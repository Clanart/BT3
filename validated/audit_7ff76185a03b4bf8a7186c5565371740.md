### Title
`ProofOfInclusion::valid()` Never Verifies Against an External Root Hash — Forged Inclusion Proofs Pass Validation - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate contains a tautological final check that makes the function self-referential: it only verifies internal consistency of the proof chain but never checks the computed root against any external, trusted tree root. An attacker can construct a fully forged `ProofOfInclusion` — claiming any arbitrary key is in any DataLayer store — and `valid()` will return `true`.

### Finding Description

The `valid()` method in `ProofOfInclusion` walks each layer, computing `calculate_internal_hash(existing_hash, side, other_hash)` and comparing it to `layer.combined_hash`. After the loop, it performs a final check:

```rust
existing_hash == self.root_hash()
```

However, `root_hash()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← returns the last layer's combined_hash
    } else {
        self.node_hash
    }
}
```

After the loop body, `existing_hash` has been set to `calculated_hash`, and the loop only continues if `calculated_hash == layer.combined_hash`. Therefore, at loop exit, `existing_hash` is guaranteed to equal `last_layer.combined_hash`. Since `root_hash()` also returns `last_layer.combined_hash`, the final check `existing_hash == self.root_hash()` is a tautology — it is always `true` whenever the loop completes without returning `false`.

The result is that `valid()` never verifies the proof against any external, trusted root. An attacker who knows the structure of `ProofOfInclusion` (a `Streamable` type, fully deserializable from bytes) can craft a proof with:
- An arbitrary `node_hash` (claiming any leaf is in the tree)
- Layers whose `combined_hash` values are each correctly computed from the previous hash and a chosen `other_hash`

Such a proof passes `valid()` with `true`, but its `root_hash()` will not match the actual DataLayer store root.

This is the direct analog of the external report's vulnerability: just as `computeValidatorUptime` distributes uptime across epochs without verifying actual activity in each epoch, `ProofOfInclusion::valid()` validates the proof chain across layers without verifying the chain terminates at the actual tree root.

### Impact Explanation

The `ProofOfInclusion` type is exposed via Python bindings (`datalayer.pyi` exposes `valid()` and `root_hash()` as separate methods) and is `Streamable` (deserializable from untrusted bytes). If any DataLayer verifier calls only `proof.valid()` to accept a proof — which the API name strongly implies is sufficient — an attacker can forge inclusion of arbitrary key-value pairs in any DataLayer store. This lets untrusted input prove invalid state, matching the allowed High impact: "DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."

### Likelihood Explanation

`ProofOfInclusion` is a public, `Streamable`, Python-exposed type. The `valid()` method's name implies complete validation. Any DataLayer client that calls `proof.valid()` without also separately checking `proof.root_hash() == expected_root` is vulnerable. The API design provides no combined "verify against root" method, making incorrect usage the natural path.

### Recommendation

`valid()` must accept an expected root hash parameter and verify the computed chain root against it:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root  // ← compare against external trusted root
}
```

Alternatively, update `valid()` to require an `expected_root: &Hash` argument and remove the tautological `existing_hash == self.root_hash()` check.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side, ProofOfInclusion, ProofOfInclusionLayer, calculate_internal_hash};

// Attacker wants to forge proof that node_hash is in a tree with root R
let fake_node_hash = Hash(/* arbitrary leaf hash */);
let fake_other_hash = Hash(/* arbitrary sibling hash */);
let fake_combined = calculate_internal_hash(&fake_node_hash, Side::Right, &fake_other_hash);

let forged_proof = ProofOfInclusion {
    node_hash: fake_node_hash,
    layers: vec![ProofOfInclusionLayer {
        other_hash_side: Side::Right,
        other_hash: fake_other_hash,
        combined_hash: fake_combined,  // correctly computed → loop passes
    }],
};

// valid() returns true even though forged_proof.root_hash() ≠ actual tree root
assert!(forged_proof.valid());
// forged_proof.root_hash() == fake_combined, not the real store root
```

The `valid()` call returns `true` because the loop check passes and the final `existing_hash == self.root_hash()` is a tautology. The verifier is deceived into accepting a forged inclusion proof.
