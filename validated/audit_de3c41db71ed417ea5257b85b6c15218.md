### Title
Non-EVM-native (IBC/CosmWasm) token balances orphaned at self-destructed contract addresses enable theft via CREATE2 address reuse - (`x/evm/statedb/statedb.go`)

### Summary
`StateDB.Commit()` only burns the EVM-denom balance of self-destructed contracts. Non-EVM-native tokens (IBC, CosmWasm bridge tokens) held by the destroyed address are explicitly skipped and remain as orphaned Cosmos bank balances. Because `DeleteAccount` removes the auth account and clears EVM storage, the address appears completely empty to the EVM, allowing a new contract to be deployed at the same address via CREATE2. The new contract can then drain the orphaned IBC tokens, constituting unauthorized theft of Cosmos bank funds.

### Finding Description
In `x/evm/statedb/statedb.go`, the `Commit()` function processes self-destructed accounts by burning the EVM denom balance and deleting the account metadata. The code explicitly acknowledges it skips non-EVM-native tokens:

```go
// Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.
if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
    coin := sdk.NewCoin(s.evmDenom, sdkmath.NewIntFromBigInt(remaining.ToBig()))
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct balance")
    }
}
if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil {
    return errorsmod.Wrap(err, "failed to delete account")
}
writeCache()
``` [1](#0-0) 

After `Commit()` processes a self-destructed account:
- The auth account is removed via `accountKeeper.RemoveAccount` inside `DeleteAccount`
- EVM contract storage is cleared
- The EVM denom balance is burned via `SubBalance`
- **IBC tokens and other non-EVM-native Cosmos bank balances remain at the address in the bank module** [2](#0-1) 

Since the auth account is removed and the EVM denom balance is zero, the address satisfies the EVM's emptiness check for CREATE2 redeployment: `GetNonce(address) == 0` and `GetCodeHash(address) == emptyCodeHash`. The EVM will permit a new contract to be deployed at the same address.

When a new contract is deployed at the same address via CREATE2:
1. A new auth account is created at the address
2. The bank module still holds the IBC tokens at that address (bank balances are keyed by address only)
3. The new contract can invoke an IBC precompile or native action to transfer the IBC tokens to the attacker

The `DeleteAccount` function confirms it only removes auth metadata and storage, never touching bank balances: [3](#0-2) 

The `SubBalance` path used for the EVM denom burn goes through `SendCoinsFromAccountToModule` + `BurnCoins`, which only operates on the single specified denom: [4](#0-3) 

### Impact Explanation
An attacker who controls a CREATE2 factory can deploy a contract, allow IBC tokens to accumulate at the deterministic address, trigger SELFDESTRUCT, and then redeploy at the same address to drain the orphaned IBC tokens. This is unauthorized theft of Cosmos bank funds enabled directly by the EVM state transition and `stateDB` commit logic. The impact matches: **Critical — unauthorized theft of Cosmos bank funds through Ethermint stateDB/native action logic.**

### Likelihood Explanation
The attack requires:
1. A contract deployed via CREATE2 that holds IBC tokens (common in cross-chain DeFi)
2. SELFDESTRUCT to be called on the contract — either by the attacker (who controls the contract), by a legitimate emergency-shutdown function, or by a reentrancy/logic bug
3. The attacker to redeploy at the same CREATE2 address (requires the same deployer address, salt, and init code hash)

All three steps are reachable via unprivileged Ethereum transactions. No validator compromise, governance action, or privileged key is required. The scenario is realistic for any cross-chain DeFi contract that (a) holds IBC tokens, (b) is deployed via CREATE2, and (c) exposes a SELFDESTRUCT path.

### Recommendation
In `StateDB.Commit()`, when processing self-destructed accounts, drain **all** bank balances at the address before calling `DeleteAccount`, not just the EVM denom. The keeper already exposes the necessary primitives:

```go
// Drain ALL bank balances, not just the EVM denom
allBalances := s.keeper.GetAllBalances(cacheCtx, cosmosAddr)
for _, coin := range allBalances {
    if coin.Amount.IsPositive() {
        if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
            return errorsmod.Wrap(err, "failed to burn post-selfdestruct non-EVM balance")
        }
    }
}
```

This ensures no Cosmos bank funds survive at a deleted address, eliminating the orphaned-balance theft vector.

### Proof of Concept
1. Attacker deploys factory contract `F` on Ethermint.
2. Factory `F` deploys contract `A` at address `X` via `CREATE2(salt=S, initCode=I)`.
3. IBC tokens (e.g., `ibc/...`) accumulate at address `X` — either sent by users interacting with `A`, or pre-funded by the attacker to the predicted address.
4. Attacker calls `SELFDESTRUCT` on contract `A` (valid if `A` was created in the same tx under EIP-6780, or in any tx pre-Prague).
5. `StateDB.Commit()` executes: EVM denom balance at `X` is burned; `DeleteAccount` removes the auth account and storage. IBC tokens remain in the bank module at `X`.
6. Address `X` now has nonce=0, codeHash=emptyHash, EVM balance=0 — the EVM considers it empty.
7. Attacker calls factory `F` to redeploy contract `B` at `X` via `CREATE2(salt=S, initCode=I)`. Redeployment succeeds because the address appears empty.
8. A new auth account is created at `X`. The bank module shows the IBC tokens at `X`.
9. Contract `B` calls an IBC transfer precompile (or any native action) to move the IBC tokens to the attacker's address. Theft complete. [5](#0-4)

func (k *Keeper) SubBalance(ctx sdk.Context, addr sdk.AccAddress, coin sdk.Coin) (uint256.Int, error) {
	coins := sdk.NewCoins(coin)
	prevBalance := k.GetBalance(ctx, addr, coin.Denom)
	if err := k.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, coins); err != nil {
		return uint256.Int{}, err

		return uint256.Int{}, err

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account
//
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
