### Title
Stale Pre-Transaction Sequencer Balance in `complete_fee_transfer_flow` Overwrites Explicit Transfer, Corrupting State Diff and Block Hash — (File: `crates/blockifier/src/concurrency/fee_utils.rs`)

---

### Summary

In concurrent execution mode, `complete_fee_transfer_flow` reads the sequencer's fee-token balance from the versioned state pinned **before** the current transaction's own writes. When a transaction's main execution explicitly transfers tokens to the sequencer address **and** also pays a fee, `add_fee_to_sequencer_balance` computes `pre_tx_balance + actual_fee` and inserts it into `execution_output.state_diff`, overwriting the correct value `pre_tx_balance + transfer_amount` that was already present. The explicit transfer amount is silently dropped from the state diff, producing a wrong storage value for the sequencer's fee-token balance, a wrong `state_diff_commitment`, and a wrong block hash.

---

### Finding Description

**Root cause — stale balance read in `complete_fee_transfer_flow`:**

In `commit_tx`, after validation passes, `complete_fee_transfer_flow` is called with `tx_versioned_state` pinned at `tx_index`:

```rust
let mut tx_versioned_state = self.state.pin_version(tx_index);
// ...
complete_fee_transfer_flow(
    &tx_context,
    tx_execution_info,
    &mut execution_output.state_diff,
    &mut tx_versioned_state,
    tx.as_ref(),
);
``` [1](#0-0) 

Inside `complete_fee_transfer_flow`, the sequencer balance is read from `state` (i.e., `tx_versioned_state`):

```rust
let sequencer_balance = state.get_fee_token_balance(
    tx_context.block_context.block_info.sequencer_address,
    tx_context.fee_token_address()
)...;
``` [2](#0-1) 

`VersionedStateProxy::get_fee_token_balance` reads from writes at indices `0..tx_index-1` — it does **not** see the current transaction's own writes at `tx_index`. This is the balance from **before** the current transaction.

**How the explicit transfer write is established:**

During `execute_raw` in concurrent mode, `concurrency_execute_fee_transfer` creates a nested transactional state, sets the initial sequencer balance to `Felt::ZERO`, executes the fee transfer, then **removes** the sequencer balance writes before committing:

```rust
let storage_writes = &mut transfer_state.cache.get_mut().writes.storage;
storage_writes.remove(&(fee_address, sequencer_balance_key_low));
storage_writes.remove(&(fee_address, sequencer_balance_key_high));
transfer_state.commit();
``` [3](#0-2) 

The outer transactional state's writes are **not** touched by this removal. If the main execution included an explicit ERC20 `transfer(sequencer_address, T)`, the outer state's writes already contain:

```
storage[(fee_token_address, sequencer_balance_key_low)] = pre_tx_balance + T
```

This value is stored in `execution_output.state_diff`.

**The overwrite:**

`add_fee_to_sequencer_balance` then computes `new_value_low = pre_tx_balance + actual_fee` and unconditionally inserts it:

```rust
if sequencer_balance_low_as_u128 != new_value_low {
    state_diff.storage.insert(
        (fee_token_address, sequencer_balance_key_low),
        Felt::from(new_value_low)
    );
}
```
