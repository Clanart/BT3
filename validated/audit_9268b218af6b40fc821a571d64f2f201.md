### Title
Unbounded L2-to-L1 Message `payload_size` Allows Unprovable Blocks — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/deprecated_execute_syscalls.cairo` and `execution/syscall_impls.cairo`)

---

### Summary

The StarkNet OS Cairo code processes L2-to-L1 messages via `send_message_to_l1` syscall without enforcing any upper bound on `payload_size`. Unlike calldata, constructor calldata, and signature arrays — all of which are explicitly checked against `SIERRA_ARRAY_LEN_BOUND` — the message payload size is accepted verbatim and passed directly to `memcpy`. A malicious Cairo 0 (deprecated) contract can set `payload_size` to an arbitrarily large felt value, causing the OS to attempt a `memcpy` of an astronomically large number of elements, exhausting Cairo VM step/memory limits and rendering the block unprovable.

---

### Finding Description

**Root cause 1 — Deprecated (Cairo 0) path:**

In `execute_deprecated_syscalls`, the `SendMessageToL1` handler reads `payload_size` directly from the contract-controlled syscall struct and passes it to `memcpy` with no bound check:

```cairo
let syscall = [cast(syscall_ptr, SendMessageToL1SysCall*)];

assert [outputs.messages_to_l1] = MessageToL1Header(
    from_address=execution_context.execution_info.contract_address,
    to_address=syscall.to_address,
    payload_size=syscall.payload_size,   // ← attacker-controlled, unchecked
);
memcpy(
    dst=outputs.messages_to_l1 + MessageToL1Header.SIZE,
    src=syscall.payload_ptr,
    len=syscall.payload_size,            // ← passed directly to memcpy
);
``` [1](#0-0) 

**Root cause 2 — Cairo 1 path:**

In `execute_send_message_to_l1`, `payload_size` is computed as a felt subtraction of two contract-supplied pointers with no range check:

```cairo
tempvar payload_start = request.payload_start;
tempvar payload_size = request.payload_end - payload_start;  // ← felt subtraction, no assert_nn_le

memcpy(
    dst=outputs.messages_to_l1 + MessageToL1Header.SIZE, src=payload_start, len=payload_size
);
``` [2](#0-1) 

**Contrast with other array sizes that ARE bounded:**

Every other variable-length array in the OS is explicitly checked:

```cairo
assert_nn_le(tx_execution_context.calldata_size, SIERRA_ARRAY_LEN_BOUND - 1);
``` [3](#0-2) 

```cairo
assert_nn_le(constructor_calldata_size, SIERRA_ARRAY_LEN_BOUND - 1);
``` [4](#0-3) 

```cairo
assert_nn_le(signature_len, SIERRA_ARRAY_LEN_BOUND - 1);
``` [5](#0-4) 

`SIERRA_ARRAY_LEN_BOUND = 4294967296` (2^32) is defined in `constants.cairo`: [6](#0-5) 

The `payload_size` for L2-to-L1 messages has no analogous check anywhere in the OS code.

**Secondary surface — virtual OS output hashing:**

In `os_utils__virtual.cairo`, `output_message_to_l1_hashes` reads `message_header.payload_size` from the message buffer to advance its pointer and compute the hash:

```cairo
let message_size = MessageToL1Header.SIZE + message_header.payload_size;
let (message_hash) = poseidon_hash_many(n=message_size, elements=messages_ptr_start);
...
return output_message_to_l1_hashes(
    messages_ptr_start=&messages_ptr_start[message_size], messages_ptr_end=messages_ptr_end
);
``` [7](#0-6) 

If `payload_size` is a huge felt, `poseidon_hash_many` is called with an astronomically large `n`, and the pointer advance skips far past `messages_ptr_end`. The equality termination check `messages_ptr_start == messages_ptr_end` is never satisfied, causing unbounded recursion and VM failure.

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

A malicious Cairo 0 contract sets `payload_size` to a value near the field prime (~2^251). The OS attempts `memcpy` of ~2^251 elements. The Cairo VM exhausts its step/memory budget. The block containing this transaction cannot be proven. If the sequencer commits this block to L1 before detecting the failure, the network halts until the block is reverted. Even if the sequencer detects the failure in simulation, it must re-run the block, and repeated attacks force continuous re-execution, degrading liveness.

---

### Likelihood Explanation

**Medium-High.** Cairo 0 contracts are still supported (the `execute_deprecated_syscalls` path is live). Any unprivileged user can deploy a Cairo 0 contract and invoke a function that writes an arbitrary felt to `payload_size` in the syscall struct. The `SEND_MESSAGE_TO_L1_GAS_COST = 14470` is a flat fee with no per-element component, so there is no gas-based deterrent at the syscall level. [8](#0-7) 

---

### Recommendation

Add an explicit upper bound check on `payload_size` in both the deprecated and Cairo 1 `send_message_to_l1` handlers, mirroring the pattern used for all other array sizes:

```cairo
// In execute_send_message_to_l1 (syscall_impls.cairo):
tempvar payload_size = request.payload_end - payload_start;
assert_nn_le(payload_size, SIERRA_ARRAY_LEN_BOUND - 1);

// In deprecated_execute_syscalls.cairo:
assert_nn_le(syscall.payload_size, SIERRA_ARRAY_LEN_BOUND - 1);
```

Similarly, add a bound check in `output_message_to_l1_hashes` before using `message_header.payload_size` to advance the pointer.

---

### Proof of Concept

1. Deploy a Cairo 0 contract with the following logic in its entry point:
   ```
   // Write syscall selector
   [syscall_ptr] = SEND_MESSAGE_TO_L1_SELECTOR
   // Write to_address (any valid L1 address)
   [syscall_ptr + 1] = 0xdeadbeef
   // Write payload_ptr (any valid pointer)
   [syscall_ptr + 2] = some_valid_ptr
   // Write payload_size = P - 1 (near field prime, huge value)
   [syscall_ptr + 3] = 3618502788666131213697322783095070105623107215331596699973092056135872020480
   ```
2. Submit a transaction calling this contract function.
3. The sequencer includes the transaction in a block.
4. The OS executes `execute_deprecated_syscalls`, reaches the `SEND_MESSAGE_TO_L1_SELECTOR` branch, reads `payload_size ≈ 2^251`, and calls `memcpy(..., len=2^251)`.
5. The Cairo VM exhausts its step limit. The block is unprovable.
6. If committed to L1, the network halts pending revert.
