### Title
Missing Duplicate-Key Sanity Check in `batch_insert()` Fast Path Corrupts DataLayer Merkle Tree Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert()` skips the duplicate-key and duplicate-hash guards that `insert()` enforces for all items beyond the first two when the tree already has ≥ 2 leaves, and for every item in the fast path when the tree starts with ≥ 2 leaves. Passing a batch that contains a repeated `KeyId` (or a `KeyId` already present in the tree) silently writes two leaf nodes for the same key into the blob, corrupts the `block_status_cache`, and produces a wrong Merkle root that is committed on-chain.

---

### Finding Description

`insert()` guards against both conditions before touching the blob:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert()` uses `insert()` only for the first two items when the tree starts with ≤ 1 leaf. Every subsequent item goes through the fast path, which calls `insert_entry_to_blob()` directly with no duplicate check:

```rust
// lines 587-602  — no KeyAlreadyPresent / HashAlreadyPresent guard
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob()` calls `block_status_cache.add_leaf()`, which performs a plain `HashMap::insert` — silently overwriting the existing cache entry for the duplicate key:

```rust
// lines 188-193
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);        // overwrites silently
    self.leaf_hash_to_index.insert(leaf.hash, index); // overwrites silently
}
``` [3](#0-2) 

The result is:
1. **Two leaf nodes** for the same `KeyId` exist in the raw blob and are wired into the tree's internal-node structure.
2. **The cache** (`key_to_index`) only tracks the last-written leaf; the earlier one is orphaned from the cache but still present in the blob.
3. **The root hash** is computed from the full blob structure (both leaves), so it reflects a tree that contains a duplicate key — an invalid state.
4. **`get_proof_of_inclusion`** uses the cache to locate the leaf, so it finds only the last-written copy and generates a proof against the corrupted root.

The Python binding `py_batch_insert` exposes this path directly to callers and only validates that the lengths of the two input lists match:

```rust
// lines 1503-1519
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    if keys_values.len() != hashes.len() { ... }
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [4](#0-3) 

---

### Impact Explanation

The DataLayer Merkle root is committed on-chain. A corrupted root — one that encodes a duplicate key — means:

- The on-chain commitment no longer represents a valid key-value store.
- Two different values can be "proven" for the same key by generating proofs against the two different leaf nodes.
- Inclusion proofs produced by `get_proof_of_inclusion` are valid against the corrupted root but invalid against any correctly-constructed root for the same data, causing cross-node disagreement on DataLayer state.
- `check_integrity()` would detect the corruption (`IntegrityKeyToIndexCacheLength` mismatch), but it is not called inside `batch_insert()` and is only invoked explicitly or on drop in debug mode.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state.**

---

### Likelihood Explanation

`batch_insert` is the primary bulk-load API and is called from the Python layer for every DataLayer store update that involves multiple keys. Any caller — including a DataLayer client submitting a batch of operations — that inadvertently or deliberately includes a repeated key in a batch of ≥ 3 items (when the tree already has ≥ 2 leaves, any batch size) will trigger the corruption silently. No privilege is required; the Python binding is the public API surface.

---

### Recommendation

Add the same duplicate guards at the top of the fast path in `batch_insert()`, mirroring what `insert()` does:

```rust
for ((key, value), hash) in keys_values_hashes {
    // --- add these two guards ---
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ----------------------------
    let new_leaf_index = self.get_new_index();
    ...
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

Alternatively, pre-validate the entire input vector for uniqueness before touching the blob, and update the cache eagerly so that intra-batch duplicates are also caught.

---

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn main() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();

    // Pre-populate with 2 leaves so leaf_count() > 1 → fast path is taken
    for i in 0i64..2 {
        blob.insert(KeyId(i), ValueId(i), &Hash(Bytes32::new([i as u8; 32])), InsertLocation::Auto {}).unwrap();
    }

    // Batch with a duplicate KeyId(0) — no error is returned
    let batch = vec![
        ((KeyId(2), ValueId(2)), Hash(Bytes32::new([2u8; 32]))),
        ((KeyId(0), ValueId(99)), Hash(Bytes32::new([99u8; 32]))), // duplicate key!
    ];
    blob.batch_insert(batch).unwrap(); // succeeds silently

    // Root hash now encodes two leaves for KeyId(0) — corrupted state
    // check_integrity() will return IntegrityKeyToIndexCacheLength error
    blob.check_integrity().unwrap_err();
}
```

The `batch_insert` call returns `Ok(())` despite the duplicate, and `check_integrity()` subsequently detects the corruption — confirming that the tree root is silently committed in an invalid state. [5](#0-4) [6](#0-5)

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
