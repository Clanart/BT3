### Title
Missing Duplicate Key Validation in `MerkleBlob::batch_insert` Bulk Path Corrupts Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash checks that `MerkleBlob::insert` enforces for every item beyond the first two, allowing an attacker-controlled batch to silently insert duplicate leaf nodes. The resulting tree has an inconsistent `block_status_cache`, a corrupted structural layout, and a wrong root hash — enabling forged or invalid DataLayer inclusion proofs.

### Finding Description
`MerkleBlob::insert` guards against duplicate keys and hashes at lines 369–373:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `insert` (with its guards) only for the first two items when `leaf_count <= 1`. For all remaining items — the bulk path — it calls `insert_entry_to_blob` directly, with **no duplicate check**:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` writes raw bytes to the blob but does **not** update `block_status_cache`. Consequently, if a caller passes a batch containing a duplicate `KeyId` or `Hash` when the tree already holds two or more leaves, both entries are written to the blob without error, the cache remains unaware of the duplicates, and the tree's internal-node hashes are computed over the corrupted leaf set.

This function is exposed as a public Python binding:

```rust

    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,
) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [3](#0-2)

### Impact Explanation
After a batch with duplicate keys is processed:

1. Two leaf nodes sharing the same `KeyId` exist in the blob at different `TreeIndex` positions.
2. `block_status_cache` is unaware of both, so subsequent `insert` calls cannot detect the collision.
3. The root hash is computed over the corrupted subtree, producing a hash that does not correspond to any valid committed state.
4. `get_proof_of_inclusion` returns a proof anchored to whichever duplicate leaf is found first; the proof verifies against the wrong root, enabling forged inclusion claims.
5. Two DataLayer nodes that process the same batch but differ in internal ordering can compute different root hashes, causing committed-state divergence.

This matches the allowed High impact: *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.*

### Likelihood Explanation
`batch_insert` is the primary bulk-load API used in production DataLayer operations (called directly in `test_datalayer.py` and `test_delta_file_cache`). [6](#0-5) [7](#0-6) 

Any Python caller — including the `chia-blockchain` DataLayer node that processes user-submitted key-value updates — can pass a list containing a repeated `KeyId`. Because the function accepts the input silently, the corruption is not detected until integrity checks (if any) are run later. The attacker-controlled entry path is direct: craft a DataLayer update batch with a repeated key.

### Recommendation
Add the same duplicate-key and duplicate-hash guards to the bulk path of `batch_insert` before writing each leaf to the blob:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insert_entry_to_blob logic, then update cache
}
```

Alternatively, pre-deduplicate the input vector and return an error if duplicates are detected, mirroring the contract of `insert`. Add a regression test that calls `batch_insert` with a duplicate key on a tree with two or more existing leaves and asserts an error is returned.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def h(n: int) -> bytes:
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

blob = MerkleBlob(blob=bytearray())

# Pre-populate with 3 leaves so leaf_count > 1 (bypasses the guarded path)
for i in range(3):
    blob.insert(KeyId(i), ValueId(i), h(i))

# batch_insert with a duplicate key (KeyId(0) already exists)
# Expected: KeyAlreadyPresent error
# Actual:   silently succeeds, tree is corrupted
blob.batch_insert(
    [(KeyId(0), ValueId(99)), (KeyId(100), ValueId(100))],
    [h(0), h(100)],
)

blob.calculate_lazy_hashes()
# Root hash is now computed over a tree containing two leaves with KeyId(0)
root = blob.get_root_hash()
print("Corrupted root:", root.hex())

# Proof for KeyId(0) will be anchored to one of the two duplicates
proof = blob.get_proof_of_inclusion(KeyId(0))
print("Proof valid (against corrupted root):", proof.valid())
