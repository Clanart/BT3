### Title
Missing `token::authority = user` Constraint Allows Delegate to Steal Victim Tokens via Cross-Chain Transfer — (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

### Summary
The `InitTransfer` account struct imposes no ownership constraint linking the `from` token account to the `user` signer. Because SPL Token's `transfer_checked` accepts a delegate as a valid authority, any attacker who holds a delegate allowance on a victim's token account can call `init_transfer` with `from=victim_ata`, route the tokens into the bridge vault, and emit a Wormhole VAA crediting an attacker-controlled NEAR recipient — effectively stealing the victim's funds.

### Finding Description

The `from` account is declared with only mint and token-program constraints: [1](#0-0) 

There is no `token::authority = user` guard. The `user` account is only required to be a `Signer` and a system-program-owned account: [2](#0-1) 

In `process()`, the lock-path CPI passes `self.user` as the `authority` to `transfer_checked`: [3](#0-2) 

SPL Token's `transfer_checked` succeeds when `authority` is either the account owner **or** an approved delegate with sufficient delegated amount. Because the bridge never verifies that `user == from.owner`, a delegate satisfies this check.

The Wormhole message encodes `self.user.key()` as the sender and `payload.recipient` (fully attacker-controlled) as the NEAR recipient: [4](#0-3) 

The same missing constraint applies to the burn path for bridged tokens: [5](#0-4) 

### Impact Explanation
Victim's tokens are irreversibly locked in the bridge vault (or burned). A Wormhole VAA is emitted crediting the attacker's NEAR address. Once the VAA is relayed and finalized on NEAR, the attacker receives the bridged tokens. The victim has no recourse. This is direct, cross-chain theft of user funds — **Critical** impact.

### Likelihood Explanation
The precondition is that the attacker holds a delegate allowance on the victim's token account. In practice this is realistic:
- Users routinely `approve` DeFi protocols (DEXes, lending markets, aggregators) as delegates.
- A compromised or malicious protocol that already holds a delegate allowance can silently redirect those tokens through the bridge.
- The attacker does not need any privileged role in the bridge itself; the only requirement is a standard SPL Token `approve` that the victim previously issued for any reason.

### Recommendation
Add `token::authority = user` to the `from` account constraint:

```rust
#[account(
    mut,
    token::mint = mint,
    token::token_program = token_program,
    token::authority = user,   // <-- add this
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
```

This ensures Anchor rejects any `from` account whose owner is not `user`, regardless of delegate state, before the CPI is ever reached.

### Proof of Concept
1. Victim holds `victim_ata` (owner = `victim`).
2. Victim previously called `spl_token::approve(victim_ata, delegate=attacker, amount=X)` for any reason.
3. Attacker calls `init_transfer` with:
   - `from = victim_ata`
   - `user = attacker` (signs the transaction)
   - `payload.recipient = "attacker.near"`
   - `payload.amount = X`
4. Anchor account validation passes — `from` only checks mint and token-program.
5. `transfer_checked(from=victim_ata, authority=attacker)` succeeds because attacker is a valid delegate.
6. Vault receives `X` tokens from `victim_ata`.
7. Wormhole VAA is posted encoding sender=`attacker`, recipient=`attacker.near`, amount=`X`.
8. Relayer delivers VAA to NEAR; NEAR bridge credits `attacker.near` with `X` tokens.
9. Victim's tokens are gone; attacker controls them on NEAR.
