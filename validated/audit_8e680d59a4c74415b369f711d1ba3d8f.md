### Title
Uncapped `max_fee` Computation Causes `assert_nn_le` Failure, Enabling Network Halt - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

### Summary

In `charge_fee`, the computed `max_fee` is used as the upper bound in `assert_nn_le(actual_fee, max_fee)`. Because `max_fee` is derived from user-controlled resource bounds that are individually bounded but whose product can exceed `2^128`, the `assert_nn_le` range-check invariant is violated. A malicious unprivileged transaction sender can craft resource bounds that make `max_fee ≥ 2^128`, causing the OS Cairo program to produce an unprovable constraint, which prevents any block containing that transaction from being proven and halts the network.

### Finding Description

`compute_max_possible_fee` computes:

```
max_fee = l1_gas.max_amount * l1_gas.max_price_per_unit
        + l2_gas.max_amount * (l2_gas.max_price_per_unit + tip)
        + l1_data_gas.max_amount * l1_data_gas.max_price_per_unit
``` [1](#0-0) 

`pack_resource_bounds` enforces:
- `max_amount ≤ 2^64 − 1`
- `max_price_per_unit ≥ 0` (i.e., `max_price_per_unit < 2^128` via `assert_nn`) [2](#0-1) 

There is **no upper bound** on `max_fee` itself. With `max_amount = 2` and `max_price_per_unit = 2^127`, a single term equals `2^128`, so `max_fee ≥ 2^128`.

In `charge_fee`, the only guard is:

```cairo
assert_nn_le(calldata.amount.low, max_fee);
``` [3](#0-2) 

`assert_nn_le(a, b)` is implemented as `assert_nn(b − a); assert_nn(a)`, where `assert_nn` uses the range-check builtin to verify the value is in `[0, 2^128)`. When `max_fee ≥ 2^128` and `actual_fee` is small (bounded by actual gas consumption, far below `2^128`), the subtraction `max_fee − actual_fee ≥ 2^128` fails the range-check constraint, making the proof invalid.

The sequencer cannot escape this: setting `low_actual_fee` close to `max_fee` (to satisfy the range check) would require transferring nearly `max_fee` tokens from the user, which fails the non-reverting ERC20 transfer if the user holds only enough for the real fee. `non_reverting_select_execute_entry_point_func` asserts `is_reverted = 0`, so a failed transfer also invalidates the proof. [4](#0-3) 

### Impact Explanation

Any block containing such a transaction cannot be proven. The sequencer is forced to either exclude the transaction (requiring off-chain filtering not enforced by the OS) or produce an unprovable block. A coordinated submission of such transactions across multiple blocks causes a **total network shutdown** — the network cannot confirm new transactions.

### Likelihood Explanation

Any unprivileged V3 transaction sender can trigger this by setting `max_price_per_unit` to any value ≥ `2^127` with `max_amount ≥ 2`. The resource bounds are user-supplied fields in the transaction, validated only for individual field ranges (`max_amount < 2^64`, `max_price_per_unit < 2^128`) but not for their product. No privileged access is required. The attack is cheap: the user only needs to submit a valid-looking transaction; they do not need to execute it successfully.

### Recommendation

Add an explicit cap on `max_fee` before the `assert_nn_le` call in `charge_fee`. Specifically, assert `max_fee < 2^128` (or equivalently `assert_nn(max_fee)`) immediately after computing it in `compute_max_possible_fee`, or restructure the fee comparison to avoid relying on `assert_nn_le` for values that may exceed `2^128`. Alternatively, bound `max_price_per_unit` more tightly (e.g., `assert_nn_le(max_price_per_unit, 2^64 − 1)`) in `pack_resource_bounds` so that the product of any two bounds stays within `2^128`. [5](#0-4) 

### Proof of Concept

1. Craft a V3 invoke transaction with:
   - `l1_gas_bounds.max_amount = 2`
   - `l1_gas_bounds.max_price_per_unit = 2^127`
   - All other bounds set to 0
   - User account holds only enough tokens for the real execution fee (e.g., 10,000 units)

2. `compute_max_possible_fee` returns `max_fee = 2 × 2^127 = 2^128`.

3. Sequencer includes the transaction; actual gas consumed yields `actual_fee = 10,000`.

4. In `charge_fee`: `assert_nn_le(10000, 2^128)` → `assert_nn(2^128 − 10000)` → `2^128 − 10000 ≥ 2^128` → **range-check constraint violated → proof fails**.

5. Sequencer alternative: set `low_actual_fee = 2^128 − 1` → ERC20 transfer of `2^128 − 1` tokens → user has only 10,000 → transfer reverts → `non_reverting_select_execute_entry_point_func` asserts `is_reverted = 0` → **proof fails**.

6. Block is unprovable. Repeat across blocks → **network halt**. [1](#0-0) [3](#0-2)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```
