### Title
Fee-on-Transfer Token Accounting Desync in `initTransfer` Breaks Bridge Collateralization — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol::initTransfer` records the caller-supplied `amount` in the `InitTransfer` event without verifying the actual tokens received by the contract. For fee-on-transfer ERC20 tokens, the bridge holds `amount − transferFee` but the event (and therefore the NEAR-side proof) records the full `amount`. NEAR then mints or unlocks the full `amount`, permanently undercollateralizing the bridge.

---

### Finding Description

In `initTransfer`, the non-bridge-token, non-custom-minter branch performs a plain `safeTransferFrom` and immediately emits the event with the original caller-supplied `amount`:

```solidity
// OmniBridge.sol lines 407-412
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ← requested amount, not actual received
    );
}
``` [1](#0-0) 

The event is then emitted with the same unverified `amount`:

```solidity
// OmniBridge.sol lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // ← not the actual balance delta
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

A relayer picks up this event, constructs a proof, and submits it to the NEAR bridge via `fin_transfer`. The NEAR contract decodes the `InitTransfer` proof and mints/unlocks the full recorded `amount` to the recipient:

```rust
// near/omni-bridge/src/lib.rs lines 722-732
let transfer_message = TransferMessage {
    ...
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
};
``` [3](#0-2) 

No balance-before/after check is performed on the EVM side to confirm the actual deposit. The analog to EulerEarn is exact: the recorded accounting value (`amount` in the event) diverges from the real vault balance (`amount − transferFee` held by the contract), and the discrepancy is booked on the destination chain as a full-value mint.

---

### Impact Explanation

Every `initTransfer` call with a fee-on-transfer token creates a permanent deficit: the EVM bridge holds `amount − fee` but NEAR has minted `amount`. Repeating this across many transfers (or with a high-fee token) progressively drains the EVM collateral pool. When legitimate users later bridge back from NEAR to EVM, the bridge cannot fulfill the full unlock — either reverting for the last users or, if the deficit is large enough, leaving the bridge insolvent. This matches the allowed impact: **balance/fee/accounting corruption that breaks bridge collateralization**.

---

### Likelihood Explanation

- `initTransfer` is fully permissionless — any address can call it.
- The bridge does not whitelist tokens by fee behavior; `deployToken` only requires a valid MPC signature for the metadata payload, not a guarantee of zero-fee transfer semantics.
- Several production ERC-20 tokens (e.g., tokens with redistribution or burn-on-transfer mechanics) implement transfer fees. Any such token registered with the bridge is immediately exploitable.
- The attacker needs no special role, no flash loan, and no colluding party — only a registered fee-on-transfer token and repeated `initTransfer` calls.

---

### Recommendation

After `safeTransferFrom`, measure the actual balance delta and use that as the canonical transfer amount:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived == amount, "Fee-on-transfer tokens not supported");
// or: use actualReceived as the amount in the event
```

Alternatively, explicitly document and enforce at registration time that only zero-fee ERC-20 tokens may be registered as native (lock/unlock) bridge tokens.

---

### Proof of Concept

1. A fee-on-transfer ERC-20 token `FeeToken` (2% burn on every transfer) is registered with the bridge via `deployToken` (valid MPC signature obtained through normal protocol flow).
2. Attacker calls `OmniBridge.initTransfer(FeeToken, 1_000_000, 0, 0, nearRecipient, "")`.
3. Bridge receives `980_000` tokens (`1_000_000 − 2%`), but emits `InitTransfer(..., amount=1_000_000, ...)`.
4. Relayer submits the event proof to NEAR `fin_transfer`; NEAR mints `1_000_000` bridged tokens to `nearRecipient`.
5. Bridge EVM collateral deficit: `20_000` tokens per call.
6. Attacker repeats; after 50 calls the bridge holds `0` tokens while NEAR has `50 × 1_000_000 = 50_000_000` minted tokens outstanding — bridge is fully insolvent for that token. [4](#0-3)
