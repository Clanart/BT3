### Title
Wrong Validator Registered for `MinGasMultiplier` in `ParamSetPairs` Allows Governance to Bypass Upper-Bound Enforcement, Corrupting EIP-1559 Base Fee Accounting - (File: `x/feemarket/types/params.go`)

---

### Summary

`ParamSetPairs()` in `x/feemarket/types/params.go` registers `MinGasMultiplier` with `validateMinGasPrice` instead of `validateMinGasMultiplier`. `validateMinGasPrice` only checks that the value is non-nil and non-negative; it does **not** enforce the upper bound of `1`. A governance proposal that updates `MinGasMultiplier` individually via the legacy params subspace therefore bypasses the `≤ 1` constraint, allowing any non-negative value to be stored. A value `> 1` inflates `gasWanted` in every `EndBlock`, permanently corrupting the EIP-1559 base fee. An extreme value (e.g., `1e18`) can cause `TruncateInt().Uint64()` to overflow, halting the chain.

---

### Finding Description

**Root cause — wrong validator in `ParamSetPairs`:**

In `x/feemarket/types/params.go` the pair for `MinGasMultiplier` is:

```go
paramtypes.NewParamSetPair(ParamStoreKeyMinGasMultiplier, &p.MinGasMultiplier, validateMinGasPrice),
``` [1](#0-0) 

The correct function is `validateMinGasMultiplier`, which additionally enforces `v ≤ 1`:

```go
func validateMinGasMultiplier(i interface{}) error {
    ...
    if v.GT(sdkmath.LegacyOneDec()) {
        return fmt.Errorf("value cannot be greater than 1: %s", v)
    }
    return nil
}
``` [2](#0-1) 

`validateMinGasPrice` only checks non-nil and non-negative — it accepts any value `≥ 0`: [3](#0-2) 

The same defect is present in the migration type: [4](#0-3) 

**How `MinGasMultiplier` is consumed — `EndBlock`:**

```go
minGasMultiplier := k.GetParams(ctx).MinGasMultiplier
limitedGasWanted := sdkmath.LegacyNewDec(gw).Mul(minGasMultiplier)
gasWanted = sdkmath.LegacyMaxDec(limitedGasWanted, sdkmath.LegacyNewDec(gasUsed)).TruncateInt().Uint64()
k.SetBlockGasWanted(ctx, gasWanted)
``` [5](#0-4) 

`gasWanted` is then read by `CalculateBaseFee` as `parentGasUsed`: [6](#0-5) 

With `MinGasMultiplier = 2.0`, every block's recorded `gasWanted` is doubled. The base fee calculation treats this inflated value as real gas consumption, permanently driving the base fee upward regardless of actual network load. With an extreme value (e.g., `1e18`), `limitedGasWanted.TruncateInt().Uint64()` overflows, producing a corrupted or panicking `gasWanted` that halts the chain deterministically on every block.

**`Validate()` is not called on per-key updates:**

`Params.Validate()` correctly calls `validateMinGasMultiplier`: [7](#0-6) 

But the Cosmos SDK legacy params subspace invokes only the per-key validator registered in `ParamSetPairs` when a single key is updated via governance — `Validate()` is not called. The wrong per-key validator therefore silently accepts `MinGasMultiplier > 1`.

---

### Impact Explanation

**High — fee market accounting bug:** Setting `MinGasMultiplier` to any value `> 1` (e.g., `1.5`) causes `EndBlock` to record an inflated `gasWanted` every block. `CalculateBaseFee` then computes a base fee that is systematically higher than the true network demand warrants. All EVM transaction senders are forced to pay an artificially elevated base fee; their fees are mis-accounted relative to actual gas consumption.

**Critical (secondary) — chain halt:** Setting `MinGasMultiplier` to an extreme value (e.g., `1e36`) causes `limitedGasWanted.TruncateInt().Uint64()` to overflow. If `sdkmath.Int.Uint64()` panics on out-of-range values (as is the case in Cosmos SDK), `EndBlock` panics on every block, halting all validators deterministically.

---

### Likelihood Explanation

Requires a governance proposal that targets the `MinGasMultiplier` key individually via the legacy params subspace. This is a low-likelihood event (requires a passing governance vote), but the impact is high-to-critical. The analog to the external report is exact: a governance majority can silently bypass an intended parameter bound because the wrong validator is registered, harming all users of the fee market.

---

### Recommendation

1. Replace `validateMinGasPrice` with `validateMinGasMultiplier` in the `ParamSetPairs` entry for `ParamStoreKeyMinGasMultiplier` in both `x/feemarket/types/params.go` and `x/feemarket/migrations/v4/types/params.go`.
2. Add a test that calls `validateMinGasPrice` and `validateMinGasMultiplier` with a value of `2` to confirm the distinction is enforced at the per-key level.
3. Consider adding a safe-cast guard in `EndBlock` before calling `.Uint64()` to prevent a panic if an out-of-range value is ever stored.

---

### Proof of Concept

1. Submit a governance `ParameterChangeProposal` targeting subspace `feemarket`, key `MinGasMultiplier`, value `"2.000000000000000000"`.
2. The proposal passes; the SDK calls the registered per-key validator `validateMinGasPrice("2.0")` → returns `nil` (no error, value accepted).
3. `MinGasMultiplier = 2` is stored in state.
4. Every subsequent `EndBlock`: `limitedGasWanted = actualGasWanted × 2`; `SetBlockGasWanted` stores the doubled value.
5. `CalculateBaseFee` reads the doubled `parentGasUsed`, computes a base fee as if the block were twice as full as it actually was, and the base fee rises every block even when the chain is idle.
6. All EVM transaction senders pay an inflated base fee; the fee market is permanently corrupted until governance corrects the parameter.
