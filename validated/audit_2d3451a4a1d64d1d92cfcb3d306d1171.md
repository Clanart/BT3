### Title
Permanent ETH Lock When `finTransfer` Recipient Cannot Receive Native ETH ‚Äî (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

In `OmniBridge.sol`, the `finTransfer` function handles native ETH releases by sending `payload.amount` ETH directly to `payload.recipient` via a low-level call. If the recipient is a contract that cannot receive ETH (no `receive()` or `fallback()`), the call fails and the entire transaction reverts. Because the recipient address is cryptographically bound inside the MPC-signed payload, the transfer can never be re-routed or finalized, permanently locking the corresponding ETH inside the bridge contract. There is no admin rescue path for stranded ETH.

---

### Finding Description

In `finTransfer()`, after signature verification, native ETH is dispatched as follows:

```solidity
// OmniBridge.sol lines 317-322
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
```

The recipient address is part of the Borsh-encoded, keccak256-hashed, MPC-signed blob verified at lines 289‚Äì312:

```solidity
bytes memory borshEncoded = bytes.concat(
    ...
    Borsh.encodeAddress(payload.recipient),
    ...
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
```

Because `payload.recipient` is embedded in the signed payload, no alternative recipient can be substituted without invalidating the signature. If the low-level call fails (e.g., recipient is a contract with no `receive()` / `fallback()`), `revert FailedToSendEther()` rolls back the entire transaction ‚Äî including the `completedTransfers[payload.destinationNonce] = true` write at line 287 ‚Äî so the nonce is not consumed. The transfer is therefore permanently un-finalizable: every retry will hit the same revert.

Compounding this, the contract exposes an unconditional ETH receiver:

```solidity
// OmniBridge.sol line 574
receive() external payable {}
```

with no corresponding withdrawal or sweep function, so any ETH that accumulates (from failed operations, accidental sends, or the locked-but-unreachable transfer amounts) has no exit path.

---

### Impact Explanation

**Critical ‚Äî Permanent freezing of user funds.**

A user who initiates a NEAR‚ÜíEVM native ETH bridge transfer and specifies a contract recipient that cannot receive ETH will:

1. Have their source-chain tokens burned/locked on NEAR (irreversible once the MPC signs).
2. Find that `finTransfer` on EVM always reverts for that signed payload.
3. Have no mechanism to redirect the ETH to a different address or cancel the transfer.

The ETH corresponding to that transfer remains permanently stranded inside `OmniBridge`. There is no admin `rescueETH`, `sweep`, or equivalent function in the contract.

---

### Likelihood Explanation

**Medium.** Any bridge user who specifies a smart-contract address as the EVM recipient ‚Äî including multisigs, DAOs, or protocol vaults that lack a `receive()` function ‚Äî will trigger this condition. This is a realistic user error, not a contrived edge case. The bridge UI cannot prevent it because the contract performs no pre-flight check on the recipient's ETH-receivability.

---

### Recommendation

Replace the hard-revert pattern with a pull-payment (claimable balance) model for native ETH:

```solidity
mapping(address => uint256) public claimableEth;

// inside finTransfer, replace the direct send:
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) {
        // Store for later claim instead of reverting
        claimableEth[payload.recipient] += payload.amount;
        completedTransfers[payload.destinationNonce] = true;
    }
}

function claimEth() external {
    uint256 amount = claimableEth[msg.sender];
    require(amount > 0, "Nothing to claim");
    claimableEth[msg.sender] = 0;
    (bool success, ) = msg.sender.call{value: amount}("");
    require(success, "Claim failed");
}
```

Alternatively, add an admin `rescueETH(address to, uint256 amount)` guarded by `DEFAULT_ADMIN_ROLE` as a last-resort recovery path.

### Proof of Concept

1. **Source chain (NEAR):** User calls the NEAR bridge to initiate a transfer of native ETH equivalent to EVM, specifying `recipient = address(VaultContract)` where `VaultContract` has no `receive()` or `fallback()`.
2. **MPC signing:** The NEAR bridge MPC signs a `TransferMessagePayload` that includes `recipient = VaultContract` and `tokenAddress = address(0)`.
3. **EVM `finTransfer` attempt:** Relayer submits the signed payload. Execution reaches:
   ```solidity
   (bool success, ) = payload.recipient.call{value: payload.amount}("");
   if (!success) revert FailedToSendEther();   // always reverts
   ```
4. **Permanent failure:** Every subsequent `finTransfer` call with the same signed payload reverts identically. The nonce is never consumed. The ETH remains in `OmniBridge` with no release path.
5. **User loss:** The user's source-chain tokens are permanently burned/locked; the ETH is permanently stranded in the bridge. [1](#0-0) [2](#0-1) [3](#0-2)
