Audit Report

## Title
`pay()` consumes stranded native ETH via `address(this).balance`, enabling theft of prior-caller funds via `refundETH()` — (`File: metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

The `pay` helper in `PeripheryPayments.sol` reads `address(this).balance` (the router's total native ETH) rather than only the current call's `msg.value` when deciding how much ETH to wrap as WETH. Any ETH left in the router from a prior caller who over-sent and omitted `refundETH()` is silently consumed to settle the next caller's swap. The next caller can then invoke `refundETH()` — which has no access control and drains the entire native balance — to steal the stranded principal.

## Finding Description

In `pay()`, when `token == WETH` and `payer != address(this)`, the function reads the router's total native balance:

```solidity
// PeripheryPayments.sol L73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← total ETH, not msg.value
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

`address(this).balance` includes ETH from all prior calls, not just the current `msg.value`. The function wraps exactly `value` ETH and sends WETH to the pool, leaving any remainder in the contract.

The `receive()` guard only blocks direct ETH transfers from non-WETH addresses: [2](#0-1) 

It does **not** prevent ETH from accumulating via `msg.value` in the payable entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`). All of these are declared `payable`: [3](#0-2) [4](#0-3) 

`refundETH()` has no access control and sends the entire contract balance to `msg.sender`: [5](#0-4) 

The intended usage pattern (multicall + refundETH) means users routinely over-send ETH. Any user who forgets to append `refundETH()` to their multicall leaves stranded ETH that any subsequent caller can drain.

## Impact Explanation

Direct loss of user principal. A user who over-sends ETH (e.g., sends 1 ETH for a 0.5 ETH swap) and omits `refundETH()` leaves 0.5 ETH stranded in the router. The next caller who performs any WETH-input swap with a tiny `msg.value` and then calls `refundETH()` receives the stranded 0.5 ETH. The loss scales with the amount of ETH stranded, which can be arbitrarily large. This is a direct loss of user principal with no protocol-level recovery mechanism, meeting the Critical/High threshold under the Metric OMM allowed impact gate.

## Likelihood Explanation

The protocol's intended usage pattern (multicall + refundETH) means users routinely over-send ETH. Any user who forgets to append `refundETH()` to their multicall is immediately vulnerable. An attacker can monitor the router's ETH balance on-chain and follow up with a minimal swap + `refundETH()` to drain it atomically. No special permissions or privileged access are required — any unprivileged caller can execute this.

## Recommendation

Track `msg.value` separately — store it in transient storage at the start of each payable entry point — and use that tracked value instead of `address(this).balance` in `pay()`. Only the ETH sent in the current call should be eligible for wrapping. Alternatively, wrap `msg.value` into WETH immediately upon entry and always pull WETH from the payer via `safeTransferFrom`, eliminating the native-balance ambiguity entirely.

## Proof of Concept

**Step 1 — User A over-sends ETH:**
```
User A calls multicall{value: 1e18}([
    exactInputSingle({tokenIn: WETH, amountIn: 0.5e18, ...})
    // ← no refundETH() appended
])
```
- `pay()` sees `address(this).balance = 1e18 >= 0.5e18`, wraps 0.5e18, sends WETH to pool.
- **0.5e18 ETH remains stranded in the router.**

**Step 2 — Attacker drains stranded ETH:**
```
Attacker calls multicall{value: 1 wei}([
    exactInputSingle({tokenIn: WETH, amountIn: 1 wei, ...}),
    refundETH()
])
```
- `pay()` sees `address(this).balance = 0.5e18 + 1 wei >= 1 wei`, wraps 1 wei, sends WETH to pool.
- 0.5e18 ETH remains in router.
- `refundETH()` sends **0.5e18 ETH to attacker**.
- **Attacker gains 0.5e18 ETH (User A's principal) at a cost of 1 wei + gas.**

Foundry test plan: deploy `MetricOmmSimpleRouter` with a real WETH mock and a pool mock; fund User A with 1 ETH; call `multicall{value: 1e18}` with a single `exactInputSingle` (amountIn = 0.5e18, tokenIn = WETH) and no `refundETH`; assert `address(router).balance == 0.5e18`; then call `multicall{value: 1 wei}` from attacker with `exactInputSingle` (amountIn = 1 wei) + `refundETH`; assert attacker received 0.5e18 ETH.
