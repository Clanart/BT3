### Title
Pause on `ft_on_transfer` Accidentally Blocks Fast-Transfer and UTXO Finalization, Locking Relayer and User Funds — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

The NEAR bridge's `ft_on_transfer` entry point handles three semantically distinct operations under a single pause guard: outbound deposit initiation (`InitTransfer`), inbound fast-transfer finalization (`FastFinTransfer`), and inbound UTXO finalization (`UtxoFinTransfer`). The pause decorator's exception list — `except(roles(Role::DAO, Role::UnrestrictedDeposit))` — is named and scoped for deposit control only, but it accidentally freezes the finalization paths as well. When the bridge is paused, relayers who have already disbursed liquidity for fast transfers and users who have already committed UTXO funds cannot complete their settlements on NEAR, leaving their assets unclaimable for the duration of the pause.

---

### Finding Description

`ft_on_transfer` is the single NEAR callback that token contracts invoke when tokens are sent to the bridge. It dispatches on the message type:

```rust
// near/omni-bridge/src/lib.rs  line 252-283
#[pause(except(roles(Role::DAO, Role::UnrestrictedDeposit)))]
pub fn ft_on_transfer(&mut self, sender_id: AccountId, amount: U128, msg: String) {
    ...
    match parsed_msg {
        BridgeOnTransferMsg::InitTransfer(...)      => self.init_transfer(...),
        BridgeOnTransferMsg::FastFinTransfer(...)   => self.fast_fin_transfer(...),
        BridgeOnTransferMsg::UtxoFinTransfer(...)   => self.utxo_fin_transfer(...),
        BridgeOnTransferMsg::SwapMigratedToken      => self.swap_migrated_token(...),
    }
}
```

The pause guard's exception role is `UnrestrictedDeposit` — a name that unambiguously signals the pause was designed to gate new deposits (`InitTransfer`). However, because all four message variants share the same function, the guard also silently blocks:

- **`FastFinTransfer`** — a trusted relayer has already sent tokens to the recipient on NEAR and is waiting to record the fast-transfer settlement so it can later be reimbursed via `fin_transfer`/`claim_fee`.
- **`UtxoFinTransfer`** — a user has already committed BTC/UTXO on-chain and is waiting to receive the corresponding NEAR-side tokens.

Critically, the `UnrestrictedRelayer` role — which exists precisely to give trusted relayers elevated access — is **not** in the exception list for `ft_on_transfer`. Even a relayer holding `UnrestrictedRelayer` cannot call `ft_on_transfer` while the bridge is paused.

The EVM side correctly separates pause flags (`PAUSED_INIT_TRANSFER` vs `PAUSED_FIN_TRANSFER`) so that finalization can proceed independently of deposit gating. The NEAR side has no equivalent separation for the `ft_on_transfer` path. [1](#0-0) [2](#0-1) [3](#0-2)

### Impact Explanation

**Allowed impact matched: Critical — Permanent freezing / unclaimable settlement of user or protocol funds in bridge flows.**

When the bridge is paused (a realistic emergency response):

1. **Fast-transfer relayers** have already disbursed tokens to recipients on NEAR. They cannot call `ft_on_transfer` with `FastFinTransfer` to record the settlement, so the fast-transfer record is never created and the relayer cannot later invoke `claim_fee` to recover their liquidity. Their capital is locked for the entire pause duration.

2. **UTXO users** have already broadcast and confirmed a BTC/UTXO transaction. They cannot call `ft_on_transfer` with `UtxoFinTransfer` to receive their NEAR-side tokens. The UTXO is spent and irreversible; the user's funds are unclaimable until the bridge is unpaused.

If the pause is extended (e.g., during a prolonged security incident or governance dispute), these locks become practically irrecoverable. Unlike the EVM side — where `PAUSED_FIN_TRANSFER` can be left unset while `PAUSED_INIT_TRANSFER` is active — there is no mechanism on the NEAR side to allow finalization while blocking new deposits. ---

### Likelihood Explanation

- The bridge has a dedicated `PauseManager` role, indicating pausing is an anticipated operational action.
- Any security incident, upgrade window, or anomaly detection would trigger a pause.
- Every pause event causes this bug to manifest for all in-flight fast transfers and UTXO transfers at that moment.
- The relayer cannot work around it: `UnrestrictedRelayer` is not in the exception list, and there is no alternative entry point for `FastFinTransfer` or `UtxoFinTransfer`. [2](#0-1) [6](#0-5) 

---

### Recommendation

Split `ft_on_transfer` dispatch so that finalization paths bypass the deposit pause, mirroring the EVM design:

```rust
// Option A: check message type before applying pause
pub fn ft_on_transfer(&mut self, sender_id: AccountId, amount: U128, msg: String) {
    let parsed_msg: BridgeOnTransferMsg = ...;
    match &parsed_msg {
        BridgeOnTransferMsg::InitTransfer(_) => {
            self.assert_not_paused_for_deposit(); // existing pause check
        }
        // FastFinTransfer, UtxoFinTransfer, SwapMigratedToken: no pause check
        _ => {}
    }
    ...
}

// Option B: introduce a separate pause flag (e.g., PAUSED_FAST_FIN_TRANSFER)
// and apply it only to InitTransfer, consistent with EVM's PAUSED_INIT_TRANSFER.
```

Additionally, add `Role::UnrestrictedRelayer` to the exception list for any remaining pause guard that covers finalization paths, so trusted relayers can always complete settlements. [1](#0-0) [3](#0-2)

### Proof of Concept

1. Relayer calls `ft_transfer_call` on a NEAR token contract, sending tokens to the bridge with a `FastFinTransfer` message for an in-flight transfer from Ethereum.
2. Admin (PauseManager) calls `pause` on the NEAR bridge to halt new deposits due to a security alert.
3. Relayer attempts to call `ft_transfer_call` again (or the original call is still pending) — `ft_on_transfer` panics with `"Pausable: paused"` because `FastFinTransfer` shares the same pause guard as `InitTransfer`.
4. The fast-transfer record is never written; the relayer's tokens are returned (or never accepted), but the relayer has no path to finalize the settlement and claim reimbursement.
5. UTXO user similarly calls `ft_transfer_call` with `UtxoFinTransfer` — same panic, BTC already spent, NEAR tokens undeliverable.
6. Both parties must wait indefinitely for the bridge to be unpaused, with no alternative recovery path. [1](#0-0) [7](#0-6)
