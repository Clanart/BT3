Audit Report

## Title
`block.number` L1-Anchored Value on Arbitrum Collapses Price Velocity Budget to Minimum, Causing Swap DoS — (File: `metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

## Summary
`PriceVelocityGuardExtension.beforeSwap` uses `block.number` to compute `blockDiff` between consecutive oracle price observations and scales the allowed price-change budget by `(1 + blockDiff)`. On Arbitrum, `block.number` returns the L1 block number (~12 s cadence), so any two swaps within the same L1 block produce `blockDiff = 0`, reducing the budget to its absolute minimum. Any oracle price movement exceeding `maxChangePerBlockE18` between those two L2 swaps — a routine occurrence — causes `PriceVelocityExceeded` to revert, rendering the pool's swap flow unusable for the duration of each L1 block.

## Finding Description
In `beforeSwap` (lines 55–74), the contract reads `prevBlock = s.lastUpdateBlock`, writes `s.lastUpdateBlock = uint64(block.number)`, then computes:

```solidity
uint256 blockDiff = block.number - prevBlock;          // line 63
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);  // line 70
if (actualSq > allowedSq) revert PriceVelocityExceeded(actualSq, allowedSq);   // line 72
```

On Arbitrum, `block.number` is the L1 block number. Approximately 48 L2 blocks are produced per L1 block. When Swap A and Swap B both land within the same L1 block, `blockDiff = 0` and `allowedSq = maxChange²`. If the oracle mid-price moved by more than `maxChangePerBlockE18` between those two L2 blocks — which is expected when the parameter is calibrated for L2-block cadence as the NatSpec implies — `actualSq > allowedSq` and the swap reverts. The same stale-block stamp exists in `setLastMidPrice` (line 32), so admin resets also use the L1 block number. No existing guard compensates for the L1/L2 block-number mismatch.

## Impact Explanation
Any pool on Arbitrum (or another L2 where `block.number` returns L1 block numbers) that attaches this extension will have `beforeSwap` revert for every swap where the oracle price moved more than `maxChangePerBlockE18` since the last swap within the same L1 block. Because each L1 block spans ~48 L2 blocks, this condition is triggered routinely during normal oracle price updates, making the pool's swap flow **unusable** for ~12 seconds per L1 block. This constitutes broken core pool functionality causing an unusable swap flow, matching the allowed impact gate.

## Likelihood Explanation
Medium. The two required conditions are both routine in production: (1) the pool is deployed on an L2 where `block.number` returns L1 block numbers (Arbitrum is the primary example, and the protocol does not restrict deployment to L1); (2) the oracle price moves by more than `maxChangePerBlockE18` between two swaps within the same L1 block, which is expected when the parameter is calibrated for L2-block cadence as the NatSpec naming implies. No privileged actor or malicious setup is required; any ordinary unprivileged swap attempt is sufficient to trigger the revert.

## Recommendation
Replace `block.number` with `block.timestamp` in both `beforeSwap` (lines 58, 63) and `setLastMidPrice` (line 32), and rename the parameter to `maxChangePerSecondE18`. Timestamps are consistent across L1 and L2 environments and directly reflect elapsed real time, preserving the velocity-scaling intent. Alternatively, document explicitly that the extension must not be used on chains where `block.number` does not return the chain's own block number, and provide a chain-specific override (e.g., `ArbSys(0x0000000000000000000000000000000000000064).arbBlockNumber()` for Arbitrum).

## Proof of Concept
1. Deploy a pool on Arbitrum with `PriceVelocityGuardExtension`; set `maxChangePerBlockE18 = 1e16` (1% per L2 block, calibrated per NatSpec intent).
2. **Swap A** executes at L2 block N (L1 block M). State: `lastMidPriceX64 = P`, `lastUpdateBlock = M`.
3. Oracle price updates at L2 block N+10 (still L1 block M): new mid = `P × 1.03` (3% move, normal for 10 L2 blocks).
4. **Swap B** executes at L2 block N+10 (still L1 block M):
   - `prevBlock = M`, `block.number = M` → `blockDiff = 0`
   - `allowedSq = (1e16)² × 1 = 1e32`
   - `changeE18 = 3e16`, `actualSq = 9e32`
   - `9e32 > 1e32` → **revert `PriceVelocityExceeded`**
5. Every subsequent swap within L1 block M that observes any oracle movement above 1% also reverts. The pool is effectively frozen for ~12 seconds per L1 block.
