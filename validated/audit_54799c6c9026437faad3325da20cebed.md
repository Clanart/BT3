### Title
Unvalidated P2P-supplied `transaction_hash` stored verbatim, corrupting `transaction_hash_to_idx` table and RPC hash lookups — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

`TransactionStreamFactory::parse_data_for_block` accepts `FullTransaction` messages from an untrusted P2P peer and pushes the peer-supplied `transaction_hash` field directly into `BlockBody::transaction_hashes` without verifying it equals `hash(transaction, chain_id)`. The body is then written to storage verbatim, permanently binding the forged hash in `transaction_hash_to_idx_table` and `TransactionMetadata::tx_hash`. The code itself acknowledges the missing check with a TODO comment.

---

### Finding Description

**Entrypoint:** `TransactionStreamFactory::parse_data_for_block` [1](#0-0) 

The destructured `transaction_hash` from the peer's `FullTransaction` is pushed unconditionally:

```rust
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
```

No call to `get_transaction_hash(tx, chain_id)` is made; no comparison against the committed `transaction_commitment` in the stored header is performed.

**Storage path:** `write_to_storage` → `append_body` → `write_transactions` [2](#0-1) 

`write_transactions` iterates the `BlockBody` and stores the peer-supplied hash in two places without any recomputation or guard:

```rust
transaction_hash_to_idx_table.insert(txn, tx_hash, &transaction_index)?;   // line 507
TransactionMetadata { tx_location, tx_output_location, tx_hash: *tx_hash } // line 511
``` [3](#0-2) 

There is no point in the entire call chain — `parse_data_for_block`, `write_to_storage`, `append_body`, or `write_transactions` — that recomputes or validates the hash.

---

### Impact Explanation

A malicious P2P peer that sends `FullTransaction { transaction, transaction_output, transaction_hash: forged_hash }` causes the node to permanently store:

- `transaction_hash_to_idx_table[forged_hash] = TransactionIndex(block_number, offset)` — the forged hash is the only key that resolves to this transaction.
- `TransactionMetadata::tx_hash = forged_hash` — the metadata record carries the wrong hash.

Consequences:
1. `starknet_getTransactionByHash(forged_hash)` returns the transaction (wrong hash→tx mapping served authoritatively to clients).
2. `starknet_getTransactionByHash(real_hash)` returns `None` — the real hash is never indexed.
3. The stored hashes are inconsistent with the `transaction_commitment` committed in the block header, so any local re-derivation of the block hash from stored data will diverge from the canonical value.

This falls squarely under: **High — RPC returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

Any node participating in P2P sync is exposed. The attacker only needs to be a reachable peer that responds to a transaction query. The precondition (a header with `n_transactions=N` already stored) is the normal operating state of a syncing node. The TODO comment confirms the developers are aware the validation is absent.

---

### Recommendation

In `parse_data_for_block`, after destructuring `FullTransaction`, recompute the expected hash and reject the peer if it does not match:

```rust
let expected_hash = transaction.calculate_hash(chain_id);
if transaction_hash != expected_hash {
    return Err(ParseDataError::BadPeer(BadPeerError::InvalidTransactionHash {
        expected: expected_hash,
        got: transaction_hash,
    }));
}
```

The `chain_id` is available from the stored header or a node-level config parameter already accessible in the sync context.
