### Title
Incomplete `locked_tokens` Update in `process_fin_transfer_to_other_chain` Fast-Transfer Path Breaks Bridge Collateralization - (File: `near/omni-bridge/src/lib.rs`)

### Summary

In `process_fin_transfer_to_other_chain`, when a fast transfer has already been executed by a relayer, the destination-chain `locked_tokens` counter is only incremented by `fee.fee` but never by `amount_without_fee`. This is the direct analog of the reported `scaleVariables()` bug: a critical accounting variable is only partially updated under one branch of a conditional, leaving the bridge's collateralization ledger understated by `amount_without_fee` for every fast-finalized cross-chain transfer.

### Finding Description

`process_fin_transfer_to_other_chain` handles the case where a proof arrives from the origin chain (e.g., Eth) for a transfer whose destination is a non-NEAR chain (e.g., Sol). The function always locks `fee.fee` on the destination chain unconditionally, then branches on whether a fast transfer was already executed:

```
// Always executed
self.lock_tokens_if_needed(destination_chain, &token, transfer_message.fee.fee.into());

// Only in the NON-fast-transfer branch
} else {
    self.lock_tokens_if_needed(
        destination_chain,
        &token,
        transfer_message.amount_without_fee()...,
    );
    None
};
```

In the fast-transfer branch the `amount_without_fee` lock is skipped entirely. The relayer is paid back on NEAR via `send_tokens`, but the `locked_tokens[destination_chain][token]` counter is only increased by `fee`, not by `fee + amount_without_fee = amount`. [1](#0-0) 

Compare with the non-fast path in `process_fin_transfer_to_near`, which correctly unlocks the full `amount` from the origin chain: [2](#0-1) 

And with `init_transfer_internal`, which correctly locks the full `amount` (including fee) on the destination chain when a transfer is initiated: [3](#0-2) 

The `locked_tokens` map is the bridge's authoritative ledger of outstanding token supply per chain. `unlock_tokens` enforces `available >= amount` before decrementing: [4](#0-3) 

### Impact Explanation

For every fast-finalized transfer to a non-NEAR destination chain, `locked_tokens[destination_chain][token]` is understated by `amount_without_fee`. When the recipient later bridges those tokens back to NEAR (or any other chain), `process_fin_transfer_to_near` calls `unlock_tokens_if_needed(origin_chain=Sol, token, amount)`. If the cumulative understatement causes `locked_tokens[Sol][token] < amount`, the call panics with `ERR_INSUFFICIENT_LOCKED_TOKENS`, permanently freezing the recipient's funds in the bridge.

This matches the **High** allowed impact: *balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value*, and **Critical**: *permanent freezing or irrecoverable lock of user funds*. [5](#0-4) 

### Likelihood Explanation

Fast transfers are a core protocol feature actively used by relayers. Any bridged token (non-native to the destination chain) flowing through the fast-transfer path accumulates the understatement. A single fast transfer of 1000 tokens with a 10-token fee leaves `locked_tokens[Sol]` understated by 990. The next user bridging 990+ tokens back from Sol to NEAR will be permanently blocked. No privileged access is required; any unprivileged user can trigger the fast-transfer path via `ft_on_transfer` with a `FastFinTransferMsg`. [6](#0-5) 

### Recommendation

In `process_fin_transfer_to_other_chain`, lock `amount_without_fee` on the destination chain unconditionally (before the fast-transfer branch check), mirroring the non-fast path. The fast-transfer branch should not skip this lock; the tokens are outstanding on the destination chain regardless of who fronted them:

```rust
// Lock fee unconditionally (already present)
self.lock_tokens_if_needed(destination_chain, &token, transfer_message.fee.fee.into());

// Also lock amount_without_fee unconditionally
self.lock_tokens_if_needed(
    destination_chain,
    &token,
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
);

// Then branch on fast-transfer status (no additional locking needed)
let recipient = if let Some(status) = self.get_fast_transfer_status(&fast_transfer.id()) {
    ...
```

### Proof of Concept

**Setup:** Bridged token `T` originates from Eth, deployed on Sol by the bridge. `locked_tokens[(Sol, T)] = 0`.

1. **Fast transfer initiated:** Relayer calls `ft_on_transfer` with `FastFinTransferMsg` for a transfer of 1000 T (fee = 100) from Eth to Sol. Relayer pays out 900 T to recipient on Sol.

2. **Finalization:** Relayer submits Eth proof via `fin_transfer`. `fin_transfer_callback` → `process_fin_transfer_to_other_chain`:
   - `unlock_tokens_if_needed(Eth, T, 1000)` → Eth locked decremented ✓
   - `lock_tokens_if_needed(Sol, T, 100)` → Sol locked = 100 (fee only)
   - Fast transfer found → `amount_without_fee` lock is **skipped**
   - `send_tokens(T, relayer, 900)` on NEAR ✓
   - **Result:** `locked_tokens[(Sol, T)] = 100` (should be 1000)

3. **Recipient bridges back:** Recipient bridges 900 T from Sol to NEAR. `fin_transfer_callback` → `process_fin_transfer_to_near`:
   - `unlock_tokens_if_needed(Sol, T, 900)` → requires `locked_tokens[(Sol, T)] >= 900`
   - **Actual:** `locked_tokens[(Sol, T)] = 100 < 900`
   - **Panics:** `ERR_INSUFFICIENT_LOCKED_TOKENS`
   - **Recipient's 900 T are permanently frozen.** [7](#0-6) [2](#0-1)

    fn unlock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(available) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        require!(
            available >= amount,
            TokenLockError::InsufficientLockedTokens.as_ref()
        );

        let remaining = available - amount;
        self.locked_tokens.insert(&key, &remaining);

        LockAction::Unlocked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }
