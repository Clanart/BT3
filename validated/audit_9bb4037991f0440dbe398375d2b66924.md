### Title
Missing shard_id position binding in `set_state_header` allows cross-shard state corruption via Merkle-position-blind chunk proof — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` accepts a `ShardStateSyncResponseHeader` for shard A containing shard B's chunk because the Merkle inclusion check (`verify_path`) is position-blind and no explicit `chunk.shard_id() == shard_id` guard exists. A malicious state-sync peer can cause shard A to be populated with shard B's trie nodes.

### Finding Description
In `set_state_header` (`chain/chain/src/state_sync/adapter.rs`, lines 368–531), the chunk Merkle inclusion check is:

```rust
if !verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
) { ... }
```

`verify_path` only checks that `compute_root_from_path(proof, hash(item)) == root`. It does **not** check which leaf index the item occupies. For a block with shards `[A, B]`, the Merkle proof for chunk B at index 1 is `[Left(hash_A)]`. Supplying chunk B with this proof satisfies the check regardless of the claimed `shard_id=A`.

No other check in `set_state_header` compares `chunk.shard_id()` to the `shard_id` parameter:
- `validate_chunk_proofs` checks body/header hash consistency only — no shard_id comparison
- Receipt proofs are validated against the `shard_id` parameter (A), not the chunk's shard_id — an attacker with chain access can supply correct receipts for shard A
- `validate_state_root_node` checks the attacker-supplied node against `chunk_inner.prev_state_root()` (shard B's root) — passes when the attacker supplies a valid node for shard B

After `set_state_header` stores the header under `StateHeaderKey(A, sync_hash)` with shard B's `prev_state_root`, `set_state_part` extracts that root and validates attacker-supplied shard B parts against it. `apply_state_part` then writes shard B's trie nodes under shard A's `ShardUId`. Finally, `set_state_finalize` applies shard B's chunk to shard A's trie.

### Impact Explanation
Shard A's trie is populated with shard B's nodes. The exact corrupted value is shard A's installed `prev_state_root` (shard B's root instead of shard A's root). All subsequent transaction and receipt execution on shard A operates against the wrong state, causing the node to diverge from the canonical chain. The corruption is persistent (written to RocksDB).

### Likelihood Explanation
Any network peer can act as a state-sync provider — no validator or operator privilege is required. The attacker needs to track shard B (to supply valid state parts) and have access to canonical block data (to supply valid receipt proofs for shard A). Both are achievable by an unprivileged full node.

### Recommendation
Add an explicit shard_id binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, consider replacing `verify_path` with `verify_path_with_index` (passing the shard index derived from `shard_id`) to enforce positional binding in the Merkle tree.

### Proof of Concept
A cargo test that:
1. Builds a two-shard block with chunks for shard 0 and shard 1
2. Obtains the `ShardStateSyncResponseHeader` for shard 1 (with shard 1's chunk and its Merkle proof)
3. Calls `set_state_header(shard_id=0, sync_hash, header_for_shard_1)`
4. Asserts that an error is returned (currently it would succeed, demonstrating the bug)
