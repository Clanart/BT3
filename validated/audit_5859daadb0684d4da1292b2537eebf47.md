### Title
Token-2022 Transfer-Fee Mint Bypasses Vault Collateralization in `init_transfer` — (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

---

### Summary

`InitTransfer::process` calls `transfer_checked(payload.amount)` on any Token-2022 mint, including those with the `TransferFee` extension. The Token-2022 program withholds the fee from the destination (vault), so the vault receives `payload.amount - fee` while the Wormhole VAA encodes the full `payload.amount`. NEAR credits the full amount to the recipient. An attacker who controls the mint's `withdraw_withheld_authority` can then drain the withheld balance from the vault, leaving unbacked tokens in circulation on NEAR.

---

### Finding Description

**Step 1 — Permissionless vault creation via `log_metadata`**

`log_metadata` is a public, unpermissioned instruction. It creates the vault PDA with `init_if_needed` for any mint whose `mint_authority` is not the bridge authority. There is no check for the `TransferFee` extension. [1](#0-0) 

**Step 2 — `transfer_checked` on a transfer-fee mint**

`init_transfer` calls `transfer_checked(payload.amount, decimals)`. For a Token-2022 mint with `TransferFee`, the SPL Token-2022 program debits the sender by `payload.amount` but credits the vault only `payload.amount - fee`, withholding `fee` in the vault's `withheld_amount` field. [2](#0-1) 

**Step 3 — Wormhole VAA encodes the full `payload.amount`**

`serialize_for_near` writes `self.amount` verbatim into the Wormhole message. NEAR reads this value and credits the full amount to the recipient. [3](#0-2) 

**Step 4 — Withheld fees are drainable by the fee authority**

Token-2022's `withdraw_withheld_tokens_from_accounts` instruction allows the `withdraw_withheld_authority` (which the attacker controls, since they created the mint) to pull withheld tokens from any token account — including the bridge vault PDA — without the bridge program's involvement.

**Step 5 — No guard exists**

`log_metadata` inspects only metadata extensions (`MetadataPointer`, `TokenMetadata`). Neither `log_metadata` nor `init_transfer` checks for the `TransferFee` extension. The Solana `SECURITY.md` acknowledges transfer hooks as unsupported but says nothing about transfer-fee mints. [4](#0-3) 

The EVM `SECURITY.md` explicitly documents fee-on-transfer tokens as unsupported, but the Solana side has no equivalent acknowledgment or guard. [5](#0-4) 

---

### Impact Explanation

Per transfer of `N` tokens:

| Step | Vault spendable | Vault withheld | Attacker Solana | Attacker NEAR |
|---|---|---|---|---|
| Before | 0 | 0 | N | 0 |
| After `init_transfer(N)` | N−fee | fee | 0 | N (credited) |
| After `withdraw_withheld` | N−fee | 0 | fee | N |
| After bridging back N−fee | 0 | 0 | fee + (N−fee) = N | fee (unbacked) |

The attacker recovers all N tokens on Solana and retains `fee` unbacked tokens on NEAR. These NEAR tokens can be sold on any DEX or transferred to victims. Any holder who later tries to redeem them via `finalize_transfer` will find the vault empty and the call will revert. The bridge's collateralization invariant — vault balance ≥ outstanding NEAR supply for that token — is permanently broken. [6](#0-5) 

---

### Likelihood Explanation

- `log_metadata` is permissionless; any attacker can register a custom Token-2022 mint with transfer fee.
- Creating a Token-2022 mint with `TransferFee` and a custom `withdraw_withheld_authority` is a standard, documented SPL operation.
- No privileged access, leaked keys, or external dependency compromise is required.
- The attack is repeatable and scales linearly with the fee rate and transfer volume.

---

### Recommendation

In `log_metadata` (and/or `init_transfer`), unpack the mint's Token-2022 extensions and reject any mint that has a `TransferFee` extension with a non-zero `transfer_fee_basis_points` or `maximum_fee`. Example check using `spl_token_2022`:

```rust
use spl_token_2022::extension::transfer_fee::TransferFeeConfig;

if let Ok(fee_config) = mint_with_extension.get_extension::<TransferFeeConfig>() {
    let epoch_fee = fee_config.get_epoch_fee(current_epoch);
    require!(
        u16::from(epoch_fee.transfer_fee_basis_points) == 0
            && u64::from(epoch_fee.maximum_fee) == 0,
        ErrorCode::UnsupportedMintExtension
    );
}
```

This mirrors the EVM bridge's documented stance that fee-on-transfer tokens are unsupported, and should be applied consistently on the Solana side.

### Proof of Concept

```
1. Create Token-2022 mint M with transfer_fee_basis_points=1000 (10%),
   withdraw_withheld_authority = attacker keypair A.

2. Mint 1000 tokens of M to attacker's token account.

3. Call log_metadata(mint=M) → vault PDA V created; NEAR registers M.

4. Call init_transfer(amount=1000, fee=0, recipient="attacker.near"):
   - transfer_checked(1000) executes:
     - attacker account: -1000
     - vault V spendable: +900
     - vault V withheld: +100
   - Wormhole VAA posted: amount=1000

5. NEAR processes VAA → credits 1000 tokens to attacker.near.

6. Call Token-2022 withdraw_withheld_tokens_from_accounts(
       withdraw_withheld_authority=A,
       source=[V],
       destination=attacker_account
   ):
   - vault V withheld: -100
   - attacker account: +100

7. Bridge back 900 tokens from NEAR to Solana via finalize_transfer:
   - vault V: -900 → 0
   - attacker Solana: +900

Assert:
  - attacker Solana balance = 100 (harvested) + 900 (redeemed) = 1000 ✓ (net zero loss)
  - attacker NEAR balance = 100 (unbacked, sellable) ✓
  - vault V balance = 0 ✓
  - NEAR outstanding supply for M = 100 (unbacked) → collateral shortfall confirmed
```
