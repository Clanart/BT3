### Title
Missing `shard_id` Binding in `set_state_header` Allows Cross-Shard State Corruption During State Sync — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for a requested `shard_id=A` that embeds a `ShardChunk` whose internal `shard_id` field is B. Every validation gate in the function passes because none of them bind the chunk's `shard_id` field to the requested shard. The header is then persisted under key `StateHeaderKey(shard_id=A, sync_hash)`, state parts for shard B's trie are installed into shard A's storage, and `set_state_finalize` applies shard B's transactions and receipts as shard A's, producing a permanently wrong post-sync state root for shard A.

### Finding Description

**Entrypoint**: `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`.

The function performs five validation steps, none of which check `chunk.shard_id() == shard_id`:

**Step 1-2 — `validate_chunk_proofs`**: Verifies the chunk's internal hash, transaction Merkle root, and outgoing-receipts root. It never inspects the `shard_id` field. [1](#0-0) 

**Step 3a — `verify_path` against `chunk_headers_root`**: Verifies that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is *somewhere* in the Merkle tree. The plain `verify_path` function only calls `compute_root_from_path` — it does **not** check which leaf index (shard position) the proof encodes. [2](#0-1) [3](#0-2) 

The position-aware variant `verify_path_with_index` exists and is used elsewhere (e.g., chunk-part validation), but is absent here. [4](#0-3) 

**Step 4 — Receipt proof validation**: Uses the *requested* `shard_id` (A) for hashing, not the chunk's `shard_id`. An attacker can supply valid on-chain receipt proofs for shard A independently of which chunk body is embedded. [5](#0-4) 

**Step 5 — `state_root_node` validation**: Validates the node against `chunk_inner.prev_state_root()` — shard B's state root. An attacker who controls a full node can supply the correct node for shard B. [6](#0-5) 

**Storage**: The header is stored under `StateHeaderKey(shard_id=A, sync_hash)` with shard B's chunk inside. [7](#0-6) 

**Downstream corruption in `set_state_finalize`**: `shard_uid` is derived from the requested `shard_id=A`, but `chunk_header.prev_state_root()` comes from shard B's chunk. `apply_state_part` (called before finalization) installs shard B's trie into shard A's storage at shard B's state root. `apply_chunk` then executes shard B's transactions against shard A's trie at shard B's state root, producing a wrong state root for shard A. [8](#0-7) 

The `chunk_headers_root` Merkle tree is built by position (shard index), so shard B's chunk hash at position B produces a valid proof against the root — but the code never checks that position B equals shard A's index. [9](#0-8) 

### Impact Explanation

Shard A's post-sync state root is computed from shard B's transactions and receipts. Any validator that subsequently tracks shard A will compute a different state root than the rest of the network, causing consensus failures or silent divergence in account balances, contract storage, and receipt routing for shard A.

### Likelihood Explanation

Any full node on the network can respond to state sync requests — no validator, block-producer, or privileged role is required. The attacker needs only to be selected as a state sync peer (achievable by running a reachable node), and all required inputs (shard B's chunk body, its Merkle proof, shard A's receipt proofs, shard B's state root node) are derivable from public chain data.

### Recommendation

Add an explicit shard identity check immediately after extracting the chunk in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, replace the bare `verify_path` call with `verify_path_with_index`, passing the shard index for `shard_id` derived from the epoch's shard layout, so the Merkle proof is bound to the correct leaf position. [2](#0-1) 

### Proof of Concept

In a two-shard integration test modelled on the existing `slow_test_catchup_gas_price_change` test: [10](#0-9) 

1. Obtain the legitimate `ShardStateSyncResponseHeader` for shard 0 from client 0.
2. Obtain the legitimate `ShardStateSyncResponseHeader` for shard 1 from client 0.
3. Swap the `chunk` field (and its `chunk_proof`) from the shard-1 header into the shard-0 header, keeping the shard-0 receipt proofs and `state_root_node` from shard 1.
4. Call `set_state_header(shard_id=0, sync_hash, crafted_header)` on client 1.
5. Assert the call returns `Err(...)` — without the fix it returns `Ok(())`.
6. Optionally continue through `set_state_part` / `set_state_finalize` and assert the resulting state root for shard 0 differs from the canonical value.

    chunk: &ShardChunk,

) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
