### Title
Missing Revert Status Verification in `check_is_reverted()` Allows Prover to Arbitrarily Skip `__execute__` While Charging Fees - (File: `execution/execution_constraints.cairo`)

---

### Summary

`check_is_reverted()` in `execution_constraints.cairo` is a completely empty function. It is called in `execute_invoke_function_transaction()` to validate a hint-supplied `is_reverted` value that controls whether `__execute__` runs. Because the function imposes no constraint, a malicious prover can freely set `is_reverted = 1` for any valid invoke transaction, causing `__execute__` to be silently skipped while fees are still charged — a direct loss of funds for every affected user.

---

### Finding Description

In `execute_invoke_function_transaction()`, the OS loads the revert status of the outer transaction from a prover-supplied hint and then calls `check_is_reverted` to validate it:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...);
} else {
    // skip __execute__
}
``` [1](#0-0) 

The function `check_is_reverted` is defined as:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

The function body is entirely empty — it accepts the hint-provided `is_reverted` value and immediately returns without asserting, constraining, or verifying anything. This is the direct analog of the reported `unstake()` bug: just as `unstake()` called `_unstake()` without checking the vesting period, `check_is_reverted()` is called without checking anything, making it indistinguishable from a no-op.

The `is_reverted` value originates exclusively from the hint `%{ IsReverted %}`, which is prover-controlled. In the Cairo proving model, hints are unconstrained unless the Cairo code explicitly constrains them. Since `check_is_reverted` imposes no constraint, the prover can set `is_reverted` to any value — `0` or `1` — for any invoke transaction, with no proof-level consequence.

Compare this with `check_proof_facts` in the same file, which performs real validation (block hash verification, version checks, config hash checks): [3](#0-2) 

The asymmetry confirms `check_is_reverted` is a stub that was never implemented.

Note also that inner entry-point revert status (inside `execute_entry_point`) is correctly derived from `entry_point_return_values.failure_flag` — a value constrained by the actual VM execution. The missing check is specifically at the outer transaction level, where the decision of whether to run `__execute__` at all is made. [4](#0-3) 

---

### Impact Explanation

**Direct loss of funds.**

A malicious prover can set `is_reverted = 1` for any invoke transaction. The `__execute__` entry point is skipped entirely. The user's intended state changes (token transfers, DeFi interactions, etc.) never occur. However, `charge_fee` is called unconditionally after the `if/else` block: [5](#0-4) 

The user pays the full fee for a transaction that was never executed. This constitutes a direct, provable loss of funds for every victim transaction. At scale, a malicious sequencer can drain fees from arbitrarily many users while producing a valid proof, since the OS imposes no constraint on the `is_reverted` value.

The inverse is also true: the prover can set `is_reverted = 0` for a transaction whose `__validate__` would have caused a revert, forcing execution of `__execute__` in a context that should have been rejected, potentially causing incorrect state transitions.

---

### Likelihood Explanation

The StarkNet OS is the trust anchor for the entire protocol — its Cairo constraints are what prevent a sequencer from cheating. The `check_is_reverted` stub removes the constraint that enforces correct revert semantics at the transaction level. Any sequencer operating the OS can exploit this without any special access, leaked keys, or external dependencies. The entry path is every invoke transaction submitted by any user. The exploit requires only that the sequencer provide a dishonest hint value, which is trivially achievable.

---

### Recommendation

Implement `check_is_reverted` to actually constrain the `is_reverted` value. The constraint must verify that `is_reverted` matches the actual execution outcome. One approach is to derive `is_reverted` from the result of `__validate__` (which is already constrained) and from the actual entry-point execution result, rather than accepting it as a free hint. At minimum, the function must assert that `is_reverted` is boolean (`0` or `1`) and that it is consistent with the observed execution trace:

```cairo
func check_is_reverted(is_reverted: felt) {
    // Assert is_reverted is boolean.
    assert is_reverted * (is_reverted - 1) = 0;
    // Additional constraint: bind is_reverted to the actual
    // execution outcome rather than a free hint.
    ...
}
```

The deeper fix is to eliminate the free hint entirely and derive `is_reverted` from the constrained return value of the validate step or from the entry-point execution result.

---

### Proof of Concept

1. User submits an invoke transaction (e.g., `transfer(recipient, amount)`).
2. The OS processes the transaction: `__validate__` runs and returns `VALIDATED` (constrained).
3. The prover sets hint `%{ IsReverted %}` → `is_reverted = 1`.
4. `check_is_reverted(1)` is called — returns immediately, no assertion fires.
5. The `if (is_reverted == FALSE)` branch is skipped; `__execute__` never runs.
6. `charge_fee` executes normally, deducting the full fee from the user's account.
7. The OS produces a valid proof. The block is accepted. The user has paid fees for a transaction that was never executed.

The prover can repeat this for every invoke transaction in a block, collecting fees while performing no actual computation on behalf of users. The resulting proof is indistinguishable from an honest execution because `check_is_reverted` imposes no constraint that would falsify it. [2](#0-1) [6](#0-5)
