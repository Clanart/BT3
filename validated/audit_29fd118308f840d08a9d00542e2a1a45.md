### Title
Cross-Shard State Root Substitution via Missing Shard-ID Binding in `set_state_header` — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` whose embedded `ShardChunk` belongs to shard Y but is submitted for shard X, because the merkle-inclusion check uses the position-agnostic `verify_path` (not `verify_path_with_index`) and there is no assertion that `chunk.shard_id() == shard_id`. The header is then stored under `StateHeaderKey(shard_id=X, sync_hash)` with shard Y's `prev_state_root`. All subsequent `set_state_part` calls validate trie nodes against shard Y's state root (validation is purely hash-based, `shard_id` is only used for metrics), and `apply_state_part` writes those trie nodes into shard X's `ShardUId` storage, permanently corrupting shard X's state.

---

### Finding Description

**Step 1 — Entrypoint is reachable by an unprivileged peer.**

`StateSyncDownloader::ensure_shard_header` downloads a `ShardStateSyncResponseHeader` from a peer-supplied source and forwards it to `set_state_header` for validation. [1](#0-0) 

**Step 2 — `verify_path` is NOT position-aware.**

The merkle-inclusion check at lines 394–403 uses `verify_path`, which only checks `compute_root_from_path(path, item_hash) == root`. It does not verify which leaf index the chunk occupies. A separate `verify_path_with_index` exists and IS position-aware, but is not used here. [2](#0-1) [3](#0-2) 

An attacker provides the chunk from shard Y (at merkle-tree index Y) together with the correct merkle proof for index Y. `verify_path` returns `true` because the proof is cryptographically valid for that chunk — regardless of the `shard_id` argument passed to `set_state_header`.

**Step 3 — No `chunk.shard_id() == shard_id` assertion anywhere in `set_state_header`.**

After the merkle check, the function extracts `chunk_inner.prev_state_root()` — which is shard Y's state root — and validates the `state_root_node` against it. No step checks that the chunk's embedded `shard_id` field matches the `shard_id` parameter. [4](#0-3) 

The header is then stored under `StateHeaderKey(shard_id=X, sync_hash)` containing shard Y's chunk and state root. [5](#0-4) 

**Step 4 — `validate_state_part` ignores `shard_id` for actual validation.**

The wrapper passes `shard_id` only to a metrics label. The real validation delegates to `Trie::validate_state_part(state_root, part_id, partial_state)`, which is purely hash-based and shard-agnostic. [6](#0-5) 

So `set_state_part` for shard X accepts any trie nodes that are consistent with shard Y's state root. [7](#0-6) 

**Step 5 — `apply_state_part` writes shard Y's trie nodes into shard X's storage.**

`apply_state_part` derives `shard_uid` from the claimed `shard_id` (X), then calls `tries.apply_all(&trie_changes, shard_uid, ...)`. The `trie_changes` contain shard Y's nodes (validated against shard Y's state root). They are written permanently into shard X's column. [8](#0-7) 

**Step 6 — Receipt proof check does not block the attack.**

Step 4 of `set_state_header` verifies incoming receipts for the claimed `shard_id` (X). An attacker who is a full node can supply the real on-chain receipt proofs for shard X — these are public chain data. The receipt check is orthogonal to which chunk's state root is embedded. [9](#0-8) 

---

### Impact Explanation

After the attack completes, shard X's trie contains shard Y's account/contract state, with shard Y's state root recorded as shard X's canonical state root. Any subsequent chunk application for shard X will read the wrong state, producing wrong execution results, wrong outgoing receipts, and wrong post-state roots. The node will diverge from the canonical chain and be unable to re-sync without a full state wipe.

---

### Likelihood Explanation

Any peer that serves state sync responses can mount this attack. State sync peers are not required to be validators or operators — any full node can act as a state sync provider. The attacker only needs access to real on-chain block and chunk data (publicly available), a valid chunk from shard Y, and the correct merkle proof for that chunk's position in the block.

---

### Recommendation

Add an explicit shard-ID binding check immediately after extracting the chunk in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, replace the `verify_path` call with `verify_path_with_index` (which already exists in the codebase) so that the merkle proof is verified to be at the correct leaf index corresponding to `shard_id`'s position in the shard layout. [10](#0-9) 

---

### Proof of Concept

```
1. Run a two-shard testnet (shard 0, shard 1).
2. Obtain the legitimate ShardStateSyncResponseHeader for shard 1 at sync_hash H
   (chunk_1, chunk_proof_1, prev_chunk_header_1, prev_chunk_proof_1,
    state_root_node_1, incoming_receipts_proofs for shard 1).
3. Replace incoming_receipts_proofs with the real proofs for shard 0 at H
   (obtainable from chain data).
4. Call set_state_header(shard_id=0, sync_hash=H, crafted_header).
5. Assert the call returns Ok(()).
6. Read back get_state_header(shard_id=0, H).
7. Assert stored_header.chunk.shard_id() == 1  (chunk belongs to shard 1).
8. Assert stored_header.chunk.prev_state_root() == shard_1_state_root.
9. Proceed to call set_state_part / apply_state_part for shard 0.
10. Assert shard 0's trie root equals shard 1's state root.
```

/// Verify merkle path for given item and corresponding path.
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}

pub fn verify_hash(root: MerkleHash, path: &MerklePath, item_hash: MerkleHash) -> bool {
    compute_root_from_path(path, item_hash) == root
}

pub fn verify_path_with_index<T: BorshSerialize>(
    root: MerkleHash,
    path: &MerklePath,
    item: T,
    part_idx: u64,
    num_merklized_parts: u64,
) -> bool {
    verify_path_matches_index(path, part_idx, num_merklized_parts) && verify_path(root, path, item)
}

            // Storage error should not happen

                StatePartValidationResult::Invalid
