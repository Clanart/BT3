### Title
Missing Validation of Declared Class Hash in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash provided by a contract is actually a declared class in the system. This is an acknowledged gap (marked with a `TODO`). An attacker can deploy a contract, attract user funds into it, and then call `replace_class` with an arbitrary undeclared class hash, permanently bricking the contract and freezing all funds held within it.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the requested class hash directly from the syscall request and updates the contract's state entry without any check that the class hash corresponds to a declared class:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The `TODO` comment at line 898 explicitly acknowledges the missing check. The OS accepts any arbitrary felt value as the new class hash and commits it to state without verifying it exists in `contract_class_changes`.

This is the direct analog to the reported `CreditVault.sol` vulnerability: just as `addCollateral()` accepts any receiver address without checking it is a whitelisted trader (causing tokens to be locked), `execute_replace_class` accepts any class hash without checking it is a declared class (causing the contract to be permanently bricked and its funds frozen). [1](#0-0) 

---

### Impact Explanation

Once a contract's class hash is set to an undeclared value, every future call to that contract will fail at the class-lookup stage because no bytecode exists for that hash. The contract becomes permanently non-functional. Any ERC-20 tokens, ETH, or other assets held in the contract's storage are permanently frozen with no recovery path, since the contract can no longer execute any entry point (including any withdrawal or rescue function).

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The attack requires only an unprivileged contract deployer:

1. Attacker deploys a contract that appears legitimate (e.g., a yield vault or token bridge).
2. Users deposit funds into the contract.
3. Attacker (or a time-locked trigger embedded in the contract) calls the `replace_class` syscall with an arbitrary, undeclared felt value as the class hash.
4. The OS processes the syscall, skips the missing declared-class check, and commits the invalid class hash to state.
5. The contract is permanently bricked; all deposited funds are frozen.

No privileged role, leaked key, or external dependency is required. The entry path is a standard `INVOKE_FUNCTION` transaction from an unprivileged sender. [2](#0-1) 

---

### Recommendation

Before committing the new class hash to `contract_state_changes`, verify that the class hash exists as a key in `contract_class_changes` (i.e., it has been declared in the current or a prior block). The acknowledged `TODO` at line 898 should be resolved by adding this check:

```cairo
// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the pattern already enforced in `execute_declare_transaction`, where `assert_not_zero(compiled_class_hash)` is called before updating `contract_class_changes`. [3](#0-2) 

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` contract (class hash `0xAAA`, legitimately declared).
2. Users call `deposit()` on `MaliciousVault`, transferring tokens into its storage.
3. Attacker sends an `INVOKE_FUNCTION` transaction calling `MaliciousVault::trigger_replace()`, which internally issues the `replace_class` syscall with `class_hash = 0xDEAD` (never declared).
4. The OS executes `execute_replace_class`:
   - `request.class_hash = 0xDEAD` is read.
   - The `TODO` check is absent; no lookup into `contract_class_changes` is performed.
   - `dict_update` commits `class_hash=0xDEAD` to `contract_state_changes` for `MaliciousVault`'s address.
5. Block is finalized. `MaliciousVault`'s on-chain class hash is now `0xDEAD`.
6. Any subsequent call to `MaliciousVault` (including `withdraw()`) fails: the OS reads `class_hash=0xDEAD`, finds no compiled class, and the execution reverts.
7. All user funds in `MaliciousVault`'s storage are permanently frozen. [4](#0-3)
