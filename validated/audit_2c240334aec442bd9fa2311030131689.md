Audit Report

## Title
`SwapAllowlistExtension` gates on router address instead of originating user, allowing any router caller to bypass per-user swap restrictions — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` checks `allowedSwapper[pool][sender]` where `sender` is the pool's immediate `msg.sender`. When swaps route through `MetricOmmSimpleRouter`, `sender` is always the router address. A pool admin who allowlists the router to enable legitimate users to swap through it inadvertently opens the pool to every caller of the router, rendering the per-user restriction unenforceable.

## Finding Description
**Step 1 — Extension check:** `SwapAllowlistExtension.beforeSwap` gates on `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is the address the pool passes in. [1](#0-0) 

**Step 2 — Pool passes `msg.sender` as `sender`:** `MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)`. When the router calls the pool, `msg.sender` is the router address. [2](#0-1) 

**Step 3 — Extension receives router address as `sender`:** `ExtensionCalling._beforeSwap` forwards `sender` (= router) verbatim to every configured extension. [3](#0-2) 

**Step 4 — Router calls pool directly:** `exactInputSingle` (and all other `exact*` entry points) calls `pool.swap(...)` directly, making the router the pool's `msg.sender`. The originating user's address is stored only in transient storage as the payer, never forwarded to the extension. [4](#0-3) 

**Why existing guards fail:** The only allowlist check is `allowedSwapper[pool][sender]`. Once the pool admin calls `setAllowedToSwap(pool, router, true)` — a necessary step for any allowlisted user to use the router — the check passes for every caller of the router because `sender` is always the router address regardless of who initiated the transaction. [5](#0-4) 

## Impact Explanation
The `SwapAllowlistExtension` is the sole mechanism for restricting swap access on a pool (e.g., KYC-gated or institutional pools). Its core invariant — that only allowlisted addresses may swap — is broken for all router-mediated swaps once the router is allowlisted. Any address can execute swaps on a pool configured with `allowAllSwappers=false`, bypassing the intended access control entirely. This constitutes broken core pool functionality and an admin-boundary break reachable by an unprivileged caller.

## Likelihood Explanation
The pool admin must allowlist the router for any allowlisted user to use it through `MetricOmmSimpleRouter`. This is a standard, expected operational step — not a malicious configuration. Once done, the bypass is available to any address with no further preconditions, is repeatable, and requires no special privileges or capital.

## Recommendation
The router should forward the originating user's address via the `extensionData` field on every `pool.swap` call, and `SwapAllowlistExtension.beforeSwap` should decode and verify it when `sender` is a known router. Alternatively, the pool interface should support a separate `originator` argument that the router populates with `msg.sender` before calling `pool.swap`, and the extension should gate on that value instead of `sender`.

## Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` attached; call `setAllowAllSwappers(pool, false)`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` so that legitimate users can swap through the router.
3. From an address `attacker` not present in `allowedSwapper`, call `router.exactInputSingle({pool: pool, ...})`.
4. Pool calls `extension.beforeSwap(sender=router, ...)`. The check `allowedSwapper[pool][router]` is `true` → passes.
5. The swap executes: `attacker`'s ERC20 balance decreases and they receive output tokens, confirming the bypass.
