### Title
Missing `verify_block_signature` Call Despite `verify_blocks = true` ‚Äî Feeder-Gateway Can Persist Cryptographically Invalid `BlockSignature` to Header Storage ‚Äî (`crates/apollo_central_sync/src/lib.rs`, `crates/apollo_central_sync/src/sources/central.rs`)

---

### Summary

The `verify_blocks` configuration flag is supposed to enable cryptographic verification of block signatures received from the feeder gateway. In practice it only triggers `track_sequencer_public_key_changes()`, which stores the sequencer public key but never passes it to `verify_block_signature`. The actual ECDSA check (`starknet_api::block::verify_block_signature`) is never invoked anywhere in the central-sync production path. Any `BlockSignature` ‚Äî including one with all-zero `(r, s)` bytes ‚Äî passes through `client_to_central_block` and is committed to header storage via `append_block_signature` without verification.

---

### Finding Description

**Step 1 ‚Äî `verify_blocks` guard is incomplete.**

`sync_while_ok` checks the flag and calls `track_sequencer_public_key_changes()`: [1](#0-0) 

`track_sequencer_public_key_changes` only fetches the key and errors if it changes; it never calls `verify_block_signature`: [2](#0-1) 

The `sequencer_pub_key` field on `GenericStateSync` is populated but is never subsequently passed to any verification function. [3](#0-2) 

**Step 2 ‚Äî `client_to_central_block` discards all binding context.**

Both `Deprecated` and `V0_13_2` arms of the match extract only the raw `(r, s)` bytes; the `block_hash` and `signature_input` fields that would be needed for verification are silently dropped: [4](#0-3) 

**Step 3 ‚Äî `store_block` writes the unverified signature directly.**

`process_sync_event` routes `SyncEvent::BlockAvailable` straight to `store_block`: [5](#0-4) 

`store_block` calls `append_block_signature` with no intervening check: [6](#0-5) 

**Step 4 ‚Äî `verify_block_signature` is dead code in production.**

A global grep confirms `verify_block_signature` appears only in its definition and in a unit test; it is never called from any production sync, storage, or proof path: [7](#0-6) [8](#0-7) 

The commitment invariant documented in `block.rs` ‚Äî that a stored `BlockSignature` must be a valid ECDSA signature over `Poseidon(block_hash, state_diff_commitment)` under the sequencer public key ‚Äî is never enforced.

---

### Impact Explanation

The `BlockSignature` stored in header storage is the authoritative sequencer attestation that downstream proof-manager and SNOS consumers read. Persisting a cryptographically invalid signature (e.g., all-zero `(r, s)`) means:

- Any component that reads the stored signature and treats it as a valid sequencer attestation over `(block_hash, state_diff_commitment)` is operating on a forged or corrupted value.
- Proof production or verification logic that relies on the stored signature to bind a block hash to a state-diff commitment will accept an invalid attestation, breaking the commitment-binding invariant for that block.
- The `verify_blocks = true` default in `SyncConfig` creates a false sense of security: operators believe signatures are being checked when they are not. [9](#0-8) 

---

### Likelihood Explanation

The feeder-gateway URL is operator-configured, but a compromised or MITM'd feeder gateway ‚Äî exactly the threat model `verify_blocks` is designed to address ‚Äî can trivially supply all-zero or otherwise invalid `(r, s)` bytes. The deployment config for `state_sync` sets `verify_blocks: false`, so production nodes are currently unprotected even by the incomplete guard: [10](#0-9) 

The schema default is `true`, meaning any node that relies on the schema default rather than the deployment override believes it is protected. [11](#0-10) 

---

### Recommendation

1. In `store_block`, after `verify_parent_block_hash`, add a call to `verify_block_signature` using `self.sequencer_pub_key` (when `verify_blocks` is true and the key is available). Return a `StateSyncError` variant if verification fails or returns `false`.
2. In `client_to_central_block`, preserve the `block_hash` and `state_diff_commitment` from the `V0_13_2` / `Deprecated` arms and thread them through `SyncEvent::BlockAvailable` so `store_block` has the inputs needed for verification without re-deriving them.
3. Add a `StateSyncError::BlockSignatureVerificationFailed` variant so the error is surfaced clearly and is non-recoverable (preventing silent retry with the same invalid block).

---

let mut mock = MockStarknetReader::new();

// Current behaviour: Ok(()) ‚Äî signature stored without verification
// Expected behaviour: Err(StateSyncError::BlockSignatureVerificationFailed { .. })

    .get_block_signature(BlockNumber(0)).unwrap().unwrap();

The test demonstrates that `append_block_signature` is reached with `(r=0, s=0)` and the storage write succeeds, confirming the missing verification call.

    async fn track_sequencer_public_key_changes(&mut self) -> StateSyncResult {
        let sequencer_pub_key = self.central_source.get_sequencer_pub_key().await?;
        match self.sequencer_pub_key {
            // First time setting the sequencer public key.
            None => {
                info!("Sequencer public key set to {sequencer_pub_key:?}.");
                self.sequencer_pub_key = Some(sequencer_pub_key);
            }
            Some(cur_key) => {
                if cur_key != sequencer_pub_key {
                    warn!(
                        "Sequencer public key changed from {cur_key:?} to {sequencer_pub_key:?}."
                    );
                    // TODO(Yair): Add alert.
                    self.sequencer_pub_key = Some(sequencer_pub_key);
                    return Err(StateSyncError::SequencerPubKeyChanged {
                        old: cur_key,
                        new: sequencer_pub_key,
                    });
                }
            }
        };
        Ok(())
    }

pub fn verify_block_signature(
    sequencer_pub_key: &SequencerPublicKey,
    signature: &BlockSignature,
    state_diff_commitment: &GlobalRoot,
    block_hash: &BlockHash,
) -> Result<bool, BlockVerificationError> {
    let message_hash = Poseidon::hash_array(&[block_hash.0, state_diff_commitment.0]);
    verify_message_hash_signature(&message_hash, &signature.0, &sequencer_pub_key.0).map_err(
        |err| BlockVerificationError::BlockSignatureVerificationFailed {
            block_hash: *block_hash,
            error: err,
        },
    )
}
