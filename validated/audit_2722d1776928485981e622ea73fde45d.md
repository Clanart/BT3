### Title
`native_fee` and Token Fee Permanently Lost When `ft_transfer_call` Fails in `fin_transfer_send_tokens_callback` - (File: near/omni-bridge/src/lib.rs)

### Summary

When a `fin_transfer` to NEAR includes a non-empty `msg` field, the bridge uses `ft_transfer_call` to deliver tokens to the recipient contract. If the recipient's `ft_on_transfer` returns `0` (or panics), `fin_transfer_send_tokens_callback` enters the refund branch. In that branch, the principal (`amount_without_fee`) is correctly burned/re-locked, but `transfer_message.fee.fee` and `transfer_message.fee.native_fee` are silently dropped — never sent to the fee recipient, never burned, and never returned to the original sender. Because the destination nonce is already consumed, the source-chain tokens backing those fees are permanently locked with no recovery path.

### Finding Description

`fin_transfer_send_tokens_callback` handles two outcomes of the downstream `ft_transfer_call`:

**Success path** (lines 1719–1746): distributes `fee.fee` to the fee recipient and mints `fee.native_fee` wrapped tokens to the fee recipient.

**Failure path** (lines 1702–1718): triggered when `is_refund_required` returns `true` (i.e., `ft_on_transfer` returned `U128(0)`):

```rust
if Self::is_refund_required(is_ft_transfer_call) {
    self.burn_tokens_if_needed(token.clone(), U128(amount_without_fee));
    self.revert_lock_actions(&lock_actions);
    self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);
    env::log_str(&OmniBridgeEvent::FailedFinTransferEvent { ... });
    // ← fee.fee and fee.native_fee are never touched
}
``` [1](#0-0) 

`lock_actions` was built from `unlock_tokens_if_needed(..., transfer_message.amount.0)` — the **full** amount including the fee: [2](#0-1) 

`revert_lock_actions` therefore re-locks the full `amount`, but only `amount_without_fee` is burned. The `fee.fee` delta is re-locked in the accounting ledger yet never minted (for deployed tokens) or permanently stranded in the bridge contract (for native tokens). `fee.native_fee` is similarly never minted to anyone.

Because `completedTransfers[destinationNonce]` is set to `true` before the callback chain begins, the nonce is consumed and the transfer cannot be replayed. There is no `recoverTransfer` or equivalent on the EVM side: [3](#0-2) 

The `FailedFinTransferEvent` is only logged; no cross-chain message is sent back to unlock the source-chain tokens.

### Impact Explanation

**For deployed (bridged) tokens**: `fee.fee` tokens are never minted on NEAR. The source-chain `amount` (including fee) is burned/locked permanently. The `locked_tokens` counter is restored to `amount`, but only `amount_without_fee` ever existed on NEAR. The `fee.fee` portion of source-chain tokens is irrecoverably burned with no NEAR-side representation.

**For native NEAR tokens**: `fee.fee` tokens remain in the bridge contract but are re-locked in `locked_tokens`. No account can ever claim them — `claim_fee` operates on `pending_transfers` (not `finalised_transfers`), and the fin-transfer record has been removed.

**For `native_fee`**: The wrapped native-token mint is skipped entirely in the failure branch. The source-chain native tokens backing `native_fee` are permanently locked.

This matches the allowed impact: **permanent freezing / irrecoverable lock of user funds in bridge flows**, and **fee/accounting corruption that misdirects value**. [4](#0-3) 

### Likelihood Explanation

The trigger condition is a `fin_transfer` with a non-empty `msg` field whose recipient contract's `ft_on_transfer` returns `U128(0)`. This is a standard NEP-141 outcome (the receiver signals it consumed zero tokens). It can occur:

- Accidentally: recipient contract has a bug, runs out of gas, or is in an unexpected state.
- Deliberately: the original sender on the source chain specifies a recipient contract they control that always returns `0`, causing their own fee to be permanently locked (griefing the relayer and themselves).

The `is_refund_required` logic explicitly documents this case: [5](#0-4) 

Any user who initiates a cross-chain transfer with a `msg` field is exposed to this risk.

### Recommendation

In the failure branch of `fin_transfer_send_tokens_callback`, handle the fee components explicitly:

1. **`fee.fee`**: Either burn it (for deployed tokens) or transfer it back to the original `transfer_message.sender` (if recoverable), or send it to the fee recipient as compensation for the relayer's work.
2. **`fee.native_fee`**: Similarly, either burn the wrapped representation or refund it.
3. Alternatively, revert the entire `fin_transfer` when `ft_on_transfer` returns `0`, so the nonce is not consumed and the source-chain tokens remain claimable. This mirrors the fix described in the referenced PR #17446.

### Proof of Concept

1. User on EVM calls `initTransfer(tokenAddress, amount=1000, fee=100, nativeFee=0, recipient=nearRecipientContract, message="trigger_refund")`. Bridge burns 1000 tokens on EVM.
2. Relayer calls `fin_transfer` on NEAR with the proof.
3. NEAR bridge calls `ft_transfer_call(nearRecipientContract, 900, "trigger_refund")`.
4. `nearRecipientContract.ft_on_transfer` returns `U128(0)`.
5. `fin_transfer_send_tokens_callback` enters the refund branch:
   - Burns 900 tokens (principal). ✓
   - Re-locks 1000 in `locked_tokens`. ✓
   - Does **not** burn or distribute the 100-token fee. ✗
6. Result: 100 tokens are permanently locked on EVM with no NEAR-side representation. The relayer receives no fee. The `locked_tokens` counter overstates the actual circulating supply by 100 tokens. [1](#0-0) [6](#0-5)
