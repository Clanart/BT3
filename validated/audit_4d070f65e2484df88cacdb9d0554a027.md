### Title
Unauthorized Execution on Behalf of Any Account via `execute_meta_tx_v0` Missing Caller Authorization Check - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary

The `execute_meta_tx_v0` syscall implementation in the StarkNet OS allows any contract to call `__execute__` on an arbitrary target account contract without verifying that the caller is authorized to act on behalf of that account. Because `__validate__` (the standard signature-verification entry point) is entirely bypassed, and standard account contracts do not re-verify signatures inside `__execute__`, an unprivileged attacker can drain funds from any account that uses a version-0 account implementation.

### Finding Description

`execute_meta_tx_v0` is a syscall callable from any executing contract. It constructs a synthetic `TxInfo` with `version=0`, `nonce=0`, and an attacker-supplied `signature`, then directly invokes `__execute__` on an arbitrary `contract_address` supplied in the request: [1](#0-0) 

The function reads `contract_address` and `calldata` entirely from the attacker-controlled `MetaTxV0Request`: [2](#0-1) 

It then builds a new `ExecutionContext` with `caller_address=ORIGIN_ADDRESS` (0) — the same value a legitimate top-level transaction would have — and calls `contract_call_helper`, which invokes `__execute__` on the target: [3](#0-2) 

**There is no check anywhere in `execute_meta_tx_v0` that the calling contract is the target account, or that the target account has authorized this call.** The only guard is that `selector` must equal `EXECUTE_ENTRY_POINT_SELECTOR`.

Contrast this with the normal invoke flow, where `run_validate` is always called before `__execute__`: [4](#0-3) 

`meta_tx_v0` skips `run_validate` entirely. Standard account contracts perform signature verification only in `__validate__`, not in `__execute__`. Therefore, the attacker-supplied signature in `request.signature_start/end` is never cryptographically checked by the OS or by the target account.

Additionally, `check_and_increment_nonce` is never called for `meta_tx_v0`: [5](#0-4) 

The hardcoded `nonce=0` in the synthetic `TxInfo` means the OS-level nonce guard is absent. If the target account's `__execute__` reads the nonce from calldata (as the original Cairo 0 account does), the attacker can supply the victim's current nonce — which is publicly readable — to pass any in-contract nonce check.

### Impact Explanation

**Critical — Direct loss of funds.**

An attacker deploys a malicious contract and invokes it. Inside that invocation, the malicious contract issues a `meta_tx_v0` syscall targeting any victim account that uses a version-0 account implementation. The victim's `__execute__` entry point runs with attacker-controlled calldata (e.g., a `transfer` call sending all tokens to the attacker), with `caller_address=0` satisfying the standard `assert_caller_is_zero()` guard. No valid signature is required. All funds held or approved by the victim account can be drained in a single transaction.

### Likelihood Explanation

**High.** The attack requires only:
1. Deploying a malicious contract (permissionless).
2. Invoking it with a `meta_tx_v0` syscall targeting the victim.

No privileged role, leaked key, or operator cooperation is needed. The victim's account address is public. The victim's current nonce is publicly readable from state. The attack is fully self-contained in one transaction.

### Recommendation

Add an authorization check inside `execute_meta_tx_v0` requiring that the calling contract address equals the target `contract_address`. This mirrors the fix applied in the BakerFi report: the `from` address must equal `msg.sender`. Concretely:

```cairo
// In execute_meta_tx_v0, after reading contract_address:
with_attr error_message("meta_tx_v0: caller must be the target account") {
    assert caller_execution_info.contract_address = contract_address;
}
```

This ensures only the account contract itself can issue a `meta_tx_v0` on its own behalf, preserving the invariant that `__execute__` is only reached after the account has authorized the action (either via `__validate__` in the normal flow, or via its own code in the meta-tx flow).

### Proof of Concept

1. Victim account `V` (version-0 implementation) holds 1000 STRK. Its current nonce is `N` (readable from state).
2. Attacker deploys contract `A` with the following logic in its `__execute__`:
   - Issue syscall `meta_tx_v0` with:
     - `contract_address = V`
     - `selector = __execute__`
     - `calldata = [1, {to=attacker, selector=transfer, calldata_offset=0, calldata_len=2}, 2, attacker_address, 1000_STRK, N]`
     - `signature = []` (empty)
3. Attacker submits an invoke transaction calling `A.__execute__`.
4. The OS processes `A`'s syscall via `execute_meta_tx_v0`:
   - Builds `new_tx_info` with `version=0`, `nonce=0`, empty signature.
   - Sets `caller_address = ORIGIN_ADDRESS`.
   - Calls `V.__execute__` with the attacker's calldata.
5. `V.__execute__` sees `caller_address == 0` ✓, executes the transfer call.
6. 1000 STRK is transferred to the attacker. `V` is drained.

The root cause — absence of `assert caller_execution_info.contract_address = contract_address` — is at: [6](#0-5)

func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
