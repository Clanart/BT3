### Title
`From<CommitmentStateDiff> for ThinStateDiff` Always Drops `deprecated_declared_classes`, Producing Wrong `state_diff_commitment`, Wrong `state_diff_length`, and Wrong Block Hash for Any Block Containing a Deprecated Class Declaration — (`crates/blockifier/src/state/cached_state.rs`)

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` conversion unconditionally sets `deprecated_declared_classes: Vec::new()`. Because this `ThinStateDiff` is the sole input to `calculate_block_commitments`, any block that includes a deprecated (Cairo 0) class declaration will have its `state_diff_commitment`, `state_diff_length`, `concatenated_counts`, and final block hash computed over an incomplete state diff. The corrupted `PartialBlockHashComponents` is then persisted to storage and broadcast to peers.

### Finding Description

`CommitmentStateDiff` — the blockifier's post-execution state summary — has no field for deprecated declared classes: [1](#0-0) 

When the blockifier executes a `DeclareV0`/`DeclareV1` transaction it records the class hash in `StateMaps.declared_contracts`. The conversion `CommitmentStateDiff::from(StateMaps)` silently drops `declared_contracts`: [2](#0-1) 

The downstream conversion to `ThinStateDiff` then hard-codes an empty vector with a `TODO` comment acknowledging the gap: [3](#0-2) 

`BlockExecutionArtifacts::new` passes this incomplete `ThinStateDiff` directly to `calculate_block_commitments`: [4](#0-3) 

Inside `calculate_block_commitments`, two values are derived from the incomplete diff:

1. `state_diff.len()` — used for `concat_counts` — undercounts by the number of deprecated declares: [5](#0-4) 

2. `calculate_state_diff_hash` — the Poseidon hash chains `deprecated_declared_classes` but receives an empty slice: [6](#0-5) 

The resulting `BlockHeaderCommitments` (wrong `state_diff_commitment` + wrong `concatenated_counts`) is stored in `PartialBlockHashComponents`: [7](#0-6) 

`calculate_block_hash` chains both corrupted fields into the final block hash: [8](#0-7) 

The wrong `state_diff_length` is also written into the block header. The P2P sync client reads this field as the authoritative termination condition for state-diff chunk collection: [9](#0-8) 

A syncing peer that receives the header with the understated `state_diff_length` will stop collecting chunks before all deprecated-class chunks arrive, storing an incomplete state diff and diverging from the canonical state.

### Impact Explanation

- **Wrong `state_diff_commitment`** — the Poseidon hash over the state diff omits all deprecated declared classes; the value stored in the block header and chained into the block hash is incorrect.
- **Wrong `concatenated_counts`** — `state_diff_length` is encoded too small; this field is also chained into the block hash.
- **Wrong block hash** — both corrupted fields feed `calculate_block_hash`, so every block containing a deprecated declare carries a wrong block hash.
- **Wrong synced state** — peers using `state_diff_length` from the header as a termination signal will accept a truncated state diff, diverging from the sequencer's actual state.

Matches: *Critical — Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.*

### Likelihood Explanation

Deprecated (`DeclareV0`/`DeclareV1`) transactions remain part of the Starknet protocol and the blockifier still processes them. Any user who submits such a transaction that is included in a block triggers the issue without any privileged access. The TODO comment in the conversion confirms the gap is known but unresolved.

### Recommendation

`CommitmentStateDiff` should be extended with a `deprecated_declared_classes` field populated from `StateMaps.declared_contracts` (filtering entries where the value is `true`). The `From<CommitmentStateDiff> for ThinStateDiff` conversion should then propagate this field instead of substituting `Vec::new()`. Alternatively, the `ThinStateDiff` used for commitment calculation should be derived from the full `StateDiff` (which correctly populates `deprecated_declared_classes` via `ThinStateDiff::from_state_diff`) rather than from `CommitmentStateDiff`.

### Proof of Concept

1. Submit a `DeclareV0` or `DeclareV1` transaction declaring a Cairo 0 class. The gateway accepts it; the batcher includes it in a block.
2. The blockifier records the class hash in `StateMaps.declared_contracts`.
3. `CommitmentStateDiff::from(StateMaps)` drops `declared_contracts`; `ThinStateDiff::from(CommitmentStateDiff)` sets `deprecated_declared_classes = []`.
4. `calculate_block_commitments` computes `state_diff.len()` = N (missing the deprecated-declare count) and `calculate_state_diff_hash` hashes a diff that omits the class.
5. `concat_counts(txs, events, N, da_mode)` encodes the wrong length; `state_diff_commitment` is wrong.
6. `calculate_block_hash` chains both corrupted values → wrong block hash stored and broadcast.
7. A syncing peer reads `state_diff_length = N` from the header, collects only N chunks, and stores a state diff that is missing the deprecated class declaration — diverging from the sequencer's actual state.
