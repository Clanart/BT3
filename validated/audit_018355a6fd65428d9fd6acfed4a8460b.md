### Title
Missing Duplicate-Key/Hash Validation in `MerkleBlob::batch_insert` Corrupts DataLayer Merkle Tree Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces for all items beyond the first two. An attacker-controlled batch containing a repeated `KeyId` or `Hash` silently writes a second leaf node into the blob, orphans the original leaf in the tree structure, and produces a root hash that no longer faithfully represents the committed key-value set. Proofs of inclusion derived from the corrupted tree are structurally invalid, enabling forged inclusion/exclusion proofs against a wrong root.

---

### Finding Description

`MerkleBlob::insert` enforces two invariants before writing any node:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::batch_insert` calls `insert` (with those guards) only for the first two items when the tree has ≤ 1 existing leaf:

```rust
// lines 578-585
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { return Ok(()); };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;   // ← guards active
    }
}
```

Every remaining item in the batch is written directly via `insert_entry_to_blob` with **no duplicate check**:

```rust
// lines 587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { … Node::Leaf(LeafNode { hash, key, value, … }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no guard
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` updates `block_status_cache.key_to_index` and `block_status_cache.leaf_hash_to_index` for the new node. When a duplicate key is present, the cache entry is silently overwritten to point to the new index, while the original leaf node remains physically embedded in the blob and still referenced by its parent internal node. The subtree built from the batch (lines 607-654) is then grafted onto the existing tree via `insert_subtree_at_key`, leaving the tree with two leaf nodes carrying the same key — one reachable through the cache, one reachable through the structural parent links — and a root hash computed over both.

The structural analog to the Hats Protocol bug is exact: just as a hat could be created under a non-existent admin (bypassing the admin-existence check) and later overwritten when the admin was created, here a leaf can be inserted via `batch_insert` (bypassing the uniqueness check) and the cache silently overwritten, while the old leaf remains in the committed tree structure.

---

### Impact Explanation

A corrupted `MerkleBlob` produces an incorrect root hash. Any `ProofOfInclusion` generated by `get_proof_of_inclusion` walks the lineage from the cache-registered leaf to the root; the orphaned duplicate leaf is not walked, so the proof is internally self-consistent but anchored to a root that encodes phantom state. A verifier comparing the proof root against the expected DataLayer root will either:

- Accept a proof for a key-value pair that has been superseded (the orphaned leaf's value is still in the tree hash), or
- Reject a valid proof because the root has drifted from the expected committed value.

Both outcomes satisfy the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

---

### Likelihood Explanation

`batch_insert` is exposed directly to Python callers via `py_batch_insert`:

```rust
// lines 1503-1518
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    …
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
```

Any Python DataLayer code that assembles a batch without pre-deduplicating keys — or that an attacker can influence to include a repeated key — triggers the bug. The fuzz target for `batch_insert` does not exist (only `merkle_blob_insert.rs` and `merkle_blob_insert_and_delete.rs` fuzz the single-item `insert` path), so this path has never been fuzz-tested for duplicate inputs.

---

### Recommendation

Add the same duplicate guards at the top of `batch_insert` (or inside the per-item loop) that `insert` already enforces:

```rust
pub fn batch_insert(&mut self, keys_values_hashes: Vec<((KeyId, ValueId), Hash)>) -> Result<(), Error> {
    // Pre-validate the entire batch before touching the blob
    for ((key, _), hash) in &keys_values_hashes {
        if self.block_status_cache.contains_key(*key) {
            return Err(Error::KeyAlreadyPresent());
        }
        if self.block_status_cache.contains_leaf_hash(hash) {
            return Err(Error::HashAlreadyPresent());
        }
    }
    // … existing logic …
}
```

Additionally, add a fuzz target that exercises `batch_insert` with duplicate keys and hashes, and add a regression test analogous to `test_double_insert_fails` for the batch path.

### Proof of Concept

```rust
// Demonstrates that batch_insert silently accepts a duplicate key
// when the batch has ≥ 3 items (so the first two go through insert()
// and the third bypasses the guard).

let mut blob = MerkleBlob::new(vec![]).unwrap();

// Pre-populate with 2 leaves so leaf_count > 1 — the guard branch is skipped entirely.
blob.insert(KeyId(0), ValueId(0), &sha256_num(&0i64), InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(1), ValueId(1), &sha256_num(&1i64), InsertLocation::Auto {}).unwrap();

// Batch with KeyId(0) duplicated — third item bypasses contains_key check.
let batch = vec![
    ((KeyId(2), ValueId(2)), sha256_num(&2i64)),
    ((KeyId(3), ValueId(3)), sha256_num(&3i64)),
    ((KeyId(0), ValueId(99)), sha256_num(&99i64)),  // ← duplicate key, new value
];
blob.batch_insert(batch).unwrap();   // succeeds — no error raised

blob.calculate_lazy_hashes().unwrap();

// The cache now reports KeyId(0) → ValueId(99), but the original leaf
// (KeyId(0) → ValueId(0)) is still embedded in the blob and included
// in the root hash computation, producing a root that encodes both values.
let root = blob.get_root_hash().unwrap();
// root is computed over a tree containing two leaves for KeyId(0) —
// the committed state is corrupted.
