### Title
Integer Wrapping in Relative Timelock Arithmetic Enables Timelock Bypass When `nowrap=false` - (File: crates/chia-consensus/src/check_time_locks.rs)

### Summary

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` contains four relative-timelock comparisons that use `wrapping_add` when the `nowrap` parameter is `false`. An unprivileged spender can craft a CLVM spend with `ASSERT_SECONDS_RELATIVE = u64::MAX` (or a precisely chosen value that causes a u64/u32 wrap), causing the addition `coin_timestamp.wrapping_add(seconds_relative)` to produce a value smaller than the current timestamp. The timelock check then passes immediately, bypassing the intended lock period. The inverse effect applies to `ASSERT_BEFORE_SECONDS_RELATIVE` and `ASSERT_BEFORE_HEIGHT_RELATIVE`, where wrapping causes a valid spend to be incorrectly rejected, producing consensus divergence between nodes using different `nowrap` values.

### Finding Description

`check_time_locks` accepts a `nowrap: bool` parameter. When `nowrap=false`, all four relative-timelock branches use `wrapping_add` instead of `saturating_add`:

```rust
// ASSERT_SECONDS_RELATIVE (nowrap=false path)
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
```

```rust
// ASSERT_HEIGHT_RELATIVE (nowrap=false path)
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

```rust
// ASSERT_BEFORE_HEIGHT_RELATIVE (nowrap=false path)
} else if prev_transaction_block_height
    >= unspent.confirmed_block_index.wrapping_add(before_height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertBeforeHeightRelativeFailed));
}
```

```rust
// ASSERT_BEFORE_SECONDS_RELATIVE (nowrap=false path)
} else if timestamp >= unspent.timestamp.wrapping_add(before_seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertBeforeSecondsRelativeFailed));
}
```

The codebase's own unit tests explicitly document the divergent outcomes:

- `seconds_relative = 0xffff_ffff_ffff_ffff`, `nowrap=false` → `Ok(())` (spend accepted, timelock bypassed)
- `seconds_relative = 0xffff_ffff_ffff_ffff`, `nowrap=true` → `Err(AssertSecondsRelativeFailed)` (correctly rejected)
- `before_seconds_relative = 0xffff_ffff_ffff_ffff`, `nowrap=false` → `Err(AssertBeforeSecondsRelativeFailed)` (incorrectly rejected)
- `before_seconds_relative = 0xffff_ffff_ffff_ffff`, `nowrap=true` → `Ok(())` (correctly accepted)

The Python binding `py_check_time_locks` forwards the caller-supplied `nowrap` value directly to the Rust function without any sanitization or height-gating inside chia_rs itself.

### Impact Explanation

**Timelock bypass (High):** A spender who sets `ASSERT_SECONDS_RELATIVE = u64::MAX` in their puzzle output causes `coin_timestamp.wrapping_add(u64::MAX) = coin_timestamp - 1`. The guard `timestamp < coin_timestamp - 1` is false for any realistic timestamp, so the spend is accepted immediately regardless of how long the coin has existed. This bypasses the intended relative timelock and enables unauthorized spend acceptance — a coin that was supposed to be unspendable for ~584 billion years can be spent in the next block.

**Consensus divergence (High):** Two nodes calling `check_time_locks` with different `nowrap` values for the same spend bundle will reach opposite accept/reject decisions. A spend with `seconds_relative = u64::MAX` is `Ok` under `nowrap=false` and `Err` under `nowrap=true`. This is a deterministic, attacker-triggerable consensus split.

### Likelihood Explanation

The `nowrap` parameter is controlled entirely by the Python caller (chia-blockchain). The `nowrap=false` path is the legacy pre-hard-fork behavior. Any node that calls `check_time_locks` with `nowrap=False` for a block containing a crafted spend is vulnerable. An attacker who controls a puzzle (e.g., a custom singleton or CAT) can embed `ASSERT_SECONDS_RELATIVE = 0xffffffffffffffff` in the conditions returned by that puzzle. The condition value passes `sanitize_uint` (it is a valid 8-byte unsigned integer) and is stored verbatim in `spend.seconds_relative`. The overflow only manifests at the `check_time_locks` call site.

### Recommendation

1. Remove the `nowrap=false` / `wrapping_add` code paths entirely. Replace all four `wrapping_add` calls with `saturating_add` unconditionally, or with `checked_add` that returns an error on overflow.
2. If backward compatibility with pre-hard-fork blocks is required, gate the `nowrap=false` path on a block-height check inside `check_time_locks` itself (using a `ConsensusConstants` parameter), rather than relying on the Python caller to pass the correct boolean.
3. Add a validation step in `parse_args` / `sanitize_uint` that rejects `ASSERT_SECONDS_RELATIVE` values that would overflow when added to any plausible coin timestamp (e.g., reject values `> u64::MAX - MAX_COIN_TIMESTAMP`).

### Proof of Concept

Craft a CLVM puzzle that outputs the condition `(ASSERT_SECONDS_RELATIVE . 0xffffffffffffffff)`. Submit a spend bundle spending a coin created by this puzzle. Call `check_time_locks` with `nowrap=False`:

```
coin_timestamp = 1_700_000_000   # any realistic Unix timestamp
seconds_relative = 0xffff_ffff_ffff_ffff  # u64::MAX

# wrapping_add:
deadline = coin_timestamp.wrapping_add(seconds_relative)
         = 1_700_000_000 + 18_446_744_073_709_551_615  (mod 2^64)
         = 1_699_999_999   # wraps to coin_timestamp - 1

# guard: timestamp < deadline
# 1_700_000_001 < 1_699_999_999  →  False  →  Ok(())  →  spend accepted
```

The spend is accepted at the very next block, bypassing the intended ~584-billion-year timelock. The same spend is rejected by a node using `nowrap=True` (`saturating_add` clamps to `u64::MAX`, so `timestamp < u64::MAX` is true → `Err`), producing a consensus split. #[cfg(feature = "py-bindings")]
#[pyfunction]
#[pyo3(name = "check_time_locks")]
#[allow(clippy::needless_pass_by_value)] // pyo3 prefers pass_by_value
pub fn py_check_time_locks(
    removal_coin_records: HashMap<Bytes32, CoinRecord>,
    bundle_conds: &OwnedSpendBundleConditions,
    prev_transaction_block_height: u32,
    timestamp: u64,
    nowrap: bool,
) -> PyResult<Option<u32>> {
    let res = check_time_locks(
        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,
        nowrap,
    );

    match res {
        Ok(()) => Ok(None),
        Err(ec) => Ok(Some(ec.error_code().into())),
    }
}
