### Title
Hardcoded `failure_flag=0` in `execute_deploy` Silently Ignores Constructor Revert ‚Äî (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

In `execute_deploy`, the `ResponseHeader.failure_flag` is unconditionally hardcoded to `0` (success) regardless of whether the constructor execution actually reverted. A calling contract therefore always receives a "success" signal from the `deploy` syscall, even when the constructor failed and the contract was never deployed. Any subsequent fund transfer to the reported `contract_address` results in a direct, permanent loss of funds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_deploy` function writes the syscall response header with a hardcoded `failure_flag=0`:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` helper is called just before this and returns `(retdata_size, retdata)` ‚Äî it does **not** surface an `is_reverted` flag to the caller:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}
``` [2](#0-1) 

The `execute_entry_point` function (which `deploy_contract` calls internally) **does** detect and handle reverts ‚Äî it calls `handle_revert` to roll back state changes and returns `is_reverted=1`: [3](#0-2) 

However, that revert status is never propagated back through `deploy_contract` to `execute_deploy`. The response header is therefore always written with `failure_flag=0`, and the `DeployResponse` always contains the computed `contract_address`:

```cairo
assert [response] = DeployResponse(
    contract_address=contract_address,
    constructor_retdata_start=retdata,
    constructor_retdata_end=retdata + retdata_size,
);
``` [4](#0-3) 

The embedded `TODO` comment confirms this is a known, unimplemented case ‚Äî not intentional design.

This is structurally identical to the BetExpress `C-01` pattern: a hardcoded `0` is passed where a dynamic value (the actual revert status) must be used, causing the accounting/state-transition layer to produce an incorrect result.

---

### Impact Explanation

**Direct loss of funds / Permanent freezing of funds.**

When a constructor reverts:
1. `handle_revert` rolls back all state changes ‚Äî the contract is **not** deployed.
2. `execute_deploy` still writes `failure_flag=0` and returns the computed `contract_address`.
3. The calling contract reads `failure_flag=0` and concludes the deployment succeeded.
4. The calling contract transfers ERC-20 tokens (or native assets) to `contract_address`.
5. Because no contract exists at that address, the tokens are stored in the ERC-20 contract's storage under an address that has no owner capable of withdrawing them ‚Äî **permanent loss of funds**.

---

### Likelihood Explanation

**High.** The incorrect value is unconditionally hardcoded; it cannot be correct under any execution path where the constructor reverts. Any contract that:
- calls the `deploy` syscall, and
- conditionally transfers funds to the returned address based on the (always-zero) `failure_flag`

is affected. This is a standard and common pattern in factory contracts on StarkNet.

---

### Recommendation

1. Modify `deploy_contract` to return `(is_reverted, retdata_size, retdata)`.
2. In `execute_deploy`, use the returned `is_reverted` value for `failure_flag`:

```cairo
with remaining_gas {
    let (is_reverted, retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}
// ...
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

3. When `is_reverted=1`, the `DeployResponse` should reflect a zero/null `contract_address` so the caller cannot mistakenly use it.

---

### Proof of Concept

1. Attacker declares a contract class whose constructor always reverts (e.g., `assert 1 = 0`).
2. A victim factory contract calls the `deploy` syscall with that class hash.
3. The OS executes the constructor ‚Üí it reverts ‚Üí `handle_revert` rolls back state.
4. `execute_deploy` writes `ResponseHeader(gas=..., failure_flag=0)` and `DeployResponse(contract_address=X, ...)`.
5. The factory contract reads `failure_flag=0`, concludes deployment succeeded, and calls `transfer(X, amount)` on an ERC-20 token.
6. The ERC-20 transfer succeeds (tokens are credited to address `X` in the ERC-20 storage).
7. No contract exists at `X`; no one can call `transfer` from `X` to recover the tokens ‚Üí **permanent loss of funds**.
