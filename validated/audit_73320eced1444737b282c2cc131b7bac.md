### Title
`CommitmentStateDiff` → `ThinStateDiff` Conversion Silently Drops `deprecated_declared_classes`, Producing Wrong `state_diff_commitment` and Block Hash for Blocks Containing Cairo 0 Class Declarations - (File: `crates/blockifier/src/state/cached_state.rs`)

---

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` implementation in the blockifier unconditionally sets `deprecated_declared_classes: Vec::new()`. Because `BlockExecutionArtifacts::new()` feeds this truncated `ThinStateDiff` directly into `calculate_block_commitments()`, any block that contains a `DeclareV1` (Cairo 0) transaction will have its `state_diff_commitment` — and therefore its block hash — computed without the deprecated class entries. The wrong commitment is then stored in the forest metadata, written to the MDBX state-diff table, and returned by the RPC layer, while the correct Starknet protocol value would include those entries.

---

### Finding Description

**Root cause — the silent drop**

`CommitmentStateDiff` (the blockifier's execution-output type) has no `deprecated_declared_classes` field:

```rust
pub struct CommitmentStateDiff {
    pub address_to_class_hash: IndexMap<ContractAddress, ClassHash>,
    pub address_to_nonce: IndexMap<ContractAddress, Nonce>,
    pub storage_updates: IndexMap<ContractAddress, IndexMap<StorageKey, Felt>>,
    pub class_hash_to_compiled_class_hash: IndexMap<ClassHash, CompiledClassHash>,
}
```

The `From` implementation therefore hard-codes an empty vector:

```rust
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            deployed_contracts: commitment_state_diff.address_to_class_hash,
            storage_diffs: commitment_state_diff.storage_updates,
            class_hash_to_compiled_class_hash: commitment_state_diff
                .class_hash_to_compiled_class_hash,
            nonces: commitment_state_diff.address_to_nonce,
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),   // ← always empty
        }
    }
}
``` [1](#0-0) 

**Propagation into block commitment**

`BlockExecutionArtifacts::new()` calls `ThinStateDiff::from(commitment_state_diff.clone())` and passes the result directly to `calculate_block_commitments()`:

```rust
let (header_commitments, measurements) = calculate_block_commitments(
    &transactions_data,
    ThinStateDiff::from(commitment_state_diff.clone()),   // deprecated_declared_classes = []
    l1_da_mode,
    &block_info.starknet_version,
)
.await;
``` [2](#0-1) 

`calculate_block_commitments()` spawns `calculate_state_diff_hash()` on that `ThinStateDiff`:

```rust
let state_diff_task = spawn_measured_task(move || calculate_state_diff_hash(&state_diff));
``` [3](#0-2) 

**The hash function requires the field**

`calculate_state_diff_hash()` explicitly chains `deprecated_declared_classes` into the Poseidon hash:

```rust
hash_chain = chain_deprecated_declared_classes(&state_diff.deprecated_declared_classes, hash_chain);
``` [4](#0-3) 

When `deprecated_declared_classes` is `[]` but the block actually declared one or more Cairo 0 classes, the resulting `StateDiffCommitment` differs from the protocol-correct value. That wrong commitment is then chained into `calculate_block_hash()`:

```rust
.chain(&block_commitments.state_diff_commitment.0.0)
``` [5](#0-4) 

producing a wrong `BlockHash`.

**Self-consistency masks the error**

`BlockExecutionArtifacts::thin_state_diff()` also calls `ThinStateDiff::from(self.commitment_state_diff.clone())`:

```rust
pub fn thin_state_diff(&self) -> ThinStateDiff {
    ThinStateDiff::from(self.commitment_state_diff.clone())
}
``` [6](#0-5) 

So the same truncated diff is written to the MDBX state-diff table via `append_state_diff()`. The committer's optional `verify_state_diff_hash` check compares the provided commitment against `calculate_state_diff_hash(&state_diff)` on the same truncated diff, so it passes — both sides are wrong in the same way. No internal alarm fires. [7](#0-6) 

**Analog to the external report**

The external report describes how `Transceiver::_checkImmutables()` can be bypassed, allowing the NTT Manager address to change silently and breaking the ownership-synchronicity invariant. Here the analog is: the `CommitmentStateDiff` → `ThinStateDiff` conversion acts as a silent bypass of the invariant that the state diff used for commitment must faithfully represent all execution-output state changes. The dropped field (`deprecated_declared_classes`) is the sequencer's equivalent of the silently changed NTT Manager address — it is structurally absent from the intermediate type, so no check can catch it.

---

### Impact Explanation

Every block that includes at least one `DeclareV1` (Cairo 0) transaction will have:

1. A wrong `state_diff_commitment` stored in the forest metadata (`ForestMetadataType::StateDiffHash`).
2. A wrong `BlockHash` stored in MDBX and returned by `starknet_getBlockWithTxHashes` / `starknet_getBlockWithTxs`.
3. A wrong `ThinStateDiff` (missing the deprecated class) stored in the state-diff table and returned by `starknet_getStateUpdate`.
4. A wrong parent hash chained into the next block's hash, propagating the error forward.
5. Potential proof-verification failure: `SnosProofFacts` embeds `block_hash`; if the on-chain block hash diverges from the sequencer's stored value, proof facts validation in `validate_proof_facts()` will reject valid transactions or accept facts for the wrong block. [8](#0-7) 

This matches the **Critical** impact scope: *Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input* (wrong block hash and state diff from execution output), and **High**: *RPC execution … returns an authoritative-looking wrong value* (wrong block hash and state update from RPC).

---

### Likelihood Explanation

`DeclareV1` transactions are still structurally valid in the blockifier and are not explicitly rejected at the gateway layer in the reviewed code. Any external account can craft and submit a `DeclareV1` transaction. The sequencer will execute it, produce a `CommitmentStateDiff` that silently omits the declared class, and commit a wrong block hash. No special privilege or coordination is required; the trigger is a single ordinary transaction.

---

### Recommendation

1. **Add `deprecated_declared_classes` to `CommitmentStateDiff`** and populate it in the blockifier when a `DeclareV1` transaction is executed, so the field is available for the conversion.
2. **Remove the hard-coded `Vec::new()`** in `From<CommitmentStateDiff> for ThinStateDiff` and propagate the actual value.
3. **Enable `verify_state_diff_hash` by default** in the committer config so that any future divergence between the provided commitment and the recalculated one is caught at commit time rather than silently stored.
4. If `DeclareV1` transactions are intentionally unsupported by the current sequencer, **reject them explicitly at the gateway** and document the invariant so the TODO comment becomes a hard enforcement boundary.

---

### Proof of Concept

```
1. Craft a DeclareV1 transaction T declaring Cairo 0 class C with class_hash H.

2. Submit T to the sequencer gateway. The gateway accepts it (no explicit DeclareV1 rejection).

3. The sequencer batches T into block B at height N.

4. BlockExecutionArtifacts::new() runs:
     commitment_state_diff = blockifier.close_block()
     // commitment_state_diff.class_hash_to_compiled_class_hash is empty (Cairo 0 has no compiled hash)
     // commitment_state_diff has NO deprecated_declared_classes field

     thin = ThinStateDiff::from(commitment_state_diff)
     // thin.deprecated_declared_classes = []   ← H is missing

     state_diff_commitment_actual = calculate_state_diff_hash(&thin)
     // Poseidon(..., chain_deprecated_declared_classes([]), ...)

5. Compute the correct value independently:
     thin_correct = thin with deprecated_declared_classes = [H]
     state_diff_commitment_correct = calculate_state_diff_hash(&thin_correct)
     // Poseidon(..., chain_deprecated_declared_classes([H]), ...)

6. Assert state_diff_commitment_actual != state_diff_commitment_correct.

7. The block hash stored in MDBX and returned by starknet_getBlockWithTxHashes
   uses state_diff_commitment_actual, which is wrong.

8. Any downstream consumer (proof verifier, L1 bridge, cross-chain relayer)
   that independently recomputes the state diff commitment from the declared
   classes will observe a mismatch.
```
