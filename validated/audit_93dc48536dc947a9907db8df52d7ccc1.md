### Title
Fee-Escalation Replacement Permanently Deletes Pending Transaction When Mempool Is Full and Account Has a Nonce Gap — (File: `crates/apollo_mempool/src/mempool.rs`)

---

### Summary

In `add_tx_validations`, a fee-escalation replacement first removes the old transaction via `handle_fee_escalation`, then checks capacity. When `handle_capacity_overflow` is reached, it reads `account_has_txs = false` (old tx already removed from `tx_pool`) but `account_has_gap = true` (stale `accounts_with_gap` set, not yet updated). This causes `creating_gap = true`, which unconditionally returns `MempoolFull`. The old transaction is permanently gone; the replacement is never inserted. A user's pending transaction is silently deleted from the mempool.

---

### Finding Description

`add_tx_validations` executes three steps in sequence:

```
validate_incoming_tx  →  handle_fee_escalation(false)  →  exceeds_capacity / handle_capacity_overflow
```

`handle_fee_escalation` with `validation_only = false` removes the existing transaction from both `tx_pool` and `tx_queue` before returning:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 762-765
self.tx_queue.remove_txs(&[existing_tx_reference]);
self.tx_pool
    .remove(existing_tx_reference.tx_hash)
    .expect("Transaction hash from pool must exist.");
```

If the replacement transaction is larger than the removed one, `exceeds_capacity` fires and `handle_capacity_overflow` is called:

```rust
// lines 950-967
fn handle_capacity_overflow(&mut self, tx: &InternalRpcTransaction, account_nonce: Nonce)
    -> Result<(), MempoolError>
{
    let address = tx.contract_address();
    let account_has_gap  = self.accounts_with_gap.contains(&address);   // ← stale
    let account_has_txs  = self.tx_pool.contains_account(address);      // ← false (old tx removed)
    let closing_gap      = tx.nonce() == account_nonce;
    let creating_gap     = (account_has_gap || !account_has_txs) && !closing_gap;

    if !creating_gap && self.try_make_space(tx.total_bytes()) {
        return Ok(());
    }
    Err(MempoolError::MempoolFull)
}
```

`accounts_with_gap` is only updated in `update_accounts_with_gap`, which is called **after** `add_tx_validations` returns (line 481). At the point `handle_capacity_overflow` runs, `accounts_with_gap` still reflects the state from when the old transaction was inserted. If the old transaction had a nonce gap (`tx.nonce > account_nonce`), `account_has_gap = true`.

Combined with `account_has_txs = false` (old tx removed) and `closing_gap = false` (nonce > account_nonce), the expression evaluates:

```
creating_gap = (true || true) && true = true
```

The function returns `MempoolFull` without calling `try_make_space`. `add_tx_validations` propagates the error, `add_tx` returns `Err(MempoolFull)`, and the old transaction — already removed — is never restored. The mempool ends up with one fewer transaction than before the call.

The `validate_tx` path does **not** reproduce this: it calls `handle_fee_escalation(true)` (validation-only, no removal) and never checks capacity, so it returns `Ok(())` for the same input that `add_tx` rejects.

---

### Impact Explanation

A user whose pending transaction has a nonce gap (e.g., nonce 5 in pool, account nonce 3) and who attempts a fee-escalation replacement with a slightly larger transaction (e.g., longer signature) when the mempool is at capacity will:

1. Have their existing pending transaction **permanently deleted** from the mempool.
2. Receive `MempoolFull` for the replacement.
3. End up with **no pending transaction** at all, despite having had one before the call.

This is a valid transaction being rejected before sequencing, matching the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Additionally, `validate_tx` returns `Ok` for the same input, creating a divergence between the gateway's pre-admission check and the actual admission result.

---

### Likelihood Explanation

Conditions required:
- `enable_fee_escalation = true` (production default).
- Mempool at or near `capacity_in_bytes` (1 GB default; reachable under load or deliberate flooding).
- Replacement transaction is larger than the original (e.g., longer signature, additional calldata).
- The original transaction is the only one for that account in the pool.
- The original transaction has a nonce gap (`tx.nonce > account_nonce`).

All conditions are realistic in production. An adversary can fill the mempool with their own transactions to reliably trigger the condition against any victim with a gap transaction.

---

### Recommendation

Do not remove the old transaction before the capacity check passes. Restructure `add_tx_validations` so that `handle_fee_escalation` is called in removal mode only after `exceeds_capacity` / `handle_capacity_overflow` confirms there is room. Alternatively, pass a `is_replacement: bool` flag to `handle_capacity_overflow` and skip the `creating_gap` guard when replacing an existing transaction, since the gap status is unchanged by the replacement.

---

### Proof of Concept

          → returns Err(MempoolFull)

5. add_tx returns Err(MempoolFull).
   tx_A_old is gone. tx_A_new is not inserted.

    fn handle_fee_escalation(
        &mut self,
        incoming_tx_reference: TransactionReference,
        validation_only: bool,
    ) -> MempoolResult<()> {
        let TransactionReference { address, nonce, .. } = incoming_tx_reference;

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {
            if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
                return Err(MempoolError::DuplicateNonce { address, nonce });
            };

            return Ok(());
        }

        let Some(existing_tx_reference) = self.tx_pool.get_by_address_and_nonce(address, nonce)
        else {
            // Replacement irrelevant: no existing transaction with the same nonce for address.
            return Ok(());
        };

        if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
            info!(
                "{existing_tx_reference} was not replaced by {incoming_tx_reference} due to \
                 insufficient fee escalation."
            );
            // TODO(Elin): consider adding a more specific error type / message.
            return Err(MempoolError::DuplicateNonce { address, nonce });
        }

        if validation_only {
            return Ok(());
        }

        debug!("{existing_tx_reference} will be replaced by {incoming_tx_reference}.");

        self.tx_queue.remove_txs(&[existing_tx_reference]);
        self.tx_pool
            .remove(existing_tx_reference.tx_hash)
            .expect("Transaction hash from pool must exist.");

        Ok(())
    }

    fn update_accounts_with_gap(&mut self, address_to_nonce: AddressToNonce) {
        for (address, account_nonce) in address_to_nonce {
            // Assumption: Future declares are not allowed — their nonce must match the account
            // nonce, so they fill a gap if one exists.
            if self.delayed_declares.contains(address, account_nonce) {
                self.accounts_with_gap.swap_remove(&address);
                continue;
            }

            // Gap exists when lowest transaction nonce is higher than account nonce.
            let gap_exists = match self.tx_pool.get_lowest_nonce(address) {
                Some(lowest_nonce) => account_nonce < lowest_nonce,
                None => false, // No transactions for the account, so no gap.
            };

            // Update the eviction tracking set accordingly.
            if gap_exists {
                self.accounts_with_gap.insert(address);
            } else {
                self.accounts_with_gap.swap_remove(&address);
            }
        }
    }
