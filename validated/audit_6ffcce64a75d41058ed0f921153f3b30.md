### Title
Attacker Can Permanently Block Bridged Token Deployment on NEAR by Pre-Creating the Deterministic Token Account - (File: `near/token-deployer/src/lib.rs`, `near/omni-bridge/src/lib.rs`)

---

### Summary

The NEAR Omni Bridge derives a fully deterministic, publicly computable NEAR account ID for every bridged token. The `TokenDeployer` contract unconditionally calls `Promise::new(account_id).create_account()`, which fails silently if the account already exists. An attacker who pre-creates that account causes the deployment to fail, the bridge rolls back all token mappings, and the token can never be deployed — permanently blocking `fin_transfer` for any user who subsequently locks funds on the source chain.

---

### Finding Description

**Step 1 — Deterministic account ID derivation (`near/omni-types/src/lib.rs`, `near/omni-bridge/src/lib.rs`)**

`deploy_token_internal` computes the NEAR account ID for a bridged token as:

```
token_id = format!("{prefix}.{deployer}")
```

where `prefix` is derived from the source-chain token address via `get_token_prefix()`:

- For EVM tokens: the raw hex address string (e.g. `23ddd3e3692d1861ed57ede224608875809e127f`)
- For Solana/Fogo/Starknet: a keccak-truncated hex prefix [1](#0-0) [2](#0-1) 

Both the `deployer` account ID and the token address are public on-chain data. Any observer can compute the exact NEAR account ID that will be used before `deploy_token` is ever called.

**Step 2 — Unconditional `create_account()` in the deployer (`near/token-deployer/src/lib.rs`)**

```rust
pub fn deploy_token(&mut self, account_id: AccountId, metadata: &BasicMetadata) -> Promise {
    Promise::new(account_id)
        .create_account()          // ← fails if account already exists
        .transfer(env::attached_deposit())
        .use_global_contract(self.global_code_hash)
        .function_call("new".to_string(), ...)
}
``` [3](#0-2) 

In NEAR, `create_account()` on an already-existing account causes the entire promise chain to fail. There is no `create_account_if_not_exists` or existence check.

**Step 3 — Full rollback on failure (`near/omni-bridge/src/lib.rs`)**

When the deployer promise fails, `deploy_token_by_deployer_callback` removes the token from every mapping:

```rust
} else {
    self.deployed_tokens.remove(&token_id);
    self.deployed_tokens_v2.remove(&token_id);
    self.token_id_to_address.remove(&(token_address.get_chain(), token_id));
    self.token_address_to_id.remove(token_address);
    self.token_decimals.remove(token_address);
    PromiseOrValue::Value(())
}
``` [4](#0-3) 

The proof is **not consumed** — there is no used-proof registry for `deploy_token`. The caller can retry with the same proof, but the attacker's pre-created account persists, so every retry fails identically.

**Step 4 — `fin_transfer` panics for the undeployed token**

`fin_transfer_callback` requires `token_decimals` to contain the token address:

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
``` [5](#0-4) 

Because the rollback removed the entry from `token_decimals`, every `fin_transfer` call for that token panics with `ERR_TOKEN_DECIMALS_NOT_FOUND`. Funds locked on the source chain via `init_transfer` can never be released.

---

### Impact Explanation

Any user who calls `init_transfer` on the EVM (or other source-chain) bridge after the attacker has pre-created the NEAR account will have their tokens permanently locked in the source-chain vault. The `fin_transfer` relayer cannot complete the settlement because the NEAR-side token mapping is absent and cannot be restored through the normal `deploy_token` path. The `add_deployed_tokens` DAO bypass also cannot help because the token contract itself was never deployed to the pre-created account — calling `storage_deposit` on it would fail.

This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows**.

---

### Likelihood Explanation

- The target NEAR account ID is fully deterministic and computable offline from the source-chain token address and the public deployer account ID.
- Creating a NEAR account costs a trivial amount of NEAR (< 0.1 NEAR).
- No mempool monitoring is required — the attacker can pre-create accounts for any token address at any time, including before the `log_metadata` event is even emitted on the source chain.
- The attack is repeatable across all tokens for all chains (ETH, ARB, BASE, BNB, POL, SOL, STRK, etc.) at negligible cost.

---

### Recommendation

1. **Check account existence before deploying.** In `deploy_token` (`near/token-deployer/src/lib.rs`), use `env::account_exists(account_id)` before issuing `create_account()` and panic early with a clear error rather than silently failing in the callback.

2. **Alternatively, include a nonce or salt in the account ID derivation** so the deployer can retry with a fresh sub-account if the first one is squatted.

3. **Do not roll back `token_decimals` / `token_address_to_id` on deployer failure** if the proof has already been verified — or mark the proof as consumed so a retry with a fresh account ID is possible.

    pub fn get_token_prefix(&self) -> String {
        match self {
            Self::Sol(address) => Self::hashed_token_prefix("sol", &H256(address.0)),
            Self::Fogo(address) => Self::hashed_token_prefix("fogo", &H256(address.0)),
            Self::Strk(address) => Self::hashed_token_prefix("strk", address),
            Self::Eth(address) => {
                if self.is_zero() {
                    "eth".to_string()
                } else {
                    address.to_string()[2..].to_string()
                }
            }
            _ => self.encode('-', true),
        }
    }
