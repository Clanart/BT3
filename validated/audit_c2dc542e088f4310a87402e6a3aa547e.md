### Title
Unguarded Shared ETH Balance in `pay()` Lets Any Caller Drain Stranded Native ETH from the Router ŌĆö (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary
`PeripheryPayments.pay()` unconditionally consumes the router's entire native ETH balance to settle WETH swap payments before pulling from the designated payer. Because the router's ETH balance is a shared, unguarded pool across all callers and transactions, any user who initiates a WETH-input swap while the router holds ETH left over from a prior user's incomplete `multicall` receives the output tokens for free, permanently stealing the stranded ETH.

### Finding Description
`pay()` is the internal settlement function called from `_justPayCallback` and `_exactOutputIterateCallback` during every swap. When `token == WETH`, it branches on the router's current native balance:

```solidity
// PeripheryPayments.sol lines 73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // payer never charged
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

When `nativeBalance >= value`, the router wraps its own ETH and sends it to the pool. The `payer` ŌĆö set in transient storage to `msg.sender` at swap entry ŌĆö is **never charged**. There is no check that the ETH in the router was contributed by the current caller.

ETH accumulates in the router whenever any `payable` entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`) is called with `msg.value` exceeding the amount consumed by the swap and the caller omits a `refundETH()` call. The `receive()` guard:

```solidity
// PeripheryPayments.sol line 32-34
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
```

only blocks direct ETH pushes; it does **not** prevent ETH from being left over from a `payable` function call, because `msg.value` is delivered to the contract before `receive()` is ever consulted.

The `_justPayCallback` path that reaches `pay()`:

```solidity
// MetricOmmSimpleRouter.sol lines 192-198
function _justPayCallback(int256 amount0Delta, int256 amount1Delta) private {
    pay(
        _getTokenToPay(),
        _getPayer(),
        msg.sender,
        uint256(MetricOmmSwapResults.extractPositiveAmount(amount0Delta, amount1Delta))
    );
}
```

`_getPayer()` returns the address stored in transient storage at swap entry ŌĆö the original `msg.sender` ŌĆö but `pay()` ignores that address entirely when the router's ETH balance is sufficient.

### Impact Explanation
Any attacker who observes a non-zero ETH balance in the router can call `exactInputSingle` (or any WETH-input swap function) with `amountIn Ōēż router.balance` and receive the output tokens without paying anything. The router wraps its own ETH to settle the pool's callback; the attacker's WETH allowance is never touched. The prior user permanently loses their stranded ETH.

**Concrete loss:** User A calls `multicall{value: 1 ETH}([exactInputSingle(WETHŌåÆtoken1, amountIn=0.5 ETH)])` without appending `refundETH()`. 0.5 ETH remains in the router. An attacker immediately calls `exactInputSingle(WETHŌåÆtoken1, amountIn=0.5 ETH)` with zero ETH sent and zero WETH approval. During the callback, `pay(WETH, attacker, pool, 0.5 ETH)` finds `address(this).balance == 0.5 ETH`, wraps it, and sends WETH to the pool. The attacker receives token1 worth 0.5 ETH while paying nothing. User A's 0.5 ETH is permanently stolen.

### Likelihood Explanation
The `multicall` + `refundETH` pattern is the documented way to use native ETH with the router, but omitting `refundETH` is a common integration mistake ŌĆö especially for integrators who send a round `msg.value` and let the swap consume only part of it. Any MEV bot monitoring the router's ETH balance on-chain can atomically exploit the window. The attack requires no special permissions, no token approvals, no prior setup, and no privileged role ŌĆö only the ability to call a public swap function.

### Recommendation
Track per-transaction ETH contributions in transient storage rather than reading the global `address(this).balance`. At each `payable` entry point, record `msg.value` in a transient slot. In `pay()`, consume only from that recorded amount and revert if it is insufficient, forcing the remainder to be pulled from the payer via `transferFrom`. Alternatively, remove the native-ETH shortcut from `pay()` entirely and require callers to wrap ETH themselves before calling the router.
