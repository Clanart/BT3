### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Fund Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary felt as the new class hash without verifying it corresponds to a declared class. A malicious contract can replace its own class with an undeclared hash, permanently freezing all funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads `request.class_hash` and immediately writes it into `contract_state_changes` with no check that the hash exists in the class tree:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment explicitly acknowledges the missing validation. Once this state update is committed to the Patricia tree via `compute_contract_state_commitment`, the contract's `class_hash` field permanently holds the undeclared value. [1](#0-0) 

When any subsequent transaction targets this contract, `get_invoke_tx_execution_context` reads the class hash from `contract_state_changes` and uses it to build the `ExecutionContext`. Since the class hash is undeclared, the OS cannot resolve any entry point, and execution fails unconditionally. [2](#0-1) 

The state commitment path in `compute_contract_state_commitment` → `hash_contract_state_changes` hashes and commits whatever `class_hash` is stored in the `StateEntry`, with no cross-check against the class tree. [3](#0-2) 

This is the direct analog of the `changeTreasury()` bug: a state-update function accepts a new value (class hash / treasury address) and writes it without validating it is a legitimate, previously-registered value, leaving the system in a broken state with no recovery path.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared value, every future call to that contract (including any `withdraw`, `transfer`, or `execute` entry point) fails at the OS level. Any ERC-20 tokens, ETH, or STRK held in the contract's storage become permanently inaccessible. There is no admin override or recovery mechanism in the OS for this condition.

---

### Likelihood Explanation

The attack requires only:
1. Deploying a contract (permissionless on StarkNet).
2. Including a `replace_class` syscall with an arbitrary undeclared felt (e.g., `1`) in the contract's logic.
3. Attracting user deposits before triggering the replacement.

No privileged role, leaked key, or external dependency is needed. Any unprivileged contract deployer is a valid attacker. The attack is irreversible once the block is finalized.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify it is present in `contract_class_changes` with a non-zero compiled class hash. Concretely, perform a `dict_read` on `contract_class_changes` for `class_hash` and assert the result is non-zero, mirroring the `assert_not_zero(compiled_class_hash)` guard already used in `execute_declare_transaction`. [4](#0-3) 

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` whose Sierra source contains:
   - `deposit()` — accepts STRK transfers, records balances in storage.
   - `rug()` — calls `replace_class(syscall_ptr, 0x1)` where `0x1` is never declared.
2. Users call `deposit()`, sending STRK to `MaliciousVault`.
3. Attacker submits an invoke transaction calling `rug()`.
4. The OS executes `execute_replace_class`:
   - `class_hash = 0x1` is read from the request.
   - No declared-class check is performed (the TODO is unimplemented).
   - `contract_state_changes` is updated: `MaliciousVault.class_hash = 0x1`.
5. Block is finalized; `compute_contract_state_commitment` commits the new state root containing `class_hash = 0x1`.
6. Any subsequent call to `MaliciousVault` (e.g., `withdraw()`) causes the OS to build an `ExecutionContext` with `class_hash = 0x1`, which has no entry points. Execution fails.
7. All user STRK is permanently frozen. [5](#0-4)
