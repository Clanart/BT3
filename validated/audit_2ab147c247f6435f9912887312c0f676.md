### Title
`MerkleBlob::batch_insert` Bypasses Duplicate-Key and Duplicate-Hash Guards Present in `insert`, Enabling Corrupted Tree Root — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::insert` enforces two invariants before writing any leaf: it rejects a key that already exists (`KeyAlreadyPresent`) and rejects a leaf hash that already exists (`HashAlreadyPresent`). `MerkleBlob::batch_insert`, the bulk-insertion path exposed directly to Python callers via `py_batch_insert`, silently skips both guards for every item beyond the first two, allowing duplicate keys and duplicate hashes to be written into the blob. The resulting tree has structurally invalid duplicate leaf nodes, a corrupted root hash, and produces forged or ambiguous proofs of inclusion.

---

### Finding Description

`insert` opens with two mandatory guard checks: [1](#0-0) 

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` calls `self.insert(…)` (with both guards active) only for the first two items when the tree has ≤ 1 existing leaf: [2](#0-1) 

For every remaining item it falls through to a raw `insert_entry_to_blob` call with **no duplicate check at all**: [3](#0-2) 

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { … Node::Leaf(LeafNode { hash, key, value, … }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no guard
    indexes.push(new_leaf_index);
}
```

This is the exact structural analog of the ERC-721 report: a "safe" variant (`insert`) performs a required validation callback; the "bulk" variant (`batch_insert`) omits it for performance, just as `transferFrom` omits `onERC721Received`.

The Python binding `py_batch_insert` is a first-class public API on `MerkleBlob`: [4](#0-3) 

and is declared in the published type stub: [5](#0-4) 

---

### Impact Explanation

A caller (or an attacker who can influence the key-value list passed to `batch_insert`) can supply a batch containing a key that already exists in the tree, or two identical keys within the same batch. Because neither guard fires, two `LeafNode` entries with the same `key` (or the same `hash`) are written into the blob. Consequences:

1. **Corrupted root hash** — internal node hashes are computed bottom-up from the raw leaf hashes; a duplicate leaf changes every ancestor hash, producing a root that does not correspond to any valid DataLayer state.
2. **Forged / ambiguous proof of inclusion** — `get_proof_of_inclusion` walks the tree by key; with two leaves sharing a key, the proof path is non-deterministic and can be made to "prove" a value that was never legitimately committed.
3. **`check_integrity` failure** — the `block_status_cache` leaf-hash-to-index map enforces uniqueness; a duplicate hash inserted via `batch_insert` leaves the cache inconsistent with the blob, causing integrity checks to fail or panic.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic … corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

`batch_insert` is the recommended high-throughput insertion path and is called by the DataLayer service when applying bulk updates derived from blockchain-committed DataLayer store deltas. A DataLayer store owner (an unprivileged network participant) publishes key-value updates; if the consuming service passes those updates directly to `batch_insert` without a prior deduplication pass, a malicious store owner can include a repeated key. No privileged access, key compromise, or node compromise is required — only the ability to publish a DataLayer update, which is open to any participant.

---

### Recommendation

Add the same two guards at the top of the bulk loop in `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // … existing raw insertion …
}
```

Alternatively, deduplicate the input vector before the loop and update `block_status_cache` atomically after the subtree is attached, mirroring the invariant that `insert` maintains.

### Proof of Concept

```rust
let mut blob = MerkleBlob::new(vec![]).unwrap();

// Pre-populate so leaf_count > 1, bypassing the safe insert path
for i in 0i64..3 {
    blob.insert(KeyId(i), ValueId(i), &sha256_num(&i), InsertLocation::Auto {}).unwrap();
}

let dup_key   = KeyId(0);          // already present
let dup_hash  = sha256_num(&0i64); // already present
let batch = vec![((dup_key, ValueId(99)), dup_hash)];

// Should return Err(KeyAlreadyPresent) — instead returns Ok(())
blob.batch_insert(batch).unwrap();

// Tree now has two leaves with KeyId(0); root hash is corrupted.
// check_integrity() will fail or produce an ambiguous proof.
blob.check_integrity().unwrap_err(); // demonstrates corruption
