### Title
Missing `state_diff_commitment` Verification in P2P Sync Allows Tampered `CompiledClassHash` to Persist in Storage — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary

The P2P sync client assembles a `ThinStateDiff` from peer-delivered `StateDiffChunk` messages and validates only the **count** of entries against `header.state_diff_length`. It never computes `calculate_state_diff_hash` on the assembled diff and never compares the result against the `state_diff_commitment` already stored in the block header. A malicious peer can therefore deliver a `DeclaredClass` chunk with the correct `class_hash_K` but a wrong `compiled_class_hash C'`, pass all guards, and cause `append_state_diff` to write `C'` into `compiled_class_hash_table`. Every subsequent CASM lookup for `class_hash_K` returns `C'`, causing wrong bytecode to be selected for execution.

### Finding Description

**Entrypoint — `parse_data_for_block`** in `crates/apollo_p2p_sync/src/client/state_diff.rs`:

The function reads `target_state_diff_len` from the stored block header and loops until `current_state_diff_len == target_state_diff_len`. [1](#0-0) 

For each received chunk it calls `unite_state_diffs`, which inserts `declared_class.compiled_class_hash` into `result.class_hash_to_compiled_class_hash` with no content check: [2](#0-1) 

After the loop, the only additional check is `validate_deprecated_declared_classes_non_conflicting`, which is unrelated to compiled class hashes. The assembled `ThinStateDiff` is returned and immediately written to storage: [3](#0-2) [4](#0-3) 

**Storage write — `append_state_diff`** in `crates/apollo_storage/src/state/mod.rs`:

`write_compiled_class_hashes` blindly inserts every `(class_hash, compiled_class_hash)` pair from the peer-supplied map into `compiled_class_hash_table` with no commitment check: [5](#0-4) [6](#0-5) 

**The missing guard — `calculate_state_diff_hash`** in `crates/starknet_api/src/block_hash/state_diff_hash.rs`:

`calculate_state_diff_hash` hashes `class_hash_to_compiled_class_hash` as part of the Poseidon commitment: [7](#0-6) 

The block header already carries `state_diff_commitment` (synced before the state diff), so the information needed to verify is present. The p2p sync path simply never uses it.

**Downstream impact — `get_compiled_class_hash`** in `crates/apollo_rpc_execution/src/state_reader.rs`:

Execution reads the compiled class hash directly from the stored state diff: [8](#0-7) 

If `C'` was stored, every call to `get_compiled_class_hash(class_hash_K)` returns `C'`, and the wrong CASM artifact is selected for execution.

**The committer's `verify_state_diff_hash` flag** (`crates/apollo_committer/src/committer.rs:167-176`) is a separate component that operates on the sequencer's own produced state diffs, not on p2p-synced storage. It provides no protection here. [9](#0-8) 

**Header signature verification** is also absent in the p2p header sync path — `parse_data_for_block` for headers checks only block number ordering and signature vector length, not the cryptographic signature: [10](#0-9) 

This means an attacker can forge both the header (with any `state_diff_commitment`) and the state diff, or — even if the header were verified — substitute a state diff whose length matches but whose `compiled_class_hash` values differ.

### Impact Explanation

`compiled_class_hash_table[class_hash_K]` is set to attacker-chosen `C'`. All subsequent executions that invoke a contract whose class hash is `class_hash_K` will look up CASM by `C'` instead of the correct `C`, causing **wrong bytecode to be executed**. This maps directly to the allowed Critical impact: *"Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution."*

### Likelihood Explanation

Any node running the p2p sync client is reachable by any peer in the network. No special privilege is required. The attacker only needs to be a peer that responds to a state diff query for a block that the victim has not yet synced. The length check is trivially satisfied by sending the correct number of entries with wrong values.

### Recommendation

In `parse_data_for_block` (or immediately before `write_to_storage`), after assembling the full `ThinStateDiff`, retrieve `header.state_diff_commitment`, call `calculate_state_diff_hash(&result)`, and compare. If they differ, return `Err(ParseDataError::BadPeer(...))` and report the peer. This is the same pattern already used by the committer's `verify_state_diff_hash` path.

### Proof of Concept

```rust
// Pseudocode for a Rust unit test (production storage, no mocks)
let correct_compiled = CompiledClassHash(Felt::from(0xCAFE));
let tampered_compiled = CompiledClassHash(Felt::from(0xDEAD));
let class_hash_k = ClassHash(Felt::from(0xABCD));

// Simulate what the p2p sync path does: store a ThinStateDiff
// with a tampered compiled class hash, bypassing commitment check.
let tampered_diff = ThinStateDiff {
    class_hash_to_compiled_class_hash: indexmap! { class_hash_k => tampered_compiled },
    ..Default::default()
};
writer.begin_rw_txn().unwrap()
    .append_state_diff(BlockNumber(0), tampered_diff).unwrap()
    .commit().unwrap();

// Read back via the execution state reader path.
let txn = reader.begin_ro_txn().unwrap();
let state_diff = txn.get_state_diff(BlockNumber(0)).unwrap().unwrap();
let stored = state_diff.class_hash_to_compiled_class_hash[&class_hash_k];

// This assertion passes — the wrong hash is stored and served.
assert_eq!(stored, tampered_compiled);  // C' not C
assert_ne!(stored, correct_compiled);
```
