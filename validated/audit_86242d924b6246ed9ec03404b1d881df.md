### Title
Missing `chunk.shard_id() == shard_id` guard in `set_state_header` allows cross-shard state header substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer without verifying that the embedded chunk's `shard_id()` matches the `shard_id` parameter. The `verify_path` call proves only that the chunk is *somewhere* in the block's chunk merkle tree, not that it occupies the position corresponding to `shard_id`. An attacker can supply shard 0's chunk data paired with shard 1's genuine receipt proofs, causing the node to persist `StateHeaderKey(1, sync_hash)` containing shard 0's `prev_state_root`.

### Finding Description
In `set_state_header` (`chain/chain/src/state_sync/adapter.rs`, lines 368–532):

- **Step 3a** calls `verify_path(chunk_headers_root, chunk_proof, ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))`. This verifies inclusion in the tree but not position. Shard 0's chunk with shard 0's proof passes this check even when `shard_id=1`. [8](#0-7) 

- **Step 4e** computes `receipts_hash = hash(ReceiptList(shard_id, receipts))` using the **parameter** `shard_id`. An attacker who provides shard 1's genuine on-chain receipt proofs (where `to_shard_id=1`) satisfies this check. [3](#0-2) 

- **Step 5** validates `state_root_node` against `chunk.prev_state_root()` — shard 0's state root node is consistent with shard 0's state root, so this passes. [5](#0-4) 

- **Storage**: the header is written under `StateHeaderKey(shard_id=1, sync_hash)` containing shard 0's chunk (and thus shard 0's `prev_state_root`). [9](#0-8) 

The crafted header passes all five validation steps because:
1. `validate_chunk_proofs` checks internal chunk consistency only — no shard_id comparison. [10](#0-9) 
2. `verify_path` is not `verify_path_with_index`; it does not enforce leaf position. [11](#0-10) 
3. Receipt proofs for shard 1 are public, derivable from block data.
4. Check 4g passes when both shards have the same `prev_chunk_height_included` (normal case).
5. `validate_state_root_node` is a pure hash check with no shard_id binding.

### Impact Explanation
`set_state_finalize` retrieves the stored header via `get_state_header(shard_id=1, sync_hash)`, gets shard 0's chunk, and calls `apply_chunk` with `RuntimeStorageConfig::new(chunk_header.prev_state_root(), true)` — shard 0's state root — for shard 1's `shard_uid`. This installs the wrong state root for shard 1, corrupting its committed state. [12](#0-11) 

### Likelihood Explanation
Any peer that can respond to state sync requests (no special privilege required) can craft this header using only public chain data. In a normal multi-shard network where all shards produce chunks at every height, all validation checks pass. The attack requires no cryptographic forgery — only recombination of valid public data.

### Recommendation
Add an explicit shard identity check immediately after extracting the chunk, before any other validation:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
``` [13](#0-12) 

### Proof of Concept
In a two-shard test-loop environment:
1. Obtain shard 0's `ShardStateSyncResponseHeader` via `compute_state_response_header(shard_id=0, sync_hash)`.
2. Obtain shard 1's genuine incoming receipt proofs from the chain store.
3. Construct a crafted `ShardStateSyncResponseHeaderV2` with shard 0's `chunk`, `chunk_proof`, `prev_chunk_header`, `prev_chunk_proof`, `state_root_node`, but shard 1's `incoming_receipts_proofs` and `root_proofs`.
4. Call `set_state_header(shard_id=1, sync_hash, crafted_header)`.
5. Assert `Ok(())` is returned (no rejection).
6. Read `StateHeaderKey(1, sync_hash)` from `DBCol::StateHeaders` and assert `chunk.shard_id() == 0` and `chunk.prev_state_root() == shard_0_prev_state_root`.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
