### Title
Native ETH Push-Transfer in `finTransfer` Can Be Permanently Blocked by a Recipient Contract, Irrecoverably Locking Bridge Funds - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` uses a push model to deliver native ETH to `payload.recipient`. If the recipient is a contract that reverts on ETH reception, every finalization attempt reverts and the ETH is permanently locked in the bridge with no recovery path.

---

### Finding Description

In `OmniBridge.finTransfer`, when `payload.tokenAddress == address(0)` (native ETH transfer), the contract pushes ETH directly to the user-supplied recipient address:

```solidity
// OmniBridge.sol lines 317-322
if (payload.tokenAddress == address(0)) {
    // slither-disable-next-line arbitrary-send-eth
    (bool success, ) = payload.recipient.call{value: payload.amount}(
        ""
    );
    if (!success) revert FailedToSendEther();
}
``` [1](#0-0) 

The `completedTransfers[payload.destinationNonce] = true` write at line 287 occurs before the ETH push: [2](#0-1) 

Because the entire transaction reverts when `FailedToSendEther` is thrown, the nonce is never durably marked as used. Every subsequent call to `finTransfer` with the same valid signature also reverts. There is no admin rescue function, no timeout withdrawal, and no alternative recipient path in the contract.

The `payload.recipient` is an EVM address chosen by the user when initiating the transfer on the NEAR side. An attacker can:

1. Deploy a contract with a togglable `receive()` that initially accepts ETH (to pass any off-chain solver checks).
2. Initiate a NEAR→EVM bridge transfer specifying this contract as the EVM recipient.
3. Flip the contract to reject ETH before any relayer calls `finTransfer`.
4. Every `finTransfer` attempt reverts with `FailedToSendEther`.

The NEAR tokens are already burned/locked on the NEAR side at step 2. The ETH held by the bridge for this transfer is permanently unclaimable. [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent, irrecoverable lock of user and bridge funds.**

- The user's NEAR tokens are burned at transfer initiation; they cannot be recovered.
- The corresponding ETH in the bridge contract is permanently locked; no admin function or timeout mechanism exists to rescue it.
- An attacker can repeat this across many transfers, progressively draining the bridge's ETH liquidity without ever receiving the ETH themselves (pure griefing / collateral damage to other users who need that ETH for their own withdrawals).

This matches the allowed impact: *"Critical. Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Medium-High.** The recipient address is fully attacker-controlled (set by the NEAR-side sender). No privileged role is required. The attack requires only:

- Deploying a cheap contract with a togglable `receive()`.
- Initiating a NEAR→EVM transfer (standard bridge user action).
- Toggling the contract before any relayer finalizes.

On low-fee L2s the cost is negligible. The attack is repeatable and anonymous.

---

### Recommendation

Replace the push model with a pull model for native ETH delivery:

1. Credit `payload.recipient` in an internal `pendingWithdrawals` mapping instead of pushing ETH immediately.
2. Expose a `withdraw()` function that lets the recipient pull their ETH at any time.
3. This decouples finalization success from recipient behavior, eliminating the revert-blocking vector.

Alternatively, wrap ETH as WETH before delivery so that `safeTransfer` to any address (including contracts) cannot revert due to missing `receive()`.

---

### Proof of Concept

```solidity
// Attacker contract
contract MaliciousRecipient {
    bool public acceptEther = true;

    function toggle() external { acceptEther = !acceptEther; }

    receive() external payable {
        require(acceptEther, "rejecting ETH");
    }
}
```

**Steps:**

1. Deploy `MaliciousRecipient` on the EVM chain; `acceptEther = true`.
2. On NEAR, call `init_transfer` with `recipient = address(MaliciousRecipient)` and `token = native ETH`, locking NEAR tokens.
3. Call `MaliciousRecipient.toggle()` → `acceptEther = false`.
4. Any relayer calls `OmniBridge.finTransfer(sig, payload)`:
   - Line 287: `completedTransfers[nonce] = true` (in memory, not yet committed).
   - Line 319: `payload.recipient.call{value: amount}("")` → `MaliciousRecipient.receive()` reverts.
   - Line 322: `revert FailedToSendEther()` → entire tx reverts, nonce never marked.
5. Repeat step 4 indefinitely — always reverts.
6. NEAR tokens are burned; ETH is permanently locked in `OmniBridge`. [4](#0-3)
