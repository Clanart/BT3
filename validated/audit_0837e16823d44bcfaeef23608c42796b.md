### Title
Unconditional Success Response in `execute_deploy` Syscall Regardless of Constructor Outcome - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall handler in the StarkNet OS unconditionally writes `failure_flag=0` (success) in its response header, regardless of whether the deployed contract's constructor actually succeeded or reverted. This is a direct analog to the reported bug: a state transition (reporting deployment success) is performed without checking the required condition (constructor did not revert). A calling contract will always observe a successful deployment, even when the constructor failed, enabling it to act on a false premise — including sending funds to an uninitialized contract address.

---

### Finding Description

In `execute_deploy` (`syscall_impls.cairo`, lines 527–554), after calling `deploy_contract`, the OS writes the response header with a hardcoded `failure_flag=0`:

```cairo
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
```

The `deploy_contract` call is made with `remaining_gas` and `revert_log` as implicit arguments, meaning the constructor can revert (the revert log mechanism exists precisely to undo constructor storage changes on failure). However, the response to the calling contract is always `failure_flag=0` — success — regardless of the constructor's actual outcome.

The developer-inserted TODO comment at line 538 explicitly acknowledges that failure handling is not implemented.

This is structurally identical to the reported bug pattern: a state-modifying action (reporting success to the caller) is taken unconditionally, without checking the required condition (constructor succeeded). In the original report, `purchasedToUnlocked(holder)` was called without verifying `_totalDelegated[holder] >= _purchased[holder]`. Here, `failure_flag=0` is asserted without verifying `is_reverted == 0`. [1](#0-0) 

For comparison, `contract_call_helper` — which handles `call_contract` — correctly propagates the revert flag:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

The `execute_deploy` function has `revert_log` as an implicit argument (it is in scope), confirming the infrastructure for revert propagation exists but is not used for the response flag. [3](#0-2) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

A calling contract that uses the `deploy` syscall will always receive `failure_flag=0` and a valid `contract_address` in the response. If the constructor reverted:

1. The revert log undoes the constructor's storage writes, leaving the new contract in an uninitialized state.
2. The calling contract, observing `failure_flag=0`, proceeds as if deployment succeeded.
3. The calling contract may transfer ERC-20 tokens or native assets to the returned `contract_address`, expecting a properly initialized contract to handle them.
4. Because the contract is uninitialized (constructor reverted), those funds are permanently locked — no withdrawal path exists.

This constitutes a **permanent freezing / direct loss of funds** reachable through normal contract execution.

---

### Likelihood Explanation

Any unprivileged user can submit an invoke transaction that calls a contract using the `deploy` syscall. The attacker controls:
- The class hash of the deployed contract (which can have a reverting constructor).
- The calldata passed to the constructor.

No privileged access, leaked keys, or operator cooperation is required. The attack is fully self-contained within a single transaction. The TODO comment confirms the developers are aware the failure path is unimplemented, meaning this is a known gap in the current production code.

---

### Recommendation

The `execute_deploy` function must propagate the constructor's revert status to the response header, mirroring the pattern used in `contract_call_helper`:

```cairo
// After calling deploy_contract, capture is_reverted from the constructor result.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

The `deploy_contract` function should be updated to return `is_reverted` alongside `retdata_size` and `retdata`, and the response should only report success when `is_reverted == 0`. This mirrors the fix in the original report: only perform the state transition (report success) when the required condition (constructor did not revert) is satisfied.

---

### Proof of Concept

1. Attacker deploys class `A` (a factory contract) and class `B` (a contract with a reverting constructor: `assert 1 == 0`).
2. Attacker submits an invoke transaction calling `A.__execute__`.
3. `A.__execute__` calls the `deploy` syscall with class `B`'s hash.
4. The OS executes `execute_deploy` → calls `deploy_contract` → constructor of `B` reverts → revert log undoes storage changes → `B` is uninitialized.
5. OS writes `ResponseHeader(gas=remaining_gas, failure_flag=0)` — unconditionally.
6. `A.__execute__` receives `failure_flag=0` and `contract_address=addr_B`.
7. `A.__execute__` calls ERC-20 `transfer(addr_B, amount)` — sending funds to the uninitialized contract.
8. Funds are permanently locked at `addr_B` with no recovery path.

The root cause is exclusively in `execute_deploy` at line 539 of `syscall_impls.cairo`, within the mandated scope `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/`. [4](#0-3)
