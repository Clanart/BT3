### Title
Non-Standard ERC20 Token `transfer()` Return-Value Check in `fin_transfer` Causes Irrecoverable Fund Lock — (`starknet/src/omni_bridge.cairo`)

---

### Summary

The StarkNet `OmniBridge.fin_transfer()` calls `IERC20Dispatcher.transfer()` on arbitrary custom (non-bridge) tokens and asserts the boolean return value. On StarkNet, tokens that do not conform to the standard ERC20 ABI (e.g., tokens whose `transfer` does not return a `bool`, or returns a non-standard type) will cause the dispatcher call to panic and revert the entire transaction. Because the source-chain funds are already committed (locked or burned) before `fin_transfer` is invoked, the recipient can never receive their tokens on StarkNet, and the funds are permanently frozen in the source-chain bridge.

---

### Finding Description

In `starknet/src/omni_bridge.cairo`, `fin_transfer` handles the release of custom (non-bridge-deployed) tokens via a raw `IERC20Dispatcher`:

```cairo
} else {
    let success = IERC20Dispatcher { contract_address: payload.token_address }
        .transfer(payload.recipient, payload.amount.into());
    assert(success, 'ERR_TRANSFER_FAILED');
}
``` [1](#0-0) 

`IERC20Dispatcher` is the strict (non-safe) OpenZeppelin dispatcher. If the target token's `transfer` function does not return a `bool` (or returns a type that cannot be decoded as `bool`), the Starknet ABI decoder panics and the entire transaction reverts. The `assert(success, ...)` line is a secondary failure point: even if the call does not panic, a token that returns `false` on success (or `true` on failure) will also cause a revert.

The same pattern appears in `init_transfer` for `transfer_from` and for the native-fee STRK transfer:

```cairo
let success = IERC20Dispatcher { contract_address: token_address }
    .transfer_from(caller, get_contract_address(), amount.into());
assert(success, 'ERR_TRANSFER_FROM_FAILED');
``` [2](#0-1) 

```cairo
let success = IERC20Dispatcher { contract_address: native_token }
    .transfer_from(caller, get_contract_address(), native_fee.into());
assert(success, 'ERR_FEE_TRANSFER_FAILED');
``` [3](#0-2) 

The `fin_transfer` path is the critical one. The nonce is marked finalised in the same transaction as the transfer:

```cairo
_set_transfer_finalised(ref self, payload.destination_nonce);
``` [4](#0-3) 

Because the revert rolls back the nonce marking, the nonce is not permanently consumed — but the transfer will **always** revert for that token, so the user can never successfully call `fin_transfer`. The source-chain funds (locked or burned on NEAR/EVM/Solana) are permanently unrecoverable without admin contract upgrade.

The codebase already acknowledges non-standard token metadata on StarkNet — `log_metadata` explicitly handles both old-style `felt252` and new-style `ByteArray` name/symbol fields:

```cairo
let name = if res.len() == 1 {
    // Old standard (felt252)
    ...
} else {
    // New standard (ByteArray)
    ...
};
``` [5](#0-4) 

This confirms the developers are aware that StarkNet tokens do not uniformly conform to a single interface standard, yet the transfer path does not apply the same defensive handling.

By contrast, the EVM `OmniBridge.sol` correctly uses `SafeERC20` throughout:

```solidity
using SafeERC20 for IERC20;
...
IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
``` [6](#0-5) [7](#0-6) 

The StarkNet contract lacks an equivalent safe-transfer wrapper.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

A user bridges a non-standard ERC20 token from NEAR (or EVM/Solana) to StarkNet. The source-chain bridge locks or burns the tokens. The relayer calls `fin_transfer` on StarkNet. The `IERC20Dispatcher.transfer()` call panics or returns `false` for the non-standard token. The transaction reverts. This repeats on every retry. The user's funds are permanently locked in the source-chain bridge with no on-chain recovery path.

---

### Likelihood Explanation

**Low-to-Medium.** StarkNet has a heterogeneous token ecosystem. Older tokens (pre-SNIP-2/SNIP-6 standardisation) may not return a `bool` from `transfer`, or may use `felt252` return values. Any such token that is registered as a custom (non-bridge-deployed) token in the OmniBridge — either by the admin or via the `addCustomToken`-equivalent flow — triggers this path. The `log_metadata` defensive code in the same contract confirms this is a realistic concern.

---

### Recommendation

1. Replace `IERC20Dispatcher` with `IERC20SafeDispatcher` (the safe variant that returns a `Result`) for all external token calls in `fin_transfer` and `init_transfer`, and handle the `Result` explicitly rather than asserting the boolean return value.
2. Remove the `assert(success, ...)` checks and instead propagate or handle errors gracefully.
3. Consider adding an integration test with a mock token that does not return a boolean to confirm the fix.

---

### Proof of Concept

1. Deploy a non-standard ERC20 token on StarkNet whose `transfer` function does not return a `bool` (e.g., returns `()` or a `felt252`).
2. Register this token as a custom token in the StarkNet OmniBridge (non-bridge-token path).
3. Initiate a transfer of this token from NEAR to StarkNet. The NEAR bridge locks the tokens.
4. The relayer calls `fin_transfer` on StarkNet with a valid MPC signature.
5. `IERC20Dispatcher { contract_address: payload.token_address }.transfer(...)` panics due to ABI mismatch.
6. The transaction reverts. The nonce is not consumed, but every subsequent retry also reverts.
7. The user's tokens remain locked in the NEAR bridge indefinitely with no on-chain recovery.
