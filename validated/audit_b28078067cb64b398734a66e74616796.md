### Title
`is_refund_required` Silently Ignores `ft_transfer_call`/`mint` Promise Failures, Permanently Freezing User Funds - (`near/omni-bridge/src/lib.rs`)

### Summary

When the NEAR bridge finalizes an inbound transfer and the downstream `ft_transfer_call` or `mint` call fails with a promise error, `is_refund_required` returns `false` ("Unexpected case: don't refund"). The callback `fin_transfer_send_tokens_callback` then takes the success path: it does not revert `unlock_tokens_if_needed`, does not remove the finalization record, and emits a `FinTransferEvent` as if the transfer succeeded. The recipient never receives tokens, the source-chain tokens remain locked/burned, and the finalization record blocks any re-submission — a permanent, irrecoverable freeze of user funds.

---

### Finding Description

**Step 1 — `process_fin_transfer_to_near` commits state before token delivery** [1](#0-0) 

`add_fin_transfer` writes the transfer ID into `finalised_transfers` and `unlock_tokens_if_needed` decrements `locked_tokens` — both happen synchronously, before the async `send_tokens` call.

**Step 2 — `send_tokens` dispatches `ft_transfer_call` or `mint` with a callback** [2](#0-1) 

When `msg` is non-empty, `send_tokens` uses `ft_transfer_call` (for native tokens) or `mint` with a message (for deployed tokens). The `is_ft_transfer_call` flag passed to the callback is `!msg.is_empty()`. [3](#0-2) 

**Step 3 — `is_refund_required` treats promise errors as "no refund"** [4](#0-3) 

When `is_ft_transfer_call` is `true` and the promise result is `Err` (the `ft_transfer_call` or `mint` panicked), the function returns `false`. The comment reads *"Unexpected case: don't refund."* This is the root cause: a failed delivery is silently treated as a success.

**Step 4 — `fin_transfer_send_tokens_callback` takes the success path on failure** [5](#0-4) 

Because `is_refund_required` returned `false`:
- `revert_lock_actions` is **not** called → `locked_tokens` remains decremented
- `remove_fin_transfer` is **not** called → the finalization record stays in `finalised_transfers`
- Fee is minted/transferred to the fee recipient
- `FinTransferEvent` is emitted

The recipient received nothing. The finalization record blocks any future re-submission of the same proof. The source-chain tokens are permanently locked/burned.

**Step 5 — `burn_tokens_if_needed` is also fire-and-forget** [6](#0-5) 

In the refund path (when `is_refund_required` correctly returns `true`), the burn is `.detach()`-ed. If the burn fails, the bridge holds extra unbacked tokens while `locked_tokens` has already been restored — a secondary accounting desync that inflates bridged supply.

---

### Impact Explanation

**Primary (High — Permanent freeze):** Any transfer whose `ft_transfer_call` or `mint` promise fails with an error results in:
- Source-chain tokens permanently locked/burned (no proof of failure exists to unlock them)
- NEAR-side finalization record blocking re-submission
- Recipient receiving zero tokens

This matches: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

**Secondary (High — Accounting corruption):** The detached `burn_tokens_if_needed` in the refund path means failed burns silently inflate the deployed token supply without backing, breaking bridge collateralization. This matches: *"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."*

---

### Likelihood Explanation

The primary path is reachable by any unprivileged bridge user:

1. A user initiates a transfer on EVM to a NEAR recipient address that is a contract whose `ft_on_transfer` panics (e.g., a contract that rejects unknown tokens, or a contract under upgrade).
2. The relayer submits the proof; `fin_transfer_callback` runs.
3. `send_tokens` dispatches `mint` with the non-empty `msg`.
4. The recipient's `ft_on_transfer` panics → promise result is `Err`.
5. `is_refund_required` returns `false` → permanent freeze.

The user need not be malicious; any legitimate recipient contract that panics on `ft_on_transfer` (e.g., due to an unexpected token, insufficient gas, or a contract bug) triggers this path. The `MINT_TOKEN_GAS` constant is only 5 TGas, which may itself be insufficient for complex token contracts, making this reachable even without a malicious recipient. [7](#0-6) 

---

### Recommendation

1. **Fix `is_refund_required` for the `Err` case:** When the promise result is `Err`, the delivery definitively failed and the tokens were returned to the bridge. Return `true` to trigger the refund path:

```rust
Err(_) => true,  // Promise failed: tokens returned, must refund
```

2. **Fix the `ft_transfer` (non-`ft_transfer_call`) path:** When `is_ft_transfer_call` is `false`, the callback currently never checks the promise result. Add a check:

```rust
} else {
    // ft_transfer: check if it failed
    env::promise_result_checked(0, 0).is_err()
}
```

3. **Await `burn_tokens_if_needed`:** Replace `.detach()` with a chained callback that verifies the burn succeeded before committing state changes, analogous to how `fin_transfer_send_tokens_callback` handles `send_tokens`.

---

### Proof of Concept

```
1. Attacker deploys a NEAR contract `panic_receiver.near` whose ft_on_transfer always panics.

2. Attacker calls EVM OmniBridge.initTransfer(token, amount, fee, nativeFee,
       recipient="panic_receiver.near", message="trigger")
   → EVM locks/burns `amount` tokens, emits InitTransfer event.

3. Relayer submits proof to NEAR bridge.fin_transfer().
   → fin_transfer_callback runs:
       add_fin_transfer(transfer_id)          // finalised_transfers[id] = true
       unlock_tokens_if_needed(Eth, token, amount)  // locked_tokens[Eth][token] -= amount
       send_tokens(token, panic_receiver.near, amount, "trigger")
           → mint(panic_receiver.near, amount, Some("trigger"))
               → panic_receiver.near.ft_on_transfer() PANICS
               → mint promise result = Err

4. fin_transfer_send_tokens_callback is called:
       is_refund_required(true) → Err branch → returns false
       → success path: fee minted, FinTransferEvent emitted
       → finalised_transfers[id] still = true
       → locked_tokens[Eth][token] still decremented

5. Result:
   - EVM: tokens permanently locked/burned
   - NEAR: recipient has 0 tokens
   - Re-submission: blocked (finalised_transfers[id] = true)
   - locked_tokens: permanently decremented (accounting corrupted)
```

    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.
                        amount.0 == 0
                    } else {
                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }
    }

    fn send_tokens(
        &self,
        token: AccountId,
        recipient: AccountId,
        amount: U128,
        msg: &str,
    ) -> Promise {
        let ft_transfer_call_gas = env::prepaid_gas()
            .saturating_sub(env::used_gas())
            .saturating_sub(SEND_TOKENS_CALLBACK_GAS) // TODO: not all send_tokens callbacks has the same gas.
            .saturating_sub(MINT_TOKEN_GAS)
            .min(FT_TRANSFER_CALL_GAS);

        let is_deployed_token = self.is_deployed_token(&token);

        if token == self.wnear_account_id && msg.is_empty() {
            // Unwrap wNEAR and transfer NEAR tokens
            ext_wnear_token::ext(self.wnear_account_id.clone())
                .with_static_gas(WNEAR_WITHDRAW_GAS)
                .with_attached_deposit(ONE_YOCTO)
                .near_withdraw(amount)
                .then(
                    Self::ext(env::current_account_id())
                        .with_static_gas(NEAR_WITHDRAW_CALLBACK_GAS)
                        .near_withdraw_callback(recipient, NearToken::from_yoctonear(amount.0)),
                )
        } else if is_deployed_token {
            let deposit = if msg.is_empty() {
                NO_DEPOSIT
            } else {
                ONE_YOCTO
            };

            require!(
                ft_transfer_call_gas >= MIN_FT_TRANSFER_CALL_GAS,
                BridgeError::NotEnoughGasForTokenTransfer(ft_transfer_call_gas).as_ref()
            );

            ext_token::ext(token)
                .with_attached_deposit(deposit)
                .with_static_gas(MINT_TOKEN_GAS.saturating_add(ft_transfer_call_gas))
                .mint(
                    recipient,
                    amount,
                    (!msg.is_empty()).then(|| msg.to_string()),
                )
        } else if msg.is_empty() {
            ext_token::ext(token)
                .with_attached_deposit(ONE_YOCTO)
                .with_static_gas(FT_TRANSFER_GAS)
                .ft_transfer(recipient, amount, None)
        } else {
            require!(
                ft_transfer_call_gas >= MIN_FT_TRANSFER_CALL_GAS,
                BridgeError::NotEnoughGasForTokenTransfer(ft_transfer_call_gas).as_ref()
            );

            ext_token::ext(token)
                .with_attached_deposit(ONE_YOCTO)
                .with_static_gas(ft_transfer_call_gas)
                .ft_transfer_call(recipient, amount, None, msg.to_string())
        }
