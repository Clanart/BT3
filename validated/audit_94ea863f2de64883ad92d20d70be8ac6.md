### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Allows Minting More Tokens Than Locked — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.initTransfer` records the caller-supplied `amount` in the `InitTransfer` event without verifying the actual tokens received by the contract. For ERC-20 tokens that deduct a transfer fee on every `transferFrom` call (fee-on-transfer tokens), the bridge holds `amount − transferFee` but broadcasts `amount` as the locked value. The NEAR bridge mints the full `amount` to the recipient, creating unbacked supply and permanently undercollateralizing the EVM vault.

---

### Finding Description

In `OmniBridge.initTransfer`, the non-bridge-token path performs:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ← user-supplied, not verified
);
``` [1](#0-0) 

Immediately after, the function emits the event with the same unverified `amount`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // ← not the actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

No balance-before / balance-after delta is computed. The `amount` value in the event is therefore the *requested* transfer amount, not the *actual* amount credited to the bridge contract.

The NEAR bridge's `fin_transfer_callback` processes the proof derived from this event and mints exactly `amount` tokens to the recipient: [3](#0-2) 

(The callback trusts the `InitTransferMessage.amount` field verbatim, which originates from the EVM event.)

---

### Impact Explanation

For every `initTransfer` call with a fee-on-transfer token:

- EVM bridge holds: `amount − transferFee`
- NEAR bridge mints: `amount`
- Surplus minted: `transferFee` worth of unbacked tokens per transfer

Repeated transfers inflate the NEAR-side supply beyond the EVM collateral. When holders bridge back, the EVM vault cannot satisfy all redemptions — the last redeemers find the vault drained, permanently freezing their funds. This matches **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

---

### Likelihood Explanation

Any ERC-20 token registered in the bridge (via `deployToken` or `addCustomToken`) that charges a transfer fee is exploitable. The `initTransfer` function contains no whitelist or token-type guard; it accepts any address. Tokens such as USDT (historically had a fee switch), STA, PAXG, and others have or have had on-chain transfer fees. A single registered fee-on-transfer token is sufficient to trigger the bug. No privileged access is required — any unprivileged user calling `initTransfer` with such a token triggers the discrepancy.

---

### Recommendation

Compute the actual received amount using a balance delta:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived > fee, "ERR_INSUFFICIENT_RECEIVED");
// use actualReceived (cast to uint128) in the event and downstream logic
```

This mirrors the mitigation recommended in H-06: check the relevant token balance before and after the operation and use the delta as the authoritative locked amount.

---

### Proof of Concept

1. Register a fee-on-transfer ERC-20 token (e.g., one charging 1% on every transfer) via `deployToken`.
2. Call `initTransfer(tokenAddress, 1000, 0, 0, "alice.near", "")`.
3. Bridge receives 990 tokens; event records `amount = 1000`.
4. NEAR bridge proof is submitted; `fin_transfer_callback` mints 1000 tokens to Alice.
5. Alice now holds 1000 NEAR-side tokens backed by only 990 EVM-side tokens.
6. Repeat until the vault is drained; the final redeemer cannot withdraw. [4](#0-3)
