### Title
Unprivileged HyperCore user can permanently burn bridged tokens by supplying an invalid `OmniAddress` recipient string through `coreReceiveWithData` — (`evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

### Summary
`coreReceiveWithData` passes a user-controlled `recipient` string directly to `OmniBridge.initTransfer()` without validating that it is a parseable `OmniAddress`. The EVM side burns tokens and emits `InitTransfer` with the malformed recipient. Every NEAR-side prover then fails to parse the recipient, making the transfer permanently unclaimable. Tokens are irrecoverably lost.

### Finding Description
In `HlBridgeToken.coreReceiveWithData`, the `ACTION_INIT_TRANSFER` branch ABI-decodes `(uint128 fee, string recipient, string message)` from the caller-supplied `data` bytes and forwards `recipient` verbatim to `OmniBridge.initTransfer()`:

```solidity
// HlBridgeToken.sol lines 124-135
(uint128 fee, string memory recipient, string memory message) = abi
    .decode(tail, (uint128, string, string));
uint128 amount128 = amount.toUint128();
_update(_systemAddress, address(this), amount);
IOmniBridgeInitTransfer(owner()).initTransfer(
    address(this), amount128, fee, 0, recipient, message
);
``` [9](#0-8) 

`OmniBridge.initTransfer()` performs no recipient format validation — it burns the tokens and emits `InitTransfer(sender, tokenAddress, originNonce, amount, fee, nativeFee, recipient, message)` with the raw string. [10](#0-9) 

On the NEAR side, all three prover paths call `.parse()` on the recipient string to construct an `OmniAddress`:

- EVM light-client prover: `event.data.recipient.parse().map_err(stringify)?`
- Wormhole prover: `transfer.recipient.parse().map_err(stringify)?`
- Starknet prover: `recipient_str.parse().map_err(stringify)?`

`OmniAddress::from_str` returns `Err(...)` for any unrecognized chain prefix or malformed address. The `?` operator propagates the error, causing the prover to return an error result. `fin_transfer_callback` then panics on `InvalidProofMessage`. The transfer can never be finalized. [11](#0-10) 

The `data` bytes originate from the HyperCore user's `sendToEvmWithData` call; the system address is a transparent relay. The user fully controls the `recipient` field. [12](#0-11) 

### Impact Explanation
Tokens are burned on HyperEVM (supply destroyed, `totalSupply` decremented) but can never be claimed on any destination chain. The `InitTransfer` event is emitted and the nonce is consumed, but no prover can produce a valid `ProverResult::InitTransfer` from it. There is no refund, no admin recovery path, and no way to re-submit the transfer. This is a **permanent, irrecoverable loss of user funds** — a Critical impact matching "Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds."

### Likelihood Explanation
Any HyperCore user who can call `sendToEvmWithData` targeting a `HyperliquedBridgeToken` can trigger this. No special privilege is required beyond holding a HyperCore spot balance. The malformed recipient can be as simple as `""` or `"invalid:xyz"`. The attack is trivially constructable and requires no coordination.

### Recommendation
Validate the `recipient` string in `coreReceiveWithData` before calling `initTransfer`. The simplest fix is to attempt `OmniAddress` parsing on the EVM side (or via a Solidity-level format check) and revert if it fails, so the HyperCore-side balance is never debited for an unroutable transfer. At minimum, reject empty strings and strings whose chain prefix is not in the supported set. The guard should be added at the `ACTION_INIT_TRANSFER` branch before `_update(_systemAddress, address(this), amount)` is called. [2](#0-1)
