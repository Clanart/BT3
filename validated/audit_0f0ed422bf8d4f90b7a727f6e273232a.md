### Title
Missing shard_id binding check in `set_state_header` allows cross-shard state root substitution — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a peer-supplied `ShardStateSyncResponseHeader` and stores it under `StateHeaderKey(shard_id, sync_hash)` without ever verifying that the embedded chunk's `shard_id` field matches the `shard_id` parameter. Because `validate_chunk_proofs` does not check shard identity and `verify_path` does not check the leaf's position in the Merkle tree, a malicious peer can supply a structurally valid header for shard_B while claiming it is for shard_A. The corrupted entry is then consumed by `set_state_part`, which derives the expected `state_root` from the stored header, causing the node to validate and install shard_B's trie state under shard_A.

---

### Finding Description

**Entrypoint**: `set_state_header` in `chain/chain/src/state_sync/adapter.rs`

The function performs five validation steps before writing to `DBCol::StateHeaders`:

**Step 1–2 (`validate_chunk_proofs`)**: Verifies the chunk's internal consistency — header hash, body hash, tx_root, and outgoing_receipts_root. [1](#0-0) 

This function checks only that the chunk body is self-consistent. It never reads or compares `chunk.shard_id()` against any external parameter.

**Step 3a (`verify_path` against `chunk_headers_root`)**: Verifies that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in the block's chunk Merkle tree. [2](#0-1) 

`verify_path` is defined as: [3](#0-2) 

It calls `compute_root_from_path`, which walks the path and combines hashes without checking the leaf's position (index) in the tree. A valid Merkle proof for chunk_B at position `shard_B_index` will produce the correct `chunk_headers_root` and pass this check, even though the caller requested `shard_A_index`.

**Step 3b**: Same `verify_path` logic applied to `prev_chunk_header` — same position-blindness applies. [4](#0-3) 

**Step 4 (receipt proofs)**: The loop computes `receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))` using the *requested* `shard_id` (shard_A). [5](#0-4) 

An attacker who replaces the `incoming_receipts_proofs` field with valid receipt proofs for shard_A (which are publicly derivable from block headers) will pass this check, while the chunk body and `prev_state_root` remain those of shard_B.

**Step 5**: Validates `state_root_node` against `chunk_inner.prev_state_root()` — this is shard_B's state root, and the check passes because the node is internally consistent with shard_B's trie. [6](#0-5) 

**Storage**: The header is written under `StateHeaderKey(shard_id_A, sync_hash)` — keyed by the *requested* shard_id, not the chunk's actual shard_id. [7](#0-6) 

**Missing guard**: There is no check equivalent to:
```rust
if chunk.shard_id() != shard_id { return Err(...); }
```
`ShardChunkHeaderInner::shard_id()` exists and is accessible: [8](#0-7)

### Impact Explanation

After `set_state_header(shard_id_A, sync_hash, header_for_shard_B)` succeeds, `set_state_part` reads the stored header to obtain the expected `state_root`:

```rust
let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
let chunk = shard_state_header.take_chunk();
let state_root = *chunk.take_header().take_inner().prev_state_root();
``` [9](#0-8) 

`state_root` is now shard_B's `prev_state_root`. All subsequent `validate_state_part` calls for shard_A are checked against shard_B's trie root. If the attacker also supplies matching state parts for shard_B, the node installs shard_B's trie under shard_A's `ShardUId`. `set_state_finalize` then applies shard_B's chunk (transactions, receipts) against shard_B's state root while believing it is operating on shard_A: [10](#0-9) 

The corrupted `ChunkExtra` is saved for shard_A's `ShardUId`, permanently diverging the node's view of shard_A's state from the canonical chain.

---

### Likelihood Explanation

State sync downloads headers from peers. Any peer reachable over the NEAR P2P network can serve a `StateResponseInfo` with an arbitrary `shard_id` field wrapping a valid header for a different shard. The attacker needs only:
1. A valid `ShardStateSyncResponseHeader` for shard_B (obtainable by syncing shard_B normally or requesting it from any honest node).
2. Valid receipt proofs for shard_A (derivable from public block headers).
3. Network access to the syncing node.

No validator, chunk producer, or operator privilege is required.

---

### Recommendation

Add an explicit shard_id binding check immediately after extracting the chunk in `set_state_header`, before any other validation:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: enforce shard identity before any further validation
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Apply the same check to `prev_chunk_header` if present:
```rust
if let Some(ref prev_hdr) = prev_chunk_header {
    if prev_hdr.shard_id() != shard_id {
        return Err(Error::Other(
            "set_shard_state failed: prev_chunk shard_id does not match requested shard_id".into(),
        ));
    }
}
```

### Proof of Concept

A cargo test would:
1. Build a two-shard `TestEnv` and produce enough blocks to reach a sync point.
2. Call `get_state_response_header(shard_B, sync_hash)` on an honest node to obtain a valid `ShardStateSyncResponseHeader` for shard_B.
3. Replace its `incoming_receipts_proofs` with valid proofs for shard_A (or use an empty list if the block has no included chunks for that range).
4. Call `set_state_header(shard_A, sync_hash, header_for_shard_B)` on the syncing node.
5. Assert the call returns `Err(...)` — currently it returns `Ok(())`.
6. Assert that `get_state_header(shard_A, sync_hash)` does **not** return a header whose `chunk_prev_state_root()` equals shard_B's state root.

The test demonstrates that the missing `chunk.shard_id() != shard_id` guard allows the corrupted `StateHeaderKey(shard_A, sync_hash)` to be written to `DBCol::StateHeaders`.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
