Audit Report

## Title
Unattributed ETH balance in `pay()` silently subsidises any subsequent WETH swap, permanently destroying the original depositor's funds — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay` uses the raw `address(this).balance` to fund WETH payments without verifying that the ETH was deposited by the current `payer`. Because `multicall` and all four swap entry-points are `payable`, any ETH sent to the router that is not explicitly refunded is silently consumed to cover a completely different user's WETH swap obligation, resulting in permanent, unrecoverable loss of the original depositor's principal.

## Finding Description

**Root cause — unattributed balance read in `pay`:** [1](#0-0) 

`address(this).balance` is a global, unattributed value. The function never checks that the ETH on the contract was deposited by `payer` in the current call. Any ETH left on the router from any prior call is treated as freely available to cover the current WETH payment.

**Stranding path — `receive()` guard is insufficient:** [2](#0-1) 

The `receive()` guard only fires when ETH arrives with *no calldata*. ETH sent via a payable function call (e.g., `multicall{value: B}(...)`) routes to the function selector, not `receive()`, so the guard is bypassed entirely.

`multicall` is `payable`: [3](#0-2) 

All four swap entry-points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) are also `payable`. Any call to these functions that sends more ETH than is consumed leaves the surplus stranded on the router with no owner record and no automatic refund.

**Consumption path — callback calls `pay` with stranded balance:** [4](#0-3) 

When a subsequent caller swaps with `tokenIn = WETH`, the pool fires `metricOmmSwapCallback` → `_justPayCallback` → `pay(WETH, payer, pool, value)`. If `address(this).balance > 0`, the partial-ETH branch wraps and forwards the stranded ETH to the pool, reducing the `transferFrom` pull on the actual `payer` by exactly `nativeBalance`. The original depositor's ETH is gone; the subsequent swapper's WETH approval requirement is silently reduced.

**No ETH attribution exists anywhere in the transient context:** [5](#0-4) 

`TransientCallbackPool.set` records pool, callback mode, payer address, and token — but never `msg.value`. There is no mechanism to bind a deposited ETH amount to a specific caller.

## Impact Explanation

Direct, permanent loss of user principal. The original ETH depositor receives nothing in return; the subsequent swapper's WETH cost is silently reduced. The pool receives the correct input amount, but it is sourced from the wrong party — a swap conservation failure. For any realistic stranded amount `B`, this exceeds Sherlock's High/Critical threshold. No privileged access is required and no funds are recoverable.

## Likelihood Explanation

The stranding condition is trivially reachable by any unprivileged user:
- `multicall{value: B}([])` with an empty call array strands `B` ETH immediately.
- Any payable swap call that sends more ETH than `amountIn` strands the surplus.
- Users routinely omit `refundETH` when they believe their swap will consume exactly their ETH.

An attacker can passively monitor the router's ETH balance and trigger the exploit with a single `exactInputSingle` call where `tokenIn = WETH` and their WETH approval is only `value - nativeBalance`, requiring no privileged access, no malicious pool, and no special timing.

## Recommendation

Introduce per-call ETH attribution. At the entry of each `payable` function, store `msg.value` keyed to `msg.sender` (e.g., in transient storage). In `pay`, only consume ETH up to the amount attributed to `payer` in the current top-level call, not the global `address(this).balance`. Alternatively, restrict the ETH-wrapping path to fire only when `payer == msg.sender` and the attributed `msg.value` for that call covers `value`, reverting otherwise rather than drawing from an unattributed pool.

## Proof of Concept

```
1. userB calls router.multicall{value: 0.5 ether}([])
   → no calls executed, 0.5 ETH stranded on router
   → router.balance == 0.5 ETH

2. userA approves router for 0.5 WETH (not 1 WETH).

3. userA calls router.exactInputSingle(
       pool = WETH/TOKEN pool,
       tokenIn = WETH,
       amountIn = 1 ether,
       ...
   )
   → _setNextCallbackContext(pool, JUST_PAY, userA, WETH)

4. Pool fires metricOmmSwapCallback
   → _justPayCallback
   → pay(WETH, userA, pool, 1 ether)
       nativeBalance = 0.5 ether  ← userB's stranded ETH
       branch: nativeBalance > 0
       → WETH.deposit{value: 0.5 ETH}()
       → WETH.transfer(pool, 0.5 WETH)
       → WETH.transferFrom(userA, pool, 0.5 WETH)  ← succeeds (userA had 0.5 WETH approved)

5. Swap settles. userA receives full output for 1 WETH input.
   userB's 0.5 ETH is permanently destroyed.
```
