### Title
Missing Recipient String Validation in `initTransfer` Enables Permanent Irrecoverable Token Loss — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The EVM `OmniBridge.initTransfer` function accepts a user-supplied `recipient` string but performs no validation that it is non-empty or parseable as a valid cross-chain address before locking or burning the caller's tokens. A user who submits an empty or malformed recipient causes their tokens to be permanently destroyed (bridge tokens) or permanently locked (native tokens) with no on-chain recovery path.

---

### Finding Description

`OmniBridge.initTransfer` at lines 373–437 of `evm/src/omni-bridge/contracts/OmniBridge.sol` accepts `string calldata recipient` as a user-controlled parameter. The only pre-transfer guard is the fee check:

```solidity
if (fee >= amount) {
    revert InvalidFee();
}
``` [1](#0-0) 

No check is performed on `recipient`. Immediately after, tokens are either burned (for bridge tokens) or pulled into the contract (for native tokens):

```solidity
} else if (isBridgeToken[tokenAddress]) {
    BridgeToken(tokenAddress).burn(msg.sender, amount);
} else {
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
}
``` [2](#0-1) 

The `InitTransfer` event is then emitted with the empty/invalid recipient string: [3](#0-2) 

The NEAR bridge's `fin_transfer` callback receives this event via the prover and attempts to deserialize the recipient field into an `OmniAddress`: [4](#0-3) 

An empty string cannot be parsed into any `OmniAddress` variant. The `fin_transfer` call on NEAR reverts. The EVM-side tokens are already burned or locked, and there is no on-chain refund or recovery function in `OmniBridge.sol`.

The same missing validation exists in `initTransfer1155`: [5](#0-4) 

The analogous gap in the Solana bridge is explicitly acknowledged in `solana/SECURITY.md` as a known issue:

> **No validation of `recipient` string in `InitTransferPayload`** — An invalid recipient causes the transfer to fail on the NEAR side after tokens are locked/burned on Solana. Manual intervention would be needed. [6](#0-5) 

For Solana, tokens are locked in a vault and manual intervention is theoretically possible. For EVM bridge tokens that are **burned**, no manual intervention can recover them — the supply is destroyed.

The StarkNet `init_transfer` does validate `amount > 0` and `fee < amount` but also omits any recipient validation: [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent irrecoverable loss of user funds.**

- For `isBridgeToken[tokenAddress] == true`: `BridgeToken.burn` destroys the tokens. There is no mint-back or refund path in `OmniBridge.sol`. Funds are permanently gone.
- For native ERC-20 tokens: tokens are locked in the bridge contract. No `recoverTransfer` or equivalent function exists in `OmniBridge.sol`, so recovery requires a contract upgrade — not guaranteed and not user-accessible.
- In both cases the `InitTransfer` event is emitted and the nonce is incremented, so the state is committed before the failure is discovered on NEAR.

---

### Likelihood Explanation

Any unprivileged user can call `initTransfer` directly. The empty-recipient scenario can occur through:

1. **User error**: a frontend bug, SDK misconfiguration, or direct contract call with `recipient = ""`.
2. **Griefing**: an attacker can cause another user's tokens to be lost if they control the frontend or relay layer that constructs the call.

No special role, key, or privileged access is required. The function is `external payable` with only a pause guard.

---

### Recommendation

Add an explicit non-empty recipient check at the top of `initTransfer` (and `initTransfer1155`) before any token movement:

```solidity
if (bytes(recipient).length == 0) {
    revert InvalidRecipient();
}
```

Optionally, add a minimum-length sanity check (e.g., `>= 5` bytes) to reject obviously malformed strings. The NEAR bridge already enforces `recipient.get_chain() != ChainKind::Near` on its side: [8](#0-7) 

The EVM entry point should mirror this defensive posture before committing any token state change.

---

### Proof of Concept

1. Attacker or user calls:
   ```solidity
   OmniBridge.initTransfer(
       bridgeTokenAddress,  // isBridgeToken[addr] == true
       100,                 // amount
       0,                   // fee  (0 < 100, passes fee check)
       0,                   // nativeFee
       "",                  // recipient — empty string, no validation
       ""
   );
   ```
2. `fee >= amount` → `0 >= 100` → false → no revert. [9](#0-8) 
3. `BridgeToken(bridgeTokenAddress).burn(msg.sender, 100)` executes — 100 tokens destroyed. [10](#0-9) 
4. `InitTransfer` event emitted with `recipient = ""`. [3](#0-2) 
5. NEAR relayer picks up the event and calls `fin_transfer`. The prover result contains `recipient = ""`.
6. NEAR attempts `"".parse::<OmniAddress>()` — fails. `fin_transfer` reverts. [4](#0-3) 
7. 100 bridge tokens are permanently destroyed. No recovery path exists.
