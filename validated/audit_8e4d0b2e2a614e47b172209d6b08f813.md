### Title
Zero `max_fee` Bypass Allows Fee-Free Transaction Execution — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `charge_fee` function in the StarkNet OS unconditionally skips all fee collection when `compute_max_possible_fee` returns `0`. Because `max_fee` is computed as a product of user-supplied price fields, and because the L2 gas *amount* (which governs execution budget) is sourced from a completely separate field, an unprivileged sender can craft a valid V3 transaction that carries a non-zero L2 gas budget for execution while producing a `max_fee` of exactly `0`. The OS then executes the full validate + execute pipeline and transfers nothing to the sequencer.

---

### Finding Description

`compute_max_possible_fee` (lines 87–102) computes the maximum chargeable fee as:

```
max_fee = l1_gas.max_amount * l1_gas.max_price_per_unit
        + l2_gas.max_amount * (l2_gas.max_price_per_unit + tip)
        + l1_data_gas.max_amount * l1_data_gas.max_price_per_unit
```

`charge_fee` (lines 111–165) then gates the entire fee-transfer path on this value:

```cairo
if (max_fee == 0) {
    return ();
}
```

Separately, `get_initial_user_gas_bound` (lines 75–78) derives the transaction's execution gas budget from only one field:

```cairo
return common_tx_fields.resource_bounds[L2_GAS_INDEX].max_amount;
```

These two quantities are **independent**. A sender can set:

| Field | Value |
|---|---|
| `l2_gas.max_amount` | `N > 0` (execution gas) |
| `l2_gas.max_price_per_unit` | `0` |
| `tip` | `0` |
| `l1_gas.max_amount` | `0` |
| `l1_data_gas.max_amount` | `0` |

This yields `max_fee = N * (0 + 0) + 0 + 0 = 0` while `initial_user_gas_bound = N > 0`. The OS runs the full validate and execute entry points with gas budget `N`, then calls `charge_fee`, which immediately returns at the `max_fee == 0` branch without executing the ERC-20 transfer to the sequencer. ---

### Impact Explanation

**Critical — Direct loss of funds.**

The sequencer's only on-chain compensation mechanism is the ERC-20 fee transfer executed inside `charge_fee`. When `max_fee == 0` the transfer is never initiated. The sequencer expends real computational resources (Cairo VM steps, builtins, state reads/writes) executing the transaction's validate and execute entry points, but receives zero fee tokens in return. An attacker who repeatedly submits such transactions drains sequencer resources without payment, constituting a direct, provable loss of fee-token funds that should have been credited to `block_context.block_info_for_execute.sequencer_address`. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

**High.** Any unprivileged V3 transaction sender can set `max_price_per_unit = 0` and `tip = 0` for all resource types while keeping `l2_gas.max_amount > 0`. This requires no special privilege, no key compromise, and no third-party dependency. The OS-level Cairo code enforces no lower bound on prices. The attack is repeatable across invoke, deploy-account, and declare transaction types, all of which call `charge_fee` through the same path. [6](#0-5) [7](#0-6) 

---

### Recommendation

Remove the early-return shortcut that treats `max_fee == 0` as a signal to skip fee collection entirely. Instead, enforce that for any non-bootstrap V3 transaction the actual fee charged is verified against the resource bounds independently of whether the product of price × amount happens to be zero. Concretely:

1. Add a protocol-level assertion that at least one of `l2_gas.max_price_per_unit` or `tip` is non-zero for any transaction that consumes L2 gas, before entering the execute pipeline.
2. Alternatively, compute the fee obligation from *actual gas consumed* rather than from the user-declared maximum, and enforce a non-zero minimum fee for any transaction that performs state-changing execution.

---

### Proof of Concept

**Attacker constructs a V3 invoke transaction with:**

```
resource_bounds = [
  L1_GAS:      { max_amount: 0,   max_price_per_unit: 0 },
  L2_GAS:      { max_amount: 1_000_000, max_price_per_unit: 0 },
  L1_DATA_GAS: { max_amount: 0,   max_price_per_unit: 0 },
]
tip = 0
```

**OS execution trace:**

1. `execute_invoke_function_transaction` calls `get_initial_user_gas_bound` → returns `1_000_000` (non-zero, execution proceeds).
2. `run_validate` executes the account's `__validate__` entry point consuming gas from the budget.
3. `non_reverting_select_execute_entry_point_func` executes `__execute__`, performing arbitrary state changes (e.g., ERC-20 transfers).
4. `charge_fee` is called. `compute_max_possible_fee` returns `0 + 1_000_000*(0+0) + 0 = 0`.
5. Branch `if (max_fee == 0) { return (); }` fires — **no ERC-20 transfer to sequencer occurs**.
6. The proven block output records the state changes from step 3 with zero fee paid.

The attacker has executed an arbitrary transaction, including token transfers, at zero cost. [8](#0-7) [9](#0-8)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
