### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Bricking — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary class hash as the replacement target without verifying that the hash corresponds to a previously declared class. A contract can replace its own class hash with an undeclared or zero value, permanently rendering itself unexecutable and freezing all funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall. After deducting gas, it reads `class_hash` directly from the caller-supplied request and writes it into the contract's `StateEntry` with no validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment explicitly acknowledges the missing check. [1](#0-0) 

By contrast, the `execute_declare_transaction` function enforces that a class can only be declared once by asserting `prev_value=0` in the class changes dict: [2](#0-1) 

The two state structures are independent: `contract_class_changes` (Sierra hash → compiled hash) and `contract_state_changes` (contract address → `StateEntry` containing `class_hash`). The OS resolves a contract's executable code by looking up `StateEntry.class_hash` in the declared class map. If `StateEntry.class_hash` is set to a value that has no entry in `contract_class_changes`, execution of that contract will fail at the class-lookup step. [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that calls `replace_class` with an undeclared class hash (including `0`, i.e., `UNINITIALIZED_CLASS_HASH`) will have its `StateEntry.class_hash` set to an invalid value. All subsequent invocations of that contract — including calls to withdraw or transfer funds — will fail because the OS cannot resolve the class to execute. Since the class declaration is a one-time operation (`prev_value=0` enforced), there is no protocol-level mechanism to repair the contract's class hash from outside. Funds held by the contract are permanently inaccessible.

---

### Likelihood Explanation

**Medium-High.** The `replace_class` syscall is callable by any contract during execution — no privileged role is required. A malicious contract deployer can intentionally trigger this by deploying a contract whose constructor or any entry point calls `replace_class(0)` or `replace_class(<undeclared_hash>)`. The TODO comment in the production code confirms the check was intentionally deferred, meaning the gap is present in the current deployed OS program. Any user who can submit a transaction to a contract that exposes `replace_class` (directly or via a library call) can trigger this path.

---

### Recommendation

Before updating `contract_state_changes` in `execute_replace_class`, verify that `request.class_hash` exists as a key in `contract_class_changes` (i.e., has a non-zero compiled class hash). This mirrors the invariant already enforced in `execute_declare_transaction` via `assert_not_zero(compiled_class_hash)` and `prev_value=0`. The check should be:

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This ensures `replace_class` can only target classes that have been properly declared and have a valid compiled class hash.

---

### Proof of Concept

1. Attacker deploys contract `C` with a valid class hash `H_valid`.
2. `C`'s `__execute__` entry point calls `replace_class(class_hash=0)` (or any undeclared felt).
3. The OS processes `execute_replace_class`: gas is deducted, `request.class_hash = 0` is read, the TODO check is absent, and `StateEntry(class_hash=0, ...)` is written for `C`'s address.
4. The block is proven and committed. `C`'s on-chain state now has `class_hash = 0`.
5. Any subsequent transaction targeting `C` (e.g., to withdraw ETH or ERC-20 tokens held by `C`) causes the OS to look up compiled class for hash `0`. `UNINITIALIZED_CLASS_HASH = 0` has no compiled class entry; execution fails.
6. All funds held by `C` are permanently frozen with no recovery path. [4](#0-3) [5](#0-4)
