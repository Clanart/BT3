### Title
Missing Recipient Validation in `ACTION_TRANSFER` Allows Permanent Token Lock — (`evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

---

### Summary

`coreReceiveWithData` with `ACTION_TRANSFER` decodes the recipient address directly from attacker-controlled `data` with no guard against `recipient == address(this)`. A HyperCore user can set the EVM recipient to the token contract's own address, causing `_update(_systemAddress, address(token), amount)` to move tokens into the contract's own balance with no release path.

---

### Finding Description

In `coreReceiveWithData`, the `ACTION_TRANSFER` branch is:

```solidity
if (action == ACTION_TRANSFER) {
    address recipient = abi.decode(tail, (address));
    _update(_systemAddress, recipient, amount);
}
``` [1](#0-0) 

The `data` field is supplied by the HyperCore user via `sendToEvmWithData` and relayed verbatim by the system address. The only access guard is `msg.sender != _systemAddress`, which is satisfied by the system address acting as a faithful relay. [2](#0-1) 

There is no check that `recipient != address(this)`. If the attacker encodes `address(token)` as the recipient, `_update` moves `amount` tokens from `_systemAddress` into the token contract's own balance.

---

### Impact Explanation

Once tokens land at `address(token)`, there is no release path:

- `BridgeToken.burn` is `onlyOwner` (OmniBridge admin) and destroys tokens rather than recovering them to a user. [3](#0-2) 
- `OmniBridge.initTransfer` for bridge tokens calls `BridgeToken(tokenAddress).burn(msg.sender, amount)` — `msg.sender` would need to be `address(token)` itself, which has no function to initiate this call. [4](#0-3) 
- The `ACTION_INIT_TRANSFER` path always pulls from `_systemAddress`, not from any existing balance at `address(this)`. [5](#0-4) 
- There is no sweep or rescue function anywhere in `HyperliquedBridgeToken` or `BridgeToken`.

The tokens are permanently irrecoverable from the user's perspective. This matches the allowed impact: **permanent freezing / irrecoverable lock of user funds in bridge flows**.

---

### Likelihood Explanation

The attacker is unprivileged — any HyperCore account can call `sendToEvmWithData`. The `data` payload is fully attacker-controlled. The system address is a trusted relay that passes the data through without sanitization. The attack requires a single HyperCore transaction with a crafted `data` field. No privileged access, leaked keys, or external dependency compromise is needed.

The scenario also covers accidental self-locking: a user who mistakenly passes `address(token)` as the EVM destination (e.g., copy-paste error) suffers the same permanent loss with no recourse.

---

### Recommendation

Add a recipient validation check in the `ACTION_TRANSFER` branch:

```solidity
if (action == ACTION_TRANSFER) {
    address recipient = abi.decode(tail, (address));
    require(recipient != address(this), "Invalid recipient: token contract");
    require(recipient != address(0), "Invalid recipient: zero address");
    _update(_systemAddress, recipient, amount);
}
``` [1](#0-0) 

---

### Proof of Concept

1. Attacker holds tokens on HyperCore.
2. Attacker calls `sendToEvmWithData` on HyperCore with:
   - EVM target = `address(token)`
   - `data` = `0x00` + `abi.encode(address(token))`
   - `amount` = N
3. System address calls `coreReceiveWithData(from, ..., N, ..., data)`.
4. `action == 0` (ACTION_TRANSFER), `recipient = address(token)`.
5. `_update(_systemAddress, address(token), N)` executes — `token.balanceOf(address(token)) == N`.
6. No function exists to move these tokens out. Funds are permanently locked.
