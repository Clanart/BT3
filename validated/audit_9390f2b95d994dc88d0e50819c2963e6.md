Audit Report

## Title
Swap Conservation Violation in `buyToken1InBinSpecifiedIn` and `buyToken0InBinSpecifiedIn`: Undershoot Fallback Charges Full Input Without Updating Output — (`metric-core/contracts/libraries/SwapMath.sol`)

## Summary
In `SwapMath.buyToken1InBinSpecifiedIn`, when both the analytical Stage 1 and the linear-refinement Stage 2 undershoot (`totalIn0Scaled < state.amountSpecifiedRemainingScaled` with `targetPos > minFinalBinPos`), the fallback at lines 758-760 sets `totalIn0Scaled = state.amountSpecifiedRemainingScaled` without recomputing `out1Scaled`. The settlement at lines 783-784 then charges the trader the full remaining token0 input while crediting only the token1 output computed for the partial `targetPos` from Stage 2, permanently transferring the excess token0 to LP balances. The identical defect exists in `buyToken0InBinSpecifiedIn` at lines 620-622.

## Finding Description
The function resolves the partial-fill position through three stages:

**Stage 1 (lines 720-736):** Analytical closed-form solution computes `targetPos`, `out1Scaled`, and `totalIn0Scaled`.

**Stage 2 (lines 738-756):** If `totalIn0Scaled < remaining && targetPos > minFinalBinPos`, a linear-scaling refinement recomputes `targetPos` using `scaledDelta = ceil(delta * remaining / totalIn0Scaled)`, then recomputes both `out1Scaled` and `totalIn0Scaled` for the new position.

**Stage 3 (lines 758-760):** If Stage 2 still undershoots:
```solidity
if (totalIn0Scaled < state.amountSpecifiedRemainingScaled && targetPos > minFinalBinPos) {
    totalIn0Scaled = state.amountSpecifiedRemainingScaled;
}
```
`out1Scaled` is not updated. Settlement then executes:
```solidity
state.amountSpecifiedRemainingScaled -= totalIn0Scaled;   // line 783: full remaining input consumed
state.amountCalculatedScaled += out1Scaled;               // line 784: partial output credited
```

Stage 3 is reachable because Stage 2's linear scaling assumes cost is proportional to position distance, but `calculateRequiredToken(out1Scaled, avgPriceX64)` uses an arithmetic mean of two prices that are themselves non-linear functions of `targetPos`. In bins with wide `upperPriceX64 / lowerPriceX64` ratios, the cost curve is sufficiently convex that one linear refinement step leaves `totalIn0Scaled < remaining` while `targetPos` has not yet reached `minFinalBinPos` (i.e., `scaledDelta` did not push `targetPos` past the clamp boundary).

The overshoot branch at lines 762-773 correctly handles the symmetric case by proportionally rescaling `out1Scaled` alongside `totalIn0Scaled`:
```solidity
out1Scaled = (out1Scaled * state.amountSpecifiedRemainingScaled) / totalIn0Scaled;
totalIn0Scaled = state.amountSpecifiedRemainingScaled;
```
The undershoot fallback has no equivalent `out1Scaled` update.

The symmetric defect in `buyToken0InBinSpecifiedIn` at lines 620-622 sets `totalIn1Scaled = state.amountSpecifiedRemainingScaled` without updating `out0Scaled`, with settlement at lines 643-644 exhibiting the same pattern.

## Impact Explanation
When Stage 3 fires, `binState.token0BalanceScaled` is incremented by the full `totalIn0Scaled - protocolFeeAmountScaled` (lines 780-781) while `binState.token1BalanceScaled` is decremented by only the partial `out1Scaled` (line 779). The trader pays `state.amountSpecifiedRemainingScaled` token0 but receives `out1Scaled` token1 where `out1Scaled * avgInvertedPrice < totalIn0Scaled`. The excess token0 accrues permanently to LP balances. This is a direct, permanent loss of trader principal — a swap conservation failure meeting Critical/High severity thresholds.

## Likelihood Explanation
The condition requires: (a) the analytical solution undershoots, AND (b) one linear refinement still undershoots. This occurs in bins where `upperPriceX64 / lowerPriceX64` is significantly above 1 and the input amount lands in the non-linear region of the quadratic cost curve. These are normal, permissionless pool configurations. No privileged setup is required; any unprivileged trader swapping token0→token1 (or token1→token0) through such a bin can trigger it by submitting an appropriately sized swap.

## Recommendation
Replace the fallback at lines 758-760 with a full traversal to `minFinalBinPos`, recomputing `out1Scaled` and `totalIn0Scaled` consistently:
```solidity
if (totalIn0Scaled < state.amountSpecifiedRemainingScaled && targetPos > minFinalBinPos) {
    targetPos = minFinalBinPos;
    out1Scaled = calculateOutputToken1FromBinPosition(binState.token1BalanceScaled, currBinPos, targetPos);
    invertedFinalPriceX64 =
        invertPriceX64(calculatePriceAtBinPosition(lowerPriceX64, upperPriceX64, targetPos, Math.Rounding.Floor));
    avgPriceX64 = calculateArithmeticMean(invertedStartingPriceX64, invertedFinalPriceX64);
    in0WithoutFeeScaled = calculateRequiredToken(out1Scaled, avgPriceX64);
    totalIn0Scaled = grossInputWithBinFeeCeil(in0WithoutFeeScaled, onePlusSellFeeX64);
}
```
Apply the same fix to `buyToken0InBinSpecifiedIn` lines 620-622, recomputing `out0Scaled` and `totalIn1Scaled` at `maxFinalBinPos`.

## Proof of Concept
Configure a pool bin with `upperPriceX64 / lowerPriceX64 >= 100` (wide bin), `currBinPos` near `type(uint104).max / 2`, and submit a swap with `amountSpecifiedRemainingScaled` set to approximately `1.05 * totalIn0Scaled_after_analytical` — large enough that Stage 1 undershoots but small enough that Stage 2's linear refinement does not push `targetPos` to `minFinalBinPos`. After the swap, verify:
```
out1Scaled * avgInvertedPrice / (1 << 64) < charged_token0
```
where `charged_token0 = amountSpecifiedRemainingScaled_before - amountSpecifiedRemainingScaled_after`. The shortfall in token1 output relative to token0 charged confirms the conservation violation. A Foundry property test fuzzing `(lowerPriceX64, upperPriceX64, currBinPos, remaining)` with the invariant `assertLe(charged, out1 * invertedAvgPrice / (1<<64) + TOLERANCE)` will reproduce the failure.
