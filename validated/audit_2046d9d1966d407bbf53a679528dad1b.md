### Title
Missing Declared Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the caller-supplied class hash has been declared before accepting it as the contract's new class. Any contract can replace its own class with an arbitrary, undeclared class hash, rendering itself permanently uncallable and freezing any funds it holds. The code even contains an explicit TODO acknowledging this missing check.

---

### Finding Description

In `syscall_impls.cairo`, the function `execute_replace_class` (lines 877–916) reads the new class hash directly from the `ReplaceClassRequest` struct and writes it into `contract_state_changes` with no validation that the hash exists in `contract_class_changes` (i.e., that it was ever declared):

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO at line 898 explicitly acknowledges the missing check. No lookup into `contract_class_changes` is performed to confirm the new class hash is a known, declared class.

**Analog to the external report:** In the `ProposedOwnable` bug, `acceptProposedOwner` accepted a new owner without verifying the caller was the proposed new owner — the wrong party could finalize a privileged state transition. Here, `execute_replace_class` accepts a new class hash without verifying it is a declared class — an invalid, non-existent class can finalize a privileged state transition (class replacement), with no recourse.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

After `execute_replace_class` commits an undeclared class hash into `contract_state_changes`, the contract's on-chain state permanently points to a class that does not exist in `contract_class_changes`. Every subsequent call to that contract will fail at class resolution time because the OS cannot find the class definition. Any ERC-20 balances, ETH, or other assets held by the contract become permanently inaccessible. There is no recovery path: the class hash is already committed to state, and the contract cannot call `replace_class` again because execution of its entry points is impossible.

---

### Likelihood Explanation

**High.** The `replace_class` syscall is a standard, publicly available syscall callable by any contract on itself. No privileged role is required. A malicious contract author, a buggy upgrade path, or a social-engineering attack on a contract's admin can trigger this with a single transaction. The missing check is already flagged as a known gap in the codebase (the TODO comment), confirming the developers are aware the guard is absent.

---

### Recommendation

Before updating `contract_state_changes`, perform a lookup into `contract_class_changes` to confirm the requested class hash has been declared (i.e., its compiled class hash is non-zero). This is the exact fix noted in the TODO:

```cairo
// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the guard already present in `execute_declare_transaction`, where `prev_value=0` enforces that a class may only be declared once and `assert_not_zero(compiled_class_hash)` enforces a valid compiled class hash.

---

### Proof of Concept

1. Deploy contract `C` that holds an ERC-20 balance (or any funds).
2. From within contract `C`, issue the `replace_class` syscall with an arbitrary undeclared felt value as the new class hash (e.g., `0xdeadbeef`).
3. The OS processes the syscall via `execute_replace_class` in `syscall_impls.cairo`. At line 898, the TODO check is absent — no lookup into `contract_class_changes` is performed.
4. `dict_update` at line 906 commits `class_hash=0xdeadbeef` into `contract_state_changes` for contract `C`.
5. The block is proven and finalized on L1.
6. Any subsequent invoke transaction targeting contract `C` fails: the OS reads `class_hash=0xdeadbeef` from state, looks it up in `contract_class_changes`, finds nothing, and cannot execute any entry point.
7. All funds in contract `C` are permanently frozen with no recovery path.

    range_check_ptr,
    syscall_ptr: felt*,

    alloc_locals;
    let request = cast(syscall_ptr + RequestHeader.SIZE, ReplaceClassRequest*);

        total_gas_cost=REPLACE_CLASS_GAS_COST, request_struct_size=ReplaceClassRequest.SIZE
    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    );

        key=contract_address,

    );

    return ();
}
