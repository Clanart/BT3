### Title
`batch_insert` Skips Duplicate-Key/Hash Validation Present in `insert`, Corrupting the MerkleBlob Cache and Tree Root — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces for every single-item insertion. When duplicate `KeyId` or `Hash` values are present in the batch, the `block_status_cache` is silently corrupted: the `key_to_index` and `leaf_hash_to_index` maps are overwritten while the orphaned leaf nodes remain in the raw blob. The resulting mismatch between the blob and its cache produces an incorrect tree root and enables forged Merkle inclusion/exclusion proofs.

### Finding Description

`MerkleBlob::insert` (lines 362–413) enforces two invariants before writing any node:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::batch_insert` (lines 570–657) calls `self.insert()` only for the first two items when the tree has ≤ 1 leaf. For every subsequent item it calls `self.insert_entry_to_blob()` directly:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // no dup check
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which does:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // silently overwrites
    self.leaf_hash_to_index.insert(leaf.hash, index);  // silently overwrites
}
```

A duplicate `KeyId` causes `HashMap::insert` to overwrite the previous `(key → old_index)` mapping. The old leaf node at `old_index` is still physically present in the blob but is now unreachable through the cache. The cache's `leaf_count()` (= `key_to_index.len()`) is therefore lower than the actual number of leaf blocks in the blob. The tree-building loop then constructs internal nodes and a root hash over only the newly inserted leaves, ignoring the orphaned duplicate, producing a root that does not commit to all stored data.

`check_integrity()` would detect this (`leaf_count != key_to_index_cache_length`), but it is only enabled in test builds (`check_integrity_on_drop: cfg!(test)`); production builds never call it automatically.

### Impact Explanation

A corrupted `MerkleBlob` root hash means:

- **Forged inclusion proofs**: a leaf that was silently overwritten can no longer be proven present, while the attacker-supplied replacement leaf can be proven present under the same key.
- **Corrupted tree root**: the committed root no longer reflects the true set of key-value pairs, breaking the integrity guarantee of the DataLayer.
- **Invalid exclusion proofs**: the orphaned leaf is invisible to the cache, so exclusion proofs for its key will incorrectly succeed.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`batch_insert` is a public method exposed through the Python bindings (`wheel/python/chia_rs/datalayer.pyi`) and is called from `crates/chia-datalayer/src/merkle/deltas.rs` during delta synchronization. An untrusted peer supplying a delta that contains duplicate `KeyId` values triggers the bug without any privileged access. Likelihood is **Medium**: the caller must supply duplicate keys, which is an abnormal but externally controllable input.

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the bulk path of `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insertion logic
}
```

Alternatively, deduplicate the input vector before processing and return an error if duplicates are detected.

### Proof of Concept

1. Create a `MerkleBlob` with at least 2 existing leaves (so `leaf_count() > 1`, bypassing the `insert()` path).
2. Call `batch_insert` with a vector containing two entries sharing the same `KeyId` (e.g., `KeyId(42)`).
3. Observe that `batch_insert` returns `Ok(())`.
4. Call `check_integrity()` — it returns `Err(IntegrityKeyToIndexCacheLength(...))` because the blob contains more leaf blocks than the cache tracks.
5. Generate a Merkle inclusion proof for `KeyId(42)` — it proves the second (overwriting) entry, while the first entry's data is silently lost and the root hash does not commit to it.

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
