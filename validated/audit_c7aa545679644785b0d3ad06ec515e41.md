Looking at the actual code, this is a real vulnerability. Here is the analysis:

**The race window is confirmed.** `initSubscriptions()` has zero synchronization — the grep for `sync.Mutex`, `sync.Once`, `sync.RWMutex` in `rpc/stream/rpc.go` returns no matches. The nil check at line 87 and the subsequent writes to `s.headerStream`/`s.logStream` are completely unguarded.

**The call path is public and concurrent.** `ServeHTTP` in `rpc/websockets.go` is called by Go's HTTP server, which spawns one goroutine per connection. ### Title
Unguarded TOCTOU Race in `initSubscriptions` Enables Duplicate CometBFT Subscriptions and Process Panic — (`rpc/stream/rpc.go`)

### Summary
`RPCStream.initSubscriptions()` uses an unsynchronized nil-check to guard one-time initialization. Two concurrent WebSocket callers can both observe `s.headerStream == nil` before either assigns it, causing two calls to `evtClient.Subscribe` under the same subscriber name `"ethermint-json-rpc"`, which panics the process on the second call.

### Finding Description

`initSubscriptions` performs a check-then-act on `s.headerStream` with no mutex or `sync.Once`: [1](#0-0) 

Both `HeaderStream()` and `LogStream()` call it directly with no external lock: [2](#0-1) 

The `RPCStream` struct holds a `sync.WaitGroup` but no mutex protecting `headerStream` or `logStream`: [3](#0-2) 

The WebSocket server calls `readLoop` (and thus `pubSubAPI` methods) in a separate goroutine per connection, with no shared lock around `HeaderStream()`/`LogStream()` calls: [4](#0-3) 

**Race sequence:**
1. Goroutine A calls `HeaderStream()` → `initSubscriptions()`, reads `s.headerStream == nil` (line 87), is preempted.
2. Goroutine B calls `HeaderStream()` → `initSubscriptions()`, also reads `s.headerStream == nil`, proceeds to assign `s.headerStream` and calls `evtClient.Subscribe(ctx, "ethermint-json-rpc", blockEvents, ...)` successfully.
3. Goroutine A resumes, also assigns `s.headerStream` (overwriting B's pointer) and calls `evtClient.Subscribe(ctx, "ethermint-json-rpc", blockEvents, ...)` a second time with the same subscriber name.
4. CometBFT's pubsub returns an error for the duplicate subscriber name → `panic(err)` at line 99 crashes the process. [5](#0-4) 

Even if CometBFT silently allows the duplicate, two `start()` goroutines are launched, both writing to `s.headerStream`/`s.logStream`, producing duplicate or out-of-order headers and logs for all WebSocket subscribers.

### Impact Explanation

A process panic in a goroutine without a recovery handler terminates the entire node process. This is a chain halt reachable from a public, unauthenticated WebSocket endpoint. Even without a panic, the data race on `s.headerStream` is undefined behavior in Go and can corrupt the stream pointer, causing all subsequent `newHeads`/`logs` subscribers to receive no events or garbage data.

### Likelihood Explanation

The attack requires only two simultaneous WebSocket connections sending `eth_subscribe('newHeads')` or `eth_subscribe('logs')` before the first `initSubscriptions` completes. This is trivially achievable under any real-world load, and also reproducible deterministically with a Go race-detector test. No privileges, keys, or special chain state are required.

### Recommendation

Replace the nil-check guard with `sync.Once`:

```go
type RPCStream struct {
    // ...
    initOnce sync.Once
    // ...
}

func (s *RPCStream) initSubscriptions() {
    s.initOnce.Do(func() {
        s.headerStream = NewStream[RPCHeader](headerStreamSegmentSize, headerStreamCapacity)
        s.logStream = NewStream[*ethtypes.Log](logStreamSegmentSize, logStreamCapacity)
        // ... Subscribe calls ...
        go s.start(&s.wg, chBlocks, chLogs)
    })
}
```

`sync.Once` guarantees exactly one execution regardless of concurrent callers and provides the necessary memory barrier so all goroutines see the fully initialized fields.
