### Title
OS `check_proof_facts` Hard-Asserts Only `PROOF_VERSION_V1` While Blockifier Accepts `PROOF_VERSION_V0` — (`crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

The Cairo OS function `check_proof_facts` contains a comment acknowledging that proof version V0 is a valid legacy version, but the assertion immediately below it only allows V1. Simultaneously, the blockifier's Rust `validate_proof_facts` checks against a per-protocol-version allowlist (`allowed_proof_versions`) that, for protocol version 0.14.2, contains only `PROOF_VERSION_V0`. This creates a direct divergence: a transaction carrying V0 proof facts passes blockifier pre-validation and is admitted into a block, but the OS rejects it during execution/proving, producing a wrong execution result for accepted input.

---

### Finding Description

**Root cause — OS Cairo code:** [1](#0-0) 

```cairo
// Proof version may be V0 (legacy) or V1 (current).
with_attr error_message("Unsupported proof version") {
    assert proof_header.proof_version = PROOF_VERSION_V1;
}
```

The comment explicitly acknowledges V0 as a valid legacy version, but the `assert` only allows V1. Any transaction whose `proof_facts[0]` equals `PROOF_VERSION_V0` (`0x50524f4f4630`) will cause the OS to abort with "Unsupported proof version".

**Blockifier pre-validation — Rust code:** [2](#0-1) 

```rust
if !os_constants.allowed_proof_versions.contains(&snos_proof_facts.proof_version.as_felt()) {
    return Err(TransactionPreValidationError::InvalidProofFacts(format!(
        "Proof version {} is not allowed under this protocol version.",
        snos_proof_facts.proof_version
    )));
}
```

This check is correct in structure — it consults the per-protocol allowlist. The problem is what that allowlist contains.

**Protocol 0.14.2 versioned constants:** [3](#0-2) 

```json
"allowed_proof_versions": [
    "0x50524f4f4630"
]
```

`0x50524f4f4630` is `PROOF_VERSION_V0`, as confirmed by: [4](#0-3) 

So for protocol 0.14.2, the blockifier's `validate_proof_facts` accepts V0 proof facts (V0 is in the allowlist), but the OS's `check_proof_facts` rejects them (only V1 is asserted). The two layers disagree on what constitutes a valid proof version.

**Where `check_proof_facts` is called in the OS execution path:** [5](#0-4) 

The OS calls `check_proof_facts` for every invoke transaction that carries proof facts, directly inside the block execution loop.

**`ProofVersion` enum and felt constants:** [6](#0-5) 

Both V0 and V1 are recognized by the Rust parser (`ProofFactsVariant::try_from`), so a V0 transaction is parsed successfully and reaches the allowlist check — where it passes for 0.14.2 — before being handed to the OS, which then rejects it.

---

### Impact Explanation

A transaction with `proof_facts[0] = PROOF_VERSION_V0` submitted when client-side proving is enabled:

1. Passes `StatelessTransactionValidator::validate` (no proof-version check there).
2. Passes `validate_proof_facts` in the blockifier (V0 is in `allowed_proof_versions` for 0.14.2).
3. Is included in a block by the batcher.
4. Causes the OS's `check_proof_facts` to abort with "Unsupported proof version" during block execution/proving.

The result is a block that the blockifier considers valid but the OS rejects — a wrong execution result for accepted input. This matches the **Critical** impact: *Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.* [7](#0-6) 

---

### Likelihood Explanation

- Client-side proving must be enabled (`allow_client_side_proving = true`).
- The attacker only needs to submit a well-formed invoke V3 transaction with `proof_facts[0] = PROOF_VERSION_V0` and a valid V0-shaped proof facts body.
- No privileged access is required; any user can submit such a transaction via the public RPC.
- The divergence is structural and deterministic: every such transaction will be accepted by the blockifier and rejected by the OS.

---

### Recommendation

The OS assertion must be updated to mirror the blockifier's allowlist check. Instead of hard-coding a single version, the OS should accept any version present in the protocol-version-gated allowlist. Concretely, the Cairo code should be changed from:

```cairo
// Proof version may be V0 (legacy) or V1 (current).
with_attr error_message("Unsupported proof version") {
    assert proof_header.proof_version = PROOF_VERSION_V1;
}
```

to a check that accepts both `PROOF_VERSION_V0` and `PROOF_VERSION_V1` (or whichever versions are in the active protocol's allowlist), consistent with the comment that already acknowledges V0 as a valid legacy version. Alternatively, the blockifier's `allowed_proof_versions` for 0.14.2 should be updated to contain only V1 if V0 is no longer intended to be accepted by the OS.

---

### Proof of Concept

1. Enable client-side proving (`allow_client_side_proving = true`).
2. Construct an invoke V3 transaction with:
   - `proof_facts[0] = 0x50524f4f4630` (`PROOF_VERSION_V0`)
   - `proof_facts[1] = VIRTUAL_SNOS`
   - Remaining fields: valid SNOS proof facts body (valid block number, block hash, config hash, program hash).
3. Submit via RPC. The gateway's `StatelessTransactionValidator` passes it. The blockifier's `validate_proof_facts` passes it (V0 is in `allowed_proof_versions` for 0.14.2).
4. The transaction is included in a block.
5. The OS's `check_proof_facts` asserts `proof_header.proof_version = PROOF_VERSION_V1`, which fails for V0, aborting block execution with "Unsupported proof version". [4](#0-3) [3](#0-2) [1](#0-0) [2](#0-1)

pub enum ProofVersion {

impl ProofVersion {
    /// Felt (Cairo short-string) representation written into proof facts.
    pub const fn as_felt(self) -> Felt {
        match self {
            ProofVersion::V0 => PROOF_VERSION_V0,
            ProofVersion::V1 => PROOF_VERSION_V1,
        }
    }

    /// Human-readable short-string label (matches the Cairo constant value).
    pub const fn as_str(self) -> &'static str {
        match self {
            ProofVersion::V0 => "PROOF0",
            ProofVersion::V1 => "PROOF1",
        }
    }
}

impl TryFrom<Felt> for ProofVersion {
    type Error = ();
    fn try_from(value: Felt) -> Result<Self, Self::Error> {

            Ok(ProofVersion::V0)
        } else if value == PROOF_VERSION_V1 {
            Ok(ProofVersion::V1)
        } else {
            Err(())
        }
    }
