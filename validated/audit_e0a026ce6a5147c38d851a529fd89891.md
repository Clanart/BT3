### Title
Permanent Fund Lock When Transferred Amount Normalizes to Zero in `sign_transfer` — (`File: near/omni-bridge/src/lib.rs`)

### Summary

`init_transfer` accepts and locks user tokens for any amount where `fee < amount`, but `sign_transfer` — the only path to finalize the transfer on the destination chain — enforces a separate check that `normalize_amount(amount_without_fee) > 0`. When a user transfers an amount smaller than the decimal-normalization threshold (e.g., less than `10^(origin_decimals − decimals)` units), `sign_transfer` always reverts with `InvalidAmountToTransfer`, there is no cancel or refund path, and the tokens are permanently locked in the bridge contract.

### Finding Description

**Step 1 — `init_transfer` accepts the transfer and locks tokens.**

`init_transfer_internal` stores the `TransferMessage` in `pending_transfers`, calls `lock_tokens_if_needed`, and returns `U128(0)` to the NEP-141 `ft_transfer_call`, confirming the tokens are consumed by the bridge. [1](#0-0) 

The only guard at this stage is `fee < amount`: [2](#0-1) 

There is no check that `normalize_amount(amount_without_fee) > 0`.

**Step 2 — `sign_transfer` is the sole path to finalize the transfer.**

A trusted relayer must call `sign_transfer` to obtain an MPC signature. Inside, the contract normalizes the amount: [3](#0-2) 

`normalize_amount` uses floor division: [4](#0-3) 

For a NEAR-native token with `origin_decimals = 24` bridging to an EVM chain with `decimals = 18`, `diff_decimals = 6`, so any `amount_without_fee < 1_000_000` normalizes to `0`. The `require!` then panics unconditionally, and the call reverts before the MPC signer is ever contacted.

**Step 3 — No recovery path exists.**

`pending_transfers` entries are removed only via:
- `remove_transfer_message` in `claim_fee_callback` — requires a `FinTransfer` proof from the destination chain, which can only exist after a successful `sign_transfer` + destination-chain `finTransfer`. Impossible here.
- `sign_transfer_callback` when `fee.is_zero()` — the callback is never reached because `sign_transfer` panics before calling the MPC signer.
- `remove_transfer_message_without_refund` in `init_transfer_internal` on storage failure — already past this point. [5](#0-4) 

There is no `cancel_transfer` or user-accessible refund function. The `transfer_token_as_dao` escape hatch exists but requires DAO-role access, which is a privileged path outside the user's control. [6](#0-5) 

### Impact Explanation

Any NEAR-native token transferred with `amount_without_fee < 10^(origin_decimals − decimals)` is permanently locked in the bridge contract. Both the principal and any fee component are irrecoverable without DAO intervention. This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge flows.**

### Likelihood Explanation

Low. The user must transfer an amount below the decimal-normalization threshold (e.g., less than `0.000001` of a 24-decimal NEAR token bridging to an 18-decimal EVM chain). This can happen accidentally during testing, with dust amounts, or via a malicious actor deliberately locking another user's small transfer. The `update_transfer_fee` function cannot rescue the transfer because increasing the fee only reduces `amount_without_fee` further, keeping the normalized value at zero. [7](#0-6) 

### Recommendation

Add a normalization check inside `init_transfer` (or `init_transfer_internal`) before accepting the transfer:

```rust
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    token_id.clone(),
).near_expect(BridgeError::FailedToGetTokenAddress);
let decimals = self.token_decimals.get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);
let normalized = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

This mirrors the existing guard in `sign_transfer` and ensures the bridge never accepts a transfer it cannot later finalize.

### Proof of Concept

1. A NEAR-native token is registered with `origin_decimals = 24`, `decimals = 18` (EVM destination). The normalization divisor is `10^6 = 1_000_000`.
2. User calls `ft_transfer_call` on the token contract, transferring `amount = 500_000` (with `fee = 0`) to the bridge. `fee < amount` passes. `init_transfer_internal` stores the `TransferMessage` in `pending_transfers`, calls `lock_tokens_if_needed(ChainKind::Eth, token_id, 500_000)`, and returns `U128(0)` — tokens are consumed.
3. Relayer calls `sign_transfer(transfer_id, fee_recipient, None)`. Inside: `amount_without_fee = 500_000`, `normalize_amount(500_000, {decimals:18, origin_decimals:24}) = 500_000 / 1_000_000 = 0`. The `require!(amount_to_transfer > 0)` panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
4. The call reverts. `pending_transfers` still contains the entry. `locked_tokens` still holds `500_000`. No MPC signature is produced.
5. Every subsequent `sign_transfer` call for this `transfer_id` produces the same panic. `claim_fee` cannot be called (no destination-chain proof). The `500_000` units are permanently locked. [3](#0-2) [4](#0-3) [8](#0-7)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
