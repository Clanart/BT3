### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Signature Replay — (`crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` computes a transaction hash that omits a nonce from the signed payload. Because the hash is identical across repeated invocations with the same parameters, any valid signature produced for a `meta_tx_v0` call can be replayed indefinitely by an attacker who observed it on-chain, causing the authorized action to execute multiple times.

---

### Finding Description

`compute_meta_tx_v0_hash` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,
        additional_data=cast(0, felt*),   // ← no nonce
    );
    return tx_hash;
}
``` [1](#0-0) 

Contrast this with `compute_l1_handler_transaction_hash`, which correctly passes the nonce as `additional_data`:

```cairo
deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
    ...
    additional_data_size=1,
    additional_data=&nonce,   // ← nonce included
);
``` [2](#0-1) 

`deprecated_get_transaction_hash` feeds `additional_data` directly into the Pedersen hash chain:

```cairo
let (hash_state_ptr) = hash_update(
    hash_state_ptr=hash_state_ptr, data_ptr=additional_data, data_length=additional_data_size
);
``` [3](#0-2) 

When `additional_data_size=0`, the hash is a pure function of `(prefix, version=0, contract_address, selector, calldata_hash, max_fee=0, chain_id)`. None of these values change between executions of the same logical call, so the hash — and therefore the valid signature — is identical every time.

The `check_and_increment_nonce` guard that protects top-level account transactions is never invoked for `meta_tx_v0`: [4](#0-3) 

That call exists only inside `execute_invoke_function_transaction` and `execute_declare_transaction`, not in any `meta_tx_v0` syscall path.

---

### Impact Explanation

An account contract that uses `meta_tx_v0` to authorize a token transfer or any state-mutating call signs a hash that carries no replay protection at the OS level. An attacker who observes the (signature, calldata) pair on-chain can re-submit the same syscall with the same signature in a subsequent transaction. The OS will compute the identical hash, the account's `__validate__` will accept the same signature, and the authorized action executes again. If the action is a token transfer, the attacker drains the victim's balance in repeated rounds until it is exhausted.

**Impact: Critical — Direct loss of funds.**

---

### Likelihood Explanation

The attack requires only:
1. Observing a `meta_tx_v0` transaction on-chain (public information).
2. Submitting a new transaction that re-invokes the same syscall with the same parameters and signature.

No privileged access, leaked keys, or social engineering is required. Any unprivileged network participant can execute this once a single `meta_tx_v0` transaction is visible on-chain.

---

### Recommendation

Include a per-account nonce in the `meta_tx_v0` hash, mirroring the pattern used by `compute_l1_handler_transaction_hash`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,          // ← add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=1,   // ← include nonce
        additional_data=&nonce,
    );
    return tx_hash;
}
```

The OS must also enforce nonce increment for `meta_tx_v0` calls, analogous to `check_and_increment_nonce` used for standard account transactions.

---

### Proof of Concept

1. **Victim** calls a contract via `meta_tx_v0` syscall with `(contract_address=C, selector=transfer, calldata=[attacker, 100_tokens])` and provides signature `(r, s)` over the hash `H = compute_meta_tx_v0_hash(C, transfer, [attacker, 100], chain_id)`.

2. **Attacker** observes `(r, s)` and the calldata on-chain.

3. **Attacker** submits a new transaction that triggers the same `meta_tx_v0` syscall with identical parameters and the same `(r, s)`.

4. The OS recomputes `H' = compute_meta_tx_v0_hash(C, transfer, [attacker, 100], chain_id)`. Since `additional_data_size=0`, `H' == H`.

5. The account contract's `__validate__` verifies `(r, s)` against `H'` — it passes.

6. The transfer executes again. Steps 3–6 repeat until the victim's balance is zero.
