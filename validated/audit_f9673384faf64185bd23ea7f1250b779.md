Audit Report

## Title
Missing `deadline` parameter in `addLiquidityWeighted` and `addLiquidityExactShares` allows stale-price liquidity deposits — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
All four public `addLiquidity*` entry points in `MetricOmmPoolLiquidityAdder` accept no `deadline` parameter and perform no time-of-execution check, while every swap entry point in `MetricOmmSimpleRouter` calls `_checkDeadline` as its first action. A transaction that sits in the mempool and executes after a significant oracle price move will deposit at a composition the LP never consented to, with no on-chain mechanism to prevent it.

## Finding Description
`MetricOmmPoolLiquidityAdder` exposes four public entry points, none of which accept or enforce a deadline:

- `addLiquidityExactShares(address,address,uint80,LiquidityDelta,uint256,uint256,bytes)` [1](#0-0) 
- `addLiquidityExactShares(address,uint80,LiquidityDelta,uint256,uint256,bytes)` [2](#0-1) 
- `addLiquidityWeighted(address,address,uint80,LiquidityDelta,uint256,uint256,int8,uint104,int8,uint104,bytes)` [3](#0-2) 
- `addLiquidityWeighted(address,uint80,LiquidityDelta,uint256,uint256,int8,uint104,int8,uint104,bytes)` [4](#0-3) 

By contrast, every swap entry point in `MetricOmmSimpleRouter` calls `_checkDeadline` as its first statement: [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

The `_checkDeadline` helper exists in `MetricOmmSwapRouterBase` but `MetricOmmPoolLiquidityAdder` does not inherit from it and has no equivalent guard: [9](#0-8) 

The `addLiquidityWeighted` path is the most dangerous. It performs a **probe** call to determine `need0`/`need1` at the current oracle price, then immediately scales shares and executes the real deposit: [10](#0-9) 

Both the probe and the paying add execute within the same transaction, so there is no intra-transaction gap. The vulnerability is that the **entire transaction** can be withheld by a validator or sit in a congested mempool. When it finally lands, the oracle price may have moved substantially. The probe then reads the new price, scales shares accordingly, and the deposit settles at a composition the user never intended.

The only existing guards are:

1. **`maxAmountToken0`/`maxAmountToken1` caps** — bound the absolute quantity spent but do not bound the time of execution or the deposit ratio: [11](#0-10) 

2. **`_validateBinAndBinPosition`** — provides partial protection for `addLiquidityWeighted` only when the oracle cursor has moved outside the user-supplied bin bounds, but does not catch oracle price drift within the same bin, and is entirely absent from `addLiquidityExactShares`: [12](#0-11) 

Neither guard prevents execution at an arbitrarily stale oracle price when the cursor has not crossed a bin boundary.

## Impact Explanation
An LP submits `addLiquidityWeighted` intending to deposit at the current market price. The transaction is delayed (network congestion or validator withholding). When it executes, the oracle price has moved significantly. The probe computes `need0`/`need1` at the new price, shares are scaled to that composition, and the deposit settles at a price far from the user's intent. The LP immediately holds a position with embedded impermanent loss relative to the price at submission time. Because the `maxAmount` caps are set by the user to accommodate normal slippage, they do not prevent execution at a price that is minutes or hours stale. This constitutes direct loss of LP principal through adverse composition at stale entry price — matching the allowed impact of "Critical/High/Medium direct loss of user principal or owed LP assets."

## Likelihood Explanation
Any pending transaction in a public mempool is subject to validator reordering or simple delay. On L2s with sequencer-controlled ordering, deliberate delay is trivial. No special privilege is required; the trigger is a normal, valid user transaction. A user who submits during a period of price volatility is at elevated risk. The condition is repeatable and requires no attacker capital.

## Recommendation
Add a `deadline` parameter to all four `addLiquidity*` entry points and validate it at the top of each function, mirroring the pattern already used in `MetricOmmSimpleRouter`. The `_checkDeadline` helper already exists in `MetricOmmSwapRouterBase`; extract it to a shared base or utility so both contracts can use it.

```solidity
// In MetricOmmPoolLiquidityAdder
function addLiquidityWeighted(
    address pool,
    address owner,
    uint80 salt,
    LiquidityDelta calldata weightDeltas,
    uint256 maxAmountToken0,
    uint256 maxAmountToken1,
    int8 minimalCurBin,
    uint104 minimalPosition,
    int8 maximalCurBin,
    uint104 maximalPosition,
    uint256 deadline,          // <-- add
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _checkDeadline(deadline);  // <-- add
    ...
}
```

Apply the same change to both `addLiquidityWeighted` overloads and both `addLiquidityExactShares` overloads.

## Proof of Concept
1. Alice calls `addLiquidityWeighted` with `maxAmountToken0 = 1000e18`, `maxAmountToken1 = 1000e18`, targeting the current oracle price of 1 token0 = 1 token1.
2. The transaction enters the mempool but is not included immediately (network congestion or sequencer delay).
3. The oracle price moves to 2 token0 = 1 token1 (token0 halves in value).
4. The transaction is included. The probe reads the new price: `need0 = 1000e18`, `need1 = 500e18`.
5. Scaling factor = `min(1000/1000, 1000/500)` = 1. Alice deposits 1000 token0 and 500 token1.
6. Alice intended a 50/50 split at 1:1 but instead deposited a 2:1 ratio of token0:token1 at the new price. Her position is now skewed toward the depreciating asset. Any reversion of the price move crystallizes impermanent loss at this stale entry point.
7. A Foundry test can reproduce this by: deploying the pool and liquidity adder, setting an initial oracle price, submitting `addLiquidityWeighted`, using `vm.warp` to advance time, updating the oracle price to simulate the move, then executing the transaction and asserting the deposit composition differs from the user's intent with no revert.

  function _validateBinAndBinPosition(
    address pool,
    int8 minimalCurBin,
    uint104 minimalPosition,
    int8 maximalCurBin,
    uint104 maximalPosition
  ) internal view {
    if (minimalCurBin > maximalCurBin) {
      revert CursorOutOfBounds(0, 0, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }

    (, int8 curBinIdx, uint104 curPosInBin,,,) = PoolStateLibrary._slot0(pool);

    int256 curBin = curBinIdx;
    if (curBin < minimalCurBin || curBin > maximalCurBin) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == minimalCurBin && curPosInBin < minimalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == maximalCurBin && curPosInBin > maximalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
  }
