### Title
Intermediate State During Token Deployment Causes Permanent Loss of User Funds via `fin_transfer` - (File: `near/omni-bridge/src/lib.rs`)

### Summary
In `deploy_token_internal`, all token registry mappings are written to contract state **before** the external deployer call completes. This creates an intermediate state window where `fin_transfer` can be called for the token, permanently consuming the transfer proof and marking the transfer as finalized, while the actual token contract does not yet exist and the `mint` call fails silently — permanently locking user funds.

### Finding Description

`deploy_token_internal` writes all token mappings atomically into state before the cross-contract deployer call: [1](#0-0) 

Specifically, `add_token` populates `token_id_to_address`, `token_address_to_id`, and `token_decimals`: [2](#0-1) 

Then the external deployer call is made asynchronously: [3](#0-2) 

Only in `deploy_token_by_deployer_callback` — which executes in a **later receipt** — are the mappings removed on failure: [4](#0-3) 

Between these two receipts, the token appears fully registered in state but the token contract does not yet exist on-chain.

During this window, a relayer can submit `fin_transfer` for the same token. `fin_transfer_callback` successfully reads `token_decimals` (already written): [5](#0-4) 

`process_fin_transfer_to_near` then **permanently marks the transfer as finalized** via `add_fin_transfer` before attempting to send tokens: [6](#0-5) 

The `send_tokens` call (which calls `mint` on the non-existent token contract) fails. However, `fin_transfer_send_tokens_callback` only reverts finalization when `is_refund_required` is true, which only occurs for `ft_transfer_call` flows (i.e., when `msg` is non-empty): [7](#0-6) 

For standard transfers (no message), `is_ft_transfer_call = false`, so `is_refund_required` returns `false`, and `remove_fin_transfer` is **never called**: [8](#0-7) 

The transfer ID is permanently consumed in `finalised_transfers`. The user cannot re-submit the proof. Tokens are never minted.

### Impact Explanation

The transfer is irrecoverably finalized on NEAR without the recipient ever receiving tokens. The user's funds on the source chain are already locked/burned (the `InitTransfer` event was emitted), and the NEAR-side proof is consumed. This constitutes **permanent, irrecoverable loss of bridged user funds** — matching the Critical impact class: "Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds."

### Likelihood Explanation

The intermediate state window spans at least one NEAR block (the time for the deployer's `deploy_token` cross-contract call to complete and be confirmed). During initial deployment of a new token, it is realistic that:
1. A user initiates a transfer on the source chain before the NEAR token is deployed.
2. A relayer, observing the valid source-chain proof, submits `fin_transfer` while `deploy_token` is in-flight.

This is a normal operational scenario, not a contrived attack. No malicious behavior is required from any party.

### Recommendation

Defer writing token registry mappings (`add_token`, `deployed_tokens.insert`, `deployed_tokens_v2.insert`) until **after** the deployer call succeeds, i.e., move all state writes into the success branch of `deploy_token_by_deployer_callback`. This makes token registration atomic with respect to the deployer call and eliminates the intermediate state window.

### Proof of Concept

1. Token `T` (EVM address `0xABC`) does not yet exist on NEAR.
2. Relayer A calls `deploy_token` with a valid `LogMetadata` proof for `0xABC`.
3. `deploy_token_internal` runs: writes `token_address_to_id[0xABC] = t.deployer`, `token_decimals[0xABC] = {18, 18}`, `deployed_tokens = {t.deployer}`. Schedules `ext_deployer.deploy_token(t.deployer, ...)`.
4. Before `deploy_token_by_deployer_callback` executes (next receipt), Relayer B calls `fin_transfer` with a valid `InitTransfer` proof for a user who sent 100 T from EVM.
5. `fin_transfer_callback`: `token_decimals.get(0xABC)` succeeds → proceeds.
6. `process_fin_transfer_to_near`: `add_fin_transfer(transfer_id)` → transfer permanently finalized.
7. `send_tokens(t.deployer, recipient, 100)` → calls `mint` on `t.deployer` which does not exist → receipt fails.
8. `fin_transfer_send_tokens_callback` with `is_ft_transfer_call = false` → `is_refund_required = false` → finalization NOT reverted.
9. `deploy_token_by_deployer_callback` eventually runs (success or failure) — irrelevant, transfer already consumed.
10. User's 100 T are permanently lost: locked on EVM, proof consumed on NEAR, tokens never minted.

    fn add_token(
        &mut self,
        token_id: &AccountId,
        token_address: &OmniAddress,
        decimals: u8,
        origin_decimals: u8,

                    &Decimals {
                        decimals,
                        origin_decimals,
