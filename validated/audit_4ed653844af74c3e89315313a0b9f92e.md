Let me check the NEAR side to understand what the MPC actually signs, and verify the Borsh encoding library used on EVM.

### Title
Cross-Chain Signature Replay on `deploy_token`: EVM MPC Signature Accepted on Solana Due to Identical Payload Encoding and Absent Chain-ID Binding — (`solana/programs/bridge_token_factory/src/state/message/deploy_token.rs` / `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`DeployTokenPayload::serialize_for_near` on Solana and `OmniBridge.deployToken`'s inline Borsh encoder on EVM produce byte-for-byte identical byte strings for the same `(token, name, symbol, decimals)` tuple. Both chains hash that string with raw keccak256 and verify against the same MPC-derived key. Neither encoding includes a destination chain ID or any domain separator. An unprivileged relayer holding a valid MPC ECDSA signature obtained from an EVM `deployToken` call can replay it verbatim (with a trivial recovery-byte adjustment) against Solana's `deploy_token`, causing an unauthorized canonical wrapped SPL mint to be created for that token without any Solana-specific MPC signing event.

---

### Finding Description

**EVM serialization** (`OmniBridge.sol` lines 142–148):

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),  // = 0x01
    Borsh.encodeString(metadata.token),               // [LE-u32 len][utf8]
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) revert InvalidSignature();
``` [1](#0-0) 

`BridgeTypes.PayloadType.Metadata` is the second enum variant (index 1). `Borsh.encodeString` emits a 4-byte little-endian length prefix followed by the UTF-8 bytes. [2](#0-1) [3](#0-2) 

**Solana serialization** (`deploy_token.rs` lines 19–26):

```rust
fn serialize_for_near(&self, _params: Self::AdditionalParams) -> Result<Vec<u8>> {
    let mut writer = BufWriter::new(Vec::with_capacity(DEFAULT_SERIALIZER_CAPACITY));
    IncomingMessageType::Metadata.serialize(&mut writer)?;  // = 0x01
    self.serialize(&mut writer)?;  // borsh({token,name,symbol,decimals})
    ...
}
``` [4](#0-3) 

`IncomingMessageType::Metadata` is also the second variant (index 1). Borsh serialization of `{token: String, name: String, symbol: String, decimals: u8}` emits exactly `[LE-u32 len][utf8]` per string, then one byte for `decimals` — identical to the EVM encoding. [5](#0-4) [6](#0-5) 

**Solana signature verification** (`mod.rs` lines 24–47):

```rust
let serialized = self.payload.serialize_for_near(params)?;
let hash = keccak::hash(&serialized);          // raw keccak256, no prefix
let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)?;
require!(signer.0 == *derived_near_bridge_address, ...);
``` [7](#0-6) 

Both sides use **raw keccak256** with no Ethereum signed-message prefix and no domain separator. The only structural difference is the recovery byte convention: EVM uses `v ∈ {27, 28}` while Solana uses `recovery_id ∈ {0, 1}`. An attacker trivially converts by subtracting 27 from the last byte of the 65-byte signature. The `r` and `s` components are unchanged.

**No chain ID anywhere in the payload.** The EVM encoding contains `[0x01][token][name][symbol][decimals]`; the Solana encoding contains exactly the same bytes. There is no field that distinguishes "this signature authorizes deployment on EVM chain X" from "this signature authorizes deployment on Solana."

---

### Impact Explanation

An attacker who observes a successful EVM `deployToken` transaction for token `T` (name `N`, symbol `S`, decimals `D`) extracts the 65-byte MPC signature from calldata, adjusts the recovery byte (`v - 27`), and submits a `deploy_token` instruction to Solana with the same `(T, N, S, D)` fields and the adjusted signature. Solana's `verify_signature` passes because the hash and the signing key are identical. The `DeployToken` accounts context then calls `init` on a new SPL mint PDA seeded by `[WRAPPED_MINT_SEED, hash(T)]`, creating a canonical wrapped mint for token `T` on Solana. [8](#0-7) 

This violates the invariant that each chain's token deployment must be authorized by a chain-specific MPC signing event. The unauthorized mint is indistinguishable from a legitimately deployed one; any subsequent `finalize_transfer` targeting that token on Solana will mint against it.

---

### Likelihood Explanation

The precondition is simply that EVM `deployToken` has been called for any token — a normal, observable on-chain event. The attacker needs no privileged access, no leaked key, and no colluding MPC signers. The only manipulation required is a one-byte arithmetic adjustment to the signature's recovery parameter. The attack is fully local-testable with a unit test using a known ECDSA key.

---

### Recommendation

Bind the signed payload to the destination chain by including a chain ID field in the serialized message before it is hashed. For example, prepend `SOLANA_OMNI_BRIDGE_CHAIN_ID` in `serialize_for_near` on the Solana side and include the corresponding chain ID in the EVM Borsh encoding. This ensures a signature issued for chain A cannot be accepted on chain B.

```rust
// In DeployTokenPayload::serialize_for_near
IncomingMessageType::Metadata.serialize(&mut writer)?;
SOLANA_OMNI_BRIDGE_CHAIN_ID.serialize(&mut writer)?;  // add this
self.serialize(&mut writer)?;
```

```solidity
// In OmniBridge.deployToken
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),   // add this
    Borsh.encodeString(metadata.token),
    ...
);
``` [9](#0-8)

### Proof of Concept

1. Deploy a test EVM `OmniBridge` with a known ECDSA key as `nearBridgeDerivedAddress`.
2. Construct the EVM Borsh payload for token `"usdc.near"`, name `"USD Coin"`, symbol `"USDC"`, decimals `6`.
3. Sign `keccak256(payload)` with the test key → get `(r, s, v=27)`.
4. On Solana, construct a `SignedPayload<DeployTokenPayload>` with the same fields and signature `[r || s || (v-27)]`.
5. Call `deploy_token`. Assert the transaction succeeds and a wrapped SPL mint PDA for `"usdc.near"` is created — using only the EVM-issued signature, with no Solana-specific MPC signing event.

```rust
    pub fn verify_signature(
        &self,
        params: P::AdditionalParams,
        derived_near_bridge_address: &[u8; 64],
    ) -> Result<()> {
        let serialized = self.payload.serialize_for_near(params)?;
        let hash = keccak::hash(&serialized);

        let signature_bytes = &self.signature[0..64];

        let signature = libsecp256k1::Signature::parse_standard_slice(signature_bytes)
            .map_err(|_| ProgramError::InvalidArgument)?;
        require!(!signature.s.is_high(), ErrorCode::MalleableSignature);

        let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)
            .map_err(|_| error!(ErrorCode::SignatureVerificationFailed))?;

        require!(
            signer.0 == *derived_near_bridge_address,
            ErrorCode::SignatureVerificationFailed
        );

        Ok(())
    }
