### Title
Hardcoded `failure_flag=0` in `execute_deploy` Silently Swallows Constructor Failures, Enabling Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall implementation in the StarkNet OS unconditionally writes `failure_flag=0` (success) to the response header regardless of whether the deployed contract's constructor actually succeeded or reverted. This means any calling contract that relies on the `deploy` syscall's failure flag to gate fund transfers or state initialization will be deceived into treating a failed deployment as successful, permanently freezing any funds sent to the uninitialized contract.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_deploy` function computes the contract address, calls `deploy_contract` (which internally runs the constructor via `execute_entry_point` and handles reverts via the revert log), and then writes the response. The critical flaw is at line 539:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `failure_flag` is hardcoded to `0` (success). The `deploy_contract` call returns only `(retdata_size, retdata)` — no `is_reverted` flag — so there is no mechanism to propagate a constructor revert to the response.

Compare this with the correct pattern used in `contract_call_helper` for `call_contract`:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

`call_contract` correctly propagates `is_reverted` into `failure_flag`. `execute_deploy` does not. The TODO comment at line 538 explicitly acknowledges this is incomplete.

The vulnerability class maps directly to the external report: multiple external calls are chained (constructor execution → response write), and a failure in the inner call (constructor revert) is silently swallowed rather than propagated, causing the outer caller to proceed on a false success signal.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any factory contract, DEX, or protocol that:
1. Calls `deploy` to deploy a user-specified or conditionally-failing contract, and
2. Sends funds to the returned `contract_address` upon receiving a success response

will transfer funds to a contract whose constructor reverted and whose storage is therefore uninitialized. The contract address is computed and registered before the constructor runs; after a constructor revert, the revert log rolls back storage writes but the address slot may remain. Funds sent to this address are permanently inaccessible because the contract's initialization logic never completed.

Additionally, the OS proves this incorrect state transition as valid (the proof attests `failure_flag=0`), meaning the on-chain verified state diverges from the true execution outcome. This constitutes a provably incorrect state commitment.

---

### Likelihood Explanation

**Medium-High.** The attacker path requires no privileged access:

1. An attacker declares a contract class whose constructor reverts unconditionally (or conditionally, e.g., based on calldata).
2. The attacker interacts with any factory or protocol contract that calls the `deploy` syscall with attacker-influenced parameters (class hash, constructor calldata).
3. The constructor reverts; the OS writes `failure_flag=0`.
4. The factory contract reads `failure_flag=0`, concludes success, and sends funds to the returned address.
5. Funds are permanently frozen.

Factory patterns are common in DeFi (e.g., pair creation, vault deployment). The attacker only needs to find one such contract that trusts the `deploy` syscall's failure flag.

---

### Recommendation

1. Modify `deploy_contract` to return an `is_reverted: felt` flag alongside `(retdata_size, retdata)`.
2. In `execute_deploy`, replace the hardcoded `failure_flag=0` with the actual `is_reverted` value returned by `deploy_contract`, mirroring the pattern in `contract_call_helper`.
3. When `is_reverted=1`, append `ERROR_ENTRY_POINT_FAILED` to the retdata (consistent with `contract_call_helper` at line 421).

Victim factory contract:
