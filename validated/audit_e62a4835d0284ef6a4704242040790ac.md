### Title
Silent u128 Overflow in `denormalize_amount` Corrupts Bridged Token Amounts - (File: near/omni-bridge/src/lib.rs)

### Summary

`denormalize_amount` performs an unchecked `u128` multiplication. In Rust release mode (used by NEAR contracts), integer overflow wraps silently rather than panicking. When a user initiates a transfer from EVM with an amount large enough to overflow after decimal scaling, the NEAR bridge stores and mints a silently-wrapped (drastically smaller) amount, while the user's EVM tokens are permanently burned. This is a direct analog to the MIPS ADD/SUB overflow class: arithmetic that should error on overflow instead silently produces an incorrect result, corrupting bridge accounting.

---

### Finding Description

`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← unchecked multiplication
}
``` [1](#0-0) 

The multiplication uses Rust's default `*` operator on `u128`. In release builds, this wraps modulo 2¹²⁸ without any trap or error.

This function is called in `fin_transfer_callback` to compute the amount to mint/transfer to the recipient on NEAR:

```rust
let transfer_message = TransferMessage {
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    fee: Self::denormalize_fee(&init_transfer.fee, decimals),
    ...
};
``` [2](#0-1) 

The `init_transfer.amount.0` value comes directly from the EVM `InitTransfer` event, which carries the raw `uint128` amount the user locked/burned on EVM — **no normalization is applied on the EVM side before emitting the event**:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,   // raw uint128, no scaling
    fee, nativeFee, recipient, message
);
``` [3](#0-2) 

The NEAR-side prover parses this amount directly into `InitTransferMessage.amount`:

```rust
amount: near_sdk::json_types::U128(event.data.amount),
``` [4](#0-3) 

**Overflow condition**: For a NEAR-native token with `origin_decimals = 24` bridged to EVM with `decimals = 18`, `diff_decimals = 6`. Overflow occurs when:

```
evm_amount > u128::MAX / 10^6  ≈  3.4 × 10³²
```

`uint128` on EVM can hold values up to `≈ 3.4 × 10³⁸`, so values in the range `(3.4 × 10³², 3.4 × 10³⁸]` are valid EVM amounts that trigger the overflow. The wrapped result `(evm_amount × 10⁶) mod 2¹²⁸` is a drastically smaller (or zero) value.

`denormalize_fee` has the same flaw and is called in the same callback:

```rust
fn denormalize_fee(fee: &Fee, decimals: Decimals) -> Fee {
    Fee {
        fee: U128(Self::denormalize_amount(fee.fee.0, decimals)),
        ...
    }
}
``` [5](#0-4) 

---

### Impact Explanation

When overflow occurs:

1. The user's EVM tokens are **permanently burned** (or locked) by `initTransfer`.
2. `fin_transfer_callback` stores a silently-wrapped, near-zero amount as `transfer_message.amount`.
3. The bridge mints or transfers this tiny wrapped amount to the recipient on NEAR.
4. The difference — potentially the entire bridged value — is **irrecoverably lost**; there is no refund path.

This is balance/accounting corruption that breaks bridge collateralization: the EVM side records a large burn while the NEAR side mints a negligible amount, permanently destroying the peg for that transfer.

---

### Likelihood Explanation

The overflow threshold for `diff_decimals = 6` is `≈ 3.4 × 10³²` raw token units (i.e., `≈ 3.4 × 10¹⁴` tokens with 18 decimals). While this is a large number, it is a valid `uint128` and is reachable for high-supply tokens (e.g., meme tokens with supplies in the quadrillions). Any user who holds such an amount and bridges it triggers the bug without any privileged access. The EVM `initTransfer` is a public, permissionless function callable by any token holder.

---

### Recommendation

Replace the unchecked multiplication with a checked variant and panic with a descriptive error on overflow:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    let scale = 10_u128.checked_pow(diff_decimals)
        .near_expect(BridgeError::AmountOverflow);
    amount.checked_mul(scale)
        .near_expect(BridgeError::AmountOverflow)
}
```

This ensures the transaction reverts rather than silently corrupting the bridged amount. The same fix applies to `denormalize_fee`.

---

### Proof of Concept

1. Deploy a high-supply ERC-20 token on EVM with 18 decimals, registered on NEAR with `origin_decimals = 24`, `decimals = 18` (`diff_decimals = 6`).
2. Mint `amount = u128::MAX / 10^6 + 1` (≈ `3.4 × 10³²`) tokens to an attacker-controlled address.
3. Call `OmniBridge.initTransfer(token, amount, 0, 0, "near:victim.near", "")` — tokens are burned on EVM.
4. A relayer submits the resulting `InitTransfer` proof to NEAR's `fin_transfer`.
5. `fin_transfer_callback` computes `denormalize_amount(3.4×10³² + 1, {decimals:18, origin_decimals:24})` = `(3.4×10³² + 1) × 10⁶ mod 2¹²⁸` ≈ a small value near zero.
6. The bridge mints this tiny amount to `victim.near`; the `3.4×10³²` EVM tokens are permanently gone. [1](#0-0) [6](#0-5) [7](#0-6)
