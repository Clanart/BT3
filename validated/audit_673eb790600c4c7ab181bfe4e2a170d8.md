### Title
Missing Class Hash Existence Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS Cairo program does not validate that the caller-supplied class hash corresponds to a declared class before committing the state update. This is an exact structural analog to the reported `setClientOwner` bug: a state-mutating function accepts an unvalidated input and writes it into a critical mapping without checking the target's existing or required state. The result is that any contract can permanently replace its own class hash with an undeclared (non-existent) hash, making the contract permanently non-executable and freezing all funds it holds.

---

### Finding Description

`execute_replace_class` (lines 877–916 of `syscall_impls.cairo`) handles the `replace_class` syscall. It reads the new class hash from the caller-supplied `ReplaceClassRequest`, then unconditionally writes it into `contract_state_changes` via `dict_update`:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The TODO comment on line 898 is an explicit, in-code acknowledgment that the required validation — confirming the new class hash exists in the declared class tree — is absent. No other code path performs this check before the state update is committed.

Contrast this with `deploy_contract.cairo`, which correctly validates the pre-condition before writing state:

```cairo
assert state_entry.class_hash = UNINITIALIZED_CLASS_HASH;
assert state_entry.nonce = 0;
```

`execute_replace_class` has no equivalent guard on the post-condition (the new class hash).

The structural parallel to the reported bug is exact:

| Original (`setClientOwner`) | StarkNet analog (`execute_replace_class`) |
|---|---|
| Transfers a name to a new address | Replaces a contract's class hash |
| Missing check: target address already has a name | Missing check: new class hash is not declared |
| Result: existing privilege overwritten | Result: contract class set to non-existent hash |

---

### Impact Explanation

Once `replace_class` is called with an undeclared class hash, the state update is committed to the blockchain. The contract's `StateEntry.class_hash` is permanently set to a hash that has no corresponding class in the class commitment tree. Every subsequent call to that contract — whether via `call_contract`, `library_call`, or a direct transaction — will fail at class resolution time because the OS cannot find executable code for the hash. The contract becomes permanently non-executable. Any ERC-20 tokens, ETH, STRK, or other assets held in the contract's storage are permanently frozen with no recovery path.

**Impact class: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The `replace_class` syscall is callable by any deployed contract without any privilege requirement. The missing validation is explicitly acknowledged in the production code via a TODO comment dated 2026, meaning it is a known gap that has not been closed. A contract that exposes any user-controlled upgrade path (e.g., a proxy pattern where the implementation hash is passed as calldata) is directly exploitable: an attacker supplies an undeclared hash, the OS accepts it, and the contract is frozen. DeFi protocols, bridges, and multi-sig wallets using upgradeable proxy patterns are the highest-risk targets.

---

### Recommendation

Before committing the `dict_update` in `execute_replace_class`, add a lookup into `contract_class_changes` (or the underlying class commitment tree) to assert that the supplied `class_hash` has a corresponding declared entry. The check should mirror the guard already present in `deploy_contract.cairo` for the pre-deployment state. Specifically, resolve the TODO on line 898 by asserting:

```cairo
// Assert the new class hash is declared (non-zero entry in contract_class_changes).
let (declared_class) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(declared_class);
```

Additionally, add a unit/integration test that attempts `replace_class` with an undeclared hash and asserts the transaction reverts.

---

### Proof of Concept

1. Attacker deploys Contract A (a proxy holding user funds) with a valid class hash `H_valid`.
2. Attacker calls `replace_class` on Contract A, supplying `class_hash = 0xdeadbeef` (an arbitrary, undeclared hash).
3. `execute_replace_class` reads `request.class_hash = 0xdeadbeef`, skips the missing existence check (line 898 TODO), and calls `dict_update` to set `contract_state_changes[contract_A].class_hash = 0xdeadbeef`.
4. The block is proven and the state is committed. Contract A's class hash is now `0xdeadbeef` on-chain.
5. Any subsequent `call_contract` or `invoke` targeting Contract A reaches `execute_call_contract`, which reads `state_entry.class_hash = 0xdeadbeef` and attempts to resolve the class. No class exists for this hash; execution fails unconditionally.
