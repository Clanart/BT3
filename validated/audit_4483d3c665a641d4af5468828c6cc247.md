### Title
Cross-Chain Replay of `MetadataPayload` MPC Signature Enables Permanent Token Deployment Blocking â€” (`near/omni-bridge/src/lib.rs`, `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The MPC signature produced by NEAR's `log_metadata_callback` over a `MetadataPayload` contains no destination chain ID, no contract address, and no chain-specific binding. The identical signature is accepted verbatim by `deployToken` on every EVM chain, Starknet, and Solana. An unprivileged attacker who observes the publicly emitted `LogMetadataEvent` on NEAR can replay the signature on any chain where the token has not yet been deployed, squatting the `nearToEthToken` / `near_to_starknet_token` / wrapped-mint mapping and permanently blocking the legitimate deployment. Any in-flight or future user transfer of that token to the targeted chain becomes permanently unclaimable.

---

### Finding Description

**Signing side â€” NEAR `log_metadata_callback`**

`log_metadata_callback` constructs a `MetadataPayload` and submits it to the MPC signer:

```rust
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,
};
let payload = near_sdk::env::keccak256_array(
    borsh::to_vec(&metadata_payload).near_expect(BridgeError::Borsh),
);
ext_signer::ext(self.mpc_signer.clone())
    .sign(SignRequest { payload, path: SIGN_PATH.to_owned(), key_version: 0 })
``` [1](#0-0) 

The `MetadataPayload` struct contains only `prefix`, `token`, `name`, `symbol`, and `decimals` â€” no chain ID, no contract address, no nonce. [2](#0-1) 

The resulting signature is broadcast publicly in a `LogMetadataEvent` on NEAR.

---

**Verification side â€” EVM `deployToken`**

The EVM contract reconstructs the same chain-agnostic borsh blob and verifies the signature:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [3](#0-2) 

No chain ID is included. The same `nearBridgeDerivedAddress` is configured on every EVM deployment (Ethereum, BNB, Arbitrum, Base, etc.), so the signature is valid on all of them simultaneously.

---

**Verification side â€” Starknet `deploy_token`**

Starknet's `MetadataPayloadImpl::to_borsh` encodes identically â€” no chain ID:

```cairo

    borsh_bytes

``` [4](#0-3) 

The Starknet `deploy_token` verifies against the same derived NEAR bridge address. [5](#0-4)

---

**Contrast with `finTransfer` â€” chain binding IS present there**

`finTransfer` on EVM encodes `omniBridgeChainId` twice (for destination chain and recipient chain), preventing cross-chain replay of transfer signatures: [8](#0-7) 

Starknet's `TransferMessagePayloadImpl::to_borsh` similarly binds `chain_id`: [9](#0-8) 

The Starknet CLAUDE.md explicitly documents this design decision for `fin_transfer` but the same protection was never applied to `deploy_token`. The `MetadataPayload` signature is the only MPC-signed message in the protocol that carries no chain binding.

### Impact Explanation

After a successful replay on chain C:

1. `nearToEthToken[token]` (EVM) or `near_to_starknet_token[token_id_hash]` (Starknet) is set to the attacker-triggered phantom token address.
2. The duplicate-token guard fires on any subsequent legitimate `deployToken` call: `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")` (EVM line 155â€“157) or `assert(existing_token.is_zero(), 'ERR_TOKEN_ALREADY_DEPLOYED')` (Starknet line 209).
3. NEAR's `deploy_token_internal` is never reached for chain C, so `token_id_to_address` for chain C is never populated.
4. `sign_transfer` on NEAR panics with `FailedToGetTokenAddress` for any transfer destined for chain C, because `get_token_address(chain_C, token_id)` returns `None`.
5. Every user who has initiated or will initiate a transfer of that token to chain C has their funds permanently locked in the NEAR bridge â€” the transfer can never be finalized and there is no recovery path in the contract. [10](#0-9) [11](#0-10) 

---

### Likelihood Explanation

- The `LogMetadataEvent` containing the MPC signature is emitted on-chain on NEAR and is publicly observable by any party.
- The attacker needs no special role, no key material, and no privileged access â€” only the ability to call `deployToken` on the target chain with the replayed signature.
- The attacker can target any chain where the token has not yet been deployed, including chains that are added to the bridge after the signature was originally produced.
- The attack is a single transaction and requires no coordination.
- Likelihood: **High**.

---

### Recommendation

Bind the `MetadataPayload` to a specific destination chain by including the destination chain ID in the signed message, mirroring the existing protection in `TransferMessagePayload`. On the NEAR signing side, produce one signature per target chain (each including that chain's ID). On each destination contract, verify that the chain ID in the signed blob matches its own configured chain ID. This is the same pattern already correctly applied to `finTransfer` across all three destination chains.

---

### Proof of Concept

1. Relayer calls `log_metadata("foo.near")` on the NEAR bridge. NEAR fetches metadata and calls MPC to sign `keccak256(borsh({prefix: Metadata, token: "foo.near", name: "Foo", symbol: "FOO", decimals: 18}))`. The resulting `(r, s, v)` signature is emitted in `LogMetadataEvent`.

2. Attacker observes the NEAR transaction receipt and extracts the signature.

3. Legitimate relayer calls `deployToken(sig, {token:"foo.near",...})` on Ethereum â€” succeeds, token deployed at address `0xAAA`.

4. Before the legitimate relayer reaches BNB chain, attacker calls `deployToken(sig, {token:"foo.near", name:"Foo", symbol:"FOO", decimals:18})` on the BNB `OmniBridge`. The signature passes `ECDSA.recover(hashed, sig) == nearBridgeDerivedAddress` because the hash is identical (no chain ID). A phantom ERC20 is deployed at `0xBBB`. `nearToEthToken["foo.near"] = 0xBBB`, `isBridgeToken[0xBBB] = true`.

5. Legitimate relayer now calls `deployToken(sig, ...)` on BNB â€” reverts with `"ERR_TOKEN_EXIST"`.

6. NEAR's `token_id_to_address` for BNB is never set. Any user who calls `ft_transfer_call` on NEAR to bridge `foo.near` â†’ BNB will have their transfer stored in `pending_transfers`. When `sign_transfer` is called, `get_token_address(BNB, foo.near)` returns `None` and the call panics with `FailedToGetTokenAddress`. The user's tokens are permanently locked in the NEAR bridge with no recovery path. [12](#0-11) [13](#0-12) [4](#0-3) [6](#0-5)

```rust
impl<P: Payload> SignedPayload<P> {
    pub fn verify_signature(
        &self,
        params: P::AdditionalParams,
        derived_near_bridge_address: &[u8; 64],
    ) -> Result<()> {
        let serialized = self.payload.serialize_for_near(params)?;
        let hash = keccak::hash(&serialized);

        let signature_bytes = &self.signature[0..64];

        let signature = libsecp256k1::Signature::parse_standard_slice(signature_bytes)
            .map_err(|_| ProgramError::InvalidArgument)?;
        require!(!signature.s.is_high(), ErrorCode::MalleableSignature);

        let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)
            .map_err(|_| error!(ErrorCode::SignatureVerificationFailed))?;

        require!(
            signer.0 == *derived_near_bridge_address,
            ErrorCode::SignatureVerificationFailed
        );

        Ok(())
    }
