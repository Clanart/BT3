### Title
Fee-on-Transfer Token Accounting Discrepancy Enables Bridge Undercollateralization - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

The `initTransfer` function in `OmniBridge.sol` pulls tokens from the user via `safeTransferFrom` but emits the `InitTransfer` event using the caller-supplied `amount` rather than the actual received balance. For fee-on-transfer tokens, the bridge receives less than `amount` but signals the full `amount` to the NEAR side, causing the NEAR bridge to mint more wrapped tokens than are actually locked on EVM — permanently breaking collateralization.

### Finding Description

In `OmniBridge.initTransfer`, when the token is neither a bridge token nor a custom-minter token, the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
``` [1](#0-0) 

Immediately after, the event is emitted with the user-supplied `amount`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // <-- user-specified, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

No balance-before/balance-after check is performed. For fee-on-transfer tokens (tokens that deduct a percentage or flat fee during `transferFrom`), the bridge's actual EVM balance increases by `amount - transfer_fee`, but the emitted event records `amount`.

The NEAR bridge consumes this event in `fin_transfer_callback`, reconstructing the `TransferMessage` directly from the proof-decoded `init_transfer.amount`:

```rust
let transfer_message = TransferMessage {
    ...
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
};
``` [3](#0-2) 

The NEAR side then mints or releases the full `amount` to the recipient, with no awareness that the EVM bridge only holds `amount - transfer_fee`.

### Impact Explanation

Every deposit of a fee-on-transfer token creates a deficit: the EVM bridge holds `amount - fee` but the NEAR side mints wrapped tokens for `amount`. Over multiple deposits, the cumulative shortfall grows. When users attempt to bridge back from NEAR to EVM, the bridge's EVM token balance is insufficient to cover all outstanding wrapped-token claims. The last redeemers cannot withdraw their funds — their wrapped tokens on NEAR become permanently unclaimable, constituting an irrecoverable lock of user funds and a direct break of bridge collateralization.

This matches the allowed impact: **"High. Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

### Likelihood Explanation

Fee-on-transfer tokens are a well-established ERC-20 pattern (e.g., tokens with redistribution mechanics, deflationary tokens, or tokens with protocol fees on transfer). The bridge does not whitelist tokens — any ERC-20 that is not registered as a `isBridgeToken` or `customMinters` entry falls into the unguarded `safeTransferFrom` branch. An unprivileged user can trigger this by simply calling `initTransfer` with any fee-on-transfer ERC-20 that has been mapped to a NEAR token ID. No special role or privileged access is required.

### Recommendation

Measure the actual received amount by comparing the contract's token balance before and after the `safeTransferFrom` call, and use the measured delta — not the caller-supplied `amount` — for the emitted event and all downstream accounting:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived in the InitTransfer event and nonce accounting
```

Alternatively, explicitly disallow fee-on-transfer tokens via a token registry check or a revert if `actualReceived != amount`.

### Proof of Concept

1. A fee-on-transfer token `FOT` (e.g., 1% fee on every transfer) is registered in the NEAR bridge's token mapping.
2. Alice calls `OmniBridge.initTransfer(FOT, 1000e18, 0, 0, "near:alice.near", "")`.
3. `safeTransferFrom` pulls `1000e18` from Alice; the token contract deducts 1% → bridge receives `990e18`.
4. `InitTransfer` event is emitted with `amount = 1000e18`.
5. NEAR relayer submits proof; `fin_transfer_callback` decodes `amount = 1000e18` and mints `1000e18` wrapped FOT to `alice.near`.
6. Bridge EVM balance: `990e18`. Outstanding NEAR wrapped supply: `1000e18`. Deficit: `10e18`.
7. After enough such deposits, the last redeemers calling `initTransfer` on NEAR to bridge back receive a signed transfer for the full amount, but the EVM bridge reverts or short-pays because its actual balance is insufficient. [4](#0-3)

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,

    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );
