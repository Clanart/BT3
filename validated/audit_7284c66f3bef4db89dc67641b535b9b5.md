### Title
EIP-712 `Tx` Type Omits `timeout_height`, Enabling Transaction Expiry Bypass - (File: `ethereum/eip712/types.go`)

### Summary
The EIP-712 `Tx` type definition in both the current and legacy Ethermint EIP-712 encoding paths deliberately omits `timeout_height`. Because EIP-712 only hashes fields declared in the type schema, the signed digest does not commit to the transaction's expiry height. An attacker who obtains a signed EIP-712 Cosmos transaction from the mempool can zero out `timeout_height` and resubmit the transaction after the user's intended expiry block, bypassing the user's authorization intent.

### Finding Description

**Root cause — type schema omission:**

In `ethereum/eip712/types.go` the `Tx` type is defined as:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id",       Type: "string"},
    {Name: "fee",            Type: "Fee"},
    {Name: "memo",           Type: "string"},
    {Name: "sequence",       Type: "string"},
    // Note timeout_height was removed because it was not getting filled with the legacyTx
},
```

The identical omission exists in the legacy path:

```go
// Note timeout_height was removed because it was not getting filled with the legacyTx
// {Name: "timeout_height", Type: "string"},
``` [1](#0-0) [2](#0-1) 

**`timeout_height` IS present in the Amino sign-doc bytes but is silently ignored by EIP-712:**

`VerifySignature` in the legacy ante handler reconstructs `txBytes` by calling `legacytx.StdSignBytes` with `tx.GetTimeoutHeight()`, so the Amino JSON payload does contain the field. However, because `timeout_height` is absent from the `Tx` type schema, EIP-712's `hashStruct` never includes it in the digest. [3](#0-2) 

The same applies to the non-legacy path: `decodeAminoSignDoc` passes the full Amino sign-doc bytes (which include `timeout_height`) to `WrapTxToTypedData`, but the `Tx` type schema still omits the field. [4](#0-3) 

Note: the Protobuf decode path (`decodeProtobufSignDoc`) does guard against this by explicitly rejecting `TimeoutHeight != 0`, but the Amino path has no such guard. [5](#0-4) 

**Ante handler ordering enables the bypass:**

In `newLegacyCosmosAnteHandlerEip712`, `authante.NewTxTimeoutHeightDecorator()` runs at position 5, **before** `cosmos.NewLegacyEip712SigVerificationDecorator()` at position 12. If an attacker sets `timeout_height = 0`, the timeout decorator passes (0 means "no expiry"), and the signature decorator then verifies successfully because the hash never covered `timeout_height`. [6](#0-5) 

The same ordering exists in `newCosmosAnteHandler` for the non-legacy path. [7](#0-6) 

### Impact Explanation

An attacker who obtains a signed EIP-712 Cosmos transaction (e.g., from the public mempool after the user broadcast it) can:

1. Modify `timeout_height` from the user's intended expiry value to `0`.
2. Resubmit the modified transaction after the original expiry block.
3. `TxTimeoutHeightDecorator` does not reject it (`timeout_height = 0` = no expiry).
4. `LegacyEip712SigVerificationDecorator` (or `ethsecp256k1.PubKey.VerifySignature`) verifies the signature successfully because the EIP-712 digest never committed to `timeout_height`.
5. The transaction executes — transferring funds, voting, delegating, etc. — after the user intended it to be invalid.

This is an **EIP-712 authorization bypass enabling replay**: the user's signature authorizes a strictly broader set of transactions than intended (any `timeout_height` value, including none).

### Likelihood Explanation

- **Attacker access to signed bytes**: Any transaction broadcast to the mempool is visible to all full nodes. A user who signs with `timeout_height = X` and broadcasts before block X gives the attacker the signed bytes.
- **Sequence number constraint**: The attack succeeds only while the user's on-chain sequence number equals the one in the signed transaction. If the user submits any other transaction first, the sequence advances and the replayed tx is rejected. This limits the window but does not eliminate it — a user who signed a tx, saw it expire, and then paused activity is fully exposed.
- **No special privilege required**: The attacker is an unprivileged network participant who can observe the mempool and submit transactions.

### Recommendation

Add `{Name: "timeout_height", Type: "string"}` to the `Tx` type definition in both `ethereum/eip712/types.go` and `ethereum/eip712/eip712_legacy.go`. This ensures the EIP-712 digest commits to `timeout_height`, making any post-signing modification detectable. The Amino sign-doc already serializes the field (with `omitempty`), so no changes to the serialization layer are needed — only the type schema must be updated.

### Proof of Concept

```
1. Alice signs an EIP-712 Cosmos tx:
      timeout_height = 500, sequence = 7, msg = MsgSend(Alice→Bob, 100atom)
   She broadcasts it; it sits in the mempool.

2. Block 500 passes. The tx is evicted from the mempool by TxTimeoutHeightDecorator.
   Alice decides not to resubmit (she changed her mind about the transfer).
   Alice's on-chain sequence is still 7 (she sent no other tx).

3. Attacker (who observed the mempool) takes the signed tx bytes and sets
      timeout_height = 0   (field absent from Amino JSON due to omitempty)

4. Attacker broadcasts the modified tx at block 600.
   - TxTimeoutHeightDecorator: timeout_height == 0 → no expiry check → PASS
   - LegacyEip712SigVerificationDecorator reconstructs txBytes with
     tx.GetTimeoutHeight() == 0, calls LegacyWrapTxToTypedData.
     The Tx type schema has no timeout_height field → EIP-712 hash is
     identical to the hash Alice signed → ECDSA verification PASSES.
   - IncrementSequenceDecorator increments Alice's sequence to 8.
   - MsgSend executes: 100atom transferred from Alice to Bob.

5. Alice's funds are transferred after she intended the authorization to expire.
```
