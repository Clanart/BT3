Audit Report

## Title
Unaccounted Router ETH Balance Silently Subsidizes Subsequent WETH Swaps, Stealing Prior User's Refund — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary

`PeripheryPayments.pay` reads `address(this).balance` as an unattributed global pool when paying with WETH. ETH stranded on the router from a prior user's `msg.value` overpayment (when `refundETH()` is not called in the same transaction) is silently consumed to fund a subsequent caller's WETH swap, causing direct loss of the prior user's principal with no recovery path.

## Finding Description

`exactInputSingle` is `payable`. [1](#0-0) 

When `tokenIn == WETH` and a user sends `msg.value = Y > amountIn = X`, the swap callback fires `_justPayCallback`: [2](#0-1) 

which calls `pay(WETH, payer, pool, X)`. Inside `pay`, the first branch deposits exactly `X` ETH and transfers WETH to the pool: [3](#0-2) 

This leaves `Y - X` ETH stranded on the router with no per-user accounting. If the user does not call `refundETH()` in the same multicall, the residual ETH persists.

A subsequent attacker calls `exactInputSingle(tokenIn=WETH, amountIn=Z, msg.value=0)`. The callback fires `pay(WETH, attacker, pool, Z)`. `nativeBalance = address(this).balance` now equals the prior user's stranded ETH. The branch at line 75 (`nativeBalance >= value`) fires: [4](#0-3) 

The router deposits the prior user's ETH and transfers WETH to the pool. The attacker's `transferFrom` is never called. The prior user's ETH is permanently lost.

The `receive()` guard: [5](#0-4) 

only blocks direct ETH transfers via `receive()`. It does not prevent ETH accumulation from `msg.value` in payable function calls, which bypass `receive()` entirely. No other guard exists to prevent `pay` from consuming ETH belonging to a prior caller.

## Impact Explanation

Direct loss of user principal. The prior user's stranded ETH (`Y - X`) is consumed to fund an unrelated user's swap. The prior user receives neither WETH nor an ETH refund for that amount. The attacker receives a fully or partially subsidized swap output. This is a Critical/High direct-loss-of-user-principal impact under Sherlock contest thresholds.

## Likelihood Explanation

Moderate. The precondition — ETH stranded on the router — arises naturally whenever a user calls `exactInputSingle` with `msg.value > amountIn` without bundling `refundETH()` in the same multicall. An attacker can monitor the router's ETH balance on-chain and immediately follow with a zero-value WETH swap to drain it. No privileged access, malicious pool, or non-standard token is required.

## Recommendation

Track the ETH attributable to the current transaction's `msg.sender` separately from any pre-existing router balance. The simplest fix is to record `address(this).balance - msg.value` at entry and restrict `pay` to consuming at most `msg.value` of ETH (i.e., only the amount the current caller explicitly sent). Alternatively, enforce that `pay` only uses ETH equal to `msg.value` of the outermost call and revert if `address(this).balance > msg.value` at the time of payment.

## Proof of Concept

```solidity
// 1. User A calls exactInputSingle(tokenIn=WETH, amountIn=0.5e18) with msg.value=1e18.
//    pay() deposits 0.5e18 ETH → 0.5e18 ETH remains on router.
//    User A does NOT call refundETH().

// 2. Attacker calls exactInputSingle(tokenIn=WETH, amountIn=0.5e18) with msg.value=0.
//    Callback: pay(WETH, attacker, pool, 0.5e18).
//    nativeBalance = 0.5e18 >= value = 0.5e18 → line 75 branch fires.
//    Router deposits User A's 0.5e18 ETH; attacker's transferFrom never called.

// Assert: router.balance == 0, User A's 0.5e18 ETH is gone,
//         attacker received full swap output paying nothing.
```
