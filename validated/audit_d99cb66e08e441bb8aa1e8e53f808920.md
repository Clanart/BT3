### Title
Constructor Failure Not Propagated in `execute_deploy` Syscall — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall handler in the StarkNet OS unconditionally writes `failure_flag=0` (success) to the syscall response, regardless of whether the deployed contract's constructor actually succeeded or reverted. This is an exact analog of the ERC-20 `approve()` unchecked-return-value pattern: a sub-operation's failure indicator is silently discarded, and the caller proceeds as if the operation succeeded.

---

### Finding Description

In `execute_deploy` inside `syscall_impls.cairo`, after calling `deploy_contract` (which internally calls `execute_entry_point` for the constructor), the OS writes the response header with a hardcoded `failure_flag=0`:

```cairo
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` call returns `(retdata_size, retdata)` — it does not surface an `is_reverted` flag to the caller. Even if the constructor reverts (and the revert log correctly undoes state changes), the calling contract receives `failure_flag=0` and a valid-looking `contract_address` in the `DeployResponse`. [2](#0-1) 

The `EntryPointReturnValues` struct explicitly carries a `failure_flag` field that `execute_entry_point` populates from the contract's execution result, but this information is never threaded back through `deploy_contract` to `execute_deploy`. [3](#0-2) 

The TODO comment at line 538 confirms this is a known, unresolved gap in the implementation.

---

### Impact Explanation

When a constructor reverts:

1. The OS correctly undoes the constructor's state changes via the revert log.
2. The contract at the computed address is **not deployed** (its class hash remains 0 / unset in state).
3. However, the calling contract receives `failure_flag=0` and a `contract_address` in the `DeployResponse`.
4. The calling contract has no protocol-level way to detect the failure.
5. If the calling contract subsequently transfers tokens or native assets to that address (e.g., seeding initial liquidity in a factory pattern), those funds are sent to an address with no deployed contract.
6. Because no contract exists at that address to handle incoming calls, the funds are **permanently frozen** — they cannot be recovered.

This satisfies the **Critical: Permanent freezing of funds** impact category.

---

### Likelihood Explanation

- The `deploy` syscall is available to any Sierra contract without privilege restrictions.
- Any unprivileged user who can invoke a factory-style contract (one that deploys sub-contracts and immediately funds them) can supply constructor calldata crafted to trigger a constructor revert.
- The OS-level misreporting of `failure_flag=0` is unconditional — it applies to every `deploy` syscall invocation where the constructor fails, regardless of the class or caller.
- Factory patterns (deploy + fund) are a standard DeFi primitive on StarkNet, making realistic targets plentiful.

---

### Recommendation

The `deploy_contract` function must be modified to return an `is_reverted` flag (analogous to how `execute_entry_point` returns `is_reverted`). The `execute_deploy` handler must then propagate this flag into the `ResponseHeader`:

```cairo
// Correct fix (pseudocode):
with remaining_gas {
    let (is_reverted, retdata_size, retdata) = deploy_contract(...);
}
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

Until this is fixed, calling contracts cannot distinguish a successful deploy from a failed one, and any contract that acts on the returned `contract_address` without an independent verification step is vulnerable.

---

### Proof of Concept

**Attack scenario — factory contract draining:**

1. Attacker identifies a factory contract `F` that:
   - Accepts a `class_hash` and `constructor_calldata` from the caller.
   - Calls the `deploy` syscall.
   - On receiving the response, reads `failure_flag` (which is always 0).
   - Immediately transfers `N` tokens to `response.contract_address`.

2. Attacker declares a contract class `C` whose constructor unconditionally reverts (e.g., `assert 0 = 1`).

3. Attacker calls `F` with `class_hash = C` and any `constructor_calldata`.

4. OS execution path:
   - `execute_deploy` is entered.
   - `deploy_contract` calls the constructor of `C`, which reverts.
   - Revert log undoes constructor state changes; address `A` has no deployed class.
   - OS writes `ResponseHeader(gas=..., failure_flag=0)` and `DeployResponse(contract_address=A, ...)`.

5. `F` reads `failure_flag=0`, concludes success, and transfers `N` tokens to address `A`.

6. Address `A` has no contract. The tokens are permanently frozen. [4](#0-3)
