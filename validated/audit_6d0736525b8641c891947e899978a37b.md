### Title
`CalculateBaseFee` Panics on Valid CometBFT `MaxGas = -1` Configuration, Halting the Chain - (File: `x/feemarket/keeper/eip1559.go`)

### Summary

`CalculateBaseFee` in the feemarket keeper unconditionally panics when `ConsensusParams.Block.MaxGas <= -1`. CometBFT explicitly documents `-1` as a valid value meaning "unlimited block gas." Any chain deployed with this configuration will experience a deterministic node crash on every `BeginBlock` call after `EnableHeight`, halting consensus.

### Finding Description

In `x/feemarket/keeper/eip1559.go`, `CalculateBaseFee` reads the consensus parameters and panics if `MaxGas` is zero or negative:

```go
if consParams.Block == nil || consParams.Block.MaxGas <= -1 {
    panic(fmt.Sprintf("get invalid consensus params: %s", consParams))
}
``` [1](#0-0) 

This function is called unconditionally from `BeginBlock` on every block after `EnableHeight`:

```go
func (k *Keeper) BeginBlock(ctx sdk.Context) error {
    baseFee := k.CalculateBaseFee(ctx)
    ...
}
``` [2](#0-1) 

CometBFT's own documentation and proto definition state that `MaxGas = -1` means "no limit on the maximum gas consumption for transactions." This is a valid, documented, and commonly used configuration. The Ethermint codebase itself acknowledges this in `types/block.go`, where `BlockGasLimit` handles `MaxGas == -1` gracefully by returning `math.MaxUint64`:

```go
if maxGas == -1 {
    return math.MaxUint64
}
``` [3](#0-2) 

The inconsistency is the root cause: `BlockGasLimit` handles `-1` gracefully, but `CalculateBaseFee` treats it as an invalid state and panics. A panic in `BeginBlock` propagates through the Cosmos SDK module manager and crashes the node process, causing a chain halt.

### Impact Explanation

**Critical.** A chain deployed with `MaxGas = -1` (unlimited gas, a valid CometBFT configuration) and `NoBaseFee = false` (EIP-1559 enabled) will panic on every `BeginBlock` after `EnableHeight`. This crashes every validator node deterministically, halting the chain. The panic is in the block-processing path (`BeginBlock`), not in a recoverable handler.

### Likelihood Explanation

Any chain operator who sets `MaxGas = -1` in genesis (to allow unlimited gas) while also enabling the feemarket module will trigger this. The two settings are independently valid and there is no validation at genesis or parameter-update time that prevents their combination. The feemarket module's own parameter validation does not check consensus params.

### Recommendation

Replace the `panic` with a graceful fallback. Mirror the behavior of `types/block.go`:

```go
if consParams.Block == nil {
    return nil
}
if consParams.Block.MaxGas == -1 {
    // unlimited gas: skip dynamic base fee adjustment
    return new(big.Int).Set(parentBaseFee)
}
if consParams.Block.MaxGas <= 0 {
    return nil
}
```

This matches the pattern already established in `types/block.go` and avoids a chain-halting panic on a valid CometBFT configuration.

### Proof of Concept

1. Deploy an Ethermint chain with genesis `consensus_params.block.max_gas = -1` and `feemarket.params.no_base_fee = false`, `enable_height = 0`.
2. Submit any transaction to advance past block 0.
3. On block 1, `BeginBlock` calls `CalculateBaseFee`, which reads `consParams.Block.MaxGas == -1`, satisfies `<= -1`, and panics.
4. The panic propagates through the module manager, crashing the node process.
5. All validators crash deterministically on the same block, halting the chain. [4](#0-3) [5](#0-4) [6](#0-5)

func (k Keeper) CalculateBaseFee(ctx sdk.Context) *big.Int {
	params := k.GetParams(ctx)

	// Ignore the calculation if not enabled
	if !params.IsBaseFeeEnabled(ctx.BlockHeight()) {
		return nil
	}
	consParams := ctx.ConsensusParams()

	// If the current block is the first EIP-1559 block, return the base fee
	// defined in the parameters (DefaultBaseFee if it hasn't been changed by
	// governance).
	if ctx.BlockHeight() == params.EnableHeight {
		return params.BaseFee.BigInt()
	}

	// get the block gas used and the base fee values for the parent block.
	// NOTE: this is not the parent's base fee but the current block's base fee,
	// as it is retrieved from the transient store, which is committed to the
	// persistent KVStore after EndBlock (ABCI Commit).
	parentBaseFee := params.BaseFee.BigInt()
	if parentBaseFee == nil {
		return nil
	}

	parentGasUsed := k.GetBlockGasWanted(ctx)

	// NOTE: a MaxGas equal to -1 means that block gas is unlimited
	if consParams.Block == nil || consParams.Block.MaxGas <= -1 {
		panic(fmt.Sprintf("get invalid consensus params: %s", consParams))
	}

func (k *Keeper) BeginBlock(ctx sdk.Context) error {
	baseFee := k.CalculateBaseFee(ctx)

	// return immediately if base fee is nil
	if baseFee == nil {
		return nil
	}

	k.SetBaseFee(ctx, baseFee)

	defer func() {
		telemetry.SetGauge(float32(baseFee.Int64()), "feemarket", "base_fee") //nolint:staticcheck
	}()

	// Store current base fee in event
	ctx.EventManager().EmitEvents(sdk.Events{
		sdk.NewEvent(
			types.EventTypeFeeMarket,
			sdk.NewAttribute(types.AttributeKeyBaseFee, baseFee.String()),
		),
	})
	return nil
}

func BlockGasLimit(ctx sdk.Context) uint64 {
	// Otherwise get from the consensus parameters
	cp := ctx.ConsensusParams()
	if cp.Block == nil {
		return 0
	}

	maxGas := cp.Block.MaxGas

	// Setting max_gas to -1 in CometBFT means there is no limit on the maximum gas consumption for transactions
	// https://github.com/cometbft/cometbft/blob/v0.37.2/proto/tendermint/types/params.proto#L25-L27
	if maxGas == -1 {
		return math.MaxUint64
	}

	if maxGas > 0 {
		return uint64(maxGas)
	}

	return 0
