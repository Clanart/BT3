### Title
`deploy_token` Is Not Pausable on Solana Despite Admin Pause Mechanism — (`solana/programs/bridge_token_factory/src/lib.rs`)

### Summary

The Solana `bridge_token_factory` program exposes a `pause()` instruction that sets `config.paused = ALL_PAUSED`. However, `ALL_PAUSED` only covers `INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED` — there is no `DEPLOY_TOKEN_PAUSED` flag, and the `deploy_token` instruction performs no pause check at all. This means an admin emergency pause cannot halt token deployments on Solana, even though the EVM `OmniBridge` correctly implements a `PAUSED_DEPLOY_TOKEN` flag and enforces it on `deployToken`.

### Finding Description

In `constants.rs`, the pause flags are defined as:

```rust
pub const INIT_TRANSFER_PAUSED: u8 = 1 << 0;
pub const FINALIZE_TRANSFER_PAUSED: u8 = 1 << 1;
pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED;
``` [1](#0-0) 

There is no `DEPLOY_TOKEN_PAUSED` bit. The `pause()` instruction sets `config.paused = ALL_PAUSED`: [2](#0-1) 

The `deploy_token` entry point in `lib.rs` performs no pause check:

```rust
pub fn deploy_token(
    ctx: Context<DeployToken>,
    data: SignedPayload<DeployTokenPayload>,
) -> Result<()> {
    msg!("Deploying token");
    data.verify_signature((), &ctx.accounts.common.config.derived_near_bridge_address)?;
    ctx.accounts.initialize_token_metadata(data.payload)?;
    Ok(())
}
``` [3](#0-2) 

Compare this to `init_transfer` and `finalize_transfer`, which both explicitly check the pause flag before proceeding: [4](#0-3) [5](#0-4) 

On the EVM side, the equivalent `deployToken` function correctly enforces `whenNotPaused(PAUSED_DEPLOY_TOKEN)`: [6](#0-5) 

And the EVM `pauseAll()` includes `PAUSED_DEPLOY_TOKEN` in its flag set: [7](#0-6) 

The Solana `pause()` instruction has no equivalent coverage for token deployment.

### Impact Explanation

When an admin calls `pause()` on Solana to halt all bridge operations during an emergency (e.g., a discovered vulnerability in token deployment logic, a token with incorrect decimals being deployed, or a token-mapping collision being exploited), the `deploy_token` instruction remains fully callable. Any holder of a valid MPC-signed `DeployTokenPayload` — including a user who requested a token deployment on NEAR before the pause — can still submit it to Solana and complete the deployment. This can result in:

- A token being deployed with incorrect or malicious metadata (wrong decimals, wrong symbol) that corrupts the Solana token-mapping and breaks bridge accounting/collateralization.
- A token-mapping entry being created that the admin intended to block, enabling subsequent `finalize_transfer` calls (once unpaused) to mint against a corrupted mapping.

This fits the allowed impact: **High — token-mapping or accounting corruption that breaks bridge collateralization or misdirects value**.

### Likelihood Explanation

Any user who submitted a token deployment request on NEAR before the emergency pause will have received a valid MPC-signed `DeployTokenPayload`. They can submit this payload to Solana at any time, including after the admin has called `pause()`. No special privilege is required beyond possession of the signed payload. The scenario is realistic whenever an admin needs to halt bridge operations urgently.

### Recommendation

1. Add a `DEPLOY_TOKEN_PAUSED` constant to `constants.rs`:
   ```rust
   pub const DEPLOY_TOKEN_PAUSED: u8 = 1 << 2;
   pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED | DEPLOY_TOKEN_PAUSED;
   ```

2. Add a pause check at the top of `deploy_token` in `lib.rs`:
   ```rust
   pub fn deploy_token(
       ctx: Context<DeployToken>,
       data: SignedPayload<DeployTokenPayload>,
   ) -> Result<()> {
       require!(
           ctx.accounts.common.config.paused & DEPLOY_TOKEN_PAUSED == 0,
           error::ErrorCode::Paused
       );
       // ...
   }
   ```

### Proof of Concept

1. Admin discovers an emergency and calls `pause()` on Solana — `config.paused` is set to `ALL_PAUSED` (= `0x03`).
2. A user who previously requested a token deployment on NEAR holds a valid MPC-signed `DeployTokenPayload`.
3. The user calls `deploy_token` on Solana with the signed payload.
4. `lib.rs:deploy_token` performs no pause check — it proceeds directly to `verify_signature` and `initialize_token_metadata`.
5. The token is deployed on Solana despite the admin's emergency pause, potentially with incorrect decimals or a colliding token mapping, corrupting bridge accounting. [3](#0-2) [1](#0-0)
