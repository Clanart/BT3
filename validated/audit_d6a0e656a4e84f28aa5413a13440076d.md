### Title
Unvalidated `compiled_class_hash` in Declare Transaction Enables Class Hash Poisoning — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

In `execute_declare_transaction`, the `compiled_class_hash` supplied by the transaction sender is stored directly into the state with only a non-zero check. The OS never validates that this value corresponds to the actual Casm compiled class. An unprivileged attacker can front-run any open-source class declaration by submitting the correct Sierra class with an arbitrary `compiled_class_hash`, poisoning the class-hash-to-compiled-class mapping and permanently freezing funds in any contract subsequently deployed under that class hash.

---

### Finding Description

In `execute_declare_transaction`, the `compiled_class_hash` is loaded from a prover hint that is ultimately derived from the transaction sender's signed field: [1](#0-0) 

It is then committed to the state with only an `assert_not_zero` guard: [2](#0-1) 

The OS **does** enforce that `class_hash` is correctly derived from the Sierra class components via `finalize_class_hash`: [3](#0-2) 

However, there is **no analogous enforcement** that `compiled_class_hash` corresponds to the actual Casm compiled class. The separate `validate_compiled_class_facts` function does verify compiled class hashes: [4](#0-3) 

But there is no cross-check between the `compiled_class_hash` written into `contract_class_changes` by the declare transaction and the set of validated compiled class facts. The two paths are entirely independent.

Additionally, `dict_update` enforces `prev_value=0`, meaning a class hash can only be declared once: [5](#0-4) 

This one-time-write property is the mechanism that makes the poisoning permanent.

---

### Impact Explanation

**Permanent freezing of funds.**

Once an attacker's declare transaction is accepted with `class_hash = H` mapped to a bogus `compiled_class_hash = X`, the legitimate declaration of `H` is permanently blocked (the `prev_value=0` constraint is no longer satisfiable). Any contract subsequently deployed under class hash `H` will reference compiled class `X`. Because `X` does not correspond to any real compiled class fact that the prover can supply, every call into such a contract will fail at the class-loading step. All funds held by those contracts are irrecoverably frozen.

---

### Likelihood Explanation

- For open-source or predictable contracts (the common case in DeFi), the Sierra class is publicly known, so the attacker can compute the correct `class_hash = H` independently.
- Front-running is feasible via mempool observation; the attacker submits the same Sierra class with a higher gas price and an arbitrary non-zero `compiled_class_hash`.
- The OS provides no protocol-level defense: the only guard is `assert_not_zero(compiled_class_hash)`.
- No privileged role is required; any transaction sender can execute this attack.

---

### Recommendation

Inside `execute_declare_transaction`, after computing the transaction hash, add an explicit assertion that `compiled_class_hash` is present in the set of compiled class facts that have been validated by `validate_compiled_class_facts`. Concretely, maintain a set (or dict) of validated compiled class hashes and assert membership before the `dict_update` call. This mirrors the existing pattern where `class_hash` is verified against `finalize_class_hash` before being accepted.

---

### Proof of Concept

1. Legitimate user constructs a declare transaction: Sierra class `S` → `class_hash = H`, `compiled_class_hash = C` (the correct Casm hash).
2. Attacker observes the pending transaction in the mempool.
3. Attacker constructs their own declare transaction: same Sierra class `S` (so `class_hash = H` passes `finalize_class_hash` validation), but sets `compiled_class_hash = 1` (any non-zero value that has no corresponding compiled class fact).
4. Attacker's transaction is sequenced first (higher tip or direct sequencer submission).
5. OS executes `execute_declare_transaction` for the attacker's transaction:
   - `finalize_class_hash` passes (correct Sierra class provided).
   - `assert_not_zero(1)` passes.
   - `dict_update(key=H, prev_value=0, new_value=1)` succeeds.
6. Legitimate user's transaction is now rejected: `dict_update(key=H, prev_value=0, ...)` fails because `prev_value` is now `1`, not `0`.
7. Any contract deployed with `class_hash = H` references compiled class `1`, which does not exist in the validated compiled class facts.
8. All calls to those contracts revert at class loading; funds are permanently frozen.

func validate_compiled_class_facts{poseidon_ptr: PoseidonBuiltin*, range_check_ptr}(
    n_compiled_class_facts, compiled_class_facts: CompiledClassFact*, builtin_costs: felt*
) {
    if (n_compiled_class_facts == 0) {
        return ();
    }
    alloc_locals;

    let compiled_class_fact = &compiled_class_facts[0];
    let compiled_class = compiled_class_fact.compiled_class;

    validate_entry_points(
        n_entry_points=compiled_class.n_external_functions,
        entry_points=compiled_class.external_functions,
    );

    validate_entry_points(
        n_entry_points=compiled_class.n_l1_handlers, entry_points=compiled_class.l1_handlers
    );
    // Compiled classes are expected to end with a `ret` opcode followed by a pointer to the
    // builtin costs.
    assert compiled_class.bytecode_ptr[compiled_class.bytecode_length] = 0x208b7fff7fff7ffe;
    assert compiled_class.bytecode_ptr[compiled_class.bytecode_length + 1] = cast(
        builtin_costs, felt
    );

    // Calculate the compiled class hash.
    // This hint enters a new scope that contains the bytecode segment structure of the class.
    %{ EnterScopeWithBytecodeSegmentStructure %}
    let (hash) = blake_compiled_class_hash(compiled_class, full_contract=FALSE);
    %{ LoadClass %}

    assert compiled_class_fact.hash = hash;
