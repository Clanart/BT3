### Title
`ProofOfInclusion::valid()` Performs No External Root Check, Allowing Forged DataLayer Inclusion Proofs - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate only verifies internal hash-chain consistency within the proof itself. Its final equality check is tautological — it compares the proof's own last `combined_hash` against itself — meaning any attacker-crafted `ProofOfInclusion` with a self-consistent hash chain will pass `valid()` regardless of whether it corresponds to any real tree state. This is the direct analog of the Witch `buy`/`payAll` missing-state-check: an operation that should verify an external authoritative condition (the known tree root) proceeds without doing so.

### Finding Description

`ProofOfInclusion::valid()` is defined as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()   // ← tautological
}
``` [1](#0-0) 

`root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← taken directly from the proof itself
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

After the loop, `existing_hash` equals the last `calculated_hash`, which the loop already asserted equals `layer.combined_hash`. `root_hash()` returns that same `last.combined_hash`. Therefore `existing_hash == self.root_hash()` is always `true` when the loop completes — the final guard is a no-op.

The function never accepts an external, authoritative root hash to compare against. Any `ProofOfInclusion` whose layers form an internally consistent hash chain (trivially constructable by an attacker) will return `valid() == true`, regardless of what the actual DataLayer tree root is.

`ProofOfInclusion` is a `Streamable` type exposed to Python via `py_valid()`: [3](#0-2) [4](#0-3) 

The struct is fully deserializable from bytes (`from_bytes`, `parse_rust`), making it directly reachable from untrusted network input. [5](#0-4) 

### Impact Explanation

An attacker who can deliver a crafted `ProofOfInclusion` to a DataLayer client can prove that an arbitrary key-value pair is present in a DataLayer store when it is not. The client calls `proof.valid()` (the only exposed validation method), receives `True`, and accepts the forged state. This allows untrusted input to prove invalid DataLayer state, matching the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."*

### Likelihood Explanation

The DataLayer is a peer-to-peer system; proofs are exchanged between nodes. The Python binding exposes `valid()` as the sole self-contained check. The fuzz target and Python tests both call only `proof.valid()` without checking `root_hash()` against an external value: [6](#0-5) 

Any DataLayer consumer that follows this pattern — calling `proof.valid()` without separately asserting `proof.root_hash() == known_root` — is vulnerable. Constructing a passing forged proof requires only computing a valid SHA-256 hash chain, which is trivial.

### Recommendation

`valid()` must accept an external `expected_root: Hash` parameter and assert `self.root_hash() == expected_root` as a non-tautological final step:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    // ... existing chain check ...
    &self.root_hash() == expected_root
}
```

All call sites — including the Python binding, fuzz targets, and any DataLayer peer verification code — must be updated to supply the authoritative root obtained from a trusted source (e.g., the committed on-chain root hash), not from the proof itself.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Attacker wants to "prove" node_hash is in the tree
node_hash = bytes([0xAA] * 32)   # arbitrary claimed leaf
other_hash = bytes([0xBB] * 32)  # arbitrary sibling

# Compute a valid combined_hash using the real calculate_internal_hash logic
# (left < right → sha256(left || right), else sha256(right || left))
left, right = sorted([node_hash, other_hash])
combined = hashlib.sha256(left + right).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=0,       # LEFT
    other_hash=other_hash,
    combined_hash=combined,
)
forged_proof = ProofOfInclusion(node_hash=node_hash, layers=[layer])

assert forged_proof.valid()          # ← returns True
# root_hash() == combined, which is attacker-chosen, not the real tree root
print("Forged proof accepted:", forged_proof.root_hash().hex())
```

The forged proof passes `valid()` with no knowledge of the real tree, because the final check compares `combined` against `combined` — both sourced from the proof itself.
