### Title
Unchecked Multiplication Overflow in `denormalize_amount` Permanently Freezes In-Flight Bridge Funds - (File: `near/omni-bridge/src/lib.rs`)

### Summary
`Contract::denormalize_amount` performs a bare `amount * 10_u128.pow(diff_decimals)` with no overflow guard. Because the NEAR workspace compiles with `overflow-checks = true`, any amount large enough to overflow `u128` causes `fin_transfer_callback` to panic. The panic reverts all state changes, so the transfer is never recorded as finalised on NEAR, while the user's tokens are already locked or burned on the source chain — permanently irrecoverable.

### Finding Description
`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← unchecked multiplication
}
``` [1](#0-0) 

It is called unconditionally inside `fin_transfer_callback` before any state is written:

```rust
let transfer_message = TransferMessage {
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    fee:    Self::denormalize_fee(&init_transfer.fee, decimals),
    ...
};
``` [2](#0-1) 

`init_transfer.amount.0` is the raw amount from the source-chain proof — the EVM bridge emits it directly from the user-supplied `uint128 amount` parameter without any normalization:

```solidity
emit BridgeTypes.InitTransfer(msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message);
``` [3](#0-2) 

For a token whose NEAR representation uses `origin_decimals = 24` and whose EVM representation uses `decimals = 18` (the cap enforced by `_normalizeDecimals`), `diff_decimals = 6`. The multiplication overflows `u128` whenever:

```
amount > u128::MAX / 10^6  ≈  3.4 × 10^32  (in 18-decimal EVM units)
                           ≈  3.4 × 10^14  tokens
``` [4](#0-3) 

With `overflow-checks = true` (confirmed present in `near/Cargo.toml`), the multiplication panics. Because `fin_transfer_callback` is a NEAR callback, the panic reverts every state mutation made inside it — including the `destination_nonce` increment — so the transfer is never inserted into `finalised_transfers` or `pending_transfers`. The proof nonce is not consumed either, so the relayer can retry, but every retry hits the same panic. The user's EVM tokens are already burned or locked with no refund path. [5](#0-4) 

The same overflow applies to `denormalize_fee` (called on the same line), which delegates to `denormalize_amount` for the token-fee component. [6](#0-5) 

**Note on CLAUDE.md false-positive #2:** That entry covers the *subtraction* underflow when `origin_decimals < decimals` (a misconfiguration). The present issue is a *multiplication* overflow triggered by a valid decimal configuration and a user-controlled amount — a distinct root cause not addressed by that entry.

### Impact Explanation
Permanent, irrecoverable freezing of user funds. Tokens are burned/locked on the EVM side before the NEAR callback is invoked; the callback panic leaves no on-chain record of the transfer, and no refund mechanism exists on either side. This matches the **Critical — permanent freezing / irrecoverable lock** impact class.

### Likelihood Explanation
Medium. The condition requires:
1. A token registered with `origin_decimals > decimals` (e.g., any NEAR-native token with 24 decimals bridged to EVM at 18 decimals — the standard configuration).
2. A single transfer exceeding `u128::MAX / 10^diff` in EVM units. For `diff = 6` this is ~3.4 × 10^14 tokens (18-decimal units). High-supply tokens (e.g., meme tokens with supplies in the hundreds of trillions) satisfy this threshold. For larger `diff` values the threshold drops sharply (e.g., `diff = 20` → overflow at ~3.4 tokens).

Any bridge user who holds a sufficient balance of such a token and calls `initTransfer` on EVM can trigger this path; no privileged role is required on the user side.

### Recommendation
Replace the bare multiplication with a checked variant and handle overflow gracefully rather than panicking:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> Option<u128> {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    let factor = 10_u128.checked_pow(diff_decimals)?;
    amount.checked_mul(factor)
}
```

In `fin_transfer_callback`, if `denormalize_amount` returns `None`, the callback should log an error and return a refund nonce (or emit a `FailedFinTransfer` event) rather than panicking, so the transfer can be unwound on the source chain.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 18` (standard NEAR→EVM bridge configuration).
2. On EVM, call `initTransfer` with `amount = u128::MAX / 10^6 + 1` (≈ 3.4 × 10^32 in 18-decimal units). The EVM bridge accepts this value because it only checks `fee < amount`.
3. A relayer calls `fin_transfer` on NEAR with the resulting proof.
4. `fin_transfer_callback` executes `denormalize_amount(u128::MAX / 10^6 + 1, {decimals:18, origin_decimals:24})` = `(u128::MAX / 10^6 + 1) * 10^6`, which overflows `u128`.
5. With `overflow-checks = true`, the NEAR runtime panics; all state changes revert.
6. The transfer is never finalised. The EVM tokens are permanently locked. Retrying produces the same panic. [1](#0-0) [7](#0-6) [8](#0-7)
