Audit Report

## Title
Stranded ETH in Router Is Consumed by Subsequent WETH Swaps — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay` uses `address(this).balance` as the ETH funding source for any WETH-input swap, with no per-caller accounting. A user who sends excess ETH via a `payable` entry-point (e.g., `exactInputSingle`) and does not atomically call `refundETH()` leaves ETH stranded in the router. Any subsequent caller whose `tokenIn == WETH` will have their swap funded by that stranded ETH, consuming it without the original depositor's consent and without drawing the attacker's own WETH allowance.

## Finding Description

`exactInputSingle` (and all other `payable` swap entry-points) accept `msg.value` but do not enforce that the full value is consumed or refunded atomically. The swap callback invokes `_justPayCallback`, which calls `pay(tokenIn, payer, pool, amountOwed)`. Inside `pay`, when `token == WETH`, the branch reads `address(this).balance` — the **total** ETH held by the router — and wraps it as WETH to fund the swap. There is no check that this ETH was contributed by the current `payer`. `refundETH()` is a separate, optional external call that is never invoked inside `exactInputSingle` before it returns. Any ETH left over between two independent transactions is therefore freely accessible to the next caller whose `tokenIn == WETH`.

## Impact Explanation

Direct loss of user principal. User A's unrefunded ETH is wrapped and transferred to a pool to pay for User B's swap; User A receives nothing in return. User B's WETH allowance is never drawn (`safeTransferFrom` is never reached). This is a direct, unconditional loss of funds for User A and an unearned gain for User B. Severity: **High**.

## Likelihood Explanation

No special permissions, malicious pool, or non-standard tokens are required. Any user who sends `msg.value > amountIn` (a common pattern to avoid reverts on partial fills) and does not bundle `refundETH()` in the same `multicall` creates the vulnerable state. An attacker only needs to observe the router's ETH balance on-chain and call `exactInputSingle` with `tokenIn=WETH` and `amountIn <= router.balance`. The attack is repeatable and permissionless.

## Recommendation

Track the ETH contributed by the current top-level call in transient storage (store `msg.value` at entry, deduct from it inside `pay`). Only use the tracked per-call ETH balance, not `address(this).balance`, when wrapping ETH for WETH payments. Alternatively, enforce that `refundETH()` is called atomically at the end of every `payable` entry-point (e.g., inside `exactInputSingle` itself before returning), so no ETH can ever be left stranded between transactions.

## Proof of Concept

```
1. User A: exactInputSingle{value: 1 ether}(tokenIn=WETH, amountIn=0.5 ether, ...)
   - Callback: pay(WETH, userA, pool, 0.5 ether)
     → nativeBalance = 1 ether ≥ 0.5 ether
     → wraps 0.5 ETH, sends WETH to pool ✓
   - 0.5 ETH remains in router. User A does NOT call refundETH().

2. User B: exactInputSingle{value: 0}(tokenIn=WETH, amountIn=0.5 ether, ...)
   - Callback: pay(WETH, userB, pool, 0.5 ether)
     → nativeBalance = 0.5 ether (User A's leftover) ≥ 0.5 ether
     → wraps 0.5 ETH (User A's), sends WETH to pool ✓
     → User B's WETH allowance: NOT touched (safeTransferFrom never called)

Result: User A loses 0.5 ETH. User B's swap is fully subsidized.
```
