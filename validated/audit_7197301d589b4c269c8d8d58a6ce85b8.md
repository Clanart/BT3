### Title
Missing Shard-Index Binding in `set_state_header` Merkle Proof Allows Cross-Shard State Substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` verifies that the supplied chunk is *a* leaf in the block's `chunk_headers_root` Merkle tree, but never verifies that it is the leaf *at the position corresponding to the requested `shard_id`*. A malicious state-sync peer can supply chunk I's header while the syncing node requests shard J, pass all validation checks, and cause the node to store shard I's `prev_state_root` as the authoritative root for shard J. Every subsequent `set_state_part` call then validates parts against shard I's state root, and `set_state_finalize` overwrites shard J's trie with shard I's state.

---

### Finding Description

**Entrypoint and call chain**

The syncing node calls `StateSyncDownloader::ensure_shard_header(shard_id=J, ...)`, downloads a `ShardStateSyncResponseHeader` from a peer, and forwards it to `ClientActor` as `StateHeaderValidationRequest { shard_id: J, header }`. The actor calls:

```
set_state_header(shard_id=J, sync_hash, header_containing_chunk_I)
``` [1](#0-0) 

**The missing position check**

Inside `set_state_header`, step 3a verifies the chunk is in the block:

```rust
if !verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
) { … }
``` [2](#0-1) 

`verify_path` only computes the Merkle root from the supplied path and compares it to `chunk_headers_root`. It does **not** check the leaf's position (shard index):

```rust
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}
pub fn verify_hash(root: MerkleHash, path: &MerklePath, item_hash: MerkleHash) -> bool {
    compute_root_from_path(path, item_hash) == root
}
``` [3](#0-2) 

`verify_path_with_index`, which does enforce position, exists but is never called here. [4](#0-3) 

**No shard-ID binding on the chunk itself**

`validate_chunk_proofs` checks internal chunk consistency (header hash, tx root, receipts root) but never compares `chunk.shard_id()` against the caller-supplied `shard_id`: [5](#0-4) 

There is no other guard in `set_state_header` that binds the chunk to the requested shard.

**Receipt proof check does not block the attack**

Step 4e computes `receipts_hash = hash(ReceiptList(shard_id=J, receipts))` and verifies it against block-committed outgoing-receipts roots: [6](#0-5) 

All inputs (`receipts`, `proof`, `root`, `block_proof`) come from the attacker. A full-node attacker can supply the *actual* incoming receipts for shard J (public blockchain data) with valid Merkle proofs, satisfying this check while still embedding chunk I's header.

**Corrupted storage and downstream effect**

After all checks pass, the header is stored under the wrong key:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
``` [7](#0-6) 

`set_state_part` then reads this header and extracts the state root:

```rust
let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
let chunk = shard_state_header.take_chunk();
let state_root = *chunk.take_header().take_inner().prev_state_root();
``` [8](#0-7) 

`state_root` is now shard I's `prev_state_root`. The attacker supplies state parts for shard I, which validate correctly against this root. `set_state_finalize` then applies shard I's trie data to shard J's slot, permanently overwriting shard J's state.

---

### Impact Explanation

A syncing node that accepts a cross-shard header installs the wrong trie for shard J. All accounts, balances, and contract state that belong to shard J are replaced with shard I's data. The node will produce incorrect execution results for shard J indefinitely, or fail to participate in consensus for that shard.

---

### Likelihood Explanation

The attacker must be a peer that the syncing node selects as a state-sync provider. In the peer-to-peer state sync path any reachable full node can serve headers; no validator or operator privilege is required. The attacker must also be a full node with access to both shards' data, which is a standard configuration. The crafted header requires only public blockchain data (chunk bodies, Merkle paths, receipt proofs) and no cryptographic forgery.

---

### Recommendation

Replace the position-unaware `verify_path` call with `verify_path_with_index`, passing the shard index derived from `shard_id` as the expected leaf position:

```rust
let shard_index = shard_layout.get_shard_index(shard_id)?;
if !verify_path_with_index(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
    shard_index as u64,
    num_shards as u64,
) { … }
```

Apply the same fix to the `prev_chunk_proof` verification. Additionally, add an explicit assertion `chunk.shard_id() == shard_id` immediately after `validate_chunk_proofs` returns.

### Proof of Concept

```rust
// Two-shard setup: shard_I = 0, shard_J = 1
// 1. Obtain the legitimate header for shard_I from a full node.
let header_for_shard_I = adapter.get_state_response_header(shard_I, sync_hash).unwrap();

// 2. Obtain valid incoming-receipt proofs for shard_J (public blockchain data).
//    Splice them into the header_for_shard_I to produce the crafted header.
let crafted_header = splice_receipts_for_shard_J(header_for_shard_I, shard_J, &chain);

// 3. Call set_state_header claiming the header is for shard_J.
//    Without the fix this succeeds; with the fix it must return Err.
adapter.set_state_header(shard_J, sync_hash, crafted_header).unwrap();

// 4. Confirm the stored state root is shard_I's, not shard_J's.
let stored = adapter.get_state_header(shard_J, sync_hash).unwrap();
let stored_root = stored.chunk_prev_state_root();
let correct_root = prev_block.chunks().get(shard_J_index).unwrap().prev_state_root();
assert_ne!(stored_root, correct_root, "shard J's state root was replaced by shard I's");

```rust
pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
