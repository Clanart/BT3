### Title
`Simulator.applyCall` Omits `MinGasMultiplier` Floor Applied in `ApplyMessageWithConfig` — Simulation Undercharges Gas and Misreports `GasUsed` - (File: x/evm/keeper/simulate.go)

---

### Summary

`ApplyMessageWithConfig` enforces a minimum gas-used floor (`minimumGasUsed = gasLimit × MinGasMultiplier`) before returning `gasUsed` and computing the actual bank refund. The parallel execution path in `Simulator.applyCall` — used by the public `eth_simulateV1` JSON-RPC endpoint — omits this floor entirely, computing `gasUsed` as the raw post-refund EVM value. This is the direct Ethermint analog of M-19: one accounting path applies a correction, the parallel path does not.

---

### Finding Description

**Primary path — `ApplyMessageWithConfig`** applies the `MinGasMultiplier` floor:

```go
// x/evm/keeper/state_transition.go:571-587
gasLimit := sdkmath.LegacyNewDec(limit)
minGasMultiplier := cfg.FeeMarketParams.MinGasMultiplier
...
minimumGasUsed := gasLimit.Mul(minGasMultiplier)
...
gasUsed = sdkmath.LegacyMaxDec(minimumGasUsed, sdkmath.LegacyNewDec(tempGasUsed)).TruncateInt().Uint64()
leftoverGas = msg.GasLimit - gasUsed
```

`ApplyTransaction` then refunds `leftoverGas × gasPrice` to the sender, so the net fee charged is `max(minimumGasUsed, actualGasUsed) × effectiveGasPrice`.

**Secondary path — `Simulator.applyCall`** computes `gasUsed` without the floor:

```go
// x/evm/keeper/simulate.go:479-486
gasUsed := msg.GasLimit - leftoverGas   // raw EVM value, no minimum floor

if msg.GasPrice.Sign() > 0 {
    cost := new(uint256.Int).SetUint64(gasUsed)
    cost.Mul(cost, gasPriceU256)
    sim.state.SubBalance(sender, cost, tracing.BalanceDecreaseGasBuy)
}
```

The sender's balance is reduced by `rawGasUsed × gasPrice`, not `max(minimumGasUsed, rawGasUsed) × gasPrice`. The `GasUsed` field in the returned `applyCallResult` and the simulation receipt is also the raw value.

The `feemarket` `EndBlock` applies the same floor to `gasWanted` for base-fee calculation:

```go
// x/feemarket/keeper/abci.go:73-74
limitedGasWanted := sdkmath.LegacyNewDec(gw).Mul(minGasMultiplier)
gasWanted = sdkmath.LegacyMaxDec(limitedGasWanted, sdkmath.LegacyNewDec(gasUsed)).TruncateInt().Uint64()
```

This confirms the floor is an intentional, protocol-level accounting rule — not an optional optimization — making its absence in `applyCall` a genuine inconsistency.

---

### Impact Explanation

With the default `MinGasMultiplier = 0.5` and a transaction with `gasLimit = 1,000,000` that actually consumes `21,000` gas (e.g., a simple ETH transfer):

| Path | `gasUsed` charged | Net fee |
|---|---|---|
| Real execution (`ApplyMessageWithConfig`) | `500,000` | `500,000 × gasPrice` |
| Simulation (`applyCall`) | `21,000` | `21,000 × gasPrice` |

The simulation undercharges by `479,000 × gasPrice` per call.

In **validation mode** (`sim.validate = true`), `applyCall` is supposed to faithfully predict real-execution outcomes. Because the sender's balance is reduced by less than real execution would reduce it, subsequent calls in a multi-call simulation block see a higher remaining balance than they would in real execution. A simulation can therefore show a sequence of calls as fully succeeding when the real block execution would fail mid-sequence due to insufficient balance — the simulation feeds incorrect data about whether transactions would commit.

The `GasUsed` values in simulation receipts are also systematically lower than real execution, causing fee mis-accounting visible to any consumer of the `eth_simulateV1` JSON-RPC endpoint.

---

### Likelihood Explanation

- `MinGasMultiplier` defaults to `0.5` and is a live feemarket parameter on all Ethermint-based chains.
- Any user calling `eth_simulateV1` with a transaction whose actual EVM gas is below `gasLimit × 0.5` (which is common for simple transfers with large gas limits) triggers the discrepancy.
- No special privileges are required; any unprivileged JSON-RPC caller can reach this path.

---

### Recommendation

Apply the same `MinGasMultiplier` floor in `Simulator.applyCall` after computing `gasUsed`, mirroring `ApplyMessageWithConfig`:

```go
// After: gasUsed := msg.GasLimit - leftoverGas
if minGasMultiplier := sim.keeper.GetFeeMarketParams(sim.state.Context()).MinGasMultiplier; !minGasMultiplier.IsNil() {
    limit, _ := ethermint.SafeInt64(msg.GasLimit)
    minimumGasUsed := sdkmath.LegacyNewDec(limit).Mul(minGasMultiplier)
    gasUsed64, _ := ethermint.SafeInt64(gasUsed)
    gasUsed = sdkmath.LegacyMaxDec(minimumGasUsed, sdkmath.LegacyNewDec(gasUsed64)).TruncateInt().Uint64()
}
```

---

### Proof of Concept

1. Deploy an Ethermint chain with `MinGasMultiplier = 0.5` (default).
2. Fund a sender with exactly `500,000 × gasPrice` wei (enough for one real execution of a 21,000-gas transfer with `gasLimit = 1,000,000`).
3. Call `eth_simulateV1` with two calls from the same sender, each with `gasLimit = 1,000,000` and `gasPrice = 1`.
4. **Simulation result**: both calls succeed — first call deducts `21,000`, leaving `479,000`; second call also deducts `21,000`, leaving `458,000`.
5. **Real execution**: first call deducts `500,000` (minimum floor), leaving `0`; second call fails with `ErrInsufficientFunds`.

The simulation incorrectly reports both calls as succeeding, feeding incorrect data about transaction viability. [1](#0-0) [2](#0-1) [3](#0-2)
