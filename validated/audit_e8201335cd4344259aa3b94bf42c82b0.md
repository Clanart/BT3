### Title
Missing `chunk.shard_id() == shard_id` Guard in `set_state_header` Allows Cross-Shard State Corruption — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for a requested `shard_id=A` that embeds a `ShardChunk` whose internal `shard_id` field is B. All five validation steps pass without ever comparing `chunk.shard_id()` to the requested `shard_id`. The corrupted header is then persisted and consumed by `set_state_finalize`, which applies shard B's transactions and receipts under shard A's `shard_uid`, producing a wrong post-sync state root for shard A.

### Finding Description

**Step 1 — `validate_chunk_proofs` does not check `shard_id`.** [1](#0-0) 

`validate_chunk_proofs` only verifies the chunk's internal hash, tx root, and outgoing-receipts root. There is no comparison of `chunk.shard_id()` to any expected value.

**Step 2 — `verify_path` is position-blind.** [2](#0-1) 

`verify_path` calls `compute_root_from_path`, which simply follows Left/Right directions to reconstruct the root. It does not verify which leaf index (i.e., which shard slot) the item occupies. A proof for shard B's chunk at position B in the `chunk_headers_root` Merkle tree reconstructs the same root as a proof for shard A's chunk at position A. Supplying shard B's chunk with shard B's proof therefore passes the check at: [3](#0-2) 

**Step 3 — No `chunk.shard_id() == shard_id` guard exists anywhere in `set_state_header`.**

The grep confirms zero occurrences of such a comparison in `adapter.rs`. After all five validation steps, the header is stored under key `StateHeaderKey(shard_id_A, sync_hash)` with shard B's chunk inside: [4](#0-3) 

**Step 4 — Receipt validation uses the requested `shard_id`, not the chunk's.** [5](#0-4) 

`ReceiptList(shard_id, receipts)` hashes with the requested shard A. The attacker supplies valid incoming receipts for shard A (public chain data), so this check passes independently of the chunk's shard_id field.

**Step 5 — `set_state_finalize` applies shard B's data under shard A's `shard_uid`.** [6](#0-5) 

`shard_uid` is derived from the requested `shard_id` (A), but `chunk_header.prev_state_root()` and all transactions/receipts come from the stored header, which contains shard B's chunk. The `apply_chunk` call therefore executes shard B's workload against shard A's trie slot, producing a wrong state root for shard A.

### Impact Explanation

A syncing node that accepts the malicious header will:
1. Download and validate state parts against shard B's `prev_state_root` (attacker-supplied, valid for shard B).
2. Apply shard B's transactions and receipts to shard A's trie.
3. Commit a post-sync state root for shard A that does not match the canonical chain.

The node becomes unable to produce or validate correct chunks for shard A, effectively ejecting it from participation in shard A consensus.

### Likelihood Explanation

Any peer in the NEAR P2P network can respond to state sync header requests — there is no cryptographic binding of state sync responses to a trusted identity. A single malicious peer that wins the race to respond to a syncing node's header request can trigger this path. The syncing node retries on validation failure, but since all five checks pass, no retry occurs.

### Recommendation

Add an explicit shard_id equality guard immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: reject cross-shard substitution
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

The same guard should be added in `set_state_finalize` as a defence-in-depth check.

### Proof of Concept

Build an integration test that:
1. Produces a two-shard chain with distinct chunks for shard 0 and shard 1.
2. Constructs a `ShardStateSyncResponseHeader` for shard 0 by taking shard 1's `ShardChunk` and shard 1's Merkle proof from `chunk_headers_root`, but using shard 0's valid incoming receipts.
3. Calls `set_state_header(shard_id=0, sync_hash, malicious_header)` and asserts it returns an error.
4. If no error is returned (confirming the bug), calls `set_state_finalize` and asserts the resulting state root differs from the canonical shard 0 state root.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
