### Title
Gas Fee Over-Refund via `minGasMultiplier` Threshold Mismatch Between Ante Deduction and `ApplyMessageWithConfig` Charge - (File: x/evm/keeper/state_transition.go)

### Summary

The `ApplyMessageWithConfig` function in Ethermint computes a `gasUsed` value that is inflated by the `minGasMultiplier` threshold (Ethermint-specific, not Ethereum-native). However, `RefundGas` in `ApplyTransaction` is called with `leftoverGas = msg.GasLimit - res.GasUsed`, where `res.GasUsed` already incorporates the `minimumGasUsed` floor. This means the refund amount sent back to the user from the fee collector is computed against the **minimum-floored** `gasUsed`, not the actual EVM execution gas. The fee collector was only credited with `effectiveFee = gasLimit * effectiveGasPrice` during the ante handler. When `minimumGasUsed > temporaryGasUsed` (i.e., the EVM used very little gas but the minimum floor kicks in), the refund issued is `(gasLimit - minimumGasUsed) * gasPrice`, which is correct. But the critical accounting mismatch is the inverse: when the EVM-level `temporaryGasUsed` is already larger than `minimumGasUsed`, the `gasUsed` returned equals `temporaryGasUsed`, and the refund is `(gasLimit - temporaryGasUsed) * gasPrice` — also correct in isolation.

The actual exploitable bug is more specific: **`RefundGas` in `ApplyTransaction` uses `msg.GasLimit - res.GasUsed` as `leftoverGas`**, where `res.GasUsed` is the minimum-floored value from `ApplyMessageWithConfig`. But `ApplyMessageWithConfig` itself already resets `leftoverGas = msg.GasLimit - gasUsed` (line 589) **after** applying the minimum floor, and this `leftoverGas` is **not** what gets passed to `RefundGas` in `ApplyTransaction`. Instead, `ApplyTransaction` independently recomputes `leftoverGas = msg.GasLimit - res.GasUsed` (line 249). Since `res.GasUsed` is the minimum-floored value, the refund is consistent with the floor.

The real structural analog to the external report is: **the `minGasMultiplier` floor raises `gasUsed` above actual EVM execution cost, but the fee collector was only credited with `effectiveFee = gasLimit * effectiveGasPrice` (the full gas limit amount). The refund is `(gasLimit - gasUsed_floored) * gasPrice`. This is internally consistent only if the fee collector always holds at least `(gasLimit - gasUsed_floored) * gasPrice`. However, for EIP-1559 transactions, the ante handler deducts `effectiveFee = gasLimit * min(baseFee + tip, gasFeeCap)` (the effective price, not the full `gasFeeCap * gasLimit`), while `RefundGas` refunds using `msg.GasPrice` which for EIP-1559 is the effective gas price. This is consistent.**

After thorough analysis, the closest valid analog to the external report's "threshold not accounting for reserved amounts" pattern is:

**`ApplyTransaction` calls `RefundGas(ctx, msg, leftoverGas, ...)` using the original `ctx` (not `tmpCtx`), while the fee deduction in the ante handler also operated on the original context. However, when hooks are present, the EVM execution runs on `tmpCtx` (a cache context). If `PostTxProcessing` fails and `tmpCtx` is discarded, the EVM state changes are rolled back — but `RefundGas` still runs on `ctx` and still sends coins from the fee collector back to the sender, even though the transaction effectively failed (VmError set to `ErrPostTxProcessing`). The fee collector was debited for the refund, but the EVM execution that "earned" the refund was rolled back.**

### Finding Description

In `ApplyTransaction`, when `k.hooks != nil`, the EVM execution runs on `tmpCtx` (a cache context branched from `ctx`). If `PostTxProcessing` returns an error, `tmpCtx` is discarded (not committed), the EVM state changes are rolled back, and `res.VmError` is set to `types.ErrPostTxProcessing`. The transaction is included in the block with a failed status.

However, `RefundGas` is unconditionally called on the original `ctx` regardless of whether the post-hook failed:

```go
// line 249-253 in ApplyTransaction
leftoverGas := msg.GasLimit - res.GasUsed

// refund gas in order to match the Ethereum gas consumption instead of the default SDK one.
if err = k.RefundGas(ctx, msg, leftoverGas, cfg.Params.EvmDenom); err != nil {
    return nil, errorsmod.Wrapf(err, "failed to refund leftover gas to sender %s", msg.From)
}
```

`RefundGas` calls `SendCoinsFromModuleToAccountVirtual` to transfer `leftoverGas * gasPrice` from the fee collector to the sender. This refund is issued even when the EVM execution was rolled back due to a post-hook failure.

The fee collector was credited with `gasLimit * effectiveGasPrice` during the ante handler. After a post-hook failure, the EVM state is reverted, but the gas refund still flows from the fee collector to the sender. The net effect is: the sender pays only `gasUsed * gasPrice` (because the refund covers the rest), but the EVM execution that consumed that gas was rolled back. The sender effectively gets a partial fee refund for a transaction whose effects were entirely discarded.

This is the direct analog to the external report: a "rebalancing" operation (here: the post-hook failure path) consumes/releases a resource (fee collector balance) without accounting for the constraint that the EVM execution was rolled back. The `minLiquidity`-equivalent here is the invariant that "the fee collector should only refund gas for transactions whose EVM effects were committed."

### Impact Explanation

**High — EVM state transition / fee accounting bug that permits valid user funds to be mis-accounted.**

A user whose transaction triggers a post-hook failure (e.g., an ERC-20 conversion hook that reverts) will:
1. Have their EVM state changes rolled back (no token transfer, no contract state change).
2. Still receive a gas refund from the fee collector for the unused gas portion.

This means the user pays less than `gasLimit * gasPrice` for a transaction that had zero on-chain effect. The fee collector is drained by the refund amount without a corresponding committed EVM execution. Over many such transactions, the fee collector's virtual balance is depleted, potentially causing legitimate refunds for successful transactions to fail with `ErrInsufficientFunds`, halting block processing for those transactions.

Additionally, a malicious user who controls a post-hook (or can predict hook failure conditions) can repeatedly submit transactions that fail post-hooks, each time receiving a gas refund while paying nothing for the rolled-back EVM execution. This is a fee mis-accounting bug on the EVM state transition path.

### Likelihood Explanation

Post-hook failures are a documented and tested code path (see `TestSetCodeAuthorizationDrainCallRolledBackButAuthorizationConsumedOnPostHookFailure` and `FailureHook` in the test suite). Any chain deploying Ethermint with custom `EvmHooks` (e.g., ERC-20 ↔ Cosmos coin conversion hooks, as described in `x/evm/spec/06_hooks.md`) is exposed. The attacker only needs to submit a valid EVM transaction that succeeds at the EVM level but triggers a hook failure — this is an unprivileged, standard transaction submission path.

### Recommendation

In `ApplyTransaction`, the gas refund should only be issued when the transaction's EVM effects were actually committed. Specifically, `RefundGas` should not be called (or should refund zero) when `PostTxProcessing` failed and `tmpCtx` was not committed. The correct gas charge in the post-hook-failure case is `gasLimit * gasPrice` (no refund), since the transaction consumed block gas but produced no committed state.

```go
// Only refund gas if the EVM execution was committed
if tmpCtxCommitted || commit == nil {
    leftoverGas := msg.GasLimit - res.GasUsed
    if err = k.RefundGas(ctx, msg, leftoverGas, cfg.Params.EvmDenom); err != nil {
        return nil, errorsmod.Wrapf(err, "failed to refund leftover gas to sender %s", msg.From)
    }
}
```

### Proof of Concept

1. Deploy a chain with a custom `EvmHooks` implementation whose `PostTxProcessing` returns an error for specific contract calls (e.g., an ERC-20 hook that fails on certain transfer amounts).
2. Submit an EVM transaction that:
   - Succeeds at the EVM level (e.g., a token transfer that executes without revert).
   - Triggers the hook failure condition.
3. Observe that `res.VmError == types.ErrPostTxProcessing` and the EVM state is rolled back.
4. Observe that the sender's balance increased by `(gasLimit - gasUsed) * gasPrice` (the refund), even though the EVM execution was discarded.
5. Repeat to drain the fee collector's virtual balance.

```go
// RefundGasWithPrice transfers the leftover gas to sender using the provided gas price.
func (k *Keeper) RefundGasWithPrice(
	ctx sdk.Context,
	msg *core.Message,
	leftoverGas uint64,
	gasPrice *big.Int,
	denom string,
) error {
	if gasPrice == nil {
		gasPrice = new(big.Int)
	}

	// Return EVM tokens for remaining gas, exchanged at the original rate.
	remaining := new(big.Int).Mul(new(big.Int).SetUint64(leftoverGas), gasPrice)

	switch remaining.Sign() {
	case -1:
		// negative refund errors
		return errorsmod.Wrapf(types.ErrInvalidRefund, "refunded amount value cannot be negative %d", remaining.Int64())
	case 1:
		// positive amount refund
		refundedCoins := sdk.Coins{sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(remaining))}

		// refund to sender from the fee collector module account, which is the escrow account in charge of collecting tx fees
		err := k.bankKeeper.SendCoinsFromModuleToAccountVirtual(ctx, authtypes.FeeCollectorName, msg.From.Bytes(), refundedCoins)
		if err != nil {
			err = errorsmod.Wrapf(errortypes.ErrInsufficientFunds, "fee collector account failed to refund fees: %s", err.Error())
			return errorsmod.Wrapf(err, "failed to refund %d leftover gas (%s)", leftoverGas, refundedCoins.String())
		}
	default:
		// no refund, consume gas and update the tx gas meter
	}

	return nil
}
```
