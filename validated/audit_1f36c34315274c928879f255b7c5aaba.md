### Title
Hardcoded `Fee(1)` Substituted for Real `paid_fee_on_l1` in RPC Re-execution and Consensus Conversion Causes Wrong Authoritative Execution Result for L1Handler Transactions - (File: `crates/apollo_rpc/src/v0_8/api/mod.rs`, `crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

Multiple code paths in the sequencer substitute a hardcoded `Fee(1)` for the real `paid_fee_on_l1` value when converting a stored or consensus-received `L1HandlerTransaction` into an executable form. Because `paid_fee_on_l1` is the Starknet analog of Solidity's `msg.value` for L1→L2 messages, using a fabricated value causes the blockifier's fee-sufficiency check to produce a wrong result: a transaction that actually paid `Fee(0)` on L1 (which should be rejected) is accepted, and the resulting execution trace, fee estimation, and state diff returned by the RPC are authoritative-looking but wrong.

### Finding Description

The original M-02 bug was: a Solidity function uses `msg.value` but is not marked `payable`, so the value is silently ignored. The Starknet analog is: `paid_fee_on_l1` is the ETH value attached to the L1 `sendMessageToL2` call and is the only guard against zero-fee L1 handler execution. Several production code paths discard the real value and substitute `Fee(1)`:

**1. RPC `traceTransaction` / `traceBlockTransactions` re-execution path**

`stored_txn_to_executable_txn` in `crates/apollo_rpc/src/v0_8/api/mod.rs` is called for every transaction in a block when the RPC re-executes it for tracing. For `L1Handler` transactions it unconditionally writes:

```rust
let paid_fee_on_l1 = Fee(1);
Ok(ExecutableTransactionInput::L1Handler(value, paid_fee_on_l1, false))
```

The comment reads: *"This is a temporary solution until we have a better way to get the l1 fee."* The real `paid_fee_on_l1` is stored in the `L1HandlerTransaction` struct in the L1 provider / storage path (see `papyrus_base_layer/src/eth_events.rs` which correctly reads `event.fee`), but it is never retrieved here.

**2. Consensus-to-internal conversion path**

`convert_consensus_l1_handler_to_internal_l1_handler` in `crates/apollo_transaction_converter/src/transaction_converter.rs` also hardcodes `Fee(1)`:

```rust
// TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
Fee(1),
```

**3. `TryFrom<(Transaction, &ChainId)>` for `executable_transaction::Transaction`**

`crates/starknet_api/src/transaction.rs` line 180 also hardcodes `Fee(1)` with the same TODO comment.

The blockifier's fee check in `execute_raw` for `L1HandlerTransaction` is:

```rust
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(...InsufficientFee...));
}
```

Because `Fee(1) != Fee(0)`, this check always passes regardless of what was actually paid on L1.

### Impact Explanation

**High – RPC execution, fee estimation, tracing, simulation returns an authoritative-looking wrong value.**

- `starknet_traceTransaction` and `starknet_traceBlockTransactions` re-execute stored L1 handler transactions with `Fee(1)` instead of the real paid fee. If the real paid fee was `Fee(0)` (which should have caused a revert), the re-execution succeeds and returns a trace showing a successful execution — a wrong authoritative result.
- `starknet_simulateTransactions` and `starknet_estimateFee` called with a stored L1 handler transaction (via `stored_txn_to_executable_txn`) will similarly produce wrong fee estimates and traces.
- The consensus conversion path means that when the batcher/blockifier executes an L1 handler received through consensus with `paid_fee_on_l1 = Fee(1)`, the fee check is bypassed for any transaction that actually paid zero fee on L1, producing a wrong execution result and wrong state diff committed to the block.

### Likelihood Explanation

Any L1 handler transaction that reaches the sequencer through the consensus path or is queried via `traceTransaction`/`traceBlockTransactions` will trigger this. An attacker who can craft an L1 message with zero ETH value (if the L1 contract allows it, or in test/devnet environments) will have their transaction accepted and traced as successful. In production, the L1 Starknet core contract enforces non-zero fee at the Ethereum level, but the sequencer's own fee check is bypassed, meaning the sequencer's execution result diverges from what the L1 contract enforces — a broken invariant.

### Recommendation

1. In `stored_txn_to_executable_txn`, retrieve the real `paid_fee_on_l1` from storage (it is stored alongside the transaction in the L1 provider storage path) instead of hardcoding `Fee(1)`.
2. In `convert_consensus_l1_handler_to_internal_l1_handler`, propagate the real `paid_fee_on_l1` from the consensus message rather than substituting `Fee(1)`.
3. In `TryFrom<(Transaction, &ChainId)>`, require `paid_fee_on_l1` as an explicit parameter rather than defaulting to `Fee(1)`.
4. Add a storage column or field to persist `paid_fee_on_l1` alongside the stored `L1HandlerTransaction` so it can be retrieved during re-execution.

### Proof of Concept

1. An L1 handler transaction is scraped from L1 with `paid_fee_on_l1 = Fee(0)` (possible in devnet or if the L1 contract check is bypassed).
2. The transaction is stored in the sequencer's storage. The stored form (`starknet_api::transaction::L1HandlerTransaction`) does **not** carry `paid_fee_on_l1` — it is only on the executable wrapper.
3. A caller invokes `starknet_traceTransaction` for that transaction hash.
4. `stored_txn_to_executable_txn` is called, substituting `Fee(1)` for the real `Fee(0)`.
5. The blockifier executes the transaction. The check `if paid_fee == Fee(0)` is **not** triggered because `paid_fee` is now `Fee(1)`.
6. The RPC returns a successful trace — an authoritative-looking wrong value — when the correct result would be a revert with `InsufficientFee`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
