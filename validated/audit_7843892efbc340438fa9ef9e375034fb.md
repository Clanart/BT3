Audit Report

## Title
Missing Zero-Address Validation on `recipient` in `unwrapWETH9` Permanently Burns User ETH - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`unwrapWETH9` accepts a caller-supplied `recipient` address and forwards all unwrapped ETH to it via `_transferETH`, which uses a low-level `.call`. No guard prevents `recipient == address(0)`. On the EVM, `address(0).call{value: v}("")` succeeds silently, so the ETH is permanently burned with no revert and no recovery path.

## Finding Description
`unwrapWETH9` (L37–45) performs no validation on `recipient` before passing it to `_transferETH(recipient, balanceWETH)`. `_transferETH` (L90–93) executes `to.call{value: value}("")`; when `to == address(0)`, the EVM returns `(true, "")`, so `ok` is `true`, the `ETHTransferFailed` revert is never triggered, and all ETH is destroyed. The canonical multicall pattern — `exactInput*(…, recipient: address(router))` followed by `unwrapWETH9(0, address(0))` — routes the entire swap output through this path. No `recipient != address(0)` guard exists anywhere in the call chain.

## Impact Explanation
Direct, irreversible loss of user ETH principal. Any ETH held by the router at the time `unwrapWETH9` is called with `recipient == address(0)` is permanently burned. This matches the allowed impact gate: direct loss of user principal with no recovery path.

## Likelihood Explanation
`recipient` is a plain `address` parameter supplied by any unprivileged caller (EOA or contract). A zero-initialized variable in an integration, a UI bug, or a copy-paste error is sufficient to trigger the loss. No privileged role, malicious setup, or non-standard token behavior is required. The function is `public payable` and callable by anyone.

## Recommendation
Add a zero-address guard at the top of `unwrapWETH9`:

```solidity
error ZeroRecipient();

function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    if (recipient == address(0)) revert ZeroRecipient();
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);
    }
}
```

Apply the same guard to `sweepToken` for defence-in-depth (though standard ERC20 implementations revert on `transfer(address(0), ...)`, making that path self-protecting for in-scope tokens).

## Proof of Concept
1. Seed a pool with token1/WETH liquidity.
2. User approves the router for token1.
3. User calls `router.multicall([...])` with:
   - `exactInputSingle({tokenIn: token1, tokenOut: WETH, recipient: address(router), amountIn: 3000, ...})`
   - `unwrapWETH9(0, address(0))`
4. First sub-call succeeds; router holds `X` WETH.
5. `unwrapWETH9` calls `IWETH9(WETH).withdraw(X)` → router holds `X` ETH.
6. `_transferETH(address(0), X)` executes `address(0).call{value: X}("")` → returns `(true, "")`.
7. `ok == true`, no revert. `X` ETH is permanently burned.
8. User receives zero ETH; `address(router).balance == 0`. [1](#0-0) [2](#0-1)
