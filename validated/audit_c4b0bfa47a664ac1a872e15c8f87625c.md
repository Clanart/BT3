### Title
Integer Overflow in `check_time_locks` via `wrapping_add` Bypasses Relative Timelocks When `nowrap=false` - (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `check_time_locks` is invoked with `nowrap = false`, the relative timelock checks for `ASSERT_SECONDS_RELATIVE` and `ASSERT_HEIGHT_RELATIVE` use `wrapping_add` on unsigned integers. An unprivileged spender can craft a CLVM condition with a near-maximum value for the relative delay that causes the addition to wrap around to a small number, making the timelock comparison evaluate to `false` and the spend to be accepted unconditionally — regardless of how recently the coin was confirmed.

---

### Finding Description

In `check_time_locks`, two branches exist depending on the `nowrap` flag:

- `nowrap = true` → uses `saturating_add` (safe, clamps at `u32::MAX` / `u64::MAX`)
- `nowrap = false` → uses `wrapping_add` (legacy path, silently overflows)

The vulnerable arithmetic for `ASSERT_HEIGHT_RELATIVE`:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

And for `ASSERT_SECONDS_RELATIVE`:

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [1](#0-0) [2](#0-1) 

When `height_relative = u32::MAX` (a valid 4-byte CLVM atom), `confirmed_block_index.wrapping_add(u32::MAX)` equals `confirmed_block_index - 1`. The check `prev_height < confirmed_block_index - 1` is always `false` for any coin that has been confirmed, so the height timelock is silently bypassed.

Identically, when `seconds_relative = u64::MAX` (a valid 9-byte CLVM atom `0x00FFFFFFFFFFFFFFFF`), `timestamp.wrapping_add(u64::MAX)` equals `timestamp - 1`. The check `current_timestamp < coin_timestamp - 1` is always `false`, bypassing the seconds timelock.

The condition parser in `conditions.rs` accepts these values as valid:

```rust
ASSERT_SECONDS_RELATIVE => {
    match sanitize_uint(a, node, 8, ...) {
        SanitizedUint::PositiveOverflow => Err(...),   // > 8 bytes → error
        SanitizedUint::NegativeOverflow => Ok(Condition::SkipRelativeCondition),
        SanitizedUint::Ok(r) => Ok(Condition::AssertSecondsRelative(r)),  // u64::MAX accepted
    }
}
``` [3](#0-2) 

`sanitize_uint` with `max_size = 8` allows a 9-byte atom with a leading zero byte, so `u64::MAX` encoded as `0x00FFFFFFFFFFFFFFFF` passes validation and is stored as `seconds_relative = u64::MAX`. [4](#0-3) 

The codebase's own tests explicitly document and confirm this wrapping behavior as a divergence between the two paths:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),
)]
``` [5](#0-4) 

The Python binding still exposes `nowrap` as a caller-controlled parameter, meaning the `nowrap = false` path remains reachable in production for blocks validated under the pre-softfork rules: [6](#0-5) 

---

### Impact Explanation

This is a **timelock validation bypass**. A coin protected by `ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE` is intended to be unspendable until a minimum time or block height has elapsed since the coin's confirmation. By supplying a crafted near-maximum delay value, an attacker causes the `wrapping_add` to produce a result smaller than the coin's own confirmation timestamp/height, making the guard comparison permanently false. The spend is accepted at any block height or timestamp — including immediately after the coin is confirmed — without the timelock ever being enforced.

This matches the allowed High impact: **timelock validation bypass enables unauthorized spend acceptance**.

---

### Likelihood Explanation

- The attacker only needs to author a CLVM puzzle that outputs `(ASSERT_SECONDS_RELATIVE . 0x00FFFFFFFFFFFFFFFF)` or `(ASSERT_HEIGHT_RELATIVE . 0x00FFFFFFFF)`.
- Both values are valid CLVM atoms that pass `sanitize_uint` without error.
- No privileged access, leaked keys, or network-level attack is required.
- The `nowrap = false` path is still present and reachable via the Python binding.
- Exploitability is conditional on whether the full node calls `check_time_locks` with `nowrap = false` for the targeted block; if the network has fully transitioned to `nowrap = true` for all new transaction blocks, new spends are not affected, but historical block re-validation paths remain vulnerable.

---

### Recommendation

1. Remove the `nowrap = false` / `wrapping_add` branch entirely from `check_time_locks`, or gate it behind a hard block-height cutoff so it is never reachable for blocks above the softfork activation height.
2. Replace all `wrapping_add` calls in timelock arithmetic with `saturating_add` unconditionally, matching the `nowrap = true` path.
3. Add a consensus-layer assertion that `nowrap` is always `true` for any block above the activation height, so the Python caller cannot accidentally pass `false`.

---

### Proof of Concept

**Setup**: Coin confirmed at block height 500, timestamp 1_000_000.

**Crafted CLVM condition**:
```
(ASSERT_SECONDS_RELATIVE . 0x00FFFFFFFFFFFFFFFF)   ; seconds_relative = u64::MAX
```

**Validation with `nowrap = false`**:
```
unspent.timestamp.wrapping_add(u64::MAX)
= 1_000_000u64.wrapping_add(18_446_744_073_708_551_615u64)
= 999_999   (wraps around)

check: current_timestamp < 999_999
     = 1_000_001 < 999_999
     = false  → no error returned → timelock bypassed
```

The spend is accepted at timestamp 1_000_001 (one second after confirmation), even though the puzzle author intended a delay of `u64::MAX` seconds. The same construction works for `ASSERT_HEIGHT_RELATIVE` using `height_relative = u32::MAX`.
