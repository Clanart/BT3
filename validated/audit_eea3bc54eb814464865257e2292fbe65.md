### Title
`revert_state_diff` unconditionally deletes `stateless_compiled_class_hash_v2` entries for migrated classes, breaking the invariant that declared classes must retain their v2 hash - (`crates/apollo_storage/src/state/mod.rs`)

---

### Summary

`revert_state_diff` calls `delete_compiled_class_hashes_v2` for **every** class in `thin_state_diff.class_hash_to_compiled_class_hash`. That set includes both newly-declared classes **and** classes whose CASM hash was migrated from v1 (Poseidon) to v2 (Blake) in the reverted block. For migrated classes the v2 entry in `stateless_compiled_class_hash_v2` was written during the **original declaration block**, not the migration block. Reverting the migration block therefore silently destroys a live, cross-block record, leaving the class declared but without a v2 hash. Any subsequent execution that calls `get_compiled_class_hash_v2` for that class will receive `StateError::MissingCompiledClassHashV2` and fail.

---

### Finding Description

**Write path – original declaration (block N)**

In `native_blockifier/src/storage.rs` only classes that are `class_undeclared` are pushed into `undeclared_casm_contracts`, and only those classes receive a call to `set_executable_class_hash_v2`:

```
for (class_hash, casm_contract_class, compiled_class_hash_v2) in undeclared_casm_contracts {
    append_txn = append_txn.append_casm(&class_hash, &casm_contract_class)?;
    append_txn = append_txn.set_executable_class_hash_v2(&class_hash, compiled_class_hash_v2)?;
}
``` [1](#0-0) 

Migrated classes are already declared, so they are **not** in `undeclared_casm_contracts` and `set_executable_class_hash_v2` is **never** called for them during the migration block.

**Write path – migration (block M)**

Migrated class hashes are inserted directly into `thin_state_diff.class_hash_to_compiled_class_hash`:

```
for (class_hash, compiled_class_hash) in migrated_class_hash_to_compiled_class_hash {
    thin_state_diff.class_hash_to_compiled_class_hash.insert(class_hash, compiled_class_hash);
}
``` [2](#0-1) 

`append_state_diff` then writes `declared_classes_block_table` only if `not_declared` (first-declaration guard), so migrated classes do not update that table either. [3](#0-2) 

**Revert path – block M reverted**

`revert_state_diff` calls `delete_compiled_class_hashes_v2` over **all** keys of `class_hash_to_compiled_class_hash`, which now includes the migrated class hashes:

```
delete_compiled_class_hashes_v2(
    &self.txn,
    thin_state_diff.class_hash_to_compiled_class_hash.keys(),
    &compiled_class_hash_v2_table,
)?;
``` [4](#0-3) 

`delete_compiled_class_hashes_v2` performs an unconditional delete with no block-number guard:

```
fn delete_compiled_class_hashes_v2(...) -> StorageResult<()> {
    for class_hash in class_hashes {
        compiled_class_hash_v2_table.delete(txn, class_hash)?;
    }
    Ok(())
}
``` [5](#0-4) 

The `stateless_compiled_class_hash_v2` table is a flat key→value store (no block-number dimension):

```
fn set_executable_class_hash_v2(self, class_hash: &ClassHash, ...) -> StorageResult<Self> {
    let table = self.open_table(&self.tables.stateless_compiled_class_hash_v2)?;
    table.upsert(&self.txn, class_hash, &executable_class_hash_v2)?;
    Ok(self)
}
``` [6](#0-5) 

Because the table has no block-number key, the revert code cannot distinguish "v2 hash written in block M" from "v2 hash written in block N". It deletes the entry regardless of origin.

**Contrast with the correct pattern**

`delete_declared_classes_block` correctly guards the delete with a block-number equality check:

```
if class_block_entry == block_number {
    declared_classes_block_table.delete(txn, class_hash)?;
}
``` [7](#0-6) 

`delete_compiled_class_hashes_v2` has no equivalent guard.

**Post-revert execution failure**

`ApolloReader::get_compiled_class_hash_v2` calls `get_executable_class_hash_v2`, which reads from `stateless_compiled_class_hash_v2`. If the entry is absent it returns `None`, and the caller converts that to `StateError::MissingCompiledClassHashV2`: [8](#0-7) 

Any transaction that executes the affected class after the revert will fail with this error even though the class is still validly declared.

---

### Impact Explanation

After reverting a migration block, every class that was migrated (but originally declared in an earlier block) loses its entry in `stateless_compiled_class_hash_v2`. The class remains present in `declared_classes_block_table` (correct), but the v2 hash is gone. Subsequent blockifier execution that calls `get_compiled_class_hash_v2` for such a class receives `MissingCompiledClassHashV2` and aborts the transaction. This produces a wrong execution result for a validly accepted transaction — matching **Critical: Wrong state or revert result from blockifier/syscall/execution logic for accepted input**.

---

### Likelihood Explanation

The condition is: (a) a block containing `migrated_compiled_classes` is committed, then (b) that block is reverted. Block reverts occur during normal reorganisation handling. The migration feature (`migrate_classes_to_v2_casm_hash`) is an active OS feature. Both conditions are reachable in production without any privileged access.

---

### Recommendation

Mirror the block-number guard used by `delete_declared_classes_block`. Before deleting from `stateless_compiled_class_hash_v2`, check whether the class was first declared in the block being reverted (i.e., `declared_classes_block_table` returns `block_number` for that class). Only delete the v2 entry if the class was newly declared in the reverted block, not merely migrated:

```rust
fn delete_compiled_class_hashes_v2<'env>(
    txn: &'env DbTransaction<'env, RW>,
    thin_state_diff: &ThinStateDiff,
    declared_classes_block_table: &'env DeclaredClassesBlockTable<'env>,
    compiled_class_hash_v2_table: &'env StatelessCompiledClassHashV2Table<'env>,
    block_number: BlockNumber,
) -> StorageResult<()> {
    for class_hash in thin_state_diff.class_hash_to_compiled_class_hash.keys() {
        // Only delete if the class was first declared in this block (not just migrated).
        if let Some(declared_block) = declared_classes_block_table.get(txn, class_hash)? {
            if declared_block == block_number {
                compiled_class_hash_v2_table.delete(txn, class_hash)?;
            }
        }
    }
    Ok(())
}
```

---

### Proof of Concept

1. **Block N**: Declare class `C`. `set_executable_class_hash_v2(C, hash_v2)` is called. `stateless_compiled_class_hash_v2[C] = hash_v2`. `declared_classes_block_table[C] = N`.

2. **Block M > N**: OS migrates class `C` from v1 to v2 CASM hash. `migrated_compiled_classes = {C: hash_v2}`. This is merged into `thin_state_diff.class_hash_to_compiled_class_hash`. `set_executable_class_hash_v2` is **not** called (class already declared). `stateless_compiled_class_hash_v2[C]` remains `hash_v2` (unchanged). `declared_classes_block_table[C]` remains `N` (unchanged, because `not_declared` is false).

3. **Revert block M**: `revert_state_diff(M)` is called. `delete_compiled_class_hashes_v2` iterates over `{C}` (from block M's `class_hash_to_compiled_class_hash`). It deletes `stateless_compiled_class_hash_v2[C]`. No block-number check is performed.

4. **Post-revert state**: `declared_classes_block_table[C] = N` (class still declared). `stateless_compiled_class_hash_v2[C]` = **absent** (incorrectly deleted). Invariant broken.

5. **Next transaction using class C**: Blockifier calls `get_compiled_class_hash_v2(C)` → `get_executable_class_hash_v2(C)` → `None` → `StateError::MissingCompiledClassHashV2(C)`. Transaction fails with a wrong result.
