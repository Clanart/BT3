### Title
Permanent Freezing of User Funds via Reverting ETH Recipient in `finTransfer` — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol::finTransfer` unconditionally reverts when the native-ETH delivery to `payload.recipient` fails. Because the signed payload fixes the recipient address and there is no cancel/refund path on the NEAR side, a user whose recipient contract rejects ETH will have their locked NEAR-side tokens frozen permanently.

---

### Finding Description

`finTransfer` is the EVM-side finalisation entry point for cross-chain transfers originating on NEAR. The function first marks the destination nonce as consumed, then dispatches the asset:

```solidity
// OmniBridge.sol line 287
completedTransfers[payload.destinationNonce] = true;

// ... signature verification ...

// lines 317-322
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [1](#0-0) 

If `payload.recipient` is a contract whose `receive()` reverts, the low-level call returns `success = false`, and the function reverts with `FailedToSendEther()`. Because the entire transaction reverts, the `completedTransfers` write at line 287 is also rolled back — the nonce is never consumed. Every subsequent relay attempt with the same MPC-signed payload (which encodes the same recipient) will hit the identical revert path.

The recipient address is embedded in the signed `TransferMessagePayload`:

```solidity
struct TransferMessagePayload {
    ...
    address recipient;   // fixed at signing time
    ...
}
``` [2](#0-1) 

No relayer or user can alter the recipient without invalidating the MPC signature. There is no `cancel_transfer` or refund function visible in the NEAR bridge contract (`near/omni-bridge/src/lib.rs`); `init_transfer` locks tokens and the only unlock path is a successful `fin_transfer` callback. Consequently, if EVM finalisation is permanently blocked, the NEAR-side locked balance is irrecoverable.

The same revert-on-failure pattern applies to the ERC-1155 branch, where `safeTransferFrom` calls `onERC1155Received` on the recipient contract and reverts if the callback is absent or reverts:

```solidity
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,
    multiToken.tokenId,
    payload.amount,
    ""
);
``` [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds.**

A user whose bridged ETH (or ERC-1155) transfer targets a contract that rejects the delivery will have their NEAR-side locked tokens frozen indefinitely. The bridge provides no escape hatch: the nonce is never consumed (so the transfer is not "completed"), but the NEAR lock cannot be reversed either. The full bridged amount is permanently unclaimable.

---

### Likelihood Explanation

**Medium.** Realistic triggering scenarios include:

1. **Accidental mis-specification** — a user supplies a smart-wallet or multisig address that does not implement a payable `receive()`.
2. **Contract upgrade** — the recipient contract is upgraded between transfer initiation and finalisation to one that rejects ETH.
3. **Deliberate self-grief** — an attacker intentionally locks their own funds to demonstrate the protocol has no recovery path (reputational/protocol-level harm).
4. **ERC-1155 recipient without callback** — any contract lacking `IERC1155Receiver` support permanently blocks that transfer.

No privileged access is required; the attack path is fully reachable by an ordinary bridge user.

---

### Recommendation

1. **Introduce a pull-payment / escrow fallback**: if the low-level ETH call fails, store the amount in a per-recipient claimable mapping rather than reverting. The recipient can then call a `claimFailed(nonce)` function.
2. **Add a NEAR-side cancellation path**: allow the original sender to reclaim locked tokens after a timeout if the destination nonce is never finalised.
3. **Validate recipient at initiation**: on the NEAR side, reject transfers whose EVM recipient is a known non-payable contract, or require the user to acknowledge contract recipients explicitly.

---

### Proof of Concept

1. Attacker (or naive user) deploys `RejectETH` on EVM:
   ```solidity
   contract RejectETH { receive() external payable { revert(); } }
   ```
2. User initiates a NEAR → EVM native-ETH transfer specifying `RejectETH` as recipient. NEAR locks the user's tokens.
3. MPC signs the `TransferMessagePayload` with `recipient = address(RejectETH)`.
4. Relayer calls `OmniBridge.finTransfer(sig, payload)`.
5. Line 319: `RejectETH.receive()` reverts → `success = false` → `revert FailedToSendEther()`.
6. The entire transaction reverts; `completedTransfers[nonce]` is rolled back.
7. Every retry with the same signed payload produces the same revert.
8. The user's NEAR-side locked balance has no recovery path — funds are permanently frozen. [4](#0-3)
