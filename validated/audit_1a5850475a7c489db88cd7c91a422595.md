The `MerkleBlob::insert()` function has explicit duplicate-key and duplicate-hash guards, but `MerkleBlob::batch_insert()` bypasses those guards for all items beyond the first two, directly writing unchecked leaves into the blob via `insert_entry_to_blob()`. This is structurally identical to the ERC1155 report: one entry-point enforces the invariant, the batch entry-point silently skips it.

### Title
`MerkleBlob::batch_insert()` skips duplicate-key and duplicate-hash validation present in `insert()`, corrupting the DataLayer Merkle tree - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::insert()` guards against duplicate `KeyId` and duplicate leaf `Hash` before writing to the blob. `MerkleBlob::batch_insert()` only routes the last two items through `insert()` when the tree has ≤1 leaves; all remaining items are written directly via `insert_entry_to_blob()` with no duplicate checks. Supplying a batch containing a repeated key or hash silently corrupts the blob and the in-memory `BlockStatusCache`, producing an inconsistent Merkle tree whose root hash and proofs of inclusion/exclusion are wrong.

### Finding Description

**`insert()` — has both guards:**

```rust
pub fn insert(...) -> Result<TreeIndex, Error> {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(hash) {
        return Err(Error::HashAlreadyPresent());
    }
    ...
}
``` [1](#0-0) 

**`batch_insert()` — guards are absent for the bulk path:**

```rust
pub fn batch_insert(
    &mut self,
    mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
) -> Result<(), Error> {
    if self.block_status_cache.leaf_count() <= 1 {
        for _ in 0..2 {
            let Some(((key, value), hash)) = keys_values_hashes.pop() else { ... };
            self.insert(key, value, &hash, InsertLocation::Auto {})?; // ← checked
        }
    }

    for ((key, value), hash) in keys_values_hashes {   // ← NO duplicate check
        let new_leaf_index = self.get_new_index();
        let new_block = Block { node: Node::Leaf(LeafNode { hash, key, value, ... }), ... };
        self.insert_entry_to_blob(new_leaf_index, &new_block)?; // ← writes unchecked
        indexes.push(new_leaf_index);
    }
    ...
}
``` [2](#0-1) 

When `leaf_count > 1` (the common production case), **every** item in the batch bypasses the guards. When `leaf_count ≤ 1`, only the last two items are checked; all earlier items in the `Vec` are unchecked.

`insert_entry_to_blob()` calls `block_status_cache.add_leaf()`, which performs a plain `HashMap::insert` — silently overwriting the previous mapping for a duplicate key or hash:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // overwrites silently
    self.leaf_hash_to_index.insert(leaf.hash, index); // overwrites silently
}
``` [3](#0-2) 

After the call:
- The raw blob contains **two leaf nodes** with the same `KeyId` at different block indexes.
- The cache (`key_to_index`) only tracks the **last** one; the first is orphaned.
- `check_integrity()` detects the divergence via `leaf_count != key_to_index_cache_length`, but only after the corruption has already been committed. [4](#0-3) 

### Impact Explanation

The corrupted blob produces an incorrect Merkle root hash. Any proof of inclusion generated for the orphaned duplicate leaf is unverifiable; any proof of exclusion for a key that is actually present (but whose cache entry was overwritten) will falsely succeed. This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

### Likelihood Explanation

`batch_insert` is a public method exposed through the Python bindings (`MerkleBlob.batch_insert`). [5](#0-4) 

Any caller — including DataLayer sync code that processes peer-supplied key/value sets — can pass a `Vec` containing a repeated `KeyId` or `Hash`. No privilege is required; the attacker only needs to supply a crafted batch to a node that calls `batch_insert`.

### Recommendation

Add the same duplicate guards at the top of `batch_insert()` that exist in `insert()`, checking every `(key, hash)` pair in the incoming batch against the current `block_status_cache` before any writes begin:

```rust
pub fn batch_insert(
    &mut self,
    mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
) -> Result<(), Error> {
+   for ((key, _value), hash) in &keys_values_hashes {
+       if self.block_status_cache.contains_key(*key) {
+           return Err(Error::KeyAlreadyPresent());
+       }
+       if self.block_status_cache.contains_leaf_hash(hash) {
+           return Err(Error::HashAlreadyPresent());
+       }
+   }
    ...
}
```

Additionally, check for duplicates **within** the batch itself (a batch may contain two entries with the same key even if neither is already in the tree).

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

let mut blob = MerkleBlob::new(vec![]).unwrap();

// Pre-populate so leaf_count > 1, bypassing the insert() path entirely
blob.insert(KeyId(0), ValueId(0), &Hash(Bytes32::new([0u8; 32])), InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(1), ValueId(1), &Hash(Bytes32::new([1u8; 32])), InsertLocation::Auto {}).unwrap();

// Now batch_insert with a duplicate key — no error is returned
let dup_key = KeyId(0);
let batch = vec![
    ((dup_key, ValueId(99)), Hash(Bytes32::new([99u8; 32]))),
];
blob.batch_insert(batch).unwrap(); // succeeds — should have returned KeyAlreadyPresent

// The blob now has two leaves with KeyId(0); the cache only tracks one.
// check_integrity() will fail with IntegrityKeyToIndexCacheLength.
blob.check_integrity().unwrap_err(); // panics/errors — tree is corrupted
```

The root hash computed after `calculate_lazy_hashes()` reflects the duplicate leaf, making it diverge from any honest node's root, and any proof of inclusion/exclusion derived from this blob is invalid.
