### Title
Reverted L1-Handler Transactions Leave ETH Permanently Frozen in L1 Core Contract — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

When an L1→L2 message is sent, ETH is locked in the L1 StarkNet core contract. The OS is responsible for recording consumed messages in `outputs.messages_to_l2`, which signals the L1 contract to release/mark the message. However, when `execute_l1_handler_transaction` marks a transaction as reverted, it returns early without calling `consume_l1_to_l2_message`, and no cancellation or refund path exists. Because `non_reverting_select_execute_entry_point_func` asserts `is_reverted = 0` (causing proof failure if the handler would actually revert), the sequencer is forced to pre-mark any unexecutable L1 handler as reverted. The result is that ETH paid for L1→L2 messages targeting a broken or upgraded contract is permanently frozen with no recovery mechanism in the OS.

---

### Finding Description

**Root cause — three cooperating code facts:**

**Fact 1:** `check_is_reverted` is a no-op.

In `execution_constraints.cairo`:
```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [1](#0-0) 

There is zero protocol-level constraint on the `is_reverted` value for L1 handler transactions. It is set entirely by the sequencer hint `%{ IsReverted %}`.

**Fact 2:** `non_reverting_select_execute_entry_point_func` asserts `is_reverted = 0`.

```cairo
let (is_reverted, retdata_size, retdata, is_deprecated) = select_execute_entry_point_func{...}(...);
assert is_reverted = 0;
``` [2](#0-1) 

This means if the L1 handler execution would actually revert (entry point not found, out-of-gas, panic), the proof itself fails. The sequencer therefore **cannot** include such a transaction with `is_reverted = FALSE`.

**Fact 3:** When `is_reverted != FALSE`, `execute_l1_handler_transaction` returns early without consuming the message.

```cairo
local is_reverted;
%{ IsReverted %}
// Skip the execution step for reverted transaction.
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
// ...
// Consume L1-to-L2 message.
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
``` [3](#0-2) 

`consume_l1_to_l2_message` is the only place that writes the message to `outputs.messages_to_l2`: [4](#0-3) 

If the message is not written to `outputs.messages_to_l2`, the L1 StarkNet core contract never marks it as consumed, and the ETH locked for that message has no recovery path in the OS.

**Entry point not found returns `is_reverted=1`**, confirming that a missing L1 handler selector causes the execution path that would fail the proof: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

ETH locked in the L1 StarkNet core contract for an L1→L2 message is permanently irrecoverable if the target contract's L1 handler is unexecutable. The OS has no cancel, timeout, or refund path for such messages. The `outputs.messages_to_l2` segment is the sole mechanism by which the L1 contract learns a message was processed; if a message never appears there, the ETH is frozen forever.

---

### Likelihood Explanation

**Medium.** The trigger requires a contract whose L1 handler becomes unexecutable after an L1→L2 message is already queued. This is reachable by an unprivileged **contract deployer** (explicitly listed as an in-scope role):

1. Deployer deploys a contract advertising an L1 handler (e.g., a bridge or token mint).
2. Users send L1→L2 messages with ETH locked in the L1 core contract.
3. Deployer calls `replace_class` (an unprivileged syscall available to any contract owner) to swap the contract's class to one that has no L1 handler entry point.
4. The sequencer cannot execute the pending L1 handlers (proof would fail via `assert is_reverted = 0`).
5. The sequencer is forced to mark them as reverted; `consume_l1_to_l2_message` is never called.
6. ETH is permanently frozen.

Additionally, any contract with a buggy L1 handler that always panics or runs out of gas creates the same outcome for any user who sends it an L1→L2 message.

---

### Recommendation

- **Short term:** Add a cancellation/refund output type to `OsCarriedOutputs` for reverted L1 handler transactions, so the L1 contract can refund the ETH to the original sender when a message is marked reverted.
- **Long term:** Model and document all L1→L2 message lifecycle states (pending, consumed, cancelled/refunded) and enforce state-transition completeness in the OS. Implement test cases covering the reverted-L1-handler path to validate that no ETH is permanently stranded.

---

### Proof of Concept

1. Deploy contract `BridgeV1` with a valid `@l1_handler` entry point selector `0xABC`.
2. User sends L1→L2 message to `BridgeV1` via the L1 StarkNet core contract, locking 1 ETH.
3. Before the sequencer processes the message, deployer calls `replace_class(BridgeV2_class_hash)` where `BridgeV2` has no `@l1_handler` entry point.
4. Sequencer attempts to include the L1 handler transaction:
   - `is_reverted = FALSE` path: `select_execute_entry_point_func` returns `is_reverted=1` (entry point not found, `execute_entry_point.cairo:172-176`); `non_reverting_select_execute_entry_point_func` asserts `is_reverted = 0` → proof fails.
   - Sequencer is forced to set `is_reverted = TRUE` (hint `%{ IsReverted %}`).
5. `execute_l1_handler_transaction` hits the early-return branch at line 387-389; `consume_l1_to_l2_message` is never called.
6. `outputs.messages_to_l2` does not contain this message; the L1 contract never marks it consumed.
7. The 1 ETH is permanently locked in the L1 StarkNet core contract with no recovery path.
