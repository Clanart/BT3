I need to carefully trace the validation logic in `set_state_header` and `set_state_finalize` to determine whether the cross-shard header substitution attack is feasible.

### Title
Wrong-Shard Header Accepted in `set_state_header` Due to Missing `chunk.shard_id()` Check and Non-Positional Merkle Proof — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` whose embedded chunk belongs to shard X while the caller supplies `shard_id=Y`. Two independent guards that should prevent this are absent: (1) no check that `chunk.shard_id() == shard_id`, and (2) the Merkle-inclusion check uses `verify_path` (position-blind) instead of `verify_path_with_index` (position-enforcing). The accepted header is stored under `StateHeaderKey(shard_Y, sync_hash)`. When `set_state_finalize` later reads it, it applies shard X's chunk under `shard_uid=Y`, writing shard X's post-state root into shard Y's `ChunkExtra`.

### Finding Description

**Guard 1 — missing `chunk.shard_id() == shard_id` check**

`set_state_header` receives `shard_id` (the target shard, Y) and `shard_state_header` (attacker-supplied). It extracts the embedded chunk but never asserts that `chunk.shard_id() == shard_id`. The only place `shard_id` is used is to key the storage write:

```
key = StateHeaderKey(shard_id, sync_hash)   // shard_id = Y
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
```

So shard X's chunk is silently stored under shard Y's key. [1](#0-0) 

**Guard 2 — `verify_path` does not enforce position**

The Merkle-inclusion check is:

```rust
verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
)
``` [2](#0-1) 

`verify_path` only checks `compute_root_from_path(path, hash(item)) == root`. It does not verify which leaf index the path corresponds to. [3](#0-2) 

The position-enforcing variant `verify_path_with_index` exists and calls `verify_path_matches_index` before `verify_path`, but it is not used here. [4](#0-3) 

A valid Merkle proof for shard X's chunk at leaf index X will satisfy `verify_path` against the same `chunk_headers_root` regardless of what `shard_id` the caller claims. The proof's Left/Right directions encode position X, but `verify_path` never checks that those directions correspond to position Y.

**Propagation through `set_state_finalize`**

`set_state_finalize` retrieves the stored header by `(shard_Y, sync_hash)` — which is now shard X's chunk — and passes it directly to `chain_update.set_state_finalize(shard_id=Y, ...)`: [5](#0-4) 

Inside `chain_update::set_state_finalize`, `shard_uid` is derived from `shard_id=Y`, but `chunk_header.prev_state_root()` comes from shard X's chunk: [6](#0-5) 

`apply_chunk_postprocessing` then writes the result keyed by `shard_uid=Y`: [7](#0-6) 

Shard Y's `ChunkExtra` now contains shard X's post-state root.

### Impact Explanation

Every subsequent block that processes shard Y reads `ChunkExtra` for shard Y to obtain the previous state root. With shard X's post-state root installed there, `apply_chunk` for shard Y will attempt to open shard X's trie under shard Y's `ShardUId`, producing either a trie-not-found error (halting shard Y processing on the synced node) or, if shard X's trie data was also installed, silently executing shard Y's transactions against shard X's state — a complete state-root corruption for shard Y on the affected node.

### Likelihood Explanation

The attack requires a malicious peer that:
1. Participates in the network (no special privilege).
2. Tracks shard X and can supply a valid `state_root_node` for shard X's `prev_state_root` and valid state parts for shard X.
3. Constructs valid incoming-receipt proofs for shard Y (all public chain data).

The syncing node downloads the header from any peer via `StateSyncDownloader::ensure_shard_header` and forwards it verbatim to `set_state_header` with the requested `shard_id=Y`. [8](#0-7) 

No validator or operator privilege is required.

### Recommendation

1. **Add an explicit shard-id check** in `set_state_header` immediately after extracting the chunk:
   ```rust
   if chunk.shard_id() != shard_id {
       return Err(Error::Other("set_shard_state failed: chunk shard_id mismatch".into()));
   }
   ``` [9](#0-8) 

2. **Replace `verify_path` with `verify_path_with_index`** for the chunk-inclusion check, passing the shard's index in the epoch's shard layout as the expected leaf position. This enforces that the proven chunk is at the position corresponding to `shard_id`, not merely somewhere in the tree. [2](#0-1)
