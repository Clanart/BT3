### Title
`NewDynamicFeeChecker` Integer-Division Truncation Produces Zero `effectiveFee`, Allowing Fee-Free Cosmos SDK Transactions - (File: `ante/evm/fee_checker.go`)

### Summary

`NewDynamicFeeChecker` in `ante/evm/fee_checker.go` computes the per-gas fee cap via integer division (`fee / gas`). When the total fee amount is smaller than the gas limit, this truncates to zero. With `baseFee = 0` (the state when `NoBaseFee = true` and London is enabled), the EIP-1559 effective price formula also collapses to zero, so the returned `effectiveFee` is zero. The downstream `DeductFeeDecorator` then skips deduction entirely. A user who submits a Cosmos SDK transaction with any non-zero fee smaller than the gas limit pays nothing.

### Finding Description

In `ante/evm/fee_checker.go`, `NewDynamicFeeChecker` derives the per-gas fee cap from the total fee and gas limit using integer (truncating) division:

```go
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))   // line 83 – truncates
``` [1](#0-0) 

When `fee < gas` (e.g., `fee = 1000`, `gas = 200000`), `feeCap` truncates to `0`.

The guard that follows only rejects the transaction when `feeCap < baseFeeInt`:

```go
if feeCap.LT(baseFeeInt) { ... }   // 0 < 0 is false → passes
``` [2](#0-1)

### Impact Explanation

Any unprivileged user can submit a Cosmos SDK transaction (governance, staking, IBC, bank send, etc.) with a non-zero fee smaller than the gas limit and have **zero fees deducted** from their balance. The fee collector module receives nothing. This is a direct mis-accounting of user funds in the ante handler, matching the allowed High impact: *"ante handler bug that permits valid user funds/fees to be mis-accounted."*

### Likelihood Explanation

`NoBaseFee = true` is a documented, governance-settable parameter explicitly designed for chains that want a static or zero base fee. The `DynamicFeeChecker` option is the recommended configuration for EIP-1559-enabled Ethermint chains. `MinGasPrice` defaults to `0`, and even when set to a sub-unit value (e.g., `0.001 aphoton/gas`), the required fee satisfies `MinGasPriceDecorator` while still being less than the gas limit, triggering the truncation. The attacker needs only to craft a standard Cosmos SDK tx with `fee < gas`.

### Recommendation

Replace the truncating integer division with a ceiling division or reject the transaction when `feeCap` rounds to zero:

```go
// Option A: reject when feeCap truncates to zero
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))
if feeCap.IsZero() && !fee.IsZero() {
    return nil, 0, errorsmod.Wrapf(errortypes.ErrInsufficientFee,
        "fee amount too small to cover gas: fee %s, gas %d", fee, gas)
}

// Option B: use ceiling division so feeCap >= 1 whenever fee > 0
feeCap := fee.Add(sdkmath.NewIntFromUint64(gas - 1)).Quo(sdkmath.NewIntFromUint64(gas))
```

Additionally, when `baseFee = 0` and `feeCap = 0`, the checker should not silently return a zero `effectiveFee`; it should either reject the tx or fall back to `checkTxFeeWithValidatorMinGasPrices`.

### Proof of Concept

**Preconditions:**
- London hardfork enabled
- `NoBaseFee = true` (governance parameter)
- `DynamicFeeChecker = true` (ante handler option)
- `MinGasPrice = 0` (default) or any value `< 1 aphoton/gas`

**Attack:**
1. Attacker constructs a Cosmos SDK tx (e.g., `MsgSend`) with `gas = 200000` and `fee = 100 aphoton`.
2. `MinGasPriceDecorator` passes (100 ≥ 0 × 200000).
3. `NewDynamicFeeChecker` computes `feeCap = 100 / 200000 = 0`.
4. `baseFeeInt = 0` (because `NoBaseFee = true`).
5. Guard `0 < 0` is false → no rejection.
6. `effectivePrice = min(MaxInt64 + 0, 0) = 0`.
7. `effectiveFee = 0 × 200000 = 0`.
8. `checkDeductFee` skips deduction because `fee.IsZero() = true`.
9. Tx executes; attacker's balance is not reduced by 100 aphoton.
