### Title
Missing `set_controller` in `OmniToken` Permanently Locks Bridged Assets on Bridge Migration — (`near/omni-token/src/lib.rs`, `near/token-deployer/src/lib.rs`)

### Summary

`OmniToken` stores a single immutable `controller` (set to the omni-bridge account at deployment) that gates all `mint`, `burn`, and metadata operations. No `set_controller` function exists anywhere in the contract. If the omni-bridge ever needs to migrate to a new NEAR account ID, every previously deployed `OmniToken` becomes permanently inoperable: the new bridge address cannot mint tokens to recipients or burn tokens from senders, irrecoverably freezing all bridged assets that flow through those tokens.

### Finding Description

**Token deployment flow**

`near/token-deployer/src/lib.rs` `deploy_token` (line 60–73) is called by the omni-bridge. Inside the function call that initialises the new `OmniToken`, the `controller` argument is hardcoded to `env::predecessor_account_id()` — the omni-bridge account at call time:

```rust
json!({"controller": env::predecessor_account_id(), "metadata": metadata})
``` [1](#0-0) 

**`OmniToken` controller is immutable**

`near/omni-token/src/lib.rs` stores the controller in a plain field and exposes no setter:

```rust
pub struct OmniToken {
    controller: AccountId,   // set once in new(), never changed
    ...
}
``` [2](#0-1) 

Every privileged operation — `mint`, `burn`, `set_metadata`, `set_withdraw_relayer_address`, `attach_full_access_key` — is gated by `assert_controller()`:

```rust
fn assert_controller(&self) {
    require!(caller == self.controller, ...);
}
``` [3](#0-2) 

There is no `set_controller` function anywhere in `lib.rs` or the `omni_ft` trait definitions. [4](#0-3) 

**The existing `update_tokens_controller` does not help**

The omni-bridge exposes `update_tokens_controller` (line 1678–1688), but it calls `set_controller_for_tokens` on an external **legacy bridge token factory** (`ext_bridge_token_facory`), not on `OmniToken` instances deployed by the token-deployer. `OmniToken` does not implement `set_controller_for_tokens`. [5](#0-4) 

### Impact Explanation

If the omni-bridge must migrate to a new NEAR account ID (e.g., after a key compromise, a critical bug requiring redeployment, or an architectural upgrade that cannot be done in-place), every `OmniToken` deployed by the token-deployer retains `controller = old_bridge_account`. The new bridge account cannot call `mint` or `burn` on any of those tokens. Incoming cross-chain transfers cannot be finalised (mint reverts), and outgoing transfers cannot be completed (burn reverts). All user funds locked in or destined for those tokens are permanently frozen with no recovery path.

This matches: **Critical — Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

### Likelihood Explanation

NEAR contract upgrades are normally done in-place (same account ID), so the controller address is unchanged in routine upgrades. However, migration to a new account is a realistic operational scenario — it is the standard response to a compromised account key, a storage-layout breaking change, or a governance-mandated redeployment. The omni-bridge already has `migrate_deployed_token` (line 1604–1664) showing that migration scenarios are anticipated by the protocol designers, yet the corresponding controller-update path for `OmniToken` is absent. [6](#0-5) 

### Recommendation

Add a `set_controller` function to `OmniToken`, callable only by the current `controller`:

```rust
pub fn set_controller(&mut self, new_controller: AccountId) {
    self.assert_controller();
    self.controller = new_controller;
}
```

Additionally, add a corresponding batch helper (e.g., `set_controller_for_tokens`) to the token-deployer so the omni-bridge's `update_tokens_controller` can reach OmniToken instances, mirroring the existing mechanism for legacy factory tokens. [7](#0-6) 

### Proof of Concept

1. Omni-bridge (account `omni-bridge.near`) calls `deploy_token` on the token-deployer.
2. Token-deployer deploys `OmniToken` with `controller = "omni-bridge.near"`.
3. A security incident forces redeployment of the bridge to `omni-bridge-v2.near`.
4. `omni-bridge-v2.near` calls `mint` on any previously deployed `OmniToken` to finalise an incoming transfer.
5. `assert_controller` fires: `"omni-bridge-v2.near" != "omni-bridge.near"` → panic.
6. All incoming cross-chain transfers to NEAR for those tokens revert permanently; all user funds are frozen.

    #[access_control_any(roles(Role::DAO))]
    #[payable]
    pub fn migrate_deployed_token(
        &mut self,
        origin_chain: ChainKind,
        old_token: AccountId,
        new_token: AccountId,
    ) {
        require!(
            env::attached_deposit() >= NEP141_DEPOSIT,
            BridgeError::NotEnoughAttachedDeposit.as_ref()
        );

        require!(
            self.deployed_tokens.remove(&old_token),
            BridgeError::OldTokenNotDeployed.as_ref(),
        );
        require!(
            self.deployed_tokens.insert(&new_token),
            BridgeError::TokenExists.as_ref()
        );
        self.deployed_tokens_v2.remove(&old_token);
        self.deployed_tokens_v2.insert(&new_token, &origin_chain);

        let origin_address = self
            .token_id_to_address
            .remove(&(origin_chain, old_token.clone()))
            .near_expect(BridgeError::FailedToGetTokenAddress);

        require!(
            self.token_id_to_address
                .insert(&(origin_chain, new_token.clone()), &origin_address)
                .is_none(),
            BridgeError::TokenExists.as_ref()
        );

        self.token_address_to_id
            .insert(&origin_address, &new_token)
            .near_expect(BridgeError::ExpectedToOverwriteTokenAddress);

        require!(
            self.migrated_tokens
                .insert(&old_token, &new_token)
                .is_none(),
            BridgeError::TokenAlreadyMigrated.as_ref()
        );

        ext_token::ext(new_token.clone())
            .with_static_gas(STORAGE_DEPOSIT_GAS)
            .with_attached_deposit(NEP141_DEPOSIT)
            .storage_deposit(&env::current_account_id(), Some(true))
            .detach();

        env::log_str(
            &OmniBridgeEvent::MigrateTokenEvent {
                old_token_id: old_token,
                new_token_id: new_token,
            }
            .to_log_string(),
        );
    }
