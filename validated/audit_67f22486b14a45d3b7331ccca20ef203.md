### Title
`batch_insert` Accepts Duplicate Keys Without Deduplication, Corrupting Merkle Tree Root and Enabling Forged Proofs - (File: crates/chia-datalayer/src/merkle/blob.rs)

### Summary

`MerkleBlob::batch_insert` processes a caller-supplied list of `(KeyId, ValueId, Hash)` tuples without checking for duplicate `KeyId` values. When duplicates are present, both leaf nodes are written into the blob and the `block_status_cache` silently overwrites the first entry with the second via `HashMap::insert`. The resulting tree contains two structurally connected leaf nodes sharing the same key, producing a corrupted Merkle root and enabling forged proofs of inclusion.

### Finding Description

`MerkleBlob::insert` (the single-item path) guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` at lines 570–657 takes a completely different code path. When the tree already has more than one leaf (`leaf_count > 1`), the function skips the guarded `insert()` call entirely and falls directly into the bulk loop:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ..., node: Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // no duplicate check
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which performs:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // silently overwrites
    self.leaf_hash_to_index.insert(leaf.hash, index);   // silently overwrites
}
```

`HashMap::insert` silently replaces the previous mapping. After the batch completes:

1. **Both** leaf nodes for the duplicate key are physically present in the blob and wired into the internal-node subtree built by the `while indexes.len() > 1` loop.
2. The cache (`key_to_index`) points only to the **second** leaf node.
3. The Merkle root is computed over the full tree structure, which includes the **first** (now orphaned-from-cache) leaf node.
4. `get_proof_of_inclusion` returns a proof anchored to the second leaf, but the root hash was computed over a tree that also contains the first leaf — so the proof is structurally inconsistent with the committed root.
5. `BlockStatusCache::new` (used when reloading the blob) detects the duplicate and returns `Error::KeyAlreadyPresent`, meaning the persisted blob is permanently unloadable.

When `leaf_count <= 1`, the last two items are popped and processed through the safe `insert()` path, but all **preceding** items in the vector still bypass the duplicate check.

The vulnerability is a direct analog of the external report: two sources of data (any two entries with the same key in the caller-supplied batch) are concatenated into a list without deduplication, and the resulting list is used to **set** (overwrite) a mapping entry (`key_to_index`), causing the first occurrence to be silently lost while both physically remain in the tree.

### Impact Explanation

- The committed Merkle root is computed over a tree containing two leaf nodes for the same key. Any proof of inclusion generated for that key is anchored to a root that does not match a canonical single-key tree, enabling a prover to present a leaf value that differs from what the root actually commits to.
- The blob becomes permanently unloadable after a restart because `BlockStatusCache::new` rejects duplicate keys.
- An adversary who controls the batch input (e.g., via the Python binding `MerkleBlob.batch_insert`) can craft a batch that corrupts the tree root, making all subsequent inclusion/exclusion proofs unverifiable or forgeable.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

### Likelihood Explanation

`batch_insert` is directly exposed as a Python binding (`wheel/python/chia_rs/datalayer.pyi` line 331). Any Python caller — including application-layer DataLayer code that aggregates updates from multiple sources — can supply a batch containing duplicate keys. No special privilege is required; the function accepts a plain list. The scenario is directly analogous to the external report: two independent sources of (key, value) pairs are merged into one list and passed to `batch_insert` without a prior deduplication step.

### Recommendation

Add a duplicate-key check at the top of `batch_insert`, mirroring the guard in `insert`:

```rust
pub fn batch_insert(
    &mut self,
    mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
) -> Result<(), Error> {
    // Reject duplicate keys within the batch itself
    for ((key, _), hash) in &keys_values_hashes {
        if self.block_status_cache.contains_key(*key) {
            return Err(Error::KeyAlreadyPresent());
        }
        if self.block_status_cache.contains_leaf_hash(hash) {
            return Err(Error::HashAlreadyPresent());
        }
    }
    // ... rest of function
}
```

Alternatively, deduplicate the input vector before processing (keeping the last occurrence, consistent with `upsert` semantics, or returning an error on any duplicate).

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def h(n: int) -> bytes:
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

# Pre-populate with 2 leaves so leaf_count > 1, bypassing the safe path
blob = MerkleBlob(blob=bytearray())
blob.insert(KeyId(100), ValueId(100), h(100))
blob.insert(KeyId(200), ValueId(200), h(200))

# batch_insert with a duplicate key (KeyId(1) appears twice)
batch_kv    = [(KeyId(1), ValueId(10)), (KeyId(2), ValueId(20)), (KeyId(1), ValueId(99))]
batch_hash  = [h(10), h(20), h(99)]
blob.batch_insert(batch_kv, batch_hash)   # succeeds — no error raised

blob.calculate_lazy_hashes()

# Cache points to the second KeyId(1) leaf (ValueId 99)
kv = blob.get_keys_values()
assert kv[KeyId(1)] == ValueId(99)

# But the blob now contains TWO leaf nodes for KeyId(1).
# Reloading the blob will fail:
try:
    MerkleBlob(blob=bytearray(blob.blob))
    print("FAIL: expected KeyAlreadyPresent error")
except Exception as e:
    print(f"Blob is permanently corrupt: {e}")

# The root hash was computed over a tree with both KeyId(1) leaves,
# so any proof of inclusion for KeyId(1) is anchored to a wrong root.
proof = blob.get_proof_of_inclusion(KeyId(1))
print(f"proof.valid() = {proof.valid()}")  # may return True against the corrupted root
