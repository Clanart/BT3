### Title
Fee Escalation Bypass via Integer Division Truncation to Zero in `increased_enough` - (File: crates/apollo_mempool/src/mempool.rs)

### Summary

The `increased_enough` function in the mempool computes the required fee escalation threshold using integer division `(existing_value * percentage) / 100`. When `existing_value * percentage < 100`, this division truncates to zero, making the escalation threshold equal to the existing value itself. An attacker can therefore replace a queued transaction with an identical tip and gas price, completely bypassing the fee escalation requirement.

### Finding Description

In `crates/apollo_mempool/src/mempool.rs`, the `increased_enough` function enforces that a replacement transaction must offer a sufficiently higher tip and max L2 gas price:

```rust
fn increased_enough(&self, existing_value: u128, incoming_value: u128) -> bool {
    let percentage = u128::from(self.config.static_config.fee_escalation_percentage);

    let Some(escalation_qualified_value) = existing_value
        .checked_mul(percentage)
        .map(|v| v / 100)          // <-- integer division truncates to zero
        .and_then(|increase| existing_value.checked_add(increase))
    else {
        return false;
    };

    incoming_value >= escalation_qualified_value
}
```

The computed `increase = (existing_value * percentage) / 100` uses integer (floor) division. When `existing_value * percentage < 100`, the increase truncates to `0`, so `escalation_qualified_value = existing_value + 0 = existing_value`. The check then becomes `incoming_value >= existing_value`, which passes for any replacement with the same or higher values — including an identical transaction.

**Concrete scenario** with the production default `fee_escalation_percentage = 10`:

| existing_value | percentage | product | increase (floor) | threshold | incoming passes at |
|---|---|---|---|---|---|
| 9 | 10 | 90 | 0 | 9 | tip = 9 (same) |
| 5 | 10 | 50 | 0 | 5 | tip = 5 (same) |
| 1 | 10 | 10 | 0 | 1 | tip = 1 (same) |

Both `tip` and `max_l2_gas_price` are checked independently via `should_replace_tx`, so an attacker sets both fields to the same small values as the existing transaction and the replacement is accepted.

The analog to the external report is exact: just as `ownerShare = (totalAmount * ownerFeePercent) / FLOAT_HANDLER_TEN_4` rounds to zero for small `totalAmount`, here `increase = (existing_value * percentage) / 100` rounds to zero for small `existing_value`, nullifying the protection.

### Impact Explanation

The mempool accepts a replacement transaction that does not satisfy the configured fee escalation requirement. An attacker with a queued transaction whose tip or max L2 gas price is below `100 / fee_escalation_percentage` (i.e., below 10 for the default 10% setting) can:

1. Continuously replace their own transaction with an identical one, refreshing its position or TTL without paying any escalation premium.
2. Spam the mempool with zero-cost replacements, consuming pool capacity and evicting other users' transactions.
3. Prevent legitimate higher-fee replacements from being correctly prioritized, since the pool believes the existing transaction already "passed" escalation.

This falls under: **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

The condition `existing_value * fee_escalation_percentage < 100` is reachable by any user who submits a transaction with a small tip or max L2 gas price. With the default `fee_escalation_percentage = 10`, any tip value of 0–9 (inclusive) triggers the truncation. Tip is a user-controlled field in every V3 transaction. No privileged access is required.

### Recommendation

Replace the floor-division escalation with a ceiling-division or a minimum-increase-of-1 guard:

```rust
fn increased_enough(&self, existing_value: u128, incoming_value: u128) -> bool {
    let percentage = u128::from(self.config.static_config.fee_escalation_percentage);

    let Some(escalation_qualified_value) = existing_value
        .checked_mul(percentage)
        .map(|v| v.div_ceil(100))   // ceil instead of floor
        .and_then(|increase| existing_value.checked_add(increase))
    else {
        return false;
    };

    incoming_value >= escalation_qualified_value
}
```

With ceiling division, `increase = ceil(9 * 10 / 100) = ceil(0.9) = 1`, so the threshold becomes `9 + 1 = 10`, correctly requiring a strictly higher value.

### Proof of Concept

**Setup:**
- `fee_escalation_percentage = 10` (production default)
- Alice submits `tx_A` with `tip = 9`, `max_l2_gas_price = 9`, `nonce = 5`

**Exploit:**
1. Alice submits `tx_B` with `tip = 9`, `max_l2_gas_price = 9`, `nonce = 5` (identical fields, different hash).
2. `increased_enough(9, 9)` computes: `increase = (9 * 10) / 100 = 90 / 100 = 0`; `threshold = 9 + 0 = 9`; `9 >= 9` → `true`.
3. `should_replace_tx` returns `true` for both tip and gas price checks.
4. `handle_fee_escalation` removes `tx_A` and accepts `tx_B` — a replacement with zero fee increase.
5. Alice can repeat indefinitely, refreshing her transaction at no escalation cost.

        let percentage = u128::from(self.config.static_config.fee_escalation_percentage);

        // Note: To reduce precision loss, we first multiply by the percentage and then divide by
        // 100. This could cause an overflow and an automatic rejection of the transaction, but the

            .map(|v| v / 100)
            .and_then(|increase| existing_value.checked_add(increase))

            // Overflow occurred during calculation; reject the transaction.
            return false;
        };

        incoming_value >= escalation_qualified_value
    }
