### Title
`MerkleBlob::upsert` Missing Hash-Uniqueness Check Enables Duplicate Leaf Hashes, Corrupting Tree Root and Forging Inclusion Proofs — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::upsert` (the update branch) does not verify that the caller-supplied `new_hash` is absent from the tree before writing it. `insert` enforces this invariant with an explicit `HashAlreadyPresent` guard, but the update path in `upsert` skips the check entirely. An unprivileged DataLayer participant can call `upsert` with a hash already held by a different leaf, silently producing two leaves with identical hashes. This corrupts the Merkle root and makes inclusion proofs for the two affected keys structurally interchangeable, enabling proof forgery.

---

### Finding Description

**Invariant enforced by `insert`, absent from `upsert`**

`insert` guards hash uniqueness before any mutation:

```rust

    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`upsert` (update branch) performs no equivalent check:

```rust
// lines 792-810

    let Ok((leaf_index, mut leaf, mut block)) = self.get_leaf_by_key(key) else {
        self.insert(key, value, new_hash, InsertLocation::Auto {})?; // insert path: checks hash
        return Ok(());

    self.block_status_cache.remove_leaf(&leaf)?;  // evicts old hash from cache

    block.node = Node::Leaf(leaf);
    self.insert_entry_to_blob(leaf_index, &block)?;
    ...
}
``` [2](#0-1) 

`insert_entry_to_blob` calls `add_leaf`, which unconditionally overwrites any existing `leaf_hash_to_index` entry for the new hash:

```rust
// lines 188-193
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index); // silently clobbers existing entry
}
``` [3](#0-2) 

**Step-by-step exploit**

1. `insert(key_A, val_A, H1)` → blob: `{A→H1, B→H2}` (two leaves, distinct hashes)
2. `insert(key_B, val_B, H2)`
3. `upsert(key_A, val_A_new, H2)` — caller supplies `H2`, which is already B's leaf hash
   - `remove_leaf(A)` removes `H1→idx_A` from `leaf_hash_to_index`; `H2→idx_B` still present
   - No guard fires; `leaf.hash` is set to `H2`
   - `insert_entry_to_blob` → `add_leaf` inserts `H2→idx_A`, **overwriting** `H2→idx_B`
   - Blob now contains two leaves both storing hash `H2` (at `idx_A` and `idx_B`)
   - `leaf_hash_to_index` has only one entry: `H2→idx_A`; B's cache entry is silently lost

**Resulting state corruption**

After `calculate_lazy_hashes()`, the internal node above both leaves computes its hash as `internal_hash(H2, H2)` (or a permutation), producing a root that encodes a tree with two identical leaf hashes. `check_integrity()` would detect `IntegrityLeafHashToIndexCacheLength` (2 leaves in blob, 1 entry in hash cache). [4](#0-3) 

**Proof-of-inclusion forgery**

`get_proof_of_inclusion` reads `node_hash` directly from the leaf stored in the blob:

```rust
// lines 1192-1195
Ok(proof_of_inclusion::ProofOfInclusion {
    node_hash: node.hash,   // ← H2 for both key_A and key_B
    layers,
})
``` [5](#0-4) 

`ProofOfInclusion::valid()` verifies only that the hash chain from `node_hash` to the root is consistent:

```rust
// proof_of_inclusion.rs lines 40-58
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(...);
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()
}
``` [6](#0-5) 

Because both leaves carry `H2`, the proof generated for `key_A` (with `node_hash = H2`) is structurally valid for `key_B`'s position in the tree as well, and vice versa. A verifier checking `proof.valid()` against the corrupted root cannot distinguish which key the proof belongs to.

---

### Impact Explanation

**High — DataLayer Merkle proof/blob/delta logic corrupts tree roots and lets untrusted input prove invalid state.**

- The Merkle root is computed over a tree containing two leaves with identical hashes, making the root meaningless as a commitment to distinct key-value pairs.
- Inclusion proofs become ambiguous: a proof generated for `key_A` passes `valid()` when presented as a proof for `key_B`, because both leaves share `node_hash = H2` and the path hashes are identical.
- Any downstream system that trusts `proof.valid()` to authenticate which key-value pair is committed at a given root is vulnerable to substitution attacks.

---

### Likelihood Explanation

The attacker needs only:
1. Knowledge of any existing leaf's hash — trivially obtained via `get_hashes_indexes()` or `get_proof_of_inclusion()`, both of which are public API.
2. The ability to call `upsert` on a key they control — a standard DataLayer operation requiring no privilege.

No cryptographic break, no key compromise, no network-level access is required. The attack is a single API call. [7](#0-6) 

---

### Recommendation

Add the same `HashAlreadyPresent` guard to the update branch of `upsert`, mirroring `insert`:

```rust
pub fn upsert(&mut self, key: KeyId, value: ValueId, new_hash: &Hash) -> Result<(), Error> {
    let Ok((leaf_index, mut leaf, mut block)) = self.get_leaf_by_key(key) else {
        self.insert(key, value, new_hash, InsertLocation::Auto {})?;
        return Ok(());
    };

    // Guard: reject if new_hash already belongs to a different leaf
    if leaf.hash != *new_hash && self.block_status_cache.contains_leaf_hash(new_hash) {
        return Err(Error::HashAlreadyPresent());
    }

    self.block_status_cache.remove_leaf(&leaf)?;
    leaf.hash.clone_from(new_hash);
    ...
}
```

Additionally, add a fuzz target that exercises `upsert` with hashes drawn from the existing tree to catch regressions.

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

let mut blob = MerkleBlob::new(vec![]).unwrap();
let h1 = Hash(Bytes32::new([1u8; 32]));
let h2 = Hash(Bytes32::new([2u8; 32]));

// Insert two distinct leaves
blob.insert(KeyId(1), ValueId(10), &h1, InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(2), ValueId(20), &h2, InsertLocation::Auto {}).unwrap();

// Upsert key_A with key_B's hash — no error returned
blob.upsert(KeyId(1), ValueId(99), &h2).unwrap();

// Tree now has two leaves both with hash h2
blob.calculate_lazy_hashes().unwrap();

// Both proofs share node_hash = h2 and are structurally identical
let proof_a = blob.get_proof_of_inclusion(KeyId(1)).unwrap();
let proof_b = blob.get_proof_of_inclusion(KeyId(2)).unwrap();
assert_eq!(proof_a.node_hash, proof_b.node_hash); // both == h2

// proof_a.valid() == true against the corrupted root
// proof_b.valid() == true against the same corrupted root
// A verifier cannot distinguish which key either proof belongs to

// check_integrity detects the corruption
blob.check_integrity().expect_err("IntegrityLeafHashToIndexCacheLength");
