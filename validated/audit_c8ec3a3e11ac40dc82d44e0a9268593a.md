### Title
Precommit Vote Message Digest Omits Chain ID, Enabling Cross-Chain Replay of Consensus Signatures — (File: crates/apollo_signature_manager/src/signature_manager.rs)

### Summary

`build_precommit_vote_message_digest` constructs the consensus precommit vote message as `blake2s("PRECOMMIT_VOTE" || block_hash)` with no chain ID. Because `calculate_block_hash` also omits chain ID, any block whose parameters are identical across two Starknet networks (e.g., mainnet and a fork that shares the same genesis) produces the same block hash and therefore the same precommit vote digest. A validator's precommit signature produced on chain A is cryptographically valid on chain B, allowing an unprivileged observer to replay it and inject foreign consensus votes into the target chain.

### Finding Description

**Root cause — missing chain ID in the vote digest:**

`build_precommit_vote_message_digest` in `crates/apollo_signature_manager/src/signature_manager.rs`:

```rust
fn build_precommit_vote_message_digest(block_hash: BlockHash) -> MessageDigest {
    let block_hash = block_hash.to_bytes_be();
    let mut message = Vec::with_capacity(PRECOMMIT_VOTE.len() + block_hash.len());
    message.extend_from_slice(PRECOMMIT_VOTE);   // b"PRECOMMIT_VOTE"
    message.extend_from_slice(&block_hash);       // 32-byte block hash
    MessageDigest(blake2s_to_felt(&message))
    // ← no chain_id field anywhere
}
``` [1](#0-0) 

The domain separator is only the ASCII string `"PRECOMMIT_VOTE"`. Chain ID is never mixed in.

**Root cause — block hash also omits chain ID:**

`calculate_block_hash` in `crates/starknet_api/src/block_hash/block_hash_calculator.rs` hashes: version constant, block number, state root, sequencer address, timestamp, concatenated counts, state-diff commitment, transaction commitment, event commitment, receipt commitment, gas prices, starknet version, a hardcoded zero, and `previous_block_hash`. Chain ID is absent from every field. [2](#0-1) 

The Cairo OS implementation confirms the same layout: [3](#0-2) 

**Consequence — identical block hashes across a fork:**

Because `previous_block_hash` chains back to genesis, every block before a fork point has an identical hash on both chains. For the first post-fork block N+1, if the same sequencer produces the same transactions on the same pre-fork state (state root identical at height N), all inputs to `calculate_block_hash` are equal, so `hash(block N+1)` is equal on both chains. The precommit vote digest `blake2s("PRECOMMIT_VOTE" || hash(block N+1))` is therefore identical, and a signature produced by a validator on chain A is a valid ECDSA signature on chain B.

**Contrast with transaction hashes (which are protected):**

Every transaction hash variant (`get_invoke_transaction_v3_hash`, `get_declare_transaction_v3_hash`, etc.) chains `Felt::try_from(chain_id)?` into the hash, making transactions chain-specific. [4](#0-3) 

Proof facts are similarly protected via `virtual_os_config_hash`, which embeds chain ID through `OsChainInfo::compute_os_config_hash`. [5](#0-4) 

Precommit votes have no equivalent protection.

**`verify_block_signature` shares the same gap:**

The sequencer block signature also omits chain ID: `Poseidon(block_hash, state_diff_commitment)`. [6](#0-5) 

### Impact Explanation

An attacker who can observe the P2P network on chain A can collect valid precommit vote signatures from any validator whose key is shared across both chains (the normal case during a fork). For every block height where the two chains produce the same block hash, the attacker replays those signatures on chain B. Chain B's consensus layer counts them as genuine votes from those validators, potentially reaching a 2/3 quorum with votes that were never cast on chain B. This causes chain B to finalize a block that its own validators did not independently vote for — the signature/hash logic binds the wrong chain to the vote, satisfying the **High** impact: *Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.*

### Likelihood Explanation

Requires a Starknet network fork where both chains share the same genesis and a common block history. This is a realistic scenario for contentious protocol upgrades or emergency forks. Once a fork exists, the attack is fully unprivileged: any observer of the P2P layer can collect and replay signatures. The condition that block N+1 has the same hash on both chains is satisfied whenever the same sequencer produces the same block content (same transactions, same timestamp, same gas prices) on both chains immediately after the fork — a plausible scenario in the first few blocks of a fork.

### Recommendation

Include chain ID in the precommit vote message digest:

```rust
fn build_precommit_vote_message_digest(
    block_hash: BlockHash,
    chain_id: &ChainId,
) -> MessageDigest {
    let block_hash = block_hash.to_bytes_be();
    let chain_id_felt: Felt = chain_id.try_into().expect("chain id to felt");
    let chain_id_bytes = chain_id_felt.to_bytes_be();
    let mut message = Vec::with_capacity(
        PRECOMMIT_VOTE.len() + block_hash.len() + chain_id_bytes.len()
    );
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash);
    message.extend_from_slice(&chain_id_bytes);
    MessageDigest(blake2s_to_felt(&message))
}
```

Apply the same fix to `verify_block_signature` by including chain ID in the Poseidon hash. Consider also adding chain ID to `calculate_block_hash` (matching the pattern already used for transaction hashes) to make the block hash itself chain-specific.

### Proof of Concept

1. Starknet forks at block N. Both chains share identical block hashes for blocks 0…N because they share the same history.
2. The sequencer produces block N+1 with the same transactions, timestamp, and gas prices on both chains. Since the pre-fork state root is identical, the post-execution state root is also identical. All inputs to `calculate_block_hash` are equal → `hash(block N+1)` is equal on both chains.
3. Validator V signs a precommit vote on chain A: `sig_A = ecdsa_sign(key_V, blake2s("PRECOMMIT_VOTE" || hash(block N+1)))`.
4. Because `hash(block N+1)` is the same on chain B, `sig_A` is a valid precommit vote for block N+1 on chain B.
5. An attacker observing chain A's P2P traffic collects `sig_A` (and analogous signatures from other validators) and injects them into chain B's consensus network via `SignatureManagerRequest::SignPrecommitVote` or direct P2P broadcast.
6. Chain B's consensus counts these replayed votes toward its 2/3 quorum, finalizing block N+1 on chain B with votes that were never independently cast by chain B's validators. [7](#0-6) [8](#0-7)

func calculate_block_hash{poseidon_ptr: PoseidonBuiltin*}(
    block_info: BlockInfo*,
    header_commitments: BlockHeaderCommitments*,
    gas_prices_hash: felt,
    state_root: felt,
    previous_block_hash: felt,
    starknet_version: felt,
) -> felt {
    static_assert BlockInfo.SIZE == 3;
    static_assert BlockHeaderCommitments.SIZE == 5;

    let hash_state = hash_init();
    with hash_state {
        hash_update_single(BLOCK_HASH_VERSION);
        hash_update_single(block_info.block_number);
        hash_update_single(state_root);
        hash_update_single(block_info.sequencer_address);
        hash_update_single(block_info.block_timestamp);
        hash_update_single(header_commitments.packed_lengths);
        hash_update_single(header_commitments.state_diff_commitment);
        hash_update_single(header_commitments.transaction_commitment);
        hash_update_single(header_commitments.event_commitment);
        hash_update_single(header_commitments.receipt_commitment);
        hash_update_single(gas_prices_hash);
        hash_update_single(starknet_version);
        hash_update_single(0);
        hash_update_single(previous_block_hash);
    }

    let block_hash = hash_finalize(hash_state=hash_state);
    return block_hash;
}
