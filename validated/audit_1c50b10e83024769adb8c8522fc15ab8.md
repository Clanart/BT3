### Title
`ProofOfInclusion.valid()` Omits Trusted-Root Binding, Enabling Forged Inclusion Proofs — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary
The DataLayer `ProofOfInclusion::valid()` method verifies only the internal self-consistency of a proof structure. It never accepts or checks against a caller-supplied trusted root hash. Because `root_hash()` is derived entirely from the proof's own fields, any attacker who can supply a `ProofOfInclusion` to a verifier can forge an internally-consistent proof for an arbitrary key, and `valid()` will return `true`. This is the direct analog of the external report's pattern: the caller has no mechanism to enforce the expected output (the tree root) within the validation call itself.

### Finding Description

`ProofOfInclusion` is a `Streamable` struct with full Python bindings, meaning it can be deserialized from untrusted network bytes. [1](#0-0) 

Its `valid()` method walks the layer chain and checks that each `combined_hash` is correctly computed from the previous hash and `other_hash`: [2](#0-1) 

The final check is `existing_hash == self.root_hash()`, where `root_hash()` returns `self.layers.last().combined_hash` — a field that is part of the proof itself and is fully attacker-controlled: [3](#0-2) 

No trusted external root is ever compared. An attacker can construct a `ProofOfInclusion` with an arbitrary `node_hash` (claiming any key-value hash) and a chain of `layers` whose `combined_hash` values are computed correctly from each other, producing a proof where `valid()` returns `true` while the `root_hash()` it claims is completely fabricated.

The Python binding exposes exactly this incomplete interface: [4](#0-3) 

The type stub confirms `valid()` takes no root parameter: [5](#0-4) 

The fuzz target and all internal tests call only `proof.valid()` without any root comparison, establishing the pattern that callers are expected to follow: [6](#0-5) 

By contrast, the consensus-layer `validate_merkle_proof` function (used for the `MerkleSet`) correctly requires the caller to supply the trusted root and rejects the proof if `tree.get_root() != *root`: [7](#0-6) 

The DataLayer `ProofOfInclusion` API has no equivalent guard.

### Impact Explanation

A DataLayer peer or client that receives a `ProofOfInclusion` over the network, deserializes it via `ProofOfInclusion::from_bytes()`, and calls `proof.valid()` will accept a forged proof claiming any key is present in any tree. This lets untrusted input prove invalid DataLayer state — matching the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."*

### Likelihood Explanation

The `ProofOfInclusion` struct is `Streamable` and has Python bindings with `from_bytes` / `from_json_dict` constructors, making deserialization from untrusted sources trivial. The method name `valid()` strongly implies complete validation, and neither the Rust API nor the Python stub provides a `valid_for_root(root)` alternative. Any DataLayer verifier that follows the pattern established by the internal tests and fuzz target — calling only `proof.valid()` — is vulnerable. The attacker needs only to craft a self-consistent proof chain, which requires no cryptographic secrets.

### Recommendation

Add a `trusted_root` parameter to the validation path, mirroring the consensus `validate_merkle_proof` design:

```rust
pub fn valid_for_root(&self, trusted_root: &Hash) -> bool {
    self.valid() && &self.root_hash() == trusted_root
}
```

Deprecate or remove the root-free `valid()` from the public API, or at minimum rename it to `internally_consistent()` to make clear it does not bind to any external tree state. Update the Python binding to expose only the root-bound variant.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer

# Attacker constructs a fake leaf hash for a key they do not own
fake_node_hash = bytes(range(32))          # arbitrary "key hash"
fake_other_hash = bytes(range(32, 64))     # arbitrary sibling

# Compute a valid combined_hash so the layer chain is internally consistent
import hashlib
# DataLayer uses calculate_internal_hash internally; replicate it:
# (exact hash function depends on Side; simplified here)
combined = hashlib.sha256(fake_node_hash + fake_other_hash).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=0,          # Left
    other_hash=bytes(fake_other_hash),
    combined_hash=bytes(combined),
)
forged_proof = ProofOfInclusion(
    node_hash=bytes(fake_node_hash),
    layers=[layer],
)

# valid() returns True — no trusted root was ever checked
assert forged_proof.valid(), "Forged proof accepted!"
# root_hash() returns the attacker-chosen combined value
print("Claimed root:", forged_proof.root_hash().hex())
```

A verifier that only calls `proof.valid()` accepts this proof unconditionally, regardless of what the actual DataLayer tree root is.

    pub fn valid(&self) -> bool {
        let mut existing_hash = self.node_hash;

        for layer in &self.layers {
            let calculated_hash = crate::calculate_internal_hash(
                &existing_hash,
                layer.other_hash_side,
                &layer.other_hash,
            );

            if calculated_hash != layer.combined_hash {
                return false;
            }

            existing_hash = calculated_hash;
        }

        existing_hash == self.root_hash()
    }
