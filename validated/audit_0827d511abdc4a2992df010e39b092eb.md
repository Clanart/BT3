### Title
u128 Overflow in `denormalize_amount` Causes Permanent DoS of `fin_transfer` for Large-Amount Transfers — (`near/omni-bridge/src/lib.rs`)

### Summary

`overflow-checks = true` is confirmed in the release profile. [1](#0-0)  The `denormalize_amount` function in `near/omni-bridge/src/lib.rs` multiplies a normalized amount by `10^(origin_decimals - decimals)`. For a token with `origin_decimals=24` and `decimals=6` (diff=18), any normalized amount exceeding `u128::MAX / 10^18 ≈ 3.4 × 10^20` will overflow u128 and panic at runtime. [2](#0-1) 

### Finding Description

The `Decimals` struct is stored per-token in contract storage. [2](#0-1)  The call chain is:

```
fin_transfer (public)
  → prover.verify_proof
  → fin_transfer_callback
      → denormalize_amount(normalized_amount, Decimals { decimals: 6, origin_decimals: 24 })
          → normalized_amount * 10^18  ← OVERFLOW PANIC
```

With `overflow-checks = true`, Rust panics (does not silently wrap) on the multiplication. In NEAR, a panicking callback rolls back all state changes made within that call — so the finalization nonce is **not** permanently consumed. However, the proof is deterministic: every re-submission of the same proof will produce the same overflow and panic. The transfer is therefore **permanently unfinalizeable**.

### Impact Explanation

User funds locked on the source chain (EVM/Solana/StarkNet) with no recovery path. The NEAR-side finalization always panics for that transfer ID. This matches the allowed impact: **permanent freezing / irrecoverable lock of user funds**.

### Likelihood Explanation

- EVM token amounts are `uint256`. A token with 24 native decimals and a large supply can produce a normalized amount (after dividing by `10^18`) that still exceeds `u128::MAX / 10^18`.
- No guard on the normalized amount is visible before `denormalize_amount` is called.
- The attack requires the attacker (or an innocent whale) to lock a very large amount on the source chain — not free, but requires no privilege.
- The `overflow-checks = true` setting is confirmed in the workspace release profile. [3](#0-2) 

### Recommendation

Replace the bare multiplication in `denormalize_amount` with a checked or saturating variant:

```rust
amount.checked_mul(10u128.pow(diff as u32))
    .expect("denormalize_amount: overflow")
```

Or, better, return a `Result` and propagate the error up through `fin_transfer_callback` so the call fails gracefully (without consuming the nonce) and the relayer can be informed of the invalid amount.

### Proof of Concept

```rust
// Pseudocode unit test
let decimals = Decimals { decimals: 6, origin_decimals: 24 };
let diff = 18u32;
let amount: u128 = u128::MAX / 10u128.pow(17) + 1; // > u128::MAX / 10^18
// With overflow-checks=true, the next line panics:
let _ = amount * 10u128.pow(diff); // PANIC: attempt to multiply with overflow
```

The nonce is rolled back by NEAR's panic semantics, but every re-attempt panics identically, permanently blocking finalization and locking source-chain funds.
