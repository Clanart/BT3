Audit Report

## Title
Unguarded `refundETH()` allows any caller to drain excess ETH stranded on the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` is a public, zero-access-control function that transfers the router's entire native ETH balance to `msg.sender`. When a user calls `multicall{value}` with a WETH-input swap but omits `refundETH`, the `pay()` helper consumes only the exact swap amount and leaves the remainder on the router. Any subsequent caller can immediately drain it by calling `refundETH()` directly.

## Finding Description
`refundETH()` contains no ownership or origin check — it sends the full contract ETH balance to whoever calls it: [1](#0-0) 

The `pay()` helper, when `token == WETH` and `nativeBalance >= value`, deposits exactly `value` wei as WETH and leaves `nativeBalance - value` on the router untouched: [2](#0-1) 

`multicall` is `payable` and forwards `msg.value` into the execution context, making it straightforward for a user to send more ETH than the swap consumes: [3](#0-2) 

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses — it does **not** intercept `msg.value` attached to `payable` function calls, so it provides no protection here: [4](#0-3) 

The protocol's own tests confirm the intended pattern is `multicall{value}([swap, refundETH])`. If the user omits `refundETH`, the excess ETH is permanently accessible to any caller: [5](#0-4) 

## Impact Explanation
Direct, irreversible loss of the victim's excess ETH. The attacker calls one public function with no preconditions other than timing. No privileged role, malicious pool, or non-standard token is required. This constitutes a Critical/High direct loss of user principal.

## Likelihood Explanation
The ETH-input multicall pattern is explicitly documented and tested. Users who overpay (common with exact-input swaps where the exact WETH cost is unknown off-chain) and omit `refundETH` — whether by mistake, wallet UX error, or a failed second call — leave funds claimable by any observer. MEV bots routinely monitor for exactly this class of stranded balance, making exploitation highly likely and repeatable.

## Recommendation
Record `msg.sender` in transient storage at the start of `multicall` and require `msg.sender == _transientDepositor()` inside `refundETH`. Alternatively, restrict `refundETH` to only return ETH to the address that sent `msg.value` in the current multicall context. The simplest safe fix is a transient depositor registry set at `multicall` entry and checked in `refundETH`.

## Proof of Concept
```
1. User calls router.multicall{value: 1 ether}([
       exactInputSingle(tokenIn=WETH, amountIn=0.5 ether, ...)
       // refundETH omitted
   ])
2. pay() deposits 0.5 ETH as WETH → pool; 0.5 ETH remains on router.
3. Attacker calls router.refundETH() in the next transaction.
4. _transferETH(attacker, 0.5 ether) executes; attacker receives victim's 0.5 ETH.
5. assert attacker.balance increased by 0.5 ETH; victim cannot recover funds.
```
Modify the existing test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` to omit `calls[1]` (the `refundETH` call), then have a second address call `router.refundETH()` and assert it receives the stranded balance.
