Audit Report

## Title
ETH Sent With Non-WETH Swaps Is Silently Consumed by Subsequent WETH Swaps, Causing Permanent Loss of User Funds — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` function in `PeripheryPayments.sol` reads `address(this).balance` — the router's entire ETH balance — when deciding how much native ETH to wrap for a WETH payment. Because all swap and liquidity entry-points are `payable` with no guard rejecting ETH when `tokenIn != WETH`, any ETH accidentally sent with a non-WETH swap is silently stranded in the router. Any subsequent caller whose `tokenIn` is WETH will have their payment covered by the stranded ETH, permanently draining the original sender's funds with no recourse.

## Finding Description
**Root cause — `pay()` uses total contract balance, not per-call ETH:**

In `PeripheryPayments.sol` lines 73–84, when `token == WETH`, the function reads `uint256 nativeBalance = address(this).balance` rather than the ETH that arrived in the current transaction. If `nativeBalance >= value`, it wraps and transfers that ETH without pulling any tokens from `payer`. [1](#0-0) 

**ETH stranding — every entry-point is `payable` with no tokenIn guard:**

`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, and all `addLiquidity*` variants are declared `external payable` with no check that `msg.value == 0` when `tokenIn != WETH`. When a user sends ETH alongside a non-WETH swap, the callback takes the `else` branch (`safeTransferFrom`) and the ETH is never consumed — it accumulates in the router. [2](#0-1) 

**`receive()` guard is irrelevant here:**

The `receive()` guard only blocks plain ETH transfers (no calldata). ETH sent alongside a function call is accepted by the `payable` modifier before `receive()` is ever consulted. [3](#0-2) 

**Exploit flow:**
1. User A calls `exactInputSingle({tokenIn: USDC, amountIn: 1000e6, ...})` with `msg.value = 1 ether`. USDC is pulled via `safeTransferFrom`; 1 ETH sits in the router.
2. User B calls `exactInputSingle({tokenIn: WETH, amountIn: 1e18, ...})` with `msg.value = 0`. The callback fires `pay(WETH, userB, pool, 1e18)`. `address(this).balance == 1e18` (User A's ETH), so `nativeBalance >= value` → `WETH.deposit{value: 1e18}()` → WETH transferred to pool. User B's own WETH is never touched.
3. User A calls `refundETH()` → balance is 0; nothing returned. User A has permanently lost 1 ETH. [4](#0-3) 

**Same vulnerability in `MetricOmmPoolLiquidityAdder`:**

`addLiquidityExactShares` and `addLiquidityWeighted` are also `external payable` and call `pay()` in their callback, inheriting the same flaw. [5](#0-4) [6](#0-5) 

## Impact Explanation
Direct, permanent loss of user principal (ETH) with no on-chain recourse once a WETH swap consumes the stranded balance. The wrong value is `address(this).balance` being used as the payment source instead of only the ETH belonging to the current caller. This meets the Critical/High threshold for direct loss of user funds: an unprivileged user can lose an arbitrary amount of ETH, and an unprivileged attacker (or MEV bot) can drain it passively by submitting any WETH swap.

## Likelihood Explanation
Sending ETH with a non-WETH swap is a realistic user error — common for users migrating from native-ETH DEX UIs or reusing scripts. Exploitation requires no privileged access: any ordinary WETH swap passively drains the stranded balance. MEV bots monitoring the router's ETH balance can exploit this atomically by front-running `refundETH()` calls, making the escape hatch unreliable in a public mempool.

## Recommendation
Add an input guard in every `payable` swap and liquidity entry-point:
```solidity
if (msg.value > 0 && params.tokenIn != WETH) revert ETHNotAcceptedForNonWETHSwap();
```
Alternatively, restrict `pay()` to use only the ETH that arrived in the current transaction by passing `msg.value` explicitly as a parameter rather than reading `address(this).balance`. This eliminates the cross-caller contamination entirely.

## Proof of Concept
```
1. User A calls exactInputSingle({tokenIn: USDC, amountIn: 1000e6, ...})
   with msg.value = 1 ether.
   → USDC pulled from User A via safeTransferFrom; 1 ETH sits in router.

2. User B calls exactInputSingle({tokenIn: WETH, amountIn: 1e18, ...})
   with msg.value = 0.
   → Callback fires: pay(WETH, userB, pool, 1e18)
   → address(this).balance = 1e18 (User A's ETH)
   → nativeBalance >= value → WETH.deposit{value: 1e18}(); transfer to pool
   → User B's WETH never touched; User B receives a free swap.

3. User A calls refundETH() → balance is 0; nothing returned.
   User A has lost 1 ETH with no recourse.
```
Reproducible as a Foundry unit test: deploy `MetricOmmSimpleRouter` with a mock WETH and mock pool that calls back with the expected delta; execute steps 1–3 and assert User A's ETH balance decreased by 1 ETH while User B's WETH balance is unchanged.
