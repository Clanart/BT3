### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Enables Unbacked Cross-Chain Minting — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` emits the caller-supplied `amount` in the `InitTransfer` event after calling `safeTransferFrom`, without measuring the actual tokens received. For fee-on-transfer tokens the bridge locks fewer tokens than the event advertises, and the NEAR bridge uses that event value to mint/release the full `amount` on the destination chain, creating unbacked supply.

---

### Finding Description

In `OmniBridge.sol`, the `initTransfer` function handles native ERC-20 tokens (those that are neither bridge tokens nor custom-minter tokens) with:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ← user-supplied
);
``` [1](#0-0) 

Immediately after, the function emits the event using the same unverified `amount`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // ← not the actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

The `InitTransfer` event's `amount` field is defined as `uint128` in `BridgeTypes.sol`: [3](#0-2) 

On the NEAR side, `fin_transfer_callback` decodes the proven event and constructs a `TransferMessage` directly from `init_transfer.amount`:

```rust
let transfer_message = TransferMessage {
    ...
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
};
``` [4](#0-3) 

There is no reconciliation between the event-advertised `amount` and the tokens actually held by the EVM bridge contract.

---

### Impact Explanation

For any ERC-20 token that deducts a transfer fee (fee-on-transfer or deflationary token), the EVM bridge receives `amount - fee_deducted` but the `InitTransfer` event records `amount`. The NEAR bridge proves this event and mints or releases the full `amount` on the destination chain. The EVM vault is undercollateralized by exactly `fee_deducted` per transfer. An attacker can repeat this to drain the bridge's reserves of any co-listed token, or to mint unbacked wrapped tokens on NEAR/other chains.

This matches the allowed impact: **balance/accounting corruption that breaks bridge collateralization** (High), and potentially **direct unauthorized mint of bridged assets** (Critical) depending on the token path.

---

### Likelihood Explanation

Any unprivileged user can call `initTransfer` with a fee-on-transfer ERC-20 token that has been registered with the bridge. No special role or key is required. The attacker only needs to hold the token and pay the native fee. The discrepancy is introduced on every single such transfer, making it continuously exploitable.

---

### Recommendation

Measure the actual received amount by comparing balances before and after the transfer, and use that value for the event:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint128 receivedAmount = uint128(IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore);
// use receivedAmount in place of amount for the event and fee validation
```

Alternatively, explicitly disallow fee-on-transfer tokens by verifying `receivedAmount == amount` and reverting otherwise, to prevent silent accounting drift.

---

### Proof of Concept

1. A fee-on-transfer ERC-20 token `FOT` (2% fee) is registered with the EVM `OmniBridge`.
2. Attacker calls `initTransfer(FOT, 1_000_000, fee, nativeFee, nearRecipient, "")`.
3. `safeTransferFrom` moves `1_000_000` from attacker; the token contract deducts 2%, so the bridge receives `980_000`.
4. The `InitTransfer` event is emitted with `amount = 1_000_000`.
5. A relayer submits proof of this event to the NEAR `fin_transfer` endpoint.
6. `fin_transfer_callback` constructs a `TransferMessage` with `amount = 1_000_000` and mints/releases `1_000_000` tokens to the NEAR recipient.
7. The EVM bridge holds only `980_000` FOT but the NEAR side has released `1_000_000` — a `20_000`-token deficit per call.
8. Repeating this drains the bridge's collateral or inflates the wrapped token supply without backing. [5](#0-4) [6](#0-5)
