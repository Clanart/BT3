### Summary

The `execute_deploy` syscall implementation in the StarkNet OS unconditionally writes `failure_flag=0` (success) in the response header, regardless of whether the deployed contract's constructor actually reverted. This is a direct analog to M-02: just as `XC20Wrapper` assumed `mint()` could never revert and thereby permanently locked tokens, the OS assumes a constructor can never revert and thereby gives the caller contract a false success signal. A caller that trusts this signal and subsequently transfers funds to the "deployed" address can permanently lose those funds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_deploy` function processes the `DEPLOY` syscall. After calling `deploy_contract` (which internally calls the constructor entry point), the OS writes the response header with a hardcoded `failure_flag=0`:

```cairo
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `TODO` comment explicitly acknowledges that failure handling is not implemented. The `deploy_contract` call that precedes this line can internally encounter a reverted constructor â€” `execute_entry_point` handles the revert via `handle_revert` and returns `is_reverted=1` â€” but `execute_deploy` discards that outcome and always reports success to the calling contract. [2](#0-1) 

The constructor can revert for any reason a Sierra contract can revert: out-of-gas, assertion failure, arithmetic overflow, or explicit `panic`. The `execute_entry_point` machinery already handles this correctly for normal contract calls (returning `is_reverted=1` and calling `handle_revert`): [3](#0-2) 

But `execute_deploy` never checks the return value for revert status and always writes `failure_flag=0`.

By contrast, `non_reverting_select_execute_entry_point_func` â€” used for fee charging and validation â€” at least asserts `is_reverted = 0` so a revert would surface as a proof failure rather than a silent lie: [4](#0-3) 

`execute_deploy` does neither: it does not assert success nor propagate failure.

### Impact Explanation

**Critical â€” Permanent freezing of funds.**

When a caller contract receives `failure_flag=0` and a valid `contract_address` from the deploy syscall, it has no protocol-level way to distinguish a successful deployment from a silently-failed one. A common and reasonable pattern is:

1. Deploy a token vault or escrow contract.
2. Immediately transfer tokens to the newly deployed address.

If the constructor reverted:
- The constructor's state changes are rolled back via `handle_revert` inside `execute_entry_point`.
- Depending on `deploy_contract`'s implementation, the class hash at the computed address may also be rolled back, leaving the address with no contract code.
- The caller, having received `failure_flag=0`, proceeds to transfer tokens to that address.
- In StarkNet, an ERC-20 `transfer` updates the token contract's storage mapping `balances[recipient]`. The tokens arrive at the recipient address regardless of whether a contract exists there.
- With no contract code at the address, there is no `transfer` function to call to move the tokens out. The funds are permanently frozen.

This matches the M-02 impact exactly: the cross-chain processing step (constructor execution) can fail indefinitely, and the protocol-level response gives the caller no signal to fall back to a safe path.

---

### Likelihood Explanation

**High.** The `deploy` syscall is a standard, unprivileged operation available to any contract. The attacker-controlled entry path requires only:

1. Declaring a contract class whose constructor reverts under attacker-chosen conditions (e.g., always, or after a specific calldata value).
2. Having a victim contract (or the attacker's own contract acting as an intermediary) call `deploy` on that class and then transfer funds to the returned address.

No privileged role, leaked key, or operator cooperation is required. The `failure_flag=0` guarantee is enforced by the OS itself, so the caller contract cannot detect the failure even with careful defensive coding.

---

### Recommendation

Replace the hardcoded `failure_flag=0` with the actual outcome of `deploy_contract`. The `deploy_contract` function already returns `(retdata_size, retdata)` and internally uses `handle_revert`; it should also return an `is_reverted` flag. The response header should then be:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

If the constructor reverted, the response should carry the revert data as the `constructor_retdata` and set `failure_flag=1`, consistent with how `contract_call_helper` handles reverted calls: [5](#0-4) 

Additionally, if the constructor reverts, the class hash assignment at the computed address must be rolled back before returning, so no zombie address is left in state.

3. Address `X` has no contract code (constructor reverted, class hash rolled back).
