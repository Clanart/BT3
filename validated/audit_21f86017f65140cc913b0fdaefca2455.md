Looking at the EVM contracts, I can see `OmniBridge.sol` already uses `call{value: ...}("")` (not `.transfer()`), but there is a closely related analog vulnerability in the same function. Let me verify the exact flow.

### Title
Native ETH Delivery to Non-Payable Smart Contract Recipient Permanently Freezes Bridged Funds — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer()` delivers native ETH to `payload.recipient` via a low-level `call`. If the recipient is a smart contract that lacks a `payable` fallback (or explicitly reverts on ETH receipt), every attempt to finalize the cross-chain transfer reverts. Because the source-chain funds are already irrevocably locked or burned at `initTransfer` time, and no admin rescue or redirect path exists, the bridged ETH becomes permanently unclaimable.

---

### Finding Description

In `OmniBridge.finTransfer()`, when `payload.tokenAddress == address(0)` (native ETH bridge-out), the contract attempts delivery:

```solidity
// OmniBridge.sol lines 317-322
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [1](#0-0) 

The `revert FailedToSendEther()` rolls back the entire transaction, including the `completedTransfers[payload.destinationNonce] = true` write at line 287: [2](#0-1) 

This means the destination nonce is never consumed, so `finTransfer` can be called again — but if the recipient contract always reverts on ETH receipt, every retry fails identically. There is no admin function to redirect the delivery to a different address, no pull-withdrawal fallback, and no refund path back to the source chain.

On the source side, `initTransfer` with `tokenAddress == address(0)` accepts `msg.value` and retains `amount` inside the bridge contract: [3](#0-2) 

Once the MPC has signed the cross-chain payload, the source-chain ETH is committed. There is no mechanism to reverse or reclaim it.

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds.**

Any native ETH bridged to a smart-contract recipient that does not implement a `payable` fallback — including common multisig wallets (Gnosis Safe with certain module configurations), proxy contracts, or any contract that explicitly reverts on plain ETH receipt — will have its funds locked in the bridge contract forever. The bridge holds the ETH (via `receive() external payable {}`), the nonce is never finalized, and no recovery path exists. [4](#0-3) 

---

### Likelihood Explanation

**Moderate-to-High.** The recipient EVM address is specified by the user when initiating the transfer on the source chain (NEAR or another EVM). Smart contract wallets — multisigs, account-abstraction wallets, protocol treasuries — are common bridge recipients. Many such contracts do not implement a plain-ETH `receive()` or `payable fallback`. A user who mistakenly (or deliberately) routes native ETH to such an address triggers the freeze with no recourse. No privileged access is required; any bridge user can reach this path.

---

### Recommendation

1. **Adopt a pull-payment pattern for native ETH.** Instead of pushing ETH to `payload.recipient` in `finTransfer`, credit an internal `pendingWithdrawals[recipient]` mapping and expose a separate `withdraw()` function the recipient calls at their convenience.

2. **If push delivery is retained**, wrap the `call` failure in a fallback that credits `pendingWithdrawals[payload.recipient]` rather than reverting, so the nonce is consumed and the funds remain claimable.

3. **Add an admin rescue function** that can redirect a stuck pending withdrawal to an alternate address after a timeout, guarded by `DEFAULT_ADMIN_ROLE`.

---

### Proof of Concept

1. **Source chain (NEAR or EVM):** User calls `initTransfer(address(0), 1 ether, 0, nativeFee, recipient, "")` sending `1 ether + nativeFee`. The bridge retains `1 ether`. [5](#0-4) 

2. **MPC signs** a `TransferMessagePayload` with `tokenAddress = address(0)`, `amount = 1 ether`, `recipient = <NonPayableContract>`.

3. **Destination chain:** Anyone calls `finTransfer(sig, payload)`.
   - Line 287: `completedTransfers[nonce] = true` (tentative).
   - Line 319: `NonPayableContract.call{value: 1 ether}("")` → `success = false` (no payable fallback).
   - Line 322: `revert FailedToSendEther()` — entire tx rolls back, nonce not consumed. [6](#0-5) 

4. **Result:** Every subsequent `finTransfer` call reverts identically. The `1 ether` sits in the bridge contract permanently. The source-chain funds are already committed and cannot be reclaimed. Funds are frozen forever.
