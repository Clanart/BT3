### Title
EIP-7702 SetCode Authorizations Applied Without Checking Prague Hardfork Activation - (File: x/evm/keeper/state_transition.go)

### Summary
`ApplyMessageWithConfig` processes EIP-7702 `SetCodeAuthorizations` (nonce bump + delegation code write) without first verifying `rules.IsPrague`. A type-4 (SetCode) transaction submitted before Prague activation passes all ante-handler checks and causes permanent, unauthorized EOA code mutation and nonce manipulation — state changes that the protocol explicitly forbids before Prague.

### Finding Description

In `ApplyMessageWithConfig`, every other hardfork-gated feature is guarded by the corresponding rule flag before touching state:

```go
// EIP-7623 — correctly gated
if rules.IsPrague {
    floorDataGas, err := core.FloorDataGas(msg.Data)
    ...
}

// Shanghai init-code size — correctly gated
if rules.IsShanghai && contractCreation && len(msg.Data) > params.MaxInitCodeSize {
    ...
}
```

But the EIP-7702 authorization loop at line 484 has **no `rules.IsPrague` guard**:

```go
if msg.SetCodeAuthorizations != nil {
    for _, auth := range msg.SetCodeAuthorizations {
        authority, err := k.applyAuthorization(&auth, stateDB)
        ...
        validAuths = append(validAuths, validAuth{auth: auth, authority: authority})
    }
    // durable commit also runs unconditionally
    if commit && cfg.DurableSetCodeAuthorizationCtx != nil && len(validAuths) > 0 {
        ...
        k.applyDurableAuthorization(&va.auth, va.authority, durableStateDB)
        durableStateDB.Commit()
    }
}
``` [1](#0-0) 

`applyAuthorization` calls `setAuthorizationDelegation`, which unconditionally writes `SetNonce(authority, auth.Nonce+1)` and `SetCode(authority, AddressToDelegation(auth.Address))` to the stateDB: [2](#0-1) 

The durable-authorization path (`applyDurableAuthorization`) then commits these effects to the persistent KV store even if the outer EVM call reverts: [3](#0-2) 

The same gap exists in `ValidateEthBasic`. It correctly rejects `DynamicFeeTxType` when `baseFee == nil` (London not active), but has **no equivalent rejection of `SetCodeTxType` when Prague is not active**:

```go
if baseFee == nil && tx.Type() == ethtypes.DynamicFeeTxType {
    return errorsmod.Wrap(ethtypes.ErrTxTypeNotSupported, "dynamic fee tx not supported")
}
// ↓ EIP-7702 list is validated structurally but Prague activation is never checked
if tx.SetCodeAuthorizations() != nil {
    if tx.To() == nil { ... }
    if len(tx.SetCodeAuthorizations()) == 0 { ... }
}
``` [4](#0-3) 

The RPC ingress path (`SendRawTransaction`) uses `ethtypes.LatestSignerForChainID(b.chainID)` — a signer that is **not** block-height-aware and accepts type-4 transactions regardless of whether Prague is active: [5](#0-4) 

The same simulator path (`applyCall`) also processes `msg.SetCodeAuthorizations` without a `rules.IsPrague` guard: [6](#0-5) 

### Impact Explanation

When Prague is not active, `rules.IsPrague = false` and `EVMBlockConfig.Rules` reflects this. Nevertheless, a crafted type-4 transaction causes:

1. **Nonce manipulation**: `stateDB.SetNonce(authority, auth.Nonce+1)` is committed, invalidating any pending transactions the authority has in the mempool with nonce `auth.Nonce`.
2. **Unauthorized code mutation**: `stateDB.SetCode(authority, AddressToDelegation(auth.Address))` permanently installs delegation bytecode on an EOA. After this, `VerifyEthAccount` rejects the authority as a sender (the `!rules.IsPrague && acct.IsContract()` check at `ante/eth.go:93`) — effectively freezing the account from sending further transactions until Prague activates.
3. **Durable persistence**: The durable-authorization commit path ensures these effects survive even a reverted EVM call, making the state change irreversible at the Cosmos layer.

This matches the allowed High impact: **EIP-7702 authorization bypass enabling unauthorized account/code mutation**. [7](#0-6) 

### Likelihood Explanation

**Medium.** The precondition is that Prague has not yet been activated on the chain (e.g., `prague_time` is set to a future timestamp). During this window an unprivileged user can:

- Craft a valid type-4 transaction (standard go-ethereum tooling supports this regardless of chain state).
- Submit it via `eth_sendRawTransaction`; the RPC layer accepts it because `LatestSignerForChainID` is not block-height-aware.
- The ante handler passes it because `ValidateEthBasic` has no `SetCodeTxType`/Prague guard.
- `ApplyMessageWithConfig` applies the authorization effects without a `rules.IsPrague` check.

No privileged role, governance action, or validator compromise is required.

### Recommendation

1. **`ValidateEthBasic`** (`ante/interfaces/setup.go`): Add a rejection analogous to the DynamicFeeTx guard:
   ```go
   if !rules.IsPrague && tx.Type() == ethtypes.SetCodeTxType {
       return errorsmod.Wrap(ethtypes.ErrTxTypeNotSupported, "set-code tx not supported before Prague")
   }
   ```

2. **`ApplyMessageWithConfig`** (`x/evm/keeper/state_transition.go`): Gate the authorization loop:
   ```go
   if rules.IsPrague && msg.SetCodeAuthorizations != nil {
       ...
   }
   ```

3. **`applyCall` in `simulate.go`**: Apply the same `rules.IsPrague` guard.

### Proof of Concept

1. Deploy an Ethermint chain with `prague_time` set to a future block time (Prague not yet active).
2. Generate an ECDSA key pair `(authorityKey, authorityAddr)`.
3. Sign an EIP-7702 authorization: `auth = SignSetCode(authorityKey, {ChainID: chainID, Address: delegateAddr, Nonce: 0})`.
4. Build a type-4 transaction: `SetCodeTx{ChainID, Nonce: senderNonce, To: anyAddr, AuthList: [auth]}`, sign with any funded sender key.
5. Submit via `eth_sendRawTransaction`.
6. **Observe**: `eth_getCode(authorityAddr)` returns `0xef0100<delegateAddr>` and `eth_getTransactionCount(authorityAddr)` returns `1`, both committed on-chain — before Prague is active.
7. Any subsequent transaction from `authorityAddr` is rejected by `VerifyEthAccount` with "the sender is not EOA", freezing the account. [8](#0-7) [9](#0-8)

func (k *Keeper) applyAuthorization(auth *types.SetCodeAuthorization, stateDB vm.StateDB) (common.Address, error) {
	authority, err := k.validateAuthorization(auth, stateDB)
	if err != nil {
		return authority, err
	}

	// If the account already exists in state, refund the new account cost
	// charged in the intrinsic calculation.
	if stateDB.Exist(authority) {
		stateDB.AddRefund(params.CallNewAccountGas - params.TxAuthTupleGas)
	}

	k.setAuthorizationDelegation(auth, authority, stateDB)
	return authority, nil
}
```

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}
