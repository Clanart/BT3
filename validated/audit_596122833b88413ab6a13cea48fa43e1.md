### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash supplied by a contract corresponds to a previously declared class. The OS unconditionally writes the caller-supplied class hash into the contract state. Any future call to that contract will then fail irrecoverably inside the OS proof, permanently freezing all funds held by the contract.

---

### Finding Description

`execute_replace_class` in `syscall_impls.cairo` processes the `replace_class` syscall. After deducting gas, it reads the caller-supplied `class_hash` from the request and immediately writes it into `contract_state_changes` without any validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment explicitly acknowledges the missing check. The analogous prerequisite in the external report is the missing `approve()` call: just as `ASDRouter` never verified that `crocSwapAddress` had spending permission before calling `swap()`, the OS never verifies that the replacement class hash is declared before committing the state change.

When a future transaction calls the affected contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
// ...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
```

If `class_hash` was never declared, `dict_read` returns 0 (the default), and `find_element` cannot locate a compiled class with hash 0. The OS proof for any block that attempts to call the contract becomes unprovable, and the contract is permanently unexecutable.

---

### Impact Explanation

**Permanent freezing of funds.** Once a contract's class hash is replaced with an undeclared value:

1. The state change is committed in the block where `replace_class` was called (the OS proof for that block is valid because no check is performed at call time).
2. Every subsequent block that attempts to invoke the contract fails at the OS proof level.
3. All assets (tokens, ETH, NFTs) held in the contract's storage are permanently inaccessible.

This matches the "Critical – Permanent freezing of funds" impact category.

---

### Likelihood Explanation

The `replace_class` syscall is callable by any contract on itself. A malicious contract developer can deploy a contract that:

- Accepts user deposits.
- Exposes a function (or uses a time-lock / trigger) that calls `replace_class` with an arbitrary undeclared hash.
- After triggering, all deposited funds are permanently frozen.

No privileged role, leaked key, or external dependency is required. The attacker only needs to deploy a contract and submit a transaction — both are unprivileged operations open to any network participant.

---

### Recommendation

Before committing the new class hash to state, verify that it exists in `contract_class_changes` (i.e., it has been declared in the current or a prior block):

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the check already present in `execute_entry_point` and closes the gap identified in the TODO comment.

---

### Proof of Concept

1. Attacker deploys `VaultContract` that accepts ETH deposits and exposes a `freeze()` function.
2. Users deposit funds into `VaultContract`.
3. Attacker calls `freeze()`, which internally calls `replace_class(0xdeadbeef)` where `0xdeadbeef` is never declared.
4. The OS processes the block: `execute_replace_class` writes `class_hash=0xdeadbeef` into `contract_state_changes` with no validation. The block proof succeeds.
5. State is finalized: `VaultContract.class_hash = 0xdeadbeef`.
6. Any user who calls `VaultContract` in a future block triggers `execute_entry_point`, which calls `dict_read(key=0xdeadbeef)` → returns 0, then `find_element(key=0)` → not found → OS proof for that block is invalid.
7. The sequencer cannot include any transaction calling `VaultContract` in a valid block. All deposited funds are permanently frozen.
