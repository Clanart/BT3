### Title
Cross-Chain Replay of `deployToken` Signatures Due to Missing Chain ID in Message Hash ‚Äî (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`, `starknet/src/bridge_types.cairo`)

---

### Summary

The `deployToken` / `deploy_token` signed message hash omits the destination chain ID on every chain (EVM, Solana, StarkNet). Because the byte-level serialization is identical across all three chains for the same token metadata, a single MPC-signed `deployToken` message is valid on every chain simultaneously. An unprivileged attacker who observes a legitimate `deployToken` call on one chain can replay the same signature on any other chain to deploy the same wrapped token there without a new MPC authorization.

---

### Finding Description

**Asymmetry between `deployToken` and `finTransfer` hash construction**

`finTransfer` / `finalize_transfer` correctly binds the signed message to the destination chain by embedding the chain ID at two positions (token-chain and recipient-chain fields):

- EVM: `bytes1(omniBridgeChainId)` at lines 294 and 297
- Solana: `SOLANA_OMNI_BRIDGE_CHAIN_ID` at lines 30 and 35 of `finalize_transfer.rs`
- StarkNet: `chain_id` at lines 67 and 70 of `bridge_types.cairo`

`deployToken` / `deploy_token` performs no equivalent binding. The hash input on every chain is:

```
0x01  ||  borsh_string(token)  ||  borsh_string(name)  ||  borsh_string(symbol)  ||  u8(decimals)
```

**Byte-level equivalence confirmed**: The StarkNet `encode_byte_array` test (borsh.cairo line 75) shows `"hello"` ‚Üí `\x05\x00\x00\x00hello`, which is identical to Rust borsh `String` encoding and to Solidity `Borsh.encodeString`. The Solana `IncomingMessageType::Metadata` borsh-serializes to `0x01`, matching `PayloadType::Metadata` on EVM and StarkNet. All three chains hash the same byte sequence for the same token metadata.

### Impact Explanation

**High ‚Äî Proof/signature verification bypass enabling unauthorized token deployment.**

A single MPC signature authorizing `deployToken` for one chain is simultaneously valid on all other chains. An attacker can:

1. Observe a legitimate `deployToken` transaction on EVM (or any chain) and extract the `(payload, signature)` pair from calldata.
2. Submit the identical pair to Solana's `deploy_token` instruction and/or StarkNet's `deploy_token` entry point.
3. Both will pass `verify_signature` / `_verify_borsh_signature` because the hash is identical.

The result is that wrapped tokens are deployed on chains where the MPC/operator has not authorized deployment. This bypasses the intended per-chain authorization model. Users who bridge to a prematurely deployed token on an unsupported chain may find no liquidity, no relayer, or no redemption path, leading to irrecoverable fund lock. Additionally, the attacker controls the timing of deployment, which can be used to front-run legitimate deployments and interfere with bridge initialization.

---

### Likelihood Explanation

**High.** The attack requires only:
1. Monitoring any public blockchain for a `deployToken` transaction (zero-privilege, trivially automated).
2. Submitting the extracted `(payload, signature)` to another chain's bridge contract (standard user-level transaction).

No privileged keys, no MPC collusion, no leaked secrets. Any token deployment on any chain immediately creates a replayable credential for all other chains.

---

### Recommendation

Include the destination chain ID in the `deployToken` / `deploy_token` message hash, mirroring the pattern already used in `finTransfer`. Specifically:

- **EVM**: Add `bytes1(omniBridgeChainId)` to the `borshEncoded` concatenation in `deployToken`.
- **Solana**: Add `writer.write_all(&[SOLANA_OMNI_BRIDGE_CHAIN_ID])?;` in `DeployTokenPayload::serialize_for_near`.
- **StarkNet**: Add `borsh_bytes.append_byte(self.omni_bridge_chain_id.read());` (or pass `chain_id` as a parameter to `to_borsh`) in `MetadataPayload::to_borsh`.

The NEAR MPC signer must include the target chain ID when producing the signature so that each chain's signature is cryptographically distinct.

```rust
impl<P: Payload> SignedPayload<P> {
    pub fn verify_signature(
        &self,
        params: P::AdditionalParams,
        derived_near_bridge_address: &[u8; 64],
    ) -> Result<()> {
        let serialized = self.payload.serialize_for_near(params)?;
        let hash = keccak::hash(&serialized);

        let signature_bytes = &self.signature[0..64];

        let signature = libsecp256k1::Signature::parse_standard_slice(signature_bytes)
            .map_err(|_| ProgramError::InvalidArgument)?;
        require!(!signature.s.is_high(), ErrorCode::MalleableSignature);

        let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)
            .map_err(|_| error!(ErrorCode::SignatureVerificationFailed))?;

        require!(
            signer.0 == *derived_near_bridge_address,
            ErrorCode::SignatureVerificationFailed
        );

        Ok(())
    }
```
