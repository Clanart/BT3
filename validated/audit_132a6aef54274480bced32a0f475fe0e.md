### Title
`ProofOfInclusion::valid()` Never Validates Against an External Trusted Root Hash, Allowing Forged DataLayer Inclusion Proofs — (`File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate only checks internal self-consistency of the proof chain. The final comparison `existing_hash == self.root_hash()` is a tautology — `self.root_hash()` is derived from the proof's own last `combined_hash`, which was already verified in the loop. No external trusted root hash is ever compared. An unprivileged attacker can construct a fully fabricated `ProofOfInclusion` for any arbitrary `node_hash` (i.e., any claimed key-value pair), and `valid()` will return `true`.

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()  // ← tautology
}
``` [1](#0-0) 

And `root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← comes from the proof itself
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

**The tautology:** After the loop completes without returning `false`, `existing_hash` holds the last `calculated_hash`. The loop already asserted `calculated_hash == layer.combined_hash` for every layer, so after the last iteration `existing_hash == last.combined_hash`. `self.root_hash()` also returns `last.combined_hash`. Therefore `existing_hash == self.root_hash()` is always `true` when layers are present — it adds zero validation.

The function never accepts an external trusted root hash as a parameter. The "root" it validates against is entirely attacker-controlled (it is part of the proof itself).

Contrast this with the consensus-layer `validate_merkle_proof` in `merkle_tree.rs`, which correctly takes an external `root` parameter and rejects any proof whose computed root does not match:

```rust
pub fn validate_merkle_proof(
    proof: &[u8],
    item: &[u8; 32],
    root: &[u8; 32],          // ← external trusted root
) -> Result<bool, SetError> {
    let tree = MerkleSet::from_proof(proof)?;
    if tree.get_root() != *root {
        return Err(SetError);
    }
    Ok(tree.generate_proof(item)?.0)
}
``` [3](#0-2) 

The DataLayer `ProofOfInclusion` is `Streamable` (serializable/deserializable) and its `valid()` method is exposed directly through Python bindings: [4](#0-3) [5](#0-4) 

### Impact Explanation

Any caller that receives a `ProofOfInclusion` from an untrusted source and calls `valid()` to verify it will accept a completely fabricated proof. An attacker can:

1. Choose any `node_hash` (claiming any key-value pair is included in the DataLayer store).
2. Build one or more layers where each `combined_hash = calculate_internal_hash(prev_hash, side, other_hash)` using arbitrary `other_hash` values.
3. Submit this proof; `valid()` returns `true`.

This allows an untrusted party to prove invalid DataLayer state — claiming a key-value mapping exists in a committed tree when it does not. This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

### Likelihood Explanation

`ProofOfInclusion` is `Streamable` and fully exposed through Python bindings. Any DataLayer client that receives proofs over the network and calls `proof.valid()` without separately comparing `proof.root_hash()` against a locally-trusted root is vulnerable. The API design actively encourages this misuse — `valid()` appears to be a complete verification function, but it is not. The fuzz target and all tests call `proof.valid()` without any external root check, confirming this is the intended (but broken) API. [6](#0-5) 

### Recommendation

`valid()` must accept an external trusted root hash and compare against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }

    &existing_hash == trusted_root   // compare against external trusted root
}
```

The no-argument `valid()` should either be removed or clearly documented as only checking internal self-consistency (not authenticity). All callers that receive proofs from untrusted sources must supply the locally-committed root hash.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side};
use chia_datalayer::ProofOfInclusion;
use chia_datalayer::ProofOfInclusionLayer;
use chia_datalayer::blob::calculate_internal_hash;

// Attacker wants to forge a proof that some arbitrary node_hash is in the tree.
let fake_node_hash = Hash([0xAA; 32]);  // arbitrary claimed leaf hash
let fake_other_hash = Hash([0xBB; 32]); // arbitrary sibling hash

// Compute a combined_hash that is internally consistent
let fake_combined = calculate_internal_hash(&fake_node_hash, Side::Right, &fake_other_hash);

let forged_proof = ProofOfInclusion {
    node_hash: fake_node_hash,
    layers: vec![ProofOfInclusionLayer {
        other_hash_side: Side::Right,
        other_hash: fake_other_hash,
        combined_hash: fake_combined,  // attacker controls this
    }],
};

// valid() returns true for a completely fabricated proof
assert!(forged_proof.valid());
// forged_proof.root_hash() == fake_combined, which is not the real tree root
```

The proof is accepted as valid despite having no relationship to any real `MerkleBlob` state.
