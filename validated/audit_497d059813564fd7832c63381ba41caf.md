### Title
Silent Stale ETH/STRK Rate Fallback Produces Wrong `price_in_fri` in Block Hash and Fee Accounting — (`crates/apollo_l1_gas_price/src/eth_to_strk_oracle.rs`)

---

### Summary

`EthToStrkOracleClient::eth_to_fri_rate` silently returns the **previous** quantized-interval rate when the current interval's HTTP query is still in-flight. The caller `get_l1_prices_in_fri_and_wei_and_conversion_rate` receives an `Ok(stale_rate)` — indistinguishable from a fresh rate — and uses it to compute `price_in_fri` for every gas token. Those stale FRI prices are locked into `PartialBlockHashComponents`, committed into the block hash via `gas_prices_to_hash`, and used to charge fees for every transaction in the block. The production `lag_interval_seconds` is 900 s, so the silently-reused rate can be up to 1 800 s (30 min) old.

---

### Finding Description

**Root cause — `eth_to_fri_rate` silent stale fallback**

With `lag_interval_seconds = 900` (production config), `quantized_timestamp = (T − 900) / 900`. The fallback returns the rate for `(quantized_timestamp − 1) × 900` seconds — up to **1 800 s** before the block timestamp.

**Propagation — stale rate treated as authoritative**

`get_l1_prices_in_fri_and_wei_and_conversion_rate` (utils.rs:147-174) calls `get_eth_to_fri_rate` and, on `Ok(rate)`, immediately computes:

```rust
let l1_gas_prices_fri_result = L1PricesInFri::convert_from_wei(&prices_in_wei, eth_to_fri_rate);
```

There is no staleness check. The stale rate bypasses the explicit fallback chain (previous block info → defaults) because the function returns `Ok`, not `Err`.

**Commitment — stale FRI prices enter the block hash**

`BlockExecutionArtifacts::new` (block_builder.rs:168-169) calls `PartialBlockHashComponents::new(&block_info, header_commitments)`, which captures `l1_gas_price.price_in_fri`, `l1_data_gas_price.price_in_fri`, and `l2_gas_price.price_in_fri` from `block_info.gas_prices` (block_hash_calculator.rs:228-230). `calculate_block_hash` then chains these through `gas_prices_to_hash` (block_hash_calculator.rs:265-273), permanently committing the stale FRI prices into the block hash.

---

### Impact Explanation

Every transaction in the affected block is charged fees computed from a stale ETH/STRK rate. If the rate has risen (ETH more expensive in STRK terms), `price_in_fri` is understated — users pay less STRK than the ETH-denominated cost warrants, and the sequencer absorbs the shortfall. If the rate has fallen, users overpay. Because the stale rate is committed into the block hash, the RPC, fee-estimation endpoint, and any proof-input that reads `gas_prices_hash` all reflect the wrong authoritative value. The block hash itself is a commitment to incorrect gas prices, which diverges from what an honest verifier would compute from the live oracle.

Matching impacts:
- **Critical** — Incorrect fee / resource accounting with economic impact (wrong `price_in_fri` applied to every transaction's fee).
- **High** — RPC fee estimation returns an authoritative-looking wrong value (derived from the same stale rate).

---

### Likelihood Explanation

The condition fires at the **start of every new 900-second quantized bucket** — a recurring, unprivileged, normal-operation event. No attacker action is required; the oracle simply hasn't finished its HTTP query for the new interval yet. With a 10-second query timeout and blocks produced every few seconds, the window spans multiple consecutive blocks per interval. The ETH/STRK rate can move 0.5–2 % in 30 minutes under normal market conditions, and more during volatility.

---

### Recommendation

1. **Propagate staleness to the caller.** Return `Err(QueryNotReadyError)` instead of silently returning the previous rate. The existing fallback chain in `get_l1_prices_in_fri_and_wei_and_conversion_rate` (previous block info → defaults) is already designed to handle oracle errors and will produce a more consistent result.

2. **Or tag the returned rate.** Return a `(rate, is_stale: bool)` pair so the caller can decide whether to accept it or fall through to the previous-block-info path.

3. **Bound the staleness.** If the silent fallback is intentional, enforce a maximum age: reject the cached rate if `(quantized_timestamp - NUMBER_OF_TIMESTAMPS_BACK) * lag_interval_seconds` is more than one interval behind the requested timestamp.

---

### Proof of Concept

1. At wall-clock time `T`, a new quantized bucket begins: `q = (T − 900) / 900` increments.
2. The oracle spawns a new HTTP query for `q`; the query is in-flight (not finished).
3. The proposer calls `get_eth_to_fri_rate(T)`.
4. `eth_to_fri_rate` finds `handle.is_finished() == false`, looks up `cache[q − 1]` (the previous interval's rate, cached from the prior bucket), and returns `Ok(stale_rate)`.
5. `get_l1_prices_in_fri_and_wei_and_conversion_rate` receives `Ok(stale_rate)`, computes `price_in_fri = wei_price * stale_rate`, and returns it as authoritative.
6. `BlockExecutionArtifacts::new` → `PartialBlockHashComponents::new` captures the stale `price_in_fri` values.
7. `calculate_block_hash` commits them into the block hash via `gas_prices_to_hash`.

        const NUMBER_OF_TIMESTAMPS_BACK: u64 = 1;
        let quantized_timestamp = (timestamp - self.config.lag_interval_seconds)
            .checked_div(self.config.lag_interval_seconds)
            .expect("lag_interval_seconds should be non-zero");

        let mut cache = self.cached_prices.lock().unwrap();

        if let Some(rate) = cache.get(&quantized_timestamp) {
            debug!("Cached conversion rate for timestamp {timestamp} is {rate}");
            return Ok(*rate);
        }

        // Check if there is a query already sent out for this timestamp, if not, start one.
        let mut queries = self.queries.lock().unwrap();
        let handle = queries
            .get_or_insert_mut(quantized_timestamp, || self.spawn_query(quantized_timestamp));
        // If the query is not finished, return an error.
        if !handle.is_finished() {
            debug!("Query not yet resolved: timestamp={timestamp}");
            // If the previous quantized timestamp is in the cache, use it.
