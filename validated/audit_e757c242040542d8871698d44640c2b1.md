### Title
`within_margin` computes margin relative to proposed price instead of reference price, allowing proposers to submit inflated L1 gas prices — (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The `within_margin` function used to validate L1 gas prices in block proposals computes the acceptable deviation as a percentage of the **proposed** price (`number1`) rather than the **reference** price (`number2`). This is the direct sequencer analog of the Velocimeter `(PRECISION - teamRate)` denominator bug: the wrong base value is used in the percentage calculation, producing an asymmetric bound that is systematically more permissive when the proposed price is above the reference. A proposer can submit an L1 gas price ~11.11% above the reference while the intended ceiling is 10%, and every validator will accept the proposal. The inflated price is then committed into the block hash, stored in the block header, and used for all fee calculations in that block.

---

### Finding Description

`within_margin` is defined at line 323 of `validate_proposal.rs`:

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;   // ← wrong base
    number1.0.abs_diff(number2.0) <= margin
}
```

It is called at lines 302–307 with the proposed price as `number1` and the oracle reference price as `number2`:

```rust
within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
```

The margin is therefore `proposed × 10 / 100`. When `proposed > reference`, this margin is larger than `reference × 10 / 100`, so the check passes for a wider range of proposed values than intended. The correct base is `number2` (the reference price), not `number1` (the proposed price).

The `l1_gas_price_margin_percent` constant is 10 in every deployed versioned-constants file (`orchestrator_versioned_constants_0_14_0/1/2.json`).

The validated prices flow directly into `calculate_block_hash` (via `PartialBlockHashComponents::new` → `gas_prices_to_hash`) and into every fee computation for transactions in the block (`get_fee_by_gas_vector`).

---

### Impact Explanation

**Wrong L1 gas price accepted into block commitment.** A proposer that sets `l1_gas_price_fri_proposed = reference × 10/9` (≈ 11.11 % above reference) satisfies:

```
margin = (reference × 10/9) × 10 / 100 = reference / 9
|proposed − reference| = reference / 9  ≤  margin  → PASSES
```

With the correct formula the same price fails:

```
margin = reference × 10 / 100 = reference / 10
|proposed − reference| = reference / 9  >  reference / 10  → FAILS
```

The accepted inflated price is then:

1. **Committed into the block hash** via `calculate_block_hash` (chained into the Poseidon hash at the `gas_prices_to_hash` step), producing a block commitment that diverges from what honest nodes would compute if they enforced the correct bound.
2. **Used for all fee charges** in the block (`get_fee_by_gas_vector` multiplies gas amounts by the block's L1 gas price), so every transaction in the block is overcharged by up to ~1.11 % relative to the intended ceiling.
3. **Stored in `StorageBlockHeader.l1_gas_price`** and propagated to P2P sync peers and RPC callers, so `starknet_estimateFee` and `starknet_getBlockWithTxs` return an authoritative-looking but inflated price.

The same asymmetry applies to `l1_data_gas_price_fri`, which is validated with the same call.

---

### Likelihood Explanation

Any consensus participant that is selected as a proposer for a round can trigger this. No privileged access beyond normal proposer selection is required. The proposer simply sets both `l1_gas_price_fri` and `l1_data_gas_price_fri` in the `ProposalInit` message to `reference × 10/9`. All validators run `is_block_info_valid`, compute the same asymmetric margin, and accept the proposal. The exploit is deterministic and requires no brute-force or timing.

---

### Recommendation

Replace the margin base with the reference price (`number2`):

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
-   let margin = (number1.0 * margin_percent) / 100;
+   let margin = (number2.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}
```

This makes the allowed band symmetric around the reference price, matching the documented intent of `l1_gas_price_margin_percent` as a tolerance on the oracle-derived value.

---

### Proof of Concept

**Setup:** `l1_gas_price_margin_percent = 10` (all deployed constants files). Oracle reference price = 1 000 FRI.

**Intended upper bound:** 1 000 × 1.10 = **1 100 FRI**.

**Actual upper bound with current code:**

```
proposed = 1 111
margin   = (1111 × 10) / 100 = 111
|1111 − 1000| = 111  ≤  111  → PASSES   ← should be rejected
```

```
proposed = 1 112
margin   = (1112 × 10) / 100 = 111
|1112 − 1000| = 112  >  111  → FAILS
```

So the effective ceiling is **1 111 FRI** (11.11 % above reference) instead of 1 100 FRI.

**Fee impact:** A transaction consuming 1 000 000 L1 gas units pays `1 111 × 1 000 000 = 1.111 × 10⁹ FRI` instead of the intended maximum of `1 100 × 1 000 000 = 1.1 × 10⁹ FRI` — an overcharge of ~1 % per transaction, compounding across every transaction in every block proposed by the malicious validator.

**Block hash impact:** `calculate_block_hash` chains the inflated `l1_gas_price` field into the Poseidon hash. The resulting `BlockHash` differs from what would be produced under the correct 10 % bound, so any external verifier or proof system that independently enforces the 10 % rule will compute a mismatching commitment. fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
```rust
    // For small numbers (e.g., less than 10 wei, if margin is 10%), even an off-by-one
    // error might be bigger than the margin, even if it is just a rounding error.
    // We make an exception for such mismatch, and don't bother checking percentages
    // if the difference in price is only one wei.
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}
```

```rust
pub fn calculate_block_hash(
    partial_block_hash_components: &PartialBlockHashComponents,
    state_root: GlobalRoot,
    previous_block_hash: BlockHash,
) -> StarknetApiResult<BlockHash> {
    let block_hash_version: BlockHashVersion =
        partial_block_hash_components.starknet_version.try_into()?;
    let block_commitments = &partial_block_hash_components.header_commitments;
    Ok(BlockHash(
        HashChain::new()
            .chain(&block_hash_version.clone().into())
            .chain(&partial_block_hash_components.block_number.0.into())
            .chain(&state_root.0)
            .chain(&partial_block_hash_components.sequencer.0)
            .chain(&partial_block_hash_components.timestamp.0.into())
            .chain(&block_commitments.concatenated_counts)
            .chain(&block_commitments.state_diff_commitment.0.0)
            .chain(&block_commitments.transaction_commitment.0)
            .chain(&block_commitments.event_commitment.0)
            .chain(&block_commitments.receipt_commitment.0)
            .chain_iter(
                gas_prices_to_hash(
                    &partial_block_hash_components.l1_gas_price,
                    &partial_block_hash_components.l1_data_gas_price,
                    &partial_block_hash_components.l2_gas_price,
                    &block_hash_version,
                )
                .iter(),
            )
            .chain(
                &Felt::try_from(&partial_block_hash_components.starknet_version)
                    .expect("Expect ASCII version"),
            )
            .chain(&Felt::ZERO)
            .chain(&previous_block_hash.0)
            .get_poseidon_hash(),
    ))
}
```
