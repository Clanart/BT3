### Title
Missing Destination-Chain Domain Separator in `deployToken` Signature Allows Cross-Chain Replay — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.deployToken` verifies a NEAR-bridge ECDSA signature over a hash that contains only token metadata (type byte, token ID, name, symbol, decimals). It omits any chain-specific identifier (`omniBridgeChainId`, EVM `block.chainid`, or contract address). Because the same `nearBridgeDerivedAddress` is configured on every EVM deployment, a signature produced for one chain is cryptographically valid on every other EVM chain where the bridge is deployed, enabling an unprivileged attacker to replay it and deploy an unauthorized bridge token.

---

### Finding Description

`deployToken` constructs its signed digest as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

The hash contains **no** chain-specific field. Contrast this with `finTransfer`, which correctly embeds `omniBridgeChainId` twice in its digest:

```solidity
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.tokenAddress),
Borsh.encodeUint128(payload.amount),
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.recipient),
``` [2](#0-1) 

Because `nearBridgeDerivedAddress` is a single NEAR-derived key shared across all EVM deployments (set identically at `initialize` time on each chain), any `deployToken` signature that passes on Ethereum is byte-for-byte valid on BSC, Arbitrum, or any other EVM chain running the same contract. [3](#0-2) 

After a successful replay the contract unconditionally writes:

```solidity
isBridgeToken[address(bridgeTokenProxy)] = true;
ethToNearToken[address(bridgeTokenProxy)] = metadata.token;
nearToEthToken[metadata.token] = address(bridgeTokenProxy);
``` [4](#0-3) 

This permanently registers the replayed token as a legitimate bridge token on the target chain.

---

### Impact Explanation

Once the replayed token is registered:

- `isBridgeToken` is `true`, so any subsequent `finTransfer` call on the target chain that references this token address will **mint tokens** to the recipient (the `finTransfer` path for bridge tokens calls `IBridgeToken.mint`).
- `nearToEthToken[metadata.token]` is overwritten, so the canonical NEAR→EVM mapping on the target chain now points to the attacker-triggered deployment rather than any future official one.
- The NEAR bridge has no record of authorizing this deployment on the target chain, meaning the on-chain supply of the replayed token is unbacked.

This satisfies the allowed impact: **High — signature verification bypass enabling unauthorized token deployment**, and potentially escalates to **Critical — unauthorized mint of bridged assets** if a `finTransfer` signature (which is chain-scoped) is later obtained for the target chain.

---

### Likelihood Explanation

- The attacker needs only a valid `deployToken` calldata (signature + metadata) from any one EVM chain where the token has been legitimately deployed. This is fully public on-chain data.
- No privileged access, leaked key, or colluding party is required.
- The only precondition is that the same `nearBridgeDerivedAddress` is used on the target chain, which is the standard deployment pattern.

Likelihood: **Medium** (requires the bridge to be live on at least two EVM chains, which is the intended multi-chain design).

---

### Recommendation

Include `omniBridgeChainId` (and optionally `address(this)`) in the `deployToken` signed digest, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to destination chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

The NEAR bridge must include the same field when producing the signature. This is a zero-storage-cost fix identical in spirit to the external report's recommendation of adding `questID` and `period` to the Merkle leaf.

---

### Proof of Concept

1. Token `"usdc.near"` is legitimately deployed on Ethereum via `deployToken(sig_eth, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})`. The transaction is public; attacker extracts `sig_eth` and the metadata struct.

2. Attacker calls `deployToken(sig_eth, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on BSC, where the same `nearBridgeDerivedAddress` is configured.

3. `keccak256(borshEncoded)` produces the identical 32-byte digest on BSC (no chain field), so `ECDSA.recover` returns `nearBridgeDerivedAddress` — the check passes.

4. A fresh `BridgeToken` proxy is deployed on BSC; `isBridgeToken[proxy] = true` and `nearToEthToken["usdc.near"] = proxy` are written.

5. The BSC chain now has an unauthorized, NEAR-untracked USDC bridge token. Any `finTransfer` signed for BSC that references this token address will mint unbacked USDC to the specified recipient. [5](#0-4)

    function deployToken(
        bytes calldata signatureData,
        BridgeTypes.MetadataPayload calldata metadata
    ) external payable whenNotPaused(PAUSED_DEPLOY_TOKEN) returns (address) {
        if (tokenImplementationAddress == address(0)) {
            revert TokenImplementationNotSet();
        }
        bytes memory borshEncoded = bytes.concat(
            bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
            Borsh.encodeString(metadata.token),
            Borsh.encodeString(metadata.name),
            Borsh.encodeString(metadata.symbol),
            bytes1(metadata.decimals)
        );
        bytes32 hashed = keccak256(borshEncoded);

        if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
            revert InvalidSignature();
        }

        require(
            !isBridgeToken[nearToEthToken[metadata.token]],
            "ERR_TOKEN_EXIST"
        );
        uint8 decimals = _normalizeDecimals(metadata.decimals);

        // slither-disable-next-line reentrancy-no-eth
        address bridgeTokenProxy = address(
            new ERC1967Proxy(
                tokenImplementationAddress,
                abi.encodeWithSelector(
                    BridgeToken.initialize.selector,
                    metadata.name,
                    metadata.symbol,
                    decimals
                )
            )
        );

        deployTokenExtension(
            metadata.token,
            bridgeTokenProxy,
            decimals,
            metadata.decimals
        );

        emit BridgeTypes.DeployToken(
            bridgeTokenProxy,
            metadata.token,
            metadata.name,
            metadata.symbol,
            decimals,
            metadata.decimals
        );

        isBridgeToken[address(bridgeTokenProxy)] = true;
        ethToNearToken[address(bridgeTokenProxy)] = metadata.token;
        nearToEthToken[metadata.token] = address(bridgeTokenProxy);

        return bridgeTokenProxy;
    }
