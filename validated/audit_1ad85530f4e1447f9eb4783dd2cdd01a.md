### Title
Fee Token `transfer` Return Value Silently Discarded in `charge_fee` — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `charge_fee` function in `transaction_impls.cairo` invokes the fee token's `transfer` entry point via `non_reverting_select_execute_entry_point_func` but **completely discards the return value**. The OS only asserts that the call did not revert; it never verifies that the transfer actually succeeded (i.e., that the return value is `true`/`1`). If the fee token's `transfer` returns `false` without reverting, the OS treats the fee as charged, executes the transaction, and produces a valid proof — while the sequencer never receives the fee.

---

### Finding Description

In `charge_fee`, the fee token's `transfer` entry point is dispatched at line 161:

```cairo
let remaining_gas = DEFAULT_INITIAL_GAS_COST;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=&execution_context
);
return ();
``` [1](#0-0) 

`non_reverting_select_execute_entry_point_func` is declared to return `(retdata_size: felt, retdata: felt*, is_deprecated: felt)`: [2](#0-1) 

The function internally asserts `is_reverted = 0` (the call must not revert), but it **returns the retdata to the caller** for the caller to inspect. In `charge_fee`, those three return values are silently dropped — Cairo allows discarding return values without a compile error.

The fee token's `transfer` selector is `TRANSFER_ENTRY_POINT_SELECTOR` (the `transfer` function), which per the StarkNet ERC20 standard (SNIP-2) returns a `bool`. A conforming but non-reverting-on-failure implementation would return `false` (felt `0`) when the transfer fails. The OS never reads this value.

**Contrast with every other critical entry point call in the same OS:**

`run_validate` captures and asserts the return value:
```cairo
let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func(...);
if (is_deprecated == 0) {
    assert retdata_size = 1;
    assert retdata[0] = VALIDATED;
}
``` [3](#0-2) 

`execute_deploy_account_transaction` does the same for `__validate_deploy__`: [4](#0-3) 

`execute_declare_transaction` does the same for `__validate_declare__`: [5](#0-4) 

`charge_fee` is the **only** critical OS-level entry point call that discards the return value entirely.

The `TRANSFER_ENTRY_POINT_SELECTOR` constant confirms the OS is calling the ERC20 `transfer` function: [6](#0-5) 

`charge_fee` is called for all three account transaction types — invoke, deploy-account, and declare: [7](#0-6) [8](#0-7) [9](#0-8) 

---

### Impact Explanation

**Impact: Critical — Direct loss of funds.**

If the fee token's `transfer` function returns `false` without reverting:

1. `non_reverting_select_execute_entry_point_func` does not panic (since `is_reverted = 0` holds).
2. The return value `retdata[0] = 0` (false) is silently discarded.
3. The OS proceeds as if the fee was successfully charged.
4. The transaction is executed, state changes are committed, and a valid STARK proof is generated and accepted on L1.
5. The sequencer's balance in the fee token is **never incremented** — the fee is not transferred.

Every transaction processed under this condition results in the sequencer providing execution service for free. Over many transactions, this constitutes a direct, provable loss of fee revenue that is cryptographically finalized on L1 and cannot be reversed.

---

### Likelihood Explanation

**Likelihood: Low.**

The fee token address is set at the protocol level in `StarknetOsConfig.fee_token_address` and is not user-controlled. The current StarkNet fee tokens (STRK, ETH bridged) use standard OpenZeppelin-style implementations that **revert** on transfer failure rather than returning `false`. An unprivileged user cannot substitute the fee token.

However, the risk is non-zero because:
- The fee token contract can be upgraded (it is a proxy or replaceable contract in practice).
- A future upgrade or a bug in the fee token that causes it to return `false` without reverting would silently activate this vulnerability for every subsequent transaction.
- The OS code provides no defense-in-depth: the missing check means any deviation from "revert on failure" in the fee token immediately becomes exploitable.

---

### Recommendation

Capture and validate the return value of the fee token's `transfer` call in `charge_fee`, mirroring the pattern used for `__validate__`, `__validate_declare__`, and `__validate_deploy__`:

```cairo
let remaining_gas = DEFAULT_INITIAL_GAS_COST;
let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func{
    remaining_gas=remaining_gas
}(block_context=block_context, execution_context=&execution_context);
// For non-deprecated fee tokens, assert the transfer returned true.
if (is_deprecated == 0) {
    assert retdata_size = 1;
    assert retdata[0] = 1;  // TRUE / success
}
return ();
```

This ensures that a fee token returning `false` without reverting is caught at the OS level and causes the proof to be invalid, preventing fee-free transaction execution.

---

### Proof of Concept

1. The fee token at `fee_token_address` is upgraded (or is buggy) so that its `transfer` function returns `felt 0` (false) without reverting when the sender has insufficient balance or under any attacker-chosen condition.
2. An attacker submits an invoke transaction (v3) with valid signature and nonce but with a fee token balance of 0.
3. The OS executes `execute_invoke_function_transaction`:
   - `run_validate` passes (signature is valid).
   - `__execute__` runs and state changes are applied.
   - `charge_fee` is called: it calls `transfer(sequencer, amount)` on the fee token.
   - The fee token returns `false` (felt 0) without reverting.
   - `non_reverting_select_execute_entry_point_func` asserts `is_reverted = 0` — passes.
   - The return value `(1, [0], 0)` is discarded.
   - `charge_fee` returns normally.
4. The block is finalized, the STARK proof is generated, and submitted to L1.
5. L1 verifies the proof and accepts the block.
6. The sequencer's fee token balance is unchanged — the fee was never paid.
7. The attacker's transaction was executed at zero cost.
