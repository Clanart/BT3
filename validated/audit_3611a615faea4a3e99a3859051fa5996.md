Audit Report

## Title
No Factory Validation for Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User Tokens via Callback — (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

`MetricOmmPoolLiquidityAdder` accepts a caller-supplied `pool` address in every `addLiquidity*` entry point without verifying it against `MetricOmmPoolFactory`. Because the callback `metricOmmModifyLiquidityCallback` trusts `msg.sender` solely by comparing it to the user-supplied pool address stored in transient context, a malicious pool can call back with `KIND_PAY` and drain up to `maxAmountToken0` of any `token0` and `maxAmountToken1` of any `token1` from the user — where both token addresses are chosen by the attacker via `getImmutables()`. The contract's own NatSpec explicitly acknowledges this gap.

## Finding Description

Every `addLiquidity*` entry point accepts an arbitrary `pool` address with no factory check. The constructor takes only `weth`: [1](#0-0) 

`_addLiquidity` immediately stores the caller-supplied pool as the trusted callback caller in transient storage: [2](#0-1) 

`metricOmmModifyLiquidityCallback` validates only that `msg.sender == expectedPool` — the address the user supplied: [3](#0-2) 

After that check passes, token addresses are fetched **from the pool itself** via `getImmutables()`, and `pay` is called unconditionally: [4](#0-3) 

A malicious pool controls both the `callbackData` it passes back (trivially encoding `KIND_PAY = 1`) and the `token0`/`token1` returned from `getImmutables()`. The only guards are the `max0`/`max1` caps, which the user sets themselves believing they are interacting with a legitimate pool.

The `addLiquidityWeighted` probe path (lines 106–110) is **not** an additional attack surface: `_setPayContext` has not been called during the probe, so `expectedPool == address(0)` and the `CallbackContextNotActive` revert fires. The vulnerability is confined to the paying call inside `_addLiquidity`.

The factory already exposes `isPool(address) → bool` via `poolToIdx[pool] != 0`, making a fix straightforward: [5](#0-4) 

The NatSpec at lines 19–21 explicitly acknowledges the missing check: [6](#0-5) 

## Impact Explanation

Direct loss of user principal. A user who calls `addLiquidityExactShares` or `addLiquidityWeighted` with a malicious pool address loses up to `maxAmountToken0` of an attacker-chosen `token0` and `maxAmountToken1` of an attacker-chosen `token1`. The user receives no LP position. Because `LiquidityAdder` is a general-purpose periphery contract that users must approve for their tokens, the attack surface is broad. This meets the "Critical/High/Medium direct loss of user principal" allowed impact gate.

## Likelihood Explanation

Medium. The attack requires a user to supply a malicious pool address, which is realistic via a phishing UI substituting a fake pool address, a third-party integration that does not whitelist pools, or a user copy-pasting an address from an untrusted source. No privileged role is required; any unprivileged attacker can deploy a contract implementing the pool interface.

## Recommendation

Add a `factory` immutable to `MetricOmmPoolLiquidityAdder` and validate every caller-supplied `pool` before `_setPayContext` or any pool interaction:

```solidity
address public immutable factory;

constructor(address weth, address _factory) PeripheryPayments(weth) {
    factory = _factory;
}

function _requireFactoryPool(address pool) internal view {
    if (!IMetricOmmPoolFactory(factory).isPool(pool)) revert NotAFactoryPool(pool);
}
```

Call `_requireFactoryPool(pool)` at the top of every `addLiquidity*` entry point. `IMetricOmmPoolFactory.isPool` is already implemented in `MetricOmmPoolFactory` and returns `poolToIdx[pool] != 0`.

## Proof of Concept

```
1. Attacker deploys MaliciousPool implementing IMetricOmmPoolActions and IMetricOmmPool:
   - addLiquidity(...) calls back metricOmmModifyLiquidityCallback(max0, max1, abi.encode(uint8(1)))
   - getImmutables() returns token0=USDC, token1=WETH (attacker-chosen)

2. Victim approves LiquidityAdder for USDC (max0) and WETH (max1).

3. Victim calls adder.addLiquidityExactShares(maliciousPool, victim, 0, deltas, max0, max1, "")

4. adder._addLiquidity: _setPayContext(maliciousPool, victim, max0, max1) — stores maliciousPool as expectedPool

5. adder calls maliciousPool.addLiquidity(...)

6. maliciousPool calls back metricOmmModifyLiquidityCallback(max0, max1, abi.encode(KIND_PAY=1))

7. Callback: msg.sender==expectedPool ✓, amounts<=caps ✓

8. Callback: imm = maliciousPool.getImmutables() → token0=USDC, token1=WETH

9. pay(USDC, victim, maliciousPool, max0) + pay(WETH, victim, maliciousPool, max1)

Result: victim loses max0 USDC + max1 WETH; receives no LP position.
```
