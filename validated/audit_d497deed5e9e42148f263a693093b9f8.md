### Title
Empty `check_is_reverted` Leaves Transaction Revert Status Unconstrained in ZK Proof — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

`check_is_reverted` in `execution_constraints.cairo` is completely empty — it accepts the hint-supplied `is_reverted` value and immediately returns without any assertion. Because the Cairo OS is a ZK proof program, every hint must be constrained by Cairo assertions to be sound. Since `is_reverted` is set exclusively by the `%{ IsReverted %}` hint and the only "validation" function is a no-op, a malicious prover can freely set `is_reverted` to any value and still produce a valid proof. This is the direct analog of the external report's pattern: a check that is *supposed* to enforce a critical invariant but is entirely ineffective.

---

### Finding Description

In `execution_constraints.cairo`, the function meant to validate the revert status of a transaction is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [1](#0-0) 

It contains no assertion, no range check, and no constraint of any kind.

This function is called in `execute_invoke_function_transaction` in `transaction_impls.cairo` immediately after the hint fills `is_reverted`:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...);
} else {
    // skip execution, align stack
    ...
}
``` [2](#0-1) 

In Cairo's constraint system, a `local` variable filled only by a hint and never constrained by any `assert` is a **free variable** — the prover can assign it any value and the proof remains valid. `check_is_reverted` is the sole designated place to enforce this constraint, and it does nothing.

Fee charging always runs after the branch, regardless of `is_reverted`:

```cairo
// Charge fee.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [3](#0-2) 

`charge_fee` enforces only that `actual_fee ≤ max_fee`:

```cairo
assert_nn_le(calldata.amount.low, max_fee);
``` [4](#0-3) 

It does not verify that execution actually occurred.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A malicious prover can set `is_reverted = TRUE` for any transaction that was not actually reverted. The result:

- The `__execute__` entry point is skipped entirely.
- `charge_fee` still runs and transfers `actual_fee` from the user's account to the sequencer.
- The user's intended state change never happens, but they are charged as if it did.

This is a provably valid proof of an invalid state transition: the L1 verifier accepts the proof, the fee transfer is committed on-chain, and the user's funds are permanently lost with no recourse.

The inverse is also exploitable: setting `is_reverted = FALSE` for a transaction that genuinely reverted causes the OS to apply state changes that should have been rolled back, enabling arbitrary invalid state transitions.

---

### Likelihood Explanation

The StarkNet OS is a ZK proof program. Its entire security model rests on the assumption that every hint is constrained by Cairo assertions. The OS itself acknowledges this in `os.cairo`:

> *"The following code deals with the problem that untrusted code (contract code) could potentially move builtin pointers backward, compromising the soundness of those builtins."* [5](#0-4) 

The empty `check_is_reverted` is a direct violation of this principle. Any prover who runs the OS program can exploit this without any special access — the hint value is under the prover's control by design, and the constraint that was supposed to validate it is absent.

---

### Recommendation

`check_is_reverted` must assert that `is_reverted` is a valid boolean (0 or 1) **and** that its value is consistent with the actual execution outcome. At minimum:

```cairo
func check_is_reverted(is_reverted: felt) {
    assert is_reverted * (1 - is_reverted) = 0;  // must be boolean
    // Additionally, tie is_reverted to the actual execution result
    // via a storage read or execution trace constraint.
    return ();
}
```

The revert status must be derived from a constrained source (e.g., the execution trace, a storage flag written during execution) rather than accepted as a free hint.

---

### Proof of Concept

1. User submits an invoke transaction with `max_l2_gas_amount = X` and `max_price_per_unit = P`.
2. The transaction executes successfully off-chain (not reverted).
3. Malicious prover sets hint `IsReverted = 1` (TRUE).
4. `check_is_reverted(1)` is called — returns immediately, no assertion fires.
5. The `if (is_reverted == FALSE)` branch is skipped; `__execute__` is never called.
6. `charge_fee` runs, transferring `actual_fee` (up to `X * P`) from the user to the sequencer.
7. The proof is submitted to L1 and accepted — it is cryptographically valid because `is_reverted` was never constrained.
8. User loses `actual_fee` worth of tokens; their intended state change never occurred. [1](#0-0) [6](#0-5)
