Audit Report

## Title
Router-Mediated Swaps Bypass `SwapAllowlistExtension` Per-User Gate — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender = router` for all router-mediated swaps because `MetricOmmPool.swap` passes `msg.sender` (the immediate caller) as `sender` to extensions. If the pool admin allowlists the router — the natural configuration for a pool that accepts router traffic — every unpermissioned user can swap on a restricted pool by routing through `MetricOmmSimpleRouter`, completely defeating the per-user allowlist invariant.

## Finding Description
`MetricOmmPool.swap` calls `_beforeSwap(msg.sender, ...)` at lines 230–240, where `msg.sender` is the router address for any router-mediated swap. `ExtensionCalling._beforeSwap` encodes that value as the `sender` argument and dispatches it to the extension unchanged (lines 160–176). `SwapAllowlistExtension.beforeSwap` then evaluates `allowedSwapper[msg.sender][sender]` (line 37), where `msg.sender` is the pool and `sender` is the router — the original user address is never present in this check. Meanwhile, `exactInputSingle` stores the original `msg.sender` only in the transient callback context via `_setNextCallbackContext` (line 71) for payment purposes; it is never forwarded to the pool as the swap initiator. Any existing guard (`allowAllSwappers` short-circuit) is irrelevant when the admin has explicitly set it to `false` to enforce per-user restrictions.

## Impact Explanation
A pool admin who configures `allowAllSwappers[pool] = false` and populates `allowedSwapper[pool]` with individual trusted addresses intends to restrict swaps to those addresses only. If the router is also allowlisted (required for the pool to accept router traffic), the allowlist invariant is completely broken for router-mediated paths: any address can execute swaps on the restricted pool. This constitutes an admin-boundary break where an unprivileged caller bypasses a pool-level access control gate enforced by the pool admin.

## Likelihood Explanation
The router is a public, permissionless contract. No privileged access, special token behavior, or non-standard setup is required. Any pool using `SwapAllowlistExtension` with `allowAllSwappers = false` and the router in its allowlist is immediately and repeatably exploitable by any caller. The precondition (router allowlisted) is the natural operational configuration for such a pool.

## Recommendation
The extension must gate the original user, not the immediate caller of `pool.swap`. The preferred fix is to have `MetricOmmSimpleRouter` encode the original `msg.sender` into a standardized field within `extensionData` before calling the pool, and have `SwapAllowlistExtension.beforeSwap` decode and check that field instead of (or in addition to) the `sender` parameter. Alternatively, redesign the pool/extension interface to carry a separate `originator` field that the router populates and the pool forwards to all extensions alongside `sender`. As a minimal mitigation, document that `SwapAllowlistExtension` is incompatible with the router and add a guard that reverts if `sender` is a known router address — though this is incomplete and does not scale.

## Proof of Concept
```solidity
// Setup (pool admin)
swapAllowlistExtension.setAllowAllSwappers(pool, false);
swapAllowlistExtension.setAllowedToSwap(pool, address(router), true);  // router allowed for operational use
swapAllowlistExtension.setAllowedToSwap(pool, address(user), false);   // user NOT allowed

// Attack: unpermissioned user routes through router
// beforeSwap receives sender=router → allowedSwapper[pool][router]=true → check passes
router.exactInputSingle(ExactInputSingleParams({
    pool: pool,
    recipient: user,
    // ... other params
}));
// Swap succeeds despite user not being in the allowlist
```
A Foundry integration test can confirm this by asserting the swap succeeds when called via the router but reverts when called directly by the same user address.
