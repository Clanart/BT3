### Title
Non-EVM-native Cosmos Bank Tokens Permanently Locked on Contract Self-Destruct — (File: `x/evm/statedb/statedb.go`)

### Summary
When an EVM contract self-destructs, Ethermint's `StateDB.Commit()` only burns the EVM-native denom balance and deletes the auth account. Any non-EVM-native Cosmos bank tokens (IBC tokens, CosmWasm bridge tokens, etc.) held by the contract address are silently abandoned in the bank module with no recovery path, permanently locking those funds.

### Finding Description
In `x/evm/statedb/statedb.go`, the `Commit()` function processes self-destructed accounts as follows: [1](#0-0) 

The code explicitly acknowledges the gap: [2](#0-1) 

Only the EVM denom balance is drained before `DeleteAccount()` removes the auth-module record. The bank module, however, stores balances keyed purely by address — it does not require an auth account to exist. After `DeleteAccount()` executes, the bank module still holds any IBC or other non-EVM tokens at that address, but:

1. The auth account no longer exists, so no signer can authorize a `MsgSend` from that address.
2. No EVM code exists at the address (it was deleted), so no contract call can move the tokens.
3. There is no sweep, recovery, or governance-rescue function anywhere in the EVM module for orphaned bank balances.

The `SelfDestruct` path that feeds into this is triggered by the EVM opcode `SELFDESTRUCT` (and `SELFDESTRUCT6780` post-Cancun), both of which are reachable by any unprivileged user who deploys a contract: [3](#0-2) [4](#0-3) 

The `SelfDestruct` call marks the object and burns the EVM-denom balance at that moment, but any non-EVM tokens that arrive *after* `SelfDestruct` is called (or that were already present and are not the EVM denom) are never touched. `Commit()` only re-checks the EVM denom for any post-selfdestruct residual: [5](#0-4) 

There is no analogous loop over all bank-module denominations.

### Impact Explanation
Any Cosmos bank funds (IBC tokens, wrapped assets, etc.) held by a self-destructed contract are permanently unrecoverable. The bank module retains the balance at the orphaned address indefinitely; no account, no contract, and no governance path can spend it. This is a permanent, irreversible loss of Cosmos bank funds caused directly by Ethermint's EVM state-transition logic, matching the allowed impact: *"EVM state transition… bug that permits… valid user funds/fees to be mis-accounted."*

### Likelihood Explanation
Medium. IBC token transfers to EVM contracts are a standard use-case on Cosmos EVM chains (e.g., wrapping IBC assets via a contract). Any contract that (a) receives non-EVM-denom tokens and (b) calls `SELFDESTRUCT` — whether intentionally or via a rug/exploit — triggers the loss. The entry path requires no privilege: any user can deploy a contract, fund it with IBC tokens, and call `SELFDESTRUCT`.

### Recommendation
Before calling `DeleteAccount()` in `Commit()`, iterate over all bank-module balances held by the contract address and either:
- Burn them (send to the EVM module's burn address or use `BurnCoins`), or
- Transfer them to a designated community/recovery pool.

A minimal fix in `x/evm/statedb/statedb.go` inside the `obj.selfDestructed` branch:

```go
// Drain ALL bank balances, not just the EVM denom.
allBalances := s.keeper.GetAllBalances(cacheCtx, cosmosAddr)
if !allBalances.IsZero() {
    if err := s.keeper.BurnCoins(cacheCtx, evmtypes.ModuleName, allBalances); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct non-EVM balances")
    }
}
```

This requires exposing `GetAllBalances` and `BurnCoins` on the bank-keeper interface in `x/evm/types`.

### Proof of Concept

1. Deploy a Solidity contract on an Ethermint chain that accepts IBC token transfers and exposes a `destroy()` function calling `selfdestruct(payable(address(0)))`.
2. Transfer 100 `uatom` (IBC denom) to the contract via a Cosmos `MsgSend` or IBC transfer.
3. Call `destroy()` from any EOA.
4. Observe: the contract account is deleted from the auth module; `eth_getCode` returns `0x`; but `bank query balances <contract-address>` still shows `100uatom`.
5. Attempt any recovery: `MsgSend` from the address fails (no auth account); no contract code exists to call; no governance rescue path exists.
6. The `100uatom` is permanently locked. [6](#0-5)

func (s *StateDB) SelfDestruct(addr common.Address) uint256.Int {
	stateObject := s.getStateObject(addr)
	var prevBalance uint256.Int
	if stateObject == nil {
		return prevBalance
	}
	prevBalance = *(stateObject.Balance())
	s.journal.append(selfDestructChange{
		account:     &addr,
		prev:        stateObject.selfDestructed,
		prevbalance: new(uint256.Int).Set(&prevBalance),
	})
	stateObject.markSelfDestructed()
	// clear balance
	balance := s.GetBalance(addr)
	if balance.Sign() > 0 {
		s.SubBalance(addr, balance, tracing.BalanceDecreaseSelfdestructBurn)
	}
	return prevBalance
}
