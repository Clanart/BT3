### Title
Password Authentication Bypass in `personal_sign` and `personal_sendTransaction` JSON-RPC Methods — (File: `rpc/namespaces/ethereum/personal/api.go`)

---

### Summary

The `personal_sign` and `personal_sendTransaction` JSON-RPC methods accept a password parameter per the Ethereum JSON-RPC specification, which is supposed to decrypt the key before use. Ethermint's implementation silently discards the password entirely. Any caller who can reach the JSON-RPC endpoint with the `personal` namespace enabled can sign arbitrary data or broadcast transactions from any address in the node's keyring without knowing the password, bypassing the only authentication gate for these operations.

---

### Finding Description

**Root cause — password parameter silently discarded:**

In `rpc/namespaces/ethereum/personal/api.go`, both signing methods use the blank identifier `_` for the password argument:

```go

    return api.backend.Sign(addr, data)

**Attack path:**

1. Attacker identifies that the node exposes the `personal` JSON-RPC namespace (common in development and many production deployments).
2. Attacker calls `personal_sendTransaction` with `from` set to any address whose key is in the node's keyring, supplying an empty or wrong password.
3. The password is discarded; the backend signs and broadcasts the transaction using the node's keyring unconditionally.
4. Funds are transferred or contracts are called from the victim address without the key owner's consent.

Alternatively, the attacker calls `personal_sign` with an arbitrary payload and the victim's address to obtain a valid ECDSA signature usable for off-chain authorization schemes (EIP-712 permit, meta-transactions, etc.).

---

### Impact Explanation

**High — Ethereum transaction authorization bypass enabling unauthorized account mutation and fund transfer.**

Any caller who can reach the JSON-RPC endpoint (network-accessible or local process) can:
- Send EVM transactions from any address managed by the node's keyring without the password, directly transferring funds or mutating contract state.
- Obtain valid `personal_sign` signatures over attacker-chosen data for any managed address, enabling off-chain authorization forgery (EIP-712 permits, meta-transactions).

This is a direct authorization bypass: the password is the sole authentication mechanism for these operations, and it is unconditionally ignored.

---

### Likelihood Explanation

**Medium.** The `personal` namespace must be enabled in the node's JSON-RPC configuration and the endpoint must be reachable. This is a common configuration for validator nodes, development nodes, and any node that manages keys locally (e.g., for `eth_sendTransaction` workflows). The attacker needs no credentials, no privileged role, and no leaked secrets — only network or local access to the RPC port.

---

### Recommendation

1. **Enforce password verification before signing.** The `Sign` and `SendTransaction` methods must accept and validate the password parameter before invoking `Keyring.SignByAddress`. Use `Keyring.ExportPrivKeyArmor` + `crypto.UnarmorDecryptPrivKey` with the supplied password to verify it before proceeding, or implement a proper unlock/lock state machine.
2. **Do not silently discard the password.** Replace `_ string` with a named parameter and return an error if the password is empty or fails verification.
3. **Implement `LockAccount`/`UnlockAccount` or remove them.** The current no-op stubs create a false sense of security. Until implemented, document clearly that no password protection is enforced.
4. **Restrict the `personal` namespace** to localhost-only or authenticated transports in production configurations.

---

### Proof of Concept

```bash
# Node has address 0xABCD...1234 in its keyring, protected by password "secret"
# Attacker sends a transaction with wrong/empty password — succeeds unconditionally

curl -X POST http://<node-rpc>:8545 \
  -H "Content-Type: application/json" \
  -d '{
    "jsonrpc":"2.0",
    "method":"personal_sendTransaction",
    "params":[{
      "from":"0xABCD...1234",
      "to":"0xAttacker...addr",
      "value":"0xDE0B6B3A7640000"
    }, "wrongpassword"],
    "id":1
  }'
# Returns transaction hash — funds transferred without correct password
```

The password `"wrongpassword"` is discarded at [5](#0-4)  before reaching the backend at [6](#0-5) , which signs and broadcasts unconditionally.

func (b *Backend) Sign(address common.Address, data hexutil.Bytes) (hexutil.Bytes, error) {
	from := sdk.AccAddress(address.Bytes())

	_, err := b.clientCtx.Keyring.KeyByAddress(from)
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", address.String())
		return nil, fmt.Errorf("%s; %s", keystore.ErrNoMatch, err.Error())
	}

	// Apply EIP-191 signed-message prefix to domain-separate personal
	// signatures from transaction signatures (matching Geth's eth_sign).
	signature, _, err := b.clientCtx.Keyring.SignByAddress(from, accounts.TextHash(data), signingtypes.SignMode_SIGN_MODE_TEXTUAL)
	if err != nil {
		b.logger.Error("keyring.SignByAddress failed", "address", address.Hex())
		return nil, err
	}

	signature[crypto.RecoveryIDOffset] += 27 // Transform V from 0/1 to 27/28 according to the yellow paper
	return signature, nil
