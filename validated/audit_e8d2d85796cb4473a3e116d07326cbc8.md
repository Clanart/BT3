### Title
`initTransfer1155` Does Not Validate `multiTokens` Registration Before Accepting ERC1155 Tokens, Enabling Permanent Token Lock — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`initTransfer1155` accepts and locks ERC1155 tokens into the bridge without verifying that the corresponding `multiTokens[deterministicToken]` entry was previously set by `logMetadata1155`. If a user initiates a transfer before the token is registered, the ERC1155 tokens are irrecoverably locked: the NEAR-side `fin_transfer_callback` panics because the token is absent from `token_decimals`, and no on-chain recovery path exists.

---

### Finding Description

The protocol's ERC1155 flow requires two steps:

1. **`logMetadata1155(tokenAddress, tokenId)`** — registers the `(tokenAddress, tokenId)` pair in `multiTokens[deterministicToken]` and emits a `LogMetadata` event that the NEAR relayer uses to register the token (including its decimals) in `token_decimals`.
2. **`initTransfer1155(tokenAddress, tokenId, amount, ...)`** — locks the ERC1155 tokens and emits `InitTransfer` with `deterministicToken` as the token identifier.

`initTransfer1155` performs no check that `multiTokens[deterministicToken].tokenAddress != address(0)` before accepting the transfer:

```solidity
// OmniBridge.sol L453-L490
address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);

IERC1155(tokenAddress).safeTransferFrom(   // tokens locked here
    msg.sender, address(this), tokenId, amount, ""
);
// ← no require(multiTokens[deterministicToken].tokenAddress != address(0))

emit BridgeTypes.InitTransfer(
    msg.sender, deterministicToken, currentOriginNonce, ...
);
```

This is directly analogous to the reported `assignTeam` bug: just as `createYeet` credited deposits to a team without validating the user's existing assignment, `initTransfer1155` locks tokens and emits a cross-chain event without validating that the token's cross-chain identity has been established.

On the NEAR side, `fin_transfer_callback` attempts to look up the token's decimal configuration:

```rust
// near/omni-bridge/src/lib.rs L715-L718
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
```

Because `logMetadata1155` was never called, `deterministicToken` is absent from `token_decimals`. The NEAR transaction panics and is rolled back. The EVM-side lock is permanent — there is no `emergencyWithdraw` or admin-rescue path in the base contract.

---

### Impact Explanation

ERC1155 tokens transferred via `initTransfer1155` without a prior `logMetadata1155` call are **permanently and irrecoverably locked** in the bridge contract. `finTransfer` cannot release them because:

- `multiTokens[deterministicToken].tokenAddress == address(0)` → ERC1155 release path skipped.
- `customMinters[deterministicToken] == address(0)` → custom-minter path skipped.
- `isBridgeToken[deterministicToken] == false` → bridge-token mint path skipped.
- `IERC20(deterministicToken).safeTransfer(...)` → reverts because `deterministicToken` is not an ERC20 contract.

No valid MPC-signed `finTransfer` payload can ever release the locked ERC1155 tokens. This matches the allowed impact: **permanent freezing / irrecoverable lock of user funds in bridge flows**.

---

### Likelihood Explanation

`initTransfer1155` is a public, permissionless function. Any token holder can call it directly. The prerequisite call to `logMetadata1155` is not documented in the function signature, not enforced by any modifier, and not obvious from the ABI alone. A user interacting with the bridge for the first time — or integrating via a third-party frontend that omits the metadata step — will permanently lose their tokens. The entry path requires no privileges, no colluding parties, and no special conditions beyond holding an ERC1155 token.

---

### Recommendation

Add a guard at the top of `initTransfer1155` that requires the `multiTokens` entry to be populated before accepting the transfer:

```solidity
function initTransfer1155(
    address tokenAddress,
    uint256 tokenId,
    uint128 amount,
    uint128 fee,
    uint128 nativeFee,
    string calldata recipient,
    string calldata message
) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);

    // Analog of the assignTeam fix: validate state consistency before processing
    MultiTokenInfo storage multiToken = multiTokens[deterministicToken];
    require(
        multiToken.tokenAddress == tokenAddress && multiToken.tokenId == tokenId,
        "ERR_TOKEN_NOT_REGISTERED"
    );

    currentOriginNonce += 1;
    if (fee >= amount) revert InvalidFee();
    // ... rest of function
}
```

### Proof of Concept

1. Alice holds ERC1155 token `(tokenAddress=0xABC, tokenId=1)`.
2. Alice calls `initTransfer1155(0xABC, 1, 100, 0, 0, "alice.near", "")` **without** first calling `logMetadata1155(0xABC, 1)`.
3. `deterministicToken = address(bytes20(keccak256(abi.encodePacked(0xABC, 1))))`.
4. `IERC1155(0xABC).safeTransferFrom(alice, bridge, 1, 100, "")` succeeds — tokens are locked.
5. `InitTransfer` event is emitted with `deterministicToken`.
6. NEAR relayer submits the EVM proof to `fin_transfer` → `fin_transfer_callback`.
7. `self.token_decimals.get(&OmniAddress::Eth(deterministicToken))` returns `None` → `near_expect(BridgeError::TokenDecimalsNotFound)` panics.
