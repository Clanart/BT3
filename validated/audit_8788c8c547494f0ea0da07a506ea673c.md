Audit Report

## Title
Stranded native ETH on the router is silently consumed by any subsequent WETH swap from an unrelated caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` funds WETH payments by reading `address(this).balance` without any per-caller attribution. Any native ETH left on the router from a prior transaction (e.g., excess `msg.value` not refunded via `refundETH()`) is silently consumed by the next caller who initiates a WETH swap, causing direct, irrecoverable loss of the original depositor's ETH.

## Finding Description
`pay()` at lines 74–77 reads the router's entire native balance and uses it to fund WETH payments:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
``` [1](#0-0) 

The design assumes all native ETH on the router belongs to the current caller. This assumption is broken across transaction boundaries. All swap entry-points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) are `payable`, so a caller can send `msg.value > amountIn`. The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH accumulation via `msg.value` in payable calls: [2](#0-1) 

`refundETH()` is a separate, optional call that users must explicitly include (e.g., in a `multicall`). If omitted — by user error, frontend bug, or a failed multicall leg — the surplus ETH remains on the router indefinitely with no per-caller tracking: [3](#0-2) 

The callback path for single-hop exact-input swaps calls `_justPayCallback`, which calls `pay(_getTokenToPay(), _getPayer(), msg.sender, amount)`. When `tokenIn == WETH`, `pay()` will consume stranded ETH from any prior transaction before pulling from the actual payer: [4](#0-3) 

**Exploit flow:**
1. Alice calls `exactInputSingle{value: 1 ether}(amountIn: 0.5 ether, tokenIn: WETH, ...)`. `pay()` deposits 0.5 ETH as WETH for the pool; 0.5 ETH remains stranded on the router.
2. Bob calls `exactInputSingle{value: 0}(amountIn: 0.5 ether, tokenIn: WETH, ...)` with no WETH approval. `pay()` sees `nativeBalance = 0.5 ether >= value = 0.5 ether`, deposits Alice's ETH as WETH, and transfers it to the pool. Bob receives the swap output; Alice's 0.5 ETH is gone.

No privileged access is required. Bob needs only to observe the stranded ETH (trivially visible on-chain) and submit a WETH swap for that amount.

## Impact Explanation
Direct, irrecoverable loss of user ETH. The stranded ETH is transferred to the pool as WETH on behalf of the attacker's swap. The original depositor has no recovery mechanism once the subsequent swap executes. Loss is bounded by the stranded amount and the attacker's chosen `amountIn`. This meets the High severity threshold for direct loss of user principal.

## Likelihood Explanation
Medium. Sending a round-number `msg.value` as a buffer when swapping WETH is a common pattern. The correct mitigation — including `refundETH()` in the same `multicall` — is not enforced by the router. Omitting it via user error, a frontend bug, or a failed multicall leg leaves ETH stranded. A MEV bot monitoring the mempool or router balance can drain it in the very next block with a zero-cost WETH swap.

## Recommendation
Track the ETH contributed by the current caller in transient storage at the start of each payable entry-point (e.g., store `msg.value` in a transient slot keyed to the call). Cap `pay()`'s native-ETH consumption to that tracked amount rather than `address(this).balance`. Alternatively, require `msg.value` to equal exactly the WETH `amountIn` needed and revert otherwise, or remove the native-ETH shortcut entirely and require callers to pre-wrap ETH before calling the router.

## Proof of Concept
```solidity
// Step 1 — Alice strands ETH (realistic user error, omits refundETH())
router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
    pool: pool, tokenIn: WETH, tokenOut: token1,
    zeroForOne: true, amountIn: 0.5 ether,
    amountOutMinimum: 0, recipient: alice,
    deadline: block.timestamp, priceLimitX64: 0, extensionData: ""
}));
// 0.5 ETH remains on the router, unattributed.

// Step 2 — Bob steals Alice's ETH (no ETH sent, no WETH approval)
router.exactInputSingle{value: 0}(ExactInputSingleParams({
    pool: pool, tokenIn: WETH, tokenOut: token1,
    zeroForOne: true, amountIn: 0.5 ether,
    amountOutMinimum: 0, recipient: bob,
    deadline: block.timestamp, priceLimitX64: 0, extensionData: ""
}));
// pay() sees nativeBalance=0.5e >= value=0.5e → deposits Alice's ETH as WETH.
// Bob receives token1 output; Alice's 0.5 ETH is irrecoverably lost.
```

Root cause: `address(this).balance` at line 74 of `PeripheryPayments.sol` is read without any per-caller attribution, making all router ETH available to fund any WETH swap regardless of who deposited it. [5](#0-4)
