Audit Report

## Title
Missing Deadline Check in `addLiquidityExactShares` Allows Stale Pending Transactions to Execute at Unfavorable Pool Prices — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

Both overloads of `addLiquidityExactShares` in `MetricOmmPoolLiquidityAdder` accept no `deadline` parameter and perform no `block.timestamp` comparison, allowing a pending mempool transaction to be mined arbitrarily late. When the pool price has moved significantly by execution time, the user's shares are deposited into bins that are entirely off-price, resulting in a single-sided deposit with a composition radically different from what the user intended and immediate impermanent loss relative to the current market price.

## Finding Description

`MetricOmmSimpleRouter` correctly guards all four swap entry points with `_checkDeadline(params.deadline)`: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

Neither overload of `addLiquidityExactShares` contains a `deadline` parameter or any `block.timestamp` comparison: [5](#0-4) [6](#0-5) 

The only guards present are `_validateOwner` (checks `owner != address(0)`) and `_validateDeltas` (checks array length consistency). The `maxAmountToken0`/`maxAmountToken1` caps enforced in the callback bound the *quantity* of tokens pulled but say nothing about *when* the transaction executes or *which bins* are active at that time: [7](#0-6) 

By contrast, `addLiquidityWeighted` has a partial mitigation: `_validateBinAndBinPosition` reverts if the pool cursor has moved outside user-supplied bounds: [8](#0-7) 

`addLiquidityExactShares` has no such cursor-bounds check, making it the more exposed path. A user who submits a transaction targeting bins near the current price will have those same bins filled regardless of how far the price has moved by the time the transaction is mined.

## Impact Explanation

In a bin-based AMM, each bin is active only when the pool price is inside that bin's range. If the price has moved far from the user's target bins by execution time, all target bins may be above or below the current price, causing the pool to request only one token. The user intended a balanced deposit but receives a single-sided LP position. The position's mark-to-market value at current prices is lower than the deposited token value because the deposited token is now the "wrong" side of the market — constituting a direct loss of user principal relative to simply holding the tokens or executing the deposit at the intended time. The `maxAmountToken0`/`maxAmountToken1` caps do not prevent this: they were sized to accommodate the expected balanced composition, so a fully single-sided deposit can still fall within the caps while being entirely unintended.

## Likelihood Explanation

The scenario requires only a low-fee transaction sitting in the mempool during a period of price volatility — a common occurrence on congested networks. No privileged access, malicious setup, or non-standard token behavior is required. Any user of `addLiquidityExactShares` is exposed.

## Recommendation

Add a `deadline` parameter to both `addLiquidityExactShares` overloads and validate it before any state-changing call:

```solidity
function addLiquidityExactShares(
    address pool,
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 maxAmountToken0,
    uint256 maxAmountToken1,
+   uint256 deadline,
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
+   if (block.timestamp > deadline) revert DeadlineExpired();
    _validateOwner(owner);
    ...
}
```

Apply the same change to both `addLiquidityWeighted` overloads for defense-in-depth, even though they already have cursor-bounds validation via `_validateBinAndBinPosition`.

## Proof of Concept

1. Alice targets bins `[-1, 0, 1]` near the current ETH/USDC price of $2,000. She calls `addLiquidityExactShares` with `maxAmountToken0 = 1,000 USDC` and `maxAmountToken1 = 0.5 ETH`, expecting a balanced deposit.
2. Alice submits with a low gas fee. The transaction sits in the mempool for two days.
3. ETH price drops to $1,200. The pool cursor is now far below bin `-1`.
4. The transaction is mined. All three target bins are now above the current price.
5. The pool callback requests `1,000 USDC` and `0 ETH` — within Alice's caps.
6. Alice receives an LP position worth ~$1,000 at current prices, but her position is 100% USDC concentrated in bins priced at $1,800–$2,200. If ETH never recovers to those bins, Alice's liquidity earns no fees and she holds a single-sided position she never intended.
7. Had Alice's transaction been rejected after a deadline, she could have resubmitted at the new price with appropriate bin targets.

A Foundry fork test can reproduce this by: (a) deploying the liquidity adder, (b) submitting `addLiquidityExactShares` targeting bins near the current price, (c) warping `block.timestamp` forward and moving the pool price via swaps, then (d) executing the original call and asserting the returned `amount1Added == 0` while `amount0Added == maxAmountToken0`.
