### Title
Missing Nonce in `meta_tx_v0` Hash Enables Unbounded Signature Replay — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo` and `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall constructs a version-0 transaction hash that omits any nonce or replay-prevention field. Because the hash is a pure deterministic function of `(contract_address, selector, calldata, chain_id)`, a signature that was valid once remains valid forever for the same parameters. No nonce is checked or incremented for meta transactions. An unprivileged attacker who observes a single valid `meta_tx_v0` execution can replay it an unlimited number of times across separate outer transactions, draining funds from any account that authorizes value-transferring calls through this mechanism.

---

### Finding Description

`execute_meta_tx_v0` in `syscall_impls.cairo` is a syscall that lets a contract invoke another contract under a synthetic version-0 transaction context, supplying a custom signature. The OS computes the hash that the target account will verify against:

```cairo

    contract_address=contract_address,
    entry_point_selector=selector,
    calldata=calldata_start,
    calldata_size=calldata_size,
    chain_id=old_tx_info.chain_id,
);
``` [1](#0-0) 

Inside `compute_meta_tx_v0_hash`, the hash is built with `additional_data_size=0` — no nonce, no block number, no counter of any kind:

The result: for any fixed `(contract_address, selector, calldata, chain_id)` tuple, the meta-tx hash is identical across every block and every outer transaction. A signature that satisfies the target account's `__execute__` for this hash is permanently valid and can be replayed without restriction.

---

### Impact Explanation

**Critical — Direct loss of funds.**

Any account contract that authorizes a value transfer (e.g., ERC-20 `transfer`, ETH withdrawal) through a `meta_tx_v0` call is vulnerable. Once a legitimate execution is observed on-chain, an attacker can:

1. Extract the `(contract_address, calldata, signature)` from the original outer transaction.
2. Submit new outer transactions (with fresh nonces on the outer account) to any relayer or multicall contract that accepts `meta_tx_v0` requests.
3. Each new outer transaction replays the identical meta-tx hash, which the target account's `__execute__` accepts because the signature is still valid.
4. Funds are drained with each replay.

---

### Likelihood Explanation

**Medium.**

- The `meta_tx_v0` syscall is a new, explicitly supported protocol feature (it has its own gas constants `META_TX_V0_GAS_COST`, `META_TX_V0_CALLDATA_FACTOR_GAS_COST`), so it will be used in production contracts.
- All parameters needed for replay (`contract_address`, `calldata`, `signature`) are visible on-chain after the first execution.
- The attacker needs no privileged access — only the ability to submit outer transactions to a contract that invokes `meta_tx_v0`.
- The only friction is finding a relayer/multicall contract that accepts arbitrary `meta_tx_v0` parameters from external callers, which is the intended use-case of the syscall.

---

### Recommendation

Include a monotonically incrementing nonce in the `meta_tx_v0` hash. The nonce should be sourced from the target account's on-chain state and incremented by the OS after each successful meta-tx execution, mirroring the pattern used for L1 handler transactions:

```cairo
// In compute_meta_tx_v0_hash: add nonce as additional_data
additional_data_size=1,
additional_data=&nonce,
```

The OS must also read the current nonce from `contract_state_changes` for the target `contract_address`, assert it matches the nonce in the request, and write back `nonce + 1` — analogous to `check_and_increment_nonce` for regular account transactions.

---

### Proof of Concept

1. **Block N**: Legitimate user calls outer contract `R` (a relayer). `R` invokes `meta_tx_v0` targeting account `A` with `calldata = [transfer(victim_token, attacker, 1000)]` and a valid ECDSA signature `S` from `A`'s key. Funds transfer succeeds.

2. **Block N+1**: Attacker submits a new outer transaction to `R` with the same `meta_tx_v0` parameters: same `contract_address=A`, same `calldata`, same `signature=S`.

3. The OS calls `compute_meta_tx_v0_hash(A, __execute__, calldata, chain_id)` → produces the **identical hash** as in Block N (no nonce in the hash).

4. Account `A`'s `__execute__` verifies `S` against the hash → **passes**, because `S` was generated for exactly this hash.

5. The transfer executes again. Repeat indefinitely until `A`'s balance is zero. [7](#0-6) [8](#0-7)

    local calldata_size = request.calldata_end - calldata_start;

    let specific_base_gas_cost = (
        META_TX_V0_GAS_COST + META_TX_V0_CALLDATA_FACTOR_GAS_COST * calldata_size
    );
    let (success, remaining_gas) = reduce_syscall_base_gas(
        specific_base_gas_cost=specific_base_gas_cost, request_struct_size=MetaTxV0Request.SIZE
    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    local contract_address = request.contract_address;
    local selector = request.selector;
    local caller_execution_info: ExecutionInfo* = caller_execution_context.execution_info;
    local old_tx_info: TxInfo* = caller_execution_info.tx_info;

    if (selector != EXECUTE_ENTRY_POINT_SELECTOR) {
        write_failure_response(remaining_gas=remaining_gas, failure_felt=ERROR_INVALID_ARGUMENT);
        return ();

    assert_nn_le(request.signature_end - request.signature_start, SIERRA_ARRAY_LEN_BOUND - 1);

    with pedersen_ptr {

            entry_point_selector=selector,
            calldata=calldata_start,
            calldata_size=calldata_size,
            chain_id=old_tx_info.chain_id,
        );

    // Prepare execution context.

        account_contract_address=contract_address,
        max_fee=0,
        signature_start=request.signature_start,
        signature_end=request.signature_end,
        transaction_hash=meta_tx_hash,
        chain_id=old_tx_info.chain_id,
        nonce=0,

        resource_bounds_end=cast(0, ResourceBounds*),

        account_deployment_data_end=cast(0, felt*),
        proof_facts_start=cast(0, felt*),
        proof_facts_end=cast(0, felt*),

        class_hash=state_entry.class_hash,
        calldata_size=calldata_size,

            tx_info=new_tx_info,

// Computes the hash of a v0 meta transaction. See the `meta_tx_v0` syscall.
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,
        additional_data=cast(0, felt*),
    );
    return tx_hash;
}
