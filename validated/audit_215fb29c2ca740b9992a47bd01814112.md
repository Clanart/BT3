### Title
Meta-TX V0 Signature Replay Due to Missing Nonce in Hash — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall constructs a version-0 transaction context with a hardcoded `nonce=0` and computes its hash via `compute_meta_tx_v0_hash`, which passes `additional_data_size=0` — omitting any nonce from the hash. Because the OS also explicitly skips nonce checking for version-0 transactions, a valid meta-tx v0 signature is permanently replayable by any unprivileged caller, enabling repeated execution of the signed action (e.g., fund transfers) until the target account is drained.

---

### Finding Description

`execute_meta_tx_v0` in `syscall_impls.cairo` builds a synthetic `TxInfo` with `nonce=0` hardcoded: [1](#0-0) 

The transaction hash for this synthetic context is computed by `compute_meta_tx_v0_hash`, which calls `deprecated_get_transaction_hash` with `additional_data_size=0`: [2](#0-1) 

The resulting hash is therefore a pure function of `(INVOKE_HASH_PREFIX, 0, contract_address, selector, calldata, 0, chain_id)` — no nonce, no block number, no expiry. For regular account transactions, the nonce is passed as `additional_data` to `deprecated_get_transaction_hash`; here it is absent entirely.

The OS-level nonce guard `check_and_increment_nonce` explicitly skips version-0 transactions and is never called inside `execute_meta_tx_v0` anyway: [3](#0-2) 

The syscall is reachable by any contract via the `META_TX_V0_SELECTOR` dispatch path: [4](#0-3) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

A meta-tx v0 is designed to call the `__execute__` entry point of an account contract with a caller-supplied signature. If the signed action transfers tokens or ETH, an attacker who observes the original signature on-chain can replay it an unlimited number of times by deploying a contract that issues the same `meta_tx_v0` syscall with the same `(contract_address, selector, calldata, signature)`. Because the hash is identical on every replay, the account contract's signature verification passes every time, and the transfer executes repeatedly until the account is drained.

---

### Likelihood Explanation

All meta-tx v0 signatures are published on-chain as part of the transaction data. Any unprivileged observer can extract the signature and replay it. No privileged access, leaked key, or social engineering is required. The only precondition is that a user has ever signed a meta-tx v0 that moves value.

---

### Recommendation

Include a per-account nonce in the meta-tx v0 hash by passing it as `additional_data` in `compute_meta_tx_v0_hash`, and increment the account's nonce in `execute_meta_tx_v0` after successful execution. Concretely:

1. In `compute_meta_tx_v0_hash` (`transaction_hash.cairo`), add a `nonce: felt` parameter and pass it as `additional_data` (size 1) to `deprecated_get_transaction_hash`.
2. In `execute_meta_tx_v0` (`syscall_impls.cairo`), read the current nonce from `contract_state_changes` for `contract_address`, pass it to the hash function, and write back `nonce + 1` to `contract_state_changes` before executing the call.

---

### Proof of Concept

1. Alice signs a meta-tx v0 authorising a transfer of 10 ETH from her account (`contract_address=ALICE`, `selector=__execute__`, `calldata=[transfer, BOB, 10_ETH]`). The OS computes `H = hash(PREFIX, 0, ALICE, __execute__, calldata, 0, chain_id)`. Alice's account verifies `sig` over `H` and executes the transfer.

2. Attacker deploys `ReplayContract` with a single function that issues the `meta_tx_v0` syscall with the same `(ALICE, __execute__, calldata, sig)`.

3. Attacker submits a transaction calling `ReplayContract`. The OS recomputes the identical `H` (no nonce), Alice's account verifies the same `sig` over the same `H`, and executes another 10 ETH transfer to BOB.

4. Steps 2–3 repeat until Alice's account balance is zero. No new signature from Alice is ever required.
