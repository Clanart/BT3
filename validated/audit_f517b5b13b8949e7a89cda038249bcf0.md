### Title
Wrong Base in `within_margin` L1 Gas Price Validation Allows Proposer to Exceed Intended Price Margin — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The `within_margin` function used to validate a proposed L1 gas price computes the allowed deviation relative to the **proposed** price (`number1`) rather than the **reference** price (`number2`). With a configured `l1_gas_price_margin_percent` of 10, this allows a proposer to embed an L1 gas price up to ~11.1% above the locally-computed reference and still pass validator checks — instead of the intended 10%. Because the L1 gas price flows directly into every transaction's fee calculation for that block, the excess is silently charged to users and credited to the sequencer.

---

### Finding Description

`within_margin` is defined as:

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;   // ← base is number1 (proposed)
    number1.0.abs_diff(number2.0) <= margin
}
``` [1](#0-0) 

It is called during proposal validation as:

```rust
within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
``` [2](#0-1) 

Here `number1 = proposed` and `number2 = reference`. The margin is therefore `proposed × 10 / 100`, not `reference × 10 / 100`.

**Arithmetic consequence.** When `proposed > reference`:

```
proposed - reference  ≤  proposed × 0.10
proposed × 0.90       ≤  reference
proposed              ≤  reference / 0.90  ≈  reference × 1.111
```

A proposer can set `l1_gas_price_fri` (and `l1_data_gas_price_fri`) up to **11.1 %** above the reference and every validator will accept the block. The intended ceiling is 10 %.

This is structurally identical to M-09: the denominator used is `proposed` (analogous to `BASE − feePct = 95`) instead of `reference` (analogous to `BASE = 100`), causing the effective tolerance to be larger than specified.

The validated gas price is then embedded in `BlockInfo` via `convert_to_sn_api_block_info`:

```rust
strk_gas_prices: GasPriceVector {
    l1_gas_price: l1_gas_price_fri,
    ...
}
``` [3](#0-2) 

and used by `get_fee_by_gas_vector` for every transaction fee in the block:

```rust
pub fn get_fee_by_gas_vector(...) -> Fee {
    gas_vector.cost(block_info.gas_prices.gas_price_vector(fee_type), tip)
}
``` [4](#0-3) 

The `l1_gas_price_margin_percent` is set to `10` in all versioned-constants files: [5](#0-4) 

---

### Impact Explanation

Every transaction in the accepted block has its L1 gas fee computed against the inflated price. The ~1.1 % excess above the intended 10 % ceiling is charged to users and transferred to the sequencer's balance via `add_fee_to_sequencer_balance`. [6](#0-5) 

This is a **Critical** impact: *Incorrect fee / L1 gas price effect with economic impact* — the block's authoritative gas price is wrong, every receipt's fee field is wrong, and the sequencer balance state diff is wrong.

---

### Likelihood Explanation

The proposer role rotates among all validators. Any validator node, on its proposer turn, can set `l1_gas_price_fri = reference × 1.111` in the `ProposalInit` it broadcasts. All honest validators will accept it because `within_margin` passes. No special privilege beyond normal proposer rotation is required. The `l1_gas_price_margin_percent = 10` configuration is the production default.

---

### Recommendation

Compute the margin relative to the **reference** price (`number2`), not the proposed price:

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number2.0 * margin_percent) / 100;  // use reference as base
    number1.0.abs_diff(number2.0) <= margin
}
```

This mirrors the M-09 fix: replace the wrong denominator (`proposed`) with the correct one (`reference`), so the effective tolerance is exactly `margin_percent %` of the reference price in both directions.

---

### Proof of Concept

Let `reference = 1_000_000` fri, `margin_percent = 10`.

**With current code** (`base = proposed`):
- Set `proposed = 1_111_112` (≈ 11.11 % above reference).
- `margin = 1_111_112 × 10 / 100 = 111_111`
- `|1_111_112 − 1_000_000| = 111_112 > 111_111` → **fails** (just barely).
- Set `proposed = 1_111_111`:
  - `margin = 111_111`
  - `|1_111_111 − 1_000_000| = 111_111 ≤ 111_111` → **passes**.
  - Actual deviation from reference: **11.1111 %**, not 10 %.

**With corrected code** (`base = reference`):
- `margin = 1_000_000 × 10 / 100 = 100_000`
- `proposed = 1_100_001`: `|1_100_001 − 1_000_000| = 100_001 > 100_000` → **rejected**.
- `proposed = 1_100_000`: `100_000 ≤ 100_000` → accepted (exactly 10 %).

A malicious proposer using the current code embeds `l1_gas_price_fri = 1_111_111` in `ProposalInit`. All validators accept it. Every transaction in the block is charged fees at 11.1 % above the reference L1 gas price. The sequencer collects the excess.

fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    // For small numbers (e.g., less than 10 wei, if margin is 10%), even an off-by-one
    // error might be bigger than the margin, even if it is just a rounding error.
    // We make an exception for such mismatch, and don't bother checking percentages
    // if the difference in price is only one wei.
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}

pub fn add_fee_to_sequencer_balance(
    fee_token_address: ContractAddress,
    state: &mut impl UpdatableState,
    actual_fee: Fee,
    block_context: &BlockContext,
    sequencer_balance: (Felt, Felt),
    sender_address: ContractAddress,
    state_diff: &mut StateMaps,
) {
    assert_ne!(
        sender_address, block_context.block_info.sequencer_address,
        "The sender cannot be the sequencer."
    );
    let (low, high) = sequencer_balance;
    let sequencer_balance_low_as_u128 =
        low.to_u128().expect("sequencer balance low should be u128");
    let sequencer_balance_high_as_u128 =
        high.to_u128().expect("sequencer balance high should be u128");
    let (new_value_low, overflow_low) = sequencer_balance_low_as_u128.overflowing_add(actual_fee.0);
    let (new_value_high, overflow_high) =
        sequencer_balance_high_as_u128.overflowing_add(overflow_low.into());
    assert!(
        !overflow_high,
        "The sequencer balance overflowed when adding the fee. This should not happen."
    );
    let (sequencer_balance_key_low, sequencer_balance_key_high) =
        get_sequencer_balance_keys(block_context);
    let writes = StateMaps {
        storage: HashMap::from([
            ((fee_token_address, sequencer_balance_key_low), Felt::from(new_value_low)),
            ((fee_token_address, sequencer_balance_key_high), Felt::from(new_value_high)),
        ]),
        ..StateMaps::default()
    };

    // Modify state_diff to accurately reflect the post tx-execution state, after fee transfer to
    // the sequencer. We assume that a non-sequencer sender cannot reduce the sequencer's
    // balance—only increases are possible.

    if sequencer_balance_high_as_u128 != new_value_high {
        // Update the high balance only if it has changed.
        state_diff
            .storage
            .insert((fee_token_address, sequencer_balance_key_high), Felt::from(new_value_high));
    }

    if sequencer_balance_low_as_u128 != new_value_low {
        // Update the low balance only if it has changed.
        state_diff
            .storage
            .insert((fee_token_address, sequencer_balance_key_low), Felt::from(new_value_low));
    }
    state.apply_writes(&writes, &ContractClassMapping::default());
