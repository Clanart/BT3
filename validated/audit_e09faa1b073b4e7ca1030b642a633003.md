Audit Report

## Title
Unprivileged `refundETH()` drains any ETH stranded on the router by excess `msg.value` in payable swap calls — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` contains no access control and sends the router's entire ETH balance to `msg.sender`. Because all four swap entry points are `payable` and `pay()` wraps only exactly `value` ETH (leaving any surplus on the contract), a user who sends excess `msg.value` in a direct swap call leaves residual ETH on the router after the transaction. Any unprivileged caller can immediately drain it via `refundETH()`.

## Finding Description
`refundETH()` unconditionally transfers the full ETH balance to any caller: [1](#0-0) 

The `pay()` helper, when `token == WETH` and `nativeBalance >= value`, wraps exactly `value` ETH and leaves the remainder on the contract: [2](#0-1) 

All four swap entry points are `external payable`, accepting arbitrary `msg.value`: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent `msg.value` from entering via payable function calls: [7](#0-6) 

**Exploit path:**
1. Victim calls `exactInputSingle{value: 2 ether}(amountIn=1 ether, tokenIn=WETH)` directly (not via multicall).
2. The swap callback invokes `pay()` with `value=1 ether`; since `nativeBalance (2 ETH) >= value (1 ETH)`, exactly 1 ETH is wrapped and forwarded to the pool. The remaining 1 ETH stays on the router.
3. The transaction completes; 1 ETH is now stranded on the router.
4. Attacker calls `refundETH()` in a subsequent transaction and receives the full 1 ETH.

The `multicall{value}([swap, refundETH])` pattern is atomic and safe, but it is not enforced on-chain. No guard prevents a user from calling a swap function directly with excess ETH.

## Impact Explanation
Direct theft of user ETH principal. Any ETH stranded on the router between transactions is fully claimable by an unprivileged attacker. The victim loses the entire excess `msg.value` they sent. This is a direct loss of user principal — **High** severity.

## Likelihood Explanation
**Medium.** The attack requires the victim to call a swap function directly with excess `msg.value` rather than using the `multicall` + `refundETH` atomic pattern. This is a realistic user error: the interface is `payable`, and no on-chain enforcement prevents the non-atomic call. A MEV bot can monitor the mempool and backrun any such transaction to steal the stranded ETH.

## Recommendation
Add a caller-binding check to `refundETH()`. The simplest approach is to track the originating caller in transient storage at the start of each payable entry point and enforce that only that address can call `refundETH()` within the same context. Alternatively, restrict `refundETH()` to only be callable via `multicall` (e.g., by checking `msg.sender == address(this)`), ensuring atomicity is enforced on-chain.

## Proof of Concept
```solidity
function test_attacker_steals_excess_eth() public {
    address victim   = makeAddr("victim");
    address attacker = makeAddr("attacker");
    vm.deal(victim, 2 ether);

    // Victim calls exactInputSingle directly with 2 ether but amountIn = 1 ether
    vm.prank(victim);
    router.exactInputSingle{value: 2 ether}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: 1 ether,       // only 1 ether used; 1 ether excess stays on router
            amountOutMinimum: 0,
            recipient: victim,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );

    assertEq(address(router).balance, 1 ether); // 1 ether stranded

    // Attacker steals it
    vm.prank(attacker);
    router.refundETH();

    assertEq(attacker.balance, 1 ether);
    assertEq(address(router).balance, 0);
}
```
