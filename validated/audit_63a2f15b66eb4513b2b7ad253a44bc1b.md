### Title
Off-by-one in `insert_forwarded_chunk` allows `part_ord == num_total_parts` to be cached, causing an index-out-of-bounds panic during chunk reconstruction — (`File: chain/chunks/src/shards_manager_actor.rs`)

---

### Summary

`insert_forwarded_chunk` guards against out-of-range `part_ord` values with `part_ord > num_total_parts` instead of `part_ord >= num_total_parts`. This is the exact same class of missing upper-bound check as the external report: a value that must be strictly less than a limit is only checked for being strictly greater than it, leaving the boundary value unguarded. A `part_ord` equal to `num_total_parts` passes the filter, is stored in `chunk_forwards_cache`, later merged into `encoded_chunks` without re-validation, and finally causes an index-out-of-bounds panic when the node attempts chunk reconstruction.

---

### Finding Description

In `insert_forwarded_chunk` (lines 1347 and 1360), the guard is:

```rust
if part_ord > num_total_parts {
    // warn and drop
} else {
    Some((part_ord, part))   // cached
}
```

Valid part ordinals are `0..num_total_parts-1`. A part with `part_ord == num_total_parts` is **not** a valid ordinal, but it satisfies `!(part_ord > num_total_parts)` and is therefore cached. [1](#0-0) [2](#0-1) 

`insert_forwarded_chunk` is called from `process_partial_encoded_chunk_forward` in two cases where `validate_partial_encoded_chunk_forward` (which correctly uses `<`) has not yet run to completion: when the chunk header is unknown (`Error::UnknownChunk`) and when the previous block is not yet processed (`DBNotFoundErr`). Both are normal, frequent conditions during block propagation. [3](#0-2) 

When the header later arrives, `insert_header_if_not_exists_and_process_cached_chunk_forwards` pops the cached parts and merges them directly into `encoded_chunks` with an explicit comment that no further validation is needed — because the code assumes the earlier `validate_partial_encoded_chunk_forward` already ran, which it did not for these cached parts. [4](#0-3) 

The invalid part (keyed by `num_total_parts`) now lives in `entry.parts`. When enough legitimate parts arrive to satisfy `can_reconstruct`, the reconstruction loop executes:

```rust
for (part_ord, part_entry) in &entry.parts {
    encoded_chunk.content_mut().parts[*part_ord as usize] =
        Some(part_entry.part.clone());
}
```

`encoded_chunk.content_mut().parts` is a `Vec` of length `num_total_parts` (allocated by `EncodedShardChunk::from_header`). Indexing it at `*part_ord as usize == num_total_parts` is an out-of-bounds access → **panic**. [5](#0-4) 

For contrast, `validate_part` — used on the non-cached path — correctly uses `<`:

```rust
if (part.part_ord as usize) < num_total_parts { /* valid */ } else { Err(Error::InvalidChunkPartId) }
``` [6](#0-5) 

---

### Impact Explanation

A panic in the `ShardsManagerActor` crashes the node process. Any peer on the network can send a `PartialEncodedChunkForwardMsg` — it is an unauthenticated network message at the point of caching. The attacker does not need to be a validator or chunk producer. Crashing a validator node causes it to miss block/chunk production slots, degrading chain liveness. Crashing enough validators simultaneously can halt the chain.

---

### Likelihood Explanation

The precondition — receiving a forward before the chunk header or before the previous block is processed — is the **normal, expected case** during every epoch. The code explicitly handles it as a non-error path. An attacker only needs to send one crafted `PartialEncodedChunkForwardMsg` per target chunk, timed to arrive before the header is known (trivially achievable by sending it slightly ahead of the block). The invalid part then sits silently in the cache until enough legitimate parts arrive to trigger reconstruction, at which point the panic fires.

---

### Recommendation

Change both occurrences of `>` to `>=` in `insert_forwarded_chunk`:

```rust
// Before (incorrect — allows part_ord == num_total_parts)
if part_ord > num_total_parts { ... }

// After (correct — rejects part_ord == num_total_parts)
if part_ord >= num_total_parts { ... }
``` [7](#0-6) [8](#0-7) 

Additionally, add a bounds check before the indexing in the reconstruction loop as a defense-in-depth measure:

```rust
for (part_ord, part_entry) in &entry.parts {
    let idx = *part_ord as usize;
    if idx < encoded_chunk.content().parts.len() {
        encoded_chunk.content_mut().parts[idx] = Some(part_entry.part.clone());
    }
}
```

---

### Proof of Concept

1. Attacker connects as a peer to a validator node.
2. Attacker observes a new chunk hash `H` being gossiped (or predicts it from the block producer schedule) before the node has received the corresponding chunk header.
3. Attacker sends a `PartialEncodedChunkForwardMsg` with `chunk_hash = H`, `parts = [PartialEncodedChunkPart { part_ord: num_total_parts, part: <any bytes>, merkle_proof: <any> }]`.
4. Node calls `process_partial_encoded_chunk_forward` → `validate_partial_encoded_chunk_forward` fails with `UnknownChunk` → `insert_forwarded_chunk` is called.
5. The guard `part_ord > num_total_parts` evaluates to `false` (since `num_total_parts > num_total_parts` is false), so the invalid part is inserted into `chunk_forwards_cache`.
6. The legitimate chunk header arrives (from a block). `insert_header_if_not_exists_and_process_cached_chunk_forwards` merges the cached parts (including the invalid one) into `encoded_chunks` without re-validation.
7. Legitimate validators forward their parts; once `entry.parts.len() >= num_data_parts`, `can_reconstruct` becomes true.
8. The reconstruction loop at line 1910 executes `encoded_chunk.content_mut().parts[num_total_parts]` → **index out of bounds → panic → node crash**.
