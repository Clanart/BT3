### Title
Missing Shard-ID Binding Check in State-Sync Header Acceptance Allows Wrong-Shard State Injection — (`File: chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a peer-supplied `ShardStateSyncResponseHeader` for a requested `shard_id` without ever verifying that the embedded chunk actually belongs to that shard. A malicious peer can substitute a chunk from a different shard (with a valid merkle proof for that other shard's position) and the function accepts it, stores the wrong shard's `prev_state_root` under the requested shard's `StateHeaderKey`, and the syncing node subsequently downloads, validates, and applies state parts that belong to the wrong shard — permanently corrupting its trie for the requested shard.

---

### Finding Description

`set_state_header` (lines 368–531) performs five groups of checks on the incoming header:

1. **`validate_chunk_proofs`** — verifies chunk hash, `tx_root`, and `outgoing_receipts_root`. It does **not** check `chunk.shard_id()`.
2. **`verify_path` (line 394)** — verifies the chunk hash is reachable from `chunk_headers_root`. `verify_path` is **not position-aware**: it is implemented as `compute_root_from_path(path, item_hash) == root`, with no index constraint. A chunk from shard B with shard B's valid merkle proof passes this check even when shard A was requested.
3. **`verify_path` for `prev_chunk`** — same non-positional check.
4. **Incoming receipts proofs** — validated against the caller-supplied `shard_id`, not against `chunk.shard_id()`.
5. **`validate_state_root_node`** — validates the state root node against `chunk_inner.prev_state_root()`, which is shard B's root.

After all checks pass, the header is committed:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
```

The `state_root` later extracted in `set_state_part` (line 543) comes from this stored header:

```rust
let state_root = *chunk.take_header().take_inner().prev_state_root();
```

That root is shard B's root. All subsequent state-part validation and application use it, writing shard B's trie nodes into shard A's `ShardUId` slot.

The missing check is:

```rust
// Never present anywhere in set_state_header:
if chunk.shard_id() != shard_id {
    return Err(Error::Other("set_shard_state failed: chunk shard_id mismatch".into()));
}
```

---

### Impact Explanation

A syncing node that accepts the malicious header will:

1. Store shard B's `prev_state_root` under `StateHeaderKey(shard_A, sync_hash)`.
2. Download and validate state parts against shard B's root (they pass `validate_state_part_impl` because the trie proof is cryptographically correct for shard B).
3. Call `apply_state_part` which writes shard B's trie nodes into shard A's `ShardUId` in `DBCol::Tries` and shard B's flat-state entries into shard A's flat storage.
4. Attempt to process blocks: the block's chunk header for shard A commits to shard A's `prev_state_root`; the node's stored root is shard B's — every block application fails.
5. The node is permanently stuck; it cannot produce or validate chunks for shard A without a full wipe and re-sync.

The corrupted value is exact: `StateHeaders[(shard_A, sync_hash)]` contains shard B's chunk, and `DBCol::Tries` / flat storage for `shard_A`'s `ShardUId` contains shard B's trie nodes.

---

### Likelihood Explanation

State sync is triggered for every new node joining the network and for validators reassigned to a new shard each epoch. The syncing node requests state from any reachable peer; no privileged role is required to respond. An attacker operating a single honest-looking peer can intercept these requests and serve a cross-shard header. The attack requires no cryptographic forgery: the attacker uses a legitimately signed chunk from a different shard with its own valid merkle proof.

---

### Recommendation

Add an explicit shard-ID binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// Analog of check_account_owner: verify the chunk belongs to the requested shard.
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Optionally, replace the non-positional `verify_path` call with `verify_path_with_index` (already used in `validate_part`) to also enforce that the chunk occupies the correct index in the block's chunk-headers merkle tree, providing defence-in-depth.

---

### Proof of Concept

**Setup:** 4-shard network. Node S is syncing shard 0. Attacker A is a reachable peer.

**Attack steps:**

1. A intercepts S's `StateRequestHeader { shard_id: 0, sync_hash }` message.
2. A constructs a `ShardStateSyncResponseHeader` containing shard 1's `ShardChunk` (legitimately signed by the shard-1 chunk producer) and shard 1's merkle proof from the same block.
3. S calls `set_state_header(shard_id=0, sync_hash, malicious_header)`.
4. `validate_chunk_proofs` passes — shard 1's chunk is internally consistent.
5. `verify_path(chunk_headers_root, shard1_proof, ChunkHashHeight(shard1_hash, …))` passes — shard 1's chunk is in the block.
6. No `chunk.shard_id() != shard_id` check exists; the header is committed under `StateHeaderKey(0, sync_hash)`.
7. S downloads state parts for shard 0; they are validated against shard 1's `prev_state_root` and pass.
8. `apply_state_part` writes shard 1's trie into shard 0's `ShardUId`.
9. S attempts block processing: shard 0's chunk header has shard 0's `prev_state_root`; S's stored root is shard 1's — every block fails with a state-root mismatch.
10. S is permanently unable to process blocks for shard 0 without a full re-sync.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
