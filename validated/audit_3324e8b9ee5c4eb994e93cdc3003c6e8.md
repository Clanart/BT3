### Title
Meta Transaction v0 Hash Omits Nonce, Enabling Signature Replay to Drain Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` computes a transaction hash that does **not** include a nonce or any replay-protection field. An attacker who observes a valid `meta_tx_v0` signature on-chain can replay it an unlimited number of times, executing the same authorized action (e.g., a token transfer) repeatedly without the signer's consent.

---

### Finding Description

The `compute_meta_tx_v0_hash` function constructs a Pedersen hash over: `INVOKE_HASH_PREFIX`, `version=0`, `contract_address`, `entry_point_selector`, `calldata` (as a hash-chain), `max_fee=0`, and `chain_id`. It passes `additional_data_size=0` and `additional_data=cast(0, felt*)`, meaning **no nonce is committed to in the signed hash**.

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,
        additional_data=cast(0, felt*),   // ← no nonce
    );
``` [1](#0-0) 

Contrast this with every other transaction type in the same file:

- **L1 handler**: explicitly passes `additional_data_size=1, additional_data=&nonce`. [2](#0-1) 

- **Invoke v3 / Declare v3 / DeployAccount v3**: nonce is committed via `hash_tx_common_fields` → `poseidon_hash_update_single(item=common_fields.nonce)`. [3](#0-2) 

The function signature of `compute_meta_tx_v0_hash` accepts no `nonce` parameter at all, so there is no path by which a nonce could be incorporated into the signed digest. Because the hash is the only thing the account contract's signature verification covers, the signature is valid for any replay of the same `(contract_address, selector, calldata)` tuple on the same chain, forever.

This is structurally identical to the permit2 vulnerability in the external report: the authorization (signature) does not bind to the specific *use* (nonce/invocation), so the same authorization can be reused for unintended repeated executions.

---

### Impact Explanation

**Critical — Direct loss of funds.**

If a `meta_tx_v0` syscall is used to authorize a token transfer (e.g., calling an ERC-20 `transfer` or `transferFrom`), any party who observes the first on-chain use of the signature can immediately resubmit the same `(contract_address, selector, calldata, signature)` tuple in subsequent transactions. Because the OS hash function commits no nonce, the account contract's `is_valid_signature` check will accept the replayed signature as valid every time. The attacker can drain the account's balance up to the authorized amount per replay, with no bound on the number of replays.

---

### Likelihood Explanation

**High.** The attacker's entry path is entirely unprivileged:

1. A legitimate user submits a `meta_tx_v0` syscall (e.g., a gasless relay scenario).
2. The transaction is included in a block and its calldata — including the signature — becomes public on-chain.
3. Any observer constructs a new transaction that re-invokes the same syscall with the identical parameters and signature.
4. The OS re-runs `compute_meta_tx_v0_hash` over the same inputs, produces the same hash, and the account contract's signature check passes again.

No privileged access, leaked key, or network-level attack is required.

---

### Recommendation

Add a `nonce` parameter to `compute_meta_tx_v0_hash` and pass it as `additional_data`, mirroring the pattern used by `compute_l1_handler_transaction_hash`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,          // ← add nonce
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=1,
        additional_data=&nonce,   // ← commit nonce into hash
    );
    return tx_hash;
}
```

The OS must also enforce that each nonce value is consumed exactly once (e.g., via `check_and_increment_nonce` as done for all other account transaction types). [4](#0-3) 

---

### Proof of Concept

1. **Victim** calls a contract via `meta_tx_v0` syscall with:
   - `contract_address = TOKEN_CONTRACT`
   - `entry_point_selector = transfer_selector`
   - `calldata = [attacker_address, amount]`
   - Signs the hash produced by `compute_meta_tx_v0_hash(TOKEN_CONTRACT, transfer_selector, calldata, 2, chain_id)`.

2. The OS computes:
   ```
   H = Pedersen(INVOKE_PREFIX, 0, TOKEN_CONTRACT, transfer_selector,
                H_chain(calldata), 0, chain_id)
   ```
   No nonce is included. Signature verifies. Transfer executes.

3. **Attacker** observes the transaction on-chain, extracts `(contract_address, selector, calldata, signature)`.

4. Attacker submits a new transaction that triggers the same `meta_tx_v0` syscall with the identical parameters.

5. The OS recomputes the **identical** hash (same inputs, no nonce), the account contract's `is_valid_signature` returns `VALID`, and the transfer executes again.

6. Attacker repeats step 4–5 until the victim's balance is exhausted. [5](#0-4)
