Audit Report

## Title
Partial-ETH branch in `pay()` silently consumes stranded router ETH to subsidize a subsequent WETH swap ‚Äî (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` helper in `PeripheryPayments.sol` contains a middle branch (lines 78‚Äì81) that fires when `0 < address(this).balance < value` during a WETH payment. It unconditionally drains the router's entire ETH balance as a partial payment before pulling the remainder from the actual payer's allowance. Because the router's ETH balance carries no per-user attribution, any ETH stranded by a prior user (who sent `msg.value` but omitted `refundETH()`) is silently consumed by the next WETH swap callback, causing a direct, irreversible loss of principal for the original depositor.

## Finding Description
`pay()` has three branches when `token == WETH` and `payer != address(this)`:

```
nativeBalance >= value  ‚Üí  wrap value from ETH, pull nothing from payer       (L75‚Äì77)
nativeBalance > 0       ‚Üí  wrap nativeBalance from ETH, pull remainder from payer  (L78‚Äì81) ‚Üê root cause
nativeBalance == 0      ‚Üí  pull all from payer                                 (L82‚Äì84)
``` [1](#0-0) 

The middle branch unconditionally drains `address(this).balance` regardless of which user deposited it. The callback chain that reaches this branch is:

`exactInputSingle` (L67) ‚Üí `_setNextCallbackContext(..., msg.sender, ...)` (L71) ‚Üí `pool.swap(...)` ‚Üí `metricOmmSwapCallback` (L46) ‚Üí `_justPayCallback` (L192) ‚Üí `pay(WETH, payer, pool, amount)` (L193‚Äì198). [2](#0-1) 

`payer` is the original `msg.sender` stored in transient storage. When `tokenIn == WETH` and the router holds any residual ETH, the partial branch fires and consumes it before touching the payer's allowance. The `receive()` guard (L32‚Äì34) only blocks plain ETH transfers; it does not prevent `msg.value` from being deposited via payable entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, etc.). [3](#0-2) 

## Impact Explanation
Direct, irreversible loss of user principal. User A sends `msg.value = 200e18` with `amountIn = 100e18`; the callback wraps 100e18 ETH (full-cover branch), leaving 100e18 ETH stranded. If User A omits `refundETH()`, User B's subsequent `exactInputSingle` with `tokenIn = WETH`, `amountIn = 150e18` triggers the partial branch: the router wraps User A's 100e18 ETH and pulls only 50e18 WETH from User B. User A loses 100e18 ETH; User B receives the full swap output while underpaying by 100e18 WETH. The pool is made whole, so no pool insolvency occurs, but the loss to User A is concrete and unrecoverable. This meets the Sherlock critical/high threshold for direct loss of user principal.

## Likelihood Explanation
- `refundETH()` is not enforced by any entry point; it must be manually composed into a multicall, making omission common.
- Any payable entry point can leave residual ETH on the router.
- The exploit requires no privileged access: any address can call `exactInputSingle` with `tokenIn = WETH` to trigger the partial-ETH branch.
- The router's ETH balance is publicly readable on-chain; an attacker can monitor it and time the call immediately after a victim's transaction. [4](#0-3) 

## Recommendation
Replace the partial-ETH branch with the Uniswap v3 pattern: only use router ETH when it fully covers the payment; otherwise fall through to `safeTransferFrom`:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

This eliminates cross-user ETH consumption while preserving the intended "pay with native ETH" convenience for users who send exactly the right `msg.value`.

## Proof of Concept

```
State before:
  router.balance = 0
  User A WETH balance = 1000e18
  User B WETH balance = 1000e18, allowance to router = 1000e18

Step 1 ‚Äì User A strands ETH:
  User A calls exactInputSingle{value: 200e18}(
      pool, tokenIn=WETH, amountIn=100e18, ...
  )
  Callback: pay(WETH, UserA, pool, 100e18)
    nativeBalance=200e18 >= value=100e18 ‚Üí wraps 100e18 ETH, sends to pool
  After swap: router.balance = 100e18  (User A omits refundETH)

Step 2 ‚Äì User B exploits:
  User B calls exactInputSingle(
      pool, tokenIn=WETH, amountIn=150e18, ...
  )
  Callback: pay(WETH, UserB, pool, 150e18)
    nativeBalance=100e18 > 0, < 150e18 ‚Üí partial branch fires:
      wraps 100e18 ETH ‚Üí sends to pool
      safeTransferFrom(UserB, pool, 50e18)  ‚Üê only 50e18 pulled from User B

Result:
  User A lost 100e18 ETH (stranded, now consumed)
  User B paid 50e18 WETH instead of 150e18 WETH
  Pool received correct 150e18 WETH (no pool loss)
``` [5](#0-4)
