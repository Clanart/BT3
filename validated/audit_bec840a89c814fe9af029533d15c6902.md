### Title
Sub-unit Transfer Amount After Decimal Normalization Permanently Locks User Funds — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer` accepts any amount where `fee < amount`, but does not verify that the post-fee amount survives decimal normalization to the destination chain's precision. When `normalize_amount(amount - fee)` floors to zero, every subsequent call to `sign_transfer` panics with `InvalidAmountToTransfer`. Because no cancellation path exists for pending transfers, the user's tokens are permanently frozen in the bridge.

---

### Finding Description

**Step 1 — `init_transfer` stores the transfer without checking normalized amount**

`init_transfer` only enforces `fee.fee < transfer_message.amount`: [1](#0-0) 

It does **not** verify that `normalize_amount(amount - fee) > 0`. The transfer message is then stored in `pending_transfers` and the user's tokens are locked/burned by the `ft_transfer_call` mechanism.

**Step 2 — `normalize_amount` uses floor division** [2](#0-1) 

For a token with `origin_decimals = 24` (e.g., a NEAR-native token) and `decimals = 18` (EVM representation), the divisor is `10^6 = 1_000_000`. Any `amount_without_fee < 1_000_000` normalizes to **zero**.

**Step 3 — `sign_transfer` panics on zero normalized amount** [3](#0-2) 

The relayer calls `sign_transfer`, which computes `normalize_amount(amount_without_fee, decimals)`. If the result is 0, the function panics. The MPC signing call is never reached, so `sign_transfer_callback` is never invoked.

**Step 4 — No cancellation path exists**

`sign_transfer_callback` only removes the transfer message when the MPC signing succeeds **and** `fee.is_zero()`: [4](#0-3) 

`remove_transfer_message` is otherwise only called from `claim_fee_callback`, which requires a proof of a completed `fin_transfer` on the destination chain — impossible if the transfer was never signed. There is no `cancel_transfer` or user-initiated refund function. The transfer remains in `pending_transfers` indefinitely.

**Step 5 — `update_transfer_fee` cannot rescue the transfer**

`update_transfer_fee` only allows the fee to be **increased** (`fee.fee >= current_fee.fee`): [5](#0-4) 

Increasing the fee makes `amount_without_fee` even smaller, worsening the situation. The user cannot reduce the fee to recover a viable amount.

---

### Impact Explanation

User tokens are permanently frozen in the bridge. The transfer is stored in `pending_transfers` with the user's tokens already locked or burned, but `sign_transfer` will revert on every call. There is no recovery path. This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user funds in bridge flows.**

---

### Likelihood Explanation

The condition is reachable by any unprivileged bridge user:

- Tokens with `origin_decimals > decimals` (e.g., 24 → 18, diff = 6) are a normal configuration for NEAR-native tokens bridged to EVM chains.
- A user transferring fewer than `10^6` base units of such a token (with any fee ≥ 0) triggers the bug.
- The user may not be aware of the decimal normalization threshold, especially for tokens with high precision.
- No special role or privileged access is required — only a standard `ft_transfer_call` to the bridge.

---

### Recommendation

Add a normalization check inside `init_transfer` before storing the transfer:

```rust
let normalized = Self::normalize_amount(
    transfer_message.fee.fee.0
        .checked_sub(transfer_message.fee.fee.0) // amount_without_fee
        ...,
    decimals,
);
require!(
    normalized > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

Concretely, after computing `transfer_message` and looking up `decimals` for the destination token, verify:

```rust
let decimals = self.token_decimals.get(&token_address)...;
let normalized = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(...),
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

This mirrors the check already present in `sign_transfer` but moves it to the point where the transfer is created, before tokens are locked.

---

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 18` (diff = 6, divisor = 1,000,000).
2. User calls `ft_transfer_call` on the token contract with `amount = 500_000` and `fee = 0`, targeting the bridge with an `InitTransfer` message to an EVM recipient.
3. `init_transfer` passes: `fee (0) < amount (500_000)`. Transfer stored; 500,000 tokens locked.
4. Relayer calls `sign_transfer(transfer_id, ...)`.
5. `normalize_amount(500_000, {origin_decimals: 24, decimals: 18})` = `500_000 / 1_000_000` = **0**.
6. `require!(0 > 0, ...)` panics — transaction reverts.
7. Transfer remains in `pending_transfers`. No signing, no `fin_transfer`, no `claim_fee` possible.
8. User's 500,000 tokens are permanently locked with no recovery path. [2](#0-1) [3](#0-2) [1](#0-0)
