### Title
`ProofOfInclusion::valid()` Tautological Root Check Allows Forged DataLayer Inclusion Proofs - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary

`ProofOfInclusion::valid()` only verifies the internal hash-chain consistency of the proof. Its final root comparison is a tautology — it compares the proof's own embedded `combined_hash` against itself. No external authoritative root is ever checked. An unprivileged attacker can construct a `ProofOfInclusion` with an arbitrary `node_hash` (representing any key-value pair they wish to "prove"), supply consistent sibling hashes, and `valid()` will return `true`, regardless of whether the claimed leaf exists in the actual DataLayer tree.

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // ← existing_hash = layer.combined_hash
    }

    existing_hash == self.root_hash()      // ← tautology
}
``` [1](#0-0) 

`root_hash()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // ← same field just verified in the loop
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

After the loop completes without returning `false`, `existing_hash` holds the last iteration's `calculated_hash`, which was already asserted equal to `layer.combined_hash`. `root_hash()` returns that same `layer.combined_hash`. Therefore `existing_hash == self.root_hash()` is unconditionally `true` whenever the loop body passes — the final check adds no security.

The function validates only that the hash chain is internally self-consistent (each `combined_hash` is correctly derived from the previous hash and the supplied sibling). It never compares the resulting root against any external, authoritative tree root. The "fixed" value used is the proof's own embedded `combined_hash` — fully attacker-controlled — rather than the actual current DataLayer tree root.

`ProofOfInclusion` is `Streamable` (deserializable from raw bytes) and is exposed to Python with `pyclass(get_all, from_py_object)`, meaning untrusted callers can construct arbitrary instances: [3](#0-2) 

The `valid()` and `root_hash()` methods are both exposed to Python: [4](#0-3) 

### Impact Explanation

An attacker who can supply a serialized `ProofOfInclusion` to any Python DataLayer consumer that calls `proof.valid()` as its sole check can forge a proof of inclusion for any arbitrary key-value pair. The attacker:

1. Chooses a target `node_hash` (e.g., `H(fake_key || fake_value)`).
2. Constructs any number of layers with arbitrary `other_hash` values and correctly computed `combined_hash` values.
3. Submits the proof; `valid()` returns `true`.
4. The `root_hash()` will be some attacker-chosen value, not the actual tree root — but if the caller does not separately verify `root_hash()` against the known authoritative root, the forged proof is accepted.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` type exposed to Python with `from_py_object`. Any code path that receives a proof from an untrusted peer and validates it solely with `proof.valid()` — without also asserting `proof.root_hash() == known_root` — is exploitable. The misleading name `valid()` makes it likely that callers treat it as a complete validity check. Likelihood is **High** given the API design.

### Recommendation

The `valid()` method must accept an external authoritative root and compare against it:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root   // compare against external root, not self
}
```

The existing `valid()` method (which only checks internal consistency) should be renamed to `is_internally_consistent()` or removed, to prevent callers from mistaking it for a complete security check. All Python-facing call sites must be updated to pass the known authoritative tree root.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side};
use chia_datalayer::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};

fn forge_proof() {
    // Attacker wants to "prove" that fake_node_hash is in the tree.
    let fake_node_hash: Hash = [0xAA; 32];
    let fake_sibling:   Hash = [0xBB; 32];

    // Compute a valid combined_hash from the fake inputs.
    let combined = chia_datalayer::calculate_internal_hash(
        &fake_node_hash, Side::Left, &fake_sibling,
    );

    let forged = ProofOfInclusion {
        node_hash: fake_node_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Left,
            other_hash: fake_sibling,
            combined_hash: combined,   // attacker-controlled root
        }],
    };

    // valid() returns true — no external root is checked.
    assert!(forged.valid());
    // root_hash() is the attacker-chosen `combined`, not the real tree root.
    assert_eq!(forged.root_hash(), combined);
}
```
