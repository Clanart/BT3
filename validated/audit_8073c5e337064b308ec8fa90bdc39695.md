### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by a contract is a previously declared class. This is directly analogous to the reported "missing access control in burn" pattern: just as `burn()` lacked a check on *who* is being burned, `execute_replace_class` lacks a check on *what value* is being written. A contract can set its own class hash to any arbitrary, undeclared felt value. Once committed to state, all subsequent calls to that contract will fail at the OS level because no compiled class exists for the hash, permanently freezing any funds the contract holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the requested new class hash directly from the syscall request and writes it into `contract_state_changes` without any validation that the hash corresponds to a declared class:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The developer-acknowledged TODO at line 898 confirms the missing check is known but unimplemented. The OS accepts any felt as a valid class hash replacement with no cross-reference against the `compiled_class_facts` bundle or `contract_class_changes` dictionary.

By contrast, the `execute_declare_transaction` path enforces `prev_value=0` to prevent re-declaration and calls `finalize_class_hash` to verify the class hash pre-image: [2](#0-1) 

No equivalent guard exists in `execute_replace_class`.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value and the block proof is accepted on L1, the state change is final. Every subsequent transaction that attempts to invoke the contract will fail at the OS execution layer because the OS cannot resolve the class hash to a compiled class entry in `compiled_class_facts`. The contract becomes permanently inert. Any ERC-20 balances, ETH, or other assets held in the contract's storage are irrecoverably frozen with no upgrade or rescue path, since the upgrade mechanism (`replace_class`) itself requires a working contract execution context.

---

### Likelihood Explanation

The `replace_class` syscall is a standard, documented StarkNet syscall reachable by any Sierra contract. The attack surface includes:

1. **Malicious contract pattern**: An attacker deploys a contract (e.g., a vault or shared pool), attracts user deposits, then calls `replace_class` with `class_hash = 0xdeadbeef` (any undeclared felt). The OS accepts the state transition. Funds are frozen.
2. **Buggy contract pattern**: A legitimate contract with a logic error passes an unvalidated user-supplied value to `replace_class`. The OS provides no safety net.
3. **Reentrancy/cross-contract pattern**: A callee contract manipulates a caller into invoking `replace_class` with an attacker-controlled hash via `call_contract`.

The syscall is dispatched unconditionally in `execute_syscalls` when `selector == REPLACE_CLASS_SELECTOR` with no pre-flight checks: [3](#0-2) 

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, the OS must verify that the hash exists as a key in `contract_class_changes` (i.e., it was declared in the current block) or exists in the pre-existing state's class mapping. Concretely, add a `dict_read` on `contract_class_changes` for `class_hash` and assert the returned compiled class hash is non-zero before proceeding with the `dict_update`. This mirrors the guard already present in `execute_declare_transaction`: [4](#0-3) 

---

### Proof of Concept

1. Attacker deploys `VaultContract` (class hash `0xAAAA`, declared) with an `__execute__` that accepts deposits and exposes a `drain_via_replace` entry point.
2. Users deposit 1000 STRK into `VaultContract`.
3. Attacker calls `drain_via_replace`, which internally issues:
   ```
   replace_class(class_hash=0xDEAD)  // 0xDEAD is never declared
   ```
4. The OS processes `execute_replace_class`:
   - Reads `request.class_hash = 0xDEAD`.
   - Skips the missing declared-class check (line 898 TODO).
   - Writes `StateEntry(class_hash=0xDEAD, ...)` into `contract_state_changes`.
5. Block proof is generated and verified on L1. State is committed: `VaultContract.class_hash = 0xDEAD`.
6. Any subsequent `call_contract` to `VaultContract` causes the OS to look up compiled class `0xDEAD`, find nothing, and fail. The transaction reverts.
7. The 1000 STRK in `VaultContract`'s storage is permanently inaccessible. [5](#0-4)
