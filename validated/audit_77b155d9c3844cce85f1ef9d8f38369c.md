### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Signature Replay Leading to Direct Loss of Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall in the StarkNet OS allows any contract to execute a version-0 style call on behalf of a user, authenticated by a user-provided signature. The hash committed to by that signature — computed in `compute_meta_tx_v0_hash` — does **not** include a nonce or any other replay-prevention field. Because version-0 transactions also explicitly skip nonce checking in `check_and_increment_nonce`, the same meta_tx_v0 signature can be replayed an unlimited number of times within the same or subsequent blocks, causing the victim's account to execute the same operation repeatedly and draining funds beyond the user's intent.

---

### Finding Description

`execute_meta_tx_v0` is a syscall that any executing contract can invoke. It:

1. Reads a caller-supplied `(contract_address, selector, calldata, signature)` from the syscall pointer.
2. Computes a transaction hash via `compute_meta_tx_v0_hash`.
3. Constructs a synthetic `TxInfo` with `version=0` and `nonce=0`.
4. Calls `__execute__` on the target contract with that synthetic context.

The hash function is:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),  // ← no nonce
    );
    return tx_hash;
}
```

`additional_data_size=0` means the nonce is entirely absent from the signed digest. The hash is a pure function of `(prefix, version=0, contract_address, selector, calldata, chain_id)`.

In `execute_meta_tx_v0`, the synthetic `TxInfo` is built with `nonce=0` and `version=0`:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,
    ...
);
```

Because `check_and_increment_nonce` explicitly returns early for `version == 0` without touching state, no on-chain counter is ever incremented for this call path:

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
```

`execute_meta_tx_v0` never calls `check_and_increment_nonce` at all — it goes directly to `contract_call_helper`. There is no other replay-prevention mechanism anywhere in the function.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A user who signs a meta_tx_v0 payload (e.g., to authorize a token transfer from their account) produces a signature that is valid for the same `(contract_address, selector, calldata, chain_id)` tuple forever. Any observer of the original transaction can replay the same signature in a new transaction, causing the victim's `__execute__` to run again with identical calldata. If the calldata encodes a token transfer, the victim's balance is drained by the replayed amount on each replay. Because there is no nonce and no state update to invalidate the signature, this can be repeated until the victim's account is empty.

---

### Likelihood Explanation

The `execute_meta_tx_v0` syscall is a public protocol feature reachable by any unprivileged contract. The attacker's only requirement is to observe a submitted meta_tx_v0 signature — available from the mempool or from on-chain transaction data. No privileged access, leaked key, or social engineering is required. The attacker simply submits a new v3 transaction that calls the same syscall with the replayed signature.

---

### Recommendation

Include a nonce in the `compute_meta_tx_v0_hash` digest. Concretely, pass `additional_data_size=1` and supply a per-account, monotonically incrementing nonce as `additional_data`. The OS must then verify and increment this nonce in state (analogous to how `check_and_increment_nonce` works for v1/v3 transactions) before executing the meta call. This ensures each signed meta_tx_v0 payload is single-use.

---

### Proof of Concept

1. Alice holds 1000 tokens in her StarkNet account. She signs a meta_tx_v0 payload authorizing a transfer of 500 tokens to Bob. The signed hash covers only `(INVOKE_PREFIX, 0, alice_address, __execute__, calldata, chain_id)` — no nonce.

2. A relayer submits a v3 transaction that calls `execute_meta_tx_v0` with Alice's signature. Alice's account executes `__execute__`, transferring 500 tokens to Bob. Alice now has 500 tokens.

3. Mallory observes Alice's signature from the submitted transaction. Mallory submits a second v3 transaction calling `execute_meta_tx_v0` with the identical signature. Because `compute_meta_tx_v0_hash` produces the same hash (no nonce), Alice's account validates the signature again and executes `__execute__` a second time, transferring another 500 tokens to Bob. Alice now has 0 tokens.

4. Mallory can repeat step 3 in subsequent blocks until Alice's account is empty, as no on-chain state is ever updated to invalidate the signature.

**Root cause lines:** [1](#0-0) 

`additional_data_size=0` omits the nonce from the signed digest entirely. [2](#0-1) 

`nonce=0` and `version=0` are hardcoded in the synthetic `TxInfo`, ensuring `check_and_increment_nonce` is never invoked. [3](#0-2) 

`check_and_increment_nonce` explicitly skips all version-0 transactions, providing no replay protection for this path.
