### Title
Stale Pending Class Served by `get_class` After Block Commit Due to Missing `read_pending_data` Guard — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary

`get_class` for `BlockId::Tag(Tag::Pending)` reads directly from `self.pending_classes` without calling `read_pending_data`, bypassing the parent-hash staleness guard that every other pending-aware RPC method enforces. After a new block is committed to storage, there is a window before the sync loop clears `pending_classes` during which `starknet_getClass(Pending, H)` returns a class body from the old pending block while all other pending endpoints return an empty synthetic pending block.

---

### Finding Description

**The guard that other methods use**

`read_pending_data` enforces the invariant that pending data is only served when `pending_data.block.parent_block_hash() == latest_header.block_hash`. When the hashes diverge it returns a synthetic empty `PendingData` with a zeroed state diff: [1](#0-0) 

Every other pending-aware method calls this guard before touching pending state:
- `get_nonce` — [2](#0-1) 
- `get_state_update` — [3](#0-2) 
- `maybe_get_class_hash_at` — [4](#0-3) 

**`get_class` skips the guard entirely**

For `BlockId::Tag(Tag::Pending)`, `get_class` reads directly from `self.pending_classes` with no parent-hash check: [5](#0-4) 

**When `pending_classes` is cleared**

`pending_classes` is only cleared inside the sync loop's `get_pending_data` helper, and only when it detects a parent-hash change in the freshly-fetched pending data: [6](#0-5) 

The sync loop itself only enters the pending-sync path when `header_marker == central_block_marker`: [7](#0-6) 

**The race window**

Between the moment a new block is committed to storage (advancing `latest_header.block_hash`) and the moment the sync loop next polls and clears `pending_classes`, the following inconsistency exists:

| Endpoint | Behaviour |
|---|---|
| `starknet_getClass(Pending, H)` | Returns stale class body from old pending block |
| `starknet_getStateUpdate(Pending)` | Returns empty state diff (no declared classes) |
| `starknet_getBlockWithTxHashes(Pending)` | Returns empty synthetic block |
| `starknet_getNonce(Pending, addr)` | Returns committed nonce (no pending overlay) |

The class `H` is presented as belonging to the current pending block even though the current pending block (as seen by every other endpoint) is empty and does not declare `H`.

---

### Impact Explanation

**High — RPC pending view returns an authoritative-looking wrong value.**

A caller receives a `SierraContractClass` body for class hash `H` under `Pending` tag. Every other pending endpoint simultaneously reports that the pending block is empty and `H` is not declared. The caller cannot distinguish this from a legitimately declared pending class. Any tooling that caches or acts on `starknet_getClass(Pending, ...)` responses (e.g., declare-before-deploy checks, SDK class caches, fee estimators that resolve class bodies) will operate on a class that is not part of the current pending block.

---

### Likelihood Explanation

The window is bounded by the sync loop's polling interval. During normal operation the sync loop polls continuously, so the window is short but non-zero and repeatable on every block boundary. Any RPC client that polls `starknet_getClass(Pending, H)` at a high rate around block boundaries will reliably hit this window. No special privileges are required.

---

### Recommendation

Apply the same `read_pending_data` guard in `get_class` before consulting `pending_classes`. Concretely, when `block_id` is `Pending`, open a storage transaction, call `read_pending_data`, and only proceed to look up `pending_classes` if the returned `PendingData` has a non-empty state diff (i.e., the pending block is not the synthetic empty one). Alternatively, add a `parent_block_hash` field to `PendingClasses` that is set atomically with `pending_data` and checked in `get_class` before returning a cached entry.

---
