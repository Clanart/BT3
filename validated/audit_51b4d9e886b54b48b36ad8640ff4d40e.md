### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Breaks Bridge Collateralization - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
The `initTransfer` function in `OmniBridge.sol` emits the user-supplied `amount` parameter in the `InitTransfer` event without verifying that the actual amount received by the bridge contract matches it. For fee-on-transfer ERC-20 tokens, the bridge receives `amount - fee` but the NEAR side processes the event and releases the full `amount`, creating unbacked supply and breaking bridge collateralization.

### Finding Description
In `initTransfer`, when a non-bridge, non-custom-minter ERC-20 token is deposited, the contract pulls tokens from the caller and then emits an event using the caller-supplied `amount`:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol lines 407-412
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // pulls `amount` from sender
    );
}
```

Immediately after, the same caller-controlled `amount` is broadcast in the cross-chain event:

```solidity
// lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,             // user-supplied, NOT measured balance delta
    fee,
    nativeFee,
    recipient,
    message
);
```

The bridge never measures its own balance before and after the `safeTransferFrom` call. For fee-on-transfer tokens, `safeTransferFrom` succeeds and returns `true`, but the bridge receives only `amount - transferFee`. The emitted `InitTransfer` event still carries the full `amount`, which the NEAR bridge uses to determine how many tokens to mint or unlock for the recipient.

This is the direct analog of the reported vulnerability: the original converter tracked sent-out tokens by measuring the **recipient's** balance change (manipulable via ERC-777 hook); here the bridge tracks received tokens by trusting the **caller-supplied** `amount` (manipulable by using any fee-on-transfer token), rather than measuring the bridge's own balance delta.

### Impact Explanation
Every `initTransfer` call with a fee-on-transfer token creates a collateral shortfall equal to the transfer fee. The NEAR side mints or releases `amount` tokens while the EVM bridge holds only `amount - transferFee`. An attacker can:

1. Repeatedly bridge fee-on-transfer tokens EVM → NEAR, receiving full `amount` on NEAR while depositing only `amount - fee` on EVM.
2. Bridge the surplus back NEAR → EVM, draining the bridge's reserves of that token.
3. Honest users who bridged the same token earlier will find their withdrawals permanently unclaimable once the bridge's actual balance is exhausted.

This matches the allowed impact: **Balance/accounting corruption that breaks bridge collateralization or misdirects value**, and **permanent freezing/unclaimable settlement of user funds**.

### Likelihood Explanation
The `else` branch (lines 407–412) is the generic path for any ERC-20 token that is neither a bridge-deployed token nor a custom-minter token. Fee-on-transfer tokens (e.g., PAXG, STA, tokens with built-in tax mechanisms) are a well-known, deployed token class. Any user can call `initTransfer` with such a token without any privileged role. The entry path is fully unprivileged and reachable on mainnet.

### Recommendation
Measure the bridge's actual token balance before and after the `safeTransferFrom` call and use the delta as the canonical `amount` for the event and any downstream accounting:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived in place of amount for the event and initTransferExtension
```

This mirrors the fix applied in the referenced Venus Protocol commit (939da0a), which switched from measuring the recipient's balance to measuring the sender/contract's balance delta.

### Proof of Concept

1. Deploy a fee-on-transfer ERC-20 token `FeeToken` that deducts 10% on every `transferFrom`.
2. Approve the OmniBridge to spend 1000 `FeeToken`.
3. Call `initTransfer(feeTokenAddress, 1000, 0, 0, nearRecipient, "")`.
4. Inside `safeTransferFrom`, `FeeToken` transfers 900 to the bridge and burns 100 as fee. The call succeeds.
5. The bridge emits `InitTransfer(..., amount=1000, ...)` — the full user-supplied value.
6. The NEAR relayer observes the event and finalizes the transfer, minting/releasing 1000 tokens to `nearRecipient`.
7. The bridge now holds 900 `FeeToken` but the NEAR side has issued 1000 — a 100-token collateral shortfall per call.
8. Repeat until the bridge's `FeeToken` reserve is exhausted; subsequent legitimate withdrawals revert because the bridge cannot cover the full `amount`. [1](#0-0) [2](#0-1) [3](#0-2)
