The code evidence is conclusive. Let me trace the exact path:

**`parse_data_for_block`** in `crates/apollo_p2p_sync/src/client/state_diff.rs` reads only `state_diff_length` from the stored header and validates only that `current_state_diff_len == target_state_diff_len`. The string `state_diff_commitment` does not appear anywhere in that file. No call to `calculate_state_diff_hash` is made, and no comparison against the header's `state_diff_commitment` field is performed before `write_to_storage` calls `append_state_diff`.

The `CommitterConfig.verify_state_diff_hash` guard exists in `crates/apollo_committer_config/src/config.rs` (default `true` in production), but it is in the **Committer** component, which receives state diffs from the **batcher's** storage path — not from the p2p sync storage path. The batcher reads via `batcher_storage_reader.get_state_diff(height)`, which is a separate storage from what `append_state_diff` writes to in the p2p sync path.

The proposed **Critical** impact — wrong state diff applied to the Patricia trie producing a wrong global state root — is **not reachable** through this path. The Committer/Patricia trie computation is driven by the batcher's own storage, not by the p2p sync storage. A full node running p2p sync does not run the Committer at all.

The **actual** reachable impact is **High**: the wrong `ThinStateDiff` is stored in the state sync storage and served authoritatively via RPC (e.g., `starknet_getStorageAt`, `starknet_getClassAt`). This is a real commitment-binding gap — the stored `ThinStateDiff` for block N is not bound to the `state_diff_commitment` in the header — but it does not propagate to the Patricia trie root.

### Title
Missing `state_diff_commitment` hash verification in `StateDiffStreamBuilder::parse_data_for_block` allows a malicious p2p peer to store a wrong `ThinStateDiff` — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary
`parse_data_for_block` validates only the byte-length of assembled state diff chunks against `state_diff_length` from the stored header. It never reads `state_diff_commitment` from the header, never calls `calculate_state_diff_hash`, and never compares the assembled `ThinStateDiff` against the committed hash. A malicious peer can supply chunks whose lengths sum to `target_state_diff_len` but whose content is entirely different from the committed state diff, and the node will store and serve the wrong state.

### Finding Description
In `crates/apollo_p2p_sync/src/client/state_diff.rs`, `parse_data_for_block` (lines 51–110):

1. Reads `target_state_diff_len` from `header.state_diff_length` only. [1](#0-0) 
2. Loops accumulating chunks until `current_state_diff_len == target_state_diff_len`. [2](#0-1) 
3. Calls `validate_deprecated_declared_classes_non_conflicting` (duplicate-class check only). [3](#0-2) 
4. Returns `Ok(Some((result, block_number)))` — no hash check. [4](#0-3) 

`write_to_storage` then calls `append_state_diff` directly with no further verification. [5](#0-4) 

The `CommitterConfig.verify_state_diff_hash` guard (default `true`) exists only in the Committer component and is never invoked in this path. [6](#0-5) 

Header signature validity is also not checked in `HeaderStreamBuilder::parse_data_for_block` — only signature count is validated — so a peer can also supply a header with an arbitrary `state_diff_commitment`. [7](#0-6) 

### Impact Explanation
**High** — not Critical as proposed. The wrong `ThinStateDiff` is stored in the state sync storage and served via RPC (e.g., `starknet_getStorageAt`, `starknet_getClassAt`, `starknet_getClassHashAt`). The Patricia trie / global state root is **not** affected: the Committer reads from the batcher's separate storage path (`batcher_storage_reader.get_state_diff`), not from the p2p sync storage. [8](#0-7) 

The proposed Critical impact (wrong global state root for all subsequent blocks) is not reachable through this path alone.

### Likelihood Explanation
Any peer the node connects to can exploit this. The p2p sync client sends queries to network peers and accepts their responses without cryptographic binding of the state diff content to the commitment. The attack requires only that the peer craft chunks whose `len()` values sum to `target_state_diff_len` with different content.

### Recommendation
After assembling the full `ThinStateDiff`, call `calculate_state_diff_hash(&result)` and compare against `header.state_diff_commitment`. Reject with `BadPeerError` if they differ. This mirrors the existing guard in the Committer: [9](#0-8) 

### Proof of Concept
1. Store a header for block N with `state_diff_length = 2` and `state_diff_commitment = H(correct_diff)`.
2. Send two `StateDiffChunk` values each with `len() == 1` but with wrong addresses/storage keys/values (total length = 2 = `target_state_diff_len`).
3. `parse_data_for_block` returns `Ok(Some((wrong_diff, N)))`.
4. `write_to_storage` stores `wrong_diff` for block N.
5. Assert `calculate_state_diff_hash(&wrong_diff) != header.state_diff_commitment` — the commitment invariant is violated.
6. RPC `starknet_getStorageAt` for any key in `wrong_diff` returns the attacker-chosen value.
