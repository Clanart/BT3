### Title
AnteCache LRU Eviction Enables Targeted DoS on EVM Transaction Replacement/Cancellation - (File: ante/cache/antecache.go)

### Summary
The global `AnteCache` uses LRU eviction bounded by `mempoolMaxTxs`. An unprivileged attacker can flood the cache with valid transactions from many addresses, evicting a victim's `(address, nonce)` entry. Once evicted, the victim can no longer submit a replacement or cancellation transaction for their pending tx: `CheckAndSetEthSenderNonce` rejects it with `ErrInvalidSequence` because the `checkState` sequence was already incremented by the original tx, and the cache shortcut that would bypass the nonce check is gone.

### Finding Description

**Root cause — `AnteCache.Set` evicts the oldest global entry when full:** [1](#0-0) 

When `maxTx > 0` and the cache is at capacity, the oldest `(address, nonce)` pair across **all senders** is evicted. The cache is initialized in production with `mempoolMaxTxs`: [2](#0-1) 

**How the replacement shortcut works — `CheckAndSetEthSenderNonce`:** [3](#0-2) 

During `CheckTx`, if `(address, nonce)` is present in the cache, the nonce sequence check is skipped entirely (`continue`), allowing a replacement tx with the same nonce to pass. If the entry is absent, the code falls through to the strict sequence check: [4](#0-3) 

**Why the sequence is already N+1 after the first CheckTx:** [5](#0-4) 

The `checkState` sequence is incremented during `CheckTx`. After the original tx passes `CheckTx`, `expectedNonce = N+1`. A replacement tx with `nonce = N` will fail `txNonce != expectedNonce` unless the cache entry `(victim, N)` is present.

**The cache entry is committed only after the full ante chain succeeds:** [6](#0-5) 

**Attack steps:**
1. Victim submits tx with nonce N → `(victim, N)` added to `AnteCache`; `checkState` sequence incremented to N+1.
2. Attacker submits `maxTx` valid transactions from distinct funded addresses → each new entry evicts the oldest; `(victim, N)` is eventually evicted.
3. Victim attempts to replace/cancel their tx (same nonce N, higher gas price) → `AnteCache.Exists(victim, N)` returns `false` → `txNonce(N) != expectedNonce(N+1)` → `ErrInvalidSequence`.
4. Victim's original tx is stuck in the mempool and cannot be replaced or cancelled.

The test comment in the codebase itself acknowledges this class of issue: [7](#0-6) 

### Impact Explanation
The victim cannot replace or cancel any pending transaction. If the original tx sends funds to an unintended recipient (e.g., a mistaken address), the victim cannot cancel it before it is included in the next block. The attacker can repeat the flood after each block commit (when `checkState` resets) to persistently deny the victim the ability to replace any transaction. This is a mempool handling bug that causes valid user funds to be mis-accounted (sent to an unintended destination with no recourse).

### Likelihood Explanation
The attacker needs `maxTx` funded accounts, each submitting one valid transaction that passes `CheckTx`. With a typical `maxTx` of a few thousand and a low minimum fee, the cost per attack round is modest. No privileged access, governance, or validator collusion is required — only the ability to broadcast standard EVM transactions via `eth_sendRawTransaction`.

### Recommendation
1. **Per-address cache limits**: Partition the cache by sender address so one sender cannot evict another's entries.
2. **Mempool-backed lookup**: Replace the `AnteCache` with a direct mempool query for `(address, nonce)` existence, eliminating the eviction problem entirely (the TODO comment in `antecache.go` already notes this as the intended long-term fix).
3. **Protect victim entries**: When evicting, skip entries whose corresponding tx is still present in the mempool.

### Proof of Concept
```
// 1. Victim submits tx with nonce N
eth_sendRawTransaction(sign(victim_key, nonce=N, gasPrice=low))
// AnteCache: {(victim, N), ...}, checkState.sequence[victim] = N+1

// 2. Attacker floods cache with maxTx transactions from distinct accounts
for i in 1..maxTx:
    eth_sendRawTransaction(sign(attacker_key_i, nonce=0, gasPrice=min))
// AnteCache: full with attacker entries; (victim, N) evicted

// 3. Victim tries to replace with higher gas price
eth_sendRawTransaction(sign(victim_key, nonce=N, gasPrice=high))
// CheckAndSetEthSenderNonce:
//   expectedNonce = N+1 (from checkState)
//   txNonce       = N
//   cache.Exists(victim, N) = false  ← evicted
//   N != N+1  →  ErrInvalidSequence
// Result: replacement rejected; original tx stuck in mempool
```
