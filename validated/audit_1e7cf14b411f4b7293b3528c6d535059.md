### Title
Missing `chunk.shard_id() == shard_id` Guard in `set_state_header` Allows Cross-Shard State Root Substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for `shard_id_A` whose embedded `chunk` belongs to `shard_id_B`. The `chunk_headers_root` Merkle path check (`verify_path`) confirms only that the chunk hash is *somewhere* in the block's Merkle tree; it does not bind the chunk to the shard-index position corresponding to `shard_id_A`. Because no explicit `chunk.shard_id() == shard_id` guard exists, the header is stored under `StateHeaderKey(shard_id_A, sync_hash)`. Subsequent `set_state_part` calls extract `chunk_B.prev_state_root()` as the state root for shard_id_A, validate trie parts against shard_id_B's trie, and `apply_state_part` installs those nodes under shard_id_A's `ShardUId`, corrupting shard_id_A's trie.

### Finding Description

**Entrypoint:** `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`.

The function performs five validation steps before persisting the header:

**Step 1-2** — `validate_chunk_proofs(&chunk, ...)` verifies the chunk's internal hash, tx root, and receipts root. It does not check `chunk.shard_id()`. [1](#0-0) 

**Step 3a** — `verify_path` confirms that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in `sync_prev_block_header.chunk_headers_root()`. [2](#0-1) 

The critical flaw: `verify_path` is a pure root-reconstruction check — it does not verify the leaf's *position* (shard index) in the tree. [3](#0-2) 

A position-aware variant `verify_path_with_index` exists but is not used here. [4](#0-3) 

Therefore, an attacker who supplies chunk_B (shard_id_B's chunk, at Merkle-tree position `j`) with its valid Merkle path will pass the check even though `shard_id_A` corresponds to position `i ≠ j`. The path for position `j` reconstructs the same root as the path for position `i` — the root is shared by all leaves.

**Step 5** — `validate_state_root_node` only checks `hash(state_root_node.data) == chunk_inner.prev_state_root()`. It is shard-agnostic. [5](#0-4) 

**Storage** — The header is stored under `StateHeaderKey(shard_id_A, sync_hash)` with no shard-id binding inside the value. [6](#0-5) 

**`set_state_part` propagation** — reads the stored header, extracts `state_root = chunk.take_header().take_inner().prev_state_root()` (chunk_B's root), and validates parts against it. [7](#0-6) 

`validate_state_part_impl` delegates to `Trie::validate_state_part(state_root, part_id, partial_state)`. The `shard_id` argument is used only for metrics; the trie validation is purely root-hash-based. [8](#0-7) 

**`apply_state_part` installation** — uses `shard_uid = get_shard_uid_from_epoch_id(shard_id_A, epoch_id)` and calls `tries.apply_all(&trie_changes, shard_uid, ...)`, writing shard_id_B's trie nodes into shard_id_A's storage namespace. [9](#0-8) 

**Receipt proof bypass** — The receipt proof loop (step 4) hashes receipts as `ReceiptList(shard_id_A, receipts)`. The attacker supplies shard_id_A's actual receipt proofs (public chain data) for the `incoming_receipts_proofs` field, and shard_id_A's `prev_chunk_header` (also public) for the `prev_chunk_header` field. Both are verifiable against on-chain Merkle roots. Step 4g's termination check then passes because the receipt chain terminates at shard_id_A's `prev_chunk_header.height_included()`. [10](#0-9) 

The `chunk` field alone carries shard_id_B's identity and state root; all other fields can be legitimately sourced from shard_id_A's on-chain data.

### Impact Explanation

A syncing node installs shard_id_B's trie under shard_id_A's `ShardUId`. The receipt queue state, account balances, contract storage, and all other trie-backed state for shard_id_A are replaced with shard_id_B's data. The node then applies shard_id_A's pending chunk on top of the wrong state root, producing an invalid `ChunkExtra` and diverging permanently from the canonical chain. The node cannot participate in consensus for shard_id_A.

### Likelihood Explanation

The attacker needs only to be a peer reachable by the syncing node during state sync — no validator, chunk producer, or operator privileges are required. All data needed to construct the malicious header (chunk_B, its Merkle path, shard_id_A's receipt proofs, shard_id_A's prev_chunk_header, shard_id_B's state_root_node) is publicly available on-chain. The syncing node retries on failure but does not re-validate a header already stored on disk, so a single successful injection persists. [11](#0-10) 

### Recommendation

Add an explicit shard-id binding check immediately after `validate_chunk_proofs` in `set_state_header`:
