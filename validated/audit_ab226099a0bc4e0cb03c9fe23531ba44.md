### Title
`MerkleBlob::batch_insert` Skips Duplicate-Key Validation, Silently Corrupting the DataLayer Tree Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-`KeyId` guard that `insert` enforces for all but the last two items in a batch. When a caller supplies a batch containing a repeated `KeyId`, the `BlockStatusCache` silently overwrites its `key_to_index` entry for that key while the raw blob retains both leaf nodes. The resulting tree root hash is computed over the ghost leaf, making it inconsistent with the cache-visible state. Any proof of inclusion derived from this root is therefore based on a corrupted committed state.

---

### Finding Description

`MerkleBlob::insert` guards against duplicate keys and hashes before writing anything:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `insert` (with its guards) only for the last two items popped from the vector when the tree starts with ≤ 1 leaves. Every other item in the batch is written directly through `insert_entry_to_blob` with **no duplicate check**:

```rust
// lines 587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no guard
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which performs an unconditional `HashMap::insert` — silently overwriting the previous index for a repeated key:

```rust
// lines 188-193
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // silent overwrite
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
``` [3](#0-2) 

After the batch completes:
- The raw blob contains **two** leaf nodes with the same `KeyId` at different `TreeIndex` positions.
- `block_status_cache.key_to_index` maps that key to only the **last** inserted index; the earlier leaf is a ghost.
- `calculate_lazy_hashes` traverses the full blob tree (including the ghost leaf) and computes the root hash over both leaves.
- The root hash therefore does not match the key-value state visible through the cache.

`check_just_integrity` would detect this (`leaf_count != key_to_index_cache_length`, or `IntegrityKeyToIndexCacheIndex` for the ghost leaf), but `check_integrity` is only enabled in test builds (`check_integrity_on_drop: cfg!(test)`): [4](#0-3) 

The Python binding `py_batch_insert` passes caller-supplied `(KeyId, ValueId)` pairs directly into `batch_insert` with no deduplication: [5](#0-4) 

The Python stub confirms `batch_insert` is part of the public `MerkleBlob` API: [6](#0-5) 

---

### Impact Explanation

The corrupted root hash is the committed state anchor for DataLayer Merkle proofs. After a batch with a duplicate `KeyId`:

1. `get_proof_of_inclusion` for the ghost key's index will traverse a path that does not correspond to any cache-visible key, producing a proof that verifies against the corrupted root but proves inclusion of a state that was never legitimately committed.
2. `get_proof_of_inclusion` for the surviving key will produce a correct-looking proof, but the root it anchors to also covers the ghost leaf — so the root itself is forged relative to the intended key-value set.
3. Any downstream consumer that trusts the root hash (e.g., a DataLayer sync peer) will accept proofs for key-value pairs that were never validly inserted, or reject valid proofs because the root they hold differs from the corrupted one.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

---

### Likelihood Explanation

`batch_insert` is the primary bulk-insertion path for DataLayer and is called directly from Python with caller-supplied `KeyId` values. `KeyId` is a plain `i64` — any caller (including one processing externally-sourced data) can supply a list containing a repeated key. No privilege is required; the Python binding imposes no deduplication. The only existing test for duplicate rejection (`test_double_insert_fails`) exercises `insert`, not `batch_insert`. [7](#0-6) 

---

### Recommendation

Add a duplicate-key and duplicate-hash check at the top of `batch_insert` (or inside the fast-path loop) before calling `insert_entry_to_blob`, mirroring the guards already present in `insert`:

```rust
// inside the fast-path loop in batch_insert
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(&hash) {
    return Err(Error::HashAlreadyPresent());
}
```

Alternatively, pre-deduplicate the input vector before processing, or collect all keys into a `HashSet` and reject the batch if its size differs from the input length.

---

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn sha256_of(n: i64) -> Hash {
    use chia_sha2::Sha256;
    let mut h = Sha256::new();
    h.update(&n.to_be_bytes());
    Hash(Bytes32::new(h.finalize()))
}

fn main() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();

    // Pre-populate with 2 leaves so batch_insert skips the guarded path entirely
    blob.insert(KeyId(0), ValueId(0), &sha256_of(0), InsertLocation::Auto {}).unwrap();
    blob.insert(KeyId(1), ValueId(1), &sha256_of(1), InsertLocation::Auto {}).unwrap();

    // Batch with a duplicate KeyId(2) — both entries bypass the duplicate guard
    let batch = vec![
        ((KeyId(2), ValueId(100)), sha256_of(100)),
        ((KeyId(2), ValueId(200)), sha256_of(200)), // duplicate key
        ((KeyId(3), ValueId(3)),   sha256_of(3)),
    ];
    blob.batch_insert(batch).unwrap(); // succeeds — no error raised

    blob.calculate_lazy_hashes().unwrap();

    // The blob now contains two leaf nodes for KeyId(2).
    // The cache points only to the last-inserted one (ValueId(200)).
    // The root hash covers both leaves — it is corrupted relative to
    // the cache-visible {0,1,2→200,3} state.
    //
    // In a test build, blob.check_integrity() would return
    // Err(IntegrityKeyToIndexCacheLength(...)) or
    // Err(IntegrityKeyToIndexCacheIndex(...)) here.
    println!("root: {:?}", blob.get_root_hash());
}
```

The `batch_insert` call succeeds without error despite the duplicate `KeyId(2)`, and the resulting root hash is computed over a tree that contains a ghost leaf — corrupting the committed DataLayer state.
