### Title
Missing Class Hash Validation in `execute_replace_class` Enables Permanent Freezing of Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the caller-supplied class hash corresponds to a declared contract class. An explicit `TODO` comment in the code acknowledges this missing check. Because the OS blindly writes any arbitrary felt value as the contract's new class hash, a contract deployer can permanently render any contract they control uncallable — freezing all funds held within it — by supplying an undeclared class hash (e.g., `0`).

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads the new class hash directly from the syscall request and writes it into `contract_state_changes` without any check that the hash corresponds to a previously declared class:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The `TODO` comment at line 898 explicitly acknowledges the missing validation. [2](#0-1) 

When a future transaction attempts to call a contract whose class hash was set to an undeclared value, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // undeclared hash → returns 0
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=cast(compiled_class_facts_bundle.compiled_class_facts, felt*),
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,           // key=0, not found → assertion failure
);
``` [3](#0-2) 

`dict_read` returns `0` for an unknown key. `find_element` then searches for a compiled class with hash `0`; if none exists, it raises a Cairo assertion failure, halting the OS. The sequencer must therefore permanently exclude all transactions targeting that contract, making its funds irrecoverable.

The `replace_class` syscall is dispatched from `execute_syscalls` using the currently executing contract's address, so the attacker can only replace their own contract's class — but that is sufficient to freeze all user funds deposited into it: [4](#0-3) 

---

### Impact Explanation

**Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value:
- The OS cannot resolve the compiled class for any future call to that contract.
- The sequencer is forced to permanently exclude all transactions targeting the contract.
- There is no recovery path: `replace_class` is itself a syscall that requires the contract to be callable, so the contract cannot self-repair.
- All ERC-20 balances, ETH, or other assets stored in the contract's storage are permanently inaccessible.

---

### Likelihood Explanation

**Medium–High.**

- Any contract deployer can trigger this with a single transaction: deploy a contract, attract user deposits (e.g., as a vault, staking contract, or token), then call `replace_class(0)`.
- No privileged role, leaked key, or external dependency is required beyond controlling the contract.
- The missing check is explicitly acknowledged in the source code (`TODO` comment), confirming it is a known gap in the current implementation.
- The attack is cheap (one syscall) and irreversible.

---

### Recommendation

Before writing the new class hash to `contract_state_changes`, verify that it exists as a key in `contract_class_changes` (i.e., it has been declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` with the requested class hash and assert the returned compiled class hash is non-zero:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("replace_class: class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the validation already implicitly performed in `execute_entry_point` and closes the gap identified by the `TODO`.

---

### Proof of Concept

1. **Attacker deploys** `MaliciousVault` — a contract that accepts user deposits and exposes a standard ERC-20-like interface. Users deposit funds.
2. **Attacker sends** an invoke transaction calling `replace_class(0)` (or any undeclared felt) from within `MaliciousVault`.
3. **OS processes** the `REPLACE_CLASS` syscall via `execute_replace_class`. No validation is performed; `contract_state_changes` is updated: `MaliciousVault.class_hash ← 0`.
4. **State is committed** to the block. The block proof succeeds because the current block does not call `MaliciousVault` after the replacement.
5. **In any subsequent block**, any transaction targeting `MaliciousVault` causes `execute_entry_point` to call `dict_read(key=0)` → `compiled_class_hash=0`, then `find_element(..., key=0)` → assertion failure → OS halt. The sequencer must permanently blacklist `MaliciousVault`.
6. **All user funds** in `MaliciousVault`'s storage are permanently frozen with no recovery mechanism.
