### Title
`run_proof_verification` TOCTOU skips cryptographic proof check when `proof_facts` hash is pre-populated, allowing invalid proofs to be admitted — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

`run_proof_verification` uses `contains_proof(proof_facts)` as a gate to skip the expensive `verify_proof` call. Because `proof_facts` are block-level values (not transaction-specific), any user who previously submitted a valid proof for the same block can cause all subsequent transactions with the same `proof_facts` to bypass cryptographic proof verification entirely. The gateway then admits those transactions — carrying arbitrary, unverified proof bytes — into the mempool.

### Finding Description

`run_proof_verification` implements a check-then-act pattern:

```
contains_proof(proof_facts)  →  if true, return Ok(false) [skip]
                              →  if false, verify_proof(proof_facts, proof)
``` [1](#0-0) 

The key is that `proof_facts` are block-level fields — `proof_version`, `program_hash`, `block_number`, `block_hash`, `config_hash` — with no per-transaction component: [2](#0-1) 

So the same `proof_facts` value is valid for any number of distinct Invoke V3 transactions referencing the same block. Once a single valid proof is stored under `hash(proof_facts)`, every subsequent call to `run_proof_verification` with the same `proof_facts` — regardless of what `proof` bytes are supplied — returns `Ok(false)` (skipped) without calling `verify_proof`.

In the **gateway flow**, `spawn_proof_verification` wraps this in a task: [3](#0-2) 

`await_verification_task_and_extract_proof_data` treats `Ok(())` from the task as success and returns `Some((proof_facts, proof))` — the original, unverified proof bytes — to `store_proof_and_spawn_archiving`: [4](#0-3) 

`store_proof_and_spawn_archiving` then calls `store_proof_in_proof_manager(proof_facts, proof)`: [5](#0-4) 

`ProofManager::set_proof` also short-circuits on `contains_proof`, so the invalid proof bytes are silently discarded — but the transaction has already passed all gateway checks and proceeds to `mempool_client.add_tx`: [6](#0-5) 

The blockifier's `validate_proof_facts` validates only the `proof_facts` fields (block hash, program hash, config hash) — it never touches the `proof` bytes: [7](#0-6) 

So the transaction is executed normally despite carrying an invalid proof.

In the **consensus flow**, `spawn_verify_and_store_proof` has the same skip path: [8](#0-7) 

### Impact Explanation

**High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

An attacker can submit an unbounded number of Invoke V3 transactions with arbitrary (invalid) proof bytes, all of which are admitted to the mempool and executed on-chain, as long as one valid proof for the target block was previously stored. The cryptographic guarantee of the client-side proving feature — that every accepted transaction carries a verifiable proof of correct Virtual OS execution — is completely bypassed.

### Likelihood Explanation

The attacker must first obtain one valid proof for a target block (computationally expensive, but a one-time cost). After that, the attack is free and repeatable for the entire window during which that block's `proof_facts` remain valid (governed by `validate_proof_block_number`'s range check). Any user with access to the gateway HTTP endpoint can trigger this.

### Recommendation

Move the `contains_proof` short-circuit out of `run_proof_verification` and into the **store** path only. Verification must always run against the supplied proof bytes, regardless of whether a proof for the same `proof_facts` was previously stored:

```rust
async fn run_proof_verification(
    proof_facts: ProofFacts,
    proof: Proof,
    proof_manager_client: SharedProofManagerClient,
) -> Result<bool, TransactionConverterError> {
    // Always verify — do NOT skip based on contains_proof.
    tokio::task::spawn_blocking(move || {
        starknet_proof_verifier::verify_proof(proof_facts, proof)
    })
    .await
    .expect("proof verification task panicked")?;
    Ok(true)
}
```

The `contains_proof` check should only guard the subsequent `set_proof` call (to avoid redundant filesystem writes), which `ProofManager::set_proof` already does internally.

### Proof of Concept

1. Attacker submits **Tx1**: `proof_facts = X` (valid block hash/number/program hash), `proof = P1` (valid cryptographic proof).
2. Gateway calls `run_proof_verification(X, P1, pmc)`:
   - `contains_proof(X)` → `false`
   - `verify_proof(X, P1)` → `Ok(())` (valid)
   - Returns `Ok(true)`
3. Gateway stores `P1` under `hash(X)` in the proof manager.
4. Attacker submits **Tx2**: `proof_facts = X` (same), `proof = P2` (random garbage bytes).
5. Gateway calls `run_proof_verification(X, P2, pmc)`:
   - `contains_proof(X)` → `true` (P1 was stored in step 3)
   - Returns `Ok(false)` — **`verify_proof` is never called**
6. `await_verification_task_and_extract_proof_data` returns `Ok(Some((X, P2)))`.
7. `store_proof_and_spawn_archiving` calls `set_proof(X, P2)` → silently no-ops (already stored).
8. `mempool_client.add_tx(Tx2)` is called — **Tx2 with invalid proof P2 is admitted**.
9. Blockifier executes Tx2: `validate_proof_facts` checks only `X`'s fields (valid) — proof bytes are never inspected.
10. Tx2 is included in a block with an unverified proof.

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        if !os_constants.allowed_proof_versions.contains(&snos_proof_facts.proof_version.as_felt())
        {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof version {} is not allowed under this protocol version.",
                snos_proof_facts.proof_version
            )));
        }

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let virtual_os_config_hash = block_context.virtual_os_config_hash();
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
