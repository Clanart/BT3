### Title
Unauthenticated `add_events` on `RemoteL1ProviderServer` Allows Injection of Fabricated L1 Events into Block Proposals — (File: `crates/apollo_l1_provider/src/l1_provider.rs`, `crates/apollo_l1_provider/src/communication.rs`)

---

### Summary

`L1Provider::add_events` accepts arbitrary `Event` payloads with no caller authentication. In distributed and hybrid deployments the `RemoteL1ProviderServer` binds this endpoint to `0.0.0.0` with no token, TLS-client-cert, or any other access-control layer. Any network peer that can reach the L1 provider's HTTP/2 port can inject fabricated `Event::L1HandlerTransaction` records (causing fake L1 handler transactions to be sequenced and executed, corrupting state roots and block hashes) or fabricated `Event::TransactionConsumed` / `Event::TransactionCanceled` records (suppressing legitimate L1 handler transactions so they are never sequenced).

---

### Finding Description

**Root cause — no caller identity check in `add_events`**

`L1Provider::add_events` performs exactly one guard: it rejects calls made before `initialize` has been called (`ProviderState::Uninitialized`). There is no check on *who* is calling. [1](#0-0) 

Every event variant is then applied unconditionally to the internal `TransactionManager`: [2](#0-1) 

**Root cause — `RemoteL1ProviderServer` has no authentication layer**

`ComponentRequestHandler` for `L1Provider` dispatches `L1ProviderRequest::AddEvents` directly to `add_events` with no identity check: [3](#0-2) 

`RemoteComponentServer::remote_component_server_handler` deserializes the HTTP/2 body and forwards it to the local client. There is no bearer token, mTLS, or IP allowlist check anywhere in the handler: [4](#0-3) 

**Root cause — production deployments bind to `0.0.0.0`**

Both the `distributed` and `hybrid` L1-service deployment configs set `bind_ip: "0.0.0.0"` for the L1 provider remote server, making it reachable by any pod in the cluster (or beyond, depending on network policy): [5](#0-4) 

The Kubernetes service exposes port 55004 as a named `l1-provider` endpoint: [6](#0-5) 

**Attack vector 1 — inject fake `Event::L1HandlerTransaction`**

An attacker crafts an `L1ProviderRequest::AddEvents` payload containing an `Event::L1HandlerTransaction` with an arbitrary `L1HandlerTransaction` (any contract address, any calldata, `paid_fee_on_l1 > 0`). After `add_events` stores it via `tx_manager.add_tx`, the batcher's next `get_txs` call returns it as a proposable transaction: [7](#0-6) 

The blockifier executes it, producing a state diff that feeds into `calculate_block_commitments`, yielding a wrong global state root and wrong block hash. The only execution-time guard is `paid_fee == Fee(0)`: [8](#0-7) 

This is trivially bypassed by setting `paid_fee_on_l1` to any non-zero value in the crafted event.

**Attack vector 2 — suppress real L1 handler transactions**

An attacker sends `Event::TransactionConsumed { tx_hash, timestamp }` for a real pending L1 handler transaction. `consume_tx` moves it to the consumed set, from which it is never proposed: [9](#0-8) 

Similarly, `Event::TransactionCanceled` calls `finalize_cancellation`, permanently removing the transaction from the provider's records: [10](#0-9) 

The `L1ProviderClient` trait exposes `add_events` as a first-class method callable by any holder of a `RemoteComponentClient`: [11](#0-10) 

---

### Impact Explanation

**Critical — wrong L1 message / wrong state from blockifier execution:**
A fabricated `Event::L1HandlerTransaction` causes the blockifier to execute a transaction that was never posted on Ethereum L1. The resulting state diff is committed to storage, the Patricia trie root is updated, and the block hash is computed over the corrupted global root. Every downstream consumer (state sync, proof inputs, RPC) sees a wrong authoritative state.

**High — valid L1 handler transactions rejected before sequencing:**
A fabricated `Event::TransactionConsumed` or `Event::TransactionCanceled` for a real pending L1 handler transaction permanently removes it from the provider's queue. The batcher never proposes it; the L1 message is silently dropped.

---

### Likelihood Explanation

The L1 provider remote server is deployed with `bind_ip: "0.0.0.0"` and no application-level authentication. Any workload co-located in the same Kubernetes cluster (or any host with network access to the service port) can send a well-formed HTTP/2 POST with a serialized `L1ProviderRequest::AddEvents` payload. No credentials, signatures, or privileged OS capabilities are required. The `L1ProviderRequest` enum is `Serialize + Deserialize`, so the wire format is straightforward to construct.

---

### Recommendation

1. **Restrict `AddEvents` and `Initialize` to the L1 scraper only.** Introduce a caller-identity mechanism (e.g., a shared secret header, mTLS client certificate, or a separate internal-only listener) so that only the authenticated `L1Scraper` component can call `add_events` and `initialize`.

2. **Do not bind the L1 provider remote server to `0.0.0.0` without a network policy.** Either restrict `bind_ip` to the loopback or pod-local address for scraper-facing endpoints, or enforce a Kubernetes `NetworkPolicy` that allows connections to port 55004 only from the L1 scraper pod.

3. **Validate event provenance inside `add_events`.** At minimum, verify that the `L1HandlerTransaction` embedded in an `Event::L1HandlerTransaction` has a transaction hash consistent with the Starknet chain ID and the event's calldata, so that a fabricated transaction with an arbitrary hash cannot be injected.

---

### Proof of Concept

```
# 1. Serialize a fabricated L1ProviderRequest::AddEvents payload
#    containing an Event::L1HandlerTransaction with:
#      - arbitrary contract_address / entry_point_selector / calldata
#      - paid_fee_on_l1 = 1  (bypasses the only execution-time fee guard)
#      - block_timestamp / scrape_timestamp = any value

# 2. POST it to the L1 provider remote server (no auth required):
curl -s --http2-prior-knowledge \
     -H "Content-Type: application/octet-stream" \
     -H "x-request-id: 1" \
     --data-binary @fabricated_add_events.bin \
     http://<l1-provider-service>:55004/

# 3. The provider returns HTTP 200 with AddEvents(Ok(())).
#    The fake L1HandlerTransaction is now in the TransactionManager queue.

# 4. On the next batcher get_txs call, the fake transaction is returned
#    and executed by the blockifier, producing a wrong state root and
#    wrong block hash committed to storage.
```

The `RemoteComponentServer` handler returns `StatusCode::OK` for any well-formed `L1ProviderRequest`, with no authentication step between TCP accept and request dispatch. [12](#0-11)

    #[instrument(skip_all, fields(request_id = %request_id, remote_addr = %client_peer))]
    async fn remote_component_server_handler(
        http_request: HyperRequest<Incoming>,
        request_id: RequestId,
        client_peer: SocketAddr,
        local_client: LocalComponentClient<Request, Response>,
        metrics: &'static RemoteServerMetrics,
    ) -> Result<HyperResponse<Full<Bytes>>, hyper::Error> {
        trace!("Received HTTP request: {http_request:?}");
        let body_bytes = http_request.into_body().collect().await?.to_bytes();
        trace!("Extracted {} bytes from HTTP request body", body_bytes.len());

        metrics.increment_total_received();

        let http_response = match SerdeWrapper::<Request>::wrapper_deserialize(&body_bytes)
            .map_err(|err| ClientError::ResponseDeserializationFailure(err.to_string()))
        {
            Ok(request) => {
                trace!(
                    remote_addr = %client_peer,
                    request_id = %request_id,
                    request_type = request.request_label(),
                    "remote component request",
                );
                trace!("Successfully deserialized request: {request:?}");
                metrics.increment_valid_received();

                // Wrap the send operation in a tokio::spawn as it is NOT a cancel-safe operation.
                // Even if the current task is cancelled, the inner task will continue to run.
                // Note: this creates a new request ID for the local client.
                let response = tokio::spawn(async move { local_client.send(request).await })
                    .await
                    .expect("Should be able to extract value from the task");

                metrics.increment_processed();

                match response {
                    Ok(response) => {
                        trace!("Local client processed request successfully: {response:?}");
                        HyperResponse::builder()
                            .status(StatusCode::OK)
                            .header(CONTENT_TYPE, APPLICATION_OCTET_STREAM)
                            .body(Full::new(Bytes::from(
                                SerdeWrapper::new(response)
                                    .wrapper_serialize()
                                    .expect("Response serialization should succeed"),
                            )))
                    }
                    Err(error) => {
                        panic!(
                            "Remote server failed sending with its local client. Error: {error:?}"
                        );
                    }
                }
            }
            Err(error) => {
                error!("Failed to deserialize request: {error:?}");
                let server_error = ServerError::RequestDeserializationFailure(error.to_string());
                HyperResponse::builder().status(StatusCode::BAD_REQUEST).body(Full::new(
                    Bytes::from(
                        SerdeWrapper::new(server_error)
                            .wrapper_serialize()
                            .expect("Server error serialization should succeed"),
                    ),
                ))
            }
        }
        .expect("Response building should succeed");
        trace!("Built HTTP response: {http_response:?}");

        Ok(http_response)
    }
