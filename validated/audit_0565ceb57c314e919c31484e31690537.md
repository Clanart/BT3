### Title
Deploy Syscall Unconditionally Reports Success Regardless of Constructor Revert - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` function in the StarkNet OS always writes `failure_flag=0` (success) in the deploy syscall response, regardless of whether the deployed contract's constructor reverted. A TODO comment in the code explicitly acknowledges this: `// TODO(Yoni, 1/1/2026): support failures.` This is the direct analog of the Chainlink unhandled-revert pattern: a critical call's failure is silently swallowed and success is unconditionally reported to the caller.

---

### Finding Description

In `execute_deploy`, after calling `deploy_contract` (which invokes the constructor entry point), the OS writes the response header with a hardcoded `failure_flag=0`:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

Compare this with the correct pattern used in `contract_call_helper`, which properly propagates the revert status:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

The `deploy_contract` call does not return an `is_reverted` flag at all — it only returns `(retdata_size, retdata)` — so there is no mechanism to propagate constructor failure to the response: [3](#0-2) 

The blockifier (the sequencer's off-chain execution engine) is a separate implementation that correctly handles constructor reverts and reports `failure_flag=1` to the calling contract when a constructor fails. The OS Cairo program, however, always reports `failure_flag=0`. This creates a provable divergence between the sequencer's execution trace and the OS-proven execution trace for any block that contains a `deploy` syscall whose constructor reverts.

---

### Impact Explanation

**Impact: High — Network not being able to confirm new transactions (total network halt).**

When the sequencer includes a transaction containing a `deploy` syscall with a reverting constructor:

1. The blockifier correctly writes `failure_flag=1` in the deploy response. The calling contract's subsequent execution branches on this failure.
2. The OS Cairo program re-executes the same block to generate a proof, but unconditionally writes `failure_flag=0`. The calling contract's execution therefore takes a different branch.
3. The two executions produce different state roots.
4. The OS proof cannot be reconciled with the sequencer's committed state — proof generation fails.
5. The block cannot be confirmed on L1. No further blocks can be proven until the issue is resolved.

Because the OS is the authoritative source for the proof, and the proof is what L1 verifies, a persistent mismatch between the OS and the blockifier for any class of transactions causes a permanent inability to finalize blocks containing that transaction type.

---

### Likelihood Explanation

**Likelihood: High.**

- Any unprivileged user can submit an invoke transaction that calls the `deploy` syscall with a contract whose constructor reverts. Constructor reverts are a normal, expected outcome (e.g., invalid constructor arguments, failed preconditions, out-of-gas in constructor).
- No privileged access, leaked keys, or external dependency compromise is required.
- The attacker does not need to know about this bug — a legitimate user whose constructor accidentally reverts triggers the same condition.
- The sequencer has no reason to exclude such transactions; a reverting constructor is a valid transaction outcome that should be handled gracefully.

---

### Recommendation

The `execute_deploy` function must be refactored to mirror `contract_call_helper`. Specifically:

1. `deploy_contract` must be updated to return an `is_reverted` flag alongside `(retdata_size, retdata)`.
2. The response header must be written as `ResponseHeader(gas=remaining_gas, failure_flag=is_reverted)` instead of the hardcoded `failure_flag=0`.
3. When `is_reverted != 0`, the OS must append `ERROR_ENTRY_POINT_FAILED` to the retdata (consistent with `contract_call_helper`) and ensure state changes from the constructor are rolled back via the revert log. [4](#0-3) 

---

### Proof of Concept

1. Deploy a contract `Caller` that calls the `deploy` syscall targeting a contract `Bomb` whose constructor always reverts (e.g., `assert 1 = 0`).
2. Submit an invoke transaction calling `Caller.__execute__`.
3. The sequencer's blockifier executes the block: `Bomb`'s constructor reverts → blockifier writes `failure_flag=1` in the deploy response → `Caller` sees failure and takes the failure branch → state root R₁ is committed.
4. The OS Cairo program re-executes the same block: `Bomb`'s constructor reverts → OS writes `failure_flag=0` → `Caller` sees success and takes the success branch → OS computes state root R₂ ≠ R₁.
5. The OS proof attests to R₂ but the sequencer committed R₁. Proof verification on L1 fails. The block cannot be finalized. All subsequent blocks are also blocked. [5](#0-4)
