### Title
`set_state_header` stores a state-sync header without binding the chunk's `shard_id` to the caller-supplied shard — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `shard_id` argument and a `ShardStateSyncResponseHeader` that embeds a `ShardChunk`. The function validates chunk proofs, Merkle inclusion in the block, incoming receipts, and the state-root node, but **never checks that `chunk.shard_id() == shard_id`**. A malicious peer can supply a fully valid header for shard B when shard A is requested. Every existing guard passes, the header is persisted under `StateHeaderKey(shard_A, sync_hash)` carrying shard B's `prev_state_root`, and all subsequent `set_state_part` / `apply_state_part` calls write shard B's trie data into shard A's storage.

---

### Finding Description

`set_state_header` performs five checks before committing the header:

1. `validate_chunk_proofs` — internal chunk consistency (tx root, receipts root, etc.)
2. `verify_path(sync_prev_block_header.chunk_headers_root(), chunk_proof, ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` — Merkle inclusion in the block
3. `verify_path` for `prev_chunk_header`
4. Incoming-receipts chain walk
5. `validate_state_root_node` against `chunk_inner.prev_state_root()` [1](#0-0) 

None of these checks bind the chunk's own `shard_id` field to the `shard_id` parameter. Check 2 only proves that the chunk hash appears *somewhere* in the block's chunk-headers Merkle tree; it does not verify that the chunk occupies the slot corresponding to the requested shard. A malicious peer can therefore:

- Provide shard B's chunk (with `chunk.shard_id() == B`) together with shard B's valid Merkle proof.
- `verify_path(root, proof_for_B, ChunkHashHeight(B_chunk_hash, B_height))` returns `true` because shard B's chunk genuinely is in the tree.
- The function stores the header under `StateHeaderKey(shard_A, sync_hash)`. [2](#0-1) 

`set_state_part` then retrieves this header, extracts `prev_state_root` from shard B's chunk, and validates incoming parts against shard B's trie root: [3](#0-2) 

`apply_state_part` writes the validated trie nodes into shard A's `ShardUId` slot (derived from the caller-supplied `shard_id`): [4](#0-3) 

The result is shard B's trie data committed to shard A's storage column.

The analogous invariant *is* enforced in `compute_state_response_header` (the server-side path), which resolves the chunk by looking up `prev_shard_layout.get_shard_index(shard_id)` and reading `prev_block.chunks().get(shard_index)`, so the server always returns the correct shard's chunk. The client-side acceptance path (`set_state_header`) has no equivalent guard. [5](#0-4) 

---

### Impact Explanation

A syncing node that accepts a crafted header will:

1. Persist shard B's state root under shard A's key.
2. Download and validate state parts against shard B's root (all pass cryptographically).
3. Write shard B's trie into shard A's `ShardUId` column via `apply_state_part`.
4. Call `set_state_finalize` / `chain_update.set_state_finalize`, which applies shard B's chunk in shard A's execution context, producing a post-state root that does not match the canonical chain. [6](#0-5) 

For a validator node catching up to a new shard, this causes the node to hold incorrect account balances, contract state, and flat-storage entries for shard A. The node will produce invalid chunks that are rejected by other validators, causing missed block/chunk production and forfeited staking rewards. Recovery requires a full re-sync.

---

### Likelihood Explanation

State sync is triggered routinely: whenever a node falls more than two epochs behind, or whenever a validator is assigned to a new shard it does not currently track. During state sync the node downloads headers from any reachable peer. No privilege is required to respond to `StateRequestHeader` messages — any connected peer can serve a crafted response. The attack requires only that the adversary be a peer of the syncing node at the moment the header is fetched.

---

### Recommendation

Add an explicit shard-binding check at the top of `set_state_header`, immediately after extracting the chunk:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: enforce the invariant that the header's chunk belongs to the requested shard.
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

This mirrors the pattern already used in `compute_state_response_header` (server side) and in `validate_block_impl`, which checks `chunk_header.shard_id() != shard_id` for every chunk in a block. [7](#0-6) 

---

### Proof of Concept

Assume a 2-shard network. Syncing node N requests the state header for shard 0 from malicious peer M.

1. M constructs a `ShardStateSyncResponseHeaderV2` where:
   - `chunk` = the genuine shard-1 chunk from the sync block's predecessor (valid signatures, valid internal proofs).
   - `chunk_proof` = the genuine Merkle path proving shard-1's chunk hash is in `sync_prev_block.chunk_headers_root`.
   - `state_root_node` = the genuine state-root node for shard 1's trie.
   - `incoming_receipts_proofs` / `root_proofs` = genuine receipts for shard 1.

2. N calls `set_state_header(shard_0, sync_hash, crafted_header)`.

3. All five checks pass:
   - `validate_chunk_proofs` passes (shard-1 chunk is internally valid).
   - `verify_path(root, shard1_proof, ChunkHashHeight(shard1_hash, shard1_height))` returns `true`.
   - Prev-chunk proof passes (genuine shard-1 prev chunk).
   - Receipts walk passes (genuine shard-1 receipts).
   - `validate_state_root_node` passes (genuine shard-1 state root node).

4. Header stored under `StateHeaderKey(shard_0, sync_hash)` with shard-1's `prev_state_root`.

5. N downloads state parts validated against shard-1's root; all pass `validate_state_part`.

6. `apply_state_part(shard_0, shard1_state_root, ...)` writes shard-1's trie nodes into shard-0's `ShardUId` column.

7. N's shard-0 trie now contains shard-1's account state. N produces invalid shard-0 chunks and is unable to participate in consensus for shard 0.

    fn apply_state_part(
        &self,
        shard_id: ShardId,
        state_root: &StateRoot,
        part_id: PartId,
        part: &StatePart,
        epoch_id: &EpochId,
    ) -> Result<(), Error> {
        let _timer = metrics::STATE_SYNC_APPLY_PART_DELAY
            .with_label_values(&[&shard_id.to_string()])
            .start_timer();

        let part = part
            .to_partial_state()
            .expect("Part was already validated earlier, so could never fail here");
        let ApplyStatePartResult { trie_changes, flat_state_delta, contract_codes } =
            Trie::apply_state_part(state_root, part_id, part);

        let shard_uid = self.get_shard_uid_from_epoch_id(shard_id, epoch_id)?;

        flat_state_delta.apply_to_flat_state(&mut store_update.flat_store_update(), shard_uid);
        self.precompile_contracts(epoch_id, contract_codes)?;
        store_update.commit();
