### Title
Unchecked Detached Burn Promise Allows Unbacked Cross-Chain Token Minting — (File: `near/omni-bridge/src/lib.rs`)

### Summary
In `near/omni-bridge/src/lib.rs`, the `burn_tokens_if_needed` function fires the cross-contract `burn` call with `.detach()`, meaning its result is never checked. `init_transfer_internal` then unconditionally emits `InitTransferEvent` regardless of whether the burn succeeded. If the burn fails silently, relayers observe the event and finalize the transfer on the destination chain, minting or releasing tokens there while the original bridged tokens remain unburned in the NEAR bridge contract — creating unbacked supply across chains.

### Finding Description

`burn_tokens_if_needed` is called inside `init_transfer_internal` for every outbound transfer of a bridge-deployed token:

```rust
// near/omni-bridge/src/lib.rs:1806-1813
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) {
    if self.is_deployed_token(&token) {
        ext_token::ext(token)
            .with_static_gas(BURN_TOKEN_GAS)
            .burn(amount)
            .detach();   // ← result is never observed
    }
}
``` [1](#0-0) 

`.detach()` in NEAR SDK means the promise is scheduled but its success or failure is never awaited or checked by the calling contract. Execution continues as if the burn succeeded.

Immediately after, `init_transfer_internal` emits the `InitTransferEvent` unconditionally:

```rust
// near/omni-bridge/src/lib.rs:1850-1863
if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
    self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
    self.lock_tokens_if_needed(...);
} else { ... }

env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
U128(0)  // ← signals "keep all tokens" (no refund to user)
``` [2](#0-1) 

The function returns `U128(0)`, which in the `ft_transfer_call` callback protocol means "consume all transferred tokens." The tokens are already held by the bridge at this point. If the detached burn fails (e.g., due to insufficient `BURN_TOKEN_GAS`, a reentrancy guard in the token contract, or any other revert condition), the tokens remain in the bridge contract unburned, but the `InitTransferEvent` has already been emitted and is indistinguishable from a successful burn.

### Impact Explanation

Relayers monitor `InitTransferEvent` logs and submit finalization proofs to the destination chain. The destination chain's `finTransfer` (EVM) or equivalent mints or releases tokens to the recipient. If the NEAR-side burn silently failed, the same token supply now exists on both NEAR (held by the bridge, unburned) and the destination chain (freshly minted/released). This is a direct unbacked mint of bridged assets — a Critical impact under the allowed scope. [3](#0-2) 

### Likelihood Explanation

The most realistic trigger is gas exhaustion: `BURN_TOKEN_GAS` is a fixed static allocation. If the bridge token's `burn` function consumes more gas than allocated (e.g., due to a storage-heavy implementation or a future upgrade to the token contract), the cross-contract call fails silently. Because `.detach()` is used, no panic propagates back to `init_transfer_internal`. The entry path is fully unprivileged: any user holding bridge-deployed tokens can call `ft_transfer_call` on the token contract, which routes through `ft_on_transfer` → `init_transfer` → `init_transfer_internal` → `burn_tokens_if_needed`.

### Recommendation

Replace the detached burn with an awaited callback that verifies success before emitting `InitTransferEvent`. The burn result must be checked, and if it fails, the transfer message must be removed and the tokens refunded:

```rust
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) -> Option<Promise> {
    if self.is_deployed_token(&token) {
        Some(
            ext_token::ext(token)
                .with_static_gas(BURN_TOKEN_GAS)
                .burn(amount)
        )
    } else {
        None
    }
}
```

Then chain a `.then(Self::ext(...).burn_callback(...))` that checks `env::promise_result` and only emits `InitTransferEvent` on success, reverting (refunding) on failure. This mirrors the fix applied in the referenced Amp.sol report: validate the critical operation's success before proceeding.

### Proof of Concept

1. User holds 1000 units of a bridge-deployed NEAR token (originally bridged from EVM).
2. User calls `ft_transfer_call` on the token contract, transferring 1000 tokens to the bridge with an `InitTransferMsg` targeting an EVM recipient.
3. Bridge's `ft_on_transfer` → `init_transfer` → `init_transfer_internal` is called.
4. `burn_tokens_if_needed` schedules a detached `burn(1000)` with `BURN_TOKEN_GAS`.
5. The burn cross-contract call runs out of gas (or reverts for any reason) — silently, because `.detach()` was used.
6. `init_transfer_internal` returns `U128(0)` and emits `InitTransferEvent { amount: 1000, recipient: <EVM address> }`.
7. A relayer picks up the event and calls `finTransfer` on the EVM bridge with a valid MPC signature.
8. EVM bridge mints 1000 tokens to the EVM recipient.
9. The 1000 NEAR tokens remain in the bridge contract, unburned.
10. Total supply is now inflated by 1000 tokens — the attacker holds 1000 EVM tokens while the NEAR bridge holds 1000 unburned NEAR tokens, breaking collateralization. [1](#0-0) [2](#0-1)
