### Title
Fee-on-Transfer / Rebasing Token Accounting Corruption in `initTransfer` Breaks Bridge Collateralization — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol`'s `initTransfer` function accepts a user-supplied `amount` parameter, transfers that amount from the caller, and immediately emits it in the `InitTransfer` event — without performing a before/after balance snapshot to verify what was *actually* received. For fee-on-transfer (FoT) tokens or rebasing tokens, the recorded `amount` diverges from the actual balance held, causing the NEAR bridge to sign and finalize transfers for more tokens than the EVM vault actually holds, breaking collateralization.

---

### Finding Description

In `initTransfer`, when the token is a plain ERC20 (not a bridge token or custom minter), the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
``` [1](#0-0) 

Immediately after, the unverified `amount` is emitted in the event that drives the entire cross-chain settlement:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // <-- user-supplied, not actual received balance
    ...
);
``` [2](#0-1) 

A relayer submits this event as proof to the NEAR `fin_transfer_callback`, which constructs a `TransferMessage` using the emitted `amount`: [3](#0-2) 

`sign_transfer` then signs a `TransferMessagePayload` carrying that same `amount`, and the destination chain's `finTransfer` releases it. On the EVM side, `finTransfer` calls:

```solidity
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount
);
``` [4](#0-3) 

There is no mechanism anywhere in this path to reconcile the signed `amount` against the vault's actual token balance.

---

### Impact Explanation

**EVM → NEAR direction (FoT token):**
- EVM vault receives `amount − transfer_fee` but records and signs `amount`.
- NEAR mints `amount` tokens backed by only `amount − transfer_fee` on EVM.
- Every such deposit inflates unbacked supply on NEAR — a direct collateralization break.

**NEAR → EVM direction (FoT token, or rebasing token that rebased down):**
- NEAR burns `amount` tokens.
- EVM `finTransfer` attempts `safeTransfer(recipient, amount)` but the vault holds less than the sum of all pending signed amounts.
- The transfer reverts, permanently locking the user's funds, or drains the residual balance of other depositors.

**Rebasing token (e.g., AMPL-style):**
- User deposits 1000 tokens when `_gonsPerFragment = 1`. Vault holds 1000. Event records 1000.
- Token rebases down (`_gonsPerFragment = 2`). Vault now holds 500 in real terms.
- NEAR signs and finalizes a release of 1000 tokens.
- `finTransfer` on EVM drains 1000 tokens from the vault, stealing 500 tokens that belonged to other depositors.

Both sub-cases map to **Critical** allowed impacts: direct theft / unbacked mint, and permanent fund lock.

---

### Likelihood Explanation

The bridge is permissionless for token registration (`logMetadata` → `deployToken` requires only a valid MPC signature, which any relayer can obtain for any ERC20). Any FoT token (e.g., tokens with configurable transfer taxes, deflationary tokens) or rebasing token that gets registered and bridged triggers this path. No privileged access is required — a normal bridge user initiating a transfer with such a token is sufficient. [5](#0-4) 

---

### Recommendation

Replace the direct `safeTransferFrom` call with a before/after balance snapshot, and use the *actual received amount* for the emitted event and all downstream accounting:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived instead of amount going forward
```

This is the same pattern already used in THORChain's `safeTransferFrom` helper (referenced in the external report) and is the standard mitigation for both FoT and rebasing token accounting bugs.

---

### Proof of Concept

1. Deploy a FoT ERC20 that deducts 10% on every transfer.
2. Register it with the bridge via `logMetadata` + `deployToken`.
3. Call `initTransfer(tokenAddress, 1000, 0, 0, nearRecipient, "")`. The vault receives 900 tokens; the event emits `amount = 1000`.
4. Relayer submits the proof to NEAR. NEAR `fin_transfer_callback` constructs a `TransferMessage` with `amount = 1000` and `sign_transfer` signs it.
5. On the NEAR side, 1000 tokens are minted to the recipient — 100 tokens are unbacked.
6. Repeat with many users. Each deposit inflates unbacked supply by 10%.
7. When users bridge back from NEAR to EVM, `finTransfer` on EVM will eventually fail with insufficient balance, permanently locking the last users' funds. [6](#0-5) [7](#0-6)

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }
