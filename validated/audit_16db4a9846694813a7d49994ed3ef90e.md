### Title
Unvalidated Immediate Class Hash Replacement in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall in the StarkNet OS accepts any caller-supplied class hash and immediately commits it to state without verifying the hash corresponds to a declared contract class. This is the direct StarkNet analog of the zAuction `SetZauction` bug: an upgrade reference is updated atomically with no existence check, leaving the contract in a permanently broken state. Any contract — including an account contract holding user funds — can be irreversibly bricked by a single syscall, permanently freezing all funds it holds.

---

### Finding Description

`execute_replace_class` in `syscall_impls.cairo` (lines 878–916) performs the following steps:

1. Reads the `ReplaceClassRequest` from `syscall_ptr`, which contains the caller-supplied `class_hash`.
2. Deducts gas.
3. Reads the current `StateEntry` for `contract_address` from `contract_state_changes`.
4. Writes a new `StateEntry` with `class_hash = request.class_hash` — the raw, unvalidated caller input.
5. Commits this to `contract_state_changes` via `dict_update`.

```cairo
// Replaces the class.
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The TODO comment at line 898 explicitly acknowledges the missing check. There is **no assertion** that `class_hash` exists in `contract_class_changes` (the dict of declared Sierra→CASM mappings) or in the `compiled_class_facts_bundle` provided to the OS. The OS blindly commits the new class hash to state.

This is structurally identical to the zAuction `SetZauction(address)` pattern: a reference is updated immediately and unconditionally, with no validation of the new target's existence or validity.

---

### Impact Explanation

Once a contract's class hash is set to an undeclared value:

- The OS cannot resolve a compiled class for that hash. Any subsequent transaction that calls into the contract (including `__validate__` and `__execute__` on account contracts) will fail at the proof level — the sequencer cannot include such calls in a provable block.
- The contract is permanently unexecutable. There is no recovery path: `replace_class` can only be called from within the contract's own execution context, which is now impossible.
- **All funds held in the contract are permanently frozen.** For account contracts, this means the user's ETH/STRK balance is irrecoverable. For vault or escrow contracts, all deposited assets are lost.

This satisfies the **Critical: Permanent freezing of funds** impact category.

---

### Likelihood Explanation

The attack surface is reachable by any unprivileged contract deployer or user:

1. **Malicious contract pattern**: A contract developer publishes a contract that appears legitimate (e.g., a vault or multisig) but includes a hidden or governance-gated `replace_class` call. After users deposit funds, the developer triggers `replace_class` with an arbitrary undeclared hash (e.g., `felt(1)`), permanently bricking the contract and freezing all deposits.
2. **Accidental self-bricking**: A contract with an upgrade mechanism that computes the new class hash incorrectly (e.g., off-by-one in a hash, or a hash of an undeclared class) will permanently brick itself. The OS provides no safety net.
3. **No privileged access required**: `replace_class` is a standard syscall available to any Cairo 1 contract. The attacker only needs to be the contract's own execution context — i.e., any caller that can invoke a function on the contract that internally calls `replace_class`.

---

### Recommendation

Before committing the new class hash to `contract_state_changes`, assert that it exists in the declared class set. Concretely, perform a `dict_read` on `contract_class_changes` for the new `class_hash` and assert the result is non-zero (i.e., a compiled class hash has been registered for it). This is already noted as a TODO at line 898 and should be treated as a security-critical fix, not a deferred cleanup item.

---

### Proof of Concept

1. Deploy contract `Vault` (Cairo 1) that accepts ETH deposits and exposes an `upgrade(new_hash: felt)` function that calls `replace_class(new_hash)`.
2. Users deposit 1000 ETH into `Vault`.
3. Attacker calls `Vault.upgrade(0x1)` — `0x1` is not a declared class hash.
4. The OS executes `execute_replace_class`:
   - `request.class_hash = 0x1`
   - No check against `contract_class_changes` (TODO at line 898 is unimplemented)
   - `dict_update` commits `StateEntry(class_hash=0x1, ...)` for `Vault`'s address
5. In the next block, any call to `Vault` requires the OS to resolve compiled class `0x1`. It does not exist in `compiled_class_facts_bundle`. The sequencer cannot construct a valid proof for any transaction touching `Vault`.
6. `Vault` is permanently unexecutable. The 1000 ETH is permanently frozen.
