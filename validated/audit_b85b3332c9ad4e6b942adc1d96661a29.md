### Title
DataLayer `Block` Struct Stores Redundant `node_type` in `NodeMetadata` Without Cross-Validation, Enabling Crafted-Blob Forged Proof-of-Inclusion — (File: `crates/chia-datalayer/src/merkle/format.rs`)

---

### Summary

The `Block` struct in the DataLayer Merkle implementation stores the node type in two independent, never-cross-validated locations: `NodeMetadata.node_type` (a serialized byte) and the `Node` enum variant (`Node::Internal` / `Node::Leaf`). The code itself acknowledges this with the comment `// NOTE: metadata node type and node's type not verified for agreement`. Because `Block::from_bytes` uses only `metadata.node_type` to decide how to interpret the data bytes, an attacker who supplies a crafted blob to `MerkleBlob::new` (reachable via the Python binding `py_init`) can set `metadata.node_type = Leaf` on bytes that are actually an `InternalNode` layout. This causes the system to misinterpret the `left`/`right` child-index fields as `key`/`value` fields, registering a phantom leaf with an attacker-controlled key and value in the `BlockStatusCache`, and ultimately producing a valid-looking proof of inclusion for state that was never committed.

**`Block::from_bytes` never checks that the resulting `node` variant agrees with `metadata.node_type`:** [3](#0-2) 

**Serialized layouts of the two node types:**

`InternalNode` (45 bytes, padded to 53):
- `hash` (32 B) · `parent` (5 B) · `left: TreeIndex/u32` (4 B) · `right: TreeIndex/u32` (4 B) · 8 B padding

`LeafNode` (53 bytes):
- `hash` (32 B) · `parent` (5 B) · `key: KeyId/i64` (8 B) · `value: ValueId/i64` (8 B) [4](#0-3) 

When `metadata.node_type = Leaf` but the data bytes are an `InternalNode` layout, the parser reads:
- `key` ← bytes of `left` (4 B) ‖ `right` (4 B) → attacker-controlled `i64`
- `value` ← 8 bytes of padding → attacker-controlled `i64`

**`BlockStatusCache` trusts `metadata.node_type` unconditionally** and registers the node as a leaf with the crafted key/value: [5](#0-4) 

**`get_proof_of_inclusion` then produces a proof for the phantom key:** [6](#0-5) 

**`check_integrity` is only active in test/debug builds** (via the `Drop` impl), so production code never automatically detects the inconsistency: [7](#0-6) 

---

### Impact Explanation

An attacker who can supply a crafted blob to `MerkleBlob::new` can:

1. Register any key K and value V as a phantom leaf by setting `metadata.node_type = Leaf` on an `InternalNode`-layout block whose `left`/`right` bytes encode K and whose padding bytes encode V.
2. Obtain a structurally valid `ProofOfInclusion` for K from `get_proof_of_inclusion(K)`.
3. Cause the DataLayer to accept forged inclusion proofs, proving state that was never committed to the tree.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

---

### Likelihood Explanation

The Python binding `py_init` accepts an arbitrary byte buffer and passes it directly to `MerkleBlob::new`: [8](#0-7) 

Any caller with access to the Python `MerkleBlob` constructor (e.g., a DataLayer client receiving a peer-supplied blob) can trigger this path. No privileged role is required. The `BlockStatusCache` performs no cross-validation of `metadata.node_type` against the actual data bytes, and `check_integrity` is disabled in production.

---

### Recommendation

1. **Eliminate the redundancy**: derive `NodeType` from the `Node` enum variant at serialization time rather than storing it as an independent field. Remove `metadata.node_type` from the on-disk format and reconstruct it from the `Node` variant after deserialization.
2. **If the field must remain**: add a validation step in `Block::from_bytes` (or a new `Block::new` constructor) that asserts `metadata.node_type` matches the deserialized `Node` variant, returning an error on mismatch.
3. **Enable `check_integrity` in production** (or at least on blob load) so structural violations are caught before they affect proof generation.

        if index == self.right {
            Ok(self.left)

    pub fn get_proof_of_inclusion(
        &self,
        key: KeyId,
    ) -> Result<proof_of_inclusion::ProofOfInclusion, Error> {
        let mut index = *self
            .block_status_cache
            .get_index_by_key(key)
            .ok_or(Error::UnknownKey(key))?;

        let node = self
            .get_node(index)?
            .expect_leaf("key to index mapping should only have leaves");

        let parents = self.get_lineage_blocks_with_indexes(index)?;
        let mut layers: Vec<proof_of_inclusion::ProofOfInclusionLayer> = Vec::new();
        let mut parents_iter = parents.iter();
        // first in the lineage is the index itself, second is the first parent
        parents_iter.next();
        for (next_index, block) in parents_iter {
            if block.metadata.dirty {
                return Err(Error::Dirty(*next_index));
            }
            let parent = block
                .node
                .expect_internal("all nodes after the first should be internal");
            let sibling_index = parent.sibling_index(index)?;
            let sibling_block = self.get_block(sibling_index)?;
            let sibling = sibling_block.node;
            let layer = proof_of_inclusion::ProofOfInclusionLayer {
                other_hash_side: parent.get_sibling_side(index)?,
                other_hash: sibling.hash(),
                combined_hash: parent.hash,
            };
            layers.push(layer);
            index = *next_index;
        }

        Ok(proof_of_inclusion::ProofOfInclusion {
            node_hash: node.hash,
            layers,
        })
