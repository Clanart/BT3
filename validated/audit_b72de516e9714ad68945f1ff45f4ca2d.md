### Title
OS `get_block_hashes` Accepts Arbitrary `BlockHeaderCommitments` as Unconstrained Hints — Block Hash Does Not Commit to Actual Transaction/Event/State-Diff Data - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/block_hash.cairo`)

### Summary

The Starknet OS `get_block_hashes` function populates all five `BlockHeaderCommitments` fields (`transaction_commitment`, `event_commitment`, `receipt_commitment`, `state_diff_commitment`, `packed_lengths`) exclusively through the non-deterministic hint `%{ GetBlockHashes %}`, with no Cairo-level assertion that these values are correctly derived from the block's actual transactions, events, or state diff. The only check, `%{ CheckBlockHashConsistency %}`, is itself a hint (not a Cairo constraint) and is circular: it compares the computed hash against `os_input.new_block_hash`, which was pre-computed by the prover using the same unverified commitments. The STARK proof therefore does not enforce that the output block hash commits to the correct block content.

### Finding Description

In `block_hash.cairo`, `get_block_hashes` declares `header_commitments` as a local variable and fills it entirely via a hint:

```cairo
// Currently, the header commitments and gas prices are not computed by the OS.
// TODO(Yoni, 1/1/2027): compute the header commitments and gas prices.
local header_commitments: BlockHeaderCommitments*;
...
%{ GetBlockHashes %}
let block_hash = calculate_block_hash(
    block_info=block_info,
    header_commitments=header_commitments,
    ...
);
%{ CheckBlockHashConsistency %}
``` [1](#0-0) 

In Cairo 0, hints are executed by the prover but are invisible to the verifier. The verifier only checks algebraic constraints. Because no `assert` or `tempvar` constraint ties `header_commitments` to the block's actual transaction list, event list, or state diff, the prover may supply any five field elements and the resulting STARK proof remains valid.

The Rust hint implementation `get_block_hashes` reads `os_input.block_hash_commitments` verbatim and writes them into the Cairo VM:

```rust
let commitments = &os_input.block_hash_commitments;
ctx.insert_values_to_fields(header_commitments_ptr, CairoStruct::BlockHeaderCommitments, &[
    ("transaction_commitment", commitments.transaction_commitment.0.into()),
    ("event_commitment",       commitments.event_commitment.0.into()),
    ("receipt_commitment",     commitments.receipt_commitment.0.into()),
    ("state_diff_commitment",  commitments.state_diff_commitment.0.0.into()),
    ("packed_lengths",         commitments.concatenated_counts.into()),
])?;
``` [2](#0-1) 

`check_block_hash_consistency` then compares the computed hash against `os_input.new_block_hash`:

```rust
if calculated_block_hash != os_input.new_block_hash.0 {
    return Err(OsHintError::AssertionFailed { ... });
}
``` [3](#0-2) 

But `new_block_hash` was itself computed by the prover using the same `block_hash_commitments` (see `OsBlockInput`): [4](#0-3) 

The check is therefore circular: it verifies internal consistency between two prover-supplied values, not correctness against the block's actual content.

The OS flow tests confirm the OS accepts all-zero commitments without error:

```rust

    &PartialBlockHashComponents::new(&block_info, block_hash_commitments.clone()),
    ...
).unwrap();
``` [5](#0-4) 

The `BlockHeaderCommitments` struct carries all five commitment shards that are chained into the block hash: [6](#0-5) 

### Impact Explanation

The STARK proof output includes the new block hash, which is the authoritative on-chain commitment to the block's content. Because the OS does not constrain `transaction_commitment`, `event_commitment`, `receipt_commitment`, `state_diff_commitment`, or `packed_lengths` to be derived from the actual executed transactions and state diff, a prover can produce a valid STARK proof whose output block hash commits to fabricated (e.g., all-zero) values for these fields. The L1 verifier accepts the proof (the Cairo constraints are satisfied), and the resulting on-chain block hash does not actually commit to the correct transactions, events, receipts, or state diff. This breaks the core soundness invariant: the block hash is supposed to be an unforgeable commitment to the block's full content.

Concretely, the five "shards" of the block hash that are not individually verified by the OS are the direct analog of the unblinded h1/h2/h3 quotient polynomial shards in the external Plonk report — both represent commitment components that are included in the final hash/proof without the proof system individually enforcing their correctness.

### Likelihood Explanation

Exploitation requires the prover (the sequencer operator) to deliberately supply wrong `block_hash_commitments`. This is a privileged position. However, the invariant is broken by design and is explicitly acknowledged with a `TODO(Yoni, 1/1/2027)` comment, meaning it is a known gap in the current proof system that has not yet been closed. Any party operating a sequencer node — including a compromised or malicious operator — can exploit this without any additional access beyond running the prover.

### Recommendation

Implement Cairo-level computation of each `BlockHeaderCommitments` field directly from the block's transaction list, event list, and state diff within the OS program, replacing the hint-only approach. Add Cairo `assert` statements (not hints) to verify each commitment against the computed value before it is chained into the block hash. This is already planned per the `TODO(Yoni, 1/1/2027)` comment in `block_hash.cairo`.

### Proof of Concept

1. Construct an `OsBlockInput` with `block_hash_commitments = BlockHeaderCommitments::default()` (all zeros) and a non-empty transaction list.
2. Pre-compute `new_block_hash = calculate_block_hash(&PartialBlockHashComponents::new(&block_info, BlockHeaderCommitments::default()), state_root, prev_hash)`.
3. Run the OS. The `GetBlockHashes` hint injects the all-zero commitments; `CheckBlockHashConsistency` passes because `new_block_hash` was pre-computed with the same zeros.
4. The OS produces a valid STARK proof whose output block hash commits to zero transaction/event/receipt/state-diff commitments, regardless of the actual block content.

This is exactly what the OS flow test harness does at `test_manager.rs:903` — it passes `BlockHeaderCommitments::default()` and the OS runs to completion without error. [7](#0-6) [2](#0-1) [5](#0-4)

// Checks that the calculated block hash is consistent with the expected block hash (sanity check).
pub(crate) fn check_block_hash_consistency<S: StateReader>(
    hint_processor: &mut SnosHintProcessor<'_, S>,
    ctx: HintContext<'_>,
) -> OsHintResult {
    let os_input = &hint_processor.get_current_execution_helper()?.os_block_input;
    let calculated_block_hash = ctx.get_integer(Ids::BlockHash)?;

    if calculated_block_hash != os_input.new_block_hash.0 {
        return Err(OsHintError::AssertionFailed {
            message: format!(
                "Calculated block hash {calculated_block_hash} does not match expected block hash \
                 {}",
                os_input.new_block_hash.0
            ),
        });
    }

    Ok(())
}

pub(crate) fn get_block_hashes<S: StateReader>(
    hint_processor: &mut SnosHintProcessor<'_, S>,
    mut ctx: HintContext<'_>,
) -> OsHintResult {
    let os_input = &hint_processor.get_current_execution_helper()?.os_block_input;
    let block_info = &os_input.block_info;
    let gas_prices = &block_info.gas_prices;
    let commitments = &os_input.block_hash_commitments;

    ctx.insert_value(Ids::PreviousBlockHash, os_input.prev_block_hash.0)?;

    let header_commitments_ptr = ctx.vm.add_memory_segment();
    ctx.insert_values_to_fields(
        header_commitments_ptr,
        CairoStruct::BlockHeaderCommitments,
        &[
            ("transaction_commitment", commitments.transaction_commitment.0.into()),
            ("event_commitment", commitments.event_commitment.0.into()),
            ("receipt_commitment", commitments.receipt_commitment.0.into()),
            ("state_diff_commitment", commitments.state_diff_commitment.0.0.into()),
            ("packed_lengths", commitments.concatenated_counts.into()),
        ],
    )?;
    ctx.insert_value(Ids::HeaderCommitments, header_commitments_ptr)?;

    // The on-chain block hash commits to the exact ASCII encoding of the version string, which
    // may be newer than the latest known `StarknetVersion` variant.
    let starknet_version_string = os_input
        .starknet_version_override
        .clone()
        .unwrap_or_else(|| block_info.starknet_version.to_string());
    ctx.insert_value(Ids::StarknetVersion, ascii_as_felt(&starknet_version_string)?)?;

    let [gas_prices_hash]: [Felt; 1] = gas_prices_to_hash(
        &gas_prices.l1_gas_price_per_token(),
        &gas_prices.l1_data_gas_price_per_token(),
        &gas_prices.l2_gas_price_per_token(),
        &block_info.starknet_version.try_into()?,
    )
    .try_into()
    .map_err(|_| {
        OsHintError::UnsupportedStarknetVersionForBlockHash(block_info.starknet_version)
    })?;

    ctx.insert_value(Ids::GasPricesHash, gas_prices_hash)?;

    Ok(())
}
