### Title
No Minimum Transfer Amount Check Allows Permanently Unresolvable NEAR-Origin Transfers That Irrecoverably Lock User Funds - (File: `near/omni-bridge/src/lib.rs`)

### Summary

The NEAR bridge's `init_transfer` function only validates `fee < amount` but does not enforce that the net transfer amount survives decimal normalization. When a user initiates a NEAR-origin transfer with `amount - fee < 10^(origin_decimals - decimals)`, the `normalize_amount` call inside `sign_transfer` returns 0. The subsequent `require!(amount_to_transfer > 0)` guard panics on every relay attempt, leaving the transfer permanently stuck in `pending_transfers` with no cancellation or recovery path. The user's tokens are irrecoverably locked or burned.

### Finding Description

**Root cause — missing minimum amount guard in `init_transfer`**

`init_transfer` (called from `ft_on_transfer`) stores the transfer and locks/burns the user's tokens after only one arithmetic check:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

There is no check that `amount - fee` is large enough to survive the decimal normalization that will be applied later.

**Normalization in `sign_transfer` uses floor division**

When a trusted relayer calls `sign_transfer`, the net amount is normalized:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

`normalize_amount` is floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [3](#0-2) 

If `amount - fee < 10^(origin_decimals - decimals)`, the division yields 0 and `sign_transfer` panics unconditionally on every call.

**No recovery path exists**

- `update_transfer_fee` cannot help: fee must satisfy `fee < amount`, so the maximum settable fee is `amount - 1`, giving `amount_without_fee = 1`, which still normalizes to 0 for any non-trivial decimal gap.
- There is no `cancel_transfer` or admin-rescue function.
- `claim_fee` requires a `FinTransfer` proof from the destination chain, which can never exist because the transfer was never signed.
- `storage_unregister(force=true)` refunds only the `available` storage balance; the tokens locked/burned by `init_transfer_internal` are not returned. [4](#0-3) 

**The code comment covers only the remainder case, not the zero-normalized case**

The existing comment at `normalize_amount` reads: *"When fee = 0, dust stays locked/burned."* This describes the sub-unit **remainder** after a successful normalization (e.g., `1_000_001 / 10^6 = 1`, remainder `1`). It does not address the distinct case where the **entire** net amount normalizes to 0 and the transfer is permanently stuck. [5](#0-4) 

### Impact Explanation

A user whose NEAR-origin transfer has `amount - fee < 10^(origin_decimals - decimals)` suffers:

1. **Irrecoverable token loss** — tokens are locked (native) or burned (bridged) by `init_transfer_internal` before `sign_transfer` is ever called.
2. **Permanent storage deposit lock** — the NEAR storage deposit charged for the `pending_transfers` entry is never refunded because the entry can never be removed.
3. **No on-chain remedy** — no function in the contract can cancel, refund, or re-route the stuck transfer.

This satisfies the allowed Critical impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

For the most common decimal gap (NEAR 24 → EVM 18, factor `10^6`), the maximum per-transfer loss is `999 999` yoctoNEAR (≈ 0.000001 NEAR), which is dust. However, for tokens with a smaller decimal gap (e.g., origin 10 decimals, destination 8 decimals, factor `10^2 = 100`), the threshold rises to 99 token units, which can represent meaningful value for high-unit-price tokens. Additionally, an attacker can create many such transfers at low cost to permanently consume storage deposits and nonces.

### Likelihood Explanation

Any unprivileged user who calls `ft_transfer_call` with an amount below the normalization threshold triggers this path. The entry point is fully public. The user need not be malicious — a UI rounding error or a token with an unusual decimal configuration is sufficient. The `fee < amount` guard that currently exists gives no protection against this class of input.

### Recommendation

Add a pre-storage check in `init_transfer` (or in `init_transfer_internal`) that rejects transfers whose net amount normalizes to zero:

```rust
// After constructing transfer_message, before add_transfer_message:
let decimals = self.token_decimals.get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);
require!(
    Self::normalize_amount(
        transfer_message.amount_without_fee()
            .near_expect(BridgeError::InvalidFee),
        decimals,
    ) > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

This mirrors the guard already present in `sign_transfer` and prevents the transfer from ever being stored when it cannot be completed.

### Proof of Concept

**Setup**: NEAR token with `origin_decimals = 24`, destination EVM token with `decimals = 18` (normalization factor = `10^6`).

1. User calls `ft_transfer_call` with `amount = 500_000` (yoctoNEAR units) and `fee = 0`.
2. `init_transfer` passes the only guard: `0 < 500_000` ✓.
3. `init_transfer_internal` stores the transfer in `pending_transfers`, locks/burns `500_000` yoctoNEAR.
4. Relayer calls `sign_transfer` for this `transfer_id`.
5. `normalize_amount(500_000, {decimals:18, origin_decimals:24})` = `500_000 / 10^6` = **0**.
6. `require!(0 > 0, BridgeError::InvalidAmountToTransfer)` → **panic**.
7. Transaction reverts; transfer remains in `pending_transfers`.
8. Steps 4–7 repeat identically on every future relay attempt.
9. User's `500_000` yoctoNEAR and storage deposit are permanently locked with no on-chain recovery path. [6](#0-5) [2](#0-1) [3](#0-2)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
