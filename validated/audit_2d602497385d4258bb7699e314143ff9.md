### Title
`ProofOfInclusion::valid()` Tautological Root-Hash Check Accepts Forged DataLayer Inclusion Proofs — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` performs only internal self-consistency checks. Its final comparison `existing_hash == self.root_hash()` is tautologically true whenever all per-layer checks pass, because `root_hash()` returns `last.combined_hash` — the exact value `existing_hash` holds at loop exit. No external trusted root is ever compared. An attacker who can supply a crafted `ProofOfInclusion` (via the Streamable `from_bytes` / `from_json_dict` Python binding) can forge a proof of inclusion for any key-value pair in any DataLayer tree and have `valid()` return `True`.

### Finding Description

`ProofOfInclusion::valid()` is defined as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;   // ← existing_hash := layer.combined_hash
    }
    existing_hash == self.root_hash()      // ← always true: root_hash() == last.combined_hash
}
``` [1](#0-0) 

`root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← same value as existing_hash at loop exit
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

After the loop, `existing_hash` equals `last.combined_hash` (because the loop body sets `existing_hash = calculated_hash` only after verifying `calculated_hash == layer.combined_hash`). `root_hash()` returns that same `last.combined_hash`. The final comparison is therefore always `true` when all per-layer checks pass — it never compares against any externally trusted tree root.

The struct is `Streamable` and exposed via Python bindings with `from_bytes`, `from_json_dict`, and `py_valid()`: [3](#0-2) 

`ProofOfInclusion` is also fully exposed in the Python type stubs: [4](#0-3) 

### Impact Explanation

An attacker can:
1. Choose any arbitrary `node_hash` (claiming to be the leaf hash of a key-value pair they want to forge).
2. Build any number of `ProofOfInclusionLayer` entries where each `combined_hash` is computed as `calculate_internal_hash(prev_hash, side, other_hash)` using attacker-chosen `other_hash` and `side` values.
3. Deserialize this blob via `ProofOfInclusion.from_bytes(...)` or `from_json_dict(...)`.
4. Call `proof.valid()` → returns `True`.

Any DataLayer consumer that calls `proof.valid()` without separately comparing `proof.root_hash()` against a trusted on-chain root will accept the forged proof. This matches the allowed impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, letting untrusted input prove invalid state.**

### Likelihood Explanation

The `valid()` method is the sole proof-verification API exposed to Python callers. Its name implies completeness. No parameter for a trusted root is provided, so callers have no obvious mechanism to perform the missing check. Any Python integration that calls `proof.valid()` as the sole gate — a natural and expected usage — is vulnerable. The attack requires only crafting a valid byte sequence, which is trivially achievable with the public `Streamable` format.

### Recommendation

`valid()` must accept a trusted root hash as a parameter and compare `existing_hash` against it at the end of the loop, rather than against `self.root_hash()`:

```rust
pub fn valid(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // compare against externally supplied root
}
```

The Python binding and all call sites must be updated accordingly. `root_hash()` should remain as a convenience accessor but must not be used as the verification target inside `valid()`.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer, Side
import hashlib

def sha256(*parts):
    h = hashlib.sha256()
    for p in parts:
        h.update(p)
    return h.digest()

def calculate_internal_hash(left_hash, right_hash):
    # mirrors crate::calculate_internal_hash with left side
    prefix = bytes(30)
    return sha256(prefix, b'\x00', b'\x00', left_hash, right_hash)

# Attacker chooses arbitrary leaf hash (claims key X maps to value Y)
fake_node_hash = bytes([0xAA] * 32)
fake_other_hash = bytes([0xBB] * 32)

# Build one internally-consistent layer
combined = calculate_internal_hash(fake_node_hash, fake_other_hash)

layer = ProofOfInclusionLayer(
    other_hash_side=Side.Left,   # or Right
    other_hash=fake_other_hash,
    combined_hash=combined,
)

proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True despite proof never touching any real tree
assert proof.valid(), "Forged proof accepted!"
print("root_hash reported by forged proof:", proof.root_hash().hex())
```

The forged proof passes `valid()` with a completely fabricated tree root, demonstrating that any caller relying solely on `valid()` will accept the forgery.
