### Title
Stale `feemarketParams` Snapshot in `NewDynamicFeeChecker` Allows Cosmos Native Transactions to Bypass Current BaseFee Requirement - (File: `ante/evm/fee_checker.go`)

### Summary

`NewDynamicFeeChecker` captures `feemarketParams` (including `BaseFee`) at ante handler construction time (app initialization) rather than reading the live on-chain value per transaction. Because the EIP-1559 `BaseFee` adjusts every block, every Cosmos native transaction processed after the first block uses a stale baseFee for fee validation. During network congestion — when the baseFee rises — Cosmos native transactions can be admitted and committed with fees below the current protocol-required baseFee, bypassing the fee market entirely.

`feemarketParams` is a local variable in `newCosmosAnteHandler`. Its address is taken and captured by the closure. After `newCosmosAnteHandler` returns, the pointed-to struct is never updated — it permanently holds the genesis/initialization-time `BaseFee`. Every subsequent call to the returned `TxFeeChecker` computes `baseFee` from this frozen snapshot, not from the live keeper state.

### Impact Explanation

The `NewDynamicFeeChecker` closure enforces `feeCap >= baseFee` and computes `effectivePrice = min(stale_baseFee + tip, feeCap)`. When the live baseFee has risen above the stale snapshot (e.g., during congestion), a Cosmos native transaction with `feeCap = stale_baseFee` passes the check and has `effectiveFee = stale_baseFee * gasLimit` deducted from the sender — far below what the current fee market requires. The transaction is committed to state with mis-accounted fees. This is a fee market bypass for all Cosmos SDK native transactions (bank sends, staking, governance, IBC, etc.) processed after the first block.

This matches the allowed High impact: **"fee market, ante handler, or proposal handling bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."**

### Likelihood Explanation

The baseFee changes every block. After block 1, the stale snapshot is already wrong. Any user submitting a Cosmos native transaction with `feeCap` equal to the genesis baseFee (e.g., 1 Gwei) will bypass the current baseFee check whenever the live baseFee exceeds that value. No special privileges are required — any unprivileged user can submit a Cosmos native transaction. The condition is trivially reachable on any active chain with EIP-1559 enabled and `DynamicFeeChecker: true`.

### Recommendation

Replace the stale-snapshot pattern with a live keeper read inside the closure. In `NewDynamicFeeChecker`, accept the `FeeMarketKeeper` interface and call `keeper.GetBaseFee(ctx)` on each invocation instead of deriving baseFee from the captured `feemarketParams` pointer:

```go
// Instead of:
baseFee := types.GetBaseFee(ctx.BlockHeight(), ethCfg, feemarketParams)

// Use:
baseFee := feeMarketKeeper.GetBaseFee(ctx)
```

Apply the same fix to `MinGasPriceDecorator`: read `MinGasPrice` from the keeper on each `AnteHandle` call rather than from the captured pointer.

### Proof of Concept

1. Chain starts with genesis `BaseFee = 1_000_000_000` (1 Gwei). `newCosmosAnteHandler` is called; `feemarketParams.BaseFee = 1 Gwei` is captured.
2. Network activity causes the live `BaseFee` to rise to `10_000_000_000` (10 Gwei) after several blocks.
3. Attacker submits a `MsgSend` (Cosmos native) with `feeCap = 1 Gwei` and `tip = 0`.
4. `NewDynamicFeeChecker` computes `baseFee = 1 Gwei` (stale), passes `feeCap >= baseFee`, deducts `1 Gwei * gasLimit`.
5. Transaction is committed. Attacker paid 10× less than the current fee market requires.
6. Repeat for any Cosmos native message type; EVM transactions are unaffected because `newEthAnteHandler` reads the live baseFee.

			// london hardfork is not enabled: fallback to min-gas-prices logic
