### Title
P2P Sync Accepts `SignedBlockHeader` With Unverified Signature Content, Allowing Injection of Wrong Block Hash, State Root, and State Diff Commitment — (`crates/apollo_p2p_sync/src/client/header.rs`)

---

### Summary

The P2P sync client enforces that a `SignedBlockHeader` carries exactly one `BlockSignature` (a count/quorum check), but never calls `verify_block_signature` to cryptographically validate that the signature was produced by the expected sequencer over the correct `(block_hash, state_diff_commitment)` pair. Any connected peer can therefore transmit a `SignedBlockHeader` carrying an arbitrary or zeroed signature, and the node will store the supplied `block_hash`, `state_root`, and `state_diff_commitment` as authoritative chain state.

---

### Finding Description

**Root cause — count check without content check**

`ALLOWED_SIGNATURES_LENGTH` is defined as `1` and enforced in `parse_data_for_block`:

```

``` [2](#0-1) 

After this check the header is passed to `write_to_storage`, which calls `append_header` and `append_block_signature` with no cryptographic check: [3](#0-2) 

The function `verify_block_signature` exists and is correct — it hashes `(block_hash, state_diff_commitment)` with Poseidon and verifies the ECDSA signature against the sequencer's public key: [4](#0-3) 

A `grep` over the entire repository shows `verify_block_signature` is referenced only in `block_test.rs` and its own definition; it is **never called** in any sync path.

**Analog to the external bug**

The Chainlink bug: `verifyTransmitSigners` iterates over `rs[]` but never checks `rs.length` against the expected signer count, so an empty array passes. The fix is to add `require(rs.length == verificationVars[…].signers.length)`.

The sequencer analog: `parse_data_for_block` checks `signatures.len() == 1` (the count/quorum check), but never checks whether the single signature is cryptographically valid. The missing call is `verify_block_signature(&sequencer_pub_key, &sig, &state_diff_commitment, &block_hash)`. A peer can supply `BlockSignature { r: Felt::ONE, s: Felt::ONE }` and the check passes.

**Attack path**

1. Attacker operates a Starknet P2P node and connects to a syncing victim node.
2. Attacker responds to a `HeaderQuery` with a `SignedBlockHeader` that contains:
   - A real-looking `block_number` matching the victim's sync position.
   - Attacker-chosen `block_hash`, `state_root`, `state_diff_commitment`, gas prices, etc.
   - A single `BlockSignature` with arbitrary `r`, `s` values (e.g., both `Felt::ONE`).
3. `parse_data_for_block` accepts the message because `signatures.len() == 1`.
4. `write_to_storage` persists the attacker-supplied `block_hash`, `state_root`, and `state_diff_commitment` into the MDBX header table.
5. All subsequent RPC calls (`starknet_getBlockWithTxHashes`, `starknet_getStateUpdate`, fee estimation, `starknet_call` against the stored state root, etc.) return the attacker-controlled values as authoritative.

---

### Impact Explanation

The stored `block_hash`, `state_root` (Patricia root of contract and class tries), and `state_diff_commitment` are the authoritative inputs for:
- RPC block-header and state-update responses.
- Fee estimation and simulation (which read state from the stored root).
- The committer's global-root chain (the stored root is the base for the next Patricia update).
- L1 state commitment verification (the node's view of what was committed on L1).

An attacker who injects a wrong `state_root` causes the node to serve incorrect storage values, class hashes, nonces, and balances to all callers, matching the **High** impact: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

---

### Likelihood Explanation

The P2P network is open: any node can connect and respond to sync queries. No privileged role is required. The victim node issues `HeaderQuery` messages and accepts the first well-formed response; a Sybil or man-on-path attacker can race a legitimate peer. The missing call is a single function that already exists in the codebase, confirming the check was simply never wired in.

---

### Recommendation

In `parse_data_for_block` (or immediately before `write_to_storage`), after confirming `signatures.len() == 1`, call `verify_block_signature` with the sequencer's known public key:

```rust
use starknet_api::block::verify_block_signature;

let sig = &signed_block_header.signatures[0];
let block_hash = &signed_block_header.block_header.block_hash;
// state_diff_commitment is used as GlobalRoot in verify_block_signature
let state_diff_commitment = signed_block_header
    .block_header
    .state_diff_commitment
    .map(|c| GlobalRoot(c.0.0))
    .unwrap_or_default();

match verify_block_signature(&sequencer_pub_key, sig, &state_diff_commitment, block_hash) {
    Ok(true) => {}
    Ok(false) => return Err(ParseDataError::BadPeer(BadPeerError::InvalidSignature { … })),
    Err(e)   => return Err(ParseDataError::BadPeer(BadPeerError::SignatureVerificationError(e))),
}
```

The sequencer public key must be supplied to the `HeaderStreamBuilder` from the node's chain configuration (it is already available in the `BlockContext` / chain-info used by other components).

/// Verifies that the the block header was signed by the expected sequencer.
pub fn verify_block_signature(
    sequencer_pub_key: &SequencerPublicKey,
    signature: &BlockSignature,
    state_diff_commitment: &GlobalRoot,
    block_hash: &BlockHash,
) -> Result<bool, BlockVerificationError> {
    let message_hash = Poseidon::hash_array(&[block_hash.0, state_diff_commitment.0]);
    verify_message_hash_signature(&message_hash, &signature.0, &sequencer_pub_key.0).map_err(
        |err| BlockVerificationError::BlockSignatureVerificationFailed {
            block_hash: *block_hash,
            error: err,
        },
    )
}
