### Title
Double-Finalization via Recipient-Controlled `ft_transfer_call` Return Value Deleting Finalization Record — (`near/omni-bridge/src/lib.rs`)

### Summary
When an inbound `fin_transfer` proof is processed and the recipient is a NEAR contract whose `ft_on_transfer` returns `0`, the callback `fin_transfer_send_tokens_callback` calls `remove_fin_transfer`, which **deletes the transfer ID from `finalised_transfers`**. This allows the identical proof to be submitted a second time, passing the duplicate-check in `add_fin_transfer`, and minting or unlocking tokens a second time for the same origin-chain event.

### Finding Description

The call sequence for a NEAR-recipient inbound transfer is:

```
fin_transfer (public, any relayer)
  → verify_proof
  → fin_transfer_callback
    → process_fin_transfer_to_near
      → add_fin_transfer          ← inserts transfer_id into finalised_transfers
      → send_tokens               ← calls ft_transfer_call on recipient
        → fin_transfer_send_tokens_callback
            if is_refund_required (ft_transfer_call returned U128(0)):
              → remove_fin_transfer  ← DELETES transfer_id from finalised_transfers
```

After step 6, `finalised_transfers` no longer contains the transfer ID. A second call to `fin_transfer` with the **same proof bytes** will:
- Pass proof verification (the prover has no per-transfer replay guard of its own beyond what the bridge records)
- Pass `add_fin_transfer`'s duplicate check (the record was deleted)
- Mint or unlock tokens again

The attacker controls the recipient: they initiate the origin-chain transfer specifying a NEAR contract they own as recipient, then make that contract's `ft_on_transfer` return `0` (i.e., return the full amount). This triggers `is_refund_required = true` at: [4](#0-3) 

and `remove_fin_transfer` at: [5](#0-4) 

The attacker then replays the same proof, receiving tokens again.

### Impact Explanation
- For bridged (deployed) tokens: the bridge mints unbacked tokens on NEAR, inflating supply without a corresponding lock on the origin chain.
- For native NEAR tokens locked in the bridge: the bridge unlocks and transfers them twice, draining the vault.

This directly matches the **High** impact category: *Cross-chain replay / double-finalization enabling double-spend or unbacked supply.*

### Likelihood Explanation
The attacker only needs to:
1. Initiate a legitimate transfer on the origin chain with a recipient they control on NEAR.
2. Deploy a trivial NEAR contract whose `ft_on_transfer` returns the full amount (i.e., `U128(amount)`), causing `ft_transfer_call` to return `U128(0)` to the bridge.
3. Submit `fin_transfer` once, wait for the callback to delete the record, then submit again.

No privileged role, leaked key, or external dependency compromise is required. The attacker is the legitimate originator of the transfer.

### Recommendation
**Do not delete the finalization record on refund.** The `finalised_transfers` set must be a permanent append-only ledger. When a refund is required, the bridge should keep the transfer ID in `finalised_transfers` and handle the refund separately (e.g., re-credit the relayer or burn the minted tokens), but never call `remove_fin_transfer` in the callback path.

Specifically, in `fin_transfer_send_tokens_callback`: [6](#0-5) 

Remove the `self.remove_fin_transfer(...)` call. The storage deposit can be refunded to `storage_owner` without removing the finalization record.

### Proof of Concept

```rust
// 1. Deploy a malicious recipient contract on NEAR:
//    ft_on_transfer returns amount (all tokens returned)
// 2. Initiate transfer on EVM: recipient = malicious_contract.near
// 3. Submit fin_transfer with valid proof → add_fin_transfer inserts ID
// 4. malicious_contract.ft_on_transfer returns U128(amount) → ft_transfer_call returns U128(0)
// 5. fin_transfer_send_tokens_callback: is_refund_required = true
//    → remove_fin_transfer deletes ID from finalised_transfers
// 6. Assert: finalised_transfers.contains(transfer_id) == false
// 7. Submit fin_transfer again with identical proof args
//    → add_fin_transfer succeeds (no duplicate)
//    → tokens minted/unlocked a second time
// 8. Assert: recipient received tokens twice; origin event settled once
```
