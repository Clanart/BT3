### Title
Missing Declared Class Validation in `execute_replace_class` Enables Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash supplied by a contract is actually a declared class. The OS unconditionally writes the caller-supplied hash into the contract's state entry. Any contract can therefore set its own class hash to an arbitrary, undeclared value, rendering itself permanently uncallable and freezing all funds it holds.

---

### Finding Description

**Vulnerability class:** Authorization/ownership check bypass — the function correctly restricts *who* may act (only the contract itself), but omits the complementary validity check on *what* is being set (the new class hash must be declared). This is structurally identical to BAB-2, where `checkApproval` verified the operator but not the owner.

In `execute_syscalls.cairo` the syscall is dispatched with the currently-executing contract's address:

```cairo
if (selector == REPLACE_CLASS_SELECTOR) {
    execute_replace_class(contract_address=execution_context.execution_info.contract_address);
``` [1](#0-0) 

Inside `execute_replace_class`, the new class hash is read directly from the syscall request and written into `contract_state_changes` with no check against `contract_class_changes`:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [2](#0-1) 

The in-code TODO explicitly acknowledges the missing check. Because the OS is the authoritative source of truth for state transitions (the L1 verifier accepts whatever the proof attests to), this omission is a protocol-level gap, not merely an off-chain validation gap.

After the state transition is committed, every subsequent call to the affected contract will fail: the OS will look up the contract's class hash, find it absent from `contract_class_changes`, and be unable to locate executable code. The contract is permanently bricked.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that holds user assets (ERC-20 token contract, multisig, vault, bridge escrow) and whose upgrade path allows `replace_class` to be invoked with an attacker-controlled hash will have all held funds permanently frozen. The state transition is proven and committed to L1; there is no recovery path. The OS produces a valid proof for the invalid state, so the L1 verifier accepts it.

---

### Likelihood Explanation

The `replace_class` syscall is available to every Sierra contract. Realistic trigger paths include:

1. **Governance/upgrade attack:** A shared contract (e.g., a token or vault) exposes an upgrade entry point gated by governance. An attacker who wins a governance vote (or exploits a governance bug) calls `replace_class` with an undeclared hash. The OS accepts it; the contract is frozen.
2. **Malicious contract developer:** A developer deploys a contract that accumulates user deposits, then calls `replace_class` with `0xdeadbeef`. The OS commits the state; funds are frozen.
3. **Buggy upgrade logic:** A contract's upgrade function fails to validate the new class hash before calling `replace_class`. An unprivileged user supplies an undeclared hash through the public interface.

No privileged sequencer cooperation is required; the syscall is reachable from any ordinary transaction.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that it exists in `contract_class_changes` (i.e., that it was previously declared). Concretely, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the returned value is non-zero:

```cairo
// Proposed fix inside execute_replace_class:
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the check already present in `execute_declare_transaction`, which asserts `assert_not_zero(compiled_class_hash)` before writing to `contract_class_changes`. [3](#0-2) 

---

### Proof of Concept

1. Attacker deploys `VaultContract` which accepts ETH/token deposits from users and exposes an `upgrade(new_class_hash)` entry point callable by the owner (or via governance).
2. Attacker (as owner, or after winning a governance vote) calls `upgrade(0xdeadbeef)` where `0xdeadbeef` is never declared.
3. `VaultContract.__execute__` calls the `replace_class` syscall with `class_hash = 0xdeadbeef`.
4. The OS executes `execute_replace_class`: it reads `request.class_hash = 0xdeadbeef`, skips the missing declared-class check, and writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
5. The sequencer generates a valid STARK proof; the L1 verifier accepts the new state root.
6. In the next block, any user who calls `VaultContract` causes the OS to look up class `0xdeadbeef` in `contract_class_changes`, find nothing, and fail. Every call reverts.
7. All user deposits inside `VaultContract` are permanently frozen with no recovery mechanism.
