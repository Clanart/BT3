### Title
Unprivileged Bootstrap Declare Bypasses All Validation, Enabling Permanent Class Registry Poisoning - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

### Summary
`execute_declare_transaction` contains a special "bootstrap" code path that completely bypasses signature validation, nonce enforcement, and fee charging for any transaction where `sender_address == 'BOOTSTRAP'`, `nonce == 0`, `version == 3`, and `max_fee == 0`. These conditions are fully attacker-controllable. An unprivileged user can exploit this to register any Sierra class hash with an arbitrary (incorrect) `compiled_class_hash`, permanently poisoning the class registry and freezing funds in any contract subsequently deployed under that class hash.

### Finding Description

In `execute_declare_transaction`, lines 764–776 implement a bootstrap shortcut:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

When this branch is taken, the function returns immediately, skipping:

- `check_and_increment_nonce` (line 779) — no nonce enforcement
- `non_reverting_select_execute_entry_point_func` for `__validate_declare__` (line 804) — no signature verification
- `charge_fee` (line 822) — no fee payment [2](#0-1) 

The `sender_address == 'BOOTSTRAP'` check compares against a Cairo short-string felt literal. `'BOOTSTRAP'` is simply the big-endian ASCII encoding `0x424F4F545354524150` — a plain felt value that any transaction sender can supply as their `sender_address`. There is no cryptographic binding between `sender_address` and the submitting party in this path, because the entire account-contract validation step is skipped.

The `class_hash` is validated to be a genuine Sierra class hash via `finalize_class_hash`: [3](#0-2) 

However, `compiled_class_hash` is only required to be non-zero (`assert_not_zero(compiled_class_hash)`). There is **no constraint** that `compiled_class_hash` corresponds to the actual CASM of the Sierra class. An attacker can supply any non-zero felt (e.g., `1`) as the `compiled_class_hash`.

The `dict_update` call uses `prev_value=0`, which is a Cairo-level constraint enforcing that a class may be declared **at most once**: [4](#0-3) 

Once an entry `(class_hash → wrong_compiled_class_hash)` is written, no subsequent transaction can overwrite it (the `prev_value=0` assertion would fail). The class registry entry is permanently corrupted.

### Impact Explanation

**Critical. Permanent freezing of funds / Direct loss of funds.**

1. An attacker front-runs a legitimate class declaration by submitting a bootstrap declare transaction for the same `class_hash` with `compiled_class_hash = <arbitrary non-zero value>`.
2. The OS writes `(class_hash → attacker_compiled_class_hash)` into `contract_class_changes`. The legitimate class can never be declared for that `class_hash` again.
3. Any contract subsequently deployed under that `class_hash` will reference the wrong `compiled_class_hash`. The OS will be unable to locate valid CASM for execution, permanently breaking the contract.
4. All funds held by or routed through such a contract are permanently frozen.

### Likelihood Explanation

**High.** The bootstrap path is unconditionally reachable by any transaction sender. The four required conditions (`sender_address == 'BOOTSTRAP'`, `nonce == 0`, `version == 3`, `max_fee == 0`) are all freely chosen by the transaction submitter with no privileged role required. The attack is a straightforward front-run of any pending class declaration visible in the mempool.

### Recommendation

Restrict the bootstrap declare path to a privileged address (e.g., the sequencer's own address or a protocol-controlled guardian address) verified against a value committed in `BlockContext` or `StarknetOsConfig`. Alternatively, remove the bootstrap path entirely and require all class declarations to go through the standard `__validate_declare__` flow. At minimum, add a Cairo-level `assert` that `sender_address` equals a specific protocol-controlled constant that cannot be chosen by an arbitrary user.

### Proof of Concept

1. Observe a pending declare transaction for Sierra class `C` with correct `compiled_class_hash = H_correct`.
2. Craft a declare transaction:
   - `sender_address = 'BOOTSTRAP'` (felt `0x424F4F545354524150`)
   - `nonce = 0`
   - `version = 3`
   - All resource bounds set to `0` (so `max_possible_fee == 0`)
   - `class_hash = class_hash(C)` (valid Sierra class hash, satisfies `finalize_class_hash` check)
   - `compiled_class_hash = 1` (non-zero, but wrong)
3. Submit this transaction to the sequencer before the legitimate one is included.
4. The OS executes the bootstrap branch at lines 764–776, writing `(class_hash(C) → 1)` into `contract_class_changes` with no signature check, no nonce check, and no fee.
5. The legitimate declare transaction now fails: `dict_update` with `prev_value=0` is inconsistent because the entry already exists.
6. Any contract deployed with `class_hash(C)` references `compiled_class_hash = 1`, which has no corresponding CASM. The contract is permanently broken and all funds within it are frozen.
