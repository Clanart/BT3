### Title
`add_fee_to_sequencer_balance` Overwrites Execute-Phase Sequencer Balance Writes in Concurrent Mode, Producing Wrong State Root - (File: `crates/blockifier/src/concurrency/fee_utils.rs`)

### Summary

In concurrent execution mode, when a transaction's `execute` phase transfers tokens directly to the sequencer address, the commit-time `add_fee_to_sequencer_balance` call reads the sequencer balance from a versioned-state view that excludes the current transaction's own writes. It then computes `new_balance = prior_balance + actual_fee` and unconditionally overwrites the sequencer balance entry in both `execution_output.state_diff` and the versioned state. The execute-phase transfer amount is silently erased. The resulting `CommitmentStateDiff` carries the wrong sequencer balance storage value, which propagates to the Patricia trie, the global state root, and the block commitment.

### Finding Description

**Concurrent fee-transfer design (background)**

During concurrent execution, `concurrency_execute_fee_transfer` deliberately sets the sequencer balance to `Felt::ZERO` in a nested transactional state, executes the ERC-20 transfer, then strips the sequencer balance writes before committing, so the sequencer balance is not touched during the parallel execution phase. [1](#0-0) 

At commit time, `commit_tx` calls `complete_fee_transfer_flow`, which reads the *current* sequencer balance from `tx_versioned_state = self.state.pin_version(tx_index)` and delegates to `add_fee_to_sequencer_balance`. [2](#0-1) 

**The versioned-state read excludes the current transaction's own writes**

`VersionedStateProxy::get_storage_at` calls `state.storage.read(self.tx_index, ...)`. The `VersionedState` documentation states that `get_writes_up_to_index` returns writes "up to the given transaction index (excluding)". A proxy pinned at `tx_index` therefore sees only writes from transactions `0 … tx_index-1`; it cannot see the writes that the current transaction committed to slot `tx_index` during `execute_tx`. [3](#0-2) [4](#0-3) 

**The overwrite**

`add_fee_to_sequencer_balance` receives `sequencer_balance = (prev_low, prev_high)` — the balance *before* the current transaction ran — and computes:

```
new_value_low = prev_low + actual_fee
```

It then unconditionally inserts this value into `state_diff.storage` for `sequencer_balance_key_low`, overwriting whatever was already there, and calls `state.apply_writes` to push the same value into the versioned state at `tx_index`. [5](#0-4) 

**The lost write**

If the transaction's `execute` phase transferred `TRANSFER_AMOUNT` tokens to the sequencer (a normal ERC-20 `transfer` call), the execute-phase write `sequencer_balance_key_low → prev_low + TRANSFER_AMOUNT` was already present in `execution_output.state_diff` and in the versioned state at `tx_index`. After `add_fee_to_sequencer_balance` runs, both are overwritten with `prev_low + actual_fee`. The `TRANSFER_AMOUNT` is gone.

The existing test `test_concurrency_execute_fee_transfer` (Case 2) confirms that after concurrent execution with a user-initiated transfer to the sequencer, the write in the outer state is `SEQUENCER_BALANCE_LOW_INITIAL + TRANSFER_AMOUNT` — exactly the value that `add_fee_to_sequencer_balance` will later erase. [6](#0-5) 

**Propagation to state root and block commitment**

The corrupted `execution_output.state_diff` (`StateMaps`) is converted to `CommitmentStateDiff` and then to `ThinStateDiff`, which is written to batcher storage and fed to the committer. The committer applies it to the Patricia trie and derives the `GlobalRoot`. [7](#0-6) [8](#0-7) 

The wrong storage value for the sequencer's fee-token balance slot produces a wrong leaf hash in the contracts trie, a wrong `GlobalRoot`, and therefore a wrong `state_diff_commitment` / `PartialBlockHash` in the final block commitment. [9](#0-8) 

### Impact Explanation

Any user can submit an invoke transaction whose `execute` phase calls `transfer(sequencer_address, amount)` on the fee token contract. When this transaction is processed in concurrent mode, the sequencer's fee-token balance in the committed state diff is `actual_fee` short of the correct value. The resulting state root is wrong. Every downstream consumer — the proof system (SNOS), L1 verifier, RPC `starknet_getStorageAt`, and state-sync peers — receives an incorrect authoritative storage value for the sequencer's balance slot. This matches the Critical impact: *"Wrong state … storage value … from blockifier/syscall/execution logic for accepted input."*

### Likelihood Explanation

Concurrent execution mode is the production path for high-throughput block building. The trigger is a standard ERC-20 `transfer` to the sequencer address inside an invoke transaction's execute phase — a completely valid, unprivileged operation. No special permissions or malicious infrastructure are required.

### Recommendation

In `complete_fee_transfer_flow`, before calling `add_fee_to_sequencer_balance`, check whether `execution_output.state_diff.storage` already contains a write for `sequencer_balance_key_low` (from the execute phase). If it does, use that value as the base for the fee addition rather than re-reading from the versioned state proxy (which cannot see the current transaction's own writes). Concretely:

```rust
// Prefer the already-accumulated execute-phase write as the base balance.
let base_low = state_diff.storage
    .get(&(fee_token_address, sequencer_balance_key_low))
    .copied()
    .unwrap_or(sequencer_balance.0);
// Then add actual_fee on top of base_low instead of sequencer_balance.0.
```

This mirrors the fix for the external `transferVesting` bug: account for the already-accumulated but not-yet-finalized portion before computing the new total.

### Proof of Concept

1. Deploy a standard ERC-20 fee-token contract (already present in the test environment).
2. Fund account `A` with enough STRK to pay fees.
3. Submit an invoke transaction from `A` whose calldata calls `fee_token.transfer(sequencer_address, TRANSFER_AMOUNT)` in its execute phase.
4. Enable concurrent execution (`ConcurrencyConfig { enabled: true, chunk_size: 2, .. }`).
5. After the block is built, read the sequencer's balance from the committed state diff.

**Expected:** `initial_sequencer_balance + TRANSFER_AMOUNT + actual_fee`

**Actual:** `initial_sequencer_balance + actual_fee`

The `TRANSFER_AMOUNT` is missing from the state diff, the Patricia trie leaf, and the global state root. The existing test infrastructure in `crates/blockifier/src/transaction/account_transactions_test.rs` (`test_concurrency_execute_fee_transfer` Case 2) already sets up the exact scenario; extending it to call `complete_fee_transfer_flow` and then verify the final state diff value will reproduce the discrepancy. [10](#0-9) [5](#0-4)

pub fn complete_fee_transfer_flow(
    tx_context: &TransactionContext,
    tx_execution_info: &mut TransactionExecutionInfo,
    state_diff: &mut StateMaps,
    state: &mut impl UpdatableState,
    tx: &Transaction,
) {
    if tx_context.is_sequencer_the_sender() {
        // When the sequencer is the sender, we use the sequential (full) fee transfer.
        return;
    }

    if let Some(fee_transfer_call_info) = tx_execution_info.fee_transfer_call_info.as_mut() {
        let sequencer_balance = state
        .get_fee_token_balance(
            tx_context.block_context.block_info.sequencer_address,
            tx_context.fee_token_address()
        )
        // TODO(barak, 01/07/2024): Consider propagating the error.
        .unwrap_or_else(|error| {
            panic!(
                "Access to storage failed. Probably due to a bug in Papyrus. {error:?}: {error}"
            )
        });

        // Fix the transfer call info.
        fill_sequencer_balance_reads(fee_transfer_call_info, sequencer_balance);
        // Update the balance.
        add_fee_to_sequencer_balance(
            tx_context.fee_token_address(),
            state,
            tx_execution_info.receipt.fee,
            &tx_context.block_context,
            sequencer_balance,
            tx_context.tx_info.sender_address(),
            state_diff,
        );
    } else {
        // Sanity check.
        match tx {
            Transaction::Account(tx) => assert!(
                !tx.execution_flags.charge_fee || tx_execution_info.receipt.fee == Fee(0),
                "Transaction with no fee transfer info must not enforce a fee charge."
            ),
            // No fee transfer info for L1 handler transactions.
            Transaction::L1Handler(_) => {}
        };
    }
}

pub fn add_fee_to_sequencer_balance(
    fee_token_address: ContractAddress,
    state: &mut impl UpdatableState,
    actual_fee: Fee,
    block_context: &BlockContext,
    sequencer_balance: (Felt, Felt),
    sender_address: ContractAddress,
    state_diff: &mut StateMaps,
) {
    assert_ne!(
        sender_address, block_context.block_info.sequencer_address,
        "The sender cannot be the sequencer."
    );
    let (low, high) = sequencer_balance;
    let sequencer_balance_low_as_u128 =
        low.to_u128().expect("sequencer balance low should be u128");
    let sequencer_balance_high_as_u128 =
        high.to_u128().expect("sequencer balance high should be u128");
    let (new_value_low, overflow_low) = sequencer_balance_low_as_u128.overflowing_add(actual_fee.0);
    let (new_value_high, overflow_high) =
        sequencer_balance_high_as_u128.overflowing_add(overflow_low.into());
    assert!(
        !overflow_high,
        "The sequencer balance overflowed when adding the fee. This should not happen."
    );
    let (sequencer_balance_key_low, sequencer_balance_key_high) =
        get_sequencer_balance_keys(block_context);
    let writes = StateMaps {
        storage: HashMap::from([
            ((fee_token_address, sequencer_balance_key_low), Felt::from(new_value_low)),
            ((fee_token_address, sequencer_balance_key_high), Felt::from(new_value_high)),
        ]),
        ..StateMaps::default()
    };

    // Modify state_diff to accurately reflect the post tx-execution state, after fee transfer to
    // the sequencer. We assume that a non-sequencer sender cannot reduce the sequencer's
    // balance—only increases are possible.

    if sequencer_balance_high_as_u128 != new_value_high {
        // Update the high balance only if it has changed.
        state_diff
            .storage
            .insert((fee_token_address, sequencer_balance_key_high), Felt::from(new_value_high));
    }

    if sequencer_balance_low_as_u128 != new_value_low {
        // Update the low balance only if it has changed.
        state_diff
            .storage
            .insert((fee_token_address, sequencer_balance_key_low), Felt::from(new_value_low));
    }
    state.apply_writes(&writes, &ContractClassMapping::default());
