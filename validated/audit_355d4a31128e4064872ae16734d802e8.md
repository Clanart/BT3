### Title
Bootstrap Declare Path Bypasses All Signature, Nonce, and Fee Validation — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `execute_declare_transaction` function in the StarkNet OS contains a special "bootstrap" code path that unconditionally skips signature verification (`__validate_declare__`), nonce enforcement, and fee charging. The only guard is a plain felt comparison: `sender_address == 'BOOTSTRAP'`. Because the OS never cryptographically proves that the submitter controls the address `'BOOTSTRAP'`, any unprivileged transaction sender can craft a declare transaction that satisfies all four conditions and have it accepted by the OS without any authorization.

---

### Finding Description

Inside `execute_declare_transaction`, after the transaction hash is computed and before the nonce is incremented, the following branch exists:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

When this branch is taken, the function returns immediately, skipping:

- `check_and_increment_nonce` — nonce verification and increment
- The `__validate_declare__` entry point call — the account's signature check
- `charge_fee` — fee deduction from the sender [2](#0-1) 

The four conditions that trigger the bypass are all attacker-controlled fields in the transaction:

| Field | Required value | Attacker control |
|---|---|---|
| `sender_address` | `'BOOTSTRAP'` (felt `0x424f4f545354524150`) | Yes — any user can set this field |
| `nonce` | `0` | Yes — freely chosen |
| `version` | `3` | Yes — freely chosen |
| all resource bounds | `0` (so `max_possible_fee == 0`) | Yes — freely chosen |

The `sender_address` field is loaded from the hint `%{ DeclareTxFields %}` and is included in the transaction hash computation. [3](#0-2) 

The transaction hash is computed and verified, but the hash itself only commits to the field values — it does **not** prove that the submitter controls the address `'BOOTSTRAP'`. That proof is exactly what `__validate_declare__` (the skipped step) is supposed to provide.

The `run_validate` function in `execute_transaction_utils.cairo` is the normal enforcement point for signature verification: [4](#0-3) 

The bootstrap path returns before this is ever called.

Additionally, the `compiled_class_hash` is accepted as-is from the hint without being verified to correspond to the `class_hash`. The only check on `compiled_class_hash` in the bootstrap path is `assert_not_zero(compiled_class_hash)`. [5](#0-4) 

---

### Impact Explanation

An attacker who crafts a declare transaction with `sender_address = 'BOOTSTRAP'`, `nonce = 0`, `version = 3`, and all resource bounds set to zero can:

1. **Declare any previously-undeclared class hash** without owning or controlling any account at address `'BOOTSTRAP'`.
2. **Pair that class hash with an arbitrary `compiled_class_hash`** — one that does not correspond to the declared Sierra class. The OS only checks `assert_not_zero(compiled_class_hash)`. [6](#0-5) 

A mismatched `compiled_class_hash` means that any contract subsequently deployed from that class hash will execute the CASM bytecode of the attacker's choosing rather than the bytecode that corresponds to the declared Sierra class. Contracts that hold or transfer funds and are deployed from a poisoned class hash will execute attacker-controlled logic, resulting in **direct loss of funds**.

Impact: **Critical — Direct loss of funds**.

---

### Likelihood Explanation

The four trigger conditions are all freely settable by any transaction submitter. No privileged key, leaked secret, or Sybil attack is required. The attacker only needs to submit a well-formed declare transaction with the four specific field values. The OS Cairo code enforces no cryptographic ownership proof over the address `'BOOTSTRAP'` in this path. Any sequencer that does not independently re-validate signatures (or a malicious sequencer) will include such a transaction, and the resulting OS proof will be accepted by the L1 verifier.

---

### Recommendation

Remove the bootstrap special-case path entirely, or gate it behind a cryptographic check that cannot be satisfied by an arbitrary transaction sender. Concretely:

- If bootstrap declarations are needed for system initialization, they should be handled outside the transaction execution path (e.g., as a pre-block OS initialization step that is not reachable via user-submitted transactions).
- If the path must remain, require that `sender_address == 'BOOTSTRAP'` is backed by a valid signature from a known, hard-coded public key, rather than a plain felt comparison.
- At minimum, `compiled_class_hash` must be verified to correspond to `class_hash` even in the bootstrap path, to prevent class poisoning.

---

### Proof of Concept

1. Construct a V3 declare transaction with:
   - `sender_address = 0x424f4f545354524150` (felt encoding of `'BOOTSTRAP'`)
   - `nonce = 0`
   - `version = 3`
   - `resource_bounds` for L1\_GAS, L2\_GAS, L1\_DATA all set to `max_amount = 0`, `max_price_per_unit = 0`
   - `class_hash` = any valid Sierra class hash (passes `finalize_class_hash` check)
   - `compiled_class_hash` = hash of attacker-controlled CASM (any nonzero value)

2. Compute the transaction hash over these fields (no signature is needed or checked).

3. Submit the transaction. The OS reaches the branch at line 764 of `transaction_impls.cairo`, satisfies all four conditions, and executes the `dict_update` that registers `compiled_class_hash` under `class_hash` — skipping `check_and_increment_nonce`, `run_validate`, and `charge_fee`.

4. Deploy a contract from the poisoned `class_hash`. The contract executes the attacker's CASM, not the Sierra source it purports to implement. [1](#0-0)

func run_validate{
    range_check_ptr,
    remaining_gas: felt,
    builtin_ptrs: BuiltinPointers*,
    contract_state_changes: DictAccess*,

    outputs: OsCarriedOutputs*,

    alloc_locals;
    local tx_execution_info: ExecutionInfo* = tx_execution_context.execution_info;

    // Do not run "__validate__" for version 0.
    if (tx_execution_info.tx_info.version == 0) {

    // "__validate__" is expected to get the same calldata as "__execute__".
    local validate_execution_context: ExecutionContext* = new ExecutionContext(

        class_hash=tx_execution_context.class_hash,
        calldata_size=tx_execution_context.calldata_size,
