### Title
Integer Overflow in Relative Timelock Arithmetic Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Enforcement — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `check_time_locks` is called with `nowrap=false` (the pre-soft-fork legacy path), relative timelock conditions `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE` are evaluated using `wrapping_add` instead of `saturating_add`. An attacker who controls the CLVM puzzle output can craft a `height_relative` or `seconds_relative` value that causes the addition to overflow and wrap to a small integer, making the guard comparison evaluate to `false` and silently returning `Ok(())` — i.e., the timelock is accepted as satisfied when it is not.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` branches on the `nowrap` flag for every relative timelock check:

```rust
// lines 55-68
if let Some(height_relative) = spend.height_relative {
    if nowrap {
        if prev_transaction_block_height
            < unspent.confirmed_block_index.saturating_add(height_relative)
        {
            return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
        }
    } else if prev_transaction_block_height
        < unspent.confirmed_block_index.wrapping_add(height_relative)  // ← overflow path
    {
        return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
    }
}
``` [1](#0-0) 

When `nowrap=false`, the expression `unspent.confirmed_block_index.wrapping_add(height_relative)` silently wraps on overflow. For example, if `confirmed_block_index = 100` and `height_relative = 0xFFFF_FFFF` (u32::MAX), the result wraps to `99`. The guard then checks `prev_height < 99`, which is `false` for any `prev_height ≥ 99`, so the function returns `Ok(())` — the timelock is bypassed entirely.

The same pattern applies to `ASSERT_SECONDS_RELATIVE` with u64 arithmetic:

```rust
// lines 70-77
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [2](#0-1) 

The test suite explicitly documents and confirms this divergence:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),   // ← nowrap=false silently passes
)]
``` [3](#0-2) 

The function is exported as a Python binding `py_check_time_locks` and called by the Chia full node with a caller-supplied `nowrap` value: [4](#0-3) 

The `nowrap=false` path is the pre-activation legacy branch kept for backward compatibility with older blocks. Any block or spend bundle validated against that path is subject to the bypass.

---

### Impact Explanation

**High — Timelock condition validation bypass enables unauthorized spend acceptance.**

A coin protected by `ASSERT_HEIGHT_RELATIVE N` (or `ASSERT_SECONDS_RELATIVE N`) is intended to be unspendable until `N` blocks (or seconds) after it was confirmed. With the wrapping bug, an attacker who controls the puzzle output can choose a `height_relative` value such that `confirmed_block_index + height_relative` overflows u32, wrapping to a value smaller than the current block height. The timelock check then passes immediately, allowing the coin to be spent at any block height — effectively nullifying the timelock entirely.

This maps directly to the allowed High impact: *"timelock … validation bypass enables unauthorized spend acceptance."*

---

### Likelihood Explanation

- The `nowrap=false` path is reachable via the exported Python binding `py_check_time_locks` whenever the full node processes blocks or mempool transactions under the pre-soft-fork rules.
- The attacker input is fully unprivileged: the `height_relative` value comes from the CLVM puzzle output, which any coin owner controls.
- The overflow value is trivially computable: `height_relative = u32::MAX - confirmed_block_index + 1` causes wrap to 0, bypassing any height check.
- No special keys, governance roles, or network-level access are required.

---

### Recommendation

Replace all `wrapping_add` calls in `check_time_locks` with `saturating_add`, unconditionally. The `nowrap` flag was introduced to preserve legacy behavior, but the wrapping path produces incorrect consensus semantics. The `saturating_add` path is the correct behavior in all cases: a `height_relative` so large that it overflows should be treated as an unsatisfiable constraint (i.e., the coin can never be spent under that condition), not as a trivially satisfied one.

Concretely, remove the `nowrap` branch entirely and always use:

```rust
if prev_transaction_block_height
    < unspent.confirmed_block_index.saturating_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
``` [5](#0-4) 

Apply the same fix to `seconds_relative`, `before_height_relative`, and `before_seconds_relative`.

---

### Proof of Concept

**Setup:** A coin confirmed at block height 100 (`confirmed_block_index = 100`). The puzzle outputs:

```
ASSERT_HEIGHT_RELATIVE 0xFFFFFF96   ; = u32::MAX - 100 + 1 = 4294967196 + 1 = wraps to 0
```

**Computation (nowrap=false):**
```
100u32.wrapping_add(0xFFFFFF97) = 100 + 4294967191 = 4294967291 mod 2^32 = ... 
```

More precisely, choose `height_relative = 0xFFFFFF9C` so that `100 + 0xFFFFFF9C = 0x1_0000_0000 mod 2^32 = 0`. Then:

```
prev_height < 0   →   false   →   Ok(())   ← timelock bypassed
```

The coin is spendable at block height 101 (or any height ≥ 0), despite the puzzle asserting a relative lock of ~4.3 billion blocks.

The test case `height_relative_wrap` in the test suite already confirms this exact outcome: [6](#0-5)
