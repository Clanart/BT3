### Title
Tokens Minted to Recipient Then Immediately Diverted to `_systemAddress` in `HyperliquedBridgeToken` 3-arg `mint` When `message` Is Non-Empty — (File: `evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

---

### Summary

The 3-arg `mint` function in `HyperliquedBridgeToken` mints tokens to the bridge recipient and then immediately transfers the entire minted amount to `_systemAddress`, leaving the recipient with zero tokens. `OmniBridge.finTransfer` selects the 3-arg variant whenever `payload.message.length != 0`. Because the `message` field originates from the user-controlled `msg` parameter of the originating `initTransfer`, any cross-chain transfer to a `HyperliquedBridgeToken` that carries a non-empty message will silently divert the full bridged amount away from the intended recipient.

---

### Finding Description

**Root cause — `HlBridgeToken.sol` lines 76–83:**

```solidity
function mint(
    address account,
    uint256 value,
    bytes memory
) external override onlyOwner {
    _mint(account, value);
    _update(account, _systemAddress, value);   // ← immediately moves all tokens away
}
```

`_mint(account, value)` correctly credits `account` with `value` tokens (emitting `Transfer(address(0) → account)`). The very next call `_update(account, _systemAddress, value)` transfers the entire balance back out to `_systemAddress` (emitting `Transfer(account → _systemAddress)`). The net balance change for `account` is **zero**.

**Dispatch path — `OmniBridge.sol` lines 337–349:**

```solidity
} else if (isBridgeToken[payload.tokenAddress]) {
    if (payload.message.length == 0) {
        IBridgeToken(payload.tokenAddress).mint(
            payload.recipient, payload.amount
        );
    } else {
        IBridgeToken(payload.tokenAddress).mint(   // ← 3-arg path
            payload.recipient, payload.amount, payload.message
        );
    }
}
```

The selector between the 2-arg (safe) and 3-arg (destructive) path is solely `payload.message.length != 0`. There is no check that the token is a `HyperliquedBridgeToken`, no check that the message encodes a valid HyperCore action, and no check that the recipient is a HyperCore user.

**Message origin — `near/omni-bridge/src/lib.rs` lines 487–499:**

```rust

    .and_then(|s| s.destination_msg())
    .unwrap_or_default();

let transfer_payload = TransferMessagePayload {
    ...

```

The `message` bytes in the MPC-signed payload are derived from the user-supplied `msg` string in the original `ft_transfer_call` / `initTransfer`. Any non-empty application-level message (DeFi routing, memo, callback data) propagates directly into the EVM `finTransfer` payload and triggers the 3-arg mint.

---

### Impact Explanation

Every cross-chain transfer to a `HyperliquedBridgeToken` that carries a non-empty `message` results in the recipient receiving **zero tokens**. The full bridged amount is instead credited to `_systemAddress` (the HyperLiquid system address). There is no on-chain mechanism for the original recipient to reclaim those tokens; recovery would require out-of-band intervention by HyperLiquid operators. This constitutes balance/accounting corruption that misdirects value and breaks bridge collateralization for the affected token.

---

### Likelihood Explanation

The condition is reachable by any unprivileged bridge user:

- A user bridging tokens from NEAR/Solana to a HyperEVM `HyperliquedBridgeToken` with any non-empty `msg` field (e.g., a DeFi protocol that always attaches routing data, a wallet that includes a memo, or a deliberate griefing attempt against a third-party recipient) will trigger the 3-arg path.
- The `HyperliquedBridgeToken` is registered as a standard `isBridgeToken`, so `finTransfer` reaches the vulnerable branch through the normal bridge flow.
- No privileged role or special condition is required beyond initiating a normal bridge transfer with a non-empty message.

---

### Recommendation

1. **Decouple HyperCore routing from the ERC-20 `mint` interface.** The 3-arg `mint` override should not be the mechanism for parking tokens at `_systemAddress`. Instead, `finTransfer` should always call the 2-arg mint (crediting the recipient), and a separate, explicitly-gated function should handle HyperCore-specific token parking.

2. **Guard the 3-arg mint.** If the 3-arg variant must exist, add a check inside `HyperliquedBridgeToken.mint` that validates the `bytes` argument encodes a recognized HyperCore action before diverting tokens, and revert otherwise.

3. **Alternatively**, override `IBridgeToken` in `HyperliquedBridgeToken` so that the 3-arg `mint` simply ignores the `bytes` argument and behaves identically to the 2-arg version, relying solely on `coreReceiveWithData` for the HyperCore parking flow.

---

### Proof of Concept

1. A NEAR user calls `ft_transfer_call` on the NEAR bridge with `msg = '{"InitTransfer":{"recipient":"eth:0xRecipient","fee":"0","native_token_fee":"0","msg":"0x01"}}'` — any non-empty `msg` suffices.
2. `sign_transfer` on NEAR produces a signed `TransferMessagePayload` where `message = [0x01]` (non-empty).
3. A relayer calls `OmniBridge.finTransfer(signatureData, payload)` on HyperEVM where `payload.tokenAddress` is a `HyperliquedBridgeToken` and `payload.message = [0x01]`.
4. Signature verification passes (the payload was legitimately signed by the NEAR MPC).
5. `isBridgeToken[payload.tokenAddress]` is `true`; `payload.message.length == 1 != 0` → 3-arg `mint` is called.
6. Inside `HyperliquedBridgeToken.mint(0xRecipient, amount, [0x01])`:
   - `_mint(0xRecipient, amount)` → `0xRecipient` balance = `amount`.
   - `_update(0xRecipient, _systemAddress, amount)` → `0xRecipient` balance = `0`; `_systemAddress` balance += `amount`.
7. `0xRecipient` receives zero tokens. The bridged amount is permanently held by `_systemAddress`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
