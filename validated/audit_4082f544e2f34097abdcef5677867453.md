### Title
ETH and ERC20/ERC1155 Tokens Can Be Permanently Locked in OmniBridge Without Recovery Mechanism — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.sol` exposes a bare `receive() external payable {}` function and accumulates non-bridge ERC20 and ERC1155 tokens as part of normal bridge operation, yet contains no `sweep()` or fund-recovery function. Any ETH or tokens that end up in the contract outside the expected bridge flow — through accidental sends, airdrops, or mistaken returns — are permanently irrecoverable without an emergency contract upgrade.

---

### Finding Description

`OmniBridge.sol` can accumulate assets in three distinct ways:

**1. ETH via `receive()`** [1](#0-0) 

The bare `receive()` at line 574 accepts any ETH sent to the contract. ETH is also legitimately held when `finTransfer` is called for native ETH transfers (`payload.tokenAddress == address(0)`), meaning the contract is expected to hold an ETH balance.

**2. Non-bridge ERC20 tokens locked in `initTransfer`** [2](#0-1) 

When a user bridges a non-bridge-issued ERC20 (i.e., not a `BridgeToken` and not a `customMinter` token), the full `amount` is pulled into `address(this)`. These tokens are the collateral backing the bridge.

**3. ERC1155 tokens locked in `initTransfer1155`** [3](#0-2) 

ERC1155 tokens are similarly transferred into `address(this)` and held as bridge collateral.

**No recovery function exists anywhere in the contract or its subclass:** [4](#0-3) 

Neither `OmniBridge` nor `OmniBridgeWormhole` defines any `sweep()`, `rescueTokens()`, `emergencyWithdraw()`, or equivalent admin-callable recovery function. A full-text search across `evm/src/**/*.sol` confirms zero matches for any such pattern.

---

### Impact Explanation

Any of the following realistic events causes funds to be permanently locked with no on-chain recovery path:

- A user or protocol accidentally sends ETH directly to the contract address (e.g., copy-paste error, wallet auto-send).
- A token issuer airdrops ERC20 tokens to all holders of a bridged token; the bridge contract itself is a holder and receives the airdrop.
- A recipient of a bridged transfer recognizes a mistake and returns tokens directly to the contract address instead of initiating a new bridge transfer.
- An ERC1155 token issuer airdrops additional token IDs to the contract.

In all cases, the only recovery path is an emergency UUPS upgrade to inject a sweep function — a high-risk, time-sensitive admin operation that may not be feasible under incident conditions. Until such an upgrade is deployed and executed, the funds are irrecoverable.

This matches **Critical: Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

Moderate. The `receive()` function is intentionally present to support the ETH bridge flow, making the contract a well-known ETH-holding address. Airdrops to contracts holding bridged tokens are a documented real-world occurrence (e.g., UNI airdrop to Uniswap LP contracts, staking contract airdrops). Mistaken direct-sends to bridge contract addresses are common in production. The contract is deployed on mainnet EVM chains where these events are routine.

---

### Recommendation

Add an admin-gated `sweep()` function to `OmniBridge.sol` that recovers ETH, arbitrary ERC20 tokens, and ERC1155 tokens to a designated admin address or multisig, while explicitly preventing withdrawal of tokens that are part of the active bridge collateral (i.e., tokens tracked in `ethToNearToken`). Example pattern:

```solidity
function sweep(address token, uint256 amount, address to)
    external
    onlyRole(DEFAULT_ADMIN_ROLE)
{
    if (token == address(0)) {
        (bool ok,) = to.call{value: amount}("");
        require(ok);
    } else {
        IERC20(token).safeTransfer(to, amount);
    }
}
```

For ERC1155, a corresponding `sweepERC1155` variant should be added.

---

### Proof of Concept

1. Deploy `OmniBridge` (or `OmniBridgeWormhole`) on a local fork.
2. Send 1 ETH directly to the contract address from any EOA — accepted silently by `receive()`.
3. Attempt to call any function to recover the ETH — none exists; the ETH is stuck.
4. Separately, call `initTransfer` with a non-bridge ERC20 token. The token is pulled into `address(this)` at lines 407–410.
5. Simulate an airdrop: transfer additional units of that ERC20 directly to the contract address.
6. Confirm the airdropped surplus has no recovery path — `finTransfer` will only release exactly the amount recorded in the cross-chain message, leaving the surplus permanently locked. [5](#0-4)
