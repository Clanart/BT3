### Title
Unhandled `.expect()` panic in `resolve_query` on malformed ETH-to-STRK oracle price string silently falls back to stale/default gas prices embedded in block hash - (File: crates/apollo_l1_gas_price/src/eth_to_strk_oracle.rs)

---

### Summary

`resolve_query` in `eth_to_strk_oracle.rs` calls `.expect()` to parse the `price` field from the external ETH-to-STRK oracle HTTP response. If the oracle returns a syntactically valid JSON body whose `price` value is not a valid hexadecimal string (e.g., an empty string, a decimal number, or a string containing non-hex characters), the code panics instead of returning an `Err`. The panic is caught by the `tokio::spawn` task boundary and surfaces as a `JoinError`, which the caller treats as an oracle failure. The fallback path in `get_l1_prices_in_fri_and_wei_and_conversion_rate` then silently substitutes stale previous-block prices or hardcoded default prices (`min_l1_gas_price_wei` / `DEFAULT_ETH_TO_FRI_RATE`) into the block header. Because gas prices are a committed field in the block hash, every block produced under this condition carries a wrong, authoritative-looking gas-price commitment.

---

### Finding Description

**Root cause — line 166-167 of `eth_to_strk_oracle.rs`:**

```rust
let rate = u128::from_str_radix(price.trim_start_matches("0x"), 16)
    .expect("Failed to parse price as u128");   // panics on any non-hex price
``` [1](#0-0) 

The function correctly guards against a missing `price` field (returning `MissingFieldError`) and against a wrong `decimals` value (returning `InvalidDecimalsError`), but it does **not** guard against a `price` field that is present and is a string yet is not a valid hex number. Examples that trigger the panic:

- `{"price": "", "decimals": 18}` — empty string → `ParseIntError { kind: Empty }`
- `{"price": "12345", "decimals": 18}` — decimal string → `ParseIntError { kind: InvalidDigit }` (digit `1` is valid hex but `2` is not after `0x` strip; actually all decimal digits are valid hex, but `"12345"` without `0x` prefix would parse fine — however `"1.5e3"` or `"NaN"` would not)
- `{"price": "0xGGGG", "decimals": 18}` — invalid hex chars → `ParseIntError { kind: InvalidDigit }`
- `{"price": "0x", "decimals": 18}` — empty hex body → `ParseIntError { kind: Empty }`

The panic fires inside the async block passed to `tokio::spawn` in `spawn_query`: [2](#0-1) 

Rust's tokio runtime catches the panic and the `JoinHandle` resolves to `Err(JoinError { is_panic: true })`. The `eth_to_fri_rate` method handles this in the `Err(e)` arm: [3](#0-2) 

This propagates as `EthToStrkOracleClientError::JoinError` up to `get_l1_prices_in_fri_and_wei_and_conversion_rate`, which treats it as an oracle failure and falls through to the fallback paths: [4](#0-3) 

**Fallback path 1** reuses the previous block's FRI and WEI prices (stale). **Fallback path 2** (no previous block info) uses `gas_price_params.min_l1_gas_price_wei` and `DEFAULT_ETH_TO_FRI_RATE` — hardcoded defaults that may be far from current market prices.

**Secondary issue — zero rate accepted without validation:**

`resolve_query` also does not reject a rate of `0`. If the oracle returns `{"price": "0x0", "decimals": 18}`, `resolve_query` returns `Ok(0)`. In `get_l1_prices_in_fri_and_wei_and_conversion_rate`, `L1PricesInFri::convert_from_wei(&prices_in_wei, 0)` calls `GasPrice::wei_to_fri(0)`: [5](#0-4) 

`checked_mul_u128(0)` returns `Some(GasPrice(0))` (no overflow), and `checked_div(WEI_PER_ETH)` returns `Some(GasPrice(0))`, so the conversion succeeds with zero FRI prices. The function returns `(zero_fri_prices, wei_prices, 0)` **without triggering the fallback**, bypassing the safety net entirely. The zero FRI prices then reach `convert_to_sn_api_block_info`: [6](#0-5) 

`NonzeroGasPrice::new(GasPrice(0))` returns `Err(ZeroGasPrice)`, causing block building to abort for the proposer — a liveness failure.

---

### Impact Explanation

**Wrong gas prices committed to block hash (High — incorrect fee/gas with economic impact):**

Gas prices (`l1_gas_price`, `l1_data_gas_price`, `l2_gas_price` in both FRI and WEI) are chained into the block hash via `PartialBlockHashComponents`: [7](#0-6) 

When the oracle fails due to the panic, the proposer embeds stale or default prices into the block header. Every transaction in that block is charged fees computed against wrong gas prices. RPC fee-estimation calls (`starknet_estimateFee`, `starknet_estimateMessageFee`) return authoritative-looking values derived from these wrong prices. The block hash itself is computed over the wrong gas-price fields, so the committed state is permanently incorrect for that block.

**Proposer liveness failure (zero-rate path):**

A zero rate bypasses the fallback and causes `convert_to_sn_api_block_info` to return `Err(ZeroGasPrice)`, aborting block construction. The proposer cannot produce a valid proposal, stalling consensus for its turn.

---

### Likelihood Explanation

The external ETH-to-STRK oracle is a third-party HTTP service. API format changes, partial outages, CDN-level response mangling, or a provider returning a decimal string instead of a hex string are all realistic non-adversarial triggers. The CoinGecko analog in the seed report was fixed precisely because such inconsistencies are observed in production. No privileged access is required; the oracle endpoint is queried automatically on every block proposal.

---

### Recommendation

1. Replace `.expect()` with proper error propagation in `resolve_query`:

```rust
let rate = u128::from_str_radix(price.trim_start_matches("0x"), 16)
    .map_err(|e| EthToStrkOracleClientError::ParseError(
        format!("Failed to parse price '{}' as u128: {e}", price)
    ))?;
```

2. Validate that the parsed rate is non-zero before returning:

```rust
if rate == 0 {
    return Err(EthToStrkOracleClientError::ParseError(
        "Oracle returned a zero ETH-to-STRK rate".to_string()
    ));
}
```

With these changes, both the malformed-string and zero-rate cases return a typed `Err`, which the existing fallback logic in `get_l1_prices_in_fri_and_wei_and_conversion_rate` already handles correctly.

### Proof of Concept

1. Stand up a mock oracle that returns `{"price": "0x", "decimals": 18}` (empty hex body).
2. The sequencer's `EthToStrkOracleClient` calls `spawn_query`, which calls `resolve_query`.
3. `u128::from_str_radix("", 16).expect(...)` panics inside the spawned task.
4. The tokio runtime catches the panic; `JoinHandle` resolves to `Err(JoinError { is_panic: true })`.
5. `eth_to_fri_rate` returns `Err(EthToStrkOracleClientError::JoinError(...))`.
6. `get_l1_prices_in_fri_and_wei_and_conversion_rate` falls through to fallback path 2 (no previous block on first block) and returns `(DEFAULT_ETH_TO_FRI_RATE-derived prices, min_l1_gas_price_wei, DEFAULT_ETH_TO_FRI_RATE)`.
7. The proposer builds a block with these default prices embedded in `PartialBlockHashComponents`.
8. `calculate_block_hash` commits the wrong gas prices into the block hash.
9. All fee estimates and transaction charges for that block are computed against the wrong prices. [8](#0-7) [9](#0-8) [10](#0-9)

fn resolve_query(body: String) -> Result<u128, EthToStrkOracleClientError> {
    let Ok(json): Result<serde_json::Value, _> = serde_json::from_str(&body) else {
        return Err(EthToStrkOracleClientError::ParseError(format!(
            "Failed to parse JSON: {body}"
        )));
    };
    // Extract price from API response. Also returns MissingFieldError if value is not a string.
    let price = match json.get("price").and_then(|v| v.as_str()) {
        Some(price) => price,
        None => {
            return Err(EthToStrkOracleClientError::MissingFieldError("price".to_string(), body));
        }
    };
    let rate = u128::from_str_radix(price.trim_start_matches("0x"), 16)
        .expect("Failed to parse price as u128");
    // Extract decimals from API response. Also returns MissingFieldError if value is not a number.
    let decimals = match json.get("decimals").and_then(|v| v.as_u64()) {
        Some(decimals) => decimals,
        None => {
            return Err(EthToStrkOracleClientError::MissingFieldError(
                "decimals".to_string(),
                body,
            ));
        }
    };
    if decimals != ETH_TO_STRK_QUANTIZATION {
        return Err(EthToStrkOracleClientError::InvalidDecimalsError(
            ETH_TO_STRK_QUANTIZATION,
            decimals,
        ));
    }
    ETH_TO_STRK_SUCCESS_COUNT.increment(1);
    set_unix_now_seconds(&ETH_TO_STRK_LAST_SUCCESS_TIMESTAMP_SECONDS);
    ETH_TO_STRK_RATE.set_lossy(rate);
    Ok(rate)
}

    pub fn wei_to_fri(self, eth_to_fri_rate: u128) -> Result<GasPrice, StarknetApiError> {
        // We use integer division since wei * eth_to_fri_rate is expected to be high enough to not
        // cause too much precision loss.
        Ok(self
            .checked_mul_u128(eth_to_fri_rate)
            .ok_or_else(|| {
                StarknetApiError::GasPriceConversionError(format!(
                    "Gas price is too high: {:?}, eth to fri rate: {:?}",
                    self, eth_to_fri_rate
                ))
            })?
            .checked_div(WEI_PER_ETH)
            .expect("WEI_PER_ETH must be non-zero"))
