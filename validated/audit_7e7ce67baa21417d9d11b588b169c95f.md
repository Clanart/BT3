### Title
Native Fee ETH Paid by Bridge Users Is Permanently Locked in `OmniBridge` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
Every user who calls `initTransfer` or `initTransfer1155` with a non-zero `nativeFee` sends ETH to the `OmniBridge` contract. The `nativeFee` portion of `msg.value` is retained in the contract and never forwarded to any relayer or fee recipient. There is no withdrawal function in `OmniBridge.sol` or `OmniBridgeWormhole.sol` to recover this ETH, causing it to be permanently locked.

### Finding Description
In `OmniBridge.initTransfer`, the caller sends `msg.value` covering the bridged `amount` (if native ETH), the `nativeFee`, and the Wormhole message fee (`extensionValue`):

```
extensionValue = msg.value - nativeFee          // ERC20 path
extensionValue = msg.value - amount - nativeFee // native ETH path
```

Only `extensionValue` is forwarded to Wormhole via `_wormhole.publishMessage{value: value}(...)` in `OmniBridgeWormhole.initTransferExtension`. The `nativeFee` portion stays in the contract. [1](#0-0) [2](#0-1) 

The same pattern applies to `initTransfer1155`: [3](#0-2) 

The entire `OmniBridge.sol` contains no function to withdraw accumulated ETH. The only ETH-related outflow is in `finTransfer` when `tokenAddress == address(0)`, which releases the bridged `amount` to the recipient — not the accumulated fees: [4](#0-3) 

The contract has a bare `receive()` function that accepts ETH but provides no withdrawal path: [5](#0-4) 

There is no `withdrawNativeFee`, `rescueETH`, or any equivalent function anywhere in `OmniBridge.sol` or `OmniBridgeWormhole.sol`. [6](#0-5) 

### Impact Explanation
Every `nativeFee` payment made by any bridge user on the EVM side is permanently locked in the contract. The ETH is irrecoverable — neither the user, the relayer, nor the admin can retrieve it. This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

The `nativeFee` is the mechanism by which users pay relayers to process their cross-chain transfers on the destination chain. Locking this ETH means relayers are never compensated from the EVM-side fee, and the ETH is lost to the protocol permanently.

### Likelihood Explanation
This is triggered by any unprivileged bridge user who calls `initTransfer` or `initTransfer1155` with `nativeFee > 0`. This is a standard, documented user-facing flow (the README explicitly describes `nativeFee` as a supported fee payment method). Every such call permanently locks ETH. Likelihood is **high** — it affects every user who pays a native fee on the EVM side.

### Recommendation
Add a privileged withdrawal function to allow the admin or a designated fee recipient to recover accumulated native fees:

```solidity
function withdrawNativeFees(address payable recipient) external onlyRole(DEFAULT_ADMIN_ROLE) {
    uint256 balance = address(this).balance;
    (bool success, ) = recipient.call{value: balance}("");
    if (!success) revert FailedToSendEther();
}
```

Alternatively, forward the `nativeFee` directly to a designated relayer/fee-recipient address at the time of `initTransfer`, rather than retaining it in the contract.

### Proof of Concept
1. User calls `OmniBridgeWormhole.initTransfer(tokenAddress, amount, fee, nativeFee=0.01 ether, recipient, message)` with `msg.value = 0.01 ether` (Wormhole fee = 0, so `extensionValue = 0`).
2. Inside `initTransfer`: `extensionValue = msg.value - nativeFee = 0`.
3. `initTransferExtension` is called with `value = 0`; `_wormhole.publishMessage{value: 0}(...)` is called.
4. The `0.01 ether` `nativeFee` remains in `address(this).balance`.
5. No function exists to withdraw it. The ETH is permanently locked.
6. Repeat for every user who pays a `nativeFee` — the contract balance grows monotonically with no recovery path.
