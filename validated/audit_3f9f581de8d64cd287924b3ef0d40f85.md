### Title
Invariant Violation: v0 `ProofOfSpace` with Both Pool Fields Absent Parses Successfully but Panics on `compute_plot_id()` — (`crates/chia-protocol/src/proof_of_space.rs`)

### Summary
The `parse()` implementation for v0 `ProofOfSpace` accepts both `pool_public_key=None` and `pool_contract_puzzle_hash=None` without error, but `compute_plot_id()` → `compute_plot_id_v1()` unconditionally panics when both are `None`. An attacker can craft a serialized v0 `ProofOfSpace` with both pool fields absent, have it accepted by `parse()`, and then trigger a process-aborting panic when any code path calls `compute_plot_id()` on it.

### Finding Description
In `ProofOfSpace::parse()`: [1](#0-0) 

The v0 branch constructs and returns `Ok(ProofOfSpace {...})` with both `pool_public_key` and `pool_contract_puzzle_hash` as `None`, with no validation. The v1 branch at lines 322–323 has the required guard, but v0 does not. [2](#0-1) 

When `compute_plot_id()` is subsequently called on this struct: [3](#0-2) 

It delegates to `compute_plot_id_v1()`, which panics: [4](#0-3) 

The test suite at line 539 explicitly acknowledges the lenient v0 parse accepts both-None: [5](#0-4) 

The Python binding `py_compute_plot_id` directly exposes this panic path across the pyo3 FFI boundary: [6](#0-5) 

The standalone `compute_plot_id_v1` Python binding in `wheel/src/api.rs` has the same issue: [7](#0-6) 

### Impact Explanation
A Rust `panic!` crossing the pyo3 FFI boundary causes process abort in CPython (undefined behavior per the Rust FFI rules). The Chia full node calls `compute_plot_id()` during block validation to derive the plot ID used in proof-of-space verification. An attacker who submits a block containing a crafted v0 `ProofOfSpace` with both pool fields absent will cause every validating node that processes it to abort. If propagated to enough peers before detection, this constitutes a network-wide chain halt.

### Likelihood Explanation
The attack requires only crafting a valid-looking block with a malformed `ProofOfSpace` — no keys, no privileged access, no chain reorg. The serialization format is public and the lenient v0 parse path is confirmed by the test suite. The attacker needs only to get the block into the validation pipeline of target nodes.

### Recommendation
Add the same mutual-exclusion guard to the v0 parse branch that already exists for v1:

```rust
if version == 0 {
    // ...
    if pool_public_key.is_none() && pool_contract_puzzle_hash.is_none() {
        return Err(Error::InvalidPoS);
    }
    // ...
}
```

Alternatively, change `compute_plot_id_v1` (and `compute_plot_group_id_v2`) to return `Result<Bytes32>` instead of panicking, so the invariant violation is surfaced as an error rather than an abort.

### Proof of Concept
```rust
use chia_protocol::{ProofOfSpace, Streamable};
use std::io::Cursor;

// Serialize a v0 ProofOfSpace with both pool fields absent
// pool_public_key Option prefix: 0x00 (None)
// pool_contract prefix byte: 0x00 (version=0, contract absent)
// Then plot_public_key (48 bytes), size (1 byte), proof (length-prefixed)
let pos = ProofOfSpace::new(
    Bytes32::default(),
    None,  // pool_public_key
    None,  // pool_contract_puzzle_hash
    plot_pk(),
    0,     // version = 0
    0, 0, 0, 32,
    Bytes::from(vec![0u8; 32]),
);
let buf = pos.to_bytes().unwrap();

// parse() returns Ok — no error
let parsed = ProofOfSpace::parse::<false>(&mut Cursor::new(&buf)).unwrap();
assert!(parsed.pool_public_key.is_none());
assert!(parsed.pool_contract_puzzle_hash.is_none());

// compute_plot_id() panics unconditionally → process abort via pyo3
let _id = parsed.compute_plot_id(); // PANIC: "invalid proof of space. Neither pool pk nor contract puzzle hash set"
```
