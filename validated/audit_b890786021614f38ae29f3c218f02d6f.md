### Title
Unconstrained `is_reverted` Hint Allows Malicious Prover to Bypass Invoke Transaction Execution While Charging Fee — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is a complete no-op. The `is_reverted` flag that gates whether `__execute__` runs in `execute_invoke_function_transaction` is loaded from an unconstrained hint. A malicious sequencer/prover can set `is_reverted = 1` for any invoke transaction, causing the execution step to be silently skipped while the fee is still charged and committed to state.

---

### Finding Description

In `execute_invoke_function_transaction`, the decision of whether to run the `__execute__` entry point is controlled by a hint-provided local variable:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    with remaining_gas {
        cap_remaining_gas(max_gas=EXECUTE_MAX_SIERRA_GAS);
        non_reverting_select_execute_entry_point_func(...);
    }
} else {
    // Execution is entirely skipped.
    tempvar _dummy_return_value: non_reverting_select_execute_entry_point_func.Return;
}
// Fee is charged regardless.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [1](#0-0) 

The function `check_is_reverted` is supposed to validate this flag. Its actual implementation is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

It performs **zero validation**. In Cairo, hints are not part of the proof — they are non-deterministic inputs provided by the prover. Because `check_is_reverted` imposes no constraints on `is_reverted`, the prover is free to set it to any value without invalidating the proof.

By contrast, when execution *does* run, the inner `is_reverted` flag returned by `execute_entry_point` is cryptographically constrained — it comes from `entry_point_return_values.failure_flag`, which is the actual return value of the contract call: [3](#0-2) 

The outer hint-provided `is_reverted` in `execute_invoke_function_transaction` has no such constraint. This is the missing enforcement — the analog to the zAuction "user flow specified but not implemented" pattern.

---

### Impact Explanation

**Critical — Direct loss of funds.**

When a malicious prover sets `is_reverted = 1`:

1. The `__execute__` entry point is never called. No state changes from the user's intended operation (token transfers, contract interactions, etc.) are applied.
2. `charge_fee` is still called unconditionally after the branch. The actual fee is loaded from the hint `%{ LoadActualFee %}` and constrained only by `assert_nn_le(calldata.amount.low, max_fee)`. [4](#0-3) 

3. The ERC20 transfer to the sequencer is executed via `non_reverting_select_execute_entry_point_func`, modifying the fee token contract's storage. This state change is included in the state diff and committed to L1.
4. The proof is valid and accepted by the L1 verifier.

The user's fee is permanently transferred to the sequencer, but their intended operation never executes. For time-sensitive operations (liquidations, loan repayments, DEX trades), the indirect loss can far exceed the fee itself.

---

### Likelihood Explanation

The sequencer controls all hints in the Cairo OS program execution. A malicious or compromised sequencer can selectively apply this bypass to any subset of invoke transactions — targeting specific users or contracts — without any on-chain evidence of wrongdoing, since the proof is valid. The attack requires no special privileges beyond operating as a sequencer, and leaves no distinguishable trace in the proof output.

---

### Recommendation

Implement `check_is_reverted` to actually constrain the `is_reverted` value. The correct approach is to always execute the transaction and derive `is_reverted` from the actual execution result (as is done inside `execute_entry_point`), rather than accepting it as an unconstrained hint. If the optimization of skipping known-reverted transactions is desired, the hint must be constrained against the actual execution outcome — for example, by running the execution and asserting that the returned `failure_flag` matches the hint.

---

### Proof of Concept

1. User submits an invoke transaction (e.g., `transfer(recipient, amount)` on an ERC20 contract).
2. Malicious sequencer includes the transaction in a block.
3. During OS execution, the sequencer sets `is_reverted = 1` in the `%{ IsReverted %}` hint for this transaction.
4. `check_is_reverted(1)` is called — it returns immediately with no assertions.
5. The `if (is_reverted == FALSE)` branch is not taken; `non_reverting_select_execute_entry_point_func` (the `__execute__` call) is skipped entirely.
6. `charge_fee` executes unconditionally, performing an ERC20 transfer of `low_actual_fee` tokens from the user to the sequencer.
7. The state diff includes the fee deduction but not the user's intended transfer.
8. The proof is generated and verified on L1. The L1 verifier accepts it — the proof is cryptographically valid.
9. The user's fee is lost; the token transfer never occurred. [2](#0-1) [5](#0-4) [6](#0-5)
