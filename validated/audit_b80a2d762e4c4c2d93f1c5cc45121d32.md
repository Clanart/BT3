### Title
Missing Duplicate-Key Validation in `batch_insert` Corrupts DataLayer Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` skips the per-key uniqueness check that `MerkleBlob::insert` enforces, allowing duplicate `KeyId` values to be written directly into the blob. This silently corrupts the Merkle tree structure and produces an incorrect root hash, enabling forged inclusion/exclusion proofs.

### Finding Description

`MerkleBlob::insert` guards against duplicate keys at the very start:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a different path. When the tree already has two or more leaves (`leaf_count > 1`), it skips the `self.insert()` call entirely and writes every item in the batch directly via `insert_entry_to_blob`, with no duplicate-key check at all:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { key, value, hash, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // no KeyAlreadyPresent guard
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

Even when the tree starts with 0–1 leaves, only the last two items popped from the vector go through `self.insert()`. All earlier items in the vector still bypass the check: [3](#0-2) 

Because `insert_entry_to_blob` writes raw blocks without updating `block_status_cache`, a duplicate key causes the cache and the blob to diverge: the cache holds one index for the key while the blob contains two leaf nodes with the same `KeyId`. The subsequent subtree-stitching pass (`insert_subtree_at_key`) then computes internal-node hashes over this malformed subtree, producing a wrong root hash. [4](#0-3) 

The fuzz targets for `MerkleBlob` only exercise `insert` (which correctly rejects `KeyAlreadyPresent`) and never exercise `batch_insert` with duplicate keys, so this path is untested: [5](#0-4) 

### Impact Explanation

A corrupted root hash means that any proof-of-inclusion or proof-of-exclusion generated from the tree after a duplicate-key batch insert will be cryptographically invalid relative to the true committed state. Consumers of the DataLayer who rely on these proofs to verify key membership will accept forged proofs or reject valid ones, constituting corrupted tree roots and forged inclusion/exclusion — matching the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`batch_insert` is called from delta synchronization logic (`deltas.rs`) and proof-of-inclusion construction (`proof_of_inclusion.rs`), both of which process data received from peers: [6](#0-5) 

A malicious or buggy peer that sends a delta containing duplicate keys will trigger the silent corruption path. No special privilege is required beyond the ability to submit a DataLayer delta.

### Recommendation

Add a duplicate-key check at the top of `batch_insert`, mirroring the guard in `insert`:

```rust
pub fn batch_insert(
    &mut self,
    mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
) -> Result<(), Error> {
    // Validate uniqueness across the entire batch and against existing keys
    for ((key, _), hash) in &keys_values_hashes {
        if self.block_status_cache.contains_key(*key) {
            return Err(Error::KeyAlreadyPresent());
        }
        if self.block_status_cache.contains_leaf_hash(hash) {
            return Err(Error::HashAlreadyPresent());
        }
    }
    // ... rest of function
}
```

Alternatively, deduplicate the input vector before processing, or call `self.insert()` for every item (accepting the performance trade-off).

### Proof of Concept

```rust
let mut blob = MerkleBlob::new(vec![]).unwrap();

// Seed the tree with 2 leaves so batch_insert skips the insert() path entirely
blob.insert(KeyId(0), ValueId(0), &sha256_num(&0i64), InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(1), ValueId(1), &sha256_num(&1i64), InsertLocation::Auto {}).unwrap();

// batch_insert with a duplicate key — no error is returned
let batch = vec![
    ((KeyId(2), ValueId(2)), sha256_num(&2i64)),
    ((KeyId(2), ValueId(99)), sha256_num(&99i64)),  // duplicate KeyId(2)
];
blob.batch_insert(batch).unwrap();  // succeeds silently

// Tree is now corrupt: two leaf nodes share KeyId(2)
// check_integrity() will fail or the root hash is wrong
blob.check_integrity().unwrap_err(); // demonstrates corruption
``` [7](#0-6)

        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {

        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {

            let min_height_leaf = self.get_min_height_leaf()?;
