### Title
Missing Duplicate-Key Check in `batch_insert` Silently Corrupts DataLayer Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key guard that `MerkleBlob::insert` enforces, allowing an attacker-controlled call with repeated or already-present keys to write multiple leaf nodes sharing the same `KeyId` into the blob. The resulting tree commits an incorrect root hash, and any proof of inclusion or exclusion derived from it is invalid.

### Finding Description

`MerkleBlob::insert` (the single-item path) guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` only calls `self.insert()` (with its duplicate guard) for the last two items popped from the input vector when the tree has ≤ 1 existing leaves. All remaining items — and **all** items when the tree already has ≥ 2 leaves — are written directly via `insert_entry_to_blob` with no key-existence check:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ..., node: Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which uses `HashMap::insert` — silently overwriting the cache entry for a key that already exists:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
``` [3](#0-2) 

The result: the raw blob contains **two** `LeafNode` blocks with the same `KeyId`, but the `block_status_cache` only tracks the last one. The tree's internal-node hashes are computed over both leaves, so the committed root hash reflects a state that is inconsistent with the cache and with any valid key-value mapping.

The Python binding `py_batch_insert` exposes this path directly to callers:

```rust
pub fn py_batch_insert(
    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,
) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [4](#0-3) 

`check_integrity` would detect the corruption (the leaf count in the blob exceeds the cache length, and the first duplicate leaf's index does not match the cache), but `check_integrity_on_drop` is only enabled in test builds (`cfg!(test)`), so production trees are never automatically validated: [5](#0-4) 

### Impact Explanation

A corrupted `MerkleBlob` produces a root hash that does not correspond to any valid key-value set. Any `ProofOfInclusion` generated from the corrupted tree will either fail to verify against the committed root or will "prove" the wrong value for a key. This directly satisfies the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

### Likelihood Explanation

`batch_insert` is the primary bulk-load API for the DataLayer and is called whenever a DataLayer store is populated or updated in bulk. Any caller — including a Python application layer that feeds user-supplied key-value pairs — can trigger the bug by supplying a batch with a repeated key or a key already present in the tree. No special privilege is required beyond the ability to call `batch_insert` or `py_batch_insert`.

### Recommendation

Add a duplicate check at the start of the bulk loop in `batch_insert`, mirroring the guard in `insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing direct-insert logic
}
```

Alternatively, pre-deduplicate the input vector and check it against the existing cache before any blob writes begin. Also consider enabling `check_integrity` assertions in production (or at least after every `batch_insert`) to catch any future regressions.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(blob=bytearray())

# Pre-populate with 2 leaves so leaf_count > 1 (bypasses the insert() path entirely)
blob.insert(KeyId(0), ValueId(0), hashlib.sha256(b"h0").digest())
blob.insert(KeyId(1), ValueId(1), hashlib.sha256(b"h1").digest())

# Now batch_insert with a duplicate key (KeyId(0) already exists)
# and a duplicate within the batch itself (KeyId(2) appears twice)
blob.batch_insert(
    [(KeyId(0), ValueId(99)), (KeyId(2), ValueId(2)), (KeyId(2), ValueId(3))],
    [hashlib.sha256(b"hX").digest(),
     hashlib.sha256(b"h2").digest(),
     hashlib.sha256(b"h3").digest()],
)
# No error is raised. The blob now contains duplicate leaf nodes for KeyId(0) and KeyId(2).
# The root hash is corrupted; proofs derived from it are invalid.
# check_integrity() (only active in test builds) would raise an error here.
```

The single-item path correctly rejects duplicates: [6](#0-5) 

But no equivalent test exists for `batch_insert` with duplicate or pre-existing keys, confirming the gap. [7](#0-6)
