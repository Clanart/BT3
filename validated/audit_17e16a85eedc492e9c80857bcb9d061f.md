### Title
Permanent Fund Lock via Blacklisted Recipient in `finTransfer()` — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.finTransfer()` transfers tokens directly to `payload.recipient` without any fallback. If the recipient is blacklisted by the token contract (e.g., USDC) or implements a reverting ERC777 `tokensReceived` hook, every finalization attempt reverts. Because the origin chain has already locked or burned the user's tokens and no rescue mechanism exists, the funds are permanently frozen.

---

### Finding Description

In `OmniBridge.sol::finTransfer()`, the function first marks the destination nonce as used, verifies the MPC signature, then dispatches the token transfer to `payload.recipient`:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287
// ... signature verification ...
IERC20(payload.tokenAddress).safeTransfer(             // line 351-354
    payload.recipient,
    payload.amount
);
``` [1](#0-0) [2](#0-1) 

If `safeTransfer` reverts (USDC blacklist, ERC777 hook, etc.), the entire transaction reverts — including the `completedTransfers` write — so the nonce is never permanently consumed. The relayer can retry, but every attempt will revert for the same reason.

The same pattern exists in the StarkNet bridge:

```cairo
_set_transfer_finalised(ref self, payload.destination_nonce);  // line 250
// ...
let success = IERC20Dispatcher { contract_address: payload.token_address }
    .transfer(payload.recipient, payload.amount.into());
assert(success, 'ERR_TRANSFER_FAILED');                        // line 262
``` [3](#0-2) 

On the origin chain (NEAR), once `init_transfer` is accepted, the user's tokens are locked or burned and a `TransferMessage` is stored. There is no public cancel or refund entry point in the NEAR bridge contract. [4](#0-3) 

The ERC1155 path in `finTransfer` has the same exposure: `safeTransferFrom` invokes `onERC1155Received` on the recipient, which a malicious contract can cause to revert. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

- Origin chain: tokens are locked in the NEAR locker or burned from the deployed token supply. This state is irreversible without an explicit cancel/refund path, which does not exist.
- Destination chain: `finTransfer` will revert on every call for that transfer, so the nonce is never consumed and the tokens are never released.
- The user loses the full bridged amount with no recovery path.

---

### Likelihood Explanation

**Medium.**

- USDC maintains an active on-chain blacklist. A user can be blacklisted between the time they initiate a bridge transfer and the time the relayer finalizes it on EVM — a window that can span minutes to hours.
- An adversary who controls a USDC-blacklisted address can deliberately trigger this to permanently destroy their own bridged funds (griefing) or, more critically, to demonstrate the absence of a recovery path.
- ERC777 tokens with a reverting `tokensReceived` hook present the same vector for any token whitelisted as a native (non-bridge) token.
- No privileged access is required; any unprivileged bridge user who initiates a transfer is exposed.

---

### Recommendation

1. **Pull-over-push pattern**: Instead of pushing tokens to `payload.recipient` inside `finTransfer`, credit the amount to a per-recipient claimable balance mapping and let the recipient pull funds in a separate call. This decouples finalization from delivery.
2. **Try/catch with escrow fallback**: Wrap the token transfer in a low-level call; on failure, escrow the funds in the contract and emit an event so the recipient (or an admin) can trigger a retry or redirect.
3. **Recipient override by admin**: Allow a privileged role to redirect escrowed funds to an alternate address when the original recipient is permanently unreachable, with appropriate time-lock and event logging.

---

### Proof of Concept

1. Alice initiates a USDC bridge transfer from NEAR to EVM, specifying her EVM address as recipient. The NEAR bridge locks her USDC equivalent and stores the `TransferMessage`.
2. The MPC network signs the transfer payload.
3. Before the relayer calls `finTransfer`, Alice's EVM address is added to the USDC blacklist (either by Circle or as part of a targeted attack).
4. The relayer calls `OmniBridge.finTransfer(signatureData, payload)`.
5. Execution reaches `IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount)` — this reverts because Alice is blacklisted.
6. The entire transaction reverts; `completedTransfers[nonce]` is rolled back to `false`.
7. The relayer retries — same revert every time.
8. Alice's USDC on NEAR remains locked forever. There is no cancel, no refund, and no admin override in the NEAR bridge contract. Alice's funds are permanently frozen.
