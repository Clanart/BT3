### Title
Gas Refund Overcrediting on `ErrStateConflict` Commit Failure — (File: `x/evm/keeper/state_transition.go`)

---

### Summary

In `ApplyMessageWithConfig`, when `stateDB.Commit()` returns `ErrStateConflict`, the function returns an `EVMResult` whose `GasUsed` field has already been reduced by EVM gas refunds (from SSTORE operations). Because the stateDB was never committed, those storage operations never actually occurred on-chain. `ApplyTransaction` then calls `RefundGas` using `leftoverGas = msg.GasLimit - res.GasUsed`, which is inflated by the uncommitted refunds, causing the fee collector to send more tokens back to the sender than were legitimately earned.

---

### Finding Description

**Step 1 — EVM gas refund is computed and baked into `gasUsed` before commit.**

In `ApplyMessageWithConfig`, after EVM execution completes:

```
temporaryGasUsed := msg.GasLimit - leftoverGas
refund := GasToRefund(stateDB.GetRefund(), temporaryGasUsed, refundQuotient)
leftoverGas += refund
``` [1](#0-0) 

Then `gasUsed` is finalised:

```
gasUsed = max(minimumGasUsed, tempGasUsed)   // tempGasUsed = msg.GasLimit - leftoverGas (post-refund)
leftoverGas = msg.GasLimit - gasUsed
``` [2](#0-1) 

At this point `gasUsed` is already reduced by the EVM SSTORE refund.

**Step 2 — `stateDB.Commit()` fails with `ErrStateConflict`; the function returns the already-reduced `gasUsed`.**

```go
if errors.Is(err, statedb.ErrStateConflict) {
    return &types.EVMResult{
        GasUsed: gasUsed,          // ← post-refund value
        VmError: statedb.ErrStateConflict.Error(),
        ...
    }, nil
}
``` [3](#0-2) 

`ErrStateConflict` is defined as the error returned when an EVM-dirty storage key was also written by a nested native action via `ExecuteNativeAction`: [4](#0-3) 

Because `Commit()` returned an error, the stateDB changes — including the SSTORE operations that generated the refund — were **never written to the KVStore**.

**Step 3 — `ApplyTransaction` calls `RefundGas` with the inflated leftover gas.**

```go
leftoverGas := msg.GasLimit - res.GasUsed   // inflated by uncommitted SSTORE refund
if err = k.RefundGas(ctx, msg, leftoverGas, cfg.Params.EvmDenom); err != nil { ... }
``` [5](#0-4) 

`RefundGas` transfers `leftoverGas × gasPrice` tokens from the fee collector module account to the sender:

```go
err := k.bankKeeper.SendCoinsFromModuleToAccountVirtual(
    ctx, authtypes.FeeCollectorName, msg.From.Bytes(), refundedCoins)
``` [6](#0-5) 

The sender receives a real token transfer for gas "saved" by storage operations that never happened.

---

### Impact Explanation

This is a fee mis-accounting bug matching the **High** allowed impact: *"EVM state transition, gas refund, fee market, ante handler, mempool, or proposal handling bug that permits … valid user funds/fees to be mis-accounted."*

The ante handler deducts `gasLimit × effectiveGasPrice` from the sender. The correct refund for a failed transaction should be `(gasLimit − actualGasUsed) × gasPrice`, where `actualGasUsed` does **not** include refunds from uncommitted SSTORE operations. Instead, the code refunds `(gasLimit − (actualGasUsed − uncommittedRefund)) × gasPrice`, overpaying the sender by up to `uncommittedRefund × gasPrice`. After EIP-3529 the refund is capped at `gasUsed / 5`, so the maximum per-transaction overcrediting is `gasUsed / 5 × gasPrice` — a direct, repeatable drain of the fee collector.

---

### Likelihood Explanation

`ErrStateConflict` is triggered when the EVM writes to a storage slot that a nested native action (precompile using `ExecuteNativeAction`) also writes to in the same transaction. Any chain that exposes a precompile writing to EVM-addressable storage slots gives an unprivileged user a reachable path:

1. Identify a precompile that writes to a predictable EVM storage slot `S` via `ExecuteNativeAction`.
2. Craft a transaction that first clears slot `S` via `SSTORE S 0` (generating a refund), then calls that precompile (which writes to `S`, causing the conflict).
3. `stateDB.Commit()` returns `ErrStateConflict`; the SSTORE clear is discarded but the refund is already baked into `gasUsed`.
4. `RefundGas` overcredits the sender.

The path is reachable through a normal, unprivileged `MsgEthereumTx` submission.

---

### Recommendation

In the `ErrStateConflict` branch of `ApplyMessageWithConfig`, return the **pre-refund** `gasUsed` (i.e., `temporaryGasUsed` before `leftoverGas += refund` is applied), so that `RefundGas` in `ApplyTransaction` does not credit the sender for uncommitted SSTORE operations. Alternatively, recompute `leftoverGas` without the EVM refund component before returning the `EVMResult` in the conflict path.

---

### Proof of Concept

1. Attacker identifies precompile `P` that calls `ExecuteNativeAction` and writes to EVM storage slot `S` of contract `C`.
2. Attacker pre-populates slot `S` of `C` with a non-zero value (one-time setup).
3. Attacker sends `MsgEthereumTx` calling `C`:
   - `SSTORE S 0` — clears slot `S`; EVM adds `~15,000` gas to the refund counter (EIP-3529 capped at `gasUsed/5`).
   - Calls precompile `P` — native action writes to slot `S` via `ExecuteNativeAction`.
4. `ApplyMessageWithConfig` computes `refund = GasToRefund(stateDB.GetRefund(), ...)`, reduces `gasUsed` accordingly, then calls `stateDB.Commit()`.
5. `Commit()` detects the EVM-dirty / native-dirty conflict on slot `S` and returns `ErrStateConflict`.
6. `ApplyMessageWithConfig` returns `EVMResult{GasUsed: gasUsed}` with the already-reduced value; the SSTORE clear is never written to the KVStore.
7. `ApplyTransaction` computes `leftoverGas = msg.GasLimit − res.GasUsed` (inflated) and calls `RefundGas`, which transfers the excess tokens from the fee collector to the attacker.
8. Attacker repeats steps 2–7 (re-populating slot `S` each iteration) to continuously drain the fee collector.
