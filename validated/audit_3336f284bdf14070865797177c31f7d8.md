### Title
Missing Class Declaration Validation in `execute_replace_class` Allows Permanent Contract Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by a contract corresponds to a previously declared contract class. This is an exact structural analog to the SapienVault missing-expiration-check bug: in both cases, a state-mutating operation proceeds without validating a prerequisite condition on the new value, allowing an invalid state to be committed on-chain. In the StarkNet OS, the result is that a contract's class hash can be set to an arbitrary, undeclared value, rendering the contract permanently uncallable and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the function `execute_replace_class` (lines 878–916) handles the `replace_class` syscall. It reads the requested new class hash from the syscall request and directly writes it into `contract_state_changes` with no check that the hash corresponds to a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The developer-acknowledged TODO at line 898 explicitly confirms the missing check:

> `// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.` [2](#0-1) 

The `execute_replace_class` function is reached from the syscall dispatch path inside `execute_syscalls.cairo` and `execute_syscalls__virtual.cairo`, which are called during normal contract execution. The syscall is available to any contract running on the network.

The parallel to the SapienVault bug is direct:

| SapienVault | StarkNet OS |
|---|---|
| `stake()` / `increaseAmount()` add tokens without checking lockup expiry | `replace_class` replaces class hash without checking class is declared |
| Stake immediately unlocked; user keeps multiplier benefit | Contract immediately uncallable; state committed with invalid class |
| Missing: `weightedStartTime + effectiveLockUpPeriod < block.timestamp` check | Missing: `class_hash ∈ contract_class_changes` check |

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once the OS commits a block containing a `replace_class` call with an undeclared class hash, the contract's `StateEntry.class_hash` is permanently set to that invalid value in the Merkle-Patricia state tree. Every subsequent transaction targeting that contract will fail at class lookup, because no Sierra or Casm bytecode exists for the hash. There is no recovery path: the contract cannot call `replace_class` again (execution is impossible), and no external party can restore the class hash. All ERC-20 balances, ETH, or other assets held in the contract's storage are permanently inaccessible.

---

### Likelihood Explanation

The attack surface is reachable by any unprivileged contract deployer or by any user who can trigger a `replace_class` syscall in a target contract (e.g., through a governance mechanism, an upgradeable proxy, or a reentrancy path). The OS provides no guard, so the sequencer will include such a transaction, the prover will prove it, and the L1 verifier will accept the resulting state root. No privileged role or leaked key is required.

---

### Recommendation

Before writing the new class hash to `contract_state_changes`, assert that the hash exists as a key in `contract_class_changes` (i.e., it was declared in the current block or in a prior committed block). The check should mirror the pattern used in `execute_declare_transaction`, where `dict_update` with `prev_value=0` enforces prior non-existence:

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

If the class hash is not declared, the syscall should write a failure response (analogous to `ERROR_INVALID_ARGUMENT`) and return without modifying state.

---

### Proof of Concept

1. Attacker deploys contract `C` holding user funds (e.g., an upgradeable vault).
2. Attacker submits an invoke transaction that calls `replace_class` on `C` with `class_hash = 0xdeadbeef` (any value not present in `contract_class_changes`).
3. `execute_replace_class` in `syscall_impls.cairo` (line 896–913) reads `class_hash = 0xdeadbeef`, skips the missing declaration check, and writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
4. The OS produces a valid proof; the L1 verifier accepts the new state root.
5. All subsequent calls to `C` fail at class dispatch — no bytecode exists for `0xdeadbeef`.
6. All funds in `C`'s storage are permanently frozen with no recovery path. [3](#0-2)
