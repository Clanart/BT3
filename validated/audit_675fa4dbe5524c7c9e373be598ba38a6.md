### Title
Integer Overflow in Relative Timelock Arithmetic Causes Consensus Divergence and Timelock Bypass — (`File: crates/chia-consensus/src/check_time_locks.rs`)

### Summary

`check_time_locks` computes relative timelock thresholds using either `wrapping_add` or `saturating_add` depending on the `nowrap` boolean flag. When `nowrap = false`, a large attacker-controlled `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` value causes the addition to wrap around to a small integer, making the comparison pass when it should fail. This lets a spend bundle bypass its own relative timelock on nodes running with `nowrap = false`, while nodes running with `nowrap = true` correctly reject it — producing deterministic consensus divergence.

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` enforces four relative timelock conditions per spend. For each condition it computes a threshold by adding the coin's confirmed height/timestamp to the condition value, then compares that threshold against the current block height/timestamp.

When `nowrap = false`, the addition uses `wrapping_add`:

```rust
// ASSERT_HEIGHT_RELATIVE — nowrap=false path
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

If `height_relative` is chosen so that `confirmed_block_index + height_relative` overflows `u32`, the result wraps to a small value. The comparison `prev_height < small_value` then evaluates to `false`, so the spend is **accepted** even though the coin has not been confirmed for the required number of blocks.

The same overflow applies to `seconds_relative` (u64), `before_height_relative` (u32), and `before_seconds_relative` (u64), each producing the opposite wrong outcome depending on the direction of the comparison (`<` vs `>=`).

The in-tree tests explicitly document and confirm the divergence:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),  // nowrap=true
    Ok(()),                                                           // nowrap=false
)]
```

The `nowrap` flag is exposed directly through the Python binding `py_check_time_locks`, meaning callers at the Python layer control which arithmetic mode is used.

### Impact Explanation

**Timelock bypass (High):** On a node running with `nowrap = false`, a spend bundle containing `ASSERT_HEIGHT_RELATIVE = u32::MAX - confirmed_block_index + 1` (or any value that causes wrapping) is accepted immediately, bypassing the intended relative height lock. The coin can be spent far earlier than the puzzle author intended.

**Consensus divergence (Critical):** Nodes running `nowrap = true` reject the same spend bundle. Any block that includes such a spend is valid to `nowrap = false` nodes and invalid to `nowrap = true` nodes, causing a chain split.

### Likelihood Explanation

The `nowrap` parameter is an unprivileged caller-controlled input exposed through the Python binding. An attacker only needs to craft a CLVM puzzle that emits a relative timelock condition with a value near the integer maximum. The exact overflow threshold is trivially computable from the coin's `confirmed_block_index` (a public on-chain value). No special privileges, keys, or governance access are required.

### Recommendation

Replace `wrapping_add` with `saturating_add` unconditionally for all four relative timelock comparisons, or remove the `nowrap = false` code path entirely if the wrapping behavior is no longer required for backward compatibility. If the flag must be retained for a transition period, document the exact block height at which `nowrap` becomes permanently `true` and enforce it in consensus rather than leaving it as a caller-supplied parameter.

### Proof of Concept

Coin confirmed at block index `100`. Attacker crafts a puzzle that outputs:

```
(ASSERT_HEIGHT_RELATIVE . 0xFFFFFF96)   ; 0xFFFFFF96 = u32::MAX - 100 + 1 = 4294967142
```

- `nowrap = true` (saturating): `100 + 4294967142` saturates to `u32::MAX = 4294967295`. `prev_height (200) < 4294967295` → **Err** (correctly rejected).
- `nowrap = false` (wrapping): `100 + 4294967142 = 4294967242`, which overflows u32 to `(4294967242 % 2^32) = 4294967242 - 4294967296 = -54`, i.e. wraps to `4294967242 mod 2^32 = 4294967242 - 4294967296`... let me recalculate: `100 + (2^32 - 100 + 1) = 2^32 + 1`, which wraps to `1`. `prev_height (200) < 1` → **Ok** (timelock bypassed).

    bundle_conds: &OwnedSpendBundleConditions,

    timestamp: u64,
    nowrap: bool,

        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,
        nowrap,
    );

        Ok(()) => Ok(None),
        Err(ec) => Ok(Some(ec.error_code().into())),
    }
}
