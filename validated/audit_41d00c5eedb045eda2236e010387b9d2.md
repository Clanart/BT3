### Title
Tautological Final Check in `ProofOfInclusion::valid()` Allows Forged DataLayer Merkle Proofs to Pass Validation - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary

`ProofOfInclusion::valid()` performs a final check that is always true when the loop completes without returning `false`. The check compares `existing_hash` against `self.root_hash()`, but `self.root_hash()` is derived directly from the last layer's `combined_hash`, which was already verified equal to `existing_hash` inside the loop. This makes the final check tautological. As a result, any attacker-supplied `ProofOfInclusion` that is internally self-consistent will pass `valid()` regardless of what tree root it claims to prove, enabling forged DataLayer inclusion proofs.

### Finding Description

`ProofOfInclusion::valid()` is the sole validation method on the `ProofOfInclusion` struct exposed to Python and Rust callers:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // ← existing_hash = layer.combined_hash
    }

    existing_hash == self.root_hash()      // ← always true: compares last combined_hash to itself
}
``` [1](#0-0) 

After the loop, `existing_hash` holds the last `calculated_hash`, which was already asserted equal to `layer.combined_hash` on the iteration before `existing_hash` was updated. `self.root_hash()` returns exactly `self.layers.last().combined_hash`:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // ← same value as existing_hash after the loop
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

The final comparison `existing_hash == self.root_hash()` is therefore comparing a value against itself. It can never be `false` when the loop exits normally. The function only verifies internal self-consistency of the proof's own fields; it never checks the computed root against any external trusted root hash.

The `ProofOfInclusion` struct is `Streamable` and exposed to Python via `from_py_object`, meaning untrusted callers can construct arbitrary instances: [3](#0-2) 

The Python binding exposes `valid()` directly: [4](#0-3) 

### Impact Explanation

An attacker can craft a `ProofOfInclusion` with an arbitrary `node_hash` (the leaf they wish to falsely prove is included) and an arbitrary chain of `ProofOfInclusionLayer` values, as long as each layer satisfies `calculate_internal_hash(prev, side, other) == combined_hash`. This is trivially achievable by choosing any `other_hash` and computing the corresponding `combined_hash`. The resulting proof will pass `valid()` with no reference to any real tree root.

This allows an untrusted party to prove that any key-value pair is present in a DataLayer store when it is not, or to prove exclusion of a key that is actually present. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

### Likelihood Explanation

The `ProofOfInclusion` type is `Streamable` and constructible from Python (`from_py_object`). Any caller that receives a `ProofOfInclusion` from an untrusted source and calls only `proof.valid()` — the documented and sole validation method — is vulnerable. The fuzz target and all tests call only `proof.valid()` without checking `root_hash()` against an external trusted root, confirming this is the expected usage pattern. [5](#0-4) 

### Recommendation

Replace the tautological final check with a comparison against a caller-supplied trusted root hash. The `valid()` method should accept the expected root as a parameter:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }

    &existing_hash == trusted_root   // ← compare against external trusted root
}
```

Alternatively, keep `valid()` as an internal-consistency check but clearly document that callers **must** separately verify `proof.root_hash() == trusted_root` obtained from a trusted source (e.g., the on-chain committed root). Update all call sites accordingly.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer, Side
import hashlib

# Arbitrary leaf we want to falsely prove is included
fake_node_hash = bytes([0xAA] * 32)
other_hash     = bytes([0xBB] * 32)

# Compute combined_hash so the layer is internally consistent
h = hashlib.sha256()
h.update(b"\x02")          # internal node prefix used by calculate_internal_hash
h.update(fake_node_hash)   # left child
h.update(other_hash)       # right child
combined = h.digest()

layer = ProofOfInclusionLayer(
    other_hash_side=Side.Right,
    other_hash=other_hash,
    combined_hash=combined,
)

forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True even though this proof was never generated from any real tree
assert forged_proof.valid(), "Forged proof incorrectly rejected"
# root_hash() is attacker-controlled
assert forged_proof.root_hash() == combined
```

The forged proof passes `valid()` because the final check `existing_hash == self.root_hash()` compares the last `calculated_hash` against `layers[-1].combined_hash`, which is the same value — a comparison of a variable against itself.
