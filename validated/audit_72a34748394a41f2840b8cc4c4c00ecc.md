### Title
Migration v5 Reads Boolean Params via `store.Has()` Instead of Value Unmarshal, Permanently Corrupting `AllowUnprotectedTxs` to `true` — (File: `x/evm/migrations/v5/migrate.go`)

---

### Summary

The v5 store migration reads three boolean parameters (`EnableCreate`, `EnableCall`, `AllowUnprotectedTxs`) using `store.Has()` rather than fetching and unmarshalling the stored byte value. `store.Has()` returns `true` whenever the key exists in the KV store — regardless of whether the stored value encodes `true` or `false`. Because the Cosmos SDK params subspace writes every parameter key at genesis (including those with value `false`), all three keys are present. The migration therefore unconditionally sets all three booleans to `true`, overwriting the actual on-chain governance value. For `AllowUnprotectedTxs`, whose default is `false`, this silently enables acceptance of pre-EIP-155 (unprotected) transactions on every chain that ran this upgrade path, permanently until corrected by a subsequent governance proposal.

---

### Finding Description

In `x/evm/migrations/v5/migrate.go`, lines 34–36:

```go
params.EnableCreate        = store.Has(v0types.ParamStoreKeyEnableCreate)
params.EnableCall          = store.Has(v0types.ParamStoreKeyEnableCall)
params.AllowUnprotectedTxs = store.Has(v0types.ParamStoreKeyAllowUnprotectedTxs)
```

The Cosmos SDK params subspace persists every parameter key at genesis initialisation, encoding the value (including `false`) as its amino/protobuf bytes. `store.Has(key)` only tests key existence; it does not decode the stored byte slice. Consequently, a key whose stored value is the amino encoding of `false` (a non-empty byte slice) still causes `store.Has()` to return `true`.

The correct pattern — used for the non-boolean params in the same migration — is to call `store.Get(key)` and unmarshal the result:

```go
chainCfgBz := store.Get(v0types.ParamStoreKeyChainConfig)
cdc.MustUnmarshal(chainCfgBz, &chainConfig)          // correct
```

The boolean params skip this step entirely.

The critical consequence is `AllowUnprotectedTxs`. Its default is `false` (`DefaultAllowUnprotectedTxs = false` in `x/evm/types/params.go` line 33). Any chain that ran with the default (or with governance explicitly setting it to `false`) will have the key present in the store with a `false`-encoded value. After the v5 migration executes, the consolidated `KeyPrefixParams` blob is written back with `AllowUnprotectedTxs = true`, and the old individual keys are deleted (lines 42–47). The corruption is now the sole authoritative state. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

`AllowUnprotectedTxs = true` instructs the ante handler to accept Ethereum transactions that carry no EIP-155 chain-ID replay protection. An attacker can:

1. Capture any valid legacy (pre-EIP-155) transaction broadcast on the affected Ethermint chain.
2. Replay it verbatim on any other EVM-compatible chain that also accepts unprotected transactions (or vice-versa).
3. Because the transaction carries no chain-ID binding, the signature is valid on both chains simultaneously.

This is a direct EIP-155 / chain-ID verification bypass enabling cross-chain replay of fund-transferring or contract-calling transactions — matching the High impact category: *"Ethereum transaction, EIP-155/EIP-712/EIP-7702 authorization, nonce, chain-id, or signer verification bypass enabling replay, forged execution, or unauthorized account/code mutation."*

---

### Likelihood Explanation

- Every chain that performed the v4→v5 consensus-version upgrade is affected if `AllowUnprotectedTxs` was `false` at migration time (the default for all chains that did not explicitly enable it via governance).
- The corrupted value is written into the single consolidated `KeyPrefixParams` blob and persists across all subsequent migrations (v6, v7, v8 each read and re-write the same blob without touching `AllowUnprotectedTxs`).
- No privileged access is required to exploit the resulting state: any user can submit an unprotected transaction. [4](#0-3) [5](#0-4) 

---

### Recommendation

Replace the three `store.Has()` calls with proper value reads and unmarshalling, mirroring the pattern already used for `chainConfig` and `extraEIPs` in the same function:

```go
// Correct pattern
var enableCreate bool
bz := store.Get(v0types.ParamStoreKeyEnableCreate)
cdc.MustUnmarshal(bz, &enableCreate)
params.EnableCreate = enableCreate

var enableCall bool
bz = store.Get(v0types.ParamStoreKeyEnableCall)
cdc.MustUnmarshal(bz, &enableCall)
params.EnableCall = enableCall

var allowUnprotectedTxs bool
bz = store.Get(v0types.ParamStoreKeyAllowUnprotectedTxs)
cdc.MustUnmarshal(bz, &allowUnprotectedTxs)
params.AllowUnprotectedTxs = allowUnprotectedTxs
```

For chains that have already run the v5 migration, a follow-on migration or governance parameter-update proposal is needed to restore the correct `AllowUnprotectedTxs` value.

---

### Proof of Concept

1. Chain genesis sets `AllowUnprotectedTxs = false` (the default). The params subspace writes the key `ParamStoreKeyAllowUnprotectedTxs` → amino(`false`) into the EVM KV store.
2. Chain upgrades to consensus version 5; `v5.MigrateStore` executes.
3. `store.Has(v0types.ParamStoreKeyAllowUnprotectedTxs)` returns `true` (key exists with value amino(`false`)).
4. `params.AllowUnprotectedTxs` is set to `true`.
5. The consolidated params blob is written; old keys are deleted. The chain now permanently accepts unprotected transactions.
6. Attacker constructs a legacy Ethereum transaction (no `v`, `r`, `s` chain-ID encoding per EIP-155) transferring funds from victim address.
7. Transaction passes ante-handler validation because `AllowUnprotectedTxs = true`.
8. Same signed bytes are replayed on a second chain, draining the victim's funds there as well. [6](#0-5)
