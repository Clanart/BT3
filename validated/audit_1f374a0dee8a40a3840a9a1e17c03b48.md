### Title
Missing Declared Class Hash Validation in `execute_replace_class` Enables Permanent Fund Freezing — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in `syscall_impls.cairo` does not verify that the new class hash supplied by the caller is actually declared (i.e., exists in `contract_class_changes`). An unprivileged contract deployer can call `replace_class` with an arbitrary, undeclared hash, permanently breaking the contract and irreversibly freezing any funds it holds. This is the direct OS-level analog of M-07: just as `Pair.mint()` lacked the `isGenesis` guard that `RouterV2.addLiquidity()` was supposed to enforce, `execute_replace_class` lacks the declared-class-hash guard that must protect the contract state from invalid class assignments.

---

### Finding Description

In `execute_replace_class`, the new class hash is taken directly from the user-controlled syscall request and written to `contract_state_changes` with no validation that the hash corresponds to a declared class: [1](#0-0) 

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The `TODO` comment at line 898 explicitly acknowledges the missing check. The function accepts any felt as `class_hash` and commits it to state unconditionally.

**What happens next.** After `replace_class` with undeclared hash `X` succeeds:

- `contract_state_changes[addr].class_hash = X`
- `contract_class_changes[X]` is never written → `dict_read` returns `0` (the Cairo dict default)

When any subsequent transaction calls the contract, `execute_entry_point` performs: [2](#0-1) 

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // = X (undeclared) → returns 0
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,           // = 0 → not in facts → assertion failure
);
```

`find_element` asserts the key is present; if it is not, the Cairo VM raises an assertion failure, halting the entire OS execution. The sequencer therefore cannot include any transaction that calls the broken contract. Any funds held by the contract become permanently inaccessible.

**Analog to M-07.** M-07 identified two layers of missing guards:

| M-07 (Blackhole) | This codebase |
|---|---|
| `RouterV2.addLiquidity` guard bypassable once `totalSupply > 0` | `execute_replace_class` guard absent: no check that new class hash is declared |
| `Pair.mint()` has no `isGenesis` check at all | `execute_replace_class` has no declared-class-hash check at all (acknowledged by TODO) |

Both share the same root cause class: **a lower-level state-mutating function that is missing the guard enforced (or intended to be enforced) at a higher level**, allowing an attacker to create an invalid protocol state.

---

### Impact Explanation

Any contract holding funds (STRK, ETH, or any ERC-20) can have its class permanently replaced with an undeclared hash. Once committed:

1. All subsequent calls to the contract fail at the OS level (not a graceful revert — an assertion failure).
2. The sequencer cannot include any transaction targeting the contract.
3. Funds held by the contract are permanently inaccessible to all parties.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The attack is reachable by any unprivileged StarkNet user with the ability to deploy a contract:

1. Deploy contract `C` (class `A`, declared) with a function `f(new_hash)` that issues the `replace_class(new_hash)` syscall.
2. Attract user deposits into `C` (e.g., by presenting it as a yield vault or bridge).
3. Call `C.f(0xDEAD)` where `0xDEAD` is any undeclared felt.
4. The OS processes `execute_replace_class` without validation; state is committed.
5. All subsequent calls to `C` fail; deposited funds are permanently frozen.

No privileged access, leaked keys, or operator cooperation is required. The only precondition is the ability to deploy a contract and invoke a syscall — available to every StarkNet user.

---

### Recommendation

Add a declared-class-hash check inside `execute_replace_class` before updating state:

```cairo
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    contract_class_changes: DictAccess*,   // add this implicit arg
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

+   // Verify the new class hash is declared.
+   let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
+       key=class_hash
+   );
+   if (compiled_class_hash == 0) {
+       write_failure_response(remaining_gas=remaining_gas, failure_felt=ERROR_INVALID_ARGUMENT);
+       return ();
+   }

    local state_entry: StateEntry*;
    ...
}
```

This mirrors the fix recommended in M-07: enforce the guard at the point of the state-mutating operation, not only at a higher-level caller.

---

### Proof of Concept

1. Declare class `A` with compiled class hash `H_A` (legitimate).
2. Deploy contract `C` at address `addr` with class `A`. `C` exposes `f(new_hash)` which calls `replace_class(new_hash)`.
3. Users deposit 1 000 STRK into `C`.
4. Attacker calls `C.f(0xDEAD)` where `0xDEAD` is not in `contract_class_changes`.
5. OS executes `execute_replace_class`:
   - `class_hash = 0xDEAD` (no validation performed — TODO acknowledged)
   - `contract_state_changes[addr].class_hash ← 0xDEAD`
   - Transaction succeeds; state committed to chain.
6. User submits withdrawal transaction to `C`.
7. OS calls `execute_entry_point` for `C`:
   - `dict_read(contract_class_changes, 0xDEAD)` → `0`
   - `find_element(..., key=0)` → assertion failure; OS halts.
8. Sequencer cannot include the withdrawal; 1 000 STRK permanently frozen in `C`.
