### Title
Hardcoded `return true` in `FakeProver` Bypasses All Proof Verification in `ENearProxy.finaliseNearToEthTransfer` - (File: `evm/src/eNear/contracts/FakeProver.sol`)

---

### Summary

`FakeProver.sol` implements the production `INearProver` interface with a function that unconditionally returns `true` for every proof, regardless of the actual proof data or block height. If this contract is deployed and configured as the `prover` in `ENearProxy`, the proof-validity gate in `ENearProxy.finaliseNearToEthTransfer` is completely bypassed, allowing any caller to finalize arbitrary NEAR-to-ETH transfers with fabricated proof data and mint eNEAR tokens without any corresponding NEAR being locked.

---

### Finding Description

`FakeProver` is a concrete contract in the production scope directory `evm/src/eNear/contracts/` that implements `INearProver`:

```solidity
contract FakeProver is INearProver {
    function proveOutcome(bytes calldata, uint64) external pure returns (bool) {
        return true;
    }
}
``` [1](#0-0) 

It ignores both arguments and always returns `true`. This is structurally identical to the external report's `return (30_000e18, 0)` pattern: a hardcoded result that makes the function's logic permanently inert.

`ENearProxy.finaliseNearToEthTransfer` relies on `prover.proveOutcome` as its sole proof-validity gate before forwarding the call to the underlying `eNear` contract:

```solidity
function finaliseNearToEthTransfer(
    bytes memory proofData,
    uint64 proofBlockHeight
) external whenNotPaused(PAUSED_LEGACY_FIN_TRANSFER) {
    require(
        prover.proveOutcome(proofData, proofBlockHeight),
        "Proof should be valid"
    );
    eNear.finaliseNearToEthTransfer(proofData, proofBlockHeight);
}
``` [2](#0-1) 

The `prover` address is set at initialization time and is a mutable `INearProver` storage variable:

```solidity
INearProver public prover;
``` [3](#0-2) 

```solidity
prover = INearProver(_prover);
``` [4](#0-3) 

If `FakeProver` is the deployed address passed as `_prover`, the `require` check becomes a no-op and any caller can supply arbitrary `proofData` to `finaliseNearToEthTransfer`.

---

### Impact Explanation

With `FakeProver` as the configured prover, an unprivileged attacker can call `ENearProxy.finaliseNearToEthTransfer` with crafted `proofData` encoding an arbitrary recipient address and amount. The call passes the proof gate unconditionally and is forwarded to `eNear.finaliseNearToEthTransfer`, which mints eNEAR tokens to the attacker-controlled recipient. This constitutes **unauthorized mint of bridged assets** with no corresponding NEAR locked on the source chain, directly breaking bridge collateralization.

Impact class: **High — Proof verification bypass enabling unauthorized transfer finalization.**

---

### Likelihood Explanation

The likelihood depends on whether `FakeProver` is actually deployed and wired as the `prover` in the live `ENearProxy`. The contract exists in the production contract directory (not a `test/` subdirectory), implements the production `INearProver` interface, and is deployable by anyone. If an operator mistakenly (or intentionally) passes its address during `initialize`, the vulnerability is immediately exploitable by any unprivileged user with no further preconditions. The `ENearProxy.mint` function already demonstrates that the codebase constructs and submits synthetic proof data internally, confirming that crafted proof bytes are a realistic attack vector. [5](#0-4) 

---

### Recommendation

Remove `FakeProver.sol` from the production contracts directory entirely. If a stub is needed for local testing, move it to a dedicated `test/` subdirectory and ensure it is never referenced in deployment scripts. Additionally, add a deployment-time check in `ENearProxy.initialize` that validates the supplied `_prover` address is not a known stub (e.g., by calling `proveOutcome` with a zero-length proof and asserting it returns `false`), or enforce an allowlist of approved prover implementations.

---

### Proof of Concept

1. Attacker (or misconfigured deployment) deploys `FakeProver`.
2. `ENearProxy.initialize` is called with `_prover = address(FakeProver)`.
3. Attacker constructs `proofData` encoding:
   - `nearConnector` bytes matching the configured connector
   - An arbitrary `amount` (e.g., `1_000_000e18`)
   - `to = attacker_address`
4. Attacker calls `ENearProxy.finaliseNearToEthTransfer(proofData, 0)`.
5. `prover.proveOutcome(proofData, 0)` returns `true` unconditionally.
6. `eNear.finaliseNearToEthTransfer(proofData, 0)` executes, minting eNEAR to the attacker with no NEAR locked on NEAR chain.
7. Bridge supply is unbacked; attacker profits by selling the minted eNEAR. [6](#0-5) [2](#0-1)
