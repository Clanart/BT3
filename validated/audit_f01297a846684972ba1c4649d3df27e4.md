### Title
Unvalidated `l1_gas_price_wei`/`l1_data_gas_price_wei` in `ProposalInit` Allows Proposer to Corrupt ETH-Denomination Fee Calculations and Block Commitment - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The validator's `is_block_info_valid` function checks only the FRI-denominated L1 gas prices (`l1_gas_price_fri`, `l1_data_gas_price_fri`) against its own oracle, but silently discards the WEI-denominated counterparts (`l1_gas_price_wei`, `l1_data_gas_price_wei`) without any independent verification. A malicious proposer can supply arbitrary WEI prices that pass validation, which are then used verbatim to populate `eth_gas_prices` in `BlockInfo` and to derive `l2_gas_price_wei` via the implied ETH-to-FRI rate. This corrupts fee calculations for ETH-paying (deprecated V1/V2) transactions, produces a wrong state diff, and ultimately a wrong block commitment.

---

### Finding Description

**Root cause — `is_block_info_valid` discards the WEI prices it fetches:**

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs:286-292
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,
    block_info_validation.previous_block_info.as_ref(),
    gas_price_params,
)
.await;
```

The validator fetches both FRI and WEI prices from its own oracle, but the WEI result is bound to `_l1_gas_prices_wei` (explicitly discarded). Only the FRI prices are compared against the proposer's values:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs:302-307
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
    && within_margin(
        l1_data_gas_price_fri_proposed,
        l1_data_gas_price_fri,
        l1_gas_price_margin_percent,
    ))
```

`init_proposed.l1_gas_price_wei` and `init_proposed.l1_data_gas_price_wei` are never compared to anything. [1](#0-0) 

**Propagation — the unchecked WEI prices flow directly into `BlockInfo`:**

`convert_to_sn_api_block_info` (called for both proposer and validator paths) takes the proposer-supplied `init.l1_gas_price_wei` and `init.l1_data_gas_price_wei` verbatim:

---

### Impact Explanation

A malicious proposer sets `l1_gas_price_wei` to an attacker-chosen value (e.g., `u128::MAX / 2`) while keeping `l1_gas_price_fri` within the validator's margin. The validator's `is_block_info_valid` passes because only FRI prices are checked. The resulting `BlockInfo` has:

1. **Corrupted `eth_gas_prices.l1_gas_price` and `eth_gas_prices.l1_data_gas_price`** — directly set from the manipulated WEI values. All deprecated V1/V2 transactions (which pay fees in ETH/WEI) are charged wrong fees. Setting WEI prices very high causes ETH-paying transactions to fail with insufficient balance; setting them very low causes near-zero fee collection.

2. **Corrupted `eth_gas_prices.l2_gas_price`** — derived from `l2_gas_price_fri / eth_to_fri_rate`, where `eth_to_fri_rate` is computed from the manipulated `l1_gas_price_wei` vs `l1_gas_price_fri` ratio. An attacker who sets `l1_gas_price_wei` very high makes `eth_to_fri_rate` very small, making `l2_gas_price_wei` very large.

3. **Wrong fee token balance changes in the state diff** — the actual fee charged (and transferred to the sequencer) is wrong, so the ERC-20 storage slots for the fee token are wrong.

4. **Wrong block commitment** — the state diff commitment, receipt commitment, and ultimately the `ProposalCommitment`/block hash are computed over the wrong execution outputs.

This matches the Critical impact: "Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact" and "Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input."

---

### Likelihood Explanation

Any consensus participant who is selected as the proposer for a round can trigger this. In Tendermint-style consensus, proposer selection rotates among validators. The attack requires no special privilege beyond being the proposer for one block. The manipulation is invisible to validators because the WEI prices are never compared to the validator's own oracle output. The only constraint is that `l1_gas_price_fri` must remain within `l1_gas_price_margin_percent` of the validator's oracle reading, which is a loose bound (configurable, defaulting to a percentage margin). The WEI prices can be set to any value that keeps `NonzeroGasPrice::new` from returning an error (i.e., non-zero).

---

### Recommendation

In `is_block_info_valid`, after fetching `_l1_gas_prices_wei` from the oracle, compare the proposed WEI prices against the oracle's WEI prices using the same `within_margin` check applied to FRI prices:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;

// existing FRI check ...

// Add WEI check:
if !(within_margin(init_proposed.l1_gas_price_wei, l1_gas_prices_wei.l1_gas_price, margin)
    && within_margin(init_proposed.l1_data_gas_price_wei, l1_gas_prices_wei.l1_data_gas_price, margin))
{
    return Err(ValidateProposalError::InvalidBlockInfo(...));
}
```

Alternatively, enforce that the implied ETH-to-FRI rate derived from the proposed WEI/FRI pair is consistent with the validator's own oracle rate, bounding the ratio within an acceptable range.

### Proof of Concept

1. Validator `V` is running with `l1_gas_price_margin_percent = 10` (10% margin).
2. Oracle reports `l1_gas_price_wei = 1000`, `eth_to_fri_rate = 1000`, so `l1_gas_price_fri = 1_000_000`.
3. Malicious proposer `P` constructs `ProposalInit` with:
   - `l1_gas_price_fri = 1_000_000` (within 10% of oracle → passes `within_margin`)
   - `l1_gas_price_wei = u128::MAX / 2` (not checked → passes validation)
   - `l1_data_gas_price_fri` and `l1_data_gas_price_wei` similarly manipulated
4. `V` calls `is_block_info_valid`:
   - FRI check passes (1_000_000 within 10% of 1_000_000). [9](#0-8) 
   - WEI check is absent (`_l1_gas_prices_wei` discarded). [10](#0-9) 
5. `V` calls `convert_to_sn_api_block_info(&init)`:
   - `eth_gas_prices.l1_gas_price = u128::MAX / 2` (from proposer). [11](#0-10) 
   - `eth_to_fri_rate = 1_000_000 * WEI_PER_ETH / (u128::MAX / 2) ≈ 0` (rounds to 0 or 1).
   - `l2_gas_price_wei = l2_gas_price_fri / eth_to_fri_rate` → enormous or error.
6. All deprecated V1/V2 transactions in the block are executed with `eth_gas_prices.l1_gas_price = u128::MAX / 2`, causing them to fail with `InsufficientFeeTokenBalance` or be charged astronomically wrong fees.
7. The resulting state diff has wrong fee token balances; the block commitment is wrong and accepted by all validators who received the same `ProposalInit`.
