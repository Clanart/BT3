### Title
EIP-7702 Authorization Chain-ID Zero Bypasses Chain-Binding Protection, Enabling Cross-Chain Replay of Code Delegation — (File: `x/evm/keeper/set_code_authorizations.go`)

---

### Summary

`validateAuthorization` explicitly skips the chain-ID check when an EIP-7702 authorization carries `ChainID = 0`. An unprivileged attacker who observes such an authorization on any other EVM chain can replay it on Ethermint — provided the authority's nonce on Ethermint matches — installing arbitrary code delegation on the victim's EOA without the victim's knowledge or consent on this chain.

---

### Finding Description

`validateAuthorization` in `x/evm/keeper/set_code_authorizations.go` enforces chain-ID binding only when the authorization's chain ID is non-zero:

```go
// Verify chain ID is null or equal to current chain ID.
if !auth.ChainID.IsZero() && auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
    return authority, core.ErrAuthorizationWrongChainID
}
``` [1](#0-0) 

When `auth.ChainID == 0` the entire chain-ID check is short-circuited. The authorization is then accepted as long as the ECDSA signature is valid and the authority's on-chain nonce matches `auth.Nonce`. [2](#0-1) 

The ante-handler validation in `ValidateEthBasic` checks only that the EIP-7702 transaction is not a contract creation and that the auth list is non-empty. It performs **no per-authorization chain-ID inspection**:

```go
if tx.SetCodeAuthorizations() != nil {
    if tx.To() == nil { ... }
    if len(tx.SetCodeAuthorizations()) == 0 { ... }
}
``` [3](#0-2) 

There is therefore no layer — ante handler or state-transition — that rejects a chain-ID-0 authorization submitted on Ethermint that was originally signed on a different chain.

Once accepted, `setAuthorizationDelegation` unconditionally writes the delegation bytecode and bumps the nonce:

```go
stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), ...)
``` [4](#0-3) 

The durable path (`applyDurableAuthorization`) then commits these changes to the persistent KV store, making the delegation permanent. [5](#0-4) 

---

### Impact Explanation

An attacker who replays a chain-ID-0 authorization on Ethermint installs arbitrary delegation bytecode (`0xef0100<address>`) on the victim's EOA. From that point the victim's account executes the attacker-chosen contract's code on every call. If the delegation target implements a `transfer`/`approve` or self-destruct path, the attacker can drain all EVM-denom and ERC-20 balances held by the victim's address. This satisfies the **High** impact criterion: *EIP-7702 authorization chain-id verification bypass enabling unauthorized account/code mutation*.

---

### Likelihood Explanation

The nonce constraint (`stateDB.GetNonce(authority) == auth.Nonce`) is the only practical barrier. It is satisfied in the common case of:

- A fresh Ethermint account (nonce 0) whose owner previously signed a chain-ID-0 authorization with nonce 0 on Ethereum mainnet or any other EVM chain.
- Any account whose Ethermint nonce happens to equal the nonce embedded in a publicly visible chain-ID-0 authorization on another chain.

Chain-ID-0 authorizations are observable on-chain on any EVM network. Wallets that implement EIP-7702 may present "valid on all chains" as a convenience option (the direct analog of "skip encryption" in the external report). The attacker's only action is to submit a well-formed EIP-7702 transaction on Ethermint containing the replayed authorization — a fully unprivileged operation.

---

### Recommendation

**Short term:** Add a per-authorization chain-ID check in `ValidateEthBasic` (or in `validateAuthorization`) that rejects authorizations with `ChainID == 0`:

```go
if auth.ChainID.IsZero() {
    return errorsmod.Wrap(errortypes.ErrInvalidRequest,
        "EIP-7702 authorization must specify a non-zero chain ID")
}
```

**Long term:** Enforce this at the `ValidateBasic` level of `MsgEthereumTx` so the rejection happens before any state access, and document the policy clearly so integrators do not inadvertently re-enable chain-ID-0 authorizations.

---

### Proof of Concept

1. On Ethereum mainnet (or any EVM chain), victim signs an EIP-7702 authorization with `chainId = 0`, `address = ATTACKER_CONTRACT`, `nonce = N` and broadcasts a SetCode transaction. The signed authorization `(v, r, s)` is now publicly visible on-chain.

2. Attacker observes the authorization. Victim's Ethermint account also has nonce `N` (e.g., both accounts are fresh with nonce 0).

3. Attacker constructs an EIP-7702 (`type 4`) transaction on Ethermint:
   - `to`: victim's address
   - `authorizationList`: `[{chainId: 0, address: ATTACKER_CONTRACT, nonce: N, v, r, s}]`
   - Signs the outer transaction with the attacker's own key and broadcasts via `eth_sendRawTransaction`.

4. `VerifyEthSig` validates the outer transaction signature (attacker's key) — passes.

5. `ValidateEthBasic` checks: not a contract creation, auth list non-empty — passes. [3](#0-2) 

6. `validateAuthorization` is called: `auth.ChainID.IsZero()` is `true`, so the chain-ID guard is skipped. Signature recovery yields victim's address. Nonce check `stateDB.GetNonce(victim) == N` — passes. [6](#0-5) 

7. `setAuthorizationDelegation` writes `0xef0100<ATTACKER_CONTRACT>` as the victim's code and bumps the victim's nonce to `N+1`. [4](#0-3) 

8. `applyDurableAuthorization` commits the delegation to the durable stateDB. [5](#0-4) 

9. All subsequent calls to the victim's address on Ethermint execute `ATTACKER_CONTRACT`'s logic. The attacker calls a drain function, transferring the victim's EVM-denom balance to the attacker.

func (k *Keeper) validateAuthorization(auth *types.SetCodeAuthorization, stateDB vm.StateDB) (authority common.Address, err error) {
	// Verify chain ID is null or equal to current chain ID.
	if !auth.ChainID.IsZero() && auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
		return authority, core.ErrAuthorizationWrongChainID
	}
	// Limit nonce to 2^64-1 per EIP-2681.
	if auth.Nonce+1 < auth.Nonce {
		return authority, core.ErrAuthorizationNonceOverflow
	}
	// Validate signature values and recover authority.
	authority, err = auth.Authority()
	if err != nil {
		return authority, fmt.Errorf("%w: %v", core.ErrAuthorizationInvalidSignature, err)
	}
	// Check the authority account
	//  1) doesn't have code or has exisiting delegation
	//  2) matches the auth's nonce
	//
	// Note it is added to the access list even if the authorization is invalid.
	stateDB.AddAddressToAccessList(authority)
	code := stateDB.GetCode(authority)
	if _, ok := types.ParseDelegation(code); len(code) != 0 && !ok {
		return authority, core.ErrAuthorizationDestinationHasCode
	}
	if have := stateDB.GetNonce(authority); have != auth.Nonce {
		return authority, core.ErrAuthorizationNonceMismatch
	}
	return authority, nil
```

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
```
