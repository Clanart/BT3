Audit Report

## Title
`buyToken0InBinSpecifiedIn` fallback sets full input without rescaling output, causing swap conservation failure — (`metric-core/contracts/libraries/SwapMath.sol`)

## Summary
In `SwapMath.buyToken0InBinSpecifiedIn`, when both the analytical solution and the proportional refinement underestimate the required input for a partial bin fill, a fallback at lines 620-622 sets `totalIn1Scaled = amountSpecifiedRemainingScaled` without updating `out0Scaled`. The subsequent rescale guard at line 624 uses strict greater-than (`>`), which evaluates to `false` since the values are now equal, causing it to be skipped entirely. The pool charges the trader's full specified input while crediting only the under-computed output, retaining the excess token1 as pool balance. The identical bug exists in `buyToken1InBinSpecifiedIn` at lines 758-760.

## Finding Description
The outer check at line 577 establishes that the cost to traverse the full bin exceeds `amountSpecifiedRemainingScaled`, so a partial fill is needed. The analytical solution (lines 581-596) computes an initial `targetPos`, `out0Scaled`, and `totalIn1Scaled`. If `totalIn1Scaled < amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos`, the proportional refinement (lines 598-618) scales `targetPos` up and recomputes both values. When `scaledTarget <= maxFinalBinPos` (i.e., the scaled target does not reach the bin boundary), `targetPos` is set to `scaledTarget` and both outputs are recomputed. Due to the non-linearity of the geometric-mean price curve, the recomputed `totalIn1Scaled` can still be less than `amountSpecifiedRemainingScaled` while `targetPos < maxFinalBinPos`.

At this point the fallback fires:
```solidity
// lines 620-622
if (totalIn1Scaled < state.amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos) {
    totalIn1Scaled = state.amountSpecifiedRemainingScaled;  // bumped to full input
    // out0Scaled is NOT updated — still computed for smaller targetPos
}
```
After the fallback, `totalIn1Scaled == amountSpecifiedRemainingScaled`. The rescale guard immediately below:
```solidity
// line 624
if (totalIn1Scaled > state.amountSpecifiedRemainingScaled) {  // FALSE — equal, not greater
    out0Scaled = (out0Scaled * state.amountSpecifiedRemainingScaled) / totalIn1Scaled;  // SKIPPED
    ...
}
```
is skipped because the condition is strict `>`. Settlement then executes with the mismatched pair:
```solidity
state.amountSpecifiedRemainingScaled -= totalIn1Scaled;  // full input consumed
state.amountCalculatedScaled         += out0Scaled;      // under-computed, not rescaled
```
The pool's token1 balance increases by the full input (minus protocol fee), while token0 decreases only by the under-computed `out0Scaled`. The difference is silently absorbed into the pool's token1 balance with no corresponding token0 given to the trader.

## Impact Explanation
This is a direct swap conservation failure causing loss of trader principal. The trader specifies an exact input amount and expects the maximum achievable output. When the fallback fires, the pool charges `amountSpecifiedRemainingScaled` in token1 but credits `out0Scaled` computed for a strictly smaller `targetPos`. The excess `amountSpecifiedRemainingScaled − original_totalIn1Scaled` in token1 is retained by the pool without any corresponding token0 output. This constitutes a High/Critical severity direct loss of user principal: `amountIn × price > amountOut + fee` within rounding tolerance, violating the core swap conservation invariant. The symmetric bug in `buyToken1InBinSpecifiedIn` affects token0→token1 exact-input swaps identically.

## Likelihood Explanation
The fallback is reachable by any unprivileged trader executing a standard exact-input swap through `_swapToken1ForToken0SpecifiedInput` → `SwapMath.buyToken0InBinSpecifiedIn`. No privileged access or special setup is required. The condition is triggered when the price curve's non-linearity causes both the quadratic analytical estimate and the single proportional refinement to underestimate the required input, with `scaledTarget` landing strictly below `maxFinalBinPos`. Wide bins (large `lengthE6`), moderate-to-high fees, and input amounts that land in the middle of a bin are the most likely triggers. This is a deterministic arithmetic condition reproducible with specific but non-exotic parameter combinations.

## Recommendation
Replace the fallback at lines 620-622 with a correct resolution that advances to `maxFinalBinPos` and recomputes `out0Scaled`, so the subsequent rescale guard at line 624 fires correctly:
```solidity
// Replace lines 620-622 with:
if (totalIn1Scaled < state.amountSpecifiedRemainingScaled && targetPos < maxFinalBinPos) {
    targetPos = maxFinalBinPos;
    out0Scaled = calculateOutputToken0FromBinPosition(binState.token0BalanceScaled, currBinPos, targetPos);
    finalPriceX64 = calculatePriceAtBinPosition(lowerPriceX64, upperPriceX64, targetPos, Math.Rounding.Ceil);
    avgPriceX64 = calculateArithmeticMean(startingPriceX64, finalPriceX64);
    in1WithoutFeeScaled = calculateRequiredToken(out0Scaled, avgPriceX64);
    totalIn1Scaled = grossInputWithBinFeeCeil(in1WithoutFeeScaled, onePlusBuyFeeX64);
}
```
This ensures `totalIn1Scaled > amountSpecifiedRemainingScaled` (since the outer check at line 577 established that the full-bin cost exceeds the remaining input), causing the rescale guard at line 624 to fire and proportionally reduce `out0Scaled` before settlement. Apply the symmetric fix to `buyToken1InBinSpecifiedIn` lines 758-760.

## Proof of Concept
1. Deploy a pool with a wide bin (`lengthE6 = 50_000`, i.e. 5% width), moderate fee (e.g. 0.3%), and token0 liquidity.
2. Execute an exact-input token1→token0 swap with an input amount chosen such that the analytical solution and proportional refinement both produce `targetPos < maxFinalBinPos` and `totalIn1Scaled < amountSpecifiedRemainingScaled` after both passes (mid-bin landing zone on a convex price curve).
3. Observe that `state.amountSpecifiedRemainingScaled` is reduced to zero (full input consumed) while `state.amountCalculatedScaled` reflects `out0Scaled` computed for the smaller `targetPos`.
4. Verify the conservation invariant `consumed × avgPrice ≤ out0 + fee` is violated: the pool retains excess token1 with no corresponding token0 output to the trader.
5. Apply the symmetric test to `buyToken1InBinSpecifiedIn` with a token0→token1 exact-input swap.
