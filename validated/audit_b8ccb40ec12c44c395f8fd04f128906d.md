### Title
`initTransfer` Does Not Verify Actual Token Balance Received After `safeTransferFrom` — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` calls `safeTransferFrom` to lock native ERC20 tokens but never verifies that the contract's balance actually increased by `amount`. The caller-supplied `amount` is emitted verbatim in the `InitTransfer` event, which NEAR uses as the authoritative locked amount to mint or unlock tokens on the destination chain. A token whose `transferFrom` returns `true` without updating balances (e.g., after an upgrade introducing silent blacklisting) allows an attacker to fabricate a fully-valid bridge event with zero real deposit, receiving destination-chain tokens for free.

---

### Finding Description

In `OmniBridge.sol`, the native ERC20 lock path inside `initTransfer` is:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
``` [1](#0-0) 

Immediately after, the function emits:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // ← caller-supplied, never cross-checked against actual balance delta
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

`SafeERC20.safeTransferFrom` only checks that the low-level call does not revert and that the return value (if any) is `true`. It does **not** verify that `address(this).balanceOf(token)` increased by `amount`. If a token's `transferFrom` implementation returns `true` while silently withholding the balance update (e.g., a blacklist that freezes the sender without reverting, introduced via an upgradeable proxy), the bridge contract emits a valid `InitTransfer` event for `amount` tokens it never received.

The `InitTransfer` event is the sole input that NEAR's prover pipeline uses to authorize the corresponding `fin_transfer` on NEAR, which then mints or unlocks `amount` tokens for the attacker on the destination chain. [3](#0-2) 

The same structural gap exists in `initTransfer1155` for ERC-1155 tokens:

```solidity
IERC1155(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    tokenId,
    amount,
    ""
);
``` [4](#0-3) 

---

### Impact Explanation

An attacker who can trigger a `safeTransferFrom` that returns `true` without actually moving tokens can:

1. Emit a legitimate `InitTransfer` event for an arbitrary `amount`.
2. Have a relayer submit the corresponding proof to NEAR.
3. Receive `amount` tokens minted or unlocked on the destination chain with zero real collateral deposited on the EVM side.

This is **direct theft / unauthorized mint of bridged assets** — a Critical impact under the allowed scope. The bridge's collateralization invariant (locked EVM tokens ↔ minted NEAR tokens) is broken: the NEAR side mints unbacked supply.

---

### Likelihood Explanation

Many high-value ERC-20 tokens supported by the bridge (USDC, USDT, and others) are upgradeable proxies. A future upgrade that introduces silent blacklisting — where `transferFrom` returns `true` but does not move tokens for blacklisted senders — is a realistic scenario (USDC's existing blacklist already reverts; a future variant that silently succeeds is the risk). The attacker does not need to control the token contract; they only need to be in a state where the token silently no-ops their transfer. The attack is permissionless: any user can call `initTransfer`.

---

### Recommendation

Record the contract's token balance before and after `safeTransferFrom` and require the delta equals `amount`:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
    uint256 balanceAfter = IERC20(tokenAddress).balanceOf(address(this));
    require(balanceAfter - balanceBefore == amount, "ERR_TRANSFER_AMOUNT_MISMATCH");
}
```

Apply the same pattern to `initTransfer1155` using `IERC1155.balanceOf(address(this), tokenId)` before and after the transfer. [1](#0-0) 

---

### Proof of Concept

1. Deploy a malicious ERC-20 token `MaliciousToken` whose `transferFrom(from, to, amount)` emits a `Transfer` event and returns `true` but does **not** update `_balances[to]` when `from` is on a "frozen" list.
2. Register `MaliciousToken` with the bridge (it is not a bridge token and has no custom minter, so it falls into the native lock path).
3. Place the attacker's address on the frozen list.
4. Attacker calls `OmniBridge.initTransfer(address(MaliciousToken), 1_000_000e18, 0, 0, "attacker.near", "")`.
5. `safeTransferFrom` returns `true`; bridge balance of `MaliciousToken` is unchanged (0).
6. Bridge emits `InitTransfer(..., amount=1_000_000e18, ...)` with a valid nonce.
7. Relayer submits the event proof to NEAR's prover.
8. NEAR's `fin_transfer` processes the proof and mints/unlocks `1_000_000e18` tokens to `attacker.near`.
9. Attacker has received 1 million tokens on NEAR backed by zero EVM collateral. [5](#0-4)
