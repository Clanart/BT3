### Title
3-arg `mint` in `HyperliquedBridgeToken` routes all minted tokens to `_systemAddress`, permanently stealing bridged assets from the intended recipient — (`evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

---

### Summary

`HyperliquedBridgeToken.mint(address, uint256, bytes)` calls `_mint(account, value)` followed immediately by `_update(account, _systemAddress, value)`, which transfers the entire freshly-minted balance away from the recipient to `_systemAddress`. `OmniBridge.finTransfer` invokes this 3-arg path whenever `payload.message.length > 0`. Because the `message` field is attacker-controlled on the source chain, any unprivileged user can cause a legitimate, MPC-signed cross-chain transfer to credit zero tokens to the intended recipient.

---

### Finding Description

**`HyperliquedBridgeToken.mint` (3-arg path)** [1](#0-0) 

```solidity
function mint(address account, uint256 value, bytes memory) external override onlyOwner {
    _mint(account, value);
    _update(account, _systemAddress, value);
}
```

`_mint(account, value)` credits `value` tokens to `account`. The immediately following `_update(account, _systemAddress, value)` is a standard ERC-20 transfer that moves those same `value` tokens from `account` to `_systemAddress`. Net result: `account.balance == 0`, `_systemAddress.balance += value`.

**`OmniBridge.finTransfer` branching** [2](#0-1) 

```solidity
} else if (isBridgeToken[payload.tokenAddress]) {
    if (payload.message.length == 0) {
        IBridgeToken(payload.tokenAddress).mint(payload.recipient, payload.amount);
    } else {
        IBridgeToken(payload.tokenAddress).mint(payload.recipient, payload.amount, payload.message);
    }
}
```

The branch condition is purely `payload.message.length == 0`. The `message` field is a user-supplied string that originates from the source-chain `initTransfer` call and is included verbatim in the MPC-signed payload. Any non-empty byte string — even a single `0x00` byte — triggers the 3-arg path.

**Attack path (concrete, no privileged role required):**

1. Attacker calls `initTransfer` on any source chain (NEAR, Solana, another EVM) with `message = "x"` (any non-empty value) and `recipient` set to the victim's HyperEVM address, targeting a `HyperliquedBridgeToken`.
2. NEAR MPC signs the `TransferMessagePayload` including the non-empty `message` bytes — this is normal protocol operation; the MPC does not filter message content.
3. Relayer submits `OmniBridge.finTransfer` with the valid MPC signature.
4. `finTransfer` passes signature validation, marks the nonce used, and enters the `isBridgeToken` branch (assuming `customMinters[tokenAddress] == address(0)`).
5. Because `payload.message.length > 0`, the 3-arg `mint` is called.
6. `_mint(recipient, amount)` runs, then `_update(recipient, _systemAddress, amount)` immediately drains the recipient to zero.
7. Recipient receives 0 tokens; `_systemAddress` receives `amount` tokens.

---

### Impact Explanation

**Critical — direct theft of bridged assets.** The intended recipient is permanently deprived of the full bridged amount. The tokens are not burned; they accumulate at `_systemAddress`. The invariant that `finTransfer` must credit exactly `payload.amount` tokens to `payload.recipient` is broken for every transfer that carries a non-empty message to a `HyperliquedBridgeToken`.

---

### Likelihood Explanation

**High.** The `message` field is a free-form, attacker-controlled string accepted by every source-chain `initTransfer` entry point. No special role, leaked key, or colluding MPC signer is required — the MPC signs the payload as presented. The condition is trivially satisfiable by any user who can initiate a cross-chain transfer.

---

### Recommendation

The 3-arg `mint` in `HyperliquedBridgeToken` must not silently re-route tokens away from the recipient. Two complementary fixes:

1. **Remove the `_update` call from `mint`** and instead handle HyperCore accounting separately (e.g., only park tokens at `_systemAddress` when the transfer is explicitly destined for HyperCore, not when arriving from the bridge).
2. **In `OmniBridge.finTransfer`**, treat `HyperliquedBridgeToken` as a special case: always call the 2-arg `mint` regardless of `payload.message`, or validate that a non-empty message is only permissible for tokens that are explicitly designed to handle it safely.

---
