### Title
Unbounded Staleness in `previous_block_info` Fallback Propagates Stale L1 Gas Prices Into Block Hash Commitment — (`crates/apollo_consensus_orchestrator/src/utils.rs`)

---

### Summary

When the L1 gas price provider fails (returning `StaleL1GasPricesError` or `MissingDataError`), `get_l1_prices_in_fri_and_wei_and_conversion_rate` falls back to `previous_block_info` — the gas prices from the last committed block — without any time-based staleness check. The `PreviousBlockInfo.timestamp` field is stored but never compared against the current time. Because `previous_block_info` is updated from each committed block's `ProposalInit`, stale prices from block N become the fallback for block N+1, whose prices then become the fallback for block N+2, and so on indefinitely. These prices are committed into `PartialBlockHashComponents` and hashed into the canonical block hash via `calculate_block_hash`, and are also used for transaction fee calculation.

---

### Finding Description

**Root cause — no staleness guard on `previous_block_info`:**

In `get_l1_prices_in_fri_and_wei_and_conversion_rate`, when the primary path fails, the code unconditionally uses `previous_block_info`:

```rust
if let Some(block_info) = previous_block_info {
    let prev_l1_gas_price_wei = block_info.l1_prices_wei.clone();
    let prev_l1_gas_price = block_info.l1_prices_fri.clone();
    let eth_to_fri_rate = calculate_eth_to_fri_rate(block_info);
    match eth_to_fri_rate {
        Ok(eth_to_fri_rate) => {
            return (prev_l1_gas_price, prev_l1_gas_price_wei, eth_to_fri_rate);
        }
        ...
    }
}
``` [1](#0-0) 

The `PreviousBlockInfo` struct carries a `timestamp` field:

```rust
pub(crate) struct PreviousBlockInfo {
    pub timestamp: u64,
    pub l1_prices_wei: L1PricesInWei,
    pub l1_prices_fri: L1PricesInFri,
}
``` [2](#0-1) 

That `timestamp` is **never compared against the current wall-clock time** before the prices are returned. There is no guard equivalent to `if current_time - block_info.timestamp > max_staleness { reject }`.

**Contrast with the provider's own staleness check:**

`L1GasPriceProvider::get_price_info` does enforce a `max_time_gap_seconds` (default 900 s):

```rust
if timestamp.0 > (*last_timestamp + self.config.max_time_gap_seconds) {
    return Err(L1GasPriceProviderError::StaleL1GasPricesError { ... });
}
``` [3](#0-2) 

This error is precisely what triggers the fallback — but the fallback itself has no equivalent time bound.

**How `previous_block_info` is updated:**

After every `decision_reached`, the context writes:

```rust
self.previous_block_info = Some(PreviousBlockInfo::from(&init));
``` [4](#0-3) 

So if the L1 scraper is down for N consecutive blocks, the prices from the last successful block are re-used for all N blocks, each time becoming the new `previous_block_info` for the next block.

**How stale prices reach the block hash:**

The prices are placed into `PartialBlockHashComponents`:

```rust
impl PartialBlockHashComponents {
    pub fn new(block_info: &BlockInfo, header_commitments: BlockHeaderCommitments) -> Self {
        Self {
            l1_gas_price: block_info.gas_prices.l1_gas_price_per_token(),
            l1_data_gas_price: block_info.gas_prices.l1_data_gas_price_per_token(),
            ...
        }
    }
}
``` [5](#0-4) 

And then hashed into the canonical block hash:

```rust
.chain_iter(
    gas_prices_to_hash(
        &partial_block_hash_components.l1_gas_price,
        &partial_block_hash_components.l1_data_gas_price,
        &partial_block_hash_components.l2_gas_price,
        &block_hash_version,
    )
    .iter(),
)
``` [6](#0-5) 

The same stale prices are used for transaction fee calculation via `convert_to_sn_api_block_info`, which constructs the `BlockInfo` passed to the blockifier. [7](#0-6) 

**Validator agreement on wrong value:**

The validator calls the same `get_l1_prices_in_fri_and_wei` with the same `previous_block_info` (derived from the same committed block), so both proposer and validator independently compute the same stale prices and agree. The `is_block_info_valid` check passes because both sides use the same fallback. [8](#0-7) 

---

### Impact Explanation

- **Wrong gas prices committed to block hash**: `l1_gas_price` and `l1_data_gas_price` in `PartialBlockHashComponents` reflect prices from an arbitrarily old block rather than current L1 conditions. The canonical block hash is therefore wrong relative to actual L1 state.
- **Incorrect fee accounting**: Transactions are charged fees based on stale L1 gas prices. If actual L1 gas prices have risen significantly, users underpay; if they have fallen, users overpay. This is a direct economic impact on fee correctness.
- **Unbounded propagation**: Unlike the provider's 15-minute `max_time_gap_seconds` guard, the fallback has no bound. Prices from hours or days ago can propagate indefinitely through the chain of blocks.

Matching impact scope: **Critical — Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact.**

---

### Likelihood Explanation

The trigger is the L1 gas price scraper failing to deliver new samples for more than `max_time_gap_seconds` (900 s by default). This occurs during:
- L1 network disruptions or RPC endpoint outages
- L1 reorgs that cause `UnexpectedBlockNumberError` in `add_price_info`, halting the scraper
- Infrastructure restarts where the scraper has not yet re-initialized

Once triggered, the fallback silently propagates stale prices for every subsequent block until the scraper recovers, with no log-level escalation or circuit-breaker.

---

### Recommendation

Add a staleness check on `previous_block_info` before using it as a fallback. Compare `block_info.timestamp` against the current wall-clock time and reject the fallback if the age exceeds a configurable threshold (e.g., the same `max_time_gap_seconds` used by the provider):

```rust
if let Some(block_info) = previous_block_info {
    let age = current_timestamp.saturating_sub(block_info.timestamp);
    if age <= max_fallback_age_seconds {
        // use previous_block_info
    }
}
```

Alternatively, surface a metric or warning that escalates when the fallback has been used for more than one consecutive block, so operators can detect prolonged L1 scraper outages before they affect many blocks.

---

### Proof of Concept

1. Start the sequencer with a working L1 gas price scraper. Record the L1 gas prices at block N.
2. Stop the L1 gas price scraper (or simulate `StaleL1GasPricesError` by advancing the block timestamp beyond `last_timestamp + max_time_gap_seconds`).
3. Continue producing blocks N+1, N+2, ..., N+K for any K.
4. Observe that every block from N+1 onward carries the same `l1_gas_price` and `l1_data_gas_price` as block N in its `PartialBlockHashComponents`, regardless of actual L1 gas price movements.
5. Verify via the block hash calculator that the committed block hashes for N+1..N+K all encode the stale prices from block N.

The `oracle_fails_on_second_block` test in `sequencer_consensus_context_test.rs` already demonstrates that the fallback is used and that the prices from the previous block are propagated — but it does not assert any staleness bound, confirming the absence of such a check. [9](#0-8)

    deps.state_sync_client.expect_add_new_block().times(1).return_once(|_| Ok(()));
