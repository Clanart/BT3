### Title
`L1EventsScraperConfig` defaults `finality` to `0`, allowing re-orged L1 handler transactions to corrupt the committed L2 state root — (`File: crates/apollo_l1_events_config/src/config.rs`)

---

### Summary

`L1EventsScraperConfig::default()` sets `finality: 0`. With zero finality the L1 scraper reads events from the very tip of the Ethereum chain. An ordinary Ethereum re-org (no attacker required) can then cause the sequencer to commit an L2 block whose state root encodes the effects of an L1 handler transaction that no longer exists on the canonical L1 chain. The `config_schema.json` also publishes `0` as the authoritative default, and no `#[validate(range(min = N))]` guard is present on the field, so any operator who does not explicitly override the value is silently exposed.

---

### Finding Description

`L1EventsScraperConfig` is the configuration struct for the component that scrapes L1 handler events from Ethereum and feeds them to the `L1EventsProvider`, which in turn gates their inclusion in L2 block proposals.

The `finality` field controls how many L1 blocks must be mined on top of an event-bearing block before that event is considered stable enough to scrape:

```rust
// crates/apollo_l1_events_config/src/config.rs  L141-L156
impl Default for L1EventsScraperConfig {
    fn default() -> Self {
        Self {
            ...
            finality: 0,          // ← zero confirmations
            ...
        }
    }
}
``` [1](#0-0) 

The field carries no minimum-value validator (contrast `max_blocks_per_fetch` which has `#[validate(range(min = 1))]`):

```rust
// crates/apollo_l1_events_config/src/config.rs  L122-L139
pub struct L1EventsScraperConfig {
    ...
    pub finality: u64,                    // no #[validate(range(min = N))]
    ...
    #[validate(range(min = 1))]
    pub max_blocks_per_fetch: u64,
}
``` [2](#0-1) 

The same `0` is published as the authoritative schema default:

```json
// crates/apollo_node/resources/config_schema.json  L3307-L3311
"l1_events_scraper_config.finality": {
    "description": "Number of blocks to wait for finality",
    "privacy": "Public",
    "value": 0
}
``` [3](#0-2) 

`fetch_start_block` and `fetch_events` both use `finality` as the ceiling offset:

```rust
// crates/apollo_l1_events/src/l1_scraper.rs  L138-L143
let latest_l1_block_number = latest_l1_block_number.checked_sub(finality).ok_or(
    L1EventsScraperError::LatestBlockNumberTooLow { ... }
)?;
``` [4](#0-3) 

With `finality = 0` the subtraction is a no-op: events from the very latest L1 block are immediately scraped and forwarded to the `L1EventsProvider`. The provider marks them as `Validated` and the batcher includes them in the next L2 block proposal. Once `commit_block` is called the transaction is removed from the pending pool and the L2 state root is updated to reflect its execution.

If Ethereum then re-orgs that L1 block away, the L1 handler transaction no longer exists on the canonical chain, but the L2 block has already been committed with its effects baked into the state root. There is no rollback path for the L2 side.

The deployment overlay files do override the value to `10`:

```json
// crates/apollo_deployments/resources/app_configs/l1_events_scraper_config.json
{ "l1_events_scraper_config.finality": 10, ... }
``` [5](#0-4) 

However, the Rust `Default` impl and the published `config_schema.json` both expose `0`, so any operator who generates a config from the schema without applying the overlay, or who calls `L1EventsScraperConfig::default()` directly (as every test harness does), silently inherits zero-confirmation scraping.

---

### Impact Explanation

An L1 re-org of depth ≥ 1 (routine on Ethereum; ~1–2 block re-orgs occur multiple times per day) is sufficient to trigger the condition. The sequencer commits an L2 block whose state root encodes the storage writes, nonce increments, and fee transfers produced by executing an L1 handler transaction that was subsequently erased from L1. The committed global root stored by the committer, the block hash written to storage, and the state diff commitment all reflect this phantom message. Downstream consumers (state sync, proof manager, SNOS input) read the corrupted root as authoritative.

This matches: **Critical — Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

---

### Likelihood Explanation

- Ethereum re-orgs of depth 1–2 are a normal, unprivileged, daily occurrence.
- Any operator who does not apply the deployment overlay (e.g., a new chain, a testnet, a self-hosted node) inherits `finality: 0` from the schema default.
- No code-level guard prevents `finality: 0` from being used; the `Validate` derive on the struct has no range constraint on the field.
- The `L1GasPriceScraperConfig` has the same `finality: 0` default, widening the surface. [6](#0-5) 

---

### Recommendation

1. **Enforce a safe minimum in the type system.** Add `#[validate(range(min = 3))]` (or a chain-appropriate constant) to `finality` in both `L1EventsScraperConfig` and `L1GasPriceScraperConfig`. Ethereum's expected re-org depth is ≤ 2 blocks; a minimum of 3–6 is a reasonable safe floor.

2. **Raise the `Default` value.** Change `finality: 0` to `finality: 6` (or the same constant) in both `Default` impls so that any code path that calls `::default()` is safe out of the box.

3. **Update `config_schema.json`.** The schema value must match the new default so that operators who generate configs from the schema receive a safe starting point.

4. **Add a startup assertion.** Log a prominent warning (or hard-error) if `finality < MINIMUM_SAFE_FINALITY` at scraper construction time, analogous to the existing `LatestBlockNumberTooLow` error path.

---

### Proof of Concept

1. Deploy the sequencer with `L1EventsScraperConfig::default()` (i.e., `finality: 0`).
2. Submit an L1→L2 message on Ethereum (e.g., via `sendMessageToL2`). The message lands in block `N`.
3. The scraper's next poll calls `fetch_events` with ceiling = `latest_block` (no finality offset). The event from block `N` is scraped immediately and forwarded to the `L1EventsProvider`.
4. The batcher includes the resulting `L1HandlerTransaction` in the next L2 block proposal. Consensus reaches decision; `commit_block` is called; the L2 state root is updated.
5. Ethereum re-orgs block `N` (a 1-block re-org). The `sendMessageToL2` call no longer exists on the canonical chain.
6. The L2 state root now permanently encodes the storage writes and fee effects of a message that was never sent on L1. The committed global root, block hash, and state diff commitment all reflect this phantom state. [1](#0-0) [4](#0-3) [3](#0-2)
