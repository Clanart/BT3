### Title
Detached Burn Promise in `burn_tokens_if_needed` Emits `InitTransferEvent` Before Confirming Token Destruction, Enabling Unbacked Cross-Chain Minting - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

`burn_tokens_if_needed` schedules a cross-contract burn call with `.detach()`, meaning its result is never awaited or checked. The `InitTransferEvent` (and `FastTransferEvent`) that drives destination-chain minting is emitted in the same synchronous transaction, before the burn executes. If the burn fails in its subsequent receipt, the NEAR-side bridged tokens remain unburned while the relayer has already observed the event and will mint tokens on the destination chain, creating unbacked supply.

---

### Finding Description

`burn_tokens_if_needed` is defined as:

```rust
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) {
    if self.is_deployed_token(&token) {
        ext_token::ext(token)
            .with_static_gas(BURN_TOKEN_GAS)
            .burn(amount)
            .detach();   // ← result is never observed
    }
}
``` [1](#0-0) 

`.detach()` in NEAR SDK schedules the promise but discards its outcome entirely. The calling function proceeds synchronously and logs the cross-chain event in the same transaction receipt, before the burn receipt has executed.

**Instance 1 — `init_transfer_internal`:**

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);  // detached
self.lock_tokens_if_needed(...);
// ...
env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
// ↑ emitted unconditionally, before burn result is known
``` [2](#0-1) 

**Instance 2 — `fast_fin_transfer_to_other_chain`:**

```rust
self.burn_tokens_if_needed(fast_transfer.token_id.clone(), amount_without_fee.into()); // detached
// ...
env::log_str(&OmniBridgeEvent::FastTransferEvent { ... }.to_log_string());
// ↑ emitted unconditionally
``` [3](#0-2) 

**Instance 3 — `resolve_fast_transfer`:**

```rust
self.burn_tokens_if_needed(token_id.clone(), amount); // detached, result ignored
``` [4](#0-3) 

In all three cases the event that the off-chain relayer monitors is written to the NEAR log before the burn receipt settles. NEAR receipts are atomic per-receipt, not per-transaction: the burn fires in a child receipt that can fail independently of the parent receipt that already logged the event.

---

### Impact Explanation

When `burn_tokens_if_needed` fails silently (e.g., gas exhaustion with the fixed `BURN_TOKEN_GAS` budget, a panic in the token contract, or the bridge losing its controller role on the token):

1. The bridged NEAR-side tokens are **not destroyed** — they remain held by the bridge contract.
2. The `InitTransferEvent` / `FastTransferEvent` has already been written to the NEAR receipt log.
3. The relayer reads that log and submits a `fin_transfer` on the destination chain (EVM, Solana, StarkNet), minting the full amount to the recipient.
4. The result is **unbacked supply**: destination-chain tokens exist without a corresponding burn on NEAR, breaking bridge collateralization.

This matches the allowed impact: *"Critical. Direct theft, unauthorized release, or unauthorized mint of native or bridged assets across NEAR, EVM, Solana, or StarkNet"* and *"High. Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."*

---

### Likelihood Explanation

The entry path is fully unprivileged: any token holder calls `ft_transfer_call` on a bridge-deployed token, which triggers `ft_on_transfer` → `init_transfer` → `init_transfer_internal`. The burn is then scheduled with a fixed static gas allocation (`BURN_TOKEN_GAS`). [5](#0-4) 

Gas exhaustion is a realistic failure mode in NEAR because the total prepaid gas for a `ft_transfer_call` chain is bounded, and the bridge already subtracts several gas reservations before reaching the burn step. Any future upgrade to the token contract that adds logic to `burn` also increases the risk. The failure is silent — no panic, no revert of the parent receipt — so neither the user nor the relayer has any indication that the burn did not occur.

---

### Recommendation

Replace the fire-and-forget `.detach()` pattern with a chained callback that checks the burn result before emitting the cross-chain event. The `InitTransferEvent` must only be logged after the burn is confirmed:

```rust
ext_token::ext(token_id.clone())
    .with_static_gas(BURN_TOKEN_GAS)
    .burn(transfer_message.amount)
    .then(
        Self::ext(env::current_account_id())
            .with_static_gas(INIT_TRANSFER_CALLBACK_GAS)
            .init_transfer_burn_callback(transfer_message, storage_owner),
    )
```

In `init_transfer_burn_callback`, check `env::promise_result_checked(0, ...)`: if the burn failed, revert the transfer message storage and return the full amount to the caller (refund via `ft_transfer_call` return value). Only on success emit `InitTransferEvent`. The same pattern must be applied in `fast_fin_transfer_to_other_chain` and `resolve_fast_transfer`.

---

### Proof of Concept

1. Alice holds 1000 units of a bridge-deployed NEAR token (e.g., `usdc.bridge.near`).
2. Alice calls `ft_transfer_call` on `usdc.bridge.near`, sending 1000 tokens to the bridge with an `InitTransfer` message targeting an EVM recipient.
3. The bridge's `ft_on_transfer` → `init_transfer` → `init_transfer_internal` executes:
   - `burn_tokens_if_needed` schedules a detached burn receipt for 1000 tokens.
   - `InitTransferEvent` is logged in the same receipt.
   - `init_transfer_internal` returns `U128(0)` — no refund — so the 1000 tokens stay in the bridge.
4. The burn receipt executes next and fails (e.g., gas exhaustion). The 1000 tokens remain in the bridge contract. No error is surfaced to the parent receipt.
5. The relayer observes the `InitTransferEvent` log (already finalized in step 3) and calls `fin_transfer` on the EVM bridge, minting 1000 USDC to Alice's EVM address.
6. Result: Alice holds 1000 EVM USDC (minted) **and** 1000 NEAR-side tokens remain unburned in the bridge contract — unbacked supply has been created. [1](#0-0) [6](#0-5)
