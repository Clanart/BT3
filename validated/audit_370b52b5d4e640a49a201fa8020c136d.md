### Title
`VirtualBlockExecutor` Hardcodes `NativeClassesWhitelist::All`, Bypassing Configured Whitelist and Causing Prover/Batcher Compiled-Class Divergence — (`crates/starknet_transaction_prover/src/running/virtual_block_executor.rs`)

---

### Summary

The transaction prover's `VirtualBlockExecutor::execute()` constructs its `StateReaderAndContractManager` via the no-whitelist constructor `::new()`, which unconditionally sets `NativeClassesWhitelist::All`. The batcher's `BlockBuilderFactory` correctly passes the operator-configured whitelist (which may be `Limited([…])`) through `::new_with_native_classes_whitelist()`. When the batcher is configured with a `Limited` whitelist, any class hash outside that list is executed as CASM by the batcher but as Cairo Native by the prover. Because native and CASM are distinct compiled artifacts, they can produce divergent execution outputs (different storage writes, events, return values, gas). The prover feeds those divergent outputs into `OsHints` → `StarknetOsInput` → `run_virtual_os()`, generating a Cairo PIE that attests to wrong execution results.

---

### Finding Description

**Root cause — `VirtualBlockExecutor::execute()` ignores the configured whitelist:** [1](#0-0) 

```rust
let state_reader_and_contract_manager =
    StateReaderAndContractManager::new(state_reader, contract_class_manager, None);
```

`StateReaderAndContractManager::new()` is a convenience constructor that hardcodes `NativeClassesWhitelist::All`: [2](#0-1) 

**Correct path — batcher passes the configured whitelist:**

`BlockBuilderFactory::preprocess_and_create_transaction_executor()` uses the explicit constructor: [3](#0-2) 

The `native_classes_whitelist` argument comes from `BatcherDynamicConfig`, which is operator-configurable and can be `Limited(Vec<ClassHash>)`: [4](#0-3) 

**How the whitelist controls compiled-class selection:**

`NativeClassManager::get_runnable()` downgrades a `V1Native` class to CASM when the class hash is absent from a `Limited` whitelist: [5](#0-4) 

So when the batcher runs with `Limited([A, B])` and a transaction invokes class hash `C` (not in the list), the batcher executes `C` as `RunnableCompiledClass::V1` (CASM). The prover, using `All`, executes `C` as `RunnableCompiledClass::V1Native`. These are distinct compiled artifacts.

**Prover feeds execution outputs directly into OS hints and proof generation:** [6](#0-5) [7](#0-6) 

The `tx_execution_infos` (storage diffs, events, return values, L2→L1 messages) from the divergent native execution are embedded verbatim into `OsBlockInput` and passed to `run_virtual_os()`, producing a Cairo PIE that proves wrong state.

**Secondary affected path — gateway stateful validator also uses `::new()`:** [8](#0-7) 

The gateway validates and estimates fees using `NativeClassesWhitelist::All` while the batcher may execute the same transaction with CASM, causing fee estimation and validation results to diverge from actual execution.

---

### Impact Explanation

When the batcher is configured with `NativeClassesWhitelist::Limited([…])`:

1. Any class hash `C` not in the list is executed as CASM by the batcher (committed state diff reflects CASM semantics).
2. The prover re-executes `C` as Cairo Native (whitelist bypassed), potentially producing different storage writes, events, return values, or gas consumption.
3. The prover's `tx_execution_infos` — which feed directly into `OsHints` and `StarknetOsInput` — reflect the native execution, not the CASM execution that was actually committed.
4. `run_virtual_os()` generates a Cairo PIE attesting to the wrong execution result.
5. The resulting ZK proof is for a state transition that was never committed, or the proof fails to verify against the committed state root.

This matches: **Critical — Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution**, and **Critical — Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

---

### Likelihood Explanation

- Trigger requires the batcher to be configured with `NativeClassesWhitelist::Limited` (non-default). The default is `All`, so the mismatch is latent until an operator restricts the whitelist.
- Once `Limited` is configured (e.g., to gate native execution to audited contracts only), any user who deploys or invokes a contract whose class hash is outside the list will trigger the divergence — no special privilege required.
- The `cairo_native` feature must be enabled and native compilation must succeed for the class in question.

---

### Recommendation

Pass the configured `NativeClassesWhitelist` through to `VirtualBlockExecutor::execute()` and use `StateReaderAndContractManager::new_with_native_classes_whitelist()` instead of `::new()`. The whitelist should be sourced from the same configuration object used by the batcher so that both execution contexts select identical compiled artifacts for every class hash.

Apply the same fix to `StatefulTransactionValidatorFactory::instantiate_validator()` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` line 80–84.

---

### Proof of Concept

1. Configure the batcher with `native_classes_whitelist = Limited(["0xAAA"])` (only class `0xAAA` runs natively).
2. Deploy a Cairo 1 contract with class hash `0xBBB` whose native and CASM execution produce different storage writes (e.g., a contract that behaves differently under native due to a known native-compiler edge case).
3. Submit an Invoke transaction calling `0xBBB`.
4. The batcher executes `0xBBB` as CASM (`V1`), writes storage value `X`, commits state diff with `storage[0xBBB] = X`.
5. The transaction prover calls `VirtualBlockExecutor::execute()` → `StateReaderAndContractManager::new(…)` → `NativeClassesWhitelist::All` → executes `0xBBB` as `V1Native`, writes storage value `Y ≠ X`.
6. `tx_execution_infos` contains `storage[0xBBB] = Y`; this is embedded into `OsBlockInput` and passed to `run_virtual_os()`.
7. The resulting Cairo PIE and ZK proof attest to `storage[0xBBB] = Y`, contradicting the committed state diff `storage[0xBBB] = X`.
