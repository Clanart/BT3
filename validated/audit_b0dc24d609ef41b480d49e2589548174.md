### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS accepts any arbitrary `class_hash` value from the `ReplaceClassRequest` without validating that the new class hash corresponds to a previously declared class. This is the direct analog of the ERC20 approve bug: just as the Alchemix code failed to enforce a prerequisite state (allowance = 0) before updating, the OS here fails to enforce a prerequisite state (class must be declared) before accepting a class replacement. A contract can call `replace_class` with an undeclared hash (including `0`), permanently making itself uncallable and freezing any funds it holds.

---

### Finding Description

**Vulnerability class:** Missing prerequisite state validation before a state update.

In `execute_replace_class` (lines 878–916 of `syscall_impls.cairo`), the OS reads the new `class_hash` directly from the syscall request and writes it into `contract_state_changes` without any check that the hash is a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing check. Contrast this with `execute_declare_transaction`, which correctly enforces `prev_value=0` to ensure a class can only be declared once:

```cairo
// Note that prev_value=0 enforces that a class may be declared only once.
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [2](#0-1) 

The `execute_replace_class` function has no equivalent prerequisite guard. A contract can call `replace_class(0)` or `replace_class(any_undeclared_hash)` and the OS will commit that invalid class hash to state.

When any subsequent transaction attempts to call the affected contract, `execute_entry_point` reads the invalid `class_hash`, performs `dict_read{dict_ptr=contract_class_changes}(key=0)` which returns `0` (since class `0` was never declared), and then calls `find_element` searching for a compiled class with hash `0`:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [3](#0-2) 

Since no compiled class with hash `0` exists in the bundle, `find_element` fails. The sequencer will reject every future transaction targeting this contract, making the contract permanently uncallable.

---

### Impact Explanation

**Critical. Permanent freezing of funds.**

Any contract that calls `replace_class` with an undeclared class hash (including `0`) is permanently frozen. The contract becomes uncallable because the OS cannot resolve its class hash to a compiled class. All funds (tokens, ETH, STRK) held by the contract at the time of the invalid `replace_class` call are permanently inaccessible. There is no recovery path: the state is committed on-chain with the invalid class hash, and no transaction can successfully execute against the contract.

---

### Likelihood Explanation

**Medium.** The `replace_class` syscall is callable by any contract from within its own execution context — no privileged role is required. Realistic trigger paths include:

1. A contract with a flawed upgrade mechanism that allows calling `replace_class` with an attacker-supplied or zero class hash.
2. A contract that conditionally calls `replace_class` based on user input without validating the hash.
3. A contract intentionally designed to self-destruct by calling `replace_class(0)`, locking user deposits.

The OS-level missing check is the necessary vulnerable step: the sequencer's off-chain simulation would accept the `replace_class` syscall (since the OS does not reject it), commit the invalid state, and then reject all future calls to the contract.

---

### Recommendation

Add a validation in `execute_replace_class` that the new `class_hash` is a declared class before committing the state update. Specifically, perform a `dict_read` on `contract_class_changes` for the new `class_hash` and assert the result is non-zero (i.e., the class has a valid compiled class hash):

```cairo
// Validate that the new class hash is a declared class.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the pattern already used in `execute_declare_transaction` where `assert_not_zero(compiled_class_hash)` is enforced before the dict update.

---

### Proof of Concept

1. Deploy a contract `VictimVault` that holds user funds and exposes a public `upgrade(new_hash)` function that calls `replace_class(new_hash)` without validating `new_hash`.
2. An attacker calls `VictimVault.upgrade(0)`.
3. The OS executes `execute_replace_class` with `class_hash = 0`. No validation is performed. The state is committed with `VictimVault.class_hash = 0`.
4. Any user attempts to call `VictimVault.withdraw()`. The sequencer simulates the transaction: `execute_entry_point` reads `class_hash = 0`, looks up `compiled_class_hash = 0`, and `find_element` fails to locate a compiled class with hash `0`.
5. The sequencer rejects the transaction. All funds in `VictimVault` are permanently frozen.

The root cause is exclusively in the OS at `execute_replace_class` (lines 878–916 of `syscall_impls.cairo`): the missing prerequisite validation that the new class hash is declared — the direct analog of the missing `approve(spender, 0)` reset in the external report.
