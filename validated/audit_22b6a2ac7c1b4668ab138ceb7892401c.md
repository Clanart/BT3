### Title
Wrapping Integer Arithmetic in Relative Timelock Checks Enables Timelock Bypass — (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary
`check_time_locks` contains a `nowrap: bool` parameter that selects between `saturating_add` (safe) and `wrapping_add` (unsafe) for all four relative timelock conditions. When called with `nowrap=false`, an attacker can supply a near-maximum relative timelock value that wraps around to a small integer, causing the timelock guard to pass immediately and allowing a spend to be accepted before its required height or timestamp.

### Finding Description
In `crates/chia-consensus/src/check_time_locks.rs`, the function `check_time_locks` evaluates four relative conditions:

- `ASSERT_HEIGHT_RELATIVE` (lines 55–68)
- `ASSERT_SECONDS_RELATIVE` (lines 70–78)
- `ASSERT_BEFORE_HEIGHT_RELATIVE` (lines 79–98)
- `ASSERT_BEFORE_SECONDS_RELATIVE` (lines 100–112)

For each, the `nowrap=false` branch uses `wrapping_add`:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

`confirmed_block_index.wrapping_add(height_relative)` silently wraps on overflow. For example, if a coin was confirmed at block 100 and the spend asserts `height_relative = 0xffff_ffff` (u32::MAX), the sum wraps to `99`. The guard then checks `prev_height < 99`, which passes for any `prev_height >= 99`, so the spend is accepted immediately — even though the intended deadline is block `100 + 4,294,967,295`, far in the future.

The same overflow applies to `seconds_relative` (u64 wrapping), `before_height_relative`, and `before_seconds_relative`, with the last two producing the inverse error (a valid spend is incorrectly rejected).

The `nowrap` flag is exposed directly through the Python binding `py_check_time_locks` in the same file (lines 122–141), meaning the Python consensus layer controls which arithmetic mode is active. If the Python caller passes `nowrap=False` (the original pre-fix behavior), the wrapping path is live in production consensus.

This is the direct analog of the Solidity report: code originally written for an environment with implicit wrapping arithmetic (old Python/Rust behavior) was not updated to use explicit saturating arithmetic when the behavior was made configurable.

### Impact Explanation
**High.** Timelock bypass via `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` with a crafted near-maximum value allows a spend to be accepted before its required block height or timestamp. This enables unauthorized spend acceptance — a coin locked until a future height can be spent immediately. This matches the allowed High impact: "timelock or condition validation bypass enables unauthorized spend acceptance."

### Likelihood Explanation
The attacker-controlled entry path is a standard unprivileged spend bundle. Any user can include `ASSERT_HEIGHT_RELATIVE` with value `u32::MAX - confirmed_height + 1` in a CLVM puzzle solution. No special privileges, keys, or network access are required. Likelihood is **High** if `nowrap=False` is the active mode in the Python consensus path (the original default before the flag was introduced).

### Recommendation
1. Remove the `nowrap=false` / `wrapping_add` branch entirely from `check_time_locks`. Saturating arithmetic (`saturating_add`) is the only semantically correct behavior for relative timelock addition.
2. If backward compatibility with old chain state is required, document precisely which block height activates `nowrap=true` and hard-code that transition rather than leaving it as a caller-controlled parameter.
3. Audit all Python callers of `check_time_locks` to confirm `nowrap=True` is passed unconditionally.

### Proof of Concept
Craft a spend for a coin confirmed at block height `H`:

```
ASSERT_HEIGHT_RELATIVE = (u32::MAX - H + 1)  # e.g., H=100 → value=0xffff_ff9b
```

With `nowrap=False`:
```
H.wrapping_add(0xffff_ff9b) = 0  (wraps to 0)
prev_height < 0  →  false  →  check passes → spend accepted at any block height
```

The test suite in `check_time_locks.rs` (lines 276–281) and `tests/test_check_time_locks.py` (lines 241–243) explicitly document this bypass:

```

#[cfg(feature = "py-bindings")]
#[pyfunction]
#[pyo3(name = "check_time_locks")]
#[allow(clippy::needless_pass_by_value)] // pyo3 prefers pass_by_value
pub fn py_check_time_locks(
    removal_coin_records: HashMap<Bytes32, CoinRecord>,
    bundle_conds: &OwnedSpendBundleConditions,
    prev_transaction_block_height: u32,
    timestamp: u64,
    nowrap: bool,
) -> PyResult<Option<u32>> {
    let res = check_time_locks(
        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,
        nowrap,
    );

    match res {
        Ok(()) => Ok(None),
        Err(ec) => Ok(Some(ec.error_code().into())),
    }
}
