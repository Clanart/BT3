### Title
Fee Bound Check Uses Raw Denormalized Amount Instead of Normalized Transfer Amount, Enabling Permanent Fund Lock - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

The fee validity check in `init_transfer` and `update_transfer_fee` compares `fee.fee` against the raw (denormalized, NEAR-native) `transfer.message.amount`. However, the actual amount delivered to the destination chain is `normalize(amount - fee.fee)`, which applies floor division by `10^(origin_decimals - decimals)`. When a user sets a fee such that `fee.fee < amount` (passes the bound check) but `normalize(amount - fee.fee) == 0` (fails the downstream check in `sign_transfer`), the transfer is permanently uncompletable and the user's tokens are irrecoverably locked or burned.

---

### Finding Description

**Root cause — wrong quantity in the bound check:**

In `init_transfer` (`near/omni-bridge/src/lib.rs`, line 554):

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

And identically in `update_transfer_fee` (line 399):

```rust
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

Both checks operate on the raw NEAR-native (denormalized) units. [1](#0-0) 

However, in `sign_transfer`, the amount actually signed and delivered to the destination chain is computed as:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

`normalize_amount` performs floor division by `10^(origin_decimals - decimals)`:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [3](#0-2) 

The fee bound check and the actual transfer-amount check are therefore operating on **different quantities**: the bound check uses the raw amount, while the downstream check uses the normalized (destination-chain-scaled) amount. This is the direct analog of the external report's "incorrect bound check" — the wrong quantity is being validated.

**Concrete scenario:**

- Token: NEAR-native token with `origin_decimals = 24`, bridging to EVM with `decimals = 18`. `diff_decimals = 6`, granularity = `10^6`.
- User initiates transfer with `amount = 10^6` (1 micro-token in NEAR units).
- User sets `fee.fee = 10^6 - 1`.
- Bound check: `10^6 - 1 < 10^6` → **passes**.
- `amount_without_fee = 10^6 - (10^6 - 1) = 1`.
- `normalize_amount(1) = 1 / 10^6 = 0` (floor division).
- `sign_transfer` panics with `InvalidAmountToTransfer` — **every signing attempt fails**.

**No recovery path exists.** In `init_transfer_internal`, tokens are either burned (deployed/bridged tokens) or locked before the transfer message is stored:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
``` [4](#0-3) 

There is no user-accessible cancel or refund function. `remove_transfer_message_without_refund` does not return tokens. The transfer message remains in `pending_transfers` permanently, and the locked/burned tokens are irrecoverable.

Additionally, `update_transfer_fee` only allows fee **increases** (`fee.fee >= current_fee.fee`), so once a user sets a fee that makes `normalize(amount - fee) == 0`, they cannot reduce it to recover the transfer. [5](#0-4) 

---

### Impact Explanation

Any user who initiates a NEAR-origin transfer for a token with `origin_decimals > decimals` (e.g., any NEAR-native token bridging to EVM or Solana) and sets a fee within the sub-unit granularity of the destination chain will have their tokens permanently locked or burned with no recovery. This matches the allowed impact: **Permanent freezing, irrecoverable lock of user funds in bridge flows**.

---

### Likelihood Explanation

NEAR-native tokens commonly have 24 decimals while EVM tokens use 18, giving a granularity of `10^6`. Any transfer amount below `10^6` NEAR-native units (or any fee set within `10^6` of the full amount) triggers this. A user can trigger this on themselves accidentally (e.g., setting fee = amount - 1 to maximize relayer incentive) or it can be triggered at `init_transfer` time without any `update_transfer_fee` call. The entry path is fully unprivileged: any user calling `ft_transfer_call` on a NEAR token contract.

---

### Recommendation

Replace the raw-amount fee bound check with a check that validates the normalized residual is nonzero:

```rust
// In init_transfer and update_transfer_fee:
let decimals = self.token_decimals.get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);
let amount_after_fee = transfer_message.amount.0
    .checked_sub(fee.fee.0)
    .near_expect(BridgeError::InvalidFee);
require!(
    Self::normalize_amount(amount_after_fee, decimals) > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

This ensures the bound check and the downstream `sign_transfer` check are operating on the **same quantity** — the normalized destination-chain amount — eliminating the discrepancy.

---

### Proof of Concept

1. Register a NEAR-native token with `origin_decimals = 24`, `decimals = 18` (diff = 6, granularity = `10^6`).
2. Call `ft_transfer_call` with `amount = 5 * 10^6` and `fee = 5 * 10^6 - 1`.
3. Bound check `fee < amount` passes: `5*10^6 - 1 < 5*10^6`.
4. `init_transfer_internal` burns/locks `5 * 10^6` NEAR-native units.
5. Relayer calls `sign_transfer`:
   - `amount_without_fee = 5*10^6 - (5*10^6 - 1) = 1`
   - `normalize_amount(1, {origin=24, dest=18}) = 1 / 10^6 = 0`
   - Panics: `ERR_INVALID_AMOUNT_TO_TRANSFER`
6. No cancel/refund function exists. Tokens are permanently locked. [2](#0-1) [3](#0-2) [5](#0-4) [1](#0-0) [4](#0-3)
