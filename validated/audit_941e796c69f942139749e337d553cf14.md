### Title
`MerkleBlob::batch_insert` Bypasses Duplicate-Key/Hash Validation for Items Beyond the Bootstrap Phase, Silently Corrupting the Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` uses two distinct code paths: a bootstrap path (≤1 existing leaves) that delegates to the validated `insert()` method, and a fast-path (≥2 existing leaves) that writes leaf nodes directly to the blob without checking the `block_status_cache` for duplicate keys or hashes. An unprivileged caller supplying a batch containing a key or hash already present in the tree — or a batch with internal duplicates — silently inserts orphaned or colliding leaf nodes, corrupting the Merkle root hash and invalidating all subsequent proofs of inclusion.

### Finding Description

`MerkleBlob::insert()` enforces two invariants before writing any node:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert()` calls `insert()` only for the first two items when the tree has ≤1 existing leaves:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { return Ok(()); };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;
    }
}
``` [2](#0-1) 

All remaining items are written directly via `insert_entry_to_blob` with **no duplicate check**:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [3](#0-2) 

The `block_status_cache` — which is the sole runtime guard against duplicates — is never consulted for these items. When a duplicate key or hash is present, a second leaf node is written into the blob. The cache's `key_to_index` map is then updated to point to the new index, leaving the original leaf node as a phantom still referenced by its parent internal node. The root hash is subsequently computed over both the live and phantom subtrees, producing a root that does not correspond to any valid, duplicate-free tree state.

The `get_proof_of_inclusion` path reads the sibling hash from the blob at the index stored in the cache:

```rust
let sibling_block = self.get_block(sibling_index)?;
let layer = proof_of_inclusion::ProofOfInclusionLayer {
    other_hash_side: parent.get_sibling_side(index)?,
    other_hash: sibling.hash(),
    combined_hash: parent.hash,
};
``` [4](#0-3) 

Because the parent internal node still references the phantom leaf, the `combined_hash` stored in the parent diverges from what `ProofOfInclusion::valid()` recomputes bottom-up:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(...);
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()
}
``` [5](#0-4) 

The result is that `proof.valid()` returns `false` for legitimately inserted keys, or `true` for a proof anchored to a corrupted root — depending on which leaf the cache resolves to and which phantom nodes remain in the lineage.

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

Concretely:
- The Merkle root stored and published by the DataLayer node is computed over a structurally invalid tree containing phantom leaf nodes.
- Proofs of inclusion generated from this root are either spuriously invalid (legitimate keys fail `valid()`) or anchored to a root that no honest peer can reproduce from the same key-value set.
- A peer receiving the corrupted root and a proof cannot distinguish a legitimately corrupted tree from a forged one, enabling either denial of valid state or acceptance of invalid state.

### Likelihood Explanation

`batch_insert` is the primary high-throughput insertion path for DataLayer, exposed directly via the Python binding `py_batch_insert`:

```rust
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [6](#0-5) 

Any caller — including the DataLayer Python layer — that supplies a batch containing a key already present in the tree (e.g., an update operation implemented as a batch re-insert rather than `upsert`) silently triggers the corruption. No existing fuzz target exercises `batch_insert` with duplicate keys; all fuzz targets use only `insert()`. [7](#0-6) 

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the fast path of `batch_insert` that `insert()` enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing write logic
}
```

Alternatively, pre-validate the entire batch (including intra-batch duplicates) before writing any node, to preserve the atomicity guarantee. Add a fuzz target that exercises `batch_insert` with duplicate keys and hashes, and add `check_integrity` assertions after every `batch_insert` call in the existing test suite.

### Proof of Concept

```rust
let mut blob = MerkleBlob::new(vec![]).unwrap();

// Insert 3 unique keys to move past the bootstrap phase
for i in 0i64..3 {
    blob.insert(KeyId(i), ValueId(i), &sha256_num(&i), InsertLocation::Auto {}).unwrap();
}

// batch_insert with a key already present in the tree (KeyId(0))
// No error is returned — the duplicate is silently written
blob.batch_insert(vec![
    ((KeyId(0), ValueId(99)), sha256_num(&99i64)),  // duplicate key
    ((KeyId(100), ValueId(100)), sha256_num(&100i64)),
]).unwrap();

blob.calculate_lazy_hashes().unwrap();

// check_integrity now fails: leaf_count in blob > key_to_index cache length
// OR the root hash is computed over phantom nodes
blob.check_integrity().expect_err("tree is corrupted");

// Proof for KeyId(0) is now invalid or anchored to a corrupted root
let proof = blob.get_proof_of_inclusion(KeyId(0)).unwrap();
assert!(!proof.valid()); // fails — proof is broken
