### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Causes Permanent Fund Lock — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol::initTransfer` uses the caller-supplied `amount` parameter in the `InitTransfer` event without verifying the actual amount received from `safeTransferFrom`. For fee-on-transfer ERC-20 tokens the bridge receives `amount − fee_amount`, but the event records the full `amount`. The NEAR bridge then mints the full (denormalized) `amount` to the recipient, creating an unbacked surplus. When the user later redeems on EVM, `finTransfer` attempts `safeTransfer(recipient, amount)` against a contract balance of only `amount − fee_amount`, causing a revert and permanently locking the user's NEAR-side tokens.

---

### Finding Description

In `initTransfer`, the non-bridge-token branch transfers tokens from the caller:

```solidity
// OmniBridge.sol lines 406-412
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ← user-specified, not actual received
    );
}
```

Immediately after, the event is emitted with the same unverified `amount`:

```solidity
// OmniBridge.sol lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,             // ← original param, not balanceOf delta
    fee,
    nativeFee,
    recipient,
    message
);
```

The NEAR prover trusts this event. In `fin_transfer_callback`, the emitted `amount` is denormalized and stored as the canonical transfer amount:

```rust
// near/omni-bridge/src/lib.rs line 725
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
```

The NEAR bridge then mints `denormalize(amount)` tokens to the recipient — more than the EVM bridge actually holds. On the return leg, `sign_transfer` normalizes back to `amount` and the EVM `finTransfer` calls:

```solidity
// OmniBridge.sol lines 351-354
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount      // ← amount the bridge does not fully hold
);
```

This reverts. Because the NEAR side has already burned/locked the user's tokens before the EVM transaction is submitted, the revert leaves those tokens irrecoverable.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

Any user who bridges a fee-on-transfer ERC-20 token from EVM to NEAR receives the correct NEAR-side balance, but can never redeem it on EVM. The EVM bridge is permanently under-collateralized by the cumulative transfer fees, and every subsequent redemption attempt reverts. The NEAR tokens are burned before the EVM call is made, so there is no rollback path.

---

### Likelihood Explanation

**Medium.** The trigger requires a fee-on-transfer token to be accepted by the bridge (i.e., not a bridge token and not a custom-minter token). Such tokens exist on mainnet (e.g., tokens with reflection or tax mechanics). Any ordinary user who calls `initTransfer` with such a token triggers the condition; no privileged access is required.

---

### Recommendation

Replace the fixed-`amount` transfer with a before/after balance check, mirroring the recommended fix from H-01:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 actualReceived = uint128(
        IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore
    );
    // use actualReceived in place of amount below
    emit BridgeTypes.InitTransfer(
        msg.sender, tokenAddress, currentOriginNonce,
        actualReceived, fee, nativeFee, recipient, message
    );
}
```

This ensures the emitted amount always matches the bridge's actual custody, keeping EVM and NEAR accounting in sync.

---

### Proof of Concept

1. A fee-on-transfer token (1 % fee) is registered with the bridge (not `isBridgeToken`, no `customMinters` entry).
2. User calls `initTransfer(token, 1000, 0, 0, nearRecipient, "")`.
3. `safeTransferFrom` executes; bridge receives **990** tokens (10 deducted as transfer fee).
4. `InitTransfer` event emits `amount = 1000`.
5. NEAR prover verifies the event; `fin_transfer_callback` denormalizes `1000` and mints **1000 × 10¹²** NEAR-side tokens to the recipient.
6. Recipient bridges back: NEAR burns **1000 × 10¹²** tokens, `sign_transfer` normalizes to **1000**, MPC signs.
7. Relayer submits signed payload to EVM `finTransfer`.
8. `IERC20(token).safeTransfer(recipient, 1000)` reverts — bridge holds only **990**.
