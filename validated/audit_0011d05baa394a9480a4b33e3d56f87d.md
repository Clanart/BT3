### Title
Unrestricted `meta_tx_v0` Syscall Allows Any Contract to Bypass `__validate__` and Invoke `__execute__` on Arbitrary Victim Accounts — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` function in the StarkNet OS implements a syscall that any user-deployed contract can invoke to call the `__execute__` entry point of **any other contract** with `caller_address = ORIGIN_ADDRESS` (0), bypassing `__validate__` entirely. This is the direct StarkNet analog of the reported tx.origin vulnerability: just as a relayer's `tx.origin` identity can be exploited by untrusted code in `xReceive`, the OS's privileged `ORIGIN_ADDRESS` identity can be impersonated by any contract via `meta_tx_v0`, enabling an attacker to execute arbitrary calls on behalf of a victim account without a valid signature.

---

### Finding Description

`execute_meta_tx_v0` in `syscall_impls.cairo` (lines 286–400) is a syscall handler callable by any contract during execution. It constructs a new execution context targeting the `__execute__` entry point of an attacker-specified `contract_address`: [1](#0-0) 

The constructed `ExecutionContext` sets:

- `caller_address = ORIGIN_ADDRESS` (the constant `0`) — the same identity the OS uses for all top-level transaction executions
- `version = 0` — a special version that many account contracts treat as a trusted legacy path
- `signature_start/end` = attacker-supplied bytes
- `transaction_hash` = a hash computed only from attacker-controlled calldata [2](#0-1) 

Critically, **`__validate__` is never called**. The OS's normal transaction flow calls `run_validate` before `__execute__`: [3](#0-2) 

`meta_tx_v0` skips this entirely and jumps directly to `contract_call_helper` → `execute_entry_point` on the victim's `__execute__`.

There is **no access control** on which contract may issue this syscall. Any contract executing inside the OS can emit a `MetaTxV0Request` syscall targeting any address.

The only selector restriction is: [4](#0-3) 

This restriction forces the call to land on `__execute__` — the function that actually moves funds — making the restriction counterproductive from a security standpoint.

Standard account contracts (e.g., OpenZeppelin-style) check `get_caller_address() == 0` in `__execute__` as the sole guard (trusting that only the OS can call with address 0), then execute the provided calldata without re-validating the signature. The `meta_tx_v0` syscall satisfies the `caller_address == 0` check while supplying attacker-controlled calldata.

---

### Impact Explanation

**Critical — Direct loss of funds.**

An attacker can drain the entire token balance of any account contract whose `__execute__` function:
1. Checks `get_caller_address() == 0` as its trust gate (standard pattern), and
2. Does not independently re-validate the signature against the transaction hash (standard for non-meta-tx accounts).

The attacker controls the calldata passed to `__execute__`, so they can encode an ERC-20 `transfer` call sending the victim's funds to the attacker's address. Because `__validate__` is bypassed, no cryptographic proof of the victim's consent is required.

---

### Likelihood Explanation

**High.** The attack requires only:
- Deploying a malicious contract (permissionless on StarkNet).
- Submitting one invoke transaction that calls `meta_tx_v0` with the victim's address and attacker-crafted calldata.

The vast majority of deployed account contracts follow the standard pattern where `__execute__` trusts `caller_address == 0` without re-checking the signature. No privileged access, leaked key, or operator cooperation is needed.

---

### Recommendation

1. **Restrict the syscall to authorized callers**: Only allow `meta_tx_v0` to be issued from a whitelisted set of relayer contracts or from the OS itself, not from arbitrary user contracts.
2. **Enforce `__validate__` before `__execute__`**: The OS should call the target contract's `__validate_meta_tx__` (or equivalent) entry point before dispatching to `__execute__`, mirroring the protection provided in the normal invoke flow.
3. **Document the security invariant**: If the current design is intentional, the protocol must explicitly document that any account contract wishing to be safe must re-validate the signature inside `__execute__` when `tx_info.version == 0`, and that `caller_address == 0` is **not** a sufficient trust check.

---

### Proof of Concept

```
1. Attacker deploys MaliciousRelayer contract containing:

   fn attack(victim: ContractAddress, attacker: ContractAddress) {
       // Encode: transfer(attacker, victim_balance)
       let calldata = encode_transfer_call(attacker, MAX_UINT256);
       meta_tx_v0(
           contract_address = victim,
           selector         = EXECUTE_ENTRY_POINT_SELECTOR,  // __execute__
           calldata         = calldata,
           signature        = [],   // empty — __validate__ is never called
       );
   }

2. Attacker submits invoke tx → MaliciousRelayer.attack(victim, attacker).

3. OS dispatches execute_meta_tx_v0:
   - Constructs ExecutionContext with caller_address = ORIGIN_ADDRESS (0),
     version = 0, calldata = [transfer(attacker, MAX_UINT256)].
   - Calls victim.__execute__ directly (no __validate__).

4. victim.__execute__ sees get_caller_address() == 0  ✓  (trust check passes).
   Executes transfer(attacker, MAX_UINT256) on the fee token or any ERC-20
   the victim holds.

5. Victim's funds are transferred to attacker. __validate__ was never invoked.
   No signature was verified.
```
