### Title
Missing `chunk.shard_id()` == `shard_id` binding in `set_state_header` allows cross-shard trie corruption via crafted `ShardStateSyncResponseHeader` — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for a caller-supplied `shard_id=B` but never asserts that the embedded chunk's `shard_id()` equals B. Because `verify_path` checks only Merkle membership (not leaf position), a chunk belonging to shard A passes the `chunk_headers_root` proof check. A malicious peer can craft a header containing shard A's chunk with shard B's valid on-chain receipt proofs, pass all five validation steps, and cause `StateHeaderKey(B, sync_hash)` to store shard A's chunk. All subsequent `set_state_part` and `set_state_finalize` calls for shard B then install shard A's trie into shard B's `ShardUId` column.

### Finding Description

`set_state_header` performs five checks before persisting the header:

1. **`validate_chunk_proofs`** — verifies internal chunk consistency (hash, tx root, receipts root). No shard_id binding. [1](#0-0) 

2. **`verify_path` for chunk** — verifies `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a member of `sync_prev_block_header.chunk_headers_root()`. This is a pure membership proof; it does not verify the leaf's index (i.e., which shard slot the chunk occupies). Shard A's chunk IS in the block's Merkle tree, so its proof verifies regardless of which shard is requested. [2](#0-1) [3](#0-2) 

3. **`verify_path` for prev_chunk** — same membership-only check for shard A's prev chunk header. Passes for the same reason. [4](#0-3) 

4. **Receipt proof check** — computes `receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))` using the **caller-supplied** `shard_id=B`, not `chunk.shard_id()`. The attacker supplies shard B's actual on-chain incoming receipt proofs (publicly available), which pass `verify_path` against the block's `prev_chunk_outgoing_receipts_root`. [5](#0-4) 

5. **`validate_state_root_node`** — validates the `state_root_node` against `chunk_inner.prev_state_root()`. The attacker supplies shard A's state root node, which is valid for shard A's `prev_state_root`. Passes. [6](#0-5) 

After all checks pass, the header is stored under `StateHeaderKey(B, sync_hash)` using the caller-supplied `shard_id=B`, with shard A's chunk embedded inside. [7](#0-6) 

The missing guard is simply: `if chunk.shard_id() != shard_id { return Err(...) }`.

**Downstream corruption in `set_state_part`:** retrieves the stored header for `shard_id=B`, extracts `state_root = chunk.prev_state_root()` (shard A's root), and calls `validate_state_part(B, shard_A_state_root, ...)`. Shard A's state parts validate correctly against shard A's root, and are stored under `StatePartKey(sync_hash, B, part_id)`. [8](#0-7) 

**Downstream corruption in `set_state_finalize`:** computes `shard_uid` from `shard_id=B` but calls `apply_chunk` with `prev_state_root = chunk_header.prev_state_root()` (shard A's root) and shard A's transactions/receipts. The result is committed to shard B's `ShardUId` column. [9](#0-8) 

### Impact Explanation

The exact corrupted value is: `StateHeaderKey(B, sync_hash)` stores a `ShardStateSyncResponseHeader` where `chunk.shard_id() == A` and `chunk.prev_state_root()` == shard A's state root. After `set_state_finalize`, shard B's `ShardUId` column in RocksDB contains shard A's trie state. The node will operate with the wrong state for shard B for the remainder of the epoch, producing incorrect execution results, wrong state roots in chunk headers, and diverging from the honest chain.

### Likelihood Explanation

The attack requires the adversary to be a peer that the victim node sends state sync requests to. Any node can advertise itself as a snapshot host via `SnapshotHostEvent::SnapshotCreated` with no authentication. State sync responses are accepted from any peer via `PeerMessage::VersionedStateResponse` → `StateResponseReceived` → `apply_peer_message`. The crafted header requires only publicly available on-chain data (shard A's chunk, its Merkle proof, shard B's receipt proofs). The attack is feasible whenever shard A and shard B share the same `prev_chunk_header.height_included()` (the common case when all shards are included in every block). [10](#0-9) [11](#0-10) 

### Recommendation

Add an explicit shard_id binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// ADD: verify the chunk belongs to the requested shard
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Similarly, add the same check for `prev_chunk_header.shard_id()` if present. Consider also switching the `verify_path` call to `verify_path_with_index` (which checks both membership and position) once the shard index is computed from `shard_id`. [12](#0-11) 

### Proof of Concept

Build a two-shard test-loop environment. On the syncing node, intercept the `StateHeaderValidationRequest` before it reaches the chain actor and substitute a crafted `ShardStateSyncResponseHeader` where:

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
