### Title
Stale-Pending Fallback Drops `l1_data_gas_price`, `l2_gas_price`, and `l1_da_mode`, Causing Wrong Fee Estimation for Pending Block ÔÇö (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

When the in-memory pending data is stale (its `parent_block_hash` no longer matches the latest committed block hash), `read_pending_data` synthesises a fallback block of type `PendingBlockOrDeprecated::Deprecated(DeprecatedPendingBlock)`. That type has no fields for `l1_data_gas_price`, `l2_gas_price`, or `l1_da_mode`, so the accessor methods for those fields unconditionally return `GasPricePerToken::default()` (zero) and `L1DataAvailabilityMode::Calldata`. Any unprivileged RPC caller who issues `estimate_fee(block_id=Pending)` during the window between a new block being committed and the pending-sync loop refreshing the shared `PendingData` receives fee estimates computed with zeroed data-gas prices and the wrong DA mode.

---

---

### Impact Explanation

When the network is running with Blob DA mode (the current Starknet mainnet configuration), the fallback block forces `use_kzg_da = false` and `l1_data_gas_price = NonzeroGasPrice::MIN`. Fee estimates returned to callers are therefore computed with the wrong DA mode and effectively-zero data-gas prices, producing authoritative-looking but incorrect values. Callers who rely on these estimates to set `max_fee` / resource bounds may under-pay (transaction reverts on-chain) or over-pay.

---

### Likelihood Explanation

The stale window is a normal operational state: it occurs on every block transition, lasting until the pending-sync loop (`sync_pending_data`) fetches fresh data from the feeder gateway. On a chain producing blocks every ~6 seconds, this window is non-trivial. Any unprivileged caller who issues `starknet_estimateFee` with `block_id=pending` during that window hits the bug. No special privileges or timing attack is required beyond a standard JSON-RPC call.

---

### Recommendation

Replace the fallback `DeprecatedPendingBlock` with a `PendingBlock` (`PendingBlockOrDeprecated::Current`) so that `l1_data_gas_price`, `l2_gas_price`, and `l1_da_mode` can be populated from the latest committed header:

```rust
block: PendingBlockOrDeprecated::Current(PendingBlock {
    parent_block_hash: latest_header.block_hash,
    l1_gas_price: latest_header.block_header_without_hash.l1_gas_price,
    l1_data_gas_price: latest_header.block_header_without_hash.l1_data_gas_price,
    l2_gas_price: latest_header.block_header_without_hash.l2_gas_price,
    l1_da_mode: latest_header.block_header_without_hash.l1_da_mode,
    timestamp: latest_header.block_header_without_hash.timestamp,
    sequencer_address: latest_header.block_header_without_hash.sequencer,
    starknet_version: latest_header.block_header_without_hash.starknet_version.to_string(),
    ..Default::default()
}),
```

### Proof of Concept

The proof idea in the question is mechanically correct. Concretely:

1. Set `pending_data.block = PendingBlockOrDeprecated::Current(PendingBlock { l1_data_gas_price: non_zero, l2_gas_price: non_zero, l1_da_mode: Blob, parent_block_hash: H1, ... })`.
2. Commit a new block to storage with `block_hash = H2 Ôëá H1` (making the pending data stale).
3. Call `read_pending_data` ÔÇö the hash check at line 1570 fails, the fallback `DeprecatedPendingBlock` is returned.
4. Pass the result through `client_pending_data_to_execution_pending_data`.
5. Assert `execution_pending_data.l1_data_gas_price == GasPricePerToken::default()`, `execution_pending_data.l2_gas_price == GasPricePerToken::default()`, and `execution_pending_data.l1_da_mode == L1DataAvailabilityMode::Calldata`. [6](#0-5) [7](#0-6)
