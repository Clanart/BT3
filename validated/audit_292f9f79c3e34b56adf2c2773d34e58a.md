### Title
Missing Shard-ID Binding Check in `set_state_header` Allows Cross-Shard State Substitution — (File: `chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for a caller-supplied `shard_id` but never verifies that the embedded chunk's own `shard_id` field matches the requested one. Because the block's `chunk_headers_root` is a flat merkle root over **all** shards' chunks, a malicious snapshot host can supply a legitimately-signed chunk from shard N with a valid merkle proof while the syncing node believes it is receiving shard M's state. All downstream validation (state-root-node check, state-part validation, finalization) then operates on the wrong shard's data.

---

### Finding Description

In `set_state_header`, the chunk extracted from the peer-supplied header is validated in two steps:

**Step 1 — internal chunk proofs:** [1](#0-0) 

**Step 2 — merkle inclusion in the block:** [2](#0-1) 

The `verify_path` call proves that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in `sync_prev_block_header.chunk_headers_root()`. That root is a single flat merkle tree over **every** chunk in the block, one leaf per shard. A chunk belonging to shard 1 is a perfectly valid leaf in that tree and carries a valid `MerklePath` proof. The call succeeds for any shard's chunk, not only the one at the position corresponding to the requested `shard_id`.

Nowhere in the function is there a check of the form `chunk.shard_id() == shard_id`. The function then proceeds to validate the state-root-node against the chunk's own `prev_state_root()`: [3](#0-2) 

And finally stores the header under the **caller-supplied** `shard_id` key: [4](#0-3) 

So the stored entry `StateHeaderKey(shard_id=0, sync_hash)` can contain shard 1's chunk and shard 1's `state_root_node`.

`set_state_part` then retrieves this header and validates incoming parts against the embedded (wrong) state root: [5](#0-4) 

Shard 1's state parts validate correctly against shard 1's state root, so they are accepted and written to `DBCol::StateParts` under shard 0's key.

`set_state_finalize` later applies the chunk and state using the caller-supplied `shard_id` (0) but the chunk header from shard 1: [6](#0-5) 

The `ShardStateSyncResponseHeaderV2` struct itself carries no shard-id field separate from the embedded chunk: [7](#0-6) 

The analog to the external report is exact: just as the Solana program accepted a `metadata` account without verifying it was derived from the specific `mint`, `set_state_header` accepts a chunk without verifying it was produced for the specific `shard_id`.

---

### Impact Explanation

A syncing node that accepts a cross-shard header will:

1. Store shard N's state (trie nodes) into shard M's `DBCol::StateParts` slot.
2. Apply shard N's trie into shard M's flat storage and memtrie via `apply_state_part`.
3. Execute shard N's transactions/receipts in `set_state_finalize` under shard M's `ShardUId`.
4. Produce a `ChunkExtra` for shard M whose `state_root` is shard N's post-state root.

Every subsequent chunk the node produces for shard M will reference the wrong state root, making those chunks invalid. If the node is a chunk producer for shard M it will be unable to produce valid chunks, and if it is a validator it will attest to an incorrect state root — both outcomes are critical correctness violations that can lead to slashing or permanent desync.

---

### Likelihood Explanation

Any network participant can advertise itself as a snapshot host by broadcasting a signed `SnapshotHostInfo` message: [8](#0-7) 

The `SnapshotHostsCache` selects from these advertised hosts when routing state-header and state-part requests: [9](#0-8) 

No stake, validator role, or privileged position is required to become a snapshot host. A single malicious node that wins the random host-selection lottery for a syncing peer can execute this substitution. The syncing node has no other guard: the `validate_sync_hash` check only verifies the sync hash belongs to a known recent epoch, not the shard binding of the returned header. [10](#0-9) 

---

### Recommendation

Immediately after extracting the chunk from the header in `set_state_header`, add an explicit shard-id binding check:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: verify the chunk belongs to the requested shard
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} does not match requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

This mirrors the shard-id validation already present in `compute_state_response_header` (the server side): [11](#0-10) 

and in `get_state_response_part`: [12](#0-11) 

The same binding check should be applied to `prev_chunk_header.shard_id()` when it is present.

---

### Proof of Concept

1. Attacker node calls `SnapshotHostInfo::new(peer_id, sync_hash, epoch_height, vec![shard_0_id], &key)` and broadcasts it so the victim's `SnapshotHostsCache` registers it as a host for shard 0.
2. Victim's `StateSyncDownloader::ensure_shard_header` selects the attacker and calls `download_shard_header(shard_id=0, sync_hash)`.
3. Attacker responds with a `ShardStateSyncResponseHeader::V2` whose `chunk` field is the legitimately-produced chunk for **shard 1** at the same `sync_hash` block, and whose `chunk_proof` is the valid `MerklePath` for shard 1's position in `sync_prev_block.chunk_headers_root`.
4. `set_state_header` runs:
   - `validate_chunk_proofs(&chunk_1, ...)` → passes (chunk 1 has valid internal proofs).
   - `verify_path(chunk_headers_root, chunk_1_proof, ChunkHashHeight(chunk_1_hash, h))` → passes (chunk 1 IS in the tree).
   - `validate_state_root_node(state_root_node_1, chunk_1.prev_state_root())` → passes.
   - Stores under `StateHeaderKey(shard_id=0, sync_hash)`.
5. Attacker serves valid state parts for shard 1; `set_state_part` validates them against shard 1's state root → passes; parts written under shard 0's `StatePartKey`.
6. `apply_state_part(shard_id=0, state_root=shard_1_root, ...)` writes shard 1's trie into shard 0's storage.
7. `set_state_finalize(shard_id=0, ...)` applies shard 1's chunk under shard 0's `ShardUId`, producing a `ChunkExtra` with shard 1's post-state root attributed to shard 0.
8. Victim node is now permanently desynced on shard 0 with an undetectable wrong state root.

    /// against B_prev.chunk_headers_root
    pub chunk_proof: MerklePath,
    /// This is None if sync_hash is the genesis hash. Otherwise, it's B_prev_chunk.chunks[shard_id]
    pub prev_chunk_header: Option<ShardChunkHeader>,
    /// A merkle path for (Self::prev_chunk_header.hash, Self::prev_chunk_header.height_included), verifiable
    /// against B_prev_chunk.chunk_headers_root
    pub prev_chunk_proof: Option<MerklePath>,
    /// This field contains the incoming receipts for shard_id for B_sync and B_prev_chunk.
    /// So, this field has at most two elements.
    /// These receipts are used to apply `chunk` after downloading state
    pub incoming_receipts_proofs: Vec<ReceiptProofResponse>,
    /// This field contains the info necessary to verify that the receipt proofs in Self::incoming_receipts_proofs
    /// are actually the ones referenced on chain
    ///
    /// The length of this field is the same as the length of Self::incoming_receipts_proofs, and elements
    /// of the two at a given index are taken together for verification. For a given index i,
    /// root_proofs[i] is a vector of the same length as incoming_receipts_proofs[i].1 , which itself is a

    /// contains a merkle root equal to the prev_outgoing_receipts_root field of the corresponding chunk

    /// that block's prev_chunk_outgoing_receipts_root field.
    pub root_proofs: Vec<Vec<RootProof>>,
    /// The state root with hash equal to B_prev.chunks[shard_id].prev_state_root.
    /// That is, the state root node of the trie before applying the chunks in B_prev

pub struct SnapshotHostInfo {
    /// Id of the node serving the snapshot
    pub peer_id: PeerId,
    /// Hash of the snapshot's state root
    pub sync_hash: CryptoHash,
    /// Ordinal of the epoch of the state root
    pub epoch_height: EpochHeight,
    /// List of shards included in the snapshot.
    pub shards: Vec<ShardId>,
    /// Signature on (sync_hash, epoch_height, shards)
    pub signature: Signature,
}

impl SnapshotHostInfo {
    fn build_hash(
        sync_hash: &CryptoHash,
        epoch_height: &EpochHeight,
        shards: &Vec<ShardId>,
    ) -> CryptoHash {
        CryptoHash::hash_borsh((sync_hash, epoch_height, shards))
    }

    pub(crate) fn new(
        peer_id: PeerId,
        sync_hash: CryptoHash,
        epoch_height: EpochHeight,
        shards: Vec<ShardId>,
        secret_key: &SecretKey,
    ) -> Self {
        #[cfg(not(test))]
        assert_eq!(&secret_key.public_key(), peer_id.public_key());
        let hash = Self::build_hash(&sync_hash, &epoch_height, &shards);
        let signature = secret_key.sign(hash.as_ref());
        Self { peer_id, sync_hash, epoch_height, shards, signature }
    }

    pub(crate) fn hash(&self) -> CryptoHash {
        Self::build_hash(&self.sync_hash, &self.epoch_height, &self.shards)
    }

    pub(crate) fn verify(&self) -> Result<(), SnapshotHostInfoVerificationError> {
        // Number of shards must be limited, otherwise it'd be possible to create malicious
        // messages with millions of shard ids.
        if self.shards.len() > MAX_SHARDS_PER_SNAPSHOT_HOST_INFO {
            return Err(SnapshotHostInfoVerificationError::TooManyShards(self.shards.len()));
        }

        if !self.signature.verify(self.hash().as_ref(), self.peer_id.public_key()) {
            return Err(SnapshotHostInfoVerificationError::InvalidSignature);
        }

        Ok(())
    }
