### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary
The `execute_replace_class` syscall handler in the StarkNet OS does not validate that the new class hash supplied by a contract is actually a declared class. A malicious contract deployer can exploit this to permanently replace a contract's class with an undeclared hash, rendering the contract permanently uncallable and freezing any funds it holds.

### Finding Description
In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall by directly writing the caller-supplied class hash into `contract_state_changes` without verifying that the hash corresponds to a declared class in `contract_class_changes`:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment at line 898 explicitly acknowledges the missing check. The OS accepts any arbitrary felt value as the new class hash with no validation.

When a subsequent transaction attempts to call the affected contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
// ...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
```

If `execution_context.class_hash` is an undeclared hash, `dict_read` on `contract_class_changes` returns 0 (the default), and `find_element` with key `0` will panic or fail to locate a compiled class, making the contract permanently uncallable.

### Impact Explanation
**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared value and the block is committed to the chain, the contract's state entry permanently records the invalid class hash. No subsequent transaction can successfully execute against that contract because the OS cannot resolve the class to a compiled class fact. Any funds (tokens, ETH bridged via L1→L2 messages, etc.) held in the contract's storage are permanently inaccessible. There is no recovery path: `replace_class` cannot be called again because the contract itself is uncallable.

### Likelihood Explanation
A contract deployer can craft a contract whose logic calls `replace_class` with an arbitrary undeclared felt value (e.g., `1`). The deployer can:
1. Deploy the contract and advertise it as a legitimate vault or escrow.
2. Attract user deposits.
3. Trigger the `replace_class` call (e.g., via a time-lock or a privileged function).

The OS unconditionally accepts the syscall. The missing validation is confirmed by the developer TODO comment, indicating this is a known gap, not a design choice.

### Recommendation
Before committing the `dict_update` in `execute_replace_class`, verify that the supplied `class_hash` has a non-zero entry in `contract_class_changes` (i.e., it has been declared in the current or a prior block):

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the check implicitly performed in `execute_entry_point` and closes the gap acknowledged by the TODO.

### Proof of Concept

1. Deploy contract `MaliciousVault` whose `__execute__` calls `replace_class(class_hash=0x1)` where `0x1` is not a declared class hash.
2. Users deposit funds into `MaliciousVault`.
3. Attacker sends an invoke transaction calling `MaliciousVault.__execute__`.
4. The OS processes `execute_replace_class`:
   - [1](#0-0) 
   - No validation of `class_hash=0x1` is performed; `contract_state_changes[MaliciousVault].class_hash` is set to `0x1`.
5. Block is committed. `MaliciousVault`'s on-chain class hash is now `0x1` (undeclared).
6. Any future call to `MaliciousVault` reaches `execute_entry_point`:
   - [2](#0-1) 
   - `dict_read(contract_class_changes, key=0x1)` returns `0`; `find_element(..., key=0)` fails to locate any compiled class.
7. The contract is permanently uncallable. All deposited funds are frozen.
