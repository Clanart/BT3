### Title
Wormhole VAA `emitter_chain` Field Never Validated — Cross-Chain Emitter Spoofing Enables Unauthorized Token Minting - (File: `near/omni-prover/wormhole-omni-prover-proxy/src/parsed_vaa.rs`)

---

### Summary

The `WormholeOmniProverProxy` parses the `emitter_chain` field from every Wormhole VAA body but never validates it against the chain kind embedded in the payload. An attacker who deploys a contract at the same address as a registered bridge factory on a different EVM chain can submit a guardian-signed VAA that passes all downstream checks, causing the NEAR bridge to accept a fraudulent cross-chain message as if it originated from the legitimate factory.

---

### Finding Description

`ParsedVAA::parse()` extracts six header fields from the VAA body — `timestamp`, `nonce`, `emitter_chain`, `emitter_address`, `sequence`, and `consistency_level` — and stores them all in the struct. The struct is annotated `#[allow(dead_code)]`, signalling that most of these fields are never consumed downstream. [1](#0-0) 

In `verify_vaa_callback`, after the Wormhole guardian signature is confirmed via `verify_vaa`, the only check performed on the parsed VAA is that `payload[0]` matches the expected `proof_kind`. The `emitter_chain` field is never read again. [2](#0-1) 

The `TryInto` implementations for all four message types (`InitTransferMessage`, `FinTransferMessage`, `DeployTokenMessage`, `LogMetadataMessage`) construct the `emitter_address` field by combining the **chain kind from the payload** (`transfer.token_address.get_chain()`) with the **raw address bytes from the VAA header** (`self.emitter_address`). The VAA's own `emitter_chain` is never consulted. [3](#0-2) [4](#0-3) 

The downstream factory check in `fin_transfer_callback` validates only that `emitter_address` (as an `OmniAddress`) matches a registered factory for the chain kind derived from the payload: [5](#0-4) 

Because the chain kind used to construct `emitter_address` comes from the **payload** (attacker-controlled), not from the VAA's `emitter_chain` header (Wormhole-attested), the factory check can be satisfied by a contract on a different chain that happens to share the same address bytes as the legitimate factory.

The `WormholeOmniProverProxy` contract stores no trusted emitter registry — only `prover_account`: [6](#0-5) 

---

### Impact Explanation

An attacker who can deploy a contract at the same 20-byte address as a registered bridge factory on a second EVM chain (feasible with CREATE2 vanity mining) can:

1. Emit a properly formatted `InitTransfer` event from their malicious contract on chain B, with the payload's `token_address` claiming the token belongs to chain A.
2. Obtain a guardian-signed VAA (`emitter_chain = B`, `emitter_address = 0xFACTORY`).
3. Submit the VAA to the NEAR bridge via `fin_transfer`.
4. The bridge constructs `emitter_address = OmniAddress::A(0xFACTORY)`, which matches the registered factory for chain A.
5. The bridge mints or releases tokens on NEAR without any real deposit having occurred on chain A.

This is **unauthorized mint of bridged assets** — a Critical/High impact per the allowed scope.

---

### Likelihood Explanation

EVM addresses are 20 bytes. CREATE2 allows deterministic address pre-computation. An attacker with moderate compute resources can mine a salt that produces the same address as a known factory contract on a target chain. This is a well-documented technique (e.g., address-collision griefing in Uniswap v3). The Wormhole guardian network will sign any message from any contract; it does not filter by emitter identity. The only barrier is the CREATE2 mining cost, which is feasible for a motivated attacker targeting a high-value bridge.

---

### Recommendation

1. **Store a trusted emitter registry** in `WormholeOmniProverProxy`: a mapping from Wormhole chain ID (`u16`) to the expected emitter address bytes. Populate it at initialization with the known factory addresses per chain.

2. **Validate `emitter_chain` and `emitter_address` together** inside `verify_vaa_callback` before dispatching to the `TryInto` converters:

```rust
// pseudocode
let expected = self.trusted_emitters.get(&parsed_vaa.emitter_chain)
    .ok_or("UnknownEmitterChain")?;
require!(parsed_vaa.emitter_address == *expected, "UnknownEmitterAddress");
```

3. **Remove the `#[allow(dead_code)]` suppression** on `ParsedVAA` so the compiler surfaces unused fields as warnings, making future omissions visible.

### Proof of Concept

**Setup:**
- Legitimate Ethereum factory registered in NEAR bridge: `OmniAddress::Eth(0xDEAD...BEEF)`.
- Attacker mines a CREATE2 salt and deploys a malicious contract on BSC at `0xDEAD...BEEF`.

**Attack steps:**

1. Attacker's BSC contract calls `publishMessage` on the Wormhole BSC core contract, emitting:
   ```
   payload[0] = 0x00  // ProofKind::InitTransfer
   token_address = OmniAddress::Eth(some_eth_token)  // chain kind = Eth in payload
   amount = 1_000_000
   recipient = attacker.near
   ...
   ```

2. Wormhole guardians observe the BSC transaction and produce a VAA:
   ```
   emitter_chain = 4  (BSC Wormhole ID)
   emitter_address = 0xDEAD...BEEF
   payload = <above>
   ```

3. Attacker calls `fin_transfer` on the NEAR bridge with this VAA.

4. `verify_vaa_callback` runs:
   - `gov_idx` is `Ok` (guardian signatures valid) ✓
   - `payload[0] == 0x00 == ProofKind::InitTransfer` ✓
   - `emitter_chain = 4` — **never checked** ✗

5. `TryInto<InitTransferMessage>` constructs:
   ```
   emitter_address = OmniAddress::Eth(0xDEAD...BEEF)
   // chain = Eth (from payload's token_address), bytes = VAA's emitter_address
   ```

6. `fin_transfer_callback` checks:
   ```
   self.factories.get(&ChainKind::Eth) == Some(OmniAddress::Eth(0xDEAD...BEEF))
   // TRUE — matches the legitimate Ethereum factory
   ```

7. Bridge mints tokens to `attacker.near` with no real Ethereum deposit. [7](#0-6) [3](#0-2) [5](#0-4)

    pub fn verify_vaa_callback(
        &mut self,
        proof_kind: ProofKind,
        vaa: String,
        #[callback_result] gov_idx: &Result<u32, PromiseError>,
    ) -> Result<ProverResult, String> {
        if gov_idx.is_err() {
            return Err("Proof is not valid!".to_owned());
        }

        let h = hex::decode(vaa).expect("invalidVaa");
        let parsed_vaa = parsed_vaa::ParsedVAA::parse(&h);

        require!(
            u8::from(proof_kind) == parsed_vaa.payload[0],
            "Invalid proof kind"
        );

        match proof_kind {
            ProofKind::InitTransfer => Ok(ProverResult::InitTransfer(parsed_vaa.try_into()?)),
            ProofKind::FinTransfer => Ok(ProverResult::FinTransfer(parsed_vaa.try_into()?)),
            ProofKind::DeployToken => Ok(ProverResult::DeployToken(parsed_vaa.try_into()?)),
            ProofKind::LogMetadata => Ok(ProverResult::LogMetadata(parsed_vaa.try_into()?)),
        }
    }
