### Title
Wrong Value Hardcoded in Receipt Commitment Hash Ignores Actual L2 Gas Consumed — (`crates/starknet_api/src/block_hash/receipt_commitment.rs`)

### Summary

`chain_gas_consumed` in `receipt_commitment.rs` unconditionally chains `Felt::ZERO` as the L2 gas consumed field instead of the actual `gas_consumed.l2_gas` value from the `GasVector`. This is the direct sequencer analog of the ContinuousGDA bug: a wrong constant is substituted for the correct variable in a commitment formula, producing a systematically incorrect `ReceiptCommitment` for every block that contains transactions with non-zero L2 gas (Starknet ≥ 0.14.2).

### Finding Description

`chain_gas_consumed` is the leaf-level function that serialises per-transaction execution resources into the Poseidon hash chain used to build each receipt leaf:

```rust
// crates/starknet_api/src/block_hash/receipt_commitment.rs  lines 86-90
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&Felt::ZERO)              // ← hardcoded zero, should be gas_consumed.l2_gas
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

The function receives a `GasVector` that carries three fields — `l1_gas`, `l1_data_gas`, and `l2_gas` — but silently discards `l2_gas` and substitutes a literal zero. [1](#0-0) 

`GasVector` is defined with all three fields:

```rust
pub struct GasVector {
    pub l1_gas: GasAmount,
    pub l1_data_gas: GasAmount,
    pub l2_gas: GasAmount,
}
``` [2](#0-1) 

`TransactionOutputForHash`, the struct whose `gas_consumed` field is passed to `chain_gas_consumed`, carries a full `GasVector`:

```rust
pub struct TransactionOutputForHash {
    pub actual_fee: Fee,
    pub events: Vec<Event>,
    pub execution_status: TransactionExecutionStatus,
    pub gas_consumed: GasVector,   // ← l2_gas is present but ignored
    pub messages_sent: Vec<MessageToL1>,
}
``` [3](#0-2) 

Starting with Starknet 0.14.2, transactions carry non-zero L2 gas. The test resource file `block_post_0_14_2.json` shows every transaction with `"l2_gas": 76014735` in `total_gas_consumed`. [4](#0-3) 

The receipt commitment is computed by `calculate_receipt_commitment`, which calls `calculate_receipt_hash` → `chain_gas_consumed` for every transaction in the block, then builds a Patricia root over the resulting leaves: [5](#0-4) 

That `ReceiptCommitment` is then chained directly into `calculate_block_hash`: [6](#0-5) 

All existing unit tests set `l2_gas: GasAmount(0)` in their `GasVector` fixtures, so the hardcoded zero produces the same result as the correct field and the bug is invisible to the test suite: [7](#0-6) 

### Impact Explanation

For every block produced on Starknet ≥ 0.14.2 that contains at least one transaction with non-zero L2 gas:

1. Each affected receipt leaf hash is computed with `0` instead of the actual `l2_gas` value, producing a wrong leaf.
2. The Patricia root over those leaves — the `ReceiptCommitment` — is wrong.
3. `calculate_block_hash` chains the wrong `ReceiptCommitment`, producing a wrong `BlockHash`.
4. The wrong `BlockHash` is stored in `StorageBlockHeader` and returned by the RPC as an authoritative value.
5. Any external verifier (proof system, light client, cross-chain bridge) that independently recomputes the receipt commitment from the actual transaction data will obtain a different value and reject the block.

This matches the allowed impact scope: **"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input"** (wrong `ReceiptCommitment` and derived `BlockHash`) and **"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value"** (RPC returns the wrong block hash and receipt commitment).

### Likelihood Explanation

The trigger condition is any block on Starknet ≥ 0.14.2 with at least one transaction that consumes non-zero L2 gas. The test resource `block_post_0_14_2.json` shows this is the normal case for current mainnet blocks. No privileged access or adversarial input is required; the bug fires automatically during normal block production.

### Recommendation

Replace the hardcoded `Felt::ZERO` with the actual `l2_gas` field from the passed `GasVector`:

```rust
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&gas_consumed.l2_gas.into())   // was: &Felt::ZERO
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

Add a regression test that sets `l2_gas` to a non-zero value and verifies the receipt commitment changes relative to the zero case.

### Proof of Concept

1. Take any transaction from `block_post_0_14_2.json` with `"l2_gas": 76014735` in `total_gas_consumed`.
2. Construct a `ReceiptElement` with `gas_consumed.l2_gas = GasAmount(76014735)`.
3. Call `calculate_receipt_hash` on it — the current code chains `Felt::ZERO` for L2 gas.
4. Manually compute the Poseidon hash with `76014735` in the L2 gas slot instead.
5. The two hashes differ, demonstrating that `calculate_receipt_commitment` produces a wrong root for any block containing such transactions, and therefore `calculate_block_hash` produces a wrong `BlockHash`. [1](#0-0) [8](#0-7)

fn calculate_receipt_hash(receipt_element: &ReceiptElement) -> Felt {
    let hash_chain = HashChain::new()
        .chain(&receipt_element.transaction_hash)
        .chain(&receipt_element.transaction_output.actual_fee.0.into())
        .chain(&calculate_messages_sent_hash(&receipt_element.transaction_output.messages_sent))
        .chain(&get_revert_reason_hash(&receipt_element.transaction_output.execution_status));
    chain_gas_consumed(hash_chain, &receipt_element.transaction_output.gas_consumed)
        .get_poseidon_hash()
}
