### Title
Missing Class Hash Validation in `replace_class` Allows Permanent Freezing of Contract Funds — (`File: execution/syscall_impls.cairo`)

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary felt value as a new class hash without verifying that the hash corresponds to a previously declared class. A contract can therefore replace its own class with an undeclared hash, rendering itself permanently non-executable and freezing any funds it holds. The missing check is explicitly acknowledged by a TODO comment in the production code.

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function updates a contract's class hash in the state without performing any validation that the supplied hash exists in the declared class tree: [1](#0-0) 

The critical lines are:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The OS writes the new `class_hash` directly into the contract's `StateEntry` and commits it to the Patricia Merkle Tree via `compute_contract_state_commitment` without any cross-check against `contract_class_changes` (the declared class tree). [2](#0-1) 

Once the state root is committed with an undeclared class hash, no future transaction can execute against that contract — there is no compiled class (CASM) to load — and the state is irrecoverable.

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ERC-20 balance, ETH, or other asset stored in the contract's storage becomes permanently inaccessible. Because the OS commits the invalid class hash into the global state root, the contract is frozen at the protocol level with no recovery path. This matches the allowed impact "Critical. Permanent freezing of funds."

### Likelihood Explanation

**High.**

The attack surface is broad:

1. **Direct self-attack**: An attacker deploys a contract that exposes a public entry point accepting an arbitrary class hash and calling `replace_class`. The attacker can lure victims into depositing funds before triggering the freeze.
2. **Victim contract with exposed `replace_class`**: Any contract that delegates class replacement to an external caller (e.g., an upgradeable proxy with insufficient access control) can be targeted. The attacker calls the exposed function with a hash that was never declared, permanently bricking the contract.
3. **Buggy contract**: A contract with a bug that allows an attacker to control the argument to `replace_class` is silently exploitable because the OS imposes no guard.

No privileged role, leaked key, or operator cooperation is required. The attacker only needs to be a standard transaction sender.

### Recommendation

Before writing the new `StateEntry`, verify that `class_hash` exists in `contract_class_changes` (i.e., it was declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` for the supplied `class_hash` and assert the returned compiled class hash is non-zero:

```cairo
// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the check already enforced during `execute_declare_transaction`: [3](#0-2) 

### Proof of Concept

1. Attacker deploys contract `VictimVault` (holds user funds) with an upgradeable class that exposes:
   ```
   fn upgrade(new_class_hash: ClassHash) { replace_class_syscall(new_class_hash); }
   ```
2. Users deposit tokens into `VictimVault`.
3. Attacker calls `upgrade(0xdeadbeef)` — a hash that was never declared on-chain.
4. The OS executes `execute_replace_class`, skips the missing validation, and writes `class_hash=0xdeadbeef` into the contract's `StateEntry`.
5. The state root is updated via `compute_contract_state_commitment` with the invalid hash.
6. All subsequent calls to `VictimVault` fail at class resolution — no CASM exists for `0xdeadbeef`.
7. All deposited funds are permanently frozen with no recovery mechanism.
