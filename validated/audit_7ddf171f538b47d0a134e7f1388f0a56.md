### Title
Child `ChunkExtra.proposed_split` Inherited Uncleared from Parent Shard After Resharding Causes `InvalidChunkHeaderShardSplit` on First Child Chunk — (`chain/chain/src/resharding/manager.rs`)

---

### Summary

When a shard is split during resharding, `ReshardingManager::process_memtrie_resharding_storage_update` creates each child shard's initial `ChunkExtra` by cloning the parent's `ChunkExtra` and updating only `state_root` and `congestion_info`. The `proposed_split` field — which records the parent shard's trie-split proposal — is **not cleared**. The child shard's first `ChunkExtra` therefore carries the parent's `proposed_split = Some(X)`. When the child shard's first chunk is produced in the next epoch, the chunk producer computes `proposed_split = None` (the cooldown prevents a new proposal), and `validate_chunk_with_chunk_extra_and_receipts_root` rejects the chunk with `InvalidChunkHeaderShardSplit` because `prev_chunk_extra.proposed_split() ≠ chunk_header.proposed_split()`. This halts chunk production for every child shard.

---

### Finding Description

In `process_memtrie_resharding_storage_update` (`chain/chain/src/resharding/manager.rs`, lines 258–265), the child `ChunkExtra` is constructed as:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
``` [1](#0-0) 

The accompanying TODO explicitly acknowledges the incompleteness:

```
// TODO(resharding): set all fields of `ChunkExtra`. Consider stronger
// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.
``` [2](#0-1) 

The `proposed_split` field of `ChunkExtraV5` is defined as:

```rust
pub proposed_split: Option<TrieSplit>,
``` [3](#0-2) 

Because the parent shard was above the memory threshold (that is what triggered resharding), its `ChunkExtra.proposed_split` is `Some(X)`. The clone propagates this value into both child shards' initial `ChunkExtra` records, which are persisted via `chain_store_update.save_chunk_extra`.

In epoch N+1, the chunk producer for each child shard runs `compute_proposed_split`. The resharding cooldown (`can_reshard`) returns `false` (because `min_epochs_between_resharding > 0`), so the producer sets `proposed_split = None` in the chunk header. Validation then compares:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
``` [4](#0-3) 

`prev_chunk_extra.proposed_split()` is `Some(X)` (inherited from parent); `chunk_header.proposed_split()` is `None`. The mismatch is fatal.

The dynamic resharding documentation acknowledges this exact failure mode:

> `min_epochs_between_resharding` must be `> 0`: allowing back-to-back reshardings is unsafe because a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk freshly computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`. [5](#0-4) 

The cooldown is a config-level workaround, not a code-level fix. The invariant violation — that a child shard's initial `ChunkExtra` carries a stale `proposed_split` binding from its parent — is present in the committed code regardless of the cooldown value.

---

### Impact Explanation

If `min_epochs_between_resharding` is set to `0` (or if the cooldown check is bypassed), every child shard's first chunk is rejected with `InvalidChunkHeaderShardSplit`. No chunk producer can produce a valid first chunk for any child shard, halting the chain at the resharding boundary. Even with the cooldown enforced, the underlying invariant is broken: the child `ChunkExtra` carries a `proposed_split` commitment that no chunk header can legitimately satisfy, because the cooldown prevents the child from proposing a split and the inherited value is non-`None`.

**Impact: Critical** — chain halt for all child shards after resharding.

---

### Likelihood Explanation

In production, `min_epochs_between_resharding > 0` is enforced by documentation and the dynamic resharding config. However:

1. The root cause is a code defect (missing field clear), not a config constraint.
2. Any test or staging environment that sets `min_epochs_between_resharding = 0` will reproduce the halt.
3. A future code change that adjusts the cooldown logic without fixing the `ChunkExtra` clone could re-expose the issue in production.
4. The TODO comment at the exact site of the bug confirms the field is known to be incorrectly propagated.

**Likelihood: Medium** (low in current production due to cooldown, but the defect is latent and acknowledged).

---

### Recommendation

In `process_memtrie_resharding_storage_update`, after cloning the parent `ChunkExtra`, explicitly clear `proposed_split` on the child:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
*child_chunk_extra.proposed_split_mut() = None;  // child shard has no inherited split proposal
```

This mirrors the fix described in the external report: clear the inherited assignment on transfer rather than relying on a config-level workaround. The `proposed_split` field is a per-shard, per-chunk commitment; it must not cross the resharding boundary.

Additionally, resolve the TODO at `chain/chain/src/resharding/manager.rs:255` by auditing all `ChunkExtra` fields for correct initialization on child shards.

---

### Proof of Concept

1. Configure dynamic resharding with `min_epochs_between_resharding = 0` and a low `memory_usage_threshold` so that a shard triggers resharding.
2. Submit transactions to inflate one shard's trie memory above the threshold.
3. At the last block of epoch N, the block producer includes `shard_split = Some((shard_id, boundary))` in the block header.
4. `EpochManager::finalize_epoch` derives the new shard layout for epoch N+2.
5. `ReshardingManager::process_memtrie_resharding_storage_update` creates child `ChunkExtra` records by cloning the parent's, leaving `proposed_split = Some(X)`.
6. In epoch N+1, the chunk producer for each child shard computes `proposed_split = None` (cooldown = 0 but `is_next_block_possibly_last_in_epoch` is false for the first block of the epoch, or the child shard is below threshold).
7. `validate_chunk_with_chunk_extra_and_receipts_root` compares `prev_chunk_extra.proposed_split() = Some(X)` against `chunk_header.proposed_split() = None` and returns `Err(InvalidChunkHeaderShardSplit)`.
8. No valid chunk can be produced for any child shard; the chain halts.

The exact corrupted value is `ChunkExtra.proposed_split` for each child shard's initial `ChunkExtra`, stored at `DBCol::ChunkExtra` keyed by `(resharding_block_hash, child_shard_uid)`.
