Audit Report

## Title
Unprivileged caller can steal stranded ETH via `refundETH()` after a victim's payable multicall omits the refund step — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` is an unrestricted `external payable` function that transfers the router's entire native ETH balance to `msg.sender`. When a user sends excess ETH via a payable `multicall` without appending a `refundETH()` call, the unused ETH is stranded on the router after the transaction completes. Any subsequent caller can invoke `refundETH()` in a separate transaction and receive all stranded ETH, causing a direct, irreversible loss of the victim's ETH principal.

## Finding Description
`refundETH()` contains no access control and unconditionally sends `address(this).balance` to `msg.sender`: [1](#0-0) 

The `pay()` function, when `token == WETH` and `nativeBalance >= value`, deposits only exactly `value` wei into WETH and forwards it to the pool. Any `msg.value` in excess of `value` remains on the router as raw ETH after the transaction: [2](#0-1) 

The `receive()` fallback only blocks *direct* ETH transfers from non-WETH addresses: [3](#0-2) 

ETH attached to a payable function call (e.g., `multicall{value: 2e18}(...)`) bypasses `receive()` entirely — the ETH is credited to the contract balance without triggering the fallback. The `multicall` dispatcher is `payable` and uses `delegatecall`, so `msg.value` is available to all sub-calls: [4](#0-3) 

The intended usage pattern — including `refundETH()` as the last call in the same multicall batch — is documented but not enforced at the code level: [5](#0-4) 

No existing guard prevents a separate-transaction call to `refundETH()` from draining the stranded balance.

## Impact Explanation
Direct loss of user ETH principal. A victim who calls `multicall{value: 2e18}([exactInputSingle(amountIn=1e18, tokenIn=WETH)])` without appending `refundETH()` will have 1e18 ETH stranded on the router. An attacker can immediately call `refundETH()` in the next block and receive the full stranded balance. The loss is irreversible with no protocol-level recovery path. This meets the Sherlock High/Medium threshold for direct loss of user principal via an unprivileged attacker path.

## Likelihood Explanation
The attack requires no special privileges, no malicious pool, and no non-standard token behavior. Any integrator, wallet, or user who constructs a multicall with excess ETH but omits the refund step — a common mistake — creates an immediately exploitable condition. The design documentation explicitly states the requirement, but omission is easy and the window of exploitation is open until the next `refundETH()` caller (attacker or otherwise) drains the balance. Mempool monitoring makes this trivially automatable.

## Recommendation
Restrict `refundETH()` so it can only be called as part of a `multicall` by tracking the original `msg.sender` in transient storage at `multicall` entry and checking it inside `refundETH()`. Only the address that initiated the multicall should be permitted to receive the refund. A `recipient` parameter alone does not fix the issue — the core fix must bind the refund to the original multicall initiator via transient storage.

## Proof of Concept
```solidity
function test_attacker_steals_stranded_eth() public {
    uint128 amountIn = 1e18;
    uint256 msgValue = 2e18;

    // Victim sends 2 ETH but only 1 ETH is consumed; no refundETH() included
    vm.prank(victim);
    bytes[] memory calls = new bytes[](1);
    calls[0] = abi.encodeWithSelector(
        router.exactInputSingle.selector,
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: amountIn,
            amountOutMinimum: 0,
            recipient: victim,
            deadline: block.timestamp + 60,
            priceLimitX64: 0,
            extensionData: ""
        })
    );
    router.multicall{value: msgValue}(calls);

    // 1e18 ETH is now stranded on the router
    assertEq(address(router).balance, 1e18);

    // Attacker calls refundETH() in the next transaction
    uint256 attackerBefore = attacker.balance;
    vm.prank(attacker);
    router.refundETH();

    // Attacker received the victim's stranded ETH
    assertEq(attacker.balance - attackerBefore, 1e18);
    assertEq(address(router).balance, 0);
}
```
