### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Locks Bridge Collateral - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`OmniBridge.initTransfer` records the caller-supplied `amount` in the `InitTransfer` event without verifying the actual tokens received after `safeTransferFrom`. For fee-on-transfer ERC20 tokens, the bridge emits a larger amount than it actually holds, causing the destination chain to mint/release more tokens than are locked, permanently breaking bridge collateralization.

### Finding Description

In `OmniBridge.initTransfer`, when the token is a plain ERC20 (not a bridge token or custom minter), the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
``` [1](#0-0) 

Immediately after, the event is emitted using the caller-supplied `amount` — not the actual balance delta received:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // ← pre-specified, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

No balance-before / balance-after check is performed. For fee-on-transfer tokens (tokens that deduct a percentage on every transfer), `safeTransferFrom(sender, bridge, amount)` results in the bridge receiving `amount - transferFee`, while the emitted event records `amount`. The NEAR-side bridge processes this event and finalizes `amount` tokens to the recipient on the destination chain — more than the bridge actually holds.

### Impact Explanation

This is a **High** severity accounting corruption issue. The bridge's lock/unlock collateral model is broken for fee-on-transfer ERC20 tokens:

- Bridge holds: `amount - transferFee` tokens
- Destination chain releases: `amount` tokens (unbacked)
- Repeated bridging drains the bridge's reserves
- The last users to bridge back cannot withdraw because the bridge is undercollateralized — their funds are permanently locked or unclaimable

This matches: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

### Likelihood Explanation

Any unprivileged user can call `initTransfer` with any ERC20 token address. The bridge does not whitelist tokens or restrict which ERC20s can be bridged via the lock/unlock path. Fee-on-transfer tokens (reflection tokens, tokens with configurable transfer taxes) are common in DeFi. A single honest user bridging such a token triggers the discrepancy; a malicious user can exploit it deliberately to drain the bridge's reserves of that token. [3](#0-2) 

### Recommendation

Record the actual received amount by measuring the balance difference before and after the transfer:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// Use actualReceived instead of amount in the event and downstream logic
```

Alternatively, explicitly disallow fee-on-transfer tokens by requiring `actualReceived == amount` and reverting otherwise.

### Proof of Concept

1. Deploy a fee-on-transfer ERC20 token `FeeToken` with a 1% transfer tax.
2. Call `OmniBridge.initTransfer(FeeToken, 1000e18, 0, 0, "recipient.near", "")`.
3. `safeTransferFrom` transfers 1000e18 from the user; the bridge receives 990e18 (1% fee deducted).
4. The `InitTransfer` event is emitted with `amount = 1000e18`.
5. The NEAR bridge processes the event and mints/releases 1000e18 tokens to the recipient.
6. The bridge is now undercollateralized by 10e18 tokens.
7. Repeat until the bridge's `FeeToken` balance is exhausted; subsequent users bridging back receive a revert on `safeTransfer`, permanently locking their cross-chain claims. [4](#0-3)

                    amount
                );
