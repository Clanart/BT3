Audit Report

## Title
Unguarded `refundETH()` allows any caller to steal ETH stranded in the router between transactions — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.refundETH()` has no access control and unconditionally sends the router's entire ETH balance to `msg.sender`. ETH is routinely stranded in the router when users call payable swap functions with `msg.value` exceeding the actual swap cost, because `pay()` wraps only the exact amount required and leaves the remainder in the contract. Any attacker can call `refundETH()` in a subsequent transaction to steal that ETH.

## Finding Description
`refundETH()` is unrestricted:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // sends to ANY caller
    }
}
```

ETH stranding occurs in `pay()` when `token == WETH` and `nativeBalance >= value`:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol L75-77
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();   // only `value`, not nativeBalance
    IERC20(WETH).safeTransfer(recipient, value);
}
```

Any `msg.value` above `value` remains in the router after the transaction ends. All four swap entry points — `exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput` — are `payable` and route through this `pay()` path when `tokenIn == WETH`. The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH entering via `msg.value` on payable functions.

The intended recovery pattern is to append `refundETH()` as the last call in a `multicall`. However, if a user calls any swap function directly with excess ETH, or omits `refundETH()` from their multicall, the excess ETH is stranded and immediately claimable by any address in a subsequent transaction.

For `exactOutputSingle` and `exactOutput`, callers cannot know the precise `amountIn` before execution, so sending excess ETH is the normal usage pattern — making stranding a routine occurrence, not an edge case.

## Impact Explanation
Direct theft of user ETH principal. Any ETH stranded in the router after a swap is visible to all and claimable by any address via `refundETH()`. This is a direct loss of user funds meeting the Critical/High threshold under the allowed impact gate (direct loss of user principal).

## Likelihood Explanation
`exactOutputSingle` and `exactOutput` are the primary affected paths. Callers routinely over-provision ETH because the exact input is unknown before execution. The test suite itself demonstrates this pattern: `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` sends `2 ether` for a `1_000` unit swap, explicitly relying on `refundETH()` being included in the multicall. Any user who calls these functions directly (not via multicall) or omits `refundETH()` from their multicall loses their excess ETH to the first back-runner. The attack is repeatable, requires no special privilege, and is trivially automatable via mempool monitoring.

## Recommendation
Restrict `refundETH()` so it can only refund the original `msg.sender` of the outermost call. Two viable approaches:
1. Track the original multicall initiator in transient storage at the start of `multicall` and require `msg.sender == storedInitiator` inside `refundETH()`.
2. Require `msg.sender == address(this)` when `refundETH()` is called as a delegatecall sub-call within `multicall`, preventing standalone external invocation.

## Proof of Concept
```solidity
// 1. User A calls exactOutputSingle with excess ETH (actual cost = 1 ether)
router.exactOutputSingle{value: 2 ether}(ExactOutputSingleParams({
    tokenIn: address(weth),
    amountOut: someAmount,
    amountInMaximum: 2 ether,   // actual amountIn = 1 ether
    ...
}));
// After tx: router holds 1 ether of stranded ETH

// 2. Attacker (separate tx) drains it
vm.prank(attacker);
router.refundETH();
assertEq(attacker.balance, 1 ether); // attacker stole user A's ETH
```

This maps directly to the existing test pattern at `metric-periphery/test/MetricOmmSimpleRouter.native.t.sol` L106-133, which demonstrates that `2 ether` sent for a `1_000` unit swap leaves `~2 ether - 1_000 wei` in the router until `refundETH()` is called.
