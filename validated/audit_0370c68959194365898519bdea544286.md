### Title
Stale Pending-Epoch Class Served by `get_class` RPC Due to Missing Staleness Check on `pending_classes` — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary
`starknet_getClass(block_id=Pending, class_hash=H)` returns a class body from a previous pending epoch because the handler reads directly from `pending_classes` without any parent-hash staleness check, while the only clearing mechanism for `pending_classes` lives in the async sync loop and runs with a delay after a new block is committed.

### Finding Description

The `get_class` handler in `api_impl.rs` handles the `Pending` tag by immediately reading from `self.pending_classes` and returning on a hit: [1](#0-0) 

There is no call to `read_pending_data` on this path. `read_pending_data` is the function that performs the staleness check — it compares `pending_data.block.parent_block_hash()` against the latest committed block hash and returns a synthetic empty pending block when they diverge: [2](#0-1) 

`pending_classes` is a separate `Arc<RwLock<PendingClasses>>` that is only cleared by the sync loop in `pending_sync.rs`, and only when the sync loop itself observes a parent-hash change: [3](#0-2) 

Between the moment a new block is committed (making the pending block stale) and the moment the sync loop polls, detects the change, and calls `clear()`, there is a window during which `pending_classes` still holds classes from the previous pending epoch. Any RPC call to `starknet_getClass(Pending, H)` during this window hits the early-return at line 606 and serves the stale class body as if it belongs to the current pending epoch.

By contrast, other pending-path handlers — e.g. `get_nonce` and `maybe_get_class_hash_at` — correctly call `read_pending_data` first, which enforces the staleness invariant: [4](#0-3) 

### Impact Explanation

The RPC pending view returns an authoritative-looking wrong value: the `SierraContractClass` or `DeprecatedContractClass` body for class hash H is served as belonging to the current pending epoch when it actually belongs to a stale one. If class H was not included in the committed block that closed the previous epoch, it is not declared in any committed or current-pending state, yet the endpoint returns it successfully. This misleads clients (wallets, explorers, fee estimators) that rely on the pending view for class availability.

This fits the allowed impact: **High — RPC pending view returns an authoritative-looking wrong value.**

### Likelihood Explanation

The race window is bounded by the sync loop's polling interval (`sleep_duration` in `sync_pending_data`). During normal operation this is a short but non-zero interval. Any unprivileged user who calls `starknet_getClass(Pending, H)` immediately after a block is committed — before the sync loop clears `pending_classes` — triggers the issue. No special privileges are required; the endpoint is public.

### Recommendation

In the `get_class` handler, before reading from `pending_classes`, call `read_pending_data` (or an equivalent staleness check) to verify that the current pending block's parent hash matches the latest committed block hash. If the pending block is stale, skip the `pending_classes` lookup entirely and fall through to the committed-state path (i.e., treat `Pending` as `Latest`), consistent with how every other pending-path handler behaves.

### Proof of Concept

1. Commit block N. The sync loop populates `pending_classes` with class H for pending epoch 1 (parent = block N hash).
2. Commit block N+1. The pending block is now stale (its parent hash no longer matches the latest committed block hash).
3. Before the sync loop polls and calls `pending_classes.clear()`, call `starknet_getClass(Pending, H)`.
4. The handler reads `pending_classes.get_class(H)`, finds the stale entry, and returns the class body — even though H was not declared in any committed block or in the current pending epoch.
5. Assert: the response should be `CLASS_HASH_NOT_FOUND` (or the committed-state result if H was committed), not the stale pending class body.
