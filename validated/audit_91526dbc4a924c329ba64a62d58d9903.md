### Title
Missing Chain ID in `deployToken` Signature Enables Cross-Chain Replay — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `deployToken` function in `OmniBridge.sol` verifies an MPC signature over a `MetadataPayload` that contains no destination chain identifier. Because the NEAR bridge uses a single, chain-agnostic MPC signing path (`"bridge-1"`) for all EVM deployments, the same `nearBridgeDerivedAddress` is authoritative on every EVM chain. A valid `deployToken` signature obtained from one EVM chain (e.g., Ethereum mainnet) can be replayed verbatim on any other EVM chain (Arbitrum, Base, BNB, Polygon, HyperEVM, Abstract) to deploy the same token without the NEAR bridge ever authorizing that deployment for the target chain.

---

### Finding Description

**Root cause — NEAR side (`near/omni-bridge/src/lib.rs`)**

`log_metadata_callback` constructs the payload that MPC signs. It contains no chain identifier:

```rust
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,   // no chain_id field
};
let payload = near_sdk::env::keccak256_array(
    borsh::to_vec(&metadata_payload).near_expect(BridgeError::Borsh),
);
```

The same `SIGN_PATH = "bridge-1"` constant is used for every signing call — both `sign_transfer` and `log_metadata` — so the MPC-derived key, and therefore `nearBridgeDerivedAddress`, is identical across all EVM deployments.

**Root cause — EVM side (`evm/src/omni-bridge/contracts/OmniBridge.sol`)**

`deployToken` reconstructs and verifies the payload without including `omniBridgeChainId`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)          // no chain ID
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
```

**Contrast with `finTransfer`** — the transfer-finalization path correctly binds the signature to the destination chain by embedding `omniBridgeChainId` twice in the signed payload, preventing cross-chain replay of transfer signatures. `deployToken` has no equivalent protection.

**StarkNet parity** — `starknet/src/bridge_types.cairo` `MetadataPayloadTrait::to_borsh` encodes the same chain-ID-free layout, so the same class of replay is present there as well.

---

### Impact Explanation

An attacker who replays a valid `deployToken` signature on chain B before the NEAR bridge's relayer does so:

1. Deploys the bridge token on chain B at an address the NEAR bridge has never registered.
2. Sets `nearToEthToken[metadata.token]` on chain B to that attacker-chosen address.
3. When the NEAR bridge's relayer later attempts the legitimate deployment, `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")` reverts permanently.
4. The NEAR bridge's `token_id_to_address` mapping for that `(ChainKind::Evm(chainB), tokenId)` pair is never populated.
5. All `fin_transfer` calls targeting chain B for that token fail because the bridge cannot resolve the token address, and all `init_transfer` calls from chain B are unrecognized.

Result: **permanent, irrecoverable freezing of that token's bridge path on chain B** without any admin key or privileged access required from the attacker.

---

### Likelihood Explanation

**Medium.** The attack is a simple race: the attacker monitors any EVM chain for a `DeployToken` event, extracts the `signatureData` from the transaction calldata (fully public), and calls `deployToken` on every other EVM chain before the relayer does. The window is the time between the first legitimate deployment and subsequent ones. For newly listed tokens this window can be hours or days. No special capital, no MPC collusion, and no privileged role is needed.

---

### Recommendation

Bind the metadata signature to the destination chain by including `omniBridgeChainId` in the signed payload on both sides:

**NEAR (`near/omni-types/src/lib.rs` — `MetadataPayload`):**
Add a `chain_id: u8` field and populate it in `log_metadata_callback` with the target chain's identifier before hashing.

**EVM (`evm/src/omni-bridge/contracts/OmniBridge.sol` — `deployToken`):**
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // add this
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the same fix to `starknet/src/bridge_types.cairo` `MetadataPayloadTrait::to_borsh`.

---

### Proof of Concept

1. Token `"usdc.near"` is legitimately deployed on Ethereum mainnet. The relayer calls `deployToken(sigS, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})`. Signature `sigS` is visible in the transaction calldata.
2. Attacker reads `sigS` from the Ethereum transaction.
3. Attacker calls `deployToken(sigS, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` on Arbitrum (`omniBridgeChainId = 3`). `ECDSA.recover` succeeds because the payload is chain-ID-free and `nearBridgeDerivedAddress` is the same on both chains.
4. A new `BridgeToken` proxy is deployed on Arbitrum at address `X`. `nearToEthToken["usdc.near"] = X` is written on Arbitrum.
5. The NEAR bridge's relayer later attempts the legitimate Arbitrum deployment. `require(!isBridgeToken[nearToEthToken["usdc.near"]], "ERR_TOKEN_EXIST")` reverts.
6. The NEAR bridge's `token_id_to_address` for `(Arbitrum, "usdc.near")` is never set. All USDC bridging to/from Arbitrum is permanently broken.

    fn to_borsh(self: @MetadataPayload) -> ByteArray {
        let mut borsh_bytes: ByteArray = Default::default();
        borsh_bytes.append_byte(PayloadType::Metadata.into());
        borsh_bytes.append(@borsh::encode_byte_array(self.token));
        borsh_bytes.append(@borsh::encode_byte_array(self.name));
        borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
        borsh_bytes.append_byte(*self.decimals);

    }
