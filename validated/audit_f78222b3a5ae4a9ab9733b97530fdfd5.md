### Title
Deploy Signature Replay After `removeCustomToken` Corrupts NEAR→EVM Token Mapping — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`deployToken` validates MPC signatures but tracks **no per-signature nonce or used-signature set**. The sole re-deployment guard — `!isBridgeToken[nearToEthToken[metadata.token]]` — is silently cleared by `removeCustomToken`. An unprivileged relayer holding any previously-issued MPC `deployToken` signature for a NEAR token ID can replay it after the admin removes that token, deploying a second BridgeToken proxy at a new address for the same NEAR token ID.

### Finding Description

**`deployToken` signature structure (lines 142–153):**

The signed payload covers only `(PayloadType.Metadata, token, name, symbol, decimals)` — no nonce, no block number, no expiry, no "used" bitmap. [1](#0-0) 

**The only re-deployment guard (lines 155–158):**

```solidity
require(
    !isBridgeToken[nearToEthToken[metadata.token]],
    "ERR_TOKEN_EXIST"
);
``` [2](#0-1) 

**`removeCustomToken` clears exactly the state this guard reads (lines 120–127):**

```solidity
delete isBridgeToken[tokenAddress];
delete nearToEthToken[ethToNearToken[tokenAddress]];   // nearToEthToken[nearTokenId] → address(0)
delete ethToNearToken[tokenAddress];
delete customMinters[tokenAddress];
``` [3](#0-2) 

After `removeCustomToken`:
- `nearToEthToken[nearTokenId]` = `address(0)`
- `isBridgeToken[address(0)]` = `false`
- Guard evaluates `!false` = `true` → **passes unconditionally**

The attacker then calls `deployToken` with the old (or any valid) MPC signature. A new `ERC1967Proxy` is deployed at a fresh address, and the mappings are updated: [4](#0-3) 

### Impact Explanation

After the replay:

- `nearToEthToken[nearTokenId]` points to the **new** proxy (zero supply)
- The **old** proxy retains whatever ERC-20 supply existed; `ethToNearToken[oldProxy]` is already gone (deleted by `removeCustomToken`), so holders of the old token cannot initiate a bridge-back via `initTransfer` in the normal flow
- Any subsequent `finTransfer` signed by the MPC for the new proxy address will mint tokens against NEAR-side collateral that was already accounted for under the old proxy — **unbacked supply on the new proxy**
- The invariant "each NEAR token ID maps to at most one canonical EVM address" is permanently broken

This matches the **High** impact category: token-mapping corruption that breaks bridge collateralization and misdirects value.

### Likelihood Explanation

The precondition is that a valid MPC `deployToken` signature exists for the target NEAR token ID. Two realistic paths:

1. **Replay of the original deploy signature**: If the token was first deployed via `deployToken` (signature consumed once), then the admin later calls `removeCustomToken` on that proxy (e.g., to replace it with a custom minter), the original signature has no "used" record and can be replayed immediately.
2. **Fresh signature obtained from NEAR side**: `logMetadata` is permissionless. An attacker can call it for any EVM token, causing the NEAR bridge to emit a `LogMetadata` event. The MPC will sign a `deployToken` payload for the corresponding NEAR token ID. This signature is then held and replayed after `removeCustomToken`.

Neither path requires any privileged access beyond the admin's own `removeCustomToken` call (which is a normal operational action, not attacker-controlled).

### Recommendation

Track consumed deploy signatures the same way `finTransfer` tracks nonces:

```solidity
mapping(bytes32 => bool) public usedDeploySignatures;
// ...
bytes32 hashed = keccak256(borshEncoded);
require(!usedDeploySignatures[hashed], "ERR_SIGNATURE_USED");
usedDeploySignatures[hashed] = true;
```

Alternatively, include a monotonic nonce in the `MetadataPayload` and have the MPC sign it, so each deploy authorization is single-use.

### Proof of Concept

```solidity
// 1. Admin registers a custom token for nearTokenId
bridge.addCustomToken(nearTokenId, customTokenAddr, minter, 18);

// 2. Attacker (or anyone) obtains a valid MPC deployToken signature
//    for (nearTokenId, name, symbol, decimals) — either by replaying
//    a previously-used signature or by triggering logMetadata on EVM
//    and waiting for the NEAR MPC to sign.
bytes memory sig = <valid MPC signature for nearTokenId metadata>;

// 3. Admin removes the custom token (normal operational action)
bridge.removeCustomToken(customTokenAddr);
// Now: nearToEthToken[nearTokenId] == address(0)
//      isBridgeToken[address(0)] == false  → guard bypassed

// 4. Attacker replays the signature
address newProxy = bridge.deployToken(sig, metadata);

// 5. Assert invariant violation:
assert(bridge.nearToEthToken(nearTokenId) == newProxy);   // new proxy
assert(bridge.ethToNearToken(customTokenAddr) == "");     // old addr orphaned
// customTokenAddr still holds any ERC-20 balance users deposited
// newProxy has 0 supply; future finTransfer mints are unbacked
```
