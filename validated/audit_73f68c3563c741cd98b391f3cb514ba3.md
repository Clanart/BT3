### Title
Missing `freeze_authority` Check on Native SPL Token Mint Allows Permanent Vault Lock - (File: `solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

The `LogMetadata` instruction registers a native SPL token for bridging by creating a vault PDA token account. It validates that the mint's `mint_authority` is not the bridge's authority, but performs **no check on the mint's `freeze_authority`**. A token issuer retaining a `freeze_authority` can freeze the bridge vault at any time, permanently locking all deposited tokens and making both inbound and outbound transfers impossible for every user of that token.

---

### Finding Description

When a native SPL token is registered for bridging, `LogMetadata` creates a vault PDA (`seeds = [VAULT_SEED, mint.key()]`) that holds all tokens locked on the Solana side. The only constraint on the mint is:

```rust
constraint = !mint.mint_authority.contains(authority.key),
``` [1](#0-0) 

There is no corresponding check that `mint.freeze_authority` is `None`. The vault is initialized as a standard token account:

```rust
#[account(
    init_if_needed,
    payer = common.payer,
    token::mint = mint,
    token::authority = authority,
    seeds = [VAULT_SEED, mint.key().as_ref()],
    bump,
    token::token_program = token_program,
)]
pub vault: Box<InterfaceAccount<'info, TokenAccount>>,
``` [2](#0-1) 

Any SPL token account — including this vault PDA — can be frozen by the holder of the mint's `freeze_authority` via the `FreezeAccount` instruction. Once frozen, the account's state becomes `Frozen` and all token instructions (transfer, burn, mint-to) targeting it will fail until a `ThawAccount` is issued. If the `freeze_authority` is subsequently set to `None`, the freeze becomes permanent and irrecoverable.

On `init_transfer`, the bridge transfers user tokens **into** the vault:

```rust
transfer_checked(
    CpiContext::new(..., TransferChecked { from: self.from, to: vault, ... }),
    ...
)?;
``` [3](#0-2) 

On `finalize_transfer`, the bridge transfers tokens **out of** the vault to the recipient:

```rust
transfer_checked(
    CpiContext::new_with_signer(..., TransferChecked { from: vault, to: self.token_account, ... }),
    ...
)?;
``` [4](#0-3) 

Both operations will revert with `AccountFrozen` if the vault has been frozen, completely DoS-ing the bridge for that token.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds in bridge vault flows.**

The vault is a single shared PDA that holds the aggregate balance of all users who have bridged a given native SPL token. Freezing it in one transaction:

- Blocks every pending `init_transfer` (users cannot bridge tokens out of Solana).
- Blocks every pending `finalize_transfer` (users cannot receive tokens back from NEAR/EVM).
- Locks the entire vault balance — potentially millions of dollars — with no recovery path inside the bridge program.

If the freeze authority is then renounced (`SetAuthority` to `None`), the freeze is permanent and the funds are irrecoverably lost.

---

### Likelihood Explanation

Any token whose issuer retains a `freeze_authority` is vulnerable. This is a common configuration for regulated stablecoins (e.g., USDC, USDT on Solana both have freeze authorities). The attack requires only that:

1. The token is registered via `log_metadata` (permissionless, callable by anyone).
2. The token issuer (or a compromised key) calls `FreezeAccount` on the vault PDA.

No privileged bridge role is needed. The token issuer is an external party whose behavior the bridge cannot control once the vault is created.

---

### Recommendation

In the `LogMetadata` account constraint, assert that the mint's `freeze_authority` is `None`:

```rust
#[account(
    constraint = !mint.mint_authority.contains(authority.key),
    constraint = mint.freeze_authority == COption::None
        @ ErrorCode::MintHasFreezeAuthority,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
```

For Token-2022 mints, also check that the `PermanentDelegate` extension is absent, as it provides an equivalent capability to drain or freeze-equivalent behavior on token accounts.

---

### Proof of Concept

1. Deploy a native SPL token mint with `freeze_authority = <attacker_keypair>`.
2. Call `log_metadata` — it succeeds because the only constraint checked is `mint_authority`. The vault PDA is created.
3. Multiple users call `init_transfer`, depositing tokens into the vault.
4. Attacker calls SPL Token's `FreezeAccount` on the vault PDA using `<attacker_keypair>`.
5. Any subsequent `init_transfer` or `finalize_transfer` for this mint reverts with `AccountFrozen`.
6. Attacker calls `SetAuthority(freeze_authority, None)` — the vault is now permanently frozen and all deposited tokens are irrecoverably locked. [1](#0-0) [2](#0-1)
