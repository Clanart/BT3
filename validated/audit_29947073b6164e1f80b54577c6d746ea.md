### Title
Timestamp Floor-Rounding in `block_info_for_validate` Allows Expired Transactions to Pass Account Validation — (File: `crates/blockifier/src/context.rs`)

---

### Summary

`BlockContext::block_info_for_validate()` floors the block timestamp to the nearest 3600-second (1-hour) boundary before handing it to every account contract's `__validate__` entry point. The floored value can lag the real block timestamp by up to 3 599 seconds. Any account contract that enforces a transaction deadline with `get_block_timestamp()` inside `__validate__` will therefore accept transactions up to 3 599 seconds after the deadline has already passed — the exact same rounding-direction error as M-2, transposed from option-token expiry to account-validation timestamps.

---

### Finding Description

**Root cause — `block_info_for_validate` in `crates/blockifier/src/context.rs`** [1](#0-0) 

```rust
pub fn block_info_for_validate(&self) -> BlockInfo {
    let block_timestamp = self.block_info.block_timestamp.0;
    let validate_timestamp_rounding =
        self.versioned_constants.get_validate_timestamp_rounding(); // 3600
    let rounded_timestamp =
        (block_timestamp / validate_timestamp_rounding) * validate_timestamp_rounding;
    BlockInfo {
        block_timestamp: BlockTimestamp(rounded_timestamp),
        ...
    }
}
```

The same floor-rounding is replicated in three places that must agree:

1. **Blockifier (Rust, Cairo 1 path)** — `allocate_block_info_segment` selects `block_info_for_validate()` when `ExecutionMode::Validate`. [2](#0-1) 

2. **Blockifier (Rust, Cairo 0 deprecated-syscall path)** — `get_block_timestamp` applies the same floor formula. [3](#0-2) 

3. **Cairo OS program** — `get_block_context` in `block_context.cairo` applies `unsigned_div_rem(block_info.block_timestamp, VALIDATE_TIMESTAMP_ROUNDING)` and multiplies back, producing the same floor. [4](#0-3) 

The constant is 3600 in every versioned-constants file and in the OS constants: [5](#0-4) 

**How the invariant breaks**

An account contract that enforces a per-transaction deadline writes something like:

```cairo
// inside __validate__
let ts = get_block_timestamp();          // receives floored value
assert(ts <= deadline, 'TX_EXPIRED');
```

Because the floored value is always ≤ the real timestamp, the check is systematically *too permissive*:

| Real block timestamp | Deadline | Floored timestamp seen in `__validate__` | Check result |
|---|---|---|---|
| `D + 1` | `D` | `D` (if `D` is on an hour boundary) | **PASS** (should FAIL) |
| `D + 3599` | `D` | `D` | **PASS** (should FAIL) |
| `D + 3600` | `D` | `D + 3600` | FAIL (correct) |

The window of false acceptance is up to **3 599 seconds** (≈ 1 hour minus 1 second).

The gateway's `run_validate_entry_point` builds the `BlockContext` from the previous block's info and increments the block number, but it does not add any independent timestamp-validity guard before calling `blockifier_validator.validate(account_tx)`: [6](#0-5) 

So the floored timestamp is the only time signal the account contract receives during gateway-level validation, and the same floored value is used again during block execution.

---

### Impact Explanation

**Critical — Invalid transaction accepted through account validation logic.**

An account contract that uses `get_block_timestamp()` in `__validate__` to enforce a transaction expiry (session keys, time-locked operations, order-book cancel-after timestamps, etc.) will accept transactions submitted up to 3 599 seconds after the deadline. The sequencer includes the transaction in a block, the OS re-runs `__validate__` with the same floored timestamp and reaches the same wrong conclusion, and the transaction is committed to the canonical state. The account's intended expiry invariant is silently violated.

---

### Likelihood Explanation

The pattern `assert(get_block_timestamp() <= deadline)` inside `__validate__` is a natural and common account-contract design for session keys and time-bounded authorizations. The rounding is not surfaced as a security warning in the versioned-constants comments or in the OS constants file — only as a functional note ("Round down the block number and timestamp when queried inside `__validate__`"). A developer reading the Starknet documentation would not immediately realize that their deadline check has a ≈1-hour blind spot. The trigger requires no privilege: any user can submit a transaction.

---

### Recommendation

Mirror the M-2 fix direction: **round up** the timestamp in validate mode instead of rounding down. This ensures the timestamp seen by `__validate__` is always ≥ the real block timestamp, making deadline checks conservative (they may reject a transaction slightly early) rather than permissive (they accept expired transactions). Concretely, replace:

```rust
let rounded_timestamp =
    (block_timestamp / validate_timestamp_rounding) * validate_timestamp_rounding;
```

with ceiling division:

```rust
let rounded_timestamp =
    block_timestamp.div_ceil(validate_timestamp_rounding) * validate_timestamp_rounding;
```

The same change must be applied in `hint_processor.rs` (deprecated syscall path) and in `block_context.cairo` (OS path) to keep all three implementations consistent.

If rounding up is undesirable for other reasons (e.g., it would expose a future timestamp to contracts), the minimum safe alternative is to require that `deadline + validate_timestamp_rounding ≤ expiry` at account-contract deployment time, analogous to the Bond Protocol's fix of requiring `eligibleDuration` to exceed `epochDuration` by at least one full rounding period.

---

### Proof of Concept

```
validate_timestamp_rounding = 3600

deadline = 7_200_000   (exactly 2000 hours, on an hour boundary)

Attacker submits transaction at real block_timestamp = 7_203_599
  (3599 seconds = ~1 hour past the deadline)

block_info_for_validate computes:
  rounded = (7_203_599 / 3600) * 3600
           = 2000 * 3600
           = 7_200_000

Account __validate__ receives block_timestamp = 7_200_000
  assert(7_200_000 <= 7_200_000)  →  PASSES

Transaction is included in the block and committed.
The account's expiry invariant is violated by 3599 seconds.
```

    fn get_block_timestamp(
        _request: GetBlockTimestampRequest,
        _vm: &mut VirtualMachine,
        syscall_handler: &mut Self,
    ) -> DeprecatedSyscallResult<GetBlockTimestampResponse> {
        let versioned_constants = syscall_handler.context.versioned_constants();
        let block_timestamp = syscall_handler.get_block_info().block_timestamp;
        let block_timestamp = match syscall_handler.execution_mode() {
            ExecutionMode::Validate => {

                    versioned_constants.get_validate_timestamp_rounding();
                BlockTimestamp(
                    (block_timestamp.0 / validate_timestamp_rounding) * validate_timestamp_rounding,
                )
            }
            ExecutionMode::Execute => block_timestamp,
        };
        Ok(GetBlockTimestampResponse { block_timestamp })
    }
