### Title
Stale Base Fee Captured at Ante Handler Construction Bypasses EIP-1559 Fee Enforcement for Cosmos Transactions - (`evmd/ante/handler_options.go`)

### Summary
`newCosmosAnteHandler` reads `feemarketParams` (including `BaseFee`) once at construction time and passes a pointer to the captured snapshot into `NewDynamicFeeChecker`. Because `feemarket.BeginBlock` updates the base fee in the KV store every block, the Cosmos-tx ante handler always validates fees against the stale base fee from app startup, not the current block's base fee.

### Finding Description

`newCosmosAnteHandler` is called once at app startup to build the Cosmos ante handler. It reads `feemarketParams` from the startup context and passes `&feemarketParams` (a pointer to a local copy) to `NewDynamicFeeChecker`:

### Impact Explanation

Any Cosmos native transaction (non-EVM) submitted after the base fee has risen from its initial value passes fee validation using the old (lower) base fee. The user pays less than the protocol-required base fee. This is a fee market bypass for Cosmos txs: the EIP-1559 base fee adjustment mechanism is enforced for EVM txs but silently ignored for Cosmos txs. Fees are mis-accounted — the fee collector receives less than the current base fee requires, and the base fee's congestion-control purpose is defeated for the Cosmos tx path.

### Likelihood Explanation

The base fee changes every block whenever block gas usage deviates from the target. On any active chain, the base fee will diverge from its genesis value within minutes. Every Cosmos tx submitted after that point exploits the stale base fee. No special privileges, keys, or coordination are required — any unprivileged user submitting a Cosmos tx with fees calibrated to the startup base fee triggers this.

### Recommendation

Remove the `ctx sdk.Context` parameter from `newCosmosAnteHandler` and `newLegacyCosmosAnteHandlerEip712`. Instead of capturing params at construction time, read them live inside the returned closure on each tx invocation, mirroring the pattern used by `newEthAnteHandler` with `EVMBlockConfig`. Specifically, `NewDynamicFeeChecker` should accept a keeper interface and call `GetParams` per invocation rather than closing over a pointer to a construction-time snapshot.

### Proof of Concept

1. Chain starts with `BaseFee = 1000` (genesis default).
2. `setAnteHandler` is called; `newCosmosAnteHandler` captures `feemarketParams.BaseFee = 1000`.
3. Several blocks pass with high gas utilization; `feemarket.BeginBlock` raises `BaseFee` to `2000` in the KV store.
4. User submits a Cosmos tx (e.g., `MsgSend`) with `fee = 1000 * gasLimit` (calibrated to the old base fee).
5. `NewDynamicFeeChecker` reads `baseFee` from the stale captured pointer → `baseFee = 1000`.
6. `feeCap = fee / gas = 1000 >= baseFee = 1000` → check passes; tx is accepted and committed.
7. The correct check against the live `baseFee = 2000` would have rejected the tx as `feeCap < baseFee`.
8. The user pays half the required base fee; the fee market's congestion signal is bypassed for Cosmos txs.

func GetBaseFee(height int64, ethCfg *params.ChainConfig, feemarketParams *feemarkettypes.Params) *big.Int {
	if !IsLondon(ethCfg, height) {
		return nil
	}
	baseFee := feemarketParams.GetBaseFee()
	// should not be nil if london hardfork enabled
	if baseFee == nil {
		return new(big.Int)
	}
	return baseFee

	baseFee := k.CalculateBaseFee(ctx)
