### Title
Integer Wrapping in Relative Time-Lock Arithmetic Enables Timelock Bypass and Consensus Divergence — (`File: crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` computes relative time-lock deadlines as `confirmed_height.wrapping_add(height_relative)` (or `timestamp.wrapping_add(seconds_relative)`) when called with `nowrap=false`. An unprivileged spend bundle can supply a max-value relative time-lock argument (e.g., `ASSERT_HEIGHT_RELATIVE = 0xFFFF_FFFF`) that wraps the addition to a small number, making the condition trivially satisfied and bypassing the intended lock. Nodes calling the function with `nowrap=true` (saturating arithmetic) reject the same spend. This produces deterministic consensus divergence between nodes using different `nowrap` values.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` accepts a `nowrap: bool` parameter that selects between two fundamentally different arithmetic semantics for all four relative time-lock conditions: [1](#0-0) 

```rust
if let Some(height_relative) = spend.height_relative {
    if nowrap {
        if prev_transaction_block_height
            < unspent.confirmed_block_index.saturating_add(height_relative)
        { ... }
    } else if prev_transaction_block_height
        < unspent.confirmed_block_index.wrapping_add(height_relative)
    { ... }
}
```

The same dual-path exists for `seconds_relative`, `before_height_relative`, and `before_seconds_relative`: [2](#0-1) 

The project's own test suite explicitly documents the divergent outcomes for max-value arguments: [3](#0-2) 

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
height_relative = 0xffff_ffff:  nowrap=true → Err,  nowrap=false → Ok

// 2000 < 1000 + u64::MAX -> Err with nowrap (saturates), Ok without (wraps)
seconds_relative = 0xffff_ffff_ffff_ffff:  nowrap=true → Err,  nowrap=false → Ok
```

And for the `BEFORE_*` variants: [4](#0-3) 

```
// 200 >= 100 + 0xffff_ffff -> Ok with nowrap (saturates), Err without (wraps)
before_height_relative = 0xffff_ffff:  nowrap=true → Ok,  nowrap=false → Err
```

The Python binding exposes `nowrap` as a plain caller-supplied `bool`: [5](#0-4) 

The Python test suite confirms the same divergence at the integration level: [6](#0-5) 

---

### Impact Explanation

**Timelock bypass (High):** When `nowrap=false`, a spend with `ASSERT_HEIGHT_RELATIVE = 0xFFFF_FFFF` computes `confirmed_height + 0xFFFF_FFFF` via `wrapping_add`. For a coin confirmed at height 10, this wraps to `9`. The check `prev_height < 9` is false at any reasonable chain height, so the spend is **accepted immediately** — bypassing a time-lock that should require ~4.3 billion blocks. The same applies to `ASSERT_SECONDS_RELATIVE = 0xFFFF_FFFF_FFFF_FFFF`.

**Consensus divergence (Critical):** Nodes calling `check_time_locks` with `nowrap=true` reject the same spend. If the network contains a mix of nodes using `nowrap=true` and `nowrap=false` — which is structurally possible because `nowrap` is a caller-supplied parameter with no consensus-enforced value — the two populations will permanently disagree on whether the spend is included in a valid block, causing a chain split.

For `ASSERT_BEFORE_HEIGHT_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE` the divergence is inverted: `nowrap=true` nodes accept the spend (saturating to `u32::MAX`/`u64::MAX`, a deadline never reached), while `nowrap=false` nodes reject it (wrapping to a small value already exceeded).

---

### Likelihood Explanation

The attacker-controlled entry path is a standard CLVM spend bundle. Any unprivileged user can submit a spend with a relative time-lock condition set to `0xFFFF_FFFF` (u32) or `0xFFFF_FFFF_FFFF_FFFF` (u64). The `sanitize_uint` parser accepts these as valid 4-byte / 8-byte unsigned integers: [7](#0-6) 

No special privilege, key material, or network position is required. The divergence is triggered deterministically by the arithmetic mismatch, not probabilistically.

---

### Recommendation

1. **Eliminate the `nowrap` branch entirely.** Saturating arithmetic (`saturating_add`) is the only semantically correct behavior for time-lock deadlines. A relative lock of `u32::MAX` blocks should mean "never spendable within the u32 height range," not "spendable immediately due to wrap." Remove the `wrapping_add` path and the `nowrap` parameter.

2. **If a soft-fork transition is required**, gate the behavior change on a consensus-enforced block height (like `hard_fork2_height` in `get_flags_for_height_and_constants`) rather than a caller-supplied boolean, so all nodes deterministically switch at the same height.

3. **Reject max-value relative time-lock arguments at parse time** in `parse_args` / `sanitize_uint` if they are known to cause wrap-around, treating them as `PositiveOverflow` (already the pattern for other conditions).

---

### Proof of Concept

Craft a spend bundle for a coin confirmed at height 10 with condition:

```
ASSERT_HEIGHT_RELATIVE 0xFFFF_FFFF
```

Call `check_time_locks` at current height 200:

- `nowrap=true`: `10.saturating_add(0xFFFF_FFFF) = u32::MAX`; `200 < u32::MAX` → **rejected** (correct)
- `nowrap=false`: `10u32.wrapping_add(0xFFFF_FFFF) = 9`; `200 < 9` is false → **accepted** (timelock bypassed)

The project's own test fixture confirms this outcome: [8](#0-7) 

```python
# 10 + (2^32 - 10) overflows to 0, wrapping: 15 < 0 -> Ok
# saturating: clamps to u32::MAX, 15 < u32::MAX -> Err
(make_test_conds(height_relative=0xFFFF_FFF6), 13, None),
# 10 + u32::MAX overflows to 9, wrapping: 15 < 9 -> Ok
(make_test_conds(height_relative=0xFFFF_FFFF), 13, None),
```

A node running with `nowrap=False` includes this spend in a block. A node running with `nowrap=True` rejects the block. The chain forks.
