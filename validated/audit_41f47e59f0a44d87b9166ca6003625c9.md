The code confirms the vulnerability. Here is the complete analysis:

**Key finding**: `parse_data_for_block` in `state_diff.rs` validates only `state_diff_length` (chunk count) against the stored header, but **never verifies** `calculate_state_diff_hash(assembled_diff) == header.state_diff_commitment`.

---

### Title
Missing State Diff Commitment Verification in P2P Sync Allows Malicious Peer to Corrupt Stored State — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary
`StateDiffStreamBuilder::parse_data_for_block` assembles a `ThinStateDiff` from peer-supplied `StateDiffChunk` messages and writes it to storage with no cryptographic binding check. The only guard is a length counter (`state_diff_length`) from the stored header. A malicious peer can send chunks whose total `len()` equals `target_state_diff_len` but whose content differs from the committed diff, causing the wrong state diff to be persisted and served authoritatively.

### Finding Description

`parse_data_for_block` reads `target_state_diff_len` from the stored header: [1](#0-0) 

It accumulates chunks until the length matches, then returns the assembled diff: [2](#0-1) 

The only structural checks are:
- `current_state_diff_len == target_state_diff_len` (line 99)
- No duplicate keys per chunk (`ConflictingStateDiffParts`)
- No duplicate deprecated class hashes (`validate_deprecated_declared_classes_non_conflicting`)

There is **no call** to `calculate_state_diff_hash` and **no comparison** against `header.state_diff_commitment`. The assembled `ThinStateDiff` is written directly to storage: [3](#0-2) 

`StateDiffChunk::len()` counts fields, not content: [4](#0-3) 

A peer can trivially craft chunks with the same total `len()` but arbitrary `storage_diffs`, `nonces`, or `class_hash_to_compiled_class_hash` values. The header sync stores `state_diff_commitment` as-is from the peer with no block-hash or signature verification either: [5](#0-4) 

`calculate_state_diff_hash` exists and is used in the committer/batcher path but is never called in the P2P sync client path: [6](#0-5) 

### Impact Explanation

After `append_state_diff` commits the corrupted `ThinStateDiff`, every downstream consumer reads the wrong values:
- `starknet_getStateUpdate` RPC returns authoritative-looking wrong `storage_diffs`, `nonces`, and `class_hash_to_compiled_class_hash`.
- The execution state reader uses the same storage table, so contract storage reads during fee estimation, simulation, and tracing return wrong values.
- The stored `state_diff_commitment` in the header does not match `calculate_state_diff_hash(stored_diff)`, so the node is permanently in an inconsistent state for that block.

### Likelihood Explanation

Any peer that can respond to a `StateDiffQuery` can exploit this. No special privilege is required. The attacker only needs to respond before a legitimate peer does, or be the only peer available. The `BadPeer` path (line 181–188 of `block_data_stream_builder.rs`) is never triggered because the crafted chunks pass all structural checks. [7](#0-6) 

### Recommendation

After assembling `result` and before returning `Ok(Some(...))`, verify the commitment:

```rust
let header = storage_reader.begin_ro_txn()?.get_block_header(block_number)? ...;
let expected_commitment = header.state_diff_commitment
    .ok_or(P2pSyncClientError::OldHeaderInStorage { ... })?;
if calculate_state_diff_hash(&result) != expected_commitment {
    return Err(ParseDataError::BadPeer(BadPeerError::WrongStateDiffCommitment));
}
```

This mirrors the pattern already used in the committer/batcher path.

### Proof of Concept

```
Precondition: header for block N is stored with
  state_diff_length = 2
  state_diff_commitment = calculate_state_diff_hash(correct_diff)
  where correct_diff = ThinStateDiff {
      storage_diffs: { addr_A: { key1: val1 } },
      nonces: { addr_A: nonce_correct },
      ...
  }

Attack:
  Peer sends two StateDiffChunk messages:
    chunk1 = ContractDiff { address: addr_B, nonce: Some(nonce_evil) }  // len=1
    chunk2 = ContractDiff { address: addr_C, nonce: Some(nonce_evil2) } // len=1
  Total len = 2 == target_state_diff_len ✓
  No conflicts ✓
  No empty chunks ✓

Result:
  stored_diff = ThinStateDiff { nonces: { addr_B: nonce_evil, addr_C: nonce_evil2 } }
  calculate_state_diff_hash(stored_diff) != header.state_diff_commitment
  get_state_diff(N) != correct_diff
  starknet_getStateUpdate(N) returns wrong nonces/storage
```
