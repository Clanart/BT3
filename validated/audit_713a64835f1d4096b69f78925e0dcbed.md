Audit Report

## Title
`_fillBids` subtracts `lenAbove` (length of bin `b+1`) instead of `lengthE6` (length of bin `b`) when decrementing `walkDistE6`, corrupting every bid-side price below `curBinIdx` — (`metric-periphery/contracts/lens/MetricOmmPoolDataProvider.sol`)

## Summary

In `_fillBids`, the downward walk through bins reads `lenAbove` from bin `b+1` and subtracts it from `walkDistE6` to compute the lower edge of bin `b`. The correct value to subtract is `lengthE6` of bin `b` itself. When adjacent bins have different widths, `walkDistE6` drifts from the true lower edge on every iteration, causing `lowerX64`, `upperX64`, and `binAvgExecPriceX64` to be wrong for every bin below `curBinIdx`. Integrators consuming these prices may set `amountOutMinimum` too low and receive worse execution than quoted.

## Finding Description

`walkDistE6` is initialised to `curBinDistFromProvidedPriceE6`, the lower edge of `curBinIdx`. [1](#0-0) 

Inside the loop, the code reads `lenAbove` from bin `b+1` (the bin above the current iteration target) and subtracts it: [2](#0-1) 

Then it reads `lengthE6` of the actual current bin `b` — but only uses it for the `upperX64` offset, not for the walk step: [3](#0-2) 

The geometry invariant requires `lower_edge(b) = lower_edge(b+1) − length(b)`. The code instead computes `lower_edge(b+1) − length(b+1)`, which is only correct when all bins share the same width. When `length(b+1) ≠ length(b)`, the error accumulates with each step, corrupting `lowerX64`, `upperX64`, and therefore `binAvgExecPriceX64` for every bin below `curBinIdx`.

**Contrast with `_fillAsks`:** the ask-side correctly advances `cumDistE6` by `lengthE6` of the *current* bin `b`: [4](#0-3) 

## Impact Explanation

`getLiquidityDepth` is the off-chain depth oracle consumed by routers, aggregators, and UI order-sizers. When `length(b+1) > length(b)` for any bin below `curBinIdx`, `walkDistE6` is decremented by more than the true bin width, shifting the computed price range downward. `binAvgExecPriceX64` is understated. An integrator that trusts the quoted average execution price sets `amountOutMinimum` too low; the actual pool swap executes at a price worse than quoted, causing direct loss of user principal. This satisfies the **bad-price execution / lens-quoter path inducing fund-losing trades** impact gate.

## Likelihood Explanation

`lengthE6` is a per-bin `uint16` stored independently for every bin in pool storage. Heterogeneous bin widths are a normal, supported pool configuration — not a privileged or malicious setup. Any pool whose bins differ in width (e.g., narrower bins near mid-price, wider bins further out) triggers the bug on every call to `getLiquidityDepth` that spans more than one bin on the bid side. No special attacker capability is required; any user or integrator calling the lens function on such a pool is affected.

## Recommendation

Remove the redundant read of bin `b+1` and subtract `lengthE6` of the current bin `b` instead:

```solidity
// Before (buggy):
(,, uint16 lenAbove,,) = PoolStateLibrary._binState(pool, int8(b + 1));
walkDistE6 -= int256(uint256(lenAbove));
(, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx);

// After (correct):
(, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx);
walkDistE6 -= int256(uint256(lengthE6));
```

This mirrors the ask-side pattern where `cumDistE6 += int256(uint256(lengthE6))` uses the current bin's own length. [4](#0-3) 

## Proof of Concept

Deploy a pool with two bins where `curBinIdx = 0` has `lengthE6 = 20_000` and bin `-1` has `lengthE6 = 10_000`. Call `getLiquidityDepth`.

- **Correct** lower edge of bin `-1`: `0 − 10_000 = −10_000 E6`
- **Buggy** lower edge (code subtracts `length(0) = 20_000`): `0 − 20_000 = −20_000 E6`

`binAvgExecPriceX64` for bin `-1` is ~1% understated. A Foundry test asserting `bids[1].binAvgExecPriceX64 == correctValue` will fail whenever `length(curBinIdx) != length(curBinIdx - 1)`, confirming the corrupted price is returned to integrators. [5](#0-4)
