### Title
No Refund Mechanism for Failed L1 Handler Transactions Causes Permanent Freezing of Bridged Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The StarkNet OS `execute_l1_handler_transaction` function has no mechanism to refund or recover user funds when an L1→L2 message cannot be executed on L2. When the L1 handler is skipped (marked reverted), the L1→L2 message is silently dropped from the OS output with no on-chain refund path, permanently freezing any assets the user bridged from L1.

---

### Finding Description

In `execute_l1_handler_transaction`, the OS first checks a hint-provided `is_reverted` flag. If `is_reverted != FALSE`, the function returns immediately at lines 387–390 **without** calling `consume_l1_to_l2_message` and **without** any refund or cancellation action:

```cairo
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
``` [1](#0-0) 

When `is_reverted = FALSE`, the message is consumed unconditionally at line 444 **before** the entry point runs:

```cairo
// Consume L1-to-L2 message.
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
let remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=tx_execution_context
);
``` [2](#0-1) 

`non_reverting_select_execute_entry_point_func` hard-asserts that the entry point did **not** revert:

```cairo
assert is_reverted = 0;
``` [3](#0-2) 

`execute_entry_point` returns `is_reverted=1` in two reachable cases triggered by user-controlled inputs:

- **Entry point not found** (selector not in the contract's `l1_handlers` list): [4](#0-3) 

- **Out of gas** before entry point execution: [5](#0-4) 

Because `non_reverting_select_execute_entry_point_func` asserts `is_reverted = 0`, any L1 handler that would revert makes the proof invalid. The sequencer is therefore **forced** to mark such transactions as `is_reverted = TRUE` and skip them. The skipped path returns without consuming the message and without any refund. The `consume_l1_to_l2_message` function itself writes the message to `outputs.messages_to_l2` unconditionally with no conditional rollback: [6](#0-5) 

There is no OS-level path to emit a refund message back to L1 or to cancel the consumed message when the handler fails.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a user bridges assets from L1 to L2 via an L1 bridge contract, the bridge locks the user's tokens on L1 and sends an L1→L2 message. If the targeted L2 contract:

- Does not implement the specified L1 handler selector (entry point not found),
- Has been upgraded via `replace_class` to a version that removed the handler, or
- Has a handler that always reverts,

the sequencer cannot include the transaction in a provable block. It must skip it. The message is never consumed in the OS output, so the L1 bridge never receives confirmation that the message was processed. The user's tokens remain locked in the L1 bridge with no OS-provided mechanism to recover them. The message parameters (including the selector and payload) are fixed at send time and cannot be changed for a replay.

---

### Likelihood Explanation

**High.** This is reachable by any unprivileged L1/L2 message sender. Concrete scenarios:

1. A user sends an L1→L2 message with a selector that does not match any `l1_handler` in the target contract's class — a simple user mistake.
2. A contract is upgraded via `replace_class` after an L1→L2 message is in flight, removing the handler that was present when the message was sent.
3. A malicious actor deliberately sends messages to contracts without matching handlers to grief other users or to lock bridge liquidity.

No privileged access is required. The L1 message sender is the only actor needed.

---

### Recommendation

1. **Add a revert path for L1 handlers**: Instead of using `non_reverting_select_execute_entry_point_func`, use `select_execute_entry_point_func` with a proper revert log, and handle the `is_reverted=1` case by emitting a refund message back to L1 (analogous to how `execute_entry_point` discards L2→L1 messages on revert via the dummy `OsCarriedOutputs`).

2. **Emit a cancellation/refund message to L1 on failure**: When an L1 handler is skipped or reverts, the OS should write a corresponding `MessageToL1Header` to `outputs.messages_to_l1` directed at the originating L1 address, allowing the L1 bridge to release the locked funds.

3. **Consume the message even on revert**: Consume the L1→L2 message in the OS output regardless of execution outcome (to prevent replay), and pair it with a refund message to L1 so the bridge can unlock the user's assets.

---

### Proof of Concept

1. User calls an L1 bridge contract, locking 100 ETH and sending an L1→L2 message with `to_address = CONTRACT_A`, `selector = HANDLER_SELECTOR`, `payload = [amount]`.
2. `CONTRACT_A` does not implement `HANDLER_SELECTOR` as an `l1_handler`.
3. The sequencer receives the L1 handler transaction. It evaluates execution: `execute_entry_point` → `get_entry_point` → `search_sorted_optimistic` finds no match → returns `(success=0)` → returns `(is_reverted=1, retdata=[ERROR_ENTRY_POINT_NOT_FOUND])`. [7](#0-6) 
4. Because `non_reverting_select_execute_entry_point_func` would assert `is_reverted = 0` and fail the proof, the sequencer sets the hint `is_reverted = TRUE`.
5. `execute_l1_handler_transaction` hits the early-return branch at line 387–390. `consume_l1_to_l2_message` is never called. No refund message is emitted. [8](#0-7) 
6. The block is proven successfully. The L1→L2 message is absent from `outputs.messages_to_l2`. The L1 bridge never receives a consumption confirmation. The user's 100 ETH remains locked in the L1 bridge permanently with no OS-level recovery path.

func consume_l1_to_l2_message{outputs: OsCarriedOutputs*}(
    execution_context: ExecutionContext*, nonce: felt
) {
    assert_not_zero(execution_context.calldata_size);
    // The payload is the calldata without the from_address argument (which is the first).
    let payload: felt* = execution_context.calldata + 1;
    tempvar payload_size = execution_context.calldata_size - 1;

    tempvar execution_info = execution_context.execution_info;

    // Write the given transaction to the output.
    assert [outputs.messages_to_l2] = MessageToL2Header(
        from_address=[execution_context.calldata],
        to_address=execution_info.contract_address,
        nonce=nonce,
        selector=execution_info.selector,
        payload_size=payload_size,
    );

    let message_payload = cast(outputs.messages_to_l2 + MessageToL2Header.SIZE, felt*);
    memcpy(dst=message_payload, src=payload, len=payload_size);

    let (outputs) = os_carried_outputs_new(
        messages_to_l1=outputs.messages_to_l1,
        messages_to_l2=outputs.messages_to_l2 + MessageToL2Header.SIZE +
        outputs.messages_to_l2.payload_size,
    );
    return ();
```
