### Title
Epoch-Unbound `descendant_of_tracked_shard_cache` Returns Stale Shard-Tracking Decision, Causing GC to Delete Live State — (`File: chain/epoch-manager/src/shard_tracker.rs`)

---

### Summary

`ShardTracker::check_if_descendant_of_tracked_shard` caches the result of "is shard X a descendant of any configured tracked shard?" using only `ShardId` as the cache key, with no `EpochId` component. After a resharding event, the same numeric `ShardId` can appear in a new epoch with a completely different ancestor relationship. The cache is never invalidated across epochs, so it returns the stale answer from the first epoch in which that `ShardId` was queried. When `gc_state` consults this cache to decide which shards to delete, it can incorrectly conclude that a shard the node is actively tracking is untracked, and permanently delete its trie state.

---

### Finding Description

`ShardTracker` holds a `descendant_of_tracked_shard_cache` field declared as:

```rust
descendant_of_tracked_shard_cache: Arc<Mutex<HashMap<ShardId, bool>>>,
``` [1](#0-0) 

The cache is populated and read in `check_if_descendant_of_tracked_shard`:

```rust
pub fn check_if_descendant_of_tracked_shard(
    &self,
    shard_id: ShardId,
    tracked_shards: &Vec<ShardUId>,
    epoch_id: &EpochId,
) -> Result<bool, EpochError> {
    if let Some(is_tracked) = self.descendant_of_tracked_shard_cache.lock().get(&shard_id) {
        return Ok(*is_tracked);   // ← epoch_id is ignored on cache hit
    }
    let is_tracked = check_if_descendant_of_tracked_shard_impl(
        shard_id, &tracked_shards, &epoch_id, &self.epoch_manager,
    )?;
    self.descendant_of_tracked_shard_cache.lock().insert(shard_id, is_tracked);
    Ok(is_tracked)
}
``` [2](#0-1) 

The cache key is `ShardId` alone. The `epoch_id` argument is passed to `check_if_descendant_of_tracked_shard_impl` only on a cache miss, and is completely ignored on a cache hit. The `HashMap` is never cleared between epochs.

`check_if_descendant_of_tracked_shard_impl` computes the answer by walking the shard layout history for the given `epoch_id`, constructing `ShardUId` values (which embed the layout version) and comparing them against the configured `tracked_shards`. The result is therefore epoch-dependent: the same `ShardId` integer can be a descendant of a tracked shard in epoch B but not in epoch A (or vice versa), because resharding assigns new `ShardUId` versions to child shards while the raw `ShardId` integer may be reused across layouts. [3](#0-2) 

This function is called from `tracks_shard_at_epoch`, which is invoked for the current epoch, the next epoch, and the previous epoch: [4](#0-3) 

The stale cache answer propagates into `gc_state` via `cares_about_shard_this_or_next_epoch`:

```rust

If the cache returns a stale `false` for a shard that the node is actually tracking in the current epoch, `gc_state` includes that shard in `shards_to_cleanup` and permanently deletes its trie state.

---

### Impact Explanation

**Severity: High**

A node configured with `TrackedShardsConfig::Shards(...)` that experiences a resharding event can have its trie state for a tracked shard silently deleted by GC. After deletion, the node cannot apply chunks for that shard, cannot produce or validate state witnesses for it, and cannot serve state sync parts for it. The node effectively loses the ability to participate in consensus for the affected shard without a full re-sync. The deletion is irreversible at runtime.

---

### Likelihood Explanation

The bug is triggered deterministically whenever:
1. The node uses `TrackedShardsConfig::Shards` (a supported production configuration for partial-shard-tracking nodes).
2. A resharding event occurs (already scheduled on mainnet via dynamic resharding).
3. A `ShardId` integer that was queried in an earlier epoch (and cached as `false`) reappears in a later epoch as a descendant of a configured tracked shard.

No privileged role, no network attacker, and no special timing is required. The bug fires on the node's own GC path as a consequence of its own cached state.

---

### Recommendation

Change the cache key from `ShardId` to `(ShardId, EpochId)` so that results are epoch-scoped:

```rust
descendant_of_tracked_shard_cache: Arc<Mutex<HashMap<(ShardId, EpochId), bool>>>,
```

Update `check_if_descendant_of_tracked_shard` to look up and insert `(shard_id, *epoch_id)` instead of `shard_id` alone. This matches the pattern already used by `tracked_accounts_shard_cache`, which correctly uses `EpochId` as its key. [6](#0-5) 

---

### Proof of Concept

**Setup:** Node configured with `TrackedShardsConfig::Shards(vec![parent_shard_uid])` where `parent_shard_uid` is a shard in the genesis layout.

**Step 1 — Epoch A (pre-resharding):** `tracks_shard_at_epoch(shard_id=X, epoch_A)` is called. In epoch A, `ShardId(X)` exists but is NOT a descendant of `parent_shard_uid`. `check_if_descendant_of_tracked_shard_impl` returns `false`. Cache stores `(X → false)`.

**Step 2 — Resharding occurs:** The shard layout changes. `ShardId(X)` is now assigned to a child shard that IS a descendant of `parent_shard_uid` in epoch B.

**Step 3 — Epoch B (post-resharding):** `tracks_shard_at_epoch(shard_id=X, epoch_B)` is called. The shard layout check at line 96 passes (X exists in epoch B's layout). `check_if_descendant_of_tracked_shard` is called. Cache hit on `X` → returns `false` (stale). The node incorrectly concludes it does not track shard X in epoch B.

**Step 4 — GC fires:** `gc_state` calls `cares_about_shard_this_or_next_epoch` for shard X. The stale cache answer propagates. Shard X is included in `shards_to_cleanup`. `delete_shard_uid_prefixed_state` permanently removes the trie state for shard X.

**Result:** The node has lost the state for a shard it is configured to track. It can no longer apply chunks or validate witnesses for that shard. [7](#0-6) [8](#0-7)

fn check_if_descendant_of_tracked_shard_impl(
    shard_id: ShardId,
    tracked_shards: &Vec<ShardUId>,
    epoch_id: &EpochId,
    epoch_manager: &Arc<dyn EpochManagerAdapter>,
) -> Result<bool, EpochError> {
    let tracked_shards: HashSet<ShardUId> = tracked_shards.into_iter().cloned().collect();
    let protocol_version = epoch_manager.get_epoch_protocol_version(epoch_id)?;
    let shard_layout = epoch_manager.get_shard_layout(&epoch_id)?;

    // `ShardLayoutV3` stores all ancestor shards, no need to iterate through protocol versions
    if let Some(ancestors) = shard_layout.ancestor_uids(shard_id) {
        let ancestors = HashSet::from_iter(ancestors);
        return Ok(!ancestors.is_disjoint(&tracked_shards));
    }

    let mut shard_uid = ShardUId::from_shard_id_and_layout(shard_id, &shard_layout);
    if tracked_shards.contains(&shard_uid) {
        // We explicitly track `shard_id` (the shard is a descendant of itself).
        return Ok(true);
    }

    // `shard_uid` does not belong to `tracked_shards`, but it might be a descendant of one.
    // Iterate through consecutive pairs of historical shard layouts (newest to oldest) to trace
    // the ancestry. Each pair represents a resharding transition.
    let layout_history = epoch_manager.get_shard_layout_history(protocol_version, None);
    for window in layout_history.windows(2) {
        let current_layout = &window[0];
        let prev_layout = &window[1];
        let Some(parent_shard_id) = current_layout.try_get_parent_shard_id(shard_uid.shard_id())?
        else {
            debug_assert!(
                false,
                "Parent shard is missing for shard {} in shard layout {:?}",
                shard_uid, current_layout,
            );
            return Ok(false);
        };
        shard_uid = ShardUId::from_shard_id_and_layout(parent_shard_id, &prev_layout);
        if tracked_shards.contains(&shard_uid) {
            return Ok(true);
        }
    }
    Ok(false)
