### Title
Receipt Commitment Omits Actual L2 Gas Consumed, Producing Wrong Block Hash for v0.14.2+ Blocks — (`crates/starknet_api/src/block_hash/receipt_commitment.rs`)

---

### Summary

`chain_gas_consumed` in the receipt commitment path hardcodes `Felt::ZERO` for the L2-gas-consumed slot instead of reading `gas_consumed.l2_gas`. Because `TransactionOutputForHash.gas_consumed` carries the real L2 gas value (non-zero in Starknet ≥ 0.14.2 blocks), every receipt leaf hash is computed with a wrong field, the receipt commitment tree root is wrong, and the final block hash that chains that root is wrong.

---

### Finding Description

`chain_gas_consumed` is the innermost function that serialises execution-resource consumption into a receipt leaf hash:

```rust
// crates/starknet_api/src/block_hash/receipt_commitment.rs  lines 86-91
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&Felt::ZERO)                    // ← hardcoded, ignores gas_consumed.l2_gas
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
``` [1](#0-0) 

The comment acknowledges the field is "always 0 in the current RPC", but `gas_consumed` is a `GasVector` that already carries the real `l2_gas` value from execution:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs  line 103
pub struct TransactionOutputForHash {
    ...
    pub gas_consumed: GasVector,   // l1_gas, l1_data_gas, l2_gas all populated
    ...
}
``` [2](#0-1) 

`GasVector.l2_gas` is populated from `total_gas_consumed` when syncing blocks from the feeder gateway:

```rust
// crates/apollo_starknet_client/src/reader/objects/transaction.rs  lines 752-760
gas_consumed: match execution_resources.total_gas_consumed {
    Some(total_gas_consumed) => total_gas_consumed,   // l2_gas = 76014735 in v0.14.2 blocks
    ...
},
``` [3](#0-2) 

Real block data confirms non-zero values: `block_post_0_14_2.json` shows `"l2_gas": 76014735` in `total_gas_consumed` for every transaction in that block. [4](#0-3) 

The call chain that propagates the wrong value into the block hash is:

1. `BlockExecutionArtifacts::new` calls `calculate_block_commitments`
2. `calculate_block_commitments` calls `calculate_receipt_commitment::<Poseidon>(&receipt_elements)`
3. `calculate_receipt_commitment` maps `calculate_receipt_hash` over every leaf
4. `calculate_receipt_hash` calls `chain_gas_consumed(hash_chain, &receipt_element.transaction_output.gas_consumed)` — the wrong `Felt::ZERO` is chained here
5. The resulting `ReceiptCommitment` is stored in `BlockHeaderCommitments.receipt_commitment`
6. `calculate_block_hash` chains that commitment directly into the Poseidon block hash [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Every block that contains at least one transaction with non-zero `l2_gas` consumed (all Starknet ≥ 0.14.2 blocks) will have:

- A wrong per-transaction receipt leaf hash
- A wrong `receipt_commitment` Patricia root
- A wrong final block hash stored in storage and broadcast to consensus peers

Any external verifier, proof system, or sync peer that computes the receipt commitment correctly (using the actual `l2_gas` value) will derive a different block hash, breaking cross-node agreement on the canonical chain and invalidating any proof that covers the receipt commitment field.

This matches the scope: **Wrong receipt/block hash from execution logic for accepted input — Critical**.

---

### Likelihood Explanation

The trigger is automatic and unprivileged: it fires for every block produced or synced on Starknet ≥ 0.14.2, because those blocks contain transactions whose `total_gas_consumed.l2_gas` is non-zero. No special transaction crafting is required.

---

### Recommendation

Replace the hardcoded `Felt::ZERO` with the actual field from the parameter:

```rust
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&gas_consumed.l2_gas.into())      // use actual L2 gas consumed
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
```

If the Starknet specification intentionally defers L2 gas consumed to a later version, the version gate must be applied explicitly (e.g., only chain `Felt::ZERO` when `starknet_version < V0_14_2`), and the `gas_consumed` parameter should not be accepted at all for those older versions to avoid silent misuse.

---

### Proof of Concept

The regression test in `receipt_commitment_test.rs` uses `l2_gas: GasAmount(0)` in its fixture, so it does not catch the bug:

```rust
// crates/starknet_api/src/block_hash/test_utils.rs  lines 27-31
gas_consumed: GasVector {
    l1_gas: GasAmount(16580),
    l2_gas: GasAmount(0),          // ← zero, masks the bug
    l1_data_gas: GasAmount(32),
},
``` [7](#0-6) 

A test that exposes the divergence:

```rust
#[test]
fn test_l2_gas_consumed_affects_receipt_hash() {
    let mut output_zero = get_transaction_output();
    output_zero.gas_consumed.l2_gas = GasAmount(0);

    let mut output_nonzero = get_transaction_output();
    output_nonzero.gas_consumed.l2_gas = GasAmount(76014735); // real v0.14.2 value

    let elem_zero    = ReceiptElement { transaction_hash: tx_hash!(1), transaction_output: output_zero };
    let elem_nonzero = ReceiptElement { transaction_hash: tx_hash!(1), transaction_output: output_nonzero };

    // With the bug, both hashes are identical because l2_gas is ignored.
    // After the fix, they must differ.
    assert_ne!(
        calculate_receipt_hash(&elem_zero),
        calculate_receipt_hash(&elem_nonzero),
        "receipt hash must differ when l2_gas differs"
    );
}
```

With the current code the assertion **fails** (both hashes are equal), confirming that `l2_gas` is silently dropped from the commitment.
