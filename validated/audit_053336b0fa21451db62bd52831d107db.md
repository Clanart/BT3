### Title
Missing Lower-Bound Timestamp Guard in `is_block_info_valid` Allows Stale Block Timestamps and Wrong L1 Gas Prices in Committed Blocks - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

`is_block_info_valid` enforces only a **one-sided** timestamp window: it rejects timestamps too far in the **future** but places no lower bound on how far in the **past** a proposed timestamp may be. A proposer can set `ProposalInit.timestamp` to any value ≥ `last_block_timestamp`, which may be arbitrarily stale. Because L1 gas prices are looked up using the proposed timestamp, the committed block records L1 gas prices from a past moment rather than the current market, producing wrong fees for every transaction in the block.

### Finding Description

`is_block_info_valid` reads the current wall-clock time and applies two checks:

```rust
// lower bound: only the previous block's timestamp
if init_proposed.timestamp < last_block_timestamp { return Err(...); }
// upper bound: now + window (1 second by default)
if init_proposed.timestamp > now + block_info_validation.block_timestamp_window_seconds {
    return Err(...);
}
``` [1](#0-0) 

The config description for `block_timestamp_window_seconds` reads *"Maximum allowed deviation (seconds) of a proposed block's timestamp from the current time"*, implying a symmetric window, but only the future direction is enforced. [2](#0-1) 

The default value is **1 second**: [3](#0-2) 

After the timestamp passes both checks, L1 gas prices are fetched **using the proposed timestamp**:

```rust
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,   // ← stale timestamp drives price lookup
    ...
).await;
``` [4](#0-3) 

The validator then checks that the proposer's declared prices are within margin of the prices it computed for the **same stale timestamp**. Because both sides use the same timestamp, the margin check passes even when the timestamp is hours old.

The stale timestamp and the prices derived from it are then embedded in the committed block header via `convert_to_sn_api_block_info`: [5](#0-4) 

This path is reachable for both current-round proposals and for **queued future-round proposals**. When a proposal for a future round arrives, it is stored without any timestamp check:

```rust
std::cmp::Ordering::Greater => {
    trace!("Queueing proposal for future round.");
    self.queued_proposals
        .insert(init.round, ((init, timeout, content_receiver), fin_sender));
    fin_receiver
}
``` [6](#0-5) 

When the round eventually advances, the queued proposal is dequeued and validated with the same one-sided check, by which time the timestamp is even more stale: [7](#0-6) 

The `L1GasPriceProvider.get_price_info` only rejects a timestamp that is **ahead** of the latest scraped L1 block by more than `max_time_gap_seconds`; it does not reject past timestamps: [8](#0-7) 

### Impact Explanation

Every transaction in the committed block has its fee computed against the L1 gas prices recorded in the block header. A stale timestamp causes the block to record L1 gas prices from a past moment. If L1 gas was cheaper at that moment, users underpay and the sequencer under-collects fees. If L1 gas was more expensive, users are overcharged. The wrong prices propagate into:

- `l1_gas_price_fri` / `l1_data_gas_price_fri` in the committed `BlockHeaderWithoutHash`
- The block hash (which commits to gas prices)
- Fee deduction for every transaction in the block

This matches **Critical: Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact**.

### Likelihood Explanation

Any validator selected as proposer for a round can exploit this. In a multi-validator network, this is an unprivileged network-participant action. The attack is especially easy via the queued-proposal path: send a proposal for a future round with a timestamp set to `last_block_timestamp` (the minimum allowed value), wait for the current round to time out, and the stale proposal is automatically dequeued and validated.

### Recommendation

Add a symmetric lower-bound check in `is_block_info_valid`:

```rust
if now > block_info_validation.block_timestamp_window_seconds
    && init_proposed.timestamp
        < now - block_info_validation.block_timestamp_window_seconds
{
    return Err(ValidateProposalError::InvalidBlockInfo(
        init_proposed.clone(),
        block_info_validation.clone(),
        format!(
            "Timestamp is too old: now={}, block_timestamp_window_seconds={}, proposed={}",
            now,
            block_info_validation.block_timestamp_window_seconds,
            init_proposed.timestamp
        ),
    ));
}
```

Additionally, validate the timestamp of queued proposals at queuing time (or re-validate it when dequeued) to prevent stale proposals from being accepted after a round timeout.

### Proof of Concept

1. Validator V is selected as proposer for round R+1 (future round).
2. V constructs `ProposalInit` with `timestamp = last_block_timestamp` (e.g., 60 seconds ago when L1 gas was 50% cheaper).
3. V sets `l1_gas_price_fri` and `l1_data_gas_price_fri` to the values returned by `get_price_info(last_block_timestamp)`.
4. V sends the proposal to peers; since `R+1 > current_round`, it is stored in `queued_proposals` without timestamp validation.
5. Round R times out; `set_height_and_round(height, R+1)` is called.
6. The queued proposal is dequeued; `is_block_info_valid` is called with `now = current_time`.
7. Check 1: `last_block_timestamp >= last_block_timestamp` → passes.
8. Check 2: `last_block_timestamp <= current_time + 1` → passes (timestamp is in the past).
9. L1 gas prices are fetched for `last_block_timestamp` → same stale prices V declared → margin check passes.
10. Block is committed with `block_timestamp = last_block_timestamp` and artificially low L1 gas prices; all transaction fees in the block are computed against the stale prices.
