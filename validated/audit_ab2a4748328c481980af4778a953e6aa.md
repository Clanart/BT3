### Title
Malicious P2P Peer Injects Crafted `state_diff_length` in Unverified Header, Causing `parse_data_for_block` to Accept Peer-Controlled `StateDiffChunk` Count and Persist a Corrupted `ThinStateDiff` — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

`HeaderStreamBuilder::parse_data_for_block` accepts a `SignedBlockHeader` from any p2p peer with no cryptographic block-hash or signature verification. A malicious peer can supply a header whose `state_diff_length` field is set to an arbitrary value `N+1`. That header is written verbatim to storage. `StateDiffStreamBuilder::parse_data_for_block` then reads `target_state_diff_len` exclusively from that stored header field and uses it as the **sole loop-termination condition**, consuming exactly `N+1` peer-supplied `StateDiffChunk` messages before assembling and persisting a `ThinStateDiff` whose content and length are entirely peer-controlled. No post-assembly check compares the stored diff against `state_diff_commitment`.

---

### Finding Description

**Step 1 — Header accepted without block-hash or signature verification.**

`HeaderStreamBuilder::parse_data_for_block` performs exactly two checks on the incoming `SignedBlockHeader`:

- `block_number` matches the expected sequence number.
- `signatures.len() == ALLOWED_SIGNATURES_LENGTH` (count only, no cryptographic verification). [1](#0-0) 

`verify_block_signature` (which performs the actual ECDSA check over `hash(block_hash, state_diff_commitment)`) is defined in `starknet_api` but is **never called** in the p2p sync path. The TODO at line 102 explicitly acknowledges that parent-hash checking is also absent. [2](#0-1) 

The accepted header — including its peer-supplied `state_diff_length`, `state_diff_commitment`, and `block_hash` — is written to storage unconditionally: [3](#0-2) 

**Step 2 — `parse_data_for_block` uses the stored `state_diff_length` as the sole termination condition.**

`StateDiffStreamBuilder::parse_data_for_block` reads `target_state_diff_len` from the already-stored header: [4](#0-3) 

The loop runs until `current_state_diff_len >= target_state_diff_len`, consuming exactly as many `StateDiffChunk` messages as the peer-controlled value dictates: [5](#0-4) 

The only post-loop guard checks that the accumulated length equals `target_state_diff_len` — a condition the attacker trivially satisfies by sending exactly `N+1` chunks: [6](#0-5) 

**Step 3 — Corrupted `ThinStateDiff` is written to storage.**

The assembled diff is stored without any comparison against `state_diff_commitment`: [7](#0-6) 

The `state_diff_commitment` field in the block hash commits to both the diff content and its length (via `packed_lengths` / `concat_counts` in the block hash preimage): [8](#0-7) 

Because the peer controls the entire header (including `state_diff_commitment` and `block_hash`), it can supply a self-consistent but fabricated triple `(block_hash, state_diff_commitment, state_diff_length)` that the node stores and then uses to accept a fabricated diff of the matching length.

---

### Impact Explanation

The stored `ThinStateDiff` is the authoritative source for:

- **RPC state queries** — `get_storage_at`, `get_nonce`, `get_class_hash_at` all read from the state trie built from stored diffs. A wrong diff produces wrong RPC answers (High impact: RPC returns an authoritative-looking wrong value).
- **State root / global root** — the trie root derived from the corrupted diff feeds into subsequent block hash computation and proof inputs, propagating the corruption.
- **Proof production** — SNOS input is built from stored state diffs; a corrupted diff produces a proof over wrong state.

The corrupted value is concrete: the `ThinStateDiff` for the targeted block number contains peer-chosen storage slots, nonces, deployed contracts, and declared classes instead of the real ones.

---

### Likelihood Explanation

Any node participating in p2p sync is reachable by any peer in the network. No operator privilege is required. The attacker needs only to be selected as the responding peer for a header or state-diff query, which is a normal network condition. The attack requires crafting a valid-looking `SignedBlockHeader` protobuf with an arbitrary `state_diff_length` and a matching sequence of `StateDiffChunk` messages — both are straightforward to construct.

---

### Recommendation

1. **Cryptographically verify the block signature** in `HeaderStreamBuilder::parse_data_for_block` before accepting any header. Call `verify_block_signature` (already implemented in `starknet_api::block`) against a known sequencer public key.
2. **Verify `state_diff_length` against `state_diff_commitment`** after assembling the `ThinStateDiff` in `parse_data_for_block`: recompute the state-diff commitment from the assembled diff and compare it to the value stored in the header before writing to storage.
3. **Verify `block_hash`** by recomputing it from the header fields (including `packed_lengths` which encodes `state_diff_length`) and rejecting headers whose hash does not match the peer-supplied `block_hash`.

---

### Proof of Concept

```rust
// Integration test sketch (production path, no mocks for the sync logic itself)
//
// 1. Craft a SignedBlockHeader for block 0 with state_diff_length = 2
//    but a state_diff_commitment that commits to a fabricated diff.
// 2. Feed it through HeaderStreamBuilder::parse_data_for_block → write_to_storage.
// 3. Send exactly 2 StateDiffChunk messages encoding fabricated storage writes
//    through StateDiffStreamBuilder::parse_data_for_block → write_to_storage.
// 4. Assert:
//    a. storage.get_state_diff(BlockNumber(0)) == fabricated_diff  (not the real diff)
//    b. fabricated_diff.len() == 2  (peer-controlled, not the real block's length)
//    c. The stored state_diff_commitment does NOT match
//       compute_state_diff_commitment(&real_diff).
```

The test would confirm that `parse_data_for_block` accepts the fabricated chunks because `target_state_diff_len` (read from the peer-supplied header) equals 2, and that the stored diff does not match the real block's `state_diff_commitment`, demonstrating the invariant violation.
