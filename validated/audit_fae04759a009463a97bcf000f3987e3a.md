Audit Report

## Title
Missing Slippage Protection When Removing Liquidity - (File: metric-core/contracts/MetricOmmPool.sol)

## Summary
`MetricOmmPool.removeLiquidity()` accepts no `minAmount0Out`/`minAmount1Out` parameters, and no periphery wrapper for removal exists. A swap that moves the oracle cursor one bin before the LP's transaction lands converts the LP's mixed-token active bin into a single-token bin, causing the LP to receive 0 of one token with no on-chain revert or recourse.

## Finding Description
`MetricOmmPool.removeLiquidity()` delegates directly to `LiquidityLib.removeLiquidity()`, which computes each token's payout proportionally from the bin's live `token0BalanceScaled` and `token1BalanceScaled`:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
```

These balances reflect the bin's current composition at execution time, not at submission time. In the OMM model, the active bin holds both tokens; bins above the cursor hold only token0; bins below hold only token1. A `zeroForOne` swap large enough to exhaust the active bin and advance the cursor to bin N-1 sets `token1BalanceScaled = 0` in the previously active bin. Any subsequent `removeLiquidity` call targeting that bin returns `amount1Removed = 0` with no revert.

A grep across all `metric-core/contracts/**/*.sol` files confirms zero occurrences of `minAmount`, `minOut`, `slippage`, or `InsufficientOutput`, meaning no guard exists anywhere in the call chain. The periphery `MetricOmmPoolLiquidityAdder` provides `maxAmountToken0`/`maxAmountToken1` caps for `addLiquidityExactShares`, but a grep across all `metric-periphery/**/*.sol` files returns zero matches for `removeLiquidity` — no removal entry point exists in the periphery at all, forcing users to call the core pool directly.

Exploit path:
1. Pool cursor is at bin 0 (active bin holds both tokens).
2. LP holds shares in bin 0 and submits `pool.removeLiquidity(owner, salt, deltas_bin0, "")`.
3. Attacker front-runs with a `zeroForOne` swap exhausting bin 0, moving cursor to bin -1; bin 0 now has `token1BalanceScaled == 0`.
4. LP's `removeLiquidity` executes: `amount1Scaled = 0 * sharesToRemove / binTotalSharesVal = 0`; LP receives only token0.
5. Attacker back-runs with the reverse swap, restoring the cursor.
6. LP received 0 token1 with no revert.

## Impact Explanation
The LP suffers a direct loss of owed LP assets: they receive 0 of one token when their proportional claim entitled them to both. To recover the missing token they must execute a swap at the pool's bid/ask spread, paying a real economic cost. On a sandwich, the attacker extracts this spread as profit. The loss scales with position size and spread width, and is repeatable on every `removeLiquidity` call targeting the active bin on any public-mempool chain.

## Likelihood Explanation
Any LP removing liquidity from the active bin is exposed. Moving the cursor requires only a swap large enough to exhaust the active bin — a normal, unprivileged pool operation. On EVM chains with public mempools (Ethereum mainnet, most L2s) front-running is routine. Natural oracle price movements between block submission and inclusion produce the same single-sided outcome without any attacker. The active bin is the most common target for LPs seeking fee income, so the exposed population is large.

## Recommendation
Add `minAmount0Out` and `minAmount1Out` to `MetricOmmPool.removeLiquidity()` and revert if the returned amounts fall below them:

```diff
 function removeLiquidity(
     address owner,
     uint80 salt,
     LiquidityDelta calldata deltas,
+    uint256 minAmount0Out,
+    uint256 minAmount1Out,
     bytes calldata extensionData
 ) external nonReentrant(PoolActions.REMOVE_LIQUIDITY)
   returns (uint256 amount0Removed, uint256 amount1Removed)
 {
     ...
     (amount0Removed, amount1Removed) = LiquidityLib.removeLiquidity(...);
+    if (amount0Removed < minAmount0Out || amount1Removed < minAmount1Out)
+        revert InsufficientOutput(amount0Removed, amount1Removed, minAmount0Out, minAmount1Out);
     _afterRemoveLiquidity(...);
 }
```

Additionally, add a `removeLiquidity` wrapper to `MetricOmmPoolLiquidityAdder` that forwards these bounds, mirroring the `maxAmountToken0`/`maxAmountToken1` pattern already present for `addLiquidityExactShares`.

## Proof of Concept
Foundry integration test:
1. Deploy pool with bins `[-1, 0, 1]`; oracle places cursor at bin 0.
2. LP calls `pool.addLiquidity(owner, salt, deltas_bin0, ...)` depositing equal token0 + token1 into bin 0.
3. Assert `binState[0].token0BalanceScaled > 0` and `binState[0].token1BalanceScaled > 0`.
4. Attacker calls `pool.swap(attacker, true, largeAmount, 0, ...)` exhausting bin 0 and moving cursor to bin -1.
5. Assert `binState[0].token1BalanceScaled == 0`.
6. LP calls `pool.removeLiquidity(owner, salt, deltas_bin0, "")`.
7. Assert `amount1Removed == 0` and transaction did not revert.
8. Attacker calls reverse swap to restore cursor.
9. LP must now swap token0 → token1 at the pool's spread to recover the missing leg.
