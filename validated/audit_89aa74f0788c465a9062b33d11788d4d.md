Audit Report

## Title
Cross-User ETH Consumption in `PeripheryPayments.pay` WETH Branch — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay` uses `address(this).balance` as the ETH source when settling WETH payments. ETH stranded in the router from a prior transaction (a user who sent excess ETH without calling `refundETH`) is silently consumed to partially fund a subsequent user's WETH swap, causing permanent, unrecoverable loss of the stranded ETH.

## Finding Description
The `pay` function's WETH branch reads the contract's entire native ETH balance at line 74: [1](#0-0) 

When `0 < address(this).balance < value`, it deposits all available ETH as WETH and pulls only the remainder from `payer` via `safeTransferFrom`. This is correct when the ETH was sent by the current caller in the same transaction. It is incorrect when ETH from a *prior* transaction is present, because that ETH belongs to a different user.

ETH can be stranded between transactions because all swap entry points are `external payable`: [2](#0-1) 

A user who calls `exactInputSingle{value: X}(amountIn < X)` without a subsequent `refundETH` leaves `X - amountIn` ETH in the contract. The `receive()` guard only blocks plain ETH transfers from non-WETH addresses; it does not prevent ETH from accumulating via payable function calls: [3](#0-2) 

`refundETH` is not enforced on-chain and sends the entire `address(this).balance` to `msg.sender`, with no per-caller accounting: [4](#0-3) 

## Impact Explanation
Direct, permanent loss of user principal. The stranded ETH owner loses their ETH with no recourse; the subsequent swapper benefits by having their WETH requirement reduced. The pool receives the correct total, so pool solvency is unaffected, but the ETH owner suffers a full loss of the stranded amount. This meets the Critical/High threshold for direct loss of user principal.

## Likelihood Explanation
Any user who calls a payable swap function directly with excess ETH and omits `refundETH` will strand ETH. The stranded ETH is then silently consumed by the next WETH swap rather than reverting or being recoverable. The intended usage pattern (`multicall{value}([swap, refundETH])`) is not enforced on-chain, making this a realistic user error with irreversible consequences.

## Recommendation
Scope the ETH available for WETH payment to `msg.value` rather than `address(this).balance`. Pass `msg.value` (or a tracked per-call ETH budget) down through the call stack to `pay`, and use only that amount in the WETH branch. Alternatively, track consumed ETH within the transaction and refund the remainder atomically at the end of each swap entry point, rather than relying on the caller to include `refundETH`.

## Proof of Concept
```
1. userB calls router.exactInputSingle{value: 1 ether}(
       tokenIn=WETH, amountIn=0.5 ether, ...
   )
   // pay() uses 0.5 ETH for the swap, 0.5 ETH stranded in router

2. userA (with only 0.5 WETH approved) calls router.exactInputSingle(
       tokenIn=WETH, amountIn=1 ether, ...
   )
   // pay(WETH, userA, pool, 1 ether):
   //   nativeBalance = 0.5 ether (userB's stranded ETH)
   //   deposits 0.5 ETH → WETH → pool
   //   safeTransferFrom(userA, pool, 0.5 WETH)  ← only 0.5 WETH pulled
   // swap succeeds; userB's 0.5 ETH permanently lost
```
