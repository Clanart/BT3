### Title
Uninitialized `eip155ChainID` Causes Nil-Pointer Dereference in EIP-7702 `validateAuthorization` via gRPC Query Path — (`File: x/evm/keeper/set_code_authorizations.go`)

---

### Summary

`Keeper.eip155ChainID` is a `*big.Int` field that is **never set in `NewKeeper`**, leaving it `nil` until `WithChainID` is called by `InitGenesis` (first boot only) or `BeginBlock` (every subsequent restart). The EIP-7702 `validateAuthorization` function dereferences `k.eip155ChainID` unconditionally via `auth.ChainID.CmpBig(k.eip155ChainID)`. Any gRPC query (`eth_call`, `eth_estimateGas`, `eth_createAccessList`) that carries a SetCode authorization with a non-zero `auth.ChainID` and arrives **before the first `BeginBlock` after a node restart** will call `CmpBig(nil)`, triggering a nil-pointer dereference panic that crashes the node process.

---

### Finding Description

`NewKeeper` constructs the `Keeper` struct and explicitly initializes every field **except** `eip155ChainID`:

```go
return &Keeper{
    cdc:               cdc,
    authority:         authority,
    accountKeeper:     ak,
    ...
    // eip155ChainID is NOT set here — remains nil
}
``` [1](#0-0) 

`eip155ChainID` is only populated lazily:

- `InitGenesis` → `k.WithChainID(ctx)` (first-boot only)
- `BeginBlock` → `k.WithChainID(ctx)` (every block) [2](#0-1) [3](#0-2) 

After a **node restart**, `eip155ChainID` is `nil` until the first `BeginBlock` fires. During this window, gRPC query handlers are fully reachable. The EIP-7702 `validateAuthorization` function uses `k.eip155ChainID` without any nil guard:

```go
func (k *Keeper) validateAuthorization(auth *types.SetCodeAuthorization, stateDB vm.StateDB) (authority common.Address, err error) {
    // Verify chain ID is null or equal to current chain ID.
    if !auth.ChainID.IsZero() && auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
        return authority, core.ErrAuthorizationWrongChainID
    }
``` [4](#0-3) 

`auth.ChainID` is a `uint256.Int`; `CmpBig` accepts a `*big.Int`. When `k.eip155ChainID` is `nil`, `CmpBig(nil)` dereferences the nil pointer inside the holiman/uint256 library, causing a panic.

`validateAuthorization` is called from `applyAuthorization`, which is called inside `ApplyMessageWithConfig` whenever `msg.SetCodeAuthorizations != nil`:

```go
for _, auth := range msg.SetCodeAuthorizations {
    authority, err := k.applyAuthorization(&auth, stateDB)
``` [5](#0-4) 

`ApplyMessageWithConfig` is invoked by gRPC query handlers (`EthCall`, `EstimateGas`, `CreateAccessList`) that accept user-supplied `TransactionArgs` including an `AuthorizationList`. These handlers pass the caller-controlled `chainID` to `EVMConfig` but **never call `k.WithChainID`**, so `k.eip155ChainID` remains nil:

```go
cfg, err := k.EVMConfig(ctx, chainID, common.Hash{})   // chainID ≠ k.eip155ChainID
``` [6](#0-5) 

The CHANGELOG itself acknowledges the pre-`BeginBlock` nil-chain-ID window as a real operational concern:
> `(rpc) [#501] Avoid invalid chain id for signer error when rpc call before chain id set in BeginBlock.`

That fix addressed the RPC signer path; `validateAuthorization` (added later for EIP-7702) was not protected.

---

### Impact Explanation

An attacker who observes a validator node restart (e.g., via monitoring public RPC liveness) can immediately send a gRPC `eth_call` or `eth_estimateGas` request containing a SetCode (EIP-7702) authorization with any non-zero `ChainID`. This triggers `CmpBig(nil)` in `validateAuthorization`, causing a nil-pointer dereference panic. If the gRPC server does not recover from the panic (Cosmos SDK query handlers do not universally wrap panics), the node process crashes. Repeated targeting of restarting validators can prevent them from rejoining consensus, constituting a targeted chain-halt attack against the validator set.

This matches: **High — Public gRPC/simulation path exposes a reachable route to chain halt.**

---

### Likelihood Explanation

- EIP-7702 (`SetCode`) transactions are a newly supported transaction type; the `validateAuthorization` code path is production-active.
- Node restarts are routine (upgrades, crashes, maintenance). The window between restart and first `BeginBlock` is seconds to minutes.
- The attack requires only a single unauthenticated gRPC call with a crafted `eth_call` payload — no funds, no keys, no privileged access.
- The CHANGELOG confirms the pre-`BeginBlock` nil-chain-ID window is a known operational reality on this codebase.

---

### Recommendation

**Short term**: Add a nil guard in `validateAuthorization` before calling `CmpBig`:

```go
if !auth.ChainID.IsZero() && k.eip155ChainID != nil && auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
    return authority, core.ErrAuthorizationWrongChainID
}
```

Or, return an explicit error when `k.eip155ChainID` is nil:

```go
if k.eip155ChainID == nil {
    return authority, errors.New("chain ID not yet initialized")
}
```

**Long term**: Audit all uses of `k.eip155ChainID` (found in `keeper.go`, `state_transition.go`, `grpc_query.go`, and `set_code_authorizations.go`) for nil-dereference exposure. Consider initializing `eip155ChainID` eagerly in `NewKeeper` by accepting the chain ID string as a constructor parameter, mirroring how `authority` is validated at construction time. [7](#0-6) [8](#0-7)

### Proof of Concept

1. Start (or restart) a node running this Ethermint build.
2. Before the first block is committed (i.e., before `BeginBlock` fires), send the following gRPC request to the `EthCall` endpoint:

```json
{
  "args": {
    "from": "0xdeadbeef...",
    "to":   "0xcafebabe...",
    "gas":  "0x100000",
    "authorizationList": [
      {
        "chainId": "0x2328",   // non-zero: any value != 0
        "address": "0x1234...",
        "nonce":   "0x0",
        "v": "0x1",
        "r": "0xaaaa...",
        "s": "0xbbbb..."
      }
    ]
  },
  "gasCap": 1000000
}
```

3. `ApplyMessageWithConfig` processes `msg.SetCodeAuthorizations`, calls `applyAuthorization` → `validateAuthorization`.
4. `auth.ChainID.IsZero()` is `false` (non-zero chain ID supplied).
5. `auth.ChainID.CmpBig(k.eip155ChainID)` is called with `k.eip155ChainID == nil`.
6. Nil-pointer dereference panic → node process terminates. [4](#0-3) [9](#0-8)
