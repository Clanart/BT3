Audit Report

## Title
Router `pay()` consumes contract-wide ETH balance without scoping to caller's `msg.value`, enabling theft of stranded ETH — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

The `pay()` helper in `PeripheryPayments.sol` funds WETH swaps using `address(this).balance`, which is not scoped to the current transaction's `msg.value`. Any ETH left in the router by a prior caller who omitted `refundETH()` can be silently consumed by a subsequent caller who sends 0 ETH with a WETH swap, causing permanent loss of the prior user's principal.

## Finding Description

In `PeripheryPayments.sol` lines 73–84, when `token == WETH` and `payer != address(this)`, `pay()` reads the entire native balance of the contract:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

`address(this).balance` accumulates across all callers and all transactions. The `receive()` guard at lines 32–34 only rejects direct ETH transfers (no calldata) from non-WETH addresses; it does not prevent ETH from entering via `payable` entry points such as `exactInputSingle` (line 67), `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, `refundETH`, `unwrapWETH9`, and `sweepToken`. None of these entry points verify that `msg.value >= amountIn` for WETH swaps.

The exploit path through `exactInputSingle`:
1. `exactInputSingle` (line 67) is `payable` and stores `msg.sender` as payer via `_setNextCallbackContext` (line 71).
2. The pool calls `metricOmmSwapCallback` → `_justPayCallback` (lines 192–199) → `pay(WETH, payer, pool, amountOwed)`.
3. `pay()` reads `address(this).balance`, which may include ETH from prior callers, and uses it to wrap and forward WETH to the pool — without verifying the ETH originated from the current `msg.value`.

The same path is reachable through `MetricOmmPoolLiquidityAdder.metricOmmModifyLiquidityCallback()` at lines 172–177 when either pool token is WETH.

No existing guard prevents this: there is no transient tracking of per-call ETH contributed, no `msg.value >= amountIn` check at entry, and no mechanism to distinguish "ETH I sent now" from "ETH left by a prior caller."

## Impact Explanation

Direct loss of user principal. ETH stranded in the router (e.g., because `refundETH()` was omitted from a multicall, or a partial revert left excess ETH behind) is permanently transferred to the pool on behalf of a subsequent attacker who contributes 0 ETH. The loss equals the stranded ETH amount and is irreversible. This meets the Metric OMM allowed impact gate for direct loss of user principal.

## Likelihood Explanation

Medium. Sending excess ETH to avoid slippage-induced reverts and relying on `refundETH()` for recovery is standard router usage. Omitting `refundETH()` from a multicall or experiencing a partial revert that leaves ETH in the router is a realistic user error. An attacker can monitor `address(router).balance` on-chain (zero cost) and immediately exploit any non-zero balance by calling `exactInputSingle` with `tokenIn = WETH` and `msg.value = 0`. The attack is repeatable and requires no special privileges.

## Recommendation

Track the ETH contributed by the current call in transient storage at the entry of each `payable` swap function (e.g., store `msg.value` alongside the existing callback context in `TransientCallbackPool`). In `pay()`, use only that tracked per-call amount — not `address(this).balance` — when funding WETH payments. Alternatively, remove native-ETH support from `pay()` entirely and require callers to wrap ETH before invoking the router, eliminating the ambiguity between current-call ETH and accumulated ETH.

## Proof of Concept

1. User A calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = 1 ETH`, `msg.value = 2 ETH`. The swap callback uses 1 ETH; 1 ETH remains in the router. User A omits `refundETH()`.
2. User B calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = 1 ETH`, `msg.value = 0`.
3. The pool calls `metricOmmSwapCallback` → `_justPayCallback` → `pay(WETH, userB, pool, 1e18)`.
4. `address(this).balance == 1e18 >= 1e18` → the `nativeBalance >= value` branch fires; the router wraps User A's 1 ETH and transfers WETH to the pool.
5. User B receives full swap output. User A's 1 ETH is permanently lost; User B paid nothing.
