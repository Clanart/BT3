### Title
Insufficient Amount Validation in `init_transfer` Causes Permanent Fund Lock When Normalized Amount Rounds to Zero - (`File: near/omni-bridge/src/lib.rs`)

### Summary

`init_transfer` only validates that `fee < amount`, but does not verify that `normalize_amount(amount - fee) > 0`. When a user bridges a small token amount across a chain pair with a large decimal difference (e.g., 24 NEAR decimals → 6 EVM decimals), the normalized transfer amount floors to zero. The relayer's subsequent `sign_transfer` call permanently reverts with `InvalidAmountToTransfer`, and there is no cancel or refund path. The user's tokens are irrecoverably locked or burned in the bridge.

### Finding Description

**Step 1 — User initiates transfer, tokens are locked/burned.**

In `init_transfer`, the only fee/amount guard is:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

This allows `amount = 1` with `fee = 0` to pass. The call then proceeds to `init_transfer_internal`, which burns or locks the tokens and stores the transfer message in `pending_transfers`: [2](#0-1) 

**Step 2 — Relayer calls `sign_transfer`, which permanently reverts.**

`sign_transfer` normalizes the amount for the destination chain's decimal precision:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [3](#0-2) 

The codebase itself documents that `normalize_amount` uses floor division:

> "Since `denormalize(normalize(x)) <= x` due to floor division, the difference naturally captures the normalization remainder." [4](#0-3) 

For a token with 24 NEAR decimals and 6 EVM decimals, any amount below `10^18` normalizes to 0. The `require!` at line 482 will always panic, making `sign_transfer` permanently uncallable for this transfer.

**Step 3 — No recovery path exists.**

The only function that removes a transfer from `pending_transfers` is `claim_fee_callback`, which requires a valid proof from the destination chain — impossible since the transfer was never finalized. There is no user-initiated cancel or refund function. [5](#0-4) 

### Impact Explanation

User tokens are permanently locked (for native tokens) or permanently burned (for deployed bridge tokens) with no recovery mechanism. This matches the **Critical** impact category: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.*

### Likelihood Explanation

This is reachable by any unprivileged bridge user. It is triggered whenever:
- A user transfers a token whose NEAR-side decimals significantly exceed the destination chain's decimals (e.g., 24 vs 6, a common pairing for EVM tokens wrapped on NEAR).
- The transferred amount (minus fee) is less than `10^(origin_decimals - dest_decimals)`.

This is a realistic scenario: a user bridging a small dust amount, or a token with an unusual decimal configuration. No special role or privileged access is required.

### Recommendation

Add a normalization check inside `init_transfer` (or `init_transfer_internal`) before locking/burning tokens:

```rust
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(token_address) = token_address {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee().unwrap_or(0),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

This mirrors the fix in the referenced report: validate the constraint at the entry point, before any irreversible state change (token lock/burn), rather than only at the signing step.

### Proof of Concept

1. Deploy a token with 24 decimals on NEAR, registered on the bridge with 6 decimals on EVM.
2. Call `ft_transfer_call` on the token contract with `amount = 1` (1 yocto-unit), `fee = 0`, and a valid EVM recipient.
3. `init_transfer` passes the `fee < amount` check (0 < 1 ✓), locks 1 yocto-unit of the token, and stores the transfer in `pending_transfers`.
4. The relayer calls `sign_transfer` for this transfer ID.
5. `normalize_amount(1, {decimals: 6, origin_decimals: 24})` = `1 / 10^18` = 0 (floor division).
6. `require!(0 > 0, ...)` panics with `InvalidAmountToTransfer`.
7. Every subsequent `sign_transfer` call for this transfer ID panics identically.
8. The 1 yocto-unit is permanently locked in the bridge with no cancel or refund path.
