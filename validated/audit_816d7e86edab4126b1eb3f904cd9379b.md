### Title
Stale Fee-Market Params Snapshot in `NewDynamicFeeChecker` Allows Cosmos Transactions to Bypass the EIP-1559 Base Fee - (`ante/evm/fee_checker.go`, `evmd/ante/handler_options.go`)

### Summary

`newCosmosAnteHandler` (and `newLegacyCosmosAnteHandlerEip712`) capture `feemarketParams` — including `BaseFee` — once at handler-construction time and pass a pointer to that frozen snapshot into `NewDynamicFeeChecker` and `MinGasPriceDecorator`. Because the EIP-1559 base fee is recalculated and written to the store on every `BeginBlock`, the fee checker used for all Cosmos (non-EVM) transactions operates against a permanently stale base fee, allowing those transactions to be included with fees below the actual current base fee.

### Impact Explanation

When the chain's base fee rises above the genesis value (which happens whenever blocks are consistently above the gas target), `NewDynamicFeeChecker` accepts Cosmos transactions whose `feeCap` satisfies only the stale genesis base fee. The check at line 86:

```go
if feeCap.LT(baseFeeInt) { return error }   // baseFeeInt = stale genesis value
``` [8](#0-7) 

…passes for any `feeCap >= genesis_baseFee`, even if `feeCap < actual_baseFee`. The effective fee deducted is `min(stale_baseFee + tip, feeCap) * gas`, which is less than `actual_baseFee * gas`. This constitutes a fee-market bypass: Cosmos transactions are committed with fees below the protocol-required base fee, mis-accounting user funds and undermining the EIP-1559 fee market for the Cosmos transaction path. This matches the allowed High impact: *"ante handler… bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."*

### Likelihood Explanation

The base fee changes every block. On any chain that has experienced sustained load since genesis, the live base fee will exceed the genesis value. Any Cosmos SDK transaction (bank sends, IBC relays, governance votes, etc.) submitted with a `feeCap` calibrated to the genesis base fee will be incorrectly accepted. This is a persistent, always-on condition — not a race — and requires no special attacker capability beyond submitting a normal Cosmos transaction.

### Recommendation

`newCosmosAnteHandler` (and `newLegacyCosmosAnteHandlerEip712`) must not snapshot `feemarketParams` at construction time. Instead, `NewDynamicFeeChecker` should read the live base fee from the keeper on every invocation, mirroring the pattern used by `newEthAnteHandler`:

```go
// Instead of capturing feemarketParams at construction, pass the keeper
// and read live params inside the closure:
func NewDynamicFeeChecker(ethCfg *params.ChainConfig, evmKeeper ..., fmKeeper ...) authante.TxFeeChecker {
    return func(ctx sdk.Context, tx sdk.Tx) (sdk.Coins, int64, error) {
        feemarketParams := fmKeeper.GetParams(ctx)   // live read per invocation
        baseFee := types.GetBaseFee(ctx.BlockHeight(), ethCfg, &feemarketParams)
        ...
    }
}
```

Similarly, `MinGasPriceDecorator` should call `mpd.feesKeeper.GetParams(ctx).MinGasPrice` inside `AnteHandle` rather than reading the frozen snapshot field.

### Proof of Concept

1. Chain starts; genesis `BaseFee = 1_000_000_000` (1 Gwei). `newCosmosAnteHandler` is called once, capturing `feemarketParams.BaseFee = 1 Gwei`.

2. The chain runs with blocks above the gas target. After N blocks, `BeginBlock` has updated the live `BaseFee` to `10_000_000_000` (10 Gwei) in the KV store. The Cosmos ante handler's snapshot still holds `1 Gwei`.

3. An unprivileged user constructs a Cosmos `MsgSend` with:
   - `gas = 200_000`
   - `fee = 200_000 * 1 Gwei = 200_000_000_000` (calibrated to genesis base fee)

4. `NewDynamicFeeChecker` computes `feeCap = fee / gas = 1 Gwei`. It checks `feeCap.LT(baseFeeInt)` where `baseFeeInt = 1 Gwei` (stale). The check passes.

5. The transaction is included in a block and committed. The user paid `200_000_000_000` instead of the required `2_000_000_000_000` — a 10× underpayment relative to the actual base fee. The fee market is bypassed for this transaction class.
