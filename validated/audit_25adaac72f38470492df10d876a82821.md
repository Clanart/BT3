### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the new class hash supplied by a contract corresponds to a previously declared contract class. This allows any contract to replace its own class hash with an arbitrary, undeclared value — permanently rendering itself non-executable and freezing any funds held within it. The missing check is explicitly acknowledged by a TODO comment in the production code.

---

### Finding Description

In `execute_replace_class` (lines 877–916 of `syscall_impls.cairo`), the function reads the requested `class_hash` directly from the syscall request buffer and immediately writes it into the contract's `StateEntry` in `contract_state_changes` without any validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 explicitly confirms the missing invariant enforcement. No check is performed to verify that `class_hash` exists in `contract_class_changes` (the declared class registry) or that it is non-zero (i.e., not `UNINITIALIZED_CLASS_HASH`). [2](#0-1) 

The `UNINITIALIZED_CLASS_HASH` constant is defined as `0` in `commitment.cairo`: [3](#0-2) 

The `StateEntry` struct that is updated holds the contract's class hash, storage pointer, and nonce — the complete on-chain identity of the contract: [4](#0-3) 

Once the class hash is set to an undeclared value, the contract's state entry is committed to the global state tree via `compute_contract_state_commitment` → `hash_contract_state_changes`, which hashes whatever `class_hash` is present without re-validating it: [5](#0-4) 

The corrupted class hash is then permanently committed to the Merkle state root, making the damage irreversible at the protocol level.

---

### Impact Explanation

**Critical — Permanent Freezing of Funds.**

Once a contract's class hash is replaced with an undeclared value:

1. The OS cannot locate a compiled class for that hash in the `compiled_class_facts_bundle`.
2. All future calls to the contract revert at the class-lookup stage.
3. The contract cannot call `replace_class` again to self-recover, because execution of the contract itself is impossible.
4. All token balances, collateral, or other assets stored in the contract's storage are permanently inaccessible.

The corrupted state is committed to the global Merkle root and propagated to L1 via the OS output, making recovery impossible without a protocol-level hard fork.

---

### Likelihood Explanation

The attack path is reachable by any unprivileged contract deployer:

1. A malicious actor deploys a contract that accepts user deposits (e.g., a fake yield vault or bridge).
2. Users deposit funds into the contract.
3. The malicious contract issues a `replace_class` syscall with `class_hash` set to an arbitrary undeclared felt (or `0`).
4. The OS processes the syscall without validation, updating the contract's state entry.
5. The block is finalized and the corrupted state root is committed to L1.
6. All deposited funds are permanently frozen.

No privileged role, leaked key, or operator compromise is required. The attacker only needs to deploy a contract and have users interact with it — a standard DeFi interaction pattern.

---

### Recommendation

Add an explicit validation in `execute_replace_class` to assert that the requested `class_hash` is present in `contract_class_changes` (i.e., has been declared in the current or a prior block) and is non-zero before updating the contract's state entry. This is the check the TODO comment already identifies as necessary:

```cairo
// Ensure the class hash is declared and non-zero.
assert_not_zero(class_hash);
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the invariant enforced during `execute_declare_transaction`, where `prev_value=0` is asserted to prevent re-declaration, and `compiled_class_hash` is verified non-zero before writing: [6](#0-5)

### Proof of Concept

**Step 1**: Deploy a malicious Cairo contract `MaliciousVault` that:
- Exposes a `deposit()` entry point accepting ERC-20 transfers.
- Exposes a `freeze()` entry point that calls the `replace_class` syscall with `class_hash = 0xdeadbeef` (an undeclared felt).

**Step 2**: Users call `deposit()`, transferring tokens into `MaliciousVault`. The contract's storage now holds user balances.

**Step 3**: The attacker calls `freeze()`. The OS executes `execute_replace_class`:

```cairo
// request.class_hash = 0xdeadbeef (undeclared)
let class_hash = request.class_hash;
// No validation — proceeds directly to dict_update
dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new StateEntry(class_hash=0xdeadbeef, ...), felt),
);
``` [7](#0-6) 

**Step 4**: `state_update` in `state.cairo` squashes and commits the corrupted state entry to the Merkle tree: [8](#0-7) 

**Step 5**: The block is finalized. `MaliciousVault` now has `class_hash = 0xdeadbeef` in the global state. Any subsequent call to `MaliciousVault` fails at class lookup. User funds are permanently frozen. The state root reflecting this is posted to L1 via the OS output.
