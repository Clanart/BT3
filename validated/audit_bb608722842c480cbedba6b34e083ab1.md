### Title
Missing `_grantRole(MINTER_ROLE, ...)` in `ENearProxy.initialize()` Permanently Blocks eNEAR Bridge Mint and Burn — (`File: evm/src/eNear/contracts/ENearProxy.sol`)

---

### Summary

`ENearProxy` defines `MINTER_ROLE` and gates both `mint()` and `burn()` behind it, but `initialize()` never grants `MINTER_ROLE` to any address. Because `OmniBridge` calls `ENearProxy.mint()` and `ENearProxy.burn()` during `finTransfer()` and `initTransfer()` respectively, every eNEAR bridge transfer in both directions will revert immediately after deployment until an admin manually grants the role — a step that is absent from the initializer and from any deployment script in scope.

---

### Finding Description

`ENearProxy.initialize()` grants only `DEFAULT_ADMIN_ROLE` and `PAUSABLE_ADMIN_ROLE`: [1](#0-0) 

`MINTER_ROLE` is declared but never granted to any address during initialization: [2](#0-1) 

Both `mint()` and `burn()` are gated exclusively behind `onlyRole(MINTER_ROLE)`: [3](#0-2) [4](#0-3) 

`ENearProxy` is registered as the `customMinter` for the eNEAR token via `OmniBridge.addCustomToken()`. When `OmniBridge.finTransfer()` processes an eNEAR destination transfer, it dispatches: [5](#0-4) 

And when `OmniBridge.initTransfer()` processes an eNEAR source transfer, it dispatches: [6](#0-5) 

Both calls target `ENearProxy`, which will revert with `AccessControlUnauthorizedAccount` because `OmniBridge` (the caller) holds no `MINTER_ROLE`. The `ICustomMinter` interface confirms these are the only two entry points: [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent freezing / unclaimable settlement of user funds in bridge flows.**

- **NEAR → EVM (finTransfer)**: A user locks eNEAR on NEAR and a relayer calls `OmniBridge.finTransfer()`. The call reaches `ENearProxy.mint()`, reverts, and the destination nonce is never marked complete (the `completedTransfers` write at line 287 is also rolled back). The user's funds are locked on NEAR with no way to claim the EVM-side settlement until an admin grants the role.
- **EVM → NEAR (initTransfer)**: A user approves eNEAR and calls `OmniBridge.initTransfer()`. The call reaches `ENearProxy.burn()` and reverts. The user's tokens are not transferred (the `safeTransferFrom` at line 395 precedes the `burn` call, so the token pull also reverts), but the bridge path is entirely non-functional.

Both directions of the eNEAR bridge are completely frozen from the moment of deployment.

---

### Likelihood Explanation

**High.** The missing grant is structural — it is absent from `initialize()` and there is no evidence in any in-scope deployment script that `MINTER_ROLE` is granted post-deployment. Any deployment of `ENearProxy` without a manual follow-up `grantRole` call leaves the contract in a broken state. This exactly mirrors the original report's finding that the role was missing in all relevant `initialize()` functions and deployment scripts.

---

### Recommendation

Add `_grantRole(MINTER_ROLE, _minterAddress)` inside `initialize()`, where `_minterAddress` is the `OmniBridge` proxy address (or a dedicated keeper address), analogous to how `DEFAULT_ADMIN_ROLE` is granted to `_adminAddress`:

```solidity
function initialize(
    address _eNear,
    address _prover,
    bytes memory _nearConnector,
    uint256 _currentReceiptId,
    address _adminAddress,
    address _minterAddress   // <-- add parameter
) public initializer {
    // ... existing init ...
    _grantRole(DEFAULT_ADMIN_ROLE, _adminAddress);
    _grantRole(PAUSABLE_ADMIN_ROLE, _msgSender());
    _grantRole(MINTER_ROLE, _minterAddress);  // <-- add this
}
```

---

### Proof of Concept

1. Deploy `ENearProxy` via its proxy, calling `initialize(eNear, prover, nearConnector, 0, adminAddr)`.
2. Deploy `OmniBridge` and call `addCustomToken(nearTokenId, eNearAddr, eNearProxyAddr, decimals)` — this registers `ENearProxy` as the custom minter for eNEAR.
3. Any user calls `OmniBridge.finTransfer(sig, payload)` where `payload.tokenAddress == eNearAddr`.
4. Execution reaches `ICustomMinter(eNearProxyAddr).mint(eNearAddr, recipient, amount)` at `OmniBridge.sol:332`.
5. `ENearProxy.mint()` checks `onlyRole(MINTER_ROLE)` — `OmniBridge` has no such role → **reverts with `AccessControlUnauthorizedAccount`**.
6. The entire `finTransfer` transaction reverts; the user's NEAR-side funds remain locked and the EVM-side settlement is permanently unclaimable until an admin manually calls `ENearProxy.grantRole(MINTER_ROLE, omniBridgeAddr)`.
