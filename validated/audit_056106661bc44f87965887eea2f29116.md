### Title
Non-EVM-Native (IBC) Token Balances Permanently Orphaned on Contract SELFDESTRUCT — (`File: x/evm/statedb/statedb.go`)

---

### Summary

`StateDB.Commit()` only drains the EVM-native denomination when finalizing a selfdestructed account. Non-EVM-native Cosmos bank tokens (IBC tokens, CosmWasm bridge tokens, etc.) held by the contract address are never burned or transferred — they remain in the bank module under the now-deleted account address, permanently inaccessible to anyone.

---

### Finding Description

In `StateDB.Commit()`, when processing a selfdestructed state object, the code performs two operations:

1. Burns any remaining EVM-denom balance via `s.keeper.SubBalance(...)` using only `s.evmDenom`.
2. Calls `s.keeper.DeleteAccount(...)` which removes auth metadata and contract storage.

The code itself documents the gap:

```go
// Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.
if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
    coin := sdk.NewCoin(s.evmDenom, sdkmath.NewIntFromBigInt(remaining.ToBig()))
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct balance")
    }
}
if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil {
    return errorsmod.Wrap(err, "failed to delete account")
}
``` [1](#0-0) 

`DeleteAccount` removes only auth metadata and contract storage — it never touches bank balances of any denomination:

```go
// NOTE: balance should be cleared separately

``` [2](#0-1) 

After `DeleteAccount`, the Cosmos `AccAddress` derived from the EVM address has no auth account, but the bank module still holds any non-EVM-denom coins under that address key. Those coins are permanently inaccessible: no account exists to authorize a spend, and no recovery path exists in the EVM module.

This is the direct analog to the `renounceVesting` report: that function handled ERC-20 tokens but not ETH; here `Commit()` handles the EVM denom but not non-EVM Cosmos bank tokens.

---

### Impact Explanation

Any non-EVM-native Cosmos bank tokens (IBC-transferred assets, CosmWasm bridge tokens, or any `sdk.Coin` with a denom other than `evmDenom`) held by a contract address at the time of SELFDESTRUCT are permanently orphaned. The bank module retains the balance entry under the deleted address, but:

- No auth account exists to authorize a spend.
- No EVM account exists to receive a transfer.
- No recovery function exists in the EVM module.

The tokens are effectively burned without proper accounting — they inflate the bank module's total supply tracking while being permanently inaccessible. This constitutes permanent loss of valid user Cosmos bank funds through EVM state transition, matching the High impact category: *"EVM state transition bug that permits valid user funds to be mis-accounted."*

---

### Likelihood Explanation

IBC tokens are routinely sent to EVM contract addresses on Cosmos EVM chains (e.g., for DeFi vaults, liquidity pools, or cross-chain bridges). Any such contract that executes SELFDESTRUCT — whether intentionally, via a bug, or via EIP-6780 (same-transaction create+destroy) — will permanently orphan its non-EVM-denom holdings. The SELFDESTRUCT opcode is reachable by any unprivileged EVM transaction. No privileged role or governance action is required.

---

### Recommendation

Replace the single-denom burn with a full drain of all bank balances held by the selfdestructed address before calling `DeleteAccount`. Use `bankKeeper.GetAllBalances(ctx, cosmosAddr)` to enumerate every denomination and burn or redirect each coin:

```go
allBalances := s.keeper.GetAllBalances(cacheCtx, cosmosAddr)
for _, coin := range allBalances {
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn selfdestruct balance for denom "+coin.Denom)
    }
}
```

This mirrors the existing EVM-denom drain but covers all denominations, ensuring no Cosmos bank funds are orphaned when a contract is selfdestructed.

---

### Proof of Concept

1. Deploy a factory contract via CREATE2 that, in a single transaction:
   a. Deploys a child contract at a deterministic address.
   b. Sends IBC tokens (e.g., `ibc/...` denom) to the child address via a Cosmos `MsgSend` or native action.
   c. Calls `SELFDESTRUCT` on the child (EIP-6780: deletion applies because child was created in this tx).

2. After the transaction commits, `StateDB.Commit()` runs:
   - EVM denom balance at child address → burned via `SubBalance(evmDenom)`. ✓
   - IBC token balance at child address → **not touched**. ✗
   - `DeleteAccount` removes the auth account for the child address.

3. Query `bankKeeper.GetAllBalances(ctx, childCosmosAddr)`:
   - Returns the IBC token balance (non-zero).
   - But `accountKeeper.GetAccount(ctx, childCosmosAddr)` returns `nil`.

4. The IBC tokens are permanently orphaned: no account owns them, no EVM address maps to them, and no recovery path exists. [3](#0-2) [2](#0-1)

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account
//
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
