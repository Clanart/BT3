### Title
MPC Prover Initialized with `EvmFinality::Latest` for Abstract Chain Enables Block-Reorg Double-Spend — (File: near/omni-prover/mpc-omni-prover/src/lib.rs)

---

### Summary

The `MpcOmniProver` contract is hardcoded at initialization to accept proofs for `ChainKind::Abs` (Abstract chain) only at `EvmFinality::Latest`. `Latest` is the chain tip — the block with zero finality guarantee and the highest natural reorg exposure. Because the NEAR bridge finalizes a transfer (mints or unlocks tokens) as soon as the MPC proof is accepted, a natural reorg that removes the source `initTransfer` transaction after NEAR has already settled the transfer produces an unbacked mint: the user recovers their tokens on the Abs chain while keeping the NEAR-side tokens.

---

### Finding Description

In `near/omni-prover/mpc-omni-prover/src/lib.rs`, the `init` constructor inserts a per-chain finality policy into the `finalities` map:

```rust
finalities.insert(ChainKind::Abs, MpcFinality::Evm(EvmFinality::Latest));
``` [1](#0-0) 

`verify_proof` then enforces that the caller-supplied proof's finality field exactly matches the stored policy via `request_matches_finality`:

```rust
require!(
    Self::request_matches_finality(&payload_v1.request, finality),
    ProverError::FinalityMismatch.as_ref()
);
``` [2](#0-1) 

The matching logic for EVM/Abstract requests compares the request's `finality` field directly to the stored `MpcFinality`:

```rust
(
    ForeignChainRpcRequest::Ethereum(args) | ForeignChainRpcRequest::Abstract(args),
    MpcFinality::Evm(finality),
) => args.finality == *finality,
``` [3](#0-2) 

Because the stored policy is `EvmFinality::Latest`, the MPC network is asked to verify the transaction at the chain tip. The MPC network's `verify_foreign_transaction` response is then trusted by the NEAR bridge to finalize the transfer:

```rust
let Ok(ProverResult::InitTransfer(init_transfer)) = Self::decode_prover_result(0) else { ... };
// ... builds transfer_message ...
self.process_fin_transfer_to_near(recipient, ..., transfer_message, ...)
``` [4](#0-3) 

Finalization inserts the `TransferId` into `finalised_transfers`, which prevents a second finalization of the same `(origin_chain, origin_nonce)` pair:

```rust
require!(
    self.finalised_transfers.insert(transfer_id),
    BridgeError::TransferAlreadyFinalised.as_ref()
);
``` [5](#0-4) 

This replay guard is correct but does not address the reorg scenario: the attacker only needs **one** successful finalization on NEAR. If the source-chain block is subsequently reorganized away, the attacker's lock transaction disappears from the canonical chain while the NEAR-side mint remains permanent.

The `set_finality` function that could correct the policy is marked `#[private]`, meaning it can only be invoked as a self-callback — it is not callable by any admin role after deployment without a dedicated upgrade path:

```rust
#[private]
pub fn set_finality(&mut self, chain_kind: ChainKind, finality: MpcFinality) {
    self.finalities.insert(chain_kind, finality);
}
``` [6](#0-5) 

---

### Impact Explanation

A user who initiates an `initTransfer` on the Abstract chain and whose transaction lands in the latest block can receive bridged tokens on NEAR (via a relayer submitting the `Latest`-finality MPC proof) and then have the source transaction removed by a natural reorg. The result is:

- Tokens returned to the user on the Abs chain (the lock transaction never existed in the canonical chain).
- Tokens already minted/unlocked on NEAR (the finalization is irreversible).
- Bridge collateralization is broken: NEAR-side supply is unbacked.

This matches the allowed impact: **High — cross-chain double-spend enabling unbacked supply**.

---

### Likelihood Explanation

Natural one- or two-block reorgs occur on every EVM chain, including newer or lower-hashrate chains. `EvmFinality::Latest` is the chain tip — the block most exposed to reorg. A relayer operating under the protocol's own finality policy (which permits `Latest`) will submit proofs immediately after transaction inclusion. No 51% attack is required; the attacker only needs a natural reorg to occur in the window between proof submission and the next block confirmation. On a chain with lower security than Ethereum mainnet, this window is wider and reorgs are more frequent.

---

### Recommendation

Change the initial finality for `ChainKind::Abs` from `EvmFinality::Latest` to at minimum `EvmFinality::Safe` (Ethereum's safe-head checkpoint, ~2 epochs) or `EvmFinality::Finalized`. Additionally, expose a privileged (DAO-gated) `set_finality` entry point so the policy can be corrected without a full contract redeployment.

```rust
// Before (vulnerable)
finalities.insert(ChainKind::Abs, MpcFinality::Evm(EvmFinality::Latest));

// After (recommended)
finalities.insert(ChainKind::Abs, MpcFinality::Evm(EvmFinality::Safe));
```

---

### Proof of Concept

1. Attacker calls `initTransfer` on the Abstract chain EVM contract, locking tokens. The transaction lands in block N (the latest block).
2. A trusted relayer observes the event and calls `fin_transfer` on the NEAR bridge with a `MpcVerifyProofArgs` whose embedded `ForeignChainRpcRequest::Abstract` carries `finality: EvmFinality::Latest`.
3. `MpcOmniProver::verify_proof` passes `request_matches_finality` (stored policy is `Latest`, request is `Latest` — match).
4. The MPC network's `verify_foreign_transaction` confirms the transaction exists at the latest block and returns a valid `VerifyForeignTransactionResponse`.
5. `verify_callback` validates `SHA-256(borsh(sign_payload)) == response.payload_hash`, parses the EVM log, and returns `ProverResult::InitTransfer`.
6. `fin_transfer_callback` calls `add_fin_transfer` (inserts nonce into `finalised_transfers`) and mints/unlocks tokens to the recipient on NEAR.
7. A natural reorg on the Abstract chain replaces block N with block N′, which does not contain the attacker's `initTransfer` transaction. The attacker's tokens are returned on the Abs chain.
8. The attacker now holds tokens on both the Abstract chain and NEAR. The NEAR-side supply is unbacked.
