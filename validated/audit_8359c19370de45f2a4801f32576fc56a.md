### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Enables Theft of Bridge Collateral — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer()` records the caller-supplied `amount` parameter directly in the cross-chain message without measuring the actual balance change. For fee-on-transfer ERC20 tokens, the bridge receives less than `amount` but the Wormhole/NEAR message carries the full `amount`, causing NEAR to mint more tokens than the bridge holds. When those tokens are bridged back, the bridge must release more than it received, draining other users' locked collateral.

---

### Finding Description

In `OmniBridge.initTransfer()`, the branch handling native (non-bridge, non-custom-minter) ERC20 tokens is: [1](#0-0) 

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
```

After this transfer, the function calls `initTransferExtension(...)` passing the original `amount` parameter unchanged: [2](#0-1) 

In `OmniBridgeWormhole.initTransferExtension()`, this `amount` is Borsh-encoded and published to Wormhole verbatim: [3](#0-2) 

The `InitTransfer` event also emits the original `amount`: [4](#0-3) 

There is **no balance-before / balance-after check** anywhere in the deposit path. For a fee-on-transfer ERC20 token, `safeTransferFrom(..., amount)` causes the bridge to receive `amount - transferFee`, but the cross-chain message carries `amount`. NEAR's `fin_transfer_callback` processes the proof and mints/unlocks the full `amount` on the destination chain: [5](#0-4) 

When the user bridges those tokens back, NEAR signs a `TransferMessagePayload` for the full `amount`, and the EVM bridge's `finTransfer` releases `amount` tokens to the recipient: [6](#0-5) 

The bridge only holds `amount - transferFee` for this deposit, so the deficit is covered by other users' locked tokens — a direct theft.

---

### Impact Explanation

**Critical — Direct theft of native assets locked in the bridge.**

Every deposit of a fee-on-transfer token creates an undercollateralization gap equal to the transfer fee. An attacker who repeatedly deposits and bridges back extracts value from the pool of other users' locked tokens. The bridge's invariant — that locked collateral equals outstanding bridged supply — is permanently broken for any registered fee-on-transfer token.

---

### Likelihood Explanation

**Medium.** The bridge does not restrict which ERC20 tokens can be registered. Fee-on-transfer tokens are a well-known ERC20 variant (e.g., PAXG, STA, tokens with configurable fee switches). Once any such token is registered via `addCustomToken` or the standard `deployToken` / `logMetadata` flow, the accounting gap is immediately exploitable by any unprivileged user calling `initTransfer`. No special role or leaked key is required after registration.

---

### Recommendation

Replace the direct `safeTransferFrom` with a balance-before / balance-after measurement and use the **actual received amount** in the cross-chain message:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint128 actualReceived = uint128(
    IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore
);
require(actualReceived == amount, "Fee-on-transfer tokens not supported");
// — or —
// use actualReceived instead of amount in initTransferExtension and the event
```

The simplest safe mitigation is to assert `actualReceived == amount`, rejecting fee-on-transfer tokens entirely. If such tokens must be supported, `actualReceived` must propagate through `initTransferExtension`, the Wormhole payload, and the emitted event.

---

### Proof of Concept

1. A fee-on-transfer ERC20 token `FeeToken` (1% fee) is registered in the bridge (admin step, not part of the exploit itself).
2. Attacker calls `initTransfer(FeeToken, 1000e18, 0, 0, nearRecipient, "")`.
3. `safeTransferFrom` deducts 1% fee; bridge receives **990e18** tokens.
4. `initTransferExtension` publishes `amount = 1000e18` to Wormhole.
5. NEAR processes the VAA and mints **1000e18** bridged `FeeToken` to `nearRecipient`.
6. Attacker bridges back 1000e18 bridged tokens to EVM.
7. NEAR MPC signs a `TransferMessagePayload` for `amount = 1000e18`.
8. Attacker calls `finTransfer` on the EVM bridge; bridge releases **1000e18** native tokens.
9. Bridge held only 990e18 from this deposit → **10e18 tokens stolen from other depositors**.

Repeating this at scale drains the bridge's entire collateral pool for that token.
