### Title
P2P Sync Stores Attacker-Supplied `block_hash` Without Validation — (`crates/apollo_p2p_sync/src/client/header.rs`)

### Summary
`HeaderStreamBuilder::parse_data_for_block` accepts a `SignedBlockHeader` from an untrusted p2p peer and performs only two checks: that the block number matches the expected value and that exactly one signature is present. It never verifies the `block_hash` field against the Poseidon-computed hash of the header fields, and never verifies the signature over `(block_hash, state_diff_commitment)`. The raw peer-supplied `block_hash` is then passed directly to `write_to_storage` → `append_header` → `update_hash_mapping`, which inserts it into both the `headers` table and the `block_hash_to_number` table with no further validation.

### Finding Description

**Step 1 — `parse_data_for_block` (the only gate):** [1](#0-0) 

The two guards are:
- `block_number == signed_block_header.block_header.block_header_without_hash.block_number` — checks ordering only.
- `signed_block_header.signatures.len() != ALLOWED_SIGNATURES_LENGTH` — checks count only; the signature bytes are never cryptographically verified.

There is no call to `verify_block_signature` or `calculate_block_hash` anywhere in the p2p sync client crate. The `block_hash` field of `BlockHeader` is never touched.

**Step 2 — `write_to_storage` passes the header verbatim:** [2](#0-1) 

**Step 3 — `append_header` stores the peer-supplied hash directly:** [3](#0-2) 

`block_hash: block_header.block_hash` — the attacker's value is copied verbatim into `StorageBlockHeader`.

**Step 4 — `update_hash_mapping` inserts the wrong hash into the reverse-lookup table:** [4](#0-3) 

`block_hash_to_number_table.insert(txn, &block_header.block_hash, &block_number)` — the attacker's `0xdeadbeef` is now the canonical key for block N.

**Confirmed missing guard — parent hash check is a TODO:** [5](#0-4) 

The comment `// TODO(shahak): Check that parent_hash is the same as the previous block's hash` confirms that even the chaining guard is absent.

### Impact Explanation

After the commit, the storage contains:
- `headers[N].block_hash = 0xdeadbeef` (wrong)
- `block_hash_to_number[0xdeadbeef] = N` (wrong reverse mapping)

Downstream consequences:
1. **RPC** — `get_block_header(N)` returns `block_hash = 0xdeadbeef`. Any client or tool that trusts this value (e.g., to construct a `parent_hash` for block N+1, or to look up a block by hash) receives a wrong authoritative value. This matches the allowed impact: *"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*
2. **Subsequent sync** — `get_block_number_by_hash(correct_hash)` returns `None`; `get_block_number_by_hash(0xdeadbeef)` returns `Some(N)`. Any logic that resolves a block by its canonical hash is silently broken.
3. **`StateSync::get_block_hash`** reads `header.block_hash` directly from storage and returns it to callers (batcher, consensus, proof manager). [6](#0-5) 

### Likelihood Explanation

Any node participating in the p2p network can act as a responding peer. The syncing node sends a `BlockHeadersRequest` and accepts the first well-formed response. No peer authentication, no stake, no operator privilege is required. The attacker only needs to:
1. Connect as a p2p peer.
2. Respond to a header query for block N with a `SignedBlockHeader` where `block_number = N` and `block_hash = <arbitrary>`, with a vector of exactly one (arbitrary) signature.

### Recommendation

In `parse_data_for_block`, after receiving the `SignedBlockHeader`:
1. **Recompute the block hash** from the header fields using `calculate_block_hash` and assert it equals `signed_block_header.block_header.block_hash`. Reject with `BadPeerError` on mismatch.
2. **Verify the block signature** using `verify_block_signature` against the known sequencer public key. Reject with `BadPeerError` on failure.
3. **Implement the parent hash check** noted in the existing TODO at line 102–103.
