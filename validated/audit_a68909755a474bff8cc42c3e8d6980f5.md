### Title
Missing Factory Pool Validation in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User Funds - (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

---

### Summary

`MetricOmmPoolLiquidityAdder` accepts an arbitrary caller-supplied `pool` address without verifying it against the factory. A malicious pool contract can exploit the callback settlement path to pull up to `maxAmountToken0` / `maxAmountToken1` from any user who has approved the adder.

---

### Finding Description

`MetricOmmSwapRouterBase._setNextCallbackContext` and `_updateCallbackContextforRecursiveOutput` both call `_requireFactoryPool(pool)` before storing the pool in transient context, ensuring only factory-registered pools can trigger the swap callback. [1](#0-0) [2](#0-1) 

`MetricOmmPoolLiquidityAdder` performs no equivalent check. Its own NatDoc acknowledges this explicitly:

> "This contract does not verify the pool against the factory; a malicious pool can request token pulls up to the caller-provided max caps during callback settlement." [3](#0-2) 

`_addLiquidity` stores the caller-supplied `pool` as the `expectedPool` in transient context, then calls `pool.addLiquidity(...)`: [4](#0-3) 

`metricOmmModifyLiquidityCallback` then validates only that `msg.sender == expectedPool` — which is trivially satisfied by the malicious pool that was just called: [5](#0-4) 

After passing that check, the callback reads token addresses from `msg.sender` (the malicious pool) and calls `pay(token0, payer, msg.sender, amount0Delta)` / `pay(token1, payer, msg.sender, amount1Delta)`: [6](#0-5) 

The `pay` function then executes `safeTransferFrom(payer, recipient, value)`, pulling real tokens from the user's wallet: [7](#0-6) 

---

### Impact Explanation

A user who has approved `MetricOmmPoolLiquidityAdder` and calls any `addLiquidityExactShares` or `addLiquidityWeighted` overload with a malicious pool address will have up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 transferred directly from their wallet to the malicious pool. This is a direct loss of user principal with no recovery path.

In the `addLiquidityWeighted` variant the attacker additionally controls the probe revert values (`need0`, `need1`), allowing them to manipulate the share-scaling step before the paying callback fires, but the cap is still `max0`/`max1`. [8](#0-7) 

---

### Likelihood Explanation

The trigger requires a user to call the adder with a malicious pool address (e.g., via a phishing UI, a compromised front-end, or a misleading integration). The user must also have pre-approved the adder. This is a realistic attack surface: the adder is a public, permissionless contract intended for broad use, and the router's analogous factory guard demonstrates the protocol is aware of this class of risk and chose to enforce it there but not here.

---

### Recommendation

Add a factory validation call at the top of `_addLiquidity` (or in each public entry point), mirroring the router pattern:

```solidity
function _addLiquidity(address pool, ...) internal returns (...) {
    if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
    _setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
    ...
}
```

This requires storing the factory address as an immutable in `MetricOmmPoolLiquidityAdder`, exactly as `MetricOmmSwapRouterBase` does. [9](#0-8)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import {IMetricOmmPoolActions} from "@metric-core/interfaces/IMetricOmmPool/IMetricOmmPoolActions.sol";
import {LiquidityDelta} from "@metric-core/types/PoolOperation.sol";
import {MetricOmmPoolLiquidityAdder} from "metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol";

interface IAdder {
    function metricOmmModifyLiquidityCallback(
        uint256 amount0Delta,
        uint256 amount1Delta,
        bytes calldata callbackData
    ) external;
}

contract MaliciousPool {
    address public token0;
    address public token1;
    IAdder adder;

    constructor(address _token0, address _token1, address _adder) {
        token0 = _token0;
        token1 = _token1;
        adder = IAdder(_adder);
    }

    // Called by LiquidityAdder._addLiquidity
    function addLiquidity(
        address, uint80, LiquidityDelta calldata, bytes calldata, bytes calldata
    ) external returns (uint256, uint256) {
        // Callback with KIND_PAY (1) and full max amounts
        adder.metricOmmModifyLiquidityCallback(
            MAX0, MAX1, abi.encode(uint8(1))
        );
        return (MAX0, MAX1);
    }
}

// Attack:
// 1. Deploy MaliciousPool(token0, token1, adderAddress)
// 2. Victim approves adder for token0 and token1
// 3. Attacker calls adder.addLiquidityExactShares(
//        maliciousPool, victim, 0, deltas, MAX0, MAX1, ""
//    )
// 4. adder stores maliciousPool as expectedPool, calls maliciousPool.addLiquidity
// 5. maliciousPool calls back metricOmmModifyLiquidityCallback(MAX0, MAX1, KIND_PAY)
// 6. msg.sender == expectedPool check passes
// 7. adder pulls MAX0 token0 and MAX1 token1 from victim → transferred to maliciousPool
```
