### Title
Missing Recipient String Validation in `initTransfer` Enables Permanent Token Lock — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol::initTransfer` accepts an arbitrary `string calldata recipient` with no format validation. Tokens are burned or locked before the event is emitted. The NEAR-side prover then attempts to parse the recipient string into an `OmniAddress` via `OmniAddress::from_str()`; if parsing fails, the proof submission panics and the transfer can never be finalized. No rescue or refund path exists in `OmniBridge.sol`.

---

### Finding Description

`initTransfer` performs no validation on `recipient`: [1](#0-0) 

Tokens are burned (bridge tokens) or locked (native ERC20/ETH) before the event is emitted: [2](#0-1) 

The NEAR-side prover then parses the recipient string from the emitted log:

```rust
recipient: event.data.recipient.parse().map_err(stringify)?
``` [3](#0-2) 

`OmniAddress::from_str` enforces strict format rules. An empty string defaults to `("eth", "")` and fails hex parsing. A string without a recognized `chain:` prefix returns `Err("Chain X is not supported")`. A `near:` prefix with an invalid account ID fails NEAR `AccountId` parsing: [4](#0-3) 

When `parse_evm_event` returns `Err`, `fin_transfer_callback` panics: [5](#0-4) 

`OmniBridge.sol` has no admin rescue, refund, or emergency withdrawal function for locked tokens. The contract is UUPS-upgradeable, but recovery requires a privileged upgrade — not a standard user-accessible path.

---

### Impact Explanation

- **Bridge tokens** (`isBridgeToken`): burned via `BridgeToken.burn` — permanently destroyed, no recovery.
- **Native ERC20 tokens**: locked in the contract with no retrieval path.
- **Native ETH**: locked in the contract with no retrieval path.

This matches: *Critical — Permanent freezing, irrecoverable lock, or unclaimable settlement of user funds.*

---

### Likelihood Explanation

Any unprivileged user can trigger this by passing `""`, `"alice.near"` (missing `near:` prefix), `"near:"` (empty account), `"unknown:address"`, or non-UTF-8 bytes as `recipient`. No special role or precondition is required beyond holding tokens and calling `initTransfer`.

**Mitigating factor**: The Solana `SECURITY.md` explicitly acknowledges the identical pattern as a known low-severity issue:

> *"No validation of `recipient` string in `InitTransferPayload` — An invalid recipient causes the transfer to fail on the NEAR side after tokens are locked/burned on Solana. Manual intervention would be needed."* [6](#0-5) 

However, `evm/SECURITY.md` does **not** document this issue for the EVM side: [7](#0-6) 

The NEAR-side `init_transfer` enforces `OmniAddress` typing at the Rust level, so this asymmetry is unique to EVM (and Solana). The "manual intervention" noted for Solana implies a contract upgrade — not a user-accessible recovery path.

---

### Recommendation

Add a recipient format check in `initTransfer` (and `initTransfer1155`) before burning/locking tokens. At minimum, require:
- `bytes(recipient).length > 0`
- The string contains exactly one `:` separator
- The chain prefix is one of the known supported values

Alternatively, mirror the NEAR-side approach: accept a typed struct rather than a raw string, so invalid formats are rejected at ABI decode time.

---

### Proof of Concept

```solidity

token.approve(address(bridge), 1000);

// 2. Call initTransfer with empty recipient — no revert

# Security Notes

## Design Decisions (Non-Issues)

These patterns have been reviewed and confirmed as intentional. Do not flag or "fix" them.

- **Fee-on-transfer tokens not supported**: `initTransfer` emits the requested `amount`, not the actual received balance. Fee-on-transfer and rebasing tokens are intentionally unsupported
- **`logMetadata` and `deployToken` are permissionless**: Anyone can call `logMetadata` for any ERC20, and anyone can submit a valid MPC signature to `deployToken`. This is by design — the bridge is fully permissionless
- **`ENearProxy.burn` uses empty NEAR recipient**: `eNear.transferToNear(amount, "")` is intentional — `transferToNear` is a legacy method used purely as a burn mechanism. The actual NEAR recipient is tracked in the OmniBridge `InitTransfer` event
- **`deployToken` signature has no chain ID**: Metadata signatures are intentionally chain-agnostic — one NEAR-side signature deploys the same token on all EVM chains

## Known Issues

Low-severity items acknowledged but not yet addressed:

- **`addCustomToken` can overwrite existing mappings** (H-01): Admin-only function. No existence check — calling with an already-mapped `nearTokenId` silently overwrites `nearToEthToken`. Accepted as operational risk
- **`pause(flags)` replaces all flags** (H-02): `_pause(flags)` does full replacement, not bitwise OR. Calling `pause(PAUSED_INIT_TRANSFER)` when `PAUSED_FIN_TRANSFER` is set will unpause finTransfer. Use `pauseAll()` for emergencies
- **`BridgeToken.initialize` stores metadata redundantly** (L-01): `__ERC20_init(name_, symbol_)` writes to parent storage that is never read (getters are overridden). Minor gas waste on init
- **`require` strings instead of custom errors** (L-02): Several locations use `require` with string messages instead of custom errors (`OmniBridge.sol:150,204,556`, `SelectivePausableUpgradable.sol:100,107`, `ENearProxy.sol:56,76,86`)
- **`OmniBridgeWormhole` has no `__gap`** (L-04): Three storage variables with no gap array. Safe as a leaf contract but would need a gap if inherited from
- **`PayloadType.ClaimNativeFee` defined but unused** (L-05): Enum value 2 is never referenced. Native fees are recovered via `finTransfer` with `tokenAddress=address(0)`
```
