### Title
`execute_deploy` Syscall Hardcodes `failure_flag=0`, Making Blocks With Failed Constructor Deployments Unprovable — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall implementation in the StarkNet OS unconditionally writes `failure_flag=0` into the syscall response header, regardless of whether the deployed contract's constructor actually succeeded or reverted. This is an acknowledged gap (marked with a `TODO` comment). If a block contains a transaction where a deploy syscall fails (constructor reverts), the OS re-execution will diverge from the sequencer's execution — the calling contract will observe `failure_flag=0` in the OS but `failure_flag=1` from the sequencer — causing irreconcilable state divergence and an unprovable block.

---

### Finding Description

In `execute_deploy` (`syscall_impls.cairo`, lines 527–555), after calling `deploy_contract`, the OS writes the syscall response header with a hardcoded `failure_flag=0`:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` call (line 528) can internally execute a constructor that reverts. The revert is handled by `execute_entry_point` → `handle_revert`, which rolls back state changes. However, `deploy_contract` only returns `(retdata_size, retdata)` — it does not surface an `is_reverted` flag to `execute_deploy`: [2](#0-1) 

Contrast this with `contract_call_helper`, which correctly propagates `is_reverted` into the response header: [3](#0-2) 

The OS processes transactions sequentially in `execute_transactions_inner`, and each transaction's syscalls are re-executed deterministically: [4](#0-3) 

When the OS re-executes a calling contract that issued a deploy syscall whose constructor reverted:
- The sequencer recorded `failure_flag=1` in the syscall response.
- The OS writes `failure_flag=0` unconditionally.
- The calling contract reads `failure_flag=0` in the OS re-execution and follows a different code path than it did under the sequencer.
- This produces different storage writes, different messages, and different state changes.
- The OS's state dict (`contract_state_changes`) diverges from the sequencer's committed state, making the proof unsatisfiable.

---

### Impact Explanation

**Impact: High — Network not being able to confirm new transactions (total network shutdown) / Critical — Permanent freezing of funds.**

A block containing a transaction with a failed deploy syscall cannot be proven by the OS. Since the StarkNet proving pipeline requires a valid OS proof to update L1 state, an unprovable block permanently stalls L1 state updates. Users cannot withdraw funds to L1 (permanent freezing of funds). New blocks piled on top of the stuck block also cannot be settled, constituting a total network shutdown from the L1 perspective.

---

### Likelihood Explanation

Any unprivileged user can:
1. Declare a Sierra class whose constructor always panics/reverts.
2. Deploy a calling contract that invokes the `deploy` syscall with that class hash and inspects `failure_flag` in the response (a normal, supported pattern).
3. Submit the transaction. The sequencer executes it: constructor reverts, `failure_flag=1` is recorded, the calling contract handles the failure gracefully, the transaction is included in a block.
4. The prover attempts to generate an OS proof for that block and fails due to the state divergence described above.

The attack requires no privileged access, no leaked keys, and no operator cooperation. It requires only the ability to declare a class and submit a transaction — both are permissionless on StarkNet.

---

### Recommendation

`execute_deploy` must propagate the constructor's revert status. `deploy_contract` should be updated to return an `is_reverted` flag (analogous to `execute_entry_point`), and `execute_deploy` should write `failure_flag=is_reverted` into the response header, matching the pattern already used in `contract_call_helper`:

```cairo
// Correct pattern (from contract_call_helper):
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

The `DeployResponse` struct and the calling contract ABI may also need updating to carry constructor retdata on failure, consistent with how `CallContractResponse` carries failure retdata.

---

### Proof of Concept

1. Declare class `RevertingConstructor` with constructor body: `panic_with_felt252('always fails')`.
2. Deploy a `Caller` contract with an `__execute__` that calls:
   ```
   let result = deploy_syscall(RevertingConstructor_class_hash, salt, [], false);
   // result.failure_flag == 1 from sequencer; result.failure_flag == 0 from OS
   assert result.failure_flag == 1;  // passes in sequencer, fails in OS re-execution
   storage_write(key=0, value=result.failure_flag);  // writes 1 in sequencer, 0 in OS
   ```
3. Submit an invoke transaction calling `Caller.__execute__`.
4. Sequencer includes the transaction (it succeeds at the transaction level; the deploy failure is handled).
5. Prover attempts OS proof: the `storage_write` in step 2 writes `0` in the OS but `1` in the sequencer's committed state. The `dict_update` for `contract_state_changes` receives a mismatched `prev_value`, causing an unsatisfiable constraint and proof failure. [5](#0-4)

// Writes a value to the current contract's storage.
func execute_storage_write{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    alloc_locals;
    let request = cast(syscall_ptr + RequestHeader.SIZE, StorageWriteRequest*);

    // Reduce gas.
    let success = reduce_syscall_gas_and_write_response_header(
        total_gas_cost=STORAGE_WRITE_GAS_COST, request_struct_size=StorageWriteRequest.SIZE
    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    local prev_value: felt;
    local state_entry: StateEntry*;
    %{ WriteSyscallResult %}

    // Update the contract's storage.
    static_assert StorageWriteRequest.SIZE == 3;
    assert request.reserved = 0;
    tempvar storage_ptr = state_entry.storage_ptr;
    tempvar storage_key = request.key;
    assert [storage_ptr] = DictAccess(
        key=storage_key, prev_value=prev_value, new_value=request.value
    );
    let storage_ptr = storage_ptr + DictAccess.SIZE;

    assert [revert_log] = RevertLogEntry(selector=storage_key, value=prev_value);
    let revert_log = &revert_log[1];

    // Update the state.
    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(
            new StateEntry(
                class_hash=state_entry.class_hash, storage_ptr=storage_ptr, nonce=state_entry.nonce
            ),
            felt,
        ),
    );

    return ();
