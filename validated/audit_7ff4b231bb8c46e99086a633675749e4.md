### Title
Reachable `panic!` in `compute_plot_id_v1()` via Parsed `ProofOfSpace` with Neither Pool Key Nor Contract Hash - (File: `crates/chia-protocol/src/proof_of_space.rs`)

### Summary

`ProofOfSpace::parse()` for version-0 proofs accepts a serialized object where both `pool_public_key` and `pool_contract_puzzle_hash` are `None`. When `compute_plot_id()` is subsequently called on such an object — as happens during block validation — `compute_plot_id_v1()` reaches an unconditional `panic!()`. A single crafted block sent over the network is sufficient to crash any node that attempts to validate it.

### Finding Description

`ProofOfSpace::parse()` reads `pool_public_key` as `Option<G1Element>` (prefix byte `0x00` → `None`) and derives `pool_contract_puzzle_hash` from a second prefix byte whose low bit controls presence. For version 0, no invariant is enforced that at least one of the two fields is `Some`: [1](#0-0) 

The version-1 branch does enforce this invariant: [2](#0-1) 

But the version-0 branch has no equivalent check, so a `ProofOfSpace` with `version=0`, `pool_public_key=None`, and `pool_contract_puzzle_hash=None` is accepted by the parser without error.

`compute_plot_id()` dispatches to `compute_plot_id_v1()` for version-0 objects: [3](#0-2) 

`compute_plot_id_v1()` contains an unconditional `panic!()` when both optional arguments are `None`: [4](#0-3) 

The same pattern exists in `compute_plot_group_id_v2()`, which is called by `compute_plot_id_v2()` for version-1 proofs: [5](#0-4) 

The Python binding `py_compute_plot_id()` directly delegates to `compute_plot_id()`: [6](#0-5) 

`update_digest()` for version-1 proofs also calls `compute_plot_id()` indirectly via `quality_string()`, but for version-0 proofs it does not — so the hashing path is not the trigger here. The trigger is any code path that calls `compute_plot_id()` directly, which the Chia Python consensus layer does during proof-of-space validation.

### Impact Explanation

A crafted block containing a version-0 `ProofOfSpace` with `pool_public_key=None` and `pool_contract_puzzle_hash=None` parses without error. When the receiving node's consensus code calls `pos.compute_plot_id()` to validate the proof, the process panics and crashes. This halts the node, preventing it from syncing with the network — directly analogous to the parachain crash described in the external report. The impact matches: **serialized network object triggers node crash / chain halt**.

### Likelihood Explanation

The crafted bytes are trivial to construct: set the `Option<G1Element>` prefix byte to `0x00` (None) and the pool-contract prefix byte to `0x00` (version=0, no contract). The parser accepts this without error. Any peer on the network can broadcast such a block. No special privileges, keys, or governance access are required.

### Recommendation

Add a validation check in `parse()` for version-0 `ProofOfSpace` that rejects objects where both `pool_public_key` and `pool_contract_puzzle_hash` are `None`, mirroring the existing check for version-1:

```rust
// In the version == 0 branch of parse():
if pool_public_key.is_none() && pool_contract_puzzle_hash.is_none() {
    return Err(Error::InvalidPoS);
}
```

Alternatively, replace the `panic!()` in `compute_plot_id_v1()` and `compute_plot_group_id_v2()` with a `Result`-returning signature so the error propagates gracefully instead of crashing the process.

### Proof of Concept

Construct a minimal serialized `ProofOfSpace`:

```
[challenge: 32 zero bytes]
[pool_public_key Option prefix: 0x00]          ← None
[pool_contract prefix byte: 0x00]              ← version=0, no contract
[plot_public_key: 48 bytes of valid G1 point]
[size: 0x20]                                   ← k=32
[proof length prefix + proof bytes]
```

Parse this with `ProofOfSpace::parse::<false>()` — it succeeds. Then call `.compute_plot_id()` (or the Python binding `.compute_plot_id()`). The process panics at: [7](#0-6)
