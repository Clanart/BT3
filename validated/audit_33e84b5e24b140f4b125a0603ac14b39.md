### Title
Unauthenticated `SetProof` on `RemoteProofManagerServer` Allows Pre-Storing Garbage Proofs, Bypassing Proof Verification for Client-Side Proving Transactions - (File: `crates/apollo_proof_manager/src/proof_manager.rs`)

---

### Summary

The `ProofManager` exposes a `SetProof` endpoint via an unauthenticated `RemoteComponentServer` that binds to `0.0.0.0`. An attacker who can reach the proof manager's network port can pre-store arbitrary (garbage) proof bytes for any predictable `proof_facts` hash. When a legitimate Invoke V3 transaction with those `proof_facts` subsequently arrives at the gateway, `run_proof_verification` unconditionally skips cryptographic verification because `contains_proof` returns `true`. The gateway accepts the transaction with an unverified proof, violating the invariant that every proof stored in the proof manager has been cryptographically verified by `verify_proof`.

---

### Finding Description

**Root cause — `ProofManager.set_proof` stores without verifying:**

`set_proof` accepts any proof bytes and stores them keyed by `proof_facts.hash()`, with no call to `verify_proof`:

```rust
pub async fn set_proof(&self, proof_facts: ProofFacts, proof: Proof) -> Result<(), FsProofStorageError> {
    if self.contains_proof(proof_facts.clone()).await? {
        return Ok(());          // ← already present: silently no-op, never overwrites
    }
    let facts_hash = proof_facts.hash();
    self.proof_storage.set_proof(facts_hash, proof.clone()).await?;
    self.cache.insert(facts_hash, proof);
    Ok(())
}
``` [1](#0-0) 

**Root cause — `run_proof_verification` skips verification when proof is already present:**

```rust
async fn run_proof_verification(...) -> Result<bool, TransactionConverterError> {
    let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;
    if contains_proof {

    // ... starknet_proof_verifier::verify_proof(proof_facts, proof) ...
}
``` [2](#0-1) 

The design assumption is: *"if a proof is already stored, it was previously verified."* This assumption is broken by the unauthenticated write path below.

**Exploit chain:**

1. Attacker computes a valid `proof_facts` value (all fields are predictable: `PROOF_VERSION`, `VIRTUAL_SNOS`, an allowed `program_hash`, `VIRTUAL_OS_OUTPUT_VERSION`, any sufficiently old `block_number`, its known `block_hash`, and the public `config_hash`).
2. Attacker sends a raw HTTP/2 `SetProof(proof_facts, garbage_bytes)` request directly to the proof manager's remote port. `set_proof` stores the garbage bytes without verification.
3. A legitimate user submits an Invoke V3 transaction carrying those same `proof_facts` and a cryptographically valid proof.
4. The gateway calls `run_proof_verification` → `contains_proof` returns `true` → verification is **skipped** → `Ok(false)` returned.
5. `store_proof_and_spawn_archiving` calls `set_proof` again → `contains_proof` is `true` → no-op; the garbage bytes remain stored. [8](#0-7) 
6. The transaction is accepted into the mempool with an unverified (garbage) proof stored under the correct `proof_facts` hash.
7. When the batcher propagates the block via consensus, `convert_internal_rpc_tx_to_rpc_tx` retrieves the garbage proof from the proof manager and embeds it in the outgoing `ConsensusTransaction`. [9](#0-8) 
8. Peer nodes that do not yet have the proof call `run_proof_verification` → `contains_proof` = `false` → `verify_proof(proof_facts, garbage_bytes)` → **fails** → transaction/block rejected by peers.

---

### Impact Explanation

**Impact: High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

The gateway accepts an Invoke V3 transaction whose proof bytes have never been cryptographically verified. The client-side proving invariant — *"every accepted transaction with non-empty `proof_facts` is backed by a valid proof"* — is broken. Downstream, peer nodes reject the transaction during consensus validation, causing the proposing node to include a transaction that the network will not finalize. An attacker can systematically pre-populate garbage proofs for all valid `proof_facts` combinations (the input space is small: one allowed program hash, a finite window of valid block numbers, and a fixed config hash), permanently poisoning the proof manager for those entries and ensuring that every future client-side-proving transaction for those blocks bypasses verification.

---

### Likelihood Explanation

**Likelihood: Medium.**

The attacker requires network-level reachability to the proof manager's TCP port. In a distributed deployment the port is bound to `0.0.0.0` with no authentication. Any co-tenant, misconfigured firewall rule, or insider with network access can exploit this. The `proof_facts` input space is small and fully predictable from public chain data, so pre-population of all valid entries is computationally trivial.

---

### Recommendation

1. **Restrict `SetProof` to internal callers only.** Remove `SetProof` (and `GetProof`) from the `RemoteProofManagerServer` interface, or add a caller-identity check (mTLS, shared secret, or IP allowlist) so that only the gateway and consensus components can write proofs.
2. **Verify before storing.** Add a call to `starknet_proof_verifier::verify_proof` inside `ProofManager::set_proof` before persisting, so that no unverified bytes can ever enter the store regardless of the call path.
3. **Do not skip verification based solely on `contains_proof`.** In `run_proof_verification`, the `contains_proof` shortcut should only be trusted if the write path is already authenticated and verified. If the write path cannot be fully trusted, remove the shortcut.

---

# Step 4: Gateway run_proof_verification → contains_proof=true → skip → Ok(false)

# Step 5: Batcher includes tx; consensus propagates block with garbage proof embedded.
