### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary felt value as a new class hash without verifying that the hash corresponds to a previously declared contract class. A contract can therefore replace its own class hash with an undeclared value. Once committed to state, every subsequent call to that contract causes the OS to look up a compiled class that does not exist, making the contract permanently non-callable and permanently freezing any funds it holds. The developers themselves acknowledge the missing check with a TODO comment at the exact vulnerable line.

---

### Finding Description

In `execute_replace_class` (lines 878–916 of `syscall_impls.cairo`), the OS reads the requested new class hash directly from the syscall request buffer and writes it into `contract_state_changes` with no validation:

```cairo
func execute_replace_class{
    ...
}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing check. [2](#0-1) 

When a subsequent block calls the affected contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [3](#0-2) 

If `class_hash` was never declared, `dict_read` returns the default value `0`. `find_element` then searches for a compiled class with hash `0`; if none exists, the assertion inside `find_element` fails and the proof is invalid. The sequencer is therefore forced to exclude every future transaction targeting that contract. The contract is permanently dead.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value and the block is finalized on L1:

- Every call to the contract causes the OS proof to be invalid.
- The sequencer cannot include any transaction that touches the contract.
- There is no on-chain recovery path: StarkNet has no admin override for contract state, and an arbitrary felt value cannot be declared as a Sierra class.
- All ERC-20 balances, LP positions, or other assets held by the contract are permanently inaccessible.

---

### Likelihood Explanation

The `replace_class` syscall is the standard StarkNet upgradeability primitive. Any contract that exposes an upgrade path (vault, DEX, lending protocol, multisig) is a potential victim if:

1. The upgrade logic contains a bug that allows an attacker to supply an arbitrary class hash (e.g., missing access control on the upgrade function, reentrancy into the upgrade path, or a logic error in the new-class validation).
2. A malicious contract author intentionally deploys a contract that calls `replace_class` with an undeclared hash after collecting user deposits.

Because the OS itself performs no check, the sequencer has no protocol-level protection to fall back on. The missing validation is the root cause; any contract-level mistake is sufficient to trigger it.

---

### Recommendation

Inside `execute_replace_class`, before writing the new class hash to `contract_state_changes`, verify that the requested class hash exists in `contract_class_changes` with a non-zero compiled class hash:

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the enforcement already present for `declare` transactions, where `assert_not_zero(compiled_class_hash)` is called before writing to `contract_class_changes`. [4](#0-3) 

---

### Proof of Concept

1. **Attacker deploys a contract** (`VictimVault`) that accepts user deposits and exposes an `upgrade(new_class_hash)` function protected only by a storage-based owner flag.

2. **Attacker deposits funds** into `VictimVault` alongside legitimate users, establishing TVL.

3. **Attacker calls `upgrade(0xdeadbeef)`** — an arbitrary felt that is not a declared Sierra class hash. Because `execute_replace_class` performs no declared-class check, the OS writes `class_hash = 0xdeadbeef` into `contract_state_changes` for `VictimVault`. The transaction succeeds and is included in the proven block.

4. **In the next block**, any transaction targeting `VictimVault` causes `execute_entry_point` to call `dict_read(contract_class_changes, 0xdeadbeef)`, which returns `0`. `find_element` then searches for compiled class hash `0`, fails its internal assertion, and the proof is invalid.

5. **The sequencer is forced to exclude all future calls** to `VictimVault`. All user funds are permanently frozen with no recovery path.
