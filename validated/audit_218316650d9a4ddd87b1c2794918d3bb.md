### Title
Unconstrained `is_reverted` Flag in `execute_l1_handler_transaction` Allows Silent Dropping of L1-to-L2 Messages, Permanently Freezing L1 Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

In `execute_l1_handler_transaction`, the `is_reverted` flag is populated entirely by a hint (`%{ IsReverted %}`) and is passed to `check_is_reverted`, which is an **empty no-op function**. When `is_reverted != FALSE`, the function returns immediately — skipping both the entry-point execution **and** the call to `consume_l1_to_l2_message`. Because `consume_l1_to_l2_message` is the only place that writes the L1-to-L2 message into `outputs.messages_to_l2`, a malicious prover can silently drop any L1 handler message from the OS output while still producing a valid proof. The L1 funds sent with that message are never marked as consumed and are permanently frozen.

---

### Finding Description

**Step 1 — `is_reverted` is set by an unconstrained hint.** [1](#0-0) 

```cairo
%{ StartTx %}
local is_reverted;
%{ IsReverted %}
// Skip the execution step for reverted transaction.
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
```

`is_reverted` is a `local` variable filled exclusively by the hint `%{ IsReverted %}`. There is no Cairo constraint that ties this value to any actual execution outcome.

**Step 2 — `check_is_reverted` is an empty no-op.** [2](#0-1) 

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
```

The function imposes zero constraints. Any felt value for `is_reverted` satisfies the Cairo proof system.

**Step 3 — Early return skips `consume_l1_to_l2_message`.**

When `is_reverted != FALSE`, the function returns at line 388–390 before reaching: [3](#0-2) 

```cairo
// Consume L1-to-L2 message.
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
let remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=tx_execution_context
);
```

`consume_l1_to_l2_message` is the **only** function that writes the L1-to-L2 message header into `outputs.messages_to_l2`: [4](#0-3) 

If this call is skipped, the message never appears in the OS output. The L1 verifier therefore never sees the message as consumed.

**Structural analogy to the external report:**

| External Report | StarkNet OS |
|---|---|
| `if (_strategyNumber == 1)` updates `uTokenProportions`; otherwise returns empty array | `if (is_reverted != FALSE)` returns early; otherwise calls `consume_l1_to_l2_message` |
| Empty array returned → token proportions reset downstream | Message not written to `outputs.messages_to_l2` → L1 funds never released |
| Condition is unchecked (any strategy ≥ 1 triggers the outer call) | `is_reverted` is unconstrained (hint + empty check) |

---

### Impact Explanation

**Critical. Permanent freezing of funds.**

An L1-to-L2 message carries ETH or ERC-20 tokens locked on L1. The L1 StarkNet core contract releases those funds only when the OS output proves the message was consumed. If `consume_l1_to_l2_message` is never called for a given message, the message is absent from the OS output, the L1 contract never marks it consumed, and the locked funds are permanently inaccessible. There is no on-chain mechanism that automatically re-queues or refunds a message that was silently omitted from the OS output.

---

### Likelihood Explanation

The prover controls hint values. Because `check_is_reverted` imposes no constraint, setting `is_reverted` to any non-zero value for an L1 handler is a valid proof witness. Any L1-to-L2 message sender is a potential victim: they send funds from L1, the sequencer/prover marks the corresponding L1 handler as "reverted" (cost: zero), and the message is silently dropped from the OS output. No privileged key or external dependency is required beyond the ability to produce a proof — which is the sequencer's normal role.

---

### Recommendation

1. **Constrain `is_reverted` for L1 handlers.** `check_is_reverted` must enforce that `is_reverted` is consistent with the actual execution result. For L1 handlers, a pre-execution "reverted" state is semantically undefined and should be disallowed entirely (assert `is_reverted = 0` before the branch, or remove the early-return path).

2. **Always consume the L1 message.** Even if the L1 handler execution fails, `consume_l1_to_l2_message` must be called so the message appears in the OS output and the L1 contract can mark it as processed. State changes from a failed execution should be rolled back via the revert log, but the message consumption record must remain.

---

### Proof of Concept

1. Alice sends an L1-to-L2 message from Ethereum, locking 10 ETH in the StarkNet core contract.
2. The sequencer includes the corresponding L1 handler transaction in a block.
3. The prover sets `is_reverted = 1` via the `%{ IsReverted %}` hint.
4. `check_is_reverted(1)` returns immediately — no constraint fires.
5. `execute_l1_handler_transaction` returns at line 388 without calling `consume_l1_to_l2_message`.
6. The OS output for this block contains no entry for Alice's message in `messages_to_l2`.
7. The proof is valid; the L1 verifier accepts it.
8. The L1 StarkNet core contract never receives proof that Alice's message was consumed.
9. Alice's 10 ETH remains locked in the L1 contract indefinitely — permanent freezing of funds.
