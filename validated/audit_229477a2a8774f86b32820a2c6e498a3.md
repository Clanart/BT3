### Title
`batch_insert` Bypasses Duplicate-Key and Duplicate-Hash Guards Present in `insert` — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::insert` enforces two integrity invariants before writing a leaf: no duplicate `KeyId` and no duplicate leaf `Hash`. `MerkleBlob::batch_insert` enforces those invariants only for the first two items it processes (via a delegating call to `insert`), then writes all remaining leaves directly to the blob with `insert_entry_to_blob`, skipping both checks entirely. An untrusted caller can therefore inject duplicate keys or duplicate hashes into the authenticated DataLayer Merkle tree, corrupting its root and invalidating all subsequent proofs of inclusion/exclusion.

---

### Finding Description

### Impact Explanation

The DataLayer Merkle tree's correctness guarantees — that every `KeyId` appears exactly once and that every leaf hash is unique — are the foundation of its proof-of-inclusion and proof-of-exclusion logic. [5](#0-4) 

Inserting a duplicate `KeyId` via `batch_insert` produces a tree with two leaves sharing the same key. Any subsequent `get_proof_of_inclusion` for that key will resolve to one of the two leaves non-deterministically (depending on traversal order), making the proof either wrong or unverifiable against the committed root. Inserting a duplicate leaf hash produces two structurally identical leaves, causing internal node hashes to be computed over repeated data, corrupting the Merkle root. Both outcomes match the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**. [6](#0-5) 

---

### Likelihood Explanation

The attack requires only a crafted list of `(KeyId, ValueId, Hash)` tuples where a duplicate appears at position ≥ 3 (after the first two items that are protected). The Python binding is the primary interface used by the Chia full node's DataLayer subsystem, and it accepts arbitrary caller-supplied lists with no pre-validation. No privileged role or key material is needed. [4](#0-3) 

---

### Recommendation

Add the same duplicate-key and duplicate-hash checks at the top of the `batch_insert` fast path (the `for ((key, value), hash) in keys_values_hashes` loop), mirroring the guards already present in `insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insertion logic
}
```

Alternatively, update `block_status_cache` incrementally as each leaf is staged, so that duplicates within the batch itself are also caught before any write occurs. [3](#0-2)

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(bytearray())

# Pre-populate with 2 leaves so batch_insert takes the fast path
# (leaf_count > 1 after these two inserts)
blob.insert(KeyId(0), ValueId(0), hashlib.sha256(b"h0").digest())
blob.insert(KeyId(1), ValueId(1), hashlib.sha256(b"h1").digest())

# Now batch_insert with a duplicate KeyId(0) at position >= 1 in the batch.
# The first two items are NOT re-checked because leaf_count is already > 1.
dup_key   = KeyId(0)          # already present
dup_value = ValueId(99)
dup_hash  = hashlib.sha256(b"hDUP").digest()

new_key   = KeyId(2)
new_value = ValueId(2)
new_hash  = hashlib.sha256(b"h2").digest()

# This should raise KeyAlreadyPresent but does not:
blob.batch_insert(
    [(new_key, new_value), (dup_key, dup_value)],
    [new_hash, dup_hash],
)

# Tree now contains two leaves with KeyId(0); root is corrupt.
blob.calculate_lazy_hashes()
proof = blob.get_proof_of_inclusion(dup_key)
# proof.valid() may return True for a structurally invalid tree
print("Duplicate key silently inserted:", proof.valid())
``` [7](#0-6)

    pub fn batch_insert(
        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {
        // OPT: perhaps go back to taking an iterator?
        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),
                };

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {
            // OPT: can we avoid this extra min height leaf traversal?
            let min_height_leaf = self.get_min_height_leaf()?;
            self.insert_subtree_at_key(min_height_leaf.key, indexes[0], Side::Left)?;
        }

        Ok(())
    }

    pub fn py_batch_insert(
        &mut self,
        keys_values: Vec<(KeyId, ValueId)>,
        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
            ))?;
        }

        self.batch_insert(zip(keys_values, hashes).collect())?;

        Ok(())
    }
