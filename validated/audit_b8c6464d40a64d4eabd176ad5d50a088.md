### Title
`set_state_header` does not verify that the embedded chunk's `shard_id` matches the caller-supplied `shard_id`, allowing a malicious peer to install a cross-shard state header and corrupt a syncing node's trie - (File: chain/chain/src/state_sync/adapter.rs)

---

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `shard_id` parameter and a `ShardStateSyncResponseHeader` that embeds a `ShardChunk`. It validates that the chunk is included in the block via a Merkle path proof, but it never checks that `chunk.shard_id() == shard_id`. A malicious peer can supply the legitimately-included chunk for shard N together with its valid Merkle proof while claiming the header is for shard M. The proof passes, and the header is committed to `DBCol::StateHeaders` under the key `StateHeaderKey(shard_id=M, sync_hash)`. Subsequent state-part downloads and `apply_state_part` calls then write shard N's trie data into shard M's storage, permanently corrupting the syncing node's state.

---

### Finding Description

`set_state_header` performs five checks before persisting the header:

1. **Chunk internal proofs** – `validate_chunk_proofs(&chunk, …)` verifies the chunk's own Merkle roots are self-consistent.
2. **Block inclusion** – `verify_path(*sync_prev_block_header.chunk_headers_root(), shard_state_header.chunk_proof(), &ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` verifies the chunk hash appears somewhere in the block's chunk-headers Merkle tree.
3. **Prev-chunk inclusion** – analogous `verify_path` for the previous chunk.
4. **Incoming-receipt proofs** – sequential block-hash chain and per-shard receipt Merkle proofs.
5. **State-root-node validity** – `validate_state_root_node(shard_state_header.state_root_node(), chunk_inner.prev_state_root())`.

None of these checks bind the chunk to the requested `shard_id`. The `chunk_headers_root` Merkle tree is indexed by shard position; `verify_path` only proves that the supplied chunk hash exists at *some* position in the tree. A chunk for shard N at position N carries a valid proof for position N, and that proof passes `verify_path` regardless of what `shard_id` the caller requested.

After all checks pass, the header is stored:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
```

The stored header now contains shard N's chunk (and shard N's `prev_state_root`) filed under shard M's key. Every downstream consumer of this header — `set_state_part`, `apply_state_part`, `set_state_finalize` — reads the state root from the stored chunk and applies it to the storage identified by the caller-supplied `shard_id` (M). The result is that shard N's trie data is written into shard M's `ShardUId` column, and shard M's trie is never populated. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

A syncing node that accepts a cross-shard header will:

1. Download and cryptographically validate state parts against shard N's `prev_state_root` (validation passes — the parts are genuine).
2. Call `apply_state_part(shard_id=M, state_root=N_root, …)`, which resolves `shard_uid` from M and writes shard N's trie nodes into M's column.
3. Call `set_state_finalize` for shard M, which re-executes the chunk (shard N's chunk) under shard M's `shard_uid`, further mixing state.

After finalization, shard M's committed state root is shard N's root. Every subsequent block the node processes will compute the wrong post-state root for shard M, causing the node to diverge from the canonical chain. The node cannot recover without a full re-sync. If the node is a validator or chunk producer for shard M, it will produce invalid chunks or fail endorsement, degrading network liveness.

**Severity: High** — permanent state corruption of a syncing node, requiring full re-sync; no key material or funds are directly at risk on the corrupted node itself, but validator liveness is affected.

---

### Likelihood Explanation

State sync headers are fetched from arbitrary peers over the P2P network. Any peer that has processed the relevant epoch's blocks can serve headers. A single malicious or compromised peer that is selected as the state-sync source for a given shard can execute this attack without any special privilege. The attack requires no cryptographic forgery: the chunk and its Merkle proof are taken verbatim from the canonical block. The only requirement is that the attacker controls a peer node and can respond to `StateHeaderRequest` messages.

Nodes joining the network or recovering from a lag are the primary targets. The attack is silent — the syncing node logs no error and believes sync succeeded.

---

### Recommendation

After extracting the chunk from the header, add an explicit shard-id binding check before any further validation:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: verify the chunk belongs to the requested shard
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Apply the same check to `prev_chunk_header` if present:

```rust
if let Some(ref prev) = prev_chunk_header {
    if prev.shard_id() != shard_id {
        byzantine_assert!(false);
        return Err(Error::Other(
            "set_shard_state failed: prev_chunk shard_id does not match requested shard_id".into(),
        ));
    }
}
```

These checks should be inserted immediately after the chunk is extracted from the header, before `validate_chunk_proofs`. [5](#0-4)

### Proof of Concept

```
Setup:
  - Canonical block B at height H contains chunks for shard 0 (C0) and shard 1 (C1).
  - Syncing node S requests the state header for shard 0 from peer P.
  - P is malicious.

Attack:
  1. P constructs a ShardStateSyncResponseHeader where:
       .chunk        = C1  (shard 1's chunk, legitimately in block B)
       .chunk_proof  = valid Merkle proof for C1's position in B's chunk_headers_root
       .state_root_node = valid state root node for C1.prev_state_root()
       (incoming receipt proofs are constructed for shard 1)
  2. P sends this header in response to S's StateHeaderRequest{shard_id=0, sync_hash=B}.
  3. S calls set_state_header(shard_id=0, sync_hash=B, header=<above>).
       - validate_chunk_proofs(C1) → passes (C1 is internally valid).
       - verify_path(B.chunk_headers_root, proof_for_C1, C1.hash) → passes.
       - validate_state_root_node(…, C1.prev_state_root()) → passes.
       - NO check: C1.shard_id() == 0  ← missing
       - Stored: StateHeaders[StateHeaderKey(shard_id=0, B)] = header_with_C1
  4. S downloads state parts for C1.prev_state_root() (shard 1's state).
       - validate_state_part(shard_id=0, state_root=C1.prev_state_root(), …) → passes.
       - Stored: StateParts[StatePartKey(B, shard_id=0, part_id)] = shard_1_part
  5. S calls apply_state_part(shard_id=0, state_root=C1.prev_state_root(), …).
       - shard_uid = uid_for(shard_id=0, epoch)
       - Trie nodes from shard 1's state are written into shard 0's column.
  6. S calls set_state_finalize(shard_id=0, B).
       - Executes C1 (shard 1's chunk) under shard 0's shard_uid.
  Result: shard 0's committed state root = C1.prev_state_root() (shard 1's root).
          Shard 0's trie contains shard 1's data. Node S is permanently corrupted.
```

    pub fn set_state_part(
        &self,
        shard_id: ShardId,
        sync_hash: CryptoHash,
        part_id: PartId,
        part: &StatePart,
    ) -> Result<(), Error> {
        let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
        let chunk = shard_state_header.take_chunk();
        let state_root = *chunk.take_header().take_inner().prev_state_root();
        if matches!(
            self.runtime_adapter.validate_state_part(shard_id, &state_root, part_id, part),
            StatePartValidationResult::Invalid
        ) {
            byzantine_assert!(false);
            return Err(Error::Other(format!(
                "set_state_part failed: validate_state_part failed. state_root={:?}",
                state_root
            )));
        }
        // Saving the part data.
        let mut store_update = self.chain_store.store().store_update();
        let key = borsh::to_vec(&StatePartKey(sync_hash, shard_id, part_id.idx)).unwrap();
        let bytes = part.to_bytes();
        store_update.set(DBCol::StateParts, &key, &bytes);
        store_update.commit();
        Ok(())

    fn apply_state_part(
        &self,
        shard_id: ShardId,
        state_root: &StateRoot,
        part_id: PartId,
        part: &StatePart,
        epoch_id: &EpochId,
    ) -> Result<(), Error> {
        let _timer = metrics::STATE_SYNC_APPLY_PART_DELAY
            .with_label_values(&[&shard_id.to_string()])
            .start_timer();

        let part = part
            .to_partial_state()
            .expect("Part was already validated earlier, so could never fail here");
        let ApplyStatePartResult { trie_changes, flat_state_delta, contract_codes } =
            Trie::apply_state_part(state_root, part_id, part);
        let tries = self.get_tries();
        let shard_uid = self.get_shard_uid_from_epoch_id(shard_id, epoch_id)?;
        let mut store_update = tries.store_update();
        tries.apply_all(&trie_changes, shard_uid, &mut store_update);
        tracing::debug!(target: "chain", %shard_id, values_count = %flat_state_delta.len(), "inserting values to flat storage");
        // TODO: `apply_to_flat_state` inserts values with random writes, which can be time consuming.
        //       Optimize taking into account that flat state values always correspond to a consecutive range of keys.
        flat_state_delta.apply_to_flat_state(&mut store_update.flat_store_update(), shard_uid);
        self.precompile_contracts(epoch_id, contract_codes)?;
        store_update.commit();
        Ok(())
