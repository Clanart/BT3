### Title
Missing Zero-Address Validation for EVM Recipient in `init_transfer` / `finTransfer` Allows Permanent Loss of Bridged Native ETH — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `near/omni-bridge/src/lib.rs`)

---

### Summary

The NEAR-side `init_transfer` function does not validate that an EVM recipient address is non-zero. A user can initiate a cross-chain transfer specifying `OmniAddress::Eth(H160::ZERO)` as the recipient. The MPC signer will sign this payload, and the EVM-side `finTransfer` will execute the transfer to `address(0)`, permanently destroying the bridged native ETH with no refund path.

---

### Finding Description

**Root cause 1 — NEAR `init_transfer` (near/omni-bridge/src/lib.rs)**

The only recipient validation performed is a chain-kind check:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
``` [1](#0-0) 

`OmniAddress::Eth(H160::ZERO)` passes this check because its chain kind is `Eth`, not `Near`. No check is made that the inner EVM address is non-zero. The transfer message is stored and proceeds to MPC signing.

**Root cause 2 — EVM `finTransfer` (evm/src/omni-bridge/contracts/OmniBridge.sol)**

When `payload.tokenAddress == address(0)` (native ETH path), the contract sends ETH directly to `payload.recipient` with no zero-address guard:

```solidity
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [2](#0-1) 

A low-level `.call` to `address(0)` succeeds in EVM (it does not revert), so `success == true`, the ETH is permanently destroyed, and no error is raised.

The `sign_transfer` function on NEAR also performs no zero-address check before requesting the MPC signature: [3](#0-2) 

---

### Impact Explanation

**Permanent, irrecoverable loss of bridged native ETH.**

- Tokens on NEAR are locked/burned at `init_transfer` time.
- The MPC signs the payload containing `address(0)` as recipient.
- On EVM, `finTransfer` marks the destination nonce as used and sends ETH to `address(0)`.
- ETH sent to `address(0)` via `.call` is permanently destroyed.
- The NEAR-side transfer message is removed after finalization; no refund mechanism exists.
- The nonce is consumed, so the transfer cannot be retried.

This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows.**

---

### Likelihood Explanation

Any unprivileged bridge user can trigger this by calling `ft_transfer_call` (or the equivalent entry point) on NEAR with `recipient: OmniAddress::Eth("0x0000000000000000000000000000000000000000")`. No special role or key is required. The scenario is analogous to a user accidentally pasting a zero address — a realistic user-error scenario that the protocol provides no protection against, exactly as in the original FETH report.

---

### Recommendation

1. **In `near/omni-bridge/src/lib.rs` `init_transfer`**: After the chain-kind check, add a zero-address guard for EVM-family recipients:

```rust
if let OmniAddress::Eth(addr) | OmniAddress::Arb(addr) | OmniAddress::Base(addr) | ... = &init_transfer_msg.recipient {
    require!(!addr.is_zero(), BridgeError::InvalidRecipientAddress.as_ref());
}
```

2. **In `evm/src/omni-bridge/contracts/OmniBridge.sol` `finTransfer`**: Add an explicit guard before any asset transfer:

```solidity
if (payload.recipient == address(0)) revert InvalidRecipient();
```

The NEAR-side fix is the more important one because it prevents the invalid transfer from ever being committed to the chain.

---

### Proof of Concept

1. User on NEAR calls `ft_transfer_call` on a wNEAR-equivalent token with:
   ```json
   { "msg": { "InitTransfer": { "recipient": "Eth:0x0000000000000000000000000000000000000000", "fee": "0", "native_token_fee": "0" } } }
   ```
2. `init_transfer` passes the chain-kind check (`Eth != Near`) and stores the transfer message with `recipient = H160::ZERO`. [4](#0-3) 
3. A trusted relayer calls `sign_transfer`; the MPC signs the payload containing `address(0)` as recipient.
4. Relayer submits `finTransfer` on EVM with `payload.recipient = address(0)` and `payload.tokenAddress = address(0)` (native ETH).
5. `completedTransfers[nonce] = true` is set; then `address(0).call{value: amount}("")` executes and returns `success = true`. [5](#0-4) 
6. ETH is permanently destroyed. The NEAR-side transfer record is removed. No refund is possible.
