### Title
`MerkleBlob::batch_insert` Bypasses Duplicate-Key Guard, Enabling Inconsistent Tree State and Forged DataLayer Proofs — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces when the tree already contains two or more leaves. An attacker who can supply a batch containing repeated `KeyId` values causes multiple leaf nodes with the same key to be written into the blob. The resulting Merkle root is computed over all leaves (including the phantom duplicates), while the in-memory `block_status_cache` only tracks one of them. This produces a root hash that no honest set of key-value pairs can reproduce, corrupts proofs of inclusion, and can be used to prove inclusion or exclusion of states that do not exist.

---

### Finding Description

`MerkleBlob::insert` enforces two invariants before writing a leaf:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-373
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `insert` (with its guards) only for the first two items when the tree has ≤ 1 existing leaf. For every subsequent item — and for **all** items when the tree already has ≥ 2 leaves — it writes directly to the blob via `insert_entry_to_blob`, bypassing both guards entirely:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block {
        metadata: NodeMetadata { node_type: NodeType::Leaf, dirty: false },
        node: Node::Leaf(LeafNode { parent: Parent(None), hash, key, value }),
    };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

The guard that routes the first two items through `insert` is:

```rust
// lines 578-585
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { return Ok(()); };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;
    }
}
``` [3](#0-2) 

Once the tree has two or more leaves (the normal operating state), the entire batch bypasses the guard. A batch containing `(key_A, value_1, hash_1)` and `(key_A, value_2, hash_2)` inserts two leaf nodes for `key_A`. The tree-building loop then pairs them into internal nodes and computes a root hash that includes both phantom leaves. The `block_status_cache` (which drives `get_proof_of_inclusion`) only records one of the two indexes for `key_A`, so the proof it generates traverses a subtree that does not match the actual root.

The analogous inconsistency in the Linea bridge was: two tokens with the same address could both acquire `NATIVE_STATUS` because neither side could observe the other's state during the bridging call. Here, two leaves with the same `KeyId` can both be committed to the blob because `batch_insert` does not observe the cache state for items beyond the first two.

---

### Impact Explanation

- **Corrupted root hash.** The Merkle root stored after `batch_insert` encodes phantom leaves. Any external party that recomputes the root from the declared key-value set will obtain a different hash, breaking cross-node consistency.
- **Forged inclusion proofs.** `get_proof_of_inclusion` returns a proof anchored to the cached leaf index. Because the root was computed over additional phantom leaves, the proof's `combined_hash` chain does not reconstruct the actual root, causing `valid()` to return `false` for a legitimately present key — or, depending on tree layout, allowing a proof for a phantom leaf to verify against the corrupted root.
- **Forged exclusion.** A key that was inserted twice may appear absent to a verifier who recomputes the root from the canonical key-value set, because the canonical root differs from the stored root. [4](#0-3) 

---

### Likelihood Explanation

The Python binding exposes `batch_insert` directly:

```rust
// wheel/src/api.rs (DataLayer Python surface)
// MerkleBlob::py_batch_insert calls self.batch_insert(...)
``` [5](#0-4) 

The DataLayer sync protocol receives batches of key-value-hash triples from remote peers. A malicious peer that sends a batch containing a repeated `KeyId` (with distinct `ValueId`/`Hash`) triggers the unchecked path as soon as the local tree has ≥ 2 leaves — which is the normal state after any prior sync. No cryptographic capability is required; the attacker only needs to craft a well-formed batch with a duplicated key.

---

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the bulk path of `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing blob write ...
}
```

Alternatively, pre-deduplicate the input vector and assert uniqueness before entering the loop. The `check_integrity` method should also be called (or its invariants enforced inline) after every `batch_insert` in production paths. [6](#0-5)

### Proof of Concept

```rust
use chia_datalayer::{MerkleBlob, KeyId, ValueId, Hash};
use chia_protocol::Bytes32;

let mut blob = MerkleBlob::new(vec![]).unwrap();

// Seed the tree with 2 legitimate leaves so leaf_count > 1
// (bypassing the guarded path for all subsequent batch items)
blob.insert(KeyId(1), ValueId(1), &Hash(Bytes32::new([1u8;32])),
            chia_datalayer::InsertLocation::Auto{}).unwrap();
blob.insert(KeyId(2), ValueId(2), &Hash(Bytes32::new([2u8;32])),
            chia_datalayer::InsertLocation::Auto{}).unwrap();

// Batch with duplicate KeyId(3) — both entries bypass the guard
let batch = vec![
    ((KeyId(3), ValueId(10)), Hash(Bytes32::new([3u8;32]))),
    ((KeyId(3), ValueId(20)), Hash(Bytes32::new([4u8;32]))),  // same key, different hash
];
blob.batch_insert(batch).unwrap();  // succeeds — no error raised

blob.calculate_lazy_hashes().unwrap();

// Root hash now encodes 4 leaves (two for KeyId(3)).
// An honest recomputation from {1,2,3} yields a different root.
// get_proof_of_inclusion(KeyId(3)) returns a proof that may not
// verify against the corrupted root.
let proof = blob.get_proof_of_inclusion(KeyId(3)).unwrap();
// proof.valid() behaviour is now undefined / incorrect
