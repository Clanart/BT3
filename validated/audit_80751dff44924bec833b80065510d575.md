### Title
Unchecked Felt Arithmetic Overflow in `compute_max_possible_fee` Allows Fee-Free Transaction Execution — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

### Summary
`compute_max_possible_fee` computes the maximum chargeable fee using raw felt arithmetic (mod P ≈ 2²⁵¹) with no range checks on `ResourceBounds` fields. A user can craft a V3 transaction whose resource-bound values sum to exactly 0 (mod P), causing `charge_fee` to return early and execute the transaction with zero fee charged.

### Finding Description
`compute_max_possible_fee` sums three products of user-controlled felt values:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
```

All arithmetic is modular over the Cairo field prime P. No `assert_nn_le` or `assert_le` range checks are applied to `max_amount`, `max_price_per_unit`, or `tip` before or after the computation. The result is a felt that can wrap to any value in `[0, P-1]`, including 0.

`charge_fee` then immediately gates all fee logic on this result:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
```

If `max_fee` evaluates to 0 due to field overflow, the function returns without executing the ERC-20 `transfer` call to the sequencer, so no fee token is debited from the user's account.

**Concrete overflow example** (values are felt elements):

| Field | Value |
|---|---|
| `l1_gas_bounds.max_amount` | 1 |
| `l1_gas_bounds.max_price_per_unit` | P − 1 |
| `l2_gas_bounds.max_amount` | 1 |
| `l2_gas_bounds.max_price_per_unit` | 1 |
| `tip` | 0 |
| `l1_data_gas_bounds.max_amount` | 0 |

Sum = (P − 1) + 1 + 0 = P ≡ 0 (mod P). The OS skips fee charging entirely.

The OS does not enforce that `max_amount` or `max_price_per_unit` fit within u64/u128 bounds; they are loaded from hints as raw felts and passed directly into arithmetic.

### Impact Explanation
**Critical — Direct loss of funds.**

The fee token (STRK or ETH at `fee_token_address`) is the on-chain asset transferred from the user's account to the sequencer for every executed transaction. When `max_fee` overflows to 0, the ERC-20 `transfer` call is never made. The sequencer expends real computational resources (and pays L1 data-availability costs) but receives zero fee-token compensation. Because the OS proof is what L1 verifies, a proof that omits fee charging for a non-trivial transaction is accepted as valid by the L1 verifier, making the loss permanent and protocol-level rather than merely off-chain.

### Likelihood Explanation
A V3 transaction sender controls all three `ResourceBounds` entries and `tip` as part of the signed transaction payload. The OS loads these values from hints without any range-check constraints. Finding a combination that sums to 0 (mod P) is straightforward discrete arithmetic (one example is given above). The only external gate is the sequencer's gateway, which may apply u64/u128 type bounds; however, the OS itself provides no such enforcement, so any path that delivers a raw transaction to the OS (including future protocol changes or direct sequencer operation) exposes this path.

### Recommendation
Apply `assert_nn_le` range checks to `max_amount` (≤ 2⁶⁴ − 1) and `max_price_per_unit` (≤ 2¹²⁸ − 1) for each `ResourceBounds` entry before performing the multiplication in `compute_max_possible_fee`. With those bounds, the maximum possible sum is 3 × 2⁶⁴ × 2¹²⁸ = 3 × 2¹⁹² ≪ P, making overflow impossible. Alternatively, compute the fee as a `Uint256` and compare against a `Uint256` max-fee, eliminating modular truncation entirely.

### Proof of Concept

1. Construct a V3 `INVOKE_FUNCTION` transaction with:
   - `resource_bounds[L1_GAS] = {max_amount: 1, max_price_per_unit: P-1}`
   - `resource_bounds[L2_GAS] = {max_amount: 1, max_price_per_unit: 1}`
   - `resource_bounds[L1_DATA] = {max_amount: 0, max_price_per_unit: 0}`
   - `tip = 0`
2. Sign and submit the transaction.
3. The sequencer includes it in a block and runs the OS.
4. `compute_max_possible_fee` returns `(P-1) + 1 + 0 = P ≡ 0`.
5. `charge_fee` hits `if (max_fee == 0) { return (); }` and exits immediately.
6. No ERC-20 transfer is issued; the sequencer's balance is unchanged.
7. The OS proof is generated and accepted by L1 — the fee bypass is finalized on-chain.

```text
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```

```text
func charge_fee{
    range_check_ptr,
```
