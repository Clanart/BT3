### Title
Missing Chain ID in `deployToken` Signed Message Enables Cross-Chain Replay — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `deployToken` signed message on every chain (EVM, Solana, StarkNet) is hashed over an identical borsh-encoded payload that contains **no chain identifier**. A single NEAR MPC signature for a `MetadataPayload` is therefore cryptographically valid on every chain simultaneously. An attacker who observes a legitimate `deployToken` transaction on one chain can replay the same signature on every other chain, permanently blocking the legitimate deployment of that token on those chains and freezing any in-flight user funds that depend on it.

---

### Finding Description

**Root cause — identical signed payload across all chains:**

On EVM (`OmniBridge.sol`), the message hashed for `deployToken` is:

```
keccak256( PayloadType::Metadata(1) || borsh(token) || borsh(name) || borsh(symbol) || decimals )
``` [1](#0-0) 

On Solana (`deploy_token.rs`), `serialize_for_near` produces:

```
IncomingMessageType::Metadata(1) || borsh(token) || borsh(name) || borsh(symbol) || decimals
``` [2](#0-1) 

On StarkNet (`bridge_types.cairo`), `MetadataPayload::to_borsh` produces:

```
PayloadType::Metadata(1) || encode_byte_array(token) || encode_byte_array(name) || encode_byte_array(symbol) || decimals
``` [3](#0-2) 

All three encodings are byte-for-byte identical (borsh `u32_le(len) + bytes` for strings). All three verification paths use raw `keccak256` of that encoding followed by raw secp256k1 ECDSA recovery against the same `nearBridgeDerivedAddress` / `derived_near_bridge_address` key, which is a single MPC-derived address shared across all chains. [4](#0-3) [5](#0-4) 

**Contrast with `finTransfer`:** The `finTransfer` message hash on EVM explicitly embeds `omniBridgeChainId` twice (for token chain and recipient chain), preventing cross-chain replay for transfers. `deployToken` has no equivalent binding. [6](#0-5) 

---

### Impact Explanation

When a relayer submits a legitimate `deployToken` on chain A (e.g., Ethereum), the transaction is public. An attacker extracts the `(signatureData, metadata)` pair and submits the identical call to chain B (e.g., Arbitrum, Base, Solana, StarkNet) before the legitimate relayer does.

On chain B:
- A new proxy token contract is deployed at an attacker-triggered address.
- `nearToEthToken[metadata.token]` is set to that address.
- `isBridgeToken[proxy]` is set to `true`. [7](#0-6) 

When the legitimate relayer later calls `deployToken` on chain B, the guard `!isBridgeToken[nearToEthToken[metadata.token]]` reverts with `ERR_TOKEN_EXIST`. The legitimate deployment is permanently blocked.

Because NEAR's `token_id_to_address` mapping for chain B is never populated (NEAR did not initiate the deployment), `sign_transfer` cannot resolve the token address for chain B and will panic. Any user funds locked on NEAR for a transfer destined for chain B with this token are permanently irrecoverable without privileged admin intervention. [8](#0-7) 

---

### Likelihood Explanation

- All `deployToken` transactions are public on-chain; no privileged access is needed.
- The attacker only needs to submit the same calldata to a different chain's RPC endpoint.
- The attack is profitable to execute preemptively on any chain where a popular token has not yet been deployed, permanently capturing that slot.
- Multiple EVM chains (Eth, Arb, Base, BNB, Pol, HyperEvm, Abs) plus Solana and StarkNet are all vulnerable simultaneously from a single observed signature.

---

### Recommendation

Bind the `deployToken` signed payload to the destination chain by including the chain identifier in the hashed message, mirroring the pattern already used in `finTransfer`:

**EVM:**
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // ADD THIS
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent change to `serialize_for_near` in `deploy_token.rs` (Solana) and `MetadataPayload::to_borsh` in `bridge_types.cairo` (StarkNet), and update the NEAR MPC signing logic to include the destination chain ID in the `MetadataPayload` before hashing.

---

### Proof of Concept

1. Deploy `OmniBridge` on Ethereum (chain ID `0x01`) and Arbitrum (chain ID `0x03`) with the same `nearBridgeDerivedAddress`.
2. Observe a legitimate `deployToken(sig, {token:"wrap.near", name:"Wrapped NEAR", symbol:"wNEAR", decimals:24})` transaction on Ethereum.
3. Submit the identical call to Arbitrum's `OmniBridge.deployToken` with the same `sig` and `metadata`.
4. The call succeeds: `nearToEthToken["wrap.near"]` is set on Arbitrum to an attacker-triggered proxy address.
5. The legitimate relayer's subsequent `deployToken` call on Arbitrum reverts with `ERR_TOKEN_EXIST`.
6. NEAR's `token_id_to_address` for `(Arb, "wrap.near")` is never set; any user who locked `wrap.near` on NEAR targeting Arbitrum cannot complete their transfer.
