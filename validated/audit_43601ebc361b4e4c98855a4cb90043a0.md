### Title
Missing State-Diff Commitment Verification in P2P Sync Allows Malicious Peer to Inject Wrong State Diff - (File: `crates/apollo_p2p_sync/src/client/state_diff.rs`)

---

### Summary

The P2P sync client assembles a block's `ThinStateDiff` from peer-supplied chunks and validates only that the assembled length equals `header.state_diff_length`. It never verifies the Poseidon hash of the assembled diff against `header.state_diff_commitment`. A malicious peer can therefore supply state diff chunks whose total length is correct but whose content hashes to a different value, causing the syncing node to permanently store a wrong state diff. This wrong diff propagates into the Patricia trie committer, the global state root, RPC responses, and proof inputs.

---

### Finding Description

**The ERC4626 analog.** The original report describes a two-sided invariant (`totalAssets` / `totalShares`) where one side can be inflated without touching the other. The sequencer has an identical structural flaw in its P2P state-diff ingestion path: the block header carries two independent representations of the state diff ŌĆö `state_diff_length` (a scalar count) and `state_diff_commitment` (a Poseidon hash) ŌĆö but the sync client validates only the scalar count and ignores the hash.

**Root cause.** `StateDiffStreamBuilder::parse_data_for_block` in `crates/apollo_p2p_sync/src/client/state_diff.rs` reads `target_state_diff_len` from the stored block header and loops until the accumulated chunk length equals that target:

```
while current_state_diff_len < target_state_diff_len { ŌĆ” }
if current_state_diff_len != target_state_diff_len { return Err(WrongStateDiffLength) }
validate_deprecated_declared_classes_non_conflicting(&result)?;
Ok(Some((result, block_number)))
``` [1](#0-0) 

After the loop the function returns the assembled `ThinStateDiff` directly. There is no call to `calculate_state_diff_hash` and no comparison against `header.state_diff_commitment`. The only semantic check is `validate_deprecated_declared_classes_non_conflicting`, which tests for duplicate deprecated class hashes, not content integrity. [2](#0-1) 

The assembled diff is then written unconditionally to MDBX storage:

```rust
storage_writer.begin_rw_txn()?.append_state_diff(self.1, self.0)?.commit()?;
``` [3](#0-2) 

**What the header actually contains.** `BlockHeader` stores both fields independently:

```rust
pub state_diff_commitment: Option<StateDiffCommitment>,
pub state_diff_length: Option<usize>,
``` [4](#0-3) 

`state_diff_commitment` is the Poseidon hash `calculate_state_diff_hash(&thin_state_diff)` and is a direct input to the block hash via `calculate_block_hash`. `state_diff_length` is the scalar `thin_state_diff.len()` and is packed into `concatenated_counts`, also a block-hash input. Both are authenticated by the block signature, but only `state_diff_length` is checked during state-diff ingestion. [5](#0-4) 

**How `state_diff_length` and `state_diff_commitment` can diverge.** `ThinStateDiff::len()` counts entries:

```rust
result += self.deployed_contracts.len();
result += self.class_hash_to_compiled_class_hash.len();
result += self.deprecated_declared_classes.len();
result += self.nonces.len();
for (_, storage_diffs) in &self.storage_diffs { result += storage_diffs.len(); }
``` [6](#0-5) 

A malicious peer can craft a `StateDiffChunk` sequence whose total `len()` equals the authentic `state_diff_length` but whose Poseidon hash differs from `state_diff_commitment` ŌĆö for example by swapping storage values, substituting class hashes, or reordering nonces across contracts while keeping per-contract entry counts identical. The length check passes; the hash check is never performed.

**P2P server fallback amplifies the surface.** The P2P sync server already patches missing `state_diff_length` from the stored state diff:

```rust
if header.state_diff_length.is_none() {
    header.state_diff_length = Some(
        txn.get_state_diff(block_number)?.ok_or(ŌĆ”)?.len(),
    );
}
``` [7](#0-6) 

This means old blocks (synced via central sync before `state_diff_length` was populated) have their length derived from the stored diff at serve time, making the length field even less trustworthy as a content authenticator.

---

### Impact Explanation

Once the wrong `ThinStateDiff` is committed to MDBX, the `ApolloCommitter` feeds it into `BlockCommitter::commit_block`, which updates the Patricia trie and returns a `global_root` that does not match the authentic post-block state root. [8](#0-7) 

This wrong global root is stored as `ForestMetadataType::StateRoot` and propagates to:
- **Block hash calculation**: `calculate_block_hash` chains `state_root` directly; a wrong root produces a wrong block hash for all subsequent blocks.
- **RPC `starknet_getStorageAt` / `starknet_getStateUpdate`**: returns wrong storage values and wrong state diff to callers.
- **Proof inputs**: `RpcStorageProofsProvider` builds `CommitmentInfo` from the stored trie; a wrong trie root causes the OS/SNOS to receive inconsistent `previous_root` / `updated_root` pairs, producing an invalid or wrong proof. [9](#0-8) 

Matching impacts: **Wrong state, receipt, event, storage value from blockifier/execution logic** (Critical) and **RPC execution / simulation returns an authoritative-looking wrong value** (High).

---

### Likelihood Explanation

Any unprivileged node that establishes a P2P connection to the syncing node can serve state diff chunks. The Starknet P2P protocol is open; peers are untrusted by design. The attacker does not need to forge block signatures ŌĆö it only needs to serve valid-length but wrong-content `StateDiffChunk` messages after the target node has already stored an authentic signed block header (which could have come from a different, honest peer). The attack requires no special timing and is repeatable for every block the victim syncs via P2P.

---

### Recommendation

After assembling the full `ThinStateDiff` and before writing it to storage, verify its Poseidon hash against the stored commitment:

```rust
let expected_commitment = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("header must exist")
    .state_diff_commitment
    .ok_or(P2pSyncClientError::OldHeaderInStorage {
        block_number,
        missing_field: "state_diff_commitment",
    })?;

let actual_commitment = calculate_state_diff_hash(&result);
if actual_commitment != expected_commitment {
    return Err(ParseDataError::BadPeer(BadPeerError::InvalidStateDiffCommitment {
        block_number,
        expected: expected_commitment,
        actual: actual_commitment,
    }));
}
```

This mirrors the check already present in `ApolloCommitter::commit_block_inner` (when `verify_state_diff_hash` is enabled) and closes the gap between the two authenticated fields. [10](#0-9) 

---

### Proof of Concept

1. **Honest peer** sends a valid `SignedBlockHeader` for block N containing authentic `state_diff_commitment = C` and `state_diff_length = L`.
2. The syncing node stores the header (header marker advances to N+1).
3. **Malicious peer** serves `StateDiffChunk` messages for block N. The chunks are crafted so that `sum(chunk.len()) == L` (length check passes) but the assembled `ThinStateDiff` has storage values swapped ŌĆö e.g., contract A's balance slot contains contract B's value and vice versa. The Poseidon hash of the assembled diff is `C' ŌēĀ C`.
4. `parse_data_for_block` exits the loop with `current_state_diff_len == target_state_diff_len`, calls `validate_deprecated_declared_classes_non_conflicting` (passes, no deprecated classes involved), and returns `Ok(Some((wrong_diff, N)))`. [11](#0-10) 

5. `write_to_storage` calls `append_state_diff(N, wrong_diff)` and commits.
6. `ApolloCommitter` receives `CommitBlockRequest { state_diff: wrong_diff, state_diff_commitment: None, height: N }` (commitment is `None` when not supplied by the batcher path), computes `calculate_state_diff_hash(wrong_diff) = C'`, stores `C'` as the authoritative commitment, and derives a wrong `global_root`. [10](#0-9) 

7. All subsequent RPC calls to `starknet_getStorageAt` for the affected contracts return the swapped values. The node's state root diverges from the canonical chain silently.

    /// Creates commitment infos from RPC storage proof without state changes.
    ///
    /// This function assumes that the new state roots equal the previous state roots.
    /// It sets `updated_root` equal to `previous_root` for all commitment infos (contracts,
    /// classes, and storage tries).
    fn create_commitment_infos_without_state_changes(
        rpc_proof: &RpcStorageProof,
        query: &RpcStorageProofsQuery,
    ) -> Result<StateCommitmentInfos, ProofProviderError> {
        let contracts_tree_root = HashOutput(rpc_proof.global_roots.contracts_tree_root);
        let classes_tree_root = HashOutput(rpc_proof.global_roots.classes_tree_root);

        let contracts_trie_commitment_info = CommitmentInfo {
            previous_root: contracts_tree_root,
            // The assumption is that the txs don`t change the state.
            updated_root: contracts_tree_root,
            tree_height: SubTreeHeight::ACTUAL_HEIGHT,
            commitment_facts: flatten_preimages(&Self::rpc_nodes_to_preimage_map(
                &rpc_proof.contracts_proof.nodes,
            )),
        };

        let classes_trie_commitment_info = CommitmentInfo {
            // The assumption is that the txs don`t change the state.
            previous_root: classes_tree_root,
            updated_root: classes_tree_root,
            tree_height: SubTreeHeight::ACTUAL_HEIGHT,
            commitment_facts: flatten_preimages(&Self::rpc_nodes_to_preimage_map(
                &rpc_proof.classes_proof,
            )),
        };

        let storage_tries_commitment_infos =
            Self::build_storage_commitment_infos(rpc_proof, query)?;

        Ok(StateCommitmentInfos {
            contracts_trie_commitment_info,
            classes_trie_commitment_info,
            storage_tries_commitment_infos,
        })
    }
