### Title
Wrapping Integer Arithmetic in `check_time_locks` Enables Timelock Condition Bypass When `nowrap=false` — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` accepts a `nowrap: bool` parameter that switches between `saturating_add` (correct) and `wrapping_add` (legacy). When `nowrap=false`, an unprivileged CLVM spend embedding a near-maximum `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` value causes the addition to wrap to a small integer, making the timelock check trivially pass and the coin spendable immediately. The inverse wrapping on `ASSERT_BEFORE_HEIGHT_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE` causes a valid spend to be incorrectly rejected. Both outcomes are confirmed by the codebase's own unit tests and Python integration tests.

---

### Finding Description

`check_time_locks` computes the required unlock height/timestamp for relative timelocks as:

```rust
// nowrap=false path
unspent.confirmed_block_index.wrapping_add(height_relative)
``` [1](#0-0) 

For `ASSERT_HEIGHT_RELATIVE`, the guard is:

```
prev_height < confirmed_block_index.wrapping_add(height_relative)  →  Err
```

If `height_relative = u32::MAX - confirmed_block_index + 1`, the sum wraps to `0`. The check becomes `prev_height < 0`, which is always `false` for any `u32`, so the timelock is silently bypassed and the spend is accepted.

The same wrapping applies to `ASSERT_SECONDS_RELATIVE` with `u64` arithmetic: [2](#0-1) 

For `ASSERT_BEFORE_HEIGHT_RELATIVE`, the guard is:

```
prev_height >= confirmed_block_index.wrapping_add(before_height_relative)  →  Err
```

With `before_height_relative = u32::MAX`, the sum wraps to `confirmed_block_index - 1`. Since `prev_height >= confirmed_block_index - 1` is almost always true, a spend that should be valid is incorrectly rejected. [3](#0-2) 

The codebase's own unit tests explicitly document and confirm this divergence:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),   // ← timelock bypassed
)]
``` [4](#0-3) 

```
// 200 >= 100 + 0xffff_ffff -> Ok with nowrap (saturates), Err without (wraps)
#[case::before_height_relative_wrap(
    Osc { before_height_relative: Some(0xffff_ffff), ..Default::default() },
    Ok(()),
    Err(ValidationErr::Err(ErrorCode::AssertBeforeHeightRelativeFailed)),  // ← valid spend rejected
)]
``` [5](#0-4) 

The Python integration tests confirm the same for both `height_relative` and `seconds_relative`: [6](#0-5) 

The `nowrap` flag is not controlled by the attacker; it is passed by the Python caller (chia-blockchain). However, the codebase explicitly supports `nowrap=false` as a production path (the legacy pre-soft-fork behavior), and the Python binding exposes it as a first-class parameter: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

**Timelock validation bypass (High).** When `nowrap=false` is active (pre-soft-fork consensus path), any unprivileged CLVM spend that encodes a carefully chosen near-maximum `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` value causes the wrapping sum to collapse to a small integer. The timelock check passes immediately, allowing the coin to be spent before the puzzle-enforced waiting period has elapsed. This matches the allowed High impact: *"timelock or condition validation bypass enables unauthorized spend acceptance."*

**Consensus divergence (Critical).** If nodes running different software versions disagree on the `nowrap` value for the same block height, they will reach different conclusions about whether a spend bundle is valid. A spend with `height_relative = 0xFFFF_FFFF` is accepted by a `nowrap=false` node and rejected by a `nowrap=true` node, causing a committed chain split.

---

### Likelihood Explanation

The `nowrap=false` path is the documented legacy behavior used before the relevant soft fork activation. Any node that has not yet activated the soft fork, or that calls `check_time_locks` with `nowrap=false` for historical block re-validation, is exposed. The attacker-controlled input is the `height_relative` / `seconds_relative` field in a CLVM condition, which is parsed directly from the spend bundle submitted to the network. No privileged role is required.

---

### Recommendation

1. Remove the `nowrap=false` branch entirely once the soft fork is universally activated, or gate it behind a hard-coded height check inside `check_time_locks` itself rather than delegating the decision to the Python caller.
2. Until removal, add a runtime assertion or a compile-time lint that prevents `nowrap=false` from being passed in any consensus-critical call site.
3. Consider replacing the dual-path arithmetic with a single `checked_add` that returns an error on overflow, mirroring the approach already used in `pot_iterations.rs`: [9](#0-8) 

---

### Proof of Concept

Coin confirmed at block index `10`. Node uses `nowrap=false`.

```
height_relative = 0xFFFF_FFF6   (= u32::MAX - 10 + 1 + 1)
confirmed_block_index.wrapping_add(0xFFFF_FFF6) = 0
check: prev_height (15) < 0  →  false  →  spend ACCEPTED immediately
```

Expected behavior with `nowrap=true`:
```
confirmed_block_index.saturating_add(0xFFFF_FFF6) = u32::MAX
check: prev_height (15) < u32::MAX  →  true  →  spend REJECTED (timelock enforced)
```

This is confirmed verbatim by the unit test `height_relative_wrap` and the Python test `test_wrapping_conditions`. [10](#0-9)
