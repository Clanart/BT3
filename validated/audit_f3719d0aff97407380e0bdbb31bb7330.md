Audit Report

## Title
Stale Router ETH Balance Consumed by Subsequent User's WETH Payment — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
The `pay()` function in `PeripheryPayments` reads `address(this).balance` unconditionally when settling a WETH obligation, with no per-transaction accounting. ETH sent as `msg.value` to any payable swap function (e.g., `exactInputSingle`) that exceeds the swap's cost is stranded on the router across transaction boundaries. Any subsequent WETH-input swap by any user silently drains that stranded balance, causing the original depositor to permanently lose their ETH while the later user pays nothing from their own wallet.

## Finding Description
`exactInputSingle` is declared `payable` (MetricOmmSimpleRouter.sol L67). A user may send `msg.value` exceeding `params.amountIn`; the surplus is not refunded unless `refundETH()` is explicitly called in the same multicall. The `receive()` function (PeripheryPayments.sol L32-34) only blocks direct ETH pushes from non-WETH addresses — it does not prevent ETH from accumulating via `msg.value` in payable entry points, so excess ETH persists on the router across blocks.

In a later transaction, when any user calls a WETH-input swap, the callback chain reaches `_justPayCallback` (MetricOmmSimpleRouter.sol L192-199) → `pay(WETH, newUser, pool, amount)`. Inside `pay()` (PeripheryPayments.sol L74-84), `nativeBalance = address(this).balance` captures the entire router balance — including the prior user's stranded ETH. If `nativeBalance >= value`, the function deposits that ETH as WETH and transfers it to the pool, fully satisfying the new user's obligation without touching their wallet or allowance. The transient payer context correctly identifies the new user as payer, but `pay()` ignores that identity when native ETH is available.

No existing guard prevents this: `_requireExpectedCallbackCaller` only validates the pool address; the callback mode and payer checks do not constrain which ETH is consumed.

## Impact Explanation
Direct loss of user principal: User A's stranded ETH is permanently consumed by User B's swap. User A receives no compensation and has no recovery path. User B receives the full token output without spending any WETH or ETH. The pool receives the correct WETH amount and is unaffected. Loss magnitude equals the stranded ETH amount, which is unbounded (any excess `msg.value` without `refundETH()`). This meets the Critical/High threshold for direct loss of user principal.

## Likelihood Explanation
`exactInputSingle` is `payable` and callable directly without a multicall, so any wallet or integrator that sends excess ETH without pairing `refundETH()` will strand funds. No special permissions are required for the attacker: any subsequent WETH-input swap in any later block triggers the drain. The condition is repeatable and requires no coordination.

## Recommendation
Track only the ETH that arrived in the current transaction as eligible for WETH conversion. The standard approach is to record `msg.value` at the multicall or swap entry point, pass it explicitly through the call stack, and use only that tracked amount in `pay()` — decrementing it as it is consumed. Alternatively, compare `address(this).balance` before and after the top-level call and restrict conversion to the delta. Do not rely on `address(this).balance` as a proxy for the current caller's deposited ETH.

## Proof of Concept
```
Setup:
  - Router deployed with WETH address.
  - Pool: WETH (token0) / ERC20 (token1).
  - User A: 2000 wei ETH.
  - User B: 0 ETH, 0 WETH, wants to swap 1000 WETH → token1.

Step 1 (User A, Tx 1):
  User A calls exactInputSingle{value: 2000}(amountIn=1000, tokenIn=WETH, ...)
  directly (no multicall, no refundETH).
  → pay() sees nativeBalance=2000 >= 1000.
  → Deposits 1000 ETH as WETH, transfers to pool. Swap succeeds.
  → 1000 ETH remains stranded on router.

Step 2 (User B, Tx 2):
  User B calls exactInputSingle{value: 0}(amountIn=1000, tokenIn=WETH, ...)
  → _justPayCallback → pay(WETH, UserB, pool, 1000).
  → nativeBalance = 1000 (User A's stranded ETH).
  → nativeBalance >= 1000: deposits 1000 ETH as WETH, transfers to pool.
  → User B's WETH/ETH balance: unchanged (nothing spent).
  → User B receives token1 output.

Result:
  User A: lost 1000 ETH permanently.
  User B: received token1 output at zero cost.
  Pool: received correct WETH, unaware of misattribution.
```
Reproducible as a Foundry integration test by deploying the router and a mock WETH/pool, executing the two transactions in sequence, and asserting User A's ETH loss and User B's zero-cost output.
