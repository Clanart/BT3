Audit Report

## Title
`SwapAllowlistExtension` checks the router's address instead of the originating user, breaking per-user access control — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary
`SwapAllowlistExtension.beforeSwap` receives `sender` from the pool, which is the pool's `msg.sender` — the router contract — not the economic actor who initiated the trade. When swaps are routed through `MetricOmmSimpleRouter`, the allowlist check resolves to `allowedSwapper[pool][router]`, making it impossible to gate individual users: either the router is allowlisted and all users bypass the check, or the router is not allowlisted and all users (including explicitly allowlisted ones) are blocked.

## Finding Description
`MetricOmmPool.swap()` passes `msg.sender` directly as the `sender` argument to `_beforeSwap`:

```solidity
// MetricOmmPool.sol:230-231
_beforeSwap(
  msg.sender,   // whoever called pool.swap() — the router, not the user
```

`SwapAllowlistExtension.beforeSwap` then checks:

```solidity
// SwapAllowlistExtension.sol:37
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
```

Here `msg.sender` is the pool and `sender` is the router. The check resolves to `allowedSwapper[pool][router]`.

`MetricOmmSimpleRouter.exactInputSingle` stores the original caller only in transient callback context for payment, and calls `pool.swap()` directly:

```solidity
// MetricOmmSimpleRouter.sol:71-80
_setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
(int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
  .swap(params.recipient, params.zeroForOne, ...);
```

The original `msg.sender` is never forwarded to the pool as the swap initiator. Two broken scenarios result:

**Scenario A — Allowlist bypass:** Admin allowlists the router (natural configuration for router-mediated swaps). Any unprivileged user calls `exactInputSingle`; the check sees `sender=router` → `true` → swap proceeds. The per-user allowlist is completely bypassed.

**Scenario B — Legitimate users blocked:** Admin allowlists individual user addresses. Those users call `exactInputSingle`; the check sees `sender=router` → `false` → reverts with `NotAllowedToSwap`. Allowlisted users cannot use the router at all.

No existing guard compensates for this: the extension has no mechanism to recover the original caller, and the pool passes no additional identity context to extension hooks.

## Impact Explanation
`SwapAllowlistExtension` is a production access-control primitive whose stated purpose is to gate `swap` by swapper address per pool. Under Scenario A, any unprivileged address executes swaps on a pool the admin intended to restrict — a direct admin-boundary bypass by an unprivileged path. Under Scenario B, the standard periphery swap path is entirely unusable for allowlisted users, constituting broken core pool functionality. Both outcomes are reachable without any special attacker capability.

## Likelihood Explanation
The router is the primary public swap entrypoint. Any pool deployer who configures `SwapAllowlistExtension` and uses the router will encounter one of these two failure modes immediately. No special attacker capability is required beyond calling `exactInputSingle` through the public router.

## Recommendation
The pool should forward the true economic actor's identity to extension hooks rather than `msg.sender`. The cleanest fix is to add an optional `swapper` parameter to `pool.swap()` that defaults to `msg.sender` but can be overridden by trusted periphery contracts, and pass that value to `_beforeSwap` instead of `msg.sender`. Alternatively, `SwapAllowlistExtension` could read an ABI-encoded `address` from `extensionData` as the identity to check, with the router forwarding `msg.sender` there — but this requires router cooperation and is fragile. The pool-level fix is preferred.

## Proof of Concept
1. Deploy `MetricOmmPool` with `SwapAllowlistExtension` in `beforeSwapOrder`.
2. Pool admin calls `setAllowedToSwap(pool, routerAddress, true)` — allowlisting the router.
3. Unprivileged `attacker` (not individually allowlisted) calls `router.exactInputSingle(...)`.
4. Pool calls `_beforeSwap(msg.sender=router, ...)` → extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
5. `attacker` successfully swaps on a pool they were never individually authorized to use.

For Scenario B:
1. Pool admin calls `setAllowedToSwap(pool, alice, true)`.
2. Alice calls `router.exactInputSingle(...)`.
3. Extension checks `allowedSwapper[pool][router]` → `false` → reverts with `NotAllowedToSwap`.
4. Alice cannot use the router despite being explicitly allowlisted.
