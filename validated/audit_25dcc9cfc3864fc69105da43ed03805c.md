### Title
Unprotected `initialize()` Allows Anyone to Frontrun and Seize Admin Control of OmniBridge - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`OmniBridge.initialize()` is declared `public initializer` with no access-control modifier. Any unprivileged actor who submits the initialization transaction before the protocol can seize `DEFAULT_ADMIN_ROLE` and inject a malicious `nearBridgeDerivedAddress`, the sole ECDSA key used to authorize every `finTransfer()` and `deployToken()` call. This enables the attacker to forge signatures and mint or release arbitrary bridge assets.

### Finding Description
`OmniBridge.initialize()` carries only OpenZeppelin's `initializer` modifier, which exclusively prevents re-initialization; it imposes no restriction on *who* may be the first caller. [1](#0-0) 

The function performs two security-critical writes in a single call:

1. **`nearBridgeDerivedAddress`** — the address recovered via `ECDSA.recover` in every `finTransfer()` and `deployToken()` call. Whoever controls the corresponding private key can authorize any cross-chain settlement.
2. **`DEFAULT_ADMIN_ROLE`** — granted to `_msgSender()`, giving full administrative control over the bridge (pause, upgrade, token management, etc.). [2](#0-1) 

The constructor correctly calls `_disableInitializers()` to lock the *implementation* contract: [3](#0-2) 

However, the *proxy* contract is a separate deployment. If the proxy is deployed without initialization calldata (a common two-step deployment pattern), `initialize()` remains callable by anyone until the protocol's own transaction lands. An attacker monitoring the mempool can frontrun that transaction.

The same pattern applies to `OmniBridgeWormhole.initializeWormhole()`, which is also `external initializer` with no access control and internally delegates to `initialize()`: [4](#0-3) 

### Impact Explanation
If the attacker wins the race:

- They become the sole `DEFAULT_ADMIN_ROLE` holder.
- They supply their own `nearBridgeDerivedAddress` (an address whose private key they control).
- `finTransfer()` verifies signatures exclusively against `nearBridgeDerivedAddress`: [5](#0-4) 

- The attacker can craft and sign arbitrary `TransferMessagePayload` structs, calling `finTransfer()` to mint unbacked bridge tokens (`IBridgeToken.mint`) or release locked native ETH and ERC-20 tokens held by the bridge.
- `deployToken()` is similarly gated only by this signature check, so the attacker can register malicious token mappings.

This is **direct unauthorized mint and theft of bridged assets** — a Critical impact under the allowed scope.

### Likelihood Explanation
The attack requires only that the proxy be deployed without atomic initialization (i.e., the `ERC1967Proxy` constructor is called with empty `data`, and `initialize()` is called in a subsequent transaction). This two-step pattern is common in deployment scripts and upgrade workflows. A mempool-watching bot can detect the pending `initialize()` transaction and submit a higher-gas frontrun. No privileged access, leaked key, or colluding party is required — only a standard Ethereum transaction.

### Recommendation
Add an `onlyOwner` or equivalent access-control check to `initialize()`, or — the preferred pattern — always deploy the proxy with initialization calldata so that initialization is atomic with proxy creation (as is already done for `BridgeToken` proxies in `deployToken()`): [6](#0-5) 

For the main bridge proxy, pass the encoded `initialize(...)` call as the `data` argument to the `ERC1967Proxy` constructor. If a two-step deployment is unavoidable, restrict `initialize()` to a deployer address set in the constructor (stored in immutable storage), mirroring the `onlyOwner` pattern recommended in the external report.

### Proof of Concept
1. Protocol deploys `OmniBridgeWormhole` implementation and then deploys an `ERC1967Proxy(implementation, "")` with empty data.
2. Attacker observes the pending `initializeWormhole(...)` transaction in the mempool.
3. Attacker submits `initialize(attackerControlledAddress, attackerAddress, chainId)` with higher gas, where `attackerControlledAddress` is an address whose private key the attacker holds.
4. Attacker's transaction is mined first: `DEFAULT_ADMIN_ROLE` is granted to the attacker; `nearBridgeDerivedAddress` is set to `attackerControlledAddress`.
5. Protocol's `initializeWormhole()` reverts (`Initializable: contract is already initialized`).
6. Attacker constructs a `TransferMessagePayload` targeting any ERC-20 token held by the bridge (or a bridge token), signs it with the private key for `attackerControlledAddress`, and calls `finTransfer(signature, payload)`.
7. `ECDSA.recover(hashed, signatureData) == nearBridgeDerivedAddress` passes; the bridge mints or transfers tokens to the attacker. [7](#0-6)

        bytes calldata signatureData,
        BridgeTypes.TransferMessagePayload calldata payload

        if (completedTransfers[payload.destinationNonce]) {
            revert NonceAlreadyUsed(payload.destinationNonce);
        }

        completedTransfers[payload.destinationNonce] = true;

        bytes memory borshEncoded = bytes.concat(
            bytes1(uint8(BridgeTypes.PayloadType.TransferMessage)),
            Borsh.encodeUint64(payload.destinationNonce),
            bytes1(payload.originChain),
            Borsh.encodeUint64(payload.originNonce),
            bytes1(omniBridgeChainId),
            Borsh.encodeAddress(payload.tokenAddress),
            Borsh.encodeUint128(payload.amount),
            bytes1(omniBridgeChainId),
            Borsh.encodeAddress(payload.recipient),
            bytes(payload.feeRecipient).length == 0 // None or Some(String) in rust
                ? bytes("\x00")
                : bytes.concat(
                    bytes("\x01"),
                    Borsh.encodeString(payload.feeRecipient)
                ),

                ? bytes("")
                : Borsh.encodeBytes(payload.message)
        );
        bytes32 hashed = keccak256(borshEncoded);
