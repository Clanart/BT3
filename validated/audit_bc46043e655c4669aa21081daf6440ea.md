### Title
`batch_insert` Skips Duplicate-Key Guard in Bulk Path, Leaving Orphaned Leaves and Corrupting the DataLayer Merkle Root — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` has two code paths. When the tree already has more than one leaf, **all** supplied items bypass the duplicate-key check that exists in `MerkleBlob::insert`. A caller who supplies a key that already exists in the tree causes a second leaf to be written at a fresh index while the original leaf remains physically present in the blob and still wired into the parent-child graph. The `block_status_cache` is updated to point the key to the new index, but the old index is never added to `free_indexes` and is never removed from the tree's structural links. The result is a tree that contains two leaves sharing the same key, a Merkle root that covers both, and a cache that only knows about one — a direct state-synchronization split between the blob and its in-memory cache.

---

### Finding Description

`MerkleBlob::insert` (the single-item path) guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `self.insert(…)` for at most two items only when `leaf_count <= 1`:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else {
            return Ok(());
        };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;
    }
}
``` [2](#0-1) 

Every remaining item — and **all** items when `leaf_count > 1` — goes through the bulk path, which calls `insert_entry_to_blob` directly with no duplicate check:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { … Node::Leaf(LeafNode { parent: Parent(None), hash, key, value }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [3](#0-2) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf(index, leaf)`:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // silently overwrites old entry
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
``` [4](#0-3) 

`HashMap::insert` returns the displaced old index but `add_leaf` discards it. The old index is never pushed into `free_indexes`, and the old leaf node — still physically present in the blob with its parent pointer intact — is never unlinked from the tree. After `insert_subtree_at_key` grafts the new subtree in, the tree contains **two** leaves with the same key, both reachable from the root. [5](#0-4) 

---

### Impact Explanation

The Merkle root is computed over all structurally reachable nodes. With two leaves sharing key K (old leaf at `old_index`, new leaf at `new_index`), the root hash covers both. `get_proof_of_inclusion(K)` builds a proof anchored at `new_index` (the only entry in `key_to_index`), but the root it must match was computed including the orphaned old leaf. The proof is therefore invalid against the stored root, or — if the root is recomputed after the corruption — it encodes a root that no honest peer will accept. Either way, inclusion/exclusion proofs for the affected key are broken.

`check_integrity` will also detect the split: iterating the blob yields `leaf_count = N+1` while `key_to_index_cache_length = N`, triggering `IntegrityKeyToIndexCacheLength`. [6](#0-5) 

---

### Likelihood Explanation

`MerkleBlob` is a `pyclass` exposed to Python callers. Any DataLayer operation that calls `batch_insert` with a list containing a key already present in the tree (e.g., an update disguised as an insert, or a replayed delta) triggers the corruption. The tree must have at least two existing leaves for the guard to be skipped entirely, which is the normal operating state of any non-trivial DataLayer store. [7](#0-6) 

---

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the bulk path of `batch_insert` that already exist in `insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // … existing bulk insertion logic …
}
```

Alternatively, `add_leaf` should detect when `key_to_index.insert` displaces an existing entry and immediately add the displaced index to `free_indexes` (and unlink the old leaf from the tree), mirroring the fix in the referenced report where `updateSupply` was added to keep the two state components synchronized.

    pub fn batch_insert(
        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {
        // OPT: perhaps go back to taking an iterator?
        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),
                };

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {
            // OPT: can we avoid this extra min height leaf traversal?
            let min_height_leaf = self.get_min_height_leaf()?;
            self.insert_subtree_at_key(min_height_leaf.key, indexes[0], Side::Left)?;
        }

        Ok(())
    }
