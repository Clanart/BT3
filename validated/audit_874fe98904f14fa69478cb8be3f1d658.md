Audit Report

## Title
Permissionless `refundETH()` allows any caller to drain excess ETH left on the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`refundETH()` sends the router's full ETH balance to `msg.sender` with no access control. When a user calls `exactInputSingle` (or any payable swap function) with `msg.value > amountIn` and `tokenIn == WETH`, the `pay()` helper wraps exactly `amountIn` ETH and silently leaves the remainder on the router. Any third party can then call `refundETH()` in a subsequent transaction to claim that ETH, causing a direct, unrecoverable loss to the original caller.

## Finding Description

`refundETH()` is unconditionally callable by any address and transfers the entire native balance to `msg.sender`: [1](#0-0) 

The `pay()` helper, when `token == WETH` and `nativeBalance >= value`, wraps exactly `value` ETH via `IWETH9.deposit` and leaves any surplus untouched on the contract: [2](#0-1) 

`exactInputSingle` is `payable` and records `msg.sender` as the payer in transient storage, but performs no refund of excess ETH before returning: [3](#0-2) 

The `receive()` guard that rejects non-WETH ETH senders provides no protection here — it does not prevent ETH accumulation via `msg.value` in payable swap functions: [4](#0-3) 

The codebase's own test confirms the design requires `refundETH()` as an explicit second multicall entry to recover unused ETH, and only asserts the router is empty after that second call: [5](#0-4) 

If a user omits `refundETH()` from their multicall, or calls `exactInputSingle` directly with `msg.value > amountIn`, the excess ETH sits on the router unguarded. Because `multicall` uses `delegatecall` (preserving `msg.sender`), `refundETH()` correctly refunds the original caller only when bundled in the same multicall transaction. In a separate transaction, `msg.sender` is the attacker. [6](#0-5) 

## Impact Explanation

Direct, permanent loss of user ETH principal. The attacker receives the full excess balance; the victim has no recovery path because `refundETH()` sends to `msg.sender` with no record of the original depositor. This meets the Critical/High direct loss of user principal threshold under the allowed impact gate.

## Likelihood Explanation

- Any user who sends `msg.value > amountIn` without bundling `refundETH()` in the same multicall is immediately vulnerable.
- Integrators (wallets, aggregators) building on top of the router may omit the refund step, as it is not enforced by the contract.
- MEV bots can monitor the mempool for transactions that leave ETH on the router and back-run them atomically in the same block.
- The pattern is not self-evident: the `receive()` guard gives a false sense of security, and the contract provides no on-chain enforcement that `refundETH()` must be called.

## Recommendation

Add a `msg.sender`-binding mechanism: record the original multicall initiator at entry (e.g., in transient storage) and restrict `refundETH()` to that address only. Alternatively, auto-refund excess ETH at the end of each payable swap function rather than relying on the caller to include a separate `refundETH()` step.

## Proof of Concept

```solidity
// Foundry integration test
function test_attacker_steals_excess_eth() public {
    uint128 amountIn = 1_000;
    uint256 msgValue = 2 ether;

    // Victim calls exactInputSingle directly (or via multicall with only 1 entry, no refundETH)
    vm.prank(victim);
    router.exactInputSingle{value: msgValue}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: amountIn,
            amountOutMinimum: 0,
            recipient: victim,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );
    // (2 ether - 1_000) now sits on the router

    uint256 attackerBefore = attacker.balance;
    vm.prank(attacker);
    router.refundETH();

    assertEq(attacker.balance - attackerBefore, 2 ether - 1_000); // attacker receives excess
    assertEq(address(router).balance, 0);
    // victim cannot recover the excess ETH
}
```

  function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(
        params.recipient,
        params.zeroForOne,
        MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
        priceLimitX64,
        "",
        params.extensionData
      );
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
