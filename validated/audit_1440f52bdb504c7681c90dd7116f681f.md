### Title
Permanent Freezing of User Funds When `normalize_amount` Returns Zero Before `sign_transfer` — (File: near/omni-bridge/src/lib.rs)

---

### Summary

`init_transfer` accepts and locks/burns user tokens without verifying that the net transfer amount (after fee deduction) will survive decimal normalization. When `normalize_amount(amount_without_fee)` floors to zero, `sign_transfer` permanently reverts with `InvalidAmountToTransfer`, leaving the user's tokens irrecoverably locked or burned on NEAR with no cancel path.

---

### Finding Description

`normalize_amount` performs integer floor division to scale a NEAR-precision amount down to the destination chain's precision: [1](#0-0) 

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

For a token whose NEAR representation has more decimals than the destination chain (e.g., NEAR native: 24 origin decimals → 18 EVM decimals, `diff_decimals = 6`), any `amount_without_fee < 10^6` normalizes to **zero**.

`sign_transfer` correctly guards against emitting a zero-amount payload: [2](#0-1) 

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee()...,
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

However, `init_transfer` — which locks or burns the user's tokens — performs **no equivalent check**: [3](#0-2) 

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

Only `fee < amount` is enforced. There is no check that `normalize_amount(amount - fee) > 0`. Once `init_transfer` succeeds, the transfer is stored and the tokens are gone from the user. Every subsequent `sign_transfer` call for that transfer ID will revert, and no cancel/refund path was found in the codebase.

The documented "dust stays locked/burned" comment refers only to the sub-unit *remainder* after normalization of a valid amount — not to the case where the *entire net amount* floors to zero: [4](#0-3) 

---

### Impact Explanation

A user who initiates a NEAR→EVM transfer where `amount_without_fee < 10^diff_decimals` will have their tokens permanently locked or burned on NEAR with no delivery on the destination chain and no recovery mechanism. This matches the **Critical/High** impact class: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.*

Concrete example with NEAR native token (24 NEAR decimals → 18 EVM decimals, `diff_decimals = 6`):

| Field | Value |
|---|---|
| `amount` | 1,000,001 yoctoNEAR |
| `fee.fee` | 1,000,000 yoctoNEAR |
| `amount_without_fee` | 1 yoctoNEAR |
| `normalize_amount(1, diff=6)` | **0** |
| Result | `sign_transfer` always reverts; 1,000,001 yoctoNEAR permanently frozen |

For tokens with larger `diff_decimals` (e.g., 18), the threshold below which the entire net amount is lost rises to `10^18` smallest units — a potentially significant value.

---

### Likelihood Explanation

- Any unprivileged user calling `ft_transfer_call` → `init_transfer` with a small amount or a fee that leaves a sub-threshold net amount triggers this path.
- No special role or key is required.
- The `update_transfer_fee` function only allows *increasing* the fee, so the user cannot self-rescue by lowering the fee after the fact: [5](#0-4) 

```rust
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

---

### Recommendation

Add a normalization check inside `init_transfer` (or in `init_transfer_internal`) before storing the transfer and consuming the user's tokens:

```rust
// Fetch decimals for the destination token and validate net amount survives normalization
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some((token_addr, decimals)) = token_address.zip(
    token_address.and_then(|a| self.token_decimals.get(&a))
) {
    let net = transfer_message.amount_without_fee()
        .expect("fee < amount already checked");
    require!(
        Self::normalize_amount(net, decimals) > 0,
        BridgeError::InvalidAmountToTransfer.as_ref()
    );
}
```

Alternatively, expose a cancel/refund path for transfers that can never be signed, so users are not permanently locked out of their funds.

---

### Proof of Concept

1. Token: NEAR native, `origin_decimals = 24`, EVM `decimals = 18`, `diff_decimals = 6`.
2. User calls `ft_transfer_call` with `amount = 1_000_001`, `fee = 1_000_000`.
3. `init_transfer` passes the `fee < amount` check and stores the transfer; tokens are burned.
4. Relayer calls `sign_transfer`:
   - `amount_without_fee()` = 1
   - `normalize_amount(1, diff=6)` = `1 / 1_000_000` = **0**
   - `require!(0 > 0)` → **panics** with `InvalidAmountToTransfer`
5. No relayer can ever sign this transfer. No cancel function exists. User's 1,000,001 yoctoNEAR are permanently frozen. [6](#0-5) [2](#0-1) [1](#0-0)
