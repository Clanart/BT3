### Title
Nonce Overwrite in `applyCall` CALL Path Discards Nested-CREATE Increments, Corrupting Multi-Call `eth_simulateV1` State - (File: `x/evm/keeper/simulate.go`)

### Summary
`Simulator.applyCall` unconditionally overwrites the sender's nonce to `msg.Nonce+1` after `evm.Call()` returns. When the called code (e.g., via an EIP-7702 delegation) executes nested `CREATE` opcodes, those opcodes increment the sender's nonce inside `evm.Call()`. The hard-coded post-call `SetNonce(msg.From, msg.Nonce+1)` then silently discards every nonce increment beyond the first, leaving the shared simulation `StateDB` with a stale, too-low nonce for all subsequent calls in the same `eth_simulateV1` block.

### Finding Description

In `applyCall` (`x/evm/keeper/simulate.go`), the nonce management for the CALL branch is:

```go
// line 432 – reset to msg.Nonce before execution
sim.state.SetNonce(msg.From, msg.Nonce, tracing.NonceChangeUnspecified)

// ... apply EIP-7702 authorizations ...

// line 449 – execute the call (nested CREATEs inside here bump the nonce)
ret, leftoverGas, vmErr = evm.Call(msg.From, *msg.To, msg.Data, leftoverGas, value)

// line 450 – UNCONDITIONAL overwrite: loses every nested-CREATE increment
sim.state.SetNonce(msg.From, msg.Nonce+1, tracing.NonceChangeUnspecified)
``` [1](#0-0) 

Inside `evm.Call()`, if the callee is an EIP-7702-delegated EOA whose delegated contract executes `CREATE` opcodes, go-ethereum's `evm.Create()` increments the sender's nonce for each deployment. After `evm.Call()` returns the nonce in `sim.state` may be `msg.Nonce + 1 + N` (N nested creates). Line 450 then unconditionally resets it to `msg.Nonce + 1`, erasing the N extra increments.

This is structurally identical to the external report's pattern: a single tracked value per address is overwritten on every operation, discarding intermediate state changes.

The production path `ApplyMessageWithConfig` does **not** have this bug for the CALL branch — it never resets the nonce after `evm.Call()` because the ante handler pre-incremented it and nested creates are left to accumulate naturally:

```go
// state_transition.go line 517 – no post-call nonce overwrite
ret, leftoverGas, vmErr = evm.Call(sender, *msg.To, msg.Data, leftoverGas, uint256.MustFromBig(msg.Value))
``` [2](#0-1) 

The CREATE branch in `applyCall` correctly avoids this problem by reading `afterCreateNonce` and computing `nestedCreates` before writing back:

```go
// simulate.go line 438-439 – CREATE path: no unconditional overwrite
ret, _, leftoverGas, vmErr = evm.Create(msg.From, msg.Data, leftoverGas, value)
// (no SetNonce after this)
``` [3](#0-2) 

But the CALL branch at line 450 has no equivalent reconciliation.

The `sanitizeCall` function auto-fills `call.Nonce` from `state.GetNonce()` when the caller omits it:

```go
if call.Nonce == nil {
    nonce := state.GetNonce(call.GetFrom())
    call.Nonce = (*hexutil.Uint64)(&nonce)
}
``` [4](#0-3) 

Because the stale nonce is committed to `sim.state`, every subsequent auto-filled call in the same simulation block inherits the wrong (too-low) nonce.

### Impact Explanation

**Impact: High** — Public JSON-RPC simulation path feeds incorrect nonce data, corrupting multi-call `eth_simulateV1` results.

1. **Validation mode (`validation: true`)**: Subsequent calls from the same sender in the same simulated block fail with `nonce too high` because the state nonce was reset to `msg.Nonce+1` instead of the correct `msg.Nonce+1+N`. The simulation aborts or returns incorrect error results.

2. **Non-validation mode (default)**: `sanitizeCall` auto-fills nonces from the stale state. All subsequent calls from the same sender use nonces that are N too low. Contract addresses derived via `CREATE` inside those calls are computed from wrong nonces, producing incorrect `receipt.ContractAddress` values and wrong `eth_simulateV1` output.

3. **EIP-7702 authorization nonce interaction**: `applyAuthorization` calls `setAuthorizationDelegation` which also calls `SetNonce(authority, auth.Nonce+1)`. If `authority == msg.From` (self-delegation), line 450 then overwrites that increment too. [5](#0-4) 

The codebase's own test comment documents the analogous CREATE-path bug and its consequences:

> "BUG: After evm.Create() returns, nonce is reset to N+1, losing all increments" [6](#0-5) 

The CALL-path overwrite at line 450 is the same class of bug, unaddressed in the CALL branch.

### Likelihood Explanation

**Likelihood: Medium.** The trigger requires:
1. An `eth_simulateV1` request with `blockStateCalls` containing multiple calls from the same sender in one block, **or** a single call where the sender has an EIP-7702 delegation that deploys contracts.
2. EIP-7702 is live on Prague-enabled chains.

Any unprivileged user can submit such a request to the public JSON-RPC endpoint. No special privileges, keys, or governance are required.

### Recommendation

Replace the unconditional `SetNonce` at line 450 with a reconciliation that preserves nested-CREATE increments, mirroring the CREATE-branch logic already present in `ApplyMessageWithConfig`:

```go
// After evm.Call() returns:
afterCallNonce := sim.state.GetNonce(msg.From)
nestedCreates := uint64(0)
if afterCallNonce > msg.Nonce {
    nestedCreates = afterCallNonce - msg.Nonce
}
// Only write if we need to advance the nonce (i.e., no nested creates already did it)
if afterCallNonce <= msg.Nonce {
    sim.state.SetNonce(msg.From, msg.Nonce+1, tracing.NonceChangeUnspecified)
}
// If afterCallNonce > msg.Nonce+1, nested creates already advanced it correctly — leave it.
```

Alternatively, remove line 450 entirely and instead set the nonce to `msg.Nonce+1` **before** calling `evm.Call()` (as go-ethereum's `StateTransition` does), so nested creates can only increment from there.

### Proof of Concept

1. Deploy an EIP-7702 delegation target contract that executes `CREATE` in its body.
2. Submit an `eth_simulateV1` request with `validation: true` and two calls in one `blockStateCalls` entry:
   - **Call 1**: EIP-7702 CALL to the delegated EOA (triggers 2 nested CREATEs → nonce should advance to N+3).
   - **Call 2**: Any call from the same sender with auto-filled nonce.
3. **Observed**: Call 2 fails with `nonce too high` because `sim.state` nonce was reset to N+1 (not N+3) by line 450.
4. **Expected**: Call 2 succeeds with nonce N+3, matching go-ethereum behavior.

The existing integration test `test_eip7702_nonce_increment` demonstrates the identical pattern for the CREATE path and confirms the codebase is aware of this class of bug; the CALL path at line 450 is the unpatched analog. [7](#0-6)
