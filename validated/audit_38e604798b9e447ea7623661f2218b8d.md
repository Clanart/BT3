### Title
`MerkleBlob::upsert` Clears Cache Entry Without Re-Adding It, Corrupting Tree State and Merkle Roots — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::upsert` calls `block_status_cache.remove_leaf(&leaf)` to clear the old cache entry, then writes the updated leaf to the blob via `insert_entry_to_blob`, but **never calls `add_leaf` to re-register the leaf in the cache**. This leaves the `BlockStatusCache` permanently inconsistent with the blob: the key is absent from `key_to_index`, the new hash is absent from `leaf_hash_to_index`, and the leaf's index is marked as free (available for reuse). All subsequent operations that rely on the cache — including duplicate-key checks, proof generation, and index allocation — operate on corrupted state.

---

### Finding Description

`MerkleBlob::upsert` (lines 792–810) follows this sequence:

1. Looks up the existing leaf by key.
2. Calls `self.block_status_cache.remove_leaf(&leaf)` — this removes the key from `key_to_index`, removes the old hash from `leaf_hash_to_index`, and **inserts the leaf's index into `free_indexes`** (marking it available for reuse).
3. Updates `leaf.hash` and `leaf.value` in local variables.
4. Calls `self.insert_entry_to_blob(leaf_index, &block)` — this writes the updated leaf bytes to the blob only; it does **not** update the cache.
5. Returns `Ok(())`.

`add_leaf` is never called. Compare with the normal insert path (`insert_first`, `insert_second`, `insert_third_or_later`), which always calls `block_status_cache.add_leaf(index, leaf)` after writing to the blob:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);   // un-marks as free
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
```

After `upsert` returns, the cache state is:
- `key_to_index` does **not** contain the upserted key → `get_leaf_by_key` returns `Err`, treating the key as non-existent.
- `leaf_hash_to_index` does **not** contain the new hash → the duplicate-hash guard in `insert` is bypassed.
- `free_indexes` **contains** the leaf's index → a subsequent `get_new_index()` call can return this index, causing a new node to overwrite the upserted leaf in the blob. [1](#0-0) 

`remove_leaf` marks the index as free: [2](#0-1) 

`add_leaf` (which is never called in `upsert`) would have un-marked it and re-registered the key and hash: [3](#0-2) 

---

### Impact Explanation

After any `upsert` call:

1. **Tree root corruption**: The blob contains the updated leaf, but the cache does not reflect it. `calculate_lazy_hashes` traverses the blob and recomputes hashes, so the root hash itself may be correct transiently — but the cache-driven operations that follow are wrong.

2. **Index reuse / blob overwrite**: Because the leaf's index is in `free_indexes`, the next `insert` or `batch_insert` call can allocate that same index via `get_new_index()`, overwriting the upserted leaf in the blob. This silently destroys a key-value pair and produces a structurally invalid tree.

3. **Duplicate key insertion**: Because the key is absent from `key_to_index`, the guard `if self.block_status_cache.contains_key(key)` in `insert` returns `false`, allowing the same key to be inserted again. Two leaves with the same key now exist in the blob, violating the tree's invariant and producing an incorrect root.

4. **Forged inclusion/exclusion proofs**: `get_proof_of_inclusion` relies on `get_leaf_by_key`, which uses `key_to_index`. After `upsert`, the upserted key is not found, so a proof of inclusion cannot be generated for a key that genuinely exists in the blob. Conversely, a proof of exclusion for that key would be accepted even though the key is present. This directly enables forged Merkle state. [4](#0-3) 

---

### Likelihood Explanation

`upsert` is exposed directly as a Python binding (`py_upsert`) and is called during DataLayer delta-sync operations. Any DataLayer store update that modifies an existing key-value pair triggers `upsert`. This is a normal, unprivileged operation in the DataLayer workflow — no special privileges or adversarial input are required. The corruption occurs on every `upsert` call without exception. [5](#0-4) 

---

### Recommendation

After writing the updated leaf to the blob, call `block_status_cache.add_leaf(leaf_index, updated_leaf)` to re-register the key, new hash, and index in the cache. The corrected `upsert` should be:

```rust
pub fn upsert(&mut self, key: KeyId, value: ValueId, new_hash: &Hash) -> Result<(), Error> {
    let Ok((leaf_index, mut leaf, mut block)) = self.get_leaf_by_key(key) else {
        self.insert(key, value, new_hash, InsertLocation::Auto {})?;
        return Ok(());
    };

    self.block_status_cache.remove_leaf(&leaf)?;
    leaf.hash.clone_from(new_hash);
    leaf.value = value;
    block.node = Node::Leaf(leaf.clone());
    self.insert_entry_to_blob(leaf_index, &block)?;
    self.block_status_cache.add_leaf(leaf_index, leaf); // <-- missing call

    if let Some(parent) = block.node.parent().0 {
        self.mark_lineage_as_dirty(parent)?;
    }

    Ok(())
}
```

let mut blob = MerkleBlob::new(vec![]).unwrap();

let hash1 = Hash(...);
let hash2 = Hash(...); // different from hash1

blob.insert(key, ValueId(10), &hash1, InsertLocation::Auto {}).unwrap();
blob.upsert(key, ValueId(20), &hash2).unwrap();

// BUG 1: key appears non-existent in cache
assert!(blob.block_status_cache.get_index_by_key(key).is_none()); // passes — key is gone

// BUG 2: index is marked free and can be overwritten
let new_key = KeyId(2);
let hash3 = Hash(...);
blob.insert(new_key, ValueId(30), &hash3, InsertLocation::Auto {}).unwrap();
// The new insert may reuse the upserted leaf's index, overwriting it in the blob.

// BUG 3: duplicate key insertion succeeds
blob.insert(key, ValueId(99), &hash1, InsertLocation::Auto {}).unwrap(); // should fail, but succeeds

// BUG 4: proof of inclusion for the upserted key fails even though it's in the blob
blob.get_proof_of_inclusion(key).unwrap_err(); // UnknownKey — but key IS in the blob
