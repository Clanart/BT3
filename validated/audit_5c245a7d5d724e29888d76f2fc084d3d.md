### Title
MetadataPayload Signature Lacks Chain Binding, Enabling Cross-Chain Replay in `deployToken` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `deployToken` function constructs a Borsh-encoded payload for MPC signature verification that does **not** include `omniBridgeChainId`. Because all deployed OmniBridge instances share the same `nearBridgeDerivedAddress` (derived from the single NEAR bridge contract), a valid MPC signature obtained for chain A is cryptographically valid on chain B. An unprivileged relayer can replay it, deploying the token and hijacking `nearToEthToken[token]` on the second chain.

---

### Finding Description

In `deployToken`, the signed payload is constructed as: [1](#0-0) 

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

`omniBridgeChainId` is **absent**. The signature check is: [2](#0-1) 

Contrast this with `finTransfer`, which **does** bind the payload to the chain by embedding `omniBridgeChainId` twice: [3](#0-2) 

The `MetadataPayload` struct itself contains no chain field: [4](#0-3) 

The only replay guard in `deployToken` is: [5](#0-4) 

This guard only prevents re-deployment on the **same** chain instance. On chain B, `nearToEthToken[metadata.token]` is `address(0)`, so `isBridgeToken[address(0)]` is `false`, and the check passes.

---

### Impact Explanation

An attacker who observes a valid `deployToken` call on chain A (Ethereum, `omniBridgeChainId=0`) can submit the identical `(signatureData, metadata)` tuple to chain B (Arbitrum, `omniBridgeChainId=3`). Because both instances share the same `nearBridgeDerivedAddress` and the signed hash is identical (no chain discriminator), the ECDSA recovery succeeds on chain B. The result:

- A bridge token is deployed on chain B at an address the protocol did not intend.
- `nearToEthToken[token]` and `ethToNearToken[bridgeTokenProxy]` are set on chain B.
- All subsequent legitimate `deployToken` calls for that NEAR token on chain B revert with `"ERR_TOKEN_EXIST"`.
- The canonical token mapping on chain B is permanently hijacked to the attacker-replayed deployment. [6](#0-5) 

---

### Likelihood Explanation

- `nearBridgeDerivedAddress` is derived from the NEAR bridge contract account ID, which is the same across all EVM deployments — this is the standard deployment model.
- All `deployToken` calls are public on-chain; the signature and payload are visible in the mempool or block history of chain A.
- No privileged access is required. Any EOA can call `deployToken` on chain B with chain A's data.

---

### Recommendation

Include `omniBridgeChainId` in the Borsh-encoded `MetadataPayload` before hashing, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

The NEAR MPC signing logic must be updated in parallel to include the destination `omniBridgeChainId` in the signed payload.

---

### Proof of Concept

1. Deploy two `OmniBridge` instances: chain A (`omniBridgeChainId=0`) and chain B (`omniBridgeChainId=3`), both initialized with the same `nearBridgeDerivedAddress`.
2. Obtain a valid MPC-signed `MetadataPayload` for NEAR token `"token.near"` targeting chain A (e.g., via a legitimate `deployToken` call or by observing the transaction).
3. Submit the identical `(signatureData, metadata)` to chain B's `deployToken`.
4. Observe: `ECDSA.recover(keccak256(borshEncoded), signatureData) == nearBridgeDerivedAddress` succeeds on chain B (same hash, same signer address).
5. `nearToEthToken["token.near"]` is now set on chain B to the replayed deployment address.
6. Any subsequent legitimate `deployToken` for `"token.near"` on chain B reverts with `"ERR_TOKEN_EXIST"`.
