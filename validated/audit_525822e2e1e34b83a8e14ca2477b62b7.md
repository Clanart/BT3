Audit Report

## Title
Router `pay()` Unconditionally Consumes Any Native ETH on the Router to Settle WETH Obligations, Enabling Cross-User ETH Theft — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` helper in `PeripheryPayments.sol` reads `address(this).balance` — the total native ETH held by the router — when settling a WETH payment obligation, with no per-user attribution. Because the router accumulates native ETH from `multicall{value: N}(...)` calls that are not automatically refunded, a subsequent unprivileged caller whose `tokenIn` is WETH can have their entire swap settled using another user's stranded ETH, stealing it without spending any of their own funds.

## Finding Description
`pay()` at lines 73–84 of `PeripheryPayments.sol` contains the following logic when `token == WETH` and `payer != address(this)`:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
```

`address(this).balance` is the **global** contract balance with no per-caller attribution. When `nativeBalance >= value`, the function deposits and transfers ETH belonging to a prior user without pulling anything from `payer`. When `nativeBalance > 0` but less than `value`, it silently consumes the prior user's ETH first and only pulls the remainder from `payer`.

The router accumulates native ETH legitimately via `multicall{value: N}(...)`. The `multicall` function is `payable` and uses `delegatecall`, so `msg.value` is accessible across all steps but is **not** automatically refunded:

```solidity
function multicall(bytes[] calldata data) public payable override returns (bytes[] memory results) {
    results = new bytes[](data.length);
    for (uint256 i = 0; i < data.length; i++) {
        results[i] = Address.functionDelegateCall(address(this), data[i]);
    }
}
```

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` in `multicall`:

```solidity
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
```

`refundETH()` sends the entire balance to whoever calls it first, meaning Alice's refund is empty after Bob drains the balance via `pay()`.

The exploit path is:
1. Alice calls `multicall{value: 1 ether}(...)` with a swap consuming only 0.3 ETH of WETH, omitting `refundETH()`. 0.7 ETH remains on the router.
2. Bob calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = 0.7 ether`. The swap callback triggers `_justPayCallback` → `pay(WETH, bob, pool, 0.7 ether)`. `nativeBalance == 0.7 ETH >= value`, so the router deposits Alice's ETH as WETH and transfers it to the pool. Bob pays nothing from his own balance.
3. Alice calls `refundETH()` and receives 0 ETH.

No privileged access is required. Bob only needs to call a standard public swap function.

## Impact Explanation
Direct loss of user principal. Alice's stranded ETH is consumed to pay for Bob's WETH swap. Alice receives 0 ETH on refund. Bob's swap costs him zero WETH. The loss equals the amount of ETH consumed from the router on Bob's behalf, up to the full stranded balance. This is a direct, unprivileged theft of user funds meeting Sherlock High/Critical thresholds.

## Likelihood Explanation
The precondition — native ETH stranded on the router — occurs whenever a user sends `msg.value` in a `multicall` and omits `refundETH()`, includes it after a reverting step, or swaps less than `msg.value`. An attacker can monitor the mempool for such transactions and immediately follow with a WETH-input swap to drain the residue. The attack is repeatable, requires no special permissions, and can be automated.

## Recommendation
The `pay()` function must not consume ETH beyond what the current caller authorized. Concretely:
- Pass the caller-authorized native amount (e.g., `msg.value` captured at entry) explicitly into `pay()` as a parameter, and cap native ETH consumption to that amount rather than reading `address(this).balance`.
- Alternatively, track per-call ETH attribution in transient storage alongside the existing callback context, and only consume up to the attributed amount in `pay()`.

## Proof of Concept
1. Deploy router with a WETH/TokenA pool.
2. Alice: `router.multicall{value: 1 ether}([exactInputSingle(WETH→TokenA, amountIn=0.3e18)])` — no `refundETH` step. Router now holds 0.7 ETH.
3. Bob (zero WETH balance): `router.exactInputSingle(WETH→TokenA, amountIn=0.7e18)`. The swap callback calls `pay(WETH, bob, pool, 0.7e18)`. `address(this).balance == 0.7e18 >= value`, so the router deposits Alice's ETH and transfers WETH to the pool. Bob receives TokenA at zero cost.
4. Alice: `router.refundETH()` — receives 0 ETH.
5. Assert: Alice's net ETH loss = 0.7 ETH; Bob's net TokenA gain = expected swap output for 0.7 ETH.
