### Title
Unbounded `proof_facts_size` in OS Transaction Hash Computation Enables Proof-Capacity Exhaustion — (`execution/transaction_impls.cairo`)

### Summary

The StarkNet OS processes the `proof_facts` array during invoke transaction hash computation without enforcing any upper-bound on its length. Unlike `signature_len` and `calldata_size`, which are both explicitly bounded by `SIERRA_ARRAY_LEN_BOUND - 1`, `proof_facts_size` is loaded from a hint and passed directly into an unbounded Poseidon hash loop. A user who submits an invoke transaction with a sufficiently large `proof_facts` array can force the OS to consume an unbounded number of Poseidon builtin instances during its own (non-gas-metered) transaction hash computation, exhausting the proof's builtin capacity and making the block permanently unprovable.

### Finding Description

In `execute_invoke_function_transaction`, three user-controlled array sizes are loaded from hints:

```cairo
local proof_facts_size;
local proof_facts: felt*;
%{ TxProofFacts %}
``` [1](#0-0) 

Immediately after, the OS calls `compute_invoke_transaction_hash`, which passes `proof_facts_size` directly into `poseidon_hash_update_with_nested_hash`:

```cairo
if (proof_facts_size != 0) {
    poseidon_hash_update_with_nested_hash(
        data_ptr=proof_facts, data_length=proof_facts_size
    );
}
``` [2](#0-1) 

This hash loop iterates over every element of `proof_facts`, consuming approximately one Poseidon builtin instance per element. This computation is performed by the OS itself — it is **not** subject to the Sierra gas metering that limits contract execution.

Contrast this with the explicit bounds enforced on the other two user-controlled arrays in the same function:

```cairo
assert_nn_le(signature_len, SIERRA_ARRAY_LEN_BOUND - 1);
``` [3](#0-2) 

```cairo
assert_nn_le(tx_execution_context.calldata_size, SIERRA_ARRAY_LEN_BOUND - 1);
``` [4](#0-3) 

No equivalent check exists for `proof_facts_size`. The constant `SIERRA_ARRAY_LEN_BOUND = 2^32` is defined in `constants.cairo`: [5](#0-4) 

Even `SIERRA_ARRAY_LEN_BOUND - 1` (≈4 billion) would be catastrophic, but the absence of any check means the field is completely unconstrained at the OS level.

### Impact Explanation

The StarkNet proof has a fixed capacity for each builtin (e.g., Poseidon). A typical proof supports on the order of 2^20–2^24 Poseidon builtin instances. A transaction with `proof_facts_size` set to a value exceeding this capacity would cause the OS to attempt more Poseidon operations than the proof can accommodate. The block containing that transaction becomes permanently unprovable. Since the OS has already accepted and processed the transaction (the hash computation happens before `check_proof_facts`), the sequencer cannot simply skip it — the block is stuck. This constitutes a **network halt**: the network cannot confirm new transactions until the block is abandoned and re-built without the malicious transaction.

**Impact class**: High — Network not being able to confirm new transactions (total network shutdown).

### Likelihood Explanation

The `proof_facts` field is a live, user-submitted field in invoke transactions (used for client-side proving). An unprivileged transaction sender controls its content and length. If the gateway does not independently enforce a size limit on `proof_facts_size` (which the OS code does not mandate), the attacker only needs to submit a single well-formed invoke transaction with an oversized `proof_facts` array. The sequencer, seeing a valid-looking transaction, may include it in a block. The OS then exhausts its Poseidon builtin budget during hash computation, before any gas-metered execution begins.

### Recommendation

Add an explicit upper-bound assertion on `proof_facts_size` immediately after it is loaded from the hint, consistent with the pattern used for `signature_len` and `calldata_size`:

```cairo
local proof_facts_size;
local proof_facts: felt*;
%{ TxProofFacts %}
assert_nn_le(proof_facts_size, SIERRA_ARRAY_LEN_BOUND - 1);
```

Additionally, a tighter protocol-level bound (e.g., a few thousand elements) should be chosen based on the actual proof capacity budget allocated to OS-level hash computation, and enforced both here and at the gateway layer.

### Proof of Concept

1. Construct an invoke transaction (version 3) with a `proof_facts` array of length `N` where `N` exceeds the Poseidon builtin capacity of a single proof (e.g., `N = 2^22`).
2. Submit the transaction to the gateway. If the gateway does not validate `proof_facts_size`, it is accepted into the mempool.
3. The sequencer includes the transaction in a block.
4. The OS prover begins executing the block. In `execute_invoke_function_transaction`, it loads `proof_facts_size = N` from the hint and calls `compute_invoke_transaction_hash`, which calls `poseidon_hash_update_with_nested_hash` with `data_length = N`.
5. The Poseidon hash loop runs `N` iterations, each consuming one Poseidon builtin instance. The proof's Poseidon builtin capacity is exhausted.
6. The block cannot be proven. The network halts on this block until the sequencer abandons it and rebuilds without the malicious transaction — but the attacker can repeat the attack with the next block.
