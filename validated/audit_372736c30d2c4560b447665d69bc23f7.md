### Title
Gateway Skips `__validate__` for Invoke Transactions When Any Pending Deploy-Account Exists, Allowing Signature-Invalid Transactions Into the Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The ERC20 permit front-running pattern — where a signature is consumed before the legitimate caller uses it, causing the original call to fail — has a direct analog in the Sequencer's UX-skip path. When a `deploy_account` is pending in the mempool for address X, the gateway unconditionally skips the `__validate__` entry-point call (i.e., signature verification) for any incoming `invoke` with `nonce=1` from that address. An attacker who observes the pending deploy-account can submit an invoke with an **invalid signature** and a higher fee; the gateway admits it without verifying the signature, and fee escalation then evicts the legitimate user's valid invoke from the mempool.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` returns `true` (skip validation) when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0` (account not yet deployed)
3. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true` [1](#0-0) 

When `skip_validate = true` is returned, `run_pre_validation_checks` passes it to `run_validate_entry_point`, which sets `execution_flags.validate = false`: [2](#0-1) 

With `validate = false`, the blockifier's `validate_tx` returns `Ok(None)` immediately without calling the account contract's `__validate__` entry point: [3](#0-2) 

The gateway then forwards the transaction to the mempool. The mempool's `validate_tx` only checks nonce ordering and fee escalation — it never inspects the signature: [4](#0-3) 

Fee escalation (`handle_fee_escalation`) will replace an existing valid invoke at the same `(address, nonce)` if the attacker's transaction offers a sufficiently higher tip and max-gas-price: [5](#0-4) 

`account_tx_in_pool_or_recent_block` returns `true` as long as the account has **any** transaction in the pool or was seen in a recent committed block — not specifically a deploy-account: [6](#0-5) 

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject an invoke with an arbitrary (invalid) signature for any address that has a pending deploy-account. The gateway admits it without signature verification. Via fee escalation, the attacker's invalid invoke can evict the legitimate user's valid invoke from the mempool. When the block is built, the blockifier *does* call `__validate__`, the signature check fails, and the transaction reverts — but the legitimate user's invoke has already been displaced and is lost.

---

### Likelihood Explanation

The trigger is fully unprivileged and observable: pending deploy-account transactions are visible via the P2P network and the RPC mempool. The attacker only needs to:

1. Observe a pending deploy-account for address X (public information).
2. Craft an invoke with `nonce=1`, sender=X, any invalid signature, and a fee ≥ `existing_fee × (1 + fee_escalation_percentage/100)`.
3. Submit it to the gateway.

The default `fee_escalation_percentage` is 10%, so the attacker needs only a 10% fee premium to displace the victim's invoke. [7](#0-6) 

---

### Recommendation

Mirror the ERC20 permit mitigation: instead of unconditionally skipping `__validate__`, fall back gracefully. Specifically, attempt to run `__validate__` and only skip it if the account contract does not yet exist (i.e., `get_class_hash_at` returns the zero hash). This preserves the UX benefit for the legitimate deploy-account+invoke flow while preventing an attacker from injecting signature-invalid invokes.

Alternatively, restrict the skip to cases where the gateway can confirm the pending transaction for the address is specifically a `deploy_account` (not just any transaction), by adding a typed query to the mempool.

---

### Proof of Concept

1. Legitimate user submits `deploy_account` for address `X` (nonce=0). Gateway validates signature, admits to mempool.
2. Attacker observes the pending deploy-account via P2P/RPC.
3. Attacker submits `invoke` for address `X`, `nonce=1`, **garbage signature**, fee = victim's fee × 1.11.
4. Gateway calls `skip_stateful_validations`:
   - `tx.nonce() == Nonce(Felt::ONE)` ✓
   - `account_nonce == Nonce(Felt::ZERO)` ✓
   - `account_tx_in_pool_or_recent_block(X)` → `true` ✓
   - Returns `true` → `execution_flags.validate = false`
5. `run_validate_entry_point` skips `__validate__`. No signature check occurs.
6. `validate_by_mempool` passes (nonce ordering is valid, fee escalation threshold met).
7. Attacker's invalid invoke is admitted to the mempool and evicts the victim's valid invoke via fee escalation.
8. Block is built: blockifier executes the attacker's invoke, calls `__validate__`, signature fails, transaction reverts. Victim's invoke is gone. [8](#0-7) [9](#0-8)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    fn handle_fee_escalation(
        &mut self,
        incoming_tx_reference: TransactionReference,
        validation_only: bool,
    ) -> MempoolResult<()> {
        let TransactionReference { address, nonce, .. } = incoming_tx_reference;

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {
            if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
                return Err(MempoolError::DuplicateNonce { address, nonce });
            };

            return Ok(());
        }

        let Some(existing_tx_reference) = self.tx_pool.get_by_address_and_nonce(address, nonce)
        else {
            // Replacement irrelevant: no existing transaction with the same nonce for address.
            return Ok(());
        };

        if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
            info!(
                "{existing_tx_reference} was not replaced by {incoming_tx_reference} due to \
                 insufficient fee escalation."
            );
            // TODO(Elin): consider adding a more specific error type / message.
            return Err(MempoolError::DuplicateNonce { address, nonce });
        }

        if validation_only {
            return Ok(());
        }

        debug!("{existing_tx_reference} will be replaced by {incoming_tx_reference}.");

        self.tx_queue.remove_txs(&[existing_tx_reference]);
        self.tx_pool
            .remove(existing_tx_reference.tx_hash)
            .expect("Transaction hash from pool must exist.");

        Ok(())
    }
