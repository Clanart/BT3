### Title
Tokens Permanently Frozen When Transfer Amount Normalizes to Zero Before Signing - (File: near/omni-bridge/src/lib.rs)

### Summary

When a user initiates a NEAR-side bridge transfer with an amount that is smaller than the minimum representable unit on the destination chain, the tokens are burned/locked during `init_transfer_internal`, but `sign_transfer` subsequently panics with `ERR_INVALID_AMOUNT_TO_TRANSFER` because the normalized amount is 0. No user-callable cancellation path exists, leaving the tokens permanently irrecoverable.

### Finding Description

The NEAR bridge normalizes token amounts from NEAR-native precision to destination-chain precision using floor division: [1](#0-0) 

For a token registered with `origin_decimals = 24` and `decimals = 18` (a common NEAR token bridged to EVM), `diff_decimals = 6`, so any amount below `1_000_000` normalizes to `0`.

The `init_transfer` entry point only validates `fee < amount`, not that the amount will survive normalization: [2](#0-1) 

`init_transfer_internal` then burns or locks the full raw amount and stores the transfer message, returning `U128(0)` to the token contract (signalling all tokens consumed): [3](#0-2) 

Later, when a relayer calls `sign_transfer`, the normalized amount is computed and the function panics: [4](#0-3) 

Because `sign_transfer` panics, the transfer message is never removed from `pending_transfers`. The `sign_transfer_callback` only removes the message on a successful signing: [5](#0-4) 

There is no user-callable cancel or withdraw function anywhere in the contract. The tokens are permanently burned/locked.

### Impact Explanation

**Critical — Permanent freezing of user funds.** Any user who sends an amount below the minimum representable unit on the destination chain (e.g., fewer than `1_000_000` yoctoNEAR for a 24→18 decimal token) will have their tokens irreversibly burned or locked in the bridge with no recovery path. The transfer message persists in `pending_transfers` indefinitely, and every relayer attempt to sign it will revert.

### Likelihood Explanation

**Low.** The condition requires a token with a significant decimal gap (`origin_decimals > decimals`) and a user sending a sub-unit amount. However, NEAR tokens commonly use 24 decimals while EVM tokens use 18, making the 6-decimal gap the standard case. A user sending fewer than `0.000001` of such a token (e.g., a dust amount or a mistaken small transfer) triggers the freeze. No privileged access is required — any unprivileged bridge user can trigger this.

### Recommendation

Add a normalization check inside `init_transfer` (before tokens are burned/locked) to reject transfers whose amount would normalize to zero:

```rust
// In init_transfer, after building transfer_message and before init_transfer_internal:
if let Some(token_address) = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
) {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

Alternatively, add a user-callable `cancel_transfer` function that removes the pending transfer message and unlocks/re-mints the tokens back to the sender, so that even if a transfer gets stuck, the user can recover their funds.

### Proof of Concept

1. A token is registered with `origin_decimals = 24`, `decimals = 18` (standard NEAR→EVM setup). `diff_decimals = 6`.
2. User calls `ft_transfer_call` on the token contract, sending `amount = 500_000` (below `1_000_000`), with `fee = 0`.
3. `init_transfer` passes the `fee < amount` check (`0 < 500_000`).
4. `init_transfer_internal` burns/locks `500_000` yoctoNEAR and stores the transfer message. Returns `U128(0)` — tokens consumed.
5. Relayer calls `sign_transfer` for this transfer ID.
6. `normalize_amount(500_000, {origin_decimals:24, decimals:18}) = 500_000 / 1_000_000 = 0`.
7. `require!(0 > 0, ...)` panics — transaction reverts.
8. Transfer message remains in `pending_transfers`. Tokens are permanently burned. No recovery path exists. [6](#0-5) [7](#0-6) [8](#0-7)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }

    /// Uses floor division — any sub-unit remainder ("dust") is truncated and not transferred
    /// to the destination chain. When fee > 0, dust is absorbed into the fee via `claim_fee`.
    /// When fee = 0, dust stays locked/burned. See SECURITY.md for details.
    fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
    }
