### Title
`get_sortable_validator_online_ratio` Ignores `endorsement_cutoff_threshold`, Causing Wrong Validators to Be Exempted from Kickout — (`File: chain/epoch-manager/src/validator_stats.rs`)

### Summary

`get_sortable_validator_online_ratio` always calls `get_validator_online_ratio(stats, None)`, permanently ignoring the `endorsement_cutoff_threshold` that is applied during reward calculation. The exemption-from-kickout mechanism sorts validators by this ratio to decide who is protected from being kicked out. Because the sort key uses the raw endorsement ratio while the reward path uses a binarized (0 or 1) cutoff-adjusted ratio, the two paths rank validators in a different order. A validator whose raw endorsement ratio is just below the cutoff (e.g. 69 % when the cutoff is 70 %) is sorted as if it has 69 % endorsement performance, but is treated as 0 % for reward purposes. A competing validator with 71 % endorsement ratio is sorted as 71 % but treated as 100 % for rewards. Under the `validator_max_kickout_stake_perc` cap, the exemption mechanism can protect the lower-effective-performance validator and kick out the higher-effective-performance one — the opposite of the intended design.

### Finding Description

In `finalize_epoch` (`chain/epoch-manager/src/lib.rs`, lines 896–904), the reward calculator is called with `endorsement_cutoff_threshold: Some(epoch_config.chunk_validator_only_kickout_threshold)`. This binarizes each validator's endorsement contribution: below the threshold → 0, at or above → 1. [1](#0-0) 

In `compute_validators_to_reward_and_kickout` (`chain/epoch-manager/src/lib.rs`, lines 500–506), validators are sorted for the exemption mechanism using `get_sortable_validator_online_ratio`: [2](#0-1) 

`get_sortable_validator_online_ratio` unconditionally passes `None` to `get_validator_online_ratio`, discarding the cutoff entirely: [3](#0-2) 

`get_validator_online_ratio` with `None` uses the raw `(produced, expected)` endorsement pair, while with `Some(threshold)` it collapses the ratio to `(0,1)` or `(1,1)`: [4](#0-3) 

The exemption logic then uses the sort order to decide which validators are shielded from kickout: [5](#0-4) 

### Impact Explanation

Consider two validators with equal block/chunk production ratios (both below the kickout threshold, so both are candidates for kickout), but different endorsement ratios:

- **Validator A**: endorsement 69 % (below 70 % cutoff) → sort key uses 69 %, reward treats as 0 %
- **Validator B**: endorsement 71 % (above 70 % cutoff) → sort key uses 71 %, reward treats as 100 %

Sort order: B > A (correct). Both are kicked-out candidates. Under `validator_max_kickout_stake_perc`, only one can be exempted. The mechanism correctly exempts B.

Now swap the block/chunk ratios slightly so that A's block/chunk production is marginally higher than B's:

- **Validator A**: blocks 70 %, chunks 70 %, endorsement 69 % → sort average ≈ 69.7 %
- **Validator B**: blocks 68 %, chunks 68 %, endorsement 71 % → sort average ≈ 69.0 %

Sort order: A > B → A is exempted, B is kicked out.  
Reward order: B (78.7 %) > A (46.7 %) → B should be exempted.

The mechanism kicks out the validator with higher effective performance and retains the one with lower effective performance, breaking the economic model's intent that the best-performing validators survive.

### Likelihood Explanation

The `endorsement_cutoff_threshold` is active in every production epoch: `finalize_epoch` always sets it to `chunk_validator_only_kickout_threshold` (70 % on mainnet/testnet). The `validator_max_kickout_stake_perc` cap (30 % on mainnet) is routinely hit during network instability, which is exactly when the exemption mechanism fires. Any epoch where multiple validators fall below kickout thresholds and have endorsement ratios straddling the 70 % cutoff will trigger the wrong exemption decision. No privileged access is required; the endorsement stats are derived from ordinary validator behavior.

### Recommendation

Pass the active `endorsement_cutoff_threshold` into `get_sortable_validator_online_ratio` so that the sort key is computed with the same binarization used by the reward calculator:

```rust
// In compute_validators_to_reward_and_kickout, pass the cutoff:
pub(crate) fn get_sortable_validator_online_ratio(
    stats: &BlockChunkValidatorStats,
    endorsement_cutoff_threshold: Option<u8>,   // add parameter
) -> BigRational {
    let ratio = get_validator_online_ratio(stats, endorsement_cutoff_threshold);
    // ... rest unchanged
}
```

Call it as:
```rust
.map(|(account, stats)| (
    get_sortable_validator_online_ratio(stats, Some(chunk_validator_only_kickout_threshold)),
    account
))
```

This ensures the exemption sort order is consistent with the reward calculation, so the validators with the highest effective performance are always protected first.

### Proof of Concept

1. Configure an epoch with `chunk_validator_only_kickout_threshold = 70`, `validator_max_kickout_stake_perc = 30`, equal stake for all validators.
2. Create two validators A and B, both block+chunk producers, both below the block/chunk kickout threshold.
3. Set A's endorsement ratio to 69 % and B's to 71 %, but give A slightly higher block/chunk ratios so A's raw 3-way average exceeds B's.
4. Call `EpochManager::compute_validators_to_reward_and_kickout`.
5. Observe: A is exempted (higher sort key), B is kicked out.
6. Compute rewards with `endorsement_cutoff_threshold = Some(70)`: B's effective uptime is higher than A's.
7. The validator with lower effective reward performance (A) survives; the one with higher effective reward performance (B) is removed — the inverse of the intended design. [6](#0-5) [1](#0-0)

```rust
pub(crate) fn get_sortable_validator_online_ratio(stats: &BlockChunkValidatorStats) -> BigRational {
    let ratio = get_validator_online_ratio(stats, None);
    let mut bytes: [u8; size_of::<U256>()] = [0; size_of::<U256>()];
    ratio.numer().to_little_endian(&mut bytes);
    let bignumer = BigUint::from_bytes_le(&bytes);
    ratio.denom().to_little_endian(&mut bytes);
    let bigdenom = BigUint::from_bytes_le(&bytes);
    BigRational::new(bignumer.try_into().unwrap(), bigdenom.try_into().unwrap())
}
```

```rust
fn get_endorsement_ratio(stats: &ValidatorStats, cutoff_threshold: Option<u8>) -> (u64, u64) {
    let (numer, denom) = if stats.expected == 0 {
        debug_assert_eq!(stats.produced, 0);
        (0, 0)
    } else if let Some(threshold) = cutoff_threshold {
        if stats.less_than(threshold) { (0, 1) } else { (1, 1) }
    } else {
        (stats.produced, stats.expected)
    };
    (numer, denom)
}
```
