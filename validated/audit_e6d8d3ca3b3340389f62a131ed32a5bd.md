Let me look at the NEAR `deploy_token_callback` factory check and the wormhole prover emitter validation more carefully.

### Title
Unauthenticated Token-2022 Embedded Metadata Allows Fraudulent Token Identity Registration on NEAR — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

An unprivileged attacker can create a Token-2022 mint with arbitrary embedded `TokenMetadata` (e.g., `name="USD Coin"`, `symbol="USDC"`) and call `log_metadata`. The Solana bridge reads the name/symbol verbatim from attacker-controlled embedded metadata, emits a Wormhole VAA signed by the legitimate bridge emitter, and NEAR's `deploy_token` registers a new NEAR token with the fraudulent identity. No guard exists to authenticate or deduplicate name/symbol.

---

### Finding Description

**Step 1 — Solana `log_metadata` constraint is insufficient.**

The only guard on the mint account is:

```rust
constraint = !mint.mint_authority.contains(authority.key),
``` [1](#0-0) 

This rejects bridge-deployed tokens (whose `mint_authority` is the bridge PDA) but accepts any externally-created mint, including one crafted by an attacker.

**Step 2 — Embedded metadata is read without any validation.**

For Token-2022 mints where `metadata_pointer.metadata_address == mint.key()`, the code reads `name` and `symbol` directly from the embedded `TokenMetadata` extension:

```rust
if metadata_pointer.metadata_address.0 == self.mint.key() {
    // Embedded metadata
    let metadata = mint_with_extension.get_variable_len_extension::<TokenMetadata>()?;
    (metadata.name, metadata.symbol)
``` [2](#0-1) 

The attacker fully controls these strings. There is no check against a canonical registry, no uniqueness constraint, and no authority requirement on the metadata update authority.

**Step 3 — The payload is emitted verbatim via Wormhole.**

```rust
let payload = LogMetadataPayload {
    token: self.mint.key(),
    name: name.trim_end_matches('\0').to_string(),
    symbol: symbol.trim_end_matches('\0').to_string(),
    decimals: self.mint.decimals,
}
.serialize_for_near(())?;
self.common.post_message(payload)?;
``` [3](#0-2) 

The Wormhole emitter is the bridge's config PDA — the registered factory address — so the VAA carries the bridge's authority.

**Step 4 — NEAR `deploy_token_callback` only validates the emitter, not the metadata content.**

```rust
let chain = metadata.emitter_address.get_chain();
require!(
    self.factories.get(&chain) == Some(metadata.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
self.deploy_token_internal(chain, &metadata.token_address,
    BasicMetadata { name: metadata.name, symbol: metadata.symbol, decimals: metadata.decimals },
    attached_deposit,
)
``` [4](#0-3) 

The factory check passes because the emitter is the legitimate Solana bridge program. The attacker-supplied `name` and `symbol` flow directly into `deploy_token_internal` with no further validation.

**Step 5 — `deploy_token_internal` only deduplicates by mint address, not by name/symbol.**

```rust
require!(
    self.deployed_tokens.insert(&token_id),
    BridgeError::TokenExists.as_ref()
);
``` [5](#0-4) 

`token_id` is derived from the attacker's mint address prefix, which is unique. Multiple tokens with identical `name`/`symbol` can be registered as long as their mint addresses differ.

---

### Impact Explanation

A fraudulent NEAR token with `name="USD Coin"` and `symbol="USDC"` is permanently registered in the bridge's token mapping. Its NEAR account ID is derived from the attacker's Solana mint address (e.g., `<attacker_prefix>.token-deployer.near`), distinct from the real USDC token ID. Users who discover tokens by name/symbol (wallets, UIs, relayers) will encounter the fraudulent entry. Any user who bridges assets to or from this fraudulent token receives worthless attacker-minted tokens instead of USDC, constituting direct misdirection of value. The fraudulent registration is permanent and cannot be undone without DAO intervention.

---

### Likelihood Explanation

The attack requires only:
1. Creating a Token-2022 mint with embedded metadata (a standard Solana operation, no special privileges)
2. Calling the public `log_metadata` instruction
3. Relaying the resulting Wormhole VAA to NEAR

No privileged access, no key compromise, no colluding validators. The cost is minimal (Solana rent + Wormhole fee + NEAR storage deposit). The attack is fully executable on mainnet by any party.

---

### Recommendation

Add a name/symbol uniqueness check in `deploy_token_internal` (reject if any existing token shares the same `name` or `symbol`), or — more robustly — require that the Token-2022 mint's `metadata_update_authority` be the bridge's authority PDA before accepting embedded metadata as canonical. A simpler mitigation is to require an explicit allowlist of trusted mints (similar to how the bridge already maintains a `factories` allowlist) before accepting `log_metadata` for Token-2022 embedded metadata.

---

### Proof of Concept

```rust
// 1. Attacker creates Token-2022 mint with embedded metadata
let mint = create_token_2022_mint_with_embedded_metadata(
    &attacker_keypair,
    name: "USD Coin",
    symbol: "USDC",
    decimals: 6,
    metadata_update_authority: attacker_keypair.pubkey(),
);

// 2. Attacker calls log_metadata — passes the only guard because
//    mint_authority != bridge_authority_pda
bridge_token_factory::log_metadata(
    mint: mint.pubkey(),
    token_program: spl_token_2022::ID,
    ...
);

// 3. Wormhole VAA is emitted with:
//    emitter = bridge config PDA (registered factory)
//    payload.name = "USD Coin", payload.symbol = "USDC"

// 4. Relay VAA to NEAR deploy_token — factory check passes,
//    name/symbol accepted verbatim, new NEAR token registered
near_bridge.deploy_token(DeployTokenArgs {
    chain_kind: ChainKind::Sol,
    prover_args: wormhole_vaa_bytes,
});

// Result: NEAR token "<attacker_mint_prefix>.token-deployer.near"
//         with name="USD Coin", symbol="USDC" is now live in the bridge
```
