### Title
Unregistered ERC20 Token Permanently Locked in OmniBridge with No Refund Path — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`initTransfer` on the EVM side accepts any ERC20 token address with no check that the token is registered in NEAR's `token_decimals` map. Tokens are locked in the bridge contract, but NEAR's `fin_transfer_callback` panics with `TokenDecimalsNotFound` for unregistered tokens, meaning no MPC signature is ever produced and no `finTransfer` can ever release the tokens on EVM. There is no on-chain refund or rescue mechanism.

### Finding Description

**Step 1 — EVM `initTransfer` has no token registration guard.**

The `else` branch at lines 406–411 of `OmniBridge.sol` executes for any `tokenAddress` that is not in `customMinters` and not in `isBridgeToken`. It unconditionally pulls tokens from the caller and locks them in the contract:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
```

There is no check that the token has been registered via `logMetadata` + NEAR-side `deployToken`, or via `addCustomToken`. Any ERC20 passes through. [1](#0-0) 

**Step 2 — NEAR `fin_transfer_callback` panics for unregistered tokens.**

When a relayer submits proof of the EVM `InitTransfer` event, `fin_transfer_callback` on NEAR looks up `token_decimals` for the token address. If the token was never registered, `near_expect` panics:

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
```

This panic reverts all state changes in the callback. No `TransferMessage` is stored, no MPC signing is triggered, and no `SignTransferEvent` is ever emitted. [2](#0-1) 

**Step 3 — No EVM refund path exists.**

`finTransfer` on EVM requires a valid ECDSA signature from `nearBridgeDerivedAddress` (the MPC key). Since NEAR never processes the transfer, no signature is ever produced. The `finTransfer` function will always revert with `InvalidSignature` for this transfer. There is no admin rescue function, no emergency withdrawal, and no timeout-based refund in `OmniBridge.sol`. [3](#0-2) 

### Impact Explanation

ERC20 tokens locked via `initTransfer` for an unregistered token are permanently irrecoverable unless an admin manually registers the token on NEAR (via `add_deployed_tokens` or equivalent) and a relayer retries `fin_transfer`. If the admin declines or the token is not a valid bridge token, the funds are frozen forever. This matches the Critical impact: **permanent freezing / irrecoverable lock of user funds in bridge flows**. [4](#0-3) 

### Likelihood Explanation

The `logMetadata` function is permissionless and must be called before `initTransfer` for native EVM tokens, but this ordering is not enforced on-chain. A user who calls `initTransfer` before `logMetadata` is processed by NEAR, or who uses a token that NEAR operators choose not to register, will permanently lose funds. This is a realistic accidental path, not just a theoretical one. [5](#0-4) 

### Recommendation

Add a registration guard in `initTransfer` for the native ERC20 lock path. Before executing `safeTransferFrom`, verify that the token has a known NEAR mapping (e.g., `ethToNearToken[tokenAddress]` is non-empty, or a dedicated `isRegisteredToken` mapping is checked). Alternatively, emit a revert if the token is not registered, preventing the lock from occurring.

### Proof of Concept

1. Deploy or use any ERC20 token `T` that has never been passed through `logMetadata` + NEAR `deployToken`.
2. Approve `OmniBridge` to spend `amount` of `T`.
3. Call `OmniBridge.initTransfer(T, amount, 0, 0, "victim.near", "")`.
4. Observe: `safeTransferFrom` succeeds, `amount` of `T` is now held by `OmniBridge`, `InitTransfer` event is emitted.
5. Relayer submits proof to NEAR `fin_transfer`. NEAR's `fin_transfer_callback` panics at `token_decimals.get(&init_transfer.token).near_expect(BridgeError::TokenDecimalsNotFound)`.
6. No MPC signature is produced. No `SignTransferEvent` is emitted.
7. Attempt `OmniBridge.finTransfer(...)` on EVM — reverts with `InvalidSignature` because no valid signature exists.
8. Assert: `T.balanceOf(OmniBridge) == amount`, user balance is zero, no recovery path exists on-chain. [6](#0-5) [7](#0-6)
