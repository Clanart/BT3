### Title
ERC1155 Tokens Permanently Locked When `initTransfer1155` Is Called Without Prior `logMetadata1155` Registration — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`initTransfer1155` accepts and locks ERC1155 tokens into the bridge without enforcing that `logMetadata1155` was called first to populate the `multiTokens[deterministicToken]` mapping. When `finTransfer` is later invoked to release those tokens back to the user, the missing mapping causes the function to fall through to the ERC20 code path, which calls `IERC20(deterministicToken).safeTransfer(...)` on a non-contract address, reverting permanently. The ERC1155 tokens are irrecoverably locked.

---

### Finding Description

`OmniBridge` supports ERC1155 bridging through two separate, independent functions:

1. **`logMetadata1155(tokenAddress, tokenId)`** — registers the `(tokenAddress, tokenId)` pair by writing to `multiTokens[deterministicToken]` and emitting a `LogMetadata` event so the NEAR side learns about the token.
2. **`initTransfer1155(tokenAddress, tokenId, amount, ...)`** — pulls ERC1155 tokens from the caller into the bridge and emits `InitTransfer` using `deterministicToken` as the token identifier.

`initTransfer1155` contains **no check** that `multiTokens[deterministicToken]` is already populated: [1](#0-0) 

The `multiTokens` mapping is **only** written inside `logMetadata1155`: [2](#0-1) 

When `finTransfer` is later called to release the tokens, it reads `multiTokens[payload.tokenAddress]`. If the mapping was never set, `multiToken.tokenAddress == address(0)`, so the ERC1155 branch is skipped and execution falls through to the ERC20 fallback: [3](#0-2) 

`deterministicToken` is computed as `address(bytes20(keccak256(abi.encodePacked(tokenAddress, tokenId))))`: [4](#0-3) 

This address is a hash-derived value with no deployed bytecode. `IERC20(deterministicToken).safeTransfer(...)` will always revert, making the locked ERC1155 tokens unrecoverable.

The existing test suite confirms `initTransfer1155` is callable without `logMetadata1155`: [5](#0-4) 

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds.**

Once `initTransfer1155` is called without a prior `logMetadata1155`:

- ERC1155 tokens are transferred into the bridge contract.
- The NEAR side receives an `InitTransfer` event for `deterministicToken`, but because no `LogMetadata` event was ever emitted for that address, the NEAR bridge has no registered decimals for the token and cannot finalize the transfer.
- Every subsequent `finTransfer` call on EVM for this token reverts because `deterministicToken` is not an ERC20 contract.
- There is no cancel, refund, or rescue path in the EVM contract for locked ERC1155 tokens.

Calling `logMetadata1155` retroactively fixes the EVM-side mapping but cannot retroactively cause the NEAR side to re-process an already-failed or unprocessable `InitTransfer` event, leaving the tokens stranded.

---

### Likelihood Explanation

Any unprivileged user can call `initTransfer1155` directly. The function is public, unguarded by access control, and imposes no prerequisite check. A user who is unaware of the required two-step flow (`logMetadata1155` → `initTransfer1155`) will lose their tokens. The protocol documentation or UI could easily omit this ordering requirement, and the contract itself provides no enforcement or revert to guide the caller.

---

### Recommendation

Add a prerequisite guard inside `initTransfer1155` that reverts if the `multiTokens` mapping for the derived address has not been populated:

```solidity
function initTransfer1155(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    ...
    address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);

    // Enforce that logMetadata1155 was called first
    if (multiTokens[deterministicToken].tokenAddress == address(0)) {
        revert ERC1155MappingMismatch(); // or a dedicated error
    }

    IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "");
    ...
}
```

Alternatively, `initTransfer1155` can call the registration logic inline (as `logMetadata1155` does) so the mapping is always set atomically with the token lock.

---

### Proof of Concept

1. Deploy `OmniBridge` and an ERC1155 token. Mint tokens to `user`.
2. User approves the bridge and calls `initTransfer1155(erc1155, tokenId, 1, 0, 0, "user.near", "")` **without** calling `logMetadata1155` first.
3. Confirm `erc1155.balanceOf(bridge, tokenId) == 1` — tokens are locked.
4. Confirm `bridge.multiTokens(deterministicToken).tokenAddress == address(0)` — mapping is empty.
5. Construct a valid MPC-signed `TransferMessagePayload` with `tokenAddress = deterministicToken` and call `bridge.finTransfer(sig, payload)`.
6. Observe the call reverts at `IERC20(deterministicToken).safeTransfer(...)` because `deterministicToken` has no bytecode.
7. The nonce is not consumed (transaction reverted), but every retry produces the same revert. Tokens are permanently locked. [6](#0-5) [7](#0-6)
