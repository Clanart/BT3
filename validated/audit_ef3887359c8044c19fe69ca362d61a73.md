### Title
Cross-EVM-Chain Signature Replay in `deployToken` Enables Unauthorized Token Deployment and Permanent Bridging DoS — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `deployToken` function in `OmniBridge.sol` constructs a signature digest that contains only token metadata fields and omits any chain-specific context (`omniBridgeChainId` or contract address). Because the same `nearBridgeDerivedAddress` (derived from the NEAR MPC key) is used across every EVM deployment (Ethereum, Arbitrum, Base, BNB, Polygon, HyperEvm, Abstract), a valid `deployToken` signature observed on one EVM chain can be replayed verbatim on every other EVM chain. This allows an unprivileged attacker to deploy bridge tokens on chains NEAR has not yet authorized, and — critically — to permanently block NEAR from ever deploying the legitimate token on those chains.

---

### Finding Description

`OmniBridge.sol::deployToken` builds its verification digest as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);

if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

The digest contains **no chain identifier**. Contrast this with `finTransfer`, which embeds `omniBridgeChainId` twice (once for the token address field, once for the recipient field):

```solidity
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.tokenAddress),
...
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.recipient),
``` [2](#0-1) 

The `nearBridgeDerivedAddress` is a single NEAR MPC-derived key shared by all EVM chain deployments: [3](#0-2) 

Because the digest is chain-agnostic and the signer is identical across all EVM chains, a signature produced by NEAR for deploying token `X` on Ethereum is cryptographically valid on Arbitrum, Base, BNB, Polygon, HyperEvm, and Abstract.

After a successful replay, the token is registered in the target chain's mappings:

```solidity
isBridgeToken[address(bridgeTokenProxy)] = true;
ethToNearToken[address(bridgeTokenProxy)] = metadata.token;
nearToEthToken[metadata.token] = address(bridgeTokenProxy);
``` [4](#0-3) 

When NEAR later attempts to deploy the same token on the target chain, it hits:

```solidity
require(
    !isBridgeToken[nearToEthToken[metadata.token]],
    "ERR_TOKEN_EXIST"
);
``` [5](#0-4) 

This check permanently prevents NEAR from deploying the legitimate token on that chain.

---

### Impact Explanation

**Primary — Unauthorized token deployment (High):** An attacker can deploy any bridge token on any EVM chain by replaying a signature that NEAR produced for a different chain. No NEAR authorization for the target chain is required.

**Secondary — Permanent bridging DoS (Critical):** Because `nearToEthToken[metadata.token]` is set to the attacker's proxy address (which differs from the address NEAR would have computed), NEAR's own subsequent `deployToken` call on that chain reverts with `ERR_TOKEN_EXIST`. NEAR can never register the correct token address for that chain, permanently blocking all bridging of that token to the affected chain. Users who hold the token on NEAR and wish to bridge it to the affected chain are permanently frozen out.

---

### Likelihood Explanation

The attack is straightforward and requires no privileged access:

1. Monitor any EVM chain for a `DeployToken` event and extract the `signatureData` from the corresponding transaction calldata — this is fully public.
2. Call `deployToken` on any other EVM OmniBridge deployment with the same `signatureData` and `metadata` before NEAR deploys there.

NEAR deploys tokens sequentially across chains; the window between the first deployment and subsequent chains is observable and exploitable by any on-chain observer. The attacker needs only a small amount of gas on the target chain.

---

### Recommendation

Include `omniBridgeChainId` in the `deployToken` signature digest, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // ← add chain binding
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

NEAR's signing logic must be updated in parallel to include the destination chain ID when producing `deployToken` signatures, so that each signature is bound to exactly one chain.

### Proof of Concept

```
Setup:
  - OmniBridgeEth  deployed on Ethereum  (omniBridgeChainId = 0, nearBridgeDerivedAddress = NEAR_KEY)
  - OmniBridgeArb  deployed on Arbitrum  (omniBridgeChainId = 3, nearBridgeDerivedAddress = NEAR_KEY)

Step 1: NEAR signs and deploys token "usdc.near" on Ethereum.
  Tx calldata on Ethereum contains: (signatureData_S, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})
  Digest signed by NEAR = keccak256(0x01 || borsh("usdc.near") || borsh("USD Coin") || borsh("USDC") || 0x06)
  → No chain ID in digest.

Step 2: Attacker observes signatureData_S from the Ethereum transaction.

Step 3: Attacker calls OmniBridgeArb.deployToken(signatureData_S, {token:"usdc.near",...}) on Arbitrum.
  Digest recomputed on Arbitrum = keccak256(0x01 || borsh("usdc.near") || borsh("USD Coin") || borsh("USDC") || 0x06)
  → Identical digest. ECDSA.recover returns NEAR_KEY. Signature accepted.
  → "usdc.near" is now registered in OmniBridgeArb at attacker-chosen proxy address A_arb.

Step 4: NEAR later tries to deploy "usdc.near" on Arbitrum.
  → nearToEthToken["usdc.near"] is already set to A_arb.
  → isBridgeToken[A_arb] == true.
  → require(!isBridgeToken[nearToEthToken["usdc.near"]], "ERR_TOKEN_EXIST") REVERTS.
  → NEAR can never deploy the legitimate token on Arbitrum.
  → All NEAR→Arbitrum bridging of USDC is permanently blocked.
```
