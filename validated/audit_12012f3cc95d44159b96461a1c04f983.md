### Title
EIP-7702 Authorization Durably Committed to Cosmos State Even When EVM Execution Reverts or Post-Hook Fails — Unauthorized Mutation of Victim Account's Nonce and Code - (File: `x/evm/keeper/state_transition.go`)

---

### Summary

In `ApplyTransaction`, when EVM hooks are registered, a `durableAuthorizationCtx` cache branch is created and `commitDurableAuthorization` is called whenever `tmpCtxCommitted` is `false` — which includes the case where the EVM execution itself reverted (`vmErr != nil`, `res.Failed() == true`). This means an unprivileged outer transaction sender can force a permanent nonce increment and delegation-code installation on any victim EOA whose private key signed an EIP-7702 authorization tuple, even if the EVM call that was supposed to use that delegation reverts entirely.

---

### Finding Description

In `ApplyTransaction` (`x/evm/keeper/state_transition.go`), when `k.hooks != nil` and the transaction carries `SetCodeAuthorizations`, two sibling cache contexts are created:

```
tmpCtx,  commit                  = ctx.CacheContext()   // main EVM execution
durableAuthorizationCtx, commitDurableAuthorization = ctx.CacheContext()   // auth effects
``` [1](#0-0) 

`ApplyMessageWithConfig` is called on `tmpCtx`. Inside it, before `evm.Call` is invoked, every valid authorization in the list is applied to the main `stateDB` **and** replayed into `durableStateDB` (backed by `durableAuthorizationCtx`), which is immediately committed:

```go
if commit && cfg.DurableSetCodeAuthorizationCtx != nil && len(validAuths) > 0 {
    durableStateDB := statedb.NewWithParams(*cfg.DurableSetCodeAuthorizationCtx, ...)
    for _, va := range validAuths {
        k.applyDurableAuthorization(&va.auth, va.authority, durableStateDB)
    }
    if err := durableStateDB.Commit(); err != nil { ... }
}
``` [2](#0-1) 

`applyDurableAuthorization` calls `setAuthorizationDelegation`, which unconditionally bumps the authority's nonce and installs delegation code:

```go
stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), ...)
``` [3](#0-2) 

Back in `ApplyTransaction`, after `ApplyMessageWithConfig` returns, the commit decision is:

```go
if commitDurableAuthorization != nil && !tmpCtxCommitted {
    commitDurableAuthorization()
}
``` [4](#0-3) 

`tmpCtxCommitted` is only set to `true` when `PostTxProcessing` succeeds **and** `commit()` is called (line 241-242). In all other cases — including when `res.Failed()` is `true` (EVM reverted) — `tmpCtxCommitted` remains `false`, so `commitDurableAuthorization()` is called unconditionally, permanently writing the authority's nonce bump and delegation code to the parent Cosmos KV store. [5](#0-4) 

The design intent (documented in comments and tests) is that authorization effects should survive a *post-hook* failure, because EIP-7702 specifies that valid authorizations are consumed even when the EVM call reverts. However, the current condition `!tmpCtxCommitted` is too broad: it fires even when `tmpCtxCommitted` is `false` because the EVM execution itself reverted (not just because of a post-hook failure). The test `TestSetCodeAuthorizationSurvivesFailedExecutionWithAndWithoutHooks` explicitly asserts this behavior as intentional, but it creates the attack surface described below. [6](#0-5) 

---

### Impact Explanation

An unprivileged attacker who obtains a signed EIP-7702 authorization tuple from a victim (e.g., a tuple the victim signed for a legitimate future use, or one obtained via a phishing/social-engineering vector) can:

1. Craft an EIP-7702 type-4 transaction whose `authorizationList` contains the victim's signed tuple but whose `to` target is a contract that immediately reverts (`REVERT` opcode).
2. Submit this transaction. The EVM call reverts, so no EVM state changes persist in `tmpCtx`. However, `commitDurableAuthorization()` is still called, permanently writing to the Cosmos KV store:
   - The victim authority's **account nonce is incremented** by 1 (consuming the authorization, preventing its legitimate future use).
   - The victim authority's **account code is set** to the EIP-7702 delegation prefix pointing to the attacker-chosen `auth.Address`.

This constitutes unauthorized mutation of another account's nonce and code without the victim's consent to the specific transaction outcome. The victim's EOA is now delegated to an arbitrary contract address chosen by the attacker, and the authorization is consumed so the victim cannot revoke it with the same nonce.

The impact maps to: **High — Ethereum transaction EIP-7702 authorization bypass enabling forged execution or unauthorized account/code mutation.**

---

### Likelihood Explanation

- Entry path is a standard unprivileged EIP-7702 type-4 transaction submitted via the normal JSON-RPC or Cosmos transaction broadcast path.
- No privileged role, validator collusion, or governance action is required.
- The attacker only needs a valid signed authorization tuple from the victim (the victim's signature on `{chainId, address, nonce}`), which could be obtained through social engineering, a malicious dApp, or a replay of a previously broadcast but not-yet-consumed authorization.
- The attack is cheap: the attacker pays gas for a transaction that reverts, but the authorization effect is permanent.
- Likelihood is **Medium-High** given that EIP-7702 is a new feature and users may sign authorizations for future use without expecting them to be consumed by a third-party-crafted reverting transaction.

---

### Recommendation

The `commitDurableAuthorization` call should only fire when the EVM execution itself succeeded (i.e., `!res.Failed()`) but the post-hook failed, not when the EVM call reverted. The condition should be changed from:

```go
if commitDurableAuthorization != nil && !tmpCtxCommitted {
    commitDurableAuthorization()
}
```

to:

```go
// Only commit durable authorization when the EVM execution succeeded
// but the post-hook failed (tmpCtx was not committed due to hook failure).
// Do NOT commit when the EVM call itself reverted.
if commitDurableAuthorization != nil && !tmpCtxCommitted && !res.Failed() {
    commitDurableAuthorization()
}
```

This aligns with EIP-7702's specification: authorizations are consumed when the EVM processes them, but in Ethermint's Cosmos context, the "durable" path exists only to handle the Cosmos-specific post-hook failure scenario. When the EVM itself reverts, the authorization should not be durably committed either, since the entire transaction's EVM effects are discarded. [4](#0-3) 

---

### Proof of Concept

1. Victim `V` signs an EIP-7702 authorization: `auth = Sign(V.key, {chainId, delegateAddr, nonce=N})`.
2. Attacker `A` deploys a contract `RevertTarget` containing only `REVERT` bytecode.
3. Attacker submits a type-4 EIP-7702 transaction:
   - `from`: attacker's address
   - `to`: `RevertTarget`
   - `authorizationList`: `[auth]` (victim's signed tuple)
4. `ApplyTransaction` is called with `k.hooks != nil`.
5. `durableAuthorizationCtx` is created; `cfg.DurableSetCodeAuthorizationCtx` is set.
6. Inside `ApplyMessageWithConfig`, `applyAuthorization` validates `auth` (victim's nonce matches, no non-delegation code), bumps victim's nonce in `stateDB`, and replays into `durableStateDB` which is immediately committed to `durableAuthorizationCtx`.
7. `evm.Call` executes `RevertTarget` → `vmErr = ErrExecutionReverted`, `res.Failed() = true`.
8. Back in `ApplyTransaction`: `res.Failed()` is `true`, so `PostTxProcessing` is not called, `commit()` is not called, `tmpCtxCommitted = false`.
9. `commitDurableAuthorization != nil && !tmpCtxCommitted` → `commitDurableAuthorization()` is called.
10. Victim `V`'s nonce is now `N+1` and their code is `0xef0100 || delegateAddr` in the committed Cosmos state.
11. The victim's original authorization is consumed; any legitimate transaction `V` planned to use it for will now fail with `ErrAuthorizationNonceMismatch`.

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}
```
