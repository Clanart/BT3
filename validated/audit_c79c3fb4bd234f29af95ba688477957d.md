### Title
Meta-Transaction v0 Hash Omits Nonce, Enabling Unlimited Signature Replay — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall constructs a transaction hash via `compute_meta_tx_v0_hash` that commits to no nonce, no deadline, and no block number. Combined with the OS explicitly skipping nonce tracking for version-0 transactions in `check_and_increment_nonce`, any meta-tx-v0 signature is permanently valid and can be replayed by any unprivileged actor who has observed it on-chain.

---

### Finding Description

`execute_meta_tx_v0` in `syscall_impls.cairo` computes the meta-transaction hash by calling `compute_meta_tx_v0_hash`:

```cairo
let meta_tx_hash = compute_meta_tx_v0_hash(
    contract_address=contract_address,
    entry_point_selector=selector,
    calldata=calldata_start,
    calldata_size=calldata_size,
    chain_id=old_tx_info.chain_id,
);
``` [1](#0-0) 

`compute_meta_tx_v0_hash` passes `additional_data_size=0` and `additional_data=cast(0, felt*)` to the underlying hash function — meaning **no nonce is committed to in the hash**:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(...) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,
        additional_data=cast(0, felt*),
    );
    return tx_hash;
}
``` [2](#0-1) 

The resulting `TxInfo` hardcodes `nonce=0`:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,
    ...
);
``` [3](#0-2) 

The OS nonce-increment gate in `check_and_increment_nonce` explicitly returns early for version-0 transactions, so no on-chain nonce state is ever updated:

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
``` [4](#0-3) 

Similarly, `run_validate` skips `__validate__` for version 0, so no account-level replay guard runs either: [5](#0-4) 

The hash is therefore a pure deterministic function of `(contract_address, selector, calldata, chain_id)`. Once a signature over this hash is broadcast, it is valid forever with no mechanism to cancel or invalidate it.

---

### Impact Explanation

**Critical — Direct loss of funds.**

Any contract that uses the `meta_tx_v0` syscall to authorize a value-moving operation (e.g., ERC-20 `transfer`, ownership change, withdrawal) produces a signature that is permanently replayable. An attacker who observes the signature in any historical transaction can re-submit the same meta-tx-v0 in a new transaction, causing the target contract's `__execute__` to run again with the same calldata and the same verified signature. This can be repeated indefinitely, draining the victim's funds with no recourse.

---

### Likelihood Explanation

**Medium.**

The `meta_tx_v0` syscall is a protocol-level feature callable from any contract. All transaction calldata — including the signature passed in `MetaTxV0Request` — is publicly visible on-chain. Any observer of a historical block can extract the signature and replay it. No privileged access, leaked key, or social engineering is required; the attacker only needs to read public chain data and submit a new transaction.

---

### Recommendation

Include a nonce (or equivalent replay-prevention field such as a block number or monotonic counter) in the `compute_meta_tx_v0_hash` computation by passing it as `additional_data`. The OS should track and increment a per-account meta-tx nonce in `contract_state_changes`, analogous to how `check_and_increment_nonce` works for version-1/3 transactions. This ensures each meta-tx-v0 signature is single-use and can be invalidated by the account owner by advancing the nonce.

---

### Proof of Concept

1. **Victim** calls a contract that internally issues a `meta_tx_v0` syscall authorizing `transfer(attacker, 1000_tokens)` on a token contract. The OS computes `H = Pedersen(INVOKE_PREFIX, 0, token_contract, __execute__, calldata, 0, chain_id)` with no nonce. The victim's signature `S` over `H` is included in the transaction calldata and is publicly visible on-chain.

2. **Attacker** reads `S` and `calldata` from the historical transaction.

3. **Attacker** submits a new transaction to any contract that calls the `meta_tx_v0` syscall with `contract_address=token_contract`, `selector=__execute__`, the same `calldata`, and `signature=S`.

4. The OS recomputes the identical `H` (same inputs, no nonce), the signature verifies, `check_and_increment_nonce` is skipped (version=0), and `transfer(attacker, 1000_tokens)` executes again.

5. Step 3–4 can be repeated in every subsequent block, draining the victim's token balance.
