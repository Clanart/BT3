### Title
Missing Duplicate-Hash Guard in `MerkleBlob::upsert` Corrupts `leaf_hash_to_index` Bijection — (`crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::upsert` updates a leaf's hash without checking whether the new hash is already held by a different leaf. `insert` performs this check explicitly and returns `HashAlreadyPresent`, but `upsert` omits it entirely. The result is that `leaf_hash_to_index` silently loses one leaf's entry, two blob leaves share the same hash, and the DataLayer Merkle tree enters a permanently corrupt state that enables proof forgery.

---

### Finding Description

`insert` guards against duplicate hashes at lines 372–374: [1](#0-0) 

`upsert` performs no equivalent check. Its update path is: [2](#0-1) 

Step-by-step cache mutation for `upsert(key1, val1, H2)` when `key2` already owns `H2`:

1. **`remove_leaf(&leaf)`** (line 798): removes `H1 → idx1` from `leaf_hash_to_index` and adds `idx1` to `free_indexes`. [3](#0-2) 

2. **`leaf.hash = H2`** (line 799): the in-memory leaf now carries `H2`.

3. **`insert_entry_to_blob`** (line 803) calls `add_leaf`, which does `leaf_hash_to_index.insert(H2, idx1)`: [4](#0-3) 

`HashMap::insert` silently overwrites the existing `H2 → idx2` entry. After the call:

| Structure | State |
|---|---|
| `leaf_hash_to_index` | `{H2 → idx1}` — only 1 entry for 2 leaves |
| `key_to_index` | `{key1 → idx1, key2 → idx2}` — correct |
| Blob leaf at `idx1` | hash = H2 |
| Blob leaf at `idx2` | hash = H2 (unchanged) |

`check_integrity` would detect this at lines 868–874 (leaf count 2 ≠ cache length 1): [5](#0-4) 

but `check_integrity` is not called on every mutation in production.

---

### Impact Explanation

- **`get_node_by_hash(H2)`** (lines 1198–1207) returns `key1`'s `(KeyId, ValueId)` because `leaf_hash_to_index[H2] = idx1`. `key2`'s leaf is permanently unreachable via hash lookup. [6](#0-5) 

- **`get_proof_of_inclusion(key2)`** produces a proof with `node_hash = H2` — identical to `key1`'s proof node hash. Both proofs are structurally valid against the (now-corrupt) root, but they prove different key-value pairs under the same leaf hash, enabling cross-key proof substitution. [7](#0-6) 

- The tree root hash is recomputed from the actual blob bytes (both leaves carry H2), so the root silently reflects the corrupted state — no error is surfaced to callers.

This matches the **High** impact category: *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.*

---

### Likelihood Explanation

`upsert` is a `pub` method whose `new_hash` parameter is fully caller-controlled. The Python `pyclass` binding on `MerkleBlob` exposes it directly to Python callers. Any DataLayer participant that can issue an `upsert` call with a hash already present in the tree can trigger the corruption deterministically — no hash collision, no privileged access, no key material required.

---

### Recommendation

Add the same guard that `insert` already has, immediately after the existing-key lookup succeeds (i.e., on the update path):

```rust
pub fn upsert(&mut self, key: KeyId, value: ValueId, new_hash: &Hash) -> Result<(), Error> {
    let Ok((leaf_index, mut leaf, mut block)) = self.get_leaf_by_key(key) else {
        self.insert(key, value, new_hash, InsertLocation::Auto {})?;
        return Ok(());
    };

    // NEW: reject if new_hash is already owned by a *different* leaf
    if leaf.hash != *new_hash && self.block_status_cache.contains_leaf_hash(new_hash) {
        return Err(Error::HashAlreadyPresent());
    }

    self.block_status_cache.remove_leaf(&leaf)?;
    leaf.hash.clone_from(new_hash);
    ...
}
```

The guard must be placed **before** `remove_leaf` so that `contains_leaf_hash` still sees the full, unmodified map.

### Proof of Concept

```rust
use chia_datalayer::{Hash, KeyId, MerkleBlob, ValueId};

let mut blob = MerkleBlob::new(vec![]).unwrap();
let h1 = Hash::from([1u8; 32]);
let h2 = Hash::from([2u8; 32]);
let key1 = KeyId(1);
let key2 = KeyId(2);

// Step 1 & 2: insert two leaves with distinct hashes
blob.insert(key1, ValueId(10), &h1, InsertLocation::Auto {}).unwrap();
blob.insert(key2, ValueId(20), &h2, InsertLocation::Auto {}).unwrap();

// Step 3: upsert key1 with H2 — no error returned
blob.upsert(key1, ValueId(10), &h2).unwrap();

// Invariant violated: get_node_by_hash(H2) returns key1, not key2
let (k, _) = blob.get_node_by_hash(h2).unwrap();
assert_eq!(k, key1); // key2's entry is silently lost

// check_integrity exposes the corruption
blob.check_integrity().unwrap_err(); // leaf_count(2) != leaf_hash_to_index_cache_length(1)
```
