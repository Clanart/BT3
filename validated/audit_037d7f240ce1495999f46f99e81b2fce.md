### Title
`deprecated_declared_classes` Unconditionally Dropped from State Diff Commitment, Producing Wrong Block Hash and `state_diff_length` — (File: `crates/blockifier/src/state/cached_state.rs`)

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` conversion hardcodes `deprecated_declared_classes: Vec::new()`. This `ThinStateDiff` is the sole input to `calculate_block_commitments` in `BlockExecutionArtifacts::new()`. For any block that includes a deprecated (Cairo 0, V0/V1) class declaration, the resulting `state_diff_commitment` (Poseidon hash) and `state_diff_length` embedded in `concatenated_counts` are both wrong, yielding an incorrect `PartialBlockHashComponents` and therefore an incorrect block hash.

### Finding Description

**Root cause — the unconditional drop:**

`CommitmentStateDiff` has no `deprecated_declared_classes` field. When a V0/V1 `DeclareTransaction` is executed, the blockifier records the class hash only in `StateMaps.declared_contracts` (a `HashMap<ClassHash, bool>`). The conversion `CommitmentStateDiff::from(StateMaps)` copies only `compiled_class_hashes`, so the deprecated class hash is silently absent from `CommitmentStateDiff`.

The subsequent conversion to `ThinStateDiff` then hardcodes the field to empty:

```rust
// crates/blockifier/src/state/cached_state.rs  L690-L701
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            deployed_contracts: commitment_state_diff.address_to_class_hash,
            storage_diffs: commitment_state_diff.storage_updates,
            class_hash_to_compiled_class_hash: commitment_state_diff
                .class_hash_to_compiled_class_hash,
            nonces: commitment_state_diff.address_to_nonce,
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),   // ← always empty
        }
    }
}
``` [1](#0-0) 

**Propagation into the block commitment:**

`BlockExecutionArtifacts::new()` passes this incomplete `ThinStateDiff` directly to `calculate_block_commitments`:

```rust
// crates/apollo_batcher/src/block_builder.rs  L160-L166
let (header_commitments, measurements) = calculate_block_commitments(
    &transactions_data,
    ThinStateDiff::from(commitment_state_diff.clone()),  // deprecated_declared_classes = []
    l1_da_mode,
    &block_info.starknet_version,
).await;
``` [2](#0-1) 

Inside `calculate_block_commitments`, two values are derived from the incomplete diff:

1. **`state_diff_commitment`** — computed by `calculate_state_diff_hash(&state_diff)`, which chains `deprecated_declared_classes` into the Poseidon hash. With an empty vec, the hash omits every deprecated class declaration. [3](#0-2) 

2. **`state_diff_length`** — computed by `state_diff.len()`, which adds `deprecated_declared_classes.len()`. With an empty vec, the count is too small by exactly the number of deprecated class declarations in the block. [4](#0-3) 

`state_diff_length` is packed into `concatenated_counts` (a single felt encoding tx count, event count, state-diff length, and DA mode), which is part of the block hash input: [5](#0-4) 

Both `state_diff_commitment` and `concatenated_counts` are stored in `partial_block_hash_components` and flow into the final block hash and the `StorageBlockHeader.state_diff_length` field.

**Contrast with the RPC execution path:**

The RPC simulation helper `induced_state_diff` correctly handles this by accepting `deprecated_declared_class_hash` as an explicit parameter and populating the field:

```rust
// crates/apollo_rpc_execution/src/execution_utils.rs  L137-L144
Ok(ThinStateDiff {
    ...
    deprecated_declared_classes: deprecated_declared_class_hash
        .map_or_else(Vec::new, |class_hash| vec![class_hash]),
    ...
})
``` [6](#0-5) 

The block-production path has no equivalent mechanism.

### Impact Explanation

For any block containing a V0/V1 `DeclareTransaction`:

- The `state_diff_commitment` stored in the block header is wrong (missing deprecated class hashes from the Poseidon hash).
- The `state_diff_length` in `concatenated_counts` is wrong (too small), corrupting the `concatenated_counts` felt that is part of the block hash.
- The resulting `PartialBlockHash` / block hash is wrong.
- The `StorageBlockHeader.state_diff_length` written to storage is wrong, which causes P2P sync peers to reject or miscount valid state diff chunks (the sync loop in `StateDiffStreamBuilder` terminates based on `target_state_diff_len` read from the header).

This matches: **Critical — Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

### Likelihood Explanation

`DeclareTransactionV0V1` (V0 and V1) is a fully supported transaction type throughout the codebase — it is serialized, deserialized, protobuf-encoded, and executed by the blockifier. Any user who can submit a transaction to the gateway can trigger this path. The gateway's stateful validation does not filter out deprecated declare transactions by type. The trigger requires no privileged access.

### Recommendation

`CommitmentStateDiff` should be extended with a `deprecated_declared_classes: Vec<ClassHash>` field, populated during blockifier execution of V0/V1 declare transactions (analogously to how `compiled_class_hashes` is populated for V2/V3). The `From<CommitmentStateDiff> for ThinStateDiff` conversion should then forward this field instead of hardcoding `Vec::new()`. The TODO comment at line 698–699 already acknowledges this is a placeholder.

### Proof of Concept

1. Construct a valid `DeclareTransaction::V1` for a Cairo 0 class and submit it to the sequencer gateway.
2. The blockifier executes it; `StateMaps.declared_contracts` records the class hash, but `StateMaps.compiled_class_hashes` is unchanged.
3. `CommitmentStateDiff::from(state_maps)` produces a diff with no entry for the class hash.
4. `ThinStateDiff::from(commitment_state_diff)` sets `deprecated_declared_classes = []`.
5. `calculate_block_commitments` computes `state_diff_commitment` = Poseidon hash without the class hash, and `state_diff_length` = N (missing the +1 for the declaration).
6. The block header is written with these wrong values; the block hash is wrong.
7. Any downstream consumer (L1 verifier, P2P sync peer, proof system) that recomputes the state diff commitment or checks `state_diff_length` will observe a mismatch.
