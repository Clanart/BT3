### Title
Missing Nonce in `meta_tx_v0` Hash Enables Unbounded Signature Replay — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`, `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall computes a transaction hash that deliberately omits a nonce and passes `additional_data_size=0` to `deprecated_get_transaction_hash`. The OS also explicitly skips nonce checking for version-0 transactions. Together, these two design choices mean that any `meta_tx_v0` signature is permanently valid and can be replayed by any observer in any future block, enabling direct loss of funds from victim account contracts.

---

### Finding Description

`execute_meta_tx_v0` in `syscall_impls.cairo` is a syscall available to any executing contract. It allows a caller to invoke another contract's `__execute__` entry point with a caller-supplied signature, simulating a version-0 transaction. The hash that the target account contract uses to verify the signature is computed by `compute_meta_tx_v0_hash`:

```cairo
// transaction_hash.cairo lines 295–315
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),  // ← no nonce
    );
    return tx_hash;
}
```

The hash is a pure function of `(contract_address, selector, calldata, chain_id)`. No nonce, block number, or any other sequencing field is included.

The OS-level nonce guard explicitly skips version-0 transactions:

```cairo
// execute_transaction_utils.cairo lines 63–67
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
```

And the constructed `TxInfo` for the meta-tx hardcodes `nonce=0`:

```cairo
// syscall_impls.cairo lines 343–363
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,   // ← always zero
    ...
);
```

Because `__validate__` is also skipped for version-0 transactions (see `run_validate` line 128: `if (tx_execution_info.tx_info.version == 0) { return (); }`), the only signature check is whatever the target account's `__execute__` performs internally — and it receives a `tx_info.transaction_hash` that is identical every time the same `(contract_address, calldata, chain_id)` tuple is used.

**Attack path:**

1. A user signs a `meta_tx_v0` authorizing a specific `calldata` (e.g., a token transfer of 1000 STRK to address X).
2. A relayer submits a transaction whose contract calls `meta_tx_v0` with that signature. It executes successfully.
3. An attacker (any network observer) extracts `(contract_address, calldata, signature)` from the block.
4. The attacker deploys or uses any contract that calls `meta_tx_v0` with the same parameters.
5. The OS computes the identical hash (no nonce → same inputs → same hash).
6. The target account's `__execute__` receives the same `tx_info.transaction_hash` and the same signature, passes its own signature check, and executes the transfer again.
7. Steps 4–6 can be repeated indefinitely until the victim's balance is drained.

---

### Impact Explanation

**Critical — Direct loss of funds.**

Any account contract that accepts `meta_tx_v0` calls (i.e., any account that can be targeted by this syscall) is vulnerable to having its authorized operations replayed without limit. A single user-signed authorization for a fund transfer becomes a permanent, open-ended authorization for that exact transfer. An attacker with no privileged access can drain the victim's token balance by replaying the same signature across multiple blocks.

---

### Likelihood Explanation

**High.** The `meta_tx_v0` syscall is callable by any contract during execution — no privileged role is required. All data needed for replay (contract address, calldata, signature) is publicly visible on-chain after the first execution. The attacker only needs to deploy a contract that issues the `meta_tx_v0` syscall with the observed parameters.

---

### Recommendation

Include a nonce in the `meta_tx_v0` hash. The nonce should be sourced from the target contract's on-chain state (as is done for regular v3 transactions via `check_and_increment_nonce`) and passed as `additional_data` to `deprecated_get_transaction_hash`. The OS should then verify and increment this nonce after each successful `meta_tx_v0` execution, analogously to how `check_and_increment_nonce` works for non-zero-version transactions.

---

### Proof of Concept

**Step 1 — Victim signs a meta-tx:**
User signs `H(prefix, 0, victim_account, __execute__, calldata=[transfer, 1000, attacker], 0, chain_id)` → `sig`.

**Step 2 — Legitimate execution:**
Relayer submits tx → contract calls `meta_tx_v0(victim_account, __execute__, calldata, sig)` → OS computes hash H → victim's `__execute__` verifies sig against H → transfer executes.

**Step 3 — Replay:**
Attacker deploys `ReplayContract` with:
```
fn execute():
    syscall meta_tx_v0(
        contract_address = victim_account,
        selector         = __execute__,
        calldata         = [transfer, 1000, attacker],
        signature        = sig,   // same sig from step 1
    )
```
Attacker submits a transaction calling `ReplayContract.execute()`. The OS recomputes the identical hash H (no nonce → same inputs). Victim's `__execute__` sees the same valid `(hash, sig)` pair and authorizes another transfer. Repeat until drained.
