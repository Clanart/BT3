Let me examine the key code paths in detail.

### Title
Catchup path silently drops `rejected_txs`, allowing historically-rejected L1 handler transactions to be re-proposed after sync — (`crates/apollo_l1_provider/src/l1_provider.rs`, `crates/apollo_l1_provider/src/catchupper.rs`)

---

### Summary

When the L1 provider enters `CatchingUp` state, all three code paths that advance the provider's height call `apply_commit_block` with an empty `rejected_txs` set. Any L1 handler transaction that was rejected in a historical block is never marked `Rejected`, remains `Pending`, stays in `proposable_index`, and is returned by `get_txs` for the next block after catchup completes.

---

### Finding Description

**Three independent paths all drop rejected tx information:**

**Path 1 — `l2_sync_task` (sync-driven catchup):** [1](#0-0) 

The comment `// No rejected txs in sync blocks.` is explicit. `l1_handler_rejected_tx_hashes` is always `Default::default()`.

**Path 2 — `accept_commit_while_catching_up` (batcher-driven, `Equal` branch):** [2](#0-1) [3](#0-2) 

The function signature does not even accept a `rejected_txs` parameter. The caller (`commit_block`) receives `rejected_txs` from the batcher but passes only `committed_txs` to this function: [4](#0-3) 

**Path 3 — backlog drain (batcher-driven, `Greater` branch):** [5](#0-4) [6](#0-5) 

`CommitBlockBacklog` stores only `committed_txs`; `rejected_txs` is structurally absent. The backlog drain also calls `apply_commit_block(..., Default::default())`.

There is an explicit TODO acknowledging the gap: [3](#0-2) 

**Effect on `TransactionRecord` state:**

`apply_commit_block` partitions `consumed_txs` by `rejected_txs` and calls `commit_txs`: [7](#0-6) 

With empty `rejected_txs`, `commit_txs` never calls `mark_rejected()`: [8](#0-7) 

`mark_rejected()` is the only path that sets `state = TransactionState::Rejected`: [9](#0-8) 

Without that call, `maintain_indices` never removes the tx from `proposable_index`: [10](#0-9) 

`is_proposable()` returns `true` only for `Pending`: [11](#0-10) 

So the tx stays in `proposable_index` and is returned by `get_txs` after catchup.

---

### Impact Explanation

After catchup completes, `get_txs` returns a tx whose L1 message was already included in a finalized historical block. The batcher includes it in a new block. Depending on whether the L1 message was already consumed on L1:

- **If not yet consumed** (block not yet proved): the execution re-processes the same L1 message, potentially succeeding and producing a duplicate committed L1 handler transaction across two blocks.
- **If already consumed**: execution fails again, but the tx is still included in the new block as a rejected/reverted entry — a stale L1 message in a committed block.

Both outcomes violate the invariant that a tx included in a finalized historical block must not appear in a future block.

---

### Likelihood Explanation

The condition is triggered on every node restart or startup where the provider falls behind the batcher height. Any L1 handler tx that was rejected in a block between `start_height` and the catchup `target_height` is affected. No attacker privilege is required beyond submitting an L1 handler transaction that fails execution on L2.

---

### Recommendation

1. **`CommitBlockBacklog`** must store both `committed_txs` and `rejected_txs`.
2. **`accept_commit_while_catching_up`** must accept and forward `rejected_txs` to `apply_commit_block`.
3. **`l2_sync_task`** must retrieve rejected tx hashes from the sync block data (if available) or the sync client must expose them.
4. The backlog drain loop must pass `committed_block.rejected_txs` to `apply_commit_block`.

---

let mut provider = L1Provider::new(...);
provider.initialize(BlockNumber(0), vec![scraped_tx(TX_HASH)]).await;

// Simulate: batcher commits block 0 with TX_HASH as rejected
// (normal path, not catchup)
provider.commit_block(
    IndexSet::new(),           // committed_txs: empty
    IndexSet::from([TX_HASH]), // rejected_txs: TX_HASH was rejected
    BlockNumber(0),
).await.unwrap();

// Verify TX_HASH is Rejected and NOT in proposable_index
let snap = provider.get_l1_provider_snapshot().unwrap();
assert!(snap.rejected_transactions.contains(&TX_HASH));

// Now simulate catchup scenario: reset provider, re-scrape tx
let mut provider2 = L1Provider::new(...);
provider2.initialize(BlockNumber(0), vec![scraped_tx(TX_HASH)]).await;

// Batcher sends commit_block at height 1 (triggers catchup to height 0)
// l2_sync_task processes block 0 with empty rejected_txs
provider2.commit_block(
    IndexSet::new(), IndexSet::new(), BlockNumber(0), // sync path: no rejected
).await.unwrap();

// After catchup: TX_HASH is still Pending
let snap2 = provider2.get_l1_provider_snapshot().unwrap();
assert!(snap2.uncommitted_transactions.contains(&TX_HASH)); // BUG: should be rejected

// get_txs returns TX_HASH for next block
provider2.start_block(BlockNumber(1), SessionState::Propose).unwrap();
let txs = provider2.get_txs(10, BlockNumber(1)).unwrap();
assert!(txs.iter().any(|tx| tx.tx_hash == TX_HASH)); // BUG: re-proposed

    pub fn commit_txs(
        &mut self,
        committed_txs: &[TransactionHash],
        rejected_txs: &[TransactionHash],
    ) {
        self.rollback_staging();

        for &tx_hash in committed_txs {
            self.create_record_if_not_exist(tx_hash);
            self.with_record(tx_hash, |r| r.mark_committed()).unwrap();
        }
        for &tx_hash in rejected_txs {
            self.with_record(tx_hash, |r| r.mark_rejected()).expect(
                "Storage inconsistency: a transaction sent to the batcher was removed \
                 unexpectedly.",
            );
        }
    }

                // We haven't scraped this tx yet, so it isn't indexed.
                return;

            let tx_hash = hash;

            if record.is_proposable() {
                // Assumption: txs will only be added to the index once, on arrival, so this
                // preserves arrival order.
                let tx_hashes = self.proposable_index.entry(scrape_timestamp).or_default();
                if !tx_hashes.contains(&tx_hash) {
                    tx_hashes.push(tx_hash);
                }

                // This tx needs to be removed from the proposable index if it was on it.
                // Remove from the vec for this timestamp, and drop the entry if it becomes empty.

                        let tx_hashes = entry.get_mut();

                            tx_hashes.remove(index_in_vec);
                            if tx_hashes.is_empty() {
                                entry.remove();
                            }
                        }
                    }
                    Entry::Vacant(_) => {}
                }
