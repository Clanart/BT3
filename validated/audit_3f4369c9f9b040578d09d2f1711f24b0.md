Audit Report

## Title
Unguarded `refundETH` and `sweepToken` Allow Any Caller to Drain Stranded ETH/ERC20 from the Router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` and `sweepToken()` carry no access control. When a user over-funds a WETH swap, `pay()` wraps exactly the required amount and leaves the surplus ETH on the router. Any third party can then call `refundETH()` to receive the full ETH balance, or call `sweepToken()` with an arbitrary `recipient` to drain any ERC20 balance. The victim has no recourse.

## Finding Description
**ETH stranding path:**
In `pay()`, when `token == WETH` and `nativeBalance >= value`, exactly `value` wei is wrapped and forwarded to the pool; the remainder stays on the router: [1](#0-0) 

The `receive()` guard only blocks plain ETH transfers (not `msg.value` sent with payable function calls), so surplus ETH from `exactInputSingle{value: X}(...)` accumulates freely: [2](#0-1) 

**Unguarded drains:**
`refundETH()` sends the full ETH balance to `msg.sender` with no check on who the caller is: [3](#0-2) 

`sweepToken()` sends the full ERC20 balance to a caller-supplied `recipient` with no check: [4](#0-3) 

**`multicall` does not mitigate this:**
`multicall` uses `delegatecall`, so `msg.sender` is the original caller within each sub-call. A user who correctly bundles `[exactInputSingle, refundETH]` is safe, but a user who calls `exactInputSingle` directly — or omits `refundETH` from the bundle — leaves ETH exposed to any subsequent caller: [5](#0-4) 

## Impact Explanation
An attacker monitoring the router's ETH balance can call `refundETH()` after any victim swap that leaves surplus ETH, receiving the full stranded amount. For `sweepToken`, the attacker supplies their own address as `recipient` and drains any ERC20 balance on the router. This is a direct loss of user funds with no recourse. Severity is **Medium**: the loss is bounded to the surplus the victim failed to reclaim (not the full swap principal), and requires the victim to omit the `refundETH` call.

## Likelihood Explanation
- Users calling `exactInputSingle` or `exactInput` directly (not via a frontend that auto-bundles `refundETH`) will routinely leave ETH on the router.
- The attack requires no special permissions, no malicious pool, and no oracle manipulation — only a public call.
- Front-running is not required; the attacker can poll the router balance and call at any time.

## Recommendation
1. **Auto-refund excess ETH** at the end of each top-level swap function rather than relying on the caller to bundle a separate `refundETH` call.
2. **Track the original sender's ETH contribution** in transient storage and restrict `refundETH` to return only to that address.
3. **Restrict `sweepToken` recipient** to `msg.sender`, or add an `onlySelf` guard so it can only be called from within a `multicall` bundle.

## Proof of Concept
```solidity
// Victim sends 1 ETH; swap only needs 0.9 ETH.
// pay() wraps 0.9 ETH → WETH → pool; 0.1 ETH stays on router.
router.exactInputSingle{value: 1 ether}(params); // victim tx

// Attacker (separate tx, no special role):
router.refundETH(); // attacker receives 0.1 ETH
// OR
router.sweepToken(address(weth), 0, attacker); // drains any WETH balance
```
Foundry test: deploy router, call `exactInputSingle{value: 1 ether}` with a mock pool that consumes 0.9 ETH worth of WETH, assert `address(router).balance == 0.1 ether`, then call `refundETH()` from a different address and assert that address received 0.1 ETH.
