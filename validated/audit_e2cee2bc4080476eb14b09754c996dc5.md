### Title
`nativeFee` ETH Permanently Locked in `OmniBridge` — No Withdrawal Mechanism - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.sol` accepts ETH `nativeFee` payments from users in `initTransfer` and `initTransfer1155`, but only forwards the `extensionValue` portion to Wormhole. The `nativeFee` portion is silently retained in the contract. There is no `withdraw`, `rescueETH`, or any admin function to recover this accumulated ETH. A `receive()` fallback further allows arbitrary ETH to be sent and permanently locked.

---

### Finding Description

`initTransfer` (line 373) and `initTransfer1155` (line 439) are both `external payable`. The ETH accounting splits `msg.value` into two parts:

**For ERC20 transfers** (`tokenAddress != address(0)`):
```solidity
extensionValue = msg.value - nativeFee;   // line 393
```

**For native ETH transfers** (`tokenAddress == address(0)`):
```solidity
extensionValue = msg.value - amount - nativeFee;   // line 391
```

Only `extensionValue` is passed to `initTransferExtension`, and in `OmniBridgeWormhole` that value is forwarded to Wormhole:

```solidity
// OmniBridgeWormhole.sol line 143
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
```

The `nativeFee` portion is never transferred anywhere — it stays in the contract balance. A grep across all EVM Solidity files confirms there is **no** `withdraw`, `rescueETH`, `rescueFunds`, or `emergencyWithdraw` function anywhere in scope.

Additionally, the contract exposes an unconditional `receive()` fallback:
```solidity
receive() external payable {}   // line 574
```
Any ETH sent directly to the contract is also permanently locked.

---

### Impact Explanation

Every user who calls `initTransfer` or `initTransfer1155` with a non-zero `nativeFee` contributes ETH that is irrecoverably locked in the bridge contract. Over time, as the bridge accumulates volume, this becomes a growing pool of permanently frozen protocol funds. There is no admin escape hatch. This matches the allowed impact: **Permanent freezing / irrecoverable lock of user or protocol funds in bridge flows**.

---

### Likelihood Explanation

`nativeFee` is a first-class parameter of the public `initTransfer` interface, documented and emitted in the `InitTransfer` event. Any user who sets `nativeFee > 0` to incentivize relayers triggers the lock. This is a normal, expected user action with no special preconditions.

---

### Recommendation

Add an admin-gated ETH withdrawal function, accessible only to `DEFAULT_ADMIN_ROLE`, to recover accumulated `nativeFee` ETH:

```solidity
function withdrawETH(address payable to, uint256 amount) external onlyRole(DEFAULT_ADMIN_ROLE) {
    (bool success, ) = to.call{value: amount}("");
    require(success, "ETH transfer failed");
}
```

Alternatively, enforce that `nativeFee` is always zero on the EVM side (since relayer compensation is handled on NEAR), and revert if `msg.value != extensionValue` (i.e., the Wormhole fee only).

---

### Proof of Concept

1. User calls `initTransfer(usdcAddress, 1000e6, 0, 1e16, "alice.near", "")` sending `msg.value = 1e16 + wormholeFee`.
2. Inside `initTransfer`: `extensionValue = msg.value - nativeFee = wormholeFee`.
3. `initTransferExtension` is called with `value = wormholeFee`; Wormhole receives exactly `wormholeFee`.
4. The `1e16` wei (`nativeFee`) remains in `OmniBridge`'s balance.
5. No function exists to retrieve it. Repeat for every bridge user who sets `nativeFee > 0`.
6. Confirm: `grep -r "withdraw\|rescueETH" evm/src/` returns zero matches.
