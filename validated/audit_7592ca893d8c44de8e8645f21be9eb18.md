### Title
Unchecked `global_roots` Across Chunked RPC Proof Responses Produces Inconsistent SNOS Proof Inputs - (`crates/starknet_transaction_prover/src/running/storage_proofs.rs`)

### Summary

`merge_storage_proofs` silently takes `global_roots` only from the first chunk response and discards the roots returned by every subsequent chunk, with no cross-chunk equality check. When the total storage-key count exceeds `MAX_KEYS_PER_REQUEST`, the query is split and each chunk is fetched with a separate RPC call. A buggy or adversarially-controlled RPC node can return a different `contracts_tree_root` / `classes_tree_root` for each chunk. The merged proof then carries the first chunk's roots while its trie-node preimages come from a mixture of states, producing `StateCommitmentInfos` whose `previous_root` / `updated_root` fields are inconsistent with the actual Patricia nodes — wrong proof inputs to the SNOS.

### Finding Description

`fetch_proofs` splits large queries into chunks and calls `fetch_single_proof` once per chunk: [1](#0-0) 

The results are merged by `merge_storage_proofs`. The very first line of that function reads: [2](#0-1) 

All subsequent chunks' `global_roots` fields are consumed in the loop but never compared to `proofs[0].global_roots`: [3](#0-2) 

The merged `RpcStorageProof` is then passed to `create_commitment_infos_with_state_changes`, which extracts the initial state roots directly from `rpc_proof.global_roots`: [4](#0-3) 

Those roots seed `commit_state_diff`, which builds a `FactsDb` from the (potentially mixed-state) trie nodes and computes new state roots: [5](#0-4) 

The resulting `StateCommitmentInfos` — with `previous_root` from chunk 0 and `updated_root` derived from trie nodes that may belong to a different state — is returned as the authoritative OS proof input.

The same unchecked path exists in `create_commitment_infos_without_state_changes`: [6](#0-5) 

The code comment acknowledges that ordering validation is expected ("Validation is performed in `to_storage_proofs`") but no such validation covers `global_roots` consistency across chunks.

### Impact Explanation

The `StateCommitmentInfos` produced here are the direct proof inputs consumed by the SNOS. A `previous_root` that does not match the actual root of the supplied Patricia preimages means the OS receives a commitment that is inconsistent with the witness data. This produces a wrong SNOS output — specifically wrong `previous_root` / `updated_root` values in the commitment infos — matching the impact category: **High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value**, and potentially **Critical. Wrong state … from blockifier/syscall/execution logic for accepted input** if the inconsistency propagates through proof verification.

### Likelihood Explanation

A correct RPC node queried at a fixed `block_number` will always return the same `global_roots`. However:
- The RPC node is an external, operator-configured dependency with no trust boundary enforced in code.
- A buggy node (e.g., returning stale cached roots for some chunks) or a node that experiences a reorg between sequential chunk requests can return divergent roots.
- The code itself acknowledges the need for response-order validation but omits root consistency validation.

Likelihood is **Medium** — requires a non-standard RPC node behaviour, but no privileged access is needed and the trigger is a normal large-query code path.

### Recommendation

After collecting all chunk proofs and before calling `merge_storage_proofs`, assert that every chunk's `global_roots` equals the first chunk's `global_roots`. Reject the entire batch if any mismatch is detected:

```rust
// After collecting all chunk proofs:
let reference_roots = &proofs[0].global_roots;
for (i, proof) in proofs.iter().enumerate().skip(1) {
    if proof.global_roots != *reference_roots {
        return Err(ProofProviderError::InconsistentGlobalRoots {
            chunk: i,
            expected: reference_roots.clone(),
            got: proof.global_roots.clone(),
        });
    }
}
```

This mirrors the recommendation in the original report: treat the "constant" value (`global_roots`, analogous to `decimals()`) as a precondition that must be validated on every external call, and reject the batch if the invariant is violated.

### Proof of Concept

1. Configure the transaction prover to use an RPC node that, for a given `block_number`, returns `global_roots = {contracts_tree_root: R1, classes_tree_root: C1}` for the first chunk and `global_roots = {contracts_tree_root: R2, classes_tree_root: C2}` (where `R2 ≠ R1`) for the second chunk, while returning trie nodes consistent with `R2`/`C2` for both chunks.
2. Submit a transaction batch large enough that `count_total_keys(query) > MAX_KEYS_PER_REQUEST`, triggering the chunked path in `fetch_proofs`.
3. `merge_storage_proofs` returns a proof with `global_roots` = `{R1, C1}` but trie nodes rooted at `R2`/`C2`.
4. `create_commitment_infos_with_state_changes` seeds `commit_state_diff` with `contracts_trie_root_hash = R1`, but the `FactsDb` contains preimages for the trie rooted at `R2`. The committer traverses the wrong trie, producing a `new_roots` that is inconsistent with the actual post-execution state.
5. The returned `StateCommitmentInfos` carries `previous_root = R1` and `updated_root` derived from the wrong trie — wrong SNOS proof inputs.

        &self,
        block_number: BlockNumber,
        query: &RpcStorageProofsQuery,
    ) -> Result<RpcStorageProof, ProofProviderError> {
        if count_total_keys(query) <= MAX_KEYS_PER_REQUEST {
            return self.fetch_single_proof(block_number, query).await;

        }

        Ok(merge_storage_proofs(proofs, &chunks, query))

/// Commits the state diff, populates the DB with the new facts and returns the new state roots.
pub async fn commit_state_diff(
    facts_db: &mut FactsDb<MapStorage>,
    contracts_trie_root_hash: HashOutput,
    classes_trie_root_hash: HashOutput,
    state_diff: StateDiff,
) -> Result<StateRoots, ProofProviderError> {
    let warn_on_trivial_modifications = false;
    let build_storage_tries_concurrently = false; // Map storage is used, no need for concurrency.
    let config = ReaderConfig::new(warn_on_trivial_modifications, build_storage_tries_concurrently);
    let initial_read_context =
        FactsDbInitialRead(StateRoots { contracts_trie_root_hash, classes_trie_root_hash });
    let input = Input { state_diff, initial_read_context, config };

    let (filled_forest, _deleted_nodes) = commit_block(input, facts_db, &mut NoMeasurements)
        .await
        .map_err(|e| ProofProviderError::BlockCommitmentError(e.to_string()))?;
    facts_db.write(&filled_forest).await?;

    Ok(StateRoots {
        contracts_trie_root_hash: filled_forest.get_contract_root_hash(),
        classes_trie_root_hash: filled_forest.get_compiled_class_root_hash(),
    })
