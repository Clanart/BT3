### Title
Missing EVM-Side Recipient Format Validation Causes Permanent Token Burn With No NEAR Settlement - (`evm/src/omni-bridge/contracts/HlBridgeToken.sol`, `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`HlBridgeToken.coreReceiveWithData` with `ACTION_INIT_TRANSFER` passes the user-supplied `recipient` string directly to `OmniBridge.initTransfer` without any format validation. `initTransfer` burns the tokens and emits `InitTransfer`. On the NEAR side, the prover parses the event and calls `recipient.parse::<OmniAddress>()`, which for a `near:` prefix calls `AccountId::from_str()`. NEAR's `AccountId` enforces strict rules (lowercase only, max 64 chars, valid charset). An invalid recipient string causes the parse to fail, the prover returns an error, `fin_transfer_callback` panics with `InvalidProofMessage`, and the transfer can never be finalized. The tokens are permanently burned on EVM with no recovery path.

---

### Finding Description

**Step 1 — EVM entry point, no validation:**

In `HlBridgeToken.coreReceiveWithData`, the `ACTION_INIT_TRANSFER` branch decodes `(uint128 fee, string memory recipient, string memory message)` from the caller-controlled `data` and passes `recipient` verbatim to `OmniBridge.initTransfer`: [1](#0-0) 

**Step 2 — `OmniBridge.initTransfer` burns tokens, emits event, no recipient check:**

`initTransfer` performs only a fee sanity check (`fee >= amount`), then burns the bridge token and emits `InitTransfer` with the raw `recipient` string. There is no format validation of `recipient` anywhere in the function: [2](#0-1) 

**Step 3 — NEAR prover parses the event and enforces `AccountId` rules:**

In `near/omni-types/src/evm/events.rs`, `TryFromLog<Log<InitTransfer>>` for `InitTransferMessage` calls:

```rust
recipient: event.data.recipient.parse().map_err(stringify)?,
``` [3](#0-2) 

This calls `OmniAddress::from_str()`. For a `near:` prefix, it calls `recipient.parse::<AccountId>()`: [4](#0-3) 

NEAR's `AccountId` type rejects: uppercase letters, characters outside `[a-z0-9._-]`, strings longer than 64 bytes, leading/trailing dots, and consecutive dots. Any of these causes `parse()` to return `Err`, which propagates via `?` and causes `parse_evm_proof` to return `Err`.

**Step 4 — `fin_transfer_callback` panics permanently:**

`fin_transfer_callback` treats any prover error as a hard panic: [5](#0-4) 

There is no retry, no refund, and no alternative settlement path. The `origin_nonce` is already consumed on EVM. The burned tokens are irrecoverable.

---

### Impact Explanation

Any HyperCore user who supplies a `recipient` string that is valid UTF-8 but violates NEAR `AccountId` rules (e.g., `near:ALICE.NEAR`, `near:alice near`, `near:` + 65-char string, `near:.alice`) will have their tokens permanently burned on EVM with zero NEAR-side settlement. This is a direct, irrecoverable loss of user funds matching the Critical impact class: **permanent freezing / irrecoverable lock of user funds in bridge flows**.

---

### Likelihood Explanation

The path is reachable by any unprivileged HyperCore user. The `coreReceiveWithData` function is callable only by `_systemAddress`, but the `data` payload (including `recipient`) is fully user-controlled at the HyperCore application layer — the system address is a pass-through for user intent. A user who mistypes their NEAR address (e.g., uses uppercase, adds a space, exceeds 64 chars) triggers the bug. This is also exploitable by a malicious user to grief themselves or, in a shared-pool model, to cause accounting confusion. The lack of any EVM-side guard makes this a latent production risk on every `ACTION_INIT_TRANSFER` call.

---

### Recommendation

Add recipient format validation in `OmniBridge.initTransfer` (or in `HlBridgeToken.coreReceiveWithData` before calling it) for `near:` prefixed recipients. The validation must enforce:
- Lowercase only (`[a-z0-9._-]`)
- Length ≤ 64 bytes
- No leading/trailing/consecutive dots

Alternatively, validate the `OmniAddress` format on-chain using a Solidity regex or a dedicated `validateRecipient(string)` helper that mirrors NEAR's `AccountId` rules. The check must happen **before** the burn/lock so that an invalid recipient causes a revert rather than a permanent loss.

---

### Proof of Concept

1. Deploy `HyperliquedBridgeToken` registered with `OmniBridge` (as in the existing test setup).
2. Call `coreReceiveWithData` from `_systemAddress` with `ACTION_INIT_TRANSFER` and `recipient = "near:ALICE.NEAR"` (uppercase).
3. Assert: `InitTransfer` is emitted, `totalSupply == 0` (tokens burned).
4. On NEAR, call `fin_transfer` with the RLP-encoded log. The prover calls `"ALICE.NEAR".parse::<AccountId>()`, which returns `Err` because uppercase is forbidden.
5. `fin_transfer_callback` panics with `InvalidProofMessage`.
6. Assert: no tokens minted on NEAR, no recovery path exists, funds permanently lost.

The same path is reachable via the standard `OmniBridge.initTransfer` public entrypoint (not just the HyperCore path), since `initTransfer` accepts any `string calldata recipient` with no validation. [6](#0-5) [7](#0-6)
