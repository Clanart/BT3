### Title
Missing Underflow Guard on `current_block_number - STORED_BLOCK_HASH_BUFFER` Causes OS Program Crash — (`File: execution/execution_constraints.cairo`)

---

### Summary

In `check_proof_facts`, the expression `current_block_number - STORED_BLOCK_HASH_BUFFER` is computed and passed directly to `assert_nn_le` without first verifying that `current_block_number >= STORED_BLOCK_HASH_BUFFER`. When `current_block_number < STORED_BLOCK_HASH_BUFFER` (i.e., during the first 10 blocks of the network), the subtraction wraps around the Cairo prime field to a value near `p ≈ 2^251`, which is far outside the `[0, 2^128)` range that `assert_nn_le` requires. This causes the assertion to fail, crashing the entire OS program and invalidating the block proof.

---

### Finding Description

The vulnerability class from M-38 is: **an incorrect or missing arithmetic bounds check that causes a function to revert/crash instead of handling the edge case gracefully**. The analog here is in `check_proof_facts`:

```cairo
// execution_constraints.cairo, lines 66-68
assert_nn_le(
    os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
);
```

`STORED_BLOCK_HASH_BUFFER = 10` (constants.cairo line 65). When `current_block_number` is in `[0, 9]`, the subtraction `current_block_number - STORED_BLOCK_HASH_BUFFER` underflows in the Cairo field, producing a value of approximately `p - (STORED_BLOCK_HASH_BUFFER - current_block_number)`, which is a ~251-bit number. `assert_nn_le` internally calls `assert_le(a, b)`, which checks that `b - a` is in `[0, 2^128)`. With `b` being a ~251-bit field element, `b - a` is also ~251 bits, causing the range check to fail and the OS program to abort.

The call site in `execute_invoke_transaction` (transaction_impls.cairo line 313) invokes `check_proof_facts` directly with no revert handler:

```cairo
check_proof_facts(
    proof_facts_size=proof_facts_size,
    proof_facts=proof_facts,
    current_block_number=block_context.block_info_for_execute.block_number,
    virtual_os_config_hash=block_context.os_global_context.virtual_os_config_hash,
);
```

`proof_facts_size` is populated from the transaction data via the hint `%{ TxProofFacts %}` (transaction_impls.cairo lines 279-281), which reads from the user-submitted transaction. If `proof_facts_size > 0`, `check_proof_facts` proceeds past its early-return guard (line 40) and reaches the vulnerable `assert_nn_le`.

A secondary occurrence of the same pattern exists in `execute_get_block_hash` (syscall_impls.cairo line 753):

```cairo
assert_nn_le(request_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER);
```

That site has a hint-based guard (`IsBlockNumberInBlockHashBuffer`), but the Cairo-level verification of the hint (`assert_lt(current_block_number, request_block_number + STORED_BLOCK_HASH_BUFFER)`) does not enforce `current_block_number >= STORED_BLOCK_HASH_BUFFER` independently of `request_block_number`, leaving the guard incomplete for a malicious prover. However, the primary exploitable path is through `check_proof_facts`, which has no guard at all.

---

### Impact Explanation

When the OS program aborts due to the failed `assert_nn_le`, the entire block proof is invalidated. The sequencer cannot produce a valid proof for any block that contains such a transaction. This maps directly to:

**High — Network not being able to confirm new transactions (total network shutdown)**

The network is unable to finalize any block that includes the triggering transaction, halting transaction confirmation.

---

### Likelihood Explanation

The window is the first 10 blocks of the network (`current_block_number` in `[0, 9]`). An attacker who submits an invoke transaction with a non-empty `proof_facts` field during this window triggers the crash. The `proof_facts` field is a legitimate part of the invoke transaction format (used for client-side proving / virtual OS), so the sequencer has no protocol-level reason to reject it. The attacker needs no special privileges — only the ability to submit a standard invoke transaction.

Likelihood is low in absolute terms (narrow time window at network genesis), but the impact is total and the trigger is trivially constructed.

---

### Recommendation

Before computing `current_block_number - STORED_BLOCK_HASH_BUFFER`, add an explicit non-negativity guard analogous to the one already present in `write_block_number_to_block_hash_mapping` (os_utils.cairo lines 52-58):

```cairo
// In check_proof_facts, before the assert_nn_le:
let is_current_block_ge_buffer = is_nn(current_block_number - STORED_BLOCK_HASH_BUFFER);
if (is_current_block_ge_buffer == FALSE) {
    // No stored block hashes exist yet; proof facts referencing any block are invalid.
    // Write a failure or simply return/assert false here.
    assert 1 = 0;  // or handle gracefully
    return ();
}
assert_nn_le(
    os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
);
```

The same fix should be applied to `execute_get_block_hash` (syscall_impls.cairo line 753) to make the guard Cairo-enforced rather than hint-dependent.

---

### Proof of Concept

1. Network launches. Block 5 is being processed (`current_block_number = 5`, `STORED_BLOCK_HASH_BUFFER = 10`).
2. Attacker submits an invoke transaction with `proof_facts = [<any non-empty data>]` (e.g., a single felt).
3. The sequencer includes the transaction in block 5 and runs the OS program.
4. `execute_invoke_transaction` calls `check_proof_facts` with `proof_facts_size = 1`.
5. `check_proof_facts` reaches line 66: `assert_nn_le(base_block_number, 5 - 10)`.
6. `5 - 10` in the Cairo field = `p - 5` ≈ `2^251`, which is outside `[0, 2^128)`.
