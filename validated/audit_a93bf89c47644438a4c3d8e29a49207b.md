### Title
`normalize_amount` Integer Division Truncation to Zero Permanently Locks User Funds - (`near/omni-bridge/src/lib.rs`)

### Summary

When a user initiates a NEAR-side transfer (`ft_transfer_call` → `init_transfer`), their tokens are immediately burned or locked. However, no check is performed at that point to verify that `normalize_amount(amount - fee, decimals) > 0`. That check only occurs later in `sign_transfer`. If the `amount_without_fee` is below the decimal-normalization divisor (`10^(origin_decimals - decimals)`), `normalize_amount` returns zero via integer division truncation, causing every subsequent `sign_transfer` call to revert with `ERR_INVALID_AMOUNT_TO_TRANSFER`. Because there is no public cancel or refund path for pending transfers, the user's funds are permanently frozen.

### Finding Description

`normalize_amount` performs integer floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

When `origin_decimals > decimals` (e.g., a NEAR token with 24 decimals bridging to an EVM chain with 18 decimals, giving `diff_decimals = 6` and `divisor = 1_000_000`), any `amount_without_fee < divisor` produces zero.

**Step 1 — Tokens are locked/burned at `init_transfer` time with no normalization check:**

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [2](#0-1) 

The only validation is `fee < amount`. There is no check that `normalize_amount(amount - fee, decimals) > 0`. Immediately after, `init_transfer_internal` burns or locks the full `amount`: [3](#0-2) 

**Step 2 — `sign_transfer` checks the normalized amount and reverts:**

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [4](#0-3) 

If `amount_without_fee < 10^(origin_decimals - decimals)`, this always reverts. The transfer is stuck in `pending_transfers` forever.

**Step 3 — No public cancel/refund path exists.** `remove_transfer_message` and `remove_transfer_message_without_refund` are private internal functions. [5](#0-4) 

`update_transfer_fee` only allows increasing the fee (not decreasing it), so the user cannot fix the situation by reducing the fee to make `amount_without_fee` larger. [6](#0-5) 

### Impact Explanation

User tokens are permanently frozen in the NEAR bridge contract (or permanently burned if the token is a deployed bridge token). The pending transfer record remains in `pending_transfers` indefinitely with no recovery path. This matches: **Critical — Permanent freezing, irrecoverable lock of user funds in bridge flows.**

### Likelihood Explanation

Any user who sends a transfer where `amount - fee < 10^(origin_decimals - decimals)` triggers this. For a NEAR-native token with 24 decimals bridging to an EVM chain with 18 decimals, the threshold is `1_000_000` base units — a realistic small transfer. The user does not need to be malicious; this can happen accidentally. No special role or privilege is required; the entry point is the public `ft_transfer_call` NEP-141 interface.

### Recommendation

Add a normalization check inside `init_transfer` (before locking/burning) to ensure the net amount after fee is non-zero when normalized to the destination chain's decimals:

```rust
// After building transfer_message, before init_transfer_internal:
let token_address = self.get_token_address(destination_chain, token_id.clone())
    .near_expect(BridgeError::FailedToGetTokenAddress);
let decimals = self.token_decimals.get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);
require!(
    Self::normalize_amount(
        transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
        decimals,
    ) > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

Alternatively, add a public `cancel_transfer` function that allows the original sender to reclaim locked/burned tokens for a pending transfer that has not yet been signed.

### Proof of Concept

1. Register a NEAR-native token with `origin_decimals = 24`, `decimals = 18` for an EVM destination chain. The normalization divisor is `10^6 = 1_000_000`.
2. Call `ft_transfer_call` with `amount = 500_000`, `fee = 0`, recipient on the EVM chain.
3. `init_transfer` passes the `fee < amount` check and calls `init_transfer_internal`, which locks `500_000` tokens in the bridge.
4. A relayer calls `sign_transfer` for this transfer. `normalize_amount(500_000, {18, 24}) = 500_000 / 1_000_000 = 0`. The call reverts with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
5. No public cancel function exists. The `500_000` tokens are permanently locked. Every future `sign_transfer` call for this transfer ID will also revert.

        let storage_usage = env::storage_usage();
        let transfer = self

            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {

            self.accounts_balances.insert(&transfer.owner, &storage);
