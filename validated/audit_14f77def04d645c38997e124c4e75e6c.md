### Title
Missing `shard_id` Binding in `set_state_header` Allows Cross-Shard State Substitution During State Sync — (`File: chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` validates that a chunk is present in the block's Merkle tree but never verifies that the chunk's own `shard_id` field matches the `shard_id` argument. A malicious peer can supply a `ShardStateSyncResponseHeader` for shard A that embeds shard B's chunk (with a valid Merkle proof for shard B's position). The header passes all checks and is stored under `StateHeaderKey(shard_A, sync_hash)` carrying shard B's `prev_state_root`. Subsequent state-part downloads are validated against that wrong root, and `apply_state_part` writes shard B's trie data into shard A's storage, permanently corrupting the syncing node's state for shard A.

### Finding Description

In `set_state_header` the only structural check that binds the supplied chunk to the block is:

```rust
if !verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
) { … }
``` [1](#0-0) 

`verify_path` confirms that the leaf `ChunkHashHeight(chunk_hash, height_included)` lies somewhere in the Merkle tree rooted at `chunk_headers_root`. The Merkle path encodes the leaf's position implicitly through its left/right sibling sequence, but **the code never asserts that this position equals the shard index corresponding to `shard_id`**. A proof for shard B's chunk at position B is cryptographically valid and will satisfy `verify_path` regardless of the `shard_id` argument passed to `set_state_header`.

No other check in the function compares `chunk.shard_id()` to the caller-supplied `shard_id`: [2](#0-1) 

After passing validation the header is stored under the key `StateHeaderKey(shard_id, sync_hash)`:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
``` [3](#0-2) 

`set_state_part` then reads the stored header to obtain the state root used for part validation:

```rust

So shard B's state parts validate correctly against shard B's state root, are stored under `StatePartKey(sync_hash, shard_A, part_id)`, and are then applied to shard A's `ShardUId` storage by `apply_state_part`:

```rust

tries.apply_all(&trie_changes, shard_uid, &mut store_update);
flat_state_delta.apply_to_flat_state(&mut store_update.flat_store_update(), shard_uid);
``` [6](#0-5) 

The `StatePartsApplied` idempotency guard uses `StatePartKey(sync_hash, shard_A, part_id)` as its key, so it does not prevent the cross-shard write: [7](#0-6) 

### Impact Explanation

A syncing node that accepts the malicious header will:

1. Store shard B's `prev_state_root` as the authoritative state root for shard A.
2. Accept and apply shard B's trie nodes and flat-state deltas into shard A's `ShardUId` column.
3. Execute `set_state_finalize` using shard B's chunk (transactions, receipts) against shard A's `shard_uid`, producing a `ChunkExtra` for shard A derived from shard B's execution.

The node's on-disk state for shard A is permanently replaced by shard B's data. All subsequent chunk production and validation for shard A will be based on the wrong state root, causing the node to produce invalid chunks and be unable to participate correctly in consensus. This is a direct, irreversible state-corruption outcome, not merely resource waste.

### Likelihood Explanation

State sync headers are downloaded from arbitrary peers. Any peer that can respond to a `StateRequestHeader` message — which requires no special role — can supply the malicious header. The attacker must also supply state parts consistent with shard B's state root (trivially available from the honest network for shard B), so the full attack requires only a single colluding peer. No validator key, chunk-producer role, or storage-operator privilege is needed.

### Recommendation

Add an explicit shard-identity check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: enforce that the chunk belongs to the requested shard
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {} does not match requested shard_id {}",
        chunk.shard_id(), shard_id
    )));
}
```

This mirrors the existing pattern of explicit field-level binding checks throughout the function (height, receipt-proof continuity, state-root-node hash).

### Proof of Concept

**Setup**: multi-shard network; syncing node N requests state for shard 0 at `sync_hash` H.

**Step 1 — Malicious header injection**  
Attacker peer P intercepts the `StateRequestHeader { shard_id: 0, sync_hash: H }` and responds with a `ShardStateSyncResponseHeader` where:
- `chunk` = the genuine chunk for shard 1 at height H−1 (obtained from the honest network).
- `chunk_proof` = the genuine Merkle path proving shard 1's chunk is in block H−1's `chunk_headers_root` (position 1, not position 0).
- `state_root_node` = the genuine `StateRootNode` for shard 1's `prev_state_root`.
- `incoming_receipts_proofs` = valid receipt proofs for shard 1 (or empty if shard 1 received no receipts).

**Step 2 — Validation passes**  
`set_state_header(shard_id=0, sync_hash=H, header)`:
- `validate_chunk_proofs` passes — shard 1's chunk is internally consistent.
- `verify_path` passes — shard 1's chunk hash IS in the block's Merkle tree (at position 1); the path is cryptographically valid.
- `validate_state_root_node` passes — the node matches shard 1's `prev_state_root`.
- No check fires on `chunk.shard_id() != 0`.

Header stored: `StateHeaderKey(shard_id=0, H)` → contains shard 1's `prev_state_root` R₁.

**Step 3 — State parts applied to wrong shard**  
N downloads state parts for shard 0. P supplies shard 1's genuine state parts (each validates against R₁). `apply_state_part(shard_id=0, state_root=R₁, …)` writes shard 1's trie nodes and flat-state entries into shard 0's `ShardUId` storage.

**Step 4 — Finalization**  
`set_state_finalize(shard_id=0, H)` applies shard 1's chunk (transactions, receipts) against shard 0's `shard_uid` using R₁ as the pre-state root. The resulting `ChunkExtra` for shard 0 is derived from shard 1's execution. Node N now has permanently corrupted state for shard 0.

        let prev_chunk_header = shard_state_header.cloned_prev_chunk_header();

            return Err(Error::Other(

        // 3a. Checking that chunk `chunk` is included into block at last height before sync_hash
        // 3aa. Also checking chunk.height_included

            self.chain_store.get_block_header(sync_block_header.prev_hash())?;
        if !verify_path(
            *sync_prev_block_header.chunk_headers_root(),
            shard_state_header.chunk_proof(),
            &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
        ) {

            return Err(Error::Other(
                "set_shard_state failed: chunk isn't included into block".into(),

            &self.chain_store,

                    *prev_block_header.chunk_headers_root(),
                    prev_chunk_proof,
                    &ChunkHashHeight(prev_chunk_header.chunk_hash().clone(), prev_chunk_header.height_included()),
                ) {
                    byzantine_assert!(false);
                    return Err(Error::Other(
                        "set_shard_state failed: prev_chunk isn't included into block".into(),

                }
            }

        if shard_state_header.root_proofs().len()

            byzantine_assert!(false);
            return Err(Error::Other("set_shard_state failed: invalid proofs".into()));
        }
        let mut hash_to_compare = sync_hash;
        for (i, receipt_response) in
            shard_state_header.incoming_receipts_proofs().iter().enumerate()

            let ReceiptProofResponse(block_hash, receipt_proofs) = receipt_response;

            // 4b. Checking that there is a valid sequence of continuous blocks
            if *block_hash != hash_to_compare {
                byzantine_assert!(false);
                return Err(Error::Other(
                    "set_shard_state failed: invalid incoming receipts".into(),
