### Title
Stale shard-to-peer routing in `SnapshotHostsCache` after peer updates `SnapshotHostInfo` with a reduced shard list — (File: `chain/network/src/snapshot_hosts/mod.rs`)

---

### Summary

`Inner::insert()` in `SnapshotHostsCache` adds a peer to `hosts_for_shard` for its new shards but never removes it from shards that were present in the old `SnapshotHostInfo` and absent in the new one. After a peer publishes an updated `SnapshotHostInfo` with a higher `epoch_height` and a smaller shard set, the derived routing index `hosts_for_shard` retains stale entries. State-part and state-header requests for those dropped shards continue to be routed to the peer, which no longer serves them, causing `Busy`/`Error` responses and potentially stalling state sync for the affected shard.

---

### Finding Description

`SnapshotHostsCache` maintains two coupled data structures inside `Inner`:

- `hosts: LruCache<PeerId, Arc<SnapshotHostInfo>>` — the authoritative per-peer record.
- `hosts_for_shard: HashMap<ShardId, HashSet<PeerId>>` — a derived routing index used by `select_host_for_header` and `select_host_for_part`. [1](#0-0) 

When a peer publishes a new `SnapshotHostInfo` with a strictly higher `epoch_height` (which `is_new()` requires), `insert()` is called: [2](#0-1) 

The logic is:

1. `add_to_shard_hosts(d)` — inserts the peer into `hosts_for_shard` for every shard in the **new** info (only if `d.sync_hash == current_state_sync_hash`).
2. `self.hosts.push(d.peer_id.clone(), d.clone())` — overwrites the old entry; because the key is the same, `LruCache::push` returns `Some((displaced, old_info))` where `displaced == d.peer_id`.
3. The guard `if displaced != d.peer_id` is **false**, so `remove_from_shard_hosts` is **never called**. [3](#0-2) 

The comment explicitly encodes the wrong assumption: *"only an eviction should leave the shard caches."* This is correct when the shard list is unchanged, but wrong when the new info drops one or more shards.

`add_to_shard_hosts` only **adds** shards; it never removes them: [4](#0-3) 

`remove_from_shard_hosts` removes a peer from **all** shards, but it is only called on LRU eviction of a **different** peer: [5](#0-4) 

**Concrete scenario (resharding):**

| Step | Action | `hosts_for_shard` state |
|------|--------|------------------------|
| 1 | Peer A broadcasts `SnapshotHostInfo{sync_hash=H, epoch_height=5, shards=[0,1,2]}` | `{0:{A}, 1:{A}, 2:{A}}` |
| 2 | Syncing node calls `select_host_for_header(H, …)` → `current_state_sync_hash = H` | same |
| 3 | Peer A creates epoch-E+1 snapshot; broadcasts `SnapshotHostInfo{sync_hash=H', epoch_height=6, shards=[1,2]}` (shard 0 dropped after resharding) | `{0:{A}, 1:{A}, 2:{A}}` ← **shard 0 stale** |
| 4 | Peer A deletes its epoch-E snapshot | — |
| 5 | Syncing node calls `select_host_for_part(H, shard_id=0, part_id=k)` | Returns A (stale) |
| 6 | Peer A responds `Busy`/`Error` | State sync for shard 0 stalls |

The `peer_selector` for `(shard_0, part_k)` is populated exclusively from `hosts_for_shard[shard_0]`: [6](#0-5) 

If Peer A is the only entry in `hosts_for_shard[shard_0]`, `tried_everybody()` becomes true and `selector.len() == available_hosts.len()`, so no new peers are ever added. The selector loops on Peer A indefinitely.

The only recovery path is `update_current_state_sync_hash`, which clears and rebuilds `hosts_for_shard` — but that only fires when the syncing node advances to a new epoch's sync hash: [7](#0-6) 

---

### Impact Explanation

State sync requests for a shard are permanently routed to a peer that no longer serves that shard for the current epoch. If the stale peer is the sole entry in `hosts_for_shard` for that shard, the syncing node cannot obtain any state parts for it and cannot complete state sync, preventing the node from joining the network until the epoch advances and the sync hash changes.

---

### Likelihood Explanation

This triggers whenever a peer publishes a new `SnapshotHostInfo` with a reduced shard set while a syncing node is actively using the old info. This is a normal, expected event during resharding (a parent shard is split; the peer no longer tracks one of the children) or when a node changes its tracked shards between epochs. No privileged role or malicious intent is required; any network peer can trigger this by legitimately rotating its snapshot.

---

### Recommendation

In `Inner::insert()`, when the LRU push returns a same-key displacement (`displaced == d.peer_id`), compute the set difference between the old shard list and the new shard list and remove the peer from `hosts_for_shard` for each dropped shard. Concretely:

```rust
fn insert(&mut self, d: &Arc<SnapshotHostInfo>) {
    // Remove peer from shards it no longer serves (same-key update case).
    if let Some(old) = self.hosts.peek(&d.peer_id) {
        let new_shards: HashSet<_> = d.shards.iter().collect();
        for shard_id in &old.shards {
            if !new_shards.contains(shard_id) {
                if let Some(set) = self.hosts_for_shard.get_mut(shard_id) {
                    set.remove(&d.peer_id);
                }
            }
        }
    }
    self.add_to_shard_hosts(d.as_ref());
    if let Some((displaced, _)) = self.hosts.push(d.peer_id.clone(), d.clone()) {
        if displaced != d.peer_id {
            self.remove_from_shard_hosts(&displaced);
        }
    }
}
```

    /// Updates the current state sync hash. This is called when a local state sync request is initiated.
    /// Resets peer selectors if the sync hash has changed.
    fn update_current_state_sync_hash(&mut self, sync_hash: &CryptoHash) {
        if self.current_state_sync_hash == Some(*sync_hash) {
            return;
        }

        self.current_state_sync_hash = Some(*sync_hash);
        // Reset peer selectors and shard-specific caches for the new sync hash
        self.hosts_for_shard.clear();
        self.peer_selector.clear();

        // Rebuild the shard-specific caches with hosts that match the new sync hash
        let known_hosts: Vec<_> = self.hosts.iter().map(|(_, info)| info.clone()).collect();
        for info in &known_hosts {
            self.add_to_shard_hosts(info.as_ref());
        }
    }

    pub fn select_host_for_part(
        &mut self,
        sync_hash: &CryptoHash,
        shard_id: ShardId,
        part_id: u64,
    ) -> Option<PeerId> {
        self.update_current_state_sync_hash(sync_hash);

        let selector =
            self.peer_selector.entry((shard_id, part_id)).or_insert(PartPeerSelector::default());

        // Insert more hosts into the selector if needed
        let available_hosts = self.hosts_for_shard.get(&shard_id)?;
        if selector.tried_everybody() && selector.len() < available_hosts.len() {
            let mut new_peers = BinaryHeap::new();
            let already_included = selector.peer_set();

            for peer_id in available_hosts {
                if already_included.contains(peer_id) {
                    continue;
                }

                let score = priority_score(peer_id, shard_id, part_id);

                // Wrap entries with `Reverse` so that we pop the *least* desirable options
                new_peers.push(std::cmp::Reverse(StatePartHost {
                    peer_id: peer_id.clone(),
                    score,
                    num_requests: 0,
                }));

                if new_peers.len() > self.part_selection_cache_batch_size {
                    new_peers.pop();
                }
            }

            selector.insert_peers(new_peers.drain().map(|e| e.0));
        }

        let res = selector.next();
        res
    }
