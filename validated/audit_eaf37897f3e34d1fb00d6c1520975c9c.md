### Title
Integer Wrapping in `check_time_locks` Enables Timelock Bypass via `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` uses `wrapping_add` (when `nowrap=false`) to compute the unlock threshold for relative timelock conditions. An attacker-controlled `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` value at or near the integer maximum wraps the computed deadline to a value smaller than the current block height/timestamp, causing the timelock check to pass immediately — bypassing the intended lock entirely.

---

### Finding Description

`check_time_locks` accepts a `nowrap: bool` parameter that selects between two arithmetic modes for relative timelock evaluation:

- `nowrap=true` → `saturating_add` (safe, clamps to `u32::MAX` / `u64::MAX`)
- `nowrap=false` → `wrapping_add` (legacy, wraps on overflow)

For `ASSERT_HEIGHT_RELATIVE`:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
``` [1](#0-0) 

For `ASSERT_SECONDS_RELATIVE`:

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [2](#0-1) 

The condition parser accepts `height_relative` as a `u32` (4-byte max via `sanitize_uint(..., 4, ...)`) and `seconds_relative` as a `u64` (8-byte max): [3](#0-2) [4](#0-3) 

So `height_relative = 0xFFFF_FFFF` and `seconds_relative = 0xFFFF_FFFF_FFFF_FFFF` are both valid, parseable CLVM condition arguments.

**Concrete overflow example (height):**

- Coin confirmed at block `10`
- `height_relative = 0xFFFF_FFFF` (u32::MAX)
- `10u32.wrapping_add(0xFFFF_FFFF) = 9`
- Check: `prev_height < 9` → always `false` for any block ≥ 10 (i.e., any block after confirmation)
- Result: spend accepted immediately, despite the intended lock of ~4.3 billion blocks

**Concrete overflow example (seconds):**

- Coin timestamp `T`
- `seconds_relative = 0xFFFF_FFFF_FFFF_FFFF` (u64::MAX)
- `T.wrapping_add(u64::MAX) = T - 1`
- Check: `timestamp < T - 1` → always `false` for any timestamp ≥ T
- Result: spend accepted immediately

The codebase's own tests document and confirm this divergence: [5](#0-4) [6](#0-5) 

The Python binding exposes `nowrap` as a caller-controlled parameter: [7](#0-6) 

---

### Impact Explanation

When the Chia full node calls `check_time_locks` with `nowrap=False` (the pre-fork legacy path), any coin spend carrying `ASSERT_HEIGHT_RELATIVE(0xFFFF_FFFF)` or `ASSERT_SECONDS_RELATIVE(0xFFFF_FFFF_FFFF_FFFF)` bypasses its timelock entirely. The spend is accepted as consensus-valid at any block height after the coin's confirmation, regardless of the intended lock duration. This constitutes an unauthorized spend acceptance — a **High** impact timelock validation bypass.

---

### Likelihood Explanation

The `nowrap=False` path is the legacy pre-fork behavior, still reachable via the Python binding for pre-fork block validation. An unprivileged user can craft a spend bundle with a maximally large `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` value — both are valid CLVM atoms within the 4-byte and 8-byte limits enforced by `sanitize_uint`. No special privileges, keys, or network access beyond submitting a spend bundle are required.

---

### Recommendation

Replace `wrapping_add` with `saturating_add` unconditionally in both the `height_relative` and `seconds_relative` branches of `check_time_locks`, or gate the `nowrap=false` path behind a hard fork height check so it is never reachable for new spend bundles. The `nowrap=true` (saturating) path already implements the correct semantics: a `height_relative` or `seconds_relative` value large enough to overflow simply means the condition can never be satisfied, which is the safe and intended behavior. [8](#0-7) 

---

### Proof of Concept

1. Create a coin `C` confirmed at block height `H_c` with timestamp `T_c`.
2. Construct a spend of `C` with condition `ASSERT_HEIGHT_RELATIVE(0xFFFF_FFFF)` (encoded as the 4-byte atom `\xff\xff\xff\xff`).
3. Submit the spend bundle to a node running with `nowrap=False`.
4. `check_time_locks` computes `H_c.wrapping_add(0xFFFF_FFFF) = H_c - 1`.
5. The check `prev_height < H_c - 1` is `false` for any `prev_height >= H_c`, so the spend passes immediately.
6. The coin is spent despite the intended lock of ~4.3 billion blocks.

The same attack applies to `ASSERT_SECONDS_RELATIVE(0xFFFF_FFFF_FFFF_FFFF)` with `T_c.wrapping_add(u64::MAX) = T_c - 1`, bypassing the seconds-based timelock at any timestamp ≥ `T_c`.

pub fn py_check_time_locks(
    removal_coin_records: HashMap<Bytes32, CoinRecord>,
    bundle_conds: &OwnedSpendBundleConditions,
    prev_transaction_block_height: u32,
    timestamp: u64,
    nowrap: bool,
) -> PyResult<Option<u32>> {
    let res = check_time_locks(
        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,
        nowrap,
    );

    match res {
        Ok(()) => Ok(None),
        Err(ec) => Ok(Some(ec.error_code().into())),
    }
}
