### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS accepts any arbitrary felt value as a new class hash without verifying that the hash corresponds to a previously declared class. An attacker can exploit this to replace a contract's class hash with an undeclared value, rendering the contract permanently non-executable and freezing any funds it holds.

---

### Finding Description

The `execute_replace_class` function in `syscall_impls.cairo` processes the `replace_class` syscall. It reads the requested new class hash directly from the syscall request and writes it into the contract state without performing any validation that the hash exists in the `contract_class_changes` dictionary (i.e., that it was previously declared via a `declare` transaction).

The code at lines 896–910 is:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment at line 898 explicitly acknowledges the missing check. The `contract_class_changes` dictionary — which tracks all declared classes for the current block — is available as an implicit argument to the function but is never consulted. Any felt value, including one that has never been declared, is accepted as a valid new class hash.

This is directly analogous to the ETH2.0 staking vulnerability: just as the deposit contract fails to verify the expected deposit root before accepting a deposit (allowing an attacker to manipulate which account receives the deposit), the StarkNet OS fails to verify the expected class commitment before accepting a class replacement, allowing an attacker to corrupt a contract's class pointer irreversibly.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared value, the OS has no corresponding compiled class (CASM) to execute. Any subsequent call to the contract — including calls to withdraw or transfer funds — will fail at the class-lookup stage. Because the state transition is committed to the global state root and proven on L1, the corruption is permanent and irreversible. All assets held in the contract (ERC-20 token balances, bridged ETH via L1→L2 messages, etc.) are permanently frozen with no recovery path.

---

### Likelihood Explanation

The attack is straightforward and requires no privileged access:

1. An attacker deploys a contract that appears legitimate (e.g., a token vault or DeFi protocol).
2. The contract attracts user deposits via L1→L2 messages or direct token transfers.
3. The attacker calls `replace_class` from within the contract, supplying an arbitrary undeclared felt (e.g., `1`) as the new class hash.
4. The OS's `execute_replace_class` accepts the call without validation.
5. The contract's class hash in the global state is updated to the invalid value.
6. All future calls to the contract fail; user funds are permanently frozen.

The attacker is an unprivileged transaction sender. No leaked keys, privileged roles, or external dependencies are required. The missing check is confirmed by the in-code TODO comment, indicating the developers are aware of the gap.

---

### Recommendation

Inside `execute_replace_class`, before writing the new class hash to `contract_state_changes`, verify that `class_hash` exists as a key in `contract_class_changes` (i.e., it was declared in the current or a prior block). Concretely:

```cairo
// Verify the new class hash has been declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the invariant already enforced in `execute_declare_transaction`, where `prev_value=0` ensures a class is declared at most once and `assert_not_zero(compiled_class_hash)` ensures the compiled hash is non-zero.

---

### Proof of Concept

1. Declare a valid class `C` and deploy a contract `A` with class `C`. Fund `A` with tokens.
2. From within `A`, issue the `replace_class` syscall with `class_hash = 0xdeadbeef` (an undeclared felt).
3. The OS executes `execute_replace_class`:
   - `class_hash = 0xdeadbeef` is read from the request.
   - The TODO check is absent; no lookup into `contract_class_changes` occurs.
   - `dict_update` writes `StateEntry(class_hash=0xdeadbeef, ...)` for contract `A`.
4. The block is proven and the state root is updated on L1.
5. Any subsequent call to `A` (e.g., a withdrawal) causes the OS to look up class `0xdeadbeef` in the compiled class facts — it does not exist — and execution fails permanently.
6. All funds in `A` are frozen with no recovery mechanism.
