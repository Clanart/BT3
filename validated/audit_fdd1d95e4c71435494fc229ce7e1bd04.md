### Title
`StateDB.Commit()` Silently Overwrites Native-Action Account Nonce/CodeHash Without Conflict Detection — (File: `x/evm/statedb/statedb.go`)

---

### Summary

The conflict-detection guard in `StateDB.Commit()` only covers EVM-dirty **storage slots**. It does not cover EVM-dirty **account metadata** (nonce, code hash). When a native action (via `ExecuteNativeAction`) modifies an account's nonce or code hash, and the EVM also marks the same account's nonce or code hash dirty in the same transaction, the EVM's `SetAccount` write silently overwrites the native action's write — with no detection and no error. This is the direct structural analog to the SBT balance-entry overwrite: an unconditional insert/set clobbers pre-existing state belonging to the same key.

---

### Finding Description

**Root cause — missing conflict check for account metadata.**

`Commit()` enforces a non-overlap invariant for storage slots before flushing native cache layers:

```go
// x/evm/statedb/statedb.go  lines 766-788
for _, addr := range s.journal.sortedDirties() {
    obj, exist := s.stateObjects[addr]
    if !exist || obj.selfDestructed {
        continue
    }
    for _, key := range obj.dirtyStorage.SortedKeys() {
        origin := obj.originStorage[key]
        dirty  := obj.dirtyStorage[key]
        if dirty == origin { continue }
        store := s.keeper.GetState(s.ctx, obj.Address(), key)
        if store != origin && store != dirty {
            return fmt.Errorf("%w: ...", ErrStateConflict, ...)
        }
    }
}
