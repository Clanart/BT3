### Title
Native Fee (NEAR) Permanently Locked When Pending Transfer Is Never Finalized — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

When a user initiates a NEAR-origin transfer with a non-zero `native_token_fee`, that NEAR amount is immediately deducted from the user's storage balance and held inside the bridge contract. If the transfer is never finalized on the destination chain — and therefore `claim_fee` is never called — the native fee NEAR has no recovery path. The only user-facing recovery function, `storage_unregister(force=true)`, refunds only `required_balance_for_account() + storage.available`, which no longer includes the already-deducted native fee. No DAO-level NEAR rescue function exists. The native fee is permanently locked.

---

### Finding Description

**Step 1 — Native fee is deducted from user's storage balance at transfer initiation.**

In `init_transfer_internal` the required storage balance is computed as the on-chain storage cost of the pending-transfer entry **plus** the full native fee amount:

```rust
let required_storage_balance = self
    .add_transfer_message(transfer_message.clone(), storage_owner.clone())
    .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));
```

`try_update_storage_balance` then subtracts this combined amount from `storage_balance.available` (without touching `storage_balance.total`). The native fee NEAR is now held inside the contract's account balance, tracked only implicitly through the pending `TransferMessage`.

**Step 2 — Transfer removal only refunds the storage cost, not the native fee.**

`remove_transfer_message` measures the freed storage bytes and credits only that amount back to the owner's `available` balance:

```rust
let refund = env::storage_byte_cost()
    .saturating_mul((storage_usage - env::storage_usage()).into());
if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
    storage.available = storage.available.saturating_add(refund);
    ...
}
```

The native fee portion is never returned here.

**Step 3 — `storage_unregister(force=true)` cannot recover the native fee.**

The only user-callable recovery path is:

```rust
let refund = self
    .required_balance_for_account()
    .saturating_add(storage.available);
Promise::new(account_id).transfer(refund).detach();
```

Because `storage.available` was already reduced by `storage_cost + native_fee` at initiation, the refund is `required_balance_for_account() + (initial_available − storage_cost − native_fee)`. The native fee is silently excluded.

**Step 4 — No DAO-level NEAR rescue exists.**

`transfer_token_as_dao` only dispatches NEP-141 `ft_transfer` / `ft_transfer_call` calls; it cannot transfer native NEAR out of the contract. There is no `transfer_near_as_dao` or equivalent.

**Step 5 — `claim_fee` is the only path that pays out the native fee, but it requires a destination-chain finalization proof.**

`claim_fee` calls `verify_proof` and then `send_fee_internal`, which executes `Promise::new(fee_recipient).transfer(native_fee)`. If the transfer is never finalized on the destination chain, no valid `FinTransferMessage` proof can be produced, so `claim_fee` can never be called and the NEAR is irrecoverable.

---

### Impact Explanation

Any user who initiates a NEAR-origin transfer with `native_token_fee > 0` and whose transfer is never finalized on the destination chain permanently loses that NEAR. The amount is user-controlled and unbounded. Neither the user nor the DAO can recover it under the current protocol. This constitutes **permanent freezing / irrecoverable lock of user funds in the bridge flow**.

---

### Likelihood Explanation

The scenario is reachable by any unprivileged bridge user:

1. User calls `ft_transfer_call` → `ft_on_transfer` → `init_transfer` with `native_token_fee > 0`.
2. No relayer picks up the transfer (fee too low, relayer outage, destination chain congestion), **or** a relayer signs it but the destination chain permanently rejects / never processes it.
3. User calls `storage_unregister(force=true)` to exit — native fee is gone.

The yield-resume mechanism (`init_transfer_resume`) shows the protocol already anticipates transfers that stall; the native fee loss in that stall path is the same.

---

### Recommendation

Add a user-callable `cancel_transfer` function that:
1. Verifies the caller is the transfer owner.
2. Removes the pending transfer from `pending_transfers`.
3. Refunds both the storage cost **and** the native fee (`transfer_message.fee.native_fee.0` yoctoNEAR) to the owner's storage balance or directly to their account.

Alternatively, modify `remove_transfer_message` to also credit `native_fee` back to the owner's available balance whenever a transfer is removed without a successful `claim_fee`.

---

### Proof of Concept

1. User deposits storage and initiates a transfer with `native_token_fee = 10 NEAR`:
   - `storage.available` is reduced by `storage_cost + 10 NEAR`.
2. No relayer finalizes the transfer on the destination chain.
3. User calls `storage_unregister(force=Some(true))`.
4. Refund = `required_balance_for_account() + storage.available` — the 10 NEAR native fee is absent.
5. The 10 NEAR remains in the contract's balance with no callable function to extract it.
