### Title
Push-Pattern `finTransfer` Permanently Locks Funds When EVM Recipient Is Blacklisted in a Restricted Token (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` uses a push pattern to deliver tokens to the recipient in a single atomic EVM transaction. When the bridged token enforces transfer restrictions (e.g., USDC's blacklist), a blacklisted recipient address causes `safeTransfer` to revert, rolling back the entire EVM transaction. Because the NEAR side has already irrevocably burned or locked the user's tokens and no on-chain cancellation path exists for the pending transfer, the funds are permanently frozen.

---

### Finding Description

In `OmniBridge.finTransfer`, the destination nonce is marked used and then tokens are pushed to the recipient in one atomic call:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol
completedTransfers[payload.destinationNonce] = true;   // line 287
...
IERC20(payload.tokenAddress).safeTransfer(             // line 351
    payload.recipient,
    payload.amount
);
```

If `payload.tokenAddress` is USDC (or any ERC-20 with a blacklist / transfer-restriction hook) and `payload.recipient` is on that token's blacklist, `safeTransfer` reverts. Because Solidity reverts are atomic, the `completedTransfers` write also rolls back â€” the nonce is not permanently consumed. However, the NEAR-side state **is** permanently committed:

- For bridged tokens: `burn_tokens_if_needed` has already burned the tokens.
- For native tokens: `lock_tokens_if_needed` has already incremented the locked-token counter.
- The `pending_transfers` entry on NEAR has no expiry and no user-callable cancellation function.

Every subsequent attempt to call `finTransfer` with the same payload will always revert for the same reason. There is no alternative delivery path (pull pattern, redirect, or admin rescue) in the EVM contract, and no `cancel_transfer` / refund function exists on the NEAR bridge contract for this case.

The NEAR-side callback `fin_transfer_send_tokens_callback` does handle failed `ft_transfer` calls on NEAR-to-NEAR flows by reverting lock actions, but this callback is never invoked for EVM-destination transfers â€” the EVM `finTransfer` is a standalone atomic call with no cross-chain callback.

---

### Impact Explanation

**Critical â€” Permanent freezing / irrecoverable lock of user funds.**

A user who bridges USDC (or any blacklist-enforcing token) from NEAR to an EVM chain loses their tokens permanently if the recipient address is or becomes blacklisted:

- Tokens are burned/locked on NEAR and cannot be recovered.
- The EVM transfer can never succeed.
- No protocol-level rescue path exists.

---

### Likelihood Explanation

**Medium-High.** USDC is one of the most commonly bridged stablecoins and Circle actively blacklists addresses (OFAC sanctions, exchange compliance). A user whose address is blacklisted after initiating a transfer â€” or who mistakenly sends to a known blacklisted address â€” triggers this condition. The scenario requires no privileged access and no attacker cooperation; it is a natural consequence of normal protocol usage combined with token-level enforcement.

---

### Recommendation

1. **Pull pattern for restricted tokens**: Instead of pushing tokens to the recipient in `finTransfer`, credit an internal balance mapping and let the recipient call a `claim` function. This decouples delivery failure from transfer finalization.
2. **Off-chain pre-check**: The relayer should verify that the recipient is not blacklisted in the token contract before submitting `finTransfer`. If blacklisted, the transfer should be routed to a separate recovery flow.
3. **NEAR-side cancellation**: Add a time-locked `cancel_transfer` function on the NEAR bridge that allows refunding a pending transfer whose EVM finalization has been permanently blocked, mirroring the `fin_transfer_send_tokens_callback` recovery logic already used for NEAR-to-NEAR flows.

---

### Proof of Concept

**Setup:**
- NEAR bridge has USDC (a token with blacklist enforcement) registered.
- Alice holds 1,000 USDC on NEAR and initiates a transfer to her EVM address `0xAlice`.

**Step 1 â€” NEAR side (irreversible):**
Alice calls `ft_transfer_call` â†’ `init_transfer_internal` is invoked:
- `burn_tokens_if_needed` burns 1,000 USDC on NEAR. [1](#0-0) 
- A `TransferMessage` is stored in `pending_transfers`. [2](#0-1) 
- `InitTransferEvent` is emitted; NEAR state is committed.

**Step 2 â€” Between chains:**
Circle blacklists `0xAlice` (e.g., OFAC sanction). The relayer picks up the event and prepares to call `finTransfer` on EVM.

**Step 3 â€” EVM side (always reverts):**
Relayer calls `OmniBridge.finTransfer(signatureData, payload)`:

```solidity
completedTransfers[payload.destinationNonce] = true;  // line 287 â€” will be rolled back
...
IERC20(payload.tokenAddress).safeTransfer(            // line 351
    payload.recipient,   // 0xAlice â€” blacklisted
    payload.amount
);
// USDC reverts â†’ entire tx reverts â†’ nonce not consumed
``` [3](#0-2) 

**Step 4 â€” Permanent lock:**
- Every retry of `finTransfer` reverts identically.
- The NEAR `pending_transfers` entry has no expiry and no cancellation path.
- Alice's 1,000 USDC is permanently destroyed on NEAR and undeliverable on EVM.

The same revert path applies to the native-ETH branch (`FailedToSendEther` at line 322) when the recipient is a contract that rejects ETH, and to the ERC-1155 branch (`safeTransferFrom` at line 324) when the recipient does not implement `onERC1155Received`. [4](#0-3)
