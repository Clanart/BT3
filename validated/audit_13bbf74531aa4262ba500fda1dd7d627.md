### Title
`batch_insert` Bypasses Duplicate Key/Hash Validation, Silently Corrupting DataLayer Merkle Tree Root — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` omits the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. When a caller supplies a batch containing a key or hash that already exists in the tree (or that appears twice within the same batch), `batch_insert` silently writes both leaf nodes into the blob, advances all internal state, and returns `Ok(())`. The resulting Merkle root is computed over the duplicate leaves and is therefore incorrect; any inclusion/exclusion proof generated against it is based on a corrupted root.

---

### Finding Description

`MerkleBlob::insert` (the single-item path) guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a different code path for all items when `leaf_count > 1`, and for all items except the last two when `leaf_count <= 1`. In that fast path (lines 587–603) it calls `get_new_index()` and `insert_entry_to_blob()` directly, with **no duplicate check**:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` writes the block to the blob and calls `block_status_cache.add_leaf`, which overwrites the key→index mapping for any duplicate key: [3](#0-2) 

After a duplicate insertion the blob contains **two leaf nodes with the same key** at different indexes. The cache's `key_to_index` map points only to the second one, while the first remains an orphaned-but-reachable node in the blob. `calculate_lazy_hashes` traverses both nodes and folds them into the root hash, producing a root that does not correspond to any valid set of key-value pairs. `check_integrity` will subsequently detect the mismatch: [4](#0-3) 

but by then the corrupted root has already been committed and any proofs derived from it are invalid.

The Python binding `py_batch_insert` exposes this path directly to callers without adding any deduplication: [5](#0-4) 

The existing fuzz targets for `insert` explicitly handle `KeyAlreadyPresent` and `HashAlreadyPresent`, confirming the developers know duplicates can arrive from untrusted input — but no fuzz target exercises `batch_insert` with duplicates: [6](#0-5) 

---

### Impact Explanation

**High — DataLayer Merkle tree root corruption enabling forged inclusion/exclusion proofs.**

A corrupted root means:
- Proofs of inclusion for the orphaned duplicate leaf are verifiable against the wrong root.
- Proofs of exclusion for a key that is actually present can be constructed against the wrong root.
- Any downstream system that trusts the root (e.g., a DataLayer proof verifier) will accept invalid state as valid.

---

### Likelihood Explanation

**Medium.** The DataLayer service calls `batch_insert` with key-value data supplied by DataLayer store operators or their clients. If the service does not deduplicate keys before calling `batch_insert` — or if an operator deliberately submits a batch with a repeated key — the tree is silently corrupted. The Python binding is directly reachable from application code with no additional privilege required.

---

### Recommendation

1. Add the same duplicate-key and duplicate-hash guards to the fast path of `batch_insert` that `insert` already enforces, returning `Error::KeyAlreadyPresent` or `Error::HashAlreadyPresent` on detection.
2. Add a fuzz target that calls `batch_insert` with batches containing duplicate keys and hashes, and asserts `check_integrity` passes (or that the expected error is returned).
3. Consider making `batch_insert` transactional: collect all new indexes before writing any of them to the blob, so that a validation failure leaves the tree unmodified.

---

### Proof of Concept

```rust
let mut blob = MerkleBlob::new(vec![]).unwrap();

// Pre-populate so leaf_count > 1, forcing ALL items through the fast path.
blob.insert(KeyId(0), ValueId(0), &sha256_num(&0i64), InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(1), ValueId(1), &sha256_num(&1i64), InsertLocation::Auto {}).unwrap();

// Batch with a duplicate key (KeyId(0) already exists).
let batch = vec![
    ((KeyId(2), ValueId(2)), sha256_num(&2i64)),
    ((KeyId(0), ValueId(99)), sha256_num(&99i64)), // duplicate key
];

// Returns Ok(()) — no error despite the duplicate.
blob.batch_insert(batch).unwrap();

// Tree is now corrupted: two leaf nodes with KeyId(0) exist in the blob.
// check_integrity() will fail, and the Merkle root is wrong.
blob.check_integrity().unwrap_err(); // IntegrityKeyToIndexCacheIndex or similar
```

The blob now contains two leaves for `KeyId(0)` at different indexes. The root hash folds both in, producing a root that no honest tree would ever generate. Any proof verified against this root is based on forged state.

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
```
