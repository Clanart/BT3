### Title
Missing Upper Bound on Computed `max_fee` in `charge_fee` Renders Block Unprovable (Network Halt) — (File: `execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` computes a fee ceiling as a raw felt arithmetic product of user-supplied resource bounds. Individual bounds are validated in `pack_resource_bounds` to be at most `max_amount < 2^64` and `max_price_per_unit < 2^128`, but **no check is placed on their product**. The product can reach ~2^192. When `charge_fee` subsequently calls `assert_nn_le(actual_fee, max_fee)`, the Cairo range-check builtin requires both `actual_fee < 2^128` and `max_fee − actual_fee < 2^128`. If `max_fee ≥ 2^129`, no valid `actual_fee` can satisfy both constraints simultaneously, making the block permanently unprovable.

---

### Finding Description

**Root cause — `compute_max_possible_fee` (transaction_impls.cairo:87–102):**

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

The individual bounds are validated in `pack_resource_bounds` (transaction_hash.cairo:103–108):

```cairo
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // max_amount < 2^64
assert_nn(resource_bounds.max_price_per_unit);             // max_price_per_unit < 2^128
``` [2](#0-1) 

These per-field checks are sound in isolation, but the **product** `max_amount × max_price_per_unit` can be as large as `(2^64 − 1) × (2^128 − 1) ≈ 2^192`, far exceeding 2^128.

**Failure site — `charge_fee` (transaction_impls.cairo:121–135):**

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);

if (max_fee == 0) { return (); }

local low_actual_fee;
%{ LoadActualFee %}
...
assert_nn_le(calldata.amount.low, max_fee);   // ← fails when max_fee ≥ 2^129
``` [3](#0-2) 

`assert_nn_le(a, b)` decomposes into:
1. `assert_nn(a)` → requires `a ∈ [0, 2^128)`
2. `assert_nn(b − a)` → requires `b − a ∈ [0, 2^128)`

For both to hold: `a < 2^128` **and** `b − a < 2^128`, which together require `b < 2^129`.

When `max_fee ≥ 2^129`, the sequencer-supplied `actual_fee` (loaded via hint `LoadActualFee`) cannot satisfy both constraints regardless of its value:
- If `actual_fee < 2^128` (required by constraint 1), then `max_fee − actual_fee ≥ 2^129 − (2^128 − 1) = 2^128 + 1 ≥ 2^128` → constraint 2 fails.
- No escape exists.

The `tip` field is bounded to `< 2^64` by `assert_nn_le(tip, 2 ** 64 - 1)` in `hash_fee_fields`, so it does not change the analysis. [4](#0-3) 

---

### Impact Explanation

If a sequencer includes a transaction whose `max_fee ≥ 2^129`, the Cairo OS proof for that block **cannot be generated**. The block is permanently stuck — it cannot be finalized on L1, and no subsequent block can build on it. This constitutes a **total network halt** (High severity per the allowed impact scope).

---

### Likelihood Explanation

Any unprivileged transaction sender can craft a valid-looking V3 invoke, declare, or deploy-account transaction with:
- `l1_gas.max_amount = 3`
- `l1_gas.max_price_per_unit = 2^128 − 1`

This yields `max_fee = 3 × (2^128 − 1) = 3 × 2^128 − 3 > 2^129`. Each individual field passes `pack_resource_bounds` validation. The transaction hash is computed correctly. The OS has no gate between hash validation and `charge_fee` that checks the product. A sequencer that does not independently enforce `max_fee < 2^129` before block inclusion (which the OS protocol does not mandate) will produce an unprovable block upon including such a transaction.

---

### Recommendation

Add an explicit upper-bound check on the computed `max_fee` before calling `assert_nn_le`. The minimal fix is to assert `max_fee < 2^128` (or `< 2^129`) immediately after computing it in `charge_fee`, or equivalently to add a `assert_nn_le(max_fee, MAX_FEE_BOUND)` guard in `compute_max_possible_fee`. Alternatively, bound `max_price_per_unit` more tightly in `pack_resource_bounds` (e.g., to 64 bits) so that the product of any two validated fields cannot exceed 2^128.

---

### Proof of Concept

1. Attacker submits a V3 invoke transaction with resource bounds:
   - `L1_GAS`: `max_amount = 3`, `max_price_per_unit = 2^128 − 1`
   - `L2_GAS`: `max_amount = 0`, `max_price_per_unit = 0`
   - `L1_DATA_GAS`: `max_amount = 0`, `max_price_per_unit = 0`

2. `pack_resource_bounds` is called for each resource type during hash computation — all three pass individually (`3 ≤ 2^64 − 1`, `2^128 − 1 < 2^128`). [5](#0-4) 

3. Transaction hash is computed and verified normally. Nonce is incremented. Execution proceeds.

4. `charge_fee` calls `compute_max_possible_fee`:
   ```
   max_fee = 3 × (2^128 − 1) = 3 × 2^128 − 3  ≈ 2^129.58
   ``` [6](#0-5) 

5. `max_fee ≠ 0`, so execution continues to `assert_nn_le(actual_fee, max_fee)`.

6. The sequencer hint `LoadActualFee` provides any `actual_fee`. For any choice:
   - `actual_fee < 2^128` → `max_fee − actual_fee > 2^128` → `assert_nn(max_fee − actual_fee)` **fails**.
   - `actual_fee ≥ 2^128` → `assert_nn(actual_fee)` **fails**. [7](#0-6) 

7. The Cairo proof for the block cannot be generated. The block is permanently unprovable → **network halt**.

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```

func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn(resource_bounds.max_price_per_unit);
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
