Audit Report

## Title
Unguarded `refundETH()` allows any caller to steal ETH stranded on the router from a prior transaction — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.refundETH()` contains no access control and unconditionally transfers `address(this).balance` to `msg.sender`. Because `pay()` wraps only the exact swap-required ETH and leaves any surplus on the contract, a user who calls a payable swap entry point (e.g., `exactInputSingle`) with excess `msg.value` — without atomically bundling `refundETH()` in the same `multicall` — leaves residual ETH on the router that any subsequent caller can immediately steal.

## Finding Description

`refundETH()` sends the router's entire native balance to the caller with no per-caller accounting or access restriction: [1](#0-0) 

Inside `pay()`, when `token == WETH` and `nativeBalance >= value`, exactly `value` wei is wrapped and forwarded; the remainder (`nativeBalance - value`) is never returned and stays on the router after the transaction ends: [2](#0-1) 

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses: [3](#0-2) 

It does **not** prevent ETH from arriving via `msg.value` on any `payable` entry point. `exactInputSingle` is `external payable` and performs no auto-refund after the swap completes: [4](#0-3) 

The same applies to `exactInput`, `exactOutputSingle`, `exactOutput`, and `multicall` — all are `payable` with no surplus-ETH recovery at the end of standalone calls.

## Impact Explanation

Any ETH left on the router between transactions is unconditionally claimable by any arbitrary address via `refundETH()`. This is a **direct, unconditional theft of user principal** — the attacker receives ETH that the victim overpaid, with no preconditions beyond the victim's excess ETH being present on the router. Impact: **High**.

## Likelihood Explanation

- `exactInputSingle` and the other swap functions are standalone `payable` entry points; users and integrators routinely send a round-number ETH value to cover an uncertain `amountIn`.
- The multicall-plus-`refundETH` pattern is not enforced at the router level; a standalone swap call leaves no automatic refund.
- A mempool observer can trivially detect a transaction that leaves ETH on the router and immediately call `refundETH()` in the next block.

Likelihood: **Medium** (requires victim to not use the multicall pattern, but this is a realistic and common mistake).

## Recommendation

1. **Track per-payer ETH deposits** in transient storage at the start of each payable entry point and restrict `refundETH()` to return only the amount attributed to `msg.sender`.
2. **Auto-refund excess ETH** at the end of each standalone swap function (mirror the `multicall` + `refundETH` pattern internally), so no ETH can ever be stranded between transactions.

## Proof of Concept

The existing test `test_refundETH_sendsBalanceToCaller` already demonstrates that any caller receives the full router balance: [5](#0-4) 

Full exploit path:

```
1. Victim calls:
   router.exactInputSingle{value: 2 ether}(
     ExactInputSingleParams({
       tokenIn: WETH,
       amountIn: 1 ether,   // only 1 ether consumed by pay()
       ...
     })
   );
   → pay() wraps exactly 1 ether; 1 ether remains on router.

2. Attacker (any EOA) calls in the next transaction:
   router.refundETH();
   → _transferETH(msg.sender, 1 ether) executes.
   → Attacker receives victim's 1 ether.
```
