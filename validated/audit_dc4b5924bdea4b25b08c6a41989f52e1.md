### Title
Missing `max_l2_gas_amount` Bound Enforcement for `Declare` Transactions in Gateway Stateless Validator - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

The `StatelessTransactionValidator::validate_resource_bounds` function enforces a `max_l2_gas_amount` ceiling on `l2_gas.max_amount` for `Invoke` and `DeployAccount` transactions, but explicitly skips this check for `Declare` transactions. Any user can submit a `Declare` transaction with `l2_gas.max_amount` set to `u64::MAX` (or any value exceeding `max_l2_gas_amount`, which is `1_200_000_000` in production) and the gateway will admit it, bypassing the admission-control invariant that is enforced for every other transaction type.

---

### Finding Description

Inside `validate_resource_bounds`, the guard is written as an explicit no-op branch for `Declare`:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← nothing; check is intentionally skipped
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The production default for `max_l2_gas_amount` is `1_200_000_000` (matching the block's `receipt_l2_gas` bouncer cap of `5_800_000_000`). [2](#0-1) 

The asymmetry is confirmed by the dedicated test `valid_l2_gas_amount_on_declare`, which asserts that a `Declare` with `max_amount: GasAmount(200)` passes even when `max_l2_gas_amount: 100`: [3](#0-2) 

Conversely, the same over-limit amount on `Invoke` or `DeployAccount` is rejected: [4](#0-3) 

The top-level `validate` dispatcher calls `validate_resource_bounds` for all transaction types before any type-specific branch, so there is no compensating check elsewhere in the stateless path. [5](#0-4) 

---

### Impact Explanation

**High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

The `max_l2_gas_amount` limit exists to prevent transactions whose declared gas ceiling exceeds what any single block can accommodate from entering the mempool. A `Declare` transaction with `l2_gas.max_amount = u64::MAX` (≈ 18 × 10¹⁸, versus the block cap of ~5.8 × 10⁹) passes all gateway stateless checks and is forwarded to the mempool. The mempool uses the declared `max_amount × max_price_per_unit` to compute the maximum possible fee for ordering and balance checks; an astronomically large declared amount can cause the `max_possible_fee` computation to overflow or produce a fee that no account can satisfy, leading to incorrect admission decisions or incorrect fee-bound enforcement downstream. The gateway's admission-control invariant — that no transaction claiming more L2 gas than the block can hold is accepted — is broken for `Declare` transactions.

---

### Likelihood Explanation

**High.** The bypass requires no privilege, no special account, and no knowledge of internal state. Any user submitting a `Declare` transaction can set `l2_gas.max_amount` to an arbitrary value. The code path is unconditionally reachable through the public HTTP endpoint. The TODO comment in the source confirms the gap is known but unresolved.

---

### Recommendation

Remove the `Declare`-exemption branch and apply the same `max_l2_gas_amount` guard uniformly to all transaction types:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

Delete the `valid_l2_gas_amount_on_declare` test (which encodes the broken behaviour) and add a corresponding failure case for `Declare` to `test_invalid_max_l2_gas_amount`. [1](#0-0) 

---

### Proof of Concept

1. Construct a valid `RpcDeclareTransaction::V3` with:
   ```
   resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)
   resource_bounds.l2_gas.max_price_per_unit = GasPrice(min_gas_price)   // satisfies min_gas_price check
   ```
2. Submit to the gateway's `add_transaction` endpoint.
3. Observe: `StatelessTransactionValidator::validate` returns `Ok(())` — the transaction is admitted to the mempool.
4. Repeat with an `Invoke` transaction carrying the same `l2_gas.max_amount`; observe `Err(MaxGasAmountTooHigh)` — the check fires correctly.

The test fixture `valid_l2_gas_amount_on_declare` already encodes step 3 as an expected-pass assertion, confirming the bypass is reachable with zero additional setup. [3](#0-2)
