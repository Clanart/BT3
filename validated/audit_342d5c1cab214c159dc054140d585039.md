### Title
Unconstrained Block Hash Written to Registry Enables Fake Proof-Fact Acceptance — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/os_utils.cairo`)

---

### Summary

The `write_block_number_to_block_hash_mapping` function writes a hint-supplied, Cairo-unconstrained block hash value into the on-chain block hash registry. Because no Cairo assertion verifies this value against any canonical source, a malicious prover can inject an arbitrary hash for any historical block. The corrupted registry is then consumed by `check_proof_facts`, which uses it as the sole on-chain anchor to validate virtual OS proof facts submitted inside invoke transactions. A fake registry entry allows fake proof facts to pass validation, enabling acceptance of invalid state transitions and direct loss of funds.

---

### Finding Description

**Root cause — `write_block_number_to_block_hash_mapping`** [1](#0-0) 

```cairo
// Currently, the block hash mapping is not enforced by the OS.
// TODO(Yoni, 1/1/2026): output this hash.
local old_block_hash;
%{ GetOldBlockNumberAndHash %}

// Update mapping.
assert state_entry.class_hash = 0;
assert state_entry.nonce = 0;
tempvar storage_ptr = state_entry.storage_ptr;
assert [storage_ptr] = DictAccess(key=old_block_number, prev_value=0, new_value=old_block_hash);
```

`old_block_hash` is a `local` variable populated exclusively by the hint `GetOldBlockNumberAndHash`. There is **no Cairo `assert` or range-check** that constrains `old_block_hash` to equal the actual canonical hash of block `N - STORED_BLOCK_HASH_BUFFER`. The comment itself confirms the absence of enforcement. Because hints are not part of the STARK proof, a malicious prover can assign any field element to `old_block_hash` and produce a valid proof.

The value is then written as `new_value` into the `DictAccess` entry for the block hash contract at address `BLOCK_HASH_CONTRACT_ADDRESS = 0x1`. [2](#0-1) 

**Secondary confirmation — `get_block_hashes` / `get_block_os_output_header`** [3](#0-2) 

`previous_block_hash` is also a hint-supplied `local` with no Cairo constraint. `%{ CheckBlockHashConsistency %}` is a hint-only check — it is invisible to the verifier. The `get_block_os_output_header` comment explicitly states: *"both the previous block hash and previous state root are guessed, and the OS does not verify their consistency."* [4](#0-3) 

**Downstream consumer — `check_proof_facts`** [5](#0-4) 

`check_proof_facts` is the security gate for virtual OS proof facts embedded in invoke transactions. Its only on-chain anchor is the call to `read_block_hash_from_storage`, which reads `base_block_hash` from the same storage dict that was populated by `write_block_number_to_block_hash_mapping`. If that dict entry holds a fake hash, then any proof fact claiming `base_block_hash = fake_hash` for the corresponding `base_block_number` passes the check. [6](#0-5) 

`read_block_hash_from_storage` asserts `prev_value = expected_block_hash`. If the dict was seeded with the fake value, the dict-squashing mechanism will enforce consistency with that fake value — not with the true canonical hash.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A malicious prover injects `fake_hash` as the stored hash for block `N-10`. An attacker then submits an invoke transaction whose proof facts are anchored to `base_block_hash = fake_hash`. `check_proof_facts` reads the corrupted registry, finds a match, and accepts the proof facts as a valid virtual OS run. The virtual OS output can encode arbitrary state transitions — including minting tokens or transferring balances — that are then applied to the live state. Funds can be created from nothing or stolen from existing accounts.

---

### Likelihood Explanation

The prover in a ZK system is architecturally **not** a trusted role — the entire security model of STARK proofs is that correctness holds even against a malicious prover. The missing Cairo constraint is a direct violation of this model. Any entity that can submit a block proof (i.e., the sequencer/prover, which in a decentralized future is permissionless) can exploit this. The code's own TODO comment (`TODO(Yoni, 1/1/2026): output this hash`) confirms the gap is known and unresolved.

---

### Recommendation

**Short term:** Add a Cairo `assert` in `write_block_number_to_block_hash_mapping` that constrains `old_block_hash` to equal the value committed in the OS output of the corresponding prior block. Specifically, the OS output for block `N-10` already contains `new_block_hash`; that value must be threaded into the current block's execution context and asserted equal to `old_block_hash` before it is written to storage.

**Long term:** Implement the TODO at `block_hash.cairo:61` — compute `header_commitments` and `gas_prices_hash` inside the OS rather than guessing them via hints, so that `new_block_hash` is fully determined by Cairo constraints. Remove the `previous_block_hash` guess and instead derive it from the verified `new_block_hash` of the preceding block, establishing a cryptographically enforced chain linkage analogous to the block hash chain in Ethereum.

---

### Proof of Concept

1. Prover processes block `N`. During `pre_process_block`, `write_block_number_to_block_hash_mapping` is called.
2. Prover's hint implementation for `GetOldBlockNumberAndHash` returns `old_block_hash = ATTACKER_CHOSEN_VALUE` instead of the true hash of block `N-10`.
3. The Cairo code writes `DictAccess(key=N-10, prev_value=0, new_value=ATTACKER_CHOSEN_VALUE)` — no assertion fails because `old_block_hash` is unconstrained.
4. The proof for block `N` is valid and accepted on L1.
5. In block `N+k` (where `k ≥ 0`), the attacker submits an invoke transaction containing virtual OS proof facts with `base_block_number = N-10` and `base_block_hash = ATTACKER_CHOSEN_VALUE`.
6. `check_proof_facts` calls `read_block_hash_from_storage(block_number=N-10, expected_block_hash=ATTACKER_CHOSEN_VALUE)`.
7. The dict lookup finds `prev_value = ATTACKER_CHOSEN_VALUE` (written in step 3), the assertion `prev_value = expected_block_hash` passes.
8. The virtual OS proof facts — encoding arbitrary state transitions — are accepted as valid, and the attacker's fabricated state changes (e.g., balance credits) are applied.
