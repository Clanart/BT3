### Title
Storage Read/Write Gas Costs Omit Patricia Tree Update and Dict Squash Costs, Enabling Underpriced Computational Resource Consumption — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/constants.cairo`)

---

### Summary

The `STORAGE_READ_GAS_COST` and `STORAGE_WRITE_GAS_COST` constants charged to users per storage syscall do not include the cost of Patricia Merkle tree updates and dict squash operations that the OS must perform during state commitment. An unprivileged transaction sender can access arbitrarily many unique storage slots, forcing the prover to perform unbounded Patricia tree work without paying for it — a direct analog to the EVM accesslist intrinsic gas omission.

---

### Finding Description

In `constants.cairo`, the per-syscall gas costs for storage operations are defined as:

```cairo
// TODO(Yoni, 1/1/2026): take into account Patricia updates and dict squash.
const STORAGE_READ_GAS_COST = 18070;
const STORAGE_WRITE_GAS_COST = 44970;
``` [1](#0-0) 

The inline TODO is an explicit developer acknowledgment that the Patricia tree update cost and dict squash cost are **not** included in these figures.

When a transaction executes a `STORAGE_READ` or `STORAGE_WRITE` syscall, the OS deducts the corresponding flat gas cost from `remaining_gas` (tracked in `execute_invoke_function_transaction`):

```cairo
let initial_user_gas_bound = get_initial_user_gas_bound(common_tx_fields=common_tx_fields);
let remaining_gas = initial_user_gas_bound;
``` [2](#0-1) 

After **all** transactions in a block finish executing, `state_update` is called, which triggers `squash_state_changes` and then `compute_contract_state_commitment`. These functions iterate over every unique storage slot touched across the block and perform a full Patricia Merkle tree update for each contract's storage trie:

```cairo
let (squashed_os_state_update, state_update_output) = state_update{hash_ptr=pedersen_ptr}(...)
``` [3](#0-2) 

The Patricia update itself, in `hash_contract_state_changes`, performs Pedersen hashing and a 251-level Merkle tree traversal (`MERKLE_HEIGHT = 251`) for every unique storage key:

```cairo
local n_updates = (state_dict_end - state_dict_start) / DictAccess.SIZE;
...
patricia_update_using_update_constants(
    ...
    n_updates=n_updates,
    height=MERKLE_HEIGHT,
    ...
);
``` [4](#0-3) 

The cost of this work scales linearly with the number of **unique** storage slots accessed across the block. Because `STORAGE_READ_GAS_COST` and `STORAGE_WRITE_GAS_COST` do not include this cost, a user pays only the flat per-syscall fee while the prover bears unbounded Patricia tree computation.

Additionally, `squash_state_changes` in `squash.cairo` calls `squash_dict` on both the outer contract dict and each contract's inner storage dict — again proportional to unique slots accessed — and this dict squash cost is equally unaccounted for:

```cairo
let (squashed_dict_end) = squash_dict(
    dict_accesses=contract_state_changes_start,
    dict_accesses_end=contract_state_changes_end,
    squashed_dict=squashed_dict,
);
``` [5](#0-4) 

---

### Impact Explanation

The prover must perform Patricia tree updates and dict squash for every unique storage slot touched in a block. Because these costs are not charged to the transaction sender, an attacker can craft transactions that access a large number of distinct storage slots at the cost of only `STORAGE_READ_GAS_COST` (18 070 L2 gas) per slot. The actual proving work per slot involves a 251-level Pedersen-hashed Merkle traversal, which is orders of magnitude more expensive than the flat fee charged. Flooding the block with such transactions can overwhelm the prover, preventing it from generating proofs in time and halting L1 finalization — matching the **High: Network not being able to confirm new transactions** impact.

---

### Likelihood Explanation

The attack requires no special privileges. Any account that can submit an `INVOKE_FUNCTION` transaction (the standard user-facing transaction type) can call a contract that reads or writes many distinct storage keys. The attacker controls the number of unique slots accessed and can maximize the Patricia update burden while staying within the L2 gas limit. The attack is cheap relative to the proving cost it imposes.

---

### Recommendation

Update `STORAGE_READ_GAS_COST` and `STORAGE_WRITE_GAS_COST` to include the amortized cost of Patricia tree updates and dict squash operations per unique storage slot, as the existing TODO comment already identifies. A per-unique-slot surcharge (analogous to EIP-2929's "cold" access cost) should be added on top of the base syscall cost to ensure the prover's work is fully compensated.

---

### Proof of Concept

1. Attacker deploys a contract `Spammer` with a function that loops over `N` distinct storage keys and calls `STORAGE_READ` on each.
2. Attacker submits an `INVOKE_FUNCTION` transaction calling `Spammer` with a large `N`, staying within the L2 gas limit (`EXECUTE_MAX_SIERRA_GAS = 1_100_000_000`). At `STORAGE_READ_GAS_COST = 18_070`, this allows roughly **60 000 unique storage reads** per transaction.
3. The sequencer charges the attacker `N × 18_070` L2 gas and accepts the transaction.
4. After block execution, `state_update` → `squash_state_changes` → `compute_contract_state_commitment` → `hash_contract_state_changes` performs a 251-level Patricia Merkle traversal for each of the `N` unique slots — work that was never charged.
5. By submitting many such transactions per block, the attacker forces the prover to perform unbounded Patricia tree work, delaying or preventing proof generation and blocking L1 finalization.
