### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS Cairo program does not verify that the new class hash supplied by a contract actually corresponds to a declared class in the contract class tree. An unprivileged contract deployer or transaction sender can call `replace_class` with an arbitrary, undeclared class hash. The OS will accept the state transition, produce a valid proof, and commit it to L1. The affected contract's class hash will then point to a non-existent class, making the contract permanently non-functional and permanently freezing all funds held within it.

---

### Finding Description

In `execute_replace_class` (lines 877–916 of `syscall_impls.cairo`), after gas is deducted and the new `class_hash` is read from the request, the OS immediately writes the new class hash into `contract_state_changes` without any check that the hash exists in `contract_class_changes`:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The developer-inserted `TODO` comment explicitly acknowledges this missing validation. The `contract_class_changes` dictionary — which tracks which class hashes have been declared in the current block — is never consulted. The OS therefore accepts any felt value as a valid replacement class hash.

This is structurally analogous to the reported `TokenSaleDistributor` bug: just as `setTokenAddress` overwrites the token pointer without providing a way to recover the old tokens, `execute_replace_class` overwrites the contract's class pointer without verifying the new pointer is reachable, making the contract's stored value (funds) permanently inaccessible.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once the OS commits a block containing a `replace_class` call with an undeclared hash:

1. The contract's `StateEntry.class_hash` in the global state tree is set to a hash that has no corresponding compiled class.
2. Every subsequent call to that contract — including any entry point that would allow fund withdrawal — will fail at class resolution time, because the class does not exist.
3. The state transition is proven and committed to L1 via the OS output. There is no protocol-level rollback mechanism.
4. All ERC-20 balances, ETH, or other assets held in the contract's storage are permanently frozen with no recovery path.

---

### Likelihood Explanation

**Medium.**

The `replace_class` syscall is callable by any contract from within its own execution context — no privileged role is required. Realistic trigger paths include:

- A contract deployer intentionally deploys a contract that calls `replace_class(arbitrary_felt)` to grief users who deposited funds.
- A buggy contract (e.g., one that reads the new class hash from user-supplied calldata) accidentally passes an undeclared hash.
- A reentrancy or logic-error attack in a shared contract (vault, bridge, multisig) causes `replace_class` to be called with attacker-controlled input.

The missing check is confirmed by the in-code `TODO` comment, meaning the gap is known and present in the current production code path.

---

### Recommendation

Before writing the new `class_hash` into `contract_state_changes`, verify that the hash exists in `contract_class_changes` (or in the pre-existing class tree). Concretely, perform a `dict_read` on `contract_class_changes` for `class_hash` and assert the result is non-zero (i.e., the class has been declared either in this block or in a prior block). This is exactly what the `TODO` comment calls for and what prevents the irrecoverable state transition.

---

### Proof of Concept

**Step-by-step attacker path:**

1. Attacker deploys a contract `VaultWithTrap` that:
   - Accepts ETH/token deposits from users (funds accumulate in its storage).
   - Exposes a `trap()` entry point that calls `replace_class(0xdeadbeef)` — an arbitrary, never-declared felt.

2. Attacker (or any caller) invokes `trap()` on `VaultWithTrap`.

3. Inside `execute_replace_class` (line 896–910 of `syscall_impls.cairo`), the OS reads `class_hash = 0xdeadbeef` from the request and writes it directly into `contract_state_changes` for `VaultWithTrap`'s address — **no check against `contract_class_changes` is performed**.

4. The block is proven. The OS output includes the state diff with `VaultWithTrap.class_hash = 0xdeadbeef`. The L1 Starknet core contract accepts the proof and updates the global state root.
