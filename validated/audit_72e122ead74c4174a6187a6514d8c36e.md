### Title
Missing Zero-Value Validation in `execute_replace_class` Enables Permanent Contract Bricking — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not validate that the supplied `class_hash` is non-zero. Any contract can call `replace_class(0)`, setting its own class hash to the uninitialized sentinel value (`UNINITIALIZED_CLASS_HASH = 0`). This permanently bricks the contract — no future calls can execute — and freezes any funds held in its storage with no recovery path.

---

### Finding Description

In `execute_replace_class` the OS reads `class_hash` directly from the syscall request and writes it into the contract state with no validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

There is no `assert_not_zero(class_hash)` guard. The value `0` is exactly `UNINITIALIZED_CLASS_HASH`, the sentinel used in `deploy_contract.cairo` to assert a contract slot is empty before deployment:

```cairo
assert state_entry.class_hash = UNINITIALIZED_CLASS_HASH;
``` [2](#0-1) 

The developers explicitly acknowledge the missing validation with a TODO comment at the same site:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
``` [3](#0-2) 

After `replace_class(0)` succeeds, every subsequent call to the contract will attempt to dispatch against class hash `0`. No compiled class with hash `0` exists in the `contract_class_changes` dictionary, so every call fails. The contract is permanently non-executable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ERC-20 balances or other assets stored in the contract's storage slots become permanently inaccessible. The contract cannot execute a withdrawal, transfer, or any other function. There is no OS-level recovery mechanism: the state update is committed to the Patricia Merkle Tree and propagated to L1 via the OS output, making the freeze irreversible on-chain.

---

### Likelihood Explanation

**Medium.** The `replace_class` syscall is available to every deployed contract. A malicious contract deployer — an unprivileged protocol participant — can:

1. Deploy a contract that accepts token deposits and contains a hidden or conditionally triggered path that calls `replace_class(0)`.
2. Accumulate user funds.
3. Trigger the freeze path, causing the OS to commit `class_hash = 0` for the contract with no validation.

The OS is the necessary vulnerable step: it is the only layer that could enforce `class_hash != 0` universally, and it currently does not. The TODO comment confirms the developers intended to add this check but have not yet done so.

---

### Recommendation

Add an explicit non-zero guard in `execute_replace_class` immediately after reading the request:

```cairo
let class_hash = request.class_hash;
assert_not_zero(class_hash);  // <-- add this
```

Additionally, implement the acknowledged TODO: verify that `class_hash` corresponds to a class that has been declared in `contract_class_changes` before committing the state update.

---

### Proof of Concept

1. Attacker deploys contract `C` containing a `freeze()` entry point that internally calls the `replace_class` syscall with `class_hash = 0`.
2. Users deposit tokens into `C` (e.g., via a legitimate-looking `deposit()` function).
3. Attacker sends a transaction calling `C.freeze()`.
4. The OS executes `execute_replace_class` with `request.class_hash = 0`.
5. No zero-check exists; the OS writes `StateEntry { class_hash: 0, ... }` for contract `C` into `contract_state_changes`. [4](#0-3) 

6. The state diff is squashed and committed to the Patricia Merkle Tree via `state_update`. [5](#0-4) 

7. All future calls to `C` fail at class dispatch — no class with hash `0` exists. All deposited funds are permanently frozen with no recovery path.
