### Title
Missing `shard_id` Binding Validation in `set_state_header` Allows Cross-Shard State Substitution During State Sync - (`File: chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header()` accepts a caller-supplied `shard_id` and a peer-supplied `ShardStateSyncResponseHeader`, but never checks that the chunk embedded inside the header actually belongs to the requested shard. A malicious peer can substitute a valid chunk from a different shard (e.g., shard 1 when shard 0 was requested), pass all five validation steps, and cause the syncing node to store and later apply the wrong shard's trie data under the target shard's storage key — permanently corrupting that shard's local state.

---

### Finding Description

`set_state_header` in `chain/chain/src/state_sync/adapter.rs` performs five validation steps on the incoming `ShardStateSyncResponseHeader`:

1. `validate_chunk_proofs` — internal chunk consistency
2. Merkle path: chunk hash is somewhere in `sync_prev_block_header.chunk_headers_root()`
3. Merkle path: prev-chunk hash is in the preceding block's root
4. Incoming-receipt proofs are structurally valid
5. `state_root_node` hashes to `chunk.prev_state_root()`

None of these steps check that `chunk.shard_id() == shard_id`. [1](#0-0) 

The Merkle path check at step 2 proves only that the chunk hash appears *somewhere* in the block's `chunk_headers_root` Merkle tree — it does not verify that the leaf position corresponds to the requested shard's index. A chunk from shard 1 carries a valid Merkle proof for its own position (index 1), and that proof passes `verify_path` against the same root even when `shard_id = 0` was requested. [2](#0-1) 

The receipt-proof loop at step 4 computes `receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))` using the *caller-supplied* `shard_id`, not the chunk's actual shard. When incoming receipts are empty (the common case for any shard with no cross-shard traffic in the sync window), this hash is identical regardless of which shard the chunk belongs to, so the Merkle path check at step 4e passes trivially. [3](#0-2) 

The `to_shard_id` field of each `ShardProof` is explicitly discarded (`to_shard_id: _`) and never validated against the requested shard. [4](#0-3) 

After all checks pass, the header is stored under `StateHeaderKey(shard_id, sync_hash)` — keyed by the *caller-supplied* shard, not the chunk's actual shard. [5](#0-4) 

Downstream, `set_state_finalize` in `chain_update.rs` derives `shard_uid` from the caller-supplied `shard_id` (e.g., shard 0) and calls `apply_chunk` with that `shard_uid`, but uses the chunk body (transactions, receipts, `prev_state_root`) from the substituted chunk (shard 1). The trie writes land in shard 0's storage with shard 1's data. [6](#0-5) 

Similarly, `apply_state_part` in `chain/chain/src/runtime/mod.rs` derives `shard_uid` from the caller-supplied `shard_id` and writes the trie nodes from the substituted shard's state parts into that shard's storage. [7](#0-6) 

---

### Impact Explanation

A syncing node that accepts a substituted header will:

1. Store shard 1's `ShardStateSyncResponseHeader` under `StateHeaderKey(shard_id=0, sync_hash)`.
2. Validate and apply shard 1's state parts against shard 1's `prev_state_root`, writing them into shard 0's trie storage.
3. Complete `set_state_finalize` believing shard 0 is fully synced, but with shard 1's account/contract data.
4. Produce chunks for shard 0 whose state root does not match the canonical chain — those chunks are rejected by stateless validators, the node cannot advance, and if it is a chunk producer it loses rewards or risks slashing.

The corruption is permanent until the node re-syncs from scratch.

---

### Likelihood Explanation

State sync is initiated by any node that falls behind or needs to catch up a new shard (e.g., after shard shuffling). The `StateResponseInfo` is received over the P2P network from any peer — no privileged role is required to respond to a `StateRequestHeader` message. The attack requires only that the target shard has no incoming cross-shard receipts in the sync window, which is the common case for lightly-loaded shards. The attacker needs to be a reachable peer selected as a snapshot host for the target shard. [8](#0-7) 

---

### Recommendation

Add an explicit shard-binding check immediately after extracting the chunk from the header in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: bind the embedded chunk's shard to the requested shard
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {} does not match requested shard_id {}",
        chunk.shard_id(), shard_id
    )));
}
```

Additionally, restore the `to_shard_id` check in the receipt-proof loop to verify that each proof is destined for the requested shard, rather than discarding it with `to_shard_id: _`.

---

### Proof of Concept

**Setup**: A 2-shard network. The syncing node needs to catch up shard 0. A malicious peer is selected as a snapshot host.

**Attack**:
1. Syncing node sends `StateRequestHeader { shard_id: ShardId(0), sync_hash }`.
2. Malicious peer responds with a `VersionedStateResponse` where `StateResponseInfo.shard_id = 0` but the embedded `ShardStateSyncResponseHeader` contains shard 1's chunk (which has a valid Merkle proof against the block's `chunk_headers_root`, valid `validate_chunk_proofs`, and a valid `state_root_node` for shard 1's state root). Incoming receipts are empty.
3. `set_state_header(shard_id=0, sync_hash, header_for_shard_1)` passes all five checks and writes `StateHeaderKey(0, sync_hash) → header_for_shard_1` to `DBCol::StateHeaders`.
4. `set_state_part` retrieves this header, extracts shard 1's `prev_state_root`, and validates shard 1's state parts against it — all pass.
5. `apply_state_part` calls `get_shard_uid_from_epoch_id(shard_id=0, epoch_id)` → `shard_uid_0`, then writes shard 1's trie nodes into shard 0's storage.
6. `set_state_finalize` applies shard 1's chunk body under `shard_uid_0`.
7. Shard 0's local state is now shard 1's data. The node produces chunks for shard 0 with the wrong state root, which are rejected by the network. [9](#0-8) [10](#0-9)

```rust
async fn apply_state_part(
    store: Store,
    runtime: Arc<dyn RuntimeAdapter>,
    computation_task_tracker: TaskTracker,
    cancel: CancellationToken,
    sync_hash: CryptoHash,
    shard_id: ShardId,
    part_id: u64,
    num_parts: u64,
    state_root: CryptoHash,
    epoch_id: EpochId,
) -> Result<StatePartApplyResult, near_chain::Error> {
    let key = StatePartKey(sync_hash, shard_id, part_id);
    let key_bytes = borsh::to_vec(&key).unwrap();
    let already_applied = store.exists(DBCol::StatePartsApplied, &key_bytes);
    if already_applied {
        tracing::debug!(target: "sync", ?key, "state part already applied, skipping");
        return Ok(StatePartApplyResult::AlreadyApplied);
    }
    return_if_cancelled!(cancel);
    let handle =
        computation_task_tracker.get_handle(&format!("shard {} part {}", shard_id, part_id)).await;
    return_if_cancelled!(cancel);
    handle.set_status("Loading part data from store");
    let bytes = store
        .get(DBCol::StateParts, &key_bytes)
        .ok_or_else(|| {
            near_chain::Error::DBNotFoundErr(format!(
                "No state part {} for shard {}",
                part_id, shard_id
            ))
        })?
        .to_vec();
    let state_part = StatePart::from_bytes(bytes)?;
    handle.set_status("Applying part data to runtime");
    runtime.apply_state_part(
        shard_id,
        &state_root,
        PartId { idx: part_id, total: num_parts },
        &state_part,
        &epoch_id,
    )?;
    tracing::debug!(target: "sync", ?key, "applied state part");

    // Mark part as applied.
    let mut store_update = store.store_update();
    store_update.set_ser(DBCol::StatePartsApplied, &key_bytes, &true);
    store_update.commit();

    Ok(StatePartApplyResult::Applied)
```
