### Title
Unchecked Detached Burn Promise in `init_transfer_internal` Enables Unbacked Minting on Destination Chain - (File: `near/omni-bridge/src/lib.rs`)

### Summary
`burn_tokens_if_needed` fires the token burn with `.detach()`, meaning its result is never observed. When called from `init_transfer_internal`, the transfer message is recorded and `InitTransferEvent` is emitted regardless of whether the burn succeeded. If the burn fails silently, the source tokens remain in the bridge while the destination chain mints an equal amount, creating unbacked supply.

### Finding Description
`burn_tokens_if_needed` is defined as:

```rust
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) {
    if self.is_deployed_token(&token) {
        ext_token::ext(token)
            .with_static_gas(BURN_TOKEN_GAS)
            .burn(amount)
            .detach();   // ← result never observed
    }
}
``` [1](#0-0) 

`BURN_TOKEN_GAS` is only 3 TGas: [2](#0-1) 

`init_transfer_internal` calls this function and then unconditionally records the transfer and emits the event:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);  // detached, result ignored

self.lock_tokens_if_needed(...);

env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
U128(0)   // ← tells ft_transfer_call to keep the tokens in the bridge
``` [3](#0-2) 

The same pattern appears in `fast_fin_transfer_to_other_chain` (line 932) and `resolve_fast_transfer` (line 904): [4](#0-3) [5](#0-4) 

The project's own security checklist in `near/CLAUDE.md` explicitly flags this pattern: *"Check .detach() usage: Detached promises should only be used for non-critical operations"* — yet the burn of bridged tokens is a critical operation. [6](#0-5) 

### Impact Explanation
If the burn promise fails (gas exhaustion with only 3 TGas, or any revert in the token contract):

1. The user's tokens remain in the bridge contract (the `ft_transfer_call` already transferred them; `ft_on_transfer` returns `U128(0)` meaning "keep all tokens").
2. The `InitTransferEvent` is emitted with a valid transfer ID and nonce.
3. Relayers observe the event and submit proof to the destination chain.
4. The destination chain mints the full amount to the recipient.

Result: the destination chain holds newly minted tokens with no corresponding burned supply on NEAR — **unbacked minting / unauthorized creation of bridged assets**. This matches the Critical impact class: *"Direct theft, unauthorized release, or unauthorized mint of native or bridged assets."*

### Likelihood Explanation
- Any unprivileged user can trigger this path by calling `ft_transfer_call` on a DAO-registered deployed token pointing to the bridge.
- 3 TGas is a very tight budget for a cross-contract burn. A token contract that performs storage reads/writes or emits events (standard NEP-141 behavior) can easily exceed this budget, causing a silent failure.
- The failure is completely silent — no revert, no log, no on-chain indication that the burn did not execute.
- The `InitTransferEvent` is emitted in the same execution frame as the detached promise, so relayers cannot distinguish a successful burn from a failed one.

### Recommendation
Replace the fire-and-forget pattern with a chained callback that checks the burn result before recording the transfer and emitting the event:

```rust
ext_token::ext(token)
    .with_static_gas(BURN_TOKEN_GAS)
    .burn(amount)
    .then(
        Self::ext(env::current_account_id())
            .burn_callback(transfer_message, storage_owner)
    )
```

Only emit `InitTransferEvent` and return `U128(0)` (keep tokens) inside `burn_callback` if the burn succeeded. If the burn failed, return the full amount to refund the user. Also increase `BURN_TOKEN_GAS` from 3 TGas to at least 10 TGas to accommodate standard NEP-141 token implementations.

### Proof of Concept
1. DAO registers a deployed token `bridge-token.near` with a burn function that costs slightly more than 3 TGas (e.g., due to storage operations or event emission).
2. Alice calls `ft_transfer_call` on `bridge-token.near` with `receiver_id = bridge.near`, `amount = 1_000_000`, and an `InitTransferMsg` targeting an EVM recipient.
3. Bridge's `ft_on_transfer` → `init_transfer` → `init_transfer_internal`:
   - Transfer message is stored in `pending_transfers`.
   - `burn_tokens_if_needed` fires a detached burn with 3 TGas — the burn runs out of gas and fails silently.
   - `InitTransferEvent` is emitted with `origin_nonce = N`.
   - Returns `U128(0)` — bridge keeps Alice's 1,000,000 tokens.
4. A relayer observes `InitTransferEvent` and calls `fin_transfer` on the EVM `OmniBridge`, which mints 1,000,000 tokens to Alice's EVM address.
5. Alice now holds 1,000,000 tokens on EVM. The bridge also holds 1,000,000 tokens on NEAR that were never burned — total supply is inflated by 1,000,000.
