### Title
Truncated-Hash Address Collision in `deriveDeterministicAddress` Enables Unauthorized ERC1155 Token Theft - (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.deriveDeterministicAddress` maps ERC1155 `(tokenAddress, tokenId)` pairs to a virtual 20-byte address by truncating a keccak256 hash to 160 bits. Because the output space is only 2^160 and the attacker controls both input sets in a birthday attack, a collision between two different `(tokenAddress, tokenId)` pairs can be found with ~2^80 work. Exploiting such a collision allows an attacker to deposit worthless ERC1155 tokens and drain valuable ERC1155 tokens locked in the bridge.

---

### Finding Description

`deriveDeterministicAddress` computes:

```solidity
address(bytes20(keccak256(abi.encodePacked(tokenAddress, tokenId))))
``` [1](#0-0) 

The `bytes20()` cast discards the upper 96 bits of the 256-bit keccak256 digest, reducing collision resistance to 2^160. By the birthday bound, two independently generated sets of 2^80 hashes each have an ~86% probability of containing a matching pair.

The attacker controls both sides of the birthday search:

- **Set A**: `deriveDeterministicAddress(evilERC1155_i, evilTokenId_i)` — the attacker deploys their own ERC1155 contract and varies `tokenId` freely across the full uint256 range.
- **Set B**: `deriveDeterministicAddress(legitimateERC1155_j, legitimateTokenId_j)` — the attacker varies `tokenId` against any valuable ERC1155 token already bridged (or expected to be bridged).

Once a collision `D` is found (i.e., both inputs hash to the same 20-byte address), the attack proceeds:

**Step 1 — Register the legitimate mapping:**
`logMetadata1155` is permissionless. The attacker calls `logMetadata1155(legitimateERC1155_B, legitimateTokenId_B)`, which sets `multiTokens[D] = {legitimateERC1155_B, legitimateTokenId_B}` and emits `LogMetadata(D, ...)`. [2](#0-1) 

**Step 2 — NEAR deploys a bridge token for address D:**
The NEAR bridge processes the `LogMetadata` event and deploys a NEAR-side token for the virtual address `D`.

**Step 3 — Deposit worthless ERC1155 tokens:**
The attacker calls `initTransfer1155(evilERC1155_A, evilTokenId_A, amount, nearRecipient, ...)`. The function computes `deterministicToken = D` and emits `InitTransfer(msg.sender, D, amount, ...)`. Crucially, `initTransfer1155` performs **no check** on whether `multiTokens[D]` is already set. [3](#0-2) 

**Step 4 — NEAR mints bridge tokens:**
NEAR sees an `InitTransfer` for token `D` and mints the corresponding NEAR-side bridge tokens to `nearRecipient`.

**Step 5 — Bridge back and drain:**
`nearRecipient` initiates a transfer back to EVM. NEAR signs a `finTransfer` payload with `tokenAddress = D`. On EVM, `finTransfer` checks `multiTokens[D]` first:

```solidity
} else if (multiToken.tokenAddress != address(0)) {
    IERC1155(multiToken.tokenAddress).safeTransferFrom(
        address(this),
        payload.recipient,
        multiToken.tokenId,
        payload.amount,
        ""
    );
``` [4](#0-3) 

Since `multiTokens[D]` points to `{legitimateERC1155_B, legitimateTokenId_B}`, the bridge releases the legitimate ERC1155 tokens to the attacker — despite the attacker having deposited only worthless ERC1155 tokens.

---

### Impact Explanation

**Critical — Direct theft of bridged ERC1155 assets.**

The attacker deposits tokens from a self-deployed, worthless ERC1155 contract and withdraws valuable ERC1155 tokens locked in the bridge. The bridge's collateralization is broken: the `multiTokens` mapping is poisoned to point to a legitimate token, but the locked balance for that token was never incremented by the attacker's deposit. All ERC1155 tokens of type `(legitimateERC1155_B, legitimateTokenId_B)` held by the bridge can be drained.

---

### Likelihood Explanation

The birthday attack requires ~2^80 keccak256 evaluations. As established in the KyberSwap M-2 discussion, the Bitcoin network's current hashrate (~4.7×10^20 H/s) can perform 2^80 keccak256-equivalent hashes in roughly 2–3 hours. Even at 1% of that compute, the attack completes in days. The attacker controls both input sets (their own ERC1155 contract with arbitrary token IDs, and any legitimate ERC1155 with arbitrary token IDs), making this a true birthday search rather than a preimage search. The `OmniBridge` contract is upgradeable, so the admin can patch it — but any ERC1155 tokens locked before the patch are at risk.

---

### Recommendation

Replace the truncated-hash scheme with a collision-resistant mapping. Two concrete options:

1. **Use a full 256-bit key**: Store `multiTokens` keyed by `bytes32(keccak256(abi.encodePacked(tokenAddress, tokenId)))` instead of a truncated 20-byte address. Pass the full `bytes32` key through the bridge protocol rather than casting to `address`.

2. **Use a counter-based deterministic ID**: Assign a monotonically increasing `uint256` ID to each new `(tokenAddress, tokenId)` pair on first registration, and use that ID as the canonical identifier. This eliminates hash truncation entirely.

Either approach removes the birthday-attack surface by ensuring the identifier space is not truncated to 160 bits.

### Proof of Concept

```python
# Pseudocode birthday search
import hashlib, random, struct

def derive(token_addr_bytes, token_id_int):
    data = token_addr_bytes + token_id_int.to_bytes(32, 'big')
    h = hashlib.sha3_256(data).digest()  # approximate keccak256
    return h[:20]  # bytes20 truncation

table = {}
# Generate 2^40 entries for Set A (attacker's evil ERC1155)
evil_addr = b'\xde\xad' + b'\x00' * 18
for i in range(2**40):
    d = derive(evil_addr, i)
    table[d] = ('evil', i)

# Generate 2^40 entries for Set B (legitimate ERC1155)
legit_addr = b'\xbe\xef' + b'\x00' * 18
for j in range(2**40):
    d = derive(legit_addr, j)
    if d in table and table[d][0] == 'evil':
        print(f"Collision at D={d.hex()}: evil tokenId={table[d][1]}, legit tokenId={j}")
        break
# Expected: collision found after ~2^40 iterations (birthday bound for 2*2^40 = 2^41 total hashes)
```

The collision probability for 2^81 total hashes against a 2^160 space is ~86.4% (matching the KyberSwap calculator). Once found, the five-step attack above drains all locked ERC1155 tokens of the colliding type.
