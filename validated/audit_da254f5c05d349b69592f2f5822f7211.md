### Title
Deprecated (Cairo 0) Class Declarations Silently Dropped from `CommitmentStateDiff`, Producing Wrong `state_diff_commitment`, `state_diff_length`, and Block Hash — (`File: crates/blockifier/src/state/cached_state.rs`)

---

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` conversion unconditionally hardcodes `deprecated_declared_classes: Vec::new()`. Because `CommitmentStateDiff` itself has no field for deprecated (Cairo 0) class declarations, and because `From<StateMaps> for CommitmentStateDiff` silently drops `StateMaps::declared_contracts`, any DeclareV1 transaction that is accepted and executed by the batcher produces a `ThinStateDiff` that is missing the declared class hash. The downstream `calculate_state_diff_hash`, `ThinStateDiff::len`, and `concatenated_counts` all operate on this truncated diff, yielding a wrong `state_diff_commitment`, a wrong `state_diff_length` in the block header, and a wrong block hash for every block that contains a deprecated Declare transaction.

---

### Finding Description

**Step 1 – `StateMaps` tracks deprecated declarations but `CommitmentStateDiff` drops them.**

`StateMaps` has a `declared_contracts: HashMap<ClassHash, bool>` field that is set for every declared class, including Cairo 0 (deprecated) ones. [1](#0-0) 

When `CommitmentStateDiff` is built from `StateMaps`, `declared_contracts` is entirely absent from the conversion: [2](#0-1) 

Cairo 1 classes survive because they also appear in `compiled_class_hashes`. Cairo 0 classes exist **only** in `declared_contracts` and are therefore silently lost.

**Step 2 – `From<CommitmentStateDiff> for ThinStateDiff` hardcodes an empty list.** [3](#0-2) 

The `TODO(AlonH)` comment acknowledges this is a placeholder, but no guard prevents the code from being reached in production.

**Step 3 – `BlockExecutionArtifacts::new` feeds this truncated diff into commitment calculation.** [4](#0-3) 

`ThinStateDiff::from(commitment_state_diff.clone())` produces the diff with `deprecated_declared_classes = []`.

**Step 4 – `calculate_block_commitments` computes `state_diff_commitment` and `concatenated_counts` from the truncated diff.**

`calculate_state_diff_hash` chains deprecated declared classes into the Poseidon hash: [5](#0-4) 

`ThinStateDiff::len()` counts them toward `state_diff_length`: [6](#0-5) 

`concatenated_counts` encodes `state_diff_length` into the block hash: [7](#0-6) 

All three values are wrong when any deprecated class was declared in the block.

**Step 5 – The RPC execution path shows the correct pattern is known but not applied to the batcher.**

`induced_state_diff` in the RPC path explicitly accepts a `deprecated_declared_class_hash` parameter and injects it manually, proving the developers know `CommitmentStateDiff` cannot carry this information: [8](#0-7) 

The batcher path has no equivalent injection.

---

### Impact Explanation

For every block that contains at least one accepted DeclareV1 (deprecated Cairo 0) transaction:

1. **Wrong `state_diff_commitment`** — the Poseidon hash omits the deprecated class hash, so the value stored in the block header diverges from what the Starknet OS independently computes.
2. **Wrong `state_diff_length`** — the count stored in the block header is too small by the number of deprecated declared classes.
3. **Wrong `concatenated_counts` → wrong block hash** — `concatenated_counts` encodes `state_diff_length`; the resulting `PartialBlockHash` and final block hash are incorrect.
4. **SNOS/proof mismatch** — the OS computes the correct state diff commitment (including deprecated classes); the sequencer committed to a different value, so proof verification fails for those blocks.
5. **P2P sync breakage** — `StateDiffStreamBuilder` uses `state_diff_length` from the header as the termination condition for consuming state diff chunks; a too-small value causes it to stop early, accepting an incomplete state diff. [9](#0-8) 

---

### Likelihood Explanation

Any unprivileged user can submit a DeclareV1 transaction through the public RPC endpoint. The gateway accepts it, the batcher includes it, and the commitment path executes without any check that deprecated declared classes are present. No special privilege or race condition is required. The trigger is a single valid transaction type that has been part of the Starknet protocol since its inception.

---

### Recommendation

1. Add a `deprecated_declared_classes: Vec<ClassHash>` field to `CommitmentStateDiff` and populate it from `StateMaps::declared_contracts` (filtering for entries where the class has no compiled class hash, i.e., Cairo 0 classes).
2. Remove the hardcoded `Vec::new()` in `From<CommitmentStateDiff> for ThinStateDiff` and propagate the field.
3. In `BlockExecutionArtifacts::new`, ensure the `ThinStateDiff` passed to `calculate_block_commitments` includes deprecated declared classes, mirroring the explicit injection already done in `induced_state_diff`.
4. Add a regression test that builds a block containing a DeclareV1 transaction and asserts that the resulting `ThinStateDiff::deprecated_declared_classes` is non-empty and that the `state_diff_commitment` matches an independently computed reference value.

---

### Proof of Concept

1. Submit a DeclareV1 transaction (Cairo 0 class) to the sequencer gateway.
2. The batcher executes it; `StateMaps::declared_contracts` is set for the class hash, but `CommitmentStateDiff` has no corresponding field.
3. `ThinStateDiff::from(commitment_state_diff)` produces `deprecated_declared_classes: []`.
4. `calculate_state_diff_hash(&thin_state_diff)` omits the class hash from the Poseidon chain.
5. `ThinStateDiff::len()` returns a value that is 1 less than the correct length.
6. `concatenated_counts` encodes the wrong `state_diff_length`.
7. The `PartialBlockHash` and final block hash are computed from the wrong `concatenated_counts`.
8. The Starknet OS, given the same block, computes the correct state diff commitment (including the deprecated class hash) and produces a different block hash, causing proof verification to fail. [3](#0-2) [10](#0-9) [5](#0-4)
