### Title
Gateway Rejects Valid Invoke V3 Proof-Facts Transactions Due to Off-by-One Between Incremented Block Number and Actual State Availability — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`, `crates/blockifier/src/transaction/account_transaction.rs`)

---

### Summary

The gateway's `run_validate_entry_point` increments the block number by one (`block_info.block_number.unchecked_next()`) to simulate the next block, but uses the **current committed state** (not the state after `pre_process_block` runs for that next block). This creates an off-by-one discrepancy: `validate_proof_block_number` permits proof facts referencing block `current − STORED_BLOCK_HASH_BUFFER + 1`, but `validate_proof_block_hash` reads from the current state which only contains block hashes up to `current − STORED_BLOCK_HASH_BUFFER`. The hash for block `current − STORED_BLOCK_HASH_BUFFER + 1` is written by `pre_process_block` only when block `current + 1` is actually executed. As a result, the gateway rejects valid Invoke V3 transactions that would be accepted by the blockifier at execution time.

---

### Finding Description

**Step 1 — Gateway increments block number but keeps current state.**

In `run_validate_entry_point`:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();   // current + 1
```

The `CachedState` passed to `StatefulValidator` is the current committed state (after block `current`). [1](#0-0) 

**Step 2 — `validate_proof_block_number` uses the incremented block number.**

```rust
let max_allowed =
    current_block_number.0.checked_sub(STORED_BLOCK_HASH_BUFFER)...;
// With current_block_number = current+1:
// max_allowed = (current+1) - 10 = current - 9
```

A proof referencing block `current − 9` passes this check. [2](#0-1) 

**Step 3 — `validate_proof_block_hash` reads from the current state.**

```rust
let stored_block_hash = state
    .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;
if stored_block_hash != proof_block_hash {
    return Err(TransactionPreValidationError::InvalidProofFacts(...));
}
```

The current state has block hashes only for blocks `0 … current − 10`. The hash for block `current − 9` is **zero** (not yet written). Any non-zero proof hash causes a "Block hash mismatch" error. [3](#0-2) 

**Step 4 — `pre_process_block` writes the missing hash only at execution time.**

```rust
state.set_storage_at(block_hash_contract_address,
    StorageKey::from(number.0), hash.0)?;
```

`pre_process_block` writes the hash for block `(next_block_number − STORED_BLOCK_HASH_BUFFER)` = `current − 9` when block `current + 1` is processed. The gateway never calls this before validation. [4](#0-3) 

**Step 5 — The OS Cairo code enforces the same constraint, confirming the intended semantics.**

```cairo
assert_nn_le(
    os_output_header.base_block_number,
    current_block_number - STORED_BLOCK_HASH_BUFFER
);
```

The OS uses `current_block_number` (not `current + 1`), so the OS would accept proof_block_number ≤ `current − 10`, not `current − 9`. The gateway's incremented block number is inconsistent with both the OS constraint and the available state. [5](#0-4) 

---

### Impact Explanation

A user submits a valid Invoke V3 transaction whose `proof_facts` reference block `current − STORED_BLOCK_HASH_BUFFER + 1` (i.e., `current − 9`). The proof is cryptographically correct and the block hash is publicly known. The gateway rejects the transaction with `InvalidProofFacts("Block hash mismatch … stored block hash: 0")`. The same transaction, submitted after block `current + 1` is finalized, is accepted without modification. This is a **gateway admission rejects valid transactions before sequencing** impact.

---

### Likelihood Explanation

Any user of the client-side proving feature who constructs proof facts against the most recently allowed block (a natural choice to minimize proof staleness) will hit this rejection. The off-by-one is deterministic and reproducible on every block boundary.

---

### Recommendation

In `run_validate_entry_point`, either:

1. **Do not increment the block number** for proof-facts validation (use `current` instead of `current + 1`), aligning with the OS constraint `proof_block_number ≤ current − STORED_BLOCK_HASH_BUFFER`; or
2. **Simulate `pre_process_block`** before constructing the `CachedState` used for proof-facts validation, so the hash for block `current − STORED_BLOCK_HASH_BUFFER + 1` is present in the state when the incremented block number is used.

---

### Proof of Concept

Let `STORED_BLOCK_HASH_BUFFER = 10` and `current = 100`.

1. The gateway sets `block_number = 101`.
2. `validate_proof_block_number`: `max_allowed = 101 − 10 = 91`. A proof referencing block `91` passes.
3. `validate_proof_block_hash`: reads `state.get_storage_at(BLOCK_HASH_CONTRACT, key=91)` → returns `0` (written only when block 101 is executed).
4. Check `0 != proof_hash` → `InvalidProofFacts("Block hash mismatch … stored block hash: 0")` → transaction rejected.
5. After block 101 is finalized, `pre_process_block` writes `hash(block 91)` at key `91`.
6. Resubmit the identical transaction → accepted.

The discrepancy is rooted in `run_validate_entry_point` using `block_number = current + 1` while the state reflects only `current`, mirroring the external report's pattern where the "pending" entity (the incremented block context) is denied access to a resource (the stored block hash) that only the "current" entity (the actual executed block) can provide.

    fn validate_proof_block_hash(
        proof_block_hash: Felt,
        proof_block_number: u64,
        os_constants: &OsConstants,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        if proof_block_hash == Felt::ZERO {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof block hash is zero for block {proof_block_number}."
            )));
        }

        // Compare the proof's block hash with the stored block hash.
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();

        let stored_block_hash = state
            .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

        if stored_block_hash != proof_block_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Block hash mismatch for block {proof_block_number}. Proof block hash: \
                 {proof_block_hash}, stored block hash: {stored_block_hash}."
            )));
        }

        Ok(())
    }

pub fn pre_process_block(
    state: &mut dyn State,
    old_block_number_and_hash: Option<BlockHashAndNumber>,
    next_block_number: BlockNumber,
    os_constants: &OsConstants,
) -> StateResult<()> {
    let should_block_hash_be_provided =
        next_block_number >= BlockNumber(constants::STORED_BLOCK_HASH_BUFFER);
    if let Some(BlockHashAndNumber { number, hash }) = old_block_number_and_hash {
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();

        state.set_storage_at(block_hash_contract_address, block_number_as_storage_key, hash.0)?;
    } else if should_block_hash_be_provided {
        return Err(StateError::OldBlockHashNotProvided);
    }

    Ok(())
