Audit Report

## Title
Unverified Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain Approved User Tokens — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

`MetricOmmPoolLiquidityAdder` accepts a caller-supplied `pool` address and stores it verbatim in transient context without validating it against any factory allowlist. The `metricOmmModifyLiquidityCallback` authenticates the caller only against this unverified address, then unconditionally trusts `msg.sender` (the attacker-controlled pool) for both token addresses and payment amounts. A malicious pool can therefore drain up to `maxAmountToken0` / `maxAmountToken1` of any ERC-20 tokens from a victim who has approved this contract.

## Finding Description

The NatSpec at lines 19–21 explicitly acknowledges the missing guard: [1](#0-0) 

All four public entry points (`addLiquidityExactShares` ×2, `addLiquidityWeighted` ×2) funnel into `_addLiquidity`, which stores the unverified `pool` address via `_setPayContext` without any factory check: [2](#0-1) [3](#0-2) 

In `metricOmmModifyLiquidityCallback`, the sole authenticity check compares `msg.sender` against `expectedPool` — which is whatever address the user supplied: [4](#0-3) 

After passing this check, the callback calls `getImmutables()` on `msg.sender` (the attacker-controlled contract) to obtain token addresses, then calls `pay()` sending those tokens from the victim to `msg.sender`: [5](#0-4) 

`pay()` resolves to `safeTransferFrom(payer, recipient, value)` where `payer` is the victim and `recipient` is the malicious pool: [6](#0-5) 

The exploit flow is:
1. `_addLiquidity` stores `maliciousPool` as `expectedPool` in transient storage.
2. `_addLiquidity` calls `maliciousPool.addLiquidity(...)`.
3. `maliciousPool.addLiquidity` immediately calls back `metricOmmModifyLiquidityCallback(MAX0, MAX1, abi.encode(KIND_PAY))`.
4. The `msg.sender == expectedPool` check passes (both are `maliciousPool`).
5. `maliciousPool.getImmutables()` returns attacker-chosen `token0`/`token1`.
6. `pay(token0, victim, maliciousPool, MAX0)` and `pay(token1, victim, maliciousPool, MAX1)` execute, draining the victim.

## Impact Explanation

Direct loss of user principal. Any victim who has approved `MetricOmmPoolLiquidityAdder` for any ERC-20 token (the normal prerequisite for LP operations) and is induced to call any of the four public entry points with a malicious pool address loses up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 in a single transaction. The attacker receives the tokens directly with no recovery path. This meets the Critical/High direct loss of user principal threshold.

## Likelihood Explanation

The attack requires no privileged role. The attacker only needs to deploy a trivial contract implementing `addLiquidity` and `getImmutables`. The victim must have approved the adder (standard LP prerequisite) and be induced to call an entry point with the malicious address — achievable via a spoofed front-end, phishing link, or malicious aggregator integration. The attack is repeatable and atomic within a single transaction.

## Recommendation

Validate the supplied `pool` address against the `MetricOmmPoolFactory` before storing it in the transient pay context. In `_addLiquidity` (or at each public entry point):

```solidity
require(IMetricOmmPoolFactory(factory).isPool(pool), "Unknown pool");
```

The `factory` address should be stored as an immutable set in the constructor, mirroring the pattern used in the router. This closes the gap without restricting permissionless LP flows for legitimate pools.

## Proof of Concept

```solidity
contract MaliciousPool {
    address immutable adder;
    address immutable token0; // e.g. USDC
    address immutable token1; // e.g. WETH
    uint256 immutable MAX0;
    uint256 immutable MAX1;

    constructor(address _adder, address _t0, address _t1, uint256 _max0, uint256 _max1) {
        adder = _adder; token0 = _t0; token1 = _t1; MAX0 = _max0; MAX1 = _max1;
    }

    function addLiquidity(address, uint80, LiquidityDelta calldata, bytes calldata, bytes calldata)
        external returns (uint256, uint256)
    {
        IMetricOmmPoolLiquidityAdder(adder).metricOmmModifyLiquidityCallback(
            MAX0, MAX1, abi.encode(uint8(1)) // KIND_PAY
        );
        return (MAX0, MAX1);
    }

    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = token0;
        imm.token1 = token1;
    }
}

// Test steps:
// 1. victim.approve(adder, USDC, MAX0); victim.approve(adder, WETH, MAX1);
// 2. attacker deploys MaliciousPool(adder, USDC, WETH, MAX0, MAX1)
// 3. victim (tricked) calls adder.addLiquidityExactShares(maliciousPool, victim, 0, deltas, MAX0, MAX1, "")
// 4. MaliciousPool.addLiquidity fires → callback → pay(USDC, victim, maliciousPool, MAX0)
//                                                 → pay(WETH, victim, maliciousPool, MAX1)
// 5. Attacker receives MAX0 USDC + MAX1 WETH from victim's wallet.
```
