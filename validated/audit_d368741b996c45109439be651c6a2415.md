### Title
Non-Atomic Snapshot of `pending_data` and `pending_classes` in RPC Pending Execution Causes Wrong Fee Estimation and Simulation Results - (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

The RPC handlers for `starknet_estimateFee`, `starknet_simulateTransactions`, and `starknet_call` read `pending_data` and `pending_classes` through two **separate, non-atomic** lock acquisitions. The central-sync loop updates these two shared objects in a non-atomic sequence (clear `pending_classes` → write `pending_data`). A concurrent RPC request can observe a torn state where `pending_data.state_update.state_diff.declared_classes` references class hashes that are absent from the simultaneously-read `pending_classes`. The resulting `ExecutionPendingData` is internally inconsistent, causing execution to fail with `UndeclaredClassHash` or to silently fall through to stale storage state, producing an authoritative-looking wrong fee estimate, wrong simulation trace, or wrong call result.

---

### Finding Description

**Root cause — non-atomic read in RPC handlers**

In `estimate_fee` (and identically in `simulate_transactions` and `call`):

```rust
// crates/apollo_rpc/src/v0_8/api/api_impl.rs  lines 1009-1013
let maybe_pending_data = if let BlockId::Tag(Tag::Pending) = block_id {
    Some(client_pending_data_to_execution_pending_data(
        read_pending_data(&self.pending_data, &storage_txn).await?,  // acquires pending_data lock, releases it
        self.pending_classes.read().await.clone(),                    // acquires pending_classes lock separately
    ))
``` [1](#0-0) 

The two `.read().await` calls are on **different** `Arc<RwLock<_>>` objects. There is no combined lock or snapshot that covers both.

**Root cause — non-atomic write in the sync loop**

In `get_pending_data` inside `sync_pending_data`:

```rust
// crates/apollo_central_sync/src/pending_sync.rs  lines 179-182
if current_pending_parent_hash != new_pending_parent_hash {
    pending_classes.write().await.clear();   // step 1: classes wiped
}
*pending_data.write().await = new_pending_data;  // step 2: data updated
``` [2](#0-1) 

Between step 1 and step 2 there is a window where `pending_classes` is empty but `pending_data` still holds the old block's `declared_classes` list. There is a symmetric window after step 2 where `pending_data` references new class hashes that have not yet been inserted into `pending_classes` (those insertions happen asynchronously via `get_pending_class` / `get_pending_compiled_class` tasks).

**How the torn snapshot is assembled**

`client_pending_data_to_execution_pending_data` blindly merges the two independently-read snapshots:

```rust
// crates/apollo_rpc/src/pending.rs  lines 5-23
pub(crate) fn client_pending_data_to_execution_pending_data(
    client_pending_data: ClientPendingData,
    pending_classes: PendingClasses,
) -> ExecutionPendingData {
    ExecutionPendingData {
        declared_classes: client_pending_data.state_update.state_diff.declared_classes,
        // ...
        classes: pending_classes,   // may be empty or belong to a different block
        // ...
    }
}
``` [3](#0-2) 

**How execution consumes the torn state**

`ExecutionStateReader::get_compiled_class` first checks `pending_data.classes`; if the class hash is absent it falls through to storage:

```rust
// crates/apollo_rpc_execution/src/state_reader.rs  lines 89-113
if let Some(pending_classes) = self.maybe_pending_data.as_ref().map(|pd| &pd.classes) {
    if let Some(api_contract_class) = pending_classes.get_class(class_hash) {
        // use pending class — NOT reached when classes is empty/stale
    }
}
// falls through to storage or returns UndeclaredClassHash
``` [4](#0-3) 

**Concrete race scenario**

1. Pending block N declares class `C1` (not yet in finalized storage). `pending_data` holds `declared_classes = [C1]`; `pending_classes` holds `{C1: <bytecode>}`.
2. A new block is finalized. The sync loop enters `get_pending_data` for block N+1.
3. Sync: `pending_classes.write().clear()` — `pending_classes` is now `{}`.
4. **RPC request arrives**: reads `pending_data` (still block N, `declared_classes = [C1]`).
5. Sync: `*pending_data.write() = new_pending_data` (block N+1).
6. **RPC**: reads `pending_classes` — gets `{}` (cleared in step 3, new classes not yet downloaded).
7. `client_pending_data_to_execution_pending_data` produces `ExecutionPendingData { declared_classes: [C1], classes: {} }`.
8. Execution calls `get_compiled_class(C1)` → not in `pending_classes` → not in storage → `StateError::UndeclaredClassHash` → fee estimation returns an error or wrong result.

---

### Impact Explanation

`starknet_estimateFee`, `starknet_simulateTransactions`, and `starknet_call` with `block_id = "pending"` return an authoritative-looking wrong value (spurious execution error, wrong fee, wrong trace) for any transaction that invokes a class declared only in the pending block. This matches the **High** impact scope: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

---

### Likelihood Explanation

The race window is narrow but real: it opens every time a new block is finalized while a pending block with newly declared classes exists. On a busy network this occurs continuously. Any RPC client that polls `estimateFee` or `simulateTransactions` against the pending block (a common pattern for wallets and dApps) can hit this window. No special privileges are required; a normal unprivileged RPC caller triggers it.

---

### Recommendation

Read both `pending_data` and `pending_classes` under a single atomic snapshot. The simplest fix is to hold both read-guards simultaneously before cloning:

```rust
// Acquire both locks together, then clone
let pending_data_guard  = self.pending_data.read().await;
let pending_classes_guard = self.pending_classes.read().await;
let maybe_pending_data = if let BlockId::Tag(Tag::Pending) = block_id {
    Some(client_pending_data_to_execution_pending_data(
        (*pending_data_guard).clone(),
        (*pending_classes_guard).clone(),
    ))
} else {
    None
};
drop(pending_data_guard);
drop(pending_classes_guard);
```

Alternatively, wrap `pending_data` and `pending_classes` in a single `Arc<RwLock<(PendingData, PendingClasses)>>` so that the sync loop updates them atomically and readers always see a consistent pair.

---

### Proof of Concept

1. Deploy a contract that declares a new Sierra class in the pending block (class `C1` not yet finalized).
2. Concurrently, trigger a block finalization so the sync loop calls `pending_classes.write().clear()` followed by `*pending_data.write() = new_data`.
3. Between those two writes, issue `starknet_estimateFee` with `block_id = "pending"` for a transaction that invokes `C1`.
4. The RPC handler reads `pending_data` (still referencing `C1`) then reads `pending_classes` (now empty).
5. `client_pending_data_to_execution_pending_data` produces `ExecutionPendingData { declared_classes: [C1], classes: {} }`.
6. `exec_estimate_fee` → `ExecutionStateReader::get_compiled_class(C1)` → `pending_classes.get_class(C1)` returns `None` → falls through to storage → `C1` not in storage → `StateError::UndeclaredClassHash` → RPC returns a `TRANSACTION_EXECUTION_ERROR` instead of a valid fee estimate.
