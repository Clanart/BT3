### Title
Missing Metaplex Metadata Mint-Binding Check Allows Fake Token Registration with Stolen Name/Symbol on NEAR — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

### Summary

`parse_metadata_account` verifies that the supplied metadata account's key matches the address stored in the Token-2022 `MetadataPointer` extension, but never verifies that the Metaplex metadata account's embedded `mint` field equals the mint being logged. An attacker can create a Token-2022 mint whose `MetadataPointer.metadata_address` points to the Metaplex PDA of any high-value token (e.g., USDC), causing `log_metadata` to emit a Wormhole VAA that pairs the attacker's mint pubkey with USDC's name, symbol, and decimals. NEAR's `deploy_token_callback` then registers the attacker's mint as a bridged token carrying USDC's identity, after which the attacker can lock self-minted tokens on Solana and mint the fake "USDC" token on NEAR to any recipient.

### Finding Description

In `process()`, when a Token-2022 mint has a `MetadataPointer` extension whose `metadata_address` differs from the mint's own key, the code takes the "Third-party metadata" branch: [1](#0-0) 

This calls `parse_metadata_account` with the pointer's target address. That function checks only that the supplied account's key equals the pointer address: [2](#0-1) 

When the account is owned by Metaplex, it deserializes and returns `metadata.name` and `metadata.symbol` without any check that `metadata.mint == self.mint.key()`: [3](#0-2) 

The payload is then built using the attacker's mint pubkey as `token` but the victim token's name/symbol/decimals: [4](#0-3) 

On NEAR, `deploy_token_callback` trusts the VAA content entirely — it only checks that the emitter is the known Solana factory, then calls `deploy_token_internal` with whatever `token_address`, `name`, `symbol`, and `decimals` the VAA carries: [5](#0-4) 

### Impact Explanation

The attacker registers their own mint on NEAR under USDC's name and symbol. Because the `LogMetadata` constraint only requires that the bridge authority is **not** the mint authority: [6](#0-5) 

the attacker retains full mint authority over their Token-2022 mint. They can freely mint tokens into it, lock them in the bridge vault on Solana, and trigger a `finalize_transfer` VAA. NEAR's `fin_transfer_callback` looks up the token by the Solana mint address (the attacker's), finds the registered "USDC" entry, and mints the corresponding NEAR token to any recipient the attacker specifies — unbacked by any real USDC. This constitutes unauthorized minting of a bridged asset on NEAR.

### Likelihood Explanation

The attack requires no privileged access. Any account can create a Token-2022 mint, set its `MetadataPointer` to an arbitrary address, and call the public `log_metadata` instruction. The USDC Metaplex PDA is a publicly known, on-chain account. The only external dependency is a relayer willing to submit the VAA to NEAR, which is a permissionless role in the bridge.

### Recommendation

In `parse_metadata_account`, after deserializing the Metaplex `MplMetadata`, add an explicit check that the metadata's embedded mint field matches the mint being logged:

```rust
require_keys_eq!(
    metadata.mint,
    self.mint.key(),
    ErrorCode::InvalidTokenMetadataAddress,
);
```

This single guard closes the gap between the pointer-address check and the actual mint-binding invariant.

### Proof of Concept

1. Create a Token-2022 mint `attacker_mint` with `MetadataPointer.metadata_address = usdc_metaplex_pda` (the well-known Metaplex PDA for USDC's mint).
2. Call `log_metadata` passing `attacker_mint` as `mint` and `usdc_metaplex_pda` as `metadata`.
3. `process()` enters the "Third-party metadata" branch; `parse_metadata_account` passes the key check and returns `("USD Coin", "USDC")`.
4. The Wormhole VAA carries `token = attacker_mint.pubkey`, `name = "USD Coin"`, `symbol = "USDC"`, `decimals = attacker_mint.decimals`.
5. Submit the VAA to NEAR via `deploy_token`. `deploy_token_callback` registers `attacker_mint.pubkey` as a Solana-origin token with USDC's metadata.
6. Mint tokens freely into `attacker_mint` (attacker controls mint authority), lock them in the bridge vault, emit a transfer VAA.
7. Call `fin_transfer` on NEAR; the bridge mints the fake "USDC" token to any attacker-controlled recipient.
