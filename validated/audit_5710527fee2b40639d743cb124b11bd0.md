Audit Report

## Title
Unguarded `unwrapWETH9` and `sweepToken` Allow Any Caller to Drain Router's Entire WETH/Token Balance to Arbitrary Recipient — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`unwrapWETH9` and `sweepToken` are `public payable` with no caller restriction and accept a fully attacker-controlled `recipient` address. Any address can call either function at any time, draining the router's entire WETH or ERC-20 balance to an arbitrary address. The intended usage pattern — setting `recipient: address(router)` in `exactInput` and then calling `unwrapWETH9` — is explicitly tested, confirming that WETH legitimately lands on the router and can be stolen if the two steps are not bundled atomically.

## Finding Description
`unwrapWETH9` reads the router's full WETH balance, withdraws it, and forwards ETH to the caller-supplied `recipient` with zero access control:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol L37-45
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);
    }
}
```

The only guard is `balanceWETH < amountMinimum`, trivially bypassed by passing `amountMinimum = 0`. `sweepToken` has the identical structure with no access control. By contrast, `refundETH` hardcodes `msg.sender` as the recipient, demonstrating the correct pattern was known but not applied to the other two functions.

The `multicall` implementation uses `Address.functionDelegateCall` and is atomic — a revert in any step rolls back the entire call. However, atomicity is not enforced: nothing prevents a caller from issuing `exactInput(recipient=router)` and `unwrapWETH9` as two separate transactions. The test suite at `metric-periphery/test/MetricOmmSimpleRouter.native.t.sol` L219–235 explicitly confirms the pattern of routing output to `address(router)` then unwrapping, meaning WETH legitimately resides on the router between the two steps when they are not bundled.

Stranding paths:
1. **Non-atomic usage**: user or integrator calls `exactInput(recipient=router)` in one transaction and `unwrapWETH9` in a separate transaction.
2. **Direct WETH transfer**: any party sends WETH directly to the router address.
3. **Smart-contract wallet / batching integrator**: issues the swap and unwrap as separate calls across blocks.

In all cases, an attacker calls `router.unwrapWETH9(0, attacker)` and receives 100% of the stranded balance.

## Impact Explanation
Direct, complete loss of user principal. The victim's swap output (WETH) is redirected in full to the attacker as ETH. The victim's subsequent `unwrapWETH9` call reverts with `InsufficientWETH`. There is no recovery path. This meets the Critical/High threshold for direct loss of user principal under the allowed impact gate.

## Likelihood Explanation
The usage pattern is documented and tested. Any user, integrator, or smart-contract wallet that does not bundle both steps into a single `multicall` is silently exposed. Front-running is straightforward: an attacker monitors the mempool for `exactInput`/`exactInputSingle` calls with `recipient=address(router)` and immediately calls `unwrapWETH9(0, attacker)` in the next block or via a sandwich. No special privileges are required.

## Recommendation
Remove the `recipient` parameter from `unwrapWETH9` and `sweepToken` and always send to `msg.sender`, matching the behavior of `refundETH`:

```solidity
function unwrapWETH9(uint256 amountMinimum) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(msg.sender, balanceWETH);
    }
}
```

Alternatively, add `require(recipient == msg.sender, "recipient must be caller")` at the top of both functions.

## Proof of Concept
1. Victim calls `exactInput({..., recipient: address(router)})` — WETH output lands on the router.
2. Attacker observes the pending or confirmed transaction.
3. Attacker calls `router.unwrapWETH9(0, attacker)`:
   - `balanceWETH` = victim's full WETH amount
   - `amountMinimum = 0` → check passes
   - WETH is withdrawn and ETH is sent to attacker
4. Victim calls `unwrapWETH9(amount, victim)` → reverts with `InsufficientWETH(amount, 0)`.
5. Victim has lost their full swap output; attacker has received it as ETH.

Foundry test: fork the existing `test_multicall_multihop_tokenToWeth_thenUnwrapEth` test, split the two `calls` entries into separate transactions with an attacker `vm.prank` between them calling `router.unwrapWETH9(0, attacker)`, and assert the attacker receives the ETH while the victim's call reverts.
