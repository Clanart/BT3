### Title
Missing Class Hash Existence Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary class hash from the caller without verifying that a contract class with that hash has actually been declared on-chain. An unprivileged transaction sender who can trigger a `replace_class` call with attacker-controlled input can permanently freeze all funds held by the targeted contract by replacing its class with a non-existent hash.

---

### Finding Description

In `execute_replace_class`, the OS reads `request.class_hash` from the syscall request and directly writes it into the contract's `StateEntry` without any validation that the hash corresponds to a declared class:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing check. The `contract_class_changes` dictionary (which maps class hash → compiled class hash for all declared classes) is never consulted. The OS commits the new `StateEntry` with an arbitrary, potentially non-existent class hash into the Patricia state tree.

This is the direct analog to the external report's vulnerability class: a user-supplied value (the new class hash) is accepted and written into protocol state without being validated against the actual on-chain declared-class registry. In the external report, the attacker manipulates the actual balance to break a balance equality check; here, the attacker supplies a class hash that was never declared, causing the OS to commit an invalid state transition.

The `contract_class_changes` dictionary is available as an implicit argument throughout the syscall execution context and is used elsewhere (e.g., in `execute_declare_transaction` to enforce `prev_value=0` uniqueness). The validation is simply absent. [2](#0-1) 

---

### Impact Explanation

Once the OS commits a `StateEntry` with a non-existent class hash into the global state root, every subsequent call to that contract will fail at class-loading time because no compiled class exists for the hash. The contract becomes permanently inoperable. Any ERC-20 balances, ETH, or other assets held in the contract's storage are permanently frozen with no recovery path, since no entry point can execute. This satisfies **Critical: Permanent freezing of funds**.

---

### Likelihood Explanation

The attack requires a contract that calls `replace_class` with caller-controlled input — a realistic pattern for:

- Upgradeable/proxy contracts where the upgrade selector is exposed to governance or users without strict access control
- Contracts that allow a user-specified class hash as part of a migration or self-upgrade flow

The attacker does not need any privileged role. They only need to submit a standard invoke transaction targeting such a contract. The OS-level missing check is the necessary vulnerable step: even a correctly-audited contract cannot defend against this if the OS itself does not reject non-existent class hashes at the syscall boundary.

---

### Recommendation

Before writing the new `StateEntry`, validate that `request.class_hash` exists as a declared class by performing a lookup in `contract_class_changes`. Specifically, perform a `dict_read` on `contract_class_changes` for `request.class_hash` and assert the result is non-zero (i.e., a compiled class hash has been registered). This mirrors the existing pattern used in `compute_class_commitment` and `hash_class_changes`, which operate on the same dictionary. [3](#0-2) 

---

### Proof of Concept

1. Deploy contract `V` that holds user funds and exposes an upgrade function: `fn upgrade(new_class_hash: felt252)` which calls the `replace_class` syscall with the caller-supplied `new_class_hash`.
2. Attacker calls `V.upgrade(0xdeadbeef)` where `0xdeadbeef` is a felt that has never been declared as a class on-chain.
3. The OS processes the `replace_class` syscall via `execute_replace_class`. It reads `class_hash = 0xdeadbeef` from the request, skips the missing existence check, and calls `dict_update` to write a new `StateEntry(class_hash=0xdeadbeef, ...)` for contract `V`.
4. `state_update` squashes and commits this entry into the Patricia contract state tree, producing a new valid global state root that includes `V`'s class hash as `0xdeadbeef`.
5. Any subsequent transaction calling any entry point of `V` fails at class resolution time — no compiled class exists for `0xdeadbeef`.
6. All funds stored in `V`'s storage slots are permanently inaccessible. [4](#0-3) [5](#0-4)
