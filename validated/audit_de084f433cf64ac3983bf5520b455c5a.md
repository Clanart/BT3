### Title
Felt Arithmetic Overflow in `compute_max_possible_fee` Enables Complete Fee Bypass — (File: `execution/transaction_impls.cairo`)

---

### Summary

The `compute_max_possible_fee` function performs unchecked felt-field arithmetic on attacker-controlled `ResourceBounds` values. Because Cairo arithmetic is modular (mod the Stark prime P ≈ 2²⁵¹ + 17·2¹⁹² + 1), an attacker can craft resource bounds whose product-sum wraps to exactly 0 mod P. The `charge_fee` caller treats a zero result as "no fee required" and returns immediately, allowing the transaction to execute without paying any fee.

---

### Finding Description

**Root cause — `compute_max_possible_fee`** (`transaction_impls.cairo`, lines 87–102):

```cairo
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    ...
    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
``` [1](#0-0) 

Every field of `ResourceBounds` is a Cairo `felt`. No `assert_nn_le` or `assert_lt` range-check is applied to `max_amount` or `max_price_per_unit` before the multiplication. The entire expression is evaluated modulo P.

**Trigger — `charge_fee`** (`transaction_impls.cairo`, lines 111–165):

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);

if (max_fee == 0) {
    return ();          // ← fee charging is completely skipped
}
``` [2](#0-1) 

When `max_fee == 0`, the function returns before the `%{ LoadActualFee %}` hint is ever reached, before `assert_nn_le(calldata.amount.low, max_fee)` is evaluated, and before the ERC-20 `transfer` call is dispatched. The sequencer receives nothing.

**Concrete overflow construction:**

Let P be the Stark prime. Set:

| Field | Value |
|---|---|
| `l1_gas_bounds.max_amount` | `P − 1` (felt −1) |
| `l1_gas_bounds.max_price_per_unit` | `1` |
| `l2_gas_bounds.max_amount` | `1` |
| `l2_gas_bounds.max_price_per_unit` | `1` |
| `tip` | `0` |
| `l1_data_gas_bounds.max_amount` | `0` |

Result: `(P−1)·1 + 1·(1+0) + 0 = P ≡ 0 (mod P)` → `max_fee = 0`.

The resource bounds are loaded via hints (`%{ LoadCommonTxFields %}`) and placed directly into `TxInfo` with no range validation anywhere in the OS execution path. [3](#0-2) 

The transaction hash commits to these values, so the hash is valid; the OS simply never checks that the individual fields are within the protocol-specified u64/u128 bounds before multiplying them.

---

### Impact Explanation

**Direct loss of funds (Critical).** The sequencer's off-chain validation uses native integer arithmetic and computes a large positive `max_fee`, so it includes the transaction and expects to receive a fee. The OS, using felt arithmetic, computes `max_fee = 0` and skips the ERC-20 transfer entirely. The user's fee-token balance is not debited; the sequencer's balance is not credited. The transaction's `__execute__` entry point still runs to completion. The sequencer provides execution services and receives zero compensation.

---

### Likelihood Explanation

**Medium.** The attack requires only that an unprivileged sender submit a V3 transaction with carefully chosen `ResourceBounds` values. The gateway and sequencer both validate using native integer arithmetic (not felt), so they see a large, apparently valid `max_fee` and admit the transaction. The discrepancy only manifests inside the OS prover. No privileged role, leaked key, or malicious operator is required — only the ability to submit a standard transaction.

---

### Recommendation

Add explicit range checks on each `ResourceBounds` field before the multiplication in `compute_max_possible_fee`. For example, enforce that `max_amount < 2^64` and `max_price_per_unit < 2^128` using `assert_nn_le`, matching the StarkNet protocol specification. This prevents any product from reaching or wrapping around the field prime.

```cairo
// Example guard (to be added before the return statement):
assert_nn_le(l1_gas_bounds.max_amount, MAX_U64);
assert_nn_le(l1_gas_bounds.max_price_per_unit, MAX_U128);
// ... repeat for l2 and l1_data bounds ...
```

---

### Proof of Concept

1. Construct a V3 invoke transaction with:
   - `l1_gas_bounds = { max_amount: P−1, max_price_per_unit: 1 }`
   - `l2_gas_bounds = { max_amount: 1, max_price_per_unit: 1 }`, `tip = 0`
   - `l1_data_gas_bounds = { max_amount: 0, max_price_per_unit: 0 }`
2. Submit to the gateway. The gateway computes `max_fee ≈ P−1` (integer), a valid large bound, and forwards the transaction.
3. The sequencer includes the transaction in a block.
4. The OS calls `compute_max_possible_fee`: `(P−1)·1 + 1·1 + 0 = P ≡ 0 (mod P)`.
5. `charge_fee` hits `if (max_fee == 0) { return (); }` and exits immediately.
6. The ERC-20 transfer is never executed. The transaction's `__execute__` body runs normally.
7. The proven state update reflects zero fee paid; the sequencer's fee-token balance is unchanged. [4](#0-3)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
