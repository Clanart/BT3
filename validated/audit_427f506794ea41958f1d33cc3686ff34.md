### Title
`ProofOfInclusion::valid()` Never Verifies Against External Tree Root ŌĆö Forged Inclusion Proofs Always Pass - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary
`ProofOfInclusion::valid()` contains a tautological final check that is always `true` when the loop completes, and the function accepts no external root hash parameter. Any internally-consistent but entirely fabricated `ProofOfInclusion` ŌĆö including one with an empty `layers` vector ŌĆö passes validation unconditionally. This is the direct analog of the reported pattern: a value (the proof's claimed root) is included in the verification calculation but is never compared against the actual, trusted tree root.

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows: [1](#0-0) 

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // ŌåÉ always equals layer.combined_hash
    }

    existing_hash == self.root_hash()      // ŌåÉ tautology
}
```

`root_hash()` is: [2](#0-1) 

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // same value as existing_hash after loop
    } else {
        self.node_hash              // same value as existing_hash when layers is empty
    }
}
```

**The tautology:** The loop only continues when `calculated_hash == layer.combined_hash`, and then sets `existing_hash = calculated_hash`. After the last iteration, `existing_hash` equals `last.combined_hash`. `root_hash()` also returns `last.combined_hash`. Therefore `existing_hash == self.root_hash()` reduces to `last.combined_hash == last.combined_hash`, which is unconditionally `true`. For the empty-layers case, `existing_hash` is never modified from `self.node_hash`, and `root_hash()` returns `self.node_hash`, so the check is again `self.node_hash == self.node_hash` ŌĆö always `true`.

**The missing check:** The function takes no external root hash parameter and never compares the proof's claimed root against the actual, trusted root stored in the `MerkleBlob`. The function only verifies internal self-consistency of the proof chain, not that the chain terminates at the real tree root.

`ProofOfInclusion` is a `Streamable` type exposed through Python and wasm bindings: [3](#0-2) 

It can be deserialized from arbitrary bytes received over the network. All callers ŌĆö including the Python test suite ŌĆö rely solely on `valid()` as the complete verification step: [4](#0-3) 

### Impact Explanation

An attacker can forge a `ProofOfInclusion` for any key-value pair that is **not** in the DataLayer tree and have it accepted as valid. The simplest exploit:

1. Choose any fake `(key, value)` pair.
2. Compute `fake_leaf_hash = hash(fake_key, fake_value)`.
3. Construct `ProofOfInclusion { node_hash: fake_leaf_hash, layers: vec![] }`.
4. Call `proof.valid()` ŌåÆ returns `true` unconditionally.

A more sophisticated attacker can build an arbitrary-depth internally-consistent chain of fake layers, all of which pass `valid()`. The DataLayer Merkle proof logic therefore accepts forged inclusion proofs, letting untrusted serialized input prove invalid state ŌĆö matching the **High** impact category: *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.*

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` network object exposed through Python and wasm bindings. Any code path that deserializes a `ProofOfInclusion` from an untrusted peer and calls `valid()` is immediately exploitable with a trivially crafted payload. No special privileges, keys, or chain state are required.

### Recommendation

`valid()` must accept an external trusted root hash and compare against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // compare against the actual tree root
}
```

All call sites must supply the root hash obtained from the trusted `MerkleBlob` (via `merkle_blob.get_root()` or equivalent), not from the proof itself.

### Proof of Concept

```rust
use chia_datalayer::{Hash, ProofOfInclusion};

// Forge a proof for a key-value pair that does not exist in any tree.
let fake_node_hash: Hash = [0xde; 32];  // arbitrary

let forged_proof = ProofOfInclusion {
    node_hash: fake_node_hash,
    layers: vec![],   // empty ŌĆö no real tree path needed
};

// valid() returns true unconditionally:
assert!(forged_proof.valid());   // passes ŌĆö forged proof accepted

// root_hash() returns fake_node_hash, not the real tree root:
assert_eq!(forged_proof.root_hash(), fake_node_hash);
```

The same attack works with non-empty `layers` as long as each layer's `combined_hash` equals the hash computed from the previous layer and the attacker-chosen `other_hash` ŌĆö a trivially satisfiable constraint since the attacker controls all fields.
