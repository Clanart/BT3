### Title
Missing Chain Binding in `deployToken` Signature Verification Enables Cross-Chain Replay — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

The `deployToken` function in `OmniBridge.sol` verifies a MPC/NEAR-signed payload that contains no chain identifier. A signature produced for one EVM deployment can be replayed verbatim on any other EVM deployment that shares the same `nearBridgeDerivedAddress`, permanently poisoning the token registry on the target chain and blocking all future legitimate bridging of that token there.

---

### Finding Description

`deployToken` constructs the signed message as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);

if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

The five-field payload contains **no `omniBridgeChainId`, no EVM chain ID, and no contract address**. The same `(signatureData, metadata)` tuple is therefore valid on every EVM chain where the bridge is deployed with the same `nearBridgeDerivedAddress`.

Contrast this with `finTransfer`, which explicitly encodes `omniBridgeChainId` twice — once for the token address context and once for the recipient context — providing chain binding:

```solidity
bytes memory borshEncoded = bytes.concat(
    ...
    bytes1(omniBridgeChainId),
    Borsh.encodeAddress(payload.tokenAddress),
    Borsh.encodeUint128(payload.amount),
    bytes1(omniBridgeChainId),
    Borsh.encodeAddress(payload.recipient),
    ...
);
``` [2](#0-1) 

`deployToken` is the only public entry point that writes to the `nearToEthToken` / `ethToNearToken` / `isBridgeToken` mappings without chain binding.

After a successful replay, the guard:

```solidity
require(
    !isBridgeToken[nearToEthToken[metadata.token]],
    "ERR_TOKEN_EXIST"
);
``` [3](#0-2) 

permanently prevents the NEAR bridge from ever legitimately deploying that token on the target chain through the normal flow, because the slot is already occupied by the attacker-triggered proxy.

---

### Impact Explanation

An attacker who observes a valid `deployToken` call on chain A (e.g., Ethereum mainnet) can immediately replay the identical `(signatureData, metadata)` on chain B (e.g., Base, Arbitrum, or any other deployment sharing `nearBridgeDerivedAddress`). The result:

1. A bridge-token proxy is deployed on chain B at an address the NEAR bridge did not authorize for chain B.
2. `nearToEthToken[metadata.token]` on chain B is set to this attacker-triggered address.
3. All subsequent legitimate `deployToken` calls for the same `metadata.token` on chain B revert with `ERR_TOKEN_EXIST`.
4. `finTransfer` calls from NEAR for that token on chain B will reference the NEAR-side canonical address (which the NEAR bridge computed for chain B), not the attacker-triggered address, so minting will either go to the wrong token or fail — permanently breaking bridging of that asset to chain B.

This matches the allowed High impact: **"Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized transfer finalization, token deployment, or message execution."**

---

### Likelihood Explanation

- The Omni Bridge is explicitly designed to be deployed on multiple EVM chains simultaneously.
- `deployToken` calls are public on-chain events; any observer can extract `signatureData` and `metadata` from chain A's transaction data.
- No privileged access, leaked key, or colluding operator is required — a single unprivileged user can perform the replay.
- The only precondition is that the target chain B deployment has not yet registered the token, which is the normal state before a token is officially bridged to a new chain.

---

### Recommendation

Include `omniBridgeChainId` (and optionally the contract address) in the signed payload for `deployToken`, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // add chain binding
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

The NEAR-side MPC signer must include the target chain ID when producing `deployToken` signatures, just as it already does for `finTransfer`.

---

### Proof of Concept

1. NEAR bridge authorizes token `"usdc.near"` on Ethereum mainnet (chain A, `omniBridgeChainId = 1`). The resulting transaction contains `signatureData_A` and `metadata = {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6}`.

2. Attacker observes the transaction on chain A and extracts `signatureData_A`.

3. Attacker calls `deployToken(signatureData_A, metadata)` on Base (chain B, `omniBridgeChainId = 2`), where the same `nearBridgeDerivedAddress` is configured.

4. `ECDSA.recover(keccak256(borshEncoded), signatureData_A)` returns `nearBridgeDerivedAddress` — the check passes because the payload is identical (no chain field).

5. A `BridgeToken` proxy is deployed on chain B at address `0xATTACKER_PROXY`. `nearToEthToken["usdc.near"]` on chain B is set to `0xATTACKER_PROXY`.

6. When the NEAR bridge later tries to officially deploy `"usdc.near"` on chain B, `deployToken` reverts with `ERR_TOKEN_EXIST`.

7. All `finTransfer` calls for `"usdc.near"` on chain B that reference the NEAR-computed canonical address will fail or mint to the wrong token, permanently breaking USDC bridging to chain B.
