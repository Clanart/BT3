Audit Report

## Title
Router `pay()` silently consumes stranded native ETH from any prior caller, enabling theft of victim's excess ETH — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` helper in `PeripheryPayments` uses `address(this).balance` — the router's entire native ETH balance — when settling WETH-input swaps, completely bypassing the `payer` argument whenever sufficient native ETH is present. Because `exactOutputSingle` and `exactInputSingle` are `payable` but never auto-refund excess ETH, any ETH left on the router after one user's call is silently consumed by the next caller who swaps with WETH as `tokenIn`, giving that caller a free swap and permanently losing the original depositor's ETH.

## Finding Description

**Root cause — `pay()` ignores `payer` when native ETH covers the full amount**

In `PeripheryPayments.sol` lines 73–84:
```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // payer never touched
    } else if (nativeBalance > 0) {
        ...
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```
When `nativeBalance >= value`, the function deposits the router's own ETH balance and transfers the resulting WETH to the pool. The `payer` (set to `msg.sender` of the swap call via transient storage) is never consulted or charged.

**How ETH becomes stranded**

`exactOutputSingle` (L130–147) and `exactInputSingle` (L67–86) are both `payable` and accept `msg.value`, but neither auto-refunds excess ETH. For `exactOutputSingle`, the user cannot know the exact `amountIn` in advance and must send up to `amountInMaximum`. The pool determines the actual `amountIn`; the callback calls `pay(WETH, user, pool, amountIn)`. If `amountIn < msg.value`, the difference remains on the router with no refund.

The protocol's own test (`test_multicall_ethInput_exactInputSingle_refundsUnusedEth`, L106–133) acknowledges this: it requires an explicit `refundETH()` call inside a `multicall` to recover excess. A user calling any swap function directly (outside a multicall) has no atomic way to reclaim the excess.

**Attack path**

1. Victim calls `exactOutputSingle{value: 1_000}` with WETH as `tokenIn`, `amountOut = X`. Pool requires only 600 wei. Callback: `pay(WETH, victim, pool, 600)` — `nativeBalance = 1000 >= 600`, deposits 600 wei as WETH, pays pool. 400 wei remains on router.
2. Attacker calls `exactInputSingle{value: 0}` with WETH as `tokenIn`, `amountIn = 400`. Callback: `pay(WETH, attacker, pool, 400)`. `address(this).balance = 400 >= 400` → router deposits victim's 400 wei as WETH, pays pool. Attacker receives token output at zero cost.

The attacker's `payer` address (set via `_setNextCallbackContext` at L71, read via `_getPayer()` at L195) is never charged because the `nativeBalance >= value` branch in `pay()` never calls `safeTransferFrom`.

**Why `receive()` does not prevent stranding**

`receive()` (L32–34) blocks plain ETH transfers from non-WETH addresses, but `msg.value` attached to a payable function call bypasses `receive()` entirely. ETH sent with `exactOutputSingle{value: X}` lands on the router's balance without triggering `receive()`.

## Impact Explanation
Direct loss of user principal. The victim's excess ETH — which can be an arbitrarily large fraction of `amountInMaximum` — is consumed by the attacker's swap. The attacker receives real token output at zero cost. This meets the Critical/High threshold of direct loss of user principal above Sherlock thresholds.

## Likelihood Explanation
Medium. The trigger condition — ETH stranded on the router — arises naturally whenever a user calls `exactOutputSingle` directly (not via a `multicall` that includes `refundETH()`), which is the common case since the user cannot know the exact `amountIn` in advance. An attacker monitoring the mempool or the router's ETH balance can exploit this immediately after the victim's transaction confirms. The attack requires no special privileges and is repeatable.

## Recommendation
1. **Auto-refund excess ETH** at the end of `exactOutputSingle` and `exactOutput`: after the swap completes and `amountIn` is known, refund `msg.value - amountIn` to `msg.sender`.
2. **Alternatively**, track only the current call's ETH contribution: compute `uint256 ethBefore = address(this).balance - msg.value` at entry, and in `pay()` only allow spending up to `msg.value` worth of native ETH (not any pre-existing balance).
3. **At minimum**, document prominently that every payable swap call must be wrapped in a `multicall` ending with `refundETH()`, and consider reverting if `msg.value > 0` when `tokenIn != WETH`.

## Proof of Concept

```solidity
function test_strandedEthStolenByAttacker() external {
    uint128 amountOut = 1_000;
    // Victim sends 1_000 wei directly — no multicall, no refundETH
    // Actual amountIn = 600 wei; 400 wei stranded on router
    vm.prank(victim);
    router.exactOutputSingle{value: 1_000}(
        IMetricOmmSimpleRouter.ExactOutputSingleParams({
            pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
            zeroForOne: true, amountOut: amountOut, amountInMaximum: 1_000,
            recipient: victim, deadline: type(uint256).max, priceLimitX64: 0, extensionData: ""
        })
    );
    assertEq(address(router).balance, 400); // stranded

    // Attacker has zero ETH and zero WETH
    vm.prank(attacker);
    router.exactInputSingle{value: 0}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
            zeroForOne: true, amountIn: 400, amountOutMinimum: 0,
            recipient: attacker, deadline: type(uint256).max, priceLimitX64: 0, extensionData: ""
        })
    );
    // pay() path: nativeBalance=400 >= value=400 → deposits victim's ETH, attacker never charged
    assertGt(token1.balanceOf(attacker), 0, "attacker got free output");
    assertEq(address(router).balance, 0, "stranded ETH consumed");
}
```

The `pay()` call resolves to the `nativeBalance >= value` branch (L75–77), depositing the victim's stranded ETH as WETH and transferring it to the pool, while the attacker's `payer` address stored in transient storage is never accessed.
