### Title
Missing Recipient Account Validation in `FinalizeTransfer` and `FinalizeTransferSol` Allows Attacker to Redirect Bridged Assets - (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`, `finalize_transfer_sol.rs`)

---

### Summary

In the Solana bridge program's `FinalizeTransfer` and `FinalizeTransferSol` instructions, the `recipient` account passed by the caller is never validated against the `recipient` field embedded in the MPC-signed `FinalizeTransferPayload`. Any unprivileged caller who obtains a valid signed payload (publicly observable from NEAR bridge events) can substitute their own address as `recipient`, redirecting the bridged tokens or SOL to themselves. The destination nonce is consumed in the same transaction, permanently preventing the legitimate recipient from ever claiming their funds.

---

### Finding Description

In `FinalizeTransfer`, the accounts struct declares `recipient` as a bare `UncheckedAccount` with no Anchor constraint tying it to the payload:

```rust
/// CHECK: this can be any type of account
pub recipient: UncheckedAccount<'info>,
```

The `token_account` is then derived as the Associated Token Account of whichever `recipient` pubkey the caller supplies:

```rust
#[account(
    init_if_needed,
    payer = common.payer,
    associated_token::mint = mint,
    associated_token::authority = recipient,   // ← derived from caller-supplied recipient
    token::token_program = token_program,
)]
pub token_account: Box<InterfaceAccount<'info, TokenAccount>>,
```

Inside `process()`, the actual token transfer (vault→token_account or mint→token_account) targets `self.token_account`, which belongs to the attacker-supplied `recipient`, not the one in `data.recipient`:

```rust
TransferChecked {
    from: vault.to_account_info(),
    to: self.token_account.to_account_info(),   // ← attacker's ATA
    ...
}
```

and for the bridged (mint) path:

```rust
MintTo {
    mint: self.mint.to_account_info(),
    to: self.token_account.to_account_info(),   // ← attacker's ATA
    ...
}
```

The same structural flaw exists in `FinalizeTransferSol`, where native SOL is transferred directly to the caller-supplied `recipient` with no validation:

```rust
/// CHECK: this can be any type of account
#[account(mut)]
pub recipient: UncheckedAccount<'info>,
```

```rust
transfer(
    CpiContext::new_with_signer(..., Transfer {
        from: self.sol_vault.to_account_info(),
        to: self.recipient.to_account_info(),   // ← attacker-controlled
    }, ...),
    data.amount...,
)?;
```

Critically, `UsedNonces::use_nonce` is called in the same transaction, consuming the destination nonce. The legitimate recipient can never replay the same signed payload. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Critical — Direct theft of bridged assets on Solana.**

An attacker can steal 100% of any in-flight cross-chain transfer destined for Solana. Both SPL token transfers (native vault release and bridged mint) and native SOL transfers are affected. The nonce is consumed on theft, making the loss permanent and irrecoverable for the legitimate recipient. This breaks bridge collateralization: tokens are burned/locked on the source chain (NEAR/EVM) but delivered to the wrong party on Solana.

---

### Likelihood Explanation

**High.** The signed `FinalizeTransferPayload` is broadcast publicly as a NEAR chain event (via `SignTransferEvent` in `near/omni-bridge/src/lib.rs`). Any observer can extract the valid MPC signature and payload, then front-run or race the legitimate relayer by submitting `finalize_transfer` with their own `recipient` address. No privileged access, leaked key, or colluding party is required — only the ability to submit a Solana transaction. [8](#0-7) 

---

### Recommendation

Add an Anchor constraint in both `FinalizeTransfer` and `FinalizeTransferSol` that enforces the `recipient` account matches the recipient encoded in the signed payload:

```rust
// In FinalizeTransfer
/// CHECK: validated against signed payload recipient
#[account(
    constraint = recipient.key() == data.payload.recipient @ ErrorCode::InvalidRecipient
)]
pub recipient: UncheckedAccount<'info>,
```

```rust
// In FinalizeTransferSol
/// CHECK: validated against signed payload recipient
#[account(
    mut,
    constraint = recipient.key() == data.payload.recipient @ ErrorCode::InvalidRecipient
)]
pub recipient: UncheckedAccount<'info>,
```

Similarly, add a constraint that `mint.key() == data.payload.token_address` in `FinalizeTransfer` to prevent token substitution attacks on the same payload.

---

### Proof of Concept

1. Alice initiates a transfer of 1000 USDC from NEAR to Solana. The NEAR bridge emits a `SignTransferEvent` containing the MPC-signed `FinalizeTransferPayload` with `recipient = Alice_Solana_Pubkey` and `destination_nonce = N`.
2. Attacker Bob observes this event on-chain and extracts the signed payload.
3. Bob calls `finalize_transfer` on Solana with:
   - `data` = the valid signed payload (unchanged, passes MPC signature verification)
   - `recipient` = `Bob_Solana_Pubkey` (substituted)
   - `token_account` = Bob's ATA for the mint (derived from Bob's pubkey, created by `init_if_needed`)
   - `mint` = the correct USDC mint
4. The program verifies the MPC signature (passes — payload is unmodified), marks nonce `N` as used, and transfers 1000 USDC to Bob's ATA.
5. Alice's nonce `N` is now consumed. Any subsequent attempt by Alice or a relayer to call `finalize_transfer` with the same payload fails with `NonceAlreadyUsed`.
6. Alice has permanently lost her 1000 USDC with no recourse.

The same attack applies to `finalize_transfer_sol` for native SOL transfers.
