### Title
Fee Market Base Fee Fails to Increase When Block Is Full by gasLimit Due to MinGasMultiplier Capping ŌĆö (`x/feemarket/keeper/abci.go`, `x/feemarket/keeper/eip1559.go`)

---

### Summary

An unprivileged attacker can submit a transaction with `gasLimit = blockGasLimit` and minimal `gasUsed`, causing `EndBlock` to store a `blockGasWanted` equal to the EIP-1559 target rather than the full block limit. As a result, the base fee does not increase even though the block is at capacity by `gasLimit`, violating the core EIP-1559 invariant.

---

### Finding Description

**`EndBlock` logic** (`x/feemarket/keeper/abci.go`):

```go
gasWanted := ctx.BlockGasWanted()          // sum of tx gasLimits = blockGasLimit (L)
gasUsed   := ctx.BlockGasUsed()            // actual execution gas = ~1

minGasMultiplier := k.GetParams(ctx).MinGasMultiplier   // 0.5 (default)
limitedGasWanted := sdkmath.LegacyNewDec(gw).Mul(minGasMultiplier)  // L * 0.5
gasWanted = sdkmath.LegacyMaxDec(limitedGasWanted, sdkmath.LegacyNewDec(gasUsed)).TruncateInt().Uint64()
// = max(L/2, 1) = L/2
k.SetBlockGasWanted(ctx, gasWanted)        // stores L/2
``` [1](#0-0) 

**`CalculateBaseFee` logic** (`x/feemarket/keeper/eip1559.go`):

```go
parentGasUsed   := k.GetBlockGasWanted(ctx)   // L/2 (stored above)
parentGasTarget := gasLimit / ElasticityMultiplier  // L / 2 = L/2 (default ElasticityMultiplier=2)

if parentGasUsed == parentGasTarget {
    return new(big.Int).Set(parentBaseFee)    // base fee UNCHANGED
}
``` [2](#0-1) 

With default parameters (`MinGasMultiplier = 0.5`, `ElasticityMultiplier = 2`), the cap `gasWanted * MinGasMultiplier` produces exactly `L/2`, which equals `parentGasTarget`. The base fee comparison sees `parentGasUsed == parentGasTarget` and returns the fee unchanged.

The default `MinGasMultiplier` is `0.5`: [3](#0-2) 

---

### Impact Explanation

The EIP-1559 invariant requires the base fee to **increase** when a block is filled to its gas limit. Here, an attacker who submits a single transaction with `gasLimit = blockGasLimit` and `gasUsed Ōēł 0`:

1. Fills the block (no other transactions can be included ŌĆö censorship).
2. Pays fees only on actual `gasUsed` (near-zero cost).
3. Causes `EndBlock` to store `blockGasWanted = L/2 = parentGasTarget`.
4. Causes `CalculateBaseFee` to return the base fee **unchanged** instead of increasing it.

Repeating this every block keeps the base fee permanently suppressed while blocking all other users' transactions. This is a concrete fee market mis-accounting path reachable by any unprivileged user through normal transaction submission.

**Correction on the question's claim:** With the exact parameters described (`gasWanted = blockGasLimit`, `gasUsed = 1`, `MinGasMultiplier = 0.5`, `ElasticityMultiplier = 2`), the base fee stays **the same** ŌĆö it does not decrease. Decrease would require `gasWanted < blockGasLimit`, but that would leave room for other transactions. The actual harm is that the base fee **fails to increase** when the block is at capacity, not that it decreases.

---

### Likelihood Explanation

- Requires no privileges ŌĆö any account can set `gasLimit` to any value up to `blockGasLimit`.
- Default parameters (`MinGasMultiplier = 0.5`, `ElasticityMultiplier = 2`) make the coincidence exact: `limitedGasWanted = parentGasTarget`.
- Cost to attacker: `baseFee * gasUsed` per block (near-zero if `gasUsed` is minimal, e.g., a simple ETH transfer using 21,000 gas against a block limit of millions).
- Sustained attack is economically viable at low base fees.

---

### Recommendation

The `EndBlock` capping should account for the relationship between `MinGasMultiplier` and `ElasticityMultiplier`. One approach: when `gasWanted >= blockGasLimit`, bypass the `MinGasMultiplier` cap and store `gasWanted` as-is (or store `blockGasLimit`), so that a full block always signals congestion to the base fee algorithm. Alternatively, base the fee calculation on `max(limitedGasWanted, gasUsed)` only when `gasWanted < blockGasLimit`, and use `blockGasLimit` directly when the block is at capacity.
