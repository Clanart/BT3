### Title
Stale Declaration-Block Compiled Class Hash Returned by `ExecutionStateReader` After Migration — (`File: crates/apollo_rpc_execution/src/state_reader.rs`)

### Summary

`ExecutionStateReader::get_compiled_class_hash` resolves the compiled class hash by looking up the **declaration block** via `get_class_definition_block_number` and then reading that block's state diff. After the compiled-class-hash migration (v1 Pedersen → v2 Poseidon) is committed to the Patricia trie, the declaration-block state diff still holds the old v1 hash. Every subsequent RPC call that exercises this path (`starknet_estimateFee`, `starknet_simulateTransactions`, `starknet_traceTransaction`) therefore receives a stale v1 hash instead of the current v2 hash, producing authoritative-looking wrong fee estimates and simulation state diffs.

### Finding Description

`ExecutionStateReader::get_compiled_class_hash` (used by the RPC execution layer for fee estimation and simulation) resolves the compiled class hash by:

1. Calling `get_class_definition_block_number(&class_hash)` — which reads from `declared_classes_block_table` and always returns the **original declaration block**, never the migration block.
2. Calling `get_state_diff(block_number)` on that declaration block and extracting `class_hash_to_compiled_class_hash`. [1](#0-0) 

The declaration-block state diff permanently records the v1 (Pedersen) hash that was submitted with the original `DeclareV3` transaction. When `set_compiled_class_hash_migration` writes the v2 (Poseidon) hash into the block state cache during sequencing, that write is committed to the Patricia trie in a **later** migration block's state diff — but `get_class_definition_block_number` is never updated; it always points to the original declaration block. [2](#0-1) 

By contrast, `ApolloReader::get_compiled_class_hash` (used by the batcher/sequencer itself) correctly calls `get_compiled_class_hash_at(state_number, &class_hash)`, which reads the Patricia-trie value at the requested state number and therefore reflects the post-migration v2 hash. [3](#0-2) 

The storage layer exposes `get_compiled_class_hash_at(state_number, class_hash)` that accumulates all state-diff updates (including migration blocks) up to the requested state number, as confirmed by the storage tests: [4](#0-3) 

`ExecutionStateReader` has access to the same `storage_reader` and `state_number` fields but never calls this function.

### Impact Explanation

Every RPC endpoint that invokes `exec_estimate_fee` or `exec_simulate_transactions` constructs an `ExecutionStateReader` and wraps it in a `CachedState`. When the blockifier calls `get_compiled_class_hash` on a migrated class, it receives v1 instead of v2. Two concrete wrong-value scenarios follow:

1. **Spurious migration in simulation state diff.** The `CachedState` records v1 as the initial read. If the simulated transaction writes v2 (e.g., a re-declare or a migration-triggering invoke), `to_state_diff()` emits a `class_hash_to_compiled_class_hash` entry `{C → v2}` even though the on-chain state already holds v2. `starknet_simulateTransactions` returns this diff as authoritative, misleading callers about what state changes will actually occur. [5](#0-4) 

2. **Wrong fee estimate for declare transactions.** When estimating a `DeclareV3` for a class whose on-chain compiled hash is already v2, the blockifier sees v1 as the current hash. It therefore treats the transaction as a valid new declaration (v1 ≠ v2 in the initial read), computes a non-zero fee, and returns success. The actual sequenced transaction would fail because the class is already declared at v2, making the fee estimate an authoritative-looking wrong value. [6](#0-5) 

Impact: **High** — RPC fee estimation and simulation return authoritative-looking wrong values.

### Likelihood Explanation

The migration flag `enable_casm_hash_migration` is a `versioned_constants` field toggled per block context. Tests confirm it is exercised in production code paths: [7](#0-6) 

Once any Cairo 1 class is migrated (i.e., executed after the flag is enabled), every subsequent RPC call that queries the compiled class hash for that class via `ExecutionStateReader` will receive the stale v1 value. The trigger is unprivileged: any user can call `starknet_estimateFee` or `starknet_simulateTransactions` with a migrated class hash.

### Recommendation

Replace the declaration-block state-diff lookup in `ExecutionStateReader::get_compiled_class_hash` with a direct Patricia-trie read at `self.state_number`:

```rust
fn get_compiled_class_hash(&self, class_hash: ClassHash) -> StateResult<CompiledClassHash> {
    // Check pending data first (unchanged).
    if let Some(pending_data) = &self.maybe_pending_data {
        for DeclaredClassHashEntry { class_hash: other_class_hash, compiled_class_hash }
            in &pending_data.declared_classes
        {
            if class_hash == *other_class_hash {
                return Ok(*compiled_class_hash);
            }
        }
    }

    // Read from the Patricia trie at the requested state_number, not the declaration block.
    let compiled_class_hash = self
        .storage_reader
        .begin_ro_txn()
        .map_err(storage_err_to_state_err)?
        .get_state_reader()
        .map_err(storage_err_to_state_err)?
        .get_compiled_class_hash_at(self.state_number, &class_hash)
        .map_err(storage_err_to_state_err)?;

    Ok(compiled_class_hash.unwrap_or_default())
}
```

This mirrors `ApolloReader::get_compiled_class_hash` and ensures that post-migration v2 hashes are visible to RPC execution.

### Proof of Concept

1. Deploy and declare Cairo 1 class `C` in block `N`; the declaration state diff records `{C → compiled_hash_v1}`. `get_class_definition_block_number(C)` returns `N`.

2. In block `M > N`, an invoke transaction executes `C` with `enable_casm_hash_migration = true`. `set_compiled_class_hash_migration` writes `{C → compiled_hash_v2}` into the block state cache; the committer commits this to the Patricia trie. `get_compiled_class_hash_at(state_M, C)` now returns `compiled_hash_v2`.

3. A user calls `starknet_estimateFee` at block `M` for a `DeclareV3` transaction with `compiled_class_hash = compiled_hash_v2`.

4. `ExecutionStateReader::get_compiled_class_hash(C)`:
   - `get_class_definition_block_number(C)` → `N` (unchanged, always the declaration block).
   - `get_state_diff(N).class_hash_to_compiled_class_hash[C]` → `compiled_hash_v1` (stale).
   - Returns `compiled_hash_v1`.

5. `CachedState` records `compiled_hash_v1` as the initial read for `C`. The blockifier sees `initial = v1`, `write = v2` → state diff shows `{C → v2}`, and the declare is treated as valid. Fee estimation returns a non-zero fee and success.

6. The actual sequenced transaction fails: the on-chain state already has `compiled_hash_v2` for `C`, so the declare is rejected as a duplicate. The fee estimate was wrong. [8](#0-7) [9](#0-8) [10](#0-9)
