Looking at the full validation path in `set_state_header` and the downstream consumers, I need to determine whether the missing `chunk.shard_id() == shard_id` check is exploitable end-to-end.

### Title
Missing `chunk.shard_id() == shard_id` Guard in `set_state_header` Allows Cross-Shard State Root Corruption — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for shard A that contains shard B's chunk and a valid Merkle proof for shard B's position in the block's `chunk_headers_root` tree. Because `verify_path` only checks that the leaf hashes to the root — it does not check which position (shard index) the leaf occupies — and because no code in the function asserts `chunk.shard_id() == shard_id`, all five validation steps pass. The header is then stored under `StateHeaderKey(shard_id=A, sync_hash)` with shard B's `prev_state_root`. Downstream, `apply_state_part` writes shard B's trie nodes into shard A's `shard_uid` column, and `set_state_finalize` applies shard A's chunk against shard B's state root, producing a permanently corrupted shard A trie.

---

### Finding Description

**Entrypoint:** `set_state_header` in `chain/chain/src/state_sync/adapter.rs`, called by the state-sync downloader after receiving a `ShardStateSyncResponseHeader` from any peer.

**Step-by-step validation trace:**

**Step 1–2 (`validate_chunk_proofs`):** Checks the chunk's internal hash, tx root, and receipts root. [1](#0-0) 
No check on `chunk.shard_id()`. Passes for shard B's chunk.

**Step 3a (`verify_path` against `chunk_headers_root`):** [2](#0-1) 
`verify_path` computes `compute_root_from_path(path, hash(leaf))` and checks equality with the root. [3](#0-2) 
It does **not** check which index (shard position) the leaf occupies. Shard B's chunk with shard B's valid Merkle proof passes because shard B's chunk IS in the tree.

**Step 3b (`verify_path` for `prev_chunk_header`):** [4](#0-3) 
Same issue — no `prev_chunk_header.shard_id() == shard_id` check.

**Step 4 (incoming receipts proofs):** The receipts hash is computed as `ReceiptList(shard_id, receipts)` using the *requested* `shard_id` (A). [5](#0-4) 
The attacker provides valid incoming receipts for shard A (public blockchain data) with valid `root_proofs` derived from the block's `prev_chunk_outgoing_receipts_root`. When shard A and shard B have the same `height_included` and `prev_chunk_header.height_included()` (the common case in a healthy network), step 4g also passes. [6](#0-5) 

**Step 5 (`validate_state_root_node`):** [7](#0-6) 
Checks `hash(&state_root_node.data) == state_root`. The attacker provides shard B's `state_root_node`, which matches shard B's `prev_state_root` from shard B's chunk. Passes.

**Storage (the commit):** [8](#0-7) 
The header — containing shard B's chunk and shard B's `prev_state_root` — is stored under `StateHeaderKey(shard_id=A, sync_hash)`.

**Downstream corruption in `set_state_part`:** [9](#0-8) 
`state_root` is read from the stored header (shard B's state root). `validate_state_part` accepts shard B's parts against shard B's state root.

**Downstream corruption in `apply_state_part`:** [10](#0-9) 
`shard_uid` is derived from `shard_id=A`, but `state_root` is shard B's. Shard B's trie nodes are written into shard A's column.

**Downstream corruption in `set_state_finalize`:** [11](#0-10) 
`apply_chunk` is called with `RuntimeStorageConfig::new(chunk_header.prev_state_root(), true)` where `prev_state_root()` is shard B's state root, applied against shard A's `shard_uid`. The shard A trie is permanently corrupted.

---

### Impact Explanation

The exact corrupted value is `prev_state_root` in the `ShardStateSyncResponseHeader` stored under `StateHeaderKey(shard_id=A, sync_hash)` — it is shard B's state root instead of shard A's. This causes:
- Shard B's trie nodes to be written into shard A's `shard_uid` storage column
- `set_state_finalize` to apply shard A's chunk against shard B's state root, producing an incorrect or failed state transition for shard A
- The syncing node to operate with a permanently wrong shard A state, diverging from the canonical chain

---

### Likelihood Explanation

**Preconditions:**
1. State sync is in progress (the node is syncing)
2. The attacker controls a peer that responds to state sync requests — any network peer qualifies; no validator or operator privilege is required
3. Shard A and shard B both have new chunks in the block before `sync_hash` (the common case)
4. The attacker has access to public blockchain data (block headers, chunk data, receipt proofs)

All preconditions are satisfiable by an unprivileged network peer. The `set_state_header` validation path is explicitly designed to validate data from untrusted peers, making this the intended attack surface.

---

### Recommendation

Add an explicit shard identity check immediately after extracting the chunk, before any Merkle proof verification:

```rust
// In set_state_header, after line 376:
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
// Similarly for prev_chunk_header:
if let Some(ref prev_chunk_header) = prev_chunk_header {
    if prev_chunk_header.shard_id() != shard_id {
        byzantine_assert!(false);
        return Err(Error::Other(
            "set_shard_state failed: prev_chunk shard_id does not match requested shard_id".into(),
        ));
    }
}
```

The `compute_state_response_header` function already enforces this invariant by construction (it selects chunks by `prev_shard_index` derived from `shard_id`), but `set_state_header` — the validation path for untrusted peer data — does not. [12](#0-11)
