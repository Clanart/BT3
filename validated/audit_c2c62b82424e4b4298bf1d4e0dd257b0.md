### Title
Native ETH Transfer to Non-Receivable Contract Recipient Causes Permanent Fund Lock — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
In `OmniBridge.finTransfer`, when the bridged asset is native ETH (`payload.tokenAddress == address(0)`), the contract delivers funds via a bare `.call{value: payload.amount}("")` to `payload.recipient`. If the recipient is a smart contract without a `receive()` or `fallback()` function, the call fails and the entire transaction reverts. Because the recipient address is cryptographically bound inside the MPC-signed payload and there is no admin recovery path, the corresponding cross-chain transfer becomes permanently unfinalisable, locking the user's funds.

### Finding Description

`finTransfer` in `OmniBridge.sol` handles native-ETH delivery at lines 317–322:

```solidity
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [1](#0-0) 

The `payload.recipient` is an EVM address embedded in the MPC-signed `TransferMessagePayload`. The signature is verified against `nearBridgeDerivedAddress` at line 311, meaning the payload — including the recipient — is immutable once signed. [2](#0-1) 

`completedTransfers[payload.destinationNonce] = true` is set at line 287 before the transfer attempt. [3](#0-2) 

Because `revert FailedToSendEther()` rolls back the entire transaction, the nonce is not permanently consumed — the call can be retried. However, the recipient is fixed in the signed message and cannot be changed. Every retry against the same signed payload will hit the same non-receivable contract and revert indefinitely.

There is no admin function in `OmniBridge` to:
- force-mark a nonce as completed without executing the transfer, or
- redirect a transfer to an alternative recipient. [4](#0-3) 

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds.**

The source-chain side of the transfer (NEAR or another chain) has already burned or locked the user's tokens when `initTransfer` was processed. The EVM bridge holds the corresponding ETH. Because `finTransfer` can never succeed for a non-receivable contract recipient, the ETH is permanently stranded in the bridge contract and the user's source-chain tokens are permanently burned. This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation

**Medium.** Smart contract addresses are common EVM recipients: multisig wallets (Gnosis Safe), DAO treasuries, protocol vaults, and smart contract wallets. Many such contracts do not implement `receive()`. A user who specifies any such address as the EVM recipient of a native-ETH bridge transfer will permanently lose their funds. No privileged access or attacker cooperation is required — the user's own legitimate bridge action triggers the lock.

### Recommendation

1. **Wrap-and-transfer pattern:** Instead of sending raw ETH, wrap it to WETH and transfer the ERC-20 token to the recipient. This eliminates the dependency on the recipient having a `receive()` function.
2. **Pull-payment pattern:** Record the owed ETH amount in a mapping keyed by recipient and expose a `withdraw()` function. The recipient pulls funds at their convenience, avoiding a push-to-contract failure.
3. **Admin recovery escape hatch:** Add a privileged function that can mark a nonce as completed and redirect the ETH to an alternative address, callable only after a timeout or with MPC co-signature, to handle stuck transfers.

### Proof of Concept

1. Alice holds native ETH on NEAR (wrapped) and calls the NEAR bridge to initiate a transfer to EVM, specifying `recipient = 0xDeadBeef...` — a deployed contract with no `receive()` function (e.g., a plain ERC-20 token contract or a DAO governor).
2. The NEAR bridge burns Alice's tokens and the MPC signs a `TransferMessagePayload` with `tokenAddress = address(0)`, `amount = 1 ether`, `recipient = 0xDeadBeef...`.
3. A relayer calls `OmniBridge.finTransfer(signatureData, payload)`.
4. Line 287 sets `completedTransfers[nonce] = true`.
5. Line 319 executes `0xDeadBeef....call{value: 1 ether}("")` — the call returns `success = false` because the contract has no `receive()`.
6. Line 322 executes `revert FailedToSendEther()`, rolling back the entire transaction including the nonce flag.
7. Steps 3–6 repeat indefinitely. The ETH remains in the bridge; Alice's NEAR tokens are gone.
