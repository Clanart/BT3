### Title
Unchecked Return Value of Fee Token `transfer` in `charge_fee` Allows Silent Fee Payment Failure — (File: `execution/transaction_impls.cairo`)

---

### Summary

The `charge_fee` function calls the fee token's `transfer` entry point via `non_reverting_select_execute_entry_point_func` but completely discards the return data `(retdata_size, retdata, is_deprecated)`. If the fee token's `transfer` returns a failure indicator (e.g., `false`) without reverting, the OS proceeds as if the fee was successfully paid, allowing transactions to be processed without the sequencer receiving the fee.

---

### Finding Description

In `transaction_impls.cairo`, `charge_fee` (lines 111–165) constructs an `ExecutionContext` targeting the fee token's `TRANSFER_ENTRY_POINT_SELECTOR` and calls:

```cairo
let remaining_gas = DEFAULT_INITIAL_GAS_COST;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=&execution_context
);
return ();
``` [1](#0-0) 

The return tuple `(retdata_size: felt, retdata: felt*, is_deprecated: felt)` is silently dropped. `non_reverting_select_execute_entry_point_func` only asserts `is_reverted = 0` (i.e., the call did not revert), but never inspects the actual return data from the `transfer` call. [2](#0-1) 

By contrast, every `__validate__` call in the same file explicitly checks:

```cairo
assert retdata_size = 1;
assert retdata[0] = VALIDATED;
``` [3](#0-2) 

No equivalent check exists for the fee token's `transfer` return data. This is the direct analog of the ERC20 unchecked-return-value class: the OS enforces non-revert but ignores the semantic success/failure encoded in the return data.

`charge_fee` is invoked for all three account transaction types: [4](#0-3) [5](#0-4) [6](#0-5) 

---

### Impact Explanation

If the fee token's `transfer` entry point returns `false` (or any non-success value) without reverting — a valid behavior for ERC20-like tokens — the OS does not detect the failure. The transaction is committed to the block and the state transition is finalized, but the sequencer never receives the fee. This constitutes **direct loss of funds** for the sequencer across every invoke, deploy-account, and declare transaction type.

---

### Likelihood Explanation

The fee token address is set in the OS config (`starknet_os_config.fee_token_address`). [7](#0-6) 

Any fee token contract whose `transfer` implementation returns `false` on insufficient balance (rather than reverting) — a standard ERC20 pattern — triggers this path. An unprivileged user with insufficient fee-token balance submits a transaction; the sequencer loads the actual fee via the `LoadActualFee` hint; the fee token's `transfer` returns `false`; the OS ignores it and finalizes the block. No privileged access is required beyond submitting a normal transaction.

---

### Recommendation

After calling `non_reverting_select_execute_entry_point_func` in `charge_fee`, check the return data analogously to how `__validate__` return data is checked:

```cairo
let (retdata_size, retdata, is_deprecated) =
    non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
        block_context=block_context, execution_context=&execution_context
    );
if (is_deprecated == 0) {
    assert retdata_size = 1;
    assert retdata[0] = TRUE;  // transfer must return true
}
```

This mirrors the SafeERC20 / SafeTransferLib mitigation recommended in the original report.

---

### Proof of Concept

1. An unprivileged user submits an invoke transaction with `max_fee > 0`.
2. The OS executes the transaction body and then calls `charge_fee`.
3. `charge_fee` constructs a `transfer(sequencer, actual_fee)` call against the fee token contract.
4. The fee token's `transfer` returns `[0]` (false) without reverting — e.g., because the sender's balance is insufficient and the token follows the non-reverting ERC20 pattern.
5. `non_reverting_select_execute_entry_point_func` asserts only `is_reverted = 0` (satisfied, since no revert occurred) and returns `(retdata_size=1, retdata=[0], is_deprecated=0)`.
6. `charge_fee` discards all return values and returns `()`.
7. The OS finalizes the block with the transaction included and the state committed, but the sequencer's fee-token balance is unchanged — the fee was never transferred. [8](#0-7)
