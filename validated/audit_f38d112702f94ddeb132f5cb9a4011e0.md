### Title
`delete_declared_classes` unconditionally erases Sierra class bodies for migration-only entries on state diff revert — (`File: crates/apollo_storage/src/state/mod.rs`)

### Summary

When `revert_state_diff` is called on a block that contains compiled-class-hash migration entries (`migrated_compiled_classes`), `delete_declared_classes` unconditionally deletes the Sierra class body from `declared_classes_table` for every key in `class_hash_to_compiled_class_hash`, regardless of which block originally declared the class. The sibling helper `delete_declared_classes_block` carries a correct `class_block_entry == block_number` guard; `delete_declared_classes` carries none. After the revert, `declared_classes_block_table[C]` still points to the original declaration block N while `declared_classes_table[C]` is gone, producing a permanently inconsistent storage state that causes `get_class_definition_at` to return `StorageError::DBInconsistency`.

### Finding Description

`ThinStateDiff::class_hash_to_compiled_class_hash` is populated from two sources. The comment at line 71 of `starknet_api/src/state.rs` states explicitly:

> "class hash to compiled class hash is affected by both declared_classes and migrated_compiled_classes." [1](#0-0) 

`ThinStateDiff::from_state_diff` chains both sources into the same field: [2](#0-1) 

`enable_casm_hash_migration` is `true` in the current production versioned-constants file: [3](#0-2) 

When enabled, `finalize_block` calls `set_compiled_class_hash_migration`, which writes the new v2 hash into `CachedState`. This flows into `CommitmentStateDiff::class_hash_to_compiled_class_hash` and then into the `ThinStateDiff` stored by `commit_proposal` via `append_state_diff`. [4](#0-3) 

During `revert_state_diff`, two functions handle Sierra class cleanup:

**`delete_declared_classes_block`** (correct): checks `if class_block_entry == block_number` before deleting. For migrated class C first declared at block N, `class_block_entry = N ≠ M` (the reverted block), so the block-number entry is preserved. [5](#0-4) 

**`delete_declared_classes`** (buggy): iterates over ALL keys in `class_hash_to_compiled_class_hash` and deletes from `declared_classes_table` with **no block-number guard**. For migrated class C, `declared_classes_table[C]` (written at block N by `append_classes` during central/P2P sync) is unconditionally deleted. [6](#0-5) 

The contrast with `delete_deprecated_declared_classes`, which does carry the guard `if block_number == declared_block_number`, makes the omission clear: [7](#0-6) 

After the revert:
- `declared_classes_block_table[C] = N` — preserved (correct)
- `declared_classes_table[C]` — **deleted** (incorrect)

`get_class_definition_at` then finds the block-number entry, determines the class should be visible, fails to find the class body, and since `state_number.is_after(class_marker)` is false (the class marker was decremented to M during revert), returns `StorageError::DBInconsistency`: [8](#0-7) 

### Impact Explanation

After the bug triggers, any RPC call to `starknet_getClass` for class C returns a `DBInconsistency` error instead of the class definition. The state is permanently inconsistent: the class appears declared (block-number entry exists, state diff commitment for block N includes C) but its body is inaccessible. This matches **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value**: the RPC returns an error where a valid class definition is expected, and the committed state diff hash for block N still encodes C as declared, creating a mismatch between the committed state root and the accessible storage state.

### Likelihood Explanation

Low-Medium. Requires:
1. `enable_casm_hash_migration = true` — default in v0.14.1+ (current production)
2. A class with a v1 compiled hash in state — all classes declared before v0.14.1 qualify
3. A transaction executing such a class to be included in a block — unprivileged
4. That block to be reverted during consensus — normal operation when a proposal is rejected

### Recommendation

Add a block-number guard to `delete_declared_classes`, mirroring the existing guard in `delete_declared_classes_block`:

```rust
fn delete_declared_classes<'env>(
    txn: &'env DbTransaction<'env, RW>,
    thin_state_diff: &ThinStateDiff,
    declared_classes_table: &'env DeclaredClassesTable<'env>,
    declared_classes_block_table: &'env DeclaredClassesBlockTable<'env>,
    file_handlers: &FileHandlers<RW>,
    block_number: BlockNumber,
) -> StorageResult<IndexMap<ClassHash, SierraContractClass>> {
    let mut deleted_data = IndexMap::new();
    for class_hash in thin_state_diff.class_hash_to_compiled_class_hash.keys() {
        // Only delete if this block is the one that first declared the class.
        let Some(declared_block) = declared_classes_block_table.get(txn, class_hash)? else {
            continue;
        };
        if declared_block != block_number {
            continue; // Migration-only entry; class was declared in a different block.
        }
        let Some(location) = declared_classes_table.get(txn, class_hash)? else {
            continue;
        };
        deleted_data.insert(*class_hash, file_handlers.get_contract_class_unchecked(location)?);
        declared_classes_table.delete(txn, class_hash)?;
    }
    Ok(deleted_data)
}
```

### Proof of Concept

1. Sync a node to block N where class C was declared with a v1 compiled hash. Both `declared_classes_table[C]` and `declared_classes_block_table[C] = N` are populated by `append_classes` during central/P2P sync.
2. `enable_casm_hash_migration = true` (default in v0.14.1+).
3. Submit a transaction that executes class C. The batcher produces block M containing `class_hash_to_compiled_class_hash = {C: v2_hash}` (migration entry, not a new declaration).
4. During consensus, block M is rejected and `revert_state_diff(M)` is called.
5. `delete_declared_classes` deletes `declared_classes_table[C]` (the body written at block N).
6. `delete_declared_classes_block` does **not** delete `declared_classes_block_table[C]` because `N ≠ M`.
7. Call `get_class_definition_at(state_after_N, C)`:
   - Finds `declared_classes_block_table[C] = N` → class should be visible
   - Fails to find `declared_classes_table[C]` → `state_number.is_after(class_marker)` is false → returns `StorageError::DBInconsistency`
8. `starknet_getClass(C)` returns an error. The state is permanently inconsistent: the committed state diff hash for block N encodes C as declared, but the storage no longer holds its body.

fn delete_declared_classes_block<'env>(
    txn: &'env DbTransaction<'env, RW>,
    thin_state_diff: &ThinStateDiff,
    declared_classes_block_table: &'env DeclaredClassesBlockTable<'env>,
    block_number: BlockNumber,
) -> StorageResult<Vec<ClassHash>> {
    let mut deleted_data = Vec::new();
    for class_hash in thin_state_diff.class_hash_to_compiled_class_hash.keys() {
        let class_block_entry =
            declared_classes_block_table.get(txn, class_hash)?.ok_or_else(|| {
                StorageError::DBInconsistency {
                    msg: format!(
                        "Attempting to revert declaration of class {class_hash} but it doesn't \
                         exist in the DB"
                    ),
                }
            })?;
        if class_block_entry == block_number {
            declared_classes_block_table.delete(txn, class_hash)?;
            deleted_data.push(*class_hash);
        }
    }
    Ok(deleted_data)
}
