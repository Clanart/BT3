### Title
ERC1155 `safeTransferFrom` Callback to Recipient Contract Causes Irrecoverable Fund Lock in `finTransfer` — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

In `OmniBridge.finTransfer`, when releasing ERC1155 tokens to a contract recipient, the ERC1155 standard's `safeTransferFrom` triggers an `onERC1155Received` callback on the recipient. If the recipient contract reverts in that callback (or does not implement `IERC1155Receiver`), the entire `finTransfer` transaction reverts. Because there is no admin recovery path and no fallback delivery mechanism, the source-chain funds become permanently irrecoverable.

---

### Finding Description

`finTransfer` in `OmniBridge.sol` is the permissionless settlement function that releases tokens to the bridge recipient on EVM. For ERC1155 tokens it executes:

```solidity
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,   // attacker-controlled contract
    multiToken.tokenId,
    payload.amount,
    ""
);
```

Per the ERC1155 standard, `safeTransferFrom` calls `onERC1155Received` on `payload.recipient` when it is a contract. If that call reverts — because the contract does not implement `IERC1155Receiver`, returns the wrong selector, or deliberately reverts — the entire `finTransfer` transaction reverts.

The nonce guard is written *before* the external call:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287
// ... signature verification ...
IERC1155(...).safeTransferFrom(..., payload.recipient, ...);  // line 324
```

Because the whole transaction reverts on a failed transfer, `completedTransfers` is also rolled back. The nonce is never consumed, so the transfer can be retried — but every retry will revert for the same reason. The transfer is permanently unfinalisable.

The same mechanism applies to native ETH transfers:

```solidity
(bool success, ) = payload.recipient.call{value: payload.amount}("");
if (!success) revert FailedToSendEther();   // line 322
```

A contract recipient with no `receive()` function, or one that explicitly reverts, produces the same permanent lock.

Inspecting the full contract confirms there is **no admin escape hatch**: the only privileged functions are `pause`, `pauseAll`, `upgradeToken`, and `setNearBridgeDerivedAddress`. None can redirect a stuck transfer, force-consume a nonce, or return funds to the sender. The `evm/SECURITY.md` does not list this as a known issue.

---

### Impact Explanation

When a user bridges ERC1155 tokens from EVM to NEAR and back, the round-trip is:

1. `initTransfer1155` on EVM → ERC1155 tokens locked in the bridge contract.
2. NEAR bridge finalises → wrapped tokens minted on NEAR.
3. User initiates return transfer on NEAR, specifying a malicious EVM contract as recipient.
4. MPC signs the `TransferMessagePayload` with that recipient.
5. Every call to `finTransfer` on EVM reverts because `onERC1155Received` on the recipient reverts.
6. The ERC1155 tokens remain locked in `OmniBridge` forever with no recovery path.

Impact: **permanent, irrecoverable lock of bridged ERC1155 assets (or native ETH) in the EVM bridge contract**, matching the allowed impact class "Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."

---

### Likelihood Explanation

The recipient address is freely chosen by the user when calling `initTransfer` on the source chain. Realistic triggering scenarios:

- A user specifies a multisig wallet, DAO treasury, or smart-contract wallet that does not implement `IERC1155Receiver` — a common situation since most contract wallets predate or ignore ERC1155.
- A user specifies a contract that conditionally reverts in `onERC1155Received` (e.g., based on token ID or amount), causing selective lock.
- A user deliberately self-inflicts the lock (e.g., to grief a relayer who pre-paid fees).

No privileged access is required. Any bridge user initiating a transfer from any source chain to EVM can trigger this path.

---

### Recommendation

1. **Pull-over-push delivery**: Instead of pushing tokens directly to the recipient in `finTransfer`, store the claimable balance in a mapping (`claimable[recipient][token] += amount`) and let the recipient pull via a separate `claim()` function. This eliminates the revert-on-callback risk entirely.

2. **Try/catch wrapper**: Wrap the `safeTransferFrom` and ETH `.call` in a `try/catch`. On failure, store the amount in a claimable mapping rather than reverting, and emit a `TransferFailed` event so relayers and users can observe the stuck state.

3. **Recipient validation**: Before accepting a contract address as recipient, verify it returns the correct `onERC1155Received` selector (e.g., via a static call), and reject the `initTransfer` on the source chain if it does not.

### Proof of Concept

```solidity
// MaliciousReceiver.sol — deployed by attacker on EVM
contract MaliciousReceiver {
    function onERC1155Received(
        address, address, uint256, uint256, bytes calldata
    ) external pure returns (bytes4) {
        revert("blocked");   // always revert
    }
    function supportsInterface(bytes4) external pure returns (bool) {
        return true;
    }
}
```

**Attack steps:**

1. Attacker calls `initTransfer1155(erc1155Token, tokenId, amount, 0, 0, "near:attacker.near", "")` on EVM — ERC1155 tokens are locked in `OmniBridge`.
2. NEAR bridge finalises; attacker receives wrapped tokens on NEAR.
3. Attacker calls NEAR bridge `initTransfer` with `recipient = "eth:0x<MaliciousReceiver>"`.
4. MPC signs `TransferMessagePayload{recipient: MaliciousReceiver, tokenAddress: deterministicToken, ...}`.
5. Anyone calls `OmniBridge.finTransfer(sig, payload)`:
   - Line 287: `completedTransfers[nonce] = true`
   - Line 324: `IERC1155.safeTransferFrom(bridge, MaliciousReceiver, ...)` → `onERC1155Received` reverts → entire tx reverts → nonce rolled back.
6. Step 5 can be repeated indefinitely; it always reverts.
7. The original ERC1155 tokens remain locked in `OmniBridge` with no recovery path.
