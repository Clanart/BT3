### Title
Child `ChunkExtra` Inherits Parent's `proposed_split` Across Resharding Boundary, Causing `InvalidChunkHeaderShardSplit` on First Child Chunk — (`File: chain/chain/src/resharding/manager.rs`)

### Summary

When a shard is split via dynamic resharding, `ReshardingManager::process_memtrie_resharding_storage_update` creates child `ChunkExtra` records by cloning the parent's `ChunkExtra` and updating only `state_root` and `congestion_info`. The `proposed_split` field — which encodes a pending trie-split proposal — is **not reset to `None`** in the child. The first chunk produced for each child shard in epoch N+2 will therefore carry a stale, parent-epoch `proposed_split` value in its `ChunkExtra`. When that chunk header arrives and `validate_chunk_with_chunk_extra_and_receipts_root` compares `prev_chunk_extra.proposed_split()` against `chunk_header.proposed_split()`, the mismatch causes an `InvalidChunkHeaderShardSplit` error, permanently blocking chunk acceptance for both child shards.

### Finding Description

**Root cause — `process_memtrie_resharding_storage_update` in `chain/chain/src/resharding/manager.rs`:**

```rust
// line 258-260
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// proposed_split is NOT cleared — it inherits the parent's value
```

The parent's `ChunkExtra` at the resharding block may contain a non-`None` `proposed_split` (a `TrieSplit` value) if the parent shard proposed a split near the epoch boundary. This value is cloned verbatim into both child `ChunkExtra` records and committed to the store.

**Validation path that rejects the first child chunk:**

When the first new chunk for a child shard is produced in epoch N+2, the chunk producer computes `proposed_split = None` (the child shard is brand-new and has not yet crossed any memory threshold). The chunk header therefore carries `proposed_split = None`. However, `validate_chunk_with_chunk_extra_and_receipts_root` reads `prev_chunk_extra` — which is the child's `ChunkExtra` written during resharding — and finds `proposed_split = Some(TrieSplit { … })`. The comparison at line 176 fails:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
```

This error is returned from `validate_chunk_with_chunk_extra` (called in `Chain::create_shard_update_job` at line 3455) and from `validate_chunk_with_chunk_extra_and_roots` (called in `ChunkValidationActor::start_validating_chunk` at line 439). Both paths reject the chunk.

**The existing `min_epochs_between_resharding > 0` guard does not protect against this.** That guard prevents a *second* resharding from being scheduled back-to-back; it does not prevent the child's inherited `proposed_split` from being non-`None` at the moment of the first split.

**Acknowledged in the codebase:** The TODO comment at line 255 of `manager.rs` explicitly notes that not all `ChunkExtra` fields are set correctly during resharding:

```rust
// TODO(resharding): set all fields of `ChunkExtra`. Consider stronger
// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.
```

The dynamic resharding wiki also lists this as an open issue: *"The resharding manager doesn't set all `ChunkExtra` fields (notably the new `proposed_split` field)."*

### Impact Explanation

Both child shards produced by a dynamic resharding event are permanently unable to accept their first new chunk in epoch N+2. Every chunk producer for those shards will produce a chunk header with `proposed_split = None`, which will be rejected by every validating node with `InvalidChunkHeaderShardSplit`. The child shards stall at their first block, halting all transaction processing and state progression for those shards. Because the stale `proposed_split` is committed to the DB and there is no recovery path that resets it, the condition is permanent until a node restart with a patched binary.

**Severity: Critical** — a dynamic resharding event (which is a normal, automatic protocol operation) renders both child shards permanently non-functional.

### Likelihood Explanation

The trigger is automatic: dynamic resharding fires whenever a shard's trie memory usage exceeds `memory_usage_threshold` near an epoch boundary. The `proposed_split` field is set in `ChunkExtra` whenever `compute_proposed_split` returns `Some(...)`, which happens at the last few blocks of any epoch where the threshold is crossed. Because the parent shard is the one that is large enough to be split, it is highly likely that its final `ChunkExtra` carries a non-`None` `proposed_split`. The bug therefore fires on every dynamic resharding event where the parent shard proposed a split (i.e., the common case — the split was selected *because* the parent proposed it).

### Recommendation

In `process_memtrie_resharding_storage_update`, after cloning the parent `ChunkExtra` into `child_chunk_extra`, explicitly reset `proposed_split` to `None`:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// Reset proposed_split: child shards start fresh with no pending split proposal.
*child_chunk_extra.proposed_split_mut() = None;
```

This requires adding a `proposed_split_mut()` accessor to `ChunkExtra` (analogous to the existing `state_root_mut()` and `congestion_info_mut()`). Alternatively, construct the child `ChunkExtra` from scratch using `ChunkExtra::new(...)` with `proposed_split: None` rather than cloning the parent.

Additionally, the `min_epochs_between_resharding` cooldown should be verified to be at least 1 (it already is, per the existing documentation), but the fix above is the necessary correctness change.

### Proof of Concept

1. Configure a node with `DynamicResharding` enabled and a low `memory_usage_threshold`.
2. Deploy contracts to a single shard until its trie memory exceeds the threshold.
3. Near the end of epoch N, the runtime calls `compute_proposed_split` and stores `Some(TrieSplit{...})` in the parent shard's `ChunkExtra`.
4. The last block of epoch N embeds `shard_split` in its header; `finalize_epoch` creates epoch N+2's layout with two child shards.
5. At the epoch N+1 → N+2 boundary, `split_shard` → `process_memtrie_resharding_storage_update` writes child `ChunkExtra` records that clone the parent's `proposed_split = Some(TrieSplit{...})`.
6. In epoch N+2, the first chunk producer for either child shard produces a chunk header with `proposed_split = None` (correct — the child is new and below threshold).
7. `validate_chunk_with_chunk_extra_and_receipts_root` reads `prev_chunk_extra.proposed_split() = Some(TrieSplit{...})` and `chunk_header.proposed_split() = None`; the comparison fails and returns `Err(InvalidChunkHeaderShardSplit(...))`.

/// Validate that all next chunk information matches previous chunk extra.
pub fn validate_chunk_with_chunk_extra_and_receipts_root(
    prev_chunk_extra: &ChunkExtra,
    chunk_header: &ShardChunkHeader,
    outgoing_receipts_root: &CryptoHash,
) -> Result<(), Error> {
    if *prev_chunk_extra.state_root() != chunk_header.prev_state_root() {
        return Err(Error::InvalidStateRoot);
    }

    if prev_chunk_extra.outcome_root() != chunk_header.prev_outcome_root() {
        return Err(Error::InvalidOutcomesProof);
    }

    let chunk_extra_proposals = prev_chunk_extra.validator_proposals();
    let chunk_header_proposals = chunk_header.prev_validator_proposals();
    if chunk_header_proposals.len() != chunk_extra_proposals.len()
        || !chunk_extra_proposals.eq(chunk_header_proposals)
    {
        return Err(Error::InvalidValidatorProposals);
    }

    if prev_chunk_extra.gas_limit() != chunk_header.gas_limit() {
        return Err(Error::InvalidGasLimit);
    }

    if prev_chunk_extra.gas_used() != chunk_header.prev_gas_used() {
        return Err(Error::InvalidGasUsed);
    }

    if prev_chunk_extra.balance_burnt() != chunk_header.prev_balance_burnt() {
        return Err(Error::InvalidBalanceBurnt);
    }

    if outgoing_receipts_root != chunk_header.prev_outgoing_receipts_root() {
        return Err(Error::InvalidReceiptsProof);
    }

    validate_congestion_info(prev_chunk_extra.congestion_info(), chunk_header.congestion_info())?;
    validate_bandwidth_requests(
        prev_chunk_extra.bandwidth_requests(),
        chunk_header.bandwidth_requests(),
    )?;

    if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
        DYNAMIC_RESHARDING_VALIDATION_FAILURES.with_label_values(&["chunk_header"]).inc();
        return Err(Error::InvalidChunkHeaderShardSplit(format!(
            "header has {:?}, expected {:?} (prev block hash: {:?} height created: {:?})",
            chunk_header.proposed_split(),
            prev_chunk_extra.proposed_split(),
            chunk_header.prev_block_hash(),
            chunk_header.height_created(),
        )));
    }

    Ok(())
}
