Audit Report

## Title
Unattributed Router ETH Balance Consumed by Any WETH Swap, Enabling Theft of Stranded User Funds — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` function in `PeripheryPayments.sol` uses the router's entire native ETH balance to settle WETH swap payments without any per-caller attribution. ETH left on the router from a prior `multicall{value: X}` call where the user omitted `refundETH()` can be silently consumed by any subsequent WETH swap initiated by an attacker, who receives the full swap output while paying nothing.

## Finding Description
The `pay()` function contains a WETH-specific branch that checks `address(this).balance` before pulling from the declared `payer`:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // payer never touched
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

In the `nativeBalance >= value` branch, `safeTransferFrom(payer, ...)` is never called; the router deposits its own ETH and transfers the resulting WETH to the pool. The `payer` stored in transient context is completely irrelevant to the actual payment.

ETH arrives on the router via `multicall{value: X}(...)`, which is `payable`: [2](#0-1) 

The `receive()` guard only blocks direct ETH transfers from non-WETH addresses and does not block ETH attached to a `multicall` call: [3](#0-2) 

**Exploit path:**

1. Victim calls `multicall{value: 1 ETH}([exactInputSingle(WETH→tokenOut, amountIn=0.5 ETH)])` without appending `refundETH()`. The swap consumes 0.5 ETH; 0.5 ETH remains stranded on the router.

2. Attacker calls `exactInputSingle(WETH→tokenOut, amountIn=0.5 ETH)` with no ETH sent and no WETH allowance. `exactInputSingle` stores `msg.sender` (attacker) as payer in transient storage and calls the pool: [4](#0-3) 

3. The pool calls back `metricOmmSwapCallback`, which dispatches to `_justPayCallback`: [5](#0-4) 

4. `pay(WETH, attacker, pool, 0.5e18)` is called. `nativeBalance (0.5 ETH) >= value (0.5e18)`, so the router deposits the victim's 0.5 ETH as WETH and transfers it to the pool. The attacker's address stored as `payer` is never used.

5. The attacker receives `tokenOut` output and pays nothing. The victim's 0.5 ETH is permanently lost.

The partial-subsidy branch (`0 < nativeBalance < value`) similarly allows the attacker to pay only `value − nativeBalance`, with the remainder subsidized by the victim's stranded ETH.

## Impact Explanation
Direct loss of user principal. The victim's stranded ETH is consumed by the attacker's swap with no recourse. The attacker receives the full swap output without providing any input tokens or ETH. Loss is bounded only by the amount of ETH stranded on the router, which can be arbitrarily large. This meets the Critical/High threshold for direct loss of user funds.

## Likelihood Explanation
The `multicall` pattern is the standard composition mechanism for permit + swap + refund flows. Omitting `refundETH()` is a common integration mistake. An attacker can monitor the router's ETH balance on-chain (via `eth_getBalance`) and immediately follow any transaction that leaves ETH behind. No special permissions, approvals, or privileged access are required — a plain `exactInputSingle` call with `tokenIn = WETH` suffices. The attack is repeatable and fully automated.

## Recommendation
Track ETH attribution per-caller using transient storage (analogous to how the callback context tracks `payer` and `token`). In `pay()`, only allow ETH deposited by the current `msg.sender` (recorded at `multicall` entry) to be used for their own WETH payment. Alternatively, remove the native-ETH-first logic from `pay()` entirely and require callers to explicitly wrap ETH via a dedicated `wrapETH` multicall step before invoking the router, making ETH handling explicit and attributable.

## Proof of Concept
```solidity
// 1. Victim leaves 0.5 ETH stranded on router
router.multicall{value: 1 ether}([
    abi.encodeCall(router.exactInputSingle, (ExactInputSingleParams({
        tokenIn: WETH, tokenOut: tokenOut, pool: pool,
        recipient: victim, amountIn: 0.5 ether, amountOutMinimum: 0,
        zeroForOne: true, priceLimitX64: 0, deadline: block.timestamp, extensionData: ""
    })))
    // refundETH() intentionally omitted
]);
// router.balance == 0.5 ether

// 2. Attacker calls with no ETH and no WETH allowance
router.exactInputSingle(ExactInputSingleParams({
    tokenIn: WETH, tokenOut: tokenOut, pool: pool,
    recipient: attacker, amountIn: 0.5 ether, amountOutMinimum: 0,
    zeroForOne: true, priceLimitX64: 0, deadline: block.timestamp, extensionData: ""
}));
// Attacker receives tokenOut; victim's 0.5 ETH is consumed; attacker paid nothing
```
