Audit Report

## Title
`SwapAllowlistExtension` checks the router address instead of the economic actor, allowing any user to bypass per-pool swap access control — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

## Summary

`MetricOmmPool.swap()` passes `msg.sender` (the calling router) as the `sender` argument to `_beforeSwap`. `SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[pool][sender]`, which resolves to `allowedSwapper[pool][router]` — never the original user. When a pool admin allowlists the router to permit router-mediated swaps, every unprivileged user can bypass the allowlist. Conversely, when individual users are allowlisted directly, they cannot swap through the router at all.

## Finding Description

`MetricOmmPool.swap()` passes `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`SwapAllowlistExtension.beforeSwap` checks: [2](#0-1) 

Here `msg.sender` is the pool and `sender` is whoever called `pool.swap()` — i.e., the router. The check resolves to `allowedSwapper[pool][router]`.

`MetricOmmSimpleRouter.exactInputSingle` stores the original `msg.sender` only in transient callback context for payment, and calls `pool.swap()` directly without forwarding the user identity: [3](#0-2) 

The original user address is never forwarded to the pool or to extension hooks. The same pattern applies to `exactOutputSingle` and `exactInput`. [4](#0-3) 

**Scenario A — Allowlist bypass:** Pool admin allowlists the router (the natural configuration to permit router-mediated swaps). Any unprivileged user calls `exactInputSingle`. The extension sees `sender=router`, which is allowlisted → swap proceeds. The per-user allowlist is completely bypassed.

**Scenario B — Legitimate users locked out:** Pool admin allowlists individual user addresses. Those users call `exactInputSingle`. The extension sees `sender=router`, which is not allowlisted → reverts with `NotAllowedToSwap`. Allowlisted users cannot use the standard periphery path.

No existing guard in the pool or extension compensates for this: the extension's `isAllowedToSwap` view and `setAllowedToSwap` admin function both operate on the same `allowedSwapper` mapping keyed by the caller of `pool.swap()`, not the economic actor. [5](#0-4) 

## Impact Explanation

`SwapAllowlistExtension` is documented as a production access-control primitive that "Gates `swap` by swapper address, per pool." [6](#0-5) 

Under Scenario A, any unprivileged address executes swaps on a pool the admin intended to restrict — a direct violation of the access-control invariant and broken core pool functionality. Under Scenario B, explicitly allowlisted users are locked out of the standard periphery swap path, rendering the extension unusable with the router. Both outcomes constitute broken core pool functionality.

## Likelihood Explanation

The router is the primary public swap entrypoint. Any pool deployer who configures `SwapAllowlistExtension` and uses the router will encounter this bug. No special attacker capability is required — only calling `exactInputSingle` through the public router. The bug is triggered by the standard, expected usage pattern.

## Recommendation

The pool should forward the original economic actor's identity to extension hooks rather than `msg.sender`. The cleanest fix is to add an optional `swapper` parameter to `pool.swap()` that defaults to `msg.sender` but can be explicitly set by trusted periphery contracts, and pass that value to `_beforeSwap` instead of `msg.sender`. Alternatively, `SwapAllowlistExtension.beforeSwap` could accept an ABI-encoded `address` in `extensionData` as the identity to check, with the router forwarding `msg.sender` there — but this requires router cooperation and is fragile. The pool-level fix is preferred.

## Proof of Concept

1. Deploy `MetricOmmPool` with `SwapAllowlistExtension` in `beforeSwapOrder`.
2. Pool admin calls `setAllowedToSwap(pool, routerAddress, true)` — allowlisting the router.
3. Unprivileged `attacker` (not individually allowlisted) calls `router.exactInputSingle(...)`.
4. Pool calls `_beforeSwap(msg.sender=router, ...)` → extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
5. `attacker` successfully swaps on a pool they were never individually authorized to use.

To demonstrate Scenario B:
1. Pool admin calls `setAllowedToSwap(pool, alice, true)`.
2. Alice calls `router.exactInputSingle(...)`.
3. Extension checks `allowedSwapper[pool][router]` → `false` → reverts with `NotAllowedToSwap`.
4. Alice cannot use the router despite being explicitly allowlisted.
