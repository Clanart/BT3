### Title
`execute_meta_tx_v0` Signature Replay Within a Single Transaction Due to Missing Nonce in Hash and No Consumption of Used Meta-Tx - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall handler in the StarkNet OS does not include a nonce in the meta-transaction hash and does not consume or mark the meta-tx hash as used after execution. Because `check_and_increment_nonce` explicitly skips version-0 transactions, a malicious relayer can replay the same meta-tx signature multiple times within a single outer transaction, causing the victim's `__execute__` to run repeatedly and resulting in direct loss of funds.

---

### Finding Description

`execute_meta_tx_v0` (in `syscall_impls.cairo`) constructs a version-0 execution context for a target contract's `__execute__` entry point. The meta-transaction hash is computed by `compute_meta_tx_v0_hash` (in `transaction_hash.cairo`):

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,          // ← no nonce included
        additional_data=cast(0, felt*),
    );
``` [1](#0-0) 

The hash is purely a function of `(contract_address, selector, calldata, chain_id)`. No nonce is committed to.

The new `TxInfo` created for the meta-tx hardcodes `nonce=0` and `version=0`:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,
    ...
);
``` [2](#0-1) 

After execution, no state is updated to record that this meta-tx hash was used. The OS's `check_and_increment_nonce` explicitly returns early for version-0 transactions:

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
``` [3](#0-2) 

Similarly, `run_validate` skips `__validate__` for version-0 transactions:

```cairo
// Do not run "__validate__" for version 0.
if (tx_execution_info.tx_info.version == 0) {
    return ();
}
``` [4](#0-3) 

The `meta_tx_v0` syscall is reachable from any contract's execution via the syscall dispatch in `execute_syscalls.cairo`:

```cairo
assert selector = META_TX_V0_SELECTOR;
execute_meta_tx_v0(block_context=block_context, caller_execution_context=execution_context);
``` [5](#0-4) 

Because the meta-tx hash does not commit to a nonce and no consumed-hash tracking exists at the OS level, the same `(contract_address, calldata, signature)` tuple can be submitted to `meta_tx_v0` an arbitrary number of times within a single outer transaction.

---

### Impact Explanation

A malicious relayer who holds a victim's valid meta-tx signature can include multiple `meta_tx_v0` syscall invocations — each with the identical `contract_address`, `calldata`, and `signature` — inside a single outer invoke transaction. Each invocation causes the victim's `__execute__` to run with the same calldata (e.g., a token transfer to the attacker). Since the OS enforces no uniqueness constraint on meta-tx hashes and increments no nonce for version-0 contexts, every invocation succeeds. The result is **direct loss of funds** (Critical) proportional to the number of replays the attacker includes.

---

### Likelihood Explanation

The `meta_tx_v0` syscall is the standard mechanism for meta-transaction relaying on StarkNet. Any unprivileged outer transaction sender who acts as a relayer and holds a victim-signed meta-tx can perform this attack. No privileged access, leaked key, or network-level capability is required beyond possession of the victim's signature — which is the normal precondition for any relayer. The attack is self-contained within a single block and requires no coordination beyond crafting the outer transaction.

---

### Recommendation

1. **Include a nonce in the meta-tx hash**: Pass `additional_data_size=1` and `additional_data=&nonce` to `deprecated_get_transaction_hash` inside `compute_meta_tx_v0_hash`, where `nonce` is a per-account counter stored in `contract_state_changes` and incremented after each successful meta-tx execution.
2. **Consume the meta-tx identifier**: After executing a meta-tx, record the used hash (or the consumed nonce) in the target contract's state so that duplicate submissions within the same or future transactions are rejected at the OS level.

---

### Proof of Concept

1. Victim signs a meta-tx: `H = hash(victim_address, __execute__, [transfer 100 tokens → attacker], chain_id)` and produces signature `σ`.
2. Attacker constructs an outer invoke transaction whose `__execute__` calls `meta_tx_v0` **K times**, each time supplying `(contract_address=victim, calldata=[transfer 100 tokens → attacker], signature=σ)`.
3. The OS computes the same `meta_tx_hash = H` for every call (no nonce in hash), creates a version-0 `TxInfo` with `nonce=0`, skips `check_and_increment_nonce` (version == 0 early return), and calls `victim.__execute__` K times.
4. Each call succeeds because `σ` is valid for `H` and no consumed-hash check exists.
5. Attacker receives `K × 100` tokens; victim loses `K × 100` tokens from a single signed authorization. [6](#0-5) [7](#0-6) [8](#0-7)

func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;
    }

    // Update contract_state_changes.
    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
