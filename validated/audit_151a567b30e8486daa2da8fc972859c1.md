## Full Trace

**Step 1 — EVM entry point: `coreReceiveWithData` + `ACTION_INIT_TRANSFER`**

A HyperCore user triggers `sendToEvmWithData` on HyperLiquid, which causes `_systemAddress` to call `coreReceiveWithData`. The `data` payload is user-controlled. For `ACTION_INIT_TRANSFER`, the contract ABI-decodes `(uint128 fee, string memory recipient, string memory message)` from the tail and passes `recipient` verbatim to `OmniBridge.initTransfer`: [1](#0-0) 

**Step 2 — `OmniBridge.initTransfer` burns tokens with no recipient validation**

`OmniBridge.initTransfer` accepts `string calldata recipient` and performs zero validation on it. It burns the tokens unconditionally (for a registered bridge token), then emits `InitTransfer` with the raw, unvalidated recipient string: [2](#0-1) 

The only checks are `fee >= amount` and pause state. No chain-prefix validation exists anywhere in this function.

**Step 3 — NEAR prover fails to parse the recipient**

`OmniAddress::from_str` is the canonical parser. For any unrecognized prefix it returns `Err("Chain {chain} is not supported")`: [3](#0-2) 

In the Wormhole path, the prover explicitly calls `.parse().map_err(stringify)?` on the recipient string, which propagates the error and causes `verify_vaa_callback` to return `Err(...)`: [4](#0-3) 

In the EVM light-client path, `parse_evm_proof` → `parse_evm_event` → `TryFromLog` similarly parses the recipient string into an `OmniAddress`, which fails identically.

**Step 4 — `fin_transfer_callback` panics; no settlement occurs**

When the prover returns an error, `fin_transfer_callback` panics with `BridgeError::InvalidProofMessage`: [5](#0-4) 

There is no retry path, no refund path, and no mechanism to recover the burned EVM tokens. The `OmniBridge.initTransfer` burn is irreversible.

**Step 5 — No refund path exists**

`OmniBridge.initTransfer` has no rollback. Once `BridgeToken.burn` executes and `InitTransfer` is emitted, the tokens are gone. The NEAR bridge has no "cancel pending transfer" function for EVM-originated burns.

---

## Verdict

The attack path is concrete and locally testable. However, the root cause is **not unique to `coreReceiveWithData`** — it is a general property of `OmniBridge.initTransfer`, which accepts any arbitrary `recipient` string with no chain-prefix validation. The `coreReceiveWithData` path is one entry point; a direct caller of `initTransfer` with `recipient = "unknownchain:addr"` has the identical outcome.

The question's framing is correct: the invariant that *every EVM-side burn must route to a supported destination chain* is violated. The EVM side enforces no such constraint, and the NEAR side cannot complete settlement for an unrecognized chain prefix.

### Title
Unvalidated `recipient` chain prefix in `OmniBridge.initTransfer` causes permanent irrecoverable burn — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

### Summary
`OmniBridge.initTransfer` accepts an arbitrary `string recipient` with no validation that the chain prefix is recognized by NEAR's `OmniAddress` parser. Any caller — including a HyperCore user via `coreReceiveWithData` — can supply an unrecognized prefix (e.g. `"unknownchain:0xdeadbeef"`), causing tokens to be burned on EVM and `InitTransfer` to be emitted, while every NEAR prover path fails to parse the recipient and `fin_transfer` can never succeed.

### Finding Description
`OmniBridge.initTransfer` burns bridge tokens and emits `InitTransfer` without validating the `recipient` string. The NEAR-side provers (both Wormhole and EVM light-client) call `OmniAddress::from_str` on the recipient field, which returns `Err("Chain {chain} is not supported")` for any unrecognized prefix. This error propagates through `verify_vaa_callback` / `parse_evm_proof`, causing `fin_transfer_callback` to panic. No settlement occurs on any chain, and there is no refund path for the burned EVM tokens.

The `coreReceiveWithData` path in `HlBridgeToken.sol` is a concrete unprivileged entry point: a HyperCore user controls the `data` payload (including the `recipient` string) passed through HyperLiquid's system address callback.

### Impact Explanation
Permanent, irrecoverable loss of user funds. Tokens are burned on EVM; no mint or unlock occurs on any destination chain. The transfer ID is never finalized, and no recovery mechanism exists.

### Likelihood Explanation
Any user who can call `OmniBridge.initTransfer` directly, or any HyperCore user who can trigger `coreReceiveWithData`, can cause this. No privileged access is required. The only prerequisite is holding bridge tokens.

### Recommendation
Validate the `recipient` string in `OmniBridge.initTransfer` before burning tokens. Either:
1. Parse and validate the chain prefix on-chain (maintain an allowlist of valid chain prefixes as a contract storage variable), or
2. Revert if the recipient string does not match a known prefix pattern before executing any token burn.

The same validation should be applied in `initTransfer1155`.

### Proof of Concept
1. Register a `HyperliquedBridgeToken` with `OmniBridge` (standard setup).
2. Seed the system-address pool via 3-arg `mint`.
3. As HyperCore user, trigger `coreReceiveWithData` with:
   ```
   data = 0x01 || abi.encode(uint128(0), "unknownchain:0xdeadbeef", "")
   ```
4. Assert: `InitTransfer` emitted, `totalSupply == 0` (tokens burned).
5. Submit the proof to NEAR `fin_transfer`. Assert: prover returns `Err("Chain unknownchain is not supported")`; `fin_transfer_callback` panics; no tokens minted on NEAR.
6. Confirm: no EVM refund path; funds permanently lost.
