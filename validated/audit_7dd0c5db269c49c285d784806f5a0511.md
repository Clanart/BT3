Looking at the `coreReceiveWithData` function in `HlBridgeToken.sol` and the recovery mechanisms in `BridgeToken.sol` and `OmniBridge.sol`.

The `recover` hits in `OmniBridge.sol` are `ECDSA.recover` calls (signature verification), not token-rescue functions. There is no sweep or emergency-withdrawal path anywhere in the contract hierarchy.

---

### Title
Missing `recipient != address(this)` Guard in `ACTION_TRANSFER` Permanently Locks Tokens — (`evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

### Summary
`coreReceiveWithData` with `ACTION_TRANSFER` blindly decodes the recipient address from user-supplied `data` and calls `_update(_systemAddress, recipient, amount)`. When a HyperCore user sets the recipient to `address(this)` (the token contract itself), the tokens are moved into the contract's own ERC-20 balance with no automatic release path, permanently locking the bridged assets.

### Finding Description
The `coreReceiveWithData` function is the HyperCore → HyperEVM relay entry point. Its only caller-level guard is:

```solidity
if (msg.sender != _systemAddress) revert NotSystemAddress();
``` [1](#0-0) 

The `data` payload is fully user-controlled: a HyperCore user encodes it in `sendToEvmWithData` and the HyperLiquid system address relays it verbatim. For `ACTION_TRANSFER`, the contract decodes the recipient without any validation:

```solidity
if (action == ACTION_TRANSFER) {
    address recipient = abi.decode(tail, (address));
    _update(_systemAddress, recipient, amount);
}
``` [2](#0-1) 

There is no `require(recipient != address(this))` guard. If the attacker encodes `address(token)` as the recipient, `_update` moves `amount` tokens from `_systemAddress` into the token contract's own balance.

Neither `HlBridgeToken` nor its parent `BridgeToken` contains a sweep, rescue, or emergency-withdrawal function: [3](#0-2) 

The only owner-callable functions on the token are `mint`, `burn`, `setMetadata`, and `upgradeToAndCall`. `burn(address(token), amount)` would destroy the tokens (not return them to the user), and `mint(user, amount)` would require the admin to manually identify the correct beneficiary from off-chain HyperCore data — there is no on-chain linkage between the `from` HyperCore address and any EVM address.

The `ACTION_INIT_TRANSFER` path intentionally moves tokens to `address(this)` as a transient step before immediately calling `initTransfer` (which burns from `msg.sender == address(this)`): [4](#0-3) 

No such follow-up burn exists for `ACTION_TRANSFER` when `recipient == address(this)`.

### Impact Explanation
Tokens transferred via `ACTION_TRANSFER` to `address(this)` accumulate in the token contract's ERC-20 balance. There is no on-chain path to release them back to any user. The `_systemAddress` pool is debited, so the HyperCore-side accounting drifts: the pool shrinks by `amount` but no EVM user receives the tokens. Recovery requires privileged admin intervention (burn + re-mint) with off-chain identification of the victim — which is not guaranteed and is not part of the protocol's normal operation. This constitutes an irrecoverable lock of bridged assets.

### Likelihood Explanation
Any HyperCore user can trigger this by calling `sendToEvmWithData` with `data = bytes([0x00]) + abi.encode(address(token))`. No special privilege, leaked key, or colluding party is required. The path is fully reachable from a single unprivileged HyperCore account. A user could do this accidentally (mistyping the recipient) or deliberately (griefing the pool accounting).

### Recommendation
Add a recipient sanity check in the `ACTION_TRANSFER` branch:

```solidity
if (action == ACTION_TRANSFER) {
    address recipient = abi.decode(tail, (address));
    require(recipient != address(this) && recipient != address(0), "Invalid recipient");
    _update(_systemAddress, recipient, amount);
}
``` [2](#0-1) 

### Proof of Concept
1. Deploy `HyperliquedBridgeToken` with a known `_systemAddress`.
2. Seed `_systemAddress` with a token balance (simulating the HyperCore pool via the 3-arg `mint`).
3. Call `coreReceiveWithData` from `_systemAddress` with:
   - `action = 0x00` (ACTION_TRANSFER)
   - `tail = abi.encode(address(token))`
   - `amount = N`
4. Assert `token.balanceOf(address(token)) == N`.
5. Confirm no function exists on the contract that can transfer those tokens to a user without admin `burn` + `mint`. [5](#0-4)

    function coreReceiveWithData(
        address from,
        bytes32 /*destinationRecipient*/,
        uint32 /*destinationChainId*/,
        uint256 amount,
        uint64 /*coreNonce*/,
        bytes calldata data
    ) external override {
        if (msg.sender != _systemAddress) revert NotSystemAddress();
        if (data.length == 0) revert EmptyActionData();

        uint8 action = uint8(data[0]);
        bytes calldata tail = data[1:];

        if (action == ACTION_TRANSFER) {
            address recipient = abi.decode(tail, (address));
            _update(_systemAddress, recipient, amount);
        } else if (action == ACTION_INIT_TRANSFER) {
            (uint128 fee, string memory recipient, string memory message) = abi
                .decode(tail, (uint128, string, string));
            uint128 amount128 = amount.toUint128();
            _update(_systemAddress, address(this), amount);
            IOmniBridgeInitTransfer(owner()).initTransfer(
                address(this),
                amount128,
                fee,
                0,
                recipient,
                message
            );
        } else {
            revert UnknownAction(action);
        }

        emit CoreReceived(from, action, amount, data);
    }
```
