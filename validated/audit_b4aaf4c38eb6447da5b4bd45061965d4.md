### Title
`initTransfer1155` Locks ERC1155 Tokens Without Updating `multiTokens` Mapping, Causing Irrecoverable Fund Lock - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
`initTransfer1155` in `OmniBridge.sol` locks ERC1155 tokens in the bridge contract but never writes to the `multiTokens` mapping. The `finTransfer` function relies exclusively on `multiTokens` to identify the ERC1155 contract and token ID needed to release those tokens. If `logMetadata1155` was not called before `initTransfer1155`, the bridge holds ERC1155 tokens it cannot release, and the NEAR side rejects the corresponding `InitTransfer` event because the token is unregistered — permanently locking the user's funds.

### Finding Description

The bridge maintains two separate tracking systems for ERC1155 tokens:

1. **Actual ERC1155 custody** — the real token balance held by the bridge contract, updated by `safeTransferFrom` inside `initTransfer1155`.
2. **`multiTokens` mapping** — the bridge's internal record of `(tokenAddress, tokenId)` for each deterministic address, updated only by `logMetadata1155`.

`initTransfer1155` updates system (1) but never touches system (2): [1](#0-0) 

There is no `require` that `multiTokens[deterministicToken].tokenAddress != address(0)` before locking tokens. The function computes `deterministicToken`, transfers ERC1155 tokens into the bridge, and emits `InitTransfer` — all without checking or updating `multiTokens`.

`finTransfer` then reads `multiTokens[payload.tokenAddress]` to decide how to release tokens: [2](#0-1) 

If `multiTokens[deterministicToken].tokenAddress == address(0)` (never set), the ERC1155 branch is skipped entirely. The function falls through to `IERC20(deterministicToken).safeTransfer(...)`, which calls a non-contract address and reverts — or, worse, if `deterministicToken` collides with a registered bridge token, mints ERC20 tokens to the recipient while leaving the ERC1155 tokens permanently locked.

`logMetadata1155` is the only function that populates `multiTokens`: [3](#0-2) 

It is permissionless and entirely optional — `initTransfer1155` imposes no ordering constraint on it.

The existing test suite confirms this gap: the "validates ERC1155 receiver hooks" test calls `initTransfer1155` directly without any prior `logMetadata1155`, and the tokens are successfully locked with no mapping entry created: [4](#0-3) 

On the NEAR side, `fin_transfer_callback` requires the token to be registered (have decimals) before it can process an `InitTransfer` event: [5](#0-4) 

`logMetadata1155` emits a `LogMetadata` event that the NEAR indexer uses to register the token. If `initTransfer1155` is called without a prior `logMetadata1155`, the `InitTransfer` event arrives on NEAR before the token is registered, causing NEAR to panic with `TokenDecimalsNotFound` and reject the transfer. The ERC1155 tokens remain locked in the EVM bridge with no corresponding NEAR-side mint and no recovery path.

### Impact Explanation

A user who calls `initTransfer1155` without first calling `logMetadata1155` will have their ERC1155 tokens permanently locked in the bridge:

- The ERC1155 tokens are transferred into the bridge (system 1 updated).
- `multiTokens` is never set (system 2 not updated).
- The NEAR side rejects the `InitTransfer` event (token unregistered) — no NEAR tokens are minted.
- `finTransfer` cannot be called because NEAR never signed a release message.
- There is no admin rescue function for stuck ERC1155 tokens.

Even if `logMetadata1155` is called after the fact, the NEAR side has already processed (and rejected) the `InitTransfer` event in block order; the event cannot be replayed. The user's ERC1155 tokens are irrecoverably locked.

This matches the allowed impact: **Critical — permanent freezing / irrecoverable lock of user funds in bridge flows.**

### Likelihood Explanation

`initTransfer1155` is a public, permissionless function callable by any token holder. The protocol documentation and contract comments do not enforce or warn about the required ordering. Any user unfamiliar with the two-step registration flow (`logMetadata1155` → `initTransfer1155`) will trigger this condition. The existing test suite itself demonstrates the gap by calling `initTransfer1155` without `logMetadata1155`.

### Recommendation

Add a guard in `initTransfer1155` that requires the `multiTokens` mapping to be populated before locking tokens:

```solidity
function initTransfer1155(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);
    
    // Require logMetadata1155 to have been called first
    require(
        multiTokens[deterministicToken].tokenAddress != address(0),
        "ERR_ERC1155_NOT_REGISTERED"
    );
    // ... rest of function
}
```

Alternatively, inline the `logMetadata1155` logic inside `initTransfer1155` so registration and locking are atomic.

### Proof of Concept

1. Deploy `OmniBridge` and an ERC1155 token.
2. Mint 10 tokens of `tokenId = 7` to Alice.
3. Alice approves the bridge and calls `initTransfer1155(erc1155, 7, 5, 0, 0, "alice.near", "")` — **without** calling `logMetadata1155` first.
4. The bridge now holds 5 ERC1155 tokens. `multiTokens[deterministicToken]` is `{address(0), 0}`.
5. The NEAR side receives the `InitTransfer` event but panics with `ERR_TOKEN_DECIMALS_NOT_FOUND` because the token was never registered via `LogMetadata`.
6. No NEAR tokens are minted; no MPC signature for `finTransfer` is ever produced.
7. Alice's 5 ERC1155 tokens are permanently locked in the bridge with no recovery path. [6](#0-5) [7](#0-6) [8](#0-7)

        } else {
