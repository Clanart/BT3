### Title
Missing Transaction Hash Validation Against Commitment in P2P Sync — (`crates/apollo_p2p_sync/src/client/transaction.rs`)

### Summary

`TransactionStreamFactory::parse_data_for_block` accepts `transaction_hash` values from an untrusted P2P peer and stores them verbatim without verifying them against the header's `transaction_commitment`. A malicious peer can inject arbitrary felts as transaction hashes, corrupting the node's storage and causing RPC endpoints to serve authoritative-looking wrong transaction hash values.

### Finding Description

In `parse_data_for_block`, the code explicitly acknowledges the missing guard:

```rust
// TODO(eitan): Validate transaction hash from untrusted sources
block_body.transaction_hashes.push(transaction_hash);
``` [1](#0-0) 

The `transaction_hash` field of each `FullTransaction` received from the network is pushed directly into `block_body.transaction_hashes` with no check against:
1. The cryptographic hash of the transaction body (via `validate_transaction_hash`), and
2. The `transaction_commitment` stored in the already-accepted block header (the Poseidon Patricia tree root over all transaction leaf hashes).

The `target_transaction_len` is read from the stored header's `n_transactions` field, so the count is bounded, but the *values* are fully attacker-controlled. [2](#0-1) 

`write_to_storage` then calls `append_body` unconditionally:

```rust
storage_writer.begin_rw_txn()?.append_body(self.1, self.0)?.commit()?;
``` [3](#0-2) 

Inside `write_transactions`, the attacker-supplied hash is written to two tables with no further validation:

```rust
transaction_hash_to_idx_table.insert(txn, tx_hash, &transaction_index)?;
transaction_metadata_table.append(txn, &transaction_index,
    &TransactionMetadata { tx_location, tx_output_location, tx_hash: *tx_hash })?;
``` [4](#0-3) 

The `validate_transaction_hash` function exists in the codebase but is never invoked on the P2P sync path: [5](#0-4) 

The `transaction_commitment` in the header (a Poseidon Patricia tree root over `H(tx_hash, tx_signature)` leaves) is stored but never compared against the hashes being written: [6](#0-5) 

### Impact Explanation

After a successful attack:

- `get_transaction_hash_by_idx(TransactionIndex(block_number, offset))` returns the attacker-chosen felt instead of the real hash. RPC `get_transaction_by_block_id_and_index` serves this wrong value as authoritative. [7](#0-6) 

- `get_transaction_idx_by_hash(real_hash)` returns `None` (TRANSACTION_HASH_NOT_FOUND), making the real transaction unfindable by hash. [8](#0-7) 

- The stored `transaction_hash_to_idx_table` maps the fake hash to the transaction index, so lookups by the fake hash succeed and return the wrong transaction body.

- The commitment invariant is permanently violated: `calculate_transaction_commitment` over the stored hashes will not equal the header's `transaction_commitment`. [9](#0-8) 

### Likelihood Explanation

Any peer that the syncing node connects to in P2P mode can trigger this. No special privileges are required — the node actively queries peers for `FullTransaction` data and accepts whatever `transaction_hash` field is returned. The `BLOCK_NUMBER_LIMIT: BlockNumberLimit::HeaderMarker` constraint only limits which blocks are requested, not the content of responses. [10](#0-9) 

### Recommendation

Before pushing `transaction_hash` into `block_body.transaction_hashes`, validate it in one of two ways (or both):

1. **Per-transaction hash check**: Call `validate_transaction_hash(transaction, block_number, chain_id, transaction_hash, &TransactionOptions::default())` and reject the peer if it returns `false`.

2. **Commitment check after collecting all transactions**: After the `while` loop, compute `calculate_transaction_commitment::<Poseidon>` over the collected `(transaction_hash, transaction_signature)` pairs and compare the result against `header.transaction_commitment`. Return `ParseDataError::BadPeer` on mismatch.

Option 2 is the stronger guard because it enforces the exact invariant the header commits to, and it is consistent with how `state_diff_commitment` is verified in the committer. [11](#0-10) 

### Proof of Concept

The attack path in production code:

1. Syncing node accepts a `SignedBlockHeader` for block N with `transaction_commitment = C` and `n_transactions = 1`.
2. Malicious peer responds to the transaction query with `FullTransaction { transaction: <valid tx>, transaction_output: <valid output>, transaction_hash: Felt::from(0xdeadbeef) }`.
3. `parse_data_for_block` pushes `Felt::from(0xdeadbeef)` into `block_body.transaction_hashes` (line 89) — no validation occurs.
4. `write_to_storage` calls `append_body(block_number, block_body)` which calls `write_transactions`, inserting `(Felt::from(0xdeadbeef) → TransactionIndex(N, 0))` into `transaction_hash_to_idx_table` and storing `tx_hash: Felt::from(0xdeadbeef)` in `transaction_metadata_table`.
5. `calculate_transaction_commitment::<Poseidon>` over `[Felt::from(0xdeadbeef)]` produces a root ≠ `C`, confirming the commitment invariant is violated in storage.
6. RPC `get_transaction_by_block_id_and_index(N, 0)` returns `transaction_hash: 0xdeadbeef` to any client.
