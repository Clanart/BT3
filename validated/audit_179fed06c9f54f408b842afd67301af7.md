### Title
`execute_deploy` Syscall Unconditionally Reports Success Regardless of Constructor Outcome, Enabling Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall in the StarkNet OS always writes `failure_flag=0` in the response header, regardless of whether the constructor execution succeeded or reverted. Any contract calling the `deploy` syscall always receives a success response. A calling contract that relies on this flag to gate subsequent fund transfers will proceed incorrectly — sending funds to an uninitialized or non-existent contract address — permanently freezing those funds.

---

### Finding Description

**Vulnerability class:** Fee/accounting bug — state-transition bypass. The OS emits an incorrect state-transition signal (success) after a failed sub-execution, causing the caller to act on false information.

**Root cause — exact location:**

In `execute_deploy` at line 539 of `syscall_impls.cairo`, after calling `deploy_contract`, the response header is unconditionally written with `failure_flag=0`:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` call returns only `(retdata_size, retdata)` — there is no `is_reverted` return value propagated to the caller:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}
``` [2](#0-1) 

This is structurally inconsistent with `contract_call_helper`, which correctly propagates `is_reverted` into the response header:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [3](#0-2) 

The calling contract receives `failure_flag=0` and a contract address in the `DeployResponse`, making it impossible to distinguish a successful deploy from a failed one at the OS level.

---

### Impact Explanation

A factory contract or DeFi protocol that calls `deploy` and then transfers funds to the returned contract address will always proceed with the transfer, even when the constructor reverted. If the constructor failed (e.g., due to out-of-gas, logic failure, or attacker-crafted calldata), the deployed contract is either uninitialized or non-existent at that address. Funds transferred via ERC20 to such an address are stored in the token contract's storage under that key, but there is no initialized contract logic at the address to recover them.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The attacker-controlled entry path is fully unprivileged:

1. An attacker declares a class whose constructor reverts under attacker-controlled conditions (e.g., when a specific storage slot is set, or when gas is below a threshold).
2. The attacker submits a transaction that causes a victim factory contract to call `deploy` with that class hash and insufficient gas for the constructor.
3. The constructor reverts; the OS writes `failure_flag=0`.
4. The victim contract proceeds to transfer funds to the returned address.
5. Funds are permanently frozen.

Factory patterns (deploy-then-fund) are standard in DeFi. The likelihood is moderate-to-high for any protocol using the `deploy` syscall followed by a fund transfer.

---

### Recommendation

Fix `execute_deploy` to propagate the constructor's revert status. The `deploy_contract` function should return an `is_reverted` flag (analogous to `execute_entry_point`), and `execute_deploy` should write:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

This mirrors the existing correct pattern in `contract_call_helper`.

---

### Proof of Concept

1. Attacker declares class `C` whose constructor reverts when called with specific calldata or when gas is below a threshold.
2. Victim factory contract `V` calls `deploy(class_hash=C, constructor_calldata=..., ...)`.
3. The OS executes `deploy_contract(...)` → constructor of `C` reverts internally.
4. `execute_deploy` writes `ResponseHeader(gas=remaining_gas, failure_flag=0)` — success.
5. `V` reads `failure_flag=0`, reads the returned `contract_address=addr` from `DeployResponse`.
6. `V` calls ERC20 `transfer(addr, amount)` — funds are stored in ERC20 storage under `addr`.
7. No contract exists at `addr` (constructor reverted; contract not initialized).
8. Funds at `addr` are permanently frozen — no contract logic exists to recover them. [4](#0-3)
