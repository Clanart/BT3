### Title
Missing Minimum Normalized-Amount Check in `init_transfer` Causes Permanent Fund Lock for Sub-Threshold Transfers — (`File: near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer` accepts and locks/burns user tokens for any amount satisfying `fee < amount`, but never verifies that `normalize_amount(amount - fee, decimals) > 0`. When a user bridges an amount smaller than the normalization scale (i.e., `amount - fee < 10^(origin_decimals - decimals)`), the tokens are irrecoverably locked or burned on NEAR, and every subsequent `sign_transfer` call panics with `InvalidAmountToTransfer`. There is no user-callable cancel path, making the lock permanent.

---

### Finding Description

**Step 1 — `init_transfer` accepts the transfer and locks tokens.**

`init_transfer` only validates `fee < amount`:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

It then calls `init_transfer_internal`, which burns (for deployed tokens) or locks (for native tokens) the full `amount`:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
```

The function returns `U128(0)`, signalling to the NEP-141 token contract that all tokens were consumed. No refund is issued.

**Step 2 — `sign_transfer` always panics for sub-threshold amounts.**

When the relayer calls `sign_transfer`, it computes:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

`normalize_amount` uses floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

If `amount - fee < 10^(origin_decimals - decimals)`, the result is `0`, and `sign_transfer` panics unconditionally. The transfer message remains in `pending_transfers` forever.

**Step 3 — No cancel mechanism exists.**

There is no user-callable `cancel_transfer` or `withdraw_pending` function. The only refund paths are:
- Storage balance insufficient during `init_transfer_internal` (does not apply here — the transfer succeeded)
- `ft_transfer_call` callback returning non-zero (does not apply — `init_transfer_internal` returned `U128(0)`)

The DAO's `set_locked_tokens` can adjust the accounting counter but cannot return the actual tokens to the user.

---

### Impact Explanation

**For deployed (bridged) tokens**: The tokens are burned in `init_transfer_internal`. They are permanently destroyed with no recovery path.

**For native tokens**: The tokens are transferred into the bridge and the `locked_tokens` counter is incremented. The counter is never decremented because `sign_transfer` always panics and `claim_fee` is never reachable. The tokens are permanently locked in the bridge contract.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

Any token registered with `origin_decimals > decimals` (e.g., a 24-decimal NEAR-native token bridging to an 18-decimal EVM chain, where scale = `10^6 = 1,000,000`) is affected. A user bridging fewer than `1,000,000` base units (less than `0.000001` of the token in 18-decimal representation) will permanently lose their funds. This is realistic for:
- Users attempting to bridge a small remaining balance ("clear their position")
- Automated systems or bots sending dust amounts
- Any user whose balance is below the normalization threshold after fees

The entry path is fully unprivileged: any user can call `ft_transfer_call` on the token contract.

---

### Recommendation

Add a minimum normalized amount check inside `init_transfer`, before tokens are locked or burned, so the transfer is rejected and tokens are refunded:

```rust
// After computing transfer_message, before init_transfer_internal:
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(decimals) = token_address.and_then(|a| self.token_decimals.get(&a)) {
    let normalized = Self::normalize_amount(
        transfer_message.amount_without_fee().unwrap_or(0),
        decimals,
    );
    require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
}
```

Alternatively, add a user-callable `cancel_pending_transfer` that removes the transfer message and returns locked/minted tokens to the sender.

---

### Proof of Concept

**Setup**: Token registered with `origin_decimals = 24`, `decimals = 18` (scale = `10^6`).

1. User calls `ft_transfer_call` with `amount = 500_000` (< 1,000,000), `fee = 0`.
2. `init_transfer` passes the `fee < amount` check.
3. `init_transfer_internal` burns/locks `500_000` tokens and returns `U128(0)`.
4. Relayer calls `sign_transfer`.
5. `normalize_amount(500_000, {origin_decimals: 24, decimals: 18}) = 500_000 / 1_000_000 = 0`.
6. `require!(0 > 0, ...)` panics with `InvalidAmountToTransfer`.

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
