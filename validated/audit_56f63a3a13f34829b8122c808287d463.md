### Title
P2P Sync State Diff Validates Length But Not Poseidon Commitment, Allowing Wrong State to Be Written to Storage — (File: `crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary

`parse_data_for_block` in the P2P sync state diff stream builder accepts an assembled `ThinStateDiff` from a peer if its total element count matches `state_diff_length` from the stored block header, but never verifies the assembled diff's Poseidon hash against `state_diff_commitment` — the cryptographic commitment that is stored in the same header. A peer can send chunks whose element count is correct but whose content is entirely different from the committed state diff. The wrong diff is written directly to storage, corrupting every downstream consumer: the state root computed by the committer, the block hash, RPC state queries, and proof inputs.

### Finding Description

The block header stored during header sync carries two independent fields that together describe the state diff:

- `state_diff_length: Option<usize>` — the total element count
- `state_diff_commitment: Option<StateDiffCommitment>` — the Poseidon hash of the canonical diff

`parse_data_for_block` reads only the first:

```rust
// crates/apollo_p2p_sync/src/client/state_diff.rs  lines 62-70
let target_state_diff_len = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("A header with number lower than the header marker is missing")
    .state_diff_length          // ← only length is read
    .ok_or(P2pSyncClientError::OldHeaderInStorage { ... })?;
```

It then accumulates chunks until the count matches and writes the result to storage:

```rust
// lines 72-107
while current_state_diff_len < target_state_diff_len {
    // ... receive chunk, add to result ...
    current_state_diff_len += state_diff_chunk.len();
    unite_state_diffs(&mut result, state_diff_chunk)?;
}
if current_state_diff_len != target_state_diff_len { /* reject */ }
validate_deprecated_declared_classes_non_conflicting(&result)?;
Ok(Some((result, block_number)))   // ← no hash check before returning
```

The `state_diff_commitment` field is never read from the header, and `calculate_state_diff_hash(&result)` is never called. The assembled diff is then persisted unconditionally:

```rust
// lines 33-35
storage_writer.begin_rw_txn()?.append_state_diff(self.1, self.0)?.commit()?;
```

The `StorageBlockHeader` struct confirms both fields are present and independently stored:

```rust
// crates/apollo_storage/src/header.rs  lines 99-107
pub state_diff_commitment: Option<StateDiffCommitment>,
// ...
pub state_diff_length: Option<usize>,
```

The analog to the external vault bug is exact: the vault's `requestWithdrawal()` checked only share count (one dimension) while ignoring asset value vs. liquid capital (the second dimension). Here, `parse_data_for_block` checks only element count (one dimension) while ignoring the Poseidon commitment hash (the second dimension). In both cases the missing check is the one that actually binds the content to a canonical value.

### Impact Explanation

A peer that sends chunks with the correct total element count but arbitrary content causes:

1. **Wrong `ThinStateDiff` written to storage** — storage values, nonces, class hashes, and deployed contracts for the block are all attacker-controlled.
2. **Wrong state root** — the committer (`apollo_committer`) calls `commit_state_diff` on whatever is in storage; the resulting Patricia trie root is wrong.
3. **Wrong block hash** — `calculate_block_hash` chains `state_root` and `state_diff_commitment`; both are now inconsistent with the canonical chain.
4. **Wrong proof inputs/outputs** — the transaction prover and SNOS consume the stored state diff; proofs generated from it are invalid or prove a false statement.
5. **Wrong RPC responses** — `starknet_getStorageAt`, `starknet_getNonce`, `starknet_getClassHashAt` all read from the corrupted storage.

This matches the allowed Critical impact: *"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input"* and the High impact: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

### Likelihood Explanation

Any peer participating in the P2P sync protocol can trigger this. No privileged access is required. The attacker only needs to:
1. Participate as a peer in the network.
2. Respond to a state diff query with chunks whose `len()` values sum to `target_state_diff_len` but whose content differs from the canonical diff.

The `state_diff_commitment` is already present in the stored header at the time `parse_data_for_block` runs (headers are synced before state diffs, enforced by `BlockNumberLimit::HeaderMarker`), so the fix requires only reading and comparing it — the data is already available.

### Recommendation

After assembling the full `result` diff and before returning it, compute and compare its Poseidon hash against the stored commitment:

```rust
// After the while loop and length check:
let stored_commitment = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("header must exist")
    .state_diff_commitment
    .ok_or(P2pSyncClientError::OldHeaderInStorage {
        block_number,
        missing_field: "state_diff_commitment",
    })?;

let computed_commitment = calculate_state_diff_hash(&result);
if computed_commitment != stored_commitment {
    return Err(ParseDataError::BadPeer(BadPeerError::StateDiffCommitmentMismatch {
        expected: stored_commitment,
        got: computed_commitment,
    }));
}
```

This mirrors the check already present in the committer's `commit_block_inner` when `verify_state_diff_hash` is enabled, and closes the gap in the sync path.

### Proof of Concept

1. A syncing node has completed header sync for block N. The stored header contains:
   - `state_diff_length = 3`
   - `state_diff_commitment = Poseidon("STARKNET_STATE_DIFF0", ..., canonical_entries...)`

2. A malicious peer responds to the state diff query with three chunks that each have `len() == 1` (total = 3, matching `target_state_diff_len`), but whose content sets arbitrary storage values — e.g., setting the fee token balance of an attacker-controlled address to `u128::MAX`.

3. `parse_data_for_block` accepts the diff (length matches, no conflicts, no duplicate deprecated classes).

4. `write_to_storage` calls `append_state_diff(block_number, poisoned_diff)` and commits.

5. The committer later calls `commit_state_diff(poisoned_diff)`, producing a wrong Patricia root. The wrong root is stored as the state root for block N.

6. Any RPC call to `starknet_getStorageAt` for the manipulated slot returns the attacker-injected value. The block hash computed from this root diverges from the canonical chain. pub fn calculate_state_diff_hash(state_diff: &ThinStateDiff) -> StateDiffCommitment {
    let mut hash_chain = HashChain::new();
    hash_chain = hash_chain.chain(&STARKNET_STATE_DIFF0);
    hash_chain = chain_deployed_contracts(&state_diff.deployed_contracts, hash_chain);
    hash_chain = chain_declared_classes(&state_diff.class_hash_to_compiled_class_hash, hash_chain);
    hash_chain =
        chain_deprecated_declared_classes(&state_diff.deprecated_declared_classes, hash_chain);
    hash_chain = hash_chain.chain(&Felt::ONE) // placeholder.
        .chain(&Felt::ZERO); // placeholder.
    hash_chain = chain_storage_diffs(&state_diff.storage_diffs, hash_chain);
    hash_chain = chain_nonces(&state_diff.nonces, hash_chain);
    StateDiffCommitment(PoseidonHash(hash_chain.get_poseidon_hash()))
}
