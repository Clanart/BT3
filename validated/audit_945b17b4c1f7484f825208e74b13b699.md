### Title
Missing Class Hash Existence Validation in `execute_replace_class` Enables Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS Cairo program does not validate that the caller-supplied class hash corresponds to any declared contract class before committing the state update. An unprivileged contract deployer can exploit this to permanently brick a contract — freezing all funds it holds — by calling `replace_class` with an arbitrary, non-existent class hash. The OS itself acknowledges the missing check via an explicit TODO comment.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` processes the `replace_class` syscall. It reads `class_hash` directly from the syscall request and writes it into the contract's `StateEntry` without any check that the hash corresponds to a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The hint `GetContractAddressStateEntry` is not a Cairo-enforced constraint — it is a prover hint. The only Cairo-enforced logic is the `dict_update`, which unconditionally commits the new (potentially invalid) class hash to the state. There is no lookup into `contract_class_changes` to confirm the hash was ever declared.

This is the direct analog of the external report's pattern: a one-way state transition (deposit/replace) exists with no guard ensuring the reverse path (withdrawal/execution) remains possible.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to a non-existent value, every subsequent call to that contract will fail at class resolution time. No entry point — including any withdrawal or recovery function — can ever execute again. Any ERC-20 tokens, ETH, or other assets held in the contract's storage are permanently inaccessible with no protocol-level recovery mechanism.

This satisfies the allowed impact: *Critical. Permanent freezing of funds.*

---

### Likelihood Explanation

**Medium.**

The attack requires deploying a contract that deliberately (or through a bug) calls `replace_class` with an invalid hash. A realistic scenario is a rug-pull: a malicious deployer publishes a contract with a `deposit` entry point and a hidden `brick` function. After accumulating user funds, the deployer calls `brick`, which internally calls `replace_class(0xDEAD_CLASS)`. The OS accepts the syscall without validation, the state is committed, and all deposited funds are frozen. The deployer needs no privileged role — only the ability to deploy a contract and call its own entry points.

---

### Recommendation

Inside `execute_replace_class`, before committing the `dict_update`, add a Cairo-enforced assertion that the requested `class_hash` exists in `contract_class_changes` (i.e., has a non-zero compiled class hash). Concretely:

```cairo
// Enforce that the class has been declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the existing pattern used in `execute_declare_transaction`, which asserts `assert_not_zero(compiled_class_hash)` before writing to `contract_class_changes`. [2](#0-1) 

---

### Proof of Concept

1. **Attacker deploys** a malicious contract (class `A`) containing:
   - `deposit(token, amount)` — transfers ERC-20 tokens into the contract.
   - `brick()` — calls the `replace_class` syscall with `class_hash = 0xBADC0DE` (a felt that has never been declared).

2. **Users deposit** tokens by calling `deposit(...)`. Funds accumulate in the contract's storage.

3. **Attacker calls** `brick()`. The OS executes `execute_replace_class`:
   - Reads `class_hash = 0xBADC0DE` from the syscall request.
   - Skips the missing existence check (line 898 TODO).
   - Commits `StateEntry(class_hash=0xBADC0DE, ...)` to `contract_state_changes`.

4. **State is finalized.** The contract's class hash in the Merkle trie is now `0xBADC0DE`.

5. **All future calls fail.** Any invoke transaction targeting this contract causes the OS to look up class `0xBADC0DE`, find nothing, and revert. No withdrawal, transfer, or recovery function can execute.

6. **Funds are permanently frozen.** The deposited tokens remain in the contract's storage with no reachable code path to move them. [3](#0-2)
