### Title
Permanent Fund Lock When `normalize_amount` Floor Division Yields Zero for `amount_without_fee` — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

The NEAR bridge contract's `init_transfer` path accepts any transfer where `fee < amount`, but does not verify that `amount - fee >= 10^diff_decimals`. When `normalize_amount` is later called in `sign_transfer`, floor division can reduce `amount_without_fee` to zero, causing an irrecoverable panic. Because there is no cancel or admin-refund path, the locked tokens are permanently frozen.

---

### Finding Description

**Root cause — `normalize_amount` uses floor division:** [1](#0-0) 

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))   // floor division
}
```

For any token whose NEAR representation has fewer decimals than its origin chain (e.g. 8-decimal WBTC on EVM → 6-decimal on NEAR, `diff_decimals = 2`), any `amount_without_fee < 10^diff_decimals` normalises to **zero**.

**`init_transfer` only checks `fee < amount`, not the normalisation floor:** [2](#0-1) 

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

There is no guard that `amount - fee >= 10^diff_decimals`. Tokens are locked immediately after this check passes: [3](#0-2) 

**`sign_transfer` then panics unconditionally for such transfers:** [4](#0-3) 

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

Because NEAR panics revert the transaction without modifying state, the transfer message remains in `pending_transfers` and the tokens remain locked. Every subsequent call to `sign_transfer` for this transfer ID will panic identically.

**`update_transfer_fee` cannot rescue the transfer:** [5](#0-4) 

```rust
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

The fee can only be *increased*, never decreased. If `amount < 10^diff_decimals`, no valid fee exists that makes `amount - fee >= 10^diff_decimals`, so the transfer is irrecoverable regardless of fee updates.

**No cancel or admin-refund function exists.** The only removal paths are `remove_transfer_message` (called on successful signing or fee claim) and `remove_transfer_message_without_refund` (called on storage failure), neither of which is reachable for a stuck transfer. [6](#0-5) 

---

### Impact Explanation

Any user who initiates a NEAR→EVM (or NEAR→Solana/StarkNet) transfer where `amount - fee < 10^diff_decimals` will have their tokens permanently locked in the bridge with no on-chain recovery path. The locked amount is bounded by `10^diff_decimals - 1` units of the origin token per transfer, but:

- For a token with `diff_decimals = 2` (e.g. WBTC 8→6), up to 99 satoshis (~$0.05) per transfer are frozen.
- For a token with `diff_decimals = 12` (e.g. ETH 18→6), up to `10^12 - 1` wei (~$0.003) per transfer are frozen.
- A user who sets an initial fee of `amount - (10^diff_decimals - 1)` (e.g. fee = 901 on amount = 1000 with diff = 2) locks the **entire** `amount` (1000 units), not just the sub-unit remainder, because `sign_transfer` panics before any tokens are released.

This satisfies the **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds"** impact class.

---

### Likelihood Explanation

- Any user bridging a token with `origin_decimals > decimals` (a common configuration for EVM→NEAR) is exposed.
- A user who transfers a dust-level amount (e.g. testing with 1 unit) or who sets a fee that leaves `amount_without_fee < 10^diff_decimals` triggers the lock.
- The UI/relayer does not validate this condition before accepting the transfer.
- The condition is distinct from the acknowledged "dust stays locked" note in the code comment, which describes the *remainder* of a successful normalisation, not the case where normalisation yields zero and the transfer is permanently unserviceable.

---

### Recommendation

1. **Add a minimum-amount guard in `init_transfer`** (and the `ft_transfer_call` handler) before locking tokens:

```rust
let decimals = self.token_decimals.get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);
let min_transferable = 10_u128.pow(
    (decimals.origin_decimals - decimals.decimals).into()
);
require!(
    transfer_message.amount.0.saturating_sub(transfer_message.fee.fee.0) >= min_transferable,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

2. **Add a DAO-gated `cancel_transfer` function** that removes the pending transfer and returns the locked tokens to the original sender, as a safety net for edge cases.

### Proof of Concept

**Setup:** WBTC is registered with `origin_decimals = 8`, `decimals = 6` → `diff_decimals = 2`, `10^diff = 100`.

**Step 1 — User initiates transfer with amount = 50, fee = 0:**
- `init_transfer` check: `0 < 50` ✓ — passes.
- `init_transfer_internal` locks 50 WBTC units in the bridge.

**Step 2 — Relayer calls `sign_transfer`:**
```
amount_without_fee = 50 - 0 = 50
normalize_amount(50, {origin=8, decimals=6}) = 50 / 100 = 0
require!(0 > 0, ...) → PANIC: ERR_INVALID_AMOUNT_TO_TRANSFER
```

**Step 3 — User attempts `update_transfer_fee`:**
- Any fee `f` satisfying `0 <= f < 50` leaves `amount_without_fee` in `[1, 50]`, all of which normalise to 0.
- No valid fee exists. Transfer is permanently stuck.

**Result:** 50 WBTC units are irrecoverably locked in the bridge contract. [4](#0-3) [2](#0-1) [1](#0-0)
