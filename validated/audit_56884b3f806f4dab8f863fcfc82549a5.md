### Title
Wrong-epoch shard index used to access `prev_block.chunks()` in `get_state_response_part` silently reads parent shard's state root for child shards after resharding — (File: `chain/chain/src/state_sync/adapter.rs`)

---

### Summary

In `get_state_response_part`, the shard index is derived from the **new epoch's** shard layout but used to index into `prev_block.chunks()`, which holds the **old epoch's** chunk headers. After a resharding event, child shard IDs map to indices in the new layout that either (a) fall within the old chunk array but point to the **parent shard's** chunk — silently returning the wrong `prev_state_root` — or (b) exceed the old chunk array's length, returning `None` and an error. This is the direct nearcore analog of `burnFees[feeCycle.length]`: an index that is valid in the new context is out-of-range (or wrong) in the old context, causing the operation to fail or produce a corrupted value.

---

### Finding Description

`get_state_response_part` in `chain/chain/src/state_sync/adapter.rs` (lines 305–316) computes the shard index from the **sync block's epoch** (the new epoch) and then uses it to index into `prev_block.chunks()`, where `prev_block` is the **last block of the old epoch**:

```rust
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;   // new epoch
// ...
let prev_block = self.chain_store.get_block(header.prev_hash())?;     // old epoch block
let shard_index = shard_layout.get_shard_index(shard_id)?;            // index in NEW layout
let state_root = prev_block
    .chunks()
    .get(shard_index)                                                  // indexes OLD chunks
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
``` [1](#0-0) 

After resharding (e.g., 4 shards → 5 shards, where shard 3 splits into child shards 4 and 5):

- **New layout**: shard IDs `[0, 1, 2, 4, 5]` → indices `[0, 1, 2, 3, 4]`
- **Old epoch block**: 4 chunks at indices `[0, 1, 2, 3]`

For **child shard 4** (index 3 in new layout): `prev_block.chunks().get(3)` returns the **parent shard 3's** chunk header. `prev_state_root()` returns the parent's state root — not the child's. The function silently continues with the wrong state root, generating state parts for the wrong trie.

For **child shard 5** (index 4 in new layout): `prev_block.chunks().get(4)` returns `None`. The function returns `Err(Error::InvalidShardId(shard_id))`, blocking state sync entirely.

There is no resharding-boundary check in this function. The correct approach (used elsewhere, e.g., `get_prev_shard_id_from_prev_hash`) is to detect the resharding boundary and look up the **parent shard's** index in the old layout. [2](#0-1)
