### Title
Double-Mint via `remove_fin_transfer` Race: Proof Replay After Recipient-Triggered Refund — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When a deployed (mint/burn) token transfer to a NEAR recipient includes a non-empty `msg`, the bridge mints tokens and calls `ft_transfer_call` on the recipient. If the recipient's `ft_on_transfer` returns the full amount (triggering a refund), `fin_transfer_send_tokens_callback` fires a **detached** burn and then **synchronously removes** the `transfer_id` from `finalised_transfers`. Because the prover has no independent replay guard, the same proof can be submitted a second time by any trusted relayer — including one that retries in good faith after observing `FailedFinTransferEvent` — minting the tokens a second time.

---

### Finding Description

**Entry point — `fin_transfer`**

`fin_transfer` is gated by `#[trusted_relayer]` and calls `verify_proof`, then `fin_transfer_callback`. [1](#0-0) 

**Finalization guard — `add_fin_transfer`**

`process_fin_transfer_to_near` inserts `transfer_id` into `finalised_transfers` with a `require!` that panics on duplicate — this is the only replay guard. [2](#0-1) [3](#0-2) 

**Token delivery — `send_tokens`**

For a deployed token with a non-empty `msg`, `send_tokens` calls `mint(recipient, amount, Some(msg))` on the bridge token. The token contract mints and then calls `ft_transfer_call` on the recipient; the promise result is the *used* amount (0 if the recipient returned everything). [4](#0-3) 

**Refund detection — `is_refund_required`**

Returns `true` when the `ft_transfer_call` result is `0` — i.e., the recipient's `ft_on_transfer` returned the full amount. [5](#0-4) 

**The vulnerable callback — `fin_transfer_send_tokens_callback`**

When `is_refund_required` is `true`:
1. `burn_tokens_if_needed` fires a **detached** (fire-and-forget) burn promise.
2. `remove_fin_transfer` **synchronously** removes `transfer_id` from `finalised_transfers`.
3. `FailedFinTransferEvent` is emitted. [6](#0-5) 

**`remove_fin_transfer` clears the replay guard**

```rust
fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
    self.finalised_transfers.remove(transfer_id);   // guard gone
    ...
}
``` [7](#0-6) 

**No prover-level replay guard**

The EVM prover verifies the Merkle proof against the light client but does not record used proofs. The bridge's `finalised_transfers` is the sole deduplication mechanism. [8](#0-7) 

**Sequence of events**

| Step | Action | State |
|------|--------|-------|
| 1 | Attacker initiates transfer on source chain with non-empty `msg`, recipient = attacker's NEAR contract | Source tokens locked |
| 2 | Relayer calls `fin_transfer` (valid proof, nonce N) | `finalised_transfers` ∋ N |
| 3 | `mint(attacker, amount, Some(msg))` → `ft_transfer_call` → attacker's `ft_on_transfer` returns full amount | Tokens refunded to bridge |
| 4 | `fin_transfer_send_tokens_callback`: detached burn fired, `remove_fin_transfer` removes N, `FailedFinTransferEvent` emitted | `finalised_transfers` ∌ N |
| 5 | Relayer observes `FailedFinTransferEvent`, retries with same proof | — |
| 6 | `add_fin_transfer` succeeds (N absent), tokens minted again | Attacker holds tokens; source still locked |
| 7 | Attacker's `ft_on_transfer` returns 0 (accepts tokens) | `FinTransferEvent` emitted; double-mint complete |

---

### Impact Explanation

The attacker receives bridged tokens on NEAR without the corresponding source-chain tokens ever being released. The bridge token supply is inflated beyond the locked collateral, breaking bridge collateralization. This is a direct unauthorized mint of bridged assets.

---

### Likelihood Explanation

- The attacker only needs to deploy a NEAR contract that returns the full amount from `ft_on_transfer` on the first call and `0` on the second — trivial to implement.
- The attacker sets a non-empty `msg` when initiating the transfer on the source chain — fully under their control.
- Relayers are automated systems explicitly designed to retry transfers that emit `FailedFinTransferEvent`. No malicious relayer behavior is required; the retry is the intended protocol response to a failed finalization.
- The prover has no independent replay guard, so the same proof is accepted on the second submission.

---

### Recommendation

**Do not remove `transfer_id` from `finalised_transfers` on refund.** The finalization record must be permanent. The correct fix is to keep the entry in `finalised_transfers` even when the token delivery fails, so the proof can never be replayed.

If storage reclamation is needed, use a separate `failed_transfers` set or a status flag rather than removing the entry. The burn can remain detached, but the replay guard must not be cleared.

```rust
// REMOVE this line from fin_transfer_send_tokens_callback:
// self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);
```

Alternatively, if the intent is to allow re-delivery (e.g., with a different recipient), the retry path must re-verify the proof and use a separate nonce or a dedicated "re-deliver" function that does not accept the original proof again.

### Proof of Concept

```rust
// Attacker's recipient contract
#[near]
impl AttackerContract {
    static mut CALL_COUNT: u32 = 0;

    pub fn ft_on_transfer(
        &mut self,
        sender_id: AccountId,
        amount: U128,
        msg: String,
    ) -> PromiseOrValue<U128> {
        unsafe {
            CALL_COUNT += 1;
            if CALL_COUNT == 1 {
                // First call: return full amount → triggers refund path
                PromiseOrValue::Value(amount)
            } else {
                // Second call: accept tokens → double-mint complete
                PromiseOrValue::Value(U128(0))
            }
        }
    }
}
```

**Test assertions:**
1. After step 4: `contract.is_transfer_finalised(transfer_id) == false` — guard removed.
2. After step 6: second `fin_transfer` with same proof succeeds (no `TransferAlreadyFinalised` panic).
3. Attacker token balance = `amount_without_fee` (minted twice, burned once from bridge, net +1 mint to attacker).

    fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
        let storage_usage = env::storage_usage();
        self.finalised_transfers.remove(transfer_id);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(storage_owner, &storage);
        }
    }
