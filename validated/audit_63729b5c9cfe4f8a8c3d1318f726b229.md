### Title
Dust-Amount Transfer Permanently Locks User Tokens Due to Missing Pre-Normalization Check — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer` on NEAR accepts and locks user tokens without verifying that the post-normalization amount (i.e., the amount after converting from NEAR's high-precision decimals to the lower-precision destination-chain decimals) is greater than zero. Any subsequent call to `sign_transfer` for such a transfer will always panic with `InvalidAmountToTransfer`, leaving the locked tokens permanently irrecoverable.

---

### Finding Description

The bridge stores token amounts in NEAR's native precision (up to 24 decimals). When a transfer targets a lower-precision chain (e.g., 6-decimal USDC on EVM), `sign_transfer` normalizes the amount before MPC signing: [1](#0-0) 

```rust
/// Uses floor division — any sub-unit remainder ("dust") is truncated and not transferred
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

`sign_transfer` then guards against a zero result: [2](#0-1) 

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

However, `init_transfer` — which is the entry point that actually locks the tokens — only checks that `fee < amount`: [3](#0-2) 

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

There is **no check** that `normalize_amount(amount - fee, decimals) > 0` at the time of locking. A user can therefore lock any positive amount that is smaller than the normalization divisor (e.g., any amount < 10^18 for a 24→6 decimal conversion) and the transfer will be accepted and tokens locked, but `sign_transfer` will always revert.

The `sign_transfer_callback` only removes the transfer message when MPC signing **succeeds**: [4](#0-3) 

```rust
if let Ok(signature) = call_result {
    if fee.is_zero() {
        self.remove_transfer_message(message_payload.transfer_id);
    }
```

Because `sign_transfer` panics before even reaching the MPC call, the transfer message is never removed and no recovery path exists.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

Any user who initiates a NEAR→EVM transfer with an amount below the decimal-normalization threshold (e.g., < 10^18 yocto-units for a 24→6 decimal token pair) will have their tokens permanently locked in the bridge contract. There is no cancel, timeout, or admin-recovery function visible in the production code that can release these funds.

---

### Likelihood Explanation

**Medium-High.** The condition is trivially reachable by any unprivileged user via a standard `ft_transfer_call`. A user sending a small test amount, a UI rounding error, or a deliberate griefing attempt can all trigger this. The only prerequisite is that the token pair has `origin_decimals > decimals` (e.g., any NEAR-native token bridged to a 6- or 18-decimal EVM representation), which is the common case for NEAR tokens (24 decimals) bridged to EVM.

---

### Recommendation

Add a normalization check inside `init_transfer` (or `init_transfer_internal`) before locking tokens. Retrieve the destination-chain `Decimals` for the token and verify:

```rust
let normalized = Self::normalize_amount(
    amount.0.checked_sub(init_transfer_msg.fee.0)
        .expect(BridgeError::InvalidFee.as_ref()),
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

This mirrors the existing guard in `sign_transfer` and ensures the rejection happens at lock-time, before any tokens are transferred to the bridge.

---

### Proof of Concept

**Setup:** NEAR-native token with `origin_decimals = 24`, bridged to EVM USDC with `decimals = 6`. Normalization divisor = 10^18.

1. User calls `ft_transfer_call` on the token contract:
   - `receiver_id` = bridge contract
   - `amount` = `500_000_000_000_000_000` (5 × 10^17, i.e., 0.5 in 18-decimal terms, well below 10^18)
   - `msg` = `InitTransferMsg { fee: 0, recipient: "eth:0xABCD...", ... }`

2. Bridge's `ft_on_transfer` fires → `init_transfer` is called.

3. Check at line 554: `fee (0) < amount (5×10^17)` → **passes**. Tokens are locked.

4. Relayer calls `sign_transfer` for this transfer ID.

5. `normalize_amount(5×10^17, {origin_decimals:24, decimals:6})` = `5×10^17 / 10^18` = **0** (integer floor division).

6. `require!(0 > 0, ...)` → **PANIC** with `InvalidAmountToTransfer`.

7. No MPC call is made. `sign_transfer_callback` is never reached. Transfer message stays in storage. Tokens are permanently locked with no recovery path. [5](#0-4) [2](#0-1) [6](#0-5)
