### Title
Missing `chunk.shard_id() == shard_id` Guard in `set_state_header` Allows Cross-Shard State Root Substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` whose embedded chunk belongs to a different shard than the one being synced. Because `verify_path` only checks that a chunk hash is *somewhere* in the `chunk_headers_root` Merkle tree (not *which position/index* it occupies), and because no code in the function ever asserts `chunk.shard_id() == shard_id`, a malicious snapshot-host peer can substitute shard 1's chunk — and therefore shard 1's `prev_state_root` — into the `StateHeaderKey(0, sync_hash)` DB entry. Every downstream call (`set_state_part`, `apply_state_part`) then operates against the wrong state root.

### Finding Description

**Entrypoint**: `set_state_header` in `chain/chain/src/state_sync/adapter.rs`.

The function receives `shard_id` (the shard being synced) and `shard_state_header` (the peer-supplied header). It extracts the embedded chunk:

```rust
let chunk = shard_state_header.cloned_chunk();
``` [1](#0-0) 

**Guard 1 — `validate_chunk_proofs`**: Checks the chunk's internal hash, tx root, and receipts root. It does **not** compare `chunk.shard_id()` against the `shard_id` parameter. [2](#0-1) 

**Guard 2 — `verify_path`**: Checks that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` hashes to the `chunk_headers_root`. This is a plain Merkle membership proof — it confirms the hash is *in the tree*, not *at which leaf index (shard position)* it sits. [3](#0-2) 

Compare with `verify_path_with_index`, which additionally calls `verify_path_matches_index` to enforce the leaf position. `set_state_header` uses the weaker `verify_path`. [4](#0-3) 

Because shard 1's chunk hash is a genuine leaf in the same `chunk_headers_root` tree, a valid Merkle proof for it passes the check even when the request is for shard 0.

**Guard 3 — receipt proof validation (step 4)**: Uses the `shard_id` *parameter* (0), not `chunk.shard_id()` (1), to hash receipts:

```rust
let receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts));
``` [5](#0-4) 

An attacker supplies shard 0's legitimate incoming receipts (publicly available on-chain) alongside shard 1's chunk, so this check passes.

**Guard 4 — `validate_state_root_node` (step 5)**: Validates the `state_root_node` against `chunk_inner.prev_state_root()`. Since `chunk_inner` is from shard 1's chunk, this validates shard 1's state root node — which the attacker, as a full node tracking shard 1, can supply correctly. [6](#0-5) 

**Storage**: The header is then written under `StateHeaderKey(shard_id=0, sync_hash)` but contains shard 1's chunk and shard 1's `prev_state_root`. [7](#0-6) 

**Downstream corruption**: `set_state_part` reads the stored header and extracts `state_root` from it:

```rust
let state_root = *chunk.take_header().take_inner().prev_state_root();
``` [8](#0-7) 

Parts are then validated against shard 1's state root and applied into shard 0's `ShardUId`, writing shard 1's trie data into shard 0's storage.

### Impact Explanation

A syncing node that accepts the crafted header will:
1. Store `StateHeaderKey(0, sync_hash)` containing shard 1's `prev_state_root`.
2. Validate all subsequent state parts against shard 1's state root.
3. Call `apply_state_part` with shard 1's trie data for shard 0's `ShardUId`.

The result is that shard 0's local state is replaced with shard 1's state. The node will then operate with a corrupted view of shard 0, producing wrong execution results, wrong chunk extras, and potentially failing to finalize state sync or producing invalid blocks if it later becomes a validator.

### Likelihood Explanation

**Attacker capability**: Any peer can advertise as a snapshot host by broadcasting a signed `SnapshotHostInfo` message (signed with the peer's own key — no validator or operator privileges required). [9](#0-8) 

The syncing node selects a snapshot host from the advertised set and sends `StateRequestHeader` to it. The response is matched by `(shard_id, sync_hash, peer_id)` but the *content* of the header is only validated by `set_state_header`, which has the missing guard. [10](#0-9) 

The attacker needs to be a full node tracking both shards (no special privilege) to obtain shard 1's chunk body, Merkle proof, and state root node, plus shard 0's receipt proofs.

### Recommendation

Add an explicit shard identity check immediately after extracting the chunk in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

This should be inserted at approximately line 386 of `chain/chain/src/state_sync/adapter.rs`, after `validate_chunk_proofs` and before the `verify_path` call. [11](#0-10) 

Similarly, verify `prev_chunk_header.shard_id() == shard_id` when `prev_chunk_header` is `Some`.

### Proof of Concept

In a two-shard test-loop setup:
1. Let node A produce blocks for two epochs so both shards have state.
2. Intercept the state sync flow for node B syncing shard 0.
3. Instead of calling `get_state_response_header(shard_id=0, sync_hash)`, call `get_state_response_header(shard_id=1, sync_hash)` to obtain shard 1's header.
4. Call `set_state_header(shard_id=0, sync_hash, shard_1_header)` on node B.
5. Assert: `get_state_header(0, sync_hash).chunk().shard_id() == 1` (shard mismatch stored).
6. Assert: the stored `prev_state_root` differs from node A's canonical shard 0 `prev_state_root`.

The existing test infrastructure in `integration-tests/src/tests/client/process_blocks.rs` (the `slow_test_catchup_gas_price_change` test) provides a direct template for this reproducer. [12](#0-11)

        byzantine_assert!(false);

    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);

    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
