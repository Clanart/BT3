### Title
L1 Provider Drops `rejected_txs` During `CatchingUp` State, Permanently Losing Rejected L1 Handler Transactions - (File: crates/apollo_l1_provider/src/l1_provider.rs)

### Summary

`L1Provider::accept_commit_while_catching_up` silently discards the `rejected_txs` parameter, passing `Default::default()` (empty set) to `apply_commit_block` for both the current-height commit and every entry in the backlog. As a result, every L1 handler transaction that was consumed-but-rejected during a block processed while the provider is in `CatchingUp` state is incorrectly promoted to `TransactionState::Committed` instead of `TransactionState::Rejected`. Once committed, the transaction is removed from the proposable index and can never be re-proposed, permanently losing the corresponding L1 message.

### Finding Description

**Normal path** (`commit_block` when `Pending`/`Propose`/`Validate`):

```rust
// crates/apollo_l1_provider/src/l1_provider.rs:320-323
Ok(_) => {
    self.apply_commit_block(committed_txs, rejected_txs);  // ← real rejected set
    self.state = self.state.transition_to_pending();
    Ok(())
}
```

**Catchup path** — `rejected_txs` is silently dropped for both the current-height commit and every backlogged commit:

```rust
// crates/apollo_l1_provider/src/l1_provider.rs:426-427
// TODO(guyn): check what about rejected txs here and in the backlog?
Equal => self.apply_commit_block(committed_txs, Default::default()),
```

```rust
// crates/apollo_l1_provider/src/l1_provider.rs:463-465
for committed_block in backlog {
    self.apply_commit_block(committed_block.committed_txs, Default::default());
}
```

The `CommitBlockBacklog` struct structurally cannot carry rejected transactions because the field was never added:

```rust
// crates/apollo_l1_provider/src/catchupper.rs:184-188
pub struct CommitBlockBacklog {
    pub height: BlockNumber,
    pub committed_txs: IndexSet<TransactionHash>,
    // no rejected_txs field
}
```

Inside `apply_commit_block`, with an empty `rejected_txs`, the partition always puts every consumed transaction into the committed bucket:

```rust
// crates/apollo_l1_provider/src/l1_provider.rs:371-373
let (rejected_and_consumed, committed_txs): (Vec<_>, Vec<_>) =
    consumed_txs.iter().copied().partition(|tx| rejected_txs.contains(tx));
self.tx_manager.commit_txs(&committed_txs, &rejected_and_consumed);
```

`mark_committed()` then sets `state = TransactionState::Committed` and `committed = true`, which is irreversible:

```rust
// crates/apollo_l1_provider/src/transaction_record.rs:50-60
pub fn mark_committed(&mut self) {
    assert!(!self.committed, "L1 handler transaction {} committed twice ...", ...);
    self.state = TransactionState::Committed;
    self.committed = true;
}
```

The `l2_sync_task` in `catchupper.rs` also explicitly passes no rejected transactions when replaying blocks from state sync:

```rust
// crates/apollo_l1_provider/src/catchupper.rs:166-167
// No rejected txs in sync blocks.
let l1_handler_rejected_tx_hashes = Default::default();
```

This means that even the blocks the sync task replays to reach the target height treat all consumed L1 handler transactions as committed.

The batcher sends the real `rejected_l1_handler_tx_hashes` to `commit_block`:

```rust
// crates/apollo_batcher/src/batcher.rs:885-893
let rejected_l1_handler_tx_hashes = rejected_tx_hashes
    .iter()
    .copied()
    .filter(|tx_hash| consumed_l1_handler_tx_hashes.contains(tx_hash))
    .collect();
self.l1_provider_client
    .commit_block(consumed_l1_handler_tx_hashes, rejected_l1_handler_tx_hashes, height)
    .await;
```

But when the provider is in `CatchingUp` state, the call returns `L1ProviderError::UnexpectedHeight` and the batcher only logs the error without retrying:

```rust
// crates/apollo_batcher/src/batcher.rs:897-916
if let Err(err) = l1_provider_result {
    // ... only logs, does not retry or abort
    BATCHER_L1_PROVIDER_ERRORS.increment(1);
}
```

The rejected-transaction information is therefore permanently discarded.

### Impact Explanation

Any L1 handler transaction that was included in a block but whose execution was rejected by the blockifier, and whose corresponding `commit_block` notification arrived while the L1 provider was in `CatchingUp` state, is permanently marked `Committed`. Consequences:

1. The transaction is removed from the proposable index and will never be re-proposed.
2. The L1 message is permanently lost — it will never be executed on L2.
3. A future proposer that tries to include the same L1 message will not find it in the provider's proposable set.
4. A future validator that receives the same L1 message hash will see `AlreadyIncludedOnL2` (committed), causing block validation to fail if the proposer somehow re-includes it.

This matches **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing**: the L1 provider is the admission gate for L1 handler transactions; incorrectly marking a rejected (re-proposable) transaction as committed permanently prevents it from being re-sequenced.

### Likelihood Explanation

`CatchingUp` is entered whenever the provider receives a `commit_block` for a height different from its current height. This occurs routinely during:
- Node startup (provider must catch up to the current chain tip)
- Node restart after a crash
- Any transient lag between the batcher and the L1 provider

If any L1 handler transaction is rejected by the blockifier during a block that coincides with a catchup window, the bug triggers. Rejected L1 handler transactions are not rare — any L1 message whose target contract reverts will be rejected.

### Recommendation

1. Add a `rejected_txs: IndexSet<TransactionHash>` field to `CommitBlockBacklog` in `crates/apollo_l1_provider/src/catchupper.rs`.
2. Thread the `rejected_txs` parameter through `accept_commit_while_catching_up` and store it in the backlog entry.
3. Pass the stored `rejected_txs` to `apply_commit_block` when draining the backlog.
4. For blocks replayed by `l2_sync_task`, consider sourcing rejected-transaction information from the storage layer (e.g., from the stored block's execution receipts) rather than always passing an empty set.

### Proof of Concept

```
1. L1 provider is at height N, state = Pending.
2. L1 handler tx T is scraped and is in Pending state in the provider.
3. Batcher builds block N, includes T, T's execution fails → T is in
   consumed_l1_handler_tx_hashes AND rejected_l1_handler_tx_hashes.
4. Batcher calls commit_block(consumed={T}, rejected={T}, height=N+2)
   (height mismatch because provider is still at N).
   → Provider enters CatchingUp, returns UnexpectedHeight error.
   → Batcher logs error, continues. T's rejected status is lost.
5. Sync task replays blocks N, N+1, N+2 from state sync.
   For block N: commit_block(consumed={T}, rejected={}, N)
   → apply_commit_block({T}, {}) → T is partitioned into committed_txs
   → tx_manager.commit_txs([T], []) → T.mark_committed()
   → T.state = Committed, T.committed = true.
6. Provider exits CatchingUp, state = Pending.
7. T is now Committed. It is absent from the proposable index.
8. T will never be re-proposed. The L1 message is permanently lost.
