### Title
Missing Zero-Rate Validation in `resolve_query` Allows Oracle to Return Zero ETH-to-STRK Rate, Bypassing Fallback and Embedding Zero L1 Gas Prices in Block Proposals - (`File: crates/apollo_l1_gas_price/src/eth_to_strk_oracle.rs`)

---

### Summary

The `resolve_query` function in `EthToStrkOracleClient` parses the ETH-to-STRK conversion rate from the oracle's HTTP response but never checks whether the returned `rate` is zero. A zero rate is a structurally valid `u128` that passes all existing checks (field presence, `decimals == 18`), gets cached in the LRU cache, and is returned as `Ok(0)`. Downstream, `GasPrice::wei_to_fri(0)` silently returns `Ok(GasPrice(0))` — a successful conversion — so the three-tier fallback in `get_l1_prices_in_fri_and_wei_and_conversion_rate` is never reached. The result is that `l1_gas_price_fri = 0` and `l1_data_gas_price_fri = 0` are embedded in every block proposal built or validated while the zero rate is cached, causing incorrect L1 gas fee accounting with direct economic impact.

---

### Finding Description

**Root cause — `resolve_query` accepts zero rate without error:** [1](#0-0) 

After parsing `price` from the JSON body, the function immediately returns `Ok(rate)` with no check for `rate == 0`. A response body of `{"price":"0x0","decimals":18}` passes every guard and is returned as `Ok(0)`.

**Zero rate is cached and reused:** [2](#0-1) 

The zero rate is stored in the `cached_prices` LRU cache under `quantized_timestamp`. Every subsequent call for the same quantized bucket returns `Ok(0)` from the cache without re-querying the oracle.

**`wei_to_fri(0)` returns `Ok(GasPrice(0))` — not an error:** [3](#0-2) 

`self.checked_mul_u128(0)` = `Some(0)` (no overflow), then `0.checked_div(WEI_PER_ETH)` = `Some(0)`. The conversion succeeds and returns `Ok(GasPrice(0))`.

**Fallback is bypassed because the conversion succeeds:** [4](#0-3) 

The three-tier fallback (previous block info → configured defaults) is only entered when `eth_to_fri_rate` or `price_info` is `Err`, or when `convert_from_wei` returns `Err`. With `rate = 0`, all three calls return `Ok`, so the function returns `(prices_in_fri_with_zeros, prices_in_wei, 0)` at line 174 and the fallback is never reached.

**Zero FRI prices are embedded in the block proposal:** [5](#0-4) 

`l1_gas_price_fri` and `l1_data_gas_price_fri` in `ProposalInit` are set to zero. Both proposer and validators call the same oracle with the same quantized timestamp, so they all compute zero FRI prices and reach consensus on a block with zero L1 gas prices.

---

### Impact Explanation

**Incorrect L1 gas price effect with economic impact (Critical).**

When `l1_gas_price_fri = 0` and `l1_data_gas_price_fri = 0` are embedded in the block, every transaction in that block is charged zero L1 gas fees in FRI. The sequencer/protocol receives no L1 gas fee revenue for those blocks. Because the zero rate is cached for the entire quantized time window (`lag_interval_seconds`), the impact persists across multiple consecutive blocks until the cache entry expires and the oracle returns a non-zero value.

Additionally, the `eth_to_fri_rate = 0` is propagated into `PreviousBlockInfo` after `decision_reached`. If the oracle continues to fail, the fallback path `calculate_eth_to_fri_rate` will also fail (it explicitly rejects a zero rate at line 509 of `utils.rs`), forcing the system to fall back to `DEFAULT_ETH_TO_FRI_RATE` — but only after the zero-rate block has already been committed. [6](#0-5) 

---

### Likelihood Explanation

The oracle is an external HTTP API (`EthToStrkOracleClient`) whose response is fully controlled by the server at the configured URL. A zero price response (`{"price":"0x0","decimals":18}`) is structurally indistinguishable from a valid response — it passes JSON parsing, field-presence checks, and the `decimals == 18` guard. This can occur due to:

- A bug or misconfiguration in the oracle backend returning a zero price for a specific timestamp bucket.
- A transient data-feed issue where the oracle has no price data and defaults to zero.

The `L1GasPriceProvider.get_price_info` path already has a staleness guard (`StaleL1GasPricesError`) and a missing-data guard, demonstrating that the codebase is aware of the need to validate oracle data. The ETH-to-STRK oracle path has no equivalent zero-value guard. [7](#0-6) 

---

### Recommendation

**Short term:** Add a zero-rate check in `resolve_query` before returning:

```rust
if rate == 0 {
    return Err(EthToStrkOracleClientError::ParseError(
        "ETH-to-STRK rate is zero".to_string()
    ));
}
```

This causes `eth_to_fri_rate` to be `Err`, which triggers the existing fallback chain (previous block info → configured defaults) in `get_l1_prices_in_fri_and_wei_and_conversion_rate`.

**Long term:** Mirror the staleness/validity pattern already present in `L1GasPriceProvider.get_price_info` — add a minimum-rate threshold check analogous to `max_time_gap_seconds`, and add a metric/alert for zero or near-zero oracle rates.

### Proof of Concept

1. Configure the oracle URL to serve `{"price":"0x0","decimals":18}` for any timestamp query.
2. The sequencer calls `eth_to_fri_rate(timestamp)` → `spawn_query` → `resolve_query` → returns `Ok(0)`.
3. `0` is stored in `cached_prices` under `quantized_timestamp`.
4. `get_l1_prices_in_fri_and_wei_and_conversion_rate` receives `Ok(0)` for `eth_to_fri_rate` and `Ok(price_info)` for `price_info`.
5. `apply_fee_transformations` clamps WEI prices to configured minimums (non-zero).
6. `L1PricesInFri::convert_from_wei(&prices_in_wei, 0)` calls `wei_to_fri(0)` for each price → returns `Ok(GasPrice(0))` for both.
7. `get_l1_prices_in_fri_and_wei_and_conversion_rate` returns `(L1PricesInFri { l1_gas_price: 0, l1_data_gas_price: 0 }, prices_in_wei, 0)` at line 174 — fallback never entered.
8. `ProposalInit` is built with `l1_gas_price_fri = 0`, `l1_data_gas_price_fri = 0`.
9. Validators call the same oracle, get the same cached zero rate, compute the same zero FRI prices, and accept the proposal via `within_margin(0, 0, margin)` = `true`.
10. Block is committed with zero L1 gas prices in FRI; all transactions in the block pay zero L1 gas fees. [8](#0-7) [9](#0-8) [10](#0-9)

fn resolve_query(body: String) -> Result<u128, EthToStrkOracleClientError> {
    let Ok(json): Result<serde_json::Value, _> = serde_json::from_str(&body) else {
        return Err(EthToStrkOracleClientError::ParseError(format!(
            "Failed to parse JSON: {body}"
        )));
    };
    // Extract price from API response. Also returns MissingFieldError if value is not a string.
    let price = match json.get("price").and_then(|v| v.as_str()) {
        Some(price) => price,
        None => {
            return Err(EthToStrkOracleClientError::MissingFieldError("price".to_string(), body));
        }
    };
    let rate = u128::from_str_radix(price.trim_start_matches("0x"), 16)
        .expect("Failed to parse price as u128");
    // Extract decimals from API response. Also returns MissingFieldError if value is not a number.
    let decimals = match json.get("decimals").and_then(|v| v.as_u64()) {
        Some(decimals) => decimals,
        None => {
            return Err(EthToStrkOracleClientError::MissingFieldError(
                "decimals".to_string(),
                body,
            ));
        }
    };
    if decimals != ETH_TO_STRK_QUANTIZATION {
        return Err(EthToStrkOracleClientError::InvalidDecimalsError(
            ETH_TO_STRK_QUANTIZATION,
            decimals,
        ));
    }
    ETH_TO_STRK_SUCCESS_COUNT.increment(1);
    set_unix_now_seconds(&ETH_TO_STRK_LAST_SUCCESS_TIMESTAMP_SECONDS);
    ETH_TO_STRK_RATE.set_lossy(rate);
    Ok(rate)
}
