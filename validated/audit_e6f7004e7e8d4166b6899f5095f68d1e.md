### Title
Gas Refund After Execution Enables Block-Filling DoS via Oversized Gas Limit — (`ante/eth.go`, `x/evm/keeper/gas.go`, `x/evm/keeper/state_transition.go`)

---

### Summary

Ethermint's `CheckEthGasConsume` ante handler permits a single EVM transaction whose `gasLimit` equals the full block gas limit. Because `RefundGas` returns all unused gas tokens to the sender after execution, and `ResetGasMeterAndConsumeGas` resets the tx gas meter to actual gas used, an attacker can submit a near-zero-work transaction (e.g., a simple ETH transfer consuming 21,000 gas) with `gasLimit = blockGasLimit`. During `PrepareProposal` the block gas meter is incremented by the full `gasLimit`, crowding out every other transaction. The attacker's net cost is only `gasUsed × gasPrice` — the cost of the trivial operation — while the entire block is monopolised.

---

### Finding Description

**Step 1 — Ante handler allows `gasLimit == blockGasLimit`**

`CheckEthGasConsume` accumulates each message's gas limit and rejects only when the sum *strictly exceeds* the block gas limit:

```go
// ante/eth.go:155-163
gasWanted += gasLimit
if gasWanted > blockGasLimit {          // strictly greater-than: == is allowed
    return ctx, errorsmod.Wrapf(
        errortypes.ErrOutOfGas,
        "tx gas (%d) exceeds block gas limit (%d)",
        gasWanted, blockGasLimit,
    )
}
```

A transaction with `gasLimit == blockGasLimit` passes this check. [1](#0-0) 

**Step 2 — Full gas limit is charged upfront and counted against the block gas meter**

The ante handler deducts `effectiveGasPrice × gasLimit` from the sender's balance and sets the tx gas meter to `gasWanted` (the full gas limit):

```go
// ante/eth.go:196-198
newCtx := ctx.
    WithGasMeter(ethermint.NewInfiniteGasMeterWithLimit(gasWanted)).
    WithPriority(minPriority)
``` [2](#0-1) 

During `PrepareProposal`, the Cosmos SDK block gas meter is incremented by each tx's `gasWanted` (gas limit) before execution. A single tx with `gasWanted == blockGasLimit` therefore fills the entire block, preventing any other transaction from being included.

**Step 3 — Unused gas is fully refunded after execution**

In `ApplyTransaction`, after the EVM runs:

```go
// x/evm/keeper/state_transition.go:249-254
leftoverGas := msg.GasLimit - res.GasUsed

// refund gas in order to match the Ethereum gas consumption instead of the default SDK one.
if err = k.RefundGas(ctx, msg, leftoverGas, cfg.Params.EvmDenom); err != nil {
    return nil, errorsmod.Wrapf(err, "failed to refund leftover gas to sender %s", msg.From)
}
``` [3](#0-2) 

`RefundGas` sends `leftoverGas × gasPrice` back to the sender from the fee collector:

```go
// x/evm/keeper/gas.go:67-82
remaining := new(big.Int).Mul(new(big.Int).SetUint64(leftoverGas), gasPrice)
// ...
err := k.bankKeeper.SendCoinsFromModuleToAccountVirtual(ctx, authtypes.FeeCollectorName, msg.From.Bytes(), refundedCoins)
``` [4](#0-3) 

**Step 4 — Tx gas meter is reset to actual gas used**

```go
// x/evm/keeper/gas.go:92-96
func (k *Keeper) ResetGasMeterAndConsumeGas(ctx sdk.Context, gasUsed uint64) {
    ctx.GasMeter().RefundGas(ctx.GasMeter().GasConsumed(), "reset the gas count")
    ctx.GasMeter().ConsumeGas(gasUsed, "apply evm transaction")
}
``` [5](#0-4) 

This resets the *tx* gas meter to actual gas used, but the *block* gas meter was already incremented by the full `gasLimit` during proposal preparation. The block slot is consumed regardless of actual execution cost.

**Step 5 — Base fee is also manipulable via `gasWanted`**

`EndBlock` stores `gasWanted` (sum of gas limits, not actual gas used) for the next block's base fee calculation:

```go
// x/feemarket/keeper/abci.go:58-75
gasWanted := ctx.BlockGasWanted()
// ...
limitedGasWanted := sdkmath.LegacyNewDec(gw).Mul(minGasMultiplier)
gasWanted = sdkmath.LegacyMaxDec(limitedGasWanted, sdkmath.LegacyNewDec(gasUsed)).TruncateInt().Uint64()
k.SetBlockGasWanted(ctx, gasWanted)
``` [6](#0-5) 

`CalculateBaseFee` uses this stored `gasWanted` as `parentGasUsed`:

```go
// x/feemarket/keeper/eip1559.go:57
parentGasUsed := k.GetBlockGasWanted(ctx)
``` [7](#0-6) 

An attacker who fills blocks with high-`gasLimit` / low-actual-gas txs drives `gasWanted` artificially high, causing the base fee to rise and pricing out legitimate users. The `MinGasMultiplier` cap partially limits this but does not eliminate it.

---

### Impact Explanation

An unprivileged attacker can:

1. **Block-fill DoS**: Submit one EVM transaction per block with `gasLimit == blockGasLimit` and a trivial payload (e.g., ETH self-transfer, `gasUsed = 21,000`). The block gas meter is saturated during `PrepareProposal`; no other transaction is included. The attacker's net cost per block is `21,000 × gasPrice` — the cost of a simple transfer — while the entire block is monopolised.

2. **Base fee inflation**: Repeated high-`gasLimit` / low-actual-gas blocks drive `gasWanted` to the block gas limit every block, causing the base fee to increase monotonically, pricing out legitimate users.

This matches the **High** impact category: *EVM gas refund / ante handler / mempool bug that permits valid user funds/fees to be mis-accounted and enables a reachable route to block-level DoS.*

---

### Likelihood Explanation

- No special privileges required — any account with enough balance to cover `gasLimit × gasPrice` upfront (immediately refunded minus `gasUsed × gasPrice`) can execute this.
- The attack is cheap: net cost per block ≈ 21,000 gas units at the current base fee.
- The check `gasWanted > blockGasLimit` (strictly greater-than) is a single off-by-one that makes the attack trivially constructible.
- Repeatable every block with a fresh nonce.

---

### Recommendation

1. **Reject `gasLimit == blockGasLimit`**: Change the check in `CheckEthGasConsume` to `>=`:
   ```go
   if gasWanted >= blockGasLimit {
   ```
   Or enforce a configurable maximum per-tx gas limit that is a fraction of the block gas limit (e.g., ≤ 50%).

2. **Use actual gas used for block gas accounting**: After execution, update the block gas meter to reflect actual gas consumed rather than the declared gas limit, so `PrepareProposal` can pack more transactions.

3. **Charge a non-refundable minimum fee**: Require that at least a minimum fraction of the declared gas limit fee is non-refundable, making large-gas-limit / low-actual-gas attacks economically costly.

---

### Proof of Concept

```
Attacker setup:
  blockGasLimit = 30_000_000  (example)
  gasPrice      = baseFee     (minimum valid)
  gasLimit      = 30_000_000  (== blockGasLimit, passes the > check)
  payload       = simple ETH self-transfer (gasUsed ≈ 21,000)

Ante handler (CheckEthGasConsume):
  gasWanted = 30_000_000
  30_000_000 > 30_000_000 → false → tx admitted
  Fee deducted: 30_000_000 × baseFee

PrepareProposal:
  block gas meter += 30_000_000 → block is full
  No other tx can be included

ApplyTransaction:
  res.GasUsed = 21_000
  leftoverGas = 30_000_000 - 21_000 = 29_979_000
  RefundGas: 29_979_000 × baseFee returned to attacker

Net attacker cost per block: 21_000 × baseFee  (≈ cost of one ETH transfer)
Legitimate txs excluded: all of them
```

		return err
