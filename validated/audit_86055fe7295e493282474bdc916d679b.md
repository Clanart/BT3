Audit Report

## Title
`PeripheryPayments.pay()` Strands Excess Native ETH on the Router, Drainable by Any Caller via `refundETH()` - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
When a user calls `exactOutputSingle` or `exactOutput` with WETH as `tokenIn` and sends more native ETH than the swap requires, `pay()` deposits only the exact required amount as WETH and leaves the remainder on the router. Because `refundETH()` is a public, attribution-free function that unconditionally transfers the router's entire ETH balance to `msg.sender`, any third party can drain the stranded ETH in a subsequent transaction, causing a direct loss of user principal.

## Finding Description
`PeripheryPayments.pay()` handles the WETH-as-tokenIn case at lines 73–84. When `nativeBalance >= value`, exactly `value` wei is deposited and forwarded to the pool; the surplus `nativeBalance - value` remains on the router with no attribution record and no automatic refund:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) { ... }
}
```

`refundETH()` at lines 58–63 is `external payable` with no access control — it transfers the router's full ETH balance to `msg.sender`:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

The entry point `exactOutputSingle` (lines 130–147 of `MetricOmmSimpleRouter.sol`) is `external payable` and can be called directly without a multicall wrapper. It sets the callback context, calls the pool, which triggers `metricOmmSwapCallback` → `_justPayCallback` → `pay()`. After `pay()` returns, `exactOutputSingle` performs no ETH balance check and issues no refund. The surplus ETH is silently left on the router. There is no per-sender attribution, no transient lock, and no check that the caller of `refundETH()` is the same address that sent the ETH.

The existing test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` (lines 106–133 of `MetricOmmSimpleRouter.native.t.sol`) confirms that excess ETH does remain on the router after a swap and is only recovered if the user explicitly includes `refundETH` as the final step of a `multicall`. A direct call to `exactOutputSingle` provides no such protection.

## Impact Explanation
Direct loss of user ETH principal. A user who calls `exactOutputSingle{value: X}(...)` with WETH as `tokenIn` where `X` exceeds the actual swap input loses `X - actualInput` ETH to the first caller of `refundETH()`. For exact-output swaps the required input is not known in advance, so users naturally send a buffer above the expected amount. The stolen ETH is transferred out of the router in the same block with no recovery path. Severity: High.

## Likelihood Explanation
Two conditions must hold: (1) the user calls an exact-output swap with WETH as `tokenIn` and sends more native ETH than the pool requires; (2) the user does not include `refundETH()` as the final step of a `multicall` in the same transaction. Condition 1 is the normal usage pattern for exact-output swaps. Condition 2 is a documentation/UX gap — `exactOutputSingle` is `external payable` and callable directly. A front-running bot monitoring the mempool can observe the stranded ETH and call `refundETH()` in the next block. Likelihood: Medium.

## Recommendation
1. **Auto-refund in exact-output entry points.** At the end of `exactOutputSingle` and `exactOutput`, when `tokenIn == WETH`, compare `address(this).balance` before and after the swap and transfer any surplus back to `msg.sender`.
2. **Alternatively, enforce strict ETH consumption.** After `pay()` returns in the WETH branch, revert if `address(this).balance` is non-zero (i.e., `msg.value` was not fully consumed).
3. **Add access control to `refundETH`.** Record the `msg.sender` of the outermost `multicall` or swap entry point in transient storage and restrict `refundETH` to that address within the same transaction.

## Proof of Concept
```
Setup:
  - Pool: WETH / Token1
  - Alice wants exactly 1500 Token1 out; actual required input = 1 ETH

Step 1 — Alice calls exactOutputSingle directly (no multicall):
  router.exactOutputSingle{value: 2 ether}(ExactOutputSingleParams({
      tokenIn: address(weth), amountOut: 1500, amountInMaximum: 2 ether, ...
  }));

Step 2 — Inside metricOmmSwapCallback → _justPayCallback → pay():
  nativeBalance = 2 ether
  value         = 1 ether  (actual required input)
  → deposit(1 ether), transfer 1 WETH to pool
  → 1 ether remains on router, no refund, no revert

Step 3 — Bob observes the stranded ETH:
  router.refundETH();
  // _transferETH(bob, 1 ether) succeeds
  // Bob receives Alice's 1 ETH

Net result: Alice receives 1500 Token1 but loses 2 ETH instead of 1 ETH.
           Bob gains 1 ETH at zero cost.
```

The existing test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` in `metric-periphery/test/MetricOmmSimpleRouter.native.t.sol` (lines 106–133) confirms that excess ETH remains on the router after a swap and is claimable via `refundETH` — it only assumes the victim includes the refund call themselves, which is not enforced by the production entry points.
