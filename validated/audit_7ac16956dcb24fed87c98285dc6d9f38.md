Audit Report

## Title
Stranded ETH in Router Consumed by Subsequent WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` reads `address(this).balance` to decide whether to wrap native ETH for a WETH payment, with no check that the ETH was deposited by the current caller. ETH left in the router from a prior user's call (e.g., excess `msg.value` with no `refundETH`) is silently consumed to fund a subsequent user's WETH swap, with zero pull from the attacker's wallet.

## Finding Description

The WETH branch of `pay()` reads the entire contract ETH balance and, if it covers `value`, wraps and forwards it to the pool without touching `payer`: [1](#0-0) 

The `receive()` guard only blocks plain ETH transfers from non-WETH addresses; it does **not** block ETH sent as `msg.value` in payable function calls such as `multicall` or `exactInputSingle`: [2](#0-1) 

So ETH can legitimately accumulate in the router (e.g., User A calls `exactInputSingle{value: 1 ether}` with `amountIn = 0.9 ether`; the callback pays only 0.9 ETH, leaving 0.1 ETH stranded). The `payer` stored in transient context is always `msg.sender` of the swap entry-point: [3](#0-2) 

And `_justPayCallback` passes that payer directly to `pay()`: [4](#0-3) 

There is no per-caller ETH accounting. `address(this).balance` is a shared, unattributed pool. Additionally, `refundETH` sends the entire balance to `msg.sender`, not the original depositor, so a third-party call would steal rather than return the stranded ETH: [5](#0-4) 

## Impact Explanation

Direct loss of user principal. User A's ETH is irreversibly consumed to settle User B's swap obligation. User A receives nothing; User B receives full swap output while paying zero WETH from their wallet. Loss equals the stranded ETH amount, up to the full excess `msg.value` of User A's call. This is a Critical/High direct loss of user principal meeting Sherlock thresholds.

## Likelihood Explanation

The precondition — ETH stranded in the router — is realistic and reachable by any unprivileged user. Users routinely send excess `msg.value` (e.g., to cover slippage) and omit `refundETH`. An attacker can monitor the mempool for such calls and immediately follow with a WETH swap sized to exactly the stranded amount, requiring no special privileges and no malicious token behavior.

## Recommendation

Track per-call deposited ETH using transient storage. At the start of each top-level entry point, record `msg.value` as the authorized native balance for that call. In `pay()`, deduct only from that authorized amount rather than from `address(this).balance` globally, and revert if the authorized balance is insufficient. Clear the authorized balance at the end of the call. Alternatively, enforce that `address(this).balance` is zero at the start of every non-multicall entry point and that multicall tracks and enforces per-call ETH budgets.

## Proof of Concept

```solidity
// Step 1: User A sends 1 ETH, swaps 0.9 ETH worth of WETH, omits refundETH
router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
    tokenIn: WETH, tokenOut: token1, amountIn: 0.9 ether, ...
}));
// router.balance == 0.1 ETH (User A's stranded ETH)

// Step 2: Attacker (User B) calls exactInputSingle for WETH, sends 0 ETH
router.exactInputSingle{value: 0}(ExactInputSingleParams({
    tokenIn: WETH, tokenOut: token1, amountIn: 0.1 ether, ...
}));
// Inside pay(WETH, UserB, pool, 0.1 ether):
//   nativeBalance = 0.1 ETH >= value = 0.1 ETH
//   → deposit{value: 0.1 ETH}() + safeTransfer(pool, 0.1 WETH)
//   → UserB pulls 0 WETH from wallet

// Result:
assert(address(router).balance == 0);   // User A's 0.1 ETH is gone
// UserB received full swap output; UserA lost 0.1 ETH with no recourse
```
