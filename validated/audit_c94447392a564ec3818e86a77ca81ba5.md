## Conflict Detection Logic — Exact Code Trace

**`Commit()` conflict check (lines 766–788):**

```go
store := s.keeper.GetState(s.ctx, obj.Address(), key)
if store != origin && store != dirty {
    return ErrStateConflict
}
```

The comment states: *"A native action wrote this slot iff the store value differs from origin."*

**`GetCommittedState` — how `originStorage` is populated:**

```go
value := s.db.keeper.GetState(s.db.ctx, s.Address(), key)
s.originStorage[key] = value
return value
```

Both the origin-population read and the conflict-check read use `s.ctx` — the same context that is updated by `ExecuteNativeAction` to point to the post-native-write cache layer.

## The Exact Failure Scenario

**Step 1 — Native action writes K=V_native:**
`ExecuteNativeAction` branches `s.cacheMS`, runs the action (writing K=V_native + MintCoins), then updates `s.ctx` to point to the new layer containing V_native. [1](#0-0) 

**Step 2 — EVM reads slot K (SLOAD):**
`GetCommittedState` reads from `s.db.ctx` (which now has V_native) and caches it:
`originStorage[K] = V_native` [2](#0-1) 

**Step 3 — EVM writes K=V_evm (SSTORE):**
`dirtyStorage[K] = V_evm` [3](#0-2) 

**Step 4 — `Commit()` conflict check:**
- `origin = V_native` (from step 2)
- `dirty = V_evm`
- `dirty != origin` → enters the check
- `store = s.keeper.GetState(s.ctx, ...)` → reads from `s.ctx` → returns **V_native**
- Condition: `store != origin && store != dirty` → `V_native != V_native` → **FALSE**
- **No conflict detected.** [4](#0-3) 

**Step 5 — `flushNativeCacheLayers()` commits the native write (including MintCoins):** [5](#0-4) 

**Step 6 — EVM dirty-storage flush overwrites V_native with V_evm:**
`value = V_evm`, `origin = V_native`, `V_evm != V_native` → `keeper.SetState(s.origCtx, addr, K, V_evm)` [6](#0-5)

## Root Cause

The invariant the comment asserts — *"store differs from origin iff a native action wrote the slot"* — is **false** when the EVM reads the slot after the native write. In that case `origin` is already set to V_native, so `store == origin` is tautologically true regardless of whether a native write occurred. The guard is blind to the exact scenario it is supposed to catch.

The fix requires comparing against the pre-StateDB value (i.e., `s.keeper.GetState(s.origCtx, ...)`) rather than `s.ctx`, so that native writes are always visible as a delta from the true baseline. [7](#0-6)

### Title
Silent overwrite of native-action storage write via EVM SLOAD-then-SSTORE bypasses conflict detection — (`x/evm/statedb/statedb.go`)

### Summary
When a native action writes storage slot K=V_native and the EVM subsequently SLOADs K (caching V_native as `originStorage[K]`), then SSTOREs K=V_evm, the `Commit()` conflict guard computes `store == origin == V_native` and concludes no conflict exists. The EVM dirty-flush then overwrites V_native with V_evm, silently discarding the native action's storage write while its bank-balance side effects (e.g., MintCoins) are already committed.

### Finding Description
`Commit()` detects native-vs-EVM storage conflicts by checking whether `store != origin`, where `store` is read from `s.ctx` and `origin` is `originStorage[key]` — also populated from `s.ctx`. [8](#0-7) 

Because `ExecuteNativeAction` advances `s.ctx` to the post-native-write cache layer, any subsequent `GetCommittedState` call reads the native-written value and stores it as `origin`. [9](#0-8) 

At conflict-check time, `s.keeper.GetState(s.ctx, ...)` returns the same V_native that is already in `origin`, making `store != origin` permanently false for this slot. The guard never fires, and the EVM dirty-flush at lines 836–842 overwrites V_native with V_evm. [6](#0-5) 

### Impact Explanation
A precompile that (a) writes a storage slot K=V_native and (b) mints coins within a single `ExecuteNativeAction` call, followed by an EVM contract that SLOADs K then SSTOREs K=V_evm, produces committed state where:
- The bank balance reflects the mint (coins exist)
- The storage slot holds V_evm (the native write is gone)

This is committed state corruption with an orphaned mint: real tokens are created but the storage invariant that was supposed to gate or record the mint is silently erased. Any precompile that uses storage to track minting authority or supply caps is fully bypassed.

### Likelihood Explanation
Exploitability requires a precompile that writes both a storage slot and performs a bank operation in the same `ExecuteNativeAction`. This is a realistic pattern for supply-tracking or access-control precompiles. An unprivileged user submits a normal EVM transaction calling such a precompile followed by an SLOAD+SSTORE on the same slot — no special privileges required.

### Recommendation
Replace `s.ctx` with `s.origCtx` in the conflict-check `GetState` call so the baseline is always the pre-StateDB committed state, not the post-native-write cache:

```go
// Before (buggy):
store := s.keeper.GetState(s.ctx, obj.Address(), key)

// After (correct):
store := s.keeper.GetState(s.origCtx, obj.Address(), key)
```

This ensures `store != origin` is true whenever a native action wrote the slot, regardless of whether the EVM read the slot before or after the native write. [10](#0-9) 

### Proof of Concept
```
1. Deploy contract C with storage slot K initially = 0x0.
2. Precompile P: ExecuteNativeAction { SetState(K, 0x1); MintCoins(user, 1000) }
3. Contract C calls precompile P → K=0x1 in s.ctx, 1000 coins minted.
4. Contract C executes: val = SLOAD(K)  // returns 0x1; originStorage[K]=0x1
5. Contract C executes: SSTORE(K, 0x2) // dirtyStorage[K]=0x2
6. Commit():
   origin=0x1, dirty=0x2, store=GetState(s.ctx,K)=0x1
   store != origin → 0x1 != 0x1 → FALSE → no ErrStateConflict
   flushNativeCacheLayers() → 1000 coins committed
   dirty loop: 0x2 != 0x1 → SetState(origCtx, K, 0x2)
7. Final state: K=0x2 (native write 0x1 gone), bank +1000 coins (mint kept).
   Assert K==0x1 fails → invariant broken.
```

	// A nested native action (via ExecuteNativeAction) commits its writes into s.cacheMS

	// store value visible through s.ctx will differ from originStorage. Detecting this
	// before flushing means we can abort cleanly — the parent context is never touched.
	//
	// Note: only EVM-dirty keys are checked; native-only writes have no EVM dirty bit
	// and are not in scope.
	for _, addr := range s.journal.sortedDirties() {
		obj, exist := s.stateObjects[addr]
		if !exist || obj.selfDestructed {
			continue
		}

func (s *stateObject) GetCommittedState(key common.Hash) common.Hash {
	if s.overrideStorage != nil {
		if value, ok := s.overrideStorage[key]; ok {
			return value
		}
		return common.Hash{}
	}

		return value
	}
	// If no live objects are available, load it from keeper

	return value
