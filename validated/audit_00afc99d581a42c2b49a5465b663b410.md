### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezing of Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary `class_hash` from the caller without verifying that the hash corresponds to a previously declared contract class. This is directly analogous to the `preSign` bug: just as `preSign` accepted any `orderUid` without validating the embedded `receiver`/`tokenOut`/`minOut` fields, `execute_replace_class` accepts any `class_hash` without validating it against the set of declared classes. The missing check is even acknowledged by a developer TODO comment in the code. An attacker-controlled contract can exploit this to permanently freeze funds by replacing its class with an undeclared hash, rendering the contract permanently uncallable.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads `class_hash` directly from the syscall request and writes it into the contract state without any validation:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    let request = cast(syscall_ptr + RequestHeader.SIZE, ReplaceClassRequest*);
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
``` [1](#0-0) 

The developer TODO at line 898 explicitly acknowledges the missing check: `// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.` [2](#0-1) 

By contrast, when a class is declared via `execute_declare_transaction`, the OS enforces that the class hash is a valid Sierra class hash pre-image and stores it in `contract_class_changes` with `prev_value=0` (ensuring uniqueness):

```cairo
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [3](#0-2) 

The `execute_replace_class` function never cross-references `contract_class_changes` to confirm the supplied `class_hash` was ever declared. Any felt value — including `0xdeadbeef` or any random value — is accepted and committed to state.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a contract's class hash is set to an undeclared value, the contract becomes permanently uncallable: any future invocation will fail at class resolution because no class body exists for that hash. All assets held by the contract — ERC-20 token balances stored in its storage, or any other value — become permanently inaccessible with no recovery path at the protocol level.

Concrete scenario:
1. Attacker deploys a contract that accepts user token deposits.
2. Users deposit funds.
3. Attacker calls a function in the contract that internally invokes the `replace_class` syscall with an arbitrary undeclared hash (e.g., `1`).
4. The OS writes the invalid class hash to `contract_state_changes` without any validation.
5. The contract is now permanently non-callable.
6. All deposited funds are permanently frozen.

This is a rug-vector at the protocol level: the OS is supposed to enforce that `replace_class` can only point to a valid declared class, but it does not.

---

### Likelihood Explanation

**Medium-High.**

- The `replace_class` syscall is callable by any contract, reachable by any unprivileged transaction sender who can deploy a contract.
- No privileged role, leaked key, or external dependency is required.
- The missing check is explicitly acknowledged in the source code with a TODO comment dated `01/01/2026`, confirming the developers are aware it is absent.
- A malicious contract author can deliberately exploit this. A buggy contract can accidentally trigger it.

---

### Recommendation

In `execute_replace_class`, before writing the new `class_hash` to `contract_state_changes`, verify that the hash exists as a key in `contract_class_changes` (i.e., it was previously declared). Concretely:

```cairo
// Verify the new class hash has been declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the validation already enforced during `execute_declare_transaction`, where `prev_value=0` guarantees a class is declared at most once and `assert_not_zero(compiled_class_hash)` guarantees the compiled hash is non-zero. [4](#0-3) 

---

### Proof of Concept

1. **Declare a malicious contract class** that contains an entry point `__execute__` which calls the `replace_class` syscall with `class_hash = 0x1` (an undeclared hash).
2. **Deploy** an instance of this contract and have users deposit ERC-20 tokens into it.
3. **Invoke** the contract's `__execute__` entry point. The OS dispatches to `execute_replace_class`:
   - `request.class_hash = 0x1` is read from the syscall segment.
   - The TODO-guarded check is absent; no lookup into `contract_class_changes` occurs.
   - `dict_update` writes `class_hash=0x1` into `contract_state_changes` for the contract address.
4. The block is proven and finalized. The contract's on-chain class hash is now `0x1`.
5. Any subsequent `call_contract` or `invoke` targeting this contract fails at class resolution — the class body for `0x1` does not exist.
6. All ERC-20 balances stored in the contract's storage are permanently inaccessible. [5](#0-4) [6](#0-5)
