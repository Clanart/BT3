### Title
Meta-Transaction Signature Replayability: `execute_meta_tx_v0` Omits Nonce from Hash and Skips Nonce Consumption — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall constructs a transaction hash that does not include a nonce, and the OS explicitly skips nonce checking for version-0 transactions. As a result, any signature that was valid for a given `(contract_address, calldata)` pair remains valid forever. An unprivileged attacker who observes a prior meta-tx signature on-chain can replay it an unlimited number of times to re-execute the target account's `__execute__` entry point with the same calldata, bypassing all replay protection.

---

### Finding Description

**Root cause 1 — Hash excludes nonce:**

`compute_meta_tx_v0_hash` in `transaction_hash.cairo` computes the hash as:

```
H(INVOKE_HASH_PREFIX, version=0, contract_address, selector, calldata, max_fee=0, chain_id, additional_data_size=0)
```

`additional_data_size=0` means **no nonce is committed to the hash**. The hash is a pure function of `(contract_address, selector, calldata, chain_id)`. [1](#0-0) 

**Root cause 2 — TxInfo nonce hardcoded to 0, no nonce consumed:**

`execute_meta_tx_v0` in `syscall_impls.cairo` constructs the new `TxInfo` with `nonce=0` hardcoded and never calls `check_and_increment_nonce`. [2](#0-1) 

**Root cause 3 — Version-0 explicitly bypasses nonce enforcement:**

`check_and_increment_nonce` in `execute_transaction_utils.cairo` contains an explicit early-return for version 0, so even if it were called, no nonce would be checked or incremented. [3](#0-2) 

**Root cause 4 — `__validate__` is also skipped for version 0:**

`run_validate` skips the account's `__validate__` entry point for version-0 transactions, so the account contract has no opportunity to enforce its own replay protection at the validate stage. [4](#0-3) 

**Combined effect:** The meta-tx hash is static for a fixed `(contract_address, calldata)` pair. A signature that verified once will verify again on every replay, with no on-chain state change (nonce increment) to invalidate it.

---

### Impact Explanation

**Critical — Direct loss of funds.**

An account contract's `__execute__` entry point is called directly (bypassing `__validate__`). If the calldata encodes a token transfer (e.g., a multicall that calls an ERC-20 `transfer`), replaying the meta-tx with the same signature and calldata re-executes the transfer. An attacker who extracts the signature from a prior block can drain the account by replaying the call repeatedly within a single block or across blocks, limited only by the account's balance and gas.

The `execute_meta_tx_v0` syscall is reachable by any unprivileged actor: they deploy a contract that issues the `meta_tx_v0` syscall with the replayed signature, then invoke it via a standard `INVOKE_FUNCTION` transaction. [5](#0-4) 

---

### Likelihood Explanation

**High.** Signatures submitted in prior meta-tx calls are visible in the transaction calldata on-chain. Any observer can extract them. The replay requires only deploying a trivial contract that issues the `meta_tx_v0` syscall — no privileged access, no leaked keys, no Sybil attack. The attacker controls `contract_address`, `calldata`, and `signature` fields of the request entirely. [6](#0-5) 

---

### Recommendation

1. **Include a nonce in the meta-tx hash.** `compute_meta_tx_v0_hash` must commit to a per-use nonce (e.g., the caller account's current nonce, or a dedicated meta-tx nonce stored in the target contract's state).

2. **Consume the nonce on use.** After executing a meta-tx, the OS must increment the relevant nonce in `contract_state_changes` so the same hash cannot be re-signed or re-used.

3. **Alternatively, bind the meta-tx hash to the outer transaction hash.** Including the parent transaction's hash in the meta-tx hash would make each meta-tx unique per outer transaction, preventing cross-transaction replay.

---

### Proof of Concept

1. Alice submits a v3 `INVOKE_FUNCTION` transaction whose `__execute__` issues a `meta_tx_v0` syscall targeting Bob's account contract with calldata encoding `transfer(attacker, 1000_STRK)`. The OS computes `H = H(INVOKE_PREFIX, 0, bob_addr, __execute__, calldata, 0, chain_id)`. Alice's signature `σ` over `H` is embedded in the syscall request and is visible on-chain.

2. Attacker deploys `ReplayContract` with a single function that calls `meta_tx_v0(contract_address=bob_addr, selector=__execute__, calldata=<same>, signature=σ)`.

3. Attacker submits an `INVOKE_FUNCTION` calling `ReplayContract`. The OS re-computes the identical hash `H` (no nonce in hash), verifies `σ` against `H` — it passes — and calls `bob_addr.__execute__(calldata)` again, executing the transfer a second time.

4. Steps 2–3 repeat until Bob's account is drained. No nonce is ever incremented for the meta-tx path. [7](#0-6) [8](#0-7)
