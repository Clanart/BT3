### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the caller-supplied `class_hash` corresponds to a previously declared class. An unprivileged user who can trigger a `replace_class` call in any contract (e.g., through a public upgrade function) can supply an undeclared hash, permanently bricking the contract and freezing all funds it holds. The OS developers explicitly acknowledge this missing check with a TODO comment in the code.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads the new `class_hash` directly from the syscall request and immediately writes it into `contract_state_changes` without verifying that the hash exists in `contract_class_changes` (the dictionary of declared classes):

```cairo
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing check. [2](#0-1) 

By contrast, when a class is legitimately declared via `execute_declare_transaction`, it is registered into `contract_class_changes` with `prev_value=0` enforcing uniqueness:

```cairo
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [3](#0-2) 

The `execute_replace_class` function never performs a corresponding lookup into `contract_class_changes` to confirm the new hash is present. The same missing validation exists in the deprecated path: [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

If a contract's class is replaced with an undeclared hash, the OS state entry for that contract is updated to reference a class that has no corresponding compiled class in `contract_class_changes`. Any subsequent call to that contract will fail at class resolution time. Because the OS has already committed the state transition, the contract is permanently non-functional. Any ERC-20 tokens, ETH, or other assets held in the contract's storage become permanently inaccessible — a direct, irreversible loss of user funds.

---

### Likelihood Explanation

**Medium.**

The attack requires a contract that exposes a code path allowing an external caller to supply an arbitrary value to `replace_class`. This is a realistic pattern: upgradeable contracts commonly expose a public `upgrade(new_class_hash)` function. If such a function lacks access control or input validation (relying on the OS to reject invalid hashes), an unprivileged user can pass any felt value. The OS, as shown above, will accept it unconditionally. The attacker does not need any privileged role, leaked key, or special network position — only the ability to submit a standard invoke transaction.

---

### Recommendation

Before writing the new `class_hash` into `contract_state_changes`, verify that it exists as a declared class by performing a `dict_read` on `contract_class_changes` and asserting the returned compiled class hash is non-zero:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the invariant already enforced during `execute_declare_transaction` and closes the gap the TODO comment identifies.

---

### Proof of Concept

1. Deploy an upgradeable contract `Vault` that holds user funds and exposes:
   ```
   func upgrade(new_class_hash: felt) {
       replace_class(new_class_hash);   // no access control, no hash validation
   }
   ```
2. An unprivileged attacker submits an invoke transaction calling `Vault.upgrade(0xdeadbeef)`, where `0xdeadbeef` is not present in `contract_class_changes`.
3. The OS executes `execute_replace_class` at `syscall_impls.cairo:878`. The function reads `class_hash = 0xdeadbeef` from the request, skips the missing declared-class check (line 898 TODO), and calls `dict_update` to set `Vault`'s `StateEntry.class_hash = 0xdeadbeef`.
4. The state transition is committed. `Vault`'s class is now `0xdeadbeef`.
5. Any subsequent call to `Vault` (e.g., `withdraw`) fails because the OS cannot resolve `0xdeadbeef` to a compiled class.
6. All funds in `Vault`'s storage are permanently frozen.

        // Not enough gas to execute the syscall.
