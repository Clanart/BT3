### Title
Missing Class Hash Validation in `execute_replace_class` Enables Contract Re-Deployment with Pre-Populated Storage — (File: `execution/syscall_impls.cairo` + `execution/deploy_contract.cairo`)

---

### Summary

`execute_replace_class` in `syscall_impls.cairo` accepts any `class_hash` value — including `0` (`UNINITIALIZED_CLASS_HASH`) — without validation. Combined with the fact that `deploy_contract` in `deploy_contract.cairo` uses only `class_hash == 0 AND nonce == 0` as its "uninitialized" proxy (omitting a `storage_root == 0` check), an unprivileged attacker can reset a deployed contract's class hash to zero via `replace_class(0)`, then re-deploy a new contract at the same address. The re-deployed contract inherits the attacker-controlled pre-existing storage, enabling direct theft of funds.

---

### Finding Description

**Root Cause 1 — `execute_replace_class` (syscall_impls.cairo, lines 878–916):**

The function accepts the caller-supplied `class_hash` from `ReplaceClassRequest` and writes it directly to state with no validation:

```cairo
let class_hash = request.class_hash;
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}
tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);
dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO at line 898 explicitly acknowledges the missing check. There is no assertion that `class_hash != 0` or that `class_hash` is a declared class. Passing `class_hash = 0` (`UNINITIALIZED_CLASS_HASH`) is accepted silently.

**Root Cause 2 — `deploy_contract` (deploy_contract.cairo, lines 51–54):**

The initialization guard checks only `class_hash` and `nonce`:

```cairo
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}
assert state_entry.class_hash = UNINITIALIZED_CLASS_HASH;
assert state_entry.nonce = 0;
``` [2](#0-1) 

There is **no check on `storage_root`**. This is weaker than the canonical "uninitialized" definition in `get_contract_state_hash` (`commitment.cairo`, lines 55–61), which requires all three conditions — `class_hash == 0`, `storage_root == 0`, and `nonce == 0` — to return the zero (uninitialized) leaf hash:

```cairo
if (class_hash == UNINITIALIZED_CLASS_HASH) {
    if (storage_root == 0) {
        if (nonce == 0) {
            return (hash=0);
        }
    }
}
``` [3](#0-2) 

The `deploy_contract` guard is therefore an **insufficient proxy** for the uninitialized state — exactly the vulnerability class described in the external report.

---

### Impact Explanation

**Critical — Direct loss of funds.**

An attacker can:
1. Deploy contract A at address X (class_hash = C1, nonce = 0, storage = empty).
2. Call contract A (via `call_contract` from their account), causing it to:
   - Write attacker-controlled values into storage (e.g., `balance[attacker] = 10^18`).
   - Call `replace_class(0)` — resetting class_hash to `UNINITIALIZED_CLASS_HASH`.
3. Address X now has: `class_hash = 0`, `nonce = 0`, `storage = {balance_slot: 10^18}`.
4. Deploy contract B (a token/financial contract, class_hash = C2) at address X.
   - `deploy_contract` passes: `class_hash == 0` ✓, `nonce == 0`
