### Title
Birthday Attack on Truncated Hash in `deriveDeterministicAddress` Enables Permanent Freezing of ERC1155 Bridge Funds - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.deriveDeterministicAddress()` truncates a 256-bit keccak256 hash to 160 bits to produce a pseudo-address key for the `multiTokens` mapping. An attacker can exploit the birthday paradox to find a `(attackerToken, attackerTokenId)` pair whose deterministic address collides with a legitimate `(victimToken, victimTokenId)` pair, then pre-register the collision via the permissionless `logMetadata1155()`. This permanently freezes any ERC1155 tokens the victim subsequently bridges, because `finTransfer()` resolves the deterministic address to the attacker's token info and fails to release the victim's locked tokens.

---

### Finding Description

`deriveDeterministicAddress` is defined as:

```solidity
function deriveDeterministicAddress(
    address tokenAddress,
    uint256 tokenId
) public pure returns (address) {
    return
        address(
            bytes20(keccak256(abi.encodePacked(tokenAddress, tokenId)))
        );
}
``` [1](#0-0) 

The 256-bit keccak256 output is cast to `bytes20`, discarding the top 96 bits. This means for every 160-bit deterministic address, there are `2^96` distinct `(tokenAddress, tokenId)` inputs that produce it. The birthday attack complexity to find any collision in this 160-bit space is `2^80` hash evaluations.

The `multiTokens` mapping uses this 160-bit address as its key:

```solidity
mapping(address => MultiTokenInfo) public multiTokens;
``` [2](#0-1) 

`logMetadata1155()` is permissionless and writes to this mapping on first call:

```solidity
if (multiToken.tokenAddress == address(0)) {
    multiToken.tokenAddress = tokenAddress;
    multiToken.tokenId = tokenId;
} else {
    if (
        multiToken.tokenAddress != tokenAddress ||
        multiToken.tokenId != tokenId
    ) {
        revert ERC1155MappingMismatch();
    }
}
``` [3](#0-2) 

`initTransfer1155()` does **not** check or update the `multiTokens` mapping — it only computes the deterministic address and locks the ERC1155 tokens: [4](#0-3) 

`finTransfer()` resolves the deterministic address through `multiTokens` to decide which ERC1155 contract and token ID to release:

```solidity
MultiTokenInfo memory multiToken = multiTokens[payload.tokenAddress];
...
} else if (multiToken.tokenAddress != address(0)) {
    IERC1155(multiToken.tokenAddress).safeTransferFrom(
        address(this),
        payload.recipient,
        multiToken.tokenId,
        payload.amount,
        ""
    );
}
``` [5](#0-4) 

If the mapping was poisoned by the attacker's colliding pair, `finTransfer` attempts to release a token the bridge does not hold, causing a permanent revert. The nonce is already consumed at this point (`completedTransfers[payload.destinationNonce] = true`), so the transfer can never be retried. [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of user ERC1155 bridge funds.**

Once the attacker pre-registers a colliding `(attackerToken, attackerTokenId)` via `logMetadata1155()`, any victim who subsequently calls `initTransfer1155(victimToken, victimTokenId, ...)` will have their ERC1155 tokens locked in the bridge with no recovery path:

1. The victim's `logMetadata1155(victimToken, victimTokenId)` call reverts with `ERC1155MappingMismatch` (the slot is already taken by the attacker's pair).
2. `initTransfer1155` still succeeds and locks the tokens (it does not consult `multiTokens`).
3. NEAR signs a `finTransfer` referencing the deterministic address.
4. `finTransfer` resolves to the attacker's `(attackerToken, attackerTokenId)`, which the bridge does not hold, causing a permanent revert.
5. The destination nonce is consumed; the transfer is irrecoverable.

---

### Likelihood Explanation

The birthday attack requires `2^80` keccak256 evaluations to find a collision in the 160-bit output space. The attacker controls both `attackerToken` (any EVM address, deployable via CREATE2) and `attackerTokenId` (any `uint256`), giving a search space far exceeding `2^256`. The victim's `victimToken` is a fixed known address (e.g., a popular NFT contract); the attacker can vary `victimTokenId` across the token's full ID range to build the second set.

As the external report's hardware analysis shows, `2^80` evaluations are achievable with ~300 modern ASIC miners over ~1 year at a cost of ~$4–5M. For high-value NFT collections (e.g., CryptoPunks, BAYC) with significant bridged liquidity, this attack is economically rational. The attack also requires no on-chain privileges — `logMetadata1155` is fully permissionless.

---

### Recommendation

Replace the 160-bit truncated hash key with the full 256-bit keccak256 output stored in a `bytes32`-keyed mapping, eliminating the birthday-attack surface entirely:

```diff
-mapping(address => MultiTokenInfo) public multiTokens;
+mapping(bytes32 => MultiTokenInfo) public multiTokens;

 function deriveDeterministicKey(
     address tokenAddress,
     uint256 tokenId
-) public pure returns (address) {
-    return address(bytes20(keccak256(abi.encodePacked(tokenAddress, tokenId))));
+) public pure returns (bytes32) {
+    return keccak256(abi.encodePacked(tokenAddress, tokenId));
 }
```

All callers (`logMetadata1155`, `initTransfer1155`, `finTransfer`) should use the `bytes32` key. The `finTransfer` payload's `tokenAddress` field for ERC1155 tokens should carry the `bytes32` key rather than a truncated address, or a separate field should be added.

### Proof of Concept

**Setup:**
- `victimToken` = `0xBC4CA0EdA7647A8aB7C2061c2E118A18a936f13D` (BAYC contract)
- `victimTokenId` = `1337`
- `deterministicTarget` = `deriveDeterministicAddress(victimToken, 1337)`

**Offline computation (birthday attack):**

The attacker iterates over `attackerTokenId ∈ [0, 2^80)` for a fixed `attackerToken` they control, computing `bytes20(keccak256(abi.encodePacked(attackerToken, attackerTokenId)))` until it equals `deterministicTarget`. By the birthday paradox, a collision is found with ~39% probability after `2^80` evaluations.

**On-chain exploit:**

```solidity
// Step 1: Attacker pre-registers the collision (no tokens needed, logMetadata1155 is permissionless)
bridge.logMetadata1155(attackerToken, collidingTokenId);
// multiTokens[deterministicTarget] = {attackerToken, collidingTokenId}

// Step 2: Victim attempts to bridge BAYC #1337
// logMetadata1155(victimToken, 1337) → reverts ERC1155MappingMismatch
// initTransfer1155(victimToken, 1337, amount, ...) → SUCCEEDS, locks BAYC in bridge

// Step 3: NEAR signs finTransfer with tokenAddress = deterministicTarget

// Step 4: finTransfer resolves multiTokens[deterministicTarget] = {attackerToken, collidingTokenId}
//         IERC1155(attackerToken).safeTransferFrom(bridge, recipient, collidingTokenId, ...) → REVERTS
//         Nonce is consumed. BAYC #1337 is permanently locked.
```
