### Title
`batch_insert` Bypasses Key/Hash Uniqueness Checks, Corrupting DataLayer Merkle Tree State — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. Any caller — including the Python-exposed `py_batch_insert` — can supply a batch containing repeated `KeyId` or `Hash` values and silently insert them into the tree, violating the uniqueness invariant that all Merkle-proof logic depends on.

### Finding Description

`MerkleBlob::insert` enforces two uniqueness guards before touching the blob:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-373
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a completely different path. When the tree already has ≥ 2 leaves it skips the `insert` call entirely; when it has ≤ 1 leaf it calls `insert` for only the last two items popped from the vector, then falls through to a raw loop for every remaining item:

```rust
// lines 587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

No call to `block_status_cache.contains_key` or `block_status_cache.contains_leaf_hash` is made for these items. A caller can therefore insert two leaves with the same `KeyId` or the same `Hash` without receiving any error.

The Python binding `py_batch_insert` is a direct, unprivileged entry point:

```rust
// lines 1503-1518
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [3](#0-2) 

### Impact Explanation

Inserting duplicate `KeyId` values produces two leaf nodes with the same key in the tree. The `block_status_cache` key-to-index map stores only one index per key, so one of the two leaves becomes unreachable through normal lookup while still contributing to internal-node hashes. This means:

- The computed Merkle root no longer faithfully represents the key-value set.
- An inclusion proof for the "visible" leaf is valid against a root that also encodes a hidden, attacker-chosen second leaf — enabling forged inclusion/exclusion proofs.
- `check_integrity` will detect the mismatch between `leaf_count` and `key_to_index_cache_length`, but only if explicitly called; the tree is silently corrupt until then.

This matches the **High** allowed impact: *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.*

### Likelihood Explanation

The Python binding is a public API. Any DataLayer operator or integrator who calls `batch_insert` with a repeated key — whether accidentally or maliciously — triggers the corruption. No privileged access is required.

### Recommendation

Add the same uniqueness guards at the top of `batch_insert` (or inside the raw-insert loop) that `insert` already enforces:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(&hash) {
    return Err(Error::HashAlreadyPresent());
}
```

Alternatively, refactor the fast path to reuse the validated `insert` call for every item, accepting the small performance cost in exchange for invariant safety.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(blob=bytearray())

# Pre-populate so leaf_count > 1 (bypasses the insert() path entirely)
for i in range(2):
    blob.insert(KeyId(i), ValueId(i), hashlib.sha256(str(i).encode()).digest())

dup_key   = KeyId(99)
dup_value = ValueId(99)
dup_hash  = hashlib.sha256(b"dup").digest()

# batch_insert with the same key twice — no error raised
blob.batch_insert(
    [(dup_key, dup_value), (dup_key, ValueId(100))],
    [dup_hash, hashlib.sha256(b"dup2").digest()],
)

# Tree now contains two leaves with KeyId(99); root is corrupt.
# check_integrity() will raise, proving the invariant was violated.
blob.check_integrity()  # expected to fail
``` [4](#0-3)

    pub fn batch_insert(
        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {
        // OPT: perhaps go back to taking an iterator?
        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),
                };

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {
            // OPT: can we avoid this extra min height leaf traversal?
            let min_height_leaf = self.get_min_height_leaf()?;
            self.insert_subtree_at_key(min_height_leaf.key, indexes[0], Side::Left)?;
        }

        Ok(())
    }
