### Title
`ProofOfInclusion::valid()` Final Root-Hash Check Is a Tautology — Forged DataLayer Inclusion Proofs Always Pass - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate verifies only the internal self-consistency of a proof's layer chain. Its final check compares `existing_hash` against `self.root_hash()`, but `root_hash()` is derived from the proof itself (the last layer's `combined_hash`), not from any external trusted root. After the loop, `existing_hash` is guaranteed to equal the last `layer.combined_hash` (the loop already enforced this), so the final comparison is always `last.combined_hash == last.combined_hash` — a tautology that is always `true`. An attacker can forge a `ProofOfInclusion` for any arbitrary `node_hash` that will pass `valid()`.

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()   // ← tautology
}
``` [1](#0-0) 

And `root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← derived from the proof itself
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

**Invariant broken:** After the loop body, `existing_hash` has been set to `calculated_hash`, which was just asserted to equal `layer.combined_hash`. Therefore, after the last iteration, `existing_hash == last.combined_hash`. `self.root_hash()` also returns `last.combined_hash`. The final check `existing_hash == self.root_hash()` reduces to `last.combined_hash == last.combined_hash`, which is unconditionally `true`.

The function never compares the proof's root against any externally-trusted root hash. An attacker can construct a `ProofOfInclusion` with:
- Any chosen `node_hash` (the leaf they want to falsely prove is included)
- Any chain of `ProofOfInclusionLayer` values where each `combined_hash` is correctly computed from the previous hash and `other_hash` (trivially satisfiable by the attacker)

Such a forged proof will pass `valid()` and will report any attacker-chosen `root_hash()`.

The struct is `Streamable` and exposed via Python bindings with `PyStreamable`, meaning it can be deserialized from untrusted network bytes and its `valid()` method called directly from Python. [3](#0-2) 

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

A DataLayer client that receives a `ProofOfInclusion` from an untrusted peer and calls `proof.valid()` — the only validation method the API provides — will accept the proof as valid regardless of whether the claimed `node_hash` is actually present in any real tree. The attacker controls both the `node_hash` and the resulting `root_hash()`. The client has no way to distinguish a genuine proof from a forged one using only the `valid()` API.

### Likelihood Explanation

The `valid()` method is the sole validation entry point exposed to Python callers via `py_valid()`. Its name strongly implies it performs a complete proof verification. DataLayer clients that receive proofs over the network and call `proof.valid()` without separately comparing `proof.root_hash()` against a blockchain-committed root will be fully deceived. The API design actively encourages this mistake by not accepting a trusted root as a parameter. [4](#0-3) 

### Recommendation

`valid()` must accept a trusted root hash as a parameter and compare against it, rather than deriving the root from the proof itself:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // compare against external trusted root
}
```

The current `valid()` / `py_valid()` should be removed or deprecated to prevent misuse.

### Proof of Concept

```rust
use chia_datalayer::{Hash, ProofOfInclusion, ProofOfInclusionLayer, Side};
use chia_datalayer::calculate_internal_hash;

// Attacker wants to forge a proof that node_hash_victim is in some tree
let node_hash_victim: Hash = [0xAA; 32];
let other_hash: Hash = [0xBB; 32];

// Attacker computes a valid combined_hash for one layer
let combined_hash = calculate_internal_hash(&node_hash_victim, Side::Left, &other_hash);

let forged_proof = ProofOfInclusion {
    node_hash: node_hash_victim,
    layers: vec![ProofOfInclusionLayer {
        other_hash_side: Side::Left,
        other_hash,
        combined_hash,   // attacker controls this
    }],
};

// valid() returns true — forged proof accepted
assert!(forged_proof.valid());
// root_hash() returns attacker-chosen combined_hash, not any real tree root
assert_eq!(forged_proof.root_hash(), combined_hash);
```

The loop sets `existing_hash = calculated_hash = combined_hash`, then the final check is `combined_hash == combined_hash` — always `true`. [1](#0-0)
