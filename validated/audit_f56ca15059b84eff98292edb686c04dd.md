Audit Report

## Title
Stranded ETH on Router Consumed for Subsequent Caller's WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay` uses `address(this).balance` — the router's total native ETH balance — when deciding how to fund a WETH payment. Any ETH left on the router from a prior user's unrefunded `msg.value` is silently consumed to fund a later user's WETH swap, violating the invariant that each user's payment is funded exclusively from their own assets.

## Finding Description
`exactInputSingle` is `payable` and stores `msg.sender` as the payer in transient storage via `_setNextCallbackContext`. [1](#0-0) 

The pool's swap callback fires `_justPayCallback`, which calls `pay(_getTokenToPay(), _getPayer(), msg.sender, value)` where `_getPayer()` returns the original `msg.sender`. [2](#0-1) 

Inside `pay`, when `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` — the router's global ETH balance — and uses it to fund the WETH deposit without any per-call scoping: [3](#0-2) 

```solidity
} else if (token == WETH) {
  uint256 nativeBalance = address(this).balance;
  if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
  } else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
  } else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
  }
```

`address(this).balance` is not scoped to the current call's `msg.value`. ETH deposited by a prior caller that was not refunded is indistinguishable from ETH the current caller sent. The `receive()` guard only blocks non-WETH ETH arriving via `receive()` — it does not prevent ETH accumulation from `msg.value` in payable calls. [4](#0-3) 

`refundETH()` is a separate, optional call that is never invoked automatically at the end of `exactInputSingle`. [5](#0-4) 

**Exploit flow:**
1. User A calls `exactInputSingle{value: 1 ether}(tokenIn=WETH, amountIn=0.5 ether)`. `pay` sees `address(this).balance == 1 ether >= 0.5 ether`, wraps 0.5 ETH, sends WETH to pool. 0.5 ETH remains on the router.
2. User A does not call `refundETH()`.
3. User B calls `exactInputSingle{value: 0}(tokenIn=WETH, amountIn=0.5 ether)`. `pay` sees `address(this).balance == 0.5 ether >= 0.5 ether`, wraps User A's 0.5 ETH, sends WETH to pool. User B's WETH allowance is never touched.

## Impact Explanation
User A loses 0.5 ETH of principal without consent. User B receives a fully-funded WETH swap at zero cost — their WETH allowance is not spent. This is a direct loss of user principal triggered by an unprivileged caller with no special access. Severity: High.

## Likelihood Explanation
Any two sequential WETH-input swaps where the first caller sends excess ETH and omits `refundETH()` trigger this. Forgetting to bundle `refundETH` in a `multicall` is a realistic user error. An attacker can also deliberately monitor the mempool for unrefunded ETH on the router and sequence a follow-up call to drain it. No privileged access is required; the exploit is repeatable and permissionless.

## Recommendation
Track per-call ETH contribution in transient storage (set to `msg.value` at entry, cleared at exit) and limit `pay`'s native-balance draw to that per-call amount rather than `address(this).balance`. Alternatively, automatically refund excess ETH at the end of each swap entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`), or require that the native balance consumed in `pay` equals exactly the current call's `msg.value`.

## Proof of Concept
1. Deploy router with WETH address.
2. User A: `exactInputSingle{value: 1 ether}(tokenIn=WETH, amountIn=0.5 ether, ...)` — 0.5 ETH consumed for swap, 0.5 ETH stranded on router. User A does not call `refundETH()`.
3. User B: `exactInputSingle{value: 0}(tokenIn=WETH, amountIn=0.5 ether, ...)` — `pay` reads `address(this).balance == 0.5 ether >= 0.5 ether`, wraps User A's ETH, sends WETH to pool. User B's WETH allowance is untouched.
4. Assert: User A's 0.5 ETH is gone from the router; User B received a fully-funded swap at zero ETH/WETH cost.
