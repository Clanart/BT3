### Title
Single `MAX_REF_STALENESS` Applied to Both Legs of Synthetic Ratio Feeds Allows Stale Quote-Leg Price to Reach Pool Swap — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

---

### Summary

`AnchoredPriceProvider` supports a two-feed synthetic ratio mode (`quoteFeedId != 0`). Both the base leg and the quote leg are validated for staleness through the same helper `_readLeg`, which applies the single immutable `MAX_REF_STALENESS` to both. Because the two feeds can have fundamentally different update frequencies (heartbeats), a single threshold cannot be correct for both simultaneously: setting it to the slower feed's heartbeat allows the faster feed to be arbitrarily stale; setting it to the faster feed's heartbeat causes the slower feed to revert on every call.

---

### Finding Description

`_getBidAndAskPrice()` calls `_readLeg(baseFeedId)` and then `_readLeg(quoteFeedId)`:

```solidity
// AnchoredPriceProvider.sol lines 258-272
function _getBidAndAskPrice() internal returns (uint128, uint128) {
    (uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
    if (!ok) return (0, type(uint128).max);

    bytes32 _quote = quoteFeedId;
    if (_quote != bytes32(0)) {
        (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
        ...
    }
    return _computeBidAsk(mid, spreadBps);
}
```

Inside `_readLeg`, the staleness check is:

```solidity
// AnchoredPriceProvider.sol line 283
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (..., false);
```

`MAX_REF_STALENESS` is a **single immutable** set at construction time and applied identically to both legs. The factory's envelope validation (`stalenessMin`/`stalenessMax`) is keyed exclusively on `baseFeedId`'s class — there is no separate envelope or parameter for the quote leg's heartbeat.

A concrete example: `baseFeedId` = BTC/ETH (86400s heartbeat), `quoteFeedId` = ETH/USD (27s heartbeat). The deployer must set `MAX_REF_STALENESS ≥ 86400s` to avoid constant reverts on the base leg. With that setting, the ETH/USD quote leg can be up to 86400s stale without triggering `ok = false`. During a period of high ETH volatility, the stale ETH/USD price flows into the synthetic ratio:

```solidity
// AnchoredPriceProvider.sol line 267
mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);  // mid2 is stale ETH/USD
```

This produces a wrong synthetic mid-price, which then passes through `_computeBidAsk` and is returned as the pool's bid/ask. No other guard catches this: `spreadBps >= ORACLE_BPS` only catches the oracle's own stall marker, not temporal staleness; `priceGuard` bounds are set at deployment and are typically wide; `MAX_SPREAD_BPS` checks oracle uncertainty, not age.

---

### Impact Explanation

A stale quote-leg price produces a wrong synthetic ratio bid/ask that reaches the pool swap. An informed trader who observes the real market price has moved significantly (e.g., ETH dropped 5% in the last hour while the oracle is 86400s stale) can swap against the pool at the stale (wrong) price, extracting value directly from LP principal. This is a **bad-price execution** impact: the pool's oracle-driven bid/ask is materially incorrect, and the pool's conservation invariant (trader receives no more than the oracle permits) is violated.

---

### Likelihood Explanation

- Synthetic ratio mode (`quoteFeedId != 0`) is an explicitly documented and supported deployment path in `AnchoredProviderFactory.createAnchoredProvider`.
- Feed pairs with mismatched heartbeats are common in practice (e.g., any illiquid/liquid pair like BTC/ETH + ETH/USD, or any RWA feed + ETH/USD).
- The deployer is forced into the dilemma at construction time: there is no correct single value for `MAX_REF_STALENESS` when the two feeds have different heartbeats.
- The trigger is unprivileged: any user initiating a swap through the pool causes `getBidAndAskPrice()` to be called.

---

### Recommendation

Introduce a separate staleness parameter for the quote leg. Replace the single `MAX_REF_STALENESS` with two immutables — `MAX_BASE_STALENESS` and `MAX_QUOTE_STALENESS` — and pass the appropriate one to `_readLeg` for each leg. The factory's `createAnchoredProvider` interface and envelope validation should be extended to accept and bound both parameters independently.

---

### Proof of Concept

1. Deploy `AnchoredPriceProvider` with:
   - `baseFeedId` = BTC/ETH (Chainlink heartbeat: 86400s)
   - `quoteFeedId` = ETH/USD (Chainlink heartbeat: 27s)
   - `MAX_REF_STALENESS` = 86400s (required to avoid constant base-leg reverts)

2. Advance time by 3600s (1 hour) without an ETH/USD oracle update (simulating a period of oracle inactivity or network congestion).

3. ETH price has moved 5% in real markets during this hour.

4. Call `getBidAndAskPrice()` through the pool:
   - `_readLeg(baseFeedId)`: `refTime` is recent → passes staleness check.
   - `_readLeg(quoteFeedId)`: `refTime` is 3600s old, `3600 ≤ 86400` → **passes staleness check** despite being 133× the feed's intended heartbeat.
   - `mid = mulDiv(btcEthMid, 1e8, staleEthUsdMid)` → wrong BTC/USD synthetic price.
   - Pool quotes bid/ask based on the stale price.
