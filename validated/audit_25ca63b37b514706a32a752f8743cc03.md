### Title
Unvalidated Recipient Address in `initTransfer` Causes Permanent Token Lock or Burn — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` (and `initTransfer1155`) on EVM accept any arbitrary string as `recipient` without validating that it represents a valid destination-chain address. Tokens are burned or locked in the contract before any validation occurs. If the recipient string is malformed or invalid, the transfer can never be finalized on the destination chain, permanently locking or destroying user funds with no on-chain recovery path.

The same root cause exists in the Solana and StarkNet bridge entry points, and in the NEAR-side `OmniAddress::Btc`/`OmniAddress::Zcash` variants, which accept any arbitrary string.

---

### Finding Description

**EVM — `OmniBridge.initTransfer` (primary)**

`initTransfer` accepts `string calldata recipient` with zero format validation: [1](#0-0) 

Before any check on `recipient`, the function either burns bridge tokens or locks native tokens in the contract: [2](#0-1) 

The `recipient` string is then emitted in the `InitTransfer` event and consumed by the off-chain prover/relayer. If the string is not a valid `OmniAddress` (e.g., an invalid NEAR account ID, a malformed Solana base58 key, or an empty string), the prover cannot construct a valid proof, no `fin_transfer` is ever submitted to NEAR, and the burned/locked tokens are irrecoverable.

**NEAR — `OmniAddress::Btc` / `OmniAddress::Zcash` (supporting)**

`UTXOChainAddress` is a plain `String` alias: [3](#0-2) 

Both `FromStr` and `new_from_slice` for `OmniAddress::Btc`/`OmniAddress::Zcash` perform only UTF-8 validity checks — no bech32, no length, no checksum: [4](#0-3) [5](#0-4) 

The codebase itself acknowledges this in a test comment: *"The bridge doesn't validate Zcash address format — `OmniAddress::Zcash(String)` accepts anything."*

A user calling NEAR `ft_transfer_call` with `recipient: OmniAddress::Btc("INVALID")` causes the bridge to lock tokens and create a pending transfer. The BTC connector will reject the invalid address, and depending on the callback implementation, tokens may be permanently stuck.

**Solana — `InitTransferPayload.recipient: String`** [6](#0-5) 

No validation of `recipient` before tokens are burned or vaulted: [7](#0-6) 

**StarkNet — `init_transfer(recipient: ByteArray)`** [8](#0-7) 

Tokens are burned/transferred before any recipient check: [9](#0-8) 

---

### Impact Explanation

**For burned bridge tokens (EVM → NEAR with invalid recipient):** `BridgeToken.burn` destroys the tokens on EVM. The NEAR side never mints the corresponding amount because `fin_transfer` cannot be finalized with an invalid recipient. The bridge's cross-chain token supply is permanently corrupted — tokens are destroyed on EVM but no corresponding supply is released on NEAR. There is no on-chain recovery function in `OmniBridge.sol`.

**For locked native tokens (EVM → NEAR with invalid recipient):** Tokens are transferred into the `OmniBridge` contract via `safeTransferFrom`. With no `fin_transfer` ever succeeding, and no emergency withdrawal function, the tokens are permanently frozen in the contract.

This matches: *Critical — Permanent freezing, irrecoverable lock of user funds in bridge flows.*

---

### Likelihood Explanation

`initTransfer` is a public, permissionless function callable by any token holder. Cross-chain address formats (NEAR account IDs, Solana base58 keys, bech32 BTC addresses) are non-trivial and error-prone for end users. A user who mistypes a recipient, pastes an address from the wrong chain, or uses a UI with a bug will trigger this path. No privileged role or special condition is required — only a normal token approval and call.

---

### Recommendation

1. **EVM**: Add a minimum sanity check on `recipient` in `initTransfer` and `initTransfer1155` — at minimum reject empty strings. Optionally validate the `chain:address` prefix format on-chain or via a library.

2. **NEAR (`OmniAddress`)**: Add length bounds and basic format checks for `OmniAddress::Btc` and `OmniAddress::Zcash` in `FromStr` and `new_from_slice`. BTC bech32/bech32m addresses are capped at 90 characters (BIP-173); Zcash unified addresses at ~512 characters. Reject strings that cannot possibly be valid.

3. **Solana / StarkNet**: Apply the same recipient format validation before burning/locking tokens.

4. **All chains**: Consider adding an admin-accessible `recoverStuckTransfer` function as a defense-in-depth measure for cases where funds become locked due to off-chain failures.

### Proof of Concept

**EVM → NEAR permanent burn (bridge token path):**

```solidity
// 1. Attacker (or user by mistake) approves OmniBridge to spend their bridge tokens
IERC20(bridgeTokenAddress).approve(address(omniBridge), 1_000e18);

// 2. Call initTransfer with an invalid NEAR recipient (empty string, wrong format, etc.)
omniBridge.initTransfer(
    bridgeTokenAddress,
    1_000e18,   // amount
    0,          // fee
    0,          // nativeFee
    "",         // INVALID recipient — empty string, no chain prefix
    ""
);
// BridgeToken.burn(msg.sender, 1_000e18) executes here — tokens are GONE from EVM.
// The emitted InitTransfer event contains recipient = "".
// Off-chain prover cannot parse "" as a valid OmniAddress.
// No proof is ever submitted to NEAR.
// 1_000e18 tokens are permanently destroyed with no recovery path.
```

        require!(payload.amount > payload.fee, ErrorCode::InvalidFee);

        if payload.native_fee > 0 {
            transfer(
                CpiContext::new(
                    self.common.system_program.to_account_info(),
                    Transfer {
                        from: self.user.to_account_info(),
                        to: self.sol_vault.to_account_info(),
                    },
                ),
                payload.native_fee,
            )?;
        }

        if let Some(vault) = &self.vault {
            // Native version. We have a proof of token registration by vault existence
            transfer_checked(
                CpiContext::new(
                    self.token_program.to_account_info(),
                    TransferChecked {
                        from: self.from.to_account_info(),
                        to: vault.to_account_info(),
                        authority: self.user.to_account_info(),
                        mint: self.mint.to_account_info(),
                    },
                ),
                payload.amount.try_into().map_err(|_| error!(ErrorCode::InvalidArgs))?,
                self.mint.decimals,
            )?;
        } else {
            // Bridged version. May be a fake token with our authority set but it will be ignored on the near side
            require!(
                self.mint.mint_authority.contains(self.authority.key),
                ErrorCode::InvalidBridgedToken
            );

            burn(
                CpiContext::new(
                    self.token_program.to_account_info(),
                    Burn {
                        mint: self.mint.to_account_info(),
                        from: self.from.to_account_info(),
                        authority: self.user.to_account_info(),
                    },
                ),
                payload.amount.try_into().map_err(|_| error!(ErrorCode::InvalidArgs))?,
            )?;
        }

        self.common.post_message(payload.serialize_for_near((
            self.common.sequence.sequence,
            self.user.key(),
            self.mint.key(),
        ))?)?;

        Ok(())
    }
