### Title
`sweepToken` and `unwrapWETH9` accept arbitrary `recipient` with no access control, allowing anyone to drain router token balances — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.sweepToken` and `PeripheryPayments.unwrapWETH9` are `public payable` with no caller restriction and accept a caller-supplied `recipient` address. Any tokens or WETH that accumulate in `MetricOmmSimpleRouter` can be drained by an unprivileged attacker to an arbitrary address, including by front-running a legitimate recovery attempt.

---

### Finding Description

`PeripheryPayments.sol` exposes two settlement helpers that transfer the contract's entire balance of a token or WETH to a caller-specified address:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken);
    }
}

function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);
    }
}
``` [1](#0-0) 

Neither function checks `msg.sender` nor restricts `recipient`. Any caller can pass `recipient = attacker` and receive the full balance.

Tokens accumulate in the router in realistic production scenarios:

1. **Accidental direct sends** — users or integrators mistakenly transfer ERC-20 tokens (e.g., USDC, USDT) directly to the router address.
2. **Excess ETH from native WETH payment** — the `pay` helper wraps only the required ETH amount; any surplus `msg.value` stays as ETH in the contract until `refundETH()` is called. If a user's `multicall` omits the refund step, the ETH is stranded and sweepable via `unwrapWETH9` after wrapping.
3. **Multi-hop intermediate residue** — in `exactInput`, intermediate tokens are routed through `address(this)` between hops; any rounding or partial-fill residue that is not consumed by the next hop remains in the router. [2](#0-1) 

---

### Impact Explanation

**Severity: Medium**

Any tokens stranded in `MetricOmmSimpleRouter` — whether from accidental sends, ETH surplus, or intermediate residue — can be permanently stolen by an unprivileged attacker. The attacker calls:

```solidity
sweepToken(USDC, 0, attacker);   // drains all USDC
unwrapWETH9(0, attacker);        // drains all WETH → ETH
```

A victim attempting to recover their own accidentally-sent tokens by calling `sweepToken(token, 0, victim)` can be front-run: the attacker submits the same call with `recipient = attacker` at a higher gas price, draining the balance before the victim's transaction executes. The victim's transaction then silently succeeds (because `amountMinimum = 0`) while transferring nothing.

This is a direct loss of user principal with no privileged precondition required.

---

### Likelihood Explanation

- Accidental token sends to router addresses are a common user error, especially for USDC/USDT which are explicitly in scope.
- The attack requires zero setup: any EOA can call `sweepToken` or `unwrapWETH9` at any time.
- Front-running is trivially achievable on Ethereum mainnet and Base via standard mempool monitoring.

---

### Recommendation

Restrict `recipient` to `msg.sender` in both functions, matching the behavior of `refundETH()`:

```solidity
function sweepToken(address token, uint256 amountMinimum) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(msg.sender, balanceToken);
    }
}

function unwrapWETH9(uint256 amountMinimum) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(msg.sender, balanceWETH);
    }
}
```

This ensures only the caller who initiates the sweep receives the funds, eliminating the front-running vector. Update `IPeripheryPayments` and all call sites accordingly. [3](#0-2) 

---

### Proof of Concept

1. User accidentally sends 10,000 USDC to `MetricOmmSimpleRouter`.
2. User broadcasts `sweepToken(USDC, 0, user_address)` to recover funds.
3. Attacker observes the pending transaction in the mempool.
4. Attacker submits `sweepToken(USDC, 0, attacker_address)` with higher gas.
5. Attacker's transaction is mined first; attacker receives 10,000 USDC.
6. User's transaction executes: `balanceToken = 0`, `amountMinimum = 0`, no revert, no transfer — user loses all funds silently.

The same attack applies to `unwrapWETH9` for any WETH balance stranded in the router. [4](#0-3)
