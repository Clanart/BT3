### Title
DataLayer `ProofOfInclusion` Does Not Bind Proof to Key/Value ŌĆö Forged Inclusion Accepted - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

---

### Summary

`ProofOfInclusion::valid()` verifies only the internal Merkle hash chain. It does not bind the proof to any `KeyId` or `ValueId`. An untrusted party who holds a valid proof for key K2 can present it as a proof for key K1; `valid()` returns `true` and the root hash matches, so the verifier accepts forged DataLayer state.

---

### Finding Description

The `ProofOfInclusion` struct stores only two fields:

```rust
pub struct ProofOfInclusion {
    pub node_hash: Hash,
    pub layers: Vec<ProofOfInclusionLayer>,
}
``` [1](#0-0) 

`valid()` starts from `node_hash` and walks up the layer chain, verifying each `combined_hash`:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash, layer.other_hash_side, &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()
}
``` [2](#0-1) 

The proof is generated by `get_proof_of_inclusion(key: KeyId)`, which looks up the leaf by key and records `node_hash: node.hash` ŌĆö the externally-supplied leaf hash, **not** a hash of `(key, value)`):

```rust
Ok(proof_of_inclusion::ProofOfInclusion {
    node_hash: node.hash,
    layers,
})
``` [3](#0-2) 

The `LeafNode` stores `key`, `value`, and `hash` as independent fields; the leaf hash is not derived from the key or value:

```rust
pub struct LeafNode {
    pub hash: Hash,
    pub parent: Parent,
    pub key: KeyId,
    pub value: ValueId,
}
``` [4](#0-3) 

Because `ProofOfInclusion` carries no `KeyId` or `ValueId`, and `valid()` never checks them, the proof is structurally decoupled from the key it was generated for. A proof for key K2 is indistinguishable from a proof for key K1 to any verifier that only calls `proof.valid()` and `proof.root_hash()`.

This is the direct analog of the MarinateV2 NFT bug: the NFT contract tracked the NFT collection type but not the `tokenId`, allowing one staker's token to be substituted for another's. Here, the DataLayer proof tracks the Merkle hash chain but not the `KeyId`, allowing one key's proof to be substituted for another's.

---

### Impact Explanation

**High ŌĆö DataLayer Merkle proof logic lets untrusted input prove invalid state.**

A verifier that receives a `ProofOfInclusion` from an untrusted source and calls only `proof.valid()` + `proof.root_hash() == known_root` will accept the proof as valid for any key the attacker claims, as long as the attacker holds *any* valid proof rooted in the same tree. This allows:

- Proving that a key K1 is present in the tree when it is absent (using a proof for a different key K2 that is present).
- Proving that key K1 maps to hash H2 (belonging to K2) rather than its actual hash H1.

Any CLVM puzzle, Python full-node logic, or off-chain DataLayer client that uses `ProofOfInclusion.valid()` to gate state transitions is vulnerable to this substitution.

The Python binding exposes this directly:

```python
proof = merkle_blob.get_proof_of_inclusion(key)
proof.valid()   # returns True ŌĆö but for which key?
``` [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

**Medium.** The attacker must already hold a valid proof for some key in the same tree (obtainable by being a legitimate participant or by observing network traffic). No privileged access is required. The attack is triggered by submitting a serialized `ProofOfInclusion` (a `Streamable` type) over any protocol boundary ŌĆö Python API, wasm binding, or deserialized network object ŌĆö to a verifier that does not separately re-derive and check the key binding. [7](#0-6) 

---

### Recommendation

Bind the proof to its key and value. Two complementary fixes:

1. **Add `key: KeyId` and `value: ValueId` to `ProofOfInclusion`**, populated by `get_proof_of_inclusion`. Verifiers can then assert the claimed key matches the proof's key before calling `valid()`.

2. **Derive `node_hash` from `(key, value, external_hash)`** rather than storing the external hash directly as the leaf hash. This makes the hash cryptographically commit to the key-value pair, so a proof for K2 cannot be reused for K1 even if the verifier forgets to check the key field. [8](#0-7)

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib, struct

def make_hash(n):
    return bytes(hashlib.sha256(struct.pack(">q", n)).digest())

blob = MerkleBlob(blob=bytearray())

# Insert two entries
K1, V1, H1 = KeyId(1), ValueId(10), make_hash(1)
K2, V2, H2 = KeyId(2), ValueId(20), make_hash(2)
blob.insert(K1, V1, H1)
blob.insert(K2, V2, H2)
blob.calculate_lazy_hashes()

known_root = blob.get_root_hash()

# Attacker obtains a legitimate proof for K2
proof_for_k2 = blob.get_proof_of_inclusion(K2)

# Attacker presents proof_for_k2 as proof for K1
# valid() passes ŌĆö no key binding is checked
assert proof_for_k2.valid(), "proof chain valid"
assert proof_for_k2.root_hash() == known_root, "root matches"

# Verifier is deceived: they believe K1 is proven in the tree,
# but the proof actually belongs to K2.
# The proof carries node_hash=H2, not H1.
assert proof_for_k2.node_hash != bytes(H1), "wrong key's hash accepted"
print("Forged inclusion accepted for wrong key.")

    pub fn get_proof_of_inclusion(
        &self,
        key: KeyId,
    ) -> Result<proof_of_inclusion::ProofOfInclusion, Error> {
        let mut index = *self
            .block_status_cache
            .get_index_by_key(key)
            .ok_or(Error::UnknownKey(key))?;

        let node = self
            .get_node(index)?
            .expect_leaf("key to index mapping should only have leaves");

        let parents = self.get_lineage_blocks_with_indexes(index)?;
        let mut layers: Vec<proof_of_inclusion::ProofOfInclusionLayer> = Vec::new();
        let mut parents_iter = parents.iter();
        // first in the lineage is the index itself, second is the first parent
        parents_iter.next();
        for (next_index, block) in parents_iter {
            if block.metadata.dirty {
                return Err(Error::Dirty(*next_index));
            }
            let parent = block
                .node
                .expect_internal("all nodes after the first should be internal");
            let sibling_index = parent.sibling_index(index)?;
            let sibling_block = self.get_block(sibling_index)?;
            let sibling = sibling_block.node;
            let layer = proof_of_inclusion::ProofOfInclusionLayer {
                other_hash_side: parent.get_sibling_side(index)?,
                other_hash: sibling.hash(),
                combined_hash: parent.hash,
            };
            layers.push(layer);
            index = *next_index;
        }

        Ok(proof_of_inclusion::ProofOfInclusion {
            node_hash: node.hash,
            layers,
        })
    }
