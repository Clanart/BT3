### Title
Missing Declared Class Validation in `execute_replace_class` Enables Permanent Freezing of Contract Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the replacement class hash corresponds to a declared contract class. Any contract can call `replace_class` with an arbitrary undeclared class hash, causing the contract to become permanently inaccessible and freezing all funds held within it. The missing check is explicitly acknowledged in a TODO comment that is past its stated deadline.

---

### Finding Description

In `execute_replace_class` (`syscall_impls.cairo`, lines 878–916), the OS reads `class_hash` from the `ReplaceClassRequest` and directly writes it to `contract_state_changes` without any check that the hash corresponds to a declared class. The missing check is explicitly acknowledged in a TODO comment: [1](#0-0) 

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO deadline of `1/1/2026` has passed (today is 2026-07-03) and the check remains absent.

When the committed state contains a contract with an undeclared class hash, any subsequent call to that contract in a future block reaches `execute_entry_point`, which calls `find_element` on `compiled_class_facts` to look up the compiled class: [2](#0-1) 

Since no compiled class facts exist for an undeclared class hash, the proof for any future block that calls the contract will be invalid. The contract is permanently inaccessible.

The same missing check exists in the deprecated syscall handler in `deprecated_execute_syscalls.cairo`, affecting Cairo 0 contracts as well: [3](#0-2) 

---

### Impact Explanation

**Critical: Permanent freezing of funds.**

A contract that calls `replace_class` with an undeclared class hash becomes permanently inaccessible. All funds (tokens, ETH equivalents) held in the contract are frozen forever, as no valid proof can be generated for any future block that attempts to call the contract. The state transition is accepted as valid by the OS proof system, making the freeze irrecoverable at the protocol level.

---

### Likelihood Explanation

**Medium.** The attack requires a contract to call `replace_class` with an undeclared class hash. A malicious contract deployer can deploy a contract that attracts user funds (e.g., a fake DeFi vault or staking protocol) and then call `replace_class` with an arbitrary undeclared hash to freeze all user funds. The OS's missing validation means this state transition is accepted as valid by the proof system. The explicit TODO comment confirms the check is known to be missing and has not been implemented despite the deadline passing.

---

### Recommendation

Add a validation check in `execute_replace_class` (both in `syscall_impls.cairo` and `deprecated_execute_syscalls.cairo`) to verify that the new class hash corresponds to a declared contract class before updating `contract_state_changes`. Specifically, verify that the class hash exists in `contract_class_changes` (i.e., has been declared in the current or a previous block) before accepting the replacement.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract that accepts user deposits and exposes a `freeze()` function that calls `replace_class(UNDECLARED_HASH)` where `UNDECLARED_HASH` is any felt value with no corresponding declared class.
2. Users deposit funds into `MaliciousVault`.
3. Attacker calls `freeze()` on `MaliciousVault`.
4. The OS processes `replace_class(UNDECLARED_HASH)` in `execute_replace_class` at `syscall_impls.cairo:878` without checking if `UNDECLARED_HASH` is declared — the TODO check is absent.
5. The state is committed with `MaliciousVault`'s class hash set to `UNDECLARED_HASH`.
6. In the next block, any call to `MaliciousVault` reaches `execute_entry_point` at `execute_entry_point.cairo:154`, which calls `find_element` on `compiled_class_facts` for `UNDECLARED_HASH`. No entry exists; the proof is invalid for any block containing such a call.
7. All user funds in `MaliciousVault` are permanently frozen.

**Analogy to H-02:** Just as anyone could call `incrementWindow` by implementing a fake `IRoyaltyVault` interface — bypassing the caller check because the check relied on a condition anyone could satisfy — any contract can call `replace_class` with an arbitrary class hash, bypassing the missing declared-class check. In H-02, the result was unauthorized token accounting manipulation; here, the result is unauthorized and irreversible class state manipulation leading to permanent fund freezing.
