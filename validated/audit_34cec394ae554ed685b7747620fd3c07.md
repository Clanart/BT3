### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS Cairo program does not verify that the replacement class hash supplied by a contract corresponds to a previously declared class. Any contract can call `replace_class` with an arbitrary, undeclared felt value as the new class hash. The OS accepts and commits this state change. All subsequent calls to that contract will fail because the class does not exist, permanently freezing any funds held by the contract.

---

### Finding Description

`execute_replace_class` reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` without consulting `contract_class_changes` to confirm the hash is declared:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The developer-acknowledged TODO on line 898 confirms this check is intentionally absent from the current implementation. [2](#0-1) 

By contrast, `deploy_contract` correctly enforces that the deployed address has `UNINITIALIZED_CLASS_HASH` before writing a new class hash, demonstrating that the OS does enforce class-hash invariants elsewhere — just not in `replace_class`. [3](#0-2) 

The analog to the external report's blocklist issue is exact: the blocklist only checked the *source* of a transfer but not the *destination*, leaving the restriction half-enforced. Here, the OS enforces that a class hash is declared when a contract is *first deployed*, but does **not** enforce the same invariant when a contract *replaces* its class — the same restriction applied asymmetrically.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once `replace_class` commits an undeclared class hash to `contract_state_changes`, the contract's leaf in the Patricia state tree permanently records that hash. Every future call to the contract — whether via `call_contract`, `library_call`, or a direct transaction — will attempt to load the class from `contract_class_changes`. Because the hash is not present there, execution cannot proceed and the call reverts. There is no recovery path: the OS has no `unset_class` or administrative override syscall. Any ERC-20 tokens, ETH, STRK, or other assets stored in the contract's storage are irrecoverably locked.

---

### Likelihood Explanation

The attack is reachable by any unprivileged user:

1. An attacker deploys a contract (standard `deploy_account` + `deploy` flow — no privileged role required).
2. The contract exposes a function that internally calls the `replace_class` syscall with an arbitrary felt (e.g., `1` or any value not present in `contract_class_changes`).
3. Victims deposit funds into the contract (e.g., believing it is a legitimate vault or token wrapper).
4. The attacker calls the trigger function; the OS executes `execute_replace_class`, writes the undeclared hash, and the state is committed and proven.
5. All subsequent calls to the contract revert; funds are permanently frozen.

No privileged key, operator access, or network-level capability is required. The only prerequisite is the ability to deploy a contract and submit transactions — both are available to any StarkNet user.

---

### Recommendation

Before writing the new class hash to `contract_state_changes`, assert that the hash exists in `contract_class_changes` (i.e., that it was previously declared). Concretely, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the returned compiled-class-hash is non-zero (i.e., `!= UNINITIALIZED_CLASS_HASH`). This mirrors the invariant already enforced by `execute_declare_transaction`, which uses `prev_value=0` to guarantee a class is declared at most once. [4](#0-3) 

---

### Proof of Concept

```cairo
// Malicious contract (pseudocode, Sierra/Cairo 1.0 equivalent)
#[external]
fn freeze_self() {
    // 0xdeadbeef is not declared in contract_class_changes
    starknet::replace_class_syscall(0xdeadbeef_felt252).unwrap();
    // After this call returns, the contract's class_hash = 0xdeadbeef.
    // All future calls to this contract will revert.
}
```

**Steps:**
1. Declare and deploy the malicious contract above.
2. Have victim(s) transfer tokens into the contract.
3. Call `freeze_self()`.
4. The OS executes `execute_replace_class` at lines 878–915 of `syscall_impls.cairo`, writes `class_hash = 0xdeadbeef` into `contract_state_changes` with no validation against `contract_class_changes`.
5. The block is proven; the state root now encodes the undeclared class hash.
6. Any call to the contract — including attempts to withdraw funds — reverts. Funds are permanently frozen. [5](#0-4)
