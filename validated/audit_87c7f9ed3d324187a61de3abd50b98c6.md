### Title
`BouncerWeights.state_diff_size` Excludes Post-Finalization Alias-Allocation Writes, Allowing Blocks to Exceed the DA State-Diff Capacity Bound - (File: crates/blockifier/src/blockifier/transaction_executor.rs)

### Summary
`finalize_block` calls `allocate_aliases_in_storage` **after** the bouncer has finished accumulating per-transaction weights. The alias-contract storage writes produced by that call are included in the final `ThinStateDiff` (and therefore in the block hash's `concatenated_counts` / `state_diff_commitment`), but they are never reflected in `bouncer_weights.state_diff_size`. The bouncer therefore enforces its DA capacity limit against a value that is systematically smaller than the actual on-chain state-diff length, allowing any unprivileged user to push the real DA payload beyond the configured `block_max_capacity.state_diff_size`.

### Finding Description

In `finalize_block` (`crates/blockifier/src/blockifier/transaction_executor.rs`):

1. The bouncer lock is taken and its accumulated `state_diff_size` is frozen (line 239). No further calls to `Bouncer::try_update` or `Bouncer::update` occur.
2. `allocate_aliases_in_storage(block_state, alias_contract_address)` is called (line 252). This function reads the current state diff, then writes one new storage entry per new address/storage-key that qualifies for an alias (addresses > `MAX_NON_COMPRESSED_CONTRACT_ADDRESS = 0xf`), plus an update to the alias counter key. All of these writes land in `block_state`'s `CachedState` write-set.
3. `block_state.to_state_diff()?.state_maps` is called (line 273). The returned `StateMaps` **includes** the alias-contract writes from step 2.
4. `BlockExecutionSummary` is returned with `state_diff: state_diff.into()` (the full diff, including aliases) and `bouncer_weights: *bouncer.get_bouncer_weights()` (the weights that do **not** include aliases).

Downstream, `BlockExecutionArtifacts::new` calls `calculate_block_commitments` with `ThinStateDiff::from(commitment_state_diff.clone())` — the full diff — so `state_diff_length` in `concatenated_counts` and `state_diff_commitment` are both computed over the larger, correct diff. The bouncer's `state_diff_size`, however, was never updated to reflect the alias writes.

Because `Bouncer::try_update` is only called during per-transaction execution (before `finalize_block`), and `allocate_aliases_in_storage` runs entirely outside that loop, the alias-contract writes are invisible to the bouncer's capacity check.

The direct analog to the external report: `totalAllocatedTokens` tracked allocations made during the crowdfund but missed the final transfer at `finalizeCrowdfund`; here `bouncer_weights.state_diff_size` tracks state-diff entries added during transaction execution but misses the alias entries written at `finalize_block`.

### Impact Explanation

`BouncerWeights.state_diff_size` is the primary guard against exceeding the DA (data-availability) blob/calldata capacity. If the actual state diff is larger than what the bouncer tracked, the sequencer will produce and commit blocks whose on-chain DA payload exceeds the configured limit. This has direct economic impact: the sequencer pays for a larger blob than budgeted, and downstream systems (provers, sync nodes) must process a state diff that violates the advertised capacity bound. The `state_diff_length` encoded in `concatenated_counts` (part of the block hash) will correctly reflect the larger value, creating a permanent divergence between the enforced limit and the committed value.

### Likelihood Explanation

Any unprivileged user can trigger alias allocations by submitting transactions that write to new storage keys at addresses above `MAX_NON_COMPRESSED_CONTRACT_ADDRESS` (`0xf`). Each such new key causes one additional alias-contract storage write that is not counted by the bouncer. A block filled to the bouncer's `state_diff_size` limit with all-new storage keys will have an actual state diff roughly twice as large (one entry per key + one alias entry per key + the counter update). The condition `enable_stateful_compression = true` must be set, which is the production configuration.

### Recommendation

Either:
1. After calling `allocate_aliases_in_storage`, compute the incremental state-diff size contributed by the alias writes and add it to `bouncer_weights.state_diff_size` before returning `BlockExecutionSummary`; or
2. Enforce the `state_diff_size` limit against the actual post-finalization state diff length (i.e., call `block_state.to_state_diff()` before returning and compare its length against `block_max_capacity.state_diff_size`).

### Proof of Concept

1. Configure `block_max_capacity.state_diff_size = N` and `enable_stateful_compression = true`.
2. Submit transactions that collectively write to exactly `N` new storage keys at addresses > `0xf` (all qualify for alias allocation).
3. The bouncer accepts the block: accumulated `state_diff_size == N ≤ block_max_capacity.state_diff_size`.
4. `finalize_block` calls `allocate_aliases_in_storage`, which writes ≈ N alias entries + 1 counter update to the alias contract — none of which are reflected in `bouncer_weights.state_diff_size`.
5. `block_state.to_state_diff()` returns a diff of length ≈ 2N + 1.
6. `calculate_block_commitments` encodes `state_diff_length ≈ 2N + 1` into `concatenated_counts` and computes `state_diff_commitment` over the full diff.
7. The committed block's actual DA payload is ≈ 2× the configured limit, while `bouncer_weights.state_diff_size` still reads `N`. [1](#0-0) [2](#0-1) [3](#0-2)

pub fn allocate_aliases_in_storage<S: StateReader>(
    state: &mut CachedState<S>,
    alias_contract_address: ContractAddress,
) -> StateResult<()> {
    let state_diff = state.to_state_diff()?.state_maps;

    // Collect the contract addresses and the storage keys that need aliases.
    let contract_addresses: BTreeSet<ContractAddress> =
        state_diff.get_contract_addresses().into_iter().collect();
    let mut contract_address_to_sorted_storage_keys = HashMap::new();
    for (contract_address, storage_key) in state_diff.storage.keys() {
        if contract_address > &MAX_NON_COMPRESSED_CONTRACT_ADDRESS {
            contract_address_to_sorted_storage_keys
                .entry(contract_address)
                .or_insert_with(BTreeSet::new)
                .insert(storage_key);
        }
    }

    // Iterate over the addresses and the storage keys and update the aliases.
    let mut alias_updater = AliasUpdater::new(state, alias_contract_address)?;
    for contract_address in contract_addresses {
        if let Some(storage_keys) = contract_address_to_sorted_storage_keys.get(&contract_address) {
            for key in storage_keys {
                alias_updater.insert_alias(key)?;
            }
        }
        alias_updater.insert_alias(&StorageKey(contract_address.0))?;
    }

    alias_updater.finalize_updates()
}
