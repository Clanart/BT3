### Title
Missing Token-Support Validation at `init_transfer` Allows Permanent Loss of Deployed Tokens - (File: near/omni-bridge/src/lib.rs)

### Summary

`init_transfer_internal` burns deployed tokens and stores the transfer message **before** verifying that the token is registered for the destination chain. The only gate that checks token-address existence is `sign_transfer`, which runs in a later, separate transaction. If the token has no address on the destination chain, `sign_transfer` panics and the transfer is permanently stuck — with the user's tokens already burned.

### Finding Description

**Entry path** — any unprivileged user calls `ft_transfer_call` on a deployed (bridged) token, which triggers the bridge's `ft_on_transfer` → `init_transfer` → `init_transfer_internal`.

`init_transfer` performs only one validation:

```rust
// near/omni-bridge/src/lib.rs  L531-534
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
``` [1](#0-0) 

There is **no check** that the token is registered for the destination chain. Execution proceeds to `init_transfer_internal`:

```rust
// near/omni-bridge/src/lib.rs  L1850-1857
if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
    self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);   // ← tokens burned here
    self.lock_tokens_if_needed(
        transfer_message.get_destination_chain(),
        &token_id,
        transfer_message.amount.0,
    );
``` [2](#0-1) 

`burn_tokens_if_needed` fires a detached burn call — the tokens are destroyed regardless of what happens next:

```rust
// near/omni-bridge/src/lib.rs  L1806-1813
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) {
    if self.is_deployed_token(&token) {
        ext_token::ext(token)
            .with_static_gas(BURN_TOKEN_GAS)
            .burn(amount)
            .detach();   // fire-and-forget
    }
}
``` [3](#0-2) 

`lock_tokens_if_needed` → `lock_tokens` silently returns `LockAction::Unchanged` when the `(chain_kind, token_id)` key is absent from `locked_tokens`, so no error is raised:

```rust
// near/omni-bridge/src/token_lock.rs  L54-57
let key = (chain_kind, token_id.clone());
let Some(current_amount) = self.locked_tokens.get(&key) else {
    return LockAction::Unchanged;   // silent no-op
};
``` [4](#0-3) 

The `InitTransferEvent` is emitted and the transfer message is stored. Later, when a relayer calls `sign_transfer`, the first real check occurs:

```rust
// near/omni-bridge/src/lib.rs  L462-469
let token_address = self
    .get_token_address(
        transfer_message.get_destination_chain(),
        self.get_token_id(&transfer_message.token),
    )
    .unwrap_or_else(|| {
        env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
    });
``` [5](#0-4) 

`get_token_address` simply looks up `token_id_to_address`:

```rust
// near/omni-bridge/src/lib.rs  L1360-1366
pub fn get_token_address(
    &self,
    chain_kind: ChainKind,
    token: AccountId,
) -> Option<OmniAddress> {
    self.token_id_to_address.get(&(chain_kind, token))
}
``` [6](#0-5) 

If the token was never registered for the destination chain, this returns `None` and `sign_transfer` panics. The transfer message remains in `pending_transfers` indefinitely — there is no public cancel/abort function — and the tokens are already burned.

### Impact Explanation

A user who sends a deployed (bridged) token to any chain for which that token has no registered address loses their tokens permanently. The burn is irreversible (detached promise), the transfer message is unrecoverable (no public abort path), and `sign_transfer` will always panic for that transfer. This is an irrecoverable lock of user funds in the bridge flow, matching the **Critical** impact tier: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation

The entry point is fully permissionless: any holder of a deployed bridge token can call `ft_transfer_call` with an arbitrary `OmniAddress` recipient. A user who mistakenly (or experimentally) specifies a chain on which the token has not yet been deployed (e.g., sending an ETH-origin token to Solana before the Solana deployment is registered) will silently lose their funds. No privileged access, no colluding parties, and no special conditions are required.

### Recommendation

Add a token-support check inside `init_transfer` (or at the top of `init_transfer_internal`) **before** any burn or lock occurs:

```rust
let destination_chain = init_transfer_msg.get_destination_chain();
require!(
    self.get_token_address(destination_chain, token_id.clone()).is_some(),
    BridgeError::TokenNotRegistered.as_ref()
);
```

This mirrors the existing guard in `sign_transfer` but places it at the point where the user's tokens are still safe, allowing the `ft_on_transfer` to return the full amount as a refund instead of burning it.

### Proof of Concept

1. Token `eth-usdc.bridge.near` is a deployed (ETH-origin) USDC token on NEAR; it is registered for `ChainKind::Eth` but **not** for `ChainKind::Sol`.
2. User calls `ft_transfer_call` on `eth-usdc.bridge.near` with `msg = InitTransferMsg { recipient: OmniAddress::Sol(<some_address>), fee: 0, ... }`.
3. `init_transfer_internal` is reached:
   - `burn_tokens_if_needed` fires a detached `burn` call — USDC is destroyed.
   - `lock_tokens_if_needed(Sol, "eth-usdc.bridge.near", amount)` → `lock_tokens` finds no `(Sol, eth-usdc.bridge.near)` entry → returns `LockAction::Unchanged` silently.
   - `InitTransferEvent` is emitted; transfer message stored.
4. Relayer calls `sign_transfer` for this transfer ID.
5. `get_token_address(Sol, "eth-usdc.bridge.near")` returns `None` → `env::panic_str(FailedToGetTokenAddress)`.
6. Transaction reverts; transfer message remains in `pending_transfers` forever.
7. User's USDC is permanently burned with no path to recovery.
