### Title
`MerkleBlob::batch_insert()` Bypasses Duplicate-Key/Hash Validation, Corrupting Tree State and Enabling Forged Inclusion Proofs — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`MerkleBlob::batch_insert()` omits the duplicate-key and duplicate-hash guards that `insert()` enforces. When a batch containing repeated `KeyId` or `Hash` values is submitted (e.g., via the Python `batch_insert` binding), every entry is written directly into the blob and the `BlockStatusCache` without rejection. The cache silently overwrites the earlier mapping, leaving the blob with multiple leaf nodes sharing the same key while the cache tracks only the last one. The resulting tree is structurally inconsistent: proofs of inclusion are generated for the wrong node, and the computed root hash does not reflect the actual tree contents.

### Finding Description
`insert()` (lines 362–374) guards against duplicates before touching any state:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert()` (lines 578–603) calls `insert()` for at most the last two items when the tree has ≤ 1 leaf, but then processes all remaining items in a raw loop that calls `insert_entry_to_blob()` directly — with no duplicate check at all:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // no guard
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob()` (lines 1013–1027) calls `block_status_cache.add_leaf()`, which uses `HashMap::insert` — silently overwriting any prior mapping for the same key or hash:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // silent overwrite
    self.leaf_hash_to_index.insert(leaf.hash, index);   // silent overwrite
}
```

After `batch_insert` returns `Ok(())`, the blob contains two (or more) leaf nodes with the same `KeyId`, but `key_to_index` maps that key to only the last-inserted index. The tree's parent/child links connect **both** nodes into the tree structure, so the blob and the cache are permanently out of sync.

The Python binding `py_batch_insert()` (lines 1503–1519) is the direct unprivileged entry point. It only validates that the lengths of the two input lists match; it performs no intra-batch duplicate check:

```rust
pub fn py_batch_insert(
    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,
) -> PyResult<()> {
    if keys_values.len() != hashes.len() {
        Err(Error::UnmatchedKeysAndValues(...))?;
    }
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
```

### Impact Explanation
After a batch with duplicate keys is accepted:

1. **Corrupted root hash.** The tree contains phantom leaf nodes that are structurally linked but not tracked by the cache. `calculate_lazy_hashes()` propagates hashes up through all linked nodes, so the root hash incorporates the phantom leaf. Any downstream consumer that trusts the root hash (e.g., a DataLayer store root committed on-chain) now reflects a hash that does not correspond to the canonical key-value set.

2. **Forged inclusion proof.** `get_proof_of_inclusion(key)` looks up the key in `key_to_index` (which points to the last-inserted duplicate) and builds a Merkle path from that node to the root. The phantom earlier duplicate is also in the tree at a different position. A caller can craft a batch so that the "real" leaf is at one position and the phantom is at another, then use the proof path of the phantom to assert inclusion of a key-value pair that was never legitimately inserted — or to assert inclusion of a different `ValueId` for the same `KeyId`.

3. **`check_integrity()` detects the corruption only if explicitly called.** Between `batch_insert` returning `Ok` and any integrity check, the tree is silently corrupted. The Python API does not call `check_integrity()` automatically after `batch_insert`.

### Likelihood Explanation
The Python `batch_insert` API is the primary interface used by the Chia DataLayer reference implementation to bulk-load key-value pairs. Any caller that constructs the batch list programmatically and accidentally (or maliciously) includes a repeated key will trigger this path. No privileged role or key material is required — only the ability to call `MerkleBlob.batch_insert()` with a list containing a duplicate `KeyId` or `Hash`.

### Recommendation
Add the same duplicate-key and duplicate-hash guards to `batch_insert()` that `insert()` already enforces. The check should be performed **before** any call to `insert_entry_to_blob()`, either by pre-scanning the input batch for duplicates against each other and against the existing cache, or by delegating every insertion through `insert()` (accepting the performance trade-off). The Python binding should propagate the resulting error as it already does for `UnmatchedKeysAndValues`.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def h(n: int) -> bytes:
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

blob = MerkleBlob(bytearray())

# Seed with 2 legitimate entries so leaf_count > 1,
# ensuring batch_insert skips the guarded insert() path entirely.
blob.insert(KeyId(100), ValueId(100), h(100))
blob.insert(KeyId(101), ValueId(101), h(101))

# Build a batch where KeyId(1) appears twice with different ValueIds.
keys_values = [(KeyId(1), ValueId(999)), (KeyId(2), ValueId(2)),
               (KeyId(3), ValueId(3)), (KeyId(1), ValueId(42))]
hashes      = [h(999), h(2), h(3), h(42)]

# Accepted without error — duplicate KeyId(1) silently corrupts the tree.
blob.batch_insert(keys_values, hashes)

# Cache reports ValueId(42) for KeyId(1) (last write wins).
kv = blob.get_keys_values()
assert kv[KeyId(1)] == ValueId(42), "cache shows last duplicate"

# But the blob contains a phantom leaf for ValueId(999) at a different index,
# and the root hash incorporates it — the committed root is now wrong.
# check_integrity() will raise IntegrityKeyToIndexCacheLength or
# IntegrityKeyToIndexCacheIndex, confirming the corruption.
blob.check_integrity()   # raises Error

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
