### Title
Missing Nonce in `meta_tx_v0` Hash Enables Unbounded Signature Replay — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`, `transaction_hash/transaction_hash.cairo`, `execution/execute_transaction_utils.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall computes a transaction hash that contains **no nonce**, and the nonce-increment logic explicitly skips version-0 transactions. Any attacker who obtains a valid `meta_tx_v0` signature from a victim can replay it an unlimited number of times across different outer transactions, repeatedly executing the victim's `__execute__` entry point with the same calldata.

---

### Finding Description

**Root cause 1 — nonce absent from hash.**

`compute_meta_tx_v0_hash` calls `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`: [1](#0-0) 

The resulting hash is a pure function of `(INVOKE_PREFIX, 0, contract_address, selector, calldata, 0, chain_id)`. It is **identical** for every invocation with the same calldata, regardless of how many times it has already been executed.

**Root cause 2 — nonce check explicitly bypassed for version 0.**

`check_and_increment_nonce` returns immediately when `tx_info.version == 0`: [2](#0-1) 

**Root cause 3 — `execute_meta_tx_v0` injects `version=0, nonce=0` into the inner `TxInfo`.** [3](#0-2) 

The inner call then proceeds directly to `contract_call_helper` (targeting `__execute__`) with this nonce-free `TxInfo`, bypassing any nonce-based replay protection: [4](#0-3) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

A victim signs a `meta_tx_v0` message once (e.g., "transfer 100 STRK to Bob"). Because the hash is nonce-free and the OS never increments a nonce for version-0 inner calls, an attacker can embed the same `(contract_address, calldata, signature)` tuple in arbitrarily many distinct outer invoke transactions (each with its own valid nonce). Each outer transaction is accepted by the OS, and each one causes the victim's `__execute__` to run again with the original calldata, draining the victim's balance until it is exhausted.

---

### Likelihood Explanation

**High.** The attacker only needs:
1. A single valid `meta_tx_v0` signature from the victim (obtainable off-chain, e.g., via a dApp that uses meta-transactions).
2. The ability to submit ordinary invoke transactions — available to any unprivileged network participant.

No privileged access, key compromise, or Sybil attack is required. The outer transactions each have unique nonces, so they are all accepted by the sequencer.

---

### Recommendation

1. **Include a nonce in the `meta_tx_v0` hash.** Pass `additional_data_size=1` and `additional_data=&nonce` in `compute_meta_tx_v0_hash`, analogously to how `compute_l1_handler_transaction_hash` includes the L1→L2 nonce: [5](#0-4) 

2. **Enforce and increment the nonce for meta_tx_v0 calls** in the contract state, or alternatively enforce a per-contract replay-protection map keyed on the meta-tx hash.

3. **Do not unconditionally skip nonce handling for version 0** without an explicit, audited justification for each call site.

---

### Proof of Concept

```
1. Victim V signs meta_tx_v0 message:
     hash = H(INVOKE_PREFIX, 0, V_addr, __execute__, calldata="transfer(100,Bob)", 0, chain_id)
     sig  = ECDSA_sign(V_privkey, hash)

2. Attacker A submits outer invoke tx (nonce=N):
     calls contract C.foo() which internally calls syscall meta_tx_v0{
         contract_address = V_addr,
         selector         = __execute__,
         calldata         = "transfer(100,Bob)",
         signature        = sig,
     }
   → OS computes same hash, V.__execute__ verifies sig, transfer executes.

3. Attacker A submits outer invoke tx (nonce=N+1) with identical inner meta_tx_v0 data.
   → Same hash, same sig still valid (no nonce in hash, no nonce incremented for V).
   → Transfer executes again.

4. Repeat until V's balance is zero.
```

The outer transactions are distinct (different nonces), so the sequencer accepts all of them. The inner `meta_tx_v0` carries no replay protection of its own.
