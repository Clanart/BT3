Now I have enough context to analyze the analog. Let me look at one more critical detail — whether `chunk.shard_id()` is ever compared to the `shard_id` parameter inside `set_state_header`.

### Title
State-sync header `shard_id` never validated against the embedded chunk's own `shard_id` in `set_state_header` — (File: chain/chain/src/state_sync/adapter.rs)

---

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a caller-supplied `shard_id` and a peer-supplied `ShardStateSyncResponseHeader` that embeds a `ShardChunk`. The function validates the chunk's internal consistency and its Merkle inclusion in the block, but **never asserts that `chunk.shard_id() == shard_id`**. A malicious peer can therefore respond to a state-sync header request for shard A with a structurally valid header whose embedded chunk belongs to shard B. The header is stored under `StateHeaderKey(shard_id_A, sync_hash)`, and when `set_state_finalize` later applies it, shard B's transactions and receipts are executed against shard A's trie, permanently corrupting the syncing node's state root for shard A.

---

### Finding Description

**Root cause — missing shard-id binding check in `set_state_header`**

`set_state_header` performs five checks on the incoming header:

1. `validate_chunk_proofs` — verifies the chunk's internal hash, tx-root, and receipts-root.
2. `verify_path(sync_prev_block_header.chunk_headers_root(), chunk_proof, ChunkHashHeight(...))` — verifies the chunk hash appears somewhere in the block's chunk-headers Merkle tree.
3. `verify_path` for `prev_chunk_header`.
4. Receipt-proof chain validation.
5. `validate_state_root_node`. [1](#0-0) 

None of these checks compare `chunk.shard_id()` to the `shard_id` parameter. The Merkle-inclusion check (step 2) uses the plain `verify_path` function: [2](#0-1) 

`verify_path` only reconstructs the root from the path and the item hash; it does **not** verify which leaf index the path encodes. The index-aware variant `verify_path_with_index` exists and is used elsewhere (e.g. in `validate_part`), but is not used here. [3](#0-2) 

Consequently, a chunk from shard B with a valid Merkle proof for shard B's position passes the check even when `shard_id = A` is supplied.

After passing all checks, the header is stored under `StateHeaderKey(shard_id, sync_hash)`: [4](#0-3) 

**Propagation into `set_state_finalize`**

`Chain::set_state_finalize` retrieves the stored header by `(shard_id, sync_hash)` and passes it to `ChainUpdate::set_state_finalize`: [5](#0-4) 

`ChainUpdate::set_state_finalize` extracts the chunk from the header, derives `shard_uid` from the **parameter** `shard_id` (not from the chunk's own `shard_id`), and applies the chunk's transactions and receipts to that `shard_uid`'s trie: [6](#0-5) 

There is no assertion that `chunk.shard_id() == shard_id` anywhere in this call chain.

**Attacker-controlled data path**

The `shard_state_header` originates from a peer over the network. The `StateSyncDownloader` downloads it and forwards it to `StateHeaderValidationRequest`, which calls `set_state_header`: [7](#0-6) [8](#0-7) 

Any peer that advertises itself as a state-sync provider (via `SnapshotHostInfo`) can be selected to serve the header. This is an unprivileged role — no validator key or special permission is required.

---

### Impact Explanation

**Impact: High**

A syncing node that accepts a cross-shard header will:

1. Store shard B's chunk under shard A's `StateHeaderKey`.
2. Apply shard B's transactions and receipts to shard A's trie, producing a wrong `new_root`.
3. Persist the corrupted `ChunkExtra` (wrong state root) for shard A.
4. Diverge from the canonical chain on every subsequent block that touches shard A.

If the node is a chunk producer for shard A, it will produce chunks with an invalid `prev_state_root`, which will be rejected by validators and may trigger slashing. If it is a validator, its attestations will be wrong. The corruption is permanent — there is no self-healing path without a full re-sync.

---

### Likelihood Explanation

**Likelihood: Low**

- State sync only occurs at epoch boundaries when a node needs to track a new shard.
- The attacker must be reachable by the syncing node and must successfully advertise `SnapshotHostInfo` for the target `(sync_hash, shard_id)`.
- Crafting the forged header requires knowledge of the block structure (chunk hashes and Merkle paths), which is public on-chain data.
- The window of opportunity is narrow (the duration of state sync for a given shard).

---

### Recommendation

Inside `set_state_header`, immediately after extracting the chunk, assert that the chunk's embedded `shard_id` matches the `shard_id` parameter:

```rust
// After: let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} does not match \
         requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

Additionally, replace the plain `verify_path` call with `verify_path_with_index`, supplying the shard index derived from `shard_id`, so that the Merkle proof is also bound to the correct leaf position in the chunk-headers tree.

---

### Proof of Concept

1. Syncing node N requests `StateHeaderValidationRequest { shard_id: 0, sync_hash: H }`.
2. Malicious peer M holds a valid `ShardStateSyncResponseHeaderV2` for shard 1 at the same `sync_hash` (public on-chain data).
3. M responds with that header. `set_state_header` is called with `shard_id=0` and the shard-1 header.
4. `validate_chunk_proofs` passes — the chunk is internally consistent.
5. `verify_path(chunk_headers_root, shard1_chunk_proof, ChunkHashHeight(shard1_chunk_hash, ...))` passes — the shard-1 chunk IS in the tree; the proof reconstructs the correct root.
6. No check compares `chunk.shard_id() (= 1)` to `shard_id (= 0)`.
7. The header is stored under `StateHeaderKey(shard_id=0, sync_hash=H)`.
8. `set_state_finalize(shard_id=0, sync_hash=H)` retrieves it, computes `shard_uid` for shard 0, and applies shard 1's transactions/receipts to shard 0's trie.
9. Node N's state root for shard 0 is permanently corrupted. [9](#0-8) [10](#0-9)
