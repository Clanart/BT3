### Title
Silent or Panicking Overflow in `denormalize_amount` Permanently Locks User Funds During Cross-Chain Finalization - (File: near/omni-bridge/src/lib.rs)

### Summary

`Contract::denormalize_amount` performs an unchecked `u128` multiplication when scaling amounts from destination-chain decimals back to NEAR-side decimals. For tokens whose NEAR-side `origin_decimals` significantly exceeds the bridged chain's `decimals`, a sufficiently large transfer amount causes this multiplication to overflow. Depending on whether `overflow-checks = true` is set in the workspace `Cargo.toml` (the grep confirms one match there), the result is either a panic that permanently reverts `fin_transfer_callback` — trapping the user's already-locked source-chain tokens — or a silent wrap that causes the NEAR bridge to credit a drastically incorrect (near-zero) amount.

### Finding Description

`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← unchecked multiplication
}
``` [1](#0-0) 

`Decimals.origin_decimals` is the NEAR-side decimal precision; `Decimals.decimals` is the bridged-chain precision (capped at 18 by `_normalizeDecimals` on EVM). For any token where `origin_decimals > decimals` (e.g., a NEAR token with 24 decimals bridged to an EVM chain where it appears with 18 decimals, giving `diff = 6`), the multiplication `amount * 10^diff` can exceed `u128::MAX`.

This function is called unconditionally inside `fin_transfer_callback`:

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
``` [2](#0-1) 

`init_transfer.amount.0` is the raw source-chain token amount parsed directly from the on-chain `InitTransfer` event log — for EVM it is a `uint128`, for StarkNet a `u128`. Both can be up to `u128::MAX`. [3](#0-2) 

The EVM `initTransfer` emits the raw `uint128 amount` with no normalization: [4](#0-3) 

There is no analogous overflow guard in `denormalize_amount` comparable to the explicit `amountScaled > type(uint64).max` check that exists in the reference `TrimmedAmount::trim`.

### Impact Explanation

**Scenario A — overflow-checks = true (panic path):**  
The multiplication panics, causing `fin_transfer_callback` to revert. The source-chain bridge has already locked/burned the user's tokens (EVM `initTransfer` already executed). Because the NEAR finalization always panics for this transfer, the user's tokens are **permanently and irrecoverably locked** in the source-chain bridge. There is no refund path back to the source chain.

**Scenario B — overflow-checks = false (silent wrap path):**  
The multiplication wraps to a small value. `transfer_message.amount` is set to this tiny wrapped value. `process_fin_transfer_to_near` then calls `send_tokens` with `amount_without_fee()` of the wrapped value, minting or releasing a near-zero amount to the recipient. The remainder is permanently locked in the source-chain bridge. [5](#0-4) 

Both outcomes satisfy the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows.**

### Likelihood Explanation

The overflow threshold depends on `diff = origin_decimals − decimals`:

| NEAR decimals | EVM decimals | diff | Overflow threshold (EVM units) | Human-readable threshold |
|---|---|---|---|---|
| 24 | 18 | 6 | ~3.4 × 10³² | ~3.4 × 10¹⁴ tokens |
| 30 | 18 | 12 | ~3.4 × 10²⁶ | ~3.4 × 10⁸ tokens (340 M) |
| 36 | 18 | 18 | ~3.4 × 10²⁰ | ~340 tokens |

NEAR's native token already uses 24 decimals. Any token registered with `origin_decimals ≥ 30` makes the overflow trivially reachable with ordinary transfer sizes. The bridge supports arbitrary token registration, so this is a realistic configuration. An unprivileged user triggers the bug simply by calling `initTransfer` on EVM with a sufficiently large `uint128 amount`.

### Recommendation

Replace the unchecked multiplication in `denormalize_amount` with a checked variant that reverts with a clear error before the source-chain tokens are committed:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    let factor = 10_u128.pow(diff_decimals);
    amount
        .checked_mul(factor)
        .unwrap_or_else(|| env::panic_str(BridgeError::Cast.as_ref()))
}
```

Similarly apply `checked_mul` in `denormalize_fee` which delegates to `denormalize_amount`. [6](#0-5) 

### Proof of Concept

1. Register a NEAR-native token with `origin_decimals = 30`, `decimals = 18` (diff = 12).
2. On EVM, call `initTransfer(tokenAddress, amount=400_000_000 * 10^18, fee=0, ...)`. This is 400 million tokens in 18-decimal EVM units — a plausible large holder.
3. The EVM bridge locks `400_000_000 * 10^18` tokens and emits `InitTransfer`.
4. A relayer calls `fin_transfer` on NEAR with the proof.
5. `fin_transfer_callback` calls `denormalize_amount(400_000_000 * 10^18, {decimals:18, origin_decimals:30})`.
6. Computation: `400_000_000 * 10^18 * 10^12 = 4 * 10^38 > u128::MAX ≈ 3.4 * 10^38`.
7. **Overflow**: with `overflow-checks = true` the callback panics and reverts; with `overflow-checks = false` the result wraps to `≈ 6.5 * 10^37 - u128::MAX ≈ small value`.
8. In either case the user's `400_000_000 * 10^18` EVM tokens are permanently locked with no recovery path. [1](#0-0) [7](#0-6)
