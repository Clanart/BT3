### Title
`fin_transfer_send_tokens_callback` Does Not Revert State on Failed Token Transfer — (`near/omni-bridge/src/lib.rs`)

### Summary

In the NEAR bridge contract, when finalizing an inbound transfer to a NEAR recipient, critical state mutations (`finalised_transfers` insertion and `locked_tokens` decrement) are performed **before** the actual token transfer promise. The callback `fin_transfer_send_tokens_callback` is supposed to revert these mutations on failure, but it only does so when `is_refund_required` returns `true`. Due to two gaps in `is_refund_required`, a failed token transfer (promise result `Failed`) is silently treated as a success, leaving state permanently desynced and the recipient's funds irrecoverably lost.

### Finding Description

The `process_fin_transfer_to_near` function performs state mutations before dispatching the token transfer:

1. `add_fin_transfer` inserts the transfer ID into `finalised_transfers` (line 1875)
2. `unlock_tokens_if_needed` decrements `locked_tokens` (lines 1881–1885)
3. `send_tokens` dispatches the actual `ft_transfer` / `mint` promise (lines 1957–1966)
4. `.then(fin_transfer_send_tokens_callback(...))` is chained to handle the result (lines 1967–1977) [1](#0-0) [2](#0-1) 

The callback delegates to `is_refund_required` to decide whether to revert: [3](#0-2) 

There are two paths where `is_refund_required` returns `false` even on a failed promise:

**Gap 1 — Plain transfer (empty `msg`):** When `msg` is empty, `is_ft_transfer_call = false` is passed to the callback. `is_refund_required` immediately returns `false` without ever inspecting the promise result. If `ft_transfer` or `mint(recipient, amount, None)` panics (e.g., token contract paused, recipient storage unregistered after the pre-check, or bridge balance insufficient), the failure is invisible to the callback. [4](#0-3) 

**Gap 2 — `ft_transfer_call` promise failure:** When `msg` is non-empty, `is_ft_transfer_call = true`. If the promise itself fails (NEAR `PromiseResult::Failed`), `env::promise_result_checked` returns `Err(_)`, and `is_refund_required` returns `false` with the comment "Unexpected case: don't refund". A failed cross-contract call is not unexpected; it is a normal failure mode. [5](#0-4) 

In both gaps, the callback falls through to the `else` branch: fees are dispatched (detached, fire-and-forget), `FinTransferEvent` is emitted, but `remove_fin_transfer` is never called and `revert_lock_actions` is never called. [6](#0-5) [7](#0-6) 

The `remove_fin_transfer` function, which is the correct revert path, is only reachable when `is_refund_required` returns `true`: [8](#0-7) 

### Impact Explanation

**For non-deployed (native NEAR) tokens:** `unlock_tokens_if_needed` decrements `locked_tokens` before the transfer. If `ft_transfer` then fails, the bridge's accounting records the tokens as unlocked/released, but they remain physically in the bridge contract. The transfer ID is permanently consumed in `finalised_transfers`. There is no admin function or recovery path to re-attempt the transfer. The recipient's funds are irrecoverably locked in the bridge.

**For deployed (bridged) tokens:** `mint` is never executed (or panics before completing). No tokens reach the recipient. The transfer ID is consumed. The user's tokens on the origin chain were already burned/locked. The user permanently loses their assets.

This matches: **Critical — Permanent freezing / irrecoverable lock of user funds in bridge flows.**

### Likelihood Explanation

The failure condition requires `ft_transfer` or `mint` to return a failed promise result. Concrete triggers:

- The token contract is temporarily paused (the bridge has no pause-awareness for the token contract itself).
- The recipient unregisters their storage between the pre-check in `fin_transfer_callback` and the actual `ft_transfer` execution (NEAR async model; these are separate transactions).
- The bridge's `locked_tokens` accounting drifts (e.g., due to a separate bug), causing `ft_transfer` to fail with insufficient balance.
- A token contract upgrade introduces a transient bug.

These are low-probability but realistic events in a live bridge. The impact is irreversible, making this a high-severity finding.

### Recommendation

In `fin_transfer_send_tokens_callback`, always inspect the promise result regardless of `is_ft_transfer_call`. If the promise failed, treat it as a refund case:

```rust
pub fn fin_transfer_send_tokens_callback(...) {
    let token = self.get_token_id(&transfer_message.token);

    // Check promise result directly, regardless of transfer type
    let promise_failed = env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT).is_err();

    let refund_required = promise_failed || Self::is_refund_required(is_ft_transfer_call);

    if refund_required {
        self.burn_tokens_if_needed(...);
        self.revert_lock_actions(&lock_actions);
        self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);
        env::log_str(&OmniBridgeEvent::FailedFinTransferEvent { transfer_message }.to_log_string());
    } else {
        // send fees, emit FinTransferEvent
    }
}
```

Additionally, the `Err(_) => false` branch in `is_refund_required` should be changed to `Err(_) => true` to treat promise failures as requiring a refund.

### Proof of Concept

1. Alice initiates a transfer from EVM → NEAR for a native NEAR token (e.g., `usdc.near`).
2. A relayer calls `fin_transfer` with a valid proof. `fin_transfer_callback` → `process_fin_transfer_to_near` runs:
   - `add_fin_transfer` marks the transfer ID in `finalised_transfers`.
   - `unlock_tokens_if_needed` decrements `locked_tokens` by the transfer amount.
   - `send_tokens` dispatches `ft_transfer(alice.near, amount, None)`.
3. Between the storage pre-check and the `ft_transfer` execution, Alice unregisters her storage on `usdc.near` (or the token contract is paused by its own admin).
4. `ft_transfer` panics → promise result is `Failed`.
5. `fin_transfer_send_tokens_callback` is called with `is_ft_transfer_call = false`.
6. `is_refund_required` returns `false` immediately (line 1801).
7. Callback emits `FinTransferEvent`, sends fees to relayer — no revert.
8. Result: `finalised_transfers` permanently contains the transfer ID; `locked_tokens` is decremented; Alice's tokens remain in the bridge with no recovery path. [9](#0-8) [10](#0-9)

impl Contract {
    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.
                        amount.0 == 0
                    } else {
                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }
    }

    fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
        let storage_usage = env::storage_usage();
        self.finalised_transfers.remove(transfer_id);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(storage_owner, &storage);
        }
    }
