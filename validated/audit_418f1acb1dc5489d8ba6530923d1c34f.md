### Title
`ProofOfInclusion.valid()` Performs No External Root Validation, Enabling Forged DataLayer Inclusion Proofs — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary
The `ProofOfInclusion.valid()` method exposed through the DataLayer Python/Rust API only verifies internal hash-chain consistency within the proof struct itself. It never checks the proof's computed root against any external, authoritative tree root. An attacker who can supply a serialized `ProofOfInclusion` to any verifier that calls only `valid()` can forge a proof claiming inclusion of an arbitrary key-value pair under an arbitrary tree root, and the check will return `true`.

### Finding Description

`ProofOfInclusion` is a `Streamable` struct (deserializable from raw bytes) exposed to Python via `pyclass(get_all, from_py_object)`. Its sole public validation entry point is `valid()`:

```rust
// crates/chia-datalayer/src/merkle/proof_of_inclusion.rs  lines 40-58
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()   // ← tautology
}
```

`root_hash()` is defined as:

```rust
// lines 32-38
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // ← taken directly from the proof itself
    } else {
        self.node_hash
    }
}
```

After the loop, `existing_hash` holds the last `calculated_hash`, which the loop already asserted equals `layer.combined_hash`. `root_hash()` returns that same `combined_hash`. Therefore the final comparison `existing_hash == self.root_hash()` is a **tautology** — it is always `true` when the loop completes without returning `false`.

The method therefore only verifies that the hash chain inside the proof is self-consistent. It never compares the proof's root to any externally-known, authoritative tree root. An attacker can freely choose any `node_hash` (representing any key-value pair) and any sibling hashes, construct a chain where each `combined_hash` equals the computed internal hash of the previous step, and `valid()` will return `true` for that entirely fabricated proof.

Contrast this with the consensus-layer `validate_merkle_proof` in `crates/chia-consensus/src/merkle_tree.rs`, which correctly rejects any proof whose computed root does not match the caller-supplied authoritative root:

```rust
// lines 334-344
pub fn validate_merkle_proof(
    proof: &[u8],
    item: &[u8; 32],
    root: &[u8; 32],
) -> Result<bool, SetError> {
    let tree = MerkleSet::from_proof(proof)?;
    if tree.get_root() != *root {   // ← external root check present
        return Err(SetError);
    }
    Ok(tree.generate_proof(item)?.0)
}
```

The DataLayer `ProofOfInclusion` API lacks this equivalent guard entirely.

### Impact Explanation

Any DataLayer verifier that deserializes a `ProofOfInclusion` from an untrusted source (network peer, RPC caller, cross-process message) and calls only `proof.valid()` will accept a completely fabricated proof. The attacker controls:

- `node_hash` — can encode any `(key, value)` pair the attacker wishes to assert is present.
- `layers[*].other_hash` and `combined_hash` — freely chosen to form a self-consistent chain.
- The resulting `root_hash()` — an arbitrary value the attacker selects.

Because `ProofOfInclusion` is `Streamable` and exposed as a Python class with `from_bytes` / `from_py_object`, the attacker-controlled entry path is: craft bytes → `ProofOfInclusion.from_bytes()` → `proof.valid()` → accepted. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, letting untrusted input prove invalid state**.

### Likelihood Explanation

The `valid()` method name strongly implies complete proof validation. The Python stub (`datalayer.pyi`) exposes only `valid()` and `root_hash()` with no documentation requiring callers to separately compare `root_hash()` against the live tree root. Any DataLayer consumer that follows the natural API usage pattern — receive proof, call `valid()` — is silently vulnerable. The struct is `Streamable`, so no special construction is needed; raw bytes suffice.

### Recommendation

1. **Add an authoritative root parameter to `valid()`** (or add a separate `valid_for_root(expected: &Hash) -> bool` method) that asserts `self.root_hash() == expected` before returning `true`. This mirrors the pattern already used in `validate_merkle_proof`.

2. **Rename the current method** to `internally_consistent()` to make clear it does not constitute a security check against a known tree state.

3. **Update the Python stub and documentation** to require callers to supply the expected root when validating proofs received from untrusted sources.

### Proof of Concept

```rust
use chia_datalayer::{Hash, ProofOfInclusion, ProofOfInclusionLayer, Side};
use chia_sha2::Sha256;

// Attacker wants to forge: key K maps to value V in some tree
let fake_node_hash = Hash(/* sha256(K || V) or any 32 bytes */);
let fake_sibling   = Hash([0xAB; 32]);

// Compute a valid combined_hash so the chain is self-consistent
let mut h = Sha256::new();
h.update(b"\x02");
h.update(fake_node_hash.0);
h.update(fake_sibling.0);
let fake_combined = Hash(h.finalize().into());

let forged = ProofOfInclusion {
    node_hash: fake_node_hash,
    layers: vec![ProofOfInclusionLayer {
        other_hash_side: Side::Right,
        other_hash: fake_sibling,
        combined_hash: fake_combined,   // attacker-chosen root
    }],
};

assert!(forged.valid());   // returns true — no external root checked
// forged.root_hash() == fake_combined, which is NOT the real tree root
```
