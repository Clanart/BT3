### Title
Missing Declared Class Existence Check in `execute_replace_class` Allows Permanent Contract Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS Cairo program does not verify that the new class hash supplied by a contract actually exists as a declared class before committing the state update. This is an exact structural analog to the reported vesting vulnerability: a resource (a declared class) is assumed to exist but its existence is never enforced. A contract can replace its own class hash with an arbitrary, undeclared value. The OS will commit this invalid state to the state trie, permanently bricking the contract and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class`, the OS reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` without any cross-check against `contract_class_changes` (the dictionary of declared classes):

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The developer-authored TODO at line 898 explicitly acknowledges the missing check. The OS has access to `contract_class_changes` (the squashed dictionary of all declared classes in the block) but `execute_replace_class` does not consume it. There is no `dict_read` on `contract_class_changes` to assert `class_hash` maps to a valid compiled class hash before the state update is committed. [1](#0-0) 

Compare this to `execute_declare_transaction` in `transaction_impls.cairo`, which enforces `assert_not_zero(compiled_class_hash)` and uses `dict_update` with `prev_value=0` to guarantee a class is declared exactly once before it can be used: [2](#0-1) 

No equivalent guard exists in `execute_replace_class`.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once the OS commits a `StateEntry` with an undeclared `class_hash` to the state trie via `state_update`, the contract's on-chain state permanently points to a class that does not exist in the class trie. Every subsequent call to that contract will fail at class resolution, with no recovery path. Any ERC-20 tokens, ETH, or other assets held in that contract's storage are irrecoverably frozen. The STARK proof produced by the OS for this block will be valid (the OS enforces no constraint that was violated), so the L1 verifier will accept it and the state root will be updated accordingly. [3](#0-2) 

---

### Likelihood Explanation

Any contract can invoke the `replace_class` syscall. A malicious contract deployer deploys a contract, then triggers `replace_class` with an arbitrary felt value as the class hash. Because the OS is the proven enforcement layer and it performs no existence check, a sequencer that includes this transaction will produce a valid proof. The sequencer's off-chain blockifier may have a separate guard, but the OS — the only component whose output is cryptographically verified by L1 — does not. The missing check is confirmed by the in-code TODO dated 2026, meaning it has been known and unresolved.

---

### Recommendation

Inside `execute_replace_class`, before writing the new `StateEntry`, perform a `dict_read` on `contract_class_changes` for `class_hash` and assert the returned compiled class hash is non-zero:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the pattern already used in `execute_declare_transaction` and ensures the OS enforces class existence at the proof level, not merely at the sequencer level.

---

### Proof of Concept

1. Attacker deploys contract `C` holding user funds (e.g., an ERC-20 vault).
2. Contract `C` calls the `replace_class` syscall with `class_hash = 0xdeadbeef` (an arbitrary felt never declared on-chain).
3. The OS executes `execute_replace_class`:
   - Reads `class_hash = 0xdeadbeef` from the syscall request.
   - Skips the (missing) existence check.
   - Calls `dict_update` on `contract_state_changes`, setting `C`'s `class_hash` to `0xdeadbeef`.
4. `state_update` squashes `contract_state_changes` and commits the new state root, which now records `C.class_hash = 0xdeadbeef`.
5. The STARK proof is generated and accepted by L1.
6. All subsequent calls to `C` fail: the class `0xdeadbeef` does not exist in the class trie, so no entry point can be resolved.
7. All funds in `C`'s storage are permanently frozen with no recovery mechanism. [4](#0-3) [5](#0-4)
