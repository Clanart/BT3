Audit Report

## Title
Stranded Native ETH on Router Is Consumed by Any Subsequent WETH-Input Swap — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`PeripheryPayments.pay()` uses `address(this).balance` — the router's global, unattributed ETH pool — to cover WETH-input swap obligations. Any native ETH left on the router by a prior user (e.g., by sending excess `msg.value` without appending `refundETH()`) is silently consumed to settle a subsequent, unrelated user's WETH payment. The original depositor receives nothing in return and suffers a direct, permanent loss of principal.

## Finding Description

`exactInputSingle` (and `exactInput`, `exactOutputSingle`, `exactOutput`) are all declared `payable`, so callers may attach arbitrary `msg.value`. [1](#0-0) 

When the swap callback fires, `_justPayCallback` calls `pay(_getTokenToPay(), _getPayer(), msg.sender, value)` where `_getPayer()` is the original `msg.sender` stored in transient storage and `_getTokenToPay()` is `WETH`. [2](#0-1) 

Inside `pay()`, the WETH branch reads `address(this).balance` — the router's total live ETH balance — with no per-user accounting: [3](#0-2) 

If `nativeBalance >= value`, the router wraps and forwards that ETH to the pool without pulling any WETH from `payer`. There is no variable tracking how much of the current balance belongs to the current caller's `msg.value`.

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH attached to a payable swap call from persisting on the router across transactions. [4](#0-3) 

The `multicall` dispatcher is also payable and loops over delegatecalls, meaning ETH sent with a multicall that is not fully consumed (and for which `refundETH()` is omitted) remains on the router indefinitely. [5](#0-4) 

**Exploit flow:**
1. User A calls `exactInputSingle{value: 1 ether}` with `amountIn: 1000` and `tokenIn: WETH`. The swap consumes 1000 wei of native ETH; the remaining `1 ether − 1000 wei` is stranded on the router.
2. User B calls `exactInputSingle` (zero ETH, zero WETH approved) with `tokenIn: WETH`, `amountIn: 500`. `pay(WETH, userB, pool, 500)` fires; `nativeBalance = 1 ether − 1000 wei ≥ 500`, so the router wraps 500 wei of User A's ETH and forwards it to the pool. User B receives token output at zero cost; User A's ETH is permanently lost.

The transient-storage payer slot correctly records User B as `payer`, but `pay()` never pulls from `payer` when `nativeBalance >= value`, so the WETH approval/balance of User B is never checked. [6](#0-5) 

## Impact Explanation
Direct loss of user principal. User A's native ETH, stranded on the router, is consumed to settle User B's WETH swap obligation. User A receives nothing; User B receives a free or partially-free swap. The loss is bounded only by the stranded amount, which can be arbitrarily large (e.g., a user who sends 1 ETH for a 1000-wei swap and omits `refundETH()`). This meets the Critical/High threshold for direct loss of user principal.

## Likelihood Explanation
High. `refundETH()` is optional and must be manually composed by the caller. The test suite itself demonstrates the correct-but-unenforced pattern: [7](#0-6) 

Any user who sends excess ETH without appending `refundETH()` — a common mistake with payable multicalls — leaves exploitable residue. An attacker can monitor the router's ETH balance on-chain and immediately follow with a WETH-input swap to drain it. No special privileges are required; any unprivileged trader can execute this.

## Recommendation
Track the ETH attributable to the current transaction's `msg.value` in a dedicated transient slot (e.g., `T_SLOT_MSG_VALUE`). At each payable swap entry point, store `msg.value` in that slot. In `pay()`, deduct from the slot before falling through to `safeTransferFrom`; if the slot is exhausted, pull from `payer` directly. Clear the slot at the end of the call. This ensures only the current caller's ETH is ever used to cover their own WETH obligation.

Alternatively, require that `msg.value` equals `amountIn` exactly when `tokenIn == WETH` and revert if `address(this).balance` exceeds the declared amount at entry, preventing any residue from accumulating.

## Proof of Concept

```solidity
// Foundry test outline
function test_strandedEthConsumedByAttacker() public {
    address victim  = makeAddr("victim");
    address attacker = makeAddr("attacker");
    vm.deal(victim, 1 ether);

    // Step 1: victim sends 1 ETH but swap only needs 1000 wei; no refundETH()
    vm.prank(victim);
    router.exactInputSingle{value: 1 ether}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
            zeroForOne: true, amountIn: 1000, amountOutMinimum: 0,
            recipient: victim, deadline: type(uint256).max, priceLimitX64: 0, extensionData: ""
        })
    );
    // router.balance == 1 ether - 1000 wei (victim's ETH, stranded)
    assertGt(address(router).balance, 0);

    // Step 2: attacker has zero ETH, zero WETH approved
    assertEq(attacker.balance, 0);
    assertEq(weth.balanceOf(attacker), 0);

    uint256 attackerToken1Before = token1.balanceOf(attacker);
    vm.prank(attacker);
    router.exactInputSingle(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
            zeroForOne: true, amountIn: 500, amountOutMinimum: 0,
            recipient: attacker, deadline: type(uint256).max, priceLimitX64: 0, extensionData: ""
        })
    );

    // attacker received token1 output spending none of their own funds
    assertGt(token1.balanceOf(attacker) - attackerToken1Before, 0);
    // victim's ETH was consumed
    assertEq(address(router).balance, 1 ether - 1000 - 500);
}
```
