### Title
`OmniBridge` Permanently Locks Any ETH Sent Directly via `receive()` — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`OmniBridge` exposes a bare `receive() external payable {}` function with no logic and no corresponding ETH-withdrawal mechanism. Any ETH delivered to the contract outside of the `initTransfer` flow is permanently trapped with no recovery path.

### Finding Description
`OmniBridge.sol` declares a no-op `receive()` function at line 574, making the contract unconditionally accept raw ETH transfers. [1](#0-0) 

The only path by which ETH ever leaves the contract is `finTransfer` when `payload.tokenAddress == address(0)`, which requires a valid MPC-signed payload and a matching `destinationNonce`. [2](#0-1) 

ETH sent directly to the contract address (i.e., not through `initTransfer`) is never recorded in any nonce, mapping, or balance entry. It therefore has no corresponding signed payload that could ever authorize its release via `finTransfer`. A full search of all EVM contracts confirms there is no `withdraw`, `rescue`, or `recoverEth` function anywhere in the codebase.


`OmniBridgeWormhole`, which inherits `OmniBridge`, is equally affected and adds no recovery path. [3](#0-2) 

### Impact Explanation
Any ETH sent directly to the contract — whether by a user mistake, a wallet that resolves the contract address as a payable target, or a future protocol integration — is permanently and irrecoverably locked. Because the protocol already handles native ETH as a bridgeable asset (`tokenAddress == address(0)` in `initTransfer`/`finTransfer`), users and integrators have a reasonable expectation that the contract can hold and release ETH, making accidental direct sends plausible. The locked ETH can never be released without a contract upgrade.

This matches the allowed impact: **Permanent freezing / irrecoverable lock of user or protocol funds in bridge flows.**

### Likelihood Explanation
The entry path requires no privilege — any externally-owned account or contract can trigger `receive()` by sending ETH to the bridge address. Wallet UIs, smart-contract routers, and refund mechanisms routinely send ETH to contract addresses. The protocol's own native-ETH bridging feature increases the probability that users or integrators will send ETH directly to the contract address by mistake.

### Recommendation
Either remove the `receive()` function entirely if the contract is not intended to accept unsolicited ETH, or add an admin-gated ETH-rescue function:

```solidity
function rescueEth(address payable to, uint256 amount) external onlyRole(DEFAULT_ADMIN_ROLE) {
    (bool ok, ) = to.call{value: amount}("");
    require(ok, "ETH transfer failed");
}
```

### Proof of Concept
1. Deploy `OmniBridge` (or `OmniBridgeWormhole`).
2. Call `address(omniBridge).call{value: 1 ether}("")` from any EOA — succeeds silently.
3. Observe `address(omniBridge).balance == 1 ether`.
4. Attempt any path to recover the ETH — no function exists; the ETH is permanently locked. [1](#0-0)
