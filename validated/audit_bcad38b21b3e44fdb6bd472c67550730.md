Audit Report

## Title
`pay()` consumes entire `address(this).balance` for WETH-input swaps, enabling cross-transaction ETH theft — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` reads `address(this).balance` — the router's total native ETH balance — when settling a WETH-input swap, rather than the current call's `msg.value`. ETH sent by a prior caller via a payable entry point (e.g., `multicall`, `exactInputSingle`) that was never refunded persists in the contract and is silently consumed to fund a subsequent caller's swap. The subsequent caller pays nothing from their own wallet yet receives the full swap output.

## Finding Description

In `PeripheryPayments.sol` lines 73–84, the WETH branch of `pay()` reads `address(this).balance`:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // entire contract balance
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

This balance is not scoped to the current transaction's `msg.value`. It includes ETH deposited by any prior caller that was never refunded via `refundETH()`.

The `receive()` guard at line 32–34 (`if (msg.sender != WETH) revert NotWETH()`) prevents arbitrary direct ETH sends but does **not** prevent ETH from accumulating via `msg.value` in the payable entry points: `multicall` (line 39), `exactInputSingle` (line 67), `exactInput` (line 92), `exactOutputSingle` (line 130), and `exactOutput` (line 154) in `MetricOmmSimpleRouter`, and `multicall` (line 42) and all `addLiquidity*` variants in `MetricOmmPoolLiquidityAdder`.

`pay()` is invoked from two production callback paths:
1. `_justPayCallback` (line 192–199, `MetricOmmSimpleRouter`) — fires for all single-hop and multi-hop exact-input swaps when `tokenIn == WETH`.
2. `metricOmmModifyLiquidityCallback` (lines 172–177, `MetricOmmPoolLiquidityAdder`) — fires when either pool token is WETH.

Exploit flow:
- **Step 1**: Alice calls `router.multicall{value: 2 ether}([exactInputSingle(tokenIn=WETH, amountIn=1 ether, ...)])` without a trailing `refundETH()`. `pay()` wraps 1 ETH; 1 ETH remains in the router.
- **Step 2**: Bob calls `router.exactInputSingle{value: 0}(tokenIn=WETH, amountIn=1 ether, ...)` with no WETH approval. `pay()` reads `address(this).balance == 1 ether >= 1 ether`, wraps Alice's ETH, and delivers WETH to the pool. Bob receives the full token output; Alice's 1 ETH is unrecoverable.

The existing `refundETH()` function (lines 58–63) is opt-in and not enforced; omitting it is a common SDK/user error.

## Impact Explanation

Direct loss of user principal. Any ETH left in the router by a prior caller is permanently available to any subsequent WETH-input swap caller. The attacker receives full swap output for free, and the victim's ETH is irretrievably consumed. This meets the Critical/High threshold for direct loss of user principal with no preconditions beyond a prior caller omitting `refundETH()`.

## Likelihood Explanation

The `multicall` + `refundETH()` pattern is standard but not enforced on-chain. SDK-generated calldata frequently omits the trailing refund call. The attack requires no special role, no flash loan, no approval, and no front-running beyond standard mempool observation. Any block containing a multicall with residual ETH is immediately exploitable in the same or next block by any unprivileged address.

## Recommendation

Track the current call's ETH budget in transient storage at each payable entry point and pass it into `pay()` instead of reading `address(this).balance`. Decrement the tracked value as ETH is consumed. The corrected WETH branch should use `_getMsgValueRemaining()` (a transient slot set at entry) rather than `address(this).balance`, ensuring only ETH explicitly sent in the current transaction is eligible for wrapping.

## Proof of Concept

```
Setup:
  - MetricOmmSimpleRouter deployed with WETH = <weth>
  - Pool: WETH / TokenX

Step 1 — Alice leaves ETH in the router:
  Alice calls router.multicall{value: 2 ether}([
      abi.encodeCall(exactInputSingle, (tokenIn=WETH, amountIn=1 ether, recipient=Alice, ...))
      // no refundETH() call
  ])
  pay(WETH, Alice, pool, 1 ether):
    nativeBalance = 2 ether >= 1 ether → wraps 1 ETH, sends WETH to pool.
  1 ETH remains in router.

Step 2 — Bob steals Alice's ETH:
  Bob calls router.exactInputSingle{value: 0}(
      tokenIn=WETH, amountIn=1 ether, recipient=Bob, ...
  )
  // Bob has zero WETH approval and sends zero ETH.
  pay(WETH, Bob, pool, 1 ether):
    nativeBalance = 1 ether >= 1 ether → wraps Alice's 1 ETH, sends WETH to pool.
  Bob receives TokenX output.

Assert:
  Bob.tokenXBalance > 0
  router.balance == 0
  Alice's 1 ETH is unrecoverable
```

Foundry test: deploy router + mock WETH + mock pool; fund Alice's multicall with 2 ETH omitting `refundETH()`; call `exactInputSingle` as Bob with `value: 0` and no WETH approval; assert Bob receives output and `address(router).balance == 0`.
