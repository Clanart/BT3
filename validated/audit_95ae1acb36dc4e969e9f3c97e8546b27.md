### Title
Zero `max_price_per_unit` Allows Complete Fee Evasion in V3 Transactions — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

In the StarkNet OS Cairo program, a V3 transaction sender can set `max_price_per_unit = 0` for all three resource bounds (L1\_GAS, L2\_GAS, L1\_DATA\_GAS) and `tip = 0`. This causes `compute_max_possible_fee` to return `0`, which triggers an early-return guard in `charge_fee` that skips the ERC-20 fee transfer entirely. Because `get_initial_user_gas_bound` is derived solely from `max_amount` (not price), the transaction still receives execution gas and can produce valid state changes — all without paying any fee to the sequencer.

---

### Finding Description

**Root cause — `pack_resource_bounds` (transaction\_hash.cairo, line 105):**

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn(resource_bounds.max_price_per_unit);          // ← allows 0
    ...
}
```

`assert_nn` checks `0 ≤ x < PRIME/2`. It does **not** enforce `x > 0`. A user may therefore supply `max_price_per_unit = 0` for every resource bound, and `tip = 0` (also unconstrained below its upper bound).

**Propagation — `compute_max_possible_fee` (transaction\_impls.cairo, lines 99–101):**

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
```

With all prices and tip equal to 0, this expression evaluates to `0` regardless of the `max_amount` values.

**Fee bypass — `charge_fee` (transaction\_impls.cairo, lines 123–125):**

```cairo
if (max_fee == 0) {
    return ();          // ← fee transfer skipped entirely
}
```

**Gas still available — `get_initial_user_gas_bound` (transaction\_impls.cairo, line 77):**

```cairo
return common_tx_fields.resource_bounds[L2_GAS_INDEX].max_amount;
```

`initial_user_gas_bound` is derived from `max_amount`, not from `max_price_per_unit`. Setting `max_amount_l2 > 0` while keeping all prices at 0 gives the transaction a non-zero gas budget for execution, so validate and execute entry points run normally and state changes are committed — with zero fee paid.

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

The fee mechanism is the sole on-chain economic incentive for the sequencer to process transactions. When the OS proof system accepts zero-fee transactions as cryptographically valid:

1. Any user can submit fully-executed, state-changing transactions that pay the sequencer nothing.
2. The sequencer's fee revenue — the STRK tokens it is entitled to receive — is never transferred; this constitutes a direct, provable loss of funds per transaction.
3. At scale, sustained fee evasion eliminates the sequencer's economic viability, forcing it to halt block production and causing a total network shutdown.

The OS is the authoritative validity oracle. Off-chain mempool filtering is the only remaining guard, mirroring exactly the acknowledged-but-unmitigated posture of the GlueX team.

---

### Likelihood Explanation

**High.** Any unprivileged V3 transaction sender can craft this condition by setting three fields in their transaction's resource bounds to 0. No special privilege, leaked key, or external dependency is required. The OS imposes no minimum-price constraint anywhere in the transaction validation path.

---

### Recommendation

In `pack_resource_bounds`, replace `assert_nn` with a strict positivity check for `max_price_per_unit`:

```cairo
// Before (allows 0):
assert_nn(resource_bounds.max_price_per_unit);

// After (enforces at least 1):
assert_nn_le(1, resource_bounds.max_price_per_unit);
```

Alternatively, enforce a minimum `max_fee` floor directly in `charge_fee` or in `compute_max_possible_fee`, so that any V3 transaction with a non-zero `max_amount_l2` must also carry a non-zero price commitment. The bootstrap path (`sender_address == 'BOOTSTRAP'`) already demonstrates the pattern for intentional zero-fee exceptions and can be preserved as-is.

### Proof of Concept

A V3 invoke transaction is crafted with:

```
resource_bounds[L1_GAS]      = { max_amount: 0,    max_price_per_unit: 0 }
resource_bounds[L2_GAS]      = { max_amount: 1000, max_price_per_unit: 0 }  ← gas budget present
resource_bounds[L1_DATA_GAS] = { max_amount: 0,    max_price_per_unit: 0 }
tip = 0
```

Step-by-step OS execution:

1. `pack_resource_bounds` is called for each bound. `assert_nn(0)` passes for all three `max_price_per_unit` values. [1](#0-0) 

2. `get_initial_user_gas_bound` returns `1000` (from `max_amount_l2`). The transaction proceeds to validate and execute with a full gas budget. [2](#0-1) 

3. `compute_max_possible_fee` returns `0 * 0 + 1000 * (0 + 0) + 0 * 0 = 0`. [3](#0-2) 

4. `charge_fee` hits `if (max_fee == 0) { return (); }` and exits without executing the ERC-20 transfer. The sequencer receives 0 STRK. [4](#0-3) 

5. The OS proof is valid. The L1 verifier accepts the state transition. The transaction's effects (e.g., a token transfer, contract deployment) are finalized on-chain with no fee paid.
