Audit Report

## Title
`pay()` ignores router-held WETH balance for external-payer WETH swaps, causing payer overcharge and stealable stranded funds — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` helper in `PeripheryPayments.sol` only inspects `address(this).balance` (native ETH) when deciding how much WETH to pull from an external payer, never reading `IERC20(WETH).balanceOf(address(this))`. When a multicall step deposits WETH onto the router and a subsequent `exactInputSingle` step uses WETH as input, the router ignores its own WETH balance, pulls the full amount from the user's wallet, and leaves the earlier WETH stranded on the contract where any caller can steal it via the public, access-control-free `sweepToken`.

## Finding Description
`pay()` at lines 69–88 of `PeripheryPayments.sol` branches on `payer == address(this)` first; if false and `token == WETH`, it reads only `address(this).balance`:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← native ETH only
    if (nativeBalance >= value) { ... }
    else if (nativeBalance > 0) { ... safeTransferFrom(payer, ..., value - nativeBalance); }
    else { IERC20(WETH).safeTransferFrom(payer, recipient, value); }  // ← full pull from payer
}
``` [1](#0-0) 

`exactInputSingle` always sets `payer = msg.sender`: [2](#0-1) 

So in a multicall where step 1 is `exactInputSingle(token1 → WETH, recipient=router)` and step 2 is `exactInputSingle(WETH → token2)`, step 2's callback calls `pay(WETH, msg.sender, pool, value)`. Because `payer != address(this)`, the `payer == address(this)` fast-path is skipped. Because `address(this).balance == 0`, the code falls through to `safeTransferFrom(payer, pool, value)`, pulling the full amount from the user's wallet while the router's WETH from step 1 sits untouched.

The `payer == address(this)` guard is only set for intermediate hops inside `exactInput` (line 103), not for independent `exactInputSingle` calls inside a multicall: [3](#0-2) 

The stranded WETH is immediately drainable because `sweepToken` is `public payable` with no access control and accepts a caller-chosen `recipient`: [4](#0-3) 

## Impact Explanation
Direct loss of user principal: Alice is charged the full WETH swap amount from her own wallet even though the router already holds W WETH from a prior multicall step. The stranded W WETH is immediately stealable by any third party via `sweepToken(WETH, 0, attacker)`. Alice loses `min(W, V)` WETH where V is the required swap input. This is a Critical/High direct loss of user funds with no privileged access required.

## Likelihood Explanation
The multicall pattern of "swap token → WETH to router, then swap WETH → token2" is the standard multi-step flow explicitly tested in this codebase (`test_multicall_tokenForWeth_thenUnwrapEth`). Any user composing two `exactInputSingle` calls via multicall with WETH as an intermediate triggers the bug. An attacker only needs to monitor the mempool for a stranded WETH balance and call `sweepToken`. No privileged access, no special setup, and no non-standard token behavior is required.

## Recommendation
In the WETH branch of `pay()`, read the router's existing WETH balance and credit it before pulling from the external payer:

```solidity
} else if (token == WETH) {
    uint256 wethBalance   = IERC20(WETH).balanceOf(address(this));
    uint256 nativeBalance = address(this).balance;

    if (wethBalance >= value) {
        IERC20(WETH).safeTransfer(recipient, value);
    } else {
        uint256 remaining = value - wethBalance;
        if (wethBalance > 0) IERC20(WETH).safeTransfer(recipient, wethBalance);
        if (nativeBalance >= remaining) {
            IWETH9(WETH).deposit{value: remaining}();
            IERC20(WETH).safeTransfer(recipient, remaining);
        } else if (nativeBalance > 0) {
            IWETH9(WETH).deposit{value: nativeBalance}();
            IERC20(WETH).safeTransfer(recipient, nativeBalance);
            IERC20(WETH).safeTransferFrom(payer, recipient, remaining - nativeBalance);
        } else {
            IERC20(WETH).safeTransferFrom(payer, recipient, remaining);
        }
    }
}
```

## Proof of Concept
```
Setup:
  pool_A: token1/WETH
  pool_B: WETH/token2
  Alice holds token1 and WETH, both approved to router

Alice's multicall:
  calls[0] = exactInputSingle(token1 → WETH, amountIn=X, recipient=address(router))
             → router receives W WETH

  calls[1] = exactInputSingle(WETH → token2, amountIn=V, recipient=alice)
             → _setNextCallbackContext(..., msg.sender, WETH)
             → callback: pay(WETH, alice, pool_B, V)
             → nativeBalance = 0, wethBalance ignored
             → safeTransferFrom(alice, pool_B, V)  ← Alice charged full V from wallet
             → router's W WETH untouched

After multicall:
  router.balanceOf(WETH) == W  ← stranded

Bob (separate tx):
  sweepToken(WETH, 0, bob)  ← Bob receives W WETH

Alice's loss: W WETH
```
