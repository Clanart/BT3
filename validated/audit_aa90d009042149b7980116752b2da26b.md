### Title
`uint32 wormholeNonce` Exhaustion Permanently Bricks All Bridge Operations on Wormhole Variant - (File: evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol)

---

### Summary

`OmniBridgeWormhole` stores a `uint32 wormholeNonce` that is incremented unconditionally on every `initTransfer`, `finTransfer`, `deployToken`, and `logMetadata` call. Solidity 0.8+ reverts on arithmetic overflow. Once the counter reaches `type(uint32).max` (4,294,967,295), every subsequent call to any of those four functions permanently reverts, irrecoverably freezing the entire Wormhole-variant bridge.

---

### Finding Description

`OmniBridgeWormhole` declares a 32-bit counter: [1](#0-0) 

Every bridge operation increments it before publishing a Wormhole message. For example, in `initTransferExtension`: [2](#0-1) 

The same pattern appears in `finTransferExtension`, `deployTokenExtension`, and `logMetadataExtension`: [3](#0-2) [4](#0-3) [5](#0-4) 

Because the contract is compiled with `pragma solidity ^0.8.24`, the `++` operator on a `uint32` at its maximum value (`0xFFFFFFFF`) triggers a checked-arithmetic revert. There is no unchecked block, no wrapping, and no reset mechanism. After the 4,294,967,295th message, every call to `initTransfer`, `finTransfer`, `deployToken`, and `logMetadata` reverts unconditionally and permanently.

By contrast, the base `OmniBridge.currentOriginNonce` is a `uint64`, giving ~1.8 × 10¹⁹ values — effectively inexhaustible. The `wormholeNonce` is a separate, narrower counter that was not given the same width. [6](#0-5) 

---

### Impact Explanation

Once `wormholeNonce` overflows:

- **`initTransfer` / `initTransfer1155`** — users can no longer lock tokens and initiate cross-chain transfers; all funds already deposited in the bridge remain locked with no outbound path.
- **`finTransfer`** — incoming transfers from NEAR cannot be settled; recipients cannot claim bridged assets.
- **`deployToken`** and **`logMetadata`** — token registration and metadata propagation are permanently halted.

The bridge contract is non-upgradeable via the standard path once bricked (the overflow happens inside the extension hooks called before state changes complete). All user funds held in the vault or awaiting finalization become permanently unclaimable. This matches the allowed impact: **Critical — Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows**.

---

### Likelihood Explanation

The Wormhole variant is deployed on Arbitrum One and Base (evidenced by the OpenZeppelin upgrade manifests): [7](#0-6) [8](#0-7) 

On these chains, transaction fees are fractions of a cent. An attacker calling `initTransfer` with the minimum viable amount (1 wei of a bridge token they control, burning it each time) can increment `wormholeNonce` at near-zero marginal cost per call. At $0.001 per transaction, exhausting 2³² slots costs roughly $4.3 million — significant but within reach of a well-funded adversary targeting a high-value bridge. Organic traffic also contributes: every legitimate `finTransfer` call (which requires no token spend by the caller, only a valid MPC signature) increments the counter, accelerating exhaustion passively.

Likelihood is **Low-to-Medium** over a multi-year horizon on high-throughput L2 deployments, but the consequence is irreversible.

---

### Recommendation

Widen `wormholeNonce` from `uint32` to `uint64`:

```solidity
// Before
uint32 public wormholeNonce;

// After
uint64 public wormholeNonce;
```

Note that the Wormhole `publishMessage` interface accepts `uint32 nonce`, so the stored counter must be cast on use:

```solidity
_wormhole.publishMessage{value: value}(
    uint32(wormholeNonce),   // safe: wraps silently, Wormhole nonce is advisory only
    payload,
    _consistencyLevel
);
wormholeNonce++;
```

Because the Wormhole `nonce` parameter is used only for message batching (not for replay protection — Wormhole uses its own `sequence` for that), silent wrapping of the cast is acceptable and does not introduce a security regression. Alternatively, use `unchecked { wormholeNonce++; }` with a `uint32` if storage layout must be preserved, accepting wrap-around semantics that Wormhole already tolerates.

---

### Proof of Concept

1. Deploy `OmniBridgeWormhole` on a local fork of Arbitrum or Base.
2. Mint a bridge token to an attacker EOA.
3. In a loop, call `initTransfer(tokenAddress, 1, 0, 0, "recipient.near", "")` 4,294,967,295 times (scriptable with a Foundry fuzz/loop test or a simple script).
4. On the 4,294,967,296th call, the transaction reverts with a Solidity arithmetic overflow panic (`0x11`).
5. All subsequent calls to `initTransfer`, `finTransfer`, `deployToken`, and `logMetadata` revert permanently — the bridge is bricked. [9](#0-8) [10](#0-9)
