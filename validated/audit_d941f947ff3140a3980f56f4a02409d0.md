### Title
`batch_insert` Bypasses Key and Hash Uniqueness Validation Enforced by `insert` — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::insert` enforces two invariants before writing a leaf: no duplicate `KeyId` and no duplicate leaf `Hash`. `MerkleBlob::batch_insert` calls `insert` only for the first two items when the tree has ≤ 1 existing leaf; all remaining items are written directly to the blob via `insert_entry_to_blob` with no uniqueness checks at all. An unprivileged caller supplying a batch with duplicate keys or hashes silently corrupts the tree's internal cache, producing an incorrect Merkle root and unreliable proof-of-inclusion results.

### Finding Description

`MerkleBlob::insert` guards both uniqueness invariants before any structural mutation: [1](#0-0) 

`batch_insert` delegates only the first two items (when `leaf_count <= 1`) to `insert`, which carries those guards. Every subsequent item is written through a raw loop that calls `insert_entry_to_blob` directly: [2](#0-1) 

`insert_entry_to_blob` eventually calls `BlockStatusCache::add_leaf`, which uses `HashMap::insert` — silently overwriting any existing entry for the same key or hash: [3](#0-2) 

When a duplicate `KeyId` is supplied, the cache (`key_to_index`) points to the newer index while the older leaf with the same key remains in the blob. When a duplicate `Hash` is supplied, `leaf_hash_to_index` loses track of the first leaf. In both cases the blob and the cache diverge: `leaf_count` in the blob exceeds the number of entries in the cache maps, the tree structure is invalid, and the computed Merkle root is wrong.

`check_integrity` would detect this divergence, but it is only enabled in test builds (`check_integrity_on_drop: cfg!(test)`): [4](#0-3) 

The Python binding `py_batch_insert` accepts `keys_values` and `hashes` directly from caller-supplied Python objects with no additional filtering: [5](#0-4) 

### Impact Explanation

A corrupted `MerkleBlob` produces an incorrect tree root. Proof-of-inclusion generated from this root either fails to verify for legitimately present keys (the "hidden" duplicate leaf is unreachable via the cache) or verifies against a root that does not reflect the true committed state. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state.**

### Likelihood Explanation

Any code path that calls `batch_insert` with ≥ 3 items and supplies a repeated `KeyId` or `Hash` triggers the bug. The Python binding is directly reachable from DataLayer sync logic that processes externally-sourced key-value pairs. No privilege is required beyond the ability to supply input to the batch.

### Recommendation

Add the same uniqueness guards at the top of the bulk loop in `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insertion logic
}
```

Alternatively, refactor `batch_insert` to route every item through `insert` (accepting the performance cost) or maintain a local seen-set for the batch before committing any writes.

### Proof of Concept

```rust
let mut blob = MerkleBlob::new(vec![]).unwrap();

// Pre-populate two leaves so leaf_count > 1, causing batch_insert
// to skip the insert() path for all subsequent items.
blob.insert(KeyId(0), ValueId(0), &hash_a, InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(1), ValueId(1), &hash_b, InsertLocation::Auto {}).unwrap();

// batch_insert with a duplicate KeyId(0) — no error is returned.
blob.batch_insert(vec![
    ((KeyId(2), ValueId(2)), hash_c),
    ((KeyId(0), ValueId(99)), hash_d), // duplicate key, bypasses check
]).unwrap();

// The blob now has two leaves with KeyId(0).
// key_to_index cache points only to the second one.
