### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Signature Replay for Meta-Transactions — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_meta_tx_v0_hash` function omits a nonce from the transaction hash, and `execute_meta_tx_v0` performs no nonce check or increment. This means any valid meta-tx v0 signature is permanently replayable by any unprivileged contract caller on the same chain, enabling direct loss of funds from victim accounts.

---

### Finding Description

The `meta_tx_v0` syscall allows a contract to call another contract's `__execute__` entry point with a caller-supplied signature and a freshly computed version-0 transaction hash. The hash is computed in `compute_meta_tx_v0_hash`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        chain_id=chain_id,
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),  // ← no nonce
    );
``` [1](#0-0) 

The hash inputs are: `(INVOKE_HASH_PREFIX, version=0, contract_address, entry_point_selector, calldata_hash, max_fee=0, chain_id)`. **No nonce is included.**

Compare this to `compute_l1_handler_transaction_hash`, which correctly passes `additional_data_size=1, additional_data=&nonce` to bind the hash to a unique nonce: [2](#0-1) 

In `execute_meta_tx_v0`, the constructed `new_tx_info` hardcodes `nonce=0`, and no nonce check or increment is ever performed: [3](#0-2) 

The `check_and_increment_nonce` function explicitly skips version-0 transactions:

```cairo
// Do not handle nonce for version 0.
if (tx_info.version == 0) {
    return ();
}
``` [4](#0-3) 

Similarly, `run_validate` skips `__validate__` for version-0 transactions, so the account contract's own replay guard is also bypassed: [5](#0-4) 

The result: for any fixed `(contract_address, calldata, chain_id)` tuple, the meta-tx v0 hash is **deterministic and permanent**. A signature that was valid once is valid forever.

---

### Impact Explanation

**Critical — Direct loss of funds.**

If a user signs a meta-tx v0 authorizing a token transfer or any fund-moving action, an attacker who observes that signature can replay it an unlimited number of times by deploying a contract that calls the `meta_tx_v0` syscall with the same `(contract_address, selector, calldata, signature)`. The target account's `__execute__` will verify the signature against the same deterministic hash and approve each replay, draining the victim's funds.

---

### Likelihood Explanation

**High.** The `meta_tx_v0` syscall is accessible to any deployed contract. Any on-chain or off-chain observer of a valid meta-tx v0 invocation can extract the signature and calldata from the transaction trace and immediately replay it. No privileged access, leaked key, or operator cooperation is required.

---

### Recommendation

Include a nonce in the meta-tx v0 hash, analogous to how `compute_l1_handler_transaction_hash` includes a nonce via `additional_data`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,   // ← add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        chain_id=chain_id,
        additional_data_size=1,
        additional_data=&nonce,
    );
```

Additionally, `execute_meta_tx_v0` must read the current nonce from `contract_state_changes` for the target `contract_address`, include it in the hash, and increment it after execution — mirroring the `check_and_increment_nonce` logic used for all non-version-0 transactions. [6](#0-5) 

---

### Proof of Concept

1. **Victim** signs a meta-tx v0 authorizing `transfer(attacker, 1000_tokens)` on their account contract. The signature covers `hash(INVOKE_HASH_PREFIX, 0, victim_account, __execute__, calldata, 0, chain_id)`.
2. **Legitimate caller** submits the meta-tx v0 via a contract; the transfer executes successfully.
3. **Attacker** deploys `ReplayContract` that calls `meta_tx_v0(contract_address=victim_account, selector=__execute__, calldata=<same>, signature=<same>)`.
4. The OS calls `compute_meta_tx_v0_hash` with the same inputs → produces the **identical hash**.
5. The victim account's `__execute__` verifies the signature against the same hash → **passes**.
6. No nonce is checked or incremented → the transfer executes again.
7. Attacker repeats step 3–6 until the victim account is drained. [7](#0-6) [8](#0-7)
