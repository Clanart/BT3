### Title
`nativeFee` ETH Permanently Locked in `OmniBridge` — No Withdrawal Mechanism (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

Every call to `initTransfer` with a non-zero `nativeFee` deposits ETH into the `OmniBridge` contract that can never be recovered. The contract has a bare `receive()` fallback that accepts ETH but no corresponding withdrawal path for accumulated native fees.

---

### Finding Description

In `OmniBridge.initTransfer()`, the user's `msg.value` is split into two logical parts:

- **For ERC-20 transfers** (line 393): `extensionValue = msg.value - nativeFee`
- **For native ETH transfers** (line 391): `extensionValue = msg.value - amount - nativeFee` [1](#0-0) 

`extensionValue` is the only value forwarded downstream. In `OmniBridgeWormhole.initTransferExtension()`, it is passed directly to `_wormhole.publishMessage{value: value}(...)` as the Wormhole message fee: [2](#0-1) 

The `nativeFee` portion is **never forwarded anywhere**. It silently accumulates in the contract's ETH balance. The contract also exposes a bare `receive()` fallback: [3](#0-2) 

Searching the entire `OmniBridge.sol` and `OmniBridgeWormhole.sol`, there is no `withdraw`, `rescueETH`, or admin drain function. The only ETH egress path is `finTransfer` when `tokenAddress == address(0)`, which requires a valid MPC signature from the NEAR side and is scoped to bridging ETH back to users — not to recovering accumulated fee balances: [4](#0-3) 

---

### Impact Explanation

Every `initTransfer` call where `nativeFee > 0` permanently locks that ETH in the contract. The `nativeFee` is a real user payment intended to compensate the NEAR-side relayer; the NEAR side reads its value from the `InitTransfer` event and may pay the relayer in NEAR tokens, but the ETH equivalent is never disbursed on the EVM side. Over the lifetime of the bridge, this accumulates to an irrecoverable pool of user-paid ETH with no on-chain mechanism to retrieve it.

**Matched allowed impact:** *Critical — Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.*

---

### Likelihood Explanation

`nativeFee` is a standard, documented parameter of the bridge's public `initTransfer` API. Any user who pays a native fee to incentivize relayer processing contributes to the locked balance. This is a normal, expected usage path — not an edge case. Likelihood is **High**.

---

### Recommendation

Add an admin-gated ETH rescue function, for example:

```solidity
function rescueETH(address payable to, uint256 amount)
    external onlyRole(DEFAULT_ADMIN_ROLE)
{
    (bool ok,) = to.call{value: amount}("");
    require(ok, "ETH transfer failed");
}
```

Alternatively, forward `nativeFee` directly to a designated fee recipient inside `initTransfer` rather than leaving it in the contract balance.

---

### Proof of Concept

1. User calls `initTransfer(erc20Token, 100e18, fee=1e18, nativeFee=0.01 ether, "alice.near", "")` with `msg.value = 0.01 ether + wormholeFee`.
2. Inside `initTransfer`: `extensionValue = msg.value - nativeFee = wormholeFee`.
3. `initTransferExtension` is called with `value = wormholeFee`; Wormhole receives exactly `wormholeFee`.
4. `nativeFee = 0.01 ether` remains in `address(this).balance`.
5. No function in `OmniBridge.sol` or `OmniBridgeWormhole.sol` can send this ETH out.
6. After N users each pay `0.01 ether` in `nativeFee`, `N × 0.01 ether` is permanently locked. [5](#0-4) [6](#0-5)
