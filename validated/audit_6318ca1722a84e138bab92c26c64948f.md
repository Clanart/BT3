### Title
Integer Overflow in Relative Timelock Arithmetic Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Conditions — (`File: crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` contains two arithmetic paths for relative timelock validation, selected by the caller-supplied `nowrap: bool` flag. When `nowrap=false`, the function uses `wrapping_add` for the addition `confirmed_block_index + height_relative` (u32) and `timestamp + seconds_relative` (u64). When the sum overflows, the result wraps to a small value, causing the timelock guard to evaluate as satisfied even though the required height or timestamp has not been reached. This is the direct analog of the external report's underflow bug: just as `netBorrowLimit = 0` followed by a subtraction produces an astronomically large borrow limit, here `confirmed_block_index.wrapping_add(height_relative)` wraps to near-zero, making the comparison `prev_height < near_zero` trivially false and the timelock trivially bypassed.

---

### Finding Description

`check_time_locks` accepts a `nowrap: bool` parameter and branches on it for every relative timelock check:

```rust
// ASSERT_HEIGHT_RELATIVE
if nowrap {
    if prev_transaction_block_height
        < unspent.confirmed_block_index.saturating_add(height_relative)
    { return Err(...); }
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{ return Err(...); }
```

When `nowrap=false`:

- `confirmed_block_index = 10`, `height_relative = 0xFFFF_FFF6`
- `10u32.wrapping_add(0xFFFF_FFF6)` = `0x1_0000_0000 & 0xFFFF_FFFF` = **0**
- Guard: `prev_height < 0` → **false** → no error → spend accepted

The intended timelock is ~4.29 billion blocks in the future, but the wrapping makes it appear the deadline is block 0, which has already passed. The same overflow applies to `ASSERT_SECONDS_RELATIVE` using `u64::wrapping_add`.

The inverse effect hits `ASSERT_BEFORE_HEIGHT_RELATIVE` and `ASSERT_BEFORE_SECONDS_RELATIVE`: wrapping to a small deadline causes a spend that should be accepted (deadline far in the future) to be incorrectly rejected.

The condition values that trigger overflow are fully attacker-controlled: `ASSERT_HEIGHT_RELATIVE` accepts any u32 value (parsed by `sanitize_uint` with `max_size=4`), and `ASSERT_SECONDS_RELATIVE` accepts any u64 value (`max_size=8`). A CLVM puzzle can emit these conditions freely.

The `nowrap` parameter is exposed through the Python binding `py_check_time_locks` and is passed by the Chia full node. The codebase's own tests explicitly document and confirm the divergence:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),   // <-- timelock bypassed when nowrap=false
)]
```

---

### Impact Explanation

**Timelock bypass (unauthorized spend acceptance):** A coin whose puzzle emits `ASSERT_HEIGHT_RELATIVE` with a value chosen so that `confirmed_block_index + height_relative` overflows u32 can be spent at any block height when the node evaluates with `nowrap=false`. The timelock condition, which is the sole enforcement mechanism for height-locked coins (vaults, vesting contracts, HTLCs), is silently skipped.

**Consensus divergence / chain split:** Nodes running with `nowrap=true` (saturating) reject the spend; nodes running with `nowrap=false` (wrapping) accept it. Because `nowrap` is a caller-supplied parameter rather than a consensus constant, any inconsistency in how the full node passes it across versions or configurations produces irreconcilable disagreement on block validity — a committed chain split.

Both impacts fall within the allowed scope:
- *High*: timelock condition validation bypass enables unauthorized spend acceptance.
- *Critical*: consensus divergence between nodes on a valid spend bundle.

---

### Likelihood Explanation

The overflow values are reachable with a single CLVM condition opcode (`ASSERT_HEIGHT_RELATIVE = 82`, `ASSERT_SECONDS_RELATIVE = 83`) carrying a near-maximum integer argument. No privileged role, leaked key, or network-level attack is required. Any unprivileged coin creator can embed the condition in a puzzle. The `sanitize_uint` parser explicitly accepts values up to `u32::MAX` for height conditions and `u64::MAX` for seconds conditions, so no input sanitization blocks the trigger value. The codebase's own unit tests confirm the divergent outcome is reachable and reproducible.

---

### Recommendation

1. **Remove the `nowrap` branch entirely.** Replace all four `wrapping_add` paths with `saturating_add` (the `nowrap=true` path). Saturating arithmetic is the only semantically correct behavior: a timelock of `u32::MAX` blocks should be treated as "never spendable at any realistic height," not as "already spendable."

2. **Make the behavior a consensus constant, not a caller parameter.** The `nowrap` flag should not be a runtime argument to `check_time_locks` or `py_check_time_locks`. Consensus-critical arithmetic must be deterministic across all nodes regardless of caller.

3. **Add a rejection rule in `sanitize_uint` / condition parsing** for `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE` values that would overflow when added to any plausible `confirmed_block_index` or `timestamp`, or document the maximum safe value and enforce it.

### Proof of Concept

The existing unit tests in `crates/chia-consensus/src/check_time_locks.rs` serve as a direct proof of concept:

```rust
// confirmed_block_index = 100, prev_height = 200
// 100u32.wrapping_add(0xffff_ffff) = 99  (wraps)
// 200 < 99 → false → Ok(()) — timelock bypassed
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(...),  // nowrap=true: correctly rejected
    Ok(()),    // nowrap=false: timelock bypassed
)]
```

And the Python-level test in `tests/test_check_time_locks.py`:

```python
# 10 + 0xFFFF_FFF6 overflows to 0; wrapping: 15 < 0 → Ok (bypass)
# saturating: clamps to u32::MAX; 15 < u32::MAX → Err (correct)
(make_test_conds(height_relative=0xFFFF_FFF6), 13, None),
```

A concrete exploit spend bundle would contain a coin confirmed at any height `h`, with a puzzle that outputs `(ASSERT_HEIGHT_RELATIVE . (u32::MAX - h + 1))`. Under `nowrap=false`, the spend is accepted at the very next block. Under `nowrap=true`, it is correctly rejected for ~4.29 billion blocks.
