### Title
`BlockStatusCache::move_index` Fails to Update Leaf-Key and Leaf-Hash Maps After Sibling Promotion, Corrupting DataLayer Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::delete()` contains a stale-state bug structurally identical to the Oracle `removeMember` report: when a leaf is deleted and its sibling must be promoted to the root position (index 0), `BlockStatusCache::move_index()` only updates `free_indexes` but leaves `key_to_index` and `leaf_hash_to_index` pointing at the now-freed old index. Subsequent `upsert`, `get_proof_of_inclusion`, or `get_node_by_hash` calls on the surviving leaf operate on the wrong blob slot, silently corrupting the Merkle root or causing a panic.

---

### Finding Description

**Root cause — `move_index` is incomplete:** [1](#0-0) 

`move_index(source, destination)` marks `source` as free in `free_indexes` but never updates `key_to_index` or `leaf_hash_to_index`. Both maps continue to hold `source` as the authoritative index for the moved leaf.

**Trigger path — `delete()` when the deleted leaf's parent is the root:** [2](#0-1) 

When `grandparent_index` is `None` (the deleted leaf's parent is the root), the sibling block is physically written to `destination = TreeIndex(0)` and `move_index(sibling_index, destination)` is called. If the sibling is a **leaf** (the canonical 2-leaf tree case), the cache is now inconsistent:

| Map | Stored value | Correct value |
|---|---|---|
| `key_to_index[sibling_key]` | `sibling_index` (freed) | `TreeIndex(0)` |
| `leaf_hash_to_index[sibling_hash]` | `sibling_index` (freed) | `TreeIndex(0)` | [3](#0-2) 

`add_leaf` (called on every correct insert) shows the pattern that `move_index` should replicate but does not.

**Downstream corruption — `upsert` on the surviving leaf:** [4](#0-3) 

`upsert` calls `get_leaf_by_key` which resolves through the stale `key_to_index` to `sibling_index`. It then writes the updated leaf back to `sibling_index` (a freed slot), leaving `TreeIndex(0)` — the actual tree root — holding the **old, unmodified leaf hash**. The Merkle root computed from the blob is now wrong.

**Downstream panic — `get_proof_of_inclusion` on the surviving leaf:** [5](#0-4) 

`get_proof_of_inclusion` resolves `index = sibling_index` (stale), reads the old block there (which still carries `parent = Some(0)`), then traverses the lineage. The block at index 0 is now the promoted leaf (not an internal node), so `expect_internal()` panics unconditionally.

**Integrity check is disabled in production:** [6](#0-5) 

`check_integrity_on_drop` is `cfg!(test)` — `false` in production — so the stale cache is never detected at runtime.

---

### Impact Explanation

After a 2-leaf → 1-leaf deletion:

1. **Merkle root corruption**: `upsert` on the surviving leaf writes to the freed slot, leaving the root blob slot with the old hash. All subsequent `get_root_hash()` calls return a hash that does not reflect the updated value. Proofs of inclusion generated from this root are invalid; proofs of exclusion can be forged against the stale root.

2. **Deterministic panic**: `get_proof_of_inclusion` on the surviving leaf always panics via `expect_internal()` on a leaf node, crashing the DataLayer node process.

3. **Index reuse confusion**: After a subsequent insert reuses `sibling_index`, `get_leaf_by_key(sibling_key)` and `get_node_by_hash(sibling_hash)` return data belonging to the newly inserted node, enabling cross-key data confusion.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots and lets untrusted input prove invalid state.**

---

### Likelihood Explanation

The trigger is a routine operational sequence: insert two leaves, delete one. This is a normal DataLayer lifecycle event (e.g., a store with two entries where one is removed). No special privileges are required; any DataLayer store owner can reach this state. The bug is silent — no error is returned from `delete()` — so the corruption persists undetected until a subsequent read or update operation.

---

### Recommendation

`move_index` must update all three cache maps, not just `free_indexes`. The fix mirrors what `remove_leaf` + `add_leaf` already do correctly:

```rust
fn move_index(&mut self, source: TreeIndex, destination: TreeIndex) -> Result<(), Error> {
    if self.free_indexes.contains(&source) {
        return Err(Error::MoveSourceIndexNotInUse(source));
    }
    if self.free_indexes.contains(&destination) {
        return Err(Error::MoveDestinationIndexNotInUse(destination));
    }

    // Update key_to_index: any key pointing at source must now point at destination
    for index in self.key_to_index.values_mut() {
        if *index == source {
            *index = destination;
        }
    }
    // Update leaf_hash_to_index similarly
    for index in self.leaf_hash_to_index.values_mut() {
        if *index == source {
            *index = destination;
        }
    }

    self.free_indexes.insert(source);
    Ok(())
}
```

Alternatively, pass the moved `LeafNode` into `move_index` so it can do a targeted O(1) update via `key_to_index.insert(leaf.key, destination)` and `leaf_hash_to_index.insert(leaf.hash, destination)`.

Additionally, enable `check_integrity_on_drop` (or an equivalent assertion) in debug/staging builds to catch future regressions.

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};

fn sha256_val(v: u8) -> Hash {
    use chia_sha2::Sha256;
    let mut h = Sha256::new(); h.update(&[v]); h.finalize().into()
}

fn main() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();
    blob.check_integrity_on_drop = false; // as in production

    // Insert two leaves → 2-leaf tree
    let key_a = KeyId(1); let hash_a = sha256_val(1);
    let key_b = KeyId(2); let hash_b = sha256_val(2);
    blob.insert(key_a, ValueId(10), &hash_a, InsertLocation::Auto {}).unwrap();
    blob.insert(key_b, ValueId(20), &hash_b, InsertLocation::Auto {}).unwrap();
    blob.calculate_lazy_hashes().unwrap();

    let root_before = blob.get_root_hash().unwrap();

    // Delete leaf_A → sibling leaf_B promoted to index 0; cache is now stale
    blob.delete(key_a).unwrap();

    // Upsert leaf_B with a new hash → writes to freed sibling_index, NOT index 0
    let new_hash_b = sha256_val(99);
    blob.upsert(key_b, ValueId(99), &new_hash_b).unwrap();
    blob.calculate_lazy_hashes().unwrap();

    let root_after = blob.get_root_hash().unwrap();

    // Root should have changed to reflect new_hash_b, but it still equals root_before
    // because index 0 was never updated.
    assert_ne!(root_before, root_after, "BUG: root unchanged after upsert");
    // ^^^ This assertion FAILS, demonstrating silent Merkle root corruption.

    // get_proof_of_inclusion panics unconditionally on the surviving leaf
    let _ = blob.get_proof_of_inclusion(key_b); // panics: expect_internal on a leaf
}
```

    pub fn get_proof_of_inclusion(
        &self,
        key: KeyId,
    ) -> Result<proof_of_inclusion::ProofOfInclusion, Error> {
        let mut index = *self
            .block_status_cache
            .get_index_by_key(key)
            .ok_or(Error::UnknownKey(key))?;

        let node = self
            .get_node(index)?
            .expect_leaf("key to index mapping should only have leaves");

        let parents = self.get_lineage_blocks_with_indexes(index)?;
        let mut layers: Vec<proof_of_inclusion::ProofOfInclusionLayer> = Vec::new();
        let mut parents_iter = parents.iter();
        // first in the lineage is the index itself, second is the first parent
        parents_iter.next();
        for (next_index, block) in parents_iter {
            if block.metadata.dirty {
                return Err(Error::Dirty(*next_index));
            }
            let parent = block
                .node
                .expect_internal("all nodes after the first should be internal");
            let sibling_index = parent.sibling_index(index)?;
            let sibling_block = self.get_block(sibling_index)?;
            let sibling = sibling_block.node;
            let layer = proof_of_inclusion::ProofOfInclusionLayer {
                other_hash_side: parent.get_sibling_side(index)?,
                other_hash: sibling.hash(),
                combined_hash: parent.hash,
            };
            layers.push(layer);
            index = *next_index;
        }

        Ok(proof_of_inclusion::ProofOfInclusion {
            node_hash: node.hash,
            layers,
        })
    }
