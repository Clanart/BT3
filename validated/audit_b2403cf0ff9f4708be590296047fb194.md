### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary class hash from the calling contract without verifying that the hash corresponds to a previously declared class. An unprivileged contract can invoke `replace_class` with a non-existent class hash, permanently rendering itself uncallable and freezing all funds held in its storage.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall:

```cairo
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The `class_hash` value is taken directly from `request.class_hash`, which is the raw syscall request data supplied by the calling contract. The OS then unconditionally writes this value into `contract_state_changes` as the contract's new class hash. There is **no check** that `class_hash` exists in `contract_class_changes` (the declared-class registry) or in any prior state. The developer-acknowledged TODO comment at line 898 confirms this missing validation:

> `// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.`

This is structurally identical to the reported `_collateralAddress` misconfiguration class: a critical address/hash parameter is accepted and committed to state without cross-checking its validity against a companion registry, leaving the system in a permanently broken configuration.

By contrast, the `execute_deploy` path (`syscall_impls.cairo` lines 452–556) immediately executes the constructor with the provided class hash, so an invalid hash causes an immediate execution failure that is caught and reverted. `replace_class` has no such immediate execution — the new class hash is only resolved the next time the contract is called, after the state has already been finalized and committed.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once `replace_class` is called with an undeclared class hash and the block is finalized:

1. The contract's `StateEntry.class_hash` is permanently set to a hash that has no corresponding class definition.
2. Every subsequent call to the contract will attempt to load the class, fail to find it, and revert.
3. The contract is permanently uncallable.
4. All assets held in the contract's storage (ERC-20 balances, ETH, NFT ownership records, etc.) are permanently inaccessible — a complete, irreversible freeze of funds.

There is no recovery path: the OS has no mechanism to undo a committed state entry, and the contract cannot call any entry point (including one that would call `replace_class` again) because every call reverts at class resolution.

---

### Likelihood Explanation

The attack surface is broad and realistic:

- **Malicious deployer**: An attacker deploys a contract that appears legitimate (e.g., a yield vault or multisig). After users deposit funds, the attacker calls a function that internally invokes `replace_class` with a fabricated, undeclared class hash. All deposited funds are frozen.
- **Buggy contract**: A contract with a logic error passes an incorrect class hash to `replace_class`. The OS accepts it silently, permanently breaking the contract.
- **Composability attack**: A contract that accepts an arbitrary class hash as a parameter and passes it to `replace_class` can be exploited by any caller who provides a non-existent hash.

The entry point is a standard, unprivileged syscall available to any deployed contract. No privileged role, leaked key, or operator cooperation is required.

---

### Recommendation

Before committing the new class hash to `contract_state_changes`, the OS must verify that the provided `class_hash` exists in `contract_class_changes` (i.e., it was declared in the current block) **or** in the pre-existing class trie (i.e., it was declared in a prior block). The check should be analogous to how `execute_deploy` implicitly validates the class by executing the constructor. Concretely, resolve the TODO at line 898 of `syscall_impls.cairo`:

```cairo
// Before dict_update, assert class_hash is declared:
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

If the class was declared in a prior block (not in `contract_class_changes`), a hint-assisted lookup into the pre-state class trie with a Cairo-enforced assertion is required.

---

### Proof of Concept

1. Attacker deploys contract `VaultAttack` with the following logic in its `__execute__` entry point:
   - Accept a `class_hash` argument from the caller.
   - Call `replace_class(class_hash=<fabricated_nonexistent_hash>)`.
2. Users deposit tokens into `VaultAttack` (e.g., via an ERC-20 `transfer` to the contract's address, or by calling a `deposit` function that records balances in storage).
3. Attacker calls `VaultAttack.__execute__` with the fabricated hash.
4. The OS processes `execute_replace_class`:
   - `request.class_hash` = fabricated hash (not in `contract_class_changes`, not in any prior class trie).
   - No validation is performed (line 898 TODO is unimplemented).
   - `dict_update` writes the fabricated hash as `VaultAttack`'s new class hash.
5. Block is finalized. `VaultAttack.StateEntry.class_hash` = fabricated hash.
6. Any subsequent call to `VaultAttack` (including withdrawal functions) fails at class resolution.
7. All user funds are permanently frozen.

The missing cross-check is the absence of a lookup into `contract_class_changes` (or the pre-state class trie) before the `dict_update` at line 906. Compare with the declared-class registry populated by `execute_declare_transaction`: [2](#0-1)
