### Title
Unchecked `max_price_per_unit` Upper Bound Enables Field-Arithmetic Wrap-Around in Fee Calculation — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` multiplies user-supplied `max_price_per_unit` and `max_amount` fields directly in Cairo felt arithmetic without enforcing an upper bound on `max_price_per_unit`. Because Cairo arithmetic is modular (mod the Stark field prime P ≈ 2^251), a crafted `max_price_per_unit` value can cause the product to wrap around to a very small felt, making `max_fee` appear near-zero. The subsequent `assert_nn_le(actual_fee, max_fee)` check then constrains the sequencer to charge essentially no fee, constituting a direct loss of fee revenue.

---

### Finding Description

`compute_max_possible_fee` in `transaction_impls.cairo` computes:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

`max_amount` is bounded to `[0, 2^64 - 1]` by `pack_resource_bounds` during hash computation:

```cairo
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
assert_nn(resource_bounds.max_price_per_unit);
``` [2](#0-1) 

`assert_nn(max_price_per_unit)` only enforces `max_price_per_unit ∈ [0, P/2)` — a ~250-bit range. No upper bound (e.g., `≤ 2^128 - 1`) is enforced. `compute_max_possible_fee` itself performs no additional validation on these fields.

With `max_amount ≤ 2^64 - 1` and `max_price_per_unit < P/2`, the product `max_amount * max_price_per_unit` can be as large as `~2^313`, wrapping around P approximately `2^62` times. By choosing `max_price_per_unit` appropriately (e.g., `max_price_per_unit ≡ inv(max_amount) mod P` when that inverse is `< P/2`), the product wraps to a very small felt value such as 1 or 2.

The resulting `max_fee` is then used as the cap in `charge_fee`:

```cairo
assert_nn_le(calldata.amount.low, max_fee);
``` [3](#0-2) 

If `max_fee` is 1, the sequencer is provably constrained to charge at most 1 unit of fee token, regardless of actual gas consumed.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A transaction sender can craft a V3 transaction with a `max_price_per_unit` value that causes `compute_max_possible_fee` to return a near-zero felt (e.g., 1). The OS program then enforces that the sequencer charges at most 1 unit of fee token. The user's transaction is executed and state changes are committed, but the sequencer receives essentially no fee. At scale, this allows systematic fee evasion, draining sequencer fee revenue.

---

### Likelihood Explanation

**Medium.** The attack requires:
1. Selecting `max_price_per_unit` such that `max_amount * max_price_per_unit mod P` is small — this is mathematically straightforward using modular inverse arithmetic (for roughly half of all `max_amount` values, `inv(max_amount) mod P < P/2` and passes `assert_nn`).
2. The sequencer's off-chain code not independently validating `max_price_per_unit` bounds before block inclusion.

The OS program is the authoritative on-chain enforcement layer. If the sequencer trusts it to enforce fee caps, the off-chain code may not re-validate this bound, making exploitation realistic.

---

### Recommendation

Add an explicit upper-bound check on `max_price_per_unit` in `pack_resource_bounds` (which is already called during hash computation and is the natural enforcement point):

```cairo
assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);
```

This mirrors the `max_amount` bound and ensures the product `max_amount * max_price_per_unit` cannot exceed `(2^64 - 1) * (2^128 - 1) < 2^192 << P`, preventing field wrap-around. [4](#0-3) 

---

### Proof of Concept

**Setup:** StarkNet field prime `P = 2^251 + 17·2^192 + 1`.

**Goal:** Find `max_amount = A ≤ 2^64 - 1` and `max_price_per_unit = B < P/2` such that `A * B mod P = 1`.

**Construction:**
1. Choose `A` such that `inv(A) mod P < P/2`. For example, `A = 5`. Compute `B = inv(5) mod P`. Since `P ≡ 1 (mod 5)`, `inv(5) = (4P + 1)/5`. Whether this is `< P/2` depends on the exact value; iterate over small `A` values until one satisfies the constraint.
2. Craft a V3 transaction with:
   - `l1_gas_bounds.max_amount = A`
   - `l1_gas_bounds.max_price_per_unit = B`
   - `l2_gas_bounds.max_amount = 0`
   - `l1_data_gas_bounds.max_amount = 0`
3. `compute_max_possible_fee` returns `A * B mod P = 1`.
4. `charge_fee` enforces `assert_nn_le(actual_fee, 1)`.
5. Sequencer must charge ≤ 1 unit of fee token to produce a valid proof.
6. Transaction executes; user pays 1 unit of fee token regardless of actual gas consumed. [5](#0-4) [6](#0-5)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}

func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn(resource_bounds.max_price_per_unit);
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;
}
