### Title
Token-2022 MetadataPointer Spoofing: Missing `metadata.mint == self.mint` Check Enables Cross-Identity Token Deployment — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

`parse_metadata_account` verifies that the passed `metadata` account key matches the address stored in the Token-2022 `MetadataPointer` extension, but **never verifies that the Metaplex metadata account's `mint` field equals `self.mint.key()`**. An unprivileged attacker can create a Token-2022 mint whose `MetadataPointer` points to a legitimate token's Metaplex PDA, call `log_metadata`, and obtain a Wormhole VAA that binds the victim's `name`/`symbol` to the attacker's mint pubkey. Submitting that VAA to NEAR's `deploy_token` deploys a NEAR token carrying the victim's identity but mapped to the attacker's Solana mint.

---

### Finding Description

In `process()`, when `token_program == token_2022::ID`, the code reads the `MetadataPointer` extension: [1](#0-0) 

If the pointer address differs from the mint's own key (i.e., it is a "third-party" metadata account), it delegates to `parse_metadata_account(metadata_pointer.metadata_address.0)`.

Inside `parse_metadata_account`, the only address check is: [2](#0-1) 

This confirms the passed `metadata` account key equals the pointer address. Then, if the account is owned by `MetaplexID`, it deserializes and returns `name`/`symbol` directly: [3](#0-2) 

**There is no `require_keys_eq!(metadata.mint, self.mint.key())` check.** The Metaplex `MetadataAccount` struct contains a `mint` field identifying which mint the metadata belongs to, but it is never consulted.

The `metadata` account itself has no Anchor constraints: [4](#0-3) 

The resulting `LogMetadataPayload` uses `self.mint.key()` (attacker's mint) as `token` but the victim's `name`/`symbol`: [5](#0-4) 

---

### Impact Explanation

The VAA payload is deserialized on NEAR by the Wormhole prover proxy into a `LogMetadataMessage`: [6](#0-5) 

`deploy_token_callback` on NEAR then deploys a token using `metadata.token_address` (attacker's mint) and `metadata.name`/`metadata.symbol` (victim's): [7](#0-6) 

The deployed NEAR token carries the victim's name/symbol (e.g., "USD Coin" / "USDC") but is permanently mapped to the attacker's Solana mint address. Since the attacker controls that mint, they can mint arbitrary supply on Solana and bridge it to NEAR, where it appears under the victim's identity. This constitutes **token-mapping corruption enabling unauthorized token deployment and misdirected value**.

---

### Likelihood Explanation

The attack requires no privileges. Any account can:
1. Create a Token-2022 mint with `MetadataPointer` set to any existing Metaplex PDA.
2. Call the public `log_metadata` instruction.
3. Relay the resulting VAA to NEAR `deploy_token`.

The only precondition is that the target NEAR token has not already been deployed (the `deployed_tokens.insert` check at line 2422 would reject a duplicate). For any token not yet deployed on NEAR, the attack is immediately executable. [8](#0-7) 

---

### Recommendation

After deserializing the Metaplex metadata in `parse_metadata_account`, add a binding check:

```rust
if metadata.owner == &MetaplexID {
    let data = metadata.try_borrow_data()?;
    let mpl = MplMetadata::try_deserialize(&mut data.as_ref())?;
    // ADD THIS:
    require_keys_eq!(
        mpl.mint,
        self.mint.key(),
        ErrorCode::InvalidTokenMetadataAddress
    );
    Ok((mpl.name.clone(), mpl.symbol.clone()))
}
```

This ensures the Metaplex metadata account actually belongs to the mint being logged. [3](#0-2) 

---

### Proof of Concept

1. Identify victim: e.g., USDC on Solana with Metaplex PDA `victim_meta_pda` (owned by `MetaplexID`, `mint` field = `usdc_mint`).
2. Attacker creates a Token-2022 mint `attacker_mint` with `MetadataPointer` extension set to `victim_meta_pda`.
3. Attacker calls `log_metadata` with `mint = attacker_mint`, `metadata = victim_meta_pda`.
4. In `process()`: `metadata_pointer.metadata_address.0 == victim_meta_pda != attacker_mint` → calls `parse_metadata_account(victim_meta_pda)`.
5. In `parse_metadata_account`: `require_keys_eq!(victim_meta_pda, victim_meta_pda)` passes; `owner == MetaplexID` passes; returns `("USD Coin", "USDC")`.
6. `LogMetadataPayload { token: attacker_mint, name: "USD Coin", symbol: "USDC", decimals: attacker_decimals }` is posted as a Wormhole message.
7. Attacker relays the VAA to NEAR `deploy_token`.
8. NEAR deploys a token with `name="USD Coin"`, `symbol="USDC"`, mapped to `sol:attacker_mint`.
9. Attacker mints tokens on `attacker_mint` and bridges them to NEAR, where they appear as "USDC".
