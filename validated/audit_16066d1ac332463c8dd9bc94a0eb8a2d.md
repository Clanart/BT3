### Title
Dust-Amount-to-Zero Rounding in `sign_transfer` Permanently Locks User Tokens - (File: `near/omni-bridge/src/lib.rs`)

### Summary

`init_transfer` accepts any `(amount, fee)` pair where `fee < amount`, but `sign_transfer` later rejects the transfer if `normalize_amount(amount - fee) == 0`. Because `update_transfer_fee` only allows fee increases, a transfer that reaches this state can never be finalized and the locked tokens are irrecoverable.

### Finding Description

`init_transfer` enforces only one arithmetic constraint on the fee:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

It does **not** verify that the net amount after fee subtraction survives decimal normalization. Later, `sign_transfer` computes:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

`normalize_amount` performs integer floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [3](#0-2) 

For any token where `origin_decimals > decimals` (e.g., a NEAR-native token with 24 decimals bridged to an EVM chain at 18 decimals, giving a divisor of 10^6), any `(amount - fee) < 10^(origin_decimals - decimals)` rounds to zero. The `require!(amount_to_transfer > 0)` check then permanently blocks `sign_transfer`.

The fee cannot be reduced after creation — `update_transfer_fee` enforces `fee.fee >= current_fee.fee`:

```rust
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [4](#0-3) 

There is no `cancel_transfer` or admin-rescue path. The only removal paths are `sign_transfer_callback` (unreachable because `sign_transfer` reverts before reaching MPC) and `claim_fee_callback` (requires a finalization proof from the destination chain, which can never exist). The transfer message and the locked tokens remain in the bridge contract forever.

### Impact Explanation

**Critical — Permanent, irrecoverable lock of user funds in the NEAR bridge vault.**

Concrete example with a 6-decimal-difference token (origin_decimals=24, decimals=18, divisor=10^6):

| Parameter | Value |
|---|---|
| `amount` | 999,999 NEAR units |
| `fee` | 0 |
| `amount_without_fee` | 999,999 |
| `normalize_amount(999,999)` | 0 (floor division) |
| `sign_transfer` result | permanent revert `ERR_INVALID_AMOUNT_TO_TRANSFER` |
| User funds | permanently locked |

A second variant: `amount = 2,000,000`, `fee = 1,500,001` (passes `fee < amount`), leaving `amount_without_fee = 499,999`, which also normalizes to 0.

### Likelihood Explanation

- Any NEAR-originated transfer targeting a chain where the token's `origin_decimals > decimals` is affected.
- The condition is reachable by any unprivileged user via `ft_transfer_call` → `ft_on_transfer` → `init_transfer`.
- It can be triggered accidentally (user transfers a small amount of a high-precision token) or deliberately (user sets fee to `amount - 1` for a token with large decimal differences).
- No privileged access, no colluding parties, and no external dependency is required.

### Recommendation

Add a normalization check inside `init_transfer` (or `ft_on_transfer`) before storing the transfer message. If the token's decimals are already registered at that point, reject the transfer early:

```rust
if let Some(decimals) = self.token_decimals.get(&token_address) {
    let normalized = Self::normalize_amount(
        transfer_message.amount_without_fee()
            .near_expect(BridgeError::InvalidFee),
        decimals,
    );
    require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
}
```

If decimals are not yet registered at `init_transfer` time, add a `cancel_transfer` function callable by the original sender so that stuck transfers can be unwound and tokens returned.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 18` (divisor = 10^6).
2. Call `ft_transfer_call` with `amount = 500_000` and `fee = 0` targeting an EVM chain.
3. `init_transfer` succeeds: `fee (0) < amount (500_000)` passes.
4. Call `sign_transfer` for the resulting `transfer_id`.
5. `normalize_amount(500_000, {decimals:18, origin_decimals:24})` = `500_000 / 1_000_000` = `0`.
6. `require!(0 > 0, ...)` panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
7. `update_transfer_fee` cannot reduce the fee (only increases allowed).
8. No cancel path exists. The 500,000 NEAR-unit tokens are permanently locked. [2](#0-1) [3](#0-2) [1](#0-0) [4](#0-3)
