### Title
ERC1155 `safeTransferFrom` to a Non-Receiver Contract Causes Permanently Unclaimable Funds — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

In `OmniBridge.finTransfer`, when the destination token is an ERC1155 multi-token, the bridge uses `IERC1155.safeTransferFrom` to deliver tokens to the recipient. If the recipient is a smart contract that does not implement `IERC1155Receiver`, the call reverts. Because the recipient address is immutably encoded in the MPC-signed payload, no retry with a different recipient is possible on-chain. The ERC1155 tokens remain locked in the bridge contract with no recovery path, while the source-chain tokens are already burned or locked.

---

### Finding Description

`OmniBridge.finTransfer` marks the destination nonce as consumed **before** executing the token delivery:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287
``` [1](#0-0) 

Then, for ERC1155 tokens, it calls:

```solidity
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,
    multiToken.tokenId,
    payload.amount,
    ""
);
``` [2](#0-1) 

`safeTransferFrom` in the ERC1155 standard calls `onERC1155Received` on the recipient if it is a contract. If the recipient contract does not implement `IERC1155Receiver`, the call reverts. Because the entire transaction reverts, the nonce assignment at line 287 is also rolled back — the nonce is **not** permanently consumed. However, the recipient address is fixed inside the MPC-signed `TransferMessagePayload`. No caller can alter the recipient and re-submit a valid signature; every future call to `finTransfer` with the same signed payload will revert identically. The ERC1155 tokens remain locked in the bridge contract indefinitely.

The ETH delivery path has the same structure:

```solidity
(bool success, ) = payload.recipient.call{value: payload.amount}("");
if (!success) revert FailedToSendEther();
``` [3](#0-2) 

If the recipient contract rejects ETH, the same permanent-unclaimability outcome applies.

---

### Impact Explanation

Matches **Critical — Permanent freezing / irrecoverable lock of user funds in bridge flows**.

- ERC1155 tokens deposited via `initTransfer1155` are held by the bridge contract.
- The source-chain side (NEAR) has already burned or locked the corresponding tokens.
- The EVM-side delivery permanently fails with no on-chain recovery mechanism.
- Tokens are irrecoverably locked in the bridge contract. [4](#0-3) 

---

### Likelihood Explanation

**Low.** A user must specify, as the EVM recipient, a smart contract address that does not implement `IERC1155Receiver`. This can happen accidentally (e.g., specifying a multisig, DAO treasury, or other contract wallet that lacks ERC1155 support). No malicious actor is required; ordinary user error is sufficient to trigger the loss.

---

### Recommendation

1. **Wrap the ERC1155 delivery in a try/catch** and, on failure, record the unclaimed amount in a claimable mapping keyed by `(destinationNonce, recipient)`. Expose a separate `claimFailed` function that allows the original recipient (or an admin) to redirect the tokens to a different address.
2. Alternatively, **validate off-chain** (in the MPC signing layer) that the recipient implements `IERC1155Receiver` before signing the payload, rejecting payloads targeting non-receiver contracts.
3. Apply the same pattern to the ETH delivery path.

---

### Proof of Concept

1. User holds ERC1155 tokens on EVM and calls `initTransfer1155`, locking them in the bridge.
2. NEAR side credits the user's NEAR account.
3. User initiates a return transfer on NEAR, specifying as EVM recipient a contract address (e.g., a simple multisig) that does not implement `onERC1155Received`.
4. NEAR burns the tokens and the MPC signs a `TransferMessagePayload` with `recipient = <non-receiver-contract>`.
5. Anyone calls `OmniBridge.finTransfer` with the signed payload.
6. `IERC1155.safeTransferFrom(address(this), <non-receiver-contract>, ...)` reverts.
7. The entire transaction reverts; `completedTransfers[nonce]` is rolled back.
8. No valid alternative payload exists (recipient is MPC-signed and immutable).
9. ERC1155 tokens remain locked in the bridge contract forever; NEAR-side tokens are already burned. [5](#0-4)
