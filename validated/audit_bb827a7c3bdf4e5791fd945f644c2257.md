### Title
`ProofOfInclusion::valid()` Tautological Root Check Allows Forged DataLayer Merkle Proofs - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer Merkle proof module contains a tautological final root-hash check. The function verifies internal chain consistency but its terminal assertion `existing_hash == self.root_hash()` is always true after the loop, meaning it never validates the proof against any external trusted root. Because `ProofOfInclusion` is exposed via Python bindings (`from_py_object`) and implements `Streamable`, an unprivileged attacker can construct a fully forged proof that `valid()` accepts for any claimed key/value pair in any claimed tree root.

### Finding Description

`ProofOfInclusion::valid()` is defined as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()   // ← tautological
}
``` [1](#0-0) 

And `root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

**The tautology:** After the loop body executes without returning `false`, `existing_hash` holds the value of the last `calculated_hash`, which was already asserted equal to `layer.combined_hash`. `self.root_hash()` returns `self.layers.last().combined_hash` — the identical value. Therefore `existing_hash == self.root_hash()` is unconditionally `true` after any loop that completes, providing zero security.

The function only checks that the hash chain is *internally self-consistent* (each layer's `combined_hash` is correctly derived from the previous hash and `other_hash`). It never compares the resulting root against any external, trusted root hash.

**Attacker-controlled entry path:** `ProofOfInclusion` is a `pyclass` with `from_py_object` and derives `Streamable`, making it directly constructable from untrusted Python input or deserialized bytes: [3](#0-2) 

The Python binding exposes `valid()` directly: [4](#0-3) 

The existing test suite only validates that corrupting a *middle* layer's `combined_hash` breaks the chain — it does not test whether a fully forged but internally consistent proof is rejected: [5](#0-4) 

### Impact Explanation

An attacker can craft a `ProofOfInclusion` with:
1. An arbitrary `node_hash` (the leaf they wish to falsely claim is included).
2. Arbitrary `other_hash` values per layer.
3. `combined_hash` values computed correctly via `calculate_internal_hash` from the chosen inputs.

`valid()` returns `true` for this forged proof. Any DataLayer consumer that calls `proof.valid()` as the sole validity gate — the pattern used throughout the Python test suite — will accept the forged proof, allowing an attacker to prove inclusion of arbitrary data in any claimed tree root, or to prove exclusion of data that is actually present. This matches the allowed High impact: **DataLayer Merkle proof logic accepts forged inclusion/exclusion and lets untrusted input prove invalid state.**

### Likelihood Explanation

`ProofOfInclusion` is a public, Python-exposed, streamable type. The `valid()` method is the documented and sole validation API. Any DataLayer integration that deserializes a proof from an untrusted peer and calls `proof.valid()` is immediately exploitable with no cryptographic work required — the attacker only needs to compute SHA-256 hashes to build a consistent chain.

### Recommendation

`valid()` must accept a trusted external root hash and compare against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    // existing chain-consistency checks ...
    existing_hash == *trusted_root   // compare against caller-supplied root
}
```

Alternatively, rename the current function to `is_internally_consistent()` to make its limited scope explicit, and add a separate `valid_against_root(root: &Hash) -> bool` that performs the authoritative check. Update all call sites — including the Python binding — to use the root-anchored variant.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId, ProofOfInclusion, ProofOfInclusionLayer, Hash, Side
import hashlib

# Attacker wants to forge a proof that key=999 is in some tree

def sha256_two(a: bytes, b: bytes) -> bytes:
    # mirrors calculate_internal_hash (exact domain depends on implementation)
    return hashlib.sha256(a + b).digest()

fake_leaf_hash = bytes([0xAA] * 32)   # attacker-chosen leaf hash
fake_sibling   = bytes([0xBB] * 32)   # arbitrary sibling

# Compute combined_hash exactly as calculate_internal_hash would
combined = sha256_two(fake_leaf_hash, fake_sibling)

layer = ProofOfInclusionLayer(
    other_hash_side=Side.Right,
    other_hash=Hash(fake_sibling),
    combined_hash=Hash(combined),
)

forged_proof = ProofOfInclusion(
    node_hash=Hash(fake_leaf_hash),
    layers=[layer],
)

# valid() returns True despite this proof being entirely fabricated
assert forged_proof.valid(), "forged proof accepted"
# root_hash() returns the attacker-chosen combined value, not any real tree root
print("forged root:", forged_proof.root_hash())
```
