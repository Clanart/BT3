### Title
Arithmetic Overflow in `denormalize_amount` for High-Decimal Tokens Causes Permanent Fund Freeze or Unbacked Minting — (`near/omni-bridge/src/lib.rs`)

---

### Summary

The `denormalize_amount` helper in `near/omni-bridge/src/lib.rs` performs unchecked `u128` exponentiation and multiplication. For any token whose `origin_decimals − decimals ≥ 39`, the expression `10_u128.pow(diff_decimals)` overflows `u128`. Depending on whether NEAR's WASM runtime is compiled with overflow-checks, this either traps (permanent fund freeze) or silently wraps (unbacked minting / accounting corruption). The token registration path is fully permissionless, so an unprivileged attacker can trigger this for any victim who bridges the token.

---

### Finding Description

`denormalize_amount` and `normalize_amount` are the two functions that scale amounts between the NEAR-side decimal precision (`decimals`) and the origin-chain decimal precision (`origin_decimals`):

```rust
// near/omni-bridge/src/lib.rs  lines 2776-2787
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))          // ← no overflow guard
}

fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))           // ← pow can overflow/wrap
}
```

`u128::MAX ≈ 3.4 × 10^38`, so `10_u128.pow(39)` already exceeds the type's range. Any token for which `origin_decimals − decimals ≥ 39` triggers the overflow.

**How the overflow is reached**

The EVM bridge normalises the on-chain decimal representation to at most 18 via `_normalizeDecimals`:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  lines 586-592
function _normalizeDecimals(uint8 decimals) internal pure returns (uint8) {
    uint8 maxAllowedDecimals = 18;
    if (decimals > maxAllowedDecimals) {
        return maxAllowedDecimals;
    }
    return decimals;
}
```

The EVM bridge token is therefore deployed with `decimals = 18`, but the original token's raw decimal count is preserved as `originDecimals` and emitted in the `DeployToken` event. On the NEAR side, `bind_token_callback` stores both values verbatim:

```rust
// near/omni-bridge/src/lib.rs  lines 1262-1267
self.add_token(
    &deploy_token.token,
    &deploy_token.token_address,
    deploy_token.decimals,          // normalized (≤ 18)
    deploy_token.origin_decimals,   // raw, unbounded
);
```

`add_token` stores them in the `Decimals` struct without any upper-bound validation on `origin_decimals`:

```rust
// near/omni-bridge/src/lib.rs  lines 2724-2735
self.token_decimals.insert(
    token_address,
    &Decimals { decimals, origin_decimals },
);
```

When a transfer from EVM to NEAR is finalised, `fin_transfer_callback` calls `denormalize_amount` with the stored `Decimals`:

```rust
// near/omni-bridge/src/lib.rs  lines 725-727
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
...
fee: Self::denormalize_fee(&init_transfer.fee, decimals),
```

For a token with `origin_decimals = 57` and `decimals = 18`, `diff_decimals = 39`, and `10_u128.pow(39)` overflows.

**Registration is fully permissionless**

`logMetadata` on the EVM bridge accepts any ERC-20 address:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  lines 224-232
function logMetadata(address tokenAddress) external payable {
    ...
    uint8 decimals = IERC20Metadata(tokenAddress).decimals();
    ...
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
```

`deploy_token` and `bind_token` on NEAR require only a valid proof, not a privileged role:

```rust
// near/omni-bridge/src/lib.rs  lines 1136-1144
#[payable]
#[pause(except(roles(Role::DAO)))]
pub fn deploy_token(&mut self, #[serializer(borsh)] args: DeployTokenArgs) -> Promise {
    self.verify_proof(args.chain_kind, args.prover_args).then(...)
}
```

---

### Impact Explanation

Two outcomes depending on WASM compilation flags:

1. **Overflow-checks enabled (trap)**: `10_u128.pow(39)` causes a WASM trap. `fin_transfer_callback` always reverts. The user's tokens are already locked on EVM; the NEAR-side finalisation can never succeed. Funds are permanently frozen — matching the "irrecoverable lock" critical impact.

2. **Overflow-checks disabled (wrap)**: The exponentiation wraps to an arbitrary `u128` value. The multiplication `amount * wrapped_value` also wraps. The NEAR bridge mints a completely wrong token quantity — either far more than the locked collateral (unbacked minting, direct theft) or far less (user loss). This matches the "balance/decimal/accounting corruption that breaks bridge collateralization" high impact.

---

### Likelihood Explanation

- Tokens with `origin_decimals ≥ 57` are uncommon but valid ERC-20 contracts (`decimals()` returns `uint8`, so values up to 255 are legal).
- The registration path (`logMetadata` → `deploy_token` → `bind_token`) requires no privileged role and only a small gas payment.
- A single attacker can deploy such a token, register it, and then any user who bridges it is affected.
- Likelihood is **low-medium**: exotic decimal values are rare in production, but the attack is fully permissionless and requires no cooperation from the victim beyond initiating a bridge transfer.

---

### Recommendation

1. **Validate `origin_decimals` at registration time.** In `add_token` (or `bind_token_callback`), assert that `origin_decimals − decimals < 39` (i.e., `diff_decimals` keeps `10^diff` within `u128`).

2. **Use checked or saturating arithmetic** in `denormalize_amount` and `normalize_amount`:
   ```rust
   fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
       let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
       let scale = 10_u128.checked_pow(diff_decimals)
           .expect("decimal scale overflows u128");
       amount.checked_mul(scale)
           .expect("denormalized amount overflows u128")
   }
   ```

3. **Mirror the EVM cap on NEAR.** Reject any `origin_decimals > 38 + decimals` during `bind_token_callback` to prevent the overflow class entirely.

---

### Proof of Concept

**Setup**

1. Deploy a minimal ERC-20 on EVM whose `decimals()` returns `57`.
2. Call `OmniBridge.logMetadata(tokenAddress)` — permissionless, emits `LogMetadata` with `decimals = 57`.
3. Call `OmniBridge.deployToken(proof, metadata)` on EVM — deploys a bridge token with `_normalizeDecimals(57) = 18` decimals, emits `DeployToken(decimals=18, originDecimals=57)`.
4. Call `near_bridge.deploy_token(proof_of_LogMetadata)` — NEAR deploys omni-token.
5. Call `near_bridge.bind_token(proof_of_DeployToken)` — NEAR stores `Decimals { decimals: 18, origin_decimals: 57 }`.

**Trigger**

6. Victim approves and calls `OmniBridge.initTransfer(tokenAddress, amount, ...)` on EVM — tokens are locked.
7. Relayer calls `near_bridge.fin_transfer(proof_of_InitTransfer)`.
8. `fin_transfer_callback` executes:
   ```rust
   let decimals = Decimals { decimals: 18, origin_decimals: 57 };
   // diff_decimals = 57 - 18 = 39
   // 10_u128.pow(39) > u128::MAX  →  overflow
   Self::denormalize_amount(init_transfer.amount.0, decimals)
   ```
9. With overflow-checks: WASM trap → transaction reverts → victim's tokens permanently locked on EVM.
   With wrapping: arbitrary amount minted on NEAR → accounting corruption / unbacked supply.
