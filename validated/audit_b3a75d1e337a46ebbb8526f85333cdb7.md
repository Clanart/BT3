### Title
Missing Zero-Address Recipient Validation Enables Permanent Fund Lock on EVM Finalization - (File: near/omni-bridge/src/lib.rs, evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

The NEAR-side `init_transfer` function accepts any `OmniAddress` recipient, including EVM zero addresses, without checking `recipient.is_zero()`. The `OmniAddress::is_zero()` helper exists in the codebase but is never called during transfer validation. When a user (accidentally or otherwise) specifies `recipient = eth:0x0000000000000000000000000000000000000000`, the NEAR bridge locks/burns the tokens and the MPC signs a `finTransfer` payload with `payload.recipient = address(0)`. On the EVM side, `finTransfer` in `OmniBridge.sol` also performs no zero-address check before dispatching tokens. The result is either a permanently unclaimable transfer (for bridge tokens and ERC20, where OpenZeppelin reverts on mint/transfer to zero) or permanent ETH loss (for native ETH transfers, where `address(0).call{value:...}` succeeds silently).

---

### Finding Description

**Root cause 1 — NEAR `init_transfer` (near/omni-bridge/src/lib.rs)**

The `init_transfer` function validates only that the recipient chain is not `ChainKind::Near`:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
``` [1](#0-0) 

There is no call to `recipient.is_zero()`. The `OmniAddress::is_zero()` method is fully implemented in `near/omni-types/src/lib.rs` and correctly identifies zero addresses for all EVM chains (`H160::ZERO`), Solana (`SolAddress::ZERO`), StarkNet (`H256::ZERO`), and NEAR (`ZERO_ACCOUNT_ID`): [2](#0-1) 

But it is never invoked in the transfer validation path.

**Root cause 2 — EVM `finTransfer` (evm/src/omni-bridge/contracts/OmniBridge.sol)**

`finTransfer` marks the nonce as used and then dispatches tokens to `payload.recipient` with no zero-address guard:

```solidity
completedTransfers[payload.destinationNonce] = true;
// ... signature verification ...
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
} else if (...) {
    IBridgeToken(payload.tokenAddress).mint(payload.recipient, payload.amount);
} else {
    IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
}
``` [3](#0-2) 

`payload.recipient` is typed as `address` in `BridgeTypes.TransferMessagePayload`: [4](#0-3) 

---

### Impact Explanation

Two distinct permanent-loss scenarios:

1. **Bridge tokens / ERC20 — permanent freeze.** OpenZeppelin's `ERC20._mint` and `ERC20.transfer` both revert when the recipient is `address(0)`. Because `completedTransfers[nonce] = true` is set *before* the token dispatch, the entire transaction reverts (including the nonce flag). The nonce remains valid but every subsequent `finTransfer` call with the same payload will also revert. The tokens on NEAR are already burned/locked. The transfer is permanently unclaimable — funds are irrecoverably frozen.

2. **Native ETH — permanent loss.** When `payload.tokenAddress == address(0)`, the contract sends ETH via `address(0).call{value: payload.amount}("")`. A low-level `.call` to `address(0)` succeeds (no code at that address, treated as an EOA). The call returns `(true, "")`, the nonce is consumed, and the ETH is permanently destroyed.

Both outcomes match the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

Any unprivileged bridge user can trigger this by calling `init_transfer` on NEAR with `recipient = eth:0x0000000000000000000000000000000000000000` (or `arb:`, `base:`, etc.). No privileged role, leaked key, or colluding MPC signer is required — the MPC faithfully signs whatever valid transfer message the NEAR bridge produces. Accidental zero-address entry (e.g., copy-paste error, uninitialized frontend field) is a realistic user-error scenario that the protocol should guard against. The absence of any validation at either the NEAR ingress or the EVM egress makes this reachable with a single user transaction.

---

### Recommendation

**NEAR side (`near/omni-bridge/src/lib.rs`):** Add a zero-address check immediately after the chain check in `init_transfer`:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
require!(
    !init_transfer_msg.recipient.is_zero(),
    BridgeError::InvalidRecipient.as_ref()  // add this error variant
);
```

**EVM side (`evm/src/omni-bridge/contracts/OmniBridge.sol`):** Add a defense-in-depth guard at the top of `finTransfer`:

```solidity
if (payload.recipient == address(0)) revert InvalidRecipient();
```

Both fixes are needed: the NEAR check prevents the MPC from ever signing a zero-recipient payload; the EVM check provides defense-in-depth against any future path that could produce such a payload.

---

### Proof of Concept

1. Attacker (or user) holds a NEAR-side bridged token (e.g., `wrapped-near.near`).
2. Calls `init_transfer` on the NEAR bridge with:
   - `token_id = wrapped-near.near`
   - `amount = 1_000_000`
   - `recipient = OmniAddress::Eth(H160::ZERO)` (i.e., `"eth:0x0000000000000000000000000000000000000000"`)
3. NEAR bridge passes the chain check (`Eth != Near`) and the fee check, burns the tokens, and stores the `TransferMessage` with `recipient = OmniAddress::Eth(H160::ZERO)`.
4. MPC signs the corresponding `TransferMessagePayload` with `payload.recipient = address(0)`.
5. Relayer calls `OmniBridge.finTransfer(signature, payload)` on EVM.
6. `completedTransfers[nonce] = true` is set.
7. `IBridgeToken(payload.tokenAddress).mint(address(0), amount)` is called → OpenZeppelin reverts with `ERC20InvalidReceiver(address(0))`.
8. Entire transaction reverts; nonce flag is cleared.
9. Every future attempt to call `finTransfer` with this payload reverts identically.
10. Tokens on NEAR are permanently burned; EVM tokens are permanently unclaimable. [5](#0-4) [3](#0-2) [6](#0-5)
