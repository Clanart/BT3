### Title
Missing `max_l2_gas_amount` Admission Limit for Declare Transactions in Gateway Stateless Validator - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate_resource_bounds` enforces an upper bound on `l2_gas.max_amount` for `Invoke` and `DeployAccount` transactions but explicitly skips that check for `Declare` transactions. Any unprivileged user can submit a `Declare` transaction with `l2_gas.max_amount = u64::MAX` and have it admitted through the gateway and into the mempool, bypassing the operator-configured admission limit.

---

### Finding Description

Inside `validate_resource_bounds`, the guard reads:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

For `Invoke` and `DeployAccount` transactions the check fires and rejects the transaction when `l2_gas.max_amount > max_l2_gas_amount` (default `1_200_000_000`). [2](#0-1) 

For `Declare` transactions the branch is an empty `if` body — the check is silently skipped. This is confirmed by the dedicated test `valid_l2_gas_amount_on_declare`, which asserts that a declare with `max_amount: GasAmount(200)` passes even when `max_l2_gas_amount: 100`: [3](#0-2) 

The full `validate` call-chain for a `Declare` transaction is:

1. `validate_contract_address` ✓  
2. `validate_empty_account_deployment_data` ✓  
3. `validate_empty_paymaster_data` ✓  
4. `validate_resource_bounds` — **skips `max_l2_gas_amount` check** ✗  
5. `validate_tx_size` — `validate_tx_extended_calldata_size` returns `Ok(())` immediately for `Declare` ✗  
6. `validate_declare_tx` — checks Sierra version and class size only [4](#0-3) 

`validate_tx_extended_calldata_size` also early-returns for `Declare`: [5](#0-4) 

---

### Impact Explanation

**Impact: High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

A `Declare` transaction with `l2_gas.max_amount = u64::MAX` and `l2_gas.max_price_per_unit >= min_gas_price` passes all stateless gateway checks and is forwarded to the mempool. The gateway's own admission policy — which exists precisely to prevent oversized gas-bound transactions from entering the pipeline — is not enforced for this transaction type. The downstream bouncer enforces block capacity during execution, but it enforces a different invariant (actual gas consumed vs. block capacity) and does not substitute for the gateway's admission-limit invariant.

---

### Likelihood Explanation

**Likelihood: High.** The trigger requires no privilege. Any account that is authorized to declare (or when `authorized_declarer_accounts` is `None`, any account) can craft a `Declare` transaction with an arbitrarily large `l2_gas.max_amount`. The bypass is unconditional — there is no configuration flag that re-enables the check for declares. [6](#0-5) 

---

### Recommendation

Apply the same `max_l2_gas_amount` guard to `Declare` transactions. Remove the empty `if let RpcTransaction::Declare(_) = tx {}` branch and let the existing `else if` apply uniformly to all transaction types, or add an explicit check inside `validate_declare_tx`:

```rust
// In validate_resource_bounds — remove the Declare exemption:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

Update the test `valid_l2_gas_amount_on_declare` to expect a rejection, and add a passing case where `max_amount <= max_l2_gas_amount`.

---

### Proof of Concept

1. Construct an `RpcDeclareTransaction::V3` with:
   - `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)`
   - `resource_bounds.l2_gas.max_price_per_unit >= config.min_gas_price`
   - A valid Sierra class (passes `validate_declare_tx`)
2. Submit to the gateway's `add_tx` endpoint.
3. `StatelessTransactionValidator::validate` returns `Ok(())` — the transaction is forwarded to the mempool despite `u64::MAX >> max_l2_gas_amount (1_200_000_000)`.
4. An equivalent `Invoke` or `DeployAccount` transaction with the same `l2_gas.max_amount` would be rejected with `MaxGasAmountTooHigh`. [7](#0-6)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }
