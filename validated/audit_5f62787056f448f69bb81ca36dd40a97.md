Audit Report

## Title
Stranded Router ETH Silently Consumed by Subsequent WETH Payers and Directly Claimable via Unguarded `refundETH` - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`PeripheryPayments.pay()` uses `address(this).balance` — the router's total ETH balance — without per-caller attribution, allowing any subsequent WETH swap to silently consume ETH stranded by a prior user. Independently, `refundETH()` has no access control and transfers the router's entire ETH balance to any caller, giving any address a direct path to claim stranded ETH deposited by others.

## Finding Description
**Root cause — `pay()` (L74):** When `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` globally. If ETH was stranded from a prior `multicall{value}` call that omitted `refundETH`, that ETH is silently consumed to fund the current caller's WETH payment. The current caller's `amountIn` is fully satisfied (the pool receives the correct amount), but part or all of it was funded by a prior user's ETH.

**Root cause — `refundETH()` (L58-63):** No access control. Any EOA or contract can call `refundETH()` at any time and receive `address(this).balance` in full.

**Stranding path (confirmed by test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth`):** A user calls `multicall{value: 2 ether}([exactInputSingle(amountIn=1_000)])` without appending a `refundETH` call. The swap consumes 1_000 wei; the remaining ~2 ether − 1_000 wei is left on the router. The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks direct ETH transfers, not ETH sent with payable function calls, so ETH sent via `multicall{value}` is not restricted.

**Exploit path A — `refundETH()` theft:**
1. User A calls `multicall{value: 2 ether}([exactInputSingle(amountIn=1_000)])` (no `refundETH`). ~2 ether − 1_000 stranded.
2. Attacker calls `router.refundETH()`. `address(this).balance` = stranded amount → transferred to attacker. User A loses their ETH.

**Exploit path B — `pay()` subsidy:**
1. User A strands 1 ether on the router (same as above).
2. Attacker calls `router.exactInputSingle{value: 0}(tokenIn=WETH, amountIn=1 ether, ...)`.
3. In `metricOmmSwapCallback`, `pay(WETH, attacker, pool, 1 ether)` is called. `nativeBalance = 1 ether >= value = 1 ether` → router wraps User A's 1 ether and transfers WETH to pool. Attacker pays 0 WETH from their own balance and receives the full swap output.

Existing guards are insufficient: `receive()` only blocks direct ETH pushes, not payable-function ETH; there is no per-caller ETH accounting in transient storage; `refundETH()` has no `msg.sender` restriction.

## Impact Explanation
Direct loss of user principal. User A's ETH is either silently consumed to subsidize an attacker's WETH swap (attacker receives swap output at zero cost) or directly stolen via `refundETH()`. Both paths result in complete loss of the stranded ETH with no recourse for User A. This meets the Critical/High direct-loss-of-user-principal threshold.

## Likelihood Explanation
The stranding condition is a normal user error explicitly anticipated by the protocol's own test suite (the `refundETH` call in multicall is optional, not enforced). Any user who omits `refundETH` from a multicall with ETH input exposes their ETH. An attacker monitoring the mempool or router balance can immediately follow with `refundETH()` (zero preconditions, zero cost) or a WETH swap. The `refundETH()` path requires no setup, no approvals, and no capital.

## Recommendation
1. **`refundETH()`**: Track per-caller ETH deposits in transient storage at `multicall` entry (store `msg.value` keyed by `msg.sender`). Only refund the amount the current caller deposited in the current transaction, not `address(this).balance`.
2. **`pay()`**: Similarly, only consume `address(this).balance` up to the amount the current `msg.sender` deposited in this call context (i.e., the `msg.value` forwarded through `multicall`). Reject or ignore any ETH balance exceeding that amount.
