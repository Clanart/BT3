### Title
Malicious P2P Peer Can Store Empty State Diff by Sending `state_diff_length=0` in Header — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary
`HeaderStreamBuilder::parse_data_for_block` accepts a `SignedBlockHeader` with `state_diff_length=0` from any peer without verifying the signature cryptographically or cross-checking `state_diff_length` against `state_diff_commitment`. This causes `StateDiffStreamBuilder::parse_data_for_block` to immediately return `ThinStateDiff::default()` for block N, writing an empty state diff to storage while the header's `state_diff_commitment` covers non-empty content.

### Finding Description
The attack path is:

1. Malicious peer sends `SignedBlockHeader { state_diff_length: Some(0), state_diff_commitment: Some(<non_zero>), ... }` for block N.
2. `HeaderStreamBuilder::parse_data_for_block` passes it — only the block number and signature-vector length are checked. [1](#0-0) 
3. `append_header` stores `state_diff_length = Some(0)` and `state_diff_commitment = Some(<non_zero>)` verbatim. [2](#0-1) 
4. `StateDiffStreamBuilder::parse_data_for_block` reads `target_state_diff_len = 0`; the `while` loop body is never executed; `current_state_diff_len (0) != target_state_diff_len (0)` is false; `validate_deprecated_declared_classes_non_conflicting` trivially passes on an empty diff; returns `Ok(Some((ThinStateDiff::default(), N)))`. [5](#0-4) 
5. `write_to_storage` calls `append_state_diff(N, ThinStateDiff::default())`, persisting an empty state diff for block N. [4](#0-3) 

No guard anywhere in the sync path verifies that the stored `ThinStateDiff` hashes to the `state_diff_commitment` recorded in the header.

### Impact Explanation
All storage updates, nonce updates, deployed contracts, and declared classes for block N are silently dropped. Any downstream consumer of the state (RPC, execution, proof verification) will observe the wrong committed state for block N. The stored `state_diff_commitment` in the header is non-zero while the stored `ThinStateDiff` is empty — a concrete, persistent mismatch between committed and stored values.

### Likelihood Explanation
Any peer that the syncing node queries for headers can trigger this. No cryptographic key material, operator access, or special privilege is required — only the ability to respond to a header query, which is the normal role of any p2p peer.

### Recommendation
1. **Verify `state_diff_length` against `state_diff_commitment`** after assembling the full `ThinStateDiff` in `StateDiffStreamBuilder::parse_data_for_block`: compute the commitment of `result` and compare it to the `state_diff_commitment` stored in the header; reject the peer if they differ.
2. **Verify the block signature** cryptographically in `HeaderStreamBuilder::parse_data_for_block` before accepting and storing any header field.
3. As a defence-in-depth measure, reject a header where `state_diff_length == 0` but `state_diff_commitment` is non-zero (or vice-versa) at the header-parsing stage.
