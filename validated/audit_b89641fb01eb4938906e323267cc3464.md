### Title
`DBCol::StateChangesForSplitStates` Is Never Garbage-Collected, Causing Unbounded Disk Growth After Resharding — (`core/store/src/columns.rs`)

### Summary

`DBCol::StateChangesForSplitStates` stores per-block, per-shard state changes produced during shard splits. It is written during resharding and copied to cold storage, but its `gc_policy()` returns `GcPolicy::Other`, which permanently excludes it from every GC path. No dedicated cleanup path exists either. The data accumulates indefinitely on both hot and cold storage after each resharding event, with no mechanism to reclaim it. A TODO comment in the codebase explicitly acknowledges the anomaly.

---

### Finding Description

During a shard split, state changes are recorded into `DBCol::StateChangesForSplitStates` (keyed by `BlockHash || ShardId`). After resharding completes, these entries are no longer needed by any runtime path, but they are never deleted.

The GC policy for this column is `GcPolicy::Other`: [1](#0-0) 

`GcPolicy::Other` means the generic `gc_col()` dispatcher will `unreachable!()` if it is ever called on this column: [2](#0-1) 

No dedicated GC path for `StateChangesForSplitStates` exists anywhere in `clear_block_data()`, `clear_head_block_data()`, `clear_state_transition_data()`, or `gc_parent_shard_after_resharding()`: [3](#0-2) 

Despite being excluded from GC, the column is listed as cold-storage-eligible (`is_cold() = true`), meaning it is also copied to the cold store and accumulates there. A TODO comment in the same function explicitly flags the inconsistency: [4](#0-3) 

The column description itself carries a pending rename TODO, confirming it is still considered active: [5](#0-4) 

---

### Impact Explanation

Every resharding event writes `StateChangesForResharding` blobs (one per block per shard involved in the split) into `DBCol::StateChangesForSplitStates`. Because `GcPolicy::Other` permanently excludes the column from all GC passes and no dedicated cleanup path exists, these blobs accumulate indefinitely in both hot RocksDB and cold storage. On non-archival nodes — which are supposed to bound their disk usage via GC — this constitutes unbounded disk growth proportional to the number of resharding events and the size of the state at each split boundary. With dynamic resharding enabling more frequent splits, the growth rate increases over time.

---

### Likelihood Explanation

Resharding has already occurred on mainnet multiple times. Every past resharding event has already written data into this column that will never be reclaimed. Future resharding events (including dynamic resharding) will continue to add to it. The bug is reachable through the normal protocol upgrade path and requires no special attacker action.

---

### Recommendation

Add a dedicated GC path for `DBCol::StateChangesForSplitStates`. The correct trigger point is `gc_parent_shard_after_resharding()` in `chain/chain/src/garbage_collection.rs`: once GC reaches the last block of the epoch in which a shard split occurred and confirms no child shard still maps to the parent, iterate over all `StateChangesForSplitStates` keys for the affected `(block_hash, shard_id)` pairs and delete them. Alternatively, change the column's `gc_policy()` to `GcPolicy::Delete` and add it to the per-block deletion loop in `clear_block_data()`, mirroring how `DBCol::StateChanges` is handled. The `is_cold()` classification should also be re-evaluated: if the column is not needed for archival replay, it should be removed from cold storage.

---

### Proof of Concept

1. Run a node through a resharding event (static or dynamic).
2. After resharding completes and several GC epochs pass, inspect the RocksDB column `StateChangesForSplitStates` using `ldb` or the state-viewer tool.
3. Observe that all entries written at the resharding block height are still present, while all other per-block columns (e.g., `IncomingReceipts`, `ChunkExtra`, `StateChanges`) for the same heights have been deleted by GC.
4. The TODO comment at `core/store/src/columns.rs` line 578 (`// TODO StateChangesForSplitStates is not GC-ed, why is it here?`) confirms the developers have already observed this anomaly. [6](#0-5) [7](#0-6)

fn gc_parent_shard_after_resharding(
    chain_store_update: &mut ChainStoreUpdate,
    epoch_manager: &dyn EpochManagerAdapter,
    block_hash: &CryptoHash,
) -> Result<(), Error> {
    // Clear out state for the parent shard. Note that this function is called at every epoch boundary,
    // even if there is no resharding.
    // It's fine to do that as after the first call to `trie_store_update.delete_shard_uid_prefixed_state`
    // the rest of the calls in future epochs are no-ops.
    if !epoch_manager.is_last_block_in_finished_epoch(block_hash)? {
        return Ok(());
    }

    tracing::debug!(target: "garbage_collection", ?block_hash, "resharding state cleanup");
    // Given block_hash is the resharding block, shard_layout is the shard layout of the next epoch
    // Important: We are not allowed to call `epoch_manager.get_shard_layout_from_prev_block()` as
    // the function relies on `self.get_block_info(block_info.epoch_first_block())` but epoch_first_block
    // has already been cleaned up.
    // We instead need to rely on chain_store to get the next block hash and use the block_info to get
    // the next epoch id and shard layout.
    let store = chain_store_update.store();
    let next_block_hash = store.chain_store().get_next_block_hash(block_hash)?;
    let next_epoch_id = epoch_manager.get_epoch_id(&next_block_hash)?;
    let shard_layout = epoch_manager.get_shard_layout(&next_epoch_id)?;
    let mut trie_store_update = store.trie_store().store_update();
    for parent_shard_uid in shard_layout.get_split_parent_shard_uids() {
        // Check if any child shard still map to this parent shard
        let children_shards =
            shard_layout.get_children_shards_uids(parent_shard_uid.shard_id()).unwrap();
        let has_active_mapping = children_shards.into_iter().any(|child_shard_uid| {
            let mapped_shard_uid = maybe_get_shard_uid_mapping(&store, child_shard_uid);
            mapped_shard_uid.as_ref() == Some(&parent_shard_uid)
        });
        if !has_active_mapping {
            // Delete the state of the parent shard
            tracing::debug!(target: "garbage_collection", ?parent_shard_uid, "resharding state cleanup for shard");
            trie_store_update.delete_shard_uid_prefixed_state(parent_shard_uid);
        } else {
            tracing::debug!(target: "garbage_collection", ?parent_shard_uid, "skipping parent shard cleanup - active mappings exist");
        }
    }

    chain_store_update.merge(trie_store_update.into());
    Ok(())
}
