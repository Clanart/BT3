### Title
`ProofOfInclusion::valid()` Is Self-Referential and Never Verifies Against a Trusted Root Hash — (`File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

---

### Summary

`ProofOfInclusion::valid()` only checks internal consistency of the proof chain. Its final step compares `existing_hash` against `self.root_hash()`, but `root_hash()` is derived from the proof's own last `combined_hash` field — not from any externally trusted committed root. An attacker can construct a fully self-consistent `ProofOfInclusion` for any arbitrary `node_hash` (i.e., any key-value pair they wish to claim is included) and `valid()` will return `true`. The struct is fully constructable from Python via `from_py_object`, and `valid()` is exposed as a Python binding.

---

### Finding Description

`root_hash()` returns the `combined_hash` of the last layer in the proof: [1](#0-0) 

`valid()` iterates over layers, verifying each `calculated_hash == layer.combined_hash`, then sets `existing_hash = calculated_hash`. After the loop, `existing_hash` equals the last `calculated_hash`, which was just asserted to equal the last `layer.combined_hash`. The final check is therefore:

```
existing_hash == self.root_hash()
  ≡ last.combined_hash == last.combined_hash
  ≡ always true
``` [2](#0-1) 

The method never accepts an external `expected_root` parameter. There is no comparison against any committed, trusted tree root. The proof is entirely self-validating.

`ProofOfInclusion` is exposed to Python with `pyclass(get_all, from_py_object)`, meaning Python code can construct an arbitrary proof object and call `.valid()` on it: [3](#0-2) 

The Python binding directly exposes `py_valid()`: [4](#0-3) 

The Python type stubs and wheel API also expose `ProofOfInclusion` and `ProofOfInclusionLayer` as first-class Python types: [5](#0-4) 

---

### Impact Explanation

Any Python-level DataLayer code that calls `proof.valid()` to decide whether a key-value pair is committed in a DataLayer store root is vulnerable. An attacker who can supply a `ProofOfInclusion` object (e.g., via a network peer, a DataLayer offer, or any deserialized proof) can forge a proof for any `node_hash` they choose. The forged proof will pass `valid()` as long as each layer's `combined_hash` is set to the hash computed from the previous step — a trivial computation. The attacker-controlled `root_hash()` will be whatever they put in the last `combined_hash`, with no check against the actual committed DataLayer root.

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

---

### Likelihood Explanation

The `valid()` API is the only validation method on `ProofOfInclusion`. Any downstream consumer of this API (in chia-blockchain or other Python code) that relies on `proof.valid()` alone to gate DataLayer state transitions is directly exploitable. The `ProofOfInclusion` struct is deserializable from bytes via `Streamable` and constructable from Python via `from_py_object`, giving an unprivileged attacker a direct entry path. No privileged role or key material is required.

---

### Recommendation

`valid()` must accept an external trusted root hash and compare against it instead of `self.root_hash()`:

```rust
pub fn valid(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }

    &existing_hash == expected_root  // compare against caller-supplied trusted root
}
```

The Python binding `py_valid()` must be updated to require the trusted root as a parameter. All call sites — including the fuzz target at `crates/chia-datalayer/fuzz/fuzz_targets/proofs_of_inclusion.rs` — must be updated to supply the root obtained from `MerkleBlob` directly. [6](#0-5)

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer, Side
import hashlib

# Attacker wants to forge a proof that node_hash X is in the tree.
# They pick any node_hash they want to claim is included.
fake_node_hash = bytes([0xAB] * 32)
other_hash     = bytes([0xCD] * 32)

# Compute what combined_hash must be for the layer to be internally consistent.
# (using whatever calculate_internal_hash does — SHA256 of the two hashes in order)
# The attacker just needs to compute one hash to make the chain consistent.
combined = hashlib.sha256(fake_node_hash + other_hash).digest()  # simplified

layer = ProofOfInclusionLayer(
    other_hash_side=Side.Right,
    other_hash=other_hash,
    combined_hash=combined,   # attacker sets this to match their computation
)

proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True even though fake_node_hash is not in any real tree.
# root_hash() returns `combined` — the attacker's chosen value, not any committed root.
assert proof.valid() == True
```

The analog to the Jet staking bug is direct: just as `unbond()` let an attacker independently adjust `shares_bonded` to shift the conversion rate to any desired value, here an attacker independently sets each `combined_hash` field to shift the "root" to any desired value — and `valid()` accepts it because it checks the proof only against itself.
