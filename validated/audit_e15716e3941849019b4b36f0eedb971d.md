### Title
Dust-Amount `init_transfer` Permanently Burns/Locks User Tokens When Normalized Amount Is Zero — (`File: near/omni-bridge/src/lib.rs`)

---

### Summary

When a user initiates a NEAR→EVM/Solana/StarkNet bridge transfer via `ft_transfer_call`, the bridge immediately burns or locks the full token amount in `init_transfer_internal`. The zero-normalized-amount guard only exists in `sign_transfer`, which is called later by a relayer. If the transferred amount (minus fee) is smaller than one unit of the destination token's precision (`10^(origin_decimals - decimals)`), `normalize_amount` returns 0, `sign_transfer` permanently reverts, and the user's tokens are irrecoverably burned/locked with no cancel path.

---

### Finding Description

**Step 1 — Token burn/lock happens unconditionally in `init_transfer_internal`:** [1](#0-0) 

`init_transfer_internal` burns deployed tokens or locks native tokens for the full `transfer_message.amount` and returns `U128(0)` to the NEP-141 token contract, meaning "keep all tokens — no refund." No check on the normalized amount is performed here.

**Step 2 — `normalize_amount` uses floor division:** [2](#0-1) 

For a token with `origin_decimals = 24` and `decimals = 18`, `diff_decimals = 6`, so any amount below `10^6 = 1,000,000` normalizes to 0.

**Step 3 — The zero-amount guard is only in `sign_transfer`, called later by a relayer:** [3](#0-2) 

`sign_transfer` computes `normalize_amount(amount_without_fee(), decimals)` and panics with `BridgeError::InvalidAmountToTransfer` if the result is 0. This panic is permanent — every future call to `sign_transfer` for this transfer ID will also revert.

**Step 4 — No cancel/refund path exists for the user:**

`remove_transfer_message` is only called from:
- `sign_transfer_callback` — only on successful MPC signing (unreachable here)
- `claim_fee_callback` — only when fee > 0 and a proof is submitted (inapplicable)
- `submit_transfer_to_utxo_chain_connector` — only for BTC/UTXO chains [4](#0-3) 

There is no public `cancel_transfer` function. The `update_transfer_fee` function only allows increasing the fee (not decreasing it), so the user cannot adjust their way out. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds.**

Any user who sends a token amount below the minimum transferable unit (i.e., `amount - fee < 10^(origin_decimals - decimals)`) loses all their tokens permanently. For a token with `origin_decimals = 24` and `decimals = 18`, any transfer of fewer than 1,000,000 base units (with zero fee) results in total loss. The tokens are burned (for deployed bridge tokens) or locked in the bridge (for native tokens) with no recovery mechanism.

---

### Likelihood Explanation

**Medium.** This requires a token pair where `origin_decimals > decimals` (decimal normalization is active). Such pairs are explicitly supported and registered via `token_decimals`. A user who is unaware of the minimum transferable unit — or who is testing with a small amount — can trigger this silently. The `ft_transfer_call` succeeds (returns 0 tokens refunded), the `InitTransferEvent` is emitted, and the user has no indication that their transfer can never be completed until a relayer attempts `sign_transfer` and it reverts.

---

### Recommendation

Add a normalized-amount check in `init_transfer` (or `init_transfer_internal`) **before** burning/locking tokens. If `normalize_amount(amount - fee, decimals) == 0`, return the full amount as a refund (return `transfer_message.amount` from `init_transfer_internal`) rather than proceeding to burn/lock.

```rust
// In init_transfer_internal, before burn_tokens_if_needed:
let normalized = Self::normalize_amount(
    transfer_message.amount_without_fee()
        .expect("invalid fee"),
    decimals,
);
if normalized == 0 {
    self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
    return transfer_message.amount; // refund all tokens
}
```

This mirrors the fix applied in the referenced AToken report (MR#79): revert/refund when the normalized output is zero, before consuming the input.

---

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 18` (diff = 6, minimum unit = 1,000,000).
2. User calls `ft_transfer_call` on the token contract with `amount = 500,000` (below minimum unit), `fee = 0`, targeting an EVM recipient.
3. Bridge's `ft_on_transfer` → `init_transfer` → `init_transfer_internal`:
   - Transfer message stored in `pending_transfers`.
   - `burn_tokens_if_needed` burns 500,000 tokens (for a deployed token).
   - Returns `U128(0)` → NEP-141 keeps all tokens (no refund to user).
4. Relayer calls `sign_transfer` for this transfer ID:
   - `normalize_amount(500_000, Decimals { origin: 24, decimals: 18 })` = `500_000 / 1_000_000` = **0**.
   - `require!(0 > 0, ...)` → **panics** with `InvalidAmountToTransfer`.
5. Every subsequent `sign_transfer` call for this ID also panics.
6. No cancel function exists. The 500,000 tokens are permanently burned. The transfer message remains in `pending_transfers` indefinitely. [6](#0-5) [3](#0-2) [7](#0-6)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }

    fn remove_transfer_message(&mut self, transfer_id: TransferId) -> TransferMessage {
        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);
        }

        transfer.message
    }

    /// Uses floor division — any sub-unit remainder ("dust") is truncated and not transferred
    /// to the destination chain. When fee > 0, dust is absorbed into the fee via `claim_fee`.
    /// When fee = 0, dust stays locked/burned. See SECURITY.md for details.
    fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
    }
