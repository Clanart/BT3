### Title
Missing Zero-Amount Check After Decimal Normalization in `sign_transfer` Permanently Locks User Tokens — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

When a user initiates a NEAR→EVM bridge transfer with an amount that normalizes to zero after decimal scaling (i.e., `amount - fee < 10^(origin_decimals - decimals)`), the tokens are irreversibly locked or burned inside `init_transfer_internal`, but `sign_transfer` always reverts with `ERR_INVALID_AMOUNT_TO_TRANSFER`. No cancel or recovery path exists, making the lock permanent.

---

### Finding Description

**Step 1 — Insufficient validation in `init_transfer`**

`init_transfer` enforces only:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

This check passes for any `amount ≥ 1` with `fee = 0`. It does **not** verify that `normalize_amount(amount - fee, decimals) > 0`.

**Step 2 — Tokens are locked/burned unconditionally**

`init_transfer_internal` immediately burns (for deployed tokens) or locks the full `amount`:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
``` [2](#0-1) 

**Step 3 — `sign_transfer` always reverts for sub-unit amounts**

`sign_transfer` computes the normalized amount via integer division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [3](#0-2) 

If `amount - fee < 10^(origin_decimals - decimals)`, the result is `0`. The subsequent guard:

```rust
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [4](#0-3) 

causes `sign_transfer` to revert on every call for this transfer. The MPC signer is never reached, so `sign_transfer_callback` — the only place that calls `remove_transfer_message` — is never executed. [5](#0-4) 

**Step 4 — No recovery path**

`update_transfer_fee` only allows the fee to increase (`fee.fee >= current_fee.fee`), so it cannot be used to make `amount - fee` larger. [6](#0-5) 

There is no public `cancel_transfer` function. The pending transfer entry remains in `pending_transfers` indefinitely, and the locked/burned tokens are irrecoverable.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

Any user who initiates a NEAR→EVM transfer where `amount - fee` is below the decimal scaling threshold has their tokens permanently locked in the bridge contract (or burned, for deployed bridge tokens) with no recovery mechanism. The transfer record persists in `pending_transfers` but can never be progressed or cancelled.

Concrete example: a token registered with `origin_decimals = 24` (NEAR-native yoctoNEAR scale) and `decimals = 18` (EVM scale) has a scaling factor of `10^6`. Any transfer with `amount - fee < 1_000_000` normalizes to zero. A user sending `amount = 500_000` with `fee = 0` passes the `fee < amount` guard, has 500,000 tokens locked, and can never retrieve them.

---

### Likelihood Explanation

**Medium.** The condition is reachable by any unprivileged user calling `ft_transfer_call` on the NEAR bridge contract. It requires:
1. A token whose `origin_decimals > decimals` (common for NEAR-native tokens bridged to EVM, e.g., 24 → 18).
2. A transfer amount below the scaling threshold (e.g., < 10^6 for a 24→18 token).

A user can trigger this accidentally (sending a "dust" amount) or an attacker can deliberately construct such a transfer to permanently destroy their own tokens (griefing is not the concern here — the concern is accidental self-lock). The protocol provides no guardrail at the entry point.

---

### Recommendation

Add a zero-amount check at the entry point in `init_transfer` (or `init_transfer_internal`) before tokens are locked or burned:

```rust
// After building transfer_message, before locking tokens:
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(addr) = token_address {
    if let Some(decimals) = self.token_decimals.get(&addr) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee()
                .unwrap_or(0),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

This mirrors the fix recommended in POSU-1: add an explicit zero check before the operation that would otherwise become permanently stuck.

---

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 18` (scaling factor = `10^6`).
2. User calls `ft_transfer_call` on the NEAR bridge with `amount = 500_000`, `fee = 0`, `recipient = eth:0x...`.
3. `init_transfer` passes: `0 < 500_000` ✓. Tokens are locked in `init_transfer_internal`.
4. Relayer calls `sign_transfer` for this `transfer_id`.
5. `normalize_amount(500_000, {decimals:18, origin_decimals:24}) = 500_000 / 1_000_000 = 0`.
6. `require!(0 > 0, ...)` → panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
7. Relayer retries indefinitely — same result every time.
8. `update_transfer_fee` cannot help (fee can only increase, making `amount - fee` smaller).
9. No cancel function exists. The 500,000 tokens are permanently locked. [7](#0-6) [3](#0-2) [1](#0-0) [2](#0-1)
