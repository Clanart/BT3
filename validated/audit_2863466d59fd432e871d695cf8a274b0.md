Audit Report

## Title
Unguarded Shared ETH Balance in `pay()` Lets Any Caller Drain Stranded Native ETH from the Router — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` uses `address(this).balance` as a shared, unguarded ETH pool when settling WETH-input swaps. When the router holds ETH left over from a prior caller's incomplete `multicall` (i.e., `msg.value` exceeded the swap amount and `refundETH()` was omitted), any subsequent WETH-input swap caller receives output tokens for free — the router wraps the stranded ETH to pay the pool and never charges the attacker's address.

## Finding Description
`pay()` at lines 73–84 of `PeripheryPayments.sol` branches on `address(this).balance` when `token == WETH`:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);   // payer never charged
}
```

There is no per-caller accounting. The `payer` argument (set to `msg.sender` via transient storage in `_setNextCallbackContext` at line 71 of `MetricOmmSimpleRouter.sol`) is completely ignored in this branch.

ETH accumulates in the router because all swap entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`) are `payable`. The `receive()` guard at lines 32–34 only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from being left over from a `payable` call where `msg.value` exceeds the consumed amount.

The full call path for the exploit:
1. `exactInputSingle` (payable, line 67) → sets payer to `msg.sender` in transient storage
2. Pool calls `metricOmmSwapCallback` (line 46) → `_justPayCallback` (line 192)
3. `_justPayCallback` calls `pay(WETH, attacker, pool, amountIn)` (lines 193–198)
4. `pay()` finds `address(this).balance >= amountIn`, wraps the router's own ETH, sends WETH to pool — attacker pays nothing

## Impact Explanation
Direct theft of user principal. Any ETH stranded in the router (from any prior caller's incomplete `multicall`) can be stolen by an unprivileged attacker calling a WETH-input swap. The attacker receives output tokens worth the full stranded ETH amount while paying zero. The prior user permanently loses their ETH. This is a direct loss of user funds meeting Critical/High severity thresholds.

## Likelihood Explanation
The `multicall` + `refundETH` pattern is the documented way to use native ETH with the router, but omitting `refundETH` is a common integration mistake. No special permissions, token approvals, or privileged roles are required — only the ability to call a public swap function. Any MEV bot monitoring the router's ETH balance can atomically exploit the window in the same block. The attack is repeatable and requires no prior setup.

## Recommendation
Track per-transaction ETH contributions in transient storage rather than reading the global `address(this).balance`. At each `payable` entry point, record `msg.value` in a transient slot keyed to the current transaction. In `pay()`, consume only from that recorded amount and fall back to `safeTransferFrom` for any remainder, reverting if the recorded amount is insufficient. Alternatively, remove the native-ETH shortcut from `pay()` entirely and require callers to wrap ETH themselves (e.g., via a `wrapETH` helper in `multicall`) before calling the router.

## Proof of Concept

```solidity
// Step 1 – User A leaves 0.5 ETH stranded in the router
router.multicall{value: 1 ether}([
    abi.encodeCall(router.exactInputSingle, (ExactInputSingleParams({
        pool:             pool,
        tokenIn:          WETH,
        tokenOut:         token1,
        zeroForOne:       true,
        amountIn:         0.5 ether,
        amountOutMinimum: 0,
        recipient:        userA,
        deadline:         block.timestamp + 1,
        priceLimitX64:    0,
        extensionData:    ""
    })))
    // ← no refundETH(); 0.5 ETH stays in router
]);

// Step 2 – Attacker exploits the stranded ETH
// Attacker has zero WETH, sends zero ETH, has zero WETH approval
router.exactInputSingle(ExactInputSingleParams({
    pool:             pool,
    tokenIn:          WETH,
    tokenOut:         token1,
    zeroForOne:       true,
    amountIn:         0.5 ether,   // <= router.balance
    amountOutMinimum: 0,
    recipient:        attacker,
    deadline:         block.timestamp + 1,
    priceLimitX64:    0,
    extensionData:    ""
}));
// Pool calls metricOmmSwapCallback → _justPayCallback → pay(WETH, attacker, pool, 0.5e18)
// address(this).balance == 0.5 ETH >= 0.5 ETH
// → router wraps its own ETH, sends WETH to pool
// → attacker receives token1 worth 0.5 ETH; User A's 0.5 ETH is permanently stolen
```
