### Title
`batch_insert` Missing Duplicate-Key Validation Corrupts DataLayer Merkle Tree and Cache — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` does not validate for duplicate keys — either within the supplied batch or against keys already present in the tree — unlike the single-item `insert` path. When a duplicate key reaches the batch fast-path, two leaf nodes with the same `KeyId` are written into the blob. The `key_to_index` cache is silently overwritten to point only to the later leaf, leaving the earlier leaf as an unreachable "ghost" node that is still woven into the tree structure and therefore still contributes to the Merkle root hash. The result is a permanently corrupted tree whose root hash cannot be reproduced by any valid proof path.

---

### Finding Description

**Single-item `insert` guards against duplicates:** [1](#0-0) 

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

**`batch_insert` omits both guards for the bulk path:** [2](#0-1) 

When `leaf_count > 1` the entire batch bypasses `insert` and goes directly to the raw write loop. When `leaf_count <= 1` only the last two items (popped via `Vec::pop`) go through `insert`; every earlier item in the vector is written without any duplicate check.

**`insert_entry_to_blob` unconditionally calls `add_leaf`, which silently overwrites the cache:** [3](#0-2) 

```rust
match block.node {
    Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
    Node::Internal(..) => self.block_status_cache.add_internal(index),
}
```

`add_leaf` does: [4](#0-3) 

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // overwrites silently
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
```

**Concrete corruption sequence (tree already has ≥ 2 leaves):**

1. `batch_insert` is called with `[(key_A, val1, hash1), (key_A, val2, hash2)]`.
2. First iteration: leaf at index `I1` written; `key_to_index[A] = I1`.
3. Second iteration: leaf at index `I2` written; `key_to_index[A] = I2` (overwrites).
4. `I1` is **not** in `free_indexes` (removed by `add_leaf`) and **not** in `key_to_index` — it is a ghost node permanently embedded in the tree structure.
5. The Merkle root is computed over both leaves; `get_proof_of_inclusion(A)` returns a proof only for `I2`.
6. The root hash is now unreproducible from any single valid proof path.

The same corruption occurs when a key already present in the tree is re-submitted in a batch (scenario: existing `key_A` at `I_old`; batch inserts `key_A` again at `I_new`; `I_old` becomes a ghost).

**`check_integrity` would detect this**, but it is only enabled in test builds: [5](#0-4) 

```rust
let self_ = Self {
    blob,
    block_status_cache,
    check_integrity_on_drop: cfg!(test),   // false in production
};
```

---

### Impact Explanation

The DataLayer Merkle root hash is the commitment published on-chain. A corrupted root — one that includes ghost leaf nodes — cannot be reproduced by any proof the API can generate. This means:

- **Forged exclusion**: a key that is genuinely in the tree cannot be proven present (its proof path traverses the ghost node whose index is no longer reachable via the cache).
- **Root corruption**: the committed root hash diverges from what any honest proof can reconstruct, breaking all subsequent inclusion/exclusion proofs for the affected store.
- **Persistent**: the ghost node is never reclaimed (its index is absent from both `key_to_index` and `free_indexes`), so the corruption survives across serialise/deserialise cycles via `MerkleBlob::new` → `BlockStatusCache::new`.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

`batch_insert` is the primary bulk-write path used during DataLayer delta synchronisation. Any DataLayer update message that carries a repeated `KeyId` — whether crafted by an adversary or produced by a buggy peer — reaches this path without sanitisation. The Python binding exposes `batch_insert` directly: [6](#0-5) 

No deduplication is enforced at the binding layer. An unprivileged peer that can submit a DataLayer delta containing a duplicate key can trigger the corruption deterministically.

---

### Recommendation

Add duplicate-key validation inside the batch fast-path, mirroring the guard already present in `insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    // ... existing insertion logic
}
```

Alternatively, deduplicate the input vector before entering the loop, keeping only the last occurrence of each key (consistent with an "upsert" semantic) and emitting an error for strict-insert semantics.

---

### Proof of Concept

```rust
use chia_datalayer::merkle::blob::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

let mut blob = MerkleBlob::new(vec![]).unwrap();

// Seed the tree with 2 leaves so leaf_count > 1 (bypasses the insert path entirely)
for i in 0i64..2 {
    blob.insert(KeyId(i), ValueId(i),
        &Hash(Bytes32::new([i as u8; 32])),
        InsertLocation::Auto {}).unwrap();
}

// batch_insert with a duplicate key — no error is returned
blob.batch_insert(vec![
    ((KeyId(99), ValueId(1)), Hash(Bytes32::new([0xAA; 32]))),
    ((KeyId(99), ValueId(2)), Hash(Bytes32::new([0xBB; 32]))),  // same key!
]).unwrap();

// check_integrity reveals the corruption (leaf_count ≠ key_to_index cache length,
// or IntegrityKeyToIndexCacheIndex mismatch for the ghost leaf)
blob.check_integrity().unwrap_err(); // panics / errors in production-equivalent run
```

The ghost leaf at the first `KeyId(99)` index is permanently embedded in the tree, making the root hash unreproducible by any proof the public API can generate.
