Looking at the actual code in `_fillBids`:

Audit Report

## Title
Wrong Bin Length Subtracted in `_fillBids` Corrupts Bid-Side Price Boundaries ‚Äî (`metric-periphery/contracts/lens/MetricOmmPoolDataProvider.sol`)

## Summary

In `MetricOmmPoolDataProvider._fillBids`, the downward walk loop decrements `walkDistE6` by `lenAbove` ‚Äî the `lengthE6` of bin `b+1` (the bin above) ‚Äî instead of the `lengthE6` of the current bin `b`. This produces wrong lower and upper price boundaries for every bin below `curBinIdx`, corrupting `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` in the bid depth ladder returned by `getLiquidityDepth`. Integrations that size slippage guards from this data are induced into fund-losing trades.

## Finding Description

`walkDistE6` is initialized to `curBinDistFromProvidedPriceE6`, the lower edge of `curBinIdx`. The correct invariant for the downward walk is:

```
lower(b) = lower(b+1) - lengthE6(b)
```

The ask-side walk (`_fillAskRow`) implements the analogous invariant correctly ‚Äî after processing bin `b` it advances `cumDistE6` by `lengthE6` of that same bin `b`:

```solidity
// line 441 ‚Äî correct: advances by current bin's own length
ctx.cumDistE6 += int256(uint256(lengthE6));
```

The bid-side loop does the opposite. At lines 496‚Äì498:

```solidity
(,, uint16 lenAbove,,) = PoolStateLibrary._binState(pool, int8(b + 1)); // reads bin b+1
walkDistE6 -= int256(uint256(lenAbove));                                  // subtracts b+1's length ‚Üê WRONG
```

It reads `lenAbove` from bin `b+1` and subtracts that, so what the code actually computes is:

```
walkDistE6 = lower(b+1) - lengthE6(b+1)   ‚Üê uses the wrong bin's length
```

Then at lines 503‚Äì506 it uses this wrong `walkDistE6` to derive both `lowerX64` and `upperX64` for bin `b`:

```solidity
uint256 lowerX64 = _priceFromMidAndDistE6(midPriceX64, walkDistE6, Math.Rounding.Floor);
uint256 upperX64 =
    _priceFromMidAndDistE6(midPriceX64, walkDistE6 + int256(uint256(lengthE6)), Math.Rounding.Floor);
```

Whenever any two adjacent bins have different `lengthE6` values, both `lowerX64` and `upperX64` are wrong, and the error accumulates with each hop downward. The `mStartX64`/`mEndX64` fed into `_accumulateBidLevel` are therefore wrong, producing wrong `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` for every bid level below `curBinIdx`.

No existing guard catches this: the function is a pure view walk with no cross-check against actual pool bin boundaries.

## Impact Explanation

`getLiquidityDepth` is the protocol's off-chain depth oracle. Integrations (aggregators, front-ends, market-making bots) consume `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` to estimate execution price for a given sell size and to set `amountOutMinimum` / `priceLimitX64` before calling `exactInput` or `exactInputSingle` on the router. When the bid depth ladder reports systematically shifted prices (wrong bin boundaries ‚Üí wrong marginal prices ‚Üí wrong fee-adjusted execution prices), callers who size their slippage guard from this data will accept a worse price than intended ‚Äî a direct loss of output tokens ‚Äî or revert unnecessarily. This matches the "Lens/quoter path" Smart Audit Pivot and the "Bad-price execution" allowed impact gate.

## Likelihood Explanation

The bug fires on every call to `getLiquidityDepth` for any pool where at least two adjacent bins below `curBinIdx` have different `lengthE6` values. Heterogeneous bin lengths are a normal and expected pool configuration (the factory explicitly supports per-bin `lengthE6`). No special attacker capability is required; any unprivileged caller of `getLiquidityDepth` or any integration consuming its output triggers the corrupted output. The error is deterministic and repeatable.

## Recommendation

Remove the `lenAbove` read entirely and subtract the current bin's own `lengthE6` from `walkDistE6` after reading the current bin's state:

```solidity
for (int256 b = int256(curBinIdx) - 1; b >= int256(lowCap); b--) {
    int8 binIdx = int8(b);
    (, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx);
    walkDistE6 -= int256(uint256(lengthE6)); // subtract current bin's own length

    uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(uint256(addFeeSellE6), Q64, ONE_E6);
    uint256 lowerX64 = _priceFromMidAndDistE6(midPriceX64, walkDistE6, Math.Rounding.Floor);
    uint256 upperX64 =
        _priceFromMidAndDistE6(midPriceX64, walkDistE6 + int256(uint256(lengthE6)), Math.Rounding.Floor);
    // ... rest unchanged
}
```

This mirrors the ask-side pattern where `cumDistE6 += lengthE6` uses the just-read current bin's length.

## Proof of Concept

Deploy a pool with three bins where `lengthE6(curBinIdx) = 10_000`, `lengthE6(curBinIdx-1) = 20_000`, `lengthE6(curBinIdx-2) = 15_000`. Call `getLiquidityDepth`. Observe:

- Bin `curBinIdx-1`: code subtracts `10_000` (bin above's length) ‚Üí `walkDistE6 = -10_000`; correct value is `-20_000`.
- Bin `curBinIdx-2`: code subtracts `20_000` (bin `curBinIdx-1`'s length) ‚Üí `walkDistE6 = -30_000`; correct value is `-35_000`.

Both `lowerX64`/`upperX64` and the resulting `binAvgExecPriceX64`/`cumulativeAvgExecPriceX64` are wrong for every bin below `curBinIdx`. A Foundry fork test asserting `bids[1].binAvgExecPriceX64 == expectedCorrectPrice` will fail with the current code and pass after the fix.
