### Title
`batch_insert` Bypasses Duplicate Key Check, Corrupting DataLayer Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces, allowing a caller to silently insert multiple leaf nodes with the same `KeyId` or `Hash` into the Merkle tree. The resulting blob contains more leaf nodes than the `BlockStatusCache` tracks, producing a corrupted tree root hash and breaking proof-of-inclusion validity.

### Finding Description

`MerkleBlob::insert` guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` calls `self.insert()` (with its guards) only for the last two items popped from the input vector when the tree has ≤ 1 existing leaves. All remaining items are written directly via `insert_entry_to_blob`, which has no duplicate check: [1](#0-0) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which calls `HashMap::insert` — silently overwriting the previous entry for a duplicate key: [2](#0-1) 

The result is:
- The raw blob contains **two** leaf nodes with the same `KeyId`.
- `block_status_cache.key_to_index` only tracks the **last** one (HashMap overwrites).
- `leaf_count` (from iterating the blob) exceeds `key_to_index_cache_length` (from the HashMap), so `check_integrity()` will return `IntegrityKeyToIndexCacheLength`.
- The Merkle root hash is computed over both duplicate leaves, producing a root that does not correspond to the intended key-value set.

The Python binding `py_batch_insert` exposes this path directly: [3](#0-2) 

### Impact Explanation

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state.**

A caller supplying a `batch_insert` list with a repeated `KeyId` (or repeated `Hash`) produces a tree whose root hash is computed over phantom duplicate leaves. Any `get_proof_of_inclusion` generated from this tree is verified against a wrong root. Downstream DataLayer consumers that trust the root hash will accept proofs for a state that was never committed, or reject proofs for a state that was.

### Likelihood Explanation

`batch_insert` is the primary bulk-load path used by the Python DataLayer node (called via `py_batch_insert`). DataLayer updates are assembled from peer-supplied data; a peer that can influence the key-value list passed to `batch_insert` can trigger this path. No privileged role is required — only the ability to supply a list with a repeated key.

### Recommendation

Add the same duplicate guards at the top of the `batch_insert` fast path (the `for ((key, value), hash) in keys_values_hashes` loop), either by:

1. Checking `self.block_status_cache.contains_key(key)` and `self.block_status_cache.contains_leaf_hash(&hash)` before each `insert_entry_to_blob` call, returning `Err(Error::KeyAlreadyPresent())` / `Err(Error::HashAlreadyPresent())` on collision; or
2. Pre-deduplicating the input vector before entering the fast path (analogous to the nested-loop fix applied in the referenced Tidal Finance commit).

### Proof of Concept

```rust
use chia_datalayer::{Hash, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

let mut blob = MerkleBlob::new(vec![]).unwrap();

// Pre-populate with 2 leaves so batch_insert skips the guarded path entirely
for i in 0i64..2 {
    blob.insert(KeyId(i), ValueId(i), &Hash(Bytes32::new([i as u8; 32])),
        chia_datalayer::InsertLocation::Auto {}).unwrap();
}

// Now batch_insert with a duplicate key — no guard fires
let dup_key = KeyId(99);
let batch = vec![
    ((dup_key, ValueId(1)), Hash(Bytes32::new([10u8; 32]))),
    ((dup_key, ValueId(2)), Hash(Bytes32::new([11u8; 32]))),  // same key
];
blob.batch_insert(batch).unwrap();  // succeeds — should have returned KeyAlreadyPresent

// Tree is now corrupted: two leaf nodes share KeyId(99)
// check_integrity() will return IntegrityKeyToIndexCacheLength
assert!(blob.check_integrity().is_err());
```

The blob now contains a root hash computed over the duplicate leaf, which diverges from any root a correctly-operating node would compute for the same key-value set.

    pub fn py_batch_insert(
        &mut self,

        hashes: Vec<Hash>,
