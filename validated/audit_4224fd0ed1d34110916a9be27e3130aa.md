### Title
Unvalidated User-Controlled `tokenAddress` in `logMetadata` + `initTransfer` Enables Unbacked Minting on NEAR ‚Äî (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

---

### Summary

`OmniBridge.logMetadata` and `OmniBridge.initTransfer` accept a fully attacker-controlled `tokenAddress` with no whitelist or registry check. In the `OmniBridgeWormhole` deployment, both functions publish Wormhole messages that the NEAR bridge treats as authoritative factory events. An attacker can register a malicious ERC-20 as a bridge token and then initiate a fake transfer, causing the NEAR bridge to mint real bridged tokens against zero deposited collateral.

---

### Finding Description

**`logMetadata` ‚Äî unvalidated token address published to Wormhole**

`OmniBridge.logMetadata` is a public, payable function that accepts any `tokenAddress` from the caller:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name = IERC20Metadata(tokenAddress).name();
    string memory symbol = IERC20Metadata(tokenAddress).symbol();
    uint8 decimals = IERC20Metadata(tokenAddress).decimals();
    logMetadataExtension(tokenAddress, name, symbol, decimals);
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
``` [1](#0-0) 

There is no check that `tokenAddress` is a legitimate, registered, or previously-known token. In `OmniBridgeWormhole`, the override of `logMetadataExtension` immediately publishes a Wormhole message containing the attacker-supplied address and attacker-supplied metadata:

```solidity
function logMetadataExtension(...) internal override {
    bytes memory payload = bytes.concat(
        bytes1(uint8(MessageType.LogMetadata)),
        bytes1(omniBridgeChainId),
        Borsh.encodeAddress(tokenAddress),
        ...
    );
    _wormhole.publishMessage{value: msg.value}(wormholeNonce, payload, _consistencyLevel);
``` [2](#0-1) 

The NEAR bridge's `deploy_token_callback` processes this Wormhole VAA. It only checks that the emitter address equals the registered factory ‚Äî which it does, because the emitter is `OmniBridgeWormhole` itself:

```rust
require!(
    self.factories.get(&chain) == Some(metadata.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
``` [3](#0-2) 

After this call, the NEAR bridge registers the malicious EVM address in `token_decimals` and `token_id_to_address`, treating it as a legitimate bridged token.

**`initTransfer` ‚Äî fake deposit via malicious ERC-20**

`initTransfer` also accepts a fully attacker-controlled `tokenAddress`. For tokens that are neither bridge tokens nor custom minters, it calls:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
``` [4](#0-3) 

A malicious ERC-20 can return `true` from `transferFrom` without moving any tokens. `safeTransferFrom` only checks the boolean return value, so the call succeeds. `OmniBridgeWormhole.initTransferExtension` then publishes a Wormhole message for the full `amount`:

```solidity
bytes memory payload = bytes.concat(
    bytes1(uint8(MessageType.InitTransfer)),
    ...
    Borsh.encodeAddress(tokenAddress),
    Borsh.encodeUint64(originNonce),
    Borsh.encodeUint128(amount),
    ...
);
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
``` [5](#0-4) 

The NEAR bridge's `fin_transfer_callback` processes this VAA. It again checks only that the emitter is the registered factory and that the token is in `token_decimals` (which it is, after the `logMetadata` step):

```rust
require!(
    self.factories.get(&init_transfer.emitter_address.get_chain())
        == Some(init_transfer.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
``` [6](#0-5) 

Both checks pass, and the NEAR bridge mints the full `amount` of bridged tokens to the attacker's NEAR recipient ‚Äî with zero real collateral deposited on the EVM side.

---

### Impact Explanation

**Critical ‚Äî Direct unauthorized mint of bridged assets.**

The attacker mints an arbitrary quantity of NEAR-side bridged tokens backed by no real EVM collateral. Every legitimate holder of that bridged token on NEAR is diluted; the bridge's collateralization invariant (locked EVM tokens = minted NEAR tokens) is permanently broken for the affected token. The attacker can immediately swap or redeem the unbacked tokens for real value.

---

### Likelihood Explanation

**Medium.** The attack is fully permissionless ‚Äî no privileged role, no leaked key, no colluding guardian is required. Wormhole guardians sign any valid message emitted by the registered bridge contract; they have no way to distinguish a legitimate `logMetadata` call from a malicious one. The only cost is two Wormhole message fees and the gas for four transactions. The attack is repeatable for any amount.

---

### Recommendation

1. **Whitelist `tokenAddress` in `logMetadata`**: Require that `tokenAddress` is already known to the bridge (e.g., `ethToNearToken[tokenAddress]` is non-empty, or a separate admin-maintained allowlist), or restrict `logMetadata` to an admin/operator role.

2. **Whitelist `tokenAddress` in `initTransfer`**: Before accepting a deposit, verify that `tokenAddress` is a registered bridge token or a known native token. Reject transfers for unregistered addresses.

3. **Alternatively**, enforce that `initTransfer` only accepts tokens that have a non-zero entry in `ethToNearToken` (i.e., tokens that have already been deployed/registered through the canonical `deployToken` path).

### Proof of Concept

```
1. Attacker deploys MaliciousToken at address M:
   - name() ‚Üí "Fake USDC"
   - symbol() ‚Üí "fUSDC"
   - decimals() ‚Üí 6
   - transferFrom(from, to, amount) ‚Üí true  // no actual transfer

2. Attacker calls OmniBridgeWormhole.logMetadata(M)
   with msg.value = wormhole.messageFee()
   ‚Üí Wormhole VAA published: LogMetadata(chainId, M, "Fake USDC", "fUSDC", 6)

3. Wormhole guardians sign the VAA (automatic ‚Äî valid message from registered emitter).

4. Attacker calls NearBridge.deploy_token(wormhole_proof)
   ‚Üí NEAR bridge registers M as a bridged token with decimals=6
   ‚Üí token_decimals[Eth(M)] = {near: 6, origin: 6}

5. Attacker calls OmniBridgeWormhole.initTransfer(
       tokenAddress = M,
       amount = 1_000_000_000_000,   // 1,000,000 fUSDC (6 decimals)
       fee = 0, nativeFee = 0,
       recipient = "attacker.near",
       message = ""
   ) with msg.value = wormhole.messageFee()
   ‚Üí M.transferFrom(attacker, bridge, 1e12) returns true (no tokens moved)
   ‚Üí Wormhole VAA published: InitTransfer(chainId, M, nonce, 1e12, ...)

6. Wormhole guardians sign the VAA (automatic).

7. Attacker calls NearBridge.fin_transfer(wormhole_proof)
   ‚Üí factory check passes (emitter = OmniBridgeWormhole = registered factory)
   ‚Üí token_decimals check passes (M registered in step 4)
   ‚Üí NEAR bridge mints 1,000,000 fUSDC-equivalent NEAR tokens to attacker.near

Result: Attacker holds 1,000,000 unbacked bridged tokens on NEAR.
        Zero real EVM tokens were ever deposited.
```
