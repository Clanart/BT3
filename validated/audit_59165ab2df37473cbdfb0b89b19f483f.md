### Title
Removed Custom Tokens Bypass Registry Check in `initTransfer`, Causing Accounting Corruption or Permanent Fund Freeze — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`removeCustomToken` deletes all EVM-side registry entries for a custom bridge token, but `initTransfer` contains no check that the supplied token is currently registered. After removal, the token silently falls into the native-lock path (`safeTransferFrom` to the bridge), emitting an `InitTransfer` event. Depending on whether the NEAR side still has the token registered, this either corrupts bridge collateralization or permanently freezes user funds.

### Finding Description
`removeCustomToken` (lines 120–127) clears every EVM-side mapping for a custom token:

```solidity
delete isBridgeToken[tokenAddress];
delete nearToEthToken[ethToNearToken[tokenAddress]];
delete ethToNearToken[tokenAddress];
delete customMinters[tokenAddress];
``` [1](#0-0) 

`initTransfer` (lines 373–437) performs no check that `tokenAddress` is currently registered. Its token-handling logic is a three-way branch:

```solidity
if (customMinters[tokenAddress] != address(0)) {
    // custom-minter burn path
} else if (isBridgeToken[tokenAddress]) {
    BridgeToken(tokenAddress).burn(msg.sender, amount);
} else {
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
}
``` [2](#0-1) 

After `removeCustomToken`, both `customMinters[tokenAddress]` and `isBridgeToken[tokenAddress]` are false/zero, so the call falls into the `else` branch: the token is **locked** in the bridge instead of burned, and an `InitTransfer` event is emitted with the raw `tokenAddress`. [3](#0-2) 

The NEAR bridge's `token_address_to_id` and `token_decimals` mappings are entirely independent of the EVM registry; they are not cleared when `removeCustomToken` is called on the EVM side. [4](#0-3) 

### Impact Explanation
Two distinct outcomes arise depending on NEAR-side state at the time of the transfer:

**Scenario A — NEAR still has the token registered (accounting corruption, High):**
The NEAR bridge's `fin_transfer_callback` finds the token in `token_decimals`, denormalizes the amount, and mints or releases tokens to the recipient. The EVM bridge now holds tokens that should have been burned (the mint/burn model is broken). Bridge collateralization is corrupted: EVM holds locked supply while NEAR has minted an equivalent unbacked supply.

**Scenario B — NEAR has also removed the token (permanent freeze, Critical):**
`fin_transfer_callback` panics at `BridgeError::TokenDecimalsNotFound`. The NEAR-side transfer is rejected. The EVM side has already locked the user's tokens with no on-chain refund or recovery path, permanently freezing user funds in the bridge contract.

Both outcomes match the allowed impact scope: accounting corruption that breaks bridge collateralization (High), and irrecoverable lock of user funds (Critical).

### Likelihood Explanation
The scenario is realistic and reachable by any unprivileged bridge user:
1. Admin calls `removeCustomToken` to delist a token (e.g., after discovering it is malicious or deprecated).
2. Any user — unaware of the removal, or deliberately — calls `initTransfer` with the delisted token address.
3. No privileged access is required for step 2; `initTransfer` is fully public.

The window between removal and user awareness can be significant, especially for automated or scripted bridge users.

### Recommendation
Add an explicit registry check at the top of `initTransfer` for non-ETH tokens:

```solidity
require(
    tokenAddress == address(0)
        || isBridgeToken[tokenAddress]
        || customMinters[tokenAddress] != address(0)
        || multiTokens[tokenAddress].tokenAddress != address(0),
    "ERR_TOKEN_NOT_REGISTERED"
);
```

This mirrors the fix recommended in the external report: validate the current whitelist status of the token before allowing the operation to proceed.

### Proof of Concept
1. Admin calls `addCustomToken("near.token", tokenAddress, customMinter, 18)` — token registered, `isBridgeToken[tokenAddress] = true`, `customMinters[tokenAddress] = customMinter`.
2. Admin calls `removeCustomToken(tokenAddress)` — all mappings cleared.
3. User calls `initTransfer(tokenAddress, 1000, 0, 0, "alice.near", "")`.
4. `customMinters[tokenAddress]` is zero, `isBridgeToken[tokenAddress]` is false → `else` branch executes: `IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), 1000)` succeeds; tokens are locked.
5. `InitTransfer` event emitted with `tokenAddress`.
6. NEAR prover picks up the event and calls `fin_transfer_callback`:
   - **If NEAR still has `token_decimals[tokenAddress]`**: transfer is processed, NEAR mints 1000 tokens — bridge accounting is corrupted (EVM holds locked tokens, NEAR has minted unbacked supply).
   - **If NEAR has removed `token_decimals[tokenAddress]`**: `fin_transfer_callback` panics with `TokenDecimalsNotFound`; user's 1000 tokens are permanently locked in the EVM bridge with no recovery path. [5](#0-4) [6](#0-5)
