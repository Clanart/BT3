### Title
`batch_insert` Bypasses Duplicate Key/Hash Validation, Corrupting DataLayer Merkle Tree Root - (File: crates/chia-datalayer/src/merkle/blob.rs)

### Summary
`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces, allowing any caller to silently write multiple leaf nodes with the same `KeyId` or `Hash` into the blob. The resulting tree has an inconsistent `block_status_cache`, a corrupted structural layout, and an incorrect root hash, which invalidates all subsequent proofs of inclusion and exclusion.

### Finding Description

`MerkleBlob::insert` enforces two invariants before writing a leaf:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a different code path. When the tree already has two or more leaves (`leaf_count > 1`), the function skips the call to `self.insert()` entirely and writes every remaining item directly via `insert_entry_to_blob`, with no duplicate check:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // no duplicate guard
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

Even when `leaf_count <= 1`, only the last two items popped from the vector go through `self.insert()`. All earlier items in the batch still bypass the guards. [3](#0-2) 

The Python binding `py_batch_insert` exposes this path directly to any caller:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 1503-1518
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [4](#0-3) 

After a batch with duplicate keys, the `block_status_cache` maps the key to only one index, but the blob contains two leaf nodes with the same key. The internal hash tree is built over all inserted nodes including the duplicates, so the computed root hash diverges from what a correctly-built tree would produce. `get_proof_of_inclusion` then generates a proof anchored to the wrong leaf, and `ProofOfInclusion::valid()` verifies it against a corrupted root. [5](#0-4) 

### Impact Explanation

Corrupting the Merkle tree root and leaf layout allows:
- **Forged proofs of inclusion**: a proof generated for key K after a duplicate-key batch will hash through the wrong sibling chain, producing a root that does not match any honest tree state. A verifier comparing against the stored root will accept or reject incorrectly.
- **Corrupted tree roots**: every subsequent `calculate_lazy_hashes` call propagates the corrupted internal-node hashes upward, permanently poisoning the root stored on disk.
- **Proof-of-exclusion bypass**: because the `block_status_cache` is inconsistent with the blob, exclusion proofs for keys that are actually present (in the duplicate leaf) may be generated and accepted.

This matches the allowed High impact: *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.*

### Likelihood Explanation

The Python binding is a public API surface. Any DataLayer node operator or client that calls `MerkleBlob.batch_insert` with a list containing a repeated `KeyId` or `Hash` — whether by mistake or by a malicious peer supplying crafted delta data — triggers the corruption silently (no error is returned). The `test_double_insert_fails` test only covers the single-item `insert` path, not `batch_insert`. [6](#0-5) 

### Recommendation

Add the same duplicate guards at the top of the bulk loop in `batch_insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insert_entry_to_blob logic
}
```

Alternatively, build a local `HashSet` of keys and hashes seen within the batch itself before writing any of them, to catch intra-batch duplicates as well as duplicates against the existing tree.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(bytearray())

# Pre-populate with 2 leaves so leaf_count > 1 and the fast path is taken
blob.insert(KeyId(1), ValueId(1), hashlib.sha256(b"a").digest())
blob.insert(KeyId(2), ValueId(2), hashlib.sha256(b"b").digest())

# batch_insert with a duplicate key (KeyId(1) already present)
# insert() would raise KeyAlreadyPresentError; batch_insert does not
blob.batch_insert(
    [(KeyId(1), ValueId(99)), (KeyId(3), ValueId(3))],
    [hashlib.sha256(b"a").digest(), hashlib.sha256(b"c").digest()],
)

blob.calculate_lazy_hashes()

# The tree now contains two leaves with KeyId(1).
# The root hash is corrupted; proofs generated here will not match
# any honest tree containing the same logical key-value mapping.
proof = blob.get_proof_of_inclusion(KeyId(1))
print("valid() on corrupted tree:", proof.valid())  # may return True against wrong root
```
