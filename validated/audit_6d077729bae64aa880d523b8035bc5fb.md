### Title
Cross-Chain Replay of `deployToken` Signature Due to Missing Chain ID Binding in `MetadataPayload` — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `MetadataPayload` signed by the NEAR MPC key for `deployToken` operations does not include the destination chain ID in the signed data. In contrast, `finTransfer` payloads on every chain explicitly include `omniBridgeChainId` twice. Because the same NEAR MPC-derived key (`nearBridgeDerivedAddress`) is used across EVM, StarkNet, and Solana, a valid `deployToken` signature obtained for one chain can be replayed verbatim on any other chain to deploy the same token without NEAR's authorization for that chain. This constitutes a signature verification bypass enabling unauthorized token deployment.

---

### Finding Description

**Root cause — missing chain ID in `MetadataPayload` serialization:**

On EVM, `OmniBridge.sol::deployToken` constructs the signed blob as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
```

No `omniBridgeChainId` is included. [1](#0-0) 

On StarkNet, `MetadataPayload::to_borsh()` serializes:

```cairo
borsh_bytes.append_byte(PayloadType::Metadata.into());
borsh_bytes.append(@borsh::encode_byte_array(self.token));
borsh_bytes.append(@borsh::encode_byte_array(self.name));
borsh_bytes.append(@borsh::encode_byte_array(self.symbol));
borsh_bytes.append_byte(*self.decimals);
```

No chain ID is included. [2](#0-1) 

On Solana, `DeployTokenPayload::serialize_for_near` serializes:

```rust
IncomingMessageType::Metadata.serialize(&mut writer)?;
self.serialize(&mut writer)?; // token, name, symbol, decimals only
```

No chain ID is included. [3](#0-2) 

**Contrast with `finTransfer` — which correctly binds to chain:**

On EVM, `finTransfer` includes `omniBridgeChainId` twice (for token address and recipient address):

```solidity
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.tokenAddress),
Borsh.encodeUint128(payload.amount),
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.recipient),
``` [4](#0-3) 

On StarkNet, `TransferMessagePayload::to_borsh(chain_id)` also includes `chain_id` twice:

```cairo
borsh_bytes.append_byte(chain_id);
borsh_bytes.append(@borsh::encode_address(*self.token_address));
...
borsh_bytes.append_byte(chain_id);
borsh_bytes.append(@borsh::encode_address(*self.recipient));
``` [5](#0-4) 

On Solana, `FinalizeTransferPayload::serialize_for_near` includes `SOLANA_OMNI_BRIDGE_CHAIN_ID` twice: [6](#0-5) 

**Same key used across all chains:**

The NEAR bridge uses a single MPC signer with a fixed `SIGN_PATH` for all signing operations. The `nearBridgeDerivedAddress` on EVM and the equivalent on StarkNet/Solana are all derived from the same NEAR MPC key. The `sign_transfer` function on NEAR uses a single `SIGN_PATH`: [7](#0-6) 

**Replay protection for `deployToken` is chain-local only:**

On EVM, the only guard against re-deployment is `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")` — this only prevents re-deployment on the same EVM instance, not on a different chain. [8](#0-7) 

On StarkNet, the guard is `assert(existing_token.is_zero(), 'ERR_TOKEN_ALREADY_DEPLOYED')` — again, chain-local only. [9](#0-8) 

---

### Impact Explanation

An attacker who observes a valid `deployToken` transaction on chain A (e.g., EVM) can extract the `(signatureData, metadata)` tuple and submit it to chain B (e.g., StarkNet or Solana) to deploy the same NEAR token there without NEAR's authorization for that chain.

**Direct impact:**
- Unauthorized token deployment on a chain the NEAR bridge operator never authorized. This matches the allowed High impact: *"Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized transfer finalization, **token deployment**, or message execution."*

**Secondary impact:**
- Once the attacker deploys the token on chain B, the legitimate deployment of that token on chain B is permanently blocked (`ERR_TOKEN_ALREADY_DEPLOYED`). The attacker-deployed token contract has no legitimate backing or mint authority from the bridge, so any future `fin_transfer` to that token on chain B will fail or mint to an uncontrolled contract. This permanently prevents the token from being bridged to chain B, matching the allowed Critical impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds."*

---

### Likelihood Explanation

- **No privilege required.** Any observer of on-chain transactions can extract a `deployToken` signature from chain A's transaction history.
- **Trivially replayable.** The attacker only needs to call `deploy_token(signature, payload)` on chain B with the same arguments.
- **Realistic scenario.** As the bridge expands to new chains, operators will deploy tokens on EVM first. Attackers can immediately replay those signatures on StarkNet or Solana before the legitimate deployment occurs, permanently blocking it.

---

### Recommendation

**Short term:** Include the destination chain ID in the `MetadataPayload` serialization on all chains, mirroring the pattern already used in `finTransfer`/`fin_transfer`:

- EVM: add `bytes1(omniBridgeChainId)` to the `borshEncoded` blob in `deployToken`.
- StarkNet: add `chain_id` as a parameter to `MetadataPayload::to_borsh()` and append it before the token address, as done in `TransferMessagePayload::to_borsh(chain_id)`.
- Solana: add `SOLANA_OMNI_BRIDGE_CHAIN_ID` to `DeployTokenPayload::serialize_for_near`, as done in `FinalizeTransferPayload::serialize_for_near`.

**Long term:** Adopt a uniform signing convention that always includes `(chain_id, contract_address, payload_type)` as a domain prefix for every signed message, regardless of payload type. Do not assume that a payload type prefix alone provides domain separation across chains.

---

### Proof of Concept

1. NEAR bridge operator signs a `MetadataPayload` for token `"usdc.near"` (name=`"USD Coin"`, symbol=`"USDC"`, decimals=`6`) to deploy it on EVM. The signature `sig` is broadcast in the EVM `deployToken` transaction.

2. Attacker observes the EVM transaction and extracts `(sig, metadata)`.

3. Attacker calls StarkNet's `deploy_token(sig, metadata)` before the legitimate StarkNet deployment occurs.

4. StarkNet's `_verify_borsh_signature` computes `keccak(Metadata | "usdc.near" | "USD Coin" | "USDC" | 6)` — identical to the EVM hash because no chain ID is included — and recovers the same `near_bridge_address`. Signature passes.

5. StarkNet deploys a new bridge token contract for `"usdc.near"` under attacker-controlled conditions and writes the mapping `near_to_starknet_token["usdc.near"] = attacker_deployed_address`.

6. All subsequent legitimate attempts to deploy `"usdc.near"` on StarkNet revert with `ERR_TOKEN_ALREADY_DEPLOYED`. The token is permanently undeployable on StarkNet through the legitimate bridge flow.
