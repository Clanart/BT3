Audit Report

## Title
Unrestricted `refundETH()` Allows Any Caller to Drain Residual ETH Left on the Router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` transfers the router's entire ETH balance to `msg.sender` with no access control. When a user calls any payable swap entry-point directly with `msg.value` exceeding `amountIn`, the `pay()` function wraps only the exact amount needed and leaves the surplus on the router. Any subsequent caller — including an attacker — can invoke `refundETH()` to claim that stranded ETH.

## Finding Description
`refundETH()` is implemented with no caller restriction: [1](#0-0) 

The internal `pay()` function, when `token == WETH` and the router holds native ETH, wraps exactly `value` and leaves `nativeBalance - value` on the contract: [2](#0-1) 

All four swap entry-points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) are `payable` and accept `msg.value`: [3](#0-2) 

The intended safe usage is `multicall{value}([swap(...), refundETH()])`, where `delegatecall` preserves `msg.sender` so the refund returns to the original caller: [4](#0-3) 

However, when a user calls a swap function directly (not via `multicall`) with `msg.value > amountIn`, the surplus ETH is stranded on the router after the call returns. The `receive()` guard does not prevent this — it only blocks plain ETH transfers; `msg.value` attached to a payable function call bypasses `receive()` entirely: [5](#0-4) 

An attacker then calls `refundETH()` standalone, which sends the full router ETH balance to the attacker's address with no check.

The existing test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` confirms the router is expected to hold residual ETH mid-multicall, proving the ETH-stranding state is reachable: [6](#0-5) 

## Impact Explanation
Direct theft of user ETH principal. A victim who calls `exactInputSingle{value: 2 ether}(amountIn: 1000)` loses `2 ether − 1000 wei` to any attacker who calls `refundETH()` in the next transaction. The loss is bounded only by the victim's excess `msg.value`, which can be arbitrarily large. This satisfies the High threshold (direct loss of user principal).

## Likelihood Explanation
The swap functions are documented and tested with native ETH input via `multicall`, but nothing prevents a user from calling them directly with excess ETH. An attacker can monitor the mempool for payable swap calls with `msg.value > amountIn` and back-run them with a standalone `refundETH()` call. The attack requires no special privileges, no setup, and is repeatable against any victim who sends excess ETH.

## Recommendation
Restrict `refundETH()` so it can only be called as part of a `multicall` (i.e., via `delegatecall` from the same contract), for example by checking `address(this) == msg.sender` is not sufficient — instead, record the original `msg.sender` of the outermost `multicall` in transient storage and require `refundETH()` to refund only that stored address. Alternatively, remove the standalone `refundETH()` entrypoint and expose refund logic only internally at the end of each swap function when `address(this).balance > 0`.

## Proof of Concept
1. Deploy `MetricOmmSimpleRouter` with a WETH/token pool.
2. Fund victim with 2 ETH.
3. Victim calls `exactInputSingle{value: 2 ether}(amountIn: 1000, tokenIn: WETH, ...)` directly (not via multicall). `pay()` wraps 1000 wei; ~2 ether − 1000 wei remains on the router.
4. Attacker calls `router.refundETH()` in the next transaction.
5. Assert attacker received `~2 ether − 1000 wei` and `address(router).balance == 0`.

The test fixture in `MetricOmmSimpleRouter.native.t.sol` already sets up the required pool and WETH infrastructure; the PoC above extends it by splitting the multicall into two separate transactions and asserting the attacker drains the residual.
