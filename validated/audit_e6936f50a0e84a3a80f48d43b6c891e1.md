### Title
Unauthorized Class Declaration via Unauthenticated `BOOTSTRAP` Sender Address Bypass — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

A hardcoded special-case branch inside `execute_declare_transaction` allows **any unprivileged transaction sender** to declare an arbitrary Sierra class hash into the global class registry **without signature verification, without a deployed account contract, and without paying any fee**, simply by setting `sender_address = 'BOOTSTRAP'`, `nonce = 0`, `version = 3`, and all resource bounds to zero.

---

### Finding Description

In `execute_declare_transaction`, after the transaction hash is computed and `tx_info` is filled, the following branch executes unconditionally before any account validation or nonce enforcement:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

The three conditions are all fully attacker-controllable fields in the transaction payload:

- `sender_address` is a felt loaded from the hint `%{ DeclareTxFields %}` and is part of the transaction hash input — no external constraint prevents it from being the felt encoding of the ASCII string `'BOOTSTRAP'`.
- `nonce = 0` is trivially satisfiable.
- `version = 3` is the only supported version for declare transactions in this codebase.
- `max_possible_fee == 0` is achieved by setting all three resource bounds (`L1_GAS`, `L2_GAS`, `L1_DATA_GAS`) `max_amount` and `max_price_per_unit` to zero. [2](#0-1) 

When the branch is taken, the function:
1. **Skips `check_and_increment_nonce`** — no account state is read or modified.
2. **Skips `run_validate` / `__validate_declare__`** — no signature is verified.
3. **Skips `charge_fee`** — no ERC-20 transfer is executed.
4. Directly writes `class_hash → compiled_class_hash` into `contract_class_changes` via `dict_update`. [3](#0-2) 

The normal path (lines 778–825) enforces all three of those steps. The BOOTSTRAP path silently bypasses all of them.

The only constraint that remains is `prev_value=0`, meaning a class hash can only be declared once this way. The class hash itself is validated to be a genuine Sierra class hash via `finalize_class_hash`: [4](#0-3) 

So the attacker can declare **any valid Sierra class** they possess the pre-image for, without owning any account contract at the `'BOOTSTRAP'` address.

---

### Impact Explanation

**Critical — Direct loss of funds.**

Once an attacker-controlled class is registered in the global class registry, it becomes available for contract deployment by any party (including the attacker themselves via a `DEPLOY_ACCOUNT` or `deploy_syscall`). A malicious class can implement arbitrary logic — for example, a token contract that silently mints to the attacker, or an account contract whose `__validate__` always returns `VALIDATED` regardless of the caller, enabling theft of funds from users who interact with it.

More directly: the fee bypass itself constitutes a state-transition that the protocol accepts as valid but which was never authorized by any account. The OS proof will attest to a class declaration that no legitimate account signed, meaning the verifier accepts a state root containing unauthorized class registrations. Any contract subsequently deployed from that class operates under a fraudulently obtained registration, and any funds sent to or locked in such contracts are at risk.

---

### Likelihood Explanation

**High.** The conditions are trivially satisfiable by any party who can submit a transaction to the sequencer:

- `sender_address = 'BOOTSTRAP'` is a known constant (the felt encoding of the 9-byte ASCII string).
- `nonce = 0` is the initial nonce.
- `version = 3` is the only accepted version.
- Zero resource bounds require no special access.

No privileged key, leaked secret, or operator collusion is required. The attacker only needs to construct a well-formed declare transaction with these fields and submit it to the mempool.

---

### Recommendation

Remove the `BOOTSTRAP` special-case branch entirely from `execute_declare_transaction`. If bootstrapping of system classes is a genuine operational requirement, it must be enforced through a separate, out-of-band mechanism (e.g., genesis block initialization, a privileged sequencer-only pre-block hook that is not reachable via the normal transaction dispatch path in `execute_transactions_inner`), not through a felt-value comparison on an attacker-supplied `sender_address` field inside the standard transaction execution loop. [5](#0-4) 

---

### Proof of Concept

1. Attacker prepares a valid Sierra contract class `C` and computes its `class_hash` and `compiled_class_hash`.
2. Attacker constructs a declare transaction with:
   - `sender_address = 0x424f4f545354524150` (felt encoding of `'BOOTSTRAP'`)
   - `nonce = 0`
   - `version = 3`
   - `tip = 0`, all three `ResourceBounds` entries with `max_amount = 0` and `max_price_per_unit = 0`
   - `class_hash` = hash of class `C`
   - `compiled_class_hash` = compiled hash of class `C`
3. The transaction hash is computed correctly by `compute_declare_transaction_hash` — no forgery needed.
4. The OS processes the transaction via `execute_declare_transaction`.
5. The branch at line 764 evaluates to true; `compute_max_possible_fee` returns `0` because all bounds are zero.
6. `dict_update` writes `class_hash → compiled_class_hash` into `contract_class_changes`.
7. `state_update` commits this into the contract class Patricia tree.
8. The OS proof attests to a new global state root containing the attacker's class — with no signature ever verified and no fee ever paid.
9. Attacker deploys a contract from class `C` and uses it to drain funds from victims. [1](#0-0)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
