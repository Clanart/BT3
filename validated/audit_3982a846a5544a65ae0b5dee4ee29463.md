### Title
Integer Division Truncation in `feeCap` Calculation Enables Zero-Fee Cosmos Tx Bypass When BaseFee Is Zero - (File: `ante/evm/fee_checker.go`)

### Summary

`NewDynamicFeeChecker` in `ante/evm/fee_checker.go` computes `feeCap` via integer division `fee / gas`. When a user submits a Cosmos tx with `fee < gas`, the division truncates to zero. If the EIP-1559 base fee has naturally reached zero (possible when `minGasPrice = 0` and blocks are underutilized), the zero `feeCap` passes the base-fee sufficiency check, the effective price collapses to zero, and the `effectiveFee` returned to the SDK's `DeductFeeDecorator` is zero â€” meaning the user pays nothing.

### Finding Description

In `ante/evm/fee_checker.go`, `NewDynamicFeeChecker` computes the per-gas fee cap using integer (truncating) division:

```go
// line 83
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))
```

`sdkmath.Int.Quo` is integer division with truncation toward zero. When `fee < gas` (e.g., `fee = 1 aevmos`, `gas = 1_000_000`), `feeCap = 0`.

The sufficiency check that follows is:

```go
// line 86
if feeCap.LT(baseFeeInt) {
    return nil, 0, errorsmod.Wrapf(errortypes.ErrInsufficientFee, ...)
}
```

When `baseFee = 0`, the condition `0 < 0` is `false`, so the check is silently bypassed.

The effective price is then:

```go
// line 91
effectivePrice := sdkmath.NewIntFromBigInt(
    types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()))
```

`EffectiveGasPrice` computes `min(baseFee + tipCap, feeCap)`. With `feeCap = 0`, the result is `0` regardless of `maxPriorityPrice`.

The effective fee returned to the SDK is:

```go
// lines 94-99
effectiveFee := sdk.Coins{{
    Denom:  denom,
    Amount: effectivePrice.Mul(sdkmath.NewIntFromUint64(gas)), // 0 * gas = 0
}}
```

The `DeductFeeDecorator` deducts exactly what `TxFeeChecker` returns. With `effectiveFee = 0`, the user's account is not charged.

`baseFee = 0` is reachable without governance action: `CalculateBaseFee` in `x/feemarket/keeper/eip1559.go` applies:

```go
// line 104
return ethermint.BigMax(x.Sub(parentBaseFee, baseFeeDelta), minGasPrice)
```

When `minGasPrice = 0` (the default) and blocks are consistently underutilized, the base fee decreases each block until it reaches zero.

### Impact Explanation

An unprivileged user can submit Cosmos SDK transactions (wrapping any `sdk.Msg`) with a nominal fee of 1 aevmos and a large gas limit, paying zero effective fees. This is a fee-market ante handler bug that permits transactions to commit without paying the required fees â€” mis-accounting user funds and enabling fee-free spam or execution of arbitrary Cosmos messages at no cost.

**Impact class:** High â€” ante handler bug that permits invalid transactions to commit and valid user funds/fees to be mis-accounted.

### Likelihood Explanation

- `minGasPrice` defaults to `0` in the feemarket module.
- On any chain with low transaction volume, `CalculateBaseFee` will drive the base fee to zero over time.
- No privileged role, governance action, or key compromise is required; the attacker only needs to craft a Cosmos tx with `fee < gas`.
- The condition is stable once reached: a zero base fee stays zero as long as blocks remain underutilized.

### Recommendation

Replace the truncating integer division with a ceiling division or a precision-scaled comparison, so that any non-zero fee amount is not silently rounded to zero:

```go
// Instead of:
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))

// Use ceiling division to avoid truncation to zero:
gasInt := sdkmath.NewIntFromUint64(gas)
feeCap := fee.Add(gasInt).Sub(sdkmath.OneInt()).Quo(gasInt) // ceil(fee/gas)
```

Alternatively, enforce a minimum `feeCap > 0` when `fee > 0`, or validate that `fee >= baseFee * gas` directly without the intermediate per-gas division.
