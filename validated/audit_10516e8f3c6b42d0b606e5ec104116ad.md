### Title
Permanent Fund Lock When Native ETH Recipient Is a Contract Without `receive()`/`fallback()` — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

In `OmniBridge.finTransfer`, when the bridged token is native ETH (`payload.tokenAddress == address(0)`), the contract attempts a raw ETH call to `payload.recipient`. If the recipient is a contract that lacks a `receive()` or `fallback()` function, the call fails and the entire transaction reverts. Because the NEAR side has already irreversibly burned or locked the user's tokens before the EVM finalization is attempted, and no cancel/recovery mechanism exists anywhere in the protocol, the user's funds are permanently lost.

---

### Finding Description

`OmniBridge.finTransfer` handles native ETH delivery at lines 317–322:

```solidity
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [1](#0-0) 

The `completedTransfers[payload.destinationNonce] = true` flag is written at line 287, before the ETH transfer. When the ETH transfer fails and `revert FailedToSendEther()` is triggered, the entire transaction is rolled back — including the nonce flag — so the nonce is never permanently consumed. [2](#0-1) 

This means `finTransfer` will revert on every retry for the same signed payload, because the recipient address is embedded in the MPC-signed message and cannot be changed without a new MPC signature.

On the NEAR side, `init_transfer_internal` burns or locks the user's tokens **before** any EVM finalization occurs:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(...);
``` [3](#0-2) 

There is no `cancelTransfer`, `recoverFunds`, or equivalent function anywhere in the protocol — confirmed absent by search. Once tokens are burned on NEAR, they cannot be recovered regardless of what happens on EVM.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

A user who specifies a contract address (e.g., a multisig wallet, DAO treasury, smart contract vault, or any contract without `receive()`) as the EVM recipient for a native ETH bridge transfer will have their NEAR-side tokens permanently destroyed with zero delivery on EVM. The signed payload is immutable (MPC-signed), so no relayer can reroute the transfer. The nonce is never consumed, but the funds are gone from NEAR and can never arrive on EVM.

---

### Likelihood Explanation

**Moderate.** Contract addresses are a common and legitimate choice for bridge recipients: multisig wallets (Gnosis Safe), DAO treasuries, protocol vaults, and smart contract wallets are all widely used. Many such contracts do not implement `receive()` or `fallback()`. Users are given no warning in the protocol that specifying such an address for a native ETH transfer will result in permanent fund loss. The scenario requires only a standard, unprivileged bridge user action.

---

### Recommendation

Replace the hard-revert pattern with a WETH fallback, mirroring the pattern from the referenced report:

```solidity
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) {
        IWETH(WETH_ADDRESS).deposit{value: payload.amount}();
        IERC20(WETH_ADDRESS).safeTransfer(payload.recipient, payload.amount);
    }
}
```

Alternatively, enforce that `payload.recipient` is an EOA (check `payload.recipient.code.length == 0`) before accepting the transfer. A third option is to add a protocol-level cancel/refund path on NEAR that can be triggered when EVM finalization is provably impossible.

---

### Proof of Concept

1. User holds bridged tokens on NEAR and calls `ft_transfer_call` targeting the NEAR OmniBridge, specifying an EVM contract address (e.g., a Gnosis Safe without ETH receive support) as the recipient, with `tokenAddress = address(0)` (native ETH).
2. NEAR bridge executes `init_transfer_internal`: tokens are burned via `burn_tokens_if_needed` and the transfer is stored in `pending_transfers`. [4](#0-3) 
3. MPC signs the `TransferMessagePayload` with the contract address embedded as `payload.recipient`.
4. Relayer calls `OmniBridge.finTransfer` on EVM with the signed payload.
5. `completedTransfers[payload.destinationNonce] = true` is set. [5](#0-4) 
6. `payload.recipient.call{value: payload.amount}("")` fails (no `receive()`/`fallback()`). [6](#0-5) 
7. `revert FailedToSendEther()` rolls back the entire transaction including the nonce flag.
8. Every subsequent retry reverts identically. The NEAR tokens are already burned. No cancel/recovery function exists. Funds are permanently lost.
