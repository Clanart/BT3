### Title
Wrong Config Account Used for Signature Verification in `finalize_transfer_sol` — (File: solana/programs/bridge_token_factory/src/lib.rs)

### Summary
In `finalize_transfer_sol`, the pause guard correctly reads `ctx.accounts.common.config.paused`, but the subsequent `verify_signature` call reads `ctx.accounts.config.derived_near_bridge_address` — a **different** config account field — instead of `ctx.accounts.common.config.derived_near_bridge_address`. This is a direct analog of the "wrong variable used for comparison" class: one config is checked for liveness, a different config is used for the security-critical signature check.

### Finding Description
In `finalize_transfer_sol` the two security checks reference different config accounts:

```
// pause check — uses common.config
ctx.accounts.common.config.paused & FINALIZE_TRANSFER_PAUSED == 0

// signature verification — uses a DIFFERENT top-level config
&ctx.accounts.config.derived_near_bridge_address
```

The parallel function `finalize_transfer` is consistent: both its pause check and its `verify_signature` call use `ctx.accounts.common.config`. [1](#0-0) 

Because the pause check compiles against `ctx.accounts.common.config`, the `FinalizeTransferSol` account struct must expose **both** a `common` sub-struct (containing a `config`) **and** a separate top-level `config` field. The `verify_signature` call therefore reads `derived_near_bridge_address` from the wrong account.

Compare with the correct pattern in `finalize_transfer`: [2](#0-1) 

### Impact Explanation
`derived_near_bridge_address` is the 64-byte uncompressed public key against which every MPC/NEAR-signed payload is verified on Solana. If `ctx.accounts.config` in `FinalizeTransferSol` is not seed-constrained to the canonical program config PDA, an attacker can supply an arbitrary account whose `derived_near_bridge_address` field contains their own key, sign a crafted `FinalizeTransferPayload` with that key, and pass `verify_signature` for SOL-destination transfers. This enables unauthorized finalization of SOL releases from the vault — direct theft of native SOL locked in the bridge.

This maps to the allowed impact: **Proof/signature verification bypass enabling unauthorized transfer finalization**.

### Likelihood Explanation
`finalize_transfer_sol` is a public, permissionless entry point callable by any Solana user. Exploitability depends on whether the `config` account in `FinalizeTransferSol` carries Anchor seed constraints (`seeds = [b"config"], bump`). If it does, the account is implicitly validated and the bug is a latent correctness issue. If it lacks seed constraints — which is plausible given the inconsistency — any caller can substitute a malicious config and bypass signature verification for every SOL-destination bridge transfer.

### Recommendation
Replace the inconsistent reference in `finalize_transfer_sol`:

```rust
// Before (wrong)
&ctx.accounts.config.derived_near_bridge_address,

// After (correct, consistent with finalize_transfer and the pause check above)
&ctx.accounts.common.config.derived_near_bridge_address,
```

Additionally, audit the `FinalizeTransferSol` account struct to confirm that the top-level `config` field (if it exists separately from `common.config`) carries the correct PDA seed constraints, or remove it entirely if it is redundant.

### Proof of Concept
1. Deploy a fake `Config` account whose `derived_near_bridge_address` is an attacker-controlled Ethereum key `K_attacker`.
2. Construct a `FinalizeTransferPayload` that releases SOL to an attacker-controlled recipient.
3. Sign the Borsh-encoded payload with `K_attacker`.
4. Call `finalize_transfer_sol` passing the fake config as `ctx.accounts.config`.
5. The pause check passes (it reads `common.config.paused`, which is the real config).
6. `verify_signature` passes (it reads `ctx.accounts.config.derived_near_bridge_address` = `K_attacker`, which matches the attacker's signature).
7. SOL is released to the attacker without a valid MPC/NEAR signature.
