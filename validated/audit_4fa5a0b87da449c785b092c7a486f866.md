### Title
`execute_deploy` Syscall Hardcodes `failure_flag=0`, Masking Constructor Reverts and Enabling Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall unconditionally writes `failure_flag=0` in its response header, regardless of whether the deployed contract's constructor actually reverted. Because `deploy_contract` is called with the outer `revert_log` as an implicit argument (meaning constructor reverts are silently undone at the state level), the calling contract is told the deployment succeeded while the contract was never actually deployed. A calling contract that subsequently transfers funds to the "deployed" address will permanently freeze those funds at an address with no class hash.

---

### Finding Description

In `execute_deploy` the response header is written with a hardcoded success flag:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The function signature carries `revert_log: RevertLogEntry*` as an implicit argument:

```cairo
func execute_deploy{
    ...
    revert_log: RevertLogEntry*,
    ...
}(block_context: BlockContext*, caller_execution_context: ExecutionContext*)
``` [2](#0-1) 

`deploy_contract` is then called without an explicit `revert_log`, meaning the outer `revert_log` is passed implicitly:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}
``` [3](#0-2) 

Compare with `execute_deploy_account_transaction`, where `deploy_contract` is explicitly called with a fresh `revert_log`, confirming the function supports constructor reverts:

```cairo
let revert_log = init_revert_log();
deploy_contract{revert_log=revert_log}(
    block_context=block_context, constructor_execution_context=constructor_execution_context
);
``` [3](#0-2) [4](#0-3) 

The `handle_revert` function in `revert.cairo` processes `CHANGE_CLASS_ENTRY` entries, meaning a reverted constructor also reverts the class hash assignment — the contract is never deployed:

```cairo
if (selector == CHANGE_CLASS_ENTRY) {
    // Change class entry.
    let class_hash = revert_log_end[0].value;
    return revert_contract_changes();
}
``` [5](#0-4) 

The net result: the constructor reverts → state is rolled back → contract address has `class_hash = 0` → but the syscall response unconditionally reports `failure_flag=0`.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A calling contract that reads `failure_flag=0` and proceeds to transfer tokens (via ERC-20 `transfer`) to the returned `contract_address` will deposit funds at an address that has no class hash. StarkNet ERC-20 transfers do not validate that the recipient is a deployed contract. Those funds become permanently unrecoverable because no entry point exists at that address to move them.

---

### Likelihood Explanation

Any unprivileged user can:
1. Deploy a contract whose constructor conditionally reverts (e.g., based on a storage value or calldata).
2. Call that contract from another contract that also transfers funds to the freshly "deployed" address.
3. Trigger the constructor revert path.

No privileged role, leaked key, or external dependency is required. The attack is fully on-chain and deterministic.

---

### Recommendation

Propagate the actual constructor outcome to `failure_flag`. Specifically:

- Have `deploy_contract` return an `is_reverted` flag (analogous to `select_execute_entry_point_func`).
- Use that flag to set `failure_flag` in the `ResponseHeader`:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

If `is_reverted = 1`, also ensure the contract address is not returned as a valid deployment in the `DeployResponse`.

---

### Proof of Concept

1. Attacker deploys **Contract A** containing logic:
   - Call `deploy` syscall targeting **Class B** (whose constructor always reverts).
   - Read the response — `failure_flag` is `0` (success).
   - Call ERC-20 `transfer(contract_b_address, amount)`.

2. At the OS level:
   - `execute_deploy` calls `deploy_contract` with the outer `revert_log`.
   - The constructor of Class B reverts; `handle_revert` rolls back the `CHANGE_CLASS_ENTRY`, leaving `contract_b_address` with `class_hash = 0`.
   - `execute_deploy` writes `ResponseHeader(gas=remaining_gas, failure_flag=0)`.

3. Contract A reads `failure_flag=0`, proceeds with the transfer.

4. `amount` tokens now reside at `contract_b_address`, which has no class hash. No entry point can be called to recover them. **Funds are permanently frozen.** [6](#0-5) [7](#0-6)

func handle_revert{contract_state_changes: DictAccess*}(
    contract_address, revert_log_end: RevertLogEntry*
) {
    alloc_locals;

    local state_entry: StateEntry*;

    %{ PrepareStateEntryForRevert %}

    let class_hash = state_entry.class_hash;
    let storage_ptr = state_entry.storage_ptr;
    with class_hash, storage_ptr, revert_log_end {
        revert_contract_changes();

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
