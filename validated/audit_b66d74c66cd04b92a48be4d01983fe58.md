### Title
Fee Bypass via Unchecked Felt Arithmetic Wrap-Around in `compute_max_possible_fee` — (File: `execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` in `transaction_impls.cairo` performs felt arithmetic (mod P ≈ 2^251) on resource-bound fields without range-checking the final result. An unprivileged V3 transaction sender can craft `max_price_per_unit` values that cause the sum to wrap to exactly 0 modulo the field prime. This makes `charge_fee` skip the ERC-20 fee transfer entirely, granting free execution of an arbitrary transaction.

---

### Finding Description

`compute_max_possible_fee` computes:

```
max_fee = l1.max_amount * l1.max_price
        + l2.max_amount * (l2.max_price + tip)
        + l1d.max_amount * l1d.max_price
```

all in felt arithmetic (mod P). [1](#0-0) 

The only bounds enforced on the inputs (during hash computation in `pack_resource_bounds`) are:
- `max_amount ≤ 2^64 − 1` (via `assert_nn_le`)
- `max_price_per_unit ≥ 0` (via `assert_nn`, so ≤ (P−1)/2 ≈ 2^250)
- `tip ≤ 2^64 − 1` [2](#0-1) 

The product `max_amount × max_price_per_unit` can reach up to `(2^64)(2^250) = 2^314`, which wraps around P roughly 2^63 times. No range check is applied to the returned felt from `compute_max_possible_fee`.

`charge_fee` then does:

```cairo
if (max_fee == 0) {
    return ();
}
``` [3](#0-2) 

If `max_fee` wraps to 0, the entire fee-collection path — including the ERC-20 `transfer` call — is skipped. [4](#0-3) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

A user can execute an arbitrary transaction (invoke, deploy-account, declare) with zero fee paid. The sequencer provides execution resources and receives nothing. Because the OS proof is valid (all individual field checks pass), the L1 verifier accepts the block. The fee-accounting invariant — that every executed transaction pays at least some fee — is broken at the protocol level, not just at the sequencer level.

---

### Likelihood Explanation

**Medium.** Any V3 transaction sender can craft the required values. The arithmetic is deterministic and the solution is closed-form (shown in PoC). The only gate is sequencer inclusion; however, a sequencer that does not independently recompute `compute_max_possible_fee` in integer arithmetic (outside Cairo) will not detect the attack. The OS itself accepts the transaction as fully valid.

---

### Recommendation

1. **Bound `max_price_per_unit` tightly.** In `pack_resource_bounds`, replace `assert_nn` with `assert_nn_le(resource_bounds.max_price_per_unit, 2**128 - 1)`. This keeps the product `max_amount × max_price_per_unit` below `2^192 ≪ P`, eliminating wrap-around. [2](#0-1) 

2. **Range-check the result.** After computing `max_fee`, add `assert_nn(max_fee)` (or `assert_nn_le`) to guarantee the result is a non-negative, non-wrapped felt before using it as an upper bound. [5](#0-4) 

---

### Proof of Concept

Let P = the Stark field prime = 2^251 + 17·2^192 + 1. Since P ≡ 1 (mod 4), the value (P+3)/4 is an integer.

Craft a V3 transaction with:

| Resource | `max_amount` | `max_price_per_unit` |
|---|---|---|
| L1_GAS | 2 | (P+3)/4 |
| L2_GAS | 1 | (P−3)/2 |
| L1_DATA_GAS | 0 | 0 |
| tip | 0 | — |

**Verification of bounds:**
- `max_amount = 2 ≤ 2^64 − 1` ✓
- `(P+3)/4 ≤ (P−1)/2` ✓ (passes `assert_nn`)
- `(P−3)/2 ≤ (P−1)/2` ✓ (passes `assert_nn`)

**Arithmetic:**
```
max_fee = 2 · (P+3)/4  +  1 · (P−3)/2  +  0
        = (P+3)/2      +  (P−3)/2
        = (P+3 + P−3) / 2
        = 2P / 2
        = P
        ≡ 0  (mod P)
```

`compute_max_possible_fee` returns 0. `charge_fee` hits the `if (max_fee == 0) { return (); }` branch and exits without executing the ERC-20 transfer. The transaction executes with zero fee paid, and the resulting OS proof is valid. [6](#0-5)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}

// Charges a fee from the user.
// If max_fee is not 0, validates that the selector matches the entry point of an account contract
// and executes an ERC20 transfer on the behalf of that account contract.
//
// Arguments:
// block_context - a global context that is fixed throughout the block.
// tx_execution_context - The execution context of the transaction that pays the fee.
func charge_fee{
    range_check_ptr,
    builtin_ptrs: BuiltinPointers*,
    contract_state_changes: DictAccess*,
    contract_class_changes: DictAccess*,
    outputs: OsCarriedOutputs*,
}(block_context: BlockContext*, tx_execution_context: ExecutionContext*) {
    alloc_locals;

    local tx_info: TxInfo* = tx_execution_context.execution_info.tx_info;
    let max_fee = compute_max_possible_fee(tx_info=tx_info);

    if (max_fee == 0) {
        return ();
    }

    local low_actual_fee;
    %{ LoadActualFee %}
    local calldata: TransferCallData = TransferCallData(
        recipient=block_context.block_info_for_execute.sequencer_address,
        amount=Uint256(low=low_actual_fee, high=0),
    );

    // Verify that the charged amount is not larger than the transaction's max_fee field.
    assert_nn_le(calldata.amount.low, max_fee);
