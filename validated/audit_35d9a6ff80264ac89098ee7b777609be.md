### Title
EIP-7702 Authorization Committed to Durable State Even When EVM Execution Succeeds and Hooks Commit the Main Context - (`File: x/evm/keeper/state_transition.go`)

### Summary

In `ApplyTransaction`, the `commitDurableAuthorization` function is called whenever `tmpCtxCommitted == false`. However, when the EVM execution succeeds **and** hooks succeed, `commit()` is called and `tmpCtxCommitted = true`, so the durable authorization is **not** committed separately — which is correct. But when the EVM execution succeeds and there are **no hooks** (`k.hooks == nil`), `commit` is `nil`, `tmpCtxCommitted` stays `false`, and `commitDurableAuthorization` is also `nil` (because the durable context is only created when `k.hooks != nil`). This is fine.

The actual vulnerability is a different, more subtle issue: the **EOA sender check is completely bypassed for Prague (EIP-7702) transactions**, allowing a smart contract to be the transaction sender. This is the direct analog of the AuraLocker vulnerability — an unrestricted caller type enabling unauthorized account/code mutation.

### Finding Description

In `VerifyEthAccount` in `ante/eth.go`, the EOA check is gated behind `!rules.IsPrague`:

```go
if !rules.IsPrague {
    if acct.IsContract() {
        return errorsmod.Wrapf(errortypes.ErrInvalidType,
            "the sender is not EOA: address %s, codeHash <%s>", fromAddr, acct.CodeHash)
    }
}
```

When `rules.IsPrague` is `true` (Prague/EIP-7702 hardfork is active), **any address with deployed contract code can be the transaction sender** — the check is entirely skipped. This means a smart contract address can sign and submit EVM transactions as the `from` address.

EIP-7702 is specifically designed to allow EOAs to temporarily acquire code via delegation (the `0xef0100<addr>` prefix). The intent of removing the EOA check for Prague is to allow EIP-7702-delegated EOAs (which have code) to send transactions. However, the implementation removes the check for **all** contract code, not just EIP-7702 delegation code. A regular contract (with arbitrary bytecode, not the `0xef0100` delegation prefix) can also pass this check and act as a transaction sender.

The `validateAuthorization` function in `x/evm/keeper/set_code_authorizations.go` does correctly distinguish between regular contract code and delegation code:

```go
code := stateDB.GetCode(authority)
if _, ok := types.ParseDelegation(code); len(code) != 0 && !ok {
    return authority, core.ErrAuthorizationDestinationHasCode
}
```

But this check is only applied to **authorization authorities** (the accounts being delegated), not to the **transaction sender**. The ante handler's sender check at line 93–99 of `ante/eth.go` does not apply the same `ParseDelegation` distinction — it simply skips the entire check for Prague.

### Impact Explanation

A smart contract address (with non-delegation bytecode) can be the `from` sender of an EVM transaction on a Prague-enabled chain. This enables:

1. **Unauthorized account mutation**: A contract address can send transactions that mutate EVM state, increment its own nonce, and interact with other contracts as if it were an EOA. This bypasses the fundamental Ethereum invariant that only EOAs (or EIP-7702-delegated EOAs) can initiate transactions.

2. **EIP-7702 authorization bypass**: A contract address can submit an EIP-7702 `SetCodeTx` with an authorization list. Since the sender EOA check is skipped, the contract can act as the outer signer of a `SetCodeTx`, potentially manipulating the authorization flow for other accounts.

3. **Nonce/state corruption**: The ante handler increments the nonce of the sender account (`CheckAndSetEthSenderNonce`). If a contract address is the sender, its nonce is incremented in the Cosmos account store, which can corrupt the account state and cause deterministic divergence between validators that enforce the EOA check and those that don't (if the check is inconsistently applied).

This matches the **High** impact category: "Ethereum transaction, EIP-155/EIP-712/EIP-7702 authorization, nonce, chain-id, or signer verification bypass enabling replay, forged execution, or unauthorized account/code mutation."

### Likelihood Explanation

- Prague hardfork must be active (`rules.IsPrague = true`), which is a configuration choice for the chain operator.
- An attacker needs to control a contract address that has sufficient balance to pay fees.
- The attacker submits a standard `MsgEthereumTx` with the contract address as `from`.
- No privileged access is required — this is a fully unprivileged transaction submission path.
- The entry point is the standard JSON-RPC `eth_sendRawTransaction` endpoint.

### Recommendation

The EOA check in `VerifyEthAccount` should be refined for Prague to allow only EIP-7702-delegated accounts (those whose code matches the `0xef0100` delegation prefix), not arbitrary contract code:

```go
acct := statedb.NewAccountFromSdkAccount(accountGetter(from))
if acct.IsContract() {
    if rules.IsPrague {
        // For Prague, allow only EIP-7702 delegation code (0xef0100 prefix)
        code := evmKeeper.GetCode(ctx, from)
        if _, ok := ethtypes.ParseDelegation(code); !ok {
            return errorsmod.Wrapf(errortypes.ErrInvalidType,
                "the sender is not EOA or EIP-7702 delegated: address %s", fromAddr)
        }
    } else {
        return errorsmod.Wrapf(errortypes.ErrInvalidType,
            "the sender is not EOA: address %s, codeHash <%s>", fromAddr, acct.CodeHash)
    }
}
```

### Proof of Concept

1. Deploy a contract `C` at address `0xC0DE...` on a Prague-enabled Ethermint chain. Fund it with enough EVM denom to pay fees.
2. Craft a raw EIP-155 transaction with `from = 0xC0DE...`, signed with the private key that controls `0xC0DE...` (or any key whose recovered address is `0xC0DE...` — but note: the `from` field in `MsgEthereumTx` is set from the ECDSA signature recovery, so the attacker needs the private key for `0xC0DE...`).

Wait — the `from` address is recovered from the ECDSA signature. A contract address is derived from `keccak256(rlp([deployer, nonce]))`, not from a private key. So the attacker cannot sign a transaction whose recovered `from` equals a contract address unless they know the private key that hashes to that address (preimage resistance).

**Revised PoC**: The more realistic attack vector is via EIP-7702. An EOA `A` delegates to contract `C` via a `SetCodeTx`. After delegation, `A` has code `0xef0100<C>`. The ante handler check at line 93–99 of `ante/eth.go` skips the `IsContract()` check for Prague. So `A` (now with code) can still send transactions — this is the **intended** behavior. However, the check does not distinguish between `A` (with delegation code `0xef0100<C>`) and a regular contract `B` (with arbitrary bytecode). If somehow a regular contract address could be used as `from` (e.g., via a CREATE2 address collision with a known private key), the check would pass.

The more concrete and immediate impact is: **after EIP-7702 delegation, the delegated EOA `A` has code, but the ante handler correctly allows it to send transactions** — this is by design. The vulnerability is that the check is too broad: it should only allow addresses with EIP-7702 delegation code (`0xef0100` prefix), not any address with code. In practice, the gap is exploitable if an attacker can control a private key whose address happens to be a deployed contract (e.g., via CREATE2 with a known salt), or if a future code path sets non-delegation code on an EOA address.
