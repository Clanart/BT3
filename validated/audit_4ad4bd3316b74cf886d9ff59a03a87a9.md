### Title
`RefundGas` Failure in `ApplyTransaction` Blocks EVM Transaction Inclusion and Mis-accounts User Fees - (File: `x/evm/keeper/gas.go`, `x/evm/keeper/state_transition.go`)

### Summary
`ApplyTransaction` unconditionally propagates errors from `RefundGas` back to the Cosmos message server. If the fee collector module account has insufficient balance to cover the gas refund â€” for any reason â€” the entire EVM transaction fails at the Cosmos level after EVM execution has already completed. The ante handler's fee deduction is not reverted, so the sender loses their full gas fee while their transaction is not executed. This is the direct analog of the RocketPool bond-return blocking the kick action.

### Finding Description

In `x/evm/keeper/gas.go`, `RefundGasWithPrice` calls `SendCoinsFromModuleToAccountVirtual` to return leftover gas to the sender from the fee collector module account. If that call fails, it returns a hard error:

```go
err := k.bankKeeper.SendCoinsFromModuleToAccountVirtual(ctx, authtypes.FeeCollectorName, msg.From.Bytes(), refundedCoins)
if err != nil {
    err = errorsmod.Wrapf(errortypes.ErrInsufficientFunds, "fee collector account failed to refund fees: %s", err.Error())
    return errorsmod.Wrapf(err, "failed to refund %d leftover gas (%s)", leftoverGas, refundedCoins.String())
}
``` [1](#0-0) 

`ApplyTransaction` calls `RefundGas` **after** EVM execution has already completed and state has been committed to `tmpCtx`. If `RefundGas` returns an error, `ApplyTransaction` returns `nil, error` unconditionally:

```go
if err = k.RefundGas(ctx, msg, leftoverGas, cfg.Params.EvmDenom); err != nil {
    return nil, errorsmod.Wrapf(err, "failed to refund leftover gas to sender %s", msg.From)
}
``` [2](#0-1) 

`EthereumTx` (the message server) propagates this error directly:

```go
evmResult, err := k.ApplyTransaction(ctx, msg)
if err != nil {
    return nil, errorsmod.Wrap(err, "failed to apply transaction")
}
``` [3](#0-2) 

When the message server returns an error, the Cosmos SDK reverts all state changes from message execution. However, the ante handler's fee deduction â€” which ran in a separate committed context â€” is **not** reverted. The ante handler deducts `gasLimit * gasPrice` via `SendCoinsFromAccountToModuleVirtual`:

```go
if err := bankKeeper.SendCoinsFromAccountToModuleVirtual(ctx, acc.GetAddress(), authtypes.FeeCollectorName, fees); err != nil {
    return errorsmod.Wrap(errortypes.ErrInsufficientFunds, err.Error())
}
``` [4](#0-3) 

The `BankKeeper` interface exposes both virtual operations, confirming they are distinct from standard bank transfers and may check actual module account balances: [5](#0-4) 

The fee collector's actual bank balance can be zero at the start of a new block (after end-of-block distribution to validators). If `SendCoinsFromModuleToAccountVirtual` checks the actual bank balance rather than a virtual shadow balance, the refund fails for every transaction with leftover gas in that block.

### Impact Explanation

**High â€” valid user funds/fees are mis-accounted.** The sender pays the full `gasLimit * gasPrice` fee via the ante handler. If `RefundGas` subsequently fails, the transaction is reverted at the Cosmos level: the EVM state changes are discarded, but the fee deduction stands. The sender loses their entire fee without their transaction being executed. If the fee collector balance is insufficient for all transactions in a block (e.g., at block start after distribution), every EVM transaction with leftover gas fails, causing a deterministic block-processing failure across all validators â€” a chain halt.

### Likelihood Explanation

The fee collector's actual balance is zero at the start of each block after end-of-block fee distribution. If `SendCoinsFromModuleToAccountVirtual` falls back to checking the actual module account balance (rather than a purely in-memory virtual ledger), the first transaction in any block that has leftover gas and a non-zero gas price will trigger this failure. The condition is reachable by any unprivileged user submitting a standard EVM transaction with a gas limit higher than actual gas used â€” which is the normal case for every EVM transaction.

### Recommendation

Mirror the RocketPool fix: in `RefundGasWithPrice`, if `SendCoinsFromModuleToAccountVirtual` returns an error, log the failure and continue rather than returning the error. The transaction has already executed; blocking its inclusion over a refund failure mis-accounts fees and breaks Ethereum semantics. The fee collector retains the unrefunded amount, which is preferable to failing the transaction entirely.

```go
err := k.bankKeeper.SendCoinsFromModuleToAccountVirtual(ctx, authtypes.FeeCollectorName, msg.From.Bytes(), refundedCoins)
if err != nil {
    // Log and continue: do not block transaction inclusion over a refund failure.
    k.Logger(ctx).Error("fee collector failed to refund gas, continuing without refund",
        "sender", msg.From, "refund", refundedCoins, "error", err)
}
```

### Proof of Concept

1. Deploy Ethermint with a fee distribution module that sweeps the fee collector at end-of-block (standard Cosmos SDK behavior).
2. At the start of block N+1, the fee collector's actual balance is 0.
3. Submit a standard EVM transfer with `gasLimit = 100_000` and `gasPrice > 0`. Actual gas used is 21_000, leaving `leftoverGas = 79_000`.
4. Ante handler deducts `100_000 * gasPrice` from sender to fee collector (virtual).
5. EVM executes successfully.
6. `RefundGas` attempts to send `79_000 * gasPrice` from fee collector (actual balance = 0) to sender.
7. `SendCoinsFromModuleToAccountVirtual` fails with `ErrInsufficientFunds`.
8. `ApplyTransaction` returns error; Cosmos SDK reverts EVM state.
9. Sender's balance is reduced by `100_000 * gasPrice` (full fee, not refunded), but the transfer never executed.
10. Repeat for every transaction in the block â€” all fail deterministically, halting the chain. [6](#0-5) [7](#0-6)

// RefundGasWithPrice transfers the leftover gas to sender using the provided gas price.
func (k *Keeper) RefundGasWithPrice(
	ctx sdk.Context,
	msg *core.Message,
	leftoverGas uint64,
	gasPrice *big.Int,
	denom string,
) error {
	if gasPrice == nil {
		gasPrice = new(big.Int)
	}

	// Return EVM tokens for remaining gas, exchanged at the original rate.
	remaining := new(big.Int).Mul(new(big.Int).SetUint64(leftoverGas), gasPrice)

	switch remaining.Sign() {
	case -1:
		// negative refund errors
		return errorsmod.Wrapf(types.ErrInvalidRefund, "refunded amount value cannot be negative %d", remaining.Int64())
	case 1:
		// positive amount refund
		refundedCoins := sdk.Coins{sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(remaining))}

		// refund to sender from the fee collector module account, which is the escrow account in charge of collecting tx fees
		err := k.bankKeeper.SendCoinsFromModuleToAccountVirtual(ctx, authtypes.FeeCollectorName, msg.From.Bytes(), refundedCoins)
		if err != nil {
			err = errorsmod.Wrapf(errortypes.ErrInsufficientFunds, "fee collector account failed to refund fees: %s", err.Error())
			return errorsmod.Wrapf(err, "failed to refund %d leftover gas (%s)", leftoverGas, refundedCoins.String())
		}
	default:
		// no refund, consume gas and update the tx gas meter
	}

	return nil
}
```
