### Title
Missing `rejected_txs` Logic in `CatchingUp` State Causes Permanent Loss of Rejected L1 Handler Transactions — (File: `crates/apollo_l1_provider/src/l1_provider.rs`)

### Summary

During the `CatchingUp` transitional state of the L1 provider, the `rejected_txs` parameter is silently dropped in `commit_block()`. Rejected L1 handler transactions are permanently marked as `Committed` instead of remaining `Pending` for re-proposal, causing them to be permanently excluded from future block proposals.

### Finding Description

`L1Provider::commit_block()` accepts both `committed_txs` and `rejected_txs`: [1](#0-0) 

When the provider is in `CatchingUp` state, it routes to `accept_commit_while_catching_up`, **silently dropping `rejected_txs`**: [2](#0-1) 

Inside `accept_commit_while_catching_up`, both the immediate application and the backlog replay call `apply_commit_block` with `Default::default()` (empty set) for `rejected_txs`. The code even contains a TODO acknowledging the gap: [3](#0-2) [4](#0-3) 

The `CommitBlockBacklog` struct itself has no field for `rejected_txs`, so the information is structurally unrecoverable once backlogged: [5](#0-4) 

`apply_commit_block` partitions `consumed_txs` into committed vs. rejected by checking membership in `rejected_txs`. With an empty `rejected_txs`, every consumed transaction — including those that were rejected during execution — is passed to `commit_txs` as a committed transaction: [6](#0-5) 

`commit_txs` then permanently marks those transactions as `Committed` in the `TransactionManager`, removing them from the `proposable_index`: [7](#0-6) 

### Impact Explanation

`rejected_txs` in `commit_block` are L1 handler transactions that were consumed (included in a block by the batcher) but whose execution was reverted. Per the normal flow, these should remain `Pending` so they can be re-proposed in a future block. During `CatchingUp`, they are instead permanently marked `Committed`. The `proposable_index` invariant ("contains all hashes of transactions that are proposable, and only them") is violated: these transactions are removed from the index and will never be proposed again. The corresponding L1→L2 messages are permanently unprocessed on L2.

This matches the allowed impact: **High — valid L1 handler transactions are permanently rejected before sequencing** by the L1 provider's admission logic during the `CatchingUp` transitional state.

### Likelihood Explanation

`CatchingUp` is entered on every node restart where the batcher's committed height is ahead of the L1 provider's current height — a routine operational scenario. Any block in the catch-up range that contained a rejected L1 handler transaction (execution revert of an L1 handler) will silently corrupt the provider's transaction state. No privileged access or adversarial peer is required; normal sequencer operation suffices.

### Recommendation

1. Pass `rejected_txs` through `accept_commit_while_catching_up` and store it in `CommitBlockBacklog`.
2. Forward the stored `rejected_txs` to `apply_commit_block` in both the `Equal` branch and the backlog replay loop.
3. Remove the `TODO(guyn)` comment and add a regression test covering the `CatchingUp` path with rejected transactions.

### Proof of Concept

```
1. L1 provider initializes at height 5 (Pending state).
2. Batcher calls commit_block(consumed={tx_A}, rejected={tx_A}, height=7).
   → Height mismatch triggers start_catching_up(7); state = CatchingUp.
3. Sync task sends commit_block(consumed={}, rejected={}, height=5).
   → accept_commit_while_catching_up: Equal branch, apply_commit_block({}, {}).
4. Sync task sends commit_block(consumed={}, rejected={}, height=6).
   → Equal branch, apply_commit_block({}, {}).
   → is_caught_up(7) = true; backlog is empty; state = Pending.
5. Batcher calls commit_block(consumed={tx_A}, rejected={tx_A}, height=7).
   → Normal path: apply_commit_block({tx_A}, {tx_A}).
   → rejected_and_consumed = [tx_A]; committed_txs = [].
   → commit_txs([], [tx_A]) → tx_A marked Rejected (kept Pending). ✓

Now repeat with CatchingUp active at the moment height=7 arrives:

1. L1 provider at height 5, state = CatchingUp (target=7).
2. Batcher calls commit_block(consumed={tx_A}, rejected={tx_A}, height=7).
   → is_catching_up() = true → accept_commit_while_catching_up({tx_A}, 7).
   → rejected_txs silently dropped.
   → Greater branch: add_commit_block_to_backlog({tx_A}, 7). rejected_txs lost.
3. Sync delivers heights 5, 6, 7 via commit_block.
   → At height 7: is_caught_up(8) = true.
   → Backlog replay: apply_commit_block({tx_A}, Default::default()).
   → rejected_and_consumed = [] (tx_A not in empty rejected set).
   → committed_txs = [tx_A].
   → commit_txs([tx_A], []) → tx_A permanently marked Committed.
   → tx_A removed from proposable_index; L1→L2 message never re-proposed.
```

        &mut self,
        consumed_txs: IndexSet<TransactionHash>,
        rejected_txs: IndexSet<TransactionHash>,
    ) {
        debug!("Applying commit_block to height: {}", self.current_height);
        let (rejected_and_consumed, committed_txs): (Vec<_>, Vec<_>) =
            consumed_txs.iter().copied().partition(|tx| rejected_txs.contains(tx));
        self.tx_manager.commit_txs(&committed_txs, &rejected_and_consumed);

        self.current_height = self.current_height.unchecked_next();

    pub fn commit_txs(
        &mut self,
        committed_txs: &[TransactionHash],
        rejected_txs: &[TransactionHash],
    ) {
        self.rollback_staging();

        for &tx_hash in committed_txs {
            self.create_record_if_not_exist(tx_hash);
            self.with_record(tx_hash, |r| r.mark_committed()).unwrap();
        }
        for &tx_hash in rejected_txs {
            self.with_record(tx_hash, |r| r.mark_rejected()).expect(
                "Storage inconsistency: a transaction sent to the batcher was removed \
                 unexpectedly.",
            );
        }
    }
