### Title
Missing Class Hash Validation in `execute_replace_class` Enables Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary
The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the new class hash supplied by a contract is actually a declared class. Any contract can call `replace_class` with an arbitrary undeclared felt value, permanently corrupting its own class hash entry in the state. Once corrupted, no subsequent transaction can successfully call that contract — the OS execution itself would fail — permanently freezing all funds held by the contract.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall and directly writes the caller-supplied `request.class_hash` into `contract_state_changes` with no check that the hash corresponds to a declared class:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The OS developers themselves acknowledge the missing check with the `TODO` comment at line 898. The `class_hash` value written here is fully attacker-controlled — it is taken directly from the syscall request with no range or existence check.

When any subsequent transaction attempts to call the affected contract, `execute_entry_point` reads the corrupted class hash and performs a lookup in `contract_class_changes`:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
``` [2](#0-1) 

Because the invalid hash was never declared, `dict_read` returns the default value `0`. The OS then calls `find_element` to locate a `CompiledClassFact` with hash `0`:

```cairo
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [3](#0-2) 

`find_element` is a Cairo assertion-based search: if no element with the given key exists, the prover cannot satisfy the constraint and the entire OS proof fails. The sequencer therefore cannot include any transaction that calls the corrupted contract. The contract becomes permanently uncallable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract holding assets (tokens, ETH bridged via L1→L2, NFTs, DeFi positions) that calls `replace_class` with an undeclared hash becomes permanently inaccessible. No transaction can ever call it again without causing the OS proof to fail. The funds are irrecoverably frozen on-chain with no recovery path, because the OS state update has already committed the corrupted class hash to the state root.

---

### Likelihood Explanation

**Medium.** The attack surface is any contract that exposes a code path reaching `replace_class`. Concrete scenarios:

1. **Malicious contract**: An attacker deploys a contract that accepts user deposits and then calls `replace_class` with an arbitrary felt (e.g., `1`) to freeze deposited funds — a rug-pull variant.
2. **Buggy contract**: A contract with a logic error passes an unvalidated user-supplied value to `replace_class`, allowing an external caller to trigger the freeze.
3. **Griefing**: An attacker deploys a contract, deposits a small amount, then freezes it to demonstrate the vulnerability or disrupt protocol integrations.

The syscall is reachable by any unprivileged transaction sender who can invoke a contract that calls `replace_class`. No special role, key, or operator access is required.

---

### Recommendation

Add an existence check inside `execute_replace_class` before writing the new class hash. Specifically, verify that `request.class_hash` has a non-zero entry in `contract_class_changes` (i.e., it has been declared and has an associated compiled class hash). If the check fails, write a failure response and return without modifying state, consistent with how other syscall failures are handled (e.g., `write_failure_response`).

---

### Proof of Concept

1. Attacker deploys contract `C` (holding user funds) whose code contains:
   ```
   replace_class(class_hash=0xdeadbeef)  // 0xdeadbeef is never declared
   ```
2. Attacker (or a tricked user) calls the function on `C`. The OS processes `execute_replace_class` successfully — no validation occurs — and commits `C.class_hash = 0xdeadbeef` to `contract_state_changes`.
3. The state root is updated with `C`'s corrupted class hash entry.
4. Any subsequent transaction targeting `C` causes `execute_entry_point` to call `dict_read(contract_class_changes, 0xdeadbeef)` → returns `0`, then `find_element(..., key=0)` → no matching `CompiledClassFact` → OS proof constraint violated → sequencer cannot include the transaction.
5. `C`'s funds are permanently frozen with no recovery mechanism.
