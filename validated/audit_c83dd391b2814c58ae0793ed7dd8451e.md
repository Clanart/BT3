### Title
Incorrect `memcpy` Length in `replace_storage_diff` Corrupts State Diff Output — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/state/aliases.cairo`)

---

### Summary

In `replace_storage_diff`, the length passed to `memcpy` for copying small-key storage entries is computed as the number of `FullStateUpdateEntry` **elements** rather than the number of **felts**. Because each `FullStateUpdateEntry` is 3 felts wide, only one-third of the required data is copied, and the destination pointer `res` is advanced by the same wrong amount. This corrupts the serialized state diff that is published on L1 for data availability.

---

### Finding Description

In `aliases.cairo`, `replace_storage_diff` copies storage entries whose keys are below `MIN_VALUE_FOR_ALIAS_ALLOC` (i.e., keys < 128) verbatim into `res`:

```cairo
func replace_storage_diff{range_check_ptr, res: felt*}(
    aliases: Aliases,
    storage_diff_start: FullStateUpdateEntry*,
    storage_diff_end: FullStateUpdateEntry*,
) {
    alloc_locals;
    let storage_diff_big_keys_start = get_storage_diff_big_keys_start(...);
    local small_keys_len = storage_diff_big_keys_start - storage_diff_start;  // BUG: in elements
    memcpy(dst=res, src=storage_diff_start, len=small_keys_len);              // BUG: len in felts expected
    let res = &res[small_keys_len];                                            // BUG: advances by elements
    ...
}
``` [1](#0-0) 

In Cairo, subtracting two `FullStateUpdateEntry*` pointers yields the number of **elements**, not felts. Since `FullStateUpdateEntry.SIZE = 3` (fields: `key`, `prev_value`, `new_value`), if there are N small-key entries, `small_keys_len = N`, but `memcpy` needs `N * 3` felts to copy all entries. The call copies only N felts (just the `key` field of each entry), and `res` is advanced by only N felts instead of 3N. [2](#0-1) 

Compare with the **correct** pattern used in `replace_contract_state_diff` in the same file, which explicitly casts to `felt*` before subtraction to obtain a felt-based length:

```cairo
tempvar diff_len = cast(storage_diff_end, felt*) - contract_state_diff;
memcpy(dst=res, src=contract_state_diff, len=diff_len);
``` [3](#0-2) 

The `replace_storage_diff` function lacks this cast, making it the direct analog of the `strncpy` bug: an incorrect offset/size causes the copy to write/advance by the wrong amount, corrupting the destination buffer.

---

### Impact Explanation

`replace_storage_diff` is called from `replace_contract_state_diff` → `replace_aliases_and_serialize_full_contract_state_diff` → `serialize_contract_state_diff_conditional` → `output_contract_state` → `serialize_os_output`. [4](#0-3) 

The output of this pipeline is the state diff written to `state_updates_ptr`, which is then compressed and published on L1 as data availability. [5](#0-4) 

When the bug fires:
- Only the `key` field of each small-key entry is copied; `prev_value` and `new_value` are omitted.
- `res` is advanced by N felts instead of 3N, so the subsequent `replace_storage_diff_big_keys` writes its output overlapping the partially-copied small-key region.
- The resulting state diff is structurally malformed.

The STARK proof still verifies (it proves the Cairo execution was correct, including the buggy path). The L1 contract accepts the proof and updates the state root. However, the data availability published on L1 is corrupted. Users and full nodes cannot reconstruct the correct L2 state from the on-chain data, making it impossible to prove balances or initiate L1 exits. This constitutes **permanent freezing of funds**.

---

### Likelihood Explanation

The bug is triggered whenever:
1. `replace_keys_with_aliases` is enabled (the alias-compression feature, active in production StarkNet).
2. A contract with address > `MAX_NON_COMPRESSED_CONTRACT_ADDRESS` (= 15) has at least one storage write to a slot < `MIN_VALUE_FOR_ALIAS_ALLOC` (= 128). [6](#0-5) 

Storage slot 0 is the most common slot used by contracts (e.g., ERC-20 balances, counters, ownership). Any standard contract deployed at address > 15 that writes to slot 0 triggers this path. This is trivially achievable by any unprivileged transaction sender with no special privileges.

---

### Recommendation

Replace the element-count subtraction with a felt-count subtraction by casting both pointers to `felt*` before subtracting, matching the pattern already used correctly in `replace_contract_state_diff`:

```cairo
// Correct fix:
local small_keys_felt_len = cast(storage_diff_big_keys_start, felt*) - cast(storage_diff_start, felt*);
memcpy(dst=res, src=cast(storage_diff_start, felt*), len=small_keys_felt_len);
let res = &res[small_keys_felt_len];
```

---

### Proof of Concept

1. Deploy a contract at address > 15 that writes to storage slot 0 (e.g., `storage_write(address=0, value=1)`).
2. Submit a transaction invoking this contract.
3. The OS processes the block with `replace_keys_with_aliases = 1`.
4. `replace_storage_diff` is called with `storage_diff_start` pointing to the slot-0 entry (key < 128), `small_keys_len = 1`.
5. `memcpy` copies 1 felt (only the `key` field = 0) instead of 3 felts.
6. `res` advances by 1 felt instead of 3.
7. `replace_storage_diff_big_keys` writes its output starting 2 felts too early, overwriting the partially-copied entry.
8. The published state diff on L1 is malformed; the block is accepted by L1 (valid STARK proof) but the data availability is corrupted, permanently freezing user funds.
