### Title
Missing Chain ID in `deployToken` MPC Signature Payload Enables Cross-Chain Replay — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
The Borsh-encoded message signed by the MPC key for `deployToken` contains no destination chain identifier. Because all EVM OmniBridge deployments share the same `nearBridgeDerivedAddress` (derived from the single NEAR MPC key), a valid `deployToken` signature obtained for chain A is cryptographically identical and accepted on chain B, allowing an unprivileged relayer to deploy a NEAR-origin token on any EVM chain without NEAR-side authorization for that chain.

### Finding Description
In `OmniBridge.deployToken`, the message committed to by the MPC signature is:

```
keccak256(
  PayloadType.Metadata (1 byte)
  | borsh(token)
  | borsh(name)
  | borsh(symbol)
  | decimals (1 byte)
)
``` [1](#0-0) 

No chain identifier appears anywhere in this encoding. By contrast, `finTransfer` explicitly embeds `omniBridgeChainId` twice in its signed payload: [2](#0-1) 

The `MetadataPayload` struct itself has no chain field: [3](#0-2) 

The same chain-ID-absent encoding is replicated verbatim on StarkNet: [4](#0-3) 

And on Solana: [5](#0-4) 

Because the MPC key is a single NEAR bridge key shared across all destination chains, `nearBridgeDerivedAddress` is identical on every EVM OmniBridge instance. A signature that satisfies `ECDSA.recover(hash, sig) == nearBridgeDerivedAddress` on chain A will satisfy the same check on chain B with the same `(token, name, symbol, decimals)` tuple.

### Impact Explanation
An attacker who observes (or front-runs) a valid `deployToken` call on chain A can immediately replay the same `(signatureData, metadata)` arguments on chain B. The call passes signature verification and succeeds unless the token was already deployed on chain B. The result is:

- A `BridgeToken` proxy is deployed on chain B without NEAR-side authorization for that chain.
- `nearToEthToken[metadata.token]` is permanently set on chain B to the attacker-triggered proxy address.
- Any subsequent legitimate attempt to deploy the canonical token on chain B reverts with `ERR_TOKEN_EXIST`. [6](#0-5) 

The hijacked mapping is canonical: all future `finTransfer` calls on chain B for that NEAR token will route through the attacker-triggered proxy. While the proxy itself is a standard `BridgeToken` (minting is gated to the bridge), the NEAR side has no record of this deployment and may never issue a corresponding `finTransfer` for chain B, effectively orphaning the mapping and blocking legitimate deployment permanently.

### Likelihood Explanation
The precondition — same `nearBridgeDerivedAddress` across chains — is the normal production configuration, not a special case. A valid `deployToken` signature is publicly observable in any chain A transaction or mempool. The replay requires only submitting the same calldata to chain B's `deployToken`, which is a public, permissionless function. No privileged access, leaked key, or colluding MPC signer is needed.

### Recommendation
Include `omniBridgeChainId` in the Borsh-encoded payload that the MPC key signs for `deployToken`, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to this chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent fix to the StarkNet `MetadataPayload.to_borsh()` and the Solana `DeployTokenPayload.serialize_for_near()` implementations, and update the NEAR-side MPC signing logic to include the destination chain ID in the metadata payload it signs.

### Proof of Concept
1. Deploy two `OmniBridge` instances — `BridgeA` (chain ID = 1) and `BridgeB` (chain ID = 2) — both initialized with the same `nearBridgeDerivedAddress`.
2. Obtain a valid MPC signature `sig` for `deployToken` targeting `BridgeA` with `metadata = {token: "foo.near", name: "Foo", symbol: "FOO", decimals: 18}`.
3. Call `BridgeA.deployToken(sig, metadata)` → succeeds, token deployed on chain A.
4. Call `BridgeB.deployToken(sig, metadata)` with the **identical** `sig` and `metadata` → also succeeds, token deployed on chain B without any chain-B-specific MPC authorization.
5. Assert `BridgeB.nearToEthToken("foo.near") != address(0)` — the canonical mapping on chain B is now hijacked.

The Borsh encoding is identical for both calls because neither `omniBridgeChainId` nor any chain-specific field appears in the signed payload, so the same `keccak256` hash is produced and the same signature recovers to `nearBridgeDerivedAddress` on both instances.
