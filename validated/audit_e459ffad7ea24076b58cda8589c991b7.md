### Title
Re-org Attack Corrupts Token Address Mapping via `CREATE`-based Proxy Deployment — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.deployToken()` deploys each bridged token proxy using the `new ERC1967Proxy(...)` constructor, which uses the EVM `CREATE` opcode. The resulting proxy address is derived solely from the `OmniBridge` contract's nonce. On L2 chains (where Omni Bridge is deployed and re-orgs are frequent), a block reorganization can change the nonce at deployment time, causing the proxy to land at a different address than expected. This corrupts the `nearToEthToken` / `ethToNearToken` bidirectional mapping and can cause the MPC to mint the wrong token to bridge users.

---

### Finding Description

In `OmniBridge.deployToken()`, the bridge proxy is deployed as:

```solidity
address bridgeTokenProxy = address(
    new ERC1967Proxy(
        tokenImplementationAddress,
        abi.encodeWithSelector(
            BridgeToken.initialize.selector,
            metadata.name,
            metadata.symbol,
            decimals
        )
    )
);
``` [1](#0-0) 

`new ERC1967Proxy(...)` compiles to the `CREATE` opcode. The deployed address is `keccak256(rlp(OmniBridge_address, nonce))[12:]` — it depends entirely on the `OmniBridge` contract's current nonce, not on any property of the NEAR token being registered.

After deployment, the address is stored in both directions:

```solidity
ethToNearToken[address(bridgeTokenProxy)] = metadata.token;
nearToEthToken[metadata.token] = address(bridgeTokenProxy);
``` [2](#0-1) 

The `MetadataPayload` signed by the MPC covers only `token`, `name`, `symbol`, and `decimals` — it does **not** commit to the expected deployed address:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
``` [3](#0-2) 

`finTransfer` then uses `payload.tokenAddress` (the EVM proxy address) to decide which token to mint:

```solidity
} else if (isBridgeToken[payload.tokenAddress]) {
    IBridgeToken(payload.tokenAddress).mint(payload.recipient, payload.amount);
``` [4](#0-3)

### Impact Explanation

If two `deployToken` calls are pending simultaneously (e.g., during initial multi-token onboarding) and a re-org reorders them, the nonces shift:

| | Pre-reorg | Post-reorg |
|---|---|---|
| `nearToEthToken["usdc.near"]` | `0xAAA` (nonce 1) | `0xBBB` (nonce 2) |
| `nearToEthToken["usdt.near"]` | `0xBBB` (nonce 2) | `0xAAA` (nonce 1) |

The NEAR MPC, having observed the pre-reorg `DeployToken` events, signs `finTransfer` payloads referencing `0xAAA` for USDC and `0xBBB` for USDT. After the re-org those addresses now belong to the swapped proxies. Submitting those signed payloads causes `finTransfer` to mint USDC tokens on the USDT proxy and vice versa — delivering the wrong asset to every recipient. This is token-mapping corruption that breaks bridge collateralization and misdirects value.

**Allowed impact matched:** *Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value (High).*

---

### Likelihood Explanation

Omni Bridge explicitly targets L2 deployments (Arbitrum, Base, etc.) where multi-block re-orgs are well-documented. The `deployToken` function is callable by any address holding a valid MPC signature, and multiple token onboardings can realistically occur in the same block during bridge bootstrapping or batch token additions. No privileged access is required beyond a valid MPC-signed `MetadataPayload`, which is a normal protocol operation.

---

### Recommendation

Replace `new ERC1967Proxy(...)` with a `CREATE2` deployment using a salt derived from the NEAR token identifier (`metadata.token`). This makes the proxy address deterministic and independent of nonce:

```solidity
bytes32 salt = keccak256(abi.encodePacked(metadata.token));
address bridgeTokenProxy = address(
    new ERC1967Proxy{salt: salt}(
        tokenImplementationAddress,
        abi.encodeWithSelector(
            BridgeToken.initialize.selector,
            metadata.name,
            metadata.symbol,
            decimals
        )
    )
);
```

Because `metadata.token` is the unique NEAR account ID of the token (e.g., `"usdc.near"`), the salt is collision-resistant and the address is fully predictable before deployment, surviving any re-org.

---

### Proof of Concept

1. Two `deployToken` transactions are broadcast in the same block:
   - Tx A: `metadata.token = "usdc.near"` → proxy lands at `0xAAA` (OmniBridge nonce = 1)
   - Tx B: `metadata.token = "usdt.near"` → proxy lands at `0xBBB` (OmniBridge nonce = 2)
2. NEAR MPC observes `DeployToken(0xAAA, "usdc.near")` and `DeployToken(0xBBB, "usdt.near")` and begins signing `finTransfer` payloads with those addresses.
3. A re-org drops the block. In the reorganized chain Tx B is included first:
   - Tx B: `"usdt.near"` → `0xAAA` (nonce = 1)
   - Tx A: `"usdc.near"` → `0xBBB` (nonce = 2)
4. `nearToEthToken["usdc.near"] = 0xBBB`, `nearToEthToken["usdt.near"] = 0xAAA`.
5. A relayer submits the pre-reorg MPC-signed `finTransfer` for a USDC transfer with `tokenAddress = 0xAAA`. The bridge calls `IBridgeToken(0xAAA).mint(recipient, amount)` — minting USDT tokens to the user instead of USDC.
6. The user receives the wrong asset; the USDC proxy at `0xBBB` has unbacked supply while the USDT proxy at `0xAAA` is over-minted.
