### Title
Gateway `validate_proof_block_number` Accepts Proof Block Number That `validate_proof_block_hash` Cannot Verify Due to Missing `pre_process_block` State — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful validator increments the latest committed block number by 1 to simulate the next block for `validate_proof_block_number`, but reads the block-hash contract storage from the state at the **un-incremented** latest block (without calling `pre_process_block`). This creates a one-block gap: a transaction whose `proof_block_number` equals exactly `latest_block_number − STORED_BLOCK_HASH_BUFFER + 1` passes the number-range check but fails the hash-lookup check at the gateway, while the blockifier — which calls `pre_process_block` before executing any transaction — would accept it. The result is that a structurally valid, sequencer-acceptable Invoke V3 transaction carrying client-side proof facts is permanently rejected at the admission layer.

---

### Finding Description

**Root cause — two independent checks use inconsistent block-number contexts.**

In `run_validate_entry_point` the gateway builds a `BlockContext` whose `block_number` is `latest_block_number + 1`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  ~L296-L303
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();   // ← N+1
let block_context = BlockContext::new(block_info, ...);
let state = CachedState::new(state_reader_and_contract_manager);      // ← state at N
let mut blockifier_validator = StatefulValidator::create(state, block_context);
blockifier_validator.validate(account_tx)
```

`StatefulValidator::validate` calls `perform_pre_validation_stage`, which calls `validate_proof_facts`:

**Check 1 — `validate_proof_block_number`** uses `current_block_number = N+1`:
```rust
// crates/blockifier/src/transaction/account_transaction.rs  L244-L257
let max_allowed = current_block_number.0.checked_sub(STORED_BLOCK_HASH_BUFFER)?;
// max_allowed = (N+1) − STORED_BLOCK_HASH_BUFFER = N − STORED_BLOCK_HASH_BUFFER + 1
if proof_block_number > max_allowed { return Err(...) }
```
→ **passes** for `proof_block_number = N − STORED_BLOCK_HASH_BUFFER + 1`.

**Check 2 — `validate_proof_block_hash`** reads from the state at block N:
```rust
// crates/blockifier/src/transaction/account_transaction.rs  L278-L285
let stored_block_hash = state.get_storage_at(block_hash_contract_address,
    StorageKey::from(proof_block_number))?;
if stored_block_hash != proof_block_hash { return Err("Block hash mismatch") }
```
The hash for block `N − STORED_BLOCK_HASH_BUFFER + 1` is written to the block-hash contract **only** during block `N+1`'s execution via `pre_process_block`:

```rust
// crates/blockifier/src/blockifier/block.rs  L18-L33
pub fn pre_process_block(state, old_block_number_and_hash, next_block_number, ...) {
    // writes hash of (next_block_number − STORED_BLOCK_HASH_BUFFER) into contract storage
    state.set_storage_at(block_hash_contract_address,
        StorageKey::from(number.0), hash.0)?;
}
```

The gateway's `StatefulValidator::create` does **not** call `pre_process_block` (only `TransactionExecutor::pre_process_and_create` does). Therefore `stored_block_hash = Felt::ZERO` at gateway validation time, and Check 2 fails with `"Block hash mismatch"`.

**The batcher's `TransactionExecutor`**, by contrast, calls `pre_process_block` before executing any transaction:
```rust
// crates/blockifier/src/blockifier/transaction_executor.rs  L108-L115
pre_process_block(&mut block_state, old_block_number_and_hash,
    block_context.block_info().block_number, ...)?;
```
After this call the hash for `N − STORED_BLOCK_HASH_BUFFER + 1` is present in the state, so both checks pass and the transaction is accepted.

**Structural parallel to the external report:**
| External (Escrow) | Sequencer analog |
|---|---|
| `loanStatus == LIQUIDATABLE` passes (deadline expired) | `validate_proof_block_number` passes (`proof_block_number ≤ N+1 − BUFFER`) |
| `minimumCollateralRatio > _getLatestCollateralRatio()` fails (ratio still above threshold) | `validate_proof_block_hash` fails (`stored_hash = 0 ≠ proof_block_hash`) |
| Arbiter cannot liquidate a legitimately liquidatable loan | Gateway rejects a legitimately sequenceable transaction |

---

### Impact Explanation

Any user who constructs a valid Invoke V3 transaction whose `proof_block_number` equals exactly `latest_committed_block − STORED_BLOCK_HASH_BUFFER + 1` will have that transaction permanently rejected by the gateway with `"Block hash mismatch"`, even though the blockifier would accept and execute it correctly. This satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission … rejects valid transactions before sequencing."* The wrong authoritative value is the `stored_block_hash = 0` returned by the gateway's state reader for a key that has not yet been written by `pre_process_block`, causing the gateway to emit an incorrect rejection verdict.

---

### Likelihood Explanation

The condition is deterministic and reproducible: it is triggered whenever a user submits a proof-bearing Invoke V3 transaction referencing the newest block whose hash is permitted by the number check but not yet stored. Because `STORED_BLOCK_HASH_BUFFER = 10`, this is a narrow but predictable one-block window that recurs with every new committed block. No privileged access is required; any unprivileged user can trigger it by choosing the boundary `proof_block_number`.

---

### Recommendation

Align the two checks by either:

1. **Reducing the allowed range at the gateway**: use `latest_block_number` (not `latest_block_number + 1`) as `current_block_number` when calling `validate_proof_block_number` in the gateway context, so the maximum allowed `proof_block_number` is `latest_block_number − STORED_BLOCK_HASH_BUFFER`, which is guaranteed to be present in the state.

2. **Simulating `pre_process_block` in the gateway validator**: before creating the `StatefulValidator`, apply `pre_process_block` to the cached state so that the hash for `(N+1) − STORED_BLOCK_HASH_BUFFER` is present when `validate_proof_block_hash` reads it.

Option 1 is simpler and mirrors the fix applied in the external report (removing the secondary check that was inconsistent with the primary status check).

---

### Proof of Concept

Let `N` = latest committed block number, `B` = `STORED_BLOCK_HASH_BUFFER` (= 10).

1. User constructs an Invoke V3 transaction with `proof_block_number = N − B + 1` and a correct `proof_block_hash` (the real hash of block `N − B + 1`).
2. Gateway calls `run_validate_entry_point` with `block_context.block_number = N + 1`, state = state at block N.
3. `validate_proof_block_number`: `max_allowed = (N+1) − B = N − B + 1`; `proof_block_number = N − B + 1 ≤ N − B + 1` → **passes**.
4. `validate_proof_block_hash`: reads `state.get_storage_at(BLOCK_HASH_CONTRACT, N−B+1)` → returns `Felt::ZERO` (not yet written; `pre_process_block` for block N+1 has not run). `Felt::ZERO ≠ proof_block_hash` → **fails** with `"Block hash mismatch"`.
5. Gateway returns error; transaction is rejected and never reaches the mempool.
6. If the same transaction were submitted directly to the batcher (block N+1 execution): `pre_process_block` writes `hash(N−B+1)` into the contract; `validate_proof_block_hash` reads the correct hash → **passes**; transaction executes successfully.     fn validate_proof_block_hash(
        proof_block_hash: Felt,
        proof_block_number: u64,
        os_constants: &OsConstants,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        if proof_block_hash == Felt::ZERO {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof block hash is zero for block {proof_block_number}."
            )));
        }

        // Compare the proof's block hash with the stored block hash.
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();

        let stored_block_hash = state
            .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

        if stored_block_hash != proof_block_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Block hash mismatch for block {proof_block_number}. Proof block hash: \
                 {proof_block_hash}, stored block hash: {stored_block_hash}."
            )));
        }

        Ok(())
    }

pub fn pre_process_block(
    state: &mut dyn State,
    old_block_number_and_hash: Option<BlockHashAndNumber>,
    next_block_number: BlockNumber,
    os_constants: &OsConstants,
) -> StateResult<()> {
    let should_block_hash_be_provided =
        next_block_number >= BlockNumber(constants::STORED_BLOCK_HASH_BUFFER);
    if let Some(BlockHashAndNumber { number, hash }) = old_block_number_and_hash {
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();
        let block_number_as_storage_key = StorageKey::from(number.0);
        state.set_storage_at(block_hash_contract_address, block_number_as_storage_key, hash.0)?;
    } else if should_block_hash_be_provided {
        return Err(StateError::OldBlockHashNotProvided);
    }

    Ok(())
