### Title
Unauthenticated Bootstrap Declare Path Allows Arbitrary `compiled_class_hash` Injection, Permanently Poisoning Any Class - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_declare_transaction` contains a special "bootstrap" code path that completely bypasses signature verification, nonce enforcement, and fee payment for any declare transaction whose `sender_address` equals the felt literal `'BOOTSTRAP'`, `nonce == 0`, `version == 3`, and all resource bounds are zero. Within this path the only constraint on `compiled_class_hash` is `assert_not_zero`. An unprivileged user can craft such a transaction, supply an arbitrary (wrong) `compiled_class_hash` for any valid Sierra class hash, and permanently register that poisoned mapping in the global state. Because the OS enforces `prev_value=0` on every class declaration, the class can never be re-declared correctly, making every contract that uses it permanently un-executable and freezing any funds held by those contracts.

---

### Finding Description

In `execute_declare_transaction` the following branch is reached before any account validation or fee logic:

```cairo
// Do not run validate or perform any account-related actions for declare transactions that
// meet the following conditions.
// This flow is used for the sequencer to bootstrap a new system.
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        // Declare the class hash and skip the rest of the transaction.
        // Note that prev_value=0 enforces that a class may be declared only once.
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

The four triggering conditions are entirely under attacker control:

| Field | Required value | Attacker control |
|---|---|---|
| `sender_address` | felt `'BOOTSTRAP'` | Free choice in tx |
| `nonce` | `0` | Free choice in tx |
| `version` | `3` | Free choice in tx |
| all resource bounds | `0` | Free choice in tx |

Once inside the branch the only check on `compiled_class_hash` is `assert_not_zero`. The Sierra class hash itself **is** validated cryptographically via `finalize_class_hash` (lines 738–743), so the attacker cannot forge the class hash — but they can supply any non-zero felt as the CASM hash for a legitimately-structured Sierra class. [2](#0-1) 

The normal declare path (lines 816–818) also calls `assert_not_zero(compiled_class_hash)` before writing to `contract_class_changes`, but it additionally runs `__validate_declare__` on the sender's account contract, increments the nonce, and charges a fee — none of which happen in the bootstrap branch. [3](#0-2) 

The `dict_update` call uses `prev_value=0`, which is the OS-level enforcement that a class may be declared **at most once**. After the attacker's transaction is included in a block, the mapping `class_hash → attacker_compiled_class_hash` is committed to the state and can never be overwritten. [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the OS later executes any contract whose class was poisoned this way, it must look up the CASM for that class hash and verify that its hash matches the value stored in state. Because the stored value is an attacker-chosen garbage felt (e.g., `1`), the compiled-class-facts validation (`validate_compiled_class_facts_post_execution`, called at the end of every block) will fail for every block that attempts to execute such a contract. No valid proof can ever be generated for those blocks. The contracts are permanently un-executable, and any funds they hold are permanently frozen.

If the attacker targets a system-critical class (fee token, account class, bridge contract class) before it is legitimately declared, the entire network's ability to process transactions that depend on that class is destroyed.

---

### Likelihood Explanation

**High.** The four triggering conditions require no privileged key, no special role, and no prior on-chain state. Any user who can submit a declare transaction to the sequencer can craft one with `sender_address = 'BOOTSTRAP'`, `nonce = 0`, `version = 3`, and zero resource bounds. The OS itself — the ground truth for proof validity — accepts such a transaction unconditionally. A race condition (front-running a legitimate bootstrap declare) is the only timing requirement, and the attacker has an indefinite window before the legitimate bootstrap transaction is included.

---

### Recommendation

1. **Restrict the bootstrap path to a sequencer-controlled mechanism.** The bootstrap sender address should be validated against a hard-coded, protocol-level address that cannot be used by ordinary transaction senders, or the path should be gated by a block-level flag set only during genesis.
2. **Validate `compiled_class_hash` against the actual CASM.** Even within the bootstrap path, the `compiled_class_hash` should be verified against the compiled class facts bundle (as done in `validate_compiled_class_facts_post_execution`) before the mapping is committed to state.
3. **Enforce nonce uniqueness for the bootstrap sender.** If the bootstrap path must remain, at minimum the nonce should be incremented to prevent replay of the same bootstrap sender across multiple declarations without any account state.

---

### Proof of Concept

1. Attacker identifies a Sierra class `C` with a valid, well-formed class hash `H` (e.g., a future system class not yet declared on-chain).
2. Attacker constructs a v3 declare transaction:
   - `sender_address = 'BOOTSTRAP'` (felt `0x424f4f545354524150`)
   - `nonce = 0`
   - `version = 3`
   - `resource_bounds = [(L1_GAS, 0, 0), (L2_GAS, 0, 0), (L1_DATA_GAS, 0, 0)]`
   - `class_hash = H` (valid Sierra class hash, passes `finalize_class_hash`)
   - `compiled_class_hash = 1` (non-zero, passes `assert_not_zero`, but wrong CASM hash)
3. The sequencer includes this transaction. The OS executes `execute_declare_transaction`:
   - `finalize_class_hash` validates `H` ✓
   - Bootstrap branch is entered (`sender_address == 'BOOTSTRAP'`, `nonce == 0`, `version == 3`, `max_fee == 0`) ✓
   - `assert_not_zero(1)` passes ✓
   - `dict_update(key=H, prev_value=0, new_value=1)` commits the poisoned mapping ✓
   - No signature check, no nonce increment, no fee charged.
4. State now contains `H → compiled_class_hash=1`.
5. Any future block attempting to deploy or call a contract of class `H` will fail `validate_compiled_class_facts_post_execution` because the actual CASM hash ≠ 1. No valid proof can be generated. Funds in any such contract are permanently frozen.
6. The legitimate declare of `H` with the correct CASM hash is rejected by `dict_update`'s `prev_value=0` assertion — the class is permanently poisoned. [1](#0-0)
