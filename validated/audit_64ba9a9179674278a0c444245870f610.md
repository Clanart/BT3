### Title
Stale pending-epoch class body served by `starknet_getClass(Pending, H)` due to missing staleness guard in `get_class` — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary
`JsonRpcServerImpl::get_class` reads from `pending_classes` before performing any parent-hash staleness check. After a new block is committed, there is a window during which `pending_classes` still contains classes from the previous pending epoch. Any caller of `starknet_getClass(block_id=Pending, class_hash=H)` during this window receives a class body that belongs to a stale (superseded) pending epoch.

### Finding Description
In `get_class` (`api_impl.rs` lines 603–606), when `block_id` is `BlockId::Tag(Tag::Pending)`, the handler reads `self.pending_classes.read().await.get_class(class_hash)` and returns immediately if a class is found. [1](#0-0) 

The staleness check — comparing `pending_data.block.parent_block_hash()` against the latest committed block hash — is implemented in `read_pending_data` (`api_impl.rs` lines 1569–1571), but `get_class` never calls this function. [2](#0-1) 

`pending_classes` is only cleared in the sync loop (`pending_sync.rs` lines 179–181) when the sync loop detects a parent hash change in the pending source. [3](#0-2) 

`pending_classes` and `pending_data` are separate `Arc<RwLock<...>>` objects; clearing one does not affect the other. [4](#0-3) 

All other pending-path RPC handlers (`get_nonce`, `get_state_update`, `get_block_transaction_count`, `maybe_get_class_hash_at`) call `read_pending_data` and thus respect the staleness boundary. `get_class` is the sole exception. [5](#0-4) 

### Impact Explanation
During the window between a new block being committed to storage and the sync loop clearing `pending_classes`, any unprivileged caller of `starknet_getClass(Pending, H)` receives the class body from the previous pending epoch. If class H was not included in the newly committed block (e.g., the pending block was reorganized or the declaration was dropped), the returned class body is cross-epoch data that does not belong to the current pending view. This is a High-severity incorrect authoritative-looking value from the RPC pending view.

### Likelihood Explanation
The staleness window is bounded by the sync loop's polling interval (`sleep_duration` in `sync_pending_data`). During normal operation this window is short but non-zero and occurs on every block. Any client polling `starknet_getClass(Pending, ...)` at block boundaries will hit this window. No special privileges are required.

### Recommendation
In `get_class`, before reading from `pending_classes`, perform the same staleness check used by all other pending-path handlers. Concretely, open a storage transaction, call `read_pending_data`, and only consult `pending_classes` if the pending data is current (i.e., `read_pending_data` returned the real pending data, not the synthetic empty block). Alternatively, refactor `read_pending_data` to also return a snapshot of `pending_classes` filtered to the current epoch, so all pending-path handlers share a single consistent staleness boundary.

### Proof of Concept
1. Populate `pending_classes` with class H (parent = block N).
2. Commit block N+1 to storage (making the pending block stale).
3. Do NOT yet trigger the sync loop to clear `pending_classes`.
4. Call `starknet_getClass(Pending, H)`.
5. Observe that the stale class body for H is returned, even though the current pending epoch (parent = block N+1) has no knowledge of H.

The existing test `get_class` in `crates/apollo_rpc/src/v0_8/api/test.rs` (lines 1139–1152) adds a class to `pending_classes` and verifies it is returned for `BlockId::Pending`, but it never advances the committed chain to make the pending data stale, so the staleness path is untested. [6](#0-5)
