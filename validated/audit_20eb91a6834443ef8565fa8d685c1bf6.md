### Title
`ft_transfer` Failure During `fin_transfer` Finalization Permanently Locks User Funds in NEAR Bridge — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When the NEAR bridge finalizes an inbound transfer (EVM → NEAR) for a native token (e.g., USDC), it marks the transfer as finalized and decrements the locked-token counter **before** the `ft_transfer` to the recipient completes. If `ft_transfer` fails (e.g., the token is paused), the callback `fin_transfer_send_tokens_callback` does not detect the failure and does not revert the state. The transfer is permanently marked as finalized, the locked-token accounting is permanently corrupted, and the user's tokens are irrecoverably stuck in the bridge contract.

---

### Finding Description

**Vulnerability class:** Callback/state desync — token transfer failure during finalization is silently ignored, leaving bridge state permanently inconsistent.

**Root cause in `near/omni-bridge/src/lib.rs`:**

`process_fin_transfer_to_near` (line 1867) executes the following sequence:

1. **Marks transfer as finalized** — `add_fin_transfer` inserts the transfer ID into `finalised_transfers` (line 1875). This is irreversible; any retry panics with `BridgeError::TransferAlreadyFinalised`.
2. **Decrements locked-token counter** — `unlock_tokens_if_needed` reduces `locked_tokens` (lines 1881–1885).
3. **Calls `send_tokens`** — for a native token with no message (`msg.is_empty()`), this dispatches `ft_transfer` (lines 2102–2106 in `send_tokens`).
4. **Registers `fin_transfer_send_tokens_callback` as the `.then()` callback** (lines 1967–1977), passing `is_ft_transfer_call = !msg.is_empty()` — which is `false` when `ft_transfer` was used.

In `fin_transfer_send_tokens_callback`, the revert path is gated on `is_refund_required`:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
            Ok(value) => { ... amount.0 == 0 }
            Err(_) => false,   // ← also does NOT revert on ft_transfer_call failure
        }
    } else {
        false   // ← NEVER reverts when ft_transfer was used
    }
}
```

When `is_ft_transfer_call = false`, the function returns `false` unconditionally — **the promise result of `ft_transfer` is never inspected**. If `ft_transfer` panicked (token paused, recipient blacklisted, etc.), the callback still takes the success branch, emits `FinTransferEvent`, and sends the fee. The unlock action is never reverted, and the transfer ID can never be retried. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds.**

- The user's tokens on the origin chain (EVM) are already locked/burned — the `InitTransfer` event was emitted and the NEAR bridge accepted the proof.
- The NEAR bridge's `finalised_transfers` set permanently records the transfer ID; no retry is possible.
- The `locked_tokens` counter is decremented even though the tokens were never released, corrupting bridge collateralization accounting.
- The tokens remain in the NEAR bridge contract with no admin function to recover them (no `remove_fin_transfer` path is exposed for already-finalized transfers).
- The `FinTransferEvent` is emitted, giving a false signal of success to off-chain monitors. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

**Moderate.** USDC (Circle) has a documented pause mechanism and has exercised it in production. Any user bridging a pausable native token (USDC, USDT, etc.) from EVM to NEAR is at risk during any pause window. The window is the gap between when `fin_transfer_callback` runs (marking the transfer finalized and unlocking tokens) and when `ft_transfer` completes. Because NEAR cross-contract calls are asynchronous, these are separate receipts — a pause that activates between them triggers the bug. No attacker action is required; the user is simply a victim of the token pause.

---

### Recommendation

In `fin_transfer_send_tokens_callback`, check the promise result even when `is_ft_transfer_call = false`. If the `ft_transfer` promise failed, treat it identically to the refund path: revert the lock actions, remove the finalized transfer record, and emit `FailedFinTransferEvent`.

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
            Ok(value) => {
                near_sdk::serde_json::from_slice::<U128>(&value)
                    .map_or(false, |a| a.0 == 0)
            }
            Err(_) => true,  // ft_transfer_call itself failed → revert
        }
    } else {
        // ft_transfer: check whether the promise succeeded
        env::promise_result_checked(0, 0).is_err()
    }
}
```

Additionally, `add_fin_transfer` should be deferred until after the token transfer succeeds (or the finalization should be made retryable), so that a failed transfer does not permanently consume the nonce.

---

### Proof of Concept

**Scenario:** User bridges 1000 USDC from Ethereum to NEAR. USDC is a native token on NEAR (locked on EVM, held by the NEAR bridge).

1. User calls `initTransfer` on EVM `OmniBridge`, locking 1000 USDC. `InitTransfer` event emitted.
2. Relayer submits proof to NEAR `fin_transfer`. `fin_transfer_callback` runs:
   - `add_fin_transfer` inserts transfer ID into `finalised_transfers`. *(irreversible)*
   - `unlock_tokens_if_needed` decrements `locked_tokens[Eth][usdc.near]` by 1000. *(accounting corrupted)*
   - `send_tokens` dispatches `ft_transfer(recipient, 1000)` to `usdc.near`.
3. Between step 2 and the `ft_transfer` receipt executing, Circle pauses USDC on NEAR.
4. `ft_transfer` panics. NEAR runtime calls `fin_transfer_send_tokens_callback` with a failed promise.
5. `is_refund_required(false)` returns `false`. The `else` branch runs: fee is sent, `FinTransferEvent` is emitted.
6. **Result:** 1000 USDC sits in the NEAR bridge contract. `finalised_transfers` blocks any retry. `locked_tokens` is wrong. User has lost 1000 USDC permanently. [7](#0-6) [3](#0-2)
