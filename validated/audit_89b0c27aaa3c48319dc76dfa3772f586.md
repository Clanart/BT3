### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts an arbitrary user-supplied `class_hash` without verifying that the hash corresponds to a previously declared class. A malicious or buggy contract can call `replace_class` with an undeclared class hash, permanently corrupting its own on-chain class pointer. Any subsequent call to that contract will cause the OS to fail when looking up the compiled class, permanently freezing the contract and all funds it holds.

---

### Finding Description

In `execute_replace_class`, the OS reads the caller-supplied `class_hash` directly from the syscall request and writes it into `contract_state_changes` without any validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The developer-left TODO comment at line 898 explicitly acknowledges the missing check. [1](#0-0) 

The OS enforces no constraint that `class_hash` must exist in `contract_class_changes` (the declared-class dictionary). Compare this to `execute_declare_transaction` in `transaction_impls.cairo`, which explicitly enforces `prev_value=0` and `assert_not_zero(compiled_class_hash)` before writing to `contract_class_changes`. [2](#0-1) 

When any future transaction calls the frozen contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // the undeclared hash
);
// compiled_class_hash == 0 (uninitialized default)

let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,   // key == 0, not present
);
```

`find_element` is a Cairo assertion-based lookup; if the key is absent, it causes an irrecoverable Cairo assertion failure, making it impossible to generate a valid proof for any block that includes a call to the frozen contract. [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is overwritten with an undeclared hash and the state is committed on L1 (via the OS proof), the contract becomes permanently uncallable. Any ERC-20 tokens, ETH, or other assets held in that contract's storage are irrecoverably frozen. There is no upgrade path because `replace_class` itself requires a callable contract to execute.

---

### Likelihood Explanation

**Medium.**

The attacker must:
1. Deploy a contract (permissionless on StarkNet).
2. Have that contract call `replace_class` with an arbitrary felt value that is not a declared class hash.

No privileged role is required. The syscall is available to any deployed contract. A malicious contract deployer can intentionally target their own contract (e.g., a token contract holding user funds) or any contract they control. The sequencer's blockifier may have a separate runtime check, but the OS Cairo code — which governs what is provable and therefore what can be committed to L1 — contains no such enforcement, as confirmed by the TODO comment.

---

### Recommendation

Remove the TODO and enforce the validation inline in `execute_replace_class`. Before writing the new `StateEntry`, assert that `class_hash` is present in `contract_class_changes` (i.e., it has been declared):

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the pattern already used in `execute_declare_transaction` and in `execute_entry_point` itself, and closes the gap between sequencer-level validation and OS-level proof enforcement.

---

### Proof of Concept

1. Attacker deploys `MaliciousToken` — a standard ERC-20 contract that also exposes a `freeze()` entry point.
2. Users deposit funds into `MaliciousToken` (it behaves normally until triggered).
3. Attacker calls `freeze()`, which internally calls `replace_class(0xdeadbeef)` — an arbitrary undeclared felt.
4. `execute_replace_class` writes `class_hash=0xdeadbeef` into `contract_state_changes` with no validation. [1](#0-0) 
5. The OS generates a valid proof for this block; the state is committed on L1.
6. Any subsequent `transfer` or `balanceOf` call to `MaliciousToken` causes `execute_entry_point` to call `dict_read` on `0xdeadbeef` (returning 0), then `find_element` with key=0, which fails with a Cairo assertion. [3](#0-2) 
7. No valid proof can be generated for any block containing a call to `MaliciousToken`. All user funds are permanently frozen.
