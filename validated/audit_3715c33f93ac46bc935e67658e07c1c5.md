### Title
P2P Sync Server Computes Wrong Block Number Due to u64 Overflow Before i128 Widening in `calculate_block_number` — (File: `crates/apollo_p2p_sync/src/server/utils.rs`)

---

### Summary

`calculate_block_number` in the P2P sync server is intended to use `i128` arithmetic to safely handle large block-number deltas. However, the multiplication `query.step * read_blocks_counter` is performed entirely in `u64` space **before** the cast to `i128`, so the protective widening is applied too late. In release mode this wraps silently; in debug mode it panics. A remote peer can craft a query that causes the server to compute the wrong block number and serve data for the wrong block — an exact structural analog of M-38's "protective bound applied after the overflowing operation."

---

### Finding Description

The vulnerable function is:

```rust
// crates/apollo_p2p_sync/src/server/utils.rs
pub(crate) fn calculate_block_number(
    query: &Query,
    start_block: u64,
    read_blocks_counter: u64,
) -> Result<u64, P2pSyncServerError> {
    let direction_factor: i128 = match query.direction {
        Direction::Forward => 1,
        Direction::Backward => -1,
    };
    // TODO(shahak): Fix this code.
    let blocks_delta: i128 = direction_factor * i128::from(query.step * read_blocks_counter);
    //                                                      ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
    //                                                      u64 * u64 — overflows BEFORE cast
    let block_number: i128 = i128::from(start_block) + blocks_delta;

    u64::try_from(block_number).map_err(|_| P2pSyncServerError::BlockNumberOutOfRange { ... })
}
``` [1](#0-0) 

The expression `query.step * read_blocks_counter` is a `u64 × u64` multiplication. When the product exceeds `u64::MAX`, Rust wraps in release mode (panics in debug mode). Only the already-corrupted `u64` result is then widened to `i128` — the widening that was supposed to prevent overflow is applied after the overflow has already occurred.

The function is called in the hot path of every inbound P2P sync query (headers, state diffs, transactions, classes, events):

### Impact Explanation

**Release mode (wrapping overflow):** The wrapped `u64` value is cast to `i128` and used as the block delta. The server fetches and streams data for the **wrong block number** to the querying peer. Concretely, a block that was already served (e.g., `start_block`) can be served again, or an entirely different block is served in place of the expected one. The receiving peer gets an authoritative-looking response containing data for the wrong block.

**Debug mode (overflow panic):** The server task panics, terminating the sync session for that peer.

Both outcomes match the allowed impact: *"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value"* — the P2P sync server is the authoritative data source for state sync; serving the wrong block header, state diff, or transaction set is a wrong authoritative value in the sync path.

---

### Likelihood Explanation

Any unauthenticated peer on the P2P network can send a crafted `Iteration` query. No special privilege is required. The `step` field accepts the full `uint64` range. The overflow is deterministic and reproducible with a fixed input.

---

### Recommendation

Cast both operands to `i128` **before** multiplying, so the widening precedes the potentially overflowing operation:

```rust
// Correct: widening happens before multiplication
let blocks_delta: i128 =
    direction_factor * i128::from(query.step) * i128::from(read_blocks_counter);
```

This mirrors the fix applied to `boundedSub()` in M-38: move the protective operation to before the dangerous arithmetic, not after it.

---

### Proof of Concept

Send a backward query with:
- `direction = Backward`
- `start_block = u64::MAX / 2 + 1` (= `9_223_372_036_854_775_808`)
- `step = u64::MAX / 2 + 1`
- `limit = 3`

Trace through `calculate_block_number` in release mode:

| `block_counter` | `step * counter` (u64) | wrapped value | `block_number` | Expected? |
|---|---|---|---|---|
| 0 | 0 | 0 | `start_block` | ✓ correct |
| 1 | `u64::MAX/2+1` | `u64::MAX/2+1` | `start_block − (u64::MAX/2+1) = 0` | ✓ correct |
| 2 | `(u64::MAX/2+1)×2 = u64::MAX+1` | **wraps to 0** | `start_block − 0 = start_block` | ✗ **wrong — block `start_block` served again** |

At `counter = 2`, the server re-serves `start_block`'s data instead of returning `BlockNumberOutOfRange`. The peer receives a duplicate block response for an unexpected block number — an authoritative-looking wrong value from the sync server.
