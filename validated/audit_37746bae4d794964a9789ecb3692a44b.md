### Title
Missing Destination Chain ID in `deployToken` Signed Hash Enables Cross-Chain Replay of Token Deployment Signatures — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`)

---

### Summary

The `deployToken` function in `OmniBridge.sol` and the `deploy_token` function in the Starknet bridge construct their signed message hash without including the destination chain identifier (`omniBridgeChainId`). Because the same NEAR MPC-derived key (`nearBridgeDerivedAddress`) signs for all supported EVM chains, a valid `deployToken` signature produced for one EVM chain (e.g., Ethereum) is cryptographically identical and accepted on every other EVM chain (Arbitrum, Base, BNB, Polygon, HyperEVM, Abstract). In contrast, `finTransfer` on the same contract correctly binds its hash to the destination chain. An unprivileged attacker can capture any on-chain `deployToken` signature and replay it on a different EVM chain, permanently occupying the `nearToEthToken` slot for that token ID on the target chain and blocking all future legitimate deployments there.

---

### Finding Description

**Root cause — missing chain ID in `deployToken` hash:**

`OmniBridge.sol` `deployToken` (lines 142–149) hashes only:

```
PayloadType::Metadata | token | name | symbol | decimals
``` [1](#0-0) 

No `omniBridgeChainId` byte is included. Compare this to `finTransfer` (lines 289–309), which embeds `omniBridgeChainId` **twice** — once for the token address field and once for the recipient field — explicitly binding the signed payload to the destination chain: [2](#0-1) 

The same omission exists in the Starknet bridge. `MetadataPayloadTrait::to_borsh()` (bridge_types.cairo lines 36–44) encodes only `PayloadType | token | name | symbol | decimals` with no chain ID, while `TransferMessagePayloadTrait::to_borsh()` (lines 61–84) correctly accepts and embeds `chain_id`: [3](#0-2) [4](#0-3) 

The Starknet `deploy_token` passes `payload.to_borsh()` (no chain ID argument) to `_verify_borsh_signature`: [5](#0-4) 

**Why the same key is used across chains:**

Each EVM bridge contract stores a single `nearBridgeDerivedAddress` — the Ethereum address recovered from NEAR MPC signatures: [6](#0-5) 

The NEAR MPC network signs with a single key pair per derivation path. All EVM-chain bridge deployments share the same `nearBridgeDerivedAddress`. The `omniBridgeChainId` field exists precisely to distinguish chains (values 0 = Eth, 3 = Arb, 4 = Base, 5 = BNB, 8 = Pol, 9 = HyperEvm, 11 = Abs per `ChainKind`): [7](#0-6) 

Because `deployToken`'s hash omits this byte, the signature is chain-agnostic.

**Attack flow:**

1. NEAR MPC signs a `MetadataPayload` for token `T` to be deployed on Ethereum (`omniBridgeChainId = 0`). The signature is submitted on-chain (publicly visible in the Ethereum transaction).
2. Attacker copies the raw `signatureData` and the `MetadataPayload` calldata.
3. Attacker calls `deployToken(signatureData, metadata)` on the Arbitrum `OmniBridge` contract before any legitimate deployment occurs there.
4. `ECDSA.recover(keccak256(borshEncoded), signatureData)` returns the same `nearBridgeDerivedAddress` because the hash is identical — no chain byte differentiates it.
5. The check `!isBridgeToken[nearToEthToken[metadata.token]]` passes (token not yet deployed on Arbitrum).
6. A `BridgeToken` proxy is deployed on Arbitrum; `nearToEthToken[metadata.token]` is permanently set.
7. Any subsequent legitimate `deployToken` call for the same `metadata.token` on Arbitrum reverts with `"ERR_TOKEN_EXIST"`. [8](#0-7) 

---

### Impact Explanation

**Permanent freezing of token deployment on target chains.** Once the attacker occupies `nearToEthToken[metadata.token]` on a target chain, no legitimate deployment can ever succeed for that token on that chain. The `"ERR_TOKEN_EXIST"` guard has no admin override path in `deployToken`. The only remediation would require a contract upgrade. Any token that has been legitimately deployed on one EVM chain is immediately replayable to all other EVM chains, permanently blocking those chains from ever receiving the correct bridge-token deployment. This prevents users from bridging that token to the affected chain — an irrecoverable lock of the token's cross-chain availability on that chain.

This matches the allowed impact: **Permanent freezing / irrecoverable lock of user or protocol funds in bridge or token flows.**

---

### Likelihood Explanation

- **No privilege required.** Any observer of on-chain transactions can extract a `deployToken` calldata and replay it.
- **Timing window is large.** The attacker does not need to front-run the original chain's deployment; they only need to act before the legitimate deployment on the *target* chain, which may happen days or weeks later.
- **Seven EVM chains are in scope** (Eth, Arb, Base, BNB, Pol, HyperEvm, Abs). Every `deployToken` signature for any one of them is simultaneously valid on all six others.
- **Economically motivated.** A competitor or griefing attacker can permanently block a token from being bridged to a specific chain at near-zero cost (only gas on the target chain).

---

### Recommendation

Include `omniBridgeChainId` in the `deployToken` Borsh-encoded hash, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to destination chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the same fix to `MetadataPayloadTrait::to_borsh()` in `starknet/src/bridge_types.cairo` — add `chain_id: u8` as a parameter and prepend it (or append it in a consistent position) to the encoded bytes, matching the pattern in `TransferMessagePayloadTrait::to_borsh()`.

The NEAR MPC signing layer must also be updated to include the destination chain ID when signing `MetadataPayload` messages, so that signatures are chain-specific.

### Proof of Concept

```python
# Attacker observes a deployToken tx on Ethereum (chain 0) and replays on Arbitrum (chain 3)

from web3 import Web3

ETH_RPC   = "https://mainnet.infura.io/v3/..."
ARB_RPC   = "https://arb1.arbitrum.io/rpc"

w3_eth = Web3(Web3.HTTPProvider(ETH_RPC))
w3_arb = Web3(Web3.HTTPProvider(ARB_RPC))

# Step 1: Fetch the original deployToken tx from Ethereum
tx = w3_eth.eth.get_transaction("0x<legitimate_deploy_token_tx_hash>")

# Step 2: Decode calldata — extract signatureData and MetadataPayload
# (ABI-decode using OmniBridge ABI)
decoded = eth_bridge_contract.decode_function_input(tx.input)
sig_data = decoded[1]['signatureData']
metadata  = decoded[1]['metadata']

# Step 3: Replay on Arbitrum OmniBridge — same sig, same metadata, different chain
arb_bridge = w3_arb.eth.contract(address=ARB_BRIDGE_ADDR, abi=OMNI_BRIDGE_ABI)
tx_hash = arb_bridge.functions.deployToken(sig_data, metadata).transact({'from': attacker})

# Result: token deployed on Arbitrum using Ethereum's signature.
# nearToEthToken[metadata.token] is now permanently set on Arbitrum.
# Any future legitimate deployToken for this token on Arbitrum reverts with ERR_TOKEN_EXIST.
```
