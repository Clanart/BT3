### Title
Integer Wrapping in Relative Timelock Arithmetic Bypasses `ASSERT_SECONDS_RELATIVE` / `ASSERT_HEIGHT_RELATIVE` Validation — (`File: crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` performs relative timelock arithmetic using `wrapping_add` when the `nowrap` flag is `false`. An attacker who crafts a coin spend with a near-maximum `ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE` value causes the addition `coin_timestamp + seconds_relative` (or `confirmed_block_index + height_relative`) to silently wrap around the u64/u32 boundary, producing a small result. The timelock comparison then passes immediately, allowing the coin to be spent far earlier than the puzzle author intended — a direct timelock validation bypass.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` implements two arithmetic paths for relative timelock conditions, selected by the `nowrap: bool` parameter:

- `nowrap = true` → `saturating_add` (correct: large values clamp to `u64::MAX` / `u32::MAX`, timelock always enforced)
- `nowrap = false` → `wrapping_add` (incorrect: large values silently wrap to small values, timelock bypassed)

The four affected checks are:

**`ASSERT_SECONDS_RELATIVE` (lines 70–78):**
```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [1](#0-0) 

**`ASSERT_HEIGHT_RELATIVE` (lines 55–68):**
```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
``` [2](#0-1) 

**`ASSERT_BEFORE_HEIGHT_RELATIVE` (lines 79–98) and `ASSERT_BEFORE_SECONDS_RELATIVE` (lines 100–112)** are also affected, but in the opposite direction: wrapping causes a valid spend to be incorrectly rejected. [3](#0-2) 

The Rust unit tests and the Python integration tests both explicitly document and confirm this divergence:

```
// 2000 < 1000 + u64::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::seconds_relative_wrap(
    Osc { seconds_relative: Some(0xffff_ffff_ffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed)),
    Ok(()),
)]
``` [4](#0-3) 

The Python test confirms the same for the exported binding:
```python
# 10000 + u64::MAX overflows to 9999, wrapping: 10150 < 9999 -> Ok
(make_test_conds(seconds_relative=0xFFFF_FFFF_FFFF_FFFF), 105, None),
``` [5](#0-4) 

The `nowrap` parameter is a caller-controlled `bool` exposed directly through the Python binding: [6](#0-5) 

The Python full node passes this flag based on whether a soft-fork / hard-fork rule is active. Any block or mempool validation path that passes `nowrap=False` is vulnerable.

---

### Impact Explanation

**Impact: High — Timelock validation bypass enables unauthorized spend acceptance.**

When `nowrap=False` is in effect (legacy/pre-hard-fork validation path), a coin spend carrying `ASSERT_SECONDS_RELATIVE(u64::MAX)` or `ASSERT_HEIGHT_RELATIVE(u32::MAX)` passes `check_time_locks` immediately, regardless of the actual elapsed time or block height. The intended timelock — which could be set to an astronomically large value to make a coin permanently unspendable, or to enforce a long lock-up period — is silently bypassed. The coin is accepted as validly spent when it should be rejected.

Concretely:
- `coin_timestamp = T`, `seconds_relative = u64::MAX - T + 1`
- `wrapping_add`: `T + (u64::MAX - T + 1) = 0` (wraps)
- Check: `current_timestamp < 0` → `false` → **no error, spend accepted**

This matches the "High" impact category: *timelock validation bypass enables unauthorized spend acceptance*.

---

### Likelihood Explanation

The attacker must:
1. Create a coin whose puzzle emits `ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE` with a carefully chosen near-maximum value.
2. Submit the spend during a block validation window where the full node calls `check_time_locks` with `nowrap=False`.

Step 1 requires only crafting a valid CLVM puzzle — fully within unprivileged attacker capability. Step 2 depends on whether the full node ever passes `nowrap=False` for live block validation. The fact that the parameter is exposed and both paths are tested in production code strongly suggests `nowrap=False` is used for some validation contexts (e.g., pre-hard-fork blocks or mempool pre-validation). Likelihood is **Medium** — exploitable whenever the legacy path is active.

---

### Recommendation

Replace all four `wrapping_add` calls in `check_time_locks` with `saturating_add`, making the safe behavior unconditional and eliminating the `nowrap` branch entirely:

```rust
// Before (vulnerable):
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {

// After (safe):
if timestamp < unspent.timestamp.saturating_add(seconds_relative) {
```

If backward-compatible legacy behavior is genuinely required for historical block replay, add an explicit upper-bound check on the relative value before performing the addition (analogous to the external report's recommendation to check `newItemId < LAZY_MINT_TOKENID_START`):

```rust
// Reject any relative value that would overflow the type boundary
if seconds_relative > u64::MAX - unspent.timestamp {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
```

### Proof of Concept

Using the exported Python binding:

```python
from chia_rs import check_time_locks, SpendBundleConditions, SpendConditions, CoinRecord, Coin
from chia_rs.sized_ints import uint32, uint64
from chia_rs.sized_bytes import bytes32

# Coin confirmed at timestamp 10000
coin_timestamp = 10000
coin = Coin(bytes32(b'\x00'*32), bytes32(b'\x00'*32), uint64(1))
coin_id = coin.name()
record = CoinRecord(coin, uint32(0), uint32(0), False, uint64(coin_timestamp))

# Craft seconds_relative so that wrapping_add produces 0:
# coin_timestamp + seconds_relative wraps to 0
seconds_relative = (2**64 - coin_timestamp) % 2**64  # = u64::MAX - 9999

conds = SpendBundleConditions([
    SpendConditions(coin_id, ..., seconds_relative=uint64(seconds_relative), ...)
], ...)

# With nowrap=True (correct): returns error code 105 (AssertSecondsRelativeFailed)
result_safe = check_time_locks({coin_id: record}, conds, uint32(15), uint64(10150), True)
assert result_safe == 105  # correctly blocked

# With nowrap=False (vulnerable): returns None (spend accepted!)
result_vuln = check_time_locks({coin_id: record}, conds, uint32(15), uint64(10150), False)
assert result_vuln is None  # timelock bypassed
```
