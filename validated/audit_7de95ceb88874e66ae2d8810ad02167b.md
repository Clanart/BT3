Audit Report

## Title
Missing Payer-Attribution Check in `pay()` Allows Theft of Router-Held Native ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` uses `address(this).balance` — the router's total native-ETH balance — to fund WETH deposits without verifying that the ETH was contributed by the current payer. Any ETH stranded on the router from a prior user's `msg.value` overshoot (i.e., no `refundETH()` call) is fully consumable by a subsequent unprivileged caller who submits `exactInputSingle` with `tokenIn = WETH` and zero `msg.value`, stealing the prior user's ETH in exchange for swap output tokens.

## Finding Description
`pay()` in `PeripheryPayments.sol` handles the WETH branch as follows: [1](#0-0) 

When `token == WETH` and `payer != address(this)`, the function reads `address(this).balance` — the router's **total** native-ETH balance — and uses it to fund the WETH deposit before pulling any shortfall from the payer via `safeTransferFrom`. There is no check that this native ETH was deposited by the current payer or the current transaction.

ETH accumulates on the router whenever a user calls `exactInputSingle{value: X}` with `amountIn < X` and omits a `refundETH()` call. The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks plain ETH transfers (no calldata); it does **not** prevent ETH from being received via `msg.value` in payable function calls. [2](#0-1) 

The exploit path:
1. `exactInputSingle` sets `payer = msg.sender` and `tokenToPay = params.tokenIn` in transient storage, then calls the pool. [3](#0-2) 

2. The pool calls `metricOmmSwapCallback`, which dispatches to `_justPayCallback`. [4](#0-3) 

3. `_justPayCallback` calls `pay(WETH, attacker, pool, amountIn)`. Since `address(this).balance >= amountIn` (from the victim's stranded ETH), the router deposits the victim's ETH as WETH and transfers it to the pool — the attacker receives the swap output with no ETH or WETH approval required.

No guard in `exactInputSingle`, `metricOmmSwapCallback`, or `_justPayCallback` checks whether `msg.value >= amountIn` or whether the router's native balance is attributable to the current caller.

## Impact Explanation
Direct loss of user principal. Any ETH stranded on the router is fully claimable by the next caller who submits `exactInputSingle` (or `exactInput`) with `tokenIn = WETH` and zero `msg.value`. The attacker receives the full swap output; the victim's ETH is consumed by the pool as WETH payment. Loss is bounded only by the router's instantaneous ETH balance, which can equal the victim's entire `msg.value`. This is a Critical/High direct loss of user principal meeting Sherlock thresholds.

## Likelihood Explanation
High. The residue condition is created by any WETH-input swap where the user does not append `refundETH()` — a common omission when calling `exactInputSingle` directly rather than through a multicall bundle. The exploit requires no special role, no flash loan, and no prior state setup. MEV bots can detect the router's ETH balance on-chain and front-run the victim's `refundETH()`. The test suite explicitly demonstrates the residue pattern and the need for `refundETH()` to clear it.

## Recommendation
Add a payer-attribution guard in the WETH branch of `pay()`. Track the "current-call ETH budget" (i.e., `msg.value` of the top-level entry-point call) in transient storage at `multicall`/`exactInputSingle` entry, deduct from it inside `pay()`, and revert if the budget is exhausted. Alternatively, require `msg.value >= amountIn` at the `exactInputSingle` level when `tokenIn == WETH` and the payer is external, and pull any shortfall exclusively via `safeTransferFrom`. This ensures only ETH explicitly contributed by the current transaction can be used to fund a WETH deposit.

## Proof of Concept
```
1. Victim calls exactInputSingle{value: 1 ETH}(
       pool=WETH/TOKEN1, tokenIn=WETH, amountIn=0.5 ETH, recipient=victim, ...
   )
   — pay() deposits 0.5 ETH as WETH, transfers to pool.
   — Router retains 0.5 ETH (victim omitted refundETH()).

2. Attacker (no ETH, no WETH approval) calls exactInputSingle{value: 0}(
       pool=WETH/TOKEN1, tokenIn=WETH, amountIn=0.5 ETH, recipient=attacker, ...
   )
   — _setNextCallbackContext sets payer=attacker, tokenToPay=WETH.
   — Pool executes swap, calls metricOmmSwapCallback.
   — _justPayCallback calls pay(WETH, attacker, pool, 0.5 ETH).
   — address(this).balance == 0.5 ETH >= 0.5 ETH → branch taken.
   — Router deposits victim's 0.5 ETH as WETH, transfers to pool.
   — Attacker receives TOKEN1 output; victim's 0.5 ETH is gone.
```
