### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Causes Bridge Undercollateralization — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.sol::initTransfer` locks ERC20 tokens by calling `safeTransferFrom(msg.sender, address(this), amount)` and then emits an `InitTransfer` event carrying the caller-supplied `amount`. For fee-on-transfer ERC20 tokens the contract actually receives `amount − transfer_fee`, but the event records the full `amount`. The NEAR bridge processes the event and releases/mints the full `amount` on the destination chain. The EVM vault is therefore systematically undercollateralized, and when users bridge back the last withdrawals will fail because the contract cannot satisfy `safeTransfer(recipient, payload.amount)`.

---

### Finding Description

In `OmniBridge.sol::initTransfer`, the native-ERC20 lock path is:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // user-supplied, not balance-delta
    );
}
```

followed immediately by:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,             // same user-supplied value, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
``` [1](#0-0) 

The contract never measures the balance before and after the transfer. For a fee-on-transfer token (e.g., a deflationary ERC20 that deducts 1 % on every transfer), the vault receives `amount * 0.99` but the event asserts `amount`. The NEAR bridge reads the event and releases the full `amount` on the destination chain.

On the return path, `finTransfer` on the EVM side executes:

```solidity
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,
        payload.amount   // full amount from the signed payload
    );
}
``` [2](#0-1) 

Because the vault holds less than the sum of all recorded `amount` values, the last users to withdraw will receive a revert from `safeTransfer`, permanently locking their funds.

---

### Impact Explanation

**High — Balance/accounting corruption that breaks bridge collateralization.**

Every `initTransfer` call with a fee-on-transfer token widens the gap between the vault's actual token balance and the total obligations recorded in cross-chain events. After N such transfers the vault is short by `N × transfer_fee`. The final users who attempt to bridge back to EVM cannot withdraw; their tokens are irrecoverably locked in the bridge.

---

### Likelihood Explanation

**Medium.** The "else" branch in `initTransfer` is reached for any ERC20 that is neither a `BridgeToken` nor has a `customMinter` — i.e., any natively-locked token. No privileged access is required; any user can call `initTransfer`. Fee-on-transfer tokens are a well-known ERC20 variant (deflationary tokens, tokens with protocol fees). A single user bridging such a token begins the undercollateralization; repeated use compounds it silently until withdrawals start failing.

---

### Recommendation

Measure the actual received amount using a balance-before / balance-after pattern and use that delta in the event:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived > 0, "Zero received");
// use actualReceived (cast to uint128) in the event and downstream logic
```

Alternatively, explicitly disallow fee-on-transfer tokens via a whitelist or a pre-transfer balance check that reverts if `actualReceived != amount`.

---

### Proof of Concept

1. Deploy or use any ERC20 token that deducts a 1 % fee on every `transfer`/`transferFrom`.
2. Register the token with the Omni Bridge (NEAR ↔ EVM mapping).
3. Alice calls `OmniBridge.initTransfer(tokenAddress, 1000, 0, 0, "alice.near", "")`.
   - `safeTransferFrom` moves 1000 tokens from Alice; the token contract deducts 1 %, so the vault receives **990**.
   - The event emits `amount = 1000`.
4. The NEAR bridge processes the event and releases **1000** tokens to `alice.near`.
5. Repeat steps 3–4 for 100 users. Vault holds `100 × 990 = 99 000` tokens; total obligations = `100 × 1000 = 100 000`.
6. The first 99 users bridge back to EVM successfully (each `finTransfer` calls `safeTransfer(recipient, 1000)`; vault drains to ~0 after ~99 withdrawals).
7. The 100th user's `finTransfer` reverts because the vault balance is insufficient — their **1000 tokens are permanently locked**.
