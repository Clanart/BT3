### Title
uint64 Underflow in Nonce Reconciliation After Failed `evm.Create()` Corrupts Account Nonce — (`File: x/evm/keeper/state_transition.go`)

### Summary

`ApplyMessageWithConfig` performs a nonce reset-and-reconcile dance around `evm.Create()` to compensate for Ethermint's ante handler pre-incrementing nonces for all messages in a batch. The reconciliation arithmetic at line 479 is an unchecked `uint64` subtraction that underflows when `evm.Create()` returns without incrementing the nonce (e.g., balance check failure). The underflowed value is then added to `oldNonce`, wrapping the account nonce to `oldNonce − 1` and committing that corrupted value to the Cosmos store.

### Finding Description

In `ApplyMessageWithConfig`, the contract-creation branch reads the current (ante-incremented) nonce, resets it to `msg.Nonce`, calls `evm.Create()`, and then reconciles:

```go
// x/evm/keeper/state_transition.go  lines 468-482
if contractCreation {
    oldNonce := stateDB.GetNonce(sender)          // e.g. N+2 (ante pre-incremented)
    stateDB.SetNonce(sender, msg.Nonce, ...)       // reset to N+1
    ret, _, leftoverGas, vmErr = evm.Create(...)
    afterCreateNonce := stateDB.GetNonce(sender)   // still N+1 if Create() bailed early
    nestedCreates := afterCreateNonce - msg.Nonce - 1  // (N+1) - (N+1) - 1 → uint64 underflow → MaxUint64
    stateDB.SetNonce(sender, oldNonce+nestedCreates, ...) // (N+2) + MaxUint64 → N+1 (wraps)
}
```

go-ethereum's `create()` increments the nonce **after** two early-exit checks:

1. Call-depth limit (`evm.depth > 1024`) — returns `ErrDepth` without touching the nonce.
2. Balance check (`CanTransfer`) — returns `ErrInsufficientBalance` without touching the nonce.

When either fires, `afterCreateNonce` equals the value just written (`msg.Nonce`), so `afterCreateNonce - msg.Nonce - 1` is `0 - 1` in `uint64` arithmetic, yielding `18446744073709551615` (`MaxUint64`). Adding that to `oldNonce` wraps to `oldNonce − 1`.

**Reachable trigger — batched transaction with balance depletion:**

Ethermint's ante handler (`CheckAndSetEthSenderNonce`) increments the Cosmos account sequence for **every** message in a batch upfront, before any EVM execution. `CheckEthCanTransfer` checks each message's balance against the **pre-execution** state independently, so it cannot detect that an earlier message in the same batch will drain the balance.

Concrete path:
1. Sender has balance B. Batch = [msg0: CALL transferring B−ε, msg1: CREATE with value ε+1].
2. Ante handler: sequence N → N+2; `canTransfer` passes for both (balance B ≥ ε+1 at check time).
3. msg0 executes, balance drops to ε−1 < ε+1.
4. msg1 executes: `oldNonce = N+2`, reset to `N+1`, `evm.Create()` hits `ErrInsufficientBalance` and returns without incrementing the nonce.
5. `afterCreateNonce = N+1`, `nestedCreates = MaxUint64`, committed nonce = `N+1`.

The stateDB commit writes `N+1` back to the Cosmos account keeper, overwriting the ante-handler's `N+2`. The account's authoritative sequence is now `N+1` instead of `N+2`. [1](#0-0) [2](#0-1) 

### Impact Explanation

The committed nonce `N+1` is lower than the correct post-batch value `N+2`. Consequences:

- **Nonce reuse**: The account's sequence is `N+1`; the ante handler will accept a new transaction with nonce `N+1`, which was already "consumed" by the batch. This breaks the replay-protection invariant.
- **Contract address mis-derivation**: Any contract whose address was pre-computed using nonce `N+2` (e.g., by a factory or a user who pre-funded the expected address) will not be deployed there; funds sent to the pre-computed address are permanently lost or claimable by whoever deploys first at that address.
- **Persistent state corruption**: The corrupted nonce is committed to the Cosmos store and persists across blocks, affecting all future transactions from the account.

This matches the allowed High impact: *"EVM state transition… bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."* [3](#0-2) 

### Likelihood Explanation

- Requires batched `MsgEthereumTx` (Cosmos-specific; ADR-003 discourages but does not prohibit it).
- Requires a CREATE message in the batch whose value exceeds the balance remaining after earlier messages execute.
- The ante handler's per-message balance check does not account for intra-batch balance changes, making this a realistic user mistake or deliberate self-trigger.
- No privileged role, validator cooperation, or chain reorg is needed; any unprivileged account can submit such a batch. [4](#0-3) [5](#0-4) 

### Recommendation

Guard the subtraction against underflow before computing `nestedCreates`:

```go
afterCreateNonce := stateDB.GetNonce(sender)
// evm.Create() may return without incrementing the nonce (depth/balance early exit).
// Guard against underflow: if the nonce was not incremented, treat nestedCreates as 0.
var nestedCreates uint64
if afterCreateNonce > msg.Nonce {
    nestedCreates = afterCreateNonce - msg.Nonce - 1
}
stateDB.SetNonce(sender, oldNonce+nestedCreates, tracing.NonceChangeUnspecified)
```

This ensures that a failed `evm.Create()` (nonce not incremented) leaves the account nonce at `oldNonce` — the correct ante-handler value — rather than wrapping it to `oldNonce − 1`. [6](#0-5)

		if rules.IsLondon {

				)
			}
			if tx.GasFeeCap().Cmp(baseFee) < 0 {
				return errorsmod.Wrapf(

					"max fee per gas less than block base fee (%s < %s)",
					tx.GasFeeCap(), baseFee,
				)
			}
		}

				value,
				from,
			)
		}
	}

// CheckAndSetEthSenderNonce handles incrementing the sequence of the signer (i.e sender). If the transaction is a
// contract creation, the nonce will be incremented during the transaction execution and not within
// this AnteHandler decorator.
func CheckAndSetEthSenderNonce(
	ctx sdk.Context, tx sdk.Tx, ak evmtypes.AccountKeeper, unsafeUnOrderedTx bool, accountGetter AccountGetter, nonceCache *cache.AnteCache,
) ([]cache.TxNonce, error) {
	var pending []cache.TxNonce

	for _, msg := range tx.GetMsgs() {
		msgEthTx, ok := msg.(*evmtypes.MsgEthereumTx)
		if !ok {
			return nil, errorsmod.Wrapf(errortypes.ErrUnknownRequest, "invalid message type %T, expected %T", msg, (*evmtypes.MsgEthereumTx)(nil))
		}

		tx := msgEthTx.AsTransaction()

		from := msgEthTx.GetFrom()
		acc := accountGetter(from)
		if acc == nil {
			return nil, errorsmod.Wrapf(
				errortypes.ErrUnknownAddress,
				"account %s is nil", common.BytesToAddress(from.Bytes()),
			)
		}
		expectedNonce := acc.GetSequence()
		txNonce := tx.Nonce()
		fromStr := from.String()

		// if flag is set, we bypass nonce all check verification
		if !unsafeUnOrderedTx {
			ex := nonceCache.Exists(fromStr, txNonce)
			// to support tx replacement, we check if the transaction nonce exists in the cache and if yes we skip
			// nonce verification, and we don't set the sequence
			// We allow skip verification only during CheckTx to keep sequence safe during the execution.
			if ctx.IsCheckTx() && !ctx.IsReCheckTx() && ex {
				continue
			}

			// nonce verification, the sequence needs to be in order
			if txNonce != expectedNonce {
				// delete in case of recheck tx
				if ex {
					nonceCache.Delete(fromStr, txNonce)
				}
				return nil, errorsmod.Wrapf(
					errortypes.ErrInvalidSequence,
					"invalid nonce; got %d, expected %d", txNonce, expectedNonce,
				)
			}

			if ctx.IsCheckTx() {
				if !ctx.IsReCheckTx() {
					pending = append(pending, cache.TxNonce{Address: fromStr, Nonce: txNonce})
				}
			} else if ex {
				// delete in case of deliver tx
				nonceCache.Delete(fromStr, txNonce)
			}
		}

		// increase sequence of sender
		if err := acc.SetSequence(expectedNonce + 1); err != nil {
			return nil, errorsmod.Wrapf(err, "failed to set sequence to %d", acc.GetSequence()+1)
		}

		ak.SetAccount(ctx, acc)
	}

	return pending, nil
}

### The Ante Handler Problem

In Cosmos SDK, the ante handler processes transactions before execution. Ethermint's ante handler increments account nonces for **all messages in a batch upfront** before EVM execution begins. For a batch with 3 messages from the same sender:

- **Before ante handler:** Account nonce = N
- **After ante handler:** Account nonce = N+3

### EIP-7702 Self-Delegation Complexity

EIP-7702 allows Externally Owned Accounts (EOAs) to delegate their code execution to smart contracts. When an EOA self-delegates (authorizes delegation to itself in the same transaction), the authorization processing increments the nonce before the EVM call executes. This adds another layer of nonce management complexity:

- Authorization processing: +1 nonce per authorization
- EVM transaction execution: +1 nonce
- Nested contract creations: +N nonces (one per CREATE opcode)

In batched transactions with self-delegation, calculating the correct nonce for contract address derivation becomes increasingly complex and error-prone.
```
