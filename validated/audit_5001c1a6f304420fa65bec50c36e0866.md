### Title
`swap_migrated_token` Assumes 1:1 Raw Amount Between Old and New Token Without Decimal Validation — (`near/omni-bridge/src/lib.rs`)

### Summary
`swap_migrated_token` burns a raw `amount` of the old token and mints the identical raw `amount` of the new token, with no adjustment for decimal differences between the two tokens. If the DAO registers a migration where old and new tokens have different decimals — a realistic scenario for token upgrades — any unprivileged user can exploit the mismatch to receive far more (or far fewer) new tokens than the equivalent value, breaking bridge collateralization.

### Finding Description

`migrate_deployed_token` (DAO-only) registers an `old_token → new_token` mapping in `migrated_tokens` with no validation that the two tokens share the same decimal precision: [1](#0-0) 

Once registered, any user can call `ft_transfer_call` on the old token with `msg: "SwapMigratedToken"`, which routes through `ft_on_transfer`: [2](#0-1) 

`swap_migrated_token` then burns the raw `amount` of old_token and mints the identical raw `amount` of new_token: [3](#0-2) 

The function performs no decimal normalization whatsoever. The rest of the bridge correctly uses `normalize_amount` / `denormalize_amount` to handle decimal differences: [4](#0-3) 

But `swap_migrated_token` skips this entirely, hardcoding a 1:1 raw-amount assumption between old and new tokens.

### Impact Explanation

**High — Balance/decimal accounting corruption that breaks bridge collateralization.**

Concrete scenario: old_token has 18 decimals, new_token has 6 decimals.

- User sends `1e18` raw units of old_token (= 1 human-readable old_token).
- `swap_migrated_token` burns `1e18` old_token and mints `1e18` new_token.
- `1e18` raw units of new_token = **1,000,000,000,000 human-readable new_token**.
- The user receives 1 trillion new tokens instead of 1, draining the entire new token supply and creating unbacked minted supply on the destination chain.

The reverse (old_token 6 decimals, new_token 18 decimals) causes users to receive near-zero new tokens, permanently locking their value.

### Likelihood Explanation

**Medium.** Token migrations are a real operational event (e.g., upgrading a 6-decimal bridged USDC representation to an 18-decimal version). The DAO does not need to act maliciously — an honest decimal mismatch during a legitimate upgrade is sufficient. Once the migration is registered, the exploit is open to any unprivileged user who holds old tokens.

### Recommendation

In `migrate_deployed_token`, fetch and compare the decimals of both tokens before registering the migration, and reject if they differ. Alternatively, store the decimal ratio at migration time and apply it inside `swap_migrated_token`:

```rust
// In swap_migrated_token, apply decimal adjustment:
let adjusted_amount = adjust_for_decimals(amount, old_decimals, new_decimals);
let mint = ext_token::ext(new_token).mint(sender_id, adjusted_amount, None);
```

### Proof of Concept

1. DAO calls `migrate_deployed_token(ChainKind::Eth, "usdc-v1.near", "usdc-v2.near")` where `usdc-v1` has 6 decimals and `usdc-v2` has 18 decimals.
2. Attacker holds 1,000 `usdc-v1` (raw: `1_000_000_000` units at 6 decimals).
3. Attacker calls `usdc-v1.near::ft_transfer_call(receiver_id: bridge, amount: 1_000_000_000, msg: "\"SwapMigratedToken\"")`.
4. `ft_on_transfer` dispatches to `swap_migrated_token(attacker, "usdc-v1.near", 1_000_000_000)`.
5. Bridge burns `1_000_000_000` usdc-v1 and mints `1_000_000_000` usdc-v2.
6. At 18 decimals, `1_000_000_000` raw units = `0.000000001` usdc-v2 — attacker loses all value.

Reversed (v1=18 dec, v2=6 dec): attacker sends `1e18` raw units (= 1 v1 token) and receives `1e18` raw units of v2 = **1,000,000,000,000 v2 tokens**, minting unbacked supply and draining the bridge. [5](#0-4)

    fn swap_migrated_token(
        &mut self,
        sender_id: AccountId,
        old_token: AccountId,
        amount: U128,
    ) -> Promise {
        let new_token = self
            .migrated_tokens
            .get(&old_token)
            .near_expect(BridgeError::TokenNotMigrated);

        let burn = ext_token::ext(old_token).burn(amount);
        let mint = ext_token::ext(new_token).mint(sender_id, amount, None);

        burn.and(mint)
    }

    fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount * (10_u128.pow(diff_decimals))
