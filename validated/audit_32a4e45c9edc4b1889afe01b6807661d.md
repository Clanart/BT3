### Title
Stale `key_to_index` Cache After Leaf-Sibling Move in `MerkleBlob::delete` Corrupts DataLayer Tree State - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
When `MerkleBlob::delete` removes a leaf from a two-leaf tree, it physically moves the sibling leaf to `TreeIndex(0)` in the blob but calls `move_index`, which only updates `free_indexes` — it never updates `key_to_index` or `leaf_hash_to_index`. The sibling's old slot is marked free while the cache still maps the sibling's key to that freed slot. Every subsequent operation that looks up the sibling by key (`upsert`, `delete`, proof generation) reads from or writes to the wrong blob index, silently corrupting the Merkle tree root and invalidating all inclusion/exclusion proofs.

### Finding Description

`MerkleBlob::delete` handles the case where the deleted leaf's parent is the root (i.e., the tree has exactly two leaves) with this branch: [1](#0-0) 

The sibling block is written to `destination = TreeIndex(0)` via `insert_entry_to_blob`, and then `move_index(sibling_index, destination)` is called: [2](#0-1) 

`move_index` only inserts `source` into `free_indexes`. It never touches `key_to_index` or `leaf_hash_to_index`: [3](#0-2) 

Compare with `add_leaf`, which correctly maintains all three maps: [4](#0-3) 

After `delete` returns, the `BlockStatusCache` is in a contradictory state:
- `key_to_index[sibling.key]` → `sibling_index` (stale; that slot is now free)
- `leaf_hash_to_index[sibling.hash]` → `sibling_index` (stale)
- `free_indexes` contains `sibling_index`
- The blob has the sibling leaf at `TreeIndex(0)`, not at `sibling_index`

The `check_integrity` function explicitly checks for this exact mismatch: [5](#0-4) 

**Concrete corruption trace (2-leaf tree, leaves A at index 1, B at index 2, root at index 0):**

1. `delete(A)`:
   - `remove_leaf(A)` → `key_to_index = {B→2}`, `free_indexes = {1}`
   - Sibling B written to `TreeIndex(0)`
   - `move_index(2, 0)` → `free_indexes = {1, 2}`
   - **`key_to_index` still = `{B→2}` (stale; B is now at index 0)**

2. `upsert(B, new_value, new_hash)`:
   - `get_leaf_by_key(B)` returns index 2 (wrong; reads stale blob data)
   - `remove_leaf(B)` → `key_to_index = {}`, `free_indexes = {1, 2}` (2 already there)
   - `insert_entry_to_blob(2, updated_block)` → writes new data to free slot 2
   - `TreeIndex(0)` retains the **old** leaf B hash — root hash is now stale
   - `key_to_index` is empty; B appears non-existent to all future lookups

3. A subsequent `insert` may allocate index 2 for a new node, overwriting the updated-but-orphaned leaf B data, completing the corruption.

### Impact Explanation

The DataLayer Merkle root stored at `TreeIndex(0)` becomes permanently stale after any `upsert` or `delete` on the surviving leaf. All proofs of inclusion and exclusion generated from this point forward are computed against the wrong root hash. An operator or counterparty relying on DataLayer proofs receives cryptographically invalid attestations — either proving inclusion of a value that has since changed, or failing to prove inclusion of a value that genuinely exists. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots and lets untrusted input prove invalid state**.

### Likelihood Explanation

The trigger condition — a two-leaf tree reduced to one leaf by deletion — is a routine, low-privilege DataLayer operation. Any DataLayer store that starts with two entries and has one deleted hits this path. No special permissions, keys, or network access are required beyond the ability to submit DataLayer updates. The bug is deterministic and reproducible on every affected deletion.

### Recommendation

`move_index` must update `key_to_index` and `leaf_hash_to_index` when the moved node is a leaf, or the `delete` function should call `add_leaf(destination, sibling_leaf)` instead of `move_index` when the sibling is a leaf. Concretely, in the `grandparent_index = None` branch of `delete`, after writing the sibling to `TreeIndex(0)`, the cache should be updated as:

```rust
// instead of move_index:
if let Node::Leaf(ref leaf) = sibling_block.node {
    // remove old stale mapping
    self.block_status_cache.key_to_index.insert(leaf.key, destination);
    self.block_status_cache.leaf_hash_to_index.insert(leaf.hash, destination);
}
self.block_status_cache.free_indexes.insert(sibling_index);
```

### Proof of Concept

```rust
use chia_datalayer::{MerkleBlob, KeyId, ValueId, InsertLocation};

let mut blob = MerkleBlob::new(vec![]).unwrap();
let hash_a = sha256_num(&1u64);
let hash_b = sha256_num(&2u64);
let hash_b_new = sha256_num(&3u64);

// Build a 2-leaf tree
blob.insert(KeyId(1), ValueId(10), &hash_a, InsertLocation::AsRoot {}).unwrap();
blob.insert(KeyId(2), ValueId(20), &hash_b, InsertLocation::Auto {}).unwrap();

// Delete leaf A — triggers the stale-cache bug
blob.delete(KeyId(1)).unwrap();

// Upsert leaf B — writes to wrong blob index; root hash becomes stale
blob.upsert(KeyId(2), ValueId(99), &hash_b_new).unwrap();

// Integrity check reveals the corruption
blob.check_integrity().unwrap_err(); // IntegrityKeyToIndexCacheIndex or similar

// Root hash at TreeIndex(0) still reflects old hash_b, not hash_b_new
let root_hash = blob.get_hash_at_index(TreeIndex(0)).unwrap().unwrap();
assert_ne!(root_hash, expected_new_root); // root is stale
```
