### Title
Detached `send_tokens` Promise in Fast-Transfer Relayer Reimbursement Silently Discards Failures, Permanently Locking Relayer Funds in Bridge — (`near/omni-bridge/src/lib.rs`)

---

### Summary

In two fast-transfer finalization paths, the NEAR bridge calls `send_tokens(…).detach()` to reimburse the relayer. Because `.detach()` fires the cross-contract call and immediately discards the result, any failure of the underlying `ft_transfer` or `mint` is silently swallowed. The fast-transfer record is simultaneously marked as finalised (or removed), so the reimbursement can never be retried. The tokens remain permanently locked inside the bridge contract with no user-accessible recovery path.

---

### Finding Description

**Vulnerable path 1 — `process_fin_transfer_to_other_chain`**

When a proof arrives for a transfer whose fast-transfer leg was already executed by a relayer, the bridge reimburses the relayer:

```rust
// near/omni-bridge/src/lib.rs  lines 2028-2040
if let Some(relayer) = recipient {
    self.send_tokens(
        token,
        relayer,
        U128(transfer_message.amount_without_fee()...),
        "",
    )
    .detach();                                          // ← result ignored
    self.mark_fast_transfer_as_finalised(&fast_transfer.id());
}
``` [1](#0-0) 

The state mutation (`mark_fast_transfer_as_finalised`) executes in the same transaction regardless of whether `send_tokens` succeeds. If the cross-contract call fails, the fast-transfer entry is permanently finalised and the relayer's reimbursement tokens are stranded in the bridge.

**Vulnerable path 2 — `utxo_fin_transfer_fast`**

The same pattern appears for UTXO-chain fast transfers:

```rust
// near/omni-bridge/src/lib.rs  lines 2529-2548
let amount = if fast_transfer.get_destination_chain() == ChainKind::Near {
    self.remove_fast_transfer(&fast_transfer.id());   // state mutated first
    fast_transfer.amount
} else {
    self.mark_fast_transfer_as_finalised(&fast_transfer.id()); // state mutated first
    ...
};

self.send_tokens(
    fast_transfer.token_id.clone(),
    fast_transfer_status.relayer,
    amount,
    "",
)
.detach();                                            // ← result ignored
``` [2](#0-1) 

Here the state is mutated *before* the detached send, making the desync even more explicit.

**Contrast with the safe path**

The non-fast `process_fin_transfer_to_near` path correctly chains a callback:

```rust
// near/omni-bridge/src/lib.rs  lines 1957-1977
self.send_tokens(token.clone(), recipient, ...)
    .then(
        Self::ext(env::current_account_id())
            .fin_transfer_send_tokens_callback(...)
    )
``` [3](#0-2) 

That callback can handle failures. The fast-transfer reimbursement paths have no equivalent safety net.

**`send_tokens` failure modes**

`send_tokens` dispatches either `ft_transfer` (for locked native tokens) or `mint` (for deployed bridged tokens): [4](#0-3) 

Both can fail if the relayer account lacks a storage deposit on the token contract at the moment of finalization (e.g., the deposit was withdrawn between fast-transfer execution and proof arrival, or the token contract enforces additional invariants). For deployed tokens the `mint` call can also fail if the token contract is paused or has non-standard behavior.

**No user-accessible recovery**

There is no `withdrawERC20`-equivalent callable by the relayer. The only recovery path is the DAO-gated `transfer_token_as_dao`: [5](#0-4) 

This requires privileged intervention and is not guaranteed to be exercised promptly or at all.

---

### Impact Explanation

A relayer who pre-funded a fast transfer can permanently lose their reimbursement. The tokens remain locked in the bridge contract. Because the fast-transfer record is already finalised/removed, no retry is possible. This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

The failure condition (relayer's storage deposit absent or token transfer rejected at finalization time) is an edge case but is reachable without any privileged access. A relayer's storage deposit can be withdrawn by the relayer themselves between the two phases, or a non-standard token contract can reject the transfer. The two-phase window (fast-transfer execution → proof submission) can span minutes to hours, giving ample time for the storage state to change. The impact when triggered is irreversible without DAO action.

---

### Recommendation

Replace `.detach()` with a proper callback in both reimbursement paths, analogous to `fin_transfer_send_tokens_callback`. The callback should:
1. On success: leave the fast-transfer record finalised as today.
2. On failure: revert the finalisation state (or store the pending reimbursement) so the relayer can retry or the tokens can be reclaimed.

Additionally, add a relayer-callable recovery function (analogous to the report's recommended `withdrawERC20()`) that allows a relayer to claim a stuck reimbursement if the fast-transfer record shows finalised but the send failed.

---

### Proof of Concept

1. Relayer R executes a fast transfer: sends 1000 USDC to user U on NEAR.
2. R's storage deposit on the USDC token contract is subsequently withdrawn (R calls `storage_withdraw` on the token contract).
3. A proof submitter calls `fin_transfer` with the origin-chain proof.
4. `fin_transfer_callback` → `process_fin_transfer_to_other_chain` detects the fast transfer, calls `send_tokens(token, R, 1000, "").detach()`, then calls `mark_fast_transfer_as_finalised`.
5. The `ft_transfer` to R fails (no storage deposit) — the failure is silently discarded.
6. The fast-transfer record is now permanently finalised; no retry path exists.
7. 1000 USDC remain locked in the bridge contract. R has lost their reimbursement with no recourse except DAO intervention.

    fn send_tokens(
        &self,
        token: AccountId,
        recipient: AccountId,
        amount: U128,
        msg: &str,
    ) -> Promise {
        let ft_transfer_call_gas = env::prepaid_gas()
            .saturating_sub(env::used_gas())
            .saturating_sub(SEND_TOKENS_CALLBACK_GAS) // TODO: not all send_tokens callbacks has the same gas.
            .saturating_sub(MINT_TOKEN_GAS)
            .min(FT_TRANSFER_CALL_GAS);

        let is_deployed_token = self.is_deployed_token(&token);

        if token == self.wnear_account_id && msg.is_empty() {
            // Unwrap wNEAR and transfer NEAR tokens
            ext_wnear_token::ext(self.wnear_account_id.clone())
                .with_static_gas(WNEAR_WITHDRAW_GAS)
                .with_attached_deposit(ONE_YOCTO)
                .near_withdraw(amount)
                .then(
                    Self::ext(env::current_account_id())
                        .with_static_gas(NEAR_WITHDRAW_CALLBACK_GAS)
                        .near_withdraw_callback(recipient, NearToken::from_yoctonear(amount.0)),
                )
        } else if is_deployed_token {
            let deposit = if msg.is_empty() {
                NO_DEPOSIT
            } else {
                ONE_YOCTO
            };

            require!(
                ft_transfer_call_gas >= MIN_FT_TRANSFER_CALL_GAS,
                BridgeError::NotEnoughGasForTokenTransfer(ft_transfer_call_gas).as_ref()
            );

            ext_token::ext(token)
                .with_attached_deposit(deposit)
                .with_static_gas(MINT_TOKEN_GAS.saturating_add(ft_transfer_call_gas))
                .mint(
                    recipient,
                    amount,
                    (!msg.is_empty()).then(|| msg.to_string()),
                )
        } else if msg.is_empty() {
            ext_token::ext(token)
                .with_attached_deposit(ONE_YOCTO)
                .with_static_gas(FT_TRANSFER_GAS)
                .ft_transfer(recipient, amount, None)
        } else {
            require!(
                ft_transfer_call_gas >= MIN_FT_TRANSFER_CALL_GAS,
                BridgeError::NotEnoughGasForTokenTransfer(ft_transfer_call_gas).as_ref()
            );

            ext_token::ext(token)
                .with_attached_deposit(ONE_YOCTO)
                .with_static_gas(ft_transfer_call_gas)
                .ft_transfer_call(recipient, amount, None, msg.to_string())
        }
