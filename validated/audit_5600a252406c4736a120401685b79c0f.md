Let me look at the `ShardStateSyncResponseHeader` structure and the `shard_id()` accessor on `ShardChunk` to confirm the chunk's shard_id is accessible but never compared.

### Title
Missing shard_id binding in `set_state_header` allows cross-shard state root substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` for shard B that contains shard A's chunk and Merkle proof. Because `verify_path` does not check the leaf's position in the tree, and because no code in `set_state_header` ever compares `chunk.shard_id()` to the requested `shard_id`, the header passes all validation and is stored under `StateHeaderKey(shard_id=B, sync_hash)` with shard A's `prev_state_root`. All subsequent state part downloads and the final `set_state_finalize` call then use shard A's state root for shard B's storage, corrupting the syncing node's state for shard B.

---

### Finding Description

**Production entrypoint:** `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`.

The function performs five checks before persisting the header:

1. `validate_chunk_proofs(&chunk, ...)` — verifies the chunk's internal hash/tx/receipt consistency. No shard_id comparison. [1](#0-0) 

2. `verify_path(*sync_prev_block_header.chunk_headers_root(), chunk_proof, &ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` — verifies the chunk hash appears somewhere in the block's chunk-headers Merkle tree. [2](#0-1) 

3. A symmetric `verify_path` for `prev_chunk`. Same issue. [3](#0-2) 

4. Receipt-proof chain validation using the function-parameter `shard_id` (shard B), not the chunk's embedded shard_id. [4](#0-3) 

5. `validate_state_root_node(state_root_node, chunk_inner.prev_state_root())` — validates the trie root node against the chunk's `prev_state_root` (shard A's root). [5](#0-4) 

Then the header is stored keyed by the **function-parameter** `shard_id` (B), not the chunk's embedded shard_id: [6](#0-5) 

**The root cause:** `verify_path` only checks that `compute_root_from_path(path, hash(item)) == root`. It does not verify that the leaf sits at the index corresponding to `shard_layout.get_shard_index(shard_id_B)`. The function `verify_path_with_index`, which does enforce position, is used for chunk erasure-coding parts but is never called here. [7](#0-6) [8](#0-7) 

The chunk's `shard_id` is cryptographically bound inside `ShardChunkHeaderInner` and is accessible via `ShardChunkHeaderInner::shard_id()`, but `set_state_header` never reads it. [9](#0-8) 

**Attack construction (all inputs are public chain data):**

| Attacker-supplied field | Value |
|---|---|
| `chunk` | Shard A's `ShardChunk` (fetched from any full node) |
| `chunk_proof` | Valid Merkle path for shard A's `ChunkHashHeight` in `sync_prev_block.chunk_headers_root` |
| `prev_chunk_header` / `prev_chunk_proof` | Shard A's prev chunk header + its Merkle path |
| `incoming_receipts_proofs` | Actual incoming receipts for shard B (public) |
| `root_proofs` | Actual outgoing-receipt roots + block proofs for those receipts (public) |
| `state_root_node` | Shard A's trie root node (public) |

All five checks pass. The header is stored under `StateHeaderKey(shard_id=B, sync_hash)` carrying shard A's `prev_state_root`.

**Downstream propagation:**

`set_state_part` reads the stored header to obtain `state_root` and validates each downloaded part against it — now shard A's root: [10](#0-9) 

`set_state_finalize` then calls `apply_chunk` with `shard_uid` derived from shard B but `prev_state_root` from shard A's chunk header, writing shard A's state into shard B's trie slot: [11](#0-10) 

---

### Impact Explanation

A syncing node installs shard A's state under shard B's storage key. After finalization, every block the node processes will compute an incorrect state root for shard B, causing it to reject all valid blocks for that shard and be permanently unable to participate in consensus or serve correct data for shard B. The corruption is silent during state sync itself; it surfaces only when the node attempts to apply the first post-sync block.

---

### Likelihood Explanation

State sync requests headers from any peer that advertises availability. No authentication or trust level is required to respond to a `StateRequestHeader` message. A single malicious peer on the network can target any syncing node. The attack requires only public on-chain data (chunk bodies, Merkle paths, receipt proofs, trie root nodes) and no validator or operator privileges.

---

### Recommendation

Add an explicit shard_id binding check immediately after extracting the chunk in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Apply the same check for `prev_chunk_header.shard_id()`. Consider also replacing the bare `verify_path` calls with `verify_path_with_index` (already used for erasure-coded parts) so that the Merkle position is explicitly bound to `shard_layout.get_shard_index(shard_id)`. [8](#0-7)
