### Title
User Funds Permanently Locked When Transfer Amount Normalizes to Zero Due to Decimal Truncation - (File: near/omni-bridge/src/lib.rs)

### Summary

The NEAR bridge contract locks or burns a user's tokens during `init_transfer_internal` before validating that the decimal-normalized transfer amount is non-zero. The zero-amount guard only fires later in `sign_transfer`, called by a relayer, at which point the tokens are already irrecoverably consumed and no public cancel path exists.

### Finding Description

`normalize_amount` performs floor integer division to convert a NEAR-side token amount to the destination chain's decimal precision: [1](#0-0) 

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

For any token whose NEAR-side decimals exceed the destination chain's decimals (e.g., 24 on NEAR vs 18 on Ethereum, giving `diff_decimals = 6`), any `amount_without_fee < 10^6` normalizes to **0**.

The guard that catches this is placed in `sign_transfer`, which is called by a relayer *after* the user's tokens have already been consumed: [2](#0-1) 

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

By the time `sign_transfer` is called, `init_transfer_internal` has already executed the success path — returning `U128(0)` to the token contract (meaning zero tokens are refunded) and calling `burn_tokens_if_needed` / `lock_tokens_if_needed`: [3](#0-2) 

```rust
if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
    self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
    self.lock_tokens_if_needed(
        transfer_message.get_destination_chain(),
        &token_id,
        transfer_message.amount.0,
    );
} else { ... }
...
U128(0)  // tokens fully consumed
```

The transfer record is stored in `pending_transfers`: [4](#0-3) 

There is no public cancel or refund function exposed to users. The internal `remove_transfer_message_without_refund` and `remove_transfer_message` helpers are only reachable through privileged or callback-gated paths. The transfer is permanently stuck: the relayer's `sign_transfer` call will always panic, and the user has no recovery mechanism.

Neither `ft_on_transfer` nor `init_transfer_internal` validates that `normalize_amount(amount_without_fee, decimals) > 0` before consuming tokens: [5](#0-4) 

### Impact Explanation

A user who initiates a cross-chain transfer with an `amount_without_fee` smaller than `10^(origin_decimals - dest_decimals)` will have their NEAR-side tokens permanently burned or locked. The transfer record sits in `pending_transfers` indefinitely; `sign_transfer` always reverts on it; and no public cancel path exists. This is an irrecoverable loss of user funds matching the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

### Likelihood Explanation

The condition is reachable by any unprivileged bridge user without any front-running or privileged access. Tokens with large decimal differences (e.g., NEAR-native tokens at 24 decimals bridging to an 18-decimal EVM token) are common in the protocol. A user sending a small "dust" amount — or a user who simply does not know the minimum bridgeable unit — can trigger this silently. The `init_transfer` entry point imposes no minimum normalized-amount check, making accidental triggering realistic.

### Recommendation

Add a validation in `init_transfer_internal` (or in `ft_on_transfer` before tokens are consumed) that computes `normalize_amount(amount_without_fee, decimals)` and requires it to be greater than zero, reverting the transfer and refunding the user if not. Concretely:

```rust
let normalized = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

This check must occur **before** `burn_tokens_if_needed` / `lock_tokens_if_needed` are called, so that a failing check causes `ft_on_transfer` to return the full amount to the token contract (triggering a refund to the user).

### Proof of Concept

1. A token `T` is registered with `origin_decimals = 24` (NEAR) and `decimals = 18` (Ethereum), so `diff_decimals = 6`.
2. User calls `ft_transfer_call` on token `T` with `amount = 500_000` (i.e., 0.5 micro-token in 18-decimal terms) and `fee = 0`, targeting an Ethereum recipient.
3. `ft_on_transfer` → `init_transfer` → `init_transfer_internal` executes:
   - Transfer message stored in `pending_transfers`.
   - `burn_tokens_if_needed` burns 500,000 units of `T` from the bridge.
   - Returns `U128(0)` → token contract does not refund the user.
4. Relayer calls `sign_transfer` for this transfer ID.
5. `normalize_amount(500_000, Decimals { origin_decimals: 24, decimals: 18 })` = `500_000 / 1_000_000` = **0**.
6. `require!(0 > 0, ...)` panics → `sign_transfer` always reverts.
7. User's 500,000 units of `T` are permanently burned; the transfer can never be finalized or cancelled. [6](#0-5) [2](#0-1) [7](#0-6)

            }

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
