### Title
Fee-on-Transfer Token Accounting Corruption in `initTransfer` - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.initTransfer()` records the caller-supplied `amount` in the cross-chain message and event, but when the token is a fee-on-transfer ERC20, the contract actually receives `amount - transfer_fee`. The NEAR side finalizes the transfer for the full nominal `amount`, minting or releasing more tokens than the EVM bridge holds, permanently breaking bridge collateralization.

### Finding Description

In `OmniBridge.initTransfer()`, for plain ERC20 tokens (the non-bridge-token, non-custom-minter branch), the contract pulls tokens from the sender and then passes the nominal `amount` directly to `initTransferExtension` and the `InitTransfer` event:

```solidity

} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // actual received = amount - transfer_fee

}
...

    msg.sender, tokenAddress, currentOriginNonce,
    amount,             // nominal, not actual received
    fee, nativeFee, recipient, message, extensionValue

emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount,             // nominal, not actual received
```

The `InitTransfer` event (or Wormhole message in `OmniBridgeWormhole`) carries `amount` to the NEAR bridge. NEAR's `fin_transfer_callback` denormalizes and credits the full `amount` to the recipient. The EVM bridge, however, only holds `amount - transfer_fee` of the token. Every such transfer leaves the bridge short by exactly the token's transfer fee. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

Each `initTransfer` call with a fee-on-transfer token causes the EVM bridge to be undercollateralized by the fee amount. When users later bridge back from NEAR to EVM, `finTransfer` attempts to release the full `amount` but the bridge holds less. Repeated use drains the bridge's reserves, eventually causing `finTransfer` to revert for legitimate recipients — permanently freezing their funds — or, if the shortfall is small enough to be covered by other depositors' tokens, silently stealing from them. This matches the allowed impact: **balance/accounting corruption that breaks bridge collateralization**. [4](#0-3) 

### Likelihood Explanation

Any unprivileged user can call `initTransfer` with any ERC20 token address. Fee-on-transfer tokens (e.g., USDT on some chains, STA, PAXG, etc.) are widely deployed. No special role or privileged access is required. The attacker simply calls `initTransfer` with such a token; the accounting error is automatic and cumulative. [5](#0-4) 

### Recommendation

Before and after the `safeTransferFrom` call, snapshot the contract's token balance and use the difference as the actual credited amount:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived instead of amount in initTransferExtension and InitTransfer
```

Pass `uint128(actualReceived)` (with appropriate overflow checks) to `initTransferExtension` and the `InitTransfer` event so the cross-chain message reflects only what the bridge actually holds.

### Proof of Concept

1. Deploy a fee-on-transfer ERC20 token `FOT` with a 1% transfer fee on EVM.
2. Approve `OmniBridge` for 1000 `FOT`.
3. Call `initTransfer(FOT, 1000, 0, 0, nearRecipient, "")`.
4. `safeTransferFrom` moves 1000 `FOT` from the caller, but the bridge receives only 990 (1% fee deducted by the token).
5. `InitTransfer` event (and Wormhole message) records `amount = 1000`.
6. NEAR's `fin_transfer_callback` processes the proof and credits the recipient with 1000 tokens (minted or released from NEAR-side reserves).
7. The EVM bridge is now short 10 `FOT`.
8. Repeat N times; the bridge deficit grows by 10 `FOT` per call.
9. When any user tries to bridge 1000 `FOT` back from NEAR to EVM, `finTransfer` calls `IERC20(FOT).safeTransfer(recipient, 1000)` but the bridge only holds `1000 - N*10` tokens, causing the transfer to revert and permanently freezing the recipient's funds. [6](#0-5)

```text
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,

        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;

            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
```
