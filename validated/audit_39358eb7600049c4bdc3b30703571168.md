### Title
Missing Token-Registration Validation in `init_transfer` Allows Permanent Fund Lock - (`near/omni-bridge/src/lib.rs`)

### Summary

The NEAR bridge's `init_transfer` function (and its EVM/StarkNet counterparts) does not verify that the bridged token has a registered address on the destination chain before burning or locking user funds. A user who specifies a destination chain for which the token has no `token_id_to_address` mapping will have their tokens permanently destroyed or locked, with no recovery path.

### Finding Description

The vulnerability class from the external report — initiating a cross-chain transfer without validating the destination is a known/registered counterpart — maps directly onto the NEAR bridge's `init_transfer` entry point.

**Entry point**: `ft_on_transfer` → `init_transfer` → `init_transfer_internal`

In `init_transfer`, the only destination-chain validation performed is:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
``` [1](#0-0) 

There is no check that `token_id_to_address.get(&(destination_chain, token_id))` returns `Some(...)`, i.e., that the token is actually registered for the chosen destination chain. The transfer message is immediately stored and tokens are consumed:

```rust
// burn deployed tokens
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
// lock native tokens (silently no-ops if no locked_tokens entry exists)
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
``` [2](#0-1) 

`lock_tokens_if_needed` silently returns `LockAction::Unchanged` when no `locked_tokens` entry exists for the `(chain, token)` pair, so the accounting is skipped without any error:

```rust
let Some(current_amount) = self.locked_tokens.get(&key) else {
    return LockAction::Unchanged;
};
``` [3](#0-2) 

Later, when a relayer calls `sign_transfer`, it calls `get_token_address` for the destination chain and **panics** if no mapping exists:

```rust
let token_address = self
    .get_token_address(
        transfer_message.get_destination_chain(),
        self.get_token_id(&transfer_message.token),
    )
    .unwrap_or_else(|| {
        env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
    });
``` [4](#0-3) 

Because `sign_transfer` always panics for an unregistered token-chain pair, the transfer stored in `pending_transfers` can never be signed or completed. There is no user-callable cancel or refund function in the contract.

The same pattern exists in the EVM bridge's `initTransfer`, which accepts any `tokenAddress` without checking `ethToNearToken[tokenAddress]`:

```solidity
function initTransfer(
    address tokenAddress, uint128 amount, uint128 fee, uint128 nativeFee,
    string calldata recipient, string calldata message
) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    ...
    // no check that ethToNearToken[tokenAddress] is set
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    ...
    emit BridgeTypes.InitTransfer(...);
}
``` [5](#0-4) 

And in StarkNet's `init_transfer`, which accepts any `token_address` without checking `starknet_to_near_token`: [6](#0-5) 

### Impact Explanation

- **For deployed (bridge) tokens on NEAR**: tokens are burned via `burn_tokens_if_needed`. They are permanently destroyed with no recovery.
- **For native NEAR tokens**: tokens are transferred into the bridge contract via `ft_transfer_call` and locked. The transfer is stored in `pending_transfers` but `sign_transfer` will always panic. No cancel mechanism exists. Funds are irrecoverably locked.
- **For EVM native tokens**: tokens are locked in the EVM bridge contract. The NEAR `fin_transfer_callback` will panic at `token_decimals.get(&init_transfer.token)` because the token is unregistered. Funds are irrecoverably locked.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

### Likelihood Explanation

Any unprivileged bridge user can trigger this by calling `ft_transfer_call` on a NEAR token contract with an `InitTransferMsg` whose `recipient` is on a chain where that token has no registered address. This requires a user mistake (choosing an unsupported chain/token combination), but the protocol provides no guard against it. The `OmniAddress` type encodes the destination chain inside the recipient address, so a user who simply picks the wrong chain type for their recipient address will silently lose funds. The likelihood is medium — analogous to the original report's medium severity rating.

### Recommendation

Add a validation in `init_transfer` (and its EVM/StarkNet equivalents) that the token is registered for the destination chain before consuming user funds:

```rust
// In init_transfer, before init_transfer_internal:
require!(
    self.token_id_to_address
        .get(&(init_transfer_msg.get_destination_chain(), token_id.clone()))
        .is_some(),
    BridgeError::TokenNotRegistered.as_ref()
);
```

For EVM, add a check in `initTransfer`:
```solidity
require(
    bytes(ethToNearToken[tokenAddress]).length > 0 || tokenAddress == address(0),
    "ERR_TOKEN_NOT_REGISTERED"
);
```

### Proof of Concept

1. Token `usdc.near` is registered for `ChainKind::Eth` (has an entry in `token_id_to_address`) but **not** for `ChainKind::Sol`.
2. User calls `ft_transfer_call` on `usdc.near` with:
   ```json
   {
     "receiver_id": "omni-bridge.near",
     "amount": "1000000",
     "msg": "{\"InitTransfer\":{\"recipient\":\"Sol:2xNweLHLqbS9YpP3UyaPrxKqgqoC6yPBFyuLxA8qtgr4\",\"fee\":\"0\",\"native_token_fee\":\"0\"}}"
   }
   ```
3. `init_transfer` passes the only check (`recipient.get_chain() != ChainKind::Near`). [1](#0-0) 
4. `init_transfer_internal` burns the tokens (if deployed) or locks them. [2](#0-1) 
5. Transfer is stored in `pending_transfers`. `InitTransferEvent` is emitted.
6. Relayer calls `sign_transfer`. `get_token_address(ChainKind::Sol, "usdc.near")` returns `None`. Contract panics with `FailedToGetTokenAddress`. [4](#0-3) 
7. Transfer is permanently stuck. User's 1,000,000 USDC is irrecoverably lost.

        address tokenAddress,
        uint128 amount,

        uint128 nativeFee,

        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();

            extensionValue = msg.value - amount - nativeFee;

            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );

                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message

        fn init_transfer(
            ref self: ContractState,
            token_address: ContractAddress,
            amount: u128,
            fee: u128,
            native_fee: u128,
            recipient: ByteArray,
            message: ByteArray,
        ) {
            assert(!_is_paused(@self, PAUSE_INIT_TRANSFER), 'ERR_INIT_TRANSFER_PAUSED');

            assert(amount > 0, 'ERR_ZERO_AMOUNT');
            assert(fee < amount, 'ERR_INVALID_FEE');

            let origin_nonce = self.current_origin_nonce.read() + 1;
            self.current_origin_nonce.write(origin_nonce);

            let caller = get_caller_address();

            if self.is_bridge_token(token_address) {
                IBridgeTokenDispatcher { contract_address: token_address }
                    .burn(caller, amount.into());
            } else {
                let success = IERC20Dispatcher { contract_address: token_address }
                    .transfer_from(caller, get_contract_address(), amount.into());
                assert(success, 'ERR_TRANSFER_FROM_FAILED');
            }

            if native_fee > 0 {
                let native_token = self.strk_token_address.read();
                let success = IERC20Dispatcher { contract_address: native_token }
                    .transfer_from(caller, get_contract_address(), native_fee.into());
                assert(success, 'ERR_FEE_TRANSFER_FAILED');
            }

            self
                .emit(
                    Event::InitTransfer(
                        InitTransfer {
                            sender: caller,
                            token_address,
                            origin_nonce,
                            amount,
                            fee,
                            native_fee,
                            recipient,
                            message,
                        },
                    ),
                )
        }
