### Title
Arithmetic Underflow in `check_proof_facts` Causes Unprovable Block (Network Halt) — (File: `execution/execution_constraints.cairo`)

---

### Summary

The `check_proof_facts` function in `execution_constraints.cairo` performs an unchecked subtraction `current_block_number - STORED_BLOCK_HASH_BUFFER` without first verifying that `current_block_number >= STORED_BLOCK_HASH_BUFFER`. In Cairo's felt arithmetic (modulo a ~252-bit prime P), when `current_block_number < 10`, the subtraction wraps to a value near P (~2^251). The subsequent `assert_nn_le` range-check then fails because it cannot accommodate a value of that magnitude, causing the OS Cairo program to panic and the block to become unprovable — a network halt.

---

### Finding Description

In `execution_constraints.cairo`, `check_proof_facts` is called for every invoke transaction that carries non-empty `proof_facts`: [1](#0-0) 

```cairo
// Validate that the proof facts block number is not too recent.
assert_nn_le(
    os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
);
```

`STORED_BLOCK_HASH_BUFFER = 10`. [2](#0-1) 

When `current_block_number < 10`, the felt subtraction `current_block_number - 10` wraps modulo P to approximately `P - (10 - current_block_number)` ≈ 2^251. `assert_nn_le(a, b)` internally checks that both `a` and `b - a` fit in `[0, 2^128 - 1]` via range-check builtins. A value of ~2^251 fails this check, causing the Cairo program to panic and the proof to be invalid.

**Contrast with the correct guard in `os_utils.cairo`**, which performs the identical subtraction but first checks non-negativity before proceeding: [3](#0-2) 

```cairo
tempvar old_block_number = block_context.block_info_for_execute.block_number -
    STORED_BLOCK_HASH_BUFFER;
let is_old_block_number_non_negative = is_nn(old_block_number);
if (is_old_block_number_non_negative == FALSE) {
    return ();
}
```

`check_proof_facts` has no equivalent guard. The `proof_facts_size == 0` early-return only skips the function when the transaction carries no proof facts; any transaction with non-empty `proof_facts` reaches the vulnerable subtraction. [4](#0-3) 

The call site in `execute_invoke_function_transaction` passes `current_block_number` directly from the block context and `proof_facts` directly from the transaction hint — both are reachable by an unprivileged sender: [5](#0-4) 

---

### Impact Explanation

The StarkNet OS is a Cairo program whose successful execution is required to generate a STARK proof for each block. If the OS panics (range-check failure), no valid proof can be produced. Without a valid proof, the block cannot be verified and settled on L1. Every subsequent block depends on the previous state root, so a single unprovable block halts the entire network — matching the **High: Network not being able to confirm new transactions (total network shutdown)** impact category.

---

### Likelihood Explanation

The vulnerable window is the first 10 blocks of any chain (block numbers 0–9). This applies to:
- New chain deployments
- Chain restarts / regenesis events
- Any testnet or staging environment in its early blocks

An attacker only needs to submit one invoke transaction with a syntactically valid `proof_facts` payload (correct `program_hash`, `proof_version`, `proof_variant`, `output_version` — all public constants) before block 10. The sequencer's blockifier executes transactions in Rust and does not run the Cairo OS; it may not replicate this exact range-check guard, creating a discrepancy between accepted transactions and provable blocks. The attacker requires no privileged access, no key material, and no cryptographic capability beyond submitting a standard invoke transaction.

---

### Recommendation

Mirror the guard used in `os_utils.cairo`. Before the subtraction, verify that `current_block_number >= STORED_BLOCK_HASH_BUFFER`:

```cairo
// In check_proof_facts, before assert_nn_le:
let is_block_number_sufficient = is_nn(current_block_number - STORED_BLOCK_HASH_BUFFER);
if (is_block_number_sufficient == FALSE) {
    // No stored block hashes exist yet; proof facts referencing them are invalid.
    // Treat as an error or simply reject.
    assert 1 = 0;  // or write a failure response and return
    return ();
}
assert_nn_le(
    os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
);
```

Alternatively, add a gateway-level check that rejects transactions with non-empty `proof_facts` when the chain's block number is below `STORED_BLOCK_HASH_BUFFER`.

---

### Proof of Concept

1. Start a new StarkNet chain (or use any environment at block 0–9).
2. Craft an invoke transaction (version 3) with `proof_facts` set to a byte sequence that satisfies:
   - `proof_header.program_hash = ALLOWED_VIRTUAL_OS_PROGRAM_HASHES_0` (public constant)
   - `proof_header.proof_version = PROOF_VERSION`, `proof_header.proof_variant = VIRTUAL_SNOS`
   - `os_output_header.output_version = VIRTUAL_OS_OUTPUT_VERSION`
   - `os_output_header.base_block_number` = any value (e.g., 0)
3. Submit the transaction; the sequencer's blockifier accepts it (no OS-level range-check guard in Rust).
4. The sequencer includes it in block N where N < 10.
5. The OS Cairo program is invoked to prove the block. It reaches:
   ```cairo
   assert_nn_le(0, N - 10)   // N - 10 wraps to ~P ≈ 2^251
   ```
6. The range-check builtin rejects the value ~2^251 (exceeds 2^128 − 1). The OS panics.
7. No valid STARK proof can be generated. The block is unprovable. The network halts.
