### Title
Stale `feemarketParams` Captured at Construction Time in `newCosmosAnteHandler` Causes Fee Validation to Use Outdated Base Fee — (File: `evmd/ante/handler_options.go`)

---

### Summary

`newCosmosAnteHandler` and `newLegacyCosmosAnteHandlerEip712` both capture `feemarketParams` (including `BaseFee`) from the keeper at handler **construction time** and pass a pointer to that snapshot into `NewDynamicFeeChecker`. Because the ante handler is constructed once at app startup (via `NewAnteHandler` → `evmd/app.go`), the `BaseFee` used to validate Cosmos SDK transaction fees is permanently frozen at the genesis/startup value. As the EIP-1559 base fee rises block-by-block, native Cosmos SDK transactions (governance, staking, IBC, etc.) can be submitted with fees below the live base fee and still pass the fee checker, bypassing the fee market for the entire lifetime of the node.

This is the direct analog to the VeloOracle bug: just as `block.timestamp != blockTimestamp` was an insufficient freshness check because Velodrome pools only update every 15 seconds, `feeCap.LT(baseFeeInt)` in `NewDynamicFeeChecker` is an insufficient fee-market check because `baseFeeInt` is derived from a stale snapshot captured at construction time rather than from the live store.

---

### Finding Description

**Root cause — stale capture in `newCosmosAnteHandler`:**

```go
// evmd/ante/handler_options.go  lines 178-211
func newCosmosAnteHandler(ctx sdk.Context, options HandlerOptions, ...) sdk.AnteHandler {
    evmParams        := options.EvmKeeper.GetParams(ctx)       // captured once
    feemarketParams  := options.FeeMarketKeeper.GetParams(ctx) // captured once
    ...
    txFeeChecker = evm.NewDynamicFeeChecker(ethCfg, &evmParams, &feemarketParams)
    ...
    return sdk.ChainAnteDecorators(decorators...)
}
```

`feemarketParams` is a **value-type local variable**. `&feemarketParams` escapes to the heap inside the closure, but its contents are never refreshed. The same stale snapshot is used for every subsequent transaction.

The identical pattern exists in `newLegacyCosmosAnteHandlerEip712` (lines 29–30 of the same file).

**Where the stale value is consumed:**

```go
// ante/evm/fee_checker.go  line 56
baseFee := types.GetBaseFee(ctx.BlockHeight(), ethCfg, feemarketParams)
```

`types.GetBaseFee` reads `feemarketParams.GetBaseFee()` — the frozen snapshot — not the live store:

```go
// x/evm/types/utils.go  lines 244-254
func GetBaseFee(height int64, ethCfg *params.ChainConfig, feemarketParams *feemarkettypes.Params) *big.Int {
    if !IsLondon(ethCfg, height) { return nil }
    baseFee := feemarketParams.GetBaseFee()   // reads the stale snapshot
    ...
    return baseFee
}
```

The resulting `baseFeeInt` is then used as the enforcement threshold:

```go
// ante/evm/fee_checker.go  line 86
if feeCap.LT(baseFeeInt) { return ..., ErrInsufficientFee }
```

Because `baseFeeInt` is the genesis value, any Cosmos SDK transaction whose `feeCap` exceeds the genesis base fee passes this check regardless of how high the live base fee has risen.

**Contrast with the correct pattern in `newEthAnteHandler`:**

```go
// evmd/ante/handler_options.go  lines 86-96
func newEthAnteHandler(options HandlerOptions) sdk.AnteHandler {
    return func(ctx sdk.Context, tx sdk.Tx, simulate bool) (sdk.Context, error) {
        blockCfg, err := options.EvmKeeper.EVMBlockConfig(ctx, options.EvmKeeper.ChainID())
        ...
        baseFee := blockCfg.BaseFee   // fresh per-transaction
```

`newEthAnteHandler` fetches a fresh `EVMBlockConfig` (and therefore a fresh `BaseFee`) on every transaction. `newCosmosAnteHandler` does not.

**`EVMBlockConfig` itself reads the live fee-market params:**

```go
// x/evm/keeper/config.go  lines 85, 95
feemarketParams := k.feeMarketKeeper.GetParams(ctx)
...
baseFee = feemarketParams.GetBaseFee()
```

---

### Impact Explanation

When `DynamicFeeChecker` is enabled, every Cosmos SDK transaction (governance, staking, IBC relay, etc.) is fee-validated against the genesis base fee rather than the current block's base fee. In a network where the EIP-1559 base fee has risen above its genesis value (the normal case for any active chain), an unprivileged user can submit Cosmos SDK transactions with fees between the genesis base fee and the live base fee. These transactions:

1. Pass `NewDynamicFeeChecker` (stale threshold is lower than the fee provided).
2. Pass `MinGasPriceDecorator` (also uses the stale `&feemarketParams` snapshot, line 198).
3. Are included in committed blocks with fees below what the live fee market requires.

This is a fee-market ante handler bug that **permits invalid transactions to commit** and causes **valid user funds/fees to be mis-accounted** — matching the High impact category.

---

### Likelihood Explanation

- The base fee changes every block; on any active chain it will diverge from the genesis value within minutes.
- The genesis base fee is publicly readable from the chain's genesis file.
- No special privileges are required: any account can craft a Cosmos SDK transaction with a fee above the genesis base fee but below the current live base fee.
- The only prerequisite is that `HandlerOptions.DynamicFeeChecker` is set to `true`, which is the intended configuration for any chain using the EIP-1559 fee market.

---

### Recommendation

Fetch `feemarketParams` (and `evmParams`) **inside** the returned handler closure, not at construction time, mirroring the pattern already used in `newEthAnteHandler`:

```go
func newCosmosAnteHandler(options HandlerOptions, ...) sdk.AnteHandler {
    return func(ctx sdk.Context, tx sdk.Tx, simulate bool) (sdk.Context, error) {
        evmParams       := options.EvmKeeper.GetParams(ctx)
        feemarketParams := options.FeeMarketKeeper.GetParams(ctx)
        // build decorators with fresh params and execute
        ...
    }
}
```

Alternatively, pass the keeper references into `NewDynamicFeeChecker` so it can query fresh params on each invocation.

---

### Proof of Concept

1. Start an Ethermint chain with `NoBaseFee = false`, `EnableHeight = 0`, and `DynamicFeeChecker = true`.
2. Send enough EVM transactions to fill blocks above the gas target; observe the live base fee rising above the genesis value (e.g., genesis = 1 000 000 000, live = 2 000 000 000 after N blocks).
3. Craft a Cosmos SDK governance `MsgSubmitProposal` with `fee = 1 500 000 000 * gasLimit` (above genesis base fee, below live base fee).
4. Broadcast the transaction. `NewDynamicFeeChecker` compares against the frozen genesis base fee (1 000 000 000) and accepts it; the live fee market threshold (2 000 000 000) is never consulted.
5. The transaction is included in a committed block with fees below the current EIP-1559 requirement, demonstrating the fee-market bypass.
