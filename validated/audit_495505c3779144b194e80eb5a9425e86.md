### Title
Contract Address Computation Excludes `chain_id`, Enabling Cross-Network Replay of Deployments — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/contract_address/contract_address.cairo`)

---

### Summary

The `get_contract_address` function computes deterministic contract addresses without including the network's `chain_id`. Because StarkNet already operates on multiple live networks (mainnet, Sepolia testnet), contract addresses are identical across all of them for the same deployment parameters. An attacker who observes a deployment on one network can replay it on another, seizing control of the same address and stealing any funds directed there.

---

### Finding Description

`get_contract_address` hashes exactly five inputs:

```
CONTRACT_ADDRESS_PREFIX || deployer_address || salt || class_hash || H(constructor_calldata)
``` [1](#0-0) 

`chain_id` is never included. The function is called directly inside `prepare_constructor_execution_context` (used for every `deploy_account` transaction) with `deployer_address=0`, making the address depend solely on `salt`, `class_hash`, and `constructor_calldata`: [2](#0-1) 

The same function is also invoked for `deploy` syscalls, so the vulnerability covers all contract deployment paths in the OS.

By contrast, every *transaction* hash correctly binds to the chain via `block_context.os_global_context.starknet_os_config.chain_id`: [3](#0-2) 

The OS output also carries a `starknet_os_config_hash` that encodes `chain_id`: [4](#0-3) 

But neither of these protections covers the address derivation step itself. The address is fixed before any transaction hash or config hash is computed.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A user who deploys a contract on Sepolia (or any non-mainnet StarkNet network) with a given `(salt, class_hash, constructor_calldata)` tuple produces address `A`. Because `chain_id` is absent from the hash, address `A` is identical on mainnet. An attacker who observes the Sepolia deployment can immediately replay it on mainnet — deploying the same class with the same parameters — and take ownership of address `A` on mainnet before the legitimate user does. Any funds subsequently sent to `A` on mainnet (e.g., as part of a deployment workflow, pre-funded multisig, or protocol bootstrap) are under the attacker's control. This is the on-chain equivalent of the Wintermute Gnosis Safe attack cited in the external report.

---

### Likelihood Explanation

**Medium-High.** StarkNet mainnet and Sepolia testnet are both live and widely used. Protocols routinely test deployments on Sepolia before mainnet. The attack requires no privileged access: the attacker only needs to monitor public chain state on one network and submit a matching deployment transaction on another. The class hash must already be declared on the target network, but Sierra class declarations are also public and replayable (and the declare transaction hash *does* include `chain_id`, so the class declaration itself is safe — only the subsequent address derivation is not).

---

### Recommendation

Include `chain_id` as an additional field in the `get_contract_address` hash computation, analogous to how it is included in every transaction hash:

```cairo
// Proposed addition inside get_contract_address:
let (hash_state_ptr) = hash_update_single(hash_state_ptr=hash_state_ptr, item=chain_id);
```

`chain_id` must be threaded into `get_contract_address` as a parameter (sourced from `block_context.os_global_context.starknet_os_config.chain_id`) wherever the function is called.

---

### Proof of Concept

1. Attacker monitors Sepolia for `deploy_account` or `deploy` syscall transactions.
2. User deploys contract with `salt=S`, `class_hash=C`, `calldata=D` on Sepolia → OS computes address `A = H(PREFIX || 0 || S || C || H(D))`. No `chain_id` in the hash.
3. Attacker submits the identical deployment on mainnet (same `S`, `C`, `D`). OS computes the same `A` on mainnet.
4. Attacker's transaction is included first (or the user mistakenly sends funds to `A` on mainnet before deploying there).
5. Attacker controls the contract at address `A` on mainnet and drains any funds sent to it.

The root cause is entirely within the scoped file: [5](#0-4)

func get_contract_address{hash_ptr: HashBuiltin*, range_check_ptr}(
    salt: felt,
    class_hash: felt,
    constructor_calldata_size: felt,
    constructor_calldata: felt*,
    deployer_address: felt,
) -> (contract_address: felt) {
    let (hash_state_ptr) = hash_init();
    let (hash_state_ptr) = hash_update_single(
        hash_state_ptr=hash_state_ptr, item=CONTRACT_ADDRESS_PREFIX
    );
    let (hash_state_ptr) = hash_update_single(hash_state_ptr=hash_state_ptr, item=deployer_address);
    let (hash_state_ptr) = hash_update_single(hash_state_ptr=hash_state_ptr, item=salt);
    let (hash_state_ptr) = hash_update_single(hash_state_ptr=hash_state_ptr, item=class_hash);
    let (hash_state_ptr) = hash_update_with_hashchain(
        hash_state_ptr=hash_state_ptr,
        data_ptr=constructor_calldata,
        data_length=constructor_calldata_size,
    );
    let (contract_address_before_modulo) = hash_finalize(hash_state_ptr=hash_state_ptr);
    let (contract_address) = normalize_address(addr=contract_address_before_modulo);

    return (contract_address=contract_address);
