### Title
Missing Registration Check in `initTransfer1155` Allows Permanent Locking of ERC1155 Tokens — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`initTransfer1155` accepts any `(tokenAddress, tokenId)` ERC1155 pair without verifying that the pair has been previously registered via `logMetadata1155`. Because `finTransfer` on the EVM side and `fin_transfer_callback` on the NEAR side both require the token to be registered, any ERC1155 transfer initiated for an unregistered token results in permanently locked funds with no recovery path.

### Finding Description

The EVM bridge exposes two separate ERC1155 functions:

1. `logMetadata1155` — registers a `(tokenAddress, tokenId)` pair by populating `multiTokens[deterministicToken]` and emitting a `LogMetadata` event that the NEAR side processes to register the token and its decimals.
2. `initTransfer1155` — accepts an ERC1155 transfer and emits `InitTransfer` with the computed `deterministicToken` address.

`initTransfer1155` performs only a fee check and then immediately pulls the ERC1155 tokens from the caller: [1](#0-0) 

There is no check that `multiTokens[deterministicToken].tokenAddress != address(0)`, i.e., that `logMetadata1155` was called first. The `multiTokens` mapping is only populated by `logMetadata1155`: [2](#0-1) 

When `finTransfer` is later called on EVM to release the tokens, it branches on `multiTokens[payload.tokenAddress]`. For an unregistered token this entry is empty (`address(0)`), so the code falls through to `isBridgeToken` (false) and then attempts `IERC20(deterministicToken).safeTransfer(...)` against a non-existent ERC20 address, which reverts: [3](#0-2) 

On the NEAR side, `fin_transfer_callback` requires registered decimals for the token: [4](#0-3) 

An unregistered token has no decimals entry, so `fin_transfer_callback` panics with `TokenDecimalsNotFound`, making NEAR-side finalization impossible as well.

### Impact Explanation

A user who calls `initTransfer1155` for an unregistered `(tokenAddress, tokenId)` pair has their ERC1155 tokens transferred into the bridge contract. Neither the EVM side (no `multiTokens` entry → `safeTransfer` reverts) nor the NEAR side (no decimals → panic) can finalize the transfer. There is no refund or unlock mechanism in the EVM bridge for this case. The tokens are irrecoverably locked.

This matches: **Critical — Permanent freezing / irrecoverable lock of user funds in bridge flows.**

### Likelihood Explanation

The vulnerability is reachable by any unprivileged user. A user unfamiliar with the required two-step flow (first `logMetadata1155`, then `initTransfer1155`) will naturally call `initTransfer1155` directly. There is no on-chain enforcement of the ordering, no revert, and no warning. The ERC1155 `safeTransferFrom` succeeds, giving the user false confidence that the bridge accepted the transfer.

### Recommendation

Add a registration guard at the top of `initTransfer1155`:

```solidity
address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);
if (multiTokens[deterministicToken].tokenAddress == address(0)) {
    revert TokenNotRegistered();
}
```

This mirrors the pattern already used in `finTransfer` where `multiToken.tokenAddress` is checked before dispatching the ERC1155 release.

### Proof of Concept

1. Deploy the EVM `OmniBridge` contract.
2. Obtain any ERC1155 token contract and mint tokens to an EOA.
3. **Do not** call `logMetadata1155` for the token — `multiTokens[deterministicToken]` remains zeroed.
4. Call `initTransfer1155(tokenAddress, tokenId, amount, 0, 0, "recipient.near", "")`.
   - `safeTransferFrom` succeeds; tokens are now held by the bridge.
   - `InitTransfer` event is emitted with `deterministicToken`.
5. Attempt to finalize on NEAR: `fin_transfer_callback` panics with `TokenDecimalsNotFound` because the token was never registered via `logMetadata1155`.
6. Attempt to call `finTransfer` on EVM with `payload.tokenAddress = deterministicToken`:
   - `multiTokens[deterministicToken].tokenAddress == address(0)` → skips ERC1155 branch.
   - `isBridgeToken[deterministicToken] == false` → skips bridge-token branch.
   - `IERC20(deterministicToken).safeTransfer(...)` → reverts (no code at that address).
7. The ERC1155 tokens are permanently locked with no recovery path.
