### Title
Uint64 Underflow in `CheckEthGasConsume` Overflow Guard Enables Block Gas Limit Bypass тАФ (`ante/eth.go`)

---

### Summary

The overflow guard at `ante/eth.go:152` uses uint64 subtraction `math.MaxInt64 - gasLimit` that wraps to a large value when `gasLimit > math.MaxInt64`, making the guard silently pass. A two-message batch can be crafted so the accumulated `gasWanted` wraps to zero, bypassing the subsequent `gasWanted > blockGasLimit` check and allowing EVM execution with an astronomically large per-message gas limit.

---

### Finding Description

In `CheckEthGasConsume`: [1](#0-0) 

```go
gasWanted := uint64(0)
// ...
gasLimit := msgEthTx.GetGas()           // uint64, no upper-bound cap
if gasWanted > math.MaxInt64-gasLimit { // BUG: uint64 subtraction underflows when gasLimit > MaxInt64
    return ctx, fmt.Errorf("gasWanted(%d) + gasLimit(%d) overflow", gasWanted, gasLimit)
}
gasWanted += gasLimit
if gasWanted > blockGasLimit {
    return ctx, errorsmod.Wrapf(errortypes.ErrOutOfGas, ...)
}
```

`math.MaxInt64` is the untyped constant `9223372036854775807`. When `gasLimit` is a `uint64 > math.MaxInt64` (e.g., `9223372036854775808`), the expression `math.MaxInt64 - gasLimit` is evaluated in uint64 arithmetic and **underflows**:

```
9223372036854775807 - 9223372036854775808 = 18446744073709551615  (MaxUint64, wraps)
```

The guard `gasWanted > 18446744073709551615` is always `false`, so the check never fires for any `gasLimit > math.MaxInt64`.

**Two-message wrap-around attack** (let `blockGasLimit = 30_000_000`):

| Step | gasLimit | gasWanted after | Block-gas check |
|------|----------|-----------------|-----------------|
| msg[0] | `29_999_999` | `29_999_999` | `29_999_999 > 30_000_000` тЖТ false, passes |
| msg[1] | `18_446_744_073_709_521_617` (`MaxUint64 тИТ 30_000_000 + 2`) | `29_999_999 + 18_446_744_073_709_521_617 = 2^64` тЖТ **wraps to 0** | `0 > 30_000_000` тЖТ false, passes |

`gasWanted` wraps to `0`. The gas meter is then set: [2](#0-1) 

```go
newCtx := ctx.WithGasMeter(ethermint.NewInfiniteGasMeterWithLimit(gasWanted)) // limit = 0
```

The EVM then executes msg[1] with its raw `gasLimit` of `~1.8├Ч10┬╣тБ╣`, effectively unlimited.

**Fee check does not save you when `effectiveGasPrice = 0`:** [3](#0-2) 

The fee deduction loop runs per-message *after* the gas-limit checks. When `baseFee = 0` and `gasTipCap = gasFeeCap = 0`, `effectiveGasPrice = 0`, so `fee = gasLimit ├Ч 0 = 0` and `DeductTxCostsFromUserBalance` succeeds regardless of the enormous gas limit.

`GetGas()` returns the raw Ethereum `uint64` gas field with no upper-bound cap: [4](#0-3) 

`ValidateBasic` for `MsgEthereumTx` imposes no gas-limit ceiling: [5](#0-4) 

`BlockGasLimit` returns `math.MaxUint64` when `maxGas == -1` (no block gas limit configured), which is a common deployment choice: [6](#0-5) 

---

### Impact Explanation

- The ante handler's block-gas-limit guard is bypassed entirely.
- The EVM executes with a per-message gas limit of up to `math.MaxUint64`, allowing a looping contract to consume unbounded CPU on every validator node.
- `gasWanted = 0` is committed to the infinite gas meter, so the block-level gas accounting under-counts the transaction, allowing additional transactions to fill the block beyond its intended capacity.
- Combined with a pre-deployed infinite-loop contract and `gasPrice = 0`, this is a **chain-halt / resource-exhaustion** path reachable by any unprivileged account.

---

### Likelihood Explanation

**Conditions required:**
1. `baseFee = 0` тАФ possible when the fee market is disabled or `MinGasPrice = 0` (common in testnets and some production chains).
2. `gasPrice = 0` тАФ allowed when `minGasPrice = 0` (passes `CheckEthMempoolFee` and `CheckEthMinGasPrice`).
3. No gas-limit ceiling in `ValidateEthBasic` тАФ I was unable to read `ante/interfaces/setup.go` fully, so this remains a partial uncertainty; however, no cap is present in `ValidateBasic` or `GetGas`.

Conditions 1 and 2 are satisfied on any chain with fee market disabled. Condition 3 is the only open question.

---

### Recommendation

Replace the flawed guard with a correct uint64 overflow check:

```go
// Option A: explicit pre-check
if gasLimit > math.MaxInt64 || gasWanted > math.MaxInt64-gasLimit {
    return ctx, fmt.Errorf("gasWanted(%d) + gasLimit(%d) overflow", gasWanted, gasLimit)
}

// Option B: standard uint64 overflow idiom
if gasWanted+gasLimit < gasWanted {
    return ctx, fmt.Errorf("gasWanted(%d) + gasLimit(%d) overflow", gasWanted, gasLimit)
}
```

Additionally, add an explicit cap `gasLimit <= math.MaxInt64` in `ValidateBasic` / `ValidateEthBasic` to reject oversized gas limits at the earliest possible point.
