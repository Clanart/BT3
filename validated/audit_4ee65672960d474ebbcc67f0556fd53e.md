### Title
Concurrent Fee Transfer Completion Overwrites Execute-Phase Sequencer Balance Write, Corrupting State Diff and State Root — (`File: crates/blockifier/src/concurrency/fee_utils.rs`)

### Summary

In concurrent execution mode, when a transaction's execute phase explicitly transfers fee tokens to the sequencer address, the `complete_fee_transfer_flow` / `add_fee_to_sequencer_balance` path at commit time overwrites the execute-phase sequencer balance write in both the transaction's `state_diff` and the versioned state with a value that ignores the execute-phase transfer amount. The resulting state diff is wrong, which propagates to the block commitment, state root, and proof.

### Finding Description

The concurrent fee transfer design splits the fee transfer into two phases:

**Phase 1 — `concurrency_execute_fee_transfer` (during `execute_tx`):** [1](#0-0) 

A nested `transfer_state` is created, the sequencer balance initial value is forced to `Felt::ZERO`, the ERC-20 `transfer` call is executed, and then the sequencer balance writes are **removed** from the cache before committing back to the outer state. The outer `transactional_state` therefore retains any execute-phase writes to the sequencer balance (e.g., from a user-initiated `transfer(sequencer_address, amount)` call), but not the fee-transfer write.

**Phase 2 — `complete_fee_transfer_flow` → `add_fee_to_sequencer_balance` (during `commit_tx`):** [2](#0-1) [3](#0-2) 

At commit time, `add_fee_to_sequencer_balance` reads the sequencer balance from `tx_versioned_state.pin_version(tx_index)`, which returns the state **before** this transaction's own writes. It computes:

```
new_balance = pre_tx_committed_balance + fee
```

It then calls `state_diff.storage.insert(...)` with this value, **unconditionally overwriting** any existing entry for the sequencer balance key in `execution_output.state_diff`. It also calls `state.apply_writes(...)` which overwrites the versioned state at `tx_index`.

**The collision:** If the execute phase wrote `pre_tx_committed_balance + transfer_amount` to the sequencer balance key (because the user transferred tokens to the sequencer), that write is present in `execution_output.state_diff`. The `insert` call in `add_fee_to_sequencer_balance` replaces it with `pre_tx_committed_balance + fee`, silently discarding `transfer_amount`.

The correct final value should be `pre_tx_committed_balance + transfer_amount + fee`.

The `fill_sequencer_balance_reads` assertion only guards the fee-transfer call info's read values (which are always zero due to Phase 1), and does not detect this overwrite: [4](#0-3) 

The commit path in `worker_logic.rs` passes `execution_output.state_diff` directly to `complete_fee_transfer_flow`: [5](#0-4) 

### Impact Explanation

The corrupted `state_diff` flows directly into block commitment:

1. `BlockExecutionArtifacts::new` converts `CommitmentStateDiff` → `ThinStateDiff` → `calculate_block_commitments`, which computes the state diff commitment and `PartialBlockHashComponents`. [6](#0-5) 

2. The wrong sequencer balance storage value is committed to the Patricia trie, producing a wrong global state root.

3. The wrong state root propagates into the block hash, the SNOS proof input/output, and any downstream proof facts.

4. The sequencer's actual on-chain balance is understated by `transfer_amount` — the user's tokens are deducted from their balance but the sequencer does not receive them in the committed state. This is a direct economic loss.

This matches: **Critical — Wrong state/storage value from blockifier/execution logic** and **Critical — Incorrect balance with economic impact**.

### Likelihood Explanation

- Concurrent execution mode (`concurrency_mode = true`) is the production path for high-throughput block building via `WorkerExecutor`.
- Any unprivileged user can craft an invoke transaction whose calldata calls `transfer(sequencer_address, amount)` on the fee token contract. The sequencer address is public.
- The test `test_concurrency_execute_fee_transfer` Case 2 explicitly constructs this scenario and confirms the execute-phase write is present in the state cache before commit — but the test does not call `commit_tx`, so the overwrite is never observed. [7](#0-6) 

### Recommendation

In `add_fee_to_sequencer_balance`, before inserting the new sequencer balance into `state_diff`, check whether the execute phase already wrote to the sequencer balance key. If so, use that write as the base rather than the pre-tx committed balance:

```rust
// Instead of reading pre-tx balance from committed state:
let base_low = state_diff.storage
    .get(&(fee_token_address, sequencer_balance_key_low))
    .copied()
    .unwrap_or(low);  // fall back to committed balance only if no execute-phase write
```

Alternatively, apply the fee increment on top of whatever value is already in `state_diff` for the sequencer balance key, rather than recomputing from the committed state.

### Proof of Concept

1. Concurrent block execution begins. Sequencer balance = `S`.
2. Transaction T is executed in concurrent mode. Its execute phase calls `transfer(sequencer_address, 100)` on the fee token. The execute-phase state diff records `sequencer_balance_key_low → S + 100`. This write is applied to the versioned state at `tx_index`.
3. `concurrency_execute_fee_transfer` runs the fee transfer with a forced zero sequencer balance, then removes the fee-transfer sequencer balance writes. The outer state diff retains `S + 100`.
4. At commit time, `complete_fee_transfer_flow` reads the sequencer balance from `pin_version(tx_index)` = `S` (pre-tx committed state). It computes `new_balance = S + fee` and calls `state_diff.storage.insert(seq_key_low, S + fee)`, overwriting `S + 100`. It also calls `state.apply_writes` with `S + fee`, overwriting the versioned state.
5. The final state diff has `sequencer_balance_key_low → S + fee` instead of the correct `S + 100 + fee`.
6. `BlockExecutionArtifacts::new` builds the block commitment from this wrong state diff, producing a wrong state root and wrong block hash. The sequencer is missing `100` tokens in the committed state.

pub fn complete_fee_transfer_flow(
    tx_context: &TransactionContext,
    tx_execution_info: &mut TransactionExecutionInfo,
    state_diff: &mut StateMaps,
    state: &mut impl UpdatableState,
    tx: &Transaction,
) {
    if tx_context.is_sequencer_the_sender() {
        // When the sequencer is the sender, we use the sequential (full) fee transfer.
        return;
    }

    if let Some(fee_transfer_call_info) = tx_execution_info.fee_transfer_call_info.as_mut() {
        let sequencer_balance = state
        .get_fee_token_balance(
            tx_context.block_context.block_info.sequencer_address,
            tx_context.fee_token_address()
        )
        // TODO(barak, 01/07/2024): Consider propagating the error.
        .unwrap_or_else(|error| {
            panic!(
                "Access to storage failed. Probably due to a bug in Papyrus. {error:?}: {error}"
            )
        });

        // Fix the transfer call info.
        fill_sequencer_balance_reads(fee_transfer_call_info, sequencer_balance);
        // Update the balance.
        add_fee_to_sequencer_balance(
            tx_context.fee_token_address(),
            state,
            tx_execution_info.receipt.fee,
            &tx_context.block_context,
            sequencer_balance,
            tx_context.tx_info.sender_address(),
            state_diff,
        );
    } else {
        // Sanity check.
        match tx {
            Transaction::Account(tx) => assert!(
                !tx.execution_flags.charge_fee || tx_execution_info.receipt.fee == Fee(0),
                "Transaction with no fee transfer info must not enforce a fee charge."
            ),
            // No fee transfer info for L1 handler transactions.
            Transaction::L1Handler(_) => {}
        };
    }
}

pub fn add_fee_to_sequencer_balance(
    fee_token_address: ContractAddress,
    state: &mut impl UpdatableState,
    actual_fee: Fee,
    block_context: &BlockContext,
    sequencer_balance: (Felt, Felt),
    sender_address: ContractAddress,
    state_diff: &mut StateMaps,
) {
    assert_ne!(
        sender_address, block_context.block_info.sequencer_address,
        "The sender cannot be the sequencer."
    );
    let (low, high) = sequencer_balance;
    let sequencer_balance_low_as_u128 =
        low.to_u128().expect("sequencer balance low should be u128");
    let sequencer_balance_high_as_u128 =
        high.to_u128().expect("sequencer balance high should be u128");
    let (new_value_low, overflow_low) = sequencer_balance_low_as_u128.overflowing_add(actual_fee.0);
    let (new_value_high, overflow_high) =
        sequencer_balance_high_as_u128.overflowing_add(overflow_low.into());
    assert!(
        !overflow_high,
        "The sequencer balance overflowed when adding the fee. This should not happen."
    );
    let (sequencer_balance_key_low, sequencer_balance_key_high) =
        get_sequencer_balance_keys(block_context);
    let writes = StateMaps {
        storage: HashMap::from([
            ((fee_token_address, sequencer_balance_key_low), Felt::from(new_value_low)),
            ((fee_token_address, sequencer_balance_key_high), Felt::from(new_value_high)),
        ]),
        ..StateMaps::default()
    };

    // Modify state_diff to accurately reflect the post tx-execution state, after fee transfer to
    // the sequencer. We assume that a non-sequencer sender cannot reduce the sequencer's
    // balance—only increases are possible.

    if sequencer_balance_high_as_u128 != new_value_high {
        // Update the high balance only if it has changed.
        state_diff
            .storage
            .insert((fee_token_address, sequencer_balance_key_high), Felt::from(new_value_high));
    }

    if sequencer_balance_low_as_u128 != new_value_low {
        // Update the low balance only if it has changed.
        state_diff
            .storage
            .insert((fee_token_address, sequencer_balance_key_low), Felt::from(new_value_low));
    }
    state.apply_writes(&writes, &ContractClassMapping::default());
}
