**Step 1: Parser accepts any-length proof for v2**

The `parse` function for v2 (`version == 1`) reads `proof` as a raw `Bytes` with no minimum-length constraint: [1](#0-0) 

The only v2-specific validation is that exactly one of `pool_public_key`/`pool_contract_puzzle_hash` is set (line 322–324). A 1-byte proof passes the parser cleanly.

**Step 2: `quality_string()` computes `k_size` via integer truncation** [2](#0-1) 

With `proof.len() = 1`: `1 * 8 / 128 = 0`, so `k_size = 0`. The call becomes `chia_pos2::quality_string_from_proof(&plot_id, 0, strength, proof)`. With `k_size=0`, it returns `None`. `quality_string()` propagates `None` via `.map()`.

**Step 3: `update_digest` panics unconditionally on `None`** [3](#0-2) 

```rust

    .expect("internal error. Can't compute hash of invalid ProofOfSpace")
    .update_digest(digest);
```

`update_digest` is an infallible function (returns `()`). When `quality_string()` returns `None`, `.expect()` panics. There is no `Result` return, no error propagation — it is a hard process abort.

**Step 4: Is `update_digest` reachable on unvalidated input?**

`update_digest` is the `Streamable` trait's hashing method. It is called whenever a `ProofOfSpace` is hashed as part of a containing structure (e.g., `RewardChainBlock`, `HeaderBlock`). In Chia's consensus pipeline, block hashes are computed during block processing — and if `update_digest` is invoked before proof-of-space validity is confirmed (which is the normal case for initial block hash verification), a single crafted network message triggers the panic.

**Conclusion**

The vulnerability is real and concrete:

- The parser imposes **no minimum length** on the `proof` field for v2 proofs.
- `quality_string()` silently computes `k_size=0` for short proofs and returns `None`.
- `update_digest` calls `.expect()` on that `None`, causing a **deterministic panic** (process abort).
- Any node that attempts to hash a block containing such a `ProofOfSpace` crashes.

### Title
Deterministic panic in `ProofOfSpace::update_digest` via short v2 proof field — (`crates/chia-protocol/src/proof_of_space.rs`)

### Summary
A v2 `ProofOfSpace` with a `proof` field shorter than 16 bytes passes the streamable parser but causes `update_digest` to panic unconditionally via `.expect()`, enabling a single malformed block to halt any node that processes it.

### Finding Description
`ProofOfSpace::parse` for version 1 (v2 proofs) reads the `proof` field as a raw `Bytes` with no minimum-length check. [4](#0-3) 

`quality_string()` derives `k_size` as `(proof.len() * 8 / 128) as u8`. [5](#0-4) 

For any `proof.len() < 16`, integer truncation yields `k_size = 0`. `chia_pos2::quality_string_from_proof` returns `None` for this degenerate input, so `quality_string()` returns `None`.

`update_digest` then calls `.expect()` on that `None`: [6](#0-5) 

This is an unconditional panic — no `Result`, no recovery.

### Impact Explanation
`update_digest` is the `Streamable` hashing method invoked whenever a `ProofOfSpace` is hashed as part of a parent structure (`RewardChainBlock`, `HeaderBlock`, etc.). Any consensus node that attempts to compute the hash of a block containing such a crafted `ProofOfSpace` will crash deterministically. A single malformed block broadcast to the network causes a **chain halt** across all nodes that process it before proof-of-space validation rejects it.

### Likelihood Explanation
The exploit requires only crafting a valid-looking v2 `ProofOfSpace` serialization with a 1-byte `proof` field. No keys, no privileged access, no special timing — just a well-formed byte sequence that the parser accepts. The attacker submits it as part of a block or unfinished block message.

### Recommendation
1. **In `parse`**: Enforce a minimum proof length for v2 proofs (e.g., `proof.len() >= 16`, or validate that `proof.len() * 8 / 128 >= MIN_K`) and return `Err(Error::InvalidPoS)` if violated.
2. **In `update_digest`**: Replace `.expect()` with a pattern that cannot panic — either validate proof length in `parse` (preferred) or make `update_digest` return a `Result` (larger refactor).
3. The invariant "only valid `ProofOfSpace` objects reach `update_digest`" must be enforced at parse time, not assumed.
