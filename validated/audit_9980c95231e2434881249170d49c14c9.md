### Title
Cross-Chain Replay of `deployToken` Signature Enables Unauthorized Token Deployment on Any EVM Chain ŌĆö (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`)

### Summary

The Borsh-encoded message hashed and signed by the NEAR MPC signer for `deployToken`/`deploy_token` operations does not include the destination chain ID. Because the same `nearBridgeDerivedAddress` (derived from the single NEAR MPC key) is used across all EVM deployments (Ethereum, Arbitrum, Base, BNB, Polygon, HyperEvm, Abstract) and StarkNet, a valid deployment signature for token X on chain A is cryptographically identical to a valid deployment signature for token X on chain B. An unprivileged attacker who observes a signed `LogMetadataEvent` on NEAR can replay the signature on any other supported chain, deploying the token without NEAR bridge authorization and permanently blocking the NEAR bridge from registering the correct token address for that chain.

### Finding Description

**Root cause ŌĆö missing chain ID in the `deployToken` signed payload.**

In `OmniBridge.sol::deployToken`, the message that is hashed and verified against `nearBridgeDerivedAddress` is:

```
PayloadType::Metadata | token | name | symbol | decimals
```

No chain identifier is included. [1](#0-0) 

Compare this with `finTransfer`, which correctly binds the message to the destination chain by including `omniBridgeChainId` twice (for the token address and the recipient address fields): [2](#0-1) 

The same omission exists on StarkNet. `MetadataPayload::to_borsh()` serializes only `PayloadType::Metadata | token | name | symbol | decimals` ŌĆö no chain ID: [3](#0-2) 

Again, contrast with `TransferMessagePayload::to_borsh()`, which correctly includes `chain_id` at both the token and recipient positions: [4](#0-3) 

On the NEAR side, `MetadataPayload` (the struct that is Borsh-serialized and keccak-hashed before being sent to the MPC signer) contains no chain field: [5](#0-4) 

The NEAR bridge signs this payload with a single MPC key (`mpc_signer`) and a single path (`"bridge-1"`) for all destination chains: [6](#0-5) 

**Attack path:**

1. The NEAR bridge calls `log_metadata` for token `foo.near`, which triggers the MPC signer to produce a signature over `keccak256(borsh(MetadataPayload{Metadata, "foo.near", "Foo", "FOO", 18}))`.
2. The signature is emitted in a `LogMetadataEvent` on NEAR ŌĆö publicly observable.
3. An attacker extracts the signature and calls `deployToken(signature, {token:"foo.near", name:"Foo", symbol:"FOO", decimals:18})` on Arbitrum (or any other supported EVM chain, or StarkNet) before the NEAR bridge relayer does.
4. The token is deployed on Arbitrum. `nearToEthToken["foo.near"]` and `isBridgeToken[newProxy]` are set.
5. When the NEAR bridge relayer later calls `deployToken` on Arbitrum, the check `!isBridgeToken[nearToEthToken[metadata.token]]` reverts with `"ERR_TOKEN_EXIST"`. [7](#0-6) 
6. The NEAR bridge's `deploy_token_callback` is never executed for Arbitrum, so the token address for `foo.near` on Arbitrum is never registered in NEAR bridge state.
7. `sign_transfer` cannot resolve the token address for Arbitrum, so no `finTransfer` signatures can be produced for this token on Arbitrum. [8](#0-7) 

### Impact Explanation

**Unauthorized token deployment** on any chain where the OmniBridge is deployed, using a signature that was only authorized for a different chain. This constitutes a signature/domain-separation bypass enabling unauthorized token deployment.

**Permanent freezing of bridging** for the affected token on the replayed chain: the NEAR bridge cannot re-deploy the token (already exists), cannot register the token address in its own state (deployment callback never fires), and therefore cannot sign `finTransfer` messages for that token on that chain. Users are permanently unable to bridge that token to the affected chain without privileged admin recovery (`addCustomToken` or equivalent).

Severity: **High** ŌĆö matches "Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized token deployment" and "Permanent freezing / irrecoverable lock of user or protocol funds in bridge or token flows."

### Likelihood Explanation

- Requires no privileged access. Any observer of NEAR chain events can extract a `LogMetadataEvent` signature.
- The NEAR bridge emits these signatures publicly for every token it registers.
- The attacker only needs to front-run the relayer on the target chain, which is straightforward given that NEAR event finality precedes EVM relayer submission.
- Multiple EVM chains (Eth, Arb, Base, BNB, Pol, HyperEvm, Abs) and StarkNet all share the same `nearBridgeDerivedAddress`, so a single signature is replayable across all of them simultaneously.

### Recommendation

Include the destination chain ID in the `MetadataPayload` that is signed by the NEAR MPC signer, mirroring the existing pattern used by `TransferMessagePayload`:

- In `near/omni-types/src/lib.rs`, add a `destination_chain: ChainKind` field to `MetadataPayload`.
- In `near/omni-bridge/src/lib.rs` `log_metadata_callback`, populate `destination_chain` from the token's registered destination.
- In `evm/src/omni-bridge/contracts/OmniBridge.sol` `deployToken`, include `bytes1(omniBridgeChainId)` in the `borshEncoded` message before hashing.
- In `starknet/src/bridge_types.cairo` `MetadataPayload::to_borsh()`, append the `omni_bridge_chain_id` byte, matching the pattern in `TransferMessagePayload::to_borsh()`.

### Proof of Concept

```
// 1. Observe NEAR LogMetadataEvent for token "foo.near" on Ethereum deployment:
//    signature = 0xABCD...  (valid over keccak256(borsh(Metadata|"foo.near"|"Foo"|"FOO"|18)))

// 2. Call deployToken on Arbitrum OmniBridge (same nearBridgeDerivedAddress):
arbitrumOmniBridge.deployToken(
    signature,                          // same signature, no chain binding
    MetadataPayload({
        token: "foo.near",
        name: "Foo",
        symbol: "FOO",
        decimals: 18
    })
);
// ŌåÆ succeeds; nearToEthToken["foo.near"] = <new proxy>; isBridgeToken[<new proxy>] = true

// 3. NEAR bridge relayer calls deployToken on Arbitrum:
arbitrumOmniBridge.deployToken(signature, payload);
// ŌåÆ reverts: "ERR_TOKEN_EXIST"

// 4. NEAR bridge never registers foo.near's Arbitrum address.
//    sign_transfer for foo.near ŌåÆ Arbitrum panics: BridgeError::FailedToGetTokenAddress
//    All foo.near ŌåÆ Arbitrum transfers are permanently blocked.
```
