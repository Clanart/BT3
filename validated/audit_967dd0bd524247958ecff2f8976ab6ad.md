### Title
`MerkleBlob::batch_insert` Bypasses Duplicate Key/Hash Validation, Corrupting Merkle Tree Roots and Proofs - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. When a batch containing repeated `KeyId` or `Hash` values is supplied, duplicate leaf nodes are silently written into the flat-buffer tree, the `block_status_cache` is silently overwritten, and the resulting Merkle root and all proofs of inclusion derived from it are incorrect. The function is exposed as a public Python binding (`py_batch_insert`) and is the primary bulk-insertion path for the DataLayer.

---

### Finding Description

`MerkleBlob::insert` guards against duplicates at the top of its body:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` calls `insert` (with those guards) only for the **last two** items popped from the input vector when the tree has ≤ 1 existing leaf. Every other item — and **all** items when the tree already has ≥ 2 leaves — is written directly via `insert_entry_to_blob`:

```rust
// lines 587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block {
        ...
        node: Node::Leaf(LeafNode { parent: Parent(None), hash, key, value }),
    };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which uses `HashMap::insert` and silently overwrites any existing entry for the same key or hash:

```rust
// lines 1013-1030
fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
    ...
    match block.node {
        Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
        ...
    }
    Ok(())
}

// lines 188-193
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // silently overwrites
    self.leaf_hash_to_index.insert(leaf.hash, index); // silently overwrites
}
```

After a duplicate is inserted:

- The blob contains **two** leaf nodes with the same `KeyId` (or `Hash`).
- The cache points only to the second one; the first is orphaned but still part of the tree structure.
- `calculate_lazy_hashes` propagates hashes up through a tree that contains both leaves, producing a root that encodes the duplicate.
- `get_proof_of_inclusion` builds a proof path from the cache-tracked leaf, which is structurally valid for the corrupted tree but does not represent the true committed state.

`check_integrity` would detect the mismatch (`IntegrityKeyToIndexCacheLength` / `IntegrityLeafHashToIndexCacheLength`), but it is only enabled in test builds (`check_integrity_on_drop: cfg!(test)`); production builds never call it automatically.

---

### Impact Explanation

A caller who supplies a batch with a repeated `KeyId` or `Hash` causes the DataLayer Merkle tree to commit an incorrect root hash. Any proof of inclusion generated from that tree is structurally valid against the wrong root, meaning:

- A key-value pair that was inserted twice will produce a proof that verifies against a root that does not represent the intended store state.
- A verifier using the committed root will accept proofs that prove incorrect state, or reject proofs that prove correct state, depending on which leaf the cache retained.
- The tree root stored on-chain diverges from the tree root a clean re-computation would produce, constituting committed state corruption.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

---

### Likelihood Explanation

`batch_insert` is the primary bulk-insertion path and is exposed directly as a Python binding (`py_batch_insert`, line 1503). Any DataLayer client code that passes user-controlled key-value pairs to `batch_insert` without pre-deduplicating them triggers the bug. Because the DataLayer is designed to accept external data submissions, an unprivileged participant can supply a batch with a repeated key at negligible cost. No privileged role, key material, or network-level access is required.

---

### Recommendation

Add duplicate-key and duplicate-hash checks at the start of `batch_insert` (or inside the fast-path loop) before calling `insert_entry_to_blob`:

```rust
for ((key, value), hash) in &keys_values_hashes {
    if self.block_status_cache.contains_key(*key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(hash) {
        return Err(Error::HashAlreadyPresent());
    }
}
```

Alternatively, deduplicate the input vector before processing, or refactor the fast path to reuse the validated `insert` call for all items. Additionally, consider enabling `check_integrity` in production (or at least on `MerkleBlob::new` when loading from an existing blob) to catch any corruption that reaches the store.

---

### Proof of Concept

```rust
use chia_datalayer::{Hash, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn main() {
    // Tree already has 2 leaves → the entire batch bypasses duplicate checks
    let mut blob = MerkleBlob::new(vec![]).unwrap();
    let h = |n: u8| Hash(Bytes32::new([n; 32]));

    // Seed with 2 leaves so leaf_count > 1
    blob.insert(KeyId(1), ValueId(1), &h(1),
        chia_datalayer::InsertLocation::Auto {}).unwrap();
    blob.insert(KeyId(2), ValueId(2), &h(2),
        chia_datalayer::InsertLocation::Auto {}).unwrap();

    // batch_insert with a duplicate KeyId(3) — no error is returned
    blob.batch_insert(vec![
        ((KeyId(3), ValueId(10)), h(3)),
        ((KeyId(3), ValueId(99)), h(4)), // same key, different value & hash
    ]).unwrap(); // succeeds silently

    blob.calculate_lazy_hashes().unwrap();

    // The blob now contains two leaf nodes with KeyId(3).
    // The cache only tracks the second one (ValueId(99)).
    // The root hash encodes both leaves → corrupted committed state.
    // check_integrity() would return IntegrityKeyToIndexCacheLength error.
    blob.check_integrity().unwrap_err(); // demonstrates corruption
}
```

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }

    #[pyo3(name = "batch_insert")]
    pub fn py_batch_insert(
        &mut self,
        keys_values: Vec<(KeyId, ValueId)>,
        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
            ))?;
        }

        self.batch_insert(zip(keys_values, hashes).collect())?;

        Ok(())
    }
