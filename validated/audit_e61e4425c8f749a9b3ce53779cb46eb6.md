### Title
Missing Block-Number Guard in `delete_declared_classes` Allows Revert to Erase Legitimately-Declared Sierra Class Content — (`File: crates/apollo_storage/src/state/mod.rs`)

### Summary

`revert_state_diff` calls two sibling helpers to clean up Sierra (Cairo 1) class data. `delete_declared_classes_block` correctly guards deletion with `if class_block_entry == block_number`, protecting classes declared in earlier blocks. `delete_declared_classes`, which removes the actual class content from `declared_classes_table`, has **no such guard** — it unconditionally deletes any class hash that appears in the reverted block's `class_hash_to_compiled_class_hash` map, regardless of which block originally declared it. This is the direct sequencer analog of the unguarded `recoverERC20()` in `MultiMerkleDistributor.sol`.

### Finding Description

During `revert_state_diff`, two helpers run back-to-back:

**`delete_declared_classes_block`** — has the guard:
```rust
if class_block_entry == block_number {
    declared_classes_block_table.delete(txn, class_hash)?;
    deleted_data.push(*class_hash);
}
``` [1](#0-0) 

**`delete_declared_classes`** — no guard at all:
```rust
for class_hash in thin_state_diff.class_hash_to_compiled_class_hash.keys() {
    let Some(contract_class_location) = declared_classes_table.get(txn, class_hash)? else {
        continue;
    };
    declared_classes_table.delete(txn, class_hash)?;  // unconditional
}
``` [2](#0-1) 

Both are called from `revert_state_diff` in sequence: [3](#0-2) 

The `deprecated_declared_classes` path has the guard in both its helpers (`delete_deprecated_declared_classes_block` and `delete_deprecated_declared_classes`): [4](#0-3) [5](#0-4) 

The `DeclaredClassesTable` stores only `ClassHash → LocationInFile` (no block number in the value), while `DeclaredClassesBlockTable` stores `ClassHash → BlockNumber` (the first block that declared the class): [6](#0-5) 

`append_state_diff` only skips updating `declared_classes_block_table` if the class is already there — it does **not** reject a state diff that re-references an already-declared class hash: [7](#0-6) 

### Impact Explanation

**Attack scenario:**

1. Block M: Class `C` is legitimately declared. `declared_classes_block_table[C] = M`, `declared_classes_table[C] = <location>`.
2. Block N (N > M): A peer-supplied state diff includes `C` in `class_hash_to_compiled_class_hash`. `append_state_diff` accepts it silently (skips updating the block table since `C` is already there). `declared_classes_table[C]` may be overwritten by `append_classes`.
3. Block N is reverted (chain reorg or sync correction):
   - `delete_declared_classes_block`: sees `class_block_entry = M ≠ N`, **skips** — `declared_classes_block_table[C] = M` survives.
   - `delete_declared_classes`: sees `C` in the state diff, finds it in `declared_classes_table`, **deletes it unconditionally**.
4. Post-revert state: `declared_classes_block_table[C] = M` (intact), `declared_classes_table[C]` = **gone**.

Any subsequent call to `get_class_definition_at` for class `C` at a state after block M will:
- Find `block_number = M` in `declared_classes_block_table`
- Fail to find `C` in `declared_classes_table`
- Return `StorageError::DBInconsistency` (since the class marker is past M) [8](#0-7) 

This corrupts the class storage: the class hash is permanently inaccessible, causing execution failures for every transaction that deploys or calls a contract using class `C`. The state root derived from the Patricia trie will be wrong for all subsequent blocks.

### Likelihood Explanation

The P2P sync path accepts state diffs from peers. The `unite_state_diffs` function only checks for duplicates **within** a single block's state diff, not against previously declared classes: [9](#0-8) 

A malicious peer can craft a state diff for the chain tip that re-references a class hash declared in any earlier block. `append_state_diff` will accept it without error. A subsequent revert (triggered by a normal chain reorg) then fires the bug. No privileged access is required beyond being a connected peer.

### Recommendation

Apply the same block-number guard to `delete_declared_classes` that already exists in `delete_declared_classes_block`. Consult `declared_classes_block_table` before deleting from `declared_classes_table`:

```rust
fn delete_declared_classes<'env>(
    txn: &'env DbTransaction<'env, RW>,
    thin_state_diff: &ThinStateDiff,
    declared_classes_table: &'env DeclaredClassesTable<'env>,
    declared_classes_block_table: &'env DeclaredClassesBlockTable<'env>,
    block_number: BlockNumber,
    file_handlers: &FileHandlers<RW>,
) -> StorageResult<IndexMap<ClassHash, SierraContractClass>> {
    let mut deleted_data = IndexMap::new();
    for class_hash in thin_state_diff.class_hash_to_compiled_class_hash.keys() {
        // Only delete if this block is the one that first declared the class.
        let Some(declared_at) = declared_classes_block_table.get(txn, class_hash)? else {
            continue;
        };
        if declared_at != block_number {
            continue;  // guard: protect classes declared in earlier blocks
        }
        let Some(contract_class_location) = declared_classes_table.get(txn, class_hash)? else {
            continue;
        };
        deleted_data.insert(*class_hash, file_handlers.get_contract_class_unchecked(contract_class_location)?);
        declared_classes_table.delete(txn, class_hash)?;
    }
    Ok(deleted_data)
}
```

Additionally, `append_state_diff` should reject state diffs that include a class hash already present in `declared_classes_block_table` (i.e., already declared in a prior block), mirroring the duplicate-within-block check in `unite_state_diffs`.

### Proof of Concept

```
// Setup
Block 0: append_state_diff({class_hash_to_compiled_class_hash: {C: H0}})
         append_classes([(C, sierra_class)])
// State: declared_classes_block_table[C] = 0, declared_classes_table[C] = <loc>

Block 1: append_state_diff({class_hash_to_compiled_class_hash: {C: H0}})
         // accepted silently; declared_classes_block_table[C] stays = 0
// State: declared_classes_block_table[C] = 0, declared_classes_table[C] = <loc>

// Revert block 1
revert_state_diff(BlockNumber(1)):
  delete_declared_classes_block: class_block_entry=0 != 1 → SKIP (correct)
  delete_declared_classes:       C found in declared_classes_table → DELETE (bug)

// Post-revert
declared_classes_block_table[C] = 0   // still present
declared_classes_table[C]             // GONE

// Any execution using class C:
get_class_definition_at(state_after_block_0, C)
  → finds block_number=0 in declared_classes_block_table
  → declared_classes_table.get(C) = None
  → class marker > 0
  → StorageError::DBInconsistency  ← state permanently corrupted
```
