Audit Report

## Title
Router `pay()` WETH Branch Silently Consumes Stranded Native ETH Without Per-Caller Accounting — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` helper in `PeripheryPayments.sol` uses `address(this).balance` as the ETH source for WETH payments without any per-caller accounting. ETH left on the router from a prior transaction (excess `msg.value` not reclaimed via `refundETH`) is freely consumed by the next caller's WETH swap, giving that caller a free trade at the prior user's expense.

## Finding Description
In `PeripheryPayments.pay()` (L73–84), when `token == WETH`:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);   // payer is never touched
}
```

The function reads the router's entire native ETH balance and, if it covers `value`, wraps and forwards it to the pool — the `payer` argument is completely ignored. There is no transient-storage or in-memory accounting that ties any portion of `address(this).balance` to the caller who deposited it via `msg.value`. [1](#0-0) 

The call path is:
- `exactInputSingle` (L67, `payable`) sets callback context with `msg.sender` as payer and calls `pool.swap()`.
- The pool fires `metricOmmSwapCallback`, which dispatches to `_justPayCallback` (L192–199).
- `_justPayCallback` calls `pay(_getTokenToPay(), _getPayer(), msg.sender, amount)`.
- Inside `pay()`, if `address(this).balance >= value`, the router wraps its own ETH — regardless of whether the current caller contributed any `msg.value`. [2](#0-1) 

The `receive()` guard (L32–33) only blocks ETH from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` in payable entry points. [3](#0-2) 

ETH strands naturally when a user sends `msg.value > amountIn` and omits the trailing `refundETH()` call. Once stranded, any subsequent WETH-input swap drains it.

## Impact Explanation
Direct loss of user principal. The victim's ETH is permanently transferred to the pool as payment for an attacker's trade. The attacker receives full swap output without spending any ETH or WETH. This is a concrete, quantifiable loss of funds meeting the Critical/High threshold under Sherlock rules.

## Likelihood Explanation
Medium. The precondition — ETH stranded on the router — arises naturally from two common patterns: (1) a user sends `msg.value` larger than `amountIn` to avoid partial-fill reverts, or (2) a `multicall` omits the trailing `refundETH` step. Both are realistic user behaviors. Once ETH is stranded, any observer can exploit it in the very next block with a zero-cost WETH swap requiring no special privileges.

## Recommendation
**Short term:** In the `nativeBalance >= value` branch, track per-call ETH credit in transient storage (alongside the existing payer/token context set by `_setNextCallbackContext`) and deduct from that credit rather than from the raw contract balance. Alternatively, verify that the current call's `msg.value` covers at least `value` before using native ETH.

**Long term:** Enforce an invariant that `address(this).balance == 0` at the start of every non-payable entry point, or revert in `pay()` when `address(this).balance > 0` and the current top-level call contributed zero `msg.value`.

## Proof of Concept
```
Block N:
  Alice calls exactInputSingle{value: 1000}(tokenIn=WETH, amountIn=500, ...)
  → pool.swap() fires metricOmmSwapCallback
  → _justPayCallback → pay(WETH, Alice, pool, 500)
      nativeBalance = 1000 >= 500
      router wraps 500 wei, sends WETH to pool  ← Alice's 500 wei consumed
      500 wei remains on router
  Alice does NOT call refundETH().

Block N+1:
  Bob calls exactInputSingle{value: 0}(tokenIn=WETH, amountIn=500, ...)
  → pool.swap() fires metricOmmSwapCallback
  → _justPayCallback → pay(WETH, Bob, pool, 500)
      nativeBalance = 500 >= 500
      router wraps 500 wei, sends WETH to pool  ← Alice's remaining 500 wei consumed
      Bob receives TOKEN output, pays nothing
```

Alice loses 500 wei permanently. Bob receives a full swap output at zero cost. A Foundry integration test can reproduce this by: (1) calling `exactInputSingle{value: 1000}` with `amountIn=500` from Alice without a subsequent `refundETH`, then (2) calling `exactInputSingle{value: 0}` with `amountIn=500` from Bob and asserting Bob receives output tokens while the router balance is zero.
