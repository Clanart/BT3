Audit Report

## Title
Unaccounted Router ETH Balance Consumed by Subsequent WETH Swaps — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` function reads `address(this).balance` as a shared, unaccounted pool when `token == WETH`. All swap entry-points are `payable` but none call `refundETH()` on exit, so any ETH sent with a non-WETH swap (or excess ETH with a WETH swap) is stranded on the router. A subsequent WETH swap from any caller silently consumes the stranded ETH, transferring the victim's principal to the pool on the attacker's behalf.

## Finding Description
`pay()` branches on `token == WETH` and reads the raw native balance without per-caller accounting: [1](#0-0) 

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses: [2](#0-1) 

It does not prevent ETH from arriving via `msg.value` in any payable call. All four swap entry-points are `payable` and none call `refundETH()` before returning: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

`multicall` is also `payable` and iterates via `delegatecall` with no enforced refund: [7](#0-6) 

**Stranding path 1 — non-WETH swap with `msg.value > 0`:** `pay()` takes the `else` branch (`safeTransferFrom`), never touching `msg.value`. ETH stays on the router.

**Stranding path 2 — WETH swap with `msg.value > value`:** `pay()` deposits exactly `value` ETH; the surplus remains on the router.

**Consumption path:** Any subsequent WETH swap triggers the `nativeBalance > 0` branch, wrapping and forwarding the stranded ETH to the pool, reducing the attacker's `safeTransferFrom` pull by exactly the stranded amount. [8](#0-7) 

`refundETH()` sends the *entire* router balance to `msg.sender`, not the caller's share, so it is not a safe recovery mechanism in a concurrent environment: [9](#0-8) 

## Impact Explanation
Direct, cross-caller loss of user principal (ETH) with no guaranteed recovery path. User A's ETH is transferred to the pool on User B's behalf; User B's wallet outflow is reduced by exactly the stranded amount. This is a fund-loss impact matching the "direct loss of user principal" allowed impact gate.

## Likelihood Explanation
Sending ETH with a non-WETH swap is a natural user mistake because every entry-point is `payable` with no guard. The `multicall` pattern actively encourages batching ETH-sending steps with swap steps; any ordering mismatch strands ETH. No privileged role, malicious pool, or non-standard token is required — only two sequential router calls from different callers.

## Recommendation
1. **Track `msg.value` per-call:** Use a transient variable (EIP-1153) or a local variable threaded through the call to record how much ETH the current caller is authorized to spend, and use only that amount in `pay()`.
2. **Enforce refund at every swap exit:** Call `refundETH()` (or an internal equivalent) at the end of each swap entry-point, similar to Uniswap v3's pattern.
3. **Guard non-WETH entry-points:** Revert if `msg.value > 0` and `tokenIn != WETH`.

## Proof of Concept
```solidity
// 1. User A calls exactInputSingle(tokenIn=USDC, amountIn=100e6, msg.value=1 ether)
//    pay() → else branch → safeTransferFrom(userA, pool, 100e6)
//    router.balance == 1 ether (stranded)

// 2. User B calls exactInputSingle(tokenIn=WETH, amountIn=10 ether, msg.value=0)
//    pay() → nativeBalance=1e18 > 0, nativeBalance < value=10e18
//    → deposit(1 ether) from router, safeTransfer(pool, 1 ether WETH)
//    → safeTransferFrom(userB, pool, 9 ether WETH)
//    User B pays only 9 ETH from wallet; User A's 1 ETH is consumed.

// Assert: userA lost 1 ETH; router.balance == 0; userB underpaid by 1 ETH.
```

  function exactInput(ExactInputParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    _validatePath(params.tokens, params.pools, params.extensionDatas);

    uint256 last = params.pools.length - 1;
    int128 amount = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn);

    for (uint256 i = 0; i <= last; i++) {
      address pool = params.pools[i];
      bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, i);

      _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
      (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
        .swap(
          i == last ? params.recipient : address(this),
          zeroForOne,
          amount,
          MetricOmmSwapPath.openLimit(zeroForOne),
          "",
          params.extensionDatas[i]
        );

      int128 amountInActual = MetricOmmSwapResults.extractAmountIn(zeroForOne, amount0Delta, amount1Delta);
      if (amountInActual < amount) revert InvalidInputAmountAtHop(uint8(i), amountInActual, amount);

      amount = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0Delta, amount1Delta);
    }

    if (amount <= 0) revert InvalidSwapDeltas();
    amountOut = MetricOmmSwapInputs.int128ToUint128(amount);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
