### Title
`execute_deploy` Syscall Unconditionally Reports Success Regardless of Constructor Outcome, Enabling Permanent Fund Freezing — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` function in `syscall_impls.cairo` hardcodes `failure_flag=0` in the deploy response header regardless of whether the constructor execution succeeded or failed. Any caller contract always receives a success signal, even when the constructor reverted. If the caller subsequently routes funds to the "deployed" address, those funds are permanently frozen because the contract is either undeployed or in an uninitialized/broken state.

---

### Finding Description

In `execute_deploy` (lines 527–553 of `syscall_impls.cairo`), after invoking `deploy_contract`, the response header is written with a hardcoded `failure_flag=0`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` call can internally execute a constructor that reverts (e.g., due to gas exhaustion, an assertion failure, or any other error condition). However, the return value of `deploy_contract` is `(retdata_size, retdata)` — no failure flag is propagated back to `execute_deploy`. The TODO comment explicitly acknowledges that failure handling is not yet implemented.

Contrast this with `contract_call_helper`, which correctly propagates `is_reverted` from `select_execute_entry_point_func` into the response header:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

The `execute_entry_point` function (which underlies all entry point execution) does return `(is_reverted, retdata_size, retdata)`: [3](#0-2) 

So the failure information exists in the execution pipeline but is discarded by `execute_deploy` before writing the syscall response.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a caller contract invokes the `deploy` syscall:

1. The constructor of the deployed contract fails (reverts) for any reason.
2. The OS writes `failure_flag=0` — the caller sees an unconditional success.
3. The caller stores the returned `contract_address` and routes user funds to it (e.g., as a newly created vault, escrow, or pool).
4. The deployed contract is either not present in state (constructor failure may prevent finalization) or is in an uninitialized state (storage not set up by the constructor).
5. The contract at that address cannot process withdrawals or transfers.
6. All funds sent to that address are permanently frozen with no recovery path.

This is a direct analog to the external report: just as Morpho's liquidation silently fails when pool liquidity is absent (leaving bad debt), the StarkNet OS's deploy syscall silently reports success when the constructor fails, leaving callers to route funds into a broken contract.

---

### Likelihood Explanation

Constructor failures are reachable by any unprivileged transaction sender through multiple vectors:

- **Gas exhaustion**: A caller can set a `deploy` gas budget that is sufficient to pass the `reduce_syscall_base_gas` check but insufficient for the constructor to complete. The constructor runs out of gas and reverts, but `execute_deploy` still reports success.
- **Attacker-crafted constructor**: A malicious class declarer can publish a class whose constructor always reverts under specific conditions (e.g., after a certain block number, or when called with specific calldata).
- **Legitimate protocol bug**: Any DeFi protocol that uses `deploy` syscall to create sub-contracts (vaults, pools, escrows) and then immediately funds them is vulnerable if the constructor can fail for any reason.

The `deploy` syscall is a standard, widely-used primitive. Any protocol that deploys-then-funds is exposed.

---

### Recommendation

Propagate the constructor's failure status through `deploy_contract` and use it to set `failure_flag` in the response header. Specifically:

1. Modify `deploy_contract` to return `(is_reverted, retdata_size, retdata)` analogously to `execute_entry_point`.
2. In `execute_deploy`, replace the hardcoded `failure_flag=0` with the actual `is_reverted` value returned by `deploy_contract`.
3. When `is_reverted=1`, revert all state changes made during the constructor (the existing `handle_revert` mechanism in `execute_entry_point.cairo` can be reused).
4. Remove the TODO comment once the fix is in place.

---

### Proof of Concept

**Step 1**: Declare a class `BrokenVault` whose constructor always reverts (e.g., `assert 1 = 0`).

**Step 2**: Deploy a `Factory` contract with the following logic in its `__execute__`:
```
let (vault_address) = deploy(
    class_hash=BROKEN_VAULT_CLASS_HASH,
    contract_address_salt=0,
    constructor_calldata=[],
    deploy_from_zero=FALSE
);
// OS always returns failure_flag=0, so this branch is always taken:
transfer(token=ETH, to=vault_address, amount=user_funds);
```

**Step 3**: A user calls `Factory.__execute__` with `user_funds > 0`.

**Step 4**: The OS executes `execute_deploy`:
- `deploy_contract` runs `BrokenVault`'s constructor, which reverts.
- `execute_deploy` writes `ResponseHeader(gas=remaining_gas, failure_flag=0)`.
- `Factory` reads `failure_flag=0`, concludes the vault was deployed, and transfers `user_funds` to `vault_address`.

**Step 5**: `vault_address` either has no class hash (contract not deployed) or has an uninitialized state. The `user_funds` are permanently frozen at that address with no withdrawal mechanism.
