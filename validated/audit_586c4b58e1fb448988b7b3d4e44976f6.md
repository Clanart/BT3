### Title
`MerkleBlob::batch_insert` Bypasses Duplicate-Key/Hash Checks, Corrupting DataLayer Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. When a caller supplies a batch containing a repeated `KeyId` or `Hash`, the extra leaf nodes are written directly into the blob without rejection, producing a structurally corrupt Merkle tree whose root hash is wrong. Any proof of inclusion or exclusion derived from that root is therefore invalid, enabling forged state attestations.

### Finding Description

`MerkleBlob::insert` enforces two uniqueness invariants before writing a leaf:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` takes a different code path. When the tree already contains two or more leaves (`leaf_count > 1`), the guarded `self.insert()` calls are skipped entirely. All items in the batch are written directly via `self.insert_entry_to_blob()`:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 578-603
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { ... };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;  // ← guarded
    }
}

for ((key, value), hash) in keys_values_hashes {          // ← NO duplicate check
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

Even when `leaf_count <= 1`, only the last two items popped from the vector pass through `insert`; all earlier items in the vector still bypass the check.

The Python binding `py_batch_insert` exposes this path directly to callers:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 1503-1518
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [3](#0-2) 

When a duplicate key is written, the `block_status_cache` maps that key to only one index while the blob contains two leaf nodes for the same key. The internal-node hashes built over those leaves incorporate both, so the computed root hash diverges from what a correctly-built tree would produce. The `test_double_insert_fails` unit test only covers the single-item `insert` path and does not exercise `batch_insert`: [4](#0-3) 

### Impact Explanation

A corrupted root hash means every `get_proof_of_inclusion` call returns a proof path anchored to a wrong root. A verifier checking the proof against the stored root will either reject valid inclusions or, more critically, accept a proof path that does not correspond to the actual committed key-value set. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

### Likelihood Explanation

`batch_insert` is the recommended bulk-insert API and is exercised in the Python test suite (`test_proof_of_inclusion_merkle_blob` calls `merkle_blob.batch_insert`). Any caller that passes a list containing a repeated `KeyId` — whether by programming error or by a malicious upstream data source — silently corrupts the tree. No privileged role is required; the Python binding is the public API surface. [5](#0-4) 

### Recommendation

Add the same duplicate-key and duplicate-hash guards at the top of `batch_insert` (or inside the bulk loop) that `insert` already enforces:

```rust
pub fn batch_insert(&mut self, keys_values_hashes: Vec<((KeyId, ValueId), Hash)>) -> Result<(), Error> {
    // Pre-validate uniqueness across the entire batch and against existing keys
    for ((key, _value), hash) in &keys_values_hashes {
        if self.block_status_cache.contains_key(*key) {
            return Err(Error::KeyAlreadyPresent());
        }
        if self.block_status_cache.contains_leaf_hash(hash) {
            return Err(Error::HashAlreadyPresent());
        }
    }
    // Also check for intra-batch duplicates
    ...
}
```

A test analogous to `test_double_insert_fails` should be added specifically for `batch_insert` with a repeated key.

### Proof of Concept

```python
from chia_rs import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(bytearray())

# Pre-populate so leaf_count > 1, bypassing the guarded path
blob.insert(KeyId(0), ValueId(0), hashlib.sha256(b"seed0").digest())
blob.insert(KeyId(1), ValueId(1), hashlib.sha256(b"seed1").digest())
blob.insert(KeyId(2), ValueId(2), hashlib.sha256(b"seed2").digest())

# batch_insert with a duplicate key — should raise KeyAlreadyPresent, but does not
dup_key   = KeyId(0)          # already present
dup_value = ValueId(99)
dup_hash  = hashlib.sha256(b"dup").digest()

blob.batch_insert([(dup_key, dup_value)], [dup_hash])
blob.calculate_lazy_hashes()

# Root hash is now wrong; proof for key 0 is anchored to a corrupted root
proof = blob.get_proof_of_inclusion(KeyId(0))
# proof.valid() may return True against the corrupted root,
# but the root itself no longer represents the true committed state.
```
