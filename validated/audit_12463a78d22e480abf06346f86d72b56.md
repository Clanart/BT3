### Title
Child `ChunkExtra.proposed_split` Inherited Uncleared from Parent Shard During Resharding, Breaking First-Chunk Validation — (`File: chain/chain/src/resharding/manager.rs`)

### Summary

During a shard split in the dynamic resharding path, `process_memtrie_resharding_storage_update` creates each child shard's `ChunkExtra` by cloning the parent's `ChunkExtra` and updating only `state_root` and `congestion_info`. The `proposed_split` field — which holds the `TrieSplit` that triggered the split — is never cleared. The child's first produced chunk header will carry `proposed_split = None` (because `compute_proposed_split` returns `None` at the start of an epoch), while `prev_chunk_extra.proposed_split()` still holds the parent's non-`None` `TrieSplit`. The equality check in `validate_chunk_with_chunk_extra_and_receipts_root` then fires `InvalidChunkHeaderShardSplit`, permanently blocking the child shard.

### Finding Description

`process_memtrie_resharding_storage_update` builds each child's `ChunkExtra` as:

```rust
// TODO(resharding): set all fields of `ChunkExtra`. Consider stronger
// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
``` [1](#0-0) 

Only `state_root` and `congestion_info` are overwritten. `proposed_split` is silently inherited from the parent, which at the resharding boundary block holds `Some(TrieSplit { boundary_account, left_mem, right_mem })` — the very split that triggered the epoch's resharding.

`ChunkExtraV5` declares the field:

```rust
/// Proposed split of this shard (dynamic resharding).
pub proposed_split: Option<TrieSplit>,
``` [2](#0-1) 

The validation gate that fires on mismatch:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    DYNAMIC_RESHARDING_VALIDATION_FAILURES.with_label_values(&["chunk_header"]).inc();
    return Err(Error::InvalidChunkHeaderShardSplit(format!(
        "header has {:?}, expected {:?} ...",
        chunk_header.proposed_split(),
        prev_chunk_extra.proposed_split(),
    )));
}
``` [3](#0-2) 

For the child shard's first chunk, `compute_proposed_split` returns `None` because `is_next_block_possibly_last_in_epoch` is false at epoch start:

```rust
if !self.epoch_manager.is_next_block_possibly_last_in_epoch(height, prev_block_hash)? {
    return Ok(None);
}
``` [4](#0-3) 

So the comparison is `Some(TrieSplit{…}) != None`, which always fails. The codebase's own architecture documentation acknowledges this directly:

> "allowing back-to-back reshardings is unsafe because a freshly-created child shard would inherit `proposed_split` from the parent's final chunk while its own first chunk freshly computes `proposed_split = None`, triggering `InvalidChunkHeaderShardSplit`."



And the TODO at the exact line:

> **`chain/chain/src/resharding/manager.rs:249`** — The resharding manager doesn't set all `ChunkExtra` fields (notably the new `proposed_split` field).

### Impact Explanation

Every shard split under dynamic resharding produces child shards whose `ChunkExtra.proposed_split` is permanently set to the parent's non-`None` `TrieSplit`. The first chunk header produced for each child shard carries `proposed_split = None`. The invariant check in `validate_chunk_with_chunk_extra_and_receipts_root` rejects it with `InvalidChunkHeaderShardSplit`. Because the stale value is persisted in `DBCol::ChunkExtra` and never corrected, every subsequent attempt to produce a chunk for the child shard fails identically. The child shard is permanently halted from its first block in the new epoch. **Impact: Critical** — a shard split that is a core protocol operation renders the resulting child shards inoperable.

### Likelihood Explanation

The bug fires on every execution of `split_shard` → `process_memtrie_resharding_storage_update` when `DynamicResharding` is enabled and a shard split occurs. No special attacker input is required; the trigger is the normal epoch-boundary resharding path. The feature is gated behind `ProtocolFeature::DynamicResharding`, so it does not affect mainnet until that flag is activated, but the defect is present in the production code path today.

### Recommendation

In `process_memtrie_resharding_storage_update`, after cloning the parent `ChunkExtra`, explicitly clear `proposed_split` on the child:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// ADD: child shards start with no pending split proposal
*child_chunk_extra.proposed_split_mut() = None;
```

A `proposed_split_mut()` accessor analogous to `state_root_mut()` and `congestion_info_mut()` should be added to `ChunkExtra`. Alternatively, construct the child `ChunkExtra` from scratch using `ChunkExtra::new(...)` with `proposed_split: None` rather than cloning the parent.

### Proof of Concept

1. Enable `ProtocolFeature::DynamicResharding` (set the protocol version to the feature's activation version in a test network).
2. Fill a shard's state past `DynamicReshardingConfig::memory_split_threshold` so that `compute_proposed_split` returns `Some(TrieSplit{…})` near the end of epoch N.
3. The last block of epoch N embeds `shard_split` in its header; `finalize_epoch` stores a new `ShardLayoutV3` for epoch N+2.
4. At the epoch N/N+1 boundary, `split_shard` → `process_memtrie_resharding_storage_update` runs. Inspect `DBCol::ChunkExtra` for the child shard UIDs at the boundary block hash: `proposed_split` equals the parent's `TrieSplit`, not `None`.
5. At the start of epoch N+2, the chunk producer for a child shard produces a chunk. `compute_proposed_split` returns `None` (epoch start, `is_next_block_possibly_last_in_epoch = false`). The chunk header carries `proposed_split = None`.
6. `validate_chunk_with_chunk_extra_and_receipts_root` compares `prev_chunk_extra.proposed_split() = Some(TrieSplit{…})` against `chunk_header.proposed_split() = None` and returns `Err(InvalidChunkHeaderShardSplit)`.
7. All validators reject the chunk; the child shard produces no valid chunks and is permanently stalled.
