### Title
Missing Chain-ID Domain Separation in `deploy_token` Signed Message Enables Cross-Chain Replay — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`, `starknet/src/bridge_types.cairo`)

---

### Summary

The NEAR MPC-signed payload for `deploy_token` / `deployToken` does not include a chain identifier, unlike `fin_transfer` / `finTransfer` / `finalize_transfer` which explicitly bind the signature to a specific chain via `omniBridgeChainId` / `SOLANA_OMNI_BRIDGE_CHAIN_ID`. A single `deploy_token` signature obtained (or observed on-chain) for one deployment can be replayed verbatim on every other chain that shares the same `nearBridgeDerivedAddress`, deploying the bridged token there without a fresh MPC signature. On EVM chains, where the deployed token address is non-deterministic, this corrupts the NEAR bridge's token-address registry and can permanently freeze user funds.

### Impact Explanation

The Omni Bridge is deployed on multiple chains (EVM Ethereum, EVM BSC/Polygon/etc., Solana, StarkNet), all sharing the same NEAR MPC key and thus the same `nearBridgeDerivedAddress`. Because the `deploy_token` signed payload is identical across all chains for the same token, a single valid signature is accepted by every chain's contract.

**Critical path on EVM (non-deterministic token address):**

1. NEAR MPC signs a `deploy_token` payload for token `"usdc.near"` targeting EVM chain A (e.g., Ethereum). The signature is broadcast and the transaction is visible on-chain.
2. Attacker extracts `(signatureData, metadata)` from chain A's transaction and submits the identical call to chain B's `OmniBridge.deployToken` (e.g., BSC).
3. Chain B deploys a new `ERC1967Proxy` at a non-deterministic address `X_B` (determined by chain B's contract nonce at that moment). The `nearToEthToken["usdc.near"] = X_B` mapping is set on chain B.
4. The NEAR bridge backend, which tracks token addresses per chain, either:
   - **Does not observe** the attacker's `DeployToken` event on chain B (because it did not initiate the deployment), leaving its internal registry empty for chain B. When users later bridge USDC to chain B, `sign_transfer` on NEAR resolves `get_token_address(ChainB, "usdc.near")` to nothing or a stale value, causing all `finTransfer` calls to chain B to reference a non-existent or wrong token address → **permanent fund freeze**.
   - **Does observe** the event but later attempts its own legitimate deployment, which reverts with `"ERR_TOKEN_EXIST"`. The NEAR backend is forced to accept `X_B` as the canonical address. If `X_B` differs from the address the NEAR backend computed internally, subsequent `finTransfer` messages signed with the NEAR-expected address will fail to release tokens → **fund freeze**.

**On StarkNet and Solana**, token addresses are deterministic (salt-based / PDA-based), so the deployed address is the same regardless of who triggers the deployment. The impact there is premature/unauthorized deployment and potential disruption of the NEAR backend's deployment sequencing, but not address mismatch.

---

### Likelihood Explanation

- The attack requires only observing a `deployToken` transaction on any one chain and replaying it on another. No privileged access, no key material, no MPC collusion is needed.
- The attacker is any unprivileged user who can submit transactions on the target chain.
- Multiple EVM deployments of Omni Bridge are the normal operating model (the `omniBridgeChainId` storage field exists precisely because the same contract is deployed on multiple chains).
- The window of opportunity is any time between the NEAR MPC signing for chain A and the NEAR backend's own submission on chain B.

---

### Recommendation

Bind the `deploy_token` signed payload to the destination chain by including the chain ID in the Borsh-encoded message, mirroring the existing pattern used in `fin_transfer`:

**EVM:**
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to this chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

**Solana:** add `writer.write_all(&[SOLANA_OMNI_BRIDGE_CHAIN_ID])?;` before serializing the payload fields in `DeployTokenPayload::serialize_for_near`.

**StarkNet:** change `MetadataPayloadTrait::to_borsh` to accept `chain_id: u8` and append it after the `PayloadType` byte, and update the `deploy_token` call site to pass `self.omni_bridge_chain_id.read()`.

The NEAR MPC signing side must correspondingly include the destination chain ID when constructing the `deploy_token` payload hash.

### Proof of Concept

**Setup:** Two EVM OmniBridge deployments exist — chain A (`omniBridgeChainId = 1`) and chain B (`omniBridgeChainId = 56`). Both share the same `nearBridgeDerivedAddress`.

**Step 1 — Observe chain A deployment:**
```
tx = chain_A.deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})
# sig = ECDSA over keccak256(borsh(Metadata | "usdc.near" | "USD Coin" | "USDC" | 6))
# No chain ID in the hash → sig is valid on ALL chains
```

**Step 2 — Replay on chain B:**
```
chain_B.deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})
# ECDSA.recover(keccak256(same borsh bytes), sig) == nearBridgeDerivedAddress  ✓
# isBridgeToken[nearToEthToken["usdc.near"]] == false  ✓  (not yet deployed on B)
# → new ERC1967Proxy deployed at address X_B (non-deterministic)
# → nearToEthToken["usdc.near"] = X_B  on chain B
```

**Step 3 — NEAR backend attempts legitimate deployment on chain B:**
```
chain_B.deployToken(sig2, {token:"usdc.near", ...})
# → reverts: "ERR_TOKEN_EXIST"
# NEAR backend's registry for (ChainB, "usdc.near") is either empty or points to a different address
```

**Step 4 — User bridges USDC to chain B:**
```
# NEAR sign_transfer resolves token address for ChainB → wrong/missing address
# finTransfer on chain B references wrong tokenAddress → transfer fails
# User's funds locked on NEAR with no recourse
```
