### Title
Non-deterministic `std::collections::HashMap` in NEAR Contract State Causes Potential Consensus Failure and Bridge Freeze - (File: `near/omni-bridge/src/lib.rs`, `near/omni-prover/mpc-omni-prover/src/lib.rs`)

### Summary

Two NEAR contracts in the Omni Bridge persist `std::collections::HashMap` directly inside their Borsh-serialized contract state structs. Rust's `HashMap` uses a randomly-seeded hasher (`RandomState`), so its iteration order — and therefore its Borsh serialization byte sequence — is non-deterministic across process restarts or different runtime environments. When NEAR validators independently re-serialize the contract state after executing a transaction, they can produce different byte sequences for the same logical state, yielding divergent state roots and breaking consensus. This can permanently freeze the bridge.

**Root cause — `near/omni-prover/mpc-omni-prover/src/lib.rs` line 47:**

pub struct MpcOmniProver {
    pub mpc_contract_id: AccountId,
    pub finalities: HashMap<ChainKind, MpcFinality>,  // line 47

Both structs are annotated with `#[near(contract_state)]`, meaning the entire struct is Borsh-serialized into a single storage slot on every state-mutating call. Borsh serializes a `HashMap` by iterating over its entries; the iteration order is governed by the internal hash seed, which in Rust's standard library is seeded from OS randomness (`getrandom`). In NEAR's Wasm execution environment, `getrandom` is wired to `env::random_seed()`, which is derived from the block's VRF output and is therefore **different on every block**. Consequently:

1. Block N: `utxo_chain_connectors` is deserialized from storage (fixed byte order from the previous write).
2. A user submits any state-mutating transaction (e.g., `ft_on_transfer` to initiate a bridge transfer).
3. Each validator independently re-creates the in-memory `HashMap` from the deserialized entries, but with a **new random seed** drawn from the current block's `random_seed()`.
4. When the contract state is written back, the HashMap entries are serialized in the new (block-specific but validator-consistent) order — **unless** the seed differs between validators, which can occur if the Wasm host's `getrandom` implementation is not perfectly synchronized.
5. Even a single byte difference in the serialized state blob produces a different storage trie hash, causing validators to disagree on the post-transaction state root.

The NEAR SDK explicitly provides `near_sdk::collections::LookupMap` and `UnorderedMap` (backed by the trie, not in-memory hash maps) to avoid exactly this class of bug. Using `std::collections::HashMap` in contract state is a documented anti-pattern in NEAR development.

**Affected code paths:**

- `add_utxo_chain_connector` (btc.rs line 130) inserts into `utxo_chain_connectors`; any subsequent call to `ft_on_transfer`, `fin_transfer`, `sign_transfer`, etc. re-serializes the entire `Contract` struct including the HashMap.
- `MpcOmniProver::init` (mpc-omni-prover/src/lib.rs line 55–66) populates `finalities`; any call to `verify_proof` re-serializes the struct.

### Impact Explanation

If validators produce divergent state roots, NEAR's BFT consensus rejects the block. Repeated failures halt block production for the shard hosting the bridge contract. All in-flight transfers — locked tokens on NEAR, pending mints on EVM/Solana/StarkNet — become permanently unclaimable. This matches the **Critical** impact category: *permanent freezing / irrecoverable lock of user and protocol funds in bridge flows*.

### Likelihood Explanation

The `utxo_chain_connectors` map is populated by a DAO-privileged call (`add_utxo_chain_connector`), but once populated, **any unprivileged user** triggering a state-mutating transaction (e.g., a normal `ft_on_transfer` bridge deposit) causes the full contract struct — including the HashMap — to be re-serialized. The non-determinism is latent in the state from the moment the DAO adds a UTXO chain connector. The likelihood is medium: it depends on whether NEAR's Wasm `getrandom` binding produces a per-validator-divergent seed or a per-block-uniform seed. Given that NEAR's `random_seed()` is block-level (same for all validators in the same block), the divergence may only manifest across block boundaries or Wasm runtime upgrades, but the structural risk is present and the NEAR SDK explicitly forbids this pattern.

### Recommendation

Replace both `std::collections::HashMap` fields with NEAR SDK trie-backed collections:

- `utxo_chain_connectors: HashMap<ChainKind, UTXOChainConfig>` → `LookupMap<ChainKind, UTXOChainConfig>` with a dedicated `StorageKey` variant.
- `finalities: HashMap<ChainKind, MpcFinality>` → `LookupMap<ChainKind, MpcFinality>` or, if iteration is needed, `UnorderedMap<ChainKind, MpcFinality>`.

These collections store each entry as a separate trie key-value pair, so serialization order is irrelevant and state roots remain deterministic.

### Proof of Concept

1. DAO calls `add_utxo_chain_connector(ChainKind::Btc, ...)` — inserts one entry into `utxo_chain_connectors`.
2. DAO calls `add_utxo_chain_connector(ChainKind::SomeOtherUtxoChain, ...)` — inserts a second entry.
3. Any user calls `ft_on_transfer(...)` to initiate a NEAR→EVM transfer.
4. The NEAR runtime serializes the full `Contract` struct. The two `utxo_chain_connectors` entries are written in HashMap iteration order, which is seeded by the block's `random_seed()`.
5. If the seed differs between the block where entries were inserted and the block where `ft_on_transfer` runs (guaranteed, since `random_seed()` changes every block), the serialized bytes of `utxo_chain_connectors` differ from the previous write.
6. Across a Wasm runtime upgrade or any scenario where validators' `getrandom` seeds diverge, different validators write different bytes → different storage trie hashes → consensus failure → bridge shard halted. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
