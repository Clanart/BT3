### Title
Unchecked `fee_proposal_fri` During Window Initialization Permanently Inflates `next_l2_gas_price` — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

During the first `window_size` blocks after V0_14_3 activation, `compute_fee_actual` returns `None` because the sliding window is not yet full. The validator's `is_proposal_init_valid` unconditionally skips the `fee_proposal_fri` bounds check when `fee_actual` is `None`. A proposer during this initialization window can therefore set `ProposalInit.fee_proposal_fri` to an arbitrarily large value. That value is accepted by every validator, stored in `fee_proposals_window` and in `StorageBlockHeader.fee_proposal_fri`, and once the window fills it becomes the dominant input to `fee_actual`. Because `fee_actual` is used as the floor for `effective_min` in `calculate_next_l2_gas_price_for_fin`, the resulting `next_l2_gas_price` committed to every subsequent block header is permanently inflated, causing all users to pay higher fees than the protocol intends.

---

### Finding Description

**Step 1 — `fee_actual` is `None` during initialization.**

`compute_fee_actual` requires exactly `window_size` prior heights in `fee_proposals_window`. When `height < window_size`, `height.0.checked_sub(window_size)` underflows and the function returns `None`: [1](#0-0) 

**Step 2 — Validator skips bounds enforcement when `fee_actual` is `None`.**

`is_proposal_init_valid` only enforces the `fee_proposal_fri` range check when both `fee_actual` and `fee_proposal` are `Some`. When `fee_actual` is `None` the entire block is skipped: [2](#0-1) 

The comment explicitly acknowledges this: *"During initiation (fee_actual is None, <window_size blocks), bounds are not enforced."* However, it does not restrict the proposer to the honest fallback value (`self.l2_gas_price`). Any `fee_proposal_fri` value — including `u128::MAX` — passes validation.

**Step 3 — Inflated value is stored in the window and in the block header.**

After consensus, `init.fee_proposal_fri` is forwarded to the cende pipeline and stored in `StorageBlockHeader.fee_proposal_fri`: [3](#0-2) 

On the next startup, `initialize_fee_proposals_window` reads `fee_proposal_fri` from state-sync and re-populates `fee_proposals_window`: [4](#0-3) 

**Step 4 — Inflated `fee_actual` raises `next_l2_gas_price`.**

Once the window fills, `fee_actual` is the median of the stored values. `calculate_next_l2_gas_price_for_fin` uses `fee_actual` as the floor (`effective_min`) for the EIP-1559 price update: [5](#0-4) 

`calculate_next_base_gas_price` then guarantees `next_l2_gas_price >= effective_min`: [6](#0-5) 

The resulting `next_l2_gas_price` is committed into the block header and becomes the L2 gas price for every transaction in the following block: [7](#0-6) 

**Step 5 — The inflated price is also bound into the proposal commitment.**

`proposal_commitment_from` hashes `fee_proposal_fri` into the consensus commitment, so the inflated value is cryptographically locked in and cannot be revised after the fact: [8](#0-7) 

---

### Impact Explanation

The corrupted value is `StorageBlockHeader.next_l2_gas_price` for every block after the initialization window closes. Because `next_l2_gas_price` is the L2 gas price applied to all transactions in the next block, every user's actual fee is computed against an inflated price. This is a direct, persistent, economic impact matching the Critical scope: *"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."*

Additionally, `starknet_estimateFee` and `starknet_simulateTransactions` read gas prices from the block header: [9](#0-8) 

So RPC fee estimates returned to users are also wrong, matching the High scope: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

---

### Likelihood Explanation

The attack window is the first `window_size` blocks after V0_14_3 activates (a known, scheduled protocol upgrade). In a Tendermint-style multi-validator consensus, any validator is eligible to be proposer in round-robin. A validator who is selected as proposer for even one block during this window can shift the eventual median. A validator controlling `floor(window_size/2) + 1` consecutive proposer slots can set the median to an arbitrary value. Because the upgrade height is public and the proposer schedule is deterministic, a motivated validator can plan for this in advance.

---

### Recommendation

Enforce a concrete upper bound on `fee_proposal_fri` even when `fee_actual` is `None`. The natural bound is the proposer's own honest fallback: `self.l2_gas_price`. The validator should reject any `fee_proposal_fri` that exceeds a configurable multiple of `l2_gas_price_fri` (the locally-trusted L2 gas price in `ProposalInitValidation`):

```rust
// In is_proposal_init_valid, replace the bare None-skip with:
if let Some(fee_proposal) = init_proposed.fee_proposal_fri {
    let reference = proposal_init_validation.l2_gas_price_fri;
    let (lower, upper) = match proposal_init_validation.fee_actual {
        Some(fa) => fee_proposal_bounds(fa, margin_ppt),
        None => fee_proposal_bounds(reference, INIT_WINDOW_MARGIN_PPT),
    };
    if fee_proposal.0 < lower || fee_proposal.0 > upper {
        return Err(...);
    }
}
```

Alternatively, during the initialization window, clamp `fee_proposal_fri` to `[min_gas_price, l2_gas_price * (1 + max_init_multiplier)]` on both the proposer and validator sides.

---

### Proof of Concept

1. Protocol upgrades to V0_14_3 at block `B`. `window_size = W` (e.g., 10).
2. Attacker is selected as proposer for block `B`. They craft `ProposalInit` with `fee_proposal_fri = u128::MAX`.
3. Every validator calls `is_proposal_init_valid`. `compute_fee_actual` returns `None` (height `B < B + W`). The bounds check at lines 398–416 of `validate_proposal.rs` is skipped. The proposal is accepted.
4. `fee_proposal_fri = u128::MAX` is stored in `fee_proposals_window[B]` and in `StorageBlockHeader[B].fee_proposal_fri`.
5. Attacker repeats for blocks `B+1` through `B + W/2` (controlling the majority of the window).
6. At block `B + W`, `compute_fee_actual` returns `Some(u128::MAX)` (median of the inflated window).
7. `calculate_next_l2_gas_price_for_fin` sets `effective_min = u128::MAX`. `next_l2_gas_price = u128::MAX` is committed to block `B + W`'s header.
8. All transactions in block `B + W + 1` are charged at `l2_gas_price = u128::MAX`, making the network economically unusable.
