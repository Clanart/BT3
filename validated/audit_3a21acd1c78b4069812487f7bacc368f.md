### Title
Stale `feemarketParams` Snapshot in `newCosmosAnteHandler` Causes `MinGasPriceDecorator` and `DynamicFeeChecker` to Enforce Outdated Fee Floors — (`evmd/ante/handler_options.go`)

### Summary

`newCosmosAnteHandler` (and `newLegacyCosmosAnteHandlerEip712`) reads `feemarketParams` once at construction time and passes a pointer to that local copy into both `cosmos.NewMinGasPriceDecorator` and `evm.NewDynamicFeeChecker`. Because the Cosmos SDK ante handler is constructed once at app initialization and reused for every subsequent transaction, the `MinGasPrice` and `BaseFee` values embedded in those decorators are permanently frozen at their genesis/startup values. Any on-chain change to these parameters — governance-driven `MinGasPrice` increases or the natural per-block EIP-1559 base-fee drift — is silently ignored, allowing Cosmos-SDK transactions to bypass the current fee floor.

### Finding Description

In `evmd/ante/handler_options.go`, `newCosmosAnteHandler` captures `feemarketParams` as a local value-type variable and immediately takes its address:

```go
feemarketParams := options.FeeMarketKeeper.GetParams(ctx)   // snapshot at construction
...
txFeeChecker = evm.NewDynamicFeeChecker(ethCfg, &evmParams, &feemarketParams)
...
cosmos.NewMinGasPriceDecorator(options.FeeMarketKeeper, evmDenom, &feemarketParams)
``` [1](#0-0) 

`MinGasPriceDecorator` stores this pointer and reads `mpd.feemarketParams.MinGasPrice` on every `AnteHandle` call instead of querying the keeper:

```go
minGasPrice := mpd.feemarketParams.MinGasPrice   // always the startup value
``` [2](#0-1) 

`NewDynamicFeeChecker` similarly captures the pointer and uses it to derive `baseFee` on every invocation:

```go
baseFee := types.GetBaseFee(ctx.BlockHeight(), ethCfg, feemarketParams)
``` [3](#0-2) 

The `MinGasPriceDecorator` struct even carries an unused `feesKeeper` field that could supply live params, but it is never consulted: [4](#0-3) 

By contrast, the Ethereum ante handler (`newEthAnteHandler`) correctly re-reads params on every call via `EVMBlockConfig`, so only Cosmos-SDK transactions are affected: [5](#0-4) 

The same construction-time snapshot bug exists in `newLegacyCosmosAnteHandlerEip712`: [6](#0-5) 

### Impact Explanation

**MinGasPrice bypass (governance-triggered):** After a governance proposal raises `MinGasPrice`, the stale decorator continues enforcing the old (lower) floor. Any unprivileged user can submit Cosmos-SDK transactions with fees below the new minimum; they pass `MinGasPriceDecorator`, are admitted to the mempool, and are included in committed blocks — invalid transactions commit.

**DynamicFeeChecker base-fee bypass (continuous):** The EIP-1559 base fee adjusts every block via `BeginBlock`/`EndBlock`. The `DynamicFeeChecker` closure uses the frozen startup `BaseFee` value. As the live base fee drifts upward (high-traffic periods), Cosmos transactions with `ExtensionOptionDynamicFeeTx` are validated and fee-deducted against the stale lower base fee. Senders pay less than the protocol-mandated effective price — valid user fees are mis-accounted, and the fee-market signal is corrupted.

Both paths map to the allowed High impact: *"fee market, ante handler bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."*

### Likelihood Explanation

- **MinGasPrice path:** Requires a governance parameter change, which is a normal chain-operation event. Once `MinGasPrice` is raised, every node that has not restarted is permanently vulnerable for the remainder of its uptime.
- **DynamicFeeChecker path:** Requires no special conditions. The base fee diverges from the startup snapshot on the very first block where gas usage differs from the target. Any Cosmos transaction using `ExtensionOptionDynamicFeeTx` after that point is affected. This is a continuous, always-reachable condition.

### Recommendation

Replace the construction-time snapshot with a live keeper read inside each decorator's `AnteHandle` / checker closure:

1. In `MinGasPriceDecorator.AnteHandle`, replace `mpd.feemarketParams.MinGasPrice` with `mpd.feesKeeper.GetParams(ctx).MinGasPrice` (the keeper field already exists).
2. In `NewDynamicFeeChecker`, replace the captured `feemarketParams` pointer with a per-invocation `feeMarketKeeper.GetParams(ctx)` call, or pass the keeper instead of a params pointer.
3. Apply the same fix to `newLegacyCosmosAnteHandlerEip712`.

### Proof of Concept

1. Deploy a chain with `MinGasPrice = 0` and `BaseFee = 765625000` (default).
2. Submit a governance proposal to set `MinGasPrice = 1_000_000_000`. Proposal passes; `SetParams` updates the KV store.
3. Without restarting the node, submit a Cosmos `MsgSend` with `fee = 0`.
4. `MinGasPriceDecorator.AnteHandle` reads `mpd.feemarketParams.MinGasPrice` → `0` (stale). Check passes.
5. `DeductFeeDecorator` deducts `0` fees. Transaction is included in a committed block despite violating the current `MinGasPrice = 1_000_000_000` parameter.

For the `DynamicFeeChecker` path: after several blocks of high gas usage the live base fee rises to, say, `2_000_000_000`. A Cosmos transaction with `ExtensionOptionDynamicFeeTx` and `feeCap = 1_000_000_000` (below live base fee) passes `NewDynamicFeeChecker` because it compares against the frozen startup value `765_625_000`, and the sender is charged at the stale effective price rather than the current one.
