### Title
Missing Token Registration Check in `initTransfer` Allows Permanent Fund Lock — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/omni_bridge.cairo`)

---

### Summary

The EVM `initTransfer` and StarkNet `init_transfer` functions accept any arbitrary token address without verifying that the token is registered in the bridge. When a user submits a transfer for an unregistered token, the tokens are irrevocably locked in the bridge contract because the NEAR-side finalization callback panics on `TokenDecimalsNotFound`, leaving no user-initiated recovery path.

---

### Finding Description

**EVM side (`OmniBridge.sol`):**

`initTransfer` handles three token cases: custom minter, bridge token (burn), and a catch-all `else` branch that does `safeTransferFrom` into the bridge contract for any other ERC20. [1](#0-0) 

There is no check that `tokenAddress` has been registered in the bridge (i.e., that `token_decimals` or `token_address_to_id` mappings exist on NEAR). Any ERC20 — including one that has never gone through the `logMetadata` → `deployToken` registration flow — is silently accepted and locked.

**StarkNet side (`omni_bridge.cairo`):**

`init_transfer` similarly branches on `is_bridge_token` and falls through to a raw `transfer_from` for any other token, with no registration check. [2](#0-1) 

**NEAR finalization side (`lib.rs`):**

When a relayer submits the proof for such a transfer, `fin_transfer_callback` attempts to look up the token's decimal configuration. If the token was never registered, this lookup panics: [3](#0-2) 

Because the NEAR callback panics, the entire NEAR transaction is reverted — including the `finalised_transfers` nonce update. The EVM/StarkNet tokens remain locked in the bridge contract. There is no `rescueTokens` or equivalent admin function in `OmniBridge.sol` to recover them.

Similarly, for NEAR-originated transfers via `ft_on_transfer` → `init_transfer_internal`, if the token is not a deployed bridge token, `sign_transfer` later panics with `FailedToGetTokenAddress`: [4](#0-3) 

The transfer message is stored in `pending_transfers` but can never be signed or finalized, and the tokens are locked with no user-initiated refund path.

---

### Impact Explanation

A user who calls `initTransfer` (EVM or StarkNet) with a token that has not been registered through the `logMetadata` → `deployToken` flow will have their tokens permanently locked in the bridge contract. The NEAR-side callback panics, the nonce is not consumed, and there is no admin rescue function. The lock is permanent unless the DAO separately registers the token and a relayer retries — an out-of-band manual intervention with no protocol guarantee.

This matches: **Critical. Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

- Any unprivileged user can call `initTransfer` directly with any ERC20 address.
- A user may accidentally use a token that has not yet completed the registration flow (e.g., a newly deployed token, a token on a chain where `logMetadata` was never called).
- A malicious actor could phish a user into calling `initTransfer` with an unregistered token, causing permanent loss.
- The EVM `SECURITY.md` explicitly marks `logMetadata` and `deployToken` as permissionless by design, but does not acknowledge `initTransfer` accepting unregistered tokens as a known/accepted risk. [5](#0-4) 

---

### Recommendation

Add a registration guard at the top of `initTransfer` (EVM) and `init_transfer` (StarkNet) that verifies the token is known to the bridge before accepting funds:

- **EVM**: Require that `ethToNearToken[tokenAddress]` is non-empty (i.e., the token has a registered NEAR counterpart) before proceeding with the `safeTransferFrom`.
- **StarkNet**: Require that `starknet_to_near_token.read(token_address)` is non-empty before accepting the transfer.
- **NEAR (`ft_on_transfer`)**: In `init_transfer_internal`, check that the token is registered (exists in `token_id_to_address`) and return the full amount as a refund if not, rather than storing an unresolvable transfer message.

Additionally, consider adding an admin `rescueTokens` function to recover ERC20 tokens that become stuck due to failed cross-chain transfers.

---

### Proof of Concept

1. Deploy a new ERC20 token `MalToken` on an EVM chain. Do **not** call `logMetadata` or `deployToken` for it.
2. Approve the `OmniBridge` contract to spend `MalToken`.
3. Call `OmniBridge.initTransfer(MalToken, 1000, 0, 0, "victim.near", "")`.
4. The bridge executes `MalToken.safeTransferFrom(msg.sender, address(this), 1000)` — tokens are now locked.
5. A relayer submits the proof to NEAR `fin_transfer`.
6. NEAR `fin_transfer_callback` reaches `self.token_decimals.get(&init_transfer.token).near_expect(BridgeError::TokenDecimalsNotFound)` and panics.
7. The NEAR transaction is fully reverted; `finalised_transfers` is not updated.
8. The 1000 `MalToken` remain locked in `OmniBridge` with no recovery path for the user. [6](#0-5) [3](#0-2)
