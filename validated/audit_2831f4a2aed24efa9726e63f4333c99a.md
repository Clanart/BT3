### Title
Permanent Fund Lock When `normalize_amount` Returns Zero Due to Decimal Precision Mismatch — (`near/omni-bridge/src/lib.rs`)

---

### Summary

In `sign_transfer`, the bridge enforces `amount_to_transfer > 0` after decimal normalization. However, `init_transfer` only validates `fee < amount` — it does not verify that `normalize_amount(amount − fee) > 0`. When a user initiates a transfer whose `amount_without_fee` is smaller than the normalization divisor (i.e., rounds to zero under floor division), `sign_transfer` will revert on every call. Because no cancel or refund path exists for pending transfers, the user's funds are permanently locked.

---

### Finding Description

**Step 1 — Funds are locked at `init_transfer` time.**

`ft_on_transfer` routes to `init_transfer`, which stores the `TransferMessage` in `pending_transfers` and returns `U128(0)` to the NEP-141 callback, keeping all tokens inside the bridge contract. The only validity check on the fee is:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

There is no check that `normalize_amount(amount − fee) > 0`.

**Step 2 — `sign_transfer` always reverts.**

When the relayer later calls `sign_transfer`, the bridge computes:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

`normalize_amount` uses floor division (confirmed by the inline comment: *"Since `denormalize(normalize(x)) <= x` due to floor division"*). For a NEAR-native token with 24 decimals bridged to an EVM chain with 6 decimals, the divisor is `10^18`. Any `amount_without_fee < 10^18` normalizes to zero, causing `sign_transfer` to revert unconditionally.

**Step 3 — No recovery path exists.**

- `update_transfer_fee` enforces `fee.fee >= current_fee.fee`, so the sender can only *increase* the fee, making the normalized amount even smaller.
- There is no `cancel_transfer` or `refund_transfer` function.
- `sign_transfer_callback` only removes the transfer from `pending_transfers` when `fee.is_zero()` — which is not the case here.
- `claim_fee_callback` requires a proof that the transfer was finalized on the destination chain, which can never happen if `sign_transfer` always reverts.

The transfer record stays in `pending_transfers` indefinitely, and the user's tokens are irrecoverably locked.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

A user who initiates a transfer where `amount − fee` is below the decimal normalization threshold loses all transferred tokens with no on-chain recovery mechanism. The DAO's `transfer_token_as_dao` is the only escape hatch, requiring privileged intervention for every affected user.

---

### Likelihood Explanation

**Medium.** The condition is reachable by any unprivileged bridge user:

- Tokens with large decimal gaps are common (e.g., a NEAR-native token with 24 decimals bridged to a 6-decimal EVM token yields a divisor of `10^18`).
- A user who sets a fee of `amount − 1` (e.g., to minimize the relayer's incentive) will have `amount_without_fee = 1`, which normalizes to zero.
- The user receives no warning at `init_transfer` time; the failure only surfaces when the relayer attempts `sign_transfer`.

---

### Recommendation

Add the normalization check at `init_transfer` time, before funds are locked:

```rust
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(token_address) = token_address {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

Alternatively, add a `cancel_transfer` function that allows the original sender to reclaim funds from a stuck pending transfer.

---

### Proof of Concept

1. Token `tok.near` has 24 decimals on NEAR; its EVM counterpart has 6 decimals. The bridge stores `Decimals { decimals: 6, origin_decimals: 24 }`, so `normalize_amount` divides by `10^18`.
2. User calls `ft_transfer_call` on `tok.near` with `amount = 2 * 10^18` and `msg = InitTransferMsg { fee: 10^18 + 1, ... }`.
3. `init_transfer` passes: `fee (10^18 + 1) < amount (2 * 10^18)`. Transfer stored; tokens locked.
4. Relayer calls `sign_transfer`. `amount_without_fee = 10^18 − 1`. `normalize_amount(10^18 − 1, decimals) = 0`. `require!(0 > 0)` panics → revert.
