### Title
Cache-Blob Inconsistency in `MerkleBlob::upsert` Leaves Active Index in Free List, Enabling Tree Corruption and Forged Proofs — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::upsert` calls `block_status_cache.remove_leaf()` to strip the old leaf from the in-memory index cache (which also inserts `leaf_index` into `free_indexes`), then writes the updated leaf back to the **same** `leaf_index` in the blob — but never calls `add_leaf` to re-register the index as active. The result is a permanent split between the blob (which holds live data at `leaf_index`) and the `BlockStatusCache` (which considers `leaf_index` free, the key absent, and the new hash absent). Any subsequent `get_new_index()` call will pop `leaf_index` from `free_indexes` and hand it to a new node, silently overwriting the upserted leaf and corrupting the Merkle tree.

---

### Finding Description

`MerkleBlob::upsert` (lines 792–810) follows this sequence:

```rust
// (1) Strips leaf from cache; inserts leaf_index into free_indexes
self.block_status_cache.remove_leaf(&leaf)?;

// (2) Mutates leaf fields
leaf.hash.clone_from(new_hash);
leaf.value = value;
block.node = Node::Leaf(leaf);

// (3) Writes updated leaf back to the SAME leaf_index in the blob
self.insert_entry_to_blob(leaf_index, &block)?;

// (4) ← NO add_leaf() call here; cache remains inconsistent
``` [1](#0-0) 

`remove_leaf` performs three mutations on the cache:

```rust
fn remove_leaf(&mut self, node: &LeafNode) -> Result<(), Error> {
    let Some(index) = self.key_to_index.remove(&node.key) else { ... };
    self.leaf_hash_to_index.remove(&node.hash);
    self.free_indexes.insert(index);   // ← leaf_index now marked FREE
    Ok(())
}
``` [2](#0-1) 

After step (3), the blob at `leaf_index` contains valid, active leaf data. But the cache says:
- `leaf_index` ∈ `free_indexes` (available for reuse)
- key absent from `key_to_index`
- new hash absent from `leaf_hash_to_index`

The correct counterpart, `add_leaf`, which would undo all three mutations, is never called:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
``` [3](#0-2) 

`get_new_index()` unconditionally pops from `free_indexes` when it is non-empty:

```rust
fn get_new_index(&mut self) -> TreeIndex {
    match self.block_status_cache.pop_free_index() {
        None => { /* extend blob */ ... }
        Some(new_index) => new_index,   // ← returns the "free" leaf_index
    }
}
``` [4](#0-3) 

`pop_free_index` takes the first element of `free_indexes`:

```rust
fn pop_free_index(&mut self) -> Option<TreeIndex> {
    let maybe_index = self.free_indexes.iter().next().copied();
    if let Some(index) = maybe_index {
        self.free_indexes.shift_remove(&index);
    }
    maybe_index
}
``` [5](#0-4) 

So the very next `insert`, `batch_insert`, or any other operation that calls `get_new_index()` will receive `leaf_index` and overwrite the upserted leaf's blob bytes with a completely different node.

---

### Impact Explanation

The corruption manifests in multiple ways after a single `upsert`:

1. **Silent leaf overwrite / data loss**: The next `get_new_index()` returns `leaf_index`; a new node is written there, destroying the upserted key-value pair without any error.
2. **Duplicate key insertion**: Because `key_to_index` no longer contains the key, a subsequent `insert` with the same key passes the `contains_key` guard and succeeds, creating two blob entries for the same key. The tree structure becomes invalid.
3. **Duplicate hash insertion**: Same logic applies to `leaf_hash_to_index`; the same hash can be inserted again.
4. **Incorrect Merkle root**: Any of the above corruptions produces a wrong root hash, meaning inclusion and exclusion proofs computed from the tree are invalid — proofs can be forged for states that do not exist, or valid states cannot be proven.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

`upsert` is a first-class public API of `MerkleBlob` and is the standard path for updating an existing DataLayer key. Any DataLayer store operation that updates a key and then performs any further tree modification (insert, batch insert, another upsert, or delete) in the same `MerkleBlob` session will trigger the corruption. No special privileges are required; any DataLayer participant who can submit update transactions can reach this path.

---

### Recommendation

After writing the updated leaf back to the blob, call `add_leaf` to re-register the index as active in the cache:

```rust
pub fn upsert(&mut self, key: KeyId, value: ValueId, new_hash: &Hash) -> Result<(), Error> {
    let Ok((leaf_index, mut leaf, mut block)) = self.get_leaf_by_key(key) else {
        self.insert(key, value, new_hash, InsertLocation::Auto {})?;
        return Ok(());
    };

    // Remove old hash from cache (but do NOT add leaf_index to free_indexes)
    self.block_status_cache.remove_leaf(&leaf)?;
    leaf.hash.clone_from(new_hash);
    leaf.value = value;
    block.node = Node::Leaf(leaf.clone());
    self.insert_entry_to_blob(leaf_index, &block)?;

    // Re-register the index as active with the updated leaf data
    self.block_status_cache.add_leaf(leaf_index, leaf);  // ← ADD THIS

    if let Some(parent) = block.node.parent().0 {
        self.mark_lineage_as_dirty(parent)?;
    }
    Ok(())
}
```

Alternatively, refactor `upsert` to avoid calling `remove_leaf` at all and instead update only the hash fields in the cache directly, avoiding the intermediate inconsistent state.

### Proof of Concept

```rust
// 1. Build a tree with two leaves so upsert takes the update path (not insert)
let mut blob = MerkleBlob::new(vec![]).unwrap();
blob.insert(KeyId(1), ValueId(10), &hash_of(1), InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(2), ValueId(20), &hash_of(2), InsertLocation::Auto {}).unwrap();

// 2. Upsert key 1 — this leaves leaf_index in free_indexes
blob.upsert(KeyId(1), ValueId(99), &hash_of(99)).unwrap();

// 3. Insert a new key — get_new_index() pops the "free" leaf_index and
//    overwrites the upserted leaf's blob bytes
blob.insert(KeyId(3), ValueId(30), &hash_of(3), InsertLocation::Auto {}).unwrap();

// 4. Key 1 is now inaccessible or points to corrupted data;
//    check_integrity() will panic/error because an active leaf is in free_indexes
//    and key_to_index is missing the entry.
blob.check_integrity().unwrap(); // ← FAILS
