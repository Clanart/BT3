### Title
Cairo 0 class from reverted block served from LRU cache without declaration check — (`crates/blockifier/src/state/state_reader_and_contract_manager.rs`)

### Summary
`get_compiled_from_class_manager` skips all declaration validation for `RunnableCompiledClass::V0` cache hits. A Cairo 0 class cached during a subsequently-reverted transaction or block remains in the `GlobalContractCache` LRU and is returned unconditionally to any caller of `get_compiled_class`, bypassing the committed-state invariant.

### Finding Description
The match arm `RunnableCompiledClass::V0(_) => {}` at line 74 is an intentional no-op: `is_declared` is not the right check for V0 (it always returns `false`). However, no substitute check is performed. The cache-miss path correctly calls `state_reader.get_compiled_classes`, which fails for undeclared classes, but the cache-hit path for V0 has no equivalent guard. Because `GlobalContractCache` is a process-wide LRU that survives block and transaction reverts, a V0 class written to the cache during block N (which is later reverted) persists and is returned as `Ok(RunnableCompiledClass::V0(...))` for any subsequent call with the same class hash, regardless of whether the class is present in the current committed state. [6](#0-5) 

### Impact Explanation
Any code path that calls `get_compiled_class` — including RPC fee estimation, `starknet_call`, `starknet_simulateTransactions`, and `starknet_traceTransaction` — will receive and execute the stale V0 class as if it were declared. This produces authoritative-looking wrong execution results: wrong fee estimates, wrong traces, and wrong simulation outputs. For the sequencer's own block execution, a `deploy` syscall referencing the reverted class hash would succeed, producing a state diff that deploys a contract against a class hash not present in the declared-classes trie — an inconsistency between execution output and committed state.

### Likelihood Explanation
Cairo 0 class declarations are rare but still valid on mainnet. Transaction-level reverts (the more common trigger) are routine. The LRU cache is shared across the process lifetime, so a single reverted declare transaction is sufficient to poison the cache for the duration of the process or until the entry is evicted. An attacker who can submit a declare transaction and then cause or observe its revert can reliably trigger this with a follow-up transaction.

### Recommendation
For the V0 cache-hit branch, add a state-reader-level existence check equivalent to what the cache-miss path already does. Since `is_declared` is not applicable to V0, the check should verify the class exists in committed state — for example by calling `get_compiled_classes` on the state reader (discarding the result) or by adding a dedicated `is_class_in_state(class_hash)` predicate to `FetchCompiledClasses` that works for both V0 and V1. The comment at lines 76–78 already documents the correct rationale; it simply needs to be applied to V0 as well.

### Proof of Concept
1. Pre-seed `GlobalContractCache` with a `CompiledClasses::V0(...)` entry for class hash `H`.
2. Configure the `FetchCompiledClasses` mock so `is_declared(H)` returns `false` and `get_compiled_classes(H)` returns `Err(UndeclaredClassHash(H))`.

4. Call `get_compiled_class(H)`.
5. Observe: the function returns `Ok(RunnableCompiledClass::V0(...))` instead of `Err(UndeclaredClassHash(H))`, because the V0 cache-hit arm at line 74 performs no check and the state reader is never consulted. [7](#0-6)
