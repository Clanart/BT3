### Title
Unconstrained `is_reverted` Flag Allows Malicious Prover to Skip Invoke Transaction Execution — (File: `execution/execution_constraints.cairo`)

---

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is a no-op stub that performs zero validation. The `is_reverted` flag, filled entirely by an untrusted hint, is left unconstrained by the Cairo constraint system. A malicious prover can set `is_reverted` to any non-zero value for any invoke transaction, causing the execute step to be silently skipped while the fee is still charged and the block proof remains valid.

---

### Finding Description

In `execute_invoke_function_transaction`, the `is_reverted` flag is loaded from a hint and then passed to `check_is_reverted`: [1](#0-0) 

The called function is: [2](#0-1) 

`check_is_reverted` accepts `is_reverted: felt` and immediately returns, adding **no Cairo constraints whatsoever**. Because the value is filled by `%{ IsReverted %}` (a prover-controlled hint) and no assertion constrains it, the prover is free to assign any field element. Setting `is_reverted = 1` causes the `if (is_reverted == FALSE)` branch to be skipped, meaning `non_reverting_select_execute_entry_point_func` is never called: [3](#0-2) 

The fee charge still executes unconditionally after the branch: [4](#0-3) 

The same pattern exists for L1 handler transactions, where `is_reverted` is loaded from a hint with **no call to `check_is_reverted` at all**, and an early return skips `consume_l1_to_l2_message`: [5](#0-4) [6](#0-5) 

This is the direct analog of the external report: just as `rs2_as = 1` can be set in a dummy row (`is_valid = 0`) where the execution bridge is disabled and the timestamp/address/value are unconstrained, here `is_reverted = 1` can be set in any invoke row where `check_is_reverted` is disabled (a no-op), leaving the execute step unconstrained.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A malicious prover can target any invoke transaction that performs a token transfer (e.g., ERC-20 `transfer`). By setting `is_reverted = 1`:

- The execute entry point is skipped; the recipient's balance is never updated.
- The fee transfer (`charge_fee`) still executes; the sender's balance is debited for the fee.
- The resulting proof is valid and accepted by the L1 verifier, because no constraint is violated.

The recipient permanently loses the expected tokens. The sender loses the fee for a transaction that produced no effect. This is a controlled, targeted, and repeatable attack against any account holding funds on StarkNet.

---

### Likelihood Explanation

Any entity that controls block production and proof generation (i.e., any sequencer/prover in a decentralized StarkNet) can exploit this. No special knowledge, leaked keys, or external dependencies are required. The attack applies to every invoke transaction in every block and requires only setting a single hint value to a non-zero field element. The attack is analogous to the external report's finding, where "all that is needed are 4 extra rows in any of the ALU chips."

---

### Recommendation

Implement `check_is_reverted` to enforce that `is_reverted` is consistent with the actual execution outcome. At minimum, assert that `is_reverted` is a boolean (0 or 1), and add a constraint tying its value to a verifiable execution artifact (e.g., the revert log being non-empty, or a flag produced by the entry-point runner). The fix is analogous to the external report's recommendation:

```cairo
// Ensure is_reverted is boolean.
assert is_reverted * (1 - is_reverted) = 0;
// Add further constraints linking is_reverted to the actual execution result.
```

---

### Proof of Concept

A malicious prover targets an ERC-20 `transfer` invoke transaction:

1. The prover fills `%{ IsReverted %}` with `1` instead of `0`.
2. `check_is_reverted(1)` is called and returns immediately — no constraint fires.
3. The `if (is_reverted == FALSE)` branch is not taken; `non_reverting_select_execute_entry_point_func` is never called.
4. `charge_fee` executes normally, debiting the sender.
5. The state diff written to the OS output contains no storage update for the recipient's balance.
6. The proof is generated and verified on L1 without error.
7. The recipient's balance on L2 is unchanged; the transfer never happened.
