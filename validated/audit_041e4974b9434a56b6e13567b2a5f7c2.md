### Title
Hardcoded `EvmFinality::Latest` for Abstract Chain in MPC Prover Accepts Non-Finalized Blocks, Enabling Reorg-Based Unauthorized Minting — (File: `near/omni-prover/mpc-omni-prover/src/lib.rs`)

---

### Summary

The `MpcOmniProver` contract is initialized with `EvmFinality::Latest` for `ChainKind::Abs` (Abstract EVM chain). This is the weakest possible EVM finality tag — it accepts proofs from the very latest block, which has no reorg protection. If the MPC contract signs a proof for a transaction that is subsequently reorged out of the source chain, the bridge will still finalize the transfer on NEAR, minting or releasing assets that were never durably locked on the source chain.

---

### Finding Description

In `near/omni-prover/mpc-omni-prover/src/lib.rs`, the `init` function hardcodes the finality for the Abstract EVM chain as `EvmFinality::Latest`:

```rust
finalities.insert(ChainKind::Abs, MpcFinality::Evm(EvmFinality::Latest));
``` [1](#0-0) 

The `verify_proof` function enforces that the submitted request's finality tag matches the configured finality:

```rust
require!(
    Self::request_matches_finality(&payload_v1.request, finality),
    ProverError::FinalityMismatch.as_ref()
);
``` [2](#0-1) 

`request_matches_finality` checks that the request's `finality` field equals the stored finality:

```rust
(ForeignChainRpcRequest::Ethereum(args) | ForeignChainRpcRequest::Abstract(args),
 MpcFinality::Evm(finality)) => args.finality == *finality,
``` [3](#0-2) 

Because the stored finality is `EvmFinality::Latest`, the prover **only accepts** requests tagged `Latest` and **rejects** `Finalized` or `Safe`. This means the MPC contract is asked to verify and sign a transaction that exists only in the latest (non-finalized) block. Once the MPC contract returns a valid signature, the proof is consumed by the NEAR bridge's `fin_transfer` flow, which releases or mints assets on NEAR.

The `set_finality` function that could correct this is marked `#[private]`, meaning it can only be called by the contract itself — there is no admin path to upgrade the finality level without a contract migration:

```rust
#[private]
pub fn set_finality(&mut self, chain_kind: ChainKind, finality: MpcFinality) {
    self.finalities.insert(chain_kind, finality);
}
``` [4](#0-3) 

The test suite confirms `EvmFinality::Latest` is the expected and accepted finality for Abstract chain requests:

```rust
fn abs_testnet_evm_request() -> EvmRpcRequest {
    EvmRpcRequest {
        ...
        finality: EvmFinality::Latest,
    }
``` [5](#0-4) 

---

### Impact Explanation

If a source-chain block on Abstract is reorged after the MPC contract has signed the proof payload:

1. The `InitTransfer` event no longer exists on the canonical chain — the user's tokens were never actually locked or burned.
2. The MPC signature is already issued and is valid.
3. A relayer (or the original submitter) can still call `fin_transfer` on NEAR using the signed proof.
4. NEAR's bridge releases or mints the corresponding bridged tokens to the recipient.

The result is **unbacked minting of bridged assets on NEAR** — tokens exist on NEAR with no corresponding locked collateral on Abstract. This directly matches the Critical impact class: *unauthorized mint of bridged assets*.

---

### Likelihood Explanation

Abstract is an EVM-compatible ZK-rollup L2. While L2 reorgs are less frequent than L1 reorgs, they are not impossible:
- The sequencer can reorder or drop transactions before they are committed to L1.
- Early blocks (before L1 data availability confirmation) are particularly susceptible.
- A sophisticated attacker who controls or can influence the sequencer, or who can time a proof submission to a block that is later reorganized, can exploit this.

The attack requires no privileged role on the NEAR side — any relayer or proof submitter can call `verify_proof` with a `Latest`-tagged request. The only external dependency is a reorg on the Abstract chain, which, while not trivial, is a realistic edge/attack case (exactly as described in the original report).

---

### Recommendation

Replace `EvmFinality::Latest` with `EvmFinality::Finalized` (or at minimum `EvmFinality::Safe`) for `ChainKind::Abs` in the `init` function:

```rust
// Before (vulnerable):
finalities.insert(ChainKind::Abs, MpcFinality::Evm(EvmFinality::Latest));

// After (recommended):
finalities.insert(ChainKind::Abs, MpcFinality::Evm(EvmFinality::Finalized));
```

Additionally, expose a privileged admin function (e.g., gated by `DEFAULT_ADMIN_ROLE` or DAO) to update finality settings without requiring a full contract migration, so the finality level can be adjusted as chain conditions evolve.

---

### Proof of Concept

1. Attacker initiates a transfer on Abstract chain: calls `initTransfer` on the Abstract `OmniBridge`, emitting an `InitTransfer` event in block `N` (latest, not finalized).
2. Attacker (or any relayer) immediately submits a `verify_proof` call to `MpcOmniProver` with an `EvmRpcRequest { finality: EvmFinality::Latest, ... }` referencing block `N`.
3. `MpcOmniProver.verify_proof` passes the `request_matches_finality` check (configured finality is `Latest`, request finality is `Latest` — match).
4. The MPC contract verifies the transaction exists in the latest block and returns a signed `VerifyForeignTransactionResponse`.
5. Block `N` is reorged on Abstract chain — the `InitTransfer` event never happened on the canonical chain; the attacker's tokens were never locked.
6. The MPC signature is already issued. The attacker calls `fin_transfer` on the NEAR bridge with the signed proof.
7. NEAR's `fin_transfer_callback` decodes the `InitTransfer` proof result and mints/releases tokens to the attacker's NEAR address.
8. Attacker holds bridged tokens on NEAR backed by nothing on Abstract. [6](#0-5) [7](#0-6) [8](#0-7)

    pub fn verify_proof(&self, #[serializer(borsh)] input: Vec<u8>) -> Promise {
        let args = MpcVerifyProofArgs::try_from_slice(&input).near_expect(ProverError::ParseArgs);

        let sign_payload = ForeignTxSignPayload::try_from_slice(&args.sign_payload)
            .near_expect(ProverError::ParseArgs);

        let ForeignTxSignPayload::V1(ref payload_v1) = sign_payload;

        let chain_kind = Self::request_to_chain_kind(&payload_v1.request)
            .near_expect(ProverError::UnsupportedChain);

        let finality = self
            .finalities
            .get(&chain_kind)
            .near_expect(ProverError::UnsupportedChain);

        require!(
            Self::request_matches_finality(&payload_v1.request, finality),
            ProverError::FinalityMismatch.as_ref()
        );

        let request_args = VerifyForeignTransactionRequestArgs {
            request: payload_v1.request.clone(),
            domain_id: DomainId(FOREIGN_TX_DOMAIN_ID),
            payload_version: ForeignTxPayloadVersion::V1,
        };

        ext_mpc_contract::ext(self.mpc_contract_id.clone())
            .with_static_gas(VERIFY_FOREIGN_TX_GAS)
            .with_attached_deposit(ONE_YOCTO)
            .verify_foreign_transaction(request_args)
            .then(
                Self::ext(near_sdk::env::current_account_id())
                    .with_static_gas(VERIFY_CALLBACK_GAS)
                    .verify_callback(args.proof_kind, args.sign_payload, chain_kind),
            )
    }

            PromiseOrValue::Value(destination_nonce)
