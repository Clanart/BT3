### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Signature Replay for Meta-Transactions — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` computes the transaction hash for meta-tx v0 syscalls without including any nonce. Combined with the OS explicitly skipping nonce checking and `__validate__` for version-0 transactions, a valid meta-tx v0 signature can be replayed an unlimited number of times against the same target contract, enabling direct loss of funds.

---

### Finding Description

`compute_meta_tx_v0_hash` in `transaction_hash/transaction_hash.cairo` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),  // ← no nonce
    );
    return tx_hash;
}
``` [1](#0-0) 

The resulting hash is a pure function of `(prefix, version=0, contract_address, selector, calldata, max_fee=0, chain_id)` — it is **identical** for every invocation with the same calldata.

Two OS-level mechanisms that would normally prevent replay are both disabled for version-0 transactions:

**1. Nonce checking is skipped for version 0:**

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
``` [2](#0-1) 

**2. `__validate__` is skipped for version 0:**

```cairo
// Do not run "__validate__" for version 0.
if (tx_execution_info.tx_info.version == 0) {
    return ();
}
``` [3](#0-2) 

**3. The meta-tx v0 execution context hardcodes `nonce=0`:**

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,
    ...
    transaction_hash=meta_tx_hash,
    ...
);
``` [4](#0-3) 

The `execute_meta_tx_v0` syscall is callable by any contract. It accepts an attacker-supplied `signature`, computes the nonce-less hash, and calls the target contract's `__execute__` entry point with that hash exposed via `get_tx_info()`. Any account contract whose `__execute__` verifies the signature against `tx_info.transaction_hash` will accept the same signature on every replay, because the hash never changes for the same `(contract_address, calldata)` pair. [5](#0-4) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

An attacker who observes any successful meta-tx v0 call (e.g., from a relayer, bundler, or on-chain event) obtains a `(calldata, signature)` pair whose hash is permanently valid. Because the OS never increments a nonce for version-0 transactions and the hash is nonce-free, the attacker can replay the same syscall indefinitely. If the target account contract holds or controls funds (ETH, ERC-20 tokens, or any asset), those funds can be fully drained through repeated replay.

---

### Likelihood Explanation

Meta-tx v0 is a first-class OS syscall designed for relayer/bundler use cases, meaning valid meta-tx v0 calls will appear on-chain regularly. Any on-chain observer can extract the `(calldata, signature)` from a successful call and immediately replay it. No privileged access, leaked key, or social engineering is required — only the ability to submit a transaction that invokes the `meta_tx_v0` syscall.

---

### Recommendation

Include a nonce in `compute_meta_tx_v0_hash`. The nonce should be read from the target contract's current on-chain nonce and passed as `additional_data`, then incremented after the call — mirroring how `check_and_increment_nonce` works for version ≥ 1 transactions:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
+   nonce: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
-       additional_data_size=0,
-       additional_data=cast(0, felt*),
+       additional_data_size=1,
+       additional_data=&nonce,
    );
    return tx_hash;
}
```

Additionally, `execute_meta_tx_v0` must read the target contract's current nonce, pass it into the hash, and increment it in `contract_state_changes` before returning — analogous to `check_and_increment_nonce`.

---

### Proof of Concept

1. Victim account contract `A` at address `0xA` holds funds. It implements `__execute__` to verify a signature over `get_tx_info().transaction_hash`.
2. A legitimate relayer submits a meta-tx v0 syscall with `contract_address=0xA`, `calldata=[transfer, attacker, 1_token]`, and a valid `signature`. The OS computes `H = H(invoke, 0, 0xA, __execute__, calldata, 0, chain_id)` — no nonce. The call succeeds.
3. Attacker observes the on-chain call, extracts `(calldata, signature)`.
4. Attacker deploys a contract that calls `meta_tx_v0(contract_address=0xA, calldata=same_calldata, signature=same_signature)`.
5. The OS recomputes the identical hash `H` (nonce is absent), the target contract's `__validate__` is skipped (version 0), and `__execute__` is called with the same valid signature.
6. The transfer executes again. Steps 4–6 repeat until `A` is drained. [1](#0-0) [2](#0-1) [6](#0-5)
