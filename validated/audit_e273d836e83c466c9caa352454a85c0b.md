### Title
Child `ChunkExtra.proposed_split` Inherits Parent's Non-None Split Proposal After Shard Split, Causing Permanent First-Chunk Rejection for Child Shards - (File: chain/chain/src/resharding/manager.rs)

### Summary

During dynamic resharding, `process_memtrie_resharding_storage_update` creates child `ChunkExtra` records by cloning the parent's `ChunkExtra` and updating only `state_root` and `congestion_info`. The `proposed_split` field â€” which holds the `TrieSplit` that triggered the resharding â€” is never reset to `None`. Every child shard therefore starts epoch N+2 with `ChunkExtra.proposed_split = Some(TrieSplit{...})`. When the first chunk of a child shard is produced, `compute_proposed_split` returns `None` (the child is new and `can_reshard` is false). The header carries `proposed_split = None`, but `validate_chunk_with_chunk_extra_and_receipts_root` compares it against the stored `prev_chunk_extra.proposed_split() = Some(TrieSplit{...})` and returns `InvalidChunkHeaderShardSplit`, permanently rejecting every first chunk of every child shard.

### Finding Description

In `process_memtrie_resharding_storage_update`, the child `ChunkExtra` is built as:

```rust
// TODO(resharding): set all fields of `ChunkExtra`. Consider stronger
// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
``` [1](#0-0) 

Only `state_root` and `congestion_info` are patched. The `proposed_split` field â€” defined in `ChunkExtraV5` â€” is silently inherited from the parent: [2](#0-1) 

At the resharding boundary block, the parent's `ChunkExtra.proposed_split` is `Some(TrieSplit{boundary_account, left_memory, right_memory})` â€” the exact value that caused the split to be selected. The child `ChunkExtra` saved to the store carries this same non-`None` value.

In epoch N+2, when the first chunk of a child shard is produced, `compute_proposed_split` is called: [3](#0-2) 

`can_reshard` returns `false` (the resharding cooldown has not elapsed), so `compute_proposed_split` returns `None`. The chunk header is signed with `proposed_split = None`.

Chunk validation then calls `validate_chunk_with_chunk_extra_and_receipts_root`, which enforces:

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    return Err(Error::InvalidChunkHeaderShardSplit(...));
}
``` [4](#0-3) 

`prev_chunk_extra.proposed_split()` is `Some(TrieSplit{...})` (inherited from the parent); `chunk_header.proposed_split()` is `None` (freshly computed). The mismatch is permanent: every subsequent chunk attempt produces the same `None` from `compute_proposed_split`, while the stored `ChunkExtra` never changes. The child shard is stuck.

The codebase itself documents this gap:

> **`chain/chain/src/resharding/manager.rs:249`** â€” The resharding manager doesn't set all `ChunkExtra` fields (notably the new `proposed_split` field).



### Impact Explanation

After any dynamic resharding event, both child shards are unable to produce their first valid chunk. Because the mismatch is deterministic and the stored `ChunkExtra` is never corrected, no subsequent attempt succeeds either. The child shards are permanently halted at the epoch boundary, making the resharding feature self-defeating: every split produces two dead shards. This is a **Critical** correctness failure in the dynamic resharding pipeline.

### Likelihood Explanation

The bug fires on every dynamic resharding event without exception. It requires no adversarial input â€” the protocol's own memory-threshold logic triggers the split, and the broken `ChunkExtra` clone is the only code path that initialises child shard state. Any deployment that enables `ProtocolFeature::DynamicResharding` will hit this on the first resharding.

### Recommendation

After cloning the parent `ChunkExtra`, explicitly reset `proposed_split` to `None` for each child shard:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// Child shards start fresh; they have not yet proposed any split.
*child_chunk_extra.proposed_split_mut() = None;
```

This mirrors the pattern already used for `congestion_info`, which is also computed from scratch rather than blindly inherited.

### Proof of Concept

1. Enable `ProtocolFeature::DynamicResharding` and configure a low `memory_usage_threshold` so a shard crosses it quickly.
2. Observe the parent shard's `ChunkExtra.proposed_split` become `Some(TrieSplit{...})` near the epoch boundary.
3. At the epoch-N last block, `process_memtrie_resharding_storage_update` saves child `ChunkExtra` records cloned from the parent â€” both carry `proposed_split = Some(TrieSplit{...})`.
4. In epoch N+2, the first chunk producer for either child shard calls `compute_proposed_split`; `can_reshard` returns `false`; the result is `None`; the chunk header is signed with `proposed_split = None`.
5. `validate_chunk_with_chunk_extra_and_receipts_root` compares `prev_chunk_extra.proposed_split() = Some(...)` against `chunk_header.proposed_split() = None` and returns `InvalidChunkHeaderShardSplit`.
6. Every subsequent chunk attempt for the child shard produces the same mismatch. The child shard never advances past its first block. [5](#0-4) [4](#0-3) [6](#0-5)
