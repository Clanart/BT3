### Title
Unchecked `max_price_per_unit` Upper Bound Allows `compute_max_possible_fee` to Exceed 128-bit Range, Making Blocks Unprovable — (File: `execution/transaction_impls.cairo`)

---

### Summary

In `charge_fee`, the function `compute_max_possible_fee` multiplies user-controlled `max_amount` (bounded to ≤ 2^64−1) by `max_price_per_unit` (bounded only to < 2^128 by `assert_nn`) in felt arithmetic. The product can reach ~2^192, far exceeding 2^128. The subsequent `assert_nn_le(calldata.amount.low, max_fee)` requires both arguments to satisfy range-check constraints (< 2^128). When `max_fee ≥ 2^129 − 1`, no valid `low_actual_fee` exists that satisfies the constraint, making the block permanently unprovable.

---

### Finding Description

In `compute_max_possible_fee`:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

The only validation of resource bounds occurs in `pack_resource_bounds` (called during transaction hash computation):

```cairo
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
assert_nn(resource_bounds.max_price_per_unit);
``` [2](#0-1) 

`assert_nn` checks `0 ≤ max_price_per_unit < 2^128`, but places **no upper bound** on the product `max_amount × max_price_per_unit`. With `max_amount` up to 2^64−1 and `max_price_per_unit` up to 2^128−1, the product can reach ~2^192.

In `charge_fee`, the cap check is:

```cairo
local low_actual_fee;
%{ LoadActualFee %}
local calldata: TransferCallData = TransferCallData(
    recipient=block_context.block_info_for_execute.sequencer_address,
    amount=Uint256(low=low_actual_fee, high=0),
);
assert_nn_le(calldata.amount.low, max_fee);
``` [3](#0-2) 

`assert_nn_le(a, b)` internally calls `assert_nn(a)` and `assert_nn(b − a)`. Both require the value to fit in a range-check cell (< 2^128). For `assert_nn(b − a)` to pass, we need `b − a < 2^128`, i.e., `low_actual_fee > max_fee − 2^128`. But `low_actual_fee` is also constrained to be < 2^128 (it is the `low` limb of a `Uint256`). Therefore, a valid `low_actual_fee` exists only if `max_fee < 2^129 − 1`. When `max_fee ≥ 2^129 − 1`, no valid `low_actual_fee` satisfies both constraints simultaneously, and the proof is permanently invalid.

---

### Impact Explanation

If a sequencer includes a transaction with resource bounds that cause `max_fee ≥ 2^129 − 1`, the OS Cairo program cannot produce a valid proof for that block. The block is permanently unprovable. The sequencer must detect and exclude such transactions before block construction; if it does not, the network cannot confirm new transactions until the sequencer is patched or the transaction is manually excluded.

This matches: **High — Network not being able to confirm new transactions (total network shutdown).**

---

### Likelihood Explanation

Any unprivileged transaction sender can craft a V3 transaction with:
- `max_amount = 3` (well within the 2^64−1 limit)
- `max_price_per_unit = 2^128 − 1` (passes `assert_nn`, since 2^128−1 < 2^128)

This yields a product of `3 × (2^128 − 1) = 3 × 2^128 − 3 > 2^129 − 1`, triggering the failure. The transaction passes all hash-level validation (the transaction hash is computed and verified successfully), so the sequencer has no protocol-level signal to reject it unless it explicitly validates `max_amount × max_price_per_unit < 2^129 − 1` off-chain.

---

### Recommendation

In `compute_max_possible_fee` (or in `pack_resource_bounds`), add an explicit upper bound on `max_price_per_unit` such that the product `max_amount × max_price_per_unit` is guaranteed to fit within 128 bits. For example, enforce `max_price_per_unit ≤ 2^64 − 1` (matching the bound on `max_amount`), ensuring the product is at most `(2^64−1)^2 < 2^128`. Alternatively, compute `max_fee` using `Uint256` arithmetic and compare using `uint256_le` rather than `assert_nn_le`.

---

### Proof of Concept

1. Attacker submits a V3 transaction with:
   - `l1_gas_bounds.max_amount = 3`
   - `l1_gas_bounds.max_price_per_unit = 2^128 − 1`
   - All other bounds set to 0
2. `pack_resource_bounds` validates: `assert_nn_le(3, 2^64−1)` ✓ and `assert_nn(2^128−1)` ✓ — transaction hash is accepted.
3. Sequencer includes the transaction in a block.
4. `compute_max_possible_fee` computes: `3 × (2^128 − 1) = 3 × 2^128 − 3`.
5. `charge_fee` calls `assert_nn_le(low_actual_fee, 3 × 2^128 − 3)`.
6. For this to pass, `low_actual_fee > 3 × 2^128 − 3 − (2^128 − 1) = 2 × 2^128 − 2 = 2^129 − 2`. But `low_actual_fee < 2^128`. No valid value exists.
7. The range-check constraint is violated; the block proof is invalid.
8. The network cannot finalize the block. [4](#0-3) [3](#0-2) [5](#0-4)

// Returns the maximum possible fee that can be charged for the transaction.
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```

func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
