### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezal of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary
The `execute_replace_class` syscall handler in the StarkNet OS accepts a caller-controlled `class_hash` from the `ReplaceClassRequest` without validating that the hash corresponds to a previously declared contract class. An explicit `TODO` in the code acknowledges this missing check. A malicious contract deployer (an unprivileged actor) can exploit this to permanently freeze all funds held by a contract by replacing its class with an arbitrary, undeclared class hash, rendering the contract permanently unexecutable.

### Finding Description

In `execute_replace_class`, the `class_hash` field is read directly from the syscall request and written into `contract_state_changes` without any check against the `contract_class_changes` dictionary (which maps declared class hashes to their compiled class hashes):

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

When a subsequent call is made to the contract, `execute_entry_point` reads the class hash from `contract_state_changes` and performs a lookup in `contract_class_changes` to obtain the compiled class hash:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
``` [2](#0-1) 

If the `class_hash` was never declared, `dict_read` returns the default value `0` (since `contract_class_changes` is initialized via `dict_new()` with default 0). `find_element` then searches for compiled class hash `0` in the bundle. Since no such entry exists, `find_element` fails with an assertion error. Every subsequent call to the contract reverts, making it permanently unexecutable.

The `contract_class_changes` dictionary is initialized with `dict_new()` and only populated when classes are declared: [3](#0-2) 

The `execute_declare_transaction` function is the only place that writes to `contract_class_changes` via `dict_update`, confirming that undeclared class hashes are absent from this dictionary: [4](#0-3) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract holding user assets (ERC-20 balances, NFTs, protocol TVL) that calls `replace_class` with an arbitrary undeclared class hash becomes permanently unexecutable. All funds locked in the contract are irrecoverably frozen. The OS provides no mechanism to recover from this state once the invalid class hash is committed to the state trie.

### Likelihood Explanation

**Medium.** The attack requires deploying a malicious contract that attracts user deposits before triggering `replace_class` with an invalid hash. This is a realistic rug-pull vector enabled by the missing OS-level invariant. Any unprivileged user can deploy a contract on StarkNet. The explicit `TODO` comment at line 898 confirms the developers are aware this check is absent, meaning the invariant is not enforced at the protocol level. [5](#0-4) 

### Recommendation

Before writing the new class hash into `contract_state_changes`, validate that the requested `class_hash` exists in `contract_class_changes` (i.e., has been declared). This is exactly what the existing TODO describes. The check should use `dict_read` on `contract_class_changes` and assert the returned compiled class hash is non-zero:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the fix recommended in the external report: maintain a list of approved/declared entries and validate the caller-supplied input against it before acting on it.

### Proof of Concept

1. Attacker deploys contract `MaliciousVault` with a `drain()` function that internally issues the `replace_class` syscall with `class_hash = 0xdeadbeef` (an arbitrary undeclared felt).
2. Users deposit tokens into `MaliciousVault` (e.g., believing it is a legitimate yield vault).
3. Attacker calls `drain()`. The OS executes `execute_replace_class` with `class_hash = 0xdeadbeef`. No validation is performed. The contract's state entry is updated: `class_hash = 0xdeadbeef`.
4. Any subsequent call to `MaliciousVault` reaches `execute_entry_point`. `dict_read(contract_class_changes, 0xdeadbeef)` returns `0`. `find_element(compiled_class_facts_bundle, key=0)` fails with an assertion error.
5. All calls to `MaliciousVault` permanently revert. All deposited user funds are frozen with no recovery path. [6](#0-5)
