### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Causes Bridge Undercollateralization — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The EVM `OmniBridge.initTransfer` function records and broadcasts the caller-supplied `amount` as the bridged quantity, but performs no balance-difference check after the `safeTransferFrom` call. For ERC20 tokens that deduct a fee on every transfer (fee-on-transfer tokens), the bridge receives fewer tokens than `amount`, yet the cross-chain message instructs the destination chain to release the full `amount`. This permanently undercollateralizes the bridge for every such deposit.

---

### Finding Description

In `OmniBridge.initTransfer`, when the deposited token is a plain ERC20 (neither a bridge token nor a custom-minter token), the code executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // ← caller-supplied, not verified post-transfer
);
``` [1](#0-0) 

Immediately after, the same unverified `amount` is forwarded to `initTransferExtension` and emitted in the `InitTransfer` event:

```solidity
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message, extensionValue
);
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message
);
``` [2](#0-1) 

In the Wormhole variant, `initTransferExtension` publishes a Wormhole message containing the unchecked `amount`:

```solidity
Borsh.encodeUint128(amount),
``` [3](#0-2) [4](#0-3) 

There is no pre/post balance snapshot to detect the discrepancy. The bridge assumes `safeTransferFrom` always delivers exactly `amount` tokens, which is false for fee-on-transfer tokens.

---

### Impact Explanation

Every `initTransfer` call with a fee-on-transfer ERC20 token causes the bridge to:

1. Hold `amount - transfer_fee` tokens in custody.
2. Instruct the destination chain (NEAR or another EVM chain) to release `amount` tokens to the recipient.

The bridge is undercollateralized by `transfer_fee` per deposit. Repeated deposits accumulate the deficit. Eventually, legitimate withdrawal finalizations (`finTransfer`) will fail because the bridge's ERC20 balance is insufficient to cover the full `payload.amount` at line 351:

```solidity
IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
``` [5](#0-4) 

This matches the allowed impact: **Balance/accounting corruption that breaks bridge collateralization**, and in the worst case, **permanent freezing of user funds** when the bridge cannot fulfill withdrawals.

---

### Likelihood Explanation

- Fee-on-transfer tokens are a well-known ERC20 pattern (USDT has a configurable fee switch; tokens like STA, PAXG, and others charge fees by design).
- No privileged access is required. Any user can call `initTransfer` with any ERC20 address that is not a bridge token or custom-minter token.
- The bridge does not whitelist which ERC20 tokens may be deposited via `initTransfer`; the only guard is that the token is not already a `isBridgeToken` or `customMinters` entry.
- The deficit accumulates silently with each deposit; no single transaction is obviously malicious.

---

### Recommendation

Record the bridge's token balance before and after the `safeTransferFrom` call, and use the actual received amount for all downstream accounting:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// Use actualReceived (cast to uint128 with overflow check) instead of amount
```

Alternatively, maintain an explicit allowlist of supported ERC20 tokens and reject any token not on the list.

---

### Proof of Concept

1. Deploy or use any ERC20 token that charges a 1% fee on every transfer (e.g., a token where `transfer(to, 1000)` delivers 990 to `to` and burns/redirects 10).
2. Call `OmniBridge.initTransfer(tokenAddress, 1000, 0, 0, "near-recipient.near", "")`.
3. The bridge receives 990 tokens but emits `InitTransfer` with `amount = 1000`.
4. The NEAR bridge processes the event and releases 1000 tokens to `near-recipient.near`.
5. Repeat N times. After N deposits the bridge holds `990*N` tokens but has promised `1000*N` tokens.
6. The (N+1)-th user who bridged back from NEAR calls `finTransfer` on EVM for 1000 tokens; the `safeTransfer` at line 351 reverts because the bridge is short by `10*N` tokens, permanently freezing that user's funds. [6](#0-5) [7](#0-6)
