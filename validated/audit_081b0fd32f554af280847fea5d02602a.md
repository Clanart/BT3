### Title
Inverted Callback Condition Sends Fee on BTC Connector Failure and Permanently Freezes User Funds on Success - (File: near/omni-bridge/src/btc.rs)

### Summary
`submit_transfer_to_btc_connector_callback` in `near/omni-bridge/src/btc.rs` uses an inverted condition to decide whether to send the relayer fee or restore the transfer message. Under NEAR's NEP-141 `ft_transfer_call` protocol, `result.0 > 0` means the BTC connector **returned** tokens (failure), not that it accepted them. The current code sends the fee on failure and restores the transfer message on success — the exact opposite of correct behavior. This causes permanent freezing of user funds whenever the BTC connector rejects a transfer.

### Finding Description

In `submit_transfer_to_utxo_chain_connector`, the transfer message is removed from `pending_transfers` before the `ft_transfer_call` is dispatched: [1](#0-0) 

The callback then decides what to do based on the connector's response: [2](#0-1) 

Under NEAR's NEP-141 standard, `ft_on_transfer` returns the amount to **refund** to the sender. Therefore:
- `result.0 == 0` → BTC connector kept all tokens → **transfer succeeded**
- `result.0 > 0` → BTC connector returned tokens → **transfer failed**

The current code does the opposite of what is correct:

| Actual result | Current action | Correct action |
|---|---|---|
| `result.0 > 0` (FAILURE) | `send_fee_internal` → fee paid to relayer | Restore transfer message for retry |
| `result.0 == 0` (SUCCESS) | `add_transfer_message` → message restored | `send_fee_internal` → fee paid to relayer |

The correct condition should be `result.0 == 0` (or equivalently `matches!(call_result, Ok(result) if result.0 == 0)`).

### Impact Explanation

**When the BTC connector rejects the transfer (`result.0 > 0`):**
1. The BTC connector returns `amount - fee` tokens to the bridge contract.
2. The transfer message was already removed at line 84 and is **not restored** by the callback.
3. The fee is sent to the relayer via `send_fee_internal`.
4. The user's tokens (`amount - fee`) are now held by the bridge contract with no corresponding transfer message. There is no function in the contract that allows recovery of tokens without a valid transfer message. **Funds are permanently frozen.**

**When the BTC connector accepts the transfer (`result.0 == 0`):**
1. The relayer receives no fee despite successfully completing the transfer.
2. The transfer message is re-inserted into `pending_transfers` via `add_transfer_message`.
3. The relayer can call `submit_transfer_to_utxo_chain_connector` again with the same `transfer_id`. Since the bridge contract may hold other users' locked tokens, a malicious trusted relayer could drain them by repeatedly re-submitting.

The primary unconditional impact is **permanent freezing of user funds** on any BTC connector failure, which matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation

The BTC connector can legitimately fail for multiple reasons: invalid UTXO parameters, connector-side validation errors, gas exhaustion in the connector, or transient state issues. None of these require a malicious actor. Any such failure deterministically triggers the inverted callback, permanently locking the user's tokens. The `send_fee_internal` path also calls `unlock_tokens_if_needed` on the destination chain accounting, further corrupting the locked-token ledger. [3](#0-2) 

### Recommendation

Invert the condition in `submit_transfer_to_btc_connector_callback`:

```rust
// BEFORE (buggy):
if matches!(call_result, Ok(result) if result.0 > 0) {
    // send fee  ← wrong: this fires on failure
} else {
    // restore transfer message  ← wrong: this fires on success
}

// AFTER (correct):
if matches!(call_result, Ok(result) if result.0 == 0) {
    // send fee  ← correct: fires when connector kept all tokens (success)
} else {
    // restore transfer message  ← correct: fires when connector returned tokens (failure)
}
``` [2](#0-1) 

### Proof of Concept

1. User initiates a NEAR→BTC transfer with `fee = F`, `amount = A`. Transfer message is stored in `pending_transfers`.
2. Trusted relayer calls `submit_transfer_to_utxo_chain_connector(transfer_id, msg, ...)`.
3. Contract removes the transfer message at line 84, then calls `ft_transfer_call(btc_connector, A - F, ...)`.
4. The BTC connector's `ft_on_transfer` fails validation (e.g., invalid UTXO input) and returns `A - F` (full refund). `result.0 = A - F > 0`.
5. `submit_transfer_to_btc_connector_callback` is called with `result.0 > 0`.
6. The condition `result.0 > 0` is **true**, so `send_fee_internal` is called: fee `F` is minted/transferred to the relayer.
7. The `else` branch (restore transfer message) is **not** taken.
8. The bridge contract now holds `A - F` tokens returned by the connector, but no transfer message exists for this transfer. The user cannot reclaim their `A - F` tokens. Funds are permanently frozen. [4](#0-3)
