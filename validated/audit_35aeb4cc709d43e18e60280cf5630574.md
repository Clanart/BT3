### Title
EIP-712 Domain Chain ID Silently Truncated via `Uint64()` Enables Cross-Chain Replay for Large Chain IDs - (File: `ethereum/eip712/encoding.go`, `ante/cosmos/eip712.go`)

### Summary
Ethermint's EIP-712 signing path converts the chain ID from `*big.Int` to `uint64` via `.Uint64()` without any overflow check. Because `ParseChainID` accepts chain ID numbers up to ~44 decimal digits (far exceeding `uint64` max of ~1.8×10¹⁹), two distinct chain IDs that differ by a multiple of 2⁶⁴ produce an identical EIP-712 domain separator. An attacker holding a valid EIP-712 signature for chain ID `Y` can replay it on any chain whose numeric ID satisfies `chainID mod 2⁶⁴ == Y`.

### Finding Description

`ParseChainID` in `types/chain_id.go` parses the EIP-155 number from the Ethermint chain ID string (`<identifier>_<EIP155>-<epoch>`) into an unbounded `*big.Int`. The total string length is capped at 48 characters, allowing an EIP-155 number up to 44 decimal digits — far beyond `uint64` max (~20 digits). [1](#0-0) 

In both the current and legacy EIP-712 encoding paths, this `*big.Int` is immediately narrowed to `uint64` via `.Uint64()`, which in Go **silently discards all bits above bit 63**:

**Non-legacy path** (`ethereum/eip712/encoding.go`): [2](#0-1) [3](#0-2) 

**Legacy path** (`ethereum/eip712/encoding_legacy.go`): [4](#0-3) [5](#0-4) 

The truncated `uint64` is then passed into `WrapTxToTypedData` / `LegacyWrapTxToTypedData`, which call `SafeInt64` and then `createEIP712Domain`: [6](#0-5) [7](#0-6) 

`SafeInt64` only guards against the truncated value exceeding `int64` max; it does **not** detect that truncation already occurred. If the truncated value is ≤ 2⁶³−1, no error is raised and the domain is silently built with the wrong chain ID.

The ante handler's chain-ID equality check also uses the truncated value: [8](#0-7) 

`signerChainID.Uint64()` truncates the actual chain ID, so an attacker who sets `TypedDataChainID` to the truncated value passes the guard even though the real chain ID is different.

### Impact Explanation

Any chain deployed with a numeric chain ID `X > 2⁶⁴` (e.g., `a_18446744073709560616-1`, which equals `2⁶⁴ + 9000`) will have its EIP-712 domain built with chain ID `9000` (the truncated value). A valid EIP-712 Cosmos transaction signed for chain ID `9000` (e.g., Evmos mainnet) can be submitted verbatim to the chain with ID `2⁶⁴ + 9000` and will pass signature verification. This constitutes a **cross-chain EIP-712 replay attack**, allowing unauthorized execution of Cosmos SDK messages (bank sends, staking, governance votes, etc.) on behalf of the victim.

### Likelihood Explanation

The likelihood is **low** in current deployments because no production chain uses a numeric chain ID exceeding `uint64` max. However, the Ethermint chain ID format explicitly permits such values (the 48-character cap allows up to 44-digit EIP-155 numbers), and the code contains no validation rejecting them. Any future deployment or testnet that inadvertently selects a large chain ID would be silently vulnerable without any error or warning from the node.

### Recommendation

1. **Short term**: Add an explicit upper-bound check in `ParseChainID` (or immediately after) that rejects chain IDs exceeding `math.MaxUint64`. Return a clear error rather than silently truncating.
2. **Long term**: Replace all `chainID.Uint64()` calls in the EIP-712 path with a checked conversion that returns an error on overflow. Consider using `math.NewHexOrDecimal256` directly from the `*big.Int` to avoid any intermediate narrowing.

### Proof of Concept

1. Deploy an Ethermint chain with chain ID string `a_18446744073709560616-1` (numeric value = 2⁶⁴ + 9000).
2. `ParseChainID("a_18446744073709560616-1")` returns `big.Int{18446744073709560616 + 2^64}`.
3. In `decodeAminoSignDoc`, `chainID.Uint64()` returns `9000` (lower 64 bits).
4. `WrapTxToTypedData(9000, ...)` builds a domain with `chainId = 9000`.
5. An attacker takes any valid EIP-712 Cosmos tx signed for chain ID `9000` (Evmos mainnet).
6. They submit it to the chain with ID `2⁶⁴ + 9000`, setting `TypedDataChainID = 9000` in `ExtensionOptionsWeb3Tx`.
7. The ante handler check `extOpt.TypedDataChainID != signerChainID.Uint64()` evaluates `9000 != 9000` → false, so it passes.
8. The EIP-712 domain hash matches the attacker's pre-existing signature; the transaction executes.

func ParseChainID(chainID string) (*big.Int, error) {
	chainID = strings.TrimSpace(chainID)
	if len(chainID) > 48 {
		return nil, errorsmod.Wrapf(ErrInvalidChainID, "chain-id '%s' cannot exceed 48 chars", chainID)
	}

	matches := ethermintChainID.FindStringSubmatch(chainID)
	if matches == nil || len(matches) != 4 || matches[1] == "" {
		return nil, errorsmod.Wrapf(ErrInvalidChainID, "%s: %v", chainID, matches)
	}

	// verify that the chain-id entered is a base 10 integer
	chainIDInt, ok := new(big.Int).SetString(matches[2], 10)
	if !ok {
		return nil, errorsmod.Wrapf(ErrInvalidChainID, "epoch %s must be base-10 integer format", matches[2])
	}

	return chainIDInt, nil
