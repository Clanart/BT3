### Title
Partial `ft_on_transfer` Rejection Permanently Locks Tokens in the Bridge — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When `send_tokens` routes a finalization through `ft_transfer_call` (i.e., when a non-empty `msg` is present), the bridge's refund-detection logic only reverts the transfer when the recipient consumed **zero** tokens. If the recipient's `ft_on_transfer` returns a partial refund (consumed some but not all tokens), the token contract's `ft_resolve_transfer` silently returns the unused portion to the bridge, but the bridge marks the transfer as finalized and has no mechanism to recover those returned tokens. They are permanently stuck.

---

### Finding Description

`send_tokens` in `near/omni-bridge/src/lib.rs` dispatches tokens to the recipient via `ft_transfer_call` when `msg` is non-empty: [1](#0-0) 

The promise result of `ft_transfer_call` is the return value of the token contract's `ft_resolve_transfer`, which is the **amount actually used** by the recipient. The bridge inspects this result in `is_refund_required`: [2](#0-1) 

The condition `amount.0 == 0` triggers a revert only when **no** tokens were consumed. If the recipient's `ft_on_transfer` returns a partial refund (e.g., consumed 60 out of 100 tokens), `ft_resolve_transfer` returns `60` (the amount used), `is_refund_required` evaluates to `false`, and the bridge proceeds to the success branch in `fin_transfer_send_tokens_callback`: [3](#0-2) 

The 40 unused tokens are returned to the bridge by the token contract's `ft_resolve_transfer` (per NEP-141 spec), but the bridge:
1. Does not detect the partial return.
2. Emits `FinTransferEvent` as if the full amount was delivered.
3. Has no sweep or recovery function for the returned balance.

The same flaw exists in `resolve_fast_transfer`, which also delegates to `is_refund_required`: [4](#0-3) 

And in `resolve_utxo_fin_transfer`: [5](#0-4) 

---

### Impact Explanation

For native (non-deployed) tokens, the bridge holds them in its own balance. When `ft_resolve_transfer` returns unused tokens to the bridge, those tokens accumulate in the bridge's balance without any corresponding `locked_tokens` accounting entry. They cannot be claimed by the intended recipient, cannot be re-bridged, and cannot be swept by any admin function. This constitutes **permanent, irrecoverable lock of user funds** in the bridge.

This matches the allowed impact: *"Critical. Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

Any user who initiates a cross-chain transfer with a non-empty `message` field targeting a NEAR recipient contract controls both the recipient address and the message payload. A recipient contract that implements `ft_on_transfer` and returns a non-zero partial refund (e.g., a DeFi vault with a deposit cap, a contract that rejects part of the amount due to slippage, or a deliberately crafted contract) will trigger this path. The entry point is fully unprivileged and reachable by any bridge user.

---

### Recommendation

Replace the binary `amount.0 == 0` check with a check that also handles partial refunds. After `ft_transfer_call` resolves, compare the amount used against the full amount sent. If `used < sent`, treat the difference as a failed partial delivery and either:
- Revert the entire transfer (safest), or
- Re-initiate a bridge transfer for the returned portion back to the original sender.

Additionally, consider adding an admin-callable token recovery function for any tokens that accumulate in the bridge's balance without a corresponding accounting entry.

---

### Proof of Concept

1. User on EVM initiates a transfer of 100 USDC to a NEAR recipient contract `receiver.near` with `msg = "deposit"`.
2. Relayer calls `fin_transfer` on the NEAR bridge.
3. Bridge calls `send_tokens("usdc.near", "receiver.near", 100, "deposit")`.
4. `send_tokens` calls `usdc.near.ft_transfer_call("receiver.near", 100, None, "deposit")`.
5. `receiver.near.ft_on_transfer(bridge, 100, "deposit")` returns `40` (only consumed 60, refunding 40).
6. `usdc.near.ft_resolve_transfer` transfers 40 USDC back to the bridge and returns `60` (amount used).
7. Bridge's `fin_transfer_send_tokens_callback` receives `60` → `is_refund_required` returns `false` (60 ≠ 0).
8. Bridge emits `FinTransferEvent` and sends fee to relayer. Transfer is marked finalized.
9. 40 USDC sit in the bridge's balance with no accounting entry and no recovery path. They are permanently lost. [2](#0-1) [6](#0-5)

    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.
                        amount.0 == 0
                    } else {
                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }
    }
