### Title
Non-EVM-Native Token Balances Permanently Orphaned After Contract Self-Destruct — (File: x/evm/statedb/statedb.go)

### Summary
When a contract self-destructs, `StateDB.Commit()` burns only the EVM-denom balance of the destroyed address. Non-EVM-native tokens (IBC tokens, CosmWasm bridge tokens) held by the same address are explicitly skipped and remain as permanently orphaned bank balances at an address whose auth account has been deleted, making them permanently inaccessible to anyone.

### Finding Description
In `StateDB.Commit()`, the self-destruct path burns the EVM-denom balance and then calls `DeleteAccount` to remove the auth account and contract storage. The code explicitly skips all other bank-module denominations:

```go
// Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.
if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
    coin := sdk.NewCoin(s.evmDenom, sdkmath.NewIntFromBigInt(remaining.ToBig()))
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct balance")
    }
}
if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil {
    return errorsmod.Wrap(err, "failed to delete account")
}
writeCache()
``` [1](#0-0) 

After `DeleteAccount` removes the auth account, the Cosmos bank module still holds any non-EVM-native token balances (IBC vouchers, CosmWasm bridge tokens, etc.) at that address. Because the auth account no longer exists, those balances are permanently inaccessible: no `MsgSend`, `MsgBankSend`, or any other Cosmos message can be signed for a non-existent account, and there is no recovery path in the bank module for orphaned balances.

The `keeper.DeleteAccount` implementation confirms it only removes auth metadata and contract storage, never touching bank balances:

```go
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
    ...
    k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool { ... })
    k.accountKeeper.RemoveAccount(ctx, acct)
    ...
}
``` [2](#0-1) 

The `AddBalance` path used during EVM execution mints coins directly into the bank KV store via `MintCoins` + `SendCoinsFromModuleToAccount`, so any non-EVM-denom tokens sent to a contract address before or during the self-destruct transaction land in the persistent bank KV store and are never cleaned up: [3](#0-2) 

### Impact Explanation
Any non-EVM-native Cosmos bank tokens (IBC vouchers, CosmWasm bridge tokens, staking rewards denominated in other denoms, etc.) held by a contract address at the time of self-destruct are permanently locked in the bank module. The auth account is gone, so no transaction can spend them. The total supply accounting in the bank module is also corrupted: the supply counter still includes these tokens, but they are unreachable. This constitutes valid user funds being permanently mis-accounted through an EVM state transition, matching the High impact scope.

### Likelihood Explanation
The entry path is fully unprivileged:
1. Any user deploys a contract.
2. IBC tokens are transferred to the contract address (e.g., via `ibc-transfer` or a CosmWasm bridge).
3. The contract self-destructs in the same or a later transaction (EIP-6780 or classic `SELFDESTRUCT`).

IBC-enabled Ethermint chains routinely hold IBC vouchers in contract addresses (e.g., liquidity pools, escrow contracts, token wrappers). The scenario is realistic and requires no privileged access.

### Recommendation
In `StateDB.Commit()`, replace the single-denom burn with a full sweep of all bank-module balances held by the self-destructed address before calling `DeleteAccount`. The Cosmos SDK `BankKeeper` exposes `GetAllBalances(ctx, addr)` which returns all denominations; each should be burned via `SendCoinsFromAccountToModule` + `BurnCoins` (or equivalent) inside the same `CacheContext` that wraps `DeleteAccount`, so the operation is atomic.

### Proof of Concept

```
1. Deploy contract C on an IBC-enabled Ethermint chain.
2. IBC-transfer 1000 uatom to C's Ethereum address (mapped 1:1 to its Cosmos AccAddress).
   → bank.GetAllBalances(C) = [1000uatom]
3. In a single EVM transaction, call C.selfdestruct() (EIP-6780 path: created and destroyed in same tx).
4. StateDB.Commit() runs:
   a. Burns EVM-denom balance of C (0 uphoton, no-op).
   b. Calls DeleteAccount(C) → removes auth account and storage.
   c. Does NOT touch the 1000uatom balance.
5. After commit:
   - keeper.GetAccount(C) == nil  (account deleted)
   - bank.GetBalance(C, "uatom") == 1000  (still present, permanently orphaned)
   - No transaction can spend these funds; they are lost forever.
```

The explicit acknowledgment in the production source confirms the root cause is present and unmitigated: [4](#0-3)

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account
//
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
