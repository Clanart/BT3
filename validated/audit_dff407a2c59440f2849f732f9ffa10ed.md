### Title
Missing Per-Transfer u64 Cap Check Before MPC Signing Causes Permanent Token Lock for Solana-Bound Transfers - (File: near/omni-bridge/src/lib.rs)

### Summary
The NEAR bridge's `sign_transfer` function normalizes the transfer amount to a `u128` value and requests an MPC signature without verifying that the normalized amount fits within `u64::MAX`. When the signed payload is submitted to Solana's `finalize_transfer`, the `u128 → u64` conversion fails with `AmountOverflow`, reverting the Solana transaction. Because the tokens were already burned or locked on NEAR during `init_transfer_internal` with no on-chain recovery path, the user's funds are permanently frozen.

### Finding Description

**Step 1 — Token lock/burn on NEAR (no u64 check)**

In `init_transfer_internal`, tokens are burned or locked before any amount validation against Solana's u64 limit: [1](#0-0) 

**Step 2 — `sign_transfer` normalizes to u128 but only checks `> 0`**

`sign_transfer` calls `normalize_amount` (which returns `u128`) and only asserts the result is non-zero. There is no check that `amount_to_transfer <= u64::MAX`: [2](#0-1) 

The normalized amount is then embedded in the MPC-signed payload as `U128`: [3](#0-2) 

**Step 3 — Solana `finalize_transfer` hard-fails on u128 → u64 cast**

On Solana, both the native-token and bridged-token paths perform a fallible cast: [4](#0-3) [5](#0-4) 

If `data.amount > u64::MAX`, the instruction reverts with `AmountOverflow` (error code 6010): [6](#0-5) 

**Step 4 — No recovery path on NEAR**

There is no `revert_transfer` or equivalent function that re-credits the user when a Solana finalization fails. The `locked_tokens` map is only decremented by `unlock_tokens_if_needed`, which is called only on successful `process_fin_transfer_to_other_chain`. Burned tokens (deployed/bridged tokens) are irrecoverable by design. [7](#0-6) 

**When can `normalize_amount` produce a value > u64::MAX?**

`normalize_amount` divides by `10^(origin_decimals − decimals)`: [8](#0-7) 

For a token with `origin_decimals = 24` and `decimals = 9` (diff = 15), a NEAR-side balance of `≈ 1.8 × 10^34` base units (≈ 1.8 × 10^10 tokens) produces a normalized value exceeding `u64::MAX ≈ 1.8 × 10^19`. NEAR token balances are `u128`, so such amounts are representable and can exist in practice for high-supply tokens.

### Impact Explanation

**Critical — Permanent freezing of user funds.** Once `init_transfer_internal` burns or locks the tokens and the MPC signature is issued, the only settlement path is Solana finalization. If that path is permanently blocked by an `AmountOverflow` revert, the tokens are irrecoverably lost. This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation

**Low.** The condition requires a single transfer whose normalized amount exceeds `u64::MAX`. This is possible for tokens with high total supply and a large decimal difference between NEAR and Solana representations, but is not a common scenario for typical bridge usage.

### Recommendation

In `sign_transfer`, after computing `amount_to_transfer`, add a destination-chain-aware cap check. For Solana-bound transfers, assert:

```rust
if transfer_message.get_destination_chain() == ChainKind::Sol {
    require!(
        amount_to_transfer <= u64::MAX as u128,
        BridgeError::AmountOverflow.as_ref()
    );
}
```

This mirrors the per-transaction guard already present on the Solana side and prevents the MPC from signing a payload that can never be finalized.

### Proof of Concept

1. Deploy a NEAR-native token with `origin_decimals = 24`, registered on Solana with `decimals = 9`.
2. Mint `2 × 10^34` base units to a user account on NEAR.
3. User calls `ft_transfer_call` targeting the NEAR bridge with `amount = 2 × 10^34` and `recipient = Sol(...)`.
4. `init_transfer_internal` burns/locks `2 × 10^34` tokens and stores the transfer message.
5. Trusted relayer calls `sign_transfer`; `normalize_amount(2e34, {decimals:9, origin_decimals:24})` returns `2e34 / 1e15 = 2e19 > u64::MAX`. The check `amount_to_transfer > 0` passes; MPC signs.
6. Relayer submits the signed payload to Solana `finalize_transfer`; `(2e19u128).try_into::<u64>()` fails → `AmountOverflow` → transaction reverts.
7. No NEAR-side recovery function exists; the `2 × 10^34` base units are permanently locked/burned.

    fn unlock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(available) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        require!(
            available >= amount,
            TokenLockError::InsufficientLockedTokens.as_ref()
        );

        let remaining = available - amount;
        self.locked_tokens.insert(&key, &remaining);

        LockAction::Unlocked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }
