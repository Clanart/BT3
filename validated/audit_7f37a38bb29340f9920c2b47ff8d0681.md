### Title
3-arg `mint` in `HyperliquedBridgeToken` routes all minted tokens to `_systemAddress` when triggered via `finTransfer` with a non-empty `message` field â€” (`evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

---

### Summary

`HyperliquedBridgeToken.mint(address, uint256, bytes)` unconditionally calls `_mint(account, value)` followed by `_update(account, _systemAddress, value)`, leaving the recipient with zero tokens. `OmniBridge.finTransfer` selects this 3-arg path whenever `payload.message.length > 0`. Because `message` is an attacker-supplied field that flows from `initTransfer` through the NEAR MPC signature into `finTransfer` without any EVM-side validation, any unprivileged caller can force this path for any `HyperliquedBridgeToken` transfer, permanently diverting the minted tokens to `_systemAddress`.

---

### Finding Description

**Step 1 â€” Attacker-controlled `message` enters the bridge.**

`OmniBridge.initTransfer` accepts `message` as an unrestricted `string calldata`: [1](#0-0) 

The field is emitted verbatim in `BridgeTypes.InitTransfer`: [2](#0-1) 

**Step 2 â€” MPC signs the payload including `message`.**

`finTransfer` borsh-encodes `payload.message` into the hash that `nearBridgeDerivedAddress` must sign: [3](#0-2) 

The NEAR relayer observes the `InitTransfer` event and produces a signed `TransferMessagePayload` carrying the same `message` bytes. There is no EVM-side guard that rejects a non-empty `message` for HyperEVM-bound transfers.

**Step 3 â€” `finTransfer` selects the 3-arg `mint` path.**

The sole branching condition is `payload.message.length == 0`: [4](#0-3) 

Any non-empty byte string â€” even a single `0x00` byte â€” routes execution to `IBridgeToken.mint(address, uint256, bytes)`.

**Step 4 â€” `HyperliquedBridgeToken.mint` destroys the recipient's balance.**

```solidity
function mint(address account, uint256 value, bytes memory) external override onlyOwner {
    _mint(account, value);
    _update(account, _systemAddress, value);
}
``` [5](#0-4) 

`_mint` credits `value` to `account`; `_update` immediately debits the same `value` from `account` and credits it to `_systemAddress`. Net effect: `account.balance == 0`, `_systemAddress.balance += value`. The `bytes memory` parameter is not even read â€” any non-empty message triggers this behaviour.

**Contrast with the base `BridgeToken`**, which simply mints and ignores the bytes argument: [6](#0-5) 

`HyperliquedBridgeToken` overrides this with the destructive `_update` call, but `OmniBridge` applies the same `message.length > 0` branch for all `isBridgeToken` tokens without distinguishing `HyperliquedBridgeToken` from plain `BridgeToken`.

---

### Impact Explanation

The intended recipient receives zero tokens. All minted value is parked at `_systemAddress`. `_systemAddress` is the Hyperliquid system address; `coreReceiveWithData` â€” the only function that can move tokens out of that pool â€” is gated by `msg.sender != _systemAddress`: [7](#0-6) 

Neither the recipient nor the attacker can call `coreReceiveWithData` to recover the tokens. The bridged assets are permanently diverted from the intended recipient. This satisfies **Critical â€” direct theft / irrecoverable lock of bridged assets**.

---

### Likelihood Explanation

- `initTransfer` is a fully public function; no role or stake is required.
- Passing `message = abi.encodePacked(uint8(0))` (one byte) is sufficient.
- The NEAR MPC signs whatever payload the relayer constructs from the on-chain event; there is no NEAR-side filter that strips `message` for HyperEVM-bound transfers.
- The attack is repeatable for every inbound transfer to a `HyperliquedBridgeToken` address.

---

### Recommendation

1. **In `HyperliquedBridgeToken.mint(address, uint256, bytes)`**: validate that the `bytes` argument actually encodes a HyperCore-bound intent (e.g., a non-zero action byte or a specific magic prefix) before executing the `_update` to `_systemAddress`. Revert if the message does not match the expected HyperCore format.

2. **In `OmniBridge.finTransfer`**: the `message.length > 0` branch was designed for generic message-passing, not specifically for HyperCore routing. Consider a dedicated token-type check (e.g., `supportsInterface`) or a separate registry flag to distinguish `HyperliquedBridgeToken` from plain `BridgeToken` before selecting the mint path.

3. **Alternatively**: remove the 3-arg `mint` override from `HyperliquedBridgeToken` entirely and route HyperCore-bound mints through a dedicated, access-controlled function that is never reachable from `finTransfer`.

---

### Proof of Concept

```solidity
// 1. Deploy HyperliquedBridgeToken owned by OmniBridge (standard setup).
// 2. Register it: isBridgeToken[token] = true, customMinters[token] = address(0).

// 3. Attacker calls:
omniBridge.initTransfer(
    address(hlToken),
    1_000e18,
    0,
    0,
    "evm:0xVictim",
    "x"          // <-- any non-empty string
);

// 4. NEAR MPC observes InitTransfer, produces signed TransferMessagePayload
//    with message = bytes("x").

// 5. Relayer calls:
omniBridge.finTransfer(signature, TransferMessagePayload({
    ...,
    tokenAddress: address(hlToken),
    amount: 1_000e18,
    recipient: victim,
    message: bytes("x")   // length > 0 â†’ 3-arg mint selected
}));

// 6. Inside HyperliquedBridgeToken.mint:
//    _mint(victim, 1_000e18);          // victim balance = 1_000e18
//    _update(victim, systemAddr, 1_000e18); // victim balance = 0

assert(hlToken.balanceOf(victim) == 0);
assert(hlToken.balanceOf(systemAddress) == 1_000e18);
```
