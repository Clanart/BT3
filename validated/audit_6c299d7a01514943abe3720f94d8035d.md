Audit Report

## Title
Router `pay()` WETH Branch Consumes Any Stranded Native ETH Without Per-Caller Accounting — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` helper in `PeripheryPayments.sol` uses `address(this).balance` as the ETH source for WETH payments without verifying that the current caller deposited that ETH. Any native ETH left on the router from a prior transaction is silently consumed by the next WETH-input swap, giving that caller a free trade at the prior user's expense.

## Finding Description
In `PeripheryPayments.pay()` (lines 73–84), when `token == WETH` and `address(this).balance >= value`, the function wraps the router's full native balance and forwards it to the pool — the `payer` argument is never touched:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // payer ignored
    } ...
}
```

The transient storage context set by `_setNextCallbackContext()` (MetricOmmSwapRouterBase.sol, line 29) records `pool`, `callbackMode`, `payer`, and `tokenToPay`, but stores **no record of how much ETH the current top-level call contributed via `msg.value`**. There is therefore no per-call ETH credit to deduct from; the function simply reads the raw contract balance.

ETH strands on the router whenever a caller sends `msg.value > amountIn` to `exactInputSingle` (payable, line 67) and omits the trailing `refundETH()` call. The `receive()` guard (line 32–34) only blocks plain ETH transfers from non-WETH addresses; it does not prevent ETH from arriving as `msg.value` in payable function calls.

The exploit path is:
1. Alice calls `exactInputSingle{value: 1000}(amountIn=500, tokenIn=WETH, ...)`. The callback fires `_justPayCallback` → `pay(WETH, Alice, pool, 500)`. `nativeBalance=1000 >= 500`, so 500 wei is wrapped and sent; 500 wei remains on the router. Alice omits `refundETH()`.
2. Bob calls `exactInputSingle{value: 0}(amountIn=500, tokenIn=WETH, ...)`. The callback fires `pay(WETH, Bob, pool, 500)`. `nativeBalance=500 >= 500`, so Alice's remaining 500 wei is wrapped and sent to the pool. Bob receives full swap output at zero cost.

No existing guard prevents this: the callback caller check (`_requireExpectedCallbackCaller`) only validates that the caller is a registered pool, not that the ETH balance belongs to the current user.

## Impact Explanation
Direct loss of user principal. Alice's stranded ETH is permanently transferred to the pool as payment for Bob's trade. Bob receives the full swap output without spending any ETH or WETH. This matches the "Critical/High direct loss of user principal" impact gate.

## Likelihood Explanation
Medium. The precondition — ETH stranded on the router — arises naturally from two common usage patterns: sending `msg.value > amountIn` as a slippage buffer, or constructing a `multicall` that omits the trailing `refundETH` step. Once ETH is stranded, any observer can exploit it in the very next block with a zero-cost WETH swap requiring no special privileges.

## Recommendation
**Short term:** In the `nativeBalance >= value` branch, verify that the ETH being consumed does not exceed the ETH credited to the current call. Store the per-call ETH credit in transient storage (alongside the existing payer/token context) at entry to each payable function, and deduct from that credit rather than from the raw `address(this).balance`.

**Long term:** Enforce that `address(this).balance == 0` at the start of every non-payable entry point, or revert in `pay()` when `msg.value` for the current top-level call is zero but the native-ETH branch would be taken.

## Proof of Concept
```
Block N:
  Alice calls exactInputSingle{value: 1000}(
      tokenIn=WETH, amountIn=500, recipient=Alice, ...
  )
  → pool.swap() fires metricOmmSwapCallback
  → _justPayCallback → pay(WETH, Alice, pool, 500)
      nativeBalance=1000 >= 500 → wraps 500 wei, sends WETH to pool
      500 wei remains on router
  Alice does NOT call refundETH().

Block N+1:
  Bob calls exactInputSingle{value: 0}(
      tokenIn=WETH, amountIn=500, recipient=Bob, ...
  )
  → pool.swap() fires metricOmmSwapCallback
  → _justPayCallback → pay(WETH, Bob, pool, 500)
      nativeBalance=500 >= 500 → wraps Alice's 500 wei, sends WETH to pool
      Bob receives TOKEN output, pays nothing
```

Alice loses 500 wei. Bob receives a full swap output at zero cost. A Foundry fork test can reproduce this by deploying the router, funding Alice's call with excess ETH, skipping `refundETH`, then calling from Bob's address with zero value and asserting Bob's output token balance increases while the router ETH balance drops to zero.
