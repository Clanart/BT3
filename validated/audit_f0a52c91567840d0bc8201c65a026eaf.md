### Title
`swap_migrated_token` Uses Non-Atomic `burn.and(mint)` Without a Success Callback, Enabling Permanent Loss of User Funds or Unbacked Token Minting — (`File: near/omni-bridge/src/lib.rs`)

---

### Summary

The `swap_migrated_token` function schedules a burn of old tokens and a mint of new tokens using `Promise::and()`, which runs both sub-promises in parallel with no atomicity guarantee. The promise is `.detach()`ed in `ft_on_transfer` with no callback, so a failure in either sub-promise is silently ignored. If the mint panics (e.g., because the recipient has not registered storage on the new token contract), the burn still executes, permanently destroying the user's old tokens. Conversely, if the burn panics, the mint still executes, creating unbacked new tokens.

---

### Finding Description

In `ft_on_transfer`, when the message is `SwapMigratedToken`, the bridge immediately returns `U128(0)` (keeping all transferred old tokens) and detaches the swap promise:

```rust
BridgeOnTransferMsg::SwapMigratedToken => {
    self.swap_migrated_token(sender_id, token_id, amount)
        .detach();
    PromiseOrPromiseIndexOrValue::Value(U128(0))
}
``` [1](#0-0) 

Inside `swap_migrated_token`, the burn and mint are composed with `Promise::and()`:

```rust
let burn = ext_token::ext(old_token).burn(amount);
let mint = ext_token::ext(new_token).mint(sender_id, amount, None);
burn.and(mint)
``` [2](#0-1) 

In NEAR, `Promise::and()` schedules both sub-promises to execute independently. Cross-contract calls are **not atomic** — if one panics, the other still executes. There is no rollback. Because the combined promise is `.detach()`ed, there is no callback to detect or recover from a partial failure.

The `mint` function on the new token calls `internal_deposit`, which panics if the recipient account has not registered storage with that token contract:

```rust
self.token.internal_deposit(&account_id, amount.into());
``` [3](#0-2) 

`migrate_deployed_token` only registers storage for the bridge contract itself on the new token — it does not pre-register storage for any user: [4](#0-3) 

---

### Impact Explanation

**Scenario A — Mint fails, burn succeeds (permanent loss of user funds):**
A user who has not registered storage on the new token contract sends old tokens to the bridge with `SwapMigratedToken`. The bridge keeps the old tokens (`U128(0)` return). The burn executes and destroys the old tokens from the bridge's balance. The mint panics due to missing storage registration. The user permanently loses their old tokens with no refund path. This matches: **Critical — permanent irrecoverable lock of user funds in bridge flow.**

**Scenario B — Burn fails, mint succeeds (unbacked token minting):**
If the old token's `burn` call panics for any reason (e.g., the old token contract has been upgraded, paused, or has insufficient balance due to a race condition), the mint still executes, issuing new tokens to the user without destroying the old ones. This creates unbacked circulating supply. This matches: **Critical — unauthorized mint of bridged assets.**

---

### Likelihood Explanation

Scenario A is highly realistic and requires no privileged access. Any user who holds old (migrated) tokens and calls `ft_transfer_call` with `SwapMigratedToken` without first registering storage on the new token will trigger it. Storage registration is a separate, non-obvious prerequisite that is not enforced or documented at the call site. The bridge provides no refund mechanism once `U128(0)` is returned from `ft_on_transfer`.

---

### Recommendation

Replace the parallel `burn.and(mint)` pattern with a sequential promise chain that uses a callback to verify the burn succeeded before minting, and refunds the user if either step fails:

```rust
ext_token::ext(old_token)
    .burn(amount)
    .then(
        Self::ext(env::current_account_id())
            .with_static_gas(SWAP_MIGRATED_TOKEN_CALLBACK_GAS)
            .swap_migrated_token_callback(sender_id, new_token, amount),
    )
```

In `swap_migrated_token_callback`, check `env::promise_result(0)` — if the burn failed, transfer the old tokens back to the user (or re-credit them). Only mint if the burn succeeded. This ensures the swap is either fully completed or fully rolled back.

---

### Proof of Concept

1. DAO calls `migrate_deployed_token(Eth, old_token, new_token)`. The bridge registers storage for itself on `new_token` but not for any user.
2. Alice holds 1000 `old_token`. She has never called `storage_deposit` on `new_token`.
3. Alice calls `old_token.ft_transfer_call(bridge, 1000, "SwapMigratedToken")`.
4. `ft_on_transfer` is triggered. The bridge returns `U128(0)` — Alice's 1000 `old_token` are now held by the bridge.
5. `swap_migrated_token` is called. `burn.and(mint)` is detached.
6. NEAR runtime schedules both sub-promises in parallel:
   - `old_token.burn(1000)` — succeeds; bridge's balance of `old_token` is reduced to 0.
   - `new_token.mint(alice, 1000, None)` — panics inside `internal_deposit` because Alice has no storage registration on `new_token`.
7. The burn receipt is committed. The mint receipt panics and is discarded. No callback exists to detect this.
8. Alice has lost 1000 `old_token` permanently. She receives 0 `new_token`. There is no recovery path.
