### Title
EndBlock chain halt via cross-tx block gas accumulation exceeding MaxInt64 when max_gas=-1 — (`x/feemarket/keeper/abci.go`)

### Summary
When `max_gas=-1` (blockGasLimit=`math.MaxUint64`), the ante handler's per-tx overflow guard allows each individual tx with `gasLimit=math.MaxInt64` to pass. Two such txs in the same block accumulate `ctx.BlockGasUsed() = 2*math.MaxInt64`, which exceeds `math.MaxInt64`. `EndBlock` calls `SafeInt64(ctx.BlockGasUsed())` unconditionally and returns the error, halting the chain.

### Finding Description

`BlockGasLimit` returns `math.MaxUint64` when `max_gas=-1`: [1](#0-0) 

The ante handler's overflow guard in `CheckEthGasConsume` uses a **local** `gasWanted` variable that resets to `0` for every tx call. For a single-message tx with `gasLimit=math.MaxInt64`: [2](#0-1) 

- Check: `0 > math.MaxInt64 - math.MaxInt64 = 0` → `false` → passes
- Check: `math.MaxInt64 > math.MaxUint64` → `false` → passes

A second independent tx with the same `gasLimit` goes through the same checks with `gasWanted=0` again and also passes. There is no cross-tx block-level guard that caps cumulative `gasWanted` at `math.MaxInt64`.

In `EndBlock`, the feemarket keeper calls `SafeInt64` on the total block gas used: [3](#0-2) 

`SafeInt64` returns an error for any value exceeding `math.MaxInt64`: [4](#0-3) 

With two txs each consuming `math.MaxInt64` gas, `ctx.BlockGasUsed() = 2*(2^63-1) = 2^64-2`, which is a valid `uint64` (no uint64 overflow), but `SafeInt64` rejects it. `EndBlock` propagates the error, causing a consensus-level chain halt.

### Impact Explanation
An unprivileged attacker can permanently halt the chain at `EndBlock` by submitting two transactions each with `gasLimit=math.MaxInt64` that consume all their gas in the same block. This is a **Critical** impact: deterministic validator consensus failure triggered through the public tx submission path.

### Likelihood Explanation
Requires `max_gas=-1` (a common/default configuration), EIP-1559 enabled (default in Ethermint), and sufficient balance to pay fees for `gasLimit=math.MaxInt64` at the prevailing base fee. The balance requirement is large but not prohibitive at low base fees. The attack is deterministic and reproducible.

### Recommendation
In `EndBlock` (`x/feemarket/keeper/abci.go`), clamp `ctx.BlockGasUsed()` to `math.MaxInt64` before calling `SafeInt64`, or use a saturating cast instead of returning an error. Additionally, the ante handler's overflow guard should be extended to track cumulative block-level gas across txs (not just within a single tx), capping the sum at `math.MaxInt64` when `blockGasLimit=math.MaxUint64`.

### Proof of Concept
1. Configure chain with `max_gas=-1` and EIP-1559 enabled.
2. Deploy a contract that loops until all gas is consumed.
3. Submit two transactions in the same block, each with `gasLimit=math.MaxInt64`, calling the gas-burning contract.
4. Both txs pass ante checks individually (per-tx `gasWanted` resets to 0).
5. Both txs execute, each consuming `math.MaxInt64` gas.
6. `ctx.BlockGasUsed()` = `2*math.MaxInt64 = 2^64-2`.
7. `EndBlock` → `SafeInt64(2^64-2)` → error → chain halt.
