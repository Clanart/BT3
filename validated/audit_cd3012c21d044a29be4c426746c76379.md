### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Bricking - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary felt value as a new class hash without verifying that the class has been declared. This is directly analogous to the reported `bumpTransfer()` bug: just as `bumpTransfer()` accepted any `transferId` without checking existence, `execute_replace_class` accepts any `class_hash` without checking it is a declared class. A contract can call `replace_class` with a non-existent class hash, permanently bricking itself and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `REPLACE_CLASS` syscall. After reducing gas, it reads the new class hash from the request and immediately writes it into `contract_state_changes` with no validation that the class hash corresponds to a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The developer-inserted TODO comment explicitly acknowledges the missing check. The OS proof is generated over this unchecked state transition, meaning the L1 verifier will accept a proof where a contract's class hash is set to an undeclared (non-existent) class.

Contrast this with `execute_declare_transaction`, which enforces `prev_value=0` to prevent double-declaration and validates the class hash pre-image via `finalize_class_hash`. No equivalent guard exists in `execute_replace_class`.

---

### Impact Explanation

Once a contract's class hash is set to an undeclared value and the block is proven and settled on L1, the state is final. Any subsequent call to that contract will fail at class resolution because the class does not exist in `contract_class_changes`. There is no recovery path: the OS has no mechanism to "un-replace" a class hash after the proof is accepted. All funds (ERC-20 balances, ETH, or any token held in the contract's storage) are permanently inaccessible.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The entry path is fully reachable by an unprivileged user:

1. An attacker deploys a contract (e.g., a fake vault or yield aggregator) that exposes a publicly callable upgrade function invoking `replace_class`.
2. Users deposit funds into the contract.
3. The attacker calls the upgrade function with an arbitrary, undeclared class hash (e.g., `felt(1)` or any random value).
4. The OS processes the syscall, writes the invalid class hash into `contract_state_changes`, and generates a valid proof.
5. After L1 settlement, the contract is permanently bricked and all deposited funds are frozen.

No privileged role, leaked key, or external dependency is required. The `replace_class` syscall is dispatched for any contract that issues it, with no OS-level access control beyond restricting the target to the calling contract's own address.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that `class_hash` exists in `contract_class_changes` (i.e., has a non-zero compiled class hash entry). This mirrors the guard already present in `execute_declare_transaction` (`prev_value=0` enforces uniqueness) and the pattern used in `execute_deploy` where the class hash is immediately exercised (implicitly validating existence). The fix should be applied at the OS level so it is enforced in the proof, not merely in the sequencer's off-chain validation.

---

### Proof of Concept

1. Declare a valid class `C` and deploy contract `V` (vault) using class `C`. Users deposit funds into `V`.
2. `V` exposes: `fn upgrade(new_class: ClassHash) { replace_class(new_class); }` with no access control.
3. Attacker calls `V.upgrade(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
4. The OS executes `execute_replace_class` at the line below, writing `0xdeadbeef` into `contract_state_changes` for `V`'s address with no existence check: [1](#0-0) 

5. The revert log records the old class hash (for intra-block revert only); the transaction is not reverted.
6. The OS squashes state and generates a STARK proof over the new state root where `V.class_hash = 0xdeadbeef`.
7. The L1 verifier accepts the proof. `V` is permanently bricked. All funds frozen.
