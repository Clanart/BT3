### Title
Silent `ft_transfer`/`mint` Failure Not Detected in `fin_transfer_send_tokens_callback`, Permanently Finalizing Transfer Without Token Delivery - (File: near/omni-bridge/src/lib.rs)

---

### Summary

`fin_transfer_send_tokens_callback` never checks whether the underlying `ft_transfer` or `mint` promise succeeded when no message is present (`is_ft_transfer_call = false`). If the cross-contract call fails, the transfer is permanently recorded as finalized, locked-token accounting is permanently updated, and the user's tokens are irrecoverably lost or frozen.

---

### Finding Description

The finalization flow for NEAR-destination transfers is:

```
fin_transfer → fin_transfer_callback → process_fin_transfer_to_near → send_tokens → fin_transfer_send_tokens_callback
```

Inside `process_fin_transfer_to_near`, **before** `send_tokens` is called, the bridge already:
1. Inserts the transfer into `finalised_transfers` (line 1875)
2. Unlocks the locked tokens via `unlock_tokens_if_needed` (lines 1881–1885) [1](#0-0) 

`send_tokens` then dispatches either `ft_transfer` (non-deployed token, no message) or `mint` (deployed token, no message), and chains `fin_transfer_send_tokens_callback` with `is_ft_transfer_call = !msg.is_empty()`. [2](#0-1) 

Inside the callback, `is_refund_required` is the sole gate for reverting state:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
            Ok(value) => { ... amount.0 == 0 }
            Err(_) => false,   // ← also does NOT revert on failure
        }
    } else {
        false   // ← NEVER checks the promise result
    }
}
``` [3](#0-2) 

When `is_ft_transfer_call` is `false` (no message → `ft_transfer` or 2-arg `mint`), the function unconditionally returns `false` without ever calling `env::promise_result_checked`. A failed promise is indistinguishable from a successful one. The callback then falls into the success branch: [4](#0-3) 

- Fees are dispatched to the fee recipient (via `.detach()`)
- `FinTransferEvent` is emitted
- The transfer remains in `finalised_transfers` permanently
- The lock-action revert (`revert_lock_actions`) is **never called**

The only path that reverts state is the `is_refund_required == true` branch (lines 1702–1718), which is unreachable when `is_ft_transfer_call = false`. [5](#0-4) 

---

### Impact Explanation

**For non-deployed (native NEAR) tokens:**
- `ft_transfer` fails → tokens remain locked inside the bridge contract
- `unlock_tokens_if_needed` already decremented `locked_tokens` → bridge accounting now records fewer locked tokens than it actually holds
- Transfer is in `finalised_transfers` → user cannot retry; tokens are permanently frozen in the bridge

**For deployed (bridged) tokens:**
- `mint` fails → no tokens are created for the user
- Transfer is in `finalised_transfers` → user cannot retry; bridged value is permanently lost

Both outcomes match the allowed critical impact: **permanent freezing / irrecoverable lock of user funds in bridge flows**.

---

### Likelihood Explanation

Realistic failure conditions for `ft_transfer` or `mint` without a message:

1. **Token contract paused**: Many NEP-141 tokens implement a pause mechanism. If the token is paused between the storage-balance check (in `fin_transfer_callback`) and the actual `ft_transfer` (a subsequent cross-contract call), the transfer fails silently.
2. **Recipient removes storage deposit**: In NEAR, `storage_unregister` can be called by the recipient between the `storage_balance_of` check and the `ft_transfer`. A zero-balance account can unregister, causing `ft_transfer` to panic.
3. **Token contract upgrade or bug**: A deployed bridge token upgraded to a version that rejects mints would silently finalize transfers without delivering tokens.

The entry point (`fin_transfer`) is gated by `#[trusted_relayer]`, but the relayer is an unprivileged protocol participant who submits valid cross-chain proofs. Any valid proof submission that triggers a failing `ft_transfer`/`mint` produces the corrupted state.

---

### Recommendation

In `fin_transfer_send_tokens_callback`, always check the promise result regardless of `is_ft_transfer_call`:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
            Ok(value) => {
                serde_json::from_slice::<U128>(&value)
                    .map(|a| a.0 == 0)
                    .unwrap_or(false)
            }
            Err(_) => true,  // promise failed → require refund
        }
    } else {
        // Check whether ft_transfer / mint succeeded
        env::promise_result_checked(0, 0).is_err()
    }
}
```

When the promise fails and `is_ft_transfer_call` is `false`, the callback must:
- Call `revert_lock_actions` to restore `locked_tokens`
- Call `remove_fin_transfer` to remove the transfer from `finalised_transfers`
- Emit `FailedFinTransferEvent`

---

### Proof of Concept

1. Alice bridges 1000 USDC from Ethereum to NEAR. The bridge locks 1000 USDC in `locked_tokens[(Eth, usdc.near)] = 1000`.
2. A relayer submits a valid proof via `fin_transfer`.
3. `process_fin_transfer_to_near` runs:
   - `finalised_transfers.insert(transfer_id)` ✓
   - `unlock_tokens_if_needed(Eth, usdc.near, 1000)` → `locked_tokens[(Eth, usdc.near)] = 0` ✓
   - `send_tokens(usdc.near, alice.near, 1000, "")` dispatched
4. Between the storage-balance check and the `ft_transfer`, Alice calls `storage_unregister` on `usdc.near` (she has 0 balance at this point). Her storage slot is removed.
5. `ft_transfer(alice.near, 1000)` panics: "The account alice.near is not registered".
6. `fin_transfer_send_tokens_callback` is called with `is_ft_transfer_call = false`.
7. `is_refund_required(false)` returns `false` without checking the promise result.
8. Callback emits `FinTransferEvent` and sends fees to the relayer.
9. Result: `finalised_transfers` contains the transfer ID (cannot retry), `locked_tokens[(Eth, usdc.near)] = 0` (accounting corrupted), and Alice's 1000 USDC are permanently frozen in the bridge contract. [6](#0-5) [3](#0-2) [2](#0-1)
