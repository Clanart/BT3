### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Bricking and Fund Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by a contract corresponds to a previously declared class. An unprivileged contract can call `replace_class` with any arbitrary felt value as the class hash. The OS accepts and commits this state transition unconditionally. Once committed, the contract's class hash in the state trie points to a non-existent class, making the contract permanently unexecutable and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall. After deducting gas, it reads the requested new class hash directly from the syscall request and proceeds to update `contract_state_changes` — but it performs **no check** that the supplied class hash has ever been declared on-chain.

The code contains an explicit acknowledgment of this gap:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}
``` [1](#0-0) 

The function then updates `contract_state_changes` with the new (unvalidated) class hash. The OS's post-execution validation step (`validate_compiled_class_facts_post_execution`, called in `os.cairo`) only validates compiled class facts that were **actually used during execution** in the current block — it does not retroactively validate class hashes written into the state trie via `replace_class`. [2](#0-1) 

The `execute_replace_class` function is dispatched from the syscall handler loop in `execute_syscalls.cairo` and is reachable by any contract executing a `replace_class` syscall. [3](#0-2) 

---

### Impact Explanation

Once a contract's class hash in the committed state trie is set to an undeclared value:

1. Any future transaction targeting that contract requires the OS to locate the compiled class for that hash in the `compiled_class_facts` bundle.
2. Because the hash was never declared, no such compiled class exists.
3. The contract becomes permanently unexecutable — no entry point (including another `replace_class` call) can ever run again.
4. All funds (tokens, ETH, etc.) held in the contract's storage are permanently frozen with no recovery path.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

- The `replace_class` syscall is callable by any contract on itself; no privileged role is required.
- An attacker needs only to deploy a contract and invoke `replace_class` with an arbitrary undeclared felt value.
- The attack is deterministic and requires no race condition, oracle manipulation, or external dependency.
- The missing check is self-documented in the codebase as a known gap (`TODO`), confirming it is not an implementation artifact but an intentional deferral.

---

### Recommendation

Inside `execute_replace_class`, before updating `contract_state_changes`, add a validation step that asserts the supplied `class_hash` exists in `contract_class_changes` (i.e., it was declared in the current or a prior block). Concretely:

1. Perform a `dict_read` on `contract_class_changes` for `class_hash`.
2. Assert the returned compiled class hash is non-zero (indicating a prior `declare` transaction registered it).
3. Only then proceed with the `dict_update` on `contract_state_changes`.

This mirrors the validation already enforced at the transaction level for `declare` transactions, where `prev_value=0` is asserted to prevent re-declaration. [4](#0-3) 

---

### Proof of Concept

1. **Deploy** a contract `VictimVault` that accepts token deposits and holds user funds.
2. **Deploy** an attacker contract `Bricker` that, in its `__execute__` entry point, calls `replace_class(undeclared_hash)` where `undeclared_hash` is any felt not corresponding to a declared class (e.g., `0xdeadbeef`).
3. **Invoke** `Bricker.__execute__`. The OS processes the `replace_class` syscall via `execute_replace_class`. No declared-class check is performed. The OS writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
4. **State is committed.** The block output includes the updated state root with `Bricker`'s class hash set to `0xdeadbeef`.
5. **Any subsequent transaction** targeting `Bricker` causes the OS to look up compiled class `0xdeadbeef`, which does not exist. The contract is permanently unexecutable.
6. If `Bricker` held user funds (e.g., users deposited tokens before step 3), those funds are permanently frozen with no recovery path.

The same attack applies to any contract a malicious deployer controls, including contracts that have accumulated user deposits.
