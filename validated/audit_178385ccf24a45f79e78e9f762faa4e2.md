### Title
Excess ETH Forwarded to Wormhole `publishMessage` Is Permanently Irrecoverable — (File: `evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

---

### Summary

In `OmniBridgeWormhole.sol`, three extension hooks — `deployTokenExtension`, `logMetadataExtension`, and `finTransferExtension` — blindly forward the full `msg.value` to `_wormhole.publishMessage{value: msg.value}(...)`. The Wormhole core contract only consumes exactly `messageFee()` and does not refund any excess. Any ETH sent above `messageFee()` is permanently trapped in the Wormhole core contract, with no recovery path in `OmniBridgeWormhole`.

---

### Finding Description

`OmniBridgeWormhole` overrides three internal extension hooks from `OmniBridge`:

**`deployTokenExtension`** (line 63):
```solidity
_wormhole.publishMessage{value: msg.value}(wormholeNonce, payload, _consistencyLevel);
```
Called by the `payable` public function `deployToken`.

**`logMetadataExtension`** (line 87):
```solidity
_wormhole.publishMessage{value: msg.value}(wormholeNonce, payload, _consistencyLevel);
```
Called by the `payable` public functions `logMetadata` and `logMetadata1155` — **both have no access control whatsoever**.

**`finTransferExtension`** (line 109):
```solidity
_wormhole.publishMessage{value: msg.value}(wormholeNonce, messagePayload, _consistencyLevel);
```
Called by the `payable` public function `finTransfer`.

In all three cases, the entire `msg.value` is forwarded to Wormhole. The Wormhole `IWormhole` interface exposes `messageFee()` to query the exact required fee. The Wormhole core contract accepts `msg.value >= messageFee()` but retains the surplus — it does not refund excess ETH to the caller. There is no sweep or recovery function in `OmniBridgeWormhole` or `OmniBridge` to retrieve ETH that has been forwarded to Wormhole.

The `initTransferExtension` path is slightly different — it receives `value = msg.value - nativeFee` (or `msg.value - amount - nativeFee` for native ETH transfers) and passes that to `_wormhole.publishMessage{value: value}`. If `value > messageFee()`, the same excess-loss applies there too.

---

### Impact Explanation

Any ETH sent above `_wormhole.messageFee()` in a call to `logMetadata`, `logMetadata1155`, `finTransfer`, or `deployToken` is permanently irrecoverable. The funds are forwarded into the Wormhole core contract and cannot be retrieved by the user, the bridge, or any admin of `OmniBridgeWormhole`. This constitutes a permanent, irrecoverable loss of user funds in a bridge flow, matching the **Critical** allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Likelihood: Low.**

The most reachable entry point is `logMetadata` / `logMetadata1155` — they require no signature, no role, and no special state. Any user can call them with arbitrary `msg.value`. A user who does not know the exact current Wormhole `messageFee()` (which can change) and sends a round-number ETH amount (e.g., `0.01 ETH`) instead of the precise fee will permanently lose the difference. For `finTransfer` and `deployToken`, a valid MPC signature is required, so the caller is typically a relayer, but relayers may also overpay to avoid reverts. The Wormhole fee is not validated anywhere in the call chain before forwarding.

---

### Recommendation

Before forwarding ETH to `_wormhole.publishMessage`, query the exact fee and refund any surplus to `msg.sender`:

```solidity
function _publishToWormhole(bytes memory payload) internal {
    uint256 fee = _wormhole.messageFee();
    require(msg.value >= fee, "Insufficient Wormhole fee");
    _wormhole.publishMessage{value: fee}(wormholeNonce, payload, _consistencyLevel);
    wormholeNonce++;
    uint256 refund = msg.value - fee;
    if (refund > 0) {
        (bool ok, ) = msg.sender.call{value: refund}("");
        require(ok, "ETH refund failed");
    }
}
```

Apply this pattern in `deployTokenExtension`, `logMetadataExtension`, and `finTransferExtension`. For `initTransferExtension`, apply the same check against the `value` parameter (which already has `nativeFee` subtracted).

---

### Proof of Concept

1. Deploy `OmniBridgeWormhole` pointing to a real Wormhole core contract where `messageFee() = 0.001 ether`.
2. Call `logMetadata(someERC20)` with `msg.value = 0.01 ether` (10× the actual fee).
3. Observe: `_wormhole.publishMessage{value: 0.01 ether}(...)` is called.
4. Wormhole consumes `0.001 ether` and retains the remaining `0.009 ether`.
5. The caller's `0.009 ether` is permanently lost — no refund, no sweep function in `OmniBridgeWormhole`.
