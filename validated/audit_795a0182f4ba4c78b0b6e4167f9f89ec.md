### Title
Missing Storage Gap in `BridgeToken` Corrupts `HyperliquedBridgeToken._systemAddress` on Upgrade - (File: `evm/src/omni-bridge/contracts/BridgeToken.sol`)

---

### Summary

`BridgeToken` defines three sequential storage variables (`_name`, `_symbol`, `_decimals`) with no `__gap` reserve. `HyperliquedBridgeToken` inherits `BridgeToken` and appends `_systemAddress` immediately after. Any future upgrade that adds a new storage variable to `BridgeToken` will shift `_systemAddress` in the deployed proxy, silently corrupting the authentication guard and the HyperCore token-accounting path.

---

### Finding Description

`BridgeToken` occupies three sequential storage slots with no trailing gap: [1](#0-0) 

All four parent contracts (`Initializable`, `UUPSUpgradeable`, `ERC20Upgradeable`, `Ownable2StepUpgradeable`) use OpenZeppelin v5 ERC-7201 namespaced storage, so they contribute **zero** sequential slots. The only sequential slots in the proxy's storage layout come from `BridgeToken` itself:

| Slot | Variable |
|------|----------|
| 0 | `_name` |
| 1 | `_symbol` |
| 2 | `_decimals` |
| 3 | `_systemAddress` ← from `HyperliquedBridgeToken` |

`HyperliquedBridgeToken` appends `_systemAddress` at slot 3: [2](#0-1) 

`BridgeToken` declares **no** `__gap`, unlike `OmniBridge` which correctly reserves 49 slots: [3](#0-2) 

The upgrade path is live. `OmniBridge.upgradeToken` allows `DEFAULT_ADMIN_ROLE` to push a new implementation to any registered bridge token proxy: [4](#0-3) 

If a future `BridgeToken` revision adds even one new variable after `_decimals` (e.g., a fee rate, a blacklist flag, a metadata URI), that variable occupies slot 3 in the new implementation layout, overwriting the value that the proxy's storage holds for `_systemAddress`.

---

### Impact Explanation

`_systemAddress` is the sole authentication gate for `coreReceiveWithData`: [5](#0-4) 

It is also the accounting sink for every HyperCore-bound mint: [6](#0-5) 

Two concrete outcomes after corruption:

1. **Permanent freeze (Critical):** If the overwritten slot holds any value other than the real HyperLiquid system address, every `coreReceiveWithData` call reverts with `NotSystemAddress`. All HyperCore→HyperEVM redemptions and cross-chain `initTransfer` calls from HyperCore are permanently bricked. Tokens already parked at the old `_systemAddress` pool become irrecoverable.

2. **Unauthorized mint / theft (Critical):** If the overwritten slot value is a predictable or attacker-controlled address (e.g., `address(0)` from a freshly zeroed slot, or a value derived from the new variable's initializer), an attacker who controls that address can call `coreReceiveWithData` directly, draining the standing pool of bridged tokens held at `_systemAddress` to an arbitrary recipient.

---

### Likelihood Explanation

`HyperliquedBridgeToken` is a production contract deployed as a UUPS proxy. The protocol already performs token upgrades via `upgradeToken`. Adding a new field to `BridgeToken` (e.g., a per-token fee, a metadata URI, a blacklist mapping) is a routine development action. No malicious intent is required; the corruption is a silent side-effect of a legitimate upgrade. The missing gap is the only barrier between a normal protocol upgrade and a critical storage collision.

---

### Recommendation

Add a storage gap at the end of `BridgeToken`'s storage variable declarations:

```solidity
// evm/src/omni-bridge/contracts/BridgeToken.sol
string internal _name;
string internal _symbol;
uint8  internal _decimals;

uint256[47] private __gap; // reserve slots 3–49 for future BridgeToken variables
```

Similarly add a gap to `HyperliquedBridgeToken` after `_systemAddress` so that contract can also be safely subclassed or extended:

```solidity
address internal _systemAddress;
uint256[49] private __gap;
```

---

### Proof of Concept

1. Deploy `HyperliquedBridgeToken` behind an `ERC1967Proxy`. Storage slot 3 holds the real HyperLiquid system address.
2. Prepare a new `BridgeToken` implementation that adds `uint256 public feeRate;` after `_decimals`.
3. Call `OmniBridge.upgradeToken(proxyAddress, newImpl)` as `DEFAULT_ADMIN_ROLE`.
4. The new implementation maps slot 3 to `feeRate` (initialized to `0`).
5. Read `_systemAddress` from the proxy: it now returns `address(0)`.
6. Every subsequent `coreReceiveWithData` call reverts (`msg.sender != address(0)` is always true), permanently freezing all HyperCore redemptions. Any tokens previously accumulated at the old system address pool are unclaimable.
