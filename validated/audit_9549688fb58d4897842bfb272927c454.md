### Title
Missing Zero-Address Recipient Validation in `fin_transfer` Causes Permanent Fund Lock — (`starknet/src/omni_bridge.cairo`)

### Summary
`OmniBridge::fin_transfer` does not validate that `payload.recipient` is non-zero before calling `BridgeToken::mint`. If NEAR signs a `fin_transfer` payload encoding `recipient = ContractAddress(0)` (e.g., user error or NEAR-side address derivation bug), the OZ ERC20 `mint` panics, the entire transaction reverts (including the nonce mark), and the signed payload becomes permanently unexecutable. Source-chain funds are already burned/locked on NEAR with no on-chain recovery path.

### Finding Description
In `fin_transfer`: [1](#0-0) 

The nonce is marked used at line 250, then the signature is verified, then `mint` is called at line 257 with `payload.recipient` passed directly — no `assert(!payload.recipient.is_zero(), ...)` guard exists anywhere in this path.

`BridgeToken::mint` delegates unconditionally to the OZ ERC20 component: [2](#0-1) 

OZ Cairo ERC20Component v2.0.0 (declared in `Scarb.lock`) panics with `MINT_TO_ZERO` when recipient is the zero address. Because StarkNet transactions are atomic, the panic rolls back `_set_transfer_finalised`, leaving the nonce unconsumed. Every retry with the same NEAR-signed payload (which encodes `recipient=0` in the Borsh-serialized message that was signed) panics identically. [3](#0-2) 

The recipient address is embedded in the signed Borsh payload at serialization time, so no relayer can substitute a valid recipient without invalidating the MPC signature.

### Impact Explanation
Source-chain funds (burned or locked on NEAR) are permanently irrecoverable. The StarkNet contract has no admin rescue function, no refund entrypoint, and no mechanism to re-finalize a transfer with a corrected recipient. The signed payload is cryptographically bound to `recipient=0` and cannot be reused with a different address.

### Likelihood Explanation
The precondition — NEAR MPC signing a payload with `recipient=0` — is realistic: StarkNet addresses are 252-bit field elements and users can supply arbitrary values including zero. If the NEAR bridge does not reject `recipient=0` before signing, a single user mistake or a NEAR-side address-parsing edge case triggers the lock. No privileged access is required; any user initiating a transfer to StarkNet with recipient=0 is affected.

### Recommendation
Add an explicit zero-address guard in `fin_transfer` before the mint/transfer dispatch:

```cairo
assert(!payload.recipient.is_zero(), 'ERR_ZERO_RECIPIENT');
```

This should be placed after signature verification and before the `is_bridge_token` branch, so the check applies to both the mint path and the native-token transfer path.

### Proof of Concept
1. Deploy `OmniBridge` and a `BridgeToken` via `deploy_token`.
2. Construct a `TransferMessagePayload` with `recipient = ContractAddress(0)` and a valid `destination_nonce`.
3. Sign the Borsh-encoded payload with the bridge's MPC key.
4. Call `fin_transfer(sig, payload)` — transaction panics with `MINT_TO_ZERO`.
5. Assert `is_transfer_finalised(nonce)` returns `false` (nonce not consumed).
6. Retry with the same payload — identical panic.
7. Confirm no function exists to recover or refund the locked source-chain funds.
