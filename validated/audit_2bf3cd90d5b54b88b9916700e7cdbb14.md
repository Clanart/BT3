Looking at the relevant production code:

**`submit_transfer_to_utxo_chain_connector`** (btc.rs line 84): calls `self.remove_transfer_message(transfer_id)` before the cross-contract `ft_transfer_call`, releasing the storage cost back to the user's available balance.

**`submit_transfer_to_btc_connector_callback`** (btc.rs lines 114–125): on failure (`result.0 == 0`), calls `add_transfer_message` then `update_storage_balance(transfer_owner, ...)`. If `update_storage_balance` panics (because the account is absent from `accounts_balances`), the entire callback reverts — including the `add_transfer_message` — leaving no pending entry.

**`storage_unregister`** (storage.rs lines 222–230): with `force=true` the pending-transfer balance check is skipped entirely and `accounts_balances.remove(&account_id)` executes unconditionally. Crucially, because `remove_transfer_message` already released the storage cost back to `available`, even `force=false` would pass the check at line 224–227 (`total - available == required_balance_for_account()`), making the attack viable without `force`.

The window between `remove_transfer_message` + `ft_transfer_call` dispatch and the callback resolution spans at least one NEAR block, giving the user a concrete opportunity to unregister.

I was unable to read the full `update_storage_balance` implementation in lib.rs to confirm it panics on a missing account (the grep confirmed it exists there but I hit the iteration limit before reading it). Given the codebase-wide pattern of `near_expect` (which panics on `None`) applied to `accounts_balances.get()`, a panic for an unregistered account is the expected behavior — but this is the one unverified link in the chain.

### Title
Callback panic on unregistered storage account permanently locks BTC-wrapped tokens after UTXO connector rejection — (`near/omni-bridge/src/btc.rs`)

### Summary
When the UTXO connector rejects an `ft_transfer_call` (returns 0), `submit_transfer_to_btc_connector_callback` attempts to restore the pending transfer and then update the owner's storage balance. If the owner unregistered their storage account during the cross-contract call window, `update_storage_balance` panics, reverting the re-insertion of the pending transfer. The tokens are refunded to the bridge contract by the FT standard but are permanently unrecoverable because no pending entry exists.

### Finding Description
1. A trusted relayer calls `submit_transfer_to_utxo_chain_connector`.
2. At line 84 of btc.rs, `remove_transfer_message` removes the pending entry and releases the storage cost to the user's `available` balance.
3. The function dispatches `ft_transfer_call` to the UTXO connector and schedules `submit_transfer_to_btc_connector_callback`.
4. Before the callback resolves (a separate NEAR block), the user calls `storage_unregister(force=true)` — or even `force=false`, since no pending-transfer storage is held — removing their entry from `accounts_balances`.
5. The UTXO connector rejects the transfer (`result.0 == 0`); the FT contract refunds the tokens to the bridge.
6. The callback enters the `else` branch (btc.rs line 114), calls `add_transfer_message`, then calls `update_storage_balance(transfer_owner, ...)`.
7. `update_storage_balance` panics because `transfer_owner` is absent from `accounts_balances`.
8. The panic reverts the entire callback, including the `add_transfer_message` call.
9. `pending_transfers` is empty for this transfer; the tokens sit in the bridge contract with no mechanism for the user to reclaim them.

### Impact Explanation
The user's BTC-wrapped tokens are permanently locked in the bridge contract. There is no admin recovery path, no re-submission entry, and no refund mechanism. This matches the critical impact category: **permanent irrecoverable lock of user funds in bridge flows**.

### Likelihood Explanation
- The cross-contract call window is at least one NEAR block, making the race condition reliably exploitable.
- The user does not need `force=true`; `force=false` suffices because `remove_transfer_message` already freed the storage cost.
- The UTXO connector rejection can be engineered (e.g., by crafting a transfer the connector will refuse) or can occur naturally.
- No privileged access is required beyond having a pending BTC transfer.

### Recommendation
In `submit_transfer_to_btc_connector_callback`, before calling `update_storage_balance`, check whether `transfer_owner` is still registered. If not, either:
- Re-register the account automatically (re-insert a minimal `StorageBalance` entry funded from the bridge's own balance), or
- Emit an event and store the orphaned transfer under a recoverable admin key so the user can re-register and reclaim.

Additionally, consider holding the storage cost in escrow (not releasing it to `available`) until the callback resolves, so that `storage_unregister` cannot succeed while a UTXO connector call is in flight.

    #[private]
    pub fn submit_transfer_to_btc_connector_callback(
        &mut self,
        transfer_msg: TransferMessage,
        transfer_owner: AccountId,
        fee_recipient: AccountId,
        #[callback_result] call_result: &Result<U128, PromiseError>,
    ) -> PromiseOrValue<()> {
        if matches!(call_result, Ok(result) if result.0 > 0) {
            let token_fee = transfer_msg.fee.fee.0;
            self.send_fee_internal(&transfer_msg, fee_recipient, token_fee)
        } else {
            let required_storage_balance =
                self.add_transfer_message(transfer_msg, transfer_owner.clone());

            self.update_storage_balance(
                transfer_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            );

            PromiseOrValue::Value(())
        }
    }

    pub fn storage_unregister(&mut self, force: Option<bool>) -> bool {
        assert_one_yocto();
        let account_id = env::predecessor_account_id();
        let Some(storage) = self.storage_balance_of(&account_id) else {
            return false;
        };

        if !force.unwrap_or_default() {
            require!(
                storage.total.saturating_sub(storage.available)
                    == self.required_balance_for_account(),
                BridgeError::StoragePendingTransfers.as_ref()
            );
        }

        self.accounts_balances.remove(&account_id);

        let refund = self
            .required_balance_for_account()
            .saturating_add(storage.available);
        Promise::new(account_id).transfer(refund).detach();
        true
    }
