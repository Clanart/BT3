### Title
`MerkleBlob::batch_insert` Bypasses Duplicate Key/Hash Validation, Enabling Merkle Tree Corruption — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
The `batch_insert` method in `MerkleBlob` silently skips the duplicate-key and duplicate-hash guards that the single-item `insert` method enforces. Any caller — including the exposed Python/WASM binding — can supply a batch containing repeated `KeyId` or `Hash` values and corrupt the tree's internal structure, producing an incorrect root hash and invalidating all subsequent proofs of inclusion or exclusion.

### Finding Description

**Root cause — inconsistent invariant enforcement**

`insert` explicitly rejects duplicates before touching the blob:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` routes items through `insert` only for the first two leaves when the tree is empty or has one leaf. Every other item is written directly via `insert_entry_to_blob` with no duplicate check:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 570-603
pub fn batch_insert(
    &mut self,
    mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
) -> Result<(), Error> {
    if self.block_status_cache.leaf_count() <= 1 {
        for _ in 0..2 {
            let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                return Ok(());
            };
            self.insert(key, value, &hash, InsertLocation::Auto {})?;  // ← checked
        }
    }
    // ↓ ALL remaining items (or ALL items when leaf_count > 1) bypass checks
    for ((key, value), hash) in keys_values_hashes {
        let new_leaf_index = self.get_new_index();
        let new_block = Block { ... };
        self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← unchecked
        indexes.push(new_leaf_index);
    }

**Exploit path**

When a duplicate `KeyId` is inserted via the unchecked path, `insert_entry_to_blob` writes a new leaf block and updates `block_status_cache.key_to_index` to point to the new index. The old leaf block remains in the blob at its original index, still referenced by its parent internal node. The result is a split state:

- The cache says the key lives at index N (new leaf).
- The tree's internal-node pointers still reference index M (old leaf).
- `calculate_lazy_hashes()` propagates the old leaf's hash upward, producing a root that does not reflect the new leaf.
- `get_proof_of_inclusion(key)` walks from the cache-reported index N, generating a proof path that does not match the actual tree, so `proof.valid()` returns `false` for a key that is genuinely present.
- Conversely, the old leaf at index M is now an orphan: it is reachable via internal-node traversal but absent from the cache, so `check_integrity()` raises `IntegrityKeyNotInCache`.

The same corruption occurs for a duplicate `Hash`: the `leaf_hash_to_index` cache entry is overwritten, leaving the old leaf's hash untracked and breaking exclusion proofs.

**Attacker-controlled entry point**

`batch_insert` is exposed directly in the Python wheel:

```python
# wheel/python/chia_rs/datalayer.pyi  line 331
def batch_insert(self, keys_values: list[tuple[KeyId, ValueId]], hashes: list[bytes32]): ...
``` [3](#0-2) 

Any unprivileged caller of the Python or Rust API can supply a list with repeated keys or hashes. No privilege, key material, or network access is required.

**Analog to the external report**

The external report's root cause is a fixed assumption (1 quote-token = 1 USD) that the code never validates at runtime; when the assumption breaks, accounting diverges from reality. Here, `batch_insert` makes the fixed assumption that all supplied keys and hashes are unique — the same invariant that `insert` validates — but never checks it. When the assumption breaks (duplicate input), the tree's structural invariants diverge from the cache's view, corrupting every downstream operation that depends on the root hash or proofs.

### Impact Explanation

Corrupted `MerkleBlob` state produces an incorrect root hash and invalid proofs of inclusion/exclusion. Any DataLayer store that relies on these proofs to verify committed state accepts forged or stale evidence. This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

The Python binding is the primary public interface for DataLayer operations. `batch_insert` is the recommended path for bulk updates. A single call with one repeated key is sufficient to trigger the corruption. No special privileges are needed.

### Recommendation

Before the unchecked insertion loop in `batch_insert`, validate each incoming key and hash against `block_status_cache`:

```rust
for ((key, value), hash) in &keys_values_hashes {
    if self.block_status_cache.contains_key(*key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(hash) {
        return Err(Error::HashAlreadyPresent());
    }
}
```

Alternatively, deduplicate the input before processing, or extend `insert_entry_to_blob` to enforce the same invariants that `insert` does.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(blob=bytearray())

# Pre-populate so leaf_count > 1, bypassing the guarded path entirely
for i in range(3):
    kid = KeyId(i)
    vid = ValueId(i)
    h   = hashlib.sha256(i.to_bytes(8, "big")).digest()
    blob.insert(kid, vid, h)

blob.calculate_lazy_hashes()
root_before = blob.get_root_hash()

# batch_insert with a duplicate key (key=0 already exists)
dup_key = KeyId(0)
dup_val = ValueId(999)
dup_hash = hashlib.sha256(b"duplicate").digest()

blob.batch_insert([(dup_key, dup_val)], [dup_hash])
blob.calculate_lazy_hashes()
root_after = blob.get_root_hash()

# Root hash changes even though no new unique key was added
assert root_before != root_after, "tree silently corrupted"

# Proof for the duplicate key is now invalid
proof = blob.get_proof_of_inclusion(dup_key)
assert not proof.valid(), "proof is structurally broken"
```
