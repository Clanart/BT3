### Title
Rejected L1 Handler Transactions Silently Dropped During `CatchingUp` State, Enabling Re-Proposal — (`crates/apollo_l1_provider/src/l1_provider.rs`)

### Summary
`commit_block` unconditionally discards the `rejected_txs` argument when the provider is in `CatchingUp` state. Both the immediate-apply path (line 427) and the backlog-drain loop (line 464) call `apply_commit_block` with an empty `rejected_txs`. Consequently, every L1 handler transaction that the batcher rejected during a catchup window is never transitioned to `Rejected` state, remains in `Pending`, stays in `proposable_index`, and is eligible for re-proposal in the very next block after catchup completes.

### Finding Description
The root cause is the missing parameter in the call on line 315:

```rust
// rejected_txs is received but never forwarded
return self.accept_commit_while_catching_up(committed_txs, height);
```

`accept_commit_while_catching_up` has no `rejected_txs` parameter at all, so the information is structurally unrepresentable. Both internal call sites therefore pass `Default::default()` (an empty `IndexSet`). The backlog struct (`commit_block_backlog`) stores only `committed_txs` and `height`, confirming that `rejected_txs` is never persisted for deferred application. [7](#0-6) [2](#0-1) [3](#0-2) 

### Impact Explanation
An L1 handler transaction that was included in a block proposal but whose execution was rejected by the batcher should be permanently removed from the proposable set. Because it is not, it will be re-proposed in the next block after catchup. If the underlying L1 message is still valid on L1 (e.g., not yet consumed), the re-proposal will succeed, executing the L1→L2 message a second time. This constitutes a **duplicate L1 message execution** — a wrong L1 message value accepted and stored by the sequencer — matching the "Critical: Wrong … L1 message … from blockifier/syscall/execution logic" impact category.

### Likelihood Explanation
`CatchingUp` is a normal operational state entered at every startup and after any crash or height mismatch. Any block that contains a rejected L1 handler transaction and is committed while the provider is catching up will silently lose the rejection record. No adversarial action is required; this is a deterministic code path exercised in ordinary operation.

### Recommendation
1. Add `rejected_txs: IndexSet<TransactionHash>` to `accept_commit_while_catching_up`'s signature and forward it from `commit_block`.
2. Add a `rejected_txs` field to the backlog entry struct so that deferred blocks carry their full rejection set.
3. Pass the stored `rejected_txs` (not `Default::default()`) in both the `Equal` branch and the backlog-drain loop.
