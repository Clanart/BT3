### Title
`delete_declared_classes` in `revert_state_diff` unconditionally deletes Sierra class bodies for migrated (non-newly-declared) classes, leaving a stale `declared_classes_block_table` marker that causes `DBInconsistency` errors and wrong RPC/execution results - (File: crates/apollo_storage/src/state/mod.rs)

### Summary

When `revert_state_diff` is called for a block that contains `migrated_compiled_classes` entries (compiled-class-hash upgrades for previously declared Sierra classes), the helper `delete_declared_classes` unconditionally deletes the Sierra class body from `declared_classes_table` for every key in `class_hash_to_compiled_class_hash`, including migrated class hashes that were originally declared in an earlier, non-reverted block. The companion helper `delete_declared_classes_block` correctly guards against this with a `class_block_entry == block_number` check, but `delete_declared_classes` has no such guard. After the revert, `declared_classes_block_table` still holds the stale `ClassHash → original_block_number` entry while `declared_classes_table` no longer holds the `ClassHash → LocationInFile` entry, creating a permanent inconsistency that causes `StorageError::DBInconsistency` on any subsequent class lookup and `CasmTableNotSynced` / `CLASS_HASH_NOT_FOUND` on any RPC or execution path that touches the affected class.

### Impact Explanation

After a reorg that reverts a block containing `migrated_compiled_classes`, any contract whose compiled-class hash was migrated in that block becomes permanently inaccessible through the storage layer. All downstream consumers — `starknet_getClass`, `starknet_getCompiledClass`, fee estimation, `starknet_call`, `starknet_simulateTransactions`, and `starknet_traceTransaction` — will return authoritative-looking wrong values (`CLASS_HASH_NOT_FOUND`, `CasmTableNotSynced`, or `DBInconsistency`) for those classes. Transactions invoking those contracts will be incorrectly rejected or produce wrong execution results.

### Likelihood Explanation

The trigger requires two conditions to coincide: (1) at least one block containing `migrated_compiled_classes` entries has been synced, and (2) a chain reorganization causes `revert_state_diff` to be called for that block. Compiled-class-hash migration is an active production feature (tested in `test_compiled_class_hash_migration`). Reorgs are handled by the central sync path. The combination is uncommon but entirely within normal operating parameters and requires no privileged access.

### Recommendation

Add a block-number guard to `delete_declared_classes` identical to the one already present in `delete_declared_classes_block`. Pass `block_number` and `declared_classes_block_table` as additional parameters and skip deletion for any class whose `declared_classes_block_table` entry does not equal the block being reverted:

```rust
fn delete_declared_classes<'env>(
    txn: &'env DbTransaction<'env, RW>,
    thin_state_diff: &ThinStateDiff,
    declared_classes_table: &'env DeclaredClassesTable<'env>,
    declared_classes_block_table: &'env DeclaredClassesBlockTable<'env>,
    file_handlers: &FileHandlers<RW>,
    block_number: BlockNumber,
) -> StorageResult<IndexMap<ClassHash, SierraContractClass>> {
    let mut deleted_data = IndexMap::new();
    for class_hash in thin_state_diff.class_hash_to_compiled_class_hash.keys() {
        // Only delete the class body if it was first declared in this block.
        let class_block_entry = declared_classes_block_table.get(txn, class_hash)?;
        if class_block_entry != Some(block_number) {
            continue;
        }
        let Some(contract_class_location) = declared_classes_table.get(txn, class_hash)? else {
            continue;
        };
        deleted_data.insert(*class_hash,
            file_handlers.get_contract_class_unchecked(contract_class_location)?);
        declared_classes_table.delete(txn, class_hash)?;
    }
    Ok(deleted_data)
}
```

### Proof of Concept

1. Sync block N: declare Sierra class `C` with compiled-class hash `H1`. `declared_classes_block_table[C] = N`, `declared_classes_table[C] = <location>`.
2. Sync block M > N: `migrated_compiled_classes` contains `(C, H2)`. `class_hash_to_compiled_class_hash` in the stored `ThinStateDiff` for block M contains `C`.
3. Reorg: call `revert_state_diff(M)`.
   - `delete_declared_classes_block`: reads `declared_classes_block_table[C] = N ≠ M` → skips. `declared_classes_block_table[C]` remains `N`. ✓
   - `delete_declared_classes`: reads `declared_classes_table[C] = Some(<location>)` → deletes. `declared_classes_table[C]` is now absent. ✗
4. Call `get_class_definition_at(state_after_N, C)`:
   - `declared_classes_block_table[C]` → `Some(N)` → passes `is_before` check.
   - `declared_classes_table[C]` → `None`.
   - `state_after_N.is_after(class_marker)` → false (class marker ≥ N+1) → returns `Err(StorageError::DBInconsistency)`.
5. Any RPC or execution call touching class `C` now returns an incorrect error, permanently, until the node is re-synced from scratch. [3](#0-2) [2](#0-1) [1](#0-0)
