### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `execution/syscall_impls.cairo`)

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the supplied new class hash corresponds to a declared (known) contract class. An unprivileged user can trigger a contract to replace its class with an undeclared class hash, rendering the contract permanently non-executable and freezing any funds it holds.

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the requested new class hash from the syscall request and immediately writes it into the contract state without any check that the class has been declared:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing validation. The OS writes the arbitrary `class_hash` directly into the `contract_state_changes` dictionary with no cross-reference against `contract_class_changes` (the declared class registry). [2](#0-1) 

By contrast, the `execute_declare_transaction` path enforces `assert_not_zero(compiled_class_hash)` and uses `dict_update` with `prev_value=0` to guarantee a class is declared exactly once before it can be used: [3](#0-2) 

No equivalent guard exists in `execute_replace_class`.

### Impact Explanation

Once a contract's `class_hash` field in the state tree is set to an undeclared hash, every subsequent call to that contract will fail at class-lookup time inside the OS execution engine. Because the state root is committed on-chain with this corrupted class hash, there is no recovery path: the contract is permanently non-executable. Any ERC-20 tokens, ETH, or other assets held in the contract's storage are permanently frozen. This matches the **Critical — Permanent freezing of funds** impact category.

### Likelihood Explanation

The attack surface is any upgradeable contract whose `upgrade` entry point passes caller-supplied calldata directly to the `replace_class` syscall without first verifying the new class hash exists in the declared-class registry. Such patterns are common in proxy/upgradeable wallet and DeFi contracts on StarkNet. An unprivileged user needs only to:

1. Identify a contract with a public upgrade path that does not validate the class hash.
2. Submit an invoke transaction calling that entry point with an arbitrary (undeclared) felt as the new class hash.
3. The OS accepts the syscall, commits the corrupted state root, and the contract is bricked.

No privileged role, leaked key, or operator cooperation is required beyond the normal transaction submission flow.

### Recommendation

Inside `execute_replace_class`, before writing the new class hash to `contract_state_changes`, perform a lookup in `contract_class_changes` (or the equivalent squashed class dict) to assert that `class_hash` maps to a non-zero compiled class hash. Concretely:

```cairo
// Assert the class has been declared (compiled_class_hash != 0).
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the invariant already enforced during `execute_declare_transaction` and closes the data-validation gap acknowledged in the existing TODO comment.

### Proof of Concept

1. Declare and deploy a simple upgradeable contract `Vault` that holds user funds and exposes:
   ```
   fn upgrade(new_class_hash: ClassHash) { replace_class_syscall(new_class_hash); }
   ```
2. Users deposit funds into `Vault`.
3. Attacker calls `Vault.upgrade(0xdeadbeef)` — an arbitrary felt that has never been declared.
4. The OS executes `execute_replace_class`, writes `class_hash=0xdeadbeef` into the state, and commits the block.
5. All subsequent calls to `Vault` fail at class resolution; deposited funds are permanently inaccessible.

The root cause is solely in the OS: `execute_replace_class` in `syscall_impls.cairo` (line 898) performs no validation against the declared-class registry, accepting any felt as a valid class hash. [4](#0-3)
