### Title
Unverified `state_diff_length` vs `concatenated_counts` in P2P-Synced Block Header Allows Malicious Peer to Corrupt Stored State Diff — (`crates/apollo_p2p_sync/src/client/header.rs`)

---

### Summary

`HeaderStreamBuilder::parse_data_for_block` accepts a `SignedBlockHeader` from an untrusted P2P peer and stores it after checking only block number and signature count. It never cross-checks the `state_diff_length` field against the value encoded in `concatenated_counts`, even though `concatenated_counts` is the authoritative source (it is hashed into the block hash). The stored `state_diff_length` is then used verbatim by `StateDiffStreamBuilder::parse_data_for_block` as the loop bound for accepting state diff chunks. A malicious peer can set `state_diff_length = 0` while supplying a valid-looking `concatenated_counts`, causing the node to store an empty state diff for a block that had real state changes, permanently corrupting its view of chain state.

---

### Finding Description

**Root cause — the missing invariant check (the "dead code" analog):**

`concatenated_counts` packs `transaction_count | event_count | state_diff_length | l1_da_mode` into a single Felt and is chained into the block hash. It is the canonical, commitment-bound source for all three counts. The code already extracts `n_events` from it correctly:

```rust
// crates/apollo_p2p_sync/src/client/header.rs  lines 138-141
let n_events =
    extract_event_count_from_concatenated_counts(&header_commitments.concatenated_counts);
// TODO(Shahak): Verify `n_transactions` and `state_diff_length` match values in
// concatenated_counts.
``` [1](#0-0) 

The TODO is the exact analog of the external bug's dead-code check: the enforcement that `state_diff_length` must equal the value in `concatenated_counts` is acknowledged but never executed. `state_diff_length` is instead taken from a separate, peer-controlled field.

**P2P untrusted-peer path:**

When a `SignedBlockHeader` arrives over the wire, `state_diff_length` is decoded from the protobuf `state_diff_commitment.state_diff_length` sub-field:

```rust
// crates/apollo_protobuf/src/converters/header.rs  lines 116-121
let state_diff_length = value.state_diff_commitment.as_ref().map(|state_diff_commitment| {
    state_diff_commitment
        .state_diff_length
        .try_into()
        .expect("Failed converting u64 to usize")
});
``` [2](#0-1) 

`parse_data_for_block` then validates only block number and signature count before accepting the header:

```rust
// crates/apollo_p2p_sync/src/client/header.rs  lines 104-119
if block_number != signed_block_header.block_header.block_header_without_hash.block_number {
    return Err(...HeadersUnordered...);
}
if signed_block_header.signatures.len() != ALLOWED_SIGNATURES_LENGTH {
    return Err(...WrongSignaturesLength...);
}
Ok(Some(signed_block_header))
``` [3](#0-2) 

No check compares `state_diff_length` against the value encoded in `concatenated_counts`.

**Downstream consumer — `StateDiffStreamBuilder`:**

The stored `state_diff_length` is read back as the hard loop bound for state diff chunk acceptance:

```rust
// crates/apollo_p2p_sync/src/client/state_diff.rs  lines 62-70
let target_state_diff_len = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("...")
    .state_diff_length
    .ok_or(P2pSyncClientError::OldHeaderInStorage { ... })?;

while current_state_diff_len < target_state_diff_len { ... }
``` [4](#0-3) 

If `target_state_diff_len = 0`, the loop body never executes and an empty `ThinStateDiff` is returned and written to storage.

**`concatenated_counts` is the authoritative encoding:**

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs  lines 374-393
pub fn concat_counts(
    transaction_count: usize,
    event_count: usize,
    state_diff_length: usize,
    l1_data_availability_mode: L1DataAvailabilityMode,
) -> Felt { ... }
``` [5](#0-4) 

This value is chained directly into `calculate_block_hash` at the `concatenated_counts` position. [6](#0-5) 

---

### Impact Explanation

A malicious P2P peer sends a `SignedBlockHeader` with `state_diff_commitment.state_diff_length = 0` (or any value ≠ the real diff length) while embedding a plausible `concatenated_counts`. The node stores the header with `state_diff_length = 0`. The subsequent state diff sync loop immediately exits, writing an empty `ThinStateDiff` for a block that had real storage/nonce/class changes. Every RPC query for storage values, nonces, or class hashes for contracts touched in that block returns stale or zero values — an authoritative-looking wrong answer. This matches the **High** impact: *RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.*

---

### Likelihood Explanation

Any node that participates in the P2P network can act as a peer. No privileged key or special role is required. The attacker only needs to respond to a header query with a well-formed protobuf message that passes the two existing checks (correct block number, one signature). The manipulated field (`state_diff_commitment.state_diff_length`) is a plain integer in the protobuf and requires no cryptographic forgery.

---

### Recommendation

In `HeaderStreamBuilder::parse_data_for_block`, after deserializing the header, extract `state_diff_length` and `n_transactions` from `concatenated_counts` (using the same `extract_*_from_concatenated_counts` helpers already used for `n_events`) and reject the header with `BadPeerError` if either value disagrees with the corresponding field in the received header. The TODO at line 140–141 of `header.rs` already identifies exactly this gap. [7](#0-6) 

---

### Proof of Concept

1. Attacker runs a node and connects to the victim's P2P network.
2. Victim issues a header query for block N.
3. Attacker responds with a `SignedBlockHeader` where:
   - `block_number = N` (passes the block-number check)
   - `signatures.len() = 1` (passes the signature-count check)
   - `state_diff_commitment.state_diff_length = 0` (attacker-controlled integer)
   - `concatenated_counts` encodes `state_diff_length = K` for any K > 0 (no cryptographic constraint on this field in the current validation)
4. `parse_data_for_block` returns `Ok(Some(signed_block_header))` — no rejection.
5. `write_to_storage` persists the header with `state_diff_length = Some(0)`.
6. `StateDiffStreamBuilder::parse_data_for_block` reads `target_state_diff_len = 0`; the `while` loop body never executes; `Ok(Some((ThinStateDiff::default(), N)))` is returned.
7. `append_state_diff(N, ThinStateDiff::default())` writes an empty diff to storage.
8. All subsequent `starknet_getStorageAt` / `starknet_getNonce` / `starknet_getClassHashAt` RPC calls for contracts modified in block N return wrong (stale) values.

```rust
pub fn concat_counts(
    transaction_count: usize,
    event_count: usize,
    state_diff_length: usize,
    l1_data_availability_mode: L1DataAvailabilityMode,
) -> Felt {
    let l1_data_availability_byte: u8 = match l1_data_availability_mode {
        L1DataAvailabilityMode::Calldata => 0,
        L1DataAvailabilityMode::Blob => 0b10000000,
    };
    let concat_bytes = [
        to_64_bits(transaction_count).as_slice(),
        to_64_bits(event_count).as_slice(),
        to_64_bits(state_diff_length).as_slice(),
        &[l1_data_availability_byte],
        &[0_u8; 7], // zero padding
    ]
    .concat();
    Felt::from_bytes_be_slice(concat_bytes.as_slice())
}
