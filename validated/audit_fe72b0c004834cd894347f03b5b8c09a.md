### Title
Felt Arithmetic Overflow in `compute_max_possible_fee` Allows Fee Bypass — (`File: execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` performs unchecked felt arithmetic over resource-bound fields whose upper values are not constrained. Because Cairo felt arithmetic wraps modulo the field prime P, a crafted transaction can make the function return exactly 0, causing `charge_fee` to silently skip the ERC-20 transfer and execute the transaction for free.

---

### Finding Description

In `charge_fee`, the OS computes the maximum fee a transaction may be charged and immediately gates all fee logic on whether that value is zero:

```cairo
// transaction_impls.cairo:121-125
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();   // ← fee charging entirely skipped
}
```

`compute_max_possible_fee` is:

```cairo
// transaction_impls.cairo:99-101
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
```

This is pure felt arithmetic — no range checks, no overflow guards. The only validation applied to `max_price_per_unit` anywhere in the OS is inside `pack_resource_bounds` (called during transaction-hash computation):

```cairo
// transaction_hash.cairo:104-105
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
assert_nn(resource_bounds.max_price_per_unit);   // ← only ≥ 0, no upper bound
```

`max_amount` is bounded to 64 bits, but `max_price_per_unit` is bounded only to be non-negative — it may be any value in `[0, P-1]` where P ≈ 2^251 + 17·2^192 + 1 is the StarkNet field prime.

Because felt multiplication wraps modulo P, the sum in `compute_max_possible_fee` can be made to equal exactly 0 mod P with attacker-chosen inputs.

---

### Impact Explanation

When `max_fee` wraps to 0, `charge_fee` returns immediately without executing the ERC-20 transfer. The sequencer receives no payment for the transaction. An attacker who can reliably produce zero-fee transactions can:

1. **Direct loss of funds (Critical):** The sequencer is deprived of all fee revenue for every crafted transaction; the fee tokens that should have been transferred remain in the user's account.
2. **Network halt (High):** With free execution, an attacker can flood the network with computationally expensive transactions at zero cost, exhausting sequencer resources and halting block production.

Both impacts are in the allowed scope.

---

### Likelihood Explanation

The exploit requires only crafting a standard V3 transaction with specific `max_price_per_unit` values. No privileged access, leaked keys, or external dependencies are needed. The transaction passes all hash-validation checks because `pack_resource_bounds` only enforces `max_price_per_unit >= 0`. Any unprivileged user who can submit a V3 transaction can trigger this.

---

### Proof of Concept

Let P = StarkNet field prime = 2^251 + 17·2^192 + 1.

Set the following resource bounds in a V3 invoke/declare/deploy-account transaction:

| Field | Value |
|---|---|
| `l1_gas.max_amount` | 1 |
| `l1_gas.max_price_per_unit` | P − 1 (i.e., −1 mod P) |
| `l2_gas.max_amount` | 1 |
| `l2_gas.max_price_per_unit` | 1 |
| `tip` | 0 |
| `l1_data_gas.max_amount` | 0 |
| `l1_data_gas.max_price_per_unit` | 0 |

Evaluation of `compute_max_possible_fee`:

```
max_fee = 1 * (P − 1) + 1 * (1 + 0) + 0 * 0
        = (P − 1) + 1
        = P
        ≡ 0  (mod P)
```

The condition `if (max_fee == 0)` is satisfied. `charge_fee` returns immediately. The ERC-20 transfer is never executed. The transaction is processed with zero fee paid.

All three V3 transaction types call `charge_fee`:
- Invoke: [1](#0-0) 
- Deploy-account: [2](#0-1) 
- Declare: [3](#0-2)

---

### Recommendation

1. Add an upper-bound range check on `max_price_per_unit` inside `pack_resource_bounds` (e.g., `assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1)`) so that the product `max_amount * max_price_per_unit` cannot exceed 2^192 − 1, well below the field prime.
2. Alternatively, recompute `max_fee` as a `Uint256` using safe 128-bit multiplication and addition, consistent with the `Uint256` type already used for the ERC-20 transfer amount.
3. Add an explicit assertion in `compute_max_possible_fee` that the result is non-negative and within a safe range before returning it.
