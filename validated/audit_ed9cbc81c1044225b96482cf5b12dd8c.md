### Title
Missing Validation of Declared Class Existence in `replace_class` Syscall Allows Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS accepts an arbitrary `class_hash` from the syscall request and commits it to state without verifying that the hash corresponds to a declared contract class. A malicious contract deployer can exploit this to permanently freeze all funds held in a contract by replacing its class with an undeclared hash, making the contract permanently uncallable.

---

### Finding Description

In `execute_replace_class`, the OS reads `class_hash` directly from the syscall request and writes it into `contract_state_changes` with no check that the hash exists in the declared class tree (`contract_class_changes`). The code even contains an explicit TODO acknowledging this missing check:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

Once this state change is committed to the Patricia Merkle tree via `compute_contract_state_commitment`, the contract's class hash in the global state permanently points to a non-existent class. Any subsequent transaction targeting this contract will fail to resolve the class, making the contract permanently uncallable and all funds within it permanently frozen. [2](#0-1) 

The `replace_class` syscall is dispatched unconditionally for any executing contract: [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value and the block is finalized, the contract becomes permanently uncallable. Any ERC-20 balances, NFTs, or other assets stored in the contract's storage are irrecoverably frozen. There is no recovery path: the state commitment is cryptographically finalized, and no syscall exists to restore a contract's class hash from outside the contract itself (which is now uncallable).

---

### Likelihood Explanation

The attack is reachable by any **contract deployer** — an explicitly listed unprivileged role. The attack requires:

1. Deploying a contract whose logic conditionally calls `replace_class` with an arbitrary felt (e.g., `0x1`).
2. Attracting user deposits (e.g., by presenting as a legitimate vault, pool, or token contract).
3. Triggering the `replace_class` call with an undeclared hash.

No privileged access, leaked keys, or external protocol compromise is required. The OS provides no on-chain guard against this.

---

### Recommendation

Before committing the new class hash in `execute_replace_class`, verify that `class_hash` exists as a key in `contract_class_changes` (i.e., it was declared in the current block) or that it exists in the contract class Patricia tree (i.e., it was declared in a prior block). The TODO comment at line 898 already identifies this gap. The check should be enforced at the Cairo OS level, not deferred to hints, so that the proof itself rejects invalid class replacements.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract that accepts deposits and, when triggered, calls `replace_class(class_hash=0xdeadbeef)` where `0xdeadbeef` is never declared.
2. Users deposit funds into `MaliciousVault`.
3. Attacker sends a transaction that triggers the `replace_class` call.
4. The OS executes `execute_replace_class`:
   - Reads `class_hash = 0xdeadbeef` from the request. [4](#0-3) 
   - Skips any existence check (the TODO). [5](#0-4) 
   - Writes the new `StateEntry` with `class_hash=0xdeadbeef` into `contract_state_changes`. [6](#0-5) 
5. The block is finalized; `compute_contract_state_commitment` commits `class_hash=0xdeadbeef` into the global state root. [7](#0-6) 
6. All future calls to `MaliciousVault` fail: the OS looks up `class_hash=0xdeadbeef`, finds no matching class, and the execution aborts.
7. All user funds in `MaliciousVault`'s storage are permanently frozen.
