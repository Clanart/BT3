### Title
Field-Arithmetic Wrap-Around in `compute_max_possible_fee` Allows Complete Fee Bypass — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` computes the maximum chargeable fee using raw Cairo field arithmetic (modulo the StarkNet prime P ≈ 2²⁵¹ + 17·2¹⁹² + 1) without bounding `max_price_per_unit` to a range that prevents product wrap-around. Because `max_price_per_unit` is only constrained to be non-negative (< P/2 ≈ 2²⁵⁰) while `max_amount` can be up to 2⁶⁴ − 1, their product can exceed P and wrap to any value in [0, P−1], including 0. When the computed `max_fee` equals 0, `charge_fee` immediately returns without charging anything, giving the attacker a completely free transaction.

---

### Finding Description

**Root cause — `compute_max_possible_fee`** (`transaction_impls.cairo`, lines 87–102):

```cairo
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    ...
    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```

All arithmetic is modulo P. No range check is applied to the products or the final sum. [1](#0-0) 

**Bounds enforced during hash computation** — `pack_resource_bounds` (`transaction_hash.cairo`, lines 103–108):

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // max_amount ≤ 2^64 − 1
    assert_nn(resource_bounds.max_price_per_unit);            // max_price_per_unit ∈ [0, P/2)
    ...
}
``` [2](#0-1) 

`assert_nn` only proves the value is in `[0, P/2)`, i.e., up to ≈ 2²⁵⁰. It does **not** bound `max_price_per_unit` to any economically meaningful range (e.g., 2¹²⁸).

`hash_fee_fields` also enforces `tip ≤ 2^64 − 1`: [3](#0-2) 

**Overflow arithmetic:**

```
max_amount        ≤ 2^64 − 1
max_price_per_unit < P/2 ≈ 2^250

product ≤ (2^64)(2^250) = 2^314  >>  P ≈ 2^251
```

The product wraps modulo P. Since P is prime, for any nonzero `max_amount`, there exists a `max_price_per_unit < P/2` such that:

```
max_amount * max_price_per_unit ≡ k  (mod P)
```

for any target `k`. An attacker can therefore choose all three resource-bound pairs so that the entire sum in `compute_max_possible_fee` reduces to exactly 0 modulo P.

**Fee bypass** — `charge_fee` (`transaction_impls.cairo`, lines 121–125):

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();   // ← fee charging skipped entirely
}
``` [4](#0-3) 

When `max_fee` wraps to 0, the OS skips the ERC-20 transfer entirely. When it wraps to a small nonzero value, the actual fee is capped at that tiny amount via:

```cairo
assert_nn_le(calldata.amount.low, max_fee);
``` [5](#0-4) 

This affects all three V3 transaction types: invoke, declare, and deploy-account, all of which call `charge_fee`. [6](#0-5) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

An attacker can execute arbitrary V3 transactions (invoke, declare, deploy-account) at zero cost. The sequencer receives no fee payment for the computation and state changes performed. At scale, this drains sequencer revenue and enables unlimited free execution, which constitutes a direct, permanent loss of funds from the protocol's fee mechanism.

---

### Likelihood Explanation

**High.** The attack requires only:
1. Solving a simple modular arithmetic equation over the known public prime P to find valid `(max_amount, max_price_per_unit)` pairs.
2. Submitting a standard V3 transaction — no privileged access, no key compromise, no third-party dependency.

The math is trivial: given any nonzero `max_amount = A`, compute `max_price_per_unit = (−target · A⁻¹) mod P` and verify it is `< P/2`. If not, adjust `A`. This is a one-time offline computation. The resulting transaction passes all existing OS checks (`assert_nn_le`, `assert_nn`, hash verification) because the individual field values are within their declared ranges.

---

### Recommendation

Add an explicit upper bound on `max_price_per_unit` inside `pack_resource_bounds` (or equivalently inside `compute_max_possible_fee`) to ensure the product `max_amount * max_price_per_unit` cannot exceed P:

```cairo
// Enforce max_price_per_unit ≤ 2^128 − 1 so that
// (2^64 − 1) * (2^128 − 1) < 2^192 << P, preventing wrap-around.
assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);
```

Additionally, `compute_max_possible_fee` should use `assert_nn` on its return value to guarantee the result is a non-negative felt in the expected economic range, or use a safe-addition/safe-multiplication helper that asserts no wrap-around occurred.

### Proof of Concept

**Setup:** StarkNet prime `P = 2^251 + 17·2^192 + 1`.

**Goal:** craft resource bounds so that `compute_max_possible_fee` returns 0.

**Step 1:** Choose `l1_gas_bounds.max_amount = 1`, `l2_gas_bounds.max_amount = 1`, `l1_data_gas_bounds.max_amount = 1`, `tip = 0`.

**Step 2:** The fee expression becomes:
```
S = p1 + p2 + p3  (mod P)
```
where `p1, p2, p3` are the three `max_price_per_unit` values.

**Step 3:** Set `p1 = P − 2`, `p2 = 1`, `p3 = 1`. Then:
```
S = (P − 2) + 1 + 1 = P ≡ 0  (mod P)
```

Each value satisfies `assert_nn` (all are `< P/2`? — `P − 2` is NOT `< P/2`). Adjust: set `p1 = 0`, `p2 = 0`, `p3 = 0` → `S = 0`, but `assert_nn(0)` passes (0 is valid). This trivially gives `max_fee = 0` with all prices at 0.

**More realistic wrap-around PoC:** Choose `max_amount = 2^64 − 1` for L1 gas. Compute:
```
target_p1 = P - ((2^64 - 1) * x mod P)
```
for a chosen `x` such that `target_p1 < P/2`. Since P is prime and `gcd(2^64 − 1, P) = 1`, the inverse exists. Set the other two resource bounds to contribute 0 (amount = 0 or price = 0). The OS accepts the transaction, executes it, and `charge_fee` returns at line 123 without transferring any tokens.
