Let me check the `unlock_tokens_if_needed` and `lock_tokens_if_needed` implementations, and whether any recovery/cancel mechanism exists for pending transfers.

### Title
Missing Destination-Chain Registration Guard in `fin_transfer_callback` Permanently Locks User Funds — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`fin_transfer_callback` stores a cross-chain transfer in `pending_transfers` without verifying that the recipient's chain (`ChainKind::Fogo`, or any other chain) has a registered factory or token mapping on NEAR. A user who sets `recipient='fogo:<valid_sol_address>'` in an EVM `InitTransfer` event causes the transfer to be stored with a Fogo destination. Every subsequent call to `sign_transfer` for that transfer panics with `FailedToGetTokenAddress` because no `(ChainKind::Fogo, token_id)` entry exists in `token_id_to_address`. The transfer is irrecoverable: the EVM tokens are already locked/burned, the NEAR pending entry can never be completed or cancelled, and no built-in admin path removes it.

---

### Finding Description

**Step 1 — User-controlled input parsed without chain validation**

In `TryFromLog<Log<InitTransfer>> for InitTransferMessage` the `recipient` field is parsed directly:

```rust
recipient: event.data.recipient.parse().map_err(stringify)?,
``` [1](#0-0) 

`OmniAddress::from_str` accepts `"fogo:<base58>"` and produces `OmniAddress::Fogo(SolAddress)` without any chain-registration check:

```rust
"fogo" => Ok(Self::Fogo(recipient.parse().map_err(stringify)?)),
``` [2](#0-1) 

**Step 2 — `fin_transfer_callback` validates only the source chain**

The only factory check in `fin_transfer_callback` is against the *emitter* (source chain, e.g. Eth), not the destination:

```rust
require!(
    self.factories
        .get(&init_transfer.emitter_address.get_chain())
        == Some(init_transfer.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
``` [3](#0-2) 

`get_next_destination_nonce(ChainKind::Fogo)` simply increments a counter — no registration check:

```rust
fn get_next_destination_nonce(&mut self, chain_kind: ChainKind) -> Nonce {
    if chain_kind == ChainKind::Near { return 0; }
    let mut payload_nonce = self.destination_nonces.get(&chain_kind).unwrap_or_default();
    payload_nonce += 1;
    self.destination_nonces.insert(&chain_kind, &payload_nonce);
    payload_nonce
}
``` [4](#0-3) 

Because the recipient is `OmniAddress::Fogo(...)` (not `Near`), execution falls into `process_fin_transfer_to_other_chain`:

```rust
} else {
    self.process_fin_transfer_to_other_chain(predecessor_account_id, transfer_message);
    PromiseOrValue::Value(destination_nonce)
}
``` [5](#0-4) 

**Step 3 — `process_fin_transfer_to_other_chain` succeeds silently**

`is_utxo_chain()` is false for Fogo, so the only UTXO guard is skipped. `lock_tokens_if_needed` for Fogo silently returns `LockAction::Unchanged` when no `(Fogo, token_id)` entry exists in `locked_tokens` (no panic, no error):

```rust
fn lock_tokens(&mut self, chain_kind: ChainKind, token_id: &AccountId, amount: u128) -> LockAction {
    let Some(current_amount) = self.locked_tokens.get(&key) else {
        return LockAction::Unchanged;   // silent no-op
    };
    ...
}
``` [6](#0-5) 

The transfer is then stored in `pending_transfers` and `finalised_transfers` (replay-prevention set) via `add_transfer_message` / `add_fin_transfer`. The function returns successfully and emits `FinTransferEvent`. [7](#0-6) 

**Step 4 — `sign_transfer` always panics for the stuck transfer**

When a relayer later calls `sign_transfer`, it calls `get_token_address(ChainKind::Fogo, token_id)`:

```rust
let token_address = self
    .get_token_address(
        transfer_message.get_destination_chain(),
        self.get_token_id(&transfer_message.token),
    )
    .unwrap_or_else(|| {
        env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
    });
``` [8](#0-7) 

`get_token_address` is a simple map lookup:

```rust
pub fn get_token_address(&self, chain_kind: ChainKind, token: AccountId) -> Option<OmniAddress> {
    self.token_id_to_address.get(&(chain_kind, token))
}
``` [9](#0-8) 

With no Fogo token registered, this returns `None` and panics every time. The transfer can never be signed.

**Step 5 — No recovery path exists**

`remove_transfer_message` is only called from `claim_fee_callback`, which requires a `FinTransfer` proof from the destination chain. Since Fogo never finalizes the transfer (no token registered), no such proof can ever be produced. There is no admin/DAO function to cancel or remove a stuck `pending_transfers` entry. [10](#0-9) 

---

### Impact Explanation

The user's EVM tokens are locked/burned on the EVM side at `initTransfer`. The NEAR bridge stores the transfer in `pending_transfers` and marks it in `finalised_transfers` (preventing replay). `sign_transfer` panics permanently. `claim_fee` cannot be called (no Fogo FinTransfer proof). The transfer is irrecoverable without a contract upgrade. **User funds are permanently frozen.**

---

### Likelihood Explanation

`fin_transfer` is gated by `#[trusted_relayer]`, so a trusted relayer must submit the proof. However:
- The user fully controls the `recipient` string in the EVM `initTransfer` event.
- Automated relayers that relay all cryptographically valid EVM proofs without checking destination-chain registration on NEAR will trigger this.
- The NEAR bridge contract itself provides no guard — it relies entirely on relayer-side filtering that is not enforced on-chain.
- `ChainKind::Fogo` is a fully recognized enum variant with a valid `OmniAddress` parser, making the input indistinguishable from a legitimate Fogo transfer at the proof-verification layer.

Likelihood is **medium**: requires a trusted relayer to not pre-filter by destination chain, which is a realistic operational assumption for automated relayers.

---

### Recommendation

In `fin_transfer_callback`, after parsing the recipient, add a guard that the destination chain has a registered factory (or at minimum a registered token mapping) before calling `get_next_destination_nonce` and `process_fin_transfer_to_other_chain`:

```rust
// Proposed guard (pseudocode):
let dest_chain = init_transfer.recipient.get_chain();
require!(
    dest_chain == ChainKind::Near
        || self.factories.contains_key(&dest_chain),
    BridgeError::UnknownFactory.as_ref()
);
```

Alternatively, validate inside `process_fin_transfer_to_other_chain` that `get_token_address(destination_chain, token_id)` returns `Some` before storing the transfer, and revert cleanly (refund, remove from `finalised_transfers`) if not.

---

### Proof of Concept

```
Preconditions:
  - Eth factory registered on NEAR bridge
  - Token T registered for Eth (token_decimals, token_address_to_id)
  - NO factory or token mapping registered for ChainKind::Fogo

Call sequence:
  1. User calls EVM bridge initTransfer(token=T, amount=X, recipient="fogo:11111111111111111111111111111111")
     → EVM emits InitTransfer event, locks/burns user's X tokens

  2. Relayer calls NEAR fin_transfer(chain_kind=Eth, prover_args=<proof of above event>)
     → fin_transfer_callback:
         - UnknownFactory check passes (Eth factory is registered)
         - TokenDecimalsNotFound check passes (T is registered for Eth)
         - get_next_destination_nonce(Fogo) → returns 1 (no error)
         - process_fin_transfer_to_other_chain → stores transfer in pending_transfers
         - Returns PromiseOrValue::Value(1)  ← SUCCESS, no panic

  3. Relayer calls sign_transfer(transfer_id={Eth, nonce})
     → get_token_address(Fogo, token_id) → None
     → panic: ERR_FAILED_TO_GET_TOKEN_ADDRESS

  4. Repeat step 3 → always panics

Result: Transfer stuck in pending_transfers forever.
        User's EVM tokens permanently locked.
        No recovery path without contract upgrade.
```

    fn remove_transfer_message(&mut self, transfer_id: TransferId) -> TransferMessage {
        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);
        }

        transfer.message
    }
