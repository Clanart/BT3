### Title
Missing Declared Class Hash Validation in `execute_replace_class` Enables Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by a contract is actually declared in the system. This allows any contract to permanently render itself uncallable by replacing its class hash with an arbitrary undeclared value, freezing any funds held within it. The missing check is explicitly acknowledged in the codebase via a TODO comment.

---

### Finding Description

In `execute_replace_class` (lines 877–916 of `syscall_impls.cairo`), the OS processes the `replace_class` syscall by directly updating the contract's class hash in `contract_state_changes` without verifying that the new class hash corresponds to a declared contract class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The analog to the external report is direct: just as the payment system implicitly trusted operators not to act as buyers or sellers (missing a check that `operator != buyer` and `operator != seller`), the StarkNet OS implicitly trusts that contracts will only call `replace_class` with declared class hashes — missing a check that `new_class_hash ∈ declared_classes`.

When a contract's class hash is set to an undeclared value, any subsequent call to that contract fails at `execute_entry_point` in `execute_entry_point.cairo`. Specifically, `find_element` searches the `compiled_class_facts` array for the compiled class matching the hash:

```cairo
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [2](#0-1) 

If the class hash is undeclared, `find_element` will trap (no matching element), making the contract permanently uncallable. Since `replace_class` cannot be called again on an uncallable contract, the state is irreversible.

---

### Impact Explanation

Any ERC20 tokens, ETH, or STRK held in a contract whose class hash has been replaced with an undeclared value are **permanently frozen**. The contract cannot be called to withdraw or transfer funds, and there is no protocol-level recovery mechanism. This constitutes **Critical: Permanent freezing of funds**.

---

### Likelihood Explanation

This is reachable by any unprivileged contract deployer or transaction sender:

1. A user deploys a contract containing a function that calls `replace_class` with an arbitrary undeclared hash (e.g., `0xDEAD`).
2. The contract attracts funds (e.g., acting as a vault or DeFi protocol).
3. The attacker triggers the `replace_class` call.
4. The OS accepts the invalid state transition without any validation.
5. All funds are permanently frozen.

No special privileges, leaked keys, or trusted-role access is required. The entry path is a standard user-submitted invoke transaction.

---

### Recommendation

Add a validation check in `execute_replace_class` to verify that the new class hash is present in `contract_class_changes` (i.e., it has been declared). If the class hash is not found among declared classes, the syscall should fail with an appropriate error response (analogous to how `execute_call_contract` returns `ERROR_INVALID_ARGUMENT` for disallowed selectors). [3](#0-2) 

This is already acknowledged in the TODO comment at line 898. The fix should mirror the lookup pattern used in `execute_entry_point`: [4](#0-3) 

---

### Proof of Concept

1. Deploy a contract `MaliciousVault` with:
   - A `deposit()` function that accepts ERC20 tokens.
   - A `freeze()` function that calls `replace_class(0xDEAD)` where `0xDEAD` is not a declared class hash.
2. Users deposit funds into `MaliciousVault`.
3. Attacker calls `freeze()`.
4. The OS processes `replace_class(0xDEAD)` in `execute_replace_class` without validation, committing the undeclared class hash to `contract_state_changes`.
5. Any subsequent call to `MaliciousVault` reaches `execute_entry_point`, which calls `dict_read` on `contract_class_changes` to get the compiled class hash, then calls `find_element` — which traps because no compiled class fact exists for `0xDEAD`.
6. All deposited funds are permanently frozen with no recovery path at the protocol level.

The StarkNet OS's `execute_replace_class` function is the necessary vulnerable step: it is the OS that accepts and commits the invalid state transition without enforcing the invariant that class hashes must be declared. [5](#0-4)
