### Title
`l2_sync_task` marks rejected L1Handler txs as Committed during catch-up sync — (`crates/apollo_l1_provider/src/catchupper.rs`)

---

### Summary

During the L1 provider catch-up flow, `l2_sync_task` reads `SyncBlock` from `StateSync::get_block`, which includes **all** L1Handler transaction hashes from storage — including those whose execution was reverted/rejected. The task then passes all of them as `committed_txs` with an empty `rejected_txs`, causing the `TransactionManager` to mark rejected L1Handler txs as `Committed` instead of `Rejected`. This permanently prevents those txs from being re-proposed, blocking the corresponding L1→L2 messages forever.

---

### Finding Description

**Step 1 — `StateSync::get_block` includes rejected L1Handler txs**

`StateSync::get_block` reads the block body from storage via `get_block_transactions_with_hash` and classifies every `Transaction::L1Handler` into `l1_transaction_hashes` with no distinction between committed and rejected: [1](#0-0) 

Rejected (reverted) L1Handler txs are stored in the block body and are returned by `get_block_transactions_with_hash`, so they appear in `SyncBlock.l1_transaction_hashes`.

**Step 2 — `l2_sync_task` hardcodes empty rejected set**

`l2_sync_task` passes all `l1_transaction_hashes` as `committed_txs` and unconditionally sets `l1_handler_rejected_tx_hashes = Default::default()`: [2](#0-1) 

The comment `// No rejected txs in sync blocks` is an incorrect assumption — `StateSync::get_block` does not filter them out.

**Step 3 — `apply_commit_block` marks all consumed txs as Committed**

With an empty `rejected_txs`, the partition in `apply_commit_block` puts every hash into `committed_txs`: [3](#0-2) 

**Step 4 — `commit_txs` sets state to `Committed`**

`TransactionManager::commit_txs` calls `mark_committed()` on every hash in `committed_txs`: [4](#0-3) 

**Step 5 — `validate_tx` returns `AlreadyIncludedOnL2` for the now-Committed rejected tx** [5](#0-4) 

A rejected tx that should be `Rejected` (and therefore `is_validatable() == true`) is now `Committed`, so any future `validate` call returns `AlreadyIncludedOnL2`.

**Additional compounding bug**: `accept_commit_while_catching_up` also drops `rejected_txs` from the batcher's backlog commits (there is even a `TODO` acknowledging this): [6](#0-5) [7](#0-6) 

---

### Impact Explanation

- Rejected L1Handler txs are permanently set to `TransactionState::Committed` in the `TransactionManager`.
- `validate` returns `InvalidValidationStatus::AlreadyIncludedOnL2` for those txs, causing the validator node to reject the block.
- The tx is removed from the `proposable_index` and can never be re-proposed.
- The corresponding L1→L2 message is permanently blocked with no retry path.

This falls under **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing**: a valid, retryable L1Handler tx is permanently treated as already committed.

---

### Likelihood Explanation

The catch-up path is triggered on every node restart where the L1 provider is behind the batcher. Any block in history that contained a reverted L1Handler tx will trigger the corruption. This is not a rare edge case — L1Handler execution failures are a normal operational event.

---

### Recommendation

1. **`StateSync::get_block`** should either (a) not include rejected L1Handler txs in `l1_transaction_hashes`, or (b) add a separate `l1_handler_rejected_tx_hashes` field to `SyncBlock`.
2. **`l2_sync_task`** must not assume `l1_handler_rejected_tx_hashes` is always empty. If `SyncBlock` gains a rejected field, pass it through; otherwise, derive it from receipts.
3. **`accept_commit_while_catching_up`** must not silently drop `rejected_txs` from the batcher's `commit_block` call (resolve the existing `TODO`).

---

### Proof of Concept

```
1. Start L1 provider behind batcher (triggers catch-up).
2. Block N contains L1Handler tx H that was rejected (reverted execution).
3. StateSync::get_block(N) returns SyncBlock { l1_transaction_hashes: [H], ... }.
4. l2_sync_task calls commit_block([H], [], N).
5. apply_commit_block([H], []) → commit_txs([H], []) → records[H].mark_committed().
6. Assert: records[H].state == TransactionState::Committed  // BUG: should be Rejected
7. l1_provider.validate(H, N+1) → AlreadyIncludedOnL2      // BUG: should be Validated
```

    fn apply_commit_block(
        &mut self,
        consumed_txs: IndexSet<TransactionHash>,
        rejected_txs: IndexSet<TransactionHash>,
    ) {
        debug!("Applying commit_block to height: {}", self.current_height);
        let (rejected_and_consumed, committed_txs): (Vec<_>, Vec<_>) =
            consumed_txs.iter().copied().partition(|tx| rejected_txs.contains(tx));
        self.tx_manager.commit_txs(&committed_txs, &rejected_and_consumed);

        self.current_height = self.current_height.unchecked_next();
    }

    pub fn commit_txs(
        &mut self,
        committed_txs: &[TransactionHash],
        rejected_txs: &[TransactionHash],
    ) {
        self.rollback_staging();

        for &tx_hash in committed_txs {
            self.create_record_if_not_exist(tx_hash);
            self.with_record(tx_hash, |r| r.mark_committed()).unwrap();
        }
        for &tx_hash in rejected_txs {
            self.with_record(tx_hash, |r| r.mark_rejected()).expect(
                "Storage inconsistency: a transaction sent to the batcher was removed \
                 unexpectedly.",
            );
        }
    }
