### Title
Missing Declared Class Hash Validation in `execute_replace_class` Allows Permanent Contract Fund Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash supplied by the caller has been declared (i.e., exists in `contract_class_changes`). This is an exact structural analog to the reported staking bug: a finalization step that mutates critical state without re-validating a prerequisite condition. A contract deployer can exploit this to permanently freeze any funds held by a contract by replacing its class with an undeclared hash, rendering the contract permanently unexecutable.

---

### Finding Description

In `execute_replace_class` (lines 878–916 of `syscall_impls.cairo`), the OS accepts the caller-supplied `class_hash` from the syscall request and immediately writes it into `contract_state_changes` without checking whether that hash exists in `contract_class_changes` (the set of declared classes):

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The developer-acknowledged TODO at line 898 confirms the missing invariant enforcement. Compare this to `execute_declare_transaction`, which uses `prev_value=0` to enforce that a class can only be declared once and that the compiled class hash is non-zero — no equivalent guard exists in `execute_replace_class`.

The structural parallel to the reported staking bug:

| Staking Bug | StarkNet OS Analog |
|---|---|
| `requestStake()` records token | Contract deployed with valid class hash |
| Token removed from `acceptedTokens` | Class hash never declared (or declared class later irrelevant) |
| `finalizeStake()` skips re-validation | `execute_replace_class` skips declared-class check |
| User tokens locked | Contract funds permanently frozen | [1](#0-0) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once `execute_replace_class` writes an undeclared class hash into a contract's `StateEntry`, every subsequent call to that contract will attempt to look up the class in `contract_class_changes` and fail to find it. The contract becomes permanently unexecutable. Any ERC-20 tokens, ETH, or other assets held in that contract's storage are irrecoverably locked — there is no upgrade path because the class that would implement an upgrade no longer exists.

This is not a revert: the state transition is committed to the proven block output. The L1 verifier accepts the proof (since the OS itself accepted the transition), and the corrupted state becomes canonical. [2](#0-1) 

---

### Likelihood Explanation

**Medium.**

The attack surface is reachable by any contract deployer — a listed unprivileged entry path. A deployer can:

1. Deploy a contract that appears legitimate (e.g., a vault or multisig).
2. Attract user deposits.
3. Invoke `replace_class` with a felt value that was never passed through `execute_declare_transaction`.

The OS will accept the syscall, produce a valid proof, and the contract's funds are permanently frozen. No sequencer compromise is required; the attacker only needs to control a contract that calls the `replace_class` syscall. The TODO comment confirms the development team is aware the check is absent, meaning the window of exposure is open until the fix is shipped. [3](#0-2) 

---

### Recommendation

In `execute_replace_class`, after reading `class_hash` from the request, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the returned compiled class hash is non-zero before proceeding with the `dict_update` on `contract_state_changes`. This mirrors the invariant already enforced in `execute_declare_transaction` via `prev_value=0` and `assert_not_zero(compiled_class_hash)`. [4](#0-3) 

---

### Proof of Concept

1. Attacker deploys `VaultContract` with a valid declared class hash `C_valid`. Users deposit funds.
2. Attacker crafts a transaction that calls `replace_class(class_hash=0xdeadbeef)` where `0xdeadbeef` has never been passed through `execute_declare_transaction` and therefore has no entry in `contract_class_changes`.
3. The OS executes `execute_replace_class`:
   - Reads `class_hash = 0xdeadbeef` from the syscall request.
   - Skips the declared-class check (line 898 TODO).
   - Writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
4. The block is proven and accepted by L1. `VaultContract`'s canonical class hash is now `0xdeadbeef`.
5. Any subsequent `call_contract` or `invoke` targeting `VaultContract` causes the OS to look up `0xdeadbeef` in `contract_class_changes`, find nothing, and fail.
6. All user funds in `VaultContract` are permanently frozen with no recovery path. [5](#0-4)
