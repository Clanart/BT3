### Title
`execute_l1_handler_transaction` Does Not Consume L1ŌåÆL2 Message When Handler Is Pre-Reverted, Enabling Message Replay ŌĆö (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_l1_handler_transaction` checks `is_reverted` before any execution setup and returns early when it is `TRUE`, without ever calling `consume_l1_to_l2_message`. This means the L1ŌåÆL2 message is never written to `outputs.messages_to_l2` for pre-reverted L1 handlers. The L1 StarkNet core contract uses that output to mark messages as consumed; if a message is absent, it remains in the pending-messages set and can be replayed in a future block.

---

### Finding Description

In `execute_l1_handler_transaction`, the very first conditional after `%{ IsReverted %}` is:

```cairo
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();          // ŌåÉ returns here; consume_l1_to_l2_message is never reached
}
``` [1](#0-0) 

All the execution-context setup, nonce loading, and the critical call to `consume_l1_to_l2_message` appear **after** this early return:

```cairo
// Consume L1-to-L2 message.
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
``` [2](#0-1) 

`consume_l1_to_l2_message` is the sole function that writes the L1ŌåÆL2 message header and payload into `outputs.messages_to_l2`:

```cairo
assert [outputs.messages_to_l2] = MessageToL2Header(...);
...
let (outputs) = os_carried_outputs_new(
    messages_to_l1=outputs.messages_to_l1,
    messages_to_l2=outputs.messages_to_l2 + MessageToL2Header.SIZE + ...,
);
``` [3](#0-2) 

The TODO comment in the same function confirms the reverted-L1-handler path is intentionally incomplete:

> "TODO(Yoni): currently, the contract state is not fetched for reverted L1 handlers." [4](#0-3) 

This is structurally identical to the reported `PledgeManager::refundTokens` bug: a critical accounting update (`tokensSold--` / `consume_l1_to_l2_message`) is placed inside only one branch of a conditional and is silently skipped in the other branch.

---

### Impact Explanation

`outputs.messages_to_l2` is the OS-level output that the L1 StarkNet core contract reads to determine which L1ŌåÆL2 messages were consumed in a block. A message absent from this output is never removed from the L1 contract's pending-messages set. An attacker who can cause an L1 handler to be pre-reverted can therefore keep the message alive on L1 and replay it in a subsequent block ŌĆö effectively double-spending any value (e.g., bridged ETH or ERC-20 tokens) that the original L1ŌåÆL2 message was meant to transfer.

**Impact class:** Critical ŌĆö Direct loss of funds.

---

### Likelihood Explanation

Any unprivileged L1/L2 message sender can trigger this path:

1. Deploy a StarkNet contract whose L1 handler always reverts (or deploy to an address with no handler for the given selector).
2. Send an L1ŌåÆL2 message (e.g., a bridge deposit) targeting that contract.
3. The sequencer, correctly, sets `is_reverted = TRUE` for the handler and the OS skips `consume_l1_to_l2_message`.
4. The message is absent from the OS output; the L1 contract does not mark it consumed.
5. The attacker upgrades the contract (via `replace_class`) to a version whose L1 handler succeeds.
6. The attacker replays the same L1ŌåÆL2 message in a future block; the handler now mints/transfers funds a second time.

No malicious sequencer behavior is required ŌĆö the sequencer is acting correctly by marking a reverting handler as reverted.

---

### Recommendation

Move the execution-context setup and `consume_l1_to_l2_message` call **before** the `is_reverted` guard, so the message is always written to `outputs.messages_to_l2` regardless of execution outcome. Concretely:

```diff
 func execute_l1_handler_transaction{...}(block_context: BlockContext*) {
     alloc_locals;
 
     %{ StartTx %}
+    // Set up execution context and consume the L1ŌåÆL2 message unconditionally.
+    local entry_point_selector;
+    %{ TxEntryPointSelector %}
+    let (local tx_execution_context: ExecutionContext*) = get_invoke_tx_execution_context(...);
+    ...
+    local nonce;
+    %{ LoadTxNonceL1Handler %}
+    ...
+    consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
+
     local is_reverted;
     %{ IsReverted %}
     if (is_reverted != FALSE) {
         %{ EndTx %}
         return ();
     }
-    ...
-    consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
     ...
 }
```

### Proof of Concept

1. **Attacker deploys** `VictimContract` on StarkNet with an `@l1_handler` that unconditionally panics.
2. **Attacker sends** an L1ŌåÆL2 message from the ETH bridge to `VictimContract`, depositing 1 ETH. The L1 bridge contract records the message hash as pending.
3. **Sequencer** includes the L1 handler transaction in a block. `%{ IsReverted %}` returns `TRUE` (handler would revert). The OS executes the early-return path; `consume_l1_to_l2_message` is never called. The OS output `messages_to_l2` does **not** contain this message.
4. **L1 core contract** processes the block proof. Because the message is absent from `messages_to_l2`, it does **not** remove the message hash from the pending set. The 1 ETH deposit is still "pending" on L1.
5. **Attacker calls** `replace_class` on `VictimContract`, swapping in a class whose `@l1_handler` mints 1 ETH worth of L2 tokens to the attacker.
6. **Attacker replays** the original L1ŌåÆL2 message in a new block. The handler now succeeds; `consume_l1_to_l2_message` is called; the attacker receives 1 ETH worth of L2 tokens a second time.
7. **Net result:** attacker has received 2├Ś the tokens for a single 1 ETH L1 deposit ŌĆö direct loss of funds from the bridge.
