### Title
Integer Wrapping in Relative Timelock Arithmetic Enables Timelock Bypass and Consensus Divergence — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` performs relative timelock arithmetic using either `wrapping_add` or `saturating_add` depending on the caller-supplied `nowrap: bool` flag. When `nowrap = false`, valid full-range `u64` timelock values cause the intermediate sum to wrap around, producing two distinct incorrect outcomes: (1) a spend with `seconds_relative = u64::MAX` bypasses the timelock entirely (accepted when it must be rejected), and (2) a spend with a large `before_seconds_relative` value is incorrectly rejected (rejected when it must be accepted). Both outcomes are reachable from an unprivileged spend bundle and produce results that diverge from nodes running with `nowrap = true`.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` evaluates four relative timelock conditions by adding a coin's birth height/timestamp to the condition value and comparing the result against the current block height/timestamp. [1](#0-0) 

When `nowrap = false`, the addition uses `wrapping_add`: [2](#0-1) 

When `nowrap = true`, it uses `saturating_add`: [3](#0-2) 

The same dual-path exists for `before_seconds_relative`, `height_relative`, and `before_height_relative`: [4](#0-3) 

The `nowrap` flag is a caller-supplied parameter exposed directly to Python via `py_check_time_locks`: [5](#0-4) 

The Python type stub confirms the parameter is part of the public API: [6](#0-5) 

**Case 1 — Timelock bypass (`seconds_relative`):**

With `coin_timestamp = 1000`, `seconds_relative = u64::MAX`, `nowrap = false`:
```
1000_u64.wrapping_add(u64::MAX) = 999
timestamp (2000) < 999  →  false  →  Ok (timelock passes)
```
The spend is accepted even though the timelock demands the coin cannot be spent for ~584 billion years. The test suite documents this exact outcome: [7](#0-6) 

**Case 2 — False rejection (`before_seconds_relative`):**

With `coin_timestamp = 10000`, `before_seconds_relative = 0xFFFF_FFFF_FFFF_D8F0`, `nowrap = false`:
```
10000_u64.wrapping_add(0xFFFF_FFFF_FFFF_D8F0) = 0
timestamp (10150) >= 0  →  true  →  Err (spend rejected)
```
The spend is rejected even though the deadline is far in the future. The test suite documents this: [8](#0-7) 

The Python-level test suite confirms both divergence directions across the full parameter space: [9](#0-8) 

The `seconds_relative` and `before_seconds_relative` values are parsed from CLVM atoms via `sanitize_uint` with `max_size = 8`, accepting the full `u64` range: [10](#0-9) [11](#0-10) 

No upper-bound restriction below `u64::MAX` is imposed on these values before they reach `check_time_locks`. The function accepts any `u64` as a valid input, but the wrapping arithmetic silently produces incorrect results for values near `u64::MAX`.

---

### Impact Explanation

**Timelock bypass (High):** With `nowrap = false`, a spend bundle containing `ASSERT_SECONDS_RELATIVE 0xFFFFFFFFFFFFFFFF` is accepted by the consensus engine regardless of how recently the coin was created. This bypasses the relative timelock condition entirely, enabling unauthorized spend acceptance of coins that should be time-locked.

**Consensus divergence (High):** A node running `check_time_locks` with `nowrap = false` and a node running with `nowrap = true` will reach opposite conclusions about the validity of the same spend bundle when large relative timelock values are present. This is a deterministic, input-triggered consensus split on a consensus-critical validation path (`run_block_generator` → `parse_spends` → `check_time_locks`).

Both impacts match the allowed High impact scope: timelock validation bypass enabling unauthorized spend acceptance, and consensus nondeterminism from cross-node disagreement on a serialized protocol object.

---

### Likelihood Explanation

The `nowrap` parameter is part of the public Python API and is passed by the Chia full node at call sites not visible in this repository. The existence of both `nowrap = true` and `nowrap = false` test paths, and the explicit exposure of the flag to Python callers, indicates that `nowrap = false` is used in at least one production code path (likely for backward-compatible validation of pre-fork blocks). An unprivileged attacker can craft a spend bundle with `seconds_relative = u64::MAX` encoded as a 9-byte CLVM atom `0x00FFFFFFFFFFFFFFFF`, which passes `sanitize_uint` validation and reaches `check_time_locks` with a value that triggers the wrapping path.

---

### Recommendation

- Replace `wrapping_add` with `saturating_add` unconditionally in all four relative timelock checks. The `nowrap = false` path provides no correct semantic — wrapping arithmetic on timelock sums has no valid consensus interpretation.
- If backward compatibility requires the old behavior for historical block validation, gate it behind a block-height check rather than a caller-supplied boolean, and document the exact height at which the safe path becomes mandatory.
- Add property-based tests (analogous to the Echidna approach in the external report) that assert: for all valid `u64` values of `seconds_relative` and `before_seconds_relative`, `check_time_locks` with `nowrap = true` and `nowrap = false` agree when no overflow occurs, and that `nowrap = true` always produces the semantically correct result.

### Proof of Concept

```python
from chia_rs import check_time_locks, SpendBundleConditions, SpendConditions, CoinRecord, Coin
from chia.types.blockchain_format.sized_bytes import bytes32
from chia.util.ints import uint32, uint64

coin_id = bytes32(b'\x03' * 32)
coin_record = CoinRecord(
    Coin(bytes32(b'\x00'*32), bytes32(b'\x00'*32), 1),
    confirmed_block_index=uint32(1000),
    spent_block_index=uint32(0),
    coinbase=False,
    timestamp=uint64(1000),
)

# Craft a spend with seconds_relative = u64::MAX
spend = SpendConditions(
    coin_id=coin_id,
    seconds_relative=uint64(0xFFFFFFFFFFFFFFFF),
    # ... other fields default
)
bundle = SpendBundleConditions(spends=[spend], ...)

# nowrap=False (wrapping mode): timelock bypassed, spend accepted
result_wrap = check_time_locks({coin_id: coin_record}, bundle, 2000, 2000, False)
assert result_wrap is None  # Ok — timelock bypassed

# nowrap=True (saturating mode): timelock correctly enforced, spend rejected
result_nowrap = check_time_locks({coin_id: coin_record}, bundle, 2000, 2000, True)
assert result_nowrap == 105  # AssertSecondsRelativeFailed
```

The two calls to the same function with the same spend bundle produce opposite validity verdicts, demonstrating both the timelock bypass and the consensus divergence.
