### Title
Constructor Failure Silently Swallowed in `execute_deploy` Syscall — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary
The `execute_deploy` syscall handler in the StarkNet OS unconditionally writes `failure_flag=0` in the response header regardless of whether the deployed contract's constructor actually succeeded or failed. A constructor revert is silently absorbed while the caller contract receives a success response, creating a state inconsistency that can permanently freeze funds sent to the "deployed" address.

### Finding Description

In `syscall_impls.cairo`, the `execute_deploy` function handles the `deploy` syscall. After calling `deploy_contract` to run the constructor, it writes the response header with a hardcoded `failure_flag=0`: [1](#0-0) 

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
```

The `failure_flag` field of `ResponseHeader` is the mechanism by which the OS communicates syscall failure back to the calling contract. For `call_contract` and `library_call`, this flag is correctly set based on `is_reverted`: [2](#0-1) 

```cairo
with_attr error_message("Predicted gas costs are inconsistent with the actual execution.") {
    assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
}
```

For `execute_deploy`, no such propagation exists. The `deploy_contract` function accepts `revert_log` as an implicit argument (as evidenced by its use in `execute_deploy_account_transaction` where a fresh `revert_log` is explicitly initialized and passed): [3](#0-2) 

```cairo
let revert_log = init_revert_log();
deploy_contract{revert_log=revert_log}(
    block_context=block_context, constructor_execution_context=constructor_execution_context
);
```

This confirms `deploy_contract` is designed to handle constructor reverts via the revert log mechanism. However, in `execute_deploy`, the constructor's revert outcome is never read and `failure_flag` is always written as `0`.

The asymmetry is structurally identical to the reported M-02 pattern: the external call (`deploy_contract` / constructor) can fail, but the failure path is silently discarded and the caller always receives a success response.

### Impact Explanation

**Permanent freezing of funds.**

A caller contract that invokes the `deploy` syscall receives `failure_flag=0` and a valid `contract_address` in the response even when the constructor reverted. The caller has no way to distinguish this from a genuine successful deployment. If the caller subsequently transfers tokens to the returned `contract_address` (a common post-deploy pattern, e.g., funding a newly deployed vault or escrow), those tokens are sent to an address whose class hash was reverted to zero (no code). No entry point exists at that address to recover the funds. The tokens are permanently frozen.

### Likelihood Explanation

Any unprivileged user can:
1. Declare a contract class whose constructor conditionally reverts based on calldata.
2. Induce a victim contract (e.g., a factory) to call `deploy` with calldata that triggers the revert.
3. The victim contract receives `failure_flag=0`, believes deployment succeeded, and proceeds to fund the address.

The attacker controls the constructor logic via the declared class. No privileged access is required. The `deploy` syscall is a standard, publicly accessible syscall available to all Cairo 1 contracts.

### Recommendation

Propagate the constructor's revert status to the response header. The `deploy_contract` function should return an `is_reverted` flag (analogous to `execute_entry_point`'s return value), and `execute_deploy` should write:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

Until constructor failure propagation is fully implemented (as noted by the existing TODO), the `deploy` syscall should at minimum panic on constructor revert (using `non_reverting_select_execute_entry_point_func` semantics internally) rather than silently reporting success.

### Proof of Concept

1. Declare class `MaliciousContract` whose constructor checks `calldata[0]` and reverts if it equals `1`.
2. Deploy a victim factory contract with logic: `addr = deploy(MaliciousContract, [1]); transfer(token, addr, amount)`.
3. Invoke the factory. The OS executes `execute_deploy`:
   - Constructor runs with `[1]`, reverts.
   - `deploy_contract` processes the revert via `revert_log` (class_hash assignment undone).
   - `execute_deploy` writes `ResponseHeader(gas=..., failure_flag=0)` — hardcoded.
4. The factory reads `failure_flag=0`, reads the returned `contract_address`, and calls `transfer(token, contract_address, amount)`.
5. The transfer succeeds (the token contract has no knowledge of the deploy failure).
6. `contract_address` has `class_hash=0` (reverted). No `__execute__` or withdrawal function exists. Funds are permanently frozen. [4](#0-3)
