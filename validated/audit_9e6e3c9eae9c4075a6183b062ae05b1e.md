### Title
`execute_replace_class` Does Not Verify the New Class Hash Is Declared, Enabling Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary `class_hash` value from the syscall request and writes it directly into the contract's state entry, without verifying that the hash corresponds to a previously declared contract class. This is the direct analog of the CLGauge bug: just as `CLGauge.deposit()` failed to verify the staked NFT belonged to the correct pool, `execute_replace_class` fails to verify the replacement class hash belongs to the declared class registry. A contract that replaces its class with an undeclared hash becomes permanently unexecutable, permanently freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` (lines 877–916) reads `class_hash` from the syscall request and immediately writes it into `contract_state_changes` with no membership check:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The developer-acknowledged TODO at line 898 confirms the check is intentionally deferred but not yet implemented.

When any future transaction subsequently calls an entry point on the affected contract, `execute_entry_point` (in `execute_entry_point.cairo`, line 154) performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
```

Because the replaced class hash was never declared, `dict_read` returns the default value `0`. The OS then calls:

```cairo
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,   // = 0
);
```

`find_element` with `key=0` panics if no compiled class with hash `0` exists in the bundle. Every subsequent call to the contract fails at the OS level. The contract is permanently unexecutable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared value, the contract can never be called again. All ERC-20 tokens, ETH, or other assets held in the contract's storage are permanently inaccessible. There is no recovery path: the state commitment has been updated with the invalid class hash, and no future transaction can execute the contract's entry points to withdraw funds.

---

### Likelihood Explanation

**Medium.**

The most realistic attack path targets upgradeable contracts whose upgrade mechanism accepts a caller-supplied class hash and passes it directly to `replace_class`. Such patterns are common in DeFi protocols (proxy/upgrade patterns). An attacker who can participate in governance, is an admin, or can exploit a reentrancy or input-validation bug in the upgrade function can supply an undeclared class hash. Because the OS performs no validation, the contract-level check is the only guard — and many contracts will not implement it, trusting the OS to enforce the invariant (as the TODO comment implies was the intended design).

---

### Recommendation

In `execute_replace_class`, before writing the new class hash into `contract_state_changes`, verify that `class_hash` is present in `contract_class_changes` (i.e., it was registered by a prior Declare transaction). Concretely:

```cairo
// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the fix described in the CLGauge report: compute/verify the identity of the supplied parameter against the authoritative registry before accepting it.

---

### Proof of Concept

1. Attacker deploys contract `C` that holds user funds. `C` exposes an `upgrade(new_class_hash)` function that calls the `replace_class` syscall with the supplied hash.
2. Attacker calls `upgrade(0xdeadbeef)` where `0xdeadbeef` is any value not present in `contract_class_changes`.
3. The OS executes `execute_replace_class`: gas is deducted, `contract_state_changes[C].class_hash` is set to `0xdeadbeef`. No error is raised. The transaction succeeds and is committed to the block.
4. In a subsequent block, any user calls any entry point on `C`.
5. `execute_entry_point` calls `dict_read(key=0xdeadbeef)` on `contract_class_changes` → returns `0`.
6. `find_element(key=0)` on `compiled_class_facts_bundle` panics (no compiled class with hash `0` exists).
7. The OS cannot produce a valid execution for any call to `C`. Every transaction targeting `C` is excluded from every block.
8. All funds in `C`'s storage are permanently frozen with no recovery path.
