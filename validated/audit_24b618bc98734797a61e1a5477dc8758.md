### Title
Gateway Proof-Facts Block-Number Check Uses `latest+1` Context While State Reader Lags One Block, Causing False Rejection of Valid Client-Side-Proving Transactions - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful validator increments the block number by one when building the `BlockContext` for proof-facts validation, but the underlying state reader still reflects the latest *committed* block. This creates a one-block gap: the block-number range check allows `proof_block_number` up to `latest_block − STORED_BLOCK_HASH_BUFFER + 1`, but the block-hash contract in the current state only contains hashes up to `latest_block − STORED_BLOCK_HASH_BUFFER`. Any Invoke V3 transaction whose `proof_block_number` equals exactly `latest_block − STORED_BLOCK_HASH_BUFFER + 1` passes the range check yet fails the hash lookup (returns `Felt::ZERO`), causing the gateway to reject a transaction that would be valid for inclusion in the very next block.

### Finding Description

**Root cause — `run_validate_entry_point`**

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  line 296-297
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();   // ← N → N+1
```

The `block_info` is fetched from the latest committed block (height N) and then its `block_number` is bumped to N+1 to simulate the pending block. The same `block_context` is passed to `AccountTransaction::validate_proof_facts`. [1](#0-0) 

**Range check — `validate_proof_block_number`**

```rust
// crates/blockifier/src/transaction/account_transaction.rs  line 244-257
let max_allowed = current_block_number.0.checked_sub(STORED_BLOCK_HASH_BUFFER)...;
if proof_block_number > max_allowed { return Err(...) }
```

With `current_block_number = N+1`, `max_allowed = N+1 − STORED_BLOCK_HASH_BUFFER`. A proof referencing block `N − STORED_BLOCK_HASH_BUFFER + 1` satisfies `proof_block_number ≤ max_allowed` and **passes**. [2](#0-1) 

**Hash lookup — `validate_proof_block_hash`**

```rust
// crates/blockifier/src/transaction/account_transaction.rs  line 278-286
let stored_block_hash = state
    .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;
if stored_block_hash != proof_block_hash { return Err(...) }
```

The `state` here is the `CachedState` built from the latest committed block (height N). The Starknet OS writes the hash of block `current − STORED_BLOCK_HASH_BUFFER` to the block-hash contract **during** execution of block `current`. Therefore, after block N is committed, the contract holds hashes for blocks `0 … N − STORED_BLOCK_HASH_BUFFER`. The hash of block `N − STORED_BLOCK_HASH_BUFFER + 1` is **not yet written**; `get_storage_at` returns `Felt::ZERO`, and the transaction is rejected with `"Proof block hash is zero"`. [3](#0-2) 

**Why the transaction is valid for block N+1**

The OS writes the hash of block `N+1 − STORED_BLOCK_HASH_BUFFER = N − STORED_BLOCK_HASH_BUFFER + 1` at the *start* of block N+1's execution (hint `WriteOldBlockToStorage`), before any transaction's `check_proof_facts` runs. So the same transaction would pass both checks during actual sequencing of block N+1. [4](#0-3) 

**Structural analog to M-16**

In M-16, a *global* health check (aggregated across all pools) uses Pool A's LTV to make a position unhealthy, enabling liquidation of Pool B even though Pool B's own risk parameters are satisfied. Here, a *global* block-number range check (using the simulated N+1 context) passes, but the *specific* block-hash lookup (using the actual N-state) fails — the two sub-checks operate on inconsistent "contexts", causing a valid transaction to be incorrectly rejected.

### Impact Explanation

Any user submitting a client-side-proving Invoke V3 transaction whose `proof_block_number` equals `latest_block − STORED_BLOCK_HASH_BUFFER + 1` will have their transaction rejected at the gateway with `InvalidProofFacts / "block hash is zero"`, even though the transaction would be accepted and execute correctly if included in the very next block. This is a **false rejection of a valid transaction before sequencing**.

Impact category: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

### Likelihood Explanation

- The trigger is unprivileged: any user generating a client-side proof over the most-recently-eligible block hits this window.
- The window is exactly one block wide (`proof_block_number = latest − STORED_BLOCK_HASH_BUFFER + 1`), so it is a narrow but deterministic and reproducible condition.
- As client-side proving adoption grows, the probability of hitting this boundary increases.

### Recommendation

In `run_validate_entry_point`, do **not** increment `block_info.block_number` before building the `BlockContext` used for proof-facts validation, or alternatively cap `max_allowed` in `validate_proof_block_number` to `current_block_number − STORED_BLOCK_HASH_BUFFER − 1` (i.e., use the same conservative bound that the current committed state actually satisfies). The block-number increment is appropriate for fee/nonce simulation but must not widen the proof-facts block-hash window beyond what the state reader can verify. [5](#0-4) 

### Proof of Concept

Let `STORED_BLOCK_HASH_BUFFER = B` and the latest committed block be `N`.

1. User generates a virtual-SNOS proof over block `N − B + 1` and embeds it in an Invoke V3 transaction as `proof_facts`.
2. User submits the transaction to the gateway.
3. Gateway calls `run_validate_entry_point`:
   - `block_info.block_number` is set to `N+1`.
   - `validate_proof_block_number(N − B + 1, N+1)` → `max_allowed = N+1 − B = N − B + 1` → **passes**.
   - `validate_proof_block_hash(proof_hash, N − B + 1, state_N)` → `state_N.get_storage_at(block_hash_contract, N − B + 1)` returns `Felt::ZERO` → **rejected: "block hash is zero"**.
4. The transaction is dropped. The user resubmits after block `N+1` is committed; the hash is now stored; the transaction passes.

The transaction was valid for block `N+1` but was incorrectly rejected at the gateway.

    fn validate_proof_block_number(
        proof_block_number: u64,
        current_block_number: BlockNumber,
    ) -> TransactionPreValidationResult<()> {
        // Proof block must be old enough to have a stored block hash.
        // Stored block hashes are guaranteed only up to: current - STORED_BLOCK_HASH_BUFFER.
        let max_allowed =
            current_block_number.0.checked_sub(STORED_BLOCK_HASH_BUFFER).ok_or_else(|| {
                TransactionPreValidationError::InvalidProofFacts(format!(
                    "The current block number {current_block_number} is below the required \
                     block-hash retention buffer: {STORED_BLOCK_HASH_BUFFER}."
                ))
            })?;

        if proof_block_number > max_allowed {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "The proof block number {proof_block_number} is too recent. The maximum allowed \
                 block number is {max_allowed}."
            )));
        }

        Ok(())
    }

```rust
    fn validate_proof_block_hash(
        proof_block_hash: Felt,
        proof_block_number: u64,
        os_constants: &OsConstants,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        if proof_block_hash == Felt::ZERO {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof block hash is zero for block {proof_block_number}."
            )));
        }

        // Compare the proof's block hash with the stored block hash.
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();

        let stored_block_hash = state
            .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

        if stored_block_hash != proof_block_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Block hash mismatch for block {proof_block_number}. Proof block hash: \
                 {proof_block_hash}, stored block hash: {stored_block_hash}."
            )));
        }

        Ok(())
    }
```
