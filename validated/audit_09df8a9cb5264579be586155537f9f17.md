### Title
Missing shard_id cross-check in `set_state_header` allows malicious peer to install wrong `ChunkExtra.state_root` for a tracked shard — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` validates that the peer-supplied chunk is included in the block's `chunk_headers_root` Merkle tree, but never verifies that the chunk belongs to the **requested** shard. A malicious peer can supply shard 1's chunk (with a valid Merkle proof for shard 1's position) in response to a shard 0 state-sync header request. All validation checks pass. The header is stored under key `StateHeaderKey(shard_id=0, sync_hash)`. When `set_state_finalize` is subsequently called, shard 1's chunk is applied under shard 0's `ShardUId`, and the resulting `ChunkExtra` — with a `state_root` derived from shard 1's transactions and receipts — is permanently written to `DBCol::ChunkExtra` for `(block_hash, shard_uid_0)`.

### Finding Description

**Validation gap in `set_state_header`:** [1](#0-0) 

The `validate_chunk_proofs` call checks only internal chunk consistency (hash, tx root, receipts root): [2](#0-1) 

The `verify_path` call proves the chunk hash is *somewhere* in the block's `chunk_headers_root` Merkle tree, but the tree covers all shards. A chunk from shard 1 with its own valid Merkle proof (at position 1) passes this check when shard 0 is requested. There is no check that the chunk occupies the correct shard-index position.

The `chunk_headers_root` is built by merklizing `ChunkHashHeight` values for all shards in order: [3](#0-2) 

A malicious peer can supply shard 1's chunk with shard 1's Merkle proof; `verify_path` returns `true` because shard 1's chunk IS in the tree.

**No shard_id guard in `ChainUpdate::set_state_finalize`:** [4](#0-3) 

`shard_uid` is derived from the **requested** `shard_id` (line 513-514), but `chunk_header.prev_state_root()` and all chunk fields come from the **peer-supplied** chunk (which belongs to shard 1). `apply_chunk` is called with shard 0's `shard_uid` but shard 1's `prev_state_root`, transactions, and receipts.

**Corrupted write:** [5](#0-4) 

`set_chunk_extra` writes the `ChunkExtra` (whose `state_root` = `apply_result.new_root`, derived from shard 1's execution) under `(block_hash, shard_uid_0)` in `DBCol::ChunkExtra`.

### Impact Explanation

The exact corrupted value is `ChunkExtra.state_root` written to `DBCol::ChunkExtra` for key `(block_hash, shard_uid_0)`. It equals the post-execution state root of shard 1's transactions/receipts applied to shard 1's trie, instead of shard 0's canonical state root. Every subsequent block processed by the node uses this wrong root as the starting state for shard 0, causing permanent divergence from the canonical chain for all blocks after the sync point.

### Likelihood Explanation

State sync is triggered whenever a node falls behind. Any peer reachable by the syncing node can be selected as the state-sync source. The attack requires:
1. Running a NEAR node (unprivileged)
2. Becoming a peer of the syncing node
3. Serving a `ShardStateSyncResponseHeader` for shard 0 that contains shard 1's chunk with shard 1's valid Merkle proof, plus correct incoming receipts for shard 0, plus state parts for shard 1

All three steps are feasible for an unprivileged network participant who has access to the canonical block data.

### Recommendation

In `set_state_header`, after extracting the chunk, add an explicit shard_id cross-check before any other validation:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        format!("set_shard_state failed: chunk shard_id {:?} does not match requested shard_id {:?}",
                chunk.shard_id(), shard_id).into()
    ));
}
```

Similarly, add the same guard at the top of `ChainUpdate::set_state_finalize` as a defense-in-depth check.

### Proof of Concept

Build a state-sync test-loop with two shards. In the test, intercept the `ShardStateSyncResponseHeader` for shard 0 and replace its `chunk` field with shard 1's chunk (taken from the same block) along with shard 1's Merkle proof. Call `set_state_header(shard_id=0, sync_hash, tampered_header)` — it returns `Ok(())`. Then apply state parts for shard 1 under `shard_id=0`. Call `chain.set_state_finalize(shard_id=0, sync_hash)`. Assert that the `ChunkExtra` stored for `(block_hash, shard_uid_0)` has a `state_root` equal to shard 0's canonical state root — the assertion will fail, demonstrating the corruption.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
