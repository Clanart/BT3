### Title
Legacy EIP-712 `timeout_height` Field Omission Allows Signature Reuse Across Transactions With Different Expiry — (`ethereum/eip712/eip712_legacy.go`, `ante/cosmos/eip712.go`)

### Summary

The `extractMsgTypes` function in `LegacyWrapTxToTypedData` deliberately omits `timeout_height` from the EIP-712 `Tx` type schema. Because EIP-712 only hashes fields declared in the type schema, any value of `timeout_height` in the message payload is silently ignored during hash computation. As a result, two transactions that are identical except for their `timeout_height` produce the same EIP-712 signable hash, and a signature over one is cryptographically valid for the other.

---

### Finding Description

**Root cause — `ethereum/eip712/eip712_legacy.go`, `extractMsgTypes`:**

The `Tx` type schema is defined without `timeout_height`: [1](#0-0) 

The comment explicitly acknowledges the omission. When `LegacyWrapTxToTypedData` unmarshals the amino sign-bytes JSON into `txData`, the `timeout_height` key is present in the map but is never encoded into the EIP-712 struct hash because it is absent from the type definition. [2](#0-1) 

**Affected verification path 1 — `ante/cosmos/eip712.go`, `VerifySignature`:**

`txBytes` is constructed from the *actual submitted transaction's* `timeout_height` via `legacytx.StdSignBytes`, then passed directly to `LegacyWrapTxToTypedData`. No guard checks whether `timeout_height` differs from what was originally signed. [3](#0-2) 

This path is live in production: `newLegacyCosmosAnteHandlerEip712` installs `LegacyEip712SigVerificationDecorator` for every tx carrying an `ExtensionOptionsWeb3Tx` extension. [4](#0-3) 

**Affected verification path 2 — `crypto/ethsecp256k1/ethsecp256k1.go`, `PubKey.VerifySignature`:**

The fallback `verifySignatureAsEIP712` calls `LegacyGetEIP712BytesForMsg`, which tries the amino decode path (`legacyDecodeAminoSignDoc`) first. That path has no `timeout_height` guard and feeds the sign bytes directly into `LegacyWrapTxToTypedData`. [5](#0-4) 

The protobuf decode path (`legacyDecodeProtobufSignDoc`) *does* reject `TimeoutHeight != 0`, but the amino path — which is the one exercised for legacy EIP-712 transactions — does not. [6](#0-5) 

---

### Impact Explanation

An attacker who observes a signed legacy EIP-712 transaction T1 (with `timeout_height = X`) in the mempool can:

1. Copy T1's msgs, fees, memo, chain-id, account-number, sequence, and signature verbatim.
2. Set `timeout_height = 0` (no expiry) in the new transaction T2.
3. Wait for block X to pass, causing T1 to be evicted from the mempool by `authante.NewTxTimeoutHeightDecorator`.
4. Submit T2. Signature verification passes because the EIP-712 hash is identical to T1's. The timeout-height decorator passes because `timeout_height = 0` means "no expiry."
5. T2 executes the user's messages after the user explicitly intended them to expire.

The inverse is also possible: a transaction signed with `timeout_height = 0` can be re-submitted with an arbitrary `timeout_height`, though this direction is less harmful.

**Scope of mutation:** The attacker **cannot** change `msgs`, `fees`, `memo`, `chain_id`, `account_number`, or `sequence` — those fields are all present in the type schema and are fully protected. Only `timeout_height` is unprotected.

---

### Likelihood Explanation

- The legacy `ExtensionOptionsWeb3Tx` path is still active in production (not removed, only deprecated).
- Any transaction broadcast to the mempool exposes its signature to observers.
- The attack requires the victim to have set a non-zero `timeout_height`, which is an uncommon but valid and documented usage pattern.
- No privileged access, governance, or key compromise is required.

---

### Recommendation

Add `timeout_height` back to the `Tx` type schema in `extractMsgTypes`:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id",       Type: "string"},
    {Name: "fee",            Type: "Fee"},
    {Name: "memo",           Type: "string"},
    {Name: "msgs",           Type: "Msg[]"},
    {Name: "sequence",       Type: "string"},
    {Name: "timeout_height", Type: "string"},  // restore
},
```

`legacytx.StdSignBytes` already serialises `timeout_height` into the amino JSON, so the field will be present in `txData` and will be hashed once the type entry is restored. Additionally, add an explicit guard in `VerifySignature` (analogous to the existing guard in `legacyDecodeProtobufSignDoc`) to reject any submitted transaction whose `timeout_height` differs from what was encoded in the sign bytes.

---

### Proof of Concept

```go
// Both txBytes1 and txBytes2 are produced by legacytx.StdSignBytes with
// identical fields except timeout_height (0 vs 100).
txBytes1 := legacytx.StdSignBytes("chain-1", 1, 0, 0,   fee, msgs, "")
txBytes2 := legacytx.StdSignBytes("chain-1", 1, 0, 100, fee, msgs, "")

// txBytes1 != txBytes2 (different JSON)
// But:
td1, _ := eip712.LegacyWrapTxToTypedData(cdc, chainID, msgs[0], txBytes1, fd)
td2, _ := eip712.LegacyWrapTxToTypedData(cdc, chainID, msgs[0], txBytes2, fd)
hash1, _, _ := apitypes.TypedDataAndHash(td1)
hash2, _, _ := apitypes.TypedDataAndHash(td2)
// hash1 == hash2  →  T1's signature verifies T2
```

The existing test `TestEIP712RejectsTimeoutTimestamp` in `ethereum/eip712/eip712_test.go` (lines 388–471) demonstrates the analogous `timeout_timestamp` fix and confirms the test infrastructure needed to verify this finding. [7](#0-6)

func (pubKey PubKey) verifySignatureAsEIP712(msg, sig []byte) bool {
	eip712Bytes, err := eip712.GetEIP712BytesForMsg(msg)
	if err != nil {
		return false
	}

	if pubKey.verifySignatureECDSA(eip712Bytes, sig) {
		return true
	}

	// Try verifying the signature using the legacy EIP-712 encoding
	legacyEIP712Bytes, err := eip712.LegacyGetEIP712BytesForMsg(msg)
	if err != nil {
		return false
	}

	return pubKey.verifySignatureECDSA(legacyEIP712Bytes, sig)
}

// TestEIP712RejectsTimeoutTimestamp ensures that a SignDoc whose TxBody sets
// timeout_timestamp (ADR-070) is rejected by the EIP-712 fallback conversion.
func (suite *EIP712TestSuite) TestEIP712RejectsTimeoutTimestamp() {
	suite.SetupTest()

	privKey, pubKey := suite.createTestKeyPair()
	signer := sdk.AccAddress(pubKey.Bytes())

	// buildDirectSignBytes builds the SIGN_MODE_DIRECT sign bytes for an
	// otherwise-identical MsgSend, optionally setting timeout_timestamp.
	buildDirectSignBytes := func(withTimeout bool) []byte {
		txBuilder := suite.clientCtx.TxConfig.NewTxBuilder()
		txBuilder.SetGasLimit(20000)
		txBuilder.SetFeeAmount(suite.makeCoins(suite.denom, math.NewInt(2000)))

		err := txBuilder.SetMsgs(banktypes.NewMsgSend(
			signer,
			suite.createTestAddress(),
			suite.makeCoins(suite.denom, math.NewInt(1)),
		))
		suite.Require().NoError(err)

		if withTimeout {
			txBuilder.SetTimeoutTimestamp(time.Unix(1700000000, 0).UTC())
		}

		err = txBuilder.SetSignatures(signing.SignatureV2{
			PubKey:   pubKey,
			Data:     &signing.SingleSignatureData{SignMode: signing.SignMode_SIGN_MODE_DIRECT},
			Sequence: 0,
		})
		suite.Require().NoError(err)

		signerData := authsigning.SignerData{
			ChainID:       testutil.TestnetChainID + "-1",
			AccountNumber: 25,
			Sequence:      0,
			PubKey:        pubKey,
			Address:       sdk.MustBech32ifyAddressBytes(config.Bech32Prefix, pubKey.Bytes()),
		}

		bz, err := authsigning.GetSignBytesAdapter(
			suite.clientCtx.CmdContext,
			suite.clientCtx.TxConfig.SignModeHandler(),
			signing.SignMode_SIGN_MODE_DIRECT,
			signerData,
			txBuilder.GetTx(),
		)
		suite.Require().NoError(err)
		return bz
	}

	cleanBytes := buildDirectSignBytes(false)
	timeoutBytes := buildDirectSignBytes(true)

	// Setting timeout_timestamp changes the direct sign bytes, so the two
	// transactions are genuinely different and a correct verifier must
	// distinguish them.
	suite.Require().NotEqual(cleanBytes, timeoutBytes)

	// A body without timeout_timestamp still converts on both fallback paths.
	_, err := eip712.GetEIP712BytesForMsg(cleanBytes)
	suite.Require().NoError(err)
	_, err = eip712.LegacyGetEIP712BytesForMsg(cleanBytes)
	suite.Require().NoError(err)

	// A body that sets timeout_timestamp must now be rejected on both the
	// current and legacy EIP-712 fallback paths.
	_, err = eip712.GetEIP712BytesForMsg(timeoutBytes)
	suite.Require().Error(err)
	suite.Require().ErrorContains(err, "TimeoutTimestamp")
	_, err = eip712.LegacyGetEIP712BytesForMsg(timeoutBytes)
	suite.Require().Error(err)
	suite.Require().ErrorContains(err, "TimeoutTimestamp")

	// End-to-end: a signature produced over the clean tx's EIP-712 bytes must
	// verify against the clean tx but NOT against the timeout-mutated tx.
	cleanEIP712, err := eip712.GetEIP712BytesForMsg(cleanBytes)
	suite.Require().NoError(err)
	sig, err := privKey.Sign(cleanEIP712)
	suite.Require().NoError(err)

	suite.Require().True(pubKey.VerifySignature(cleanBytes, sig), "signature should verify against the signed tx")
	suite.Require().False(pubKey.VerifySignature(timeoutBytes, sig), "signature must not verify against a body with a mutated timeout_timestamp")
