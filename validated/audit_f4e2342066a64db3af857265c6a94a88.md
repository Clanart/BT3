Audit Report

## Title
Stale ETH in `PeripheryPayments.pay` Is Silently Consumed by Subsequent WETH Swaps, Stealing Prior Users' Funds — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay` reads `address(this).balance` — the router's total native ETH balance — when settling a WETH payment leg. ETH accumulates in the router whenever a user calls `multicall{value: X}(...)` without appending `refundETH()`. Any subsequent WETH swap by any user then silently consumes that stranded ETH, giving the second user a free or discounted swap while the first user permanently loses their ETH.

## Finding Description
The root cause is in `pay` at lines 73–84 of `PeripheryPayments.sol`:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // total router ETH, not msg.value
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value); // payer pays nothing
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance); // payer underpays
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

ETH enters the router via the `payable` `multicall`: [2](#0-1) 

The `receive()` guard only blocks plain ETH transfers from non-WETH addresses; it does not prevent ETH from entering via a `payable` function call: [3](#0-2) 

When User A calls `multicall{value: X}(...)` and omits `refundETH()`, surplus ETH is permanently stranded. The next user who executes any WETH swap — even with zero `msg.value` — triggers `_justPayCallback` → `pay(WETH, userB, pool, value)`, which reads the full `address(this).balance` and silently consumes User A's stranded ETH on User B's behalf: [4](#0-3) 

The same `pay` function is shared by `MetricOmmPoolLiquidityAdder`, so the same drain path exists for WETH liquidity additions: [5](#0-4) 

## Impact Explanation
Direct loss of user principal. When `address(this).balance >= value`, User A loses up to `value` ETH and User B pays zero WETH while receiving full swap output. When `0 < address(this).balance < value`, User A loses the stranded amount and User B receives a proportional discount. The ETH is irreversibly transferred to the pool with no recourse for User A. This meets the Critical/High direct loss of user principal threshold.

## Likelihood Explanation
Medium. The trigger condition — a user sending ETH via `multicall` without `refundETH()` — is a realistic and documented usage pattern. The test suite demonstrates it as the expected pattern for native-ETH swaps. Once ETH is stranded, any subsequent WETH swap by any unprivileged address drains it; no special permissions or privileged access are required. The attack is permissionless and repeatable.

## Recommendation
Track only the ETH that belongs to the current call. The standard fix (used by Uniswap v3's `SwapRouter`) is to pass `msg.value` through the call chain and cap native consumption to that amount. A minimal targeted fix is to compare `address(this).balance` against `msg.value` inside `pay` and only use native ETH when it was explicitly provided in the current top-level call. Alternatively, enforce that `refundETH()` is always the last call in any `multicall` batch that includes a native-ETH value — but this is a social/documentation control, not a code-level invariant.

## Proof of Concept
```
1. User A calls:
       router.multicall{value: 1 ether}([
           exactInputSingle(tokenIn=WETH, amountIn=0.5 ether, ...)
           // ← refundETH() omitted
       ])
   → 0.5 ETH remains stranded in the router.

2. User B (attacker) calls (no ETH sent):
       router.exactInputSingle(
           tokenIn=WETH, amountIn=0.5 ether, recipient=attacker, ...
       )

3. Pool calls router.metricOmmSwapCallback(amount0Delta, amount1Delta, ...)
   → _justPayCallback → pay(WETH, userB, pool, 0.5 ether)

4. Inside pay():
       nativeBalance = address(this).balance = 0.5 ether   // User A's stranded ETH
       nativeBalance >= value  →  branch 1 taken
       IWETH9(WETH).deposit{value: 0.5 ether}()            // wraps User A's ETH
       IERC20(WETH).safeTransfer(pool, 0.5 ether)          // pays pool
       // safeTransferFrom(userB, ...) is NEVER called

5. Result:
   - User B receives full swap output, pays 0 WETH.
   - User A loses 0.5 ETH with no recourse.
```
