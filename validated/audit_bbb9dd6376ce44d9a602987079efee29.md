Audit Report

## Title
Unattributed Native ETH in `pay()` Allows Cross-Transaction Theft of Victim's Stranded ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` reads `address(this).balance` without any per-caller attribution when `token == WETH`. Native ETH left on the router from a prior transaction (e.g., excess `msg.value` sent without a `refundETH()` step) is silently consumed to fund a subsequent caller's WETH swap, resulting in direct loss of the victim's ETH with zero cost to the attacker.

## Finding Description

The `pay` function branches on `token == WETH` and unconditionally reads the router's full native balance: [1](#0-0) 

There is no check that the native ETH present was contributed by the current `payer`. Any ETH on the contract — regardless of origin or which transaction deposited it — is deposited as WETH and forwarded to the pool on behalf of the current caller.

Native ETH can be stranded between transactions because `exactInputSingle`, `exactInput`, `exactOutputSingle`, and `exactOutput` are all `payable`: [2](#0-1) 

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent `msg.value` from accumulating via payable entry points: [3](#0-2) 

The transient callback context correctly records the `payer` as `msg.sender` at the entry point: [4](#0-3) 

However, `pay()` ignores this attribution entirely when the native balance branch is taken — it deposits `address(this).balance` regardless of whether that ETH belongs to the current payer.

The exploit path:
1. Victim calls `exactInputSingle{value: X}(amountIn: Y)` where `X > Y`, without a `refundETH()` step. `X - Y` ETH remains on the router.
2. Attacker calls `exactInputSingle{value: 0}(amountIn: X-Y, tokenIn: WETH)`.
3. The pool triggers `metricOmmSwapCallback`, which calls `_justPayCallback`: [5](#0-4) 

4. `pay(WETH, attacker, pool, X-Y)` is called. `nativeBalance = X-Y >= value`, so the victim's stranded ETH is deposited as WETH and transferred to the pool. The attacker's WETH balance is never touched.
5. Attacker receives the full swap output at zero cost; victim's ETH is permanently lost.

The test suite confirms that omitting `refundETH()` leaves ETH stranded — the expected pattern explicitly requires it: [6](#0-5) 

## Impact Explanation

Direct loss of victim's native ETH principal. The attacker receives swap output tokens without spending any ETH or WETH. The victim's stranded ETH is permanently consumed. This is a direct fund loss to an unprivileged user triggered by another unprivileged caller, meeting Critical/High severity thresholds under Sherlock contest rules.

## Likelihood Explanation

The precondition — stranded ETH on the router — arises from two common, realistic patterns: (1) a user calls `exactOutputSingle{value: buffer}` directly (not via multicall) and has no in-transaction path to recover the surplus; (2) a user constructs a multicall with excess ETH but omits the `refundETH()` step. An attacker can monitor the router's ETH balance on-chain and immediately follow any transaction that leaves a non-zero balance, making this repeatable and low-effort.

## Recommendation

Attribute native ETH to the current call by tracking how much `msg.value` was contributed in the current transaction — e.g., via a transient storage slot written at each payable entry point and decremented in `pay()`. Only the amount attributable to the current call's `msg.value` should be eligible for the native-balance branch. Alternatively, enforce that any excess `msg.value` is refunded atomically before the entry point returns, so ETH can never persist on the router across transaction boundaries.

## Proof of Concept

```solidity
function test_crossTx_attackerStealsVictimStrandedETH() public {
    uint128 amountIn = 1_000;
    uint256 excess   = 1_000;
    uint256 msgValue = amountIn + excess;

    // Victim tx: sends excess ETH, no refundETH()
    vm.prank(victim);
    router.exactInputSingle{value: msgValue}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
            zeroForOne: true, amountIn: amountIn, amountOutMinimum: 0,
            recipient: victim, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
        })
    );
    assertEq(address(router).balance, excess, "ETH stranded");

    // Attacker tx: zero msg.value, steals victim's ETH
    uint256 attackerWethBefore   = weth.balanceOf(attacker);
    uint256 attackerToken1Before = token1.balanceOf(attacker);

    vm.prank(attacker);
    router.exactInputSingle{value: 0}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
            zeroForOne: true, amountIn: excess, amountOutMinimum: 0,
            recipient: attacker, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
        })
    );

    assertEq(address(router).balance, 0,                              "router drained");
    assertEq(weth.balanceOf(attacker), attackerWethBefore,            "attacker WETH unchanged");
    assertGt(token1.balanceOf(attacker), attackerToken1Before,        "attacker got output for free");
}
```
