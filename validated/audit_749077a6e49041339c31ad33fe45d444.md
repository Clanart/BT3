### Title
Wrong Config Account Used for Signature Verification in `finalize_transfer_sol` — (`solana/programs/bridge_token_factory/src/lib.rs`)

### Summary

`finalize_transfer_sol` reads the pause flag from `ctx.accounts.common.config` but verifies the MPC signature against `ctx.accounts.config.derived_near_bridge_address`. These are two structurally different account paths within the same function. If `FinalizeTransferSol` exposes both a `common.config` and a top-level `config` field, an attacker can supply an attacker-controlled account for `config` (containing an attacker-chosen `derived_near_bridge_address`) while the pause check still passes against the legitimate `common.config`. This is the direct analog of the PoolFactory "two different invariantCheck variables" bug: the same logical variable (`config`) is referenced through two different account paths in the same function, and the security-critical path uses the wrong one.

### Finding Description

In `lib.rs`, `finalize_transfer` consistently uses `ctx.accounts.common.config` for both the pause check and signature verification:

```rust
// finalize_transfer — consistent
ctx.accounts.common.config.paused & FINALIZE_TRANSFER_PAUSED == 0   // line 83
&ctx.accounts.common.config.derived_near_bridge_address              // line 90
```

But `finalize_transfer_sol` uses two different paths:

```rust
// finalize_transfer_sol — INCONSISTENT
ctx.accounts.common.config.paused & FINALIZE_TRANSFER_PAUSED == 0   // line 102  ← common.config
&ctx.accounts.config.derived_near_bridge_address                     // line 109  ← config (different!)
``` [1](#0-0) 

The pause check on line 102 reads from `common.config`, but the signature verification on line 109 reads from a top-level `config` field. If the `FinalizeTransferSol` accounts struct exposes both `common` (a shared sub-struct containing `config`) and a separate top-level `config` account, and if the top-level `config` is not seed-constrained to the same PDA as `common.config`, an attacker can pass an arbitrary account as `config` with an attacker-controlled `derived_near_bridge_address`.

Compare with `finalize_transfer`, which uses `common.config` for both operations: [2](#0-1) 

### Impact Explanation

If the top-level `config` account in `FinalizeTransferSol` is not constrained to the canonical `CONFIG_SEED` PDA, an attacker can:

1. Deploy a fake config account with `derived_near_bridge_address` set to a key they control.
2. Call `finalize_transfer_sol` with a `FinalizeTransferPayload` of their choosing, signed with their own key.
3. Signature verification passes (against the attacker's fake config), and the program mints SOL or SPL tokens to an arbitrary recipient.

This constitutes **unauthorized release of bridged assets** (Critical) or at minimum a **proof/signature verification bypass enabling unauthorized transfer finalization** (High).

### Likelihood Explanation

`finalize_transfer_sol` is a public instruction callable by any Solana account. The only gate is the pause flag, which is read from the legitimate `common.config`. If the top-level `config` account is unconstrained or only weakly constrained (e.g., `mut` without seed enforcement), the attack is trivially reachable by any unprivileged user who can craft a Solana transaction.

### Recommendation

Change `finalize_transfer_sol` to use `ctx.accounts.common.config.derived_near_bridge_address` for signature verification, exactly as `finalize_transfer` does:

```rust
// Before (wrong):
&ctx.accounts.config.derived_near_bridge_address

// After (correct):
&ctx.accounts.common.config.derived_near_bridge_address
```

Additionally, audit the `FinalizeTransferSol` accounts struct to confirm whether a redundant top-level `config` field exists and, if so, remove it to prevent future confusion — mirroring the recommendation in the reference report to remove the redundant `invariantCheck` struct member.

### Proof of Concept

1. Deploy a fake Solana account `fake_config` with `derived_near_bridge_address = attacker_pubkey`.
2. Construct a `FinalizeTransferPayload` specifying an arbitrary recipient and amount.
3. Sign the payload with `attacker_privkey`.
4. Call `finalize_transfer_sol` passing `fake_config` as the `config` account and the legitimate program config as `common.config` (to pass the pause check).
5. `data.verify_signature(…, &ctx.accounts.config.derived_near_bridge_address)` recovers `attacker_pubkey` and matches — verification passes.
6. `ctx.accounts.process(data.payload)` executes, releasing SOL to the attacker-specified recipient. [3](#0-2)

    pub fn finalize_transfer(
        ctx: Context<FinalizeTransfer>,
        data: SignedPayload<FinalizeTransferPayload>,
