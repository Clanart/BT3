### Title
Missing Pause Check on `deploy_token` Allows Token Deployment When Protocol Is Paused — (File: `solana/programs/bridge_token_factory/src/lib.rs`)

---

### Summary

The Solana `bridge_token_factory` program's `deploy_token` instruction is the only user-facing instruction that **lacks a pause check**. Every other state-mutating instruction (`finalize_transfer`, `finalize_transfer_sol`, `init_transfer`, `init_transfer_sol`) guards execution with a bitflag check against `config.paused`. `deploy_token` has no such guard, meaning an admin's decision to pause the protocol is silently bypassed for token deployment operations.

---

### Finding Description

In `solana/programs/bridge_token_factory/src/lib.rs`, the four state-mutating instructions consistently check the pause flag before proceeding:

`finalize_transfer` (lines 82–85):
```rust
require!(
    ctx.accounts.common.config.paused & FINALIZE_TRANSFER_PAUSED == 0,
    error::ErrorCode::Paused
);
``` [1](#0-0) 

`finalize_transfer_sol` (lines 101–104):
```rust
require!(
    ctx.accounts.common.config.paused & FINALIZE_TRANSFER_PAUSED == 0,
    error::ErrorCode::Paused
);
``` [2](#0-1) 

`init_transfer` (lines 125–128) and `init_transfer_sol` (lines 140–143) both check `INIT_TRANSFER_PAUSED` identically. [3](#0-2) 

However, `deploy_token` (lines 66–76) contains **no pause check whatsoever**:

```rust
pub fn deploy_token(
    ctx: Context<DeployToken>,
    data: SignedPayload<DeployTokenPayload>,
) -> Result<()> {
    msg!("Deploying token");
    data.verify_signature((), &ctx.accounts.common.config.derived_near_bridge_address)?;
    ctx.accounts.initialize_token_metadata(data.payload)?;
    Ok(())
}
``` [4](#0-3) 

The EVM counterpart (`OmniBridge.sol`) correctly guards `deployToken` with `whenNotPaused(PAUSED_DEPLOY_TOKEN)`: [5](#0-4) 

The Solana program defines `FINALIZE_TRANSFER_PAUSED` and `INIT_TRANSFER_PAUSED` but has no equivalent `DEPLOY_TOKEN_PAUSED` constant and no corresponding check in `deploy_token`. [6](#0-5) 

---

### Impact Explanation

When the admin triggers a full pause (setting `ALL_PAUSED` via the `pause` instruction), the intent is to halt all protocol operations. Because `deploy_token` skips the pause check, any caller holding a valid MPC-signed `DeployTokenPayload` — including one obtained before the pause was triggered — can still deploy new bridge tokens on Solana. This creates permanent on-chain token mappings (`near_to_starknet_token`-equivalent Solana mint accounts) during a security incident when the protocol is supposed to be fully halted. If the pause was triggered precisely because of a vulnerability in the token deployment flow, the missing check means that vulnerability remains exploitable on Solana even after the admin acts.

This matches the allowed High impact: **"Proof, signature, MPC, Wormhole, or light-client verification bypass enabling unauthorized transfer finalization, token deployment, or message execution."**

---

### Likelihood Explanation

**Medium.** Two conditions must hold simultaneously:

1. The protocol is paused by an admin (a reactive security action).
2. A valid MPC-signed `DeployTokenPayload` exists and has not yet been submitted to Solana (e.g., a relayer queued it before the pause, or a user obtained it from NEAR bridge events).

Both conditions are realistic during a security incident: admins pause reactively after observing an exploit, and signed payloads from the NEAR MPC signer are broadcast as on-chain events before the Solana submission occurs. The attacker does not need to forge a signature — they only need to replay a legitimately signed payload that was already in flight.

---

### Recommendation

Add a `DEPLOY_TOKEN_PAUSED` bitflag constant (analogous to `FINALIZE_TRANSFER_PAUSED`) and guard `deploy_token` with the same pattern used by every other instruction:

```rust
// In constants.rs — add:
pub const DEPLOY_TOKEN_PAUSED: u8 = 1 << 2; // or the next available bit

// In lib.rs — deploy_token:
pub fn deploy_token(
    ctx: Context<DeployToken>,
    data: SignedPayload<DeployTokenPayload>,
) -> Result<()> {
    require!(
        ctx.accounts.common.config.paused & DEPLOY_TOKEN_PAUSED == 0,
        error::ErrorCode::Paused
    );
    msg!("Deploying token");
    data.verify_signature((), &ctx.accounts.common.config.derived_near_bridge_address)?;
    ctx.accounts.initialize_token_metadata(data.payload)?;
    Ok(())
}
```

---

### Proof of Concept

1. Admin observes a security incident and calls `pause` on the Solana `bridge_token_factory`, setting `config.paused = ALL_PAUSED`.
2. A relayer had previously obtained a valid MPC-signed `DeployTokenPayload` from the NEAR bridge's on-chain event log (emitted before the pause).
3. The relayer (or any observer who copied the signed payload) calls `deploy_token` on Solana with that payload.
4. The instruction executes successfully — `config.paused` is never read — and a new mint account is created and registered in the token mapping.
5. All other instructions (`finalize_transfer`, `init_transfer`) correctly revert with `Paused`, but `deploy_token` proceeds, defeating the admin's pause.
