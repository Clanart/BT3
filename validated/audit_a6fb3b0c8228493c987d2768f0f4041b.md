### Title
`eth_call` and `eth_estimateGas` Skip `PostTxProcessing` Hooks, Producing Results That Diverge From Actual Execution - (File: `x/evm/keeper/grpc_query.go`, `x/evm/keeper/state_transition.go`)

### Summary

`eth_call` and `eth_estimateGas` invoke `ApplyMessageWithConfig` with `commit=false` and never call `PostTxProcessing` hooks. When a chain registers hooks that can fail (e.g., ERC-20 ↔ native token conversion), the simulation paths return **success with return data and logs** for transactions that would actually fail with `VmError = ErrPostTxProcessing` and `status=0` in real block execution. This causes `eth_estimateGas` to return a gas estimate for a transaction that will fail on-chain, and `eth_call` to return fabricated success data that protocols use for pre-execution checks.

### Finding Description

**Real execution path — `ApplyTransaction`:**

`ApplyTransaction` calls `ApplyMessageWithConfig(tmpCtx, msg, cfg, true)` and then, if the EVM succeeds, calls `PostTxProcessing`. If the hook returns an error, the entire transaction is reverted: `res.VmError` is set to `ErrPostTxProcessing`, logs are cleared, and the transaction is included in the block with `status=0`. [1](#0-0) 

**Simulation paths — `EthCall` and `EstimateGas`:**

Both handlers call `ApplyMessageWithConfig(ctx, msg, cfg, false)` directly and return immediately. `PostTxProcessing` is never invoked. [2](#0-1) [3](#0-2) 

`PostTxProcessing` is only wired in `ApplyTransaction`, not in `ApplyMessageWithConfig`: [4](#0-3) 

**Secondary divergence — `ErrStateConflict`:**

The code itself acknowledges a second divergence: when a transaction would produce a state conflict between EVM-dirty storage and a nested native action, `stateDB.Commit()` returns `ErrStateConflict` and the transaction is included with `status=0`. Because `commit=false` in simulation paths, this check is never reached, and `eth_call`/`eth_estimateGas` return success. [5](#0-4) 

### Impact Explanation

When a chain registers a `PostTxProcessing` hook that can fail (the canonical Ethermint use case is ERC-20 ↔ native Cosmos bank token conversion):

1. **`eth_estimateGas` returns a gas estimate for a transaction that will fail on-chain.** The user submits the transaction with the estimated gas; it is included in the block, gas fees are consumed, but the transaction has `status=0`. The user's gas fees are mis-accounted based on a simulation that omitted a mandatory execution step.

2. **`eth_call` returns fabricated success data** — including EVM return values and logs that would be cleared by hook failure — for a transaction that would actually fail. DeFi protocols and wallets that use `eth_call` for pre-execution checks (e.g., "will this ERC-20 transfer succeed before I release collateral?") receive incorrect results, exposing a reachable route to unauthorized fund transfers.

This matches: *High — Public JSON-RPC/gRPC simulation path feeds incorrect data into transaction execution or exposes a reachable route to unauthorized balance transfer.*

### Likelihood Explanation

- Any chain built on Ethermint that registers `PostTxProcessing` hooks (the documented and intended extension mechanism) is affected.
- The hook registration API is public and explicitly designed for application-level use.
- Any unprivileged user can call `eth_call` or `eth_estimateGas` via the public JSON-RPC endpoint.
- No special privileges, governance actions, or validator compromise are required.

Likelihood: **Medium** — requires hooks to be registered and reachable by the transaction under simulation, which is the normal operating condition for chains using the ERC-20 module or similar hook-based integrations.

### Recommendation

1. In `EthCall` and `EstimateGas`, after `ApplyMessageWithConfig` returns a non-failed result, invoke `k.PostTxProcessing` in a read-only (non-committing) manner. If the hook returns an error, surface it as a `VmError` in the response, mirroring `ApplyTransaction` behavior.

2. For `ErrStateConflict`: run `stateDB.Commit()` even in simulation mode (on a throwaway cache context) so that the conflict check executes and the simulation result accurately reflects real execution.

3. Document the remaining divergences (e.g., gas refund to fee collector) so integrators are aware of which simulation behaviors are intentionally approximate.

### Proof of Concept

**Setup:** Chain with a `PostTxProcessing` hook registered that fails for a specific contract call (e.g., ERC-20 transfer to a blacklisted address in the native module).

**Step 1 — Call `eth_estimateGas`:**
```json
{"method":"eth_estimateGas","params":[{"from":"0xAlice","to":"0xERC20","data":"<transfer to blacklisted>"}]}
```
Response: `{"result":"0x5208"}` — simulation succeeds, gas estimate returned.

**Step 2 — Submit transaction with estimated gas:**
Transaction is included in the block. `ApplyTransaction` runs `PostTxProcessing`; the hook fails. Result: `VmError = "ErrPostTxProcessing"`, `status=0`, Alice's gas fees consumed.

**Root cause trace:**
- `eth_estimateGas` → `Keeper.EstimateGas` (`grpc_query.go:293`) → `ApplyMessageWithConfig(ctx, msg, cfg, false)` (`grpc_query.go:394`) — `PostTxProcessing` never called.
- Real execution → `ApplyTransaction` (`state_transition.go:163`) → `ApplyMessageWithConfig(tmpCtx, msg, cfg, true)` → `PostTxProcessing(tmpCtx, msg, receipt)` (`state_transition.go:232`) → hook fails → `res.VmError = ErrPostTxProcessing`. [3](#0-2) [1](#0-0) [6](#0-5)

// PostTxProcessing delegate the call to the hooks. If no hook has been registered, this function returns with a `nil` error
func (k *Keeper) PostTxProcessing(ctx sdk.Context, msg *core.Message, receipt *ethtypes.Receipt) error {
	if k.hooks == nil {
		return nil
	}
	return k.hooks.PostTxProcessing(ctx, msg, receipt)
}

// PostTxProcessing delegate the call to underlying hooks
func (mh MultiEvmHooks) PostTxProcessing(ctx sdk.Context, msg *core.Message, receipt *ethtypes.Receipt) error {
	for i := range mh {
		if err := mh[i].PostTxProcessing(ctx, msg, receipt); err != nil {
			return errorsmod.Wrapf(err, "EVM hook %T failed", mh[i])
		}
	}
	return nil
}
