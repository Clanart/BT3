### Title
`EnableCreate` Governance Guard Bypassed for EIP-7702 Code-Deployment via `if-else` Structure тАФ (`File: ante/interfaces/setup.go`, `x/evm/keeper/state_transition.go`)

---

### Summary

The `EnableCreate` governance parameter is intended to prevent all new code deployment on the chain. However, because the guard is implemented as an `if-else if` keyed on `tx.To() == nil`, an EIP-7702 (`SetCode`) transaction тАФ which always has `To != nil` тАФ causes the `EnableCreate` branch to be skipped entirely. When `EnableCreate = false` and `EnableCall = true`, any unprivileged user can submit an EIP-7702 transaction that installs arbitrary delegation code on an EOA, bypassing the governance restriction.

---

### Finding Description

Two locations contain the same `if-else if` guard:

**`ante/interfaces/setup.go` lines 116тАУ120** (ante-handler path):

```go
if !enableCreate && tx.To() == nil {
    return errorsmod.Wrap(evmtypes.ErrCreateDisabled, "failed to create new contract")
} else if !enableCall && tx.To() != nil {
    return errorsmod.Wrap(evmtypes.ErrCallDisabled, "failed to call contract")
}
```

**`x/evm/keeper/state_transition.go` lines 352тАУ356** (execution path, also reached by `eth_call`/`eth_estimateGas`):

```go
if !cfg.Params.EnableCreate && msg.To == nil {
    return nil, errorsmod.Wrap(types.ErrCreateDisabled, "failed to create new contract")
} else if !cfg.Params.EnableCall && msg.To != nil {
    return nil, errorsmod.Wrap(types.ErrCallDisabled, "failed to call contract")
}
```

Because the two branches are mutually exclusive on `To == nil` / `To != nil`, the `else if` means **only one branch ever executes**. For an EIP-7702 (`SetCodeTx`) transaction, `To` is always non-nil (enforced at line 134тАУ136 of `ante/interfaces/setup.go`). Therefore:

- `!enableCreate && tx.To() == nil` тЖТ **always false** for EIP-7702 тЖТ `EnableCreate` check is **never reached**.
- `else if !enableCall && tx.To() != nil` тЖТ only fires when `EnableCall = false`.

When governance sets `EnableCreate = false` (to halt code deployment) while leaving `EnableCall = true`, the EIP-7702 path proceeds unchecked into `applyAuthorization` тЖТ `setAuthorizationDelegation` тЖТ `stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), ...)`, writing delegation bytecode to the authority EOA.

The documentation for `EnableCreate` states: *"When the parameter is disabled, it will prevent all contract creation functionality."* EIP-7702 is a code-deployment mechanism and falls within that intent.

---

### Impact Explanation

When governance disables `EnableCreate` to freeze code deployment on the chain, any unprivileged user can still install arbitrary delegation code on any EOA they control via an EIP-7702 transaction. This constitutes **unauthorized account/code mutation** тАФ the authority EOA's `codeHash` changes from empty to a delegation pointer, altering its execution semantics for all future calls. This matches the allowed High impact: *"EIP-7702 authorization bypass enabling unauthorized account/code mutation."*

---

### Likelihood Explanation

- `EnableCreate = false` is a realistic governance action (e.g., emergency pause of contract deployment).
- Once set, **any** user holding an EOA private key can self-sign an EIP-7702 authorization and submit a `SetCodeTx` with `To = self` and a non-zero `auth.Address`. No special privilege is required.
- The bypass is deterministic and requires no timing, race condition, or validator cooperation.
- The same structural flaw exists in both the ante-handler and the execution-layer check, so neither path independently closes the gap.

---

### Recommendation

Replace the `if-else if` with two independent `if` statements, and add a third independent check that blocks EIP-7702 transactions when `EnableCreate` is disabled:

```go
// Block traditional contract creation
if !enableCreate && tx.To() == nil {
    return errorsmod.Wrap(evmtypes.ErrCreateDisabled, "failed to create new contract")
}
// Block EIP-7702 code deployment when contract creation is disabled
if !enableCreate && tx.SetCodeAuthorizations() != nil {
    return errorsmod.Wrap(evmtypes.ErrCreateDisabled, "EIP-7702 code deployment disabled")
}
// Block contract calls
if !enableCall && tx.To() != nil {
    return errorsmod.Wrap(evmtypes.ErrCallDisabled, "failed to call contract")
}
```

Apply the same fix in `x/evm/keeper/state_transition.go` (`ApplyMessageWithConfig`) using `msg.SetCodeAuthorizations != nil`.

---

### Proof of Concept

**Setup:** Governance passes a proposal setting `EnableCreate = false`, `EnableCall = true`.

**Attack:**
1. Attacker generates EOA key `K`, derives address `A`.
2. Attacker picks any deployed contract address `D` to delegate to.
3. Attacker signs an EIP-7702 authorization: `auth = SignSetCode(K, {ChainID: chainID, Address: D, Nonce: 0})`.
4. Attacker submits a `SetCodeTx` (type 4): `{To: A, AuthList: [auth], ...}`.
5. Ante-handler: `tx.To() = A тЙа nil` тЖТ `!enableCreate && tx.To() == nil` is **false** тЖТ `else if !enableCall && tx.To() != nil` тЖТ `enableCall = true` тЖТ **no error**.
6. `ApplyMessageWithConfig`: same `if-else if` тЖТ same bypass тЖТ enters the `else` branch at line 483.
7. `applyAuthorization` validates the auth, calls `setAuthorizationDelegation` тЖТ `stateDB.SetCode(A, AddressToDelegation(D), ...)`.
8. EOA `A` now has delegation code `0xef0100 || D` committed to state, despite `EnableCreate = false`.

**Result:** Code is deployed to an EOA on a chain where contract creation was governance-disabled. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```go
func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}
```
