### Title
Fee-on-Transfer Token Accounting Corruption Breaks Bridge Collateralization — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` records the caller-supplied `amount` in the `InitTransfer` event without verifying the actual amount received by the bridge contract after `safeTransferFrom`. For fee-on-transfer tokens, the bridge receives fewer tokens than `amount`, but the NEAR side processes the full `amount` from the proof, creating an unbacked supply and breaking bridge collateralization.

---

### Finding Description

In `OmniBridge.initTransfer`, when the token is neither a bridge token nor a custom-minter token, the bridge pulls tokens from the caller and immediately emits the user-supplied `amount` in the event:

```solidity
// evm/src/omni-bridge/contracts/OmniBridge.sol  lines 406-412
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
``` [1](#0-0) 

Immediately after, the unverified `amount` is broadcast in the cross-chain event:

```solidity
// lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // ← user-supplied, never reconciled with actual receipt
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

`SafeERC20` only guards against missing or `false` boolean return values; it does **not** verify that the bridge's balance increased by exactly `amount`. For fee-on-transfer tokens (a well-known class of non-standard ERC20 tokens), the bridge receives `amount − fee_taken`, yet the event records `amount`. The NEAR bridge reads this event as proof and authorises a release of the full `amount` on the destination chain. [3](#0-2) 

The `logMetadata` entry point is fully permissionless — any caller can register any ERC20 token address:

```solidity
// lines 224-231
function logMetadata(address tokenAddress) external payable {
    string memory name = IERC20Metadata(tokenAddress).name();
    ...
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
``` [4](#0-3) 

Once a fee-on-transfer token is registered and users begin depositing, every `initTransfer` call inflates the bridge's recorded liability relative to its actual holdings.

---

### Impact Explanation

Each deposit of a fee-on-transfer token creates a gap between the bridge's actual ERC20 balance and the total amount it has committed to release on the destination chain. Over repeated deposits the bridge becomes progressively undercollateralised. When later users attempt to withdraw via `finTransfer`, the `safeTransfer` call will eventually fail because the bridge holds insufficient tokens, permanently freezing those users' funds. This matches the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

---

### Likelihood Explanation

- `logMetadata` is callable by any unprivileged address with any token address — no admin gate.
- Fee-on-transfer tokens (e.g., tokens with reflection mechanics or deflationary burns) are common on mainnet.
- The attacker does not need any special role; they only need to register the token and initiate deposits.
- The damage accumulates silently with every legitimate user deposit, requiring no further attacker interaction after registration.

---

### Recommendation

Record the bridge's ERC20 balance before and after `safeTransferFrom` and use the measured delta as the canonical transfer amount:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(received == amount, "Fee-on-transfer token not supported");
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported, and add an on-chain allowlist check so that only pre-approved tokens can be used in `initTransfer`.

---

### Proof of Concept

1. Deploy a fee-on-transfer ERC20 token `FoT` that deducts 5% on every transfer.
2. Call `OmniBridge.logMetadata(FoT)` — succeeds with no access control.
3. Submit the resulting proof to the NEAR bridge to register `FoT`; NEAR records `decimals` and the token mapping.
4. Alice calls `OmniBridge.initTransfer(FoT, 1000, ...)`.
   - `safeTransferFrom` moves 1000 tokens from Alice; bridge receives 950 (5% fee).
   - `InitTransfer` event records `amount = 1000`.
5. NEAR processes the proof and authorises release of 1000 units on the destination chain.
6. Repeat steps 4–5 many times. The bridge's actual `FoT` balance grows by 950 per iteration while its committed liability grows by 1000.
7. When users attempt `finTransfer` to reclaim tokens, the bridge's `safeTransfer` call eventually reverts due to insufficient balance, permanently freezing those withdrawals. [5](#0-4)
