### Title
Bridge Token Deployment via `CREATE` Opcode Enables Token-Address Mapping Corruption on Reorg-Susceptible Chains — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.deployToken()` deploys `BridgeToken` proxy contracts using Solidity's `new` keyword (the `CREATE` opcode), whose output address depends solely on the `OmniBridge` contract address and its current nonce. On chains susceptible to block reorganizations (Polygon, Optimism, Arbitrum), a reorg that reorders two pending `deployToken` transactions causes the deployed token addresses to swap, permanently corrupting the bridge's `nearToEthToken`/`ethToNearToken` mappings and orphaning any tokens already minted to users at the pre-reorg addresses.

---

### Finding Description

In `OmniBridge.deployToken()`, the bridge token proxy is deployed as:

```solidity
address bridgeTokenProxy = address(
    new ERC1967Proxy(
        tokenImplementationAddress,
        abi.encodeWithSelector(
            BridgeToken.initialize.selector,
            metadata.name,
            metadata.symbol,
            decimals
        )
    )
);
``` [1](#0-0) 

`new ERC1967Proxy(...)` uses the EVM `CREATE` opcode. The resulting address is deterministically derived from `keccak256(rlp(OmniBridge_address, nonce))` — it depends **only on the deployer's nonce**, not on any token-specific input such as the NEAR token ID, name, or symbol.

After deployment, the bridge registers the address in its bidirectional mappings:

```solidity
isBridgeToken[address(bridgeTokenProxy)] = true;
ethToNearToken[address(bridgeTokenProxy)] = metadata.token;
nearToEthToken[metadata.token] = address(bridgeTokenProxy);
``` [2](#0-1) 

**Reorg attack scenario:**

1. Two valid `deployToken` transactions are pending in the mempool: one for NEAR token `X` (signed by the MPC key), one for NEAR token `Y` (also signed by the MPC key). Both signatures are already public on-chain.
2. They execute in order: Token X deploys at nonce N → address A; Token Y deploys at nonce N+1 → address B. The bridge maps `X → A`, `Y → B`.
3. The bridge's `finTransfer` flow mints Token X tokens to users at address A.
4. A block reorg occurs (documented on Polygon with reorgs up to 1.5 minutes long). The two transactions execute in **reversed order**: Token Y deploys at nonce N → address A; Token X deploys at nonce N+1 → address B.
5. The bridge now maps `X → B`, `Y → A`.
6. Users who received Token X tokens at address A now hold tokens at a contract the bridge considers to be Token Y. Their Token X holdings are permanently orphaned — the bridge will never recognize address A as Token X again.

The attacker's role is simply to be a relayer who submits the second `deployToken` transaction before the first one is re-included after the reorg, or to observe the reorg and reorder the two already-signed transactions. No privileged key is needed beyond what is already public in the mempool.

---

### Impact Explanation

- **Token-mapping corruption**: `nearToEthToken` and `ethToNearToken` permanently map each NEAR token ID to the wrong EVM contract address.
- **Permanent freezing of user funds**: Tokens already minted to users at the pre-reorg address are orphaned. `finTransfer` will mint future tokens at the new (post-reorg) address, while `initTransfer` (burn/lock) will only accept tokens from the new address. Tokens at the old address cannot be bridged back and are permanently frozen.
- **Unbacked supply**: The bridge may mint new tokens at the new address while tokens at the old address remain in circulation, breaking the 1:1 collateralization invariant.

This matches the allowed impact: **"High. Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value"** and **"Critical. Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds."**

---

### Likelihood Explanation

- Polygon has documented reorgs of up to 1.5 minutes in practice (as shown in the reference report). Optimism and Arbitrum are also susceptible to reorgs under fraud-proof scenarios.
- The `OmniBridgeWormhole` contract is designed for multi-chain deployment including these networks.
- Multiple `deployToken` transactions being pending simultaneously is realistic during bridge bootstrapping or when multiple new tokens are being registered.
- The two MPC-signed `deployToken` transactions are already public in the mempool; no key compromise is needed. A relayer or any observer can reorder them after a reorg.

---

### Recommendation

Deploy bridge token proxies using `CREATE2` with a salt derived from the NEAR token identifier, making the deployed address deterministic and token-specific regardless of nonce or transaction ordering:

```solidity
bytes32 salt = keccak256(abi.encodePacked(metadata.token));
address bridgeTokenProxy = address(
    new ERC1967Proxy{salt: salt}(
        tokenImplementationAddress,
        abi.encodeWithSelector(
            BridgeToken.initialize.selector,
            metadata.name,
            metadata.symbol,
            decimals
        )
    )
);
```

With `CREATE2`, the address of Token X is always the same regardless of which nonce the deployment lands on, eliminating the reorg-induced address swap.

### Proof of Concept

**Pre-conditions:**
- `OmniBridge` is deployed on Polygon (reorg-susceptible).
- MPC has signed metadata for Token X (`tokenX.near`) and Token Y (`tokenY.near`). Both signed transactions are broadcast to the mempool.

**Steps:**

1. Mempool contains:
   - Tx1: `deployToken(sigX, {token: "tokenX.near", ...})` — valid MPC signature
   - Tx2: `deployToken(sigY, {token: "tokenY.near", ...})` — valid MPC signature

2. Block N includes Tx1 then Tx2:
   - `OmniBridge` nonce = 5 → Token X deployed at `addr_A`
   - `OmniBridge` nonce = 6 → Token Y deployed at `addr_B`
   - Bridge state: `nearToEthToken["tokenX.near"] = addr_A`, `nearToEthToken["tokenY.near"] = addr_B`

3. Bridge relayer calls `finTransfer` for a user bridging Token X from NEAR → mints Token X at `addr_A` to user Alice.

4. Polygon reorg reverts Block N. Attacker (acting as relayer) resubmits Tx2 before Tx1:
   - `OmniBridge` nonce = 5 → Token Y deployed at `addr_A`
   - `OmniBridge` nonce = 6 → Token X deployed at `addr_B`
   - Bridge state: `nearToEthToken["tokenX.near"] = addr_B`, `nearToEthToken["tokenY.near"] = addr_A`

5. Alice holds Token X tokens at `addr_A`. The bridge now considers `addr_A` to be Token Y. Alice cannot call `initTransfer` to bridge her tokens back — the bridge will reject `addr_A` as Token X. Her funds are permanently frozen. [3](#0-2)

    function deployToken(
        bytes calldata signatureData,
        BridgeTypes.MetadataPayload calldata metadata
    ) external payable whenNotPaused(PAUSED_DEPLOY_TOKEN) returns (address) {
        if (tokenImplementationAddress == address(0)) {
            revert TokenImplementationNotSet();
        }
        bytes memory borshEncoded = bytes.concat(
            bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
            Borsh.encodeString(metadata.token),
            Borsh.encodeString(metadata.name),
            Borsh.encodeString(metadata.symbol),
            bytes1(metadata.decimals)
        );
        bytes32 hashed = keccak256(borshEncoded);

        if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
            revert InvalidSignature();
        }

        require(
            !isBridgeToken[nearToEthToken[metadata.token]],
            "ERR_TOKEN_EXIST"
        );
        uint8 decimals = _normalizeDecimals(metadata.decimals);

        // slither-disable-next-line reentrancy-no-eth
        address bridgeTokenProxy = address(
            new ERC1967Proxy(
                tokenImplementationAddress,
                abi.encodeWithSelector(
                    BridgeToken.initialize.selector,
                    metadata.name,
                    metadata.symbol,
                    decimals
                )
            )
        );

        deployTokenExtension(
            metadata.token,
            bridgeTokenProxy,
            decimals,
            metadata.decimals
        );

        emit BridgeTypes.DeployToken(
            bridgeTokenProxy,
            metadata.token,
            metadata.name,
            metadata.symbol,
            decimals,
            metadata.decimals
        );

        isBridgeToken[address(bridgeTokenProxy)] = true;
        ethToNearToken[address(bridgeTokenProxy)] = metadata.token;
        nearToEthToken[metadata.token] = address(bridgeTokenProxy);

        return bridgeTokenProxy;
    }
