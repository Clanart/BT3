### Title
Asymmetric `within_margin` Anchor Computes Tolerance from Proposed Price Instead of Reference, Allowing Proposer to Inflate L1 Gas Price Beyond the 10% Bound - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The `within_margin` function used to validate a proposer's L1 gas price computes the percentage tolerance band relative to `number1` (the **proposed** price) rather than `number2` (the **locally-computed reference** price). This is the direct sequencer analog of the inverted-bounds bug in the external report: the wrong value is used as the anchor for the range check, making the check asymmetric. A malicious proposer can set an L1 gas price up to ≈11.1 % above the reference and have every validator accept it, when the intended maximum deviation is 10 %.

---

### Finding Description

`is_block_info_valid` validates the proposer's claimed L1 gas price by calling:

```rust
within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
``` [1](#0-0) 

Inside `within_margin`, the tolerance band is computed as:

```rust
let margin = (number1.0 * margin_percent) / 100;   // number1 = proposed price
number1.0.abs_diff(number2.0) <= margin
``` [2](#0-1) 

`number1` is the **proposed** price; `number2` is the **reference** price. The margin should be anchored to the reference (`number2`), not to the proposed value.

**Concrete arithmetic with `margin_percent = 10`** (the value in all deployed versioned-constants files): [3](#0-2) 

| Scenario | Reference | Proposed | Current margin (from proposed) | Accepted? | Correct margin (from reference) | Should accept? |
|---|---|---|---|---|---|---|
| Proposed 10 % above ref | 1 000 | 1 100 | 110 | ✓ | 100 | ✓ |
| Proposed **11.1 % above ref** | 1 000 | 1 111 | **111.1** | **✓ (bug)** | 100 | **✗** |
| Proposed 10 % below ref | 1 000 | 900 | 90 | **✗ (bug)** | 100 | ✓ |

Solving `P − R ≤ (P × 10) / 100` gives `P ≤ 10R/9 ≈ 1.1111R`, so the current code accepts proposals up to **11.11 %** above the reference instead of the intended 10 %.

The `l1_gas_price_margin_percent` is always read from `VersionedConstants::latest_constants()`, hardcoded to `10` across all deployed versions. [4](#0-3) 

---

### Impact Explanation

The L1 gas price from `ProposalInit` is passed directly into `convert_to_sn_api_block_info`, which constructs the `BlockInfo` used by the blockifier for every transaction's fee calculation in that block. [5](#0-4) 

It is also chained into `PartialBlockHashComponents` and ultimately into `calculate_block_hash`, so the wrong price is committed to the block hash. [6](#0-5) 

A proposer who sets `l1_gas_price_fri` to `10R/9` (≈11.1 % above the reference `R`) will pass every validator's `within_margin` check because each validator independently computes the same inflated margin from the proposed value. The result is:

1. **Wrong L1 gas price committed to the block hash** — every transaction in the block pays fees computed against an inflated price.
2. **Wrong authoritative value returned by RPC** — `starknet_getBlockWithTxs` and fee-estimation endpoints reflect the inflated price.

This matches the allowed impact: *"Critical. Incorrect fee, gas … or L1 gas price effect with economic impact"* and *"High. RPC execution, fee estimation … returns an authoritative-looking wrong value."*

---

### Likelihood Explanation

Exploitation requires the attacker to be the selected proposer for a consensus round. In Tendermint-style rotation any validator can be proposer, so this is not a permanently privileged role. The extra deviation is small (~1.1 % beyond the 10 % band), but it is systematic: every block proposed by a malicious validator can carry the inflated price, and all honest validators will accept it due to the same buggy check.

---

### Recommendation

Anchor the margin to the **reference** price (`number2`), not the proposed price (`number1`):

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    // Anchor to the reference (number2), not the proposed value (number1).
    let margin = (number2.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}
```

This makes the check symmetric: any proposed price within ±10 % of the reference is accepted, and any price outside that band is rejected, regardless of direction.

---

### Proof of Concept

Using the existing unit-test harness in `validate_proposal_test.rs`:

```rust
// Demonstrates the asymmetry: proposed = 909 is 9.1% below reference 1000,
// which is within 10% but is REJECTED by the current code.
assert_eq!(within_margin(GasPrice(909), GasPrice(1000), 10), false);
// margin = (909 * 10) / 100 = 90; diff = 91; 91 <= 90 → false  ← BUG

// Proposed = 1111 is 11.1% above reference 1000,
// which exceeds 10% but is ACCEPTED by the current code.
assert_eq!(within_margin(GasPrice(1111), GasPrice(1000), 10), true);
// margin = (1111 * 10) / 100 = 111; diff = 111; 111 <= 111 → true  ← BUG
``` [7](#0-6) 

A malicious proposer sets `init.l1_gas_price_fri = reference * 10 / 9` in the `ProposalInit`. Every validator calls `is_block_info_valid`, computes `within_margin(proposed, reference, 10)`, and accepts the proposal because the margin is computed from the inflated proposed value. The block is committed with the wrong gas price, affecting all transaction fees and the block hash. [8](#0-7)

fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    // For small numbers (e.g., less than 10 wei, if margin is 10%), even an off-by-one
    // error might be bigger than the margin, even if it is just a rounding error.
    // We make an exception for such mismatch, and don't bother checking percentages
    // if the difference in price is only one wei.
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}

pub(crate) fn convert_to_sn_api_block_info(
    init: &ProposalInit,
) -> Result<starknet_api::block::BlockInfo, StarknetApiError> {
    if init.l1_gas_price_fri.0 == 0
        || init.l1_gas_price_wei.0 == 0
        || init.l1_data_gas_price_fri.0 == 0
        || init.l1_data_gas_price_wei.0 == 0
        || init.l2_gas_price_fri.0 == 0
    {
        warn!("Zero gas price detected in block info: {:?}", init);
    }

    let l1_gas_price_fri = NonzeroGasPrice::new(init.l1_gas_price_fri)?;
    let l1_data_gas_price_fri = NonzeroGasPrice::new(init.l1_data_gas_price_fri)?;
    let l1_gas_price_wei = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
    let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
    let l2_gas_price_fri = NonzeroGasPrice::new(init.l2_gas_price_fri)?;
    let previous_block_info = PreviousBlockInfo::from(init);
    let eth_to_fri_rate = calculate_eth_to_fri_rate(&previous_block_info)?;

    let l2_gas_price_wei = NonzeroGasPrice::new(init.l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)?)
        .inspect_err(|_| {
            warn!(
                "L2 gas price in wei is zero! Conversion rate: {eth_to_fri_rate}, L2 gas price in \
                 FRI: {}",
                init.l2_gas_price_fri
            )
        })?;
    Ok(starknet_api::block::BlockInfo {
        block_number: init.height,
        block_timestamp: BlockTimestamp(init.timestamp),
        sequencer_address: init.builder,
        gas_prices: GasPrices {
            strk_gas_prices: GasPriceVector {
                l1_gas_price: l1_gas_price_fri,
                l1_data_gas_price: l1_data_gas_price_fri,
                l2_gas_price: l2_gas_price_fri,
            },
            eth_gas_prices: GasPriceVector {
                l1_gas_price: l1_gas_price_wei,
                l1_data_gas_price: l1_data_gas_price_wei,
                l2_gas_price: l2_gas_price_wei,
            },
        },
        use_kzg_da: init.l1_da_mode.is_use_kzg_da(),
        starknet_version: init.starknet_version,
    })
}

```rust
pub fn calculate_block_hash(
    partial_block_hash_components: &PartialBlockHashComponents,
    state_root: GlobalRoot,
    previous_block_hash: BlockHash,
) -> StarknetApiResult<BlockHash> {
    let block_hash_version: BlockHashVersion =
        partial_block_hash_components.starknet_version.try_into()?;
    let block_commitments = &partial_block_hash_components.header_commitments;
    Ok(BlockHash(
        HashChain::new()
            .chain(&block_hash_version.clone().into())
            .chain(&partial_block_hash_components.block_number.0.into())
            .chain(&state_root.0)
            .chain(&partial_block_hash_components.sequencer.0)
            .chain(&partial_block_hash_components.timestamp.0.into())
            .chain(&block_commitments.concatenated_counts)
            .chain(&block_commitments.state_diff_commitment.0.0)
            .chain(&block_commitments.transaction_commitment.0)
            .chain(&block_commitments.event_commitment.0)
            .chain(&block_commitments.receipt_commitment.0)
            .chain_iter(
                gas_prices_to_hash(
                    &partial_block_hash_components.l1_gas_price,
                    &partial_block_hash_components.l1_data_gas_price,
                    &partial_block_hash_components.l2_gas_price,
                    &block_hash_version,
                )
                .iter(),
            )
            .chain(
                &Felt::try_from(&partial_block_hash_components.starknet_version)
                    .expect("Expect ASCII version"),
            )
            .chain(&Felt::ZERO)
            .chain(&previous_block_hash.0)
            .get_poseidon_hash(),
    ))
}
