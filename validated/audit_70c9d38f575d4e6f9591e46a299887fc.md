### Title
Fee-Amount Interaction Deadlock Permanently Locks User Tokens in Bridge — (`near/omni-bridge/src/lib.rs`)

### Summary

The NEAR bridge enforces a fee validity check at `init_transfer` time (`fee < amount`) and a separate normalized-amount check at `sign_transfer` time (`normalize(amount - fee) > 0`). When `amount - fee` falls in the range `[1, 10^(origin_decimals - decimals) - 1]`, both checks are individually satisfiable but their interaction creates a permanent deadlock: the transfer is accepted and tokens are locked/burned, yet `sign_transfer` will always revert, leaving the funds irrecoverable.

### Finding Description

**Step 1 — `init_transfer` accepts the transfer.**

In `init_transfer` (called via `ft_transfer_call` → `ft_on_transfer`), the only fee validity check is:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

A transfer with `amount = N` and `fee = N - k` (where `1 ≤ k < 10^(origin_decimals - decimals)`) passes this check. Execution continues into `init_transfer_internal`, which locks native tokens or burns bridged tokens:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(...);
``` [2](#0-1) 

**Step 2 — `sign_transfer` permanently rejects the transfer.**

When a relayer calls `sign_transfer`, the net amount is normalized for the destination chain's decimal precision:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [3](#0-2) 

`normalize_amount` performs floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [4](#0-3) 

If `amount - fee < 10^(origin_decimals - decimals)`, then `normalize_amount` returns `0`, and `sign_transfer` panics with `InvalidAmountToTransfer` on every invocation. The transfer message remains in `pending_transfers` indefinitely.

**Step 3 — No recovery path exists.**

`update_transfer_fee` only allows the fee to be *increased* (`fee.fee >= current_fee.fee`), which makes `amount - fee` even smaller and worsens the deadlock: [5](#0-4) 

There is no cancel-transfer or admin-refund function for NEAR-originated pending transfers. The transfer message is only removed from `pending_transfers` upon a successful `sign_transfer_callback` (zero-fee path) or `claim_fee_callback`, neither of which can be reached.

### Impact Explanation

For native (non-bridged) tokens: the full `amount` is locked inside the bridge contract via `lock_tokens_if_needed` and cannot be released without DAO intervention that has no on-chain mechanism to refund the original sender.

For bridged tokens: the full `amount` is burned via `burn_tokens_if_needed` and is permanently destroyed.

In both cases, the user's funds are irrecoverably lost. This matches the **Critical** impact class: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.*

### Likelihood Explanation

The condition requires:
1. A token registered with `origin_decimals > decimals` (e.g., a NEAR-native token with 8 decimals bridging to a 6-decimal destination, giving a dead zone of `amount - fee < 100`).
2. A user who sets `fee` such that `amount - fee` falls in the dead zone — either accidentally (trying to give the relayer nearly all of the transfer as incentive) or through a UI that does not validate the post-normalization net amount.

For tokens with small decimal differences (diff = 1 or 2), the dead zone spans amounts that are economically meaningful (e.g., 1–99 base units). The entry path is fully unprivileged: any user calling `ft_transfer_call` with a crafted `InitTransferMsg` can trigger it.

### Recommendation

Add a pre-flight normalization check inside `init_transfer` (before tokens are locked/burned) to reject transfers where the net amount after fee and normalization would be zero:

```rust
let decimals = self.token_decimals.get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);
let net_normalized = Self::normalize_amount(
    transfer_message.amount.0.checked_sub(transfer_message.fee.fee.0)
        .near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(net_normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

This mirrors the fix recommended in M-03: adjust the lower-bound check to account for the upper-bound constraint (normalization floor), preventing the deadlock state from ever being entered.

### Proof of Concept

Consider a token registered with `origin_decimals = 8`, `decimals = 6` (diff = 2, divisor = 100).

1. User calls `ft_transfer_call` with `amount = 150`, `fee = 100`.
2. `init_transfer` check: `100 < 150` ✓ — transfer accepted, 150 tokens locked.
3. `init_transfer_internal` stores the transfer message and locks 150 tokens.
4. Relayer calls `sign_transfer`:
   - `amount_without_fee() = 150 - 100 = 50`
   - `normalize_amount(50, {origin: 8, dest: 6}) = 50 / 100 = 0`
   - `require!(0 > 0)` → **panic: `ERR_INVALID_AMOUNT_TO_TRANSFER`**
5. Every subsequent `sign_transfer` call panics identically.
6. `update_transfer_fee` can only raise the fee further, deepening the deadlock.
7. The 150 tokens remain locked in the bridge with no on-chain recovery path. [1](#0-0) [3](#0-2) [4](#0-3) [5](#0-4) [2](#0-1)
