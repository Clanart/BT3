### Title
Arithmetic Overflow in Relative Timelock Validation Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Conditions — (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary

`check_time_locks` uses `wrapping_add` (instead of `saturating_add`) when the `nowrap` flag is `false` to compute the deadline for relative timelock conditions. An attacker who crafts a spend with a near-maximum `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` value causes the sum `confirmed_height + height_relative` to wrap around to a value smaller than the current block height, making the timelock appear satisfied at any block height. The inverse overflow on `ASSERT_BEFORE_HEIGHT_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE` causes those guards to always fail, permanently locking coins that use them.

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` accepts a `nowrap: bool` parameter. When `nowrap = false` (the legacy/pre-fork path), relative timelock deadlines are computed with `wrapping_add`:

```rust
// ASSERT_HEIGHT_RELATIVE — nowrap=false branch
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

and similarly for `seconds_relative`, `before_height_relative`, and `before_seconds_relative`. [1](#0-0) 

When `nowrap = true`, the same paths use `saturating_add`, which clamps at the type maximum and produces the correct result: [2](#0-1) 

The condition parser in `conditions.rs` accepts `ASSERT_HEIGHT_RELATIVE` values up to `0x8000_0000` (confirmed by test) and `ASSERT_SECONDS_RELATIVE` values up to `0x8000_0000_0000_0000`: [3](#0-2) 

The overflow behavior is explicitly documented and tested in the unit tests:

```
// 10 + u32::MAX overflows to 9, wrapping: 15 < 9 -> Ok  (timelock bypassed)
height_relative=0xFFFF_FFFF  →  nowrap=false: Ok(())
// 10000 + u64::MAX overflows to 9999, wrapping: 10150 < 9999 -> Ok
seconds_relative=0xFFFF_FFFF_FFFF_FFFF  →  nowrap=false: Ok(())
``` [4](#0-3) 

The inverse effect on `ASSERT_BEFORE_HEIGHT_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE` causes those conditions to always fail when the sum wraps to a small value, permanently preventing any spend that uses them from being accepted: [5](#0-4) 

The Python binding exposes `nowrap` as a caller-controlled parameter, meaning the Chia node software decides which mode is active for each block: [6](#0-5) 

### Impact Explanation

**Timelock bypass (High):** When `nowrap=false` is active, a spend containing `ASSERT_HEIGHT_RELATIVE = 0xFFFF_FFFF` (or a value chosen so that `confirmed_height + height_relative` wraps below `prev_height`) passes the timelock check at any block height. A coin that was intended to be locked for ~136 years (u32::MAX blocks) can be spent immediately. This enables unauthorized spend acceptance — a coin protected only by a relative height or seconds timelock can be drained before the lock expires.

**Permanent coin lock (High):** The symmetric overflow on `ASSERT_BEFORE_HEIGHT_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE` causes the deadline to wrap to a value already exceeded, making the condition always fail. Any coin whose puzzle emits such a condition with a near-maximum value becomes permanently unspendable under `nowrap=false`, effectively burning it.

### Likelihood Explanation

The `nowrap=false` path is the legacy behavior retained for pre-fork block validation. The `nowrap` flag is passed by the Chia node and is not attacker-controlled, but the relative timelock value inside the spend **is** fully attacker-controlled. If any production code path calls `check_time_locks` with `nowrap=false` for blocks that include attacker-submitted transactions, the bypass is directly reachable. The condition parser accepts `ASSERT_HEIGHT_RELATIVE` values up to at least `0x8000_0000`, and the wrapping bypass requires values near `u32::MAX - confirmed_height`, which is achievable for recently-confirmed coins.

### Recommendation

1. Remove the `nowrap=false` branch entirely once the fork that mandates `saturating_add` has activated on all supported networks. Until then, gate the `wrapping_add` path so it is only reachable for blocks at heights strictly below the fork activation height.
2. Add a consensus-layer assertion that `height_relative` and `seconds_relative` values that would cause overflow under `saturating_add` are rejected at condition-parse time (they already produce `PositiveOverflow → AssertHeightRelativeFailed` for values exceeding the type width, but values just below the type maximum are silently accepted and cause the overflow at check time).
3. Consider making `nowrap` an internal constant rather than a caller-supplied parameter to prevent accidental misuse from Python bindings.

### Proof of Concept

```python
# Analog to the H-09 POC: craft a spend whose relative timelock wraps to zero.
# Coin confirmed at block height 1.
# ASSERT_HEIGHT_RELATIVE = 0xFFFF_FFFF (u32::MAX)
# wrapping: 1 + 0xFFFF_FFFF = 0  (mod 2^32)
# check: prev_height < 0  →  always False  →  timelock passes at block 2

from chia_rs import check_time_locks, SpendBundleConditions, SpendConditions, CoinRecord, Coin
from chia_rs.sized_ints import uint32, uint64

coin = Coin(b'\x00'*32, b'\x00'*32, 1000)
coin_id = coin.name()
record = CoinRecord(coin, confirmed_block_index=uint32(1), spent_block_index=uint32(0),
                    coinbase=False, timestamp=uint64(1000))

spend = SpendConditions(
    coin_id, b'\x00'*32, b'\x00'*32, 1000,
    height_relative=uint32(0xFFFF_FFFF),  # near-max relative lock
    # ... other fields zero/None
)
bundle = SpendBundleConditions(spends=[spend], ...)

# With nowrap=True (correct): AssertHeightRelativeFailed
result_safe = check_time_locks({coin_id: record}, bundle, prev_height=2, timestamp=2000, nowrap=True)
assert result_safe == 13  # AssertHeightRelativeFailed

# With nowrap=False (legacy): timelock bypassed — spend accepted at block 2
result_vuln = check_time_locks({coin_id: record}, bundle, prev_height=2, timestamp=2000, nowrap=False)
assert result_vuln is None  # No error — timelock silently bypassed
```
