### Title
`PeripheryPayments.pay()` Consumes Only the Required ETH and Leaves Excess Native Balance on the Router, Where Any Caller Can Drain It via `refundETH()` - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

---

### Summary

When a user calls `exactOutputSingle` or `exactOutput` with WETH as `tokenIn` and sends more native ETH than the swap actually requires, `PeripheryPayments.pay()` deposits only the exact required amount as WETH and silently leaves the remainder on the router. Because `refundETH()` is a public, attribution-free function that transfers the router's entire ETH balance to `msg.sender`, any third party can drain the stranded ETH in a subsequent transaction, causing a direct loss of user principal.

---

### Finding Description

`PeripheryPayments.pay()` handles the WETH-as-tokenIn case with the following branch:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();   // only `value` deposited
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) { ... }
}
``` [1](#0-0) 

When `nativeBalance >= value`, exactly `value` wei is deposited and forwarded; the surplus `nativeBalance - value` remains on the router with no attribution record and no automatic refund.

`refundETH()` is public, payable, and unconditionally transfers the router's full ETH balance to `msg.sender`:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
``` [2](#0-1) 

There is no mapping from deposited ETH to the original sender, no per-transaction lock, and no check that the caller of `refundETH` is the same address that sent the ETH. Any account that calls `refundETH` after a victim's swap transaction receives the entire stranded balance.

The entry points that expose this path are `exactOutputSingle` and `exactOutput`, both of which are `external payable`: [3](#0-2) 

The callback that triggers `pay()` is `_justPayCallback`, called from `metricOmmSwapCallback` with `payer = msg.sender` (the original caller) and `token = tokenIn`: [4](#0-3) 

The same vulnerability applies to `MetricOmmPoolLiquidityAdder`, which inherits `PeripheryPayments` and exposes the same `refundETH()` surface: [5](#0-4) 

---

### Impact Explanation

**Severity: High — direct loss of user ETH principal.**

A user who calls `exactOutputSingle{value: X}(...)` with WETH as `tokenIn` where `X` exceeds the actual swap input loses `X - actualInput` ETH to the first caller of `refundETH()`. The loss is bounded only by how much excess ETH the victim sent. For exact-output swaps the required input is not known in advance, so users naturally send a buffer above the expected amount. The stolen ETH is transferred out of the router in the same block with no recovery path.

---

### Likelihood Explanation

**Medium.** Two conditions must hold:

1. The user calls an exact-output swap (or any payable swap function) with WETH as `tokenIn` and sends more native ETH than the pool ultimately requires.
2. The user does not include `refundETH()` as the final step of a `multicall` in the same transaction.

Condition 1 is the normal usage pattern for exact-output swaps (the required input is unknown until execution). Condition 2 is a documentation/UX gap: `exactOutputSingle` is `external payable` and can be called directly without a multicall wrapper. A user who calls it directly, or who assembles a multicall without a trailing `refundETH`, is silently vulnerable. A front-running bot monitoring the mempool can observe the stranded ETH and call `refundETH()` in the next block.

---

### Recommendation

1. **Auto-refund in exact-output entry points.** At the end of `exactOutputSingle` and `exactOutput`, when `tokenIn == WETH`, compare `address(this).balance` before and after the swap and transfer any surplus back to `msg.sender`.

2. **Alternatively, enforce strict ETH consumption.** After `pay()` returns in the WETH branch, revert if `address(this).balance` is non-zero (i.e., `msg.value` was not fully consumed). This mirrors the strict-equality fix applied in the external report.

3. **Add access control to `refundETH`.** At minimum, record the `msg.sender` of the outermost `multicall` or swap entry point in transient storage and restrict `refundETH` to that address within the same transaction.

---

### Proof of Concept

```
Setup:
  - Pool: WETH / Token1, oracle bid/ask such that 1 ETH buys 1500 Token1
  - Alice wants exactly 1500 Token1 out

Step 1 — Alice calls exactOutputSingle directly (no multicall):
  router.exactOutputSingle{value: 2 ether}(ExactOutputSingleParams({
      pool:           address(pool),
      tokenIn:        address(weth),
      tokenOut:       address(token1),
      zeroForOne:     true,
      amountOut:      1500,
      amountInMaximum: 2 ether,
      recipient:      alice,
      ...
  }));

Step 2 — Inside metricOmmSwapCallback → _justPayCallback → pay():
  nativeBalance = 2 ether
  value         = 1 ether  (actual required input)
  → deposit(1 ether), transfer 1 WETH to pool
  → 1 ether remains on router, no refund, no revert

Step 3 — Bob observes the stranded ETH (mempool or on-chain):
  router.refundETH();
  // _transferETH(bob, 1 ether) succeeds
  // Bob receives Alice's 1 ETH

Net result: Alice receives 1500 Token1 but loses 2 ETH instead of 1 ETH.
           Bob gains 1 ETH at zero cost.
```

The `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` test in the test suite confirms that excess ETH does remain on the router after a swap and is claimable via `refundETH` — it just assumes the victim includes the refund call themselves: [6](#0-5)
