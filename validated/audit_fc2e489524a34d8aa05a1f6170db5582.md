### Title
Unbounded `waiting_for_block` Buffer in `SpiceChunkValidatorActor` Allows Witness-Driven Memory Exhaustion Without Size or Distance Bounds — (File: chain/client/src/spice/chunk_validator_actor.rs)

---

### Summary

`SpiceChunkValidatorActor` defers incoming `SpiceChunkStateWitness` and `SpiceChunkContractAccesses` messages into a plain, unbounded `HashMap` (`waiting_for_block`) when the referenced block is not yet in the store. No size limit, no distance-from-head check, and no eviction policy are applied before insertion. The non-spice path enforces all three of these invariants; the Spice path explicitly acknowledges they are missing via TODO comments. This is the nearcore-native analog of the external report's missing timeout: instead of a missing timeout on a network fetch, the missing bound is on a deferred witness accumulation buffer that holds objects up to 64 MiB each.

---

### Finding Description

`SpiceChunkValidatorActor` holds:

```rust
waiting_for_block:
    HashMap<CryptoHash, (Vec<SpiceChunkStateWitness>, Vec<SpiceChunkContractAccesses>)>,
``` [1](#0-0) 

This is initialized as a plain `HashMap::new()` with no capacity bound. [2](#0-1) 

When `process_chunk_state_witness` determines the block is not yet available (`WitnessProcessingReadiness::NotReady`), the witness is pushed into this map **before any signature or size validation**:

```rust
// TODO(spice): Implement additional checks (size limit, distance to head)
// before adding witness to `waiting_for_block`. See non-spice handle_orphan_witness().
self.waiting_for_block.entry(chunk_id.block_hash).or_default().0.push(witness);
``` [3](#0-2) 

The same pattern applies to `SpiceChunkContractAccesses` when the block is not in the store — the accesses are deferred without signature validation:

```rust
// TODO(spice): Implement additional checks (size limit, distance to head)
// before adding accesses to `waiting_for_block`. See non-spice handle_orphan_witness().
self.waiting_for_block.entry(chunk_id.block_hash).or_default().1.push(accesses);
``` [4](#0-3) 

By contrast, the non-spice `ChunkValidationActor` enforces three explicit invariants before accepting an orphan witness:

1. **Distance from head**: `ALLOWED_ORPHAN_WITNESS_DISTANCE_FROM_HEAD: Range<BlockHeight> = 2..6`
2. **Size limit**: `max_orphan_witness_size` — witnesses exceeding this are rejected
3. **Pool capacity**: `OrphanStateWitnessPool` backed by an `LruCache` with a configurable bound [5](#0-4) [6](#0-5) [7](#0-6) 

The Spice path has none of these guards. The `partial_chunk_data` LruCache is bounded at `MAX_PENDING_CHUNKS = 24`, but `waiting_for_block` — which feeds into it — is not: [8](#0-7) 

---

### Impact Explanation

Each `SpiceChunkStateWitness` can be up to `MAX_UNCOMPRESSED_STATE_WITNESS_SIZE` (64 MiB). An attacker who can deliver `SpiceChunkStateWitnessMessage` packets to a validator node can craft witnesses referencing distinct, non-existent block hashes. Each message passes the `witness_processing_readiness` check (block not found → `NotReady`) and is appended to `waiting_for_block` without eviction. Because the block never arrives, the entries are never drained. Memory grows without bound until the validator process is OOM-killed, halting chunk endorsement for all shards the validator tracks.

**Scoped impact**: Validator node crash → missed chunk endorsements → degraded block production liveness. Classified **High**.

---

### Likelihood Explanation

The `SpiceChunkStateWitnessMessage` handler accepts the message and calls `process_chunk_state_witness` with no pre-validation of the witness signature or sender identity: [9](#0-8) 

Signature validation only occurs later, inside `spice_pre_validate_chunk_state_witness`, which is reached only after the block arrives. For witnesses whose block never arrives, validation never runs. Any peer reachable at the network layer can trigger accumulation. The Spice feature is behind `protocol_feature_spice` and not yet enabled on mainnet, but the gap is self-acknowledged in the codebase and will be reachable when the feature activates.

---

### Recommendation

Apply the same three guards the non-spice path uses before inserting into `waiting_for_block`:

1. **Distance-from-head check**: reject witnesses whose `block_hash` height is more than a small constant (e.g., 6) above the current chain head.
2. **Size check**: reject witnesses whose serialized size exceeds a configured maximum (matching `max_orphan_witness_size`).
3. **Bounded container**: replace the plain `HashMap` with an `LruCache` or a structure that enforces a total-entry or total-byte cap, evicting the oldest entries when the limit is reached.

The TODO comments at lines 266 and 522 already point to the non-spice `handle_orphan_witness` as the reference implementation.

---

### Proof of Concept

1. Connect to a Spice-enabled validator node's network port.
2. Craft `SpiceChunkStateWitnessMessage` values with `chunk_id.block_hash` set to distinct random hashes (blocks that do not exist in the validator's store).
3. Send a large number of such messages. Each passes `witness_processing_readiness` with `NotReady` and is appended to `waiting_for_block[block_hash]`.
4. Because the referenced blocks never arrive, `process_waiting_for_block` is never triggered for these keys, and the entries are never removed.
5. Validator process memory grows proportionally to `num_messages × witness_size`, eventually triggering OOM and halting chunk endorsement. [10](#0-9) [11](#0-10)

        &mut self,
        block: Arc<Block>,
        signer: Arc<ValidatorSigner>,
    ) -> Result<(), Error> {
        let block_hash = *block.header().hash();
        let Some((witnesses, contract_accesses)) = self.waiting_for_block.remove(&block_hash)
        else {
            return Ok(());
        };

        // Process deferred contract accesses first so that partial_chunk_data entries
        // have their `missing` set before witnesses try to finalize.
        for accesses in contract_accesses {
            self.handle_spice_contract_accesses(accesses)?;
        }

        let prev_hash = *block.header().prev_hash();
        let prev_block = self.chain_store.get_block(&prev_hash)?;
        if self.core_reader.get_block_execution_results(prev_block.header())?.is_none() {
            tracing::debug!(
                target: "spice_chunk_validator",
                ?prev_hash,
                "process_waiting_for_block: new block is available, but some of the prev block execution results are still missing");
            // Put witnesses back; contract accesses already went through
            // handle_spice_contract_accesses and are now in partial_chunk_data.
            self.waiting_for_block.entry(block_hash).or_default().0 = witnesses;
            return Ok(());
        };
