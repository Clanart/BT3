### Title
Wrong Address Used in `OmniToken::mint()` When `msg` Is Present — Tokens Deposited to Bridge Contract Instead of Intended Recipient - (File: near/omni-token/src/lib.rs)

### Summary

`OmniToken::mint()` accepts an `account_id` parameter representing the intended token recipient. When a `msg` is provided (the destination-chain hook path), the function ignores `account_id` and instead calls `internal_deposit` with `env::predecessor_account_id()` — the bridge contract — before forwarding via `ft_transfer_call`. This is the direct analog of the reported pattern: a user-supplied address parameter is silently replaced by the caller's address.

### Finding Description

In `near/omni-token/src/lib.rs`, the `mint()` function has two branches:

```rust
fn mint(
    &mut self,
    account_id: AccountId,   // intended recipient
    amount: U128,
    msg: Option<String>,
) -> PromiseOrValue<U128> {
    self.assert_controller();

    if let Some(msg) = msg {
        self.token
            .internal_deposit(&env::predecessor_account_id(), amount.into()); // ← BUG: deposits to bridge, not account_id

        self.ft_transfer_call(account_id, amount, None, msg)
    } else {
        self.token.internal_deposit(&account_id, amount.into()); // ← correct path
        PromiseOrValue::Value(amount)
    }
}
``` [1](#0-0) 

When `msg` is `None`, tokens are correctly deposited to `account_id`. When `msg` is `Some`, tokens are deposited to `env::predecessor_account_id()` — the bridge contract — and then `ft_transfer_call` is used to forward them to `account_id`.

The bridge invokes this path from `send_tokens()` in `near/omni-bridge/src/lib.rs`:

```rust
ext_token::ext(token)
    .with_attached_deposit(deposit)
    .with_static_gas(MINT_TOKEN_GAS.saturating_add(ft_transfer_call_gas))
    .mint(
        recipient,
        amount,
        (!msg.is_empty()).then(|| msg.to_string()),
    )
``` [2](#0-1) 

For `internal_deposit(&env::predecessor_account_id(), ...)` to succeed, the bridge contract must have a registered storage account in the OmniToken contract. NEAR's `FungibleToken::internal_deposit` panics if the account is not registered. The `fin_transfer` flow only registers the recipient and fee recipient via `storage_deposit_actions` — it never registers the bridge contract itself in the token. [3](#0-2) 

### Impact Explanation

When a cross-chain transfer includes a non-empty `msg` field and the destination token is a deployed (bridged) OmniToken, the `mint()` call panics because the bridge contract is not registered in the token contract. The `fin_transfer` nonce is already consumed (marked finalized) before `send_tokens` is called: [4](#0-3) 

The transfer is permanently marked as finalized on NEAR, but the user never receives their tokens. The source-chain funds are already locked or burned. This constitutes an **irrecoverable lock of user funds** — matching the "Permanent freezing, irrecoverable lock, or unclaimable settlement" impact class.

Even if the bridge contract happened to be registered, the intermediate deposit to the bridge creates a second risk: if `ft_transfer_call` returns tokens (receiver rejects), `fin_transfer_send_tokens_callback` burns them rather than returning them to the user: [5](#0-4) 

### Likelihood Explanation

Any user bridging tokens to a NEAR address with a non-empty `msg` (destination-chain hook) for a deployed OmniToken triggers this path. The `msg` field is a documented, user-accessible parameter in the bridge API. The bridge contract is not registered in newly deployed OmniToken contracts by default — the `storage_deposit_actions` mechanism only covers the recipient and fee recipient.

### Recommendation

Replace `env::predecessor_account_id()` with `account_id` in the `msg` branch of `mint()`:

```rust
if let Some(msg) = msg {
    self.token
        .internal_deposit(&account_id, amount.into()); // use the supplied account_id

    self.ft_transfer_call(account_id, amount, None, msg)
}
```

This matches the `None` branch behavior and eliminates the need for the bridge contract to hold a storage deposit in every deployed token.

### Proof of Concept

1. User initiates a transfer from EVM to NEAR with a non-empty `msg` (e.g., a DeFi hook payload) for a deployed OmniToken.
2. Relayer calls `fin_transfer` on NEAR. The nonce is consumed and the transfer is marked finalized.
3. `process_fin_transfer_to_near` → `send_tokens` → `ext_token::mint(recipient, amount, Some(msg))` is called.
4. Inside `OmniToken::mint()`, `internal_deposit(&env::predecessor_account_id(), amount)` is called. `predecessor_account_id()` is the bridge contract, which has no storage deposit in the token contract.
5. `internal_deposit` panics. The `mint` cross-contract call fails.
6. `fin_transfer_send_tokens_callback` receives a failed promise. `is_ft_transfer_call = true`, so `is_refund_required` returns true, and the tokens are burned.
7. The transfer nonce is already consumed. The user's source-chain funds are permanently lost. [6](#0-5) [7](#0-6)
