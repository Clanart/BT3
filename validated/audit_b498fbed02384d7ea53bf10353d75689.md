### Title
Missing EIP-712 Domain Chain ID Verification in `SignTypedData` Enables Cross-Chain Signature Replay - (File: `rpc/backend/sign_tx.go`)

### Summary
`SignTypedData` in the JSON-RPC backend signs arbitrary EIP-712 typed data supplied by the caller without verifying that the domain separator's `chainId` matches the node's configured chain ID. Any caller who can reach the `eth_signTypedData` endpoint and knows an address held in the node's keyring can obtain a signature valid on a different chain, enabling cross-chain replay of EIP-712 authorizations.

### Finding Description
In `rpc/backend/sign_tx.go` lines 150–174, `SignTypedData` accepts a caller-supplied `apitypes.TypedData` struct, computes its hash via `apitypes.TypedDataAndHash(typedData)`, and immediately signs the result with the keyring:

```go
sigHash, _, err := apitypes.TypedDataAndHash(typedData)
...
signature, _, err := b.clientCtx.Keyring.SignByAddress(from, sigHash, signingtypes.SignMode_SIGN_MODE_TEXTUAL)
``` [1](#0-0) 

At no point is `typedData.Domain.ChainId` compared against `b.chainID`. The function's only gate is that the requested address exists in the node's keyring.

This is in direct contrast to every other signing path in the codebase:

- `SendRawTransaction` explicitly rejects transactions whose chain ID does not match `b.chainID`:
  ```go
  if tx.Protected() && tx.ChainId().Cmp(b.chainID) != 0 {
      return common.Hash{}, fmt.Errorf("incorrect chain-id; expected %d, got %d", ...)
  }
  ``` [2](#0-1) 

- `LegacyEip712SigVerificationDecorator.VerifySignature` checks that the typed-data chain ID matches the Cosmos chain ID:
  ```go
  if extOpt.TypedDataChainID != signerChainID.Uint64() {
      return errorsmod.Wrap(errortypes.ErrInvalidChainID, "invalid chain-id")
  }
  ``` [3](#0-2) 

- `SendTransaction` rejects mismatched chain IDs when the caller supplies one: [4](#0-3) 

The signing endpoint has no equivalent guard. A caller can set `domain.chainId` to any value (e.g., Ethereum mainnet chain ID `1`) and the node will sign the hash without complaint.

### Impact Explanation
The resulting 65-byte ECDSA signature is cryptographically valid for the target chain specified in the domain. If the same private key controls an account on that chain (which is common — Ethereum-compatible addresses are derived from the same secp256k1 key regardless of chain), the attacker can use the signature to authorize any EIP-712-aware operation on that chain: ERC-20 `permit` approvals, Gnosis Safe transactions, EIP-712-based DEX orders, or any other structured-data authorization. This enables unauthorized fund transfers or account/code mutations on the target chain from the node's keyring account.

This matches the allowed High impact: **EIP-712 authorization, chain-id verification bypass enabling replay, forged execution, or unauthorized account/code mutation**.

### Likelihood Explanation
`SignTypedData` is exposed as `eth_signTypedData` in the `PublicAPI` struct of the `eth` namespace: [5](#0-4) 

The `eth` namespace is enabled by default. The only prerequisite is that the target address is in the node's keyring. Validator and operator addresses are publicly visible in block headers and governance proposals; many Ethermint deployments keep these keys in the keyring for transaction signing. Any caller who can reach the JSON-RPC port — which is commonly exposed on port 8545 — can submit a crafted `TypedData` payload.

### Recommendation
In `SignTypedData`, before computing the hash, validate that `typedData.Domain.ChainId` (converted to `*big.Int`) equals `b.chainID`. If the domain chain ID is absent or mismatched, return an error. This mirrors the chain ID check already present in `SendRawTransaction` and `LegacyEip712SigVerificationDecorator.VerifySignature`.

```go
if typedData.Domain.ChainId != nil {
    domainChainID := typedData.Domain.ChainId.ToInt()
    if domainChainID.Cmp(b.chainID) != 0 {
        return nil, fmt.Errorf("domain chainId %s does not match node chainId %s",
            domainChainID, b.chainID)
    }
}
```

### Proof of Concept

The node runs on chain ID `9000` (`ethermint_9000-1`). The validator address `0xABC` is in the keyring and also controls funds on Ethereum mainnet (chain ID `1`).

1. Attacker calls `eth_signTypedData("0xABC", typedData)` where `typedData.Domain.ChainId = 1` and the message is a USDC `Permit` authorizing the attacker as spender.
2. `SignTypedData` at line 160 calls `apitypes.TypedDataAndHash(typedData)` — the hash encodes `chainId=1` in the domain.
3. At line 166, the keyring signs the hash with no chain ID check.
4. The returned signature is a valid EIP-712 `Permit` signature for Ethereum mainnet.
5. Attacker submits the signature to the USDC contract on Ethereum mainnet, draining `0xABC`'s balance.

The root cause is the absence of any `b.chainID` comparison between lines 160 and 166 of `rpc/backend/sign_tx.go`. [6](#0-5)

// SignTypedData signs EIP-712 conformant typed data
func (b *Backend) SignTypedData(address common.Address, typedData apitypes.TypedData) (hexutil.Bytes, error) {
	from := sdk.AccAddress(address.Bytes())

	_, err := b.clientCtx.Keyring.KeyByAddress(from)
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", address.String())
		return nil, fmt.Errorf("%s; %s", keystore.ErrNoMatch, err.Error())
	}

	sigHash, _, err := apitypes.TypedDataAndHash(typedData)
	if err != nil {
		return nil, err
	}

	// Sign the requested hash with the wallet
	signature, _, err := b.clientCtx.Keyring.SignByAddress(from, sigHash, signingtypes.SignMode_SIGN_MODE_TEXTUAL)
	if err != nil {
		b.logger.Error("keyring.SignByAddress failed", "address", address.Hex())
		return nil, err
	}

	signature[crypto.RecoveryIDOffset] += 27 // Transform V from 0/1 to 27/28 according to the yellow paper
	return signature, nil
}
```
