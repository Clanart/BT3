### Title
`ProofOfInclusion::valid()` Does Not Verify Computed Root Against a Trusted External Root — (`File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate checks only the internal hash-chain consistency of a proof, but its final comparison is a tautology: it compares the computed root against `self.root_hash()`, which is itself derived from the proof's own data. No external trusted root is ever consulted. An attacker who can supply a `ProofOfInclusion` object (e.g., over the DataLayer sync protocol) can craft a structurally consistent but entirely fabricated proof claiming any key is included in any tree, and `valid()` will return `true`.

### Finding Description

**Root cause — tautological final check in `valid()`** [1](#0-0) 

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // ← set to layer.combined_hash
    }

    existing_hash == self.root_hash()      // ← always true (see below)
}
```

`root_hash()` is: [2](#0-1) 

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // ← same field just checked in the loop
    } else {
        self.node_hash
    }
}
```

**Why the final check is always `true`:**

- After the loop, `existing_hash` equals the last `calculated_hash`, which the loop already asserted equals `last_layer.combined_hash`.
- `self.root_hash()` returns `last_layer.combined_hash`.
- Therefore `existing_hash == self.root_hash()` is `last_layer.combined_hash == last_layer.combined_hash` — a tautology in every non-empty case.
- In the empty-layers case, `existing_hash == self.node_hash` and `root_hash()` returns `self.node_hash` — also a tautology.

**What is missing (the analog to the sequencer-status check):**

The function never compares the computed root against an *externally supplied, trusted* root hash. The correct check would be:

```rust
pub fn valid_against(&self, trusted_root: &Hash) -> bool {
    // ... internal chain check ...
    existing_hash == *trusted_root   // compare against external authority
}
```

**Attacker-controlled entry path:**

`ProofOfInclusion` is a `Streamable` type exposed through the Python bindings: [3](#0-2) 

A DataLayer peer or any untrusted source can send a serialized `ProofOfInclusion`. The receiver deserializes it and calls `proof.valid()`. Because `valid()` only checks internal self-consistency, the attacker can craft:

1. An arbitrary `node_hash` (claiming any key/value pair is included).
2. A chain of `ProofOfInclusionLayer` entries whose hashes are self-consistent (trivially constructable by hashing forward from the chosen `node_hash`).

The resulting proof passes `valid()` with `true`, proving inclusion of a key that is not in the tree.

The Python binding exposes `valid()` as the sole named validation method: [4](#0-3) 

### Impact Explanation

**High — DataLayer Merkle proof logic accepts forged inclusion proofs.**

A remote, unprivileged attacker who can deliver a `ProofOfInclusion` object to any DataLayer client can prove the inclusion of an arbitrary key/value pair in any tree root of their choosing. Because `valid()` is the only named validation gate and its final check is a tautology, callers who rely on it (as the function name implies they should) receive no protection against fabricated proofs. This enables forged inclusion/exclusion proofs against DataLayer state, matching the allowed High impact criterion.

### Likelihood Explanation

Medium. The DataLayer sync protocol exchanges proofs between peers. Any peer can send a crafted `ProofOfInclusion`. The only precondition is network access to a DataLayer client that calls `proof.valid()` without separately verifying `proof.root_hash()` against a locally trusted root — a natural and expected usage given the function's name and signature.

### Recommendation

1. **Remove the tautological final check** and replace it with a comparison against a caller-supplied trusted root:

```rust
pub fn valid_against(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // compare against external authority
}
```

2. Deprecate or remove the current `valid()` method, or make it clearly document that it does **not** verify against any trusted root and must not be used as a standalone security gate.

3. Audit all callers (Rust and Python) to ensure they supply and check a trusted root hash.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side};
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};

fn forged_proof_passes_valid() {
    // Attacker chooses an arbitrary "included" node hash
    let fake_node_hash: Hash = [0xAA; 32];
    // Attacker builds a single consistent layer
    let fake_combined = chia_datalayer::calculate_internal_hash(
        &fake_node_hash,
        Side::Left,
        &[0xBB; 32],
    );
    let proof = ProofOfInclusion {
        node_hash: fake_node_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Left,
            other_hash: [0xBB; 32],
            combined_hash: fake_combined,  // self-consistent
        }],
    };
    // valid() returns true — no trusted root was ever consulted
    assert!(proof.valid());
    // proof.root_hash() == fake_combined, which the attacker chose freely
}
```

The proof is entirely fabricated. `valid()` returns `true` because the internal chain is self-consistent. No trusted root is ever checked.
