### Title
Missing Token Registration Check in `init_transfer` Allows Permanent Irrecoverable Lock of User Funds - (File: starknet/src/omni_bridge.cairo)

### Summary
The StarkNet bridge's `init_transfer` function accepts any ERC20 token address without verifying it is registered in the bridge's cross-chain token mapping. When a user initiates a transfer with an unregistered token, the tokens are locked in the StarkNet bridge vault, but the NEAR bridge rejects finalization because the token has no registered decimals entry. With no refund, cancel, or admin-rescue mechanism on the StarkNet side, the user's funds are permanently irrecoverable.

### Finding Description
In `starknet/src/omni_bridge.cairo`, `init_transfer` (lines 281–331) performs only two token-related checks before accepting funds:

```cairo
assert(amount > 0, 'ERR_ZERO_AMOUNT');
assert(fee < amount, 'ERR_INVALID_FEE');
```

It then branches on `is_bridge_token(token_address)` to decide whether to burn or `transfer_from`: [1](#0-0) 

`is_bridge_token` only identifies tokens deployed by the bridge itself (those present in `starknet_to_near_token`): [2](#0-1) 

There is **no check** that an external ERC20 token is registered in the bridge's cross-chain mapping before locking it. The full exploit path:

1. User calls `init_transfer` with an unregistered external ERC20 token.
2. `transfer_from` succeeds — tokens are locked in the bridge contract. [3](#0-2) 
3. `InitTransfer` event is emitted and picked up by the NEAR relayer. [4](#0-3) 
4. The NEAR relayer submits the proof to `fin_transfer`, which calls `fin_transfer_callback`.
5. `fin_transfer_callback` attempts to look up the token's registered decimals and panics: [5](#0-4) 
6. The NEAR transaction reverts; no tokens are minted on NEAR.
7. The StarkNet bridge has **no** `cancel_transfer`, `refund`, or admin rescue function — the locked tokens are permanently irrecoverable.

### Impact Explanation
Permanent irrecoverable lock of user funds in the StarkNet bridge vault. The StarkNet contract provides no mechanism to return locked tokens when the corresponding NEAR-side finalization fails. This matches **Critical: Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

### Likelihood Explanation
Moderate. Any user who attempts to bridge a token that is not yet registered (e.g., a newly listed token, a token pending registration, or a token that was recently delisted from the bridge) will permanently lose their funds. The function provides no on-chain feedback that the token is unregistered before locking funds. The risk is amplified because the function silently accepts any ERC20 token that approves the bridge as a spender.

### Recommendation
Add a token registration check in `init_transfer` before accepting the transfer. The bridge should maintain a mapping of approved external ERC20 tokens (mirroring the NEAR side's `token_decimals` / `token_address_to_id` registries) and assert membership before locking funds:

```cairo
assert(
    self.is_bridge_token(token_address) || self.is_registered_external_token(token_address),
    'ERR_TOKEN_NOT_REGISTERED'
);
```

Additionally, implement a `cancel_transfer` or admin rescue function that allows recovery of locked tokens when the NEAR-side finalization fails, analogous to the NEAR bridge's pending-transfer refund paths.

### Proof of Concept
1. Deploy an arbitrary ERC20 token on StarkNet that is **not** registered in the bridge.
2. Approve the bridge to spend the tokens: `token.approve(bridge_address, amount)`.
3. Call `bridge.init_transfer(unregistered_token_address, amount, fee, 0, near_recipient, "")`.
4. The bridge locks the tokens and emits `InitTransfer` — no revert, no warning.
5. The NEAR relayer submits the event proof to `fin_transfer` on NEAR.
6. `fin_transfer_callback` panics with `ERR_TOKEN_DECIMALS_NOT_FOUND` because the token has no entry in `token_decimals`.
7. The NEAR transaction reverts; the user's tokens remain permanently locked in the StarkNet bridge contract with no recovery path.
