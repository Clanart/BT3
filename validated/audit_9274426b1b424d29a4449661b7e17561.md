### Title
Constructor Failure Not Propagated in `execute_deploy` Syscall — Always Reports Success (`failure_flag=0`) - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` function in `syscall_impls.cairo` calls `deploy_contract` to run a constructor, captures its return data, but unconditionally writes `failure_flag=0` into the syscall response header regardless of whether the constructor succeeded or reverted. This is the StarkNet OS analog of the "unchecked return value from external call" vulnerability class: the OS performs an external sub-execution (the constructor) and silently discards its failure status, always reporting success to the calling contract.

---

### Finding Description

In `execute_deploy` (the handler for the `deploy` syscall in Cairo 1 contracts), the constructor is executed via `deploy_contract`. The function returns `(retdata_size, retdata)`. Immediately after, the OS writes the response header with a hardcoded `failure_flag=0`:

```cairo
// syscall_impls.cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `failure_flag` field in `ResponseHeader` is the mechanism by which the OS communicates syscall success or failure back to the calling contract. Setting it unconditionally to `0` means the calling contract always observes a successful deployment, even when the constructor reverted. The TODO comment explicitly acknowledges this is unimplemented: `"support failures"`.

By contrast, `execute_call_contract` (the `call_contract` syscall handler) correctly propagates the `is_reverted` flag from `contract_call_helper` into the response header:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

The `deploy` syscall is missing this same check. The return value of the constructor execution — specifically whether it reverted — is captured but never used to set `failure_flag`.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A contract that uses the `deploy` syscall and relies on the returned `failure_flag` to determine whether deployment succeeded will always be told it succeeded. A realistic exploit scenario:

1. A factory contract deploys a vault sub-contract via `deploy` syscall, passing initialization parameters in the constructor calldata.
2. The constructor reverts (e.g., due to an invalid parameter, a storage collision, or an out-of-gas condition inside the constructor).
3. The OS writes `failure_flag=0` into the syscall response. The factory contract's Sierra code reads `failure_flag == 0` and concludes the deployment succeeded.
4. The factory contract stores the computed `contract_address` and immediately transfers funds to it (e.g., as an initial deposit).
5. The deployed contract is in an uninitialized or partially-initialized state (constructor state changes were reverted). Funds sent to it may be permanently unrecoverable if the contract's withdrawal logic depends on constructor-initialized storage.

This results in a direct, permanent loss of user funds triggered by a normal user transaction.

---

### Likelihood Explanation

**High.** The entry path is fully unprivileged:

- Any user can submit an `invoke` transaction calling a contract that uses the `deploy` syscall.
- Constructor failures are a normal, expected condition (invalid calldata, resource exhaustion, business-logic guards).
- Factory-and-fund patterns (deploy then transfer) are standard in DeFi protocols on StarkNet.
- The bug is unconditional: every single `deploy` syscall invocation, regardless of constructor outcome, returns `failure_flag=0`. There is no configuration or privilege required to trigger it.

---

### Recommendation

Propagate the constructor's revert status into the `failure_flag` field of the `ResponseHeader`, mirroring the pattern used in `contract_call_helper`:

```cairo
// After deploy_contract returns, check is_reverted:
let (is_reverted, retdata_size, retdata) = deploy_contract(...);
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

This requires `deploy_contract` to return an `is_reverted` flag (analogous to `execute_entry_point`'s return signature). The TODO comment at line 538 already marks this as a known gap. [3](#0-2) 

---

### Proof of Concept

**Attacker-controlled path:**

1. Attacker deploys (or uses an existing) factory contract `F` with logic:
   ```
   fn deploy_and_fund(class_hash, salt, bad_calldata, amount):
       let (addr, success) = deploy(class_hash, salt, bad_calldata)
       assert success == true   // OS always returns true
       transfer(addr, amount)   // funds sent to uninitialized contract
   ```

2. Attacker submits an invoke transaction calling `F.deploy_and_fund` with `bad_calldata` that causes the target constructor to revert.

3. The OS executes `execute_deploy`:
   - `deploy_contract` runs the constructor → constructor reverts → `retdata` contains revert reason.
   - OS writes `ResponseHeader(gas=remaining_gas, failure_flag=0)` unconditionally. [4](#0-3) 

4. Factory contract `F` reads `failure_flag=0`, concludes success, executes `transfer(addr, amount)`.

5. Funds arrive at `addr` whose contract is in an uninitialized state (constructor reverted). Funds are permanently frozen.

**Root cause line:** `syscall_impls.cairo:539` — `failure_flag=0` hardcoded, constructor revert status discarded.
