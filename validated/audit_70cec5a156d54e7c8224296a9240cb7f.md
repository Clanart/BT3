### Title
Missing `omniBridgeChainId` in `deployToken` Borsh Encoding Enables Cross-Chain Signature Replay — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `deployToken` function constructs a Borsh-encoded payload that is signed by the NEAR MPC (`nearBridgeDerivedAddress`). That payload contains only `PayloadType.Metadata`, `token`, `name`, `symbol`, and `decimals` — **no chain identifier**. Because `finTransfer` does bind its payload to `omniBridgeChainId`, the asymmetry is clear and intentional for transfers but was omitted for deployments. Any valid `(signatureData, MetadataPayload)` pair accepted on chain A can be submitted verbatim to chain B whenever both instances share the same `nearBridgeDerivedAddress`, which is the normal multi-chain deployment topology.

---

### Finding Description

In `OmniBridge.deployToken`, the signed digest is:

```
keccak256(
  PayloadType.Metadata (1 byte)
  | encodeString(token)
  | encodeString(name)
  | encodeString(symbol)
  | decimals (1 byte)
)
``` [1](#0-0) 

There is no `omniBridgeChainId` field in `MetadataPayload`: [2](#0-1) 

By contrast, `finTransfer` embeds `omniBridgeChainId` twice in its Borsh encoding, making those signatures chain-specific: [3](#0-2) 

The contract stores `omniBridgeChainId` as a state variable and uses it in `finTransfer`, proving the field exists and is available — it was simply never included in the `deployToken` digest: [4](#0-3) 

The only guard against re-use on the same chain is `!isBridgeToken[nearToEthToken[metadata.token]]`, which is per-instance state and provides no cross-chain protection: [5](#0-4) 

---

### Impact Explanation

An unprivileged relayer who observes a successful `deployToken` call on chain A can immediately replay the identical `signatureData` + `MetadataPayload` on chain B. The ECDSA recovery will succeed (same signer, same payload bytes, no chain binding), and a `BridgeToken` proxy will be deployed and registered in chain B's `nearToEthToken` / `ethToNearToken` mappings.

**Direct consequences:**

1. **Unauthorized token deployment**: A bridged token contract is created on chain B without the NEAR bridge ever authorizing support for that token on chain B.
2. **Permanent blocking of legitimate deployment**: Once `nearToEthToken[metadata.token]` is set, any future legitimate `deployToken` call for the same NEAR token ID on chain B will revert with `ERR_TOKEN_EXIST`, permanently preventing the protocol from deploying the correct token through its normal flow.
3. **Downstream minting risk**: If the NEAR bridge observes the replayed `DeployToken` event and begins routing `finTransfer` messages to the replayed token address on chain B, tokens will be minted against that contract. While those individual mints are backed by locked NEAR-side assets, the deployment itself was never authorized for chain B, violating the invariant that each chain's token registry is independently authorized.

The "unbacked supply" framing in the question is partially overstated: `finTransfer` is chain-bound and cannot itself be replayed. However, the unauthorized deployment and permanent registry poisoning are concrete, locally testable, and bounty-grade impacts under the "cross-chain replay" category.

---

### Likelihood Explanation

- **Precondition** (shared `nearBridgeDerivedAddress`): This is the standard multi-chain deployment model. The NEAR MPC key is chain-agnostic; all EVM deployments verify against the same derived address. No special setup is required.
- **Attacker capability**: Any address can call `deployToken`. The attacker only needs to observe a successful call on chain A (public on-chain data) and submit the same calldata on chain B.
- **No privileged access required**: Entirely unprivileged, permissionless path.

---

### Recommendation

Include `omniBridgeChainId` in the `deployToken` Borsh-encoded digest, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

The NEAR-side MPC signing logic must be updated in parallel to include the destination chain ID when producing `deployToken` signatures.

---

### Proof of Concept

1. Deploy two `OmniBridge` instances (`bridgeA`, `bridgeB`) with the same `nearBridgeDerivedAddress` and different `omniBridgeChainId` values (e.g., `1` and `2`).
2. Have the MPC sign a `MetadataPayload` for NEAR token `"token.near"` targeting chain A. Call `bridgeA.deployToken(sig, payload)` — succeeds, token deployed at `addrA`.
3. Submit the **identical** `(sig, payload)` to `bridgeB.deployToken(sig, payload)`.
4. The ECDSA recovery returns `nearBridgeDerivedAddress` (same bytes, same signer), the `ERR_TOKEN_EXIST` check passes (chain B's mapping is empty), and a new `BridgeToken` is deployed at `addrB`.
5. Assert `bridgeB.nearToEthToken("token.near") != address(0)` — the replay succeeded without any authorization from the NEAR bridge for chain B. [6](#0-5)

    function deployToken(
        bytes calldata signatureData,
        BridgeTypes.MetadataPayload calldata metadata
    ) external payable whenNotPaused(PAUSED_DEPLOY_TOKEN) returns (address) {
        if (tokenImplementationAddress == address(0)) {
            revert TokenImplementationNotSet();
        }
        bytes memory borshEncoded = bytes.concat(
            bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
            Borsh.encodeString(metadata.token),
            Borsh.encodeString(metadata.name),
            Borsh.encodeString(metadata.symbol),
            bytes1(metadata.decimals)
        );
        bytes32 hashed = keccak256(borshEncoded);

        if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
            revert InvalidSignature();
        }

        require(
            !isBridgeToken[nearToEthToken[metadata.token]],
            "ERR_TOKEN_EXIST"
        );
        uint8 decimals = _normalizeDecimals(metadata.decimals);

        // slither-disable-next-line reentrancy-no-eth
        address bridgeTokenProxy = address(
            new ERC1967Proxy(
                tokenImplementationAddress,
                abi.encodeWithSelector(
                    BridgeToken.initialize.selector,
                    metadata.name,
                    metadata.symbol,
                    decimals
                )
            )
        );

        deployTokenExtension(
            metadata.token,
            bridgeTokenProxy,
            decimals,
            metadata.decimals
        );

        emit BridgeTypes.DeployToken(
            bridgeTokenProxy,
            metadata.token,
            metadata.name,
            metadata.symbol,
            decimals,
            metadata.decimals
        );

        isBridgeToken[address(bridgeTokenProxy)] = true;
        ethToNearToken[address(bridgeTokenProxy)] = metadata.token;
        nearToEthToken[metadata.token] = address(bridgeTokenProxy);

        return bridgeTokenProxy;
    }
```
