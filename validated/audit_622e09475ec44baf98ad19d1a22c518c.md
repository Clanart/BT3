### Title
Hint-Supplied `gas_prices_hash` and `header_commitments` Are Never Cairo-Asserted in Block Hash Computation — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/block_hash.cairo`)

---

### Summary

`get_block_hashes` in `block_hash.cairo` feeds four critical inputs — `previous_block_hash`, `header_commitments` (which bundles `transaction_commitment`, `event_commitment`, `receipt_commitment`, `state_diff_commitment`, and `packed_lengths`), `gas_prices_hash`, and `starknet_version` — exclusively from a single hint (`%{ GetBlockHashes %}`). No Cairo `assert` statement constrains any of these values. Because hints are not part of the STARK proof, a prover can supply arbitrary values for all four locals, producing a valid proof that commits a structurally well-formed but semantically incorrect block hash to L1.

This is the direct StarkNet analog of the deprecated Chainlink `latestAnswer` pattern: an external data source is consumed without any on-chain validity check, and the result is silently embedded in a security-critical output.

---

### Finding Description

In `get_block_hashes`:

```cairo

    previous_block_hash: felt, new_block_hash: felt

    alloc_locals;
    local previous_block_hash;
    // Currently, the header commitments and gas prices are not computed by the OS.
    // TODO(Yoni, 1/1/2027): compute the header commitments and gas prices.
    local header_commitments: BlockHeaderCommitments*;

    local starknet_version;

        block_info=block_info,

        gas_prices_hash=gas_prices_hash,
        state_root=state_root,
        previous_block_hash=previous_block_hash,

    %{ CheckBlockHashConsistency %}   // ← also a hint, not a Cairo assertion

    return (previous_block_hash=previous_block_hash, new_block_hash=block_hash);

``` [1](#0-0) 

The `%{ CheckBlockHashConsistency %}` call is a Python hint executed by the VM runner; it does not generate any constraint in the proof. The only verified input to `calculate_block_hash` is `state_root` (computed by the OS state-update machinery) and the fields of `block_info` (block number, timestamp, sequencer address). Every other field fed into the Poseidon hash — including `gas_prices_hash` and all five sub-fields of `header_commitments` — is unconstrained.

The resulting `new_block_hash` is then serialized directly into the OS output:

```cairo
serialize_word(os_output_header.new_block_hash);
``` [2](#0-1) 

The `OsOutputHeader` struct itself carries a comment acknowledging the absence of enforcement:

```cairo
// Currently, the block hash is not enforced by the OS.
new_block_hash: felt,
``` [3](#0-2) 

And `os_utils.cairo` confirms the same for `prev_block_hash`:

```cairo

### Impact Explanation

**Commitment/output flaw → Direct loss of funds / Unintended chain split.**

1. **`gas_prices_hash` manipulation.** The block hash committed to L1 encodes the gas prices for the block. If the prover supplies a `gas_prices_hash` that does not correspond to the actual per-unit prices used during execution, the L1 state-update contract accepts a block whose fee schedule is provably inconsistent with what users were charged. A sequencer can systematically overstate gas prices in the committed hash while charging users at a lower rate (or vice versa), creating an undetectable discrepancy between the on-chain fee record and actual fee transfers — a direct loss of funds for users or the protocol.

2. **`header_commitments.transaction_commitment` / `event_commitment` manipulation.** These fields are the Merkle/Poseidon roots over all transactions and events in the block. Because they are unconstrained, the prover can commit a block hash whose transaction commitment does not match the transactions actually executed. L1 consumers that rely on the block hash to verify transaction inclusion proofs will accept forged proofs, enabling censorship or misrepresentation of the transaction record.

3. **`previous_block_hash` manipulation.** The previous block hash is also hint-supplied without assertion. A prover that supplies a wrong `previous_block_hash` produces a block hash that does not chain correctly from the canonical tip. If the L1 does not independently re-derive and check this field (the OS explicitly delegates that responsibility to "the consumer"), two competing valid proofs with different `previous_block_hash` values can both pass L1 verification, causing an unintended chain split.

---

### Likelihood Explanation

The prover/sequencer controls all hints. No external attacker capability is required beyond operating the sequencer — which in a decentralized StarkNet is any validator. The OS is the mechanism that is supposed to constrain the sequencer; the absence of assertions here means the constraint simply does not exist. The TODO comments dated 2027 confirm this is a known, unresolved gap, not a transient oversight.

---

### Recommendation

Replace the hint-only pattern with Cairo assertions that cryptographically bind each field:

1. **`gas_prices_hash`**: Compute the Poseidon hash of the block's actual gas price vector inside the OS (analogous to how `state_root` is computed) and assert equality with the hint-supplied value before passing it to `calculate_block_hash`.

2. **`header_commitments`**: Compute `transaction_commitment`, `event_commitment`, `receipt_commitment`, and `state_diff_commitment` from the actual transaction and event data processed during block execution, then assert each sub-field.

3. **`previous_block_hash`**: Assert that the hint-supplied `previous_block_hash` equals the value stored in the block-hash contract for `block_number - 1`, using the same `read_block_hash_from_storage` primitive already used in `check_proof_facts`.

4. **`old_block_hash` in `write_block_number_to_block_hash_mapping`**: Output the written hash in the OS output so the L1 can verify it, as the TODO at line 65 acknowledges.

Until these assertions are added, the `%{ CheckBlockHashConsistency %}` hint provides no security guarantee and should not be relied upon.

---

### Proof of Concept

1. Operator runs the StarkNet OS prover with a modified hint implementation for `GetBlockHashes` that returns:
   - `gas_prices_hash` = Poseidon hash of a fabricated gas price vector (e.g., 10× the actual prices)
   - `header_commitments.transaction_commitment` = arbitrary felt
   - `previous_block_hash` = hash of a non-canonical predecessor block

2. The Cairo program executes `calculate_block_hash` with these values alongside the legitimately computed `state_root` and `block_info` fields.

3. No `assert` statement in `get_block_hashes` fires; `%{ CheckBlockHashConsistency %}` is a no-op in the proof.

4. The STARK proof is valid. The OS output serialized by `serialize_output_header` contains the manipulated `new_block_hash`.

5. The L1 verifier accepts the proof and records the manipulated block hash, permanently committing incorrect gas price and transaction commitment data to the canonical chain state. [1](#0-0) [6](#0-5) [7](#0-6)
