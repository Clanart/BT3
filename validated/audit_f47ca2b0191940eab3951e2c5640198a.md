### Title
Integer Wrapping in `check_time_locks()` Bypasses Relative Timelock Conditions Under Legacy `nowrap=false` Mode - (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary

`check_time_locks()` in `crates/chia-consensus/src/check_time_locks.rs` contains two distinct arithmetic paths controlled by the `nowrap` boolean parameter. When `nowrap=false` (the legacy path), the function uses `wrapping_add` for all relative timelock arithmetic. This causes integer overflow to silently wrap around, producing semantically incorrect comparison results. For `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE`, overflow causes the enforced deadline to wrap to a small value, making the timelock check trivially pass and allowing a coin to be spent immediately regardless of its intended lock duration. For `ASSERT_BEFORE_HEIGHT_RELATIVE` and `ASSERT_BEFORE_SECONDS_RELATIVE`, overflow causes the opposite: a valid spend is incorrectly rejected. Both behaviors are confirmed by the codebase's own test suite.

### Finding Description

`check_time_locks()` enforces relative timelock conditions by computing `confirmed_block_index + height_relative` (or `timestamp + seconds_relative`) and comparing against the current block height or timestamp. The function has two arithmetic modes:

- `nowrap=true`: uses `saturating_add`, which clamps at the type maximum — semantically correct.
- `nowrap=false`: uses `wrapping_add`, which silently wraps on overflow — semantically incorrect.

**`ASSERT_HEIGHT_RELATIVE` bypass (nowrap=false):**

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

If `confirmed_block_index + height_relative` overflows `u32`, `wrapping_add` produces a small value (e.g., `1u32.wrapping_add(0xFFFF_FFFF) = 0`). The comparison `prev_transaction_block_height < 0u32` is always `false`, so the error is never returned and the spend is accepted immediately — bypassing the timelock entirely.

**`ASSERT_SECONDS_RELATIVE` bypass (nowrap=false):**

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
```

Same overflow pattern on `u64`: `timestamp.wrapping_add(u64::MAX)` wraps to a small value, making the check trivially pass.

**`ASSERT_BEFORE_HEIGHT_RELATIVE` incorrect rejection (nowrap=false):**

```rust
} else if prev_transaction_block_height
    >= unspent.confirmed_block_index.wrapping_add(before_height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertBeforeHeightRelativeFailed));
}
```

Overflow wraps the deadline to a small value; `prev_transaction_block_height >= 0` is always `true`, so a valid spend is incorrectly rejected.

The codebase's own test suite explicitly documents all four overflow cases:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),  // <-- bypass confirmed
)]
```

The `height_relative` value `0xFFFFFFFF` is a valid parsed value: `sanitize_uint` with a 4-byte limit accepts it as `SanitizedUint::Ok(0xFFFF_FFFF)`, not a `PositiveOverflow`.

### Impact Explanation

When `nowrap=false` is active, an attacker who controls a coin's puzzle can output `ASSERT_HEIGHT_RELATIVE` with a value chosen to overflow when added to the coin's `confirmed_block_index`. The resulting wrapped deadline is a small number that the current block height already exceeds, so `check_time_locks` returns `Ok(())` and the spend is accepted. A coin intended to be locked for billions of blocks can be spent in the very next block. This is a **timelock condition bypass enabling unauthorized spend acceptance**, matching the High impact category.

The `ASSERT_BEFORE_*` wrapping direction causes the inverse: a valid spend is incorrectly rejected, which can cause consensus divergence between nodes running different `nowrap` modes on the same block.

### Likelihood Explanation

- The `nowrap` parameter is a caller-controlled boolean exposed through the Python binding `py_check_time_locks` and the type stub `check_time_locks(..., nowrap: bool) -> Optional[int]`. Both `True` and `False` are exercised in the test suite, indicating both are used in production.
- The attacker-controlled entry path is a CLVM puzzle that outputs `ASSERT_HEIGHT_RELATIVE` with a large `u32` value (e.g., `0xFFFF_FFF6` when the coin is confirmed at block 10, causing `10 + 0xFFFF_FFF6 = 0` after wrapping). This requires no privileged access — any coin owner can deploy such a puzzle.
- The overflow threshold is predictable: given a known `confirmed_block_index`, the attacker can compute the exact `height_relative` value needed to wrap to 0.

### Recommendation

Replace all `wrapping_add` calls in `check_time_locks` with `saturating_add`, making the `nowrap` parameter unconditionally use safe arithmetic. If backward compatibility with pre-fork blocks requires the wrapping behavior, document the exact block height at which `nowrap` transitions and ensure the legacy path is never reachable for attacker-controlled spends. Alternatively, reject any `height_relative` or `seconds_relative` value that would overflow when added to the coin's birth height/timestamp at condition-parse time in `parse_args`.

### Proof of Concept

**Attacker setup:**
- Coin confirmed at block `N = 10` (`confirmed_block_index = 10`)
- Puzzle outputs condition: `ASSERT_HEIGHT_RELATIVE 0xFFFF_FFF6`
  - `sanitize_uint(4 bytes)` accepts this as `SanitizedUint::Ok(0xFFFF_FFF6)` → stored as `height_relative = Some(0xFFFF_FFF6)`

**At spend time (block 15, `nowrap=false`):**
```
wrapping_add(10u32, 0xFFFF_FFF6u32) = 0u32   // overflows
15u32 < 0u32  →  false                        // check does NOT fire
→ Ok(())  →  spend accepted
```

**With `nowrap=true` (correct behavior):**
```
saturating_add(10u32, 0xFFFF_FFF6u32) = u32::MAX
15u32 < u32::MAX  →  true
→ Err(AssertHeightRelativeFailed)  →  spend rejected
```
