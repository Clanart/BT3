### Title
Nonce-Free `meta_tx_v0` Hash Enables Permanent Replay of Signed Meta-Transactions — (`File: execution/syscall_impls.cairo`, `transaction_hash/transaction_hash.cairo`)

---

### Summary

The `meta_tx_v0` syscall computes a transaction hash that does not include any nonce or freshness indicator. Combined with the OS explicitly skipping nonce checking and incrementing for version-0 transactions, any valid `meta_tx_v0` signature can be replayed indefinitely by any party who observed it on-chain, causing permanent, repeated execution of the signed operation and direct loss of funds.

---

### Finding Description

`execute_meta_tx_v0` in `execution/syscall_impls.cairo` constructs a synthetic version-0 transaction context for the target contract's `__execute__` entry point. The transaction hash it uses is computed by `compute_meta_tx_v0_hash`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // <-- no nonce, no counter
        additional_data=cast(0, felt*),
    );
    return tx_hash;
}
```

The hash covers only `(prefix, version=0, contract_address, selector, calldata_hash, max_fee=0, chain_id)`. No nonce, no block number, no counter of any kind is included.

The synthetic `TxInfo` constructed in `execute_meta_tx_v0` hardcodes `nonce=0`:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,          // always zero
    ...
);
```

The OS-level nonce enforcement in `check_and_increment_nonce` explicitly skips version-0 transactions:

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
```

Similarly, `run_validate` skips `__validate__` for version 0:

```cairo
// Do not run "__validate__" for version 0.
if (tx_execution_info.tx_info.version == 0) {
    return ();
}
```

The result is that the `(calldata, signature)` pair presented to the target contract's `__execute__` is permanently valid. The on-chain nonce of the target account is never incremented by the OS for meta_tx_v0 executions, and the hash the account contract sees is always identical for the same calldata. Any account contract that relies on `tx_info.nonce` to prevent replay will see `nonce=0` every time and the stored nonce will still be 0 (never incremented by the OS), so the check passes on every replay.

---

### Impact Explanation

**Critical — Direct loss of funds.**

The `meta_tx_v0` syscall is the protocol's gasless/relayer mechanism. A user signs a meta-transaction (e.g., `transfer(recipient, amount)`) and a relayer submits it. Once the transaction appears on-chain, any observer can extract the `(calldata, signature)` pair and replay it in any future block. Because the hash is identical and the OS never increments the nonce, the target account's `__execute__` will accept the replayed signature and execute the transfer again. This can be repeated until the account is drained.

---

### Likelihood Explanation

High. The `meta_tx_v0` syscall is a first-class protocol feature reachable by any unprivileged contract. Every on-chain meta_tx_v0 transaction is publicly observable. An attacker needs only to copy the `(contract_address, selector, calldata, signature)` from any historical transaction and submit a new outer transaction that calls `meta_tx_v0` with those values. No privileged access, key material, or special conditions are required beyond gas to submit the replaying transaction.

---

### Recommendation

Include the target account's current on-chain nonce in the `compute_meta_tx_v0_hash` computation, analogous to how `compute_l1_handler_transaction_hash` includes the L1→L2 nonce as `additional_data`. The OS must then read the target account's nonce from `contract_state_changes`, assert it matches the nonce committed in the hash, and increment it after successful execution — removing the version-0 nonce exemption for the meta_tx path.

---

### Proof of Concept

1. User signs a meta_tx_v0 authorizing `transfer(attacker_address, 1000)` on their account contract. Relayer submits it; it executes successfully.
2. Attacker observes the transaction on-chain and extracts `(contract_address, calldata, signature)`.
3. Attacker deploys a contract that calls `meta_tx_v0(contract_address, __execute__, calldata, signature)`.
4. Because `compute_meta_tx_v0_hash` produces the same hash (no nonce), the account's `__execute__` sees the same `transaction_hash` and `signature`, passes its own signature check, and executes the transfer again.
5. The OS never increments the account nonce (version-0 exemption), so step 4 can be repeated in every block until the account balance is zero.
