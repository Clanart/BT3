### Title
Silent Truncation in `BlsCache::py_aggregate_verify` Due to Missing Length Parity Check - (File: `crates/chia-bls/src/bls_cache.rs`)

### Summary

`BlsCache::py_aggregate_verify` (the Python-facing binding for BLS cache-accelerated aggregate signature verification) zips two separately-supplied lists — `pks` and `msgs` — without first asserting they have equal length. Rust's `zip` silently stops at the shorter iterator, causing the verification to silently skip any trailing `(pk, msg)` pairs. This is a direct analog of the Linea missing-length-check bug: iterating over two parallel collections without validating their sizes match.

### Finding Description

In `crates/chia-bls/src/bls_cache.rs`, the Python-binding method `py_aggregate_verify` collects `pks` and `msgs` into two independent `Vec`s and then zips them:

```rust
Ok(self.aggregate_verify(pks.into_iter().zip(msgs), sig))
```

No length equality check is performed before the zip. [1](#0-0) 

The same pattern appears in `py_evict`:

```rust
self.evict(pks.into_iter().zip(msgs));
``` [2](#0-1) 

By contrast, `AugSchemeMPL::aggregate_verify` in `wheel/src/api.rs` **does** perform the check and raises a `PyRuntimeError` on mismatch:

```rust
if pks.len() != msgs.len() {
    return Err(PyRuntimeError::new_err(
        "aggregate_verify expects the same number of public keys as messages",
    ));
}
``` [3](#0-2) 

This creates a cross-API behavioral inconsistency: `AugSchemeMPL.aggregate_verify` rejects mismatched inputs, while `BLSCache.aggregate_verify` silently verifies only the shorter prefix of pairs.

### Impact Explanation

BLS aggregate signature verification checks:

```
e(sig, G2) == e(pk1, H(aug_msg1)) * e(pk2, H(aug_msg2)) * ... * e(pkN, H(aug_msgN))
```

If `len(pks) > len(msgs)`, the trailing public keys are silently dropped from the pairing product. The verification then passes for a signature that only covers the shorter prefix of `(pk, msg)` pairs — the remaining public keys are never required to have signed anything. An attacker who can influence the construction of the `pks` list (e.g., by appending extra public keys) while keeping `msgs` shorter can cause the cache-accelerated verifier to accept a signature that does not cover all required signers.

This matches the allowed High impact: **Signature validation bypass enables unauthorized spend acceptance**.

### Likelihood Explanation

The Python full node (chia-blockchain) calls `BLSCache.aggregate_verify(pks, msgs, sig)` with separately-constructed lists derived from AGG_SIG conditions. Under normal operation these lists are the same length. However:

1. The inconsistency with `AugSchemeMPL.aggregate_verify` means callers have no reason to expect different behavior.
2. Any future refactor or bug in the Python full node that produces a length mismatch will silently degrade verification rather than raising an error.
3. The `py_evict` path with the same pattern can silently fail to evict cache entries, leaving stale pairings that could be exploited in combination.

Likelihood is **Medium** — not immediately exploitable from a raw spend bundle alone, but the missing guard removes a critical safety net at the Python/Rust boundary.

### Recommendation

Add a length parity check in `py_aggregate_verify` and `py_evict`, mirroring the guard already present in `AugSchemeMPL::aggregate_verify`:

```rust
pub fn py_aggregate_verify(
    &self,
    pks: &Bound<'_, PyList>,
    msgs: &Bound<'_, PyList>,
    sig: &Signature,
) -> PyResult<bool> {
    if pks.len() != msgs.len() {
        return Err(PyValueError::new_err(
            "aggregate_verify requires the same number of public keys and messages",
        ));
    }
    // ... existing collection and zip logic
}
```

Apply the same guard to `py_evict`. [2](#0-1) 

### Proof of Concept

```python
from chia_rs import BLSCache, G1Element, G2Element, PrivateKey, AugSchemeMPL

seed = b"a" * 32
sk1 = AugSchemeMPL.key_gen(seed)
sk2 = AugSchemeMPL.key_gen(b"b" * 32)
pk1 = sk1.get_g1()
pk2 = sk2.get_g1()

msg1 = b"hello"
msg2 = b"world"

# Aggregate signature covers BOTH (pk1, msg1) and (pk2, msg2)
sig = AugSchemeMPL.aggregate([
    AugSchemeMPL.sign(sk1, msg1),
    AugSchemeMPL.sign(sk2, msg2),
])

# AugSchemeMPL correctly verifies both pairs
assert AugSchemeMPL.aggregate_verify([pk1, pk2], [msg1, msg2], sig)

# Now craft a signature that only covers (pk1, msg1)
partial_sig = AugSchemeMPL.sign(sk1, msg1)

cache = BLSCache()

# Supply pks=[pk1, pk2] but msgs=[msg1] only — zip truncates to [(pk1, msg1)]
# partial_sig is valid for [(pk1, msg1)] alone, so this returns True
# even though pk2 never signed anything
result = cache.aggregate_verify([pk1, pk2], [msg1], partial_sig)
print(result)  # True — pk2's signature requirement is silently dropped
```

The `BLSCache.aggregate_verify` call returns `True` for a partial signature that does not cover `pk2`, because the mismatched-length zip silently discards the `pk2` pairing requirement. [1](#0-0)
