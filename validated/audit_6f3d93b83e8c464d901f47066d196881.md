### Title
Non-EVM-native Bank Balances Permanently Orphaned on Contract Selfdestruct — (File: x/evm/statedb/statedb.go)

### Summary
When an EVM contract selfdestructs, `StateDB.Commit()` burns only the EVM-native denom balance and then deletes the auth-module account. Any non-EVM-native Cosmos bank balances (IBC tokens, CosmWasm bridge tokens, etc.) held by the contract address are never drained. After `DeleteAccount` removes the auth account, those bank balances remain in the bank module permanently associated with an address that no longer has an auth account, making them irrecoverable.

### Finding Description
In `StateDB.Commit()`, the selfdestruct branch explicitly handles only the EVM denom:

```go
// Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.
if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {

    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct balance")
    }
}
if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil {

`DeleteAccount` in `keeper/statedb.go` removes the auth-module account and clears EVM storage, but it never touches the bank module:

```go
// clear storage
k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
    k.SetState(ctx, addr, key, nil)
    return true
})
// remove auth account
k.accountKeeper.RemoveAccount(ctx, acct)
``` [2](#0-1) 

After `RemoveAccount` executes, the Cosmos bank module still holds any non-EVM-native balances keyed to that address. Because the auth account is gone, no transaction can be signed from that address, no EVM code exists to call, and no standard recovery path exists. The tokens are permanently orphaned.

The analog to the external report is direct: just as `UniswapMarketMaker.sol` changes the `incoming` token without first removing LP positions (leaving LP tokens stranded), Ethermint's selfdestruct path changes the account state (deletes the auth account) without first draining all associated bank balances, leaving non-EVM-native tokens stranded.

### Impact Explanation
Any non-EVM-native Cosmos bank tokens (IBC denoms, CosmWasm-bridged assets, etc.) held by a selfdestructed contract address are permanently locked. The bank module retains the balance record, but with no auth account the funds can never be spent, transferred, or burned through any standard path. This constitutes permanent, irrecoverable loss of Cosmos bank funds — a mis-accounting of valid user funds through the EVM state transition path.

### Likelihood Explanation
Cosmos/Ethermint chains routinely receive IBC token transfers to EVM contract addresses (e.g., DeFi contracts that accept IBC assets). Any such contract that calls `SELFDESTRUCT` (or is selfdestructed via EIP-6780 in the same transaction it was created) will orphan those balances. The entry path requires only a normal EVM transaction — no privileged role is needed. The scenario is realistic wherever IBC tokens interact with EVM contracts.

### Recommendation
Before calling `DeleteAccount`, enumerate **all** bank module balances for the contract address (not just the EVM denom) and either burn them or transfer them to a designated recovery address (e.g., the fee collector or a governance-controlled module account). The `bankKeeper.GetAllBalances(ctx, cosmosAddr)` call provides the full balance set; each non-zero denom should be drained inside the same `CacheContext` used for the EVM-denom burn and `DeleteAccount`, so the operation remains atomic.

### Proof of Concept
1. Deploy a Solidity contract `Victim` on an Ethermint chain that supports IBC.
2. Via IBC, transfer 1000 `uatom` (or any non-EVM denom) to the `Victim` contract address.
3. Call `Victim.destroy()` which executes `SELFDESTRUCT` (or deploy + selfdestruct in one tx under EIP-6780).
4. `StateDB.Commit()` burns the EVM-denom balance and calls `DeleteAccount`.
5. Query `bankKeeper.GetBalance(ctx, victimAddr, "uatom")` — it still returns 1000.
6. Query `accountKeeper.GetAccount(ctx, victimAddr)` — it returns `nil`.
7. The 1000 `uatom` are permanently orphaned: no auth account exists to sign a spend, no contract code exists to call, and no governance mechanism automatically recovers them. [3](#0-2) [4](#0-3)

func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
