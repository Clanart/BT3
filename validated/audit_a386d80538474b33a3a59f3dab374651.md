### Title
BLOCKHASH Returns Wrong Hash for Out-of-Window Blocks Due to Unvalidated Ring-Buffer Slot Reuse вЂ” (`x/evm/keeper/keeper.go`)

### Summary

`GetHeaderHash` reads a ring-buffer slot using `height % window` but never verifies that the hash stored in that slot actually belongs to `height`. After `window` blocks have elapsed, `SetHeaderHash` overwrites the slot with a newer block's hash, and any subsequent `BLOCKHASH(oldHeight)` call returns that newer hash instead of zero, violating EIP-2935 semantics and enabling contract logic bypass.

---

### Finding Description

**`SetHeaderHash`** stores the current block hash at ring index `ringIndex = uint64(ctx.BlockHeight()) % window`: [1](#0-0) 

**`GetHeaderHash`** reads back using `ringIndex = height % window` and returns whatever is stored there if it is non-zero вЂ” with **no check that the stored hash belongs to `height`**: [2](#0-1) 

**`GetHashFn`** calls `GetHeaderHash` and short-circuits immediately on any non-zero result, before the `lower`-bound guard that would otherwise reject out-of-window heights: [3](#0-2) 

The `lower` bound (lines 114вЂ“119) is only reached when `GetHeaderHash` returns zero. Because the ring buffer returns a non-zero (but wrong) hash, the `lower` guard is never evaluated for wrapped heights.

**Concrete wrap-around scenario** (default `window = 8191`):

| Step | Action | Ring slot `N % 8191` |
|------|--------|----------------------|
| Block N | `SetHeaderHash` stores `hash(N)` | `hash(N)` |
| Block N+8191 | `SetHeaderHash` stores `hash(N+8191)` | `hash(N+8191)` в†ђ overwrites |
| EVM tx at block N+8191+k | `BLOCKHASH(N)` в†’ `GetHeaderHash(N)` в†’ reads slot `N % 8191` | returns `hash(N+8191)` вњ— |

EIP-2935 requires `BLOCKHASH(N)` to return `0` when `N < currentBlock - window`. Instead it returns the hash of a different, more recent block.

---

### Impact Explanation

Any smart contract that uses `blockhash(N) != 0` as a guard вЂ” e.g., to verify that block N is within the recent window for a commit-reveal scheme, randomness seed, or replay-protection check вЂ” will receive a non-zero value for an arbitrarily old block. The returned value is a real block hash (of block N + kВ·window), so it passes `!= 0` checks and can satisfy `bytes32` comparisons if the attacker can predict or influence which block hash ends up in the slot. This constitutes an EVM state-transition correctness violation that enables contract logic bypass.

---

### Likelihood Explanation

The precondition is simply that the chain has been running for more than `window` (8191) blocks with the `HistoryStorageAddress` contract deployed. On any live Ethermint chain that has passed the EIP-2935 upgrade, this condition is permanently satisfied. No privileged access is required; any unprivileged caller can trigger `BLOCKHASH` via a normal EVM transaction or contract call.

---

### Recommendation

In `GetHeaderHash`, after reading the ring-buffer slot, validate that the stored hash belongs to the requested height. EIP-2935 specifies that the contract storage key encodes the **block number**, not just the ring index. The fix is to store and check a `(height, hash)` pair, or to explicitly reject requests where `currentHeight - height >= window` before reading the ring buffer:

```go
// In GetHeaderHash, before returning the ring-buffer result:
currentHeight := uint64(ctx.BlockHeight()) //nolint:gosec
if currentHeight > height && currentHeight-height >= window {
    // Block is outside the serve window; ring slot has been overwritten.
    // Fall through to KV fallback (which will also return zero for old blocks).
} else {
    hash := k.GetState(ctx, ethparams.HistoryStorageAddress, key)
    if hash.Cmp(common.Hash{}) != 0 {
        return hash
    }
}
```

Similarly, `GetHashFn` should apply the `lower` bound check **before** calling `GetHeaderHash`, not only on the staking fallback path.
