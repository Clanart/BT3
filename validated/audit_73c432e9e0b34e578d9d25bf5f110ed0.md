### Title
Unprotected `execute_meta_tx_v0` Syscall Allows Any Contract to Execute `__execute__` on Behalf of Any Account Without Authorization — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_meta_tx_v0` function in the StarkNet OS allows **any contract** to invoke the `__execute__` entry point of **any other account contract** with fully attacker-controlled calldata and signature, without any authorization check and without calling `__validate__` first. This is a direct analog to the "Unprotected `executeModule`" vulnerability class: just as `executeModule` allowed anyone to execute any module with any parameters on behalf of any user, `execute_meta_tx_v0` allows any contract to execute `__execute__` on behalf of any account with arbitrary parameters.

---

### Finding Description

In `execute_syscalls.cairo`, the syscall dispatch loop handles `META_TX_V0_SELECTOR` with no restriction on the calling contract: [1](#0-0) 

This routes to `execute_meta_tx_v0` in `syscall_impls.cairo`. The function reads the target `contract_address` and `calldata` entirely from the syscall request (i.e., from the calling contract's memory), with no check that the caller has any relationship to the target: [2](#0-1) 

The only selector restriction is that `selector` **must** be `EXECUTE_ENTRY_POINT_SELECTOR` — meaning the syscall is hardwired to call `__execute__` on the victim: [3](#0-2) 

A new `TxInfo` is constructed with `version=0`, `nonce=0`, `account_contract_address=contract_address` (attacker-chosen), and `signature=request.signature_*` (attacker-provided): [4](#0-3) 

The execution context is built with `caller_address=ORIGIN_ADDRESS` (0), making it appear as if the OS itself is the caller: [5](#0-4) 

`contract_call_helper` is then called, which directly invokes `__execute__` on the victim account: [6](#0-5) 

**Critically, `__validate__` is never called.** The OS's normal transaction flow always calls `run_validate` before `__execute__`: [7](#0-6) 

But `execute_meta_tx_v0` skips this entirely. The OS explicitly blocks `EXECUTE_ENTRY_POINT_SELECTOR` in the normal `call_contract` syscall path for exactly this reason: [8](#0-7) 

`execute_meta_tx_v0` bypasses this protection with no compensating authorization check.

---

### Impact Explanation

**Critical — Direct loss of funds.**

Standard StarkNet account contracts (OpenZeppelin, Argent, Braavos) follow the pattern where `__validate__` performs signature verification and `__execute__` performs execution, trusting that `__validate__` already ran. When `execute_meta_tx_v0` calls `__execute__` directly with an attacker-supplied signature and calldata, the account contract's `__execute__` will execute the attacker's arbitrary calls (e.g., ERC-20 transfers) without any signature verification having occurred.

Additionally, account contracts that attempt to guard `__execute__` by checking `caller_address == 0` (ORIGIN_ADDRESS) as a proxy for "called by the OS" are also bypassed, because `execute_meta_tx_v0` always sets `caller_address = ORIGIN_ADDRESS` regardless of who called the syscall.

---

### Likelihood Explanation

**High.** The attack requires only:
1. An attacker account (any valid StarkNet account).
2. A deployed malicious contract that issues the `meta_tx_v0` syscall.
3. A victim account whose `__execute__` does not re-verify the signature (the standard pattern for all major account implementations).

No privileged access, leaked keys, or operator cooperation is required. The attacker pays normal gas fees for their own transaction. The victim need not take any action.

---

### Recommendation

The OS must enforce at least one of the following before `execute_meta_tx_v0` calls `__execute__` on the target:

1. **Call `__validate__` on the target account** using the meta-transaction hash and the provided signature before calling `__execute__`, mirroring the normal transaction flow.
2. **Verify the signature cryptographically at the OS level** against `meta_tx_hash` before dispatching to `__execute__`.
3. **Restrict the syscall** so that `contract_address` in the request must equal the calling contract's own address (i.e., a contract can only issue a meta-tx for itself, not for arbitrary victims).

Option 1 is most consistent with the existing OS security model.

---

### Proof of Concept

**Setup:**
- `victimAccount` is a standard account contract at address `V` with tokens.
- `attackerAccount` is the attacker's own account.
- `maliciousContract` is a contract deployed by the attacker.

**Attack flow:**

1. Attacker submits an invoke transaction from `attackerAccount` calling `maliciousContract.__execute__`.

2. `maliciousContract` issues the `meta_tx_v0` syscall with:
   ```
   request.contract_address = V          // victim's account
   request.selector         = EXECUTE_ENTRY_POINT_SELECTOR
   request.calldata         = [transfer(attacker, victim_balance)]  // arbitrary calls
   request.signature_start  = []         // empty or garbage
   request.signature_end    = []
   ```

3. The OS executes `execute_meta_tx_v0`:
   - Computes `meta_tx_hash = hash(V, __execute__, calldata, chain_id)`.
   - Constructs `new_tx_info` with `version=0`, `nonce=0`, `account_contract_address=V`, `signature=[]`, `transaction_hash=meta_tx_hash`.
   - Sets `caller_address = ORIGIN_ADDRESS (0)`.
   - Calls `contract_call_helper` → `select_execute_entry_point_func` → victim's `__execute__`.

4. Victim's `__execute__` runs with `tx_info.account_contract_address = V` and the attacker's calldata. Since `__validate__` was never called and `__execute__` does not re-verify the signature, the transfer executes.

5. Victim's tokens are transferred to the attacker. `__validate__` was never invoked at any point.
