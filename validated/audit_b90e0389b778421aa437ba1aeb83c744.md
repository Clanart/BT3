### Title
`deploy_token` Bypasses Pause Mechanism on Solana — (File: `solana/programs/bridge_token_factory/src/lib.rs`)

### Summary
The Solana `bridge_token_factory` program's `deploy_token` instruction has no pause check, while every other state-changing instruction checks pause flags. Additionally, the `ALL_PAUSED` bitmask does not include a deploy-token bit, so even a full admin pause cannot prevent new token deployments on Solana.

### Finding Description
In `solana/programs/bridge_token_factory/src/lib.rs`, the `deploy_token` function (lines 66–76) executes unconditionally:

```rust
pub fn deploy_token(
    ctx: Context<DeployToken>,
    data: SignedPayload<DeployTokenPayload>,
) -> Result<()> {
    msg!("Deploying token");
    data.verify_signature((), &ctx.accounts.common.config.derived_near_bridge_address)?;
    ctx.accounts.initialize_token_metadata(data.payload)?;
    Ok(())
}
``` [1](#0-0) 

Every other state-changing instruction checks a pause flag before proceeding:

```rust
// finalize_transfer
require!(ctx.accounts.common.config.paused & FINALIZE_TRANSFER_PAUSED == 0, ...);
// init_transfer
require!(ctx.accounts.common.config.paused & INIT_TRANSFER_PAUSED == 0, ...);
``` [2](#0-1) [3](#0-2) 

The root cause is compounded by the constants file, where `ALL_PAUSED` is defined as only the bitwise OR of `INIT_TRANSFER_PAUSED` and `FINALIZE_TRANSFER_PAUSED` — no `DEPLOY_TOKEN_PAUSED` bit exists:

```rust
pub const INIT_TRANSFER_PAUSED: u8 = 1 << 0;
pub const FINALIZE_TRANSFER_PAUSED: u8 = 1 << 1;
pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED; // = 0x03
``` [4](#0-3) 

When the admin calls `pause()`, it sets `config.paused = ALL_PAUSED = 0x03`. Because there is no deploy-token bit and no check in `deploy_token`, the instruction remains fully callable.

This directly contradicts the behavior on every other chain in the same protocol:

- **EVM** defines `PAUSED_DEPLOY_TOKEN = 1 << 2` and `deployToken` is gated by `whenNotPaused(PAUSED_DEPLOY_TOKEN)`. [5](#0-4) 
- **StarkNet** defines `PAUSE_DEPLOY_TOKEN: u8 = 0x04` and `deploy_token` asserts `!_is_paused(@self, PAUSE_DEPLOY_TOKEN)`. [6](#0-5) [7](#0-6) 

### Impact Explanation
When the bridge admin triggers a pause in response to a security incident, the intent is to halt all bridge operations on Solana. Because `deploy_token` is unguarded, a relayer holding a valid MPC-signed `DeployTokenPayload` (e.g., one generated before the pause, or during a window where the MPC is still responsive) can still register new token mint accounts and token-ID mappings on Solana. This:

1. Undermines the emergency pause by allowing new token registrations the admin explicitly intended to block.
2. Persists the new token mapping on-chain; once the bridge is unpaused, `finalize_transfer` calls referencing that mapping will mint tokens against it — enabling unauthorized token minting if the mapping was fraudulently constructed.

This matches the allowed impact: **High — pause/verification bypass enabling unauthorized token deployment**.

### Likelihood Explanation
The `deploy_token` instruction requires a valid MPC-signed payload, so an unauthenticated attacker cannot call it arbitrarily. However, the realistic attack surface is a relayer submitting a pending signed payload that was produced before the pause was triggered. This is a normal operational scenario (signed payloads are queued and submitted asynchronously), making the bypass reachable without any key compromise.

### Recommendation
1. Add a `DEPLOY_TOKEN_PAUSED` constant in `constants.rs`:
   ```rust
   pub const DEPLOY_TOKEN_PAUSED: u8 = 1 << 2;
   pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED | DEPLOY_TOKEN_PAUSED;
   ```
2. Add a pause check at the top of `deploy_token` in `lib.rs`:
   ```rust
   require!(
       ctx.accounts.common.config.paused & DEPLOY_TOKEN_PAUSED == 0,
       error::ErrorCode::Paused
   );
   ```

### Proof of Concept
1. Admin calls `pause()` on Solana bridge — sets `config.paused = ALL_PAUSED = 0x03`.
2. A relayer holds a valid MPC-signed `DeployTokenPayload` for NEAR token `token.near`.
3. Relayer calls `deploy_token(signed_payload)`.
4. The instruction executes successfully: no pause check exists, `ALL_PAUSED` has no deploy-token bit.
5. A new Solana mint is created and registered as the canonical mapping for `token.near`, despite the bridge being in a paused/emergency state.
6. After the bridge is unpaused, `finalize_transfer` calls for `token.near` will mint against this newly registered mint — which may not be the intended one if the deployment was adversarial.
