### Title
Reentrancy in `initTransfer1155` via Malicious ERC1155 Callback Causes Nonce Collision and Permanent Lock of User Funds — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`initTransfer1155` increments `currentOriginNonce` at the top of the function but reads it again at the bottom when emitting `InitTransfer`. Between those two points it makes an external call to an attacker-controlled ERC1155 contract via `safeTransferFrom`. A malicious ERC1155 can reenter `initTransfer1155` during that call, causing `currentOriginNonce` to be incremented a second time. Both the reentrant call and the original call then emit `InitTransfer` with the same `originNonce`. NEAR processes only the first event and rejects the second as a duplicate, permanently locking the tokens from the second transfer in the bridge with no recovery path.

---

### Finding Description

In `initTransfer1155`:

```solidity
function initTransfer1155(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;                          // (1) nonce = N
    ...
    IERC1155(tokenAddress).safeTransferFrom(          // (2) external call — reentrancy window
        msg.sender, address(this), tokenId, amount, ""
    );
    ...
    emit BridgeTypes.InitTransfer(
        msg.sender, deterministicToken,
        currentOriginNonce,                           // (3) reads N+1 if reentrant call ran
        amount, fee, nativeFee, recipient, message
    );
}
``` [1](#0-0) 

The nonce is captured at step (1) as `N`, but the event at step (3) reads `currentOriginNonce` from storage, not from a local variable. If a reentrant call executes between steps (2) and (3), it increments `currentOriginNonce` to `N+1` and emits its own `InitTransfer` with nonce `N+1`. When control returns to the outer call, it also emits `InitTransfer` with nonce `N+1` (the now-stale storage value). Nonce `N` is never emitted.

The `onERC1155Received` guard on the bridge only rejects direct sends where `operator != address(this)`:

```solidity
function onERC1155Received(address operator, ...) external view override returns (bytes4) {
    if (operator != address(this)) revert ERC1155DirectSendNotAllowed();
    return this.onERC1155Received.selector;
}
``` [2](#0-1) 

When the bridge itself calls `safeTransferFrom`, the `operator` passed to `onERC1155Received` is `address(this)`, so the guard passes. The malicious ERC1155 token contract can execute arbitrary logic inside its own `safeTransferFrom` implementation — including calling back into `initTransfer1155` — before or after invoking `onERC1155Received`. No `nonReentrant` modifier is present on `initTransfer1155`.

The same stale-nonce read also affects `initTransferExtension`, which receives `currentOriginNonce` as a parameter:

```solidity
initTransferExtension(
    msg.sender, deterministicToken,
    currentOriginNonce,   // wrong nonce after reentrant call
    ...
);
``` [3](#0-2) 

In the Wormhole extension variant this would post a Wormhole message with the wrong nonce.

The NEAR side's `fin_transfer_callback` uses `(origin_chain, origin_nonce)` as the transfer ID and records it in `finalised_transfers`:

```rust
let mut required_balance = self.add_fin_transfer(&transfer_message.get_transfer_id());
``` [4](#0-3) 

When two events share the same `origin_nonce`, the second is rejected as already finalized, and the tokens locked on EVM for that transfer are permanently inaccessible.

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds.**

A victim who calls `initTransfer1155` with a malicious ERC1155 token has their tokens locked in the bridge. The attacker's reentrant call emits first (inner call completes before outer call), so NEAR finalizes the attacker's event. The victim's event — emitted second with the same nonce — is rejected as a duplicate. The victim's ERC1155 tokens remain locked in the bridge contract with no mechanism to recover them: there is no refund path, no admin rescue function, and the NEAR side will never process a transfer with that nonce again.

Additionally, nonce `N` (the outer call's original nonce) is never emitted, creating a permanent gap in the nonce sequence.

---

### Likelihood Explanation

The attack requires deploying a malicious ERC1155 contract and convincing at least one user to bridge it (or the attacker bridges it themselves to disrupt accounting). ERC1155 tokens are permissionlessly bridgeable — `logMetadata1155` is callable by anyone:

```solidity
function logMetadata1155(address tokenAddress, uint256 tokenId) external payable {
    ...
    multiToken.tokenAddress = tokenAddress;
    multiToken.tokenId = tokenId;
    ...
}
``` [5](#0-4) 

No admin approval is needed to register an ERC1155 token. The attacker can deploy a malicious ERC1155, register it, and wait for users to bridge it. The ERC1155 standard's `safeTransferFrom` callback is well-known and the reentrancy pattern is straightforward to implement.

---

### Recommendation

1. **Save the nonce in a local variable** before the external call and use it in the event and `initTransferExtension`:

```solidity
function initTransfer1155(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    uint64 nonce = currentOriginNonce;   // capture before external call
    ...
    IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "");
    ...
    initTransferExtension(msg.sender, deterministicToken, nonce, ...);
    emit BridgeTypes.InitTransfer(msg.sender, deterministicToken, nonce, ...);
}
```

2. **Or add `nonReentrant`** from OpenZeppelin's `ReentrancyGuard` to `initTransfer1155` (and `initTransfer` for defense-in-depth against future ERC20 callback tokens).

Note: `initTransfer` (ERC20 path) has the same structural pattern — nonce incremented at top, event emitted at bottom with `currentOriginNonce` — but standard ERC20 tokens lack callbacks. It should be hardened with a local nonce variable as well for consistency.

---

### Proof of Concept

1. Attacker deploys `MaliciousERC1155` whose `safeTransferFrom(from, to, id, amount, data)` implementation:
   - Calls `bridge.initTransfer1155(address(this), id, largeAmount, 0, 0, attacker_near, "")` (reentrant call, `msg.sender` = `MaliciousERC1155`)
   - Then calls `bridge.onERC1155Received(bridge, from, id, amount, "")` to satisfy the callback check and returns the selector
2. Attacker calls `bridge.logMetadata1155(MaliciousERC1155, tokenId)` to register the token.
3. Victim calls `bridge.initTransfer1155(MaliciousERC1155, tokenId, victimAmount, 0, 0, victim_near, "")`.
4. Bridge increments `currentOriginNonce` to `N`, then calls `MaliciousERC1155.safeTransferFrom(victim, bridge, tokenId, victimAmount, "")`.
5. Inside `safeTransferFrom`, reentrant call executes: `currentOriginNonce` becomes `N+1`; emits `InitTransfer(MaliciousERC1155, deterministicToken, N+1, largeAmount, 0, 0, attacker_near, "")`.
6. `safeTransferFrom` returns; outer call emits `InitTransfer(victim, deterministicToken, N+1, victimAmount, 0, 0, victim_near, "")`.
7. Transaction log contains two `InitTransfer` events both with `originNonce = N+1`.
8. NEAR relayer submits both. NEAR finalizes the first (attacker's, nonce `N+1`). The second (victim's, nonce `N+1`) is rejected: `ERR_TRANSFER_ALREADY_FINALISED`.
9. Victim's `victimAmount` ERC1155 tokens are permanently locked in the bridge.
