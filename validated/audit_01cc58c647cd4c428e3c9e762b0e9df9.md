### Title
`state_diff_length` in Stored Header Diverges from `concatenated_counts` in Commitment — Wrong `concatenated_counts` Enters Block Hash - (File: `crates/apollo_p2p_sync/src/client/header.rs`)

### Summary

When a sequencer node converts an internally-produced `SyncBlock` into a `SignedBlockHeader` for storage, it sets `state_diff_length` from `sync_block.state_diff.len()` (the actual state diff) while `concatenated_counts` — which encodes `state_diff_length` as a 64-bit field and is chained directly into the block hash — was computed earlier from `ThinStateDiff::from(commitment_state_diff.clone())`, which **always produces `deprecated_declared_classes: Vec::new()`**. If the block contains deprecated (Cairo 0) declared classes, the two values diverge: `concatenated_counts` encodes a smaller length than `state_diff_length`, and the stored header carries an internally inconsistent commitment that produces a wrong block hash.

### Finding Description

**Root cause — `CommitmentStateDiff → ThinStateDiff` silently drops `deprecated_declared_classes`**

`CommitmentStateDiff` has no field for deprecated declared classes. The `From<CommitmentStateDiff> for ThinStateDiff` conversion hard-codes an empty `Vec`:

```rust
// crates/blockifier/src/state/cached_state.rs:690-701
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            ...
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),   // ← always empty
        }
    }
}
``` [1](#0-0) 

**Where `concatenated_counts` is computed**

`BlockExecutionArtifacts::new` calls `calculate_block_commitments` with `ThinStateDiff::from(commitment_state_diff.clone())`. Because `deprecated_declared_classes` is empty, `state_diff.len()` inside `calculate_block_commitments` is smaller than the real length whenever deprecated classes are present. The resulting `concatenated_counts` field encodes this smaller value:

```rust
// crates/apollo_batcher/src/block_builder.rs:160-166
let (header_commitments, measurements) = calculate_block_commitments(
    &transactions_data,
    ThinStateDiff::from(commitment_state_diff.clone()),  // deprecated_declared_classes = []
    l1_da_mode,
    &block_info.starknet_version,
).await;
``` [2](#0-1) 

`concat_counts` packs `state_diff_length` into bits [127:64] of the `concatenated_counts` felt, which is then chained into the block hash:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs:318-323
let concatenated_counts = concat_counts(
    transactions_data.len(),
    event_leaf_elements.len(),
    state_diff.len(),   // ← uses the truncated length
    l1_da_mode,
);
``` [3](#0-2) 

**Where the divergence surfaces in the stored header**

`convert_sync_block_to_block_data` in the P2P sync client constructs the `BlockHeader` that is written to storage. It sets `state_diff_length` from the actual `sync_block.state_diff.len()` (which includes deprecated classes), but uses `header_commitments.concatenated_counts` that was computed without them:

```rust
// crates/apollo_p2p_sync/src/client/header.rs:130-157
fn convert_sync_block_to_block_data(block_number, sync_block) -> SignedBlockHeader {
    let header_commitments = sync_block.block_header_commitments.expect(...);
    // TODO(Shahak): Verify `n_transactions` and `state_diff_length` match values in
    // concatenated_counts.                          ← acknowledged but unimplemented
    SignedBlockHeader {
        block_header: BlockHeader {
            state_diff_length: Some(sync_block.state_diff.len()),  // real length
            // concatenated_counts inside header_commitments encodes wrong length
            ...
        },
    }
}
``` [4](#0-3) 

The TODO comment at line 140–141 explicitly acknowledges that `n_transactions` and `state_diff_length` are **not verified** against `concatenated_counts`.

**How the wrong value propagates into the block hash**

`calculate_block_hash` chains `concatenated_counts` directly:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs:260
.chain(&block_commitments.concatenated_counts)
``` [5](#0-4) 

When `finalize_commitment_output` later calls `calculate_block_hash` using the stored `PartialBlockHashComponents` (which contains the wrong `concatenated_counts`), the resulting block hash is wrong: [6](#0-5) 

**P2P sync state diff length check is also broken by the same divergence**

The P2P sync client uses `header.state_diff_length` (the real length) as the termination condition for reassembling state diff chunks: [7](#0-6) 

Because `state_diff_length` in the header is correct but `concatenated_counts` is wrong, the block hash computed from the stored header will not match the hash computed by any other node that correctly encodes `state_diff_length` in `concatenated_counts`.

### Impact Explanation

**Wrong block hash / wrong state diff commitment in block hash** — `concatenated_counts` encodes a `state_diff_length` that is smaller than the real one whenever a block contains deprecated (Cairo 0) declared classes. The `PartialBlockHashComponents` stored for that block carries this wrong value. When `calculate_block_hash` is called with these components, it produces a block hash that differs from the canonical hash. This is a **wrong state/commitment value** that feeds the block hash, matching the impact category: *Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.*

Additionally, the stored header has `state_diff_length` (correct) inconsistent with `concatenated_counts` (wrong), so any downstream consumer that reconstructs `concatenated_counts` from the header fields (as `TryFrom<&BlockHeader> for Option<BlockHeaderCommitments>` does) will produce a different `concatenated_counts` than what was stored, causing further divergence. [8](#0-7) 

### Likelihood Explanation

Deprecated (Cairo 0) class declarations are a standard Starknet transaction type (`DeclareTransactionV0V1`). Any block that includes such a transaction triggers the divergence. The sequencer is a production node; blocks with deprecated declarations have been common historically. The bug is latent in the current code path and activates automatically without any special attacker action — any block with a deprecated Declare transaction produces a wrong `concatenated_counts` and therefore a wrong block hash.

### Recommendation

1. **Fix `CommitmentStateDiff`** to carry `deprecated_declared_classes`, or pass the full `ThinStateDiff` (including deprecated classes) into `BlockExecutionArtifacts::new` so that `calculate_block_commitments` receives the correct length.

2. **Enforce the invariant** in `convert_sync_block_to_block_data`: verify that `sync_block.state_diff.len()` matches the `state_diff_length` encoded in `concatenated_counts` before writing the header to storage. The existing TODO comment at line 140–141 of `crates/apollo_p2p_sync/src/client/header.rs` acknowledges this gap but leaves it unimplemented.

3. **Add a cross-check** in `finalize_commitment_output` that the `state_diff_length` stored in the header equals the value extracted from `concatenated_counts` before computing the final block hash.

### Proof of Concept

1. Submit a `DeclareTransactionV0V1` (deprecated Cairo 0 class declaration) to the sequencer.
2. The blockifier produces a `CommitmentStateDiff` with no `deprecated_declared_classes` field.
3. `BlockExecutionArtifacts::new` calls `ThinStateDiff::from(commitment_state_diff.clone())`, producing `deprecated_declared_classes: vec![]`.
4. `calculate_block_commitments` computes `state_diff.len()` = N (missing the deprecated class entry), encodes this into `concatenated_counts`.
5. The `PartialBlockHashComponents` stored for this block contains `concatenated_counts` encoding length N.
6. The actual `ThinStateDiff` written to storage has `deprecated_declared_classes` populated, so `sync_block.state_diff.len()` = N+1.
7. `convert_sync_block_to_block_data` writes `state_diff_length: Some(N+1)` but `concatenated_counts` encoding N into the stored header.
8. `calculate_block_hash` uses `concatenated_counts` with N, producing hash H_wrong.
9. Any verifier (OS, proof system, other node) that computes `concatenated_counts` from the correct length N+1 produces hash H_correct ≠ H_wrong.
10. The sequencer's stored block hash is wrong; proofs generated against it are invalid.
