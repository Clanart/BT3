Looking at the full validation chain in `set_state_header` and the downstream finalization path.

### Title
Missing `chunk.shard_id()` vs `shard_id` parameter check in `set_state_header` allows cross-shard state root substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer and validates it against a caller-supplied `shard_id` parameter. None of the five validation steps check that `chunk.shard_id()` equals the `shard_id` parameter. A malicious peer can supply shard B's chunk (legitimately included in `sync_prev_block`) while the call is made with `shard_id=A`, pass every guard, and cause the header to be persisted under `StateHeaderKey(shard_id=A, sync_hash)`. Subsequent `set_state_part` and `set_state_finalize` then install shard B's trie nodes under shard A's `ShardUId` and write shard B's `prev_state_root` into `DBCol::ChunkExtra` keyed by shard A's uid.

---

### Finding Description

**Entrypoint**: `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`, called from `ClientActor` via `StateHeaderValidationRequest` (dispatched by `StateSyncDownloader::ensure_shard_header`).

The five validation steps are:

**Step 1-2 — `validate_chunk_proofs`**: Checks the chunk's internal hash consistency, tx root, and outgoing receipts root. No comparison of `chunk.shard_id()` to the `shard_id` parameter. [1](#0-0) 

**Step 3a — `verify_path` on `chunk_headers_root`**: Proves the chunk is a leaf in the block's Merkle tree of `ChunkHashHeight` values. The tree is built over `(chunk_hash, height_included)` tuples for all shards; `verify_path` checks hash reachability from root but **does not verify the leaf's index (shard position)**. Shard B's chunk, legitimately at position 1, produces a valid Merkle proof against the same root. [2](#0-1) [3](#0-2) 

**Step 4 — receipt proof validation**: `receipts_hash` is computed as `CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))` using the **parameter** `shard_id` (A), not `chunk.shard_id()`. An attacker populates `incoming_receipts_proofs` with shard A's legitimate, publicly-available receipt proofs; all `verify_path` calls in the loop pass. [4](#0-3) 

**Step 5 — `validate_state_root_node`**: Validates the supplied `state_root_node` against `chunk_inner.prev_state_root()`, which is shard B's state root. The attacker supplies shard B's legitimate `state_root_node`; validation passes. [5](#0-4) 

**Storage**: The header is persisted under `StateHeaderKey(shard_id=A, sync_hash)` but contains shard B's chunk. [6](#0-5) 

**Downstream corruption in `set_state_finalize`**: `shard_uid` is derived from the `shard_id` parameter (A), but `chunk_header.prev_state_root()` is shard B's state root. `apply_chunk` is called with `shard_uid=A` and `state_root=shard_B_root`, writing shard B's trie nodes and the resulting `ChunkExtra` under shard A's uid. [7](#0-6) 

---

### Impact Explanation

A syncing node that accepts the crafted header will:
1. Apply shard B's state parts under shard A's `ShardUId` (via `apply_state_part(shard_id=A, state_root=shard_B_root, ...)`).
2. Write a `ChunkExtra` for shard A whose `state_root` equals shard B's state root.
3. Produce incorrect block execution results for shard A in all subsequent epochs, causing the node to diverge from the canonical chain or produce invalid blocks/chunks if it is a validator.

The corrupted value is concrete: `DBCol::ChunkExtra` keyed by `(block_hash, shard_uid_A)` will contain `state_root = shard_B_prev_state_root`.

---

### Likelihood Explanation

All data required to craft the attack is publicly available on the network:
- Shard B's chunk body and its Merkle proof in `sync_prev_block` (served by any full node).
- Shard A's incoming receipt proofs for the sync epoch (served by any full node).
- Shard B's `state_root_node` (served by any state sync provider).

The attacker only needs to be a peer that the syncing node selects as a state sync provider. No validator, chunk producer, or operator privileges are required. Any node can advertise state sync capability and respond to `StateRequestHeader` messages.

---

### Recommendation

Add an explicit shard identity check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// ADD: reject headers whose embedded chunk belongs to a different shard
if chunk.shard_id() != shard_id {
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {} != requested shard_id {}",
        chunk.shard_id(), shard_id
    )));
}
```

Apply the same guard to `prev_chunk_header.shard_id()` if present. [8](#0-7) 

---

### Proof of Concept

```
Setup: two-shard network, shard_id 0 (A) and shard_id 1 (B).
sync_prev_block contains chunk_A (shard 0) and chunk_B (shard 1).

Craft ShardStateSyncResponseHeaderV2:
  chunk          = chunk_B  (shard_id() == 1, legitimately in sync_prev_block)
  chunk_proof    = Merkle proof of chunk_B in sync_prev_block.chunk_headers_root
  prev_chunk_header / prev_chunk_proof = shard B's previous chunk (legitimate)
  incoming_receipts_proofs = shard A's legitimate receipt proofs
  root_proofs    = shard A's legitimate root proofs
  state_root_node = shard B's state_root_node (hash matches chunk_B.prev_state_root())

Call: set_state_header(shard_id=0, sync_hash, crafted_header)

Assert: all five validation steps return Ok(()).
Assert: get_state_header(shard_id=0, sync_hash).chunk_prev_state_root()
        == shard_B_prev_state_root   (not shard_A_prev_state_root)

Then call apply_state_part(shard_id=0, &shard_B_state_root, ...) for each part,
then set_state_finalize(shard_id=0, sync_hash).

Assert: DBCol::ChunkExtra keyed by (block_hash, shard_uid_0)
        contains state_root == shard_B_prev_state_root.
```

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
