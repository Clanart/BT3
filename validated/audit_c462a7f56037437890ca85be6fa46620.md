### Title
Gateway Skips Account `__validate__` Signature Check for Nonce-1 Invoke When Any Account Transaction Exists in Mempool — (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

---

### Summary

The `skip_stateful_validations` function in the Apollo gateway bypasses the account's `__validate__` entry-point (the Starknet equivalent of a signature check) for any Invoke transaction with nonce 1 whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. Because that helper returns `true` for **any** transaction type in the pool — not exclusively a `DeployAccount` — an unprivileged attacker who observes a victim's `DeployAccount` in the mempool can inject an Invoke with nonce 1 for the victim's address carrying an arbitrary or empty signature, and the gateway will admit it without running `__validate__`.

---

### Finding Description

`skip_stateful_validations` is the direct sequencer analog of the "seller's signature not required" pattern: just as the payment contract accepted a listing without the seller's authorization, the gateway accepts a nonce-1 Invoke without the account owner's authorization.

### Impact Explanation

An attacker who monitors the public mempool for a victim's `DeployAccount` can immediately inject a nonce-1 Invoke for the victim's address with an arbitrary signature. The gateway admits it without calling `__validate__`, placing it in the mempool under the victim's (address, nonce=1) slot. This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

Concrete consequences:
- The attacker's invalid transaction occupies the victim's nonce-1 slot. Depending on mempool replacement policy, the victim's legitimate nonce-1 Invoke may be rejected as a duplicate nonce or require a fee-escalation race.
- The attacker can repeat this for every new `DeployAccount` observed, creating a systematic DoS against newly-deploying accounts.
- The admitted transaction is eventually rejected during block execution (because `__validate__` runs with `validate: true` in the batcher), but the mempool slot is held until that point — the exact "phony listing that will never complete, locking resources for a period of time" pattern from the external report.

---

### Likelihood Explanation

- **Unprivileged trigger**: requires only observing the public mempool (P2P gossip) and submitting a crafted Invoke — no privileged access needed.
- **Condition**: victim must have submitted a `DeployAccount` that has not yet been committed. This is a normal, observable event for any new Starknet account.
- **Attacker cost**: submitting a transaction costs only the RPC call; the transaction will be rejected at execution so no fee is charged.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` predicate with a type-specific check that returns `true` only when a `DeployAccount` transaction (not any transaction) is present for the sender address. Add a dedicated mempool API such as `deploy_account_in_pool(address) -> bool` and use it in `skip_stateful_validations`. This mirrors the fix described in the external report: requiring the specific authorizing action (the deploy) rather than any action.

---

### Proof of Concept

1. Victim broadcasts `DeployAccount` for address `A` (nonce 0). It enters the mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
2. Attacker submits `Invoke { sender: A, nonce: 1, signature: [0x0] }` to the gateway.
3. Gateway evaluates `skip_stateful_validations`:
   - `tx.nonce() == 1` ✓
   - `account_nonce == 0` ✓
   - `account_tx_in_pool_or_recent_block(A) == true` ✓ (victim's DeployAccount)
   - Returns `true` → `validate: false` → `__validate__` is **not called**.
4. Gateway calls `run_validate_entry_point` with `validate: false`; `StatefulValidator::perform_validations` returns `Ok(())` immediately at line 79–81 without running `__validate__`. [6](#0-5) 

5. Attacker's Invoke is added to the mempool occupying slot `(A, nonce=1)`.
6. Victim's legitimate Invoke with nonce 1 may be rejected or require fee escalation to displace the attacker's entry.
7. When the batcher eventually executes the attacker's Invoke, `__validate__` runs with `validate: true`, the invalid signature is rejected, and the transaction is dropped — but the mempool slot was held throughout.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
