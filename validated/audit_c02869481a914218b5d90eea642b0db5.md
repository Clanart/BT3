### Title
Unchecked Field Arithmetic Overflow in `compute_max_possible_fee` Allows Fee Bypass - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` performs unchecked field arithmetic on `max_price_per_unit` values that have no enforced upper bound. A transaction sender can craft resource bounds such that the sum of products wraps to zero modulo the Cairo field prime, causing the OS to skip fee charging entirely. This is enforced at the proof level — the sequencer is provably unable to collect fees from such a transaction.

---

### Finding Description

`compute_max_possible_fee` computes the fee cap as a sum of products in the Cairo field:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

The only validation applied to `max_price_per_unit` during transaction hash computation is `assert_nn` (non-negative), which permits values up to `(P−1)/2 ≈ 2^250`:

```cairo
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
assert_nn(resource_bounds.max_price_per_unit);   // no upper bound
``` [2](#0-1) 

`max_amount` is bounded to `[0, 2^64−1]`, but `max_price_per_unit` is bounded only to `[0, (P−1)/2]`. With three independent resource types, the attacker controls six values. The sum of three products can be made to equal exactly `P` (or any multiple of `P`) modulo the field prime, yielding `max_fee = 0`.

**Concrete example:**
- Set `l1_gas_bounds = {max_amount: 1, max_price_per_unit: P−1}` → product = `P−1 ≡ −1`
- Set `l2_gas_bounds = {max_amount: 1, max_price_per_unit: 1}`, `tip = 0` → product = `1`
- Set `l1_data_gas_bounds = {max_amount: 0, max_price_per_unit: 0}` → product = `0`
- Sum = `(P−1) + 1 + 0 = P ≡ 0 (mod P)`

`compute_max_possible_fee` returns `0`. The OS then takes the early-exit path:

```cairo
if (max_fee == 0) {
    return ();
}
``` [3](#0-2) 

Fee charging is skipped entirely. Even if `max_fee` wraps to a small non-zero value, the proof-enforced constraint `assert_nn_le(actual_fee, max_fee)` caps the sequencer's collectible fee at that tiny value: [4](#0-3) 

The transaction hash commits to these resource bounds via `hash_fee_fields` → `pack_resource_bounds`, so the user signs over the crafted values — this is a deliberate, self-signed attack. [5](#0-4) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

The OS proof enforces that the sequencer cannot collect more than `max_fee`. When `max_fee` wraps to 0, the proof makes it impossible for the sequencer to collect any fee for that transaction. The user receives full execution (validate + execute entry points run, state changes are committed) at zero cost. At scale, an attacker can flood the network with computationally expensive zero-fee transactions, draining sequencer resources and causing the network to be unable to confirm legitimate transactions.

---

### Likelihood Explanation

**Medium.** Any V3 transaction sender can craft the required resource bounds. The math is straightforward: choose `max_price_per_unit` values such that the field sum equals a multiple of P. No privileged access, leaked key, or external dependency is required. The attacker's own account contract runs `__validate__` and can be written to accept any such transaction.

---

### Recommendation

Add explicit upper-bound range checks on `max_price_per_unit` in `compute_max_possible_fee` (or in `pack_resource_bounds`, which is already called during hash computation). The StarkNet specification constrains `max_price_per_unit` to 128 bits. Enforce this:

```cairo
assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);
```

With `max_amount <= 2^64−1` and `max_price_per_unit <= 2^128−1`, each product is at most `2^192−1`, and the sum of three products is at most `3 * (2^192−1) < P`, eliminating any possibility of field overflow. [6](#0-5) 

---

### Proof of Concept

1. Deploy an account contract whose `__validate__` unconditionally returns `VALIDATED`.
2. Submit a V3 invoke transaction with:
   - `l1_gas_bounds = ResourceBounds{resource: L1_GAS, max_amount: 1, max_price_per_unit: P−1}`
   - `l2_gas_bounds = ResourceBounds{resource: L2_GAS, max_amount: 1, max_price_per_unit: 1}`
   - `l1_data_gas_bounds = ResourceBounds{resource: L1_DATA_GAS, max_amount: 0, max_price_per_unit: 0}`
   - `tip = 0`
3. The OS computes `compute_max_possible_fee`:
   - `1*(P−1) + 1*(1+0) + 0*0 = P ≡ 0 (mod P)`
4. `charge_fee` hits `if (max_fee == 0) { return (); }` and exits without charging.
5. The transaction executes fully. The proof is valid. The sequencer collected 0 fees. [7](#0-6)

// Returns the maximum possible fee that can be charged for the transaction.
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}

func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);

func hash_fee_fields{range_check_ptr, poseidon_ptr: PoseidonBuiltin*}(
    tip: felt, resource_bounds: ResourceBounds*, n_resource_bounds: felt
) -> felt {
    alloc_locals;

    let (local data_to_hash: felt*) = alloc();
    assert data_to_hash[0] = tip;
    assert_nn_le(tip, 2 ** 64 - 1);

    static_assert L1_GAS_INDEX == 0;
    static_assert L2_GAS_INDEX == 1;
    static_assert L1_DATA_GAS_INDEX == 2;

    with_attr error_message("Invalid number of resource bounds: {n_resource_bounds}.") {
        assert n_resource_bounds = 3;
    }

    // L1 gas.
    let l1_gas_bounds = resource_bounds[L1_GAS_INDEX];
    assert l1_gas_bounds.resource = L1_GAS;
    assert data_to_hash[1] = pack_resource_bounds(l1_gas_bounds);

    // L2 gas.
    let l2_gas_bounds = resource_bounds[L2_GAS_INDEX];
    assert l2_gas_bounds.resource = L2_GAS;
    assert data_to_hash[2] = pack_resource_bounds(l2_gas_bounds);

    // L1 data gas.
    let l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];
    assert l1_data_gas_bounds.resource = L1_DATA_GAS;
    assert data_to_hash[3] = pack_resource_bounds(l1_data_gas_bounds);

    let (hash) = poseidon_hash_many(n=n_resource_bounds + 1, elements=data_to_hash);
    return hash;
