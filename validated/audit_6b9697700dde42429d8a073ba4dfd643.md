### Title
Asymmetric Amount Validation Causes Permanently Frozen User Funds When Normalized Transfer Amount Rounds to Zero - (File: near/omni-bridge/src/lib.rs)

### Summary

`init_transfer` on NEAR accepts any amount where `fee < amount`, but `sign_transfer` enforces an additional check that the decimal-normalized amount is greater than zero. When a user sends a small amount of a token whose origin chain has more decimals than the destination chain, the normalization truncates the amount to zero. `sign_transfer` then permanently panics, and because there is no cancel/refund path for pending transfers, the user's locked or burned tokens are irrecoverably frozen.

### Finding Description

The NEAR bridge stores token decimal metadata as a `Decimals` struct with two fields: `decimals` (destination chain decimals) and `origin_decimals` (source chain decimals). [1](#0-0) 

`normalize_amount` performs floor division by `10^(origin_decimals - decimals)`. For a token with 24 origin decimals going to an 18-decimal EVM chain, the divisor is `10^6`. Any amount less than `10^6` units (after fee deduction) normalizes to zero.

`init_transfer` only validates that `fee < amount`: [2](#0-1) 

It then locks or burns the tokens and stores the transfer message in `pending_transfers`: [3](#0-2) 

Later, when a relayer calls `sign_transfer`, the normalized amount is computed and checked: [4](#0-3) 

If `normalize_amount` returns zero, `sign_transfer` panics with `InvalidAmountToTransfer`. This panic is permanent — no amount of retries or fee updates can change the outcome, because the amount is fixed at `init_transfer` time and `update_transfer_fee` can only increase the fee (making `amount_without_fee` even smaller).

There is no public `cancel_transfer` function. The only paths that remove a pending transfer are:
- `sign_transfer_callback` — only executes if signing succeeds (it never will here)
- `claim_fee_callback` — requires a proof of a successful `fin_transfer` on the destination chain (impossible since signing never succeeds)
- `remove_transfer_message_without_refund` — only called internally when storage balance fails during `init_transfer_internal` [5](#0-4) 

### Impact Explanation

User funds are permanently frozen. The tokens are locked (for native NEAR tokens) or burned (for deployed bridge tokens) at `init_transfer` time. Because `sign_transfer` will always revert and no cancel path exists, the transfer can never complete and the tokens can never be recovered. This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user funds**.

### Likelihood Explanation

The condition is triggered whenever a user sends an amount (net of fee) smaller than `10^(origin_decimals - destination_decimals)`. For NEAR's native token (24 decimals) bridging to any EVM chain (18 decimals), the threshold is `10^6` yoctoNEAR — a tiny but non-zero amount. A user who misunderstands decimal units, or a script that sends a dust amount, can trigger this. The `init_transfer` call succeeds with no warning, making the failure non-obvious.

### Recommendation

Add a pre-flight check inside `init_transfer` (or `init_transfer_internal`) that mirrors the check in `sign_transfer`:

```rust
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(token_address) = token_address {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

Alternatively, implement a `cancel_transfer` function that allows the original sender to reclaim their tokens from a pending transfer that has not yet been signed.

### Proof of Concept

1. A NEAR token with 24 origin decimals is registered on the bridge with an EVM destination chain using 18 decimals. The `Decimals` struct is `{decimals: 18, origin_decimals: 24}`, giving a normalization divisor of `10^6`.

2. User calls `ft_transfer_call` on the token contract, transferring `amount = 500_000` units (less than `10^6`) with `fee = 0` and a valid EVM recipient. The `ft_on_transfer` callback routes to `init_transfer`.

3. `init_transfer` passes the only check (`fee (0) < amount (500_000)`), locks the 500,000 units, and stores the transfer in `pending_transfers`. [2](#0-1) 

4. Relayer calls `sign_transfer`. Inside:
   ```
   amount_without_fee = 500_000 - 0 = 500_000
   normalize_amount(500_000, {18, 24}) = 500_000 / 1_000_000 = 0
   require!(0 > 0, ...) → PANIC: InvalidAmountToTransfer
   ``` [4](#0-3) 

5. Every subsequent call to `sign_transfer` for this transfer ID panics identically. `update_transfer_fee` can only increase the fee, which reduces `amount_without_fee` further. No cancel function exists.

6. The 500,000 units remain locked/burned in the bridge contract forever.

    fn remove_transfer_message(&mut self, transfer_id: TransferId) -> TransferMessage {
        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);
        }

        transfer.message
    }

        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
