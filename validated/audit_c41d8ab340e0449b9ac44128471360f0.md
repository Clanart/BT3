### Title
EVM SSTORE Gas Refund Silently Discarded When `minGasMultiplier` Overrides `gasUsed` - (File: `x/evm/keeper/state_transition.go`)

---

### Summary

In `ApplyMessageWithConfig`, the EVM SSTORE gas refund (EIP-2200/EIP-3529) is correctly computed and added to `leftoverGas`, but is then silently discarded when the `minGasMultiplier` fee-market parameter forces `gasUsed` upward. The final `leftoverGas` is recalculated from the inflated `gasUsed`, erasing the refund entirely. Users who perform SSTORE operations lose their entitled gas refunds, and the fee collector retains funds it should not.

---

### Finding Description

`ApplyMessageWithConfig` computes gas accounting in two sequential, conflicting phases:

**Phase 1 — EVM SSTORE refund (lines 532–556):**

```go
temporaryGasUsed := msg.GasLimit - leftoverGas
refund := GasToRefund(stateDB.GetRefund(), temporaryGasUsed, refundQuotient)
leftoverGas += refund                          // refund credited here
// ...
temporaryGasUsed = msg.GasLimit - leftoverGas  // post-refund gas used
```

After this phase, `temporaryGasUsed` correctly reflects gas consumed minus the SSTORE refund, and `leftoverGas` includes the refund.

**Phase 2 — `minGasMultiplier` floor (lines 571–589):**

```go
minimumGasUsed := gasLimit.Mul(minGasMultiplier)   // e.g. gasLimit * 0.5
// ...
gasUsed = sdkmath.LegacyMaxDec(minimumGasUsed, sdkmath.LegacyNewDec(tempGasUsed)).TruncateInt().Uint64()
leftoverGas = msg.GasLimit - gasUsed               // OVERWRITES the refund-adjusted leftoverGas
```

When `minimumGasUsed > temporaryGasUsed`, `gasUsed` is set to `minimumGasUsed` and `leftoverGas` is recalculated from scratch. The SSTORE refund that was added in Phase 1 is completely overwritten and never reaches the sender.

This `leftoverGas` is then returned in `res.GasUsed` and used in `ApplyTransaction` to compute the actual coin refund:

```go
leftoverGas := msg.GasLimit - res.GasUsed
if err = k.RefundGas(ctx, msg, leftoverGas, cfg.Params.EvmDenom); err != nil { ... }
```

The `RefundGas` call transfers `leftoverGas * gasPrice` back to the sender from the fee collector. Because `leftoverGas` was computed from the inflated `minimumGasUsed`, the SSTORE refund is never paid out.

---

### Impact Explanation

This is a **High** impact fee/gas accounting bug. Any user who:
1. Submits a transaction that performs SSTORE operations generating a non-zero `stateDB.GetRefund()`, and
2. Has a gas limit set such that `gasLimit * minGasMultiplier > temporaryGasUsed` (post-refund gas used),

will silently lose their SSTORE gas refund. The fee collector module retains the difference. The loss scales with the refund amount and the gas price. For a transaction with `gasLimit = 1,000,000`, `minGasMultiplier = 0.5`, actual gas used = 600,000, and SSTORE refund = 200,000: the user pays 500,000 gas instead of 400,000 — a 100,000-gas overcharge per transaction. This fits the allowed impact: *"EVM state transition, gas refund, fee market, ante handler, mempool, or proposal handling bug that permits... valid user funds/fees to be mis-accounted."*

---

### Likelihood Explanation

- `minGasMultiplier` is a live governance parameter read from `cfg.FeeMarketParams.MinGasMultiplier`. Any chain that sets it above zero (a common production configuration to align Tendermint and EVM gas mechanics, as noted in the code comment referencing issue #1085) activates this bug.
- The trigger condition — `gasLimit * minGasMultiplier > post-refund gas used` — is easily met by any user who sets a generous gas limit (standard practice) and clears storage slots (SSTORE to zero, which generates the largest refunds under EIP-3529).
- No special privileges are required. Any unprivileged Ethereum transaction that writes to storage is a candidate.

---

### Recommendation

Apply the `minGasMultiplier` floor **before** computing the SSTORE refund, or re-apply the SSTORE refund **after** the floor is enforced. Concretely, move the `GasToRefund` calculation to after line 587, operating on the already-floored `gasUsed`:

```go
// After minGasMultiplier floor:
gasUsed = sdkmath.LegacyMaxDec(minimumGasUsed, ...).TruncateInt().Uint64()
// Now apply SSTORE refund on top of the floored value:
refund := GasToRefund(stateDB.GetRefund(), gasUsed, refundQuotient)
if refund > gasUsed { refund = gasUsed }
gasUsed -= refund
leftoverGas = msg.GasLimit - gasUsed
```

This ensures the SSTORE refund is always honoured and is never silently discarded by the `minGasMultiplier` override.
