### Title
Unvalidated `mint` Account in `FinalizeTransfer` Allows Vault Draining and Unauthorized Token Minting — (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

### Summary

The `FinalizeTransfer` instruction accepts a caller-supplied `mint` account that is never validated against the token identifier embedded in the signed `FinalizeTransferPayload`. Any permissionless relayer can substitute an arbitrary Solana mint, causing the program to release tokens from the wrong vault or mint tokens of the wrong bridged asset to the recipient.

### Finding Description

In `FinalizeTransfer`, the `mint` account constraint is:

```rust
#[account(
    mut,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
``` [1](#0-0) 

The only constraint is that `mint` belongs to the given `token_program`. There is no constraint tying `mint.key()` to `data.payload.token` (the NEAR-side token address carried in the signed Wormhole payload).

The vault is then derived purely from the caller-supplied `mint`:

```rust
seeds = [VAULT_SEED, mint.key().as_ref()],
``` [2](#0-1) 

So whichever `mint` the relayer passes, the program opens the corresponding vault and transfers from it. The `process()` function performs no additional check linking `mint` to the payload token before executing the transfer or mint:

```rust
if let Some(vault) = &self.vault {
    transfer_checked(... from: vault ..., to: self.token_account ..., ...)?;
} else {
    require!(self.mint.mint_authority.contains(self.authority.key), ...);
    mint_to(... mint: self.mint ..., to: self.token_account ..., ...)?;
}
``` [3](#0-2) 

Furthermore, the `FinalizeTransferResponse` posted back to NEAR uses `self.mint.key()` — the attacker-controlled value — as the reported token:

```rust
let payload = FinalizeTransferResponse {
    token: self.mint.key(),
    ...
``` [4](#0-3) 

The `Config` struct contains no token whitelist or mint-to-NEAR-token mapping that could be checked at runtime: [5](#0-4) 

### Impact Explanation

**Native token path (vault present):** A relayer submits a valid, Wormhole-signed payload for token A but passes mint B. The program opens vault B (a legitimate vault for a different registered token), transfers `data.amount` of token B to the recipient, and reports token B back to NEAR. Vault B is drained without a corresponding lock on the NEAR side. This is direct theft of native bridged assets — **Critical**.

**Bridged token path (no vault):** A relayer passes any bridged mint whose `mint_authority` is the bridge authority. The program mints `data.amount` of that token to the recipient regardless of which NEAR token the payload describes. This is unauthorized minting of bridged assets — **Critical**.

In both cases the nonce is consumed, so the legitimate recipient of the original transfer can never claim their funds (permanent lock of user funds — also **Critical**).

### Likelihood Explanation

`FinalizeTransfer` is a permissionless instruction — any account can act as relayer and submit it. No privileged role is required. The attacker only needs a valid, unspent Wormhole-signed `FinalizeTransferPayload` (e.g., one produced by a real user's cross-chain transfer) and knowledge of an existing vault or bridged mint. Both are publicly observable on-chain.

### Recommendation

Add an explicit constraint in the `FinalizeTransfer` account struct that derives or verifies the expected Solana mint from `data.payload.token`. For bridged tokens this is deterministic (the mint is a PDA seeded from the NEAR token hash, as done in `deploy_token.rs`):

```rust
seeds = [WRAPPED_MINT_SEED, data.payload.token.to_hashed_bytes().as_ref()],
``` [6](#0-5) 

For native tokens, maintain an on-chain mapping from NEAR token address to Solana mint (analogous to `whitelisted_tokens` in the original report) and enforce it as an account constraint on `mint`.

### Proof of Concept

1. User on NEAR initiates a transfer of 1000 USDC (NEAR token address `usdc.near`) to Solana recipient `R`. NEAR bridge emits a Wormhole message; the signed `FinalizeTransferPayload` contains `token = "usdc.near"`, `amount = 1000`, `destination_nonce = N`, `recipient = R`.
2. Attacker (acting as relayer) calls `finalize_transfer` with:
   - `data` = the legitimate signed payload above
   - `mint` = the Solana mint address of WBTC (a different registered native token with a funded vault)
   - `vault` = the WBTC vault PDA (derived from `[VAULT_SEED, wbtc_mint]`)
   - `token_account` = R's associated token account for WBTC
3. The program verifies the Wormhole signature (valid), marks nonce N as used, and executes `transfer_checked` from the WBTC vault to R's WBTC account for 1000 lamports-worth of WBTC.
4. Result: WBTC vault drained by 1000 units; nonce N consumed; the legitimate USDC recipient R receives WBTC instead of USDC (or nothing of value if amounts differ); the NEAR side receives a `FinalizeTransferResponse` claiming `token = wbtc_mint`, corrupting cross-chain accounting.
