### Title
`ProofOfInclusion::valid()` Tautological Final Check Allows Forged Inclusion Proofs Without External Root Verification - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary
`ProofOfInclusion::valid()` in the DataLayer Merkle proof module contains a tautological final check that makes the method only verify internal self-consistency of the proof structure, never verifying the proof's claimed root against any external trusted root hash. An attacker who can supply a serialized `ProofOfInclusion` object can forge a proof of inclusion for any arbitrary key-value pair and have `valid()` return `true`.

### Finding Description

The `ProofOfInclusion::valid()` method is intended to validate that a key is included in a DataLayer Merkle tree. It iterates through each `ProofOfInclusionLayer`, verifying that each layer's `combined_hash` equals the hash of the running hash and the sibling hash. After the loop, it performs a final check:

```rust
existing_hash == self.root_hash()
```

However, `root_hash()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // <-- taken directly from the proof itself
    } else {
        self.node_hash
    }
}
```

After the loop completes without returning `false`, `existing_hash` holds the value of `calculated_hash` from the last iteration. In that last iteration, the check `calculated_hash != layer.combined_hash` was already passed, meaning `calculated_hash == last_layer.combined_hash`. Since `root_hash()` also returns `last_layer.combined_hash`, the final comparison `existing_hash == self.root_hash()` is **always `true`** when the loop completes — it is a tautology.

This means `valid()` only verifies that the proof's internal hash chain is self-consistent, but **never verifies the proof's root hash against any external trusted root**. An attacker can construct a `ProofOfInclusion` with an arbitrary `node_hash` (representing any key-value pair they wish to forge), add one or more layers with attacker-chosen `other_hash` and `combined_hash` values that satisfy the internal consistency check, and `valid()` will return `true`.

The `ProofOfInclusion` struct derives `Streamable` and is exposed via Python bindings with `from_bytes` and `from_bytes_unchecked` constructors, making it trivially constructable from attacker-controlled bytes. [1](#0-0) [2](#0-1) 

### Impact Explanation

Any code that calls `proof.valid()` as the sole check for proof validity — without also comparing `proof.root_hash()` to a trusted on-chain committed root — will accept forged proofs. The DataLayer is used for off-chain key-value storage with on-chain Merkle root commitments. A malicious DataLayer server or a man-in-the-middle attacker can forge a `ProofOfInclusion` for any key-value pair and have it accepted as valid by a client that relies on `valid()` alone. This lets untrusted input prove invalid state, matching the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.**

The fuzz target and tests confirm that `valid()` is used as the sole validation call without any external root comparison: [3](#0-2) [4](#0-3) 

### Likelihood Explanation

The `ProofOfInclusion` struct is fully serializable/deserializable and exposed via Python bindings with `from_bytes`. Any code path that receives a `ProofOfInclusion` from an untrusted source (network, user input, deserialized bytes) and validates it using only `valid()` is exploitable. The misleading name `valid()` strongly implies completeness of validation, making it likely that callers omit the separate `root_hash()` comparison. [5](#0-4) [6](#0-5) 

### Recommendation

`ProofOfInclusion::valid()` must accept a trusted external root hash parameter and verify the proof's computed root against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root  // compare to EXTERNAL trusted root
}
```

The existing `valid()` method (which only checks internal consistency) should be renamed to `is_internally_consistent()` or removed to prevent misuse. All call sites must be updated to pass the on-chain committed root hash.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Attacker wants to forge proof that fake_node_hash is in the tree
fake_node_hash = bytes([0xAA] * 32)
other_hash = bytes([0xBB] * 32)

# Compute a valid combined_hash for this layer (attacker controls all inputs)
h = hashlib.sha256()
h.update(b"\x01")  # internal node prefix (left side)
h.update(fake_node_hash)
h.update(other_hash)
combined_hash = h.digest()

# Construct a forged proof - no knowledge of the real tree required
forged_proof = ProofOfInclusion(
    node_hash=fake_node_hash,
    layers=[ProofOfInclusionLayer(
        other_hash_side=1,   # right side
        other_hash=other_hash,
        combined_hash=combined_hash,  # attacker-controlled "root"
    )]
)

# valid() returns True for a completely forged proof
assert forged_proof.valid() == True
# root_hash() returns the attacker-chosen combined_hash, not any real tree root
assert forged_proof.root_hash() == combined_hash
```

The tautological final check `existing_hash == self.root_hash()` passes because both sides equal `last_layer.combined_hash`, which is entirely attacker-controlled. [1](#0-0)
