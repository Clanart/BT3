### Title
`ProofOfInclusion::valid()` Never Verifies Against an External Root Hash — Forged DataLayer Inclusion Proofs Always Pass - (`File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` only checks that the proof's internal hash chain is self-consistent. The final comparison `existing_hash == self.root_hash()` is a tautology: `root_hash()` returns `last.combined_hash`, which is exactly the value `existing_hash` was just set to inside the loop. No external, authoritative root hash is ever consulted. Any attacker who can supply a `ProofOfInclusion` object (e.g., over the DataLayer network protocol) can fabricate an internally consistent proof for an arbitrary leaf and have `valid()` return `true`.

### Finding Description

`ProofOfInclusion::valid()` iterates over `self.layers`, recomputing each `combined_hash` and comparing it to the stored value:

```rust
// crates/chia-datalayer/src/merkle/proof_of_inclusion.rs  lines 40-58
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // ← existing_hash == layer.combined_hash here
    }

    existing_hash == self.root_hash()      // ← always true; see root_hash() below
}
```

`root_hash()` is:

```rust
// lines 32-38
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← same field that existing_hash was just set to
    } else {
        self.node_hash
    }
}
```

After the loop exits normally, `existing_hash` equals `calculated_hash` from the last iteration, which the loop body already asserted equals `layer.combined_hash`. `root_hash()` returns that same `last.combined_hash`. Therefore `existing_hash == self.root_hash()` is unconditionally `true` whenever the loop completes. The function never compares against any externally known, authoritative tree root.

The analog to the external report is direct: `_lastUpdateTimestamp` was a single shared value that should have been keyed per `periodId`; here `root_hash()` is derived entirely from the proof's own fields rather than from an external reference, so the "validation" is circular and meaningless against a forged proof. [1](#0-0) 

### Impact Explanation

Any DataLayer participant that receives a `ProofOfInclusion` from an untrusted peer and calls `proof.valid()` as its sole check will accept the proof regardless of whether the claimed leaf is actually present in the tree. An attacker can:

1. Choose an arbitrary `node_hash` (the key-value pair they wish to forge).
2. Build any chain of `ProofOfInclusionLayer` values where each `combined_hash` is computed correctly from the previous hash and a chosen sibling — no real tree is needed.
3. Transmit the fabricated `ProofOfInclusion`; `valid()` returns `true`.

This lets untrusted input prove invalid DataLayer state, matching the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

The Python binding exposes `valid()` as the primary (and only named) validation entry point:

```python

Any caller that follows this pattern on an externally supplied proof is vulnerable.

### Likelihood Explanation

The DataLayer is designed to exchange proofs between nodes. The `ProofOfInclusion` type is `Streamable` (serializable/deserializable), meaning it is intended to cross trust boundaries. The Python binding exposes `valid()` as the sole named check. There is no `valid_against_root(root: Hash)` variant, so callers have no obvious API to perform the correct check. The probability that at least one DataLayer consumer calls `proof.valid()` without separately verifying `proof.root_hash()` against a stored root is high. [3](#0-2) 

### Recommendation

Replace the no-argument `valid()` with a version that accepts the expected root hash:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root   // compare against authoritative root
}
```

The current `valid()` (which only checks internal consistency) should either be removed or clearly renamed to `is_internally_consistent()` with documentation warning that it does **not** verify against any tree root.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side};
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};

// Attacker picks an arbitrary leaf hash they wish to forge
let fake_leaf: Hash = [0xAA; 32];
let fake_sibling: Hash = [0xBB; 32];

// Compute a combined_hash that is internally consistent
let fake_combined = chia_datalayer::calculate_internal_hash(
    &fake_leaf, Side::Left, &fake_sibling,
);

let forged = ProofOfInclusion {
    node_hash: fake_leaf,
    layers: vec![ProofOfInclusionLayer {
        other_hash_side: Side::Left,
        other_hash: fake_sibling,
        combined_hash: fake_combined,
    }],
};

// valid() returns true even though fake_leaf is not in any real tree
assert!(forged.valid());

// root_hash() returns fake_combined, not the real tree root
// A correct check would be: forged.root_hash() == actual_tree_root
// but valid() never performs that comparison.
