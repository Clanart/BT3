### Title
Dust-Amount Transfer Permanently Locks/Burns User Tokens Due to Missing Pre-Normalization Check - (File: `near/omni-bridge/src/lib.rs`)

### Summary

When a user initiates a NEAR-side transfer with an amount smaller than the decimal scaling factor (`10^(origin_decimals - decimals)`), the tokens are immediately locked or burned in `init_transfer_internal`, but the transfer can never be completed because `sign_transfer` will always revert with `InvalidAmountToTransfer`. There is no cancellation path, so the funds are permanently frozen.

### Finding Description

The bridge stores a `Decimals` struct per token address containing `origin_decimals` (e.g., 24 for NEAR-native tokens) and `decimals` (e.g., 6 for a 6-decimal EVM token). The `normalize_amount` helper performs floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

For a token with `origin_decimals = 24` and `decimals = 6`, `diff_decimals = 18`, so any `amount < 10^18` normalizes to `0`.

The `init_transfer` function only validates `fee < amount`:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

It does **not** validate that `normalize_amount(amount - fee, decimals) > 0`. Execution then proceeds to `init_transfer_internal`, which locks or burns the tokens:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
```

Later, when the relayer calls `sign_transfer`, the zero-check fires and the call reverts:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

Because `sign_transfer` will always revert for this transfer, no MPC signature is ever produced, no `FinTransfer` event is ever emitted on the destination chain, and no proof can ever be submitted to `claim_fee` or `fin_transfer_callback` to remove the pending transfer and release the funds. The tokens are permanently locked (for native tokens) or permanently burned (for deployed bridge tokens).

### Impact Explanation

- **For deployed/bridged tokens** (burn-on-source path): the tokens are burned in `burn_tokens_if_needed` and are irrecoverably destroyed.
- **For native tokens** (lock-on-source path): the tokens are locked in the bridge contract via `lock_tokens_if_needed` and can never be released because the only release paths (`fin_transfer_callback` for NEAR recipients, `process_fin_transfer_to_other_chain` for other chains, and `claim_fee_callback`) all require a valid proof of a finalized destination-chain event that can never exist.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

### Likelihood Explanation

Any user bridging a token with a large decimal difference (e.g., NEAR 24-decimal tokens to a 6-decimal EVM representation, `diff = 18`) who sends an amount below `10^18` base units will trigger this. For NEAR-native tokens this threshold is 1 NEAR (10^24 yoctoNEAR / 10^18 = 10^6 units on EVM), so any transfer of less than 1 NEAR to such a chain is affected. This is a realistic user mistake, not a contrived edge case.

### Recommendation

Add a pre-normalization check inside `init_transfer` (before tokens are locked/burned) to ensure the net amount after fee and after decimal normalization is non-zero:

```rust
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(token_address) = token_address {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

This mirrors the existing guard in `sign_transfer` but places it before any state mutation.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 6` (diff = 18, scaling factor = `10^18`).
2. User calls `ft_transfer_call` with `amount = 500_000_000_000_000_000` (0.5 NEAR, i.e., `< 10^18`) and `fee = 0`.
3. `init_transfer` passes the `fee < amount` check.
4. `init_transfer_internal` burns or locks the 0.5 NEAR worth of tokens.
5. Relayer calls `sign_transfer`:
   - `normalize_amount(500_000_000_000_000_000, Decimals{decimals:6, origin_decimals:24})` = `500_000_000_000_000_000 / 10^18` = `0`
   - `require!(0 > 0, ...)` → **panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`**
6. No signature is ever produced. The transfer sits in `pending_transfers` forever. The tokens are permanently lost. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
