Audit Report

## Title
Unguarded `refundETH()` allows any caller to drain excess ETH left on the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` is `external` with no access control and unconditionally transfers the router's entire ETH balance to `msg.sender`. Because `pay()` deposits only the exact `value` wei into WETH when `nativeBalance >= value`, any ETH sent above `amountIn` remains on the router as raw ETH. An attacker can call `refundETH()` in a subsequent transaction to steal that surplus.

## Finding Description
`refundETH()` performs no caller check: [1](#0-0) 

`pay()` wraps exactly `value` wei when the native balance covers it, leaving any surplus untouched on the contract: [2](#0-1) 

`receive()` blocks plain ETH pushes from non-WETH addresses, but this guard is irrelevant when ETH arrives as `msg.value` on a named `payable` function: [3](#0-2) 

`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, and `multicall` are all `payable`, so users legitimately attach ETH to these calls: [4](#0-3) [5](#0-4) 

The exploit path: user calls `exactInputSingle{value: 1 ether}` with `amountIn = 0.5 ether` and `tokenIn == WETH`. Inside the swap callback, `pay()` deposits exactly `0.5 ether` into WETH and transfers it to the pool; `0.5 ether` remains on the router. After the swap transaction is mined, an attacker calls `refundETH()` and receives the full `0.5 ether` surplus. No privileged role or special setup is required.

## Impact Explanation
Direct loss of user ETH principal. Any user who sends more ETH than `amountIn` and does not atomically bundle `refundETH()` in the same `multicall` loses the surplus to the first external caller. This is a Critical/High direct loss of user principal with no preconditions beyond a standard overpayment, which is common practice to guard against price movement.

## Likelihood Explanation
Overpaying ETH on WETH-input swaps is routine. Calling `exactInputSingle` or `exactInput` directly (without `multicall`) is a fully supported path. Any frontend or integrator that omits `refundETH()` from the multicall bundle exposes every such user. The attack is zero-cost, permissionless, and repeatable on every victim transaction.

## Recommendation
Restrict `refundETH()` to the address that initiated the top-level call. One approach: record `msg.sender` in transient storage at the entry of each `payable` swap function and require that `refundETH()` is called only by that stored address. Alternatively, automatically sweep excess ETH back to `msg.sender` at the end of each swap entry point, eliminating the need for a separate unguarded call entirely.

## Proof of Concept
1. User calls `exactInputSingle{value: 1 ether}(params)` where `params.tokenIn == WETH` and `params.amountIn == 0.5 ether`.
2. Swap callback fires; `pay()` enters the `nativeBalance >= value` branch, deposits `0.5 ether` into WETH, and transfers it to the pool. The remaining `0.5 ether` stays on the router.
3. Swap completes; user does not call `refundETH()`.
4. Attacker calls `refundETH()` in the next transaction.
5. `_transferETH(msg.sender, 0.5 ether)` executes; attacker receives `0.5 ether`.

Foundry test outline:
```solidity
vm.deal(user, 1 ether);
vm.prank(user);
router.exactInputSingle{value: 1 ether}(params); // amountIn = 0.5 ether, tokenIn = WETH

vm.prank(attacker);
router.refundETH();
assertEq(attacker.balance, 0.5 ether);
```
