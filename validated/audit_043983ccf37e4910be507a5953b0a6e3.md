### Title
Asymmetric margin computation in `within_margin` allows any consensus proposer to inflate L1 gas prices by up to ~11.1% beyond the intended tolerance — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

The `within_margin` function used to validate proposed L1 gas prices computes the tolerance margin from the **proposed** price rather than the **expected/reference** price. This creates a systematic asymmetry: a proposer can inflate L1 gas prices by up to ~11.1% above what validators independently compute and still pass block-info validation, while deflation beyond ~9.1% is rejected. The accepted inflated price is embedded in the block's `BlockInfo` and used by the blockifier to compute fees for every transaction in the block.

---

### Finding Description

In `is_block_info_valid`, the validator checks whether the proposer's L1 gas prices are "close enough" to its own computation: [1](#0-0) 

The check delegates to `within_margin`: [2](#0-1) 

The margin is computed as `(number1.0 * margin_percent) / 100` where `number1` is the **proposed** price. This is the root cause. The check resolves to:

```
|proposed − expected| ≤ proposed × 10%
```

Solving for the range of acceptable proposed prices relative to expected price `E`:

- **Maximum inflation**: proposed = E / 0.9 ≈ **1.111 × E** (+11.1%)
- **Maximum deflation**: proposed = E / 1.1 ≈ **0.909 × E** (−9.1%)

The tolerance is not symmetric around the expected price. A proposer who inflates the gas price gets a proportionally wider absolute margin (because the margin scales with the inflated proposed value), while a proposer who deflates gets a narrower margin.

The correct reference for the margin is the **expected** price (`number2`), which would yield a symmetric ±10% window:

```
|proposed − expected| ≤ expected × 10%
```

The `l1_gas_price_margin_percent` is 10 in all versioned-constants files: [3](#0-2) 

The accepted (inflated) gas price flows directly into `convert_to_sn_api_block_info`, which constructs the `BlockInfo` passed to the blockifier for fee computation: [4](#0-3) 

---

### Impact Explanation

The inflated `l1_gas_price_fri` and `l1_data_gas_price_fri` are stored in the block's `GasPrices` struct and used by `get_fee_by_gas_vector` to compute the actual fee charged to every transaction in the block: [5](#0-4) 

A proposer who sets L1 gas prices at the maximum allowed inflation (~11.1%) causes all transactions in the block to pay ~11.1% more in fees than the protocol intends. This is a wrong fee with direct economic impact, matching the Critical scope: *"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."*

---

### Likelihood Explanation

Any consensus validator can be a block proposer. The proposer has access to the same L1 gas price provider and ETH/STRK oracle data as validators, so they can precisely compute the expected price and set the proposed price to exactly the maximum allowed inflation. The asymmetry is systematic and exploitable every block the malicious validator proposes.

---

### Recommendation

Compute the margin from the **expected** price (`number2`) rather than the proposed price (`number1`):

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    // Use the reference (expected) price, not the proposed price, so the
    // tolerance window is symmetric around the expected value.
    let margin = (number2.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}
```

---

### Proof of Concept

**Setup** (10% margin, `l1_gas_price_margin_percent = 10`):

| | Current (buggy) | Fixed |
|---|---|---|
| Expected price (validator) | 1,000,000 FRI | 1,000,000 FRI |
| Proposed price (attacker) | 1,111,111 FRI (+11.1%) | 1,111,111 FRI (+11.1%) |
| Margin computed from | proposed: 111,111 | expected: 100,000 |
| abs_diff | 111,111 | 111,111 |
| Result | **PASS** (111,111 ≤ 111,111) | **FAIL** (111,111 > 100,000) |

With the current code, the inflated price passes `is_block_info_valid`: [6](#0-5) 

The accepted `init.l1_gas_price_fri = 1,111,111` is then passed to `convert_to_sn_api_block_info` and stored in `strk_gas_prices.l1_gas_price`. Every transaction in the block that consumes L1 gas pays fees computed against this inflated price, yielding ~11.1% excess fee extraction per block the attacker proposes.

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    // For small numbers (e.g., less than 10 wei, if margin is 10%), even an off-by-one
    // error might be bigger than the margin, even if it is just a rounding error.
    // We make an exception for such mismatch, and don't bother checking percentages
    // if the difference in price is only one wei.
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}
```

```rust
        || init.l1_gas_price_wei.0 == 0
        || init.l1_data_gas_price_fri.0 == 0
        || init.l1_data_gas_price_wei.0 == 0
        || init.l2_gas_price_fri.0 == 0
    {
        warn!("Zero gas price detected in block info: {:?}", init);
    }

    let l1_gas_price_fri = NonzeroGasPrice::new(init.l1_gas_price_fri)?;
    let l1_data_gas_price_fri = NonzeroGasPrice::new(init.l1_data_gas_price_fri)?;
    let l1_gas_price_wei = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
    let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
    let l2_gas_price_fri = NonzeroGasPrice::new(init.l2_gas_price_fri)?;
```
