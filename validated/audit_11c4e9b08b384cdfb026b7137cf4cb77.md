### Title
Missing Class Existence Validation in `execute_replace_class` Enables Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS Cairo program does not verify that the caller-supplied class hash is a declared class. An unprivileged contract can invoke `replace_class` with an arbitrary, undeclared class hash. The OS commits this to the global state without constraint, permanently rendering the contract non-executable and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads the new class hash directly from the syscall request buffer (controlled by the calling contract) and writes it unconditionally into `contract_state_changes`:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 is the OS developers' own acknowledgment that the check is absent. There is no Cairo constraint anywhere in the OS that cross-references `class_hash` against `contract_class_changes` (the dict that records declared classes) or against the existing committed class tree. The OS therefore generates a valid proof for a state transition that sets a contract's class to a hash that has never been declared.

Class declarations are recorded exclusively via `dict_update` on `contract_class_changes` inside `execute_declare_transaction`:

```cairo
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [2](#0-1) 

`execute_replace_class` never reads from `contract_class_changes`, so the OS proof is sound even when the replacement class hash has no corresponding entry.

---

### Impact Explanation

Once the state root containing the undeclared class hash is accepted on L1:

1. The contract's `StateEntry.class_hash` is permanently set to a hash for which no compiled class exists.
2. Every subsequent call to the contract causes the OS to look up the class in the compiled class facts bundle; the lookup fails, the execution fails, and the call reverts.
3. Any ERC-20 tokens, ETH, or other assets held in the contract's storage are permanently inaccessible — **permanent freezing of funds**.

This matches the Critical allowed impact: *Permanent freezing of funds*.

---

### Likelihood Explanation

The `replace_class` syscall is a standard, gas-metered syscall available to any Sierra contract. An unprivileged user needs only to:

- Deploy (or already own) a contract written to call `replace_class` with an arbitrary felt value as the class hash.
- Submit the invoke transaction to the network.

The OS Cairo program — which is the authoritative source of truth for proof validity — imposes no constraint on the class hash value. Even if the off-chain blockifier were to reject such a transaction today, the absence of the constraint in the OS means the protocol rule is not enforced at the proof level. Any future blockifier version, alternative sequencer implementation, or sequencer misconfiguration that omits the same check would allow the invalid state to be proven and accepted on L1 with no recourse.

---

### Recommendation

Inside `execute_replace_class`, after reading `class_hash` from the request, add a Cairo constraint that verifies the class hash exists in `contract_class_changes`. Concretely:

- Perform a `dict_read` on `contract_class_changes` keyed by `class_hash`.
- Assert the returned compiled class hash is non-zero (i.e., the class was declared).

This mirrors the pattern already used in `deploy_contract` and `execute_declare_transaction` and closes the gap between the off-chain blockifier check and the on-chain proof constraint.

---

### Proof of Concept

1. **Attacker deploys a malicious contract** whose `__execute__` function issues the `replace_class` syscall with `class_hash = 0xdeadbeef` (an arbitrary undeclared felt).
2. **Attacker submits an invoke transaction** targeting that contract. The transaction is valid (correct nonce, sufficient gas, valid signature).
3. **OS processes the block**: `execute_syscalls` dispatches to `execute_replace_class`; the function writes `DictAccess(key=contract_address, new_value=StateEntry(class_hash=0xdeadbeef, ...))` into `contract_state_changes` with no further constraint.
4. **`state_update` squashes and commits** the change; `compute_contract_state_commitment` hashes the new `StateEntry` (including `class_hash=0xdeadbeef`) into the Patricia tree. The resulting `final_root` is valid.
5. **`validate_compiled_class_facts_post_execution`** only validates classes *actually executed* in this block; `0xdeadbeef` was never executed, so validation passes.
6. **The proof is accepted on L1**; the global state root now encodes `class_hash=0xdeadbeef` for the target contract.
7. **In any subsequent block**, any call to the contract causes the OS to attempt to look up `0xdeadbeef` in the compiled class facts — it is absent — execution fails unconditionally.
8. **All funds in the contract are permanently frozen.**
