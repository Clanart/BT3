**Key code under examination:**

`set_state_header` in `chain/chain/src/state_sync/adapter.rs`: [1](#0-0) 

The function extracts the chunk from the header and runs `validate_chunk_proofs`, but **never checks `chunk.shard_id() == shard_id`**. [2](#0-1) 

The `verify_path` call verifies that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is in the block's `chunk_headers_root` merkle tree. This is a position-encoded merkle proof — it proves chunk A is at shard A's index — but it does **not** verify that the chunk is at shard B's index. The check passes for chunk A regardless of what `shard_id` parameter was supplied.

**The receipt proof check — the critical binding question:** [3](#0-2) 

At line 488, `ReceiptList(shard_id, receipts)` uses the **parameter** `shard_id` (B), not `chunk.shard_id()` (A). This means the receipt proofs in the header must be valid for shard B's incoming receipts. An attacker who constructs a crafted `ShardStateSyncResponseHeader` can supply:
- Chunk A (with its valid merkle proof for shard A's position)
- Shard B's actual incoming receipt proofs (public blockchain data, valid for shard B)
- Shard A's `state_root_node` (consistent with chunk A's `prev_state_root`)

All four validation stages pass. The header is then stored: [4](#0-3) 

Under `StateHeaderKey(shard_B, sync_hash)` — but the embedded chunk belongs to shard A.

**Downstream impact in `set_state_part`:** [5](#0-4) 

`set_state_part` retrieves the stored header for shard B, extracts `prev_state_root()` from the embedded chunk (which is shard A's state root), and uses it to validate and install state parts. The wrong state root is used for shard B.

**However — the receipt proof constraint is a practical blocker:**

The receipt proofs must span from `sync_hash` back to `prev_chunk_header.height_included`. The `prev_chunk_header` in the crafted header is shard A's previous chunk, which may have a different `height_included` than shard B's. The attacker must supply shard B's incoming receipt proofs over the range dictated by shard A's `prev_chunk.height_included`. This is feasible (all data is public), but requires careful construction.

**Reachability:**

The downloader at `chain/client/src/sync/state/downloader.rs` sends a `StateHeaderValidationRequest` with the requested `shard_id` and the peer-supplied header: [6](#0-5) 

A malicious peer responding to a shard B header request with a crafted header containing shard A's chunk reaches `set_state_header(shard_B, sync_hash, crafted_header)` with no prior shard_id binding check.

### Title
Missing `chunk.shard_id() == shard_id` guard in `set_state_header` allows wrong shard's state root to be installed — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` stores a `ShardStateSyncResponseHeader` under `StateHeaderKey(shard_id, sync_hash)` without verifying that the embedded chunk's `shard_id()` equals the `shard_id` parameter. A malicious peer can supply a header containing shard A's chunk (with its valid merkle proof) combined with shard B's incoming receipt proofs, causing the header to be stored under shard B's key with shard A's state root. Subsequent `set_state_part` calls then validate and install state parts against shard A's state root for shard B.

### Finding Description

In `set_state_header` (`chain/chain/src/state_sync/adapter.rs`, lines 368–532):

1. `validate_chunk_proofs` checks internal chunk consistency but not shard identity. [7](#0-6) 
2. `verify_path` against `chunk_headers_root` verifies the chunk hash is in the block's merkle tree at its own shard's position — it does not verify the chunk is at shard B's position. [2](#0-1) 
3. The receipt proof check hashes `ReceiptList(shard_id, receipts)` using the parameter `shard_id` (B), so an attacker who supplies shard B's actual receipt proofs satisfies this check even when the chunk belongs to shard A. [3](#0-2) 
4. No explicit `chunk.shard_id() != shard_id` rejection exists anywhere in the function.
5. The header is stored under `StateHeaderKey(shard_id=B, sync_hash)` with shard A's embedded chunk and state root. [4](#0-3) 

### Impact Explanation

`set_state_part` reads the stored header for shard B, extracts `prev_state_root()` from the embedded chunk (shard A's root), and validates/installs state parts against it. [5](#0-4)  The syncing node installs shard A's state as shard B's state, causing permanent state corruption for shard B on the syncing node.

### Likelihood Explanation

Requires a malicious state-sync peer (any network peer can serve state sync responses) and careful construction of a `ShardStateSyncResponseHeader` mixing shard A's chunk with shard B's receipt proofs. All required data is public. The receipt proof range constraint (determined by shard A's `prev_chunk.height_included`) is a practical complication but not a cryptographic barrier.

### Recommendation

Add an explicit shard identity check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
``` [8](#0-7) 

### Proof of Concept

In `chain/chain/src/state_sync/` (or an integration test), construct a `ShardStateSyncResponseHeader` for shard 0 (valid chunk, valid merkle proof, valid state root node), substitute shard 1's incoming receipt proofs, then call `set_state_header(shard_1, sync_hash, crafted_header)`. Assert that the call returns an error. Without the fix, it succeeds and `get_state_header(shard_1, sync_hash)` returns a header whose embedded chunk has `shard_id() == 0`.
