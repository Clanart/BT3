### Title
Unauthenticated `RemoteProofManagerServer` Allows Pre-Population of Proof Storage, Bypassing Cryptographic Proof Verification for Accepted Transactions — (`crates/apollo_proof_manager/src/communication.rs`, `crates/apollo_transaction_converter/src/transaction_converter.rs`)

---

### Summary

`run_proof_verification` skips cryptographic proof verification when `contains_proof` returns `true`, trusting that any proof already in storage was previously verified. However, the `RemoteProofManagerServer` exposes `SetProof` over the network bound to `0.0.0.0` with no caller authentication. Any network-reachable party can call `SetProof` with arbitrary `proof_facts` and a fake proof body, pre-populating the store. A subsequent transaction carrying those same `proof_facts` will have `verify_proof` silently skipped, and the transaction will be admitted as if it carried a valid ZK proof.

---

### Finding Description

**Root cause — missing source verification in `run_proof_verification`:**

`run_proof_verification` in `crates/apollo_transaction_converter/src/transaction_converter.rs` implements a cache-skip optimization:

```rust
let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;
if contains_proof {
    return Ok(false);   // ← verification entirely skipped
}
// Only reached when proof is NOT already stored:
starknet_proof_verifier::verify_proof(proof_facts, proof)?;
``` [1](#0-0) 

The invariant this relies on is: *only a previously-verified proof can be in storage*. That invariant is broken because `SetProof` is an unauthenticated network endpoint.

**Unauthenticated write path — `RemoteProofManagerServer`:**

`communication.rs` dispatches `ProofManagerRequest::SetProof` directly to `ProofManager::set_proof` with no caller identity check:

```rust
ProofManagerRequest::SetProof(proof_facts, proof) => ProofManagerResponse::SetProof(
    self.set_proof(proof_facts, proof).await
        .map_err(|e| ProofManagerError::ProofStorage(e.to_string())),
),
``` [2](#0-1) 

`ProofManager::set_proof` itself only checks for a duplicate key; it never verifies the proof cryptographically before writing:

```rust
pub async fn set_proof(&self, proof_facts: ProofFacts, proof: Proof) -> ... {
    if self.contains_proof(proof_facts.clone()).await? { return Ok(()); }
    let facts_hash = proof_facts.hash();
    self.proof_storage.set_proof(facts_hash, proof.clone()).await?;
    self.cache.insert(facts_hash, proof);
    Ok(())
}
``` [3](#0-2) 

The remote server is bound to `0.0.0.0` in the distributed deployment configuration, making it reachable from any host that can reach the pod:

```json
"components.proof_manager.remote_server_config.bind_ip": "0.0.0.0"
``` [4](#0-3) 

**Analog to the external bug:**

| External (Solidity) | Sequencer (Rust) |
|---|---|
| Checks `xDomainMessageSender() == INFERNAL_RIFT_ABOVE` (content) | Checks `contains_proof(proof_facts)` (content/existence) |
| Does NOT check `msg.sender == L2_CROSS_DOMAIN_MESSENGER` (source) | Does NOT check who called `SetProof` (source) |
| Anyone deploys a contract returning the right value | Anyone calls `SetProof` on the unauthenticated remote server |
| Privileged action executes without authorization | `verify_proof` is skipped; transaction admitted without ZK verification |

---

### Impact Explanation

A transaction carrying `proof_facts` that reference a valid historical block (satisfying `validate_proof_facts`'s checks on block hash, program hash, and config hash) but backed by a fake or empty proof body will be admitted by both the gateway flow (`spawn_proof_verification`) and the consensus flow (`spawn_verify_and_store_proof`). [5](#0-4) [6](#0-5) 

The accepted transaction carries unverified SNOS proof facts. Under SNIP-36, proof-carrying transactions receive special treatment (e.g., fee discounts, account-validation bypass). Accepting one without a valid proof constitutes an **invalid or unauthorized Starknet transaction accepted through account validation logic** — matching the Critical impact tier.

---

### Likelihood Explanation

The `RemoteProofManagerServer` is a plain TCP/HTTP service with no TLS, no token, and no IP allowlist in the code. In a distributed Kubernetes deployment the pod is reachable from within the cluster. Any compromised co-tenant pod, or any operator mistake that exposes the port externally, gives an attacker a direct write path. The attack requires only a single unauthenticated RPC call followed by a normal transaction submission — no special cryptographic material is needed.

---

### Recommendation

1. **Authenticate `SetProof` callers.** Add a caller-identity check in `communication.rs` (or in the `RemoteComponentServer` middleware) so that only the transaction prover service can invoke `SetProof`. This mirrors the fix in the external report: verify the *source*, not just the *content*.

2. **Verify before storing.** `ProofManager::set_proof` should call `starknet_proof_verifier::verify_proof` before persisting, so the storage invariant ("every stored proof is valid") holds regardless of who calls the endpoint.

3. **Restrict network binding.** Bind the proof manager's remote server to a loopback or cluster-internal address rather than `0.0.0.0`, and enforce a network policy that allows only the transaction prover to connect.

---

        );

        Ok(true)
    }
