### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Breaks Bridge Collateralization - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary

`OmniBridge.initTransfer` uses the caller-supplied `amount` parameter in the emitted `InitTransfer` event rather than the actual amount received by the bridge contract. For fee-on-transfer ERC20 tokens, `safeTransferFrom` deposits `amount - transfer_fee` into the bridge, but the event records `amount`. The NEAR side processes the proof and mints/releases the full `amount`, creating unbacked supply and breaking bridge collateralization.

### Finding Description

In `OmniBridge.initTransfer`, when `tokenAddress` is a regular ERC20 (not a bridge token or custom minter), the bridge pulls tokens from the caller:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
```

Immediately after, the event is emitted using the caller-supplied `amount`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // <-- caller-supplied, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
```

For fee-on-transfer tokens, the bridge vault receives `amount - transfer_fee`, but the event records `amount`. The NEAR bridge contract processes the proof of this event in `fin_transfer_callback`, which uses `init_transfer.amount` (the value from the event) to compute the denormalized amount and ultimately mint or release that full amount to the recipient:

```rust
let transfer_message = TransferMessage {
    ...
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
};
```

The NEAR side then calls `send_tokens` with `transfer_message.amount_without_fee()`, which is derived from the event's `amount`, not from what the EVM bridge actually holds.

### Impact Explanation

Every `initTransfer` with a fee-on-transfer token creates a deficit: the EVM bridge holds `amount - transfer_fee` but NEAR mints/releases `amount`. Repeated transfers drain the bridge's real reserves while inflating the bridged token supply on NEAR. This is a direct accounting corruption that breaks bridge collateralization — the canonical "unbacked mint" impact.

### Likelihood Explanation

Any registered ERC20 token that implements a transfer fee (e.g., tokens with configurable fee switches, rebasing tokens, or tokens that deduct fees on transfer) triggers this path. The `initTransfer` function has no guard against fee-on-transfer tokens. An unprivileged user only needs to call `initTransfer` with such a token to exploit this.

### Recommendation

Measure the actual balance change rather than trusting the caller-supplied `amount`. Replace the `safeTransferFrom` call with a balance-before/after check:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived == amount, "Fee-on-transfer token not supported");
```

Alternatively, explicitly document and enforce that fee-on-transfer tokens are not supported, and add a token registry check that rejects such tokens at registration time.

### Proof of Concept

1. A fee-on-transfer ERC20 token `T` (1% fee on every transfer) is registered in the Omni Bridge on EVM and NEAR.
2. Attacker calls `initTransfer(T, 1000, 0, 0, nearRecipient, "")`.
3. `safeTransferFrom` transfers 1000 from attacker; bridge receives 990 (1% fee deducted by token).
4. Event emits `amount = 1000`.
5. Relayer submits proof to NEAR `fin_transfer`.
6. NEAR `fin_transfer_callback` reads `init_transfer.amount = 1000`, mints 1000 bridged-T to `nearRecipient`.
7. Bridge EVM vault: 990 T. NEAR bridged supply: 1000 T. Deficit: 10 T per transfer.
8. After 100 such transfers, the bridge is undercollateralized by 1000 T. The last ~10 users to withdraw on EVM cannot redeem their bridged tokens. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
