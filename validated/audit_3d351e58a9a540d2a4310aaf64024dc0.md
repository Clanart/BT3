### Title
Missing Pre-Lock Validation of Normalized Transfer Amount Causes Permanent Fund Lock - (File: `near/omni-bridge/src/lib.rs`)

### Summary

`init_transfer` locks user tokens before validating that `normalize_amount(amount - fee, decimals) > 0`. The zero-amount guard only fires in `sign_transfer`, which is called after the lock. Because no public cancel path exists, any transfer whose `amount - fee` is below the minimum representable unit on the destination chain results in permanently irrecoverable funds.

### Finding Description

`init_transfer` stores the `TransferMessage` and locks/burns tokens via `init_transfer_internal` with only one fee check:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

This check does **not** verify that `amount - fee` is large enough to survive decimal normalization. The normalization uses floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [2](#0-1) 

The zero-amount guard only appears in `sign_transfer`, which is called by a relayer **after** the tokens are already locked:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee()...,
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [3](#0-2) 

Tokens are locked in `init_transfer_internal` before `sign_transfer` is ever called:

```rust
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
``` [4](#0-3) 

There is no public cancel or refund function. `remove_transfer_message` is only called internally from `claim_fee_callback` and `sign_transfer_callback`, both of which require a successful MPC signing or proof — impossible when `sign_transfer` always panics. [5](#0-4) 

### Impact Explanation

**Critical — Permanent freezing of user funds.**

For any token registered with `origin_decimals > decimals` (e.g., a NEAR-native token with 24 decimals bridging to an EVM chain with 6 decimals, giving `diff_decimals = 18`), any transfer where `amount - fee < 10^18` will:

1. Successfully lock/burn the user's tokens in `init_transfer`.
2. Permanently revert every subsequent `sign_transfer` call with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
3. Leave the `TransferMessage` in `pending_transfers` with no recovery path.

The locked tokens can never be unlocked because `unlock_tokens` is only called from `send_fee_internal` (reached via `claim_fee_callback`) and `revert_lock_actions` (reached via `fin_transfer_send_tokens_callback`), neither of which is reachable for a transfer that cannot be signed. [6](#0-5) 

### Likelihood Explanation

Any unprivileged bridge user who sends a token amount below the minimum representable unit on the destination chain triggers this. For a 24→6 decimal token pair, the threshold is `10^18` base units — a user sending 0.5 NEAR-equivalent of such a token would permanently lose their funds. The `update_transfer_fee` function cannot rescue the transfer because fees can only be increased, not decreased. [7](#0-6) 

### Recommendation

Add the normalization check inside `init_transfer` (or `init_transfer_internal`) **before** locking tokens:

```rust
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    token_id.clone(),
).near_expect(BridgeError::FailedToGetTokenAddress);

let decimals = self.token_decimals
    .get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);

let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

This mirrors the existing guard in `sign_transfer` but places it before any state mutation, preventing tokens from being locked into an unrecoverable state.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 6` (18-decimal difference; minimum transferable unit = `10^18`).
2. User calls `ft_transfer_call` with `amount = 5 * 10^17` (0.5 NEAR-equivalent), `fee = 0`, valid EVM recipient.
3. `init_transfer` passes the `fee < amount` check and locks `5 * 10^17` tokens.
4. Relayer calls `sign_transfer` → `normalize_amount(5 * 10^17, {decimals:6, origin_decimals:24}) = 0` → panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
5. No public cancel function exists; `pending_transfers` retains the entry forever.
6. The `5 * 10^17` tokens are permanently locked in the bridge contract.
