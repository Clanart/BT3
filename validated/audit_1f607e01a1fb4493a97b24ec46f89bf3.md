### Title
Missing `chunk.shard_id()` == `shard_id` Guard in `set_state_header` Allows Cross-Shard State Root Substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` whose embedded chunk belongs to a different shard than the `shard_id` argument. All five validation steps pass because none of them compare `chunk.shard_id()` to the caller-supplied `shard_id`. The header is then stored under `StateHeaderKey(shard_id, sync_hash)`, and every subsequent `set_state_part` call for that shard extracts `state_root` from the wrong chunk, validating all state parts against the wrong shard's state root.

---

### Finding Description

`set_state_header` in `chain/chain/src/state_sync/adapter.rs` performs five checks before persisting the header:

**Check 1-2 — `validate_chunk_proofs`** [1](#0-0) [2](#0-1) 

`validate_chunk_proofs` only verifies the chunk's internal hash consistency, transaction root, and outgoing receipts root. It never compares `chunk.shard_id()` to any expected shard.

**Check 3a — `verify_path` against `chunk_headers_root`** [3](#0-2) 

This calls the plain `verify_path` (not `verify_path_with_index`): [4](#0-3) 

`verify_path` only checks that the item's hash is reachable from the root via the supplied path. It does **not** check the leaf's position (shard index) in the tree. `verify_path_with_index` (line 121) exists and does check position via `verify_path_matches_index`, but it is not used here. A chunk from shard 1 with its own valid Merkle path will pass this check even when `shard_id=0` is passed.

**Check 4 — Receipt proofs** [5](#0-4) 

Receipt hashes are computed as `ReceiptList(shard_id, receipts)` using the caller's `shard_id` (0), not `chunk.shard_id()`. An attacker supplies valid incoming-receipt proofs for shard 0 (all public on-chain data) while embedding shard 1's chunk. These proofs are independent of which chunk is in the header.

**Check 5 — `validate_state_root_node`** [6](#0-5) 

This validates `state_root_node` against `chunk_inner.prev_state_root()` — the attacker-controlled chunk's state root, not shard 0's. An attacker provides a `state_root_node` consistent with shard 1's `prev_state_root`, which passes trivially.

**Storage — under the wrong key** [7](#0-6) 

The header (containing shard 1's chunk) is stored under `StateHeaderKey(shard_id=0, sync_hash)`.

**Downstream corruption in `set_state_part`** [8](#0-7) 

`set_state_part` reads the stored header, extracts `state_root = chunk.take_header().take_inner().prev_state_root()` — now shard 1's state root — and validates all incoming state parts against it. The exact corrupted value is **`state_root` used to validate `StateParts` for `shard_id=0`**, which becomes shard 1's `prev_state_root` instead of shard 0's.

---

### Impact Explanation

A syncing node installs shard 1's state as shard 0's state. After `set_state_finalize`, the node operates with the wrong committed state for shard 0. All account balances, contract storage, and access keys for shard 0 are replaced with shard 1's data. The node will diverge from the canonical chain on the first block it processes for shard 0. [9](#0-8) 

---

### Likelihood Explanation

Any peer acting as a state sync provider can trigger this. The state sync downloader requests headers from peers without authenticating the provider's identity or cross-checking the returned chunk's shard against the requested shard. [10](#0-9) 

The attacker only needs access to public on-chain data (shard 1's chunk, its Merkle proof, and shard 0's incoming receipt proofs) — no validator or privileged credentials are required.

---

### Recommendation

Add an explicit shard identity check at the top of `set_state_header`, immediately after extracting the chunk:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {} does not match expected shard_id {}",
        chunk.shard_id(), shard_id
    )));
}
```

Additionally, replace the plain `verify_path` call with `verify_path_with_index` (passing the shard index as the leaf position) so that the Merkle proof is bound to the correct position in the `chunk_headers_root` tree. [11](#0-10) 

---

### Proof of Concept

```rust
// In a cargo test (integration-tests or chain/chain):
// 1. Produce a multi-shard chain (shard 0 and shard 1).
// 2. Obtain the legitimate header for shard 1 via get_state_response_header(shard_id=1, sync_hash).
// 3. Call set_state_header(shard_id=0, sync_hash, shard_1_header).
// 4. Assert Ok(()) — all checks pass.
// 5. Read back the stored header via get_state_header(shard_id=0, sync_hash).
// 6. Assert stored_header.chunk_prev_state_root() != correct_shard_0_state_root.
//    (It equals shard 1's prev_state_root instead.)
```

The existing integration test at `integration-tests/src/tests/client/process_blocks.rs:1994` already shows the exact call pattern; substituting a cross-shard header requires no additional infrastructure. [12](#0-11)

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)

    pub fn set_state_finalize(
        &mut self,
        shard_id: ShardId,
        sync_hash: CryptoHash,
    ) -> Result<(), Error> {
        let shard_state_header = self.state_sync_adapter.get_state_header(shard_id, sync_hash)?;
        let mut height = shard_state_header.chunk_height_included();
        let mut chain_update = self.chain_update();
        let shard_uid = chain_update.set_state_finalize(shard_id, sync_hash, shard_state_header)?;
        chain_update.commit()?;

        // We restored the state on height `shard_state_header.chunk.header.height_included`.
        // Now we should build a chain up to height of `sync_hash` block.
        loop {
            height += 1;
            let mut chain_update = self.chain_update();
            // Result of successful execution of set_state_finalize_on_height is bool,
            // should we commit and continue or stop.
            if chain_update.set_state_finalize_on_height(height, shard_id, sync_hash)? {
                chain_update.commit()?;
            } else {
                break;
            }
        }

        let flat_storage_manager = self.runtime_adapter.get_flat_storage_manager();
        if let Some(flat_storage) = flat_storage_manager.get_flat_storage_for_shard(shard_uid) {
            let header = self.get_block_header(&sync_hash)?;
            flat_storage.update_flat_head(header.prev_hash()).unwrap();
        }

        Ok(())
    }
