### Title
Missing Duplicate-Key Check in `MerkleBlob::batch_insert` Corrupts DataLayer Merkle Tree Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. When the tree already contains ≥ 2 leaves — the normal production state — **every** entry in a batch bypasses the existence check. Supplying a batch that contains a key already present in the tree (or a key that appears twice within the same batch) silently writes a second leaf node for that key into the raw blob, desynchronises the `BlockStatusCache`, and produces a permanently incorrect Merkle root. The Python binding `MerkleBlob.batch_insert()` exposes this path directly to callers.

---

### Finding Description

`MerkleBlob::insert` guards against re-insertion at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` (lines 570–657) takes a different code path. It only calls `self.insert()` (with its guards) for the **last two** entries popped from the input vector, and only when the tree currently holds ≤ 1 leaf:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { ... };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;  // guarded
    }
}

for ((key, value), hash) in keys_values_hashes {   // remaining entries — NO guard
    let new_leaf_index = self.get_new_index();
    let new_block = Block { node: Node::Leaf(LeafNode { hash, key, value, .. }), .. };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // writes directly
    indexes.push(new_leaf_index);
}
```

When the tree already has ≥ 2 leaves (the `if` branch is skipped entirely), **all** entries in the batch are written through `insert_entry_to_blob` without any existence check.

`BlockStatusCache::add_leaf` (called by `insert_entry_to_blob` for leaf nodes) uses a plain `HashMap::insert`, which silently overwrites the existing cache entry for a duplicate key:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // overwrites silently
    self.leaf_hash_to_index.insert(leaf.hash, index);   // overwrites silently
}
```

After the call:
- The blob contains **two** leaf nodes for the same key at different block indices.
- The cache tracks only the **last-written** index; the first leaf's index is neither freed nor tracked.
- The Merkle root is computed over both leaves, producing a hash that does not correspond to any valid single-key state.
- `check_integrity()` will detect the mismatch (`leaf_count != key_to_index_cache_length`), but it is not called automatically on every mutation.

The Python binding `py_batch_insert` (lines 1503–1519) passes the caller-supplied list directly to `batch_insert` with no pre-deduplication:

```rust
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
```

---

### Impact Explanation

A caller (or an attacker who controls DataLayer batch-update input) can supply a batch that includes a key already present in the store, or a key repeated within the same batch. The result is a permanently corrupted `MerkleBlob`:

- The committed Merkle root no longer represents the actual key-value set.
- Proofs of inclusion generated from the corrupted tree will be structurally invalid or will prove the wrong state.
- Proofs of exclusion for the duplicate key become impossible to generate correctly.
- Any downstream system that trusts the root hash (e.g., DataLayer delta synchronisation, cross-chain proofs) will accept a forged or inconsistent state.

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

---

### Likelihood Explanation

The Python binding is a public API surface. Any code path that feeds externally-sourced key-value pairs into `batch_insert` without prior deduplication is vulnerable. Because `insert` does enforce the guard, developers may assume `batch_insert` does too — the asymmetry is not documented. The condition `leaf_count <= 1` means the partial guard only applies to a freshly-initialised tree; any store with ≥ 2 existing entries (the overwhelming majority of production stores) receives zero protection.

---

### Recommendation

Add duplicate-key and duplicate-hash checks at the top of `batch_insert`, mirroring `insert`:

```rust
pub fn batch_insert(&mut self, keys_values_hashes: Vec<((KeyId, ValueId), Hash)>) -> Result<(), Error> {
    for ((key, _), hash) in &keys_values_hashes {
        if self.block_status_cache.contains_key(*key) {
            return Err(Error::KeyAlreadyPresent());
        }
        if self.block_status_cache.contains_leaf_hash(hash) {
            return Err(Error::HashAlreadyPresent());
        }
    }
    // also check for intra-batch duplicates
    ...
}
```

Alternatively, enforce the invariant inside the fast-path loop before calling `insert_entry_to_blob`. Also add a fuzz target that specifically exercises `batch_insert` with duplicate keys against a pre-populated tree (the existing fuzz targets only call `insert`, which is already guarded).

---

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def h(n):
    return bytes(hashlib.sha256(n.to_bytes(8, "big")).digest())

blob = MerkleBlob(blob=bytearray())

# Populate tree with 3 leaves so leaf_count > 1
for i in range(3):
    blob.insert(KeyId(i), ValueId(i), h(i))

# Key 0 already exists. batch_insert with a 1-entry batch (leaf_count > 1
# means the guarded branch is skipped entirely).
blob.batch_insert(
    [(KeyId(0), ValueId(999))],   # duplicate key
    [h(999)],
)

# Tree now has two leaf nodes for KeyId(0).
# Root hash is wrong; check_integrity() will raise.
blob.check_integrity()   # raises IntegrityKeyToIndexCacheLengthError
```

The call succeeds (no exception from `batch_insert`) and the tree is silently corrupted. The root hash returned by `get_root_hash()` after `calculate_lazy_hashes()` reflects the corrupted state and will not match any legitimate single-key-set hash.
