### Title
`update_transfer_fee` Allows Fee That Makes `sign_transfer` Permanently Fail, Permanently Locking User Funds — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`update_transfer_fee` permits the sender to raise `fee.fee` up to `amount - 1` with no check that the post-fee remainder survives decimal normalization. `sign_transfer` independently requires `normalize_amount(amount - fee) > 0`. For any token whose NEAR-side decimals exceed its destination-chain decimals (the normal case for bridged tokens), a sender can set a fee that leaves a sub-unit remainder, making `normalize_amount` return 0 and causing every subsequent `sign_transfer` call to permanently revert with `ERR_INVALID_AMOUNT_TO_TRANSFER`. Because no public cancel path exists, the locked or burned tokens are irrecoverable.

---

### Finding Description

**Root cause — missing normalization guard in `update_transfer_fee`**

`update_transfer_fee` enforces only two constraints on the new token fee:

```rust
// near/omni-bridge/src/lib.rs  L399-L402
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [1](#0-0) 

The upper bound is `fee.fee < amount` — i.e., the fee may be set to `amount - 1`. There is no check that `amount - fee` survives the decimal normalization that `sign_transfer` will later apply.

**`sign_transfer` requires the normalized remainder to be non-zero**

```rust
// near/omni-bridge/src/lib.rs  L475-L485
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

`normalize_amount` uses floor division:

```rust
// near/omni-bridge/src/lib.rs  L2784-L2787
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [3](#0-2) 

For a token registered with `origin_decimals = 24` and `decimals = 18` (a common configuration for NEAR-native tokens bridging to EVM), the divisor is `10^6`. Any `amount - fee < 1_000_000` normalizes to 0.

**Exploit path**

1. Sender initiates a transfer: `amount = 1_000_000` (NEAR-side units), `fee = 0`.
2. Token is registered with `origin_decimals = 24`, `decimals = 18`.
3. Sender calls `update_transfer_fee` with `fee.fee = 999_999` — valid because `999_999 < 1_000_000`.
4. Stored state: `amount = 1_000_000`, `fee.fee = 999_999`, `amount_without_fee() = 1`.
5. Any trusted relayer calls `sign_transfer`. The contract computes `normalize_amount(1, {24,18}) = 1 / 10^6 = 0` and panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
6. `sign_transfer` will revert on every future call for this transfer ID.
7. No public cancel or refund function exists; `remove_transfer_message` is only reachable internally via `claim_fee_callback` (requires a valid proof of finalization) or `sign_transfer_callback` when `fee.is_zero()` — neither is reachable here. [4](#0-3) 

**Token fate**

During `init_transfer_internal`, deployed (bridged) tokens are burned immediately; native tokens are locked in the contract. [5](#0-4) 

In either case, once `sign_transfer` is permanently blocked, the funds are irrecoverable.

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds** — matching the allowed High/Critical impact class.

The sender's tokens (burned or locked at transfer initiation) can never be released. The transfer record stays in `pending_transfers` indefinitely with no protocol-level escape hatch. This is a direct, on-chain permanent freeze triggered by a public function callable by any bridge user on their own transfer.

---

### Likelihood Explanation

**Moderate.** The condition requires:
1. A token with `origin_decimals > decimals` — the standard configuration for any NEAR-native token bridging to an EVM chain (e.g., 24 → 18).
2. The sender setting `fee.fee` to a value within `amount - 10^(origin_decimals - decimals)` of `amount`.

A user who wants to increase their fee to attract a relayer (the intended use of `update_transfer_fee`) and sets it "as high as possible" without understanding decimal normalization will trigger this accidentally. The protocol provides no warning and accepts the call without error.

---

### Recommendation

Add a normalization guard inside `update_transfer_fee` before committing the new fee:

```rust
// After computing the new fee, before insert_raw_transfer:
let token_address = self.get_token_address(
    transfer.message.get_destination_chain(),
    self.get_token_id(&transfer.message.token),
).near_expect(BridgeError::FailedToGetTokenAddress);

let decimals = self.token_decimals
    .get(&token_address)
    .near_expect(BridgeError::TokenDecimalsNotFound);

let normalized = Self::normalize_amount(
    transfer.message.amount.0 - fee.fee.0,
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

This mirrors the guard already present in `sign_transfer` and closes the inconsistency.

### Proof of Concept

**State setup** (unit-test style, no privileged access required):

```
token: origin_decimals=24, decimals=18  (diff = 6, divisor = 1_000_000)
transfer: amount = 1_000_000, fee.fee = 0
```

**Step 1** — sender calls `update_transfer_fee` with `fee.fee = 999_999`:

```rust
// update_transfer_fee check passes:
//   999_999 >= 0  ✓
//   999_999 < 1_000_000  ✓
// No normalization check → fee stored successfully.
``` [1](#0-0) 

**Step 2** — trusted relayer calls `sign_transfer`:

```rust
// amount_without_fee() = 1_000_000 - 999_999 = 1
// normalize_amount(1, {24, 18}) = 1 / 1_000_000 = 0
// require!(0 > 0)  → PANIC: ERR_INVALID_AMOUNT_TO_TRANSFER
``` [2](#0-1) 

**Step 3** — every subsequent `sign_transfer` call for this `transfer_id` panics identically. `claim_fee` requires a proof of a finalized destination-chain event that can never exist. No cancel function is available. The `1_000_000` NEAR-side tokens (burned or locked at step 0) are permanently lost.
