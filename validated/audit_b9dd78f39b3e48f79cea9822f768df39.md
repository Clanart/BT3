### Title
Unregistered Token Accepted in `initTransfer` Causes Permanent Fund Lock — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` accepts any arbitrary ERC20 token address without verifying that the token is registered in the bridge's `ethToNearToken` mapping. Tokens transferred this way are permanently locked in the bridge because the NEAR side panics on `TokenDecimalsNotFound` and can never produce a release signature for an unregistered token.

---

### Finding Description

In `OmniBridge.sol`, `initTransfer` branches on three conditions for a non-zero `tokenAddress`:

1. `customMinters[tokenAddress] != address(0)` → custom-minter burn path
2. `isBridgeToken[tokenAddress]` → bridge-token burn path
3. **else** → `IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount)` [1](#0-0) 

The `else` branch contains no check that `ethToNearToken[tokenAddress]` is non-empty, i.e., that the token has ever been registered via `deployToken` or `addCustomToken`. Any ERC20 token — including tokens that have never been logged or deployed through the bridge — passes through this branch, is pulled from the caller, and is locked in the bridge contract.

After locking, `initTransferExtension` emits a Wormhole message carrying the raw EVM token address. [2](#0-1) 

On the NEAR side, `fin_transfer_callback` immediately looks up `token_decimals` for the incoming token address and calls `near_expect(BridgeError::TokenDecimalsNotFound)`, which panics if the token is absent from the registry: [3](#0-2) 

Because the token was never registered, this lookup always fails. The NEAR contract can never finalize the transfer, and therefore the MPC signer will never produce a `finTransfer` signature for the EVM side. The EVM `OmniBridge` contract has no admin rescue path for arbitrary locked ERC20 tokens; the only egress is `finTransfer`, which requires a valid MPC signature. [4](#0-3) 

---

### Impact Explanation

Tokens sent via `initTransfer` with an unregistered address are irrecoverably locked in the bridge contract. There is no admin sweep function, no refund path, and no way for the NEAR side to ever sign a release. This constitutes **permanent, irrecoverable lock of user funds** — matching the allowed High impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

The entry point is fully public and requires no privilege. Any user who mistakenly (or deliberately) calls `initTransfer` with a token that has not gone through the `logMetadata` → `deployToken` lifecycle triggers the lock. Given that the `else` branch is the natural fallback for native EVM tokens (the intended use case for locking real assets), the probability of accidental misuse is non-trivial. A malicious actor could also deliberately grief other users by front-running and locking their tokens if they control the token contract.

---

### Recommendation

Add an explicit allowlist check at the top of the `else` branch in `initTransfer`:

```solidity
} else {
    require(
        bytes(ethToNearToken[tokenAddress]).length != 0,
        "ERR_TOKEN_NOT_REGISTERED"
    );
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
}
```

This mirrors the pattern already used in `setMetadata` and `upgradeToken`, which both gate on `isBridgeToken` or `ethToNearToken` membership before acting. [5](#0-4) 

---

### Proof of Concept

1. Deploy or obtain any ERC20 token `FAKE` that is **not** registered in the bridge (i.e., `ethToNearToken[FAKE] == ""`).
2. Approve the bridge to spend `FAKE` tokens.
3. Call:
   ```solidity
   omniBridge.initTransfer(
       FAKE,          // unregistered token
       1e18,          // amount
       0,             // fee
       0,             // nativeFee
       "near:victim.near",
       ""
   );
   ```
4. `FAKE` tokens are transferred from the caller into `OmniBridge` (line 407–410). A Wormhole message is emitted.
5. The NEAR relayer submits the Wormhole VAA to `fin_transfer` → `fin_transfer_callback`.
6. NEAR executes `self.token_decimals.get(&init_transfer.token).near_expect(BridgeError::TokenDecimalsNotFound)` → **panics**. The callback reverts; no transfer record is created; no MPC signature is ever produced.
7. The EVM bridge holds the `FAKE` tokens with no mechanism to release them. Funds are permanently locked. [6](#0-5) [3](#0-2)
