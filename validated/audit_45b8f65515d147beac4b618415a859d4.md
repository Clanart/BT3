### Title
Hint-Supplied `header_commitments` in `get_block_hashes()` Produce a Block Hash That Does Not Commit to Actual Transactions, Enabling Chain Split — (File: `block_hash.cairo`)

---

### Summary

In `get_block_hashes()`, the entire `BlockHeaderCommitments` struct (`transaction_commitment`, `event_commitment`, `receipt_commitment`, `state_diff_commitment`, `packed_lengths`), along with `gas_prices_hash`, `starknet_version`, and `previous_block_hash`, are all loaded exclusively via the `%{ GetBlockHashes %}` hint with **zero Cairo-level assertions** verifying them against actual execution. The OS itself documents this gap: *"Currently, the header commitments and gas prices are not computed by the OS."* The resulting block hash therefore does not cryptographically bind to the actual transaction set of the block. A malicious sequencer can produce two structurally valid OS proofs for the same block number with different transaction sets, creating an unresolvable chain split — the direct StarkNet analog of the Clique single-signer fork-choice failure.

---

### Finding Description

`calculate_block_hash()` hashes fourteen fields into the block hash: [1](#0-0) 

Of those fourteen fields, only `block_info` (block number, sequencer address, timestamp) and `state_root` are derived from verified OS execution. The remaining four inputs — `header_commitments`, `gas_prices_hash`, `starknet_version`, and `previous_block_hash` — are declared as `local` variables and populated entirely by the `%{ GetBlockHashes %}` hint: [2](#0-1) 

There is no Cairo `assert` between the hint call and `calculate_block_hash()`. The `%{ CheckBlockHashConsistency %}` at line 79 is a **hint** — it executes only in the prover's Python environment and is invisible to the STARK verifier. It imposes no constraint on the proof.

The codebase explicitly acknowledges all three layers of this gap:

1. **`block_hash.cairo` line 60–61** — *"Currently, the header commitments and gas prices are not computed by the OS. TODO(Yoni, 1/1/2027): compute the header commitments and gas prices."* [3](#0-2) 

2. **`output.cairo` line 39** — *"Currently, the block hash is not enforced by the OS."* [4](#0-3) 

3. **`os_utils.cairo` lines 127–131** — *"NOTE: both the previous block hash and previous state root are guessed, and the OS does not verify their consistency (unlike the new hash and root). The consumer of the OS output should verify both."* [5](#0-4) 

A compounding secondary flaw exists in `write_block_number_to_block_hash_mapping()`: the `old_block_hash` written into the on-chain block-hash mapping contract is also hint-supplied with no Cairo verification: [6](#0-5) 

The comment at line 64–65 confirms: *"Currently, the block hash mapping is not enforced by the OS."*

---

### Impact Explanation

Because `transaction_commitment`, `event_commitment`, `receipt_commitment`, and `state_diff_commitment` are all hint-supplied and unconstrained, a malicious sequencer can produce **two structurally valid OS proofs for the same block number** with entirely different transaction sets. Both proofs satisfy all Cairo constraints because the only verified field that differs between them — `state_root` — can be made to match by choosing transactions that produce the same net state change, or the sequencer can simply publish both proofs and let the L1 verifier accept whichever arrives first.

Additionally, because `previous_block_hash` is hint-supplied and unverified, the sequencer can produce proofs that do not form a valid hash chain, breaking chain continuity at the protocol level.

**Impact: Unintended chain split (network partition) — High; Direct loss of funds — Critical** (any bridge or exchange that treats a block hash as a finality commitment for a deposit transaction can be double-spent against).

---

### Likelihood Explanation

The vulnerability is structural and present in every block the OS produces — it is not a corner case. The TODO deadline of `1/1/2027` confirms the gap will persist for an extended period. The sequencer is currently centralized, meaning a single compromised or malicious operator can exploit this without any collusion. Likelihood is **Medium** (requires sequencer compromise or malice, but no cryptographic break or external dependency).

---

### Recommendation

1. **Compute `header_commitments` inside the OS** from the actual transaction execution outputs (transaction hashes, events, receipts, state diff) rather than accepting them from a hint. Replace the `%{ GetBlockHashes %}` hint population of `header_commitments` with Cairo functions that derive each commitment from the verified execution trace.
2. **Enforce `previous_block_hash`** by reading it from the verified block-hash storage contract (already populated by `write_block_number_to_block_hash_mapping`) and asserting equality with the hint-supplied value, rather than delegating verification to an external consumer.
3. **Remove the `%{ CheckBlockHashConsistency %}` hint** and replace it with Cairo `assert` statements so the constraint is part of the proof.
4. **Enforce the block-hash mapping write** by computing `old_block_hash` from the verified stored value rather than accepting it from `%{ GetOldBlockNumberAndHash %}`.

### Proof of Concept

```
Block N execution — Fork A (published):
  Transactions T_A = [invoke deposit(exchange, 1000 STRK)]
  Sequencer supplies hint: header_commitments.transaction_commitment = H(T_A)
  OS computes state_root_A (deposit credited)
  Block hash BH_A = Poseidon(VERSION, N, state_root_A, seq, ts, packed, sdiff_A, H(T_A), ...)
  Proof_A submitted to L1 → accepted (all Cairo constraints satisfied)
  Exchange sees deposit in block N, credits user

Block N execution — Fork B (withheld, then published):
  Transactions T_B = []  (no deposit)
  Sequencer supplies hint: header_commitments.transaction_commitment = H(T_B)
  OS computes state_root_B (no deposit)
  Block hash BH_B = Poseidon(VERSION, N, state_root_B, seq, ts, packed, sdiff_B, H(T_B), ...)
  Proof_B submitted to L1 → also accepted (all Cairo constraints satisfied)

Result:
  Two valid proofs exist for block N with different transaction sets.
  Nodes that received Proof_A and nodes that received Proof_B disagree on the canonical chain.
  The deposit transaction exists in one fork and not the other.
  The exchange has credited funds that do not exist on the fork that ultimately prevails.
```

The root cause is identical in structure to the Clique single-signer vulnerability: the protocol provides no mechanism — at the proof level — to distinguish between two conflicting valid attestations for the same block height, because the block hash does not cryptographically bind to the actual transaction set. func get_block_hashes{poseidon_ptr: PoseidonBuiltin*}(block_info: BlockInfo*, state_root: felt) -> (
    previous_block_hash: felt, new_block_hash: felt
) {
    alloc_locals;
    local previous_block_hash;
    // Currently, the header commitments and gas prices are not computed by the OS.
    // TODO(Yoni, 1/1/2027): compute the header commitments and gas prices.
    local header_commitments: BlockHeaderCommitments*;
    local gas_prices_hash;
    // TODO(Yoni): move to global context, and consider enforcing a specific version for the
    // non-virtual OS.
    local starknet_version;

    %{ GetBlockHashes %}

    let block_hash = calculate_block_hash(
        block_info=block_info,
        header_commitments=header_commitments,
        gas_prices_hash=gas_prices_hash,
        state_root=state_root,
        previous_block_hash=previous_block_hash,
        starknet_version=starknet_version,
    );

    %{ CheckBlockHashConsistency %}

    return (previous_block_hash=previous_block_hash, new_block_hash=block_hash);
}
