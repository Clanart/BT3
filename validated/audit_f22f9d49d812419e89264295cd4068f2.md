### Title
Attacker-Controlled `sender_address` Bypasses `__validate__` in `skip_stateful_validations`, Admitting Unsigned Invoke Transactions to Mempool ŌĆö (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator uses the attacker-supplied `sender_address` field from an incoming invoke transaction to query `account_tx_in_pool_or_recent_block`. When that query returns `true`, the function returns `skip_validate = true`, causing the gateway to set `execution_flags.validate = false` and skip the account's `__validate__` entry point entirely. Because the `sender_address` is fully attacker-controlled and is never verified to match the actual submitter, any attacker who observes a victim's `deploy_account` transaction in the mempool can submit an invoke transaction claiming to be from the victim, bypass signature verification at the gateway, and have the transaction admitted to the mempool without a valid signature.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` implements a UX feature that allows a user to submit a `deploy_account` and a follow-up `invoke` (nonce=1) simultaneously, before the deploy is confirmed:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();   // ŌåÉ attacker-controlled
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())  // ŌåÉ checked against attacker-supplied address
                .await
                ...
        }
    }
    Ok(false)
}
``` [1](#0-0) 

When `skip_stateful_validations` returns `true`, the caller `run_validate_entry_point` constructs an `AccountTransaction` with `validate: false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

This causes `StatefulValidator::perform_validations` to return `Ok(())` immediately without calling the account's `__validate__` entry point:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [3](#0-2) 

The three conditions that trigger the skip are:
1. Transaction type is `Invoke`
2. `tx.nonce() == 1`
3. `account_nonce == 0` (on-chain state shows account not yet deployed)
4. `account_tx_in_pool_or_recent_block(tx.sender_address()) == true`

All four conditions are satisfiable by an attacker who monitors the mempool for pending `deploy_account` transactions. The attacker supplies an arbitrary `sender_address` pointing to a victim whose `deploy_account` is pending. The gateway never verifies that the submitter controls the claimed `sender_address`.

The mempool's `account_tx_in_pool_or_recent_block` checks:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

This is a pure lookup against the attacker-supplied address with no caller-identity binding.

### Impact Explanation

An attacker can submit an `Invoke V3` transaction with:
- `sender_address` = any victim address whose `deploy_account` is pending in the mempool
- `nonce` = 1
- An arbitrary or empty signature

The gateway will admit this transaction to the mempool without running the account's `__validate__` entry point. The admitted transaction occupies the `(victim_address, nonce=1)` slot in the mempool. If the victim also submitted a legitimate nonce=1 invoke, the attacker can outbid it via fee escalation rules, displacing the victim's valid transaction. When the batcher eventually executes the attacker's transaction, it will fail (the batcher runs `__validate__` with `validate=true`), but the victim's legitimate nonce=1 transaction has already been evicted.

This matches the allowed impact: **High ŌĆö Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

- `deploy_account` transactions are visible in the mempool before confirmation.
- The attacker only needs to observe a pending `deploy_account` and submit a competing nonce=1 invoke before the deploy is committed.
- No privileged access is required; any unprivileged network participant can submit transactions to the gateway.
- The attack window is the time between the victim's `deploy_account` entering the mempool and being committed to a block.

### Recommendation

Bind the skip-validation decision to the actual submitter rather than the claimed `sender_address`. Concretely, the skip should only be granted when the gateway can confirm that the incoming invoke transaction was submitted by the same party who submitted the pending `deploy_account`. One approach is to require the `deploy_account` transaction hash to be provided as an explicit parameter (as the `native_blockifier` path does via `deploy_account_tx_hash`), and verify that the hash corresponds to a pending deploy for the claimed `sender_address`. Alternatively, require that the signature on the invoke transaction is valid against the class hash declared in the pending `deploy_account`, before granting the skip.

### Proof of Concept

1. Victim submits `deploy_account` for address `V` (nonce=0, class_hash=`C`). The transaction enters the mempool.
2. Attacker observes the mempool and sees `V`'s `deploy_account` is pending.
3. Attacker constructs an `Invoke V3` transaction: `sender_address=V`, `nonce=1`, `signature=[0x0, 0x0]` (garbage), `calldata=<arbitrary>`, `resource_bounds` above minimum.
4. Attacker submits the transaction to the gateway.
5. Gateway runs `skip_stateful_validations`:
   - `tx.nonce() == 1` Ō£ō
   - `account_nonce == 0` (V not yet deployed) Ō£ō
   - `account_tx_in_pool_or_recent_block(V) == true` (V's deploy_account is in the pool) Ō£ō
   - Returns `true` ŌåÆ `validate = false`
6. `run_validate_entry_point` skips `__validate__`; the transaction is forwarded to the mempool.
7. The attacker's transaction now occupies slot `(V, nonce=1)` in the mempool with a garbage signature.
8. If the victim also submitted a nonce=1 invoke, the attacker can outbid it via fee escalation, evicting the victim's valid transaction.
9. The batcher eventually executes the attacker's transaction; `__validate__` runs and fails; the transaction reverts ŌĆö but the victim's nonce=1 transaction is permanently gone from the mempool.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
