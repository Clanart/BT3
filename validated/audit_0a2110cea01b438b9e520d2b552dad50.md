Audit Report

## Title
Unprivileged `refundETH()` drains any ETH stranded on the router by a prior payable swap call — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` is an unrestricted `external payable` function that transfers the router's entire native ETH balance to `msg.sender`. Every swap entry-point is also `payable`, and the `pay()` helper deposits only exactly `amountIn` worth of native ETH into WETH, leaving any `msg.value` surplus on the router. Any subsequent caller — including an attacker — can drain that residual balance by calling `refundETH()`.

## Finding Description
`refundETH()` contains no caller restriction: [1](#0-0) 

The `pay()` helper, when `token == WETH` and `nativeBalance >= value`, deposits only the exact `value` (= `amountIn`) into WETH and leaves any surplus native ETH untouched on the router: [2](#0-1) 

All swap entry-points are `external payable` and accept arbitrary `msg.value`: [3](#0-2) [4](#0-3) 

The `receive()` guard only blocks plain ETH transfers (no calldata), not ETH arriving via a `payable` function call: [5](#0-4) 

**Attack path (two separate transactions):**

1. Victim calls `exactInputSingle{value: V}(...)` where `V > amountIn` and `tokenIn == WETH`. `pay()` deposits `amountIn` into WETH; `V - amountIn` remains as ETH on the router.
2. Attacker calls `refundETH()`. The function reads `address(this).balance` (which includes the victim's excess) and calls `_transferETH(msg.sender, balance)`, sending the full balance to the attacker.

The same path applies to `exactOutputSingle{value: amountInMaximum}(...)` — a common pattern — where the actual `amountIn` determined by the pool is less than `amountInMaximum`, leaving the difference stranded.

The intended safe usage pattern bundles `[exactInputSingle, refundETH]` inside a single `multicall`, as shown in the test suite: [6](#0-5) 

However, `exactInputSingle` and all other swap functions are callable standalone without `multicall`, and nothing in the contract enforces the bundled pattern.

## Impact Explanation
Direct theft of user ETH. Any excess native ETH sent with a WETH swap call is immediately claimable by any address via `refundETH()`. There is no minimum amount — the attacker receives the full stranded balance. This is a **High** severity direct loss of user principal.

## Likelihood Explanation
- All swap functions are `external payable` and callable directly without `multicall`.
- For `exactOutputSingle`, sending `msg.value = amountInMaximum` is the standard pattern to guarantee the swap succeeds; the actual `amountIn` is almost always less, leaving a residual.
- The attack requires no special privileges, no malicious pool, and no non-standard token behavior.
- MEV bots can back-run the victim's transaction atomically in the same block.

## Recommendation
Restrict `refundETH()` to only refund ETH sent in the same transaction by the caller. The standard fix is to track the original `msg.sender` in transient storage (EIP-1153) or enforce that `refundETH()` is only callable via `multicall` (where `msg.sender` is preserved via `delegatecall`). Alternatively, record the depositor address when ETH enters the router and enforce that only that address can call `refundETH()`.

## Proof of Concept
The existing test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` already demonstrates that excess ETH remains on the router after a standalone swap with `msgValue > amountIn`. A minimal exploit test:

1. Deploy router with real WETH and factory using existing test fixtures.
2. Fund victim with 2 ether.
3. Victim calls `router.exactInputSingle{value: 1 ether}(...)` with `amountIn = 1_000` and `tokenIn = WETH` directly (not via `multicall`).
4. Assert `address(router).balance > 0` — excess ETH is stranded.
5. Attacker calls `router.refundETH()`.
6. Assert `attacker.balance` increased by the full stranded amount and `address(router).balance == 0`.
