### Title
No User-Initiated Refund/Cancel Mechanism for ERC20 and ETH Locked in `initTransfer` — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.initTransfer` irrevocably transfers user ERC20 tokens (or ETH) into the contract with no corresponding cancel, refund, or emergency-withdrawal path. Release of those funds depends entirely on the MPC-signed `finTransfer` path. If that path is unavailable for any reason — NEAR bridge paused, MPC signer changed to a wrong address, token not registered on NEAR, or Wormhole guardian attestation failure — the deposited assets are permanently locked with no recovery mechanism for the user or an admin.

---

### Finding Description

In `OmniBridge.initTransfer`, when `tokenAddress` is a plain ERC20 (neither a bridge token nor a custom-minter token), the contract pulls the full `amount` from the caller and holds it:

```solidity
// OmniBridge.sol L407-L410
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
```

For native ETH (`tokenAddress == address(0)`), the ETH is implicitly held by the contract via `msg.value`.

The only release path for these assets is `finTransfer`, which requires a valid ECDSA signature from `nearBridgeDerivedAddress` (the MPC-derived key):

```solidity
// OmniBridge.sol L311
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
```

There is no `cancelTransfer`, `emergencyWithdraw`, or any admin token-rescue function anywhere in `OmniBridge.sol`. The contract does have a `receive()` fallback and a `pause` mechanism, but neither provides a recovery path for locked ERC20/ETH.

The `OmniBridgeWormhole` variant compounds this: `initTransferExtension` publishes a Wormhole message, but if Wormhole guardians fail to attest it, the NEAR side never sees the transfer and the MPC never signs a `finTransfer`.

On the NEAR side, `transfer_token_as_dao` exists as a DAO-only escape hatch for NEAR-side tokens, but there is no equivalent on the EVM side.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

Any user who calls `initTransfer` with a native ERC20 token or ETH has their funds locked in the `OmniBridge` contract until a valid MPC-signed `finTransfer` is submitted. If that never happens, the funds are unrecoverable. There is no timeout, no expiry, no user-initiated cancel, and no admin rescue function on the EVM contract.

---

### Likelihood Explanation

Multiple realistic, non-malicious scenarios trigger permanent lock:

1. **NEAR bridge paused**: The NEAR bridge has a `PauseManager` role that can pause `fin_transfer`. While paused, no `finTransfer` can be processed on NEAR, so the MPC never signs a release. EVM-side funds remain locked indefinitely.
2. **Token not registered on NEAR**: `initTransfer` performs no check that `ethToNearToken[tokenAddress]` is non-empty. A user can deposit a token that has no NEAR-side mapping; `fin_transfer_callback` will panic with `BridgeError::TokenDecimalsNotFound`, and the EVM tokens are stuck.
3. **`nearBridgeDerivedAddress` misconfigured**: An admin calling `setNearBridgeDerivedAddress` with a wrong address invalidates all future `finTransfer` calls, locking all currently deposited assets.
4. **Wormhole guardian failure** (Wormhole variant): If guardians fail to attest the `InitTransfer` message, the NEAR side never processes it and the MPC never signs a release.

---

### Recommendation

1. Add a user-initiated `cancelTransfer(uint64 nonce)` function that allows the original depositor to reclaim their ERC20/ETH after a configurable timeout (e.g., 7 days) if no corresponding `finTransfer` has been executed.
2. Add an admin-accessible `rescueTokens(address token, address recipient, uint256 amount)` function as a last-resort recovery path.
3. Validate that `ethToNearToken[tokenAddress]` is non-empty before accepting a deposit in `initTransfer`.

---

### Proof of Concept

1. User calls `OmniBridge.initTransfer(USDC, 1000e6, 0, 0, "near:alice.near", "")`.
2. `USDC.safeTransferFrom(user, OmniBridge, 1000e6)` executes — tokens are now in the contract.
3. The NEAR bridge `PauseManager` pauses `fin_transfer` (or the token has no NEAR mapping, or `nearBridgeDerivedAddress` is updated to a wrong value).
4. No valid `finTransfer` can ever be submitted on the EVM side.

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

        emit BridgeTypes.InitTransfer(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message
        );
    }
