Audit Report

## Title
ERC20 Swap and Liquidity Functions Accept Non-Zero `msg.value` Without Revert, Enabling Third-Party ETH Theft via `refundETH()` - (File: metric-periphery/contracts/MetricOmmSimpleRouter.sol, metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

## Summary
All public swap and liquidity-add entry points on `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` are declared `payable` but contain no guard that reverts when `msg.value > 0` and the input token is a plain ERC20 (not WETH). Any ETH sent alongside such a call is silently accepted and stranded on the contract. Because `refundETH()` is a public, caller-controlled function that transfers the contract's entire native balance to `msg.sender` with no ownership check, a third party can immediately steal the stranded ETH in a separate transaction.

## Finding Description
`PeripheryPayments.pay()` branches on `token == WETH` to consume native ETH from the contract balance. When `token != WETH`, it falls through to the plain `safeTransferFrom` branch and ignores `address(this).balance` entirely. [1](#0-0) 

All four public router swap functions are declared `payable` with no `msg.value == 0` guard for non-WETH tokens: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

The same applies to all four `addLiquidityExactShares` / `addLiquidityWeighted` overloads on `MetricOmmPoolLiquidityAdder`: [6](#0-5) [7](#0-6) 

The `receive()` hook blocks plain ETH transfers from non-WETH addresses, but it is **not** invoked when ETH is sent alongside a function call — `payable` functions accept ETH unconditionally at the EVM level regardless of `receive()`: [8](#0-7) 

The stranded ETH is then claimable by any address through the public `refundETH()` helper, which sends the contract's full native balance to `msg.sender` with no ownership check: [9](#0-8) 

## Impact Explanation
A user who accidentally sends ETH alongside an ERC20 swap or liquidity-add call loses those funds permanently to the first third party who calls `refundETH()`. Unlike a simple freeze, the public `refundETH()` surface converts the freeze into active theft: the attacker does not need to be the original caller, does not need any approval, and can front-run the victim's own recovery attempt. The loss is the full `msg.value` sent with the ERC20 call — a direct loss of user principal meeting Sherlock High/Medium thresholds depending on amount.

## Likelihood Explanation
The victim must first make the mistake of sending ETH with a non-WETH call. This is a realistic user error: wallets and integrators that wrap calls in `multicall{value: X}` for WETH paths may reuse the same call template with a different `tokenIn`, or a user may send ETH believing it covers gas. Once the ETH is stranded, exploitation requires only a single public call with no special privileges, making it trivially easy for any on-chain observer or MEV bot to front-run the victim's recovery.

## Recommendation
Add a `msg.value == 0` guard at the top of every swap and liquidity-add function whose `tokenIn` (or pool token0/token1) is not WETH. A minimal pattern:

```solidity
if (params.tokenIn != WETH && msg.value != 0) revert NativeNotAccepted();
```

Alternatively, add a shared modifier:

```solidity
modifier noNativeUnlessWETH(address token) {
    if (token != WETH && msg.value != 0) revert NativeNotAccepted();
    _;
}
```

Apply it to `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, and both `addLiquidity*` families when the relevant token is not WETH. For multi-hop paths, check `tokens[0]` (the initial input token).

## Proof of Concept
```solidity
// Setup: pool with tokenIn = USDC (not WETH)
// Victim:
router.exactInputSingle{value: 1 ether}(
    ExactInputSingleParams({
        pool: usdcPool,
        tokenIn: address(usdc),   // NOT WETH
        ...
    })
);
// Swap succeeds via safeTransferFrom; 1 ETH is now stranded on the router.
// address(router).balance == 1 ether

// Attacker (any address, separate tx):
router.refundETH();
// Attacker receives 1 ETH. Victim's ETH is gone.
```

The same sequence applies to `exactInput`, `exactOutputSingle`, `exactOutput`, and all `addLiquidity*` overloads on `MetricOmmPoolLiquidityAdder` when neither pool token is WETH.
