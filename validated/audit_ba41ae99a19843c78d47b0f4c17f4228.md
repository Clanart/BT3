Looking at the production code in `chain/chain/src/state_sync/adapter.rs`, I need to verify whether the shard_id/chunk mismatch is actually possible and whether it survives all validation steps.

**Key findings from the code:**

**1. The missing check in `set_state_header`**

The validation at lines 394-403 verifies only that the chunk hash is present *somewhere* in the block's `chunk_headers_root` Merkle tree: [1](#0-0) 

There is no check that `chunk.shard_id() == shard_id`. The Merkle proof for shard X's chunk at index 0 is structurally valid against the root regardless of what `shard_id` the caller supplies, because `verify_path` proves membership, not positional binding to a specific shard index.

**2. Storage uses caller-supplied `shard_id`** [2](#0-1) 

The header is stored under `StateHeaderKey(shard_id, sync_hash)` where `shard_id` is the caller-supplied value (Y), but the header's `chunk.prev_state_root()` belongs to shard X.

**3. The call path from a peer**

In `downloader.rs`, the `shard_id` in the `StateHeaderValidationRequest` is the local node's intended shard (Y), while the `header` is whatever the peer returned: [3](#0-2) 

A malicious state-sync peer can return shard X's header when asked for shard Y's header. The local node passes `shard_id=Y` (its own intent) and `header=header_for_shard_X` (from the peer) directly into `set_state_header`.

**4. `validate_chunk_proofs` does not catch this** [4](#0-3) 

`validate_chunk_proofs` checks internal chunk transaction/receipt Merkle proofs, not the shard_id binding. It passes for a legitimately-signed shard X chunk.

**5. Downstream `set_state_part` uses the corrupted state root** [5](#0-4) 

`set_state_part` reads the stored header for `(shard_id=Y, sync_hash)`, extracts `prev_state_root` from it (which is shard X's root), and validates parts against that root. A malicious peer supplying shard X's parts would pass this validation, completing the installation of shard X's state under shard Y.

### Title
Missing shard_id-to-chunk binding in `set_state_header` allows malicious state-sync peer to install wrong shard's state — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`ChainStateSyncAdapter::set_state_header` accepts a caller-supplied `shard_id` and a peer-supplied `ShardStateSyncResponseHeader`, but never verifies that `header.chunk.shard_id() == shard_id`. The Merkle proof check only proves the chunk is *somewhere* in the block, not that it is the chunk for the requested shard. A malicious state-sync peer can return shard X's header when asked for shard Y, causing the node to store `StateHeaderKey(Y, sync_hash)` with shard X's `prev_state_root`, and subsequently install shard X's state for shard Y.

### Finding Description
In `set_state_header` (lines 368–532 of `chain/chain/src/state_sync/adapter.rs`):

- **Step 1 (line 380):** `validate_chunk_proofs` validates internal chunk proofs only; it does not check `chunk.shard_id()`.
- **Step 2 (lines 394–403):** `verify_path` proves `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is in `sync_prev_block_header.chunk_headers_root()`. Since shard X's chunk is legitimately in the block, this passes even when `shard_id=Y` is supplied.
- **No step checks** `chunk.shard_id() == shard_id`.
- **Step 3 (lines 526–529):** The header is stored under `StateHeaderKey(shard_id=Y, sync_hash)` with shard X's `prev_state_root`.

The call path from `StateSyncDownloader::ensure_shard_header` (downloader.rs lines 67–88) shows that `shard_id` is the local node's intent and `header` is the peer's response — these are never cross-checked.

### Impact Explanation
A node performing state sync for shard Y installs shard X's trie state under shard Y's key. All subsequent chunk execution for shard Y operates on the wrong state root, producing incorrect state transitions, wrong account balances, and divergence from the canonical chain. The node will be unable to produce or validate correct chunks for shard Y.

### Likelihood Explanation
Any peer serving state sync data (no validator or special privilege required) can trigger this by returning a mismatched header. The attack is silent — no error is returned, no byzantine assertion fires, and the node proceeds normally with the corrupted state.

### Recommendation
Add an explicit shard_id binding check immediately after extracting the chunk in `set_state_header`:

```rust
// After: let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        format!("set_shard_state failed: chunk shard_id {:?} != requested shard_id {:?}",
                chunk.shard_id(), shard_id)
    ));
}
```

This should be placed before the `validate_chunk_proofs` call at line 380.

### Proof of Concept
In a two-shard `TestEnv`:
1. Obtain `header_for_shard_0` (a valid `ShardStateSyncResponseHeader` for shard 0, with its chunk_proof).
2. Call `chain_state_sync_adapter.set_state_header(shard_id=1, sync_hash, header_for_shard_0)`.
3. Assert it returns `Ok(())` (no error).
4. Read back the stored header via `get_state_header(shard_id=1, sync_hash)`.
5. Assert `stored_header.chunk().prev_state_root() == shard_0_state_root` (not shard 1's state root).
6. Confirm the node would proceed to download and apply shard 0's state parts under shard 1's key.
