### Title
Missing Validation of Declared Class Hash in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash provided by a contract is actually a declared class. The OS itself acknowledges this with an explicit TODO comment. Because the OS accepts any arbitrary felt value as a replacement class hash without checking it against `contract_class_changes` or the compiled class facts bundle, a contract can permanently set its own class hash to a non-existent value, making itself permanently uncallable and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall:

```cairo
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The `class_hash` field is taken directly from the syscall request and written into `contract_state_changes` with no check that it corresponds to any entry in `contract_class_changes` or the `compiled_class_facts_bundle`. The OS's own TODO comment at line 898 confirms this validation is absent. [1](#0-0) 

By contrast, the `execute_declare_transaction` function in `transaction_impls.cairo` enforces `prev_value=0` to prevent re-declaration and verifies the class hash via `finalize_class_hash`, demonstrating that the OS does have the infrastructure to validate class hashes — it simply does not apply it in `execute_replace_class`. [2](#0-1) 

The compiled class facts bundle is available in `OsGlobalContext` and is validated post-execution via `validate_compiled_class_facts_post_execution`, but `execute_replace_class` never consults it. [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a contract's class hash in `contract_state_changes` is set to a value that does not correspond to any compiled class fact, every subsequent call to that contract will fail at class resolution time. The contract's storage and any token balances it holds (e.g., ERC-20 balances stored in its storage, or ETH held via the fee token contract) become permanently inaccessible. Because the state change is committed to the state root and proven on-chain, there is no recovery path: the invalid class hash is canonical.

---

### Likelihood Explanation

Any Sierra contract that exposes a `replace_class` entrypoint with caller-controlled input — a common pattern for upgradeable contracts — is directly exploitable by an unprivileged transaction sender. The attacker supplies a felt value that is not a declared class hash (e.g., `1`). The OS accepts the syscall, commits the state change, and the contract is permanently bricked. No privileged role, leaked key, or operator cooperation is required. The attack is a single transaction.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, the OS must verify that the requested class hash exists in the current block's `contract_class_changes` dictionary or in the pre-existing state (i.e., it was declared in a prior block and is present in the compiled class facts bundle). This is exactly what the existing TODO acknowledges. The check should be analogous to the `prev_value=0` enforcement in `execute_declare_transaction` — a Cairo-level assertion, not a hint, so it is part of the proof constraints. [4](#0-3) 

---

### Proof of Concept

1. Declare a Sierra contract `VictimVault` that holds ERC-20 token balances and exposes:
   ```
   fn replace_my_class(new_hash: felt252) { replace_class_syscall(new_hash); }
   ```
2. Deploy `VictimVault` and deposit funds into it.
3. Submit an invoke transaction calling `replace_my_class(1)` (class hash `1` is not declared).
4. The OS executes `execute_replace_class`: `request.class_hash = 1` is written directly into `contract_state_changes` with no validation.
5. The block is proven and the state root is updated. `VictimVault`'s class hash is now `1` on-chain.
6. Any subsequent call to `VictimVault` fails at class resolution — the class `1` does not exist in the compiled class facts.
7. All funds in `VictimVault` are permanently frozen. [5](#0-4)

func execute_replace_class{
    range_check_ptr,

    alloc_locals;
    let request = cast(syscall_ptr + RequestHeader.SIZE, ReplaceClassRequest*);

        total_gas_cost=REPLACE_CLASS_GAS_COST, request_struct_size=ReplaceClassRequest.SIZE
    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.

    );
