### Title
`MerkleBlob::batch_insert` Silently Accepts Duplicate Keys/Hashes for Items 3+, Corrupting the DataLayer Merkle Tree Root and Invalidating Proofs of Inclusion - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces for all items beyond the first two. When a caller supplies a batch of three or more entries that contains a repeated `KeyId` or `Hash`, the function silently writes duplicate leaf nodes into the blob, overwrites the `BlockStatusCache` index maps without removing the stale entry, and produces a structurally corrupt tree whose Merkle root and proofs of inclusion are wrong.

### Finding Description

`MerkleBlob::insert` enforces two invariants before writing any leaf:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` calls `insert` (with its guards) only for the first two items when the tree has ≤ 1 existing leaf. Every subsequent item is written directly via `insert_entry_to_blob`, which skips both checks entirely:

```rust
for ((key, value), hash) in keys_values_hashes {          // items 3+
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?; // no duplicate guard
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which unconditionally overwrites the `key_to_index` and `leaf_hash_to_index` `HashMap` entries:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // silently overwrites
    self.leaf_hash_to_index.insert(leaf.hash, index); // silently overwrites
}
```

The old leaf node remains physically present in the blob at its original index, but the cache now points only to the new index. The tree therefore contains two leaf nodes for the same key, while the cache tracks only one. The Merkle root is computed over both nodes (the duplicate is included in the binary tree built by the pairing loop), producing a root that does not correspond to the logical key-value set. Any proof of inclusion generated from this state is based on the wrong tree.

`check_integrity` would detect the mismatch (`leaf_count != key_to_index_cache_length`), but it is not called automatically after `batch_insert` in production paths.

### Impact Explanation

- **Corrupted Merkle root**: The root hash is computed over a tree that contains a phantom duplicate leaf, so it does not represent the actual key-value set. Any downstream consumer that trusts this root (e.g., a DataLayer sync peer) accepts a false state.
- **Forged or invalid proofs of inclusion**: `get_proof_of_inclusion` walks the tree using the cache-recorded index for the key. After corruption the cache index points to the second (duplicate) leaf, while the first leaf is an orphan still embedded in the tree. Proofs generated against the corrupted root can be used to assert inclusion of data that was never legitimately inserted, or to deny inclusion of data that was.
- **Subsequent delete/upsert failures**: `delete` calls `block_status_cache.remove_leaf`, which removes the key from `key_to_index` and frees the cached index. The orphaned first leaf at the original index is never freed, leaving the blob in a permanently inconsistent state that causes `check_integrity` to return `IntegrityKeyToIndexCacheLength` or `IntegrityTotalNodeCount` errors.

This matches the allowed High impact: *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.*

### Likelihood Explanation

`batch_insert` is exposed directly to Python callers via `py_batch_insert` in the `chia_rs` wheel. Any DataLayer node or Python script that calls `MerkleBlob.batch_insert` with a list containing a repeated `KeyId` or `Hash` triggers the corruption. The Python binding performs no deduplication. Because DataLayer operators assemble batches from external data sources, accidental or deliberate duplicates are a realistic input.

### Recommendation

Add the same duplicate guards at the top of the bulk loop in `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing code
}
```

Alternatively, deduplicate the input `Vec` before processing, or use a local `HashSet` to detect intra-batch duplicates before any blob writes occur.

### Proof of Concept

```rust
#[test]
fn test_batch_insert_duplicate_key_corrupts_tree() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();
    blob.check_integrity_on_drop = false;

    // Seed the tree with 2 leaves so batch_insert skips the guarded path
    for i in 0i64..2 {
        blob.insert(KeyId(i), ValueId(i), &sha256_num(&i), InsertLocation::Auto {})
            .unwrap();
    }

    // Build a batch of 3 items where the last two share the same KeyId
    let dup_key = KeyId(99);
    let hash_a = sha256_num(&99i64);
    let hash_b = sha256_num(&100i64);
    let batch = vec![
        ((KeyId(50), ValueId(50)), sha256_num(&50i64)),
        ((dup_key, ValueId(1)), hash_a),  // first occurrence
        ((dup_key, ValueId(2)), hash_b),  // duplicate key – bypasses guard
    ];

    // batch_insert succeeds without error despite the duplicate
    blob.batch_insert(batch).unwrap();

    // The tree is now corrupt: check_integrity detects the mismatch
    blob.check_integrity()
        .expect_err("tree should be corrupt after duplicate batch insert");
}

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }
