### Title
Proof Verification Bypassed via ProofFacts Cache Reuse Across Unrelated Transactions ŌĆö (File: `crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

The `run_proof_verification` function in `TransactionConverter` skips cryptographic proof verification when `proof_facts` are already present in the `ProofManager`. Because `proof_facts` encode only a block reference (`block_number`, `block_hash`, `program_hash`, `config_hash`) and are not bound to any specific transaction's sender, nonce, or calldata, an attacker can submit a transaction carrying valid `proof_facts` copied from a previously-verified transaction alongside a completely invalid/garbage `proof`. The gateway and consensus admission paths accept the transaction without ever verifying the proof.

### Finding Description

In `run_proof_verification`:

```rust
let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;
if contains_proof {
    return Ok(false);   // proof parameter is silently discarded
}
``` [1](#0-0) 

When `contains_proof` returns `true`, the function returns immediately without calling `starknet_proof_verifier::verify_proof`. The `proof` argument passed by the caller is never inspected. The `ProofManager` stores entries keyed by `proof_facts.hash()`:

```rust
pub fn contains(&self, facts_hash: &Felt) -> bool { ... }
``` [2](#0-1) 

The `ProofFacts` structure contains only `proof_version`, `program_hash`, `block_number`, `block_hash`, and `config_hash`: [3](#0-2) 

None of these fields encode the submitting transaction's sender address, nonce, or calldata. The `proof` field is explicitly dropped when converting `RpcInvokeTransactionV3` to the internal representation and is not included in the transaction hash: [4](#0-3) 

The transaction hash computation in the Cairo OS hashes `proof_facts` but not `proof`: [5](#0-4) 

During blockifier pre-validation, `validate_proof_facts` checks only the block hash, program hash, and config hash ŌĆö it never re-verifies the cryptographic proof: [6](#0-5) 

The same pattern holds in the Cairo OS `check_proof_facts`: [7](#0-6) 

**Attack path:**

1. Attacker submits `Tx_A` with valid `proof_facts_X` and a valid `proof_valid`. Gateway admits it; `proof_valid` is stored in the `ProofManager` under key `proof_facts_X.hash()`.
2. Attacker submits `Tx_B` with the same `proof_facts_X` but a garbage `proof_garbage`, using a different sender/nonce/calldata so the transaction hash differs.
3. `run_proof_verification` calls `contains_proof(proof_facts_X)` ŌåÆ `true`.
4. `proof_garbage` is never passed to `verify_proof`; the function returns `Ok(false)`.
5. `Tx_B` is admitted to the mempool and executed. The blockifier validates `proof_facts_X` (block hash, program hash, config hash) ŌĆö all pass ŌĆö but never checks whether `proof_garbage` actually proves `Tx_B`'s execution.

This is the direct sequencer analog of the voucher-frontrun pattern: the `proof_facts` (like a voucher) are validated for their content but are not bound to the specific transaction that presents them, so any transaction can reuse them with an arbitrary (invalid) proof.

### Impact Explanation

**High ŌĆö Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

The client-side proving security model requires that every transaction carrying `proof_facts` is accompanied by a cryptographic proof that its execution on the referenced block is correct. The cache-skip path breaks this invariant: once any valid transaction has caused `proof_facts_X` to be stored, every subsequent transaction referencing `proof_facts_X` ŌĆö regardless of its actual calldata or proof validity ŌĆö is admitted without proof verification. The `proof_facts` committed into the transaction hash and block commitment no longer attest to the correctness of the specific transaction's execution.

### Likelihood Explanation

**Medium.** The attacker must first cause valid `proof_facts` to be stored in the `ProofManager` (by submitting or observing a valid client-side-proving transaction). `proof_facts` are public (visible in the mempool and on-chain). Once stored, the bypass can be triggered for an unlimited number of subsequent transactions. The `ProofManager` uses an LRU cache backed by persistent filesystem storage, so entries survive across restarts. [8](#0-7) 

### Recommendation

Bind `proof_facts` to the specific transaction by including the transaction hash (or a commitment over sender, nonce, and calldata) as a field inside `proof_facts` before hashing. This ensures that a proof generated for one transaction cannot satisfy the `contains_proof` check for a different transaction. Alternatively, when `contains_proof` returns `true`, verify that the submitted `proof` matches the stored proof before skipping re-verification.

### Proof of Concept

```
1. Attacker calls gateway with Tx_A:
     sender=Alice, nonce=5, calldata=[...], proof_facts=PF_X, proof=PROOF_VALID
   ŌåÆ contains_proof(PF_X) = false ŌåÆ verify_proof(PF_X, PROOF_VALID) ŌåÆ OK
   ŌåÆ ProofManager stores: PF_X.hash() ŌåÆ PROOF_VALID
   ŌåÆ Tx_A admitted.

2. Attacker calls gateway with Tx_B:
     sender=Bob, nonce=0, calldata=[evil], proof_facts=PF_X, proof=GARBAGE
   ŌåÆ contains_proof(PF_X) = true  ŌåÉ PF_X already stored from step 1
   ŌåÆ run_proof_verification returns Ok(false) without calling verify_proof
   ŌåÆ GARBAGE proof is never checked
   ŌåÆ Tx_B admitted to mempool with an unverified proof.

3. Blockifier executes Tx_B:
   ŌåÆ validate_proof_facts checks PF_X.block_hash, program_hash, config_hash ŌåÆ all pass
   ŌåÆ GARBAGE proof is never re-examined
   ŌåÆ Tx_B executes and its state changes are committed.
```

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let chain_info = &block_context.chain_info;
        // TODO(Meshi): Cache this computation as part of the chain context.
        let virtual_os_config_hash = OsChainInfo::from(chain_info)
            .compute_virtual_os_config_hash()
            .expect("Failed to compute OS config hash");
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }

func check_proof_facts{range_check_ptr, contract_state_changes: DictAccess*}(
    proof_facts_size: felt,
    proof_facts: felt*,
    current_block_number: felt,
    virtual_os_config_hash: felt,
) {
    if (proof_facts_size == 0) {
        return ();
    }

    assert_le(ProofHeader.SIZE + VirtualOsOutputHeader.SIZE, proof_facts_size);

    // Validate the proof header.
    let proof_header = cast(proof_facts, ProofHeader*);
    assert is_program_hash_allowed(proof_header.program_hash) = TRUE;
    // Proof version and variant are for future compatibility.
    assert [proof_header] = ProofHeader(
        proof_version=PROOF_VERSION,
        proof_variant=VIRTUAL_SNOS,
        program_hash=proof_header.program_hash,
    );

    // Validate the virtual OS output header.
    let os_output_header = cast(&proof_facts[ProofHeader.SIZE], VirtualOsOutputHeader*);

    with_attr error_message("Virtual OS output version is not supported") {
        assert os_output_header.output_version = VIRTUAL_OS_OUTPUT_VERSION;
    }

    // Validate that the proof facts block number is not too recent.
    // (This is a sanity check - the following non-zero check ensures that the block hash is
    // not trivial).
    assert_nn_le(
        os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
    );
    // Not all block hashes are stored in the contract; Make sure the requested one is not trivial.
    assert_not_zero(os_output_header.base_block_hash);

    // validate that the proof facts block hash is the true hash of the proof facts block number.
    read_block_hash_from_storage(
        block_number=os_output_header.base_block_number,
        expected_block_hash=os_output_header.base_block_hash,
    );

    // validate that the proof facts config hash is the true hash of the OS config.
    assert os_output_header.starknet_os_config_hash = virtual_os_config_hash;

    return ();
}
