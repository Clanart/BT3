### Title
Missing Recipient Validation in `FinalizeTransfer` — Attacker Can Redirect Bridged Tokens to Arbitrary Address - (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

### Summary

The `FinalizeTransfer` Anchor accounts struct accepts a `recipient: UncheckedAccount` with no constraint tying it to the recipient encoded in the signed `FinalizeTransferPayload`. Tokens are minted or transferred to the ATA derived from the caller-supplied `recipient`, not from the payload-specified recipient. Any submitter can redirect the finalized transfer to an arbitrary address.

### Finding Description

In `finalize_transfer.rs`, the accounts struct declares:

```rust
/// CHECK: this can be any type of account
pub recipient: UncheckedAccount<'info>,

#[account(
    init_if_needed,
    payer = common.payer,
    associated_token::mint = mint,
    associated_token::authority = recipient,   // ← derived from caller-supplied recipient
    token::token_program = token_program,
)]
pub token_account: Box<InterfaceAccount<'info, TokenAccount>>,
``` [1](#0-0) 

The `recipient` account carries only the `/// CHECK: this can be any type of account` comment and zero Anchor constraints. There is no `constraint = recipient.key() == data.payload.recipient` or equivalent guard anywhere in the struct.

The `process` function then mints or transfers `data.amount` tokens into `self.token_account` (the ATA of `self.recipient`) without ever comparing `self.recipient.key()` to the recipient field carried inside the Wormhole-signed `FinalizeTransferPayload`:

```rust
pub fn process(&mut self, data: FinalizeTransferPayload) -> Result<()> {
    UsedNonces::use_nonce(data.destination_nonce, ...)?;

    if let Some(vault) = &self.vault {
        transfer_checked(... to: self.token_account ..., data.amount ...)?;
    } else {
        require!(self.mint.mint_authority.contains(self.authority.key), ...);
        mint_to(... to: self.token_account ..., data.amount ...)?;
    }
    // data.recipient is never read or compared to self.recipient
    ...
}
``` [2](#0-1) 

The `FinalizeTransferResponse` posted back to NEAR also omits the recipient entirely, so NEAR cannot detect the misdirection:

```rust
let payload = FinalizeTransferResponse {
    token: self.mint.key(),
    amount: data.amount,
    fee_recipient: data.fee_recipient.unwrap_or_default(),
    transfer_id: data.transfer_id,   // recipient absent
}
``` [3](#0-2) 

This is the direct analog of the reported `investorExists` bug: the signed payload authenticates the intended recipient, but the actual account receiving value is a caller-controlled parameter that is never validated against the authenticated value.

### Impact Explanation

**Critical — direct theft of bridged assets.**

An attacker who submits `finalize_transfer` before the legitimate relayer (front-run on Solana) can supply their own wallet as `recipient`. The nonce is consumed, the tokens are minted/transferred to the attacker's ATA, and the legitimate user's subsequent attempt fails with `NonceAlreadyUsed`. The user's funds on the origin chain are already locked/burned; they receive nothing on Solana.

### Likelihood Explanation

Solana's public mempool and fast block times make front-running straightforward. Any observer of pending Wormhole VAAs (which are publicly broadcast) can extract the signed payload and race to submit `finalize_transfer` with a substituted `recipient`. No privileged access, leaked key, or colluding party is required — only the ability to submit a Solana transaction.

### Recommendation

Add an explicit Anchor constraint in the `FinalizeTransfer` accounts struct that ties the `recipient` account to the value encoded in the signed payload:

```rust
#[account(
    constraint = recipient.key() == data.payload.recipient
        @ ErrorCode::InvalidRecipient
)]
pub recipient: UncheckedAccount<'info>,
```

Alternatively, derive `recipient` as a PDA seeded from `data.payload.recipient` so Anchor enforces the binding at the account-resolution layer.

### Proof of Concept

1. Alice initiates a transfer on NEAR/EVM; the bridge locks/burns her tokens and emits a Wormhole message containing `recipient = Alice_Solana_pubkey`.
2. Wormhole guardians sign the VAA. The signed VAA is publicly visible.
3. Attacker Bob extracts the VAA, constructs a `finalize_transfer` instruction with the valid `SignedPayload<FinalizeTransferPayload>` but sets `recipient = Bob_Solana_pubkey`.
4. Bob submits before Alice's relayer. Anchor validates the Wormhole signature (payload is authentic), marks the `destination_nonce` as used, and mints/transfers `data.amount` tokens to Bob's ATA.
5. Alice's relayer submits the same VAA; the instruction reverts because the nonce is already consumed.
6. Alice's origin-chain funds are permanently locked/burned; Bob holds the bridged tokens. [4](#0-3)

impl FinalizeTransfer<'_> {
    pub fn process(&mut self, data: FinalizeTransferPayload) -> Result<()> {
        UsedNonces::use_nonce(
            data.destination_nonce,
            &self.used_nonces,
            &mut self.config,
            self.authority.to_account_info(),
            self.common.payer.to_account_info(),
            &Rent::get()?,
            self.system_program.to_account_info(),
        )?;

        if let Some(vault) = &self.vault {
            // Native version. We have a proof of token registration by vault existence
            transfer_checked(
                CpiContext::new_with_signer(
                    self.token_program.to_account_info(),
                    TransferChecked {
                        from: vault.to_account_info(),
                        to: self.token_account.to_account_info(),
                        authority: self.authority.to_account_info(),
                        mint: self.mint.to_account_info(),
                    },
                    &[&[AUTHORITY_SEED, &[self.config.bumps.authority]]],
                ),
                data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?,
                self.mint.decimals,
            )?;
        } else {
            // Bridged version. May be a fake token with our authority set but it will be ignored on the near side
            require!(
                self.mint.mint_authority.contains(self.authority.key),
                ErrorCode::InvalidBridgedToken
            );

            mint_to(
                CpiContext::new_with_signer(
                    self.token_program.to_account_info(),
                    MintTo {
                        mint: self.mint.to_account_info(),
                        to: self.token_account.to_account_info(),
                        authority: self.authority.to_account_info(),
                    },
                    &[&[AUTHORITY_SEED, &[self.config.bumps.authority]]],
                ),
                data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?,
            )?;
        }

        let payload = FinalizeTransferResponse {
            token: self.mint.key(),
            amount: data.amount,
            fee_recipient: data.fee_recipient.unwrap_or_default(),
            transfer_id: data.transfer_id,
        }
        .serialize_for_near(())?;

        self.common.post_message(payload)?;

        Ok(())
    }
