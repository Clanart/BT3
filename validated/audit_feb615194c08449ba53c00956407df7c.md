### Title
Unchecked Cross-Contract Burn Call Enables Unbacked Token Minting on Destination Chain - (File: near/omni-bridge/src/lib.rs)

---

### Summary

`burn_tokens_if_needed` fires a cross-contract `burn` call using `.detach()`, meaning the result is never checked. If the burn fails silently, `InitTransferEvent` is still emitted unconditionally, allowing relayers to mint tokens on the destination chain without the corresponding supply destruction on NEAR — creating unbacked bridged tokens.

---

### Finding Description

In `init_transfer_internal`, after a user's tokens have been received by the bridge via `ft_transfer_call`, the function calls `burn_tokens_if_needed` to destroy the bridged token supply on NEAR before emitting the cross-chain transfer event: [1](#0-0) 

```rust
fn burn_tokens_if_needed(&self, token: AccountId, amount: U128) {
    if self.is_deployed_token(&token) {
        ext_token::ext(token)
            .with_static_gas(BURN_TOKEN_GAS)
            .burn(amount)
            .detach();   // ← result never observed
    }
}
```

`.detach()` schedules the cross-contract call but discards the promise handle entirely. The NEAR runtime will execute the burn asynchronously, but the bridge contract never receives or checks the outcome.

Immediately after this detached call, `init_transfer_internal` unconditionally emits `InitTransferEvent`: [2](#0-1) 

```rust
if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
    self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
    self.lock_tokens_if_needed(...);
} ...
env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
U128(0)
```

The `burn` function on `OmniToken` calls `internal_withdraw` on the bridge contract's own balance in the token contract: [3](#0-2) 

```rust
fn burn(&mut self, amount: U128) {
    self.assert_controller();
    self.token.internal_withdraw(&env::predecessor_account_id(), amount.into());
}
```

If this call panics (e.g., due to gas exhaustion from `BURN_TOKEN_GAS` being insufficient, or any other runtime failure), the panic is silently swallowed because the promise was detached. The bridge has already emitted `InitTransferEvent` and returned `U128(0)` (success) to the caller.

---

### Impact Explanation

Relayers observe `InitTransferEvent` on NEAR and finalize the transfer on the destination chain (EVM, Solana, StarkNet), minting the bridged token amount to the recipient. If the burn on NEAR failed silently:

- The user's tokens are **not destroyed** on NEAR (they remain in the bridge contract's account in the token contract).
- Tokens are **minted on the destination chain** based on the emitted event.
- The result is **unbacked supply**: the same economic value exists on two chains simultaneously.

This matches the Critical impact class: *unauthorized mint of bridged assets*.

---

### Likelihood Explanation

The burn call is allocated a fixed `BURN_TOKEN_GAS` budget. If this budget is insufficient for the token contract's execution (e.g., after a token contract upgrade that adds logic, or due to miscalibration), the call will fail with a gas-exceeded error. Because the promise is detached, this failure is invisible to the bridge. The user does not need to do anything special — the failure is a protocol-level condition triggered by normal bridge usage with a deployed (bridged) token. Any user bridging a `is_deployed_token` asset back to its origin chain traverses this code path.

---

### Recommendation

Replace the detached fire-and-forget pattern with a chained callback that checks the burn result before emitting `InitTransferEvent`. The event must only be emitted after confirming the burn succeeded. This mirrors the fix described in the external report: the call result must be observed and the transaction must revert (or the state must be rolled back) on failure.

```rust
ext_token::ext(token)
    .with_static_gas(BURN_TOKEN_GAS)
    .burn(amount)
    .then(
        Self::ext(env::current_account_id())
            .with_static_gas(BURN_CALLBACK_GAS)
            .burn_callback(transfer_message, storage_owner),
    )
```

The `burn_callback` should check `env::promise_result_checked(0, ...)`, panic on failure, and only emit `InitTransferEvent` on success.

---

### Proof of Concept

1. User holds 1000 units of a bridged ERC-20 token deployed on NEAR (e.g., `usdc.omni-bridge.near`).
2. User calls `ft_transfer_call` on the token contract, transferring 1000 units to the bridge with a cross-chain transfer message targeting an EVM chain.
3. Bridge's `ft_on_transfer` → `init_transfer` → `init_transfer_internal` is invoked.
4. `burn_tokens_if_needed` fires `ext_token::ext(token).burn(1000).detach()`.
5. Due to gas exhaustion (or any other runtime failure in the token contract), the burn call fails. The failure is silently discarded.
6. `InitTransferEvent` is emitted with `amount = 1000`.
7. A relayer observes the event and calls `finTransfer` on the EVM `OmniBridge`, which mints 1000 tokens to the recipient.
8. The 1000 NEAR-side tokens remain in the bridge contract's balance (burn never executed). The 1000 EVM-side tokens are freshly minted. Total supply is now inflated by 1000 units — unbacked. [1](#0-0) [2](#0-1) [3](#0-2)
