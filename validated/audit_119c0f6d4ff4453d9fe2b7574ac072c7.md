Audit Report

## Title
Excess ETH Sent to `MetricOmmSimpleRouter` Is Not Refunded and Is Stealable by Any Caller via `refundETH()` — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`, `metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

## Summary
When a user calls a `payable` swap function on `MetricOmmSimpleRouter` with `tokenIn == WETH` and sends more native ETH than the pool consumes, the surplus ETH is silently retained by the router. The only recovery path, `refundETH()`, is permissionless and sends the entire ETH balance to `msg.sender` — not the original depositor — allowing any third party to steal the stranded ETH.

## Finding Description
In `PeripheryPayments.pay()`, when `token == WETH` and `nativeBalance >= value`, the function wraps and forwards exactly `value` wei to the pool:

```solidity
// PeripheryPayments.sol L75-77
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
}
```

Any ETH above `value` that was sent with the transaction is not consumed and not returned — it remains in the router.

This is directly reachable through `exactOutputSingle` (L130-147): the user sends `msg.value == params.amountInMaximum` to guarantee the swap succeeds. The pool charges `amountIn ≤ amountInMaximum`; the callback calls `pay(WETH, msg.sender, pool, amountIn)`, which wraps only `amountIn`. The delta `msg.value − amountIn` is left in the router. `exactOutputSingle` then returns without any refund:

```solidity
// MetricOmmSimpleRouter.sol L145-146
if (amountIn > params.amountInMaximum) revert InputTooHigh(amountIn, params.amountInMaximum);
_clearExpectedCallbackPool();
// no ETH refund here
```

The recovery function is permissionless:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);  // sends to caller, not original depositor
    }
}
```

`refundETH()` has no access control and sends the **entire** ETH balance to whoever calls it. A front-runner or MEV bot observing the swap transaction can call `refundETH()` immediately after and receive the victim's surplus ETH. The same exposure exists in `exactOutput` (multi-hop, L154-188) and `exactInputSingle`/`exactInput` when the user over-sends ETH relative to `params.amountIn`.

While `multicall` (L39-44) allows batching `exactOutputSingle` + `refundETH()` in a single transaction, this is not enforced — it is optional and not documented as required. Users and front-ends calling swap functions directly without multicall are fully exposed.

## Impact Explanation
Direct loss of user ETH principal. The stranded amount equals `msg.value − actual amountIn`, which for exact-output swaps can be a meaningful fraction of the swap size (up to `amountInMaximum − amountIn`). The theft requires no special privilege — any EOA or bot can call `refundETH()` in the same block with zero setup cost.

## Likelihood Explanation
High. Exact-output ETH swaps are a standard UX pattern. Users and front-ends routinely send `amountInMaximum` in ETH to cover worst-case slippage. MEV infrastructure already monitors routers for stranded ETH. The attack requires a single permissionless call with no setup cost and is repeatable on every such swap.

## Recommendation
1. **Auto-refund in swap functions**: After each `payable` swap completes, refund any remaining `address(this).balance` to `msg.sender` before returning. Add `if (address(this).balance > 0) _transferETH(msg.sender, address(this).balance);` at the end of `exactOutputSingle`, `exactOutput`, `exactInputSingle`, and `exactInput`.
2. **Restrict `refundETH` to the original depositor**: Record the depositor in transient storage at swap entry and enforce it inside `refundETH`, or accept a `recipient` parameter. At minimum, document that `refundETH` must always be batched in the same `multicall` as the swap.

## Proof of Concept
1. Alice calls `exactOutputSingle{value: 1 ETH}(...)` with `tokenIn = WETH`, `amountOut = 1000 USDC`, `amountInMaximum = 1 ETH`.
2. The pool charges `0.8 ETH` as `amountIn`; the callback calls `pay(WETH, Alice, pool, 0.8 ETH)`.
3. `pay` wraps `0.8 ETH` and transfers it to the pool. `0.2 ETH` remains in the router (`nativeBalance (1 ETH) >= value (0.8 ETH)` branch).
4. `exactOutputSingle` returns; no refund is issued.
5. Bob (MEV bot) calls `refundETH()` in the same block. `_transferETH(Bob, 0.2 ETH)` executes.
6. Alice loses `0.2 ETH`; Bob gains it with zero cost.
