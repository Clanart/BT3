### Title
Detached `send_tokens` in `process_fin_transfer_to_other_chain` Permanently Freezes Relayer Funds and Corrupts `locked_tokens` Accounting on Fast Transfer Finalization — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

In `process_fin_transfer_to_other_chain`, when a fast transfer has already occurred, the bridge irrevocably updates `locked_tokens` accounting and marks the fast transfer as finalised, then calls `send_tokens(...).detach()` to reimburse the relayer. Because the promise is detached with no callback, any failure of the token transfer permanently freezes the relayer's fronted funds inside the bridge with no recovery path. The `locked_tokens` map is also left in a corrupted state that diverges from actual on-chain balances, directly analogous to the `lockedCollateral.base` / sETH balance desync in the external report.

---

### Finding Description

In `near/omni-bridge/src/lib.rs`, `process_fin_transfer_to_other_chain` handles finalization of a cross-chain transfer whose destination is not NEAR. When a relayer has already executed a fast transfer (fronting `amount_without_fee` to the recipient on the destination chain), the function executes the following sequence synchronously and commits all state before the actual token transfer resolves:

```
// 1. locked_tokens accounting updated — state committed immediately
self.unlock_tokens_if_needed(origin_chain, &token, transfer_message.amount.0);
self.lock_tokens_if_needed(destination_chain, &token, transfer_message.fee.fee.into());

// 2. fast transfer status checked
let recipient = if let Some(status) = self.get_fast_transfer_status(&fast_transfer.id()) { ... Some(status.relayer) } else { ... None };

// 3. Fire-and-forget token transfer — NO callback, result never checked
self.send_tokens(token, relayer, U128(amount_without_fee), "").detach();

// 4. Fast transfer permanently marked finalised — state committed
self.mark_fast_transfer_as_finalised(&fast_transfer.id());
``` [1](#0-0) 

The `.detach()` call means NEAR will never deliver the promise result back to the contract. If `send_tokens` fails — for example, `ft_transfer` panics because the relayer has no storage registration for the token, or `mint` reverts on the token contract — the following permanent state corruption results:

- `locked_tokens[origin_chain][token]` has been decremented by `amount` (or left unchanged for origin-chain tokens), and `locked_tokens[destination_chain][token]` has been incremented by `fee`, even though no actual token movement to the relayer occurred.
- The fast transfer entry is permanently marked `finalised = true`, so no retry or recovery call is possible.
- The relayer's `amount_without_fee` tokens remain locked inside the bridge contract with no mechanism to release them.

The identical pattern exists in `utxo_fin_transfer_fast`, where `mark_fast_transfer_as_finalised` is called before `send_tokens(...).detach()`: [2](#0-1) 

The project's own `CLAUDE.md` security checklist explicitly flags this risk: *"Check .detach() usage: Detached promises should only be used for non-critical operations."* Reimbursing a relayer for fronted funds is a critical operation. [3](#0-2) 

By contrast, the analogous fast-transfer-to-NEAR path (`fast_fin_transfer_to_near_callback`) correctly chains a `resolve_fast_transfer` callback so that failures are detected and the fast transfer entry is removed on refund: [4](#0-3) 

The inconsistency between the two paths confirms the detach in `process_fin_transfer_to_other_chain` is unintentional.

---

### Impact Explanation

**Permanent fund freeze (Critical):** A relayer who executes a fast transfer for a cross-chain transfer whose destination is not NEAR fronts `amount_without_fee` tokens to the recipient. If the subsequent `send_tokens` call fails during finalization, those tokens are irrecoverably locked in the bridge. The fast transfer is marked finalised, so no retry is possible. The relayer has no recourse.

**Accounting corruption (High):** `locked_tokens` diverges from actual token balances held by the bridge. The origin chain's locked count is decremented and the destination chain's locked count is incremented for the fee, even though no actual transfer to the relayer occurred. This distorts all downstream protocol decisions that read `locked_tokens` — including future unlock eligibility checks and collateralization accounting — exactly as `lockedCollateral.base` diverging from sETH balance distorted liquidity values in the referenced report. [5](#0-4) 

---

### Likelihood Explanation

For NEAR-native tokens, `send_tokens` dispatches `ft_transfer(relayer, amount, None)`. This call panics if the relayer account has no storage registration for that specific token. A relayer may be registered for some tokens but not others, or their storage registration may have been removed. For deployed (bridged) tokens, `mint` is called; this can fail if the token contract is paused or its controller has changed. In both cases the failure is silent due to `.detach()`. The same risk applies to `utxo_fin_transfer_fast`. The attacker-controlled entry path is simply submitting a valid proof for a transfer that has a matching fast transfer entry — a standard relayer operation. [6](#0-5) 

---

### Recommendation

Replace `.detach()` with a proper callback in both `process_fin_transfer_to_other_chain` and `utxo_fin_transfer_fast`. The callback should:

1. On success: emit the finalization event and leave state as-is.
2. On failure: call `revert_lock_actions` to restore `locked_tokens`, call `remove_fast_transfer` (or un-mark finalised), and log a `FailedFinTransferEvent` so the relayer can be reimbursed through an alternative path.

The pattern already used in `fast_fin_transfer_to_near_callback` → `resolve_fast_transfer` is the correct model to follow. [7](#0-6) 

---

### Proof of Concept

**Setup:**
- Token: `usdc.near` (NEAR-native, non-deployed)
- Origin chain: Ethereum, Destination chain: Solana
- Relayer has no `storage_deposit` for `usdc.near` on their own account

**Steps:**

1. User on Ethereum calls `initTransfer` for 1000 USDC to a Solana recipient, fee = 10.
2. Relayer executes a fast transfer: sends 990 USDC to the Solana recipient from their own NEAR balance.
3. Relayer submits the Ethereum proof to NEAR via `fin_transfer`.
4. `fin_transfer_callback` → `process_fin_transfer_to_other_chain` executes:
   - `unlock_tokens_if_needed(Eth, usdc.near, 1000)` — `locked_tokens[Eth][usdc.near]` decremented by 1000 (committed).
   - `lock_tokens_if_needed(Sol, usdc.near, 10)` — `locked_tokens[Sol][usdc.near]` incremented by 10 (committed).
   - `send_tokens(usdc.near, relayer, 990, "").detach()` — `ft_transfer` dispatched fire-and-forget.
   - `mark_fast_transfer_as_finalised(...)` — fast transfer permanently finalised (committed).
5. `ft_transfer` panics: relayer has no storage for `usdc.near`. Promise fails silently.
6. **Result:** Relayer's 990 USDC is permanently locked in the bridge. `locked_tokens` is corrupted. Fast transfer cannot be retried. No recovery path exists. [8](#0-7)

    fn unlock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(available) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        require!(
            available >= amount,
            TokenLockError::InsufficientLockedTokens.as_ref()
        );

        let remaining = available - amount;
        self.locked_tokens.insert(&key, &remaining);

        LockAction::Unlocked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }
