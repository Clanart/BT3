### Title
Missing `recipient` Non-Empty Validation in `initTransfer` Enables Permanent Fund Lock - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

The `initTransfer` function in `OmniBridge.sol` accepts a `string calldata recipient` parameter with no validation that it is non-empty. Any unprivileged user can call `initTransfer` with an empty `recipient` string, causing their tokens to be burned or locked on the EVM chain with no recoverable destination. Because there is no cancel or refund path, the funds are permanently frozen in the bridge.

---

### Finding Description

`initTransfer` burns or locks the caller's tokens and emits an `InitTransfer` event that the MPC/relayer layer monitors to produce a cross-chain finalization. The only input guard on the `recipient` field is the fee check:

```solidity
// OmniBridge.sol lines 382–384
if (fee >= amount) {
    revert InvalidFee();
}
``` [1](#0-0) 

No check of the form `require(bytes(recipient).length > 0)` exists anywhere in `initTransfer`. The function proceeds to burn/lock tokens and emit the event regardless of whether `recipient` is a valid, non-empty destination address string. [2](#0-1) 

By contrast, the StarkNet counterpart explicitly guards against this class of error with `assert(amount > 0, 'ERR_ZERO_AMOUNT')` and `assert(fee < amount, 'ERR_INVALID_FEE')`, but neither StarkNet nor EVM validates the recipient string. [3](#0-2) 

The NEAR bridge's `log_metadata_callback` shows the project is aware of the pattern — it explicitly rejects empty `name` and `symbol` strings — but the same discipline was not applied to the `recipient` field in `initTransfer`. [4](#0-3) 

---

### Impact Explanation

**Severity: Critical — Permanent freezing / irrecoverable lock of user funds.**

Once `initTransfer` executes:

- For a bridge token: `BridgeToken(tokenAddress).burn(msg.sender, amount)` is called — the tokens are destroyed on-chain. [5](#0-4) 
- For a native/ERC-20 token: `safeTransferFrom` moves the tokens into the bridge contract. [6](#0-5) 

There is no cancel, refund, or dispute path in the contract. The MPC layer observes the `InitTransfer` event and attempts to produce a signed `finTransfer` payload for the destination chain. With an empty or syntactically invalid recipient string, the destination chain (NEAR, Solana, StarkNet) will reject the finalization because no valid account ID or address can be derived. The MPC will not produce a valid signature for a nonsensical recipient, so `finTransfer` is never called, and the funds remain locked forever with no on-chain recovery mechanism.

---

### Likelihood Explanation

**Likelihood: Medium.**

The entry point is fully public — any token holder can call `initTransfer`. The scenario is reachable by:

1. A user making a typo or copy-paste error that results in an empty string.
2. A front-end bug that submits the transaction before the recipient field is populated.
3. A malicious actor deliberately griefing another user's funds (e.g., via a contract that calls `initTransfer` on behalf of a victim after obtaining token approval).

No privileged role, leaked key, or colluding party is required.

---

### Recommendation

Add an explicit non-empty recipient guard at the top of `initTransfer` (and the analogous `initTransfer1155`):

```solidity
require(bytes(recipient).length > 0, "ERR_EMPTY_RECIPIENT");
```

This mirrors the pattern already used in `log_metadata_callback` on NEAR and the `ERR_ZERO_AMOUNT` guard on StarkNet, and is directly analogous to the `mintTo` validation recommended in the external report (validating that `to` is not an empty/zero address to prevent accidental loss of tokens).

---

### Proof of Concept

1. Attacker (or victim) holds 1000 USDC bridged tokens on EVM.
2. Calls:
   ```solidity
   omniBridge.initTransfer(
       usdcBridgeToken,  // tokenAddress
       1000e6,           // amount
       0,                // fee
       0,                // nativeFee
       "",               // recipient — empty string, no revert
       ""                // message
   );
   ```
3. `BridgeToken.burn(msg.sender, 1000e6)` executes — 1000 USDC is destroyed. [5](#0-4) 
4. `InitTransfer` event is emitted with `recipient = ""`. [7](#0-6) 
5. The MPC/relayer observes the event but cannot construct a valid `finTransfer` payload for an empty recipient on NEAR/Solana/StarkNet.
6. No `finTransfer` is ever submitted. No refund path exists. 1000 USDC is permanently lost.
