### Title
Inverted NEP-141 Callback Success Condition Causes Permanent Fee Loss and Transfer Re-insertion on Every Successful BTC Withdrawal — (`near/omni-bridge/src/btc.rs`)

---

### Summary

`submit_transfer_to_btc_connector_callback` interprets the `ft_transfer_call` callback result backwards relative to the NEP-141 standard. The condition `result.0 > 0` treats a **partial/full token refund** (connector rejection) as success and a **zero refund** (connector accepted all tokens) as failure. Because the production BTC connector accepts all tokens and returns `0`, every successful BTC withdrawal re-inserts the transfer into `pending_transfers` and never distributes the relayer fee.

---

### Finding Description

In NEP-141, `ft_transfer_call` resolves to the amount **refunded** back to the sender:

| `ft_on_transfer` return | Meaning | Callback value |
|---|---|---|
| `0` | Connector consumed all tokens — **success** | `Ok(U128(0))` |
| `N > 0` | Connector refunded N tokens — **rejection** | `Ok(U128(N))` |
| panic | Call failed entirely | `Err(PromiseError::Failed)` |

The callback at line 111 reads:

```rust
if matches!(call_result, Ok(result) if result.0 > 0) {
    // fee distribution  ← only reached on REJECTION
    self.send_fee_internal(&transfer_msg, fee_recipient, token_fee)
} else {
    // re-insert transfer ← reached on SUCCESS (result.0 == 0)
    self.add_transfer_message(transfer_msg, transfer_owner.clone());
    ...
}
``` [1](#0-0) 

The transfer is removed from `pending_transfers` **before** the cross-contract call: [2](#0-1) 

Then `ft_transfer_call` sends the nBTC to the connector: [3](#0-2) 

When the connector's `ft_on_transfer` returns `0` (standard NEP-141 success), the callback receives `Ok(U128(0))`, the guard `result.0 > 0` is **false**, and the `else` branch fires — re-inserting the transfer as if the connector had rejected it, while the connector already holds the tokens.

The existing test explicitly documents only the failure path (mock connector panics → re-insert), leaving the success path untested and broken: [4](#0-3) 

---

### Impact Explanation

**Every** successful BTC withdrawal produces these effects simultaneously:

1. **Fee permanently lost** — `send_fee_internal` is never called; the relayer who submitted the transfer receives nothing. With nonzero fees this is a direct financial loss on every withdrawal. [5](#0-4) 

2. **Transfer permanently stuck** — The re-inserted record in `pending_transfers` can never be successfully re-submitted because the bridge has already transferred the nBTC to the connector and no longer holds those tokens. The record leaks storage and is unclaimable.

3. **Potential nBTC drain via re-submission** — A trusted relayer who notices the stuck record can call `submit_transfer_to_utxo_chain_connector` again. If the bridge holds nBTC belonging to other users, the function will deduct those tokens and send a second batch to the connector, effectively double-spending the withdrawal amount against other users' balances.

---

### Likelihood Explanation

This is triggered on **every** successful BTC withdrawal — no special conditions are needed. Any trusted relayer (permissionless after staking) submitting a normal withdrawal will hit this path. The BTC connector is a standard NEP-141 receiver and will return `0` on acceptance. The bug is not gated behind any edge case.

---

### Recommendation

Invert the success condition to match NEP-141 semantics:

```rust
// BEFORE (inverted):
if matches!(call_result, Ok(result) if result.0 > 0) {

// AFTER (correct):
if matches!(call_result, Ok(result) if result.0 == 0) {
```

Alternatively, use the idiomatic pattern already used elsewhere in the codebase — check `is_refund_required` / `Err` path for failure and treat `Ok(U128(0))` as the success signal. [6](#0-5) 

---

### Proof of Concept

1. Deploy bridge + nBTC token + BTC connector (connector implements `ft_on_transfer` returning `U128(0)`).
2. User initiates BTC withdrawal; transfer is stored in `pending_transfers`.
3. Trusted relayer calls `submit_transfer_to_utxo_chain_connector`.
4. Bridge removes transfer, calls `ft_transfer_call` → connector's `ft_on_transfer` returns `0`.
5. Callback receives `Ok(U128(0))` → `result.0 > 0` is false → `else` branch fires.
6. **Assert**: `add_transfer_message` was called (transfer re-inserted).
7. **Assert**: `send_fee_internal` was NOT called (fee not distributed).
8. **Assert**: connector's nBTC balance equals the sent amount (tokens already transferred).
9. Attempt re-submission → bridge sends another batch of nBTC from other users' deposits.
