### Title
`meta_tx_v0` Signature Replay Attack Due to Missing Nonce in Hash — (`execution/syscall_impls.cairo`, `transaction_hash/transaction_hash.cairo`)

---

### Summary

The `meta_tx_v0` syscall computes a transaction hash that omits any nonce, and the OS never enforces nonce uniqueness for the inner meta-transaction. A previously-observed `meta_tx_v0` signature is therefore permanently valid and can be replayed by any unprivileged caller in a new outer transaction, causing the victim's `__execute__` entry point to run again with identical calldata — enabling direct, repeated loss of funds.

---

### Finding Description

**Root cause 1 — hash excludes nonce:**

`compute_meta_tx_v0_hash` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0`, meaning no nonce is mixed into the hash: [1](#0-0) 

The resulting hash is purely a function of `(INVOKE_HASH_PREFIX, version=0, contract_address, selector, hash(calldata), max_fee=0, chain_id)`. Two calls with identical calldata on the same chain always produce the same hash, regardless of when or how many times they are executed.

**Root cause 2 — OS never increments a nonce for the inner transaction:**

`execute_meta_tx_v0` constructs the inner `TxInfo` with a hard-coded `nonce=0` and never calls `check_and_increment_nonce`: [2](#0-1) 

This is consistent with the general version-0 nonce bypass in `check_and_increment_nonce`: [3](#0-2) 

**Root cause 3 — only `__execute__` is invoked, not `__validate__`:**

`execute_meta_tx_v0` enforces `selector == EXECUTE_ENTRY_POINT_SELECTOR` and then calls `contract_call_helper` directly, bypassing the OS-level `run_validate` step: [4](#0-3) 

The victim account's `__execute__` receives `tx_info.transaction_hash = meta_tx_hash` and `tx_info.signature = request.signature`. Because the hash is nonce-free, the signature is cryptographically valid for every future replay.

---

### Impact Explanation

**Critical — Direct loss of funds.**

Any financial operation executed via `meta_tx_v0` (token transfer, approval, swap, etc.) can be replayed an unlimited number of times by any party who observed the original outer transaction on-chain. The victim's account contract will accept the replayed signature because the hash it verifies against is identical to the original. Funds can be drained until the account balance is exhausted or the victim manually revokes the signing key.

---

### Likelihood Explanation

- `meta_tx_v0` is a public syscall callable from any contract; no privileged role is required.
- All outer transactions are published on-chain, so every `meta_tx_v0` signature is permanently visible to any observer.
- The attacker only needs to submit a new outer invoke transaction that calls any contract which forwards the observed calldata and signature to `meta_tx_v0`.
- No key compromise, Sybil attack, or network-level capability is required.

---

### Recommendation

Include a nonce in `compute_meta_tx_v0_hash` and enforce its uniqueness in `execute_meta_tx_v0`:

1. Add a `nonce` parameter to `compute_meta_tx_v0_hash` and pass it as `additional_data` (size 1) to `deprecated_get_transaction_hash`, mirroring how `compute_l1_handler_transaction_hash` handles nonces.
2. In `execute_meta_tx_v0`, read the current nonce from `contract_state_changes` for `contract_address`, assert it matches the nonce in the request, and increment it — analogous to `check_and_increment_nonce`.
3. Expose the nonce field in `MetaTxV0Request` so callers can specify the expected nonce.

---

### Proof of Concept

**Setup:**
- Alice's account contract is deployed at `alice_addr`.
- Alice signs a `meta_tx_v0` payload: `sign(hash(INVOKE_PREFIX, 0, alice_addr, __execute__, hash(transfer_calldata), 0, chain_id))` → `sig_alice`.
- Bob (outer transaction sender) submits TX₁ that calls a relay contract, which calls `meta_tx_v0(contract_address=alice_addr, selector=__execute__, calldata=transfer_calldata, signature=sig_alice)`.
- TX₁ is included; Alice's `__execute__` runs and transfers tokens.

**Replay:**
- Bob observes `sig_alice` from TX₁ on-chain.
- Bob submits TX₂ (new nonce for Bob's account) with the same relay call and the same `sig_alice`.
- The OS computes `meta_tx_hash` identically (no nonce in hash).
- Alice's `__execute__` validates `sig_alice` against `meta_tx_hash` — **valid**.
- The transfer executes again. Bob repeats until Alice's balance is zero.

The outer transaction nonce (Bob's) advances each time, so the OS never rejects TX₂, TX₃, … on nonce grounds. The inner meta-transaction has no independent replay protection.
