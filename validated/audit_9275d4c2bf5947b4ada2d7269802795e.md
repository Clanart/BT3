### Title
Unguarded `IERC20Metadata` Calls in `logMetadata` Permanently Freeze Non-Standard ERC-20 Tokens - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.logMetadata` calls `IERC20Metadata(tokenAddress).name()`, `.symbol()`, and `.decimals()` with no `try/catch` or low-level call protection. Tokens that return `bytes32` instead of `string` (e.g., MKR, SAI) or that do not implement these optional metadata functions at all will cause the call to revert unconditionally. Because `initTransfer` has no prerequisite check that a token has been registered via `logMetadata`, a user can lock such tokens in the bridge and then find them permanently unclaimable on the NEAR side.

---

### Finding Description

`logMetadata` is a public, permissionless `external payable` function:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name = IERC20Metadata(tokenAddress).name();
    string memory symbol = IERC20Metadata(tokenAddress).symbol();
    uint8 decimals = IERC20Metadata(tokenAddress).decimals();
    logMetadataExtension(tokenAddress, name, symbol, decimals);
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
``` [1](#0-0) 

The ERC-20 standard marks `name()`, `symbol()`, and `decimals()` as **optional**. Tokens such as MKR and SAI encode these fields as `bytes32`, not `string`. When Solidity's ABI decoder attempts to decode a `bytes32` return value as a `string`, it reverts. There is no `try/catch` block and no fallback path.

`logMetadata` is the only mechanism by which an EVM token's metadata is published to the NEAR side (via Wormhole in `OmniBridgeWormhole.logMetadataExtension`, or via MPC signature in the base flow). Without a successful `logMetadata` call, the NEAR bridge never learns the token exists and cannot finalize any transfer for it. [2](#0-1) 

`initTransfer`, however, imposes **no prerequisite** that the token has been registered. It accepts any ERC-20 address, locks the tokens in the bridge, and emits an `InitTransfer` event:

```solidity
function initTransfer(
    address tokenAddress,
    uint128 amount,
    ...
) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    ...
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    ...
}
``` [3](#0-2) 

The combination means:

1. User calls `initTransfer` for MKR → MKR is locked in the bridge contract.
2. User (or relayer) calls `logMetadata(MKR)` → always reverts because MKR's `name()` returns `bytes32`.
3. NEAR side never receives the metadata → token mapping is never created → `sign_transfer` on NEAR panics with `TokenDecimalsNotFound`.
4. The locked MKR has no recovery path in the EVM contract.

---

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds.**

Any ERC-20 token that returns `bytes32` metadata (MKR, SAI, and others) or that omits the optional metadata functions entirely can be locked in the bridge via `initTransfer` but can never be claimed on the NEAR side. There is no cancel/refund function in `OmniBridge` that would return the locked tokens to the sender. The funds are permanently frozen in the bridge contract.

---

### Likelihood Explanation

**High.** `logMetadata` and `initTransfer` are both permissionless public functions. MKR is one of the largest DeFi tokens by market cap and is a well-known example of the `bytes32` metadata pattern. Any user who attempts to bridge MKR (or any similarly non-standard token) will trigger this path. No privileged access is required.

---

### Recommendation

Wrap all three metadata calls in `try/catch` blocks and assign safe default values on failure, following the same pattern recommended in the original report:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name;
    string memory symbol;
    uint8 decimals;

    try IERC20Metadata(tokenAddress).name() returns (string memory n) {
        name = bytes(n).length > 0 ? n : _tryDecodeBytes32Name(tokenAddress);
    } catch {
        name = _tryDecodeBytes32Name(tokenAddress);
    }

    try IERC20Metadata(tokenAddress).symbol() returns (string memory s) {
        symbol = bytes(s).length > 0 ? s : _tryDecodeBytes32Symbol(tokenAddress);
    } catch {
        symbol = _tryDecodeBytes32Symbol(tokenAddress);
    }

    try IERC20Metadata(tokenAddress).decimals() returns (uint8 d) {
        decimals = d;
    } catch {
        decimals = 18; // safe default
    }

    require(bytes(name).length > 0 && bytes(symbol).length > 0, "ERR_EMPTY_METADATA");

    logMetadataExtension(tokenAddress, name, symbol, decimals);
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
```

Additionally, add a `bytes32`-to-string helper (as used by Polygon zkEVM and Arbitrum bridges) to handle tokens like MKR. Also consider adding a guard in `initTransfer` that requires the token to have been successfully registered before locking funds.

---

### Proof of Concept

1. Deploy or reference MKR on Ethereum mainnet (`0x9f8F72aA9304c8B593d555F12eF6589cC3A579A2`). Its `name()` function returns `bytes32("Maker")`, not `string`.

2. Call `OmniBridge.initTransfer(MKR_ADDRESS, 1e18, 0, 0, "near:recipient.near", "")` with 1 MKR approved. The call succeeds; 1 MKR is now held by the bridge contract.

3. Call `OmniBridge.logMetadata(MKR_ADDRESS)`. The EVM reverts at `IERC20Metadata(tokenAddress).name()` because the ABI decoder cannot decode `bytes32` as `string`. [4](#0-3) 

4. No `logMetadata` call can ever succeed for MKR. The NEAR bridge never registers the token. `sign_transfer` on NEAR panics with `TokenDecimalsNotFound`. [5](#0-4) 

5. The 1 MKR is permanently locked in the bridge with no recovery mechanism.
