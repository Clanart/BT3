### Title
Permanent Fund Freeze When EVM Token Transfer Reverts in `finTransfer` With No NEAR-Side Cancel Path — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`finTransfer` on the EVM bridge makes unchecked external calls to token contracts (ERC20 `safeTransfer`, ERC1155 `safeTransferFrom`, custom minter `mint`, bridge token `mint`). If any of these calls permanently revert — due to a token-level pause, recipient blacklisting, or custom minter failure — the transfer can never be finalized. Because the NEAR locker has no cancel or refund path for pending transfers, the user's funds are irrecoverably locked.

---

### Finding Description

`finTransfer` in `OmniBridge.sol` executes one of four external token delivery branches depending on the token type: [1](#0-0) 

Each branch is an unchecked external call. If any reverts, the entire transaction reverts. Because the nonce is marked consumed before the transfer: [2](#0-1) 

…the revert also rolls back the nonce mark, so the relayer can retry. However, if the underlying cause is permanent (token paused, recipient blacklisted, custom minter non-functional), every retry will revert identically.

On the NEAR side, the user's tokens are burned or locked inside `init_transfer_internal`: [3](#0-2) 

The transfer is stored in `pending_transfers` and is only removed by `claim_fee_callback`, which requires a proof of a successful `finTransfer` on the EVM side: [4](#0-3) 

There is no public `cancel_transfer`, `refund_transfer`, or equivalent function in the NEAR locker. The only admin escape valve is `transfer_token_as_dao`, which operates on the locker's own token balance, not on pending transfer records. The recipient is permanently fixed in the stored `TransferMessage`; `update_transfer_fee` only allows fee changes: [5](#0-4) 

Concrete permanent-revert scenarios:

1. **ERC20 global pause** (e.g., USDC/USDT emergency pause): `safeTransfer` reverts for all transfers of that token.
2. **ERC20 recipient blacklisting** (e.g., OFAC-sanctioned address): `safeTransfer` reverts for that specific transfer.
3. **Custom minter failure**: `ICustomMinter(customMinters[payload.tokenAddress]).mint(...)` reverts if the minter contract is paused, self-destructed, or has a permanent bug.
4. **ERC1155 recipient incompatibility**: `safeTransferFrom` calls `onERC1155Received` on the recipient; if the recipient contract does not implement `IERC1155Receiver`, the call reverts. [6](#0-5) 

---

### Impact Explanation

User funds are permanently frozen. The tokens are burned or locked on NEAR at `init_transfer_internal` time. Because `finTransfer` on EVM will always revert, `claim_fee` can never be called with a valid proof, so `pending_transfers` is never cleared. There is no programmatic recovery path for the user. This matches the **Critical** impact class: *permanent freezing, irrecoverable lock, or unclaimable settlement of user funds in bridge flows*.

---

### Likelihood Explanation

**Medium-High.** USDC and USDT both implement global pause and per-address blacklisting. Chainlink paused the UST/ETH oracle during the UST collapse — the same class of event applies here: a token-level emergency pause during a crisis is precisely when cross-chain transfers are most active. Custom minters are external contracts with their own upgrade/pause logic. ERC1155 recipient incompatibility is a common integration mistake. Any of these conditions, once triggered, permanently freezes all in-flight transfers for the affected token/recipient combination with no on-chain remedy.

---

### Recommendation

1. **Add a NEAR-side cancel/refund function** (DAO-gated or time-locked) that removes a stuck `pending_transfers` entry and returns the locked/burned tokens to the original sender.
2. **On the EVM side**, consider wrapping the token delivery in a try/catch pattern (using low-level calls with result checks) so that a failed delivery marks the nonce as "claimable by sender" rather than reverting the entire transaction, allowing the user to reclaim funds on EVM.
3. For the ERC1155 branch specifically, validate that the recipient implements `IERC1155Receiver` before calling `safeTransferFrom`, or use a pull-payment pattern.

---

### Proof of Concept

1. Alice holds 1000 USDC on NEAR and calls `ft_transfer_call` to bridge to EVM, specifying her EVM address as recipient.
2. NEAR locker burns Alice's USDC and stores the transfer in `pending_transfers`. [7](#0-6) 
3. Before the relayer calls `finTransfer`, Circle pauses USDC globally (or Alice's EVM address is OFAC-blacklisted).
4. Relayer calls `finTransfer` on EVM. The call reaches:
   ```solidity
   IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
   ```
   USDC reverts. The entire transaction reverts; `completedTransfers[nonce]` is rolled back. [8](#0-7) 
5. Every subsequent retry reverts identically.
6. Alice's transfer remains in `pending_transfers` on NEAR indefinitely. Her USDC is burned with no recovery path. [9](#0-8)

    pub fn update_transfer_fee(&mut self, transfer_id: TransferId, fee: UpdateFee) {
        match fee {
            UpdateFee::Fee(fee) => {
                let mut transfer = self.get_transfer_message_storage(transfer_id);

                require!(
                    transfer.message.origin_transfer_id.is_none(),
                    BridgeError::UpdateFeeNotAllowedForTransfer.as_ref()
                );

                let current_fee = transfer.message.fee;
                require!(
                    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
                    BridgeError::InvalidFee.as_ref()
                );

                require!(
                    fee.fee == current_fee.fee
                        || OmniAddress::Near(env::predecessor_account_id())
                            == transfer.message.sender,
                    BridgeError::SenderCanUpdateTokenFeeOnly.as_ref()
                );

                let diff_native_fee = fee
                    .native_fee
                    .0
                    .checked_sub(current_fee.native_fee.0)
                    .near_expect(BridgeError::LowerFee);

                require!(
                    NearToken::from_yoctonear(diff_native_fee) == env::attached_deposit(),
                    BridgeError::InvalidAttachedDeposit.as_ref()
                );

                transfer.message.fee = fee;
                self.insert_raw_transfer(transfer.message.clone(), transfer.owner);

                env::log_str(
                    &OmniBridgeEvent::UpdateFeeEvent {
                        transfer_message: transfer.message,
                    }
                    .to_log_string(),
                );
            }
            UpdateFee::Proof(_) => {
                env::panic_str(BridgeError::UnsupportedFeeUpdateProof.to_string().as_str())
            }
        }
    }
