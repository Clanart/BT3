### Title
`logMetadata()` Reverts for Non-Standard ERC20 Tokens, Permanently Blocking Token Registration and Locking User Funds - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.logMetadata()` calls `IERC20Metadata(tokenAddress).name()`, `.symbol()`, and `.decimals()` directly with no safe-decode wrapper. Tokens that return `bytes32` instead of `string` for `name()` or `symbol()` (e.g., MKR, SAI) cause an ABI-decode revert. Because `initTransfer()` accepts any ERC20 without requiring prior registration, a user can lock tokens in the bridge before `logMetadata` is ever successfully called. If `logMetadata` then reverts for that token, the NEAR side never deploys the corresponding wrapped token, and the locked EVM-side funds become permanently irrecoverable.

---

### Finding Description

`logMetadata` is the sole permissionless entry point for registering a native EVM token on the OmniBridge so it can be bridged to NEAR:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name    = IERC20Metadata(tokenAddress).name();    // line 225
    string memory symbol  = IERC20Metadata(tokenAddress).symbol();  // line 226
    uint8  decimals       = IERC20Metadata(tokenAddress).decimals();// line 227
    ...
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
``` [1](#0-0) 

The three calls use the high-level Solidity ABI decoder. When the callee returns a raw `bytes32` value (as MKR and SAI do for `name()`/`symbol()`), the decoder expects a dynamic-string ABI encoding (offset pointer + length + data). The raw `bytes32` is not a valid dynamic-string encoding, so `abi.decode` reverts unconditionally. There is no `try/catch`, no low-level `staticcall`, and no fallback path.

`initTransfer`, by contrast, accepts **any** ERC20 token address with no registration check:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
}
``` [2](#0-1) 

There is no cancel/refund path in the contract. `finTransfer` only moves tokens **from** NEAR **to** EVM; there is no reverse-unlock mechanism. [3](#0-2) 

On the NEAR side, `log_metadata_callback` requires a successful `LogMetadata` event before it will sign and propagate metadata to deploy a wrapped token: [4](#0-3) 

Without a deployed NEAR-side token, any `InitTransfer` event for that token address can never be finalized on NEAR.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

Attack path:

1. User calls `initTransfer(MKR, amount, ...)` on EVM. MKR is transferred into the bridge vault. An `InitTransfer` event is emitted.
2. User (or anyone) calls `logMetadata(MKR)`. The call reverts because MKR's `name()` returns `bytes32`. No `LogMetadata` event is ever emitted.
3. The NEAR relayer never receives a signed `LogMetadata` payload, so no wrapped MKR is ever deployed on NEAR.
4. The NEAR bridge cannot finalize the `InitTransfer` event — there is no token contract to mint into.
5. The EVM bridge has no refund or cancel mechanism. The MKR locked in step 1 is permanently frozen.

Even without step 1, the DoS on `logMetadata` permanently prevents any token with non-standard metadata from being bridged, which is a functional denial of the bridge for those assets.

---

### Likelihood Explanation

**Medium.** Several widely-held, high-value ERC20 tokens (MKR, SAI) return `bytes32` for `name()`/`symbol()`. The `initTransfer` function imposes no registration prerequisite, so a user unfamiliar with the registration requirement — or one who calls `initTransfer` speculatively — can trigger the fund-lock. The revert in `logMetadata` is deterministic and unrecoverable without a contract upgrade.

---

### Recommendation

Replace the direct `IERC20Metadata` calls in `logMetadata` with a safe low-level `staticcall` that handles both `string` (ABI-encoded) and `bytes32` (legacy) return formats, analogous to BoringCrypto's `returnDataToString` pattern:

```solidity
function _safeSymbol(address token) internal view returns (string memory) {
    (bool success, bytes memory data) =
        token.staticcall(abi.encodeWithSignature("symbol()"));
    if (!success || data.length == 0) return "";
    if (data.length == 32) return string(abi.encodePacked(bytes32(data)));
    return abi.decode(data, (string));
}
```

Apply the same pattern to `name()` and `decimals()`. This mirrors the fix recommended in the referenced Timeswap M-04 report and the BoringCrypto `BoringERC20.sol` library.

### Proof of Concept

MKR token (`0x9f8F72aA9304c8B593d555F12eF6589cC3A579A2`) on Ethereum mainnet returns `bytes32` for both `name()` and `symbol()`.

```solidity
// Demonstrates the revert
address MKR = 0x9f8F72aA9304c8B593d555F12eF6589cC3A579A2;
// This call reverts:
string memory sym = IERC20Metadata(MKR).symbol();
```

Fund-lock sequence:

```
1. user.initTransfer(MKR, 1e18, 0, 0, "near_recipient", "")
   → MKR locked in OmniBridge, InitTransfer event emitted

2. user.logMetadata(MKR)
   → REVERTS (bytes32 decode failure)
   → No LogMetadata event, no NEAR-side deployment

3. NEAR relayer cannot finalize the InitTransfer
   → 1e18 MKR permanently frozen in OmniBridge
```

    #[private]
    #[result_serializer(borsh)]
    pub fn log_metadata_callback(
        &self,
        #[callback] metadata: FungibleTokenMetadata,
        token_id: &AccountId,
    ) -> Promise {
        require!(
            !metadata.name.is_empty() && !metadata.symbol.is_empty(),
            BridgeError::InvalidMetadata.as_ref()
        );

        let metadata_payload = MetadataPayload {
            prefix: PayloadType::Metadata,
            token: token_id.to_string(),
            name: metadata.name,
            symbol: metadata.symbol,
            decimals: metadata.decimals,
        };

        let payload = near_sdk::env::keccak256_array(
            borsh::to_vec(&metadata_payload).near_expect(BridgeError::Borsh),
        );

        ext_signer::ext(self.mpc_signer.clone())
            .with_static_gas(MPC_SIGNING_GAS)
            .with_attached_deposit(env::attached_deposit())
            .sign(SignRequest {
                payload,
                path: SIGN_PATH.to_owned(),
                key_version: 0,
            })
            .then(
                Self::ext(env::current_account_id())
                    .with_static_gas(SIGN_LOG_METADATA_CALLBACK_GAS)
                    .sign_log_metadata_callback(metadata_payload),
            )
    }
