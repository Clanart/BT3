### Title
Cross-Chain Replay of `deployToken` Signature Due to Missing Chain ID in Message Hash ŌĆö (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `deployToken` / `deploy_token` message hash on every destination chain (EVM, Starknet, Solana) omits the destination chain ID. Because all chains share the same NEAR MPC-derived signer key (`nearBridgeDerivedAddress` / `derived_near_bridge_address` / `omni_bridge_derived_address`), a valid `deployToken` signature produced for one chain is cryptographically valid on every other chain. An unprivileged attacker who observes a legitimate `deployToken` transaction on chain A can replay the identical signature on chain B, deploying the same NEAR token there without NEAR authorization and permanently blocking any future legitimate deployment on that chain.

---

### Finding Description

**`finTransfer` includes chain ID; `deployToken` does not.**

In `OmniBridge.sol`, `finTransfer` encodes `omniBridgeChainId` twice into the signed message:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.TransferMessage)),
    Borsh.encodeUint64(payload.destinationNonce),
    bytes1(payload.originChain),
    Borsh.encodeUint64(payload.originNonce),
    bytes1(omniBridgeChainId),          // ŌåÉ chain ID present
    Borsh.encodeAddress(payload.tokenAddress),
    Borsh.encodeUint128(payload.amount),
    bytes1(omniBridgeChainId),          // ŌåÉ chain ID present again
    ...
);
``` [1](#0-0) 

But `deployToken` encodes only token metadata ŌĆö no chain ID:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)           // ŌåÉ no chain ID anywhere
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [2](#0-1) 

The same omission exists on Starknet. `MetadataPayload.to_borsh()` serializes only `[PayloadType::Metadata, token, name, symbol, decimals]` with no chain ID: [3](#0-2) 

While `TransferMessagePayload.to_borsh(chain_id)` explicitly takes and embeds `chain_id`: [4](#0-3) 

And on Solana, `DeployTokenPayload::serialize_for_near()` writes only `[IncomingMessageType::Metadata, token, name, symbol, decimals]` ŌĆö no `SOLANA_OMNI_BRIDGE_CHAIN_ID`: [5](#0-4) 

While `FinalizeTransferPayload::serialize_for_near()` writes `SOLANA_OMNI_BRIDGE_CHAIN_ID` for both the token and recipient fields: [6](#0-5) 

The `SignedPayload::verify_signature` on Solana recovers the signer from `keccak::hash(&serialized)` and checks it against `derived_near_bridge_address`: [7](#0-6) 

Since all chains verify against the same NEAR MPC-derived key, and the `deployToken` hash is chain-agnostic, the signature is valid on every chain simultaneously.

---

### Impact Explanation

**High ŌĆö Cross-chain replay enabling unauthorized token deployment.**

1. **Unauthorized token deployment**: An attacker replays a `deployToken` signature from chain A onto chain B, deploying the NEAR token on chain B without NEAR ever authorizing it for that chain. The token is registered in `isBridgeToken` / `near_to_starknet_token` / the Solana mint PDA as a legitimate bridge token.

2. **Permanent blocking of legitimate deployment**: Once the token exists on chain B (via replay), any future legitimate `deployToken` call from NEAR for that chain fails with `ERR_TOKEN_EXIST` / `ERR_TOKEN_ALREADY_DEPLOYED`. The token mapping is permanently poisoned. [8](#0-7) [9](#0-8) 

3. **Unbacked supply risk**: The replayed deployment registers the token as a valid bridge token. If NEAR's MPC later signs `finTransfer` messages for that token on chain B (which it may, since the token is legitimately registered on chain A), those signatures include the chain B chain ID and are valid ŌĆö enabling minting against a token that was never properly authorized for chain B.

---

### Likelihood Explanation

**High.** The attack requires only:
1. Observing a legitimate `deployToken` transaction on any chain (fully public on-chain data).
2. Submitting the same `(signatureData, metadata)` to any other chain's bridge contract.

No privileged access, no key material, no special tooling. Any bridge user or relayer can execute this immediately after a `deployToken` is broadcast on chain A.

---

### Recommendation

Include the destination chain ID in the `deployToken` / `deploy_token` message hash, consistent with how `finTransfer` already handles it. On EVM:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // add chain ID
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent fix to `MetadataPayload.to_borsh()` in `starknet/src/bridge_types.cairo` and `DeployTokenPayload::serialize_for_near()` in `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`. The NEAR side that produces the signature must also include the target chain ID when signing.

---

### Proof of Concept

1. NEAR MPC signs a `deployToken` payload for EVM Ethereum (`omniBridgeChainId = 1`):
   - Borsh bytes: `[0x01, len(token), token..., len(name), name..., len(symbol), symbol..., decimals]`
   - Signature `S` is broadcast in a successful `deployToken` tx on Ethereum.

2. Attacker extracts `(signatureData=S, metadata)` from the Ethereum transaction.

3. Attacker calls `deployToken(S, metadata)` on the Starknet bridge contract (or Solana `deploy_token`, or any other EVM chain bridge).

4. Each chain computes the identical hash (no chain ID in the preimage), recovers the same NEAR MPC address, and accepts the signature.

5. The token is now deployed on Starknet/Solana/other-EVM without NEAR authorization. Any future legitimate `deployToken` call for that token on those chains reverts with `ERR_TOKEN_ALREADY_DEPLOYED` / `ERR_TOKEN_EXIST`.

```rust
impl<P: Payload> SignedPayload<P> {
    pub fn verify_signature(
        &self,
        params: P::AdditionalParams,
        derived_near_bridge_address: &[u8; 64],
    ) -> Result<()> {
        let serialized = self.payload.serialize_for_near(params)?;
        let hash = keccak::hash(&serialized);

        let signature_bytes = &self.signature[0..64];

        let signature = libsecp256k1::Signature::parse_standard_slice(signature_bytes)
            .map_err(|_| ProgramError::InvalidArgument)?;
        require!(!signature.s.is_high(), ErrorCode::MalleableSignature);

        let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)
            .map_err(|_| error!(ErrorCode::SignatureVerificationFailed))?;

        require!(
            signer.0 == *derived_near_bridge_address,
            ErrorCode::SignatureVerificationFailed
        );

        Ok(())
    }
```
