### Title
`locked_tokens` Not Decremented When Reimbursing Relayer in Fast-Transfer Finalization to Non-NEAR Chain - (`File: near/omni-bridge/src/lib.rs`)

---

### Summary

In `process_fin_transfer_to_other_chain`, when a fast transfer has already been executed by a relayer, the bridge sends `amount_without_fee` tokens back to the relayer as reimbursement but never calls `unlock_tokens_if_needed` to reverse the corresponding lock that was recorded in `fast_fin_transfer_to_other_chain`. This permanently overstates `locked_tokens[(destination_chain, token)]` by `amount_without_fee` for every completed fast cross-chain transfer, corrupting the bridge's collateralization accounting.

---

### Finding Description

The `locked_tokens` map is the bridge's on-chain ledger of how many tokens are locked on NEAR backing the circulating supply on each foreign chain. Two functions must be read together:

**`fast_fin_transfer_to_other_chain` (lines 914–973)** — called when a trusted relayer pre-pays a cross-chain transfer (e.g., Eth→Sol). The relayer sends tokens to the bridge; the bridge burns `amount_without_fee` and records the lock:

```rust
self.burn_tokens_if_needed(fast_transfer.token_id.clone(), amount_without_fee.into());

self.lock_tokens_if_needed(
    fast_transfer.get_destination_chain(),   // e.g. Sol
    &fast_transfer.token_id,
    amount_without_fee,                      // locked_tokens[(Sol, token)] += amount_without_fee
);
``` [1](#0-0) 

**`process_fin_transfer_to_other_chain` (lines 1980–2054)** — called when the origin-chain proof arrives. In the fast-transfer branch it sends `amount_without_fee` back to the relayer but only adds `fee.fee` to `locked_tokens`; it never subtracts `amount_without_fee`:

```rust
// Only fee.fee is added — amount_without_fee is NOT subtracted
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token,
    transfer_message.fee.fee.into(),         // locked_tokens[(Sol, token)] += fee.fee
);

// Fast-transfer branch: reimburse relayer
if let Some(relayer) = recipient {
    self.send_tokens(token, relayer, U128(amount_without_fee), "").detach();
    self.mark_fast_transfer_as_finalised(&fast_transfer.id());
    // ← missing: unlock_tokens_if_needed(destination_chain, token, amount_without_fee)
}
``` [2](#0-1) 

The non-fast-transfer `else` branch correctly calls `lock_tokens_if_needed(destination_chain, token, amount_without_fee)` because those tokens remain locked pending a new outbound transfer message. In the fast-transfer branch the tokens are immediately disbursed to the relayer, so the lock recorded in step 1 must be reversed — but it is not. [3](#0-2) 

**Net accounting error per completed fast cross-chain transfer:**

| Event | `locked_tokens[(dest, token)]` change |
|---|---|
| `fast_fin_transfer_to_other_chain` | `+amount_without_fee` |
| `process_fin_transfer_to_other_chain` (fast branch) | `+fee.fee` (should be `−amount_without_fee + fee.fee`) |
| **Overstatement** | **`+amount_without_fee`** |

The `lock_tokens` / `unlock_tokens` primitives in `token_lock.rs` enforce `available >= amount` before any unlock:

```rust
require!(
    available >= amount,
    TokenLockError::InsufficientLockedTokens.as_ref()
);
``` [4](#0-3) 

Because `locked_tokens` is inflated, this guard passes for amounts the bridge no longer holds, allowing the bridge to attempt token transfers it cannot fulfill.

---

### Impact Explanation

`locked_tokens` is the sole on-chain invariant that prevents the bridge from releasing more tokens than it has locked. Overstating it by `amount_without_fee` per fast cross-chain transfer means:

1. **Collateralization is broken.** The bridge believes it holds more tokens than it actually does. The gap grows with every fast cross-chain transfer.
2. **Future legitimate `fin_transfer` calls can fail.** A subsequent user bridging back from the destination chain triggers `unlock_tokens_if_needed(destination_chain, token, X)`. The guard passes (inflated counter), `send_tokens` is called, but the bridge's actual NEP-141 balance is insufficient, causing the token transfer to fail. The `locked_tokens` counter is still decremented, deepening the desync.
3. **Permanent fund freezing.** Once the actual token balance falls below what `locked_tokens` implies, honest users' finalization attempts will either revert at the token level or leave the bridge in an unrecoverable accounting state, permanently locking their funds.

This matches the allowed impact: **High — Balance/accounting corruption that breaks bridge collateralization or misdirects value.**

---

### Likelihood Explanation

The fast-transfer-to-other-chain path is a core, intended feature of the protocol. Any trusted relayer executing a fast transfer for a cross-chain route (e.g., Eth→Sol, Eth→Strk) followed by any proof submitter calling `fin_transfer` triggers the bug. No special permissions beyond being a trusted relayer are required to initiate the fast transfer; the proof submission is fully public. The bug fires on every such transfer, so the accounting error accumulates monotonically.

---

### Recommendation

In the fast-transfer branch of `process_fin_transfer_to_other_chain`, call `unlock_tokens_if_needed` for `amount_without_fee` on the destination chain before (or after) sending tokens to the relayer, to reverse the lock recorded in `fast_fin_transfer_to_other_chain`:

```rust
if let Some(relayer) = recipient {
    // Reverse the lock recorded during fast_fin_transfer_to_other_chain
    self.unlock_tokens_if_needed(
        transfer_message.get_destination_chain(),
        &token,
        transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    );
    self.send_tokens(token, relayer, U128(amount_without_fee), "").detach();
    self.mark_fast_transfer_as_finalised(&fast_transfer.id());
}
``` [5](#0-4)

### Proof of Concept

**Setup:** Token `T` originates on Eth. `locked_tokens[(Sol, T)]` starts at `L`.

1. **Relayer fast-transfer (Eth→Sol, amount=1000, fee=10):**
   - `fast_fin_transfer_to_other_chain` burns 990 tokens, calls `lock_tokens_if_needed(Sol, T, 990)`.
   - `locked_tokens[(Sol, T)]` = `L + 990`.

2. **Proof submitter calls `fin_transfer` with Eth proof:**
   - `process_fin_transfer_to_other_chain` is called.
   - `unlock_tokens_if_needed(Eth, T, 1000)` → Eth is token origin chain → `Unchanged`.
   - `lock_tokens_if_needed(Sol, T, 10)` → `locked_tokens[(Sol, T)]` = `L + 990 + 10 = L + 1000`.
   - `send_tokens(T, relayer, 990)` → bridge's actual balance decreases by 990.
   - No `unlock_tokens_if_needed(Sol, T, 990)` is called.
   - **`locked_tokens[(Sol, T)]` = `L + 1000`, but bridge only holds `L + 10` worth of tokens.**

3. **Victim bridges back Sol→NEAR (amount=990):**
   - `process_fin_transfer_to_near` calls `unlock_tokens_if_needed(Sol, T, 990)`.
   - Guard: `L + 1000 >= 990` → passes.
   - `send_tokens(T, victim, 990)` → bridge tries to transfer 990 tokens it does not have → transfer fails or bridge is drained below safe levels.
   - `locked_tokens[(Sol, T)]` is now `L + 10`, but actual balance may be 0 or negative relative to what other users expect. [6](#0-5) [7](#0-6) [8](#0-7)

    fn unlock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(available) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        require!(
            available >= amount,
            TokenLockError::InsufficientLockedTokens.as_ref()
        );

        let remaining = available - amount;
        self.locked_tokens.insert(&key, &remaining);

        LockAction::Unlocked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }
