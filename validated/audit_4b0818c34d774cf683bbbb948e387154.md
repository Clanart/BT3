### Title
Hardcoded `L1_HANDLER_L2_GAS_MAX_AMOUNT` Gas Ceiling Causes Irrecoverable Loss of Bridged Funds on L1-Handler Out-of-Gas - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_l1_handler_transaction` unconditionally assigns the hardcoded constant `L1_HANDLER_L2_GAS_MAX_AMOUNT = 100_000_000` as the gas budget for every L1-handler execution. The L1→L2 message is consumed (written to OS outputs) **before** execution begins. If the handler exhausts this fixed budget, execution fails and all contract-state changes are reverted, but the message is permanently marked as consumed. Any funds that were locked on L1 as part of that message are irrecoverably lost.

---

### Finding Description

In `constants.cairo`, the L1-handler gas ceiling is a compile-time constant: [1](#0-0) 

In `execute_l1_handler_transaction`, the message is consumed first, then execution is attempted with that fixed budget: [2](#0-1) 

The ordering is critical: `consume_l1_to_l2_message` writes to `outputs.messages_to_l2` (OS-level output, not contract state), so it is **not** rolled back when the subsequent `non_reverting_select_execute_entry_point_func` call fails due to out-of-gas. The "non-reverting" wrapper confirms the OS does not abort the block on handler failure — it silently discards the execution result while keeping the message consumed.

By contrast, invoke transactions derive their gas budget dynamically from the user-supplied `resource_bounds[L2_GAS_INDEX].max_amount`: [3](#0-2) 

L1-handler callers have no equivalent mechanism — there is no field in the L1→L2 message that can raise the gas ceiling above `L1_HANDLER_L2_GAS_MAX_AMOUNT`.

---

### Impact Explanation

**Critical — Direct / permanent loss of funds.**

The canonical use-case for L1 handlers is token bridging: a user locks ETH or ERC-20 tokens in an L1 bridge contract, which emits an L1→L2 message. The L2 bridge's L1 handler is supposed to mint the corresponding L2 tokens. If that handler runs out of gas:

- The L1→L2 message is consumed and cannot be replayed.
- The L2 mint never happens.
- The L1 tokens remain locked in the bridge with no recovery path.

The user's funds are permanently frozen.

---

### Likelihood Explanation

**Medium.**

The 100 M gas ceiling equals `VALIDATE_MAX_SIERRA_GAS` — the cap for the *validation* phase of a regular transaction, which is intentionally tight. A bridge handler that performs multiple storage writes, emits events, or iterates over variable-length calldata can realistically approach or exceed this limit. The gas cost constants themselves carry an acknowledged undercount: [4](#0-3) 

If `STORAGE_READ_GAS_COST` or `STORAGE_WRITE_GAS_COST` are later corrected upward (as the TODO signals), handlers that previously fit within 100 M gas will silently start failing, triggering the loss-of-funds path for all in-flight L1→L2 messages at that moment.

An unprivileged L1 message sender can also deliberately craft large calldata payloads targeting bridge contracts whose handlers iterate over calldata, pushing gas consumption over the ceiling.

---

### Recommendation

Mirror the pattern used for invoke transactions: allow the L1→L2 message to carry a `l2_gas_limit` field, and use `min(user_supplied_limit, protocol_maximum)` as the actual gas budget. At minimum, do not consume the L1→L2 message before confirming that execution succeeded, or implement a cancellation / refund mechanism on L1 for messages whose handlers fail due to out-of-gas.

---

### Proof of Concept

1. Deploy an L2 bridge contract whose L1 handler iterates over `calldata` (e.g., minting tokens for each recipient in a batch).
2. From L1, call the bridge's `sendMessage` with a batch large enough that the handler's gas consumption exceeds `L1_HANDLER_L2_GAS_MAX_AMOUNT = 100_000_000`.
3. The sequencer includes the L1 handler transaction. `consume_l1_to_l2_message` writes the message to OS outputs (line 444).
4. `non_reverting_select_execute_entry_point_func` is called with `remaining_gas = 100_000_000` (line 445–448). The handler exhausts the budget and returns `ERROR_OUT_OF_GAS`.
5. All L2 state changes (token mints) are discarded. The message is permanently consumed.
6. The L1 tokens locked in the bridge are irrecoverable — no replay, no refund.
