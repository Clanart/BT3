### Title
Missing Validation That EVM Recipient Can Receive Native ETH Causes Permanent Lock of User Funds — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` unconditionally attempts a low-level ETH transfer to `payload.recipient` when bridging native ETH to EVM. If the recipient is a contract without a `payable` fallback or `receive` function, every call to `finTransfer` reverts. Because the NEAR bridge contract has no cancel/refund path for a pending transfer whose EVM leg can never succeed, the user's locked NEAR-side tokens become permanently irrecoverable.

---

### Finding Description

In `OmniBridge.finTransfer`, when `payload.tokenAddress == address(0)` (native ETH delivery), the contract performs a low-level call directly to the caller-supplied `payload.recipient`:

```solidity
// OmniBridge.sol lines 317-322
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [1](#0-0) 

No pre-flight check is performed to verify that `payload.recipient` can accept ETH before the NEAR-side transfer is committed. The `completedTransfers[payload.destinationNonce] = true` write at line 287 is reverted together with the failed call, so the nonce is never consumed and the relayer's ETH is returned — but the user's tokens locked on NEAR remain in `pending_transfers` with no recovery path. [2](#0-1) 

On the NEAR side, tokens are locked inside the bridge contract when the user calls `ft_on_transfer` → `init_transfer`. The transfer is stored in `pending_transfers`: [3](#0-2) 

The NEAR contract exposes no public cancel or refund function that would return locked tokens to the sender when the EVM leg is permanently undeliverable. The only removal of a pending transfer occurs inside `sign_transfer_callback` (when fee is zero) or `fin_transfer_callback`, neither of which is reachable if `finTransfer` on EVM always reverts. [4](#0-3) 

---

### Impact Explanation

A user who specifies an EVM contract address (e.g., a multisig, DAO treasury, or any contract without a `payable` fallback) as the recipient of a native-ETH bridge transfer will have their NEAR-side tokens permanently locked. Every relayer attempt to call `finTransfer` reverts with `FailedToSendEther`, the destination nonce is never consumed, and the NEAR bridge holds the tokens indefinitely with no user-accessible recovery mechanism.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user funds in bridge flows.**

---

### Likelihood Explanation

The scenario is realistic and reachable by any unprivileged bridge user:

- Smart-contract wallets, multisigs (e.g., Gnosis Safe without ETH receive), DAO treasuries, and protocol vaults are common EVM recipients.
- Many such contracts do not implement a `payable` fallback or `receive` function.
- The user supplies `recipient` freely when initiating the transfer on NEAR; no validation is performed on either chain before tokens are locked.
- The Proof of Concept from the external report demonstrates that a zero-value (and non-zero-value) call to a plain contract without `receive` returns `success = false`, confirming the revert path is reliably triggered.

---

### Recommendation

1. **Add a pre-flight ETH-receivability check in `finTransfer`** before marking the nonce used or accepting the relayer's ETH, analogous to the fix applied in the referenced Ignite commit:

```solidity
if (payload.tokenAddress == address(0)) {
    (bool canReceive, ) = payload.recipient.call{value: 0}("");
    if (!canReceive) revert RecipientCannotReceiveEther();
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
```

2. **Add a cancel/refund function on NEAR** that allows the original sender (or a DAO role) to reclaim locked tokens for transfers that have been pending beyond a timeout, so that even if the EVM leg is permanently broken, funds are not irrecoverable.

3. Consider adding `nonReentrant` protection around the ETH delivery path in `finTransfer`.
