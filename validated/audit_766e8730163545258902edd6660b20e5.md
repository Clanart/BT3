### Title
Unbounded Predecessor Replay Loop in `TraceTx` Enables Unthrottled Node Resource Exhaustion - (File: x/evm/keeper/grpc_query.go)

### Summary
The `TraceTx` gRPC handler iterates over `req.Predecessors` without any bound or timeout guard, executing each predecessor via `ApplyMessageWithConfig`. The per-trace timeout is installed only after the predecessor loop completes. An unprivileged caller can submit a single `debug_traceTransaction` / gRPC `TraceTx` request carrying an arbitrarily large `Predecessors` slice, forcing the node to replay an unbounded number of full EVM executions with no time limit, exhausting CPU and memory.

### Finding Description

In `x/evm/keeper/grpc_query.go`, the `TraceTx` handler passes a callback to `execTrace`. Inside that callback, the predecessor replay loop runs unconditionally over every element of `req.Predecessors`:

```go
for i, tx := range req.Predecessors {
    ...
    rsp, err := k.ApplyMessageWithConfig(ctx, msg, cfg, true)
    ...
}
``` [1](#0-0) 

Each iteration calls `ApplyMessageWithConfig` with `commit=true`, which runs the full EVM state transition (intrinsic gas, EVM execution, stateDB commit) against a cache context. There is no cap on `len(req.Predecessors)` and no timeout wrapping this loop.

The `defaultTraceTimeout` of 5 seconds is set up inside `prepareTrace`, which is called only **after** the predecessor callback returns:

```go
result, _, err := k.prepareTrace(ctx, cfg, msg, req.GetTraceConfig(), false)
``` [2](#0-1) 

The timeout goroutine is created inside `prepareTrace` at:

```go
deadlineCtx, cancel := context.WithTimeout(ctx.Context(), timeout)
defer cancel()
go func() {
    <-deadlineCtx.Done()
    if errors.Is(deadlineCtx.Err(), context.DeadlineExceeded) {
        tracer.Stop(errors.New("execution timeout"))
    }
}()
``` [3](#0-2) 

The predecessor loop therefore runs entirely outside any timeout protection.

The public JSON-RPC `debug_traceTransaction` path in `rpc/backend/tracing.go` collects predecessors from the real block and forwards them to `TraceTx`:

```go
traceTxRequest := evmtypes.QueryTraceTxRequest{
    Msg:          ethMessage,
    Predecessors: predecessors,
    ...
}
traceResult, err := b.queryClient.TraceTx(rpctypes.ContextWithHeight(contextHeight), &traceTxRequest)
``` [4](#0-3) 

The gRPC endpoint itself imposes no bound on `Predecessors`, so a caller hitting the gRPC port directly (or a crafted JSON-RPC relay) can supply an arbitrary number of predecessors that do not correspond to any real block.

### Impact Explanation

A single malformed `TraceTx` request with N predecessors forces N full EVM executions (each potentially consuming up to the block gas limit in CPU work) with no timeout. This exhausts CPU and memory on the target node. If the node is also a CometBFT validator, sustained resource exhaustion causes it to miss `Propose`/`Prevote`/`Precommit` rounds, leading to validator downtime and potential slashing. Repeated requests from a single attacker can keep the node permanently degraded. This is a reachable route to deterministic validator consensus failure via the public gRPC/JSON-RPC tracing path.

### Likelihood Explanation

The gRPC query port is commonly exposed alongside the JSON-RPC port on Ethermint nodes. No authentication is required. A single HTTP/2 request carrying a `QueryTraceTxRequest` with thousands of simple ETH-transfer predecessors (each ~100 bytes) fits within typical gRPC message size limits and triggers the vulnerability. The attack is trivially scriptable and requires no on-chain funds or special privileges.

### Recommendation

1. **Cap `req.Predecessors`**: Reject requests where `len(req.Predecessors)` exceeds a reasonable bound (e.g., the maximum number of transactions per block, or a configurable `maxTracePredecessors` parameter).
2. **Apply the timeout to the entire handler**: Wrap the predecessor loop inside the same `context.WithTimeout` used for the trace execution, or move the timeout setup before the predecessor loop begins.
3. **Rate-limit tracing endpoints**: Apply per-IP or per-connection rate limiting on `debug_trace*` and gRPC tracing methods at the server layer.

### Proof of Concept

```go
// Attacker constructs a TraceTx request with 10,000 simple ETH-transfer predecessors.
predecessors := make([]*evmtypes.MsgEthereumTx, 10_000)
for i := range predecessors {
    predecessors[i] = buildSimpleTransferMsg(i) // valid signed ETH transfer
}
req := &evmtypes.QueryTraceTxRequest{
    Msg:          targetMsg,
    Predecessors: predecessors,
    BlockNumber:  someHeight,
    ...
}
// Single gRPC call; node executes 10,000 ApplyMessageWithConfig calls
// with no timeout before the trace itself begins.
client.TraceTx(ctx, req)
```

The node will execute all 10,000 predecessor EVM state transitions (each potentially running arbitrary EVM bytecode up to the gas limit) before the 5-second trace timeout is even started, blocking the gRPC worker goroutine and saturating CPU for the duration.
