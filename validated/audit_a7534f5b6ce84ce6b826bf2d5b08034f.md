### Title
Missing Validation of Declared Class Hash in `execute_replace_class` Allows Permanent Contract Bricking — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary `class_hash` value — including undeclared or zero hashes — without verifying that the hash corresponds to a previously declared contract class. This is structurally identical to the external report's pattern: a critical reference (approval/class hash) is written at action time without validating the target exists, causing the dependent execution path to become permanently broken. Any contract that calls `replace_class` with an undeclared hash is permanently bricked, freezing all funds it holds.

---

### Finding Description

In `execute_replace_class`, the OS reads `request.class_hash` and immediately writes it into `contract_state_changes` with no check that the hash exists in `contract_class_changes`:

```cairo
let class_hash = request.class_hash;
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}
tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);
dict_update{dict_ptr=contract_state_changes}(...);
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges this missing validation. [2](#0-1) 

When any future transaction targets this contract, `execute_entry_point` runs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [3](#0-2) 

If `execution_context.class_hash` is an undeclared hash, `dict_read` returns `0` (the default for unset dict entries). `find_element` then searches for a compiled class with hash `0`. Because `execute_declare_transaction` enforces `assert_not_zero(compiled_class_hash)` before writing to `contract_class_changes`, no compiled class with hash `0` can ever legitimately exist: [4](#0-3) 

`find_element` fails (hint exception), making it impossible to generate a valid OS proof for any block containing a call to the bricked contract. The contract and all funds it holds are permanently inaccessible.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value via `replace_class`, every subsequent call to that contract causes `execute_entry_point` to fail at the `find_element` step. The sequencer cannot include such calls in any provable block. There is no recovery path: the class hash is committed to state, and no syscall exists to reset it to a valid value without executing the contract (which is now impossible). Any ERC-20 tokens, ETH, or other assets held in the contract's storage are permanently frozen.

---

### Likelihood Explanation

**Low.**

The attacker must control a contract (or exploit a contract that exposes `replace_class` to external callers) and supply an undeclared class hash. This is not a default pattern, but it requires no privileged role — any deployed contract can issue the `replace_class` syscall. Contracts that expose upgrade functions callable by arbitrary users, or contracts with bugs in their upgrade logic, are directly exploitable. The missing check is confirmed by the in-code TODO, indicating it is a known gap.

---

### Recommendation

In `execute_replace_class`, before writing the new class hash to state, verify that the hash exists in `contract_class_changes` (i.e., has been declared in the current or a prior block). Specifically:

1. Perform a `dict_read` on `contract_class_changes` with `key=class_hash`.
2. Assert the returned `compiled_class_hash` is non-zero (analogous to the `assert_not_zero` guard already present in `execute_declare_transaction`).
3. Remove the TODO comment once the check is implemented.

This mirrors the external report's recommendation to validate the module address is non-zero before storing the approval.

---

### Proof of Concept

1. Attacker deploys `VaultContract` holding 1000 STRK, with an `upgrade(class_hash: felt)` function that calls `replace_class(class_hash)`.
2. Attacker calls `upgrade(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
3. OS executes `execute_replace_class`: no validation occurs; `contract_state_changes` is updated with `class_hash=0xdeadbeef`.
4. Attacker (or any user) submits a `withdraw` call to `VaultContract`.
5. OS runs `execute_entry_point` for `VaultContract`:
   - `dict_read(contract_class_changes, key=0xdeadbeef)` → returns `0` (undeclared).
   - `find_element(..., key=0)` → hint failure; OS cannot produce a proof.
6. The sequencer cannot include any call to `VaultContract` in any block. The 1000 STRK are permanently frozen.
