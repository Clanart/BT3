### Title
Gateway Skips Account `__validate__` Signature Verification for Invoke Transactions When Any Deploy-Account Is Pending in Mempool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` in the Apollo gateway unconditionally skips the `__validate__` entry point (the Starknet account's on-chain signature verification) for any invoke transaction with nonce 1 when the sender's on-chain nonce is 0 and `account_tx_in_pool_or_recent_block` returns `true`. Because the mempool check is keyed only on the sender address — not on the identity of the submitter — any attacker who observes a victim's pending deploy-account transaction can submit an invoke with a completely fake signature from the victim's address and have it accepted by the gateway without any cryptographic verification.

### Finding Description

In `skip_stateful_validations` (lines 401–433 of `crates/apollo_gateway/src/stateful_transaction_validator.rs`), the gateway decides to skip the `__validate__` entry point when three conditions are simultaneously true:

1. The transaction is an `Invoke` with `nonce == 1`
2. The account's on-chain nonce is `0` (account not yet deployed)
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

This causes `AccountTransaction::validate_tx` in the blockifier to return `Ok(None)` immediately, never calling the account contract's `__validate__` entry point: [3](#0-2) 

The guard in condition 3 is `account_tx_in_pool_or_recent_block`, which returns `true` if the address appears in the mempool's tx-pool **or** in the mempool's committed-state set: [4](#0-3) 

This check is entirely address-based. It does not verify that the submitter of the current invoke transaction is the same party who submitted the deploy-account transaction. Any address that has a pending deploy-account in the mempool satisfies the condition.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks mempool-level constraints (duplicate hashes, nonce range); it does not verify the transaction signature: [5](#0-4) 

### Impact Explanation

An attacker can submit an invoke transaction with an arbitrary (invalid) signature from any address that has a pending deploy-account transaction in the mempool. The gateway accepts the transaction without calling `__validate__`, and the transaction enters the mempool. Via fee escalation, the attacker can replace the victim's legitimate nonce-1 invoke with the fake one. When the batcher later executes the fake invoke, `__validate__` is called with `validate = true` (the batcher's default), the signature check fails, and the transaction is rejected — but the victim's legitimate invoke has already been displaced and is lost from the mempool.

This matches: **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

The attack requires only that a victim has submitted a deploy-account transaction that is visible in the mempool — a normal, observable event for any new account creation. No privileged access, no special keys, and no race condition beyond submitting before the victim's invoke is committed. The attacker controls `sender_address`, `nonce` (set to 1), and the signature (set to anything).

### Recommendation

The skip-validate path should not rely solely on the presence of any transaction from the sender address in the mempool. At minimum, the gateway should verify the invoke transaction's signature against the expected account class derived from the pending deploy-account transaction's `class_hash` and `constructor_calldata`, even before the account is deployed. Alternatively, the skip should be removed and the UX concern addressed by allowing the deploy-account and invoke to be submitted atomically in a single RPC call that the gateway processes in order.

### Proof of Concept

1. Victim submits `deploy_account` for address `X` (class `C`, salt `S`). The transaction enters the mempool. `account_tx_in_pool_or_recent_block(X)` now returns `true`.
2. Attacker submits `invoke`: `sender_address = X`, `nonce = 1`, `signature = [0x1, 0x2]` (fake), `tip = high`.
3. Gateway evaluates `skip_stateful_validations`: nonce is 1, on-chain nonce of `X` is 0, `account_tx_in_pool_or_recent_block(X) = true` → returns `true`.
4. `run_validate_entry_point` is called with `skip_validate = true` → `execution_flags.validate = false` → `__validate__` is never called. The fake invoke is accepted and enters the mempool.
5. If the victim also submitted a legitimate invoke with nonce 1 and a lower tip, the attacker's fake invoke replaces it via fee escalation.
6. Batcher executes: `deploy_account` succeeds; fake invoke is executed with `validate = true` → `__validate__` fails → fake invoke is rejected.
7. Victim's legitimate invoke is gone from the mempool and must be resubmitted.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
