### Title
Missing Class Hash Existence Validation in `execute_replace_class` Allows Contracts to Adopt Undeclared Class Hashes, Permanently Freezing Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash provided by a calling contract is a previously declared class. This allows any contract to replace its class with an arbitrary, undeclared hash. The OS commits this invalid state, rendering the contract permanently non-executable and freezing any funds held within it.

---

### Finding Description

**Vulnerability class**: Insufficient input validation / state-transition bypass — the same class as the external report, where an asset type (Hypervisor tokens) existed in one registry (`hypervisors` array) but was absent from another (`TokenManager`), allowing a critical safety check to be bypassed.

**Analog mapping**:

| External Report | This Codebase |
|---|---|
| Hypervisor tokens tracked in `hypervisors[]` but not in `TokenManager` | Class hashes tracked in `contract_class_changes` (via `execute_declare_transaction`) but not checked in `execute_replace_class` |
| `removeAsset` bypasses collateral check because `token.addr == address(0)` | `execute_replace_class` bypasses class existence check entirely |
| Result: collateral stolen, protocol left with bad debt | Result: contract adopts undeclared class, funds permanently frozen |

In `execute_declare_transaction`, declared class hashes are registered into `contract_class_changes`:

```cairo
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [1](#0-0) 

However, in `execute_replace_class`, the new class hash is written directly into `contract_state_changes` **without any lookup into `contract_class_changes`**:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [2](#0-1) 

The `TODO` comment at line 898 explicitly acknowledges the missing check. The function signature confirms it has no access to `contract_class_changes`:

```cairo
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
``` [3](#0-2) 

`contract_class_changes` is not an implicit argument of `execute_replace_class`, so there is no structural path to perform the validation even if the code attempted it.

Compare this to `execute_declare_transaction`, which enforces `prev_value=0` to guarantee uniqueness of declared classes: [4](#0-3) 

The two registries — `contract_class_changes` (declared classes) and `contract_state_changes` (per-contract class hash) — are maintained independently, and `execute_replace_class` writes to the latter without consulting the former.

---

### Impact Explanation

**Impact: Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value and the block is committed, the OS state permanently records that contract as having a non-existent class. Any subsequent transaction that calls into this contract will fail at class resolution time, because the class hash cannot be found in the declared class registry. No function on the contract can ever be executed again.

If the contract holds user funds (e.g., a shared vault, a multi-sig wallet, a liquidity pool, or any contract holding ERC-20 balances on behalf of users), those funds are permanently inaccessible. There is no recovery path: the state is committed on-chain, and the class that would be needed to execute a withdrawal does not exist.

---

### Likelihood Explanation

**Likelihood: High.**

The `replace_class` syscall is available to every deployed contract without restriction. Any unprivileged user who deploys a contract can invoke it. The attacker does not need any special role, leaked key, or privileged access. The attack requires only:

1. Deploying a contract (or controlling an existing one that holds user funds).
2. Calling `replace_class` with any felt value that is not a declared class hash (e.g., `1`, `0xdeadbeef`, or any random felt).

The OS will accept the syscall, update the state, and commit the block. No validation prevents this path.

---

### Recommendation

Add `contract_class_changes` as an implicit argument to `execute_replace_class` and verify that the requested class hash has a non-zero entry in `contract_class_changes` before updating the contract state:

```cairo
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    contract_class_changes: DictAccess*,  // Add this
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    ...
    // After reducing gas, before updating state:
    let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
    with_attr error_message("Class hash is not declared.") {
        assert_not_zero(compiled_class_hash);
    }
    ...
}
```

This mirrors the pattern used in `execute_declare_transaction`, which enforces `prev_value=0` to guarantee that only legitimately declared classes are registered.

---

### Proof of Concept

1. Attacker deploys Contract A — a shared vault holding user token balances — with a valid declared class hash `C`.
2. Users deposit funds into Contract A.
3. Attacker (as contract owner or via a malicious entry point) calls the `replace_class` syscall with `class_hash = 0x1` (an undeclared felt).
4. `execute_replace_class` executes without error. `contract_state_changes` is updated: Contract A now has `class_hash = 0x1`.
5. The block is proven and committed. The on-chain state records Contract A with class hash `0x1`.
6. In all subsequent blocks, any transaction calling Contract A triggers class resolution for `0x1`. Since `0x1` was never declared, the class does not exist. Execution fails unconditionally.
7. All user funds deposited in Contract A are permanently frozen with no recovery path.
