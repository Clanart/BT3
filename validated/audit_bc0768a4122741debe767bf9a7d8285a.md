### Title
Missing Pause Guard in `deploy_token` Allows Token Deployment and Wormhole Message Emission During Emergency Pause — (`solana/programs/bridge_token_factory/src/lib.rs`)

### Summary

`deploy_token` in `lib.rs` has no pause check. `ALL_PAUSED` is defined only as `INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED` (bits 0 and 1), with no `DEPLOY_TOKEN_PAUSED` bit. Any relayer holding a previously-issued valid `SignedPayload<DeployTokenPayload>` can call `deploy_token` while `config.paused == ALL_PAUSED`, creating a mint PDA and posting a Wormhole message to NEAR.

### Finding Description

In `constants.rs`, the pause bitmask is defined as:

```
INIT_TRANSFER_PAUSED  = 1 << 0  (0x01)
FINALIZE_TRANSFER_PAUSED = 1 << 1  (0x02)
ALL_PAUSED            = 0x01 | 0x02 = 0x03
``` [1](#0-0) 

There is no `DEPLOY_TOKEN_PAUSED` bit. `finalize_transfer`, `finalize_transfer_sol`, `init_transfer`, and `init_transfer_sol` all have explicit `require!` guards: [2](#0-1) [3](#0-2) [4](#0-3) 

`deploy_token` has none: [5](#0-4) 

`initialize_token_metadata` creates a mint PDA, creates Metaplex metadata accounts, and calls `self.common.post_message(payload)` — emitting a Wormhole message back to NEAR: [6](#0-5) 

### Impact Explanation

When the bridge is fully paused (e.g., due to a compromised MPC key or critical bug), an unprivileged relayer can still:
1. Create a new wrapped mint PDA on Solana.
2. Emit a Wormhole `DeployToken` message to NEAR, causing NEAR to register a new token mapping.

This desynchronizes the NEAR token registry with Solana state during an emergency window, and if the pause was triggered by a compromised MPC key, the attacker can use any previously-captured valid `SignedPayload<DeployTokenPayload>` to register arbitrary token mappings while operators cannot stop it.

### Likelihood Explanation

The precondition — holding a previously-issued valid `SignedPayload<DeployTokenPayload>` — is realistic. Relayers routinely cache signed payloads for submission. The call path is fully permissionless once the payload is in hand; no privileged role is needed beyond the already-issued MPC signature.

### Recommendation

Add a `DEPLOY_TOKEN_PAUSED` bit to `constants.rs` and include it in `ALL_PAUSED`:

```rust
pub const DEPLOY_TOKEN_PAUSED: u8 = 1 << 2;
pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED | DEPLOY_TOKEN_PAUSED;
```

Then add the guard at the top of `deploy_token` in `lib.rs`:

```rust
require!(
    ctx.accounts.common.config.paused & DEPLOY_TOKEN_PAUSED == 0,
    error::ErrorCode::Paused
);
```

### Proof of Concept

1. `pausable_admin` calls `pause()` → `config.paused = 0x03` (`ALL_PAUSED`).
2. Attacker submits `deploy_token(signed_payload)` with any previously-issued valid `SignedPayload<DeployTokenPayload>`.
3. No pause check fires; `verify_signature` passes (payload was legitimately signed).
4. `initialize_token_metadata` executes: mint PDA is created, Metaplex metadata is written, Wormhole message is posted.
5. NEAR processes the Wormhole message and registers the token mapping — all while the bridge is supposed to be fully halted. [7](#0-6) [5](#0-4)
