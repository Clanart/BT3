### Title
Legacy String-Prefix Fallback in `get_token_origin_chain` Causes Permanent Finalization DoS for Unrecognized Deployed Tokens — (`near/omni-bridge/src/lib.rs`)

---

### Summary

The `get_token_origin_chain` function retains a live legacy fallback that resolves a token's origin chain by matching its NEAR account name against hardcoded string prefixes. After contract migration, `deployed_tokens_v2` is reset to an empty map while `deployed_tokens` retains all previously registered tokens. Any legacy deployed token whose account name does not match a known prefix causes a deterministic `env::panic_str` inside every transfer finalization call, permanently preventing users from claiming bridged assets on NEAR.

---

### Finding Description

`get_token_origin_chain` has three resolution paths:

1. **Fast path** — look up `deployed_tokens_v2` (the current authoritative map).
2. **UTXO path** — check `utxo_chain_connectors`.
3. **Legacy fallback** — if the token is present in the old `deployed_tokens` set but absent from `deployed_tokens_v2`, fall through to a `match` on the raw account-name string. [1](#0-0) 

The legacy `match` arm covers only a fixed set of prefixes (`"eth"`, `"base"`, `"arb"`, `"bnb"`, `"pol"`, `"hlevm"`, `"abs"`, `"sol"`, `"fogo"`, `"strk"`, `"starknet"`, `"factory.bridge.near"`, `"factory.sepolia.testnet"`). Any token whose name does not start with one of these hits the wildcard arm:

```rust
_ => env::panic_str(&BridgeError::CannotDetermineOriginChain.as_ref()),
```

The contract migration resets `deployed_tokens_v2` to a brand-new empty `LookupMap` while preserving the old `deployed_tokens` set intact: [2](#0-1) 

After migration every token that was registered before the upgrade is absent from `deployed_tokens_v2` and must traverse the legacy path. The self-healing write (`deployed_tokens_v2.insert(token, &origin_chain)`) is only reached after a successful prefix match; a token with an unrecognized name never heals — it panics on every invocation.

`get_token_origin_chain` is called unconditionally inside `lock_tokens_if_needed` and `unlock_tokens_if_needed` before any check on whether locked-token state even exists: [3](#0-2) 

These helpers are invoked inside `process_fin_transfer_to_near` and `process_fin_transfer_to_other_chain`: [4](#0-3) [5](#0-4) 

Because the panic reverts the entire transaction — including the `add_fin_transfer` call that would mark the transfer as finalized — the transfer ID is never recorded, and every subsequent retry panics identically.

    pub fn get_token_origin_chain(&mut self, token: &AccountId) -> ChainKind {
        if let Some(origin_chain) = self.deployed_tokens_v2.get(token) {
            return origin_chain;
        }

        if let Some(origin_chain) = self.get_utxo_chain_by_token(token) {
            return origin_chain;
        }

        if !self.deployed_tokens.contains(token) {
            return ChainKind::Near;
        }

        let origin_chain = match token.as_str() {
            s if s.starts_with("eth")
                || s.contains("factory.bridge.near")
                || s.contains("factory.sepolia.testnet") =>
            {
                ChainKind::Eth
            }
            s if s.starts_with("base") => ChainKind::Base,
            s if s.starts_with("arb") => ChainKind::Arb,
            s if s.starts_with("bnb") => ChainKind::Bnb,
            s if s.starts_with("pol") => ChainKind::Pol,
            s if s.starts_with("hlevm") => ChainKind::HyperEvm,
            s if s.starts_with("abs") => ChainKind::Abs,
            s if s.starts_with("sol") => ChainKind::Sol,
            s if s.starts_with("fogo") => ChainKind::Fogo,
            s if s.starts_with("strk") || s.starts_with("starknet") => ChainKind::Strk,
            _ => env::panic_str(&BridgeError::CannotDetermineOriginChain.as_ref()),
        };

        if !origin_chain.is_utxo_chain() {
            self.deployed_tokens_v2.insert(token, &origin_chain);
        }

        origin_chain
    }
