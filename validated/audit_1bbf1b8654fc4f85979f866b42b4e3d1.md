Looking at the external report's core invariant: **a snapshot value must be updated before a state transition is committed**. In M-06, `lastAccumulatedETHPerFreeFloatingShare` is snapshotted at deregistration time, but two code paths deregister a knot without first refreshing `accumulatedETHPerFreeFloatingShare`, so the snapshot captures a stale value.

The nearcore analog maps directly: when a shard is split, the child shards' `ChunkExtra` is snapshotted from the parent at the resharding boundary. The `proposed_split` field must be reset to `None` in that snapshot, but the resharding manager never does so. If the parent's `ChunkExtra` carries a non-`None` `proposed_split` at the boundary (propagated through missing-chunk carry-over), the child inherits a stale value that breaks the chunk-header commitment invariant for the child's first chunk.

### Title
Stale `proposed_split` Inherited by Child-Shard `ChunkExtra` at Resharding Boundary Causes Permanent `InvalidChunkHeaderShardSplit` — (File: `chain/chain/src/resharding/manager.rs`)

---

### Summary

`process_memtrie_resharding_storage_update` creates each child shard's initial `ChunkExtra` by cloning the parent's `ChunkExtra` and updating only `state_root` and `congestion_info`. The `proposed_split` field is never cleared. If the parent's `ChunkExtra` at the resharding block carries `proposed_split = Some(TrieSplit)` — which happens whenever one or more missing chunks near the epoch boundary propagate the field via `next_for_old_chunk` — the child shards inherit that stale value. The child's first chunk header will carry `proposed_split = None` (because `can_reshard` returns `false` due to the cooldown), but `validate_chunk_with_chunk_extra_and_receipts_root` compares it against the stored `ChunkExtra` and returns `InvalidChunkHeaderShardSplit`. The child shard cannot produce any valid chunk and is permanently stalled.

---

### Finding Description

**Step 1 — How `proposed_split` enters the parent's `ChunkExtra`.**

`compute_proposed_split` is called during every chunk application. It returns `Some(TrieSplit)` when `is_next_block_possibly_last_in_epoch` is true and the shard exceeds the memory threshold. Because `is_next_block_possibly_last_in_epoch` is a conservative over-approximation, it returns `true` for several blocks before the actual epoch boundary, not just the last one. [1](#0-0) 

**Step 2 — Missing chunks carry the value forward.**

When a chunk is missing, `apply_old_chunk` is called and the result is processed by `next_for_old_chunk`, which clones the entire previous `ChunkExtra` and updates only `state_root`. Every field — including `proposed_split` — is silently carried forward. [2](#0-1) 

So if the last *new* chunk of the parent shard (say at block H) produced `proposed_split = Some(TrieSplit)`, and blocks H+1 through the resharding block are all missing chunks, the parent's `ChunkExtra` at the resharding block still holds `proposed_split = Some(TrieSplit)`.

**Step 3 — Resharding clones the stale value into the child.**

`process_memtrie_resharding_storage_update` creates the child's `ChunkExtra` by cloning the parent's and updating only `state_root` and `congestion_info`. The `proposed_split` field is never cleared. The explicit TODO in the codebase confirms this is unresolved: [3](#0-2) 

The architecture document also records this as an open defect:

> **`chain/chain/src/resharding/manager.rs:249`** — The resharding manager doesn't set all `ChunkExtra` fields (notably the new `proposed_split` field).



**Step 4 — Validation rejects the child's first chunk.**

When the child shard produces its first chunk in epoch N+2, `validate_chunk_with_chunk_extra_and_receipts_root` compares `prev_chunk_extra.proposed_split()` (= `Some(TrieSplit)`, inherited from the parent) against `chunk_header.proposed_split()` (= `None`, because `can_reshard` returns `false` due to the cooldown). The mismatch triggers `InvalidChunkHeaderShardSplit`.
