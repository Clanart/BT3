### Title
`batch_insert` Bypasses Duplicate Key/Hash Validation, Silently Corrupting DataLayer Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. When a batch of ≥3 entries contains a key or leaf hash that already exists in the tree (or appears twice within the batch itself), the extra entries are silently written into the blob and the `block_status_cache` is silently overwritten. The result is a structurally corrupt Merkle tree whose root hash is wrong, and whose inclusion/exclusion proofs are invalid or forgeable.

### Finding Description

`MerkleBlob::insert` enforces two invariants before writing any data:

```rust
// blob.rs:369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::batch_insert` uses `insert` only for the first ≤2 entries (the bootstrap phase when `leaf_count <= 1`). All remaining entries in the batch are written through a fast path that calls `insert_entry_to_blob` directly, with **no duplicate checks**:

```rust
// blob.rs:587-603
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;  // no key/hash guard
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which performs a plain `HashMap::insert` on both `key_to_index` and `leaf_hash_to_index`:

```rust
// blob.rs:188-192
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
```

`HashMap::insert` silently overwrites the previous entry when the key already exists. Consequently:

1. The blob now contains **multiple leaf nodes with the same `KeyId` or `Hash`**, all reachable from the tree structure.
2. The cache (`key_to_index`, `leaf_hash_to_index`) only tracks the **last** duplicate, so earlier duplicates become orphaned — unreachable via normal lookup but still present in the blob and included in hash computation.
3. The Merkle root is computed over the corrupt tree, producing a **wrong root hash**.
4. Inclusion proofs generated from this tree are invalid; exclusion proofs may falsely claim a key is absent when a duplicate copy is still in the tree.

The Python binding `py_batch_insert` exposes this path directly to callers:

```rust
// blob.rs:1503-1518
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(
    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,
) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
```

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

- The Merkle root stored and propagated by the DataLayer node is wrong after a batch containing duplicates.
- `get_proof_of_inclusion` for an orphaned duplicate key will fail (`UnknownKey`) even though the key's leaf node is physically present in the blob and hashed into the root — a false exclusion.
- Conversely, a key that was supposed to be unique now has two leaf nodes contributing to the root hash; any proof built against this root is invalid from the perspective of a verifier that recomputes the root from a clean tree.
- Delta synchronization (`DeltaReader`, `collect_and_return_from_merkle_blob`) traverses the raw blob and will encounter the orphaned duplicate leaves, producing incorrect delta sets and corrupting the synced state on peer nodes.

### Likelihood Explanation

The `batch_insert` / `py_batch_insert` API is the primary bulk-load path used by the DataLayer Python layer. Any caller that passes a batch where a key appears more than once — whether due to a bug in the caller, a race condition, or deliberate malicious input — will silently corrupt the tree. There is no runtime guard, no assertion, and no integrity check performed inside `batch_insert` itself. The existing `test_batch_insert` test only uses unique keys, so the gap is untested.

### Recommendation

Add the same duplicate guards at the start of `batch_insert` (or inside the fast-path loop) that `insert` already enforces:

1. Before the fast-path loop, build a local `HashSet<KeyId>` and `HashSet<Hash>` from the incoming batch and check them against `block_status_cache` and against each other.
2. Alternatively, call `block_status_cache.contains_key(key)` and `block_status_cache.contains_leaf_hash(&hash)` inside the fast-path loop and return `Err(Error::KeyAlreadyPresent())` / `Err(Error::HashAlreadyPresent())` on collision, mirroring `insert`.
3. Add a regression test that passes a batch with a duplicate key to `batch_insert` and asserts it returns an error.

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

fn main() {
    let mut blob = MerkleBlob::new(vec![]).unwrap();

    // Pre-populate with 2 entries so leaf_count > 1 (bypasses the bootstrap path)
    for i in 0i64..2 {
        blob.insert(KeyId(i), ValueId(i), &sha256_num(i), InsertLocation::Auto {}).unwrap();
    }

    // Batch of 3 entries where the first key (KeyId(0)) already exists in the tree.
    // The fast-path loop (line 587) processes all 3 without checking for duplicates.
    let batch = vec![
        ((KeyId(0), ValueId(99)), sha256_num(99i64)), // duplicate key — should error, but doesn't
        ((KeyId(2), ValueId(2)), sha256_num(2i64)),
        ((KeyId(3), ValueId(3)), sha256_num(3i64)),
    ];

    // Succeeds silently — tree is now corrupt
    blob.batch_insert(batch).unwrap();
    blob.calculate_lazy_hashes().unwrap();

    // The root hash is wrong: it includes two leaf nodes for KeyId(0)
    // get_proof_of_inclusion(KeyId(0)) returns the proof for the *last* duplicate only;
    // the first duplicate is orphaned but still hashed into the root.
    // check_integrity() will detect the cache/blob mismatch and panic/error.
    blob.check_integrity().unwrap_err(); // demonstrates corruption
}

```rust
    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
