### Title
Re-entrant `ft_on_transfer` via Self-Directed `ft_transfer_call` Enables Unauthorized Outbound Transfer of Bridged Assets — (`near/omni-bridge/src/lib.rs`)

---

### Summary

When the NEAR bridge contract finalizes an inbound transfer whose recipient is the bridge contract itself, `send_tokens` issues an `ft_transfer_call` back to the bridge. The token contract then calls `bridge.ft_on_transfer` with the attacker-supplied `msg` field from the original source-chain transfer. Because `ft_on_transfer` accepts any registered token contract as caller and processes the `msg` as a new `BridgeOnTransferMsg`, the attacker can embed a valid `InitTransfer` payload in `msg` and cause the bridge to create a new outbound transfer to an attacker-controlled address on another chain — effectively stealing the tokens.

---

### Finding Description

**Root cause — no guard against the bridge contract being the `recipient`**

`process_fin_transfer_to_near` accepts any `AccountId` as `recipient`, including `env::current_account_id()` (the bridge itself). There is no check of the form `require!(recipient != env::current_account_id(), ...)`. [1](#0-0) 

**Root cause — `send_tokens` issues `ft_transfer_call` with user-controlled `msg`**

When `msg` is non-empty and the token is not wNEAR, `send_tokens` calls `token.ft_transfer_call(recipient, amount, None, msg)`. Both `recipient` and `msg` originate from the proof of the source-chain transfer and are fully attacker-controlled. [2](#0-1) 

For bridge-deployed tokens with a non-empty `msg`, `OmniToken::mint` similarly calls `self.ft_transfer_call(account_id, amount, None, msg)`, producing the same callback. [3](#0-2) 

**Root cause — `ft_on_transfer` processes the re-entrant call as a new transfer**

When `recipient = bridge_account_id`, the token contract calls `bridge.ft_on_transfer(sender_id=bridge, amount, msg)`. Inside `ft_on_transfer`, `token_id = env::predecessor_account_id()` is the legitimate token contract, so the token is recognized. The `msg` is parsed as `BridgeOnTransferMsg` and dispatched to `init_transfer`, which creates a new outbound `TransferMessage` and emits an `InitTransferEvent` that the MPC network will sign. [4](#0-3) 

`init_transfer` only rejects `ChainKind::Near` as the destination; any other chain (Ethereum, Solana, StarkNet) is accepted. [5](#0-4) 

The `msg` field flows unmodified from the source-chain proof into `send_tokens`: [6](#0-5) [7](#0-6) [8](#0-7) 

---

### Impact Explanation

**Critical — direct theft of bridged assets.**

The attacker causes the bridge to re-route tokens that were being delivered to the bridge contract into a new MPC-signed outbound transfer to an attacker-controlled address on Ethereum, Solana, or StarkNet. The tokens are burned (for deployed tokens) or locked (for native tokens) inside the re-entrant `init_transfer`, and the MPC network signs the new transfer payload. The attacker then finalizes the claim on the destination chain and receives the full token amount.

---

### Likelihood Explanation

**Medium.** Any bridge user on any supported source chain can trigger this by:
1. Calling `initTransfer` with `recipient = "bridge.near"` (or the deployed bridge account ID) and `message = <valid InitTransfer JSON targeting attacker's address>`.
2. Waiting for a relayer to submit the proof to NEAR `fin_transfer`.

No privileged access is required. The attacker only needs tokens on the source chain and knowledge of the bridge's NEAR account ID (public information). The relayer's storage balance is used for the re-entrant `init_transfer` call, which is a prerequisite that is already satisfied in normal operation.

---

### Recommendation

1. **Reject the bridge contract as a recipient.** Add a guard in `process_fin_transfer_to_near` (or earlier in `fin_transfer_callback`):
   ```rust
   require!(
       recipient != env::current_account_id(),
       BridgeError::InvalidRecipient.as_ref()
   );
   ```

2. **Alternatively, strip `msg` when the recipient is the bridge contract.** If self-delivery is intentionally supported, pass an empty `msg` to `send_tokens` when `recipient == env::current_account_id()` to prevent the re-entrant `ft_on_transfer` from processing a crafted payload.

3. **Consider a re-entrancy guard.** A boolean flag set during `fin_transfer_callback` execution and checked at the top of `ft_on_transfer` would prevent any re-entrant bridge operation regardless of recipient.

---

### Proof of Concept

**Setup:** Bridge is deployed at `bridge.near`. Token `tok.near` is a non-deployed (native) NEAR token registered with the bridge. Attacker controls `0xAttacker` on Ethereum.

**Step 1 — Source chain:** Attacker calls on Ethereum:
```solidity
OmniBridge.initTransfer(
    tok_eth,
    amount,
    fee = 0,
    nativeFee = 0,
    recipient = "bridge.near",
    message = '{"InitTransfer":{"recipient":{"Eth":"0xAttacker"},"fee":"0","native_token_fee":"0"}}'
);
```

**Step 2 — Proof submission:** Relayer submits proof to NEAR `fin_transfer`. `fin_transfer_callback` decodes the proof:
- `init_transfer.recipient = OmniAddress::Near("bridge.near")`
- `init_transfer.msg = '{"InitTransfer":...}'`

`process_fin_transfer_to_near` is called with `recipient = "bridge.near"` and `msg = '{"InitTransfer":...}'`. [9](#0-8) 

**Step 3 — `send_tokens` issues `ft_transfer_call`:**
```
tok.near.ft_transfer_call(
    receiver_id = "bridge.near",
    amount = amount,
    msg = '{"InitTransfer":{"recipient":{"Eth":"0xAttacker"},...}}'
)
``` [10](#0-9) 

**Step 4 — Re-entrant `ft_on_transfer`:** Token contract calls `bridge.near.ft_on_transfer(sender_id="bridge.near", amount, msg)`. The bridge parses `msg` as `BridgeOnTransferMsg::InitTransfer(...)` and calls `init_transfer`, which:
- Creates a new `TransferMessage` with `recipient = OmniAddress::Eth("0xAttacker")`
- Locks/burns the tokens
- Emits `InitTransferEvent` [11](#0-10) [12](#0-11) 

**Step 5 — MPC signs:** Relayer calls `sign_transfer` for the new transfer. MPC signs the payload for `0xAttacker` on Ethereum.

**Step 6 — Claim:** Attacker calls `finTransfer` on Ethereum with the MPC signature and receives `amount` tokens at `0xAttacker`.

**Result:** Tokens that were bridged to `bridge.near` are stolen by the attacker. The original inbound transfer is marked finalized, and a new outbound transfer is created and signed — all within a single relayer transaction.

    pub fn ft_on_transfer(&mut self, sender_id: AccountId, amount: U128, msg: String) {
        let token_id = env::predecessor_account_id();
        let parsed_msg: BridgeOnTransferMsg = serde_json::from_str(&msg)
            .or_else(|_| serde_json::from_str(&msg).map(BridgeOnTransferMsg::InitTransfer))
            .near_expect(BridgeError::ParseMsg);

        // We can't trust sender_id to pay for storage as it can be spoofed.
        let signer_id = env::signer_account_id();
        let promise_or_promise_index_or_value = match parsed_msg {
            BridgeOnTransferMsg::InitTransfer(init_transfer_msg) => {
                self.init_transfer(sender_id, signer_id, token_id, amount, init_transfer_msg)
            }
            BridgeOnTransferMsg::FastFinTransfer(fast_fin_transfer_msg) => {
                self.fast_fin_transfer(token_id, amount, signer_id, fast_fin_transfer_msg)
            }
            BridgeOnTransferMsg::UtxoFinTransfer(utxo_fin_transfer_msg) => self.utxo_fin_transfer(
                token_id,
                amount,
                &signer_id,
                &sender_id,
                utxo_fin_transfer_msg,
            ),
            BridgeOnTransferMsg::SwapMigratedToken => {
                self.swap_migrated_token(sender_id, token_id, amount)
                    .detach();
                PromiseOrPromiseIndexOrValue::Value(U128(0))
            }
        };

        promise_or_promise_index_or_value.as_return();
    }

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
