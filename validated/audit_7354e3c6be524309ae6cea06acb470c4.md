### Title
Missing `RPCTxFeeCap` check in `backend.SendTransaction` allows fee-cap bypass via `personal_sendTransaction` — (`rpc/backend/sign_tx.go`)

### Summary
`backend.SendTransaction` signs and broadcasts a transaction using the node's own keyring key without ever calling `rpctypes.CheckTxFee`. Both sibling functions `Resend` and `SendRawTransaction` perform this check. A caller who can reach the `personal_sendTransaction` RPC endpoint and supply a `from` address present in the node's keyring can force the node to sign and broadcast a transaction with an arbitrarily large `gasPrice` (e.g., `MaxUint256`), draining the node key's balance through excessive fees.

### Finding Description
In `rpc/backend/sign_tx.go`, `Backend.SendTransaction` (lines 38–125):

1. Verifies the `from` address is in the keyring.
2. Calls `SetTxDefaults` — which only fills in *missing* gas price fields; it does **not** cap a caller-supplied `gasPrice`.
3. Signs the transaction with the node's key.
4. Broadcasts it — **without ever calling `rpctypes.CheckTxFee`**.

By contrast, `Resend` (line 77) and `SendRawTransaction` (line 144) both call `rpctypes.CheckTxFee(price, gas, b.RPCTxFeeCap())` before proceeding. The omission in `SendTransaction` is an inconsistency that removes the only RPC-layer fee-cap guard for this code path.

`SetTxDefaults` only sets `GasPrice` when `args.GasPrice == nil`; if the caller supplies a non-nil `GasPrice` (e.g., `MaxUint256`), it is passed through unchanged. [7](#0-6) 

### Impact Explanation
The node's keyring key (often the validator/coinbase address) signs and broadcasts a transaction paying fees equal to `gasPrice * gas`. With `gasPrice = MaxUint256` and `gas = 21000`, the fee is astronomically large. The transaction will be rejected by the ante handler if the account lacks sufficient balance, but the node will still attempt to broadcast it, and if the account has any balance, a carefully chosen `gasPrice` just below the balance threshold will drain it entirely. This constitutes **valid user funds/fees being mis-accounted** — the node's own key balance is drained via excessive fees, matching the "High" impact category.

### Likelihood Explanation
- The `personal_` namespace is commonly enabled on Ethermint/Evmos nodes (it is listed in the default API namespaces).
- The node's coinbase/validator address is publicly discoverable via `eth_coinbase` or block headers.
- No authentication beyond RPC endpoint access is required; the attacker only needs to know one address in the node's keyring.
- The exploit is a single JSON-RPC call: `personal_sendTransaction({from: <node_addr>, to: <any>, gasPrice: "0xfff...fff", gas: "0x5208"}, "")`.

### Recommendation
Add the `CheckTxFee` call at the top of `Backend.SendTransaction`, immediately after `SetTxDefaults` resolves the final gas price, mirroring the pattern in `Resend` and `SendRawTransaction`:

```go
// After SetTxDefaults resolves args.GasPrice / args.MaxFeePerGas:
price := args.ToTransaction().AsTransaction().GasPrice()
gas   := uint64(*args.Gas)
if err := rpctypes.CheckTxFee(price, gas, b.RPCTxFeeCap()); err != nil {
    return common.Hash{}, err
}
```

```go
func CheckTxFee(gasPrice *big.Int, gas uint64, feeCap float64) error {
	// Short circuit if there is no cap for transaction fee at all.
	if feeCap == 0 {
		return nil
	}
	totalfee := new(big.Float).SetInt(new(big.Int).Mul(gasPrice, new(big.Int).SetUint64(gas)))
	// 1 photon in 10^18 aphoton
	oneToken := new(big.Float).SetInt(big.NewInt(params.Ether))
	// quo = rounded(x/y)
	feeEth := new(big.Float).Quo(totalfee, oneToken)
	// no need to check error from parsing
	feeFloat, _ := feeEth.Float64()
	if feeFloat > feeCap {
		return fmt.Errorf("tx fee (%.2f ether) exceeds the configured cap (%.2f ether)", feeFloat, feeCap)
	}
	return nil
}
```
