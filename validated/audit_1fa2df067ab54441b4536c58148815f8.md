### Title
Stranded Router ETH Is Freely Consumable by Any Caller via Unattributed `pay` and Public `refundETH` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

The `pay` function in `PeripheryPayments.sol` uses the router's **entire** native ETH balance to subsidize any WETH swap, with no attribution to who deposited the ETH. The public `refundETH()` function sends all router ETH to `msg.sender` with no access control. Together, any ETH stranded on the router from a prior user's transaction is freely claimable or consumable by any subsequent caller, causing a direct loss of the original depositor's principal.

---

### Finding Description

**Root cause — `pay` uses unattributed `address(this).balance`:** [1](#0-0) 

When `token == WETH`, the function reads `address(this).balance` and uses it unconditionally:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
```

There is no check that the ETH on the router was deposited by the current `payer`. Any ETH left over from a previous transaction is silently consumed to pay for the current caller's swap.

**Root cause — `refundETH` sends to `msg.sender` with no attribution:** [2](#0-1) 

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

`refundETH` is `external` with no access control. Any caller can invoke it and receive the router's entire ETH balance, regardless of who deposited it.

**How ETH becomes stranded:**

`exactInputSingle` and `exactOutputSingle` are both `payable` and accept `msg.value` directly (not only via `multicall`): [3](#0-2) [4](#0-3) 

The test suite confirms this usage pattern — `exactOutputSingle{value: nativePart}` is called directly without a `refundETH` step: [5](#0-4) 

For `exactOutputSingle`, the actual `amountIn` is only known after the swap executes. A user who sends `msg.value = X` but the pool only requires `Y < X` will have `X - Y` ETH stranded on the router after the call returns — unless they also included `refundETH()` in the same `multicall`. Calling `exactOutputSingle` directly (without multicall) is a supported and tested pattern, making this a realistic user path.

**`sweepToken` and `unwrapWETH9` are equally unguarded:** [6](#0-5) 

Both functions are `public payable` with no access control and transfer the router's entire balance of the specified token to a caller-chosen `recipient`. Any ERC-20 or WETH stranded on the router (e.g., from a multi-hop `exactInput` where the user set `recipient = address(router)` for an intermediate unwrap step that was then omitted) is equally exposed.

---

### Impact Explanation

**Theft via `refundETH`:** An attacker monitors the router's ETH balance. When User A's excess ETH is stranded (e.g., after a direct `exactOutputSingle{value: 1 ETH}` call where only 0.6 ETH was consumed), the attacker immediately calls `refundETH()` and receives the full 0.4 ETH. User A's principal is permanently lost.

**Free swap via `pay`:** An attacker calls `exactInputSingle{value: 0}` with `tokenIn = WETH` and `amountIn ≤ stranded ETH`. The `pay` function enters the `nativeBalance >= value` branch, wraps the stranded ETH, and sends it to the pool. The attacker receives swap output without spending any ETH or WETH. The original depositor's ETH is consumed with no recourse.

Both vectors are unprivileged, require no special setup beyond the victim's own transaction, and result in a direct, unrecoverable loss of user principal.

---

### Likelihood Explanation

The `exactOutputSingle` function is the highest-risk entry point: users cannot know the exact input amount before execution, so sending a conservative `msg.value` overage is the natural pattern. The test suite itself demonstrates calling `exactOutputSingle{value: nativePart}` without a paired `refundETH`, confirming this is an expected usage path. Any user who follows this pattern without wrapping the call in a `multicall([..., refundETH()])` will strand ETH. MEV bots routinely monitor contract balances and can atomically steal stranded ETH in the same block.

---

### Recommendation

1. **Track per-caller ETH deposits in transient storage.** When `msg.value > 0` is received in a payable swap function, record `(msg.sender → msg.value)` in a transient slot. In `refundETH`, only return the amount attributed to `msg.sender`, not the entire balance.

2. **Enforce `refundETH` sends only to the original depositor.** At minimum, restrict `refundETH` so it can only be called within a `multicall` context where `msg.sender` is the same as the swap initiator, or require an explicit `recipient` parameter that is validated against the transient payer slot.

3. **Apply the same guard to `sweepToken` and `unwrapWETH9`.** These functions should either require the caller to be the entity that deposited the tokens, or be restricted to `multicall`-only contexts with transient attribution.

---

### Proof of Concept

**Scenario — ETH theft via `refundETH`:**

```
// Block N
// User A calls exactOutputSingle directly (not via multicall)
// Pool requires 0.6 ETH; user sends 1 ETH
router.exactOutputSingle{value: 1 ether}(ExactOutputSingleParams({
    tokenIn: WETH,
    amountOut: someTokenAmount,
    amountInMaximum: 1 ether,
    ...
}));
// 0.4 ETH is now stranded on the router

// Block N (same or next)
// Attacker calls refundETH — no access control, no attribution check
router.refundETH();
// Attacker receives 0.4 ETH; User A's funds are permanently lost
```

**Scenario — free swap via `pay` consuming stranded ETH:**

```
// 0.4 ETH stranded on router (from User A above)

// Attacker calls exactInputSingle with value: 0
router.exactInputSingle{value: 0}(ExactInputSingleParams({
    tokenIn: WETH,
    amountIn: 0.4 ether,   // ≤ stranded balance
    ...
}));
// pay() sees nativeBalance = 0.4 ETH >= value = 0.4 ETH
// Wraps User A's 0.4 ETH, sends to pool
// Attacker receives swap output without spending any ETH or WETH
```

The `pay` function's unconditional use of `address(this).balance` is the root cause in both cases. [7](#0-6)

  function exactOutputSingle(ExactOutputSingleParams calldata params) external payable returns (uint256 amountIn) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    int128 expectedAmountOut = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountOut);
    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(params.recipient, params.zeroForOne, -expectedAmountOut, priceLimitX64, "", params.extensionData);
    int128 amountOut = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    if (amountOut != expectedAmountOut) revert InvalidOutputAmount(amountOut, params.amountOut);

    amountIn = MetricOmmSwapInputs.int128ToUint128(
      MetricOmmSwapResults.extractAmountIn(params.zeroForOne, amount0Delta, amount1Delta)
    );

    if (amountIn > params.amountInMaximum) revert InputTooHigh(amountIn, params.amountInMaximum);
    _clearExpectedCallbackPool();
  }
