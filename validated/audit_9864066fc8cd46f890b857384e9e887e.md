### Title
Wrapping Arithmetic in `check_time_locks` Enables Relative Timelock Bypass via Integer Overflow — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` accepts a caller-controlled `nowrap: bool` parameter that switches relative-timelock addition between `saturating_add` (`nowrap=true`) and `wrapping_add` (`nowrap=false`). When `nowrap=false`, an attacker-supplied relative timelock value that overflows the integer type causes the addition to wrap to a small number, making the timelock comparison trivially pass. The same spend bundle is **rejected** under `nowrap=true` and **accepted** under `nowrap=false`. Because the parameter is forwarded verbatim from the Python binding with no enforcement of a canonical value, any caller that passes `nowrap=false` opens a timelock-bypass path reachable by unprivileged spend-bundle input.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` evaluates four relative-timelock conditions per spend:

```
height_relative:        prev_height < confirmed + height_relative
seconds_relative:       timestamp   < coin_time + seconds_relative
before_height_relative: prev_height >= confirmed + before_height_relative
before_seconds_relative:timestamp   >= coin_time + before_seconds_relative
```

The addition is performed with two mutually exclusive arithmetic modes selected by `nowrap`:

```rust
// nowrap=true  → saturating (correct, new behavior)
.saturating_add(height_relative)

// nowrap=false → wrapping (old behavior, preserved for compat)
.wrapping_add(height_relative)
``` [1](#0-0) 

The Python binding `py_check_time_locks` passes `nowrap` straight through from the Python caller with no validation or enforcement: [2](#0-1) 

The Python-facing type stub confirms `nowrap: bool` is a free parameter supplied by the caller: [3](#0-2) 

The codebase's own tests document the divergence explicitly:

```
// 200 < 100 + u32::MAX → Err with nowrap (saturates), Ok without (wraps)
height_relative = 0xffff_ffff → nowrap=true: Err, nowrap=false: Ok

// 2000 < 1000 + u64::MAX → Err with nowrap (saturates), Ok without (wraps)
seconds_relative = 0xffff_ffff_ffff_ffff → nowrap=true: Err, nowrap=false: Ok

// 200 >= 100 + 0xffff_ffff → Ok with nowrap (saturates), Err without (wraps)
before_height_relative = 0xffff_ffff → nowrap=true: Ok, nowrap=false: Err
``` [4](#0-3) 

The Python integration tests confirm the same divergence end-to-end: [5](#0-4)

### Impact Explanation

**Timelock bypass enabling unauthorized spend acceptance (High).**

Under `nowrap=false`, a spend bundle containing `height_relative = 0xFFFF_FFFF` (u32::MAX) for a coin confirmed at height `H` computes the threshold as `H.wrapping_add(0xFFFF_FFFF) = H - 1`. The check `prev_height < H - 1` is false for any `prev_height >= H - 1`, so the spend is accepted at essentially any block height. The intended semantic — "do not spend this coin for ~4 billion more blocks" — is completely bypassed.

The inverse applies to `before_height_relative` and `before_seconds_relative`: wrapping turns a large value into a small threshold, causing a spend that should be accepted (the deadline has not yet passed) to be incorrectly rejected, or vice versa.

If any consensus-critical caller passes `nowrap=false`, an attacker can:
- Spend a coin protected by a large relative-height or relative-seconds timelock before that timelock expires.
- Craft `before_*` conditions that wrap to force incorrect rejection or acceptance of time-bounded spends.

If different nodes call `check_time_locks` with different `nowrap` values for the same block, they reach different accept/reject decisions for the same spend bundle, producing deterministic consensus divergence (chain split).

---

### Likelihood Explanation

The `nowrap` parameter is fully caller-controlled from Python with no guard in the Rust layer. The Python full-node code (outside this repository) determines which value is passed. The risk materializes in two concrete scenarios:

1. **Inconsistent caller usage**: If the Python full node passes `nowrap=false` during any block-validation path (e.g., for pre-hard-fork blocks or in a code path that was not updated), an attacker who submits a spend bundle with an overflow-triggering relative timelock value bypasses the check.

2. **Cross-node divergence**: If nodes running different software versions pass different `nowrap` values for the same block, they disagree on spend validity, causing a chain split. The parameter's existence as a runtime boolean rather than a compile-time constant or consensus-flag-derived value makes this a live risk during any protocol transition.

The attacker entry path requires only a crafted spend bundle with `height_relative = u32::MAX` (or `seconds_relative = u64::MAX`), which is valid serialized input accepted by `sanitize_uint` and passed through condition parsing without rejection. [6](#0-5) 

---

### Recommendation

1. **Remove the `nowrap` parameter from the public API.** The correct arithmetic mode (saturating) should be hardcoded in `check_time_locks`. If backward-compatible wrapping behavior is needed for pre-fork blocks, derive the mode from a consensus flag or block height passed through the existing `ConsensusFlags` mechanism, not from a free boolean.

2. **If the parameter must remain**, add a `ConsensusFlags`-based gate so the arithmetic mode is determined by the same height-and-constants logic used for all other consensus transitions (`get_flags_for_height_and_constants`), preventing per-caller divergence.

3. **Audit all Python call sites** of `check_time_locks` to confirm `nowrap=True` (saturating) is passed in every consensus-critical path, and add an assertion or type-level enforcement to prevent `nowrap=False` from being used post-fork.

### Proof of Concept

Craft a spend bundle for a coin confirmed at height 100 with `height_relative = 0xFFFF_FFFF`:

```python
from chia_rs import check_time_locks

# coin confirmed at height 100, current block height 200
# height_relative = u32::MAX = 4294967295
# Expected: spend should be REJECTED (requires ~4 billion more blocks)

conds = make_test_conds(height_relative=0xFFFF_FFFF)

# nowrap=True (saturating, correct): 100 + u32::MAX = u32::MAX, 200 < u32::MAX → REJECTED
result_nowrap = check_time_locks(removals, conds, prev_height=200, timestamp=2000, nowrap=True)
assert result_nowrap == 13  # AssertHeightRelativeFailed

# nowrap=False (wrapping, bypass): 100 + u32::MAX wraps to 99, 200 < 99 → ACCEPTED
result_wrap = check_time_locks(removals, conds, prev_height=200, timestamp=2000, nowrap=False)
assert result_wrap is None  # No error — timelock bypassed
```

pub fn sanitize_uint(
    a: &Allocator,
    n: NodePtr,
    max_size: usize,
    code: ValidationErr,
) -> Result<SanitizedUint, ValidationErr> {
    assert!(max_size <= 8);

    let buf = match a.sexp(n) {
        SExp::Atom => a.atom(n),
        SExp::Pair(..) => return Err(code),
    };
    let buf = buf.as_ref();

    if buf.is_empty() {
        return Ok(SanitizedUint::Ok(0));
    }

    // if the top bit is set, it's a negative number
    if (buf[0] & 0x80) != 0 {
        return Ok(SanitizedUint::NegativeOverflow);
    }

    // we only allow a leading zero if it's used to prevent a value to otherwise
    // be interpreted as a negative integer. i.e. if the next top bit is set
    // all other leading zeros are invalid
    if buf == [0_u8] || (buf.len() > 1 && buf[0] == 0 && (buf[1] & 0x80) == 0) {
        return Err(code);
    }

    // strip the leading zero byte if there is one
    let size_limit = if buf[0] == 0 { max_size + 1 } else { max_size };

    // if there are too many bytes left in the value, it's too big
    if buf.len() > size_limit {
        return Ok(SanitizedUint::PositiveOverflow);
    }

    Ok(SanitizedUint::Ok(u64_from_bytes(buf)))
