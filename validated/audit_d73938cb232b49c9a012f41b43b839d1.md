### Title
Native Fee (STRK) Permanently Locked in StarkNet Bridge Contract With No Retrieval Mechanism - (File: starknet/src/omni_bridge.cairo)

### Summary

The StarkNet `OmniBridge` contract collects `native_fee` (STRK tokens) from users during `init_transfer`, transferring them into the contract itself. However, neither the `fin_transfer` function nor any other function in the contract distributes or withdraws these collected STRK tokens. They are permanently locked with no retrieval path for the relayer, admin, or any party.

### Finding Description

In `starknet/src/omni_bridge.cairo`, the `init_transfer` function accepts a `native_fee` parameter denominated in STRK tokens. When `native_fee > 0`, the contract pulls STRK from the caller directly into itself:

```cairo
if native_fee > 0 {
    let native_token = self.strk_token_address.read();
    let success = IERC20Dispatcher { contract_address: native_token }
        .transfer_from(caller, get_contract_address(), native_fee.into());
    assert(success, 'ERR_FEE_TRANSFER_FAILED');
}
``` [1](#0-0) 

The entire public interface of the contract is:

```cairo
pub trait IOmniBridge<TContractState> {
    fn log_metadata(...);
    fn deploy_token(...);
    fn fin_transfer(...);
    fn init_transfer(...);
    fn upgrade_token(...);
    fn set_pause_flags(...);
    fn pause_all(...);
    fn get_token_address(...);
    fn is_bridge_token(...);
    fn is_transfer_finalised(...);
}
``` [2](#0-1) 

None of these functions withdraw or distribute STRK tokens. The `fin_transfer` function only handles the bridged token amount (`payload.amount`) and does not touch the `native_fee` at all:

```cairo
fn fin_transfer(ref self: ContractState, signature: Signature, payload: TransferMessagePayload) {
    ...
    if self.is_bridge_token(payload.token_address) {
        IBridgeTokenDispatcher { contract_address: payload.token_address }
            .mint(payload.recipient, payload.amount.into());
    } else {
        let success = IERC20Dispatcher { contract_address: payload.token_address }
            .transfer(payload.recipient, payload.amount.into());
        assert(success, 'ERR_TRANSFER_FAILED');
    }
    ...
}
``` [3](#0-2) 

By contrast, the NEAR bridge has a full `claim_fee` / `claim_fee_callback` flow that explicitly sends the native fee to the relayer:

```rust
} else if origin_chain == ChainKind::Near {
    Promise::new(fee_recipient.clone())
        .transfer(NearToken::from_yoctonear(transfer_message.fee.native_fee.0))
        .detach();
} else {
    ext_token::ext(self.get_native_token_id(origin_chain))
        .with_static_gas(MINT_TOKEN_GAS)
        .mint(fee_recipient.clone(), transfer_message.fee.native_fee, None)
        .detach();
}
``` [4](#0-3) 

The StarkNet bridge has no equivalent distribution path. The STRK tokens transferred in as `native_fee` accumulate in the contract with no exit.

### Impact Explanation

Every user who calls `init_transfer` on StarkNet with `native_fee > 0` permanently loses those STRK tokens. They are transferred into the bridge contract and can never be recovered by the relayer, the user, or the admin. This constitutes an irrecoverable lock of user funds in the bridge flow, matching the "Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows" impact category.

### Likelihood Explanation

The `native_fee` parameter is a first-class feature of the bridge protocol — it is the mechanism by which users incentivize relayers to process their cross-chain transfers. Any user who sets `native_fee > 0` (which is the normal operating mode for competitive relayer incentivization) will trigger this loss. The entry path is a standard, unprivileged user action with no special preconditions.

### Recommendation

Implement a `claim_native_fee` function (analogous to the NEAR bridge's `claim_fee` flow) that allows an authorized relayer or admin to withdraw accumulated STRK native fees from the StarkNet bridge contract. Alternatively, distribute the `native_fee` directly to the fee recipient inside `fin_transfer` when the corresponding transfer is finalized, mirroring the NEAR bridge's `send_fee_internal` logic.

### Proof of Concept

1. User calls `init_transfer` on the StarkNet bridge with `native_fee = 1_000_000_000` (1 STRK).
2. The contract executes `transfer_from(caller, get_contract_address(), 1_000_000_000)` — STRK moves into the bridge contract. [1](#0-0) 
3. The `InitTransfer` event is emitted. A relayer picks it up and finalizes the transfer on NEAR.
4. On NEAR, `send_fee_internal` mints wrapped STRK to the relayer — but this is independent of the STRK locked on StarkNet. [5](#0-4) 
5. The 1 STRK in the StarkNet bridge contract has no exit path. No function in `IOmniBridge` can move it out. [2](#0-1) 
6. The STRK is permanently locked.
