### Title
Single-Step Admin Transfer with No Confirmation Enables Permanent Loss of Bridge Admin Control - (File: `solana/programs/bridge_token_factory/src/instructions/admin/change_config.rs`)

### Summary
The Solana `bridge_token_factory` program's `set_admin` instruction immediately overwrites `config.admin` with the caller-supplied `Pubkey` in a single atomic step. There is no pending-admin pattern, no acceptance step, and no way to recover if the wrong key is supplied. Loss of the admin key permanently bricks all admin-gated operations, including unpausing the bridge and updating the `derived_near_bridge_address` used for MPC signature verification on every cross-chain transfer.

### Finding Description
`ChangeConfig::set_admin` writes the new admin directly into on-chain state with no intermediate confirmation:

```rust
// change_config.rs line 22-26
pub fn set_admin(&mut self, admin: Pubkey) -> Result<()> {
    self.config.admin = admin;
    Ok(())
}
```

The `ChangeConfig` account constraint only checks that the current signer equals the current `config.admin`:

```rust
constraint = signer.key() == config.admin @ crate::error::ErrorCode::Unauthorized
```

Once the instruction executes, `config.admin` is the new value. If that value is a typo, a zeroed key, or an address whose private key is unknown, the admin slot is permanently bricked. There is no `pending_admin` field, no `accept_admin` instruction, and no recovery path anywhere in the program.

The same single-step pattern applies to `set_pausable_admin` and `set_metadata_admin` (lines 28-43 of `change_config.rs`), but the `admin` key is the most critical because it is the only account authorised to call `unpause` and `set_derived_near_bridge_address`.

### Impact Explanation
`config.derived_near_bridge_address` is the public key against which every `finalize_transfer` and `deploy_token` MPC signature is verified:

```rust
// lib.rs line 88-91
data.verify_signature(
    (ctx.accounts.mint.key(), ctx.accounts.recipient.key()),
    &ctx.accounts.common.config.derived_near_bridge_address,
)?;
```

If the admin is permanently lost:
1. **Permanent fund freeze** – if the bridge is paused (by `pausable_admin` or `admin`), only `admin` can call `unpause`. With admin gone, the bridge stays paused forever and all in-flight user funds are irrecoverably locked.
2. **Signature verification permanently broken** – if the NEAR-side MPC key rotates, only `admin` can call `set_derived_near_bridge_address`. With admin gone, every subsequent `finalize_transfer` will fail signature verification, permanently preventing settlement of any bridged asset.

Both outcomes match the allowed critical impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation
Admin key rotation is a routine operational event (multisig migration, hardware wallet replacement, key ceremony). A single-character typo in a 32-byte base58 Solana public key produces a syntactically valid but uncontrolled `Pubkey`. The absence of any confirmation step means the error is irreversible the moment the transaction lands on-chain. This is a realistic, non-exotic operational mistake.

### Recommendation
Implement a two-step transfer pattern:
1. Add a `pending_admin: Option<Pubkey>` field to `Config`.
2. Replace `set_admin` with `propose_admin` (sets `pending_admin`, does not touch `admin`).
3. Add an `accept_admin` instruction that requires the signer to equal `pending_admin`, then atomically moves `pending_admin` into `admin` and clears `pending_admin`.

This mirrors the `Ownable2Step` pattern already used for EVM `BridgeToken` contracts in this same repository (see the `transfer-token-ownership` hardhat task which explicitly documents the two-step flow).

### Proof of Concept
1. Current admin calls `set_admin` with a mistyped `Pubkey` (e.g., one byte off from the intended multisig address).
2. Transaction succeeds; `config.admin` is now an address nobody controls.
3. An emergency arises; operator calls `pause` (succeeds, `pausable_admin` still works).
4. Operator attempts `unpause` – requires `ChangeConfig` with `signer == config.admin`. No valid signer exists. Transaction reverts with `Unauthorized`.
5. All `init_transfer` and `finalize_transfer` calls revert with `Paused`.
6. User funds locked in the SOL vault and token vaults are permanently unclaimable. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
