Looking at the full call chain in the code to determine if the precondition (header poisoning) is achievable and if the subsequent `set_state_part` attack follows.

### Title
Missing shard_id binding in `set_state_header` enables cross-shard state root substitution during state sync — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` never verifies that the chunk embedded in the supplied `ShardStateSyncResponseHeader` actually belongs to the `shard_id` parameter. An unprivileged P2P peer can supply shard B's chunk (with a valid position-agnostic Merkle proof against the block's `chunk_headers_root`) as the header for shard A's state sync. This poisons `StateHeaderKey(A, sync_hash)` with shard B's `prev_state_root`. A subsequent `set_state_part` call reads that poisoned root, accepts any genuine shard-B state part as valid for shard A, and stores it under `StatePartKey(sync_hash, A, P)`. `apply_state_part` then installs shard B's trie nodes into shard A's `ShardUId` prefix.

---

### Finding Description

**Step 1 — Header poisoning (the precondition is itself reachable)**

In `set_state_header`: [1](#0-0) 

The five validation checks are:

1. `validate_chunk_proofs(&chunk, …)` — validates internal chunk consistency (hash, tx root, receipts root). It does **not** check `chunk.shard_id() == shard_id`.
2. `verify_path(*sync_prev_block_header.chunk_headers_root(), shard_state_header.chunk_proof(), &ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` — verifies the chunk is somewhere in the block's Merkle tree.

The critical point is that `verify_path` is **position-agnostic**: [2](#0-1) 

It only checks `compute_root_from_path(path, item_hash) == root`. It does **not** check which leaf index (shard index) the item occupies. Contrast with `verify_path_with_index`, which additionally calls `verify_path_matches_index`. Because shard B's chunk is a genuine member of the block's `chunk_headers_root` Merkle tree, shard B's chunk hash + shard B's Merkle proof will satisfy the check even when the call is for shard A.

3. Receipt proof validation (step 4) uses the `shard_id` parameter (shard A) to compute `ReceiptList(shard_id, receipts)`. The attacker supplies the actual on-chain incoming receipts for shard A (publicly available), so this passes.
4. `validate_state_root_node` is called against `chunk_inner.prev_state_root()` — shard B's root — with shard B's state root node. Passes.

There is **no check** `chunk.shard_id() == shard_id` anywhere in `set_state_header`. [3](#0-2) 

The header is stored under `StateHeaderKey(shard_id=A, sync_hash)` but contains shard B's chunk and shard B's `prev_state_root`.

**Step 2 — Part injection**

`set_state_part` reads the poisoned header: [4](#0-3) 

- `state_root = chunk.take_header().take_inner().prev_state_root()` → shard B's state root.
- `validate_state_part(shard_id=A, &B_root, part_id, part)` calls `Trie::validate_state_part(B_root, part_id, partial_state)`, which passes for any genuine shard-B state part. [5](#0-4) 

- The part is stored under `StatePartKey(sync_hash, A, part_id)`.

**Step 3 — Application**

`apply_state_part` derives `shard_uid` from `shard_id=A` and installs the trie changes: [6](#0-5) 

Shard B's trie nodes are written into shard A's `ShardUId` prefix in the trie store, and shard B's flat-state delta is applied to shard A's flat storage.

---

### Impact Explanation

Shard A's entire trie state is replaced with shard B's data. All RPC queries, transaction execution, and receipt routing for shard A operate on the wrong state. If the victim node is a validator or chunk producer for shard A, it will produce blocks/chunks with an incorrect `prev_state_root`, which the rest of the network will reject, potentially triggering slashing.

---

### Likelihood Explanation

The attack requires the adversary to be a P2P peer from which the victim node is downloading state sync data. Any peer can serve state sync responses — this is not a privileged role. The attacker only needs publicly available on-chain data (shard B's chunk, its Merkle proof, shard A's incoming receipts, and shard B's state parts). No validator key or operator access is required.

---

### Recommendation

Add an explicit shard-id binding check in `set_state_header` immediately after extracting the chunk:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, replace the position-agnostic `verify_path` call for the chunk Merkle proof with `verify_path_with_index`, supplying the expected shard index derived from the epoch's shard layout, so that a chunk from the correct shard but at the wrong position is also rejected.

---

### Proof of Concept

1. Spin up a 2-shard V1 nearcore test environment.
2. Intercept the state sync header request for shard A; substitute shard B's `ShardStateSyncResponseHeader` (genuine chunk, genuine Merkle proof, genuine shard-A incoming receipts, shard B's `state_root_node`).
3. Call `set_state_header(shard_id=A, sync_hash, header_B)` — all five checks pass.
4. For each part `P` in `0..num_parts_B`, call `set_state_part(shard_id=A, sync_hash, P, genuine_part_B_P)` — `validate_state_part` passes against `B_root`.
5. Call `set_state_finalize(shard_id=A, sync_hash)` → `apply_state_part` installs shard B's nodes under shard A's `ShardUId`.
6. Assert: querying shard A's trie root returns `B_root`, not `A_root`.
