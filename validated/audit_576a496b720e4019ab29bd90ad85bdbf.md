### Title
Unauthenticated `RemoteCommitterServer` Allows Any Network Peer to Inject Arbitrary `CommitBlock` Requests, Corrupting the Global State Root — (`crates/apollo_committer/src/communication.rs`, `crates/apollo_infra/src/component_server/remote_component_server.rs`)

### Summary

The `RemoteCommitterServer` exposes `CommitBlock` and `RevertBlock` endpoints over plain TCP/HTTP with no authentication. In the distributed deployment layout the committer binds to `0.0.0.0` and is reachable by any pod in the cluster. Any unprivileged network peer can inject a `CommitBlock` request carrying an attacker-chosen `ThinStateDiff`, causing the committer to apply that diff to the Patricia trie, store a wrong `GlobalRoot` and `StateDiffHash`, and advance its offset — permanently preventing the batcher from committing the correct block and corrupting the state root that feeds into block-hash and proof-input computation.

### Finding Description

**Root cause — no caller authentication on `RemoteCommitterServer`.**

`communication.rs` registers `CommitterRequest::CommitBlock` and `CommitterRequest::RevertBlock` as the only two request variants handled by the committer:

```rust
CommitterRequest::CommitBlock(commit_block_request) => {
    CommitterResponse::CommitBlock(self.commit_block(commit_block_request).await)
}
``` [1](#0-0) 

The `RemoteComponentServer` handler that receives these requests over the network performs no authentication whatsoever — it deserializes the body and forwards it directly:

```rust
let http_response = match SerdeWrapper::<Request>::wrapper_deserialize(&body_bytes) {
    Ok(request) => {
        let response = tokio::spawn(async move { local_client.send(request).await })
            .await
            .expect("...");
        // return 200 OK
    }
``` [2](#0-1) 

`RemoteServerConfig` carries no TLS, no API key, no IP allowlist — only TCP keepalive and nodelay knobs. In every production deployment layout the committer is configured with `bind_ip: "0.0.0.0"` and `remote_server_config.#is_none: false`: [3](#0-2) [4](#0-3) 

**Attack path.**

An attacker who can reach the committer's TCP port (any pod in the same Kubernetes cluster, or any process on the same host in a non-containerised deployment) sends:

```
POST /  body = CommitBlock { state_diff: <attacker_diff>, state_diff_commitment: Some(hash(<attacker_diff>)), height: N }
```

Inside `commit_block_inner`, with `verify_state_diff_hash: true` (the production default), the committer recomputes the hash and accepts it because the attacker supplied a self-consistent pair: [5](#0-4) 

The committer then applies `<attacker_diff>` to the Patricia trie, stores the resulting wrong `GlobalRoot` and wrong `StateDiffHash` under block `N`, and advances its offset to `N+1`: [6](#0-5) 

When the batcher subsequently sends the legitimate `CommitBlock` for height `N`, the committer sees `height < self.offset`, loads the stored (attacker-written) `state_diff_commitment`, compares it with the batcher's correct commitment, and returns `CommitterError::InvalidStateDiffCommitment`: [7](#0-6) 

The batcher cannot recover: the committer's Patricia trie is now rooted at the attacker's state, and the offset has moved past the legitimate block.

### Impact Explanation

The corrupted `GlobalRoot` stored under `ForestMetadataType::StateRoot(DbBlockNumber(N))` is the authoritative Starknet state root. It is read back by the commitment manager and used to compute `PartialBlockHashComponents`, which feeds `calculate_block_hash` and ultimately the SNOS proof inputs. A wrong global root means:

- The block hash committed to consensus is wrong.
- The SNOS proof input carries a wrong pre/post state root, making any generated proof invalid or binding the wrong state.
- The batcher is permanently halted for all subsequent heights because the committer's offset is stuck one block ahead with wrong state.

This matches **Critical — Wrong state … storage value … from blockifier/syscall/execution logic for accepted input** and **Critical — Incorrect … proof inputs/outputs**.

### Likelihood Explanation

The committer is deployed as a separate Kubernetes pod with a `ClusterIP` service. Any other pod in the same namespace — including a compromised gateway, mempool, or RPC pod — can reach it. No exploit primitive beyond a plain HTTP POST is required. The attack is deterministic and requires no brute-force.

### Recommendation

Add mutual TLS (mTLS) or a shared-secret HMAC header to `RemoteComponentServer` so that only the batcher can send `CommitBlock` requests. At minimum, add an IP allowlist in `RemoteServerConfig` that restricts connections to the batcher's known address. As a defence-in-depth measure, add a Kubernetes `NetworkPolicy` that permits ingress to the committer port only from the batcher pod's label selector.

### Proof of Concept

```bash
# 1. Attacker crafts a minimal ThinStateDiff and its Poseidon commitment.
# 2. Attacker sends CommitBlock for the current committer offset N.
curl -X POST http://<committer-service>:<port>/ \
  -H "Content-Type: application/octet-stream" \
  --data-binary @attacker_commit_block_request.bin

# 3. Committer applies attacker diff, stores wrong GlobalRoot, advances offset to N+1.
# 4. Batcher sends legitimate CommitBlock for height N.
# 5. Committer returns InvalidStateDiffCommitment — batcher is halted.
# 6. Wrong GlobalRoot is now the authoritative state root for block N,
#    corrupting all downstream block-hash and proof-input computations.
```

The serialised `CommitBlock` request body is a `SerdeWrapper<CommitterRequest>` (bincode/serde), trivially constructable from the public `CommitterRequest` and `CommitBlockRequest` types defined in `crates/apollo_committer_types/src/communication.rs`. [8](#0-7)
