### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Contract Bricking - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary felt value as the new class hash without verifying that a class with that hash has actually been declared on-chain. This is an exact structural analog to the `settleAuction()` bug: a state-transition function that performs an irreversible write without checking the required precondition. Any contract can permanently set its own class hash to an undeclared value, making the contract permanently non-executable and freezing all funds it holds.

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the `ReplaceClassRequest`, deducts gas, and then directly updates `contract_state_changes` with the caller-supplied `class_hash` — with no check that the hash corresponds to a previously declared class:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The TODO comment explicitly acknowledges the missing check. The deadline of `1/1/2026` has already passed (today is 2026-07-03) and the check is still absent.

The syscall is dispatched unconditionally from `execute_syscalls.cairo`:

```cairo
if (selector == REPLACE_CLASS_SELECTOR) {
    execute_replace_class(contract_address=execution_context.execution_info.contract_address);
``` [2](#0-1) 

The OS validates compiled class facts only at the end of block execution via `validate_compiled_class_facts_post_execution`, not at the point of `replace_class` invocation. Once the transaction that calls `replace_class(invalid_hash)` is committed, the contract's on-chain class hash is permanently set to the invalid value. Any subsequent transaction targeting that contract will fail to resolve the class and revert, but the state entry is already committed and cannot be recovered without a separate upgrade transaction — which itself requires the contract to be callable.

### Impact Explanation

**Critical — Permanent freezing of funds.**

If a contract holds user funds (e.g., a multisig wallet, a DeFi vault, an ERC-20 token contract) and its class hash is replaced with an undeclared hash, all future calls to that contract will fail at class resolution. The funds are permanently inaccessible. There is no recovery path because the recovery transaction itself would require calling the now-broken contract.

### Likelihood Explanation

The entry path is fully unprivileged. Any contract deployer can:

1. Deploy a contract that accepts user deposits and exposes an upgrade function calling `replace_class(attacker_controlled_hash)`.
2. Attract user deposits.
3. Call the upgrade function with an arbitrary undeclared hash (e.g., `1`).
4. The OS commits the state change without validation.
5. All user funds are permanently frozen.

Additionally, a legitimate protocol with an on-chain governance upgrade path is vulnerable if a governance participant can supply an arbitrary class hash — the OS provides no safety net.

### Recommendation

Inside `execute_replace_class`, before writing the new `StateEntry`, verify that `class_hash` is present in `contract_class_changes` (i.e., it was declared in the current block) **or** that it exists in the committed class trie (i.e., it was declared in a prior block). The check should mirror the validation already performed for `compiled_class_hash` in `execute_declare_transaction`:

```cairo
// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

If the class was declared in a prior block, a hint-assisted lookup into the committed class trie (analogous to `read_block_hash_from_storage`) should be used.

### Proof of Concept

1. Declare a valid class `C` and deploy contract `Victim` using class `C`. `Victim` holds 1000 STRK in storage.
2. `Victim` exposes an `upgrade(new_hash: felt)` entry point that calls `replace_class(new_hash)`.
3. Attacker sends an invoke transaction calling `Victim.upgrade(0xdeadbeef)` where `0xdeadbeef` is never declared.
4. The OS executes `execute_replace_class`: gas is deducted, `dict_update` writes `class_hash=0xdeadbeef` into `contract_state_changes`. No validation occurs.
5. The block is proven and settled. `Victim`'s on-chain class hash is now `0xdeadbeef`.
6. Any subsequent transaction calling any entry point of `Victim` fails at class resolution — the 1000 STRK are permanently frozen. [3](#0-2)
