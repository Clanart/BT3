### Title
`eth_getProof` Always Returns Zero `storageHash`, Breaking Storage Proof Verification for All Integrators — (`rpc/backend/account_info.go`)

### Summary

`GetProof` in Ethermint's JSON-RPC backend hardcodes `StorageHash` to `common.Hash{}` (all-zero bytes) in every `eth_getProof` response. Any integrator — bridge, light client, or cross-chain protocol — that relies on `storageHash` to anchor or verify storage Merkle proofs receives permanently incorrect data, mirroring the same class of bug as the external report's hardcoded `updatedAt = 0`.

### Finding Description

In `rpc/backend/account_info.go`, `GetProof` assembles the `AccountResult` struct returned by `eth_getProof`:

```go
return &rpctypes.AccountResult{
    Address:      address,
    AccountProof: GetHexProofs(proof),
    Balance:      (*hexutil.Big)(balance.BigInt()),
    CodeHash:     common.HexToHash(res.CodeHash),
    Nonce:        hexutil.Uint64(res.Nonce),
    StorageHash:  common.Hash{}, // NOTE: Ethermint doesn't have a storage hash. TODO: implement?
    StorageProof: storageProofs,
}, nil
``` [1](#0-0) 

`StorageHash` is unconditionally set to the zero hash (`0x000...000`). In the Ethereum specification, `storageHash` is the Keccak-256 root of the account's storage trie; it is the anchor against which each `storageProof[i].proof` Merkle path must be verified.

The `AccountResult` type exposes this field directly to callers:

```go
type AccountResult struct {
    ...
    StorageHash  common.Hash     `json:"storageHash"`
    StorageProof []StorageResult `json:"storageProof"`
}
``` [2](#0-1) 

The public RPC endpoint delegates directly to this backend function:

```go
func (e *PublicAPI) GetProof(address common.Address,
    storageKeys []string,
    blockNrOrHash *rpctypes.BlockNumberOrHash,
) (*rpctypes.AccountResult, error) {
    ...
    return e.backend.GetProof(address, storageKeys, blockNrOrHashOrLatest(blockNrOrHash))
}
``` [3](#0-2) 

The actual per-slot storage proofs (`storageProofs[i].proof`) are fetched against the real IAVL store:

```go
valueBz, proof, err := b.queryClient.GetProof(clientCtx, evmtypes.StoreKey, evmtypes.StateKey(address, hexKey))
``` [4](#0-3) 

These proofs are valid Cosmos/IAVL proofs, but they are proofs against the real storage trie root — **not** against the zero hash that is returned as `storageHash`. The two values are permanently inconsistent.

### Impact Explanation

Any integrator that follows the standard Ethereum pattern for `eth_getProof` verification:

1. Fetch `eth_getProof` → receive `storageHash = 0x000...000`
2. Verify `storageProof[i].proof` against `storageHash`

will find that every legitimate proof fails verification (the proof is against the real trie root, not zero). Conversely, an attacker who knows `storageHash` is always zero can construct a trivial proof against the empty trie and present it as valid to any verifier that trusts the `storageHash` field from the RPC response.

For cross-chain bridges or light-client relayers that use `eth_getProof` to confirm that a user has locked funds in a contract on Ethermint before releasing funds on another chain, this creates a path to unauthorized fund release: the attacker supplies a proof against the empty trie, the bridge reads `storageHash = 0x000...000` from Ethermint's RPC, the proof trivially verifies, and funds are released without any actual deposit.

This falls under the allowed High impact: **Public JSON-RPC path feeds incorrect consensus-critical data and exposes a reachable route to unauthorized fund transfer**.

### Likelihood Explanation

- Every single call to `eth_getProof` on any Ethermint node returns `storageHash = 0x000...000`, regardless of the account, block, or storage keys queried.
- No special conditions, privileges, or timing are required.
- The entry path is a standard, publicly documented Ethereum JSON-RPC method callable by any unprivileged user or integrator.
- The bug is explicitly acknowledged in the source code with a `TODO` comment, confirming it is a known unimplemented field rather than an intentional design choice.

### Recommendation

Compute the actual storage trie root for the queried account at the requested block height and return it as `storageHash`. If the Cosmos/IAVL store does not expose a per-account storage root directly, derive it by hashing the IAVL subtree root for the account's storage prefix, or document clearly that `eth_getProof` is not supported and return an explicit error rather than silently returning a zero hash that appears valid.

### Proof of Concept

```go
// Any caller of eth_getProof receives storageHash = 0x000...000
result, _ := backend.GetProof(
    common.HexToAddress("0xSomeContractWithStorage"),
    []string{"0x0"},
    rpctypes.BlockNumberOrHash{BlockNumber: &latestBlock},
)

// storageHash is always zero regardless of actual storage state
assert(result.StorageHash == common.Hash{})  // always true

// The actual proof in result.StorageProof[0].Proof is a valid IAVL proof
// against the REAL storage root — NOT against the zero hash.
// Verifying proof against result.StorageHash (zero) will always FAIL for
// legitimate proofs, and will PASS for a trivially constructed empty-trie proof.

// Attacker constructs a proof against the empty trie:
fakeProof := constructEmptyTrieProof(slot, value)
// Bridge reads storageHash = 0x000...000 from eth_getProof
// Bridge verifies fakeProof against 0x000...000 → passes
// Bridge releases funds on destination chain → unauthorized transfer
``` [5](#0-4)
