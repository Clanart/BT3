### Title
`nativeFee` ETH Permanently Locked in `OmniBridge.initTransfer` with No Release Mechanism — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

In `OmniBridge.initTransfer`, the `nativeFee` portion of `msg.value` is deducted when computing `extensionValue` (the Wormhole publishing fee), but the `nativeFee` ETH itself is never forwarded to any relayer, fee recipient, or returned to the caller. It is permanently locked in the contract with no withdrawal or distribution path.

---

### Finding Description

`OmniBridge.initTransfer` is `payable` and accepts a user-supplied `nativeFee` parameter alongside `amount` and the Wormhole message fee:

```solidity
// OmniBridge.sol lines 386–413
uint256 extensionValue;
if (tokenAddress == address(0)) {
    if (fee != 0) { revert InvalidFee(); }
    extensionValue = msg.value - amount - nativeFee;   // nativeFee stays in contract
} else {
    extensionValue = msg.value - nativeFee;             // nativeFee stays in contract
    // ... ERC20 burn/transfer
}

initTransferExtension(..., extensionValue);
```

`extensionValue` is forwarded to `_wormhole.publishMessage{value: value}` in `OmniBridgeWormhole.initTransferExtension` (line 143). The Wormhole test stub confirms the exact-fee requirement (`require(msg.value == this.messageFee(), "invalid fee")`), so `extensionValue` must equal `_wormhole.messageFee()` exactly.

This means the user must send:
- **ERC20 path**: `msg.value = nativeFee + wormhole_messageFee`
- **Native ETH path**: `msg.value = amount + nativeFee + wormhole_messageFee`

In both cases, `nativeFee` ETH is retained by the contract. Searching the entire `OmniBridge.sol` and `OmniBridgeWormhole.sol`, the only ETH outflow is in `finTransfer` for native ETH transfers:

```solidity
// OmniBridge.sol lines 317–322
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
```

This releases only `payload.amount` to the recipient — not `nativeFee`. There is no `withdraw`, no admin recovery function, and no relayer payment path for the accumulated `nativeFee` ETH. The contract has a bare `receive() external payable {}` (line 574) that accepts ETH but provides no egress for it.

---

### Impact Explanation

Every call to `initTransfer` with `nativeFee > 0` permanently locks that ETH in the bridge contract. The funds are irrecoverable: no admin withdrawal function exists, no relayer is paid from this balance on the EVM side, and no cross-chain message triggers its release. Over time, the locked ETH accumulates proportionally to bridge usage.

This matches: **Critical — Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

`nativeFee` is a standard, documented parameter of the public `initTransfer` function. Any user bridging ERC20 tokens or native ETH who sets `nativeFee > 0` (e.g., to incentivize a relayer to finalize their transfer on the destination chain) will permanently lose that ETH. The function imposes no upper bound on `nativeFee` and no warning that it is non-refundable. Likelihood is **High** — this affects every ordinary bridge user who uses the relayer-fee mechanism.

---

### Recommendation

Either:
1. **Return `nativeFee` to `msg.sender`** after the Wormhole publish call if it is not consumed on-chain, or
2. **Require `nativeFee == 0`** for EVM-side `initTransfer` if the native fee is only meaningful on the destination chain, or
3. **Distribute `nativeFee` to `msg.sender` of `finTransfer`** (the relayer) when a corresponding inbound transfer is finalized on EVM, implementing a proper relayer-payment mechanism.

At minimum, add a strict check: `require(msg.value == amount + nativeFee + _wormhole.messageFee())` so users cannot accidentally over-send, and document clearly that `nativeFee` ETH is locked in the contract.

---

### Proof of Concept

1. Deploy `OmniBridgeWormhole` with `TestWormhole` (fee = 10000 wei).
2. Call `initTransfer(USDC, 100e6, 0, 1e15, "alice.near", "")` with `msg.value = 1e15 + 10000`.
3. Observe: `extensionValue = 1e15 + 10000 - 1e15 = 10000` → forwarded to Wormhole (exact fee, succeeds).
4. `nativeFee = 1e15` wei remains in the contract.
5. Call `initTransfer` 1000 times with `nativeFee = 1e15`. Contract accumulates 1 ETH of locked `nativeFee`.
