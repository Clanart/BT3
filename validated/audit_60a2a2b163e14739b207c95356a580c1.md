Audit Report

## Title
Single `MAX_REF_STALENESS` Applied to Both Legs of Synthetic Ratio Feeds Allows Stale Quote-Leg Price to Reach Pool Swap — (`smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

## Summary

`AnchoredPriceProvider` supports a two-feed synthetic ratio mode where `_getBidAndAskPrice()` calls `_readLeg` for both `baseFeedId` and `quoteFeedId`. Both legs are staleness-checked against the single immutable `MAX_REF_STALENESS`. Because the two feeds can have fundamentally different update heartbeats, no single threshold is simultaneously correct for both: setting it to the slower feed's heartbeat allows the faster feed to be arbitrarily stale, producing a wrong synthetic mid-price that reaches the pool's bid/ask and enables value extraction from LP principal.

## Finding Description

`_getBidAndAskPrice()` reads both legs through `_readLeg`:

```solidity
// AnchoredPriceProvider.sol L258-272
function _getBidAndAskPrice() internal returns (uint128, uint128) {
    (uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
    if (!ok) return (0, type(uint128).max);

    bytes32 _quote = quoteFeedId;
    if (_quote != bytes32(0)) {
        (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
        if (!ok2 || mid2 == 0) return (0, type(uint128).max);
        mid = Math.mulDiv(mid, ORACLE_DECIMALS, mid2);  // mid2 may be stale
        spreadBps += spreadBps2;
    }
    return _computeBidAsk(mid, spreadBps);
}
```

Inside `_readLeg`, the single immutable `MAX_REF_STALENESS` is applied to whichever feed is passed:

```solidity
// AnchoredPriceProvider.sol L283
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
```

`MAX_REF_STALENESS` is a single immutable set at construction with no per-leg variant:

```solidity
// AnchoredPriceProvider.sol L77
uint256 public immutable MAX_REF_STALENESS;
```

The factory's envelope validation is keyed exclusively on `baseFeedId`'s class — there is no envelope, parameter, or validation path for the quote leg's heartbeat:

```solidity
// AnchoredProviderFactory.sol L171-180
bytes32 classId = feedClass[baseFeedId];
if (classId == bytes32(0)) classId = DEFAULT_CLASS;
Envelope storage env = envelopes[classId];
if (!env.exists) revert EnvelopeNotFound(classId);
if (
    minMargin < env.minMarginMin || minMargin > env.minMarginMax
    || maxRefStaleness < env.stalenessMin || maxRefStaleness > env.stalenessMax
    || maxSpreadBps < env.maxSpreadMin || maxSpreadBps > env.maxSpreadMax
) revert ParamsOutOfEnvelope();
```

**Exploit path:**
1. Deployer creates a synthetic ratio provider: `baseFeedId` = BTC/ETH (86400s heartbeat), `quoteFeedId` = ETH/USD (27s heartbeat). To avoid constant base-leg reverts, `MAX_REF_STALENESS` must be set to ≥ 86400s — there is no other valid choice.
2. ETH/USD oracle goes 3600s without an update (network congestion, oracle inactivity). ETH price moves 5% in real markets.
3. Trader calls swap → pool calls `getBidAndAskPrice()` → `_readLeg(quoteFeedId)`: `refTime` is 3600s old, `3600 ≤ 86400` → staleness check passes, `ok2 = true`.
4. `mid = Math.mulDiv(btcEthMid, ORACLE_DECIMALS, staleEthUsdMid)` — wrong synthetic BTC/USD price.
5. `_computeBidAsk` returns bid/ask based on the stale price; trader swaps at the stale rate, extracting LP principal.

**Existing guards are insufficient:**
- `spreadBps >= ORACLE_BPS`: catches the Chainlink off-hours/stall marker, not temporal staleness.
- `priceGuard` (min/max bounds): set at deployment, typically wide enough to accommodate normal price movement.
- `MAX_SPREAD_BPS`: checks oracle uncertainty magnitude, not data age.

## Impact Explanation

A stale `mid2` in the ratio computation `Math.mulDiv(mid, ORACLE_DECIMALS, mid2)` produces a materially wrong synthetic bid/ask that reaches the pool swap. An informed trader who observes the real market price divergence can swap against the pool at the stale price, receiving tokens worth more than the oracle-intended rate. This is a direct loss of LP principal through bad-price execution — the pool's conservation invariant (trader receives no more than the oracle permits) is violated. Severity is High: the loss is proportional to the price divergence and pool liquidity, and the attack is repeatable for the duration of the staleness window.

## Likelihood Explanation

- Synthetic ratio mode (`quoteFeedId != 0`) is an explicitly documented and supported deployment path in `createAnchoredProvider` (factory NatSpec: "optional second feed for synthetic ratio quoting").
- Feed pairs with mismatched heartbeats are common: any illiquid/liquid pair (BTC/ETH + ETH/USD), any RWA feed + ETH/USD, etc.
- The deployer has no correct single value for `MAX_REF_STALENESS` when heartbeats differ — the dilemma is structural, not a configuration mistake.
- The trigger is unprivileged: any trader initiating a swap causes `getBidAndAskPrice()` to be called.

## Recommendation

Introduce a separate staleness parameter for the quote leg. Replace the single `MAX_REF_STALENESS` immutable with two — `MAX_BASE_STALENESS` and `MAX_QUOTE_STALENESS` — and pass the appropriate one to `_readLeg` for each leg. Extend `createAnchoredProvider` and the factory's envelope validation to accept and independently bound both parameters, with the quote-leg envelope keyed on `quoteFeedId`'s class (or a separate admin-configured bound).

## Proof of Concept

1. Deploy `AnchoredPriceProvider` with `baseFeedId` = BTC/ETH (86400s heartbeat), `quoteFeedId` = ETH/USD (27s heartbeat), `MAX_REF_STALENESS` = 86400s (required to avoid constant base-leg reverts).
2. In a Foundry fork test, `vm.warp(block.timestamp + 3600)` without pushing a new ETH/USD oracle update.
3. Call `getBidAndAskPrice()` through the pool:
   - `_readLeg(baseFeedId)`: recent `refTime` → passes, `ok = true`.
   - `_readLeg(quoteFeedId)`: `refTime` is 3600s old, `3600 ≤ 86400` → passes, `ok2 = true`.
   - `mid = Math.mulDiv(btcEthMid, 1e8, staleEthUsdMid)` → wrong BTC/USD synthetic price.
4. Assert that the returned bid/ask differs from the correct price by ~5% (matching the real ETH price move).
5. Execute a swap at the stale price and assert the trader receives more value than the oracle-correct rate would permit, confirming LP principal loss.
