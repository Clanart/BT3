### Title
EIP-7702 Authorization Signature Bound Only to Nonce — Zero Chain-ID Enables Cross-Chain Replay of `SetCodeAuthorization` - (File: `x/evm/keeper/set_code_authorizations.go`)

### Summary

The `validateAuthorization` function in Ethermint's EIP-7702 implementation explicitly permits authorizations signed with a zero chain ID (`auth.ChainID.IsZero()`). An authorization signed with `chainID = 0` is valid on **every** EVM-compatible chain that runs Ethermint (or any EIP-7702-compatible chain). Because the only remaining binding is the authority's nonce, an attacker who observes a valid zero-chain-ID authorization on one Ethermint chain can replay it on any other Ethermint chain where the authority account happens to have the same nonce — installing arbitrary delegation code on the victim's EOA without their consent on the target chain.

This is the direct analog of the `IdRegistry` bug: the signed credential (`SetCodeAuthorization`) does not include enough state (specifically, the chain ID is optional/zero) to prevent it from being applied in a context the signer never intended.

---

### Finding Description

In `validateAuthorization`:

```go
// x/evm/keeper/set_code_authorizations.go:17
if !auth.ChainID.IsZero() && auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
    return authority, core.ErrAuthorizationWrongChainID
}
```

The check is skipped entirely when `auth.ChainID == 0`. This is intentional per EIP-7702 spec (zero chain ID means "any chain"), but it creates a concrete cross-chain replay surface in Ethermint's multi-chain deployment model.

The full validation path is:
1. Chain ID: skipped if zero [1](#0-0) 
2. Nonce overflow: checked [2](#0-1) 
3. Signature recovery: checked [3](#0-2) 
4. Nonce match against stateDB: checked [4](#0-3) 

Once validated, `setAuthorizationDelegation` unconditionally writes the delegation code and bumps the nonce: [5](#0-4) 

The durable authorization path in `ApplyMessageWithConfig` then commits these effects even when the outer EVM call fails (post-hook failure scenario), via `applyDurableAuthorization` → `durableStateDB.Commit()`: [6](#0-5) 

And in `ApplyTransaction`, the durable commit is triggered unconditionally when `tmpCtx` was not committed (i.e., on post-hook failure): [7](#0-6) 

The ante handler validates the outer transaction's chain ID via `EthSigVerificationDecorator` and `VerifyEthSig`, but **does not validate the chain IDs of individual authorizations inside the auth list**: [8](#0-7)

### Impact Explanation

An attacker who obtains a `SetCodeAuthorization` signed with `chainID = 0` (or observes one broadcast on chain A) can submit it inside a valid EIP-7702 transaction on chain B (a different Ethermint deployment) where the authority account has the same nonce. The result is that the victim's EOA on chain B gets its code set to `0xef0100<delegate>` — an arbitrary delegation — without the victim ever signing anything on chain B.

With delegation installed, the attacker can call the victim's EOA as if it were a smart contract, executing the delegate's code in the victim's context. If the delegate contains a drain function (e.g., `CALL(CALLER, SELFBALANCE)`), the attacker can steal all EVM-denom funds from the victim's account on chain B.

This matches the **High** allowed impact: "Ethereum transaction, EIP-7702 authorization, nonce, chain-id, or signer verification bypass enabling replay, forged execution, or unauthorized account/code mutation."

---

### Likelihood Explanation

- Zero-chain-ID authorizations are explicitly permitted by EIP-7702 and are a documented feature for "chain-agnostic" delegations. Users or wallets may sign them intentionally for one chain, not realizing they are valid everywhere.
- Ethermint is deployed as multiple independent chains (Evmos, Cronos, etc.) sharing the same codebase. An authorization valid on one is valid on all others with the same nonce state.
- The attacker only needs to observe the authorization (e.g., from a mempool, a failed transaction, or a public broadcast) and submit it on a different chain. No key compromise is required.
- The nonce constraint is the only remaining protection, and nonces on freshly funded accounts (nonce = 0) are trivially predictable.

---

### Recommendation

Ethermint should reject EIP-7702 authorizations with `chainID = 0` at the application layer, even though the EIP-7702 spec permits them. Add an explicit check in `validateAuthorization`:

```go
// Reject chain-agnostic (zero chain ID) authorizations to prevent cross-chain replay.
if auth.ChainID.IsZero() {
    return authority, core.ErrAuthorizationWrongChainID
}
if auth.ChainID.CmpBig(k.eip155ChainID) != 0 {
    return authority, core.ErrAuthorizationWrongChainID
}
```

Alternatively, if zero-chain-ID authorizations must be supported for compatibility, add a governance parameter `AllowZeroChainIDAuthorizations` defaulting to `false`, and document the cross-chain replay risk prominently.

### Proof of Concept

**Setup:**
- Chain A: Ethermint chain, `chainID = 9001`
- Chain B: Ethermint chain, `chainID = 25` (Cronos)
- Victim (`authority`): EOA with nonce = 0 on both chains, funded on chain B
- Attacker controls `delegate` contract on chain B containing drain bytecode `0x600060006000600047335af100`

**Step 1 — Victim signs a zero-chain-ID authorization on chain A:**
```python
auth = {
    "chainId": 0,          # zero = any chain
    "address": delegate,   # attacker-controlled contract
    "nonce": 0,
}
signed_auth = victim.sign_authorization(auth)
```

**Step 2 — Attacker submits the authorization on chain B:**
```python
setcode_tx = {
    "chainId": 25,         # chain B
    "type": 4,
    "to": victim.address,
    "gas": 100000,
    "nonce": attacker_nonce,
    "authorizationList": [signed_auth],  # victim's zero-chain-ID auth
}
```

**Step 3 — `validateAuthorization` on chain B:**
- `auth.ChainID.IsZero()` → `true` → chain ID check **skipped** [1](#0-0) 
- `auth.Authority()` recovers victim's address ✓
- `stateDB.GetNonce(victim) == 0 == auth.Nonce` ✓ [4](#0-3) 
- `setAuthorizationDelegation` installs `0xef0100<delegate>` on victim's account on chain B [9](#0-8) 

**Step 4 — Attacker calls victim's EOA on chain B:**
The delegation is now active. The attacker calls `victim.address` with a drain payload; the delegate's `CALL(CALLER, SELFBALANCE)` transfers all of victim's chain-B funds to the attacker.

**Result:** Victim's funds on chain B are stolen via a signature they created for chain A (or for "any chain"), without the victim ever interacting with chain B.

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.

}
