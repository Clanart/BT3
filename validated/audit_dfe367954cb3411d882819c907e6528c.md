### Title
Orphan `ChunkStateWitness` Overwrites Legitimate Witness Without Signature Validation, Suppressing Validator Endorsements — (File: `chain/client/src/stateless_validation/chunk_validation_actor.rs`)

---

### Summary

Any network peer can send a `ChunkStateWitness` carrying a valid `ChunkProductionKey` but a garbage chunk-header signature. When the witness's `prev_block` is absent, `process_chunk_state_witness_message` stores it in `OrphanStateWitnessPool` **without validating the signature**. The pool's `add_orphan_state_witness` uses LRU `push`, which **silently overwrites** any existing entry for the same key. An attacker who races a legitimate orphan witness can replace it with an invalid one; when the missing block later arrives, the invalid witness fails full validation and the validator never emits a chunk endorsement.

---

### Finding Description

**Root cause — no signature check before orphan storage.**

`process_chunk_state_witness_message` branches on whether `prev_block` is known: [1](#0-0) 

When `prev_block` is absent the code calls `handle_orphan_witness` directly. That function performs only a height-distance check and a size check: [2](#0-1) 

No signature validation occurs. The comment in `add_orphan_state_witness` acknowledges the expectation but the caller never fulfils it: [3](#0-2) 

**Root cause — LRU `push` overwrites existing entries.**

`add_orphan_state_witness` keys on `witness.chunk_production_key()` — the public triple `(epoch_id, shard_id, height_created)` — and calls `self.witness_cache.push(cache_key, cache_entry)`. The LRU `push` semantics replace any existing value for the same key and return the displaced entry: [4](#0-3) 

**Deferred validation is too late.**

Signature validation only happens inside `validate_chunk_state_witness_impl`, which is spawned asynchronously after the witness is retrieved from the pool when the block arrives: [5](#0-4) 

By that point the legitimate witness has already been evicted and is unrecoverable.

**Attack sequence:**

1. Attacker observes the network and learns `ChunkProductionKey(epoch_id, shard_id, height_created)` for an in-flight chunk (all three fields are public).
2. Attacker constructs a `ChunkStateWitness` with that key, a `prev_block_hash` unknown to the target validator, and a garbage chunk-header signature.
3. Attacker sends the message to the target validator. `process_chunk_state_witness_message` finds `prev_block` absent and calls `handle_orphan_witness`.
4. `handle_orphan_witness` passes the height-distance and size checks and calls `add_orphan_state_witness`.
5. If the legitimate witness is already in the pool under the same key, `push` overwrites it with the attacker's invalid witness.
6. When the real `prev_block` arrives, `take_state_witnesses_waiting_for_block` returns the attacker's witness.
7. `process_chunk_state_witness` → `start_validating_chunk` → `validate_chunk_state_witness` fails on the bad signature.
8. The validator never calls `send_chunk_endorsement_to_block_producers`. [6](#0-5) 

---

### Impact Explanation

A validator that fails to endorse a chunk does not contribute to the endorsement quorum. NEAR requires approximately 2/3 of chunk validators to endorse before a chunk is included. An attacker who simultaneously targets enough validators (≥ 1/3 of the validator set for a shard) can prevent the chunk from accumulating sufficient endorsements, stalling chunk inclusion and degrading chain liveness. The attack is silent — the targeted validator logs a validation error but the chunk producer receives no endorsement and has no way to distinguish a targeted attack from a slow network.

**Severity: High**

---

### Likelihood Explanation

- **Unprivileged**: any network peer can send a `ChunkStateWitness` message; no validator key is required.
- **Public key material**: `ChunkProductionKey` is fully observable from the chain.
- **Timing window**: the attack window is the interval between the legitimate witness arriving as an orphan and the missing `prev_block` arriving — typically one block time (~1 s). The attacker can send the spoofed witness repeatedly to maximise the chance of landing inside the window.
- **No cryptographic barrier**: the attacker does not need to forge a valid signature; any bytes pass the orphan-pool admission checks.

**Likelihood: Medium** (requires precise timing and targeting multiple validators for full impact, but each individual overwrite is trivially cheap).

---

### Recommendation

1. **Validate the chunk-header signature before orphan storage.** In `handle_orphan_witness`, call `verify_chunk_header_signature_by_hash` (or the equivalent anchored variant) on `witness.chunk_header()` and reject witnesses whose signature does not match the expected chunk producer. This mirrors the validation already performed on the non-orphan path in `validate_chunk_header_full`. [7](#0-6) 

2. **Use insert-if-absent semantics.** Replace `self.witness_cache.push(cache_key, cache_entry)` with a conditional insert (e.g., `get_or_insert`) so that a later arrival cannot silently displace an already-stored legitimate witness. [4](#0-3)

    /// Add an orphaned chunk state witness to the pool. The witness will be put in a cache and it'll
    /// wait there for the block that's required to process it.
    /// It's expected that this `ChunkStateWitness` has gone through basic validation - including signature,
    /// shard_id, size, epoch_id and distance from the tip. The pool would still work without it, but without
    /// validation it'd be possible to fill the whole cache with spam.
    /// `witness_size` is only used for metrics, it's okay to pass 0 if you don't care about the metrics.
    pub fn add_orphan_state_witness(&mut self, witness: ChunkStateWitness, witness_size: usize) {
        // Insert the new ChunkStateWitness into the cache
        let cache_key = witness.chunk_production_key();
        let metrics_tracker = OrphanWitnessMetricsTracker::new(&witness, witness_size);
        let cache_entry = CacheEntry { witness, _metrics_tracker: metrics_tracker };
        if let Some((_, ejected_entry)) = self.witness_cache.push(cache_key, cache_entry) {
            // Another witness has been ejected from the cache due to capacity limit
            let header = &ejected_entry.witness.chunk_header();
            tracing::debug!(
                target: "client",
                ejected_witness_height = header.height_created(),
                ejected_witness_shard = ?header.shard_id(),
                ejected_witness_chunk = ?header.chunk_hash(),
                ejected_witness_prev_block = ?header.prev_block_hash(),
                "ejecting an orphaned chunk state witness from the cache due to capacity limit, it will not be processed"
            );
        }
    }

    fn validate_chunk_header_full(
        &self,
        header: &ShardChunkHeader,

        self.verify_chunk_protocol_version(header, epoch_id)?;
