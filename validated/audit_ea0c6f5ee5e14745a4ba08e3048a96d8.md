### Title
Missing Chain Identifier in `deployToken` Signature Enables Cross-Chain Replay, Allowing Unauthorized Token Deployment and Permanent Fund Lock - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `deployToken` function in `OmniBridge.sol` verifies a NEAR MPC signature over a borsh-encoded payload that contains **no chain identifier** (no `omniBridgeChainId`, no `block.chainid`, no contract address). Because the same `nearBridgeDerivedAddress` is used across all supported EVM deployments, a valid `deployToken` signature obtained from one chain (e.g., Ethereum) can be replayed verbatim on any other supported EVM chain (e.g., Arbitrum, Base, BNB, Polygon) to deploy the same bridge token there without authorization.

---

### Finding Description

In `OmniBridge.deployToken()`, the signature verification hash is computed as:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);

if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

The hash covers only the NEAR token ID, name, symbol, and decimals. **No chain ID, no contract address, and no nonce is included.** The `nearBridgeDerivedAddress` is a single NEAR MPC-derived key shared across all EVM deployments.

Compare this to `finTransfer`, which correctly binds its signature to the destination chain by hardcoding `omniBridgeChainId` into the borsh payload at two positions:

```solidity
bytes1(omniBridgeChainId),   // destination chain for token address
...
bytes1(omniBridgeChainId),   // destination chain for recipient
``` [2](#0-1) 

`deployToken` has no equivalent binding. The Omni Bridge is confirmed deployed on multiple EVM chains (Ethereum, Arbitrum, Base, BNB, Polygon, etc.) as evidenced by the `.openzeppelin` deployment manifests, all sharing the same `nearBridgeDerivedAddress`.

The `MetadataPayload` struct contains no chain field:

```solidity
struct MetadataPayload {
    string token;
    string name;
    string symbol;
    uint8 decimals;
}
``` [3](#0-2) [4](#0-3) 

Once a token is deployed via replay, the EVM bridge registers it fully:

```solidity
isBridgeToken[address(bridgeTokenProxy)] = true;
ethToNearToken[address(bridgeTokenProxy)] = metadata.token;
nearToEthToken[metadata.token] = address(bridgeTokenProxy);
``` [5](#0-4) 

This means users can subsequently call `initTransfer` on the replayed chain, locking real tokens in the bridge contract. However, the NEAR bridge's `factories` and `token_decimals` mappings will not have a registered entry for that chain's token address, causing `fin_transfer_callback` on NEAR to fail with `TokenDecimalsNotFound` or `UnknownFactory`, permanently locking the user's funds. [6](#0-5) 

---

### Impact Explanation

**High — Permanent freezing of user funds.**

An attacker replays a `deployToken` signature to register a bridge token on an unintended EVM chain. Unsuspecting users who call `initTransfer` on that chain lock their tokens in the EVM bridge. The NEAR bridge has no corresponding `token_decimals` or `factories` entry for that chain, so `fin_transfer_callback` panics and the transfer can never be finalized. The locked tokens are irrecoverable.

---

### Likelihood Explanation

**Medium.** The attack requires no special privileges. Any observer of a `deployToken` transaction on one chain can extract `signatureData` and `metadata` from calldata and replay them on any other supported EVM chain. The Omni Bridge is live on multiple EVM chains simultaneously, making the replay surface broad. The only friction is that the attacker must pay gas and the `PAUSED_DEPLOY_TOKEN` flag must be unset on the target chain.

---

### Recommendation

Include `omniBridgeChainId` (and ideally `address(this)`) in the borsh-encoded payload that is signed for `deployToken`, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // ADD: bind to destination chain
    Borsh.encodeAddress(address(this)), // ADD: bind to contract address
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

The NEAR MPC signing logic must be updated correspondingly to include these fields when producing the `log_metadata` / `deploy_token` signature.

---

### Proof of Concept

1. The NEAR bridge operator calls `deployToken(sig, {token: "usdc.near", name: "USD Coin", symbol: "USDC", decimals: 6})` on **Ethereum mainnet**. The transaction is public; anyone can read `signatureData` and `metadata` from calldata.

2. Attacker calls `deployToken(signatureData, metadata)` on **Arbitrum** (or Base, BNB, Polygon, etc.) with the identical arguments.

3. `ECDSA.recover(keccak256(borshEncoded), signatureData)` returns the same `nearBridgeDerivedAddress` because the hash is chain-agnostic. Signature check passes. [7](#0-6) 

4. A new `BridgeToken` proxy is deployed on Arbitrum and registered: `nearToEthToken["usdc.near"] = <proxy>`, `isBridgeToken[<proxy>] = true`. [5](#0-4) 

5. Victim calls `initTransfer(<proxy>, amount, ...)` on Arbitrum, locking real USDC in the bridge. [8](#0-7) 

6. Relayer submits proof to NEAR `fin_transfer`. `fin_transfer_callback` checks `self.token_decimals.get(&init_transfer.token)` — no entry exists for the Arbitrum token address — and panics with `TokenDecimalsNotFound`. The transfer is never finalized. [9](#0-8) 

7. Victim's funds are permanently locked in the Arbitrum bridge contract with no recovery path.
