### Title
Dust Transfer Permanently Locks User Funds Due to Missing Minimum-Amount Check Before Decimal Normalization - (File: `near/omni-bridge/src/lib.rs`)

### Summary

A user can initiate a bridge transfer with a "dust" amount that is below the decimal normalization threshold. The tokens are immediately locked or burned in `init_transfer_internal`, but the subsequent `sign_transfer` call always reverts with `InvalidAmountToTransfer` because `normalize_amount` floors the amount to zero. With no cancel or rescue mechanism, the user's funds are permanently frozen in the bridge.

### Finding Description

`init_transfer` accepts any `amount` satisfying `fee < amount`. For a NEAR-native token with 24 decimals bridging to an EVM chain where the token has 18 decimals, the normalization divisor is `10^6`. Any `amount` in the range `[1, 999_999]` (yoctoNEAR-denominated units) passes the fee check but normalizes to zero.

`init_transfer_internal` immediately burns or locks the tokens:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
```

Later, when a relayer calls `sign_transfer`, the normalization is applied:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

`normalize_amount` performs integer floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

For `amount = 1` and `diff_decimals = 6`, this returns `0`. The `require!` panics, reverting `sign_transfer`. The transfer record remains in `pending_transfers` indefinitely. There is no public `cancel_transfer` or admin rescue function in the contract.

### Impact Explanation

User funds are permanently frozen. The tokens are burned (for deployed/bridged tokens) or locked (for native tokens) in `init_transfer_internal` before any normalization check occurs. Because `sign_transfer` always reverts for dust transfers and no cancellation path exists, the funds are irrecoverable. This matches the allowed impact: **Permanent freezing, irrecoverable lock of user funds in bridge flows**.

### Likelihood Explanation

This is reachable by any unprivileged bridge user. The NEAR-to-EVM path (24 → 18 decimals, divisor = 10^6) is the primary production configuration. Any transfer of fewer than 10^6 yoctoNEAR-denominated token units triggers the issue. A user who mistakenly sends a small amount, or a griefing attacker who deliberately sends dust, will trigger the permanent lock. No special role or key is required.

### Recommendation

Add a minimum-amount check in `init_transfer` (before tokens are locked/burned) that ensures the normalized amount is greater than zero:

```rust
let token_address = self.get_token_address(
    init_transfer_msg.get_destination_chain(),
    token_id.clone(),
);
if let Some(addr) = token_address {
    if let Some(decimals) = self.token_decimals.get(&addr) {
        let normalized = Self::normalize_amount(
            amount.0.checked_sub(init_transfer_msg.fee.0).unwrap_or(0),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

Alternatively, enforce a protocol-level minimum transfer amount that accounts for the worst-case decimal normalization factor.

### Proof of Concept

1. A NEAR token with 24 decimals is registered in the bridge with an EVM counterpart having 18 decimals (`diff_decimals = 6`, divisor = `10^6`).
2. User calls `ft_transfer_call` on the token contract with `amount = 1` and `msg` encoding an `InitTransferMsg` with `fee = 0` and an EVM recipient.
3. `init_transfer` passes the fee check (`0 < 1`). `init_transfer_internal` burns/locks the 1-unit amount. The transfer is stored in `pending_transfers`.
4. A trusted relayer calls `sign_transfer` for this transfer ID.
5. `normalize_amount(1, Decimals{decimals:18, origin_decimals:24})` = `1 / 1_000_000` = `0`.
6. `require!(0 > 0, ...)` panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`. The transaction reverts; the transfer record and locked tokens remain.
7. Every subsequent `sign_transfer` call for this transfer ID produces the same panic. The user's token is permanently lost. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
