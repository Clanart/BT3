### Title
`MerkleBlob::batch_insert` Bypasses Duplicate-Key Validation, Enabling Tree-Root Corruption and Forged Inclusion Proofs — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces for every single insertion. When a batch of three or more entries contains a repeated `KeyId` (or repeated leaf `Hash`) among the items processed by the fast path, the duplicate is written directly into the blob without error. The resulting tree has two leaf nodes sharing the same key, producing a corrupted Merkle root and making inclusion proofs for that key ambiguous or forgeable.

---

### Finding Description

`MerkleBlob::insert` (the single-item path) explicitly rejects duplicates before touching the blob:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`MerkleBlob::batch_insert` handles the first two items (when the tree has ≤ 1 existing leaf) by delegating to `self.insert()`, which carries those guards. All remaining items — the bulk of any large batch — are written directly via `insert_entry_to_blob` with **no duplicate check**:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

Because `keys_values_hashes` is consumed in reverse order (the last two elements are `pop()`-ed into the guarded path), any duplicate `KeyId` or `Hash` that appears among the **first `n-2` elements** of the input vector bypasses all validation and is silently committed to the blob.

The Python binding `py_batch_insert` exposes this path directly to unprivileged callers:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 1503-1518
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
``` [3](#0-2)

### Impact Explanation

After a batch containing a duplicate key is committed, the `MerkleBlob` contains two leaf nodes with the same `KeyId`. The Merkle root is computed over both leaves, so it reflects a structurally invalid tree. Consequences:

1. **Corrupted tree root**: The root hash no longer represents a valid set; it encodes a multiset. Any downstream consumer that trusts the root (e.g., a DataLayer singleton coin) is operating on a false commitment.
2. **Ambiguous / forgeable inclusion proofs**: `get_proof_of_inclusion` looks up the key in `block_status_cache.key_to_index`. If the cache was updated by the second duplicate write, it points to one of the two leaves; the other leaf is unreachable through the cache but still present in the blob. An adversary who controls which leaf the cache points to can selectively prove inclusion of either value for the same key against the same root.
3. **`ProofOfInclusion::valid()` passes for the wrong value**: The proof verifier only checks that hashes chain up to the stored root; it does not detect that the root itself was built from a duplicate-key tree. [4](#0-3) 

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

The entry point is the Python-exposed `batch_insert` API, callable by any DataLayer client. The caller supplies both the keys and the hashes; no privilege is required. The bug is triggered simply by including a repeated `KeyId` (or repeated `Hash`) in a batch of three or more entries. The first two entries (popped from the tail) are protected; everything else is not. A malicious or buggy caller can exploit this deterministically.

---

### Recommendation

Add the same duplicate guards to the fast path inside `batch_insert` before calling `insert_entry_to_blob`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insertion logic
}
```

Alternatively, collect all keys and hashes into a `HashSet` before the loop and reject the entire batch if any duplicate is detected, consistent with the atomicity semantics implied by the function name.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def h(n: int) -> bytes:
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

blob = MerkleBlob(blob=bytearray())

# Build a batch of 4 entries where key 0 appears twice (positions 0 and 2).
# Positions 2 and 3 are popped into the guarded insert() path (different keys → OK).
# Position 0 (duplicate key=0) goes through the unguarded fast path.
keys_values = [
    (KeyId(0), ValueId(10)),   # duplicate key – fast path, no check
    (KeyId(1), ValueId(20)),   # unique
    (KeyId(2), ValueId(30)),   # popped into guarded insert()
    (KeyId(0), ValueId(99)),   # popped into guarded insert() – key=0 first occurrence
]
hashes = [h(10), h(20), h(30), h(99)]

# Expected: KeyAlreadyPresentError. Actual: succeeds silently.
blob.batch_insert(keys_values, hashes)
blob.calculate_lazy_hashes()

# Tree now has two leaves with KeyId(0); root is corrupted.
# get_proof_of_inclusion returns a proof for whichever leaf the cache points to,
# and proof.valid() returns True — against a root that encodes a duplicate-key tree.
proof = blob.get_proof_of_inclusion(KeyId(0))
assert proof.valid()   # passes, but root is invalid
```

The single-item path correctly raises `KeyAlreadyPresentError` for the same input:

```python
blob2 = MerkleBlob(blob=bytearray())
blob2.insert(KeyId(0), ValueId(99), h(99))
blob2.insert(KeyId(0), ValueId(10), h(10))  # raises KeyAlreadyPresentError ✓
