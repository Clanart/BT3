### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts an arbitrary class hash from the contract request without verifying it is a declared class. A contract that exposes an upgrade path with user-controlled input can be tricked into replacing its class with an undeclared or zero class hash, permanently making the contract uncallable and freezing all funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads the new class hash directly from the syscall request and writes it into `contract_state_changes` with no Cairo-level assertion that the hash corresponds to a declared class: [1](#0-0) 

The comment at line 898 explicitly acknowledges the missing guard:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
```

The syscall dispatcher in `execute_syscalls.cairo` routes `REPLACE_CLASS_SELECTOR` directly to this function with no pre-check: [2](#0-1) 

By contrast, `execute_declare_transaction` enforces `assert_not_zero(compiled_class_hash)` and uses `prev_value=0` to guarantee a class is declared at most once: [3](#0-2) 

No equivalent guard exists for `replace_class`. The revert log does record the old class hash for rollback on revert: [4](#0-3) 

But this only helps if the *transaction itself* reverts. If the transaction succeeds (which it will, since the OS does not reject the invalid hash), the corrupted class hash is committed permanently to state via `state_update`: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with `0` or any undeclared hash, every subsequent call to that contract will fail at entry-point resolution because the OS cannot find a matching compiled class. Any ERC-20 balances, NFTs, or other assets stored in that contract's storage become permanently inaccessible. There is no recovery path: the contract cannot call `replace_class` again because it is already uncallable.

---

### Likelihood Explanation

The attack is reachable by any unprivileged transaction sender against any upgradeable contract that passes the caller-supplied class hash directly to `replace_class_syscall` without validating it first — a common pattern in StarkNet upgradeable contracts. The attacker does not need any privileged role; they only need to call a public upgrade entry point with a crafted argument. The OS is the last line of defense and currently provides none.

---

### Recommendation

Inside `execute_replace_class`, before writing the new state entry, assert that `class_hash` is present in `contract_class_changes` (i.e., was declared in the current block) **or** in the existing committed class trie. At minimum, add `assert_not_zero(class_hash)` as an immediate guard, and implement the full declared-class lookup that the TODO comment defers.

---

### Proof of Concept

1. Deploy a vault contract `V` holding user funds, with an `upgrade(new_class_hash)` entry point that calls `replace_class_syscall(new_class_hash)` without validation.
2. An attacker sends an `invoke` transaction calling `V.upgrade(0)`.
3. The OS executes `execute_replace_class`: `class_hash = 0` is written into `contract_state_changes` for `V` with no rejection.
4. The transaction succeeds; `state_update` commits `V`'s class hash as `0`.
5. Every subsequent call to `V` fails at entry-point lookup — class `0` has no compiled class.
6. All funds in `V`'s storage are permanently frozen.
