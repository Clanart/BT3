### Title
Missing Length Consistency Check Between `pks` and `msgs` in `BLSCache.aggregate_verify` Python Binding Silently Truncates Verification Pairs - (File: crates/chia-bls/src/bls_cache.rs)

### Summary
The Python binding `py_aggregate_verify` on `BlsCache` accepts two independent lists — `pks` (`list[G1Element]`) and `msgs` (`list[bytes]`) — and zips them together without checking that both lists have the same length. Rust's `Iterator::zip` silently truncates to the shorter list. If `len(pks) < len(msgs)`, the trailing messages are silently dropped and the aggregate signature is verified against only `len(pks)` pairs. If `len(msgs) < len(pks)`, the trailing public keys are silently dropped. Either case produces a verification result that does not correspond to the full set of (pk, msg) pairs the caller intended to check, creating a latent aggregate-signature bypass at the Python/Rust binding boundary.

### Finding Description
In `crates/chia-bls/src/bls_cache.rs`, the `py_aggregate_verify` method (the Python-exposed `BLSCache.aggregate_verify`) collects `pks` and `msgs` into two independent `Vec`s and then calls:

```rust
Ok(self.aggregate_verify(pks.into_iter().zip(msgs), sig))
``` [1](#0-0) 

There is no guard of the form `if pks.len() != msgs.len() { return Err(...) }` before the `zip`. Rust's `zip` iterator adapter stops as soon as either iterator is exhausted, silently discarding the tail of the longer one.

The same defect is present in `py_evict`:

```rust
self.evict(pks.into_iter().zip(msgs));
``` [2](#0-1) 

The Rust-internal `BlsCache::aggregate_verify` takes a single `impl IntoIterator<Item = (Pk, Msg)>` and is not affected — the bug is exclusively in the Python binding layer that splits the paired iterator into two separate lists. [3](#0-2) 

The public type stub documents the split-list API as the intended interface:

```python
def aggregate_verify(self, pks: list[G1Element], msgs: list[bytes], sig: G2Element) -> bool: ...
``` [4](#0-3) 

The Rust-side consensus path (`validate_signature`) correctly uses a single paired iterator and is not affected: [5](#0-4) 

### Impact Explanation
BLS aggregate verification checks `e(agg_sig, G2) == ∏ e(pk_i, H(aug_msg_i))`. If the Python caller passes `n` public keys but only `m < n` messages (or vice versa), `zip` silently verifies only `m` pairs. An aggregate signature that is valid for only `m` of the required `n` (pk, msg) pairs will pass `aggregate_verify` returning `True`, even though `n − m` required authorizations were never checked. In the context of the Chia full node, which uses `BLSCache.aggregate_verify` to pre-validate spend bundles before block inclusion, this could allow a spend bundle whose aggregate signature does not cover all required AGG_SIG conditions to be accepted as valid, enabling unauthorized coin spends.

### Likelihood Explanation
The Python full node is the primary consumer of this binding. Any code path that constructs `pks` and `msgs` from different sources, or that appends to one list but not the other due to a conditional branch, will silently produce an incorrect verification result with no error or warning. The binding provides no defensive signal (no exception, no assertion, no length mismatch error) to alert the caller. The `py_evict` companion method has the same defect, meaning stale cache entries may persist and be replayed in subsequent verifications. The likelihood of a latent triggering bug in the full node is non-trivial given the split-list API design.

### Recommendation
Add an explicit length check at the top of both `py_aggregate_verify` and `py_evict` before the `zip`:

```rust
if pks.len() != msgs.len() {
    return Err(PyValueError::new_err(
        "pks and msgs must have the same length",
    ));
}
```

Alternatively, redesign the Python API to accept a single list of `(G1Element, bytes)` tuples, eliminating the possibility of a length mismatch entirely. This mirrors the Rust-internal API which already uses a single paired iterator.

### Proof of Concept
```python
from chia_rs import BLSCache, AugSchemeMPL, G2Element

sk = AugSchemeMPL.key_gen(b"a" * 32)
pk = sk.get_g1()
msg1 = b"hello"
msg2 = b"world"

# Sign only msg1
sig = AugSchemeMPL.sign(sk, msg1)

cache = BLSCache()

# Pass 1 pk but 2 msgs: zip truncates to 1 pair (pk, msg1)
# The second message is silently dropped; verification passes
# even though msg2 was never signed.
result = cache.aggregate_verify([pk], [msg1, msg2], sig)
assert result is True  # passes — msg2 was never verified

# Pass 2 pks but 1 msg: zip truncates to 1 pair (pk, msg1)
# The second public key is silently dropped.
sk2 = AugSchemeMPL.key_gen(b"b" * 32)
pk2 = sk2.get_g1()
result2 = cache.aggregate_verify([pk, pk2], [msg1], sig)
assert result2 is True  # passes — pk2 was never verified

    pub fn aggregate_verify<Pk: Borrow<PublicKey>, Msg: AsRef<[u8]>>(
        &self,
        pks_msgs: impl IntoIterator<Item = (Pk, Msg)>,
        sig: &Signature,
    ) -> bool {
        let iter = pks_msgs.into_iter().map(|(pk, msg)| -> GTElement {
            // Hash pubkey + message
            let mut hasher = Sha256::new();
            let mut aug_msg = pk.borrow().to_bytes().to_vec();
            aug_msg.extend_from_slice(msg.as_ref());
            hasher.update(&aug_msg);
            let hash: [u8; 32] = hasher.finalize();

            // If the pairing is in the cache, we don't need to recalculate it.
            if let Some(pairing) = self.cache.lock().expect("cache").items.get(&hash).cloned() {
                return pairing;
            }

            // Otherwise, we need to calculate the pairing and add it to the cache.
            let aug_hash = hash_to_g2(&aug_msg);

            let pairing = aug_hash.pair(pk.borrow());
            self.cache.lock().expect("cache").put(hash, pairing.clone());
            pairing
        });

        aggregate_verify_gt(sig, iter)
    }
