### Title
`check_dynamic_resharding` Uses Full-Trie `memory_usage` (Including Non-Account-Keyed Columns) as Split Threshold While `find_trie_split` Measures Only Account-Keyed Columns, Enabling Premature Irreversible Shard Splits - (`chain/chain/src/runtime/mod.rs`)

---

### Summary

`check_dynamic_resharding` gates the shard-split proposal on `total_mem_usage(shard_trie)`, which returns the trie root's cumulative `memory_usage` field and therefore includes every trie column — account records, contract code, access keys, contract data, **and** delayed receipts, buffered receipts, promise yields, and other shard-global columns. The actual split algorithm `find_trie_split` descends only the four account-keyed subtrees (`ACCOUNT`, `CONTRACT_CODE`, `ACCESS_KEY`, `CONTRACT_DATA`). The returned `TrieSplit.left_memory` and `TrieSplit.right_memory` therefore reflect only account-keyed state. Any accumulation of non-account-keyed state (e.g., a large delayed-receipt queue) inflates `total_mem_usage` past `memory_usage_threshold` and triggers a split proposal even when account-keyed state alone would not cross the threshold. Because the split is embedded in `BlockHeaderInnerRestV6.shard_split` and then committed to `EpochInfoV5.shard_layout`, it is **irreversible**.

---

The documentation explicitly acknowledges the excluded columns:

> "The remaining columns (delayed receipts, buffered receipts, promise yields, etc.) store shard-global state not keyed by account ID, so they are not considered by the split search."

**Consequence:** `total_mem_usage` ≫ `TrieDescent::total_memory()` whenever non-account-keyed state is large. The threshold check fires on the inflated value; the split is computed on the smaller account-keyed value.

**Concrete scenario:**

| Quantity | Value |
|---|---|
| Account-keyed state | 80 GB |
| Delayed-receipt queue | 30 GB |
| `total_mem_usage` | 110 GB |
| `memory_usage_threshold` | 100 GB |
| `find_trie_split` left/right | 40 GB / 40 GB |
| `min_child_memory_usage` | 10 GB |

Result: split is proposed and committed. Account-keyed state alone (80 GB) would not have crossed the 100 GB threshold.

After the split, non-account-keyed state (delayed receipts, buffered receipts) is **duplicated** to both children during `split_shard_task_blocking_impl` (flat-storage copy) and `process_memtrie_resharding_storage_update` (memtrie retain). Each child inherits the full delayed-receipt queue, so total cluster memory grows from 110 GB to 160 GB (80 GB account-keyed split + 30 GB × 2 duplicated).

---

### Impact Explanation

1. **Irreversible premature shard split.** The split is committed to `EpochInfoV5.shard_layout` via `finalize_epoch → next_next_shard_layout → ShardLayoutV3::derive`. There is no merge operation; the split cannot be undone.

2. **Broken split-decision invariant.** The invariant "split when account-keyed state exceeds threshold" is violated. The split boundary computed by `find_trie_split` is based on account-keyed memory that is below the threshold, so the resulting children may be significantly smaller than intended, wasting validator slots and network resources.

3. **Memory amplification after split.** Non-account-keyed state is duplicated to both children. If the delayed-receipt queue is large, total cluster memory increases after the split, potentially pushing children toward the threshold again and causing cascading splits.

4. **Incorrect `pick_shard_to_split` selection.** `pick_shard_to_split` uses `TrieSplit::total_memory()` (account-keyed only) to rank shards. A shard whose split was triggered by delayed receipts may rank lower than a shard with genuinely large account-keyed state, causing the wrong shard to be selected when multiple proposals exist.

**Severity: High.** The split decision is irreversible, the corrupted value (`mem_usage` at line 1734) is directly derived from attacker/congestion-controlled trie state, and the downstream effect (permanent shard layout change) cannot be corrected without a protocol upgrade.

---

### Likelihood Explanation

Delayed receipts accumulate naturally during congestion events — no adversarial intent is required. Any shard that processes cross-shard function calls under load will build a delayed-receipt queue. The `memory_usage_threshold` for dynamic resharding is expected to be set near the current mainnet shard size target (~100 GB). A delayed-receipt queue of tens of GB is reachable during sustained congestion. An adversary with sufficient NEAR tokens can also deliberately flood a shard with cross-shard calls to inflate the queue.

---

### Recommendation

Replace the threshold check with a measurement that matches what `find_trie_split` actually measures. Two options:

**Option A (preferred):** Run `find_trie_split` first and use `left_memory + right_memory` (account-keyed total) as the threshold input:

```rust
let trie_split = find_trie_split(shard_trie)?;
let account_keyed_mem = trie_split.left_memory + trie_split.right_memory;
if account_keyed_mem < config.memory_usage_threshold { return Ok(None); }
if trie_split.left_memory < config.min_child_memory_usage { return Ok(None); }
if trie_split.right_memory < config.min_child_memory_usage { return Ok(None); }
Ok(Some(trie_split))
```

**Option B:** Expose a `total_account_keyed_mem_usage` function from `TrieDescent` that sums only the four account-keyed subtrees' root memory, and use that for the threshold check while keeping `total_mem_usage` for the Prometheus metric only.

The Prometheus metric `near_dynamic_resharding_shard_memory_usage` should document that it reflects total trie memory (not account-keyed only) to avoid operator confusion.

```rust
pub fn find_trie_split(trie: &Trie) -> FindSplitResult<TrieSplit> {
    match trie.lock_memtries() {
        Some(memtries) => {
            let trie_storage = MemTrieIteratorInner::new(&memtries, trie);
            TrieDescent::new(trie_storage)?.find_mem_usage_split()
        }
        None => {

            TrieDescent::new(trie_storage)?.find_mem_usage_split()
        }
    }
}
```

```rust
/// Used as a quick check before running the more expensive `find_trie_split`.
pub fn total_mem_usage(trie: &Trie) -> FindSplitResult<u64> {

            Ok(root_node.memory_usage)
        }
    }
```

```rust
fn check_dynamic_resharding(
    shard_trie: &Trie,
    shard_id: ShardId,
    shard_layout: ShardLayout,
    config: &DynamicReshardingConfig,
) -> Result<Option<TrieSplit>, FindSplitError> {
    let shard_uid = ShardUId::from_shard_id_and_layout(shard_id, &shard_layout);
    let mem_usage = total_mem_usage(shard_trie)?;

    DYNAMIC_RESHARDING_SHARD_MEMORY_USAGE
        .with_label_values(&[&shard_uid.to_string()])
        .set(mem_usage as i64);
    DYNAMIC_RESHARDING_MEMORY_USAGE_THRESHOLD.set(config.memory_usage_threshold as i64);
    DYNAMIC_RESHARDING_MIN_CHILD_MEMORY_USAGE.set(config.min_child_memory_usage as i64);
    DYNAMIC_RESHARDING_MAX_NUMBER_OF_SHARDS.set(config.max_number_of_shards as i64);

    if shard_layout.num_shards() >= config.max_number_of_shards {
        return Ok(None);
    }
    // maximum number of shards takes precedence over force-split – DO NOT REORDER
    if config.force_split_shards.contains(&shard_id) {
        return Ok(Some(find_trie_split(shard_trie)?));
    }
    if config.block_split_shards.contains(&shard_id) {
        return Ok(None);
    }

    if mem_usage < config.memory_usage_threshold {
        return Ok(None);
    }
    let trie_split = find_trie_split(shard_trie)?;
    if trie_split.left_memory < config.min_child_memory_usage {
        return Ok(None);
    }
    if trie_split.right_memory < config.min_child_memory_usage {
        return Ok(None);
    }
    Ok(Some(trie_split))
}
```

```rust
    /// Performs the bulk of [split_shard_task]. This method splits the flat storage of the parent shard.
    ///
    /// Returns `true` if the routine completed successfully.
    fn split_shard_task_blocking_impl(
        &self,
        parent_shard: ShardUId,
        split_params: &ParentSplitParameters,
        metrics: &FlatStorageReshardingShardSplitMetrics,
    ) -> FlatStorageReshardingTaskResult {
        // Exit early if the task has already been cancelled.
        if self.handle.0.is_cancelled() {
            return FlatStorageReshardingTaskResult::Cancelled;
        }

        // Determines after how many bytes worth of key-values the process stops to commit changes
        // and to check cancellation.
        let batch_size = self.resharding_config.get().batch_size.as_u64() as usize;
        metrics.set_split_shard_batch_size(batch_size);
        // Delay between every batch.
        let batch_delay = self.resharding_config.get().batch_delay.unsigned_abs();

        tracing::info!(target: "resharding", ?parent_shard, ?split_params, ?batch_delay, ?batch_size, "flat storage shard split task: starting key-values copy");

        // Prepare the store object for commits and the iterator over parent's flat storage.
        let resharding_block = split_params.resharding_blocks.iter().exactly_one().unwrap();
        let flat_store = self.runtime.store().flat_store();
        let mut iter = match self.flat_storage_iterator(
            &flat_store,
            &parent_shard,
            &resharding_block.hash,
        ) {
            Ok(iter) => iter,
            Err(err) => {
                tracing::error!(target: "resharding", ?parent_shard, block_hash=?resharding_block.hash, ?err, "failed to build flat storage iterator");
                return FlatStorageReshardingTaskResult::Failed;
            }
        };

        let mut num_batches_done: usize = 0;
        metrics.set_split_shard_processed_bytes(0);
        let mut iter_exhausted = false;

        loop {
            let _span = tracing::debug_span!(
                target: "resharding",
                "split_shard_task_impl/batch",
                batch_id = ?num_batches_done)
            .entered();
            let mut store_update = flat_store.store_update();
            let mut processed_size = 0;

            // Process a `batch_size` worth of key value pairs.
            while processed_size < batch_size && !iter_exhausted {
                match iter.next() {
                    // Stop iterating and commit the batch.
                    Some(FlatStorageAndDeltaIterItem::CommitPoint) => break,
                    Some(FlatStorageAndDeltaIterItem::Entry((key, value))) => {
                        processed_size += key.len() + value.as_ref().map_or(0, |v| v.size());
                        if let Err(err) = shard_split_handle_key_value(
                            key,
                            value,
                            &mut store_update,
                            &split_params,
                        ) {
                            tracing::error!(target: "resharding", ?err, "failed to handle flat storage key");
                            return FlatStorageReshardingTaskResult::Failed;
                        }
                    }
                    None => {
                        iter_exhausted = true;
                    }
                }
            }

            // Make a pause to commit and check if the routine should stop.
            store_update.commit();

            num_batches_done += 1;
            metrics.set_split_shard_processed_batches(num_batches_done);
            metrics.inc_split_shard_processed_bytes_by(processed_size);

            // If `iter`` is exhausted we can exit after the store commit.
            if iter_exhausted {
                return FlatStorageReshardingTaskResult::Successful { num_batches_done };
            }
            if self.handle.0.is_cancelled() {
                return FlatStorageReshardingTaskResult::Cancelled;
            }

            // Sleep between batches in order to throttle resharding and leave some resource for the
            // regular node operation.
            std::thread::sleep(batch_delay);
        }
    }
```
