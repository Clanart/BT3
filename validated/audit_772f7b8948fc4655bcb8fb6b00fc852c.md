### Title
Missing Upper-Bound Validation on `proof_facts_size` in Invoke Transaction Processing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The OS program enforces `SIERRA_ARRAY_LEN_BOUND - 1` upper-bound checks on `calldata_size`, `signature_len`, and `constructor_calldata_size`, but applies **no equivalent upper-bound check** on `proof_facts_size` in `execute_invoke_function_transaction`. A user-submitted invoke transaction carrying an oversized `proof_facts` array is accepted by the OS without size validation, causing the Poseidon hash over that array to consume an unbounded number of builtin instances and making proof generation infeasible, which halts the network.

---

### Finding Description

The OS program consistently bounds array lengths before use. Three examples:

- **`calldata_size`** — bounded at `get_invoke_tx_execution_context`, line 485:
  ```cairo
  assert_nn_le(tx_execution_context.calldata_size, SIERRA_ARRAY_LEN_BOUND - 1);
  ``` [1](#0-0) 

- **`signature_len`** — bounded at `fill_account_tx_info`, line 218:
  ```cairo
  assert_nn_le(signature_len, SIERRA_ARRAY_LEN_BOUND - 1);
  ``` [2](#0-1) 

- **`constructor_calldata_size`** — bounded at `prepare_constructor_execution_context`, line 534:
  ```cairo
  assert_nn_le(constructor_calldata_size, SIERRA_ARRAY_LEN_BOUND - 1);
  ``` [3](#0-2) 

`SIERRA_ARRAY_LEN_BOUND` is `2^32`: [4](#0-3) 

In contrast, `proof_facts_size` is loaded from a hint with **no upper-bound assertion**:

```cairo
local proof_facts_size;
local proof_facts: felt*;
%{ TxProofFacts %}
``` [5](#0-4) 

`proof_facts_size` is then passed directly into `compute_invoke_transaction_hash`, where it drives an unbounded Poseidon hash:

```cairo
if (proof_facts_size != 0) {
    poseidon_hash_update_with_nested_hash(
        data_ptr=proof_facts, data_length=proof_facts_size
    );
}
``` [6](#0-5) 

It is also passed into `fill_account_tx_info`, where the end-pointer is computed as `&proof_facts[proof_facts_size]` — felt-space pointer arithmetic that wraps silently for values above the field prime: [7](#0-6) 

And into `check_proof_facts`, which only validates the fixed-size header and does not impose an upper bound on the total size: [8](#0-7) 

Note: `account_deployment_data_size` is separately asserted to `0` inside the hash functions (`assert account_deployment_data_size = 0`), so it is effectively locked and not a live vulnerability path. `proof_facts_size` has no such lock. [9](#0-8) 

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

`poseidon_hash_update_with_nested_hash` iterates over every element of `proof_facts`. Each element consumes one Poseidon builtin instance. A `proof_facts_size` of, say, `2^20` would require ~1 million Poseidon instances for a single transaction — far exceeding any block's builtin budget. The prover cannot generate a valid STARK proof for such a block. Because the OS program does not reject the transaction, the sequencer has no protocol-level signal to exclude it, and the block cannot be proven, halting finality.

---

### Likelihood Explanation

`proof_facts` is a user-supplied field in invoke transactions (v3), used for client-side proving. Any unprivileged user can craft and sign an invoke transaction with an arbitrarily large `proof_facts` array. The user must compute the Poseidon hash of that array to produce the correct transaction hash for signing, which is feasible for moderate sizes (e.g., `2^16`–`2^20` elements). The sequencer has no OS-enforced size limit to rely on, so it may include the transaction. Once included, proof generation fails.

---

### Recommendation

Add an upper-bound assertion on `proof_facts_size` immediately after it is loaded from the hint in `execute_invoke_function_transaction`, mirroring the pattern used for all other array fields:

```cairo
local proof_facts_size;
local proof_facts: felt*;
%{ TxProofFacts %}
assert_nn_le(proof_facts_size, SIERRA_ARRAY_LEN_BOUND - 1);  // ADD THIS
``` [5](#0-4) 

This mirrors the existing pattern for `calldata_size`, `signature_len`, and `constructor_calldata_size`. [4](#0-3) 

---

### Proof of Concept

1. Craft an invoke v3 transaction with `proof_facts` set to an array of `N = 2^20` arbitrary felt values.
2. Compute `poseidon_hash_many(N, proof_facts)` locally to obtain the nested hash, then compute the full transaction hash and sign it.
3. Submit the transaction to the sequencer. The OS program has no assertion rejecting `proof_facts_size = 2^20`.
4. The sequencer includes the transaction in a block.
5. The prover invokes the OS program. `compute_invoke_transaction_hash` calls `poseidon_hash_update_with_nested_hash` with `data_length = 2^20`, consuming ~1 million Poseidon builtin instances for this single transaction.
6. The block's Poseidon builtin budget is exhausted; proof generation fails.
7. The block cannot be finalized; the network halts.
