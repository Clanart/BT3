### Title
Missing Token Registration Check in `initTransfer` Allows Permanent Locking of User Funds â€” (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer()` accepts and locks any arbitrary ERC20 token without verifying that the token has a valid NEAR-side mapping (`ethToNearToken`). If a user calls `initTransfer` with an unregistered token, or with a token whose registration was removed via `removeCustomToken`, the tokens are permanently locked in the bridge with no recovery path.

---

### Finding Description

`OmniBridge.initTransfer()` processes the token address through a cascade of checks: [1](#0-0) 

```
if (customMinters[tokenAddress] != address(0)) { ... burn via custom minter ... }
else if (isBridgeToken[tokenAddress])           { ... burn bridge token ... }
else {
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
}
```

The `else` branch executes for any ERC20 that is neither a custom-minter token nor a bridge token. There is **no check** that `ethToNearToken[tokenAddress]` is non-empty â€” i.e., that the token has a registered NEAR counterpart. The function proceeds to lock the tokens in the bridge contract and emits `InitTransfer`, but no relayer can ever finalize this transfer on NEAR because no token mapping exists. [2](#0-1) 

The `removeCustomToken` function makes this directly exploitable as a race condition: [3](#0-2) 

After `removeCustomToken(tokenAddress)` executes:
- `isBridgeToken[tokenAddress]` â†’ `false`
- `customMinters[tokenAddress]` â†’ `address(0)`
- `ethToNearToken[tokenAddress]` â†’ `""` (deleted)

A user who calls `initTransfer` with this token (e.g., a pending transaction, or unaware of the removal) hits the unguarded `else` branch. Their tokens are transferred into the bridge contract and permanently stranded â€” there is no cancel, refund, or rescue function anywhere in `OmniBridge.sol`.

There is also no check for the case where a token was never registered at all: any user can call `initTransfer` with a completely arbitrary ERC20 address and lock tokens irreversibly.

---

### Impact Explanation

**Critical â€” Permanent, irrecoverable lock of user funds in the bridge.**

Tokens locked via the unguarded `else` branch of `initTransfer` can never be released:
- `finTransfer` on EVM requires a valid MPC-signed proof originating from NEAR. NEAR cannot produce such a proof for a token with no mapping.
- There is no `cancelTransfer`, `rescueTokens`, or admin withdrawal function in `OmniBridge.sol`.
- The `currentOriginNonce` is incremented and the `InitTransfer` event is emitted, but the transfer is permanently unprocessable. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** Two realistic paths exist:

1. **`removeCustomToken` race condition**: An admin removes a custom token. A user with a pending `initTransfer` transaction (or one submitted in the same block) hits the now-unguarded else branch. This is a realistic operational scenario.
2. **Unregistered token**: Any user can intentionally or accidentally call `initTransfer` with an ERC20 that was never registered. No UI-level protection is enforced at the contract level.

Neither path requires privileged access, leaked keys, or colluding operators.

---

### Recommendation

Add a validation check at the top of `initTransfer` (and `initTransfer1155`) that requires the token to have a valid NEAR mapping before accepting funds:

```solidity
function initTransfer(
    address tokenAddress,
    ...
) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    if (tokenAddress != address(0)) {
        require(
            bytes(ethToNearToken[tokenAddress]).length > 0,
            "ERR_TOKEN_NOT_REGISTERED"
        );
    }
    ...
}
```

This mirrors the recommendation in the external report: add a state-validity check before accepting user funds, so that tokens cannot be locked into an unprocessable state. [5](#0-4) 

---

### Proof of Concept

1. Admin calls `addCustomToken(nearTokenId, tokenAddress, customMinter, decimals)` â€” token is registered.
2. Admin calls `removeCustomToken(tokenAddress)` â€” all mappings deleted; `isBridgeToken[tokenAddress] = false`, `ethToNearToken[tokenAddress] = ""`.
3. User (unaware of removal, or with a pending tx) calls:
   ```solidity
   OmniBridge.initTransfer(tokenAddress, 1000e18, 0, 0, "alice.near", "")
   ```
4. Contract enters the `else` branch at line 407:
   ```solidity
   IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), 1000e18);
   ```
   Tokens are now held by `OmniBridge`.
5. `InitTransfer` event is emitted with `currentOriginNonce` incremented.
6. No relayer can construct a valid `fin_transfer` call on NEAR â€” `ethToNearToken[tokenAddress]` is empty, so no NEAR token ID exists.
7. No `finTransfer` on EVM can release the tokens â€” that requires a NEAR-originated MPC signature.
8. Tokens are permanently locked. [6](#0-5) [3](#0-2)

        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );

                );
