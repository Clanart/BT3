Missing Duplicate-Key Validation in `batch_insert` Silently Corrupts DataLayer Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. When a caller supplies a key that already exists in the tree (or a key that appears twice within the same batch), `BlockStatusCache::add_leaf` silently overwrites the `key_to_index` and `leaf_hash_to_index` HashMap entries. The old leaf node remains physically present in the blob and is still referenced by its parent internal node, while the cache now points to the new leaf. The result is a split state: the blob encodes one tree (with both leaves), the cache encodes another (only the new leaf). Every subsequent root-hash computation, proof-of-inclusion, and proof-of-exclusion operates on this corrupted state, producing hashes and proofs that do not correspond to the actual committed data.

---

### Finding Description

`MerkleBlob::insert` (lines 362–413) guards against duplicates at the very top:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::batch_insert` (lines 570–657) applies those same guards only for the first two items and only when the tree currently holds ≤ 1 leaf:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { … };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;   // ← guarded
    }
}

for ((key, value), hash) in keys_values_hashes {          // ← ALL remaining items
    let new_leaf_index = self.get_new_index();
    …
    self.insert_entry_to_blob(new_leaf_index, &new_block)?; // ← NO duplicate check
    …
}
```

When the tree already contains ≥ 2 leaves (the common production case), **every** item in the batch skips the guard entirely. `insert_entry_to_blob` calls `block_status_cache.add_leaf`, which unconditionally overwrites the HashMap entries:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);        // silent overwrite
    self.leaf_hash_to_index.insert(leaf.hash, index); // silent overwrite
}
```

The displaced old index is neither freed nor removed from the blob. The blob now contains two leaf nodes for the same key; the cache knows only the newer one. The tree's structural integrity is broken.

The Python binding `py_batch_insert` (lines 1503–1519) exposes this path directly with no additional validation:

```rust
self.batch_insert(zip(keys_values, hashes).collect())?;
```

---

### Impact Explanation

After a duplicate-key batch insert:

1. **Corrupted root hash.** `calculate_lazy_hashes` traverses the blob, which still contains the orphaned old leaf. The computed root hash reflects a tree with both leaves, but the cache-based API (proofs, key lookups) reflects only the new leaf. The root hash committed on-chain diverges from the state the cache represents.

2. **Forged inclusion proofs.** `get_proof_of_inclusion` resolves the key through `key_to_index` (now pointing to the new leaf) and builds a Merkle path against the corrupted root. A verifier checking the proof against the on-chain root will accept it even though the proof path does not correspond to the actual tree structure.

3. **Invalid exclusion proofs.** Because the old leaf is invisible to the cache but present in the blob, exclusion proofs for the old key's hash can be constructed against a root that actually contains that hash, allowing an attacker to prove absence of data that is present.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

`py_batch_insert` is a public Python binding on `MerkleBlob`, the primary DataLayer storage object. Any code path that calls `batch_insert` with caller-controlled key lists — including delta synchronization between DataLayer nodes — can trigger the bug. No privilege escalation is required; a DataLayer peer that sends a delta containing a key already present in the recipient's store is sufficient. The bug is also triggerable by a single local caller passing a list with a repeated key.

---

### Recommendation

Add the same duplicate guards to `batch_insert` that `insert` already enforces, before writing any leaf to the blob:

```rust
for ((key, value), hash) in &keys_values_hashes {
    if self.block_status_cache.contains_key(*key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(hash) {
        return Err(Error::HashAlreadyPresent());
    }
}
```

Alternatively, make `add_leaf` return an error (instead of silently overwriting) when the key or hash is already present, so that every call site is protected uniformly.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def h(n: int) -> bytes:
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

blob = MerkleBlob(bytearray())

# Pre-populate with 3 leaves so leaf_count > 1 (bypasses the guarded path)
blob.batch_insert(
    [(KeyId(1), ValueId(1)), (KeyId(2), ValueId(2)), (KeyId(3), ValueId(3))],
    [h(1), h(2), h(3)],
)
blob.calculate_lazy_hashes()
root_before = blob.get_root_hash()

# Now batch-insert KeyId(1) again (duplicate) alongside a new key
blob.batch_insert(
    [(KeyId(1), ValueId(99)), (KeyId(4), ValueId(4))],  # KeyId(1) is a duplicate
    [h(100), h(4)],
)
blob.calculate_lazy_hashes()
root_after = blob.get_root_hash()

# root_after is computed from a blob that contains FOUR leaf nodes (two for KeyId(1))
# but the cache only knows about three.  Proofs built against root_after are invalid.
proof = blob.get_proof_of_inclusion(KeyId(1))
# proof.node_hash == h(100) (new leaf), but root_after encodes h(1) (old leaf) too.
print("Root corrupted:", root_before != root_after)
print("Proof node hash:", proof.node_hash.hex())
```

The `check_integrity()` call on the resulting blob will raise an integrity error, confirming the tree is in an inconsistent state.
