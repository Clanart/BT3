### Title
Malicious or Non-ERC1155-Compliant Recipient in `finTransfer` Permanently Freezes Bridged ERC1155 Tokens — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol::finTransfer` uses `IERC1155.safeTransferFrom` to deliver ERC1155 tokens to the recipient, which triggers the `onERC1155Received` callback on the recipient contract. A recipient contract that reverts in this callback causes the entire `finTransfer` transaction to revert. Because there is no on-chain recovery path, the user's tokens on the source chain are permanently frozen.

---

### Finding Description

In `finTransfer`, the nonce is marked used **before** the external call:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287
```

Then, for ERC1155 tokens, the function calls:

```solidity
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,          // attacker-controlled contract
    multiToken.tokenId,
    payload.amount,
    ""
);
```

`safeTransferFrom` mandates that the recipient implement `onERC1155Received` and return the correct selector. If `payload.recipient` is a contract whose `onERC1155Received` reverts (intentionally or because it does not support ERC1155), the entire `finTransfer` call reverts, rolling back the nonce marking at line 287.

Because the nonce is rolled back, the transfer is not "consumed" — but every subsequent attempt to call `finTransfer` with the same signed payload will trigger the same reverting callback. The signed payload is fixed by the MPC signature; there is no on-chain mechanism to re-issue a payload with a different recipient or to unlock the source-chain tokens. The user's funds are therefore irrecoverably locked.

The same pattern applies to the native-ETH path:

```solidity
(bool success, ) = payload.recipient.call{value: payload.amount}("");
if (!success) revert FailedToSendEther();
```

A recipient contract whose `receive()` / `fallback()` reverts produces `success = false`, causing `finTransfer` to revert with `FailedToSendEther`, with identical consequences. [1](#0-0) 

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

Once a user initiates a transfer from NEAR (or another source chain) to an EVM recipient that reverts on token receipt, the source-chain tokens are locked in the bridge and can never be released. The NEAR bridge has no on-chain mechanism to cancel a signed transfer or redirect it to a different recipient. The `finalised_transfers` set on NEAR is only updated when `fin_transfer_callback` succeeds; if EVM-side finalization never succeeds, the NEAR-side state is never updated and the tokens remain locked indefinitely. [2](#0-1) [3](#0-2) 

---

### Likelihood Explanation

**Medium.**

1. **Accidental**: A user bridges ERC1155 tokens to a multisig wallet, a DeFi vault, or any contract that does not implement `IERC1155Receiver`. This is a common mistake; many contracts accept ERC20 but not ERC1155.
2. **Malicious**: An attacker deliberately specifies a contract whose `onERC1155Received` burns all gas and reverts. The attacker sacrifices their own bridged tokens to permanently grief the relayer (repeated gas loss) and permanently freeze their own funds — a low-cost denial-of-service against the relayer's fee budget, analogous to the AutoRange gas-grief pattern in the reference report.
3. **Front-run variant**: If the recipient address is a CREATE2 address not yet deployed, an attacker can deploy a reverting contract there before the relayer submits `finTransfer`, freezing a legitimate user's funds.

---

### Recommendation

1. **Pull pattern**: Instead of pushing tokens directly to `payload.recipient`, hold them in the bridge and emit an event. Let the recipient call a separate `claimTokens` function. This eliminates the callback entirely.
2. **Use `transferFrom` instead of `safeTransferFrom`**: `IERC1155.transferFrom` (non-safe) does not invoke `onERC1155Received`, removing the callback attack surface. Note this requires the ERC1155 token to expose a non-safe transfer path.
3. **On-chain recovery**: Add a time-locked refund path that allows the source-chain bridge to reclaim tokens if the destination-chain finalization has not succeeded within a deadline.

---

### Proof of Concept

1. Attacker deploys `MaliciousReceiver` on EVM:
   ```solidity
   function onERC1155Received(address, address, uint256, uint256, bytes calldata)
       external pure override returns (bytes4) {
       revert("grief");
   }
   ```
2. Attacker initiates an ERC1155 bridge transfer from NEAR, setting `recipient = address(MaliciousReceiver)`.
3. Source-chain tokens are locked; MPC signs the transfer payload.
4. Relayer calls `finTransfer(signatureData, payload)` on EVM.
5. `safeTransferFrom` triggers `onERC1155Received` → reverts.
6. Entire transaction reverts; `completedTransfers[nonce]` is rolled back.
7. Every subsequent `finTransfer` attempt with the same signed payload repeats steps 4–6.
8. No on-chain path exists to unlock the source-chain tokens. Funds are permanently frozen. [4](#0-3) [5](#0-4)
