### Title
Unchecked Felt Arithmetic in `compute_max_possible_fee` Enables Fee Bypass via Field Wrap-Around — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `compute_max_possible_fee` function computes the maximum chargeable fee by multiplying `max_amount * max_price_per_unit` for each resource type in raw felt arithmetic, without enforcing that these values are within their protocol-specified ranges (`u64` for `max_amount`, `u128` for `max_price_per_unit`). Because Cairo arithmetic is modular over the Stark prime `p ≈ 2^251`, an attacker can supply resource bound values that cause the sum of products to wrap around to `0 (mod p)`. When `max_fee == 0`, the OS unconditionally skips fee charging, allowing a transaction to execute without paying any fee.

---

### Finding Description

`compute_max_possible_fee` is defined as:

```cairo
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    ...
    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
``` [1](#0-0) 

The result is used immediately in `charge_fee`:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
...
assert_nn_le(calldata.amount.low, max_fee);
``` [2](#0-1) 

The StarkNet protocol specifies `max_amount` as a `u64` and `max_price_per_unit` as a `u128`. Their product is at most `2^64 × 2^128 = 2^192 < p`, so no wrap-around occurs for in-range values. However, **the OS never enforces these bit-width constraints on the felt values loaded from hints**. Examining `get_account_tx_common_fields` and `fill_account_tx_info`, the resource bounds are loaded from hints and written directly into the `TxInfo` struct with no `assert_nn_le` or range-check applied to `max_amount` or `max_price_per_unit`: [3](#0-2) [4](#0-3) 

Because the OS only checks `assert_nn_le(signature_len, SIERRA_ARRAY_LEN_BOUND - 1)` for the signature array but applies no analogous bound to resource bound fields, an attacker can supply felt values outside `[0, 2^64)` / `[0, 2^128)`. The multiplication then operates in `F_p`, and the sum of products can be made to equal `0 (mod p)`.

This is directly analogous to the MIPS `MUL` bug: in that case, a 32-bit multiplication was not widened to 64 bits before truncation, producing an incorrect low-order result. Here, a multiplication intended to operate on bounded integers is instead performed in an unbounded prime field, producing an incorrect (wrapped) result.

---

### Impact Explanation

When `max_fee` evaluates to `0` due to field wrap-around, `charge_fee` returns immediately without executing the ERC-20 transfer:

```cairo
if (max_fee == 0) {
    return ();
}
``` [5](#0-4) 

The transaction's `__execute__` entry point still runs and state changes are committed. The fee recipient (sequencer) receives nothing. This constitutes **direct loss of funds**: the fee token balance that should have been transferred to the sequencer is never deducted from the user's account, and the sequencer processes work for free. At scale, this makes sequencer operation economically unviable, mapping to both **Critical (direct loss of funds)** and **High (network unable to confirm new transactions)**.

---

### Likelihood Explanation

The attacker controls all three resource bound fields and `tip` in a V3 transaction. They sign the transaction hash over these values, so any felt values are valid inputs to the OS. Finding a combination such that:

```
max_amount_l1 * max_price_l1 + max_amount_l2 * (max_price_l2 + tip) + max_amount_l1d * max_price_l1d ≡ 0 (mod p)
```

is trivially achievable. For example:

- Set `max_amount_l1 = p − 1`, `max_price_l1 = 1` → product `≡ −1 (mod p)`
- Set `max_amount_l2 = 1`, `max_price_l2 = 1`, `tip = 0` → product `= 1`
- Set `max_amount_l1d = 0` → product `= 0`
- Sum: `(−1) + 1 + 0 = 0 (mod p)` ✓

The attacker simply signs this transaction and submits it. No privileged access is required — any unprivileged transaction sender can exploit this.

---

### Recommendation

Add explicit range checks on `max_amount` and `max_price_per_unit` for each resource bound before computing the fee. Specifically, in `fill_account_tx_info` or `compute_max_possible_fee`, assert that each `max_amount` fits in 64 bits and each `max_price_per_unit` fits in 128 bits using `assert_nn_le`:

```cairo
// For each resource bound rb:
assert_nn_le(rb.max_amount, MAX_U64);           // 2^64 - 1
assert_nn_le(rb.max_price_per_unit, MAX_U128);  // 2^128 - 1
```

This ensures the products cannot exceed `2^192 < p`, preventing field wrap-around. The same check should be applied to `tip`.

### Proof of Concept

1. Attacker constructs a V3 `invoke` transaction with:
   - `resource_bounds[L1_GAS_INDEX] = { max_amount: p−1, max_price_per_unit: 1 }`
   - `resource_bounds[L2_GAS_INDEX] = { max_amount: 1, max_price_per_unit: 1 }`, `tip = 0`
   - `resource_bounds[L1_DATA_GAS_INDEX] = { max_amount: 0, max_price_per_unit: 0 }`
2. Attacker signs the Poseidon hash of this transaction (which includes these fields).
3. OS calls `compute_max_possible_fee`:
   - `(p−1)*1 + 1*(1+0) + 0*0 = (p−1) + 1 = p ≡ 0 (mod p)`
   - Returns `0`.
4. `charge_fee` hits `if (max_fee == 0) { return (); }` and exits without transferring any fee tokens.
5. The transaction's `__execute__` runs normally; state changes are committed; the sequencer receives zero fee. [6](#0-5)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
