Looking at the `upsert` function and related cache methods in detail:

### Title
Missing Duplicate-Hash Guard in `MerkleBlob::upsert` Corrupts `leaf_hash_to_index` Cache and Blob State — (`crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::upsert` removes the old leaf's hash from `leaf_hash_to_index` **before** checking whether `new_hash` is already occupied by a different leaf. `insert` performs this guard; `upsert` does not. Passing a `new_hash` that belongs to an existing sibling leaf silently overwrites that sibling's cache entry and writes two leaves with the same hash into the blob, corrupting the Merkle tree root and causing `get_node_by_hash` to return the wrong key/value.

---

### Finding Description

**`insert` — has the guard** [1](#0-0) 

```rust
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

**`upsert` — missing the guard** [2](#0-1) 

```rust
pub fn upsert(&mut self, key: KeyId, value: ValueId, new_hash: &Hash) -> Result<(), Error> {
    let Ok((leaf_index, mut leaf, mut block)) = self.get_leaf_by_key(key) else {
        self.insert(key, value, new_hash, InsertLocation::Auto {})?;
        return Ok(());
    };

    self.block_status_cache.remove_leaf(&leaf)?;   // ← old hash evicted here
    leaf.hash.clone_from(new_hash);                // ← new_hash set, no collision check
    leaf.value = value;
    block.node = Node::Leaf(leaf);
    self.insert_entry_to_blob(leaf_index, &block)?; // ← add_leaf overwrites sibling's entry
    ...
}
```

`remove_leaf` removes the **old** hash from `leaf_hash_to_index` [3](#0-2)  but does not check whether `new_hash` is already mapped to a different index.

`insert_entry_to_blob` then calls `add_leaf`, which does an **unconditional** `HashMap::insert` for `leaf_hash_to_index`: [4](#0-3) 

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index); // silent overwrite
}
```

**Step-by-step corruption:**

| Step | Action | `leaf_hash_to_index` state |
|------|--------|---------------------------|
| 1 | `insert(K1, V1, H1)` | `{H1→idx1}` |
| 2 | `insert(K2, V2, H2)` | `{H1→idx1, H2→idx2}` |
| 3 | `upsert(K1, V1_new, H2)` — `remove_leaf(K1's old leaf)` | `{H2→idx2}` |
| 4 | `add_leaf(idx1, leaf{hash=H2})` | `{H2→idx1}` ← **K2's mapping destroyed** |

After step 4:
- The blob contains **two leaves both carrying hash H2** (K1 at `idx1`, K2 at `idx2`).
- `leaf_hash_to_index[H2]` points to `idx1` (K1), not `idx2` (K2).
- `get_node_by_hash(H2)` returns K1's `(key, value)` instead of K2's. [5](#0-4) 
- All ancestor hashes of K1 are marked dirty and will be recomputed from the corrupted leaf hash, producing a wrong Merkle root.

`check_integrity` would catch this (it verifies `leaf_count == leaf_hash_to_index.len()`), but `check_integrity_on_drop` is `false` in production builds: [6](#0-5) 

```rust
check_integrity_on_drop: cfg!(test),
```

---

### Impact Explanation

Fits: **High — DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state.**

- `get_node_by_hash(H2)` returns K1's data, enabling a caller to read forged DataLayer state.
- The Merkle root stored in the DataLayer singleton coin is computed from the corrupted blob, so any subsequent inclusion proof for K2 (using H2) will verify against the wrong root.
- K2's true mapping is permanently lost from the in-memory cache; it cannot be recovered without rebuilding from the raw blob.

---

### Likelihood Explanation

**Moderate-to-low** in practice. Exploitation requires:

1. Write access to the DataLayer store (the store owner or an authorized updater).
2. Knowledge of an existing leaf's hash H2.
3. The ability to pass H2 directly as `new_hash` to `upsert` — which is possible if the Python/wasm bindings expose `upsert` with a caller-supplied hash, or if the DataLayer update path does not re-derive the hash from the value before calling `upsert`.

If the hash is always re-derived from the value by the DataLayer layer above `MerkleBlob`, an accidental collision requires a SHA-256 preimage, which is infeasible. However, the bug is a missing guard that `insert` already has, so it is a straightforward code defect regardless of likelihood.

---

### Recommendation

Add the same guard that `insert` uses, **before** calling `remove_leaf`, so the old hash is not evicted if the operation will fail:

```rust
pub fn upsert(&mut self, key: KeyId, value: ValueId, new_hash: &Hash) -> Result<(), Error> {
    let Ok((leaf_index, mut leaf, mut block)) = self.get_leaf_by_key(key) else {
        self.insert(key, value, new_hash, InsertLocation::Auto {})?;
        return Ok(());
    };

    // Guard: reject if new_hash is already used by a *different* leaf
    if leaf.hash != *new_hash && self.block_status_cache.contains_leaf_hash(new_hash) {
        return Err(Error::HashAlreadyPresent());
    }

    self.block_status_cache.remove_leaf(&leaf)?;
    leaf.hash.clone_from(new_hash);
    leaf.value = value;
    block.node = Node::Leaf(leaf);
    self.insert_entry_to_blob(leaf_index, &block)?;
    ...
}
```

The `leaf.hash != *new_hash` short-circuit avoids a false positive when upserting with the same hash (a no-op value update).

### Proof of Concept

```rust
use chia_datalayer::{MerkleBlob, InsertLocation, Hash, KeyId, ValueId};

let mut blob = MerkleBlob::new(vec![]).unwrap();
let h1 = Hash([1u8; 32]);
let h2 = Hash([2u8; 32]);
let k1 = KeyId(1);
let k2 = KeyId(2);

blob.insert(k1, ValueId(10), &h1, InsertLocation::Auto {}).unwrap();
blob.insert(k2, ValueId(20), &h2, InsertLocation::Auto {}).unwrap();

// Upsert K1 with K2's existing hash — no error returned
blob.upsert(k1, ValueId(99), &h2).unwrap();

// get_node_by_hash(h2) now returns K1's data, not K2's
let (key, value) = blob.get_node_by_hash(h2).unwrap();
assert_eq!(key, k1);   // returns K1 — WRONG, should be K2
assert_eq!(value, ValueId(99));

// K2's mapping is silently destroyed
// blob.get_node_by_hash(h2) can never return K2 again
```
