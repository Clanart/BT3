### Title
Broken `assert_nn_le` Fee Cap Check Allows Proof Failure via Oversized `max_fee` — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` returns an unbounded felt that can exceed `2^128`. The subsequent `assert_nn_le(low_actual_fee, max_fee)` call is semantically broken for any `max_fee >= 2^128 + actual_fee`, because Cairo's `assert_nn_le` is implemented via range-check and requires both operands and their difference to fit in 128 bits. A transaction sender can craft valid resource bounds that produce such a `max_fee`, making it impossible for the sequencer to construct a valid proof for any block containing that transaction.

---

### Finding Description

`compute_max_possible_fee` in `transaction_impls.cairo` computes:

```
max_fee = l1.max_amount * l1.max_price_per_unit
        + l2.max_amount * (l2.max_price_per_unit + tip)
        + l1_data.max_amount * l1_data.max_price_per_unit
``` [1](#0-0) 

The only bounds enforced on the inputs (in `pack_resource_bounds` and `hash_fee_fields`) are:
- `max_amount <= 2^64 - 1`
- `max_price_per_unit < 2^128` (via `assert_nn`, which is a range-check)
- `tip <= 2^64 - 1` [2](#0-1) [3](#0-2) 

These bounds allow `max_fee` to reach up to approximately `3 × (2^64 − 1) × (2^128 − 1) ≈ 3 × 2^192`, far exceeding `2^128`.

`charge_fee` then does:

```cairo
assert_nn_le(calldata.amount.low, max_fee);
``` [4](#0-3) 

Cairo's `assert_nn_le(a, b)` expands to:
1. `assert_nn(a)` → `a ∈ [0, 2^128)`
2. `assert_nn(b − a)` → `b − a ∈ [0, 2^128)`

Condition 2 means the check only passes when `b − a < 2^128`, i.e., `low_actual_fee > max_fee − 2^128`.

When `max_fee >= 2^128 + actual_fee + 1` (which is trivially achievable), there is **no valid `low_actual_fee`** satisfying both conditions simultaneously:
- Condition 1 requires `low_actual_fee < 2^128`
- Condition 2 requires `low_actual_fee > max_fee − 2^128 ≥ actual_fee + 1`

If `max_fee >= 2^129`, then `max_fee − 2^128 >= 2^128`, so **no value in `[0, 2^128)` satisfies condition 2**. The proof is unconditionally invalid for any block containing such a transaction.

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

If the sequencer's off-chain simulation layer (e.g., Rust-based execution) does not replicate the exact range-check semantics of `assert_nn_le`, it will accept and include a transaction with `max_fee >= 2^129` into a block. The Cairo OS prover will then fail to generate a valid STARK proof for that block. Since the block cannot be proven, the chain halts: no further blocks can be finalized until the invalid block is discarded, which requires operator intervention and a potential chain reorganization.

---

### Likelihood Explanation

Any unprivileged transaction sender can trigger this. A concrete minimal example:

- `l1_gas_bounds.max_amount = 3`
- `l1_gas_bounds.max_price_per_unit = 2^128 − 1`

This gives `l1_gas_bounds.max_amount × l1_gas_bounds.max_price_per_unit = 3 × (2^128 − 1) = 3 × 2^128 − 3 > 2^129`.

Both values pass all existing validation in `pack_resource_bounds`:
- `3 <= 2^64 − 1` ✓
- `2^128 − 1 >= 0` ✓ [5](#0-4) 

No privileged access, no key compromise, and no network-level attack is required.

---

### Recommendation

Add an explicit upper-bound assertion on `max_fee` inside `compute_max_possible_fee` or immediately after it in `charge_fee`, ensuring the result fits within 128 bits before passing it to `assert_nn_le`:

```cairo
// After computing max_fee:
assert_nn_le(max_fee, 2 ** 128 - 1);
```

Alternatively, replace `assert_nn_le` with a proper 256-bit comparison that correctly handles felt values exceeding `2^128`, consistent with the `Uint256` representation already used for the actual fee amount. [6](#0-5) 

---

### Proof of Concept

1. Craft a v3 invoke transaction with:
   - `l1_gas_bounds = { resource: L1_GAS, max_amount: 3, max_price_per_unit: 2^128 − 1 }`
   - `l2_gas_bounds = { resource: L2_GAS, max_amount: 1, max_price_per_unit: 1 }`
   - `l1_data_gas_bounds = { resource: L1_DATA_GAS, max_amount: 1, max_price_per_unit: 1 }`
   - `tip = 0`

2. `compute_max_possible_fee` returns `3 × (2^128 − 1) + 1 + 1 = 3 × 2^128 − 1 > 2^129`.

3. All `pack_resource_bounds` and `hash_fee_fields` checks pass (amounts ≤ `2^64 − 1`, prices ≥ 0, tip ≤ `2^64 − 1`).

4. The transaction is accepted by the mempool and included in a block.

5. During proving, `charge_fee` is reached. `low_actual_fee` (the real cost, e.g., `10^18 ≈ 2^60`) is loaded via `%{ LoadActualFee %}`.

6. `assert_nn_le(2^60, 3 × 2^128 − 1)` evaluates `assert_nn(3 × 2^128 − 1 − 2^60)`. The value `3 × 2^128 − 1 − 2^60 >> 2^128`, so the range-check fails.

7. The STARK proof cannot be generated. The block is stuck. The network halts. [7](#0-6) [4](#0-3)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
