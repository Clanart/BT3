### Title
Cross-Chain Replay of `deployToken` MPC Signature Enables Unauthorized Token Deployment on Any Supported Chain — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The MPC-signed payload used by `deployToken` across EVM, StarkNet, and Solana omits the destination chain identifier (`omniBridgeChainId`). Because the same NEAR MPC key derives the same `nearBridgeDerivedAddress` on every supported chain, a valid `deployToken` signature obtained for one chain (e.g., Ethereum) can be replayed verbatim on any other chain (Base, Arbitrum, StarkNet, Solana) to deploy an unauthorized bridge token. By contrast, `finTransfer` correctly binds its signed payload to the destination chain ID, making this omission in `deployToken` a deliberate inconsistency with a concrete security consequence.

---

### Finding Description

**EVM `deployToken` signed payload** (`OmniBridge.sol` lines 142–149):

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)

bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
```

**EVM `finTransfer` signed payload** (lines 289–311) — for comparison, the chain ID appears twice:

---

### Impact Explanation

Once a `deployToken` signature is broadcast on chain A (e.g., Ethereum), any observer can submit the identical `(signatureData, metadata)` tuple to the `deployToken` function on chain B (Base, Arbitrum, StarkNet, Solana). The signature check passes because the signer address is identical on all chains and the signed bytes are identical. The result:

1. **Unauthorized bridge token deployed on chain B.** The token is registered as `isBridgeToken[proxy] = true` and `nearToEthToken[metadata.token] = proxy` on chain B without NEAR's explicit authorization for that chain. [9](#0-8) 

2. **Permanent griefing of legitimate future deployment.** The `ERR_TOKEN_EXIST` guard (`!isBridgeToken[nearToEthToken[metadata.token]]`) prevents the same NEAR token ID from ever being legitimately deployed on chain B afterward. The token slot is permanently occupied by the attacker-triggered proxy. [10](#0-9) 

3. **Token-mapping corruption enabling accounting desync.** If the NEAR bridge later registers chain B as a supported destination and the replayed token address is used in `initTransfer` events, the NEAR bridge's token-address-to-id mapping will be inconsistent, breaking collateralization accounting for that token on chain B.

4. **Unauthorized `finTransfer` minting path.** If the NEAR bridge is configured with chain B as a factory and the replayed token is used in a transfer, the bridge on chain B can mint tokens via `finTransfer` (which requires its own MPC signature, but that signature is now obtainable because the token is registered). This creates an unbacked supply path.

---

### Likelihood Explanation

- The attack requires no privileged access. Any user who observes a `deployToken` transaction on chain A (public mempool or block explorer) obtains the full `(signatureData, metadata)` tuple.
- The attacker only pays gas on chain B.
- The Omni Bridge is deployed on multiple EVM chains (Ethereum, Base, Arbitrum) and StarkNet/Solana, all sharing the same MPC-derived signer address.
- The `deployToken` function is permissionless (`external payable`), so no role check blocks the replay.
- The attack is immediately executable after any legitimate `deployToken` call on any chain.

---

### Recommendation

Include `omniBridgeChainId` in the borsh-encoded payload that is signed for `deployToken`, mirroring the pattern already used in `finTransfer`. For EVM:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to destination chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent fix to `MetadataPayload::to_borsh` in `starknet/src/bridge_types.cairo` and `DeployTokenPayload::serialize_for_near` in `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`. The NEAR MPC signing side (`sign_log_metadata_callback`) must also include the target chain ID in the payload it signs so that signatures are chain-specific.

---

### Proof of Concept

1. NEAR MPC signs metadata for `wrap.near` targeting Ethereum (chain ID `0x00`). The signed borsh bytes are: `[0x01, <token_borsh>, <name_borsh>, <symbol_borsh>, <decimals>]`.
2. A relayer calls `OmniBridgeEthereum.deployToken(sig, metadata)` — succeeds. `wrap.near` is now deployed on Ethereum.
3. Attacker copies `sig` and `metadata` from the Ethereum transaction.
4. Attacker calls `OmniBridgeBase.deployToken(sig, metadata)` — the signature check at line 151 passes because `nearBridgeDerivedAddress` is identical on Base and the signed bytes are identical.
5. `wrap.near` is now deployed on Base as a bridge token without NEAR's authorization for Base.
6. `nearToEthToken["wrap.near"]` on Base is set to the attacker-triggered proxy address.
7. Any future legitimate attempt to deploy `wrap.near` on Base reverts with `ERR_TOKEN_EXIST`.
8. Repeat for Arbitrum, StarkNet (`deploy_token` with same `signature` and `payload`), and Solana.
