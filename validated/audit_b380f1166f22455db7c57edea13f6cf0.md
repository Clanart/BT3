Audit Report

## Title
`PeripheryPayments.pay()` consumes residual contract ETH from prior transactions to subsidize the current payer's WETH obligation — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`pay()` reads `address(this).balance` at line 74 to determine how much ETH to wrap toward a WETH payment. This is the contract's total ETH balance, not the ETH sent in the current call. ETH stranded from a prior payable call (where the user omitted `refundETH`) is silently consumed to reduce a subsequent caller's `safeTransferFrom` obligation, permanently destroying the prior depositor's funds.

## Finding Description
`pay()` in `PeripheryPayments.sol` handles WETH payments via a three-branch structure keyed on `address(this).balance`: [1](#0-0) 

The `receive()` guard at lines 32–34 only blocks ETH arriving via low-level `call` from non-WETH addresses: [2](#0-1) 

It does not block ETH arriving via `msg.value` in the four `payable` entry points (`addLiquidityExactShares` ×2, `addLiquidityWeighted` ×2, `multicall`): [3](#0-2) 

The callback at lines 172–177 only calls `pay()` for tokens where the pool requests a non-zero delta: [4](#0-3) 

If a user calls `addLiquidityExactShares{value: X}` on a pool where only token1 is needed (e.g., liquidity entirely in token1 bins), `pay()` is invoked for token1 only — the ETH is never consumed. If the user omits `refundETH`, the ETH is stranded. A subsequent caller paying WETH then hits the `else if (nativeBalance > 0)` branch: the contract wraps the residual ETH, transfers it to the pool, and pulls only `value - residualETH` from the new payer via `safeTransferFrom`. The pool receives the correct total; the prior depositor's ETH is permanently gone.

`refundETH()` at lines 58–63 returns `address(this).balance` to `msg.sender`, but once the residual ETH is consumed by a subsequent `pay()` call, it is unrecoverable: [5](#0-4) 

## Impact Explanation
Direct, permanent loss of user principal (ETH). The prior depositor's ETH is consumed and cannot be recovered via `refundETH`. The payment invariant — "the registered payer must supply the full token amount" — is violated: the current payer's `transferFrom` obligation is reduced by an amount they did not provide. This is a direct principal loss for the prior depositor above Sherlock thresholds (any non-dust ETH amount qualifies).

## Likelihood Explanation
The precondition is an innocent user calling a payable liquidity function with `msg.value > 0` on a pool where the WETH leg is not needed, then omitting `refundETH`. This is realistic: the `multicall` + `refundETH` pattern is easy to omit, and single-sided liquidity additions (only token1 bins active) are a normal operational scenario. An attacker monitoring the contract's ETH balance can exploit the window between the victim's deposit and their `refundETH` call. The attacker cannot profit by self-seeding (net zero), but the victim's loss is real and repeatable.

## Recommendation
Track only the ETH sent in the current transaction. Replace `address(this).balance` in `pay()` with a `msg.value`-derived amount passed through the call chain. The simplest fix: capture `msg.value` at the outermost payable entry point, store it in transient storage alongside the existing pay context (`T_SLOT_PAY_PAYER`, etc.), and pass it into `pay()` as the available native balance. This ensures residual ETH from prior transactions is never consumed on behalf of a new payer.

## Proof of Concept
1. Deploy `MetricOmmPoolLiquidityAdder` with a WETH/token1 pool.
2. Call `addLiquidityExactShares{value: 0.5 ether}(pool, owner, salt, token1OnlyDelta, 0, max1, "")` — pool requests only token1; `pay()` is called for token1 only; 0.5 ETH stays in the adder.
3. From a second account (User B), call `addLiquidityExactShares(pool, owner2, salt2, wethDelta, 1e18, max1, "")` with no ETH sent.
4. Observe: `pay()` enters the `else if (nativeBalance > 0)` branch, wraps 0.5 ETH, and calls `safeTransferFrom(userB, pool, 0.5e18)` instead of `1e18`.
5. Assert: User B's WETH allowance consumed = 0.5e18 (not 1e18). User A's `refundETH()` returns 0.
