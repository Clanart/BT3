### Title
Missing shard-id binding on `prev_chunk_header` in `set_state_header` allows wrong-shard receipt range to be accepted — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` verifies `prev_chunk_header` via a Merkle inclusion proof but never asserts `prev_chunk_header.shard_id() == shard_id`. A malicious peer can substitute another shard's `prev_chunk_header` (with its valid proof at that shard's position), causing `prev_chunk_header.height_included()` to bound the incoming-receipt range to the wrong interval, silently omitting or extending receipts for the synced shard.

### Finding Description
In `set_state_header` (lines 368–532), step-3b verifies `prev_chunk_header` with:

```rust
verify_path(
    *prev_block_header.chunk_headers_root(),
    prev_chunk_proof,
    &ChunkHashHeight(prev_chunk_header.chunk_hash(), prev_chunk_header.height_included()),
)
``` [4](#0-3) 

This proves the `(hash, height_included)` pair is *some* leaf in the Merkle tree, but does not enforce that it is the leaf at the expected shard index. No subsequent code checks `prev_chunk_header.shard_id() == shard_id`. Step-4g then uses `prev_chunk_header.height_included()` as the lower bound for the receipt range:

```rust
if header.height() != prev_chunk_header.map_or(0, |h| h.height_included()) {
``` [5](#0-4) 

If the substituted shard's `height_included` is higher than the target shard's, the receipt range is shortened and receipts from the omitted blocks are never stored for the synced shard.

### Impact Explanation
A syncing node that accepts this crafted header stores it under `StateHeaderKey(shard_id, sync_hash)` in `DBCol::StateHeaders`. Subsequent state-part application uses the stored header's `prev_state_root` (from the main chunk, which may be correct) but the incoming receipts derived from the wrong range. The resulting state root after applying parts + receipts diverges from the canonical state root, rendering the node unable to validate or produce blocks for that shard correctly.

### Likelihood Explanation
Any peer the syncing node contacts for state sync can supply this crafted response. State sync is performed by nodes that are catching up after being offline or joining the network — an unprivileged peer position is sufficient. The precondition (shard 1's `height_included` ≠ shard 0's `height_included` in the relevant epoch) is a normal network condition, not a special setup.

### Recommendation
Add an explicit shard-id equality check immediately after extracting `prev_chunk_header`:

```rust
if let Some(ref pch) = prev_chunk_header {
    if pch.shard_id() != shard_id {
        return Err(Error::Other(
            "set_shard_state failed: prev_chunk_header shard_id mismatch".into(),
        ));
    }
}
```

Additionally, verify that `prev_chunk_proof` corresponds to the expected shard index (i.e., the proof path encodes position `prev_shard_index`) rather than any arbitrary position in the tree.

### Proof of Concept
1. Syncing node requests state for `shard_id = 0`, `sync_hash = H`.
2. Attacker peer constructs a `ShardStateSyncResponseHeaderV2` where:
   - `chunk` = legitimate shard-0 chunk (passes `validate_chunk_proofs` and step-3a)
   - `prev_chunk_header` = shard-1's chunk header from `prev_block` (with `height_included = T+10` vs shard-0's `height_included = T`)
   - `prev_chunk_proof` = valid Merkle proof for shard-1's position in `prev_block.chunk_headers_root()`
   - `incoming_receipts_proofs` = valid receipt proofs only for blocks `[sync_hash .. T+11]` (10 blocks instead of 20)
3. Step-3b: `verify_path(prev_block.chunk_headers_root(), shard1_proof, ChunkHashHeight(shard1_hash, T+10))` → `true` (shard-1 is genuinely in the tree at position 1).
4. Step-4g: `header.height() == T+10 == prev_chunk_header.height_included()` → passes.
5. Header is stored. Receipts from blocks `[T+1 .. T+10]` for shard 0 are never included in the synced state.
