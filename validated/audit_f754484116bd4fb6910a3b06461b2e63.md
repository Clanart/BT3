### Title
Missing Class Hash Validation in `execute_replace_class` Enables Permanent Contract Fund Freezing — (File: `execution/syscall_impls.cairo`)

### Summary
The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the replacement class hash is a declared class. A contract deployer can craft a contract that calls `replace_class` with an arbitrary undeclared class hash, permanently rendering the contract uncallable and freezing any funds it holds.

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function accepts any `class_hash` from the syscall request and writes it directly into `contract_state_changes` without verifying that the hash exists in `contract_class_changes` (i.e., that it was previously declared):

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment at line 898 explicitly acknowledges this missing check. [1](#0-0) 

Once the contract's `class_hash` is set to an undeclared value, any subsequent call to that contract reaches `execute_entry_point`, which performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [2](#0-1) 

Because the undeclared class hash has no entry in `contract_class_changes`, `dict_read` returns 0 (the default initial value). `find_element` with `key=0` then fails to locate any compiled class fact (no class has hash 0), causing the OS proof to be invalid. The sequencer, detecting this, will permanently refuse to include any call to this contract in future blocks. The contract becomes permanently uncallable, and all funds it holds are frozen with no recovery path.

This is the direct analog to the original report: just as `WithdrawProxy.deposit` was callable without restriction and inflated `totalSupply` beyond what `PublicVault` could burn — causing `processEpoch` to revert and freeze LP funds — here `replace_class` accepts any class hash without restriction, setting the contract's class pointer to a value the OS can never resolve, permanently freezing the contract.

### Impact Explanation

Any ERC-20 tokens, ETH, or other assets held in the storage of a contract that has been `replace_class`'d to an undeclared hash are permanently frozen. There is no upgrade path, no admin override, and no recovery mechanism in the OS. The impact is **Critical: Permanent freezing of funds**.

### Likelihood Explanation

The attack requires a contract deployer to publish a contract whose code contains a `replace_class(undeclared_hash)` call reachable by an external trigger (e.g., a public function). Users who deposit funds into such a contract (believing it to be a legitimate vault, DEX, or bridge) are exposed. Once the attacker triggers the function, the freeze is irreversible. The attacker needs no special privilege beyond deploying a contract — a capability available to any unprivileged user.

### Recommendation

In `execute_replace_class`, before writing the new `class_hash` into `contract_state_changes`, verify that the hash exists in `contract_class_changes` (i.e., it was previously declared via `execute_declare_transaction`). A `dict_read` on `contract_class_changes` with the new `class_hash` as key should return a non-zero compiled class hash; if it returns 0, the syscall should write a failure response and return without updating state.

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract that (a) accepts user deposits and (b) exposes a `freeze()` function that calls `replace_class(0xDEAD)` where `0xDEAD` is never declared.
2. Users deposit funds into `MaliciousVault`, trusting its interface.
3. Attacker sends a transaction calling `MaliciousVault.freeze()`.
4. The OS processes `execute_replace_class` with `class_hash = 0xDEAD` — **no validation occurs** (line 898
