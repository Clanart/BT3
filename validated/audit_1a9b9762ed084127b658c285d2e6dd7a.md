### Title
Unguarded `refundETH()` allows any caller to steal ETH stranded on the router from a prior transaction — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`refundETH()` is an unrestricted `external` function that transfers the router's **entire** native ETH balance to `msg.sender`. Because `pay()` only wraps exactly the swap-required amount of ETH (leaving any excess on the contract), a victim who calls a payable swap function with more ETH than the swap consumes — without atomically bundling `refundETH()` in the same `multicall` — leaves residual ETH on the router that any subsequent caller can steal.

---

### Finding Description

`PeripheryPayments.refundETH()` contains no access control and no per-caller accounting: [1](#0-0) 

It unconditionally sends `address(this).balance` to `msg.sender`.

The `pay()` helper, invoked from the swap callback, wraps only the exact amount the pool requests: [2](#0-1) 

When `nativeBalance >= value`, exactly `value` wei is wrapped and forwarded; the surplus (`nativeBalance - value`) remains on the router after the transaction ends.

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses: [3](#0-2) 

It does **not** prevent ETH from arriving via `msg.value` on any `payable` entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`). [4](#0-3) 

There is no enforcement that `msg.value == amountIn`, and no automatic refund at the end of a standalone swap call.

---

### Impact Explanation

Any ETH left on the router after a transaction — from a victim who sent excess `msg.value` without atomically calling `refundETH()` — is immediately claimable by any arbitrary address in the next transaction. This is a **direct, unconditional theft of user principal** with no preconditions beyond the victim's excess ETH being present. Impact: **High**.

---

### Likelihood Explanation

- `exactInputSingle` and the other swap functions are standalone `payable` entry points; users and integrators routinely send a round-number ETH value to cover an uncertain `amountIn`.
- The multicall-plus-`refundETH` pattern is documented only in the liquidity adder interface, not enforced at the router level.
- A mempool observer can trivially detect a transaction that leaves ETH on the router and immediately call `refundETH()` in the next block.

Likelihood: **Medium** (requires victim to not use the multicall pattern, but this is a realistic and common mistake).

---

### Recommendation

Two complementary fixes:

1. **Track per-payer ETH deposits** in transient storage at the start of each payable entry point and restrict `refundETH()` to return only the amount attributed to `msg.sender`.
2. **Auto-refund excess ETH** at the end of each standalone swap function (mirror the `multicall` + `refundETH` pattern internally), so no ETH can ever be stranded between transactions.

---

### Proof of Concept

```
1. Victim calls:
   router.exactInputSingle{value: 2 ether}(
     ExactInputSingleParams({
       tokenIn: WETH,
       amountIn: 1 ether,   // only 1 ether consumed by pay()
       ...
     })
   );
   → pay() wraps exactly 1 ether; 1 ether remains on router.

2. Attacker (any EOA) calls in the next transaction:
   router.refundETH();
   → _transferETH(msg.sender, 1 ether) executes.
   → Attacker receives victim's 1 ether.
```

The existing test `test_refundETH_sendsBalanceToCaller` already demonstrates that any caller receives the full router balance: [5](#0-4)
