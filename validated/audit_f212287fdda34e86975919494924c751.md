### Title
Unconditional Token Burn Before Refund Check in `resolve_fast_transfer` Causes Permanent Loss of Relayer Funds — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

In `resolve_fast_transfer`, `burn_tokens_if_needed` is called **unconditionally** before checking whether the transfer to the recipient succeeded. When the recipient's `ft_on_transfer` reverts (returning tokens to the bridge), the bridge burns those tokens and then signals a refund of the same amount to the relayer. Because the tokens were already burned, the NEP-141 refund transfer from the bridge back to the relayer fails, permanently destroying the relayer's funds.

---

### Finding Description

The fast-transfer flow for NEAR recipients is:

1. A trusted relayer calls `ft_transfer_call(bridge, amount, FastFinTransfer{...})`.
2. The token contract transfers `amount` tokens from the relayer to the bridge and calls `bridge.ft_on_transfer`.
3. `ft_on_transfer` → `fast_fin_transfer` → `fast_fin_transfer_to_near_callback` → `send_tokens(recipient, amount_without_fee, msg)`.
4. If `msg` is non-empty, `send_tokens` issues `ft_transfer_call(recipient, amount_without_fee, msg)`.
5. If the recipient's `ft_on_transfer` reverts, `amount_without_fee` tokens are returned to the bridge.
6. The callback `resolve_fast_transfer` is invoked.

Inside `resolve_fast_transfer`:

```rust
// near/omni-bridge/src/lib.rs  lines 895–912
pub fn resolve_fast_transfer(
    &mut self,
    token_id: &AccountId,
    fast_transfer_id: &FastTransferId,
    amount: U128,
    is_ft_transfer_call: bool,
) -> U128 {
    // Burn the tokens to ensure the locked tokens are not double-minted
    self.burn_tokens_if_needed(token_id.clone(), amount);   // ← always executes

    if Self::is_refund_required(is_ft_transfer_call) {
        self.remove_fast_transfer(fast_transfer_id);
        amount                                               // ← signals refund
    } else {
        U128(0)
    }
}
```

`burn_tokens_if_needed` is called **before** the refund check. For bridge-deployed tokens (tokens minted on NEAR from a foreign chain), this burns the `amount_without_fee` tokens that were just returned to the bridge by the failed recipient transfer. Immediately after, `is_refund_required` detects the failure and returns `amount` to the NEP-141 token contract as the refund amount. The token contract then attempts `ft_transfer(relayer, amount_without_fee)` from the bridge's account, but the bridge's balance for that token is now zero — the refund silently fails and the relayer's tokens are permanently destroyed.

The call site that passes `amount_without_fee` to `resolve_fast_transfer`:


---

### Impact Explanation

For any fast transfer that uses a bridge-deployed token and routes to a NEAR recipient via `ft_transfer_call` (non-empty `msg`):

- If the recipient's `ft_on_transfer` reverts for any reason (slippage, insufficient liquidity, paused contract, intentional revert), the relayer's `amount_without_fee` tokens are **permanently burned** and the refund to the relayer fails.
- The fast-transfer record is also removed (`remove_fast_transfer`), so the relayer cannot retry or recover.
- This matches the allowed impact: **Permanent freezing / irrecoverable lock of user or protocol funds in bridge flows**.

---

### Likelihood Explanation

- The recipient of a fast transfer can be any NEAR smart contract. Contracts that auto-swap incoming tokens (e.g., DEX integrations) are common and can revert due to slippage or low liquidity.
- A malicious recipient can intentionally revert in `ft_on_transfer` to grief any relayer who services a fast transfer to that address.
- No privileged access is required; the attacker only needs to be the designated recipient of a cross-chain fast transfer.

---

### Recommendation

Move `burn_tokens_if_needed` inside the success branch (i.e., only burn when the transfer to the recipient succeeded):

```rust
pub fn resolve_fast_transfer(
    &mut self,
    token_id: &AccountId,
    fast_transfer_id: &FastTransferId,
    amount: U128,
    is_ft_transfer_call: bool,
) -> U128 {
    if Self::is_refund_required(is_ft_transfer_call) {
        // Transfer failed — do NOT burn; tokens are still at the bridge
        // and will be returned to the relayer via the NEP-141 refund.
        self.remove_fast_transfer(fast_transfer_id);
        amount
    } else {
        // Transfer succeeded — safe to burn to prevent double-minting.
        self.burn_tokens_if_needed(token_id.clone(), amount);
        U128(0)
    }
}
```

### Proof of Concept

1. A bridge-deployed token `eth.token.near` exists (minted on NEAR, origin chain ETH).
2. A cross-chain fast transfer is initiated on ETH: recipient = `malicious.near` (a contract that always reverts in `ft_on_transfer`), amount = 1000, fee = 10, msg = `"swap"`.
3. A trusted relayer calls `ft_transfer_call(bridge, 1000, FastFinTransfer{recipient: malicious.near, amount: 1000, fee: 10, msg: "swap", ...})`.
4. Bridge receives 1000 tokens. `fast_fin_transfer_to_near_callback` calls `ft_transfer_call(malicious.near, 990, "swap")`.
5. `malicious.near.ft_on_transfer` reverts → 990 tokens return to bridge.
6. `resolve_fast_transfer(eth.token.near, id, 990, true)` is called:
   - `burn_tokens_if_needed` burns 990 tokens from bridge (bridge balance = 0 for this token).
   - `is_refund_required` returns `true`.
   - Returns `U128(990)` as refund signal.
7. Token contract calls `ft_transfer(relayer, 990)` from bridge — bridge has 0 balance → **panics / fails silently**.
8. Relayer has lost 990 `eth.token.near` tokens permanently. The fast-transfer record is deleted. [1](#0-0) [2](#0-1) [3](#0-2)
