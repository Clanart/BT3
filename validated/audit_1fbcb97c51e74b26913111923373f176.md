Looking at the exact code path in question:

### Title
P2P Sync Stores Attacker-Controlled `parent_hash` Without Validation, Breaking Chain Linkage Invariant — (`crates/apollo_p2p_sync/src/client/header.rs`)

---

### Summary

`HeaderStreamBuilder::parse_data_for_block` accepts a `SignedBlockHeader` from a network peer and only validates `block_number` equality. It never checks that `parent_hash` matches the stored `block_hash` of block N-1. The peer-supplied header, including its arbitrary `parent_hash`, is written verbatim to storage via `append_header`. The exact corrupted value is `stored_header(N).parent_hash` = attacker-controlled ≠ `stored_header(N-1).block_hash`.

---

### Finding Description

**Step 1 — Missing guard in `parse_data_for_block`.**

The `_storage_reader` parameter is prefixed with `_`, confirming it is intentionally unused. The only checks performed are `block_number` equality and `signatures.len() == 1`. A developer-acknowledged TODO at line 102 explicitly marks the absent guard:

```
// TODO(shahak): Check that parent_hash is the same as the previous block's hash
// and handle reverts.
``` [1](#0-0) 

**Step 2 — `write_to_storage` passes the peer header directly to `append_header`.**

No `parent_hash` validation occurs between `parse_data_for_block` returning `Ok(Some(...))` and the storage write. The entire peer-supplied `self.block_header` — including its `parent_hash` — is forwarded unchanged. [2](#0-1) 

**Step 3 — `append_header` stores `parent_hash` verbatim.**

`StorageBlockHeader` is constructed with `parent_hash: block_header.block_header_without_hash.parent_hash` and written to the `headers` table with no cross-check against the previous block's stored `block_hash`. [3](#0-2) 

**Step 4 — Contrast: central sync enforces the invariant.**

`store_block` in `apollo_central_sync` calls `self.verify_parent_block_hash(block_number, &block)?` before any write. That function reads `stored_header(N-1).block_hash` and returns `Err(StateSyncError::ParentBlockHashMismatch)` on mismatch. The P2P path has no equivalent call. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

The concrete corrupted value is `stored_header(N).parent_hash` = arbitrary peer-supplied felt ≠ `stored_header(N-1).block_hash`. Downstream effects:

- **RPC** — Any `starknet_getBlockWithTxHashes` / `starknet_getBlockWithTxs` call for block N returns the wrong `parent_hash` as an authoritative committed value. Clients performing chain-linkage verification (e.g., light clients, bridges, proof verifiers) receive a silently incorrect field.
- **Chain integrity** — The node cannot detect a fork or reorg on the P2P path because the invariant `stored_header(N).parent_hash == stored_header(N-1).block_hash` is never enforced.
- **Proof path (if node is a prover)** — `calculate_block_hash` for block N reads `parent_hash` from the stored header as one of its inputs. A wrong stored `parent_hash` produces a wrong computed block hash for N, which propagates to `BaseBlockInfo.prev_base_block_hash` in the transaction prover and to SNOS's block hash chain.

Impact category: **High — RPC returns an authoritative-looking wrong committed value** (and Critical if the node also runs the prover).

---

### Likelihood Explanation

Any P2P peer — no privileges required — can send a syntactically valid `SignedBlockHeader` with a correct `block_number` and a single signature but an arbitrary `parent_hash`. The TODO comment confirms the developers are aware the check is absent. The `BLOCK_NUMBER_LIMIT::Unlimited` setting means the stream runs continuously, so every header synced from a malicious peer is affected. [6](#0-5) 

---

### Recommendation

In `parse_data_for_block`, use the already-available `storage_reader` (currently named `_storage_reader` and unused) to read `stored_header(block_number - 1).block_hash` and compare it to `signed_block_header.block_header.block_header_without_hash.parent_hash`. On mismatch, return `Err(ParseDataError::BadPeer(...))` — mirroring the logic already present in `verify_parent_block_hash` in `apollo_central_sync`. [7](#0-6) 

---

### Proof of Concept

```rust
// 1. Write block 0 with a known block_hash.
writer.begin_rw_txn().unwrap()
    .append_header(BlockNumber(0), &BlockHeader {
        block_hash: BlockHash(felt!("0xabc")),
        ..Default::default()
    }).unwrap().commit().unwrap();

// 2. Feed a network header for block 1 with a WRONG parent_hash.
let malicious_header = SignedBlockHeader {
    block_header: BlockHeader {
        block_hash: BlockHash(felt!("0x1")),
        block_header_without_hash: BlockHeaderWithoutHash {
            block_number: BlockNumber(1),
            parent_hash: BlockHash(felt!("0xdeadbeef")), // wrong: should be 0xabc
            ..Default::default()
        },
        state_diff_length: Some(0),
        ..Default::default()
    },
    signatures: vec![BlockSignature::default()],
};

// 3. parse_data_for_block returns Ok(Some(...)) — no rejection.
// 4. write_to_storage commits it.
// 5. Assert corruption:
let stored = reader.begin_ro_txn().unwrap()
    .get_block_header(BlockNumber(1)).unwrap().unwrap();
assert_eq!(stored.block_header_without_hash.parent_hash, BlockHash(felt!("0xdeadbeef")));
// stored parent_hash (0xdeadbeef) != stored block_hash of block 0 (0xabc)
```

    // Compares the block's parent hash to the stored block.
    fn verify_parent_block_hash(
        &self,
        block_number: BlockNumber,
        block: &Block,
    ) -> StateSyncResult {
        let prev_block_number = match block_number.prev() {
            None => return Ok(()),
            Some(bn) => bn,
        };
        let prev_hash = self
            .reader
            .begin_ro_txn()?
            .get_block_header(prev_block_number)?
            .ok_or(StorageError::DBInconsistency {
                msg: format!(
                    "Missing block {prev_block_number} in the storage (for verifying block \
                     {block_number}).",
                ),
            })?
            .block_hash;

        if prev_hash != block.header.block_header_without_hash.parent_hash {
            // Block parent hash mismatch, log and restart sync loop.
            warn!(
                "Detected reorg in central. block number {block_number} had hash {prev_hash} but \
                 the next block's parent hash is {}.",
                block.header.block_header_without_hash.parent_hash
            );
            CENTRAL_SYNC_FORKS_FROM_FEEDER.increment(1);
            return Err(StateSyncError::ParentBlockHashMismatch {
                block_number,
                expected_parent_block_hash: block.header.block_header_without_hash.parent_hash,
                stored_parent_block_hash: prev_hash,
            });
        }
