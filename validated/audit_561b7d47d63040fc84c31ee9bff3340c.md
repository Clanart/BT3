### Title
Missing Duplicate Key/Hash Validation in `MerkleBlob::batch_insert` Corrupts Merkle Tree Root — (File: crates/chia-datalayer/src/merkle/blob.rs)

---

### Summary

`MerkleBlob::batch_insert` silently bypasses the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces for every item beyond the first two. An attacker who can supply attacker-controlled `(KeyId, ValueId, Hash)` triples to a DataLayer node — e.g., via a crafted delta-sync payload — can insert duplicate leaf entries, producing an internally inconsistent tree whose root hash no longer reflects the true committed state. Downstream Merkle proofs of inclusion or exclusion computed against that root are then forged or invalid.

---

### Finding Description

`MerkleBlob::insert` (lines 362–413) enforces two invariants before writing any leaf:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::batch_insert` (lines 570–657) is the bulk-insertion path. It calls `self.insert()` — and therefore runs those guards — only for the first two items when the tree currently has ≤ 1 leaf:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { return Ok(()); };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;   // validated
    }
}

for ((key, value), hash) in keys_values_hashes {          // ← NO duplicate check
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ..., node: Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;  // written directly
    indexes.push(new_leaf_index);
}
```

Every item in the `keys_values_hashes` vector that is processed by the second loop is written directly to the blob without consulting `block_status_cache`. Consequently:

1. **Duplicate key** — a `KeyId` already present in the tree can be inserted again, creating two leaf nodes with the same key. The cache's `key_to_index` map still points to the original leaf; the new leaf is invisible to the cache but physically present in the blob.
2. **Duplicate hash** — a `Hash` already present can be inserted again, violating the uniqueness invariant that the tree relies on for correct proof generation.
3. **Dirty-lineage propagation is skipped** — the new leaves are written with `parent: Parent(None)` and are never linked into the existing tree structure during the loop; the subsequent subtree-attachment step (`insert_subtree_at_key`) only handles the batch of *new* nodes, leaving the cache and the blob in a diverged state.

The `check_integrity` method (lines 812–887) would detect the inconsistency, but it is not called automatically after `batch_insert`.

`batch_insert` is exposed to Python callers via the wheel (`wheel/python/chia_rs/datalayer.pyi`) and is invoked from `deltas.rs` during delta-sync and from `proof_of_inclusion.rs` during proof construction — both paths that can receive network-supplied data.

---

### Impact Explanation

A corrupted `MerkleBlob` produces an incorrect root hash. Any proof of inclusion or exclusion generated against that root is either unverifiable or falsely verifiable, allowing:

- **Forged inclusion proofs**: a key that was never legitimately inserted can appear to be proven present.
- **Forged exclusion proofs**: a key that is present can appear to be proven absent.
- **Committed-state corruption**: the DataLayer root stored on-chain diverges from the actual key-value set, permanently corrupting the auditable state.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

`batch_insert` is called from `deltas.rs` during delta-sync, a code path that processes data received from remote peers. A peer that sends a delta containing a key already present in the local tree (or a hash collision) will trigger the unguarded loop. No special privilege is required beyond the ability to participate in DataLayer sync. The condition `leaf_count > 1` is satisfied by any non-trivial tree, making the vulnerable branch the common case in production.

---

### Recommendation

Apply the same duplicate-key and duplicate-hash guards inside the bulk loop in `batch_insert`, or refactor the function to route every item through `self.insert()`. At minimum, call `self.check_integrity()` after `batch_insert` completes and return an error if integrity fails. Additionally, update `block_status_cache` atomically with each blob write so that the cache and the blob cannot diverge.

---

### Proof of Concept

```rust
// Tree already has 2 leaves (leaf_count > 1), so the guarded branch is skipped.
let mut blob = MerkleBlob::new(vec![]).unwrap();
let key1 = KeyId(1); let val1 = ValueId(1); let hash1 = sha256_num(&1i64);
let key2 = KeyId(2); let val2 = ValueId(2); let hash2 = sha256_num(&2i64);
blob.insert(key1, val1, &hash1, InsertLocation::Auto {}).unwrap();
blob.insert(key2, val2, &hash2, InsertLocation::Auto {}).unwrap();
// leaf_count == 2 → the guarded branch is NOT taken in batch_insert.
// Supply key1 again (duplicate) — no error is returned.
blob.batch_insert(vec![((key1, ValueId(99)), sha256_num(&99i64))]).unwrap();
// Now two leaf nodes with key1 exist in the blob; check_integrity will fail.
assert!(blob.check_integrity().is_err());
