### Title
Unhandled Constructor Revert in `deploy` Syscall Causes OS Proof Failure, Leading to Network Halt — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/deploy_contract.cairo`)

---

### Summary

The `deploy_contract` function in the StarkNet OS unconditionally asserts that the constructor execution does not revert (`assert is_reverted = 0`). The `execute_deploy` syscall handler also always writes `failure_flag=0` in the response. This creates a critical mismatch: the sequencer's off-chain simulation engine (`blockifier`) can correctly model a reverting constructor as a syscall-level failure (returning `failure_flag=1`) and allow the outer transaction to succeed if it handles the failure. However, when the OS Cairo program processes the same block, it hits the unconditional assertion and fails, making it impossible to generate a valid proof for that block, halting the network.

---

### Finding Description

In `deploy_contract.cairo`, after invoking the constructor via `select_execute_entry_point_func`, the return value `is_reverted` is checked with a hard assertion:

```cairo
// The deprecated deploy syscalls do not support reverts.
assert is_reverted = 0;
``` [1](#0-0) 

This function is called from `execute_deploy` in `syscall_impls.cairo`, which is the handler for the new `deploy` syscall callable by any user contract:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [2](#0-1) 

The TODO comment at line 538 explicitly acknowledges that failure handling is not yet implemented. The function-level TODO in `deploy_contract.cairo` also confirms this:

```cairo
// TODO(Yoni, 1/1/2027): handle failures.
func deploy_contract{...}(...) -> (retdata_size: felt, retdata: felt*) {
``` [3](#0-2) 

The `deploy` syscall is reachable from any user contract. The sequencer's off-chain simulation (`blockifier`) correctly models a reverting constructor as a syscall-level failure and propagates `failure_flag=1` to the calling contract. A calling contract can inspect this flag and continue execution successfully. The sequencer then includes the outer transaction as non-reverted. When the OS Cairo program later processes this block to generate a proof, it reaches `assert is_reverted = 0` inside `deploy_contract`, which fails, making proof generation impossible for that block.

The `execute_deploy_account_transaction` path also calls `deploy_contract` directly without any outer `is_reverted` guard:

```cairo
let revert_log = init_revert_log();
deploy_contract{revert_log=revert_log}(
    block_context=block_context, constructor_execution_context=constructor_execution_context
);
``` [4](#0-3) 

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

If an attacker successfully causes a block to be sequenced that contains a transaction where the `deploy` syscall's constructor reverts but the outer transaction is non-reverted (because the outer contract handles the failure), the OS Cairo program cannot generate a valid proof for that block. No subsequent blocks can be proven until the issue is resolved, effectively halting the network's ability to confirm new transactions.

---

### Likelihood Explanation

The attack is reachable by any unprivileged user:

1. Class declaration is permissionless — an attacker can declare a class whose constructor always reverts.
2. The `deploy` syscall is callable from any user contract.
3. A wrapper contract can call `deploy` with the reverting class, inspect the `failure_flag` in the response, and return successfully.
4. The sequencer's `blockifier` simulation correctly models this as a successful outer transaction.
5. The sequencer includes the block; the OS proof generation fails.

No privileged access, leaked keys, or operator collusion is required.

---

### Recommendation

Implement proper failure handling in `deploy_contract` and `execute_deploy`:

1. Remove the unconditional `assert is_reverted = 0` in `deploy_contract.cairo` line 91.
2. In `execute_deploy` (`syscall_impls.cairo`), propagate the constructor's `is_reverted` flag into the `ResponseHeader.failure_flag` field, mirroring the pattern used in `contract_call_helper`:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

3. When `is_reverted=1`, revert the state changes made by `deploy_contract` (the class hash update in `contract_state_changes`) using the existing `handle_revert` mechanism, consistent with how `execute_entry_point` handles reverts. [5](#0-4)

### Proof of Concept

1. **Attacker declares** a contract class `RevertingClass` whose constructor body unconditionally panics/reverts.

2. **Attacker deploys** a wrapper contract `Wrapper` with an `__execute__` function that:
   - Calls the `deploy` syscall with `RevertingClass` as the class hash.
   - Reads the `failure_flag` from the deploy response.
   - Returns successfully regardless of the flag value.

3. **Attacker submits** an invoke transaction targeting `Wrapper.__execute__`.

4. **Sequencer simulation** (`blockifier`): constructor of `RevertingClass` reverts → deploy syscall returns `failure_flag=1` → `Wrapper.__execute__` handles it and returns success → outer transaction is marked non-reverted → block is sequenced.

5. **OS proof generation**: `execute_invoke_function_transaction` runs the execute step (since `is_reverted=FALSE`), calls `execute_deploy`, which calls `deploy_contract`, which calls `select_execute_entry_point_func` for the constructor → constructor reverts → `is_reverted=1` → `assert is_reverted = 0` **fails** → Cairo program aborts → proof cannot be generated → **network halt**.
