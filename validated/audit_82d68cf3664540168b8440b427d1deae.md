Audit Report

## Title
Unguarded `refundETH` allows any caller to steal excess ETH left on the router by a prior user — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.refundETH()` transfers the router's entire native ETH balance to `msg.sender` with no access control. When a user calls `multicall{value: X}([exactInputSingle(..., amountIn < X)])` without appending a `refundETH` call, `pay()` wraps only the required amount and leaves the remainder as native ETH on the router. Any subsequent caller can invoke `refundETH()` directly and receive that ETH.

## Finding Description
`refundETH()` reads `address(this).balance` and unconditionally transfers the full amount to `msg.sender`: [1](#0-0) 

When `pay()` is called for a WETH-input swap and `nativeBalance >= value`, it wraps exactly `value` wei and sends it to the pool, leaving any excess native ETH on the contract: [2](#0-1) 

The `receive()` guard only blocks plain ETH transfers (no-calldata sends); it does not prevent ETH from being deposited via `msg.value` on a payable function call: [3](#0-2) 

`multicall` uses `delegatecall`, so `msg.value` is visible to all sub-calls and any unspent ETH remains on the router after the batch returns: [4](#0-3) 

The test suite confirms the intended pattern — users must include `refundETH` as the last call in their multicall batch — but this is not enforced at the contract level: [5](#0-4) 

If a user omits `refundETH`, the excess ETH sits on the router with no owner attribution, and any subsequent caller can drain it by calling `refundETH()` directly.

## Impact Explanation
Direct loss of user principal. The victim loses the excess ETH they sent with their swap transaction. The attacker gains it with no privileged role required. This meets the "Critical/High direct loss of user principal" impact gate.

## Likelihood Explanation
The intended usage pattern (include `refundETH` in every multicall that sends excess ETH) is not enforced at the contract level. Users who call `exactInputSingle` or `exactOutputSingle` with `msg.value > amountIn` — without wrapping in a multicall that ends with `refundETH` — are silently vulnerable. The attack requires only monitoring the router's ETH balance or the mempool, and is executable by any EOA or contract.

## Recommendation
Record the top-level `msg.sender` in transient storage at the start of `multicall` and require `refundETH` to use that stored address rather than the immediate `msg.sender`. Alternatively, set a reentrancy-style flag during `multicall` execution and revert in `refundETH` if the flag is not set, ensuring `refundETH` can only be reached as a sub-call within an active `multicall` batch from the original depositor.

## Proof of Concept
```
1. User calls router.multicall{value: 1 ether}([
       abi.encodeCall(router.exactInputSingle, (ExactInputSingleParams({
           tokenIn: WETH, amountIn: 0.5 ether, ...
       })))
       // refundETH omitted
   ]);
   // pay() wraps 0.5 ETH → WETH, sends to pool. 0.5 ETH remains on router.
   // address(router).balance == 0.5 ETH

2. Attacker calls router.refundETH();
   // balance = 0.5 ETH > 0 → _transferETH(attacker, 0.5 ETH)
   // Attacker receives 0.5 ETH. User's excess ETH is gone.
```
