### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Unbounded Transaction Replay — (File: `transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` computes a transaction hash with **no nonce or any replay-protection field**, making every signed meta_tx_v0 transaction permanently replayable by any observer. This is a direct analog to the signature malleability / replay-protection failure in the reference report.

---

### Finding Description

`compute_meta_tx_v0_hash` (lines 295–315 of `transaction_hash.cairo`) calls `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`: [1](#0-0) 

The resulting hash is a **pure deterministic function** of five static fields:

```
H(INVOKE_HASH_PREFIX, version=0, contract_address, entry_point_selector, H(calldata), max_fee=0, chain_id)
```

No nonce, no timestamp, no expiry, no counter of any kind is mixed in.

Contrast this with `compute_l1_handler_transaction_hash` (lines 220–238), which correctly passes `additional_data_size=1, additional_data=&nonce`: [2](#0-1) 

And with `hash_tx_common_fields` used by all v3 account transactions, which explicitly commits to `common_fields.nonce`: [3](#0-2) 

The function signature of `compute_meta_tx_v0_hash` does not even accept a nonce parameter, so there is no way for a caller to inject one: [4](#0-3) 

The `MetaTxV0Request` syscall is imported and dispatched in `syscall_impls.cairo`: [5](#0-4) 

Because the OS-level hash function provides no replay protection, any account contract that uses this hash for signature verification (the intended use of the function) inherits the flaw. A signature produced once over this hash is valid for every future replay of the identical `(contract_address, selector, calldata)` tuple on the same chain.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A meta_tx_v0 is a syscall that lets a contract execute a call on behalf of an account (e.g., a token transfer, an approval, or any fund-moving operation). Once a user signs and broadcasts such a transaction, the signature is permanently valid because the hash never changes. An attacker who observes the on-chain transaction can resubmit it indefinitely, draining the account of funds with each replay.

---

### Likelihood Explanation

**High.** Every meta_tx_v0 transaction is publicly visible on-chain. No privileged access, leaked key, or social engineering is required. The attacker only needs to copy the calldata and signature from a confirmed transaction and resubmit them. The entry path is fully unprivileged.

---

### Recommendation

Add a nonce parameter to `compute_meta_tx_v0_hash` and pass it as `additional_data`, mirroring the pattern already used in `compute_l1_handler_transaction_hash`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,          // <-- add nonce
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=1,
        additional_data=&nonce,   // <-- commit to nonce
    );
    return tx_hash;
}
```

The OS must then enforce that each nonce value is consumed exactly once per account, as it does for L1 handler transactions via `consume_l1_to_l2_message`.

---

### Proof of Concept

1. **User** signs and submits a meta_tx_v0 that transfers 100 STRK from their account to address `X`. The OS computes `H(INVOKE_HASH_PREFIX, 0, user_account, transfer_selector, H(calldata), 0, chain_id)` and the account's `__validate__` accepts the signature.
2. The transaction is included in block `N` and is publicly visible.
3. **Attacker** copies the exact calldata and signature from block `N`.
4. Attacker resubmits the identical meta_tx_v0 in block `N+1`. The OS recomputes the same hash (no nonce → same inputs → same hash). The account contract's signature check passes again.
5. The transfer executes a second time. Steps 3–5 repeat until the account is drained.

No privileged access is required. The attacker-controlled entry path is the public transaction submission interface, and the necessary vulnerable step is `compute_meta_tx_v0_hash` omitting the nonce from the committed hash. [1](#0-0)
