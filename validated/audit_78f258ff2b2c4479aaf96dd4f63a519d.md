### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (`crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash supplied by a contract corresponds to a previously declared class. A TODO comment in the production code explicitly acknowledges this missing check. Any contract can therefore replace its own class hash with an arbitrary, undeclared felt value. Once committed to state, the contract becomes permanently unexecutable, freezing all funds it holds.

---

### Finding Description

`execute_replace_class` reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` without any validation:

```cairo
let class_hash = request.class_hash;
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}
tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);
dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The `compiled_class_facts_bundle` is fixed at the start of OS execution and validated post-execution via `validate_compiled_class_facts_post_execution`. [2](#0-1) 

If a contract replaces its class with a hash that is not present in the bundle, any subsequent call to that contract — in the same block or any future block — will fail at the class-lookup hint, making the contract permanently unexecutable. The state transition that wrote the bad class hash is already committed and cannot be undone.

This is the direct analog of the external report's vulnerability class: a mutable configuration parameter (the class hash, analogous to the payout calculator address) is changed mid-operation without validation, causing permanent loss of access to funds.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

After a successful `replace_class` call with an undeclared hash is included in a finalized block, the contract's class hash in the global state is permanently set to an undeclared value. Every future attempt to call the contract will fail during OS execution because the class cannot be resolved from the compiled class facts bundle. Any ERC-20 tokens, ETH, or other assets held in the contract's storage are irrecoverably frozen. There is no recovery path: the state entry cannot be corrected without a protocol-level upgrade.

---

### Likelihood Explanation

The attack is reachable by any unprivileged transaction sender:

1. An attacker deploys a contract whose `__execute__` function accepts user deposits and, after accumulating funds, issues a `replace_class` syscall with an arbitrary undeclared felt (e.g., `1`).
2. The sequencer simulates the transaction; `replace_class` does not revert (no validation exists), so the transaction is included in a block.
3. The block is finalized. The contract's class hash in state is now `1`, which is not in any compiled class facts bundle.
4. All deposited funds are permanently frozen.

No privileged access, leaked keys, or external dependency compromise is required. The only prerequisite is the ability to deploy a contract and submit transactions — standard capabilities of any network participant.

---

### Recommendation

Inside `execute_replace_class`, before writing the new class hash to `contract_state_changes`, verify that the requested class hash exists in `contract_class_changes` (i.e., it was declared in the current block) or in the pre-existing state (i.e., it was declared in a prior block). The check should mirror the validation already performed for the `class_hash` field during `execute_declare_transaction`, where `prev_value=0` enforces uniqueness of declarations. [3](#0-2) 

Concretely, add a `dict_read` on `contract_class_changes` for `class_hash` and assert the result is non-zero before proceeding with the state update in `execute_replace_class`.

---

### Proof of Concept

**Step 1 — Attacker deploys a malicious vault contract** whose constructor accepts deposits and whose `__execute__` entry point, when called with a special selector, issues `replace_class(class_hash=1)`.

**Step 2 — Victims deposit funds.** The contract behaves normally; `class_hash` in state is the legitimate declared hash.

**Step 3 — Attacker calls the special selector.** The OS processes `execute_replace_class`:

```cairo
// request.class_hash = 1  (undeclared)
dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),       // old, valid class hash
    new_value=cast(new_state_entry, felt),    // new class_hash = 1
);
```

No revert occurs. The transaction is included in the block and finalized.

**Step 4 — Any future call to the contract** causes the OS to look up class hash `1` in the compiled class facts bundle. The lookup hint fails; the call reverts. The contract is permanently dead.

**Step 5 — All victim funds are frozen.** The storage entries holding balances remain in state but are inaccessible because no valid class can be executed against the contract address.
