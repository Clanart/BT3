The vulnerability claim is valid. Here is the full analysis:

### Title
EIP-712 `timeout_height` Excluded from Type Schema Enables Replay of Expired Transactions — (`ethereum/eip712/eip712_legacy.go`, `ante/cosmos/eip712.go`)

### Summary

The legacy EIP-712 ante handler path (`ExtensionOptionsWeb3Tx`) is still active in production. The `Tx` type schema used for EIP-712 hash computation explicitly omits `timeout_height`, while `StdSignBytes` includes it in the signed JSON payload. Because EIP-712 hashing only covers fields declared in the type schema, two transactions differing only in `timeout_height` produce an identical `sigHash`. An attacker can take a signed-but-expired transaction, strip its `timeout_height`, and replay it with the original `FeePayerSig` — bypassing the user's intended expiry.

---

### Finding Description

**Step 1 — The legacy EIP-712 path is live.**

`evmd/ante/ante.go` routes any tx carrying `ExtensionOptionsWeb3Tx` to `newLegacyCosmosAnteHandlerEip712`: [1](#0-0) 

That handler installs `cosmos.NewLegacyEip712SigVerificationDecorator` at position 13 of the decorator chain: [2](#0-1) 

**Step 2 — `StdSignBytes` includes `timeout_height` in the JSON payload.**

`VerifySignature` calls `legacytx.StdSignBytes` with `tx.GetTimeoutHeight()`: [3](#0-2) 

When `timeout_height > 0`, the Cosmos SDK `StdSignDoc` serialises it as `"timeout_height":"N"` in the JSON blob (`txBytes`). That blob is then passed verbatim as the `Message` field of the `TypedData` struct via `LegacyWrapTxToTypedData`.

**Step 3 — `timeout_height` is deliberately absent from the `Tx` EIP-712 type schema.**

`extractMsgTypes` in `eip712_legacy.go` defines the `Tx` struct type without `timeout_height`: [4](#0-3) 

The same omission exists in the non-legacy path: [5](#0-4) 

**Step 4 — EIP-712 hashing ignores fields not in the type schema.**

`apitypes.TypedDataAndHash` (go-ethereum) encodes only the fields declared in the type schema. A field present in `Message` but absent from `Types["Tx"]` is silently dropped. Therefore:

- Tx A: `timeout_height = 0` → field omitted from JSON → `sigHash = H`
- Tx B: `timeout_height = N` → field present in JSON but not in schema → `sigHash = H`

Both produce the same `sigHash`, so the same `FeePayerSig` satisfies `VerifySignature` for both. [6](#0-5) 

**Step 5 — `TxTimeoutHeightDecorator` is bypassable.**

The ante chain does include `authante.NewTxTimeoutHeightDecorator()`: [7](#0-6) 

This decorator rejects a tx when `current_block > timeout_height`. However, if the attacker replaces `timeout_height=N` with `timeout_height=0` in the tx body, the decorator sees "no timeout" and passes. The signature check also passes because `timeout_height` is not in the EIP-712 schema.

---

### Impact Explanation

A user who signs a Cosmos tx via legacy EIP-712 with `timeout_height=N` intends the tx to be invalid after block N. Because the signature does not commit to `timeout_height`, an attacker who observes the signed tx (mempool, broadcast, or on-chain) can:

1. Wait until block N (the tx is rejected by `TxTimeoutHeightDecorator`, sequence is **not** incremented).
2. Reconstruct the tx with `timeout_height=0`, keeping the original `FeePayerSig`.
3. Submit the modified tx — it passes both `TxTimeoutHeightDecorator` (no timeout) and `VerifySignature` (same hash).
4. The tx executes, violating the user's intended expiry window.

This is an EIP-712 authorization bypass enabling replay of timed-out transactions — matching the **High** impact category: *"EIP-712 authorization bypass enabling replay, forged execution, or unauthorized account/code mutation."*

---

### Likelihood Explanation

- The legacy EIP-712 path is still reachable by any unprivileged user submitting a tx with `ExtensionOptionsWeb3Tx`.
- The attacker needs no special privileges — only the ability to observe a broadcast tx and submit a modified version.
- The window is any time after the original tx's `timeout_height` has passed and before the account's sequence is incremented by another tx.

---

### Recommendation

Add `timeout_height` back to the `Tx` type schema in both `extractMsgTypes` (`eip712_legacy.go`) and `createEIP712Types` (`types.go`):

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id",       Type: "string"},
    {Name: "fee",            Type: "Fee"},
    {Name: "memo",           Type: "string"},
    {Name: "msgs",           Type: "Msg[]"},
    {Name: "sequence",       Type: "string"},
    {Name: "timeout_height", Type: "string"},  // restore binding
},
```

This ensures the EIP-712 hash commits to `timeout_height`, making any modification to that field invalidate the signature.

---

### Proof of Concept

```
1. Construct a Cosmos tx T with:
     msgs = [MsgSend{from=Alice, to=Bob, amount=X}]
     timeout_height = 100
     sequence = alice_seq
   Sign it via legacy EIP-712 → FeePayerSig = S

2. Broadcast T. At block 101, TxTimeoutHeightDecorator rejects it.
   Alice's sequence is still alice_seq.

3. Attacker constructs T' identical to T except timeout_height = 0.
   Attacker sets FeePayerSig = S (unchanged).

4. VerifySignature on T':
     txBytes' = StdSignBytes(..., timeout_height=0, ...)
       → JSON has no "timeout_height" key (omitempty)
     LegacyWrapTxToTypedData → Tx schema has no timeout_height field
     sigHash(T') == sigHash(T) == H
     ethcrypto.VerifySignature(pubKey, H, S) → true ✓

5. TxTimeoutHeightDecorator sees timeout_height=0 → no expiry → passes ✓

6. T' executes: MsgSend transfers X from Alice to Bob,
   despite Alice's intent that the tx expire at block 100.
```
