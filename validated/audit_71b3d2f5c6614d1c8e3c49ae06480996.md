### Title
Deploy Syscall Unconditionally Reports Success Regardless of Constructor Revert, Enabling Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall handler in `syscall_impls.cairo` hardcodes `failure_flag=0` in the response header unconditionally, reporting deployment success to the calling contract even when the constructor execution reverts. Any contract that relies on the deploy response to detect failure will send funds to an address with no valid class hash, permanently freezing those funds with no recovery path.

---

### Finding Description

In `execute_deploy` (lines 534–539 of `syscall_impls.cairo`), after calling `deploy_contract`, the response header is written with a hardcoded success flag:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` function is invoked with the caller's implicit `revert_log`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}
``` [2](#0-1) 

When the constructor reverts, the revert log mechanism (defined in `revert.cairo`) unwinds the class hash assignment for the deployed address, leaving it uninitialized (`class_hash=0`). Despite this, the response written back to the calling contract always carries `failure_flag=0` and a valid `contract_address`. The calling contract has no mechanism to distinguish a reverted deployment from a successful one.

This is structurally identical to the M-02 pattern: just as `pauseAndWithdrawToPurple` simultaneously pauses the token (blocking recovery) and drains ETH (causing loss), `execute_deploy` simultaneously reverts the constructor (leaving the address unusable) while reporting success (causing the caller to fund the unusable address).

The `execute_deploy_account_transaction` path, by contrast, creates a fresh `revert_log` for the constructor:

```cairo
let revert_log = init_revert_log();
deploy_contract{revert_log=revert_log}(
    block_context=block_context, constructor_execution_context=constructor_execution_context
);
``` [3](#0-2) 

The syscall path shares no such isolation and does not return an `is_reverted` signal to the caller. The `contract_call_helper` function, which handles `call_contract` and `library_call`, correctly propagates `is_reverted` to the response header:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [4](#0-3) 

The `execute_deploy` path is the sole syscall that omits this propagation, confirmed by the developer TODO.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A contract address whose constructor reverted has `class_hash=0` (uninitialized). No entry point — including any withdrawal, rescue, or recovery function — can ever be dispatched to it. Any tokens or assets transferred to that address after the false-success response are irrecoverable. The freeze is permanent and protocol-level; no operator action can undo it once the state is committed.

---

### Likelihood Explanation

**Low to Medium.** The deploy-then-fund pattern is ubiquitous in DeFi (pool factories, escrow deployers, vault initializers). A constructor can revert due to out-of-gas, an explicit `assert`/`panic`, or attacker-controlled constructor calldata. The developer TODO comment confirms this is a known unimplemented code path, not an intentional design choice, meaning it is present in the current production OS program.

---

### Recommendation

`deploy_contract` must return an `is_reverted` flag. The `execute_deploy` response header must be written as:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

mirroring the pattern already used in `contract_call_helper`. If the constructor reverts, the `contract_address` field in `DeployResponse` should be set to `0` or the response should be structured to signal failure, consistent with how `call_contract` failures are communicated.

---

### Proof of Concept

1. Attacker declares class `MaliciousVault` whose constructor unconditionally reverts (e.g., `assert 0 = 1`).
2. Victim contract `Factory` calls the `deploy` syscall with `class_hash = MaliciousVault`.
3. OS executes `execute_deploy`:
   - `deploy_contract` runs the constructor; it reverts.
   - The revert log unwinds the class hash assignment; the deployed address remains `class_hash=0`.
   - The response header is written with `failure_flag=0` (hardcoded).
4. `Factory` reads `failure_flag=0`, concludes deployment succeeded, and calls `transfer(token, deployed_address, amount)` to seed the vault.
5. Tokens arrive at `deployed_address`. Since `class_hash=0`, no entry point dispatch is possible.
6. Funds are permanently frozen with no recovery path.
