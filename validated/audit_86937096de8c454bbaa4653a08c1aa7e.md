Audit Report

## Title
LP Burns Non-Zero Shares But Receives Zero Tokens Due to Scaled-to-Native Floor Rounding in `removeLiquidity` — (File: metric-core/contracts/libraries/LiquidityLib.sol)

## Summary

`LiquidityLib.removeLiquidity` permanently burns an LP's shares and decrements the pool's internal scaled balances in Phase 1, then converts the accumulated scaled withdrawal to native tokens using floor division in Phase 2. When the LP's proportional scaled claim is smaller than `token*ScaleMultiplier`, the floor rounds the native transfer to zero. The LP's position is irreversibly erased from storage while they receive nothing.

## Finding Description

**Phase 1 — irreversible state mutation:** The LP's proportional scaled claim is computed with floor division, and the bin's scaled balances plus the LP's share record are permanently updated: [1](#0-0) 

**Phase 2 — native conversion with floor:** The accumulated scaled totals are converted to native token units via `_deltasScaledToExternal` with `Math.Rounding.Floor`, and transfers are gated on `> 0`: [2](#0-1) 

Inside `_deltasScaledToExternal`, the floor branch performs integer division: [3](#0-2) 

`TOKEN_0_SCALE_MULTIPLIER` is `10^(max(18, decimals) − token0.decimals())`. For a USDC (6 dec) / WETH (18 dec) pool this equals `10^12`. [4](#0-3) 

Whenever `totalToken0ToRemoveScaled < 10^12`, `amount0Removed = 0` and no USDC is transferred, even though the LP's shares have already been erased.

The `addLiquidity` path uses `Math.Rounding.Ceil` for the same conversion, so the pool always collects at least as many native tokens as its scaled credit. The remove path has no symmetric protection. [5](#0-4) 

The `MinimalLiquidity` guard only prevents leaving a *surviving* dust position (`newUserShares > 0`); it does not prevent a full burn (`newUserShares == 0`) that yields zero native tokens: [6](#0-5) 

## Impact Explanation

**High — direct, permanent loss of LP principal.** An LP who calls `removeLiquidity` with a valid, non-dust share count can have their entire position erased from pool storage while receiving zero tokens in return. The forfeited scaled balance remains inside the pool and is silently redistributed to all remaining LPs through the unchanged `binTotalShares` denominator. There is no recovery path.

## Likelihood Explanation

The condition is reachable on any pool whose two tokens have different decimal counts — the most common real-world pairing (USDC/WETH, USDT/WBTC). In such pools `token0ScaleMultiplier ≥ 10^10`. An LP whose proportional scaled claim in a bin falls below that threshold — which occurs naturally when many LPs share a bin or when the bin's token balance is small — will silently receive nothing on withdrawal. No privileged role, no malicious setup, and no non-standard token is required; the trigger is a normal `removeLiquidity` call by any LP.

## Recommendation

Add a revert guard immediately before the transfer block that fires whenever the scaled claim is non-zero but the native amount rounds to zero:

```solidity
if (totalToken0ToRemoveScaled > 0 && amount0Removed == 0) revert RemovalRoundsToZero();
if (totalToken1ToRemoveScaled > 0 && amount1Removed == 0) revert RemovalRoundsToZero();
```

Alternatively, mirror the `addLiquidity` asymmetry by using `Math.Rounding.Ceil` for the remove conversion — this slightly over-pays the LP (by at most 1 native unit) but preserves the invariant that a non-zero scaled claim always yields a non-zero native transfer. The pool's solvency buffer from the `addLiquidity` ceil-rounding absorbs this.

## Proof of Concept

**Setup:** USDC (token0, 6 dec) / WETH (token1, 18 dec) pool. `token0ScaleMultiplier = 10^12`, `minimalMintableLiquidity = 1000`.

1. **Alice** adds `1_000` shares to bin 4 (above current price, token0-only bin).
   - Bin is empty → `amount0Scaled = ceil(10^12 × 1000 / 1e18) = 1000`, `amount0Added = ceil(1000 / 10^12) = 1` USDC.
   - State: `binState.token0BalanceScaled = 1000`, `binTotalShares[4] = 1000`, Alice's shares = 1000.

2. **Bob** adds `10^9` shares to the same bin.
   - `amount0Scaled = ceil(1000 × 10^9 / 1000) = 10^9`.
   - State: `binState.token0BalanceScaled ≈ 10^9 + 1000`, `binTotalShares[4] = 10^9 + 1000`.

3. **Alice** calls `removeLiquidity` to burn all 1000 shares.
   - `amount0Scaled = (10^9 + 1000) × 1000 / (10^9 + 1000) = 1000`.
   - `amount0Removed = 1000 / 10^12 = 0` (floor).
   - Alice's 1000 shares are erased from `positionBinShares`; she receives **0 USDC**.
   - Bob's effective claim on the bin increases by Alice's forfeited 1000 scaled units.

Alice has permanently lost her 1 USDC deposit with no recourse.
