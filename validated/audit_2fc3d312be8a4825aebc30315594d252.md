Audit Report

## Title
Unguarded `refundETH` allows any caller to steal excess ETH left on the router by a prior user — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.refundETH()` transfers the router's entire native ETH balance to `msg.sender` with no access control or ownership check. When a user calls `multicall{value: X}` with `amountIn < X` and omits the `refundETH` step, the unspent ETH remains on the router. Any subsequent caller can invoke `refundETH()` and receive that ETH, causing direct loss of the original user's principal.

## Finding Description
`refundETH()` reads `address(this).balance` and unconditionally transfers the full amount to `msg.sender`: [1](#0-0) 

The `pay()` function, when `token == WETH` and `nativeBalance >= value`, wraps exactly `value` wei and leaves the remainder as native ETH on the contract: [2](#0-1) 

The `receive()` guard only blocks plain ETH transfers (no calldata); it does not intercept `msg.value` attached to function calls like `multicall{value: X}(...)`: [3](#0-2) 

`multicall` uses `delegatecall`, so `msg.value` is visible to all sub-calls, and any unspent ETH remains on the router after the outer call returns: [4](#0-3) 

The test suite confirms the intended pattern — users must include `refundETH` as the last call in their multicall batch — but this is not enforced at the contract level: [5](#0-4) 

**Exploit path:**
1. User calls `router.multicall{value: 1 ether}([exactInputSingle(..., amountIn=0.5 ether, tokenIn=WETH)])` — omitting `refundETH`.
2. `pay()` wraps 0.5 ETH → WETH and sends it to the pool. The remaining 0.5 ETH stays on the router.
3. Attacker calls `router.refundETH()`. The function reads `address(this).balance == 0.5 ETH` and calls `_transferETH(msg.sender, 0.5 ETH)`, sending the full balance to the attacker.

No existing guard prevents step 3. The `receive()` check is irrelevant here because `refundETH` is a normal external call, not a plain ETH transfer.

## Impact Explanation
Direct loss of user principal. The victim permanently loses the excess ETH they sent with their swap transaction. The attacker gains it with no privileged role required. This meets the Sherlock Critical/High threshold for direct loss of user funds by an unprivileged caller.

## Likelihood Explanation
The vulnerability requires only that a user sends `msg.value > amountIn` without appending `refundETH` to their multicall batch. This is a realistic user error — the pattern is not enforced at the contract level. An attacker monitoring the mempool or the router's ETH balance can front-run or immediately follow any such transaction. The attack is fully permissionless and repeatable.

## Recommendation
Record the top-level `msg.sender` in transient storage at the start of `multicall` and require `refundETH` to send to that stored address rather than the immediate `msg.sender`. Alternatively, add a reentrancy-style flag set only during `multicall` execution and revert in `refundETH` if the flag is not set, ensuring `refundETH` can only be reached as a sub-call within a `multicall` batch initiated by the rightful depositor.

## Proof of Concept
```
1. User calls router.multicall{value: 1 ether}([
       abi.encodeCall(router.exactInputSingle, (ExactInputSingleParams({
           tokenIn: WETH, amountIn: 0.5 ether, ...
       })))
       // refundETH omitted
   ]);
   // pay() wraps 0.5 ETH → WETH, sends to pool. 0.5 ETH remains on router.
   // address(router).balance == 0.5 ETH

2. Attacker calls router.refundETH();
   // balance = 0.5 ETH > 0 → _transferETH(attacker, 0.5 ETH)
   // Attacker receives 0.5 ETH. User's excess ETH is gone.
```
A Foundry test can reproduce this by adapting `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` — omit `calls[1]` (the `refundETH` call), assert `address(router).balance > 0` after the user's multicall, then call `router.refundETH()` from a different address and assert that address received the balance.
