### Title
Transfer Permanently Locked When Fee Reduces Net Amount Below Decimal Normalization Threshold — (`near/omni-bridge/src/lib.rs`)

---

### Summary

The `init_transfer` and `update_transfer_fee` entry points only enforce `fee < amount`, but `sign_transfer` additionally requires `normalize_amount(amount - fee) > 0`. Because `normalize_amount` performs floor division by `10^(origin_decimals - decimals)`, a user can create or update a transfer whose `amount_without_fee` is non-zero yet normalizes to zero. Once in that state, `sign_transfer` always panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`, and there is no cancel/refund path, so the locked tokens are irrecoverable.

---

### Finding Description

**Root cause — constraint mismatch across three functions:**

**1. `init_transfer` validation** (`near/omni-bridge/src/lib.rs`, line 554–557):
```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```
Only checks `fee < amount`. Does not check that the post-normalization net amount is positive. [1](#0-0) 

**2. `update_transfer_fee` validation** (`near/omni-bridge/src/lib.rs`, line 399–402):
```rust
require!(
    fee.fee >= current_fee.fee && fee.fee < transfer.message.amount,
    BridgeError::InvalidFee.as_ref()
);
```
Allows the sender to monotonically increase the fee up to `amount - 1`. No check that `normalize_amount(amount - new_fee) > 0`. Fees can never be decreased. [2](#0-1) 

**3. `sign_transfer` hard requirement** (`near/omni-bridge/src/lib.rs`, line 475–485):
```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```
`normalize_amount` performs floor division: `(amount - fee) / 10^(origin_decimals - decimals)`. If `amount - fee < 10^(origin_decimals - decimals)`, the result is 0 and the call panics. [3](#0-2) 

**4. `normalize_amount` uses floor division** (`near/omni-bridge/src/lib.rs`, line 2784–2787):
```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [4](#0-3) 

**5. No cancel/refund path exists for pending transfers.** Once a `TransferMessage` is stored in `pending_transfers`, it can only be removed via `sign_transfer_callback` (when fee is zero), `claim_fee_callback` (requires a destination-chain proof), or `process_fin_transfer_to_near`. None of these are reachable if `sign_transfer` always panics. [5](#0-4) 

---

### Impact Explanation

Once a transfer reaches the state where `normalize_amount(amount - fee) == 0`:

- `sign_transfer` always panics — the MPC signing call is never made.
- No signature is ever produced, so the destination chain never receives a finalization message.
- `claim_fee` requires a proof that the transfer was finalized on the destination chain — impossible without a signature.
- There is no `cancel_transfer` or refund function.

The user's tokens are locked in the NEAR bridge contract permanently with no recovery path. This matches the **Critical/High** impact class: *Permanent freezing, irrecoverable lock of user funds in bridge flows*. [6](#0-5) 

---

### Likelihood Explanation

**Trigger path 1 — via `init_transfer`:**
A user transfers a token with `origin_decimals = 24`, `decimals = 18` (6-decimal gap, divisor = 10^6). They send `amount = 1_000_001` with `fee = 1_000_000`. The check `fee < amount` passes (1_000_000 < 1_000_001). But `amount_without_fee = 1`, and `normalize_amount(1) = 0`. The transfer is stored and immediately unsignable.

**Trigger path 2 — via `update_transfer_fee`:**
A user initiates a valid transfer (e.g., `amount = 2_000_000`, `fee = 0`). Later they call `update_transfer_fee` to raise the fee to `1_999_999` (valid: `fee < amount`). Now `amount_without_fee = 1`, `normalize_amount(1) = 0`. The transfer is permanently stuck.

Both paths are reachable by an unprivileged bridge user (the token sender). The 6-decimal gap between NEAR (24 decimals) and EVM (18 decimals) is the standard configuration for NEAR-native tokens bridged to EVM chains, making this a realistic scenario. [4](#0-3) 

---

### Recommendation

Add a post-normalization check in both `init_transfer` and `update_transfer_fee` to ensure the net amount after fee deduction normalizes to a positive value on the destination chain:

```rust
// In init_transfer, after computing transfer_message:
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(token_address) = token_address {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee().unwrap_or(0),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

Apply the same guard in `update_transfer_fee` before persisting the new fee. Alternatively, document the minimum transfer amount per token and enforce it at the entry points.

### Proof of Concept

**Setup:** Token registered with `origin_decimals = 24`, `decimals = 18` (standard NEAR→EVM configuration).

**Step 1:** User calls `ft_transfer_call` with `amount = 1_000_001` and `InitTransferMsg { fee: 1_000_000, ... }`.

**Step 2:** `init_transfer` checks `fee (1_000_000) < amount (1_000_001)` → passes. Transfer stored in `pending_transfers`.

**Step 3:** Relayer calls `sign_transfer`:
- `amount_without_fee() = 1_000_001 - 1_000_000 = 1`
- `normalize_amount(1, {origin: 24, dest: 18}) = 1 / 10^6 = 0`
- `require!(0 > 0, ...)` → **panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`**

**Step 4:** No signature is produced. No destination-chain proof can ever be generated. `claim_fee` is unreachable. The 1,000,001 NEAR-side token units are permanently locked in the bridge. [3](#0-2) [2](#0-1) [1](#0-0)
