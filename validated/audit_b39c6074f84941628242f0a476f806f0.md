### Title
Recipient-Controlled Revert in `finTransfer` Permanently Locks Source-Chain Funds — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer()` uses a push-based delivery pattern for native ETH and ERC-1155 tokens. If the MPC-signed recipient is a contract that rejects ETH or lacks an `onERC1155Received` hook, every finalization attempt reverts. Because the recipient is immutably encoded in the signed payload and no cancel/refund path exists on the source chain, the sender's bridged funds are permanently frozen.

---

### Finding Description

In `OmniBridge.finTransfer()`, the destination nonce is marked used **before** the token delivery:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287
```

Then, for native ETH:

```solidity
(bool success, ) = payload.recipient.call{value: payload.amount}("");
if (!success) revert FailedToSendEther();              // lines 319-322
```

And for ERC-1155:

```solidity
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,
    multiToken.tokenId,
    payload.amount,
    ""
);                                                      // lines 324-330
```

If `payload.recipient` is a contract without a `receive()`/`fallback()` function (ETH case) or without an `IERC1155Receiver` implementation (ERC-1155 case), the delivery call reverts. Because Solidity reverts roll back the entire transaction, `completedTransfers[payload.destinationNonce]` is also rolled back — the nonce is never consumed.

However, the recipient address is part of the MPC-signed `TransferMessagePayload`. No relayer can alter it. Every subsequent `finTransfer` call with the same payload will fail identically.

On the NEAR side, the source-chain funds are locked inside `pending_transfers` (for NEAR-origin transfers) or burned/locked on the origin EVM chain. The NEAR bridge exposes no public `cancel_transfer` or sender-initiated refund function. `remove_transfer_message` is only reachable through `sign_transfer_callback` (fee-zero path) or `claim_fee_callback`, neither of which is callable by the original sender to recover stuck funds. ---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

A user who bridges native ETH (or ERC-1155 tokens) to a contract recipient that rejects the push-delivery loses their funds with no on-chain recovery path. The source-chain lock is permanent: the NEAR bridge has no cancel/refund entrypoint for the sender, and the EVM finalization can never succeed for that signed payload.

This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Medium.** The scenario arises naturally in several realistic situations:

1. **Contract wallets / multisigs / DAOs** — a user specifies a Gnosis Safe or DAO treasury as recipient; many such contracts do not implement a plain `receive()` or `IERC1155Receiver`.
2. **ERC-1155 recipients** — `safeTransferFrom` mandates `onERC1155Received` on any contract recipient; contracts that do not implement it will always revert.
3. **Intentional griefing** — a recipient deploys or upgrades a contract to reject ETH after the transfer is initiated on the source chain, permanently blocking finalization.

No privileged access is required. Any bridge user who initiates a transfer to a non-accepting contract address triggers the condition.

---

### Recommendation

Replace the push-based delivery with a **pull-based (claim) pattern**:

- On `finTransfer`, record `claimable[recipient][token] += amount` instead of pushing immediately.
- Expose a `claim(token, amount)` function that the recipient calls to withdraw.
- This eliminates the revert surface entirely and matches the mitigation recommended in M-11.

Additionally, add a **source-chain cancellation path** on NEAR: allow the original sender to cancel a `pending_transfer` after a timeout if no valid finalization proof has been submitted, returning the locked/burned tokens.

---

### Proof of Concept

1. Alice initiates a NEAR → EVM bridge transfer of native ETH, specifying `recipient = 0xDeadContract` (a contract with no `receive()` function).
2. NEAR bridge locks Alice's funds in `pending_transfers`.
3. Relayer calls `OmniBridge.finTransfer(signatureData, payload)` where `payload.tokenAddress == address(0)` and `payload.recipient == 0xDeadContract`.
4. `completedTransfers[payload.destinationNonce] = true` executes.
5. `0xDeadContract.call{value: payload.amount}("")` returns `success = false`.
6. `revert FailedToSendEther()` rolls back the entire transaction, including step 4.
7. Relayer retries — identical result every time.
8. Alice's funds remain locked in NEAR `pending_transfers` indefinitely; no public cancel/refund function exists. [4](#0-3) [5](#0-4) [6](#0-5)

    fn remove_transfer_message(&mut self, transfer_id: TransferId) -> TransferMessage {
        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);
        }

        transfer.message
    }
