### Title
Solana Bridge Pause Mechanism Is Completely Ineffective — All Critical User Instructions Bypass Pause State (`solana/programs/bridge_token_factory/src/instructions/user/`)

---

### Summary

The Solana bridge program implements a pause mechanism via the `config.paused` field, settable by admins through the `pause` instruction. However, none of the critical user-facing instructions — `finalize_transfer`, `finalize_transfer_sol`, `init_transfer`, `init_transfer_sol`, and `deploy_token` — check the pause state before executing. The pause mechanism is therefore completely inert for the Solana bridge: any user can mint, release, lock, burn, or deploy tokens regardless of whether the bridge has been paused.

---

### Finding Description

The `pause` instruction in `solana/programs/bridge_token_factory/src/instructions/admin/pause.rs` sets `config.paused = ALL_PAUSED`: [1](#0-0) 

The `Config` account is loaded in every user instruction (directly or via the `WormholeCPI` struct), but no instruction enforces a constraint or runtime check against `config.paused`. Examining each critical instruction:

**`finalize_transfer.rs`** — mints bridged tokens or releases locked tokens from the vault. The `config` account is loaded with only a seeds/bump constraint; `process()` never reads `config.paused`: [2](#0-1) [3](#0-2) 

**`finalize_transfer_sol.rs`** — releases native SOL from the vault to the recipient. Same pattern: config loaded, no pause check: [4](#0-3) [5](#0-4) 

**`init_transfer.rs`** — locks or burns user tokens. No pause check in `process()`: [6](#0-5) 

**`init_transfer_sol.rs`** — locks native SOL. No pause check: [7](#0-6) 

**`deploy_token.rs`** — deploys a new bridged token mint. No pause check: [8](#0-7) 

This is structurally identical to the external report: the contract inherits/implements a pause mechanism but the mechanism is never wired into the functions that actually move assets, making the pause completely ineffective.

---

### Impact Explanation

**Critical.** When the bridge is paused — the primary use case being an emergency response to a compromised MPC key, a proof-verification exploit, or a supply discrepancy — the following remain fully callable by any unprivileged user:

- `finalize_transfer`: mints unbacked bridged tokens or releases locked tokens from the vault, enabling unauthorized asset release or mint even after the admin has attempted to halt the bridge.
- `finalize_transfer_sol`: releases native SOL from the program-controlled vault to arbitrary recipients.
- `init_transfer` / `init_transfer_sol`: locks or burns user funds into the bridge, creating pending transfers that cannot be settled (since the other side is paused), permanently freezing user funds.
- `deploy_token`: deploys new token mints, potentially polluting the token registry during an incident.

The most severe scenario: an attacker exploiting a compromised MPC key submits fraudulent `finalize_transfer` instructions. The admin pauses the bridge. The attacker continues calling `finalize_transfer` unimpeded, minting unbacked tokens or draining the vault, because the pause is never enforced.

---

### Likelihood Explanation

High. The pause mechanism exists precisely for emergency use. Any attacker who has already obtained a fraudulent signed payload (e.g., via a compromised MPC key or a replayed nonce) can continue to drain the bridge after the admin triggers a pause, since the pause check is absent. No special privilege is required beyond submitting a valid (or fraudulently obtained) signed payload.

---

### Recommendation

Add a pause check at the start of every critical user instruction's `process()` function. For example, in `finalize_transfer.rs`:

```rust
impl FinalizeTransfer<'_> {
    pub fn process(&mut self, data: FinalizeTransferPayload) -> Result<()> {
        require!(
            self.config.paused & PAUSED_FIN_TRANSFER == 0,
            ErrorCode::Paused
        );
        // ... rest of logic
    }
}
```

Apply the equivalent check to `finalize_transfer_sol`, `init_transfer`, `init_transfer_sol`, and `deploy_token`, using the appropriate pause flag for each operation. The `Config` account is already loaded in all of these instructions, so no structural change is needed — only the runtime guard.

---

### Proof of Concept

1. Admin calls `pause` instruction → `config.paused = ALL_PAUSED`.
2. Attacker (or any user) constructs a `finalize_transfer` instruction with a valid `SignedPayload<FinalizeTransferPayload>` (e.g., obtained before the pause, or via a compromised MPC key).
3. The instruction loads `config` (seeds/bump check passes), calls `UsedNonces::use_nonce`, then calls `mint_to` or `transfer_checked` — no code path reads `config.paused`.
4. Tokens are minted or released to the attacker's account despite the bridge being in a fully paused state.
5. The admin's emergency pause has zero effect on the Solana side of the bridge. [9](#0-8) [1](#0-0)
