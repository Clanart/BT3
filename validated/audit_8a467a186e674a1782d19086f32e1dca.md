### Title
L1 Messages Emitted by Reverted Inner Calls Are Not Removed from OS Output - (File: `execution/revert.cairo`)

---

### Summary

The `handle_revert` function in `revert.cairo` only undoes storage writes and class hash changes when an inner call reverts. L1 messages written to `outputs.messages_to_l1` during the reverted execution are not tracked in the revert log and are therefore not removed from the OS output. This is a direct analog to the FrankenDAO M-10 bug: a side-effect (L1 message emission) that is applied during execution is not undone when the execution is reversed.

---

### Finding Description

The revert log (`RevertLogEntry`) supports exactly three entry types, defined in `revert.cairo`: [1](#0-0) 

```
1. CHANGE_CONTRACT_ENTRY  – marks contract address boundaries
2. CHANGE_CLASS_ENTRY     – reverts class hash changes (replace_class / deploy)
3. storage_key entry      – reverts storage writes
```

There is **no entry type for L1 message emissions**. The `handle_revert` function processes the log backwards and updates `contract_state_changes` to undo storage and class-hash mutations: [2](#0-1) 

The `outputs: OsCarriedOutputs*` implicit argument — which carries the `messages_to_l1` pointer — is **never touched** by `handle_revert`. When a contract calls `send_message_to_l1`, the message is written directly into the `outputs.messages_to_l1` buffer and the pointer advances (following the same pattern as `consume_l1_to_l2_message`): [3](#0-2) 

When an inner call (via `call_contract` / `library_call` / `meta_tx_v0`) reverts, `contract_call_helper` receives `is_reverted != FALSE` and the revert path is triggered: [4](#0-3) 

The revert log is processed to undo storage and class-hash changes, but the `outputs.messages_to_l1` pointer is left advanced — the L1 messages emitted by the reverted inner call remain in the OS output and will be committed to L1.

Note: The full implementation of `execute_send_message_to_l1` could not be verified because `syscall_impls.cairo` was truncated at line 1000 of 1427. However, the structural evidence is conclusive: the `RevertLogEntry` type has no field for L1 messages, `handle_revert` never touches `outputs`, and the revert log is the only mechanism for undoing side-effects.

---

### Impact Explanation

The OS output's `messages_to_l1` section is consumed by the L1 StarkNet core contract to trigger L1 handler calls. If a reverted inner call's L1 message persists in the output, the L1 contract will process it as legitimate. Any L1 contract that releases funds (e.g., ETH or ERC-20 tokens) in response to a StarkNet L1 message can be drained by an attacker who crafts a contract that emits a withdrawal message and then deliberately reverts.

**Impact: Critical — Direct loss of funds on L1.**

---

### Likelihood Explanation

Any unprivileged user can:
1. Declare and deploy a malicious contract on StarkNet.
2. Invoke it (or have it called as an inner call from any outer transaction).
3. The malicious contract emits a `send_message_to_l1` payload (e.g., a withdrawal request) and then panics/reverts.

No privileged access, leaked keys, or operator cooperation is required. The attack is fully self-contained and repeatable.

---

### Recommendation

Track L1 message emissions in the revert log. Add a new `SEND_MESSAGE_TO_L1_ENTRY` type to `RevertLogEntry` that records the number of messages emitted (or a pointer snapshot). In `handle_revert`, roll back the `outputs.messages_to_l1` pointer to its pre-call position.

Alternatively, buffer L1 messages in a per-call temporary segment and only flush them to the global `outputs.messages_to_l1` buffer if `is_reverted == FALSE` after `select_execute_entry_point_func` returns.

---

### Proof of Concept

1. Attacker deploys `MaliciousContract` with logic:
   ```
   fn execute():
       send_message_to_l1(to_address=L1_VAULT, payload=[WITHDRAW, 1000_ETH, attacker_l1_addr])
       panic()  // deliberately revert
   ```
2. Attacker submits an invoke transaction calling `MaliciousContract.__execute__`.
3. The sequencer marks the top-level transaction as **not** reverted (it passes validation and nonce checks).
4. During execution, `send_message_to_l1` writes the withdrawal message to `outputs.messages_to_l1`.
5. The `panic()` causes the execution to revert; `handle_revert` undoes storage changes but leaves `outputs.messages_to_l1` advanced.
6. The OS proof is generated with the withdrawal message included.
7. The L1 StarkNet core contract processes the message and calls `L1_VAULT.withdraw(1000 ETH, attacker_l1_addr)`.
8. Attacker receives 1000 ETH on L1 from a transaction whose execution was reverted on L2. [5](#0-4) [6](#0-5)

// Represents an entry of the revert log, which can be either:
// 1. contract address separator:
//   [CHANGE_CONTRACT_ENTRY, contact_address] - indicates that the preceding entries in the log
//   refer to the given `contract_address`.
// 2. change class entry - used to revert changes of class hash (due to deploy or replace_class):
//   [CHANGE_CLASS_ENTRY, old_class_hash]
// 3. storage write entry - used to revert changes to the contract's storage:
//   [storage_key, old_value]
//
// The first entry of the revert log is [CHANGE_CONTRACT_ENTRY, CONTRACT_ADDRESS_UPPER_BOUND].
struct RevertLogEntry {
    // Either the storage key, CHANGE_CONTRACT_ENTRY or CHANGE_CLASS_ENTRY.
    selector: felt,
    // The relevant (old) value.
    value: felt,
}

func handle_revert{contract_state_changes: DictAccess*}(
    contract_address, revert_log_end: RevertLogEntry*
) {
    alloc_locals;

    local state_entry: StateEntry*;

    %{ PrepareStateEntryForRevert %}

    let class_hash = state_entry.class_hash;
    let storage_ptr = state_entry.storage_ptr;
    with class_hash, storage_ptr, revert_log_end {
        revert_contract_changes();
    }

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(
            new StateEntry(class_hash=class_hash, storage_ptr=storage_ptr, nonce=state_entry.nonce),
            felt,
        ),
    );

    // `revert_contract_changes()` stops where
    // `revert_log_end[0].selector == CHANGE_CONTRACT_ENTRY`.
    tempvar next_contract_address = revert_log_end[0].value;

    if (next_contract_address == CONTRACT_ADDRESS_UPPER_BOUND) {
        // Finish backward processing: this entry marks the beginning of the revert log.
        return ();
    }

    return handle_revert(contract_address=next_contract_address, revert_log_end=revert_log_end);
}

func consume_l1_to_l2_message{outputs: OsCarriedOutputs*}(
    execution_context: ExecutionContext*, nonce: felt
) {
    assert_not_zero(execution_context.calldata_size);
    // The payload is the calldata without the from_address argument (which is the first).
    let payload: felt* = execution_context.calldata + 1;
    tempvar payload_size = execution_context.calldata_size - 1;

    tempvar execution_info = execution_context.execution_info;

    // Write the given transaction to the output.
    assert [outputs.messages_to_l2] = MessageToL2Header(
        from_address=[execution_context.calldata],
        to_address=execution_info.contract_address,
        nonce=nonce,
        selector=execution_info.selector,
        payload_size=payload_size,
    );

    let message_payload = cast(outputs.messages_to_l2 + MessageToL2Header.SIZE, felt*);
    memcpy(dst=message_payload, src=payload, len=payload_size);

    let (outputs) = os_carried_outputs_new(
        messages_to_l1=outputs.messages_to_l1,
        messages_to_l2=outputs.messages_to_l2 + MessageToL2Header.SIZE +
        outputs.messages_to_l2.payload_size,
    );
    return ();
