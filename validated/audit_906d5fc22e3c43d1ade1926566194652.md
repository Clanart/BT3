Audit Report

## Title
Unspent ETH Stranded in Router After Partial Fill in `exactInputSingle` When Price Limit Is Hit — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

## Summary

`exactInputSingle` is `payable` and `PeripheryPayments.pay` wraps native ETH to WETH during the swap callback. When a caller sends ETH with `tokenIn = WETH` and a non-trivial `priceLimitX64`, the pool may partially fill the swap. The callback pays only the actual consumed amount; the remainder stays in the router. Because `refundETH` is permissionless, any third party can immediately drain the stranded ETH.

## Finding Description

**Step 1 – Native ETH wrapping is live in `pay`.**

`PeripheryPayments.pay` checks `address(this).balance` when `token == WETH` and `payer != address(this)`, wrapping and forwarding native ETH to the pool: [1](#0-0) 

A caller who sends ETH with `tokenIn = WETH` will have that ETH wrapped and forwarded during the callback.

**Step 2 – `_justPayCallback` pays only the actual consumed delta.** [2](#0-1) 

`extractPositiveAmount` returns only the amount the pool actually consumed. If the pool partially fills (price limit hit), the callback wraps and forwards only that partial amount; the difference between `msg.value` and the partial amount remains as raw ETH in the router.

**Step 3 – `exactInputSingle` has no post-swap ETH refund and no partial-fill guard.** [3](#0-2) 

After the swap, only the output minimum is checked and transient context is cleared. There is no `refundETH()` call and no check that the full `amountIn` was consumed. Compare with `exactInput` (multi-hop), which explicitly reverts on partial fills: [4](#0-3) 

`exactInputSingle` has no equivalent guard, so a partial fill silently succeeds and leaves ETH stranded.

**Step 4 – `refundETH` is permissionless.** [5](#0-4) 

Any address can call this and receive all ETH currently held by the router, including the victim's unspent ETH.

## Impact Explanation

A user calling `exactInputSingle` with `tokenIn = WETH`, sending `N` ETH as `msg.value`, and setting a `priceLimitX64` that is reached mid-swap will receive fewer output tokens than `N` ETH would buy, and lose the unspent ETH portion to any caller of `refundETH()`. This is a direct, permanent loss of user principal with no recovery path — qualifying as High severity under Sherlock thresholds.

## Likelihood Explanation

Setting a price limit is a standard, documented feature of the router interface. Sending native ETH instead of pre-wrapped WETH is the natural UX for most users. MEV bots routinely monitor for stranded ETH in router contracts and call `refundETH()` atomically. No privileged access or malicious setup is required; any ordinary user transaction with a price limit and native ETH triggers the loss.

## Recommendation

Add an automatic ETH refund at the end of `exactInputSingle` (and `exactOutputSingle`) when native ETH may have been used:

```solidity
_clearExpectedCallbackPool();
if (address(this).balance > 0) {
    _transferETH(msg.sender, address(this).balance);
}
```

Alternatively, add a partial-fill guard analogous to `exactInput`'s `InvalidInputAmountAtHop` check, reverting if the actual consumed input is less than `params.amountIn`. Document that callers must wrap calls in a `multicall` appending `refundETH()` whenever native ETH is sent.

## Proof of Concept

```
1. Alice calls exactInputSingle({
       pool: ETH/T pool,
       tokenIn: WETH,
       zeroForOne: true,
       amountIn: 1e18,
       amountOutMinimum: 0,
       priceLimitX64: P   // non-trivial limit
   }, { value: 1e18 });
2. Pool fills only 0.6 ETH before hitting price P.
3. metricOmmSwapCallback fires; pay() wraps and forwards 0.6e18 ETH to the pool.
4. exactInputSingle returns; 0.4e18 ETH remains in the router.
5. Alice receives tokens worth 0.6 ETH but has spent 1 ETH.
6. Bob (MEV bot) calls router.refundETH() and receives 0.4e18 ETH.
7. Alice has permanently lost 0.4 ETH with no recourse.
```
