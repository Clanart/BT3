### Title
Native ETH Overpayment Not Refunded After WETH-Path Swaps, Excess Stealable via `refundETH()` — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

When a user calls `exactOutputSingle` (or any payable swap function) with WETH as `tokenIn` and sends ETH via `msg.value`, the `pay()` function in `PeripheryPayments` wraps only the exact amount the pool demands, leaving any excess ETH silently stranded in the router. No automatic refund is issued. Because `refundETH()` sends the entire contract ETH balance to `msg.sender` — not to the original payer — any third party can front-run the user's refund call and steal the surplus ETH.

---

### Finding Description

**Root cause — `pay()` wraps only the exact pool-demanded amount:** [1](#0-0) 

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();          // wraps exactly `value`
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
```

When `nativeBalance > value`, the branch at line 75 wraps exactly `value` wei and transfers it to the pool. The remainder `nativeBalance − value` is never touched — it stays in the router.

**No post-swap refund in any swap entry point:**

`exactOutputSingle` ends with a slippage check and `_clearExpectedCallbackPool()` — no ETH refund: [2](#0-1) 

The same is true for `exactInputSingle`, `exactInput`, and `exactOutput`.

**`refundETH()` sends to `msg.sender`, not the original payer:** [3](#0-2) 

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // ← sends to caller, not original payer
    }
}
```

Any address that calls `refundETH()` after a swap that left residual ETH in the router receives the full balance. There is no access control, no payer tracking, and no per-user accounting.

---

### Impact Explanation

**Direct loss of user ETH principal.** A user who calls `exactOutputSingle{value: amountInMaximum}(...)` with WETH as `tokenIn` sends up to `amountInMaximum` ETH. The pool determines the actual `amountIn ≤ amountInMaximum` at execution time. The surplus `amountInMaximum − amountIn` is left in the router and is immediately claimable by any address via `refundETH()`. The user cannot atomically recover it unless they explicitly bundle a `refundETH()` call in the same multicall — a step that is not enforced or documented in the function interface.

---

### Likelihood Explanation

**Medium.** The scenario is triggered by any user who:
1. Calls an exact-output swap with WETH as `tokenIn` and sends ETH as `msg.value` (the standard pattern when the exact input cost is unknown at submission time), or
2. Calls an exact-input swap and sends slightly more ETH than `params.amountIn` as a buffer.

A mempool-watching bot can detect the swap transaction, observe the residual ETH balance, and immediately call `refundETH()` in the next block (or via a bundle) to drain it. No special privileges are required.

---

### Recommendation

1. **Automatic refund at swap exit.** At the end of each payable swap function, if `address(this).balance > 0`, transfer the remainder back to `msg.sender`:
   ```solidity
   if (address(this).balance > 0) _transferETH(msg.sender, address(this).balance);
   ```
2. **Or restrict `refundETH()` to the original payer.** Store the payer address in transient storage at swap entry and require `msg.sender == storedPayer` inside `refundETH()`.
3. **Or enforce multicall usage.** If the intended pattern is `multicall([exactOutputSingle(...), refundETH()])`, enforce it at the contract level (e.g., revert if ETH remains after a swap without a bundled refund call).

---

### Proof of Concept

1. Alice calls `exactOutputSingle{value: 1 ether}(params)` where `params.tokenIn = WETH`, `params.amountInMaximum = 1 ether`, `params.amountOut = X`.
2. The pool executes the swap and calls `metricOmmSwapCallback`.
3. `_justPayCallback` calls `pay(WETH, Alice, pool, 0.8 ether)` — the pool only needed 0.8 ETH.
4. `pay()` enters the `nativeBalance >= value` branch: wraps 0.8 ETH → sends 0.8 WETH to pool. The remaining **0.2 ETH stays in the router**.
5. `exactOutputSingle` returns; no refund is issued.
6. Bob observes the router's ETH balance (0.2 ETH) and calls `refundETH()`.
7. `refundETH()` sends `address(this).balance` (0.2 ETH) to Bob.
8. **Alice loses 0.2 ETH with no recourse.**
