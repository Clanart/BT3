### Title
Relative Timelock Bypass via Integer Overflow in `check_time_locks` (`nowrap=false` Path) — (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary
When `check_time_locks` is called with `nowrap=false` (the pre-soft-fork legacy path), the relative timelock checks for `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE` use `wrapping_add` instead of `saturating_add`. An unprivileged coin owner can craft a puzzle with a maximally-large relative timelock value (e.g., `0xffff_ffff` for height, `0xffff_ffff_ffff_ffff` for seconds) that wraps the addition to a value smaller than the current block height or timestamp, causing the timelock guard to pass immediately. A coin that should be unspendable for billions of blocks or seconds becomes spendable at any height.

### Finding Description
In `check_time_locks`, the `nowrap` flag selects between two arithmetic modes for relative conditions:

```rust
// nowrap=true (safe path)
if prev_transaction_block_height
    < unspent.confirmed_block_index.saturating_add(height_relative)
{ ... }

// nowrap=false (legacy path — VULNERABLE)
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{ ... }
``` [1](#0-0) 

When `nowrap=false`, `confirmed_block_index.wrapping_add(height_relative)` overflows for large `height_relative` values. For example:

- `confirmed_block_index = 100`, `height_relative = 0xffff_ffff`
- `100u32.wrapping_add(0xffff_ffff) = 99` (wraps to 99)
- Guard: `prev_height < 99` → **false** at any block height ≥ 99 → **no error, spend accepted**

The same overflow applies to `seconds_relative` using `u64::wrapping_add`: [2](#0-1) 

The test suite explicitly documents and confirms this behavior:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),   // <-- bypass confirmed
)]
``` [3](#0-2) 

The Python binding exposes `nowrap` as a caller-controlled parameter, meaning the Python full node decides which arithmetic mode to apply per block: [4](#0-3) 

The Python test suite also confirms the wrapping bypass is reachable and produces `None` (no error) for overflow inputs when `nowrap=False`: [5](#0-4) 

A symmetric but opposite flaw exists for `before_height_relative` and `before_seconds_relative`: wrapping causes a large "before" deadline to collapse to a small value, making the coin permanently unspendable (a coin-locking DoS against the coin owner): [6](#0-5) 

### Impact Explanation
This is a **timelock validation bypass**. A coin puzzle embedding `(ASSERT_HEIGHT_RELATIVE 0xffff_ffff)` or `(ASSERT_SECONDS_RELATIVE 0xffff_ffff_ffff_ffff)` is intended to be unspendable for billions of blocks/seconds. Under the `nowrap=false` path, the overflow causes the guard to pass immediately, allowing the coin to be spent at any block height. This matches the allowed High impact: *"timelock or coin-id validation bypass enables unauthorized spend acceptance."*

The inverse case (`before_height_relative` / `before_seconds_relative` with overflow) causes a permanently unspendable coin, which is a committed state corruption of the coin's spendability.

### Likelihood Explanation
The `nowrap=false` path is the pre-soft-fork legacy behavior. Any block validated before the soft-fork activation height uses this path. An attacker who creates a coin with an overflowing relative timelock before the soft-fork activates can exploit this. The attacker-controlled entry point is a standard unprivileged CLVM puzzle spend — no privileged role is required. The overflow values are valid CLVM integers (they parse successfully through `sanitize_uint` as shown in `conditions.rs`), so the condition is accepted during CLVM execution and only evaluated incorrectly at the `check_time_locks` stage.

### Recommendation
Remove the `nowrap=false` branch entirely, or replace `wrapping_add` with `saturating_add` unconditionally. The `nowrap=true` (saturating) path is already the correct behavior and is what the soft-fork enforces. Retaining the wrapping path as a reachable code branch creates a permanent bypass window for any block validated under legacy rules. If backward compatibility with pre-soft-fork blocks is required, the wrapping behavior should be replaced with an explicit overflow check that rejects the spend rather than silently passing it.

### Proof of Concept
1. Before the soft-fork activation height, create a coin with a puzzle that outputs `(ASSERT_HEIGHT_RELATIVE 0xffff_ffff)`.
2. The condition parses successfully (it is within the 4-byte sanitized uint range).
3. The coin is confirmed at block height `H` (e.g., `H = 100`).
4. At any subsequent block height ≥ 99, call `check_time_locks(..., nowrap=False)`.
5. The guard computes `100u32.wrapping_add(0xffff_ffff) = 99`. The check `prev_height < 99` is false for `prev_height ≥ 99`.
6. `check_time_locks` returns `Ok(())` — the spend is accepted despite the coin being intended to be locked for ~4 billion blocks.

Confirmed by the existing test case `height_relative_wrap` and Python test `make_test_conds(height_relative=0xFFFF_FFFF)` with `nowrap=False` returning `None` (no error). [7](#0-6) [8](#0-7)
