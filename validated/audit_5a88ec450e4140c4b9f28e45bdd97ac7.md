### Title
Unchecked Multiplication in `denormalize_amount` Can Cause Permanent Fund Lock on Finalization - (File: near/omni-bridge/src/lib.rs)

### Summary
`denormalize_amount` performs a plain `u128` multiplication with no overflow guard. When a token's `origin_decimals` exceeds its NEAR-side `decimals`, any sufficiently large transfer amount causes the multiplication to overflow. Because `fin_transfer_callback` calls this function before writing any state, an overflow causes the callback to panic and the finalization to fail permanently, locking the user's funds in the origin-chain bridge contract with no recovery path.

### Finding Description

`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← unchecked multiplication
}
``` [1](#0-0) 

The `Decimals` struct stores two fields: `decimals` (the NEAR-side / chain-side capped value) and `origin_decimals` (the original token's decimal count). [2](#0-1) 

When a token is registered via `bind_token_callback`, these two values can differ:

```rust
self.add_token(
    &deploy_token.token,
    &deploy_token.token_address,
    deploy_token.decimals,         // chain-side (capped)
    deploy_token.origin_decimals,  // original
);
``` [3](#0-2) 

On Solana, the chain-side decimals are explicitly capped at `MAX_ALLOWED_DECIMALS`:

```rust
let origin_decimals = metadata.decimals;
metadata.decimals = std::cmp::min(MAX_ALLOWED_DECIMALS, metadata.decimals);
``` [4](#0-3) 

This guarantees `origin_decimals >= decimals`, so `diff_decimals > 0` for any token whose origin precision exceeds the chain-side cap. The `DeployTokenResponse` propagates both values to NEAR: [5](#0-4) 

`denormalize_amount` is called unconditionally inside `fin_transfer_callback` before any state is written:

```rust
let transfer_message = TransferMessage {
    ...
    amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
    ...
    fee: Self::denormalize_fee(&init_transfer.fee, decimals),
    ...
};
``` [6](#0-5) 

`denormalize_fee` also calls `denormalize_amount` for the fee field: [7](#0-6) 

The same unchecked call appears in `fast_fin_transfer`: [8](#0-7) 

### Impact Explanation

NEAR contracts compiled with `overflow-checks = true` (the standard NEAR SDK release profile) will **panic** on overflow, causing the entire `fin_transfer_callback` transaction to revert. Because the origin-chain `InitTransfer` has already been executed and the tokens are already locked or burned in the origin bridge contract, a permanently failing callback means:

- The user's funds are irrecoverably locked in the EVM/Solana bridge contract.
- No retry is possible: every subsequent finalization attempt with the same proof will hit the same overflow and panic again.
- There is no refund or escape hatch on the origin chain for a failed NEAR finalization.

This matches **Critical: Permanent freezing / irrecoverable lock of user funds**.

### Likelihood Explanation

The overflow requires `origin_decimals > decimals` (true for any token whose origin precision exceeds the chain-side cap) **and** a transfer amount large enough that `amount × 10^diff_decimals > u128::MAX`.

- With `diff_decimals = 9` (e.g., a Solana token with 18 origin decimals, capped to 9): overflow threshold ≈ `3.4 × 10^29` in 9-decimal units.
- With `diff_decimals = 6` (e.g., an EVM token with 24 origin decimals, normalized to 18): overflow threshold ≈ `3.4 × 10^32` in 18-decimal units.

For tokens with very large supplies or high-denomination transfers this threshold is reachable. The amount is entirely user-controlled via `initTransfer` on the origin chain — no privileged role is required. An attacker who is willing to lock their own tokens can also use this to permanently freeze a victim's pending finalization if they can front-run or race the finalization.

### Recommendation

Replace the plain multiplication with `checked_mul` and propagate the error, or use `u256`/`U256` arithmetic for the intermediate result and downcast only after confirming it fits in `u128`:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount
        .checked_mul(10_u128.pow(diff_decimals))
        .expect("denormalize_amount overflow")
}
```

A better approach is to return `Option<u128>` or `Result<u128, BridgeError>` and propagate the error to the caller so that `fin_transfer_callback` can return a meaningful error instead of panicking.

### Proof of Concept

Assume a token registered with `origin_decimals = 18`, `decimals = 9` (e.g., a Solana-origin token with `MAX_ALLOWED_DECIMALS = 9`). A user initiates a transfer on Solana with amount `A = u128::MAX / 10^9 + 1 ≈ 3.4 × 10^29 + 1` (in 9-decimal units). The proof is submitted to NEAR via `fin_transfer`. Inside `fin_transfer_callback`:

```
denormalize_amount(3.4e29 + 1, Decimals { decimals: 9, origin_decimals: 18 })
= (3.4e29 + 1) * 10^9
> u128::MAX   ← overflow → panic
```

The callback panics on every retry. The Solana-side tokens are already burned/locked. The user's funds are permanently lost.
