### Title
Nil Pointer Dereference Panic in `TransactionArgs.ToTransaction` SetCodeTx Branch via `eth_fillTransaction` — (`x/evm/types/tx_args.go`)

### Summary
`ToTransaction()` unconditionally dereferences `*args.To` in the `AuthorizationList != nil` branch. `SetTxDefaults` only rejects `To == nil` when no input data is present; supplying non-empty `data` bypasses the guard, leaving `args.To == nil` and causing a panic that crashes the node process.

### Finding Description
In `TransactionArgs.ToTransaction()`, the `SetCodeTx` branch performs an unconditional pointer dereference:

```go
// x/evm/types/tx_args.go:92-93
data = &types.SetCodeTx{
    To: *args.To,   // panics if args.To == nil
```

The only upstream guard is in `SetTxDefaults`:

```go
// rpc/backend/call_tx.go:294-306
if args.To == nil {
    // ...
    if len(input) == 0 {
        return args, errors.New("contract creation without any data provided")
    }
    // ← no error returned when len(input) > 0; args.To stays nil
}
```

When `AuthorizationList` is non-nil and `To` is nil but `data`/`input` is non-empty, `SetTxDefaults` returns successfully with `args.To == nil`, and the subsequent `ToTransaction()` call panics.

Note: `ToEthTransaction()` (used by the simulator) correctly guards this with:
```go
// x/evm/types/tx_args.go:393-396
to := common.Address{}
if args.To != nil {
    to = *args.To
}
``` [5](#0-4) 

`ToTransaction()` lacks this nil check entirely.

### Impact Explanation
A panic in the RPC goroutine crashes the node process (or at minimum the serving goroutine, depending on recovery middleware). A validator node crashing stops it from participating in consensus. Repeated requests from any unauthenticated caller can keep the node down, constituting a sustained DoS against the validator. This matches: *"Valid unprivileged RPC submission path can halt the chain / cause deterministic validator consensus failure."*

### Likelihood Explanation
The `eth_fillTransaction` endpoint is part of the standard `eth_` public JSON-RPC namespace, requires no authentication, and accepts arbitrary JSON. The triggering condition (non-nil `AuthorizationList`, nil `To`, non-empty `data`) is trivially constructable. No special privileges, keys, or chain state are required.

### Recommendation
In `ToTransaction()`, mirror the nil-safe pattern already used in `ToEthTransaction()`:

```go
case args.AuthorizationList != nil:
    to := common.Address{}
    if args.To != nil {
        to = *args.To
    }
    data = &types.SetCodeTx{
        To: to,
        ...
    }
```

Additionally, `SetTxDefaults` should explicitly reject `To == nil` when `AuthorizationList` is non-nil, since EIP-7702 `SetCodeTx` requires a `to` field (it is not a contract creation transaction type).
