Audit Report

## Title
Unaccounted Router ETH Balance Silently Subsidizes Subsequent WETH Swaps, Stealing Prior User's Refund — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay` reads `address(this).balance` as an unattributed ETH pool when paying with WETH. ETH stranded on the router from a prior user's `msg.value` overpayment (when `msg.value > amountIn` and `refundETH()` is not called) is silently consumed to fund a subsequent user's WETH swap, causing direct loss of the prior user's principal with no compensation.

## Finding Description

`exactInputSingle` (and `exactInput`) are `payable` with no enforcement that `msg.value == params.amountIn`. When `tokenIn == WETH`, the callback invokes `pay(WETH, payer, pool, amountIn)`. Inside `pay`, line 74 reads `nativeBalance = address(this).balance` — the entire router balance, not the current call's `msg.value`. If `nativeBalance >= value`, the router deposits that ETH and transfers WETH to the pool without ever calling `safeTransferFrom` on the payer. Any ETH left from a prior overpaying user satisfies this condition. An attacker who observes stranded ETH on the router calls `exactInputSingle(tokenIn=WETH, amountIn=stranded_amount, msg.value=0)`, triggering the callback, which consumes the stranded ETH to fund the attacker's swap for free. The `receive()` guard (lines 32–34) is irrelevant because `msg.value` in payable calls bypasses `receive()` entirely.

## Impact Explanation

Direct loss of user principal. The prior user's overpayment ETH (`msg.value - amountIn`) is permanently consumed to fund an unrelated user's swap. The prior user receives no WETH and no ETH refund for that amount. This is a Critical/High direct-loss-of-user-principal impact under Sherlock contest rules.

## Likelihood Explanation

Moderate. The precondition — ETH stranded on the router — arises naturally whenever a user calls `exactInputSingle` with `msg.value > amountIn` without bundling `refundETH()` in the same multicall. An attacker can monitor the router's ETH balance on-chain and immediately follow with a WETH swap to drain it. No privileged access, malicious pool, or non-standard token is required.

## Recommendation

Track the ETH belonging to the current transaction's `msg.value` separately from any pre-existing router balance. The simplest fix is to record `address(this).balance - msg.value` at entry and only allow `pay` to use ETH up to `msg.value`. Alternatively, pass `msg.value` through the call chain and cap ETH usage in `pay` to that amount, reverting if `address(this).balance > msg.value` at payment time.
