### Title
Legacy EIP-712 Multi-Message Type Schema Derived Only From First Message, Allowing Signature Bypass for Optional Fields in Subsequent Messages - (File: `ante/cosmos/eip712.go`)

### Summary

In `LegacyEip712SigVerificationDecorator.VerifySignature`, validation iterates over **all** messages in the transaction, but the EIP-712 type schema used to compute the signed hash is derived exclusively from `msgs[0]`. Because `legacyTraverseFields` silently skips zero-value fields when building the schema, any optional field that is zero in `msgs[0]` is absent from the schema and therefore absent from the EIP-712 hash. An attacker can craft a multi-message transaction where `msgs[0]` has a zero value for a sensitive optional field, obtain a valid user signature, then submit a modified transaction where `msgs[1]` carries a non-zero value for that field. The signature remains valid because the field is not covered by the hash.

### Finding Description

**Step 1 – Validation covers all messages.**

`LegacyValidatePayloadMessages(msgs)` iterates over every message, enforcing that all share the same Amino type and the same signer. [1](#0-0) 

**Step 2 – Type schema is built from `msgs[0]` only.**

Immediately after validation, `LegacyWrapTxToTypedData` is called with `msgs[0]` as the sole `sdk.Msg` argument. This argument is forwarded to `extractMsgTypes` → `walkFields` → `legacyTraverseFields`, which constructs the `MsgValue` type definition that will be embedded in the EIP-712 `Types` map. [2](#0-1) 

**Step 3 – Zero-value fields are silently dropped from the schema.**

Inside `legacyTraverseFields`, every struct field whose reflected value satisfies `field.IsZero()` is skipped and never added to the type map. Empty slices are also skipped at line 259. [3](#0-2) [4](#0-3) 

**Step 4 – The `Message` payload covers all messages; the `Types` schema does not.**

`LegacyWrapTxToTypedData` sets `typedData.Message = txData` (the full Amino JSON of the transaction, including every message), but `typedData.Types` contains only the schema derived from `msgs[0]`. [5](#0-4) 

In EIP-712, `encodeData` encodes only fields that appear in the declared type. A field present in `Message` but absent from `Types` is silently excluded from the hash. Therefore, any optional field that is zero in `msgs[0]` is not committed to by the signature, even though `msgs[1]` may carry a non-zero value for that field.

**Step 5 – The decorator is still reachable.**

The `LegacyEip712SigVerificationDecorator` is referenced in `evmd/ante/evm_handler.go` and remains a live code path for chains that have not migrated to the `ethsecp256k1` path. [6](#0-5) 

### Impact Explanation

An attacker who can persuade a user to sign a crafted multi-message EIP-712 transaction can execute a second message (`msgs[1]`) with field values that were never covered by the user's signature. Concretely, for any Cosmos message type that has a pointer or optional field (e.g., an `expiration *time.Time`, a secondary `sdk.Coin` amount, or any nullable proto field), the attacker can:

1. Set that field to `nil`/zero in `msgs[0]` so it is excluded from the type schema.
2. Obtain the user's signature over the resulting hash.
3. Submit a modified transaction where `msgs[1]` carries a large or malicious value for that field.
4. The chain recomputes the EIP-712 hash from the submitted `txBytes` using the same schema (still derived from `msgs[0]`), the absent field is still not hashed, and the signature verifies successfully.

The result is unauthorized execution of a Cosmos message with attacker-controlled parameters — a forged execution enabled by a signer-verification bypass. This matches the **High** impact category: *EIP-712 authorization bypass enabling forged execution or unauthorized account/code mutation*.

### Likelihood Explanation

- The `LegacyEip712SigVerificationDecorator` is still present and reachable in `evmd/ante/evm_handler.go`.
- Multi-message EIP-712 transactions are a documented and tested feature (`CreateTestEIP712MultipleSignerMsgs`).
- Many Cosmos SDK message types (authz `MsgGrant`, staking `MsgDelegate` with optional fields, IBC messages) contain pointer or optional fields that are zero in common usage.
- No privileged role is required; any unprivileged user can submit a crafted transaction.

### Recommendation

Replace the single-message schema derivation with one that merges the type schemas of **all** messages:

```go
// Instead of:
typedData, err := eip712.LegacyWrapTxToTypedData(ethermintCodec, extOpt.TypedDataChainID, msgs[0], txBytes, feeDelegation)

// Derive the schema from every message and merge, or pass the full slice:
typedData, err := eip712.LegacyWrapTxToTypedDataForMsgs(ethermintCodec, extOpt.TypedDataChainID, msgs, txBytes, feeDelegation)
```

Inside `legacyTraverseFields`, remove the `field.IsZero()` early-exit so that optional fields are always included in the type schema regardless of their value in the first message. Alternatively, accelerate the migration away from the legacy path entirely.

### Proof of Concept

1. Choose a Cosmos message type `M` that has an optional pointer field `F` (e.g., `expiration *time.Time` in `authz.MsgGrant`, or any proto `optional` field).
2. Construct a transaction with two messages of type `M`:
   - `msgs[0]`: `F = nil` (zero value → excluded from schema)
   - `msgs[1]`: `F = <attacker-chosen value>`
3. Present the transaction to the victim. The EIP-712 hash does not include `F` (absent from schema). The victim signs.
4. Submit the transaction. The chain calls `LegacyWrapTxToTypedData(..., msgs[0], txBytes, ...)`, rebuilds the same schema (no `F`), recomputes the same hash, and `ethcrypto.VerifySignature` returns `true`.
5. Both messages execute: `msgs[1]` runs with the attacker-chosen value of `F` that the victim never authorized. [7](#0-6) [8](#0-7) [3](#0-2)

// LegacyWrapTxToTypedData is an ultimate method that wraps Amino-encoded Cosmos Tx JSON data
// into an EIP712-compatible TypedData request.
func LegacyWrapTxToTypedData(
	cdc codectypes.AnyUnpacker,
	chainID uint64,
	msg sdk.Msg,
	data []byte,
	feeDelegation *FeeDelegationOptions,
) (apitypes.TypedData, error) {
	value, err := ethermint.SafeInt64(chainID)
	if err != nil {
		return apitypes.TypedData{}, err
	}
	txData := make(map[string]interface{})

	if err := json.Unmarshal(data, &txData); err != nil {
		return apitypes.TypedData{}, errorsmod.Wrap(errortypes.ErrJSONUnmarshal, "failed to JSON unmarshal data")
	}

	domain := apitypes.TypedDataDomain{
		Name:              "Cosmos Web3",
		Version:           "1.0.0",
		ChainId:           math.NewHexOrDecimal256(value),
		VerifyingContract: "cosmos",
		Salt:              "0",
	}

	msgTypes, err := extractMsgTypes(cdc, "MsgValue", msg)
	if err != nil {
		return apitypes.TypedData{}, err
	}

	if feeDelegation != nil {
		feeInfo, ok := txData["fee"].(map[string]interface{})
		if !ok {
			return apitypes.TypedData{}, errorsmod.Wrap(errortypes.ErrInvalidType, "cannot parse fee from tx data")
		}

		feeInfo["feePayer"] = feeDelegation.FeePayer.String()

		// also patching msgTypes to include feePayer
		msgTypes["Fee"] = []apitypes.Type{
			{Name: "feePayer", Type: "string"},
			{Name: "amount", Type: "Coin[]"},
			{Name: "gas", Type: "string"},
		}
	}

	typedData := apitypes.TypedData{
		Types:       msgTypes,
		PrimaryType: "Tx",
		Domain:      domain,
		Message:     txData,
	}

	return typedData, nil
}
