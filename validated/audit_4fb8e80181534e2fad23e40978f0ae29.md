### Title
ERC1155 `safeTransferFrom` in `finTransfer` Triggers Uncontrolled `onERC1155Received` Callback, Permanently Locking User Funds — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer()` uses `IERC1155.safeTransferFrom()` to deliver bridged ERC1155 tokens to the recipient. This triggers the `onERC1155Received` hook on any contract recipient. If the recipient contract does not implement `IERC1155Receiver` (or reverts/exhausts gas in the hook), `finTransfer` always reverts. Because the nonce is set before the transfer but the revert rolls it back, the nonce is never consumed — yet the transfer can never succeed. The user's NEAR-side tokens are already locked/burned with no on-chain refund path, permanently freezing their funds.

---

### Finding Description

In `OmniBridge.finTransfer()`, when the destination token is a registered ERC1155 multi-token, the bridge delivers tokens via:

```solidity
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,
    multiToken.tokenId,
    payload.amount,
    ""
);
``` [1](#0-0) 

Per the ERC1155 standard, `safeTransferFrom` calls `onERC1155Received` on `payload.recipient` if it is a contract. If that contract:

- does not implement `IERC1155Receiver` (e.g., a multisig, DAO treasury, or generic vault), or
- reverts or exhausts gas inside `onERC1155Received`,

the entire `finTransfer` call reverts.

The nonce guard is set at line 287 before the transfer:

```solidity
completedTransfers[payload.destinationNonce] = true;
``` [2](#0-1) 

Because the whole transaction reverts, this write is also rolled back. The nonce is never consumed, so the call can be retried — but every retry will hit the same revert. There is no alternative delivery path and no on-chain refund mechanism on the NEAR side once `initTransfer` has locked or burned the tokens.

The NEAR bridge stores the pending transfer in `pending_transfers` but provides no automatic refund if EVM-side finalization permanently fails: [3](#0-2) 

---

### Impact Explanation

**Permanent freezing of user funds.** Once a user initiates a NEAR→EVM bridge of ERC1155 tokens specifying a contract recipient that does not implement `IERC1155Receiver`, their NEAR-side tokens are irrecoverably locked. `finTransfer` will revert on every attempt with no on-chain escape hatch. Recovery requires privileged admin action (contract upgrade or manual intervention), which is not guaranteed and may not be timely.

This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Medium.** ERC1155 tokens are commonly sent to contract addresses — DAO treasuries, multisig wallets, yield vaults, and protocol contracts — the majority of which do not implement `IERC1155Receiver`. A user bridging to any such address triggers the permanent lock. The scenario requires no privileged access: any unprivileged bridge user initiating an ERC1155 transfer to a contract recipient is exposed. The `logMetadata1155` registration path is also permissionless: [4](#0-3) 

---

### Recommendation

Replace `safeTransferFrom` with a try/catch pattern so that a failed `onERC1155Received` does not permanently block finalization. Alternatively, wrap the call and emit a claimable-escrow event so the recipient can pull tokens later:

```solidity
try IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,
    multiToken.tokenId,
    payload.amount,
    ""
) {
    // success
} catch {
    // store as claimable or revert with a recoverable error
    pendingClaims[payload.destinationNonce] = payload;
    emit TransferEscrowed(payload.destinationNonce, payload.recipient);
}
```

Document clearly that contract recipients of ERC1155 tokens must implement `IERC1155Receiver`, analogous to the original report's recommendation to use `mint` instead of `safeMint`.

---

### Proof of Concept

1. Alice holds ERC1155 tokens on NEAR and calls `initTransfer` specifying `recipient = <DAO treasury address>` on EVM. The DAO treasury is a standard Gnosis Safe that does not implement `IERC1155Receiver`. NEAR-side tokens are locked.

2. The MPC signs a `TransferMessagePayload` with `recipient = <DAO treasury>` and `tokenAddress = deterministicToken` (the ERC1155 multi-token mapping).

3. A relayer calls `OmniBridge.finTransfer(signatureData, payload)`.

4. Execution reaches:
   ```solidity
   IERC1155(multiToken.tokenAddress).safeTransferFrom(
       address(this), payload.recipient, multiToken.tokenId, payload.amount, ""
   );
   ``` [5](#0-4) 

5. OpenZeppelin's `safeTransferFrom` calls `_checkOnERC1155Received`, which calls `onERC1155Received` on the DAO treasury. The treasury has no such function → revert.

6. The entire `finTransfer` transaction reverts. `completedTransfers[nonce]` is rolled back. The nonce is free but the call will always revert.

7. Alice's NEAR-side tokens remain locked with no refund path. Funds are permanently frozen.
