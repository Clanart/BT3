### Title
Missing Declared-Class Validation in `execute_replace_class` Enables Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the replacement class hash corresponds to a previously declared contract class. A malicious contract deployer can call `replace_class` with an arbitrary, undeclared class hash, permanently bricking the contract. Any funds held by that contract become permanently frozen because subsequent calls to it will cause a hard Cairo assertion failure inside `execute_entry_point`, making those transactions unincludable in any block.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function accepts any `class_hash` value from the syscall request and writes it directly into the contract's `StateEntry` without checking whether that hash corresponds to a declared class: [1](#0-0) 

The critical missing check is explicitly acknowledged by the developers themselves in a TODO comment at line 898:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
```

After the replacement is committed to state, any future call to the affected contract reaches `execute_entry_point`, which performs a `dict_read` on `contract_class_changes` using the new (undeclared) class hash as the key: [2](#0-1) 

Because the class was never declared, `dict_read` returns the default value `0` for `compiled_class_hash`. The subsequent `find_element` call then searches `compiled_class_facts_bundle` for a compiled class with hash `0`. If no such entry exists, `find_element` raises a hard Cairo assertion failure — not a graceful revert — making the entire block invalid if such a transaction is included. The sequencer is therefore forced to exclude all calls to the bricked contract, permanently freezing its funds.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared hash and the block is finalized, the state transition is irreversible. No future transaction can successfully call the contract (the OS will hard-fail during proof generation), so any ERC-20 balances, vault deposits, or other assets held in that contract's storage are permanently inaccessible.

---

### Likelihood Explanation

Any contract deployer can craft a contract that exposes a function calling the `replace_class` syscall with an arbitrary hash — either as a deliberate rug-pull mechanism or as a hidden backdoor. The `replace_class` syscall is available to every contract with no OS-level access control. The attacker only needs to deploy a contract, attract user deposits, and then invoke the backdoor. The missing validation is confirmed by the developer TODO comment, indicating the check was known to be absent.

---

### Recommendation

Inside `execute_replace_class`, before writing the new class hash to state, verify that the requested `class_hash` has a non-zero entry in `contract_class_changes` (i.e., it has been declared). Concretely:

1. Perform a `dict_read` on `contract_class_changes` with `key=class_hash`.
2. Assert the returned `compiled_class_hash` is non-zero (i.e., the class was declared).
3. Only then proceed with the `dict_update` to the contract's `StateEntry`.

This mirrors the validation already performed implicitly in `execute_entry_point` but must be enforced eagerly at the point of replacement to prevent undeclared hashes from ever entering the state.

---

### Proof of Concept

1. Attacker deploys `MaliciousVault` — a contract with a `deposit` entry point and a hidden `brick` entry point that calls `replace_class(0xdeadbeef)` where `0xdeadbeef` is never declared.
2. Users call `deposit`, transferring tokens into `MaliciousVault`'s storage.
3. Attacker calls `brick`. The OS processes `execute_replace_class`:
   - `class_hash = 0xdeadbeef` is written into `MaliciousVault`'s `StateEntry` with no validation.
   - Block is finalized; state is committed.
4. In any subsequent block, a user calls `withdraw` on `MaliciousVault`:
   - `execute_entry_point` reads `contract_class_changes[0xdeadbeef]` → returns `0`.
   - `find_element(..., key=0)` fails with a hard assertion error.
   - The sequencer cannot include this transaction.
5. All user funds in `MaliciousVault` are permanently frozen.

The root cause is the absent validation at: [3](#0-2) 

and the downstream hard failure at: [4](#0-3)

func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    alloc_locals;
    let request = cast(syscall_ptr + RequestHeader.SIZE, ReplaceClassRequest*);

    // Reduce gas.
    let success = reduce_syscall_gas_and_write_response_header(
        total_gas_cost=REPLACE_CLASS_GAS_COST, request_struct_size=ReplaceClassRequest.SIZE
    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );

    assert [revert_log] = RevertLogEntry(selector=CHANGE_CLASS_ENTRY, value=state_entry.class_hash);
    let revert_log = &revert_log[1];

    return ();
}
