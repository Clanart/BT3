### Title
Excess ETH Not Refunded to Caller in Wormhole-Variant Bridge Functions — (`File: evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

---

### Summary

`OmniBridgeWormhole` exposes four `payable` entry points — `finTransfer`, `logMetadata` / `logMetadata1155`, `deployToken`, and `initTransfer` — none of which validate that `msg.value` equals the exact Wormhole `messageFee()`, and none of which refund any excess ETH to the caller. Every wei above the required fee is forwarded to the Wormhole core contract and permanently lost to the user.

---

### Finding Description

`OmniBridge.sol` declares `finTransfer`, `logMetadata`, `logMetadata1155`, and `deployToken` as `external payable`. In the Wormhole deployment (`OmniBridgeWormhole`), each of these calls an internal `*Extension` hook that forwards the full `msg.value` to `_wormhole.publishMessage`:

**`finTransferExtension`** — passes `msg.value` verbatim: [1](#0-0) 

**`logMetadataExtension`** — passes `msg.value` verbatim: [2](#0-1) 

**`deployTokenExtension`** — passes `msg.value` verbatim: [3](#0-2) 

For `initTransfer`, the contract computes `extensionValue = msg.value - nativeFee` (ERC-20 path) or `msg.value - amount - nativeFee` (native-ETH path) and passes that residual to `initTransferExtension`, which forwards it to Wormhole: [4](#0-3) [5](#0-4) 

In every case the contract never checks `msg.value == _wormhole.messageFee()` (or `extensionValue == messageFee()` for `initTransfer`), and there is no code path that refunds the difference back to `msg.sender`. The Wormhole core bridge accepts any `msg.value >= messageFee()` and retains the surplus; it does not return excess ETH to the caller.

The only `messageFee` reference in production code is the interface declaration: [6](#0-5) 

It is never called to validate or bound the incoming value.

---

### Impact Explanation

Any caller who sends more ETH than `_wormhole.messageFee()` permanently loses the excess. For `finTransfer` and `initTransfer` — the two highest-frequency bridge operations — this means:

- A relayer finalizing a cross-chain transfer overpays by even a small amount and the surplus is irrecoverably forwarded to the Wormhole contract.
- A user initiating a transfer with `msg.value = amount + nativeFee + wormhole_fee + dust` loses the dust with no recourse.

This is a fee/accounting corruption that misdirects user value, matching the **High** impact class: *"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."*

---

### Likelihood Explanation

- `finTransfer` and `initTransfer` are the primary public bridge entry points, called by every relayer and every user initiating a transfer.
- Front-end tooling, gas-estimation rounding, or manual overpayment (a common defensive pattern) can trivially produce `msg.value > messageFee()`.
- No special privilege is required; any unprivileged bridge user or relayer triggers the path.
- `logMetadata` and `logMetadata1155` are unrestricted `payable` functions callable by anyone. [7](#0-6) 

---

### Recommendation

Before forwarding ETH to Wormhole, validate the exact required fee and refund any surplus:

```solidity
// In OmniBridgeWormhole — shared helper
function _publishWithExactFee(uint32 nonce, bytes memory payload, uint8 level) internal {
    uint256 fee = _wormhole.messageFee();
    require(msg.value >= fee, "insufficient wormhole fee");
    _wormhole.publishMessage{value: fee}(nonce, payload, level);
    uint256 excess = msg.value - fee;
    if (excess > 0) {
        (bool ok, ) = msg.sender.call{value: excess}("");
        require(ok, "refund failed");
    }
    wormholeNonce++;
}
```

Apply the same pattern to `initTransferExtension`, replacing `msg.value` with `value` and refunding `value - fee` back to `sender`.

---

### Proof of Concept

1. Wormhole `messageFee()` returns `X` wei on the target chain.
2. Caller invokes `OmniBridgeWormhole.finTransfer(sig, payload, {value: X + 1 ether})`.
3. `finTransferExtension` executes `_wormhole.publishMessage{value: X + 1 ether}(...)`.
4. Wormhole accepts the call (fee satisfied), retains `X + 1 ether`.
5. `finTransfer` returns successfully; caller's `1 ether` surplus is gone with no refund.

The same sequence applies to `logMetadata`, `deployToken`, and `initTransfer` (substituting `extensionValue` for `msg.value` in the last case). [8](#0-7) [9](#0-8)
