### Title
Missing `paid_fee_on_l1` Validation in L1 Handler Execution Allows Uncompensated L2 Execution - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_l1_handler_transaction` in the StarkNet OS program executes L1→L2 messages with a hardcoded gas budget of `L1_HANDLER_L2_GAS_MAX_AMOUNT = 100,000,000` without ever validating that the fee paid on L1 is sufficient to cover that L2 execution cost. The OS output struct `MessageToL2Header` carries no `paid_fee_on_l1` field, and the function never calls `charge_fee`. This means the OS proof is valid regardless of the ETH fee attached to the L1→L2 message, so the L1 StarkNet contract cannot use the OS output to enforce fee adequacy. An unprivileged L1 message sender can trigger up to 100M gas units of L2 computation without the OS enforcing any compensation, leading to direct loss of funds for the sequencer/protocol.

---

### Finding Description

Every account transaction type in the OS (invoke, deploy_account, declare) calls `charge_fee` after execution, which validates `actual_fee <= max_fee` and executes an ERC-20 transfer to the sequencer. [1](#0-0) 

`execute_l1_handler_transaction` is the sole exception. It sets a fixed gas budget from the constant `L1_HANDLER_L2_GAS_MAX_AMOUNT` and immediately calls the entry point with no fee check and no `charge_fee` call: [2](#0-1) 

The constant is defined as: [3](#0-2) 

The `TxInfo` written for the L1 handler hardcodes `max_fee=0` and zeroes out all `resource_bounds` fields: [4](#0-3) 

`compute_l1_handler_transaction_hash` also hardcodes `max_fee=0`, so the fee paid on L1 is never committed to the transaction hash: [5](#0-4) 

The OS output struct `MessageToL2Header` has no `paid_fee_on_l1` field at all: [6](#0-5) 

A grep for `paid_fee_on_l1` across the entire OS program returns zero matches, confirming the field is entirely absent from the OS program. Because the OS output does not carry the fee paid on L1, the L1 StarkNet verifier contract cannot use the proven output to enforce that the fee was sufficient.

---

### Impact Explanation

**Impact: Critical — Direct loss of funds.**

An L1 message sender can attach zero or minimal ETH to an L1→L2 message. The OS will execute the corresponding L1 handler with the full `L1_HANDLER_L2_GAS_MAX_AMOUNT = 100,000,000` gas budget, produce a valid proof, and the L1 verifier will accept it. The sequencer bears the full cost of L2 execution (compute, state writes, DA) without being compensated through the OS-enforced fee mechanism. At scale, an attacker can flood the network with zero-fee L1→L2 messages, each consuming up to 100M L2 gas units, draining sequencer resources and causing direct, unrecoverable financial loss.

---

### Likelihood Explanation

**Likelihood: Medium.**

Sending an L1→L2 message is a permissionless operation available to any Ethereum address. The only friction is the L1 gas cost of the `sendMessageToL2` call itself, which is small relative to the L2 execution value obtained. The OS-level gap is structural and persistent: every L1 handler transaction processed by this OS version is executed without fee validation. The attack requires no special privilege, no leaked key, and no operator cooperation.

---

### Recommendation

1. Add a `paid_fee_on_l1` field to `MessageToL2Header` in `output.cairo` so the fee paid on L1 is committed to the OS output and verifiable by the L1 contract.
2. In `execute_l1_handler_transaction`, read the `paid_fee_on_l1` value from the transaction data and assert that it is sufficient to cover the actual gas consumed multiplied by the current L1 gas price, analogous to the check in `charge_fee`:
   ```
   assert_nn_le(actual_gas_consumed * gas_price, paid_fee_on_l1);
   ```
3. Include `paid_fee_on_l1` in `compute_l1_handler_transaction_hash` so the fee commitment is part of the signed transaction hash, preventing post-hoc manipulation.

---

### Proof of Concept

1. Attacker calls `sendMessageToL2` on the L1 StarkNet contract targeting a gas-heavy L2 contract entry point, attaching 1 wei of ETH as fee.
2. The sequencer picks up the message and includes it as an `L1_HANDLER` transaction.
3. `execute_l1_handler_transaction` is called. It sets `remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT` (100,000,000) with no fee check. [7](#0-6) 
4. The entry point executes, consuming up to 100M gas units of L2 computation.
5. `consume_l1_to_l2_message` writes the message to the OS output with no fee field. [8](#0-7) 
6. The OS proof is generated and submitted to L1. The L1 verifier accepts it because the OS output is valid — it contains no fee field to check against.
7. The sequencer has executed 100M gas units of L2 computation for 1 wei. Repeated at scale, this constitutes direct, provable loss of funds with no OS-level defense.

func consume_l1_to_l2_message{outputs: OsCarriedOutputs*}(
    execution_context: ExecutionContext*, nonce: felt
) {
    assert_not_zero(execution_context.calldata_size);
    // The payload is the calldata without the from_address argument (which is the first).
    let payload: felt* = execution_context.calldata + 1;
    tempvar payload_size = execution_context.calldata_size - 1;

    tempvar execution_info = execution_context.execution_info;

    // Write the given transaction to the output.
    assert [outputs.messages_to_l2] = MessageToL2Header(
        from_address=[execution_context.calldata],
        to_address=execution_info.contract_address,
        nonce=nonce,
        selector=execution_info.selector,
        payload_size=payload_size,
    );

    let message_payload = cast(outputs.messages_to_l2 + MessageToL2Header.SIZE, felt*);
    memcpy(dst=message_payload, src=payload, len=payload_size);

    let (outputs) = os_carried_outputs_new(
        messages_to_l1=outputs.messages_to_l1,
        messages_to_l2=outputs.messages_to_l2 + MessageToL2Header.SIZE +
        outputs.messages_to_l2.payload_size,
    );
    return ();
