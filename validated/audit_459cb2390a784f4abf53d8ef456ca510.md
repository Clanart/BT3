### Title
Missing Zero/Undeclared Class Hash Validation in `execute_replace_class` Permanently Freezes Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the new `class_hash` is non-zero or corresponds to a declared class before committing it to the contract state. This is the direct analog of the `LPToken.set_minter()` bug: a critical privileged state variable (the contract's class hash) is overwritten without any null/validity check, permanently bricking the contract and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class`, the OS reads `class_hash` directly from the syscall request and writes it into the contract's `StateEntry` with no validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The developer-acknowledged TODO at line 898 confirms the missing check: `// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.`

There is no `assert_not_zero(class_hash)` and no verification that `class_hash` exists in the declared class registry (`contract_class_changes` dict). The state update is committed unconditionally.

The downstream consequence is in `execute_entry_point`, which is called whenever any transaction targets the bricked contract in a subsequent block:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // now 0 or an undeclared hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,           // 0 → not found → find_element panics
);
```

`dict_read` on an unset key returns 0. `find_element` with key 0 (no compiled class with hash 0 exists) causes an unrecoverable OS-level panic, making the entire block unprovable for any block that includes a call to the bricked contract.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to 0 (or any undeclared hash) via `replace_class`, the contract's `StateEntry` in the global Patricia tree is permanently updated with that invalid hash. There is no recovery path in the OS: `replace_class` can only be called by the contract itself, and the contract can no longer execute (any call to it causes `find_element` to panic). All ERC-20 tokens, ETH, or other assets held in the contract's storage are permanently inaccessible.

---

### Likelihood Explanation

Any contract that exposes a `replace_class` call path — including contracts with access-control bugs, reentrancy, or intentionally malicious constructors — can trigger this. The attacker-controlled entry path is:

1. Attacker deploys or exploits a contract that holds user funds.
2. Attacker triggers a call path that invokes `replace_class(class_hash=0)` (or any undeclared hash).
3. The OS accepts the syscall, commits `class_hash=0` to the contract's `StateEntry`, and the block is proven successfully.
4. In any subsequent block, any transaction targeting the contract causes `execute_entry_point` → `find_element` to panic, making that block unprovable.
5. Funds are permanently frozen with no recovery mechanism.

This is reachable by any unprivileged transaction sender who can interact with a vulnerable contract.

---

### Recommendation

Add an explicit non-zero check and a declared-class check in `execute_replace_class` before committing the new class hash:

```cairo
// Assert the new class hash is not zero.
assert_not_zero(class_hash);

// Assert the class hash is declared (exists in contract_class_changes).
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This directly resolves the acknowledged TODO and mirrors the check already present in `execute_declare_transaction` (`assert_not_zero(compiled_class_hash)` at line 816 of `transaction_impls.cairo`).

---

### Proof of Concept

1. Deploy contract `Vault` that holds user ETH and exposes `brick_self()` which calls `replace_class(0)`.
2. Call `brick_self()`. The OS executes `execute_replace_class` with `class_hash=0`. No assertion fires. `dict_update` commits `StateEntry(class_hash=0, ...)` for `Vault`'s address. Block is proven.
3. In the next block, any transaction calling `Vault` reaches `execute_entry_point`:
   - `dict_read(key=0)` → `compiled_class_hash = 0`
   - `find_element(..., key=0)` → element not found → OS panic
   - Block is unprovable; sequencer must skip any block containing a call to `Vault`.
4. All funds in `Vault`'s storage are permanently frozen.
