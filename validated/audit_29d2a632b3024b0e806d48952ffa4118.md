### Title
Meta Transaction v0 Hash Missing Nonce Enables Unbounded Same-Chain Signature Replay - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` omits the nonce from the signed hash, and `execute_meta_tx_v0` never checks or increments a nonce. A single valid meta-transaction signature can therefore be replayed an unlimited number of times on the same chain, draining the signing account of funds.

---

### Finding Description

**Root cause — hash construction:**

`compute_meta_tx_v0_hash` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0`:

```cairo
// transaction_hash.cairo L302-L314
let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
    tx_hash_prefix=INVOKE_HASH_PREFIX,
    version=0,
    contract_address=contract_address,
    entry_point_selector=entry_point_selector,
    calldata_size=calldata_size,
    calldata=calldata,
    max_fee=0,
    chain_id=chain_id,
    additional_data_size=0,          // ← nonce slot is empty
    additional_data=cast(0, felt*),
);
``` [1](#0-0) 

For every other transaction type the nonce travels through `additional_data` (deprecated path) or through `CommonTxFields.nonce` → `hash_tx_common_fields` (v3 path). The meta-tx v0 path is the sole exception. [2](#0-1) 

**Root cause — nonce enforcement:**

`check_and_increment_nonce` explicitly bails out for version-0 transactions:

```cairo
// execute_transaction_utils.cairo L64-L67
func check_and_increment_nonce{...}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
``` [3](#0-2) 

`execute_meta_tx_v0` never calls `check_and_increment_nonce` at all, and hard-codes `nonce=0` in the synthesised `TxInfo`:

```cairo
// syscall_impls.cairo L343-L363
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,          // ← always zero, never checked, never incremented
    ...
);
``` [4](#0-3) 

The resulting hash is therefore a pure function of `(contract_address, selector, calldata, chain_id)`. Once a user has signed it, the signature is permanently valid for any number of re-executions.

---

### Impact Explanation

**Critical — Direct loss of funds.**

The meta-tx v0 syscall is designed to execute the `__execute__` entry point of an arbitrary account contract. `__execute__` is the entry point that performs token transfers and other state-changing calls on behalf of the account owner. Because the signed hash never changes (no nonce), an attacker who observes one valid meta-transaction can replay it indefinitely within the same block or across future blocks, repeatedly draining the victim's token balance until it reaches zero.

---

### Likelihood Explanation

**High.** The `meta_tx_v0` syscall is callable by any contract (`execute_syscalls` dispatches to it on `META_TX_V0_SELECTOR`). [5](#0-4) 

All transaction data, including signatures, is public on-chain. Any observer of a successful meta-transaction can extract the `(contract_address, selector, calldata, signature)` tuple and submit it again through any contract that exposes the syscall. No privileged access, leaked key, or social engineering is required.

---

### Recommendation

Include the nonce in the meta-tx v0 hash by passing it through `additional_data`, mirroring how regular deprecated (v0/v1/v2) invoke transactions are handled:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,          // add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        chain_id=chain_id,
        additional_data_size=1,       // include nonce
        additional_data=&nonce,
    );
    return tx_hash;
}
```

Additionally, `execute_meta_tx_v0` must read the account contract's current nonce from `contract_state_changes`, assert it matches the nonce supplied in the request, and increment it — analogous to `check_and_increment_nonce` for regular transactions. [6](#0-5) 

---

### Proof of Concept

1. **Victim** signs a meta-tx v0 authorising a transfer of 100 STRK:
   - `hash = H(INVOKE_PREFIX, 0, victim_addr, __execute__, calldata, 0, chain_id)`
   - `sig = ECDSA_sign(victim_privkey, hash)`

2. **Relayer** submits the transaction via a contract that calls `meta_tx_v0(victim_addr, __execute__, calldata, sig)`. The OS computes the same hash, places `sig` in `TxInfo`, and calls `victim.__execute__`. The transfer succeeds.

3. **Attacker** observes the on-chain calldata and signature. They call the same (or any other) contract that invokes `meta_tx_v0` with identical arguments.

4. The OS recomputes `hash` — identical, because `nonce` is not an input. `sig` still verifies. `victim.__execute__` runs again and transfers another 100 STRK.

5. Steps 3–4 repeat until the victim's balance is zero. No new signature from the victim is ever needed. [7](#0-6) [8](#0-7)

func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }

    tempvar state_entry: StateEntry*;
    %{ SetStateEntryToAccountContractAddress %}

    tempvar current_nonce = state_entry.nonce;
    with_attr error_message("Unexpected nonce.") {
        assert current_nonce = tx_info.nonce;

    tempvar new_state_entry = new StateEntry(
        class_hash=state_entry.class_hash,
        storage_ptr=state_entry.storage_ptr,
        nonce=current_nonce + 1,
    );
    dict_update{dict_ptr=contract_state_changes}(
        key=tx_info.account_contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    return ();
