### Title
Missing EVM-Side Recipient Validation Causes Permanent Fund Lock on Invalid OmniAddress — (`near/omni-types/src/evm/events.rs`, `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The EVM `initTransfer` function accepts any arbitrary `string` as `recipient` with zero format validation. The NEAR-side proof parser strictly requires the recipient to be a valid `OmniAddress` (known chain prefix + valid address). Any mismatch permanently locks the user's tokens on EVM with no recovery path.

---

### Finding Description

**Step 1 — EVM accepts any recipient string.**

`OmniBridge.initTransfer` takes `string calldata recipient` and performs no validation on it before locking/burning tokens and emitting the event: [1](#0-0) 

The function locks ERC-20 tokens (or burns bridge tokens), then emits `InitTransfer` with the raw, unvalidated `recipient` string. There is no `require` or format check on `recipient` anywhere in the function.

**Step 2 — NEAR-side parsing strictly requires a valid OmniAddress.**

In `TryFromLog<Log<InitTransfer>>::try_from_log`, line 127:

```rust
recipient: event.data.recipient.parse().map_err(stringify)?,
``` [2](#0-1) 

This calls `OmniAddress::from_str`, which splits on `:` and matches against a fixed set of known chain prefixes. Any unrecognized prefix or malformed address part returns `Err`: [3](#0-2) 

Specifically, line 409 returns `Err(format!("Chain {chain} is not supported"))` for any unknown prefix, and each chain's address parser (e.g., `AccountId::parse` for NEAR, hex decode for EVM) can independently fail.

**Step 3 — The error propagates as a permanent proof rejection.**

`parse_evm_proof` propagates the `Err` via `?`: [4](#0-3) 

`verify_proof_callback` returns `Result<ProverResult, String>` and propagates the same `Err`: [5](#0-4) 

The `#[handle_result]` attribute causes the NEAR transaction to revert. Every subsequent relay attempt for the same log entry will produce the same parse failure — the EVM event is immutable.

**Step 4 — No EVM-side recovery mechanism exists.**

The EVM contract has no function to refund a stuck `initTransfer`. The `completedTransfers` mapping only guards `finTransfer` (destination nonce replay). There is no admin rescue function for locked `initTransfer` funds. [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds.**

Any user who calls `initTransfer` with a recipient string that is valid UTF-8 but not a valid `OmniAddress` (e.g., `"unknown_chain:0xabc..."`, `"near:INVALID ACCOUNT!!!"`, `"garbage"`) will have their tokens permanently locked in the EVM bridge contract with no settlement path and no refund mechanism.

---

### Likelihood Explanation

**Medium-High.** The EVM `initTransfer` is a public, permissionless function. Users bridging to NEAR or other chains must manually type or paste a recipient string. A single typo in the chain prefix (e.g., `"ethereum:0x..."` instead of `"eth:0x..."`) or an invalid NEAR account ID (uppercase letters, special characters, length violations) is sufficient to trigger permanent loss. No privileged access is required.

---

### Recommendation

1. **Add recipient format validation in the EVM contract.** At minimum, require the recipient string to contain `:` and match a known chain prefix before accepting the transfer.
2. **Add a refund/rescue path on EVM.** Allow users (or an admin) to reclaim tokens from a stuck `initTransfer` nonce if the corresponding NEAR proof has never been successfully finalized.
3. **Alternatively, make NEAR-side parsing lenient for the recipient field** — store the raw string and defer validation to `fin_transfer`, returning the funds to the sender if the recipient is unparseable.

---

impl FromStr for OmniAddress {
    type Err = String;

    fn from_str(input: &str) -> Result<Self, Self::Err> {
        let (chain, recipient) = input.split_once(':').unwrap_or(("eth", input));

        match chain {
            "eth" => Ok(Self::Eth(recipient.parse().map_err(stringify)?)),
            "near" => Ok(Self::Near(recipient.parse().map_err(stringify)?)),
            "sol" => Ok(Self::Sol(recipient.parse().map_err(stringify)?)),
            "arb" => Ok(Self::Arb(recipient.parse().map_err(stringify)?)),
            "base" => Ok(Self::Base(recipient.parse().map_err(stringify)?)),
            "bnb" => Ok(Self::Bnb(recipient.parse().map_err(stringify)?)),
            "pol" => Ok(Self::Pol(recipient.parse().map_err(stringify)?)),
            "hlevm" => Ok(Self::HyperEvm(recipient.parse().map_err(stringify)?)),
            "abs" => Ok(Self::Abs(recipient.parse().map_err(stringify)?)),
            "btc" => Ok(Self::Btc(recipient.to_string())),
            "zcash" => Ok(Self::Zcash(recipient.to_string())),
            "strk" => Ok(Self::Strk(recipient.parse().map_err(stringify)?)),
            "fogo" => Ok(Self::Fogo(recipient.parse().map_err(stringify)?)),
            _ => Err(format!("Chain {chain} is not supported")),
        }
    }
