### Title
Blacklisted EVM Recipient Causes Irrecoverable Fund Lock in NEAR-to-EVM Bridge Transfers — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

When a user initiates a NEAR-to-EVM bridge transfer specifying an EVM address that is (or later becomes) blacklisted by the destination token (e.g., USDC/USDT), the `finTransfer` call on EVM will permanently revert. Because the recipient is embedded in the MPC-signed payload and cannot be altered, and because the NEAR bridge has no built-in cancel/refund path for this case, the user's tokens are irrecoverably locked.

---

### Finding Description

**NEAR-side initiation** (`near/omni-bridge/src/lib.rs` — `init_transfer_internal`):

When a user calls `ft_on_transfer` with an `InitTransfer` message targeting an EVM recipient, the bridge immediately burns bridge tokens or locks native tokens, then stores the `TransferMessage` in `pending_transfers`: [1](#0-0) 

The `recipient` field is set from user-supplied `init_transfer_msg.recipient` and is never validated against any token blacklist: [2](#0-1) 

**EVM-side finalization** (`evm/src/omni-bridge/contracts/OmniBridge.sol` — `finTransfer`):

The relayer submits the MPC-signed payload to `finTransfer`. For a native ERC-20 token (e.g., USDC), the final step is: [3](#0-2) 

If `payload.recipient` is blacklisted by the token contract, `safeTransfer` reverts, rolling back the entire transaction (including the `completedTransfers` nonce mark at line 287). The nonce is therefore never consumed, but the relayer **cannot change the recipient** — it is Borsh-encoded into the payload that the MPC key signed: [4](#0-3) 

Every subsequent `finTransfer` attempt with the same signed payload will revert identically. No alternative payload can be produced because the MPC signature is over the specific recipient address.

**No NEAR-side recovery path:**

`remove_transfer_message` is internal and does not restore burned/locked tokens: [5](#0-4) 

There is no public `cancel_transfer` or `refund_transfer` function. The `sign_transfer_callback` removes the pending record only when fee is zero, but even then the tokens are not returned: [6](#0-5) 

The result is that burned bridge tokens are permanently destroyed and locked native tokens are permanently trapped in the NEAR bridge contract.

---

### Impact Explanation

Matches: **Critical — Permanent freezing, irrecoverable lock of user funds in bridge flows.**

- For bridge tokens (burned on NEAR, never minted on EVM): tokens are permanently destroyed.
- For native NEAR tokens (locked in bridge): tokens are permanently trapped in the NEAR bridge contract with no built-in recovery path.

The DAO could deploy a contract upgrade to add a rescue function, but no such mechanism exists in the current production code.

---

### Likelihood Explanation

**Medium.** Blacklist-bearing tokens (USDC, USDT) are among the most commonly bridged assets. Two realistic triggers exist:

1. A user's EVM address is sanctioned (OFAC) **after** they initiated the transfer but **before** `finTransfer` is executed — a window that can span hours or days depending on relayer latency.
2. A user mistakenly specifies a known blacklisted address (e.g., a sanctioned exchange deposit address) as the recipient.

No privileged access is required; any unprivileged bridge user can trigger this state.

---

### Recommendation

1. **On the NEAR side**: Add a DAO-callable `cancel_transfer` function that removes the pending transfer record and restores (mints back or unlocks) the user's tokens when a transfer has been pending beyond a configurable timeout.
2. **On the EVM side**: Consider emitting a failure event (or using a try/catch pattern) so that a failed `finTransfer` can be proven back to NEAR to trigger a refund, rather than silently reverting.
3. **Optionally**: Validate at `initTransfer`/`ft_on_transfer` time that the recipient address is not a known zero/burn address, though full blacklist validation is not feasible on-chain.

---

### Proof of Concept

1. User holds USDC on NEAR (as a bridge-deployed token).
2. User calls `ft_transfer_call` on the NEAR USDC token with `msg = InitTransfer { recipient: "Eth:0xE1D865c3D669dCc8c57c8D023140CB204e672ee4", ... }` (a USDC-blacklisted EVM address).
3. NEAR bridge burns the USDC tokens and stores the `TransferMessage` in `pending_transfers`.
4. Relayer calls `sign_transfer` → MPC produces a signature over the payload including the blacklisted recipient.
5. Relayer calls `OmniBridge.finTransfer(signature, payload)` on EVM → `USDC.safeTransfer(0xE1D865..., amount)` reverts with `"Blacklistable: account is blacklisted"`.
6. Transaction reverts; nonce is not consumed; relayer retries → same revert indefinitely.
7. User's USDC is permanently burned on NEAR with no recovery path.

    fn remove_transfer_message(&mut self, transfer_id: TransferId) -> TransferMessage {
        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);
        }

        transfer.message
    }
