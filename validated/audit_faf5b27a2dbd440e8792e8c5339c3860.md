Audit Report

## Title
Unrestricted `refundETH`, `sweepToken`, and `unwrapWETH9` allow any caller to drain router ETH/token residue — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.refundETH()`, `sweepToken()`, and `unwrapWETH9()` carry no caller restriction or ownership check. Any address can invoke them at any time and redirect the router's entire ETH, ERC-20, or WETH balance to an arbitrary recipient. ETH residue is concretely created by `exactOutputSingle` when `msg.value > actualAmountIn`: the `pay()` helper wraps only the exact swap cost into WETH, leaving the surplus as raw ETH on the router, which any attacker can immediately claim via `refundETH()`.

## Finding Description

**Root cause — no caller guard on settlement helpers:**

`PeripheryPayments.sol` L37–63 exposes three public settlement functions with no `msg.sender` check:

```solidity
// L37-45
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override { ... }

// L48-55
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override { ... }

// L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) { _transferETH(msg.sender, balance); }
}
```

`sweepToken` and `unwrapWETH9` accept a caller-controlled `recipient`; `refundETH` sends to `msg.sender`. None of the three verify who deposited the value or whether the caller is entitled to it.

**ETH residue creation path — `pay()` L73–84:**

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();      // wraps exactly `value`
        IERC20(WETH).safeTransfer(recipient, value);
    }   // ← nativeBalance - value remains as raw ETH on the router
```

Only `value` (the exact swap cost) is wrapped; the surplus `msg.value - value` stays as raw ETH.

**Call path:**

`exactOutputSingle{value: X}(...)` (L130–147) sets callback context with `msg.sender` as payer, executes the pool swap, then calls `pay()` via `_justPayCallback`. After the swap, `actualAmountIn ≤ amountInMaximum`; the difference `X − actualAmountIn` remains as raw ETH on the router. The function then calls `_clearExpectedCallbackPool()` and returns — no refund is issued. The `receive()` guard (L32–34) only blocks direct ETH pushes from non-WETH addresses; it does not affect ETH arriving as `msg.value` in a function call.

`multicall()` (L39–44) is a simple delegate-call loop with no initiator tracking in transient storage, so there is no cross-call attribution that could restrict `refundETH`.

**Existing guards reviewed and shown insufficient:**

- `receive() external payable { if (msg.sender != WETH) revert NotWETH(); }` — only blocks unsolicited ETH pushes; does not prevent residue from `msg.value` in swap calls.
- `amountInMaximum` check in `exactOutputSingle` (L145) — only prevents overpayment by the pool; does not refund or lock the ETH surplus.
- No `onlyOwner`, `onlyInitiator`, or transient-storage initiator pattern exists on any of the three settlement helpers.

## Impact Explanation

Direct loss of user principal. An attacker who monitors the mempool can call `refundETH()` immediately after a victim's `exactOutputSingle{value: X}` transaction confirms, stealing `X − actualAmountIn` ETH. For `sweepToken`, the attacker can redirect the router's entire token balance — potentially the full swap output — to an arbitrary address. This meets the Critical/High direct loss of user principal threshold under the allowed impact gate.

## Likelihood Explanation

`exactOutputSingle` with native ETH input is a standard user flow. Users routinely supply a round `msg.value` (e.g., 1 ETH) when the exact cost is unknown. Calling `exactOutputSingle` directly without a `multicall` + `refundETH` tail is a realistic and common pattern. The attack requires a single public call with no setup, no privileged role, and no special conditions — trivially repeatable by any EOA watching the mempool.

## Recommendation

Add a caller-attribution guard to all three helpers. Record `msg.sender` in transient storage at the start of each top-level router entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`) and require the same address calls the settlement helpers:

```solidity
modifier onlyInitiator() {
    require(msg.sender == _tloadAddress(T_SLOT_INITIATOR), "not initiator");
    _;
}

function refundETH() external payable onlyInitiator { ... }
function sweepToken(...) public payable onlyInitiator { ... }
function unwrapWETH9(...) public payable onlyInitiator { ... }
```

Alternatively, remove the `recipient` parameter from `sweepToken` and `unwrapWETH9` and always send to `msg.sender`, and add the initiator guard to `refundETH`.

## Proof of Concept

```
Setup:
  - Router deployed with WETH address.
  - Pool: WETH / Token1, seeded with liquidity.
  - Victim has 2 ETH and wants exactly 1000 Token1.

Step 1 — Victim calls exactOutputSingle directly (no multicall):
  router.exactOutputSingle{value: 2 ether}(ExactOutputSingleParams({
      pool: pool,
      tokenIn: WETH,
      tokenOut: Token1,
      zeroForOne: true,
      amountOut: 1000e18,
      amountInMaximum: 2 ether,
      recipient: victim,
      deadline: block.timestamp + 60,
      priceLimitX64: 0,
      extensionData: ""
  }));
  // pay() wraps 0.8 ETH into WETH and transfers it to the pool.
  // 1.2 ETH remains as raw ETH on the router.

Step 2 — Attacker (next tx or same block) calls:
  router.refundETH();
  // _transferETH(msg.sender, 1.2 ether) — attacker receives victim's 1.2 ETH.

Alternative (token theft):
  // Victim calls exactInputSingle with recipient = address(router).
  // Attacker calls sweepToken(Token1, 0, attacker) — receives all Token1.
```
