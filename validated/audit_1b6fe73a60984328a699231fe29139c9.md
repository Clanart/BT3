Audit Report

## Title
Router `pay()` Consumes Unattributed Native ETH to Settle Any Caller's WETH Swap, Enabling Theft of Stranded ETH — (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
The `pay()` function in `PeripheryPayments.sol` reads `address(this).balance` as a shared, unattributed pool when settling WETH-input swaps. Any native ETH stranded on the router by a prior caller (e.g., excess `msg.value` from a `multicall` that omits `refundETH`) is silently consumed to pay the next caller's swap. The original depositor permanently loses their ETH; the attacker receives output tokens without spending any WETH or ETH of their own.

## Finding Description
`pay()` at lines 73–84 of `PeripheryPayments.sol` branches on `token == WETH` and reads `address(this).balance` without any check that the ETH was deposited by the current execution context:

```solidity
// PeripheryPayments.sol L73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

When `nativeBalance >= value`, the named `payer` is never touched — the router pays entirely from its own balance. The `receive()` guard at lines 32–34 only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from being stranded via excess `msg.value` on any payable entry point. [2](#0-1) 

`refundETH()` at lines 58–63 is the intended cleanup mechanism, but it is a separate, optional call that must be explicitly composed into a multicall — nothing in the router enforces its inclusion. [3](#0-2) 

The exploit path:
1. Alice calls `multicall{value: 2 ether}([exactInputSingle(WETH→token1, amountIn=1 ether, ...)])` without appending `refundETH`. The swap consumes 1 ETH; 1 ETH remains stranded on the router.
2. Bob (zero WETH balance, zero WETH allowance) calls `exactInputSingle({tokenIn: WETH, amountIn: 1 ether, ...})` with `msg.value = 0`.
3. The pool triggers `metricOmmSwapCallback` → `_justPayCallback` → `pay(WETH, Bob, pool, 1 ether)`. [4](#0-3) 

4. `address(this).balance == 1 ether >= 1 ether` → router wraps Alice's ETH into WETH and transfers it to the pool. `safeTransferFrom(Bob, ...)` is never called.
5. Bob receives token1 output. Alice's 1 ETH is permanently consumed.

`exactInputSingle` is `payable` and sets `payer = msg.sender` via `_setNextCallbackContext`, but `pay()` ignores the named payer whenever native ETH is available. [5](#0-4) 

## Impact Explanation
Direct loss of user principal (ETH). When stranded ETH covers the full swap amount, the named payer is bypassed entirely — the victim's ETH is permanently converted to WETH and delivered to the pool as payment for a stranger's trade. This meets the Sherlock threshold for a direct loss of user principal with no recovery path.

## Likelihood Explanation
Medium. The precondition is ETH stranded on the router, which occurs whenever a user sends excess `msg.value` in a `multicall` and omits `refundETH`. The `multicall` function is `payable` and uses `delegatecall`, making this a realistic and documented usage pattern. [6](#0-5) 

Integrators composing multicalls are especially likely to omit the cleanup step. Once ETH is stranded, any observer can drain it in the next block with a single `exactInputSingle` call targeting a WETH pool. No special privileges are required.

## Recommendation
Track the ETH belonging to the current execution context rather than reading the raw contract balance. One approach: record `msg.value` in transient storage at each payable entry point and deduct from that tracked amount inside `pay()`, reverting if the tracked amount is exhausted. Alternatively, remove the native-ETH branch from `pay()` entirely and require callers to pre-wrap ETH to WETH before calling the router, relying solely on `safeTransferFrom`.

## Proof of Concept
1. Deploy router with a WETH/token1 pool.
2. Alice calls `router.multicall{value: 2 ether}([abi.encodeCall(router.exactInputSingle, (ExactInputSingleParams{pool: wethToken1Pool, tokenIn: WETH, amountIn: 1 ether, amountOutMinimum: 0, ...}))])` — no `refundETH` appended. After the call, `address(router).balance == 1 ether`.
3. Bob (no WETH, no allowance) calls `router.exactInputSingle(ExactInputSingleParams{pool: wethToken1Pool, tokenIn: WETH, amountIn: 1 ether, amountOutMinimum: 0, recipient: Bob, ...})` with `msg.value = 0`.
4. Assert: Bob received token1 output; `address(router).balance == 0`; Alice's 1 ETH is gone; Bob's WETH balance was never touched.
