### Title
Non-EVM-Native (IBC/CosmWasm) Token Balances Not Burned on SELFDESTRUCT, Recoverable via CREATE2 Re-deployment - (File: x/evm/statedb/statedb.go)

### Summary

When a contract self-destructs, `StateDB.Commit()` burns only the EVM-denom balance and removes the auth account. Non-EVM-native tokens (IBC coins, CosmWasm bridge tokens) held by the destroyed address are explicitly left as orphaned bank balances. Because the auth account is deleted but the bank balance persists, an attacker who controls the CREATE2 factory can re-deploy an identical contract at the same deterministic address in a subsequent transaction and immediately drain those orphaned funds.

### Finding Description

`StateDB.Commit()` processes self-destructed accounts as follows:

```go
// x/evm/statedb/statedb.go lines 800-825
if obj.selfDestructed {
    cosmosAddr := sdk.AccAddress(obj.Address().Bytes())
    cacheCtx, writeCache := s.origCtx.CacheContext()
    // Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
    // bridge) held by the destroyed address are not drained and may remain as
    // orphaned bank balances.
    if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
        ...SubBalance(cacheCtx, cosmosAddr, coin)...
    }
    if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil {
        return errorsmod.Wrap(err, "failed to delete account")
    }
    writeCache()
}

`DeleteAccount` removes the auth account record and EVM storage, but never touches the bank module for non-EVM-denom coins:

```go

After `DeleteAccount`, the Cosmos bank module still holds any IBC or other non-EVM-native coin balances at that address. The address is now "empty" from the auth-account perspective, so a CREATE2 re-deployment succeeds and the new contract inherits those orphaned bank balances.

### Impact Explanation

An attacker who controls a CREATE2 factory can:

1. Deploy a contract via CREATE2 with a known salt.
2. Have the contract accumulate non-EVM-native tokens (e.g., IBC coins transferred to the contract address by users of a DeFi protocol).
3. Call SELFDESTRUCT on the contract (valid under EIP-6780 if created in the same tx, or via a pre-existing SELFDESTRUCT path).
4. At commit time, the EVM-denom balance is burned and the auth account is deleted, but IBC coin balances remain at the address in the bank module.
5. In a subsequent transaction, re-deploy an identical contract at the same CREATE2 address.
6. The new contract can immediately transfer or drain the orphaned IBC balances.

This constitutes an unauthorized transfer of Cosmos bank funds through Ethermint's stateDB commit logic, matching the Critical/High allowed impact: *"Unauthorized theft … or balance transfer of … Cosmos bank funds through Ethermint transaction execution or stateDB/native action logic."* [3](#0-2) 

### Likelihood Explanation

- **Realistic on any Cosmos chain with IBC**: IBC tokens are routinely sent to contract addresses (e.g., DEX pools, lending vaults). Any contract that holds IBC coins and exposes a SELFDESTRUCT path (or is deployed via CREATE2 by a malicious factory) is vulnerable.
- **Attacker-controlled entry path is a standard EVM transaction**: No privileged validator or governance action is required. A single unprivileged `MsgEthereumTx` calling a factory's `attackInOneTx`-style function (as modeled in the existing test contract `SelfDestructExploitFactory`) is sufficient.
- **CREATE2 parameters are public**: The factory address, salt, and init-code are all observable on-chain, so a third party who observes the original deployment can also perform the re-deployment. [4](#0-3) 

### Recommendation

In `StateDB.Commit()`, after burning the EVM-denom balance and before calling `DeleteAccount`, iterate over **all** coin denominations held by the destroyed address and burn them:

```go
// Drain ALL bank balances, not just the EVM denom.
allBalances := s.bankKeeper.GetAllBalances(cacheCtx, cosmosAddr)
for _, coin := range allBalances {
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct non-evm balance")
    }
}
```

Alternatively, expose a `GetAllBalances` method on the `Keeper` interface and call it in the self-destruct commit path. This ensures that no orphaned bank balance survives account deletion regardless of denomination. [5](#0-4) 

### Proof of Concept

```
Tx 1 (single MsgEthereumTx):
  factory.attackInOneTx(salt, value=1_000_000_000):
    a) CREATE2 deploys child contract at deterministic address A
    b) child.destroy() → SELFDESTRUCT (EIP-6780: valid, created in same tx)
    c) factory.call{value: 1e9}(A) → AddBalance(A, 1e9 EVM-denom)
    d) Commit: EVM-denom burned ✓, auth account deleted ✓
       IBC coins at A: NOT burned ✗ → orphaned bank balance remains

Tx 2 (separate MsgEthereumTx):
  factory.redeployChild(salt):
    a) CREATE2 re-deploys identical child at address A (succeeds: no auth account)
    b) child.drain() → transfers orphaned IBC balance to tx.origin
    → attacker recovers IBC funds that users deposited before the selfdestruct

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account

// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
