### Title
Deploy Syscall Unconditionally Reports Success Regardless of Constructor Failure, Enabling Permanent Fund Loss — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` function in the StarkNet OS Cairo program unconditionally writes `failure_flag=0` (success) into the syscall response header after calling `deploy_contract`, regardless of whether the constructor actually succeeded or failed. This is structurally identical to the external report's `callWithMinGas()` returning `success = false` without reverting: the operation is marked as completed even when it silently fails, making it non-replayable and permanently bricking any funds sent by the caller based on the false success signal.

---

### Finding Description

In `execute_deploy`, after invoking `deploy_contract`, the response header is unconditionally asserted to contain `failure_flag=0`:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

In Cairo, `assert [response_header] = ResponseHeader(...)` is a memory constraint that **enforces** the value at that address. This means even if `deploy_contract` internally fails (constructor reverts, out of gas, etc.), the response written back to the calling contract's syscall segment always carries `failure_flag=0`. The calling contract has no mechanism to detect the failure.

The `deploy_contract` call itself:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}
``` [2](#0-1) 

returns only `(retdata_size, retdata)` — no `is_reverted` flag is returned or checked. The acknowledged TODO comment confirms this is a known gap: failure propagation is not yet implemented.

The `execute_deploy` function is dispatched from `execute_syscalls` whenever a contract issues a `DEPLOY_SELECTOR` syscall: [3](#0-2) 

---

### Impact Explanation

In StarkNet, a failed constructor means the contract is **not deployed** at the computed address. If the calling contract reads `failure_flag=0` (success) and proceeds to send funds (ETH or ERC-20 tokens) to the returned contract address, those funds are sent to a non-existent contract. There is no handler to receive or forward them. The transaction is finalized in the proven block; no replay is possible. This constitutes **Critical: Direct loss of funds**.

---

### Likelihood Explanation

The deploy-and-fund pattern is standard in DeFi factory contracts (e.g., a factory deploys a vault and seeds it with initial liquidity in the same transaction). Any such contract is vulnerable if the constructor can fail under any reachable condition. The entry path is fully unprivileged: any contract deployer can issue a `deploy` syscall. A constructor failure can be triggered by:
- Deploying a contract whose constructor reverts on a specific input (attacker-controlled calldata)
- A constructor that fails due to state conditions (e.g., a dependency contract not yet initialized)

The caller contract, receiving `failure_flag=0`, proceeds to fund the address — permanently losing those assets.

---

### Recommendation

Return and propagate the constructor's revert status from `deploy_contract`, and use it in the response header:

```cairo
with remaining_gas {
    let (retdata_size, retdata, is_reverted) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// Propagate actual failure status.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

If `is_reverted=1`, the response should carry the failure retdata and the contract address field should be zeroed or otherwise marked invalid, so callers can branch correctly and avoid sending funds to an undeployed address.

---

### Proof of Concept

1. Deploy a **factory contract** on StarkNet that:
   - Calls the `deploy` syscall with a target class whose constructor conditionally reverts (e.g., reverts if a storage slot is already set)
   - After the deploy call, reads the response: sees `failure_flag=0` (success) and a valid-looking contract address
   - Sends tokens to that address (standard post-deploy seeding)

2. Trigger the constructor failure condition (e.g., pre-set the storage slot the constructor checks).

3. The OS Cairo program enforces `failure_flag=0` via the `assert` at line 539 of `syscall_impls.cairo`, so the proof is valid and the block is finalized.

4. The factory's token transfer goes to the non-existent contract address. No contract exists there to handle or forward the funds.

5. The transaction is included in a proven block. The state is final. The funds are permanently lost with no replay path. [4](#0-3)
