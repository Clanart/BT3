### Title
`is_refund_required` Returns Wrong Default `false` on `ft_transfer_call` Promise Failure, Causing Permanent Freeze of User Funds and Accounting Corruption — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`is_refund_required` in `near/omni-bridge/src/lib.rs` returns `false` (the wrong default) when the underlying `ft_transfer_call` promise fails with a promise error. This causes `fin_transfer_send_tokens_callback` to skip reverting lock actions and skip removing the finalization record, permanently freezing user funds in the bridge and corrupting the locked-token accounting.

---

### Finding Description

`fin_transfer_send_tokens_callback` delegates the refund decision entirely to `is_refund_required`: [1](#0-0) 

`is_refund_required` is: [2](#0-1) 

When `ft_transfer_call` fails (i.e., the recipient contract's `ft_on_transfer` panics), `env::promise_result_checked` returns `Err`. The function falls into the `Err(_) => false` arm and returns `false` — the same default as "no refund needed". This is the wrong value: a failed `ft_transfer_call` means NEAR has already refunded the tokens back to the bridge contract, so the bridge **must** revert its own accounting.

Because `is_refund_required` returns `false`, the callback takes the `else` branch: [3](#0-2) 

Three critical cleanup steps are skipped:
1. `burn_tokens_if_needed` — for deployed tokens, the minted-but-refunded tokens are never burned, inflating supply.
2. `revert_lock_actions` — the `unlock_tokens_if_needed` call made earlier in `process_fin_transfer_to_near` is never reversed, so the locked-token counter is permanently decremented even though the tokens are back in the bridge.
3. `remove_fin_transfer` — the transfer ID stays in `finalised_transfers`, making re-finalization impossible.

The `lock_actions` vector is built and passed through the callback chain precisely to enable this revert: [4](#0-3) [5](#0-4) 

But because `is_refund_required` returns the wrong default, `revert_lock_actions` is never called.

---

### Impact Explanation

**For non-deployed (locked) tokens:**
- `unlock_tokens_if_needed` decrements the locked-token counter during `process_fin_transfer_to_near`.
- `ft_transfer_call` fails → NEAR refunds tokens back to bridge.
- `revert_lock_actions` is skipped → counter stays decremented.
- Bridge now reports fewer locked tokens than it actually holds, permanently breaking collateralization accounting.
- The transfer ID is in `finalised_transfers` → re-finalization is blocked forever.
- User's tokens are irrecoverably stuck.

**For deployed tokens:**
- `mint` is called, increasing total supply.
- `ft_on_transfer` panics → NEAR refunds minted tokens back to bridge.
- `burn_tokens_if_needed` is skipped → total supply remains inflated.
- Unbacked token supply is created.

Both outcomes match **Critical: Permanent freezing / irrecoverable lock of user funds** and **High: Balance/accounting corruption that breaks bridge collateralization**.

---

### Likelihood Explanation

The trigger path is fully reachable by an unprivileged bridge user:

1. User initiates a cross-chain transfer (e.g., Ethereum → NEAR) with a non-empty `msg` field targeting a NEAR DeFi contract.
2. Relayer calls `fin_transfer` with a valid proof; `fin_transfer_callback` → `process_fin_transfer_to_near` → `send_tokens` → `ft_transfer_call`.
3. The recipient contract's `ft_on_transfer` panics (e.g., slippage exceeded, pool drained, reentrancy guard, any revert condition).
4. NEAR refunds tokens to bridge; promise result is `Failed`.
5. `fin_transfer_send_tokens_callback` is called; `is_refund_required` returns `false`; state is not reverted.

No privileged role is required. Any transfer with a non-empty `msg` to any contract that can revert is sufficient.

---

### Recommendation

Change the `Err(_)` arm of `is_refund_required` to return `true`, so that a failed `ft_transfer_call` promise triggers the full revert path:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
            Ok(value) => {
                if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                    amount.0 == 0
                } else {
-                   // Unexpected case: don't refund
-                   false
+                   // Unexpected result format: treat as failed, refund
+                   true
                }
            }
-           // Unexpected case: don't refund
-           Err(_) => false,
+           // Promise failed (ft_on_transfer panicked): tokens are back in bridge, must revert
+           Err(_) => true,
        }
    } else {
        false
    }
}
```

### Proof of Concept

1. Deploy a NEAR contract whose `ft_on_transfer` always panics.
2. Initiate a transfer from Ethereum to NEAR with `msg = "trigger"` and `recipient = <panicking contract>`.
3. Relayer submits `fin_transfer` with valid proof.
4. `process_fin_transfer_to_near` unlocks tokens and calls `ft_transfer_call`.
5. `ft_on_transfer` panics; NEAR refunds tokens to bridge.
6. `fin_transfer_send_tokens_callback` is called; `is_refund_required` returns `false`.
7. Observe: `get_locked_tokens` shows the decremented (wrong) value; `finalised_transfers` contains the transfer ID; tokens are stuck in bridge with no recovery path.

    pub fn revert_lock_actions(&mut self, lock_actions: &[LockAction]) {
        for lock_action in lock_actions {
            match lock_action {
                LockAction::Locked {
                    chain_kind,
                    token_id,
                    amount,
                } => {
                    self.unlock_tokens(*chain_kind, token_id, *amount);
                }
                LockAction::Unlocked {
                    chain_kind,
                    token_id,
                    amount,
                } => {
                    self.lock_tokens(*chain_kind, token_id, *amount);
                }
                LockAction::Unchanged => {}
            }
        }
    }
