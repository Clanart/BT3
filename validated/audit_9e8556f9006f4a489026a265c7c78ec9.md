### Title
Stale `feemarketParams` Snapshot in `NewDynamicFeeChecker` Enables Fee Market Bypass for Cosmos Native Transactions - (File: `ante/evm/fee_checker.go`, `evmd/ante/handler_options.go`)

---

### Summary

`NewDynamicFeeChecker` in `ante/evm/fee_checker.go` receives `feemarketParams` as a pointer to a snapshot captured at handler-construction time inside `newCosmosAnteHandler`. Because the EIP-1559 base fee is updated every block in `BeginBlock`, the checker permanently uses a stale base fee for all Cosmos-native transaction fee validation. An unprivileged user can submit Cosmos-native transactions whose fee-per-gas is below the live base fee and have them accepted and committed, bypassing the fee market entirely for that transaction class.

---

---

### Impact Explanation

When `DynamicFeeChecker` is enabled, every Cosmos-native transaction (bank sends, governance votes, IBC relays, etc.) is validated by `NewDynamicFeeChecker`. The check:

```go
feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))
baseFeeInt := sdkmath.NewIntFromBigInt(baseFee)   // stale
if feeCap.LT(baseFeeInt) { return error }
``` [7](#0-6) 

uses the stale `baseFee`. If the live base fee has risen (e.g., from sustained full blocks), a user can submit a Cosmos-native transaction with `feeCap` equal to the stale (lower) base fee. The check passes, the transaction is committed, and the user pays less than the protocol requires. The `effectiveFee` deducted is also computed from the stale base fee, so the fee collector receives less than the current block's base fee demands. This is a direct mis-accounting of user fees and a bypass of the fee market for the Cosmos-native transaction path.

---

### Likelihood Explanation

The base fee changes every block via `CalculateBaseFee` / `BeginBlock`. On a chain with sustained load, the base fee grows monotonically. The staleness gap between the snapshot and the live value widens continuously. Any user who observes that the live base fee has risen above the snapshot value can immediately exploit the discrepancy by submitting Cosmos-native transactions with fees calibrated to the stale value. No special privileges, keys, or network position are required — only the ability to submit a standard Cosmos transaction.

---

### Recommendation

Replace the captured-snapshot pattern in `newCosmosAnteHandler` with a live-read pattern identical to `newEthAnteHandler`. Either:

1. Refactor `NewDynamicFeeChecker` to accept the `FeeMarketKeeper` interface and call `GetBaseFee(ctx)` on every invocation (matching the EVM path), or
2. Restructure `newCosmosAnteHandler` as a closure (like `newEthAnteHandler`) that fetches fresh `feemarketParams` from the keeper on every transaction, rather than capturing a one-time snapshot.

---

### Proof of Concept

1. Chain starts; genesis base fee = 1 000 000 000 (1 Gwei). `newCosmosAnteHandler` is constructed; `feemarketParams.BaseFee` = 1 Gwei is captured.
2. Sustained full blocks cause `CalculateBaseFee` to raise the live base fee to 10 Gwei over ~18 blocks (each block can raise it ≤12.5%).
3. Attacker submits a Cosmos bank-send with `fee = 1 Gwei × gasLimit` (calibrated to the stale snapshot).
4. `NewDynamicFeeChecker` computes `baseFee` from the captured `feemarketParams` → 1 Gwei. `feeCap` (1 Gwei) ≥ `baseFeeInt` (1 Gwei) → check passes.
5. The live base fee is 10 Gwei; the transaction should have been rejected. Instead it is committed and the fee collector receives 1/10th of the required fee.
6. The attacker can repeat this for every Cosmos-native message type, effectively paying 10× less than the protocol demands.
