### Title
Missing Chain ID in `deployToken` Signature Payload Enables Cross-Chain Replay — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `deployToken` function in `OmniBridge.sol` verifies an ECDSA signature over a `MetadataPayload` that contains no chain-specific domain separator. Because the NEAR MPC signs metadata with a fixed key path for all EVM chains, a valid `deployToken` signature obtained from one chain (e.g., Ethereum) can be replayed verbatim on any other EVM chain where the bridge is deployed (Arbitrum, Base, Polygon, BNB), deploying an unauthorized bridge token whose NEAR-side mapping does not exist, permanently locking user funds.

---

### Finding Description

`finTransfer` correctly binds its signed payload to a specific chain by embedding `omniBridgeChainId` twice: [1](#0-0) 

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.TransferMessage)),
    Borsh.encodeUint64(payload.destinationNonce),
    bytes1(payload.originChain),
    Borsh.encodeUint64(payload.originNonce),
    bytes1(omniBridgeChainId),          // ← chain-bound
    Borsh.encodeAddress(payload.tokenAddress),
    Borsh.encodeUint128(payload.amount),
    bytes1(omniBridgeChainId),          // ← chain-bound again
    ...
```

`deployToken`, however, signs only token metadata with **no chain identifier**: [2](#0-1) 

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
    // ← no omniBridgeChainId, no block.chainid
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
```

On the NEAR side, the MPC signs metadata via `log_metadata_callback` using the fixed path `"bridge-1"` for every chain: [3](#0-2) 

The `MetadataPayload` serialized there also contains no chain discriminator: [4](#0-3) 

Because the same MPC key and path are used for all EVM deployments, the resulting signature is **chain-agnostic**. The only on-chain guard against re-use is: [5](#0-4) 

```solidity
require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST");
```

This check only prevents re-deployment on the **same** chain; it does nothing to block replay on a **different** chain where `nearToEthToken[metadata.token]` is still `address(0)`.

---

### Impact Explanation

An attacker who replays a legitimate Ethereum `deployToken` call onto Arbitrum (or any other supported EVM chain) causes:

1. A bridge token is deployed on Arbitrum with correct metadata but **no corresponding NEAR-side token mapping** (`token_decimals`, `token_id_to_address`, `token_address_to_id`).
2. Users who call `initTransfer` on Arbitrum with this token have their tokens burned/locked on Arbitrum.
3. When a relayer submits the proof to NEAR's `fin_transfer`, the callback panics at: [6](#0-5) 

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
```

4. The transfer is permanently unclaimable; user funds are irrecoverably locked.

This matches two allowed impact classes:
- **Critical/High**: Permanent freezing / irrecoverable lock of user funds in bridge flows.
- **High**: Signature verification bypass enabling unauthorized token deployment.

---

### Likelihood Explanation

- The `deployToken` calldata (signature + metadata) is publicly visible on-chain the moment the legitimate deployment is executed on Ethereum.
- Any unprivileged user can copy those two arguments and call `deployToken` on any other OmniBridge instance.
- No special access, leaked key, or privileged role is required.
- The bridge is explicitly deployed on multiple EVM chains (Ethereum, Arbitrum, Base, Polygon, BNB), making the replay surface wide.

---

### Recommendation

Include `omniBridgeChainId` (or `block.chainid`) in the borsh-encoded metadata payload before it is sent to the NEAR MPC for signing, mirroring the pattern already used in `finTransfer`:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
+   bytes1(omniBridgeChainId),          // bind to this chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

The NEAR `log_metadata_callback` must include the target chain kind in the `MetadataPayload` it signs so the MPC-produced signature is chain-specific.

---

### Proof of Concept

```
1. NEAR operator calls log_metadata("usdc.near") on NEAR bridge.
2. NEAR MPC signs MetadataPayload{token="usdc.near", name="USD Coin",
   symbol="USDC", decimals=6} → sig S (no chain ID in payload).
3. Relayer calls OmniBridge(Ethereum).deployToken(S, metadata)
   → legitimate USDC bridge token deployed on Ethereum.
   → NEAR bridge registers token_decimals for Ethereum USDC.

4. Attacker reads S and metadata from Ethereum tx calldata.
5. Attacker calls OmniBridge(Arbitrum).deployToken(S, metadata)
   → nearToEthToken["usdc.near"] == address(0) on Arbitrum → check passes
   → ECDSA.recover(keccak256(same borsh), S) == nearBridgeDerivedAddress → passes
   → Unauthorized USDC bridge token deployed on Arbitrum.
   → No NEAR-side mapping exists for Arbitrum USDC.

6. Victim calls OmniBridge(Arbitrum).initTransfer(arbitrumUSDC, 1000e6, ...)
   → 1000 USDC burned on Arbitrum.

7. Relayer submits proof to NEAR fin_transfer.
8. fin_transfer_callback panics: BridgeError::TokenDecimalsNotFound
   (no token_decimals entry for Arbitrum USDC address).

9. Victim's 1000 USDC is permanently lost.
```
