The vulnerability is real. Here is the analysis:

### Title
UTXO Fast-Path Near-Destination Finalization Leaves No Deduplication Record, Enabling Double-Finalization — (`near/omni-bridge/src/lib.rs`)

### Summary

When a UTXO inbound transfer (BTC→NEAR) is finalized via the fast-transfer path with a **Near** destination, `utxo_fin_transfer_fast` calls `remove_fast_transfer` (erasing the only record of the transfer) without inserting into `finalised_utxo_transfers`. This leaves the contract in a state where a second `ft_transfer_call` from the connector with the same UTXO ID passes all guards and mints tokens a second time.

### Finding Description

**Entry point — `utxo_fin_transfer`** (called via `ft_on_transfer` when the connector does `ft_transfer_call`): [1](#0-0) 

```
if let Some(status) = self.get_fast_transfer_status(&fast_transfer.id()) {
    return self.utxo_fin_transfer_fast(fast_transfer, status, utxo_fin_transfer_msg);
}
// Only reached on the slow path:
let required_storage_balance =
    self.add_fin_utxo_transfer(&utxo_fin_transfer_msg.get_transfer_id(origin_chain));
```

When a fast transfer exists, the function returns early from `utxo_fin_transfer_fast` **without ever calling `add_fin_utxo_transfer`**.

**Inside `utxo_fin_transfer_fast`** — the Near vs. non-Near asymmetry: [2](#0-1) 

```
require!(!fast_transfer_status.finalised, ...);

let amount = if fast_transfer.get_destination_chain() == ChainKind::Near {
    self.remove_fast_transfer(&fast_transfer.id());   // ← entry DELETED
    fast_transfer.amount
} else {
    self.mark_fast_transfer_as_finalised(&fast_transfer.id()); // ← entry kept, finalised=true
    ...
};
```

- **Non-Near path**: `mark_fast_transfer_as_finalised` keeps the entry in `fast_transfers` with `finalised=true`. A second call hits the `require!(!finalised)` guard and panics. ✅ Protected.
- **Near path**: `remove_fast_transfer` deletes the entry entirely from `fast_transfers`. No entry is added to `finalised_utxo_transfers`. After this call, **both maps have zero record of this UTXO ID**. ❌ Unprotected. [3](#0-2) 

**Second call with the same UTXO ID:**

`utxo_fin_transfer` is called again. `get_fast_transfer_status` returns `None` (entry was deleted). The code falls through to `add_fin_utxo_transfer`, which succeeds because `finalised_utxo_transfers` has no record of this UTXO ID: [4](#0-3) 

The second finalization proceeds, minting tokens to the recipient a second time.

**The `fails_on_double_finalization` test does not cover this case.** It only tests the non-Near destination (uses `base_eoa_address()`), which is protected by the `mark_fast_transfer_as_finalised` path: [5](#0-4) 

**Connector access control:** The `utxo_fin_transfer` guard requires `sender_id == config.connector`. The connector is a separate contract. The mock connector's `verify_deposit` is publicly callable with no access control: [6](#0-5) 

If the production connector similarly exposes a public deposit-verification entry point (which is architecturally necessary for relayers to submit BTC proofs), any account can trigger the second `ft_transfer_call` with the same UTXO ID.

### Impact Explanation

Double-minting of bridged BTC tokens. Each exploited UTXO ID produces two token mints against one on-chain BTC deposit, creating unbacked supply and draining the bridge's collateral.

### Likelihood Explanation

The fast-transfer path for Near destinations is a normal, intended flow. The missing `add_fin_utxo_transfer` call is a straightforward omission. The `fails_on_double_finalization` test's gap (only testing non-Near destinations) confirms the case was not considered. Exploitability depends on whether the production connector's deposit-verification function is publicly callable, which is architecturally expected for a permissionless bridge.

### Recommendation

In `utxo_fin_transfer_fast`, when the destination is Near, insert into `finalised_utxo_transfers` **before** removing the fast transfer entry:

```rust
let amount = if fast_transfer.get_destination_chain() == ChainKind::Near {
    // Add to finalised set FIRST to prevent double-finalization
    self.add_fin_utxo_transfer(
        &utxo_fin_transfer_msg.get_transfer_id(origin_chain)
    );
    self.remove_fast_transfer(&fast_transfer.id());
    fast_transfer.amount
} else {
    self.mark_fast_transfer_as_finalised(&fast_transfer.id());
    ...
};
```

Also add a test case for `fails_on_double_finalization` with a Near recipient.

### Proof of Concept

State trace (all within the bridge contract):

1. Relayer calls `fast_fin_transfer` for UTXO ID `X` with `recipient = Near(alice)` → `fast_transfers[X] = {finalised: false, relayer: R}`
2. Connector calls `ft_transfer_call` → `ft_on_transfer` → `utxo_fin_transfer`:
   - `get_fast_transfer_status(X)` → `Some({finalised: false})`
   - `utxo_fin_transfer_fast` called; destination is Near → `remove_fast_transfer(X)` → `fast_transfers` is now empty for `X`
   - `finalised_utxo_transfers` is still empty for `X`
   - Returns `U128(0)` (connector keeps 0 tokens, bridge mints to relayer)
3. Connector calls `ft_transfer_call` again with same UTXO ID `X`:
   - `get_fast_transfer_status(X)` → `None`
   - Falls through to `add_fin_utxo_transfer(X)` → **succeeds** (set was empty)
   - `utxo_fin_transfer_to_near` called → tokens minted to `alice` a second time

Result: `alice` receives tokens twice for a single BTC deposit. `finalised_utxo_transfers` contains `X` only after the second call, which is too late.

    fn remove_fast_transfer(&mut self, fast_transfer_id: &FastTransferId) {
        let storage_usage = env::storage_usage();
        let fast_transfer = self
            .fast_transfers
            .remove(fast_transfer_id)
            .map(storage::FastTransferStatusStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&fast_transfer.storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances
                .insert(&fast_transfer.storage_owner, &storage);
        }
    }
