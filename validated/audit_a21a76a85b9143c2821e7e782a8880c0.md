### Title
Asymmetric `within_margin` check enables L1 gas price ratcheting via oracle-fallback reference manipulation — (`crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_block_info_valid` validates a proposed L1 gas price by calling `get_l1_prices_in_fri_and_wei` to obtain a local reference price, then calling `within_margin(proposed, reference, margin_percent)`. The `within_margin` function computes the allowed absolute deviation from the **proposed** value, not the **reference** value. When the ETH-to-STRK oracle or L1 gas price provider fails, `get_l1_prices_in_fri_and_wei_and_conversion_rate` silently falls back to `previous_block_info` prices as the reference. Because `previous_block_info` is populated directly from the last accepted `ProposalInit`, a proposer who controls consecutive blocks during oracle-unavailability windows can exploit the asymmetric margin to ratchet the L1 gas price embedded in block headers upward, block by block, far above the true L1 gas price. The resulting inflated gas price is committed into the block header, used for all fee calculations in that block, and hashed into the block commitment.

---

### Finding Description

**Root cause — asymmetric margin in `within_margin`** [1](#0-0) 

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;   // ← margin from PROPOSED, not reference
    number1.0.abs_diff(number2.0) <= margin
}
```

Called as `within_margin(proposed, reference, margin_percent)`. The margin is `proposed × margin_percent / 100`. When `proposed > reference` the margin grows proportionally to the proposed value, making the check easier to satisfy the higher the proposed price is. Concretely, the maximum allowed proposed price is `reference / (1 − margin_percent/100)`, while the minimum is `reference / (1 + margin_percent/100)` — an asymmetric window that is wider on the upside.

**Trigger — oracle-failure fallback writes proposer-controlled value as reference** [2](#0-1) 

When either `get_eth_to_fri_rate` or `get_price_info` fails, `get_l1_prices_in_fri_and_wei_and_conversion_rate` falls back to `previous_block_info`:

```rust
if let Some(block_info) = previous_block_info {
    let prev_l1_gas_price_wei = block_info.l1_prices_wei.clone();
    let prev_l1_gas_price = block_info.l1_prices_fri.clone();
    ...
    return (prev_l1_gas_price, prev_l1_gas_price_wei, eth_to_fri_rate);
}
```

`previous_block_info` is populated from the last accepted `ProposalInit` after `decision_reached`. The proposer of the previous block therefore controls the reference value used by the validator during any subsequent oracle-failure window. [3](#0-2) 

**Exploit chain**

Let `m = l1_gas_price_margin_percent` and `R₀` be the true L1 gas price.

1. **Block N (oracle available):** Proposer A proposes `P_N = R₀ × 1/(1−m/100)` — the maximum the asymmetric check allows above the oracle reference `R₀`. Block is accepted; `previous_block_info.l1_gas_price_fri = P_N`.

2. **Block N+1 (oracle fails):** Validator falls back to `previous_block_info`; reference is now `P_N`. Proposer A proposes `P_{N+1} = P_N × 1/(1−m/100) = R₀ × [1/(1−m/100)]²`. The `within_margin` check passes because the margin is computed from the inflated proposed value.

3. **Repeat:** Each oracle-failure block multiplies the embedded gas price by `1/(1−m/100)`. After `k` such blocks the committed price is `R₀ × [1/(1−m/100)]^k`, unbounded in the number of oracle-failure blocks.

The oracle client (`EthToStrkOracleClient`) already documents natural failure modes — `QueryNotReadyError` when the async HTTP query has not yet resolved, `AllUrlsFailedError` when all configured endpoints are unreachable, and `JoinError` on task failure — all of which silently activate the fallback path. [4](#0-3) 

**Corrupted value committed to chain**

The inflated `l1_gas_price_fri` / `l1_data_gas_price_fri` from `ProposalInit` is passed directly into `convert_to_sn_api_block_info`, which constructs the `BlockInfo` used for all transaction fee calculations in that block. [5](#0-4) 

It is also embedded in `PartialBlockHashComponents` and hashed into the block commitment via `calculate_block_hash`. [6](#0-5) 

---

### Impact Explanation

The L1 gas price embedded in the block header is the authoritative price used by the blockifier to charge fees for every transaction in the block. An inflated price causes:

- Every transaction in the affected block to be charged fees proportional to the manipulated price, extracting excess value from users.
- Transactions whose `max_price_per_unit` resource bound was set against the true L1 gas price to be rejected with `MaxGasPriceTooLow`, denying service.
- The block hash / state commitment to encode the wrong gas price, making the committed state diverge from what honest nodes would compute from true L1 data.

This matches **"Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."**

---

### Likelihood Explanation

The trigger requires two concurrent conditions:

1. **Oracle unavailability** — the `EthToStrkOracleClient` has documented, naturally occurring failure modes (`QueryNotReadyError`, `AllUrlsFailedError`). The production config sets `query_timeout_sec = 10` and `lag_interval_seconds = 900`; any 10-second HTTP outage during a block proposal activates the fallback.

2. **Proposer controls the previous block** — in a BFT round-robin or stake-weighted proposer selection, a single validator can propose multiple consecutive blocks. The attacker does not need to cause the oracle failure; they only need to be the proposer when a natural failure occurs.

The asymmetric margin means even a single oracle-failure block yields a price above the true reference, and the effect compounds across multiple such blocks.

---

### Recommendation

1. **Fix the margin computation** — compute the margin from the **reference** value, not the proposed value:

```rust
fn within_margin(proposed: GasPrice, reference: GasPrice, margin_percent: u128) -> bool {
    if proposed.0.abs_diff(reference.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (reference.0 * margin_percent) / 100;   // ← use reference
    proposed.0.abs_diff(reference.0) <= margin
}
```

2. **Bound the fallback** — when falling back to `previous_block_info`, apply a maximum staleness cap (e.g., reject if the previous block's timestamp is more than N seconds old) so a prolonged oracle outage does not silently propagate a stale price indefinitely.

3. **Decouple reference from proposer control** — consider requiring the validator to independently verify the L1 gas price against its own oracle before accepting a proposal, rather than accepting the proposer's price as the reference for future blocks.

### Proof of Concept

**Setup:** `margin_percent = 10`, true L1 gas price `R₀ = 1000 fri`.

**Block N (oracle available):**
- Oracle returns `R₀ = 1000`.
- Proposer sets `proposed = 1000 / (1 − 0.10) = 1111`.
- `within_margin(1111, 1000, 10)`: `margin = 1111 × 10 / 100 = 111`; `|1111 − 1000| = 111 ≤ 111`. **Passes.**
- `previous_block_info.l1_gas_price_fri = 1111`.

**Block N+1 (oracle fails — `QueryNotReadyError`):**
- Fallback: reference = `1111`.
- Proposer sets `proposed = 1111 / 0.9 = 1234`.
- `within_margin(1234, 1111, 10)`: `margin = 1234 × 10 / 100 = 123`; `|1234 − 1111| = 123 ≤ 123`. **Passes.**
- Committed `l1_gas_price_fri = 1234` (23.4% above true price after 2 blocks).

**After 10 oracle-failure blocks:** committed price ≈ `1000 × (1/0.9)^10 ≈ 2868 fri` — nearly 3× the true L1 gas price — causing every transaction in those blocks to pay ~3× the correct fee.

fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    // For small numbers (e.g., less than 10 wei, if margin is 10%), even an off-by-one
    // error might be bigger than the margin, even if it is just a rounding error.
    // We make an exception for such mismatch, and don't bother checking percentages
    // if the difference in price is only one wei.
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}
