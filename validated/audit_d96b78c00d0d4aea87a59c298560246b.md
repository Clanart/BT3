### Title
Fee Validation Does Not Account for Decimal Normalization, Causing Permanent Fund Lock — (`near/omni-bridge/src/lib.rs`)

### Summary

The `update_transfer_fee` and `init_transfer` functions validate `fee < amount` but do not verify that `normalize_amount(amount - fee) > 0`. For tokens where `origin_decimals > decimals`, if `amount - fee` is smaller than the normalization factor (`10^(origin_decimals - decimals)`), the normalized transfer amount becomes 0. This causes `sign_transfer` to permanently revert. Because fees can only be increased (never decreased) and no cancel mechanism exists, the user's bridged funds become permanently and irrecoverably locked.

### Finding Description

**Root cause — `update_transfer_fee`:** [1](#0-0) 

The only fee constraint is `fee.fee < transfer.message.amount`. There is no check that `normalize_amount(amount - fee) > 0`.

**Root cause — `init_transfer`:** [2](#0-1) 

Same missing check at transfer creation time.

**Downstream failure — `sign_transfer`:** [3](#0-2) 

`normalize_amount` applies floor division by `10^(origin_decimals - decimals)`: [4](#0-3) 

If `amount_without_fee < 10^(origin_decimals - decimals)`, the result is 0 and `sign_transfer` reverts with `InvalidAmountToTransfer`. Every subsequent relayer call hits the same revert.

**Fee can only increase, never decrease:** [1](#0-0) 

`fee.fee >= current_fee.fee` is enforced, so once the fee is set to `amount - 1`, it cannot be lowered. There is no cancel or refund path for pending transfer messages.

### Impact Explanation

A user's tokens are permanently locked in the NEAR bridge contract. `sign_transfer` always reverts (normalized amount = 0), `claim_fee` requires a destination-chain finalization proof that can never exist, and `update_transfer_fee` cannot lower the fee. The transfer message persists in storage indefinitely with no recovery path. This matches the allowed impact: **permanent freezing / irrecoverable lock of user funds in bridge flows**.

### Likelihood Explanation

Any token registered with `origin_decimals > decimals` (e.g., a 24-decimal NEAR-native token bridged to an 18-decimal EVM representation, normalization factor = 10^6) is affected. A user who sets `fee = amount - 1` — either deliberately to maximize relayer incentive or accidentally — triggers the lock. The `update_transfer_fee` path is callable by any unprivileged bridge user who is the sender of a pending transfer.

### Recommendation

In both `init_transfer` and `update_transfer_fee`, add a check that the normalized net amount is positive:

```rust
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(addr) = token_address {
    if let Some(decimals) = self.token_decimals.get(&addr) {
        let net = transfer_message.amount.0
            .checked_sub(fee.fee.0)
            .expect("fee < amount already checked");
        require!(
            Self::normalize_amount(net, decimals) > 0,
            BridgeError::InvalidAmountToTransfer.as_ref()
        );
    }
}
```

Alternatively, enforce a protocol-level minimum `amount - fee` that is at least `10^(origin_decimals - decimals)` for every token at registration time.

### Proof of Concept

1. Token `T` is registered with `origin_decimals = 24`, `decimals = 18` (normalization factor = 10^6).
2. User calls `ft_transfer_call` on NEAR with `amount = 1_000_001` and `fee = 0`, creating a pending `TransferMessage` with `amount = 1_000_001`.
3. User calls `update_transfer_fee(transfer_id, Fee { fee: 1_000_000, native_fee: 0 })`. Validation passes: `1_000_000 >= 0` and `1_000_000 < 1_000_001`. [1](#0-0) 
4. Relayer calls `sign_transfer`. `amount_without_fee = 1`. `normalize_amount(1, decimals) = 1 / 10^6 = 0`. [3](#0-2) 
5. `require!(amount_to_transfer > 0, ...)` panics. Every relayer call hits the same revert.
6. User cannot lower the fee (`fee.fee >= current_fee.fee` enforced). No cancel function exists. The 1,000,001 tokens are permanently locked.
