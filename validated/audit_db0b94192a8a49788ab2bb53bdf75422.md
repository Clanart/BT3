Looking at the full validation chain in `set_state_header` and the `verify_path` / `validate_chunk_proofs` implementations to determine whether the shard_id binding is enforced.

### Title
Missing `chunk.shard_id()` == `shard_id` Guard in `set_state_header` Allows Cross-Shard State-Root Substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from an untrusted peer and stores it under `StateHeaderKey(shard_id, sync_hash)`. The function validates that the embedded chunk is internally consistent and that it appears somewhere in the block's `chunk_headers_root` Merkle tree, but it **never checks that `chunk.shard_id()` equals the caller-supplied `shard_id`**. An unprivileged state-sync peer can therefore serve shard X's chunk (with shard X's `prev_state_root`) while the victim node believes it is receiving shard Y's header. The stored header then drives all subsequent `set_state_part` validation against the wrong state root.

---

### Finding Description

**Entrypoint:** `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`, reachable from `ClientActor::handle(StateHeaderValidationRequest)` in `chain/client/src/client_actor.rs`. [1](#0-0) 

The function performs five validation steps:

1. **`validate_chunk_proofs`** — verifies the chunk's internal hash and body consistency. It does not inspect `shard_id`. [2](#0-1) 

2. **`verify_path` against `sync_prev_block_header.chunk_headers_root()`** — proves the chunk appears *somewhere* in the block's chunk-header Merkle tree. [3](#0-2) 

   `verify_path` is a pure path-traversal check: it computes `compute_root_from_path(path, hash(item)) == root`. It does **not** verify the leaf's position/index in the tree, so a proof for shard X's chunk at tree-index 0 passes even when the caller claims shard Y (index 1). [4](#0-3) 

3. **Receipt proof validation** — uses the *requested* `shard_id` (line 488: `ReceiptList(shard_id, receipts)`), not `chunk.shard_id()`. An attacker who can observe the chain can supply valid receipt proofs for the requested shard Y while embedding shard X's chunk. [5](#0-4) 

4. **`validate_state_root_node`** — validates the `state_root_node` against `chunk_inner.prev_state_root()`. Since the attacker controls both the chunk (shard X's) and the `state_root_node` (shard X's), this passes trivially. [6](#0-5) 

5. **Storage** — the header is written under `StateHeaderKey(shard_id, sync_hash)` using the *requested* `shard_id`, not the chunk's actual shard. [7](#0-6) 

**The grep for `chunk.shard_id()` inside `adapter.rs` returns zero matches**, confirming no such guard exists anywhere in the function.

---

### Impact Explanation

After the malicious header is stored, `set_state_part` reads it back:

```rust
let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
let chunk = shard_state_header.take_chunk();
let state_root = *chunk.take_header().take_inner().prev_state_root();
// state_root is now shard X's root, not shard Y's
self.runtime_adapter.validate_state_part(shard_id, &state_root, part_id, part)
``` [8](#0-7) 

The attacker then serves valid state parts for shard X. These parts validate correctly against shard X's `state_root`. They are then installed as shard Y's state. The syncing node ends up tracking shard X's account/contract state under shard Y's identity, causing silent data corruption: wrong account balances, wrong contract storage, and incorrect chunk production for shard Y.

---

### Likelihood Explanation

Any node on the network can respond to state-sync header requests — no validator key or special privilege is required. The attacker only needs to:
- Know a valid `sync_hash` (public information from the chain)
- Serve a crafted `ShardStateSyncResponseHeader` with shard X's chunk + shard X's Merkle proof + valid shard Y receipt proofs (all computable from public chain data)

The victim node has no other guard: `set_state_header` is the sole validation gate before the header is persisted.

---

### Recommendation

Add an explicit shard-id equality check immediately after extracting the chunk, before any other validation:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk.shard_id() does not match requested shard_id".into(),
    ));
}
``` [9](#0-8)

### Proof of Concept

```rust
// In an integration test environment with two shards (shard_id 0 and 1):
// 1. Obtain a valid ShardStateSyncResponseHeader for shard 0.
let header_shard0 = adapter.get_state_response_header(shard_id_0, sync_hash).unwrap();

// 2. Call set_state_header claiming it is for shard 1.
//    All five validation steps pass because:
//    - validate_chunk_proofs: shard 0's chunk is internally valid
//    - verify_path: shard 0's chunk IS in the block's chunk_headers_root
//    - receipt proofs: supply valid proofs for shard 1 (observable from chain)
//    - validate_state_root_node: shard 0's node matches shard 0's state_root
adapter.set_state_header(shard_id_1, sync_hash, header_shard0.clone()).unwrap();

// 3. Read back the stored header for shard 1.
let stored = adapter.get_state_header(shard_id_1, sync_hash).unwrap();

// 4. Assert the embedded chunk belongs to shard 0, not shard 1.
assert_eq!(stored.cloned_chunk().shard_id(), shard_id_0); // passes — bug confirmed
assert_ne!(stored.cloned_chunk().shard_id(), shard_id_1);
```

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
