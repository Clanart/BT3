### Title
Missing Zero-Address Validation on Transfer Recipient Causes Permanent Lock of User Funds - (`evm/src/omni-bridge/contracts/OmniBridge.sol` / `near/omni-bridge/src/lib.rs`)

### Summary
NEAR's `init_transfer` does not validate that an EVM-chain recipient address is non-zero. A user who initiates a NEARãEVM bridge transfer with `recipient = address(0)` will have their tokens permanently locked: the NEAR side locks/burns the tokens, but every subsequent call to `finTransfer` on the EVM side reverts because OpenZeppelin ERC20/ERC1155 forbids transfers to `address(0)`. There is no cancel or refund path on NEAR, making the lock irrecoverable.

### Finding Description

**Step 1 ã No zero-address guard in NEAR `init_transfer`**

`init_transfer` in `near/omni-bridge/src/lib.rs` validates only that the recipient chain is not NEAR:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
```

No check is made that the EVM address component of `OmniAddress::Eth(...)` (or Arb, Base, Bnb, Pol, etc.) is non-zero. `OmniAddress::is_zero()` exists in `near/omni-types/src/lib.rs` but is never called here. [1](#0-0) [2](#0-1) 

**Step 2 ã MPC signs the payload containing the zero recipient**

`sign_transfer` reads the stored `transfer_message.recipient` (which is the zero address) and embeds it verbatim into `TransferMessagePayload`, then asks the MPC signer to sign it. No validation is performed on the recipient value at this stage. [3](#0-2) 

**Step 3 ã EVM `finTransfer` has no zero-address guard on `payload.recipient`**

After verifying the MPC signature, `finTransfer` in `OmniBridge.sol` immediately attempts to deliver tokens to `payload.recipient` without checking whether it is `address(0)`:

```solidity
} else if (isBridgeToken[payload.tokenAddress]) {
    IBridgeToken(payload.tokenAddress).mint(
        payload.recipient,   // ã no zero-address check
        payload.amount
    );
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,   // ã no zero-address check
        payload.amount
    );
}
```

OpenZeppelin's `ERC20._mint` and `ERC20._transfer` both revert with `"ERC20: mint/transfer to the zero address"`. OpenZeppelin's `ERC1155._safeTransferFrom` similarly reverts. The entire transaction reverts, including the `completedTransfers[nonce] = true` write, so the nonce is never consumed and the call can be retried ã but it will always revert. [4](#0-3) [5](#0-4) 

**Step 4 ã No cancel/refund path on NEAR**

Once `init_transfer` completes, the tokens are locked (or burned for bridge tokens) on NEAR and the transfer message is stored. No public cancel or refund function was found in `near/omni-bridge/src/lib.rs`. The funds are irrecoverable. [6](#0-5) 

### Impact Explanation

A user who bridges any ERC20 or ERC1155 token to `address(0)` on an EVM chain will have their tokens permanently locked in the NEAR bridge contract. The EVM `finTransfer` will revert on every attempt, and there is no on-chain mechanism to reclaim the locked funds. This matches the allowed impact: **Critical ã permanent freezing / irrecoverable lock of user funds in bridge flows**.

### Likelihood Explanation

Any unprivileged user can trigger this by calling `ft_transfer_call` on NEAR with `recipient = "eth:0x0000000000000000000000000000000000000000"`. This can happen accidentally (copy-paste error, uninitialized address) or intentionally. The entry path is fully public and requires no special role.

### Recommendation

1. In NEAR `init_transfer`, add a check that the recipient `OmniAddress` is not the zero address before storing the transfer:
   ```rust
   require!(
       !init_transfer_msg.recipient.is_zero(),
       BridgeError::InvalidRecipient.as_ref()
   );
   ```
2. In EVM `finTransfer`, add a guard before any token delivery:
   ```solidity
   require(payload.recipient != address(0), "ERR_ZERO_RECIPIENT");
   ```
3. Consider adding a cancel/refund mechanism on NEAR so that transfers whose EVM finalization is permanently broken can be unwound.

### Proof of Concept

1. User calls `ft_transfer_call` on NEAR token contract with `msg` containing `InitTransferMsg { recipient: OmniAddress::Eth(EvmAddress::zero()), fee: ..., ... }`.
2. NEAR `init_transfer` passes the only check (`recipient.get_chain() != ChainKind::Near`) and stores the transfer with zero EVM recipient. Tokens are locked.
3. Relayer calls `sign_transfer(transfer_id, fee_recipient, fee)`. MPC signs `TransferMessagePayload { recipient: OmniAddress::Eth(0x000...000), ... }`.
4. Anyone calls `OmniBridge.finTransfer(signature, payload)` on the EVM chain.
5. `completedTransfers[nonce] = true` is written, then `IERC20(token).safeTransfer(address(0), amount)` is called ã reverts with `"ERC20: transfer to the zero address"`. The entire transaction reverts; nonce is not consumed.
6. Step 4ã5 can be repeated indefinitely; it always reverts. Tokens remain locked on NEAR with no recovery path.
