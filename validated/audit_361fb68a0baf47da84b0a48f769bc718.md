### Title
Unconstrained `is_reverted` Hint Enables Sequencer to Skip Transaction Execution While Charging Fees — (File: `execution/execution_constraints.cairo`)

### Summary
The `check_is_reverted` function in `execution_constraints.cairo` is a documented no-op. Combined with the unverified `%{ IsReverted %}` hint in `transaction_impls.cairo`, the OS provides zero proof-level constraint on whether a transaction's execute step is skipped. A sequencer running a modified hint processor can set `is_reverted = 1` for any invoke transaction, causing the OS to skip execution while still charging the user's fee — a direct loss of funds with a valid proof.

### Finding Description

In `execute_invoke_function_transaction` (`transaction_impls.cairo`, lines 338–358), the local variable `is_reverted` is populated exclusively by the `%{ IsReverted %}` hint:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    with remaining_gas {
        cap_remaining_gas(max_gas=EXECUTE_MAX_SIERRA_GAS);
        non_reverting_select_execute_entry_point_func(...);
    }
} else {
    // Align the stack ... (no execution)
}
// Charge fee regardless.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [1](#0-0) 

The call to `check_is_reverted` is supposed to validate this value. However, the function body is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [2](#0-1) 

This is a complete no-op. There is no `assert`, no range check, and no Cairo constraint of any kind on `is_reverted`. The hint `%{ IsReverted %}` is the sole source of truth, and hints are Python code executed by the prover — they are **not** part of the proof and impose no soundness guarantee.

The analog to the external report's "undocumented assembly blocks" is exact: just as Solidity assembly bypasses the compiler's safety features, Cairo `%{ %}` hint blocks bypass the VM's constraint system. Here, the hint sets a value that controls a critical branch (execute vs. skip), and the named "check" function that should constrain it does nothing.

### Impact Explanation

When `is_reverted = 1`:
- The `execute` step is skipped entirely — no state changes are applied.
- `charge_fee` is still called unconditionally after the branch, deducting the fee from the user's account. [3](#0-2) 

The user's funds are taken as a fee, but the transaction's intended effects never occur. This is **direct loss of funds**. The resulting proof is fully valid because there are no Cairo constraints on `is_reverted`.

### Likelihood Explanation

The sequencer controls the hint processor. No key compromise or external dependency is required — only a modified hint implementation that returns `is_reverted = 1` for targeted transactions. The OS, which is the intended trust anchor constraining sequencer behavior, provides no proof-level defense against this. The `check_is_reverted` no-op is the root cause.

### Recommendation

Replace the no-op `check_is_reverted` with a Cairo assertion derived from verifiable execution state. The `is_reverted` flag should be constrained by the actual syscall and gas accounting results already present in the OS — for example, by asserting consistency with the `failure_flag` returned from `execute_entry_point`, which is already constrained via `entry_point_return_values.failure_flag`. [4](#0-3) 

### Proof of Concept

1. User submits a valid invoke transaction with sufficient gas and a correct signature.
2. Sequencer's hint processor evaluates `%{ IsReverted %}` and returns `is_reverted = 1`.
3. `check_is_reverted(1)` is called — it immediately returns, imposing no constraint.
4. The `if (is_reverted == FALSE)` branch is not taken; `non_reverting_select_execute_entry_point_func` is never called.
5. `charge_fee` executes unconditionally, deducting the fee from the user's account via an ERC-20 transfer.
6. The transaction's intended state changes are never applied.
7. The STARK proof is valid — no Cairo constraint was violated.
8. The user has lost funds with no recourse, and the sequencer has produced a provably "correct" block that silently dropped the execution.
