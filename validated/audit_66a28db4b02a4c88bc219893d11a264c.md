### Title
Integer Wrapping in `check_time_locks` Bypasses `ASSERT_SECONDS_RELATIVE` / `ASSERT_HEIGHT_RELATIVE` Timelocks on Legacy `nowrap=false` Path — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `check_time_locks` is called with `nowrap=false` (the pre-hard-fork legacy path), relative timelock arithmetic uses `wrapping_add` instead of `saturating_add`. An attacker can craft a puzzle that outputs `ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE` with a large value that causes integer overflow, wrapping the computed deadline to a small number (e.g., 0). The comparison then trivially passes, allowing the coin to be spent immediately — bypassing the intended timelock entirely.

---

### Finding Description

In `check_time_locks`, the `nowrap` flag selects between two arithmetic modes for relative timelock enforcement:

**`ASSERT_HEIGHT_RELATIVE` (lines 64–68):**
```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

**`ASSERT_SECONDS_RELATIVE` (lines 75–77):**
```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [1](#0-0) 

When `nowrap=false`, `wrapping_add` is used. If `confirmed_block_index + height_relative` overflows `u32::MAX`, the result wraps to a small value (e.g., 9 when `confirmed_block_index=100` and `height_relative=0xFFFF_FFFF`). The check `prev_height < 9` is then `false`, so no error is returned — the timelock is silently bypassed.

The same applies to `ASSERT_BEFORE_HEIGHT_RELATIVE` and `ASSERT_BEFORE_SECONDS_RELATIVE` in the opposite direction (lines 90–98 and 107–111): wrapping causes a spend that should be valid to be incorrectly rejected, producing consensus disagreement. [2](#0-1) 

The codebase's own tests explicitly document and confirm this behavior:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),  // <-- bypass: no error returned
)]
``` [3](#0-2) 

```
// 2000 < 1000 + u64::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::seconds_relative_wrap(
    Osc { seconds_relative: Some(0xffff_ffff_ffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed)),
    Ok(()),  // <-- bypass: no error returned
)]
``` [4](#0-3) 

The Python binding exposes `nowrap` as a caller-controlled parameter, and the full node passes it based on block height (pre/post hard fork): [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Impact class: High — timelock condition validation bypass enabling unauthorized spend acceptance.**

A coin whose puzzle outputs `ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE` with a carefully chosen large value (one that causes `coin_timestamp + delay` to wrap past `u64::MAX` / `u32::MAX`) can be spent immediately on the `nowrap=false` path. The timelock condition — which is the sole mechanism preventing premature spending — is silently accepted as satisfied. Any protocol (payment channel, vault, escrow, or time-locked custody) that relies on these conditions for fund protection is vulnerable on nodes using the legacy path.

---

### Likelihood Explanation

The `nowrap=false` path is the legacy pre-hard-fork code path. The Python binding accepts `nowrap` as a direct parameter from the calling full node, meaning any full node that has not yet transitioned (or that validates historical blocks) will invoke this path. The attacker-controlled entry is fully unprivileged: any user can create a coin with a puzzle that outputs an arbitrary `ASSERT_SECONDS_RELATIVE` value. No special role, key, or governance access is required. The wrapping threshold is deterministic and computable from the coin's confirmation timestamp.

---

### Recommendation

Replace `wrapping_add` with `saturating_add` unconditionally for all relative timelock arithmetic in `check_time_locks`. The `nowrap=false` branch should be removed or made equivalent to the `nowrap=true` branch. Integer overflow in a timelock deadline computation must never silently reduce the deadline — it should always clamp to the maximum value (saturating), ensuring the timelock is treated as unsatisfiable rather than trivially satisfied.

```rust
// Replace:
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {

// With (always):
if timestamp < unspent.timestamp.saturating_add(seconds_relative) {
```

---

### Proof of Concept

Using the Python test harness already present in the repository:

```python
# Coin confirmed at timestamp 10000, current timestamp 10150
# seconds_relative chosen so that 10000 + value wraps to 0
# 10000 + (2^64 - 9999 - 1) = 0 (mod 2^64)
conds = make_test_conds(seconds_relative=0xFFFF_FFFF_FFFF_D8F0)

result = check_time_locks(
    {TEST_COIN.name(): COIN_RECORD},  # coin confirmed at timestamp 10000
    conds,
    PREV_BLOCK_HEIGHT,     # 15
    PREV_BLOCK_TIMESTAMP,  # 10150
    False,                 # nowrap=False (legacy path)
)
assert result is None  # No error — timelock bypassed
``` [7](#0-6) 

The test at line 249 confirms: `(make_test_conds(seconds_relative=0xFFFF_FFFF_FFFF_D8F0), 105, None)` — `105` is the error code for `ASSERT_SECONDS_RELATIVE_FAILED` (returned with `nowrap=True`), and `None` (no error, spend accepted) is returned with `nowrap=False`. The timelock is bypassed.
