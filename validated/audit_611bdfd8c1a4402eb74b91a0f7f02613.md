### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash provided by the caller corresponds to a previously declared contract class. This allows any contract to replace its own class hash with an arbitrary, undeclared value. The OS will produce a valid STARK proof for this invalid state transition, permanently rendering the contract inaccessible and freezing any funds it holds.

---

### Finding Description

The ERC777 vulnerability class is: **a missing invariant check on a state-mutating function that allows an invalid relationship to be written into protocol state**. In ERC777, `revokeOperator` failed to check `operator != msg.sender`, allowing a holder to revoke their own irrevocable self-operator right. The StarkNet analog is: `execute_replace_class` fails to check that the new class hash is a declared class, allowing a contract to write an invalid class hash into the global state tree — a protocol invariant that the OS is supposed to enforce.

In `syscall_impls.cairo`, the `execute_replace_class` function reads the caller-supplied `class_hash` from the syscall request and directly writes it into `contract_state_changes` with no existence check: [1](#0-0) 

The code itself contains an explicit TODO acknowledging the missing check: [2](#0-1) 

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
```

After this unvalidated write, the `StateEntry` for the contract is updated in `contract_state_changes` with the arbitrary class hash: [3](#0-2) 

The `StateEntry` struct, defined in `commitment.cairo`, stores `class_hash` as a plain `felt` with no domain restriction: [4](#0-3) 

The commitment computation in `compute_contract_state_commitment` then hashes this invalid class hash into the global state root, producing a valid proof for an invalid state: [5](#0-4) 

There is no cross-check against `contract_class_changes` anywhere in `execute_replace_class` or its callers.

---

### Impact Explanation

**Critical. Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value in the proven state root, all subsequent calls to that contract will fail at class resolution — no entry points can be dispatched. Any ERC-20 tokens, ETH, or other assets held by the contract become permanently inaccessible. Because the OS produces a valid STARK proof for this state transition, it is committed on-chain and irreversible.

---

### Likelihood Explanation

Any contract can issue the `replace_class` syscall. Realistic trigger paths include:

- A buggy contract that passes an incorrect class hash (e.g., a hash computed off-chain with an error).
- A contract with a public or semi-public `replace_class` wrapper that an unprivileged caller can invoke with attacker-controlled input.
- A contract owner who intentionally or accidentally calls `replace_class` with a hash that was never declared.

The OS provides no safety net in any of these cases. The TODO comment confirms the developers are aware the check is absent.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that it exists in `contract_class_changes` (i.e., it was declared in the current block or in a prior block). Concretely, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the returned compiled class hash is non-zero:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the invariant enforced during `execute_declare_transaction`, where `assert_not_zero(compiled_class_hash)` is required before writing to `contract_class_changes`: [6](#0-5) 

---

### Proof of Concept

1. Deploy contract `C` that holds user funds (e.g., acts as a shared wallet or escrow).
2. From within `C`, issue the `replace_class` syscall with `class_hash = 0xdeadbeef` — a felt value that has never been passed to `declare`.
3. The OS executes `execute_replace_class`, skips the missing existence check, and writes `class_hash=0xdeadbeef` into `C`'s `StateEntry` in `contract_state_changes`.
4. `compute_contract_state_commitment` hashes this entry into the new global state root; a valid STARK proof is generated and submitted to L1.
5. All subsequent calls to `C` fail — no class with hash `0xdeadbeef` exists, so no entry point can be resolved.
6. All funds held by `C` are permanently frozen with no recovery path.
