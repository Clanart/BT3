### Title
`fetch_and_build_committee` reads staker weights from latest block state rather than epoch-start state, enabling cross-node committee divergence and consensus halt - (`crates/apollo_staking/src/staking_manager.rs`)

### Summary

`StakingManager::fetch_and_build_committee` builds the epoch committee by calling `CairoStakingContract::get_stakers(epoch_id)`, which executes against the **latest committed block state** via a fresh `state_reader_factory.create()` call. There is no pinning to the epoch-start block. If staker weights change mid-epoch, two nodes that build the committee for the same epoch at different times will compute different `cumulative_weights` / `total_weight` / `eligible_proposers`, and therefore select different proposers for the same height and round. The consensus manager's `handle_proposal` rejects any proposal whose `init.proposer` does not match the locally computed proposer, causing the chain to halt.

### Finding Description

**Root cause — `CairoStakingContract::call_view` always reads from the latest block:** [1](#0-0) 

`state_reader_factory.create()` produces a reader pinned to the latest committed block at the moment of the call, not to the epoch-start block. The epoch ID is passed as calldata to the on-chain contract, but the underlying state the contract executes against is the current chain tip.

**Committee build path — no epoch-start pinning:** [2](#0-1) 

`fetch_and_build_committee` calls `get_stakers_with_config(epoch, ...)` (line 298), which for `CairoStakingContract` executes `call_view(GET_STAKERS_ENTRY_POINT, ...)` against the latest state. The resulting `committee_members`, `cumulative_weights`, and `total_weight` are all derived from that snapshot. The committee is then cached per epoch.

**Cache miss is the trigger — nodes that join or restart mid-epoch re-fetch:** [3](#0-2) 

A node that restarts mid-epoch, or whose `committee_cache` has evicted epoch N (capacity is configurable via `max_cached_epochs`), will call `fetch_and_build_committee` again. At that point the latest block state may reflect staker weight changes that occurred after epoch N started, producing a different committee than nodes that built it at epoch start.

**Mock contract confirms the design gap — epoch_id is ignored for weight lookups:** [4](#0-3) 

The mock explicitly notes "epoch_id is not used in this mock" and returns whatever stakers are currently stored, regardless of the requested epoch. This is the same pattern as the external bug: the live state is used instead of the epoch-start snapshot.

**Consensus validation rejects proposals from the "wrong" proposer:** [5](#0-4) 

`get_proposer_for_height` calls `committee_provider.get_committee(height)` and then `committee.get_proposer(height, round)`. If the locally computed proposer differs from `init.proposer`, the proposal is silently dropped. If this divergence is systematic (all validators compute a different proposer than the actual proposer), no proposal is ever accepted.

**Proposer selection uses `cumulative_weights` derived from the fetched staker data:** [6](#0-5) 

The weighted random selection is entirely determined by `cumulative_weights` and `total_weight` built at committee-fetch time. A weight change of even one staker can shift the cumulative ranges and flip the selected proposer.

### Impact Explanation

If a staker increases or decreases their `staking_power` mid-epoch (a normal on-chain operation), any node that builds the committee for that epoch after the weight change will compute a different proposer for every height and round. The consensus manager will reject all proposals from the legitimate proposer (as computed by nodes that built the committee at epoch start), and the chain cannot make progress. This is a liveness-breaking consensus failure analogous to the Lighthouse `ProposalSignatureInvalid` chain halt.

The impact maps to: **High — Transaction conversion or signature/hash logic binds the wrong signer** (the proposer identity is the "signer" bound by `ProposalInit.proposer` validation in `handle_proposal`).

### Likelihood Explanation

- Staker weight changes mid-epoch are a normal, unprivileged operation in any live staking system.
- Node restarts, rolling upgrades, or new nodes joining mid-epoch are routine operational events that trigger a committee cache miss.
- The `committee_cache` capacity is configurable (`max_cached_epochs`); a small value increases the probability of cache eviction and re-fetch.
- No special privileges are required; any staker can trigger the divergence by updating their stake.

### Recommendation

Pin the state reader to the epoch-start block when building the committee. In `fetch_and_build_committee`, resolve the epoch's `start_block` from the `epoch_cache` and pass it to `state_reader_factory` so that `call_view` executes against the state as of that block, not the latest block. This ensures all nodes — regardless of when they first query the committee — compute identical staker weights and therefore identical proposers.

### Proof of Concept

1. Epoch N starts at block H_start. Node A builds the committee for epoch N at block H_start; staker X has weight W_x. The weighted selection yields proposer P_A for height H_start + 50.
2. At block H_start + 30, staker X submits an on-chain transaction increasing their `staking_power` from W_x to W_x + Δ. This is committed to the chain.
3. Node B restarts at block H_start + 40. Its `committee_cache` is empty. It calls `fetch_and_build_committee(epoch_N)`, which calls `CairoStakingContract::get_stakers(epoch_N)` against the latest state (block H_start + 40). Staker X's weight is now W_x + Δ. The new `cumulative_weights` shift the selection boundary, and the weighted random selection yields proposer P_B ≠ P_A for height H_start + 50.
4. At height H_start + 50, P_A (the legitimate proposer per Node A's committee) broadcasts a proposal with `init.proposer = P_A`.
5. Node B calls `get_proposer_for_height(H_start + 50, round=0)`, which returns P_B. Since P_B ≠ P_A, Node B drops the proposal.
6. If a quorum of nodes have restarted or joined after block H_start + 30, no proposal is ever accepted and the chain halts. [1](#0-0) [3](#0-2) [7](#0-6) [5](#0-4)

    async fn committee_at_height(
        &self,
        height: BlockNumber,
    ) -> CommitteeProviderResult<Arc<Committee>> {
        let epoch = self.epoch_at_height(height).await?;

        // Attempt to read from cache.
        {
            let cache = self.committee_cache.lock().await;
            if let Some(committee) = cache.get(epoch) {
                return Ok(committee.clone());
            }
        }

        // Otherwise, build the committee from state, and cache the result.
        info!("Committee cache miss for epoch={epoch}, fetching from state.");
        let committee = Arc::new(self.fetch_and_build_committee(epoch).await?);

        // Cache the result.
        let mut cache = self.committee_cache.lock().await;
        cache.insert(epoch, committee.clone());

        Ok(committee)
    }

    async fn fetch_and_build_committee(&self, epoch: u64) -> CommitteeProviderResult<Committee> {
        // Update dynamic config to ensure we have the latest stakers config.
        self.update_dynamic_config().await;

        // Get the config to inject and use for committee building.
        // Clone it to avoid holding the lock across await.
        let dynamic_config = self.dynamic_config.read().expect("RwLock poisoned").clone();

        // Always use get_stakers_with_config - works for all implementations.
        let contract_stakers =
            self.staking_contract.get_stakers_with_config(epoch, &dynamic_config).await?;

        // Get the active committee config for this epoch (includes size and stakers).
        let active_config = get_config_for_epoch(
            &dynamic_config.default_committee,
            &dynamic_config.override_committee,
            epoch,
        );

        let committee_members =
            self.select_committee(contract_stakers, active_config.committee_size);

        // Prepare the data needed for proposer selection.
        let cumulative_weights: Vec<u128> = committee_members
            .iter()
            .scan(0, |acc, staker| {
                *acc = u128::checked_add(*acc, staker.weight.0).expect("Total weight overflow.");
                Some(*acc)
            })
            .collect();
        let total_weight = *cumulative_weights.last().unwrap_or(&0);

        let eligible_proposers =
            self.calculate_eligible_proposers(&committee_members, &active_config.stakers);

        // Calculate the randomness block hash for this epoch.
        let randomness_block_hash = self.proposer_randomness_block_hash(epoch).await?;

        info!(
            "Fetched committee for epoch={epoch}: size={}, total_weight={total_weight}, \
             eligible_proposers={}.",
            committee_members.len(),
            eligible_proposers.len(),
        );

        // Update committee metrics.
        STAKING_COMMITTEE_TOTAL_WEIGHT.set_lossy(total_weight);

        // Calculate eligible proposers total weight.
        let eligible_proposers_total_weight: u128 = committee_members
            .iter()
            .filter(|staker| eligible_proposers.contains(&staker.address))
            .map(|staker| staker.weight.0)
            .sum();

        STAKING_COMMITTEE_ELIGIBLE_PROPOSERS_TOTAL_WEIGHT
            .set_lossy(eligible_proposers_total_weight);

        // Update per-staker metrics.
        self.committee_member_metrics.lock().await.update_committee_members(&committee_members);

        Ok(Committee {
            committee_members: Arc::new(committee_members),
            cumulative_weights,
            total_weight,
            eligible_proposers,
            random_generator: self.random_generator_factory.create_seeded(randomness_block_hash),
            proposer_cache: std::sync::Mutex::new(None),
            use_only_actual_proposer_selection: self.use_only_actual_proposer_selection,
        })
    }
