### Title
Permanent Freeze of Bridged Funds When EVM Recipient Is Blacklisted by Token Contract - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

### Summary
In `OmniBridge.finTransfer`, when the bridged token is a blacklist-enforcing ERC20 (e.g., USDC, USDT), and the MPC-signed payload's `recipient` address is blacklisted, the mandatory `safeTransfer` call reverts. Because the MPC signature cryptographically binds the recipient address and no cancel/refund path exists on the source chain for already-signed transfers, the user's tokens become permanently frozen.

### Finding Description
`OmniBridge.finTransfer` finalizes a cross-chain transfer by first marking the nonce as used, verifying the MPC signature, then dispatching tokens to the recipient. For native ERC20 tokens (the `else` branch), it calls:

```solidity
IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
``` [1](#0-0) 

If `payload.tokenAddress` is USDC and `payload.recipient` is on the USDC blacklist, this call reverts. Because Solidity reverts are atomic, the `completedTransfers[payload.destinationNonce] = true` write at line 287 is also rolled back: [2](#0-1) 

The nonce is therefore never consumed, but every subsequent retry of `finTransfer` with the same MPC-signed payload will also revert — the signature is bound to the blacklisted `payload.recipient` and cannot be redirected.

On the source chain (NEAR), the user's tokens were locked in the bridge contract when `ft_on_transfer` was called with an `InitTransfer` message. After `sign_transfer` is called with zero fee, the transfer is removed from `pending_transfers`: [3](#0-2) 

There is no `cancel_transfer` or refund function in the NEAR bridge contract. The tokens remain locked with no recovery path.

The same freeze applies to the ETH-native case: if `payload.recipient` is a contract that rejects ETH, `FailedToSendEther` is thrown and the same permanent-freeze outcome results. [4](#0-3) 

### Impact Explanation
**Critical — Permanent, irrecoverable freeze of user funds in the bridge flow.**

Tokens are burned/locked on the source chain and can never be delivered on the destination chain. The user loses the full bridged amount with no protocol-level recovery mechanism.

### Likelihood Explanation
**Medium.** USDC is one of the most commonly bridged stablecoins. USDC blacklisting is a real, documented occurrence (sanctions enforcement, exchange hacks, regulatory orders). A user can be blacklisted *after* initiating a bridge transfer — the window between `ft_on_transfer` on NEAR and `finTransfer` on EVM can span minutes to hours. No attacker action is required; the blacklisting event alone triggers the freeze.

### Recommendation
1. **Source-chain cancel mechanism**: Add a `cancel_transfer` function on the NEAR bridge that allows the original sender to reclaim locked tokens if a transfer has been pending beyond a timeout or has never been finalized.
2. **EVM pull-over-push**: Instead of reverting on transfer failure, catch the failure and store the amount in a `claimable[recipient][token]` mapping. Provide a separate `claim()` function so the user (or an admin on their behalf) can redirect funds to an unblacklisted address.
3. **Pre-flight check**: Optionally, check token-level blacklist status before signing the transfer payload on NEAR, though this is not always queryable cross-chain.

### Proof of Concept
1. User holds USDC on NEAR and calls `ft_transfer_call` to the NEAR bridge with an `InitTransfer` message specifying an EVM recipient address `R` and zero fee. Tokens are locked in the NEAR bridge; the transfer is stored in `pending_transfers`.
2. Relayer calls `sign_transfer` on NEAR. MPC signs a `TransferMessagePayload` binding `recipient = R`. With zero fee, the transfer is removed from `pending_transfers` — no cancel path remains. [5](#0-4) 
3. Between step 1 and step 3, USDC blacklists address `R` (e.g., due to a sanctions designation).
4. Relayer calls `OmniBridge.finTransfer` on EVM with the signed payload. Execution reaches:
   ```solidity
   IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
   ```
   USDC's `transfer` reverts because `R` is blacklisted. The entire transaction reverts; `completedTransfers[nonce]` stays `false`. [1](#0-0) 
5. Every subsequent `finTransfer` call with the same MPC-signed payload reverts identically. No alternative payload can be signed (the MPC key signed `recipient = R`).
6. The NEAR bridge holds the user's USDC with no cancel function. Funds are permanently frozen.

    pub fn sign_transfer(
        &mut self,
        transfer_id: TransferId,
        fee_recipient: Option<AccountId>,
        fee: &Option<Fee>,
    ) -> Promise {
        let transfer_message = self.get_transfer_message(transfer_id);

        if let Some(fee) = &fee {
            require!(
                &transfer_message.fee == fee,
                BridgeError::InvalidFee.as_ref()
            );
        }

        let token_address = self
            .get_token_address(
                transfer_message.get_destination_chain(),
                self.get_token_id(&transfer_message.token),
            )
            .unwrap_or_else(|| {
                env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
            });

        let decimals = self
            .token_decimals
            .get(&token_address)
            .near_expect(BridgeError::TokenDecimalsNotFound);
        let amount_to_transfer = Self::normalize_amount(
            transfer_message
                .amount_without_fee()
                .near_expect(BridgeError::InvalidFee),
            decimals,
        );

        require!(
            amount_to_transfer > 0,
            BridgeError::InvalidAmountToTransfer.as_ref()
        );

        let message = DestinationChainMsg::from_json(&transfer_message.msg)
            .and_then(|s| s.destination_msg())
            .unwrap_or_default();

        let transfer_payload = TransferMessagePayload {
            prefix: PayloadType::TransferMessage,
            destination_nonce: transfer_message.destination_nonce,
            transfer_id,
            token_address,
            amount: U128(amount_to_transfer),
            recipient: transfer_message.recipient,
            fee_recipient,
            message,
        };

        let payload = near_sdk::env::keccak256_array(
            transfer_payload
                .encode_hashable()
                .near_expect(BridgeError::Borsh),
        );

        ext_signer::ext(self.mpc_signer.clone())
            .with_static_gas(MPC_SIGNING_GAS)
            .with_attached_deposit(env::attached_deposit())
            .sign(SignRequest {
                payload,
                path: SIGN_PATH.to_owned(),
                key_version: 0,
            })
            .then(
                Self::ext(env::current_account_id())
                    .with_static_gas(SIGN_TRANSFER_CALLBACK_GAS)
                    .sign_transfer_callback(transfer_payload, &transfer_message.fee),
            )
    }
