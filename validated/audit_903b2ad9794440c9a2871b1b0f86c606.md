### Title
`ProofOfInclusion::valid()` Missing External Root Hash Check Allows Forged Inclusion Proofs — (`File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

---

### Summary

`ProofOfInclusion::valid()` performs only a self-referential internal consistency check. Its final assertion is tautologically true after the loop, meaning any attacker-crafted `ProofOfInclusion` with internally consistent hashes will pass `valid()` regardless of whether it corresponds to any real tree state. This is the direct analog of the GMX missing-uniqueness-check bug: just as GMX's `_setPrices()` was missing a check that prevented the same price index from being reused (allowing tampered prices to pass signature verification), chia_rs's `valid()` is missing a check against an external, known root hash (allowing a fabricated proof chain to pass inclusion verification).

---

### Finding Description

In `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`, the `valid()` method is:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()   // ← always true after the loop
}
```

And `root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← same value as `existing_hash` after the loop
    } else {
        self.node_hash
    }
}
```

After the loop body, `existing_hash` holds the last `calculated_hash`, which the loop already asserted equals `layer.combined_hash`. `root_hash()` returns that same `last.combined_hash`. Therefore the final guard `existing_hash == self.root_hash()` is **unconditionally true** whenever the loop completes without returning `false`.

The function accepts no external root hash parameter. It cannot be used to verify a proof against a known, trusted root. An attacker who can supply a `ProofOfInclusion` (the struct is `Streamable` and fully deserializable from bytes, and is exposed to Python via `py_valid()`) can construct any self-consistent chain of layers — with any chosen `node_hash` — and `valid()` will return `true`.

**Concrete forgery recipe:**
1. Choose any target `node_hash` H₀ (claiming any key is "included").
2. Choose any `other_hash` H₁ and `other_hash_side`.
3. Compute `combined_hash = calculate_internal_hash(H₀, side, H₁)`.
4. Wrap in `ProofOfInclusion { node_hash: H₀, layers: [ProofOfInclusionLayer { other_hash_side: side, other_hash: H₁, combined_hash }] }`.
5. `valid()` returns `true`. The attacker-controlled `root_hash()` is `combined_hash`.

---

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."**

Any consumer that receives a `ProofOfInclusion` from an untrusted peer and calls `.valid()` to decide whether a key-value pair is committed in a DataLayer store will accept the forged proof. Because `ProofOfInclusion` is `Streamable` and exposed to Python via `py_valid()`, it is a natural deserialization target for network-received data. A malicious peer can prove inclusion of any key-value pair in any tree, enabling false state attestation across the DataLayer protocol.

---

### Likelihood Explanation

The `ProofOfInclusion` struct is `Streamable` (deserializable from raw bytes), exposed to Python via `py_valid()`, and the `valid()` method takes no external root parameter — making it the natural and only API for callers to verify a received proof. Any Python code that calls `proof.valid()` without separately comparing `proof.root_hash()` to a locally-known root is vulnerable. The misleading name `valid()` makes this mistake highly likely.

---

### Recommendation

Add a required `expected_root: &Hash` parameter to `valid()` (and `py_valid()`), and replace the final tautological check with a comparison against the caller-supplied root:

```rust
pub fn valid(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    &existing_hash == expected_root   // ← compare against trusted external root
}
```

This forces every call site to supply the root hash it trusts, eliminating the self-referential tautology.

---

### Proof of Concept

The tautology is directly visible in the source. After the loop, `existing_hash` equals `last.combined_hash` (enforced by the loop's `if calculated_hash != layer.combined_hash { return false; }` guard). `root_hash()` returns `last.combined_hash`. Therefore `existing_hash == self.root_hash()` reduces to `last.combined_hash == last.combined_hash`, which is always `true`.

A fabricated single-layer proof:

```rust
let forged = ProofOfInclusion {
    node_hash: Hash(arbitrary_leaf_hash),
    layers: vec![ProofOfInclusionLayer {
        other_hash_side: Side::Right,
        other_hash: Hash(arbitrary_sibling),
        combined_hash: calculate_internal_hash(
            &Hash(arbitrary_leaf_hash),
            Side::Right,
            &Hash(arbitrary_sibling),
        ),
    }],
};
assert!(forged.valid()); // passes — no real tree needed
```
