### Title
3-arg `mint` in `HyperliquedBridgeToken` routes all minted tokens to `_systemAddress` instead of the intended recipient when `message.length > 0` — (`evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

---

### Summary

`HyperliquedBridgeToken.mint(address, uint256, bytes)` first mints tokens to `account` and then immediately transfers the full amount to `_systemAddress` via `_update`. `OmniBridge.finTransfer` selects this 3-arg path whenever `payload.message.length > 0`. Any unprivileged user who initiates a cross-chain transfer with a non-empty `message` field targeting a HyperEVM recipient will have their tokens silently redirected to `_systemAddress`, leaving the intended recipient with zero balance.

---

### Finding Description

`HyperliquedBridgeToken.mint` (3-arg) is:

```solidity
function mint(address account, uint256 value, bytes memory) external override onlyOwner {
    _mint(account, value);
    _update(account, _systemAddress, value);
}
``` [1](#0-0) 

`_mint(account, value)` credits `value` to `account`. The immediately following `_update(account, _systemAddress, value)` is a standard ERC-20 transfer that moves the entire balance from `account` to `_systemAddress`. Net effect: `account.balance == 0`, `_systemAddress.balance += value`.

`OmniBridge.finTransfer` selects the 3-arg path based solely on `payload.message.length > 0`:

```solidity
} else if (isBridgeToken[payload.tokenAddress]) {
    if (payload.message.length == 0) {
        IBridgeToken(payload.tokenAddress).mint(payload.recipient, payload.amount);
    } else {
        IBridgeToken(payload.tokenAddress).mint(payload.recipient, payload.amount, payload.message);
    }
}
``` [2](#0-1) 

The `message` field is a legitimate, user-controlled input in `initTransfer`: [3](#0-2) 

The MPC signs whatever was legitimately initiated on the source chain, including the `message` bytes. It has no knowledge that a non-empty `message` will cause `finTransfer` to invoke the HyperCore-specific 3-arg mint path on the destination.

---

### Impact Explanation

Any cross-chain transfer to a `HyperliquedBridgeToken` address with `message.length > 0` results in:
- `payload.recipient.balanceOf == 0` after `finTransfer`
- `_systemAddress.balanceOf += payload.amount`

The intended recipient receives nothing. The tokens are parked at `_systemAddress`, which is the Hyperliquid system precompile address — not a user-controlled address. There is no direct mechanism for the intended recipient to reclaim tokens from `_systemAddress` without going through the HyperCore system, which was not the intended destination. This constitutes **direct theft / misdirection of bridged assets** from the intended recipient.

---

### Likelihood Explanation

The `message` field is a first-class, documented feature of the bridge (used for cross-chain messaging). Any user who sends a transfer with a non-empty `message` to a HyperEVM recipient holding a `HyperliquedBridgeToken` triggers this. An attacker can also deliberately target a victim's pending transfer by front-running or by initiating a transfer on behalf of a victim with a crafted non-empty message. No privileged access is required — only a public `initTransfer` call on any supported source chain.

---

### Recommendation

The branching logic in `OmniBridge.finTransfer` must not use `message.length > 0` as the signal to invoke the HyperCore-specific 3-arg mint. Options:

1. **Separate the HyperCore path entirely**: Introduce a dedicated flag or a separate token interface for HyperCore-bound mints, rather than overloading the `message` field.
2. **Fix `HyperliquedBridgeToken.mint` (3-arg)**: If the 3-arg path is only valid for HyperCore-originated calls (i.e., via `coreReceiveWithData`), remove the 3-arg `mint` override entirely and handle HyperCore routing through a separate, access-controlled entry point.
3. **Guard in `finTransfer`**: Only invoke the 3-arg mint path when the token is explicitly registered as a HyperCore token, not based on `message.length`.

---

### Proof of Concept

1. Deploy `HyperliquedBridgeToken` owned by `OmniBridge`, registered as `isBridgeToken`.
2. Attacker calls `initTransfer` on any source chain with `message = abi.encode("x")` (non-empty), `recipient = victim_evm_address`, `tokenAddress = HyperliquedBridgeToken`.
3. MPC observes the event and signs the `TransferMessagePayload` (including the non-empty `message` bytes) — this is normal MPC behavior.
4. Relayer calls `OmniBridge.finTransfer` with the signed payload.
5. `finTransfer` enters the `isBridgeToken` branch, sees `payload.message.length > 0`, calls `HyperliquedBridgeToken.mint(victim, amount, message)`.
6. Inside `mint`: `_mint(victim, amount)` → victim balance = `amount`; `_update(victim, _systemAddress, amount)` → victim balance = 0, `_systemAddress` balance = `amount`.
7. Assert: `victim.balanceOf == 0`, `_systemAddress.balanceOf == amount`. ✓
