### Title
Missing Shard-ID Binding Check in `set_state_header` Allows Cross-Shard State Corruption During State Sync — (File: `chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` in `ChainStateSyncAdapter` accepts a `ShardStateSyncResponseHeader` from a peer and stores it under `StateHeaderKey(shard_id, sync_hash)` without ever verifying that the embedded chunk's `shard_id` matches the caller-supplied `shard_id` parameter. A malicious peer can serve a fully-valid header for shard X when the syncing node requests shard Y's header. If shard X carries no incoming receipts for the relevant block range, every validation gate passes, the wrong header is committed to `DBCol::StateHeaders` under shard Y's key, and shard X's trie data is subsequently applied into shard Y's flat storage and trie, permanently corrupting the syncing node's state for shard Y.

---

### Finding Description

`set_state_header` performs five checks before writing to the DB:

1. **`validate_chunk_proofs`** — validates the chunk's internal self-consistency (signature, encoded Merkle root). It does not compare `chunk.shard_id()` against the `shard_id` parameter.

2. **`verify_path` against `chunk_headers_root`** — proves the chunk hash is *somewhere* in the block's chunk Merkle tree. The proof is position-specific (it encodes left/right siblings), but the call site never asserts that the proven position equals the shard index of the requested `shard_id`. A valid proof for shard X at index 0 passes this check even when `shard_id = Y` (index 1).

3. **Receipt proof loop (step 4e)** — computes `CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))` using the *parameter* `shard_id`. If the injected header for shard X carries non-empty incoming receipts, the hash mismatch causes rejection. **However, if shard X has zero incoming receipts for the covered block range, the loop body never executes and the check is entirely bypassed.**

4. **Step 4g** — checks receipt height continuity; irrelevant when the receipt list is empty.

5. **`validate_state_root_node`** — validates the state root node against `chunk_inner.prev_state_root()`, which is shard X's root. This passes trivially.

After all checks pass, the header is written:

```rust
// chain/chain/src/state_sync/adapter.rs  lines 526-529
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
store_update.commit();
