### Title
Hardcoded `failure_flag=0` in `execute_deploy` Syscall Silently Swallows Constructor Failures — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall handler in the StarkNet OS unconditionally writes `failure_flag=0` into the syscall response header, regardless of whether the deployed contract's constructor actually succeeded or reverted. This is the direct analog of the Compound Finance error-propagation bug: a failure code is never surfaced to the caller, so any contract that issues a `deploy` syscall and then acts on the assumption of success (e.g., immediately transferring funds to the newly deployed address) will proceed even when the constructor silently failed.

---

### Finding Description

In `execute_deploy` (the OS handler for the `DEPLOY_SELECTOR` syscall), after calling `deploy_contract`, the response header is written with a hardcoded `failure_flag=0`:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The developer-authored TODO comment explicitly acknowledges that failure propagation is not yet implemented. Compare this with the correct pattern used in `contract_call_helper` (the handler for `call_contract` and `library_call`), which properly reads `is_reverted` from `select_execute_entry_point_func` and writes it into the response:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

The `execute_entry_point` function is capable of returning `is_reverted=1` when a constructor fails (out-of-gas, explicit revert, or entry-point-not-found): [3](#0-2) [4](#0-3) 

That `is_reverted` signal is never threaded back through `deploy_contract` into the `execute_deploy` response. The `deploy_contract` call site captures only `(retdata_size, retdata)`: [5](#0-4) 

The inconsistency is also present in the deprecated syscall path (`execute_deploy_syscall` in `deprecated_execute_syscalls.cairo`), where `deploy_contract` is called and its return values are entirely discarded: [6](#0-5) 

---

### Impact Explanation

**Direct loss of funds (Critical).**

A contract that issues a `deploy` syscall and then immediately interacts with the deployed address — for example, by transferring ERC-20 tokens or native ETH to it — will always observe `failure_flag=0` in the response. If the constructor reverted (e.g., due to an attacker-controlled argument that causes an assertion failure, or due to gas exhaustion), the calling contract has no way to detect the failure. It will proceed to send funds to an address whose contract is in an uninitialized or partially-initialized state. Those funds may be permanently unrecoverable, constituting a direct loss of funds for the caller.

The revert-log mechanism (`handle_revert`) would undo the constructor's storage writes, leaving the deployed contract's storage in its zero-initialized default state. A malicious or buggy constructor can exploit this: the contract exists at the address (its class hash is registered), but its invariants are not established. Any funds sent to it after the false-success response are at risk.

---

### Likelihood Explanation

The `deploy` syscall is available to any unprivileged contract on the network. Any user can submit an invoke transaction that calls a contract which in turn issues a `deploy` syscall with a constructor designed to fail. DeFi factory patterns (deploy-then-fund) are extremely common in practice. The attacker does not need any privileged role; they only need to be able to submit a transaction.

---

### Recommendation

Replace the hardcoded `failure_flag=0` with the actual revert status returned by `deploy_contract`. Refactor `deploy_contract` to return `(is_reverted, retdata_size, retdata)` — mirroring the signature of `select_execute_entry_point_func` — and propagate `is_reverted` into the `ResponseHeader`, exactly as `contract_call_helper` does for `call_contract`. The existing TODO comment confirms this is a known gap.

---

### Proof of Concept

1. Deploy a factory contract `F` on StarkNet. `F.__execute__` does:
   - Issues `deploy(class_hash=BOMB, calldata=[attacker_arg])` where `BOMB.__constructor__` asserts `calldata[0] == 0` (fails when `attacker_arg != 0`).
   - Reads `response.failure_flag` — always 0 due to the bug.
   - Calls `transfer(deployed_address, amount)` on the fee token, sending funds to the deployed address.
2. The constructor of `BOMB` reverts. The revert log undoes its storage writes. The contract exists at the address but is uninitialized.
3. `F` receives `failure_flag=0`, proceeds with the transfer.
4. Funds arrive at an address whose contract state violates its own invariants. Depending on `BOMB`'s logic, the funds may be permanently locked (frozen) or extractable only by the attacker.
