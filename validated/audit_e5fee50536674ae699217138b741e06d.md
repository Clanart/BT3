### Title
Wrong epoch's shard layout used to index `prev_block.chunks()` in `get_state_response_part` silently returns wrong state root for child shards after resharding — (File: chain/chain/src/state_sync/adapter.rs)

### Summary

`get_state_response_part` computes the `shard_index` using the **new epoch's** shard layout, then uses that index to access `prev_block.chunks()`, which is indexed by the **previous epoch's** shard layout. After a resharding event, child shard IDs occupy different positions in the new layout than their parent shards occupied in the old layout. This causes the function to either silently return the wrong state root (from a different shard's chunk) or return `Error::InvalidShardId`, permanently breaking state-part serving for every child shard created by a split.

### Finding Description

In `get_state_response_part` the serving node resolves the state root it will use to generate a state part as follows:

```rust
let epoch_id = block.header().epoch_id();          // sync block = first block of NEW epoch
let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;  // NEW epoch layout
...
let prev_block = self.chain_store.get_block(header.prev_hash())?;   // last block of OLD epoch
let shard_index = shard_layout.get_shard_index(shard_id)?;          // index in NEW layout ← wrong
let state_root = prev_block
    .chunks()
    .get(shard_index)                                                // indexed by OLD layout ← mismatch
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
```

`prev_block.chunks()` is a flat array whose positions correspond to the **previous epoch's** shard layout. After a shard split (e.g., old layout `[shard 0, shard 1, shard 2]` → new layout `[shard 0, shard 2, shard 3, shard 4]` where shard 1 split into shards 3 and 4):

| Child shard | Index in new layout | `prev_block.chunks().get(index)` | Result |
|---|---|---|---|
| shard 3 | 2 | chunk for old shard 2 | **wrong state root** (shard 2's root, not shard 1's) |
| shard 4 | 3 | `None` (prev_block has only 3 chunks) | `Error::InvalidShardId` |

The `shard_id` validation at line 307 passes for child shards because it checks against the **new** epoch's shard IDs. The function then proceeds to the broken index computation.

The correct fix is to resolve the state root using the **previous epoch's** shard layout and, for child shards, look up the parent shard's index in that layout. [1](#0-0) [2](#0-1) 

### Impact Explanation

Any node that needs to track a newly-created child shard after a resharding event must state-sync that shard. The serving node's `get_state_response_part` will either:

1. Return state parts built from the **wrong state root** (a sibling shard's root), which the syncing node's `validate_state_part` will reject because the Merkle proof won't match.
2. Return `Error::InvalidShardId` outright for child shards whose new-layout index exceeds the old chunk count.

In both cases the syncing node cannot complete state sync for the child shard. With dynamic resharding enabled (`ProtocolFeature::DynamicResharding`), splits happen automatically at epoch boundaries, so every split permanently disables state-part serving for the resulting child shards on all serving nodes. [3](#0-2) [4](#0-3) 

### Likelihood Explanation

The trigger is a standard `StateRequestPart` network message (unprivileged, sent by any syncing node) for a child shard after any resharding epoch boundary. With dynamic resharding active the condition is met automatically; with static resharding it is met at every protocol-version upgrade that changes the shard layout. No special attacker capability is required — the bug fires for every legitimate state-sync attempt on a child shard. [5](#0-4) 

### Recommendation

Replace the single `shard_layout` lookup with a two-layout lookup that mirrors what `compute_state_response_header` does for the header path:

```rust
let prev_epoch_id = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
// For child shards after resharding, resolve to the parent shard's index.
let lookup_shard_id = if prev_shard_layout.shard_ids().any(|id| id == shard_id) {
    shard_id
} else {
    shard_layout.get_parent_shard_id(shard_id)?
};
let shard_index = prev_shard_layout.get_shard_index(lookup_shard_id)?;
```

This mirrors the pattern already used in `get_prev_shard_infos` and `get_prev_shard_id_from_prev_hash`. [6](#0-5) 

### Proof of Concept

1. Configure a network with dynamic resharding enabled (`ShardLayoutConfig::Dynamic`) and `force_split_shards` pointing at shard 0.
2. Run until the split takes effect (epoch N+2 has child shards 3 and 4).
3. From any node, send a `StateRequestPart { shard_id: 4, sync_hash: <first block of epoch N+2>, part_id: 0 }` to a serving node.
4. The serving node enters `get_state_response_part`, computes `shard_index = new_layout.get_shard_index(4)` = 3, calls `prev_block.chunks().get(3)` on a block that has only 3 chunks → `None` → returns `Error::InvalidShardId(4)`.
5. For shard 3 (index 2 in new layout), `prev_block.chunks().get(2)` returns the chunk for old shard 2; the state root from that chunk is used to generate parts, which fail Merkle validation on the syncing side.
6. State sync for both child shards is permanently broken on all serving nodes. [7](#0-6) [8](#0-7)

    pub fn get_state_response_part(
        &mut self,
        shard_id: ShardId,
        part_id: u64,
        sync_hash: CryptoHash,
    ) -> Result<StatePart, Error> {
        let _span = tracing::debug_span!(
            target: "sync",
            "get_state_response_part",
            %shard_id,
            part_id,
            ?sync_hash)
        .entered();
        let block = self
            .chain_store
            .get_block(&sync_hash)
            .log_storage_error("block has already been checked for existence")?;
        let header = block.header();
        let epoch_id = block.header().epoch_id();
        // Check cache
        let key = borsh::to_vec(&StatePartKey(sync_hash, shard_id, part_id)).unwrap();
        if let Some(bytes) = self.chain_store.store_ref().get(DBCol::StateParts, &key) {
            metrics::STATE_PART_CACHE_HIT.inc();
            let state_part = StatePart::from_bytes(bytes.to_vec())?;
            return Ok(state_part);
        }
        metrics::STATE_PART_CACHE_MISS.inc();

        let shard_layout = self.epoch_manager.get_shard_layout(epoch_id)?;
        let shard_ids = self.epoch_manager.shard_ids(epoch_id)?;
        if !shard_ids.contains(&shard_id) {
            return Err(shard_id_out_of_bounds(shard_id));
        }
        let prev_block = self.chain_store.get_block(header.prev_hash())?;
        let shard_index = shard_layout.get_shard_index(shard_id)?;
        let state_root = prev_block
            .chunks()
            .get(shard_index)
            .ok_or(Error::InvalidShardId(shard_id))?
            .prev_state_root();
        let prev_hash = *prev_block.hash();
        let prev_prev_hash = *prev_block.header().prev_hash();
        let state_root_node = self
            .runtime_adapter
            .get_state_root_node(shard_id, &prev_hash, &state_root)
            .log_storage_error("get_state_root_node fail")?;
        let num_parts = get_num_state_parts(state_root_node.memory_usage);
        if part_id >= num_parts {
            return Err(shard_id_out_of_bounds(shard_id));
        }
        let current_time = Instant::now();
        let state_part = self
            .runtime_adapter
            .obtain_state_part(
                shard_id,
                &prev_prev_hash,
                &state_root,
                PartId::new(part_id, num_parts),
            )
            .log_storage_error("obtain_state_part fail")?;

        let elapsed_ms = (self.clock.now().signed_duration_since(current_time))
            .whole_milliseconds()
            .max(0) as u128;
        self.requested_state_parts
            .save_state_part_elapsed(&sync_hash, &shard_id, &part_id, elapsed_ms);

        // Cache the part data, but only if the corresponding header is also cached.
        // At epoch boundaries, clear_all_downloaded_parts() deletes all cached headers
        // and parts. Since serving runs on a separate actor, a part request can arrive
        // after the clear and re-create a StatePartKey without its StateHeaderKey,
        // which the storage validator treats as an inconsistency.
        let header_key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
        if self.chain_store.store_ref().exists(DBCol::StateHeaders, &header_key) {
            let mut store_update = self.chain_store.store().store_update();
            let bytes = state_part.to_bytes();
            store_update.set(DBCol::StateParts, &key, &bytes);
            store_update.commit();
        }

        Ok(state_part)
    }

        Some(response)
    }

    /// Returns the shard index for a given shard id. The shard index should be
    /// used when indexing into an array of chunk data.
    pub fn get_shard_index(&self, shard_id: ShardId) -> Result<ShardIndex, ShardLayoutError> {
        match self {
            // In V0 & V1 the shard id and shard index are the same.
            Self::V0(_) | Self::V1(_) => Ok(shard_id.into()),
            // In V2 & V3 the shard id and shard index are **not** the same.
            Self::V2(v2) => v2.get_shard_index(shard_id),
            Self::V3(v3) => v3.get_shard_index(shard_id),
        }
    }
