### Title
Finalization Record Removed After Failed `ft_transfer_call`, Enabling Proof Replay and Double-Spend - (File: `near/omni-bridge/src/lib.rs`)

### Summary

When a `fin_transfer` to a NEAR recipient uses a non-empty `msg` (triggering `ft_transfer_call`), and the recipient contract returns `0` from `ft_on_transfer`, the bridge removes the transfer from `finalised_transfers` as part of its rollback. Because the prover does not track consumed proofs, the same source-chain proof can be re-submitted and accepted a second time, causing the bridge to release or mint the same tokens twice.

### Finding Description

The NEAR bridge contract uses `finalised_transfers` as its sole replay-prevention record for incoming cross-chain transfers. When a transfer targets a NEAR recipient with a non-empty `msg`, `process_fin_transfer_to_near` calls `add_fin_transfer` to mark the transfer as finalized, then dispatches tokens via `ft_transfer_call`. The result is handled in `fin_transfer_send_tokens_callback`. [1](#0-0) 

If `ft_on_transfer` on the recipient contract returns `U128(0)` (the full amount is "unused"), `is_refund_required` returns `true`: [2](#0-1) 

The callback then reverts all state changes, including calling `remove_fin_transfer`, which **deletes the transfer from `finalised_transfers`**: [3](#0-2) [4](#0-3) 

After this rollback, `is_transfer_finalised` returns `false` for that transfer ID: [5](#0-4) 

A second call to `fin_transfer` with the identical proof now passes the `add_fin_transfer` guard (line 2228–2231) and processes the transfer a second time. The prover contract verifies only cryptographic validity of the source-chain event; it does not maintain a consumed-proof registry. Replay prevention is entirely the bridge's responsibility via `finalised_transfers`. [6](#0-5) 

This is the direct analog of the external report: the external report's `isWearerOfHat` checked a **currently-active** state (which could be false due to toggle/eligibility) instead of the **static raw balance**. Here, `finalised_transfers` is checked as the replay guard, but it is a **mutable removable set** rather than a permanent record — it can be cleared by the rollback path, allowing the same proof to be accepted again.

### Impact Explanation

**For deployed (bridged) tokens**: On the second finalization, the bridge mints `amount` tokens again. The first minted tokens were returned by `ft_transfer_call` and burned by `burn_tokens_if_needed`. The second minting is unbacked by any source-chain event, inflating the bridged token supply and breaking collateralization.

**For native (locked) tokens**: On the first failed finalization, `revert_lock_actions` restores `locked_tokens`. On the second finalization, `unlock_tokens_if_needed` decrements `locked_tokens` again and the bridge transfers the same tokens to the recipient. The bridge releases the same locked collateral twice for a single source-chain lock event, making it undercollateralized.

Both cases match the allowed impact: **direct theft / unauthorized mint of bridged assets** and **balance/accounting corruption that breaks bridge collateralization**.

### Likelihood Explanation

The attacker must:
1. Initiate a transfer on the source chain (e.g., ETH) with a NEAR recipient address they control and a non-empty `msg` field — both are attacker-controlled parameters.
2. Deploy a malicious NEAR contract as the recipient that returns `U128(0)` from `ft_on_transfer` on the first call and accepts on the second.
3. Have a trusted relayer re-submit the proof after the first attempt fails.

Step 3 is realistic: after the rollback, the transfer is absent from `finalised_transfers` and the `FailedFinTransferEvent` is emitted. A relayer monitoring for unfinalized transfers will see the transfer as pending and re-submit the proof in good faith. The attacker does not need to be a trusted relayer themselves; they only need to control the recipient contract.

### Recommendation

Replace the mutable `finalised_transfers` set with a **permanent, non-removable** record for replay prevention. The finalization record should never be deleted, even on rollback. Instead of removing the entry from `finalised_transfers` in `fin_transfer_send_tokens_callback`, use a separate `failed_fin_transfers` set (or a status enum) to track rollback state. The replay guard in `add_fin_transfer` must check the permanent record regardless of whether the transfer subsequently failed.

Alternatively, introduce a separate `pending_fin_transfers` set that is inserted before token dispatch and only promoted to `finalised_transfers` on success — but the key invariant is that once a proof has been accepted by the prover and processed by the bridge, its transfer ID must be permanently recorded and never removed.

### Proof of Concept

1. Attacker deploys `MaliciousReceiver` on NEAR: `ft_on_transfer` returns `"0"` on the first call, `"<amount>"` (accept) on the second.
2. Attacker locks 1000 USDC on ETH, setting `recipient = MaliciousReceiver`, `msg = "trigger"`.
3. Trusted relayer calls `fin_transfer` with the ETH proof. Bridge calls `add_fin_transfer` (marks finalized), unlocks 1000 USDC, calls `ft_transfer_call("MaliciousReceiver", 1000, "trigger")`.
4. `MaliciousReceiver.ft_on_transfer` returns `"0"`. Tokens are returned to bridge. `fin_transfer_send_tokens_callback` calls `revert_lock_actions` (re-locks 1000 USDC) and `remove_fin_transfer` (removes the finalization record).
5. Relayer (or attacker if also a trusted relayer) calls `fin_transfer` again with the same ETH proof. `add_fin_transfer` succeeds (entry was removed). Bridge unlocks 1000 USDC again, calls `ft_transfer_call("MaliciousReceiver", 1000, "trigger")`.
6. `MaliciousReceiver.ft_on_transfer` returns `"1000"` (accepts). Attacker receives 1000 USDC on NEAR.
7. Attacker bridges 2000 USDC back to ETH (1000 from step 6 + 1000 from a prior legitimate balance), draining the bridge of 1000 USDC beyond what was locked. [7](#0-6) [4](#0-3)

    fn remove_fin_transfer(&mut self, transfer_id: &TransferId, storage_owner: &AccountId) {
        let storage_usage = env::storage_usage();
        self.finalised_transfers.remove(transfer_id);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(storage_owner, &storage);
        }
    }
