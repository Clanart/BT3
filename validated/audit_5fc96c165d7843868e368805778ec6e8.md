### Title
Unchecked Multiplication Overflow in `denormalize_amount` Corrupts Bridge Token Amounts - (File: near/omni-bridge/src/lib.rs)

### Summary
The `denormalize_amount` helper in the NEAR bridge contract multiplies a `u128` amount by `10_u128.pow(diff_decimals)` — both operands are `u128` — with no overflow guard. This is the direct analog of the reported premia-multiplication overflow: two 128-bit values multiplied in an intermediate step that can silently wrap (or panic) before the result is used to settle a transfer or fee.

### Finding Description
`denormalize_amount` is called whenever the bridge converts a normalized (bridge-side) token amount back to the origin-chain denomination:

```rust
// near/omni-bridge/src/lib.rs  line 2776-2779
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← unchecked u128 × u128
}
``` [1](#0-0) 

`denormalize_fee` delegates directly to it:

```rust
// near/omni-bridge/src/lib.rs  line 2790-2795
fn denormalize_fee(fee: &Fee, decimals: Decimals) -> Fee {
    Fee {
        fee: U128(Self::denormalize_amount(fee.fee.0, decimals)),
        native_fee: fee.native_fee,
    }
}
``` [2](#0-1) 

Both `amount` (a `u128` representing the normalized bridge amount) and `10_u128.pow(diff_decimals)` (also a `u128`) are 128-bit values. Their product can exceed `u128::MAX` (≈ 3.4 × 10³⁸). For a token with `diff_decimals = 18` (e.g., an 18-decimal origin token normalized to 0 decimals on NEAR), overflow occurs for any `amount > 3.4 × 10²⁰`. In Rust release builds **without** `overflow-checks = true`, the multiplication wraps silently; with overflow checks enabled it panics, reverting the transaction.

### Impact Explanation
- **Silent wrap (overflow-checks off):** The denormalized amount returned is far smaller than the true value. The recipient receives far fewer tokens than they are owed, or the fee paid to the relayer is drastically under-calculated. This directly breaks bridge collateralization and misdirects value — a **High** accounting-corruption impact.
- **Panic (overflow-checks on):** Any `fin_transfer` or `claim_fee` call that triggers denormalization with a large amount reverts permanently, permanently locking the user's funds in the bridge — a **Critical** permanent-freeze impact.

### Likelihood Explanation
Any unprivileged bridge user can trigger this by initiating a transfer of a sufficiently large normalized amount for a token whose origin decimals exceed the bridge-side decimals. The decimal difference is set at token-binding time (admin-controlled), but once a token with `diff_decimals ≥ 18` is registered, any user submitting a large transfer amount reaches the overflow path. High-supply tokens (e.g., meme tokens with 18 decimals and enormous total supply) make this realistic.

### Recommendation
Replace the bare multiplication with a checked or widening variant:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount
        .checked_mul(10_u128.pow(diff_decimals))
        .expect("denormalize_amount overflow")
}
```

Or, if silent truncation/saturation is acceptable for dust, use `saturating_mul`. Either way, add a comment explaining the casting logic, mirroring the upstream recommendation.

### Proof of Concept
1. Admin registers a token with `origin_decimals = 18`, `decimals = 0` → `diff_decimals = 18`.
2. User calls `ft_transfer_call` (or the EVM `initTransfer`) with `amount = 4 × 10²⁰` (a valid token balance for a high-supply token).
3. On `fin_transfer` callback, the bridge calls `denormalize_amount(4e20, Decimals { decimals: 0, origin_decimals: 18 })`.
4. The intermediate product `4e20 * 10^18 = 4e38 > u128::MAX ≈ 3.4e38` overflows.
5. **With overflow-checks off:** the wrapped result (≈ `6.5e37`) is minted/transferred to the recipient instead of the correct `4e38` — the user loses ~84 % of their funds.
6. **With overflow-checks on:** the transaction panics and the user's funds remain locked in the bridge with no recovery path. [1](#0-0)
