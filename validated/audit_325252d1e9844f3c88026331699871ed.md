### Title
`compute_meta_tx_v0_hash` Omits Nonce, Enabling Replay of Meta-Transactions and Direct Loss of Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` computes the hash that a user signs to authorize a meta-transaction (version 0 invoke). The hash does **not** include the account nonce. Because the OS also explicitly skips nonce enforcement for version-0 transactions, the resulting signature is valid across every future account state and can be replayed an unlimited number of times by any party that possesses it, draining the signer's funds.

---

### Finding Description

`compute_meta_tx_v0_hash` is defined as:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        ...
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),  // ← no nonce
    );
    return tx_hash;
}
``` [1](#0-0) 

Compare this to `compute_l1_handler_transaction_hash`, which **does** include the nonce in `additional_data`:

```cairo
func compute_l1_handler_transaction_hash{pedersen_ptr: HashBuiltin*}(
    execution_context: ExecutionContext*, chain_id: felt, nonce: felt
) -> felt {
    let (transaction_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=1,
        additional_data=&nonce,   // ← nonce IS included
    );
    return transaction_hash;
}
``` [2](#0-1) 

The nonce omission is compounded by `check_and_increment_nonce`, which explicitly skips all version-0 transactions:

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
``` [3](#0-2) 

The two defects together mean:

1. The signed hash does not commit to any account state (no nonce in hash).
2. The OS does not enforce a nonce for version-0 transactions.

Therefore, a single user signature over a meta-transaction can be submitted to the OS an unlimited number of times.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A user who signs a meta-transaction authorizing, for example, an ERC-20 transfer of 100 tokens can have that transfer executed repeatedly by any party that observed the original submission. Each replay is accepted by the OS as valid because the hash is identical and no nonce check is performed. The user's entire token balance can be drained in a single block.

---

### Likelihood Explanation

Any unprivileged actor (a relayer, a mempool observer, or a malicious sequencer) who sees a submitted meta-transaction can immediately replay it. No privileged access, leaked key, or external dependency is required. The attacker only needs to re-submit the same calldata and signature that appeared on-chain or in the mempool.

---

### Recommendation

Include the account's current nonce in the meta-transaction hash, mirroring the pattern used by `compute_l1_handler_transaction_hash`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,           // ← add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=1,
        additional_data=&nonce,   // ← commit to nonce
    );
    return tx_hash;
}
```

Additionally, `check_and_increment_nonce` should enforce the nonce for meta-transactions (version 0) rather than skipping them, so that each signed hash can only be consumed once.

---

### Proof of Concept

1. User signs a meta-transaction: `meta_tx_v0(contract=TOKEN, selector=transfer, calldata=[recipient, 100])`.
2. Relayer submits the transaction; the OS calls `compute_meta_tx_v0_hash` with `additional_data_size=0` — nonce is absent from the hash.
3. OS executes the transfer; `check_and_increment_nonce` is skipped because `version == 0`.
4. Relayer (or any observer) re-submits the identical transaction object.
5. OS recomputes the same hash, verifies the same signature successfully, skips the nonce check again, and executes a second transfer.
6. Steps 4–5 repeat until the user's balance is zero.

The root cause is entirely within the scoped file `transaction_hash.cairo` (lines 294–315) and the nonce-skip in `execute_transaction_utils.cairo` (lines 63–68). No external dependency or privileged role is required.
