Audit Report

## Title
Router `pay()` consumes contract-wide ETH balance without scoping to caller's `msg.value`, enabling theft of stranded ETH — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

The `pay()` helper in `PeripheryPayments.sol` funds WETH swaps using `address(this).balance`, which is not scoped to the current transaction's `msg.value`. Any ETH left in the router by a prior caller who omitted `refundETH()` can be silently consumed by a subsequent caller who sends 0 ETH with `tokenIn = WETH`, causing direct, permanent loss of the prior user's principal.

## Finding Description

In `PeripheryPayments.sol` lines 73–84, when `token == WETH` and `payer != address(this)`, `pay()` reads the router's entire native balance:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
```

`address(this).balance` accumulates across transactions. A grep across all periphery contracts confirms **zero uses of `msg.value`** — no entry point validates that the caller's ETH contribution covers the requested swap amount.

`exactInputSingle` (line 67) is `external payable` with no ETH amount check. It stores `msg.sender` as payer via `_setNextCallbackContext`, then the pool calls back into `metricOmmSwapCallback` → `_justPayCallback` → `pay(WETH, originalCaller, pool, amountIn)`. If `address(this).balance >= amountIn` from a prior caller's stranded ETH, the `nativeBalance >= value` branch fires, wrapping and forwarding that ETH without touching the current caller's WETH allowance.

The same path is reachable through `MetricOmmPoolLiquidityAdder.metricOmmModifyLiquidityCallback()` at lines 172–177 when either pool token is WETH.

The `receive()` guard (line 32–34) only blocks direct ETH transfers from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` in any `payable` entry point.

## Impact Explanation

Direct loss of user principal. ETH sent by User A that is not recovered (e.g., because `refundETH()` was omitted from a multicall, or a partial revert left ETH stranded) can be consumed by User B's WETH swap at zero cost to User B. The loss equals the stranded ETH amount and is permanent — User A has no recourse once the ETH is wrapped and forwarded to the pool on User B's behalf. This meets the Critical/High threshold for direct loss of user principal.

## Likelihood Explanation

Medium. Users routinely send excess ETH to routers to avoid slippage-induced reverts and rely on `refundETH()` to recover the surplus. Omitting `refundETH()` from a multicall, or having a partial revert that leaves ETH in the router, is a realistic user error. An attacker monitoring the router's ETH balance on-chain can immediately exploit any non-zero balance by calling `exactInputSingle` with `tokenIn = WETH` and `msg.value = 0`.

## Recommendation

Track the ETH contributed by the current call in transient storage at the entry of each `payable` swap function, and in `pay()` use only that tracked amount — not `address(this).balance` — when funding WETH payments. Alternatively, remove native-ETH support from `pay()` entirely and require callers to wrap ETH themselves before invoking the router, eliminating the ambiguity between "ETH I sent now" and "ETH left by someone else."

## Proof of Concept

1. User A calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = 1 ETH`, `msg.value = 2 ETH`. The swap callback uses 1 ETH; 1 ETH remains in the router. User A omits `refundETH()`.
2. User B calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = 1 ETH`, `msg.value = 0`.
3. The pool calls `metricOmmSwapCallback`; `_justPayCallback` calls `pay(WETH, userB, pool, 1e18)`.
4. `address(this).balance == 1e18 >= 1e18` → router wraps User A's 1 ETH and transfers WETH to the pool.
5. User B receives full swap output. User A's 1 ETH is permanently lost; User B paid nothing. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
