### Title
EIP-712 Domain Chain ID Silently Truncated via `big.Int.Uint64()` Enabling Cross-Chain Replay - (File: `ethereum/eip712/encoding.go`, `ethereum/eip712/encoding_legacy.go`)

### Summary

`ParseChainID` returns an arbitrary-precision `*big.Int` that can represent chain IDs far exceeding `uint64` max (2^64−1). Both EIP-712 encoding paths immediately call `.Uint64()` on that value before constructing the domain separator, silently discarding all bits above bit 63. Two chains whose numeric IDs differ only in the upper bits (e.g., `1` and `2^64+1`) therefore produce an identical EIP-712 domain separator, making every EIP-712-signed Cosmos transaction on one chain replayable on the other.

### Finding Description

`ParseChainID` in `types/chain_id.go` accepts any chain-ID string up to 48 characters and returns a `*big.Int` with no upper-bound check on the numeric value. [1](#0-0) 

A 48-character chain-ID string can encode a number far larger than `uint64` max (≈1.8×10¹⁹). The regex `[1-9][0-9]*` imposes no numeric ceiling. [2](#0-1) 

In the Protobuf EIP-712 path, the returned `*big.Int` is immediately narrowed to `uint64` via `.Uint64()` before being passed to `WrapTxToTypedData`: [3](#0-2) 

The same truncation occurs in the Amino path: [4](#0-3) 

`WrapTxToTypedData` then calls `SafeInt64` on the already-truncated `uint64`, so the overflow guard fires only on the post-truncation value and never catches the original large chain ID: [5](#0-4) 

`createEIP712Domain` embeds this truncated value as the `chainId` field of the EIP-712 domain separator: [6](#0-5) 

The identical truncation exists in the legacy path via `LegacyWrapTxToTypedData`: [7](#0-6) [8](#0-7) 

Go's `big.Int.Uint64()` is defined to return only the low-order 64 bits with no error when the value exceeds `uint64` max. Therefore:

- Chain ID `18446744073709551617` (= 2^64 + 1) → `.Uint64()` = **1**
- Chain ID `36893488147419103233` (= 2^65 + 1) → `.Uint64()` = **1**
- Chain ID `1` → `.Uint64()` = **1**

All three produce an EIP-712 domain with `chainId = 1`.

### Impact Explanation

Any EIP-712-signed Cosmos transaction (Amino or Protobuf) submitted to a chain whose numeric ID truncates to the same `uint64` value as another chain is **fully replayable** on that other chain. The domain separator — the only chain-binding component of the EIP-712 signature — is identical across all chains sharing the same truncated value. An attacker who observes a valid signed transaction on chain `2^64+1` can broadcast it verbatim on chain `1` (Ethereum mainnet EVM-compatible chain) and it will pass signature verification. This constitutes a chain-ID signer verification bypass enabling replay and unauthorized account mutation, matching the High impact category.

### Likelihood Explanation

`ParseChainID` explicitly supports chain IDs up to 48 characters, which is the documented and enforced limit. Any operator who deploys an Ethermint chain with a numeric ID exceeding 2^64 (a 20+ digit number, well within the 48-char budget) is silently exposed. The attacker entry path is a standard unprivileged EIP-712 transaction submission — no special role or key is required. The attacker only needs to observe a signed transaction on one chain and resubmit it on the colliding chain.

### Recommendation

Replace the silent `.Uint64()` call with a checked conversion that returns an error when the chain ID exceeds `uint64` max (or `int64` max, consistent with `SafeInt64`). In both `decodeAminoSignDoc` and `decodeProtobufSignDoc`, and in `LegacyWrapTxToTypedData`, validate that `chainID.IsUint64()` (Go's `big.Int` method that returns `false` when the value overflows `uint64`) before calling `.Uint64()`, and reject the transaction with an explicit error otherwise. Alternatively, pass the `*big.Int` directly through to `createEIP712Domain` and use `math.NewHexOrDecimal256` directly on the big.Int, bypassing the lossy integer narrowing entirely.

### Proof of Concept

```
Chain A: "mychain_18446744073709551617-1"   // numeric ID = 2^64 + 1
Chain B: "mainnet_1-1"                      // numeric ID = 1

Step 1: User on Chain A signs an EIP-712 Cosmos tx.
        ParseChainID("mychain_18446744073709551617-1") → big.Int{2^64+1}
        big.Int{2^64+1}.Uint64()            → 1   (silent truncation)
        SafeInt64(1)                         → 1, nil
        createEIP712Domain(1)               → domain{chainId: 1}
        Signature covers domain{chainId: 1}

Step 2: Attacker copies the signed tx bytes verbatim.

Step 3: Attacker submits to Chain B ("mainnet_1-1").
        ParseChainID("mainnet_1-1")         → big.Int{1}
        big.Int{1}.Uint64()                 → 1
        createEIP712Domain(1)               → domain{chainId: 1}   ← identical
        Signature verification passes.      ← replay succeeds
```

func ParseChainID(chainID string) (*big.Int, error) {
	chainID = strings.TrimSpace(chainID)
	if len(chainID) > 48 {
		return nil, errorsmod.Wrapf(ErrInvalidChainID, "chain-id '%s' cannot exceed 48 chars", chainID)
	}

	matches := ethermintChainID.FindStringSubmatch(chainID)
	if matches == nil || len(matches) != 4 || matches[1] == "" {
		return nil, errorsmod.Wrapf(ErrInvalidChainID, "%s: %v", chainID, matches)
	}

	// verify that the chain-id entered is a base 10 integer
	chainIDInt, ok := new(big.Int).SetString(matches[2], 10)
	if !ok {
		return nil, errorsmod.Wrapf(ErrInvalidChainID, "epoch %s must be base-10 integer format", matches[2])
	}

	return chainIDInt, nil

func WrapTxToTypedData(
	chainID uint64,
	data []byte,
) (apitypes.TypedData, error) {
	messagePayload, err := createEIP712MessagePayload(data)
	message := messagePayload.message
	if err != nil {
		return apitypes.TypedData{}, err
	}

	types, err := createEIP712Types(messagePayload)
	if err != nil {
		return apitypes.TypedData{}, err
	}

	value, err := ethermint.SafeInt64(chainID)
	if err != nil {
		return apitypes.TypedData{}, err
	}
	domain := createEIP712Domain(value)

	typedData := apitypes.TypedData{
		Types:       types,
		PrimaryType: txField,
		Domain:      domain,
		Message:     message,
	}

	return typedData, nil
