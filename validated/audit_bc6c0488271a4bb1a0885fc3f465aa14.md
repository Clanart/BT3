### Title
Reversed Field Order in `compute_deploy_account_transaction_hash` Produces Invalid Transaction Hashes — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

### Summary
The `compute_deploy_account_transaction_hash` function hashes `constructor_calldata` **before** `class_hash` and `contract_address_salt`, which is the reverse of the order mandated by the StarkNet deploy_account v3 transaction hash specification. Every deploy_account v3 transaction processed by this OS will carry a hash that differs from the hash wallets/clients compute per spec, causing `__validate_deploy__` signature verification to fail for every such transaction.

### Finding Description
The StarkNet deploy_account v3 transaction hash specification (SNIP-12 / StarkNet docs) defines the Poseidon hash input — after the common fields — as:

```
class_hash  →  contract_address_salt  →  h(constructor_calldata)
```

Inside `compute_deploy_account_transaction_hash`, the `calldata` pointer is laid out as `[class_hash, salt, constructor_calldata…]` (assembled in `execute_deploy_account_transaction`). The function then updates the hash state in this order:

```cairo
// Hash and add the constructor calldata to the hash state.
poseidon_hash_update_with_nested_hash(data_ptr=&calldata[2], data_length=calldata_size - 2);
// Add the class hash and the contract address salt to the hash state.
poseidon_hash_update(data_ptr=calldata, data_length=2);
``` [1](#0-0) 

This produces the sequence:

```
h(constructor_calldata)  →  class_hash  →  contract_address_salt
```

which is the **inverse** of the specification. Compare with `compute_declare_transaction_hash`, which correctly places `class_hash` and `compiled_class_hash` **before** any data array:

```cairo
poseidon_hash_update_single(item=class_hash);
poseidon_hash_update_single(item=compiled_class_hash);
``` [2](#0-1) 

The calldata layout is assembled in `execute_deploy_account_transaction`:

```cairo
assert validate_deploy_calldata[0] = constructor_execution_context.class_hash;
assert validate_deploy_calldata[1] = salt;
memcpy(dst=&validate_deploy_calldata[2], …);
``` [3](#0-2) 

and then passed directly to the hash function:

```cairo
let transaction_hash = compute_deploy_account_transaction_hash(
    common_fields=common_tx_fields,
    calldata_size=validate_deploy_calldata_size,
    calldata=validate_deploy_calldata,
);
``` [4](#0-3) 

### Impact Explanation
The OS writes the incorrectly-ordered hash into `tx_info.transaction_hash` and passes it to the account contract's `__validate_deploy__` entry point. Every conforming wallet computes the correct hash per spec and signs it. The OS presents a different hash to the account contract, so ECDSA/Stark-curve signature verification fails for **every** deploy_account v3 transaction. No new StarkNet account can be deployed. This satisfies **High — Network not being able to confirm new transactions** (the entire deploy_account transaction class is permanently broken).

### Likelihood Explanation
The broken code path is exercised by every deploy_account v3 transaction — the only supported version (`assert common_fields.version = 3`). There is no fallback or version-gating that bypasses this function. Likelihood is **High** (certain to trigger on any deploy_account attempt). [5](#0-4) 

### Recommendation
Swap the two `poseidon_hash_update*` calls so that `class_hash` and `contract_address_salt` are hashed **before** the nested constructor calldata hash, matching the specification:

```cairo
// Add the class hash and the contract address salt to the hash state.
poseidon_hash_update(data_ptr=calldata, data_length=2);
// Hash and add the constructor calldata to the hash state.
poseidon_hash_update_with_nested_hash(data_ptr=&calldata[2], data_length=calldata_size - 2);
```

### Proof of Concept

1. A user constructs a deploy_account v3 transaction. Their wallet computes the spec-compliant hash:
   ```
   H_correct = poseidon(…common…, class_hash, salt, h(constructor_calldata))
   ```
   and signs it: `sig = sign(H_correct)`.

2. The sequencer submits the block to the OS. The OS calls `compute_deploy_account_transaction_hash`, which produces:
   ```
   H_wrong = poseidon(…common…, h(constructor_calldata), class_hash, salt)
   ```
   `H_wrong ≠ H_correct`.

3. The OS writes `H_wrong` into `tx_info.transaction_hash` and invokes `__validate_deploy__` on the account contract.

4. The account contract verifies `sig` against `H_wrong` — verification fails.

5. The transaction is reverted. No account is deployed. This happens for **every** deploy_account v3 transaction, permanently blocking new account creation. [6](#0-5) [7](#0-6)

func compute_deploy_account_transaction_hash{range_check_ptr, poseidon_ptr: PoseidonBuiltin*}(
    common_fields: CommonTxFields*, calldata_size: felt, calldata: felt*
) -> felt {
    alloc_locals;

    with_attr error_message("Invalid transaction version: {version}.") {
        assert common_fields.version = 3;
    }

    let hash_state: PoseidonHashState = poseidon_hash_init();
    with hash_state {
        hash_tx_common_fields(common_fields=common_fields);
        // Hash and add the constructor calldata to the hash state.
        poseidon_hash_update_with_nested_hash(data_ptr=&calldata[2], data_length=calldata_size - 2);
        // Add the class hash and the contract address salt to the hash state.
        poseidon_hash_update(data_ptr=calldata, data_length=2);
    }

    let transaction_hash = poseidon_hash_finalize(hash_state=hash_state);
    return transaction_hash;
}
