### Title
Legacy EIP-712 `Tx` Type Schema Omits `timeout_height`, Allowing Signature Reuse After Intended Expiry - (`ethereum/eip712/eip712_legacy.go`)

### Summary

The Legacy EIP-712 `Tx` type schema intentionally omits the `timeout_height` field, but `VerifySignature` in `ante/cosmos/eip712.go` builds the Amino sign bytes with the actual `tx.GetTimeoutHeight()` value. Because EIP-712 only hashes fields declared in the type schema, `timeout_height` is never covered by the signature. An attacker who observes a signed Legacy EIP-712 transaction with a non-zero `timeout_height` can strip or alter that field and resubmit the transaction with the original `FeePayerSig` after the intended expiry block, and signature verification will still pass.

### Finding Description

**Root cause — missing field in EIP-712 type schema**

In `ethereum/eip712/eip712_legacy.go`, `extractMsgTypes` defines the `Tx` type without `timeout_height`:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id",       Type: "string"},
    {Name: "fee",            Type: "Fee"},
    {Name: "memo",           Type: "string"},
    {Name: "msgs",           Type: "Msg[]"},
    {Name: "sequence",       Type: "string"},
    // Note timeout_height was removed because it was not getting filled with the legacyTx
},
``` [1](#0-0) 

**Sign bytes DO include `timeout_height`**

In `ante/cosmos/eip712.go`, `VerifySignature` builds `txBytes` using the actual `timeout_height` from the submitted transaction:

```go
txBytes := legacytx.StdSignBytes(
    signerData.ChainID,
    signerData.AccountNumber,
    signerData.Sequence,
    tx.GetTimeoutHeight(),   // ← non-zero when signer set an expiry
    legacytx.StdFee{Amount: tx.GetFee(), Gas: tx.GetGas()},
    msgs, tx.GetMemo(),
)
``` [2](#0-1) 

`legacytx.StdSignBytes` serialises `timeout_height` into the JSON when it is non-zero (the field carries `omitempty`, so it is absent only when zero). `LegacyWrapTxToTypedData` then unmarshals those bytes into `txData` and uses it as the EIP-712 message: [3](#0-2) [4](#0-3) 

Because `timeout_height` is present in `txData` but absent from the `Tx` type schema, EIP-712 silently ignores it when computing the struct hash. The resulting `sigHash` is therefore **identical** whether `timeout_height` is 0 or any positive integer.

**Verification path that accepts the forged transaction**

`VerifySignature` recovers the fee-payer public key from `sigHash` and compares it to the account's stored key: [5](#0-4) 

Since `sigHash` is the same for both the original and the modified transaction, the recovered key matches and the check passes.

### Impact Explanation

An attacker who observes a signed Legacy EIP-712 transaction in the mempool with `timeout_height = N` can:

1. Copy the `FeePayerSig` from `ExtensionOptionsWeb3Tx`.
2. Reconstruct the transaction with `timeout_height = 0` (or any other value) while keeping all other fields (messages, fees, sequence, chain ID, account number, `FeePayer`) identical.
3. Submit the modified transaction after block N.
4. `VerifySignature` computes the same `sigHash` → signature verification passes → the transaction executes.

This bypasses the `timeout_height` safety mechanism entirely for the Legacy EIP-712 path. A user who signs a bank transfer, governance vote, or staking operation with an expiry block to limit its validity window has no protection: the attacker can execute it at any future block.

This matches the allowed High impact: **EIP-712 authorization bypass enabling forged execution**.

### Likelihood Explanation

The Legacy EIP-712 ante handler (`newLegacyCosmosAnteHandlerEip712`) is still active and reachable via the `/ethermint.types.v1.ExtensionOptionsWeb3Tx` extension option path. [6](#0-5) 

Any user who constructs a Legacy EIP-712 Cosmos transaction with a non-zero `timeout_height` (a standard Cosmos SDK safety feature) is vulnerable. The attacker only needs to observe the mempool — no privileged access is required.

### Recommendation

Add `timeout_height` to the `Tx` type schema in `extractMsgTypes` and ensure `legacytx.StdSignBytes` always serialises it (even when zero, to avoid the `omitempty` gap):

```go
"Tx": {
    {Name: "account_number",  Type: "string"},
    {Name: "chain_id",        Type: "string"},
    {Name: "fee",             Type: "Fee"},
    {Name: "memo",            Type: "string"},
    {Name: "msgs",            Type: "Msg[]"},
    {Name: "sequence",        Type: "string"},
    {Name: "timeout_height",  Type: "string"},  // ← add this
},
```

Correspondingly, ensure `txBytes` always includes `timeout_height` (pass it as a string even when 0) so the field is present in `txData` and covered by the hash.

### Proof of Concept

```
Alice signs Legacy EIP-712 tx:
  timeout_height = 500, sequence = 7, msgs = [MsgSend(Alice→Bob, 100aphoton)]
  FeePayerSig = Sign(EIP712Hash(txData_without_timeout_height))

Block 500 passes; tx expires in mempool.

Attacker constructs:
  timeout_height = 0  (all other fields identical)
  FeePayerSig = Alice's original FeePayerSig  (copied verbatim)

VerifySignature:
  txBytes = StdSignBytes(..., timeoutHeight=0, ...)  → JSON omits timeout_height
  sigHash  = EIP712Hash(txData_without_timeout_height)  ← same hash as Alice's
  Ecrecover(sigHash, FeePayerSig) → Alice's pubkey  ✓
  recoveredFeePayerAcc == feePayer  ✓
  → verification passes

Result: MsgSend executes at block 600, after Alice's intended expiry.
```
