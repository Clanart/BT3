### Title
Hardcoded `Felt::ZERO` Instead of Actual `gas_consumed.l2_gas` in Receipt Commitment Produces Wrong Block Hash for L2-Gas-Bearing Transactions — (`crates/starknet_api/src/block_hash/receipt_commitment.rs`)

---

### Summary

`chain_gas_consumed` in `receipt_commitment.rs` always chains the literal `Felt::ZERO` for the L2-gas-consumed slot of every receipt leaf, ignoring the actual `gas_consumed.l2_gas` value that is present in the `GasVector`. Starting with Starknet v0.14.3, transactions carry non-zero L2 gas (e.g. `76 014 735` in the shipped test fixture). Because the L2-gas dimension is silently dropped, every receipt leaf hash is computed as if L2 gas were zero, making the receipt commitment — and therefore the block hash — wrong for any block that contains at least one L2-gas-bearing transaction.

---

### Finding Description

**Root cause — hardcoded zero instead of the variable field**

```rust
// crates/starknet_api/src/block_hash/receipt_commitment.rs  lines 86-91
fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
    hash_chain
        .chain(&Felt::ZERO)               // ← always 0, ignores gas_consumed.l2_gas
        .chain(&gas_consumed.l1_gas.into())
        .chain(&gas_consumed.l1_data_gas.into())
}
``` [1](#0-0) 

The `GasVector` type carries three independent dimensions:

```rust
pub struct GasVector {
    pub l1_gas: GasAmount,
    pub l1_data_gas: GasAmount,
    #[serde(default)]
    pub l2_gas: GasAmount,   // ← populated but never chained
}
``` [2](#0-1) 

**Data path that delivers non-zero `l2_gas`**

The feeder-gateway client converts `total_gas_consumed` from the gateway JSON into `gas_consumed` on `ExecutionResources`:

```rust
gas_consumed: match execution_resources.total_gas_consumed {
    Some(total_gas_consumed) => total_gas_consumed,
    None => GasVector { l1_gas: GasAmount(0), l2_gas: GasAmount(0), l1_data_gas: ... },
},
``` [3](#0-2) 

The shipped v0.14.3 test fixture confirms non-zero values are present in production data:

```json
"total_gas_consumed": { "l1_gas": 0, "l1_data_gas": 128, "l2_gas": 76014735 }
``` [4](#0-3) 

`ReceiptElement` is built directly from `TransactionHashingData.transaction_output.gas_consumed`, so the non-zero `l2_gas` reaches `chain_gas_consumed` — where it is silently discarded. [5](#0-4) 

**Commitment chain**

`calculate_receipt_commitment` feeds into `calculate_block_commitments`, which feeds into `calculate_block_hash`:

```rust
let receipt_task =
    spawn_measured_task(move || calculate_receipt_commitment::<Poseidon>(&receipt_elements));
``` [6](#0-5) 

```rust
.chain(&block_commitments.receipt_commitment.0)
``` [7](#0-6) 

Any block containing a transaction with `l2_gas > 0` therefore produces a receipt commitment — and a block hash — that does not match the value a correct implementation would compute.

---

### Impact Explanation

**Critical — Wrong receipt commitment / wrong block hash.**

Two transactions that differ only in `l2_gas` produce identical receipt leaf hashes. The receipt commitment over a block with non-zero L2 gas is wrong. Because `receipt_commitment` is chained directly into `calculate_block_hash`, the final block hash stored in storage, returned by RPC, and fed to the proof manager is incorrect for every v0.14.3+ block that contains L2-gas-bearing transactions. This matches:

> *Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.*

and

> *RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.*

---

### Likelihood Explanation

**High.** Starknet v0.14.3 is already deployed and its blocks carry non-zero `l2_gas` in `total_gas_consumed`. Every such block triggers the wrong receipt commitment. No special attacker action is required; ordinary user transactions are sufficient.

---

### Recommendation

Replace the hardcoded `Felt::ZERO` with the actual `l2_gas` field:

```diff
 fn chain_gas_consumed(hash_chain: HashChain, gas_consumed: &GasVector) -> HashChain {
     hash_chain
-        .chain(&Felt::ZERO) // L2 gas consumed
+        .chain(&gas_consumed.l2_gas.into())
         .chain(&gas_consumed.l1_gas.into())
         .chain(&gas_consumed.l1_data_gas.into())
 }
``` [1](#0-0) 

If the protocol spec intentionally fixes L2 gas to zero for a specific version range, add an explicit version gate and document the invariant, rather than silently ignoring the field.

---

### Proof of Concept

1. Take any v0.14.3 block (e.g. the shipped `block_post_0_14_3.json` fixture) whose receipts contain `total_gas_consumed.l2_gas > 0`.
2. Call `calculate_receipt_commitment` on those `ReceiptElement`s — the result chains `Felt::ZERO` for every L2-gas slot.
3. Construct a second set of `ReceiptElement`s identical except with `gas_consumed.l2_gas = 0` — `calculate_receipt_commitment` returns the **same** value.
4. Feed both into `calculate_block_hash` — both produce the same block hash, demonstrating that the L2-gas dimension is invisible to the commitment.
5. Compare the sequencer-computed block hash against the on-chain value for a v0.14.3 block; the mismatch confirms the wrong commitment.
