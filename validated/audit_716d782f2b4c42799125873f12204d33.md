### Title
Blacklisted ERC20 Recipient Causes Permanent Fund Lock in `finTransfer` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

In `OmniBridge.sol::finTransfer`, when the destination token is a blacklisting ERC20 (e.g., USDC, USDT) and the signed `payload.recipient` is blacklisted, the `safeTransfer` call reverts unconditionally. Because the recipient address is embedded in the MPC-signed payload and cannot be substituted, the transfer can never be finalized on EVM. The corresponding tokens are already locked or burned on NEAR with no user-accessible recovery path, resulting in permanent fund loss.

---

### Finding Description

`OmniBridge.sol::finTransfer` processes cross-chain settlement in a single atomic transaction:

```
completedTransfers[payload.destinationNonce] = true;   // line 287
...
IERC20(payload.tokenAddress).safeTransfer(             // line 351
    payload.recipient,
    payload.amount
);
``` [1](#0-0) [2](#0-1) 

The nonce is marked used at line 287 before the transfer, but because the entire transaction is atomic, a revert at line 351 also rolls back the nonce marking. This means the nonce is never consumed — but the transfer also never succeeds.

The `payload.recipient` is part of the MPC-signed Borsh-encoded message:

```solidity
Borsh.encodeAddress(payload.recipient),
``` [3](#0-2) 

Because `payload.recipient` is cryptographically bound to the signature, no alternative recipient can be substituted. Any call to `finTransfer` with the same signed payload will always revert if the recipient remains blacklisted.

On the NEAR side, tokens are locked or burned at `init_transfer` time (via `ft_on_transfer` → `init_transfer_internal`), and the transfer record is stored in `pending_transfers`. [4](#0-3) 

There is no user-accessible `cancel_transfer` or refund function visible in the NEAR bridge contract. The only token-recovery function is `transfer_token_as_dao`, which is restricted to the `DAO` role and is not a protocol-level user remedy. [5](#0-4) 

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds in bridge flows.**

A user who initiates a NEAR → EVM transfer of a blacklisting token (USDC, USDT, etc.) and whose EVM address is subsequently blacklisted will have their tokens permanently locked on NEAR. The signed payload cannot be altered, `finTransfer` will revert on every attempt, and no user-callable recovery path exists. This matches the Critical/High allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

USDC and USDT are among the most-bridged assets and both implement address blacklisting. Blacklisting events occur regularly due to regulatory enforcement, sanctions compliance, or theft response. A user whose address is blacklisted between `initTransfer` on NEAR and `finTransfer` on EVM — a window that can span minutes to hours depending on relayer latency and MPC signing time — will trigger this condition. No attacker action is required; the victim's own blacklisting is sufficient. Likelihood is **Low-Medium** (blacklisting is rare per address but the protocol handles high-value transfers, making the expected loss significant).

---

### Recommendation

1. **Wrap the token transfer in a try/catch pattern** (or use a low-level call with a success check) so that a failed delivery does not permanently block the nonce. Store the undeliverable amount in a claimable escrow keyed by `(destinationNonce, recipient)`.
2. **Provide a user-callable `claimTransfer` function** that allows the original recipient (or a DAO-approved substitute) to pull funds from escrow after a failed `finTransfer`.
3. Alternatively, **mark the nonce as used before attempting the transfer** and emit a `TransferFailed` event, then allow the user to re-route via a separate claim with a fresh recipient address (requiring a new MPC signature over the updated recipient).

---

### Proof of Concept

1. Alice holds 10,000 USDC on NEAR and calls `ft_transfer_call` to the NEAR bridge with `recipient = OmniAddress::Eth(alice_evm_address)`. Tokens are locked on NEAR; a `TransferMessage` is stored in `pending_transfers`.
2. The MPC network signs a `TransferMessagePayload` with `tokenAddress = USDC_EVM`, `recipient = alice_evm_address`, `amount = 10000e6`.
3. Before the relayer submits `finTransfer`, Circle blacklists `alice_evm_address` (e.g., due to a sanctions hit).
4. The relayer calls `OmniBridge.finTransfer(signatureData, payload)`. Execution reaches line 351: `IERC20(USDC).safeTransfer(alice_evm_address, 10000e6)`. USDC reverts with `Blacklistable: account is blacklisted`.
5. The entire transaction reverts. `completedTransfers[nonce]` is rolled back to `false`.
6. Every subsequent attempt to call `finTransfer` with the same signed payload reverts identically.
7. Alice's 10,000 USDC remain locked in the NEAR bridge contract indefinitely with no user-callable recovery function. [6](#0-5)
