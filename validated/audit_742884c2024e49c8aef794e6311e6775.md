### Title
`OmniBridge::initTransfer` Accepts ERC721 Tokens via ABI Selector Collision, Causing Permanent NFT Lock — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge::initTransfer` is intended exclusively for ERC20 tokens, but it also silently accepts ERC721 NFTs. Because ERC721's `transferFrom(address,address,uint256)` shares the exact same ABI selector as ERC20's `transferFrom(address,address,uint256)`, OpenZeppelin's `SafeERC20.safeTransferFrom` succeeds when called against an ERC721 contract. The NFT is locked inside the bridge, but the NEAR side cannot process the transfer (the token has no registered decimals), and there is no on-chain refund path. The NFT is permanently irrecoverable.

---

### Finding Description

`initTransfer` in `OmniBridge.sol` handles the "native ERC20" case with:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
``` [1](#0-0) 

There is no check that `tokenAddress` is actually an ERC20 token. ERC721's `transferFrom(address from, address to, uint256 tokenId)` has the **identical 4-byte selector** as ERC20's `transferFrom(address from, address to, uint256 amount)`. OpenZeppelin's `SafeERC20` wrapper accepts an empty return value (ERC721's `transferFrom` returns nothing), so the call succeeds and the NFT with `tokenId == amount` is transferred into the bridge. [2](#0-1) 

After the NFT is locked, the NEAR bridge's `fin_transfer_callback` attempts to look up the token's registered decimals:

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
``` [3](#0-2) 

Because `logMetadata` was never successfully called for the ERC721 token (standard ERC721 tokens do not implement `decimals()`, so `logMetadata` reverts), `token_decimals` has no entry and the NEAR callback panics. The transfer is never finalized on NEAR, but the NFT is already locked in the EVM bridge with no refund mechanism.

Even in the secondary case where an ERC721 token happens to implement `decimals()` (returning 0), allowing `logMetadata` to succeed and the NEAR side to register the token, the return leg still fails: `finTransfer` calls `IERC20(tokenAddress).safeTransfer(recipient, amount)`, but ERC721 has no `transfer(address,uint256)` function, causing a revert and leaving the NFT permanently stuck. [4](#0-3) 

The `logMetadata` function itself also performs no ERC20 type validation:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name = IERC20Metadata(tokenAddress).name();
    string memory symbol = IERC20Metadata(tokenAddress).symbol();
    uint8 decimals = IERC20Metadata(tokenAddress).decimals();
    ...
}
``` [5](#0-4) 

---

### Impact Explanation

Any ERC721 NFT bridged via `initTransfer` is permanently locked in the `OmniBridge` contract with no recovery path. This matches the allowed impact: **Critical — Permanent freezing / irrecoverable lock of user funds in bridge or vault flows.**

---

### Likelihood Explanation

The entry point is fully permissionless — any user can call `initTransfer` with an ERC721 address. The `amount` parameter is `uint128`, which accommodates the vast majority of real-world NFT tokenIds (sequential integers starting from 0 or 1). A user who mistakes an NFT for a fungible token, or who is misled by a UI, triggers the loss with a single transaction. No privileged role or special condition is required.

---

### Recommendation

Add an ERC20 type guard in `initTransfer` before the `safeTransferFrom` call. The most robust option is to require that `decimals()` succeeds and returns a non-zero value (ERC721 tokens either lack `decimals()` entirely or return 0). Alternatively, use ERC165 to reject tokens that advertise the `IERC721` interface (`0x80ac58cd`). The same guard should be applied in `logMetadata` to prevent registration of non-ERC20 tokens.

---

### Proof of Concept

1. Deploy a standard ERC721 contract (e.g., OpenZeppelin `ERC721`).
2. Mint tokenId `5` to `user`.
3. `user` approves `OmniBridge` for tokenId `5`.
4. `user` calls `OmniBridge.initTransfer(erc721Address, 5, 0, 0, "recipient.near", "")`.
5. `SafeERC20.safeTransferFrom` succeeds — tokenId `5` is now owned by `OmniBridge`.
6. The NEAR bridge's `fin_transfer_callback` panics with `TokenDecimalsNotFound` because `logMetadata` was never called (it would revert on `decimals()`).
7. The NFT is permanently locked; `user` has no on-chain mechanism to recover it.
