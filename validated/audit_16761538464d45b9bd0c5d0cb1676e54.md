### Title
Gateway `skip_stateful_validations` Admits Unauthenticated Invoke Transactions, Enabling Frontrunning DoS Against Deploy-Account Users — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the Apollo gateway unconditionally bypasses `__validate__` (signature verification) for any `Invoke` transaction with `nonce=1` sent to an undeployed account that has a `DEPLOY_ACCOUNT` in the mempool. An attacker who observes a victim's `DEPLOY_ACCOUNT` in the mempool can race-submit a crafted `Invoke(nonce=1)` with an arbitrary or empty signature for the victim's address. The gateway admits it without any signature check, it occupies the victim's nonce-1 slot in the mempool, and the victim's legitimate `Invoke` is subsequently rejected with `DuplicateNonce`. This is the direct sequencer analog of the ERC-20 permit-frontrunning DoS: a one-time-use credential (the nonce slot) is consumed by an unprivileged third party before the legitimate owner can use it.

---

### Finding Description

**Root cause — unconditional signature skip**

`skip_stateful_validations` returns `true` (skip) whenever three conditions hold simultaneously: [1](#0-0) 

```
tx is Invoke  AND  tx.nonce() == 1  AND  account_nonce == 0
AND  account_tx_in_pool_or_recent_block(sender) == true
```

When `skip_validate = true`, `run_validate_entry_point` sets `ExecutionFlags { validate: false, … }`: [2](#0-1) 

and `validate_tx` returns `Ok(None)` immediately without calling `__validate__`: [3](#0-2) 

**The three conditions are all attacker-controllable**

| Condition | Attacker control |
|---|---|
| `tx is Invoke` | Attacker chooses tx type |
| `tx.nonce() == 1` | Attacker sets nonce field |
| `account_nonce == 0` | True for any undeployed account |
| `account_tx_in_pool_or_recent_block` | True once victim's `DEPLOY_ACCOUNT` is visible in the mempool |

`account_tx_in_pool_or_recent_block` checks only whether the account has *any* entry in the pool or recent block — not whether the caller owns the account: [4](#0-3) 

**`validate_by_mempool` does not close the gap**

`validate_by_mempool` is called before `skip_stateful_validations` but only checks for duplicate *transaction hash* and nonce-too-old: [5](#0-4) [6](#0-5) 

It does **not** reject a different transaction with the same `(address, nonce)` pair. That duplicate-nonce rejection only fires in `add_tx` when the slot is already occupied — i.e., it is a race condition the attacker wins by submitting first.

**Nonce check in `validate_nonce` also passes**

For the attacker's `Invoke(nonce=1)` against an account with `account_nonce=0`, the general-case branch allows any nonce in `[account_nonce, account_nonce + max_allowed_nonce_gap]`: [7](#0-6) 

With the default `max_allowed_nonce_gap ≥ 1`, nonce=1 passes.

---

### Impact Explanation

The attacker's crafted `Invoke(nonce=1)` is admitted to the mempool without any signature check. It occupies the victim's nonce-1 slot. The victim's legitimate `Invoke(nonce=1)` is then rejected by the mempool with `DuplicateNonce`. When the batcher eventually executes the attacker's transaction, `__validate__` fails (wrong signature), the nonce is **not** incremented (state rolled back), and the victim can resubmit — but the attacker can repeat the attack indefinitely, creating a persistent DoS.

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

The gateway accepts an invalid transaction (no valid signature) and rejects a valid one.

---

### Likelihood Explanation

- The attacker needs no privileged access — only the ability to submit transactions to the public gateway.
- The victim's `DEPLOY_ACCOUNT` is visible in the mempool before the victim's `Invoke` is processed (the two are submitted together but processed sequentially).
- The attacker only needs to submit one transaction per attack round; the cost is a single gas-fee payment for a transaction that will revert.
- The attack is repeatable at low cost.

---

### Recommendation

Replace the unconditional skip with a **try-then-fallback** pattern analogous to `trustlessPermit`:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    // Never skip signature validation.
    // If __validate__ fails solely because the account is not yet deployed,
    // the blockifier already handles this gracefully during execution.
    Ok(false)
}
```

Alternatively, if the UX feature must be preserved, restrict the skip to transactions whose `sender_address` matches the `contract_address` field of the `DEPLOY_ACCOUNT` transaction already in the mempool, and verify that the `DEPLOY_ACCOUNT` was submitted in the same gateway request (e.g., via a session token or atomic batch endpoint). This prevents a third party from triggering the skip for an arbitrary account.

### Proof of Concept

```
1. Alice signs and submits:
      DEPLOY_ACCOUNT(class_hash=C, salt=S, constructor_calldata=D)
      INVOKE(sender=alice_addr, nonce=1, calldata=..., signature=alice_sig)

2. Gateway processes DEPLOY_ACCOUNT first → added to mempool.
   alice_addr is now visible via account_tx_in_pool_or_recent_block.

3. Attacker observes alice_addr in mempool.
   Attacker submits:
      INVOKE(sender=alice_addr, nonce=1, calldata=anything, signature=0x0)

4. Gateway stateful validation for attacker's tx:
      validate_nonce:          0 <= 1 <= max_allowed_nonce_gap  → PASS
      validate_by_mempool:     no dup hash, nonce not too old   → PASS
      skip_stateful_validations:
          tx is Invoke ✓, nonce==1 ✓, account_nonce==0 ✓,
          account_tx_in_pool_or_recent_block(alice_addr)==true ✓
          → returns true (SKIP)
      run_validate_entry_point: validate=false → __validate__ NOT called

5. Attacker's INVOKE(nonce=1) is added to mempool.

6. Alice's INVOKE(nonce=1) arrives at mempool → DuplicateNonce → REJECTED.

7. Batcher executes attacker's INVOKE → __validate__ fails (bad sig) →
   nonce NOT incremented, state rolled back.

8. Attacker repeats from step 3 indefinitely.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
