### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Signature Replay — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_meta_tx_v0_hash` function computes a transaction hash that omits a nonce (or any replay-prevention field), and the OS explicitly skips nonce enforcement for version-0 transactions. This is the direct analog of the external report's missing-parameter-in-signature vulnerability: just as the NFT `preMint()` signature lacked the contract address, the meta-tx-v0 hash lacks a nonce, allowing the same user-signed payload to be replayed indefinitely across different outer transactions.

---

### Finding Description

`compute_meta_tx_v0_hash` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`, meaning no nonce is ever mixed into the hash: [1](#0-0) 

The resulting `TxInfo` constructed in `execute_meta_tx_v0` hard-codes `nonce=0`: [2](#0-1) 

The OS nonce-enforcement gate in `check_and_increment_nonce` explicitly returns early for any version-0 transaction, so no nonce is ever checked or incremented: [3](#0-2) 

The hash therefore binds only: `INVOKE_HASH_PREFIX`, `version=0`, `contract_address`, `entry_point_selector`, `calldata`, `max_fee=0`, and `chain_id`. There is no field that changes between successive uses of the same signature. [4](#0-3) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

A user who signs a meta-tx-v0 payload (e.g., to transfer ERC-20 tokens via `__execute__`) produces a signature that is permanently valid for that `(contract_address, calldata, chain_id)` tuple. Any party who observes the signature — on-chain or off-chain — can wrap it in a fresh outer transaction and re-invoke `__execute__` on the victim's account contract an unlimited number of times, draining the account. [5](#0-4) 

---

### Likelihood Explanation

**High.** The `meta_tx_v0` syscall is callable by any contract, and the signature is necessarily broadcast when the first outer transaction is submitted. No privileged access, leaked key, or Sybil attack is required. Any on-chain observer can extract the signature from the first transaction's calldata and replay it in a new outer transaction.

---

### Recommendation

Include a user-controlled nonce in `compute_meta_tx_v0_hash` (passed as `additional_data`) and enforce its uniqueness in `execute_meta_tx_v0` — either by checking and incrementing the target contract's OS-level nonce, or by maintaining a separate per-contract replay-prevention mapping for meta-tx-v0 nonces.

---

### Proof of Concept

1. Alice signs a meta-tx-v0 payload: `hash = H(INVOKE_PREFIX, 0, alice_addr, __execute__, calldata, 0, chain_id)`. The calldata encodes "transfer 100 STRK to Bob."
2. Relayer submits outer transaction T1 that calls a relayer contract, which invokes the `meta_tx_v0` syscall with Alice's signature. Alice's `__execute__` runs; 100 STRK is transferred.
3. Attacker observes Alice's signature in T1's calldata.
4. Attacker submits outer transaction T2 (with attacker's own nonce) that invokes the same `meta_tx_v0` syscall with the identical signature. `compute_meta_tx_v0_hash` produces the same hash; Alice's `__execute__` runs again; another 100 STRK is transferred.
5. Steps 4 repeats until Alice's account is drained. The OS never rejects the replay because `check_and_increment_nonce` is a no-op for `version=0`. [6](#0-5) [7](#0-6)
