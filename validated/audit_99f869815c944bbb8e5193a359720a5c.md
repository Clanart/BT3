### Title
`compute_meta_tx_v0_hash` Omits Nonce, Enabling Unbounded Signature Replay of Meta-Transactions - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `meta_tx_v0` syscall computes a transaction hash that does not include a nonce or any replay-preventing counter. Combined with the OS explicitly skipping nonce enforcement for version-0 transactions, any valid `meta_tx_v0` signature can be replayed an unlimited number of times by any unprivileged actor, causing repeated unauthorized execution of the target contract's `__execute__` entry point and direct loss of user funds.

---

### Finding Description

`compute_meta_tx_v0_hash` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        ...
        additional_data_size=0,
        additional_data=cast(0, felt*),   // ← no nonce
    );
    return tx_hash;
}
```

The resulting hash commits only to: `(INVOKE_HASH_PREFIX, 0, contract_address, selector, calldata_hash, 0, chain_id)`. No nonce, no block number, no counter of any kind is included. [1](#0-0) 

The OS then constructs the meta-tx `TxInfo` with `nonce=0` and `version=0`:

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    nonce=0,
    ...
);
``` [2](#0-1) 

`check_and_increment_nonce` explicitly skips all version-0 transactions:

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
``` [3](#0-2) 

Contrast this with `compute_l1_handler_transaction_hash`, which correctly passes the nonce as `additional_data`:

```cairo
let (transaction_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
    ...
    additional_data_size=1,
    additional_data=&nonce,   // ← nonce included
);
``` [4](#0-3) 

The `meta_tx_v0` syscall is reachable from any contract execution via the syscall dispatcher: [5](#0-4)

### Impact Explanation

**Critical — Direct loss of funds.**

The `meta_tx_v0` syscall is designed to let a relayer submit a user-signed meta-transaction that calls the user's account contract's `__execute__` entry point. The account contract's `__execute__` receives `tx_info.transaction_hash` and `tx_info.signature` and is expected to verify the signature against the hash. Because the hash is nonce-free, the same `(calldata, signature)` pair produces the identical hash on every invocation.

An attacker who observes a valid meta_tx_v0 signature on-chain (from the calldata of any outer v3 transaction) can replay it by submitting a new outer v3 transaction (using their own nonce) that calls any contract which invokes `meta_tx_v0` with the captured signature. The OS will compute the same hash, the target account's `__execute__` will accept the same signature, and the operation (e.g., a token transfer) will execute again. This can be repeated indefinitely, draining the victim's account.

---

### Likelihood Explanation

**High.** The `meta_tx_v0` syscall is a first-class protocol feature callable by any unprivileged contract. Signatures are visible in transaction calldata on-chain. No special access or key compromise is required — the attacker only needs to observe a single successful meta_tx_v0 invocation and submit a new outer transaction. The OS provides no nonce enforcement at the protocol level for version-0 transactions, so the only defense would be a custom replay-guard inside the target account contract, which the OS does not mandate or verify.

---

### Recommendation

Include the target contract's current nonce in the `compute_meta_tx_v0_hash` computation, passing it as `additional_data`, mirroring the pattern used by `compute_l1_handler_transaction_hash`. Additionally, the OS should read and increment the target contract's nonce after a successful `meta_tx_v0` execution, rather than unconditionally skipping nonce handling for all version-0 transactions.

```cairo
// Proposed fix in compute_meta_tx_v0_hash:
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
    nonce: felt,          // ← add nonce parameter
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=1,
        additional_data=&nonce,   // ← commit to nonce
    );
    return tx_hash;
}
```

And in `execute_meta_tx_v0`, read the target contract's current nonce from `contract_state_changes`, pass it to the hash function, and increment it afterward.

### Proof of Concept

1. **Setup**: User account at address `A` holds 1000 tokens. A relayer contract `R` accepts meta_tx_v0 requests.

2. **Victim action**: User signs a meta_tx_v0 message authorizing a transfer of 100 tokens to address `B`. The hash is:
   ```
   H = Pedersen(invoke, 0, A, __execute__, H(calldata), 0, chain_id)
   ```
   User submits a v3 transaction calling `R` with `(signature, calldata)`. `R` calls `meta_tx_v0`. The OS computes `H`, calls `A.__execute__` with `tx_info.transaction_hash=H` and `tx_info.signature=sig`. Account `A` verifies `sig` over `H` and transfers 100 tokens to `B`.

3. **Attacker replay**: Attacker extracts `(signature, calldata)` from the v3 transaction's calldata. Attacker submits a new v3 transaction (with their own nonce) calling any contract that invokes `meta_tx_v0(contract_address=A, selector=__execute__, calldata=calldata, signature=signature)`. The OS recomputes the identical `H` (no nonce in hash), calls `A.__execute__` again with the same `sig`. Account `A` accepts the signature and transfers another 100 tokens to `B`.

4. **Repeat**: Step 3 can be repeated until `A`'s balance is exhausted. Each replay costs only the attacker's gas for the outer v3 transaction.
