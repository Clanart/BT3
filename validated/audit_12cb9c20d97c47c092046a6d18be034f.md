### Title
Stale `key_to_index` Cache After Leaf Move in `delete` Corrupts DataLayer Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary
`BlockStatusCache::move_index` only marks the source slot as free but never updates `key_to_index` or `leaf_hash_to_index`. When `MerkleBlob::delete` moves the surviving sibling leaf to `TreeIndex(0)` after removing the second-to-last leaf, the cache permanently maps that leaf's key to the old (now-free) index. Subsequent `upsert` or `delete` calls on the surviving leaf read and write the wrong blob slot, producing a stale Merkle root that diverges from the actual tree state.

### Finding Description

`BlockStatusCache::move_index` is defined as:

```rust
fn move_index(&mut self, source: TreeIndex, destination: TreeIndex) -> Result<(), Error> {
    // to be called _after_ having written to the destination index
    if self.free_indexes.contains(&source) {
        return Err(Error::MoveSourceIndexNotInUse(source));
    }
    if self.free_indexes.contains(&destination) {
        return Err(Error::MoveDestinationIndexNotInUse(destination));
    }
    self.free_indexes.insert(source);   // ← only marks source free
    Ok(())
}
``` [1](#0-0) 

It never calls `key_to_index.insert(destination, ...)` or `leaf_hash_to_index.insert(destination, ...)`. The only caller is the no-grandparent branch of `delete`:

```rust
let Some(grandparent_index) = parent.parent.0 else {
    sibling_block.node.set_parent(Parent(None));
    let destination = TreeIndex(0);
    // ... update children's parent pointers ...
    self.insert_entry_to_blob(destination, &sibling_block)?;   // blob updated
    self.block_status_cache
        .move_index(sibling_index, destination)?;              // cache NOT updated
    return Ok(());
};
``` [2](#0-1) 

This branch fires whenever the deleted leaf's parent is the tree root (i.e., the tree had exactly two leaves). After the call:

| State | Expected | Actual |
|---|---|---|
| `key_to_index[sibling.key]` | `TreeIndex(0)` | `sibling_index` (old, now free) |
| `free_indexes` | `{leaf_index, sibling_index}` | `{leaf_index, sibling_index}` ✓ |
| Blob at `TreeIndex(0)` | sibling data | sibling data ✓ |
| Blob at `sibling_index` | stale | stale |

`check_integrity` explicitly asserts this invariant and would detect the violation:

```rust
let cached_index = self.block_status_cache
    .get_index_by_key(node.key)
    .ok_or(Error::IntegrityKeyNotInCache(node.key))?;
if *cached_index != index {
    return Err(Error::IntegrityKeyToIndexCacheIndex(...));
}
``` [3](#0-2) 

**Concrete corruption scenario (3 steps):**

1. Tree has two leaves: `leaf_A` at `TreeIndex(1)`, `leaf_B` at `TreeIndex(2)`, root at `TreeIndex(0)`.
2. `delete(leaf_A.key)` → `leaf_B` is moved to `TreeIndex(0)` in the blob, but `key_to_index[leaf_B.key]` remains `TreeIndex(2)`.
3. `upsert(leaf_B.key, new_value, new_hash)` → `get_leaf_by_key` resolves to `TreeIndex(2)` (stale slot), writes the updated leaf there, and calls `mark_lineage_as_dirty` on the stale parent pointer. The actual root at `TreeIndex(0)` retains the old hash. Every subsequent `get_root()` returns the pre-update hash. [4](#0-3) 

### Impact Explanation

The DataLayer Merkle root diverges from the committed key-value state. A node serving proofs will generate inclusion proofs against the stale root, which will verify against the stale root but not against the correct one (or vice versa). This satisfies the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state**.

### Likelihood Explanation

The trigger is a routine two-step DataLayer operation: delete the first of two leaves, then upsert the remaining one. No special privileges, keys, or network conditions are required. Any DataLayer store that shrinks to one leaf and then updates it will hit this path.

### Recommendation

`move_index` must update `key_to_index` and `leaf_hash_to_index` when the moved node is a leaf. The simplest fix is to pass the node type (or the full `Block`) into `move_index` so it can remap the cache entries:

```rust
fn move_index(&mut self, source: TreeIndex, destination: TreeIndex, node: &Node) -> Result<(), Error> {
    // existing free-index checks ...
    self.free_indexes.insert(source);
    if let Node::Leaf(leaf) = node {
        self.key_to_index.insert(leaf.key, destination);
        self.leaf_hash_to_index.insert(leaf.hash, destination);
    }
    Ok(())
}
```

Alternatively, call `remove_leaf` + `add_leaf` on the cache around the move, mirroring the pattern used in `upsert`.

### Proof of Concept

```rust
// Pseudocode exercising the bug
let mut blob = MerkleBlob::new(vec![]).unwrap();

// Insert two leaves → 2-leaf tree

// Delete leaf A → sibling (leaf B) moved to TreeIndex(0), cache stale
