### Title
Stale `feemarketParams` Captured at Construction Time Causes Wrong Base Fee in Cosmos `DynamicFeeChecker` — (`ante/evm/fee_checker.go`, `evmd/ante/handler_options.go`)

---

### Summary

`newCosmosAnteHandler` captures `feemarketParams` (including `BaseFee`) once at app-startup time and passes a pointer to the stale snapshot into `NewDynamicFeeChecker`. Because the base fee changes every block, all subsequent Cosmos transactions validated by this checker use the wrong base fee — the value frozen at startup — rather than the live on-chain value. This is the direct analog of the external report's bug: a stale/approximate value is substituted for the true current value in a critical fee calculation.

---

### Finding Description

`newCosmosAnteHandler` is called once during application initialization from `setAnteHandler`:


The live base fee is stored in the KVStore via `SetBaseFee` → `SetParams`, but `NewDynamicFeeChecker` never reads from the store; it reads from the frozen pointer.

**Contrast with the Ethereum ante handler**, which correctly fetches fresh params on every invocation:

---

### Impact Explanation

**High — ante handler fee market bug that permits invalid transactions to commit with mis-accounted fees.**

When the live base fee rises above the startup-time snapshot (the normal case in a busy network, since `CalculateBaseFee` increases the fee when blocks are above the gas target), the `DynamicFeeChecker` computes `effectivePrice` using the lower stale base fee. The deducted fee is therefore less than what EIP-1559 requires. Cosmos transactions (e.g., IBC relays, governance votes, staking operations) submitted with `ExtensionOptionDynamicFeeTx` pass the ante handler and are committed to state while paying below the correct minimum fee. This is a direct fee underpayment / fee bypass for all Cosmos-native transactions on chains with `DynamicFeeChecker: true`.

Conversely, if the base fee has dropped since startup, legitimate transactions may be incorrectly rejected, causing a liveness issue.

---

### Likelihood Explanation

- `DynamicFeeChecker: true` is set in the production app (`evmd/app.go`).
- The base fee changes every block once EIP-1559 is enabled (`EnableHeight` reached, `NoBaseFee = false`).
- Any Cosmos transaction submitted after the base fee has diverged from its startup value triggers the bug.
- No special privileges are required; any user submitting a Cosmos transaction with `ExtensionOptionDynamicFeeTx` is affected.

---

### Recommendation

Replace the stale-pointer pattern in `newCosmosAnteHandler` with a live-reading fee checker that queries the keeper on every invocation, mirroring `newEthAnteHandler`:

```go
// evmd/ante/handler_options.go
func newCosmosAnteHandler(ctx sdk.Context, options HandlerOptions, ...) sdk.AnteHandler {
    // Do NOT snapshot feemarketParams here.
    // Pass the keeper so the checker can read live params per-call.
    var txFeeChecker ante.TxFeeChecker
    if options.DynamicFeeChecker {
        txFeeChecker = evm.NewDynamicFeeCheckerFromKeeper(
            options.EvmKeeper, options.FeeMarketKeeper)
    }
    ...
}
```

`NewDynamicFeeCheckerFromKeeper` should call `feeMarketKeeper.GetParams(ctx)` and `evmKeeper.GetParams(ctx)` inside the returned closure, not outside it, so each transaction sees the current block's base fee.

---

### Proof of Concept

1. Chain starts with `BaseFee = 1_000_000_000` (1 Gwei). `newCosmosAnteHandler` captures this value.
2. The chain runs for many blocks with high load. `CalculateBaseFee` raises the live base fee to `2_000_000_000` (2 Gwei) and writes it to the KVStore.
3. A user submits a Cosmos transaction with `ExtensionOptionDynamicFeeTx{MaxPriorityPrice: 0}` and `fee = 1_000_000_000 * gasLimit` (sufficient for the stale base fee, insufficient for the live base fee).
4. `NewDynamicFeeChecker` reads `feemarketParams.BaseFee = 1_000_000_000` (stale), computes `effectivePrice = 1_000_000_000`, passes the `feeCap >= baseFeeInt` check, and returns `effectiveFee = 1_000_000_000 * gasLimit`.
5. The transaction is committed. The user paid half the required fee. The correct check (using the live `2_000_000_000` base fee) would have rejected it. [7](#0-6) [8](#0-7) [4](#0-3) [9](#0-8)

```go
func NewDynamicFeeChecker(ethCfg *params.ChainConfig, evmParams *types.Params, feemarketParams *feemarkettypes.Params) authante.TxFeeChecker {
	return func(ctx sdk.Context, tx sdk.Tx) (sdk.Coins, int64, error) {
		feeTx, ok := tx.(sdk.FeeTx)
		if !ok {
			return nil, 0, fmt.Errorf("tx must be a FeeTx")
		}

		if ctx.BlockHeight() == 0 {
			// genesis transactions: fallback to min-gas-price logic
			return checkTxFeeWithValidatorMinGasPrices(ctx, feeTx)
		}

		denom := evmParams.EvmDenom

		baseFee := types.GetBaseFee(ctx.BlockHeight(), ethCfg, feemarketParams)
		if baseFee == nil {
			// london hardfork is not enabled: fallback to min-gas-prices logic
			return checkTxFeeWithValidatorMinGasPrices(ctx, feeTx)
		}

		// default to `MaxInt64` when there's no extension option.
		maxPriorityPrice := sdkmath.NewInt(math.MaxInt64)

		// get the priority tip cap from the extension option.
		if hasExtOptsTx, ok := tx.(authante.HasExtensionOptionsTx); ok {
			for _, opt := range hasExtOptsTx.GetExtensionOptions() {
				if extOpt, ok := opt.GetCachedValue().(*ethermint.ExtensionOptionDynamicFeeTx); ok {
					maxPriorityPrice = extOpt.MaxPriorityPrice
					break
				}
			}
		}

		if maxPriorityPrice.Sign() == -1 {
			return nil, 0, errorsmod.Wrapf(errortypes.ErrInsufficientFee, "priority fee is negative")
		}

		gas := feeTx.GetGas()
		feeCoins := feeTx.GetFee()
		fee := feeCoins.AmountOf(denom)

		feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))
		baseFeeInt := sdkmath.NewIntFromBigInt(baseFee)

		if feeCap.LT(baseFeeInt) {
			return nil, 0, errorsmod.Wrapf(errortypes.ErrInsufficientFee, "insufficient gas prices; got: %s required: %s", feeCap, baseFeeInt)
		}

		// calculate the effective gas price using the EIP-1559 logic.
		effectivePrice := sdkmath.NewIntFromBigInt(types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()))

		// NOTE: create a new coins slice without having to validate the denom
		effectiveFee := sdk.Coins{
			{
				Denom:  denom,
				Amount: effectivePrice.Mul(sdkmath.NewIntFromUint64(gas)),
			},
		}
```
