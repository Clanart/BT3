### Title
P2P Sync Stores Unvalidated Peer-Supplied `state_root` in Headers Table, Causing RPC `get_state_update` to Serve Wrong Global Root — (`crates/apollo_p2p_sync/src/client/header.rs`, `crates/apollo_storage/src/header.rs`, `crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

`HeaderStreamBuilder::parse_data_for_block` accepts any `state_root` value from a p2p peer without cryptographic or trie-commitment validation. The value flows through `write_to_storage` → `append_header` into the `headers` table verbatim. Every subsequent read of that header — including the RPC `get_state_update` response (`new_root`) and the `read_pending_data` fallback (`old_root`) — reflects the attacker-supplied value. The question's further claim that this corrupts the sequencer's own `calculate_block_hash` is **incorrect**: the commitment manager derives `global_root` from `CommitBlockResponse` (the actual Patricia trie output), not from the stored header field.

---

### Finding Description

**Entrypoint — `parse_data_for_block`:**

`HeaderStreamBuilder::parse_data_for_block` performs exactly two checks on a peer-supplied `SignedBlockHeader`:

1. Block number ordering.
2. Signature vector length equals `ALLOWED_SIGNATURES_LENGTH`. [1](#0-0) 

There is no recomputation of `block_hash` from the header fields, no cryptographic verification of the signature against `block_hash` (which would indirectly bind `state_root`), and no comparison of `state_root` against any Patricia trie commitment.

**Storage write — `write_to_storage` / `append_header`:**

`write_to_storage` passes the peer-supplied `SignedBlockHeader` directly to `append_header`: [2](#0-1) 

`append_header` copies `state_root` from the `BlockHeader` into `StorageBlockHeader` and persists it without any guard: [3](#0-2) 

**RPC exposure — `get_state_update` (`new_root`):**

For accepted blocks, `get_state_update` reads the stored header and returns `header.new_root` (which is `state_root`) as the authoritative `new_root` field: [4](#0-3) 

**RPC exposure — `read_pending_data` (`old_root`):**

When the cached pending block's parent hash does not match the latest stored block hash, `read_pending_data` synthesises a fallback `PendingData` whose `old_root` is taken directly from the stored header's `state_root`: [5](#0-4) 

Both paths serve the attacker-controlled value to any RPC caller.

**Why the `calculate_block_hash` claim is wrong:**

The sequencer's commitment manager calls `calculate_block_hash` with `global_root` sourced from `CommitBlockResponse` — the output of the actual Patricia trie commitment computation — not from the stored header field: [6](#0-5) 

The stored `state_root` therefore does **not** corrupt the sequencer's own block hash or proof commitment.

---

### Impact Explanation

**Impact: High** — RPC `get_state_update` returns an authoritative-looking wrong `new_root` (for accepted blocks) and wrong `old_root` (for pending blocks) to any caller. Clients relying on these values for state verification, proof construction, or cross-chain bridging receive a fabricated global root. The sequencer's own block production and proof commitment are unaffected because `calculate_block_hash` uses the trie-derived `global_root`, not the stored header field.

---

### Likelihood Explanation

Any node participating in the p2p network can send a `SignedBlockHeader` with an arbitrary `state_root`. No operator privilege is required. The syncing node accepts and persists the value unconditionally. The corrupted value is immediately observable via the public RPC endpoint.

---

### Recommendation

1. **Verify `block_hash` before storage.** Recompute `block_hash` from the received header fields (including `state_root`) using `calculate_block_hash` and reject headers where the recomputed hash does not match the peer-supplied `block_hash`.
2. **Verify the signature cryptographically.** Call `verify_block_signature` (which signs over `block_hash` and `state_diff_commitment`) against the known sequencer public key before accepting any header. This binds `state_root` to the signed `block_hash`.
3. **Cross-check against L1.** For finalized blocks, validate the stored `block_hash` against the L1 state update to detect any peer that supplies a self-consistent but fabricated `(state_root, block_hash)` pair.

### Proof of Concept

```rust
// In a Rust integration test (mirrors the production p2p write path):
use apollo_storage::header::{HeaderStorageReader, HeaderStorageWriter};
use starknet_api::block::{BlockHeader, BlockHeaderWithoutHash, BlockNumber};
use starknet_api::core::GlobalRoot;
use starknet_types_core::felt::Felt;

#[tokio::test]
async fn p2p_peer_controls_stored_state_root() {
    let ((reader, mut writer), _tmp) = get_test_storage();

    // Attacker-controlled state_root — no validation prevents this.
    let attacker_root = GlobalRoot(Felt::from(0xdeadbeef_u64));

    writer
        .begin_rw_txn().unwrap()
        .append_header(BlockNumber(0), &BlockHeader {
            block_header_without_hash: BlockHeaderWithoutHash {
                state_root: attacker_root,
                ..Default::default()
            },
            ..Default::default()
        }).unwrap()
        .commit().unwrap();

    let stored = reader
        .begin_ro_txn().unwrap()
        .get_block_header(BlockNumber(0)).unwrap().unwrap();

    // Passes: attacker value is stored and returned verbatim.
    assert_eq!(stored.block_header_without_hash.state_root, attacker_root);
    // RPC get_state_update would return new_root = 0xdeadbeef.
    // read_pending_data fallback would return old_root = 0xdeadbeef.
}
```
