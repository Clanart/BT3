### Title
Missing Normalization-Zero Guard in `init_transfer` Permanently Locks User Funds - (`near/omni-bridge/src/lib.rs`)

### Summary

`init_transfer` accepts and locks user tokens without verifying that `normalize_amount(amount - fee, decimals) > 0`. The zero-check only occurs later in `sign_transfer`. For tokens where `origin_decimals > decimals`, any transfer where `amount - fee < 10^(origin_decimals - decimals)` will have its normalized amount floor-divided to zero, causing `sign_transfer` to permanently revert with `InvalidAmountToTransfer`. Because there is no cancel/refund path for pending transfers, the user's funds are irrecoverably locked in the NEAR bridge contract.

### Finding Description

`normalize_amount` performs floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

`sign_transfer` calls this and then enforces a non-zero guard:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

However, `init_transfer` only checks `fee < amount`:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [3](#0-2) 

It does **not** check that `normalize_amount(amount - fee, decimals) > 0`. Tokens are locked into the bridge at `init_transfer` time (via `ft_on_transfer` / NEP-141 transfer), before `sign_transfer` is ever called.

Similarly, `update_transfer_fee` only validates `fee.fee < transfer.message.amount`: [4](#0-3) 

A user can raise their own fee to `amount - 1`, making `amount_without_fee() = 1`, which normalizes to 0 for any token with `origin_decimals > decimals`.

Once `sign_transfer` permanently reverts, the transfer entry stays in `pending_transfers` forever. `remove_transfer_message` is only called from `sign_transfer_callback` (on success) and `claim_fee_callback` (which requires a `FinTransfer` proof from the destination chain — impossible since the transfer was never signed and finalized): [5](#0-4) 

There is no `cancel_transfer` or emergency-refund function. The funds are permanently locked.

### Impact Explanation

**Critical — Permanent freezing of user funds in the NEAR bridge contract.**

For any token registered with `origin_decimals > decimals` (e.g., a 24-decimal origin token normalized to 18 decimals on NEAR, giving `diff = 6`), any pending transfer where `amount - fee < 1_000_000` (i.e., less than `10^6` base units) will be permanently unclaimable. The user's tokens are held by the bridge contract with no recovery path short of a DAO contract upgrade.

### Likelihood Explanation

**Moderate.** The condition is reachable by any unprivileged user who:
1. Bridges a token with `origin_decimals > decimals` (the standard normalization case for high-precision tokens), AND
2. Creates a transfer with a small `amount - fee` (below the normalization threshold).

This can happen accidentally (user sends a dust amount) or deliberately (user raises their own fee via `update_transfer_fee` to `amount - 1`). No privileged role is required. The `ft_on_transfer` entry point is fully public.

### Recommendation

Add a normalization-zero guard at `init_transfer` time, after the token decimals are known:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

Apply the same guard in `update_transfer_fee` when the token fee is changed. This mirrors the existing check in `sign_transfer` and ensures the invariant is enforced before funds are locked.

### Proof of Concept

Assume a token registered with `origin_decimals = 24`, `decimals = 18` (diff = 6, so normalization divides by `1_000_000`).

1. User calls `ft_on_transfer` with `amount = 5`, `fee = 0`. The check `fee < amount` passes (0 < 5). Tokens are locked in the bridge.
2. A relayer calls `sign_transfer` for this transfer.
3. `normalize_amount(5 - 0, Decimals{18, 24}) = 5 / 1_000_000 = 0`.
4. `require!(amount_to_transfer > 0, ...)` panics with `InvalidAmountToTransfer`.
5. `sign_transfer` permanently reverts on every call.
6. No MPC signature is ever produced; the destination chain never receives a finalization payload.
7. `claim_fee` cannot be called (no `FinTransfer` proof exists).
8. The user's 5 tokens are permanently locked in the NEAR bridge contract.

Alternatively, with `amount = 2_000_000` and `fee = 0`:
- User calls `update_transfer_fee` setting `fee = 1_999_999` (valid: `fee < amount`).
- `amount_without_fee() = 1`, `normalize_amount(1) = 0`.
- Same permanent lock outcome.

    fn remove_transfer_message(&mut self, transfer_id: TransferId) -> TransferMessage {
        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);
        }

        transfer.message
    }
