### Title
`max_l2_gas_amount` Bound Not Enforced for `Declare` Transactions in Gateway Admission - (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` enforces a `max_l2_gas_amount` upper bound on `l2_gas.max_amount` for Invoke and DeployAccount transactions, but explicitly skips this check for Declare transactions via an empty branch. This is the direct sequencer analog of the Ajna `limitIndex_` bug: a configured bound parameter is accepted by the validation function but not enforced in a specific code path, allowing Declare transactions with arbitrarily large `l2_gas.max_amount` values to bypass gateway admission control and enter the mempool.

### Finding Description

In `validate_resource_bounds`, the check for `l2_gas.max_amount > max_l2_gas_amount` is guarded by a type check that explicitly does nothing for Declare transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
``` [1](#0-0) 

The empty `if let RpcTransaction::Declare(_) = tx { }` branch is the structural analog of the Ajna `repayDebt()` bug: the `limitIndex_` / `max_l2_gas_amount` bound is accepted as a configured parameter but the enforcement check is silently skipped for a specific operation type (pure repayment / Declare transaction). The `max_l2_gas_amount` config field is read and used for Invoke and DeployAccount, but the Declare branch falls through without any check.

The test `valid_l2_gas_amount_on_declare` explicitly confirms this behavior — a Declare transaction with `max_amount: GasAmount(200)` passes when `max_l2_gas_amount: 100`: [2](#0-1) 

The full `validate` dispatch in `StatelessTransactionValidator` shows that `validate_resource_bounds` is called for all transaction types, but the Declare path silently bypasses the upper-bound check: [3](#0-2) 

No downstream check in `perform_pre_validation_stage` or `check_fee_bounds` in the blockifier enforces an upper bound on `max_amount` — those checks only verify that `max_amount` is not too *low* relative to the minimal gas estimate: [4](#0-3) 

### Impact Explanation

The gateway's `max_l2_gas_amount` admission control is bypassed for Declare transactions. An attacker can submit Declare transactions with `l2_gas.max_amount` set to an arbitrarily large value (e.g., `u64::MAX`) and they will be accepted by the gateway and enter the mempool. No downstream blockifier check rejects them on the basis of an excessive upper bound. This fits the **High** impact: *Mempool/gateway/RPC admission accepts invalid transactions before sequencing*.

### Likelihood Explanation

High — no special privileges or conditions are required. Any user can submit a Declare transaction with `l2_gas.max_amount` exceeding `max_l2_gas_amount`. The behavior is confirmed by an explicit passing test. The TODO comment acknowledges the gap has not been addressed.

### Recommendation

Remove the empty `if let RpcTransaction::Declare(_) = tx { }` branch and apply the `max_l2_gas_amount` check uniformly to all transaction types, including Declare:

```rust
// Remove the empty Declare branch:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```
