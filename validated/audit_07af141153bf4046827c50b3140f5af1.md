### Title
Integer Wrapping in Relative Timelock Arithmetic Enables Timelock Bypass — (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary
When the `nowrap` flag is `false`, the relative timelock checks in `check_time_locks` use `wrapping_add` for `u32`/`u64` arithmetic. A CLVM spend supplying `ASSERT_HEIGHT_RELATIVE(0xFFFFFFFF)` or `ASSERT_SECONDS_RELATIVE(0xFFFFFFFFFFFFFFFF)` causes the computed deadline to wrap around to a value smaller than the coin's birth height/timestamp, making the timelock trivially satisfiable at any current height/time. This is the structural analog to the Solidity underflow: a boundary arithmetic operation produces a result on the wrong side of zero, inverting the intended guard.

### Finding Description
In `check_time_locks`, two branches handle relative timelocks without the `nowrap` guard:

```rust
// height_relative, nowrap=false path
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

```rust
// seconds_relative, nowrap=false path
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [1](#0-0) [2](#0-1) 

When `height_relative = 0xFFFF_FFFF` and `confirmed_block_index = 100`:

- `100u32.wrapping_add(0xFFFF_FFFF)` = `99` (wraps to a value *below* the coin's birth height)
- The guard `prev_height < 99` is `false` at any height ≥ 99 → **no error returned → timelock passes**

The intended semantics are that `height_relative = u32::MAX` should require ~4.3 billion blocks of waiting. With wrapping, the coin becomes spendable at height ≥ `confirmed_block_index − 1`.

The same overflow applies to `seconds_relative = u64::MAX`: `timestamp.wrapping_add(u64::MAX)` wraps to `timestamp − 1`, making the seconds-relative timelock satisfiable immediately.

The `nowrap=true` path correctly uses `saturating_add`, which clamps to `u32::MAX`/`u64::MAX` and keeps the guard meaningful. The `nowrap=false` path is preserved for pre-softfork block validation. [3](#0-2) [4](#0-3) 

The test suite explicitly documents this divergence:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),
)]
``` [5](#0-4) 

The `Ok(())` result for `nowrap=false` confirms the bypass is reachable in that code path.

### Impact Explanation
This matches the **High** allowed impact: *"timelock validation bypass enables unauthorized spend acceptance."* A coin locked with `ASSERT_HEIGHT_RELATIVE(0xFFFFFFFF)` or `ASSERT_SECONDS_RELATIVE(0xFFFFFFFFFFFFFFFF)` — intended to be unspendable for billions of blocks/seconds — can be spent immediately when processed through the `nowrap=false` branch. Any spend bundle accepted under this path would be consensus-valid from the node's perspective, allowing premature coin spending.

### Likelihood Explanation
Exploitability depends on whether the `nowrap=false` branch is reachable for attacker-controlled spend bundles in the current production call path. The `nowrap` flag is derived from consensus flags passed to `check_time_locks` via `spendbundle_conditions` and `wheel/src/api.rs`. If the softfork activating `nowrap=true` has fully activated for all new transaction blocks, new spends are protected. However, if any production validation path (e.g., mempool pre-validation, pre-softfork block replay, or Python API callers passing flags without `NOWRAP_TIMELOCKS`) still invokes `check_time_locks` with `nowrap=false`, the bypass is directly exploitable by any unprivileged user submitting a crafted spend bundle. The test infrastructure confirms both paths are live in the codebase. [3](#0-2) 

### Recommendation
Replace `wrapping_add` with `saturating_add` unconditionally in both the `height_relative` and `seconds_relative` branches, removing the `nowrap` conditional entirely for these checks. The `nowrap=false` / `wrapping_add` behavior produces semantically incorrect timelock enforcement and should not be preserved for any block era. If backward compatibility with historical blocks is required, document and gate it behind a specific block-height range check rather than a caller-supplied flag.

### Proof of Concept
A CLVM puzzle that outputs `(ASSERT_HEIGHT_RELATIVE . 0xFFFFFFFF)` when solved. When the coin is confirmed at height `H`, the wrapped deadline is `H + 0xFFFFFFFF mod 2^32 = H - 1`. Any block at height ≥ `H - 1` satisfies `prev_height >= H - 1`, so `prev_height < (H - 1)` is false, the error is not raised, and the spend is accepted — bypassing a ~4.3-billion-block timelock.

```
; CLVM puzzle
(mod () (list (list 82 4294967295)))  ; 82 = ASSERT_HEIGHT_RELATIVE, arg = 0xFFFFFFFF
```

When `nowrap=false`, `check_time_locks` returns `Ok(())` for this spend at any height ≥ `confirmed_block_index − 1`, as confirmed by the `height_relative_wrap` test case. [5](#0-4) [6](#0-5)
