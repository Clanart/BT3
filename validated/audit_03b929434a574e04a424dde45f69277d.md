### Title
CEI Violation in `initTransfer1155` Allows Nonce Hijacking via ERC1155 Callback Reentrancy, Enabling Unbacked Token Minting on NEAR - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`initTransfer1155` increments `currentOriginNonce` before an external `IERC1155.safeTransferFrom` call but reads `currentOriginNonce` again **after** that call for both `initTransferExtension` and the `InitTransfer` event emission. A malicious ERC1155 contract can reenter `initTransfer1155` during `safeTransferFrom`, increment the nonce a second time, and cause the outer call to emit with the wrong (already-used) nonce. Because ERC1155 `safeTransferFrom` always invokes a callback on the recipient, this reentrancy path is unconditionally available to any attacker-controlled ERC1155 token — no special token standard (ERC777, etc.) is required.

---

### Finding Description

In `initTransfer1155`:

```solidity
function initTransfer1155(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;          // (A) nonce becomes N
    ...
    IERC1155(tokenAddress).safeTransferFrom(   // (B) external call — attacker-controlled
        msg.sender, address(this), tokenId, amount, ""
    );
    ...
    initTransferExtension(
        msg.sender, deterministicToken,
        currentOriginNonce,            // (C) reads currentOriginNonce AFTER external call
        ...
    );
    emit BridgeTypes.InitTransfer(
        ..., currentOriginNonce, ...   // (D) same post-call read
    );
}
```

The malicious ERC1155 contract's `safeTransferFrom` implementation reenters `initTransfer1155` before returning:

| Step | `currentOriginNonce` | Action |
|------|----------------------|--------|
| Outer call enters, line (A) | N | nonce incremented |
| Outer call reaches line (B) | N | external call begins |
| **Reentrant inner call enters, line (A)** | **N+1** | nonce incremented again |
| Inner call reaches line (B) | N+1 | malicious ERC1155 returns (no further reentry) |
| Inner call reaches lines (C)/(D) | N+1 | emits `InitTransfer` with nonce **N+1** ✓ |
| Outer call resumes at lines (C)/(D) | N+1 | emits `InitTransfer` with nonce **N+1** again ✗ |

Nonce N is **never emitted**. Nonce N+1 is emitted **twice**. The NEAR bridge processes the first N+1 event and rejects the second as a duplicate.

The malicious ERC1155 does not need to actually transfer tokens — it only needs to avoid reverting. The bridge has no way to verify that tokens were genuinely transferred before emitting the event.

---

### Impact Explanation

**Critical — Unauthorized minting of bridged assets / unbacked supply.**

Attack path:

1. Attacker deploys a malicious ERC1155 contract whose `safeTransferFrom` reenters `initTransfer1155` and does not actually update balances.
2. Attacker calls `logMetadata1155(maliciousERC1155, tokenId)` — this function is permissionless and emits a `LogMetadata` event that causes the NEAR bridge to deploy a bridged token for the ERC1155.
3. Once the bridged token is deployed on NEAR, attacker calls `initTransfer1155(maliciousERC1155, tokenId, amount, ...)`.
4. The reentrant inner call emits a valid `InitTransfer(nonce=N+1, amount, recipient=attacker_near_address)` without locking any real ERC1155 tokens.
5. The NEAR bridge's `fin_transfer_callback` processes the N+1 event and mints bridged tokens to the attacker's NEAR address.
6. The outer call also emits N+1 — rejected as duplicate on NEAR.

Result: attacker receives bridged tokens on NEAR backed by zero locked ERC1155 tokens on EVM. This breaks bridge collateralization and constitutes direct unauthorized minting of bridged assets.

---

### Likelihood Explanation

**Medium.** The attacker must:
- Deploy a malicious ERC1155 contract (trivial).
- Register it via `logMetadata1155` (permissionless, no access control).
- Wait for the NEAR MPC to sign and process the `LogMetadata` event (normal bridge operation; MPC has no reason to reject a well-formed event).
- Execute the reentrant `initTransfer1155` call.

No privileged role, leaked key, or colluding MPC threshold is required. The ERC1155 callback path is unconditional — unlike ERC777, ERC1155 `safeTransferFrom` **always** invokes `onERC1155Received` on the recipient, making this reachable for any ERC1155 token the attacker controls.

---

### Recommendation

Cache `currentOriginNonce` into a local variable immediately after incrementing it, and use only the local variable in all subsequent reads. This ensures the emitted nonce is the one assigned at the start of the call, regardless of any reentrant modifications:

```solidity
function initTransfer1155(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    uint64 originNonce = currentOriginNonce;  // cache before any external call
    ...
    IERC1155(tokenAddress).safeTransferFrom(msg.sender, address(this), tokenId, amount, "");
    ...
    initTransferExtension(msg.sender, deterministicToken, originNonce, ...);
    emit BridgeTypes.InitTransfer(msg.sender, deterministicToken, originNonce, ...);
}
```

Apply the same fix to `initTransfer` for consistency (relevant if ERC777 or hook-bearing ERC20 tokens are ever supported).
