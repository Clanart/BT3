### Title
Missing Class Hash Validation in `execute_replace_class` Enables Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The StarkNet OS `execute_replace_class` syscall implementation does not validate that the caller-supplied class hash corresponds to a previously declared class. An unprivileged attacker who can trigger `replace_class` on a victim contract (e.g., through a missing-access-control upgrade function) can permanently set the contract's class hash to an arbitrary, undeclared value. Because the OS accepts and proves this state transition without error, the contract becomes permanently uncallable and all funds it holds are irreversibly frozen.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads `request.class_hash` directly from the syscall request and writes it into the contract's `StateEntry` with no check that the hash corresponds to any declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing validation. The identical omission exists in the deprecated syscall path: [2](#0-1) 

The post-execution compiled-class-facts validation (`validate_compiled_class_facts_post_execution`) only checks class hashes that were **executed** during the block. Because `replace_class` updates the state entry without executing any code under the new hash, the invalid hash is never checked against the declared-class set, and the proof is accepted by the L1 verifier. [3](#0-2) 

---

### Impact Explanation

Once the state transition is proven and accepted on L1, the victim contract's on-chain class hash is permanently set to the invalid value. In every subsequent block, when the OS attempts to execute any call to that contract, it cannot locate a compiled class for the stored hash. The execution fails unconditionally. Any ERC-20 tokens, ETH, or other assets held by the contract are permanently frozen with no recovery path. This satisfies the **Critical — Permanent freezing of funds** impact category.

---

### Likelihood Explanation

The `replace_class` syscall is a standard upgrade primitive. Many DeFi and protocol contracts expose a public or governance-gated upgrade function. A contract that lacks strict access control on its upgrade path (a common pattern, especially in early deployments or contracts that rely on the OS to reject invalid hashes) is directly exploitable. An attacker needs only to call the upgrade function with `new_class_hash = 0` or any arbitrary felt. No special privilege, leaked key, or operator cooperation is required beyond the ability to submit a regular invoke transaction.

---

### Recommendation

Inside `execute_replace_class` (and its deprecated counterpart), add an explicit Cairo-level assertion that the supplied class hash exists in `contract_class_changes` (i.e., was declared in the current or a prior block). For example:

```cairo
// Assert the class hash was previously declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This ensures the constraint is enforced in the proof itself, not merely in an off-chain hint, and prevents any undeclared hash from being committed to the on-chain state.

---

### Proof of Concept

1. **Deploy victim contract** with a public upgrade entry point and no access control:
   ```cairo
   @external
   func upgrade(new_class_hash: felt) {
       replace_class(new_class_hash);   // no access check
   }
   ```
2. **Users deposit funds** into the victim contract (e.g., via an ERC-20 `transfer`).
3. **Attacker submits** an invoke transaction
