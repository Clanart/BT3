### Title
Fee-on-Transfer Token-2022 Tokens Allow Unbacked Bridged-Token Minting via Solana `init_transfer` — (File: `solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

---

### Summary

The Solana bridge's `init_transfer` instruction uses `transfer_checked` to lock tokens in the vault and then posts a Wormhole message carrying `payload.amount` as the transfer amount. For SPL Token-2022 tokens with the **transfer fee extension**, `transfer_checked` withholds the fee from the transferred amount, so the vault receives `payload.amount − fee` tokens. The Wormhole message still carries the full `payload.amount`. When NEAR finalizes the transfer it mints `payload.amount` bridged tokens, creating unbacked supply and breaking bridge collateralization.

---

### Finding Description

In `init_transfer.rs`, `InitTransfer::process` calls `transfer_checked` with the user-supplied `payload.amount`:

```rust
transfer_checked(
    CpiContext::new(
        self.token_program.to_account_info(),
        TransferChecked {
            from: self.from.to_account_info(),
            to: vault.to_account_info(),          // vault receives amount − fee
            authority: self.user.to_account_info(),
            mint: self.mint.to_account_info(),
        },
    ),
    payload.amount.try_into()...,
    self.mint.decimals,
)?;
```

Immediately after, the Wormhole message is posted with the original `payload.amount`:

```rust
self.common.post_message(payload.serialize_for_near((
    self.common.sequence.sequence,
    self.user.key(),
    self.mint.key(),
))?)?;
```

`serialize_for_near` encodes `self.amount` (i.e. `payload.amount`) verbatim into the cross-chain message.

The Solana bridge uses `token_interface` (`Interface<'info, TokenInterface>`), which explicitly supports both SPL Token and Token-2022. For a Token-2022 mint with the **transfer fee extension** enabled, `transfer_checked` withholds the configured fee percentage from the transferred amount: the vault's balance increases by `payload.amount − fee`, while the Wormhole message carries `payload.amount`.

On the NEAR side, `fin_transfer` trusts the amount in the signed Wormhole payload and mints exactly `payload.amount` bridged tokens to the recipient. No reconciliation against the actual vault balance is performed. The result is `fee` unbacked bridged tokens in circulation.

The EVM `SECURITY.md` explicitly documents "Fee-on-transfer and rebasing tokens are intentionally unsupported" for the EVM `initTransfer`, but no equivalent statement exists for the Solana bridge, which additionally uses `token_interface` signalling Token-2022 compatibility.

---

### Impact Explanation

**High — Balance/accounting corruption that breaks bridge collateralization.**

After each such transfer the Solana vault holds fewer tokens than the total outstanding bridged supply on NEAR. The deficit accumulates with every fee-bearing transfer. When users attempt to bridge back from NEAR to Solana, the vault will be short; the last redeemers cannot recover their funds. The bridge's 1:1 collateralization invariant is permanently violated for the affected token.

---

### Likelihood Explanation

Token-2022 with the transfer fee extension is a live, widely-used Solana standard (e.g. USDC on Solana uses Token-2022). The Solana bridge explicitly opts into Token-2022 support via `token_interface`. Any unprivileged token holder can call `init_transfer` with such a token; no special role or key is required. The discrepancy is introduced silently — `transfer_checked` succeeds, the Wormhole message is posted, and the NEAR relayer processes it normally.

---

### Recommendation

After `transfer_checked`, read the vault's post-transfer balance and use the **actual deposited amount** (pre-transfer vault balance subtracted from post-transfer vault balance) as the amount encoded in the Wormhole message, rather than the user-supplied `payload.amount`. This is the standard "balance-before / balance-after" pattern used to handle fee-on-transfer tokens safely.

Alternatively, explicitly reject Token-2022 mints that have the transfer fee extension enabled by inspecting the mint's extension list before proceeding.

---

### Proof of Concept

1. Deploy a Token-2022 mint on Solana with a 5 % transfer fee extension.
2. Register the token with the Solana bridge (`log_metadata`), deploy the bridged token on NEAR.
3. Attacker holds 1 000 units of the token and calls `init_transfer` with `payload.amount = 1_000`.
4. `transfer_checked` executes: vault receives **950** tokens; 50 are withheld as fee.
5. Wormhole message is posted with `amount = 1_000`.
6. NEAR relayer submits `fin_transfer`; NEAR bridge mints **1 000** bridged tokens to the recipient.
7. Vault balance: 950. Bridged supply on NEAR: 1 000. Bridge is undercollateralized by 50 tokens.
8. Attacker bridges 950 bridged tokens back to Solana (vault releases 950, NEAR burns 950). Attacker retains 50 unbacked bridged tokens.
9. Any subsequent user attempting to redeem those 50 tokens finds the vault empty and their funds permanently locked.
