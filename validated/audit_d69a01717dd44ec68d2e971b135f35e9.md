### Title
`meta_tx_v0` Signature Replay — Missing Nonce in Hash Computation - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_meta_tx_v0_hash` function computes a version-0 meta-transaction hash with **no nonce** in the preimage. The OS also never calls `check_and_increment_nonce` for meta-transactions (which explicitly skips version-0 transactions). As a result, any `meta_tx_v0` signature is permanently replayable by any unprivileged party in any future transaction, with no OS-level protection.

---

### Finding Description

`compute_meta_tx_v0_hash` delegates to `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
// transaction_hash.cairo lines 295–315
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        contract_address=contract_address,
        entry_point_selector=entry_point_selector,
        calldata_size=calldata_size,
        calldata=calldata,
        max_fee=0,
        chain_id=chain_id,
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),  // ← no nonce
    );
    return tx_hash;
}
``` [1](#0-0) 

The resulting hash is a pure function of `(chain_id, contract_address, selector, calldata)` — identical for every invocation with the same arguments, across all blocks and all time.

`execute_meta_tx_v0` in `syscall_impls.cairo` then constructs a `TxInfo` with `nonce=0` (hardcoded) and calls `contract_call_helper` directly — no nonce check is performed:

```cairo
// syscall_impls.cairo lines 343–363
tempvar new_tx_info = new TxInfo(
    version=0,
    account_contract_address=contract_address,
    ...
    transaction_hash=meta_tx_hash,
    chain_id=old_tx_info.chain_id,
    nonce=0,          // ← always zero
    ...
);
``` [2](#0-1) 

The OS-level nonce guard `check_and_increment_nonce` explicitly skips version-0 transactions:

```cairo
// execute_transaction_utils.cairo lines 63–67
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
``` [3](#0-2) 

And `execute_meta_tx_v0` never calls `check_and_increment_nonce` at all — it goes straight to `contract_call_helper`: [4](#0-3) 

---

### Impact Explanation

An attacker who observes a valid `meta_tx_v0` call (signature + calldata) from a victim account can replay it in any subsequent transaction. Because the hash contains no nonce, the victim account's `__execute__` entry point receives the same `transaction_hash` and the same `signature` — and if the account contract does not independently track used hashes, the call succeeds again.

Concrete impact: if the calldata encodes a token transfer (e.g., `transfer(recipient, amount)`), the attacker replays the signed transfer an arbitrary number of times, draining the victim's balance. This constitutes **direct loss of funds (Critical)**.

---

### Likelihood Explanation

- `meta_tx_v0` is a public syscall callable by any contract from any unprivileged v3 transaction.
- The attacker only needs to observe one on-chain `meta_tx_v0` invocation (calldata + signature are visible in the transaction trace) and then submit their own v3 transaction invoking the same syscall with the captured data.
- No privileged access, leaked keys, or operator cooperation is required.
- Account contracts that support `meta_tx_v0` and rely on the OS for replay protection (as is standard for all other transaction types) are fully exposed.

---

### Recommendation

Include a nonce in the `meta_tx_v0` hash preimage, analogous to how `compute_l1_handler_transaction_hash` passes `additional_data_size=1` and `additional_data=&nonce`:

```cairo
// compute_l1_handler_transaction_hash — correct pattern
deprecated_get_transaction_hash(
    ...
    additional_data_size=1,
    additional_data=&nonce,
);
``` [5](#0-4) 

`execute_meta_tx_v0` should accept a caller-supplied nonce, include it in the hash, and enforce uniqueness via `check_and_increment_nonce` (or a dedicated used-hash registry) before executing the inner call.

---

### Proof of Concept

1. Alice signs a `meta_tx_v0` payload: `H = hash(INVOKE_PREFIX, 0, alice, __execute__, [transfer(bob, 1000)], 0, chain_id)` with signature `σ`.
2. Alice submits v3 tx (nonce=N) calling `meta_tx_v0(alice, __execute__, [transfer(bob,1000)], σ)` → 1000 tokens sent to Bob.
3. Attacker observes `σ` and the calldata on-chain.
4. Attacker submits a new v3 tx (from attacker's account, nonce=M) calling `meta_tx_v0(alice, __execute__, [transfer(bob,1000)], σ)`.
5. The OS recomputes the same `H` (no nonce → same hash), passes `(H, σ)` to `alice.__execute__`.
6. Alice's `__execute__` verifies `σ` against `H` — succeeds — and transfers 1000 more tokens to Bob.
7. Steps 4–6 repeat until Alice's balance is drained.

The root cause is entirely within the scoped OS files: `compute_meta_tx_v0_hash` in `transaction_hash/transaction_hash.cairo` (lines 295–315) and `execute_meta_tx_v0` in `execution/syscall_impls.cairo` (lines 286–395). [6](#0-5)
