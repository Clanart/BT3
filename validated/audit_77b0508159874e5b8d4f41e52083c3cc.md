### Title
ERC1155 `finTransfer` Permanently Freezes Tokens When Recipient Is a Contract Without `IERC1155Receiver` - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`OmniBridge.finTransfer` uses `IERC1155.safeTransferFrom` to deliver bridged ERC1155 tokens to the recipient. If the recipient is a contract that does not implement `IERC1155Receiver.onERC1155Received`, the delivery reverts on every attempt. Because the recipient address is embedded in the MPC-signed payload and cannot be changed, the tokens are permanently unclaimable â€” locked in the bridge with no recovery path.

### Finding Description
In `finTransfer`, when the destination token is an ERC1155 multi-token, the bridge calls:

```solidity
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this),
    payload.recipient,   // contract that may not implement IERC1155Receiver
    multiToken.tokenId,
    payload.amount,
    ""
);
```

Per the ERC1155 standard, `safeTransferFrom` calls `onERC1155Received` on the recipient if it is a contract. If the recipient does not implement `IERC1155Receiver`, the call reverts. The entire `finTransfer` transaction reverts, including the `completedTransfers[payload.destinationNonce] = true` write at line 287, so the nonce is never consumed.

However, the recipient address is part of the Borsh-encoded, MPC-signed payload. No one â€” not the user, not the relayer, not the admin â€” can substitute a different recipient and produce a valid signature. Every future call to `finTransfer` with the only valid signed payload for this transfer will revert identically. The ERC1155 tokens remain locked in the bridge contract forever. [1](#0-0) 

### Impact Explanation
**Critical â€” Permanent freezing / irrecoverable lock of user funds.**

The user's ERC1155 tokens are locked in `OmniBridge` on the EVM side with no mechanism to recover them:
- The signed payload is immutable; the recipient cannot be changed post-signing.
- `safeTransferFrom` will revert on every finalization attempt.
- There is no admin rescue function, no fallback transfer path, and no refund mechanism for ERC1155 tokens.
- The corresponding assets on the origin chain (NEAR) have already been locked/burned by `initTransfer1155`.

### Likelihood Explanation
**Medium-High.** Contract addresses are common bridge recipients: multisig wallets (Gnosis Safe), DAO treasuries, DeFi vaults, and other protocol contracts. Many of these do not implement `IERC1155Receiver`. A user who specifies any such contract as the EVM recipient of an ERC1155 bridge transfer will permanently lose their tokens. No attacker action is required â€” this is triggered by ordinary user behavior.

### Recommendation
Replace `safeTransferFrom` with `transferFrom` (the non-safe variant) in `finTransfer` for ERC1155 delivery, or add a fallback that stores undeliverable tokens in a claimable escrow keyed by `(destinationNonce, recipient)` so the recipient can later pull them once they implement the receiver interface or via an admin rescue path.

### Proof of Concept
1. User holds ERC1155 token on EVM and calls `initTransfer1155`, locking tokens in `OmniBridge`.
2. NEAR bridge processes the lock and the MPC signs a `TransferMessagePayload` with `recipient = 0xVaultContract` (a contract without `IERC1155Receiver`).
3. Relayer calls `finTransfer(signature, payload)`.
4. Execution reaches line 324: `IERC1155(...).safeTransferFrom(address(this), 0xVaultContract, ...)`.
5. `0xVaultContract` does not implement `onERC1155Received` â†’ ERC1155 standard reverts.
6. Entire transaction reverts; `completedTransfers[nonce]` is reset to `false`.
7. No alternative valid signature exists for a different recipient.
8. Tokens are permanently locked in `OmniBridge` with no recovery path. [2](#0-1) [3](#0-2)
