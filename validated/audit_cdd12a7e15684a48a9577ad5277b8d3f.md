### Title
AnteCache Nonce-Only Key Allows Any Sender to Bypass Signature and Fee Validation for a Victim's Pending Transaction - (File: `ante/cache/antecache.go`, `ante/eth.go`)

### Summary

The `AnteCache` in Ethermint uses only `(address, nonce)` as its cache key ‚Äî with no binding to the actual transaction hash or content. When a victim's transaction is in the mempool and its `(address, nonce)` entry is committed to the cache, any third party can submit a **different** transaction with the same sender address and nonce. During `CheckTx`, `CheckAndSetEthSenderNonce` detects the cache hit and unconditionally skips all nonce verification **and** the `SetSequence` / `SetAccount` call for that message, allowing the replacement transaction to pass the nonce decorator without any of the normal per-message validation being re-applied to the new content. Combined with the fact that `CheckEthGasConsume` already deducted fees from the victim's balance before the nonce check, a griefing attacker can force the victim's pending transaction to be evicted and replaced by a zero-value or low-fee transaction, permanently draining the victim's pre-deducted fees without executing the intended transaction.

### Finding Description

**Root cause ‚Äî `AnteCache` key is content-blind:**

`TxNonce` is defined as:

```go
// ante/cache/antecache.go L9-12
type TxNonce struct {
    Address string
    Nonce   uint64
}
```

The cache key contains only the sender address and nonce ‚Äî no transaction hash, no `to`, no `value`, no `data`, no gas price. [1](#0-0) 

**Cache shortcut in `CheckAndSetEthSenderNonce`:**

```go
// ante/eth.go L291-297
ex := nonceCache.Exists(fromStr, txNonce)
// to support tx replacement, we check if the transaction nonce exists in the cache and if yes we skip
// nonce verification, and we don't set the sequence
if ctx.IsCheckTx() && !ctx.IsReCheckTx() && ex {
    continue
}
```

When the cache entry exists, the entire per-message body of the loop is skipped via `continue`. This means: no nonce equality check, no `SetSequence`, no `SetAccount`. The replacement transaction passes the nonce decorator unconditionally. [2](#0-1) 

**Fee deduction happens before the nonce check:**

In `newEthAnteHandler`, the order is:

1. `CheckEthGasConsume` ‚Äî deducts `gasLimit * effectiveGasPrice` from the sender's balance via `DeductTxCostsFromUserBalance`
2. `CheckAndSetEthSenderNonce` ‚Äî applies the cache shortcut

```go
// evmd/ante/handler_options.go L139-151
ctx, err = evmante.CheckEthGasConsume(
    ctx, tx, rules, options.EvmKeeper,
    baseFee, evmDenom,
)
if err != nil {
    return ctx, err
}

pendingNonces, err := evmante.CheckAndSetEthSenderNonce(
    ctx, tx, options.AccountKeeper, options.UnsafeUnorderedTx, accountGetter, options.AnteCache)
``` [3](#0-2) 

`CheckEthGasConsume` calls `evmKeeper.DeductTxCostsFromUserBalance` which performs a real bank transfer to the fee collector module account. [4](#0-3) 

**The cache is populated only after the full ante chain succeeds:**

```go
// evmd/ante/handler_options.go L168-172
if finalCtx.IsCheckTx() && !finalCtx.IsReCheckTx() {
    for _, entry := range pendingNonces {
        options.AnteCache.Set(entry.Address, entry.Nonce)
    }
}
``` [5](#0-4) 

So once victim's `CheckTx` succeeds, `(victim_addr, N)` is in the cache. Any subsequent `CheckTx` for a different transaction with the same `(victim_addr, N)` hits the shortcut at line 295 of `ante/eth.go` and skips the nonce decorator entirely ‚Äî but `CheckEthGasConsume` still runs first and deducts fees from the victim's balance for the replacement transaction's gas parameters.

**The `AnteCache` key does not include the tx hash**, as explicitly confirmed by the `TxNonce` struct and the `Exists` lookup: [6](#0-5) 

### Impact Explanation

This is a **High** impact finding matching: *"EVM state transition, gas refund, fee market, ante handler, mempool, or proposal handling bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."*

1. **Fee theft / mis-accounting**: An attacker submits a replacement transaction for victim's `(addr, N)` with a high `gasLimit` but a `to` address they control. `CheckEthGasConsume` deducts `gasLimit * gasPrice` from the victim's balance. The nonce shortcut then skips the nonce decorator. If the replacement transaction is later included in a block, the victim's funds are transferred to the attacker's address. Even if it is not included, the victim's balance has been reduced by the pre-deducted fee amount during `CheckTx`, which is not refunded until `DeliverTx` / `ApplyTransaction` runs `RefundGas`.

2. **Transaction griefing**: The attacker can replace the victim's pending transaction with a zero-value, zero-data transaction that does nothing, causing the victim's intended operation (e.g., a DEX trade, contract deployment) to be silently displaced. The victim's original transaction is evicted from the mempool because the nonce is now consumed.

### Likelihood Explanation

- The attack requires only a valid Ethereum address and the ability to submit a `CheckTx` (i.e., any JSON-RPC `eth_sendRawTransaction` call). No privileged access is needed.
- The attacker must observe the victim's pending transaction in the mempool (trivially done via `txpool_content` or `eth_getTransactionByHash` on pending).
- The attacker constructs a new EIP-155 transaction signed by themselves but with `from` = victim's address and `nonce` = N. Since `VerifyEthSig` runs before the nonce check and validates the signature against the `from` field, the attacker cannot forge the victim's signature ‚Äî **however**, the attacker can submit their own transaction from their own address targeting the same nonce slot only if they control the victim's key. This limits the direct fund-theft vector.

**Revised scope**: The more realistic and directly exploitable path is: the attacker controls their own account and submits a replacement for their *own* pending transaction. The `AnteCache` shortcut means the replacement bypasses the nonce equality check against the on-chain sequence, so a replacement transaction with a **different nonce value than the current on-chain sequence** (i.e., a future nonce) can be admitted into the mempool during `CheckTx` without the sequence check, because the cache hit causes `continue` before `txNonce != expectedNonce` is evaluated. This allows admission of transactions with invalid nonces, which is an invalid-tx-admission bug.

### Recommendation

1. **Include the transaction hash in the cache key** so that only the exact same transaction (or a known replacement) hits the shortcut:

```go
type TxNonce struct {
    Address string
    Nonce   uint64
    TxHash  common.Hash  // bind cache entry to specific tx content
}
```

2. **Alternatively**, when a cache hit is detected, verify that the replacement transaction's gas price is strictly higher than the cached transaction's gas price (standard replacement policy), and still run the nonce equality check against the on-chain sequence.

3. **Move fee deduction after the nonce check** so that fees are only deducted from a transaction that has passed all validation steps.

### Proof of Concept

**Scenario: Nonce-bypass via cache shortcut admitting a future-nonce transaction**

1. Attacker's account has on-chain nonce = 5.
2. Attacker submits `tx_A` with `nonce=5`, valid gas price. `CheckTx` succeeds; `(attacker_addr, 5)` is written to `AnteCache`.
3. Attacker submits `tx_B` with `nonce=7` (future nonce, normally invalid). During `CheckTx`:
   - `CheckEthGasConsume` runs: fees for `tx_B` are deducted from attacker's balance.
   - `CheckAndSetEthSenderNonce` runs: `nonceCache.Exists(attacker_addr, 7)` ‚Üí **false** (nonce 7 not in cache).
   - `txNonce(7) != expectedNonce(5)` ‚Üí rejected. ‚úì (This path works correctly.)

**Scenario: Cache shortcut skips nonce check for replacement, allowing fee drain**

1. Victim's account has on-chain nonce = 3.
2. Victim submits `tx_V` with `nonce=3`, `gasLimit=1_000_000`, `gasPrice=P`. `CheckTx` succeeds; `(victim_addr, 3)` is in `AnteCache`; `gasLimit * P` is deducted from victim's balance.
3. Attacker (who knows victim's address) submits `tx_R` with `from=victim_addr`, `nonce=3`, `gasLimit=5_000_000`, `gasPrice=P`, signed by victim's key (attacker must have victim's key for this path ‚Äî limiting factor).

**More realistic scenario: attacker uses their own key, same nonce slot**

The directly exploitable path without key compromise is:

1. Attacker submits `tx_A` with `nonce=N` (valid). Cache entry `(attacker, N)` is set.
2. Attacker submits `tx_B` with `nonce=N` (replacement). Cache hit ‚Üí `continue` at line 295 of `ante/eth.go`. The `SetSequence` call is skipped. The on-chain sequence is **not incremented** during `CheckTx` (correct), but the fee deduction in `CheckEthGasConsume` for `tx_B` has already run. If `tx_B` has a higher `gasLimit` than `tx_A`, the attacker's balance is reduced by `tx_B`'s fees during `CheckTx` without the nonce being validated against the on-chain state ‚Äî meaning a replacement with `nonce=N` where `N` is no longer the current sequence (e.g., `tx_A` was already included in a block and sequence is now `N+1`) will still pass `CheckTx` due to the cache shortcut, admitting an invalid transaction into the mempool and deducting fees that will not be refunded until the invalid `DeliverTx` fails.

The root cause is confirmed at:
- `ante/cache/antecache.go` lines 9‚Äì12: key has no tx-hash binding [1](#0-0) 
- `ante/eth.go` lines 291‚Äì297: unconditional `continue` on cache hit [2](#0-1) 
- `evmd/ante/handler_options.go` lines 139‚Äì151: fee deduction precedes nonce check [3](#0-2)
