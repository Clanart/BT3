Audit Report

## Title
Unvalidated Pool Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain Approved User Tokens via Callback — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

`MetricOmmPoolLiquidityAdder` accepts a caller-supplied `pool` address and stores it as the trusted callback authority in transient storage without validating it against the factory. A malicious pool contract can exploit `metricOmmModifyLiquidityCallback` to pull any token the victim has approved to the adder — up to the user-supplied caps — by trivially satisfying the `msg.sender == expectedPool` guard that was seeded with the attacker's own address. The NatSpec at lines 19–21 explicitly documents this gap, confirming it is a known omission rather than an oversight in the analysis.

## Finding Description

`MetricOmmSwapRouterBase._setNextCallbackContext` calls `_requireFactoryPool(pool)` before writing to transient storage, ensuring only factory-deployed pools can be set as callback authorities. [1](#0-0) 

`MetricOmmPoolLiquidityAdder` performs no equivalent check. Its NatSpec explicitly documents the gap: [2](#0-1) 

`addLiquidityExactShares` accepts an arbitrary `pool` from the caller and passes it directly to `_addLiquidity` with `msg.sender` as the payer: [3](#0-2) 

`_addLiquidity` stores the attacker-controlled pool as the trusted callback authority via `_setPayContext`, then calls into it: [4](#0-3) 

Inside `metricOmmModifyLiquidityCallback`, the only caller-identity guard is `msg.sender != expectedPool`. Because `expectedPool` was seeded with the attacker's address, this check is trivially satisfied: [5](#0-4) 

The callback then reads token addresses from `IMetricOmmPool(msg.sender).getImmutables()` — fully attacker-controlled — and calls `pay(token0/token1, payer, msg.sender, amount)`, pulling tokens from the victim to the malicious pool: [6](#0-5) 

`pay` in `PeripheryPayments` executes `safeTransferFrom(payer, recipient, value)` when `payer != address(this)`: [7](#0-6) 

**Complete exploit call path:**
1. Victim approves `MetricOmmPoolLiquidityAdder` for token X (standard prerequisite for any deposit).
2. Victim is tricked (phishing front-end / malicious aggregator) into calling `addLiquidityExactShares(maliciousPool, victim, salt, deltas, 1e18, 0, "")`.
3. `_addLiquidity` calls `_setPayContext(maliciousPool, victim, 1e18, 0)` — victim is stored as payer, malicious pool as `expectedPool`.
4. `IMetricOmmPoolActions(maliciousPool).addLiquidity(...)` is called.
5. Malicious pool immediately calls back `metricOmmModifyLiquidityCallback(1e18, 0, abi.encode(KIND_PAY))`.
6. Guard `msg.sender == expectedPool` passes (both are `maliciousPool`).
7. `getImmutables()` returns attacker-chosen `token0 = tokenToSteal`.
8. `pay(tokenToSteal, victim, maliciousPool, 1e18)` → `safeTransferFrom(victim, maliciousPool, 1e18)` executes.
9. Victim loses 1e18 of `tokenToSteal`; malicious pool receives it.

The `addLiquidityWeighted` path is equally exploitable: the malicious pool cooperates during the probe phase by reverting with a crafted `LiquidityProbe(need0, need1)`, steering the scale calculation, then drains tokens in the paying phase via the same callback mechanism. [8](#0-7) 

## Impact Explanation

Any user who has granted an ERC-20 allowance to `MetricOmmPoolLiquidityAdder` — required for any legitimate deposit — can have those tokens stolen in a single transaction. The malicious pool controls which tokens are pulled (via `getImmutables()`) and how much (up to the victim's own `maxAmountToken0`/`maxAmountToken1` caps, which users routinely set to their full intended deposit). This is a direct, unrecoverable loss of user principal, meeting the Critical/High threshold under the allowed impact gate.

## Likelihood Explanation

The attack requires the victim to call `addLiquidityExactShares` or `addLiquidityWeighted` with a malicious pool address. This is realistic because: (1) `MetricOmmPoolLiquidityAdder` is a public, permissionless entry point; (2) a phishing front-end or malicious aggregator can supply a crafted pool address while presenting a legitimate-looking UI; (3) the victim only needs to have previously approved the adder — a one-time action done for any legitimate deposit; (4) no privileged access or special conditions are required on the attacker's side beyond deploying a contract.

## Recommendation

Add a factory validation check inside `_addLiquidity` (or at the top of each public entry point) mirroring the pattern already used in `MetricOmmSwapRouterBase._setNextCallbackContext`:

```solidity
// In MetricOmmPoolLiquidityAdder._addLiquidity or each public entry point:
if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
```

This requires storing the factory address as an immutable (as `MetricOmmSwapRouterBase` does) and adding it to the constructor. The check must be applied before `_setPayContext` is called so the attacker-controlled address never reaches transient storage as a trusted callback authority.

## Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

contract MaliciousPool {
    address public immutable adder;
    address public immutable tokenToSteal;

    constructor(address _adder, address _tokenToSteal) {
        adder = _adder;
        tokenToSteal = _tokenToSteal;
    }

    // Called by LiquidityAdder._addLiquidity → pool.addLiquidity
    function addLiquidity(address, uint80, bytes calldata, bytes calldata, bytes calldata)
        external returns (uint256, uint256)
    {
        // Trigger the callback claiming 1e18 of tokenToSteal is owed
        IMetricOmmPoolLiquidityAdder(adder).metricOmmModifyLiquidityCallback(
            1e18, 0, abi.encode(uint8(1)) // KIND_PAY = 1
        );
        return (1e18, 0);
    }

    // LiquidityAdder reads token addresses from here
    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = tokenToSteal;
        imm.token1 = address(0);
    }
}

// Test steps (Foundry):
// 1. victim.approve(address(liquidityAdder), type(uint256).max) for tokenToSteal
// 2. attacker deploys MaliciousPool(address(liquidityAdder), address(tokenToSteal))
// 3. victim calls:
//    liquidityAdder.addLiquidityExactShares(
//        address(maliciousPool), victim, 0, deltas, 1e18, 0, ""
//    )
// 4. Assert: tokenToSteal.balanceOf(victim) decreased by 1e18
//            tokenToSteal.balanceOf(address(maliciousPool)) increased by 1e18
```
