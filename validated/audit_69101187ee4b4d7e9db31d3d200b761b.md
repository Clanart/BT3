### Title
Forged `n_transactions` in peer-supplied `SignedBlockHeader` causes empty `BlockBody` stored with mismatched commitments ŌĆö (`crates/apollo_p2p_sync/src/client/header.rs`, `crates/apollo_p2p_sync/src/client/transaction.rs`)

---

### Summary

`HeaderStreamBuilder::parse_data_for_block` accepts any `n_transactions` value from a peer without validation. The value is stored verbatim in `StorageBlockHeader`. `TransactionStreamFactory::parse_data_for_block` then reads that stored `n_transactions` as its loop bound (`target_transaction_len`). A forged value of `0` causes the loop to never execute, storing an empty `BlockBody` while the header retains the peer-supplied (wrong) `transaction_commitment` and `receipt_commitment`. No cross-check between the stored body and the stored commitments exists anywhere in the path.

---

### Finding Description

**Step 1 ŌĆö Header validation is incomplete.**

`HeaderStreamBuilder::parse_data_for_block` performs exactly two checks on the incoming `SignedBlockHeader`: [1](#0-0) 

- `block_number` matches the expected value
- `signatures.len() == ALLOWED_SIGNATURES_LENGTH`

No check is made on `n_transactions`, `transaction_commitment`, `receipt_commitment`, `block_hash`, or the cryptographic validity of the signature against the block hash. The header is returned as-is.

**Step 2 ŌĆö Forged `n_transactions` is stored verbatim.**

`SignedBlockHeader::write_to_storage` calls `append_header` with the full `BlockHeader` struct, including the attacker-controlled `n_transactions=0`: [2](#0-1) 

`StorageBlockHeader.n_transactions` is now `0`. [3](#0-2) 

**Step 3 ŌĆö Transaction stream uses the stored value as its loop bound.**

`TransactionStreamFactory::parse_data_for_block` reads `n_transactions` from the already-stored header and uses it as `target_transaction_len`: [4](#0-3) 

With `target_transaction_len = 0`, the condition `0 < 0` is immediately false. The loop body never executes.

**Step 4 ŌĆö Empty `BlockBody` is stored; no commitment cross-check exists.**

The function returns `Ok(Some((empty_block_body, block_number)))`: [5](#0-4) 

`(BlockBody, BlockNumber)::write_to_storage` then persists the empty body via `append_body` with no verification against the `transaction_commitment` or `receipt_commitment` already stored in the header: [6](#0-5) 

**Resulting storage state (concrete corrupted values):**

| Field | Expected | Stored |
|---|---|---|
| `StorageBlockHeader.n_transactions` | 3 | 0 (forged) |
| `StorageBlockHeader.transaction_commitment` | computed from 3 txs | peer-supplied arbitrary value |
| `StorageBlockHeader.receipt_commitment` | computed from 3 receipts | peer-supplied arbitrary value |
| `BlockBody.transactions` | 3 entries | empty |

A developer TODO in the same file confirms awareness that `n_transactions` is unverified: [7](#0-6) 

---

### Impact Explanation

Any RPC call that reads the stored header or body for the affected block will return authoritative-looking wrong values: `n_transactions=0`, an empty transaction list, and peer-fabricated `transaction_commitment`/`receipt_commitment`. This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

The mismatch is permanent until the node re-syncs the block. Because the header marker advances after the forged header is stored, the transaction stream immediately proceeds to fetch transactions for the next block, leaving the corrupted block body in place.

---

### Likelihood Explanation

Any peer the syncing node connects to can trigger this. No special privilege is required ŌĆö the attacker only needs to be a reachable p2p peer. The attack requires sending a single malformed `SignedBlockHeader` message with `n_transactions=0` (or any value lower than the real count).

---

### Recommendation

In `HeaderStreamBuilder::parse_data_for_block`, validate `n_transactions` against the `concatenated_counts` field already present in the header (which encodes the authoritative transaction count as part of the commitment structure). Alternatively, after `write_to_storage` for the body, verify that `block_body.transactions.len() == stored_header.n_transactions` and reject/report the peer if they differ. The existing TODO at line 140ŌĆō141 of `header.rs` already identifies this gap.

---

### Proof of Concept

```
1. Peer sends SignedBlockHeader { block_number: 5, n_transactions: 0,
       transaction_commitment: <arbitrary>, receipt_commitment: <arbitrary>,
       signatures: [valid_length_sig] }
2. parse_data_for_block (header): block_number matches, signatures.len()==1 ŌåÆ Ok(Some(header))
3. write_to_storage (header): StorageBlockHeader { n_transactions: 0,
       transaction_commitment: <arbitrary>, ... } committed to DB
4. TransactionStreamFactory::parse_data_for_block:
       target_transaction_len = get_block_header(5).n_transactions = 0
       while 0 < 0 { /* never executes */ }
       return Ok(Some((BlockBody { transactions: [], ... }, 5)))
5. write_to_storage (body): append_body(5, empty_body) committed to DB
6. RPC starknet_getBlockWithTxs(block_number=5) ŌåÆ { transactions: [], n_transactions: 0,
       transaction_commitment: <attacker value> }  ŌåÉ wrong authoritative response
```
