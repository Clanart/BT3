### Title
Silent Success on Self-Destructed Custom Minter in `finTransfer` Permanently Freezes User Funds - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` calls `ICustomMinter.mint` (a void-returning function) on the stored `customMinters[payload.tokenAddress]` address without verifying that the contract still has code at that address. If the custom minter has been self-destructed, the EVM `call` opcode returns `(true, "")` — success with empty return data — and Solidity silently accepts it for a void function. The destination nonce is already marked consumed before this call, so the transfer is permanently finalized with no tokens ever minted to the recipient.

---

### Finding Description

In `finTransfer`, the nonce is irrevocably marked used at line 287 before any token dispatch:

```solidity
completedTransfers[payload.destinationNonce] = true;
``` [1](#0-0) 

After MPC signature verification, the function dispatches to the custom minter path when `customMinters[payload.tokenAddress] != address(0)`:

```solidity
} else if (customMinters[payload.tokenAddress] != address(0)) {
    ICustomMinter(customMinters[payload.tokenAddress]).mint(
        payload.tokenAddress,
        payload.recipient,
        payload.amount
    );
``` [2](#0-1) 

The `ICustomMinter.mint` interface declares a void return:

```solidity
interface ICustomMinter {
    function mint(address token, address to, uint128 amount) external;
``` [3](#0-2) 

There is no `extcodesize` check before this call. In Solidity 0.8.x, a high-level call to a void function on an address with zero code size returns `(true, "")` from the EVM `call` opcode. Because the function has no return value, Solidity's ABI decoder accepts empty return data as valid and does not revert. The call silently succeeds.

The same structural issue exists in the `isBridgeToken` path, where `IBridgeToken(payload.tokenAddress).mint(recipient, amount)` is also void-returning and called without a code-existence check: [4](#0-3) 

The `customMinters` mapping is populated by `addCustomToken` (admin-only), but the custom minter contract itself is an external contract whose lifecycle is independent of the bridge admin: [5](#0-4) 

---

### Impact Explanation

When `finTransfer` is called with a valid MPC signature for a token whose registered custom minter has been self-destructed:

1. `completedTransfers[destinationNonce]` is set to `true` — permanently consuming the nonce.
2. `ICustomMinter(customMinters[tokenAddress]).mint(...)` is dispatched to an address with no code — the call returns success silently.
3. No tokens are minted to `payload.recipient`.
4. The nonce can never be replayed (`NonceAlreadyUsed` revert on retry).
5. The user's assets were already burned or locked on the source chain (NEAR/Solana/StarkNet) when `initTransfer` was submitted.

Result: **permanent, irrecoverable loss of user funds** — the cross-chain transfer is finalized on-chain with zero token delivery and no recovery path.

This matches: *Critical. Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.*

---

### Likelihood Explanation

The custom minter is an external contract registered by the bridge admin. Its self-destruction can occur if:

- The custom minter contract contains a `selfdestruct` opcode (common in older or upgradeable minter designs).
- The custom minter is itself a UUPS/transparent proxy whose implementation is self-destructed (the proxy's `delegatecall` to a zero-code implementation returns `(false, "")`, but if the proxy itself is self-destructed the address has no code).
- A minter contract owner — who is not necessarily the bridge admin — triggers self-destruction.

Once the minter is destroyed, every in-flight or future `finTransfer` for that token silently succeeds with no mint. Users have no way to detect this from the transaction receipt (it emits `FinTransfer` normally).

Likelihood: **Medium** — requires a specific precondition (minter self-destruction), but the precondition is realistic for any custom minter with a selfdestruct path, and the bridge provides no defense against it.

---

### Recommendation

**Short term:** Add an `extcodesize` guard before calling the custom minter and bridge token in `finTransfer`:

```solidity
address minter = customMinters[payload.tokenAddress];
require(minter.code.length > 0, "CustomMinter: no code at address");
ICustomMinter(minter).mint(...);
```

Apply the same guard before `IBridgeToken(payload.tokenAddress).mint(...)`.

**Long term:** Consider moving the nonce-marking (`completedTransfers[nonce] = true`) to *after* the token dispatch succeeds, so that a failed or silent dispatch does not permanently consume the nonce. Alternatively, implement a recovery mechanism (e.g., admin-callable `retryFinTransfer`) that can re-execute a finalization whose token dispatch silently failed.

### Proof of Concept

1. Admin calls `addCustomToken("near:token.near", tokenAddr, minterAddr, 18)` — `customMinters[tokenAddr] = minterAddr`.
2. `minterAddr` is a contract with a `selfdestruct` function; its owner calls `selfdestruct`. `minterAddr` now has `code.length == 0`, but `customMinters[tokenAddr]` still holds the non-zero address.
3. A user initiates a cross-chain transfer on NEAR, burning their tokens. MPC signs a `TransferMessagePayload` for `tokenAddr` with `destinationNonce = N`.
4. Anyone (relayer or user) calls `finTransfer(sig, payload)` on EVM.
5. Line 287: `completedTransfers[N] = true` — nonce consumed.
6. Line 311–313: MPC signature verifies successfully.
7. Line 331: `customMinters[tokenAddr] != address(0)` — true (stale mapping entry).
8. Line 332–336: `ICustomMinter(minterAddr).mint(tokenAddr, recipient, amount)` — EVM `call` to zero-code address returns `(true, "")`. Solidity accepts it (void function).
9. `FinTransfer` event is emitted. Transaction succeeds.
10. `recipient` receives zero tokens. Nonce `N` is permanently spent. User's source-chain assets are gone. [6](#0-5)
