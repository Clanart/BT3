### Title
`ProofOfInclusion::valid()` Final Check Is a Tautology — Forged Proofs Always Pass Without External Root Validation - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` only checks internal self-consistency of the proof chain. Its final comparison `existing_hash == self.root_hash()` is a tautology — it is always `true` when the loop completes — because `root_hash()` is derived from the same proof data (`layers.last().combined_hash`) that `existing_hash` was just set to. No external expected root hash is ever compared. An attacker who can supply a `ProofOfInclusion` (which is `Streamable` and fully deserializable) can forge a self-consistent proof for any arbitrary key-value pair and `valid()` will return `true`.

### Finding Description

`ProofOfInclusion::valid()` is implemented as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;  // existing_hash = layer.combined_hash
    }

    existing_hash == self.root_hash()     // always true — see below
}
``` [1](#0-0) 

And `root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // <-- same field as existing_hash after the loop
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

**The tautology:** After the loop, `existing_hash` holds the value of the last layer's `combined_hash` (because the loop only continues when `calculated_hash == layer.combined_hash`, and then sets `existing_hash = calculated_hash`). `self.root_hash()` also returns `layers.last().combined_hash`. Therefore `existing_hash == self.root_hash()` is always `true` when the loop completes. The final check adds no validation.

**The missing check:** `valid()` never accepts an external expected root hash to compare against. It only verifies that the proof's own internal hash chain is self-consistent. An attacker can trivially construct a self-consistent chain:

1. Choose any `node_hash` (e.g., hash of a fake key-value pair).
2. For each layer, pick any `other_hash` and `other_hash_side`, then compute `combined_hash = calculate_internal_hash(existing_hash, side, other_hash)`.
3. The resulting `ProofOfInclusion` passes `valid()` with `true`, but `root_hash()` returns an attacker-controlled value unrelated to the real tree.

`ProofOfInclusion` is `Streamable` and fully deserializable from bytes, so it can be received from untrusted network input. [3](#0-2) 

The Python binding exposes `valid()` and `root_hash()` as separate methods, placing the burden of the external root comparison entirely on the caller — with no documentation or enforcement that both must be checked. [4](#0-3) [5](#0-4) 

### Impact Explanation

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

A caller who receives a `ProofOfInclusion` from an untrusted source (e.g., a DataLayer peer) and calls `proof.valid()` — without separately asserting `proof.root_hash() == known_root` — will accept a forged proof for any key-value pair. This allows an attacker to prove arbitrary state inclusion in a DataLayer store, enabling false data attestation and state corruption at the DataLayer level.

### Likelihood Explanation

The function name `valid()` strongly implies complete proof validation. There is no documentation, type-level enforcement, or API design that forces callers to also check `root_hash()` against an external expected value. Any Python or Rust caller who relies solely on `valid()` is vulnerable. The `ProofOfInclusion` struct is `Streamable`, making it trivially injectable from network input.

### Recommendation

`valid()` should accept an expected root hash parameter and compare against it:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root  // compare against external root, not self-derived
}
```

The current `valid()` (which only checks internal consistency) should either be removed or clearly renamed to `is_internally_consistent()` with documentation warning that it does not validate against any tree root.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Attacker wants to forge proof that fake_node_hash is in the tree
fake_node_hash = bytes([0xAA] * 32)
other_hash     = bytes([0xBB] * 32)

# Compute combined_hash so the layer is internally consistent
# (using whatever calculate_internal_hash does — left-side concatenation + sha256)
combined = hashlib.sha256(fake_node_hash + other_hash).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=0,   # left
    other_hash=other_hash,
    combined_hash=combined,
)

forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True for a completely forged proof
assert forged_proof.valid() == True

# root_hash() returns attacker-controlled value, not the real tree root
print("Forged root:", forged_proof.root_hash().hex())
# A caller who only checks valid() accepts this as a valid inclusion proof
```

The forged proof passes `valid()` because the tautological final check `existing_hash == self.root_hash()` is always satisfied, and no external root is ever consulted.
