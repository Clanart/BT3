### Title
`execute_replace_class` Does Not Validate That `request.class_hash` Is a Declared Class — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the caller-supplied `class_hash` corresponds to a legitimately declared contract class. Any contract can invoke `replace_class` with an arbitrary, undeclared class hash. The OS will commit this invalid hash to the contract's state entry without complaint. Any subsequent call to that contract will then fail irrecoverably at the OS level, permanently freezing all funds held by the contract.

---

### Finding Description

In `execute_replace_class`, the OS reads `request.class_hash` directly from the syscall buffer and writes it into `contract_state_changes` without checking whether that hash was ever declared on-chain:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 is an explicit acknowledgment by the developers that this validation is missing. [2](#0-1) 

When a future call is made to the affected contract, `execute_entry_point` reads the stored `class_hash` and performs a lookup in `contract_class_changes`:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
``` [3](#0-2) 

Because the class was never declared, `contract_class_changes[undeclared_hash]` returns `0`. The subsequent `find_element` call then searches for a compiled class with hash `0`:

```cairo
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [4](#0-3) 

`find_element` panics (Cairo assertion failure) when the key is not found. Since no compiled class with hash `0` exists, every future call to the contract fails at the OS level with no recovery path. The contract is permanently bricked.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that holds user funds (tokens, ETH bridged via L1→L2, NFTs, etc.) and whose class hash is replaced with an undeclared value becomes permanently inaccessible. No withdrawal, transfer, or administrative function can ever execute again because every entry point invocation fails inside `execute_entry_point` before reaching contract code. The funds are frozen forever on-chain with no upgrade or rescue path.

---

### Likelihood Explanation

The attack surface is realistic and reachable by an unprivileged transaction sender:

1. **Contracts with public upgrade paths**: Many protocol contracts expose an upgrade function (governance-controlled or otherwise) that internally calls `replace_class`. If the upgrade function does not independently validate the class hash — reasonably expecting the OS to enforce it — an attacker who can trigger the upgrade (e.g., via a governance exploit, a reentrancy, or a misconfigured access control) can supply an undeclared hash.

2. **No OS-level guard**: The OS is the canonical last line of defense. The missing check is explicitly noted in the source (`TODO`), confirming the guard was intended but never implemented.

3. **Irreversibility**: Unlike most bugs, there is no recovery. Once the state is committed with the invalid class hash, the contract is permanently frozen.

---

### Recommendation

Before updating `contract_state_changes`, verify that `request.class_hash` maps to a non-zero compiled class hash in `contract_class_changes`:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(
    key=request.class_hash
);
with_attr error_message("replace_class: class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the validation already performed implicitly in `execute_entry_point` but makes it an explicit, enforced precondition at the syscall boundary, consistent with how `execute_deploy` is expected to be hardened.

---

### Proof of Concept

1. Declare a contract class `A` on StarkNet (legitimate).
2. Deploy a contract `C` using class `A`. Fund `C` with tokens.
3. From within `C`, expose a public function `upgrade(new_hash: felt)` that calls `replace_class(new_hash)`.
4. As an unprivileged user, call `C.upgrade(0xdeadbeef)` where `0xdeadbeef` is never declared.
5. The OS executes `execute_replace_class`: no validation occurs; `contract_state_changes[C].class_hash` is set to `0xdeadbeef`. [1](#0-0) 
6. The block is proven and the state is committed.
7. Any subsequent call to `C` reaches `execute_entry_point`, which reads `class_hash = 0xdeadbeef`, performs `dict_read{contract_class_changes}(0xdeadbeef)` → returns `0`, then calls `find_element(..., key=0)` → Cairo panic. [5](#0-4) 
8. All funds in `C` are permanently frozen with no recovery mechanism.
