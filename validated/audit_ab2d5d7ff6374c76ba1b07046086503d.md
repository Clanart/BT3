### Title
Missing Bound Check on `proof_facts_size` Enables Unbounded Poseidon Hash Computation in OS Transaction Processing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

An invoke transaction sender can supply an arbitrarily large `proof_facts` array. Unlike every other variable-length transaction field (`calldata_size`, `signature_len`, `constructor_calldata_size`), `proof_facts_size` is loaded from the transaction without any upper-bound assertion. The OS then feeds this unbounded length directly into a Poseidon hash loop during transaction-hash computation, causing O(N) builtin consumption proportional to the attacker-chosen size. A single such transaction in a block can make OS proof generation prohibitively expensive or impossible, halting the network's ability to finalize blocks.

---

### Finding Description

Every other variable-length field in the OS transaction pipeline is explicitly capped before use:

- `signature_len` → `assert_nn_le(signature_len, SIERRA_ARRAY_LEN_BOUND - 1)` [1](#0-0) 
- `calldata_size` → `assert_nn_le(tx_execution_context.calldata_size, SIERRA_ARRAY_LEN_BOUND - 1)` [2](#0-1) 
- `constructor_calldata_size` → `assert_nn_le(constructor_calldata_size, SIERRA_ARRAY_LEN_BOUND - 1)` [3](#0-2) 

`proof_facts_size`, however, is loaded from the hint with no analogous guard:

```cairo
local proof_facts_size;
local proof_facts: felt*;
%{ TxProofFacts %}
``` [4](#0-3) 

This value is then passed directly into `compute_invoke_transaction_hash`:

```cairo
let transaction_hash = compute_invoke_transaction_hash(
    common_fields=common_tx_fields,
    execution_context=tx_execution_context,
    account_deployment_data_size=account_deployment_data_size,
    account_deployment_data=account_deployment_data,
    proof_facts_size=proof_facts_size,
    proof_facts=proof_facts,
);
``` [5](#0-4) 

Inside `compute_invoke_transaction_hash`, the unbounded length is fed into a Poseidon hash loop:

```cairo
if (proof_facts_size != 0) {
    poseidon_hash_update_with_nested_hash(
        data_ptr=proof_facts, data_length=proof_facts_size
    );
}
``` [6](#0-5) 

`poseidon_hash_update_with_nested_hash` iterates over every element in `proof_facts`, consuming one `PoseidonBuiltin` instance per element. Because `proof_facts_size` is a Cairo `felt` (range `[0, P)` where P ≈ 2²⁵¹), an attacker can set it to any value up to the field prime, making the hash loop effectively unbounded.

The `proof_facts` data is committed to the transaction hash, so the OS must process exactly the array the attacker submitted — the hint cannot deviate without producing a wrong hash.

---

### Impact Explanation

The Starknet OS runs as a Cairo program whose output is a STARK proof. Every `PoseidonBuiltin` invocation adds a row to the proof trace. If `proof_facts_size` is set to a value such as 2²⁰ (≈ 1 million), the prover must generate a trace with at least 1 million Poseidon rows for a single transaction. At 2³⁰ the proof becomes computationally infeasible. Because the OS must process all transactions in a block before producing any output, a single such transaction prevents the block from ever being proven, and the network cannot confirm new transactions.

**Impact**: High — Network not being able to confirm new transactions (total network shutdown).

---

### Likelihood Explanation

Any unprivileged account can submit an invoke transaction (version 3) with an arbitrarily large `proof_facts` array. The transaction is valid from the mempool's perspective as long as the transaction hash is consistent. The sequencer has no documented protocol-level limit on `proof_facts_size` (unlike `calldata_size` which is bounded by `SIERRA_ARRAY_LEN_BOUND`). Once the sequencer includes the transaction in a block and the OS begins proof generation, the damage is done — the block cannot be finalized.

---

### Recommendation

Add an explicit upper-bound assertion on `proof_facts_size` immediately after it is loaded from the hint, mirroring the pattern used for all other variable-length fields:

```cairo
local proof_facts_size;
local proof_facts: felt*;
%{ TxProofFacts %}
assert_nn_le(proof_facts_size, SIERRA_ARRAY_LEN_BOUND - 1);  // ADD THIS
```

This should be placed in `execute_invoke_function_transaction` in `transaction_impls.cairo` directly after line 281, consistent with the bound checks applied to `signature_len` (line 218), `calldata_size` (line 485), and `constructor_calldata_size` (line 534).

---

### Proof of Concept

1. Attacker constructs a valid version-3 invoke transaction where the `proof_facts` field contains N = 2²⁰ arbitrary felt values.
2. The transaction hash is computed client-side to include all N elements, making the transaction cryptographically valid.
3. The attacker submits the transaction to the sequencer's mempool. The gateway has no protocol-mandated limit on `proof_facts_size`.
4. The sequencer includes the transaction in a block and triggers OS proof generation.
5. The OS reaches `execute_invoke_function_transaction` → `compute_invoke_transaction_hash` → `poseidon_hash_update_with_nested_hash(data_ptr=proof_facts, data_length=2²⁰)`.
6. The Poseidon builtin loop executes 2²⁰ ≈ 1,048,576 iterations, each adding a row to the proof trace.
7. Proof generation time and memory grow proportionally; at sufficiently large N the prover times out or runs out of resources.
8. The block cannot be proven; the network stalls and cannot confirm new transactions.

**Root cause line**: `transaction_impls.cairo` lines 279–281 (missing `assert_nn_le` on `proof_facts_size`). [4](#0-3) 

**Exploitation path**: `poseidon_hash_update_with_nested_hash` called with attacker-controlled length at `transaction_hash.cairo` lines 210–213. [6](#0-5)
