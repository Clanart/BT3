### Title
Unauthenticated Orphan Witness Silently Evicts Legitimate Witness via `ChunkProductionKey` Cache-Key Collision — (`chain/client/src/stateless_validation/chunk_validator/orphan_witness_pool.rs`)

---

### Summary

`OrphanStateWitnessPool` keys its LRU cache on `ChunkProductionKey` (`shard_id`, `epoch_id`, `height_created`), which does **not** include `prev_block_hash`. Witnesses are admitted to the pool without signature validation. Any unprivileged peer can send a syntactically valid but cryptographically unsigned `ChunkStateWitness` carrying the same `ChunkProductionKey` as a legitimate in-flight orphan but a different `prev_block_hash`. The `LruCache::push` call silently replaces the legitimate entry. When the real previous block later arrives, `take_state_witnesses_waiting_for_block` matches on `prev_block_hash` and finds nothing; the validator never produces a chunk endorsement for that slot.

---

### Finding Description

**Cache key definition** — `ChunkProductionKey` contains only `(shard_id, epoch_id, height_created)`: [1](#0-0) 

**Pool storage** — the cache is keyed on `ChunkProductionKey`; inserting a duplicate key silently evicts the prior entry: [2](#0-1) [3](#0-2) 

**No signature check before orphan admission** — `process_chunk_state_witness_message` calls `handle_orphan_witness` directly when the previous block is absent; signature validation only occurs inside `process_chunk_state_witness`, which is never reached for orphans: [4](#0-3) 

**`handle_orphan_witness` guards only distance and size** — no cryptographic check: [5](#0-4) 

**Retrieval matches on `prev_block_hash`** — if the stored entry carries an attacker-controlled hash, the lookup returns nothing when the real block arrives: [6](#0-5) 

**Ready-orphan processing** — called on every new block; if the pool entry's `prev_block_hash` does not match the arriving block, the witness is never processed: [7](#0-6) 

---

### Impact Explanation

A validator that fails to process a legitimate `ChunkStateWitness` never calls `send_chunk_endorsement_to_block_producers`. Chunk endorsements require ≥ 2/3 of assigned validators. An attacker who simultaneously targets ≥ 1/3 of the validator set for a given shard slot can suppress all endorsements for that chunk, preventing the block producer from including it and stalling liveness for that shard.

---

### Likelihood Explanation

- **Unprivileged**: any network peer can send a `ChunkStateWitness` message.
- **Predictable parameters**: `shard_id`, `epoch_id`, and `height_created` are deterministic from public chain state.
- **Timing window**: the attacker must deliver the fake witness after the legitimate one is stored but before the previous block arrives — a window of one network round-trip, which is feasible on a monitored network.
- **No existing guard**: the pool comment acknowledges that signature validation is *expected* but not enforced at the orphan admission path.

---

### Recommendation

1. **Validate the witness signature before storing it in the orphan pool.** Move the signature check from `process_chunk_state_witness` to a shared pre-admission step that runs for both the normal and orphan paths.
2. **Include `prev_block_hash` (or `chunk_hash`) in the cache key** so that two witnesses for the same production slot but different fork parents occupy distinct entries and neither silently evicts the other.

---

### Proof of Concept

```
Attacker observes: upcoming chunk at (shard=S, epoch=E, height=H, prev_block=B_real)

Step 1: Chunk producer broadcasts ChunkStateWitness W_real
        (shard=S, epoch=E, height=H, prev_block=B_real, valid_sig)

Step 2: W_real arrives at validator V; B_real not yet available.
        → handle_orphan_witness: distance OK, size OK, no sig check
        → pool.add_orphan_state_witness(W_real)
        → pool entry: key=(S,E,H), prev_block_hash=B_real

Step 3: Attacker sends W_fake
        (shard=S, epoch=E, height=H, prev_block=FAKE_HASH, no/invalid sig)
        to validator V.

Step 4: W_fake arrives at V; FAKE_HASH not in store.
        → handle_orphan_witness: distance OK, size OK, no sig check
        → pool.add_orphan_state_witness(W_fake)
        → LruCache::push with same key (S,E,H) EVICTS W_real
        → pool entry: key=(S,E,H), prev_block_hash=FAKE_HASH

Step 5: B_real arrives at V.
        → handle_block_notification → process_ready_orphan_witnesses(B_real)
        → take_state_witnesses_waiting_for_block(B_real)
        → iterates pool: only entry has prev_block_hash=FAKE_HASH ≠ B_real
        → returns empty list

Step 6: V never calls process_chunk_state_witness for W_real.
        V never sends a chunk endorsement for (S,E,H).
```

Repeating this against ≥ 1/3 of the validator set for shard S at height H suppresses all endorsements for that chunk slot.

    pub fn take_state_witnesses_waiting_for_block(
        &mut self,
        prev_block: &CryptoHash,
    ) -> Vec<ChunkStateWitness> {
        let mut to_remove: Vec<ChunkProductionKey> = Vec::new();
        for (cache_key, cache_entry) in &self.witness_cache {
            if cache_entry.witness.chunk_header().prev_block_hash() == prev_block {
                to_remove.push(cache_key.clone());
            }
        }
        let mut result = Vec::new();
        for cache_key in to_remove {
            let ready_witness = self
                .witness_cache
                .pop(&cache_key)
                .expect("The cache contains this entry, a moment ago it was iterated over");
            result.push(ready_witness.witness);
        }
        result
    }

    pub fn handle_orphan_witness(
        &mut self,
        witness: ChunkStateWitness,
        witness_size: ChunkStateWitnessSize,
    ) -> Result<HandleOrphanWitnessOutcome, Error> {
        let chunk_header = witness.chunk_header();
        let witness_height = chunk_header.height_created();
        let witness_shard = chunk_header.shard_id();

        let _span = tracing::debug_span!(
            target: "chunk_validation",
            "handle_orphan_witness",
            witness_height,
            ?witness_shard,
            witness_chunk = ?chunk_header.chunk_hash(),
            witness_prev_block = ?chunk_header.prev_block_hash(),
        )
        .entered();

        // Don't save orphaned state witnesses which are far away from the current chain head.
        let chain_head = self.chain_store.head()?;
        let head_distance = witness_height.saturating_sub(chain_head.height);

        if !ALLOWED_ORPHAN_WITNESS_DISTANCE_FROM_HEAD.contains(&head_distance) {
            tracing::debug!(
                target: "chunk_validation",
                head_height = chain_head.height,
                "not saving an orphaned chunk state witness because its height isn't within the allowed height range"
            );
            return Ok(HandleOrphanWitnessOutcome::TooFarFromHead {
                witness_height,
                head_height: chain_head.height,
            });
        }

        // Don't save orphaned state witnesses which are bigger than the allowed limit.
        let witness_size_u64: u64 = witness_size as u64;
        if witness_size_u64 > self.max_orphan_witness_size {
            tracing::warn!(
                target: "chunk_validation",
                witness_height,
                ?witness_shard,
                witness_chunk = ?chunk_header.chunk_hash(),
                witness_prev_block = ?chunk_header.prev_block_hash(),
                witness_size = witness_size_u64,
                "not saving an orphaned chunk state witness because it's too big, this is unexpected"
            );
            return Ok(HandleOrphanWitnessOutcome::TooBig(witness_size_u64 as usize));
        }

        // Orphan witness is OK, save it to the pool
        tracing::debug!(target: "chunk_validation", "saving an orphaned chunk state witness to orphan pool");
        self.orphan_witness_pool
            .lock()
            .add_orphan_state_witness(witness, witness_size_u64 as usize);
        Ok(HandleOrphanWitnessOutcome::SavedToPool)
    }
