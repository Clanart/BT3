### Title
Missing Re-Check of `finalised_transfers` in `fast_fin_transfer_to_near_callback` Enables Double-Spend — (File: `near/omni-bridge/src/lib.rs`)

### Summary

`fast_fin_transfer` performs a pre-check against `finalised_transfers` before initiating an async cross-contract call. The subsequent callback `fast_fin_transfer_to_near_callback` only checks the `fast_transfers` map — a strict subset of the full finalization state — and never re-checks `finalised_transfers`. If a regular `fin_transfer` for the same origin transfer is processed in the window between the pre-check and the callback, the recipient receives tokens twice.

### Finding Description

The `fast_fin_transfer` function (line 778) checks `is_unified_transfer_finalised` against `finalised_transfers` before proceeding:

```rust
if self.is_unified_transfer_finalised(&fast_fin_transfer_msg.transfer_id) {
    env::panic_str(BridgeError::TransferAlreadyFinalised.to_string().as_str());
}
``` [1](#0-0) 

It then initiates an async cross-contract call to `check_or_pay_ft_storage`, scheduling `fast_fin_transfer_to_near_callback` to run after it returns. [2](#0-1) 

Inside `fast_fin_transfer_to_near_callback`, the only guard is `add_fast_transfer`, which exclusively checks the `fast_transfers` LookupMap: [3](#0-2) 

`add_fast_transfer` panics only if the key already exists in `fast_transfers`: [4](#0-3) 

There is **no re-check** of `finalised_transfers` inside the callback. The two sets are entirely independent: [5](#0-4) 

Meanwhile, `process_fin_transfer_to_near` (called from `fin_transfer_callback`) checks `fast_transfers` to decide whether to redirect tokens to the relayer: [6](#0-5) 

If the fast transfer has not yet been recorded in `fast_transfers` at the time `fin_transfer_callback` runs, `fast_transfer_status` is `None` and tokens are sent to the original recipient — not the relayer.

### Impact Explanation

**Race condition leading to double-spend:**

1. Trusted Relayer A calls `fast_fin_transfer(T)` → pre-check passes (T not in `finalised_transfers`) → `check_or_pay_ft_storage` is dispatched asynchronously.
2. In the same or a subsequent block, Trusted Relayer B calls `fin_transfer(T)` → `verify_proof` dispatched → `fin_transfer_callback` runs → `process_fin_transfer_to_near` → `add_fin_transfer(T)` inserts T into `finalised_transfers` → `get_fast_transfer_status(T)` returns `None` (A's fast transfer not yet recorded) → **tokens sent to recipient (first payment)**.
3. A's `fast_fin_transfer_to_near_callback` runs → `add_fast_transfer(T)` succeeds (T is absent from `fast_transfers`) → **tokens sent to recipient again (second payment)**.

The recipient receives the bridged amount twice. The bridge contract loses one full transfer amount of tokens. This is a direct, unauthorized release of bridged assets.

### Likelihood Explanation

NEAR's async cross-contract call model makes this race window real and non-negligible. The `check_or_pay_ft_storage` call spans at least one receipt round-trip, during which any number of other transactions — including a concurrent `fin_transfer` proof submission — can be included and finalized. In a production bridge with multiple active relayers, both `fast_fin_transfer` and `fin_transfer` submissions for the same transfer are expected to occur in close temporal proximity. No malicious intent is required; the race is a natural consequence of concurrent honest relayer operation.

### Recommendation

Re-check `finalised_transfers` at the start of `fast_fin_transfer_to_near_callback`, before calling `add_fast_transfer` or sending any tokens:

```rust
pub fn fast_fin_transfer_to_near_callback(
    &mut self,
    fast_transfer: &FastTransfer,
    storage_payer: AccountId,
    relayer_id: AccountId,
) -> Promise {
    // Re-check: a concurrent fin_transfer may have finalized this transfer
    // in the window between the initial check and this callback.
    if self.is_unified_transfer_finalised(&fast_transfer.transfer_id) {
        // Refund storage_payer and return tokens
        env::panic_str(BridgeError::TransferAlreadyFinalised.to_string().as_str());
    }
    // ... rest of callback
}
```

This mirrors the fix described in the original report: iterate (check) the full state range rather than relying solely on the subset (`fast_transfers` / `dirtyItems`) that was touched in the current execution context.

### Proof of Concept

**State before attack:**
- Transfer T = `{origin_chain: Eth, origin_nonce: 42}` exists on Ethereum.
- `finalised_transfers` does not contain T.
- `fast_transfers` does not contain T.

**Step 1:** Relayer A submits `ft_transfer_call` with `BridgeOnTransferMsg::FastFinTransfer` for T (recipient = NEAR account X).
- `fast_fin_transfer` runs: `is_unified_transfer_finalised(T)` → `false` → passes.
- `check_or_pay_ft_storage` cross-contract call is dispatched. Execution pauses.

**Step 2 (in the async window):** Relayer B submits `fin_transfer` with a valid proof for T.
- `verify_proof` cross-contract call runs and returns.
- `fin_transfer_callback` runs → `process_fin_transfer_to_near`:
  - `add_fin_transfer(T)` → T inserted into `finalised_transfers`. ✓
  - `get_fast_transfer_status(fast_transfer.id())` → `None` (A's entry not yet in `fast_transfers`).
  - `send_tokens(token, X, amount)` → **X receives `amount` tokens. (Payment 1)**

**Step 3:** A's `fast_fin_transfer_to_near_callback` runs:
- `add_fast_transfer(fast_transfer, ...)` → T is absent from `fast_transfers` → **succeeds**.
- `send_tokens(token, X, amount_without_fee)` → **X receives tokens again. (Payment 2)**

**Result:** Account X holds `2 × amount_without_fee` tokens. The bridge has paid out twice for a single cross-chain transfer. [7](#0-6) [8](#0-7) [9](#0-8)
