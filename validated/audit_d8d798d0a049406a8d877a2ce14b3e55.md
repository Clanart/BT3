Audit Report

## Title
Missing Factory Validation in `_addLiquidity` Allows Malicious Pool to Drain Approved User Funds — (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

`MetricOmmPoolLiquidityAdder` accepts an arbitrary caller-supplied `pool` address and stores it as the authorised callback counterparty in transient storage without validating it against the `MetricOmmPoolFactory`. A malicious pool can exploit this to drain up to `maxAmountToken0`/`maxAmountToken1` of any ERC-20 tokens from a victim who has approved the adder. The contract's own NatSpec explicitly acknowledges the missing guard, and the sibling swap router (`MetricOmmSwapRouterBase`) demonstrates the correct pattern that is absent here.

## Finding Description

**Root cause:** `MetricOmmPoolLiquidityAdder` has no `factory` immutable and performs no `isPool` check before entering the transient pay context.

**Exact call path:**

1. Victim calls `addLiquidityExactShares(maliciousPool, ...)` or `addLiquidityWeighted(maliciousPool, ...)`. [1](#0-0) 

2. `_addLiquidity` calls `_setPayContext(pool, payer, maxAmountToken0, maxAmountToken1)`, storing the **unvalidated** malicious pool address as `expectedPool` in transient storage. [2](#0-1) 

3. `IMetricOmmPoolActions(pool).addLiquidity(...)` is called on the malicious pool. [3](#0-2) 

4. The malicious pool immediately re-enters `metricOmmModifyLiquidityCallback` with `KIND_PAY`. The only caller check is `msg.sender != expectedPool` — but `expectedPool` **is** the malicious pool, so the check passes trivially. [4](#0-3) 

5. The callback calls `IMetricOmmPool(msg.sender).getImmutables()` to obtain `token0`/`token1` — values the malicious pool fully controls — and immediately pulls those tokens from the victim via `pay(token0, payer, msg.sender, amount0Delta)`. [5](#0-4) 

**Why existing guards fail:** The `msg.sender != expectedPool` check is circular — it validates that the callback caller matches the pool stored in transient context, but that pool was itself set to the malicious address without any legitimacy check. The cap check (`amount0Delta > max0 || amount1Delta > max1`) only limits the drain to the victim's own slippage parameters, which can equal the victim's full approval allowance.

**Contrast with the swap router:** `MetricOmmSwapRouterBase` stores a `FACTORY` immutable and calls `_requireFactoryPool(pool)` — which calls `FACTORY.isPool(pool)` — before setting any transient callback context. This guard is entirely absent from `MetricOmmPoolLiquidityAdder`. [6](#0-5) 

The contract's own NatSpec at L19–21 explicitly documents the missing guard as a known limitation delegated to the caller, but no on-chain enforcement exists. [7](#0-6) 

## Impact Explanation

A victim who has approved `MetricOmmPoolLiquidityAdder` to spend their ERC-20 tokens loses up to `maxAmountToken0` units of any token the malicious pool declares as `token0`, and up to `maxAmountToken1` units of any token declared as `token1`. Both amounts are bounded only by the victim's slippage caps, which can equal the victim's full approval allowance. This is a direct, irreversible loss of user principal — a Critical/High impact under Sherlock thresholds.

## Likelihood Explanation

The attack requires: (1) deploying a malicious pool contract (permissionless, trivial on any EVM chain); (2) inducing the victim to call `addLiquidityExactShares` or `addLiquidityWeighted` with the malicious pool address. The second condition is achievable via a phishing UI, a compromised front-end, or a misleading pool listing, since no on-chain legitimacy check exists. Users who grant large standing approvals to the adder (a common convenience pattern) are at highest risk and can be drained in a single transaction.

## Recommendation

Add a factory immutable to `MetricOmmPoolLiquidityAdder` (mirroring `MetricOmmSwapRouterBase`) and validate the pool before entering transient context in `_addLiquidity`:

```solidity
// In _addLiquidity, before _setPayContext:
require(
    IMetricOmmPoolFactory(FACTORY).isPool(pool),
    "UnknownPool"
);
```

This mirrors the pattern already enforced in `MetricOmmSwapRouterBase._requireFactoryPool`. [8](#0-7) 

## Proof of Concept

```solidity
contract MaliciousPool {
    address victim;
    address adder;
    address stolenToken0;
    address stolenToken1;

    function addLiquidity(address, uint80, LiquidityDelta calldata, bytes calldata, bytes calldata)
        external returns (uint256, uint256)
    {
        // Re-enter adder — msg.sender == this == expectedPool, check passes
        IMetricOmmModifyLiquidityCallback(adder)
            .metricOmmModifyLiquidityCallback(
                MAX_AMOUNT_0,
                MAX_AMOUNT_1,
                abi.encode(uint8(1)) // KIND_PAY
            );
        return (MAX_AMOUNT_0, MAX_AMOUNT_1);
    }

    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = stolenToken0; // any token victim approved
        imm.token1 = stolenToken1;
    }
}

// Victim (tricked by phishing UI) calls:
adder.addLiquidityExactShares(
    address(maliciousPool),
    victim,
    0,
    deltas,
    1_000_000e6,  // maxAmountToken0 — victim's full USDC allowance
    500e18,       // maxAmountToken1 — victim's full WETH allowance
    ""
);
// Result: malicious pool receives 1 000 000 USDC + 500 WETH from victim.
```
