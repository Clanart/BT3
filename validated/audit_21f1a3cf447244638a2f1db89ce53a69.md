### Title
Sequential `currentOriginNonce` Combined with Sub-Finality Proof Acceptance Enables Re-Org-Based Unauthorized Mint and Permanent Fund Freezing — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `near/omni-prover/mpc-omni-prover/src/lib.rs`)

---

### Summary

`OmniBridge.sol` identifies every outbound transfer using a monotonically incrementing `currentOriginNonce` that is not derived from transfer content. The NEAR bridge records finalization keyed solely on `{origin_chain, origin_nonce}`. The `mpc-omni-prover` is hardcoded to accept proofs at `EvmFinality::Latest` for the Abstract chain and `StarknetFinality::AcceptedOnL2` for Starknet — both sub-finality levels that are susceptible to chain re-orgs. If a re-org reassigns a nonce to a different transfer after NEAR has already finalized the original, the replacement transfer is permanently unclaimable and the original recipient received unbacked tokens.

---

### Finding Description

**Root cause 1 — Sequential, content-independent nonce on EVM**

`OmniBridge.sol` increments a single global counter for every `initTransfer` and `initTransfer1155` call:

```solidity
// OmniBridge.sol line 381
currentOriginNonce += 1;
```

The nonce is emitted in the `InitTransfer` event and becomes the `origin_nonce` field that NEAR uses as the transfer identifier. It carries no commitment to the transfer's token, amount, recipient, or sender.

**Root cause 2 — Transfer ID on NEAR is purely `{chain, nonce}`**

`TransferMessage::get_transfer_id()` returns only the chain kind and the sequential nonce:

```rust
// near/omni-types/src/lib.rs line 578-583
pub const fn get_transfer_id(&self) -> TransferId {
    TransferId {
        origin_chain: self.get_origin_chain(),
        origin_nonce: self.origin_nonce,
    }
}
```

`add_fin_transfer` inserts this pair into `finalised_transfers` and panics if it is already present:

```rust
// near/omni-bridge/src/lib.rs line 2226-2231
fn add_fin_transfer(&mut self, transfer_id: &TransferId) -> NearToken {
    let storage_usage = env::storage_usage();
    require!(
        self.finalised_transfers.insert(transfer_id),
        BridgeError::TransferAlreadyFinalised.as_ref()
    );
```

**Root cause 3 — `mpc-omni-prover` accepts sub-finality proofs**

The prover is initialized with non-final confirmation levels for two chains:

```rust
// near/omni-prover/mpc-omni-prover/src/lib.rs line 57-61
finalities.insert(ChainKind::Abs, MpcFinality::Evm(EvmFinality::Latest));
finalities.insert(
    ChainKind::Strk,
    MpcFinality::Starknet(StarknetFinality::AcceptedOnL2),
);
```

`EvmFinality::Latest` is the tip of the canonical chain — not safe, not finalized. `StarknetFinality::AcceptedOnL2` is accepted on the Starknet sequencer but not yet proven on Ethereum L1. Both are re-org-susceptible. The `request_matches_finality` check enforces that the submitted proof's finality *matches* the configured level, meaning a relayer **must** submit at `Latest` for Abstract and at `AcceptedOnL2` for Starknet — there is no option to wait for a stronger guarantee.

**How the three roots combine**

Because the nonce is sequential and the proof is accepted at a pre-finality level, a re-org on the source chain can silently reassign a nonce that NEAR has already consumed.

---

### Impact Explanation

**Unauthorized mint (Critical):** Alice's `initTransfer` at nonce N is finalized on NEAR — tokens are minted or unlocked for Alice. The source chain then re-orgs and Alice's transaction is dropped. Alice holds tokens on NEAR that are no longer backed by any locked or burned collateral on the source chain. Bridge collateralization is broken.

**Permanent fund freezing (Critical):** After the re-org, a different transaction (Bob's) occupies nonce N on the source chain. When a relayer attempts to finalize Bob's transfer on NEAR, `add_fin_transfer` panics with `TransferAlreadyFinalised` because `{chain, N}` is already in `finalised_transfers`. Bob's tokens are permanently locked or burned on the source chain with no recovery path.

Both impacts are in the allowed scope: unauthorized mint of bridged assets and irrecoverable lock of user funds.

---

### Likelihood Explanation

- **Abstract chain (`EvmFinality::Latest`):** EVM chains at "latest" finality experience re-orgs routinely. A single-block re-org is sufficient. The relayer has no protocol-enforced waiting period before submitting the proof to NEAR. Any relayer that submits immediately upon seeing the event (the expected behavior for fast settlement) is exposed.
- **Starknet (`AcceptedOnL2`):** Starknet sequencer re-orgs (before L1 proof submission) have occurred historically. `AcceptedOnL2` is explicitly not L1-finalized.
- The scenario requires only that the relayer submits the proof before the re-org is detected — a normal race condition given that "Latest" finality is the configured level.

---

### Recommendation

1. **Use finalized confirmation levels.** Change the `mpc-omni-prover` initialization to require `EvmFinality::Finalized` (or at minimum `EvmFinality::Safe`) for the Abstract chain, and `StarknetFinality::AcceptedOnL1` for Starknet. This eliminates the re-org window.

2. **Bind the transfer ID to transfer content.** Derive the `TransferId` (or an additional content commitment) from a hash of the full transfer payload — token, amount, recipient, sender, and nonce — analogous to how `FastTransferId` is computed:
   ```rust
   // near/omni-types/src/lib.rs line 856-858
   pub fn id(&self) -> FastTransferId {
       FastTransferId(utils::sha256(&borsh::to_vec(self).unwrap()))
   }
   ```
   Applying the same pattern to `TransferId` means that even if a nonce is reused after a re-org, a proof for a different transfer payload will not collide with the already-finalized entry.

3. **Enforce a minimum confirmation depth in the relayer** as a defense-in-depth measure, even if the on-chain prover is tightened.

---

### Proof of Concept

**Setup:** Abstract chain (`ChainKind::Abs`) with `mpc-omni-prover` configured at `EvmFinality::Latest`.

1. **Block N:** Alice calls `initTransfer(USDC, 1000, "alice.near")` on Abstract chain. `currentOriginNonce` increments to 5. Event `InitTransfer(alice, USDC, 5, 1000, ...)` is emitted.

2. **Immediately:** Relayer observes the event at `EvmFinality::Latest` and submits a proof to NEAR's `fin_transfer`. `mpc-omni-prover` accepts it (finality matches `Latest`). `fin_transfer_callback` decodes `origin_nonce = 5`, calls `add_fin_transfer({Abs, 5})`, and mints 1000 USDC for Alice on NEAR.

3. **Re-org:** Block N is dropped. In the replacement block, Bob's `initTransfer(WETH, 2, "bob.near")` is included first. `currentOriginNonce` increments to 5 — Bob's transfer now owns nonce 5.

4. **Relayer attempts to finalize Bob's transfer:** Submits proof for `{Abs, 5}` (Bob's transfer) to NEAR. `add_fin_transfer` calls `self.finalised_transfers.insert({Abs, 5})` which returns `false` (already present). The `require!` macro panics with `TransferAlreadyFinalised`. Bob's WETH is permanently locked/burned on Abstract chain.

5. **Result:** Alice holds 1000 USDC on NEAR backed by nothing. Bob's 2 WETH is irrecoverably frozen. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
