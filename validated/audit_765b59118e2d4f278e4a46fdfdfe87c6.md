### Title
Too-Soft L1 Handler Fee Check Accepts Underpaying L1→L2 Messages - (File: `crates/blockifier/src/transaction/l1_handler_transaction.rs`)

---

### Summary

The blockifier's `L1HandlerTransaction` execution path checks only that `paid_fee_on_l1 != 0` before accepting an L1→L2 message. Any non-zero fee — even `Fee(1)` — passes the check regardless of the actual execution cost (`receipt.fee`). This is the direct sequencer analog of the CoverPool `received > 0` instead of `received == _amount` pattern: a "greater-than-zero" guard where an exact-or-sufficient comparison is required.

---

### Finding Description

In `execute_raw` for `L1HandlerTransaction`, after a successful execution and resource-bounds check, the code reads:

```rust
// TODO(Arni): Consider removing this check. It is covered by the starknet
// core contract.
let paid_fee = self.paid_fee_on_l1;
// For now, assert only that any amount of fee was paid.
// The error message still indicates the required fee.
if paid_fee == Fee(0) {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee {
            paid_fee,
            actual_fee: receipt.fee,
        },
    )));
}
``` [1](#0-0) 

The condition `paid_fee == Fee(0)` is the "too-soft" check. The `InsufficientFee` error variant carries both `paid_fee` and `actual_fee`, making it clear the intended invariant is `paid_fee >= actual_fee`, yet the guard only enforces `paid_fee > 0`. The test suite explicitly documents this relaxation:

```rust
// Today, we check that the paid_fee is positive, no matter what was the actual fee.
``` [2](#0-1) 

The `receipt.fee` (actual execution cost) is computed from gas prices and execution resources and is available at the point of the check, but is never compared against `paid_fee_on_l1`. [3](#0-2) 

---

### Impact Explanation

An L1→L2 message submitted with `paid_fee_on_l1 = Fee(1)` (1 wei) will be accepted and fully executed by the sequencer — state changes committed, events emitted, receipt generated — even when `receipt.fee` is orders of magnitude larger. The sequencer bears the full execution cost without compensation. Because the execution is committed before the fee check, the state diff, storage writes, and events are already finalised into the block when the (non-zero) fee passes. This constitutes an incorrect fee/resource accounting effect with direct economic impact on the sequencer, matching the **Critical** impact scope: *"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."*

---

### Likelihood Explanation

Any unprivileged user can send an L1→L2 message on Ethereum with an arbitrarily small non-zero `msg.value`. The Starknet core contract on L1 enforces a minimum fee, but that minimum is computed at message-submission time using L1 gas prices. L2 execution costs are denominated in L2 gas and priced at L2 gas prices, which can diverge significantly from the L1-side estimate. The comment "It is covered by the starknet core contract" acknowledges the reliance on L1 enforcement, but that enforcement uses a different pricing model and a different point in time, leaving a gap the sequencer's own check should close.

---

### Recommendation

Replace the zero-equality guard with a sufficient-fee guard:

```rust
// Before (too soft):
if paid_fee == Fee(0) { ... }

// After (correct):
if paid_fee < receipt.fee {
    return Err(TransactionExecutionError::TransactionFeeError(Box::new(
        TransactionFeeError::InsufficientFee {
            paid_fee,
            actual_fee: receipt.fee,
        },
    )));
}
```

This aligns the check with the semantics of `InsufficientFee` and closes the gap between L1-side fee enforcement and L2-side execution cost. If the design intent is truly to delegate all enforcement to the L1 core contract, the check should be removed entirely (as the TODO suggests) and that decision documented explicitly, rather than leaving a misleading partial guard in place.

---

### Proof of Concept

1. Craft an L1→L2 message targeting a storage-writing L1 handler contract. Set `msg.value = 1 wei` on L1 (satisfying the L1 core contract's non-zero minimum).
2. The sequencer picks up the message and constructs an `L1HandlerTransaction` with `paid_fee_on_l1 = Fee(1)`.
3. `execute_raw` runs the handler; `receipt.fee` is computed as, say, `Fee(50_000_000_000)` (50 Gwei equivalent in STRK).
4. `FeeCheckReport::check_all_gas_amounts_within_bounds` passes (resource bounds are checked against `l1_handler_max_amount_bounds`, not against `paid_fee`).
5. `execution_state.commit()` is called — state changes are now permanent.
6. The fee check `if paid_fee == Fee(0)` evaluates to `false` (since `paid_fee = 1`).
7. `Ok(l1_handler_tx_execution_info(...))` is returned; the transaction is included in the block with full state effect and zero fee charged (`receipt.fee` is zeroed out in `l1_handler_tx_execution_info`). [4](#0-3) 

The sequencer has executed and committed a transaction for which it received 1 wei instead of the actual cost, with no error returned.
