### Title
`env::signer_account_id()` (tx.origin analog) Used for Trusted-Relayer Authentication in `fast_fin_transfer` Enables Unauthorized Fast Transfer Execution and Fee Theft - (File: `near/omni-bridge/src/lib.rs`)

### Summary

`ft_on_transfer` in the NEAR omni-bridge contract uses `env::signer_account_id()` — the NEAR equivalent of Solidity's `tx.origin` — to authenticate the caller as a trusted relayer before executing `fast_fin_transfer`. An attacker who tricks any trusted relayer into calling a malicious contract can have that contract invoke `ft_transfer_call` on a registered bridge token, causing the bridge to see the trusted relayer as the `signer_id` and pass the `is_trusted_relayer` check, bypassing the relayer registration and staking requirement entirely.

### Finding Description

In `near/omni-bridge/src/lib.rs`, `ft_on_transfer` explicitly reads `env::signer_account_id()` and passes it as `signer_id` to both `fast_fin_transfer` and `utxo_fin_transfer`:

```rust
// We can't trust sender_id to pay for storage as it can be spoofed.
let signer_id = env::signer_account_id();
...
BridgeOnTransferMsg::FastFinTransfer(fast_fin_transfer_msg) => {
    self.fast_fin_transfer(token_id, amount, signer_id, fast_fin_transfer_msg)
}
``` [1](#0-0) 

Inside `fast_fin_transfer`, the only access-control gate is:

```rust
require!(self.is_trusted_relayer(&signer_id), "Relayer is not active");
``` [2](#0-1) 

In NEAR's execution model:
- `env::predecessor_account_id()` = the immediate calling contract (analogous to `msg.sender`)
- `env::signer_account_id()` = the original transaction signer across the entire call chain (analogous to `tx.origin`)

When a trusted relayer (Alice) calls any external contract, and that contract internally calls `ft_transfer_call` on a registered bridge token, the token contract calls `ft_on_transfer` on the bridge. At that point `env::signer_account_id()` still returns Alice's account ID, so `is_trusted_relayer(&signer_id)` passes even though Alice never intended to execute a fast transfer.

The comment in the code acknowledges that `sender_id` (the NEP-141 `ft_on_transfer` parameter) can be spoofed by the token contract, which is why `signer_account_id()` was chosen. However, this substitution introduces the symmetric `tx.origin` problem: any account in the call chain that is a trusted relayer will satisfy the check, regardless of whether it is the actual initiator of the bridge operation. [3](#0-2) 

### Impact Explanation

Once the trusted-relayer gate is bypassed, the attacker controls the entire `FastFinTransferMsg` payload, including:

- `transfer_id` — which pending cross-chain transfer to finalize
- `recipient` — where the fronted tokens are sent
- `relayer` — the account that will receive the fee when `claim_fee` is later called [4](#0-3) 

**Fee theft**: The attacker sets `relayer` to themselves, fronts the correct token amount to the legitimate recipient, and records themselves as the fee beneficiary. When the cross-chain proof is later submitted and `claim_fee` is called, the fee is paid to the attacker rather than the legitimate relayer who staked and registered.

**Unauthorized fast-transfer-to-other-chain**: For non-NEAR recipients, `fast_fin_transfer_to_other_chain` burns or locks the fronted tokens and emits a new `TransferMessage` with `origin_transfer_id` pointing to the attacker-created fast transfer. This creates an unbacked cross-chain transfer record that the protocol will attempt to settle. [5](#0-4) 

**Relayer staking bypass**: The trusted-relayer system requires registration and staking (`apply_for_trusted_relayer`). The bypass allows an unregistered, unstaked party to perform fast transfers, undermining the economic security model of the relayer set.

### Likelihood Explanation

Trusted relayers are active on-chain participants who regularly interact with DeFi protocols, token contracts, and other smart contracts. Any interaction where a relayer calls an external contract that subsequently calls `ft_transfer_call` on a registered bridge token — whether through a malicious airdrop, a yield aggregator, a DEX router, or a deliberately crafted honeypot — satisfies the precondition. The attacker only needs to hold enough of a registered bridge token to cover the fronted amount, which is economically rational when the fee exceeds the cost of acquiring those tokens.

### Recommendation

Replace `env::signer_account_id()` with `env::predecessor_account_id()` for the relayer identity check. The `predecessor_account_id` is the token contract that called `ft_on_transfer`; the actual human/relayer account is the `sender_id` parameter passed by the token. For the trusted-relayer check, validate `sender_id` directly (the account that called `ft_transfer_call` on the token), since that is the party who chose to initiate the bridge operation. The existing comment about storage spoofing applies only to storage-balance deductions, not to access-control identity.

Concretely, in `ft_on_transfer`, use `sender_id` (not `signer_id`) for the `is_trusted_relayer` check inside `fast_fin_transfer`, and keep `signer_id` only for storage-balance accounting where the spoofing concern is valid.

### Proof of Concept

1. Attacker deploys `MaliciousContract` holding a balance of a registered bridge token (e.g., `wbtc.bridge.near`).
2. Attacker calls `MaliciousContract.trigger(relayer_account)` — a public function that does nothing except wait to be called by a trusted relayer.
3. Trusted relayer Alice calls `MaliciousContract.trigger()` for any reason (airdrop claim, staking, etc.).
4. Inside `trigger`, `MaliciousContract` calls:
   ```
   wbtc.bridge.near.ft_transfer_call(
     receiver_id = omni_bridge,
     amount = <transfer_amount + fee>,
     msg = FastFinTransfer { transfer_id: <victim_transfer_id>, recipient: <correct_recipient>, relayer: attacker, ... }
   )
   ```
5. `wbtc.bridge.near` calls `omni_bridge.ft_on_transfer(sender_id=MaliciousContract, amount=..., msg=...)`.
6. Inside `ft_on_transfer`: `signer_id = env::signer_account_id()` = Alice (trusted relayer).

### Title
`env::signer_account_id()` (NEAR `tx.origin` Analog) Used for Trusted-Relayer Authentication in `fast_fin_transfer` Enables Bypass via Malicious Intermediary Contract - (File: `near/omni-bridge/src/lib.rs`)

### Summary

`ft_on_transfer` in the NEAR omni-bridge contract resolves the caller identity for the trusted-relayer check using `env::signer_account_id()` — NEAR's direct analog of Solidity's `tx.origin`. Because `signer_account_id()` reflects the original transaction signer rather than the immediate predecessor, any malicious contract that a trusted relayer interacts with can re-enter the bridge via `ft_transfer_call` and pass the `is_trusted_relayer` guard in `fast_fin_transfer`, without the relayer's knowledge or consent.

---

### Finding Description

In `ft_on_transfer`, the code explicitly reads the original transaction signer to use as the relayer identity:

```rust
// We can't trust sender_id to pay for storage as it can be spoofed.
let signer_id = env::signer_account_id();   // ← tx.origin equivalent
``` [6](#0-5) 

This `signer_id` is then forwarded directly into `fast_fin_transfer`:

```rust
BridgeOnTransferMsg::FastFinTransfer(fast_fin_transfer_msg) => {
    self.fast_fin_transfer(token_id, amount, signer_id, fast_fin_transfer_msg)
}
``` [7](#0-6) 

Inside `fast_fin_transfer`, the only access-control gate is:

```rust
require!(self.is_trusted_relayer(&signer_id), "Relayer is not active");
``` [2](#0-1) 

Because `signer_id` is `env::signer_account_id()` (the original transaction signer, not the immediate predecessor), any contract that a trusted relayer calls can itself call `ft_transfer_call` on a registered bridge token, causing the bridge's `ft_on_transfer` to fire with `signer_id` equal to the trusted relayer's account — even though the trusted relayer never intended to interact with the bridge in that transaction.

By contrast, `utxo_fin_transfer` correctly uses `sender_id` (the immediate predecessor) for its connector check:

```rust
require!(
    sender_id == &config.connector,
    BridgeError::SenderIsNotConnector.as_ref()
);
``` [8](#0-7) 

This shows the developers are aware of the `sender_id`/`signer_id` distinction but applied the wrong identity for the relayer gate in `fast_fin_transfer`.

---

### Impact Explanation

Once the trusted-relayer check is bypassed, the attacker controls the entire `FastFinTransferMsg` payload, including:

- `transfer_id` — which cross-chain transfer to finalize
- `recipient` — who receives the fronted tokens
- `relayer` — who is recorded as the relayer entitled to claim the fee

The attacker fronts their own tokens (via `ft_transfer_call`) and records a fast transfer with `relayer = attacker_account`. When the legitimate cross-chain proof is later submitted via `claim_fee`, the bridge reads the stored `relayer` field and pays the fee to the attacker:

```rust
let fee_recipient = fin_transfer.fee_recipient.unwrap_or_else(|| { ... });
require!(fee_recipient == *predecessor_account_id, ...);
...
self.send_fee_internal(&transfer_message, fee_recipient, fee)
``` [9](#0-8) 

The attacker recovers their fronted principal plus steals the fee that should have gone to the legitimate relayer. This is direct misdirection of bridge fee value — matching the allowed High impact: **"fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

Additionally, the attacker can create fast-transfer state records for transfers that were never legitimately initiated, corrupting the `fast_transfers` map and potentially blocking legitimate fee claims for those transfer IDs.

---

### Likelihood Explanation

The attack requires a trusted relayer to call a malicious contract in the same transaction. This is realistic because:

1. Trusted relayers are active on-chain participants who interact with DeFi protocols, airdrop contracts, token claim contracts, etc.
2. The malicious contract's re-entry into the bridge is invisible to the relayer — they see only their own top-level call.
3. The attacker only needs to deploy a contract that, when called, invokes `ft_transfer_call` on a registered bridge token with a crafted `FastFinTransfer` message. The attacker must hold a small amount of the registered token to front the transfer.
4. The attacker can monitor the mempool for pending cross-chain transfers and race to finalize them with `relayer = attacker` before the legitimate relayer does.

Likelihood: **Medium** — requires social engineering or opportunistic interaction with a trusted relayer, but the technical execution is straightforward once that condition is met.

---

### Recommendation

Replace `env::signer_account_id()` with `env::predecessor_account_id()` for the relayer identity used in `fast_fin_transfer`. The `predecessor_account_id()` is the account that directly called `ft_transfer_call` on the token contract, which is the actual party initiating the bridge action.

The comment in the code ("We can't trust sender_id to pay for storage as it can be spoofed") is correct for the storage-payment use case — `sender_id` is a parameter passed by the token contract and can be set to any value. However, `predecessor_account_id()` (the direct caller of `ft_transfer_call`) is not spoofable and is the correct identity to use for access control. The storage-payment logic can continue to use `signer_account_id()` if desired, but the relayer authentication must use `predecessor_account_id()`.

```rust
// For relayer authentication, use the direct caller, not the original signer
let relayer_id = env::predecessor_account_id(); // immediate caller of ft_transfer_call
// signer_id can still be used for storage payment if needed
let signer_id = env::signer_account_id();
```

### Proof of Concept

**Setup:**
- `alice.near` is a registered trusted relayer.
- `usdc.near` is a registered bridge token.
- A cross-chain transfer of 1000 USDC from EVM to NEAR is pending (transfer ID `T`).

**Attack steps:**

1. Attacker deploys `evil.near`. Its callable function does:
   ```
   ft_transfer_call(
     receiver_id = omni_bridge.near,
     amount = 1000,
     msg = FastFinTransfer { transfer_id: T, recipient: victim.near, relayer: attacker.near, fee: 10, ... }
   )
   ```
   on `usdc.near`.

2. Attacker lures `alice.near` into calling `evil.near` (e.g., via an airdrop claim, a DeFi interaction, or any other pretext).

3. `evil.near` executes `ft_transfer_call` on `usdc.near` with the crafted message.

4. `usdc.near` calls `omni_bridge.near::ft_on_transfer(sender_id=evil.near, amount=1000, msg=FastFinTransfer{...})`.

5. Inside `ft_on_transfer`:
   - `token_id = predecessor_account_id() = usdc.near` ✓ (registered token)
   - `signer_id = signer_account_id() = alice.near` ✓ (trusted relayer — bypass succeeds)

6. `fast_fin_transfer` passes the `is_trusted_relayer` check, records the fast transfer with `relayer = attacker.near`, and sends 990 USDC to `victim.near`.

7. When the legitimate EVM proof is submitted via `claim_fee`, the bridge pays the 10 USDC fee to `attacker.near` instead of the legitimate relayer.

    pub fn ft_on_transfer(&mut self, sender_id: AccountId, amount: U128, msg: String) {
        let token_id = env::predecessor_account_id();
        let parsed_msg: BridgeOnTransferMsg = serde_json::from_str(&msg)
            .or_else(|_| serde_json::from_str(&msg).map(BridgeOnTransferMsg::InitTransfer))
            .near_expect(BridgeError::ParseMsg);

        // We can't trust sender_id to pay for storage as it can be spoofed.
        let signer_id = env::signer_account_id();
        let promise_or_promise_index_or_value = match parsed_msg {
            BridgeOnTransferMsg::InitTransfer(init_transfer_msg) => {
                self.init_transfer(sender_id, signer_id, token_id, amount, init_transfer_msg)
            }
            BridgeOnTransferMsg::FastFinTransfer(fast_fin_transfer_msg) => {
                self.fast_fin_transfer(token_id, amount, signer_id, fast_fin_transfer_msg)
            }
            BridgeOnTransferMsg::UtxoFinTransfer(utxo_fin_transfer_msg) => self.utxo_fin_transfer(
                token_id,
                amount,
                &signer_id,
                &sender_id,
                utxo_fin_transfer_msg,
            ),
            BridgeOnTransferMsg::SwapMigratedToken => {
                self.swap_migrated_token(sender_id, token_id, amount)
                    .detach();
                PromiseOrPromiseIndexOrValue::Value(U128(0))
            }
        };

        promise_or_promise_index_or_value.as_return();
    }
