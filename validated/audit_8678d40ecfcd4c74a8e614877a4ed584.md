### Title
`deployToken` Signed Payload Lacks Chain-ID Binding, Enabling Cross-Chain Replay to Unauthorized Token Deployment — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`, `starknet/src/omni_bridge.cairo`)

---

### Summary

The `deployToken` / `deploy_token` functions on EVM, Solana, and Starknet verify a NEAR MPC ECDSA signature over a Borsh-encoded metadata payload that contains **no chain identifier**. The same NEAR MPC key is used across all chains. A signature produced for one chain is therefore byte-for-byte valid on every other chain. By contrast, `finTransfer` / `finalize_transfer` on all three chains correctly embeds `omniBridgeChainId` / `SOLANA_OMNI_BRIDGE_CHAIN_ID` in the signed preimage. The `deployToken` path is the only signed operation that omits this binding.

---

The Borsh encoding of `[Metadata, token, name, symbol, decimals]` is **identical** across EVM, Solana, and Starknet for the same token metadata, because all three implement the same Borsh wire format. The NEAR MPC key (`nearBridgeDerivedAddress` / `derived_near_bridge_address` / `omni_bridge_derived_address`) is the same key used to sign for all chains. Therefore `keccak256(borshEncoded)` is the same value on all three chains for the same token, and a single signature satisfies all three verifiers simultaneously.

---

### Impact Explanation

**Unauthorized cross-chain token deployment (High).**

An attacker who observes a valid `deployToken` signature on chain A (e.g., from the EVM mempool or after block inclusion) can immediately submit it to chain B and chain C:

- **EVM**: `new ERC1967Proxy(...)` deploys a non-deterministic contract address. The `nearToEthToken[metadata.token]` mapping is set. The NEAR bridge, which did not authorize this EVM deployment, does not know the deployed address. Any subsequent legitimate attempt to deploy the same token on EVM reverts with `ERR_TOKEN_EXIST`. The token's EVM deployment slot is **permanently blocked** — the NEAR bridge can never deploy the token on EVM through the normal flow. This matches the "permanent freezing / irrecoverable lock" impact class.

- **Solana / Starknet**: Token deployment is deterministic (PDA seed / `keccak(token_id).low` salt), so the attacker-deployed token lands at the same address the NEAR bridge would have used. The NEAR bridge may then recognize the deployment via event monitoring and begin signing transfers to it — meaning an attacker can force the NEAR bridge to activate a token on a chain before the NEAR side has set up the corresponding token mapping or fee configuration, enabling premature or unauthorized cross-chain transfers.

---

### Likelihood Explanation

- The `deployToken` signature is submitted on-chain and is publicly visible in the mempool or block data.
- No privileged access is required; any observer can extract the `signatureData` bytes from a pending or confirmed `deployToken` transaction on chain A and resubmit them on chain B.
- The NEAR MPC key is shared across all chains by design, so no key compromise is needed.
- The attack is a single transaction on the target chain.

---

### Recommendation

Include the destination chain identifier in the `deployToken` signed preimage, matching the pattern already used by `finTransfer`:

**EVM:**
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // add chain binding
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

**Solana (`serialize_for_near`):**
```rust
IncomingMessageType::Metadata.serialize(&mut writer)?;
writer.write_all(&[SOLANA_OMNI_BRIDGE_CHAIN_ID])?;  // add chain binding
self.serialize(&mut writer)?;
```

**Starknet (`MetadataPayload::to_borsh`):**
```cairo
fn to_borsh(self: @MetadataPayload, chain_id: u8) -> ByteArray {
    borsh_bytes.append_byte(PayloadType::Metadata.into());
    borsh_bytes.append_byte(chain_id);   // add chain binding
    ...
}
```

The NEAR side must include the same chain ID byte when constructing the payload before signing, so that signatures are chain-scoped.

### Proof of Concept

1. NEAR bridge signs a `deployToken` for EVM (Ethereum mainnet) for token `"usdc.near"` with name `"USD Coin"`, symbol `"USDC"`, decimals `6`. The signature `sig` is submitted in a transaction to `OmniBridge.deployToken(sig, payload)` on Ethereum.

2. Attacker observes the Ethereum transaction (mempool or block). Extracts `sig` and `payload`.

3. Attacker calls `deploy_token(sig, payload)` on the Solana `bridge_token_factory` program. `DeployTokenPayload::serialize_for_near` produces `[0x01, borsh("usdc.near"), borsh("USD Coin"), borsh("USDC"), 0x06]` — identical bytes to what the EVM contract hashed. `verify_signature` recovers the same NEAR MPC address. The call succeeds. [3](#0-2) 

4. Attacker calls `deploy_token(sig, payload)` on the Starknet `OmniBridge`. `MetadataPayload::to_borsh()` produces the same bytes. `_verify_borsh_signature` passes. The call succeeds. [5](#0-4) 

5. The token is now deployed on Solana and Starknet without NEAR authorization for those chains. On EVM, if the attacker had submitted to a second EVM deployment (e.g., a testnet or a second EVM chain supported by the bridge) before the NEAR bridge did, the `nearToEthToken` mapping is set and the legitimate deployment is permanently blocked with `ERR_TOKEN_EXIST`.

    fn serialize_for_near(&self, params: Self::AdditionalParams) -> Result<Vec<u8>> {
        let mut writer = BufWriter::new(Vec::with_capacity(DEFAULT_SERIALIZER_CAPACITY));
        // 0. prefix
        IncomingMessageType::InitTransfer.serialize(&mut writer)?;
        // 1. destination_nonce
        self.destination_nonce.serialize(&mut writer)?;
        // 2. transfer_id
        writer.write_all(&[self.transfer_id.origin_chain])?;
        self.transfer_id.origin_nonce.serialize(&mut writer)?;
        // 3. token
        writer.write_all(&[SOLANA_OMNI_BRIDGE_CHAIN_ID])?;

        self.amount.serialize(&mut writer)?;

        params.1.serialize(&mut writer)?;
        // 6. fee_recipient
        self.fee_recipient.serialize(&mut writer)?;

        writer
            .into_inner()

    fn to_borsh(self: @MetadataPayload) -> ByteArray {
        let mut borsh_bytes: ByteArray = Default::default();
        borsh_bytes.append_byte(PayloadType::Metadata.into());
        borsh_bytes.append(@borsh::encode_byte_array(self.token));
