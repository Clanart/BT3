### Title
Bootstrap Declare Bypass via Redundant `version == 3` Guard Allows Unauthorized Class Declaration — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_declare_transaction` contains a "bootstrap" fast-path that skips all account validation (nonce, signature, fee). One of its three guard conditions — `tx_info.version == 3` — is structurally always true because `get_account_tx_common_fields` hardcodes `version=3`. This is the direct analog of the `tokenState.setState` redundant-check bug: a guard that appears to restrict the bypass is in fact a tautology, leaving the path protected only by two attacker-controlled felt values (`sender_address` and `nonce`). Any unprivileged transaction sender can trigger the path and write an arbitrary `(class_hash → compiled_class_hash)` entry into the class-changes dict without any signature or account verification.

---

### Finding Description

In `execute_declare_transaction`, after computing the transaction hash and filling `tx_info`, the OS checks:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

The guard `tx_info.version == 3` is structurally dead as a security control. `tx_info.version` is set from `common_tx_fields.version`, which is hardcoded to the literal `3` inside `get_account_tx_common_fields`:

```cairo
tempvar common_tx_fields = new CommonTxFields(
    tx_hash_prefix=tx_hash_prefix,
    version=3,
    ...
``` [2](#0-1) 

`fill_account_tx_info` then copies this directly into `tx_info.version`:

```cairo
assert [tx_info_dst] = TxInfo(
    version=common_tx_fields.version,
    ...
``` [3](#0-2) 

So `tx_info.version == 3` is always satisfied for every declare transaction that reaches this point. The effective guard reduces to only two conditions:

| Condition | Attacker-controlled? |
|---|---|
| `sender_address == 'BOOTSTRAP'` | Yes — a felt literal; attacker sets it in the transaction |
| `tx_info.nonce == 0` | Yes — attacker sets nonce field to 0 |
| `tx_info.version == 3` | **Always true** — hardcoded, redundant |
| `max_possible_fee == 0` | Yes — set all resource bounds to 0 |

`compute_max_possible_fee` returns 0 when all three resource-bound `max_amount` fields are zero, which is a valid transaction encoding:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
``` [4](#0-3) 

When the bypass fires, the OS skips `check_and_increment_nonce`, `__validate_declare__`, and `charge_fee`, and directly writes the attacker-supplied `(class_hash, compiled_class_hash)` pair into `contract_class_changes`.

---

### Impact Explanation

**Critical — Direct loss of funds.**

An attacker who successfully triggers the bypass can register an arbitrary `compiled_class_hash` for any previously-undeclared `class_hash`. Because `prev_value=0` only prevents overwriting an already-declared class, the attacker can inject a new class whose Sierra bytecode is valid (passing `finalize_class_hash`) but whose compiled (Casm) hash points to malicious logic. Any contract subsequently deployed from that class hash executes the attacker's code. If users or protocols interact with such a contract (e.g., a token, a bridge, a vault), funds can be drained directly.

Additionally, a mismatched `(class_hash, compiled_class_hash)` pair corrupts the class commitment tree, which can cause proof verification to fail on L1, halting the network.

---

### Likelihood Explanation

The attacker entry path requires only submitting a well-formed declare transaction to the sequencer with:
- `sender_address` = felt encoding of `'BOOTSTRAP'` (a known constant)
- `nonce` = 0
- All resource bounds = 0

No privileged key, no operator access, no Sybil attack is needed. The felt value of `'BOOTSTRAP'` is public and fixed. The sequencer may apply mempool-level heuristics, but the OS itself enforces no additional restriction, so any block that includes such a transaction will produce a valid proof that L1 accepts.

---

### Recommendation

The `tx_info.version == 3` guard should be removed as it is structurally redundant (analogous to the `tokenState.setState` issue where dead branches created false security). More importantly, the bootstrap path must be hardened:

1. **Remove the redundant version check** to make the actual guard surface explicit.
2. **Restrict `sender_address == 'BOOTSTRAP'` at the protocol level** — either by requiring a sequencer-signed attestation, or by removing the bootstrap path entirely once the network is live and replacing it with a governance-controlled mechanism.
3. **Add a block-number or epoch gate** so the bootstrap path is only reachable during a defined initialization window (e.g., `block_number == 0`).
4. **Require a non-zero nonce or a sequencer signature** so the path cannot be triggered by an arbitrary external transaction.

---

### Proof of Concept

1. Attacker constructs a Sierra class with valid structure so `finalize_class_hash` succeeds, but with a `compiled_class_hash` pointing to malicious Casm.
2. Attacker submits a v3 declare transaction with:
   - `sender_address = 0x424f4f545354524150` (felt encoding of `'BOOTSTRAP'`)
   - `nonce = 0`
   - `L1_gas.max_amount = 0`, `L2_gas.max_amount = 0`, `L1_data_gas.max_amount = 0`
   - `tip = 0`
3. The sequencer includes the transaction in a block.
4. The OS evaluates the bootstrap condition: `sender_address == 'BOOTSTRAP'` ✓, `nonce == 0` ✓, `version == 3` ✓ (always), `max_possible_fee == 0` ✓.
5. The OS executes `dict_update(key=class_hash, prev_value=0, new_value=malicious_compiled_class_hash)` and returns — no `__validate_declare__` is called, no fee is charged, no nonce is incremented.
6. The malicious class is now part of the committed state. Any contract deployed from it runs the attacker's Casm.

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
