Audit Report

## Title
Unattributed `address(this).balance` in `pay()` and unrestricted `refundETH()` allow theft of stranded ETH - (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` uses `address(this).balance` — the router's aggregate native ETH balance — rather than the current caller's contributed `msg.value`. ETH stranded by any prior caller (e.g., excess `msg.value` from a payable entry point where `refundETH()` was omitted) is silently consumed by the next caller's WETH swap or drained outright via the access-control-free `refundETH()`. This results in direct loss of user principal with no protocol-level barrier.

## Finding Description
In `PeripheryPayments.sol` lines 73–84, the WETH branch of `pay()` reads `address(this).balance` as `nativeBalance`:

```solidity
uint256 nativeBalance = address(this).balance;   // total router balance
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
```

This is the router's **total** ETH balance, not the ETH the current caller sent. ETH accumulates whenever a payable entry point (`multicall`, `exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`) receives more `msg.value` than the swap consumes and the caller omits `refundETH()`. The test at `MetricOmmSimpleRouter.native.t.sol` lines 106–133 explicitly documents this: it sends `2 ether` for a `1_000`-wei swap and must include `refundETH()` as a second multicall step to recover the remainder. Without that step, `~2 ether - 1_000 wei` remains on the router indefinitely.

`refundETH()` at lines 58–63 has no access control:

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // no caller check
    }
}
```

Any address can call this and receive the entire router ETH balance. The same `pay()` is inherited by `MetricOmmPoolLiquidityAdder` (lines 172–177), extending the attack surface to WETH-denominated liquidity additions.

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only prevents accidental plain ETH transfers; it does not prevent stranding via `msg.value` on payable functions.

## Impact Explanation
Direct loss of user principal. Two theft paths exist:

1. **Direct drain** — attacker calls `refundETH()` and receives the entire router ETH balance at zero cost.
2. **Free swap** — attacker calls `exactInputSingle(tokenIn=WETH, amountIn=X)` with zero ETH sent and zero WETH approved; `pay()` deposits the victim's stranded ETH as WETH and settles the swap, giving the attacker full swap output at zero personal cost.

Loss magnitude equals the stranded ETH amount, which can be arbitrarily large (e.g., a user who sends `amountInMaximum` ETH for an exact-output swap and omits `refundETH()`). This meets the Critical/High direct loss of user principal threshold.

## Likelihood Explanation
- `exactOutputSingle` and `exactOutput` are `payable` and commonly called with excess ETH for slippage headroom; if the caller does not wrap the call in a `multicall` ending with `refundETH()`, the surplus is stranded.
- `multicall` itself is `payable`; any sub-call that uses less than the forwarded `msg.value` leaves a residue.
- No special role or privileged access is required — any unprivileged caller can invoke `refundETH()`.
- An attacker can monitor the mempool for transactions that strand ETH and immediately call `refundETH()` in the next block, or sandwich the victim's transaction.
- The pattern is repeatable and requires no setup beyond observing a victim transaction.

## Recommendation
Track per-transaction ETH attribution using transient storage (EIP-1153). At the start of each payable entry point, record `msg.value` in a transient slot keyed to the call. In `pay()`, consume only from that recorded budget rather than from `address(this).balance`, and zero the slot before returning. This ensures each caller's ETH is isolated.

Alternatively, add a post-condition check in `multicall` that reverts if `address(this).balance > 0` after all sub-calls complete, forcing callers to always include `refundETH()` when forwarding ETH.

## Proof of Concept
```solidity
// Step 1 – Alice strands ETH (omits refundETH)
router.multicall{value: 2 ether}(
    [abi.encodeWithSelector(router.exactInputSingle.selector,
        ExactInputSingleParams({
            tokenIn: WETH, amountIn: 1_000, ...
        })
    )]
    // No refundETH() — ~2 ether - 1_000 wei remains on router
);

// Step 2a – Bob steals via refundETH (zero cost)
router.refundETH();   // Bob receives ~2 ether - 1_000 wei

// Step 2b – Alternatively, Bob gets a free swap
// Bob sends 0 ETH, has 0 WETH approved
router.exactInputSingle(ExactInputSingleParams({
    tokenIn: WETH,
    amountIn: 1 ether,   // ≤ stranded balance
    recipient: bob,
    ...
}));
// pay() sees address(this).balance >= 1 ether, deposits Alice's ETH,
// Bob receives swap output at zero personal cost
```
