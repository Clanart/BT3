### Title
Rebaseable Token Accrued Rewards Permanently Locked in EVM OmniBridge — No Balance-Divergence Recovery Path - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.sol` locks native ERC20 tokens at `address(this)` during `initTransfer` and releases exactly the proof-recorded amount during `finTransfer`. For rebaseable tokens (e.g., stETH, aTokens), the contract's actual token balance grows between lock and unlock, but the release path uses only the original proof amount. The accrued surplus is permanently irrecoverable: there is no rescue, sweep, or balance-sync function in the contract.

---

### Finding Description

**Lock path** — `initTransfer` (lines 407–411):

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // recorded in the emitted event / proof
);
```

The `amount` is emitted in `BridgeTypes.InitTransfer` and later signed by the NEAR MPC into a `TransferMessagePayload`.

**Unlock path** — `finTransfer` (lines 351–354):

```solidity
IERC20(payload.tokenAddress).safeTransfer(
    payload.recipient,
    payload.amount  // == original deposit amount, from the signed proof
);
```

`payload.amount` is the value that was locked at deposit time. The contract never queries `balanceOf(address(this))` and has no stored-balance accounting that could be updated to reflect rebasing. Any balance increase that occurred between the `initTransfer` and `finTransfer` calls is silently left in the contract.

There is no `rescue`, `sweep`, `emergencyWithdraw`, or any other function in `OmniBridge.sol` that would allow recovery of surplus token balance.

The NEAR bridge side has an analogous issue: `locked_tokens` in `near/omni-bridge/src/token_lock.rs` (lines 71–94) tracks only the deposited amount. `unlock_tokens` enforces `available >= amount` (a floor check), so users can withdraw their principal, but any rebased surplus held by the NEAR bridge contract is also unrecoverable through any existing code path.

---

### Impact Explanation

Every unit of rebase yield (stETH rewards, aToken interest, etc.) that accrues while the token is held by `OmniBridge` is permanently frozen in the contract. There is no on-chain path to recover it. This constitutes an irrecoverable lock of user/protocol funds in the bridge vault flow.

Matches: **Critical/High — Permanent freezing, irrecoverable lock of user or protocol funds in bridge or vault flows.**

---

### Likelihood Explanation

`logMetadata` is permissionless — any caller can register any ERC20 token address with the bridge. `initTransfer` accepts any registered token. Rebaseable tokens (stETH, wstETH unwrapped variants, Aave aTokens) are widely used on Ethereum mainnet. A user who bridges such a token triggers the condition automatically; no special attacker action is required beyond normal bridge usage. The longer the token remains locked (e.g., waiting for a cross-chain finalization), the larger the irrecoverable surplus.

---

### Recommendation

1. **EVM**: Record the actual received balance delta (pre/post `balanceOf`) in `initTransfer` rather than trusting the caller-supplied `amount`. Alternatively, add an admin-callable `rescueTokens(address token, address to, uint256 amount)` function that can sweep surplus balance above the sum of all pending locked amounts.
2. **NEAR**: Add a DAO-gated function to transfer any token balance held by the bridge that exceeds the sum of all `locked_tokens` entries for that token.

---

### Proof of Concept

1. Alice calls `logMetadata(stETH_address)` to register stETH.
2. NEAR side deploys the corresponding bridge token; Alice calls `initTransfer(stETH, 1000e18, ...)`. The contract receives 1000 stETH and emits `InitTransfer` with `amount = 1000e18`.
3. NEAR MPC signs a `TransferMessagePayload` with `amount = 1000e18`.
4. 30 days pass. stETH rebases; the contract now holds 1010 stETH.
5. Alice (or a relayer) calls `finTransfer(sig, payload)` where `payload.amount = 1000e18`. The contract sends Alice 1000 stETH.
6. 10 stETH remain in `OmniBridge`. No function exists to withdraw them. They are permanently locked. [1](#0-0) [2](#0-1) [3](#0-2)

```rust
    fn unlock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(available) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        require!(
            available >= amount,
            TokenLockError::InsufficientLockedTokens.as_ref()
        );

        let remaining = available - amount;
        self.locked_tokens.insert(&key, &remaining);

        LockAction::Unlocked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }
```
