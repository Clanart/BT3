### Title
Unverified Hint-Provided Block Hash Written to State Enables Prover Manipulation of `get_block_hash` Syscall Results - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/os_utils.cairo`)

---

### Summary

The `write_block_number_to_block_hash_mapping` function in `os_utils.cairo` writes a block hash into the `BLOCK_HASH_CONTRACT_ADDRESS` (0x1) contract storage using a value that is **entirely hint-provided and explicitly not enforced by the OS**. The OS itself acknowledges this with a comment and a pending TODO. Because the OS proof does not constrain the stored hash to match the actual historical block hash, any prover can write an arbitrary value. Contracts that call the `get_block_hash` syscall to obtain historical block hashes for randomness or security-critical logic will receive a prover-controlled value, enabling direct theft of funds.

---

### Finding Description

**Vulnerability class:** Commitment/output flaw — a protocol-level parameter (`STORED_BLOCK_HASH_BUFFER`) combined with a missing enforcement constraint allows the prover to freely choose the value stored as a historical block hash.

**Step 1 — The buffer constant.**

`STORED_BLOCK_HASH_BUFFER = 10` is defined in `constants.cairo`: [1](#0-0) 

This constant controls how many blocks back the OS records a block hash. When processing block `N`, the OS writes the hash of block `N - 10` into the dedicated mapping contract.

**Step 2 — The hash is hint-provided and unverified.**

In `write_block_number_to_block_hash_mapping` (`os_utils.cairo`): [2](#0-1) 

The comment reads: *"Currently, the block hash mapping is not enforced by the OS."* The `old_block_hash` local variable is populated exclusively by the hint `%{ GetOldBlockNumberAndHash %}`. There is no Cairo-level assertion, range check, or cryptographic constraint that ties `old_block_hash` to any previously committed block header. The value is then written directly into state: [3](#0-2) 

**Step 3 — The unverified value is committed to state and readable by contracts.**

The `dict_update` call at lines 79–83 commits this arbitrary value into `contract_state_changes` under `BLOCK_HASH_CONTRACT_ADDRESS`. Any contract that subsequently calls the `get_block_hash` syscall for a block number in the range `[N - STORED_BLOCK_HASH_BUFFER]` will read back this prover-supplied value as if it were the canonical block hash. [4](#0-3) 

**Step 4 — The TODO confirms this is a known incomplete enforcement.** [5](#0-4) 

The `TODO(Yoni, 1/1/2026): output this hash` indicates the OS was designed to eventually verify and output this hash as part of the proof, but currently does not. The enforcement gap is not a design choice — it is an unimplemented constraint.

---

### Impact Explanation

**Impact: Critical — Direct loss of funds.**

Any on-chain contract that uses `get_block_hash` as a source of randomness (e.g., lottery, NFT rarity, gaming outcomes, commit-reveal schemes) is fully controllable by the prover. The prover can:

1. Observe which block hash value would produce a favorable outcome for a target contract.
2. Supply that exact value as `old_block_hash` in the hint.
3. Generate a valid STARK proof that includes the manipulated mapping.
4. The L1 verifier accepts the proof because the OS does not constrain the stored hash.

The result is that the prover can deterministically steer the output of `get_block_hash` to drain funds from any contract that relies on it for security.

---

### Likelihood Explanation

The prover is the entity that generates the execution trace and supplies all hints. In StarkNet's proving model, the OS is the trust anchor — it is supposed to enforce correctness without trusting the prover. Because the OS explicitly does not enforce the block hash value, **every proof generated** for every block silently accepts a prover-chosen block hash. The attack requires no special access beyond the ability to run the prover, which is the normal operational role. The likelihood is high for any deployment where contracts use `get_block_hash` for security-sensitive logic.

---

### Recommendation

The OS must compute and constrain the stored block hash rather than accepting it from a hint. Specifically:

1. The `old_block_hash` must be derived from a previously committed and verified block header, not from an unconstrained hint.
2. The stored hash should be included in the OS output (as the TODO acknowledges) so that the L1 verifier can cross-check it against the on-chain block hash registry.
3. Until enforcement is implemented, `STORED_BLOCK_HASH_BUFFER` should be treated as a security-critical parameter. Increasing it alone does not fix the root cause (the missing enforcement), but the buffer value of 10 is also insufficient for any chain that can experience reorgs or sequencer restarts deeper than 10 blocks.

---

### Proof of Concept

1. Deploy a contract on StarkNet that calls `get_block_hash(block_number)` and uses the result to determine a winner in a lottery (e.g., `winner = hash % n_participants`).
2. A user submits a lottery entry transaction in block `N`.
3. The prover, when building block `N + 10`, must write the hash of block `N` into the mapping.
4. Instead of supplying the real hash of block `N`, the prover supplies `H_fake` such that `H_fake % n_participants == attacker_index`.
5. The prover generates a valid STARK proof — the OS imposes no constraint on `old_block_hash`, so the proof verifies.
6. The L1 verifier accepts the proof. The lottery contract reads `H_fake` via `get_block_hash(N)` and pays out to the attacker.

    let is_old_block_number_non_negative = is_nn(old_block_number);
    if (is_old_block_number_non_negative == FALSE) {
        // Not enough blocks in the system - nothing to write.
        return ();

    // Fetch the (block number -> block hash) mapping contract state.
    local state_entry: StateEntry*;
    %{ GetBlockHashMapping %}

    // Currently, the block hash mapping is not enforced by the OS.

        prev_value=cast(state_entry, felt),
