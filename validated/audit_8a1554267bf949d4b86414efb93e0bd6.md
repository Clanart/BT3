### Title
`EnableCall = false` Governance Pause Bypassable via Contract-Constructor Internal Calls — (`x/evm/keeper/state_transition.go`)

### Summary

Ethermint's `EnableCall` governance parameter is intended to pause all EVM call execution. However, the enforcement check is applied only to the **top-level** `ApplyMessageWithConfig` entry point. When `EnableCall = false` and `EnableCreate = true` (an explicitly supported governance state), any unprivileged user can bypass the call-pause by submitting a contract-creation transaction whose constructor body executes `CALL` opcodes against arbitrary targets. The EVM's internal call dispatch never re-enters `ApplyMessageWithConfig`, so the `EnableCall` guard is never re-evaluated for nested calls.

---

### Finding Description

`ApplyMessageWithConfig` enforces the governance flags at the very top of the function, before the EVM is instantiated:

```go
// return error if contract creation or call are disabled through governance
if !cfg.Params.EnableCreate && msg.To == nil {
    return nil, errorsmod.Wrap(types.ErrCreateDisabled, "failed to create new contract")
} else if !cfg.Params.EnableCall && msg.To != nil {
    return nil, errorsmod.Wrap(types.ErrCallDisabled, "failed to call contract")
}
``` [1](#0-0) 

The identical guard exists in the ante handler:

```go
if !enableCreate && tx.To() == nil {
    return errorsmod.Wrap(evmtypes.ErrCreateDisabled, "failed to create new contract")
} else if !enableCall && tx.To() != nil {
    return errorsmod.Wrap(evmtypes.ErrCallDisabled, "failed to call contract")
}
``` [2](#0-1) 

Both checks gate on the **outer message**: `msg.To == nil` means contract creation; `msg.To != nil` means a call. Once the EVM is running, all nested `CALL`, `STATICCALL`, `DELEGATECALL`, and `CALLCODE` opcodes are dispatched entirely within go-ethereum's `vm.EVM` interpreter. That interpreter never calls back into `ApplyMessageWithConfig`, so `cfg.Params.EnableCall` is never re-checked for any sub-call.

The governance `Params` struct exposes both flags independently:

```go
bool enable_create = 2 ...
bool enable_call   = 3 ...
``` [3](#0-2) 

Setting `EnableCall = false` while leaving `EnableCreate = true` is a reachable, explicitly supported governance state (the simulation genesis generator randomises them independently). [4](#0-3) 

**Attack path:**

1. Governance passes a proposal setting `EnableCall = false`, `EnableCreate = true` (e.g., to freeze all token transfers during an incident).
2. Attacker crafts a contract whose `constructor` body contains `CALL target, value, data` — a direct call to any contract (e.g., an ERC-20 `transfer`, a DEX swap, a vault withdrawal).
3. Attacker submits a `MsgEthereumTx` with `To = nil` (contract creation). The ante handler and `ApplyMessageWithConfig` both pass the `EnableCreate` check.
4. The EVM executes the constructor; the `CALL` opcode fires inside go-ethereum's interpreter. `EnableCall` is never consulted again.
5. The nested call commits to state exactly as if `EnableCall` were `true`.

The symmetric bypass also exists: `EnableCreate = false`, `EnableCall = true` → a call to a factory contract that internally executes `CREATE`/`CREATE2` deploys new contracts despite the governance freeze.

---

### Impact Explanation

The `EnableCall` / `EnableCreate` governance flags are the only on-chain mechanism to pause EVM execution. Bypassing them allows:

- **Unauthorized token transfers** when the chain operator intended to freeze all value movement (e.g., post-exploit containment).
- **Continued interaction with a vulnerable contract** that governance tried to isolate.
- **Mis-accounted state transitions**: transactions that should be rejected at the ante handler level instead commit, violating the invariant that governance parameters are enforced.

This matches the allowed High impact: *"EVM state transition … bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."*

---

### Likelihood Explanation

- The asymmetric state (`EnableCall = false`, `EnableCreate = true`) is a reachable governance configuration — the two flags are independent booleans with no cross-validation. [5](#0-4) 
- The bypass requires only a standard `eth_sendRawTransaction` call with a crafted constructor payload — no privileged access, no leaked keys.
- The scenario is most likely during an emergency governance action, exactly when the bypass is most harmful.

---

### Recommendation

1. **Cross-validate the flags**: Add a `Validate()` rule that rejects any `Params` update where `EnableCreate = true && EnableCall = false` (or vice versa), mirroring the external report's recommendation that pausing one operation must pause the dependent one.

2. **Alternatively, enforce at the EVM opcode level**: Pass `EnableCall` / `EnableCreate` into the `vm.Config` and override the `CanTransfer` / interpreter hooks so that the EVM itself rejects `CALL` / `CREATE` opcodes when the corresponding flag is `false`. This is the only complete fix.

3. **At minimum, document the invariant** that both flags must be toggled together, and enforce it in governance proposal validation.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IERC20 {
    function transfer(address to, uint256 amount) external returns (bool);
}

// Deploy this when EnableCall=false, EnableCreate=true.
// The constructor CALL bypasses the EnableCall governance gate.
contract BypassEnableCall {
    constructor(address token, address victim, address attacker, uint256 amount) {
        // This CALL executes even though EnableCall = false on-chain.
        IERC20(token).transfer(attacker, amount);
        // Any other call to any contract works here.
    }
}
```

Submit via `eth_sendRawTransaction` with `to = nil`. The ante handler passes (`EnableCreate = true`). The constructor `CALL` to the ERC-20 contract commits to state. The `EnableCall = false` governance parameter is silently bypassed. [1](#0-0) [2](#0-1) [6](#0-5)
