### Title
Zero Resource-Bound Price Bypasses Fee Charging While Preserving Execution Gas — (`execution/transaction_impls.cairo`)

### Summary

The `charge_fee` function in the StarkNet OS skips all fee collection when `compute_max_possible_fee` returns zero. Because `compute_max_possible_fee` is a product of amounts and prices, a user can set all `max_price_per_unit` fields to zero while keeping `L2_GAS.max_amount` positive. The result is a V3 transaction that receives a full L2 gas execution budget yet pays no fee — a direct analog of the "repay zero amount resets the borrow index" pattern from the reference report.

---

### Finding Description

`compute_max_possible_fee` computes the ceiling fee as a sum of products:

```
l1_gas.max_amount * l1_gas.max_price_per_unit
+ l2_gas.max_amount * (l2_gas.max_price_per_unit + tip)
+ l1_data_gas.max_amount * l1_data_gas.max_price_per_unit
``` [1](#0-0) 

If every price term is zero the whole expression is zero regardless of the amounts. `charge_fee` then returns immediately without executing the ERC-20 transfer:

```cairo
if (max_fee == 0) {
    return ();
}
``` [2](#0-1) 

The execution gas budget, however, is derived independently from `L2_GAS.max_amount` alone:

```cairo
func get_initial_user_gas_bound(common_tx_fields: CommonTxFields*) -> felt {
    assert common_tx_fields.n_resource_bounds = 3;
    return common_tx_fields.resource_bounds[L2_GAS_INDEX].max_amount;
}
``` [3](#0-2) 

The two code paths are completely decoupled. Nothing in the OS enforces that a non-zero gas budget implies a non-zero fee. The transaction hash machinery (`pack_resource_bounds`) explicitly permits `max_price_per_unit = 0` via `assert_nn` (non-negative, not positive): [4](#0-3) 

The same `charge_fee` call site is shared by `execute_invoke_function_transaction`, `execute_deploy_account_transaction`, and `execute_declare_transaction`: [5](#0-4) [6](#0-5) [7](#0-6) 

---

### Impact Explanation

A user who submits a V3 transaction with `L2_GAS.max_amount = N > 0` and all `max_price_per_unit = 0` receives `N` units of L2 gas for execution while the OS provably skips the ERC-20 fee transfer. The proven state transition is valid under the current OS rules, so the on-chain verifier accepts the proof. The sequencer receives no STRK/ETH for the work performed. Repeated at scale this constitutes **direct loss of funds** for the fee-collection mechanism and enables zero-cost spam that can **halt the network** by saturating block capacity without economic deterrent.

---

### Likelihood Explanation

The attacker controls all resource bound fields directly in the transaction they sign. No privileged access, key material, or operator cooperation is required to craft the malformed fields. The OS hash computation (`hash_fee_fields` → `pack_resource_bounds`) accepts zero prices without error, so the transaction is well-formed and will pass hash verification. The only external dependency is sequencer mempool policy; because the OS itself does not reject zero-price transactions, any mempool that defers to OS validity will forward them.

---

### Recommendation

Add an explicit guard in `charge_fee` (or in `compute_max_possible_fee`) that rejects a transaction whose `L2_GAS.max_amount > 0` but whose computed `max_fee == 0`. Concretely, after computing `max_fee`, assert:

```cairo
if (max_fee == 0) {
    // Ensure the transaction truly requested zero gas, not just zero price.
    assert get_initial_user_gas_bound(tx_info) = 0;
    return ();
}
```

Alternatively, enforce `max_price_per_unit > 0` for L2 gas inside `pack_resource_bounds` or `hash_fee_fields` so that the transaction hash itself is invalid for zero-price V3 transactions.

---

### Proof of Concept

1. Construct a V3 invoke transaction with:
   - `L2_GAS.max_amount = 1_000_000` (sufficient for a non-trivial `__execute__`)
   - `L2_GAS.max_price_per_unit = 0`
   - `L1_GAS.max_amount = 0`, `L1_GAS.max_price_per_unit = 0`
   - `L1_DATA_GAS.max_amount = 0`, `L1_DATA_GAS.max_price_per_unit = 0`
   - `tip = 0`
2. `compute_max_possible_fee` evaluates to `0 + 1_000_000*(0+0) + 0 = 0`.
3. `charge_fee` hits `if (max_fee == 0) { return (); }` and exits without executing the ERC-20 transfer.
4. `get_initial_user_gas_bound` returns `1_000_000`; validate and execute run normally.
5. The OS produces a valid proof. The verifier accepts it. The sequencer's fee balance is unchanged.

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```

func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn(resource_bounds.max_price_per_unit);
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
        resource_bounds.max_price_per_unit;
