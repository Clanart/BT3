### Title
EIP-712 Domain Separator Uses Static `VerifyingContract: "cosmos"` Enabling Cross-Chain Signature Replay Between Chains Sharing the Same Chain ID - (File: ethereum/eip712/domain.go)

---

### Summary

Both the legacy and non-legacy EIP-712 domain constructors in Ethermint hard-code `VerifyingContract: "cosmos"` and `Salt: "0"`. Because these are static, application-agnostic constants, the domain separator is differentiated **only** by `chainId`. Any two Cosmos/Ethermint chains that share the same numeric chain ID produce an identical EIP-712 domain separator, making a signed Cosmos transaction from one chain cryptographically valid on the other.

---

### Finding Description

In `ethereum/eip712/domain.go`, `createEIP712Domain` builds the typed-data domain as:

```go
domain := apitypes.TypedDataDomain{
    Name:              "Cosmos Web3",
    Version:           "1.0.0",
    ChainId:           math.NewHexOrDecimal256(chainID),
    VerifyingContract: "cosmos",   // ← static placeholder
    Salt:              "0",        // ← static placeholder
}
``` [1](#0-0) 

The same static values are duplicated verbatim in `LegacyWrapTxToTypedData`:

```go
domain := apitypes.TypedDataDomain{
    ...
    VerifyingContract: "cosmos",
    Salt:              "0",
}
``` [2](#0-1) 

The `EIP712Domain` type schema in both the legacy and non-legacy paths declares `verifyingContract` as `string` (not `address`), so `keccak256("cosmos")` is hashed in as a fixed 32-byte value that is identical for every Cosmos chain. [3](#0-2) 

The ante-handler verifier in `VerifySignature` checks that `extOpt.TypedDataChainID == signerChainID.Uint64()` and that the recovered fee-payer matches, but it never validates any chain-unique field beyond the numeric chain ID. [4](#0-3) 

The signed message payload (`txBytes`) does include `chain_id` (as a Cosmos string, e.g. `"ethermint_9000-1"`), `account_number`, `sequence`, fees, and messages. [5](#0-4) 

However, the domain separator — the outer EIP-712 envelope — is identical across all chains that share the same numeric chain ID. The analog to the original report is exact: just as the game contract's address was not encoded in the salt hash, the specific Cosmos application's identity is not encoded in the EIP-712 domain beyond the bare integer chain ID.

**Replay scenario:**

1. Chain A and Chain B both use numeric chain ID `9000` (e.g., a mainnet and a testnet fork, or a chain that migrated while keeping its chain ID).
2. Alice signs an EIP-712 Cosmos transaction on Chain A: `MsgSend` transferring 1000 tokens, with `account_number=5`, `sequence=0`.
3. Chain B was forked from Chain A at a point where Alice's account also has `account_number=5` and `sequence=0`.
4. An attacker submits Alice's signature verbatim to Chain B. The domain separator is identical; the message payload matches; `VerifySignature` passes; the `MsgSend` executes on Chain B, draining Alice's funds there.

---

### Impact Explanation

Unauthorized execution of arbitrary Cosmos messages (including `MsgSend`, governance votes, staking operations) on a chain the signer never intended to authorize. Because EIP-712 Cosmos transactions can carry any `sdk.Msg`, the attacker can drain EVM-denom or Cosmos bank balances from the victim on the target chain. This matches the allowed High impact: **EIP-712 authorization replay enabling forged execution and unauthorized fund transfer**.

---

### Likelihood Explanation

**Low.** Exploitation requires two live Cosmos/Ethermint chains sharing the same numeric chain ID **and** a victim account whose `account_number` and `sequence` are identical on both chains. This is realistic for:
- Chains that hard-fork without changing their chain ID
- Testnets that clone mainnet genesis (same account numbers, sequences start at 0)
- Chain migrations that preserve the numeric chain ID

Once those conditions are met, any previously broadcast EIP-712 signature is immediately replayable with no additional attacker capability.

---

### Recommendation

Encode a chain-unique, immutable identifier into the EIP-712 domain separator so that signatures are cryptographically bound to a single chain instance. Two concrete options:

1. **Genesis hash as `Salt`**: Set `Salt` to the hex-encoded genesis block hash (available at chain start). This is unique per chain even when chain IDs collide.
2. **Unique `VerifyingContract`**: Set `VerifyingContract` to a deterministic, chain-specific address (e.g., a well-known module account address unique to this deployment) rather than the static string `"cosmos"`.

Either change makes the domain separator unique per chain instance, directly mirroring the original report's recommendation to encode `address(this)` in the salt.

---

### Proof of Concept

```
Chain A: chainID=9000, genesis hash=0xAAA...
Chain B: chainID=9000, genesis hash=0xBBB... (fork of A)

Both chains produce domain separator:
  keccak256(encode(
    typeHash,
    keccak256("Cosmos Web3"),
    keccak256("1.0.0"),
    9000,
    keccak256("cosmos"),   ← identical on both chains
    keccak256("0")         ← identical on both chains
  ))
  → 0xSAMEHASH

Alice on Chain A signs:
  EIP712(domain=0xSAMEHASH, message={chain_id:"ethermint_9000-1",
         account_number:"5", sequence:"0", msgs:[MsgSend(Alice→Bob, 1000)]})
  → sig = 0xALICESIG

Attacker submits to Chain B:
  ExtensionOptionsWeb3Tx{TypedDataChainID:9000, FeePayer:Alice, FeePayerSig:0xALICESIG}
  → VerifySignature passes (domain separator identical, message fields match)
  → MsgSend executes on Chain B, Alice loses 1000 tokens she never authorized
```

The root cause is `VerifyingContract: "cosmos"` and `Salt: "0"` in `createEIP712Domain` (`ethereum/eip712/domain.go:29-30`) and `LegacyWrapTxToTypedData` (`ethereum/eip712/eip712_legacy.go:70-71`), consumed without any chain-unique binding by `VerifySignature` (`ante/cosmos/eip712.go:248-253`).
