### Title
Silent LRU Eviction of Pending Spice Endorsements Permanently Loses Certification Data — (`File: chain/chain/src/spice/core_writer_actor.rs`)

---

### Summary

`SpiceCoreWriterActor` stores endorsements that arrive before their block in a fixed-capacity LRU cache (`pending_endorsements`, size 100). When the cache overflows, the least-recently-used entry is silently evicted. When the corresponding block later arrives, `pop_pending_endorsement_for_block` finds nothing for the evicted `SpiceChunkId` and returns an empty set. Those endorsements are permanently lost: the chunk may never accumulate enough stake to be certified, stalling execution-result propagation for that shard.

---

### Finding Description

In `SpiceCoreWriterActor::new`, the pending-endorsement store is created with a hard cap of 100 entries:

```rust
const PENDING_ENDORSEMENT_CACHE_SIZE: NonZeroUsize = NonZeroUsize::new(100).unwrap();
pending_endorsements: SyncLruCache::new(PENDING_ENDORSEMENT_CACHE_SIZE.into()),
``` [1](#0-0) 

When an endorsement arrives for a block not yet in the store, `process_chunk_endorsement` calls `add_pending_endorsement`:

```rust
self.pending_endorsements
    .lock()
    .get_or_insert_mut(endorsement.chunk_id().clone(), HashMap::new)
    .insert(endorsement.account_id().clone(), endorsement);
``` [2](#0-1) 

`SyncLruCache` wraps a standard `lru::LruCache`. When the 101st distinct `SpiceChunkId` is inserted, the LRU entry is silently dropped with no log, no error, and no persistence to disk. [3](#0-2) 

When the block eventually arrives, `record_block_core_statements` calls:

```rust
let pending_endorsements = self.pop_pending_endorsement_for_block(&block)?;
store_update.merge(self.record_chunk_endorsements_with_block(block, pending_endorsements)?);
``` [4](#0-3) 

`pop_pending_endorsement_for_block` iterates over the block's shard IDs and calls `pop` on the LRU cache for each `SpiceChunkId`. If the entry was evicted, `pop` returns `None` and the function returns an empty `Vec` — the endorsements are gone. [5](#0-4) 

There is no fallback: the endorsements are not persisted to `DBCol::endorsements` until they are processed against a known block, so the disk store also has no record of them.

---

### Impact Explanation

In the Spice protocol, a chunk's execution result must be endorsed by a quorum of chunk validators before it can be certified and included in the next block. If enough endorsements for a given `SpiceChunkId` are evicted from the cache before the block arrives, the chunk may permanently fail to reach the certification threshold. This breaks the `SpiceCoreWriterActor`'s core invariant: **every signature-validated endorsement must eventually be applied to its block**. The corrupted value is the `HashMap<AccountId, SpiceVerifiedEndorsement>` for the evicted `SpiceChunkId`, which is irrecoverably lost.

**Impact: High** — chunk certification can stall for affected shards, blocking execution-result propagation and delaying block production.

---

### Likelihood Explanation

The cache holds 100 `SpiceChunkId` entries, where each entry is a `(block_hash, shard_id)` pair. In a network with `N` shards, 100 / N blocks worth of out-of-order endorsements fill the cache. With 4 shards that is 25 blocks; with 8 shards it is 12. High-latency links, burst block production, or a slow block producer that delays block gossip while endorsements propagate ahead all produce this condition in normal operation — no malicious actor is required. A validator with any stake can also deliberately send endorsements for many future `(block_hash, shard_id)` pairs (each passes `validate_endorsement_without_block`'s signature and epoch checks) to force eviction of a targeted chunk's endorsements. [6](#0-5) 

---

### Recommendation

1. **Persist pending endorsements to disk** (e.g., a dedicated `DBCol`) instead of relying solely on an in-memory LRU cache. On block arrival, read from disk and delete after processing.
2. **Alternatively, bound the cache by block height** rather than by number of `SpiceChunkId` entries, and evict only entries whose block height is provably below the final head (i.e., the block can never arrive).
3. **At minimum, log a warning** when an LRU eviction occurs so operators can detect the condition.

---

### Proof of Concept

1. Start a Spice-enabled network with 4 shards.
2. Arrange for 26 distinct future blocks' endorsements to arrive at the `SpiceCoreWriterActor` before those blocks are stored (e.g., by delaying block gossip to the writer node while endorsements propagate normally).
3. The 101st `SpiceChunkId` insertion evicts the first entry.
4. When the block for the evicted `SpiceChunkId` arrives, `pop_pending_endorsement_for_block` returns `[]` for that shard.
5. `record_chunk_endorsements_with_block` is called with an empty endorsement list; no endorsements are saved to `DBCol::endorsements` for that `(block_hash, shard_id)`.
6. `try_sending_execution_result_endorsed` finds the execution result count below threshold and does not fire `ExecutionResultEndorsed`.
7. The chunk's execution result is never certified; the shard stalls. [7](#0-6) [8](#0-7) [9](#0-8)

    fn pop_pending_endorsement_for_block(
        &self,
        block: &Block,
    ) -> Result<Vec<SpiceVerifiedEndorsement>, Error> {
        let mut endorsements = Vec::new();
        let shard_layout = self.epoch_manager.get_shard_layout(block.header().epoch_id())?;
        for shard_id in shard_layout.shard_ids() {
            let Some(endorsements_map) = self
                .pending_endorsements
                .lock()
                .pop(&SpiceChunkId { block_hash: *block.hash(), shard_id })
            else {
                continue;
            };
            endorsements.extend(endorsements_map.into_values().filter(|endorsement| {
                match self.validate_verified_endorsement_with_block(endorsement, block) {
                    Ok(()) => true,
                    Err(err) => {
                        tracing::info!(
                            target: "spice_core_writer",
                            chunk_id = ?endorsement.chunk_id(),
                            ?err,
                            "encountered invalid pending endorsement"
                        );
                        false
                    }
                }
            }));
        }
        Ok(endorsements)
    }
