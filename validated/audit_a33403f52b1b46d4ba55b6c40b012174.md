### Title
`Simulator.applyCall` Nonce Reset Before EIP-7702 Authorization Processing Causes Self-Delegation to Always Silently Fail — (`File: x/evm/keeper/simulate.go`)

---

### Summary

`Simulator.applyCall` unconditionally resets the sender's nonce to `msg.Nonce` before processing EIP-7702 `SetCodeAuthorizations`. For self-delegating transactions (where `authority == msg.From`), the authorization was signed with `auth.Nonce = msg.Nonce + 1` (the post-ante-handler nonce). Because the simulator resets the nonce to `msg.Nonce` first, `validateAuthorization` always finds `have (msg.Nonce) != auth.Nonce (msg.Nonce+1)` and returns `ErrAuthorizationNonceMismatch`. The error is silently swallowed, so the delegation is never applied. The simulation then proceeds with the call as if no delegation exists, producing state that diverges from what the real chain would commit.

---

### Finding Description

In `x/evm/keeper/simulate.go`, `applyCall` sets the sender nonce to `msg.Nonce` unconditionally before any authorization processing:

```go
// Line 432
sim.state.SetNonce(msg.From, msg.Nonce, tracing.NonceChangeUnspecified)
```

Then, for the CALL branch, it processes authorizations:

```go
// Lines 441-448
if msg.SetCodeAuthorizations != nil {
    for _, auth := range msg.SetCodeAuthorizations {
        if _, err := sim.keeper.applyAuthorization(&auth, sim.state); err != nil {
            sim.keeper.Logger(...).Debug("simulation: failed to apply authorization", ...)
        }
    }
}
```

`applyAuthorization` calls `validateAuthorization`, which checks:

```go
// x/evm/keeper/set_code_authorizations.go line 39
if have := stateDB.GetNonce(authority); have != auth.Nonce {
    return authority, core.ErrAuthorizationNonceMismatch
}
```

For a self-delegating EIP-7702 transaction, the authorization is signed with `auth.Nonce = msg.Nonce + 1` (the nonce the account will have after the ante handler pre-increments it). But the simulator has just reset the nonce to `msg.Nonce`, so `have = msg.Nonce ≠ auth.Nonce = msg.Nonce+1`. The authorization fails. The error is silently discarded. `setAuthorizationDelegation` is never called, so the delegation code is never written to the account.

After `evm.Call()`, the simulator then sets:

```go
// Line 450
sim.state.SetNonce(msg.From, msg.Nonce+1, tracing.NonceChangeUnspecified)
```

This is also wrong for self-delegation: the correct post-execution nonce should be `msg.Nonce+2` (one for the transaction, one for the authorization), but the simulator leaves it at `msg.Nonce+1`.

The consensus path in `ApplyMessageWithConfig` does not have this problem: the ante handler pre-increments the nonce to `msg.Nonce+1` before `applyAuthorization` is called, so `have = msg.Nonce+1 = auth.Nonce` and the authorization succeeds.

---

### Impact Explanation

`eth_simulateV1` (the public JSON-RPC `eth_simulateV1` endpoint backed by `Simulator.processBlock` → `applyCall`) produces simulation results that diverge from real chain execution for any EIP-7702 self-delegating transaction:

1. **Delegation never applied in simulation**: The simulation shows the account without delegation code. Any call that depends on the delegated code executing (e.g., a factory contract deploying children via the delegation) will appear to fail or produce wrong output in the simulation, while succeeding on the real chain.

2. **Nonce undercount in validation mode**: After the call, the simulator leaves the sender's nonce at `msg.Nonce+1` instead of `msg.Nonce+2`. In `validation: true` mode, a subsequent call from the same sender with nonce `msg.Nonce+2` is rejected with `ErrNonceTooHigh` even though it is valid on the real chain.

Both effects cause `eth_simulateV1` to feed incorrect consensus-critical data (account code state, nonce) to callers, qualifying as a High impact under the allowed scope: *"Public JSON-RPC … simulation … path feeds incorrect consensus-critical data into transaction execution."*

---

### Likelihood Explanation

EIP-7702 self-delegation is a standard and documented pattern (e.g., the integration test `setup_eip7702_delegation` uses `auth.nonce = current_nonce + 1`, which is exactly the pattern that triggers the bug). Any developer or user calling `eth_simulateV1` to preview a self-delegating EIP-7702 transaction will hit this silently. No special privileges are required; the endpoint is public.

---

### Recommendation

In `applyCall`, process authorizations **before** resetting the nonce, or reset the nonce to `msg.Nonce + 1` (the post-ante-handler value) so that `validateAuthorization` sees the same nonce the real chain would:

```go
// Option A: set nonce to msg.Nonce+1 before authorization processing
sim.state.SetNonce(msg.From, msg.Nonce+1, tracing.NonceChangeUnspecified)

// ... process authorizations (self-delegation: nonce → msg.Nonce+2) ...

ret, leftoverGas, vmErr = evm.Call(...)
// Remove the unconditional SetNonce(msg.From, msg.Nonce+1) after evm.Call
// The nonce is already correct from authorization processing
```

For the CREATE branch, the existing reset to `msg.Nonce` is correct (go-ethereum's `evm.Create` internally bumps it). The CALL branch needs a different treatment.

func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}
