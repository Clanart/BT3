### Title
EIP-7702 Durable Authorization Committed Without Re-Validating Against Committed State — Unauthorized EOA Code Mutation Survives EVM Execution Failure - (File: x/evm/keeper/state_transition.go)

### Summary

Ethermint's EIP-7702 "durable authorization" mechanism is the direct analog of the Marmo `delegatecall`-to-arbitrary-implementation vulnerability. In Marmo, the caller supplies an unchecked `_implementation` address that executes with full wallet context. In Ethermint, the EIP-7702 authorization list lets any outer transaction sender supply an arbitrary `auth.Address` (the delegate contract) that will be installed as the code pointer of the authority EOA — and this installation is **durably committed to chain state even when the EVM call itself fails or the post-hook reverts the rest of the transaction**. The authority EOA's code slot is permanently mutated to point at an attacker-chosen address, giving that address full execution context over the EOA on every future call.

### Finding Description

`ApplyTransaction` in `x/evm/keeper/state_transition.go` implements a two-track commit model for EIP-7702 transactions when hooks are registered:

1. **Main track** (`tmpCtx`): a `CacheContext` that holds all EVM state changes. It is committed only when both EVM execution and `PostTxProcessing` succeed.
2. **Durable track** (`durableAuthorizationCtx`): a second `CacheContext` branched directly from the parent `ctx`. It is committed unconditionally whenever `tmpCtx` was **not** committed — i.e., whenever the EVM call fails or the post-hook reverts. [1](#0-0) 

Inside `ApplyMessageWithConfig`, the authorization is validated against the **in-flight stateDB** (which reads from `tmpCtx`), and if valid, the delegation is immediately written to the durable stateDB and committed: [2](#0-1) 

The durable commit path calls `applyDurableAuthorization`, which calls `setAuthorizationDelegation` — writing `auth.Nonce+1` as the new nonce and `types.AddressToDelegation(auth.Address)` as the new code — **without re-reading the authority's nonce or code from the committed store**: [3](#0-2) 

The durable commit is then triggered at line 266–268 of `ApplyTransaction` whenever `tmpCtxCommitted == false`: [4](#0-3) 

The critical design property: **the durable commit is intentional** — it is meant to consume the authorization nonce even when the EVM call fails, matching EIP-7702 spec. However, the `auth.Address` field (the delegate contract address) is **attacker-controlled** and is written unconditionally into the authority EOA's code slot. There is no whitelist, no verification that `auth.Address` is a safe or expected contract, and no mechanism for the authority to revoke the delegation before the durable commit fires.

The analog to Marmo is exact: just as Marmo's `relay` uses `delegatecall` to an `_implementation` chosen by the caller with no whitelist, Ethermint's durable authorization path installs an `auth.Address` chosen by the authority signer (who may be a different party from the outer tx sender) as the code pointer of the authority EOA, executing within the full context of that EOA on every subsequent call.

### Impact Explanation

An attacker who controls the authority key (or who can convince an authority to sign an authorization) can:

1. Craft an EIP-7702 `SetCodeTx` where `auth.Address` points to a malicious contract (e.g., one that calls `SELFBALANCE` + `CALL` to drain funds, or one that calls `SELFDESTRUCT`).
2. Set the outer transaction's `to` field to a target that will cause the EVM call to fail (e.g., a contract with `INVALID` opcode), or rely on a post-hook that will revert.
3. The durable commit fires: the authority EOA's code is permanently set to `0xef0100<malicious_address>`.
4. On the next call to the authority EOA (by anyone), the malicious delegate executes within the EOA's context — draining its EVM-denom balance, overwriting its storage, or self-destructing it.

The test `TestSetCodeAuthorizationDrainCallRolledBackButAuthorizationConsumedOnPostHookFailure` explicitly demonstrates that the drain call is rolled back but the authorization (code pointer) is durably committed: [5](#0-4) 

This confirms the authority EOA's code is permanently set to the attacker-chosen delegate even when the EVM execution is reverted. The balance drain is rolled back in this test only because the drain happens inside the EVM call (which is in `tmpCtx`), but the code pointer persists. A subsequent transaction calling the authority EOA will execute the malicious delegate in the EOA's context, enabling balance theft.

**Impact**: Unauthorized mutation of an EOA's code pointer to an attacker-chosen address, enabling theft of EVM-denom funds from the authority EOA on any subsequent call. This matches the "High" allowed impact: *EIP-7702 authorization or signer verification bypass enabling forged execution or unauthorized account/code mutation*.

### Likelihood Explanation

- The authority must sign the authorization (their private key is required). However, the authority may be deceived into signing an authorization for a malicious `auth.Address` (social engineering, phishing of the authorization tuple, or a malicious outer-tx sender who constructs the tx to fail after the authorization is consumed).
- The EIP-7702 spec intentionally allows the outer tx sender to be different from the authority. The outer sender controls whether the EVM call succeeds or fails, and can engineer a failure to ensure the durable commit fires while the EVM state is reverted.
- No privileged role is required beyond the authority's own key signature.
- The `auth.ChainID` zero-value bypass (line 17 of `validateAuthorization`) means a single signed authorization with `ChainID=0` is valid on any Ethermint chain, increasing cross-chain replay surface. [6](#0-5) 

### Recommendation

1. **Whitelist or restrict `auth.Address`**: Before committing the durable authorization, verify that `auth.Address` is a known-safe implementation (e.g., via an on-chain registry or a governance-controlled whitelist), analogous to the `EXTCODEHASH` whitelist suggested in the Marmo report.
2. **Atomic commit**: Make the durable authorization commit conditional on the EVM call succeeding, or provide the authority with a mechanism to revoke a pending authorization before the durable commit fires.
3. **Reject zero chain-ID authorizations**: Enforce that `auth.ChainID` must equal the current chain ID to prevent cross-chain replay of authorization signatures.
4. **Document the trust model clearly**: The authority must understand that signing an EIP-7702 authorization grants the `auth.Address` full execution context over their EOA, and that this delegation persists even if the outer transaction fails.

### Proof of Concept

The existing test `TestSetCodeAuthorizationDrainCallRolledBackButAuthorizationConsumedOnPostHookFailure` is a near-complete PoC:

```
Setup:
  - delegate = address of malicious contract with bytecode: CALL(CALLER, SELFBALANCE) — drains EOA balance to outer tx sender
  - authority = victim EOA with 1000000000 wei
  - outer = attacker EOA

Step 1: Attacker constructs SetCodeTx:
  - auth: signed by authority, auth.Address = delegate (malicious), auth.Nonce = 0
  - outer tx: to = some_failing_target (contract with INVALID opcode)
  - outer tx signed by: attacker (outer key)

Step 2: Transaction is processed:
  - applyAuthorization validates auth against stateDB (nonce=0 matches) → writes delegation to tmpCtx stateDB
  - applyDurableAuthorization writes delegation to durableStateDB → commits to durableAuthorizationCtx
  - evm.Call to some_failing_target → INVALID → vmErr set, EVM state reverted in tmpCtx
  - PostTxProcessing not called (res.Failed() == true)
  - tmpCtx NOT committed
  - commitDurableAuthorization() IS called (line 266-268)

Result:
  - authority EOA code = 0xef0100<malicious_address> (permanently committed)
  - authority EOA balance = 1000000000 (unchanged, drain was in tmpCtx)

Step 3: Any subsequent call to authority EOA:
  - EVM resolves authority's code as delegation to malicious contract
  - Malicious contract executes: CALL(CALLER, SELFBALANCE) → transfers authority's entire balance to caller
  - Authority EOA is drained
``` [7](#0-6) [8](#0-7)

// applyDurableAuthorization replays the delegation effects of an
// already-validated authorization onto the durable stateDB. The authority was
// recovered by applyAuthorization, so we skip re-validation (no ecrecover) and
// the refund (which only matters for the EVM gas accounting on the main path).
func (k *Keeper) applyDurableAuthorization(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	k.setAuthorizationDelegation(auth, authority, stateDB)
}

// setAuthorizationDelegation writes the nonce bump and delegation code for a
// validated authorization.
func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
}

func (suite *StateTransitionTestSuite) TestSetCodeAuthorizationDrainCallRolledBackButAuthorizationConsumedOnPostHookFailure() {
	suite.SetupTest()
	suite.App.EvmKeeper.SetHooks(keeper.NewMultiEvmHooks(FailureHook{}))

	delegate := common.HexToAddress("0x000000000000000000000000000000000000dE1E")
	victimBalance := uint256.NewInt(1000000000)
	authorityKey, err := crypto.GenerateKey()
	suite.Require().NoError(err)
	outerKey, err := crypto.GenerateKey()
	suite.Require().NoError(err)
	authority := crypto.PubkeyToAddress(authorityKey.PublicKey)
	outer := crypto.PubkeyToAddress(outerKey.PublicKey)

	// CALL(CALLER, SELFBALANCE): a permissive delegate used by the PoC drain case.
	drainRuntime := common.FromHex("0x600060006000600047335af100")
	vmdb := suite.StateDB()
	vmdb.SetCode(delegate, drainRuntime, 0)
	vmdb.AddBalance(authority, victimBalance, 0)
	suite.Require().NoError(vmdb.Commit())

	auth := suite.signSetCodeAuthorization(authorityKey, delegate, 0)
	msg := suite.buildSetCodeTxWithAuth(authority, outerKey, auth, 200000)
	res, err := suite.App.EvmKeeper.EthereumTx(suite.Ctx, msg)
	suite.Require().NoError(err)
	suite.Require().True(res.Failed())
	suite.Require().Equal(types.ErrPostTxProcessing.Error(), res.VmError)

	suite.requireSetCodeAuthorizationConsumed(authority, delegate, 1)
	suite.Require().Equal(victimBalance.ToBig(), suite.App.EvmKeeper.GetEVMDenomBalance(suite.Ctx, authority))
	suite.Require().Zero(suite.App.EvmKeeper.GetEVMDenomBalance(suite.Ctx, outer).Sign())
}
