### Title
Irrecoverable Relayer Fund Loss in `utxo_fin_transfer_fast` Due to Unchecked `send_tokens(...).detach()` — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`utxo_fin_transfer_fast` removes or finalises the fast-transfer record **before** calling `send_tokens(...).detach()`. Because `.detach()` fires the promise with no callback, any failure of the token transfer silently discards the result. The fast-transfer record is already gone, so there is no re-entry point and no recovery path. The relayer's advanced tokens remain permanently locked inside the bridge contract.

The developers themselves flagged this with a TODO comment directly at the call site.

---

### Finding Description

**Entry point:** `ft_on_transfer` → `utxo_fin_transfer` → `utxo_fin_transfer_fast` [1](#0-0) 

Inside `utxo_fin_transfer_fast`, two distinct branches both commit state before the token transfer:

**Branch 1 — destination is NEAR** (line 2530): `remove_fast_transfer` is called, permanently deleting the record from `fast_transfers`.

**Branch 2 — destination is another chain** (line 2533): `mark_fast_transfer_as_finalised` is called, setting `finalised = true`. [2](#0-1) 

After either state mutation, `send_tokens` is called with `.detach()`: [3](#0-2) 

`send_tokens` for a non-deployed token with an empty message issues a plain `ft_transfer`: [4](#0-3) 

`ft_transfer` panics if the recipient has no storage deposit in the token contract, if the token contract is paused, or if gas is exhausted. Because the promise is detached, the panic is silently swallowed. The bridge retains the tokens but has no record of the pending obligation.

**Contrast with the correctly-handled paths:**

- `fast_fin_transfer_to_near_callback` chains `send_tokens` with `resolve_fast_transfer`, which burns tokens and removes the record on failure. [5](#0-4) 

- `utxo_fin_transfer_to_near_callback` chains `send_tokens` with `resolve_utxo_fin_transfer`, which refunds on failure. [6](#0-5) 

`utxo_fin_transfer_fast` is the only settlement path that uses `.detach()` with no error handler.

---

### Impact Explanation

The relayer advanced their own tokens to the original recipient during the fast-transfer leg. When the UTXO connector later calls `utxo_fin_transfer`, the bridge is supposed to reimburse the relayer. If `send_tokens` fails:

- **Branch 1 (NEAR destination):** The fast-transfer record is already removed. Re-triggering `utxo_fin_transfer` would succeed but find no fast-transfer record, so it would execute the normal (non-fast) path and send tokens to the original recipient — not the relayer. The relayer's reimbursement is permanently lost.
- **Branch 2 (other-chain destination):** The record is marked `finalised = true`. Any retry hits `BridgeError::FastTransferAlreadyFinalised`. The relayer's reimbursement is permanently lost.

In both cases the tokens remain in the bridge with no automatic or protocol-level recovery path. The only escape is a manual DAO intervention via `transfer_token_as_dao`, which is not guaranteed.

Impact: **Critical — permanent, irrecoverable lock of relayer funds in the bridge.**

---

### Likelihood Explanation

Realistic failure conditions for `send_tokens` in this context:

1. **Relayer not registered in the UTXO token contract.** NEP-141 `ft_transfer` panics if the receiver has no storage deposit. A relayer who registered for the bridged token but not for the UTXO-native token would trigger this.
2. **Token contract paused.** If the UTXO token contract is paused between the fast-transfer leg and the settlement leg, `ft_transfer` panics.
3. **Gas exhaustion.** The detached promise receives whatever gas remains after the synchronous work; if that is insufficient, the transfer silently fails.

Condition 1 is reachable without any privileged action. The relayer controls their own storage registrations and may not have pre-registered in the UTXO token contract.

---

### Recommendation

Mirror the pattern used by `fast_fin_transfer_to_near_callback` and `utxo_fin_transfer_to_near_callback`:

1. Do **not** remove or finalise the fast-transfer record before the token transfer succeeds.
2. Chain `send_tokens` with a callback (e.g., `resolve_utxo_fast_transfer`) that:
   - On success: removes/finalises the record.
   - On failure: leaves the record intact so the settlement can be retried, or refunds the tokens back to the relayer.

The developers already identified this gap: [1](#0-0)

### Proof of Concept

```
1. Deploy a mock UTXO token contract that panics on ft_transfer when a flag is set.
2. Register a trusted relayer and perform a fast transfer (fast_fin_transfer) for a
   UTXO-origin transfer with a NEAR recipient. The fast-transfer record is created.
3. Set the mock token's panic flag.
4. Call utxo_fin_transfer (via the UTXO connector) for the same transfer ID.
   - utxo_fin_transfer_fast is entered.
   - remove_fast_transfer deletes the record.
   - send_tokens fires ft_transfer, which panics; the detached promise fails silently.
5. Assert:
   - The relayer's token balance is unchanged (no reimbursement received).
   - fast_transfers no longer contains the record (no retry possible).
   - Calling utxo_fin_transfer again succeeds but routes to the normal path,
     sending tokens to the original recipient instead of the relayer.
6. The relayer's advanced tokens are permanently locked in the bridge.
```
