### Title
Pending-deployed contract with zero storage value incorrectly returns `CONTRACT_NOT_FOUND` in `starknet_getStorageAt` — (File: `crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

`starknet_getStorageAt` for `BlockId::Tag(Tag::Pending)` checks contract existence exclusively against the **committed** state. When a contract is deployed only in the pending block and the queried storage slot is zero (never written), the RPC returns `CONTRACT_NOT_FOUND` instead of `Felt::ZERO`. This is a direct sequencer-native analog of the `LiquidationPool::empty` bug: a "pending" record is ignored when deciding whether an entity exists, causing wrong exclusion from a result.

---

### Finding Description

In `get_storage_at` the handler first overlays pending storage diffs on top of committed storage, then — if the result is `Felt::default()` — validates contract existence using only the committed-state reader:

```rust
// crates/apollo_rpc/src/v0_8/api/api_impl.rs  lines 347-382
let maybe_pending_storage_diffs = if let BlockId::Tag(Tag::Pending) = block_id {
    Some(
        read_pending_data(&self.pending_data, &txn)
            .await?
            .state_update
            .state_diff
            .storage_diffs,   // ← only storage_diffs, not deployed_contracts
    )
} else {
    None
};

let block_number = get_accepted_block_number(&txn, block_id)?;
let state_number = StateNumber::unchecked_right_after_block(block_number);
let res = execution_utils::get_storage_at(
    &txn,
    state_number,
    maybe_pending_storage_diffs.as_ref(),
    contract_address,
    key,
)
.map_err(internal_server_error)?;

if res == Felt::default() && contract_address != BLOCK_HASH_TABLE_ADDRESS {
    txn.get_state_reader()
        .map_err(internal_server_error)?
        .get_class_hash_at(state_number, &contract_address)   // ← committed state only
        .map_err(internal_server_error)?
        .ok_or_else(|| ErrorObjectOwned::from(CONTRACT_NOT_FOUND))?;
}
``` [1](#0-0) 

`maybe_pending_storage_diffs` is populated from `state_update.state_diff.storage_diffs` only — it does **not** carry `deployed_contracts` or `nonces` from the pending block. [2](#0-1) 

`get_class_hash_at(state_number, &contract_address)` queries the MDBX-backed committed state at the tip of the last finalized block. [3](#0-2) 

A contract deployed only in the pending block is absent from that table, so the lookup returns `None` and the handler raises `CONTRACT_NOT_FOUND`. The pending `deployed_contracts` list is never consulted.

The existing test at line 2121 explicitly covers the *stale-pending* variant of this scenario and marks `CONTRACT_NOT_FOUND` as the expected outcome, but it does not test the **up-to-date-pending** case where the contract is deployed in the current pending block. [4](#0-3) 

---

### Impact Explanation

Any caller of `starknet_getStorageAt` with `block_id = "pending"` for a contract that:
1. was deployed in the current (up-to-date) pending block, and
2. has never written to the queried storage slot (value is `Felt::ZERO`)

receives `CONTRACT_NOT_FOUND` instead of `0x0`. This is an authoritative-looking wrong value from the RPC layer. Wallets, SDKs, and dApps that probe pending-block contract state (e.g., to read initial storage after deployment) will observe a spurious error and may incorrectly conclude the deployment failed or the contract does not exist.

**Matching impact scope:** *High — RPC pending view returns an authoritative-looking wrong value.*

---

### Likelihood Explanation

The trigger is unprivileged: any user who deploys a contract in the pending block and immediately queries an uninitialized storage slot hits this path. The pending block is always present in a live sequencer node. The condition `res == Felt::default()` is the common case for any storage key that has never been written.

---

### Recommendation

After computing `res`, also check the pending `deployed_contracts` (and/or `nonces`) before concluding the contract does not exist. Concretely, read the pending state update's `deployed_contracts` map alongside `storage_diffs` and include it in the existence check:

```rust
let pending_state_update = read_pending_data(&self.pending_data, &txn).await?;
let maybe_pending_storage_diffs = if let BlockId::Tag(Tag::Pending) = block_id {
    Some(pending_state_update.state_update.state_diff.storage_diffs)
} else {
    None
};
// ... compute res as before ...
if res == Felt::default() && contract_address != BLOCK_HASH_TABLE_ADDRESS {
    let deployed_in_pending = maybe_pending_storage_diffs.is_some()
        && pending_state_update
            .state_update
            .state_diff
            .deployed_contracts   // or nonces map
            .contains_key(&contract_address);
    if !deployed_in_pending {
        txn.get_state_reader()?
            .get_class_hash_at(state_number, &contract_address)?
            .ok_or_else(|| ErrorObjectOwned::from(CONTRACT_NOT_FOUND))?;
    }
}
```

---

### Proof of Concept

1. Node is running; pending block is up to date (parent hash matches latest committed block).
2. Transaction T1 deploys contract `C` in the pending block. `C` is added to `pending_data.state_update.state_diff.deployed_contracts` but is **not** yet in committed storage.
3. Caller queries `starknet_getStorageAt(C, key=0x1, "pending")`.
4. `execution_utils::get_storage_at` finds no entry for `(C, 0x1)` in either pending storage diffs or committed storage → returns `Felt::ZERO`.
5. `res == Felt::default()` → true.
6. `get_class_hash_at(state_number, C)` → `None` (C not committed).
7. Handler returns `CONTRACT_NOT_FOUND`.
8. **Expected:** `0x0` (uninitialized storage of a deployed contract). **Actual:** `CONTRACT_NOT_FOUND`.
