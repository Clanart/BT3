### Title
Permanent Token Destruction in `swap_migrated_token` Due to Non-Atomic Burn+Mint with No Refund Path — (`near/omni-bridge/src/lib.rs`)

### Summary
`ft_on_transfer` returns `U128(0)` (no refund) and calls `swap_migrated_token(...).detach()`. This commits the user's old tokens to the bridge before the burn+mint async chain runs. If the mint cross-contract call fails (e.g., recipient not registered in new token), the burn has already executed and the tokens are permanently destroyed with no recovery mechanism.

### Finding Description
The NEP-141 `ft_transfer_call` flow works as follows:
1. User calls `old_token.ft_transfer_call(bridge, amount, SwapMigratedToken)`.
2. `old_token` transfers `amount` to bridge, then calls `bridge.ft_on_transfer`.
3. `ft_on_transfer` returns `U128(0)` — zero refund — committing the old tokens to the bridge.
4. The detached promise `swap_migrated_token` then calls `burn(old_token, amount)` followed by `mint(new_token, sender, amount)`.
5. If `mint` panics (e.g., `internal_deposit` panics because sender is not registered), the burn is not rolled back. NEAR has no cross-contract atomicity.
6. No outer callback exists (`.detach()`) to detect the failure and re-mint or refund. [1](#0-0) [4](#0-3) 

### Impact Explanation
**Critical — Permanent, irrecoverable destruction of user funds.** The user loses their old tokens (burned) and receives no new tokens (mint failed). There is no on-chain recovery path. The bridge's `migrated_tokens` map entry remains, so the user cannot retry with old tokens (they are gone).

### Likelihood Explanation
**High.** The most common trigger requires no attacker: a user who has not registered (called `storage_deposit`) on the new token contract before initiating the swap. This is a standard NEAR FT requirement that many users miss. Additionally, out-of-gas conditions or any panic in the new token's `mint` function produce the same outcome.

### Recommendation
1. **Do not return `U128(0)` unconditionally.** Return `amount` (full refund) from `ft_on_transfer` for `SwapMigratedToken`, and only proceed with burn+mint inside a callback that is chained to the `ft_transfer_call` resolution.
2. **Alternatively**, add a callback after `burn.and(mint)` that, on mint failure, re-mints the old tokens back to the sender (or stores a claimable balance).
3. **Pre-check** that the sender is registered in the new token contract before burning, and revert the entire operation (return `amount` from `ft_on_transfer`) if not.

### Proof of Concept
```
1. Admin adds (old_token → new_token) to migrated_tokens map.
2. User holds 100 old_token but has NOT called storage_deposit on new_token.
3. User calls: old_token.ft_transfer_call(bridge, 100, "SwapMigratedToken")
4. bridge.ft_on_transfer returns U128(0) → old tokens committed to bridge, no refund.
5. swap_migrated_token executes:
   a. burn(old_token, 100) → succeeds, 100 old tokens destroyed.
   b. mint(new_token, user, 100) → panics: user not registered in new_token.
6. Result: user has 0 old_token, 0 new_token. Tokens permanently destroyed.
```
