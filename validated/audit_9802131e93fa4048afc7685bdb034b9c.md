### Title
Unsound `py_to_slice` Lifetime Extension Causes Use-After-`PyBuffer_Release` Data Race in Consensus-Critical Block Generator Binding — (`File: wheel/src/run_generator.rs`)

---

### Summary

`py_to_slice` in `wheel/src/run_generator.rs` uses `unsafe` to forge an arbitrary lifetime `'a` for a slice derived from a `PyBuffer<u8>`, then immediately drops the `PyBuffer` (triggering `PyBuffer_Release`). The returned slice — now pointing to memory the Python buffer protocol considers released — is subsequently read inside `py.detach()`, which drops the GIL. Any Python thread holding a reference to the same mutable backing object (e.g. `bytearray`) may legally mutate that memory concurrently, producing a data race over the raw bytes of the block generator program or block-ref programs fed into `native_run_block_generator` / `native_run_block_generator2`. The same pattern is replicated across `additions_and_removals`, `generator_interned_vbytes`, `get_puzzle_and_solution_for_coin`, and `get_spends_for_trusted_block`.

---

### Finding Description

**Root cause — `py_to_slice`:**

```rust
// wheel/src/run_generator.rs  lines 20-23
pub fn py_to_slice<'a>(buf: PyBuffer<u8>) -> &'a [u8] {
    assert!(buf.is_c_contiguous(), "buffer must be contiguous");
    unsafe { std::slice::from_raw_parts(buf.buf_ptr() as *const u8, buf.len_bytes()) }
}
```

`buf` is consumed (moved in). At the closing brace of `py_to_slice`, `buf` is dropped, which calls `ffi::PyBuffer_Release`. Per the Python buffer protocol, `PyBuffer_Release` signals to the exporting object that the view is no longer held; a mutable exporter (e.g. `bytearray`, `memoryview` over a `bytearray`) is then free to resize or overwrite the backing store. The returned `&'a [u8]` carries a lifetime `'a` that is **not** tied to `buf` in any way — the compiler cannot see that the data's true lifetime ended when `buf` was dropped.

**Call site — `run_block_generator` / `run_block_generator2`:**

```rust
// wheel/src/run_generator.rs  lines 42-57
let refs = block_refs.to_list()...
    .map(|b| {
        let buf = b.extract::<PyBuffer<u8>>()...;
        py_to_slice::<'a>(buf)          // ← PyBuffer dropped here; PyBuffer_Release fired
    })
    .collect::<Vec<&'a [u8]>>();
let program = py_to_slice::<'a>(program); // ← same

py.detach(|| {                            // ← GIL released here
    match native_run_block_generator(
        program, refs, ...               // ← reads from post-release memory
    ) { ... }
})
```

The sequence is:

1. `py_to_slice` is called; `PyBuffer_Release` fires; the Python exporter is now free to mutate.
2. The GIL is still held — no other thread can run yet.
3. `py.detach()` releases the GIL.
4. A concurrent Python thread that holds a reference to the same `bytearray` (or `memoryview`) can now write to the backing store.
5. `native_run_block_generator` reads `program` / `refs` bytes while they are being mutated — a classic data race over `unsafe` memory.

The same pattern appears in:
- `additions_and_removals` (lines 128–151)
- `generator_interned_vbytes` (lines 160–169)
- `get_puzzle_and_solution_for_coin` in `wheel/src/api.rs` (lines 193–194)
- `get_spends_for_trusted_block` / `get_spends_for_trusted_block_with_conditions` in `wheel/src/api.rs` (lines 554–597)

### Impact Explanation

The bytes read by `native_run_block_generator` are the raw CLVM block-generator program. They are deserialized, executed, and their output is used to derive every `SpendConditions` record — coin IDs, puzzle hashes, amounts, aggregate-signature pairs — that the consensus layer accepts or rejects for a block.

If those bytes are mutated mid-parse by a concurrent Python thread:

- **Non-canonical / corrupted CLVM deserialization**: `node_from_bytes_backrefs` may read a structurally inconsistent tree, producing a different (or panicking) result than the bytes the caller intended to submit.
- **Cross-language disagreement**: Two nodes running the same block generator bytes through the Python binding may produce different `SpendBundleConditions` if one experiences the race and the other does not — a deterministic consensus divergence trigger.
- **Incorrect coin accounting**: Mutated `CREATE_COIN` amounts or puzzle hashes pass through `validate_conditions` with wrong values, potentially allowing `addition_amount > removal_amount` to go undetected or causing a valid block to be rejected.

This matches the allowed High impact: *"Streamable/CLVM/serde/Python/wasm boundary parsing bug causes non-canonical bytes, integer confusion, hash mismatch, or cross-language disagreement in consensus-critical data."*

---

### Likelihood Explanation

**Entry path is fully unprivileged**: any caller of the exported Python function `run_block_generator` or `run_block_generator2` controls the `program` and `block_refs` arguments. Passing a `bytearray` (which satisfies `PyBuffer<u8>`) is legal and requires no special privilege.

**Triggering the race** requires a second Python thread to mutate the `bytearray` during the `py.detach()` window. In a production full-node that processes blocks concurrently (asyncio + thread-pool workers, or explicit `threading.Thread`), this window is reachable. The race window is the entire duration of CLVM execution, which for a full block can be tens of milliseconds — wide enough to hit reliably with a spin-loop in a background thread.

Likelihood is **medium** in multi-threaded deployments (common for full nodes) and **low** in purely single-threaded scripts.

---

### Recommendation

1. **Do not drop the `PyBuffer` before the data is fully consumed.** Tie the slice lifetime to the buffer's borrow:

   ```rust
   // Correct: slice lifetime is bounded by buf's borrow
   pub fn py_to_slice<'a>(buf: &'a PyBuffer<u8>) -> &'a [u8] {
       assert!(buf.is_c_contiguous(), "buffer must be contiguous");
       unsafe { std::slice::from_raw_parts(buf.buf_ptr() as *const u8, buf.len_bytes()) }
   }
   ```

   Callers must keep the `PyBuffer` alive for the duration of use. Because `py.detach()` requires `Send` data and `PyBuffer` is not `Send`, this will force the compiler to reject the current pattern, guiding developers toward the correct fix.

2. **Alternatively, copy into an owned `Vec<u8>` before releasing the GIL:**

   ```rust
   pub fn py_to_vec(buf: PyBuffer<u8>) -> Vec<u8> {
       assert!(buf.is_c_contiguous(), "buffer must be contiguous");
       let slice = unsafe {
           std::slice::from_raw_parts(buf.buf_ptr() as *const u8, buf.len_bytes())
       };
       slice.to_vec()   // copy while GIL is held; buf dropped after copy
   }
   ```

   This eliminates the race entirely at the cost of one allocation per call.

3. **Apply the fix to all call sites**: `run_block_generator`, `run_block_generator2`, `additions_and_removals`, `generator_interned_vbytes`, `get_puzzle_and_solution_for_coin`, `get_spends_for_trusted_block`, and `get_spends_for_trusted_block_with_conditions`.

### Proof of Concept

```python
import threading
import chia_rs
from chia_rs import G2Element, ConsensusConstants

# A valid minimal CLVM generator (quote nil → empty spend list)
program = bytearray(b'\xff\x01\x80')

constants = ConsensusConstants(...)  # use TEST_CONSTANTS values

results = []

def mutate():
    """Flip bytes while run_block_generator holds the GIL-released window."""
    import time; time.sleep(0.0001)
    program[0] ^= 0xff   # corrupt first byte mid-parse

t = threading.Thread(target=mutate)
t.start()

# run_block_generator accepts bytearray via PyBuffer protocol
err, msg, conds = chia_rs.run_block_generator(
    program, [], 11_000_000_000, 0, G2Element(), None, constants
)
t.join()

# err / conds may differ from a run without the race,
# demonstrating non-deterministic consensus output from identical input bytes.
print(err, msg, conds)
```
