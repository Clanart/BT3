### Title
Fee-on-Transfer Token Accounting Discrepancy in `OmniBridge::initTransfer` Causes Bridge Undercollateralization — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge::initTransfer` uses `SafeERC20.safeTransferFrom` to pull tokens from the caller but records the caller-supplied `amount` in the emitted event without verifying the actual amount received. For ERC-20 tokens that silently deduct a transfer fee (fee-on-transfer tokens), the bridge vault receives `amount - fee` while the cross-chain message asserts `amount`. The NEAR bridge processes that message at face value and mints the full `amount` of wrapped tokens on NEAR, creating unbacked supply and breaking bridge collateralization.

---

### Finding Description

In `OmniBridge.sol`, the `initTransfer` function handles native (non-bridge) ERC-20 tokens with:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ← user-supplied, not verified against actual receipt
    );
}
``` [1](#0-0) 

Immediately after, the function emits the `InitTransfer` event with the same unverified `amount`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message
);
``` [2](#0-1) 

`SafeERC20.safeTransferFrom` only checks that `transferFrom` does not revert and returns `true`; it does **not** compare the contract's pre- and post-transfer balance. For a fee-on-transfer token the bridge vault ends up holding `amount - fee` while the event records `amount`.

The NEAR bridge's `fin_transfer_callback` ingests the event amount directly:

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
``` [3](#0-2) 

It then mints or releases the full denormalized `amount` to the recipient on NEAR, with no awareness that the EVM vault is short by `fee` tokens.

The symmetric path (`finTransfer` on EVM) later calls:

```solidity
IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
``` [4](#0-3) 

At that point the vault holds `amount - fee` but must release `amount`, either reverting (freezing funds) or, if other depositors have topped up the vault, silently draining their collateral.

An identical root cause exists on the NEAR side: `ft_on_transfer` receives the caller-supplied `amount` from the token contract and passes it directly to `init_transfer` without checking the bridge's actual post-transfer balance:

```rust
pub fn ft_on_transfer(&mut self, sender_id: AccountId, amount: U128, msg: String) {
    ...
    self.init_transfer(sender_id, signer_id, token_id, amount, init_transfer_msg)
``` [5](#0-4) 

The `locked_tokens` counter is then inflated by the full `amount`, overstating the bridge's actual holdings. [6](#0-5) 

---

### Impact Explanation

**High — Balance/accounting corruption that breaks bridge collateralization.**

For every round-trip of a fee-on-transfer token:
- EVM vault is short by `fee` tokens per inbound transfer.
- NEAR mints `amount` wrapped tokens backed by only `amount - fee` real tokens.
- On the return leg, `finTransfer` on EVM must release `amount` but the vault holds `amount - fee`, either reverting (permanent freeze of the return transfer) or draining other users' collateral.

An attacker who sells the over-minted NEAR-side tokens before bridging back extracts `fee` tokens of real value per trip at the expense of other bridge depositors, with no upper bound on repetition.

---

### Likelihood Explanation

**Medium.** The precondition is that a fee-on-transfer ERC-20 token is registered with the bridge (i.e., `ethToNearToken[tokenAddress]` is set and `isBridgeToken` is false). Bridge admins can register arbitrary tokens; several real-world tokens (e.g., STA, PAXG, tokens with dynamic fee switches) implement transfer fees. Once such a token is registered, any unprivileged user can trigger the discrepancy by calling `initTransfer`.

---

### Recommendation

After the `safeTransferFrom` call, measure the actual received amount by comparing pre- and post-transfer balances and use that value for the event and cross-chain message:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(received > fee, "InvalidFee");
amount = uint128(received); // use actual received amount downstream
```

Apply the same pattern on the NEAR side: after `ft_transfer_call` completes, verify the bridge's actual balance increase rather than trusting the `amount` parameter supplied by the token contract.

---

### Proof of Concept

1. Deploy a fee-on-transfer ERC-20 token `FeeToken` that deducts 1% on every `transferFrom`.
2. Admin registers `FeeToken` in `OmniBridge` (sets `ethToNearToken[FeeToken]`; `isBridgeToken` remains `false`).
3. Attacker calls `initTransfer(FeeToken, 1_000_000, 0, 0, nearRecipient, "")`.
   - `safeTransferFrom` moves `1_000_000` from attacker; vault receives `990_000`.
   - Event emits `amount = 1_000_000`.
4. NEAR relayer submits proof; `fin_transfer_callback` mints `1_000_000` wrapped `FeeToken` to `nearRecipient`.
5. Attacker sells `1_000_000` wrapped tokens on NEAR for market value.
6. Any subsequent `finTransfer` on EVM for a return transfer of `1_000_000` tokens will attempt `safeTransfer(recipient, 1_000_000)` but the vault only holds `990_000`, causing the call to revert and permanently freezing the return transfer — or, if other users have deposited `FeeToken`, silently consuming their collateral.
