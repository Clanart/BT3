### Title
Missing Received-Amount Verification After `safeTransferFrom` in `initTransfer` Allows Fee-on-Transfer Tokens to Undercollateralize the Bridge - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer()` calls `safeTransferFrom` to pull native ERC20 tokens from the user but never checks the contract's actual post-transfer balance. The `InitTransfer` event is emitted with the caller-supplied `amount`, not the amount actually received. For fee-on-transfer or rebasing tokens, the bridge holds fewer tokens than it attests to have locked, permanently undercollateralizing the bridge and enabling recipients on the destination chain to withdraw more than the bridge actually holds.

---

### Finding Description

In `OmniBridge.sol`, `initTransfer` handles three token paths. For native ERC20 tokens (neither a bridge token nor a custom-minter token), the code is:

```solidity
// OmniBridge.sol lines 406-412
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
```

Immediately after, the function emits the `InitTransfer` event with the caller-supplied `amount`:

```solidity
// OmniBridge.sol lines 427-436
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,          // <-- unverified; may exceed actual received tokens
    fee,
    nativeFee,
    recipient,
    message
);
```

No balance snapshot is taken before the transfer, and no post-transfer balance check is performed. The `amount` encoded in the event — and subsequently relayed cross-chain — is the caller-supplied parameter, not the actual amount received by the bridge contract.

The same structural flaw exists in the `customMinters` path (lines 394–401): `safeTransferFrom` sends tokens to the custom minter, then `burn(tokenAddress, amount)` is called with the full `amount`, which will fail or corrupt accounting if the minter received fewer tokens due to a transfer fee. ---

### Impact Explanation

**High — Balance/accounting corruption that breaks bridge collateralization.**

The `InitTransfer` event is the authoritative cross-chain message consumed by the NEAR bridge (and Wormhole variant via `initTransferExtension`) to determine how many tokens to release on the destination chain. If the bridge emits `amount = 1000` but only received `950` (due to a 5% transfer fee), the destination chain releases `1000 - bridge_fee` tokens to the recipient. Repeated over many transfers, the bridge's ERC20 reserve is drained below its liabilities. Eventually, legitimate redemptions via `finTransfer` will fail because the bridge cannot cover its obligations, permanently freezing user funds. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation

**Medium.** The bridge is permissionless for `initTransfer` — any address can call it with any ERC20 token address that is not already registered as a bridge token or custom-minter token. Fee-on-transfer tokens (e.g., USDT on some chains, STA, DEFLATIONARY tokens) and rebasing tokens exist on mainnet. A user can legitimately bridge such a token, or an attacker can deploy a minimal fee-on-transfer ERC20 and bridge it to drain the bridge's accounting. No privileged role is required. [5](#0-4) 

---

### Recommendation

Capture the bridge contract's balance of `tokenAddress` before and after the `safeTransferFrom` call, and use the delta as the authoritative `amount` for the event and all downstream accounting:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 received = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
    require(received > 0, "No tokens received");
    amount = uint128(received); // use actual received amount
}
```

Apply the same pattern to the `customMinters` path: measure the minter's balance delta before calling `burn`. [6](#0-5) 

---

### Proof of Concept

1. Deploy a fee-on-transfer ERC20 token `FeeToken` with a 10% transfer fee. It is not registered as a bridge token or custom minter.
2. Call `OmniBridge.initTransfer(feeTokenAddress, 1000, 0, 0, "near:recipient.near", "")`.
3. Inside `initTransfer`, `safeTransferFrom(msg.sender, address(this), 1000)` executes. Due to the 10% fee, the bridge receives only `900` tokens.
4. The function emits `InitTransfer(..., amount=1000, ...)`.
5. The NEAR bridge (or Wormhole relay) observes the event and authorizes a release of `1000` tokens (minus bridge fee) to `recipient.near`.
6. The bridge's ERC20 balance is `900` but its liability is `1000`. The bridge is undercollateralized by `100` tokens per transfer.
7. After enough such transfers, `finTransfer` calls that attempt `IERC20(tokenAddress).safeTransfer(recipient, amount)` will revert due to insufficient balance, permanently freezing those users' funds. [7](#0-6) [8](#0-7)

        currentOriginNonce += 1;

        uint256 extensionValue;

            if (fee != 0) {
                revert InvalidFee();

            extensionValue = msg.value - amount - nativeFee;

            extensionValue = msg.value - nativeFee;
