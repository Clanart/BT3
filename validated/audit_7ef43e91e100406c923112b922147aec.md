### Title
Arithmetic Overflow in `denormalize_amount` Permanently Freezes User Funds During EVM→NEAR Bridge Finalization - (File: `near/omni-bridge/src/lib.rs`)

### Summary

`denormalize_amount` performs an unchecked `u128` multiplication that overflows when a user bridges a sufficiently large token amount from EVM to NEAR. Because the NEAR workspace is compiled with `overflow-checks = true`, the overflow causes a hard panic in `fin_transfer_callback`, permanently locking the user's already-burned/locked EVM tokens with no recovery path.

### Finding Description

`denormalize_amount` converts a normalized (NEAR-side) token amount back to the origin-chain denomination by multiplying by `10^(origin_decimals - decimals)`:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← unchecked u128 multiply
}
``` [1](#0-0) 

The workspace `Cargo.toml` sets `overflow-checks = true` in the release profile, so any integer overflow causes an immediate NEAR contract panic (trap): [2](#0-1) 

`denormalize_amount` is called unconditionally inside `fin_transfer_callback` with the amount extracted directly from the submitted EVM proof:

```rust
amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
``` [3](#0-2) 

`init_transfer.amount` originates from the EVM `InitTransfer` event, which is emitted after the user's tokens are already burned or locked on EVM. The EVM `initTransfer` function accepts `uint128 amount`, so the full `[0, 2^128-1]` range is valid on the EVM side: [4](#0-3) 

For any token where `origin_decimals > decimals` (e.g., a 24-decimal EVM token normalized to 18 decimals on NEAR, giving `diff = 6`), the overflow threshold is:

```
overflow_threshold = u128::MAX / 10^diff
                   = 2^128 / 10^6
                   ≈ 3.4 × 10^32   (for diff = 6)
```

Any EVM `initTransfer` call with `amount > overflow_threshold` will produce a proof that, when submitted to NEAR, always panics in `fin_transfer_callback`. There is no on-chain refund or recovery mechanism on the EVM side once `initTransfer` has executed.

`denormalize_fee` has the same pattern and is called in the same callback: [5](#0-4) 

### Impact Explanation

The user's EVM tokens are burned or locked at the moment `initTransfer` executes. If the resulting proof always causes `fin_transfer_callback` to panic, the NEAR side never credits the recipient and the EVM side has no refund path. The funds are permanently and irrecoverably frozen — matching the Critical allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation

The attack is self-inflicted (the user loses their own funds) unless a griefing scenario is constructed where an attacker front-runs a victim's large transfer. Realistic likelihood is **medium**: tokens with non-trivial decimal differences (e.g., 24 vs 18, diff = 6) are common, and the overflow threshold (`~3.4 × 10^32` raw units for diff = 6, `~3.4 × 10^20` for diff = 18) is reachable for high-supply tokens or tokens with many decimals. The absence of any pre-call bounds check makes the path fully deterministic once triggered.

### Recommendation

Replace the bare multiplication with a checked variant and propagate the error:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    let multiplier = 10_u128.pow(diff_decimals);
    amount
        .checked_mul(multiplier)
        .unwrap_or_else(|| env::panic_str(BridgeError::AmountOverflow.as_ref()))
}
```

Apply the same fix to `denormalize_fee`. Additionally, add an upper-bound validation in `fin_transfer_callback` before calling `denormalize_amount`, rejecting proofs whose amounts would overflow before any state is mutated.

### Proof of Concept

1. Protocol registers a token with `origin_decimals = 24`, `decimals = 18` (`diff = 6`).
2. User calls EVM `initTransfer` with `amount = 2^128 / 10^6 + 1` (just above the overflow threshold). Tokens are burned on EVM.
3. Relayer (or user) submits the resulting proof to NEAR via `fin_transfer`.
4. NEAR verifies the proof successfully and calls `fin_transfer_callback`.
5. `denormalize_amount((2^128/10^6 + 1), Decimals{decimals:18, origin_decimals:24})` computes `(2^128/10^6 + 1) * 10^6`, which exceeds `u128::MAX`.
6. With `overflow-checks = true`, the NEAR runtime traps. The callback panics.
7. The NEAR transfer is never recorded; the EVM tokens remain burned with no refund. Funds are permanently lost.
