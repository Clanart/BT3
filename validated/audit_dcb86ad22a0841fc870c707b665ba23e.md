### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Bricking — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the supplied new class hash corresponds to a previously declared contract class. A contract can therefore replace its own class hash with an arbitrary, undeclared value. Once committed to state, every subsequent call to that contract will fail because the OS cannot resolve the class to a compiled implementation, permanently freezing any funds held by the contract.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads the caller-supplied class hash directly from the syscall request and writes it into `contract_state_changes` without any membership check against the set of declared classes. The code itself acknowledges the gap with an explicit TODO:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The identical omission exists in the Cairo-0 path:

```cairo
func execute_replace_class{contract_state_changes: DictAccess*, revert_log: RevertLogEntry*}(
    contract_address, syscall_ptr: ReplaceClass*
) {
    alloc_locals;
    let class_hash = syscall_ptr.class_hash;
    ...
    dict_update{dict_ptr=contract_state_changes}(...);
``` [2](#0-1) 

**Analogy to the external report:** In the Suzaku bug, `getVaults(epoch)` silently omits vaults that were removed after rewards were distributed, so rewards for those vaults are never claimed and remain locked. Here, `execute_replace_class` silently accepts a class hash that was never declared, so the contract's implementation can never be resolved and all funds inside it are locked. Both are cases where a dynamic registry (vault list / declared-class set) can be placed in an inconsistent state because the consuming code performs no membership check.

The declared-class registry is built during block execution via `contract_class_changes` (populated by `execute_declare_transaction`):

```cairo
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [3](#0-2) 

`execute_replace_class` never consults `contract_class_changes` or any equivalent hint to confirm the target class exists.

---

### Impact Explanation

Once a contract's class hash is set to an undeclared value and the block is proven and accepted on L1:

1. Every future call to that contract requires the OS to resolve the class hash to a compiled class. Because no such class exists, proof generation for any block containing such a call will fail or the call will be permanently skipped by the sequencer.
2. All assets (ERC-20 balances, ETH, NFTs) stored in the contract's storage become permanently inaccessible — no `transfer`, `withdraw`, or any other entry point can be executed.
3. The state root committed on L1 reflects the bricked class hash; there is no on-chain recovery path.

**Impact: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The attacker-controlled entry path requires only the ability to deploy a contract (an unprivileged operation):

1. Attacker deploys Contract A that holds or will receive user funds.
2. Contract A exposes a function (or is designed) to call `replace_class(undeclared_hash)`.
3. The OS processes the syscall, writes the undeclared hash to state, and produces a valid proof — no validation fails.
4. After the block is finalized, Contract A is permanently bricked.

The explicit `TODO` comment in the production OS code confirms the check is absent and was not accidentally omitted. The same gap exists in the deprecated (Cairo-0) syscall path, widening the attack surface to legacy contracts.

---

### Recommendation

Before updating `contract_state_changes`, verify that `class_hash` is present in the declared-class mapping. Concretely, the OS should assert that `class_hash` appears as a key in `contract_class_changes` (for classes declared in the same block) **or** confirm via a hint that the class was declared in a prior block (analogous to how `execute_get_class_hash_at` reads state entries). The check should mirror the enforcement already applied to `execute_declare_transaction`, which uses `prev_value=0` to guarantee a class is declared at most once:

```cairo
// Proposed addition inside execute_replace_class:
// Assert class_hash is a known declared class (either current block or prior state).
assert_declared_class(class_hash);  // to be implemented via contract_class_changes lookup + hint
```

The same fix must be applied to `execute_replace_class` in `deprecated_execute_syscalls.cairo`.

---

### Proof of Concept

```cairo
// Malicious contract (simplified pseudocode)
@external
func brick_self{syscall_ptr: felt*, ...}() {
    // 0xdeadbeef is not a declared class hash
    replace_class(class_hash=0xdeadbeef);
    return ();
}
```

1. Attacker deploys the contract above and induces users to deposit funds.
2. Attacker calls `brick_self()`.
3. The OS executes `execute_replace_class` with `class_hash = 0xdeadbeef`, writes it to `contract_state_changes` without any check, and includes it in the proven state update.
4. After the block is finalized on L1, the contract's class hash in the global state is `0xdeadbeef`.
5. Any subsequent transaction targeting the contract cannot be proven (no compiled class exists for `0xdeadbeef`); the sequencer must permanently skip it.
6. All deposited funds are frozen with no recovery path.
