Audit Report

## Title
Missing Deadline Check in `MetricOmmPoolLiquidityAdder` Allows Stale Liquidity Deposits at Unfavorable Prices — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
`MetricOmmPoolLiquidityAdder` exposes four public liquidity-add entry points — two overloads of `addLiquidityExactShares` and two of `addLiquidityWeighted` — none of which accept or enforce a deadline. A transaction delayed in the mempool can execute after significant price movement, depositing the caller's tokens into bins that are now far from the current cursor, causing immediate loss of LP principal through stale-price execution. `MetricOmmSimpleRouter` enforces `_checkDeadline` on every swap entry point, but this protection is entirely absent from the sibling liquidity contract.

## Finding Description
`MetricOmmSwapRouterBase._checkDeadline` (L91–94) is called at the top of all four swap functions in `MetricOmmSimpleRouter` (`exactInputSingle` L68, `exactInput` L93, `exactOutputSingle` L131, `exactOutput` L155). `MetricOmmPoolLiquidityAdder` does not inherit from `MetricOmmSwapRouterBase` and contains no `deadline` parameter or time-based guard in any of its four public entry points (L56–68, L71–81, L88–116, L123–149).

The only quantity-bounding guard is the `maxAmountToken0`/`maxAmountToken1` cap enforced in the callback at L165–167. This caps the *number of tokens* the pool may pull, but does not protect the *price* at which those tokens are committed. In a bin AMM, bins below the cursor hold only token1 and bins above hold only token0; the active bin holds both. If the cursor moves significantly while a transaction is pending, the targeted bins become single-sided. The pool then requests only one token from the callback, and the deposited tokens are priced at the stale bin price. Arbitrageurs immediately swap through those bins at the favorable price, leaving the LP with the opposite token at below-market value — a direct loss of principal.

The `_validateBinAndBinPosition` check in `addLiquidityWeighted` (L263–286) reads `slot0` at probe time and reverts if the cursor is outside the user-specified bounds. This is not equivalent to a deadline: the cursor can remain within bounds while the pool's actual price composition has shifted enough to produce a materially different deposit ratio and stale-price exposure.

## Impact Explanation
Direct loss of LP principal through bad-price execution. In a bin AMM, depositing into out-of-range bins at stale prices is immediately exploitable by arbitrageurs: the LP receives the opposite token at the old bin price, which is below fair-market value at the time of execution. The `maxAmount` caps bound quantity but not price, so the full capped amount can be lost at an unfavorable rate. This matches the "bad-price execution" and "direct loss of user principal" categories in the allowed impact gate.

## Likelihood Explanation
Any unprivileged user calling `addLiquidityExactShares` directly (not via a private mempool) is exposed. No privileged actor is required. Network congestion, gas price spikes, or deliberate transaction ordering by a block builder can delay execution by minutes to hours. The `addLiquidityWeighted` path is partially mitigated by cursor bounds but remains vulnerable when the user sets wide bounds or when the cursor stays within bounds while the deposit ratio shifts materially.

## Recommendation
Add a `deadline` parameter to all four public entry points in `MetricOmmPoolLiquidityAdder` and call `_checkDeadline(deadline)` as the first statement, mirroring the pattern in `MetricOmmSimpleRouter`. The `_checkDeadline` helper already exists in `MetricOmmSwapRouterBase`; `MetricOmmPoolLiquidityAdder` can either inherit from it or duplicate the one-liner:

```solidity
function addLiquidityExactShares(
    address pool,
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 maxAmountToken0,
    uint256 maxAmountToken1,
    uint256 deadline,          // ← add
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _checkDeadline(deadline);  // ← add
    _validateOwner(owner);
    ...
}
```

Apply the same change to both `addLiquidityWeighted` overloads.

## Proof of Concept
1. Pool cursor is at bin 0 (balanced, token0/token1 price = 1:1).
2. Alice calls `addLiquidityExactShares` targeting bins `[-1, 0, 1]` with `maxAmountToken0 = 1000e18`, `maxAmountToken1 = 1000e18`. Transaction enters the public mempool.
3. A large swap moves the cursor to bin 5 (price has risen sharply). Bins `[-1, 0, 1]` are now entirely below the cursor and hold only token1.
4. Alice's transaction is included. The pool callback requests `amount0Delta = 0`, `amount1Delta ≤ 3000e18` (all token1, single-sided). The `maxAmountToken1` cap is satisfied, so no revert occurs.
5. Alice's token1 is deposited at the old bin prices. Arbitrageurs immediately swap token0 for token1 through those bins at the favorable (below-market) price.
