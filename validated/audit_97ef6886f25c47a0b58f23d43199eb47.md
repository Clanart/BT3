Audit Report

## Title
Missing Deadline Guard on All Liquidity-Adding Entry Points Allows Stale Transactions to Deposit at Unfavorable Oracle Prices - (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

## Summary
`MetricOmmPoolLiquidityAdder` exposes four public entry points — two overloads each of `addLiquidityExactShares` and `addLiquidityWeighted` — none of which accept or enforce a `deadline` parameter. The sibling `MetricOmmSimpleRouter` enforces `_checkDeadline` on every swap entry point via `MetricOmmSwapRouterBase`. A pending liquidity transaction can sit in the mempool and execute at an oracle price the user never intended to accept, deploying capital into bins that are immediately out-of-range and earn no fees.

## Finding Description
`MetricOmmSwapRouterBase._checkDeadline` is defined at lines 91–94 and reverts with `TransactionExpired` if `block.timestamp > deadline`. Every swap function in `MetricOmmSimpleRouter` calls this guard as its first statement (lines 68, 93, 131, 155). `MetricOmmPoolLiquidityAdder`, however, inherits only from `PeripheryPayments` and carries no deadline logic. Its four public entry points at lines 56–81 and 88–149 accept no `deadline` argument and make no timestamp check before invoking `IMetricOmmPoolActions.addLiquidity`.

The `addLiquidityWeighted` overloads include `_validateBinAndBinPosition` (lines 263–286), which checks the pool's internal bin cursor against caller-supplied `minimalCurBin`/`maximalCurBin` bounds. This guard checks the pool's discrete bin index, not the oracle price. The oracle price can move substantially within a single bin range, so a transaction can pass the cursor check while the oracle price has shifted materially. `addLiquidityExactShares` has no secondary guard at all.

The exploit path is straightforward: a user submits a liquidity transaction; the oracle price moves (Pyth/Chainlink update on-chain) before the transaction is included; the transaction executes at the new oracle price, deploying the user's capital into bins that are now out-of-range relative to the user's intent.

## Impact Explanation
Metric OMM is an oracle-anchored market maker. Bin token composition is determined by the live oracle price at the moment `addLiquidity` is called. If the oracle price has moved since the user signed the transaction, capital is deployed into bins that may be immediately single-sided and earn no fees. For `addLiquidityExactShares`, the token ratio consumed is dictated by the live oracle, not the user's intended price. For `addLiquidityWeighted`, the probe step reflects the new oracle price, and the scaled shares are deployed at that price. In both cases the user's principal is locked in LP shares at an oracle price they did not consent to. Recovering requires an additional `removeLiquidity` call (gas cost) and any impermanent-loss-equivalent incurred during the window. This constitutes a direct loss of user principal meeting Medium severity thresholds.

## Likelihood Explanation
No privileged role is required. Any ordinary LP is exposed. Network congestion, gas price spikes, or deliberate transaction withholding by a block builder can delay any pending transaction by multiple blocks. Pyth and Chainlink feeds update continuously; the oracle price can move between blocks during volatile periods. The scenario is realistic and repeatable for any user adding liquidity during market volatility.

## Recommendation
Add a `uint256 deadline` parameter to all four public entry points in `MetricOmmPoolLiquidityAdder` and call `_checkDeadline` (or an equivalent inline check) as the first statement, mirroring the pattern in `MetricOmmSimpleRouter`. To access `_checkDeadline`, `MetricOmmPoolLiquidityAdder` should either inherit from `MetricOmmSwapRouterBase` or inline the equivalent check. Update `IMetricOmmPoolLiquidityAdder` accordingly.

```solidity
function addLiquidityExactShares(
    address pool,
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 maxAmountToken0,
    uint256 maxAmountToken1,
    uint256 deadline,          // ← add
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    if (block.timestamp > deadline) revert TransactionExpired(deadline, block.timestamp);
    _validateOwner(owner);
    _validateDeltas(deltas);
    return _addLiquidity(pool, owner, salt, deltas, msg.sender, maxAmountToken0, maxAmountToken1, extensionData);
}
```

Apply the same change to both `addLiquidityWeighted` overloads and the second `addLiquidityExactShares` overload.

## Proof of Concept
1. Alice calls `addLiquidityExactShares` targeting bin index 4 (above current oracle price) with `maxAmountToken0 = 1000e18`, `maxAmountToken1 = 0`. The oracle price at submission time is P.
2. The transaction sits in the mempool. A Pyth update is pushed on-chain, dropping the oracle price to P/2.
3. The transaction is included. At P/2, bin 4 is now far above the oracle price; the pool accepts Alice's 1000e18 token0 into that bin as a fully single-sided deposit.
4. Alice's 1000e18 token0 is locked in a bin that will not be traversed by swaps until the oracle price rises back above P. She receives no fees and cannot recover her tokens without removing liquidity (paying gas again).
5. Had a deadline of `block.timestamp + 60` been enforced, the transaction would have reverted at step 3, protecting Alice's capital.

A Foundry fork test can reproduce this by: deploying the contracts, submitting the liquidity call, warping `block.timestamp` forward and pushing a new oracle price via the mock feed, then executing the transaction and asserting the deposited bin is out-of-range.
