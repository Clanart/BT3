Audit Report

## Title
Cross-Transaction ETH Stranding Allows Subsequent WETH-Input Swaps to Consume Prior Callers' ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

Every swap entry-point on `MetricOmmSimpleRouter` is declared `external payable`, so ETH sent alongside a non-WETH `tokenIn` swap is silently accepted and left in the router. The `pay` function in `PeripheryPayments` unconditionally reads `address(this).balance` before pulling WETH from the actual payer, meaning any stranded ETH from a prior transaction is deposited as WETH and forwarded to the pool on behalf of the next WETH-input caller. The original ETH depositor suffers a permanent, unrecoverable loss of principal.

## Finding Description

`receive()` rejects plain ETH transfers from non-WETH addresses: [1](#0-0) 

However, `receive()` is not invoked when ETH is attached as `msg.value` to a named `payable` function. All four swap entry-points and `multicall` are `payable`: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

When `tokenIn` is not WETH, `pay` takes the `else` branch and only calls `safeTransferFrom` — the attached ETH is never touched and remains in the router: [7](#0-6) 

When the next caller swaps with `tokenIn = WETH`, `pay` reads `address(this).balance` with no per-transaction accounting: [8](#0-7) 

If `nativeBalance >= value`, the entire payment is sourced from the router's accumulated ETH — the actual payer's WETH allowance is never touched. If `0 < nativeBalance < value`, the stranded ETH partially covers the payment. In both cases the original depositor's ETH is permanently consumed with no recourse.

## Impact Explanation

Direct, permanent loss of user principal. User A's ETH is silently consumed to subsidize User B's WETH-input swap. This matches the allowed impact gate: **Critical/High direct loss of user principal**. The loss is unbounded (any amount of ETH stranded in the router is at risk), requires no privileged access, no malicious pool, and no non-standard token behavior.

## Likelihood Explanation

All swap functions are `payable`, so any wallet or front-end can silently attach ETH to a non-WETH swap. Omitting `refundETH()` in a `multicall` batch is a realistic and common user error. An on-chain attacker can monitor the mempool for transactions that strand ETH and immediately follow with a WETH-input swap to drain the balance atomically. No special permissions or setup are required; the attack is repeatable and permissionless.

## Recommendation

Remove the `nativeBalance` shortcut from `pay` entirely, or cap consumable native balance to the `msg.value` of the current transaction by storing it in transient storage at each swap entry-point. The simplest safe fix:

```solidity
function pay(address token, address payer, address recipient, uint256 value) internal {
    if (payer == address(this)) {
        IERC20(token).safeTransfer(recipient, value);
    } else {
        IERC20(token).safeTransferFrom(payer, recipient, value);
    }
}
```

Callers who wish to pay with native ETH should use an explicit `multicall([wrapETH, swap, refundETH])` pattern where `wrapETH` wraps a known amount before the swap.

## Proof of Concept

```solidity
// 1. User A sends ETH with a non-WETH swap (e.g., accidentally or via a UI bug)
router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
    tokenIn: address(token1),  // NOT WETH — pay() takes the else branch
    tokenOut: address(token2),
    amountIn: 1000,
    ...
}));
// router.balance == 1 ether; User A's ETH is stranded

// 2. User B exploits the stranded ETH
uint256 wethBefore = weth.balanceOf(userB);
router.exactInputSingle(ExactInputSingleParams({
    tokenIn: address(weth),
    amountIn: 1 ether,
    ...
}));
// pay() reads nativeBalance == 1 ETH >= value == 1 ETH
// deposits User A's ETH as WETH, transfers to pool
// weth.balanceOf(userB) == wethBefore  (User B paid nothing)
// User A's 1 ETH is permanently gone
```
