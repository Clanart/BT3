### Title
Zero Recipient Address Accepted in `init_transfer` Causes Permanent Irrecoverable Lock of User Funds — (File: `near/omni-bridge/src/lib.rs`)

### Summary

The NEAR bridge's `init_transfer` function accepts a zero EVM address (`OmniAddress::Eth(H160::ZERO)`) as the recipient without any zero-address guard. Once tokens are burned or locked on NEAR, every subsequent attempt to finalize the transfer on EVM reverts because OpenZeppelin's `_mint` rejects `address(0)`. No cancel or refund path exists on NEAR for pending transfers, making the locked/burned funds permanently irrecoverable.

### Finding Description

`init_transfer` in `near/omni-bridge/src/lib.rs` performs only one recipient check:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
``` [1](#0-0) 

The `OmniAddress` type exposes an `is_zero()` predicate that covers all chain variants, but it is never called here. [2](#0-1) 

After the chain-kind check passes, `init_transfer_internal` immediately burns or locks the user's tokens:

```rust
self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
self.lock_tokens_if_needed(
    transfer_message.get_destination_chain(),
    &token_id,
    transfer_message.amount.0,
);
``` [3](#0-2) 

The MPC signer then signs a `TransferMessagePayload` that encodes `recipient = address(0)`: [4](#0-3) 

On the EVM side, `finTransfer` attempts to mint to `payload.recipient`:

```solidity
IBridgeToken(payload.tokenAddress).mint(
    payload.recipient,   // address(0)
    payload.amount
);
``` [5](#0-4) 

`BridgeToken.mint` delegates to OpenZeppelin's `_mint`, which unconditionally reverts for `address(0)`:

```solidity
function mint(address beneficiary, uint256 amount) external onlyOwner {
    _mint(beneficiary, amount);   // reverts: "ERC20: mint to the zero address"
}
``` [6](#0-5) 

Because the MPC signature is cryptographically bound to the zero recipient, no valid `finTransfer` call can ever succeed. A grep across all NEAR bridge source files finds no `cancel_transfer`, `refund_transfer`, or equivalent public function — the pending transfer entry in `pending_transfers` can never be removed by the user.

For native ERC20 tokens (non-bridge tokens), `safeTransfer(address(0), amount)` also reverts under OpenZeppelin. For native ETH (`tokenAddress == address(0)`), the ETH call to `address(0)` succeeds, meaning ETH is permanently burned on EVM while the NEAR-side tokens are also burned — a double-sided loss.

### Impact Explanation

- **Permanent irrecoverable lock / burn of user funds.** Tokens burned or locked on NEAR cannot be recovered because `finTransfer` on EVM will always revert, and no cancel/refund path exists on NEAR.
- Matches allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation

Any unprivileged bridge user can trigger this by supplying `"eth:0x0000000000000000000000000000000000000000"` (or the equivalent for Arb, Base, Bnb, Pol, etc.) as the recipient in `ft_transfer_call`. This is a realistic user mistake (copy-paste of a placeholder address, misconfigured tooling) and requires no special privilege. The same zero-address path is reachable from StarkNet's `init_transfer` and the EVM's `initTransfer` (with an empty or zero NEAR recipient string), though the NEAR-origin path is the clearest because token burning is irreversible.

### Recommendation

Add a zero-address guard immediately after the chain-kind check in `init_transfer`:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
require!(
    !init_transfer_msg.recipient.is_zero(),
    BridgeError::InvalidRecipientAddress.as_ref()
);
```

Apply the same guard in `finish_withdraw_v2` and in the EVM `initTransfer` (reject empty `recipient` string) and StarkNet `init_transfer` (reject empty `recipient` ByteArray). Additionally, consider introducing a DAO-gated `cancel_transfer` function that can refund locked tokens and remove the pending entry for transfers that can never be finalized.

### Proof of Concept

1. User calls `ft_transfer_call` on a NEAR NEP-141 token with:
   ```json
   {
     "receiver_id": "omni-bridge.near",
     "amount": "1000000000000000000",
     "msg": "{\"InitTransfer\":{\"recipient\":\"eth:0x0000000000000000000000000000000000000000\",\"fee\":\"0\",\"native_token_fee\":\"0\"}}"
   }
   ```
2. `ft_on_transfer` → `init_transfer` passes the chain-kind check (Eth ≠ Near). [1](#0-0) 
3. `init_transfer_internal` burns the bridge token (or locks the native token) on NEAR. [3](#0-2) 
4. Relayer calls `sign_transfer`; MPC signs `TransferMessagePayload { recipient: OmniAddress::Eth(H160::ZERO), ... }`. [4](#0-3) 
5. Relayer calls `OmniBridge.finTransfer(signature, payload)` on Ethereum. `payload.recipient = address(0)`.
6. `BridgeToken.mint(address(0), amount)` → `_mint(address(0), ...)` → **reverts**. [6](#0-5) 
7. The entire EVM transaction reverts. The NEAR-side tokens remain burned/locked with no recovery path. User funds are permanently lost.
