### Title
`CommitmentStateDiff`→`ThinStateDiff` Conversion Silently Drops `deprecated_declared_classes`, Producing a Wrong `state_diff_commitment` in the Block Hash - (File: `crates/blockifier/src/state/cached_state.rs`)

---

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` implementation in `crates/blockifier/src/state/cached_state.rs` unconditionally sets `deprecated_declared_classes: Vec::new()`. This is the conversion used by `BlockExecutionArtifacts::new` in `crates/apollo_batcher/src/block_builder.rs` to compute `calculate_block_commitments` → `calculate_state_diff_hash` → `state_diff_commitment`. Any block that contains a deprecated (Cairo 0) `Declare` transaction will have its deprecated class hashes silently omitted from the state diff fed into the commitment hash, producing a wrong `state_diff_commitment`, a wrong `concatenated_counts` (wrong `state_diff_length`), and therefore a wrong `PartialBlockHash` / block hash commitment.

---

### Finding Description

**Root cause – the uninitialized/hardcoded-zero field:**

```rust
// crates/blockifier/src/state/cached_state.rs  lines 690-701

    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            deployed_contracts: commitment_state_diff.address_to_class_hash,
            storage_diffs: commitment_state_diff.storage_updates,
            class_hash_to_compiled_class_hash: commitment_state_diff
                .class_hash_to_compiled_class_hash,
            nonces: commitment_state_diff.address_to_nonce,
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),   // ← always empty
        }
    }
}
``` [1](#0-0) 

`CommitmentStateDiff` has no `deprecated_declared_classes` field at all:

```rust

---

### Impact Explanation

**Impact: Critical – Wrong state_diff_commitment and block hash commitment for any block containing a deprecated Declare transaction.**

1. `state_diff_commitment` fed into `calculate_block_hash` is wrong (deprecated class hashes are missing from the Poseidon hash).
2. `concatenated_counts` (`packed_lengths`) is wrong because `state_diff_length` is under-counted.
3. Both fields are chained into the block hash:

```rust
.chain(&block_commitments.concatenated_counts)
.chain(&block_commitments.state_diff_commitment.0.0)
``` [10](#0-9) 

4. The `PartialBlockHash` (proposal commitment) used by consensus is derived from these wrong values, so consensus nodes agree on a block hash that does not match the correct Starknet protocol hash for that block.
5. The `ThinStateDiff` written to storage is also missing the deprecated classes, so any downstream consumer (state sync, RPC `starknet_getStateUpdate`, proof manager) receives an incomplete state diff.

This matches the allowed impact scope: **"Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input"** and **"Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution"** (the class hash is omitted from the commitment entirely).

---

### Likelihood Explanation

Deprecated (Cairo 0) `Declare` transactions are still valid on Starknet and can be submitted by any unprivileged user. Any such transaction included in a block triggers the bug. The sequencer is the proposer, so the wrong commitment is produced during normal block building without any special attacker capability.

---

### Recommendation

`CommitmentStateDiff` should be extended to carry `deprecated_declared_classes`, or the conversion `From<CommitmentStateDiff> for ThinStateDiff` must be supplied the deprecated class hashes from the transaction execution context (as is already done in `induced_state_diff` in `crates/apollo_rpc_execution/src/execution_utils.rs`).

At minimum, remove the hardcoded `deprecated_declared_classes: Vec::new()` and populate it from the actual execution output before calling `calculate_block_commitments`.

---

### Proof of Concept

1. Submit a Cairo 0 `Declare` transaction to the sequencer (any unprivileged user can do this).
2. The batcher includes it in a block and calls `BlockExecutionArtifacts::new`.
3. Inside `new`, `ThinStateDiff::from(commitment_state_diff.clone())` is called; `deprecated_declared_classes` is `[]`.
4. `calculate_block_commitments` computes `state_diff_length = N` (missing the 1 deprecated class), and `calculate_state_diff_hash` hashes `0` deprecated classes instead of `1`.
5. The resulting `header_commitments.state_diff_commitment` and `header_commitments.concatenated_counts` differ from the correct values.
6. `PartialBlockHash::from_partial_block_hash_components` produces a wrong proposal commitment that consensus nodes sign and agree on.
7. The final block hash stored in the chain is wrong relative to the Starknet protocol specification for any block containing a deprecated Declare.
