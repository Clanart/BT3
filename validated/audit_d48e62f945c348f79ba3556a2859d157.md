### Title
Integer Overflow in Relative Timelock Arithmetic Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Spend Conditions — (`File: crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` evaluates relative timelock conditions using `wrapping_add` when the `nowrap` flag is `false`. An attacker-controlled puzzle that emits `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` with a near-maximum integer value causes the deadline to wrap around to a value smaller than the current block height or timestamp, making the timelock check pass immediately. This is the direct analog of the `BlockhashRegistry.searchForAvailableBlock` unaddressed overflow: both involve unchecked integer addition in a range/bound check that is the sole guard against premature execution.

---

### Finding Description

`check_time_locks` accepts a `nowrap: bool` parameter. When `nowrap = false` (the legacy pre-fix path), all four relative timelock comparisons use `wrapping_add`:

```
// height_relative, nowrap=false branch
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
```

```
// seconds_relative, nowrap=false branch
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative)
```

```
// before_height_relative, nowrap=false branch
} else if prev_transaction_block_height
    >= unspent.confirmed_block_index.wrapping_add(before_height_relative)
```

```
// before_seconds_relative, nowrap=false branch
} else if timestamp >= unspent.timestamp.wrapping_add(before_seconds_relative)
```

The `nowrap = true` path (the corrected behavior) uses `saturating_add` instead, which clamps at the type maximum and preserves the intended semantics.

**Concrete overflow scenario for `ASSERT_HEIGHT_RELATIVE`:**

- Coin confirmed at block `100`.
- Puzzle emits `ASSERT_HEIGHT_RELATIVE = 0xFFFF_FFFF` (u32::MAX).
- Expected deadline: `100 + 0xFFFF_FFFF` → should saturate to `u32::MAX` (≈4.3 billion blocks in the future).
- Actual computation with `wrapping_add`: `100u32.wrapping_add(0xFFFF_FFFF) = 99`.
- Check: `prev_height < 99` → false for any `prev_height ≥ 99`, so the check **passes** (spend accepted) immediately.

The same overflow applies to `ASSERT_SECONDS_RELATIVE` with `u64::MAX`, and the inverse wrapping for `ASSERT_BEFORE_HEIGHT_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE` causes the opposite error (a valid spend is incorrectly rejected).

The codebase's own tests explicitly document and confirm this behavior:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),
)]
```

```
// 2000 < 1000 + u64::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::seconds_relative_wrap(
    Osc { seconds_relative: Some(0xffff_ffff_ffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed)),
    Ok(()),
)]
```

The Python test suite (`tests/test_check_time_locks.py`) also parametrizes over `nowrap=True` and `nowrap=False` and confirms the wrapping bypass.

---

### Impact Explanation

**High — Timelock validation bypass enables unauthorized spend acceptance.**

A coin whose puzzle enforces `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` with a near-maximum integer value is intended to be unspendable for an astronomically long period. When `nowrap=false` is active, the wrapping arithmetic makes the deadline appear to have already passed, and the spend is accepted immediately. This allows a coin that should be locked for ~4 billion blocks (or ~584 billion years) to be spent at the block it was confirmed, bypassing the puzzle's security guarantee entirely.

For `ASSERT_BEFORE_HEIGHT_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE`, the wrapping causes the opposite: a spend that should be valid is incorrectly rejected, which can cause consensus divergence between nodes running different `nowrap` configurations.

---

### Likelihood Explanation

The `nowrap` parameter is not determined inside the Rust consensus engine — it is passed in by the Python full node at call time. The `nowrap=false` path is the legacy behavior retained for backward compatibility with blocks validated before the saturating-add fix was introduced. If the Python full node passes `nowrap=false` for any block at or after the chain tip (i.e., for new spend bundles being validated), the overflow is directly exploitable by any unprivileged user who can craft a puzzle with a large relative timelock argument. The attacker controls the puzzle and solution entirely; no privileged access is required.

---

### Recommendation

1. **Remove the `nowrap=false` code path entirely** once the activation height for the saturating-add fix has passed on all supported networks. Retaining a known-broken arithmetic path in production consensus code is unnecessary risk.
2. If backward compatibility must be preserved, **add an explicit guard** that rejects any `height_relative` or `seconds_relative` value that would cause `confirmed_index + relative > type::MAX` before performing the addition, analogous to the `checked_add` pattern used elsewhere in the codebase (e.g., `pot_iterations.rs`).
3. **Document the activation height** at which `nowrap` transitions from `false` to `true` directly in `check_time_locks.rs`, so the invariant is auditable without reading the Python full node.

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Attacker constructs a ChiaLisp puzzle that emits `(ASSERT_HEIGHT_RELATIVE . 0xFFFFFFFF)` (condition opcode 82, argument = u32::MAX).
2. Attacker funds and confirms a coin locked by this puzzle at block height `H`.
3. Attacker immediately submits a spend bundle for this coin at block `H+1`.
4. The Python full node calls `check_time_locks(..., nowrap=false)`.
5. Inside `check_time_locks`: `H.wrapping_add(0xFFFF_FFFF) = H - 1`. The check `H+1 < H-1` is false → spend passes.
6. The coin is spent at block `H+1` despite the puzzle intending a ~4.3 billion block lockup.
