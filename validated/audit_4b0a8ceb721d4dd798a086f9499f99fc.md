### Title
L1-to-L2 Message Not Consumed When L1 Handler Reverts, Enabling Permanent Fund Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_l1_handler_transaction` returns early without calling `consume_l1_to_l2_message` when the sequencer marks the transaction as reverted. This means the L1-to-L2 message is never recorded in the OS output (`outputs.messages_to_l2`), creating an irreconcilable inconsistency between L1 state (message sent, funds locked) and L2 OS output (message never consumed). An unprivileged L1 message sender can trigger this path by crafting a message that causes the L2 handler to revert.

---

### Finding Description

In `execute_l1_handler_transaction`, the sequencer hint `%{ IsReverted %}` sets `is_reverted`. When `is_reverted != FALSE`, the function returns immediately:

```cairo
local is_reverted;
%{ IsReverted %}
// Skip the execution step for reverted transaction.
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
``` [1](#0-0) 

This early return skips the entire body of the function, including the critical call to `consume_l1_to_l2_message`:

```cairo
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
``` [2](#0-1) 

`consume_l1_to_l2_message` is the function that appends the message to `outputs.messages_to_l2`, which is the OS-level record used by the L1 verifier to mark the message as consumed:

```cairo
func consume_l1_to_l2_message{outputs: OsCarriedOutputs*}(
    execution_context: ExecutionContext*, nonce: felt
) {
    // ...
    assert [outputs.messages_to_l2] = MessageToL2Header(...);
    // ...
}
``` [3](#0-2) 

The OS output (`OsCarriedOutputs`) is accumulated per block and written into the final `OsOutput`: [4](#0-3) 

When the L1 handler reverts, the L2 state changes are correctly rolled back via the revert log mechanism. However, the L1-to-L2 message is **not** recorded in the OS output at all. The L1 contract, which locked funds when the message was sent, relies on the OS output to confirm consumption. Since the message never appears in the output, the L1 contract never marks it as consumed, and the locked funds cannot be released.

The developer's own TODO comment acknowledges this path is incomplete:

```cairo
// TODO(Yoni): currently, the contract state is not fetched for reverted L1 handlers.
//   Once block hash is supported, we should fetch the contract state for them as well.
``` [5](#0-4) 

---

### Impact Explanation

This is a **Critical: Permanent freezing of funds** vulnerability.

When an L1 user sends a message (e.g., a bridge deposit locking ETH or ERC-20 tokens on L1), the L1 contract holds those funds pending L2 confirmation. The L1 verifier only releases or marks the message consumed when it appears in the proven OS output's `messages_to_l2`. If the L2 handler reverts and the message is absent from the OS output, the L1 contract has no mechanism to release the funds through normal protocol operation. The funds are permanently frozen in the L1 bridge contract for any message whose L2 handler deterministically reverts.

---

### Likelihood Explanation

Any unprivileged L1 user can trigger this path:

1. The attacker sends an L1-to-L2 message targeting an L2 contract whose handler will revert (e.g., due to out-of-gas, a failing assertion in the handler, or a contract that has been upgraded to always revert).
2. The sequencer executes the L2 handler, observes the revert, and sets `is_reverted = TRUE` via the `%{ IsReverted %}` hint.
3. The OS code takes the early-return path, omitting `consume_l1_to_l2_message`.
4. The proven block output does not contain the message, so the L1 verifier never marks it consumed.

No privileged access, leaked keys, or Sybil attacks are required. The attacker only needs to send an L1 message — a standard, permissionless protocol action.

---

### Recommendation

The L1-to-L2 message **must** be consumed (recorded in `outputs.messages_to_l2`) regardless of whether the L2 handler reverts. The correct fix mirrors the recommendation in the external report: perform the message consumption unconditionally, and separately roll back only the L2 state changes via the revert log. Concretely:

1. Move the `consume_l1_to_l2_message` call **before** the `is_reverted` check (or execute it in both branches).
2. Ensure the execution context and nonce are resolved even for reverted L1 handlers (the current TODO acknowledges this is missing).
3. Only skip the L2 state-mutating execution when `is_reverted != FALSE`, not the message consumption.

---

### Proof of Concept

1. Deploy an L2 contract whose `@l1_handler` always reverts (e.g., `assert 1 = 0`).
2. From L1, call the StarkNet core contract's `sendMessageToL2`, sending ETH to that L2 contract's handler. The ETH is locked in the L1 bridge.
3. The sequencer includes the L1 handler transaction in a block. The handler reverts. The sequencer sets `is_reverted = 1`.
4. `execute_l1_handler_transaction` hits the early return at line 387–390, skipping `consume_l1_to_l2_message`.
5. The proven OS output for the block contains no entry for this message in `messages_to_l2`.
6. The L1 verifier processes the proof and output. The message is absent, so the L1 bridge contract never marks it consumed.
7. The ETH sent in step 2 is permanently locked in the L1 bridge with no protocol-level path to recovery.

func consume_l1_to_l2_message{outputs: OsCarriedOutputs*}(
    execution_context: ExecutionContext*, nonce: felt
) {
    assert_not_zero(execution_context.calldata_size);
    // The payload is the calldata without the from_address argument (which is the first).
    let payload: felt* = execution_context.calldata + 1;
    tempvar payload_size = execution_context.calldata_size - 1;

    tempvar execution_info = execution_context.execution_info;

    // Write the given transaction to the output.
    assert [outputs.messages_to_l2] = MessageToL2Header(
        from_address=[execution_context.calldata],
        to_address=execution_info.contract_address,
        nonce=nonce,
        selector=execution_info.selector,
        payload_size=payload_size,
    );

    let message_payload = cast(outputs.messages_to_l2 + MessageToL2Header.SIZE, felt*);
    memcpy(dst=message_payload, src=payload, len=payload_size);

    let (outputs) = os_carried_outputs_new(
        messages_to_l1=outputs.messages_to_l1,
        messages_to_l2=outputs.messages_to_l2 + MessageToL2Header.SIZE +
        outputs.messages_to_l2.payload_size,
    );
    return ();
```
