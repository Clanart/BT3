### Title
Broken Overflow Guard in `CheckEthGasConsume` Allows Block Gas Limit Bypass via uint64 Underflow — (`ante/eth.go`)

---

### Summary

The overflow guard in `CheckEthGasConsume` (`ante/eth.go`) uses the expression `math.MaxInt64 - gasLimit` in a `uint64` context. When `gasLimit > math.MaxInt64`, this subtraction **underflows** as unsigned arithmetic, making the guard permanently false. An attacker can submit a batch `MsgEthereumTx` with two messages each carrying `gas = math.MaxUint64/2 + 1`, causing `gasWanted` to silently wrap to `0` and pass the block gas limit check unconditionally.

---

### Finding Description

The relevant code in `CheckEthGasConsume`:

```go
gasLimit := msgEthTx.GetGas()           // uint64, no upper bound enforced
if gasWanted > math.MaxInt64-gasLimit { // ← BUGGY: underflows when gasLimit > MaxInt64
    return ctx, fmt.Errorf("gasWanted(%d) + gasLimit(%d) overflow", ...)
}
gasWanted += gasLimit                   // ← wraps silently if guard is bypassed
``` [1](#0-0) 

`GetGas()` returns the raw `uint64` gas field of the Ethereum transaction with no upper-bound validation: [2](#0-1) 

**Go arithmetic walkthrough** — let `G = math.MaxUint64/2 + 1 = 9223372036854775808`:

| Step | Expression | Result (uint64) |
|---|---|---|
| `math.MaxInt64 - G` | `9223372036854775807 - 9223372036854775808` | **underflows → `18446744073709551615`** (MaxUint64) |
| guard check (msg 1) | `0 > MaxUint64` | **false** — no error |
| `gasWanted += G` | `0 + G` | `9223372036854775808` |
| guard check (msg 2) | `G > MaxUint64` | **false** — no error |
| `gasWanted += G` | `G + G` | **wraps → `0`** |

After the loop, `gasWanted = 0`. The block gas limit check:

```go
if gasWanted > blockGasLimit { ... }   // 0 > anything → always false
``` [3](#0-2) 

passes unconditionally, and the context is set with an infinite gas meter whose `limit = 0`: [4](#0-3) 

The `infiniteGasMeterWithLimit` never panics on `IsOutOfGas()` or `IsPastLimit()` (both hardcoded `false`), so EVM execution proceeds: [5](#0-4) 

---

### Impact Explanation

This is an ante handler bug that permits transactions whose total declared gas far exceeds the block gas limit to pass ante checks and commit EVM state. The block gas limit — the primary mechanism preventing a single transaction from monopolizing an entire block — is rendered ineffective for any tx carrying per-message gas above `math.MaxInt64`. This fits the **High** impact category: *"ante handler … bug that permits invalid transactions to commit."*

---

### Likelihood Explanation

**Moderate.** The attack requires:
1. Per-message `gas > math.MaxInt64` — valid in the Ethereum wire format (gas is `uint64`), no Ethermint validation rejects it.
2. Sufficient balance to pay `fee = gasLimit × effectiveGasPrice` per message. At any non-zero gas price this is enormous (~9.2 ETH-equivalent at 1 wei/gas). **On chains with zero minimum gas price** (common in testnets and some production deployments), the fee is zero and the barrier disappears entirely.

`BlockGasLimit` returns `math.MaxUint64` when `maxGas == -1` (no limit configured), which would make the block gas limit check vacuous anyway — but on chains with a finite block gas limit this bypass is directly exploitable at zero cost when gas price is zero. [6](#0-5) 

---

### Recommendation

Replace the subtraction-based guard with an addition-based overflow check that is safe for all `uint64` inputs:

```go
// Safe: check addition overflow before performing it
if gasLimit > math.MaxUint64-gasWanted {
    return ctx, fmt.Errorf("gasWanted(%d) + gasLimit(%d) overflow uint64", gasWanted, gasLimit)
}
gasWanted += gasLimit
```

Additionally, add a separate guard rejecting any single message whose `gasLimit > math.MaxInt64`, since Cosmos SDK gas meters and the block gas limit are `int64`-bounded in consensus params.

// Batch them in a single Cosmos tx.
// In CheckEthGasConsume:
//   Iteration 1: guard = (0 > MaxUint64) = false; gasWanted = G
//   Iteration 2: guard = (G > MaxUint64) = false; gasWanted = G+G = 0 (wraps)
//   Block gas limit check: 0 > blockGasLimit = false → PASSES
//
// Result: tx with effective gasWanted=0 commits EVM state,
// bypassing the block gas limit entirely.
```

func BlockGasLimit(ctx sdk.Context) uint64 {
	// Otherwise get from the consensus parameters
	cp := ctx.ConsensusParams()
	if cp.Block == nil {
		return 0
	}

	maxGas := cp.Block.MaxGas

	// Setting max_gas to -1 in CometBFT means there is no limit on the maximum gas consumption for transactions
	// https://github.com/cometbft/cometbft/blob/v0.37.2/proto/tendermint/types/params.proto#L25-L27
	if maxGas == -1 {
		return math.MaxUint64
	}

	if maxGas > 0 {
		return uint64(maxGas)
	}

	return 0
