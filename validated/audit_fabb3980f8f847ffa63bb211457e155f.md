### Title
Legacy EIP-712 Type Schema Derived from First Message Only, Enabling Forged Execution of Subsequent Messages - (File: `ante/cosmos/eip712.go`, `ethereum/eip712/eip712_legacy.go`, `ethereum/eip712/encoding_legacy.go`)

---

### Summary

In the legacy EIP-712 signature verification path, the EIP-712 type schema is constructed using only `msgs[0]`. The `legacyTraverseFields` function silently skips zero-value and empty-slice fields from the type schema. For multi-message transactions, this means those fields are absent from the EIP-712 hash for **all** messages — including `msgs[1]`, `msgs[2]`, etc. An attacker controlling a dApp can craft a transaction where `msgs[0]` has zero-value fields (making the type schema incomplete) while `msgs[1]` carries malicious non-zero values for those same fields. The signature verification passes, and the transaction executes with the attacker-controlled values in subsequent messages.

---

### Finding Description

**Step 1 — Only `msgs[0]` is used for type inference.**

In `ante/cosmos/eip712.go`, `VerifySignature` calls:

```go
typedData, err := eip712.LegacyWrapTxToTypedData(ethermintCodec, extOpt.TypedDataChainID, msgs[0], txBytes, feeDelegation)
``` [1](#0-0) 

The full `msgs` slice is validated (same type, same signer) but only `msgs[0]` is passed to `LegacyWrapTxToTypedData` for type schema construction. The same pattern appears in both `legacyDecodeAminoSignDoc` and `legacyDecodeProtobufSignDoc`:

```go
// Use first message for fee payer and type inference
msg := msgs[0]
``` [2](#0-1) [3](#0-2) 

**Step 2 — Zero-value and empty-slice fields are silently dropped from the type schema.**

Inside `legacyTraverseFields`, two separate guards skip fields from the type map:

```go
// If field is an empty value, do not include in types
if field.IsZero() {
    continue
}
``` [4](#0-3) 

```go
if fieldType.Kind() == reflect.Array || fieldType.Kind() == reflect.Slice {
    if field.Len() == 0 {
        // skip empty collections from type mapping
        continue
    }
    fieldType = fieldType.Elem()
    field = field.Index(0)
    isCollection = true
``` [5](#0-4) 

Any field that is zero-valued or an empty slice in `msgs[0]` is excluded from the `MsgValue` type definition. Because the same `MsgValue` type is reused when hashing every element of the `msgs[]` array in the EIP-712 payload, those fields are **not hashed for any message** — including `msgs[1]`, `msgs[2]`, etc.

**Step 3 — The full `txBytes` payload (containing all messages) is hashed with the incomplete type schema.**

`LegacyWrapTxToTypedData` sets `typedData.Message = txData` (the full Amino sign doc JSON, containing all messages), but `typedData.Types` is built only from `msgs[0]`: [6](#0-5) 

When `apitypes.TypedDataAndHash(typedData)` is called, the EIP-712 hash covers only the fields present in the type schema. Fields absent from the schema (because they were zero in `msgs[0]`) are silently ignored during hashing, even if they carry non-zero values in `msgs[1]`.

**Step 4 — The legacy EIP-712 path is still reachable.**

The `LegacyEip712SigVerificationDecorator` is invoked for any transaction carrying `ExtensionOptionsWeb3Tx`:

```go
case "/ethermint.types.v1.ExtensionOptionsWeb3Tx":
    anteHandler = newLegacyCosmosAnteHandlerEip712(ctx, options, options.ExtraDecorators...)
``` [7](#0-6) 

Any unprivileged user can submit such a transaction via `eth_sendRawTransaction` or direct Cosmos broadcast.

### Impact Explanation

An attacker controlling a dApp constructs a two-message EIP-712 transaction (both messages must be the same type, per `LegacyValidatePayloadMessages`):

- `msgs[0]`: a valid message with one or more fields set to zero/empty (e.g., an empty `msgs []Any` in a `MsgExec`, or an empty `coins []Coin` in a message type that permits it). These fields are dropped from the `MsgValue` type schema.
- `msgs[1]`: the same message type, but with those fields set to attacker-chosen values (e.g., a populated `msgs` list executing an arbitrary authz action, or a non-zero coin transfer).

The user's wallet computes and displays the EIP-712 typed data. Because the dropped fields are absent from the type schema, conformant wallets do not display or hash them. The user signs the hash. The ante handler re-derives the same incomplete type schema from `msgs[0]` and verifies the signature — which passes. The chain then executes `msgs[1]` with the attacker-controlled field values.

This constitutes **EIP-712 authorization bypass enabling forged execution** of Cosmos messages, matching the allowed High impact: *"EIP-712 authorization … verification bypass enabling … forged execution."*

---

### Likelihood Explanation

- The attacker must control the dApp that constructs and presents the transaction (realistic: malicious or compromised front-end).
- The attacker must identify a Cosmos message type where at least one field can be zero/empty in `msgs[0]` while remaining valid (passes `ValidateBasic`), yet carries meaningful value in `msgs[1]`. Optional string fields (e.g., `metadata` in governance messages) and optional slice fields (e.g., `msgs []Any` in `MsgExec` if an empty exec is accepted) are candidates.
- No privileged access, validator compromise, or network-level attack is required.
- The legacy EIP-712 path remains active and is used by Metamask-style Cosmos dApps.

Likelihood: **Medium** — requires a malicious dApp and a suitable message type, but no chain-level privilege.

---

### Recommendation

Pass all messages to the type-schema builder, not just `msgs[0]`. The type schema for `MsgValue` should be the union of all field types across every message in the transaction. Alternatively, enforce that the type schema is derived from the concrete Go type (via `reflect.TypeOf`) rather than from the runtime value of `msgs[0]`, so zero-value fields are never silently omitted.

---

### Proof of Concept

1. Identify a Cosmos message type `T` where field `F` can be zero/empty in a valid message (e.g., `metadata string` in `MsgVote`, or `msgs []Any` in `MsgExec` if empty is accepted).

2. Construct a Cosmos SDK transaction with two messages of type `T`:
   - `msgs[0]`: `T{..., F: <zero/empty>}` — valid, passes `ValidateBasic`.
   - `msgs[1]`: `T{..., F: <attacker-chosen value>}`.

3. Wrap in `ExtensionOptionsWeb3Tx` and present to the user's EIP-712-compatible wallet. The wallet hashes the typed data; because `F` is absent from the `MsgValue` type schema (dropped at `legacyTraverseFields` line 229/259), the wallet neither displays nor hashes `F`.

4. User signs. Broadcast the transaction. The ante handler in `VerifySignature` re-derives the same incomplete type schema from `msgs[0]` and verifies the signature — it passes.

5. The chain executes both messages. `msgs[1]` runs with `F = <attacker-chosen value>`, which was never covered by the signature.
