### Title
Invalid Receipt Proof Commitment at Resharding Epoch Boundary Breaks Chunk State Witness Reconstruction — (`chain/chain/src/stateless_validation/state_witness.rs`)

---

### Summary

At a resharding epoch boundary, `collect_source_receipt_proofs` calls `get_incoming_receipts_for_shard` to build the `source_receipt_proofs` map embedded in a `ChunkStateWitness`. When the traversal crosses the epoch boundary, it switches to the parent shard's ID and fetches the parent shard's stored incoming receipts. Those receipts were committed to the chain under the parent shard's outgoing-receipts merkle root, which covers the **full** receipt set (destined for both child shards). The code then inserts only the subset of receipts relevant to the specific child shard into the witness, while the merkle proof remains anchored to the full parent-shard leaf. Validators re-derive the same filtered subset and verify it against `chunk.prev_outgoing_receipts_root()`, which was computed over the full set — the proof fails. The developers explicitly acknowledge this in a TODO comment at the exact call site.

---

### Finding Description

`collect_source_receipt_proofs` in `chain/chain/src/stateless_validation/state_witness.rs` (lines 264–340) calls `get_incoming_receipts_for_shard` with `ReceiptFilter::All` to collect all incoming receipt proofs for the previous chunk. [1](#0-0) 

The function `get_incoming_receipts_for_shard` in `chain/chain/src/store/utils.rs` (lines 186–272) walks backwards through blocks. When it detects a shard-layout change at an epoch boundary, it maps `current_shard_id` to the parent shard and fetches the parent shard's stored incoming receipts. [2](#0-1) 

The parent shard's outgoing-receipts merkle root (stored in `chunk.prev_outgoing_receipts_root()`) was computed over the **complete** receipt set — receipts destined for every shard in the old layout, including both children of the split. The merkle leaf is `hash(all_receipts_to_parent_shard_id, parent_shard_id)`.

After the resharding, the code inserts into the witness only the receipts that belong to the specific child shard (the other half is discarded). The merkle proof accompanying those receipts was generated for the full parent-shard leaf, not for the filtered child-shard subset. The proof is therefore cryptographically invalid for the filtered set.

When validators call `validate_source_receipt_proofs` → `validate_receipt_proof`, they verify the proof against `chunk.prev_outgoing_receipts_root()`. [3](#0-2) 

The check `receipt_proof.verify_against_receipt_root(outgoing_receipts_root)` fails because the receipts in the proof no longer match the committed root. The witness is rejected with `InvalidChunkStateWitness`.

The developers explicitly document this broken invariant with a TODO at the call site:

> **TODO(resharding):** `get_incoming_receipts_for_shard` generates invalid proofs on resharding boundaries, because it removes the receipts that target the other half of a split shard, which makes the proof invalid. We need to collect the original proof and later, after verification, filter it to remove the receipts that were meant for the other half of the split shard. [4](#0-3) 

The analog to the external report is direct: just as `solverMetaTryCatch()` compared `endBalance` (which included `solverOp.value` and the initial ETH balance) against `bidAmount` — an incorrect comparison because the value included extra amounts — nearcore's witness builder compares a filtered receipt set against a merkle root that was committed over the unfiltered set. In both cases a variable/value is reused across two semantically different contexts (full vs. filtered), breaking the commitment invariant.

---

### Impact Explanation

Any chunk produced for a child shard whose receipt window spans the resharding boundary will embed an invalid `source_receipt_proofs` map. Every validator that processes the witness will reject it. Because **all** chunk producers for that shard hit the same code path, no valid witness can be produced for that chunk. The chain stalls at the resharding boundary for any shard that has cross-shard receipts in flight at the time of the split.

**Severity: High** — permanent chain stall for affected shards; no validator-side workaround exists once the invalid witness is the only one produced.

---

### Likelihood Explanation

The bug fires whenever cross-shard receipts are in transit at the moment of a resharding epoch boundary. Because resharding is announced two epochs in advance (embedded in the block header via `shard_split`), any network participant can observe the upcoming split and submit cross-shard transactions timed to be in-flight at the boundary. This converts a latent protocol bug into a user-triggerable chain-halt: the attacker sends ordinary cross-shard transactions (no special privilege required) to ensure receipts exist in the parent shard's window at the split block.

---

### Recommendation

Follow the fix direction stated in the TODO:

1. In `collect_source_receipt_proofs`, fetch the **full** parent-shard receipt proof (all receipts to the parent shard, with the parent shard's `to_shard_id`) so the merkle proof remains valid against `prev_outgoing_receipts_root`.
2. Embed the full proof in the witness for the purpose of merkle verification.
3. In `validate_source_receipt_proofs`, verify the full proof first, then filter the receipts to retain only those whose destination account falls within the child shard's account range before computing `applied_receipts_hash`.

This mirrors the correct pattern already used in `validate_chunk_state_witness_impl` for the resharding case, where `reassign_outgoing_receipts_for_resharding` is applied **after** the state transition, not before proof verification. [5](#0-4) 

---

### Proof of Concept

1. Observe the upcoming resharding epoch (announced in `BlockHeaderInnerRestV6.shard_split` two epochs ahead).
2. Submit cross-shard transactions from any account in a stable shard to an account in the shard that will be split, timed so the receipts are included in blocks within the receipt window of the first child-shard chunk after the split.
3. At the resharding boundary, the chunk producer for the child shard calls `collect_source_receipt_proofs`. `get_incoming_receipts_for_shard` crosses the epoch boundary, maps to the parent shard, and retrieves the stored incoming receipts. The receipts for the other child shard are absent from the returned set, but the merkle proof is anchored to the full parent-shard leaf.
4. The witness is distributed to validators. Each validator calls `validate_receipt_proof` → `receipt_proof.verify_against_receipt_root(outgoing_receipts_root)`. The verification fails because the filtered receipt set does not hash to the committed root.
5. All validators return `InvalidChunkStateWitness`. No endorsements are produced. The child shard's chunk cannot be included. The chain stalls for that shard.

    /// TODO(resharding): `get_incoming_receipts_for_shard` generates invalid proofs on resharding
    /// boundaries, because it removes the receipts that target the other half of a split shard,
    /// which makes the proof invalid. We need to collect the original proof and later, after verification,
    /// filter it to remove the receipts that were meant for the other half of the split shard.
    fn collect_source_receipt_proofs(
        &self,
        epoch_manager: &dyn EpochManagerAdapter,
        prev_block_header: &BlockHeader,
        prev_chunk_header: &ShardChunkHeader,
    ) -> Result<HashMap<ChunkHash, ReceiptProof>, Error> {
        if prev_chunk_header.is_genesis() {
            // State witness which proves the execution of the first chunk in the blockchain
            // doesn't have any source receipts.
            return Ok(HashMap::new());
        }

        // Find the first block that included `prev_chunk`.
        // Incoming receipts were generated by the blocks before this one.
        let mut cur_block;
        let prev_chunk_original_block: &BlockHeader = {
            if prev_chunk_header.is_new_chunk(prev_block_header.height()) {
                prev_block_header
            } else {
                cur_block = self.get_block_header(prev_block_header.prev_hash())?;
                loop {
                    if prev_chunk_header.is_new_chunk(cur_block.height()) {
                        break &cur_block;
                    }
                    cur_block = self.get_block_header(cur_block.prev_hash())?;
                }
            }
        };

        // Get the last block that contained `prev_prev_chunk` (the chunk before `prev_chunk`).
        // We are interested in all incoming receipts that weren't handled by `prev_prev_chunk`.
        let prev_prev_chunk_block = self.get_block(prev_chunk_original_block.prev_hash())?;
        // Find the header of the chunk before `prev_chunk`
        let prev_prev_chunk_header = epoch_manager
            .get_prev_chunk_header(&prev_prev_chunk_block, prev_chunk_header.shard_id())?;

        // Fetch all incoming receipts for `prev_chunk`.
        // They will be between `prev_prev_chunk.height_included` (first block containing `prev_prev_chunk`)
        // and `prev_chunk_original_block`
        let shard_layout = epoch_manager
            .get_shard_layout_from_prev_block(prev_chunk_original_block.prev_hash())?;
        let incoming_receipt_proofs = get_incoming_receipts_for_shard(
            &self,
            epoch_manager,
            prev_chunk_header.shard_id(),
            &shard_layout,
            *prev_chunk_original_block.hash(),
            prev_prev_chunk_header.height_included(),
            ReceiptFilter::All,
        )?;

pub fn validate_receipt_proof(
    receipt_proof: &ReceiptProof,
    from_chunk: &ShardChunkHeader,
    target_chunk_shard_id: ShardId,
    outgoing_receipts_root: CryptoHash,
) -> Result<(), Error> {
    // Validate that from_shard_id is correct. The receipts must match the outgoing receipt root
    // for this shard, so it's impossible to fake it.
    if receipt_proof.1.from_shard_id != from_chunk.shard_id() {
        return Err(Error::InvalidChunkStateWitness(format!(
            "Receipt proof for chunk {:?} is from shard {}, expected shard {}",
            from_chunk.chunk_hash(),
            receipt_proof.1.from_shard_id,
            from_chunk.shard_id(),
        )));
    }
    // Validate that to_shard_id is correct. to_shard_id is also encoded in the merkle tree,
    // so it's impossible to fake it.
    if receipt_proof.1.to_shard_id != target_chunk_shard_id {
        return Err(Error::InvalidChunkStateWitness(format!(
            "Receipt proof for chunk {:?} is for shard {}, expected shard {}",
            from_chunk.chunk_hash(),
            receipt_proof.1.to_shard_id,
            target_chunk_shard_id
        )));
    }
    // Verify that (receipts, to_shard_id) belongs to the merkle tree of outgoing receipts.
    if !receipt_proof.verify_against_receipt_root(outgoing_receipts_root) {
        return Err(Error::InvalidChunkStateWitness(format!(
            "Receipt proof for chunk {:?} has invalid merkle path, doesn't match outgoing receipts root",
            from_chunk.chunk_hash()
        )));
    }
    Ok(())
