### Title
Unbounded Stale L1 Gas Price Fallback Propagates Wrong Fees Into Block Commitments — (`crates/apollo_consensus_orchestrator/src/utils.rs`)

---

### Summary

When the L1 gas price scraper stops updating, `get_l1_prices_in_fri_and_wei_and_conversion_rate` silently falls back to `previous_block_info` prices with no bound on how many consecutive blocks may use this fallback and no check on how old `previous_block_info.timestamp` is. Because both the proposer and the validator execute the same fallback logic, proposals carrying arbitrarily stale `l1_gas_price_fri` / `l1_data_gas_price_fri` values are accepted indefinitely, embedding wrong gas prices into every block header and block hash.

---

### Finding Description

`L1GasPriceProvider::get_price_info` contains a staleness guard:

```rust
// Check if the prices are stale.
if timestamp.0 > (*last_timestamp + self.config.max_time_gap_seconds) {
    return Err(L1GasPriceProviderError::StaleL1GasPricesError { … });
}
``` [1](#0-0) 

`max_time_gap_seconds` defaults to 900 s (15 minutes). [2](#0-1) 

When `StaleL1GasPricesError` (or `MissingDataError`) is returned, `get_l1_prices_in_fri_and_wei_and_conversion_rate` falls back to `previous_block_info`:

```rust
// One or both (oracle/provider) have failed to fetch …
if let Some(block_info) = previous_block_info {
    let prev_l1_gas_price_wei = block_info.l1_prices_wei.clone();
    let prev_l1_gas_price = block_info.l1_prices_fri.clone();
    …
    return (prev_l1_gas_price, prev_l1_gas_price_wei, eth_to_fri_rate);
}
``` [3](#0-2) 

`PreviousBlockInfo` carries a `timestamp` field, but **it is never compared against the current time** in this fallback path. There is no expiry, no counter, and no maximum number of consecutive fallback blocks. [4](#0-3) 

`previous_block_info` is populated from the previous block's `ProposalInit`: [5](#0-4) 

The fallback chain therefore self-reinforces:

| Block | Provider result | Prices used |
|-------|----------------|-------------|
| N−1   | OK             | P_live      |
| N     | `StaleL1GasPricesError` | P_live (from N−1 `previous_block_info`) |
| N+1   | `StaleL1GasPricesError` | P_live (from N `previous_block_info`, itself a fallback) |
| …     | …              | P_live (frozen indefinitely) |

The validator path calls the identical function:

```rust
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(
    l1_gas_price_provider,
    init_proposed.timestamp,
    block_info_validation.previous_block_info.as_ref(),
    gas_price_params,
)
.await;
``` [6](#0-5) 

Because both proposer and validator derive the same stale reference price from the same `previous_block_info`, the `within_margin` check passes and the proposal is accepted. [7](#0-6) 

The accepted stale prices flow directly into `ProposalInit`: [8](#0-7) 

and from there into `BlockInfo::gas_prices` via `convert_to_sn_api_block_info`: [9](#0-8) 

These prices are committed into the block hash and used by the blockifier to charge fees for every transaction in the block.

---

### Impact Explanation

`l1_gas_price_fri` and `l1_data_gas_price_fri` from `ProposalInit` determine the fee charged for every transaction in the block and are included in the block hash computation. If L1 gas prices have moved significantly since the scraper stopped, every transaction in every block built during the outage pays the wrong fee. The wrong prices are permanently committed into the chain. This matches the Critical scope item: **"Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."**

Concrete economic scenarios:
- L1 gas price drops (e.g., 100 gwei → 10 gwei): users are overcharged by up to 10×.
- L1 gas price rises (e.g., 10 gwei → 100 gwei): the protocol absorbs L1 DA costs at 10× the price it charges users.

---

### Likelihood Explanation

The L1 gas price scraper can stop due to L1 node RPC failures, network partitions, or scraper bugs. The `max_time_gap_seconds = 900` threshold means the fallback activates after only 15 minutes of no L1 data. L1 gas prices can change by an order of magnitude in 15+ minutes during periods of high or low L1 activity. Once the fallback is active there is no automatic recovery mechanism — prices stay frozen until the scraper resumes and the provider's ring buffer refills.

---

### Recommendation

Add a staleness check on `previous_block_info.timestamp` inside the fallback branch. If `current_timestamp − previous_block_info.timestamp > max_time_gap_seconds`, refuse to build or validate proposals rather than silently propagating stale prices. This is the direct analog of the M-31 fix:

```rust
// Proposed guard in the fallback branch:
if let Some(block_info) = previous_block_info {
    if timestamp.saturating_sub(block_info.timestamp) <= max_time_gap_seconds {
        // use block_info prices
        …
    }
    // else: fall through to hard defaults or return an error
}
```

The `max_time_gap_seconds` value should be sourced from `L1GasPriceProviderConfig` so the threshold is consistent between the provider's own staleness check and the orchestrator's fallback guard.

---

### Proof of Concept

1. L1 gas price scraper stops (e.g., L1 node RPC failure). Last scraped price: `P_old`.
2. After 900 s, `get_price_info(timestamp)` returns `StaleL1GasPricesError`.
3. Proposer calls `get_l1_prices_in_fri_and_wei(…, previous_block_info=Some(N−1_info), …)` → fallback → returns `P_old`.
4. Proposer builds Block N with `l1_gas_price_fri = P_old`, broadcasts `ProposalInit`.
5. Validator calls `is_block_info_valid` → calls `get_l1_prices_in_fri_and_wei(…, previous_block_info=Some(N−1_info), …)` → same fallback → same `P_old` → `within_margin` passes → proposal accepted.
6. Block N committed with `l1_gas_price_fri = P_old` in block hash.
7. Block N+1: `previous_block_info` = Block N's `ProposalInit` (which already used `P_old`). Same fallback. Block N+1 committed with `P_old`.
8. This continues indefinitely. All transactions in these blocks pay fees based on `P_old`, not the current L1 gas price. If the actual L1 gas price has risen 10× since the scraper stopped, the protocol absorbs the difference on every L1 DA write.
