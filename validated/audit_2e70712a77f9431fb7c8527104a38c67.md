### Title
Incomplete `emit_event` Syscall Implementation Leaves Event Commitment Unenforced in Block Hash — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execute_syscalls.cairo`)

---

### Summary

The `EMIT_EVENT_SELECTOR` branch inside `execute_syscalls` is a stub: it charges gas and writes a response header but **never records the event data**. Simultaneously, the `event_commitment` field that feeds into the block hash is not computed by the OS — it is freely guessed via a hint. Because the OS never derives the event commitment from the actual events it processed, any prover can supply an arbitrary `event_commitment` and still produce a valid proof. This makes the event commitment in every block hash meaningless and unenforceable, enabling a malicious prover to attest to a block whose event commitment does not match the events that were actually executed, causing an unintended chain split between L1 and honest L2 nodes.

---

### Finding Description

**Step 1 — `emit_event` is silently skipped.**

In `execute_syscalls.cairo` the handler for `EMIT_EVENT_SELECTOR` explicitly states it is skipped and only deducts gas:

```cairo
if (selector == EMIT_EVENT_SELECTOR) {
    // Skip as long as the block hash is not calculated by the OS.
    // TODO(Yoni, 1/4/2022): calculate event hash in the OS.
    reduce_syscall_gas_and_write_response_header(
        total_gas_cost=EMIT_EVENT_GAS_COST, request_struct_size=EmitEventRequest.SIZE
    );
    ...
    return execute_syscalls(...);
}
```

No event key, data, or emitting contract address is written to any output segment. The event simply disappears. [1](#0-0) 

**Step 2 — The event commitment in the block hash is never computed; it is a free hint.**

`get_block_hashes` in `block_hash.cairo` explicitly documents that `header_commitments` (which contains `event_commitment`, `transaction_commitment`, `receipt_commitment`, and `state_diff_commitment`) are **not computed by the OS**:

```cairo
// Currently, the header commitments and gas prices are not computed by the OS.
// TODO(Yoni, 1/1/2027): compute the header commitments and gas prices.
local header_commitments: BlockHeaderCommitments*;
local gas_prices_hash;
...
%{ GetBlockHashes %}
```

The hint `GetBlockHashes` fills these locals with whatever the prover supplies. There is no Cairo assertion that ties `header_commitments.event_commitment` to the events actually processed in the block. [2](#0-1) 

**Step 3 — The OS output header itself acknowledges the block hash is unenforced.**

The `OsOutputHeader` struct comment reads:

```cairo
// Currently, the block hash is not enforced by the OS.
new_block_hash: felt,
``` [3](#0-2) 

**Step 4 — The block hash mapping written to storage is also a free hint.**

`write_block_number_to_block_hash_mapping` in `os_utils.cairo` similarly guesses `old_block_hash` without verification:

```cairo
// Currently, the block hash mapping is not enforced by the OS.
// TODO(Yoni, 1/1/2026): output this hash.
local old_block_hash;
%{ GetOldBlockNumberAndHash %}
``` [4](#0-3) 

**Step 5 — The block hash is computed from the unverified commitment and serialized to output.**

`calculate_block_hash` hashes `header_commitments.event_commitment` (among other fields) into the final block hash, and `get_block_os_output_header` places the result into the `OsOutputHeader` that is serialized to the proof output: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Because the OS never records events and never constrains `event_commitment` to match the events it processed, a malicious prover can:

1. Execute a block containing many `emit_event` calls (e.g., ERC-20 `Transfer` events from a DEX).
2. Supply `header_commitments.event_commitment = 0` (or any fabricated value) via the `GetBlockHashes` hint.
3. Produce a valid STARK proof — the Cairo constraints are fully satisfied because no constraint links the commitment to the actual events.
4. Submit this proof to L1. L1 accepts the proof and records the fabricated block hash.

Honest full nodes that independently compute the event commitment from the transaction receipts will derive a different block hash. L1 and honest L2 nodes now hold irreconcilably different block hashes for the same block number — an **unintended chain split (network partition)**.

Additionally, any application or bridge that relies on Merkle-style event proofs against the block hash (e.g., to prove a token transfer occurred) will be unable to produce valid proofs against the fabricated commitment, effectively **permanently freezing funds** locked in such bridges.

---

### Likelihood Explanation

- The entry path is reachable by any unprivileged user: every ERC-20 transfer, DEX swap, or NFT mint emits events via `emit_event`. The OS silently drops all of them.
- The prover (sequencer) controls the hint values. A malicious or compromised sequencer can trivially supply a false `event_commitment` with zero additional effort.
- The TODO comment dated `1/4/2022` (over three years old) and the `1/1/2027` deadline for computing header commitments confirm this is a known, long-standing gap that is not yet closed.

---

### Recommendation

1. **Record events in `execute_syscalls`**: When `EMIT_EVENT_SELECTOR` is encountered, serialize the event keys and data into a dedicated output segment (analogous to how `execute_send_message_to_l1` appends to `outputs.messages_to_l1`).
2. **Compute `event_commitment` inside the OS**: After all transactions in a block are executed, compute the Poseidon commitment over the collected events in Cairo, and assert it equals the value used in `calculate_block_hash`. Remove the reliance on the `GetBlockHashes` hint for this field.
3. **Enforce the block hash**: Remove the "not enforced" caveat from `OsOutputHeader.new_block_hash` by asserting the computed hash matches the hint-supplied value, or by deriving it entirely in Cairo.
4. **Enforce the block hash mapping**: Similarly, compute and assert `old_block_hash` in `write_block_number_to_block_hash_mapping` rather than accepting it from a hint.

---

### Proof of Concept

```
1. User deploys an ERC-20 contract and calls `transfer(recipient, amount)`.
   → The contract calls `emit_event(keys=[TRANSFER_KEY, sender, recipient], data=[amount_low, amount_high])`.

2. The OS reaches the EMIT_EVENT_SELECTOR branch in execute_syscalls.
   → Only `reduce_syscall_gas_and_write_response_header` is called.
   → No event data is written to any segment.

3. After block execution, get_block_hashes is called.
   → `%{ GetBlockHashes %}` fills `header_commitments` from the prover hint.
   → A malicious prover sets `header_commitments.event_commitment = 0`.
   → No Cairo assertion checks this against the (empty) event output.

4. calculate_block_hash hashes event_commitment=0 into new_block_hash.
   → The proof is valid; L1 accepts new_block_hash.

5. An honest node computes the correct event commitment from the Transfer event
   and derives a different block hash.
   → L1 state and honest node state diverge → chain split.

6. A bridge that issued a receipt "Transfer event in block N" cannot produce
   a Merkle proof against the fabricated event_commitment=0
   → user funds locked in the bridge are permanently frozen.
```
