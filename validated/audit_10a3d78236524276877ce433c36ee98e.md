### Title
Cross-Chain Replay of `deployToken` MPC Signature Enables Unauthorized Token Deployment on Any Chain — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `MetadataPayload` message that NEAR's MPC signs for `deployToken` does not include a destination chain ID. A single MPC signature is therefore valid on every supported chain simultaneously. Any observer can extract a `deployToken` signature from one chain's transaction history and replay it on any other chain to deploy the same bridge token there without a new MPC signing event.

---

### Finding Description

NEAR's `log_metadata_callback` constructs and signs a `MetadataPayload` containing only `prefix | token | name | symbol | decimals`: [1](#0-0) 

The borsh-serialised payload is hashed and submitted to the MPC signer with no chain identifier: [2](#0-1) 

Every destination chain verifies this same chain-agnostic hash:

**EVM** — `deployToken` encodes `PayloadType.Metadata | token | name | symbol | decimals` and recovers the signer. No `omniBridgeChainId` is present: [3](#0-2) 

**Starknet** — `MetadataPayloadImpl::to_borsh()` serialises the same five fields with no chain byte: [4](#0-3) 

`deploy_token` passes this chain-agnostic blob directly to `_verify_borsh_signature`: [5](#0-4) 

**Solana** — `DeployTokenPayload::serialize_for_near` writes `IncomingMessageType::Metadata` followed by the four token fields; `SOLANA_OMNI_BRIDGE_CHAIN_ID` is never written: [6](#0-5) 

This is in direct contrast to `finTransfer`/`finalize_transfer`, which embeds the chain ID twice in the signed payload on every chain: [7](#0-6) [8](#0-7) [9](#0-8) 

---

### Impact Explanation

An attacker who observes a successful `deployToken` call on chain A can immediately replay the identical `(signatureData, MetadataPayload)` tuple on chains B and C. Each chain's bridge will:

1. Accept the signature as valid (it is — the MPC signed exactly that hash).
2. Register the token in its `nearToEthToken` / `near_to_starknet_token` / equivalent mapping as a legitimate bridge token.
3. Mark the token as `isBridgeToken = true`, enabling future `finTransfer` minting calls for that token on that chain.

Concrete consequences:

- **Unauthorized token deployment**: Bridge tokens are deployed on chains the protocol never authorised, corrupting the canonical token registry.
- **Permanent blocking of legitimate deployment**: Each chain's guard (`ERR_TOKEN_EXIST`, `ERR_TOKEN_ALREADY_DEPLOYED`) prevents a second deployment of the same NEAR token ID. Once the attacker occupies the slot, the protocol can never deploy the token legitimately on that chain.
- **Path to unauthorized minting**: After deploying the token on chain B via replay, the attacker calls `logMetadata` on chain B for the newly deployed bridge token, then proves the resulting `LogMetadata` event on NEAR. NEAR registers the token for chain B. From that point the MPC will sign `finTransfer` messages for chain B, and the attacker (or anyone) can mint unbacked supply on chain B by finalising transfers that were never initiated on a legitimate source chain.

This matches the allowed impact: **"Proof, signature, MPC … verification bypass enabling unauthorized transfer finalization, token deployment, or message execution"** and **"token-mapping … corruption that breaks bridge collateralization."**

---

### Likelihood Explanation

- The `deployToken` transaction and its embedded signature are publicly visible on every chain.
- No privileged access, leaked key, or colluding party is required.
- The replay call is a single permissionless transaction on the target chain.
- The window of opportunity is permanent: the signature remains valid indefinitely.

Likelihood is **High**.

---

### Recommendation

Include the destination chain ID in the `MetadataPayload` before it is hashed and submitted to the MPC signer, mirroring the existing pattern used by `TransferMessagePayload`:

```rust
// near/omni-bridge/src/lib.rs — log_metadata_callback
let metadata_payload = MetadataPayload {
    prefix: PayloadType::Metadata,
    destination_chain: chain_id,   // <-- add this field
    token: token_id.to_string(),
    name: metadata.name,
    symbol: metadata.symbol,
    decimals: metadata.decimals,
};
```

Each destination chain must then include its own `omniBridgeChainId` / `omni_bridge_chain_id` / `SOLANA_OMNI_BRIDGE_CHAIN_ID` in the borsh encoding before hashing, exactly as `finTransfer` already does. A signature produced for chain A will then be cryptographically invalid on chain B.

---

### Proof of Concept

1. Observe a successful `deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` transaction on Ethereum (chain ID `0x01`).
2. Extract `sig` from the calldata.
3. On Starknet (chain ID `0x05`), call:
   ```
   deploy_token(sig, MetadataPayload { token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6 })
   ```
4. `_verify_borsh_signature` computes `keccak(0x01 | borsh("usdc.near") | borsh("USD Coin") | borsh("USDC") | 0x06)` — identical to the Ethereum hash — and accepts the signature.
5. `usdc.near` is now registered as a bridge token on Starknet without any MPC authorisation for Starknet.
6. Any subsequent legitimate attempt to deploy `usdc.near` on Starknet reverts with `ERR_TOKEN_ALREADY_DEPLOYED`.

    fn serialize_for_near(&self, params: Self::AdditionalParams) -> Result<Vec<u8>> {
        let mut writer = BufWriter::new(Vec::with_capacity(DEFAULT_SERIALIZER_CAPACITY));
        // 0. prefix
        IncomingMessageType::InitTransfer.serialize(&mut writer)?;
        // 1. destination_nonce
        self.destination_nonce.serialize(&mut writer)?;
        // 2. transfer_id
        writer.write_all(&[self.transfer_id.origin_chain])?;
        self.transfer_id.origin_nonce.serialize(&mut writer)?;

        writer.write_all(&[SOLANA_OMNI_BRIDGE_CHAIN_ID])?;

        self.amount.serialize(&mut writer)?;
