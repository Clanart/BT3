### Title
Pending Transfer Permanently Locked When `normalize_amount(amount - fee) == 0` — (`near/omni-bridge/src/lib.rs`)

### Summary

`init_transfer` validates only that `fee < amount`, but `sign_transfer` additionally requires that `normalize_amount(amount - fee) > 0`. For tokens with significant decimal differences between chains, a transfer whose `amount - fee` is below the normalization unit will pass creation-time validation but permanently fail at signing time. Because no cancel or refund path exists, the user's tokens are irrecoverably locked in `pending_transfers`.

### Finding Description

**Creation-time check (`init_transfer`, line 554–557):**

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
```

This only enforces `fee < amount`. It does not verify that the net amount survives decimal normalization. [1](#0-0) 

**Execution-time check (`sign_transfer`, lines 475–485):**

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

`normalize_amount` performs floor division by `10^(origin_decimals - decimals)`. For a token with 18 origin decimals and 6 destination decimals, any `amount - fee < 10^12` normalizes to 0, causing `sign_transfer` to panic and revert — leaving the transfer permanently in `pending_transfers`. [2](#0-1) 

**No recovery path exists.** `update_transfer_fee` only allows *increasing* the fee (enforced by `fee.fee >= current_fee.fee`), so the fee cannot be reduced to widen the net amount. There is no `cancel_transfer` or refund function. [3](#0-2) 

`sign_transfer_callback` only removes the transfer message when the fee is zero and MPC signing succeeds — a branch that is never reached if `sign_transfer` panics before calling the MPC signer. [4](#0-3) 

The `Decimals` struct that drives normalization: [5](#0-4) 

### Impact Explanation

User tokens are transferred into the bridge at `ft_on_transfer` time and locked. If the resulting `TransferMessage` can never pass `sign_transfer`'s normalization check, those tokens are permanently frozen in `pending_transfers` with no on-chain recovery mechanism. This matches the **Critical** allowed impact: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.*

### Likelihood Explanation

The condition is reachable by any unprivileged bridge user:

1. A token with `origin_decimals = 18` and `decimals = 6` (e.g., a USDC-like token) has a normalization factor of `10^12`.
2. A user calls `ft_on_transfer` → `init_transfer` with `amount = X` and `fee = X - 1` (passes `fee < amount`).
3. `amount - fee = 1`, which normalizes to `0`.
4. Every subsequent `sign_transfer` call panics; the transfer is stuck forever.

This can occur accidentally (user sets a fee very close to the amount) or deliberately (self-harm). The protocol provides no guard against it at creation time and no escape hatch afterward.

### Recommendation

Add a normalization check inside `init_transfer` (or `init_transfer_internal`) before storing the transfer:

```rust
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(token_address) = token_address {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let net = transfer_message.amount_without_fee()
            .near_expect(BridgeError::InvalidFee);
        require!(
            Self::normalize_amount(net, decimals) > 0,
            BridgeError::InvalidAmountToTransfer.as_ref()
        );
    }
}
```

Additionally, add a `cancel_transfer` function callable by the transfer owner that refunds the locked tokens and removes the entry from `pending_transfers`, providing a recovery path for already-stuck transfers.

### Proof of Concept

1. Deploy a token with `origin_decimals = 18`, `decimals = 6` (normalization factor `10^12`).
2. Call `ft_transfer_call` on the token contract with `amount = 10^12 + 1` and `msg` encoding `InitTransferMsg { fee: 10^12, ... }`.
   - `init_transfer` check: `10^12 < 10^12 + 1` → passes.
   - Transfer stored in `pending_transfers`.
3. Call `sign_transfer(transfer_id, ...)`.
   - `amount_without_fee() = 1`.
   - `normalize_amount(1, Decimals{decimals:6, origin_decimals:18}) = 1 / 10^12 = 0`.
   - `require!(0 > 0, ...)` → panics with `InvalidAmountToTransfer`.
4. Call `update_transfer_fee` attempting to lower the fee → rejected (`fee.fee >= current_fee.fee`).
5. No other function can remove the transfer. Funds are permanently locked.
