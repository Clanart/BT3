Audit Report

## Title
Stranded ETH in Router Silently Consumed to Fund Arbitrary User's WETH Payment — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay` reads `address(this).balance` without verifying that the ETH belongs to the current payer. ETH stranded in the router from a prior `multicall{value: X}` that omitted `refundETH` is silently wrapped and transferred to a pool on behalf of a completely different user, causing permanent, irrecoverable loss of the original depositor's ETH.

## Finding Description

**How ETH becomes stranded:** `multicall` is `payable` and accepts `msg.value` directly into the contract. [1](#0-0) 

The `receive()` guard only fires when ETH arrives with *empty calldata* (i.e., a bare ETH transfer). When ETH is attached as `msg.value` to a `payable` function call, `receive()` is never invoked, so the guard at L32–34 provides no protection against ETH accumulating in the router. [2](#0-1) 

**How stranded ETH is consumed:** When any subsequent user swaps with `tokenIn = WETH`, the callback reaches `pay`. The WETH branch unconditionally reads the entire contract ETH balance at L74 with no ownership check: [3](#0-2) 

- **Full-cover branch (L75–77):** If stranded ETH ≥ `value`, the entire payment is funded from the stranded balance; the actual payer pays nothing.
- **Partial-cover branch (L78–81):** Stranded ETH is wrapped and sent to the pool first; only the remainder is pulled from the actual payer via `transferFrom`. The stranded ETH is irrecoverably consumed.

No existing guard prevents this: `refundETH` is optional and caller-initiated, `receive()` is irrelevant to `msg.value` flows, and `pay` has no per-transaction ETH accounting.

## Impact Explanation

Direct, permanent loss of user principal. The user who sent ETH via `multicall` without `refundETH` loses that ETH with no recourse — it is wrapped and transferred to a pool on behalf of a different user. The PAYER_EXCLUSIVITY invariant is broken: a swap settles using ETH from one principal and WETH allowance from another. Severity: **High/Critical** — direct fund loss, no privileged role required, no non-standard token behavior required.

## Likelihood Explanation

Omitting `refundETH` in a multicall is a well-documented user mistake in Uniswap-style routers (e.g., over-sending ETH for a swap that partially fills). Any subsequent WETH swap by any user drains the stranded balance. A griefing attacker can also deliberately strand dust ETH to reduce a victim's required allowance, causing the victim's swap to fail if they approved exactly the expected amount. The condition is reachable by any unprivileged caller with no special setup.

## Recommendation

In the WETH branch of `pay`, only use `address(this).balance` when `payer == address(this)` (mid-path hop where the router itself is the source). For external payers, always pull the full amount via `transferFrom`:

```solidity
} else if (token == WETH) {
    if (payer == address(this)) {
        IERC20(WETH).safeTransfer(recipient, value);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

If native-ETH-as-WETH-input is a desired feature, gate it explicitly (e.g., require `msg.value == amountIn` at the entry point and track it in transient storage) so only the current transaction's ETH is eligible.

## Proof of Concept

1. Deploy `MetricOmmSimpleRouter` with a WETH/TOKEN pool.
2. **UserB strands ETH:** `router.multicall{value: 0.5 ether}([someAction])` — omits `refundETH`. `router.balance == 0.5 ETH`.
3. **UserA swaps:** UserA approves exactly `0.5 WETH` to the router, then calls `exactInputSingle({tokenIn: WETH, amountIn: 1 ether, ...})`.
4. **`pay` executes (L78–81):** `nativeBalance = 0.5 ETH`, `value = 1 ETH`. Wraps 0.5 ETH → WETH, transfers to pool. Calls `transferFrom(userA, pool, 0.5 WETH)` — succeeds because userA approved exactly 0.5 WETH.
5. **Result:** UserA's 1 ETH swap completes using UserB's 0.5 ETH. UserB's 0.5 ETH is permanently lost.
