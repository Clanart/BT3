### Title
Integer Division Truncation to Zero in `normalize_amount` Causes Permanent Lock/Burn of User Funds When `amount_without_fee < 10^(origin_decimals - decimals)` - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

`normalize_amount` uses integer (floor) division to scale a NEAR-side token amount down to the destination chain's decimal precision. When a user initiates a transfer with an `amount_without_fee` smaller than the decimal divisor (`10^(origin_decimals - decimals)`), the normalized result is `0`. Tokens are already locked or burned at `init_transfer` time; the subsequent `sign_transfer` call panics on the `amount_to_transfer > 0` guard, leaving the transfer message stranded in `pending_transfers` with no recovery path. The user's funds are permanently frozen or destroyed.

---

### Finding Description

**Step 1 – Tokens are locked/burned unconditionally at `init_transfer` time.**

`init_transfer_internal` locks (for native tokens) or burns (for deployed/bridged tokens) the full `amount` before any decimal normalization check is performed:

```rust
// near/omni-bridge/src/lib.rs  lines 1850-1857
if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
    self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
    self.lock_tokens_if_needed(
        transfer_message.get_destination_chain(),
        &token_id,
        transfer_message.amount.0,
    );
}
```

The only pre-condition checked at this stage is `fee.fee < amount` (line 555), which does not prevent `amount_without_fee` from being smaller than the decimal divisor.

**Step 2 – `normalize_amount` performs plain integer (floor) division.**

```rust
// near/omni-bridge/src/lib.rs  lines 2784-2787
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

For a token registered with `origin_decimals = 24` (NEAR) and `decimals = 6` (EVM), `diff_decimals = 18` and the divisor is `10^18`. Any `amount_without_fee < 10^18` normalizes to `0`.

**Step 3 – `sign_transfer` panics, but the transfer message is never removed.**

```rust
// near/omni-bridge/src/lib.rs  lines 475-485
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

The panic aborts `sign_transfer` without removing the transfer message from `pending_transfers`. No `fin_transfer` proof can ever be generated (because no MPC signature was produced), and no `claim_fee` path exists (it requires a `FinTransfer` proof from the destination chain). There is no user-callable cancel or refund function in the contract.

---

### Impact Explanation

- **For deployed/bridged tokens**: `burn_tokens_if_needed` is called at `init_transfer` time. The tokens are destroyed immediately and irrecoverably.
- **For native (locked) tokens**: `lock_tokens_if_needed` increments `locked_tokens`. The tokens remain locked indefinitely because neither `fin_transfer` nor `claim_fee` can ever succeed for this transfer.

Both outcomes satisfy the allowed impact: **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."**

---

### Likelihood Explanation

Any unprivileged bridge user can trigger this by calling `ft_transfer_call` with an amount smaller than the decimal divisor for the destination chain. For a 24→6 decimal token pair the threshold is `10^18` base units (0.000001 of the token in 6-decimal terms). A user who sends, for example, `5 × 10^17` base units with zero fee will have their tokens permanently locked/burned. No special role, leaked key, or colluding party is required.

---

### Recommendation

Perform the `normalize_amount > 0` check **before** locking or burning tokens, inside `init_transfer` (or `init_transfer_internal`), and revert the entire call if the normalized amount would be zero. This mirrors the guard already present in `sign_transfer` but moves it to the point where funds are still safe:

```rust
// Validate before locking/burning
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(amount_to_transfer > 0, BridgeError::InvalidAmountToTransfer.as_ref());
// Only then: burn_tokens_if_needed / lock_tokens_if_needed
```

---

### Proof of Concept

1. Register a token with `origin_decimals = 24` on NEAR and `decimals = 6` on EVM (a realistic NEAR→EVM pair).
2. As an unprivileged user, call `ft_transfer_call` with `amount = 5 × 10^17` and `fee = 0`.
3. `init_transfer_internal` succeeds: tokens are locked (or burned if deployed).
4. A trusted relayer calls `sign_transfer` for the resulting `TransferId`.
5. `normalize_amount(5×10^17, {decimals:6, origin_decimals:24}) = 5×10^17 / 10^18 = 0`.
6. `require!(amount_to_transfer > 0, ...)` panics → `sign_transfer` aborts.
7. The transfer message remains in `pending_transfers`; the `5 × 10^17` base-unit tokens are permanently locked/burned with no recovery path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
