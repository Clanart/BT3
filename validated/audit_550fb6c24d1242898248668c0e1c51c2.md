### Title
`wormholeNonce` Declared as `uint32` Causes Permanent Bridge Freeze After ~4.29 Billion Operations — (File: `evm/src/omni-bridge/contracts/OmniBridgeWormhole.sol`)

---

### Summary

`OmniBridgeWormhole` stores its Wormhole message counter as `uint32`, capping it at 4,294,967,295. Solidity 0.8.x's built-in overflow protection causes every bridge operation — `initTransfer`, `finTransfer`, `deployToken`, and `logMetadata` — to permanently revert once that ceiling is hit, freezing all user funds in the bridge.

---

### Finding Description

In `OmniBridgeWormhole.sol`, the counter used to tag every outbound Wormhole message is declared as:

```solidity
uint32 public wormholeNonce;   // OmniBridgeWormhole.sol line 30
``` [1](#0-0) 

It is unconditionally incremented at the end of all four extension hooks:

| Function | Increment line |
|---|---|
| `deployTokenExtension` | 69 |
| `logMetadataExtension` | 93 |
| `finTransferExtension` | 115 |
| `initTransferExtension` | 149 | [2](#0-1) [3](#0-2) [4](#0-3) 

Every public bridge entry point calls one of these hooks:

- `initTransfer` / `initTransfer1155` → `initTransferExtension` → `wormholeNonce++`
- `finTransfer` → `finTransferExtension` → `wormholeNonce++`
- `deployToken` / `addCustomToken` → `deployTokenExtension` → `wormholeNonce++`
- `logMetadata` / `logMetadata1155` → `logMetadataExtension` → `wormholeNonce++` [5](#0-4) [6](#0-5) 

Under Solidity `^0.8.24`, incrementing a `uint32` that already holds `4,294,967,295` reverts with an arithmetic overflow. Because the increment happens inside the extension hook that is called from every public entry point, **all bridge operations revert simultaneously and permanently** once the counter saturates. There is no admin setter or reset path for `wormholeNonce`.

The Wormhole interface itself accepts `uint32` for the nonce parameter:

```solidity
function publishMessage(
    uint32 nonce,
    bytes memory payload,
    uint8 consistencyLevel
) external payable returns (uint64 sequence);
``` [7](#0-6) 

Wormhole's own replay-protection relies on its internal `sequence` counter, not on this nonce field; the nonce is purely a user-defined tag. Therefore the counter could safely wrap (or be stored as `uint256` and cast) without any security consequence, but the current code does not do so.

---

### Impact Explanation

Once `wormholeNonce` saturates:

- **`initTransfer` / `initTransfer1155` revert** — users cannot lock tokens to bridge out; any ETH or ERC-20 sent with the call is returned, but the bridge is effectively closed for outbound transfers.
- **`finTransfer` reverts** — in-flight transfers whose proofs arrive after saturation can never be finalized; the corresponding tokens or native assets locked on the source chain become permanently unclaimable.
- **`deployToken` / `logMetadata` revert** — no new tokens can be registered.

This matches the allowed impact: **Permanent freezing / irrecoverable lock of user or protocol funds in bridge flows.**

---

### Likelihood Explanation

`uint32` max is 4,294,967,295. Both `initTransfer` and `finTransfer` each increment the counter, so a round-trip bridge transfer consumes two increments. At a sustained rate of 10 operations per second the counter saturates in roughly 13.6 years; at 100 ops/second, ~1.4 years. On a high-volume EVM chain (Arbitrum, Base) with automated relayers, the combined init + fin + metadata traffic can easily reach tens of operations per second. The counter is also shared across all token types (ERC-20, ERC-1155, native ETH), multiplying the rate. There is no mechanism to pause, reset, or migrate the counter short of a full UUPS upgrade — which itself requires the admin to notice the impending saturation before it occurs.

---

### Recommendation

Store the counter as `uint256` and cast to `uint32` only at the call site, accepting silent wrapping (which is harmless because Wormhole ignores the nonce for replay protection):

```solidity
// Before
uint32 public wormholeNonce;
...
wormholeNonce++;

// After
uint256 public wormholeNonce;
...
_wormhole.publishMessage{value: ...}(
    uint32(wormholeNonce),   // safe cast; Wormhole nonce is informational only
    payload,
    _consistencyLevel
);
wormholeNonce++;
```

This eliminates the overflow revert while preserving full compatibility with the Wormhole interface.

---

### Proof of Concept

1. Deploy `OmniBridgeWormhole` on a local fork.
2. Set `wormholeNonce` to `type(uint32).max` (e.g., via a storage slot write in a test).
3. Call `initTransfer(...)` with a valid ERC-20 approval.
4. Observe the transaction reverts with `Panic(0x11)` (arithmetic overflow) before any token movement occurs.
5. Repeat for `finTransfer` — the call reverts even with a valid MPC signature, leaving the recipient unable to claim their tokens. [8](#0-7) [9](#0-8) [10](#0-9)
