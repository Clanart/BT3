### Title
uint64 Underflow in `ApplyMessageWithConfig` Nonce Reconciliation Resets Sender Nonce on Failed Contract Creation, Enabling Nonce Replay - (File: x/evm/keeper/state_transition.go)

### Summary

`ApplyMessageWithConfig` in `x/evm/keeper/state_transition.go` implements a nonce reset-and-reconcile pattern for `CREATE` transactions. When `evm.Create()` fails (constructor reverts, out-of-gas, address collision), go-ethereum reverts the nonce increment via snapshot rollback, leaving the stateDB nonce at `msg.Nonce`. The subsequent reconciliation arithmetic performs an unchecked `uint64` subtraction that underflows, ultimately writing `msg.Nonce` (the pre-transaction nonce) back to the account keeper instead of `msg.Nonce + 1`. This breaks the monotonic nonce invariant and allows the sender to reuse the same nonce for a subsequent transaction.

### Finding Description

The nonce reconciliation block in `ApplyMessageWithConfig` is:

```go
if contractCreation {
    oldNonce := stateDB.GetNonce(sender)          // = msg.Nonce + 1 (ante-handler pre-incremented)
    stateDB.SetNonce(sender, msg.Nonce, ...)       // reset to msg.Nonce before evm.Create
    ret, _, leftoverGas, vmErr = evm.Create(...)
    afterCreateNonce := stateDB.GetNonce(sender)   // = msg.Nonce when Create fails (snapshot reverted)
    nestedCreates := afterCreateNonce - msg.Nonce - 1  // uint64 underflow: 0 - 1 = 2^64-1
    stateDB.SetNonce(sender, oldNonce+nestedCreates, ...) // (msg.Nonce+1) + (2^64-1) wraps to msg.Nonce
}
``` [1](#0-0) 

The ante handler pre-increments the nonce to `msg.Nonce + 1` before execution: [2](#0-1) 

In go-ethereum's `evm.create()`, the nonce increment (`SetNonce(caller, nonce+1, ...)`) is placed **inside** the snapshot scope. When creation fails (constructor `revert`, out-of-gas, or `ErrContractAddressCollision` before the snapshot), `RevertToSnapshot` restores the nonce to `msg.Nonce`. The `ErrContractAddressCollision` path returns before the snapshot is even taken, so the nonce is never incremented at all. In both cases `afterCreateNonce == msg.Nonce`.

The subtraction `afterCreateNonce - msg.Nonce - 1` is a bare `uint64` operation in Go. With `afterCreateNonce == msg.Nonce`:

```
nestedCreates = msg.Nonce - msg.Nonce - 1 = 0 - 1 = 18446744073709551615  (2^64 - 1)
```

Then:

```
oldNonce + nestedCreates = (msg.Nonce + 1) + (2^64 - 1) = msg.Nonce  (mod 2^64)
```

`stateDB.SetNonce(sender, msg.Nonce, ...)` is called, and `stateDB.Commit()` writes this value to the account keeper, overwriting the ante-handler's increment. [3](#0-2) 

The stateDB is committed unconditionally when `commit = true`, regardless of `vmErr`: [4](#0-3) 

### Impact Explanation

After a failed contract creation, the sender's on-chain nonce is `msg.Nonce` instead of `msg.Nonce + 1`. The sender can now submit a second transaction with nonce `msg.Nonce` — a nonce that was already consumed by the failed creation. This is a direct nonce replay: the same nonce slot is reusable, breaking Ethereum's replay-protection invariant. Any protocol or contract that relies on the sender's nonce being strictly monotonic (e.g., CREATE2 factory patterns, off-chain nonce tracking, EIP-712 permit flows) is affected. Additionally, any legitimate pending transaction the sender signed with nonce `msg.Nonce + 1` will be permanently invalidated, since the chain now expects nonce `msg.Nonce` next.

This matches the allowed High impact: *"Ethereum transaction … nonce … verification bypass enabling replay, forged execution, or unauthorized account/code mutation."*

### Likelihood Explanation

Any unprivileged user can trigger this by submitting a contract creation whose constructor reverts (e.g., `revert()` in the constructor body, or a constructor that consumes all gas). This is a standard, freely available transaction type. No special privileges, governance access, or validator collusion is required. The path goes through the normal `eth_sendRawTransaction` → ante handler → `ApplyTransaction` → `ApplyMessageWithConfig` consensus execution flow.

### Recommendation

Guard the subtraction against underflow. The correct fix is to check whether `evm.Create()` actually incremented the nonce before computing `nestedCreates`:

```go
afterCreateNonce := stateDB.GetNonce(sender)
if afterCreateNonce > msg.Nonce {
    nestedCreates := afterCreateNonce - msg.Nonce - 1
    stateDB.SetNonce(sender, oldNonce+nestedCreates, tracing.NonceChangeUnspecified)
} else {
    // Creation failed before nonce was incremented; restore oldNonce unchanged.
    stateDB.SetNonce(sender, oldNonce, tracing.NonceChangeUnspecified)
}
```

This ensures that a failed contract creation leaves the nonce at `oldNonce` (= `msg.Nonce + 1`, as set by the ante handler), matching go-ethereum's behavior.

### Proof of Concept

1. Alice has account nonce `N`. Ante handler pre-increments to `N+1`.
2. Alice sends a `CREATE` transaction (nonce = `N`) whose constructor always calls `revert()`.
3. `ApplyMessageWithConfig` sets stateDB nonce to `N`, calls `evm.Create()`.
4. `evm.Create()` takes a snapshot, increments nonce to `N+1`, runs constructor, constructor reverts, snapshot is restored → stateDB nonce = `N`.
5. `afterCreateNonce = N`, `nestedCreates = N - N - 1 = 2^64-1`, `stateDB.SetNonce(sender, N+1 + 2^64-1) = N`.
6. `stateDB.Commit()` writes nonce `N` to account keeper.
7. Alice's on-chain nonce is now `N`. She submits a second transaction with nonce `N` (e.g., a value transfer to herself), which is accepted by the chain.
8. Alice has effectively executed two transactions for the price of one nonce slot, with the second transaction being a replay of the nonce consumed by the first. [1](#0-0)
