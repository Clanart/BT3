### Title
Malicious ERC20 Token Can Fake `safeTransferFrom` to Trigger Unauthorized Minting on NEAR — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` accepts any user-supplied `tokenAddress` and calls `IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount)` without verifying the contract's actual token balance changed. A malicious ERC20 that returns `true` from `transferFrom` without moving tokens causes the bridge to emit a valid `InitTransfer` event, which a relayer then proves to the NEAR bridge, triggering an unbacked mint of bridged tokens on NEAR.

---

### Finding Description

`initTransfer` in `OmniBridge.sol` handles three token categories. For tokens that are neither a registered bridge token nor a custom-minter token, it falls into the `else` branch:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
``` [1](#0-0) 

There is no balance-before / balance-after check. OpenZeppelin's `safeTransferFrom` only checks the boolean return value of `transferFrom`; it does not verify that the contract's actual balance of `tokenAddress` increased by `amount`. A malicious ERC20 can return `true` from `transferFrom` while transferring nothing.

After the call, the bridge unconditionally emits:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

This event is emitted by the OmniBridge contract itself, which is the registered factory address. The NEAR bridge's `fin_transfer_callback` validates only that the emitter is the known factory:

```rust
require!(
    self.factories.get(&init_transfer.emitter_address.get_chain())
        == Some(init_transfer.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
``` [3](#0-2) 

Since the event originates from the legitimate OmniBridge contract, this check passes. The NEAR bridge then mints the full `amount` of the corresponding NEAR token to the attacker's recipient address.

The prerequisite — registering the fake token — is also permissionless. `logMetadata` accepts any `tokenAddress` and calls `name()`, `symbol()`, `decimals()` on it without any whitelist check:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name = IERC20Metadata(tokenAddress).name();
    string memory symbol = IERC20Metadata(tokenAddress).symbol();
    uint8 decimals = IERC20Metadata(tokenAddress).decimals();
    logMetadataExtension(tokenAddress, name, symbol, decimals);
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
``` [4](#0-3) 

The resulting `LogMetadata` event is emitted by the factory, so `deploy_token_callback` on NEAR accepts it and registers the fake token in `token_decimals` and `token_address_to_id`, satisfying the lookup in `fin_transfer_callback`. [5](#0-4) 

---

### Impact Explanation

An attacker can mint an unbounded quantity of any bridged NEAR token without depositing any real EVM-side collateral. Every minted token is unbacked, directly breaking bridge collateralization and enabling theft of real assets held by other users who later redeem their legitimately-backed tokens. This matches the allowed impact: **Critical — direct unauthorized mint of bridged assets**.

---

### Likelihood Explanation

The attack is fully permissionless on the EVM side. The only external dependency is a trusted relayer processing the proof on NEAR, but relayers are expected to process all valid factory events automatically — they have no mechanism to distinguish a malicious token from a legitimate one. Deploying a malicious ERC20 is trivial and costs only gas. No privileged role, leaked key, or colluding MPC signer is required.

---

### Recommendation

Apply a balance-before / balance-after guard in the `else` branch of `initTransfer`, analogous to the mitigation recommended in the external report:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 balanceAfter = IERC20(tokenAddress).balanceOf(address(this));
    require(balanceAfter - balanceBefore >= amount, "insufficient transfer");
}
```

This ensures the bridge's actual custody of the token increased by at least `amount` before emitting the `InitTransfer` event, regardless of what the token's `transferFrom` returns.

---

### Proof of Concept

1. **Deploy** a malicious ERC20 `FakeToken` that:
   - Returns arbitrary strings from `name()`, `symbol()`, and a valid `uint8` from `decimals()`.
   - Returns `true` from `transferFrom(from, to, amount)` without updating any balances.

2. **Register the fake token** — call `OmniBridge.logMetadata(FakeToken)`. OmniBridge emits `LogMetadata(FakeToken, ...)` from the factory address. A relayer submits the proof to the NEAR bridge's `deploy_token`, which passes the factory check and registers `FakeToken` in `token_decimals` and `token_address_to_id`.

3. **Initiate a fake transfer** — call `OmniBridge.initTransfer(FakeToken, 1_000_000e18, 0, 0, "near:attacker.near", "")`. The `else` branch calls `FakeToken.safeTransferFrom(attacker, bridge, 1_000_000e18)`, which returns `true` without moving tokens. OmniBridge emits `InitTransfer(attacker, FakeToken, nonce, 1_000_000e18, ...)`.

4. **Finalize on NEAR** — a trusted relayer calls `fin_transfer` with the EVM proof. `fin_transfer_callback` verifies the factory, looks up the registered `FakeToken` decimals, and calls `process_fin_transfer_to_near`, which mints `1_000_000e18` (normalized) of the corresponding NEAR token to `attacker.near`.

5. **Result**: The attacker holds 1,000,000 NEAR-side bridged tokens backed by zero EVM collateral. Other users who legitimately bridged the same token can no longer redeem their holdings.
