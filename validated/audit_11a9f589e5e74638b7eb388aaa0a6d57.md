### Title
Missing Class Hash Existence Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash provided by a contract corresponds to a previously declared class. This allows any contract to replace its own class hash with an arbitrary, undeclared value, permanently rendering the contract uncallable and freezing any funds it holds. The missing check is even acknowledged by a TODO comment in the code.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function accepts any `class_hash` value from the `ReplaceClassRequest` and directly writes it into `contract_state_changes` without checking whether that class hash exists in `contract_class_changes` (the dictionary tracking declared classes).

The code explicitly acknowledges this missing check: [1](#0-0) 

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The `contract_class_changes` dictionary — which maps declared class hashes to their compiled class hashes — is never consulted here. The OS will prove and commit a state transition where a contract's class hash is set to an arbitrary felt value that has no corresponding entry in the class trie.

This is the direct analog of the external report's vulnerability: just as `MinVotingPowerCondition` checked a mutable token balance (transient state) instead of the locked amount (committed state), `execute_replace_class` writes a new class hash into the contract state without checking the committed declared-class state (`contract_class_changes`). The invariant — that every contract's class hash must correspond to a declared class — is enforced nowhere in this path.

The `execute_replace_class` function is dispatched from `execute_syscalls` when the selector matches `REPLACE_CLASS_SELECTOR`: [2](#0-1) 

The `contract_address` passed to `execute_replace_class` is always the currently executing contract's address: [3](#0-2) 

After the block's transactions complete, `state_update` squashes and commits `contract_state_changes` into the Patricia trie: [4](#0-3) 

The invalid class hash is committed to the global state root with no further validation. Once committed, no transaction can ever execute against this contract again, because the OS cannot find a compiled class for the arbitrary hash.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that holds user funds (e.g., a shared vault, a multisig, a token contract, or any contract receiving ETH/ERC-20 tokens) can have those funds permanently frozen if `replace_class` is called with an undeclared class hash. After the state is committed:

- The contract's class hash in the Patricia trie is an arbitrary felt with no corresponding compiled class.
- Every future call to the contract fails at the OS level when it attempts to look up the class.
- There is no recovery path: since the contract is uncallable, no subsequent transaction can call `replace_class` again to fix it.
- All funds held by the contract are permanently inaccessible.

---

### Likelihood Explanation

The `replace_class` syscall is reachable by any unprivileged contract. An attacker needs only to:

1. Deploy a contract (or exploit a bug in an existing contract that calls `replace_class`).
2. Invoke the contract so it calls `replace_class` with an arbitrary felt (e.g., `0xdeadbeef`).

No privileged role, leaked key, or external dependency is required. The OS Cairo program itself accepts and proves the invalid state transition. The TODO comment confirms the Aragon team is aware the check is absent.

---

### Recommendation

Before updating `contract_state_changes` in `execute_replace_class`, verify that the new `class_hash` exists in `contract_class_changes`. Specifically, perform a `dict_read` on `contract_class_changes` with `key=class_hash` and assert the returned compiled class hash is non-zero (indicating the class was declared in a prior block or the current block). This mirrors how `execute_declare_transaction` enforces `prev_value=0` to prevent double-declaration: [5](#0-4) 

---

### Proof of Concept

1. Attacker deploys contract `A` with a valid class `C1`. Contract `A` holds user funds (e.g., received via constructor or subsequent transfers).
2. Attacker sends an invoke transaction calling contract `A`. Inside `A`'s `__execute__`, it issues the `replace_class` syscall with `class_hash = 0xdeadbeef` (a felt that was never declared).
3. The OS processes `execute_replace_class` at `syscall_impls.cairo:878–916`. The TODO check is absent; `contract_state_changes` is updated with `class_hash = 0xdeadbeef` for contract `A`.
4. `state_update` squashes and commits the state. Contract `A`'s class hash in the global state root is now `0xdeadbeef`.
5. Any future transaction targeting contract `A` fails: the OS cannot find a compiled class for `0xdeadbeef` in the compiled class facts bundle.
6. All funds in contract `A` are permanently frozen with no recovery path.
