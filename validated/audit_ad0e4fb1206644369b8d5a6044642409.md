### Title
Dust-Amount Transfer Permanently Locks User Funds via Irrecoverable `sign_transfer` Panic - (File: `near/omni-bridge/src/lib.rs`)

### Summary

A user can initiate a bridge transfer with an amount that is too small to be representable on the destination chain after decimal normalization. The `init_transfer` entry-point accepts the transfer and burns/locks the user's tokens, but the subsequent `sign_transfer` call permanently panics with `InvalidAmountToTransfer` because `normalize_amount` floors the result to zero. There is no recovery path: the pending transfer entry cannot be removed and the burned/locked tokens are irrecoverable.

### Finding Description

The bridge uses a two-step process for NEAR→EVM transfers:

**Step 1 — `init_transfer`** (via `ft_transfer_call`): Validates only that `fee < amount`, then burns or locks the user's tokens and stores the transfer in `pending_transfers`.

**Step 2 — `sign_transfer`**: Computes the destination-chain amount by applying `normalize_amount`, which performs floor division to account for decimal differences between NEAR and the destination chain. It then requires the result to be non-zero.

The critical mismatch is that `init_transfer` does **not** verify that `amount - fee` is representable on the destination chain, while `sign_transfer` enforces this with a hard panic.

`normalize_amount` is defined as:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

And `sign_transfer` enforces:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [2](#0-1) 

But `init_transfer` only checks:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [3](#0-2) 

After the check passes, `init_transfer_internal` burns or locks the tokens and stores the pending entry: [4](#0-3) 

Once stored, the only removal paths for a pending transfer are:
- `sign_transfer_callback` — only reached if `sign_transfer` succeeds (it panics instead)
- `claim_fee_callback` — requires a proof from the destination chain (impossible since the transfer was never signed)
- `remove_transfer_message_without_refund` — only called inside `init_transfer_internal` on storage failure, not reachable here [5](#0-4) 

### Impact Explanation

**Critical — Permanent freezing of user funds.**

Any user who sends a token amount smaller than `10^(origin_decimals - decimals)` will have their tokens permanently burned or locked with no recovery path. For example, with a token having 24 NEAR-side decimals and 18 EVM-side decimals (a 6-decimal gap, common for NEAR-native tokens bridged to EVM), any transfer of fewer than 1,000,000 base units results in a stuck transfer. The user loses their tokens entirely. The `update_transfer_fee` function cannot rescue the transfer because increasing the fee to `amount - 1` still yields `normalize_amount(1, decimals) = 0`.

### Likelihood Explanation

**Medium.** Tokens with decimal mismatches between NEAR and EVM are a normal part of the bridge's design (the `Decimals` struct with `origin_decimals` and `decimals` fields exists precisely for this). A user sending a "dust" amount — either accidentally or due to a UI rounding error — will trigger this permanently. No special privileges are required; any unprivileged token holder calling `ft_transfer_call` with a small amount is sufficient.

### Recommendation

Add a representability check in `init_transfer` (or `init_transfer_internal`) before burning/locking tokens. Specifically, after computing the token address and decimals, verify that `normalize_amount(amount - fee, decimals) > 0`. If the check fails, return the full token amount as a NEP-141 refund (return `transfer_message.amount` from `ft_on_transfer`) without storing the pending entry or touching the user's balance.

Alternatively, enforce a minimum transfer amount at the `ft_on_transfer` / `init_transfer` boundary that accounts for the decimal normalization factor of the destination token.

### Proof of Concept

1. Admin registers a token with `origin_decimals = 24`, `decimals = 18` (6-decimal gap, factor of 1,000,000).
2. User calls `ft_transfer_call` on the token contract with `amount = 500_000` (less than 1,000,000), `msg = InitTransfer { fee: 0, recipient: eth:0x..., ... }`.
3. `init_transfer` passes the `fee (0) < amount (500_000)` check. Tokens are burned. Transfer stored in `pending_transfers`.
4. Relayer calls `sign_transfer` for this transfer ID.
5. `normalize_amount(500_000, Decimals { origin_decimals: 24, decimals: 18 })` = `500_000 / 1_000_000` = `0` (floor division).
6. `require!(0 > 0, ...)` panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
7. Every subsequent `sign_transfer` call panics identically. The 500,000 base-unit tokens are permanently burned. The pending transfer entry is never removed. [6](#0-5) [7](#0-6)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
