### Title
`HyperliquedBridgeToken` 3-arg `mint` misdirects `finTransfer` tokens to `_systemAddress` when `DestHexMsg` is set — (`evm/src/omni-bridge/contracts/HlBridgeToken.sol`)

---

### Summary

`HyperliquedBridgeToken` overrides `mint(address,uint256,bytes)` to call `_mint(account, value)` immediately followed by `_update(account, _systemAddress, value)`, routing all freshly minted tokens away from the recipient to `_systemAddress`. `OmniBridge.finTransfer` dispatches to this 3-arg overload whenever `payload.message.length != 0`. Any user bridging from NEAR who sets `msg = {"DestHexMsg":"<hex>"}` in their `InitTransferMsg` will produce a valid MPC-signed `finTransfer` payload with a non-empty `message`, triggering this path and causing the recipient to receive zero tokens.

---

### Finding Description

**Step 1 — User-controlled `msg` on NEAR**

`InitTransferMsg.msg` is an `Option<BoundedString<MAX_INIT_TRANSFER_MSG_LEN>>` freely set by any caller. [1](#0-0) 

**Step 2 — `DestHexMsg` variant produces a non-empty `message` bytes vector**

In `sign_transfer`, the stored `transfer_message.msg` string is parsed as `DestinationChainMsg`. The `DestHexMsg` variant's `destination_msg()` returns `Some(Vec<u8>)`, which becomes the `message` field of the signed `TransferMessagePayload`. If the user sets `msg = {"DestHexMsg":"00"}`, `message = vec![0x00]`. [2](#0-1) [3](#0-2) 

**Step 3 — MPC signs the payload including the non-empty `message`**

The `message` field is included in the borsh-encoded hash that the MPC signs. The signature is therefore valid for a payload with `message = [0x00]`. [4](#0-3) 

**Step 4 — `finTransfer` dispatches to 3-arg `mint`**

`OmniBridge.finTransfer` branches on `payload.message.length`: if non-zero and `isBridgeToken[tokenAddress]`, it calls `IBridgeToken(tokenAddress).mint(recipient, amount, message)`. [5](#0-4) 

**Step 5 — `HyperliquedBridgeToken` overrides 3-arg `mint` to route tokens to `_systemAddress`**

Unlike the base `BridgeToken` which simply calls `_mint(account, value)` and ignores the bytes argument, `HyperliquedBridgeToken` additionally calls `_update(account, _systemAddress, value)`, which is an ERC-20 internal transfer that moves the entire freshly minted balance from `account` to `_systemAddress`. Net result: `account.balanceOf == 0`, `_systemAddress.balanceOf += value`. [6](#0-5) [7](#0-6) 

The `_systemAddress` pool can only be moved by the HyperLiquid system itself via `coreReceiveWithData` (guarded by `msg.sender != _systemAddress`), so the recipient has no path to recover the tokens. [8](#0-7) 

---

### Impact Explanation

Any user who bridges tokens from NEAR to a `HyperliquedBridgeToken` address on HyperEVM and includes a `DestHexMsg` payload in their `InitTransferMsg` will have their tokens minted directly to `_systemAddress` instead of the intended recipient. The recipient receives zero tokens. The tokens are not permanently destroyed (they sit in the `_systemAddress` pool), but they are irrecoverable by the recipient through any public bridge call. This constitutes a direct, permanent loss of bridged assets for the recipient.

---

### Likelihood Explanation

The `DestHexMsg` variant of `DestinationChainMsg` is a documented, public feature for forwarding hex-encoded payloads to the destination chain. A user who legitimately uses this feature when bridging to a `HyperliquedBridgeToken` (e.g., to trigger a downstream hook) will silently lose their tokens. Additionally, an attacker can deliberately initiate a transfer from NEAR with `recipient = victim` and `msg = {"DestHexMsg":"00"}`, spending their own tokens to grief the victim (victim receives nothing). No privileged access, leaked keys, or MPC collusion is required — the MPC faithfully signs whatever `message` the NEAR-side transfer message produces.

---

### Recommendation

1. **Override `mint(address,uint256,bytes)` to revert** in `HyperliquedBridgeToken`, or explicitly document that the 3-arg path must never be called via `finTransfer`. The HyperCore accounting path should be triggered only by `coreReceiveWithData`, not by `finTransfer`.
2. **Alternatively**, in `OmniBridge.finTransfer`, always call the 2-arg `mint` regardless of `message.length`, and pass the message to a separate hook (e.g., `onMintWithMessage`) if needed.
3. **Or**, in `HyperliquedBridgeToken.mint(address,uint256,bytes)`, revert unconditionally so that any `finTransfer` with a non-empty message for this token type fails safely rather than silently misdirecting funds.

pub enum DestinationChainMsg {
    MaxGasFee(U64),
    DestHexMsg(#[serde_as(as = "Hex")] Vec<u8>),
}

impl DestinationChainMsg {
    pub fn max_gas_fee(&self) -> Option<U128> {
        if let Self::MaxGasFee(fee) = self {
            Some(U128(fee.0.into()))
        } else {
            None
        }
    }

    pub fn destination_msg(&self) -> Option<Vec<u8>> {
        if let Self::DestHexMsg(msg) = self {
            Some(msg.clone())
        } else {
            None
        }
    }

    pub fn from_json(s: &str) -> Option<Self> {
        serde_json::from_str(s).ok()
    }
