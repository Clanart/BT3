### Title
`state_diff_length` in Stored Block Header Not Cross-Validated Against `concatenated_counts` in Block Hash Commitment Allows Wrong State Diff Acceptance — (File: `crates/apollo_p2p_sync/src/client/header.rs`)

### Summary

In `convert_sync_block_to_block_data`, the `state_diff_length` written into the stored `BlockHeader` is taken directly from `sync_block.state_diff.len()`, while the `concatenated_counts` field in `BlockHeaderCommitments` — which is the value actually committed to in the block hash — independently encodes its own `state_diff_length`. No cross-check between these two values is performed. An explicit TODO in the code acknowledges this gap. Because the P2P sync client uses the stored header's `state_diff_length` as the authoritative target length when reassembling state diff chunks from peers, a divergence between the two values causes the client to accept a state diff of the wrong length, storing a `ThinStateDiff` that is inconsistent with the committed block hash.

### Finding Description

`calculate_block_commitments` packs `transaction_count | event_count | state_diff_length | l1_da_mode` into a single felt called `concatenated_counts`, which is then chained directly into the block hash via `calculate_block_hash`. [1](#0-0) [2](#0-1) 

When a `SyncBlock` is received internally and converted to a `SignedBlockHeader` for storage, `convert_sync_block_to_block_data` sets `state_diff_length` from the raw state diff object and `n_transactions` from the transaction hash lists, but **never verifies these against the `concatenated_counts` already present in `header_commitments`**:

```rust
// TODO(Shahak): Verify `n_transactions` and `state_diff_length` match values in
// concatenated_counts.
SignedBlockHeader {
    block_header: BlockHeader {
        ...
        state_diff_length: Some(sync_block.state_diff.len()),   // ← from state diff object
        n_transactions: sync_block.account_transaction_hashes.len()
            + sync_block.l1_transaction_hashes.len(),           // ← from tx hash lists
        ...
    },
    ...
}
``` [3](#0-2) 

The stored header's `state_diff_length` is then the sole input used by `StateDiffStreamBuilder::parse_data_for_block` to determine `target_state_diff_len` — the number of state diff chunks the P2P sync client will accept before declaring the state diff complete:

```rust
let target_state_diff_len = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    ...
    .state_diff_length
    .ok_or(...)?;

while current_state_diff_len < target_state_diff_len { ... }
``` [4](#0-3) 

If `sync_block.state_diff.len()` (call it **X**) differs from the `state_diff_length` encoded inside `header_commitments.concatenated_counts` (call it **Y**):

1. The stored header carries `state_diff_length = X`.
2. The block hash was computed with `concatenated_counts` encoding `state_diff_length = Y`.
3. When `TryFrom<&BlockHeader> for Option<BlockHeaderCommitments>` reconstructs `concatenated_counts` from the stored header fields, it uses **X**, producing a different felt than the one committed to in the block hash. [5](#0-4) 

4. The P2P sync client accepts a `ThinStateDiff` of length **X** and writes it to storage, while the block hash commits to a state diff of length **Y**.
5. The stored `ThinStateDiff` is therefore inconsistent with the committed `state_diff_commitment` and `concatenated_counts` in the block hash.

The P2P sync server also has a fallback path that computes `state_diff_length` from the stored state diff when the header field is absent, which can further propagate the wrong value to downstream peers: [6](#0-5) 

### Impact Explanation

A node syncing via P2P stores a `ThinStateDiff` whose length does not match the `state_diff_length` encoded in the committed `concatenated_counts`. Any RPC call that reads state from that block (storage values, nonces, class hashes) operates on the wrong state diff, returning authoritative-looking wrong values. If the wrong state diff is shorter than the committed one, storage updates from the tail of the real diff are silently dropped; if longer, spurious updates are injected. Either outcome corrupts the Patricia trie root that downstream provers and the committer use as their starting state, causing wrong global roots and wrong proof inputs.

Matches: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value** and **Critical — Wrong state/storage value from blockifier/syscall/execution logic**.

### Likelihood Explanation

The divergence requires the consensus orchestrator to emit a `SyncBlock` in which `state_diff.len()` and the `state_diff_length` packed into `block_header_commitments.concatenated_counts` disagree. The code explicitly marks this invariant as unverified (the TODO comment). Any future refactor that constructs `SyncBlock` without atomically deriving both fields from the same `ThinStateDiff` instance will silently introduce the divergence. The gap is also present in the P2P external-peer path, where no check prevents a peer from advertising a `state_diff_length` in the protobuf `StateDiffCommitment` that differs from the length implied by `concatenated_counts`.

### Recommendation

In `convert_sync_block_to_block_data`, extract `state_diff_length` and `n_transactions` from `header_commitments.concatenated_counts` using `extract_event_count_from_concatenated_counts` and the analogous transaction/state-diff extractors, then assert equality with `sync_block.state_diff.len()` and the transaction hash counts before writing the header. Alternatively, derive `state_diff_length` exclusively from `concatenated_counts` so there is a single source of truth. The same cross-check should be added in the P2P external-peer header parsing path.

### Proof of Concept

1. The consensus orchestrator constructs a `SyncBlock` where `state_diff` contains **N** entries but `block_header_commitments.concatenated_counts` was computed (e.g., from a stale snapshot) with `state_diff_length = N − k` for some `k > 0`.
2. `convert_sync_block_to_block_data` stores the header with `state_diff_length = N` (from `sync_block.state_diff.len()`).
3. `StateDiffStreamBuilder::parse_data_for_block` reads `target_state_diff_len = N` from the stored header and accepts all **N** entries.
4. The block hash, however, was computed with `concatenated_counts` encoding `N − k`. Recomputing the block hash from the stored header fields yields a different `concatenated_counts` (encoding **N**), producing a hash mismatch.
5. Any verifier or prover that recomputes the block hash from storage (e.g., `compare_block_hash` in `blockifier_reexecution`) will observe a mismatch, and any downstream node that syncs this block will store a `ThinStateDiff` of length **N** while the committed hash expects **N − k**, corrupting its Patricia trie state root.
