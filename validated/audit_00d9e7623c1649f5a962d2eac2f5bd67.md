### Title
Missing Nonce in `meta_tx_v0` Hash Enables Unbounded Signature Replay — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo` and `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `execute_meta_tx_v0` syscall computes a transaction hash that contains **no nonce** and hardcodes `nonce=0` in the resulting `TxInfo`. Because `check_and_increment_nonce` explicitly skips version-0 transactions, the OS provides zero replay protection for meta_tx_v0 calls. The same `(contract_address, selector, calldata, chain_id)` tuple always produces the same hash, so any signature over that hash is valid forever and can be replayed by any unprivileged actor to drain funds from the target account.

---

### Finding Description

**Vulnerability class:** Nonce bypass / missing nonce in commitment hash → transaction replay.

**Root cause — `compute_meta_tx_v0_hash`:**

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        ...
        additional_data_size=0,          // ← NO nonce committed to hash
        additional_data=cast(0, felt*),
    );
    return tx_hash;
}
``` [1](#0-0) 

The `additional_data_size=0` means the nonce is never mixed into the Pedersen hash. Every invocation with the same inputs produces an identical `meta_tx_hash`.

**Root cause — `execute_meta_tx_v0` hardcodes `nonce=0`:**

```cairo
tempvar new_tx_info = new TxInfo(
    version=0,
    ...
    transaction_hash=meta_tx_hash,
    nonce=0,          // ← always zero, never incremented
    ...
);
``` [2](#0-1) 

**Root cause — `check_and_increment_nonce` skips version 0:**

```cairo
func check_and_increment_nonce{contract_state_changes: DictAccess*}(tx_info: TxInfo*) -> () {
    // Do not handle nonce for version 0.
    if (tx_info.version == 0) {
        return ();
    }
    ...
}
``` [3](#0-2) 

`L1_HANDLER_VERSION = 0` is the same constant, confirming version 0 is the designated "no-nonce" path. [4](#0-3) 

**Root cause — `run_validate` also skips version 0:**

```cairo
// Do not run "__validate__" for version 0.
if (tx_execution_info.tx_info.version == 0) {
    return ();
}
``` [5](#0-4) 

The OS therefore neither validates the signature itself nor increments any nonce counter for meta_tx_v0 calls. The only replay protection available is whatever the target contract implements internally — but the OS-provided `transaction_hash` (which the contract uses to verify the signature) is identical across all replays.

---

### Impact Explanation

**Critical — Direct loss of funds.**

The `execute_meta_tx_v0` syscall is explicitly designed to let a contract execute a call on behalf of another contract address using a user-supplied signature:

> *"Executes a v0 meta transaction. Specifically, calls another contract where: The signature is replaced with the given signature … The transaction hash is replaced by the corresponding version-0 transaction hash."* [6](#0-5) 

The target contract receives `tx_info.transaction_hash` and `tx_info.signature_*` and is expected to verify the signature against that hash. Because the hash is nonce-free, the same signature is cryptographically valid for every future replay. An attacker who observes a single on-chain meta_tx_v0 call (e.g., a token transfer authorization) can replay it in any subsequent block to drain the victim's funds.

---

### Likelihood Explanation

**High.** The attacker entry path requires only:
1. Observing a valid meta_tx_v0 call on-chain (the signature and calldata are public).
2. Deploying or reusing any contract that calls the `meta_tx_v0` syscall with the same `(contract_address, selector, calldata, signature)`.

No privileged access, leaked key, or trusted role is required. The attack is fully permissionless and repeatable across blocks.

---

### Recommendation

Include a nonce in `compute_meta_tx_v0_hash` — analogous to how `compute_l1_handler_transaction_hash` passes `additional_data=&nonce` — and enforce uniqueness of that nonce at the OS level (e.g., by reading and incrementing the account's on-chain nonce, or by maintaining a separate consumed-nonce bitmap for meta_tx_v0 calls).

```cairo
// Fix: pass nonce as additional_data
let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
    ...
    additional_data_size=1,
    additional_data=&nonce,   // nonce sourced from account state and incremented
);
``` [7](#0-6) 

---

### Proof of Concept

1. **Victim** calls a contract that internally invokes `meta_tx_v0` with:
   - `contract_address = VICTIM_ACCOUNT`
   - `selector = __execute__`
   - `calldata = [transfer(ATTACKER, 1000_STRK)]`
   - `signature = SIG` (victim's valid ECDSA signature)

2. The OS computes `meta_tx_hash = H('invoke', 0, VICTIM_ACCOUNT, __execute__, H(calldata), 0, chain_id)` — **no nonce**.

3. The victim's account contract verifies `SIG` against `meta_tx_hash` and executes the transfer. Block N completes.

4. **Attacker** in Block N+1 deploys `ReplayContract` that calls `meta_tx_v0(VICTIM_ACCOUNT, __execute__, calldata, SIG)`.

5. The OS recomputes the **identical** `meta_tx_hash` (same inputs, no nonce). The victim's account contract sees the same hash and the same valid `SIG`, accepts it, and executes another transfer.

6. Repeat until the victim's account is drained.
