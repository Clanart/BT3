### Title
Missing Hash-Uniqueness Validation in `MerkleBlob::upsert` Allows Duplicate Leaf Hashes, Corrupting DataLayer Merkle Tree Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::insert` enforces that every leaf hash is unique across the tree before writing. `MerkleBlob::upsert` — the update path for an existing key — performs no such check. An unprivileged caller who supplies a `new_hash` that already belongs to a different leaf silently creates two leaves with identical hashes, corrupting every ancestor hash up to the tree root and invalidating all proofs of inclusion derived from that root.

---

### Finding Description

`MerkleBlob::insert` guards against hash collisions at lines 372–374:

```rust
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::upsert` (lines 792–810) takes the update path when the key already exists. It calls `block_status_cache.remove_leaf(&leaf)` to evict the old hash from the `leaf_hash_to_index` map, then calls `insert_entry_to_blob`, which calls `add_leaf` (lines 188–193):

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index);  // silent overwrite
}
```

`HashMap::insert` silently overwrites any pre-existing entry. If `new_hash` is already present in `leaf_hash_to_index` (owned by a different key at a different index), the cache entry for that other key's hash is silently replaced. After the call:

- The raw blob contains **two leaf nodes with identical hashes** (the upserted leaf and the original owner of `new_hash`).
- `leaf_hash_to_index` has one fewer entry than the actual leaf count, so `check_integrity` will report `IntegrityLeafHashToIndexCacheLength`.
- `mark_lineage_as_dirty` is called only for the upserted leaf's ancestors; the original owner's ancestors are not dirtied, so `calculate_lazy_hashes` will recompute only part of the tree, producing a root hash that is inconsistent with the actual leaf set.
- `get_proof_of_inclusion` for the key whose hash was overwritten in the cache will silently return a proof built from the wrong node, making the proof structurally invalid against the committed root.

The analogous creation-time guard is present; the update-time guard is absent — the same pattern as the referenced report.

---

### Impact Explanation

Matches allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

Concretely:
- The committed Merkle root diverges from the actual key-value set.
- Proofs of inclusion generated after the corrupt `upsert` are structurally invalid; a verifier checking them against the stored root will reject legitimate entries.
- Conversely, a proof generated for the key whose hash was "stolen" may appear valid against the wrong root, enabling a forged-inclusion scenario.
- The corruption is silent at call time; it is only detectable by an explicit `check_integrity()` call, which is not mandatory in the hot path.

---

### Likelihood Explanation

The `upsert` method is exposed directly through the Python wheel bindings (visible in `wheel/python/chia_rs/datalayer.pyi`). Any caller that controls the `hash` argument — including DataLayer sync code that accepts hashes from remote peers — can trigger this with a hash value equal to any hash already present in the tree. No cryptographic collision is required; the attacker only needs to observe an existing leaf hash (which is stored in plaintext in the blob) and supply it as the `new_hash` for a different key.

---

### Recommendation

Add the same guard that `insert` uses, before the `remove_leaf` call, so the old hash is not evicted before the check:

```rust
pub fn upsert(&mut self, key: KeyId, value: ValueId, new_hash: &Hash) -> Result<(), Error> {
    let Ok((leaf_index, mut leaf, mut block)) = self.get_leaf_by_key(key) else {
        self.insert(key, value, new_hash, InsertLocation::Auto {})?;
        return Ok(());
    };

+   // Reject if new_hash is already owned by a *different* leaf
+   if leaf.hash != *new_hash && self.block_status_cache.contains_leaf_hash(new_hash) {
+       return Err(Error::HashAlreadyPresent());
+   }

    self.block_status_cache.remove_leaf(&leaf)?;
    leaf.hash.clone_from(new_hash);
    leaf.value = value;
    block.node = Node::Leaf(leaf);
    self.insert_entry_to_blob(leaf_index, &block)?;
    ...
}
```

The guard must be placed before `remove_leaf` to avoid a TOCTOU window where the old hash is evicted before the collision is detected.

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

let mut blob = MerkleBlob::new(vec![]).unwrap();

let hash_a = Hash(Bytes32::new([0xAA; 32]));
let hash_b = Hash(Bytes32::new([0xBB; 32]));

// Insert two distinct leaves
blob.insert(KeyId(1), ValueId(1), &hash_a, InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(2), ValueId(2), &hash_b, InsertLocation::Auto {}).unwrap();

// upsert key 1 with hash_b — the hash already owned by key 2
// insert() would return Err(HashAlreadyPresent); upsert() silently succeeds
blob.upsert(KeyId(1), ValueId(99), &hash_b).unwrap();

// Tree now has two leaves with hash_b; cache is inconsistent
// check_integrity() will return IntegrityLeafHashToIndexCacheLength
assert!(blob.check_integrity().is_err());
