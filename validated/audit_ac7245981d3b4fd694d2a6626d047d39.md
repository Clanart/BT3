### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Breaks Bridge Collateralization - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`OmniBridge.sol`'s `initTransfer` function uses the caller-supplied `amount` parameter in the emitted `InitTransfer` event without verifying the actual number of tokens received by the contract. For fee-on-transfer (FOT) ERC20 tokens, the contract receives `amount - FOT_fee` but the cross-chain message records `amount`, causing the NEAR bridge to mint or release more tokens than the EVM vault actually holds. This permanently undercollateralizes the bridge for every FOT token transfer.

### Finding Description

In `OmniBridge.sol`, the `initTransfer` function for non-bridge, non-custom-minter ERC20 tokens performs:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount          // requested amount, not actual received
);
```

Immediately after, it emits the cross-chain event using the same caller-supplied `amount`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,         // ← user-supplied, not actual received
    fee,
    nativeFee,
    recipient,
    message
);
```

For a standard ERC20 token, `safeTransferFrom` guarantees the contract receives exactly `amount`. For a FOT token (e.g., PAXG, STA, or any token with a transfer tax), the contract receives `amount - transfer_tax` while the event records `amount`. The NEAR bridge reads this event and mints or releases the full `amount` on the destination chain, but the EVM vault only holds `amount - transfer_tax` as collateral.

The same mismatch occurs in the reverse direction: `finTransfer` calls `IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount)`, which for a FOT token delivers `payload.amount - transfer_tax` to the recipient while the NEAR side has already burned the full `payload.amount`. Each round trip drains the vault by `transfer_tax` per transfer.

### Impact Explanation

Every `initTransfer` with a FOT token creates a permanent collateral deficit equal to the transfer tax. After N transfers of `amount` tokens with tax rate `r`, the vault is short by `N * amount * r` tokens. When users attempt to bridge back from NEAR to EVM, the vault cannot fulfill redemptions for the last users — their funds are permanently frozen in the NEAR contract with no corresponding EVM collateral. This matches the allowed impact: **balance/accounting corruption that breaks bridge collateralization**.

### Likelihood Explanation

The `initTransfer` function accepts any ERC20 `tokenAddress` with no restriction on token type. Any user can call it with a FOT token that has been registered or even an unregistered token (the function does not check `ethToNearToken` mapping before accepting the transfer). FOT tokens are a well-known ERC20 variant in active use on mainnet. No privileged access is required — any bridge user can trigger this.

### Recommendation

Measure the actual received amount by querying the contract's balance before and after the `safeTransferFrom` call, and use the delta as the canonical `amount` for the cross-chain message:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived (cast to uint128) in the event and extension call
```

Alternatively, explicitly disallow FOT tokens by documenting and enforcing that only standard ERC20 tokens (where `transferFrom` delivers exactly `amount`) are supported.

### Proof of Concept

1. A FOT token `FOT` with a 1% transfer tax is deployed on EVM and registered with the bridge (or a user simply calls `initTransfer` with it).
2. User calls `OmniBridge.initTransfer(FOT, 1000, 0, 0, "alice.near", "")`.
3. `safeTransferFrom` moves 1000 FOT from user; contract receives 990 FOT (1% tax taken by token).
4. `InitTransfer` event is emitted with `amount = 1000`.
5. NEAR bridge relayer reads the event and calls `fin_transfer` on NEAR, minting 1000 wrapped-FOT to `alice.near`.
6. Alice now holds 1000 wrapped-FOT on NEAR, but the EVM vault only holds 990 FOT.
7. Alice bridges back: NEAR burns 1000 wrapped-FOT, EVM `finTransfer` tries `safeTransfer(alice_evm, 1000)` — this succeeds (vault still has 990 from Alice's deposit plus prior deposits), but the vault is now 10 FOT short.
8. Repeated over many users, the vault is drained; the last redeemers cannot withdraw their funds. [1](#0-0) [2](#0-1) [3](#0-2)
