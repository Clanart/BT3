### Title
Gateway `__validate__` Bypass via `skip_stateful_validations` Pre-Deployment Window — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the Apollo gateway bypasses the `__validate__` entry point (signature verification) for invoke transactions with nonce=1 when the on-chain account nonce is 0 and `account_tx_in_pool_or_recent_block` returns `true`. The proxy check `account_tx_in_pool_or_recent_block` returns `true` for **any** pending transaction from that address — not exclusively a `deploy_account` transaction. An attacker with a freshly-deployed account (nonce=0 on-chain) can submit one valid invoke tx with nonce=0 to seed the mempool, then submit a second invoke tx with nonce=1 carrying an **invalid or arbitrary signature**, which the gateway accepts without running `__validate__`. This is the direct sequencer analog of the `isContract` constructor bypass: just as a contract has no code during construction and bypasses `isContract`, an account in the "pre-first-transaction" state bypasses signature validation.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` implements a UX feature allowing `deploy_account + invoke` to be submitted simultaneously:

```rust
async fn skip_stateful_validations(...) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await ...;
        }
    }
    Ok(false)
}
``` [1](#0-0) 

When this returns `true`, `run_validate_entry_point` is called with `validate: !skip_validate = false`, meaning the `__validate__` entry point is never invoked:

```rust
let execution_flags = ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

The guard `account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

This returns `true` for **any** transaction type in the pool for that address — not only `deploy_account`. The code comment claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this is incorrect for the case of a freshly-deployed account with nonce=0 on-chain that already has a valid nonce=0 invoke tx in the pool.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce ordering and fee escalation rules — it does not verify the cryptographic signature: [4](#0-3) 

The `validate_state_preconditions` call only checks resource bounds and nonce range — again, no signature check: [5](#0-4) 

The full gateway validation flow is:

```
extract_state_nonce_and_run_validations
  → run_pre_validation_checks
      → validate_state_preconditions  (resource bounds + nonce range only)
      → validate_by_mempool           (nonce ordering only)
      → skip_stateful_validations     ← returns true (bypass)
  → run_validate_entry_point(skip_validate=true)  ← __validate__ NOT called
``` [6](#0-5) 

### Impact Explanation

The gateway accepts an invoke transaction with an invalid or arbitrary signature without running `__validate__`. The transaction enters the mempool and will be included in a block. When the batcher executes it using `AccountTransaction::new_for_sequencing` (which sets `validate: true`), `__validate__` runs and the transaction reverts:

```rust
pub fn new_for_sequencing(tx: Transaction) -> Self {
    let execution_flags = ExecutionFlags {
        only_query: false,
        charge_fee: enforce_fee(&tx, false),
        validate: true,
        strict_nonce_check: true,
    };
    ...
}
``` [7](#0-6) 

The reverted transaction is committed to the block with a revert receipt. The gateway — the primary admission gate — accepted a transaction it should have rejected. This matches the **High** impact scope: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

The trigger requires only two sequential HTTP calls to the gateway from any unprivileged user:
1. A valid invoke tx with nonce=0 for a freshly-deployed account (nonce=0 on-chain).
2. An invoke tx with nonce=1 and an invalid/arbitrary signature.

No privileged access, no malicious peer, no special network position is required. The account only needs to be deployed (class_hash ≠ 0) with nonce=0 on-chain. This is the normal state of any account that has just been deployed and has not yet sent any transactions.

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is pending for the address. The mempool should expose a `deploy_account_tx_in_pool(address) -> bool` query that inspects the transaction type, not just the address presence. Alternatively, the gateway should only skip `__validate__` when the on-chain class hash at the sender address is zero (i.e., the account is genuinely not yet deployed), and a `deploy_account` tx is confirmed to be pending.

### Proof of Concept

**Precondition:** Account `A` is deployed on-chain with nonce=0 (class_hash ≠ 0, no transactions sent yet).

**Step 1 — Seed the mempool:**
```
POST /gateway/add_transaction
{
  "type": "INVOKE",
  "sender_address": A,
  "nonce": "0x0",
  "signature": [<valid_r>, <valid_s>],   // valid ECDSA signature
  ...
}
```
Gateway runs `__validate__`, signature passes, tx enters mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.

**Step 2 — Submit invalid-signature invoke with nonce=1:**
```
POST /gateway/add_transaction
{
  "type": "INVOKE",
  "sender_address": A,
  "nonce": "0x1",
  "signature": ["0xdeadbeef", "0xdeadbeef"],  // invalid signature
  ...
}
```

Gateway evaluates `skip_stateful_validations`:
- `tx.nonce() == 1` ✓
- `account_nonce == 0` (on-chain state) ✓
- `account_tx_in_pool_or_recent_block(A) == true` ✓ (due to nonce=0 tx from Step 1)

`skip_validate = true` → `run_validate_entry_point` sets `validate: false` → `__validate__` is **never called** → invalid-signature tx is accepted by the gateway and enters the mempool. [8](#0-7) 

**Result:** The gateway has admitted an invoke transaction with an invalid signature, violating the admission invariant that all accepted invoke transactions have passed cryptographic signature verification.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
