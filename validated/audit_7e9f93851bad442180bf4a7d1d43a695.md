Audit Report

## Title
Stranded ETH in router consumed for any subsequent WETH-input swap, enabling theft of prior user's ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` checks the router's native ETH balance before pulling from the declared `payer` when `token == WETH`. Any ETH left in the router by a prior user (excess `msg.value` not reclaimed via `refundETH()`) is silently consumed to fund the next caller's WETH swap. The prior user permanently loses their ETH while the attacker receives full swap output at zero cost.

## Finding Description
In `pay()` (L73-84), when `token == WETH` and `payer != address(this)`, the function evaluates `address(this).balance >= value`. If true, it executes `IWETH9(WETH).deposit{value: value}()` and `IERC20(WETH).safeTransfer(recipient, value)` — the `payer` argument is completely ignored and `safeTransferFrom` is never called.

ETH accumulates in the router whenever a user sends `msg.value > amountIn` to a `payable` entry point (e.g., `exactInputSingle`, `exactInput`, `multicall`) without a same-transaction `refundETH()`. `exactInputSingle` (L71) records `payer = msg.sender` via `_setNextCallbackContext`, and `_justPayCallback` (L192-199) passes that payer to `pay()`. However, the payer check is bypassed entirely if the router's ETH balance satisfies the swap amount. An attacker monitoring the router's ETH balance can call `exactInputSingle(tokenIn=WETH, amountIn=<stranded_amount>)` with `msg.value=0` to drain the stranded ETH into a pool swap credited to themselves.

## Impact Explanation
**High.** Direct theft of user principal. Any ETH stranded in the router is claimable by any subsequent caller who submits `exactInputSingle(tokenIn=WETH, amountIn=<stranded_amount>)` with `msg.value=0`. The attacker receives full swap output; the prior user permanently loses their ETH. This is a direct loss of user funds with no recovery path.

## Likelihood Explanation
**Medium.** The precondition — ETH left in the router — arises naturally when users call `exactInputSingle` with excess `msg.value` without a same-transaction `refundETH()`, or when a multicall partially fails after a payable call. The exploit requires no special permissions and is trivially executable by any EOA or MEV bot monitoring the router's ETH balance.

## Recommendation
Remove the native-ETH-first logic from `pay()` for the case where `payer` is an external user. The router's ETH balance should only be used when `payer == address(this)`:

```solidity
} else if (token == WETH) {
    if (payer == address(this)) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

If ETH-as-WETH-input is a desired UX feature, it must be handled explicitly at the entry point (e.g., wrap `msg.value` into WETH on behalf of `msg.sender` before the swap, then refund excess), not inside the callback payment path.

## Proof of Concept
```solidity
function test_strandedEthStolenViaWethSwap() public {
    // User A: sends 150 ETH, swaps 100 WETH, leaves 50 ETH in router
    vm.deal(userA, 150 ether);
    vm.prank(userA);
    router.exactInputSingle{value: 150 ether}(ExactInputSingleParams({
        tokenIn: WETH, amountIn: 100 ether, /* ... */ recipient: userA
    }));
    assertEq(address(router).balance, 50 ether); // 50 ETH stranded

    // User B: sends 0 ETH, swaps 50 WETH — paid by User A's stranded ETH
    vm.prank(userB);
    router.exactInputSingle{value: 0}(ExactInputSingleParams({
        tokenIn: WETH, amountIn: 50 ether, /* ... */ recipient: userB
    }));
    assertEq(address(router).balance, 0);
    // safeTransferFrom(userB, ...) was never called; userB paid nothing
}
```
