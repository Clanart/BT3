### Title
EIP-712 Legacy Signature Verification Accepts Arbitrary Attacker-Controlled `TypedDataChainID` Without Validating Against the Node's Actual EVM Chain ID - (File: `ante/cosmos/eip712.go`)

### Summary

The `LegacyEip712SigVerificationDecorator` in `ante/cosmos/eip712.go` validates that `extOpt.TypedDataChainID` matches the Cosmos chain-ID string parsed from `signerData.ChainID` (i.e., the value from `ctx.ChainID()`). However, it never compares `TypedDataChainID` against the authoritative EVM chain ID stored in the EVM keeper (`k.eip155ChainID`). Because the Cosmos chain-ID string is attacker-supplied in the transaction's `SignDoc.ChainId` field, an attacker can craft a transaction whose `SignDoc.ChainId` encodes an arbitrary EVM integer (e.g., `"attacker_99999-1"`) and set `TypedDataChainID = 99999` to match it. The signature is then verified against a typed-data domain with the wrong chain ID, enabling cross-chain replay of EIP-712 signed Cosmos transactions.

### Finding Description

In `VerifySignature` (`ante/cosmos/eip712.go`), the chain-ID consistency check is:

```go
signerChainID, err := ethermint.ParseChainID(signerData.ChainID)  // parses ctx.ChainID()
...
if extOpt.TypedDataChainID != signerChainID.Uint64() {
    return errorsmod.Wrap(errortypes.ErrInvalidChainID, "invalid chain-id")
}
``` [1](#0-0) 

`signerData.ChainID` is set from `ctx.ChainID()` in `AnteHandle`:

```go
chainID := ctx.ChainID()
signerData := authsigning.SignerData{ChainID: chainID, ...}
``` [2](#0-1) 

`ctx.ChainID()` returns the Cosmos chain-ID string from the transaction's own `SignDoc.ChainId` field (set by the submitter), not from a trusted on-chain source. `ParseChainID` extracts the numeric EIP-155 portion from the Cosmos chain-ID string format `<name>_<eip155>-<epoch>`. [3](#0-2) 

The authoritative EVM chain ID is `k.eip155ChainID` in the EVM keeper: [4](#0-3) [5](#0-4) 

The `LegacyEip712SigVerificationDecorator` holds only an `AccountKeeper` and a `SignModeHandler` — it has **no reference to the EVM keeper** and therefore cannot access `k.eip155ChainID`: [6](#0-5) 

The typed-data domain is then constructed using the attacker-controlled `extOpt.TypedDataChainID`:

```go
typedData, err := eip712.LegacyWrapTxToTypedData(ethermintCodec, extOpt.TypedDataChainID, msgs[0], txBytes, feeDelegation)
``` [7](#0-6) 

Which sets the EIP-712 domain `chainId` to the attacker-supplied value: [8](#0-7) 

The result: a signature produced over a typed-data domain with chain ID `X` (a different chain) passes verification on this chain as long as the attacker also sets `SignDoc.ChainId = "<anyname>_X-1"`. The only constraint is that `TypedDataChainID` must equal the numeric portion of the attacker-supplied `SignDoc.ChainId` string — both are fully attacker-controlled.

The `newLegacyCosmosAnteHandlerEip712` handler, which is still active and wired in `evmd/ante/evm_handler.go`, routes all `ExtensionOptionsWeb3Tx` transactions through this decorator: [9](#0-8) [10](#0-9) 

### Impact Explanation

An attacker who holds a valid EIP-712 signature produced for chain ID `X` (e.g., a Metamask signature for a different Ethermint-based chain) can replay that exact transaction on this chain by:

1. Setting `SignDoc.ChainId = "<anyname>_X-1"` (any valid Ethermint chain-ID string whose numeric portion equals `X`).
2. Setting `ExtensionOptionsWeb3Tx.TypedDataChainID = X`.

The signature verification passes because the typed-data domain is reconstructed with `chainId = X`, matching the original signature. The transaction executes with the signer's authority — enabling unauthorized Cosmos bank transfers, staking operations, governance votes, or any other Cosmos SDK message type supported by the legacy EIP-712 path. This is a **cross-chain replay / EIP-712 authorization bypass** enabling forged execution of Cosmos SDK messages.

### Likelihood Explanation

The legacy EIP-712 path (`ExtensionOptionsWeb3Tx`) is still active in the production ante handler. Any unprivileged user can submit a transaction via the standard Cosmos broadcast endpoint or JSON-RPC. The attacker needs only a valid EIP-712 signature from the victim on any Ethermint-compatible chain (including the same chain with a different epoch number). No special privileges, validator access, or key compromise is required.

### Recommendation

The `LegacyEip712SigVerificationDecorator` must be given access to the authoritative EVM chain ID and must validate `extOpt.TypedDataChainID` against it:

```go
// In VerifySignature or AnteHandle, after extracting extOpt:
if extOpt.TypedDataChainID != evmChainID.Uint64() {
    return errorsmod.Wrap(errortypes.ErrInvalidChainID, "TypedDataChainID does not match node chain ID")
}
```

Where `evmChainID` is obtained from the EVM keeper (e.g., passed into `LegacyEip712SigVerificationDecorator` at construction time alongside the `AccountKeeper`). The current check against `signerData.ChainID` (which is derived from the attacker-supplied `SignDoc.ChainId`) is insufficient and must be replaced or supplemented with a check against the trusted keeper value.

### Proof of Concept

1. Victim signs a legacy EIP-712 Cosmos tx on chain `ethermint_9000-1` (chain ID 9000): `MsgSend` from `victim` to `attacker`, amount 1000 tokens.
2. Attacker takes the victim's `FeePayerSig` bytes and constructs a new transaction targeting the current chain (`ethermint_9000-1` or any chain where the victim has funds):
   - Sets `SignDoc.ChainId = "replay_9000-1"` (numeric portion = 9000, same as victim's chain).
   - Sets `ExtensionOptionsWeb3Tx.TypedDataChainID = 9000`.
   - Uses the victim's original `FeePayerSig`.
3. The check at line 228 passes: `extOpt.TypedDataChainID (9000) == signerChainID.Uint64() (9000)`.
4. `LegacyWrapTxToTypedData` reconstructs the typed-data domain with `chainId = 9000` — identical to the domain the victim signed.
5. `ethcrypto.VerifySignature` succeeds; the transaction commits, transferring the victim's funds.

For cross-chain replay: if the victim signed on a different Ethermint chain also using chain ID 9000 (e.g., a testnet), the same signature replays on mainnet without any modification. [11](#0-10) [12](#0-11) [13](#0-12)

func ParseChainID(chainID string) (*big.Int, error) {
	chainID = strings.TrimSpace(chainID)
	if len(chainID) > 48 {
		return nil, errorsmod.Wrapf(ErrInvalidChainID, "chain-id '%s' cannot exceed 48 chars", chainID)
	}

	matches := ethermintChainID.FindStringSubmatch(chainID)
	if matches == nil || len(matches) != 4 || matches[1] == "" {
		return nil, errorsmod.Wrapf(ErrInvalidChainID, "%s: %v", chainID, matches)
	}

	// verify that the chain-id entered is a base 10 integer
	chainIDInt, ok := new(big.Int).SetString(matches[2], 10)
	if !ok {
		return nil, errorsmod.Wrapf(ErrInvalidChainID, "epoch %s must be base-10 integer format", matches[2])
	}

	return chainIDInt, nil
```

func newLegacyCosmosAnteHandlerEip712(ctx sdk.Context, options HandlerOptions, extra ...sdk.AnteDecorator) sdk.AnteHandler {

	feemarketParams := options.FeeMarketKeeper.GetParams(ctx)
	evmDenom := evmParams.EvmDenom
	chainID := options.EvmKeeper.ChainID()
	chainCfg := evmParams.GetChainConfig()
	ethCfg := chainCfg.EthereumConfig(chainID)
	var txFeeChecker authante.TxFeeChecker

		txFeeChecker = evm.NewDynamicFeeChecker(ethCfg, &evmParams, &feemarketParams)

	decorators := make([]sdk.AnteDecorator, 0, 15+len(extra))
	decorators = append(decorators,
		cosmos.RejectMessagesDecorator{}, // reject MsgEthereumTxs
		// disable the Msg types that cannot be included on an authz.MsgExec msgs field
		cosmos.NewAuthzLimiterDecorator(options.DisabledAuthzMsgs),
		authante.NewSetUpContextDecorator(),
		authante.NewValidateBasicDecorator(),
		authante.NewTxTimeoutHeightDecorator(),
		cosmos.NewMinGasPriceDecorator(options.FeeMarketKeeper, evmDenom, &feemarketParams),
		authante.NewValidateMemoDecorator(options.AccountKeeper),
		authante.NewConsumeGasForTxSizeDecorator(options.AccountKeeper),
		authante.NewDeductFeeDecorator(options.AccountKeeper, options.BankKeeper, options.FeegrantKeeper, txFeeChecker),
		// SetPubKeyDecorator must be called before all signature verification decorators
		authante.NewSetPubKeyDecorator(options.AccountKeeper),

		cosmos.NewLegacyEip712SigVerificationDecorator(options.AccountKeeper, options.SignModeHandler),
		authante.NewIncrementSequenceDecorator(options.AccountKeeper),
		ibcante.NewRedundantRelayDecorator(options.IBCKeeper),

	decorators = append(decorators, extra...)
	return sdk.ChainAnteDecorators(decorators...)
}
