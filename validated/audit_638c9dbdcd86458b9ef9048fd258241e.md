### Title
Blocking Check and Prover Resolve `BlockId::Latest` to Different Concrete Blocks, Allowing Security Gate Bypass — (File: `crates/starknet_transaction_prover/src/proving/virtual_snos_prover.rs`)

---

### Summary

`prove_with_blocking_check` spawns the OS prover and sends the external `starknet_checkTransaction` request in parallel, both carrying the caller-supplied `block_id` verbatim. When `block_id` is `BlockId::Latest`, each side independently resolves "latest" against the live chain at the moment it makes its own RPC call. If a new block is finalized between the two resolutions, the blocking check evaluates the transaction against block N while the prover generates a proof for block N+1. The blocking check's allow/block decision is therefore stale relative to the state actually committed to the proof, directly mirroring the Native protocol finding where off-chain signed data diverges from the on-chain state at execution time.

---

### Finding Description

In `prove_with_blocking_check`:

```rust
// line 236-237
let prove_handle =
    tokio::spawn(async move { prover.run_and_prove(block_id, vec![invoke_tx]).await });

// line 240-242
let check_outcome =
    tokio::time::timeout(timeout_duration,
        client.check_transaction(block_id, transaction))
    .await;
```

The prover task is spawned **before** the check request is sent. Both receive the same `block_id` value, but if that value is `BlockId::Latest`, each independently resolves it to whatever the chain's latest block is at the moment it issues its own downstream RPC call. The prover calls `run_virtual_os` → `virtual_block_executor.execute` → `rpc_state_reader.get_block_header()`, which resolves "latest" at that instant. The blocking check service resolves "latest" when it processes the HTTP request it receives. These two resolutions are not synchronized.

`prove_transaction` explicitly rejects `BlockId::Pending` but does **not** reject `BlockId::Latest`:

```rust
// line 161-165
if matches!(block_id, BlockId::Pending) {
    return Err(VirtualSnosProverError::ValidationError(
        "Pending blocks are not supported; only finalized blocks can be proven."
            .to_string(),
    ));
}
```

There is no analogous guard for `BlockId::Latest`, and no step that pins "latest" to a concrete block number before the two parallel operations begin.

The resulting `ProveTransactionResult` embeds `proof_facts` whose `block_number` and `block_hash` fields are taken from the SNOS output of the **prover's** resolved block, not the block the blocking check evaluated. The blocking check's decision is therefore authoritative-looking but stale.

---

### Impact Explanation

The blocking check is an external compliance/security gate (e.g., Elliptic AML screening). Its purpose is to prevent certain transactions from receiving a valid ZK proof. When the check approves a transaction against block N but the proof is generated for block N+1, the proof facts bind the proof to a block the check never evaluated. A transaction that the external service would have blocked at block N+1 receives a valid, on-chain-submittable proof. The returned `ProveTransactionResult` is authoritative-looking (it contains a real proof and real proof facts) but was produced without a valid security evaluation of the actual execution context.

This matches: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value**, and **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing**.

---

### Likelihood Explanation

`BlockId::Latest` is the natural default for client-side proving (the integration test at line 112 of `virtual_snos_prover_test.rs` uses `BlockId::Latest` directly). Block times on Starknet are on the order of seconds to tens of seconds. The prover's OS execution takes non-trivial time (the code logs `os_duration_ms`). The window in which a new block can be finalized between the prover's resolution and the check's resolution is therefore realistic and repeatable. An attacker who knows a block is about to be finalized can time the request to maximize the probability of the race.

---

### Recommendation

Pin `BlockId::Latest` to a concrete block number before spawning either the prover or the blocking check. Specifically, resolve "latest" to a `BlockId::Number(n)` once at the entry of `prove_transaction` (or at the entry of `prove_with_blocking_check`) and pass that concrete identifier to both sides. This ensures the blocking check and the prover always operate on identical chain state, eliminating the temporal divergence.

---

### Proof of Concept

1. Attacker submits `starknet_proveTransaction(block_id: "latest", transaction: T)` to the prover RPC endpoint.
2. `prove_transaction` passes `BlockId::Latest` to `prove_with_blocking_check`.
3. `prove_with_blocking_check` spawns `run_and_prove(BlockId::Latest, ...)` in a background task (line 236–237). The prover task begins fetching the block header; at this moment the latest block is N.
4. A new block N+1 is finalized on-chain before the blocking check HTTP request is dispatched.
5. `client.check_transaction(BlockId::Latest, T)` is sent to the external service (line 240–242). The external service resolves "latest" to N+1 and returns `Allowed` (transaction T is compliant at block N+1).
6. The background prover task, which started earlier, already resolved "latest" to N and generates a proof for block N. At block N, transaction T would have been flagged and blocked by the external service.
7. `prove_handle.await` returns a `ProveTransactionResult` whose `proof_facts` contain `block_number = N` and `block_hash = hash(N)`.
8. The attacker receives a valid proof for block N — a block the blocking check never evaluated — bypassing the security gate.
