### Title
Token-2022 PermanentDelegate Allows Post-`init_transfer` Vault Drain Creating Unbacked NEAR Supply — (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`, `log_metadata.rs`)

---

### Summary

The Solana bridge program accepts Token-2022 mints without checking for the `PermanentDelegate` extension. Because Token-2022's `PermanentDelegate` allows its holder to transfer tokens out of **any** token account for that mint — including PDA-owned vaults — an attacker can lock tokens in the vault via `init_transfer`, have the Wormhole message posted and settled on NEAR, then use the permanent delegate authority to drain the vault. The same tokens can be recycled into a second `init_transfer`, causing NEAR to settle again. This creates unbacked supply on NEAR with no corresponding collateral on Solana.

---

### Finding Description

**Token registration (`log_metadata`)** creates the vault PDA and posts metadata to NEAR. It only inspects the `MetadataPointer` extension to extract name/symbol; it performs no check for `PermanentDelegate`: [1](#0-0) 

The only mint-level guard is that the mint authority must not already be the bridge's authority PDA: [2](#0-1) 

**`init_transfer` (native path)** moves tokens from the user's account into the vault PDA and then posts the Wormhole message. The vault's token-account authority is the bridge's `authority` PDA: [3](#0-2) 

The `transfer_checked` CPI locks the tokens: [4](#0-3) 

The Wormhole message is posted immediately after, in the same transaction: [5](#0-4) 

**The gap**: Token-2022's `PermanentDelegate` extension grants its holder the ability to call `transfer_checked` (or `burn`) on the Token-2022 program for **any** token account of that mint, regardless of the token account's stored authority. The bridge's vault authority being a PDA provides no protection — the Token-2022 runtime honours the permanent delegate unconditionally. There is no code anywhere in `log_metadata`, `init_transfer`, or the account validation structs that reads or rejects the `PermanentDelegate` extension.

The `SECURITY.md` acknowledges Token-2022 extension awareness but only documents `TransferHook` as a known non-issue (denial, not fund loss); `PermanentDelegate` is not mentioned: [6](#0-5) 

---

### Impact Explanation

After `init_transfer` completes:
1. The Wormhole VAA is finalized and NEAR settles the transfer (mints or releases tokens to the recipient on NEAR).
2. The attacker invokes `transfer_checked` directly on the Token-2022 program using the permanent delegate authority, draining the vault back to their own account.
3. The attacker calls `init_transfer` again with the same tokens.
4. NEAR settles again.

Each cycle produces one unit of NEAR-side supply backed by zero Solana collateral. The bridge's documented invariant — *"A Wormhole message and its corresponding token lock/burn must occur in the same transaction … never change token state without posting the message"* — is satisfied at lock time, but the post-lock drain breaks the collateralization invariant permanently. [7](#0-6) 

For a legitimate Token-2022 token (e.g., a compliance-regulated asset) that has `PermanentDelegate` set to a third party, that third party can drain the vault and create unbacked supply without the bridge user's knowledge or consent.

---

### Likelihood Explanation

- **Attacker-created token path**: Fully self-contained. The attacker creates the mint, sets `PermanentDelegate` to themselves, calls `log_metadata` (permissionless), mints tokens, calls `init_transfer`, drains vault, repeats. No privileged access to the bridge is required.
- **Legitimate token path**: Any Token-2022 token with `PermanentDelegate` set (a real-world extension used for regulatory compliance and custodial products) is vulnerable the moment it is registered with the bridge. The permanent delegate holder can exploit this at any time.
- The Token-2022 `PermanentDelegate` extension is a documented, production-deployed feature of the SPL Token-2022 program. No external dependency compromise is required.

---

### Recommendation

In `log_metadata` (and optionally in `init_transfer` as a defence-in-depth check), unpack the Token-2022 mint extensions and reject any mint that has `PermanentDelegate` set to a non-zero, non-authority address:

```rust
// In log_metadata::process(), after unpacking mint_with_extension:
use spl_token_2022::extension::permanent_delegate::PermanentDelegate;
if let Ok(pd) = mint_with_extension.get_extension::<PermanentDelegate>() {
    require!(
        pd.delegate.0 == Pubkey::default(),
        ErrorCode::UnsupportedMintExtension
    );
}
```

This check belongs in `log_metadata.rs` at vault-creation time so that no vault is ever created for a mint whose collateral can be externally drained. [8](#0-7) 

---

### Proof of Concept

```
1. Create Token-2022 mint M with:
   - mint_authority = attacker keypair A
   - permanent_delegate = attacker keypair A

2. Call log_metadata(M) → vault PDA V is created, NEAR registers M.

3. Mint 1_000_000 tokens of M to attacker ATA T_A (using mint authority A).

4. Call init_transfer(from=T_A, vault=V, amount=1_000_000, recipient=<NEAR addr>)
   → vault V now holds 1_000_000 tokens
   → Wormhole VAA posted; NEAR settles, minting 1_000_000 NEAR-side tokens.

5. Directly invoke Token-2022 transfer_checked(
       from=V, to=T_A, authority=A [as permanent delegate], amount=1_000_000
   )
   → vault V is now empty; attacker has 1_000_000 tokens back.

6. Assert: vault balance = 0, NEAR-side supply = 1_000_000 (unbacked).

7. Repeat steps 4–6 N times → N × 1_000_000 unbacked NEAR-side tokens.
```

**File:** solana/SECURITY.md (L13-19)
```markdown
## Known Issues

Low-severity items acknowledged but not yet addressed:

- **No validation of `recipient` string in `InitTransferPayload`** — An invalid recipient causes the transfer to fail on the NEAR side after tokens are locked/burned on Solana. Manual intervention would be needed.
- **No validation of `fee_recipient` length in `FinalizeTransferPayload`** — Excessively large strings increase Wormhole message size. Bounded by Solana tx size limits in practice.
- **Token-2022 tokens with transfer hooks are not supported** — Transfer hook extra account metas are not included in instruction account sets. Affected tokens will fail at runtime (denial, not fund loss).
```

- **Atomic Wormhole posting**: A Wormhole message and its corresponding token lock/burn must occur in the same transaction. Never post a message without the token state change, and never change token state without posting the message — either direction is a bridge invariant violation
- **Malleable signature protection**: The program rejects signatures where `s` is high (`signature.s.is_high()`) to prevent signature malleability
