### Title
Token-2022 Metadata Pointer Allows Cross-Mint Identity Hijacking via Missing `metadata.mint` Validation — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

`LogMetadata::parse_metadata_account` validates that the caller-supplied metadata account address matches the address stored in the mint's `MetadataPointer` extension, and that the account is owned by `MetaplexID`. It does **not** validate that the deserialized `MplMetadata.mint` field equals `self.mint.key()`. An unprivileged attacker can create a Token-2022 mint whose `metadata_pointer` references the Metaplex PDA of any legitimate token, then call `log_metadata` to post a Wormhole payload that binds the attacker's mint address to the legitimate token's name, symbol, and decimals on NEAR.

---

### Finding Description

The vulnerable path is in `process()` for Token-2022 mints: [1](#0-0) 

When `metadata_pointer.metadata_address.0` is neither the mint itself nor `Pubkey::default()`, the code calls `parse_metadata_account` with the pointer address as the expected key: [2](#0-1) 

`parse_metadata_account` performs two checks:
1. `require_keys_eq!(metadata.key(), address)` — the passed account's address matches the pointer value.
2. `metadata.owner == &MetaplexID` — the account is owned by Metaplex.

After both checks pass, it deserializes and returns `metadata.name` and `metadata.symbol` **without ever checking `metadata.mint == self.mint.key()`**.

The `MplMetadata` struct contains a `mint: Pubkey` field that identifies which mint the metadata record belongs to. This field is set by Metaplex at creation time and cannot be forged. The code ignores it entirely.

**Attack steps (all permissionless):**

1. Attacker creates a Token-2022 mint `A` with a `MetadataPointer` extension whose `metadata_address` is set to `USDC_metaplex_PDA` (the canonical Metaplex PDA for USDC: `find_program_address(["metadata", MetaplexID, USDC_mint], MetaplexID)`).
2. Attacker calls `log_metadata` passing:
   - `mint = A`
   - `metadata = USDC_metaplex_PDA` (the real, on-chain Metaplex account for USDC)
   - `token_program = token_2022::ID`
3. Inside `process()`: the pointer address equals `USDC_metaplex_PDA`, so `parse_metadata_account(USDC_metaplex_PDA)` is called.
4. Inside `parse_metadata_account`: key equality passes (attacker supplied the correct account), owner check passes (Metaplex owns it), deserialization succeeds, returns `("USD Coin", "USDC")`.
5. The payload constructed is: [3](#0-2) 

```
LogMetadataPayload {
    token: A,           // attacker's mint
    name: "USD Coin",   // USDC's name
    symbol: "USDC",     // USDC's symbol
    decimals: X,        // attacker's mint decimals
}
```

6. This is posted via Wormhole. NEAR receives a `LogMetadataMessage` mapping attacker mint `A` → identity of USDC. [4](#0-3) 

---

### Impact Explanation

NEAR registers attacker mint `A` as a token with USDC's name and symbol. From this registration, the attacker can trigger `deploy_token` on EVM chains to deploy a wrapped ERC-20 that presents itself as "USD Coin / USDC" but is backed by the attacker's worthless Token-2022 mint. Users who receive this wrapped token and attempt to redeem it on Solana receive the attacker's mint tokens, not USDC. This is token-mapping corruption that breaks bridge collateralization: the canonical asset identity (name/symbol/decimals) is decoupled from the actual backing asset.

---

### Likelihood Explanation

The attack requires only:
- Creating a Token-2022 mint (permissionless, costs ~0.01 SOL rent)
- Setting a `MetadataPointer` extension to any existing Metaplex PDA (permissionless at mint creation)
- Calling `log_metadata` (public, no privileged role required)

The legitimate USDC Metaplex PDA is a well-known, permanently on-chain account. No key compromise, oracle manipulation, or privileged access is needed.

---

### Recommendation

After deserializing the `MplMetadata` account in `parse_metadata_account`, add a check that the metadata record's `mint` field matches the mint being registered:

```rust
if metadata.owner == &MetaplexID {
    let data = metadata.try_borrow_data()?;
    let metadata = MplMetadata::try_deserialize(&mut data.as_ref())?;
    // ADD THIS CHECK:
    require_keys_eq!(
        metadata.mint,
        self.mint.key(),
        ErrorCode::InvalidTokenMetadataAddress,
    );
    Ok((metadata.name.clone(), metadata.symbol.clone()))
}
```

This ensures the Metaplex metadata record was created for the specific mint being logged, not for any other token.

```rust
// Create Token-2022 mint with MetadataPointer { metadata_address: usdc_metaplex_pda }

call_log_metadata(attacker_mint, Some(usdc_metaplex_pda));

// 3. Assert: Wormhole payload contains
//    token=attacker_mint, name="USD Coin", symbol="USDC"
// NEAR now maps attacker_mint → USDC identity
```

The `parse_metadata_account` function at line 83–86 will deserialize the real USDC Metaplex account and return its name/symbol without ever checking `metadata.mint == attacker_mint`. [5](#0-4)
