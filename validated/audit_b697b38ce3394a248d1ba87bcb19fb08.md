### Title
Asymmetric `within_margin` check rejects valid proposals when L1 gas price rises between proposer and validator oracle reads - (File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs)

### Summary
The `within_margin` function used to validate the proposed L1 gas price computes the percentage tolerance relative to the **proposed** price (`number1`), not the locally-computed reference price. This makes the check asymmetric: when the proposer's oracle read is lower than the validator's oracle read (i.e., the price rose between the two reads), the effective absolute tolerance is smaller than intended, causing valid proposals to be incorrectly rejected and blocking block production.

### Finding Description
During proposal validation, `is_block_info_valid` fetches the local L1 gas price and compares it to the price embedded in the incoming `ProposalInit` using `within_margin`:

```rust
// validate_proposal.rs line 302-307
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
    && within_margin(
        l1_data_gas_price_fri_proposed,
        l1_data_gas_price_fri,
        l1_gas_price_margin_percent,
    ))
```

`within_margin` is defined as:

```rust
// validate_proposal.rs line 323-333
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;   // ← margin relative to number1 (proposed)
    number1.0.abs_diff(number2.0) <= margin
}
```

`number1` is the **proposed** price and `number2` is the **locally computed** price. The absolute tolerance is therefore `proposed * margin_percent / 100`, not `reference * margin_percent / 100`.

With `l1_gas_price_margin_percent = 10` (from `orchestrator_versioned_constants_0_14_2.json`):

| Proposed | Locally computed | Intended margin (10% of ref) | Actual margin (10% of proposed) | Result |
|---|---|---|---|---|
| 100 | 90 | 9 | 10 | PASS |
| 90 | 100 | 10 | 9 | **FAIL** (abs_diff=10 > 9) |

A proposer who read the oracle slightly earlier (when the price was lower) will have their proposal rejected even though the price difference is within the intended 10% window. The `GAS_PRICE_ABS_DIFF_MARGIN = 1` bypass only helps when the absolute difference is ≤ 1 unit, which is irrelevant for real-world gas prices in the range of billions of fri.

The root cause is that `number1` (the untrusted proposed value) is used as the base for the margin calculation instead of `number2` (the locally trusted reference value).

### Impact Explanation
A validator node will reject a valid proposal whenever the proposer's oracle read is lower than the validator's oracle read by more than `proposed_price * 10 / 100`. During any period of rising L1 gas prices, this condition is routinely met. The rejected proposal causes the consensus round to fail; the network must wait for a timeout and retry with a new proposer. Repeated rejections across multiple validators stall block production entirely. This maps to: **High — valid transactions/proposals rejected before sequencing**.

### Likelihood Explanation
L1 gas prices fluctuate continuously. The proposer and validator fetch prices at different wall-clock times (the proposer fetches during `build_proposal`, the validator fetches during `validate_proposal`). Any upward price movement between those two fetches that exceeds `proposed_price * 10 / 100` triggers the bug. During Ethereum congestion spikes (common during NFT mints, token launches, etc.) prices can move 10–20% within seconds, making this a realistic and recurring condition.

### Recommendation
Compute the margin relative to the locally-trusted reference price (`number2`), not the untrusted proposed price (`number1`):

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number2.0 * margin_percent) / 100;  // use reference price as base
    number1.0.abs_diff(number2.0) <= margin
}
```

Alternatively, use `max(number1.0, number2.0)` as the base to make the check symmetric in both directions.

### Proof of Concept
The existing unit test `test_within_margin` in `validate_proposal_test.rs` already demonstrates the asymmetry:

```rust
// validate_proposal_test.rs line 342-354
#[case::small_number_in_margin(9, 10, 10, true)]   // proposed=9, ref=10 → margin=0 → abs_diff=1 ≤ GAS_PRICE_ABS_DIFF_MARGIN=1 → PASS (only saved by the abs floor)
#[case::small_number_out_of_margin(9, 11, 10, false)] // proposed=9, ref=11 → margin=0 → abs_diff=2 > 1 → FAIL
```

For production-scale prices, the asymmetry is directly observable:

```
// l1_gas_price_margin_percent = 10 (from orchestrator_versioned_constants_0_14_2.json)
// Scenario: L1 gas price rises from 90 to 100 fri between proposer and validator oracle reads.

proposed  = GasPrice(90_000_000_000)   // proposer read earlier
reference = GasPrice(100_000_000_000)  // validator read later (price rose ~11%)

margin    = 90_000_000_000 * 10 / 100 = 9_000_000_000
abs_diff  = 10_000_000_000

10_000_000_000 <= 9_000_000_000  →  false  →  proposal REJECTED

// Reverse scenario: price fell from 100 to 90 (same absolute difference)
proposed  = GasPrice(100_000_000_000)
reference = GasPrice(90_000_000_000)

margin    = 100_000_000_000 * 10 / 100 = 10_000_000_000
abs_diff  = 10_000_000_000

10_000_000_000 <= 10_000_000_000  →  true  →  proposal ACCEPTED
```

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    // For small numbers (e.g., less than 10 wei, if margin is 10%), even an off-by-one
    // error might be bigger than the margin, even if it is just a rounding error.
    // We make an exception for such mismatch, and don't bother checking percentages
    // if the difference in price is only one wei.
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}
```
