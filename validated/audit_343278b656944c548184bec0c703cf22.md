### Title
Missing Duplicate-Key/Hash Validation in `MerkleBlob::batch_insert` Bulk Path — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` has a split code path: the first two items are routed through the validated `insert()` method, but all subsequent items bypass the duplicate-key and duplicate-hash guards and are written directly into the blob. An attacker-controlled batch containing duplicate `KeyId` or `Hash` values (beyond the first two) silently corrupts the Merkle tree, producing an invalid root hash and enabling forged inclusion proofs.

### Finding Description

`MerkleBlob::insert` enforces two invariants before writing a leaf:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `insert` (with full validation) only for the first two items when the tree has ≤ 1 existing leaf:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { return Ok(()); };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;
    }
}
``` [2](#0-1) 

All remaining items are written directly via `insert_entry_to_blob` with **no duplicate-key or duplicate-hash check**:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [3](#0-2) 

The Python binding `py_batch_insert` only checks that the lengths of the two input lists match; it performs no deduplication:

```rust
if keys_values.len() != hashes.len() {
    Err(Error::UnmatchedKeysAndValues(...))?;
}
self.batch_insert(zip(keys_values, hashes).collect())?;
``` [4](#0-3) 

The Python-facing type stub confirms `batch_insert` is a public API:

```python
def batch_insert(self, keys_values: list[tuple[KeyId, ValueId]], hashes: list[bytes32]): ...
``` [5](#0-4) 

### Impact Explanation

When a batch of ≥ 3 items contains a duplicate `KeyId` or `Hash`, two leaf nodes with the same key (or hash) are written into the blob. The `block_status_cache` maps each key to exactly one index, so the second duplicate leaf is orphaned from the cache. The resulting tree has:

- An incorrect root hash (it commits to both duplicate leaves).
- A `block_status_cache` that is inconsistent with the blob (integrity check would fail or panic).
- `get_proof_of_inclusion` returning a proof anchored to the wrong root, enabling a caller to present a proof that appears valid against a root that was computed over corrupted state.

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state.** [6](#0-5) 

### Likelihood Explanation

`batch_insert` is the primary bulk-insertion API exposed through the Python wheel. Any caller that constructs a batch with repeated keys or hashes — whether by mistake or deliberately — triggers the corruption. The `insert` path correctly rejects duplicates, so the discrepancy is non-obvious and unlikely to be caught by callers who assume `batch_insert` has the same invariants as `insert`.

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the bulk path inside `batch_insert`, before writing each leaf with `insert_entry_to_blob`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insert_entry_to_blob logic
}
```

Alternatively, pre-validate the entire input vector for duplicates before entering the fast path.

### Proof of Concept

```rust
let mut blob = MerkleBlob::new(Vec::new()).unwrap();

// Pre-populate so leaf_count > 1 (bypasses the validated path for items 3+)
blob.insert(KeyId(0), ValueId(0), &hash_a, InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(1), ValueId(1), &hash_b, InsertLocation::Auto {}).unwrap();

// batch with a duplicate key (KeyId(2) appears twice)
let batch = vec![
    ((KeyId(2), ValueId(2)), hash_c),
    ((KeyId(3), ValueId(3)), hash_d),
    ((KeyId(2), ValueId(99)), hash_e), // duplicate key — bypasses validation
];
blob.batch_insert(batch).unwrap(); // succeeds silently

// Tree now contains two leaves with KeyId(2); root hash is corrupted.
// blob.check_integrity() will fail or produce inconsistent results.

    pub fn batch_insert(
        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {
        // OPT: perhaps go back to taking an iterator?
        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),
                };

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {
            // OPT: can we avoid this extra min height leaf traversal?
            let min_height_leaf = self.get_min_height_leaf()?;
            self.insert_subtree_at_key(min_height_leaf.key, indexes[0], Side::Left)?;
        }

        Ok(())
    }
