### Title
Unauthenticated BOOTSTRAP Declare Bypass Allows Unauthorized Class Declaration Without Signature Verification — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_declare_transaction` in `transaction_impls.cairo` contains a special-case bypass that skips `__validate_declare__` (signature verification), nonce increment, and fee charging for any declare transaction whose `sender_address` equals the felt literal `'BOOTSTRAP'`, `nonce == 0`, `version == 3`, and `max_possible_fee == 0`. Because `sender_address` is a user-supplied field in a declare transaction, any unprivileged class declarer can craft such a transaction and have the OS accept it without any cryptographic proof of authorization.

---

### Finding Description

In `execute_declare_transaction`, after computing the transaction hash and filling `tx_info`, the OS checks:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

When this branch is taken, the function returns immediately — **before** `check_and_increment_nonce`, **before** `run_validate` / `non_reverting_select_execute_entry_point_func` (which calls `__validate_declare__`), and **before** `charge_fee`. The only constraints are:

1. `sender_address` equals the felt encoding of the ASCII string `'BOOTSTRAP'` — a fixed, publicly known felt value.
2. `nonce == 0` — trivially satisfiable for any fresh or non-existent account address.
3. `version == 3` — the standard current version.
4. All resource-bound `max_amount * max_price` products equal zero — trivially satisfiable by setting all bounds to zero.

The `sender_address` field in a declare transaction is entirely user-controlled. There is no cryptographic check that the submitter owns or controls the address `'BOOTSTRAP'`. The OS accepts the transaction and writes `compiled_class_hash` into `contract_class_changes` without any signature.

The class hash itself is validated via `finalize_class_hash` (the Sierra component hash check), so the attacker must supply a structurally valid Sierra class. However, the `compiled_class_hash` (the CASM hash) is accepted as any non-zero felt — the attacker can pair a valid Sierra class hash with an arbitrary, attacker-chosen compiled class hash. [2](#0-1) 

For comparison, the normal declare path (lines 778–827) always calls `check_and_increment_nonce` and `non_reverting_select_execute_entry_point_func` with `VALIDATE_DECLARE_ENTRY_POINT_SELECTOR`, enforcing that the account contract's `__validate_declare__` returns `VALIDATED`. [3](#0-2) 

The `run_validate` function, which is bypassed, is explicitly documented as "responsible for signature verification": [4](#0-3) 

---

### Impact Explanation

**Impact: Direct loss of funds (Critical).**

An attacker can execute a **front-running class-poisoning attack**:

1. Monitor the mempool for a legitimate declare transaction for a class hash `H` (e.g., a new account contract class, a token contract class, or any widely-used class).
2. Submit a BOOTSTRAP-bypass declare transaction for the same class hash `H`, pairing it with a malicious `compiled_class_hash` `M` that points to attacker-controlled CASM (e.g., CASM that drains storage or redirects transfers).
3. Because `dict_update` enforces `prev_value=0` (the class can only be declared once), the legitimate declaration is now permanently blocked — the class hash `H` is irrevocably bound to the malicious compiled class `M`.
4. Any user who subsequently deploys a contract with class hash `H` (believing it to be the legitimate class) will instead run the malicious CASM. Funds deposited into those contracts can be stolen.

Additionally, if the poisoned class hash is a system-critical class (fee token, account contract template), the impact escalates to a network-wide disruption.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The conditions (`sender_address = felt('BOOTSTRAP')`, `nonce = 0`, `version = 3`, zero fees) are all trivially constructable by any user — no privileged key or role is required.
- The OS is the protocol-level source of truth. Since the OS accepts this transaction, a sequencer following the protocol must also accept it.
- The attack requires the attacker to observe a pending class declaration and front-run it, which is feasible in a public mempool environment.
- The only practical barrier is whether the sequencer's off-chain mempool validation independently rejects `sender_address = 'BOOTSTRAP'` transactions, but this is not enforced at the OS/protocol level and is therefore not a reliable protection.

---

### Recommendation

Remove the unauthenticated BOOTSTRAP bypass entirely, or gate it behind a cryptographic proof of authorization (e.g., a multi-sig from a set of trusted bootstrap keys whose public keys are committed in the OS configuration). If bootstrapping is a one-time operation, it should be handled outside the normal transaction flow (e.g., as a genesis-state initialization step that is not reachable via user-submitted transactions), not as a special case inside `execute_declare_transaction` that any user can trigger.

---

### Proof of Concept

**Attacker-controlled inputs to the OS (via a declare transaction):**

```
sender_address  = 0x424F4F545354524150   // felt('BOOTSTRAP')
nonce           = 0
version         = 3
tip             = 0
resource_bounds = [{resource=L1_GAS, max_amount=0, max_price=0},
                   {resource=L2_GAS, max_amount=0, max_price=0},
                   {resource=L1_DATA_GAS, max_amount=0, max_price=0}]
class_hash      = H   // any valid Sierra class hash (attacker constructs a valid Sierra class)
compiled_class_hash = M  // attacker's malicious CASM hash
```

**OS execution path:**

1. `execute_declare_transaction` is called.
2. `tx_version != 0`, so the early-return is not taken.
3. `get_account_tx_common_fields` loads the attacker's fields.
4. `compute_declare_transaction_hash` computes a valid hash (no signature checked here).
5. `finalize_class_hash` validates the Sierra class structure — passes because the attacker supplied a valid Sierra class.
6. `fill_account_tx_info` fills `tx_info` with `signature_start = signature_end = 0` (empty signature).
7. The BOOTSTRAP condition at line 764 evaluates to TRUE.
8. `compute_max_possible_fee` returns 0 (all bounds are zero).
9. `dict_update` writes `contract_class_changes[H] = M` — **no signature verification performed**.
10. The function returns. Class hash `H` is now permanently bound to malicious compiled class `M` in the proven state. [1](#0-0)

// Runs the account contract's "__validate__" entry point, which is responsible for
// signature verification.
//
// Arguments:
// block_context - a global context that is fixed throughout the block.
// tx_execution_context - The execution context of the underlying invoke transaction.
func run_validate{
    range_check_ptr,
    remaining_gas: felt,
    builtin_ptrs: BuiltinPointers*,
    contract_state_changes: DictAccess*,
    contract_class_changes: DictAccess*,
    outputs: OsCarriedOutputs*,
}(block_context: BlockContext*, tx_execution_context: ExecutionContext*) {
    alloc_locals;
    local tx_execution_info: ExecutionInfo* = tx_execution_context.execution_info;

    // Do not run "__validate__" for version 0.
    if (tx_execution_info.tx_info.version == 0) {
        return ();
    }

    // "__validate__" is expected to get the same calldata as "__execute__".
    local validate_execution_context: ExecutionContext* = new ExecutionContext(
        entry_point_type=ENTRY_POINT_TYPE_EXTERNAL,
        class_hash=tx_execution_context.class_hash,
        calldata_size=tx_execution_context.calldata_size,
        calldata=tx_execution_context.calldata,
        execution_info=new ExecutionInfo(
            block_info=block_context.block_info_for_validate,
            tx_info=tx_execution_info.tx_info,
            caller_address=tx_execution_info.caller_address,
            contract_address=tx_execution_info.contract_address,
            selector=VALIDATE_ENTRY_POINT_SELECTOR,
        ),
        deprecated_tx_info=tx_execution_context.deprecated_tx_info,
    );

    // The __validate__ function should not revert.
    let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func(
        block_context=block_context, execution_context=validate_execution_context
    );
    if (is_deprecated == 0) {
        %{ CheckRetdataForDebug %}
        assert retdata_size = 1;
        assert retdata[0] = VALIDATED;
    }

    return ();
}
