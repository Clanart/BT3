### Title
`From<CommitmentStateDiff> for ThinStateDiff` silently drops `deprecated_declared_classes`, producing a wrong `state_diff_commitment` and wrong `concatenated_counts` in every block that contains a deprecated Declare transaction — (`crates/blockifier/src/state/cached_state.rs`)

---

### Summary

`CommitmentStateDiff` has no `deprecated_declared_classes` field. The `From<CommitmentStateDiff> for ThinStateDiff` conversion hard-codes `deprecated_declared_classes: Vec::new()`. `BlockExecutionArtifacts::new()` feeds this truncated `ThinStateDiff` into `calculate_block_commitments`, which uses it for both `calculate_state_diff_hash` (the `state_diff_commitment`) and `concat_counts` (the `state_diff_length` packed into `concatenated_counts`). Both values are therefore wrong whenever a block contains a deprecated Declare (v0/v1) transaction, producing a wrong `PartialBlockHashComponents` and ultimately a wrong block hash. The P2P sync client additionally uses the stored `state_diff_length` as a hard termination counter, so it will either truncate or reject the correct state diff received from a peer.

---

### Finding Description

**Step 1 – Root cause: the conversion drops deprecated declared classes**

`CommitmentStateDiff` carries only `address_to_class_hash`, `address_to_nonce`, `storage_updates`, and `class_hash_to_compiled_class_hash`. The `From` impl that converts it to `ThinStateDiff` unconditionally writes an empty vector:

```rust
// crates/blockifier/src/state/cached_state.rs  line 756-767
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            ...
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),   // ← always empty
        }
    }
}
``` [1](#0-0) 

**Step 2 – The truncated diff is the sole input to commitment computation**

`BlockExecutionArtifacts::new()` converts `commitment_state_diff` to `ThinStateDiff` and passes it directly to `calculate_block_commitments`:

```rust

    ...

**Step 4 – P2P sync client uses the wrong `state_diff_length` as a hard termination counter**

The P2P sync client reads `state_diff_length` from the stored block header and uses it as `target_state_diff_len`:

```rust
// crates/apollo_p2p_sync/src/client/state_diff.rs  line 62-103
let target_state_diff_len = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    ...
    .state_diff_length ...?;

while current_state_diff_len < target_state_diff_len { ... }

if current_state_diff_len != target_state_diff_len {
    return Err(ParseDataError::BadPeer(BadPeerError::WrongStateDiffLength { ... }));
}
``` [7](#0-6) 

If the stored `state_diff_length` is N (without deprecated declared classes) but the peer serves N+K chunks (the correct full diff), the client exits the loop at N, silently omitting K deprecated-class entries, and then rejects the peer's Fin as `WrongStateDiffLength`.

---

### Impact Explanation

A block containing one or more deprecated Declare (v0/v1) transactions will have:

- A `state_diff_commitment` that does not cover the deprecated declared classes — the Poseidon hash is computed over an empty list instead of the actual list.
- A `concatenated_counts` field (packed into the block hash) whose `state_diff_length` bits are too small by exactly the number of deprecated declared classes.
- A final block hash that is therefore wrong.

Any downstream consumer that independently recomputes the block hash (proof system, L1 verifier, another sequencer node) will derive a different value and reject the block or the proof. The P2P sync client will additionally either silently drop deprecated declared class entries from the synced state or ban the honest peer that served the correct diff.

This matches: **Critical — Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input** (wrong `state_diff_commitment` and block hash committed for an accepted block) and **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value** (wrong `state_diff_commitment` returned by `starknet_getStateUpdate`).

---

### Likelihood Explanation

Deprecated Declare transactions (transaction version 0 and 1, declaring Cairo 0 contracts) are still valid Starknet transactions. Any user can submit one. The sequencer's gateway does not reject them on the basis of being deprecated. The bug fires on the very first such transaction included in a sequenced block, with no special timing or race condition required.

---

### Recommendation

Add `deprecated_declared_classes` to `CommitmentStateDiff` and populate it from `StateMaps.declared_contracts` (filtering for the `false`/deprecated entries) in `CommitmentStateDiff::from(StateMaps)`. Remove the `Vec::new()` hard-code in `From<CommitmentStateDiff> for ThinStateDiff` and propagate the field. This ensures `calculate_state_diff_hash` and `concat_counts` both operate on the complete diff, producing a correct `state_diff_commitment`, correct `concatenated_counts`, and correct block hash.

---

### Proof of Concept

1. Submit a deprecated Declare transaction (v0 or v1) to the sequencer gateway. The gateway accepts it.
2. The transaction is included in a block. `BlockExecutionArtifacts::new()` is called.
3. `ThinStateDiff::from(commitment_state_diff.clone())` produces a `ThinStateDiff` with `deprecated_declared_classes = []`.
4. `calculate_state_diff_hash` hashes `[0]` (length 0, no elements) for the deprecated-classes segment instead of `[1, <class_hash>]`.
5. `concat_counts` packs `state_diff_length = N` instead of `N + 1`.
6. The resulting `state_diff_commitment` and `concatenated_counts` differ from the values any independent verifier (SNOS, L1 verifier, another node) would compute from the full state diff.
7. A validator node that independently recomputes the block hash from the full state diff (including the deprecated declared class) obtains a different hash and rejects the proposal.
8. A P2P sync peer that serves the correct full state diff (N+1 chunks) is banned by the client because `current_state_diff_len (N+1) != target_state_diff_len (N)`.
