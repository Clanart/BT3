### Title
Incorrect Field Ordering in `compute_deploy_account_transaction_hash` Causes All Deploy Account v3 Transactions to Fail Signature Verification — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_deploy_account_transaction_hash` function in the StarkNet OS hashes the constructor calldata **before** the class hash and contract address salt, which is the reverse of the order mandated by the StarkNet transaction hash specification (SNIP-8). Because the OS passes this incorrectly-ordered hash to the account contract's `__validate__` entry point, every deploy account v3 transaction will fail signature verification — permanently blocking new account deployments and freezing funds held in counterfactual (pre-deployment) accounts.

---

### Finding Description

In `compute_deploy_account_transaction_hash`, the calldata array is laid out as:

```
calldata[0] = class_hash
calldata[1] = contract_address_salt
calldata[2..] = constructor_calldata
```

The function builds the Poseidon hash state as follows:

```cairo
with hash_state {
    hash_tx_common_fields(common_fields=common_fields);
    // Hash and add the constructor calldata to the hash state.
    poseidon_hash_update_with_nested_hash(data_ptr=&calldata[2], data_length=calldata_size - 2);
    // Add the class hash and the contract address salt to the hash state.
    poseidon_hash_update(data_ptr=calldata, data_length=2);
}
``` [1](#0-0) 

This produces the hash over the sequence:

```
common_fields … | H(constructor_calldata) | class_hash | salt
```

The StarkNet specification (SNIP-8) requires the deploy account v3 hash to be computed over:

```
common_fields … | class_hash | salt | H(constructor_calldata)
```

The `class_hash` and `salt` must appear **before** `H(constructor_calldata)`, not after. The code has the two blocks in the wrong order.

This is directly analogous to the external report's root cause: a field that must be included in a hash/slot derivation is effectively displaced by incorrect ordering, producing a commitment that does not match what the off-chain signer computed.

For comparison, `compute_declare_transaction_hash` correctly places the class-related fields after the common fields and before any nested data hash:

```cairo
poseidon_hash_update_single(item=class_hash);
poseidon_hash_update_single(item=compiled_class_hash);
``` [2](#0-1) 

No analogous ordering inversion exists there.

---

### Impact Explanation

The OS-computed transaction hash is the value passed to the account contract's `__validate__` entry point. Wallets and clients sign the hash computed according to the specification (class_hash → salt → H(constructor_calldata)). Because the OS computes a structurally different hash, the signature presented by any conforming client will never verify against the OS-produced hash.

Concrete consequences:

1. **Every deploy account v3 transaction is rejected** at the `__validate__` step with a signature-verification failure.
2. **Funds held in counterfactual accounts** — accounts whose address was derived before deployment — cannot be recovered, because the deployment transaction can never be included in a block. This constitutes **permanent freezing of funds**.
3. The impact is universal: it applies to every user who attempts to deploy a new account via the v3 path, with no workaround available at the protocol level.

---

### Likelihood Explanation

- The entry path is fully unprivileged: any user submitting a deploy account v3 transaction triggers the vulnerable code path.
- Deploy account v3 is the current standard transaction version; all modern tooling (starknet.js, starknet.py, Argent, Braavos) generates v3 deploy account transactions.
- There is no operator or trusted-role gate between the user's transaction and this hash computation.
- The bug is deterministic: it fires on 100% of deploy account v3 transactions, not probabilistically.

---

### Recommendation

Swap the two `poseidon_hash_update` calls so that `class_hash` and `salt` are committed to the hash state **before** the nested hash of the constructor calldata, matching the specification:

```cairo
with hash_state {
    hash_tx_common_fields(common_fields=common_fields);
    // Add the class hash and the contract address salt to the hash state first.
    poseidon_hash_update(data_ptr=calldata, data_length=2);
    // Then hash and add the constructor calldata.
    poseidon_hash_update_with_nested_hash(data_ptr=&calldata[2], data_length=calldata_size - 2);
}
```

This aligns the OS hash computation with the SNIP-8 specification and with what conforming signers produce.

---

### Proof of Concept

Consider a deploy account v3 transaction with:
- `class_hash = C`
- `salt = S`
- `constructor_calldata = [A, B]`

**Client (spec-compliant) hash input sequence** (after common fields):
```
C, S, poseidon(A, B)
```

**OS (buggy) hash input sequence** (after common fields):
```
poseidon(A, B), C, S
```

These two sequences produce different Poseidon digests. The client signs over the first; the OS verifies against the second. Signature verification fails unconditionally.

A test analogous to the external report's PoC would be:

```python
# Compute hash as the spec requires
spec_hash = poseidon([
    *common_fields,
    class_hash,
    salt,
    poseidon(constructor_calldata),  # nested hash
])

# Compute hash as the OS computes it (wrong order)
os_hash = poseidon([
    *common_fields,
    poseidon(constructor_calldata),  # nested hash first
    class_hash,
    salt,
])

assert spec_hash != os_hash  # always true → signature always fails
```
