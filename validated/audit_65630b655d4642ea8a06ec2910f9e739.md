### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

`execute_replace_class` in `syscall_impls.cairo` updates a contract's class hash in `contract_state_changes` without verifying that the new class hash is present in `contract_class_changes` (i.e., that it has been declared). This is structurally identical to the C-02 pattern: a state-change operation modifies one mapping while leaving a dependent mapping unchecked, producing permanently inconsistent state. Any contract that calls `replace_class` with an undeclared hash will have its class hash set to an undeclared value in the global state tree. Because `execute_entry_point` unconditionally looks up the class hash in `contract_class_changes` and then calls `find_element` on the result, any subsequent execution of that contract will fail at the OS level, permanently freezing all funds held by the contract.

---

### Finding Description

`execute_replace_class` (`syscall_impls.cairo`, lines 877–916) accepts any arbitrary `class_hash` from the caller and writes it directly into `contract_state_changes`:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The function signature does not even receive `contract_class_changes` as an implicit argument, making it structurally impossible to perform the check. The TODO comment at line 898 explicitly acknowledges the missing validation. [2](#0-1) 

When any subsequent call to the contract is processed, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [3](#0-2) 

If `execution_context.class_hash` is not in `contract_class_changes`, `dict_read` returns `0`. `find_element` is then called with `key=0`. Because `find_element` is a non-optional search that panics when the key is absent, the OS Cairo program fails to execute for any block containing a call to the affected contract. ### Impact Explanation

**Critical. Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value, the OS cannot produce a valid proof for any block that includes a call to that contract. The sequencer must permanently exclude all transactions targeting the contract. Any ERC-20 tokens, ETH, or other assets held in the contract's storage become irrecoverable. The state change is committed to the global Merkle tree and cannot be undone without a protocol-level intervention.

---

### Likelihood Explanation

**High.** The attack requires only:
1. Deploying a contract (permissionless).
2. Calling that contract so it invokes `replace_class` with an arbitrary undeclared felt value.

No privileged role, leaked key, or external dependency is needed. The TODO comment at line 898 confirms the development team is aware the check is absent. Any contract author — including a malicious one — can trigger this path today.

---

### Recommendation

Add a validation step inside `execute_replace_class` that asserts the requested `class_hash` is present in `contract_class_changes` before writing it to `contract_state_changes`. The function must receive `contract_class_changes` as an implicit argument and perform a `dict_read` (or equivalent lookup) to confirm the class has been declared. The same fix must be applied to `execute_replace_class` in `deprecated_execute_syscalls.cairo`. This mirrors the recommendation in C-02: on a state-change operation, all dependent mappings must be validated or updated atomically.

---

### Proof of Concept

1. Attacker deploys contract `A` holding funds (e.g., an ERC-20 balance).
2. Attacker sends a transaction that calls contract `A`'s `__execute__` entry point.
3. Contract `A` internally calls the `replace_class` syscall with `class_hash = 0xdeadbeef` (any value not present in `contract_class_changes`).
4. `execute_replace_class` (`syscall_impls.cairo:906`) writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes` with no validation.
5. The transaction succeeds; the new class hash is committed to the global state tree.
6. In any subsequent block, a transaction calls contract `A`.
7. `execute_entry_point` (`execute_entry_point.cairo:154`) calls `dict_read{dict_ptr=contract_class_changes}(key=0xdeadbeef)` → returns `0`.
8. `find_element(..., key=0)` (`execute_entry_point.cairo:161`) panics — no compiled class fact with hash `0` exists.
9. The OS program fails; the sequencer cannot produce a valid proof for any block containing a call to contract `A`.
10. All funds in contract `A` are permanently frozen.
