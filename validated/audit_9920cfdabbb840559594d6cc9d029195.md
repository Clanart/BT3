### Title
User-Controlled Zero-Price Resource Bounds Bypass Fee Enforcement in `charge_fee` - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `charge_fee` function in the StarkNet OS unconditionally skips all fee collection when `compute_max_possible_fee` returns `0`. Because every component of that computation is derived from user-supplied resource-bound fields (`max_price_per_unit` for L1\_GAS, L2\_GAS, L1\_DATA\_GAS, and `tip`), and the OS imposes no minimum-price floor on any of them, an attacker can craft a V3 transaction that executes with a full L2-gas budget while paying zero fees. The OS proof remains valid, meaning the L1 verifier accepts the block even though no fee was collected.

---

### Finding Description

**`compute_max_possible_fee`** (lines 87–102) computes the ceiling fee entirely from user-controlled fields:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

**`charge_fee`** (lines 111–165) immediately returns — charging nothing — when that value is zero:

```cairo
if (max_fee == 0) {
    return ();
}
``` [2](#0-1) 

The only validation applied to `max_price_per_unit` in the entire OS is `assert_nn` (i.e., `>= 0`), which explicitly permits the value `0`:

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn(resource_bounds.max_price_per_unit);   // 0 is accepted
    ...
}
``` [3](#0-2) 

Similarly, `tip` is only bounded to `[0, 2^64-1]`, so `tip = 0` is valid: [4](#0-3) 

Meanwhile, `get_initial_user_gas_bound` grants the transaction a full L2-gas execution budget derived solely from `l2_gas_bounds.max_amount`, which is independent of price:

```cairo
func get_initial_user_gas_bound(common_tx_fields: CommonTxFields*) -> felt {
    assert common_tx_fields.n_resource_bounds = 3;
    return common_tx_fields.resource_bounds[L2_GAS_INDEX].max_amount;
}
``` [5](#0-4) 

This gas budget is used directly as `remaining_gas` for both validate and execute phases: [6](#0-5) 

The result is a structural decoupling: **a user can set `max_amount` to `EXECUTE_MAX_SIERRA_GAS` (1,100,000,000) while setting all prices and tip to `0`**, obtaining a full execution budget at zero cost. The OS produces a valid proof for this block.

---

### Impact Explanation

**Direct loss of funds (Critical):** The fee token (STRK) transfer to the sequencer is the protocol's mechanism for compensating block producers. When `max_fee == 0`, the ERC-20 `transfer` call to the sequencer address is never executed. Any block containing such transactions results in the sequencer receiving zero fee revenue for real computational work performed. Because the OS proof is valid, the L1 verifier accepts the block, making the loss permanent and unrecoverable on-chain.

**Network halt (High):** With zero-cost execution, an attacker can flood the network with transactions each consuming up to `EXECUTE_MAX_SIERRA_GAS = 1,100,000,000` L2 gas units at no financial cost. This can saturate block capacity indefinitely, preventing legitimate transactions from being confirmed — a total network shutdown.

---

### Likelihood Explanation

**High.** The attack requires no privileged access, no leaked keys, and no third-party compromise. Any account holder can sign and broadcast a V3 transaction with all `max_price_per_unit = 0` and `tip = 0`. The OS imposes no minimum-price floor; the only constraint is that the sequencer must include the transaction. Since the OS itself does not reject zero-price transactions, a sequencer that lacks an independent mempool filter (or a malicious sequencer) will include them and produce a valid proof. The attack is fully permissionless at the protocol layer.

---

### Recommendation

Enforce a minimum non-zero price floor inside the OS for at least the L2-gas resource bound. For example, add an assertion in `compute_max_possible_fee` or in `get_initial_user_gas_bound` that rejects transactions where `l2_gas_bounds.max_price_per_unit == 0` when `l2_gas_bounds.max_amount > 0`. Alternatively, require `max_fee > 0` whenever `initial_user_gas_bound > 0`, ensuring that any transaction that consumes execution gas must commit to paying a non-zero fee.

---

### Proof of Concept

1. Construct a valid V3 invoke transaction with:
   - `l1_gas_bounds = { resource: L1_GAS, max_amount: 0, max_price_per_unit: 0 }`
   - `l2_gas_bounds = { resource: L2_GAS, max_amount: 1_100_000_000, max_price_per_unit: 0 }`
   - `l1_data_gas_bounds = { resource: L1_DATA_GAS, max_amount: 0, max_price_per_unit: 0 }`
   - `tip = 0`

2. Sign and submit the transaction. The transaction hash commits to these zero-price bounds (via `hash_fee_fields`), so the signature is valid.

3. The sequencer includes the transaction. The OS executes:
   - `compute_max_possible_fee` → `0 * 0 + 1_100_000_000 * (0 + 0) + 0 * 0 = 0`
   - `charge_fee`: `max_fee == 0` → early return, no ERC-20 transfer executed
   - `get_initial_user_gas_bound` → `1_100_000_000` (full execute budget granted)
   - Transaction executes arbitrary logic up to `EXECUTE_MAX_SIERRA_GAS`

4. The OS proof is generated and verified on L1. No fee was paid; the proof is accepted.

5. Repeat with fresh nonces to spam the network at zero cost. [7](#0-6) [3](#0-2)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
