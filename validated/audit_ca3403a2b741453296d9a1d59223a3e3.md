Audit Report

## Title
Unaccounted Router ETH Balance Silently Subsidizes Subsequent WETH Swaps, Stealing Prior User's Refund — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay` uses `address(this).balance` as an unattributed global pool when `token == WETH`. ETH stranded on the router from a prior user's `msg.value` overpayment is silently consumed to fund a subsequent user's WETH swap, causing direct loss of the prior user's principal with no per-transaction accounting to prevent it.

## Finding Description

`exactInputSingle` is `payable` and accepts arbitrary `msg.value`. When a user sends `msg.value = Y` with `amountIn = X` where `Y > X`, the callback invokes `pay(WETH, user, pool, X)`. Inside `pay`: [1](#0-0) 

`nativeBalance = address(this).balance = Y`. Since `Y >= X`, the branch deposits exactly `X` ETH and transfers WETH to the pool. The remaining `Y - X` ETH is left on the router with no per-user or per-transaction accounting.

If the user does not call `refundETH()` in the same multicall, the residual ETH persists. A subsequent attacker calls `exactInputSingle(tokenIn=WETH, amountIn=X, msg.value=0)`. The callback fires: [2](#0-1) 

which calls `pay(WETH, attacker, pool, X)`. Now `nativeBalance = address(this).balance = Y - X` (the prior user's stranded ETH). If `Y - X >= X`, the branch at line 75 fires, depositing the prior user's ETH to fund the attacker's swap entirely. The attacker's `safeTransferFrom` is never reached.

The `receive()` guard: [3](#0-2) 

is irrelevant here — ETH sent via a `payable` function call is credited directly to `address(this).balance` without triggering `receive()`, so the guard provides no protection against overpayment stranding.

The partial-balance branch compounds the issue: [4](#0-3) 

Even if the stranded ETH is less than the attacker's `amountIn`, it is still consumed first, reducing the attacker's `transferFrom` obligation by the stolen amount.

## Impact Explanation

Direct loss of user principal. The prior user's overpayment ETH (`Y - X`) is consumed to fund an unrelated user's swap. The prior user receives no WETH and no ETH refund for that amount. This is a Critical/High direct-loss-of-user-principal impact: an unprivileged attacker can steal real ETH from the router with no privileged access required.

## Likelihood Explanation

Moderate. The precondition — ETH stranded on the router — arises naturally whenever a user calls `exactInputSingle` with `msg.value > amountIn` without bundling `refundETH()` in the same multicall. An attacker can monitor the router's ETH balance on-chain and immediately follow with a zero-`msg.value` WETH swap to drain it. No privileged access, malicious pool, or non-standard token is required.

## Recommendation

Introduce per-transaction ETH accounting. Record `address(this).balance - msg.value` at entry to each payable external function, and restrict `pay` to consuming at most `msg.value` of ETH (i.e., only ETH the current caller explicitly sent). Alternatively, pass `msg.value` explicitly into `pay` and use that as the ETH budget rather than reading `address(this).balance`.

## Proof of Concept

```solidity
// 1. User A calls exactInputSingle(tokenIn=WETH, amountIn=0.5e18) with msg.value=1e18.
//    pay() sees nativeBalance=1e18 >= value=0.5e18 → deposits 0.5e18 ETH.
//    0.5e18 ETH remains on router. User A does NOT call refundETH().

// 2. Attacker calls exactInputSingle(tokenIn=WETH, amountIn=0.5e18) with msg.value=0.
//    Callback: pay(WETH, attacker, pool, 0.5e18).
//    nativeBalance = 0.5e18 >= value = 0.5e18 → line 75 branch fires.
//    Router deposits User A's 0.5e18 ETH; safeTransferFrom(attacker,...) never called.

// Assert: router.balance == 0, User A's 0.5e18 ETH gone, attacker received full swap output for free.
```
