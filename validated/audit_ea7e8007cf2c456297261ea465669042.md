### Title
Rejected L1 Handler Transactions Permanently Silenced as `Committed` During L1 Provider Catch-Up - (`crates/apollo_l1_provider/src/l1_provider.rs`)

### Summary

When the L1 provider enters `CatchingUp` state, the `rejected_txs` set is silently dropped before being forwarded to `apply_commit_block`. Every L1 handler transaction that the blockifier rejected (executor-level failure) during those catch-up blocks is permanently promoted to `Committed` state instead of `Rejected`. A `Committed` transaction returns `AlreadyIncludedOnL2` on any future `validate` call, so it can never be re-proposed or re-validated. The L1 message is permanently unexecuted on L2 with no recovery path.

### Finding Description

**Step 1 – All L1 handler txs enter `consumed_l1_handler_tx_hashes` unconditionally.**

In `collect_execution_results_and_stream_txs`, the tx hash is inserted into `consumed_l1_handler_tx_hashes` *before* the execution result is inspected. If the executor returns `Err(...)`, the hash is also inserted into `rejected_tx_hashes`. Both sets are populated for the same hash. [1](#0-0) 

The unit test `failed_l1_handler_transaction_consumed` explicitly asserts this: both L1 handler txs (one of which failed with `StateError::OutOfRangeContractAddress`) appear in `consumed_l1_handler_tx_hashes`. [2](#0-1) 

**Step 2 – The batcher correctly computes `rejected_l1_handler_tx_hashes` and passes both sets to the L1 provider.** [3](#0-2) 

**Step 3 – `commit_block` drops `rejected_txs` when routing to catch-up.**

`commit_block` receives both `committed_txs` (= `consumed_l1_handler_tx_hashes`) and `rejected_txs` (= `rejected_l1_handler_tx_hashes`). When the provider is in `CatchingUp` state it calls `accept_commit_while_catching_up(committed_txs, height)` — the `rejected_txs` argument is silently discarded. [4](#0-3) 

**Step 4 – `accept_commit_while_catching_up` calls `apply_commit_block` with an empty rejected set.**

Both the `Equal` branch (current block) and the backlog replay loop call `apply_commit_block(committed_txs, Default::default())`. The developers left a `TODO` acknowledging the gap. [5](#0-4) 

**Step 5 – `apply_commit_block` partitions by `rejected_txs`; with an empty set every tx becomes `Committed`.** [6](#0-5) 

`commit_txs` then calls `mark_committed()` on every hash, including the ones that were rejected. [7](#0-6) 

**Step 6 – `Committed` state blocks all future re-inclusion.**

`is_proposable()` only returns `true` for `Pending`. [8](#0-7) 

The test `validate_rejected_transactions` shows that a `Rejected` transaction returns `ValidationStatus::Validated` (re-includable), while a `Committed` transaction returns `AlreadyIncludedOnL2` (permanently excluded). [9](#0-8) 

### Impact Explanation

An L1 handler transaction that the blockifier rejects (executor-level error, e.g. `OutOfRangeContractAddress`) is **not** included in the block body and is **not** consumed on the L1 core contract. The L1 scraper will continue to see the `LogMessageToL2` event. However, because the L1 provider has already promoted the hash to `Committed`, any subsequent `validate` call returns `AlreadyIncludedOnL2`, and `add_tx` silently ignores the re-scraped event (record already exists). The L1 message is permanently unexecuted on L2 — matching the external report's "tokens burned, no refund" pattern.

This matches: **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing** — the L1 provider permanently rejects a valid L1 handler transaction from ever being sequenced again.

### Likelihood Explanation

Catch-up is triggered on every node restart and on any height mismatch between the batcher and the L1 provider. Rejected L1 handler transactions are a documented, tested scenario (the `failed_l1_handler_transaction_consumed` test and the `reverted_l1_handler_tx_flow` integration test both exercise this path). The combination is reachable without any privileged access.

### Recommendation

Pass `rejected_txs` through the catch-up path:

1. Add a `rejected_txs: IndexSet<TransactionHash>` field to `CommitBlockBacklogEntry`.
2. Change `accept_commit_while_catching_up` to accept and forward `rejected_txs`.
3. Pass `rejected_txs` in the `Equal` branch and when replaying the backlog.
4. Remove the `TODO(guyn): check what about rejected txs here and in the backlog?` comment once fixed.

### Proof of Concept

1. Start a sequencer node. Let it process block N containing one rejected L1 handler tx (e.g. targeting an out-of-range contract address). The batcher calls `commit_block(consumed={tx}, rejected={tx}, N)`. The L1 provider is in `Pending` state → `apply_commit_block` correctly marks the tx as `Rejected`.

2. Restart the node. The L1 provider reinitialises at height 0 and enters `CatchingUp` state. The batcher replays `commit_block(consumed={tx}, rejected={tx}, N)`. The provider routes to `accept_commit_while_catching_up(consumed={tx}, N)` — `rejected_txs` is dropped. `apply_commit_block({tx}, {})` marks the tx as `Committed`.

3. The L1 scraper re-scrapes the `LogMessageToL2` event (the message was never consumed on L1). `add_tx` is called; the record already exists as `Committed` so the tx is not re-added to the proposable index.

4. Any future `validate(tx_hash)` call returns `AlreadyIncludedOnL2`. The L1 message is permanently unexecuted. [4](#0-3) [10](#0-9) [11](#0-10)

    pub fn commit_txs(
        &mut self,
        committed_txs: &[TransactionHash],
        rejected_txs: &[TransactionHash],
    ) {
        self.rollback_staging();

        for &tx_hash in committed_txs {
            self.create_record_if_not_exist(tx_hash);
            self.with_record(tx_hash, |r| r.mark_committed()).unwrap();
        }
        for &tx_hash in rejected_txs {
            self.with_record(tx_hash, |r| r.mark_rejected()).expect(
                "Storage inconsistency: a transaction sent to the batcher was removed \
                 unexpectedly.",
            );
        }
    }
