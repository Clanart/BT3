### Title
Missing `freeze_authority` Check in `log_metadata` Allows Permanent Vault Lock — (`File: solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

The `log_metadata` instruction registers any native Solana token for bridging by creating a PDA vault account. It validates that the mint is not already a bridge-controlled token (`!mint.mint_authority.contains(authority.key)`), but performs no check on `mint.freeze_authority`. Any SPL token or Token-2022 mint with a live `freeze_authority` can be registered. After tokens are locked in the vault via `init_transfer`, the freeze-authority holder can freeze the vault PDA, permanently trapping all locked tokens with no recovery path.

---

### Finding Description

`LogMetadata` accepts any mint that is not already bridge-controlled:

```rust
#[account(
    constraint = !mint.mint_authority.contains(authority.key),
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
```

No constraint checks `mint.freeze_authority`. The vault is then created unconditionally:

```rust
#[account(
    init_if_needed,
    payer = common.payer,
    token::mint = mint,
    token::authority = authority,
    seeds = [VAULT_SEED, mint.key().as_ref()],
    bump,
    token::token_program = token_program,
)]
pub vault: Box<InterfaceAccount<'info, TokenAccount>>,
```

Once the vault exists and tokens are locked in it via `init_transfer`, the freeze-authority holder can call the SPL Token `freeze_account` instruction on the vault PDA. After that, every `finalize_transfer` call for that token will fail at:

```rust
transfer_checked(
    CpiContext::new_with_signer(
        self.token_program.to_account_info(),
        TransferChecked {
            from: vault.to_account_info(),
            to: self.token_account.to_account_info(),
            authority: self.authority.to_account_info(),
            mint: self.mint.to_account_info(),
        },
        &[&[AUTHORITY_SEED, &[self.config.bumps.authority]]],
    ),
    ...
)?;
```

SPL Token rejects `transfer_checked` on a frozen source account. Because the entire Solana transaction is atomic, the nonce is not consumed on failure, so the relayer can retry — but every retry will fail identically. The tokens in the vault are permanently unclaimable.

The `SECURITY.md` does not list this as a known or accepted issue.

---

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds.**

Once the vault is frozen:
- All tokens previously locked in the vault via `init_transfer` (Solana → NEAR direction) are permanently trapped. There is no admin instruction to unfreeze the vault or drain it; the bridge program holds no freeze authority over the vault.
- Every `finalize_transfer` (NEAR → Solana) for that token will revert, leaving cross-chain transfers in a permanently unclaimable state.
- The NEAR side has already credited the sender or burned the source token; the Solana side can never release the corresponding locked tokens.

This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Moderate.**

Two realistic paths exist:

1. **Malicious token creator**: An attacker deploys an SPL token with `freeze_authority` set to themselves, calls `log_metadata` (permissionless, no admin gate), waits for users to bridge the token (e.g., via social engineering or a DeFi integration), then freezes the vault. The attacker controls the entire chain of events.

2. **Legitimate token with freeze_authority (e.g., USDC)**: USDC and other regulated stablecoins carry a `freeze_authority`. If USDC is registered via `log_metadata`, Circle could freeze the vault (intentionally or due to a compliance action targeting the vault address), permanently locking all bridged USDC.

`log_metadata` is explicitly not subject to pause controls and is callable by any payer, making the registration step fully open to unprivileged users.

---

### Recommendation

Add a constraint in `LogMetadata` rejecting mints with a live `freeze_authority`:

```rust
#[account(
    constraint = !mint.mint_authority.contains(authority.key),
    constraint = mint.freeze_authority.is_none() @ ErrorCode::MintHasFreezeAuthority,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
```

Add the corresponding error variant to `ErrorCode`. For Token-2022 mints, also inspect the `PermanentDelegate` extension, which can achieve a similar effect.

---

### Proof of Concept

1. Attacker creates an SPL token mint with `freeze_authority = attacker_keypair`.
2. Attacker (or anyone) calls `log_metadata` with that mint. The vault PDA `[VAULT_SEED, mint]` is created successfully — no freeze_authority check exists.
3. Users call `init_transfer` to bridge the token Solana → NEAR. Tokens accumulate in the vault.
4. Attacker calls SPL Token `freeze_account` on the vault PDA, signing with `attacker_keypair`.
5. Any subsequent `finalize_transfer` (NEAR → Solana) for that token executes `transfer_checked` from the frozen vault and reverts with `Custom(0x11)` (`AccountFrozen`). The nonce is not consumed (atomic revert), so the relayer retries indefinitely — all retries fail.
6. Tokens in the vault are permanently unclaimable. Users who already sent tokens to NEAR have no recourse on the Solana side. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```markdown
# Security Notes — Solana Bridge Token Factory

## Design Decisions (Non-Issues)

Items reviewed and confirmed as intentional:

- **`initialize` requires `program: Signer` with `address = crate::ID`** — Standard pattern ensuring `initialize` can only be called during program deployment. Not a vulnerability.
- **`deploy_token` and `log_metadata` are not subject to pause controls** — These require a valid MPC signature (`deploy_token`) or are read-only metadata operations (`log_metadata`). Pausing them adds no security value.
- **Initialization Wormhole message has placeholder payload (`vec![0]`)** — The init message exists solely to bootstrap the Wormhole sequence tracker. Payload content is irrelevant.
- **`unpause` accepts arbitrary `u8` value** — Only callable by admin. Naming is slightly misleading but functionally correct as a `set_pause_state` operation.
- **Wrapped tokens are always classic SPL Token, not Token-2022** — Intentional design decision. Bridged mints don't need Token-2022 extensions.

## Known Issues

Low-severity items acknowledged but not yet addressed:

- **No validation of `recipient` string in `InitTransferPayload`** — An invalid recipient causes the transfer to fail on the NEAR side after tokens are locked/burned on Solana. Manual intervention would be needed.
- **No validation of `fee_recipient` length in `FinalizeTransferPayload`** — Excessively large strings increase Wormhole message size. Bounded by Solana tx size limits in practice.
- **Token-2022 tokens with transfer hooks are not supported** — Transfer hook extra account metas are not included in instruction account sets. Affected tokens will fail at runtime (denial, not fund loss).
```
