### Title
`MerkleBlob::batch_insert()` Skips Duplicate-Key and Duplicate-Hash Validation Present in `MerkleBlob::insert()`, Corrupting the DataLayer Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::insert()` guards against duplicate keys and duplicate leaf hashes before writing to the tree. `MerkleBlob::batch_insert()` delegates only the first two items (when the tree has ≤1 existing leaf) to `insert()`, and processes all remaining items through a direct `insert_entry_to_blob()` path that performs **no** duplicate-key or duplicate-hash checks. An attacker-controlled batch containing a key or hash already present in the tree silently corrupts the blob, producing an invalid Merkle root and enabling forged inclusion proofs.

---

### Finding Description

`MerkleBlob::insert()` enforces two invariants before any structural modification:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`MerkleBlob::batch_insert()` has two code paths:

**Path A (tree has ≤1 existing leaf):** pops the last two items from the input vector and calls `self.insert()` (with checks). All *earlier* items in the vector then fall through to Path B.

**Path B (all items when tree has ≥2 leaves, or items 1…N-2 when tree has ≤1 leaf):** iterates the remaining items and calls `self.insert_entry_to_blob()` directly — with **no** `contains_key` or `contains_leaf_hash` guard:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf()`, which does:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 188-193
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // silently overwrites
    self.leaf_hash_to_index.insert(leaf.hash, index); // silently overwrites
}
``` [3](#0-2) 

`HashMap::insert` silently overwrites on collision. The old leaf node remains physically in the blob but is no longer reachable through the cache. The new subtree is built over both the old and the duplicate leaf, and the resulting internal-node hashes propagate upward, producing a Merkle root that is computed over a structurally inconsistent tree.

The Python binding `py_batch_insert` is the direct attacker-controlled entry point:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 1503-1518

    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,
) -> PyResult<()> { ... self.batch_insert(zip(keys_values, hashes).collect())?; ... }
``` [4](#0-3) 

---

### Impact Explanation

**High — DataLayer Merkle proof/blob/delta logic corrupts tree roots and lets untrusted input prove invalid state.**

When a duplicate key or hash is inserted via `batch_insert`:

1. The blob contains two leaf nodes with the same `KeyId` (or same `Hash`).
2. The `block_status_cache` silently drops the old leaf's index, making it a phantom node.
3. The Merkle root is recomputed over the corrupted subtree, yielding a root that does not correspond to any valid key-value set.
4. `get_proof_of_inclusion` will generate a proof path using the cache-tracked (new) index, but the tree structure contains the orphaned old leaf, so the proof may be internally inconsistent or verifiable against a root that encodes a state the DataLayer never committed to.
5. `check_integrity` will detect the corruption only if called explicitly; normal operation does not call it.

---

### Likelihood Explanation

**Medium.** The Python-exposed `batch_insert` is the primary DataLayer write path for bulk operations. Any caller that passes a key already present in the tree (e.g., due to a retry, a race condition, or a malicious input) will silently corrupt the tree when the tree already has ≥2 leaves. The single-item `insert` path correctly rejects this with `KeyAlreadyPresent`, so the inconsistency is non-obvious and unlikely to be caught by callers who assume batch and single-item paths have equivalent semantics.

---

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the bulk path inside `batch_insert`, before writing any leaf to the blob:

```rust
for ((key, value), hash) in &keys_values_hashes {
    if self.block_status_cache.contains_key(*key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(hash) {
        return Err(Error::HashAlreadyPresent());
    }
}
```

Alternatively, check for intra-batch duplicates (duplicate keys within the batch itself) using a local `HashSet` before the loop, mirroring the invariant enforced by `insert`.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def h(n):
    return bytes(hashlib.sha256(n.to_bytes(8, 'big')).digest())

blob = MerkleBlob(blob=bytearray())

# Pre-populate with 2 leaves so leaf_count > 1 (all batch items bypass insert())
for i in range(2):
    blob.insert(KeyId(i), ValueId(i), h(i))

# batch_insert with a duplicate of KeyId(0) — no error raised
blob.batch_insert(
    [(KeyId(100), ValueId(100)), (KeyId(0), ValueId(999))],  # KeyId(0) already exists
    [h(100), h(999)],
)

# Tree now contains two leaf nodes for KeyId(0); root is corrupted
blob.calculate_lazy_hashes()
root = blob.get_root_hash()
print("Corrupted root:", root.hex())

# get_keys_values() will show only one entry for KeyId(0) (cache overwritten),
# but the blob physically contains both leaves
kv = blob.get_keys_values()
print("Keys in cache:", list(kv.keys()))
# blob.check_integrity() will raise an integrity error
```

The call to `batch_insert` succeeds without error despite `KeyId(0)` already being present, because the bulk path at lines 587–602 bypasses the `contains_key` guard that `insert` enforces at lines 369–374. [5](#0-4) [6](#0-5)
