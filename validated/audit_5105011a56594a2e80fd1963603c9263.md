Audit Report

## Title
Router's `pay()` Consumes Stranded ETH From Prior Callers to Fund Subsequent WETH Swaps — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

The `pay()` function in `PeripheryPayments` uses `address(this).balance` — the router's total native ETH balance — to fund WETH swap inputs without verifying that the ETH belongs to the current payer. Any ETH stranded on the router from a prior caller's `msg.value` overpayment is silently consumed to subsidize a subsequent user's swap, causing direct loss of user principal.

## Finding Description

`pay()` contains a WETH-specific branch that reads the router's full native balance before deciding how to settle the swap input: [1](#0-0) 

There is no check that `address(this).balance` was contributed by the current `payer` in the current transaction. The function simply uses whatever ETH happens to be on the router.

ETH can legitimately strand on the router because all swap entry points are `payable`: [2](#0-1) 

The `receive()` guard only blocks *direct* ETH sends from non-WETH addresses: [3](#0-2) 

It does not prevent ETH from accumulating via `msg.value` in payable swap functions. When `_justPayCallback` fires, it calls `pay()` with the payer set at entry-point time: [4](#0-3) 

If the router holds stranded ETH from a prior caller, the `nativeBalance >= value` branch executes, depositing that ETH as WETH and transferring it to the pool — without ever calling `safeTransferFrom(payer, ...)`. The current payer's swap is fully or partially funded by a prior user's ETH.

## Impact Explanation

Direct loss of user principal. User A's ETH is consumed to fund User B's swap. User B receives output tokens without spending any WETH or ETH. This is a fund-loss impact above Sherlock Critical thresholds: one user loses principal, another gains a free or subsidized swap with no special permissions required.

## Likelihood Explanation

Stranding ETH on the router is trivially easy: any user who calls a payable swap function with `msg.value > amountIn` and omits `refundETH()` leaves ETH behind. The router's ETH balance is publicly readable on-chain; an attacker can monitor it and immediately call a WETH swap to drain it. No special permissions, malicious pool, or non-standard token behavior is required. The existing test suite explicitly demonstrates the `multicall + refundETH` pattern as the expected usage, confirming that omitting `refundETH()` is a realistic user error.

## Recommendation

Track how much ETH the current `msg.sender` contributed in the current transaction (e.g., via a transient storage slot set at the payable entry point), and limit `pay()`'s native ETH consumption to that amount. Alternatively, pass `msg.value` explicitly through to `pay()` as a parameter rather than reading `address(this).balance`, which aggregates ETH from all prior callers.

## Proof of Concept

```solidity
// Foundry fork test sketch
function test_strandedEthSubsidizesAttacker() public {
    // User A sends 1 ETH but only swaps 0.5 ETH worth, forgets refundETH
    vm.prank(userA);
    router.exactInputSingle{value: 1 ether}(ExactInputSingleParams({
        pool: pool, tokenIn: weth, tokenOut: token1,
        zeroForOne: true, amountIn: 0.5 ether,
        amountOutMinimum: 0, recipient: userA,
        deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
    }));
    // Router now holds 0.5 ETH belonging to userA

    // User B swaps with tokenIn=WETH, sends no ETH, has no WETH approval
    vm.prank(userB);
    router.exactInputSingle(ExactInputSingleParams({
        pool: pool, tokenIn: weth, tokenOut: token1,
        zeroForOne: true, amountIn: 0.3 ether,
        amountOutMinimum: 0, recipient: userB,
        deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
    }));

    // pay() sees nativeBalance=0.5 ETH >= value=0.3 ETH,
    // deposits 0.3 ETH as WETH, transfers to pool — no safeTransferFrom(userB) called
    assertEq(weth.balanceOf(userB), userBWethBefore, "userB spent no WETH");
    assertGt(token1.balanceOf(userB), userBToken1Before, "userB received token1");
    assertEq(address(router).balance, 0.2 ether, "userA's ETH partially stolen");
}
```
