### Title
Missing Duplicate-Hash Guard in `MerkleBlob::batch_insert` Batch Path Corrupts Merkle Tree Root — (`crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` enforces the leaf-hash uniqueness invariant only for the first two items it processes via the guarded `insert()` path. All remaining items are written directly to the blob through an unchecked loop that never calls `contains_leaf_hash`. Because `add_leaf` uses a plain `HashMap::insert` (which silently overwrites), a caller can insert a leaf whose hash already exists in the tree, placing two leaf nodes with identical hashes into the blob while the cache tracks only one of them. The resulting root hash is computed from a structurally corrupt tree.

---

### Finding Description

`batch_insert` has two distinct code paths:

**Path A — guarded (lines 578–585):** Active only when `leaf_count <= 1`. Pops up to two items from the *end* of the input vector and calls `self.insert()`, which checks both `contains_key` and `contains_leaf_hash` before writing. [1](#0-0) 

**Path B — unguarded (lines 587–603):** Processes *all remaining* items (or *all* items when `leaf_count > 1`) by constructing a `Block` directly and calling `insert_entry_to_blob` with no duplicate check of any kind. [2](#0-1) 

`insert_entry_to_blob` delegates to `add_leaf`, which calls `HashMap::insert` on both `key_to_index` and `leaf_hash_to_index`. `HashMap::insert` silently overwrites an existing entry; it does not return an error. [3](#0-2) 

By contrast, the single-item `insert()` function explicitly rejects duplicate hashes: [4](#0-3) 

**Concrete corruption sequence:**

1. Tree already has ≥ 2 leaves (so `leaf_count > 1`; Path A is skipped entirely).
2. Caller invokes `batch_insert` with a tuple `((new_key, value), H)` where `H` is the hash of an existing leaf.
3. Path B writes a new `LeafNode` with hash `H` at a fresh blob index.
4. `add_leaf` overwrites `leaf_hash_to_index[H]` with the new index; the old leaf at the original index is now unreachable by hash lookup.
5. The blob now contains two leaf nodes with identical hash `H` at different indices.
6. The subtree built over these leaves (lines 607–654) computes internal hashes from the duplicate, producing a root hash that does not correspond to any valid tree state. [5](#0-4) 

`check_integrity` would detect the inconsistency post-hoc (leaf count vs. `leaf_hash_to_index` length mismatch), but it is not called inside `batch_insert` itself. [6](#0-5) 

---

### Impact Explanation

The corrupted blob produces an invalid Merkle root. Any proof of inclusion or exclusion derived from that root is unsound: a verifier who trusts the published root can be shown a proof for a key/value pair that does not actually represent the committed state. This directly satisfies the scoped High impact: *"DataLayer Merkle proof/blob/delta logic … corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

Any caller with write access to a DataLayer store — a normal, unprivileged operation — can trigger this by supplying a batch where at least one hash duplicates an existing leaf hash. No special privileges, leaked keys, or network-level access beyond normal DataLayer write operations are required. The tree need only have ≥ 2 existing leaves (a common state in any active store) for Path A to be bypassed entirely.

---

### Recommendation

Add duplicate-hash (and duplicate-key) checks at the top of the Path B loop in `batch_insert`, mirroring the guards already present in `insert()`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing block construction ...
}
```

Alternatively, refactor `batch_insert` to route all items through `insert()` (accepting the performance trade-off), or extract the guard logic into a shared helper called by both paths.

### Proof of Concept

```rust
// Tree starts with two leaves so leaf_count > 1, bypassing Path A entirely.
let mut blob = MerkleBlob::new(vec![]).unwrap();
let hash_a = Hash::from([0xAA; 32]);
let hash_b = Hash::from([0xBB; 32]);
let hash_dup = hash_a; // same hash as the first leaf

blob.insert(KeyId(1), ValueId(1), &hash_a, InsertLocation::Auto {}).unwrap();
blob.insert(KeyId(2), ValueId(2), &hash_b, InsertLocation::Auto {}).unwrap();

// batch_insert with a third item whose hash duplicates hash_a.
// Path A is skipped (leaf_count == 2 > 1); Path B runs with no guard.
blob.batch_insert(vec![((KeyId(3), ValueId(3)), hash_dup)]).unwrap();

// The blob now contains two LeafNodes with hash_a.
// check_integrity detects the invariant violation:
assert!(blob.check_integrity().is_err()); // IntegrityLeafHashToIndexCacheLength
```
