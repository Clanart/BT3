### Title
Missing Declared Class Hash Validation in `execute_replace_class` Enables Permanent Fund Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary felt value as the new class hash without verifying it corresponds to a declared class. An unprivileged contract deployer can exploit this to permanently brick a contract holding user funds, causing irreversible fund freezing. The OS code itself contains a `TODO` acknowledging the missing check.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads the new class hash directly from the syscall request and writes it into `contract_state_changes` with no validation: [1](#0-0) 

The critical comment at line 898 reads:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
```

This is an explicit acknowledgment that the OS does **not** verify the new class hash is present in `contract_class_changes` (the declared class registry). Any felt value — including `0` or a random undeclared hash — is accepted and committed to state.

When a subsequent call is made to the now-replaced contract, `execute_entry_point` performs: [2](#0-1) 

`dict_read` on `contract_class_changes` for an undeclared class hash returns the default value `0`. `find_element` then searches for compiled class hash `0` in `compiled_class_facts_bundle`. If `0` is absent (which it always is in a valid OS run), `find_element` panics, causing the OS proof to fail entirely for any block that includes a call to the bricked contract.

The sequencer is therefore forced to permanently exclude all calls to the bricked contract from provable blocks, making any funds held by that contract irrecoverable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A contract that calls `replace_class` with an undeclared hash becomes permanently unexecutable at the OS proof level. Any ERC-20 balances, LP positions, or other assets held in that contract's storage are frozen forever. There is no recovery path: the class hash in state is committed, and no valid proof can be generated for a call to that contract.

---

### Likelihood Explanation

**Medium.** The `replace_class` syscall is callable by any contract on itself — no privileged role is required. A malicious contract deployer can:

1. Deploy a contract that accepts user deposits (e.g., a yield vault or token wrapper).
2. After accumulating funds, invoke an internal function that calls `replace_class` with an arbitrary undeclared hash (e.g., `0xdeadbeef`).
3. The OS accepts the state update unconditionally.
4. All deposited funds are permanently frozen.

This is directly analogous to the original report's assimilator-addition backdoor: just as an admin could inject arbitrary bytecode via assimilators, here a contract deployer can inject an arbitrary (non-executable) class hash via `replace_class`, with the OS providing no guard.

---

### Recommendation

Inside `execute_replace_class`, before committing the new class hash to `contract_state_changes`, add a lookup into `contract_class_changes` to assert the provided hash has a non-zero compiled class hash entry:

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the validation already implicitly required by `execute_entry_point` and closes the gap the TODO acknowledges.

---

### Proof of Concept

1. **Deploy** a contract `VaultContract` that:
   - Accepts ERC-20 deposits from users (stores balances in its own storage).
   - Exposes a function `brick_self()` that calls `replace_class(0xdeadbeef)`.

2. **Users deposit** funds; `VaultContract` accumulates balances.

3. **Attacker calls** `brick_self()`. The OS processes `execute_replace_class`: [3](#0-2) 
   `class_hash = 0xdeadbeef` is written to `contract_state_changes` with no validation.

4. **Any subsequent call** to `VaultContract` reaches `execute_entry_point`: [4](#0-3) 
   `dict_read` returns `0` for the undeclared hash; `find_element` panics. The OS proof for any block containing this call is unprovable.

5. **Result**: The sequencer must permanently exclude all calls to `VaultContract`. All user funds are frozen with no recovery mechanism.
