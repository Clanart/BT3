### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezal of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts a caller-supplied `class_hash` and unconditionally writes it into the contract state without verifying that the hash corresponds to an actually declared contract class. This is the direct structural analog of the reported `withdrawIncentives` bug: a function accepts an identifier (token address / class hash) without checking it belongs to the expected set (incentive tokens / declared classes). A contract can therefore replace its own class with an arbitrary undeclared felt value, permanently breaking its execution and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads the caller-supplied `class_hash` from the syscall request and writes it directly into `contract_state_changes` with no check that the hash is present in `contract_class_changes` (the set of declared classes):

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;          // ← user-controlled, unvalidated

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash,                    // ← written without existence check
        storage_ptr=state_entry.storage_ptr,
        nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The TODO comment at line 898 is an explicit developer acknowledgment that this check is absent. The OS never consults `contract_class_changes` to confirm the supplied hash was previously declared via a `declare` transaction. [1](#0-0) 

Compare with how `execute_deploy` and `execute_declare_transaction` handle class hashes: `execute_declare_transaction` enforces `prev_value=0` (class declared at most once) and verifies the Sierra component hash, while `execute_replace_class` performs neither check. [2](#0-1) 

---

### Impact Explanation

Once a contract's `class_hash` is set to an undeclared value, every subsequent call to that contract — including `__execute__`, `__validate__`, and any user-defined entry points — will fail at class-lookup time because no compiled class exists for that hash. The contract becomes permanently non-functional. Any ERC-20 tokens, ETH, STRK, or other assets held in the contract's storage are irrecoverably frozen, satisfying the **Critical: Permanent freezing of funds** impact criterion.

The most damaging scenario involves shared-custody contracts (multisigs, vaults, bridges) where many users' funds reside in a single contract address. A single `replace_class` call with an invalid hash freezes all of them simultaneously.

---

### Likelihood Explanation

The `replace_class` syscall is callable by any contract on itself — no privileged role is required. Realistic trigger paths include:

1. **Buggy contract logic**: A contract with a code path that calls `replace_class` with a computed or user-supplied hash (e.g., a governance-controlled upgrade mechanism) can be manipulated to supply an undeclared hash, freezing the contract.
2. **Malicious contract admin**: A contract owner who controls an upgrade function can intentionally pass an undeclared hash to freeze other users' deposited funds.
3. **Reentrancy / cross-contract manipulation**: A contract that delegates upgrade authority to a sub-call can be tricked into calling `replace_class` with an attacker-chosen hash.

The missing check is confirmed by the in-code TODO, meaning it is a known gap, not a theoretical edge case.

---

### Recommendation

Before writing the new `class_hash` into `contract_state_changes`, verify that the hash exists in `contract_class_changes` (i.e., it was declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the returned compiled class hash is non-zero:

```cairo
// Verify the class has been declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors how `execute_declare_transaction` enforces class existence before committing state. [3](#0-2) 

---

### Proof of Concept

1. **Declare** a valid contract class `C` and deploy an instance at address `A` holding user funds.
2. **Craft** an invoke transaction that calls `A.__execute__`, which internally calls the `replace_class` syscall with `class_hash = 0xdeadbeef` (any felt not present in `contract_class_changes`).
3. The OS executes `execute_replace_class`: gas is deducted, the response header is written with `failure_flag=0`, and `contract_state_changes[A].class_hash` is updated to `0xdeadbeef`.
4. The state update (`state_update` in `os.cairo`) commits this change to the Merkle tree.
5. All subsequent transactions targeting `A` fail at class resolution — no compiled class exists for `0xdeadbeef`.
6. Funds in `A`'s storage are permanently inaccessible. [4](#0-3) [5](#0-4)
