### Title
Unconditional `receive()` with No ETH Recovery Mechanism Permanently Locks Ether — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`OmniBridge.sol` contains an unconditional `receive() external payable {}` function that accepts ETH with no guards and no corresponding withdrawal or rescue path. Any ETH sent directly to the contract (not via `initTransfer`) is permanently irrecoverable, because the only ETH egress path is `finTransfer` for `tokenAddress == address(0)`, which requires a valid MPC-signed cross-chain payload. There is no admin `withdraw`, `rescue`, or sweep function anywhere in the contract.

### Finding Description
At line 574 of `OmniBridge.sol`, the contract declares:

```solidity
receive() external payable {}
``` [1](#0-0) 

This is a completely open ETH sink. The contract's only ETH egress is the `finTransfer` path, which sends ETH to a recipient only when `payload.tokenAddress == address(0)` and a valid MPC-signed `TransferMessagePayload` is presented:

```solidity
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
}
``` [2](#0-1) 

There is no `withdraw()`, `rescueETH()`, or any admin-callable function that moves ETH out of the contract outside of the bridge flow. A full search of the contract confirms no such function exists. [3](#0-2) 

Additionally, in `initTransfer` for ETH (`tokenAddress == address(0)`), the `nativeFee` component of `msg.value` is computed as:

```solidity
extensionValue = msg.value - amount - nativeFee;
``` [4](#0-3) 

In `OmniBridgeWormhole.initTransferExtension`, only `value` (i.e., `extensionValue`) is forwarded to Wormhole:

```solidity
_wormhole.publishMessage{value: value}(wormholeNonce, payload, _consistencyLevel);
``` [5](#0-4) 

The `nativeFee` portion of `msg.value` is retained in the contract balance with no release mechanism on the EVM side. It is encoded into the Borsh payload for NEAR-side accounting, but the corresponding ETH is never disbursed from the EVM contract.

### Impact Explanation
Any ETH that enters the contract via:
1. Direct `receive()` calls (e.g., accidental sends, pre-funding attempts), or
2. The `nativeFee` component of `initTransfer` calls

is permanently locked. There is no admin sweep, no withdrawal function, and no bridge-protocol path that releases this ETH. This constitutes an irrecoverable lock of user and protocol funds, matching the **Critical** impact class: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds*.

### Likelihood Explanation
- Any unprivileged user can trigger the `receive()` path by sending ETH directly to the contract address (e.g., from a wallet UI, a script, or a contract interaction).
- Every `initTransfer` call for an ERC-20 token that includes a non-zero `nativeFee` locks that ETH permanently.
- The `nativeFee` parameter is user-controlled and required by the protocol for relayer compensation signaling, making this path exercised in normal bridge usage.
- No special role or privileged access is required.

### Recommendation
1. Remove `receive() external payable {}` or replace it with a guarded version that only accepts ETH from known protocol paths (e.g., only from `initTransfer` internal flow).
2. Add an admin-callable ETH rescue function:
   ```solidity
   function rescueETH(address payable to, uint256 amount) external onlyRole(DEFAULT_ADMIN_ROLE) {
       (bool ok,) = to.call{value: amount}("");
       require(ok);
   }
   ```
3. Track the ETH balance attributable to locked bridge transfers separately from `nativeFee` deposits, and provide a release path for `nativeFee` ETH (e.g., forward it to a designated fee recipient at `initTransfer` time rather than retaining it in the contract).

### Proof of Concept
1. Deploy `OmniBridgeWormhole` on a testnet.
2. Send 1 ETH directly to the contract address (triggers `receive()`).
3. Observe that the contract balance increases by 1 ETH.
4. Confirm there is no function callable by any account (admin or otherwise) that can move this ETH out of the contract without a valid MPC-signed `finTransfer` payload for `tokenAddress == address(0)`.
5. For the `nativeFee` path: call `initTransfer(someERC20, amount, fee, 0.01 ether nativeFee, "near:alice.near", "")` with `msg.value = 0.01 ether`. The Wormhole extension forwards `extensionValue = 0` to Wormhole; the 0.01 ETH `nativeFee` remains in the contract permanently. [6](#0-5) [7](#0-6)
