### Title
ERC1155 `safeTransferFrom` Callback in `finTransfer` Permanently Locks Source-Chain Funds When Recipient Contract Reverts — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer` uses `IERC1155.safeTransferFrom` to deliver ERC1155 tokens to the recipient. `safeTransferFrom` mandates that any contract recipient implement `onERC1155Received` and return the correct selector; if the recipient contract reverts (deliberately or because it lacks the hook), the entire `finTransfer` transaction reverts. Because the MPC-signed payload fixes the recipient address and no cancellation path exists on the NEAR side, the user's source-chain tokens are permanently locked with no recovery.

---

### Finding Description

In `finTransfer`, the nonce is marked used before the external call (correct CEI order), but the ERC1155 branch uses `safeTransferFrom`:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287
// ... signature verification ...
IERC1155(multiToken.tokenAddress).safeTransferFrom(    // line 324
    address(this),
    payload.recipient,
    multiToken.tokenId,
    payload.amount,
    ""
);
```

`safeTransferFrom` calls `onERC1155Received` on the recipient if it is a contract. If that call reverts — either because the contract lacks the hook or deliberately reverts it — the entire `finTransfer` transaction reverts, rolling back `completedTransfers[nonce] = true` as well.

The consequence is:

1. The nonce is never consumed, so the transfer is never finalized on EVM.
2. The recipient address is embedded in the MPC-signed `TransferMessagePayload`; no relayer can substitute a different recipient.
3. The NEAR bridge has no cancel/refund path for a transfer whose EVM finalization permanently fails: `pending_transfers` retains the locked/burned tokens indefinitely.
4. The transfer is permanently unclaimable.

A recipient contract that conditionally reverts `onERC1155Received` can also use this as extortion leverage: it can demand a side-payment before allowing the transfer to succeed, since the NEAR-side sender's funds are held hostage.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

The user's tokens are burned or locked on NEAR at `init_transfer` time. If EVM `finTransfer` can never succeed, those tokens are irrecoverable. There is no admin escape hatch, no cancel function, and no alternative finalization path visible in the codebase. This directly matches the Critical impact class: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Medium.**

- ERC1155 tokens are a supported token type via `initTransfer1155` / `logMetadata1155`.
- Many widely-deployed contracts (multisigs, DEXes, vaults, DAOs) do not implement `onERC1155Received`. A user who bridges ERC1155 tokens to any such contract address triggers the lock.
- A malicious recipient can deliberately revert the hook to extort the sender.
- No special privilege is required; any bridge user who initiates an ERC1155 transfer to a contract recipient is exposed.

---

### Recommendation

Replace `safeTransferFrom` with the non-callback `transferFrom` (or a direct balance-update if the ERC1155 implementation supports it) in the `finTransfer` ERC1155 branch. This removes the external callback and eliminates the revert vector:

```solidity
// Instead of:
IERC1155(multiToken.tokenAddress).safeTransferFrom(
    address(this), payload.recipient, multiToken.tokenId, payload.amount, ""
);

// Use:
IERC1155(multiToken.tokenAddress).transferFrom(
    address(this), payload.recipient, multiToken.tokenId, payload.amount, ""
);
```

Alternatively, add a NEAR-side cancellation / refund function that allows the original sender to reclaim locked tokens after a configurable timeout if EVM finalization has not occurred.

---

### Proof of Concept

1. Alice bridges 100 units of an ERC1155 token from NEAR to EVM, specifying `recipient = MaliciousContract` (a contract whose `onERC1155Received` always reverts).
2. A relayer calls `OmniBridge.finTransfer` with the MPC-signed payload.
3. Execution reaches line 324; `safeTransferFrom` calls `MaliciousContract.onERC1155Received`, which reverts.
4. The entire transaction reverts; `completedTransfers[nonce]` is rolled back to `false`.
5. Every subsequent `finTransfer` attempt reverts identically — the nonce is never consumed.
6. Alice's 100 tokens remain locked/burned in the NEAR bridge contract forever.
7. `MaliciousContract` can demand a ransom from Alice before agreeing to accept the transfer.

**Root cause location:** [1](#0-0) 

The nonce guard and the `safeTransferFrom` call are both in `finTransfer`; the ERC1155 branch at lines 323–330 is the specific vulnerable path. [2](#0-1) 

The `IBridgeToken` interface confirms the non-ERC1155 mint paths do not trigger an external callback, making the ERC1155 branch uniquely vulnerable among all token types handled in `finTransfer`. [3](#0-2)
