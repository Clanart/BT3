### Title
Missing Upper-Bound Validation on `strength` in `ProofOfSpace::parse` Causes Deterministic Panic in Block-Hash Path - (File: `crates/chia-protocol/src/proof_of_space.rs`)

---

### Summary

The `strength` field of a v2 `ProofOfSpace` is parsed from the network wire as a raw `u8` (0–255) with no upper-bound check. The rest of the codebase treats `strength < 64` as a hard invariant (enforced only by a debug `assert!` in the Python-binding helper). When a node calls `update_digest` on a `ProofOfSpace` carrying `strength >= 64`, the inner call to `quality_string()` returns `None`, and the unconditional `.expect(...)` panics, crashing the node before it can reject the block.

---

### Finding Description

`ProofOfSpace::parse` deserializes the `strength` byte for v2 proofs without any range check:

```rust
// crates/chia-protocol/src/proof_of_space.rs  line 319
let strength = <u8 as Streamable>::parse::<TRUSTED>(input)?;
```

The only place a bound is enforced is in the Python-binding constructor helper:

```rust
// line 68
fn make_v2(plot_index: u16, meta_group: u8, strength: u8) -> Self {
    assert!(strength < 64);   // ← only a debug assert, absent from parse()
    ...
}
```

After a successful parse, `update_digest` (called whenever the block hash is computed) reaches this code for every v2 proof:

```rust
// lines 251-253
self.quality_string()
    .expect("internal error. Can't compute hash of invalid ProofOfSpace")
    .update_digest(digest);
```

`quality_string()` delegates to `chia_pos2::quality_string_from_proof` passing the unchecked `self.strength`. When `strength >= 64` the external library returns `None`, `quality_string()` propagates `None`, and `.expect(...)` panics unconditionally.

`update_digest` on `ProofOfSpace` is called from `RewardChainBlock::update_digest` (line 54), which is invoked whenever any code computes the hash of a received block — a mandatory step in block validation.

---

### Impact Explanation

A malicious peer crafts a `RewardChainBlock` (or any enclosing block type) whose embedded `ProofOfSpace` has `version = 1` and `strength >= 64`. The bytes parse successfully (no error is returned). The first time the node computes the block hash — which must happen before any proof-of-space or VDF validation — `update_digest` panics, crashing the node process. Because the panic is deterministic and triggered purely by the wire bytes, every node that receives the same crafted block crashes identically, producing a chain halt.

This matches the Critical allowed impact: *a serialized network object triggers deterministic consensus divergence / chain halt*.

---

### Likelihood Explanation

The attack requires only the ability to send a single crafted block to a listening node — no keys, no stake, no privileged role. The malformed byte is a single field in a freely constructable protocol message. Any external peer can trigger the crash.

---

### Recommendation

Add an explicit range check inside `ProofOfSpace::parse` immediately after reading `strength` for v2 proofs:

```rust
// after line 319
if strength >= 64 {
    return Err(Error::InvalidPoS);
}
```

Additionally, replace the `.expect(...)` in `update_digest` with a graceful error path (or make `update_digest` return a `Result`) so that even if an out-of-range value somehow reaches that point, the node returns an error rather than panicking.

---

### Proof of Concept

1. Construct a valid v2 `ProofOfSpace` wire encoding (prefix byte `0b10`, valid BLS keys, valid `plot_index`, valid `meta_group`).
2. Set the `strength` byte to any value `>= 64` (e.g., `0xFF`).
3. Embed it in a `RewardChainBlock` / `FullBlock` and send it to a target node.
4. The node calls `ProofOfSpace::parse` → succeeds (no bound check).
5. The node calls `RewardChainBlock::update_digest` → calls `ProofOfSpace::update_digest` → calls `quality_string()` → `chia_pos2::quality_string_from_proof` returns `None` → `.expect(...)` panics → node process terminates.

The conclusion that `quality_string_from_proof` returns `None` for `strength >= 64` is inferred from the `assert!(strength < 64)` invariant in the codebase. If `chia_pos2` panics internally instead of returning `None`, the crash still occurs but via a different code path; if it somehow succeeds, the panic in `update_digest` would not trigger — but the `assert!` evidence makes the `None`-return path the most likely behavior.

                    self.strength,
