Audit Report

## Title
Cross-User ETH Theft via Global `address(this).balance` in Partial-Branch of `pay()` ‚Äî (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` reads `address(this).balance` as a global, persistent value with no per-caller accounting. When a prior user's excess ETH remains in the contract and a subsequent WETH payment is requested for a larger amount, the partial branch silently consumes the prior user's ETH, wraps it, and forwards it to the pool ‚Äî pulling only the shortfall from the current payer. The prior user's `refundETH()` call then returns zero, causing an unrecoverable loss of principal.

## Finding Description

In `pay()`, when `token == WETH`, the function reads `nativeBalance = address(this).balance` (line 74) and branches on three cases. The partial branch at lines 78‚Äì81 fires when `0 < address(this).balance < value`:

```solidity
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
}
```

`address(this).balance` is global ‚Äî it accumulates ETH from all callers across all transactions. There is no per-user deposit ledger.

**Exploit path:**

1. LP-A calls `addLiquidityExactShares{value: 1 ETH}(...)` on a WETH pool. The pool requests 0.5 ETH in the callback. The first branch fires (`nativeBalance >= value`), wraps exactly 0.5 ETH, and sends it to the pool. The remaining **0.5 ETH stays in the contract** (lines 75‚Äì77).

2. LP-A has not yet called `refundETH()`. The 0.5 ETH is now unprotected in `address(this).balance`.

3. LP-B calls `addLiquidityExactShares{value: 0}(...)` for a WETH pool, with parameters causing the pool to request 2 ETH in the callback. In `pay(WETH, lpB, pool, 2 ETH)`:
   - `nativeBalance = 0.5 ETH` (LP-A's leftover)
   - `0 < 0.5 < 2` ‚Üí partial branch fires
   - LP-A's 0.5 ETH is wrapped and forwarded to the pool
   - Only 1.5 ETH of WETH is pulled from LP-B

4. LP-A calls `refundETH()` (lines 58‚Äì63), which sends `address(this).balance` to `msg.sender`. The balance is now 0 ‚Äî LP-A receives nothing.

The `receive()` guard (lines 32‚Äì34) only blocks unsolicited ETH pushes from non-WETH addresses; it does not prevent ETH arriving via `msg.value` on the payable entry points. Both `addLiquidityExactShares` overloads are `external payable` and callable standalone without `multicall`, so LP-A can legitimately send excess ETH expecting to reclaim it via `refundETH()` ‚Äî a pattern the interface documentation explicitly acknowledges (lines 15‚Äì17 of `IMetricOmmPoolLiquidityAdder.sol`) but does not enforce atomically.

## Impact Explanation

LP-A suffers a direct, unrecoverable loss of native ETH principal. LP-B receives more WETH-backed liquidity than they paid for, subsidized by LP-A's ETH. This is a cross-user fund theft through a public, unprivileged entry point with no privileged preconditions, meeting the Critical/High direct loss of user principal threshold.

## Likelihood Explanation

Any user who calls `addLiquidityExactShares` directly (not via `multicall` + `refundETH` in the same call) and sends more ETH than the pool requests is immediately vulnerable. The leftover ETH is visible on-chain. An attacker can observe it in the mempool or in the next block and call `addLiquidityExactShares` with parameters that trigger the partial branch, consuming the victim's ETH. No privileged access, special tokens, or unusual conditions are required.

## Recommendation

Introduce per-transaction ETH accounting. The simplest fix is to store `msg.value` in transient storage at the entry of each payable function and, inside `pay()`, only consume up to that stored amount as native ETH ‚Äî reverting or falling through to the ERC20 path if `address(this).balance` exceeds what the current caller deposited. Alternatively, enforce that WETH-leg liquidity adds using native ETH must always be composed via `multicall` with `refundETH` as the final call, and remove the `payable` modifier from standalone `addLiquidityExactShares` overloads (or revert if `msg.value > 0` and the call is not inside a multicall context).

## Proof of Concept

```solidity
function test_crossLpEthTheft() public {
    // LP-A sends 1 ETH; pool only needs 0.5 ETH
    vm.deal(lpA, 1 ether);
    vm.prank(lpA);
    adder.addLiquidityExactShares{value: 1 ether}(
        pool, lpA, 1, deltaRequiring0_5Eth, 1 ether, 0, ""
    );
    // 0.5 ETH remains in adder; lpA has not called refundETH
    assertEq(address(adder).balance, 0.5 ether);

    // LP-B (attacker) calls with 0 ETH; pool requests 2 ETH
    // partial branch: wraps lpA's 0.5 ETH + pulls 1.5 ETH WETH from lpB
    vm.deal(lpB, 0);
    deal(address(weth), lpB, 2 ether);
    vm.prank(lpB);
    weth.approve(address(adder), type(uint256).max);
    vm.prank(lpB);
    adder.addLiquidityExactShares{value: 0}(
        pool, lpB, 2, deltaRequiring2Eth, 2 ether, 0, ""
    );

    // lpA's ETH is gone
    assertEq(address(adder).balance, 0);
    vm.prank(lpA);
    adder.refundETH(); // returns 0 ‚Äî lpA lost 0.5 ETH
    assertEq(lpA.balance, 0); // expected 0.5 ETH, got 0
}
```
