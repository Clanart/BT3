### Title
`deprecated_declared_classes` silently dropped from `ThinStateDiff` in commitment path, producing wrong `state_diff_commitment`, `concatenated_counts`, and block hash for blocks containing Cairo 0 Declare transactions — (`crates/blockifier/src/state/cached_state.rs`)

### Summary
`CommitmentStateDiff` has no `deprecated_declared_classes` field. The `From<CommitmentStateDiff> for ThinStateDiff` conversion hardcodes `deprecated_declared_classes: Vec::new()`. Every call to `BlockExecutionArtifacts::new` feeds this truncated `ThinStateDiff` into `calculate_block_commitments`, which computes both the `state_diff_commitment` (Poseidon hash) and the `state_diff_length` packed into `concatenated_counts`. Both values are wrong for any block that includes a Cairo 0 (V0/V1) Declare transaction. Because `concatenated_counts` and `state_diff_commitment` are chained directly into the block hash, the sequencer commits and broadcasts a block hash that diverges from what the Starknet OS would compute for the same block.

### Finding Description

**Root cause — `From<CommitmentStateDiff> for ThinStateDiff` always emits an empty `deprecated_declared_classes`:** [1](#0-0) 

`CommitmentStateDiff` is produced from `StateMaps` via: [2](#0-1) 

`StateMaps` has a `declared_contracts: HashMap<ClassHash, bool>` field that tracks whether a class is declared, but no `deprecated_declared_classes` field. When a Cairo 0 Declare (V0/V1) transaction executes, the blockifier calls `state.set_contract_class(class_hash, ...)` and marks `declared_contracts[class_hash] = true`, but this never propagates into `CommitmentStateDiff.class_hash_to_compiled_class_hash` (Cairo 0 classes have no compiled class hash). The class hash is therefore invisible to the commitment path.

**Propagation into block commitments:**

`BlockExecutionArtifacts::new` calls `calculate_block_commitments` with `ThinStateDiff::from(commitment_state_diff.clone())`: [3](#0-2) 

Inside `calculate_block_commitments`, two values are derived from the truncated `ThinStateDiff`:

1. `state_diff_length = state_diff.len()` — `ThinStateDiff::len()` counts `deprecated_declared_classes.len()`, which is always 0 here, so the length is under-counted by the number of Cairo 0 declarations in the block.
2. `state_diff_commitment = calculate_state_diff_hash(&state_diff)` — the Poseidon hash is computed over a state diff that is missing the deprecated class hashes. [4](#0-3) 

Both wrong values are packed into `concatenated_counts` and `state_diff_commitment`, which are chained into the block hash: [5](#0-4) 

**The wrong `ThinStateDiff` is also written to storage:**

`thin_state_diff()` re-derives from `commitment_state_diff` with the same empty `deprecated_declared_classes`: [6](#0-5) 

This is the value passed to `commit_proposal_and_block` → `append_state_diff`: [7](#0-6) 

The stored `ThinStateDiff` is missing the deprecated declared class. The `state_diff_length` field written to the block header is also wrong, so the P2P sync client's length-based termination condition accepts a state diff that omits the deprecated class: [8](#0-7) 

**The `StarknetClientStateDiff` conversion also drops the field:** [9](#0-8) 

The cende blob sent to the DA layer also omits the deprecated declared class.

**Contrast with the RPC simulation path**, which correctly handles deprecated declared classes: [10](#0-9) 

The divergence between the simulation path (correct) and the block-building path (incorrect) confirms this is a bug, not a design decision.

**The TODO comment in the conversion acknowledges the incompleteness:** [11](#0-10) 

### Impact Explanation

For any block containing a Cairo 0 Declare transaction:

- `state_diff_commitment` (Poseidon hash) is computed over a state diff missing the declared class hash → **wrong authoritative commitment value stored in the block header and used in the block hash**.
- `state_diff_length` in `concatenated_counts` is under-counted → **wrong `concatenated_counts` field in the block hash**.
- The final block hash diverges from what the Starknet OS would compute for the same block → **proof generation fails or produces a proof for a different block hash than the one committed on-chain**.
- The stored `ThinStateDiff` is missing the deprecated declared class → **wrong state diff in storage, wrong class marker, wrong P2P sync termination**.
- The cende blob is missing the deprecated declared class → **wrong DA blob**.

This matches: **Critical — Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input.**

### Likelihood Explanation

Cairo 0 Declare transactions (V0/V1) remain valid in the Starknet protocol. The blockifier explicitly handles them: [12](#0-11) 

Any unprivileged user can submit a Cairo 0 Declare transaction. The OS flow tests confirm Cairo 0 declarations are exercised in production scenarios: [13](#0-12) 

### Recommendation

1. Add a `deprecated_declared_classes: Vec<ClassHash>` field to `CommitmentStateDiff`.
2. Populate it from `StateMaps.declared_contracts` by filtering for entries where `is_declared == true` and `compiled_class_hashes` does not contain the class hash (i.e., Cairo 0 classes).
3. Update `From<CommitmentStateDiff> for ThinStateDiff` to use this field instead of `Vec::new()`.
4. Update `From<StateMaps> for CommitmentStateDiff` to populate the new field.
5. Update `StarknetClientStateDiff` conversion to propagate `old_declared_contracts` from the new field.

### Proof of Concept

1. Submit a Cairo 0 Declare transaction (V0 or V1) to the sequencer gateway.
2. The transaction is included in a block by the batcher.
3. `BlockExecutionArtifacts::new` is called; `ThinStateDiff::from(commitment_state_diff.clone())` produces a `ThinStateDiff` with `deprecated_declared_classes: []`.
4. `calculate_block_commitments` computes `state_diff_commitment = H([...])` over the truncated diff and `state_diff_length = N` (missing the 1 deprecated class entry).
5. `concat_counts(n_txs, n_events, N, l1_da_mode)` embeds the wrong length.
6. `calculate_block_hash` chains the wrong `concatenated_counts` and `state_diff_commitment` → produces block hash `H_wrong`.
7. The Starknet OS, given the same block, computes `state_diff_commitment` over the full diff (including the deprecated class) and `state_diff_length = N+1`, producing block hash `H_correct ≠ H_wrong`.
8. Proof verification fails; the committed block hash is permanently wrong.

pub fn induced_state_diff(
    transactional_state: &mut CachedState<MutRefState<'_, CachedState<ExecutionStateReader>>>,
    deprecated_declared_class_hash: Option<ClassHash>,
) -> ExecutionResult<ThinStateDiff> {
    let blockifier_state_diff =
        CommitmentStateDiff::from(transactional_state.to_state_diff()?.state_maps);

    Ok(ThinStateDiff {
        deployed_contracts: blockifier_state_diff.address_to_class_hash,
        storage_diffs: blockifier_state_diff.storage_updates,
        class_hash_to_compiled_class_hash: blockifier_state_diff.class_hash_to_compiled_class_hash,
        deprecated_declared_classes: deprecated_declared_class_hash
            .map_or_else(Vec::new, |class_hash| vec![class_hash]),
        nonces: blockifier_state_diff.address_to_nonce,
    })
}
