### Title
Missing Minimum-Fee Enforcement in OS Allows Fee-Free V3 Transaction Execution — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` returns `0` whenever all resource-bound prices are zero, and `charge_fee` unconditionally skips the ERC-20 transfer when that result is zero. Because the OS never asserts `max_fee > 0` before executing a V3 transaction, a sender can craft a transaction with a non-zero L2-gas amount but zero prices, consume execution resources, and pay nothing. The OS — the protocol's enforcement layer — produces a valid proof for such a block.

---

### Finding Description

**`compute_max_possible_fee`** (lines 87–102) computes:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

`pack_resource_bounds` (called during hash computation) enforces `max_amount ∈ [0, 2^64−1]` and `max_price_per_unit ≥ 0` (non-negative, no lower bound of 1):

```cairo
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
assert_nn(resource_bounds.max_price_per_unit);
``` [2](#0-1) 

`assert_nn` only checks non-negativity — `max_price_per_unit = 0` is explicitly permitted. Likewise `tip ∈ [0, 2^64−1]`, so `tip = 0` is valid.

**`charge_fee`** then does:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
``` [3](#0-2) 

When `max_fee == 0` the function returns immediately — no ERC-20 transfer is executed, no fee is charged, and the proof remains valid.

**`get_initial_user_gas_bound`** returns `resource_bounds[L2_GAS_INDEX].max_amount`:

```cairo
func get_initial_user_gas_bound(common_tx_fields: CommonTxFields*) -> felt {
    assert common_tx_fields.n_resource_bounds = 3;
    return common_tx_fields.resource_bounds[L2_GAS_INDEX].max_amount;
}
``` [4](#0-3) 

So a transaction with `L2_GAS_INDEX.max_amount = N > 0` and all prices/tip = 0 receives a full gas budget of `N` for `__validate__` + `__execute__`, while `compute_max_possible_fee` returns exactly `0` and `charge_fee` exits without payment.

The OS never inserts a guard such as `assert max_fee != 0` before dispatching validate/execute. The execution flow for invoke transactions is:

1. Hash verified ✓  
2. Nonce incremented ✓  
3. `run_validate` called with gas budget `N` ✓  
4. `__execute__` called with gas budget `N` ✓  
5. `charge_fee` → `max_fee = 0` → **early return, no payment** ✗ [5](#0-4) 

---

### Impact Explanation

The OS is the protocol's proof-level enforcement layer. A block containing V3 transactions with zero prices but non-zero L2-gas amounts is provably valid under the current OS logic. A sequencer that does not independently enforce `max_fee > 0` (or that relies on the OS to do so) will include such transactions, generate a valid STARK proof, and have it accepted on-chain. At scale this fills blocks with fee-free work, starving legitimate fee-paying transactions of block space.

**Impact: High — Network not being able to confirm new transactions.**

---

### Likelihood Explanation

The attacker is an ordinary V3 transaction sender. No privileged role is required. The crafted transaction is cryptographically valid (correct hash, valid signature, correct nonce). The only gate is whether the sequencer's mempool independently rejects `max_fee == 0` transactions. Because the OS is the canonical enforcement layer and it does not reject them, any sequencer that delegates this check to the OS will include them. The attack is cheap to repeat across many accounts (each with its own nonce sequence) to sustain block saturation.

---

### Recommendation

Add an explicit assertion in `charge_fee` (or immediately before `run_validate` in each transaction execution function) that `max_fee > 0` for non-L1-handler V3 transactions:

```cairo
// Before executing validate/execute:
let max_fee = compute_max_possible_fee(tx_info=tx_info);
with_attr error_message("Transaction max fee must be non-zero.") {
    assert_not_zero(max_fee);
}
```

Additionally, `pack_resource_bounds` should enforce `max_price_per_unit > 0` (change `assert_nn` to `assert_nn_le` with a lower bound of 1, or add `assert_not_zero`) so that the hash-time validation and the fee-time validation are consistent. [6](#0-5) 

---

### Proof of Concept

1. Attacker constructs a valid V3 invoke transaction with:
   - `resource_bounds[L1_GAS]  = { max_amount: 0, max_price_per_unit: 0 }`
   - `resource_bounds[L2_GAS]  = { max_amount: 1_100_000_000, max_price_per_unit: 0 }` ← full execute budget
   - `resource_bounds[L1_DATA] = { max_amount: 0, max_price_per_unit: 0 }`
   - `tip = 0`

2. `pack_resource_bounds` passes: `max_amount ≤ 2^64−1` ✓, `max_price_per_unit = 0 ≥ 0` ✓, `tip = 0 ≤ 2^64−1` ✓. Transaction hash is computed and signed normally.

3. Sequencer (relying on OS enforcement) includes the transaction.

4. OS runs `get_initial_user_gas_bound` → returns `1_100_000_000` (full `EXECUTE_MAX_SIERRA_GAS`).

5. OS runs `__validate__` and `__execute__` with the full gas budget. Arbitrary contract logic executes.

6. OS calls `charge_fee`:
   - `compute_max_possible_fee` = `0*0 + 1_100_000_000*(0+0) + 0*0` = **0**
   - `if (max_fee == 0) { return (); }` — **fee transfer skipped**.

7. Proof is generated and accepted. Attacker paid zero fees for up to `EXECUTE_MAX_SIERRA_GAS = 1,100,000,000` gas units of execution.

8. Attacker repeats with fresh nonces across many accounts to saturate block capacity indefinitely. [7](#0-6) [8](#0-7)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
