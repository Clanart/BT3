### Title
Missing Pre-Burn Normalized-Amount Validation in `init_transfer` Causes Permanent Fund Freeze — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer` on NEAR burns or locks user tokens before any check that the decimal-normalized amount is non-zero. The zero-amount guard lives exclusively in `sign_transfer`, which is called later by a relayer. When a user sends an amount smaller than the minimum representable unit on the destination chain, their tokens are permanently consumed and the transfer can never be finalized.

---

### Finding Description

**Step 1 — `init_transfer` accepts any amount above the fee, with no normalization check.** [1](#0-0) 

The only guard is `fee.fee < amount`. There is no check that `normalize_amount(amount - fee, decimals) > 0`.

**Step 2 — `init_transfer_internal` immediately burns/locks the tokens and returns `U128(0)`, consuming them from `ft_on_transfer`.** [2](#0-1) 

`burn_tokens_if_needed` fires for deployed (bridged) tokens; for native tokens (e.g. wNEAR) the balance is locked inside the bridge. Either way the user's funds are gone from their account.

**Step 3 — `normalize_amount` uses floor division.** [3](#0-2) 

For a NEAR-native token with `origin_decimals = 24` and `decimals = 18` (EVM cap), any amount below `10^6` yoctoNEAR normalizes to zero.

**Step 4 — The zero-amount guard is only in `sign_transfer`, called later by a relayer.** [4](#0-3) 

`sign_transfer` will always revert with `InvalidAmountToTransfer` for such a transfer. Because no cancel/refund function exists in the contract, the stored `TransferMessage` is permanently stuck and the burned/locked tokens are irrecoverable.

---

### Impact Explanation

**Permanent freezing of user funds.** Tokens are burned (deployed tokens) or locked (native tokens) inside the bridge at `init_transfer_internal` time. Because `sign_transfer` unconditionally reverts for zero-normalized amounts and no cancellation path exists, the funds can never be released. This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Low-to-Medium.** Any unprivileged user calling `ft_transfer_call` with a sub-unit amount triggers the path. For NEAR tokens bridging to EVM (24 → 18 decimal difference = factor of `10^6`), amounts below `1,000,000 yoctoNEAR` are sufficient. While the monetary value is small per incident, the loss is total and irrecoverable for the affected user, and the entry path requires no special role.

---

### Recommendation

Add a normalization check inside `init_transfer` (or at the top of `init_transfer_internal`) before burning or locking tokens. The token decimals are already stored in `self.token_decimals` keyed by `OmniAddress`:

```rust
// Pseudocode addition inside init_transfer_internal, before burn_tokens_if_needed
if let Some(decimals) = self.token_decimals.get(&transfer_message.token) {
    let normalized = Self::normalize_amount(
        transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
        decimals,
    );
    require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
}
```

This mirrors the recommendation from the external report: retrieve the exchange rate (normalized amount) before committing the asset, and reject the operation if the output would be zero.

---

### Proof of Concept

1. A NEAR-native token is registered with `origin_decimals = 24`, `decimals = 18` (standard EVM cap).
2. User calls `ft_transfer_call` on the token contract, sending `amount = 500_000` yoctoNEAR to the bridge with `msg = InitTransfer { recipient: "eth:0x...", fee: 0, ... }`.
3. Bridge's `ft_on_transfer` → `init_transfer` passes the `fee < amount` check (`0 < 500_000`).
4. `init_transfer_internal` calls `burn_tokens_if_needed`, burning the 500,000 yoctoNEAR. Returns `U128(0)` → `ft_on_transfer` returns 0 → tokens are consumed.
5. Relayer calls `sign_transfer` for the stored transfer.
6. `normalize_amount(500_000, {origin_decimals:24, decimals:18})` = `500_000 / 10^6` = `0`.
7. `require!(amount_to_transfer > 0, ...)` panics → `sign_transfer` always reverts.
8. No cancel function exists. The 500,000 yoctoNEAR are permanently lost. [5](#0-4) [6](#0-5) [4](#0-3) [3](#0-2)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }

        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
