### Title
Undercounted DA Gas in `estimate_minimal_gas_vector` for Declare Transactions Causes Below-Minimum Fee Admission - (File: crates/blockifier/src/fee/gas_usage.rs)

---

### Summary

`estimate_minimal_gas_vector` in `gas_usage.rs` deliberately sets `n_compiled_class_hash_updates: 0` for `Declare` transactions, even though a V2/V3 Declare always writes one `(class_hash, compiled_class_hash)` pair to the DA segment. The code even carries a self-incriminating comment: `// TODO(Yoni): BLOCKIFIER-RESET: should be 1.` This causes the minimum-fee pre-validation check (`check_fee_bounds`) to accept Declare transactions whose resource bounds are too low to cover the actual DA cost, allowing them to be sequenced and executed at a below-true-cost fee.

---

### Finding Description

`estimate_minimal_gas_vector` computes a lower-bound gas vector that is used in `check_fee_bounds` to reject transactions whose resource bounds are provably insufficient. For a `Declare` transaction, the function builds a `StateChangesCount` with:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    // TODO(Yoni): BLOCKIFIER-RESET: should be 1.
    n_compiled_class_hash_updates: 0,
    n_modified_contracts: 1,
},
``` [1](#0-0) 

However, the actual execution of a V2/V3 Declare transaction always writes one compiled class hash entry. This is confirmed by `declare_expected_state_changes_count` in the test suite:

```rust
} else if version == TransactionVersion::TWO || version == TransactionVersion::THREE {
    StateChangesCount {
        n_storage_updates: 1,
        n_modified_contracts: 1,
        n_compiled_class_hash_updates: 1, // Also set compiled class hash.
        ..StateChangesCount::default()
    }
``` [2](#0-1) 

And by `try_declare`, which unconditionally calls `state.set_compiled_class_hash` for V2/V3: [3](#0-2) 

`get_onchain_data_segment_length` adds `n_compiled_class_hash_updates * 2` felts to the DA segment: [4](#0-3) 

With `n_compiled_class_hash_updates = 0` instead of `1`, the minimal estimate is missing **2 felts** of DA cost. In KZG-DA mode (`use_kzg_da = true`) each felt costs `DATA_GAS_PER_FIELD_ELEMENT` blob gas units; in calldata mode it costs `SHARP_GAS_PER_DA_WORD` L1 gas units. The missing DA cost propagates directly into the minimum gas vector returned to `check_fee_bounds`.

`check_fee_bounds` is called from `perform_pre_validation_stage`, which runs before execution for every transaction: [5](#0-4) 

The check compares `minimal_gas_amount_vector` against the transaction's resource bounds. Because the minimal vector is too low, a Declare transaction with resource bounds that cover only the underestimated cost passes pre-validation and is admitted to the mempool and sequenced.

---

### Impact Explanation

**Impact: High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing; also Critical — Incorrect fee/gas/resource accounting with economic impact.**

A V2/V3 Declare transaction can be submitted with resource bounds that are below the true minimum required to cover the compiled-class-hash DA cost. The transaction passes `check_fee_bounds` (because the minimum estimate omits 2 DA felts), is admitted by the gateway, enters the mempool, and is sequenced. The actual execution charges the correct (higher) DA cost, but since the resource bounds were set below that, the transaction reverts with an "Insufficient max resource" error — yet the sequencer still charges the declared (too-low) fee. The sequencer therefore absorbs the DA cost that the user did not pay, creating a systematic economic shortfall on every V2/V3 Declare transaction whose bounds are crafted to exactly match the underestimated minimum.

---

### Likelihood Explanation

Every V2/V3 Declare transaction is affected. A user who calls `starknet_estimateFee` for a Declare transaction receives an estimate derived from the same undercount (since `exec_estimate_fee` also runs through the blockifier execution path), and will naturally set their resource bounds to match the estimate. This means the bug is triggered by normal, non-malicious usage. A malicious actor can also deliberately craft bounds at exactly the underestimated minimum to maximize the shortfall.

---

### Recommendation

Change `n_compiled_class_hash_updates` from `0` to `1` for `Transaction::Declare(_)` in `estimate_minimal_gas_vector`, as the TODO comment already acknowledges:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    n_compiled_class_hash_updates: 1, // V2/V3 always writes compiled_class_hash.
    n_modified_contracts: 1,
},
``` [1](#0-0) 

Additionally, add a regression test that executes a V3 Declare with resource bounds set to exactly the `estimate_minimal_gas_vector` output and asserts it does not revert due to insufficient resources.

---

### Proof of Concept

1. Deploy an account contract on a Starknet devnet running this sequencer.
2. Prepare a V3 Declare transaction for a new Sierra class.
3. Call `starknet_estimateFee` — the returned `l1_data_gas` (KZG mode) or `l1_gas` (calldata mode) will be 2 felts × `DATA_GAS_PER_FIELD_ELEMENT` short of the true cost.
4. Submit the Declare with `l1_data_gas.max_amount` set to exactly the estimated value.
5. Observe: `check_fee_bounds` passes (the underestimated minimum ≤ declared bound), the transaction is sequenced, but execution reverts with "Insufficient max L1DataGas" because the actual DA cost includes the 2-felt compiled-class-hash entry.
6. The sequencer charges the declared (too-low) fee, not the true cost, creating an economic shortfall.

The delta in DA gas is:
- **KZG mode**: `2 × DATA_GAS_PER_FIELD_ELEMENT` blob gas units per Declare transaction.
- **Calldata mode**: `2 × SHARP_GAS_PER_DA_WORD` L1 gas units per Declare transaction. [6](#0-5) [7](#0-6)

pub fn get_onchain_data_segment_length(state_changes_count: &StateChangesCount) -> usize {
    // For each newly modified contract:
    // contract address (1 word).
    // + 1 word with the following info: A flag indicating whether the class hash was updated, the
    // number of entry updates, and the new nonce.
    let mut onchain_data_segment_length = state_changes_count.n_modified_contracts * 2;
    // For each class updated (through a deploy or a class replacement).
    onchain_data_segment_length +=
        state_changes_count.n_class_hash_updates * constants::CLASS_UPDATE_SIZE;
    // For each modified storage cell: key, new value.
    onchain_data_segment_length += state_changes_count.n_storage_updates * 2;
    // For each compiled class updated (through declare): class_hash, compiled_class_hash
    onchain_data_segment_length += state_changes_count.n_compiled_class_hash_updates * 2;

    onchain_data_segment_length
}
