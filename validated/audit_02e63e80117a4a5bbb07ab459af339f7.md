### Title
`eth_estimateGas` / `eth_call` Simulation Bypasses `stateDB.Commit()` State-Conflict Check, Diverging from Actual Execution - (File: x/evm/keeper/state_transition.go)

---

### Summary

`EstimateGas` and `EthCall` invoke `ApplyMessageWithConfig` with `commit=false`, permanently skipping the `stateDB.Commit()` call. `Commit()` is the only place where the `ErrStateConflict` invariant is enforced — detecting when both the EVM and a nested native precompile action (via `ExecuteNativeAction`) write to the same storage slot. Because simulation never reaches `Commit()`, `eth_estimateGas` returns a successful gas estimate for transactions that will fail with `VmError = "state conflict"` during actual block execution. Users who rely on the estimate to predict success will submit transactions that are included in the block, fail silently, and still be charged gas fees.

---

### Finding Description

**Root cause — `commit=false` skips the only conflict-detection path:**

In `ApplyMessageWithConfig`, the final block is:

```go
if commit {
    if err := stateDB.Commit(); err != nil {
        if errors.Is(err, statedb.ErrStateConflict) {
            return &types.EVMResult{
                GasUsed: gasUsed,
                VmError: statedb.ErrStateConflict.Error(),
                ...
            }, nil
        }
        return nil, errorsmod.Wrap(err, "failed to commit stateDB")
    }
}
``` [1](#0-0) 

When `commit=false` (all simulation paths), this entire block is skipped. The function returns a clean `EVMResult` with no `VmError`, regardless of whether `Commit()` would have detected a conflict.

**The conflict detection logic in `stateDB.Commit()`:**

`Commit()` enforces a non-overlap invariant: for every EVM-dirty storage slot, it reads the current store value. If the store value differs from both `originStorage` (what the EVM read at start) and `dirtyStorage` (what the EVM wrote), a nested native action must have written the slot concurrently — a state conflict.

```go
store := s.keeper.GetState(s.ctx, obj.Address(), key)
if store != origin && store != dirty {
    return fmt.Errorf(
        "%w: address %s key %s modified by both EVM execution and native action ...",
        ErrStateConflict, ...)
}
``` [2](#0-1) 

**The code itself acknowledges the divergence:**

```go
// Note: estimateGas and eth_call do not hit this path because commit is
// false for simulations, so they will succeed even when a real execution
// would produce a state conflict.
``` [3](#0-2) 

**Simulation entry point (`EstimateGas`):**

```go
executable := func(gas uint64) (vmError bool, rsp *types.EVMResult, err error) {
    msg.GasLimit = gas
    // pass false to not commit StateDB
    rsp, err = k.ApplyMessageWithConfig(ctx, msg, cfg, false)
    ...
    return rsp.Failed(), rsp, nil
}
``` [4](#0-3) 

`rsp.Failed()` checks `len(r.VmError) > 0`. Since `commit=false` never sets `VmError = "state conflict"`, `Failed()` returns `false`, and the binary search converges on a successful gas estimate.

**Actual execution path (`ApplyTransaction`):**

```go
res, err := k.ApplyMessageWithConfig(tmpCtx, msg, cfg, true)
``` [5](#0-4) 

Here `commit=true`, so `Commit()` runs, the conflict is detected, and the result carries `VmError = "state conflict"` — the transaction is included in the block with status=0.

---

### Impact Explanation

This is a **preview/execution inconsistency** directly analogous to the external report: the "preview" function (`eth_estimateGas`) uses a code path that omits a critical check present in the actual execution path (`ApplyTransaction`).

Concrete impact:

1. A user calls `eth_estimateGas` on a transaction that interacts with a custom precompile using `ExecuteNativeAction` and also writes to the same EVM storage slot. The estimate returns success and a gas value.
2. The user submits the transaction with the estimated gas.
3. The transaction is included in the block, `Commit()` detects the conflict, and the transaction is finalized with `VmError = "state conflict"` (status=0). No state change occurs.
4. The user is charged `max(gasLimit × minGasMultiplier, actualGasUsed) × gasPrice` — real funds lost for a transaction the simulation guaranteed would succeed.

This fits: **"EVM state transition bug that permits valid user funds/fees to be mis-accounted"** — the simulation path and the execution path diverge on a reachable code branch, causing users to pay fees for transactions that the simulation predicted as successful.

---

### Likelihood Explanation

The trigger requires a chain built on Ethermint to register a custom precompile that calls `ExecuteNativeAction` and writes to an EVM storage slot that the calling contract also writes to in the same transaction. This is a realistic pattern for precompiles that implement bank/staking operations on EVM-mapped storage (as documented in the Ethermint precompile guide). Any unprivileged user interacting with such a contract via `eth_estimateGas` + `eth_sendRawTransaction` can trigger the inconsistency without any special privileges.

---

### Recommendation

1. **Run a speculative `Commit()` in simulation**: After `ApplyMessageWithConfig(ctx, msg, cfg, false)`, perform the conflict check from `Commit()` on a read-only basis (without flushing) and surface any detected conflict as a `VmError` in the estimate response.
2. **Alternatively**, run simulation with `commit=true` on a branched cache context that is always discarded, so the full `Commit()` path executes and conflicts are detected.
3. At minimum, document in the `eth_estimateGas` JSON-RPC response that the estimate cannot predict `ErrStateConflict` failures for transactions involving native precompile interactions.

---

### Proof of Concept

```
1. Register a custom precompile P at address 0xABCD that, when called,
   executes ExecuteNativeAction writing value V2 to storage slot K of
   contract C.

2. Deploy contract C with a function f() that:
   a. Calls precompile P  → native action writes K = V2
   b. Writes K = V3 in EVM storage

3. Call eth_estimateGas({to: C, data: f()})
   → ApplyMessageWithConfig(commit=false)
   → stateDB.Commit() is NEVER called
   → No VmError, returns gas estimate G

4. Submit eth_sendRawTransaction with gas = G
   → ApplyMessageWithConfig(commit=true)
   → stateDB.Commit() runs
   → store[K] = V2 (written by native action) ≠ origin[K] AND ≠ dirty[K] = V3
   → ErrStateConflict returned
   → Transaction included in block with VmError="state conflict", status=0

5. User is charged max(G × minGasMultiplier, actualGasUsed) × gasPrice
   for a transaction that eth_estimateGas predicted would succeed.
```
