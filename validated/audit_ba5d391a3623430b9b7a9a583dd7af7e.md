### Title
Silent Proof-Manager Store Failure Allows Proof-Facts Transaction Into Mempool Without Stored Proof, Breaking Consensus Conversion - (`crates/apollo_gateway/src/gateway.rs`)

### Summary

In `GenericGateway::store_proof_and_spawn_archiving`, a failure from `store_proof_in_proof_manager` is only logged with `error!` and then silently discarded. The gateway continues to send the transaction to the mempool. Later, when the consensus orchestrator calls `convert_internal_consensus_tx_to_consensus_tx` → `convert_internal_rpc_tx_to_rpc_tx` to reconstruct the full RPC transaction for peer propagation, it calls `get_proof(proof_facts)` which fails because the proof was never persisted. This is the direct sequencer analog of the external "unchecked transfer result" pattern: a critical storage operation's failure is swallowed, and the downstream flow proceeds as if it succeeded.

### Finding Description

**Root cause — `store_proof_and_spawn_archiving` in `crates/apollo_gateway/src/gateway.rs`:**

```rust
let store_result = self
    .transaction_converter
    .store_proof_in_proof_manager(proof_facts.clone(), proof.clone())
    .await;
match store_result {
    Ok(...) => { /* record latency */ }
    Err(e) => {
        error!("Failed to set proof in proof manager: {}", e);
        // ← error is swallowed; no early return
    }
}
// flow continues unconditionally
``` [1](#0-0) 

After this block, the gateway proceeds to send the transaction to the mempool:

```rust
let mempool_client_result = self.mempool_client.add_tx(add_tx_args).await;
``` [2](#0-1) 

The internal transaction carries only `proof_facts` (the hash commitment); the actual `Proof` bytes are stripped and are expected to be retrievable from the proof manager. If `store_proof_in_proof_manager` failed, the proof is absent from both the in-memory cache and the filesystem-backed `FsProofStorage`. [3](#0-2) 

**Downstream breakage — `convert_internal_rpc_tx_to_rpc_tx`:**

When the consensus orchestrator reconstructs the full RPC transaction for propagation to peer validators, it calls:

```rust
let proof = if tx.proof_facts.is_empty() {
    Proof::default()
} else {
    self.get_proof(&tx.proof_facts).await?   // ← fails: proof not stored
};
``` [4](#0-3) 

`get_proof` delegates to `ProofManager::get_proof`, which checks the cache (miss) then the filesystem (miss), returning `None` or an error. The `?` propagates the failure up through `convert_internal_consensus_tx_to_consensus_tx`. [5](#0-4) 

### Impact Explanation

A client-side-proving invoke transaction (with non-empty `proof_facts`) passes cryptographic proof verification and stateful validation, is admitted to the mempool, and is returned to the caller as accepted. However, because the proof was never stored, the consensus layer cannot reconstruct the full RPC transaction when it needs to propagate the proposal to peer validators. This causes `convert_internal_consensus_tx_to_consensus_tx` to return an error for that transaction, breaking the consensus proposal flow for any block containing it. This matches the allowed impact: **High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload** (the proof payload is irrecoverably missing after admission).

### Likelihood Explanation

Low. The trigger requires `store_proof_in_proof_manager` to fail, which requires a transient filesystem I/O error or an IPC failure to the proof manager component. This is not directly attacker-controlled but is reachable under disk-full, permission, or network-partition conditions in a distributed deployment where the proof manager runs as a separate process. [6](#0-5) 

### Recommendation

Propagate the `store_proof_in_proof_manager` error instead of swallowing it. If the proof cannot be stored, the gateway should return an error to the caller and not forward the transaction to the mempool:

```rust
self.transaction_converter
    .store_proof_in_proof_manager(proof_facts.clone(), proof.clone())
    .await
    .map_err(|e| {
        error!("Failed to set proof in proof manager: {}", e);
        StarknetError::internal_with_logging("Failed to store proof:", &e)
    })?;
```

This mirrors the pattern already used for proof verification failures in `await_verification_task_and_extract_proof_data`, which correctly propagates errors and prevents the transaction from reaching the mempool. [7](#0-6) 

### Proof of Concept

1. Submit an `InvokeV3` transaction with valid non-empty `proof_facts` and `proof` to the gateway.
2. Arrange for `ProofManager::set_proof` to return an `FsProofStorageError::IoError` (e.g., by making the proof storage directory read-only or by injecting a fault into the local component channel).
3. Observe that `add_tx_inner` logs `"Failed to set proof in proof manager"` but still returns `Ok(GatewayOutput)` and the transaction hash appears in the mempool.
4. Trigger the consensus orchestrator to build a proposal containing that transaction. The call to `convert_internal_consensus_tx_to_consensus_tx` will fail with a proof-not-found error when attempting to reconstruct the RPC transaction for propagation, because `ProofManager::get_proof` returns `None` for the unstored `facts_hash`. [8](#0-7) [9](#0-8)

    pub async fn get_proof(
        &self,
        proof_facts: ProofFacts,
    ) -> Result<Option<Proof>, FsProofStorageError> {
        let facts_hash = proof_facts.hash();
        // Check cache first.
        if let Some(proof) = self.cache.get(&facts_hash) {
            return Ok(Some(proof));
        }
        // Fallback to filesystem.
        let proof = self.proof_storage.get_proof(facts_hash).await?;
        if let Some(proof) = &proof {
            self.cache.insert(facts_hash, proof.clone());
        }
        Ok(proof)
    }

    async fn write_proof_atomically(
        &self,
        facts_hash: Felt,
        proof: Proof,
    ) -> FsProofStorageResult<()> {
        // Write proof to a temporary directory.
        let (_tmp_root, tmp_dir) = self.create_tmp_dir(facts_hash).await?;
        self.write_proof_to_file(&tmp_dir, &proof).await?;

        // Atomically rename directory to persistent one.
        // If a concurrent write already placed the proof at the persistent path, the rename
        // will fail (e.g. ENOTEMPTY on Linux). Since proofs are deterministic for a given
        // facts_hash, the existing proof is identical and we can safely treat this as success.
        let persistent_dir = self.get_persistent_dir_with_create(facts_hash).await?;
        match tokio::fs::rename(&tmp_dir, &persistent_dir).await {
            Ok(()) => Ok(()),
            Err(_)
                if tokio::fs::try_exists(persistent_dir.join("proof")).await.unwrap_or(false) =>
            {
                Ok(())
            }
            Err(e) => Err(e.into()),
        }
    }
