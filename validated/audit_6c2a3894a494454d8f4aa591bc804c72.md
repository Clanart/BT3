### Title
Missing Shard-ID Binding in `set_state_header` Allows Cross-Shard State Root Substitution — (File: `chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` accepts a caller-supplied `shard_id` and a peer-supplied `ShardStateSyncResponseHeader` containing a `ShardChunk`. It validates that the chunk is internally consistent and that its hash appears somewhere in the block's `chunk_headers_root`, but it **never checks that `chunk.shard_id() == shard_id`**. Because `verify_path` proves only membership in the merkle tree (not position), a malicious peer can substitute shard X's chunk as shard Y's header. The header is then stored under `StateHeaderKey(shard_Y, sync_hash)` carrying shard X's `prev_state_root`. All subsequent state-part validation and application for shard Y proceeds against shard X's state root, permanently corrupting the syncing node's trie for shard Y.

---

### Finding Description

**Root cause — `set_state_header` (`chain/chain/src/state_sync/adapter.rs` lines 368–531)**

The function performs five checks:

| Check | What it proves | What it does NOT prove |
|---|---|---|
| `validate_chunk_proofs` (line 380) | Chunk body is internally consistent (hash, tx-root, receipts-root) | Nothing about which shard the chunk belongs to |
| `verify_path` against `chunk_headers_root` (lines 394–403) | The chunk hash appears *somewhere* in the block's chunk-header merkle tree | That the chunk is at the index corresponding to `shard_id` |
| `verify_path` for `prev_chunk` (lines 416–425) | Same membership proof for the previous chunk | Same gap |
| Receipt proofs (lines 438–503) | Receipts are consistent with the `shard_id` *parameter* | That the chunk itself belongs to `shard_id` |
| `validate_state_root_node` (lines 514–522) | The state-root node is internally consistent | That the state root belongs to `shard_id` |

`verify_path` is defined as:

```rust
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}
```

It computes `compute_root_from_path(path, hash(item)) == root`. It proves the item is *a* leaf in the tree; it does not prove *which* leaf position. The position-aware variant `verify_path_with_index` (used for chunk parts in `ShardsManagerActor::validate_part`) is never called here.

The grep confirms: `chunk.shard_id()` is never referenced anywhere in `adapter.rs`. The storage key at line 527 is `StateHeaderKey(shard_id, sync_hash)` — the caller-supplied `shard_id`, not the chunk's own shard field.

**Attacker-controlled data flow**

A syncing node requests a state-sync header for shard Y from a peer. Any peer on the network can respond. The peer crafts a `ShardStateSyncResponseHeader` containing:

- `chunk` = the legitimate `ShardChunk` for shard X (obtained from the block)
- `chunk_proof` = the legitimate merkle path for shard X's position in the block

Both are freely available from any full node. The syncing node calls:

```
set_state_header(shard_Y, sync_hash, crafted_header)
```

- `validate_chunk_proofs` passes — shard X's chunk is internally valid.
- `verify_path(chunk_headers_root, shard_X_proof, ChunkHashHeight(shard_X_hash, ...))` passes — shard X's chunk hash is indeed in the block.
- No check `chunk.shard_id() == shard_Y` exists.
- The header is committed to `DBCol::StateHeaders` under key `StateHeaderKey(shard_Y, sync_hash)`, embedding shard X's `prev_state_root`.

**Propagation through state-part application**

`set_state_part` (`adapter.rs` lines 534–561) reads the stored header, extracts `state_root = chunk.take_header().take_inner().prev_state_root()` — now shard X's root — and validates all incoming parts against it. The attacker supplies valid parts for shard X; they pass.

`apply_state_part` (`chain/chain/src/runtime/mod.rs` lines 1501–1528) writes shard X's trie nodes into the store keyed by `shard_uid` for shard Y.

`set_state_finalize` (`chain/chain/src/chain_update.rs` lines 452–568) reads the stored header, derives `shard_uid` from the `shard_id` parameter (shard Y), but uses `chunk_header.prev_state_root()` (shard X's root) as the execution state root. Shard Y's `ChunkExtra` is written with shard X's post-execution `new_root`.

**Exact corrupted value**: `ChunkExtra` for shard Y at the sync point carries shard X's state root. Every subsequent block applied on top of shard Y reads from the wrong trie.

---

### Impact Explanation

**Impact: High**

A syncing node (any node performing state sync — new nodes, resharded validators, nodes recovering from a gap) can be caused to install shard X's entire trie as shard Y's state. After finalization:

- The node cannot process any block that references shard Y's correct state root; it will diverge from the canonical chain.
- If the node is a chunk producer for shard Y, it will produce chunks with an incorrect `prev_state_root`, which will be rejected by all other validators, causing it to lose chunk-production rewards and potentially be slashed.
- If the node is a chunk validator for shard Y, it will endorse chunks against the wrong state, producing invalid endorsements.
- Recovery requires a full re-sync, which is expensive and time-consuming.

---

### Likelihood Explanation

**Likelihood: Medium**

- Any unprivileged peer on the NEAR P2P network can respond to state-sync header requests; no validator key or special role is required.
- The attacker only needs to win the race to respond before a legitimate peer does, or to be the only peer the syncing node contacts for that shard.
- All inputs needed (shard X's chunk, its merkle proof) are publicly available from any full node.
- The attack targets nodes during state sync, which is a well-defined, observable window.

---

### Recommendation

Add an explicit shard-ID binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: bind the chunk to the requested shard
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} does not match requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

Apply the same check to `prev_chunk_header` if present. Additionally, consider using `verify_path_with_index` (already used in `ShardsManagerActor::validate_part`) to prove the chunk is at the correct shard index within the block's chunk-header merkle tree, providing a cryptographic binding rather than a field equality check alone.

### Proof of Concept

```
Preconditions:
  - Attacker is a reachable peer of the syncing node.
  - The network is at sync_hash H; shard layout has shards X and Y.
  - Attacker has a full node and knows the block at H-1.

Steps:
1. Attacker obtains shard X's ShardChunk from block H-1 and its merkle proof
   (chunk_proof_X) from the block's chunk_headers_root.

2. Syncing node sends StateHeaderRequest { shard_id: Y, sync_hash: H }.

3. Attacker responds with ShardStateSyncResponseHeader {
       chunk: shard_X_chunk,          // shard_id() == X, not Y
       chunk_proof: chunk_proof_X,    // valid proof for shard X's position
       state_root_node: shard_X_state_root_node,
       incoming_receipts_proofs: [],  // empty or crafted to pass length checks
       ...
   }

4. Syncing node calls set_state_header(Y, H, crafted_header):
   - validate_chunk_proofs(shard_X_chunk) → Ok (internally consistent)

   - apply_chunk(state_root = shard_X_root, shard_uid = Y) → succeeds
   - ChunkExtra for shard Y written with shard X's post-state root.

Result: Shard Y's state is permanently replaced with shard X's state.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}

                .compute_transaction_validity(&prev_block_header, &chunk)

            vec![true; transactions.len()]
