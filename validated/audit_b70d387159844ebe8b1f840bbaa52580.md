### Title
Token-2022 Metadata Pointer Allows Hijacking Legitimate Token Identity in `log_metadata` — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

`LogMetadata::parse_metadata_account` validates that the caller-supplied `metadata` account key matches the address stored in the mint's `MetadataPointer` extension, and that the account is owned by `MetaplexID`. It never verifies that the deserialized `MetadataAccount.mint` field equals `self.mint.key()`. An unprivileged attacker can create a Token-2022 mint whose `metadata_pointer` points to a legitimate token's Metaplex PDA, call `log_metadata`, and cause NEAR to deploy a bridged token for the attacker's mint carrying the legitimate token's name and symbol.

---

### Finding Description

The vulnerable function is `parse_metadata_account` in `LogMetadata`:

```
solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs
lines 72–90
``` [1](#0-0) 

The two checks performed are:

1. `require_keys_eq!(metadata.key(), address, ...)` — the passed account key equals `metadata_pointer.metadata_address.0`
2. `if metadata.owner == &MetaplexID` — the account is owned by Metaplex

After both pass, the code immediately reads `metadata.name` and `metadata.symbol` without ever checking `metadata.mint == self.mint.key()`. [2](#0-1) 

The Token-2022 branch that reaches this function: [3](#0-2) 

The `metadata` account field in the `LogMetadata` struct is declared `/// CHECK: may be uninitialized` and is entirely caller-supplied with no PDA derivation constraint: [4](#0-3) 

**Attack steps:**

1. Attacker creates a Token-2022 mint `M_attacker` (retaining its own mint authority — the only constraint is `!mint.mint_authority.contains(authority.key)`, which is satisfied).
2. Attacker initializes a `MetadataPointer` extension on `M_attacker` pointing to the Metaplex PDA of a well-known token, e.g. USDC: `find_program_address([b"metadata", MetaplexID, USDC_mint], MetaplexID)`.
3. Attacker calls `log_metadata` passing `mint = M_attacker` and `metadata = USDC_Metaplex_PDA`.
4. `process()` enters the Token-2022 branch, reads `metadata_pointer.metadata_address.0 = USDC_Metaplex_PDA`, and calls `parse_metadata_account(USDC_Metaplex_PDA)`.
5. Both checks pass: the passed account key equals the pointer address, and the account is owned by `MetaplexID`.
6. The code deserializes USDC's `MetadataAccount` and returns `name = "USD Coin"`, `symbol = "USDC"`.
7. `LogMetadataPayload { token: M_attacker, name: "USD Coin", symbol: "USDC", decimals: X }` is posted via Wormhole. [5](#0-4) 

On the NEAR side, `deploy_token_callback` receives the Wormhole-attested VAA and calls `deploy_token_internal` with `token_address = M_attacker` and `metadata = { name: "USD Coin", symbol: "USDC", decimals: X }`: [6](#0-5) 

NEAR deploys a new bridged token for `M_attacker` carrying USDC's identity. Since the attacker retains mint authority over `M_attacker`, they can mint unlimited `M_attacker` tokens on Solana, lock them in the bridge vault, and bridge them to NEAR to receive "USD Coin (USDC)"-named tokens — unbacked by any real USDC.

---

### Impact Explanation

The invariant that a token's logged metadata must correspond to that token's own on-chain metadata is broken. NEAR maps `M_attacker` to a token with the name and symbol of a legitimate asset. Because the attacker controls `M_attacker`'s mint authority, they can create unlimited supply of `M_attacker` on Solana, bridge it to NEAR, and receive tokens that appear to be USDC. This constitutes token-mapping corruption that breaks bridge collateralization and misdirects value.

---

### Likelihood Explanation

The attack requires only:
- Creating a Token-2022 mint (permissionless on Solana)
- Setting a `MetadataPointer` extension to any existing Metaplex PDA (permissionless)
- Calling the public `log_metadata` instruction

No privileged access, no key compromise, no oracle manipulation. The Metaplex PDA of every major token is publicly known and permanently on-chain. The attack is fully local-testable.

---

### Recommendation

After deserializing the `MplMetadata` account in `parse_metadata_account`, add a binding check:

```rust
if metadata.owner == &MetaplexID {
    let data = metadata.try_borrow_data()?;
    let mpl = MplMetadata::try_deserialize(&mut data.as_ref())?;
    // ADD THIS CHECK:
    require_keys_eq!(
        mpl.mint,
        self.mint.key(),
        ErrorCode::InvalidTokenMetadataAddress,
    );
    Ok((mpl.name.clone(), mpl.symbol.clone()))
}
```

This ensures the deserialized metadata account actually belongs to the mint being registered, not to an arbitrary legitimate token.
