### Title
Missing Class Hash Validation in `execute_replace_class` Allows Arbitrary Class Hash Assignment — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary
The `execute_replace_class` syscall handler in the StarkNet OS accepts a user-controlled `class_hash` from the syscall request and directly writes it into `contract_state_changes` without verifying that the class hash corresponds to a previously declared contract class. A developer TODO comment at the exact location explicitly acknowledges this missing check. This is a direct analog to the ibbtc `redeem()` missing address validation: both functions accept an arbitrary user-supplied identifier and act on it without allowlist/existence validation.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads `request.class_hash` — a value fully controlled by the calling contract — and immediately uses it to overwrite the contract's class hash in `contract_state_changes`:

```cairo
// syscall_impls.cairo lines 896–910
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The only guard present is the gas check (`reduce_syscall_gas_and_write_response_header`). There is **no assertion** that `class_hash` exists in `contract_class_changes` (the declared-class registry). The TODO comment at line 898 is the developers' own acknowledgment that this check is absent and was deferred.

When a subsequent transaction calls the now-replaced contract, `execute_entry_point` performs:

```cairo
// execute_entry_point.cairo lines 154–166
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // ← the attacker-supplied hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,           // ← resolves to 0 for undeclared hash
);
``` [2](#0-1) 

If the class hash is undeclared, `dict_read` returns 0 (the default for an absent key), and `find_element` with key `0` fails to locate a compiled class, causing the OS to panic and the block proof to be unprovable.

---

### Impact Explanation

**Critical — Permanent freezing of funds / High — Network halt.**

- A contract that holds user funds and calls `replace_class` with an arbitrary undeclared hash will have its class hash permanently corrupted in the committed state. Every future call to that contract will fail at the OS proof level, making the funds irrecoverable.
- If the corrupted contract is a widely-used system contract or if many contracts are affected, the sequencer cannot produce a valid proof for any block containing calls to those contracts, resulting in a total network halt.

---

### Likelihood Explanation

Any unprivileged contract can emit a `replace_class` syscall with an arbitrary felt value as the class hash. No special role, key, or operator access is required. The attacker only needs to deploy a contract and invoke it via a standard `invoke` transaction. The missing validation is confirmed by the in-code TODO, indicating it was never implemented.

---

### Recommendation

Before writing the new class hash to `contract_state_changes`, assert that the hash exists in `contract_class_changes` (i.e., it was previously declared):

```cairo
// After reading class_hash from request:
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the validation already performed implicitly in `execute_entry_point` and makes the OS consistent with the intended protocol invariant that `replace_class` may only target declared classes.

---

### Proof of Concept

1. Attacker deploys `VictimContract` holding ETH/STRK funds.
2. `VictimContract` contains a function that calls `replace_class(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
3. Attacker submits an `invoke` transaction calling that function.
4. The OS processes `execute_replace_class`: gas check passes, `class_hash = 0xdeadbeef` is written to `contract_state_changes` with no further validation. [3](#0-2) 
5. State is committed with `VictimContract.class_hash = 0xdeadbeef`.
6. In the next block, any transaction calling `VictimContract` causes `execute_entry_point` to call `dict_read{dict_ptr=contract_class_changes}(key=0xdeadbeef)` → returns `0`. [4](#0-3) 
7. `find_element(..., key=0)` fails — no compiled class with hash `0` exists in `compiled_class_facts_bundle`. [5](#0-4) 
8. OS panics; block proof cannot be generated → **permanent fund freeze** and/or **network halt**.
