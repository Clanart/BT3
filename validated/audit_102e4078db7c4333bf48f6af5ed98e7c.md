### Title
EIP-712 `Tx` Type Schema Omits `timeout_height` Field, Enabling Replay of Expired Legacy EIP-712 Transactions — (File: `ante/cosmos/eip712.go`, `ethereum/eip712/eip712_legacy.go`, `ethereum/eip712/types.go`)

---

### Summary

The EIP-712 `Tx` type schema used for hashing in both the legacy and non-legacy Ethermint EIP-712 paths deliberately omits the `timeout_height` field. However, the Amino JSON sign bytes passed as the EIP-712 message payload **do** include `timeout_height` when it is non-zero. Because EIP-712 hashing silently ignores message fields that are absent from the type schema, `timeout_height` is never covered by the user's signature. An attacker who intercepts a signed legacy EIP-712 Cosmos transaction can freely modify `timeout_height` — including stripping an expiry the user intended — without invalidating the signature, enabling replay of transactions the user meant to expire.

---

### Finding Description

**Root cause — struct/type schema mismatch (direct analog to the external report):**

In `ethereum/eip712/eip712_legacy.go`, `extractMsgTypes` defines the `Tx` EIP-712 type schema:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id",       Type: "string"},
    {Name: "fee",            Type: "Fee"},
    {Name: "memo",           Type: "string"},
    {Name: "msgs",           Type: "Msg[]"},
    {Name: "sequence",       Type: "string"},
    // Note timeout_height was removed because it was not getting filled with the legacyTx
    // {Name: "timeout_height", Type: "string"},
},
``` [1](#0-0) 

The same omission exists in the non-legacy path: [2](#0-1) 

In `ante/cosmos/eip712.go`, `VerifySignature` reconstructs the Amino sign bytes from the live transaction, **including** `timeout_height` when it is non-zero:

```go
txBytes := legacytx.StdSignBytes(
    signerData.ChainID,
    signerData.AccountNumber,
    signerData.Sequence,
    tx.GetTimeoutHeight(),   // ← included in JSON when != 0
    legacytx.StdFee{Amount: tx.GetFee(), Gas: tx.GetGas()},
    msgs, tx.GetMemo(),
)
``` [3](#0-2) 

These `txBytes` (containing `timeout_height`) are then passed directly to `LegacyWrapTxToTypedData`: [4](#0-3) 

Inside `LegacyWrapTxToTypedData`, the JSON is unmarshalled into `txData` (so `timeout_height` is present in the message map), but the `Tx` type schema does not declare it: [5](#0-4) 

Per EIP-712, `TypedDataAndHash` hashes only the fields declared in the type schema. Any field present in the message but absent from the schema is **silently dropped** from the hash. Therefore `timeout_height` is never included in the computed `sigHash`, regardless of its value.

The same silent omission occurs in the non-legacy `decodeAminoSignDoc` path used by `ethsecp256k1.PubKey.verifySignatureAsEIP712`: [6](#0-5) 

Note the contrast: the Protobuf decode path **explicitly rejects** `timeout_height != 0` with an error, acknowledging the schema cannot cover it: [7](#0-6) 

The Amino path has no such guard.

---

### Impact Explanation

An attacker who can observe or relay a signed legacy EIP-712 Cosmos transaction (e.g., a malicious RPC node, mempool observer, or relayer) can:

1. Intercept a transaction signed with `timeout_height = X` (user's intent: expire at block X).
2. Mutate the raw transaction bytes to set `timeout_height = 0` (no expiry).
3. Resubmit the modified transaction.

Because `timeout_height` is not covered by the EIP-712 signature, the `LegacyEip712SigVerificationDecorator` accepts the modified transaction as fully valid: [8](#0-7) 

The `TxTimeoutHeightDecorator` (which enforces expiry) is bypassed because the attacker set `timeout_height = 0`. The transaction executes after the user's intended deadline, with the user's funds/authorizations consumed against their intent.

This is an **EIP-712 authorization bypass enabling replay** — the user's signature does not cover a consensus-critical transaction field, allowing unauthorized mutation of that field post-signing.

---

### Likelihood Explanation

- The legacy EIP-712 ante handler (`newLegacyCosmosAnteHandlerEip712`) is still active and reachable by any transaction carrying an `ExtensionOptionsWeb3Tx` extension option. [9](#0-8) 
- The attack requires only that the user set `timeout_height != 0` (a standard Cosmos SDK practice for time-bounded operations) and that the attacker can observe the mempool (trivially achievable by running a node or connecting to a public RPC).
- No privileged access, governance, or validator collusion is required.
- The non-legacy path (`ethsecp256k1.verifySignatureAsEIP712` → `decodeAminoSignDoc`) is also affected for Amino-encoded sign docs, broadening the attack surface. [10](#0-9) 

---

### Recommendation

1. **Add `timeout_height` to the `Tx` EIP-712 type schema** in both `ethereum/eip712/types.go` (`createEIP712Types`) and `ethereum/eip712/eip712_legacy.go` (`extractMsgTypes`), mirroring the Protobuf path's explicit rejection of the field.
2. **Alternatively**, add a guard in `ante/cosmos/eip712.go` `VerifySignature` that rejects any transaction with `tx.GetTimeoutHeight() != 0` on the legacy EIP-712 path, consistent with the Protobuf decode path's behavior.
3. Ensure the Amino decode path in `decodeAminoSignDoc` also rejects `timeout_height != 0` (matching `decodeProtobufSignDoc`).

---

### Proof of Concept

```
1. Alice constructs a legacy EIP-712 Cosmos MsgDelegate with timeout_height = 100.
2. Alice's wallet signs the EIP-712 typed data. The Tx schema has no timeout_height
   field, so the signed hash H does not depend on timeout_height.
3. Alice broadcasts the signed transaction (timeout_height=100, sig=S).
4. Block 100 passes; the transaction is not included.
5. Attacker intercepts the transaction from the mempool, sets timeout_height=0.
6. Attacker rebroadcasts the modified transaction.
7. VerifySignature in ante/cosmos/eip712.go reconstructs txBytes with timeout_height=0,
   builds the same EIP-712 typed data (timeout_height absent from schema → same hash H),
   and verifies sig S successfully.
8. TxTimeoutHeightDecorator sees timeout_height=0 → no expiry → passes.
9. Alice's delegation executes after her intended deadline.
```

func decodeAminoSignDoc(signDocBytes []byte) (apitypes.TypedData, error) {
	// Ensure codecs have been initialized
	if err := validateCodecInit(); err != nil {
		return apitypes.TypedData{}, err
	}

	var aminoDoc legacytx.StdSignDoc
	if err := aminoCodec.UnmarshalJSON(signDocBytes, &aminoDoc); err != nil {
		return apitypes.TypedData{}, err
	}

	var fees legacytx.StdFee
	if err := aminoCodec.UnmarshalJSON(aminoDoc.Fee, &fees); err != nil {
		return apitypes.TypedData{}, err
	}

	// Validate payload messages
	msgs := make([]sdk.Msg, len(aminoDoc.Msgs))
	for i, jsonMsg := range aminoDoc.Msgs {
		var m sdk.Msg
		if err := aminoCodec.UnmarshalJSON(jsonMsg, &m); err != nil {
			return apitypes.TypedData{}, fmt.Errorf("failed to unmarshal sign doc message: %w", err)
		}
		msgs[i] = m
	}

	if err := validatePayloadMessages(msgs); err != nil {
		return apitypes.TypedData{}, err
	}

	chainID, err := types.ParseChainID(aminoDoc.ChainID)
	if err != nil {
		return apitypes.TypedData{}, errors.New("invalid chain ID passed as argument")
	}

	typedData, err := WrapTxToTypedData(
		chainID.Uint64(),
		signDocBytes,
	)
	if err != nil {
		return apitypes.TypedData{}, fmt.Errorf("could not convert to EIP712 representation: %w", err)
	}

	return typedData, nil

func (pubKey PubKey) verifySignatureAsEIP712(msg, sig []byte) bool {
	eip712Bytes, err := eip712.GetEIP712BytesForMsg(msg)
	if err != nil {
		return false
	}

	if pubKey.verifySignatureECDSA(eip712Bytes, sig) {
		return true
	}

	// Try verifying the signature using the legacy EIP-712 encoding
	legacyEIP712Bytes, err := eip712.LegacyGetEIP712BytesForMsg(msg)
	if err != nil {
		return false
	}

	return pubKey.verifySignatureECDSA(legacyEIP712Bytes, sig)
}
