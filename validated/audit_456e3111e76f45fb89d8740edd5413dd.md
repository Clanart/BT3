### Title
`ProofOfInclusion.valid()` Does Not Validate Against an External Root Hash — Forged DataLayer Inclusion Proof Accepted - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion.valid()` only checks the internal hash-chain consistency of the proof. It never compares the computed root against any externally-committed tree root. Because the final guard `existing_hash == self.root_hash()` is a tautology (both sides always equal `layers.last().combined_hash` after the loop), the method returns `true` for any internally-consistent proof, regardless of which tree it actually belongs to. An attacker can craft a structurally valid `ProofOfInclusion` for a key that is not in the real tree, and any caller that relies solely on `proof.valid()` — the only validation method exposed through the Python/wasm bindings — will accept the forged proof.

### Finding Description

`ProofOfInclusion` is a `Streamable` struct (deserializable from bytes) exposed to Python via `#[pymethods]`. [1](#0-0) 

Its `valid()` method walks the layer chain and checks that each `combined_hash` equals the hash computed from the previous hash and the sibling: [2](#0-1) 

The final guard is:

```rust
existing_hash == self.root_hash()
```

`root_hash()` is defined as: [3](#0-2) 

After the loop completes without returning `false`, `existing_hash` holds the last `calculated_hash`, which was already asserted equal to `layer.combined_hash` in the same iteration. `self.root_hash()` returns that same `layers.last().combined_hash`. The comparison is therefore always `true` — it is a tautology. `valid()` never touches any externally-committed root.

The Python binding exposes exactly this method and `root_hash()` as two separate, independent calls, with no combined "validate against this root" primitive: [4](#0-3) 

The Python type stub confirms the surface: [5](#0-4) 

The fuzz target and all Rust/Python tests call only `proof.valid()` without comparing `proof.root_hash()` to the actual blob root: [6](#0-5) [7](#0-6) 

### Impact Explanation

An attacker who controls a DataLayer peer can:

1. Choose any target key `K` they wish to falsely prove is in tree `T` (with committed root `R`).
2. Construct a `ProofOfInclusion` with `node_hash = H(K)` and an arbitrary number of layers where each `combined_hash` is correctly computed from the previous hash and a chosen `other_hash`. The chain is internally consistent, so `valid()` returns `true`.
3. Serialize the struct via `Streamable` (`to_bytes()`) and send it to a victim node.
4. The victim deserializes it with `ProofOfInclusion.from_bytes()` and calls `proof.valid()` — which returns `true`.
5. The victim never checks `proof.root_hash() == R`, so the forged inclusion is accepted.

This allows untrusted input to prove invalid DataLayer state, matching the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."**

### Likelihood Explanation

The `valid()` method name strongly implies complete proof validation. The API provides no single method that checks both internal consistency and root binding. The fuzz target, all Rust unit tests, and all Python integration tests demonstrate the idiomatic usage pattern of calling only `proof.valid()`. Any DataLayer client that follows this pattern is vulnerable. The `ProofOfInclusion` struct is `Streamable` and fully attacker-controlled over the network.

### Recommendation

1. **Add a root-binding check to `valid()`** — change the signature to `valid(&self, expected_root: &Hash) -> bool` and compare `self.root_hash() == *expected_root` as the final step, or add a separate `valid_for_root(&self, expected_root: &Hash) -> bool` method.
2. **Remove or rename the current `valid()`** to something like `internally_consistent()` to prevent misuse.
3. **Update the Python binding** to expose the root-binding variant as the primary validation entry point.
4. **Fix the fuzz target and all tests** to pass the expected root hash into the validation call.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer, MerkleBlob
from chia_rs.sized_bytes import bytes32
from chia_rs.sized_ints import uint8

# Real tree with one key
blob = MerkleBlob(bytearray())
real_key_id = ...
blob.insert(real_key_id, ...)
blob.calculate_lazy_hashes()
real_root = blob.get_root()

# Attacker forges a proof for a fake key not in the tree.
# Pick any node_hash (e.g. hash of the fake key's data).
fake_node_hash = bytes32(b'\xaa' * 32)
sibling_hash   = bytes32(b'\xbb' * 32)

# Compute a valid combined_hash so the chain is internally consistent.
import hashlib
combined = hashlib.sha256(fake_node_hash + sibling_hash).digest()

forged_proof = ProofOfInclusion(
    node_hash=fake_node_hash,
    layers=[
        ProofOfInclusionLayer(
            other_hash_side=uint8(1),
            other_hash=sibling_hash,
            combined_hash=bytes32(combined),
        )
    ],
)

# valid() returns True even though this proof has nothing to do with real_root.
assert forged_proof.valid()                        # True  ← bug
assert forged_proof.root_hash() != real_root       # True  ← different root, never checked
```

    pub fn valid(&self) -> bool {
        let mut existing_hash = self.node_hash;

        for layer in &self.layers {
            let calculated_hash = crate::calculate_internal_hash(
                &existing_hash,
                layer.other_hash_side,
                &layer.other_hash,
            );

            if calculated_hash != layer.combined_hash {
                return false;
            }

            existing_hash = calculated_hash;
        }

        existing_hash == self.root_hash()
    }
