### Title
Unattributed router ETH balance allows any caller to steal stranded native ETH via `refundETH()` or receive a subsidized WETH-input swap via `pay()` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` uses the router's entire `address(this).balance` when settling a WETH-input swap, with no per-user attribution. `refundETH()` is a public function that unconditionally transfers the router's full ETH balance to `msg.sender`. Any ETH stranded on the router from a prior user's transaction is either silently consumed to subsidize a subsequent user's swap, or directly stolen by any caller of `refundETH()`.

---

### Finding Description

`PeripheryPayments.pay()` implements a native-ETH-first WETH payment path:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← entire router balance, not caller's ETH
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

`address(this).balance` is the router's **global** native ETH balance — it is never scoped to the current caller. ETH is stranded on the router whenever a user calls `exactInputSingle` or `exactOutputSingle` with `msg.value > amountIn` and omits `refundETH()` from their multicall. The test suite itself documents this pattern as the expected usage: [2](#0-1) 

Once ETH is stranded, two theft paths exist:

**Path 1 — Direct theft via `refundETH()`:**

```solidity
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // ← sends entire balance to any caller
    }
}
``` [3](#0-2) 

`refundETH()` is `external payable` with no access control and no per-user accounting. Any address can call it at any time and receive the full ETH balance of the router.

**Path 2 — Indirect theft via `pay()` subsidy:**

A second user calls `exactInputSingle` with WETH as `tokenIn` and `msg.value = 0`. Inside the swap callback, `pay()` reads `nativeBalance = address(this).balance` (User A's stranded ETH), wraps it, and transfers it to the pool on behalf of User B. User B's swap is fully or partially funded by User A's ETH without User B providing any ETH or WETH allowance. [4](#0-3) 

The same unattributed-balance pattern applies to `sweepToken()` and `unwrapWETH9()`, which also send the router's entire token/WETH balance to any caller-specified `recipient`: [5](#0-4) 

---

### Impact Explanation

Direct loss of user principal (native ETH). A user who sends excess ETH with a WETH-input swap and does not atomically include `refundETH()` in the same multicall permanently loses the excess ETH to any subsequent caller. The loss is unbounded — it equals the full excess `msg.value` sent. This is a High-severity impact under Sherlock thresholds (direct loss of user funds, no minimum threshold for ETH).

---

### Likelihood Explanation

Medium-High. The intended usage pattern (documented in tests and comments) requires users to compose `exactInput*` + `refundETH()` in a single multicall. Any user who calls `exactInputSingle` or `exactOutputSingle` directly with excess ETH — a natural pattern when the exact swap cost is unknown — leaves ETH stranded. A MEV bot monitoring the mempool can front-run the user's `refundETH()` follow-up call, or simply call `refundETH()` after the swap transaction confirms. No special privileges or setup are required.

---

### Recommendation

1. **Track per-call ETH in transient storage.** At the entry of each `exactInput*`/`exactOutput*` function, record `msg.value` in a transient slot. In `pay()`, consume only up to that recorded amount rather than `address(this).balance`. Zero the slot before returning.

2. **Restrict `refundETH()` to the original multicall initiator.** Store `msg.sender` in transient storage at the start of `multicall` and require it in `refundETH()`, or make `refundETH()` internal and only callable from within a multicall context.

3. **Alternatively**, enforce that `refundETH()` is always the last call in any multicall that includes a WETH-input swap, and document this as a hard invariant with an on-chain guard.

---

### Proof of Concept

```
Setup:
  - Router deployed with WETH and factory.
  - Pool: WETH / Token1, registered on factory.
  - User A: has 2 ETH, wants to swap 1000 wei of WETH for Token1.
  - User B: attacker, has 0 ETH.

Step 1 — User A swaps with excess ETH (no refundETH):
  router.exactInputSingle{value: 2 ether}(ExactInputSingleParams{
      pool: pool,
      tokenIn: WETH,
      amountIn: 1000,   // only 1000 wei consumed
      ...
  });
  // pay() wraps 1000 wei, sends to pool.
  // 2 ether - 1000 wei remains on router.

Step 2 — User B steals stranded ETH:
  vm.prank(attacker);
  router.refundETH();
  // attacker receives ~2 ether - 1000 wei. User A loses it.

Alternate Step 2 — User B gets a free swap:
  router.exactInputSingle{value: 0}(ExactInputSingleParams{
      pool: pool,
      tokenIn: WETH,
      amountIn: 1000,
      ...
  });
  // pay() sees nativeBalance = ~2 ether >= 1000, wraps User A's ETH,
  // pays pool. User B receives Token1 without spending any ETH or WETH.
```

The test `test_refundETH_sendsBalanceToCaller` already demonstrates that any caller receives the full router ETH balance — it uses `vm.deal(address(router), amount)` to simulate stranded ETH and confirms the caller receives it all: [6](#0-5)
