### Title
Nonce Corruption via Reentrancy in `initTransfer()` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer()` increments `currentOriginNonce` before an external `safeTransferFrom` call, but then reads `currentOriginNonce` from storage again when passing it to `initTransferExtension()` and `emit`. A reentrant call during the token transfer (e.g., via an ERC-777 `tokensToSend` hook) causes the outer call to publish the wrong nonce, producing two cross-chain messages with the same `originNonce` while skipping one nonce entirely. This permanently locks the outer caller's tokens in the bridge with no claimable counterpart on NEAR.

---

### Finding Description

In `OmniBridge.initTransfer()`, the execution order is:

1. `currentOriginNonce += 1` — storage variable incremented to `N` [1](#0-0) 
2. External call: `IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount)` — can invoke a `tokensToSend` hook on the sender for ERC-777 tokens [2](#0-1) 
3. `initTransferExtension(..., currentOriginNonce, ...)` — reads `currentOriginNonce` from storage **again** [3](#0-2) 
4. `emit BridgeTypes.InitTransfer(..., currentOriginNonce, ...)` — reads `currentOriginNonce` from storage **again** [4](#0-3) 

The nonce is never captured into a local variable before the external call. If a reentrant call to `initTransfer` occurs during step 2, the storage variable is incremented a second time (to `N+1`) before the outer call reads it in steps 3 and 4.

In `OmniBridgeWormhole`, `initTransferExtension` encodes `originNonce` directly into the Wormhole payload and publishes it: [5](#0-4) 

The result of a successful reentrant attack:

| Call | Nonce incremented to | Nonce used in message |
|---|---|---|
| Inner (reentrant) | N+1 | N+1 |
| Outer (original) | — | N+1 (stale read) |

- Nonce `N` is **never published** to Wormhole and never seen by NEAR.
- Nonce `N+1` is published **twice** with different transfer parameters.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

The NEAR bridge tracks `origin_nonce` to deduplicate incoming transfers. When two distinct Wormhole messages arrive carrying the same `originNonce = N+1`:

- The first message is finalized on NEAR normally.
- The second message is rejected as a duplicate nonce.
- The tokens locked in the EVM bridge by the outer call have **no claimable NEAR-side transfer** and are permanently irrecoverable.

Additionally, nonce `N` is permanently skipped. If the NEAR prover or relayer infrastructure enforces sequential nonce processing (common for ordered message queues), all subsequent transfers with nonces `> N` may be stalled until `N` is processed — causing a **bridge-wide freeze**.

---

### Likelihood Explanation

**Medium.** The bridge accepts arbitrary ERC-20 tokens for the non-bridge-token path (line 407–411). ERC-777 tokens are a standardized extension of ERC-20 and are deployed on mainnet (e.g., imBTC, LUKSO LYX). Any ERC-777 token holder can trigger the `tokensToSend` hook on themselves during `safeTransferFrom`, giving them full control of execution mid-function. No privileged role is required — any unprivileged token holder calling `initTransfer` with an ERC-777 token can execute this attack. [6](#0-5) 

---

### Recommendation

Capture `currentOriginNonce` into a local variable **before** any external call, and use only the local variable thereafter:

```solidity
function initTransfer(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    uint64 originNonce = currentOriginNonce; // capture before external calls

    ...
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    ...

    initTransferExtension(msg.sender, tokenAddress, originNonce, ...);
    emit BridgeTypes.InitTransfer(msg.sender, tokenAddress, originNonce, ...);
}
```

Alternatively, add OpenZeppelin's `ReentrancyGuard` (`nonReentrant` modifier) to `initTransfer`, consistent with the pattern already used in `SherlockClaimManager` in the reference report.

---

### Proof of Concept

1. Attacker deploys an ERC-777 token `AttackToken` and registers a `tokensToSend` hook on their own address that calls `initTransfer` again (with a small amount of `AttackToken`).
2. Attacker calls `initTransfer(AttackToken, amount=100, ...)`:
   - `currentOriginNonce` incremented: `N → N+1` (storage = `N+1`).
   - `safeTransferFrom` is called; ERC-777 fires `tokensToSend` on the attacker.
   - **Reentrant call**: `initTransfer(AttackToken, amount=1, ...)`:
     - `currentOriginNonce` incremented: `N+1 → N+2` (storage = `N+2`).
     - `safeTransferFrom` completes (no further hook).
     - `initTransferExtension(..., currentOriginNonce=N+2, ...)` → Wormhole message published with `originNonce=N+2`.
     - `emit InitTransfer(..., currentOriginNonce=N+2, ...)`.
   - Outer call resumes.
   - `initTransferExtension(..., currentOriginNonce=N+2, ...)` → **second** Wormhole message published with `originNonce=N+2`.
   - `emit InitTransfer(..., currentOriginNonce=N+2, ...)`.
3. Result: nonce `N+1` is skipped; nonce `N+2` is published twice with different amounts (100 and 1). The outer transfer's 100 tokens are locked in the bridge with no valid NEAR claim. [7](#0-6)
