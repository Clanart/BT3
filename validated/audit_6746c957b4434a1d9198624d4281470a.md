### Title
Transfer Amount Exceeding `u64::MAX` Causes Permanent Fund Lock on Solana Finalization - (`solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

---

### Summary

The Solana bridge's `finalize_transfer` and `finalize_transfer_sol` instructions receive a `FinalizeTransferPayload` where `amount` is typed as `u128`, but SPL token and SOL transfers require a `u64`. The conversion `data.amount.try_into()` returns `AmountOverflow` if the amount exceeds `u64::MAX`. Because Solana transactions are atomic, the nonce is not permanently consumed on failure — but the source-chain funds (locked/burned on NEAR or EVM) have no recovery path. The NEAR `sign_transfer` function imposes no upper-bound check on the normalized amount before signing, so a user can initiate a transfer whose normalized amount exceeds `u64::MAX`, permanently locking their tokens.

---

### Finding Description

**NEAR `sign_transfer` — no upper-bound guard on normalized amount**

`sign_transfer` normalizes the transfer amount and embeds it as `U128` in the signed payload:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,          // only lower-bound check
    BridgeError::InvalidAmountToTransfer.as_ref()
);

let transfer_payload = TransferMessagePayload {
    ...
    amount: U128(amount_to_transfer),   // u128, no u64 ceiling
    ...
};
```

There is no `require!(amount_to_transfer <= u64::MAX as u128, ...)` guard.

**Solana `finalize_transfer` — hard `u64` ceiling on token operations**

Both SPL-token and SOL transfer paths perform a fallible narrowing cast:

```rust
// finalize_transfer.rs line 114 (native vault path)
data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?

// finalize_transfer.rs line 134 (bridged mint path)
data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?

// finalize_transfer_sol.rs line 88
data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?
```

If `data.amount > u64::MAX`, the instruction returns `AmountOverflow` and the entire Solana transaction is rolled back. The destination nonce is **not** permanently marked as used, so the relayer can retry — but every retry will fail identically.

**No source-chain recovery path**

On NEAR, the only way to remove a transfer from `pending_transfers` is via `claim_fee_callback`, which requires a successful Solana finalization (a Wormhole `FinTransfer` proof). There is no `cancel_transfer` or admin escape hatch. The tokens locked/burned during `init_transfer_internal` are irrecoverable.

---

### Impact Explanation

A user who bridges a normalized amount exceeding `u64::MAX` to Solana will have their tokens permanently locked in the NEAR bridge contract (or burned, for deployed tokens). The transfer record persists in `pending_transfers` indefinitely. No relayer can ever produce a valid `claim_fee` proof because the Solana finalization will always revert. This matches the allowed critical impact: **permanent freezing / irrecoverable lock of user funds in bridge flows**.

---

### Likelihood Explanation

- NEAR and EVM both use `u128` / `uint128` for amounts; no ceiling is enforced at `init_transfer`.
- `sign_transfer` only checks `amount_to_transfer > 0`, not `amount_to_transfer ≤ u64::MAX`.
- For a token with equal decimals on NEAR and Solana (e.g., 9 on both), a user must transfer more than `u64::MAX` ≈ 1.84 × 10¹⁹ smallest units (≈ 18 billion whole tokens with 9 decimals). For high-supply tokens (meme coins, governance tokens) this threshold is reachable.
- For tokens with more decimals on NEAR than Solana (e.g., 18 vs. 9), `normalize_amount` divides by 10⁹, raising the threshold to ≈ 1.84 × 10²⁸ smallest NEAR units — still reachable for tokens with very large supplies.
- The scenario can occur accidentally (a whale bridging a large position) or deliberately (a user intentionally locking funds to grief the bridge's accounting).

---

### Recommendation

Add an explicit upper-bound check in `sign_transfer` before signing the payload, rejecting transfers whose normalized amount exceeds `u64::MAX` when the destination chain is Solana:

```rust
if transfer_message.recipient.get_chain() == ChainKind::Sol {
    require!(
        amount_to_transfer <= u64::MAX as u128,
        BridgeError::InvalidAmountToTransfer.as_ref()
    );
}
```

This causes `sign_transfer` to fail early (before tokens are locked), giving the user a clear error and preserving their funds. Alternatively, add a corresponding check in `init_transfer` so the transfer is rejected before tokens are ever locked.

---

### Proof of Concept

1. Register a token with 9 decimals on both NEAR and Solana.
2. Mint `u64::MAX + 1` = `18_446_744_073_709_551_616` smallest units to a NEAR user.
3. User calls `ft_transfer_call` → `init_transfer` with `amount = 18_446_744_073_709_551_616` and recipient on Solana. Tokens are locked; transfer stored in `pending_transfers`.
4. Relayer calls `sign_transfer`. `normalize_amount` returns `18_446_744_073_709_551_616` (no decimal difference). Only `> 0` check passes. MPC signs the payload with `amount = 18_446_744_073_709_551_616` (u128).
5. Relayer calls `finalize_transfer` on Solana with the signed payload.
6. `data.amount.try_into::<u64>()` fails → `AmountOverflow` → transaction rolled back.
7. Relayer retries indefinitely — same result every time.
8. `claim_fee` on NEAR can never succeed (no Wormhole proof exists).
9. User's `18_446_744_073_709_551_616` tokens are permanently locked in the NEAR bridge.
