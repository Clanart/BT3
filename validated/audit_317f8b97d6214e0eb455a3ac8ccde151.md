### Title
Missing Constraint in `check_is_reverted` Allows Prover to Arbitrarily Skip Transaction Execution While Charging Fees — (`File: execution/execution_constraints.cairo`)

### Summary

The `check_is_reverted` function in `execution_constraints.cairo` is completely empty — it adds zero Cairo constraints on the hint-loaded `is_reverted` value. Because the Cairo OS is a ZK proof system where the prover (sequencer) must not be trusted, every hint-derived value that controls execution flow must be constrained by the Cairo program itself. The absence of any constraint means the prover can freely set `is_reverted = 1` for any invoke transaction, causing execution to be skipped while fees are still charged, or set `is_reverted = 1` for any L1 handler, causing the L1-to-L2 message to not be consumed and permanently freezing the associated funds.

### Finding Description

In `execution_constraints.cairo`, the function that is supposed to validate the `is_reverted` hint is a no-op:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [1](#0-0) 

This function is called in `execute_invoke_function_transaction` after loading `is_reverted` from a hint:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...);
} else {
    tempvar _dummy_return_value: non_reverting_select_execute_entry_point_func.Return;
}
// Charge fee — always runs regardless of is_reverted.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [2](#0-1) 

Because `check_is_reverted` adds no constraints, `is_reverted` is a free variable. The prover can assign it any value. If `is_reverted = 1`, the `__execute__` entry point is skipped entirely, but `charge_fee` still runs unconditionally, transferring the user's fee to the sequencer with no corresponding execution.

The situation is even more severe for `execute_l1_handler_transaction`, where `check_is_reverted` is not even called — the hint value is used directly to gate `consume_l1_to_l2_message`:

```cairo
local is_reverted;
%{ IsReverted %}
// Skip the execution step for reverted transaction.
if (is_reverted != FALSE) {
    %{ EndTx %}
    return ();
}
...
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
``` [3](#0-2) 

If `is_reverted = 1` for an L1 handler, `consume_l1_to_l2_message` is never called, so the L1-to-L2 message is never recorded in the OS output. The L1 contract has already locked or burned the associated funds; they cannot be reclaimed.

This is the direct analog to M-03: just as `KangarooVault` imports `PauseModifier` but never applies `whenNotPaused` to its critical functions (while `LiquidityPool` does), the OS defines `check_is_reverted` and calls it on invoke transactions, but the function body is empty — making the "check" a dead letter — and omits it entirely for L1 handlers.

### Impact Explanation

**Critical — Direct loss of funds**: For any invoke transaction, the prover can set `is_reverted = 1`. Execution is skipped, but `charge_fee` still executes an ERC-20 transfer from the user's account to the sequencer. The user pays a fee and receives nothing.

**Critical — Permanent freezing of funds**: For any L1 handler transaction, the prover can set `is_reverted = 1`. The L1-to-L2 message is not consumed in the OS output. The ETH or tokens locked on L1 by the originating L1 contract cannot be recovered, because the L1 contract's message-cancellation window has passed or the contract has no cancellation path.

### Likelihood Explanation

The StarkNet OS is a ZK proof system. Its security guarantee is that the proof is valid if and only if the Cairo constraints are satisfied. The prover (sequencer) is explicitly not trusted — the constraints are the sole source of truth. Because `check_is_reverted` adds no constraints, the prover can freely choose `is_reverted` for every transaction in every block. Any sequencer operator (including a compromised or economically incentivized one) can exploit this without detection, since the resulting proof is valid by construction.

### Recommendation

Implement `check_is_reverted` to add a real Cairo constraint that ties `is_reverted` to the actual execution outcome. One approach is to derive `is_reverted` from the return value of `select_execute_entry_point_func` rather than from a free hint, and assert consistency. Alternatively, if the hint-based approach is retained for performance, add a range check asserting `is_reverted * (1 - is_reverted) = 0` (boolean) and a constraint that links `is_reverted` to observable state changes (e.g., the revert log being non-empty). Additionally, apply the same constraint call to `execute_l1_handler_transaction`.

### Proof of Concept

1. A user submits a valid invoke transaction `T` with a non-zero fee bound.
2. The sequencer includes `T` in a block and, when building the Cairo execution trace, sets the `IsReverted` hint to `1` for `T`.
3. The Cairo program calls `check_is_reverted(1)`, which returns immediately with no assertion.
4. The `if (is_reverted == FALSE)` branch is not taken; `__execute__` is never called.
5. `charge_fee` runs unconditionally, transferring the user's fee to the sequencer.
6. The proof is generated and verified successfully — all Cairo constraints are satisfied.
7. The user's account balance is reduced by the fee; the intended state change never occurs.

For the L1 handler variant:
1. A user sends an L1-to-L2 message (e.g., a deposit) from an L1 contract, locking ETH.
2. The sequencer sets `IsReverted = 1` for the corresponding L1 handler transaction.
3. `consume_l1_to_l2_message` is never called; the message does not appear in the OS output.
4. The proof is valid. The L1 contract's funds are permanently frozen with no recourse.
