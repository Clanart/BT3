### Title
Missing Cross-Validation Between Caller-Supplied `chain_kind` and Proof-Embedded Chain — (`near/omni-bridge/src/lib.rs`)

---

### Summary

Every proof-consuming entry point in the NEAR bridge (`fin_transfer`, `claim_fee`, `deploy_token`, `bind_token`) accepts a caller-controlled `chain_kind` that selects which registered prover to invoke, and a separate `prover_args` blob whose decoded content carries its own chain identity (`emitter_address.get_chain()`). The callback that processes the prover's result validates the emitter against the factory registry using the chain embedded **inside the proof**, never against the `chain_kind` the caller used to pick the prover. There is no assertion that the two agree.

---

### Finding Description

`verify_proof` selects the prover exclusively from the caller-supplied `chain_kind`: [1](#0-0) 

All four public entry points pass the caller's `chain_kind` directly into `verify_proof`: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

Each callback then derives the chain from the proof content and validates the factory against **that** chain — never against the `chain_kind` argument:

`fin_transfer_callback`: [6](#0-5) 

`claim_fee_callback`: [7](#0-6) 

`deploy_token_callback`: [8](#0-7) 

`bind_token_callback`: [9](#0-8) 

The missing invariant in every callback is:

```
require!(args.chain_kind == proof_result.emitter_address.get_chain())
```

This is structurally identical to the oracle mismatch in NFTPairWithOracle: one party (the bridge protocol) registered a prover for a specific chain, but the caller can present a proof whose embedded chain identity differs from the `chain_kind` used to select the prover, and the bridge never checks that they agree.

---

### Impact Explanation

If a prover for chain A can be made to return a `ProverResult` whose `emitter_address` belongs to chain B (e.g., because the prover is generic/MPC-based and does not enforce chain identity in the signed payload, or because two EVM-family chains share a compatible proof format), an attacker can:

- **`fin_transfer`**: finalize a transfer attributed to chain B while routing it through chain A's prover, bypassing chain B's proof security guarantees and potentially minting unbacked tokens.
- **`claim_fee`**: claim fees against a `TransferMessage` from chain B using chain A's prover.
- **`deploy_token` / `bind_token`**: register a token mapping attributed to chain B using chain A's prover, corrupting the token registry.

The impact class is: **Proof/signature verification bypass enabling unauthorized transfer finalization or token deployment** (High/Critical per the allowed scope).

---

### Likelihood Explanation

The attack is reachable by any unprivileged user who can call `fin_transfer`, `claim_fee`, `deploy_token`, or `bind_token`. The practical exploitability depends on whether any registered prover is generic enough (e.g., an MPC-based prover that signs arbitrary payloads without enforcing chain identity) or whether two chains share a compatible proof format (e.g., multiple EVM chains). The structural gap is present unconditionally; exploitability scales with prover permissiveness.

---

### Recommendation

Add an explicit assertion in each callback that the chain embedded in the proof result matches the `chain_kind` the caller supplied:

```rust
// In fin_transfer_callback, claim_fee_callback, deploy_token_callback, bind_token_callback:
require!(
    proof_result.emitter_address.get_chain() == expected_chain_kind,
    BridgeError::ChainKindMismatch.as_ref()
);
```

Because `chain_kind` is passed through the async callback chain, it should be included in the callback arguments (analogous to how `attached_deposit` is already threaded through). Alternatively, the prover interface itself should be required to attest to the chain it verified, and the bridge should assert that attestation matches the registered prover's chain.

---

### Proof of Concept

1. Attacker calls `fin_transfer` with `chain_kind: ChainKind::Eth` and `prover_args` crafted so that the Eth prover (or a generic MPC prover registered under `Eth`) returns a `ProverResult::InitTransfer` whose `emitter_address` is `sol:<solana_factory_address>`.
2. `verify_proof` dispatches to the Eth prover. The prover verifies the proof and returns the result.
3. `fin_transfer_callback` decodes the result, reads `init_transfer.emitter_address.get_chain()` → `ChainKind::Sol`, and checks `self.factories.get(&Sol) == Some(solana_factory)` → passes.
4. The transfer is finalized as if it originated from Solana, but was verified only by the Eth prover — bypassing Solana's proof security entirely.
5. No check anywhere asserts `ChainKind::Eth == ChainKind::Sol`.
