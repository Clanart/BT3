### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS program accepts an arbitrary `class_hash` from the syscall request and unconditionally writes it into the contract state without verifying that the supplied hash corresponds to a previously declared class in `contract_class_changes`. This is the direct analog of the reported "Missing Rebalance Checks" vulnerability: just as `start_rebalance` failed to verify the destination mint account that `end_rebalance` would use, `execute_replace_class` fails to verify the destination class hash before committing it to state. The missing check is explicitly acknowledged by a TODO comment in the production code.

---

### Finding Description

In `syscall_impls.cairo`, `execute_replace_class` reads `class_hash` from the raw syscall request and immediately writes it into `contract_state_changes` with no validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The function signature does not receive `contract_class_changes` as an implicit argument, making it structurally impossible to perform the required lookup: [2](#0-1) 

The same omission exists in the deprecated path (`deprecated_execute_syscalls.cairo`), where `execute_replace_class` also performs no class-existence check: [3](#0-2) 

The protocol invariant that should be enforced is: every `class_hash` stored in `contract_state_changes` must have a corresponding entry in `contract_class_changes` (i.e., it must be a declared class). The `execute_replace_class` function breaks this invariant by accepting any arbitrary felt value as the new class hash.

For comparison, the `execute_declare_transaction` path correctly enforces that a class hash is valid by verifying the Sierra component hashes and writing the mapping into `contract_class_changes`: [4](#0-3) 

No equivalent check exists in `execute_replace_class`.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

If a contract calls `replace_class(undeclared_hash)`, the OS writes `undeclared_hash` into the contract's `StateEntry.class_hash`. Every subsequent call to that contract will attempt to dispatch to a class that does not exist in the class tree, causing all executions to fail unconditionally. Any ERC-20 tokens, ETH, or other assets held in the contract's storage become permanently inaccessible — the contract can never execute again, and there is no recovery path because the class hash is now invalid and the contract cannot call `replace_class` again to fix itself (since execution itself fails).

---

### Likelihood Explanation

The likelihood is **high**:

1. The TODO comment at line 898 explicitly confirms the check is missing and was known to the developers.
2. The `replace_class` syscall is reachable by any unprivileged contract caller — no privileged role is required.
3. A malicious contract author can deploy a contract, attract user funds (e.g., as a fake vault or token), and then call `replace_class` with an arbitrary undeclared hash to permanently freeze all deposited funds.
4. A legitimate contract with a reentrancy or access-control bug could be exploited by an attacker to trigger `replace_class` with an attacker-supplied class hash, freezing the victim contract's funds.
5. The deprecated path (`deprecated_execute_syscalls.cairo`) has the same omission, widening the attack surface.

---

### Recommendation

Add a check inside `execute_replace_class` that verifies the supplied `class_hash` exists in `contract_class_changes` before committing the state update. Concretely:

1. Add `contract_class_changes: DictAccess*` as an implicit argument to `execute_replace_class`.
2. Perform a `dict_read` on `contract_class_changes` with `key=class_hash` and assert the returned value is non-zero (i.e., a compiled class hash has been registered for this class hash).
3. Apply the same fix to the deprecated `execute_replace_class` in `deprecated_execute_syscalls.cairo`.
4. Add an analogous check in `execute_deploy` / `deploy_contract` for the `request.class_hash` used during deployment.

---

### Proof of Concept

1. Attacker deploys contract `VaultContract` with a valid class hash `C_valid`. Users deposit funds into `VaultContract`.
2. Attacker calls any entry point on `VaultContract` that internally invokes the `replace_class` syscall with `class_hash = 0xdeadbeef` (an arbitrary undeclared felt).
3. The OS executes `execute_replace_class`:
   - `class_hash = 0xdeadbeef` is read from the request.
   - No check against `contract_class_changes` is performed (the TODO at line 898 is the missing guard).
   - `dict_update` writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`.
4. State commitment proceeds: `compute_contract_state_commitment` hashes the new `StateEntry` (including `class_hash=0xdeadbeef`) into the Merkle tree. The class tree has no entry for `0xdeadbeef`, but the contract state tree now references it.
5. All subsequent calls to `VaultContract` fail at class dispatch — the class `0xdeadbeef` does not exist. All user funds are permanently frozen. [5](#0-4) [6](#0-5)
