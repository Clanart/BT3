### Title
Felt-Field Arithmetic Overflow in `compute_max_possible_fee` Allows Fee-Free Transaction Execution — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` performs unchecked felt-field multiplications of user-controlled resource-bound fields. Because Cairo arithmetic is modulo the Stark field prime P, a transaction sender can choose `max_price_per_unit` values (bounded only by `assert_nn`, i.e. `< P/2`) and `max_amount` values (bounded to `2^64 - 1`) such that the entire sum wraps to **0 mod P**. `charge_fee` then hits its early-return guard `if (max_fee == 0) { return (); }` and skips the ERC-20 fee transfer entirely, letting the transaction execute without paying any fee.

---

### Finding Description

`compute_max_possible_fee` computes:

```
max_fee = l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
        + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
        + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit
``` [1](#0-0) 

All arithmetic is native felt arithmetic (mod P). The only enforced bounds on the inputs come from `pack_resource_bounds` and `hash_fee_fields`:

- `max_amount` ≤ `2^64 - 1` (via `assert_nn_le`)
- `max_price_per_unit` ∈ `[0, P/2)` (via `assert_nn` only — **no upper bound tighter than P/2**)
- `tip` ≤ `2^64 - 1` (via `assert_nn_le`) [2](#0-1) [3](#0-2) 

A product such as `max_amount * max_price_per_unit` can reach `(2^64 - 1) × (P/2 - 1) ≈ 2^63 × P`, which is many multiples of P. Modulo P the result can be any value in `[0, P)`. A sender can therefore choose three pairs of `(max_amount, max_price_per_unit)` values — all individually passing their range checks — such that the three terms sum to exactly **0 mod P**.

`charge_fee` then evaluates:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();          // ← fee transfer is skipped entirely
}
``` [4](#0-3) 

The ERC-20 transfer to the sequencer is never executed. The `assert_nn_le(calldata.amount.low, max_fee)` guard that would otherwise cap the actual fee is never reached. [5](#0-4) 

Note that `non_reverting_select_execute_entry_point_func` (used for the actual fee transfer) asserts `is_reverted = 0`, so a *reverting* fee transfer would invalidate the proof. The overflow path is different: it bypasses the transfer call entirely before that assertion is ever reached. [6](#0-5) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

A transaction sender who crafts resource bounds that cause `compute_max_possible_fee` to return 0 can have their transaction fully executed (state changes committed, L1/L2 messages emitted) without transferring any fee tokens to the sequencer. Because the OS proof is the authoritative enforcer of fee payment, a valid proof containing such a transaction irrefutably demonstrates that no fee was owed — the sequencer has no recourse. Repeated exploitation drains sequencer revenue and breaks the economic security of the fee market.

---

### Likelihood Explanation

**Medium-High.** The attacker controls all three `(max_amount, max_price_per_unit)` pairs directly in the signed transaction. Finding a combination that sums to 0 mod P is a straightforward modular arithmetic problem (one degree of freedom is sufficient: fix five of the six values and solve for the sixth, subject only to `assert_nn` and `assert_nn_le` constraints). No privileged access, leaked key, or external dependency is required. The only gate is sequencer inclusion, but a sequencer that does not independently re-derive and check `compute_max_possible_fee` for overflow will include the transaction.

---

### Recommendation

Replace the unchecked felt multiplications in `compute_max_possible_fee` with overflow-safe arithmetic. Concretely:

1. Add tight upper-bound checks on `max_price_per_unit` (e.g. `assert_nn_le(max_price_per_unit, 2**128 - 1)`) inside `pack_resource_bounds` or at the start of `compute_max_possible_fee`, so that each product `max_amount * max_price_per_unit` is at most `(2^64 - 1)(2^128 - 1) < 2^192 < P` and cannot overflow.
2. Alternatively, compute the fee using `Uint256` arithmetic and verify the result fits in a felt before using it.
3. Add a protocol-level assertion that `max_fee > 0` is required for any transaction that declares non-zero resource bounds, rejecting the transaction at hash-verification time rather than silently skipping the fee.

---

### Proof of Concept

**Setup:** Stark field prime `P = 2^251 + 17·2^192 + 1`.

**Goal:** find `(a1, p1, a2, p2, a3, p3)` with `a_i ≤ 2^64-1`, `p_i < P/2`, `tip ≤ 2^64-1` such that:

```
a1·p1 + a2·(p2 + tip) + a3·p3 ≡ 0  (mod P)
```

**Construction (simplified):**
- Set `tip = 0`, `a1 = a2 = a3 = 1`.
- Choose `p1 = k`, `p2 = m`, `p3 = P - k - m` for any `k, m` with `k + m < P/2` and `P - k - m < P/2` (i.e. `k + m > P/2`).
- Then `p1 + p2 + p3 = k + m + (P - k - m) = P ≡ 0 (mod P)`. ✓
- All `p_i < P/2` satisfies `assert_nn`. ✓
- All `a_i = 1 ≤ 2^64 - 1` satisfies `assert_nn_le`. ✓

The attacker submits this transaction. The OS computes `max_fee = 0`, enters the early-return branch in `charge_fee`, and processes the transaction without any fee transfer.

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
