### Title
Missing `state_diff_commitment` Binding Check in P2P Sync Allows Malicious Peer to Corrupt `compiled_class_hash_table` - (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary

`parse_data_for_block` in the P2P sync client assembles a `ThinStateDiff` from peer-supplied chunks and validates only the total length against `header.state_diff_length`. It never computes `calculate_state_diff_hash(result)` and never compares it to `header.state_diff_commitment`. A malicious peer can therefore supply a syntactically valid state diff of the correct length but with a wrong `class_hash_to_compiled_class_hash` mapping, which is written verbatim into `compiled_class_hash_table` via `write_compiled_class_hashes`. The stored header's `state_diff_commitment` field is itself also peer-supplied and never signature-verified in the header parsing path, so the peer can make the two values consistent with each other while both being wrong relative to the canonical chain.

### Finding Description

**Entrypoint and call chain:**

`parse_data_for_block` (state_diff.rs, lines 51–110) → `write_to_storage` (state_diff.rs, lines 28–39) → `append_state_diff` (state/mod.rs, lines 516–590) → `write_compiled_class_hashes` (state/mod.rs, lines 801–811) → `compiled_class_hash_table.insert((H, N), C1)`

**Missing guard in `parse_data_for_block`:**

The function reads `target_state_diff_len` from the stored header and accumulates chunks until `current_state_diff_len == target_state_diff_len`. The only content checks are duplicate-key detection (`ConflictingStateDiffParts`) and duplicate deprecated class detection. There is no call to `calculate_state_diff_hash(&result)` and no comparison against `header.state_diff_commitment`. [1](#0-0) 

**Missing guard in header parsing:**

`parse_data_for_block` for headers (header.rs, lines 82–123) checks only block number ordering and that `signatures.len() == 1`. It does not call `verify_block_signature` (which exists in `starknet_api/src/block.rs` lines 717–730) and does not verify the signature against any sequencer public key. The `state_diff_commitment` field in the stored header is therefore fully peer-controlled. [2](#0-1) 

**`write_compiled_class_hashes` inserts without verification:** [3](#0-2) 

**`append_state_diff` calls it unconditionally:** [4](#0-3) 

**The `verify_state_diff_hash` guard exists only in the committer, not in the p2p sync path:** [5](#0-4) 

### Impact Explanation

After the corrupt state diff is stored, `get_compiled_class_hash_at(StateNumber::right_after_block(N), H)` returns C1 instead of the canonical C2. Every downstream consumer that reads this value is affected:

- **Blockifier / execution**: selects the wrong CASM artifact for class H; all transactions invoking class H produce wrong results.
- **RPC `starknet_getClass` / `starknet_getClassAt`**: returns the wrong compiled class hash.
- **Transaction prover / committer**: `insert_compiled_class_leaves` in `committer_utils.rs` (lines 218–229) reads `initial_reads.compiled_class_hashes` which is populated from the corrupted storage, producing a wrong class trie leaf and therefore a wrong global root. [6](#0-5) 

This matches the allowed impact: **"Critical. Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution."**

### Likelihood Explanation

Any peer the p2p sync client connects to can execute this attack. No operator or validator privilege is required. The attacker only needs to be a reachable peer that the syncing node selects. The attack is silent: the node stores the wrong value, advances its state marker, and continues syncing normally. There is no error, no disconnect, and no retry that would correct the stored value.

### Recommendation

In `parse_data_for_block` for state diffs, after assembling the full `ThinStateDiff`, compute `calculate_state_diff_hash(&result)` and compare it to `header.state_diff_commitment` read from storage. Return `ParseDataError::BadPeer` if they differ. Additionally, verify the block signature in the header parsing path using `verify_block_signature` against the known sequencer public key before storing the header, so that `state_diff_commitment` itself cannot be forged.

### Proof of Concept

Concrete corrupted value: `compiled_class_hash_table[(H, N)] = C1` where the canonical value is C2.

A Rust integration test would:
1. Store a header for block N with `state_diff_commitment = calculate_state_diff_hash({H: C1})` and `state_diff_length = 1`.
2. Call `append_state_diff(N, ThinStateDiff { class_hash_to_compiled_class_hash: {H: C1}, .. })`.
3. Assert `get_compiled_class_hash_at(StateNumber::right_after_block(N), H) == C1`.
4. Assert `calculate_state_diff_hash(stored_diff) != canonical_commitment_for_C2`.

Steps 1–3 succeed today without any error because `parse_data_for_block` never performs the commitment check shown missing at: [7](#0-6)

fn write_compiled_class_hashes<'env>(
    compiled_class_hashes: &IndexMap<ClassHash, CompiledClassHash>,
    txn: &DbTransaction<'env, RW>,
    block_number: BlockNumber,
    compiled_class_hash_table: &'env CompiledClassHashTable<'env>,
) -> StorageResult<()> {
    for (class_hash, compiled_class_hash) in compiled_class_hashes {
        compiled_class_hash_table.insert(txn, &(*class_hash, block_number), compiled_class_hash)?;
    }
    Ok(())
}
