### Title
Integer Wrapping in Relative Timelock Comparison Enables Timelock Bypass for Maximum-Value Conditions — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

When `check_time_locks` is called with `nowrap=false`, it uses `wrapping_add` for relative timelock arithmetic. A CLVM spend supplying `ASSERT_SECONDS_RELATIVE = u64::MAX` or `ASSERT_HEIGHT_RELATIVE = u32::MAX` causes the deadline to wrap around to a value smaller than the coin's birth timestamp/height, making the condition trivially satisfied and bypassing the intended timelock. This is the direct analog of the LeqGadget issue: a numeric comparison that silently produces wrong results for values at the boundary of the supported integer range.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` implements two arithmetic modes controlled by the `nowrap: bool` parameter:

- `nowrap=true` → `saturating_add` (correct: clamps at `u64::MAX` / `u32::MAX`)
- `nowrap=false` → `wrapping_add` (incorrect for boundary values: wraps around)

For `ASSERT_SECONDS_RELATIVE`:

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [1](#0-0) 

With `seconds_relative = u64::MAX` and `unspent.timestamp = T`:

```
wrapping_add(T, u64::MAX) = T - 1
```

The check becomes `timestamp < T - 1`, which is false for any `timestamp >= T - 1` — i.e., almost always. The condition passes, and the spend is accepted immediately, regardless of how long the coin has actually been held.

The same wrapping flaw applies to `ASSERT_HEIGHT_RELATIVE` with `height_relative = u32::MAX`:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
``` [2](#0-1) 

`wrapping_add(confirmed_block_index, u32::MAX) = confirmed_block_index - 1`, so the check `prev_height < confirmed_block_index - 1` is false for any block after the coin was confirmed, bypassing the height lock.

The `sanitize_uint` parser accepts values up to `u64::MAX` as `SanitizedUint::Ok(r)` — only values requiring more than 8 bytes are rejected as `PositiveOverflow`. So `seconds_relative = 0xFFFF_FFFF_FFFF_FFFF` is a fully valid parsed CLVM atom that reaches `check_time_locks` without error:

```rust
SanitizedUint::PositiveOverflow => {
    Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed))
}
SanitizedUint::Ok(r) => Ok(Condition::AssertSecondsRelative(r)),
``` [3](#0-2) 

The Python binding exposes `nowrap` as a caller-controlled parameter:

``` [4](#0-3) 

The project's own tests explicitly confirm the bypass under `nowrap=false`:

```python
# 10000 + (u64::MAX - 9999) overflows to 0, wrapping: 10150 < 0 -> Ok
(make_test_conds(seconds_relative=0xFFFF_FFFF_FFFF_D8F0), 105, None),
# 10000 + u64::MAX overflows to 9999, wrapping: 10150 < 9999 -> Ok
(make_test_conds(seconds_relative=0xFFFF_FFFF_FFFF_FFFF), 105, None),
``` [5](#0-4) 

`expected_nowrap=105` (error) vs `expected_wrap=None` (no error, bypass) confirms the divergence.

---

### Impact Explanation

Any caller of `py_check_time_locks` with `nowrap=False` — including the Python chia-blockchain full node during a consensus-transition window — will accept a spend bundle containing `ASSERT_SECONDS_RELATIVE = u64::MAX` or `ASSERT_HEIGHT_RELATIVE = u32::MAX` immediately, regardless of the coin's actual age. This is a **timelock validation bypass**: a coin that should be locked for effectively forever can be spent at any block. Additionally, nodes running `nowrap=True` would reject the same spend, producing **deterministic consensus divergence** between node versions.

---

### Likelihood Explanation

The `nowrap` parameter is caller-controlled from Python. During any period where `nowrap=False` is the active consensus mode (old behavior), an unprivileged attacker can craft a standard CLVM spend bundle with a single large integer atom (`0x00ffffffffffffffff`, 9 bytes, valid per `sanitize_uint`) as the `ASSERT_SECONDS_RELATIVE` argument. No special keys, governance access, or network position is required. The spend is syntactically valid and passes all CLVM-level checks.

---

### Recommendation

- **Short term**: Add an explicit upper-bound check in `check_time_locks` (or in `sanitize_uint` for relative conditions) that rejects `seconds_relative` or `height_relative` values that would overflow when added to any plausible coin timestamp/height. Alternatively, always use `saturating_add` regardless of the `nowrap` flag for the `ASSERT_SECONDS_RELATIVE` and `ASSERT_HEIGHT_RELATIVE` comparisons.
- **Long term**: Remove the `nowrap=false` (wrapping) code path entirely once the consensus upgrade is complete, eliminating the behavioral divergence. Document the maximum safe values for each relative timelock condition.

---

### Proof of Concept

A CLVM spend bundle for a coin confirmed at block 100 / timestamp 10000, containing:

```
(ASSERT_SECONDS_RELATIVE . 0x00ffffffffffffffff)
```

When evaluated by `check_time_locks(..., nowrap=False)` at any timestamp ≥ 9999:

```
wrapping_add(10000, 18446744073709551615) = 9999
timestamp (e.g. 10150) < 9999  →  false  →  no error  →  spend accepted
```

The coin is accepted immediately despite the `u64::MAX`-second relative lock. Under `nowrap=True` the same spend is correctly rejected (`saturating_add` → `u64::MAX`; `10150 < u64::MAX` → error 105). [1](#0-0) [6](#0-5)
