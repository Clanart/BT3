After tracing the full call path through the production code, I can confirm this is a real vulnerability.

## Key Code Analysis

**`validate_chunk_proofs` does not check shard_id binding:** [1](#0-0) 

The function checks hash integrity, tx_root, and receipts_root — but never compares `chunk.shard_id()` against any externally-requested shard_id.

**`set_state_header` uses position-unaware `verify_path`:** [2](#0-1) 

The `verify_path` call proves that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is *somewhere* in the `chunk_headers_root` Merkle tree. It does **not** verify the chunk is at the leaf index corresponding to `shard_id`. Compare with `verify_path_with_index` which does enforce position: [3](#0-2) 

The `chunk_headers_root` tree is built over all shards ordered by shard index. A valid Merkle proof for shard 1's chunk at position 1 will pass `verify_path` even when the caller claims it is for shard 0.

**No shard_id binding check anywhere in `set_state_header`:** [4](#0-3) 

There is no `chunk.shard_id() != shard_id` guard in the entire function.

**`set_state_part` blindly trusts the stored header's state_root:** [5](#0-4) 

It retrieves the stored header (which contains shard 1's chunk), extracts `prev_state_root` from shard 1's chunk, and validates the `StatePart` against that root. If the attacker supplies valid state parts for shard 1's trie, `validate_state_part` passes.

**The incoming receipts check (step 4) does not block the attack:** [6](#0-5) 

The receipt hash is computed using the *requested* `shard_id` (not the chunk's shard_id), and the attacker can supply the actual valid incoming receipts for shard 0 from the network. These are publicly available data.

### Title
Missing `chunk.shard_id() == shard_id` binding check in `set_state_header` allows cross-shard state installation — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` accepts a `ShardStateSyncResponseHeader` for shard X that contains shard Y's chunk. All validation passes because `verify_path` against `chunk_headers_root` is position-unaware, and no explicit `chunk.shard_id() == shard_id` guard exists. The stored header's `prev_state_root` (from shard Y's chunk) is then used by `set_state_part` and `apply_state_part`, installing shard Y's trie data into shard X's storage.

### Finding Description
An unprivileged state-sync provider peer can craft a `ShardStateSyncResponseHeader` for shard_id=0 that embeds shard_id=1's chunk. The following checks all pass:

1. `validate_chunk_proofs` — verifies hash/tx_root/receipts_root of shard 1's chunk (valid).
2. `verify_path(chunk_headers_root, chunk_proof, ChunkHashHeight(shard1_hash, shard1_height))` — shard 1's chunk IS in the block's Merkle tree; the proof is valid. `verify_path` does not enforce leaf position.
3. Incoming receipts proofs — computed using the requested `shard_id=0`, so the attacker supplies the actual valid receipts for shard 0.
4. `validate_state_root_node` — validates against shard 1's `prev_state_root`, which the attacker provides a valid node for.

The header is stored under `StateHeaderKey(shard_id=0, sync_hash)`. Subsequent `set_state_part` calls extract `state_root` from shard 1's chunk and validate parts against it. `apply_state_part` writes shard 1's trie into shard 0's `ShardUId`.

### Impact Explanation
Shard 0 ends up with shard 1's trie state. When the node attempts to apply chunks for shard 0, the state root will not match the expected value from the chain, causing the node to produce invalid blocks or fail to participate in consensus for shard 0. This is a targeted state corruption attack against any node performing state sync.

### Likelihood Explanation
State sync is performed by any node joining the network or catching up after downtime. The attacker only needs to be a peer that responds to state sync requests — no validator or privileged role is required. All data needed (shard 1's chunk, its Merkle proof, valid incoming receipts for shard 0, shard 1's state parts) is publicly available on the network.

### Recommendation
Add an explicit shard_id binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// ADD THIS:
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Additionally, consider replacing `verify_path` with `verify_path_with_index` using the shard's index in the block to enforce positional binding.

### Proof of Concept
Integration test outline:
1. Set up a two-shard network, produce blocks, identify `sync_hash`.
2. Obtain the legitimate `ShardStateSyncResponseHeader` for shard 1 (including its chunk and Merkle proof).
3. Construct a malicious `ShardStateSyncResponseHeader` claiming to be for shard 0, but embedding shard 1's chunk and its valid Merkle proof, with valid incoming receipts for shard 0.
4. Call `state_sync_adapter.set_state_header(shard_id=0, sync_hash, malicious_header)`.
5. Assert it returns `Ok(())` (demonstrating the missing check).
6. Call `set_state_part` with valid parts for shard 1's trie.
7. Call `set_state_finalize(shard_id=0, sync_hash)` and assert the resulting state root for shard 0 equals shard 1's `prev_state_root` rather than shard 0's expected state root.

        byzantine_assert!(false);
        return Ok(false);

    // 2a. Checking chunk hash

        return Ok(false);

    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();

        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {

            return Ok(false);

        &self,
        shard_id: ShardId,
        sync_hash: CryptoHash,
