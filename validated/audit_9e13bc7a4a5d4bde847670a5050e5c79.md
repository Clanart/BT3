### Title
Unrestricted `logMetadata` Allows Any Caller to Publish Arbitrary Token Metadata via Wormhole — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary
`logMetadata` and `logMetadata1155` in `OmniBridge.sol` are `external payable` with no access control. Any unprivileged user can call them with an arbitrary `tokenAddress` — including a malicious contract returning attacker-chosen name/symbol/decimals — causing a `LogMetadata` event to be emitted and, in the `OmniBridgeWormhole` variant, a Wormhole cross-chain message to be published. Off-chain relayers and the NEAR bridge that consume these messages may register or overwrite token metadata with attacker-controlled values, corrupting decimal mappings and breaking bridge collateralization.

---

### Finding Description

`logMetadata` is declared `external payable` with no role check or bridge-token guard:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name  = IERC20Metadata(tokenAddress).name();
    string memory symbol = IERC20Metadata(tokenAddress).symbol();
    uint8 decimals       = IERC20Metadata(tokenAddress).decimals();
    logMetadataExtension(tokenAddress, name, symbol, decimals);
    emit BridgeTypes.LogMetadata(tokenAddress, name, symbol, decimals);
}
``` [1](#0-0) 

There is no check that `tokenAddress` is a registered bridge token (i.e., `isBridgeToken[tokenAddress]` is never consulted), and no role restriction. The same applies to `logMetadata1155`. [2](#0-1) 

In `OmniBridgeWormhole`, `logMetadataExtension` publishes a Wormhole message of type `LogMetadata` carrying the attacker-supplied `tokenAddress`, `name`, `symbol`, and `decimals`:

```solidity
function logMetadataExtension(...) internal override {
    bytes memory payload = bytes.concat(
        bytes1(uint8(MessageType.LogMetadata)),
        bytes1(omniBridgeChainId),
        Borsh.encodeAddress(tokenAddress),
        Borsh.encodeString(name),
        Borsh.encodeString(symbol),
        bytes1(decimals)
    );
    _wormhole.publishMessage{value: msg.value}(wormholeNonce, payload, _consistencyLevel);
    wormholeNonce++;
}
``` [3](#0-2) 

The NEAR bridge's `log_metadata` flow signs and emits `LogMetadataEvent` payloads that downstream consumers use to register token decimals and mappings: [4](#0-3) 

An attacker can deploy a malicious ERC-20 that returns an arbitrary `decimals()` value (e.g., 6 instead of 18 for USDC, or 18 instead of 6), call `logMetadata` with it, and have the Wormhole relayer deliver the corrupted `LogMetadata` message to NEAR. If the NEAR bridge processes this message to update or register the token's decimal entry in `token_decimals`, all subsequent `normalize_amount` / `denormalize_amount` conversions for that token will use the wrong scale.

---

### Impact Explanation

Corrupted decimal metadata directly breaks the bridge's amount normalization logic. The NEAR bridge uses `token_decimals` to scale amounts during `fin_transfer` and `sign_transfer`: [5](#0-4) 

If an attacker overwrites the stored decimals for a high-value token (e.g., WBTC, USDC), every subsequent cross-chain transfer for that token will mint or release an incorrect amount on the destination chain — either drastically over-minting (unbacked supply) or under-releasing (permanent fund lock). This matches the allowed impact: **High — balance/decimal/accounting corruption that breaks bridge collateralization or misdirects value**.

---

### Likelihood Explanation

The entry path requires no privilege: any EOA can call `logMetadata(maliciousTokenAddress)` while paying the Wormhole message fee (`msg.value`). The Wormhole fee is small (typically a few dollars). The attacker only needs to deploy a contract that mimics a legitimate token's address prefix or name/symbol while returning a different `decimals()`. No leaked keys, no colluding validators, and no admin access are required.

---

### Recommendation

1. Add a bridge-token guard to `logMetadata` and `logMetadata1155`:
   ```solidity
   require(isBridgeToken[tokenAddress], "ERR_NOT_BRIDGE_TOKEN");
   ```
   or restrict the function to a privileged role (e.g., `onlyRole(DEFAULT_ADMIN_ROLE)`).

2. For `logMetadata1155`, additionally verify that the `(tokenAddress, tokenId)` pair is already registered in `multiTokens` before emitting or publishing.

3. On the NEAR side, ensure that `LogMetadata` Wormhole messages can only update metadata for tokens that are already registered, and that re-registration of decimals for an existing token is either disallowed or gated behind a privileged role.

---

### Proof of Concept

1. Attacker deploys `MaliciousToken` implementing `IERC20Metadata` with `name() = "USD Coin"`, `symbol() = "USDC"`, `decimals() = 18` (real USDC has 6).
2. Attacker calls `OmniBridgeWormhole.logMetadata{value: wormholeFee}(address(MaliciousToken))`.
3. `logMetadataExtension` publishes a Wormhole VAA: `[LogMetadata | chainId | realUSDCAddress-lookalike | "USD Coin" | "USDC" | 18]`.
4. Wormhole relayer delivers the VAA to NEAR; the bridge processes it and updates `token_decimals[USDC_omni_address] = 18`.
5. A user bridges 1,000 USDC (1,000 × 10^6 raw units). NEAR now treats this as 1,000 × 10^6 / 10^18 ≈ 0 USDC, permanently locking the funds — or the inverse: 1 raw unit is treated as 10^12 USDC, minting unbacked supply.

    pub fn log_metadata(&self, token_id: &AccountId) -> Promise {
        ext_token::ext(token_id.clone())
            .with_static_gas(LOG_METADATA_GAS)
            .ft_metadata()
            .then(
                Self::ext(env::current_account_id())
                    .with_static_gas(LOG_METADATA_CALLBACK_GAS)
                    .with_attached_deposit(env::attached_deposit())
                    .log_metadata_callback(token_id),
            )
    }

    #[private]
    #[result_serializer(borsh)]
    pub fn log_metadata_callback(
        &self,
        #[callback] metadata: FungibleTokenMetadata,
        token_id: &AccountId,
    ) -> Promise {
        require!(
            !metadata.name.is_empty() && !metadata.symbol.is_empty(),
            BridgeError::InvalidMetadata.as_ref()
        );

        let metadata_payload = MetadataPayload {
            prefix: PayloadType::Metadata,
            token: token_id.to_string(),
            name: metadata.name,
            symbol: metadata.symbol,
            decimals: metadata.decimals,
        };

        let payload = near_sdk::env::keccak256_array(
            borsh::to_vec(&metadata_payload).near_expect(BridgeError::Borsh),
        );

        ext_signer::ext(self.mpc_signer.clone())
            .with_static_gas(MPC_SIGNING_GAS)
            .with_attached_deposit(env::attached_deposit())
            .sign(SignRequest {
                payload,
                path: SIGN_PATH.to_owned(),
                key_version: 0,
            })
            .then(
                Self::ext(env::current_account_id())
                    .with_static_gas(SIGN_LOG_METADATA_CALLBACK_GAS)
                    .sign_log_metadata_callback(metadata_payload),
            )
    }
