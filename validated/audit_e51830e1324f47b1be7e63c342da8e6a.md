### Title
v5 Migration `store.Has()` Incorrectly Forces `AllowUnprotectedTxs=true`, Persisting Through v6/v7/v8, Enabling EIP-155 Replay-Attack Bypass — (`x/evm/migrations/v5/migrate.go`)

---

### Summary

The v5 `MigrateStore` reads the `AllowUnprotectedTxs` boolean parameter using `store.Has()` instead of reading and decoding the actual stored value. Because the legacy params subspace stores the key regardless of whether the value is `true` or `false`, `store.Has()` always returns `true` when the key exists. This silently flips `AllowUnprotectedTxs` from `false` (the default) to `true` for every chain that runs the v5 migration. The v6, v7, and v8 migrations each read the consolidated params blob and write it back without touching `AllowUnprotectedTxs`, so the incorrect value persists indefinitely. After the upgrade, any unprivileged user can submit a non-EIP-155 (Homestead-style, no chain-ID) transaction and have it accepted by the ante handler.

---

### Finding Description

**Root cause — v5 migration, line 36:**

```go
params.AllowUnprotectedTxs = store.Has(v0types.ParamStoreKeyAllowUnprotectedTxs)
``` [1](#0-0) 

`store.Has()` returns `true` whenever the key is present in the KV store. The legacy Cosmos SDK params subspace stores every parameter key unconditionally (amino-JSON encoded), so the key `ParamStoreKeyAllowUnprotectedTxs` exists whether the on-chain value is `true` or `false`. The correct approach is `store.Get()` followed by amino/JSON decoding of the actual boolean. The same pattern is applied to `EnableCreate` and `EnableCall` (lines 34–35), but those default to `true`, so the error is masked for them. `AllowUnprotectedTxs` defaults to `false`: [2](#0-1) 

**Cross-migration persistence — v6, v7, v8:**

Each subsequent migration reads the consolidated params blob from `types.KeyPrefixParams`, modifies only `ChainConfig` fork-time fields, and writes the blob back. None of them inspect or correct `AllowUnprotectedTxs`:

- v6: adds `ShanghaiTime` [3](#0-2) 
- v7: adds `CancunTime`, `PragueTime` [4](#0-3) 
- v8: adds `OsakaTime` [5](#0-4) 

The incorrect `true` value is faithfully round-tripped through every upgrade.

**Ante handler gate — the only enforcement point:**

```go
allowUnprotectedTxs := evmParams.GetAllowUnprotectedTxs()
...
if !allowUnprotectedTxs && !tx.Protected() {
    return errorsmod.Wrapf(errortypes.ErrNotSupported,
        "rejected unprotected Ethereum transaction...")
}
``` [6](#0-5) 

With `AllowUnprotectedTxs=true` (incorrectly set by v5), the guard is permanently disabled. `tx.Protected()` returns `false` for any Homestead/pre-EIP-155 transaction (no chain-ID in the signature), and the condition `!true && !false` evaluates to `false`, so the transaction is accepted.

---

### Impact Explanation

An unprivileged attacker can submit Ethereum transactions signed with the Homestead signer (no chain-ID), which are accepted on the post-migration chain. Because the signature does not commit to a chain-ID, the same signed transaction is valid on any other chain that also accepts unprotected transactions. This is a direct EIP-155 replay-protection bypass: a transaction executed on chain A can be replayed verbatim on chain B (or vice versa), causing unauthorized fund transfers or contract calls from the victim's account on the replayed chain.

This matches the allowed High impact: *"Ethereum transaction, EIP-155 authorization, chain-id, or signer verification bypass enabling replay, forged execution, or unauthorized account/code mutation."*

---

### Likelihood Explanation

- The default `AllowUnprotectedTxs` is `false` on every standard Ethermint deployment.
- Any chain that runs the v5 upgrade (and has not explicitly set `AllowUnprotectedTxs=true` via governance before the upgrade) is affected automatically — no attacker action is required to trigger the migration bug.
- The attacker only needs to craft a Homestead-signed transaction (trivial with any standard Ethereum library) and submit it through the public mempool after the upgrade.
- No privileged access, governance participation, or key compromise is required.

---

### Recommendation

Replace `store.Has()` with a proper read-and-decode for all three boolean parameters in v5 `MigrateStore`:

```go
// Correct approach: read and amino-decode the actual stored value
allowUnprotectedBz := store.Get(v0types.ParamStoreKeyAllowUnprotectedTxs)
if allowUnprotectedBz != nil {
    cdc.MustUnmarshal(allowUnprotectedBz, &params.AllowUnprotectedTxs)
}
```

Apply the same fix to `EnableCreate` and `EnableCall` (lines 34–35) for correctness, even though their defaults happen to be `true`.

Additionally, add a corrective step in a new migration (or in v6/v7/v8 if not yet deployed) that re-reads and validates `AllowUnprotectedTxs` against the intended governance value, or at minimum hard-resets it to `false` unless explicitly enabled.

---

### Proof of Concept

1. Initialize a chain with default params (`AllowUnprotectedTxs=false`). Confirm the legacy params subspace key `ParamStoreKeyAllowUnprotectedTxs` exists in the KV store with value `false`.
2. Run v5 `MigrateStore`. Assert `params.AllowUnprotectedTxs == true` in the consolidated params blob (bug triggered by `store.Has()` returning `true`).
3. Run v6, v7, v8 `MigrateStore` in sequence. Assert `AllowUnprotectedTxs == true` after each — the value is preserved unchanged.
4. Construct a Homestead-signed (non-EIP-155) Ethereum transaction using `ethtypes.HomesteadSigner{}`.
5. Submit it through the public mempool. Observe that `ValidateEthBasic` passes the `!allowUnprotectedTxs && !tx.Protected()` guard (line 126 of `ante/interfaces/setup.go`) and the transaction is included in a block.
6. Replay the identical signed bytes on a second chain running the same post-migration state. Observe the transaction is accepted and executed on the second chain, demonstrating cross-chain replay.
