### Title
Wrong `state_diff_commitment` in Block Hash Due to Silent Drop of `deprecated_declared_classes` in `CommitmentStateDiff` → `ThinStateDiff` Conversion — (`crates/blockifier/src/state/cached_state.rs`)

---

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` conversion hardcodes `deprecated_declared_classes: Vec::new()`. This `ThinStateDiff` is then fed directly into `calculate_block_commitments`, which calls `calculate_state_diff_hash`. Because `deprecated_declared_classes` is always empty in this path, the `state_diff_commitment` committed into every block hash is structurally wrong for any block that contains a Declare V1 (Cairo 0 / deprecated class) transaction. The resulting `PartialBlockHash` — used as the consensus proposal commitment — and the final `BlockHash` both commit to an incomplete state diff.

---

### Finding Description

**Conversion that drops the field:** [1](#0-0) 

```rust
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            deployed_contracts: commitment_state_diff.address_to_class_hash,
            storage_diffs: commitment_state_diff.storage_updates,
            class_hash_to_compiled_class_hash: commitment_state_diff
                .class_hash_to_compiled_class_hash,
            nonces: commitment_state_diff.address_to_nonce,
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),   // ← always empty
        }
    }
}
```

**Where this `ThinStateDiff` is used for commitment:**

`BlockExecutionArtifacts::new` calls `calculate_block_commitments` with `ThinStateDiff::from(commitment_state_diff.clone())`: [2](#0-1) 

**`calculate_block_commitments` calls `calculate_state_diff_hash`:** [3](#0-2) 

**`calculate_state_diff_hash` hashes `deprecated_declared_classes`:** [4](#0-3) 

The hash chains `chain_deprecated_declared_classes(&state_diff.deprecated_declared_classes, ...)`. When this field is always `Vec::new()`, the hash encodes count=0 and no class hashes, regardless of how many Cairo 0 classes were actually declared in the block.

**The wrong `state_diff_commitment` propagates into the block hash:** [5](#0-4) 

**And into the `PartialBlockHash` (proposal commitment):** [6](#0-5) 

**The committer's internal verification also uses the same wrong `ThinStateDiff`**, so it passes silently: [7](#0-6) 

Both sides of the check use the same `CommitmentStateDiff`-derived `ThinStateDiff`, so the mismatch is never caught internally.

---

### Impact Explanation

Any block that includes a Declare V1 transaction (Cairo 0 deprecated class declaration) will have a `state_diff_commitment` that omits those declarations. Concretely:

1. **Wrong `BlockHash`** — `calculate_block_hash` chains `state_diff_commitment` directly; the hash is structurally incorrect.
2. **Wrong `PartialBlockHash`** — used as the consensus proposal commitment; validators accept a commitment that does not cover the full state diff.
3. **Wrong stored `state_diff_commitment`** in `StorageBlockHeader` — downstream consumers (RPC, sync, proof manager) read a commitment that does not match the actual state diff.
4. **SNOS / proof inputs inconsistency** — the `OsBlockInput` state diff commitment diverges from what an independent verifier would compute from the actual `ThinStateDiff` (which does include `deprecated_declared_classes` when constructed via `ThinStateDiff::from_state_diff`).

The internal committer verification passes because both sides use the same defective conversion, masking the error entirely.

---

### Likelihood Explanation

Declare V1 transactions (Cairo 0 class declarations) are valid Starknet transactions and are not rejected by the gateway. Any user can submit one. The bug is triggered deterministically whenever such a transaction is included in a block. No privileged access is required.

---

### Recommendation

`CommitmentStateDiff` must be extended to carry `deprecated_declared_classes`, populated from the blockifier's execution output when a Declare V1 transaction is processed. The `From<CommitmentStateDiff> for ThinStateDiff` conversion should then propagate this field instead of hardcoding `Vec::new()`. The TODO comment acknowledges the structural gap; it must be resolved before the commitment path is considered correct.

---

### Proof of Concept

1. Submit a Declare V1 transaction declaring a Cairo 0 class hash `C`.
2. The batcher includes it in a block. `CommitmentStateDiff` has no `deprecated_declared_classes` field, so `ThinStateDiff::from(commitment_state_diff)` produces `deprecated_declared_classes: []`.
3. `calculate_state_diff_hash` hashes `[]` → commitment `H_wrong`.
4. Independently compute `calculate_state_diff_hash` on the actual `ThinStateDiff` (with `deprecated_declared_classes: [C]`) → commitment `H_correct`.
5. `H_wrong ≠ H_correct`. The block hash and proposal commitment are built from `H_wrong`.
6. The committer's `verify_state_diff_hash` check compares `H_wrong` against `H_wrong` (same defective input on both sides) and passes, so no error is raised.

pub fn calculate_state_diff_hash(state_diff: &ThinStateDiff) -> StateDiffCommitment {
    let mut hash_chain = HashChain::new();
    hash_chain = hash_chain.chain(&STARKNET_STATE_DIFF0);
    hash_chain = chain_deployed_contracts(&state_diff.deployed_contracts, hash_chain);
    hash_chain = chain_declared_classes(&state_diff.class_hash_to_compiled_class_hash, hash_chain);
    hash_chain =
        chain_deprecated_declared_classes(&state_diff.deprecated_declared_classes, hash_chain);
    hash_chain = hash_chain.chain(&Felt::ONE) // placeholder.
        .chain(&Felt::ZERO); // placeholder.
    hash_chain = chain_storage_diffs(&state_diff.storage_diffs, hash_chain);
    hash_chain = chain_nonces(&state_diff.nonces, hash_chain);
    StateDiffCommitment(PoseidonHash(hash_chain.get_poseidon_hash()))
}
