### Title
Unvalidated Attacker-Controlled `payload_size` in `execute_send_message_to_l1` Enables OS Output Corruption and Network Halt — (File: `execution/syscall_impls.cairo`)

---

### Summary

`execute_send_message_to_l1` in `syscall_impls.cairo` computes `payload_size` directly from attacker-controlled syscall fields (`request.payload_end - request.payload_start`) with no bounds check. If a contract sets `payload_end < payload_start`, the subtraction wraps around the field prime (~2²⁵¹), producing an astronomically large `payload_size`. This value is then passed verbatim to `memcpy` and written into the OS output segment, causing the OS execution to exhaust all available VM steps and fail to produce a valid proof — a total network halt.

---

### Finding Description

In `execute_send_message_to_l1`:

```cairo
tempvar payload_start = request.payload_start;
tempvar payload_size = request.payload_end - payload_start;

assert [outputs.messages_to_l1] = MessageToL1Header(
    from_address=contract_address, to_address=request.to_address, payload_size=payload_size
);
memcpy(
    dst=outputs.messages_to_l1 + MessageToL1Header.SIZE, src=payload_start, len=payload_size
);
let (outputs) = os_carried_outputs_new(
    messages_to_l1=outputs.messages_to_l1 + MessageToL1Header.SIZE + payload_size,
    ...
);
``` [1](#0-0) 

There is no `assert_nn(payload_size)` or `assert_nn_le(payload_size, SIERRA_ARRAY_LEN_BOUND - 1)` before `payload_size` is used. Compare this with every other array-length computation in the same file and in `transaction_impls.cairo`, which all apply explicit bounds checks:

```cairo
// execute_meta_tx_v0 — validates signature array
assert_nn_le(request.signature_end - request.signature_start, SIERRA_ARRAY_LEN_BOUND - 1);
``` [2](#0-1) 

```cairo
// get_invoke_tx_execution_context — validates calldata array
assert_nn_le(tx_execution_context.calldata_size, SIERRA_ARRAY_LEN_BOUND - 1);
``` [3](#0-2) 

`execute_send_message_to_l1` is the only syscall handler that copies a variable-length attacker-supplied array into the OS output without any such guard. [4](#0-3) 

The gas cost for this syscall is a **fixed constant** (`SEND_MESSAGE_TO_L1_GAS_COST`), with no per-element charge for the payload, so the attacker pays only the base syscall fee regardless of the injected `payload_size`. [5](#0-4) 

---

### Impact Explanation

When `payload_end < payload_start`, Cairo felt subtraction wraps modulo the field prime P ≈ 2²⁵¹, producing `payload_size ≈ P`. The subsequent `memcpy(dst, src, len=P)` iterates one felt per step for P iterations — far exceeding any realistic VM step budget. The OS Cairo program cannot complete execution, no valid STARK proof can be generated for the block, and the network cannot finalize any further blocks until the issue is resolved. This maps directly to the allowed impact: **High — Network not being able to confirm new transactions (total network shutdown)**.

---

### Likelihood Explanation

The attack requires only:
1. Deploying a contract (permissionless on StarkNet).
2. Having that contract issue a `send_message_to_l1` syscall with `payload_end` set to any value less than `payload_start`.
3. Submitting one invoke transaction to call that contract.

No privileged role, leaked key, or network-level capability is needed. The OS is the authoritative re-execution layer; even if a sequencer-side filter rejects the transaction, the absence of the check in the OS Cairo code means any sequencer that does not independently enforce this invariant will produce an unprovable block. The vulnerability is reachable by any unprivileged transaction sender.

---

### Recommendation

Add an explicit non-negativity and upper-bound check on `payload_size` immediately after it is computed, consistent with every other array-length guard in the codebase:

```cairo
tempvar payload_start = request.payload_start;
tempvar payload_size = request.payload_end - payload_start;
// ADD: assert payload_size is a valid, non-negative array length
assert_nn_le(payload_size, SIERRA_ARRAY_LEN_BOUND - 1);
```

This mirrors the pattern already used for `signature`, `calldata`, and `constructor_calldata` throughout the OS.

---

### Proof of Concept

1. **Deploy** a Cairo contract containing:
   ```cairo
   // Inside an external entry point
   starknet::send_message_to_l1(
       to_address: 0x1,
       payload: [],   // payload_start = X, payload_end = X - 1 (set via raw syscall)
   );
   ```
   Using a raw syscall write, set `payload_end = payload_start - 1` so that `payload_end - payload_start = P - 1` in felt arithmetic.

2. **Submit** an invoke transaction calling this entry point. The sequencer executes it and, lacking its own guard, includes it in a block.

3. **OS re-execution**: `execute_send_message_to_l1` computes `payload_size = P - 1`, writes it into `MessageToL1Header`, then calls `memcpy(..., len = P - 1)`.

4. The Cairo VM exhausts its step budget inside `memcpy`. No proof is generated for the block.

5. The network cannot advance past this block → **total network shutdown**. [6](#0-5)
