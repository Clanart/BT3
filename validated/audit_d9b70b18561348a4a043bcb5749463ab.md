### Title
`denormalize_amount` Arithmetic Overflow in `fin_transfer_callback` Causes Permanent Fund Lock â€” (`near/omni-bridge/src/lib.rs`)

### Summary

`denormalize_amount` uses unchecked `*` multiplication with `overflow-checks = true` in the release profile. An attacker can call EVM `initTransfer` with `amount` large enough that `amount * 10^diff` overflows `u128`, causing `fin_transfer_callback` to panic and permanently locking the EVM-side tokens. The specific claim about `denormalize_fee` overflowing first is slightly off â€” because the EVM enforces `fee < amount`, the overflow always hits `amount` denormalization at line 725 before reaching `denormalize_fee` at line 727 â€” but the underlying vulnerability and impact are real.

---

### Finding Description

**`overflow-checks = true` in release profile:** [1](#0-0) 

This means any arithmetic overflow in release builds panics rather than wrapping.

**`denormalize_amount` uses plain `*` multiplication:** [2](#0-1) 

No `checked_mul` or `saturating_mul`. For any token where `origin_decimals > decimals`, the multiplier `10^diff` can be large. If `amount * 10^diff > u128::MAX`, this panics.

**`fin_transfer_callback` calls `denormalize_amount` on the raw EVM event amount:** [3](#0-2) 

**The EVM `InitTransfer` event emits raw (non-normalized) `amount` and `fee` as `uint128`:** [4](#0-3) 

**The NEAR parser reads them directly into `u128` with no bounds check:** [5](#0-4) 

**The EVM bridge only enforces `fee < amount` (relative cap), not an absolute cap:** [6](#0-5) 

**Overflow analysis:**

For a token with `origin_decimals = 24` (NEAR) and `decimals = 18` (EVM), `diff = 6`. The overflow threshold is `u128::MAX / 10^6 â‰ˆ 3.4 Ă— 10^32` in EVM units. For `diff = 30`, the threshold drops to `â‰ˆ 3.4 Ă— 10^8` EVM units (340 whole tokens with 6 EVM decimals). Any `amount` above the threshold causes `denormalize_amount` to panic.

**Why `denormalize_fee` is not the first overflow point:**

Since the EVM enforces `fee < amount`, if `fee * 10^diff > u128::MAX` then `amount * 10^diff > u128::MAX` as well. The panic therefore always occurs at line 725 (`amount` denormalization) before line 727 (`denormalize_fee`). The question's specific claim about `denormalize_fee` overflowing first is incorrect, but the root cause and impact are identical. [7](#0-6) 

---

### Impact Explanation

When `fin_transfer_callback` panics:
- The NEAR transaction reverts; the transfer is never finalized on NEAR.
- The EVM tokens are already locked/burned in the EVM bridge (transferred in `initTransfer`).
- There is no automatic refund path triggered by a NEAR-side panic.
- The funds are permanently locked.

This matches the allowed impact: **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."**

---

### Likelihood Explanation

**Constraints:**
1. The token must have `origin_decimals > decimals` (NEAR decimals > EVM decimals). This is a real configuration for NEAR-native tokens bridged to EVM.
2. The attacker must hold `amount > u128::MAX / 10^diff` tokens and transfer them to the EVM bridge.
3. An automated relayer must submit the proof (standard bridge operation).

**Practical threshold by `diff`:**

| diff | Overflow threshold (EVM units) | Whole tokens (18 EVM decimals) |
|------|-------------------------------|-------------------------------|
| 6    | ~3.4 Ă— 10^32                  | ~3.4 Ă— 10^14 (340 trillion)   |
| 18   | ~3.4 Ă— 10^20                  | ~3.4 Ă— 10^2 (340)             |
| 30   | ~3.4 Ă— 10^8                   | ~340 (6 EVM decimals)         |

For tokens with large decimal differences, the threshold is very low. The attacker primarily harms their own funds, but the bridge's accounting integrity and the permanent lock of user funds constitute a real security impact.

### Recommendation

Replace the unchecked multiplication in `denormalize_amount` with a checked variant that returns an error instead of panicking:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> Result<u128, BridgeError> {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    let multiplier = 10_u128.pow(diff_decimals);
    amount.checked_mul(multiplier).ok_or(BridgeError::AmountOverflow)
}
```

Propagate the `Result` through `fin_transfer_callback` and `denormalize_fee`, and handle the error gracefully (e.g., reject the transfer with an error rather than panicking). Similarly, add a pre-check on the EVM side capping `amount` and `fee` to `u128::MAX / 10^diff` for tokens with decimal normalization. [2](#0-1)
