### Title
Class Declaration Griefing via Unvalidated `compiled_class_hash` Enables Permanent Blocking of Class Declarations — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_declare_transaction` enforces a once-only class declaration constraint (`prev_value=0`) but does not validate that the caller-supplied `compiled_class_hash` corresponds to any valid CASM compilation of the Sierra class. An unprivileged attacker can front-run any pending declare transaction with a garbage `compiled_class_hash`, permanently poisoning the class registry entry and blocking the legitimate declaration forever.

---

### Finding Description

In `execute_declare_transaction`, the Sierra class hash is rigorously verified via `finalize_class_hash`, which recomputes the expected hash from the Sierra class component hashes and asserts equality:

```cairo
let expected_class_hash = finalize_class_hash(
    contract_class_component_hashes=contract_class_component_hashes
);
with_attr error_message("Invalid class hash pre-image.") {
    assert [class_hash_ptr] = expected_class_hash;
}
``` [1](#0-0) 

However, the `compiled_class_hash` — the mapping target stored in the class registry — is subject to only a non-zero check before being committed:

```cairo
// Note that prev_value=0 enforces that a class may be declared only once.
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [2](#0-1) 

There is no verification that `compiled_class_hash` corresponds to a valid CASM compilation of the Sierra class, nor any check against the `compiled_class_facts_bundle`. The `validate_compiled_class_facts_post_execution` call in `os.cairo` only validates compiled class facts that were **used during execution** in the block — a class declared with a garbage hash that is never executed in the same block escapes this check entirely. [3](#0-2) 

The `prev_value=0` constraint is the critical amplifier: once any value (including garbage) is written for a given `class_hash` key, no subsequent declare transaction can succeed for that class, because `dict_update` will assert `prev_value=0` and fail.

When a contract later attempts to execute using the poisoned class, `execute_entry_point` reads the garbage `compiled_class_hash` from `contract_class_changes` and calls `find_element` to locate it in `compiled_class_facts_bundle`. Since the garbage hash does not exist in the bundle, execution fails unconditionally for every contract referencing that class. [4](#0-3) 

---

### Impact Explanation

The class is permanently declared with an invalid `compiled_class_hash`. There is no "undeclare" mechanism in the protocol. Any contract that:
- Is deployed with the poisoned class hash, or
- Calls `replace_class` to upgrade to the poisoned class hash

will be permanently non-executable. If such a contract holds user funds and its withdrawal or recovery logic depends on the class being executable, those funds are **permanently frozen**. This satisfies the Critical impact threshold: **Permanent freezing of funds**.

---

### Likelihood Explanation

The attack requires only:
1. Monitoring the public mempool for pending declare transactions (trivial).
2. Extracting the Sierra class from the pending transaction (the class is public data in the transaction).
3. Submitting a competing declare transaction with the same Sierra class but `compiled_class_hash = 1` (any non-zero felt), paying a higher fee to win ordering.

Cost: one declare transaction's gas fee. The attack is one-shot and irreversible. No privileged access, leaked keys, or Sybil resources are required.

---

### Recommendation

At declaration time, validate that `compiled_class_hash` is present in the `compiled_class_facts_bundle` provided to the OS for the current block. Alternatively, require the declarer to supply the full compiled class and verify `hash(compiled_class) == compiled_class_hash` inline, analogous to how `finalize_class_hash` verifies the Sierra class hash. This closes the gap between the two hash verifications.

---

### Proof of Concept

1. Alice submits a declare transaction for Sierra class `X` with valid `compiled_class_hash = Y`.
2. Bob observes Alice's transaction in the mempool and extracts Sierra class `X` (public data).
3. Bob submits a declare transaction for the same Sierra class `X` with `compiled_class_hash = 1` (garbage, non-zero), paying higher fees to win ordering.
4. Bob's transaction executes first. `finalize_class_hash` succeeds (the Sierra class is valid). `assert_not_zero(1)` passes. `dict_update(key=class_hash(X), prev_value=0, new_value=1)` succeeds.
5. Alice's transaction executes next. `dict_update(key=class_hash(X), prev_value=0, ...)` **fails** — `prev_value` is now `1`, not `0`.
6. Class `X` is permanently registered as `class_hash(X) → 1` in the class registry.
7. Any contract that attempts to execute using class `X` triggers `find_element(..., key=1)` in `compiled_class_facts_bundle`, which fails because no compiled class with hash `1` exists.
8. Contracts holding funds that depend on class `X` for withdrawal or recovery are permanently frozen.
