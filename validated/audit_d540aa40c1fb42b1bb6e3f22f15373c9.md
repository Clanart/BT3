### Title
Validator Accepts L1 Handler Transactions Without Enforcing Proposal Cooldown, Enabling Proposer to Front-Run L1 Cancellations - (File: crates/apollo_l1_provider/src/transaction_manager.rs)

### Summary
`TransactionManager::validate_tx` applies no `l1_handler_proposal_cooldown_seconds` check, while `get_txs` does. A malicious proposer can include a freshly-scraped L1 handler transaction in a proposed block before the cooldown expires; every honest validator will call `validate_tx`, receive `ValidationStatus::Validated`, and accept the block. The cooldown — the only mechanism that gives users time to cancel an L1 handler before it is sequenced — is therefore unenforceable by the validator quorum.

### Finding Description

The L1 provider enforces a proposal cooldown (default 70 s, production config 70 s) to prevent a freshly-scraped `L1HandlerTransaction` from being immediately sequenced. The intent is to give users a window to submit a `MessageToL2CancellationStarted` event on L1 before the message is executed on L2.

**`get_txs` (proposer path) — cooldown enforced:** [1](#0-0) 

```rust
let cutoff = now.saturating_sub(
    self.config.l1_handler_proposal_cooldown_seconds.as_secs()
);
let past_cooldown_txs = self.proposable_index.range(..cutoff);
```

Only transactions whose `scrape_timestamp < cutoff` are returned to the proposer.

**`validate_tx` (validator path) — cooldown absent:** [2](#0-1) 

`validate_tx` checks only the cancellation timelock and the `is_validatable()` predicate (not committed, not cancelled, not consumed). There is no comparison against `l1_handler_proposal_cooldown_seconds`. A transaction scraped one second ago passes validation identically to one scraped 10 minutes ago.

The design intent is documented in the config: [3](#0-2) 

> "How long to wait before allowing new L1 handler transactions to be **proposed** (validation is available immediately)"

The asymmetry is intentional for the honest-proposer case, but it creates a broken invariant: **the validator quorum cannot enforce the cooldown**. A malicious proposer that bypasses `get_txs` (or constructs a block directly) and includes a freshly-scraped L1 handler will have every validator return `Validated` and accept the block.

    pub fn validate_tx(&mut self, tx_hash: TransactionHash, unix_now: u64) -> ValidationStatus {
        let current_staging_epoch_cloned = self.current_staging_epoch;

        let policy = TransactionRecordPolicy {
            cancellation_timelock: self.config.l1_handler_cancellation_timelock_seconds,
        };

        let validation_status = self.with_record(tx_hash, |record| {
            // If the current time affects the state, update state now.
            record.update_time_based_state(unix_now, policy);
            if !record.is_validatable() {
                match record.state {
                    TransactionState::Committed => {
                        InvalidValidationStatus::AlreadyIncludedOnL2.into()
                    }
                    TransactionState::CancelledOnL2 => {
                        InvalidValidationStatus::CancelledOnL2.into()
                    }
                    TransactionState::Consumed => InvalidValidationStatus::ConsumedOnL1.into(),
                    _ => unreachable!(),
                }
            } else if record.try_mark_staged(current_staging_epoch_cloned) {
                ValidationStatus::Validated
            } else {
                InvalidValidationStatus::AlreadyIncludedInProposedBlock.into()
            }
        });

        validation_status.unwrap_or(InvalidValidationStatus::NotFound.into())
    }
