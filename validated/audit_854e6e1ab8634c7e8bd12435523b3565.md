### Title
`MerkleBlob::batch_insert` Skips Duplicate `KeyId`/`Hash` Validation, Enabling DataLayer Tree Root Corruption — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` silently bypasses the duplicate-key and duplicate-hash guards that the single-item `insert()` enforces. An untrusted caller can supply a batch containing repeated `KeyId` or `Hash` values — or values already present in the tree — and the function will write all of them into the blob without error, producing a structurally corrupt Merkle tree with a wrong root hash and unreliable proofs of inclusion/exclusion.

---

### Finding Description

The single-item `insert()` function explicitly rejects duplicates before touching the blob:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  insert()
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert()` calls `insert()` for at most the last two items popped from the vector — and only when the tree currently has ≤ 1 leaf. Every other item is written directly to the blob via `insert_entry_to_blob` with **no duplicate check whatsoever**:

```rust
// batch_insert fast path — no contains_key / contains_leaf_hash guard
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ..., node: Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

When the tree already has ≥ 2 leaves (the common production case), the `if self.block_status_cache.leaf_count() <= 1` branch is skipped entirely, so **all** items in the batch bypass the duplicate check: [3](#0-2) 

The Python binding `py_batch_insert` exposes this path directly to callers with no additional validation — it only checks that the two input lists have equal length: [4](#0-3) 

The public Python API signature confirms the surface is reachable from untrusted input: [5](#0-4) 

This is structurally identical to the external report's root cause: a list of identifiers that must be unique is accepted and processed without a uniqueness check, while the single-item path enforces the invariant correctly.

---

### Impact Explanation

When `batch_insert` receives duplicate `KeyId` values:

1. Two `LeafNode` entries with the same `key` are written into the blob at distinct `TreeIndex` positions.
2. `block_status_cache.key_to_index` (a `HashMap`) can only hold one mapping per `KeyId`; the second write silently shadows the first, leaving one leaf permanently unreachable through the cache.
3. The internal-node subtree built over the batch hashes both leaves, producing a Merkle root that does not correspond to any valid, deduplicated tree state.
4. `get_proof_of_inclusion(key)` returns a proof anchored to the wrong root hash.
5. `ProofOfInclusion::valid()` evaluates the proof against the corrupted root, returning `true` for a root that does not match the blockchain-committed value — i.e., the node locally "proves" inclusion against an invalid state.
6. `check_integrity()` will detect the inconsistency, but it is not called automatically after `batch_insert` in production paths.

The same corruption occurs for duplicate `Hash` values, since `leaf_hash_to_index` is also a `HashMap` that silently overwrites on collision.

This satisfies: **High — DataLayer Merkle proof/blob/delta logic corrupts tree roots and lets untrusted input prove invalid state.**

---

### Likelihood Explanation

`batch_insert` is the primary bulk-load path used during DataLayer delta synchronization. Delta data originates from remote peers and is parsed into `(KeyId, ValueId, Hash)` tuples before being passed to `batch_insert`. A malicious peer can craft a delta containing repeated `KeyId` values. Because the function returns `Ok(())` on duplicate input, the corruption is silent and the caller has no signal to abort or retry. The `check_integrity_on_drop` flag is `false` in production-adjacent code paths, so the corruption persists undetected.

---

### Recommendation

Add a pre-flight uniqueness check at the top of `batch_insert` that mirrors the guards in `insert()`:

```rust
pub fn batch_insert(
    &mut self,
    keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
) -> Result<(), Error> {
    // Reject duplicates within the batch itself
    let mut batch_keys  = HashSet::new();
    let mut batch_hashes = HashSet::new();
    for ((key, _), hash) in &keys_values_hashes {
        if !batch_keys.insert(*key) || self.block_status_cache.contains_key(*key) {
            return Err(Error::KeyAlreadyPresent());
        }
        if !batch_hashes.insert(*hash) || self.block_status_cache.contains_leaf_hash(hash) {
            return Err(Error::HashAlreadyPresent());
        }
    }
    // ... existing logic unchanged
}
```

This brings `batch_insert` into parity with `insert()` and closes the gap that the fast path currently leaves open.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

blob = MerkleBlob(blob=bytearray())

# Seed the tree with 3 leaves so leaf_count > 1 (fast path is always taken)
for i in range(3):
    h = bytes(hashlib.sha256(i.to_bytes(8, "big")).digest())
    blob.insert(KeyId(i), ValueId(i), h)

blob.calculate_lazy_hashes()
root_before = blob.get_root_hash()

# Batch with a duplicate KeyId — single insert() would raise KeyAlreadyPresentError
dup_key = KeyId(999)
h1 = bytes(hashlib.sha256(b"h1").digest())
h2 = bytes(hashlib.sha256(b"h2").digest())

# No exception raised; both leaves are written to the blob
blob.batch_insert(
    [(dup_key, ValueId(1)), (dup_key, ValueId(2))],
    [h1, h2],
)

blob.calculate_lazy_hashes()
root_after = blob.get_root_hash()

# root_after is computed over a tree with two leaves sharing KeyId(999)
# check_integrity() will now report an inconsistency
blob.check_integrity()   # raises — tree is corrupt
```

The `insert()` path correctly raises `KeyAlreadyPresentError` for the same input, confirming the guard exists but is absent in `batch_insert`. [6](#0-5) [1](#0-0) [4](#0-3)

    pub fn batch_insert(
        &mut self,
        mut keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
    ) -> Result<(), Error> {
        // OPT: perhaps go back to taking an iterator?
        // OPT: would it be worthwhile to hold the entire blocks?
        let mut indexes = vec![];

        if self.block_status_cache.leaf_count() <= 1 {
            for _ in 0..2 {
                let Some(((key, value), hash)) = keys_values_hashes.pop() else {
                    return Ok(());
                };
                self.insert(key, value, &hash, InsertLocation::Auto {})?;
            }
        }

        for ((key, value), hash) in keys_values_hashes {
            let new_leaf_index = self.get_new_index();
            let new_block = Block {
                metadata: NodeMetadata {
                    node_type: NodeType::Leaf,
                    dirty: false,
                },
                node: Node::Leaf(LeafNode {
                    parent: Parent(None),
                    hash,
                    key,
                    value,
                }),
            };
            self.insert_entry_to_blob(new_leaf_index, &new_block)?;
            indexes.push(new_leaf_index);
        }

        // OPT: can we insert the top node first?  maybe more efficient to update it's children
        //      than to update the parents of the children when traversing leaf to sub-root?
        while indexes.len() > 1 {
            let mut new_indexes = vec![];

            for chunk in indexes.chunks(2) {
                let [index_1, index_2] = match chunk {
                    [index] => {
                        new_indexes.push(*index);
                        continue;
                    }
                    [index_1, index_2] => [*index_1, *index_2],
                    _ => unreachable!(
                        "chunk should always be either one or two long and be handled above"
                    ),
                };

                let new_internal_node_index = self.get_new_index();

                let mut hashes = vec![];
                for index in [index_1, index_2] {
                    let block = self.update_parent(index, Some(new_internal_node_index))?;
                    hashes.push(block.node.hash());
                }

                let new_block = Block {
                    metadata: NodeMetadata {
                        node_type: NodeType::Internal,
                        dirty: false,
                    },
                    node: Node::Internal(InternalNode {
                        parent: Parent(None),
                        hash: internal_hash(&hashes[0], &hashes[1]),
                        left: index_1,
                        right: index_2,
                    }),
                };

                self.insert_entry_to_blob(new_internal_node_index, &new_block)?;
                new_indexes.push(new_internal_node_index);
            }

            indexes = new_indexes;
        }

        if indexes.len() == 1 {
            // OPT: can we avoid this extra min height leaf traversal?
            let min_height_leaf = self.get_min_height_leaf()?;
            self.insert_subtree_at_key(min_height_leaf.key, indexes[0], Side::Left)?;
        }

        Ok(())
    }

    pub fn py_batch_insert(

        hashes: Vec<Hash>,

        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
