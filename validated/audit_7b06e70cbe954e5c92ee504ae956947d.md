### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Indefinite Signature Replay — (File: `transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_meta_tx_v0_hash` function in `transaction_hash.cairo` computes a transaction hash for v0 meta-transactions that omits any nonce or deadline. The `execute_meta_tx_v0` syscall in `syscall_impls.cairo` then constructs a `TxInfo` with `version=0` and `nonce=0`, which causes the OS-level `check_and_increment_nonce` to skip nonce enforcement entirely. As a result, any valid `meta_tx_v0` signature is permanently replayable by any unprivileged party who has observed it on-chain.

---

### Finding Description

**Root cause — `compute_meta_tx_v0_hash` omits nonce:** [1](#0-0) 

The function calls `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`. The resulting hash is a pure function of `(INVOKE_HASH_PREFIX, version=0, contract_address, selector, calldata_hash, max_fee=0, chain_id)`. No nonce, no deadline, no per-invocation uniqueness.

Compare this with `compute_l1_handler_transaction_hash`, which correctly passes `additional_data_size=1, additional_data=&nonce`: [2](#0-1) 

**Root cause — `execute_meta_tx_v0` sets `version=0` and `nonce=0`, bypassing nonce enforcement:** [3](#0-2) 

The newly constructed `TxInfo` has `version=0` and `nonce=0`. The OS-level nonce guard explicitly skips version-0 transactions: [4](#0-3) 

Because `check_and_increment_nonce` returns immediately for `version == 0`, no nonce is consumed and no state is updated to prevent re-use of the same signature.

**The syscall accepts fully attacker-controlled inputs:** [5](#0-4) 

`request.contract_address`, `request.selector`, `request.calldata_start/end`, and `request.signature_start/end` all come from the syscall pointer, which is written by the calling contract — fully attacker-controlled. The only non-attacker-controlled field in the hash is `chain_id`, which is inherited from the parent transaction's `TxInfo`: [6](#0-5) 

---

### Impact Explanation

**Direct loss of funds (Critical).**

A user signs a `meta_tx_v0` payload authorizing a token transfer (e.g., `transfer(recipient, amount)`) via their account contract's `__execute__` entry point. The signature is broadcast on-chain and becomes public. Because the signed hash contains no nonce and no deadline, an attacker can:

1. Deploy (or reuse) any contract that issues the `meta_tx_v0` syscall.
2. Supply the original `(contract_address, selector, calldata, signature)` tuple verbatim.
3. The OS recomputes the identical hash, passes the identical signature to the target account's `__execute__`, and the account contract — which verifies against `tx_info.transaction_hash` — accepts it.
4. The transfer executes again.

This can be repeated until the victim's balance is exhausted. The `__execute__` entry point is the only selector the syscall is permitted to invoke: [7](#0-6) 

---

### Likelihood Explanation

**High.** Once a `meta_tx_v0`-bearing transaction is included in a block, the signature is permanently visible in the transaction's calldata. Any party can replay it by submitting a new outer transaction that calls a contract invoking the `meta_tx_v0` syscall with the same parameters. No privileged access, leaked key, or social engineering is required. The attacker only needs to be an ordinary transaction sender.

---

### Recommendation

1. **Include a nonce in `compute_meta_tx_v0_hash`:** Pass `additional_data_size=1` and a per-invocation nonce (e.g., the target account's current storage nonce) as `additional_data`, mirroring the pattern used in `compute_l1_handler_transaction_hash`.

2. **Enforce nonce consumption for meta_tx_v0:** After the inner execution, increment the nonce associated with the target `contract_address` in `contract_state_changes`, or introduce a dedicated meta-tx nonce slot, so each signature can only be used once.

3. **Optionally add a deadline field:** Include a `valid_until` block number or timestamp in the hash and reject executions where `block_info.block_timestamp > valid_until`.

---

### Proof of Concept

**Step 1 — Victim signs a meta_tx_v0 payload:**
- `contract_address = VICTIM_ACCOUNT`
- `selector = __execute__`
- `calldata = [transfer, ATTACKER, 1000_STRK]`
- Signed hash = `H(INVOKE_HASH_PREFIX, 0, VICTIM_ACCOUNT, __execute__, H(calldata), 0, chain_id)` — fixed forever.

**Step 2 — Legitimate relayer submits the transaction.** Funds transfer once.

**Step 3 — Attacker deploys `ReplayContract` with a single function:**
```
fn replay(sig: Array<felt252>) {
    starknet::syscalls::meta_tx_v0(
        contract_address: VICTIM_ACCOUNT,
        selector: __execute__,
        calldata: [transfer, ATTACKER, 1000_STRK],
        signature: sig,
    );
}
```

**Step 4 — Attacker calls `ReplayContract.replay(original_sig)` in a loop.** Each call:
- OS computes the same hash (no nonce → same inputs → same hash).
- Passes the same signature to `VICTIM_ACCOUNT.__execute__`.
- Account verifies signature against `tx_info.transaction_hash` → passes.
- Transfer executes again.

Funds are drained until the victim's balance reaches zero.
