Audit Report

## Title
`refundETH()` Has No Access Control — Any Caller Steals Excess ETH Left on Router After a Partial-ETH Swap - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
`refundETH()` transfers the router's entire ETH balance to `msg.sender` with no ownership or origin check. When a user calls any `payable` swap entry point with more ETH than `amountIn`, `pay()` deposits only the exact swap amount as WETH and leaves the remainder on the router. Any subsequent caller can drain that remainder by calling `refundETH()`.

## Finding Description
`refundETH()` is unconditionally open: [1](#0-0) 

ETH becomes stranded via `pay()`: when `token == WETH` and `nativeBalance >= value`, exactly `value` is deposited as WETH and forwarded to the pool; the surplus stays on the router: [2](#0-1) 

The `receive()` guard only blocks direct ETH transfers not originating from WETH: [3](#0-2) 

It does not prevent ETH from being stranded via the `payable` swap entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`), all of which are `external payable`: [4](#0-3) [5](#0-4) 

The safe pattern — appending `refundETH` to a `multicall` — is tested but never enforced by the contract: [6](#0-5) 

If a user calls a swap function directly (not via multicall) or omits `refundETH` from their multicall, the excess ETH persists on the router across transaction boundaries and is claimable by anyone.

## Impact Explanation
Direct loss of user principal. The victim permanently loses `msg.value - amountIn` ETH; the attacker gains it with a single zero-cost call to `refundETH()`. The stolen amount is unbounded and equals whatever excess the victim sent. This meets the Critical/High threshold for direct loss of user funds.

## Likelihood Explanation
The documented and expected usage pattern requires users to either use multicall with `refundETH` appended, or send exactly `amountIn` as ETH. Calling swap functions directly with excess ETH (e.g., from integrators, scripts, or less experienced users) is a realistic and common mistake. An attacker can monitor the mempool or poll `address(router).balance` to detect and back-run any such transaction with a single `refundETH()` call. The attack is repeatable, permissionless, and requires no special privileges.

## Recommendation
Track per-caller ETH deposits in transient storage at the start of each `payable` entry point and restrict `refundETH()` to return only the calling address's recorded deposit. Alternatively, automatically refund any remaining ETH balance to `msg.sender` at the end of each swap function, eliminating the need for a separate `refundETH` call entirely.

## Proof of Concept
1. User calls `router.exactInputSingle{value: 1 ether}(params)` where `params.tokenIn = WETH`, `params.amountIn = 0.5 ether`, without including `refundETH` in a multicall.
2. Inside the swap callback, `pay()` is invoked with `value = 0.5 ether`. Since `address(this).balance (1 ETH) >= value (0.5 ETH)`, it deposits exactly 0.5 ETH as WETH and transfers it to the pool. The remaining 0.5 ETH stays on the router.
3. The user's transaction completes. `address(router).balance == 0.5 ether`.
4. Attacker calls `router.refundETH()` in the next transaction.
5. `balance = 0.5 ether`; `_transferETH(attacker, 0.5 ether)` executes.
6. Attacker receives 0.5 ETH. Victim's excess ETH is gone.
