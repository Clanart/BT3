Audit Report

## Title
Excess native ETH sent to `exactOutputSingle`/`exactOutput` is silently stranded and stealable via permissionless `refundETH()` — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

## Summary
`exactOutputSingle` and `exactOutput` are `payable` and accept native ETH for WETH-input swaps. The `pay()` helper wraps only the exact pool-requested amount, leaving any excess `msg.value` as raw ETH in the router. Neither function refunds the surplus, and `refundETH()` is permissionless — any caller can drain the stranded ETH to themselves.

## Finding Description
When a user calls `exactOutputSingle{value: amountInMaximum}` with WETH as `tokenIn`, the ETH arrives in the router via the `payable` function call (bypassing the `receive()` guard that rejects non-WETH senders).

Inside the swap callback, `_justPayCallback` calls `pay()`: [1](#0-0) 

When `nativeBalance >= value`, exactly `value` (the actual pool-requested `amountIn`) is wrapped and forwarded. The remainder `msg.value - actualAmountIn` stays as raw ETH in the router.

`exactOutputSingle` then returns after the slippage check with no refund: [2](#0-1) 

`exactOutput` has the same pattern: [3](#0-2) 

`refundETH()` has no access control — it sends the entire router ETH balance to whoever calls it: [4](#0-3) 

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) does not protect against this because ETH sent via a `payable` function call is not routed through `receive()`. [5](#0-4) 

## Impact Explanation
Direct loss of user principal. A user calling `exactOutputSingle{value: amountInMaximum}` loses `amountInMaximum - actualAmountIn` ETH to the first address that calls `refundETH()` after the swap settles. For exact-output swaps the caller cannot know `actualAmountIn` before execution, so sending `msg.value = amountInMaximum` is the only viable pattern. The loss is proportional to the slippage budget and can be substantial on volatile assets. This meets the Critical/High direct-loss-of-user-principal threshold.

## Likelihood Explanation
The functions are individually `payable` and callable without multicall. No NatSpec on `exactOutputSingle` or `exactOutput` warns that excess ETH is not refunded or that `refundETH()` must be appended. EOA users and integrators calling these functions directly will silently lose the excess. MEV bots routinely monitor router contracts for stranded ETH and can extract it in the same block.

## Recommendation
Add an automatic refund at the end of both exact-output entry points when native ETH was used:

```solidity
// after _clearExpectedCallbackPool()
uint256 leftover = address(this).balance;
if (leftover > 0) _transferETH(msg.sender, leftover);
```

Alternatively, add a prominent NatSpec warning on both functions that callers **must** append `refundETH()` in a `multicall` when paying with native ETH, and consider reverting if `msg.value > 0` on non-multicall paths.

## Proof of Concept

1. Victim calls `router.exactOutputSingle{value: 1 ether}(params)` where `amountInMaximum = 1 ether` and the pool fills at `actualAmountIn = 0.6 ether`.
2. `pay()` wraps 0.6 ETH → WETH and transfers to pool. 0.4 ETH remains in the router. [1](#0-0) 
3. `exactOutputSingle` returns. Router holds 0.4 ETH with no owner record. [2](#0-1) 
4. Attacker calls `router.refundETH()`. Router sends 0.4 ETH to attacker. [4](#0-3) 
5. Victim loses 0.4 ETH.

```solidity
contract Thief {
    function steal(address router) external {
        IPeripheryPayments(router).refundETH();
        payable(msg.sender).transfer(address(this).balance);
    }
    receive() external payable {}
}
```
