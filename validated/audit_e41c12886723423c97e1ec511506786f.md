### Title
Wrapping Integer Overflow in Relative Timelock Arithmetic Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Conditions — (`File: crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

In `check_time_locks.rs`, when the `nowrap` flag is `false`, relative timelock comparisons use `wrapping_add` instead of `saturating_add`. An attacker who controls the CLVM condition arguments can supply a crafted `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` value that, when added to the coin's `confirmed_block_index` or `timestamp`, overflows and wraps to a small integer. The resulting comparison then passes unconditionally, allowing the coin to be spent at any block height or timestamp — a complete timelock bypass.

---

### Finding Description

`check_time_locks` accepts a `nowrap: bool` parameter that selects between two arithmetic strategies for relative timelock evaluation:

- `nowrap = true` → `saturating_add`: overflow clamps to `u32::MAX` / `u64::MAX`, so the timelock is always enforced.
- `nowrap = false` → `wrapping_add`: overflow wraps to a small value, so the comparison `prev_height < small_wrapped_value` evaluates to `false`, and **no error is returned** — the timelock silently passes.

The affected code for `ASSERT_HEIGHT_RELATIVE`:

```rust
// crates/chia-consensus/src/check_time_locks.rs, lines 55-68
if let Some(height_relative) = spend.height_relative {
    if nowrap {
        if prev_transaction_block_height
            < unspent.confirmed_block_index.saturating_add(height_relative)
        {
            return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
        }
    } else if prev_transaction_block_height
        < unspent.confirmed_block_index.wrapping_add(height_relative)  // BUG
    {
        return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
    }
}
```

The same pattern applies to `ASSERT_SECONDS_RELATIVE` (lines 70–77), and the inverse direction for `ASSERT_BEFORE_HEIGHT_RELATIVE` (lines 79–98) and `ASSERT_BEFORE_SECONDS_RELATIVE` (lines 100–111), where wrapping causes a valid spend to be **incorrectly rejected**.

The repository's own test suite explicitly documents and confirms this divergence:

```
# 10 + u32::MAX overflows to 9, wrapping: 15 < 9 -> Ok  (timelock bypassed!)
(make_test_conds(height_relative=0xFFFF_FFFF), 13, None),
``` [1](#0-0) 

The `height_relative` value is parsed from the CLVM condition `ASSERT_HEIGHT_RELATIVE` (opcode 82), which is fully attacker-controlled inside a spend bundle. The parsed value is a `u32` (up to `0xFFFF_FFFF`). [2](#0-1) 

The `check_time_locks` function is exposed as a Python binding (`py_check_time_locks`) and called by the Python full node with a caller-supplied `nowrap` value. [3](#0-2) 

---

### Impact Explanation

**Impact: High** — Timelock validation bypass enables unauthorized spend acceptance.

When `nowrap=false` is active (the pre-softfork code path), an attacker can craft a spend bundle containing `ASSERT_HEIGHT_RELATIVE(N)` where `N` is chosen so that `confirmed_block_index + N` overflows `u32`. The wrapped result is a small integer (e.g., 9), and the check `prev_height < 9` is false at any realistic block height, so the timelock is silently satisfied. The coin is accepted as spendable immediately, regardless of the intended lock duration.

This matches the allowed High impact: *"timelock or coin-id validation bypass enables unauthorized spend acceptance."*

The same overflow in `ASSERT_BEFORE_HEIGHT_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE` causes the inverse: a spend that should be accepted is incorrectly rejected, which can cause consensus divergence between nodes running different `nowrap` settings. [4](#0-3) 

---

### Likelihood Explanation

**Likelihood: Low-to-Medium.**

- The `nowrap=false` path must be active. This is the legacy pre-softfork path; if the softfork has fully activated on mainnet with `nowrap=true`, the bypass is not reachable. However, the code path remains present and callable via the Python binding.
- The attacker must own (or be able to create) a coin with a known `confirmed_block_index`, then craft a spend with a precisely chosen `height_relative` value. This is trivially computable: `N = u32::MAX - confirmed_block_index + 1` causes the sum to wrap to 0.
- No privileged access, leaked keys, or network-level attack is required — only a valid spend bundle submitted to the mempool.

---

### Recommendation

Replace `wrapping_add` with `saturating_add` in all four relative timelock checks, unconditionally. The `nowrap` branch distinction should be removed entirely, or the `nowrap=false` path should be made unreachable:

```rust
// ASSERT_HEIGHT_RELATIVE
if prev_transaction_block_height
    < unspent.confirmed_block_index.saturating_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}

// ASSERT_SECONDS_RELATIVE
if timestamp < unspent.timestamp.saturating_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}

// ASSERT_BEFORE_HEIGHT_RELATIVE
if prev_transaction_block_height
    >= unspent.confirmed_block_index.saturating_add(before_height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertBeforeHeightRelativeFailed));
}

// ASSERT_BEFORE_SECONDS_RELATIVE
if timestamp >= unspent.timestamp.saturating_add(before_seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertBeforeSecondsRelativeFailed));
}
```

This mirrors the `nowrap=true` branch exactly and is the semantically correct behavior: a timelock that would require waiting longer than `u32::MAX` blocks should never be satisfiable, not silently bypassed. [4](#0-3) 

---

### Proof of Concept

**Setup:** Coin confirmed at block height `confirmed_block_index = 10`.

**Attacker crafts:** `ASSERT_HEIGHT_RELATIVE(0xFFFF_FFF6)` (i.e., `4294967286`).

**Arithmetic (wrapping, `nowrap=false`):**
```
10 + 4294967286 = 4294967296 → wraps to 0  (mod 2^32)
```

**Check:** `prev_transaction_block_height < 0` → `false` for any block height → **no error returned → timelock passes**.

**Expected (saturating, `nowrap=true`):**
```
10 + 4294967286 = u32::MAX (saturated)
```
**Check:** `prev_transaction_block_height < u32::MAX` → `true` → **`AssertHeightRelativeFailed` returned → spend rejected**.

This exact case is confirmed by the repository's own test: [5](#0-4) 

The attacker submits a spend bundle with this condition via the standard P2P mempool interface. No special privileges are required.

#[cfg(feature = "py-bindings")]
#[pyfunction]
#[pyo3(name = "check_time_locks")]
#[allow(clippy::needless_pass_by_value)] // pyo3 prefers pass_by_value
pub fn py_check_time_locks(
    removal_coin_records: HashMap<Bytes32, CoinRecord>,
    bundle_conds: &OwnedSpendBundleConditions,
    prev_transaction_block_height: u32,
    timestamp: u64,
    nowrap: bool,
) -> PyResult<Option<u32>> {
    let res = check_time_locks(
        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,
        nowrap,
    );

    match res {
        Ok(()) => Ok(None),
        Err(ec) => Ok(Some(ec.error_code().into())),
    }
}
