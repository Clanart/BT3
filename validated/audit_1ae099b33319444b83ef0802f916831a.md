### Title
Missing Declared Class Hash Validation in `execute_replace_class` Allows Permanent Contract Fund Freezing - (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash provided by a contract corresponds to a previously declared class. This allows any contract to replace its class with an arbitrary, undeclared class hash, permanently rendering the contract unusable and freezing any funds held in its storage.

---

### Finding Description

The `execute_replace_class` function processes the `replace_class` syscall, which allows a contract to replace its own class hash. However, the function does not verify that `request.class_hash` corresponds to a declared class in `contract_class_changes`. The missing check is explicitly acknowledged with a TODO comment: [1](#0-0) 

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The function proceeds to commit the new (potentially undeclared) class hash into `contract_state_changes` with no validation whatsoever.

When a contract's class is replaced with an undeclared hash, any subsequent call to that contract invokes `execute_entry_point`, which reads the compiled class hash via `dict_read` on `contract_class_changes`: [2](#0-1) 

Since the class hash is undeclared, `dict_read` returns `0`. The subsequent `find_element` call then searches for a compiled class with hash `0`, which does not exist, causing the OS execution to fail. The contract becomes permanently unusable, and any funds stored in its storage are permanently frozen.

This is the direct analog to the SPL freeze authority issue: just as a token with an active freeze authority can have its accounts frozen with no recourse, a StarkNet contract can have its class replaced with an undeclared hash with no recourse, permanently locking all funds inside it.

---

### Impact Explanation

Any funds held in the storage of a contract whose class has been replaced with an undeclared hash are permanently inaccessible. The contract cannot be called, upgraded, or recovered. This matches the **Critical — Permanent freezing of funds** impact category. A malicious sequencer can include a `replace_class` transaction with an undeclared class hash for any contract, and the OS will accept it (since there is no check), permanently bricking the contract and freezing its funds.

---

### Likelihood Explanation

The `replace_class` syscall is accessible to any contract at runtime: [3](#0-2) 

A malicious sequencer can include transactions that call `replace_class` with undeclared class hashes for any contract, permanently freezing its funds. Additionally, a legitimate contract with a vulnerability that allows arbitrary `replace_class` calls (e.g., an access-control bug) could have its funds permanently frozen by an unprivileged attacker. The TODO comment in the code explicitly acknowledges this missing check, confirming it is a known, unmitigated gap in the current OS implementation.

---

### Recommendation

In `execute_replace_class`, add a check to verify that the new class hash corresponds to a declared class in `contract_class_changes` before updating the contract state. Specifically, perform a `dict_read` on `contract_class_changes` with the new class hash and assert that the result is non-zero (i.e., the class has been declared). This mirrors the enforcement already present in `execute_entry_point`, which relies on the class being in `contract_class_changes`.

---

### Proof of Concept

1. A contract (e.g., a vault holding user funds) calls the `replace_class` syscall with an arbitrary, undeclared class hash (e.g., `0xdeadbeef`).
2. `execute_replace_class` in `syscall_impls.cairo` processes this without checking if the class hash is declared (line 898 TODO). [4](#0-3) 
3. The contract's `class_hash` in `contract_state_changes` is updated to `0xdeadbeef`.
4. The block containing this transaction proves successfully — `execute_replace_class` itself does not invoke `execute_entry_point` for the new class.
5. In a subsequent block, any call to the contract invokes `execute_entry_point`, which reads `compiled_class_hash = 0` (since `0xdeadbeef` is not in `contract_class_changes`). [5](#0-4) 
6. `find_element` fails to locate a compiled class with hash `0`, making any block containing a call to this contract unprovable.
7. The sequencer is forced to exclude all calls to this contract from future blocks.
8. The contract is permanently unusable, and all funds in its storage are permanently frozen with no recovery path.
