Audit Report

## Title
Missing Deadline Check in `addLiquidityExactShares` Allows Stale Pending Transactions to Execute at Unfavorable Pool Prices — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary
Both overloads of `MetricOmmPoolLiquidityAdder.addLiquidityExactShares` accept no `deadline` parameter and perform no `block.timestamp` comparison, allowing a pending mempool transaction to be mined arbitrarily late. When the pool price has moved significantly by execution time, the user's shares are deposited into bins that are entirely off-price, resulting in a fully single-sided deposit with a composition radically different from what the user intended and an LP position that earns no fees at the current market price.

## Finding Description
`MetricOmmSimpleRouter` guards every swap entry point with `_checkDeadline(params.deadline)` (lines 68, 93, 131, 155). `MetricOmmPoolLiquidityAdder` provides no equivalent protection in either `addLiquidityExactShares` overload (lines 56–68, 71–81). The only pre-call guards are `_validateOwner` (checks `owner != address(0)`) and `_validateDeltas` (checks array length consistency). The callback enforces `maxAmountToken0`/`maxAmountToken1` caps (lines 165–167), but these bound the *quantity* of tokens pulled, not *when* the transaction executes or *which bins* are active at that time. A user who submits a transaction targeting bins near the current price will have those same bins filled regardless of how far the price has moved by the time the transaction is mined. `addLiquidityWeighted` has a partial mitigation via `_validateBinAndBinPosition` (lines 263–286), which reverts if the pool cursor has moved outside user-supplied bounds; `addLiquidityExactShares` has no such check, making it the more exposed path.

## Impact Explanation
In a bin-based AMM, each bin is active only when the pool price is inside that bin's range. If the price has moved far from the user's target bins by execution time, all target bins may be above or below the current price, causing the pool to request only one token. The user receives a fully single-sided LP position concentrated in bins that are entirely off-price, earning zero fees until the market price returns to those bins. The `maxAmountToken0`/`maxAmountToken1` caps do not prevent this: they were sized to accommodate the expected balanced composition, so a fully single-sided deposit can still fall within the caps while being entirely unintended. This constitutes a medium-severity loss of expected LP yield and unintended capital allocation for any user of `addLiquidityExactShares`.

## Likelihood Explanation
The scenario requires only a low-fee transaction sitting in the mempool during a period of price volatility — a common occurrence on congested networks. No privileged access, malicious setup, or non-standard token behavior is required. Any unprivileged user of `addLiquidityExactShares` is exposed.

## Recommendation
Add a `deadline` parameter to both `addLiquidityExactShares` overloads and validate it before any state-changing call:

```solidity
function addLiquidityExactShares(
    address pool,
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 maxAmountToken0,
    uint256 maxAmountToken1,
+   uint256 deadline,
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
+   if (block.timestamp > deadline) revert DeadlineExpired();
    _validateOwner(owner);
    ...
}
```

Apply the same change to the `addLiquidityWeighted` overloads for defense-in-depth, even though they already have cursor-bounds validation.

## Proof of Concept
1. Alice targets bins `[-1, 0, 1]` near the current ETH/USDC price of $2,000. She calls `addLiquidityExactShares` with `maxAmountToken0 = 1,000 USDC` and `maxAmountToken1 = 0.5 ETH`, expecting a balanced deposit.
2. Alice submits with a low gas fee. The transaction sits in the mempool for two days.
3. ETH price drops to $1,200. The pool cursor is now far below bin `-1`.
4. The network mines Alice's transaction. All three target bins are now above the current price.
5. The pool callback requests `1,000 USDC` and `0 ETH` — within Alice's caps.
6. Alice receives an LP position that is 100% USDC concentrated in bins priced at $1,800–$2,200, earning zero fees at the current $1,200 price. Had Alice's transaction been rejected after a deadline, she could have resubmitted at the new price with appropriate bin targets.
