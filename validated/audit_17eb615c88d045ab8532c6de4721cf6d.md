### Title
Missing `chunk.shard_id == shard_id` binding in `set_state_header` allows a malicious peer to install shard X's trie state into shard Y's column — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` from a peer and stores it under `StateHeaderKey(shard_id, sync_hash)`. It validates the embedded chunk's internal consistency (`validate_chunk_proofs`) and its Merkle inclusion in the block (`verify_path` against `chunk_headers_root`), but **never asserts that `chunk.shard_id() == shard_id`**. Because `verify_path` (unlike `verify_path_with_index`) does not check the leaf's position in the tree, a chunk from shard X at position X passes the Merkle check even when `shard_id=Y` was requested. The corrupted header is then used by `apply_state_part` and `set_state_finalize` to write shard X's trie nodes and state root into shard Y's `ShardUId` column.

### Finding Description

The attack proceeds as follows:

1. **Attacker is a peer** the syncing node contacts for state sync (no validator privileges required).
2. The syncing node requests state for `shard_id=Y`. The attacker responds with a `ShardStateSyncResponseHeaderV2` whose `chunk` field is a valid, on-chain chunk for shard X (`chunk.shard_id() = X`), with the correct Merkle path for shard X's position in the block.
3. **`validate_chunk_proofs`** passes: it checks hash, tx_root, and receipts root of the chunk, but has no `shard_id` parameter and makes no cross-check against the caller's `shard_id=Y`. [2](#0-1) 
4. **`verify_path` against `chunk_headers_root`** passes: `chunk_headers_root` covers all shards; the attacker supplies the correct Merkle path for chunk X at position X. `verify_path` only checks `compute_root_from_path(path, hash) == root` — it does not verify that the path position corresponds to shard Y's index. [3](#0-2) [7](#0-6) 
5. **Receipt proof validation (step 4)** uses `shard_id=Y` for the hash (`ReceiptList(shard_id, receipts)`), but the attacker supplies valid on-chain receipts for shard Y with correct Merkle paths against `prev_chunk_outgoing_receipts_root`. This passes. [8](#0-7) 
6. **State root node validation (step 5)** checks `state_root_node` against `chunk_inner.prev_state_root()` — shard X's state root. The attacker provides a valid state root node for shard X. This passes. [9](#0-8) 
7. **Header stored under `StateHeaderKey(shard_id=Y, sync_hash)`** with shard X's chunk embedded. [10](#0-9) 
8. **`set_state_part`** reads the stored header, extracts `state_root` from shard X's chunk, and calls `validate_state_part(shard_id=Y, state_root_X, ...)`. The attacker provides valid state parts for shard X. Parts are stored under `StatePartKey(sync_hash, shard_id=Y, part_id)`. [11](#0-10) 
9. **`apply_state_part(shard_id=Y, state_root_X, ...)`** computes `shard_uid` for shard Y, then writes shard X's trie nodes into shard Y's `ShardUId` column and flat storage. [5](#0-4) 
10. **`set_state_finalize`** computes `shard_uid` for shard Y, then calls `apply_chunk` with `RuntimeStorageConfig::new(chunk_header.prev_state_root(), ...)` — shard X's state root — under shard Y's `shard_uid`. The resulting `ChunkExtra` for shard Y has shard X's state root. [6](#0-5) 

### Impact Explanation

The exact corrupted value is: **shard Y's `ChunkExtra.state_root` is set to shard X's state root**, and **shard Y's `DBCol::State` column contains shard X's trie nodes**. The node then operates on shard Y with shard X's state, causing it to produce incorrect block/chunk results for shard Y, reject valid transactions, or accept invalid ones. The corruption is committed to disk and persists across restarts.

### Likelihood Explanation

Any peer the syncing node connects to for state sync can execute this. State sync peers are not required to be validators. The attacker only needs to serve a crafted response to a state sync request — a standard network-level capability. The attack requires knowledge of a valid on-chain chunk for shard X (publicly available) and valid receipts for shard Y (also publicly available from the chain).

### Recommendation

Add an explicit shard_id binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// ADD THIS:
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk.shard_id does not match requested shard_id".into(),
    ));
}
``` [12](#0-11) 

Similarly, add a corresponding check for `prev_chunk_header.shard_id()` when it is present.

### Proof of Concept

A test-loop reproducer would:
1. Produce a multi-shard chain (at least 2 shards).
2. Obtain a valid `ShardStateSyncResponseHeader` for shard X (e.g., via `get_state_response_header(shard_id=X, sync_hash)`).
3. Call `set_state_header(shard_id=Y, sync_hash, header_for_shard_X)` on a second client.
4. Assert the call returns an `Err` containing a shard_id mismatch message.
5. Currently, the call returns `Ok(())`, demonstrating the missing guard.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
