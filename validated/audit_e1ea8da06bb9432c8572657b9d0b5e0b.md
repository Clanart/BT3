### Title
`set_locked_tokens` Absolute-Value Overwrite Corrupts Bridge Collateral Accounting â€” (File: `near/omni-bridge/src/token_lock.rs`)

---

### Summary

The `set_locked_tokens` admin function overwrites the `locked_tokens` collateral map with a caller-supplied **absolute value**. When the `TokenLockController` reads the current state off-chain, computes a correction, and submits the transaction, any bridge operations that land between the off-chain read and the on-chain write will be silently discarded. An unprivileged attacker who observes the pending correction can execute bridge operations in that window to make the resulting `locked_tokens` value diverge from the true locked supply, enabling unbacked token release.

---

### Finding Description

`set_locked_tokens` in `near/omni-bridge/src/token_lock.rs` (lines 38â€“44) is the only public write path for the `locked_tokens` map that accepts an arbitrary absolute value:

```rust
#[access_control_any(roles(Role::DAO, Role::TokenLockController))]
pub fn set_locked_tokens(&mut self, args: Vec<SetLockedTokenArgs>) {
    for arg in args {
        self.locked_tokens
            .insert(&(arg.chain_kind, arg.token_id), &arg.amount.0);
    }
}
``` [1](#0-0) 

All other mutations of `locked_tokens` are differential: `lock_tokens` adds to the current value and `unlock_tokens` subtracts from it, both operating atomically on-chain. [2](#0-1) 

The `locked_tokens` map is the bridge's collateral ledger. `unlock_tokens` enforces `available >= amount` before releasing tokens, so the map's accuracy is the sole on-chain guard against releasing more tokens than are locked on the remote chain. [3](#0-2) 

The `TokenLockController` role exists precisely to call `set_locked_tokens` for corrections and migrations (confirmed by the `Role` enum and test helpers). [4](#0-3) 

The attack window is the gap between when the `TokenLockController` reads `locked_tokens` off-chain and when the `set_locked_tokens` transaction is finalized on-chain. During that window, any user can call `ft_transfer_call` â†’ `init_transfer` (bridge-out) or `fin_transfer` (bridge-in), both of which mutate `locked_tokens` differentially. When `set_locked_tokens` lands, it blindly overwrites the current value with the stale absolute figure, erasing those interim changes.

### Impact Explanation

**Scenario â€” inflated locked_tokens leading to unbacked release:**

1. `locked_tokens[(Eth, token)] = 100`. The `TokenLockController` reads this off-chain and prepares a downward correction to `80` (e.g., because 20 tokens were already released on the Ethereum side).
2. Before the transaction lands, an attacker calls `ft_transfer_call` â†’ `init_transfer` bridging out 10 tokens. `lock_tokens` decrements the map: `locked_tokens[(Eth, token)] = 90`.
3. `set_locked_tokens` executes and writes `80` unconditionally.
4. The true locked supply is now `90 âˆ’ 10 = 80 âˆ’ 10 = 70` (the 10 tokens the attacker bridged out are in-flight to Ethereum), but the contract records `80`.
5. The 10-unit surplus allows a subsequent `fin_transfer` to pass the `available >= amount` check and release tokens to a NEAR recipient even though the corresponding Ethereum-side collateral does not exist.

This is a direct path to unbacked token minting / unauthorized release of bridged assets, matching the **High** impact class: *"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."*

---

### Likelihood Explanation

The `TokenLockController` role is a live production role (present in the `Role` enum, granted during deployment, and exercised in integration tests including `btc_storage_fix`). [5](#0-4) 

Corrections are expected to be sent periodically (e.g., after migrations, after cross-chain accounting reconciliation). The attacker needs no privileged access â€” only the ability to call the public `ft_transfer_call` entry point and observe when a `set_locked_tokens` transaction is pending. The TOCTOU window exists regardless of whether NEAR has a traditional mempool; the gap between the off-chain read and on-chain finality is sufficient.

---

### Recommendation

Replace the absolute-value setter with differential update functions that add or subtract from the current value atomically on-chain:

```rust
pub fn adjust_locked_tokens(
    &mut self,
    chain_kind: ChainKind,
    token_id: AccountId,
    delta: i128,   // positive = increase, negative = decrease
) { ... }
```

If an absolute override is ever genuinely needed (e.g., emergency recovery), it should require a time-lock or multi-sig threshold to prevent race exploitation.

2. Attacker observes the pending correction and immediately calls `ft_transfer_call` on the USDC token contract with `msg = InitTransfer { recipient: Eth(attacker_eth), amount: 100_000, fee: 0 }`. `lock_tokens` runs: `locked_tokens[(Eth, USDC)] = 900_000`.
3. `set_locked_tokens` lands: `locked_tokens[(Eth, USDC)] = 800_000`.
4. True collateral = `900_000 âˆ’ 100_000 = 800_000 âˆ’ 100_000 = 700_000`, but the contract records `800_000`.
5. Attacker (or any user) submits a `fin_transfer` for `100_000` USDC to a NEAR address. `unlock_tokens` checks `800_000 >= 100_000` âœ“ and releases the tokens.
6. The bridge has now released `100_000` USDC that has no corresponding locked collateral on Ethereum â€” unbacked supply has been created.

    fn lock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(current_amount) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        let new_amount = current_amount
            .checked_add(amount)
            .near_expect(TokenLockError::LockedTokensOverflow);

        self.locked_tokens.insert(&key, &new_amount);

        LockAction::Locked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }

    fn unlock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(available) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        require!(
            available >= amount,
            TokenLockError::InsufficientLockedTokens.as_ref()
        );

        let remaining = available - amount;
        self.locked_tokens.insert(&key, &remaining);

        LockAction::Unlocked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }
