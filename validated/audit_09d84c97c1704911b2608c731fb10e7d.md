### Title
Missing Validation of `class_hash` in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

`execute_replace_class` in `syscall_impls.cairo` does not validate that the provided `class_hash` corresponds to a declared contract class before writing it to the contract state. This is the direct analog to the "empty listings" bug: just as `_createListing()` accepted a zero-length `tokenIds` array without validation, `execute_replace_class` accepts any arbitrary `class_hash` — including `0` or any undeclared hash — without validation. A contract can call `replace_class(0)`, permanently bricking itself and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class`, the `class_hash` from the syscall request is written directly to the contract state with no check that it is a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The TODO comment explicitly acknowledges this missing check. [1](#0-0) 

After a successful `replace_class(0)` call, the contract's class hash is permanently set to `0` in the committed state. Any subsequent call to this contract reaches `execute_entry_point`, which does:

1. `compiled_class_hash = dict_read{dict_ptr=contract_class_changes}(key=0)` → returns `0` (default for undeclared class)
2. `find_element(array_ptr=compiled_class_facts, key=0)` → panics because no compiled class fact with hash `0` exists [2](#0-1) 

`find_element` panicking causes the proof to fail. An honest sequencer cannot include any transaction that calls the bricked contract. The contract's funds are permanently inaccessible.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any funds held by a contract that has called `replace_class` with an undeclared class hash are permanently frozen. No valid STARK proof can be generated for any transaction that subsequently calls the contract, because `find_element` in `execute_entry_point` will always panic when looking up the non-existent compiled class fact. The contract becomes permanently uncallable on-chain.

---

### Likelihood Explanation

**Moderate.** The attack path is:

- A malicious contract deployer deploys a contract with a public `freeze()` function that calls `replace_class(0)`.
- Users are tricked into depositing funds into the contract (e.g., it appears to be a legitimate vault or escrow).
- The attacker calls `freeze()`.
- The OS executes `execute_replace_class` with `class_hash = 0` — no validation occurs, the state is committed.
- All funds are permanently frozen.

Additionally, a buggy contract could accidentally call `replace_class` with an invalid hash, freezing its own funds unintentionally. The missing check is explicitly flagged as a known TODO by the developers, confirming awareness of the gap.

---

### Recommendation

Add a validation check in `execute_replace_class` to ensure `class_hash` corresponds to a declared contract class before updating the state. Concretely:

- Assert that `dict_read{dict_ptr=contract_class_changes}(key=class_hash)` returns a non-zero compiled class hash, **or**
- Assert that `class_hash` exists in the `compiled_class_facts` bundle.

This mirrors the fix described in the external report: just as `_createListing()` needed `assert tokenIds.length > 0`, `execute_replace_class` needs `assert class_hash is declared`.

---

### Proof of Concept

1. Attacker deploys a malicious contract `MaliciousVault` with a public `freeze()` function that calls `replace_class(0)`.
2. Attacker advertises `MaliciousVault` as a legitimate yield vault; users deposit funds.
3. Attacker calls `freeze()` on `MaliciousVault`.
4. The OS executes `execute_replace_class` with `class_hash = 0`. [1](#0-0) 
5. No validation occurs; `contract_state_changes` is updated with `class_hash = 0` for `MaliciousVault`.
6. Any subsequent transaction calling `MaliciousVault` reaches `execute_entry_point`:
   - `dict_read(key=0)` returns `0` (compiled_class_hash = 0).
   - `find_element(key=0)` panics — no compiled class fact with hash `0` exists. [3](#0-2) 
7. Proof generation fails. The sequencer cannot include any call to `MaliciousVault`. All deposited funds are permanently frozen.
