### Title
`ProofOfInclusion::valid()` Performs a Tautological Root Check, Accepting Any Internally-Consistent Forged Proof - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate verifies only the internal hash-chain consistency of a proof. Its final comparison — `existing_hash == self.root_hash()` — is a mathematical tautology: after the loop, `existing_hash` is always equal to `last.combined_hash`, which is exactly what `root_hash()` returns. The function never validates the proof against any external trusted root. An unprivileged attacker who can supply a `ProofOfInclusion` object (via Python bindings, `from_bytes()`, or network deserialization) can forge a proof for any arbitrary `node_hash` that will pass `valid()`, allowing them to prove inclusion of data that is not in the committed tree.

### Finding Description

**Root cause — tautological final check in `valid()`:**

```rust
// crates/chia-datalayer/src/merkle/proof_of_inclusion.rs

pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // <-- sourced entirely from the proof itself
    } else {
        self.node_hash
    }
}

pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;   // after loop: existing_hash == last.combined_hash
    }

    existing_hash == self.root_hash()      // always true: both sides == last.combined_hash
}
```

After the loop body enforces `calculated_hash == layer.combined_hash` at every step, `existing_hash` at loop exit equals `last_layer.combined_hash`. `root_hash()` also returns `last_layer.combined_hash`. The final comparison is therefore always `true` whenever the loop completes without an early return. For an empty `layers` vector, both sides equal `self.node_hash`, so it is also always `true`.

**Consequence:** `valid()` is equivalent to "the hash chain is internally self-consistent." It never checks whether the proof's claimed root matches any externally trusted, on-chain-committed root. An attacker can construct a `ProofOfInclusion` with an arbitrary `node_hash` (representing a key-value pair not in the real tree), build a single layer with any `other_hash`, compute `combined_hash = calculate_internal_hash(node_hash, side, other_hash)`, and `valid()` returns `true`.

**Attacker-controlled entry path:**

The struct is `Streamable` and exposed via Python bindings with `from_bytes()` / `from_py_object`:

```python
# wheel/python/chia_rs/datalayer.pyi
class ProofOfInclusion:
    node_hash: bytes32
    layers: list[ProofOfInclusionLayer]
    def valid(self) -> bool: ...
    @classmethod
    def from_bytes(cls, blob: bytes) -> Self: ...
```

Any caller that receives a `ProofOfInclusion` from an untrusted source (peer, RPC, DataLayer sync) and calls only `proof.valid()` — without separately comparing `proof.root_hash()` against the on-chain committed root — will accept the forged proof.

### Impact Explanation

An attacker can forge a `ProofOfInclusion` for any `node_hash` value (any key-value pair) and have it accepted by any verifier that relies solely on `proof.valid()`. In the DataLayer context, this allows proving inclusion of data that was never committed to the on-chain root, enabling invalid state to be accepted as proven. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, letting untrusted input prove invalid state.**

### Likelihood Explanation

The `valid()` method name strongly implies a complete validity check. The Python API exposes `valid()` and `root_hash()` as separate methods with no documentation requiring callers to check both. The fuzz target and all Rust/Python tests call only `proof.valid()` without comparing against an external root, establishing a usage pattern that omits the root check. Any downstream Chia DataLayer client that follows this pattern is vulnerable.

### Recommendation

`valid()` must accept an external trusted root and compare against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // compare against externally-supplied root
}
```

Alternatively, rename the current `valid()` to `is_internally_consistent()` and add a separate `valid(trusted_root: &Hash) -> bool` that performs the full check. Update all call sites — including the Python bindings — to require the trusted root parameter.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Attacker wants to forge a proof that fake_node_hash is in the tree
fake_node_hash = bytes([0xAB] * 32)
other_hash     = bytes([0xCD] * 32)

# Compute a valid combined_hash for one layer (left side)
h = hashlib.sha256()
h.update(b'\x00')          # internal node prefix used by calculate_internal_hash
h.update(fake_node_hash)
h.update(other_hash)
combined_hash = h.digest()  # exact hash depends on calculate_internal_hash impl

layer = ProofOfInclusionLayer(
    other_hash_side=1,       # right side
    other_hash=other_hash,
    combined_hash=combined_hash,
)
forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True even though fake_node_hash is not in any real tree
assert forged_proof.valid(), "Forged proof accepted"
# root_hash() returns the attacker-chosen combined_hash, not any on-chain root
print("Forged root:", forged_proof.root_hash().hex())
```

The proof passes `valid()` because the loop check and the final tautological comparison both succeed. A verifier that does not separately compare `forged_proof.root_hash()` against the on-chain committed root will accept this as a valid inclusion proof. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
