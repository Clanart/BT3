### Title
`ProofOfInclusion::valid()` Does Not Bind Proof to Any Expected Tree Root, Enabling Forged Inclusion Acceptance — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary
`ProofOfInclusion::valid()` only verifies the internal hash-chain consistency of the proof against itself. It never checks the computed root against any externally-supplied, trusted tree root. Any caller that uses `valid()` as the sole verification step will accept a completely fabricated `ProofOfInclusion` that is internally consistent but proves membership in an attacker-controlled tree, not the legitimate DataLayer store.

### Finding Description

`ProofOfInclusion` is a `Streamable` struct (deserializable from raw bytes) with two public methods exposed through the Python wheel binding:

```rust
pub fn root_hash(&self) -> Hash { ... }
pub fn valid(&self) -> bool { ... }
```

`valid()` is implemented as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()
}
``` [1](#0-0) 

And `root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // <-- derived entirely from the proof itself
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

The final check `existing_hash == self.root_hash()` is tautological: after the loop passes, `existing_hash` equals `layers.last().combined_hash`, which is exactly what `root_hash()` returns. The function therefore only verifies that the proof's own hash chain is self-consistent — it never compares the computed root against any trusted, externally-known tree root.

This is the direct analog of the reported bug: `commitReaction` checks that the user joined `_gameId` (one context) but never verifies that `_questionId` belongs to `_gameId` (cross-context). Here, `valid()` checks that the proof is internally consistent (one context) but never verifies the proof belongs to the expected DataLayer tree root (cross-context).

`ProofOfInclusion` is `Streamable` and is exposed to Python via the wheel binding: [3](#0-2) 

An attacker can craft arbitrary bytes that deserialize to a `ProofOfInclusion` with any `node_hash` and any `layers` whose hash chain is internally consistent. `valid()` will return `true` for this forged proof.

### Impact Explanation

Any DataLayer client or verifier that calls `proof.valid()` as the sole check — without separately asserting `proof.root_hash() == expected_root` — will accept forged inclusion proofs for arbitrary key-value pairs in any tree. This lets untrusted input prove invalid DataLayer state, matching the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion… or lets untrusted input prove invalid state."*

### Likelihood Explanation

The API design actively invites this misuse. The method is named `valid()` and returns `bool`, strongly implying it fully validates the proof. There is no parameter for an expected root hash, so callers have no in-API reminder that a separate root check is required. The Python type stub exposes `valid() -> bool` with no documentation of this limitation. [4](#0-3) 

The existing Rust unit tests call `proof.valid()` without checking the root hash against any external value, demonstrating the pattern that callers naturally follow: [5](#0-4) 

### Recommendation

`valid()` should accept an expected root hash parameter and compare the computed root against it:

```rust
pub fn valid_for_root(&self, expected_root: &Hash) -> bool {
    // ... existing chain check ...
    existing_hash == *expected_root
}
```

Alternatively, rename the current method to `is_internally_consistent()` to make its limited scope explicit, and add a separate `valid_for_root(expected: &Hash) -> bool` method. Update all callers and the Python binding accordingly.

### Proof of Concept

```rust
// Attacker constructs a fake proof for node_hash = FAKE_HASH
// in a tree they control (not the legitimate DataLayer store).
let fake_node_hash: Hash = [0xAA; 32];
let fake_other_hash: Hash = [0xBB; 32];
let fake_combined = calculate_internal_hash(&fake_node_hash, Side::Left, &fake_other_hash);

let forged_proof = ProofOfInclusion {
    node_hash: fake_node_hash,
    layers: vec![ProofOfInclusionLayer {
        other_hash_side: Side::Left,
        other_hash: fake_other_hash,
        combined_hash: fake_combined,
    }],
};

// valid() returns true — no external root is checked
assert!(forged_proof.valid());
// root_hash() returns the attacker-controlled combined hash, not the legitimate store root
assert_eq!(forged_proof.root_hash(), fake_combined);
```

A verifier that only calls `forged_proof.valid()` accepts this as a valid proof of inclusion for `fake_node_hash` in the legitimate DataLayer store, without any check that `fake_combined` matches the on-chain committed root. [1](#0-0)
