### Title
Irrecoverable Fund Lock via Blacklisted Recipient in `finTransfer` Push-Payment Pattern — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.finTransfer()` uses a push-payment model to deliver bridged assets directly to `payload.recipient`. For native ERC-20 tokens (e.g., USDT, USDC), this is done via `IERC20.safeTransfer()`. If the recipient address is blacklisted by the token contract, the call reverts unconditionally. Because the source-chain tokens are already burned or locked before `finTransfer` is ever called, and no on-chain refund path exists, the user's funds are permanently frozen.

---

### Finding Description

In `OmniBridge.finTransfer()`, after signature verification, the contract pushes tokens directly to `payload.recipient`:

```solidity
// OmniBridge.sol lines 317-355
if (payload.tokenAddress == address(0)) {
    (bool success, ) = payload.recipient.call{value: payload.amount}("");
    if (!success) revert FailedToSendEther();
} else if (multiToken.tokenAddress != address(0)) {
    IERC1155(multiToken.tokenAddress).safeTransferFrom(
        address(this), payload.recipient, multiToken.tokenId, payload.amount, ""
    );
} else if (customMinters[payload.tokenAddress] != address(0)) {
    ICustomMinter(customMinters[payload.tokenAddress]).mint(
        payload.tokenAddress, payload.recipient, payload.amount
    );
} else if (isBridgeToken[payload.tokenAddress]) {
    IBridgeToken(payload.tokenAddress).mint(payload.recipient, payload.amount);
} else {
    IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
}
```

For the native ERC-20 branch (the `else` at line 351), `SafeERC20.safeTransfer` will revert if the token contract (USDT, USDC) has blacklisted `payload.recipient`. Because Solidity reverts roll back all state changes — including the `completedTransfers[payload.destinationNonce] = true` write at line 287 — the nonce is never consumed, and the relayer can retry indefinitely. However, every retry will revert for the same reason: the recipient is permanently blacklisted.

The signed `TransferMessagePayload` is MPC-produced and binds `recipient` immutably. There is no `cancelTransfer`, `changeRecipient`, or refund function anywhere in the EVM contracts or the NEAR bridge (`near/omni-bridge/src/lib.rs`) that would allow recovery of the source-chain funds once the originating `initTransfer` has burned or locked them.

For the ETH branch (line 319), the same permanent-freeze outcome occurs if `payload.recipient` is a smart-contract wallet whose `receive()` always reverts (e.g., a paused multisig or a contract that requires more than the forwarded gas).

For the ERC-1155 branch (line 324), `safeTransferFrom` will revert if `payload.recipient` is a contract that does not implement `IERC1155Receiver`, again permanently freezing funds.

---

### Impact Explanation

**Permanent freezing / irrecoverable lock of user funds.**

1. User calls `initTransfer` on the source chain → tokens are burned (bridge tokens) or locked (native ERC-20s) in the bridge contract.
2. The NEAR MPC signs a `TransferMessagePayload` binding `recipient` to the user's destination address.
3. A relayer calls `finTransfer` on the EVM destination chain.
4. `safeTransfer` to a USDT/USDC-blacklisted address reverts every time.
5. The source-chain tokens are already gone; no refund path exists.
6. Funds are permanently unclaimable.

This matches the allowed impact: **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."**

---

### Likelihood Explanation

- USDT and USDC are among the most commonly bridged ERC-20 tokens and both implement address blacklisting enforced at the token level.
- A user can be blacklisted *after* initiating a bridge transfer (e.g., due to a compliance action), making this a realistic race condition.
- The ERC-1155 and ETH-reverting-receiver variants require only that the recipient is a contract without the appropriate interface, which is a common deployment scenario.
- No privileged role or external compromise is required; the attacker-controlled input is simply the `recipient` field chosen at `initTransfer` time.

---

### Recommendation

Replace the push-payment model with a pull pattern:

1. Instead of calling `safeTransfer` / `mint` / `.call` directly to `payload.recipient` inside `finTransfer`, credit the amount to an internal `claimable[recipient][token]` mapping.
2. Expose a separate `claimTokens(address token)` function that lets the recipient withdraw at their own discretion.
3. This ensures a blacklisted or reverting recipient cannot prevent the nonce from being finalized, and the funds remain claimable if/when the blacklist is lifted.

Alternatively, wrap each push in a `try/catch` (for Solidity external calls) and emit a `TransferFailed` event so that a governance-controlled rescue path can be triggered.

---

### Proof of Concept

1. Alice holds USDT and calls `initTransfer` on the NEAR bridge, specifying her Ethereum address as recipient. USDT is locked in the bridge.
2. NEAR MPC signs a `TransferMessagePayload` with `tokenAddress = USDT_EVM`, `recipient = Alice_EVM`, `amount = X`.
3. Before the relayer submits `finTransfer`, USDT's issuer blacklists `Alice_EVM`.
4. Relayer calls `OmniBridge.finTransfer(sig, payload)`.
5. Execution reaches line 351: `IERC20(USDT).safeTransfer(Alice_EVM, X)` → USDT reverts with `Blacklisted`.
6. The entire transaction reverts; `completedTransfers[nonce]` is rolled back.
7. Every subsequent relay attempt produces the same revert.
8. Alice's USDT is permanently locked — burned on NEAR, undeliverable on Ethereum — with no recovery function available in the contract. [1](#0-0) [2](#0-1) [3](#0-2)
