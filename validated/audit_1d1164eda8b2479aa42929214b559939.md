### Title
Decimal Normalization Mismatch Permanently Locks All Funds for High-Decimal Tokens on StarkNet→NEAR Path — (`starknet/src/omni_bridge.cairo`, `near/omni-bridge/src/lib.rs`)

### Summary
`_normalizeDecimals` silently clamps `origin_decimals=255` to 18 on StarkNet, but the original value is preserved in the `DeployToken` event and stored on NEAR as `Decimals { decimals: 18, origin_decimals: 255 }`. Any subsequent StarkNet→NEAR transfer triggers `denormalize_amount(X, {18, 255})` = `X * 10^237`, which overflows `u128` and panics, permanently locking all bridged funds for that token.

### Finding Description

**Step 1 — Obtain MPC signature (permissionless):**
Any user can deploy a NEAR NEP-141 token with `decimals = 255` and call the public `log_metadata` function on the NEAR bridge. `log_metadata_callback` reads the token's actual decimals and asks the MPC to sign `MetadataPayload { decimals: 255 }`. The MPC performs no range validation on the decimal value. [8](#0-7) 

**Step 2 — Deploy token on StarkNet (permissionless with valid signature):**
The attacker calls `deploy_token` on StarkNet with the MPC-signed payload. `_normalizeDecimals(255)` returns 18, so the bridge token is deployed with 18 decimals. However, the `DeployToken` event records both `decimals: 18` and `origin_decimals: 255`. [9](#0-8) 

**Step 3 — Bind token on NEAR (permissionless with proof):**
Anyone calls `bind_token` on NEAR with a proof of the StarkNet `DeployToken` event. `bind_token_callback` extracts `deploy_token.decimals = 18` and `deploy_token.origin_decimals = 255` from the prover result and stores them verbatim. [10](#0-9) 

`add_token` stores `Decimals { decimals: 18, origin_decimals: 255 }` keyed by the StarkNet token address: [11](#0-10) 

**Step 4 — Any user bridges the token (StarkNet→NEAR):**
A user calls `init_transfer` on StarkNet. NEAR's `fin_transfer_callback` looks up the stored decimals and calls:

```rust
Self::denormalize_amount(init_transfer.amount.0, decimals)
// = amount * 10_u128.pow(255 - 18)
// = amount * 10^237  →  u128 overflow → panic
``` [6](#0-5) 

With `overflow-checks = true` in `near/Cargo.toml`, this panics unconditionally for any `amount > 0`. The callback fails, the transfer can never be finalized, and the user's tokens are permanently locked in the StarkNet bridge contract.

### Impact Explanation
**Critical — Permanent irrecoverable lock of user funds.** Every StarkNet→NEAR transfer for the poisoned token fails at `fin_transfer_callback`. Tokens are burned/locked on StarkNet but can never be released on NEAR. There is no recovery path: the token registration cannot be overwritten (the `add_token` guard `require!(...is_none(), BridgeError::TokenExists)` prevents re-registration), and the overflow is deterministic for all non-zero amounts. [12](#0-11) 

### Likelihood Explanation
The entire attack chain is permissionless. Deploying a NEAR token with 255 decimals costs only a small NEAR storage deposit. `log_metadata`, `deploy_token` (StarkNet), and `bind_token` (NEAR) are all public functions requiring no privileged access. The MPC signs whatever decimals the token contract reports — it performs no range check. A single attacker can execute all four steps independently.

### Recommendation
1. **Reject out-of-range decimals at `deploy_token` on StarkNet** — assert `payload.decimals <= 18` (or the configured max) and revert rather than silently clamping. This prevents the mismatch from ever being recorded.
2. **Validate in `bind_token_callback` on NEAR** — assert `deploy_token.origin_decimals <= some_max` (e.g., 24 or 38) before calling `add_token`, so that any event with an extreme value is rejected.
3. **Use checked arithmetic in `denormalize_amount`** — replace `amount * (10_u128.pow(diff_decimals))` with `amount.checked_mul(10_u128.checked_pow(diff_decimals)?)` and return an error rather than panicking.
