### Title
Missing Declared Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS program does not verify that the new class hash supplied by a contract corresponds to a previously declared contract class. An acknowledged TODO comment in the code confirms this check is absent. A contract owner can call `replace_class` with an arbitrary, undeclared class hash, permanently corrupting the contract's on-chain state and irreversibly freezing any funds held by that contract.

---

### Finding Description

The `execute_replace_class` function in `syscall_impls.cairo` processes the `replace_class` syscall, which allows a contract to upgrade its own class hash. The function reads the requested class hash directly from the syscall request and writes it into `contract_state_changes` without any check that the hash exists in the declared class tree (`contract_class_changes`):

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing invariant enforcement. The new `class_hash` is committed directly to the contract state tree via `compute_contract_state_commitment` without any cross-reference against the class declaration tree. [2](#0-1) 

When `execute_entry_point` later attempts to execute a call against this contract, it reads the (now-invalid) class hash from `contract_state_changes` and performs a lookup in `contract_class_changes`:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,
);
``` [3](#0-2) 

If the class hash was never declared, `dict_read` returns 0 and `find_element` cannot locate a matching `CompiledClassFact`, causing the OS to fail. The contract becomes permanently unexecutable in all subsequent blocks.

The state commitment logic in `commitment.cairo` commits whatever class hash is stored in `StateEntry` without any validity check against the class tree: [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value and committed to the global state root, the contract becomes permanently unexecutable. No entry point — including any recovery or withdrawal function — can be invoked. All ERC-20 tokens, ETH, or other assets held by the contract are irreversibly locked. The corruption is committed to the Merkle state tree and cannot be undone without a protocol-level intervention (hard fork).

---

### Likelihood Explanation

The attack requires no privileged access. Any contract deployer can:

1. Deploy a contract whose implementation calls `replace_class` with an attacker-chosen arbitrary felt value as the class hash.
2. Attract user deposits (e.g., by presenting the contract as a legitimate vault or token wrapper).
3. Invoke the contract to trigger the `replace_class` syscall.
4. The OS processes the syscall without validation and commits the invalid class hash to state.
5. All subsequent calls to the contract fail; deposited funds are permanently frozen.

The entry path is fully attacker-controlled: the `replace_class` syscall is available to any Sierra contract, the class hash field in `ReplaceClassRequest` is caller-supplied, and the OS performs no cross-check against the declared class tree.

---

### Recommendation

Inside `execute_replace_class`, before updating `contract_state_changes`, verify that the requested `class_hash` has a non-zero entry in `contract_class_changes` (i.e., it was previously declared). This is exactly what the existing TODO comment calls for:

```cairo
// Enforce the class was declared before allowing replacement.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the invariant already enforced for `declare` transactions, where `assert_not_zero(compiled_class_hash)` is required before writing to `contract_class_changes`. [5](#0-4) 

---

### Proof of Concept

1. **Deploy** a contract `MaliciousVault` whose `__execute__` entry point calls `replace_class(class_hash=0xdeadbeef)` — an arbitrary felt that has never been declared.
2. **Attract deposits**: users call `deposit()` on `MaliciousVault`, transferring tokens into it.
3. **Trigger**: attacker sends an invoke transaction calling `MaliciousVault.__execute__`. The OS processes `execute_replace_class` at lines 877–916 of `syscall_impls.cairo`, writes `class_hash=0xdeadbeef` into `contract_state_changes`, and commits it to the state root — no validation occurs.
4. **Subsequent block**: any transaction calling `MaliciousVault` causes `execute_entry_point` to do `dict_read(key=0xdeadbeef)` on `contract_class_changes`, returning 0. `find_element(key=0)` finds no matching `CompiledClassFact`; the OS cannot execute the contract.
5. **Result**: all deposited funds are permanently frozen. The contract state is irrecoverably corrupted in the committed global state root.
