### Title
Timelock Bypass via Wrapping Integer Overflow in `ASSERT_SECONDS_RELATIVE` / `ASSERT_HEIGHT_RELATIVE` Condition Validation — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

In `check_time_locks`, the legacy `nowrap=false` code path uses `wrapping_add` for relative timelock arithmetic. An attacker-controlled CLVM puzzle that emits `ASSERT_SECONDS_RELATIVE(u64::MAX)` or `ASSERT_HEIGHT_RELATIVE(u32::MAX)` causes the computed deadline to wrap around to a value already in the past, making the timelock check pass unconditionally. A coin that should be locked for an astronomically long time can be spent immediately.

---

### Finding Description

`check_time_locks` accepts a `nowrap: bool` parameter that selects between two arithmetic modes for relative timelock conditions:

- `nowrap=true` (post-softfork): uses `saturating_add`, which clamps at `u64::MAX` / `u32::MAX`, correctly keeping the deadline in the future.
- `nowrap=false` (pre-softfork / legacy): uses `wrapping_add`, which silently wraps on overflow. [1](#0-0) 

For `ASSERT_SECONDS_RELATIVE`:

```
coin_timestamp = 1000
seconds_relative = 0xffff_ffff_ffff_ffff   (u64::MAX)

deadline = 1000u64.wrapping_add(0xffff_ffff_ffff_ffff)
         = 999   ← wraps to a value already in the past

check: timestamp (2000) < 999  →  false  →  Ok(())   ← timelock bypassed
```

The same overflow applies to `ASSERT_HEIGHT_RELATIVE` with a `u32::MAX` argument: [2](#0-1) 

The test suite explicitly documents and confirms both bypasses: [3](#0-2) [4](#0-3) 

The Python-level wrapping tests confirm the same behavior end-to-end through the `py_check_time_locks` binding: [5](#0-4) [6](#0-5) 

The inverse overflow also affects `ASSERT_BEFORE_SECONDS_RELATIVE` and `ASSERT_BEFORE_HEIGHT_RELATIVE`: a large argument wraps to a small deadline, causing a spend that should be valid to be incorrectly rejected — a denial-of-spend that can be exploited to grief specific coin designs. [7](#0-6) 

---

### Impact Explanation

**High — Timelock condition validation bypass enables unauthorized spend acceptance.**

A coin whose puzzle emits `ASSERT_SECONDS_RELATIVE(0xffff_ffff_ffff_ffff)` is intended to be unspendable for ~584 billion years. Under `nowrap=false`, the wrapping arithmetic makes the deadline evaluate to a timestamp already in the past, so the coin is accepted as spendable at any block. Any funds locked behind such a timelock can be stolen immediately by the coin's owner (or anyone who knows the puzzle/solution) without waiting the intended duration.

---

### Likelihood Explanation

The `nowrap=false` path is the pre-softfork behavior that the Chia node passes for blocks below the softfork activation height. The `py_check_time_locks` Python binding exposes `nowrap` as a caller-controlled parameter: [8](#0-7) 

An unprivileged user controls the CLVM puzzle and solution entirely. Crafting a puzzle that outputs `ASSERT_SECONDS_RELATIVE` with a large value requires no special privilege. The attacker only needs to submit a spend bundle through the normal transaction mempool.

---

### Recommendation

1. **Remove the `wrapping_add` branch entirely.** The `nowrap=false` path has no safe use: wrapping arithmetic on timelock deadlines is never semantically correct. Replace both `wrapping_add` calls with `saturating_add` unconditionally, or gate the legacy path behind a hard block-height check that prevents it from being applied to any new spend.

2. **Add an explicit overflow guard** in `parse_conditions` / `sanitize_uint` to reject `ASSERT_SECONDS_RELATIVE` values that would overflow when added to any plausible coin timestamp, returning `AssertSecondsRelativeFailed` at parse time rather than silently passing at check time.

3. **Audit all callers** of `py_check_time_locks` in the Chia node to confirm `nowrap=false` is never passed for blocks at or above the softfork activation height.

---

### Proof of Concept

Using the existing test harness in `tests/test_check_time_locks.py`:

```python
from chia_rs import check_time_locks
from chia_rs.sized_ints import uint32, uint64

# Coin confirmed at timestamp 1000, current block timestamp 2000
# ASSERT_SECONDS_RELATIVE(u64::MAX) — should lock coin for ~584 billion years

result = check_time_locks(
    removals,                        # coin record with timestamp=1000
    make_test_conds(seconds_relative=0xFFFF_FFFF_FFFF_FFFF),
    prev_height=200,
    timestamp=2000,
    nowrap=False,                    # legacy pre-softfork path
)

assert result is None   # None = no error = timelock bypassed
# Expected: error code 105 (ASSERT_SECONDS_RELATIVE_FAILED)
```

`expected_wrap=None` means `check_time_locks` returns no error — the spend is accepted — despite the coin carrying a timelock of `u64::MAX` seconds.
