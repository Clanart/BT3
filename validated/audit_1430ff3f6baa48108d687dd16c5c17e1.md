Audit Report

## Title
Missing Deadline Guard in `addLiquidityExactShares` Allows Stale Bin Deposits at Unfavorable Oracle Prices - (File: `metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

Both overloads of `addLiquidityExactShares` in `MetricOmmPoolLiquidityAdder` accept caller-specified `binIdxs` and `shares` but carry no `deadline` parameter and no cursor-position guard. A pending transaction can execute long after submission when the oracle-anchored pool cursor has moved, depositing the user's tokens into bins that are immediately out-of-range and subject to arbitrage-driven loss. The only existing protection — `maxAmountToken0`/`maxAmountToken1` caps — limits the amount spent but does not prevent execution at a stale oracle price.

## Finding Description

`MetricOmmSimpleRouter` enforces `_checkDeadline(params.deadline)` at the entry of all four swap paths: [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

The helper `_checkDeadline` reverts with `TransactionExpired` when `block.timestamp > deadline`: [5](#0-4) 

`addLiquidityWeighted` compensates for the absence of a deadline by calling `_validateBinAndBinPosition`, which reads the live cursor from `slot0` and reverts if it has drifted outside caller-specified bounds: [6](#0-5) [7](#0-6) 

Neither overload of `addLiquidityExactShares` has either guard: [8](#0-7) [9](#0-8) 

The only check in the callback is the amount cap: [10](#0-9) 

Because Metric OMM is oracle-anchored, the pool's active bin cursor moves with the oracle price. Bins that were near mid-price at submission time can be deep out-of-range at execution time. The amount caps do not prevent this: the token amounts required for the stale bins may still be within the caps (e.g., a below-cursor bin requires only token0, which may be well under `maxAmountToken0`), so the call succeeds and the user's liquidity lands in bins that are immediately arbitrageable at a below-oracle price.

## Impact Explanation

An LP who submits `addLiquidityExactShares` targeting bins near the current oracle price can have their deposit executed at a materially different oracle price. In an oracle-anchored pool, bins below the active cursor are priced below the oracle mid-price; traders immediately arbitrage by buying token0 from those bins at a discount, leaving the LP with token1 received at a below-oracle rate. This constitutes a direct loss of user principal — the deposited value is less than the equivalent spot value — with no contract-level mechanism to revert or compensate. Severity: **Medium** — direct loss of LP principal, no privileged access required, triggered by standard mempool conditions.

## Likelihood Explanation

Any user calling `addLiquidityExactShares` during oracle volatility or mempool congestion is at risk. A searcher can observe a pending liquidity transaction and delay its inclusion (e.g., by front-running with a higher-gas transaction that fills the block) until the oracle price has moved enough to make the deposit unfavorable. No special permissions are needed; the attack is a standard MEV vector repeatable on every oracle price move.

## Recommendation

Add a `uint256 deadline` parameter to both `addLiquidityExactShares` overloads and call `_checkDeadline(deadline)` at entry, mirroring the pattern in `MetricOmmSwapRouterBase`. Since `MetricOmmPoolLiquidityAdder` does not inherit `MetricOmmSwapRouterBase`, the helper can be duplicated inline or extracted to a shared base:

```solidity
function addLiquidityExactShares(
    address pool,
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 maxAmountToken0,
    uint256 maxAmountToken1,
    uint256 deadline,          // add
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    if (block.timestamp > deadline) revert TransactionExpired(deadline, block.timestamp);
    _validateOwner(owner);
    _validateDeltas(deltas);
    return _addLiquidity(...);
}
```

Alternatively, expose the cursor-bounds guard (already present in `addLiquidityWeighted`) as an optional parameter on `addLiquidityExactShares` so callers can bound acceptable oracle drift without a hard deadline.

## Proof of Concept

1. Oracle price for token0/token1 is $2,000 at block N. User constructs a `LiquidityDelta` targeting bins `[-1, 0, +1]` around the current cursor and submits `addLiquidityExactShares` with `maxAmountToken0 = 1_000e6`, `maxAmountToken1 = 0.5e18`.
2. A searcher front-runs with a high-gas transaction that fills the block; the user's transaction is not mined until block N+200 (~40 minutes on a 12-second chain).
3. By block N+200 the oracle price has moved to $1,600. The pool cursor has shifted; the user's bins are now 3–4 bins below the new cursor.
4. `addLiquidityExactShares` executes. The required token amounts for the stale bins are still within the caps (e.g., `amount0Delta = 800e6 < 1_000e6`), so the callback's cap check at line 165 does not revert.
5. The user's liquidity sits entirely below the current oracle price, is immediately one-sided in token0, and is arbitraged by traders buying token0 at the below-oracle bin price — constituting a direct loss of principal with no contract-level recovery path.

Foundry test plan: fork at block N, warp to block N+200, push oracle price to $1,600 via the oracle admin, call `addLiquidityExactShares` with the original bins, assert that the deposited position value (computed at oracle price) is materially less than the tokens spent.

  function _validateBinAndBinPosition(
    address pool,
    int8 minimalCurBin,
    uint104 minimalPosition,
    int8 maximalCurBin,
    uint104 maximalPosition
  ) internal view {
    if (minimalCurBin > maximalCurBin) {
      revert CursorOutOfBounds(0, 0, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }

    (, int8 curBinIdx, uint104 curPosInBin,,,) = PoolStateLibrary._slot0(pool);

    int256 curBin = curBinIdx;
    if (curBin < minimalCurBin || curBin > maximalCurBin) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == minimalCurBin && curPosInBin < minimalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == maximalCurBin && curPosInBin > maximalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
