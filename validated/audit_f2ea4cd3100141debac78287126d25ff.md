### Title
`check_is_reverted` Is a No-Op, Leaving `is_reverted` Hint Unconstrained in Transaction Execution — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

`execution_constraints.cairo` defines `check_is_reverted()` as a completely empty function. It is imported and called in the invoke-transaction execution flow, but because it contains no Cairo assertions, the hint-provided `is_reverted` value is entirely unconstrained in the proof. A malicious sequencer/prover can set `is_reverted = 1` for any valid transaction, causing the `__execute__` entry point to be skipped while fees are still charged — a direct, provably-valid loss of user funds.

---

### Finding Description

**Vulnerability class:** State-transition bypass / invalid transaction acceptance — a check function is defined, imported, and called, but performs no actual constraint, making the guarded value hint-free.

**Analog to the external report:** The `transferWhitelist` bug is exactly this pattern: a restriction mechanism (`transferWhitelist` mapping / `check_is_reverted` function) is declared and invoked in the critical path, but the invocation has zero effect. In both cases the "check" is a dead call.

In `execution_constraints.cairo`, the function body is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
``` [1](#0-0) 

It is imported and called in `execute_invoke_function_transaction` in `transaction_impls.cairo`:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...non_reverting_select_execute_entry_point_func(...)...
} else {
    // stack-alignment only, no execution
}

// Charge fee — always runs regardless of is_reverted.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [2](#0-1) 

In Cairo, a `local` variable filled only by a hint (`%{ IsReverted %}`) and never constrained by any `assert` or arithmetic relation is **a free variable in the proof system** — the prover can assign it any field element. `check_is_reverted` is the sole intended constraint site, and it is empty.

Compare with `check_proof_facts`, the sibling function in the same file, which contains real assertions: [3](#0-2) 

`check_proof_facts` enforces multiple `assert` statements; `check_is_reverted` enforces none.

---

### Impact Explanation

**Impact: Critical — Direct loss of funds.**

`charge_fee` is called unconditionally after the `is_reverted` branch: [4](#0-3) 

A malicious sequencer that sets `is_reverted = 1` for a valid, non-reverting transaction will:

1. Skip the `__execute__` entry point entirely (the `else` branch only realigns the stack).
2. Still execute `charge_fee`, deducting the actual fee from the user's ERC-20 balance.
3. Produce a **valid STARK proof** — because no Cairo constraint is violated.

The user's funds (fee tokens) are permanently transferred to the sequencer address with no corresponding execution of their intended transaction. This is irreversible on-chain.

---

### Likelihood Explanation

The attacker is the sequencer/prover — a role that is reachable without any leaked key or privileged access beyond operating a sequencer node. The sequencer controls hint injection. Because the proof verifier only checks Cairo constraints (not hint values), and `check_is_reverted` adds zero constraints, the malicious proof is indistinguishable from an honest one. Any sequencer operator can exploit this against any user whose transaction they include in a block.

---

### Recommendation

`check_is_reverted` must add a Cairo constraint that ties `is_reverted` to a verifiable on-chain condition. At minimum, the function should assert that `is_reverted` is a boolean (0 or 1):

```cairo
func check_is_reverted(is_reverted: felt) {
    assert is_reverted * (1 - is_reverted) = 0;
    return ();
}
```

More robustly, the revert status should be derived from the return value of the entry-point execution itself (the `failure_flag` field of `EntryPointReturnValues`), which is already constrained by the proof, rather than from an independent hint. [5](#0-4) 

---

### Proof of Concept

1. Sequencer receives a valid user invoke transaction (e.g., a token transfer).
2. During OS execution, the sequencer's hint runner sets `is_reverted = 1` via `%{ IsReverted %}` for this transaction.
3. `check_is_reverted(1)` is called — it immediately returns, adding no proof constraint.
4. The `if (is_reverted == FALSE)` branch is not taken; `__execute__` is skipped.
5. `charge_fee` executes normally, transferring the user's fee tokens to the sequencer.
6. The OS produces a STARK proof. The verifier accepts it — no assertion was violated.
7. The block is finalized on L1. The user's fee is gone; their transaction was never executed.

The attacker-controlled entry path is: sequencer node → hint injection at `%{ IsReverted %}` in `execute_invoke_function_transaction` → unconstrained `is_reverted` local → skipped execution + charged fee.
