### Title
Permanent Fund Lock Due to `normalize_amount` Rounding to Zero in `sign_transfer` - (File: `near/omni-bridge/src/lib.rs`)

### Summary

`sign_transfer` applies `normalize_amount` (integer floor division) to scale the transfer amount from NEAR's high-precision representation down to the destination chain's lower-precision representation. When the user's `amount_without_fee` is smaller than the decimal scaling factor (`10^(origin_decimals - decimals)`), the division truncates to zero. The function then panics with `InvalidAmountToTransfer`. Because the user's tokens were already locked or burned during `init_transfer`, and no cancel/refund path exists for this failure mode, the funds are permanently irrecoverable.

### Finding Description

`normalize_amount` performs integer floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
```

In `sign_transfer`, this is applied to `amount_without_fee`:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message
        .amount_without_fee()
        .near_expect(BridgeError::InvalidFee),
    decimals,
);

require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

The `init_transfer` entry point only validates `fee < amount`; it does **not** validate that `normalize_amount(amount - fee) > 0`. A user can therefore successfully lock or burn tokens via `init_transfer`, store a `TransferMessage`, and then find that every subsequent call to `sign_transfer` panics before the MPC signer is ever invoked.

The `sign_transfer_callback` only removes the stored `TransferMessage` when `fee.is_zero()` **and** MPC signing succeeded. Since `sign_transfer` panics before reaching MPC, the callback is never executed, the message is never removed, and the locked/burned tokens have no recovery path.

**Concrete example:**
- Token: NEAR-native token with `origin_decimals = 24`, bridged to EVM with `decimals = 6`.
- `diff_decimals = 18`, scaling factor = `10^18`.
- User sends `amount = 5 * 10^17` (0.5 NEAR in yoctoNEAR), fee = 0.
- `normalize_amount(5 * 10^17, ...) = 5 * 10^17 / 10^18 = 0`.
- `sign_transfer` panics. Tokens are permanently locked in the bridge.

For bridged tokens (burn path), the tokens are permanently destroyed.

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

Any user who initiates a NEAR-origin transfer with `amount_without_fee < 10^(origin_decimals - decimals)` will have their tokens permanently locked (native tokens) or burned (bridged tokens) with no recovery mechanism. The `TransferMessage` remains in storage indefinitely but can never be finalized or cancelled.

### Likelihood Explanation

**Medium-High.** The condition is reachable by any unprivileged user calling `ft_transfer_call` → `init_transfer`. For tokens with large decimal differences (e.g., 24 vs 6 = factor of `10^18`), the minimum transferable amount is 1 full token unit on the origin chain. Users sending sub-unit amounts — a natural behavior for high-value tokens — will silently lose funds. No front-end or contract-level guard at `init_transfer` time prevents this.

### Recommendation

Add a normalization check at `init_transfer` time, before tokens are locked or burned:

```rust
require!(
    Self::normalize_amount(
        transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
        decimals
    ) > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
```

This mirrors the existing guard in `sign_transfer` but places it at the point where the user still has their tokens.

### Proof of Concept

1. Deploy a NEAR-origin token registered with `origin_decimals = 24`, `decimals = 6` on the destination chain.
2. Call `ft_transfer_call` with `amount = 10^17` (0.1 NEAR), `fee = 0`, valid EVM recipient.
3. `init_transfer` succeeds: tokens are locked, `TransferMessage` is stored.
4. Trusted relayer calls `sign_transfer` for the stored `transfer_id`.
5. `normalize_amount(10^17, Decimals { origin_decimals: 24, decimals: 6 }) = 10^17 / 10^18 = 0`.
6. `require!(0 > 0, ...)` panics with `InvalidAmountToTransfer`.
7. No MPC call is made; `sign_transfer_callback` is never reached; `TransferMessage` is never removed.
8. User's `10^17` yoctoNEAR is permanently locked in the bridge with no cancel path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
