### Title
`ProofOfInclusion::valid()` Does Not Verify Root Hash Against a Trusted External Root — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer Merkle implementation only checks the internal self-consistency of the proof chain. It derives its "root" from the proof's own last `combined_hash` field rather than comparing against any externally trusted tree root. An attacker can craft a `ProofOfInclusion` that is internally consistent but proves inclusion in a completely different tree, and `valid()` will return `true`. Because the struct is `Streamable` and fully exposed via Python bindings, any consumer that calls `proof.valid()` as the sole check for proof validity — without also comparing `proof.root_hash()` against the actual DataLayer tree root — silently accepts a forged proof of inclusion.

### Finding Description

`ProofOfInclusion::valid()` is defined as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()
}
``` [1](#0-0) 

The final check `existing_hash == self.root_hash()` is tautological: after the loop, `existing_hash` equals the last `layer.combined_hash`, and `root_hash()` also returns `last.combined_hash`:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

So `valid()` only verifies that each layer's `combined_hash` is correctly computed from the previous hash and the sibling hash — it never compares the final root against any externally known, trusted tree root. An attacker who constructs a `ProofOfInclusion` with arbitrary `node_hash`, `other_hash`, and `combined_hash` values that form a self-consistent chain will always pass `valid()`.

The struct is `Streamable` and exposed via Python bindings with `from_bytes()`, `from_bytes_unchecked()`, and `valid()`: [3](#0-2) [4](#0-3)

### Impact Explanation

Any Python or Rust consumer that receives a `ProofOfInclusion` from an untrusted source (e.g., a DataLayer peer over the network) and calls `proof.valid()` as the sole validity check will accept a forged proof of inclusion. The attacker can prove that an arbitrary key-value pair is included in a DataLayer tree when it is not, or prove inclusion in a stale/deleted tree state. This maps directly to the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, letting untrusted input prove invalid state.**

### Likelihood Explanation

The `ProofOfInclusion` struct is `Streamable`, fully deserializable from bytes, and exposed via Python bindings. The `valid()` method's name strongly implies it is a complete validity check. Any DataLayer client that receives a proof over the network and calls `proof.valid()` without also asserting `proof.root_hash() == known_tree_root` is vulnerable. The misleading API design makes this omission highly likely in practice.

### Recommendation

- **Short term:** Add an `expected_root: Hash` parameter to `valid()` (or add a separate `valid_for_root(expected_root: &Hash) -> bool` method) that compares the computed root against the caller-supplied trusted root. Update all call sites accordingly.
- **Long term:** Document clearly that `valid()` alone is insufficient for security — callers must always compare `root_hash()` against the actual DataLayer tree root obtained from a trusted source. Consider making the unsafe self-consistency-only check private or renaming it to avoid confusion.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Craft a fake leaf hash
fake_node_hash = bytes([0xAA] * 32)
fake_sibling   = bytes([0xBB] * 32)

# Compute a fake combined_hash that is internally consistent
def internal_hash(left, right):
    h = hashlib.sha256()
    h.update(b'\x01')  # internal node prefix used by calculate_internal_hash
    h.update(left)
    h.update(right)
    return h.digest()

fake_combined = internal_hash(fake_node_hash, fake_sibling)

# Build a ProofOfInclusion that is internally consistent but proves
# inclusion in a completely fabricated tree (root = fake_combined)
layer = ProofOfInclusionLayer(
    other_hash_side=1,          # sibling is on the right
    other_hash=fake_sibling,
    combined_hash=fake_combined,
)
proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True even though this proof was never generated
# from any real MerkleBlob, and root_hash() is attacker-controlled
assert proof.valid()                        # passes — no external root check
assert proof.root_hash() == fake_combined   # attacker controls the root
# A consumer checking only proof.valid() accepts this forged proof
```

The `calculate_internal_hash` prefix byte may differ; the attacker simply adjusts `fake_combined` to match whatever hash function `chia-datalayer` uses. The structural point — that `valid()` never compares against an external root — holds regardless.
