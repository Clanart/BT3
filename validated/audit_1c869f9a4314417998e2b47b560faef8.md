### Title
Unregistered ERC20 Tokens Accepted by `initTransfer` Are Permanently Locked With No Recovery Path - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` accepts any ERC20 token address in its `else` branch without verifying that the token has a registered NEAR-side mapping. Tokens sent for unregistered addresses are transferred into the bridge contract and permanently locked: the NEAR `fin_transfer_callback` panics with `TokenDecimalsNotFound`, and unlike the Gravity.sol case in the reference report, `OmniBridge.sol` contains no admin rescue function.

---

### Finding Description

`initTransfer` in `OmniBridge.sol` routes ERC20 handling through three branches:

1. `customMinters[tokenAddress] != address(0)` → burn via custom minter
2. `isBridgeToken[tokenAddress]` → burn bridge token
3. **else** → `IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount)` [1](#0-0) 

The `else` branch is intended for native ERC20 tokens (e.g., USDC, WETH) that have been fully registered through the `logMetadata` → `deployToken`/`bind_token` pipeline, which populates `ethToNearToken[tokenAddress]` and `token_decimals` on the NEAR side. However, `initTransfer` performs **no check** that `ethToNearToken[tokenAddress]` is non-empty before accepting the transfer. [2](#0-1) 

A token can reach the `else` branch in a partially-registered state: `logMetadata` was called (permissionless, anyone can call it for any ERC20), but `bind_token` on NEAR was never completed, so `token_decimals` has no entry for that token address.

On the NEAR side, `fin_transfer_callback` unconditionally calls `.near_expect(BridgeError::TokenDecimalsNotFound)` on the `token_decimals` lookup: [3](#0-2) 

This panics for any token not registered in `token_decimals`, causing the NEAR-side finalization to fail permanently. The EVM-side transfer is already complete and irreversible.

`OmniBridge.sol` contains no `withdrawERC20`, sweep, or rescue function of any kind. The full function list is: `addCustomToken`, `removeCustomToken`, `acceptTokenOwnership`, `deployToken`, `setMetadata`, `logMetadata`, `pause`/`pauseAll`, `upgradeToken`, `setNearBridgeDerivedAddress`. None allow recovering locked ERC20 balances. [4](#0-3) 

The same structural flaw exists in the StarkNet bridge: `init_transfer` transfers tokens to the contract in its `else` branch without checking `starknet_to_near_token[token_address]`: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

Any ERC20 token sent via `initTransfer` for which `ethToNearToken[tokenAddress]` is empty (unregistered or partially registered) is transferred into the bridge contract and cannot be recovered. There is no admin rescue path. The NEAR-side proof submission fails permanently. The user's funds are gone with no recourse.

---

### Likelihood Explanation

**High.** The attack surface is reachable by any unprivileged user. Realistic triggering scenarios include:

- A user calls `logMetadata` for a token (permissionless), then immediately calls `initTransfer` before the NEAR-side `bind_token` is completed.
- A user mistypes or copy-pastes a token address that is not registered.
- A user bridges a token that was previously removed via `removeCustomToken` (which clears `ethToNearToken` and `isBridgeToken`) but does not burn the locked balance.

The `logMetadata` function is explicitly permissionless by design (noted in `evm/SECURITY.md`), making the partially-registered state easy to reach. [6](#0-5) 

---

### Recommendation

Add a check in `initTransfer` that the token has a registered NEAR mapping before accepting the transfer:

```solidity
} else {
    require(
        bytes(ethToNearToken[tokenAddress]).length > 0,
        "ERR_TOKEN_NOT_REGISTERED"
    );
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
```

Apply the equivalent guard in the StarkNet `init_transfer`:

```cairo
} else {
    assert(
        self.starknet_to_near_token.read(token_address).len() > 0,
        'ERR_TOKEN_NOT_REGISTERED'
    );
    // transfer_from ...
}
```

Additionally, consider adding an admin ERC20 rescue function as a defense-in-depth measure for any tokens that bypass the guard.

---

### Proof of Concept

1. Attacker (or innocent user) calls `logMetadata(tokenAddress)` for an arbitrary ERC20 — this is permissionless and succeeds.
2. Before `bind_token` is called on NEAR (so `token_decimals` has no entry for this token), the user calls:
   ```solidity
   OmniBridge.initTransfer(
       tokenAddress,   // unregistered ERC20
       1_000e18,       // amount
       0,              // fee
       0,              // nativeFee
       "user.near",    // recipient
       ""
   );
   ```
3. `tokenAddress` is not a bridge token and has no custom minter, so the `else` branch executes: `IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), 1_000e18)`. Tokens are now in the bridge.
4. `InitTransfer` event is emitted with `currentOriginNonce`.
5. A relayer submits proof to NEAR `fin_transfer`. `fin_transfer_callback` executes and hits:
   ```rust
   let decimals = self
       .token_decimals
       .get(&init_transfer.token)
       .near_expect(BridgeError::TokenDecimalsNotFound); // PANICS
   ```
6. NEAR transaction reverts. No tokens are minted. The EVM-side lock is permanent. No admin function exists to recover the 1,000 tokens from the bridge contract. [7](#0-6) [3](#0-2)
