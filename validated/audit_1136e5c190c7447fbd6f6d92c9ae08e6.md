### Title
Token-2022 MetadataPointer Cross-Mint Metadata Spoofing Enables Fake Canonical Token Registration on NEAR — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

### Summary

`parse_metadata_account` in the Token-2022 third-party metadata path verifies only that the provided metadata account's **key** matches `metadata_pointer.metadata_address.0`, but never verifies that the Metaplex metadata account's embedded `mint` field matches the Token-2022 mint being registered. An unprivileged attacker can create a Token-2022 mint whose MetadataPointer points to a legitimately Metaplex-owned account for a *different* mint that carries attacker-chosen name/symbol (e.g., "USDC"), causing the bridge to emit a Wormhole `LogMetadata` message with spoofed identity, which NEAR's `deploy_token_callback` then uses to register and deploy a new NEAR token under that spoofed identity.

### Finding Description

The vulnerable code path is in `process()`: [1](#0-0) 

When `metadata_pointer.metadata_address.0` is neither the mint itself nor `Pubkey::default()`, it delegates to `parse_metadata_account`: [2](#0-1) 

The only guard is `require_keys_eq!(metadata.key(), address, ...)` — confirming the account at the address the pointer names was actually passed in — and then an owner check `metadata.owner == &MetaplexID`. If both pass, it blindly deserializes and returns `metadata.name` / `metadata.symbol`.

**What is missing:** there is no check that `metadata.mint == self.mint.key()`. The Metaplex `Metadata` struct contains a `mint` field recording which mint the metadata was created for. The code never reads it.

**Attack construction:**

1. Attacker creates `mint_A` (any SPL or Token-2022 mint they control).
2. Attacker calls Metaplex `create_metadata_accounts_v3` for `mint_A` with `name = "USD Coin"`, `symbol = "USDC"`. This creates a valid Metaplex-owned PDA at `[b"metadata", MetaplexID, mint_A]` with `metadata.mint = mint_A`.
3. Attacker creates Token-2022 `mint_B` with a `MetadataPointer` extension whose `metadata_address` is set to the Metaplex PDA of `mint_A`.
4. Attacker calls `log_metadata` passing `mint_B` as the mint and `mint_A`'s Metaplex PDA as the `metadata` account.
5. `require_keys_eq!` passes (the account key matches the pointer value).
6. `metadata.owner == &MetaplexID` passes (Metaplex owns it).
7. `MplMetadata::try_deserialize` succeeds and returns `("USD Coin", "USDC")`.
8. `LogMetadataPayload { token: mint_B, name: "USD Coin", symbol: "USDC", decimals: ... }` is serialized and posted via Wormhole. [3](#0-2) 

On NEAR, `deploy_token_callback` receives the proven `LogMetadataMessage`: [4](#0-3) 

It checks only that the emitter is a registered factory (the legitimate Solana bridge program address — which it is, since the transaction went through the real bridge program), then calls `deploy_token_internal` with `name = "USD Coin"`, `symbol = "USDC"`, deploying a new NEAR token under that identity mapped to `mint_B`. [5](#0-4) 

Because `log_metadata` requires `!mint.mint_authority.contains(authority.key)` (the bridge authority must NOT be the mint authority), the attacker retains full mint authority over `mint_B`: [6](#0-5) 

The attacker can therefore mint unlimited `mint_B` tokens and bridge them to NEAR, where they are credited as "USDC" tokens under the spoofed NEAR token account.

### Impact Explanation

- A NEAR token with name "USD Coin" / symbol "USDC" is registered and deployed, mapped to the attacker's worthless `mint_B`.
- The attacker mints unlimited `mint_B` supply and bridges it to NEAR, creating an unbacked supply of a token that impersonates a canonical asset.
- Any user or protocol that queries the NEAR token registry by name/symbol and accepts this token as USDC receives worthless tokens.
- The `emitter_address` check in `deploy_token_callback` provides no protection because the message legitimately originates from the real Solana bridge program.

### Likelihood Explanation

Fully permissionless. No privileged role, no leaked key, no compromised guardian is required. The attacker needs only:
- SOL to pay for account creation and Wormhole fees.
- The ability to call the public `log_metadata` instruction.

Metaplex imposes no restriction on what name/symbol a creator assigns to their own mint's metadata. The Token-2022 `MetadataPointer` extension is a standard feature with no on-chain restriction on which address it may point to.

### Recommendation

In `parse_metadata_account`, after deserializing the Metaplex account, add a cross-mint check:

```rust
if metadata.owner == &MetaplexID {
    let data = metadata.try_borrow_data()?;
    let mpl = MplMetadata::try_deserialize(&mut data.as_ref())?;
    // NEW: ensure the metadata was created for THIS mint, not a different one
    require_keys_eq!(
        mpl.mint,
        self.mint.key(),
        ErrorCode::InvalidTokenMetadataAddress,
    );
    Ok((mpl.name.clone(), mpl.symbol.clone()))
}
```

This is the same binding that the classic SPL path enforces implicitly (the PDA seed includes the mint pubkey), and it must be made explicit for the Token-2022 third-party pointer path. [7](#0-6) 

### Proof of Concept

    attacker,                  // mint authority
    attacker,                  // update authority

    ...
);

// 3. Create Token-2022 mint_B with MetadataPointer → mint_A's Metaplex PDA
let mint_b = Keypair::new();
initialize_mint_with_extensions(
    token_2022_program,
    mint_b,
    MetadataPointer { metadata_address: metadata_pda_for_mint_a },
    ...
);

// 4. Call log_metadata on the bridge with mint_B + mint_A's metadata account
bridge_program.log_metadata(
    mint = mint_b.pubkey(),
    metadata = Some(metadata_pda_for_mint_a),
    token_program = token_2022::ID,
    ...
);

// 5. Observe: Wormhole message emitted with name="USD Coin", symbol="USDC", token=mint_B
// 6. On NEAR: deploy_token_callback registers a new NEAR token "USD Coin"/"USDC" → mint_B
// 7. Attacker mints unlimited mint_B tokens, bridges to NEAR, receives "USDC" NEAR tokens

    fn deploy_token_internal(
        &mut self,
        chain_kind: ChainKind,
        token_address: &OmniAddress,
        metadata: BasicMetadata,
        attached_deposit: NearToken,
    ) -> Promise {
        let deployer = self
            .token_deployer_accounts
            .get(&chain_kind)
            .unwrap_or_else(|| env::panic_str(BridgeError::DeployerNotSet.to_string().as_str()));
        let prefix = token_address.get_token_prefix();
        let token_id: AccountId = format!("{prefix}.{deployer}")
            .parse()
            .unwrap_or_else(|_| env::panic_str(BridgeError::ParseAccountId.to_string().as_str()));

        let storage_usage = env::storage_usage();
        self.add_token(
            &token_id,
            token_address,
            metadata.decimals,
            metadata.decimals,
        );

        require!(
            self.deployed_tokens.insert(&token_id),
            BridgeError::TokenExists.as_ref()
        );
        self.deployed_tokens_v2
            .insert(&token_id, &token_address.get_chain());

        let required_deposit = env::storage_byte_cost()
            .saturating_mul((env::storage_usage().saturating_sub(storage_usage)).into())
            .saturating_add(NEP141_DEPOSIT);

        require!(
            attached_deposit >= required_deposit,
            BridgeError::InsufficientStorageDeposit.as_ref()
        );

        env::log_str(
            &OmniBridgeEvent::DeployTokenEvent {
                token_id: token_id.clone(),
                token_address: token_address.clone(),
                metadata: metadata.clone(),
            }
            .to_log_string(),
        );

        ext_deployer::ext(deployer)
            .with_static_gas(DEPLOY_TOKEN_GAS)
            .with_attached_deposit(attached_deposit.saturating_sub(required_deposit))
            .deploy_token(token_id.clone(), metadata)
            .then(
                Self::ext(env::current_account_id())
                    .deploy_token_by_deployer_callback(token_address, token_id),
            )
