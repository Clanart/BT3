### Title
`ChainStateSyncAdapter::set_state_header` Does Not Verify That the Supplied Chunk Belongs to the Requested `shard_id` — (`File: chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` validates that the chunk inside a `ShardStateSyncResponseHeader` is internally self-consistent and is included somewhere in the block's `chunk_headers_root` Merkle tree, but it never checks that `chunk.shard_id() == shard_id`. A malicious peer can supply a header whose embedded chunk belongs to a different shard, pass all existing checks, and cause the syncing node to store — and later apply — the wrong shard's state under the requested shard's key.

---

### Finding Description

`set_state_header` in `chain/chain/src/state_sync/adapter.rs` accepts three parameters: `shard_id` (the shard the caller wants to sync), `sync_hash`, and `shard_state_header` (the untrusted peer-supplied response). The function performs the following checks:

1. **Step 1-2** (`validate_chunk_proofs`): verifies the chunk's internal hash, tx-root, and receipts-root consistency.
2. **Step 3a** (`verify_path`): verifies that `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is a leaf in the block's `chunk_headers_root` Merkle tree.
3. **Step 3b**: verifies the previous chunk is also in the preceding block.
4. **Step 4**: verifies incoming receipt proofs.
5. **Step 5** (`validate_state_root_node`): verifies the `state_root_node` matches `chunk_inner.prev_state_root()`.

None of these steps check `chunk.shard_id() == shard_id`. [1](#0-0) 

The Merkle proof in step 3a is position-specific (it encodes sibling hashes for the chunk's actual slot in the block), but `verify_path` only confirms that the supplied chunk hash appears somewhere in the tree — it does not confirm that the slot corresponds to the requested `shard_id`. A chunk for shard B at slot 1 carries a valid proof against the same `chunk_headers_root`, and that proof will pass `verify_path` regardless of which shard was requested.

After all checks pass, the header is persisted under the key `StateHeaderKey(shard_id, sync_hash)` — keyed to the requested shard — but the embedded chunk and state root belong to a different shard: [2](#0-1) 

Downstream, `set_state_part` reads the stored header to obtain the state root and validates incoming parts against it: [3](#0-2) 

And `set_state_finalize` applies the chunk using the requested `shard_id` to derive `shard_uid`, but uses `chunk_header.prev_state_root()` from the stored (wrong-shard) chunk: [4](#0-3) 

The result is that shard B's state is applied to shard A's trie slot.

---

### Impact Explanation

A syncing node that accepts a crafted `ShardStateSyncResponseHeader` from a malicious peer will:

1. Store shard B's state root under `StateHeaderKey(shard_A, sync_hash)`.
2. Download and validate state parts against shard B's state root (all parts pass because they are genuine parts of shard B).
3. Apply shard B's state to shard A's `shard_uid` slot via `apply_chunk`.
4. Persist the corrupted `ChunkExtra` and trie changes for shard A.

The node now has an entirely wrong state for shard A. If the node is a chunk producer or chunk validator for shard A, it will produce incorrect state witnesses and state transitions, leading to slashing or chain split. If it is an RPC node, it will serve incorrect state queries.

---

### Likelihood Explanation

State sync downloads headers from any reachable peer via `StateSyncDownloader::ensure_shard_header`. The download source is not restricted to validators; any peer that advertises the state can respond. The `StateHeaderValidationRequest` carries the peer-supplied header verbatim to the chain for validation, and the chain's `set_state_header` is the sole gatekeeper. No privilege is required beyond being a reachable network peer. [5](#0-4) 

---

### Recommendation

Add an explicit shard-binding check immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
// NEW: verify the chunk belongs to the requested shard
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} does not match requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

Similarly, add the analogous check for `prev_chunk_header.shard_id()` when it is present.

---

### Proof of Concept

**Setup**: Two shards, shard 0 and shard 1, both present in the sync block. A syncing node requests state for shard 0.

**Attack**:
1. Malicious peer intercepts the `StateRequestHeader { shard_id: 0, sync_hash }` request.
2. Peer constructs a `ShardStateSyncResponseHeader` where:
   - `chunk` = the genuine chunk for shard 1 (internally valid, passes `validate_chunk_proofs`).
   - `chunk_proof` = the genuine Merkle proof for shard 1's slot in the block (passes `verify_path` against `chunk_headers_root`).
   - `state_root_node` = the genuine state root node for shard 1 (passes `validate_state_root_node`).
   - `incoming_receipts_proofs` = genuine receipts for shard 1.
3. Syncing node calls `set_state_header(shard_id=0, sync_hash, crafted_header)`.
4. All five validation steps pass. The header is stored under `StateHeaderKey(shard_0, sync_hash)`.
5. `set_state_part` downloads and validates genuine shard 1 parts against shard 1's state root — all pass.
6. `set_state_finalize(shard_id=0, sync_hash)` reads the stored header, derives `shard_uid` for shard 0, but applies `chunk_header.prev_state_root()` from shard 1's chunk.
7. Shard 0's trie slot now contains shard 1's state. The node is permanently corrupted for shard 0.

**Exact corrupted entry**: `DBCol::StateHeaders` at key `StateHeaderKey(shard_0, sync_hash)` contains `ShardStateSyncResponseHeaderV2` whose embedded `chunk.shard_id() == 1`. [6](#0-5) [7](#0-6) [8](#0-7)
