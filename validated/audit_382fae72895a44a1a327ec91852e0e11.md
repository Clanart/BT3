### Title
Relayer Funds Permanently Lost in `utxo_fin_transfer_fast` Due to Non-Atomic `remove_fast_transfer` + `send_tokens.detach()` — (`near/omni-bridge/src/lib.rs`)

### Summary
`utxo_fin_transfer_fast` deletes the fast transfer record before attempting to reimburse the relayer, and uses `.detach()` with no failure callback. If `send_tokens` fails for any reason, the relayer's pre-advanced funds are permanently unrecoverable.

### Finding Description
In `utxo_fin_transfer_fast` (lines 2518–2561), when the destination chain is NEAR, the function:
1. Calls `remove_fast_transfer` (line 2530), permanently deleting the fast transfer entry and freeing its storage.
2. Calls `send_tokens(token, relayer, amount, "").detach()` (lines 2542–2548) with no `.then(...)` callback.

`send_tokens` for a non-deployed token issues `ft_transfer(relayer, amount, None)` on the token contract. This call panics/fails if the relayer is not registered (no storage deposit) in the token contract. With `.detach()`, the failure is silently discarded. The fast transfer entry is already gone, so there is no re-entry path. The UTXO transfer is also not recorded in `finalised_utxo_transfers`, so a re-submission by the connector takes the normal (non-fast) path and sends tokens to the original recipient, not the relayer.

The developer explicitly acknowledged this gap with `// TODO: check how to deal with failed send_tokens` at line 2484.

### Impact Explanation
The relayer pre-advanced their own tokens to the recipient (Phase 1 succeeded). On UTXO finalization, the bridge should reimburse the relayer from the UTXO connector's tokens. If `send_tokens` fails:
- The fast transfer entry is deleted — no re-claim path exists.
- The UTXO connector's tokens remain in the bridge contract (bridge returned `U128(0)` from `ft_on_transfer`, keeping all tokens).
- The relayer's pre-advanced tokens are permanently lost.

This is an irrecoverable lock of relayer funds matching the Critical impact category.

### Likelihood Explanation
The trigger requires `send_tokens` to fail after `remove_fast_transfer`. Concrete failure modes:
- **Relayer deregisters from token contract**: A relayer calls `storage_unregister` on the token contract between fast-transfer registration and UTXO finalization. This is a public, permissionless action the relayer controls themselves (possibly by mistake, or due to a separate workflow).
- **Token contract paused**: If the token contract has a pause mechanism and is paused between the two phases.
- **Gas exhaustion edge cases**: Misconfigured gas in the `send_tokens` call path.

The deregistration scenario is realistic and requires no privileged access — only the relayer's own action on the token contract.

### Recommendation
Replace the fire-and-forget `.detach()` with a `.then(callback)` that restores the fast transfer entry (or re-credits the relayer) on failure, mirroring the pattern used in `fast_fin_transfer_to_near_callback` → `resolve_fast_transfer`. Alternatively, call `remove_fast_transfer` only inside the success branch of the callback, not before the async send.

### Proof of Concept
1. Deploy bridge with a non-deployed token (e.g., wBTC).
2. Relayer calls `ft_transfer_call` with `FastFinTransfer` for a NEAR recipient → fast transfer entry created, tokens sent to recipient.
3. Relayer calls `storage_unregister` on the token contract (deregisters themselves).
4. UTXO connector calls `ft_transfer_call` with `UtxoFinTransfer` for the same UTXO.
5. `utxo_fin_transfer_fast` is called: `remove_fast_transfer` deletes the entry; `send_tokens(...).detach()` fails silently because relayer is not registered.
6. Assert: fast transfer entry is absent, relayer token balance is unchanged (pre-advanced tokens lost), UTXO connector tokens are stuck in bridge.

    fn remove_fast_transfer(&mut self, fast_transfer_id: &FastTransferId) {
        let storage_usage = env::storage_usage();
        let fast_transfer = self
            .fast_transfers
            .remove(fast_transfer_id)
            .map(storage::FastTransferStatusStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&fast_transfer.storage_owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances
                .insert(&fast_transfer.storage_owner, &storage);
        }
    }

    fn utxo_fin_transfer_fast(
        &mut self,
        fast_transfer: FastTransfer,
        fast_transfer_status: FastTransferStatus,
        utxo_fin_transfer_msg: UtxoFinTransferMsg,
    ) -> PromiseOrPromiseIndexOrValue<U128> {
        require!(
            !fast_transfer_status.finalised,
            BridgeError::FastTransferAlreadyFinalised.as_ref()
        );

        let amount = if fast_transfer.get_destination_chain() == ChainKind::Near {
            self.remove_fast_transfer(&fast_transfer.id());
            fast_transfer.amount
        } else {
            self.mark_fast_transfer_as_finalised(&fast_transfer.id());
            // With transfers to other chain the fee will be claimed after finalization on the destination chain
            U128(
                fast_transfer
                    .amount_without_fee()
                    .near_expect(BridgeError::InvalidFee),
            )
        };

        self.send_tokens(
            fast_transfer.token_id.clone(),
            fast_transfer_status.relayer,
            amount,
            "",
        )
        .detach();

        env::log_str(
            &OmniBridgeEvent::UtxoTransferEvent {
                token_id: fast_transfer.token_id,
                amount,
                utxo_transfer_message: utxo_fin_transfer_msg,
                new_transfer_id: None,
            }
            .to_log_string(),
        );

        PromiseOrPromiseIndexOrValue::Value(U128(0))
    }
