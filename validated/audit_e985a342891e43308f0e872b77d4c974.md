### Title
Non-EVM-Denom Bank Balances Permanently Locked After Contract `SELFDESTRUCT` — (`File: x/evm/statedb/statedb.go`)

### Summary
When a contract self-destructs, `StateDB.Commit()` burns only the EVM-native denom balance and then calls `DeleteAccount` to remove the auth account. Non-EVM-native tokens (IBC, CosmWasm bridge, etc.) held by the destroyed address are explicitly skipped. After `DeleteAccount` removes the auth account, those bank balances have no authorized spender and are permanently inaccessible — a direct analog to the PositionManager "no retrieval path for held assets" class.

### Finding Description

In `StateDB.Commit()`, the self-destruct branch reads:

```go
// Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.
if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
    coin := sdk.NewCoin(s.evmDenom, sdkmath.NewIntFromBigInt(remaining.ToBig()))
    if _, err := s.keeper.SubBalance(cacheCtx, cosmosAddr, coin); err != nil {
        return errorsmod.Wrap(err, "failed to burn post-selfdestruct balance")
    }
}
if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil {
    return errorsmod.Wrap(err, "failed to delete account")
}

`DeleteAccount` removes the auth account record and all EVM storage, but never touches the Cosmos bank module balances for non-EVM denoms:

```go

After `RemoveAccount`, the Cosmos address has no auth account. The bank module still holds any non-EVM-denom balances at that address, but because there is no account to sign transactions from, those funds can never be spent or recovered. There is no sweep, burn, or transfer of non-EVM-denom tokens anywhere in the self-destruct commit path.

### Impact Explanation

Any IBC tokens, CosmWasm bridge tokens, or other non-EVM-native Cosmos bank balances held by a self-destructed contract are permanently locked. The bank module retains the balance entry, but the auth account is gone, making it impossible for anyone to authorize a spend from that address. This constitutes a permanent, irreversible loss of Cosmos bank funds triggered through Ethermint's stateDB commit logic.

On chains built on Ethermint that expose IBC or bank precompiles, a more severe variant exists: an attacker could deploy a contract at a predictable CREATE2 address, induce users to send IBC tokens to it, SELFDESTRUCT the contract (burning only the EVM denom), then redeploy at the same address and use a bank/IBC precompile to drain the orphaned non-EVM-denom balance — converting loss into theft.

### Likelihood Explanation

- IBC transfers to contract addresses are standard Cosmos SDK behavior; any user can send IBC tokens to any address including EVM contracts.
- Any contract with a callable `SELFDESTRUCT` (or EIP-6780 same-tx destruction) can trigger this path.
- No privileged role is required: the contract deployer controls both the contract code and the destruction trigger.
- The code comment explicitly acknowledges the gap, confirming it is a known, unmitigated condition.

### Recommendation

Before calling `DeleteAccount`, iterate over all bank module balances held by the destroyed address and either:
1. Burn them (consistent with the EVM denom treatment), or
2. Transfer them to a designated recovery module account.

A minimal fix would extend the self-destruct commit block to query all bank balances for the address and burn each non-EVM denom, mirroring the existing `SubBalance` call for the EVM denom.

### Proof of Concept

1. Deploy a contract `C` at a deterministic CREATE2 address.
2. Send IBC tokens (e.g., `uatom`) to `C`'s address via a standard IBC transfer.
3. Call `C.destroy()` (a public `SELFDESTRUCT` function) within the same transaction (EIP-6780) or in a subsequent transaction (pre-EIP-6780 chains).
4. `StateDB.Commit()` burns only the EVM denom balance and calls `DeleteAccount`, removing the auth account.
5. Query the bank module: `uatom` balance at `C`'s address is still non-zero.
6. Attempt any spend from `C`'s address: fails — no auth account exists to authorize the transaction.
7. The `uatom` tokens are permanently locked with no recovery path. [3](#0-2) [4](#0-3)

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account
//
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
