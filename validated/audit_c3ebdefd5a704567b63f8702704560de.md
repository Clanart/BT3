### Title
User-Controlled Zero Recipient Address in `init_transfer` Causes Permanent Fund Loss — (`near/omni-bridge/src/lib.rs`)

---

### Summary

The NEAR bridge's `init_transfer` function accepts any `OmniAddress` as the recipient without validating that the address is non-zero. A user can specify `OmniAddress::Eth(H160::ZERO)` (or the zero address for any other EVM chain) as the transfer recipient. The bridge then burns/locks the user's tokens on NEAR and produces a signed payload with `recipient = address(0)`. On the EVM side, `finTransfer` either permanently burns the ETH (native-ETH path) or always reverts (ERC20/bridge-token path), leaving the NEAR-side tokens irrecoverably burned or locked.

---

### Finding Description

**Root cause — missing zero-address guard in `init_transfer`:** [1](#0-0) 

The only recipient validation is a chain-kind check:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
```

No check is made against `OmniAddress::is_zero()`, which is already implemented in the type system: [2](#0-1) 

The error variant `BridgeError::InvalidRecipientAddress` already exists in the error enum, confirming the protocol intended to validate recipients but omitted the guard here: [3](#0-2) 

**Attack path:**

1. User calls `ft_transfer_call` on a NEAR NEP-141 token with `msg = InitTransferMsg { recipient: OmniAddress::Eth(H160::ZERO), ... }`.
2. `init_transfer` passes the chain-kind check, burns/locks the user's tokens, and stores the transfer message with `recipient = 0x0000…0000`.
3. A relayer calls `sign_transfer`; the NEAR MPC signer signs the `TransferMessagePayload` containing `recipient = address(0)` — no zero-address check exists there either: [4](#0-3) 

4. The relayer submits `finTransfer` on EVM with `payload.recipient = address(0)`.

**EVM-side consequences** (`OmniBridge.sol` `finTransfer`): [5](#0-4) 

- **Native ETH path** (`tokenAddress == address(0)`): `address(0).call{value: amount}("")` succeeds — ETH is permanently burned.
- **Bridge-token path**: `IBridgeToken.mint(address(0), amount)` reverts (OpenZeppelin ERC20 rejects zero-address mint). The nonce is also reverted, so the relayer can retry, but the signed payload is fixed to `recipient = address(0)` — every retry reverts. NEAR tokens are already burned/locked with no recovery path.
- **Native ERC20 path**: `safeTransfer(address(0), amount)` reverts for the same reason.

After `sign_transfer_callback` removes the transfer message for zero-fee transfers, there is no mechanism to re-sign with a corrected recipient: [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent, irrecoverable loss of user funds.**

For the native-ETH path, ETH is burned at `address(0)`. For ERC20/bridge-token paths, the NEAR-side tokens are burned or locked and the EVM `finTransfer` can never succeed (signature is bound to `recipient = address(0)`). No admin recovery or cancel function exists in the NEAR bridge contract. The `BridgeError::InvalidRecipientAddress` error type exists but is never triggered in this code path.

---

### Likelihood Explanation

**Low-Medium.** A user must supply `OmniAddress::Eth(H160::ZERO)` (or equivalent) as the recipient. This can occur through:
- A buggy SDK or integration that defaults unset address fields to zero.
- A user who intentionally or accidentally passes a zero address.

The protocol provides no guard, so any such call succeeds and irreversibly commits the funds.

---

### Recommendation

Add a zero-address guard immediately after the chain-kind check in `init_transfer`:

```rust
require!(
    !init_transfer_msg.recipient.is_zero(),
    BridgeError::InvalidRecipientAddress.as_ref()
);
```

The `is_zero()` method is already implemented on `OmniAddress` and the error variant already exists, so this is a one-line fix.
