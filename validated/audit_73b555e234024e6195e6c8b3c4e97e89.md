### Title
EVM Bridge Locks Native ERC20 Tokens by User-Specified `amount` But Releases by the Same Static Figure, Breaking Collateralization for Rebasing Tokens — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` locks native ERC20 tokens using the caller-supplied `amount` and propagates that exact figure cross-chain. `finTransfer` later releases the same static figure from escrow. For rebasing tokens whose balance changes outside of transfers, the actual balance held by the bridge can diverge from the recorded amount, causing either a permanent freeze of user funds (negative rebase) or irrecoverable surplus accumulation (positive rebase).

---

### Finding Description

When a user bridges a native (non-bridge, non-custom-minter) ERC20 token from EVM to NEAR, `initTransfer` calls `safeTransferFrom(msg.sender, address(this), amount)` and then emits `InitTransfer` with the caller-supplied `amount`: [1](#0-0) 

The emitted `amount` is what the NEAR bridge records and eventually signs into a `TransferMessagePayload`. When the user later bridges back, `finTransfer` on EVM releases exactly `payload.amount` from escrow: [2](#0-1) 

There is no balance-before/balance-after check at lock time, and no reconciliation at release time. The bridge implicitly assumes that the ERC20 balance it holds for a given token equals the sum of all locked `amount` values. For rebasing tokens, this invariant is violated whenever a rebase occurs while tokens sit in escrow.

The `initTransferExtension` (used by `OmniBridgeWormhole`) also propagates the raw `amount` into the Wormhole message payload, not an observed balance delta: [3](#0-2) 

---

### Impact Explanation

**Negative rebase (Critical — permanent freeze):**
The bridge holds fewer tokens than the sum of locked amounts. When `finTransfer` calls `safeTransfer(recipient, payload.amount)`, the call reverts because the contract balance is insufficient. The nonce is not permanently consumed on revert (the whole tx reverts), so the user can retry — but the underlying shortfall persists. The user's NEAR-side tokens have already been burned by the NEAR bridge in the preceding step, yet the EVM tokens cannot be released. The funds are permanently frozen.

**Positive rebase (High — stuck surplus):**
The bridge holds more tokens than the sum of locked amounts. `finTransfer` releases only `payload.amount`, leaving the rebased surplus permanently stranded in the bridge contract with no recovery mechanism.

Both scenarios break bridge collateralization and constitute direct loss or irrecoverable lock of user assets.

---

### Likelihood Explanation

Any unprivileged user can call `initTransfer` with any ERC20 token address — there is no whitelist or token-type validation. Rebasing tokens (e.g., stETH, aTokens, elastic-supply tokens) are widely deployed on EVM chains. A user who bridges such a token triggers the vulnerable path without any special access. The rebase itself is an autonomous on-chain event requiring no attacker action; it occurs naturally during the cross-chain settlement window.

---

### Recommendation

1. **Measure actual received amount** at lock time using a balance-before/balance-after pattern in `initTransfer`, and emit/propagate only the observed delta — not the caller-supplied `amount`.
2. **Measure actual available amount** at release time in `finTransfer`, and release `min(payload.amount, actualBalance)` or revert with a clear error that does not permanently consume the nonce.
3. Alternatively, explicitly reject known rebasing tokens via a denylist or require tokens to pass a non-rebasing interface check before being accepted by the bridge.

---

### Proof of Concept

1. Deploy a rebasing ERC20 token `RBT` on EVM. The bridge is not a bridge token for `RBT` (`isBridgeToken[RBT] == false`).
2. User calls `initTransfer(RBT, 1000, 0, 0, "near:alice.near", "")`.
   - `safeTransferFrom(user, bridge, 1000)` executes; bridge holds 1000 RBT.
   - `InitTransfer` event emitted with `amount = 1000`.
3. NEAR bridge processes the event, mints 1000 wrapped-RBT to `alice.near`.
4. RBT undergoes a negative rebase; bridge's RBT balance drops to 800.
5. Alice sends 1000 wrapped-RBT back via NEAR bridge → NEAR bridge burns 1000 tokens, creates a transfer message with `amount = 1000`, MPC signs it.
6. Relayer calls `finTransfer(sig, {tokenAddress: RBT, amount: 1000, recipient: alice_evm, ...})`.
   - `completedTransfers[nonce] = true` is set.
   - `IERC20(RBT).safeTransfer(alice_evm, 1000)` reverts (bridge holds only 800).
   - Entire transaction reverts; nonce is not consumed.
7. Alice's 1000 NEAR-side tokens are permanently burned. She cannot recover her EVM tokens because every retry of step 6 will revert until the rebase reverses — which may never happen. Funds are permanently frozen. [4](#0-3) [5](#0-4)

        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
