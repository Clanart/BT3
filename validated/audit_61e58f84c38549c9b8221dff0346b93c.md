### Title
Missing Validation of Constructor Entry Points in `validate_compiled_class_facts` - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/contract_class/compiled_class.cairo`)

### Summary
`validate_compiled_class_facts` validates selector ordering for `external_functions` and `l1_handlers` entry points but entirely omits the same validation for `constructors`. An unprivileged class declarer can declare a compiled class whose constructor entry-point table has unsorted selectors. Because the OS entry-point lookup relies on the table being sorted (binary search), the wrong constructor — or no constructor at all — may be invoked when a contract is deployed from that class, leaving the contract in an uninitialized state and permanently freezing any funds sent to it.

### Finding Description

`validate_compiled_class_facts` iterates over every compiled class and calls `validate_entry_points` for two of the three entry-point tables:

```cairo
validate_entry_points(
    n_entry_points=compiled_class.n_external_functions,
    entry_points=compiled_class.external_functions,
);

validate_entry_points(
    n_entry_points=compiled_class.n_l1_handlers, entry_points=compiled_class.l1_handlers
);
// ← no call for compiled_class.constructors
``` [1](#0-0) 

`validate_entry_points` / `validate_entry_points_inner` enforce strict selector ordering via `assert_lt_felt`:

```cairo
func validate_entry_points_inner{range_check_ptr}(
    n_entry_points: felt, entry_points: CompiledClassEntryPoint*, prev_selector
) {
    ...
    assert_lt_felt(prev_selector, entry_points[0].selector);
    ...
}
``` [2](#0-1) 

This ordering guarantee is required for the binary-search entry-point lookup the OS performs at execution time. Because the constructor table is never subjected to this check, a class with multiple constructor entry points whose selectors are not strictly increasing passes `validate_compiled_class_facts` without error.

The deprecated class path validates `external_functions` and `l1_handlers` in the same pattern, and `DeprecatedCompiledClass` explicitly carries `n_constructors` / `constructors` fields: [3](#0-2) 

The non-deprecated `CompiledClass` follows the same layout convention; the omission of constructor validation in `validate_compiled_class_facts` is therefore a structural gap.

### Impact Explanation

When a contract is deployed (`execute_deploy_account_transaction` → `deploy_contract`), the OS performs a binary search over the constructor entry-point table to locate `CONSTRUCTOR_ENTRY_POINT_SELECTOR`. [4](#0-3) 

If the table is unsorted, the binary search may:
- **Return "not found"** → the constructor is silently skipped; the contract is deployed in a completely uninitialized state.
- **Return the wrong entry point** → an unintended code path is executed as the constructor.

A contract whose constructor is supposed to set up ownership, access control, or vault parameters is deployed without that initialization. Any funds subsequently sent to the contract are permanently frozen because the privileged functions that would allow withdrawal are never reachable.

**Impact category: Critical — Permanent freezing of funds.**

### Likelihood Explanation

- Any unprivileged **class declarer** can submit a compiled class with multiple constructor entry points whose selectors are deliberately out of order; the OS accepts it because no Cairo-level constraint rejects it.
- The class hash is computed and stored normally; nothing in the hash computation enforces selector ordering for constructors.
- A victim (or the attacker themselves) then deploys a contract from that class and deposits funds, triggering the freeze.
- While multiple constructor entry points are uncommon in typical Sierra/CASM output, the protocol imposes no upper bound on the constructor table length, and a crafted class can carry arbitrarily many entries.

### Recommendation

Add a `validate_entry_points` call for the constructor table inside `validate_compiled_class_facts`, immediately after the existing two calls:

```cairo
validate_entry_points(
    n_entry_points=compiled_class.n_constructors,
    entry_points=compiled_class.constructors,
);
```

This mirrors the treatment already applied to `external_functions` and `l1_handlers` and closes the gap at zero additional complexity cost. [5](#0-4) 

### Proof of Concept

1. Craft a Sierra class whose compiled (CASM) form contains two constructor entry points with selectors `[S2, S1]` where `S2 > S1` (deliberately reversed).
2. Submit a `declare` transaction. `validate_compiled_class_facts` runs, calls `validate_entry_points` for `external_functions` and `l1_handlers` only — the constructor table is never checked

func validate_entry_points_inner{range_check_ptr}(

    if (n_entry_points == 0) {
        return ();
    }

    return validate_entry_points_inner(
        n_entry_points=n_entry_points - 1,
        entry_points=&entry_points[1],

func validate_compiled_class_facts{poseidon_ptr: PoseidonBuiltin*, range_check_ptr}(
    n_compiled_class_facts, compiled_class_facts: CompiledClassFact*, builtin_costs: felt*
) {
    if (n_compiled_class_facts == 0) {
        return ();
    }
    alloc_locals;

    let compiled_class_fact = &compiled_class_facts[0];
    let compiled_class = compiled_class_fact.compiled_class;

    validate_entry_points(
        n_entry_points=compiled_class.n_external_functions,
        entry_points=compiled_class.external_functions,
    );

    validate_entry_points(
        n_entry_points=compiled_class.n_l1_handlers, entry_points=compiled_class.l1_handlers
    );
    // Compiled classes are expected to end with a `ret` opcode followed by a pointer to the
    // builtin costs.
    assert compiled_class.bytecode_ptr[compiled_class.bytecode_length] = 0x208b7fff7fff7ffe;
    assert compiled_class.bytecode_ptr[compiled_class.bytecode_length + 1] = cast(
        builtin_costs, felt
    );

    // Calculate the compiled class hash.
    // This hint enters a new scope that contains the bytecode segment structure of the class.
    %{ EnterScopeWithBytecodeSegmentStructure %}
    let (hash) = blake_compiled_class_hash(compiled_class, full_contract=FALSE);
    %{ LoadClass %}

    assert compiled_class_fact.hash = hash;

    return validate_compiled_class_facts(
        n_compiled_class_facts=n_compiled_class_facts - 1,
        compiled_class_facts=&compiled_class_facts[1],
        builtin_costs=builtin_costs,
    );
}
