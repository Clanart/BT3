### Title
Hardcoded `high=0` in Uint256 Fee Amount Inverts the Overcharge Guard When `max_fee ≥ 2¹²⁸` — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `charge_fee` function constructs the ERC-20 transfer amount as `Uint256(low=low_actual_fee, high=0)`, hardcoding the high 128-bit limb to zero. The guard `assert_nn_le(calldata.amount.low, max_fee)` is intended to prevent the sequencer from charging more than `max_fee`. However, `max_fee` is a raw felt product of user-supplied resource-bound fields and can legitimately exceed 2¹²⁸. When it does, the range-check semantics of `assert_nn_le` invert: instead of acting as an upper bound, the check becomes a **lower bound** on the fee, forcing the sequencer to overcharge the user. If `max_fee ≥ 2¹²⁹ − 1`, no valid `low_actual_fee` exists and the OS halts entirely.

---

### Finding Description

`compute_max_possible_fee` returns a plain felt:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

`max_amount` is a 64-bit field and `max_price_per_unit` is a 128-bit field, so each product can reach ≈ 2¹⁹², and the sum of three such products can be an arbitrary felt well above 2¹²⁸. No upper-bound check is applied to `max_fee` before it is used.

In `charge_fee`, the actual fee is then constructed and guarded:

```cairo
local low_actual_fee;
%{ LoadActualFee %}
local calldata: TransferCallData = TransferCallData(
    recipient=block_context.block_info_for_execute.sequencer_address,
    amount=Uint256(low=low_actual_fee, high=0),   // ← high hardcoded to 0
);
assert_nn_le(calldata.amount.low, max_fee);        // ← only low limb checked
``` [2](#0-1) 

`assert_nn_le(a, b)` in Cairo expands to:
- `assert_nn(a)` → range-check: `0 ≤ a < 2¹²⁸`
- `assert_nn(b − a)` → range-check: `0 ≤ b − a < 2¹²⁸`

When `max_fee ≥ 2¹²⁸`, the second check requires `low_actual_fee ≥ max_fee − (2¹²⁸ − 1)`. Because `low_actual_fee` must itself be `< 2¹²⁸` (first check), a valid value exists only when `max_fee < 2¹²⁹ − 1`. In that window the sequencer is **forced** to charge at least `max_fee − (2¹²⁸ − 1)` tokens — far more than the actual gas cost — to keep the range check from failing. If `max_fee ≥ 2¹²⁹ − 1`, no valid `low_actual_fee` exists and the OS execution aborts.

---

### Impact Explanation

**Case 1 — Direct loss of funds (Critical):** When `2¹²⁸ ≤ max_fee < 2¹²⁹ − 1`, the sequencer must set `low_actual_fee ≥ max_fee − (2¹²⁸ − 1)` to produce a valid proof. The actual execution cost is typically orders of magnitude smaller. The user is overcharged by up to `2¹²⁸ − 1` tokens, which are transferred to the sequencer. The user cannot recover these funds.

**Case 2 — Network halt (High):** When `max_fee ≥ 2¹²⁹ − 1`, the OS cannot satisfy both range checks simultaneously. If the sequencer includes such a transaction in a block, proof generation fails and the block cannot be finalized, halting the network.

---

### Likelihood Explanation

Any unprivileged transaction sender can trigger this by setting resource bounds whose product exceeds 2¹²⁸. For example, setting `max_amount = 2` and `max_price_per_unit = 2¹²⁷` for a single resource type already yields `max_fee ≥ 2¹²⁸`. The OS performs no validation that `max_fee` fits in 128 bits before calling `charge_fee`. The transaction hash computation and signature validation both succeed for such inputs, so the transaction reaches `charge_fee` without being rejected.

---

### Recommendation

1. **Bound `max_fee` at the OS level.** After computing `max_fee`, assert `max_fee < 2¹²⁸` (or the chosen token precision) before proceeding to `charge_fee`.
2. **Use a full Uint256 for the fee amount.** If fees can legitimately exceed 2¹²⁸ tokens, represent `low_actual_fee` as a proper `Uint256` with both limbs loaded from the hint, and validate `amount ≤ max_fee` using 256-bit comparison rather than a single `assert_nn_le` on the low limb.
3. **Validate resource-bound magnitudes during transaction hash computation** so that transactions with `max_fee ≥ 2¹²⁸` are rejected before they reach execution.

---

### Proof of Concept

1. Craft a V3 invoke transaction with:
   - `l2_gas_bounds.max_amount = 2`
   - `l2_gas_bounds.max_price_per_unit = 2¹²⁷`
   - Other resource bounds set to zero.
2. `compute_max_possible_fee` returns `2 × 2¹²⁷ = 2¹²⁸`.
3. In `charge_fee`, `assert_nn_le(low_actual_fee, 2¹²⁸)` requires `2¹²⁸ − low_actual_fee < 2¹²⁸`, i.e., `low_actual_fee ≥ 1`.
4. The actual execution cost is, say, 500 gas units × price = 500 tokens. The sequencer sets `low_actual_fee = 500`. Then `2¹²⁸ − 500 ≥ 2¹²⁸ − 2¹²⁸ + 1 = 1` — wait, `2¹²⁸ − 500 < 2¹²⁸`, so this actually passes.

Let me sharpen: set `max_amount = 3`, `max_price_per_unit = 2¹²⁷`. Then `max_fee = 3 × 2¹²⁷ = 2¹²⁷ × 3 = 2¹²⁸ + 2¹²⁷`. Now `assert_nn_le(low_actual_fee, 2¹²⁸ + 2¹²⁷)` requires `low_actual_fee ≥ (2¹²⁸ + 2¹²⁷) − (2¹²⁸ − 1) = 2¹²⁷ + 1`. The actual fee is 500 tokens. The sequencer must set `low_actual_fee ≥ 2¹²⁷ + 1 ≈ 1.7 × 10³⁸` tokens to avoid an OS abort — overcharging the user by `≈ 2¹²⁷` tokens. The ERC-20 transfer executes with this inflated amount, draining the user's balance. [3](#0-2) [4](#0-3)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
```
