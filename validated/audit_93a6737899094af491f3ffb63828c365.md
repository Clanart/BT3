### Title
Unchecked Multiplication Overflow in `denormalize_amount` Permanently Locks User Funds on Origin Chain - (File: near/omni-bridge/src/lib.rs)

### Summary

`denormalize_amount` performs a bare `u128` multiplication without overflow protection. With `overflow-checks = true` set in the NEAR release profile, an overflow causes a WASM trap (panic) inside `fin_transfer_callback`. Because the funds are already locked on the origin chain (EVM) before the callback executes, a panicking callback permanently freezes the user's assets with no recovery path.

### Finding Description

`denormalize_amount` at `near/omni-bridge/src/lib.rs` lines 2776–2779:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))   // ← unchecked multiplication
}

`denormalize_amount` is called unconditionally inside `fin_transfer_callback` at line 725 with the amount extracted directly from the prover result:

``` [3](#0-2) 

`fin_transfer_callback` is the NEAR-side callback that finalises an inbound cross-chain transfer. It is scheduled by `fin_transfer` after proof verification: [4](#0-3) 

The `Decimals` struct stores two independent `u8` fields: [5](#0-4) 

For a token whose origin chain has `origin_decimals = 24` (e.g. NEAR native) and whose EVM representation uses `decimals = 18`, `diff_decimals = 6` and `10_u128.pow(6) = 1_000_000`. Any EVM-side amount greater than `u128::MAX / 1_000_000 ≈ 3.4 × 10^32` (in the 18-decimal unit) causes the multiplication to overflow. The EVM bridge accepts amounts up to `uint128::MAX ≈ 3.4 × 10^38`, so the overflow range is fully reachable on-chain.

`denormalize_fee` also delegates to `denormalize_amount` for the token-denominated fee component, creating a second overflow surface on the same callback path: [6](#0-5) 

### Impact Explanation

When the overflow trap fires inside `fin_transfer_callback`, the NEAR transaction is aborted. The transfer message is never stored, so the destination-side finalisation can never succeed. The origin-chain funds (locked in the EVM bridge vault at `initTransfer` time) have no corresponding recovery or cancellation mechanism triggered by a failed NEAR callback. The result is an irrecoverable lock of the user's assets.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

### Likelihood Explanation

`fin_transfer` is gated by `#[trusted_relayer]`, but the trusted relayer is a permissionless-entry role (any account can `apply_for_trusted_relayer` and stake the required NEAR). The relayer acts as a messenger: it submits a valid proof of the EVM event and has no reason to inspect whether the encoded amount will overflow on NEAR. The amount itself is set by the unprivileged user who called `initTransfer` on EVM. [7](#0-6) 

For tokens with larger decimal differences (e.g. `diff_decimals = 18`), the overflow threshold drops to `u128::MAX / 10^18 ≈ 3.4 × 10^20` in the smallest EVM unit — roughly 340 whole tokens for an 18-decimal EVM token. For the common NEAR-native case (`diff_decimals = 6`) the threshold is much higher, making accidental triggering unlikely, but the absence of any guard means the protocol relies entirely on token economics rather than code correctness to prevent the loss.

### Recommendation

Replace the bare multiplication with a checked variant and propagate the error:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> Option<u128> {
    let diff_decimals: u32 = decimals.origin_decimals.checked_sub(decimals.decimals)?.into();
    let multiplier = 10_u128.checked_pow(diff_decimals)?;
    amount.checked_mul(multiplier)
}
```

Callers (`fin_transfer_callback`, `fast_fin_transfer`, `claim_fee_callback`) should treat `None` as a hard error and revert with a descriptive `BridgeError` variant rather than panicking, so that the NEAR transaction fails cleanly without corrupting any state that was already written.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 18` (`diff_decimals = 6`).
2. On EVM, call `initTransfer` with `amount = u128::MAX / 1_000_000 + 1` (≈ `3.4 × 10^32 + 1`). Funds are locked in the EVM vault.
3. A trusted relayer calls `fin_transfer` on NEAR with the corresponding proof.
4. Inside `fin_transfer_callback`, `denormalize_amount(amount, decimals)` executes `(3.4×10^32 + 1) * 10^6`, which exceeds `u128::MAX`.
5. With `overflow-checks = true`, the WASM runtime traps; the callback panics.
6. The NEAR transaction is rolled back; no `TransferMessage` is stored; no tokens are minted or released.
7. The EVM vault retains the locked funds indefinitely — no refund path exists.
