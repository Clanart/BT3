### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the supplied `class_hash` is non-zero (i.e., not `UNINITIALIZED_CLASS_HASH = 0`) or that it corresponds to a declared class. This is structurally identical to the ClaggSyncAdapter bug: a state-transition function that accepts an empty/null target, permanently locking the associated assets. Any contract that calls `replace_class(0)` — intentionally or due to a bug — will have its class set to the uninitialized sentinel value, making it permanently non-executable while its storage (funds) remains intact and inaccessible.

---

### Finding Description

In `execute_replace_class` (lines 877–916 of `syscall_impls.cairo`), the OS reads the requested `class_hash` directly from the syscall request and writes it into the contract's `StateEntry` without any validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The in-code `TODO` at line 898 explicitly acknowledges the missing check. The constant `UNINITIALIZED_CLASS_HASH = 0` is defined in `commitment.cairo` (line 16) and is the sentinel value used by `deploy_contract.cairo` (line 53) to assert a contract slot is empty before deployment:

```cairo
assert state_entry.class_hash = UNINITIALIZED_CLASS_HASH;
```

If `replace_class(0)` is accepted by the OS, the contract's `class_hash` is written back as `0`. All subsequent calls to that contract will resolve to `class_hash = 0`, for which no compiled class exists. Execution will fail unconditionally. The contract's storage — including any token balances or locked assets — remains committed to the Merkle tree but is permanently inaccessible.

The `get_contract_state_hash` function in `commitment.cairo` (lines 55–61) confirms that a contract with `class_hash = 0` but non-zero storage or nonce still produces a non-zero leaf hash, meaning the state is committed and the funds are provably present but unreachable:

```cairo
if (class_hash == UNINITIALIZED_CLASS_HASH) {
    if (storage_root == 0) {
        if (nonce == 0) {
            return (hash=0);
        }
    }
}
```

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract holding assets (ERC-20 balances, ETH, NFTs stored as storage slots) that transitions its `class_hash` to `0` via `replace_class` becomes permanently non-executable. The funds are committed to the global state root and cannot be recovered by any protocol mechanism. There is no upgrade path, no rescue function, and no revert possible after the block is finalized.

---

### Likelihood Explanation

The `replace_class` syscall is callable by any contract on itself. Realistic trigger paths include:

1. **Malicious deployer rug-pull**: A contract deployer creates a vault or token contract, attracts user deposits, then calls `replace_class(0)` to permanently freeze all deposited funds.
2. **Buggy contract**: A contract that does not validate the `class_hash` argument before invoking `replace_class` (e.g., passing a user-supplied value) can be tricked into calling `replace_class(0)`.
3. **Exploit of an upgradeable contract**: An attacker exploiting a vulnerability in an upgradeable contract's access control can call `replace_class(0)` to freeze all funds held by the protocol.

The OS is the last line of defense. Because it performs no validation, all three paths succeed.

---

### Recommendation

In `execute_replace_class`, add an explicit guard before writing the new state entry:

1. **Reject `class_hash = 0`**: Assert `class_hash != UNINITIALIZED_CLASS_HASH` before updating the state.
2. **Validate declared class**: Implement the acknowledged TODO — verify that `class_hash` exists in `contract_class_changes` (i.e., has been declared in the current or a prior block) before accepting the replacement.

---

### Proof of Concept

1. Attacker deploys a contract `Vault` that accepts user deposits and exposes a public `upgrade(class_hash: felt)` function that calls the `replace_class` syscall with the supplied argument.
2. Users deposit funds into `Vault`.
3. Attacker calls `upgrade(0)`.
4. The OS executes `execute_replace_class` with `class_hash = 0`. No validation is performed (line 896–910 of `syscall_impls.cairo`). The `StateEntry` for `Vault` is updated: `class_hash = 0`, storage unchanged.
5. The block is finalized. The global state root now commits `Vault` with `class_hash = 0` and non-zero storage.
6. Any subsequent `call_contract` or `invoke` targeting `Vault` resolves `class_hash = 0`, finds no compiled class, and fails unconditionally.
7. All user funds are permanently frozen. No recovery is possible at the protocol level.
