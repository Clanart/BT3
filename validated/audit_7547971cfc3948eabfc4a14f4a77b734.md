### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not validate that the requested new class hash corresponds to a previously declared contract class. An unprivileged contract deployer can call `replace_class` with any arbitrary felt value as the class hash. The OS accepts and commits this change without checking whether the class hash exists in the declared class registry. Any subsequent call to the affected contract will fail irrecoverably at the OS level, permanently freezing all funds held by that contract.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall:

```cairo
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The TODO comment at line 898 explicitly acknowledges the missing check. The function unconditionally writes `request.class_hash` — a fully attacker-controlled felt value — into `contract_state_changes` for `contract_address` without verifying that this class hash was ever declared (i.e., that it exists in `contract_class_changes` with a non-zero compiled class hash).

When any subsequent transaction attempts to call the affected contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
```

A `dict_read` on an undeclared class hash returns 0 (the default). `find_element` with key 0 will not find a match in `compiled_class_facts`, causing the OS Cairo program to abort. The contract becomes permanently uncallable, and all funds it holds are irrecoverably frozen.

This is the direct analog of the external report: just as `setTradingAccountToken` accepted any arbitrary address with no validation, `execute_replace_class` accepts any arbitrary class hash with no validation, enabling a state-transition that permanently destroys the usability of the affected contract.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that holds user funds (e.g., a vault, multi-sig, or token contract) and calls `replace_class` with an undeclared class hash will have those funds permanently frozen. The contract becomes uncallable at the OS level. There is no recovery path: the class hash is committed to the state trie, and no future transaction can call the contract to withdraw funds.

---

### Likelihood Explanation

**Medium.** The entry path requires a contract to issue the `replace_class` syscall with an invalid class hash. This is reachable by:

1. An attacker who deploys a contract, attracts user deposits, then calls `replace_class` with an arbitrary undeclared felt value to brick the contract and freeze deposited funds.
2. A contract with a publicly callable `replace_class` wrapper (no internal access control) that an attacker can invoke with a crafted class hash.
3. A buggy contract that passes an unvalidated user-supplied value to `replace_class`.

The OS is the last line of defense and should reject invalid class hashes regardless of contract-level logic.

---

### Recommendation

In `execute_replace_class`, before committing the new class hash to `contract_state_changes`, validate that `request.class_hash` is a declared class by reading `contract_class_changes` and asserting the result is non-zero:

```cairo
// Validate that the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(
    key=class_hash
);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This resolves the acknowledged TODO and mirrors the validation that `execute_entry_point` implicitly requires.

---

### Proof of Concept

1. Attacker deploys contract `VaultWithReplaceClass` that:
   - Accepts user deposits (stores balances in storage).
   - Exposes a public `replace_class(new_hash: felt)` function that calls the `replace_class` syscall with the caller-supplied `new_hash`.

2. Users deposit funds into `VaultWithReplaceClass`.

3. Attacker calls `replace_class(new_hash=0xdeadbeef)` where `0xdeadbeef` is never declared.

4. `execute_replace_class` in the OS writes `class_hash=0xdeadbeef` into `contract_state_changes` for the vault address with no validation. [1](#0-0) 

5. Any subsequent transaction calling the vault reaches `execute_entry_point`, which does `dict_read(contract_class_changes, key=0xdeadbeef)` → returns 0, then `find_element(..., key=0)` → not found → OS aborts. [2](#0-1) 

6. All user funds in the vault are permanently frozen. The contract is irrecoverably bricked.

The root cause is the missing declared-class validation at line 898, explicitly flagged by the development team's own TODO comment. [3](#0-2)
