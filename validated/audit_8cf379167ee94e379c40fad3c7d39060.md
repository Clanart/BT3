### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Bricking and Fund Freeze — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash supplied by the caller corresponds to a previously declared class. An unprivileged contract can call `replace_class` with an arbitrary, undeclared class hash, transitioning its own state to an irrecoverable configuration. Any subsequent call to that contract will fail at the proof level (not merely revert), permanently freezing all funds held by the contract.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The developer-acknowledged TODO at line 898 confirms the missing prerequisite check. The function unconditionally writes `class_hash` (attacker-controlled) into `contract_state_changes` without verifying that `class_hash` exists as a key in `contract_class_changes` (i.e., was previously declared via a `DECLARE` transaction).

**Contrast with the declare path**, which enforces `prev_value=0` to prevent re-declaration and always writes a non-zero `compiled_class_hash`:

```cairo
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
```

**Contrast with `deploy_contract`**, which also does not validate the class hash is declared but at least requires the contract address to be uninitialized (`assert state_entry.class_hash = UNINITIALIZED_CLASS_HASH`).

When a subsequent transaction calls the bricked contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
```

For an undeclared class hash, `dict_read` returns `0` (the Cairo dict default). Then:

```cairo
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,  // = 0
);
```

`find_element` with `key=0` will fail to locate any compiled class fact (no legitimate class has hash `0`), causing the Cairo proof to be **unsatisfiable**. The sequencer cannot include any transaction that calls this contract in a provable block. The contract is permanently inaccessible.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract holding token balances, acting as a vault, escrow, or DeFi pool can be permanently bricked. Once `replace_class` is called with an undeclared hash:

1. The `contract_state_changes` dict records the new (invalid) class hash for the contract address.
2. Every future call to the contract produces an unprovable execution trace.
3. The sequencer is forced to exclude all such transactions.
4. All assets stored in the contract's storage (ERC-20 balances, ETH, etc.) become permanently inaccessible — a direct, irreversible loss of user funds.

There is no recovery path: the OS has no mechanism to "un-replace" a class hash once committed to state, and the invalid state cannot be proven through.

---

### Likelihood Explanation

**Medium-High.**

The attack surface is broad:

- **Malicious contract developer (rug-pull):** A developer deploys a contract, attracts user deposits, then calls `replace_class` with `0` or any random felt. All deposited funds are frozen instantly.
- **Buggy contract:** A contract with a logic error passes an incorrect hash to `replace_class` (e.g., passing a storage value or calldata argument that was never declared). The result is identical.
- **No privileged access required:** Any deployed contract can call `replace_class` on itself. The syscall is available to all contracts without restriction.
- The TODO comment in the production code confirms the developers are aware of the missing check and have not yet implemented it.

---

### Recommendation

Inside `execute_replace_class`, before writing the new class hash to `contract_state_changes`, verify that `class_hash` is present in `contract_class_changes` (i.e., has a non-zero compiled class hash):

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=class_hash
);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the prerequisite enforcement pattern used in `execute_declare_transaction` (`assert_not_zero(compiled_class_hash)`) and closes the state-transition bypass.

---

### Proof of Concept

1. Declare class `A` (valid, compiled class hash `H_A`). Deploy contract `C` using class `A`. `C` holds user funds.
2. `C` internally calls `replace_class(class_hash=0xDEAD)` where `0xDEAD` was never declared.
3. The OS executes `execute_replace_class`: no check is performed; `contract_state_changes[C].class_hash` is set to `0xDEAD`.
4. A user submits a transaction calling `C`.
5. `execute_entry_point` calls `dict_read(contract_class_changes, key=0xDEAD)` → returns `0`.
6. `find_element(compiled_class_facts, key=0)` → no match; Cairo assertion fails.
7. The block containing this transaction cannot be proven. The sequencer must drop it.
8. No transaction can ever successfully call `C` again. All funds in `C` are permanently frozen.
