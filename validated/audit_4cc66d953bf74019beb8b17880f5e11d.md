Audit Report

## Title
Unpermissioned `refundETH()` drains ETH silently trapped by payable swap and liquidity entry-points - (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
Every swap and liquidity entry-point on `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` is `payable`, but `PeripheryPayments.pay()` only consumes native ETH when `token == WETH`. For any other ERC-20 input token, all `msg.value` is silently left in the contract; for WETH inputs, only exactly `value` is wrapped and any surplus remains. `refundETH()` is unpermissioned and forwards the entire contract balance to `msg.sender`, so any ETH stranded between transactions is claimable by an arbitrary caller.

## Finding Description
`PeripheryPayments.pay()` branches on `token == WETH`:

```solidity
// PeripheryPayments.sol L69-88
} else {
    IERC20(token).safeTransferFrom(payer, recipient, value); // ETH completely ignored
}
```

For non-WETH tokens the `else` branch executes `safeTransferFrom` and never reads `address(this).balance`, so any `msg.value` attached to the call is permanently stranded. For WETH tokens, line 76 wraps exactly `value`, leaving `nativeBalance - value` in the contract when the caller over-sends.

None of the four `payable` swap functions (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) nor the two `payable` liquidity functions (`addLiquidityExactShares`, `addLiquidityWeighted`) contain any `msg.value` guard. A grep for `msg.value ==`, `msg.value !=`, or `require.*msg.value` across all periphery contracts returns zero matches, confirming no on-chain protection exists.

The `receive()` fallback at `PeripheryPayments.sol` L32-34 only blocks plain ETH transfers (no calldata); it does not fire when ETH is delivered as `msg.value` to a named `payable` function, so it provides no protection here.

The stranded ETH is then exposed via:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // sends to caller, not original depositor
    }
}
```

`refundETH()` has no access control and sends the **entire** contract balance to whoever calls it. Any ETH left in the contract between transactions is claimable by any address.

## Impact Explanation
Direct loss of user principal. A user who sends ETH alongside a non-WETH swap loses 100% of that ETH unless they include `refundETH()` in the same `multicall` batch. A user who over-sends ETH for a WETH swap loses the surplus. In both cases a front-runner monitoring the mempool can call `refundETH()` in the same block and redirect the stranded ETH to themselves. The loss is bounded only by the amount the victim sent; there is no protocol-side cap. This constitutes a Critical/High direct loss of user principal under the allowed impact gate.

## Likelihood Explanation
All four swap functions and both liquidity functions are `payable`, making it natural for users and integrators to send ETH. The WETH-as-native-ETH pattern (send ETH, router wraps it) is standard in Uniswap-style periphery; users familiar with that pattern will routinely send ETH even for non-WETH tokens or send a rounded-up amount. `multicall` batching is the intended usage pattern; a user who omits `refundETH()` from the batch silently loses funds with no on-chain revert or warning. The attack requires no special privilege and is repeatable on every affected transaction.

## Recommendation
1. In `exactInputSingle` / `exactOutputSingle`: add a guard before the swap:
   ```solidity
   if (params.tokenIn != WETH) {
       require(msg.value == 0, "ETH not accepted for ERC20 input");
   } else {
       require(msg.value == params.amountIn, "msg.value must equal amountIn");
   }
   ```
2. In `exactInput` / `exactOutput`: apply the same check against `params.tokens[0]` and `params.amountIn`.
3. In `MetricOmmPoolLiquidityAdder`: add analogous guards in `addLiquidityExactShares` and `addLiquidityWeighted` for both pool tokens.
4. Alternatively, make `refundETH()` accept a caller-supplied `recipient` parameter and restrict it to the address that deposited ETH in the same transaction, reducing the theft surface even if ETH is accidentally trapped.

## Proof of Concept
**Scenario (Path A – non-WETH token):**
1. Alice calls `exactInputSingle` with `tokenIn = USDC`, `amountIn = 1000e6`, and attaches `msg.value = 1 ether`.
2. The pool callback fires; `pay(USDC, Alice, pool, 1000e6)` hits the `else` branch at `PeripheryPayments.sol` L86 and calls `USDC.safeTransferFrom(Alice, pool, 1000e6)`. The 1 ETH is never touched.
3. The swap completes successfully; Alice receives her output tokens. Her 1 ETH sits in the router.
4. Bob observes the transaction and calls `refundETH()`.
5. `refundETH()` at L61 sends `address(this).balance` (Alice's 1 ETH) to Bob. Alice's ETH is permanently lost.

**Scenario (Path B – WETH token, excess ETH):**
1. Alice calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = 0.5 ether`, and sends `msg.value = 1 ether`.
2. `pay(WETH, Alice, pool, 0.5e18)` at L75-77 sees `nativeBalance = 1 ether >= value = 0.5 ether`, wraps exactly 0.5 ETH, and transfers it to the pool. The remaining 0.5 ETH stays in the router.
3. Bob calls `refundETH()` and receives Alice's 0.5 ETH.
