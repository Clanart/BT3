### Title
Unverified `block_header_commitments` from P2P sync stored in `PartialBlockHashComponents` produces wrong block hash - (File: crates/apollo_batcher/src/batcher.rs)

### Summary

In `add_sync_block`, the `block_header_commitments` received from a P2P sync peer are stored directly into `PartialBlockHashComponents` without verifying `transaction_commitment`, `event_commitment`, `receipt_commitment`, or `concatenated_counts` against the actual transaction hashes and events present in the same `SyncBlock`. Only `state_diff_commitment` is later verified by the committer (when `verify_state_diff_hash = true`). A malicious P2P peer can supply a correct `state_diff_commitment` alongside a forged `transaction_commitment`, causing the sequencer to derive and permanently store a wrong block hash. That wrong hash propagates to RPC responses, to the `get_block_hash` syscall, and to subsequent block hash chains.

### Finding Description

**Root cause — `add_sync_block` stores commitments without cross-checking them against block contents**

`add_sync_block` destructures the incoming `SyncBlock` into `state_diff`, `account_transaction_hashes`, `l1_transaction_hashes`, and `block_header_commitments`, then immediately wraps the commitments into a `PartialBlockHashComponents` struct and writes it to storage: [1](#0-0) 

The `account_transaction_hashes` and `l1_transaction_hashes` fields are present in the same struct but are never used to recompute or verify `header_commitments.transaction_commitment`. Similarly, `event_commitment`, `receipt_commitment`, and `concatenated_counts` are accepted verbatim.

**Only `state_diff_commitment` is verified — the other four fields are not**

The extracted `state_diff_commitment` is forwarded to the committer: [2](#0-1) 

Inside `commit_block_inner`, the committer optionally checks that the provided commitment matches `calculate_state_diff_hash(&state_diff)`: [3](#0-2) 

`transaction_commitment`, `event_commitment`, `receipt_commitment`, and `concatenated_counts` are never touched by this check.

**P2P sync stores `SignedBlockHeader` without signature verification**

In the P2P sync path, `parse_data_for_block` only validates the number of signatures, not their cryptographic validity: [4](#0-3) 

A malicious peer can therefore send a `SignedBlockHeader` carrying an arbitrary `transaction_commitment` (or any other commitment field) and it will be accepted and stored.

**The stored `PartialBlockHashComponents` (with forged commitments) are used to compute the final block hash**

`finalize_commitment_output` reads the stored `PartialBlockHashComponents` and calls `calculate_block_hash`: [5](#0-4) 

`calculate_block_hash` chains every field of `header_commitments` — including the forged `transaction_commitment` — into the Poseidon hash: [6](#0-5) 

The resulting wrong block hash is written to storage via `set_global_root_and_block_hash`.

**The OS also accepts `block_hash_commitments` from external input without computing them**

The Cairo OS `get_block_hashes` function explicitly notes that header commitments are not computed by the OS and are instead "guessed" via the `GetBlockHashes` hint: [7](#0-6) 

The Rust hint implementation populates these from `os_block_input.block_hash_commitments`: [8](#0-7) 

If the stored `PartialBlockHashComponents` carry a forged `transaction_commitment`, the OS will embed it in the proof output without independent verification.

### Impact Explanation

**High — RPC returns authoritative-looking wrong block hash.** `starknet_getBlockWithTxHashes` and related endpoints read the block hash directly from storage. After the attack, every RPC caller receives a wrong block hash for the affected height.

**Critical — `get_block_hash` syscall returns wrong value.** Contracts that call `get_block_hash` for the affected block number receive the forged hash, corrupting any on-chain logic that depends on it (e.g., cross-chain bridges, randomness beacons, commitment schemes).

**Cascading wrong parent hashes.** The wrong block hash is used as `previous_block_hash` when computing the hash of the next block, so every subsequent block hash is also wrong.

**Wrong proof output.** The OS embeds the forged commitments in the SNOS output; the proof prover attests to a block hash that does not correspond to the actual transactions.

### Likelihood Explanation

Any node that enables P2P sync (a production feature) is reachable by an unprivileged peer. The attacker only needs to:
1. Compute the correct `state_diff_commitment` from the publicly known state diff (to pass the committer's check).
2. Set `transaction_commitment` to an arbitrary value.
3. Serve the crafted `SignedBlockHeader` to the victim node.

No privileged access, no key material, and no consensus participation is required.

### Recommendation

1. **Verify all commitments in `add_sync_block`.** After receiving `block_header_commitments`, recompute `transaction_commitment` from `account_transaction_hashes` and `l1_transaction_hashes`, recompute `event_commitment` from the transaction receipts, and verify `concatenated_counts` encodes the correct counts. Reject the block if any commitment mismatches.

2. **Verify block signatures in the P2P sync path.** `parse_data_for_block` should call `verify_block_signature` against the known sequencer public key before accepting a `SignedBlockHeader`.

3. **Extend the committer's hash check.** The `verify_state_diff_hash` flag should be accompanied by analogous flags (or always-on checks) for `transaction_commitment`, `event_commitment`, and `receipt_commitment`.

### Proof of Concept

```
1. Attacker connects as a P2P peer to a syncing sequencer node.

2. Attacker constructs a SignedBlockHeader for block N:
   - state_diff_commitment  = calculate_state_diff_hash(actual_state_diff)  // correct
   - transaction_commitment = Felt::ZERO                                     // forged
   - event_commitment       = Felt::ZERO                                     // forged
   - receipt_commitment     = Felt::ZERO                                     // forged
   - concatenated_counts    = concat_counts(n_txs, n_events, sd_len, mode)  // correct
   - signatures             = [BlockSignature::default()]                    // not verified

3. P2P sync receives the header; parse_data_for_block checks only
   signatures.len() == 1 → accepted; stored in local storage.

4. State sync client provides SyncBlock (with the forged commitments) to
   the batcher via try_sync → add_sync_block.

5. add_sync_block stores PartialBlockHashComponents { header_commitments: forged, ... }
   in storage (crates/apollo_batcher/src/batcher.rs:688-697).

6. Committer receives state_diff_commitment = correct value →
   verify_state_diff_hash check passes (crates/apollo_committer/src/committer.rs:167-176).

7. finalize_commitment_output reads the stored PartialBlockHashComponents and
   calls calculate_block_hash with transaction_commitment = Felt::ZERO
   → wrong_block_hash ≠ correct_block_hash.

8. wrong_block_hash is written to storage via set_global_root_and_block_hash.

9. starknet_getBlockWithTxHashes(block_id=N) returns wrong_block_hash.
   get_block_hash syscall for block N returns wrong_block_hash to any contract.
   Block N+1's block hash is computed with previous_block_hash = wrong_block_hash.
```

pub(crate) fn get_block_hashes<S: StateReader>(
    hint_processor: &mut SnosHintProcessor<'_, S>,
    mut ctx: HintContext<'_>,
) -> OsHintResult {
    let os_input = &hint_processor.get_current_execution_helper()?.os_block_input;
    let block_info = &os_input.block_info;
    let gas_prices = &block_info.gas_prices;
    let commitments = &os_input.block_hash_commitments;

    ctx.insert_value(Ids::PreviousBlockHash, os_input.prev_block_hash.0)?;

    let header_commitments_ptr = ctx.vm.add_memory_segment();
    ctx.insert_values_to_fields(
        header_commitments_ptr,
        CairoStruct::BlockHeaderCommitments,
        &[
            ("transaction_commitment", commitments.transaction_commitment.0.into()),
            ("event_commitment", commitments.event_commitment.0.into()),
            ("receipt_commitment", commitments.receipt_commitment.0.into()),
            ("state_diff_commitment", commitments.state_diff_commitment.0.0.into()),
            ("packed_lengths", commitments.concatenated_counts.into()),
        ],
    )?;
    ctx.insert_value(Ids::HeaderCommitments, header_commitments_ptr)?;

    // The on-chain block hash commits to the exact ASCII encoding of the version string, which
    // may be newer than the latest known `StarknetVersion` variant.
    let starknet_version_string = os_input
        .starknet_version_override
        .clone()
        .unwrap_or_else(|| block_info.starknet_version.to_string());
    ctx.insert_value(Ids::StarknetVersion, ascii_as_felt(&starknet_version_string)?)?;

    let [gas_prices_hash]: [Felt; 1] = gas_prices_to_hash(
        &gas_prices.l1_gas_price_per_token(),
        &gas_prices.l1_data_gas_price_per_token(),
        &gas_prices.l2_gas_price_per_token(),
        &block_info.starknet_version.try_into()?,
    )
    .try_into()
    .map_err(|_| {
        OsHintError::UnsupportedStarknetVersionForBlockHash(block_info.starknet_version)
    })?;

    ctx.insert_value(Ids::GasPricesHash, gas_prices_hash)?;

    Ok(())
}
