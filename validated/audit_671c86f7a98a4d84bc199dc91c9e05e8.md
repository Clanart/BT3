### Title
Integer Wrapping in Relative Timelock Arithmetic Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Enforcement — (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary

When `check_time_locks` is called with `nowrap=false` (the legacy code path still exposed through the Python binding), relative timelock conditions whose argument is near the type maximum (`u32::MAX` for height, `u64::MAX` for seconds) wrap around to a small value. The comparison then trivially passes, allowing a coin to be spent immediately even though the puzzle asserts a multi-billion-block or multi-billion-second wait.

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` accepts a `nowrap: bool` parameter that selects between two arithmetic paths for relative timelock enforcement.

**`nowrap=true` (safe path):**
```rust
if prev_transaction_block_height
    < unspent.confirmed_block_index.saturating_add(height_relative)
```

**`nowrap=false` (legacy/wrapping path):**
```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
``` [1](#0-0) 

The same pattern applies to `seconds_relative`: [2](#0-1) 

The condition parser in `conditions.rs` accepts any value that fits in a `u32` (for height) or `u64` (for seconds) as a valid `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` argument: [3](#0-2) 

So an attacker can craft a puzzle that emits `ASSERT_HEIGHT_RELATIVE(0xffff_fff6)`. With a coin confirmed at height 10 and the current height at 15:

```
10.wrapping_add(0xffff_fff6) = 0  (overflows to 0)
15 < 0  →  false  →  Ok(())   ← timelock bypassed
```

The project's own test suite documents this behavior explicitly: [4](#0-3) 

The Python binding `py_check_time_locks` forwards the `nowrap` flag directly to the caller without any default enforcement: [5](#0-4) 

The binding is registered in the Python wheel module: [6](#0-5) 

### Impact Explanation

If the Python full node (chia-blockchain) calls `check_time_locks` with `nowrap=False` — which was the only available behavior before the `nowrap` flag was introduced — any coin whose puzzle emits a near-maximum relative timelock value can be spent at any block height or timestamp. This is a **timelock validation bypass enabling unauthorized spend acceptance**, matching the High impact category: *"timelock or coin-id validation bypass enables unauthorized spend acceptance."*

### Likelihood Explanation

The `nowrap=False` path is the original legacy behavior. The `nowrap` parameter was added to chia_rs to fix the wrapping issue, but the fix is only effective if the Python full node is updated to pass `nowrap=True`. If any production caller still passes `nowrap=False`, the bypass is fully exploitable by any unprivileged user who can submit a spend bundle. The crafted condition value (`0xffff_fff6` for height, `0xffff_ffff_ffff_d8f0` for seconds relative to a typical coin timestamp) is a valid CLVM integer and passes the condition parser without error.

### Recommendation

1. Remove the `nowrap=false` branch entirely from `check_time_locks`. The wrapping behavior has no legitimate consensus use case and only exists as a backward-compatibility artifact.
2. If backward compatibility must be preserved, rename the parameter to make the safe path the default (e.g., `use_saturating: bool = true`) and audit all call sites in chia-blockchain to confirm `nowrap=True` is always passed.
3. Add a compile-time or runtime assertion that rejects `nowrap=false` in production builds.

### Proof of Concept

Craft a CLVM puzzle that outputs:
```
(ASSERT_HEIGHT_RELATIVE . 0xffff_fff6)
```

This is a valid 4-byte big-endian unsigned integer (`4294967286`), accepted by the condition parser as `AssertHeightRelative(4294967286_u32)`.

With a coin confirmed at block index `B` and the current block at `B + 5`:

```
B.wrapping_add(4294967286) = B - 10   (mod 2^32)
(B + 5) < (B - 10)  →  false  →  check passes
```

The spend is accepted immediately, despite the puzzle asserting a ~4.3-billion-block wait. The same technique applies to `ASSERT_SECONDS_RELATIVE` using a `u64`-near-max value, bypassing second-based timelocks on any coin regardless of its age. [7](#0-6) [8](#0-7)
