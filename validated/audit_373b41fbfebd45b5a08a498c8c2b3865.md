### Title
EIP-7702 Authorization List Omitted from Auto-Gas Estimation in `SetTxDefaults` — Causes Valid Transactions to Fail with Intrinsic Gas Error - (File: rpc/backend/call_tx.go)

### Summary

`SetTxDefaults` in `rpc/backend/call_tx.go` builds a stripped `callArgs` for gas auto-estimation that omits the `AuthorizationList` field present in the original EIP-7702 transaction. Because each authorization tuple contributes `TxAuthTupleGas` (25,000 gas) to intrinsic gas, the returned estimate is systematically too low. Any EIP-7702 transaction submitted via `eth_sendTransaction` or `eth_fillTransaction` without an explicit gas limit will be assigned this underestimated value and subsequently rejected at the ante handler's intrinsic gas check.

### Finding Description

When `args.Gas == nil`, `SetTxDefaults` constructs a `callArgs` struct to call `EstimateGas`:

```go
callArgs := evmtypes.TransactionArgs{
    From:                 args.From,
    To:                   args.To,
    Gas:                  args.Gas,
    GasPrice:             args.GasPrice,
    MaxFeePerGas:         args.MaxFeePerGas,
    MaxPriorityFeePerGas: args.MaxPriorityFeePerGas,
    Value:                args.Value,
    Data:                 input,
    AccessList:           args.AccessList,
    ChainID:              args.ChainID,
    Nonce:                args.Nonce,
    // AuthorizationList is NOT copied here
}
```

`AuthorizationList` is silently dropped. [1](#0-0) 

`EstimateGas` then calls `args.ToMessage(gasCap, cfg.BaseFee)`, which sets `SetCodeAuthorizations: args.AuthorizationList` on the `core.Message`. Because `callArgs.AuthorizationList` is `nil`, the message has no authorization tuples. [2](#0-1) 

The intrinsic gas calculation inside `ApplyMessageWithConfig` calls `GetEthIntrinsicGas`, which passes `msg.SetCodeAuthorizations` to `core.IntrinsicGas`. With no authorizations, the per-tuple cost (`TxAuthTupleGas = 25,000` per entry) is not included in the estimate. [3](#0-2) 

The estimated gas is then written back to `args.Gas` and the full transaction (which still carries the original `AuthorizationList`) is submitted. [4](#0-3) 

At execution time, `VerifyFee` recomputes intrinsic gas **with** `tx.SetCodeAuthorizations()` included: [5](#0-4) 

```go
intrinsicGas, err := core.IntrinsicGas(
    tx.Data(),
    accessList,
    tx.SetCodeAuthorizations(),   // ← present in real tx, absent during estimation
    isContractCreation,
    ...
)
```

During `CheckTx` this triggers: [6](#0-5) 

```go
if isCheckTx && gasLimit < intrinsicGas {
    return nil, errorsmod.Wrapf(errortypes.ErrOutOfGas,
        "gas limit too low: %d (gas limit) < %d (intrinsic gas)", ...)
}
```

And during `DeliverTx`, `ApplyMessageWithConfig` enforces the same check: [7](#0-6) 

```go
if leftoverGas < intrinsicGas {
    return nil, errorsmod.Wrap(core.ErrIntrinsicGas, "apply message")
}
```

The shortfall is exactly `N × TxAuthTupleGas` (N = number of authorization tuples, each costing 25,000 gas). A transaction with even a single authorization tuple will be underestimated by 25,000 gas.

### Impact Explanation

Every EIP-7702 (`type 4`) transaction submitted through `eth_sendTransaction` or `eth_fillTransaction` without an explicit `gas` field will receive an underestimated gas limit and be rejected at the ante handler with an intrinsic gas error. This is a gas accounting bug in the ante handler path that causes valid user transactions to be mis-accounted and fail. The user's transaction is silently dropped; no funds are deducted, but the intended state change never occurs. This matches the allowed High impact: *"ante handler … bug that permits … valid user funds/fees to be mis-accounted."*

### Likelihood Explanation

EIP-7702 is a newly supported transaction type in this codebase. The standard JSON-RPC client workflow (MetaMask, web3.py, ethers.js) omits `gas` and relies on `eth_sendTransaction` to auto-fill it via `SetTxDefaults`. Any user or dApp following this standard flow for EIP-7702 transactions will hit the bug deterministically. No special privileges or adversarial conditions are required — a normal unprivileged user submitting a type-4 transaction without specifying gas is sufficient.

### Recommendation

Copy `AuthorizationList` into `callArgs` inside `SetTxDefaults`, mirroring how `AccessList` is already forwarded:

```go
callArgs := evmtypes.TransactionArgs{
    From:                 args.From,
    To:                   args.To,
    Gas:                  args.Gas,
    GasPrice:             args.GasPrice,
    MaxFeePerGas:         args.MaxFeePerGas,
    MaxPriorityFeePerGas: args.MaxPriorityFeePerGas,
    Value:                args.Value,
    Data:                 input,
    AccessList:           args.AccessList,
    AuthorizationList:    args.AuthorizationList, // ← add this
    ChainID:              args.ChainID,
    Nonce:                args.Nonce,
}
```

### Proof of Concept

1. Deploy any contract `C` on the chain.
2. Create an EIP-7702 authorization tuple signed by account `A` delegating to `C`.
3. Call `eth_sendTransaction` with `type: 4`, `authorizationList: [auth]`, and **no `gas` field**.
4. `SetTxDefaults` calls `EstimateGas` with `callArgs` that has `AuthorizationList: nil`.
5. `EstimateGas` returns, e.g., `21,000` (bare transfer cost, no authorization overhead).
6. `args.Gas` is set to `21,000`.
7. The transaction is broadcast with `gasLimit = 21,000` but `authorizationList` containing one tuple.
8. `VerifyFee` / `ApplyMessageWithConfig` computes `intrinsicGas = 21,000 + 25,000 = 46,000`.
9. `gasLimit (21,000) < intrinsicGas (46,000)` → transaction rejected with `ErrOutOfGas` / `ErrIntrinsicGas`.
10. The user's EIP-7702 delegation never takes effect. [8](#0-7) [9](#0-8) [3](#0-2) [10](#0-9) [11](#0-10)
