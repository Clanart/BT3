### Title
Unlimited-gas ETH transfer to user-controlled recipient in `finTransfer` enables permanent unclaimable settlement of cross-chain funds - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

In `OmniBridge.finTransfer`, when the bridged token is native ETH (`payload.tokenAddress == address(0)`), the contract sends ETH to `payload.recipient` using a bare `.call{value: ...}("")` with **no gas limit**. If the recipient is a contract whose fallback function reverts, the entire `finTransfer` transaction reverts. Because the nonce-consumed flag (`completedTransfers[payload.destinationNonce] = true`) is set before the ETH transfer and is rolled back on revert, the nonce is never consumed. The relayer can retry indefinitely but will always fail, leaving the originating NEAR-side funds permanently locked with no on-chain recovery path.

---

### Finding Description

`OmniBridge.finTransfer` follows this sequence:

1. **Mark nonce used** (line 287):
   ```solidity
   completedTransfers[payload.destinationNonce] = true;
   ```
2. **Send ETH to recipient with unlimited gas** (lines 317–322):
   ```solidity
   if (payload.tokenAddress == address(0)) {
       (bool success, ) = payload.recipient.call{value: payload.amount}("");
       if (!success) revert FailedToSendEther();
   }
   ```

If `payload.recipient` is a contract whose `receive()` / `fallback()` reverts (or consumes all forwarded gas), the `revert FailedToSendEther()` at line 322 rolls back the entire transaction, including the nonce flag set at line 287. The nonce is therefore **never consumed**.

Because there is no admin escape hatch to manually mark a nonce as completed or to redirect the ETH to an alternative address, the relayer is stuck in an infinite retry loop that always reverts. The NEAR-side locked funds have no corresponding EVM settlement path.

The same pattern applies to the ERC-1155 branch (lines 323–330), where `IERC1155.safeTransferFrom` triggers `onERC1155Received` on the recipient contract with unlimited gas, giving the recipient another revert vector. [1](#0-0) 

---

### Impact Explanation

**Permanent freezing / unclaimable settlement of user funds** (matches allowed High impact).

- The NEAR-side bridge locks the user's tokens at transfer initiation.
- If the EVM `finTransfer` can never succeed, those tokens are irrecoverably locked on NEAR with no refund or cancel mechanism visible in the EVM contract.
- There is no admin function in `OmniBridge` to force-complete a nonce or reroute the ETH, so the freeze is permanent from the EVM side. [2](#0-1) 

---

### Likelihood Explanation

**Medium.** The `payload.recipient` is an EVM address embedded in the MPC-signed payload originating from NEAR. An attacker who controls an EVM contract address (e.g., via CREATE2) can:

1. Advertise the address as a legitimate recipient.
2. Have a victim initiate a NEAR→EVM transfer to that address.
3. Deploy (or upgrade) the contract at that address to a reverting fallback **before** the relayer calls `finTransfer`.
4. Every subsequent `finTransfer` attempt reverts; the victim's NEAR funds are permanently frozen.

Alternatively, a user who wishes to grief the relayer economically (unlimited gas forwarded) can point the recipient at a gas-consuming contract. While pure gas griefing is out of scope, the permanent-freeze consequence is in scope and is the primary impact here.

---

### Recommendation

1. **Cap the gas forwarded** to the recipient to a fixed stipend (e.g., `2300` gas or a configurable limit), preventing fallback abuse:
   ```solidity
   (bool success, ) = payload.recipient.call{value: payload.amount, gas: 2300}("");
   ```
2. **Use a pull-payment pattern**: credit the recipient's claimable balance in storage and let them withdraw separately, so a reverting fallback cannot block settlement.
3. For the ERC-1155 branch, consider wrapping `safeTransferFrom` in a try/catch and falling back to a claimable-balance record on failure.

---

### Proof of Concept

```
1. Attacker deploys MaliciousRecipient at address X:
       receive() external payable { revert(); }

2. Victim initiates NEAR→EVM transfer of 1 ETH, recipient = X.
   NEAR bridge locks 1 ETH equivalent on NEAR side.

3. MPC signs the TransferMessagePayload with recipient = X.

4. Relayer calls OmniBridge.finTransfer(sig, payload):
   - Line 287: completedTransfers[nonce] = true  ← written
   - Line 319: X.call{value: 1 ether}("")        ← MaliciousRecipient.receive() reverts
   - Line 322: revert FailedToSendEther()         ← entire tx reverts
   - completedTransfers[nonce] rolled back to false

5. Relayer retries → same result every time.

6. Victim's 1 ETH equivalent is permanently locked on NEAR.
   No admin function exists to force-complete the nonce or recover funds.
``` [3](#0-2)
