### Title
Wrong Epoch's Shard Layout Used to Index `prev_block.chunks()` in `get_state_response_part()` at Resharding Boundary — (File: `chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`get_state_response_part()` resolves the `shard_index` for `prev_block.chunks()` using the **sync block's (new epoch's) shard layout**, while `prev_block` belongs to the **previous epoch** with a different layout. At a resharding boundary this produces the wrong `prev_state_root`, causing every generated state part to fail the receiver's validation and permanently stalling state sync for any shard whose index shifted across the boundary.

---

### Finding Description

`get_state_response_part()` in `chain/chain/src/state_sync/adapter.rs` serves state parts to syncing nodes. Its logic is:

```
epoch_id   = sync_hash block's epoch          // NEW epoch
shard_layout = get_shard_layout(epoch_id)     // NEW layout
prev_block   = block before sync_hash         // LAST block of OLD epoch
shard_index  = shard_layout.get_shard_index(shard_id)   // index in NEW layout
state_root   = prev_block.chunks().get(shard_index)      // reads OLD block with NEW index
``` [1](#0-0) 

The sibling function `compute_state_response_header()` performs the same lookup correctly: it fetches `prev_shard_layout` from `prev_block`'s epoch and uses `prev_shard_layout.get_shard_index(shard_id)` to index into `prev_block.chunks()`. [2](#0-1) 

In `ShardLayoutV2`/`V3`, `shard_id` and `shard_index` are **not** the same: [3](#0-2) 

When a shard is split via `ShardLayoutV3::derive_impl`, the two children are spliced in at the parent's position, shifting every subsequent shard's index by +1: [4](#0-3) 

**Concrete example.** Old layout: shards `[id=0, id=1, id=2]` at indices `[0, 1, 2]`. Shard `id=0` is split; new layout: `[id=3, id=4, id=1, id=2]` at indices `[0, 1, 2, 3]`.

| Shard | Old index | New index |
|-------|-----------|-----------|
| id=1  | 1         | 2         |
| id=2  | 2         | 3         |

`get_state_response_part()` for `shard_id=1`:
- `shard_index = new_layout.get_shard_index(1)` → **2**
- `prev_block.chunks().get(2)` → chunk for **shard id=2** in the old block
- `state_root` = shard 2's `prev_state_root` — **wrong shard's root**

The state part is then generated against shard 2's trie root and served as if it were shard 1's state.

---

### Impact Explanation

The receiver calls `set_state_part()`, which validates the incoming part against the state root committed in the `ShardStateSyncResponseHeader` (computed by `compute_state_response_header()`, which uses the correct `prev_shard_layout`): [5](#0-4) 

Because the part was generated from the wrong state root, `validate_state_part` returns `Invalid` and the part is rejected. Every retry produces the same wrong part. State sync for all shards whose index shifted across the resharding boundary is permanently stalled. Validators that must state-sync to track a new shard after resharding cannot complete the process and cannot produce chunks in the new epoch.

---

### Likelihood Explanation

The bug is triggered whenever all three conditions hold simultaneously:
1. A resharding event occurs (ShardLayoutV2/V3 with dynamic resharding enabled).
2. State sync is triggered for the new epoch (any validator joining a shard it did not track in the previous epoch).
3. The requested `shard_id` has a different `shard_index` in the new layout than in the old layout (any shard positioned after the split point).

Condition 3 is guaranteed for at least one shard in every split event. With dynamic resharding active on mainnet, this becomes reachable on every resharding epoch boundary.

---

### Recommendation

Replace the `shard_layout` used to index into `prev_block.chunks()` with the **previous epoch's** shard layout, mirroring the correct logic already present in `compute_state_response_header()`:

```rust
// After fetching prev_block:
let prev_epoch_id = prev_block.header().epoch_id();
let prev_shard_layout = self.epoch_manager.get_shard_layout(&prev_epoch_id)?;
let shard_index = prev_shard_layout.get_shard_index(shard_id)?;
let state_root = prev_block
    .chunks()
    .get(shard_index)
    .ok_or(Error::InvalidShardId(shard_id))?
    .prev_state_root();
```

The `shard_ids.contains(&shard_id)` guard should remain keyed on the new epoch's layout (to validate the requested shard exists in the epoch being synced), but the chunk-array lookup must use the old epoch's layout.

---

### Proof of Concept

1. Configure a testnet with ShardLayoutV3 dynamic resharding enabled.
2. Trigger a resharding event (split one shard). Let the new epoch start.
3. Bring up a new validator node that needs to state-sync shard `id=X`, where `X` is any shard positioned after the split point (its index shifted by +1 in the new layout).
4. The node sends `StateRequestPart { shard_id: X, sync_hash, part_id: 0 }`.
5. The serving node executes `get_state_response_part()`: it computes `shard_index` from the new layout (e.g., 3), reads `prev_block.chunks().get(3)` (shard Y's chunk, not X's), and returns a state part built from shard Y's `prev_state_root`.
6. The receiving node's `set_state_part()` calls `validate_state_part(shard_id=X, state_root=correct_X_root, part)` — the part was built from Y's root, so validation fails with `StatePartValidationResult::Invalid`.
7. All retries produce the same wrong part; state sync never completes. [6](#0-5) [7](#0-6)

    pub fn get_shard_index(&self, shard_id: ShardId) -> Result<ShardIndex, ShardLayoutError> {
        match self {
            // In V0 & V1 the shard id and shard index are the same.
            Self::V0(_) | Self::V1(_) => Ok(shard_id.into()),
            // In V2 & V3 the shard id and shard index are **not** the same.
            Self::V2(v2) => v2.get_shard_index(shard_id),
            Self::V3(v3) => v3.get_shard_index(shard_id),
        }
    }
