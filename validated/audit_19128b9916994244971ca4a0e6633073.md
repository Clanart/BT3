Audit Report

## Title
Router ETH Balance Used as WETH Payment Source Regardless of `payer` Identity, Enabling Theft of Stranded ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` uses `address(this).balance` as the ETH source for WETH payments whenever the router holds native ETH, completely bypassing the `payer` argument. Any ETH stranded in the router from a prior user's overpayment (e.g., via `exactOutputSingle` without a subsequent `refundETH`) can be consumed by any attacker who calls a WETH-input swap, receiving full swap output without spending any of their own ETH or WETH.

## Finding Description

In `PeripheryPayments.pay()`, the WETH branch at lines 73–84 checks `address(this).balance` and, when `nativeBalance >= value`, wraps the router's own ETH and transfers it to the pool — the `payer` parameter is never consulted in this branch:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();   // uses router's ETH, not payer's
        IERC20(WETH).safeTransfer(recipient, value);
    }
    ...
}
``` [1](#0-0) 

The transient callback context always stores `msg.sender` as the payer: [2](#0-1) 

And `_justPayCallback` passes that stored payer to `pay()`: [3](#0-2) 

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` on any `payable` entry point (`exactInputSingle`, `exactOutputSingle`, `multicall`, etc.): [4](#0-3) 

The test suite confirms `refundETH` is an optional follow-up step, not enforced by the router itself: [5](#0-4) 

The same flaw exists in `MetricOmmPoolLiquidityAdder.metricOmmModifyLiquidityCallback()` when `token0 == WETH`, as it calls `pay(token0, payer, msg.sender, amount0Delta)` through the same vulnerable `pay()` function: [6](#0-5) 

## Impact Explanation

Direct loss of user ETH principal. A victim who sends excess ETH via `exactOutputSingle{value: X}` and omits `refundETH` leaves ETH stranded in the router. An attacker calling `exactInputSingle` with `tokenIn = WETH` and `amountIn ≤ stranded balance` receives full swap output without spending any ETH or WETH of their own. The victim's stranded ETH is permanently consumed and transferred to the pool as the attacker's payment. This constitutes a direct, unrecoverable loss of user principal above Sherlock thresholds.

## Likelihood Explanation

Medium. The `exactOutputSingle` pattern with ETH overpayment is the primary intended use of native ETH input, as evidenced by the test suite. Omitting `refundETH` is a common user mistake (it is optional, not enforced). An attacker only needs to monitor the router's ETH balance on-chain and submit a WETH-input swap in the next block. No special privileges, approvals, or setup are required — any unprivileged caller can execute this.

## Recommendation

Track the ETH contributed by the current call in transient storage (store `msg.value` at entry, decrement as consumed in `pay()`). Replace `address(this).balance` in `pay()` with this per-call ETH budget so only the ETH the current caller sent can fund their swap. Alternatively, enforce that `refundETH` is always called in the same `multicall` by checking `address(this).balance == 0` at the end of `multicall`, or revert in `pay()` if `nativeBalance > 0` but the ETH was not contributed by the current call.

## Proof of Concept

```
1. Victim calls exactOutputSingle{value: 1_000}(amountOut=X, tokenIn=WETH, amountInMaximum=1_000)
   - Actual input required: 600
   - pay(WETH, victim, pool, 600): nativeBalance=1000 >= 600 → wraps 600 ETH, sends to pool
   - Router ETH balance after swap: 400 (stranded; victim omitted refundETH)

2. Attacker calls exactInputSingle(amountIn=400, tokenIn=WETH) — zero msg.value, zero WETH approval
   - _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, attacker, WETH)
   - Pool executes swap, calls metricOmmSwapCallback → _justPayCallback
   - pay(WETH, attacker, pool, 400): nativeBalance=400 >= 400 → wraps router's 400 ETH, sends to pool
   - Attacker receives swap output tokens; payer=attacker is never consulted

Result: Victim loses 400 ETH permanently. Attacker receives output tokens for free.
```
