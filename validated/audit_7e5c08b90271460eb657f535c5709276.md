### Title
ETH Dust Permanently Trapped in `finTransfer` for Non-ETH Token Transfers — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`finTransfer` is marked `payable` but for non-ETH token transfers (ERC20, ERC1155, bridge tokens), `msg.value` is never consumed or refunded. Any ETH sent with such a call is permanently locked in the contract with no recovery mechanism.

### Finding Description
`finTransfer` in `OmniBridge.sol` is declared `external payable`: [1](#0-0) 

Inside the function, ETH (`msg.value`) is only forwarded to the recipient when `payload.tokenAddress == address(0)` (native ETH transfers): [2](#0-1) 

For all other token types — ERC20, ERC1155, bridge tokens, and custom minter tokens — the function transfers tokens but never touches `msg.value`. The `finTransferExtension` hook in the base `OmniBridge` contract is a no-op: [3](#0-2) 

The contract has a bare `receive()` fallback and no ETH withdrawal or sweep function: [4](#0-3) 

Any ETH sent alongside an ERC20 `finTransfer` call is irrecoverably locked. In the `OmniBridgeWormhole` variant, `finTransferExtension` forwards the full `msg.value` to Wormhole's `publishMessage`: [5](#0-4) 

Wormhole only requires `messageFee()`. Any `msg.value` exceeding `messageFee()` is consumed by Wormhole and not refunded, compounding the dust loss in the Wormhole deployment.

### Impact Explanation
ETH sent with `finTransfer` calls for ERC20/ERC1155/bridge-token transfers is permanently locked in the contract. There is no admin sweep, no `refundETH`, and no withdrawal path. This constitutes an irrecoverable lock of user funds in a bridge vault flow — matching the Critical impact class: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation
`finTransfer` is callable by any address holding a valid MPC-signed payload. Relayers and users interacting with the bridge may send ETH alongside ERC20 finalization calls (e.g., expecting to pay a Wormhole fee or mistakenly including ETH). The `payable` modifier provides no guard and no revert for unexpected ETH, making accidental trapping realistic across normal bridge operations.

### Recommendation
Add an ETH refund at the end of `finTransfer` for cases where `msg.value` exceeds the amount actually consumed (i.e., the Wormhole fee or zero for non-ETH transfers). A minimal fix:

```solidity
// At the end of finTransfer, after finTransferExtension:
uint256 consumed = (payload.tokenAddress == address(0)) ? payload.amount : 0;
// For Wormhole variant, consumed += _wormhole.messageFee();
uint256 dust = msg.value - consumed;
if (dust > 0) {
    (bool ok, ) = msg.sender.call{value: dust}("");
    require(ok, "ETH refund failed");
}
```

Similarly audit `deployToken`, `logMetadata`, and `logMetadata1155`, which are also `payable` with empty extension hooks in the base contract.

### Proof of Concept
1. A relayer calls `finTransfer` for a USDC transfer (ERC20), accidentally including `0.01 ETH` as `msg.value`.
2. The function verifies the MPC signature, calls `IERC20(usdc).safeTransfer(recipient, amount)`, then calls the empty `finTransferExtension`.
3. `msg.value` (0.01 ETH) is never forwarded or refunded — it remains in the contract balance.
4. No admin function exists to recover it. The ETH is permanently locked. [6](#0-5)
