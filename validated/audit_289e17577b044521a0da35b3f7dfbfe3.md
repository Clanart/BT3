### Title
Missing Array Length Check in `BlsCache::py_aggregate_verify` Python Binding Silently Truncates Verification — (File: `crates/chia-bls/src/bls_cache.rs`)

### Summary
The Python binding `BLSCache.aggregate_verify(pks, msgs, sig)` in `crates/chia-bls/src/bls_cache.rs` accepts two separate lists (`pks` and `msgs`) but never checks that they have equal length before zipping them. Rust's `Iterator::zip` silently truncates to the shorter iterator. If `len(pks) > len(msgs)`, the trailing public keys are silently dropped from verification; the aggregate signature is then checked against only `len(msgs)` pairs instead of `len(pks)`. This is a direct analog of the reported missing-length-check vulnerability class.

### Finding Description
`BlsCache::py_aggregate_verify` collects `pks` and `msgs` into two independent `Vec`s and then calls `pks.into_iter().zip(msgs)`: [1](#0-0) 

There is no guard equivalent to the one present in `AugSchemeMPL::aggregate_verify` in `wheel/src/api.rs`: [2](#0-1) 

`AugSchemeMPL::aggregate_verify` explicitly rejects mismatched lengths with a `PyRuntimeError`. `BlsCache::py_aggregate_verify` does not. The same asymmetry exists in `BlsCache::py_evict`: [3](#0-2) 

The Python-facing type stubs expose both functions with separate `list[G1Element]` / `list[bytes]` parameters, making the silent truncation invisible to callers: [4](#0-3) 

### Impact Explanation
When `len(pks) > len(msgs)`, `zip` silently drops the trailing `pks` entries. The aggregate signature is verified against only `len(msgs)` (pk, msg) pairs. A spend bundle whose aggregate signature is valid for a strict subset of the declared signers will pass `BLSCache.aggregate_verify` if the caller inadvertently passes a shorter `msgs` list. This constitutes a signature verification bypass at the Python binding boundary: keys that were supposed to authorize a spend are never checked, enabling unauthorized spend acceptance.

The inconsistency between `AugSchemeMPL.aggregate_verify` (raises on mismatch) and `BLSCache.aggregate_verify` (silently truncates) is a cross-language/cross-function API disagreement in consensus-critical signature verification code.

### Likelihood Explanation
The chia-blockchain Python layer constructs `pks` and `msgs` lists from parsed spend bundles and passes them to `BLSCache.aggregate_verify`. Any code path that builds these two lists independently (rather than as paired tuples) and makes an off-by-one or filtering error will silently produce a truncated verification. Because the API accepts two separate lists rather than a single list of pairs, this class of caller mistake is structurally encouraged. The contrast with `AugSchemeMPL.aggregate_verify` means callers who switch between the two APIs face different silent-vs-loud failure modes.

### Recommendation
Add a length equality check in `BlsCache::py_aggregate_verify` and `BlsCache::py_evict`, mirroring the guard already present in `AugSchemeMPL::aggregate_verify`:

```rust
if pks.len() != msgs.len() {
    return Err(PyValueError::new_err(
        "aggregate_verify expects the same number of public keys as messages",
    ));
}
```

This should be inserted immediately after both `Vec`s are collected, before the `zip` call, in both `py_aggregate_verify` and `py_evict`.

### Proof of Concept
```python
from chia_rs import BLSCache, G1Element, G2Element, AugSchemeMPL, PrivateKey

seed = bytes(range(32))
sk = AugSchemeMPL.key_gen(seed)
pk = sk.get_g1()
msg1 = b"authorized_message"
msg2 = b"unauthorized_message"

# Sign only msg1
sig = AugSchemeMPL.sign(sk, msg1)

cache = BLSCache()

# Pass two pks but only one msg — the second pk is silently dropped.
# Verification succeeds even though pk was never checked against msg2.
result = cache.aggregate_verify([pk, pk], [msg1], sig)
# result == True  (only (pk, msg1) was verified; second pk silently ignored)

# AugSchemeMPL raises an error for the same input:
try:
    AugSchemeMPL.aggregate_verify([pk, pk], [msg1], sig)
except RuntimeError as e:
    print(e)  # "aggregate_verify expects the same number of public keys as messages"
```
