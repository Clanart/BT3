### Title
Deflationary Token Transfer Fee Not Accounted in `initTransfer`, Causing Bridge Undercollateralization - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer()` calls `safeTransferFrom(msg.sender, address(this), amount)` for native ERC20 tokens but never verifies the actual amount received. The original caller-supplied `amount` is then encoded verbatim into the cross-chain message. If the token charges a transfer fee (deflationary token), the bridge holds fewer tokens than the amount committed on the destination chain, permanently breaking collateralization.

---

### Finding Description

In `initTransfer()`, when the token is a plain ERC20 (not a bridge token and not a custom minter), the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
``` [1](#0-0) 

No balance snapshot is taken before or after this call. The function then passes the original `amount` parameter directly to `initTransferExtension()` and the `InitTransfer` event:

```solidity
initTransferExtension(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // <-- unverified, may exceed actual received
    fee,
    ...
);
emit BridgeTypes.InitTransfer(..., amount, ...);
``` [2](#0-1) 

In the Wormhole variant, `initTransferExtension()` encodes this unverified `amount` into the cross-chain payload sent to NEAR:

```solidity
Borsh.encodeUint128(amount),   // amount committed to destination chain
``` [3](#0-2) 

The NEAR bridge receives this message and processes a `fin_transfer` releasing or minting `amount` tokens to the recipient — an amount larger than what the EVM bridge actually holds.

---

### Impact Explanation

For every `initTransfer` call with a deflationary token (one that deducts a fee on `transferFrom`), the EVM bridge records a commitment of `amount` but only holds `amount - fee_taken`. Over time, the bridge's ERC20 balance becomes insufficient to honor all return transfers from NEAR. When users bridge back (NEAR → EVM), the `finTransfer` path calls:

```solidity
IERC20(payload.tokenAddress).safeTransfer(payload.recipient, payload.amount);
``` [4](#0-3) 

This will revert for later users once the bridge's actual balance is exhausted, permanently locking their funds on NEAR with no recourse. This matches the allowed impact: **balance/accounting corruption that breaks bridge collateralization**, and **permanent freezing/irrecoverable lock of user funds**.

---

### Likelihood Explanation

- Any ERC20 token with a transfer fee (deflationary tokens, tokens with configurable fees such as USDT's dormant fee mechanism, or any custom ERC20 with `transferFrom` hooks) triggers this path.
- The entry point is fully unprivileged: any user can call `initTransfer()` with any whitelisted token.
- The bridge does not whitelist tokens — `ethToNearToken` mapping is populated by admin, but once a fee-on-transfer token is registered, every user deposit silently undercollateralizes the bridge.
- The discrepancy accumulates silently; no on-chain check will catch it until a withdrawal fails.

---

### Recommendation

Record the bridge's token balance before and after the `safeTransferFrom` call and use the delta as the canonical `amount` for the cross-chain message:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
// use actualReceived instead of amount going forward
```

Additionally, consider maintaining an explicit token whitelist that excludes fee-on-transfer and rebasing tokens, and document this restriction clearly.

---

### Proof of Concept

1. Deploy or use a deflationary ERC20 token `FeeToken` that deducts 1% on every `transferFrom`.
2. Register `FeeToken` in the bridge (admin step, one-time).
3. Alice calls `initTransfer(FeeToken, 1000, 0, 0, "alice.near", "")`.
4. Bridge receives `990` tokens (1% fee taken by the token contract).
5. The Wormhole message encodes `amount = 1000`.
6. NEAR bridge processes `fin_transfer` and releases `1000` tokens to Alice on NEAR.
7. Repeat 100 times: bridge holds `99,000` tokens but has committed `100,000` on NEAR.
8. The 100th user to bridge back from NEAR → EVM calls `finTransfer` on EVM; `safeTransfer` reverts because the bridge balance is insufficient.
9. That user's funds are permanently locked on NEAR with no EVM-side release possible. [5](#0-4) [6](#0-5)

    function initTransfer(
        address tokenAddress,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message
    ) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;
        } else {
            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );
                ICustomMinter(customMinters[tokenAddress]).burn(
                    tokenAddress,
                    amount
                );
            } else if (isBridgeToken[tokenAddress]) {
                BridgeToken(tokenAddress).burn(msg.sender, amount);
            } else {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    address(this),
                    amount
                );
            }
        }

        initTransferExtension(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message,
            extensionValue
        );

        emit BridgeTypes.InitTransfer(
            msg.sender,
            tokenAddress,
            currentOriginNonce,
            amount,
            fee,
            nativeFee,
            recipient,
            message
        );
    }

    function initTransferExtension(
        address sender,
        address tokenAddress,
        uint64 originNonce,
        uint128 amount,
        uint128 fee,
        uint128 nativeFee,
        string calldata recipient,
        string calldata message,
        uint256 value
    ) internal override {
        bytes memory payload = bytes.concat(

            bytes1(omniBridgeChainId),

            bytes1(omniBridgeChainId),
            Borsh.encodeAddress(tokenAddress),
            Borsh.encodeUint64(originNonce),
            Borsh.encodeUint128(amount),
            Borsh.encodeUint128(fee),
            Borsh.encodeUint128(nativeFee),
            Borsh.encodeString(recipient),
            Borsh.encodeString(message)
        );

            wormholeNonce,
            payload,
            _consistencyLevel
