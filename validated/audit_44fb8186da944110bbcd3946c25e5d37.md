### Title
Unverified Peer-Supplied `state_diff_commitment` in `add_sync_block` Produces Wrong Block Hash - (File: crates/apollo_batcher/src/batcher.rs)

### Summary
When the batcher ingests a synced block via `add_sync_block`, the `state_diff_commitment` field inside the peer-supplied `block_header_commitments` is stored directly into `PartialBlockHashComponents` and forwarded to the committer without being verified against `calculate_state_diff_hash(&state_diff)`. The committer's only guard — the `verify_state_diff_hash` config flag — is disabled by default. A sync peer that supplies a mismatched `state_diff_commitment` therefore causes the sequencer to compute and persist a wrong block hash, corrupting the commitment chain.

### Finding Description
`add_sync_block` destructures the incoming `SyncBlock` into `state_diff` and `block_header_commitments`, then unconditionally promotes the peer-provided `header_commitments.state_diff_commitment` into two authoritative positions:

1. **`PartialBlockHashComponents` stored in batcher storage** — used later by `finalize_commitment_output` → `calculate_block_hash` to produce the final block hash.
2. **`optional_state_diff_commitment` forwarded to the committer** — stored as `ForestMetadataType::StateDiffHash` in the forest DB.

Neither site verifies the commitment against the actual `state_diff` received in the same message. [1](#0-0) 

The committer does contain a conditional check: [2](#0-1) 

However, the test that exercises this path must explicitly set the flag to `true`, confirming the production default is `false`: [3](#0-2) 

Even when the committer flag is `true`, it only guards the forest-DB metadata path; the `PartialBlockHashComponents` already written to batcher storage by `commit_proposal` retains the peer-supplied value and is read independently by `finalize_commitment_output`: [4](#0-3) 

`calculate_block_hash` chains `state_diff_commitment.0.0` directly into the Poseidon hash: [5](#0-4) 

### Impact Explanation
A wrong `state_diff_commitment` propagates into the Poseidon hash chain of `calculate_block_hash`, producing an incorrect `BlockHash`. This wrong hash is:
- Stored as the canonical block hash for height *h*.
- Used as `previous_block_hash` when computing the hash for height *h+1*, poisoning every subsequent block hash.
- Returned by RPC `starknet_getBlockWithTxHashes` and related endpoints as an authoritative value.
- Potentially fed into proof-manager and SNOS inputs that rely on stored block hashes for proof-fact verification.

This matches: **Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input** (Critical tier) and **RPC execution … returns an authoritative-looking wrong value** (High tier).

### Likelihood Explanation
`add_sync_block` is reachable from the consensus orchestrator during normal block synchronisation: [6](#0-5) 

Any peer node participating in consensus can supply a `SyncBlock` with a crafted `block_header_commitments.state_diff_commitment`. Because `verify_state_diff_hash` defaults to `false` and no independent recomputation occurs in `add_sync_block`, the malformed value is accepted without error.

### Recommendation
Add an explicit check in `add_sync_block` immediately after extracting `block_header_commitments`:

```rust
// After extracting header_commitments from the sync block:
let expected = calculate_state_diff_hash(&state_diff);
if header_commitments.state_diff_commitment != expected {
    return Err(BatcherError::StateDiffCommitmentMismatch {
        block_number,
        provided: header_commitments.state_diff_commitment,
        calculated: expected,
    });
}
```

Additionally, set `verify_state_diff_hash = true` as the production default in `CommitterConfig` so the committer acts as a second line of defence: [7](#0-6) 

### Proof of Concept
1. Construct a `SyncBlock` for height *N* with a valid `state_diff` but with `block_header_commitments.state_diff_commitment` set to an arbitrary wrong `Felt` value.
2. Call `batcher.add_sync_block(sync_block)`.
3. Observe that `commit_proposal` stores `PartialBlockHashComponents` containing the wrong commitment — no error is raised.
4. After the committer processes the task (with `verify_state_diff_hash = false`), call the commitment manager's finalization path.
5. `calculate_block_hash` produces a hash that differs from the hash a correct node would compute for the same block, confirming the commitment chain is corrupted. [8](#0-7) [9](#0-8)
