### Title
Integer Overflow in Relative Timelock Arithmetic Allows `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` Bypass — (`File: crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks()` uses `wrapping_add` (instead of `saturating_add`) when `nowrap=false` to evaluate `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE` conditions. When the relative value is large enough to cause a `u32`/`u64` overflow, the wrapped result is a small number, making the "not yet mature" comparison evaluate to `false` — i.e., the timelock is treated as satisfied when it is not. A coin whose puzzle emits a large-but-overflowing relative height/seconds condition can be spent immediately on any node that invokes `check_time_locks` with `nowrap=false`.

---

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` branches on the `nowrap` flag for all four relative timelock conditions. For `ASSERT_HEIGHT_RELATIVE`:

```rust
// nowrap=true path (correct)
if prev_transaction_block_height < unspent.confirmed_block_index.saturating_add(height_relative) {
    return Err(...AssertHeightRelativeFailed);
}
// nowrap=false path (vulnerable)
} else if prev_transaction_block_height < unspent.confirmed_block_index.wrapping_add(height_relative) {
    return Err(...AssertHeightRelativeFailed);
}
``` [1](#0-0) 

The guard condition is `prev_height < (confirmed_height + height_relative)`. When `confirmed_height + height_relative` overflows `u32`, `wrapping_add` produces a small value (e.g., `10 + 0xFFFF_FFF6 = 0`). The comparison `prev_height < 0` is always `false`, so the function returns `Ok(())` — the spend is accepted — even though the coin has not matured by the required number of blocks.

The same flaw applies to `ASSERT_SECONDS_RELATIVE` via `u64::wrapping_add`:

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(...AssertSecondsRelativeFailed);
}
``` [2](#0-1) 

The test suite explicitly documents and asserts this divergence:

```
# 10 + (2^32 - 10) overflows to 0, wrapping: 15 < 0 -> Ok
# saturating: clamps to u32::MAX, 15 < u32::MAX -> Err
(make_test_conds(height_relative=0xFFFF_FFF6), 13, None),
# 10 + u32::MAX overflows to 9, wrapping: 15 < 9 -> Ok
(make_test_conds(height_relative=0xFFFF_FFFF), 13, None),
``` [3](#0-2) 

`expected_wrap=None` means the check passes (no error returned) — the timelock is silently bypassed.

The Python binding `py_check_time_locks` forwards the caller-supplied `nowrap` flag directly to the Rust function without any additional guard: [4](#0-3) 

The function is exported to the Python `chia_rs` module and is the authoritative timelock gate called by the full node before accepting a spend bundle: [5](#0-4) 

---

### Impact Explanation

Any coin whose CLVM puzzle emits `(ASSERT_HEIGHT_RELATIVE r)` or `(ASSERT_SECONDS_RELATIVE r)` with a value `r` large enough to overflow `confirmed_height + r` (resp. `coin_timestamp + r`) will have its timelock silently bypassed when the full node calls `check_time_locks` with `nowrap=false`. The spend is accepted as consensus-valid even though the coin has not aged by the required number of blocks or seconds. This is a direct timelock condition validation bypass enabling unauthorized spend acceptance — matching the allowed High impact: *"timelock … validation bypass enables unauthorized spend acceptance."*

---

### Likelihood Explanation

`nowrap=false` is the legacy code path retained for backward compatibility with pre-softfork blocks. Any full node that processes historical blocks (or that is misconfigured to pass `nowrap=False` from Python) will apply the wrapping path. An attacker who controls a coin's puzzle (or who can predict the puzzle of a protocol coin) can craft `ASSERT_HEIGHT_RELATIVE 0xFFFF_FFFF` (or the seconds equivalent) to make the coin immediately spendable on such nodes. The entry path is a standard unprivileged CLVM spend bundle submitted to the network.

---

### Recommendation

1. **Remove the `nowrap=false` branch entirely** if the wrapping behavior is no longer required for any live block height. Replace both `wrapping_add` calls with `saturating_add` unconditionally.
2. If backward compatibility must be preserved, **add an explicit overflow check** before the comparison: if `confirmed_height.checked_add(height_relative)` returns `None`, the condition can never be satisfied and must return `Err(AssertHeightRelativeFailed)` rather than silently passing.
3. Audit all callers of `py_check_time_locks` in the Python full-node code to confirm `nowrap=True` is always passed for any block height where the softfork is active, and add an assertion or type-level enforcement to prevent `nowrap=False` from being used on current-tip blocks.

---

### Proof of Concept

**Setup**: coin confirmed at block height `10`, current `prev_transaction_block_height = 15`.

**Puzzle condition**: `(ASSERT_HEIGHT_RELATIVE 0xFFFF_FFF6)`

**Arithmetic with `nowrap=false`**:
```
confirmed_block_index.wrapping_add(height_relative)
= 10u32.wrapping_add(0xFFFF_FFF6)
= 0x0000_0000  (overflows to 0)

check: prev_height (15) < 0  →  false  →  Ok(())  ← timelock bypassed
```

**Arithmetic with `nowrap=true` (correct)**:
```
confirmed_block_index.saturating_add(height_relative)
= 10u32.saturating_add(0xFFFF_FFF6)
= 0xFFFF_FFFF  (saturates at u32::MAX)

check: prev_height (15) < 4294967295  →  true  →  Err(AssertHeightRelativeFailed)
```

The same overflow pattern applies to `ASSERT_SECONDS_RELATIVE` with `u64` arithmetic: a coin with `seconds_relative = 0xFFFF_FFFF_FFFF_D8F0` and `coin_timestamp = 10000` wraps to `0`, making `current_timestamp < 0` always false, bypassing the seconds-relative timelock entirely. [6](#0-5)

pub fn py_check_time_locks(
    removal_coin_records: HashMap<Bytes32, CoinRecord>,
    bundle_conds: &OwnedSpendBundleConditions,
    prev_transaction_block_height: u32,
    timestamp: u64,
    nowrap: bool,
) -> PyResult<Option<u32>> {

        &removal_coin_records,
        bundle_conds,
        prev_transaction_block_height,
        timestamp,
        nowrap,

        Err(ec) => Ok(Some(ec.error_code().into())),
