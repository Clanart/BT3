### Title
`eth_estimateGas` Underestimates Gas for Try/Catch Transactions Due to 63/64 Rule — (`File: x/evm/keeper/grpc_query.go`)

### Summary

The `EstimateGas` gRPC handler in Ethermint applies an optimization that sets the binary-search lower bound to `ExecutionGasUsed - 1` from an unconstrained execution. For Solidity contracts that use `try/catch` with external calls (`this.foo()`), EIP-150's 63/64 forwarding rule means the subcall receives less gas at the estimated limit than it did during the unconstrained probe. The binary search converges to a gas value at which the subcall silently runs out of gas and is caught by the `catch` block, while the outer transaction returns no `vmError`. `eth_estimateGas` therefore returns a value that is insufficient for the intended execution path, feeding incorrect gas data into every subsequent transaction that relies on the estimate.

### Finding Description

In `x/evm/keeper/grpc_query.go`, `EstimateGas` first executes the transaction at the maximum allowable gas (`hi`) to obtain `ExecutionGasUsed`, then sets the binary-search lower bound:

```go
if result.ExecutionGasUsed > 0 {
    lo = result.ExecutionGasUsed - 1
}
hi, err = types.BinSearch(lo, hi, executable)
``` [1](#0-0) 

The comment acknowledges the assumption: *"For almost any transaction, the gas consumed by the unconstrained execution above lower-bounds the gas limit required for it to succeed."*

This assumption is violated for contracts that use Solidity `try/catch` with an external call to `this`. Under EIP-150, only 63/64 of the remaining gas is forwarded to the subcall. Concretely:

- **Unconstrained probe** (`hi` = 1,000,000): outer overhead = 5,000 gas; subcall receives `63/64 × 995,000 ≈ 979,531` gas; subcall uses 410,000 gas; `ExecutionGasUsed` = 415,000.
- **Binary search lower bound**: `lo = 414,999`.
- **At `gas = 415,000`**: outer overhead = 5,000 gas; subcall receives `63/64 × 410,000 ≈ 403,593` gas; subcall needs 410,000 gas → **out of gas in subcall**; `catch` block executes; outer transaction returns no `vmError`; `executable(415,000)` returns `failed = false`.
- **Binary search converges to `hi = 415,000`** — the minimum gas at which the outer transaction does not revert — and returns this as the estimate.

The `BinSearch` function in `x/evm/types/utils.go` finds the minimum gas where `executable` returns `failed = false`, without distinguishing between "succeeded correctly" and "succeeded because try/catch swallowed an OOG subcall":

```go
func BinSearch(lo, hi uint64, executable func(uint64) (bool, *EVMResult, error)) (uint64, error) {
    for lo+1 < hi {
        mid := (hi + lo) / 2
        failed, _, err := executable(mid)
        if failed { lo = mid } else { hi = mid }
    }
    return hi, nil
}
``` [2](#0-1) 

The `ApplyMessageWithConfig` function returns a result with an empty `VmError` whenever the top-level EVM call succeeds, even if an inner subcall ran out of gas and was caught:

```go
var vmError string
if vmErr != nil {
    vmError = vmErr.Error()
}
``` [3](#0-2) 

The `GasConsumerTryCatch.sol` contract and `test_trycatch_gas_estimation_underestimate` integration test in the repository confirm the developers are aware of this class of issue, but the production estimation logic in `grpc_query.go` does not account for the 63/64 rule when setting `lo`. [4](#0-3) 

### Impact Explanation

Any user who calls `eth_estimateGas` for a transaction that uses `try/catch` with an external call receives an underestimated gas limit. When the user submits the transaction with that limit, the subcall silently runs out of gas; the `catch` branch executes instead of the intended success path. If the subcall was responsible for transferring tokens, updating balances, or any state-critical operation, those effects are silently skipped while the transaction is recorded as successful. This constitutes the public JSON-RPC path feeding incorrect consensus-critical data into transaction execution, matching the **High** allowed impact: *"Public JSON-RPC, gRPC, simulation, tracing, receipt/log, or indexer path feeds incorrect consensus-critical data into transaction execution."*

### Likelihood Explanation

The vulnerability is triggered by any unprivileged user calling `eth_estimateGas` on a contract that uses Solidity `try/catch` with `this.externalCall()`. This is a common pattern in DeFi contracts (e.g., callback-based deposit/withdrawal handlers analogous to the GMX pattern in the reference report). No special privileges, governance access, or validator compromise is required. The entry path is the standard `eth_estimateGas` JSON-RPC endpoint exposed by every Ethermint node.

### Recommendation

Remove or correct the `lo = result.ExecutionGasUsed - 1` optimization for cases where the transaction may succeed at a lower gas limit due to try/catch swallowing OOG errors. Two options:

1. **Conservative fix**: Do not set `lo` from `ExecutionGasUsed`; keep `lo = ethparams.TxGas - 1`. This restores correctness at the cost of more binary-search iterations.
2. **Targeted fix**: After the binary search converges, verify that `executable(hi)` succeeds with the subcall also succeeding (e.g., by checking that the result matches the unconstrained execution's return value or logs), and if not, increase `hi` by `64/63 × subcall_gas_budget`.

The go-ethereum reference implementation avoids this by not applying the `ExecutionGasUsed` lower-bound optimization.

### Proof of Concept

1. Deploy `GasConsumerTryCatch` (already present in the repo at `tests/integration_tests/hardhat/contracts/GasConsumerTryCatch.sol`).
2. Call `eth_estimateGas` for `callWithTryCatch(N, false)` where `N` is large enough that `63/64 × (estimatedGas - overhead) < N × 20000`.
3. Submit the transaction with the returned gas estimate.
4. Observe that `receipt.status == 1` (success) but `totalWrites` was not incremented (the subcall ran out of gas and was caught), whereas submitting with `gas = 1,000,000` correctly increments `totalWrites`.

The `lo` shortcut at line 430–431 of `grpc_query.go` causes the binary search to converge to `ExecutionGasUsed` rather than the true minimum gas needed for the subcall to succeed, because `executable(ExecutionGasUsed)` returns `failed = false` (the `catch` block handles the OOG). [5](#0-4)

// BinSearch execute the binary search and hone in on an executable gas limit
func BinSearch(lo, hi uint64, executable func(uint64) (bool, *EVMResult, error)) (uint64, error) {
	for lo+1 < hi {
		mid := (hi + lo) / 2
		failed, _, err := executable(mid)
		// If the error is not nil(consensus error), it means the provided message
		// call or transaction will never be accepted no matter how much gas it is
		// assigned. Return the error directly, don't struggle any more.
		if err != nil {
			return 0, err
		}
		if failed {
			lo = mid
		} else {
			hi = mid
		}
	}
	return hi, nil
}
