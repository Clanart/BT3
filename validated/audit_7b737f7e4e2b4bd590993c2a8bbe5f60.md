### Title
`denormalize_amount` Multiplication Overflow Permanently Freezes Bridged Funds for High-Decimal Tokens — (`File: near/omni-bridge/src/lib.rs`)

### Summary
An unprivileged attacker can register an ERC-20 token with `origin_decimals ≥ 57` through the fully permissionless bridge registration flow. Any subsequent EVM→NEAR transfer of that token causes `denormalize_amount` to panic due to `u128` multiplication overflow (with `overflow-checks = true`), permanently locking the user's tokens on the EVM side with no recovery path.

### Finding Description

`denormalize_amount` in the NEAR bridge contract performs:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

`u128::MAX ≈ 3.4 × 10^38`, so `10_u128.pow(39)` already overflows. With `overflow-checks = true` in the workspace profile, this causes a runtime panic for any non-zero amount. [2](#0-1) 

The `Decimals` struct stores `{ decimals, origin_decimals }` where `decimals` is the EVM-normalized value (capped at 18 by `_normalizeDecimals`) and `origin_decimals` is the raw value from the source token. [3](#0-2) 

`add_token` stores these values with no validation that `origin_decimals - decimals < 39`: [4](#0-3) 

`denormalize_amount` is called in `fin_transfer_callback` (EVM→NEAR incoming transfer processing): [5](#0-4) 

and in `fast_fin_transfer`: [6](#0-5) 

and in `claim_fee_callback`: [7](#0-6) 

**Note on CLAUDE.md dismissal:** The project's CLAUDE.md dismisses "Decimal Arithmetic Underflow" (the subtraction `origin_decimals - decimals` wrapping when `origin_decimals < decimals`) as a "correct fail-safe." This multiplication overflow is a distinct bug — it occurs when `origin_decimals - decimals ≥ 39` (a valid, non-misconfigured configuration), and unlike the subtraction case, it triggers *after* user funds are already locked on the EVM side, making the panic a fund freeze rather than a fail-safe.

### Impact Explanation

When `fin_transfer_callback` panics, the NEAR-side callback fails. The user's tokens are already locked in the EVM bridge contract via `initTransfer`. There is no automatic refund mechanism: `finTransfer` on EVM requires a valid NEAR-side proof, which is never generated when the callback panics. The result is **permanent, irrecoverable lock of user funds** on the EVM side.

This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

### Likelihood Explanation

The entire token registration flow is permissionless:

1. **`logMetadata`** on EVM is explicitly permissionless — anyone can call it for any ERC-20: [8](#0-7) 

2. **`deploy_token`** on NEAR is permissionless (requires only a valid prover result): [9](#0-8) 

3. **`deployToken`** on EVM is permissionless (requires only a valid MPC signature, which is produced automatically): [10](#0-9) 

4. **`bind_token`** on NEAR is permissionless: [11](#0-10) 

An attacker deploys an ERC-20 with `decimals() = 57` (a valid `uint8` value), walks through all four steps, and the token is registered with `{ decimals: 18, origin_decimals: 57 }`. Any victim who then bridges this token from EVM to NEAR has their funds permanently locked.

### Recommendation

1. **Validate `diff_decimals` in `add_token`**: Reject registration if `origin_decimals - decimals >= 39` (i.e., `10^diff_decimals` would overflow `u128`).
2. **Use `checked_mul` / `checked_pow` in `denormalize_amount`**: Replace `amount * (10_u128.pow(diff_decimals))` with `10_u128.checked_pow(diff_decimals).and_then(|scale| amount.checked_mul(scale))` and handle `None` as an explicit error rather than a panic.
3. **Cap `origin_decimals`** at a safe maximum (e.g., 56) during `bind_token_callback` before calling `add_token`.

### Proof of Concept

1. Attacker deploys `MaliciousToken` on EVM with `decimals() = 57`.
2. Attacker calls `OmniBridge.logMetadata(maliciousToken)` — permissionless, emits `LogMetadata` event with `decimals = 57`.
3. Prover verifies the event; attacker calls `NEAR.deploy_token(prover_args)` — NEAR deploys an omni-token with 57 decimals, MPC signs the metadata payload.
4. Attacker calls `OmniBridge.deployToken(signature, metadata)` — EVM bridge calls `_normalizeDecimals(57) = 18`, deploys bridge token with 18 decimals, emits `DeployToken(decimals=18, originDecimals=57)`.
5. Attacker calls `NEAR.bind_token(prover_args)` — `bind_token_callback` calls `add_token(token, address, 18, 57)`, storing `Decimals { decimals: 18, origin_decimals: 57 }`.
6. Victim calls `OmniBridge.initTransfer(bridgeToken, 1_000_000, ...)` — 1,000,000 bridge tokens (18 decimals) locked in EVM bridge.
7. Relayer submits proof to NEAR; `fin_transfer_callback` calls `denormalize_amount(1_000_000, { decimals: 18, origin_decimals: 57 })`.
8. `diff_decimals = 39`; `10_u128.pow(39)` overflows `u128` → panic with `overflow-checks = true`.
9. NEAR callback fails; no `FinTransfer` proof is ever generated; victim's 1,000,000 bridge tokens are permanently locked in the EVM bridge.
