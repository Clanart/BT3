Audit Report

## Title
Excess Native ETH Sent to Payable Swap Functions Is Not Automatically Refunded and Can Be Stolen by Any Caller — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`, `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

All four swap entry points (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) are `payable` and accept native ETH. When `tokenIn == WETH`, `pay()` wraps only the exact amount the pool requests, leaving any excess ETH stranded in the router. `refundETH()` is permissionless and sends the full router ETH balance to `msg.sender`, so any third party can call it after the victim's swap and steal the leftover ETH.

## Finding Description

`exactOutputSingle` is `payable` and stores no binding between the depositor and any future refund recipient. [1](#0-0) 

Inside the swap callback, `pay()` is invoked with the pool-determined `value`. When `tokenIn == WETH` and the router holds native ETH, it deposits exactly `value` into WETH and forwards it to the pool; the remainder (`address(this).balance - value`) is never touched. [2](#0-1) 

After `exactOutputSingle` returns, the excess ETH sits in the router. `refundETH()` has no access control and sends the entire balance to whoever calls it: [3](#0-2) 

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only applies to plain ETH transfers with empty calldata; it does **not** block ETH sent via a `payable` function call, so users can freely send ETH through any of the four swap functions. [4](#0-3) 

The same stranding applies to `exactOutput` (multihop), `exactInputSingle`, and `exactInput` when `msg.value > amountIn`. [5](#0-4) 

The existing test suite only validates the safe multicall pattern (swap + explicit `refundETH()` in one atomic call) and does not cover a direct `exactOutputSingle` call with `msg.value > actualAmountIn` followed by a third-party `refundETH()`. [6](#0-5) 

## Impact Explanation

Any ETH sent above the pool-determined `amountIn` is permanently stranded in the router after the swap returns. Because `refundETH()` is permissionless and sends to `msg.sender`, a third party (MEV bot) can steal the entire stranded balance in the same block. This is a direct, unconditional loss of user principal with no recovery path once the attacker claims it — qualifying as a Critical/High direct loss of user funds under Sherlock thresholds.

## Likelihood Explanation

Exact-output swaps are the primary use case where callers cannot pre-compute the precise input amount; sending `amountInMaximum` as ETH is the natural usage pattern. MEV infrastructure routinely monitors for stranded ETH in router contracts. The attack requires no special privilege, no malicious setup, and no non-standard token behavior — only a standard `exactOutputSingle` call with `tokenIn == WETH` and `msg.value > actualAmountIn`.

## Recommendation

Automatically refund any remaining native ETH balance to `msg.sender` at the end of each payable swap function before returning:

```solidity
function exactOutputSingle(ExactOutputSingleParams calldata params)
    external payable returns (uint256 amountIn)
{
    // ... existing swap logic ...
    _clearExpectedCallbackPool();

    // Refund excess ETH to the caller
    uint256 remaining = address(this).balance;
    if (remaining > 0) _transferETH(msg.sender, remaining);
}
```

Apply the same pattern to `exactOutput`, `exactInputSingle`, and `exactInput`.

## Proof of Concept

```solidity
// 1. Alice calls exactOutputSingle with msg.value = 2 ETH, tokenIn = WETH
//    Pool charges actualAmountIn = 1.2 ETH
//    pay() wraps exactly 1.2 ETH → 0.8 ETH excess remains in router

uint256 aliceEthBefore = alice.balance;
vm.prank(alice);
router.exactOutputSingle{value: 2 ether}(
    IMetricOmmSimpleRouter.ExactOutputSingleParams({
        pool: address(pool),
        tokenIn: address(weth),
        tokenOut: address(token1),
        zeroForOne: true,
        amountOut: amountOut,        // results in actualAmountIn = 1.2 ETH
        amountInMaximum: 2 ether,
        recipient: alice,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// Router now holds 0.8 ETH

// 2. Bob (MEV bot) calls refundETH() in the same block
uint256 bobEthBefore = bob.balance;
vm.prank(bob);
router.refundETH();
assertEq(bob.balance - bobEthBefore, 0.8 ether); // Bob stole Alice's excess
assertEq(alice.balance, aliceEthBefore - 2 ether); // Alice lost full 2 ETH
```

  function exactOutputSingle(ExactOutputSingleParams calldata params) external payable returns (uint256 amountIn) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    int128 expectedAmountOut = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountOut);
    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(params.recipient, params.zeroForOne, -expectedAmountOut, priceLimitX64, "", params.extensionData);
    int128 amountOut = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    if (amountOut != expectedAmountOut) revert InvalidOutputAmount(amountOut, params.amountOut);

    amountIn = MetricOmmSwapInputs.int128ToUint128(
      MetricOmmSwapResults.extractAmountIn(params.zeroForOne, amount0Delta, amount1Delta)
    );

    if (amountIn > params.amountInMaximum) revert InputTooHigh(amountIn, params.amountInMaximum);
    _clearExpectedCallbackPool();
  }
