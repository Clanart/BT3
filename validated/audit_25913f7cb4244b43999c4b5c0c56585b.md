### Title
`timeout_height` Excluded from Legacy EIP-712 Signed Hash Allows Attacker to Modify Transaction Expiry Without Invalidating Signature - (File: `ethereum/eip712/eip712_legacy.go`)

---

### Summary

In the legacy EIP-712 Cosmos transaction path, the `timeout_height` field is deliberately omitted from the EIP-712 type schema used to compute the signature hash. Because `timeout_height` is not covered by the signature, an attacker can freely modify this field in a submitted transaction without invalidating the signer's EIP-712 signature. A user who signs a transaction intending it to expire at a specific block height has that constraint silently stripped, enabling the transaction to be executed at any time the attacker chooses.

---

### Finding Description

In `ethereum/eip712/eip712_legacy.go`, the `extractMsgTypes` function defines the EIP-712 `"Tx"` struct type used for hashing. The `timeout_height` field is explicitly commented out:

```go
"Tx": {
    {Name: "account_number", Type: "string"},
    {Name: "chain_id",       Type: "string"},
    {Name: "fee",            Type: "Fee"},
    {Name: "memo",           Type: "string"},
    {Name: "msgs",           Type: "Msg[]"},
    {Name: "sequence",       Type: "string"},
    // Note timeout_height was removed because it was not getting filled with the legacyTx
    // {Name: "timeout_height", Type: "string"},
},
``` [1](#0-0) 

Meanwhile, in `ante/cosmos/eip712.go`, `VerifySignature` constructs `txBytes` via `legacytx.StdSignBytes`, which **does** include `tx.GetTimeoutHeight()`:

```go
txBytes := legacytx.StdSignBytes(
    signerData.ChainID,
    signerData.AccountNumber,
    signerData.Sequence,
    tx.GetTimeoutHeight(),   // ← included in the JSON payload
    legacytx.StdFee{...},
    msgs, tx.GetMemo(),
)
``` [2](#0-1) 

These `txBytes` are then passed to `LegacyWrapTxToTypedData`, which JSON-unmarshals them into `txData` and sets `typedData.Message = txData`: [3](#0-2) 

When `apitypes.TypedDataAndHash(typedData)` is called to produce the `sigHash`, it hashes only the fields declared in the type schema. Because `timeout_height` is absent from the `"Tx"` schema, it is silently ignored during hashing regardless of its value in the message payload: [4](#0-3) 

The result: the EIP-712 signature commits to `account_number`, `chain_id`, `fee`, `memo`, `msgs`, and `sequence` — but **not** `timeout_height`. The field is user-controlled at submission time and completely uncovered by the cryptographic commitment.

---

### Impact Explanation

The Cosmos SDK enforces `timeout_height` at block processing: if `timeout_height > 0` and `current_block_height > timeout_height`, the transaction is rejected. A signer who sets `timeout_height = X` intends the transaction to be valid only before block `X`.

Because `timeout_height` is not in the EIP-712 hash, an attacker who intercepts the signed transaction (from the mempool, during broadcast, or via a malicious relayer) can:

1. **Remove a timeout**: Change `timeout_height` from `X` to `0`, making the transaction valid indefinitely. The signer's expiry intent is nullified.
2. **Shorten a timeout**: Change `timeout_height` to a past block, causing the transaction to be immediately rejected (griefing/DoS against the signer).

In scenario (1), the attacker can hold a signed delegation, governance vote, or token transfer and submit it at a strategically chosen future time — after market conditions change, after a governance window closes, or after the signer believes the tx has safely expired. The signer's EIP-712 signature remains valid for all of these modified transactions.

This matches the allowed High impact: **EIP-712 authorization bypass enabling forged execution** — the signed authorization does not fully bind the transaction's execution constraints.

---

### Likelihood Explanation

The legacy EIP-712 path (`LegacyEip712SigVerificationDecorator` / `ExtensionOptionsWeb3Tx`) is the active signing path for Metamask and Web3 wallet users submitting Cosmos SDK messages. Any transaction broadcast through a public RPC endpoint or relayer is observable in the mempool. The modification requires only changing a single integer field in the Cosmos tx body before rebroadcasting — no cryptographic capability is needed. The attacker only needs to observe a pending signed transaction.

---

### Recommendation

Add `timeout_height` back to the `"Tx"` EIP-712 type schema in `extractMsgTypes`:

```go
"Tx": {
    {Name: "account_number",  Type: "string"},
    {Name: "chain_id",        Type: "string"},
    {Name: "fee",             Type: "Fee"},
    {Name: "memo",            Type: "string"},
    {Name: "msgs",            Type: "Msg[]"},
    {Name: "sequence",        Type: "string"},
    {Name: "timeout_height",  Type: "string"},  // ← restore
},
```

The original comment states it "was not getting filled with the legacyTx." The correct fix is to ensure `timeout_height` is always serialized into `txBytes` (even as `"0"` when absent) and included in the type schema, so the field is always covered by the signature. Clients that do not set a timeout will sign over `"0"`, which is safe and consistent. [1](#0-0) 

---

### Proof of Concept

1. Alice uses a Web3 wallet to sign a legacy EIP-712 Cosmos tx (e.g., `MsgDelegate`) with `timeout_height = 500` (current block is 400). She broadcasts it.

2. The EIP-712 hash is computed over `{account_number, chain_id, fee, memo, msgs, sequence}` — `timeout_height` is absent from the schema and not hashed.

3. Bob (attacker) observes the tx in the mempool. He copies the tx, sets `timeout_height = 0` in the Cosmos tx body, and rebroadcasts.

4. In `VerifySignature`, `txBytes` is rebuilt with `timeout_height = 0`. `LegacyWrapTxToTypedData` produces a `typedData` whose `Message["timeout_height"] = "0"`. But `apitypes.TypedDataAndHash` ignores this field (not in schema), producing the **same** `sigHash` as Alice's original.

5. `ethcrypto.VerifySignature(pubKey, sigHash, feePayerSig)` passes — Alice's signature is valid for Bob's modified tx.

6. Bob's tx (with no timeout) is accepted and executed. Alice's original tx (with `timeout_height = 500`) is now a duplicate nonce and will be rejected. Bob has successfully replaced Alice's time-bounded authorization with an unbounded one. [5](#0-4) [1](#0-0)
