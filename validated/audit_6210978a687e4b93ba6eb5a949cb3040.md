Audit Report

## Title
Stranded Native ETH on `MetricOmmPoolLiquidityAdder` Is Freely Claimable as WETH Liquidity by Any Subsequent Caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` reads `address(this).balance` — the contract's total ETH balance — when settling WETH payments, rather than limiting itself to the current call's `msg.value`. ETH overpaid by a prior caller accumulates on `MetricOmmPoolLiquidityAdder` and is silently consumed by the next caller who adds liquidity to a WETH pool, giving that caller a fully-funded liquidity position at the prior caller's expense.

## Finding Description
In `PeripheryPayments.pay()`, the WETH branch reads the contract-wide native balance:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
}
``` [1](#0-0) 

`addLiquidityExactShares` is `payable` and stores `msg.sender` as the payer in transient context, but never enforces that only the current call's `msg.value` is available for wrapping: [2](#0-1) 

The callback settlement calls `pay(token0, payer, msg.sender, amount0Delta)` where `payer` is the current caller — but when `nativeBalance >= value`, no `safeTransferFrom` is issued against `payer` at all; the payment comes entirely from the contract's accumulated ETH: [3](#0-2) 

The `receive()` guard only blocks direct ETH transfers from non-WETH addresses; it does not prevent ETH from accumulating via payable function calls: [4](#0-3) 

`refundETH()` is a separate, optional call that the user must explicitly include in a multicall batch. `addLiquidityExactShares` never calls it internally, so overpaid ETH is routinely stranded: [5](#0-4) 

The `maxAmountToken0`/`maxAmountToken1` caps only bound what the pool may request in the callback; they do not prevent `pay()` from consuming stranded ETH on behalf of a different payer: [6](#0-5) 

**Exploit path:**
1. Victim calls `addLiquidityExactShares{value: 2X}(pool, ...)` for a WETH pool that requests only `X` WETH. `pay()` wraps `X` ETH; `X` ETH remains on the contract.
2. Victim omits `refundETH()` (or calls `addLiquidityExactShares` directly without multicall).
3. Attacker calls `addLiquidityExactShares{value: 0}(pool, attacker, ...)`. `pay()` reads `address(this).balance = X >= X`, wraps `X` ETH, sends WETH to pool. Attacker receives a fully-funded liquidity position. No token is pulled from the attacker.

## Impact Explanation
Direct loss of user principal: the victim's stranded ETH is transferred to the pool as WETH liquidity credited to the attacker's position. This is an unconditional theft of funds meeting the Critical/High threshold under Sherlock rules — the victim loses `X` ETH and the attacker gains an equivalent liquidity position at zero cost. [1](#0-0) 

## Likelihood Explanation
Overpayment is the normal usage pattern: callers set `msg.value = maxAmountToken0` as a safety cap, and the pool requests only what it needs. The difference is stranded unless `refundETH()` is explicitly included in the same multicall batch. Calling `addLiquidityExactShares` directly (without multicall) is a common and documented usage that leaves no refund path. An attacker can monitor the contract's ETH balance on-chain and call with `msg.value = 0` immediately after any overpayment. No special privileges are required. [2](#0-1) 

## Recommendation
Track how much of `address(this).balance` belongs to the current call by snapshotting the pre-call balance at entry and computing the delta attributable to `msg.value`. In `pay()`, limit the native ETH available for wrapping to `msg.value` (passed as a parameter or stored in transient context at the start of each outer call), and revert or fall back to `safeTransferFrom` if the contract's pre-existing balance would be consumed. Alternatively, enforce that `address(this).balance - msg.value` is unchanged after each top-level call, or automatically call `refundETH()` at the end of every liquidity-adding entry point. [7](#0-6) 

## Proof of Concept
```solidity
// Setup: WETH pool exists; token0 = WETH

vm.prank(victim);
