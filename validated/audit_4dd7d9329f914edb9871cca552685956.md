### Title
Tokens Permanently Locked When `normalize_amount(amount - fee) == 0` Due to Floor Division Without Pre-Validation — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer` locks user tokens before validating that the normalized transfer amount is non-zero. `sign_transfer` applies floor-division normalization and then panics if the result is zero, but by that point the tokens are already irrecoverably locked with no cancel path.

---

### Finding Description

`normalize_amount` uses integer floor division:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

`init_transfer` validates only that `fee < amount`, then immediately locks the tokens:

```rust
require!(
    transfer_message.fee.fee < transfer_message.amount,
    BridgeError::InvalidFee.as_ref()
);
``` [2](#0-1) 

Tokens are locked inside `init_transfer_internal` before any normalization check: [3](#0-2) 

Later, `sign_transfer` computes the normalized amount and panics if it is zero:

```rust
let amount_to_transfer = Self::normalize_amount(
    transfer_message.amount_without_fee().near_expect(BridgeError::InvalidFee),
    decimals,
);
require!(
    amount_to_transfer > 0,
    BridgeError::InvalidAmountToTransfer.as_ref()
);
``` [4](#0-3) 

Because `sign_transfer` is the only forward path for a pending transfer, and no cancel/refund function exists for pending transfers, the tokens are permanently locked.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

For any token where `origin_decimals > decimals` (e.g., a NEAR-native token with 24 decimals bridging to an EVM chain where it is registered with 6 decimals, giving `diff_decimals = 18`), any transfer where `amount - fee < 10^18` will normalize to zero. The user's tokens are locked in the bridge contract with no mechanism to recover them. This matches the allowed impact: *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."*

---

### Likelihood Explanation

**Medium.** The decimal gap between NEAR (24 decimals) and EVM tokens (commonly 6 or 8 decimals) is a standard bridge configuration. A user sending a small amount — e.g., less than 1 full unit of a 24-decimal token bridging to a 6-decimal destination — will trigger this silently. The `init_transfer` validation (`fee < amount`) gives no indication that the amount is too small to normalize. The user has no way to know the transfer is undeliverable until `sign_transfer` is called by the relayer and reverts.

---

### Recommendation

Add a pre-validation in `init_transfer` (or `init_transfer_internal`) that checks `normalize_amount(amount - fee, decimals) > 0` before locking tokens. Alternatively, implement a `cancel_transfer` function that allows the sender to reclaim locked tokens for transfers that have not yet been signed.

---

### Proof of Concept

1. A token is registered with `origin_decimals = 24`, `decimals = 6` (diff = 18, divisor = `10^18`).
2. User calls `ft_transfer_call` with `amount = 5 * 10^17` (half a unit in 24-decimal representation) and `fee = 0`, targeting an EVM recipient.
3. `init_transfer` passes the `fee < amount` check and locks `5 * 10^17` tokens in the bridge.
4. Relayer calls `sign_transfer`:
   - `amount_without_fee() = 5 * 10^17`
   - `normalize_amount(5 * 10^17, {origin: 24, dest: 6}) = 5 * 10^17 / 10^18 = 0`
   - `require!(amount_to_transfer > 0, ...)` panics with `InvalidAmountToTransfer`
5. Every subsequent call to `sign_transfer` for this transfer ID will also panic.
6. The `5 * 10^17` tokens remain locked in the bridge contract forever with no recovery path. [1](#0-0) [4](#0-3) [2](#0-1) [5](#0-4)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
