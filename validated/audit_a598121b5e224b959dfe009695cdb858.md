### Title
Permanent Token Lock When `ft_transfer_call` Fails in `fin_transfer_send_tokens_callback` — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

In `fin_transfer_send_tokens_callback`, when the cross-contract `ft_transfer_call` to the NEAR recipient fails (i.e., the receiver's `ft_on_transfer` panics), the NEP-141 standard guarantees the tokens are returned to the bridge. However, `is_refund_required` returns `false` on a promise error, so `remove_fin_transfer` is never called. The transfer remains permanently in `finalised_transfers`, and the returned tokens are irrecoverably locked in the bridge with no admin recovery path.

---

### Finding Description

The `process_fin_transfer_to_near` function first commits the transfer to `finalised_transfers` via `add_fin_transfer`, then schedules `send_tokens` (which may dispatch `ft_transfer_call` for non-deployed tokens with a `msg`), and chains `fin_transfer_send_tokens_callback` as the resolution callback. [1](#0-0) 

The callback decides whether to revert state via `is_refund_required`: [2](#0-1) 

`is_refund_required` is defined as: [3](#0-2) 

The critical branch is:
```rust
Err(_) => false,  // Unexpected case: don't refund
```

When `ft_transfer_call` fails (the receiver's `ft_on_transfer` panics), NEAR's promise result is `Err(PromiseError::Failed)`. The NEP-141 standard guarantees the token contract reverts the transfer and returns the tokens to the bridge. But `is_refund_required` returns `false`, so:

- `remove_fin_transfer` is **not** called → transfer stays in `finalised_transfers` forever
- `revert_lock_actions` is **not** called → locked-token accounting is not restored
- `FinTransferEvent` is emitted as if the transfer succeeded [4](#0-3) 

The same desync applies to plain `ft_transfer` failures (`is_ft_transfer_call = false` → `is_refund_required` always returns `false`). [5](#0-4) 

The existing integration test confirms the tokens are returned to the bridge (locker balance = 1000) when the receiver panics, but the test does not verify that `finalised_transfers` is cleared, meaning the lock is silent and permanent.

---

### Impact Explanation

For non-deployed (locked) tokens bridged to NEAR with a non-empty `msg`:

1. The bridge marks the transfer as finalized in `finalised_transfers`.
2. `ft_transfer_call` is dispatched; the receiver panics.
3. NEP-141 returns the tokens to the bridge.
4. `fin_transfer_send_tokens_callback` runs with `Err(_)` → `is_refund_required = false`.
5. `remove_fin_transfer` is skipped → the transfer ID is permanently in `finalised_transfers`.
6. The user cannot re-submit the proof (replay blocked by `TransferAlreadyFinalised`).
7. There is no admin function to remove entries from `finalised_transfers`. [6](#0-5) 

Result: **irrecoverable lock** of the user's bridged assets. This matches the allowed impact: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.*

---

### Likelihood Explanation

The `msg` field is set by the original sender on the source chain and is forwarded verbatim to the NEAR recipient contract's `ft_on_transfer`. Any of the following causes the promise to fail:

- The recipient contract's `ft_on_transfer` panics (e.g., due to a bug, unexpected input, or insufficient gas forwarded).
- The recipient contract does not implement `ft_on_transfer`.
- Gas exhaustion inside the receiver.

The entry path is fully unprivileged: any user bridging tokens from EVM/Solana/StarkNet to a NEAR contract recipient with a `msg` field can trigger this. The `msg` field is a standard bridge feature exposed to all users.

---

### Recommendation

In `is_refund_required`, treat a failed promise as requiring a refund when `is_ft_transfer_call` is true, since NEP-141 guarantees token return on failure:

```rust
Err(_) => true,  // ft_transfer_call failed; NEP-141 guarantees tokens were returned
```

Similarly, for the `ft_transfer` (no-msg) path, add a promise-result check in `fin_transfer_send_tokens_callback` and call `remove_fin_transfer` + `revert_lock_actions` if the transfer failed.

---

### Proof of Concept

1. Alice initiates a transfer of 1000 locked tokens from Ethereum to NEAR, with `recipient = malicious_contract.near` and `msg = "trigger_panic"`.
2. The relayer calls `fin_transfer` with a valid proof.
3. `fin_transfer_callback` runs: `add_fin_transfer` marks the transfer ID in `finalised_transfers`; `send_tokens` dispatches `ft_transfer_call(malicious_contract.near, 1000, "trigger_panic")`.
4. `malicious_contract.near::ft_on_transfer` panics. NEP-141 reverts the transfer; 1000 tokens return to the bridge.
5. `fin_transfer_send_tokens_callback` runs: `env::promise_result_checked(0, ...)` returns `Err(Failed)` → `is_refund_required` returns `false`.
6. The `else` branch executes: fee is sent, `FinTransferEvent` is emitted. `remove_fin_transfer` is never called.
7. Alice attempts to re-submit the proof → `BridgeError::TransferAlreadyFinalised`. The 1000 tokens are permanently locked in the bridge with no recovery path. [3](#0-2) [2](#0-1) [6](#0-5)

    fn is_refund_required(is_ft_transfer_call: bool) -> bool {
        if is_ft_transfer_call {
            match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
                Ok(value) => {
                    if let Ok(amount) = near_sdk::serde_json::from_slice::<U128>(&value) {
                        // Normal case: refund if the used token amount is zero
                        // The amount can be zero if the `ft_on_transfer` in the receiver contract returns an amount instead of `0`, or if it panics.
                        amount.0 == 0
                    } else {
                        // Unexpected case: don't refund
                        false
                    }
                }
                // Unexpected case: don't refund
                Err(_) => false,
            }
        } else {
            // Not ft_transfer_call: don't refund
            false
        }
    }

    fn add_fin_transfer(&mut self, transfer_id: &TransferId) -> NearToken {
        let storage_usage = env::storage_usage();
        require!(
            self.finalised_transfers.insert(transfer_id),
            BridgeError::TransferAlreadyFinalised.as_ref()
        );
        env::storage_byte_cost()
            .saturating_mul((env::storage_usage().saturating_sub(storage_usage)).into())
    }
