### Title
Missing Declared-Class Validation in `replace_class` Creates Permanent Contract Deadlock — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash supplied by a contract is actually declared (i.e., present in `contract_class_changes`). This allows any contract to replace its own class hash with an arbitrary, undeclared felt value. Once committed, no valid proof can ever be generated for a subsequent call to that contract, because `execute_entry_point` will look up the undeclared hash in `contract_class_changes`, receive the default value `0`, and then fail to locate a compiled class with hash `0` in `compiled_class_facts_bundle`. The contract enters a permanent deadlock: it cannot be executed to fix itself, and no external mechanism can repair the state.

---

### Finding Description

**Root cause — missing validation in `execute_replace_class`:** [1](#0-0) 

The function reads `class_hash` directly from the syscall request and writes it into `contract_state_changes` with no check that `class_hash` has a corresponding entry in `contract_class_changes`. The acknowledged TODO comment at line 898 confirms this invariant is intentionally deferred:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
```

**Consequence — deadlock in `execute_entry_point`:**

When the affected contract is called in any subsequent block, `execute_entry_point` performs: [2](#0-1) 

`dict_read` on an uninitialized key returns the default value `0` (set by `dict_new()` in `initialize_state_changes`). `find_element` then searches `compiled_class_facts_bundle` for a compiled class with hash `0`. Because no such class exists, the hint-based `find_element` assertion fails, making it impossible to generate a valid proof for any call to the contract.

**The deadlock is symmetric (analog to the external report):**

| External Report (Morpho/Aave) | StarkNet OS Analog |
|---|---|
| Collateral flag on Aave (State A) | Contract's `class_hash` in `contract_state_changes` (State A) |
| Collateral flag on Morpho (State B) | Declared class hashes in `contract_class_changes` (State B) |
| Invariant: A and B must be consistent | Invariant: A must exist in B |
| Bug: Aave can auto-disable A, breaking invariant | Bug: `replace_class` sets A to an undeclared value, breaking invariant |
| Deadlock: can't enable A (B is on), can't disable B (A is off) | Deadlock: can't execute contract (A not in B), can't call `replace_class` to fix A (requires executing the contract) |

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is set to an undeclared value, the contract is permanently unexecutable. No valid STARK proof can be generated for any call to it. Any ERC-20 tokens, ETH, or other assets held in the contract's storage are irrecoverably frozen. The state cannot be repaired because:

1. `replace_class` can only be called from within the contract's own execution context.
2. Executing the contract requires a valid proof.
3. A valid proof requires the class hash to be declared.
4. The class hash is undeclared.

This is a true deadlock with no escape path.

---

### Likelihood Explanation

**Medium-High.** The attack path is directly reachable by any unprivileged contract deployer:

1. A malicious actor deploys a contract (e.g., a fake vault or DeFi protocol) that accepts deposits from users.
2. The actor calls a function on the contract that invokes `replace_class(arbitrary_undeclared_felt)`.
3. An honest sequencer includes this transaction — it is syntactically valid and the OS accepts it without error.
4. The state is committed on-chain with the undeclared class hash.
5. All user funds deposited in the contract are permanently frozen.

No privileged access, leaked keys, or malicious sequencer is required. The attack requires only the ability to deploy and call a contract.

---

### Recommendation

In `execute_replace_class`, add an explicit check that `request.class_hash` is present in `contract_class_changes` (i.e., has been declared in the current or a prior block) before updating the contract state. Specifically, perform a `dict_read` on `contract_class_changes` with `request.class_hash` as the key and assert the result is non-zero:

```cairo
// Verify the new class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(
    key=class_hash
);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the invariant enforced implicitly by `execute_entry_point` and closes the gap acknowledged by the existing TODO comment.

---

### Proof of Concept

1. Deploy contract `MaliciousVault` with a `freeze()` function that calls `replace_class(0xDEAD)` where `0xDEAD` is never declared.
2. Users deposit funds into `MaliciousVault`.
3. Attacker calls `freeze()`. The OS processes `execute_replace_class` without error: [3](#0-2) 

4. State is committed: `MaliciousVault.class_hash = 0xDEAD`.
5. Any subsequent call to `MaliciousVault` reaches `execute_entry_point`: [4](#0-3) 

`dict_read(key=0xDEAD)` → `0`. `find_element(key=0)` → assertion failure. No valid proof can be generated. All funds are permanently frozen.
