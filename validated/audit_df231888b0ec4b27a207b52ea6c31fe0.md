### Title
Queued Future-Round `ProposalInit` Bypasses Timestamp Window Validation, Committing a Wrong Block Timestamp into the Block Hash - (`crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs`)

---

### Summary

When a `ProposalInit` arrives for a future consensus round, it is stored in `queued_proposals` without any validation. When the round is eventually reached, the proposal is dequeued and validated against `clock.unix_now()` **at dequeue time**, not at the time the proposal was received. This allows a malicious proposer to inject a `timestamp` that exceeds the `block_timestamp_window_seconds` constraint relative to the time the proposal was received, because by the time the round is reached, the clock has advanced enough to make the previously-invalid timestamp pass. The injected timestamp flows directly into `BlockInfo`, `PartialBlockHashComponents`, and `calculate_block_hash`, producing a wrong block hash committed to storage.

---

### Finding Description

In `validate_proposal`, when `init.round > self.current_round`, the full `ProposalInit` (including its `timestamp`) is inserted into `queued_proposals` with no validation:

```rust
std::cmp::Ordering::Greater => {
    trace!("Queueing proposal for future round.");
    self.queued_proposals
        .insert(init.round, ((init, timeout, content_receiver), fin_sender));
    fin_receiver
}
``` [1](#0-0) 

When `set_height_and_round` later advances to that round, the queued proposal is dequeued and a fresh `ProposalInitValidation` is constructed using the **current** `self.previous_proposal_init` and the **current** clock: [2](#0-1) 

The timestamp check inside `is_proposal_init_valid` is:

```rust
let now: u64 = clock.unix_now();
if init_proposed.timestamp > now + proposal_init_validation.block_timestamp_window_seconds {
    return Err(...)
}
``` [3](#0-2) 

Because `now` at dequeue time is `T` seconds later than `now` at queue time, a timestamp of `now_queue + window + T` passes the check at dequeue time even though it would have been rejected at queue time. The `block_timestamp_window_seconds` is the configured maximum deviation: [4](#0-3) 

The injected timestamp then flows through `convert_to_sn_api_block_info(&init)` → `BlockInfo.block_timestamp` → `PartialBlockHashComponents::new(&block_info, commitments)`: [5](#0-4) 

And is hashed into the final block hash via `calculate_block_hash`: [6](#0-5) 

The wrong timestamp is also fed into `get_l1_prices_in_fri_and_wei(init_proposed.timestamp, ...)` during validation, producing wrong L1 gas prices for the block: [7](#0-6) 

---

### Impact Explanation

The `timestamp` field is a direct input to `calculate_block_hash`. A wrong timestamp produces a wrong block hash that is written to storage via `set_global_root_and_block_hash`: [8](#0-7) 

This wrong block hash is then:
1. Returned by the `get_block_hash` syscall to executing contracts.
2. Used as `previous_block_hash` in the next block's hash chain.
3. Validated against `proof_block_hash` in SNOS proof facts submitted by users — a mismatch causes valid proof-bearing transactions to be rejected. [9](#0-8) 

This matches: **Critical — Wrong state/storage value from blockifier/syscall/execution logic for accepted input** (wrong block hash in storage) and **Critical — Incorrect fee/gas/resource accounting** (wrong L1 gas prices derived from the injected timestamp).

---

### Likelihood Explanation

Any consensus peer that is the designated proposer for a future round (a normal, unprivileged role determined by committee rotation) can trigger this. The attacker sends a `ProposalInit` for round `R+k` with `timestamp = now + window + delta`. The proposal is queued without validation. When round `R+k` is reached (after `delta` seconds), the timestamp passes. No special keys or access are required beyond being a scheduled proposer.

---

### Recommendation

Validate the `timestamp` field of `ProposalInit` at queue time, not only at dequeue time. When a proposal arrives for a future round, apply at minimum the upper-bound timestamp check (`timestamp <= now_queue + block_timestamp_window_seconds`) before inserting into `queued_proposals`. The lower-bound check (`timestamp >= last_block_timestamp`) can remain deferred to dequeue time since `previous_proposal_init` may not yet be final.

---

### Proof of Concept

```
Scenario:
  block_timestamp_window_seconds = W (e.g. 30 seconds)
  now_queue = T

Step 1: Attacker (proposer for round R+1) sends ProposalInit{round: R+1, timestamp: T + W + 5}.
        Current round is R, so the proposal is queued without validation.

Step 2: Round R times out. set_height_and_round(height, R+1) is called at now_dequeue = T + 6.

Step 3: Dequeue check: T + W + 5 <= (T + 6) + W = T + W + 6. PASSES.

Step 4: validate_current_round_proposal runs with the injected timestamp T + W + 5.
        This timestamp (5 seconds beyond the window at queue time) is accepted.

Step 5: convert_to_sn_api_block_info produces BlockInfo{block_timestamp: T + W + 5}.
        PartialBlockHashComponents{timestamp: T + W + 5} is hashed into the block hash.
        The wrong block hash is written to storage.
        get_l1_prices_in_fri_and_wei(T + W + 5, ...) computes wrong L1 gas prices.
``` [10](#0-9) [11](#0-10)

    let block_commitments = &partial_block_hash_components.header_commitments;
    Ok(BlockHash(
        HashChain::new()
            .chain(&block_hash_version.clone().into())
            .chain(&partial_block_hash_components.block_number.0.into())
            .chain(&state_root.0)
            .chain(&partial_block_hash_components.sequencer.0)
            .chain(&partial_block_hash_components.timestamp.0.into())
            .chain(&block_commitments.concatenated_counts)

            .chain(&block_commitments.event_commitment.0)

                )
                .iter(),
            )
            .chain(

                    .expect("Expect ASCII version"),

            .chain(&Felt::ZERO)
            .chain(&previous_block_hash.0)

    fn validate_proof_block_hash(
        proof_block_hash: Felt,
        proof_block_number: u64,
        os_constants: &OsConstants,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        if proof_block_hash == Felt::ZERO {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof block hash is zero for block {proof_block_number}."
            )));
        }

        // Compare the proof's block hash with the stored block hash.
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();

        let stored_block_hash = state
            .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

        if stored_block_hash != proof_block_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Block hash mismatch for block {proof_block_number}. Proof block hash: \
                 {proof_block_hash}, stored block hash: {stored_block_hash}."
            )));
        }

        Ok(())
