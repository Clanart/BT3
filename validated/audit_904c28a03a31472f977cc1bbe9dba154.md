### Title
Precision Loss in `normalize_amount` Causes Permanent Lock of User Funds When Transfer Amount Is Below Decimal Scaling Factor - (File: near/omni-bridge/src/lib.rs)

### Summary
A user can initiate a bridge transfer with an amount that, after decimal normalization, rounds down to zero. The tokens are locked/burned at deposit time in `init_transfer_internal`, but `sign_transfer` later reverts with `ERR_INVALID_AMOUNT_TO_TRANSFER` every time it is called. Because no cancel or refund path exists for pending transfers, the user's funds are permanently irrecoverable.

### Finding Description

`init_transfer` (called via `ft_on_transfer`) locks or burns the user's tokens and stores the transfer in `pending_transfers` before any normalization check is performed. [1](#0-0) 

The only validation at deposit time is `fee.fee < amount`. There is no check that `normalize_amount(amount - fee) > 0`.

Later, when a trusted relayer calls `sign_transfer`, the normalized amount is computed: [2](#0-1) 

`normalize_amount` performs integer floor division: [3](#0-2) 

For a token registered with `origin_decimals = 24` and `decimals = 18`, the divisor is `10^6 = 1_000_000`. Any `amount_without_fee` smaller than `1_000_000` normalizes to `0`, causing `sign_transfer` to panic with `ERR_INVALID_AMOUNT_TO_TRANSFER` on every invocation.

The tokens were already locked or burned in `init_transfer_internal`: [4](#0-3) 

No `cancel_transfer` or user-callable refund function exists. The only unlock path is through `process_fin_transfer_to_near` or `process_fin_transfer_to_other_chain`, both of which require a valid proof of finalization on the destination chain — which can never be produced because `sign_transfer` always reverts. [5](#0-4) 

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds.**

- For **native (non-deployed) tokens**: the tokens are locked in the bridge contract via `lock_tokens_if_needed` and can never be unlocked because the unlock path requires a finalization proof that can never be generated.
- For **deployed (bridged) tokens**: the tokens are burned via `burn_tokens_if_needed` and are permanently destroyed.

In both cases the user loses their entire deposited amount with no recovery mechanism.

### Likelihood Explanation

**Low.** The user must send an amount smaller than the decimal scaling factor (e.g., less than `1_000_000` base units for a token with a 6-decimal gap). This is an unusual but reachable scenario — tokens with large decimal differences (e.g., 24 NEAR decimals bridged to an 18-decimal EVM token) are explicitly supported by the protocol, and a user sending a "dust" amount or making a mistake about the token's precision would trigger this silently.

### Recommendation

Add a normalization check inside `init_transfer` (before locking/burning tokens) to reject transfers whose net amount normalizes to zero:

```rust
let token_address = self.get_token_address(...);
let decimals = self.token_decimals.get(&token_address)...;
let normalized = Self::normalize_amount(
    transfer_message.amount_without_fee()...,
    decimals,
);
require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
```

Alternatively, add a user-callable `cancel_transfer` function that refunds the sender when a pending transfer has never been signed.

### Proof of Concept

1. A token is registered with `origin_decimals = 24`, `decimals = 18` (6-decimal gap, divisor = `1_000_000`).
2. User calls `ft_transfer_call` with `amount = 500_000` and `fee = 0`.
3. `init_transfer` passes the `fee < amount` check (0 < 500_000). Tokens are locked. Transfer stored in `pending_transfers`.
4. Trusted relayer calls `sign_transfer`.
5. `normalize_amount(500_000, {decimals:18, origin_decimals:24}) = 500_000 / 1_000_000 = 0`.
6. `require!(amount_to_transfer > 0, ...)` panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`.
7. Relayer retries — same result every time.
8. No `cancel_transfer` exists. The 500,000 tokens are permanently locked in the bridge with no recovery path. [2](#0-1) [3](#0-2) [6](#0-5)

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }

    fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
        let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
        amount / (10_u128.pow(diff_decimals))
    }
