### Title
Arithmetic Underflow in `check_proof_facts` Causes OS Proof Failure During Early Network Blocks - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

The `check_proof_facts` function performs an unchecked subtraction `current_block_number - STORED_BLOCK_HASH_BUFFER` without first verifying that `current_block_number >= STORED_BLOCK_HASH_BUFFER`. When `current_block_number < 10` (the first 10 blocks of any network deployment), this subtraction underflows in Cairo's prime field arithmetic, producing a value close to the Stark prime P (~2^251). The subsequent `assert_nn_le` range check then fails because the result exceeds 2^128, causing a hard OS execution failure. Since this is not a transaction revert but an OS-level assertion failure, the entire block proof cannot be generated, halting the network.

---

### Finding Description

In `execution_constraints.cairo`, `check_proof_facts` is called from `execute_invoke_function_transaction` whenever a user-submitted invoke transaction includes non-empty `proof_facts`:

```cairo
// execution_constraints.cairo, lines 66-68
assert_nn_le(
    os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
);
``` [1](#0-0) 

`STORED_BLOCK_HASH_BUFFER = 10`. When `current_block_number` is 0–9, the expression `current_block_number - STORED_BLOCK_HASH_BUFFER` wraps around in the prime field to a value near P ≈ 2^251. The `assert_nn_le(a, b)` call then checks that `b - a` is in `[0, 2^128)`, which fails because the wrapped value is far outside that range. This is a hard Cairo assertion failure — not a graceful transaction revert — and aborts the entire OS execution, making the block unprovable.

The `proof_facts` are user-supplied, loaded via the `%{ TxProofFacts %}` hint in `execute_invoke_function_transaction`:

```cairo
// transaction_impls.cairo, lines 279-281
local proof_facts_size;
local proof_facts: felt*;
%{ TxProofFacts %}
``` [2](#0-1) 

`check_proof_facts` is then called unconditionally (when `proof_facts_size > 0`):

```cairo
// transaction_impls.cairo, lines 313-318
check_proof_facts(
    proof_facts_size=proof_facts_size,
    proof_facts=proof_facts,
    current_block_number=block_context.block_info_for_execute.block_number,
    virtual_os_config_hash=block_context.os_global_context.virtual_os_config_hash,
);
``` [3](#0-2) 

The correct guard pattern already exists elsewhere in the codebase. In `write_block_number_to_block_hash_mapping`, the same subtraction is performed but is protected by an `is_nn` check before proceeding:

```cairo
// os_utils.cairo, lines 52-57
tempvar old_block_number = block_context.block_info_for_execute.block_number -
    STORED_BLOCK_HASH_BUFFER;
let is_old_block_number_non_negative = is_nn(old_block_number);
if (is_old_block_number_non_negative == FALSE) {
    return ();
}
``` [4](#0-3) 

This guard is absent in `check_proof_facts`. Similarly, `execute_get_block_hash` uses a hint-driven branch (`IsBlockNumberInBlockHashBuffer`) to route to a failure response before reaching the analogous subtraction:

```cairo
// syscall_impls.cairo, lines 745-753
if (is_block_number_in_block_hash_buffer != FALSE) {
    assert_lt(current_block_number, request_block_number + STORED_BLOCK_HASH_BUFFER);
    write_failure_response(...);
    return ();
}
assert_nn_le(request_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER);
``` [5](#0-4) 

No equivalent guard exists in `check_proof_facts`.

---

### Impact Explanation

**High. Network not being able to confirm new transactions (total network shutdown).**

When the OS assertion fails, the Cairo VM execution aborts at the OS level — this is not a transaction-level revert handled by the `is_reverted` flag. The block proof cannot be generated. Any block containing such a transaction during the first 10 blocks of a network deployment becomes unprovable, halting block finalization on L1 and preventing the network from confirming any new transactions.

---

### Likelihood Explanation

The vulnerability is exploitable during the first 10 blocks (`block_number` 0–9) of any StarkNet network deployment (mainnet genesis, new testnets, devnets, or network restarts). The attacker only needs to submit an invoke transaction with `proof_facts_size > 0`. The required fields (`PROOF_VERSION`, `VIRTUAL_SNOS`, `VIRTUAL_OS_OUTPUT_VERSION`, `ALLOWED_VIRTUAL_OS_PROGRAM_HASHES_0`) are all public constants in the codebase, making it straightforward to craft a transaction that passes the initial header checks and reaches the vulnerable subtraction. The window is narrow (10 blocks) but deterministic and fully attacker-controlled.

---

### Recommendation

Add a guard in `check_proof_facts` before the subtraction, mirroring the pattern used in `write_block_number_to_block_hash_mapping`:

```cairo
func check_proof_facts{range_check_ptr, contract_state_changes: DictAccess*}(...) {
    if (proof_facts_size == 0) {
        return ();
    }

    // Guard: if not enough blocks exist, no valid proof facts are possible.
    let is_block_number_valid = is_nn(current_block_number - STORED_BLOCK_HASH_BUFFER);
    if (is_block_number_valid == FALSE) {
        // Fail gracefully: proof facts cannot reference a stored block hash.
        assert 1 = 0;  // or write a failure response if this becomes a transaction-level check
        return ();
    }

    // ... rest of the function
    assert_nn_le(
        os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
    );
```

### Proof of Concept

1. A new StarkNet network is deployed (block number starts at 0).
2. During blocks 0–9, an attacker submits an invoke transaction with `proof_facts` crafted to pass the header checks:
   - `proof_header.program_hash = ALLOWED_VIRTUAL_OS_PROGRAM_HASHES_0`
   - `proof_header.proof_version = PROOF_VERSION`
   - `proof_header.proof_variant = VIRTUAL_SNOS`
   - `os_output_header.output_version = VIRTUAL_OS_OUTPUT_VERSION`
   - `os_output_header.base_block_number = 0` (any small value)
3. The sequencer includes the transaction in a block (e.g., block 5).
4. The prover runs the OS to generate a proof for the block.
5. `check_proof_facts` is called with `current_block_number = 5`, `STORED_BLOCK_HASH_BUFFER = 10`.
6. `5 - 10` underflows to `P - 5` in the prime field.
7. `assert_nn_le(0, P - 5)` calls `assert_le(0, P - 5)`, which checks that `(P - 5) - 0 = P - 5` is in `[0, 2^128)`. Since `P - 5 ≈ 2^251 >> 2^128`, the range check fails.
8. The OS execution aborts. The block proof cannot be generated. The network halts.
