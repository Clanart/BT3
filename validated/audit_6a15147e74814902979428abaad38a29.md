### Title
Rejected L1 Handler Transactions Silently Dropped in `CatchingUp` State, Corrupting L1 Provider Transaction State - (File: crates/apollo_l1_provider/src/l1_provider.rs)

### Summary

When `L1Provider` is in `CatchingUp` state, the `rejected_txs` argument passed to `commit_block` is silently discarded before reaching `accept_commit_while_catching_up`. As a result, every L1 handler transaction that was rejected during block execution is incorrectly promoted to `TransactionState::Committed` instead of `TransactionState::Rejected`. This corrupts the provider's authoritative view of which L1 handler transactions have been successfully committed to L2, and can cause a validator node to return `AlreadyIncludedOnL2` for a transaction that is legitimately validatable, causing it to reject an otherwise valid block proposal.

`apply_commit_block` partitions `consumed_txs` by whether they appear in `rejected_txs`. With an empty `rejected_txs`, every consumed L1 handler transaction — including those that failed execution — is placed in the `committed_txs` bucket and passed to `TransactionManager::commit_txs` as committed:

```rust

        consumed_txs.iter().copied().partition(|tx| rejected_txs.contains(tx));

The batcher then computes `rejected_l1_handler_tx_hashes` as the intersection of `rejected_tx_hashes` and `consumed_l1_handler_tx_hashes` and sends both to the L1 provider:

```rust
let rejected_l1_handler_tx_hashes = rejected_tx_hashes
    .iter()
    .copied()
    .filter(|tx_hash| consumed_l1_handler_tx_hashes.contains(tx_hash))
    .collect();

self.l1_provider_client
    .commit_block(consumed_l1_handler_tx_hashes, rejected_l1_handler_tx_hashes, height)
    .await;
``` [7](#0-6) 

The `rejected_l1_handler_tx_hashes` set is never stored in the backlog struct; only `committed_txs` and `height` are preserved, so the information is permanently lost.

### Impact Explanation

`TransactionState::Committed` makes a transaction non-validatable:

```rust
pub fn is_validatable(&self) -> bool {
    !self.is_committed() && !self.is_cancelled() && !self.is_consumed()
}
``` [8](#0-7) 

`TransactionState::Rejected` is validatable (it is not committed, cancelled, or consumed), so a validator node with a correctly-updated provider would return `ValidationStatus::Validated` for a rejected-then-re-proposed L1 handler transaction. A validator node whose provider was in `CatchingUp` state during the relevant `commit_block` call would instead return `InvalidValidationStatus::AlreadyIncludedOnL2`, causing it to fail block validation for a proposal that every correctly-updated node would accept. This is a wrong authoritative value returned by the L1 provider during the pre-sequencing validation path.

### Likelihood Explanation

The `CatchingUp` state is entered whenever the L1 provider receives a `commit_block` call at a height that differs from its `current_height` — a routine occurrence during node startup, restart, or any brief height desynchronisation between the batcher and the provider. Any block in that window that contains a rejected L1 handler transaction will silently corrupt the provider's state. The TODO comment at line 426 confirms the developers are aware the path is incomplete.

### Recommendation

Pass `rejected_txs` through the entire catching-up path:

1. Add `rejected_txs: IndexSet<TransactionHash>` to `accept_commit_while_catching_up`.
2. Store `rejected_txs` alongside `committed_txs` in the backlog entry struct.
3. Pass the stored `rejected_txs` to every `apply_commit_block` call inside the backlog-replay loop and the `Equal` branch.

### Proof of Concept

1. Start a node. The L1 provider initialises at height H and enters `Pending` state.
2. The batcher commits block H+2 before the provider has processed H+1, triggering `CatchingUp` state.
3. Block H+1 contains an L1 handler transaction `tx_A` that fails execution; the batcher sends `consumed = {tx_A}`, `rejected = {tx_A}` to the provider.
4. `accept_commit_while_catching_up` is called with only `committed_txs = {tx_A}`; `rejected_txs` is dropped.
5. `apply_commit_block({tx_A}, {})` marks `tx_A` as `Committed`.
6. A second node (proposer) whose provider correctly marks `tx_A` as `Rejected` (validatable) includes `tx_A` in block H+2.
7. The first node's validator calls `validate(tx_A)`, receives `AlreadyIncludedOnL2`, and rejects the block — a consensus divergence caused solely by the dropped `rejected_txs` in the catching-up path. [1](#0-0) [3](#0-2) [4](#0-3)
