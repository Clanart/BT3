### Title
Integer Wrapping in `check_time_locks` Renders `ASSERT_SECONDS_RELATIVE` / `ASSERT_HEIGHT_RELATIVE` Timelocks Permanently Ineffective When `nowrap=false` — (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary
When `check_time_locks` is invoked with `nowrap=false` (the pre-softfork legacy path), the relative timelock conditions `ASSERT_SECONDS_RELATIVE` and `ASSERT_HEIGHT_RELATIVE` are evaluated using `wrapping_add`. A CLVM puzzle that emits either condition with the maximum representable value (`u64::MAX` for seconds, `u32::MAX` for height) causes the computed deadline to wrap below the coin's own birth timestamp/height, making the guard permanently satisfiable at any block — an exact analog to the "infinite deadline" anti-pattern in the external report.

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` branches on the `nowrap` flag for all four relative timelock conditions. For `ASSERT_SECONDS_RELATIVE` the legacy branch is:

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [1](#0-0) 

The intent is: reject the spend if the current timestamp has not yet reached `coin_birth + seconds_relative`. With `wrapping_add`, however, a `seconds_relative` value of `u64::MAX` produces:

```
coin_timestamp.wrapping_add(u64::MAX)  =  coin_timestamp − 1
```

The guard then becomes `timestamp < coin_timestamp − 1`. Because a coin can only be spent after it is confirmed (`timestamp >= coin_timestamp`), this condition is **always false**, so the spend is always accepted regardless of the actual elapsed time.

The same arithmetic applies to `ASSERT_HEIGHT_RELATIVE` with `u32::MAX`:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
``` [2](#0-1) 

Crucially, `sanitize_uint` with `max_size = 8` **accepts** a 9-byte CLVM atom `0x00 FF FF FF FF FF FF FF FF` as `u64::MAX`, so the crafted value is reachable directly from a CLVM puzzle output — no out-of-band injection is required:

```rust
ASSERT_SECONDS_RELATIVE => {
    match sanitize_uint(a, node, 8, ...) {
        SanitizedUint::PositiveOverflow => Err(...),   // > 9 bytes → rejected
        SanitizedUint::NegativeOverflow => Ok(Condition::SkipRelativeCondition),
        SanitizedUint::Ok(r) => Ok(Condition::AssertSecondsRelative(r)),  // u64::MAX accepted
    }
}
``` [3](#0-2) 

The fuzz harness confirms that a 9-byte atom with a leading `0x00` byte is a valid `SanitizedUint::Ok` result: [4](#0-3) 

The `nowrap=false` path is the pre-softfork legacy behavior exposed through the Python binding: [5](#0-4) 

The existing test suite explicitly documents and confirms the bypass:

```

    Osc { seconds_relative: Some(0xffff_ffff_ffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed)),
    Ok(()),   // ← timelock bypassed
)]
``` [6](#0-5) 

### Impact Explanation

Any CLVM puzzle that emits `(ASSERT_SECONDS_RELATIVE 0x00FFFFFFFFFFFFFFFF)` or `(ASSERT_HEIGHT_RELATIVE 0x00FFFFFFFF)` will have its timelock silently nullified on nodes running the `nowrap=false` code path. This maps directly to the **High** allowed impact: *"timelock… condition… validation bypass enables unauthorized spend acceptance."* A coin that is supposed to be unspendable for a fixed duration can be spent immediately, undermining vault, escrow, and DeFi timelock contracts.

### Likelihood Explanation

- The crafted CLVM atom is a single 9-byte value, trivially constructable by any puzzle author or solution provider.
- The `nowrap=false` path is still reachable: it is a named parameter in the public Python binding (`py_check_time_locks`) and is the designated behavior for pre-softfork block validation.
- No privileged role, key material, or network-level capability is required — only the ability to submit a spend bundle containing a puzzle that outputs the crafted condition.

### Recommendation

1. **Deprecate and remove the `nowrap=false` path** in `check_time_locks`. All callers should pass `nowrap=true` (saturating arithmetic), which correctly rejects any `seconds_relative` or `height_relative` value that would overflow.
2. If backward compatibility with pre-softfork blocks is required, document the exact block-height boundary at which `nowrap=false` was last used and add a hard assertion that it is never passed for blocks above that height.
3. Consider adding a CLVM-level sanitization step that rejects `ASSERT_SECONDS_RELATIVE` values ≥ some practical maximum (e.g., 200 years in seconds) to prevent the overflow class entirely.
