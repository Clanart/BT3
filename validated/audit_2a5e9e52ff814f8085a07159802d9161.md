### Title
`batch_insert` Fast Path Bypasses `HashAlreadyPresent` Guard, Allowing Duplicate Leaf Hashes and Tree State Corruption — (`crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

The `batch_insert` fast path (lines 587–603) directly writes leaf nodes to the blob via `insert_entry_to_blob` without performing the duplicate-hash check that `insert()` enforces. This allows two leaf nodes with identical `Hash` values to coexist in the tree, silently corrupting the `leaf_hash_to_index` cache and violating the invariant that every leaf hash is unique.

---

### Finding Description

`insert()` guards against duplicate hashes at lines 372–374:

```rust
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` calls `insert()` (with its guard) only for the first two items when `leaf_count <= 1`:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { ... };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;
    }
}
``` [2](#0-1) 

All remaining items go through the fast path, which calls `insert_entry_to_blob` directly — **no key or hash uniqueness check**:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [3](#0-2) 

`insert_entry_to_blob` calls `add_leaf`, which uses `HashMap::insert` — silently overwriting any existing entry for the same hash:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index);  // silent overwrite
}
``` [4](#0-3) 

**Concrete call sequence on an empty blob:**

```
batch_insert([(key1,val1,H), (key2,val2,H), (key3,val3,H2)])
```

- `pop()` takes from the **end**: `(key3,val3,H2)` → `insert()` ✓, then `(key2,val2,H)` → `insert()` ✓ (H not yet present).
- Remaining `[(key1,val1,H)]` enters the fast path → written directly → `add_leaf` overwrites `leaf_hash_to_index[H]` from key2's index to key1's index.
- Both key1 and key2 now exist in the blob with hash H. The cache has `leaf_hash_to_index` size = 2, but `leaf_count` = 3.

`check_just_integrity` detects this via:

```rust
if leaf_count != leaf_hash_to_index_cache_length {
    return Err(Error::IntegrityLeafHashToIndexCacheLength(...));
}
``` [5](#0-4) 

However, `check_integrity_on_drop` is only `true` in test builds:

```rust
check_integrity_on_drop: cfg!(test),
``` [6](#0-5) 

In production, `check_integrity()` is **never called automatically** after `batch_insert`, so the corrupted state persists.

---

### Impact Explanation

After the corrupted `batch_insert`:

- `get_node_by_hash(H)` returns key1's `(key, value)` — the entry for key2 is silently lost from the cache. Any DataLayer operation relying on hash-to-node lookup (e.g., delta sync, proof verification by hash) returns stale or wrong data.
- The `leaf_hash_to_index` cache is permanently desynchronized from the blob until a full rebuild. Subsequent `delete(key2)` calls `remove_leaf` which removes `leaf_hash_to_index[H]` — but key1's blob entry still carries hash H, leaving a ghost entry in the blob with no cache backing.
- The Merkle root computed over the corrupted tree is a hash of a structurally invalid tree (two leaves sharing a hash), violating the DataLayer's integrity guarantee that the root authenticates a unique key→hash mapping.

Direct proof forgery via `ProofOfInclusion.valid()` is **not** achievable from this bug alone — the Merkle paths for key1 and key2 differ, so a proof for one cannot substitute for the other. The impact is **tree state corruption and cache desynchronization**, which falls under the High-impact category: *DataLayer Merkle proof/blob/delta logic corrupts tree roots or lets untrusted input prove invalid state*.

---

### Likelihood Explanation

The Python binding `py_batch_insert` accepts caller-supplied `hashes: Vec<Hash>` with no deduplication:

```rust
pub fn py_batch_insert(&mut self, keys_values: Vec<(KeyId, ValueId)>, hashes: Vec<Hash>) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
``` [7](#0-6) 

Any caller (local or via RPC) that supplies ≥3 entries where at least one hash duplicates an earlier entry in the same batch will trigger this path. No cryptographic collision is required — the attacker simply repeats a hash value in the input list. The bug is deterministic and locally reproducible.

---

### Recommendation

Add a duplicate-hash (and duplicate-key) check at the top of the fast-path loop in `batch_insert`, mirroring the guard in `insert()`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing fast-path logic
}
```

Alternatively, pre-validate the entire input batch for uniqueness before any mutation begins.

### Proof of Concept

```rust
use chia_datalayer::{Hash, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

let mut blob = MerkleBlob::new(vec![]).unwrap();
let h = Hash(Bytes32::new([0xAB; 32]));
let h2 = Hash(Bytes32::new([0xCD; 32]));

// 3-item batch: key1 and key2 share hash `h`
// pop() takes key3/h2 first, then key2/h — both via insert() with guard.
// key1/h goes through the unguarded fast path.
blob.batch_insert(vec![
    ((KeyId(1), ValueId(1)), h),   // fast path — no guard
    ((KeyId(2), ValueId(2)), h),   // insert() — guard passes (h not yet present)
    ((KeyId(3), ValueId(3)), h2),  // insert() — guard passes
])
.unwrap(); // succeeds — no error returned

// Integrity check reveals the corruption:
blob.check_integrity().expect_err("should fail: leaf_count=3 but leaf_hash_to_index_cache_length=2");
```
