### Title
Missing shard-position check in `set_state_header` allows cross-shard chunk substitution during state sync — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::set_state_header` verifies that the supplied chunk is *a member* of the block's `chunk_headers_root` Merkle tree, but never verifies that it occupies the leaf position that corresponds to the requested `shard_id`. An unprivileged state-sync peer can supply a `ShardStateSyncResponseHeader` containing the chunk for shard index 1 together with a valid Merkle proof for that chunk's position, while the syncing node stores the result under `StateHeaderKey(shard_id=0, sync_hash)`. Subsequent `set_state_finalize` then applies the wrong shard's transactions and receipts to shard 0's state, producing a `ChunkExtra.state_root` that diverges from the canonical value.

---

### Finding Description

**Entrypoint.** Any peer can send a `PeerMessage::VersionedStateResponse` containing a `ShardStateSyncResponseHeader`. The network layer forwards it without authentication to `ClientActor`, which calls `set_state_header(shard_id, sync_hash, header)`. [1](#0-0) 

**The missing guard.** Inside `set_state_header`, step 3a verifies chunk membership with:

```rust
if !verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
) { … }
``` [2](#0-1) 

`verify_path` is a pure membership check: it reconstructs the Merkle root from the supplied path and compares it to the committed root. It carries no information about which leaf index was proven. [3](#0-2) 

The position-aware variant `verify_path_with_index` exists and is used elsewhere (e.g., for erasure-coded chunk parts), but is not used here. [4](#0-3) 

There is no subsequent check of the form `chunk.shard_id() == shard_id` anywhere in `set_state_header`. The `shard_id` parameter is used only as the DB key and in the receipt-hash computation, never to constrain which leaf the chunk must occupy. [5](#0-4) 

**`validate_chunk_proofs` does not help.** It checks internal self-consistency of the chunk (hash, tx_root, outgoing_receipts_root) but never compares `chunk.shard_id()` against the requested `shard_id`. [6](#0-5) 

**Concrete 2-shard attack.** For a block with two shards, the `chunk_headers_root` Merkle tree has two leaves:

```
leaf[0] = hash(ChunkHashHeight(chunk0_hash, h0))   ← shard 0
leaf[1] = hash(ChunkHashHeight(chunk1_hash, h1))   ← shard 1
root    = combine_hash(leaf[0], leaf[1])
```

The proof for leaf[1] is `[{direction: Left, hash: leaf[0]}]`. An attacker supplies:
- `chunk` = the actual chunk for shard 1 (passes `validate_chunk_proofs`)
- `chunk_proof` = the valid proof for leaf[1]

`verify_path(root, proof_for_leaf1, ChunkHashHeight(chunk1_hash, h1))` evaluates to `combine_hash(leaf[0], leaf[1]) == root` → **true**.

The attacker also supplies:
- `prev_chunk_header` / `prev_chunk_proof` = shard 1's previous chunk with its valid proof (passes the step-3b `verify_path` for the same reason)
- `incoming_receipts_proofs` = the legitimate receipt proofs for shard 0 (publicly available from block data; validated against `ReceiptList(shard_id=0, receipts)` at line 488, which the attacker can satisfy)
- `state_root_node` = shard 1's state root node (valid against chunk1's `prev_state_root`)

All five validation steps pass. The header is stored under `StateHeaderKey(shard_id=0, sync_hash)`. [7](#0-6) 

**`set_state_finalize` applies the wrong data.** It retrieves the stored header, extracts chunk1's transactions and `prev_state_root`, and calls `apply_chunk` for `shard_uid` of shard 0 using shard 1's state and transactions. [8](#0-7) 

The resulting `ChunkExtra.state_root` for shard 0 diverges from the canonical value.

**`set_state_part` is also affected.** It reads the stored header to obtain `state_root`, then validates incoming parts against that root. The attacker supplies shard 1's state parts, which are valid against chunk1's `prev_state_root`, so they are accepted and installed as shard 0's state. [9](#0-8) 

---

### Impact Explanation

A syncing node that accepts the malicious header installs shard 1's state as shard 0's state. After `set_state_finalize`, the node holds a `ChunkExtra` for shard 0 whose `state_root` does not match the canonical chain. The node will subsequently fail to validate or produce blocks for shard 0, effectively causing a silent fork or a crash-loop for the affected shard.

---

### Likelihood Explanation

Any peer on the NEAR network can respond to a state-sync header request — no validator key or special privilege is required. The syncing node selects a provider from `snapshot_hosts` but does not authenticate the response cryptographically beyond the checks inside `set_state_header`. The attack requires only knowledge of the block's chunk data (publicly available) and the ability to construct a well-formed `ShardStateSyncResponseHeader`, which is straightforward given the block's Merkle tree.

---

### Recommendation

Replace the `verify_path` call for `chunk_proof` (and `prev_chunk_proof`) with a position-aware check. Concretely:

1. Resolve `shard_index = shard_layout.get_shard_index(shard_id)`.
2. Use `verify_path_with_index(root, proof, item, shard_index as u64, num_shards as u64)` instead of `verify_path`.

Alternatively, after the membership check passes, explicitly assert `chunk.shard_id() == shard_id` and reject the header if they differ. The same fix must be applied to the `prev_chunk_proof` verification.

---

### Proof of Concept

Test-loop test with 2 shards:

1. Run the chain for several blocks so both shards have committed chunks.
2. Obtain the legitimate `ShardStateSyncResponseHeader` for `shard_id=1` (via `get_state_response_header(shard_id=1, sync_hash)`).
3. Construct a malicious header by taking the shard-1 header and replacing only the `shard_id` label in the storage key — i.e., call `set_state_header(shard_id=0, sync_hash, header_for_shard_1)` directly.
4. Assert `set_state_header` returns `Ok(())`.
5. Provide shard-1 state parts via `set_state_part(shard_id=0, …)` (they validate against chunk1's `prev_state_root`).
6. Call `set_state_finalize(shard_id=0, sync_hash)`.
7. Assert the resulting `ChunkExtra.state_root` for shard 0 does not match the canonical `ChunkExtra` for shard 0 obtained from a correctly-synced node.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
