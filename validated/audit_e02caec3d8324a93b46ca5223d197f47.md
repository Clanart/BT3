### Title
Unchecked Field Arithmetic Overflow in `compute_max_possible_fee` Allows Complete Fee Bypass — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `compute_max_possible_fee` function in `transaction_impls.cairo` performs direct felt-field multiplication of `max_amount * max_price_per_unit` for each resource type without any upper-bound constraint on `max_price_per_unit`. Because Cairo arithmetic is modulo the STARK prime P ≈ 2^251, a user-controlled transaction can be crafted so that the sum of the three product terms wraps to exactly 0 (mod P), causing `charge_fee` to silently skip the ERC-20 fee transfer entirely.

---

### Finding Description

`compute_max_possible_fee` computes the fee cap as a plain felt sum of three products:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

The only upstream validation of these fields occurs in `pack_resource_bounds`, called during transaction-hash computation:

```cairo
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
assert_nn(resource_bounds.max_price_per_unit);
``` [2](#0-1) 

`assert_nn_le` bounds `max_amount` to `[0, 2^64 − 1]`. `assert_nn` only checks non-negativity in felt semantics, bounding `max_price_per_unit` to `[0, (P−1)/2]` ≈ `[0, 2^250]`. No check constrains the *product* `max_amount × max_price_per_unit`.

With `max_amount` up to 2^64 and `max_price_per_unit` up to 2^250, each product can reach ~2^314, wrapping around P (~2^251) many times. The sum of three such products can be made to equal 0 (mod P) by a user who controls all six fields.

`charge_fee` then short-circuits on the zero result:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
``` [3](#0-2) 

The ERC-20 transfer to the sequencer is never executed, and the OS proof remains valid for a block containing a fee-free transaction.

---

### Impact Explanation

**Critical — Direct loss of funds.**

The fee token (STRK) that the protocol requires to be transferred to the sequencer for every V3 transaction is never transferred. The OS Cairo program — which is what gets proven — accepts the block as valid. Any sequencer that includes such a transaction produces a provably correct block in which the fee accounting is silently wrong. The sequencer loses the fee revenue it is entitled to, and the attacker's transaction is executed at zero cost.

---

### Likelihood Explanation

A V3 transaction sender controls all six resource-bound fields (`max_amount` and `max_price_per_unit` for L1 gas, L2 gas, and L1 data gas) plus `tip`. Constructing a wrapping combination requires only elementary modular arithmetic. A concrete example:

| Resource | `max_amount` | `max_price_per_unit` |
|---|---|---|
| L1 gas | 2 | (P−1)/2 |
| L2 gas | 1 | 1 |
| L1 data gas | 0 | 0 |
| tip | 0 | — |

Computation: `2 × (P−1)/2 + 1 × 1 = (P−1) + 1 = P ≡ 0 (mod P)`.

All individual field checks pass (`max_amount ≤ 2^64−1`; `max_price_per_unit = (P−1)/2` passes `assert_nn`). The transaction appears to offer an extremely high L1 gas price, which may cause a sequencer to prioritise it. The wrapping is invisible to any check that does not explicitly bound the product.

---

### Recommendation

Add an explicit upper-bound check on `max_price_per_unit` in `pack_resource_bounds` (e.g., `assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1)`) so that each product fits within 192 bits and the three-term sum fits within a felt without wrapping. Alternatively, rewrite `compute_max_possible_fee` to perform saturating or checked arithmetic and assert that the result is non-negative and within a known safe range before using it as a fee cap.

---

### Proof of Concept

1. Craft a V3 `invoke` transaction with:
   - `resource_bounds[L1_GAS] = { max_amount: 2, max_price_per_unit: (P−1)/2 }`
   - `resource_bounds[L2_GAS] = { max_amount: 1, max_price_per_unit: 1 }`
   - `resource_bounds[L1_DATA_GAS] = { max_amount: 0, max_price_per_unit: 0 }`
   - `tip = 0`
2. Submit to the sequencer. All field-level checks in `pack_resource_bounds` pass.
3. The sequencer includes the transaction (it appears to offer a very high L1 gas price).
4. The OS executes `compute_max_possible_fee`:
   - `2 × (P−1)/2 = P−1 ≡ −1 (mod P)`
   - `1 × (1 + 0) = 1`
   - Sum = `−1 + 1 = 0 (mod P)`
5. `charge_fee` sees `max_fee == 0` and returns immediately — no STRK is transferred.
6. The OS proof is generated and verified as valid for a block in which the transaction paid zero fees. [4](#0-3) [5](#0-4)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
