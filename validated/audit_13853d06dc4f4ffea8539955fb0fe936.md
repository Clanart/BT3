### Title
Missing Zero-Address Validation for `nearBridgeDerivedAddress_` in Initializer Permanently Freezes Bridge Finalization — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The `OmniBridge.initialize()` function assigns `nearBridgeDerivedAddress_` directly to the `nearBridgeDerivedAddress` storage variable without any zero-address check. This variable is the sole authority used to verify MPC/bridge signatures in both `finTransfer()` and `deployToken()`. If it is initialized to `address(0)`, every call to those two functions will permanently revert with `InvalidSignature`, freezing all incoming bridge settlements and token deployments. No setter exists in the current implementation to correct the value post-deployment.

---

### Finding Description

`OmniBridge.initialize()` stores three constructor-equivalent parameters:

```solidity
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
) public initializer {
    tokenImplementationAddress = tokenImplementationAddress_;
    nearBridgeDerivedAddress = nearBridgeDerivedAddress_;   // ← no zero check
    ...
}
``` [1](#0-0) 

`nearBridgeDerivedAddress` is then used as the expected signer in both critical bridge operations:

**`deployToken()`** (line 151):
```solidity
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [2](#0-1) 

**`finTransfer()`** (line 311):
```solidity
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [3](#0-2) 

OpenZeppelin's `ECDSA.recover` reverts on malformed signatures and never returns `address(0)` for a well-formed one. Therefore, if `nearBridgeDerivedAddress == address(0)`, the inequality `recovered != address(0)` is always `true` for any valid signature, causing both functions to always revert.

By contrast, `tokenImplementationAddress` has a defensive runtime guard added inside `deployToken()`:
```solidity
if (tokenImplementationAddress == address(0)) {
    revert TokenImplementationNotSet();
}
``` [4](#0-3) 

No equivalent guard exists for `nearBridgeDerivedAddress`, and no setter function for it appears anywhere in `OmniBridge.sol` or `OmniBridgeWormhole.sol`. The `initializer` modifier prevents `initialize()` from being called again.

`OmniBridgeWormhole.initializeWormhole()` delegates directly to `initialize()` and inherits the same gap: [5](#0-4) 

---

### Impact Explanation

If `nearBridgeDerivedAddress` is initialized to `address(0)`:

- **`finTransfer()` is permanently broken**: no cross-chain transfer can ever be finalized on the EVM side. Tokens locked or burned on NEAR/Solana/StarkNet cannot be released or minted on EVM — a permanent, irrecoverable freeze of user funds in the bridge flow.
- **`deployToken()` is permanently broken**: no new bridged token can be deployed, blocking the entire token-deployment pipeline.
- Recovery requires a UUPS implementation upgrade plus a new migration/setter function — a non-trivial operational emergency with no in-protocol self-healing path.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

The `initialize()` function is called exactly once, at deployment time, typically via a deployment script. A scripting error, copy-paste mistake, or misconfigured environment variable that supplies `address(0)` for `nearBridgeDerivedAddress_` would silently succeed — the transaction does not revert — and the bridge would be deployed in a permanently broken state. The absence of any on-chain guard means the error is not caught until the first `finTransfer` or `deployToken` call fails in production.

---

### Recommendation

Add an explicit zero-address check in `initialize()` for `nearBridgeDerivedAddress_`:

```solidity
function initialize(
    address tokenImplementationAddress_,
    address nearBridgeDerivedAddress_,
    uint8 omniBridgeChainId_
) public initializer {
    require(nearBridgeDerivedAddress_ != address(0), "ERR_ZERO_BRIDGE_ADDRESS");
    require(tokenImplementationAddress_ != address(0), "ERR_ZERO_TOKEN_IMPL");
    tokenImplementationAddress = tokenImplementationAddress_;
    nearBridgeDerivedAddress = nearBridgeDerivedAddress_;
    ...
}
```

Additionally, consider adding a privileged setter for `nearBridgeDerivedAddress` (guarded by `DEFAULT_ADMIN_ROLE`) to allow correction without a full implementation upgrade.

---

### Proof of Concept

1. Deploy `OmniBridgeWormhole` proxy and call `initializeWormhole` with `nearBridgeDerivedAddress = address(0)`.
2. The transaction succeeds; `nearBridgeDerivedAddress` is stored as `address(0)`.
3. Any user calls `finTransfer(validSignature, validPayload)`.
4. `ECDSA.recover(hashed, validSignature)` returns the legitimate MPC signer address (non-zero).
5. `recovered != address(0)` → `true` → `revert InvalidSignature()`.
6. All `finTransfer` and `deployToken` calls revert indefinitely. No user funds bridged from NEAR can be claimed on EVM. The bridge is frozen until an emergency upgrade is executed.
