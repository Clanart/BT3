### Title
Cross-Chain Replay of `MetadataPayload` MPC Signature Enables Unauthorized Token Deployment on Any Bridge Chain - (File: `starknet/src/bridge_types.cairo`, `evm/src/omni-bridge/contracts/OmniBridge.sol`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

### Summary
The `MetadataPayload` signed message used by `deploy_token` / `deployToken` across all bridge chains (EVM, Starknet, Solana) does not include a destination chain ID. A single NEAR MPC signature authorizing token deployment on one chain is structurally identical to a signature for any other chain, allowing any observer to replay it on every other supported chain without NEAR's authorization.

### Finding Description
The `MetadataPayload` Borsh-encoded message that NEAR MPC signs for token deployment is constructed identically on all three destination chains:

    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);

`deploy_token` has no equivalent chain-ID binding anywhere in the signing path.

### Impact Explanation
An attacker who observes a valid `deployToken` transaction on any chain (all calldata is public) can extract the MPC signature and replay it on every other supported chain. The result is that a token is deployed on chains NEAR never authorized. Once deployed, the token is registered in the bridge's canonical mapping (`nearToEthToken`, `near_to_starknet_token`, Solana wrapped-mint PDA). Users who subsequently call `init_transfer` on the unauthorized chain will have their tokens burned or locked, but NEAR's bridge infrastructure — which did not authorize that chain for that token — will not process the corresponding finalization. Those user funds are permanently irrecoverable. This matches the **Critical** impact class: permanent freezing / irrecoverable lock of user funds in bridge flows, and the **High** impact class: signature verification bypass enabling unauthorized token deployment.

### Likelihood Explanation
The attack requires no special privileges. All `deployToken` calldata (including the MPC signature) is publicly visible on-chain the moment the legitimate deployment transaction is confirmed. Any observer can extract the signature and submit it to the other chains' bridge contracts. The only prerequisite is that the token has not yet been deployed on the target chain, which is the normal state before the legitimate deployment reaches that chain. The attack window is the period between the first legitimate deployment on one chain and the operator's own deployment on the remaining chains.

### Recommendation
Include the destination chain ID in the `MetadataPayload` Borsh encoding, exactly as is already done for `TransferMessagePayload`. Concretely:

- **EVM**: add `bytes1(omniBridgeChainId)` to the `borshEncoded` concatenation in `deployToken`.
- **Starknet**: add a `chain_id: u8` parameter to `MetadataPayload.to_borsh()` and append it, then pass `self.omni_bridge_chain_id.read()` at the call site in `deploy_token`.
- **Solana**: write `SOLANA_OMNI_BRIDGE_CHAIN_ID` into `DeployTokenPayload.serialize_for_near()`.

NEAR MPC must then sign a separate payload for each destination chain, making signatures non-transferable across chains.

### Proof of Concept
1. NEAR MPC signs a `MetadataPayload` for token `"usdc.near"` (name `"USD Coin"`, symbol `"USDC"`, decimals `6`) to authorize deployment on EVM.
2. A relayer submits the signature to EVM's `deployToken`. The token is deployed and the transaction is confirmed on-chain with the full signature in calldata.
3. An attacker reads the signature from the EVM transaction.
4. The attacker calls Starknet's `deploy_token` with the identical `MetadataPayload` and the extracted signature. `_verify_borsh_signature` computes `keccak(Metadata || "usdc.near" || "USD Coin" || "USDC" || 6)` — the same hash as on EVM — and the signature verifies successfully.
5. `"usdc.near"` is now registered in `near_to_starknet_token` on Starknet. Users who call `init_transfer` on Starknet for this token have their tokens burned, but NEAR never authorized Starknet for `"usdc.near"` and will not finalize those transfers. Funds are permanently lost.
6. The same signature can simultaneously be replayed on Solana's `deploy_token`, registering the token there as well.

    fn _verify_borsh_signature(
        ref self: ContractState, borsh_bytes: @ByteArray, signature: Signature,
    ) {
        let message_hash_le = compute_keccak_byte_array(borsh_bytes);
        let message_hash = reverse_u256_bytes(message_hash_le);

        let sig = signature_from_vrs(signature.v, signature.r, signature.s);
        verify_eth_signature(message_hash, sig, self.omni_bridge_derived_address.read());
    }
