### Title
Missing Destination-Chain Token Validation in `init_transfer` Causes Permanent Fund Loss - (File: near/omni-bridge/src/lib.rs)

### Summary
The `init_transfer` function in the NEAR OmniBridge contract accepts any non-NEAR destination chain encoded in the user-supplied `recipient` address without verifying that the transferred token is registered for that chain. Tokens are burned (for deployed/bridged tokens) or locked (for native tokens) immediately at transfer initiation. Because `sign_transfer` will always panic when no token address exists for the specified chain, the transfer is permanently stuck with no on-chain recovery path.

### Finding Description
In `init_transfer` (called via `ft_on_transfer`), the only destination-chain check is:

```rust
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
``` [1](#0-0) 

There is no check that `(destination_chain, token_id)` exists in `token_id_to_address`. Execution then proceeds to `init_transfer_internal`, which:

1. Burns the full amount immediately for deployed (bridged) tokens via `burn_tokens_if_needed`: [2](#0-1) 

2. Attempts to update the `locked_tokens` accounting via `lock_tokens_if_needed`. If the `(destination_chain, token_id)` key is absent from `locked_tokens`, `lock_tokens` silently returns `LockAction::Unchanged` — the accounting is skipped but the transfer message is still committed: [3](#0-2) 

3. Stores the transfer message in `pending_transfers`: [4](#0-3) 

Later, when a trusted relayer calls `sign_transfer`, it calls `get_token_address(destination_chain, token_id)` and panics unconditionally if the mapping is absent:

```rust
let token_address = self
    .get_token_address(
        transfer_message.get_destination_chain(),
        self.get_token_id(&transfer_message.token),
    )
    .unwrap_or_else(|| {
        env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
    });
``` [5](#0-4) 

`get_token_address` is a pure lookup with no fallback: [6](#0-5) 

There is no public `cancel_transfer` or user-callable refund function. `remove_transfer_message` is only called internally during successful finalization flows: [7](#0-6) 

### Impact Explanation
- **Deployed/bridged tokens**: Burned immediately and irrecoverably at `init_transfer_internal` line 1851. The burn is fire-and-forget (`.detach()`), so it cannot be rolled back.
- **Native NEAR tokens**: Transferred to the bridge contract via `ft_transfer_call` and held there permanently. The transfer message sits in `pending_transfers` indefinitely because `sign_transfer` always panics for an unsupported chain, and no public cancellation path exists.

Both cases match the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds**.

### Likelihood Explanation
The `OmniAddress` type supports multiple chain variants (`Eth`, `Sol`, `Strk`, `Btc`, `Near`). A user can construct a syntactically valid recipient address on any of these chains. If a token is registered only for Ethereum but a user specifies a Solana recipient, the transfer proceeds through `init_transfer` without error, burns/locks the tokens, and then becomes permanently unresolvable. This is reachable by any unprivileged user via the standard `ft_transfer_call` → `ft_on_transfer` → `init_transfer` path with no special access required. [8](#0-7) 

### Recommendation
Add a validation step in `init_transfer` (before burning or locking) that checks whether the token has a registered address on the destination chain:

```rust
require!(
    self.get_token_address(
        init_transfer_msg.get_destination_chain(),
        token_id.clone(),
    ).is_some(),
    BridgeError::TokenNotSupportedOnDestinationChain.as_ref()
);
```

This mirrors the `enabledDepositRoutes` pattern from Across V2 and ensures that tokens are only burned/locked when a valid delivery path exists on the destination chain.

### Proof of Concept
1. Token `usdc.near` is registered only for `ChainKind::Eth` (entry in `token_id_to_address` for `(Eth, usdc.near)`).
2. User calls `ft_transfer_call` on `usdc.near` with `receiver_id = omni_bridge.near`, `amount = 1000`, and `msg = InitTransfer { recipient: OmniAddress::Sol(<valid_sol_address>), fee: 0, ... }`.
3. `ft_on_transfer` → `init_transfer`: check passes (`Sol != Near`). `init_transfer_internal` is called.
4. If `usdc.near` is a deployed token: `burn_tokens_if_needed` burns 1000 tokens immediately (detached promise, irreversible).
5. Transfer message stored in `pending_transfers` with `destination_chain = Sol`.
6. Relayer calls `sign_transfer` → `get_token_address(Sol, usdc.near)` returns `None` → panic with `FailedToGetTokenAddress`.
7. No public function can remove the pending transfer or recover the burned tokens. Funds are permanently lost.
