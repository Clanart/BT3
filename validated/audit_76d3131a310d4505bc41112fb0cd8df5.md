### Title
Unauthenticated Bootstrap Path in `execute_declare_transaction` Allows Arbitrary Class Declaration Without Signature Verification - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The StarkNet OS `execute_declare_transaction` function contains an unauthenticated "bootstrap" code path that completely bypasses signature validation, nonce checking, and fee charging. Any party who can include a transaction in a block — including a malicious or compromised sequencer, or any block proposer in a decentralized StarkNet — can declare an arbitrary class hash with an attacker-controlled `compiled_class_hash`, permanently corrupting the class registry for that class hash. Contracts that subsequently use the hijacked class hash will execute the attacker's malicious compiled class, leading to direct loss of funds.

---

### Finding Description

In `execute_declare_transaction`, lines 761–776 implement a special "bootstrap" path:

```cairo
// Do not run validate or perform any account-related actions for declare transactions that
// meet the following conditions.
// This flow is used for the sequencer to bootstrap a new system.
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        // Declare the class hash and skip the rest of the transaction.
        // Note that prev_value=0 enforces that a class may be declared only once.
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

When the four conditions are satisfied — `sender_address == 'BOOTSTRAP'` (a specific felt value), `nonce == 0`, `version == 3`, and all resource bounds zero — the OS:

1. **Skips `run_validate`** — no `__validate_declare__` entry point is called, so no signature is verified.
2. **Skips `check_and_increment_nonce`** — no nonce is consumed from any account state.
3. **Skips `charge_fee`** — no fee is deducted.
4. **Directly writes to `contract_class_changes`** — the class hash is registered with the attacker-supplied `compiled_class_hash`.

The normal declare path, by contrast, always runs `__validate_declare__` and increments the nonce:

```cairo
// Increment nonce.
check_and_increment_nonce(tx_info=tx_info);
// ...
// Run the account contract's "__validate_declare__" entry point.
let (retdata_size, retdata, is_deprecated) = non_reverting_select_execute_entry_point_func(
    block_context=block_context, execution_context=validate_declare_execution_context
);
``` [2](#0-1) 

The `sender_address` field is a plain `felt` value. The string literal `'BOOTSTRAP'` is its ASCII big-endian encoding — a specific felt constant. There is **no on-chain verification** that any deployed account contract exists at that address, no cryptographic proof of authority, and no relationship between the sender and the class being declared. The only guard is the magic constant itself, which is publicly visible in the OS source code.

The `prev_value=0` constraint at line 771 enforces that a class may be declared only once:

```cairo
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [3](#0-2) 

This means the attacker can **permanently and irreversibly** bind any class hash to a malicious `compiled_class_hash`. Once done, the legitimate class can never be declared.

---

### Impact Explanation

**Critical — Direct loss of funds.**

The class registry (`contract_class_changes`) is the authoritative mapping from Sierra class hash to compiled (CASM) class hash. Every contract deployment and `replace_class` syscall relies on this mapping to determine which bytecode to execute. If an attacker front-runs a legitimate class declaration and registers a malicious `compiled_class_hash` for a given class hash:

- Any contract subsequently deployed with that class hash will execute the attacker's malicious compiled class.
- Any existing contract that calls `replace_class` to upgrade to that class hash will execute the attacker's code.
- The attacker's compiled class can drain storage, redirect transfers, or perform arbitrary state mutations — resulting in direct loss of user funds.
- The corruption is **permanent**: `prev_value=0` means the legitimate class can never overwrite the malicious one.

---

### Likelihood Explanation

**Medium.**

- The conditions to trigger the bootstrap path are fully public and trivially constructable: `sender_address = felt('BOOTSTRAP')`, `nonce = 0`, `version = 3`, all resource bounds = 0.
- In the current centralized StarkNet architecture, the sequencer's gateway performs off-chain validation (signature check, account existence). A gateway that correctly rejects transactions from non-existent accounts would block this at ingestion. However, the OS program itself — the ground truth for proof validity — has no such check.
- A **malicious or compromised sequencer** can bypass gateway validation and directly include such a transaction in a block. The resulting proof would be valid and accepted by L1.
- In a **decentralized StarkNet** (the stated roadmap), any block proposer can include this transaction. The OS program is the only enforcement layer, and it does not enforce authority.
- The attacker needs no funds, no deployed contract, and no cryptographic key — only the ability to get a transaction included in a block.

---

### Recommendation

Remove the unauthenticated bootstrap path entirely, or gate it behind a verifiable on-chain authority check. If bootstrap functionality is genuinely required for system initialization, it must be controlled by a privileged, verifiable identity — for example, by requiring the `sender_address` to be a specific well-known contract address that is itself deployed and controlled by the protocol, and by verifying a valid signature from that contract's `__validate_declare__` entry point. The current check on a magic string constant provides no security guarantee.

A minimal fix is to require that the account at `sender_address == 'BOOTSTRAP'` actually exists in `contract_state_changes` with a non-zero class hash, and that `__validate_declare__` is still executed. Alternatively, remove the bootstrap path and handle system initialization through a separate, privileged L1→L2 message mechanism.

### Proof of Concept

An attacker constructs a declare transaction with the following fields:

| Field | Value |
|---|---|
| `sender_address` | `felt('BOOTSTRAP')` = `0x424F4F545354524150` |
| `nonce` | `0` |
| `version` | `3` |
| `l1_gas.max_amount` | `0` |
| `l2_gas.max_amount` | `0` |
| `l1_data_gas.max_amount` | `0` |
| `class_hash` | Target class hash (e.g., a popular token contract class) |
| `compiled_class_hash` | Attacker-controlled malicious CASM hash |

The attacker submits this transaction to a sequencer (or, in a decentralized setting, proposes a block containing it). The OS processes it via `execute_declare_transaction`:

1. `sender_address == 'BOOTSTRAP'` → **true**
2. `tx_info.nonce == 0` → **true**
3. `tx_info.version == 3` → **true**
4. `compute_max_possible_fee(tx_info) == 0` → **true** (all bounds are zero)

The OS enters the bootstrap branch, skips all validation, and writes `compiled_class_hash = <malicious>` for the target class hash into `contract_class_changes`. The proof is generated and submitted to L1. The class registry is now permanently corrupted. Any subsequent deployment or `replace_class` using that class hash executes the attacker's malicious compiled class. [4](#0-3)
