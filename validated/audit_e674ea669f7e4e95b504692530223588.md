After thorough analysis of the scoped files, I identified one analog to the external report's vulnerability class.

---

### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the caller-supplied class hash corresponds to a previously declared contract class. This is the direct analog of the proxy report's root cause: just as the `TokenProxy` accepted an implementation address without verifying it was initialized, the OS accepts an arbitrary class hash without verifying it is declared. A contract can replace its own class hash with any felt value, committing an invalid class hash to the global state tree. Any funds held by that contract become permanently inaccessible.

---

### Finding Description

`execute_replace_class` in `syscall_impls.cairo` reads the new class hash directly from the attacker-controlled syscall request and writes it into the contract's `StateEntry` with no validation:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 is an explicit developer acknowledgment that the check is absent. The OS therefore produces a valid proof for a state transition that sets a contract's class hash to an undeclared value.

When any subsequent block calls that contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
``` [2](#0-1) 

Because the class was never declared, `dict_read` returns `0` (`UNINITIALIZED_CLASS_HASH`). The OS then calls `find_element` to locate a `CompiledClassFact` with hash `0`:

```cairo
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
``` [3](#0-2) 

`find_element` is a non-optional search; if no matching element exists, the Cairo program fails. The OS cannot generate a valid proof for any block containing a call to the affected contract. The contract's storage and funds are permanently inaccessible.

The `UNINITIALIZED_CLASS_HASH = 0` constant and the `StateEntry` struct confirm that `0` is the sentinel for an undeployed contract: [4](#0-3) 

The `deploy_contract` function enforces `state_entry.class_hash = UNINITIALIZED_CLASS_HASH` before deployment, confirming that `0` is not a valid live class hash: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A contract that holds ERC-20 balances or acts as a vault can have its class hash replaced with an arbitrary undeclared value. Once the state commitment is finalized on L1 with this invalid class hash, no subsequent block can execute any entry point of that contract. The OS will fail to prove any block that touches the contract, making the funds permanently inaccessible. The state commitment integrity is also violated: the global state root encodes a class hash that has no corresponding compiled class, breaking the invariant that every live contract's class hash is declared.

---

### Likelihood Explanation

The `replace_class` syscall is callable by any Sierra contract from within its own execution context. An attacker needs only to:
1. Deploy a contract (or target an existing one they control) that holds funds.
2. Issue a transaction that calls `replace_class` with an arbitrary undeclared felt value.

The OS performs no validation of the class hash. The missing check is explicitly marked as a TODO, confirming it is not a deliberate design choice but an unimplemented guard. The exploit requires no privileged role, no leaked key, and no malicious operator — only the ability to submit a standard transaction.

---

### Recommendation

In `execute_replace_class`, before updating `contract_state_changes`, verify that `class_hash` is present in `contract_class_changes` (i.e., was declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` with `key=class_hash` and assert the result is non-zero (i.e., `!= UNINITIALIZED_CLASS_HASH`). This mirrors the check that `deploy_contract` already performs on the initial class hash at deployment time.

---

### Proof of Concept

1. Attacker deploys contract `C` (e.g., a vault holding user funds) with a valid declared class hash `H`.
2. Attacker submits an invoke transaction that causes `C` to execute the `replace_class` syscall with `class_hash = 0xDEAD` (an arbitrary undeclared felt).
3. The OS processes `execute_replace_class`: no validation is performed (line 898 TODO), and `C`'s `StateEntry` is updated to `class_hash=0xDEAD`.
4. The state commitment is finalized on L1 with `C` having class hash `0xDEAD`.
5. In any subsequent block, a call to `C` causes `execute_entry_point` to call `dict_read(key=0xDEAD)` on `contract_class_changes`, returning `0`.
6. `find_element` searches for a `CompiledClassFact` with hash `0` — none exists — and the Cairo program fails.
7. The OS cannot produce a valid proof for any block touching `C`. All funds in `C` are permanently frozen.
