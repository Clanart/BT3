### Title
Missing Upper Bound on `max_price_per_unit` Causes OS-Level Panic in Fee Computation — (`execution/transaction_impls.cairo`)

### Summary

`compute_max_possible_fee` multiplies `max_amount` (bounded to `2^64 - 1`) by `max_price_per_unit` (only checked to be non-negative, i.e., `< 2^128`) in the Cairo field. The product can reach `≈ 2^192`, which exceeds the `2^128` range-check bound. The subsequent `assert_nn_le(actual_fee, max_fee)` in `charge_fee` then panics the OS, invalidating the entire block proof.

### Finding Description

In `pack_resource_bounds` (called during transaction hash computation), the OS enforces:

```cairo
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // max_amount ≤ 2^64 - 1
assert_nn(resource_bounds.max_price_per_unit);            // max_price_per_unit ≥ 0 (< 2^128)
``` [1](#0-0) 

`max_amount` is tightly bounded to `2^64 - 1`, but `max_price_per_unit` is only checked to be non-negative — meaning it can be any value up to `2^128 - 1`. The `tip` field is bounded to `2^64 - 1`. [2](#0-1) 

Later, `compute_max_possible_fee` computes:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
``` [3](#0-2) 

With `max_amount = 2^64 - 1` and `max_price_per_unit = 2^128 - 1`, the product is `≈ 2^192`. Since the Cairo field prime `P ≈ 2^251`, no field-level wrapping occurs — the result is a valid felt `≈ 2^192`.

This `max_fee` value is then used in `charge_fee`:

```cairo
assert_nn_le(calldata.amount.low, max_fee);
``` [4](#0-3) 

`assert_nn_le(a, b)` internally calls `assert_nn(b - a)`, which range-checks that `b - a < 2^128`. With `max_fee ≈ 2^192` and `actual_fee < 2^128`, the value `max_fee - actual_fee ≈ 2^192` fails the range check. This is an OS-level assertion failure — not a transaction revert — causing the entire block proof to be invalid.

### Impact Explanation

An OS-level panic in `charge_fee` (which is OS code, not user contract code) invalidates the entire block proof. The sequencer cannot finalize the block and must re-run block production excluding the offending transaction. If the sequencer's pre-inclusion simulation does not catch this condition, an attacker can repeatedly trigger block proof failures, preventing the network from confirming new transactions. This matches **High: Network not being able to confirm new transactions**.

### Likelihood Explanation

Any unprivileged user can submit a V3 transaction with `max_price_per_unit` set to a value close to `2^128 - 1`. The OS's own hash-verification path (`pack_resource_bounds`) does not reject such a value — it only calls `assert_nn`, not `assert_nn_le(..., 2^64 - 1)`. If the sequencer's mempool validation does not independently enforce a tighter bound on `max_price_per_unit`, the transaction will be included in a block and trigger the panic. The OS code is the authoritative validator and should not rely on external pre-filtering.

### Recommendation

In `pack_resource_bounds`, add an upper bound check on `max_price_per_unit` analogous to the one on `max_amount`:

```cairo
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);  // existing
// Add:
assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 64 - 1);
```

This ensures `max_amount * max_price_per_unit ≤ (2^64 - 1)^2 < 2^128`, keeping `compute_max_possible_fee`'s result within the range-check domain and preventing the OS panic.

Alternatively, `compute_max_possible_fee` should use a `Uint256` accumulator and validate the result fits in `2^128` before using it in `assert_nn_le`.

### Proof of Concept

1. Craft a V3 transaction with:
   - `l1_gas_bounds.max_amount = 2^64 - 1`
   - `l1_gas_bounds.max_price_per_unit = 2^128 - 1`
   - All other resource bounds set to 0.
2. Submit to the sequencer. The transaction passes `pack_resource_bounds` (only `assert_nn` is checked for price).
3. Sequencer includes the transaction in a block and runs the OS.
4. `compute_max_possible_fee` returns `(2^64 - 1) * (2^128 - 1) ≈ 2^192`.
5. `charge_fee` calls `assert_nn_le(actual_fee, 2^192)`.
6. Internally, `assert_nn(2^192 - actual_fee)` is called; `2^192 - actual_fee > 2^128` fails the range check.
7. OS panics. Block proof is invalid. Block cannot be finalized. [5](#0-4) [6](#0-5) [1](#0-0)

// Returns the maximum possible fee that can be charged for the transaction.
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```
