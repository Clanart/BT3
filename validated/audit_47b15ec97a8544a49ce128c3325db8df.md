### Title
`ProofOfInclusion::valid()` Tautological Root Check Allows Forged DataLayer Inclusion Proofs - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` only verifies the internal self-consistency of a proof chain. Its final root-equality check is a tautology — it always evaluates to `true` if the loop completes — because `root_hash()` is derived from the same `combined_hash` field that the loop already verified. An untrusted party can supply any internally-consistent `ProofOfInclusion` (including a zero-layer proof for an arbitrary `node_hash`) and `valid()` will return `true`, without the proof corresponding to any real tree state.

### Finding Description

`ProofOfInclusion::valid()` is defined as:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()   // ← always true
}
``` [1](#0-0) 

`root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← same field the loop just verified
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

After the loop, `existing_hash` equals `last_layer.combined_hash` (because the loop would have returned `false` otherwise). `root_hash()` also returns `last_layer.combined_hash`. Therefore `existing_hash == self.root_hash()` is unconditionally `true` whenever the loop completes. The final check adds no information.

For the zero-layer case: `existing_hash = self.node_hash` and `root_hash() = self.node_hash`, so `valid()` returns `true` for any `node_hash` with no layers at all.

By contrast, the consensus-layer `validate_merkle_proof` in `merkle_tree.rs` correctly takes an external `root` parameter and rejects proofs whose computed root does not match:

```rust
pub fn validate_merkle_proof(proof: &[u8], item: &[u8; 32], root: &[u8; 32]) -> Result<bool, SetError> {
    let tree = MerkleSet::from_proof(proof)?;
    if tree.get_root() != *root {
        return Err(SetError);
    }
    ...
}
``` [3](#0-2) 

The DataLayer `ProofOfInclusion::valid()` has no equivalent external-root parameter.

`ProofOfInclusion` is a `Streamable` type exposed through the Python binding: [4](#0-3) [5](#0-4) 

It can be deserialized from untrusted bytes via `from_bytes` / `parse_rust`, then passed to `valid()`.

### Impact Explanation

A caller who receives a `ProofOfInclusion` from an untrusted source (e.g., a DataLayer peer) and calls `proof.valid()` as the sole check will accept any internally-consistent proof, regardless of whether the claimed `node_hash` is actually present in the tree. An attacker can:

1. Construct a zero-layer `ProofOfInclusion` with `node_hash` set to any leaf hash they wish to falsely prove is included.
2. Serialize it via the `Streamable` interface.
3. Deliver it to a verifier who calls `proof.valid()`.
4. `valid()` returns `true`; the verifier accepts the forged inclusion.

This lets untrusted input prove invalid DataLayer state, matching the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."*

### Likelihood Explanation

The Python binding exposes `valid()` as a standalone method returning `bool`, with no indication that callers must separately compare `proof.root_hash()` against a trusted root. The method name `valid()` implies complete validation. Existing tests call only `proof.valid()` without a root comparison: [6](#0-5) [7](#0-6) 

Any DataLayer client that follows this pattern when verifying proofs received from a peer is exploitable by an unprivileged attacker who can send a crafted serialized `ProofOfInclusion`.

### Recommendation

1. Add a `root` parameter to `valid()` (or a new `valid_against_root(root: &Hash) -> bool` method) that compares the computed root against a caller-supplied trusted root, mirroring `validate_merkle_proof` in `merkle_tree.rs`.
2. Remove or document the tautological final check so it is not mistaken for a root-binding assertion.
3. Update all call sites (Rust tests, Python tests, DataLayer client code) to supply and verify against a trusted root.

### Proof of Concept

```rust
use chia_datalayer::{ProofOfInclusion, Hash};

// Forge a zero-layer proof claiming node_hash X is in the tree
let fake_node_hash: Hash = [0xAB; 32];
let forged_proof = ProofOfInclusion {
    node_hash: fake_node_hash,
    layers: vec![],   // no layers needed
};

// valid() returns true — no real tree involved
assert!(forged_proof.valid());
// root_hash() == fake_node_hash, not the real tree root
assert_eq!(forged_proof.root_hash(), fake_node_hash);
```

A verifier calling only `forged_proof.valid()` accepts this as a valid proof of inclusion for `fake_node_hash`, even though no such node exists in any real `MerkleBlob`.
