### Title
Integer Division in `normalize_amount` Allows Zero-Amount Transfers That Permanently Lock User Funds - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

When a user initiates a bridge transfer with an amount smaller than `10^(origin_decimals - decimals)`, the `normalize_amount` function silently returns `0` due to integer floor division. No guard rejects or reverts such a transfer. The user's tokens are locked or burned on the source chain while the destination chain receives a transfer of `0` tokens, permanently destroying the user's funds with no recovery path.

---

### Finding Description

`normalize_amount` is defined as:

```rust
fn normalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount / (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

This is pure floor division. When `amount < 10^diff_decimals`, the result is exactly `0`. The code's own comment acknowledges the truncation:

> "Uses floor division — any sub-unit remainder ("dust") is truncated and not transferred to the destination chain. When fee > 0, dust is absorbed into the fee via `claim_fee`. When fee = 0, dust stays locked/burned." [2](#0-1) 

The comment only contemplates *remainder dust* (a small fraction of a larger amount). It does not address the case where the **entire amount** normalizes to `0` — i.e., when the user's full input is below the decimal threshold. There is no check anywhere in the transfer initiation path that rejects or reverts when `normalize_amount` returns `0`. The transfer message is stored with the original amount locked on the source chain, and the destination chain is instructed to release `0` tokens.

The `claim_fee_callback` compounds the problem. It computes the fee as:

```rust
let denormalized_amount = Self::denormalize_amount(
    fin_transfer.amount.0,   // = 0 if normalize returned 0
    ...
);
let fee = transfer_message.amount.0 - denormalized_amount;
// fee = original_locked_amount - 0 = original_locked_amount
``` [3](#0-2) 

This means the entire locked amount is reclassified as "fee," which a trusted relayer can then claim via `claim_fee`. Even without relayer involvement, the user's funds are irrecoverably locked when no fee claim is submitted.

---

### Impact Explanation

**Critical — Permanent irrecoverable lock of user funds.**

A user who sends any amount below `10^(origin_decimals - decimals)` base units will have their tokens locked or burned on the source chain while receiving nothing on the destination chain. There is no refund, no revert, and no recovery mechanism. For a token with `origin_decimals = 18` bridging to a chain where `decimals = 6`, the threshold is `10^12` base units (≈ 0.000001 of the token in human-readable units). Any transfer below this threshold silently destroys the user's funds.

---

### Likelihood Explanation

**Medium.** The decimal difference of 12 (e.g., 18 → 6) is common for tokens like USDC or USDT bridged from Ethereum. The threshold of `10^12` wei is small in absolute terms but reachable by any user making a small or dust transfer. No special role or attacker setup is required — a regular user initiating a transfer triggers this path directly.

---

### Recommendation

Add an explicit zero-amount guard immediately after `normalize_amount` is called in the transfer initiation flow. If the normalized amount is `0`, revert the transaction and return the user's tokens:

```rust
let normalized = Self::normalize_amount(amount, decimals);
require!(normalized > 0, BridgeError::AmountTooSmallAfterNormalization);
```

This mirrors the mitigation recommended for H-06: check for the zero result and exit cleanly rather than proceeding with a semantically invalid state.

---

### Proof of Concept

1. Token is registered with `origin_decimals = 18`, `decimals = 6` (e.g., an ERC-20 USDC-like token bridging from Ethereum to NEAR).
2. User calls `init_transfer` (or equivalent entry point) with `amount = 999_999_999_999` (just below `10^12`).
3. `normalize_amount(999_999_999_999, Decimals { origin_decimals: 18, decimals: 6 })` computes `999_999_999_999 / 1_000_000_000_000 = 0`.
4. No revert occurs. The transfer message is stored. The user's `999_999_999_999` base units are locked/burned on the source chain.
5. The destination chain receives a finalization message for `0` tokens and releases nothing.
6. The user's funds are permanently lost. If a relayer submits a `claim_fee` proof, `fee = 999_999_999_999 - denormalize(0) = 999_999_999_999`, and the relayer claims the user's entire locked amount.
