### Title
Unchecked Constructor Execution Result in `deploy_contract` Causes Unprovable Block State - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/deploy_contract.cairo`)

---

### Summary

The `deploy_contract` function in `deploy_contract.cairo` calls `select_execute_entry_point_func` to invoke a contract constructor, receives an `is_reverted` flag, and then unconditionally asserts `assert is_reverted = 0` without any graceful failure path. Simultaneously, `execute_deploy` in `syscall_impls.cairo` hardcodes `failure_flag=0` in the deploy response header regardless of the actual constructor outcome. For Cairo 1 constructors (which can legitimately revert), this means the OS has no mechanism to produce a valid proof for any block containing a deploy syscall whose constructor reverts, creating a potential network halt condition.

---

### Finding Description

In `deploy_contract.cairo`, the `deploy_contract` function invokes the constructor via `select_execute_entry_point_func`:

```cairo
// Invoke the contract constructor.
let (is_reverted, retdata_size, retdata, _is_deprecated) = select_execute_entry_point_func(
    block_context=block_context, execution_context=constructor_execution_context
);
// ...
// The deprecated deploy syscalls do not support reverts.
assert is_reverted = 0;
``` [1](#0-0) 

The comment "The deprecated deploy syscalls do not support reverts" is misleading: `deploy_contract` is also called from the **non-deprecated** Cairo 1 deploy syscall handler `execute_deploy` in `syscall_impls.cairo`. Cairo 1 constructors can legitimately return `is_reverted=1` (e.g., via a panic or explicit failure). When this happens, the `assert is_reverted = 0` constraint fails, making the execution trace invalid and preventing any valid STARK proof from being generated for the block.

Compounding this, `execute_deploy` in `syscall_impls.cairo` hardcodes `failure_flag=0` in the response header with an explicit TODO acknowledging the missing failure support:

```cairo
// Write the response header.
// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [2](#0-1) 

The `deploy_contract` function itself also carries a TODO acknowledging the missing failure handling: [3](#0-2) 

The call chain is:
- `execute_deploy` (`syscall_impls.cairo`) → `deploy_contract` (`deploy_contract.cairo`) → `select_execute_entry_point_func` → Cairo 1 constructor → `is_reverted=1` → `assert is_reverted = 0` → **proof failure** [4](#0-3) 

The same path applies to `execute_deploy_account_transaction` in `transaction_impls.cairo`, which also calls `deploy_contract`: [5](#0-4) 

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

In Cairo, a failing `assert` constraint means no valid execution trace exists for that code path. If the sequencer includes a block containing a deploy syscall (or deploy_account transaction) where the Cairo 1 constructor reverts, the OS cannot generate a valid STARK proof for that block. The block cannot be finalized on L1. If the sequencer's off-chain simulation diverges from the OS execution (e.g., due to a simulation bug, gas accounting discrepancy, or a constructor that reverts conditionally based on block context), the sequencer would produce an unprovable block, halting the network's ability to confirm new transactions.

---

### Likelihood Explanation

**Medium.** The sequencer is expected to simulate transactions before inclusion and avoid including blocks with reverting constructors. However:

1. The OS and sequencer are separate codebases. Any divergence in how they handle constructor gas, state, or revert conditions can cause the sequencer to include a transaction the OS cannot prove.
2. An attacker can deliberately deploy a Cairo 1 class whose constructor reverts under specific conditions (e.g., based on block timestamp, storage state, or gas remaining) that may not be caught by the sequencer's simulation.
3. The `deploy` syscall is callable by any unprivileged Cairo 1 contract, making the attack surface broad.
4. The explicit TODO comments confirm this is an unresolved gap in the implementation, not a deliberate design choice.

---

### Recommendation

1. **In `deploy_contract.cairo`**: Replace `assert is_reverted = 0` with a proper conditional branch that handles the `is_reverted=1` case — rolling back state changes via the revert log and propagating the failure to the caller, analogous to how `contract_call_helper` in `syscall_impls.cairo` handles reverts. [6](#0-5) 

2. **In `execute_deploy` (`syscall_impls.cairo`)**: Replace the hardcoded `failure_flag=0` with the actual `is_reverted` value returned from `deploy_contract`, and propagate the failure response correctly to the calling contract.

3. **In `execute_deploy_account_transaction` (`transaction_impls.cairo`)**: Handle the case where the constructor of a deploy_account transaction reverts — this should result in a failed (but provable) transaction rather than an unprovable block.

---

### Proof of Concept

1. Declare a Cairo 1 contract class whose constructor unconditionally panics:
   ```rust
   #[constructor]
   fn constructor(ref self: ContractState) {
       panic!("always fails");
   }
   ```
2. Deploy a Cairo 1 "attacker" contract that calls the `deploy` syscall with the above class hash.
3. Submit an invoke transaction calling the attacker contract's deploy function.
4. The sequencer executes the transaction: the constructor reverts, `is_reverted=1` is returned to `deploy_contract`.
5. `deploy_contract.cairo` line 91 executes `assert is_reverted = 0` → constraint fails.
6. The OS execution trace is invalid; no valid STARK proof can be generated for the block.
7. The block cannot be submitted to L1; the network cannot confirm new transactions. [7](#0-6) [8](#0-7)
