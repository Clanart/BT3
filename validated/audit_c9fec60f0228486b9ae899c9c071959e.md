### Title
Fire-and-Forget `send_tokens` with No Failure Callback Permanently Freezes Fast-Transfer Relayer Funds - (`File: near/omni-bridge/src/lib.rs`)

### Summary

In two code paths that reimburse a fast-transfer relayer during finalization, `send_tokens()` is called with `.detach()` (fire-and-forget) while the fast-transfer state is simultaneously updated to `finalised` or removed. If the token transfer fails for any reason, the relayer's funds are permanently lost with no recovery path.

### Finding Description

The Omni Bridge supports a "fast transfer" flow where a trusted relayer advances their own tokens to a recipient before the cross-chain proof arrives. When the proof is later submitted via `fin_transfer`, the bridge is supposed to reimburse the relayer.

**Path 1 — `process_fin_transfer_to_other_chain`** (lines 2028–2040):

```rust
if let Some(relayer) = recipient {
    self.send_tokens(token, relayer, U128(...), "")
        .detach();                                          // fire-and-forget
    self.mark_fast_transfer_as_finalised(&fast_transfer.id()); // state updated
}
```

**Path 2 — `utxo_fin_transfer_fast`** (lines 2529–2548):

```rust
let amount = if fast_transfer.get_destination_chain() == ChainKind::Near {
    self.remove_fast_transfer(&fast_transfer.id());         // state updated
    fast_transfer.amount
} else {
    self.mark_fast_transfer_as_finalised(&fast_transfer.id()); // state updated
    ...
};
self.send_tokens(fast_transfer.token_id.clone(), fast_transfer_status.relayer, amount, "")
    .detach();                                              // fire-and-forget
```

In both cases:
1. The fast-transfer record is either **removed** or **marked finalised** before or alongside the token dispatch.
2. `send_tokens()` is detached — no callback is registered to detect or handle failure.
3. If `send_tokens` fails (e.g., `ft_transfer` panics because the relayer has no storage registered for the token, the token contract is paused, or gas is insufficient), the state change is **permanent** and **irreversible**.
4. The relayer never receives their reimbursement, and the fast-transfer entry can never be retried because it is already finalised/removed.

The developer is aware of this gap in the UTXO path — a `// TODO: check how to deal with failed send_tokens` comment sits at line 2484, directly above the call to `utxo_fin_transfer_fast`.

**Contrast with the correctly handled path**: `process_fin_transfer_to_near` (lines 1957–1977) chains `send_tokens` with a proper `fin_transfer_send_tokens_callback` that reverts lock actions and removes the finalization record on failure. The other-chain and UTXO paths lack this protection entirely.

### Impact Explanation

If `send_tokens` fails in either path, the relayer who advanced funds for the fast transfer permanently loses the tokens they are owed. The fast-transfer record is already finalised/removed, so:
- The reimbursement cannot be retried.
- No admin function exists to recover the stuck funds.
- The tokens remain locked inside the bridge contract indefinitely.

This is **permanent freezing / irrecoverable lock of user funds** in the bridge flow.

### Likelihood Explanation

`send_tokens` can fail in realistic, non-adversarial conditions:
- The relayer account has no storage registered for the specific token being transferred (common for newly deployed tokens).
- The token contract is paused at the moment of finalization.
- Gas exhaustion in the detached promise (the gas budget is computed dynamically and can be zero after subtracting `SEND_TOKENS_CALLBACK_GAS` and `MINT_TOKEN_GAS`).

Any of these conditions, which can arise from normal protocol operation, triggers the permanent loss.

### Recommendation

Replace the `.detach()` pattern with a proper callback, mirroring the `process_fin_transfer_to_near` approach:

1. In `process_fin_transfer_to_other_chain`: chain `send_tokens` with a callback that, on failure, reverts `mark_fast_transfer_as_finalised` (i.e., restores the fast-transfer record so the reimbursement can be retried).
2. In `utxo_fin_transfer_fast`: do not remove or finalise the fast-transfer record before the token transfer succeeds; move the state update into a success-only callback.
3. Remove the `// TODO` comment and implement the fix.

    fn utxo_fin_transfer_fast(
        &mut self,
        fast_transfer: FastTransfer,
        fast_transfer_status: FastTransferStatus,
        utxo_fin_transfer_msg: UtxoFinTransferMsg,
    ) -> PromiseOrPromiseIndexOrValue<U128> {
        require!(
            !fast_transfer_status.finalised,
            BridgeError::FastTransferAlreadyFinalised.as_ref()
        );

        let amount = if fast_transfer.get_destination_chain() == ChainKind::Near {
            self.remove_fast_transfer(&fast_transfer.id());
            fast_transfer.amount
        } else {
            self.mark_fast_transfer_as_finalised(&fast_transfer.id());
            // With transfers to other chain the fee will be claimed after finalization on the destination chain
            U128(
                fast_transfer
                    .amount_without_fee()
                    .near_expect(BridgeError::InvalidFee),
            )
        };

        self.send_tokens(
            fast_transfer.token_id.clone(),
            fast_transfer_status.relayer,
            amount,
            "",
        )
        .detach();

        env::log_str(
            &OmniBridgeEvent::UtxoTransferEvent {
                token_id: fast_transfer.token_id,
                amount,
                utxo_transfer_message: utxo_fin_transfer_msg,
                new_transfer_id: None,
            }
            .to_log_string(),
        );

        PromiseOrPromiseIndexOrValue::Value(U128(0))
    }
