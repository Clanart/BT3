### Title
Missing Duplicate-Key Guard in `batch_insert` Fast-Path Corrupts DataLayer Merkle Tree — (`crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` contains a fast-path `for` loop that writes leaf nodes directly to the blob without checking whether the supplied `KeyId` already exists in the tree or appears more than once in the batch itself. When the tree already holds ≥ 2 leaves, the guard that routes through the duplicate-checking `insert()` is bypassed entirely, allowing two leaf nodes with the same key to be committed to the blob. This leaves `block_status_cache` in an inconsistent state and causes `check_integrity()` to return an error.

---

### Finding Description

**Guard that is bypassed**

`insert()` checks for duplicates before writing: [1](#0-0) 

**The fast-path condition**

When `leaf_count() > 1`, the `if` block that pops items and routes them through `insert()` is skipped entirely: [2](#0-1) 

**The unguarded loop**

All remaining items in the batch are written directly to the blob with no key-uniqueness check: [3](#0-2) 

**Cache corruption mechanism**

`insert_entry_to_blob` calls `add_leaf`, which uses `HashMap::insert` — silently overwriting any prior entry for the same `KeyId`: [4](#0-3) [5](#0-4) 

**Exact post-condition after `batch_insert([(k1,v1,h1),(k1,v2,h2)])` with 2 pre-existing leaves:**

| Location | State |
|---|---|
| Blob | Two leaf nodes with `key = k1` at indices `idx_a` and `idx_b` |
| `key_to_index[k1]` | `idx_b` only (idx_a's entry silently overwritten) |
| `free_indexes` | `idx_a` absent (removed by `add_leaf`) |

**`check_integrity()` failure paths**

1. When the iterator reaches the leaf at `idx_a` (key=k1), the cache lookup returns `idx_b`; the index mismatch check fires: [6](#0-5) 

2. The leaf-count vs. cache-length check also fires (`4 blob leaves` vs. `3 cache entries`): [7](#0-6)

### Impact Explanation

The DataLayer Merkle tree root is derived from the blob. A corrupted blob with two leaves sharing the same `KeyId` produces a structurally invalid tree: the root hash no longer faithfully represents the committed key-value set. Proofs of inclusion/exclusion generated from this tree are invalid. Any subscriber node that applies such a batch will hold a permanently corrupted local tree, unable to pass integrity checks or generate valid proofs. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots and lets untrusted input prove invalid state**.

---

### Likelihood Explanation

The Python binding `py_batch_insert` is directly exposed to callers: [8](#0-7) 

A DataLayer store owner (an unprivileged role — anyone can create a store) publishes updates that subscriber nodes apply via `batch_insert`. Supplying a batch with a repeated `KeyId` is trivial and requires no special privileges. The precondition (≥ 2 existing leaves) is met by any non-trivial store.

---

### Recommendation

Add a duplicate-key check at the top of the fast-path `for` loop in `batch_insert`, mirroring the check already present in `insert()`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    // ... existing code
}
```

Alternatively, collect all keys from the batch into a `HashSet` before the loop and reject the batch immediately if any duplicates are detected (both within the batch and against the existing tree).

    #[pyo3(name = "batch_insert")]
    pub fn py_batch_insert(
        &mut self,
        keys_values: Vec<(KeyId, ValueId)>,
        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
            ))?;
        }

        self.batch_insert(zip(keys_values, hashes).collect())?;

        Ok(())
    }
