### Title
Unbounded `waiting_for_block` accumulation in `SpiceChunkValidatorActor` allows any peer to exhaust validator memory - (`File: chain/client/src/spice/chunk_validator_actor.rs`)

### Summary

`SpiceChunkValidatorActor` stores incoming `SpiceChunkStateWitness` messages in an unbounded `HashMap<CryptoHash, (Vec<SpiceChunkStateWitness>, Vec<SpiceChunkContractAccesses>)>` called `waiting_for_block` when the referenced block is not yet in the store. No signature validation, no size limit, and no distance-from-head check is applied before insertion. Any peer can flood the validator with witnesses referencing non-existent block hashes, growing the map without bound until the process is OOM-killed.

### Finding Description

`SpiceChunkValidatorActor` declares `waiting_for_block` as a plain, unbounded `HashMap`: [1](#0-0) 

When `process_chunk_state_witness` is called and `witness_processing_readiness` returns `NotReady` (block not in store), the witness is pushed directly into the vector with no guard: [2](#0-1) 

The code itself carries a TODO acknowledging the missing protections:

> `// TODO(spice): Implement additional checks (size limit, distance to head) before adding witness to waiting_for_block. See non-spice handle_orphan_witness().`

The same unguarded push exists for `SpiceChunkContractAccesses`: [3](#0-2) 

Contrast with the non-spice orphan path, which enforces a distance-from-head window, a per-witness byte-size cap, and an LRU cache with a configured capacity before any storage: [4](#0-3) 

The non-spice `OrphanStateWitnessPool` is a bounded LRU cache: [5](#0-4) 

No signature check is performed before the spice witness is stored. Signature verification only occurs inside `spice_pre_validate_chunk_state_witness`, which is called only after the block arrives and the witness is moved out of `waiting_for_block`: [6](#0-5) 

The `partial_chunk_data` field is bounded by `MAX_PENDING_CHUNKS = 24` via an LRU cache, but `waiting_for_block` has no analogous bound: [7](#0-6) 

### Impact Explanation

A malicious peer can send an unbounded stream of `SpiceChunkStateWitness` messages each referencing a distinct, non-existent `block_hash`. Every message is accepted, deserialized, and pushed into `waiting_for_block` without any rejection criterion. Because each `SpiceChunkStateWitness` can be up to `MAX_UNCOMPRESSED_STATE_WITNESS_SIZE` (64 MiB), a sustained flood exhausts the validator's heap, causing an OOM crash. A crashed chunk validator cannot endorse chunks, which degrades or halts block production for the shards it covers.

### Likelihood Explanation

The `SpiceChunkStateWitnessMessage` handler accepts messages from the network layer without requiring the sender to be a known chunk producer before storage. The block-hash key space is a 256-bit hash, so an attacker can generate an effectively unlimited number of distinct keys, each accumulating its own unbounded `Vec`. The attack requires only the ability to send network messages to a validator node, which is possible for any participant in the p2p network.

### Recommendation

Apply the same three guards that the non-spice orphan path uses before inserting into `waiting_for_block`:

1. **Signature verification** — verify the witness is signed by the expected chunk producer before storing it.
2. **Distance-from-head check** — reject witnesses whose `height_created` is outside `ALLOWED_ORPHAN_WITNESS_DISTANCE_FROM_HEAD`.
3. **Bounded capacity** — replace the plain `HashMap<CryptoHash, Vec<...>>` with a structure that enforces a total byte or entry cap, evicting the oldest entries when the limit is reached, mirroring `OrphanStateWitnessPool`.

### Proof of Concept

1. Attacker connects to a Spice validator node over the p2p network.
2. Attacker repeatedly sends `SpiceChunkStateWitnessMessage` with `chunk_id.block_hash` set to random 32-byte values that do not exist in the validator's chain store.
3. Each message passes the `validator_signer.get()` check (node is a validator) and enters `process_chunk_state_witness`.
4. `witness_processing_readiness` returns `NotReady` for every message because the block is not in the store.
5. Each witness is pushed into `self.waiting_for_block.entry(chunk_id.block_hash).or_default().0` with no size or count limit.
6. The `HashMap` grows without bound; the process is OOM-killed.
7. The validator stops endorsing chunks for its assigned shards.
