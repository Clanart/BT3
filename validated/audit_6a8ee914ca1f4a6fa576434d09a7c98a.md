### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Contract Freezing — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in `syscall_impls.cairo` does not verify that the caller-supplied `class_hash` corresponds to a previously declared contract class before committing it to state. Any unprivileged transaction sender can deploy a contract and invoke the `replace_class` syscall with an arbitrary, undeclared class hash. The OS accepts and commits this invalid state transition, permanently breaking the contract and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class` (Cairo 1 / new-syscall path), after gas deduction, the function reads `request.class_hash` and immediately writes it into `contract_state_changes` with no check that the hash exists in `contract_class_changes` (i.e., was ever declared):

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 is an explicit developer acknowledgment that this validation is absent and was deferred past the date of this audit (deadline: `1/1/2026`). [2](#0-1) 

The identical missing check exists in the deprecated (Cairo 0) syscall path:

```cairo
func execute_replace_class{contract_state_changes: DictAccess*, revert_log: RevertLogEntry*}(
    contract_address, syscall_ptr: ReplaceClass*
) {
    alloc_locals;
    let class_hash = syscall_ptr.class_hash;
    // No check that class_hash is declared.
    ...
    dict_update{dict_ptr=contract_state_changes}(...);
``` [3](#0-2) 

The `replace_class` syscall is dispatched from `execute_syscalls` (Cairo 1 path) and `execute_deprecated_syscalls` (Cairo 0 path) with no pre-dispatch validation either: [4](#0-3) [5](#0-4) 

By contrast, the `declare` transaction path correctly enforces that a class hash is valid before writing it to `contract_class_changes`:

```cairo
// Ensure the given class hash is a result of a Sierra class hash calculation.
let expected_class_hash = finalize_class_hash(...);
assert [class_hash_ptr] = expected_class_hash;
...
// Note that prev_value=0 enforces that a class may be declared only once.
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash);
``` [6](#0-5) 

`replace_class` bypasses all of this — it writes directly to `contract_state_changes` with an unvalidated hash.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a contract's class hash is set to an undeclared value (one with no entry in `contract_class_changes`), the OS will look up that hash in the class dict on any subsequent call and receive a zero/invalid compiled class hash. The contract becomes permanently non-executable. Any funds (ERC-20 balances, ETH, NFTs) held by or attributed to that contract address are irrecoverably frozen, as no entry point can ever be dispatched again.

---

### Likelihood Explanation

The `replace_class` syscall is a standard, publicly accessible upgrade mechanism. An unprivileged transaction sender can:

1. Deploy any contract (no special privilege required).
2. Have that contract call `replace_class` with an arbitrary felt value as the class hash.
3. The OS commits the invalid class hash to state without rejection.

Additionally, any existing deployed contract that exposes a public upgrade entry point (a common pattern) is vulnerable if its own logic does not independently validate the class hash — a check that users reasonably expect the OS to enforce.

---

### Recommendation

In `execute_replace_class` in `syscall_impls.cairo`, before updating `contract_state_changes`, verify that `class_hash` has a non-zero entry in `contract_class_changes`. If the class is not declared, write a failure response instead of committing the state change:

```cairo
let class_hash = request.class_hash;

// Check that there is a declared contract class with the given hash.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
if (compiled_class_hash == 0) {
    write_failure_response(remaining_gas=remaining_gas, failure_felt=ERROR_INVALID_ARGUMENT);
    return ();
}

// Proceed with state update only if class is declared.
```

Apply the same fix to `execute_replace_class` in `deprecated_execute_syscalls.cairo`.

---

### Proof of Concept

**Steps:**

1. Deploy contract `C` that holds funds and exposes a public `upgrade(class_hash: felt)` entry point that calls `replace_class(class_hash)`.
2. As an unprivileged attacker, send an invoke transaction calling `C.upgrade(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
3. The OS executes `execute_replace_class`, skips the missing validation, and commits `StateEntry(class_hash=0xdeadbeef, ...)` for contract `C` into `contract_state_changes`.
4. The block is proven and the state root is updated with `C`'s class hash set to `0xdeadbeef`.
5. In any subsequent block, any call to `C` causes the OS to look up `0xdeadbeef` in `contract_class_changes`, receive `compiled_class_hash=0`, and fail to dispatch any entry point.
6. All funds held by `C` are permanently frozen — no withdrawal, transfer, or recovery is possible.
