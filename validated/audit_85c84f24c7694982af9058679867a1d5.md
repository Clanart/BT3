### Title
Gas price permanently frozen at zero via relative-only increase formula when `price = 0` — (`crates/apollo_consensus_orchestrator/src/fee_market/mod.rs`)

---

### Summary

`calculate_next_base_gas_price` applies a gradual recovery formula `price + price / MIN_GAS_PRICE_INCREASE_DENOMINATOR` when the current price is below the configured minimum. When `price = 0`, integer division yields `0 / 333 = 0`, so the adjusted price is `min(0, min_gas_price) = 0`. The function returns `GasPrice(0)` unconditionally, permanently freezing the L2 gas price at zero. This wrong value propagates into every subsequent block header's `next_l2_gas_price` field and into the block hash commitment via `calculate_block_hash`, producing an incorrect authoritative block hash and breaking fee accounting for all downstream consumers.

---

### Finding Description

The vulnerable code is in `calculate_next_base_gas_price`:

```rust
// crates/apollo_consensus_orchestrator/src/fee_market/mod.rs, lines 105-114
if price < min_gas_price {
    let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR; // 0 / 333 = 0
    let adjusted = price.0 + max_increase;                           // 0 + 0 = 0
    let adjusted_price = adjusted.min(min_gas_price.0);              // min(0, X) = 0
    return GasPrice(adjusted_price);                                  // returns GasPrice(0)
}
``` [1](#0-0) 

This is the structural analog of the Smilee Finance bug: a relative-only tolerance/adjustment check (`value / N`) that silently produces zero when the reference value is zero, causing the correction to have no effect.

**Reachability — how `price = 0` enters the function:**

`SequencerConsensusContext.l2_gas_price` is of type `GasPrice` (a plain `u128` wrapper whose `Default` is 0). [2](#0-1) 

The `try_sync` path sets `self.l2_gas_price` directly from the synced block's `next_l2_gas_price` field. Old (deprecated) block formats explicitly document that L2 gas price was 0:

```rust
// In older versions, L2 gas price was 0.
PendingBlockOrDeprecated::Deprecated(_) => GasPricePerToken::default(),
``` [3](#0-2) 

The test `test_first_height_keeps_sync_provided_l2_gas_price` confirms that `try_sync` assigns the synced block's `next_l2_gas_price` directly to `context.l2_gas_price`: [4](#0-3) 

After the first height, `update_l2_gas_price` calls `calculate_next_l2_gas_price_for_fin` → `calculate_next_base_gas_price(GasPrice(0), ...)` → returns `GasPrice(0)` forever: [5](#0-4) 

**Why the bootstrap guard does not protect subsequent heights:**

`set_height_and_round` contains a guard that clamps `l2_gas_price` to the configured minimum, but only on the **first** height (`self.current_height.is_none()`): [6](#0-5) 

If `l2_gas_price` becomes 0 after the first height (e.g., via a sync from an old block), this guard never fires again.

**Propagation to block hash commitment:**

`next_l2_gas_price = 0` is stored in block headers and fed into `calculate_block_hash` via `l2_gas_price` in `PartialBlockHashComponents`: [7](#0-6) 

The `l2_gas_price` field is chained into the Poseidon hash at lines 265–272. A permanently-zero price produces a wrong block hash for every block after the freeze, breaking the commitment invariant.

**Propagation to fee estimation and gateway:**

`calculate_next_l2_gas_price_for_fin` is called both when building the fin and when updating state at decision time: [8](#0-7) 

With `next_l2_gas_price = 0`, the gateway's `validate_tx_l2_gas_price_within_threshold` receives a zero-valued price, and the computed threshold `0 * percentage / 100 = 0` causes all transactions to pass the L2 gas price check regardless of their actual bid: [9](#0-8) 

---

### Impact Explanation

- **Wrong block hash commitment**: Every block produced after the freeze includes `l2_gas_price = 0` in its Poseidon hash chain, producing an incorrect `BlockHash` that diverges from what any correctly-operating node would compute.
- **Wrong `next_l2_gas_price` in block headers**: The stored header field is permanently 0, corrupting the authoritative fee-market state read by RPC, sync, and proof consumers.
- **Fee market permanently broken**: The EIP-1559-style adjustment never recovers; all subsequent blocks inherit `next_l2_gas_price = 0`.
- **Gateway admits under-priced transactions**: With threshold = 0, the L2 gas price admission check is bypassed for all transactions.

Matching impact categories: **Critical — Incorrect fee/gas/resource accounting with economic impact** and **Critical — Wrong block hash/state value from execution logic**.

---

### Likelihood Explanation

The trigger condition (`price = 0`) is reached whenever a node syncs from any pre-L2-gas-price block (old format), which is an explicitly documented and handled code path. The `PendingBlockOrDeprecated::Deprecated` arm already acknowledges "In older versions, L2 gas price was 0." Any sequencer node that bootstraps by syncing from historical blocks before the L2 gas price era will hit this path. No privileged access is required; the condition arises from normal sync operation.

---

### Recommendation

Replace the relative-only increase with a combined relative + absolute floor so the price can always escape zero:

```rust
// crates/apollo_consensus_orchestrator/src/fee_market/mod.rs
if price < min_gas_price {
    // Use at least 1 fri of absolute increase to escape zero.
    let max_increase = (price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR).max(1);
    let adjusted_price = price.0.saturating_add(max_increase).min(min_gas_price.0);
    return GasPrice(adjusted_price);
}
```

Alternatively, add an explicit zero-price guard at the entry of `calculate_next_base_gas_price`:

```rust
if price.0 == 0 {
    return min_gas_price;
}
```

The `try_sync` path should also clamp the loaded `next_l2_gas_price` to `max(synced_price, effective_min)` before assigning it to `self.l2_gas_price`, mirroring the bootstrap guard already present for the first height.

---

### Proof of Concept

```rust
#[test]
fn test_price_permanently_frozen_at_zero() {
    use crate::fee_market::calculate_next_base_gas_price;
    use starknet_api::block::GasPrice;
    use starknet_api::execution_resources::GasAmount;

    let price        = GasPrice(0);               // synced from old block
    let min_gas_price = GasPrice(1_000_000_000);  // configured minimum
    let gas_used     = GasAmount(1_000_000);       // non-zero usage
    let gas_target   = GasAmount(2_000_000);

    // First call: price is 0, formula gives 0/333 = 0, returns GasPrice(0).
    let result = calculate_next_base_gas_price(price, gas_used, gas_target, min_gas_price);
    assert_eq!(result, GasPrice(0), "price should have increased but is stuck at 0");

    // Second call with the returned price: still 0, still stuck.
    let result2 = calculate_next_base_gas_price(result, gas_used, gas_target, min_gas_price);
    assert_eq!(result2, GasPrice(0), "price remains frozen at 0 indefinitely");
    // Expected: result >= GasPrice(1) on first call, converging toward min_gas_price.
}
```

The test demonstrates that `calculate_next_base_gas_price` returns `GasPrice(0)` on every invocation when `price = 0`, confirming the permanent freeze. The wrong value then flows through `update_l2_gas_price` → block header `next_l2_gas_price` → `calculate_block_hash` → incorrect block hash commitment for every block produced after the freeze.
