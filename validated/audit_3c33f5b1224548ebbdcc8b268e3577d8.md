### Title
Transfer Permanently Locked When `normalize_amount(amount - fee) == 0` Due to Missing Pre-Check in `init_transfer` - (File: `near/omni-bridge/src/lib.rs`)

### Summary

The NEAR bridge contract accepts a token transfer via `ft_transfer_call` → `init_transfer` and locks user tokens, but performs no validation that the post-fee amount survives decimal normalization. When `normalize_amount(amount - fee.fee)` truncates to zero (floor division), `sign_transfer` permanently reverts with `InvalidAmountToTransfer`, and there is no cancel or recovery path. User tokens are irrecoverably locked.

### Finding Description

`init_transfer` validates only that `fee.fee < amount`: [1](#0-0) 

It does not validate that the net amount survives the decimal normalization applied later in `sign_transfer`: [2](#0-1) 

`normalize_amount` uses floor division: [3](#0-2) 

For a NEAR-native token with `origin_decimals = 24` bridging to an EVM token with `decimals = 6`, the divisor is `10^18`. Any transfer where `amount - fee.fee < 10^18` produces `normalize_amount = 0`, causing `sign_transfer` to always revert.

`update_transfer_fee` only allows the fee to be **increased**, never decreased: [4](#0-3) 

This makes the stuck state irrecoverable: increasing the fee makes `amount_without_fee` even smaller. There is no public `cancel_transfer` or emergency-withdrawal function; `remove_transfer_message` is only reachable via `claim_fee_callback`, which requires a valid proof from the destination chain — a proof that can never exist because `sign_transfer` never succeeds. [5](#0-4) 

### Impact Explanation

User tokens are permanently locked in the bridge contract. The `pending_transfers` entry persists indefinitely with no recovery path. This matches the **Critical** impact class: *Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge flows.*

### Likelihood Explanation

Any user who initiates a NEAR → EVM transfer with `amount - fee < 10^(origin_decimals - decimals)` triggers this. For a 24-decimal NEAR token bridging to a 6-decimal EVM token, the threshold is `10^18` base units — a small but non-zero amount. This can happen accidentally (dust transfers, small test amounts) or be triggered deliberately by an attacker who front-runs a victim's storage deposit to force a small-amount transfer through.

### Recommendation

Add a validation in `init_transfer` (or `init_transfer_internal`) that `normalize_amount(amount - fee.fee, decimals) > 0` before locking tokens. Additionally, implement a `cancel_transfer` function that allows the original sender to reclaim locked tokens for transfers that have not yet been signed, providing a recovery path for stuck transfers.

### Proof of Concept

**Setup:**
- Token registered with `origin_decimals = 24`, `decimals = 6` (e.g., a NEAR-native token bridging to EVM USDC-scale)
- Divisor = `10^18`

**Attack / accidental path:**
1. User calls `ft_transfer_call` with `amount = 5 * 10^17` (0.5 in 24-decimal units), `fee = 0`, recipient = EVM address.
2. `init_transfer` passes the check `fee.fee (0) < amount (5e17)` ✓ — tokens are locked.
3. Relayer calls `sign_transfer`:
   - `amount_without_fee() = 5e17 - 0 = 5e17`
   - `normalize_amount(5e17, {24, 6}) = 5e17 / 10^18 = 0`
   - `require!(0 > 0)` → **reverts with `ERR_INVALID_AMOUNT_TO_TRANSFER`**
4. User calls `update_transfer_fee` to try to fix it — only allowed to increase fee, making `amount_without_fee` smaller → still 0.
5. No `cancel_transfer` exists. `claim_fee_callback` requires a destination-chain proof that can never be generated.
6. **Tokens are permanently locked.** [2](#0-1) [1](#0-0) [3](#0-2) [6](#0-5)

    fn remove_transfer_message(&mut self, transfer_id: TransferId) -> TransferMessage {
        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);
        }

        transfer.message
    }
