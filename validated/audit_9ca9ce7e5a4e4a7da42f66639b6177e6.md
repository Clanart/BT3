### Title
`batch_insert` Bypasses Duplicate-Key Guard, Corrupting DataLayer Merkle Tree Root — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the `KeyAlreadyPresent` and `HashAlreadyPresent` guards that `MerkleBlob::insert` enforces. When a caller supplies duplicate keys — either within the batch itself or matching keys already in the tree — duplicate leaf nodes are silently written into the blob. The in-memory `block_status_cache` is left inconsistent with the on-disk blob, the computed Merkle root is wrong, and proofs of inclusion derived from it are invalid.

---

### Finding Description

`MerkleBlob::insert` opens with two explicit guards:

```rust
// blob.rs:369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert` uses `insert` only for the last two items popped from the input vector when the tree has ≤ 1 existing leaves. Every other item — the entire batch when the tree already has ≥ 2 leaves, or all but the last two items otherwise — is written directly through `insert_entry_to_blob` with no duplicate check at all:

```rust
// blob.rs:587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which performs an unconditional `HashMap::insert` — silently overwriting any prior mapping for the same key:

```rust
// blob.rs:1024-1026
match block.node {
    Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
``` [3](#0-2) 

```rust
// blob.rs:188-192
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);       // silent overwrite
    self.leaf_hash_to_index.insert(leaf.hash, index); // silent overwrite
}
``` [4](#0-3) 

The result is a blob that physically contains two leaf nodes sharing the same `KeyId`, while `block_status_cache.key_to_index` retains only the last-written index. The tree structure is corrupted: `leaf_count` (from iterating the blob) exceeds `key_to_index_cache_length`, which `check_integrity` would detect — but `check_integrity` is not called automatically after `batch_insert`. The Python binding `py_batch_insert` exposes this path directly: [5](#0-4) 

---

### Impact Explanation

After a corrupted `batch_insert`, `calculate_lazy_hashes` computes a root hash over a tree that contains duplicate leaf nodes. Every `get_proof_of_inclusion` call thereafter produces a proof anchored to this wrong root. Any verifier that trusts the root hash will accept proofs for state that was never legitimately committed, or reject proofs for state that was. This directly satisfies the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

The Python binding `py_batch_insert` is a public API callable by any DataLayer client. No privilege is required. A caller need only pass a list containing a repeated `KeyId` (or a `KeyId` already present in the tree when it has ≥ 2 leaves). The bug is triggered on the normal fast path of `batch_insert`, which is the recommended bulk-insertion API used in production DataLayer sync. [6](#0-5) 

---

### Recommendation

Add duplicate-key and duplicate-hash checks at the top of `batch_insert`, mirroring `insert`, before any item is written to the blob:

```rust
pub fn batch_insert(
    &mut self,
    keys_values_hashes: Vec<((KeyId, ValueId), Hash)>,
) -> Result<(), Error> {
    for ((key, _), hash) in &keys_values_hashes {
        if self.block_status_cache.contains_key(*key) {
            return Err(Error::KeyAlreadyPresent());
        }
        if self.block_status_cache.contains_leaf_hash(hash) {
            return Err(Error::HashAlreadyPresent());
        }
    }
    // ... existing logic
}
```

Alternatively, deduplicate within the batch itself before the pre-check, or delegate all insertions through `insert` (accepting the performance trade-off). A regression test for `batch_insert` with a duplicate key should be added alongside the fix.

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

let mut blob = MerkleBlob::new(vec![]).unwrap();

// Pre-populate with 2 leaves so batch_insert takes the all-fast-path branch
blob.insert(KeyId(100), ValueId(100), &Hash(Bytes32::new([1u8;32])), InsertLocation::Auto{}).unwrap();
blob.insert(KeyId(200), ValueId(200), &Hash(Bytes32::new([2u8;32])), InsertLocation::Auto{}).unwrap();

// Batch with a duplicate key (KeyId(100) already present)
let batch = vec![
    ((KeyId(100), ValueId(999)), Hash(Bytes32::new([3u8;32]))), // duplicate!
    ((KeyId(300), ValueId(300)), Hash(Bytes32::new([4u8;32]))),
];

// No error is returned — duplicate silently accepted
blob.batch_insert(batch).unwrap();

// blob now has two physical leaf nodes for KeyId(100);
// block_status_cache only knows about the last one.
// check_integrity() will return IntegrityKeyToIndexCacheIndex error,
// but calculate_lazy_hashes() and get_proof_of_inclusion() proceed
// on the corrupted tree, producing a wrong root and invalid proofs.
blob.calculate_lazy_hashes().unwrap();
// Root hash is now computed over a tree with a duplicate leaf — corrupted state.
