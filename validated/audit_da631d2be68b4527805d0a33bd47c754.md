### Title
Missing `normalize_amount > 0` Validation in `init_transfer` That Exists in `sign_transfer` Causes Permanent Token Lock — (`File: near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer` accepts and locks user tokens with only a `fee < amount` check. A later, mandatory processing step — `sign_transfer` — enforces an additional constraint: `normalize_amount(amount - fee) > 0`. For tokens whose NEAR-side decimals exceed the destination-chain decimals, any amount below the normalization threshold passes `init_transfer` but causes `sign_transfer` to panic permanently. Because no public cancel/refund path exists, the locked tokens are irrecoverable without DAO intervention.

---

### Finding Description

The NEAR bridge processes outbound transfers in two distinct stages:

**Stage 1 — `init_transfer` (acceptance):** [1](#0-0) 

The only amount/fee validation is `fee < amount`. Tokens are then locked or burned inside `init_transfer_internal`: [2](#0-1) 

**Stage 2 — `sign_transfer` (execution):** [3](#0-2) 

`normalize_amount` performs floor division by `10^(origin_decimals - decimals)`: [4](#0-3) 

For any token where `origin_decimals > decimals` (e.g., a NEAR token with 24 decimals bridged to an EVM chain with 6 decimals, giving a factor of `10^18`), any `amount - fee < 10^18` normalizes to zero. `sign_transfer` then panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`. Because `sign_transfer` panics before any state mutation, the transfer remains in `pending_transfers` with tokens locked.

There is no public cancel or refund function. `remove_transfer_message` is only called internally on successful signing or fee-claim finalization: [5](#0-4) 

The DAO escape hatch `transfer_token_as_dao` requires privileged access and is not a user-accessible recovery path.

---

### Impact Explanation

A user who sends any amount below the decimal-normalization threshold for a given token has their funds permanently locked in the bridge contract. For a token with `origin_decimals = 24` and `decimals = 6`, the threshold is `10^18` base units — a non-trivial amount. The transfer sits in `pending_transfers` indefinitely; `sign_transfer` will always panic on it, and no user-callable path removes it or returns the tokens.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

Any unprivileged bridge user can trigger this by calling `ft_transfer_call` with a small amount targeting a token whose NEAR decimals exceed the destination-chain decimals. The user need not be malicious — a user unaware of the normalization threshold will accidentally lock their own funds. The condition is reachable on any token pair where `origin_decimals > decimals`, which is a standard configuration for many bridged assets (e.g., NEAR-native tokens with 24 decimals bridged to 6-decimal EVM tokens).

---

### Recommendation

Add the same `normalize_amount > 0` guard to `init_transfer` (or `init_transfer_internal`) before locking tokens, mirroring the check already present in `sign_transfer`:

```rust
// In init_transfer, after building transfer_message and before locking:
let token_address = self.get_token_address(
    transfer_message.get_destination_chain(),
    self.get_token_id(&transfer_message.token),
);
if let Some(token_address) = token_address {
    if let Some(decimals) = self.token_decimals.get(&token_address) {
        let normalized = Self::normalize_amount(
            transfer_message.amount_without_fee().unwrap_or(0),
            decimals,
        );
        require!(normalized > 0, BridgeError::InvalidAmountToTransfer.as_ref());
    }
}
```

This mirrors the existing guard in `sign_transfer` at lines 475–485 and prevents tokens from being locked into an unprocessable state.

---

### Proof of Concept

1. A token is registered with `origin_decimals = 24`, `decimals = 6` (normalization factor `10^18`).
2. Attacker (or naive user) calls `ft_transfer_call` on the token contract with `amount = 1` and `msg` encoding `InitTransferMsg { fee: U128(0), recipient: <EVM address>, ... }`.
3. `init_transfer` passes: `0 < 1` ✓. `init_transfer_internal` locks `1` base unit. [1](#0-0) 
4. A trusted relayer calls `sign_transfer` for this transfer ID.
5. `normalize_amount(1 - 0, {origin_decimals:24, decimals:6}) = 1 / 10^18 = 0`. [3](#0-2) 
6. `sign_transfer` panics with `ERR_INVALID_AMOUNT_TO_TRANSFER`. No state is changed; the transfer remains in `pending_transfers`.
7. No user-callable function can remove the transfer or return the tokens. Funds are permanently locked.

    fn remove_transfer_message(&mut self, transfer_id: TransferId) -> TransferMessage {
        let storage_usage = env::storage_usage();
        let transfer = self
            .pending_transfers
            .remove(&transfer_id)
            .map(storage::TransferMessageStorage::into_main)
            .near_expect(BridgeError::TransferNotExist);

        let refund =
            env::storage_byte_cost().saturating_mul((storage_usage - env::storage_usage()).into());

        if let Some(mut storage) = self.accounts_balances.get(&transfer.owner) {
            storage.available = storage.available.saturating_add(refund);
            self.accounts_balances.insert(&transfer.owner, &storage);
        }

        transfer.message
    }
