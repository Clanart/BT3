### Title
`deploy_contract` Unconditionally Asserts Constructor Non-Revert, Causing OS Proof Failure for Reverted Deploy Account Transactions — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/deploy_contract.cairo`)

---

### Summary

`deploy_contract` calls `select_execute_entry_point_func` for the constructor and then unconditionally asserts `is_reverted = 0`. For Sierra (Cairo 1+) contracts whose constructors revert, this assertion fails, causing the StarkNet OS proof to abort. Unlike `execute_invoke_function_transaction` and `execute_l1_handler_transaction`, `execute_deploy_account_transaction` contains no `IsReverted` guard before calling `deploy_contract`, so there is no path to skip or handle a reverted constructor. Any block containing a deploy-account transaction whose Sierra constructor reverts cannot be proven, halting the network.

---

### Finding Description

**Vulnerable call chain:**

`execute_deploy_account_transaction` (transaction_impls.cairo, line 643) → `deploy_contract` (deploy_contract.cairo, line 82) → `select_execute_entry_point_func` → returns `is_reverted = 1` for a reverting Sierra constructor → `assert is_reverted = 0` (deploy_contract.cairo, line 91) **fails**.

In `deploy_contract.cairo`, after invoking the constructor through `select_execute_entry_point_func`, the function unconditionally asserts the constructor did not revert:

```cairo
// The deprecated deploy syscalls do not support reverts.
assert is_reverted = 0;
``` [1](#0-0) 

The comment reveals the intent: this assertion was written for the deprecated (Cairo 0) deploy syscall path, where `deprecated_execute_entry_point` always returns `is_reverted=0`. However, `deploy_contract` is also called from `execute_deploy_account_transaction` for Sierra contracts. For Sierra contracts, `execute_entry_point` propagates the contract's `failure_flag` directly as `is_reverted`:

```cairo
local is_reverted = entry_point_return_values.failure_flag;
``` [2](#0-1) 

So when a Sierra constructor reverts, `select_execute_entry_point_func` returns `is_reverted=1`, and the unconditional `assert is_reverted = 0` in `deploy_contract` causes the OS Cairo program to fail.

**Contrast with other transaction types:**

`execute_invoke_function_transaction` and `execute_l1_handler_transaction` both read the `IsReverted` hint and branch accordingly before executing contract code:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
} else {
    // Skip execution.
    ...
}
``` [3](#0-2) 

`execute_deploy_account_transaction` has no such guard. It unconditionally calls `deploy_contract`:

```cairo
let revert_log = init_revert_log();
deploy_contract{revert_log=revert_log}(
    block_context=block_context, constructor_execution_context=constructor_execution_context
);
``` [4](#0-3) 

This is the direct analog to the reported vulnerability: a function designed to handle a class of transactions (deploy account) calls a sub-function (`deploy_contract`) that always aborts when the constructor reverts, making it impossible to prove any block containing such a transaction.

---

### Impact Explanation

The StarkNet OS is a Cairo program whose successful execution produces the validity proof for a block. If the OS program aborts (via a failed `assert`), no proof can be generated for that block. The block cannot be finalized on L1, and the network cannot advance past it. This constitutes a **total network shutdown** — the network cannot confirm new transactions.

---

### Likelihood Explanation

An unprivileged user can:
1. Declare a Sierra contract class whose constructor contains `assert 1 = 0` (or any always-reverting logic).
2. Submit a deploy-account transaction referencing that class.

The sequencer executes the transaction, the constructor reverts, and the sequencer includes the transaction in a block (standard behavior: reverted transactions are included to charge fees). When the OS attempts to prove the block, `assert is_reverted = 0` fails, aborting the proof. The attacker needs only a valid signature and sufficient fee balance — no privileged access is required.

---

### Recommendation

Add an `IsReverted` check in `execute_deploy_account_transaction` before calling `deploy_contract`, mirroring the pattern used in `execute_invoke_function_transaction`. If the transaction is marked as reverted (constructor failed), skip the constructor execution and handle state/fee accordingly. Alternatively, modify `deploy_contract` to return `is_reverted` to its caller and let `execute_deploy_account_transaction` handle the reverted case, rather than asserting it can never happen.

---

### Proof of Concept

1. **Attacker declares** a Sierra contract class with constructor body: `assert 1 = 0; // always reverts`.
2. **Attacker submits** a `deploy_account` transaction referencing this class hash, with a valid signature and fee.
3. **Sequencer executes** the transaction: constructor reverts; transaction is marked reverted but included in the block for fee collection.
4. **OS attempts to prove the block**:
   - `execute_deploy_account_transaction` calls `deploy_contract` (no `IsReverted` guard).
   - `deploy_contract` calls `select_execute_entry_point_func` → dispatches to `execute_entry_point` (Sierra path).
   - Constructor reverts → `entry_point_return_values.failure_flag = 1` → `is_reverted = 1`.
   - `assert is_reverted = 0` **fails** → OS program aborts.
5. **No proof is generated** for the block → block cannot be finalized → **network halt**. [5](#0-4) [6](#0-5)
