### Title
Delegate Can Drain Victim's Token Account via `init_transfer`, Crediting Attacker on Destination Chain — (`solana/programs/bridge_token_factory/src/instructions/user/init_transfer.rs`)

---

### Summary

The `InitTransfer` account struct imposes no ownership constraint linking `from` to `user`. Because SPL Token's `transfer_checked` accepts an approved delegate as a valid authority, any attacker holding a delegation over a victim's token account can call `init_transfer` with `from=victim_account`, `user=attacker`, and `payload.recipient=attacker_destination_address`. The victim's tokens are locked in the bridge vault (or burned for bridged tokens) while the Wormhole message credits the attacker's address on the destination chain.

---

### Finding Description

The `from` account constraint is:

```rust
#[account(
    mut,
    token::mint = mint,
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
``` [1](#0-0) 

There is **no** `token::authority = user` constraint. The only checks are that `from` uses the correct mint and token program. The `user` account is only required to be a signer owned by the system program:

```rust
#[account(
    mut,
    owner = common.system_program.key(),
)]
pub user: Signer<'info>,
``` [2](#0-1) 

In `process()`, the transfer CPI passes `self.user` as the `authority`:

```rust
transfer_checked(
    CpiContext::new(
        self.token_program.to_account_info(),
        TransferChecked {
            from: self.from.to_account_info(),
            to: vault.to_account_info(),
            authority: self.user.to_account_info(),
            mint: self.mint.to_account_info(),
        },
    ),
    ...
)?;
``` [3](#0-2) 

SPL Token's `transfer_checked` accepts **either the owner or an approved delegate** as a valid authority. If the attacker is an approved delegate of the victim's token account, the CPI succeeds without any bridge-level rejection.

After the transfer, the Wormhole message is posted with `self.user.key()` — the attacker's key — as the `sender`, and `payload.recipient` — fully attacker-controlled — as the destination:

```rust
self.common.post_message(payload.serialize_for_near((
    self.common.sequence.sequence,
    self.user.key(),
    self.mint.key(),
))?)?;
``` [4](#0-3) 

The serialized message encodes `params.1` (i.e., `self.user.key()`) as the sender and `self.recipient` as the cross-chain destination: [5](#0-4) 

The same missing ownership check applies to the `burn` path for bridged tokens: [6](#0-5) 

---

### Impact Explanation

An attacker who holds any SPL Token delegation over a victim's account can:

1. Lock the victim's native tokens in the bridge vault (or burn bridged tokens from the victim's account).
2. Have the resulting Wormhole message credit the attacker's chosen address on the destination chain (NEAR or any other supported chain).

The victim's tokens are permanently removed from their account and irrecoverably credited to the attacker. This is direct, cross-chain theft of user assets.

---

### Likelihood Explanation

The precondition — victim has approved attacker as a delegate — is realistic. Users routinely issue SPL Token `approve` instructions to DEXes, lending protocols, and other DeFi programs. Any such approval, regardless of its original intent, can be weaponized to call `init_transfer` against the victim's account. The attacker needs only to sign the transaction themselves (`user` is a `Signer`) and supply the victim's token account as `from`.

---

### Recommendation

Add `token::authority = user` to the `from` account constraint:

```rust
#[account(
    mut,
    token::mint = mint,
    token::authority = user,
    token::token_program = token_program,
)]
pub from: Box<InterfaceAccount<'info, TokenAccount>>,
```

This enforces at the Anchor constraint level — before any CPI — that `user` is the owner of `from`, preventing delegates from initiating bridge transfers on behalf of token account owners.

---

### Proof of Concept

1. Victim creates a token account holding 1000 tokens and calls SPL Token `approve` granting attacker a delegated amount of 1000 (for any reason, e.g., a DEX interaction).
2. Attacker constructs an `init_transfer` transaction with:
   - `from` = victim's token account
   - `user` = attacker's keypair (signs the transaction)
   - `payload.recipient` = attacker's NEAR address
3. Anchor constraint validation passes: `from` has the correct mint and token program; `user` is a valid signer.
4. `transfer_checked` CPI succeeds: SPL Token sees attacker as an approved delegate with sufficient delegated amount.
5. Victim's 1000 tokens are transferred into the bridge vault.
6. Wormhole message is posted with attacker as sender and attacker's NEAR address as recipient.
7. On NEAR, the bridge finalizes the transfer and mints/releases tokens to the attacker's address.
