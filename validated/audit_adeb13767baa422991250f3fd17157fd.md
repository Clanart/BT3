### Title
Permanent Fund Lock via Indefinitely Reverting ETH Delivery in `finTransfer` — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.sol`'s `finTransfer` delivers native ETH to the recipient using a low-level call and reverts the entire transaction if the call fails. If the recipient is a contract that permanently rejects ETH, every `finTransfer` attempt reverts indefinitely. Because the MPC-signed payload fixes the recipient address and there is no on-chain recovery path, the user's source-chain tokens become permanently unclaimable.

---

### Finding Description

`finTransfer` marks the destination nonce as used **before** performing any delivery, then attempts to send ETH:

```solidity
completedTransfers[payload.destinationNonce] = true;   // line 287
// ...
(bool success, ) = payload.recipient.call{value: payload.amount}("");
if (!success) revert FailedToSendEther();              // line 322
``` [1](#0-0) 

Because the `revert` unwinds the entire transaction, the nonce write at line 287 is also rolled back. The nonce is therefore never permanently consumed — but that is precisely the trap: every subsequent retry also reverts, because the recipient address is immutably encoded in the MPC-signed `TransferMessagePayload`. There is no admin escape hatch, no pull-payment fallback, and no mechanism to redirect the delivery to a different address.

The same structural problem exists for the other delivery branches in the same function:

- **ERC-1155**: `IERC1155.safeTransferFrom` calls `onERC1155Received` on the recipient; a contract that returns an unexpected selector or reverts will block the transfer forever.
- **ERC-20 (`safeTransfer`)**: tokens with transfer blacklists (e.g. USDC, USDT) will revert if the recipient is blacklisted after the source-chain transfer was initiated.
- **`IBridgeToken.mint` / `ICustomMinter.mint`**: any revert in the bridge-token or custom-minter contract permanently blocks finalization. [2](#0-1) 

---

### Impact Explanation

When a user initiates a transfer on the source chain (e.g. NEAR), the NEAR bridge immediately burns or locks the user's tokens and emits an `InitTransferEvent`. The MPC then signs a `TransferMessagePayload` that is submitted to the EVM bridge via `finTransfer`. If `finTransfer` reverts indefinitely:

- The source-chain tokens are already burned/locked and cannot be recovered — there is no cancel or refund path on the NEAR side.
- The destination-chain ETH (or ERC-20) remains in the bridge contract, inaccessible to the user.
- The signed payload cannot be altered without a new MPC signature; no such re-signing mechanism exists.

Result: **permanent, irrecoverable loss of user funds** — matching the "Permanent freezing, irrecoverable lock, or unclaimable settlement" impact class. [3](#0-2) 

---

### Likelihood Explanation

Realistic triggering conditions (no privileged access required):

1. **Contract recipient without ETH fallback**: A user bridges ETH to a smart-contract wallet (e.g. a Gnosis Safe deployed without a `receive()` function). The Safe rejects ETH; every `finTransfer` call reverts.
2. **USDC/USDT blacklisting**: A user bridges USDC from NEAR to EVM. Between `initTransfer` on NEAR and `finTransfer` on EVM, the recipient address is added to the USDC blacklist (e.g. due to a regulatory freeze). `safeTransfer` reverts on every attempt.
3. **Upgradeable recipient contract**: A recipient contract is upgraded after the transfer is initiated to reject incoming ETH or tokens.

None of these scenarios require any privileged role or colluding party — they are reachable by ordinary bridge users.

---

### Recommendation

Apply a **pull-payment (escrow) pattern** for the ETH delivery path: if the low-level call fails, store the owed amount in a per-recipient claimable balance rather than reverting. Extend the same pattern to ERC-20 and ERC-1155 deliveries using try-catch or a checked transfer wrapper:

```solidity
// ETH path
(bool success, ) = payload.recipient.call{value: payload.amount}("");
if (!success) {
    claimable[payload.recipient] += payload.amount;  // pull-payment fallback
}

// ERC-20 path
try IERC20(payload.tokenAddress).transfer(payload.recipient, payload.amount) {
    // success
} catch {
    claimable[payload.tokenAddress][payload.recipient] += payload.amount;
}
```

Expose a `claim()` function so recipients can withdraw their owed amounts. This mirrors the fix applied in the referenced XC20Wrapper audit (`_safeMint` + fallback transfer).

### Proof of Concept

**Scenario: ETH bridge to a contract that rejects ETH**

1. Alice deploys a contract `Vault` on EVM that has no `receive()` or `fallback()` function.
2. Alice initiates a transfer of 1 ETH from NEAR to EVM, specifying `Vault` as the recipient. The NEAR bridge burns Alice's wETH tokens and emits `InitTransferEvent`.
3. The MPC signs a `TransferMessagePayload` with `tokenAddress = address(0)`, `recipient = address(Vault)`, `amount = 1 ETH`.
4. A relayer calls `finTransfer(signatureData, payload)` on `OmniBridge`.
5. Execution reaches line 319: `payload.recipient.call{value: 1 ether}("")` — `Vault` has no fallback, so `success = false`.
6. Line 322: `revert FailedToSendEther()` — the entire transaction reverts, including the nonce write at line 287.
7. Every subsequent retry by any relayer produces the same revert.
8. Alice's NEAR-side wETH is permanently burned. The 1 ETH sits in `OmniBridge` forever, unclaimable. [4](#0-3) [5](#0-4)
