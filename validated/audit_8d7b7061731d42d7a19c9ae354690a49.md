Audit Report

## Title
`pay()` Consumes Any Router-Held Native ETH to Settle WETH Obligations Regardless of Payer Identity, Enabling Cross-User ETH Theft — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` uses `address(this).balance` — the router's entire unattributed native ETH balance — to cover WETH swap payments before pulling from the recorded `payer`. ETH stranded on the router from a prior user's `exactOutputSingle` call (surplus `msg.value` not yet refunded) can be silently consumed to settle a completely different user's WETH obligation. The original depositor loses their ETH; the attacker receives swap output at zero cost.

## Finding Description
`pay()` in `PeripheryPayments.sol` (L69–88) contains the following WETH branch:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // payer never charged
    } else if (nativeBalance > 0) {
        ...
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

When `nativeBalance >= value`, the function deposits and forwards the router's native ETH and **never calls `safeTransferFrom(payer, …)`**. The `payer` stored in transient storage is completely ignored.

**How ETH becomes stranded:** `exactOutputSingle` (L130–147) is `payable` and stores `msg.sender` as `payer` in transient storage. The pool calls back with only the actual `amountIn`; `pay()` deposits exactly that amount, leaving `msg.value − amountIn` as native ETH on the router. If the user omits `refundETH()` from their multicall, the surplus persists across transactions. The `receive()` guard (L32–34) only blocks *direct* ETH transfers from non-WETH addresses; it does not prevent ETH from accumulating via `msg.value` in payable function calls.

**Exploit path:**
1. Victim calls `exactOutputSingle{value: 1 ether}(tokenIn=WETH, amountOut=X, amountInMaximum=1 ether)`. Pool requests `amountIn = 0.6 ETH` in the callback. `pay()` deposits 0.6 ETH; 0.4 ETH remains on the router. Victim omits `refundETH()`.
2. Attacker calls `exactInputSingle{value: 0}(tokenIn=WETH, amountIn=0.4 ETH, amountOutMinimum=0)`. Pool fires `metricOmmSwapCallback` → `_justPayCallback` (L192–199) → `pay(WETH, attacker, pool, 0.4 ETH)`. Since `address(this).balance = 0.4 ETH >= 0.4 ETH`, the router deposits the victim's ETH as WETH and transfers it to the pool. Attacker's address is never charged. Attacker receives TOKEN output; victim loses 0.4 ETH.

The same substitution occurs through `MetricOmmPoolLiquidityAdder.metricOmmModifyLiquidityCallback` (L162–178) when either pool token is WETH, as it calls `pay(token0/token1, payer, msg.sender, amount)` through the same `pay()` implementation.

The only existing guard — `_requireExpectedCallbackCaller` (L82–85 of `MetricOmmSwapRouterBase.sol`) — verifies the callback comes from the expected factory pool, but does not verify that the payer has actually provided ETH for the current call.

## Impact Explanation
**Critical/High — direct loss of user principal.** The victim loses the full stranded ETH amount (up to `amountInMaximum − amountIn` per exact-output swap). The attacker receives the corresponding swap output tokens at zero cost. The loss is exact and deterministic; no slippage, oracle condition, or special privilege is required.

## Likelihood Explanation
**Medium.** The precondition — ETH stranded on the router — arises naturally from the documented `exactOutputSingle + refundETH` multicall pattern whenever a user omits the `refundETH` step, sends excess `msg.value`, or has a partial multicall revert after ETH is deposited. Attackers can monitor the router's ETH balance on-chain and execute the theft in the next block with a single `exactInputSingle` call carrying `msg.value = 0`. The attack is repeatable for every stranded-ETH event.

## Recommendation
Remove the native-ETH shortcut from the external-payer branch. When `payer != address(this)`, always pull WETH directly from the payer:

```solidity
} else if (token == WETH) {
    if (payer == address(this)) {
        IERC20(WETH).safeTransfer(recipient, value);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

If native-ETH input must be supported, deposit `msg.value` to WETH at the top of `exactInputSingle`/`exactOutputSingle` before setting the callback context, then set `payer = address(this)` for the WETH leg. This ensures native ETH is attributed to the current caller before any callback fires.

## Proof of Concept
```
Setup:
  - Router deployed with WETH address.
  - Pool(WETH, TOKEN) exists with liquidity.
  - Victim has 1 ETH.

Step 1 — Victim strands ETH:
  victim.exactOutputSingle{value: 1 ether}(
      pool=Pool(WETH,TOKEN), tokenIn=WETH, tokenOut=TOKEN,
      amountOut=X, amountInMaximum=1 ether, recipient=victim
  )
  // Pool requests amountIn = 0.6 ETH in callback.
  // pay() deposits 0.6 ETH as WETH, transfers to pool.
  // 0.4 ETH remains on router. Victim omits refundETH().

Step 2 — Attacker steals:
  attacker.exactInputSingle{value: 0}(
      pool=Pool(WETH,TOKEN), tokenIn=WETH, tokenOut=TOKEN,
      amountIn=0.4 ETH, amountOutMinimum=0, recipient=attacker
  )
  // Pool fires callback requesting 0.4 ETH worth of WETH.
  // pay() sees address(this).balance = 0.4 ETH >= 0.4 ETH.
  // Deposits victim's 0.4 ETH as WETH, transfers to pool.
  // Attacker's payer address is never charged.
  // Attacker receives TOKEN output; victim loses 0.4 ETH.

Assert:
  attacker ETH/WETH spent: 0
  victim ETH lost: 0.4 ETH (the stranded surplus)
  router ETH balance after: 0
```
