### Title
Missing shard_id-to-chunk binding in `set_state_header` allows cross-shard state root substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` stores a peer-supplied `ShardStateSyncResponseHeader` under `StateHeaderKey(shard_id, sync_hash)` without ever asserting that the chunk embedded in the header actually belongs to `shard_id`. A malicious peer can supply shard Y's chunk (with shard Y's `prev_state_root`) while the node is syncing shard X, pass every validation gate, and cause `set_state_part` / `apply_state_part` to reconstruct shard X's trie from shard Y's nodes.

---

### Finding Description

**Entrypoint and data flow**

`StateSyncDownloader::ensure_shard_header` downloads a header from a peer and sends a `StateHeaderValidationRequest { shard_id: X, sync_hash, header: <peer-supplied> }` to the chain actor. [1](#0-0) 

The chain actor calls `ChainStateSyncAdapter::set_state_header(shard_id=X, sync_hash, peer_header)`. [2](#0-1) 

**The missing guard**

Inside `set_state_header` the five validation steps are:

1. `validate_chunk_proofs` — checks the chunk's internal hash, tx root, and receipts root. Does **not** inspect `chunk.shard_id()`. [3](#0-2) 

2. `verify_path(*sync_prev_block_header.chunk_headers_root(), chunk_proof, &ChunkHashHeight(...))` — proves the chunk hash appears somewhere in the block's chunk-headers Merkle tree. The path encodes the leaf position (shard Y's index), but the code never checks that this position equals the index of `shard_id=X`. [4](#0-3) 

3. Receipt-proof loop — hashes receipts as `ReceiptList(shard_id, receipts)` using the **caller-supplied** `shard_id=X`, not the chunk's actual shard. An attacker provides shard X's real incoming receipts (public chain data) alongside shard Y's chunk; the check passes. [5](#0-4) 

4. `validate_state_root_node(state_root_node, chunk_inner.prev_state_root())` — validates the node against the chunk's own `prev_state_root` (shard Y's root). No comparison to shard X's expected root. [6](#0-5) 

5. **Storage** — the header is written under `StateHeaderKey(shard_id=X, sync_hash)` with no shard binding check. [7](#0-6) 

**Downstream corruption**

`set_state_part(shard_id=X, ...)` reads back the header stored for X, extracts `prev_state_root` from the embedded chunk (shard Y's root), and validates incoming parts against it. [8](#0-7) 

`set_state_finalize` then applies the chunk (shard Y's) to `shard_uid` derived from `shard_id=X`, writing shard Y's trie nodes into shard X's ShardUId. [9](#0-8) 

---

### Impact Explanation

The exact corrupted value is `StateHeaderKey(X, sync_hash)` → `prev_state_root` = shard Y's root. Every subsequent operation that reads this key for shard X (part validation, trie reconstruction, `apply_state_part`) operates on shard Y's state. Shard X ends up with a wrong committed state root, breaking all account lookups, receipt routing, and chunk production for that shard on the syncing node.

---

### Likelihood Explanation

The attacker must:
- Be a reachable P2P peer of the syncing node (any NEAR node qualifies).
- Possess shard Y's trie data to supply a valid `state_root_node` (any node tracking shard Y has this).
- Assemble shard X's real incoming receipts from public chain data.

No validator, block-producer, or trusted-storage-service privilege is required. The attack is silent — `set_state_header` returns `Ok(())` and logs nothing anomalous.

---

### Recommendation

Add an explicit shard-id binding check immediately after extracting the chunk, before any other validation:

```rust
// In set_state_header, after: let chunk = shard_state_header.cloned_chunk();
let chunk_shard_id = chunk.shard_id();
if chunk_shard_id != shard_id {
    return Err(Error::Other(format!(
        "set_shard_state failed: header chunk shard_id {chunk_shard_id} \
         does not match requested shard_id {shard_id}"
    )));
}
``` [10](#0-9)

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
