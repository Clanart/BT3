### Title
Irreversible Eviction of Valid Transactions When Incoming Transaction Exceeds Freed Capacity — (`crates/apollo_mempool/src/mempool.rs`)

---

### Summary

`Mempool::try_make_space` permanently removes transactions from accounts-with-gap before confirming the incoming transaction will actually fit. If the freed space is still insufficient, `handle_capacity_overflow` returns `MempoolError::MempoolFull` and `add_tx` rejects the incoming transaction — but the already-evicted transactions are never restored. Any unprivileged sender can craft a transaction large enough to trigger eviction yet too large to be admitted, permanently expelling valid queued transactions from the mempool without sequencing them.

---

### Finding Description

`Mempool::add_tx` calls `add_tx_validations`, which calls `handle_capacity_overflow` when the pool is over capacity. `handle_capacity_overflow` calls `try_make_space`, which iterates over `accounts_with_gap` and calls `tx_pool.remove` on each transaction until enough bytes are freed or no evictable accounts remain. [1](#0-0) 

If `try_make_space` exhausts all evictable accounts without freeing `required_space` bytes, it returns `false`. [2](#0-1) 

`handle_capacity_overflow` then returns `Err(MempoolError::MempoolFull)`. `add_tx_validations` propagates this error, and `add_tx` returns early via `?` without inserting the incoming transaction. [3](#0-2) 

The transactions removed by `try_make_space` are gone. There is no rollback path. The codebase explicitly acknowledges this in the test: [4](#0-3) 

The structural parallel to the whitelist bug is exact:

| Whitelist bug | Mempool analog |
|---|---|
| `AddressSet.add` silently returns `false` for an already-registered caller | `try_make_space` returns `false` when freed bytes < required bytes |
| Minimum registrant is removed before the silent failure | Evictable transactions are removed before the failure is detected |
| Minimum registrant is gone; caller was never added | Evicted transactions are gone; incoming transaction was never added |

---

### Impact Explanation

Valid transactions that were already admitted to the mempool are permanently deleted without being sequenced. From the perspective of their senders, the transactions simply vanish — they will not appear in any block and will not be returned with an error. The mempool's admission invariant ("a transaction accepted into the pool will eventually be sequenced or explicitly rejected") is broken.

This matches the allowed impact: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The attack requires only a standard `add_tx` call. The attacker needs to:

1. Observe that the mempool contains at least one account with a nonce gap (evictable).
2. Submit a transaction whose `total_bytes()` exceeds the total bytes held by all evictable accounts combined, but whose nonce equals the attacker's account nonce (so `creating_gap = false` and `try_make_space` is called). [5](#0-4) 

The transaction signature field is unbounded in size, making it straightforward to craft a transaction of arbitrary byte length. No privileged access is required. The mempool's `accounts_with_gap` set is populated naturally during normal operation whenever a transaction with a nonce gap is submitted. [6](#0-5) 

---

### Recommendation

Perform a dry-run capacity check before committing any evictions. Specifically, compute the total evictable bytes across all `accounts_with_gap` entries before calling `tx_pool.remove` on any of them. Only proceed with actual removal if the total evictable space is sufficient to accommodate the incoming transaction. If it is not, return `Err(MempoolError::MempoolFull)` immediately without touching the pool.

Alternatively, collect the transactions to evict into a staging buffer, verify the freed space is sufficient, and only then commit the removals.

---

### Proof of Concept

The existing test `rejects_or_accepts_tx_based_on_freed_space` (case `insufficient_eviction_space`) already demonstrates the bug: [7](#0-6) 

Concretely:

1. Fill the mempool to capacity with `tx1` and `tx2` from address `0x1` (nonce 1 and 2, account nonce 0 → both create a gap → address `0x1` is in `accounts_with_gap`).
2. Submit `large_tx` from address `0x0` (nonce 0, account nonce 0 → `closing_gap = true`, `creating_gap = false`) with a signature large enough that `tx1.bytes + tx2.bytes < large_tx.bytes`.
3. `try_make_space` evicts `tx1` and `tx2` (freeing their bytes), finds no more evictable accounts, returns `false`.
4. `handle_capacity_overflow` returns `MempoolError::MempoolFull`; `large_tx` is rejected.
5. `tx1` and `tx2` are permanently gone from the pool — confirmed by `assert!(!mempool.tx_pool.contains_account(contract_address!("0x1")))`.

    pub fn try_make_space(&mut self, required_space: u64) -> bool {
        let mut total_space_freed = 0;

        while total_space_freed < required_space {
            let Some(address) = self.get_evictable_account() else {
                return false;
            };

            let txs: Vec<_> = self.tx_pool.account_txs_sorted_by_nonce(address).copied().collect();
            for tx_ref in txs.iter().rev() {
                let tx = self
                    .tx_pool
                    .remove(tx_ref.tx_hash)
                    .expect("Transaction must exist in the pool.");
                total_space_freed += tx.total_bytes();
                metric_count_evicted_txs(1);
                if total_space_freed >= required_space {
                    break;
                }
            }

            // Clean up if account is now empty.
            if !self.tx_pool.contains_account(address) {
                self.accounts_with_gap.swap_remove(&address);
            }
        }

        true

#[rstest]
#[case::insufficient_eviction_space(20, false)]
#[case::sufficient_eviction_space(8, true)]
fn rejects_or_accepts_tx_based_on_freed_space(
    #[case] signature_size: usize,
    #[case] expect_success: bool,
) {
    let tx1 = add_tx_input!(tx_hash: 1, address: "0x1", tx_nonce: 1, account_nonce: 0);
    let tx2 = add_tx_input!(tx_hash: 2, address: "0x1", tx_nonce: 2, account_nonce: 0);

    let large_signature = vec![felt!("0x0"); signature_size];
    let large_tx = AddTransactionArgs {
        tx: internal_invoke_tx(invoke_tx_args!(
            tx_hash: tx_hash!(3),
            signature: TransactionSignature(large_signature.into())
        )),
        account_state: AccountState { address: contract_address!("0x0"), nonce: nonce!(0) },
    };

    let capacity = tx1.tx.total_bytes() + tx2.tx.total_bytes();
    let mut mempool = Mempool::new(
        MempoolConfig {
            static_config: MempoolStaticConfig {
                capacity_in_bytes: capacity,
                ..Default::default()
            },
            ..Default::default()
        },
        Arc::new(FakeClock::default()),
    );

    for tx in [&tx1, &tx2] {
        add_tx(&mut mempool, tx);
    }

    if expect_success {
        add_tx(&mut mempool, &large_tx);
        assert!(mempool.tx_pool.get_by_tx_hash(large_tx.tx.tx_hash()).is_ok());
    } else {
        add_tx_expect_error(&mut mempool, &large_tx, MempoolError::MempoolFull);
    }

    // Transactions tx1 and tx2 are evicted regardless of whether large_tx is accepted or rejected.
    // We do not revert the eviction attempt even if adding large_tx ultimately fails.
    assert!(!mempool.tx_pool.contains_account(contract_address!("0x1")));
}
