### Title
ERC1155 Tokens Permanently Locked When `initTransfer1155` Is Called Without Prior `logMetadata1155` Registration — (`evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary

`initTransfer1155` accepts and locks ERC1155 tokens using a deterministically derived pseudo-address (`deterministicToken`) without verifying that the `multiTokens` mapping for that address has been populated by a prior `logMetadata1155` call. Because `finTransfer` on EVM releases ERC1155 tokens only when `multiTokens[payload.tokenAddress].tokenAddress != address(0)`, and because the NEAR bridge rejects the inbound proof when the token is unregistered, the locked ERC1155 tokens become irrecoverable.

### Finding Description

`deriveDeterministicAddress` computes a 20-byte pseudo-address from `keccak256(abi.encodePacked(tokenAddress, tokenId))`: [1](#0-0) 

`logMetadata1155` is the only function that writes to `multiTokens[deterministicToken]`: [2](#0-1) 

`initTransfer1155` computes the same `deterministicToken`, transfers the ERC1155 tokens into the bridge, and emits `InitTransfer` with `deterministicToken` as the token address — but **never checks** whether `multiTokens[deterministicToken]` is already populated: [3](#0-2) 

On the release side, `finTransfer` dispatches ERC1155 tokens only when `multiToken.tokenAddress != address(0)`. If the mapping is absent, execution falls through to the ERC20 `safeTransfer` branch, which calls `IERC20(deterministicToken).safeTransfer(...)` — a call to a non-contract address that reverts: [4](#0-3) 

On the NEAR side, `fin_transfer_callback` looks up `token_decimals` for the token address extracted from the EVM proof. Because `deterministicToken` is not a registered NEAR token, the lookup panics with `TokenDecimalsNotFound`, causing the proof submission to fail: [5](#0-4) 

There is no admin rescue function for ERC1155 tokens in `OmniBridge.sol`. The only release path is `finTransfer` with a valid MPC signature from NEAR, which NEAR will never produce for a transfer it rejected.

### Impact Explanation

Any ERC1155 tokens sent via `initTransfer1155` without a prior `logMetadata1155` call are permanently locked in the bridge contract with no recovery path. This satisfies the allowed impact: **Permanent freezing / irrecoverable lock of user funds in bridge flows**.

### Likelihood Explanation

`initTransfer1155` is a public, permissionless function. `logMetadata1155` is a separate permissionless call that a user or integrator can easily omit — the contract imposes no on-chain ordering requirement. Any user who calls `initTransfer1155` directly (e.g., via a script or a frontend that skips the metadata step) triggers the lock. The likelihood is **High**.

### Recommendation

Add a guard at the top of `initTransfer1155` that requires the `multiTokens` mapping to already be populated for the derived address:

```solidity
function initTransfer1155(...) external payable whenNotPaused(PAUSED_INIT_TRANSFER) {
    currentOriginNonce += 1;
    if (fee >= amount) revert InvalidFee();

    address deterministicToken = deriveDeterministicAddress(tokenAddress, tokenId);

    // Require logMetadata1155 to have been called first
    if (multiTokens[deterministicToken].tokenAddress == address(0)) {
        revert ERC1155MappingMismatch(); // or a dedicated error
    }
    ...
}
```

Alternatively, `initTransfer1155` can auto-register the mapping if absent (mirroring the logic in `logMetadata1155`), eliminating the ordering dependency entirely.

### Proof of Concept

1. Deploy `OmniBridge` on a local fork.
2. Mint ERC1155 token ID `7` to `user`.
3. `user` approves the bridge and calls `initTransfer1155(erc1155, 7, 1, 0, 0, "recipient.near", "")` — **without** calling `logMetadata1155` first.
4. Observe: `erc1155.balanceOf(bridge, 7) == 1`; `multiTokens[deterministicToken].tokenAddress == address(0)`.
5. Attempt `finTransfer` with `payload.tokenAddress = deterministicToken` and a valid MPC signature: the call reverts because `multiToken.tokenAddress == address(0)` and `IERC20(deterministicToken).safeTransfer(...)` fails on a non-contract address.
6. The ERC1155 token is permanently locked with no recovery path. [3](#0-2) [4](#0-3) [5](#0-4)
