### Title
`MerkleBlob::batch_insert` Bypasses Duplicate Key/Hash Checks, Silently Corrupting DataLayer Merkle Tree Root - (File: crates/chia-datalayer/src/merkle/blob.rs)

### Summary

`MerkleBlob::batch_insert` contains a fast-path loop that writes leaf nodes directly to the blob via `insert_entry_to_blob` without performing the duplicate-key or duplicate-hash checks that the single-item `insert` function enforces. When a batch contains a key already present in the tree, or when the same key appears twice within the batch, both copies are silently written into the blob. The in-memory `BlockStatusCache` silently overwrites its index entry for the duplicated key, leaving the old leaf node orphaned in the blob. The resulting tree has an incorrect root hash and produces invalid proofs of inclusion. If the corrupted blob is persisted and reloaded, `MerkleBlob::new` detects the duplicate and returns `Error::KeyAlreadyPresent`, permanently bricking the DataLayer store.

### Finding Description

`MerkleBlob::insert` (the single-item path) guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::batch_insert` (lines 570–657) has two code paths:

1. **Bootstrap path** (lines 578–585): when the tree has ≤ 1 leaf, the last two items are popped from the vector and processed through `self.insert(...)`, which **does** perform the duplicate checks.

2. **Fast path** (lines 587–603): all remaining items are written directly via `self.insert_entry_to_blob(new_leaf_index, &new_block)?` — **no duplicate key or hash check is performed**.

For any tree that already has ≥ 2 leaves (the common production case), **every item** in the batch bypasses the duplicate guards.

`insert_entry_to_blob` (lines 1013–1030) calls `self.block_status_cache.add_leaf(index, leaf)`, which calls `self.key_to_index.insert(leaf.key, index)` — a `HashMap::insert` that silently overwrites the existing entry without returning an error. The old leaf node remains in the blob at its original index, but the cache no longer tracks it. The tree is now structurally inconsistent: two leaf nodes share the same key, the root hash is computed over both, and `get_proof_of_inclusion` returns a proof for the new leaf while the old leaf is unreachable through the cache.

When the corrupted blob is later passed to `MerkleBlob::new`, `BlockStatusCache::new` (lines 96–130) iterates all nodes and detects the duplicate key, returning `Err(Error::KeyAlreadyPresent())`. The DataLayer store becomes permanently unloadable.

### Impact Explanation

- **Corrupted tree root**: The root hash is computed over a tree containing two leaf nodes with the same key. Any downstream consumer of the root hash (e.g., a DataLayer proof verifier) receives a hash that does not correspond to the intended key-value mapping.
- **Invalid proofs of inclusion**: `get_proof_of_inclusion` for the duplicated key returns a proof anchored to the new leaf's position. The old leaf's lineage is unreachable, so proofs for other keys that share ancestors with the old leaf may also be invalid.
- **Permanent store corruption**: Persisting and reloading the blob causes `MerkleBlob::new` to fail, permanently bricking the DataLayer store.

This matches the allowed High impact: *DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state.*

### Likelihood Explanation

Any caller of `batch_insert` (or its Python binding `py_batch_insert`) that supplies a batch containing a key already present in the tree, or a key that appears more than once within the batch, triggers the corruption. The Python binding is directly exposed to DataLayer application code. DataLayer store updates are constructed from user-supplied key-value data; if the application layer does not deduplicate before calling `batch_insert`, an attacker who can submit DataLayer updates can trigger this path.

### Recommendation

Add the same duplicate guards to the fast path of `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing fast-path logic
}
```

Alternatively, pre-validate the entire batch for internal duplicates and against the existing tree before writing any nodes.

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

let mut blob = MerkleBlob::new(vec![]).unwrap();

// Seed the tree with ≥ 2 leaves so the fast path is taken
for i in 0i64..3 {
    blob.insert(KeyId(i), ValueId(i),
        &Hash(Bytes32::new([i as u8; 32])),
        InsertLocation::Auto {}).unwrap();
}

let root_before = blob.get_root_hash().unwrap();

// batch_insert with a key already in the tree (KeyId(0))
blob.batch_insert(vec![
    ((KeyId(0), ValueId(99)), Hash(Bytes32::new([0u8; 32]))),  // duplicate key!
    ((KeyId(10), ValueId(10)), Hash(Bytes32::new([10u8; 32]))),
]).unwrap();  // succeeds silently — no error

let root_after = blob.get_root_hash().unwrap();
assert_ne!(root_before, root_after);  // root is now wrong

// Persisting and reloading the blob fails permanently
let saved = blob.blob.clone();
let reload = MerkleBlob::new(saved.to_vec());
assert!(reload.is_err());  // Error::KeyAlreadyPresent — store is bricked
```

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
