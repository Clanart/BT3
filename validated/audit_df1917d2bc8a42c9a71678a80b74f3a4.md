### Title
Division-by-Zero Panic in `NewDynamicFeeChecker` Halts Chain via Zero-Gas Cosmos SDK Transaction - (File: ante/evm/fee_checker.go)

### Summary
`NewDynamicFeeChecker` in `ante/evm/fee_checker.go` performs an unchecked integer division `fee / gas` at line 83 and `getTxPriority` performs `QuoRaw(gas)` at line 157. When a Cosmos SDK transaction carries `gas = 0`, both divisions panic, crashing the node during ante handler execution. No prior decorator in the Cosmos ante handler chain rejects a zero-gas Cosmos SDK tx before this point.

### Finding Description
`NewDynamicFeeChecker` is the `TxFeeChecker` wired into the Cosmos SDK ante handler chain (not the EVM ante handler chain) via `newCosmosAnteHandler` and `newLegacyCosmosAnteHandlerEip712`.

Inside the returned closure, after retrieving the gas value from the transaction:

```go
gas := feeTx.GetGas()          // returns 0 for a zero-gas tx
feeCoins := feeTx.GetFee()
fee := feeCoins.AmountOf(denom)

feeCap := fee.Quo(sdkmath.NewIntFromUint64(gas))  // Quo(0) → panic
```

`sdkmath.Int.Quo` wraps `big.Int.Quo`, which panics unconditionally when the divisor is zero. A second panic site exists in the fallback path `getTxPriority`:

```go
gasPrice := fee.Amount.QuoRaw(gas)  // QuoRaw(0) → panic
```

For EVM transactions, `EthereumTx.Validate()` explicitly rejects `gas == 0` before the EVM ante handler runs. No equivalent guard exists for Cosmos SDK transactions before `DeductFeeDecorator` invokes `NewDynamicFeeChecker`.

The Cosmos ante handler chain is:
1. `SetUpContextDecorator` — sets an **infinite** gas meter during `CheckTx`, so `ConsumeGasForTxSizeDecorator` does not panic first
2. `ValidateBasicDecorator` — calls `tx.ValidateBasic()`, which does not enforce `gas > 0` for Cosmos SDK txs
3. `ConsumeGasForTxSizeDecorator` — uses the infinite meter during `CheckTx`, no panic
4. `DeductFeeDecorator` → `NewDynamicFeeChecker` → **panic**

### Impact Explanation
A panic in the ante handler propagates as an unrecovered runtime panic, crashing the node process. Because the panic is deterministic (same tx → same panic on every validator), all validators processing the same block crash simultaneously, halting the chain. This satisfies: **Critical — valid unprivileged transaction can halt the chain or cause deterministic validator consensus failure**.

### Likelihood Explanation
Any user can craft a Cosmos SDK transaction (e.g., a `MsgSend` or any governance message) with `gas = 0` and broadcast it. No special privilege, key, or validator access is required. The tx passes `ValidateBasicDecorator` and reaches the vulnerable division. The attack is trivially reproducible and requires only a single transaction.

### Recommendation
Add an explicit zero-gas guard at the top of `NewDynamicFeeChecker` before any arithmetic:

```go
gas := feeTx.GetGas()
if gas == 0 {
    return nil, 0, errorsmod.Wrap(errortypes.ErrInvalidRequest, "gas limit must not be zero")
}
```

Apply the same guard in `getTxPriority` (or ensure it is never called with `gas == 0`). Additionally, add a zero-gas check in `checkTxFeeWithValidatorMinGasPrices` before `QuoRaw(gas)` is reached.

### Proof of Concept

1. Construct any valid Cosmos SDK transaction (e.g., `MsgSend`) with `GasLimit = 0` and any non-zero fee amount in the EVM denom.
2. Broadcast the transaction to a node with London HF enabled (`baseFee != nil`).
3. During `CheckTx`, the ante handler chain reaches `NewDynamicFeeChecker`.
4. `fee.Quo(sdkmath.NewIntFromUint64(0))` executes → `big.Int.Quo` panics with a division-by-zero runtime error.
5. The node crashes. Because the panic is deterministic, all validators crash on the same block, halting the chain.
