### Title
`ProofOfInclusion::valid()` Is a Self-Referential Tautology That Never Validates Against a Trusted Root — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary
`ProofOfInclusion::valid()` only checks internal consistency of the proof chain. Its final comparison is a mathematical tautology — it always returns `true` regardless of whether the proof corresponds to any real tree root. An untrusted party can construct a `ProofOfInclusion` with an arbitrary `node_hash` (representing any key-value pair) and internally consistent layers, and `valid()` will accept it. Any DataLayer verifier that calls `proof.valid()` without separately checking `proof.root_hash() == known_trusted_root` accepts forged inclusion proofs.

### Finding Description

`ProofOfInclusion::valid()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← derived from the proof itself
    } else {
        self.node_hash
    }
}

pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;   // ← existing_hash = layer.combined_hash
    }
    existing_hash == self.root_hash()      // ← always true: both sides = last layer.combined_hash
}
``` [1](#0-0) 

After the loop, `existing_hash` equals the last `layer.combined_hash` (the loop only continues when `calculated_hash == layer.combined_hash`, so `existing_hash = calculated_hash = layer.combined_hash`). Meanwhile, `self.root_hash()` also returns `self.layers.last().combined_hash`. The final comparison `existing_hash == self.root_hash()` is therefore a tautology — both sides are the same value extracted from the proof itself. [2](#0-1) 

For the zero-layers case: `existing_hash = self.node_hash` and `root_hash() = self.node_hash`, so it is also always `true`.

The method is exposed directly to Python callers via `py_valid`: [3](#0-2) 

And declared in the Python type stub as a standalone verification primitive: [4](#0-3) 

`ProofOfInclusion` also supports `from_bytes()` / `from_bytes_unchecked()` deserialization, meaning an untrusted party can supply arbitrary bytes that deserialize into a `ProofOfInclusion` with any `node_hash` and any internally consistent layer chain. [5](#0-4) 

### Impact Explanation

A DataLayer client that receives a `ProofOfInclusion` from an untrusted peer and calls `proof.valid()` to verify it will accept any internally consistent proof, regardless of whether the claimed `node_hash` (key-value pair) is actually present in the tree. The attacker can prove inclusion of any key-value pair in any tree root they choose, because `valid()` never checks against an external trusted root. This directly matches the allowed High impact: "DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."

### Likelihood Explanation

The method name `valid()` strongly implies complete proof validation. There is no documentation warning callers to also check `proof.root_hash() == known_root`. The fuzz target itself only calls `assert!(proof.valid())` without an external root check, reinforcing the incorrect usage pattern: [6](#0-5) 

Any Python consumer of the DataLayer API that follows the natural usage pattern of calling `proof.valid()` is vulnerable.

### Recommendation

`valid()` must accept a trusted external root hash as a parameter and compare the computed root against it:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root   // compare against caller-supplied trusted root
}
```

The current `valid()` method (which only checks internal consistency) should either be removed or renamed to `is_internally_consistent()` with a clear doc comment that it does **not** validate against any tree root.

### Proof of Concept

```python
from chia_rs.datalayer import (
    ProofOfInclusion, ProofOfInclusionLayer, MerkleBlob, KeyId, ValueId
)
import hashlib

# Build a real tree with one entry
blob = MerkleBlob(bytearray())
real_key   = KeyId(1)
real_value = ValueId(1)
real_hash  = bytes(range(32))
blob.insert(real_key, real_value, real_hash)
blob.calculate_lazy_hashes()
real_root = blob.get_root_hash()

# Forge a proof claiming key=999 is in the tree
fake_node_hash = hashlib.sha256(b"fake_leaf").digest()
# No layers needed: valid() returns True for empty layers too
forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[])

assert forged_proof.valid()          # passes — tautology
assert forged_proof.root_hash() != real_root  # forged root != real root
# A verifier calling only proof.valid() accepts the forgery
