### Title
Unrestricted BOOTSTRAP Declare Path Bypasses Signature Verification and Fee Payment — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_declare_transaction` contains a special code path gated only on `sender_address == 'BOOTSTRAP'`, `nonce == 0`, `version == 3`, and zero resource bounds. Because `'BOOTSTRAP'` is merely a Cairo felt literal (the ASCII encoding `0x424F4F545354524150`), any unprivileged user can craft a declare transaction that satisfies these conditions. The OS then declares the class hash with no signature verification, no nonce check, and no fee charge — a direct protocol-level bypass reachable by any transaction sender.

---

### Finding Description

In `execute_declare_transaction`, after computing the transaction hash and filling `tx_info`, the following branch is evaluated:

```cairo
if (sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3) {
    let max_possible_fee = compute_max_possible_fee(tx_info=tx_info);
    if (max_possible_fee == 0) {
        assert_not_zero(compiled_class_hash);
        dict_update{dict_ptr=contract_class_changes}(
            key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
        );
        %{ SkipTx %}
        return ();
    }
}
``` [1](#0-0) 

When this branch is taken, the function returns immediately, skipping:

- **`check_and_increment_nonce`** (line 779) — no nonce verification against on-chain account state
- **`run_validate`** (line 804) — no `__validate_declare__` entry point execution, no signature check
- **`charge_fee`** (line 822) — no fee deducted from any account [2](#0-1) 

The only guard is `assert_not_zero(compiled_class_hash)` and the dict's `prev_value=0` constraint (enforcing one-time declaration per class hash). There is no check that `sender_address` corresponds to a deployed account contract, no cryptographic proof of ownership, and no restriction to a trusted deployer address.

The `sender_address == 'BOOTSTRAP'` comparison is a felt equality check. In Cairo, `'BOOTSTRAP'` is the felt value `0x424F4F545354524150`. Any user can submit a declare transaction specifying this value as the sender address. The OS does not verify that a contract is deployed at this address before entering the bootstrap path.

The Sierra class hash is still verified via `finalize_class_hash` before this branch: [3](#0-2) 

So the attacker must provide a valid Sierra class hash pre-image, but the `compiled_class_hash` mapping is written to state with no further constraint beyond non-zero.

---

### Impact Explanation

**Direct loss of funds (Critical):** The fee mechanism is entirely bypassed. An attacker can declare an arbitrary number of valid Sierra classes without paying any fees. The fee collector — which receives fees via `charge_fee` calling an ERC20 transfer to `block_context.block_info_for_execute.sequencer_address` — receives nothing for these declarations. [4](#0-3) 

**Class declaration front-running:** Because `prev_value=0` enforces that each class hash can only be declared once, an attacker who races to declare a class first permanently blocks the legitimate declarer. If the attacker supplies a `compiled_class_hash` that differs from the one the legitimate declarer intended, any contract subsequently deployed with that class hash will execute against the attacker-chosen compiled class mapping — a state corruption that is irreversible without a protocol-level override mechanism.

---

### Likelihood Explanation

High. The attack requires no special privileges, no cryptographic keys, and no deployed account contract. The attacker only needs to:

1. Set `sender_address` to the felt literal `'BOOTSTRAP'` (`0x424F4F545354524150`)
2. Set `nonce = 0`, `version = 3`
3. Set all resource bounds to zero (so `compute_max_possible_fee` returns 0)
4. Provide a valid Sierra class hash pre-image

These are all public, statically known values. The OS is the security boundary for the StarkNet protocol — it must accept any transaction the sequencer includes, and the L1 verifier accepts whatever the OS proof attests to. A user who can submit transactions to the sequencer (i.e., any participant) can trigger this path.

---

### Recommendation

1. **Remove the BOOTSTRAP path entirely** if it is no longer needed for production. If it is needed for initial system bootstrapping, restrict it to a hardcoded trusted address (e.g., the program upgrade authority or a specific deployer public key), analogous to the remediation applied in the gorples-bridge report.
2. **Require a valid signature** even in the bootstrap path, or gate it on a hardcoded felt constant that corresponds to a key under operator control rather than an arbitrary ASCII string.
3. **Add a mechanism for ownership transfer** so that any misconfigured class declaration can be corrected by a higher authority.

---

### Proof of Concept

1. Attacker constructs a declare transaction with:
   - `sender_address = 0x424F4F545354524150` (felt encoding of `'BOOTSTRAP'`)
   - `nonce = 0`
   - `version = 3`
   - `resource_bounds`: all three entries (`L1_GAS`, `L2_GAS`, `L1_DATA_GAS`) with `max_amount = 0` and `max_price_per_unit = 0`, so `compute_max_possible_fee` returns `0`
   - A valid Sierra class hash (with a valid pre-image satisfying `finalize_class_hash`)
   - Any non-zero `compiled_class_hash`

2. The sequencer includes this transaction in a block.

3. The OS evaluates the condition at line 764: `sender_address == 'BOOTSTRAP' and tx_info.nonce == 0 and tx_info.version == 3` → **TRUE**. Then `compute_max_possible_fee(tx_info) == 0` → **TRUE**.

4. The OS executes `dict_update` writing `class_hash → compiled_class_hash` into `contract_class_changes` and returns immediately — skipping `check_and_increment_nonce`, `run_validate`, and `charge_fee`.

5. The class is now declared in the protocol state. No fee was paid. No signature was verified. The fee collector received nothing. The legitimate declarer is permanently blocked from declaring the same class hash. [1](#0-0)
