### Title
Precommit Vote Signature Omits `round`, `height`, and `vote_type` — Cross-Round Replay Allows Forged Quorum Contribution - (File: `crates/apollo_signature_manager/src/signature_manager.rs`)

---

### Summary

`build_precommit_vote_message_digest` constructs the ECDSA message as `PRECOMMIT_VOTE || block_hash` only. The `round`, `height`, `vote_type`, and `chain_id` fields present in the `Vote` struct are not covered by the signature. This is the direct sequencer analog of the TimelockTokenPool nonce-omission: a signature that is valid for one context (round R, vote type Precommit) is cryptographically valid for a different context (round R+1, vote type Prevote) for the same `block_hash`. Additionally, signature verification is not yet enforced in the production vote-handling path (`// TODO(Asmaa): verify the signature`), meaning the structural flaw is immediately reachable.

---

### Finding Description

**Signing function** (`crates/apollo_signature_manager/src/signature_manager.rs`, lines 138–145):

```rust
fn build_precommit_vote_message_digest(block_hash: BlockHash) -> MessageDigest {
    let block_hash = block_hash.to_bytes_be();
    let mut message = Vec::with_capacity(PRECOMMIT_VOTE.len() + block_hash.len());
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash);
    MessageDigest(blake2s_to_felt(&message))
}
```

The signed payload is `blake2s("PRECOMMIT_VOTE" || block_hash_bytes)`. The `Vote` struct that carries this signature also contains `height`, `round`, `vote_type`, and `voter` fields, none of which are committed to by the signature:

```rust
pub struct Vote {
    pub vote_type: VoteType,   // NOT signed
    pub height: BlockNumber,   // NOT signed
    pub round: Round,          // NOT signed
    pub proposal_commitment: Option<ProposalCommitment>,  // = block_hash, IS signed
    pub voter: ContractAddress, // NOT signed (implicit via public key only)
    pub signature: RawSignature,
}
```

Signature verification is explicitly deferred. `verify_precommit_vote_signature` is defined and tested in isolation but is never called in any production code path — confirmed by `grep`: the function appears only in `signature_manager.rs` (definition) and `signature_manager_test.rs` (tests).

**Replay scenario** (Tendermint re-propose path):

1. At height H, round R, validator V signs a precommit for `block_hash X`. The signature `S` covers only `PRECOMMIT_VOTE || X`.
2. Round R fails to reach quorum. The block is re-proposed in round R+1 with the same `proposal_commitment X` (Tendermint valid-round repropose, `SMRequest::Repropose`).
3. An attacker replays the captured vote `{height: H, round: R+1, vote_type: Precommit, voter: V, proposal_commitment: X, signature: S}`.
4. Because the signature covers only `X`, it verifies correctly against V's public key even though V never precommitted in round R+1.
5. The state machine counts this as a legitimate precommit from V in round R+1, inflating the quorum count.

The same signature can also be replayed with `vote_type: Prevote` since `vote_type` is not in the signed message, allowing a precommit to be counted as a prevote.

---

### Impact Explanation

A network peer who observes a validator's precommit for a block that is subsequently re-proposed can replay that signature in the new round. This allows a coalition that controls fewer than the required 2/3 quorum weight to reach a decision by replaying captured signatures from honest validators. The wrong `ProposalCommitment` (partial block hash) is then passed to `decision_reached`, which calls `batcher.decision_reached`, commits the proposal to storage via `commit_proposal`, and writes the resulting `block_hash` and `global_root` to the Patricia trie — permanently anchoring a fraudulently-decided block into the state commitment chain.

**Matching impact scope**: *High — Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload.*

---

### Likelihood Explanation

- **Trigger is unprivileged**: any node that can observe P2P vote traffic (all peers) can capture and replay signatures.
- **Condition is reachable in normal operation**: Tendermint re-proposes the same block in a new round whenever precommit quorum is not reached in round R. This is a standard liveness path, not an edge case.
- **No existing check prevents it**: `handle_vote` performs a height check and a committee-membership check, but no signature verification. Even after verification is added, the structural omission of `round` and `vote_type` from the signed message will persist unless the digest construction is fixed.

---

### Recommendation

1. Include `height`, `round`, `vote_type`, and `chain_id` in the signed message inside `build_precommit_vote_message_digest` (rename to a more general `build_vote_message_digest`):

```rust
fn build_vote_message_digest(
    vote_type: VoteType,
    height: BlockNumber,
    round: Round,
    block_hash: BlockHash,
    chain_id: ChainId,
) -> MessageDigest { ... }
```

2. Implement the deferred signature verification in `handle_vote` (`// TODO(Asmaa): verify the signature`) using the committee's public keys before the vote is forwarded to the state machine.
