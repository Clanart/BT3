### Title
Missing Duplicate Key/Hash Validation in `batch_insert` Bypasses Uniqueness Checks, Corrupting DataLayer Merkle Tree Root — (`File: crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert()` skips the duplicate-key and duplicate-hash guards that `MerkleBlob::insert()` enforces. When the tree already contains ≥ 2 leaves, every item in the batch is written directly to the blob without any uniqueness check. Supplying duplicate keys — either within the batch itself or matching keys already in the tree — silently produces orphaned leaf nodes, an inconsistent `BlockStatusCache`, and a corrupted Merkle root hash.

---

### Finding Description

`MerkleBlob::insert()` enforces two guards before writing a leaf:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
``` [1](#0-0) 

`batch_insert()` takes a completely different code path for items beyond the bootstrap phase. When `leaf_count >= 2`, **all** items in the batch skip `insert()` entirely and go straight to `insert_entry_to_blob()`:

```rust
// crates/chia-datalayer/src/merkle/blob.rs  lines 587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // ← no duplicate check
    indexes.push(new_leaf_index);
}
``` [2](#0-1) 

`insert_entry_to_blob()` calls `block_status_cache.add_leaf()`, which uses `HashMap::insert` — silently overwriting any existing cache entry for the same key or hash:

```rust
// lines 188-193
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // silent overwrite
    self.leaf_hash_to_index.insert(leaf.hash, index);   // silent overwrite
}
``` [3](#0-2) 

When `leaf_count <= 1`, only the first two items are routed through `insert()` (with checks); all remaining items still bypass the guards: [4](#0-3) 

The Python binding `py_batch_insert` passes caller-supplied data directly to `batch_insert` with no pre-filtering: [5](#0-4)

### Impact Explanation

When a duplicate key `K` is inserted via `batch_insert`:

1. A second `LeafNode` for `K` is written to the blob at a new index `I2`.
2. The cache entry `K → I1` is silently overwritten with `K → I2`.
3. The original leaf at `I1` remains physically present in the blob and is still referenced by its parent internal node — it becomes an **orphaned node** that is unreachable through the cache.
4. The subtree-building phase (`insert_subtree_at_key`) attaches the new batch subtree (containing `I2`) to the existing tree, so both `I1` and `I2` are now part of the tree structure.
5. The root hash is computed over a tree that contains two leaves for the same key, making the root hash **non-canonical and unverifiable**.
6. `get_proof_of_inclusion(K)` returns a proof anchored at `I2`, but the actual root hash includes `I1`; the proof is structurally invalid against any externally computed root.
7. `check_integrity()` detects the inconsistency (`leaf_count ≠ key_to_index_cache_length`), but only if called after the fact — the corruption is already committed. [6](#0-5) 

This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic corrupts tree roots and lets untrusted input prove invalid state.**

---

### Likelihood Explanation

The Python-exposed `batch_insert` is the primary bulk-load API for DataLayer synchronization. A malicious DataLayer peer that sends a sync payload containing duplicate key-value pairs — or keys already present in the receiver's tree — triggers this path with no special privileges. The attacker only needs to craft a list with a repeated `KeyId`. The condition `leaf_count >= 2` is satisfied by any non-trivial tree, making the vulnerable fast path the common case in production.

---

### Recommendation

Add the same duplicate guards at the top of the fast-path loop in `batch_insert`, mirroring `insert()`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing fast-path code
}
```

Alternatively, pre-deduplicate the input vector and cross-check against the existing cache before entering the fast path, to preserve the performance intent of `batch_insert` while still rejecting invalid inputs.

### Proof of Concept

```rust
use chia_datalayer::{Hash, InsertLocation, KeyId, MerkleBlob, ValueId};
use chia_protocol::Bytes32;

let mut blob = MerkleBlob::new(vec![]).unwrap();

// Seed the tree with 3 leaves so leaf_count >= 2 and the fast path is active
for i in 0i64..3 {
    blob.insert(KeyId(i), ValueId(i), &sha256_num(&i), InsertLocation::Auto {}).unwrap();
}

// batch_insert with a key (KeyId(0)) already present in the tree
// and a duplicate within the batch itself (KeyId(99) appears twice)
let batch = vec![
    ((KeyId(0),  ValueId(99)), sha256_num(&0i64)),   // key already in tree
    ((KeyId(99), ValueId(1)),  sha256_num(&99i64)),
    ((KeyId(99), ValueId(2)),  sha256_num(&99i64)),  // duplicate within batch
];
blob.batch_insert(batch).unwrap(); // succeeds — no error raised

// Tree is now corrupted: two leaf nodes for KeyId(0) and KeyId(99) exist in the blob
// check_integrity() will report IntegrityKeyToIndexCacheLength mismatch
blob.check_integrity().expect_err("tree is corrupted");

    pub fn py_batch_insert(
        &mut self,
        keys_values: Vec<(KeyId, ValueId)>,
        hashes: Vec<Hash>,
    ) -> PyResult<()> {
        if keys_values.len() != hashes.len() {
            Err(Error::UnmatchedKeysAndValues(
                keys_values.len(),
                hashes.len(),
            ))?;
        }

        self.batch_insert(zip(keys_values, hashes).collect())?;
