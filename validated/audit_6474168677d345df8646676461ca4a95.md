### Title
Simulation/Execution Divergence via `ErrStateConflict`: `eth_call` Reports Success While On-Chain Execution Fails with `VmError` and Consumes Full Gas — (`x/evm/keeper/state_transition.go`, `x/evm/statedb/statedb.go`)

### Summary
When a transaction triggers both an EVM `SSTORE` and a native action (via `ExecuteNativeAction`) that write different values to the same storage slot, `stateDB.Commit()` returns `ErrStateConflict`. This is surfaced as a `VmError` (tx included, status=0, full gas consumed). However, `eth_call` and `eth_estimateGas` pass `commit=false`, so `Commit()` is never called and the conflict is never detected — both simulations return success. The divergence is explicitly acknowledged in a code comment.

### Finding Description
`ApplyMessageWithConfig` accepts a `commit bool` parameter. When `commit=false` (simulation paths), the entire `stateDB.Commit()` block is skipped. When `commit=true` (real execution via `ApplyTransaction`), `Commit()` runs a conflict-detection loop over all EVM-dirty storage keys. If a native action (via `ExecuteNativeAction`) wrote a different value to the same key, `ErrStateConflict` is returned and converted to a `VmError` — the transaction is included in the block with status=0 and the sender's gas is consumed.

The conflict detection at `statedb.go:766–788` compares `keeper.GetState(s.ctx, addr, key)` (which reflects the native action's committed write into the branched `cacheMS`) against `originStorage[key]` and `dirtyStorage[key]`. If `store != origin && store != dirty`, the conflict is raised. This check is entirely absent from the `commit=false` path.

### Impact Explanation
A user who calls `eth_call` or `eth_estimateGas` before submitting a transaction receives a success response. When the transaction is actually executed on-chain, it fails with `VmError=ErrStateConflict` (status=0). The user pays full gas for a transaction that simulation predicted would succeed. This is a direct mis-accounting of user funds and falls squarely within the "High" impact category: *"EVM state transition, gas refund, fee market, ante handler, mempool, or proposal handling bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."*

### Likelihood Explanation
Requires a chain deployment with custom precompiles that use `ExecuteNativeAction` to write to EVM contract storage slots (e.g., a bank or staking precompile that updates a balance slot). An unprivileged attacker who controls a contract calling such a precompile can reliably reproduce the divergence. The precondition (native action hooks enabled) is explicitly supported and documented in Ethermint's precompile creation guide. The attacker does not need any privileged role — only the ability to deploy a contract and submit transactions.

### Recommendation
The simulation paths (`EthCall`, `EstimateGas`) should replicate the conflict-detection logic even when `commit=false`. One approach: after EVM execution completes (but before discarding the stateDB), run the same conflict-detection loop from `Commit()` and, if a conflict is found, set `VmError` on the result so that simulations accurately predict the on-chain outcome. Alternatively, document this as a known limitation and surface a distinct error in `EstimateGas` so callers are not misled.

2. Deploy contract `C` with a function `f()` that: (a) calls precompile `P` (triggering the native write of `0xBEEF` to slot `S`), then (b) executes `SSTORE S, 0xDEAD` (EVM writes `0xDEAD` to slot `S`).
3. Call `eth_call` for `C.f()` → no `Commit()` → no conflict detected → returns success.

func (s *StateDB) ExecuteNativeAction(contract common.Address, converter EventConverter, action func(ctx sdk.Context) error) error {
	prevStore := s.cacheMS
	prevLayerCount := len(s.cacheLayers)

	nextStore, ok := s.cacheMS.CacheMultiStore().(cachemulti.Store)
	if !ok {
		panic("expect nested CacheMultiStore result to be cachemulti.Store")
	}

	eventManager := sdk.NewEventManager()
	actionCtx := s.ctx.WithMultiStore(nextStore).WithEventManager(eventManager)

	if err := action(actionCtx); err != nil {
		return err
	}

	s.cacheMS = nextStore
	s.cacheLayers = append(s.cacheLayers, nextStore)
	s.ctx = s.ctx.WithMultiStore(nextStore)

	events := eventManager.Events()
	s.emitNativeEvents(contract, converter, events)
	s.nativeEvents = s.nativeEvents.AppendEvents(events)
	s.journal.append(nativeChange{
		previousStore:      prevStore,
		previousLayerCount: prevLayerCount,
		events:             len(events),
	})
