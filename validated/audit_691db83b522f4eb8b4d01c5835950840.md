### Title
Missing Class Hash Validation in `execute_replace_class` Allows Permanent Contract Freezing — (`crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not validate that the new class hash provided by a contract corresponds to a previously declared contract class. This is the direct analog of the M-05 vulnerability class: just as an unprotected initializer allowed setting a malicious strategy address that could be used to destroy a logic contract via `delegatecall`, the unvalidated `replace_class` syscall allows a contract to replace its own class with an undeclared (non-existent) class hash, permanently rendering the contract inoperable and freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall:

```cairo
// Replaces the class.
func execute_replace_class{
    range_check_ptr,
    syscall_ptr: felt*,
    contract_state_changes: DictAccess*,
    revert_log: RevertLogEntry*,
}(contract_address: felt) {
    alloc_locals;
    let request = cast(syscall_ptr + RequestHeader.SIZE, ReplaceClassRequest*);

    // Reduce gas.
    let success = reduce_syscall_gas_and_write_response_header(
        total_gas_cost=REPLACE_CLASS_GAS_COST, request_struct_size=ReplaceClassRequest.SIZE
    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The comment at line 898 explicitly acknowledges the missing check:

> `// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.`

The OS unconditionally writes `class_hash = request.class_hash` into the contract state without verifying that this hash exists in the `contract_class_changes` dictionary (i.e., was declared in the current block) or in the existing class tree. The same missing validation exists in the deprecated path in `deprecated_execute_syscalls.cairo` at `execute_replace_class` (lines 307–329), which also writes the caller-supplied class hash directly into the state without any existence check.

The `execute_replace_class` function in `syscall_impls.cairo` is invoked from `execute_syscalls.cairo` at line 196 for the new execution path, and from `deprecated_execute_syscalls.cairo` at line 677 for the deprecated path. Both paths are reachable by any contract during normal transaction execution.

---

### Impact Explanation

**Impact: Critical — Permanent Freezing of Funds.**

When a contract calls `replace_class` with an arbitrary undeclared class hash (e.g., `0x1`, or any felt value that has never been declared), the OS commits this state change to the contract state tree without error. On any subsequent call to that contract, the OS will attempt to load and execute the class identified by the new hash. Since no such class exists in the class tree, execution will fail for every future invocation. Any ERC-20 tokens, ETH, or other assets held in the contract's storage become permanently inaccessible — the contract can never execute again.

This is structurally identical to the M-05 impact: just as destroying the EVM logic contract via `selfdestruct` rendered all proxy instances inoperable, replacing a StarkNet contract's class with a non-existent hash renders that contract permanently inoperable.

---

### Likelihood Explanation

**Likelihood: Medium.**

The attack requires one of the following conditions:

1. **Attacker-controlled contract with user funds**: An attacker deploys a contract that accepts deposits from other users and then calls `replace_class` with an undeclared hash, permanently freezing all deposited funds.

2. **Vulnerable shared contract**: A contract (e.g., a vault, multisig, or DAO) exposes a function that internally calls `replace_class` with user-supplied or insufficiently validated input. An unprivileged transaction sender can trigger this path.

3. **Griefing via `library_call` composition**: A contract that uses `library_call` to delegate to an attacker-controlled class hash, where the attacker's class calls `replace_class` with an undeclared hash in the caller's context (since `library_call` executes in the caller's storage and address context, the `replace_class` would affect the calling contract).

Condition (3) is the most direct analog to M-05: `library_call` is StarkNet's `delegatecall`, and a malicious class called via `library_call` can call `replace_class` to permanently corrupt the calling contract's class pointer — exactly as a malicious strategy called via `delegatecall` could `selfdestruct` the logic contract.

The OS is the correct enforcement layer for this check. The TODO comment confirms the developers are aware this check is missing and plan to add it.

---

### Recommendation

In `execute_replace_class` (both in `syscall_impls.cairo` and `deprecated_execute_syscalls.cairo`), before writing the new class hash into the contract state, verify that the class hash exists either:

- In the `contract_class_changes` dictionary (declared in the current block), or
- In the existing contract class Patricia tree (previously declared).

This is exactly what the TODO comment at line 898 of `syscall_impls.cairo` calls for. Until this check is implemented, any contract can permanently corrupt its own class pointer (or, via `library_call`, the class pointer of any contract that delegates to it), freezing all funds held in that contract.

---

### Proof of Concept

**Attack via `library_call` composition (direct M-05 analog):**

1. Attacker declares a malicious class `MaliciousClass` whose only function calls `replace_class(0xdeadbeef)` where `0xdeadbeef` is an undeclared class hash.
2. Attacker identifies a victim contract `VaultContract` that holds user funds and uses `library_call` to delegate some operation to a caller-supplied class hash (e.g., a strategy pattern, analogous to the `strategy` address in M-05).
3. Attacker calls `VaultContract` supplying `MaliciousClass` as the class hash for the `library_call`.
4. The OS executes `execute_library_call` (`syscall_impls.cairo`, lines 237–278), which runs `MaliciousClass` in the context of `VaultContract` (same `contract_address`, same storage).
5. `MaliciousClass` calls `replace_class(0xdeadbeef)`. The OS executes `execute_replace_class` (`syscall_impls.cairo`, lines 878–916), which writes `class_hash = 0xdeadbeef` into `VaultContract`'s state entry **without any existence check** (the TODO at line 898 is not yet implemented).
6. The transaction succeeds and is committed. `VaultContract`'s class is now `0xdeadbeef`.
7. All future calls to `VaultContract` fail because class `0xdeadbeef` does not exist. All user funds are permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
