### Title
Non-Standard ERC20 Tokens Permanently Locked in EVM Bridge Due to `logMetadata` Revert and Missing Registration Guard in `initTransfer` - (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.sol`'s `logMetadata` function makes hard, non-low-level calls to `IERC20Metadata(tokenAddress).name()`, `.symbol()`, and `.decimals()`. For well-known tokens that do not conform to the `string`-returning `IERC20Metadata` ABI (e.g., USDT on Ethereum mainnet, which returns `bytes32` for `name()` and `symbol()`), these calls revert due to ABI-decoding failure. Because `initTransfer` has no guard requiring prior successful token registration, a user can lock such tokens in the EVM bridge contract. The NEAR bridge then permanently refuses to finalize the transfer, leaving the tokens irrecoverable.

---

### Finding Description

**Vulnerability class:** Token-classification/filtering logic that causes legitimate tokens to be permanently stuck — direct analog of the SushiMakerAuction `onlyToken` / `token0()` pattern.

**Step 1 — `logMetadata` reverts for non-standard tokens.**

`logMetadata` in `OmniBridge.sol` performs three direct high-level interface calls:

```solidity
function logMetadata(address tokenAddress) external payable {
    string memory name    = IERC20Metadata(tokenAddress).name();
    string memory symbol  = IERC20Metadata(tokenAddress).symbol();
    uint8  decimals       = IERC20Metadata(tokenAddress).decimals();
    ...
}
``` [1](#0-0) 

These are not low-level `call()` invocations; they are typed ABI calls. If the token returns `bytes32` instead of `string` for `name()` or `symbol()` (as USDT on Ethereum mainnet does), the Solidity ABI decoder reverts. The `LogMetadata` event is never emitted, so the NEAR bridge never deploys or registers the token.

**Step 2 — `initTransfer` has no registration guard.**

`initTransfer` accepts any ERC20 token address and immediately pulls tokens from the caller:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount
    );
}
``` [2](#0-1) 

There is no check that `logMetadata` was ever successfully called for `tokenAddress`, no check that the token is mapped in `nearToEthToken`, and no check that `isBridgeToken[tokenAddress]` is set. The `InitTransfer` event is emitted regardless.

**Step 3 — NEAR bridge permanently rejects the finalization.**

When the NEAR bridge's `fin_transfer_callback` processes the `InitTransfer` proof, it looks up the token's decimal configuration:

```rust
let decimals = self
    .token_decimals
    .get(&init_transfer.token)
    .near_expect(BridgeError::TokenDecimalsNotFound);
``` [3](#0-2) 

Because `logMetadata` never succeeded, `token_decimals` has no entry for this token. The callback panics with `TokenDecimalsNotFound` every time it is attempted. The NEAR bridge never issues an MPC-signed `finTransfer` payload for the EVM side.

**Step 4 — No recovery path exists.**

The EVM bridge has no `rescue`, `sweep`, or admin-withdrawal function for arbitrary ERC20 tokens. The only release path is `finTransfer`, which requires a valid MPC signature from the NEAR bridge — a signature that can never be produced for an unregistered token. The locked tokens are irrecoverable.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

Any amount of a non-`IERC20Metadata`-conformant ERC20 token (e.g., mainnet USDT, MKR, or any token whose `name()`/`symbol()` returns `bytes32`) that a user bridges via `initTransfer` is permanently locked in the EVM `OmniBridge` contract with no on-chain recovery mechanism.

---

### Likelihood Explanation

**Medium.**

- Mainnet USDT (Tether) is the most-bridged stablecoin on Ethereum and is a canonical example of a token that returns `bytes32` for `name()` and `symbol()`. Any user attempting to bridge USDT through this contract would trigger the issue.
- `initTransfer` is a permissionless, user-facing function. No privileged role is required.
- The ordering constraint (call `logMetadata` before `initTransfer`) is not enforced on-chain and is not obvious to users or integrators.
- Automated relayers or front-ends that call `initTransfer` without pre-validating `logMetadata` success would silently lock funds.

---

### Recommendation

1. **Guard `initTransfer` with a registration check.** Before accepting tokens, verify that the token has been successfully registered (e.g., `require(nearToEthToken[tokenAddress].length > 0 || isBridgeToken[tokenAddress], "ERR_TOKEN_NOT_REGISTERED")`).

2. **Make `logMetadata` robust to non-standard return types.** Use a low-level `call()` with a try/catch pattern (analogous to the SushiMakerAuction fix) to handle tokens that return `bytes32` instead of `string`, or add a whitelist for tokens whose metadata must be supplied manually.

3. **Add an admin rescue function** for ERC20 tokens that are not actively managed by the bridge (i.e., not `isBridgeToken` and not in `multiTokens`), to allow recovery of accidentally locked funds.

---

### Proof of Concept

1. Deploy or use mainnet USDT (`0xdAC17F958D2ee523a2206206994597C13D831ec7`). Its `name()` function returns `bytes32("Tether USD")`, not ABI-encoded `string`.

2. Call `OmniBridge.logMetadata(USDT_ADDRESS)`. The call reverts because `IERC20Metadata(USDT_ADDRESS).name()` fails ABI decoding. No `LogMetadata` event is emitted; USDT is never registered on NEAR.

3. Call `OmniBridge.initTransfer(USDT_ADDRESS, 1_000_000, 0, 0, "victim.near", "")` with sufficient USDT approval. `safeTransferFrom` succeeds; 1 USDT is locked in the bridge; `InitTransfer` event is emitted. [4](#0-3) 

4. The NEAR bridge relayer submits the proof to `fin_transfer`. `fin_transfer_callback` panics at `token_decimals.get(&init_transfer.token).near_expect(BridgeError::TokenDecimalsNotFound)` because USDT was never registered. [3](#0-2) 

5. Every subsequent attempt to finalize the transfer panics identically. The 1 USDT is permanently locked in the EVM bridge with no recovery path.
