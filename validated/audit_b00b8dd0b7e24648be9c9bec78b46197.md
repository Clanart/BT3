### Title
`validate_proof_facts` Does Not Bind Proof to the Submitting Transaction's Sender or Calldata, Allowing Any Account to Reuse Another Account's Proof — (`crates/blockifier/src/transaction/account_transaction.rs`)

---

### Summary

The `validate_proof_facts` function validates only block-level fields of a `SnosProofFacts` struct (program hash, block number, block hash, config hash). It never checks that the proof was generated for the submitting transaction's `sender_address`, `calldata`, or transaction hash. Any account can copy another account's `proof_facts` + `proof` from the chain, attach them to a completely different transaction, and pass all gateway and blockifier proof-fact checks. The proof does not actually prove the attacker's transaction; the sequencer accepts it as a valid client-side-proven transaction anyway.

---

### Finding Description

`SnosProofFacts` contains exactly five fields:

```rust
pub struct SnosProofFacts {
    pub proof_version: Felt,
    pub program_hash: StarkHash,
    pub block_number: BlockNumber,
    pub block_hash: BlockHash,
    pub config_hash: StarkHash,
}
```

There is no `sender_address`, no `transaction_hash`, and no `calldata_hash`. [1](#0-0) 

The virtual OS output format (from which proof_facts are derived) is:

```
[output_version, base_block_number, base_block_hash, starknet_os_config_hash,
 n_l2_to_l1_messages, message_hash_0, ...]
```

No transaction-level identity is committed. [2](#0-1) 

`validate_proof_facts` checks only those five block-level fields and returns `Ok(())` if they all match:

```rust
// Validate the program hash.
if !allowed.contains(&snos_proof_facts.program_hash) { ... }
// Validate the block hash and block number.
Self::validate_proof_block_number(...)?;
Self::validate_proof_block_hash(...)?;
// Validate the config hash.
if virtual_os_config_hash != proof_config_hash { ... }
Ok(())
``` [3](#0-2) 

The gateway's `run_proof_verification` verifies the proof cryptographically against the proof_facts, then stores it in the proof manager keyed by `proof_facts.hash()`. If the same proof_facts hash is already present, verification is **skipped entirely**:

```rust
let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;
if contains_proof {
    return Ok(false);   // skip verification
}
``` [4](#0-3) 

The proof manager stores proofs keyed only by `proof_facts.hash()`, with no binding to any account or transaction:

```rust
pub async fn set_proof(&self, proof_facts: ProofFacts, proof: Proof) -> ...
pub async fn get_proof(&self, proof_facts: ProofFacts) -> ...
``` [5](#0-4) 

The OS-level `check_proof_facts` call in the Cairo execution layer also only validates block-level fields:

```cairo
check_proof_facts(
    proof_facts_size=proof_facts_size,
    proof_facts=proof_facts,
    current_block_number=block_context.block_info_for_execute.block_number,
    virtual_os_config_hash=block_context.os_global_context.virtual_os_config_hash,
);
``` [6](#0-5) 

---

### Impact Explanation

**Impact: High**

An attacker (Account B) can:

1. Observe any accepted client-side-proven transaction from Account A on-chain (proof_facts and proof are public transaction fields).
2. Construct their own Invoke V3 transaction with Account A's `proof_facts` and `proof` attached, signed by Account B's key.
3. Submit to the gateway. The gateway calls `run_proof_verification`:
   - If Account A's proof is already in the proof manager, verification is **skipped** (`contains_proof → true`).
   - If not, `verify_proof(proof_facts, proof)` passes because the proof is cryptographically valid for those proof_facts — the verifier has no knowledge of Account B's sender address or calldata.
4. `validate_proof_facts` in `perform_pre_validation_stage` passes all five block-level checks.
5. Account B's transaction is admitted to the mempool and sequenced as a valid client-side-proven transaction, even though the proof proves Account A's execution, not Account B's.

The proof verification mechanism is rendered meaningless: it does not prove the submitting transaction's execution. This matches the scope item **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing"** and **"High. Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."**

A secondary consequence: the proof_facts include L2-to-L1 message hashes from Account A's execution. The virtual OS output embeds these hashes. If the OS uses them to emit messages for the current transaction, Account B could claim Account A's L2-to-L1 messages — however, confirming this requires inspecting the full `check_proof_facts` Cairo implementation in `execution_constraints.cairo`, which was not fully traced.

---

### Likelihood Explanation

**Likelihood: High**

Proof_facts and proof bytes are public fields of any accepted Invoke V3 transaction. Any observer can extract them from the chain or from the gateway response. No privileged access is required. The attacker only needs to sign their own transaction with the copied fields, which is a standard wallet operation.

---

### Recommendation

Bind the proof to the submitting transaction by including the transaction hash (or at minimum the `sender_address` and a hash of the calldata) in the virtual OS output. This requires a version bump to `VIRTUAL_OS_OUTPUT_VERSION`. On the Rust side, `validate_proof_facts` should then extract and compare the committed sender/tx-hash field:

```diff
// In SnosProofFacts:
pub struct SnosProofFacts {
    pub proof_version: Felt,
    pub program_hash: StarkHash,
    pub block_number: BlockNumber,
    pub block_hash: BlockHash,
    pub config_hash: StarkHash,
+   pub transaction_hash: TransactionHash,  // committed by the virtual OS
}

// In validate_proof_facts:
+ let proof_tx_hash = snos_proof_facts.transaction_hash;
+ let actual_tx_hash = self.tx_hash();
+ if proof_tx_hash != actual_tx_hash {
+     return Err(TransactionPreValidationError::InvalidProofFacts(
+         format!("Proof transaction hash mismatch: proof={proof_tx_hash}, tx={actual_tx_hash}")
+     ));
+ }
```

The proof manager key should also incorporate the transaction hash to prevent cross-transaction cache hits.

### Proof of Concept

```
// Setup: Account A has already submitted a valid client-side-proven transaction.
// Its proof_facts and proof are visible on-chain.

// Attacker (Account B) steps:
let stolen_proof_facts = fetch_proof_facts_from_chain(account_a_tx_hash);
let stolen_proof       = fetch_proof_from_chain(account_a_tx_hash);

// Build attacker's own transaction with Account A's proof artifacts.
let attacker_tx = InvokeTransactionV3 {
    sender_address: account_b_address,
    calldata: attacker_calldata,   // completely different from Account A's
    proof_facts: stolen_proof_facts.clone(),
    proof: stolen_proof.clone(),
    // ... sign with account_b_key ...
};

// Submit to gateway.
gateway.add_transaction(attacker_tx);

// Gateway flow:
// 1. proof_manager.contains_proof(stolen_proof_facts) → true (already stored from Account A)
//    → proof verification SKIPPED entirely.
// 2. validate_proof_facts checks: program_hash ✓, block_number ✓, block_hash ✓, config_hash ✓
//    → no sender_address or calldata check → Ok(())
// 3. Transaction admitted to mempool as a valid client-side-proven transaction.
// 4. The proof does not prove Account B's execution; the sequencer accepted it anyway.

assert!(mempool.contains(attacker_tx_hash));
// Account B's transaction is sequenced with a proof that proves Account A's execution.
```

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let chain_info = &block_context.chain_info;
        // TODO(Meshi): Cache this computation as part of the chain context.
        let virtual_os_config_hash = OsChainInfo::from(chain_info)
            .compute_virtual_os_config_hash()
            .expect("Failed to compute OS config hash");
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }

    pub async fn set_proof(
        &self,
        proof_facts: ProofFacts,
        proof: Proof,
    ) -> Result<(), FsProofStorageError> {
        if self.contains_proof(proof_facts.clone()).await? {
            return Ok(());
        }
        let facts_hash = proof_facts.hash();
        self.proof_storage.set_proof(facts_hash, proof.clone()).await?;
        self.cache.insert(facts_hash, proof);
        Ok(())
    }

    pub async fn get_proof(
        &self,
        proof_facts: ProofFacts,
    ) -> Result<Option<Proof>, FsProofStorageError> {
        let facts_hash = proof_facts.hash();
        // Check cache first.
        if let Some(proof) = self.cache.get(&facts_hash) {
            return Ok(Some(proof));
        }
        // Fallback to filesystem.
        let proof = self.proof_storage.get_proof(facts_hash).await?;
        if let Some(proof) = &proof {
            self.cache.insert(facts_hash, proof.clone());
        }
        Ok(proof)
    }
