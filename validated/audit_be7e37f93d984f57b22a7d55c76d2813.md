### Title
Missing Validation of Target Class Hash in `execute_replace_class` Allows Permanent Contract Freezing — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall in the StarkNet OS program does not validate that the supplied `class_hash` refers to a declared (non-zero) class. A contract can call `replace_class(0)` (or any undeclared hash), setting its own `class_hash` to `UNINITIALIZED_CLASS_HASH = 0`. Once the class hash is zeroed, the contract becomes permanently non-executable: no entry point can be dispatched, and any funds held in the contract's storage are permanently frozen. The OS itself acknowledges this gap with an explicit `TODO` comment.

---

### Finding Description

In `execute_replace_class` (the Sierra/new-syscall path):

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);
```

`class_hash` is taken directly from `request.class_hash` with no assertion that it is non-zero or that it exists in `contract_class_changes`. The OS writes the arbitrary value unconditionally into the contract's `StateEntry`.

The same missing check exists in the deprecated path:

```cairo
func execute_replace_class{contract_state_changes: DictAccess*, revert_log: RevertLogEntry*}(
    contract_address, syscall_ptr: ReplaceClass*
) {
    alloc_locals;
    let class_hash = syscall_ptr.class_hash;
    ...
    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );
```

No check against `UNINITIALIZED_CLASS_HASH` (= `0`) and no lookup into `contract_class_changes` to confirm the class was declared.

By contrast, `deploy_contract` correctly enforces:

```cairo
assert state_entry.class_hash = UNINITIALIZED_CLASS_HASH;
assert state_entry.nonce = 0;
```

And `execute_declare_transaction` enforces:

```cairo
// Note that prev_value=0 enforces that a class may be declared only once.
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
```

No equivalent guard exists in either `execute_replace_class` implementation.

The `get_contract_state_hash` function in `commitment.cairo` shows the consequence: when `class_hash == UNINITIALIZED_CLASS_HASH` and `storage_root == 0` and `nonce == 0`, the leaf hash collapses to `0`, effectively erasing the contract from the state tree. Even when storage is non-zero, the contract leaf is committed with `class_hash = 0`, making the contract permanently non-dispatchable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that holds token balances (ERC-20 balances stored in its own storage) or acts as a vault/escrow can be bricked by calling `replace_class(0)` from within itself. Once the class hash is set to `UNINITIALIZED_CLASS_HASH = 0`:

- No entry point selector can be dispatched (the OS looks up the class to find entry points; class `0` has none).
- The contract's storage — including token balances — remains committed in the Patricia trie but is permanently inaccessible.
- There is no recovery path: `replace_class` itself requires dispatching an entry point, which is impossible once the class is zeroed.

This is the StarkNet analog of the EIP-4758 / SELFDESTRUCT issue: just as Axelar's `DepositReceiver` relied on `SELFDESTRUCT` to reset a contract to a re-deployable state, a StarkNet contract relying on `replace_class` for upgrades can be permanently destroyed by supplying `class_hash = 0`, freezing all funds inside.

---

### Likelihood Explanation

**High likelihood.** The attack path requires only a single unprivileged transaction:

1. Deploy or use any existing contract that holds funds (e.g., an ERC-20 token balance stored in its own storage).
2. Call any external entry point on that contract that internally invokes `replace_class(0)` — or, if the attacker controls the contract logic (e.g., a malicious upgrade), directly issue the syscall.
3. The OS accepts the syscall without validation and commits `class_hash = 0` to state.

The `TODO` comment dated `1/1/2026` confirms the developers are aware this check is missing and it is not yet implemented. The vulnerability is present in both the Sierra (`syscall_impls.cairo`) and deprecated (`deprecated_execute_syscalls.cairo`) syscall paths, meaning it affects all contract versions.

---

### Recommendation

In both `execute_replace_class` implementations, add the following validations before writing the new `StateEntry`:

1. **Assert the new class hash is non-zero:**
   ```cairo
   assert_not_zero(class_hash);
   ```

2. **Assert the new class hash is declared** by reading from `contract_class_changes` (analogous to how `execute_declare_transaction` uses `dict_update` with `prev_value=0` to enforce uniqueness):
   ```cairo
   let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
   assert_not_zero(compiled_class_hash);
   ```

This mirrors the protection already present in `deploy_contract` (which asserts `class_hash = UNINITIALIZED_CLASS_HASH` before deploying) and in `execute_declare_transaction` (which asserts `compiled_class_hash != 0`).

---

### Proof of Concept

**Step 1.** Deploy a contract `Vault` that holds ERC-20 token balances in its own storage and exposes a `self_destruct()` entry point that calls `replace_class(0)`.

**Step 2.** Users deposit tokens into `Vault`. The storage trie for `Vault`'s address now contains non-zero balance entries.

**Step 3.** Call `Vault.self_destruct()`. The OS executes `execute_replace_class` with `class_hash = 0`:

```cairo
// syscall_impls.cairo line 896-910
let class_hash = request.class_hash;  // = 0

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
// <-- NO CHECK HERE -->

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash,          // = 0 = UNINITIALIZED_CLASS_HASH
    storage_ptr=state_entry.storage_ptr,
    nonce=state_entry.nonce,
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

**Step 4.** The block is proven and committed. `Vault`'s `class_hash` in the global state trie is now `0`. All token balances remain in storage but are permanently inaccessible — no entry point can ever be dispatched again.
