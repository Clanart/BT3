### Title
`add_factory()` and `add_token_deployer()` Are Add-Only — No Removal Mechanism Exists - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

The NEAR omni-bridge contract exposes `add_factory()` and `add_token_deployer()` to register trusted cross-chain factory addresses and token deployer accounts. Neither function has a corresponding removal counterpart. If a registered factory or deployer is ever compromised, the DAO has no on-chain mechanism to revoke its trusted status, leaving the bridge permanently exposed to fraudulent proof submissions from that source.

---

### Finding Description

`add_factory()` inserts an `OmniAddress` into the `self.factories` `LookupMap`, which is the authoritative registry of trusted factory contracts on each external chain (EVM, Solana, StarkNet, etc.). [1](#0-0) 

`add_token_deployer()` inserts a NEAR `AccountId` into `self.token_deployer_accounts`, keyed by `ChainKind`, designating which account is trusted to deploy bridged tokens for each chain. [2](#0-1) 

A search across the entire repository confirms there is no `remove_factory`, `remove_token_deployer`, or any equivalent function that removes entries from either collection. The `StorageKey::Factories` and `StorageKey::TokenDeployerAccounts` buckets are write-only from the protocol's perspective. [3](#0-2) 

The `factories` map is the trust anchor for cross-chain proof verification: when `fin_transfer` is called, the `emitter_address` embedded in the prover result is checked against this map to confirm the message originated from a legitimate factory. If a factory address is compromised, every proof it emits will still pass this check indefinitely.

---

### Impact Explanation

**Critical / High.** If a registered factory contract on an external chain (e.g., an EVM `OmniBridge.sol` deployment) is exploited or its private key is leaked, an attacker can craft and submit arbitrary `InitTransfer` proofs whose `emitter_address` matches the compromised-but-still-trusted factory. The NEAR bridge will accept these proofs and mint or release tokens to attacker-controlled recipients with no backing collateral. Because there is no removal path, the DAO cannot stop this even after detecting the compromise — the only recourse is a full contract redeployment or migration, during which further damage can accumulate.

This maps directly to the allowed impact: **unauthorized mint of bridged assets** and **permanent inability to revoke a compromised trust anchor**.

---

### Likelihood Explanation

Factory contracts on EVM chains are upgradeable proxies controlled by admin keys. A key compromise, a malicious upgrade, or a vulnerability in the factory itself could turn a registered factory into an attacker-controlled proof source. The NEAR bridge has no on-chain defense once this happens. The scenario is realistic for any long-lived deployment.

---

### Recommendation

Add a `remove_factory` and a `remove_token_deployer` function, both gated behind `Role::DAO`, that call `.remove()` on the respective `LookupMap`:

```rust
#[access_control_any(roles(Role::DAO))]
pub fn remove_factory(&mut self, address: OmniAddress) {
    self.factories.remove(&(&address).into());
}

#[access_control_any(roles(Role::DAO))]
pub fn remove_token_deployer(&mut self, chain: ChainKind) {
    self.token_deployer_accounts.remove(&chain);
}
```

Alternatively, follow the original report's recommendation and convert `add_factory` to `set_factory(address, enabled: bool)` so a single function handles both registration and revocation.

---

### Proof of Concept

1. DAO calls `add_factory(eth_factory_address)` — the EVM factory is now trusted.
2. The EVM factory contract is exploited; attacker gains control of it.
3. Attacker submits a `fin_transfer` call to the NEAR bridge with a crafted `ProverResult::InitTransfer` whose `emitter_address` equals `eth_factory_address`.
4. The bridge verifies `emitter_address` against `self.factories` — it matches, so the proof is accepted.
5. The bridge mints tokens to the attacker's NEAR account with no real collateral locked on the EVM side.
6. DAO attempts to remove the compromised factory — **no such function exists**. The attack can be repeated indefinitely until a full contract migration is executed. [4](#0-3)
