### Title
Blacklisted ERC20 Recipient Permanently Freezes Bridged Funds in `finTransfer` — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

In the NEAR → EVM bridge flow, `finTransfer` pushes native ERC20 tokens (e.g., USDC) directly to `payload.recipient`. If that address is blacklisted by the token contract, every `finTransfer` attempt reverts. Because the recipient is cryptographically bound to the MPC-signed payload and no protocol-level redirect or refund path exists, the user's funds locked on NEAR become permanently irrecoverable.

---

### Finding Description

`OmniBridge.finTransfer` handles native ERC20 tokens (those that are not bridge tokens, custom minters, or ERC1155) via a direct push transfer:

```solidity
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,
        payload.amount
    );
}
``` [1](#0-0) 

The nonce is marked used at line 287 before the transfer attempt, but because Solidity reverts all state changes on failure, the nonce is not permanently consumed when the transfer reverts: [2](#0-1) 

The consequence is that the nonce remains replayable, but **every replay also reverts** — because the recipient address is immutably encoded in the MPC-signed `TransferMessagePayload`. The signature covers `payload.recipient` directly: [3](#0-2) 

There is no admin function, emergency redirect, or refund path in the contract. The SECURITY.md documents no known issue for this scenario: [4](#0-3) 

The CLAUDE.md invariant "No token release without signature" explicitly forbids any admin bypass of `finTransfer`, closing the only conceivable escape hatch: [5](#0-4) 

**Identical root cause exists in the StarkNet contract.** `fin_transfer` in `omni_bridge.cairo` marks the nonce used before calling `transfer`, and if the ERC20 `transfer` returns false or panics (e.g., a blacklisted recipient), the assert reverts the whole transaction — same permanent-revert loop, same absence of a recovery path: [6](#0-5) 

---

### Impact Explanation

A user whose EVM (or StarkNet) address is blacklisted by a token like USDC after initiating a NEAR → EVM bridge transfer has their funds permanently frozen:

- Tokens were locked/burned on NEAR when the transfer was initiated.
- `finTransfer` on EVM will revert on every attempt indefinitely.
- No protocol mechanism exists to redirect the transfer or refund the origin-chain balance.
- Impact: **Permanent freezing / irrecoverable lock of user funds in the bridge flow.**

---

### Likelihood Explanation

- USDC is a major token that would be bridged as a native ERC20 (not a bridge token), hitting the `safeTransfer` code path.
- USDC blacklisting is a real, recurring event (OFAC sanctions, exchange compliance actions).
- Bridge transfers are not instant; a user can be blacklisted in the window between `initTransfer` on NEAR and `finTransfer` on EVM.
- No privileged access is required to trigger this — any user whose address is blacklisted by the token issuer after initiating a transfer is affected.

---

### Recommendation

Replace the direct push with a pull-based settlement pattern:

1. In `finTransfer`, credit `payload.amount` to a `claimable[payload.recipient][payload.tokenAddress]` mapping instead of calling `safeTransfer` immediately.
2. Add a separate `claim(address token)` function that lets the recipient (or an admin-designated substitute) withdraw their credited balance.

This decouples transfer finalization from token delivery, so a blacklisted recipient does not permanently block settlement. The nonce is consumed and the bridge accounting is correct regardless of the token's recipient-level restrictions.

---

### Proof of Concept

1. Alice holds USDC on NEAR and calls the bridge to transfer to her EVM address `0xAlice`.
2. NEAR MPC signs a `TransferMessagePayload` with `recipient = 0xAlice`, `tokenAddress = USDC_EVM`.
3. Before the relayer submits `finTransfer`, USDC blacklists `0xAlice` (e.g., OFAC action).
4. Relayer calls `OmniBridge.finTransfer(signature, payload)`:
   - `completedTransfers[nonce] = true` is written (line 287).
   - `IERC20(USDC).safeTransfer(0xAlice, amount)` reverts — blacklisted recipient.
   - Entire transaction reverts; nonce write is undone.
5. Every subsequent `finTransfer` call with the same valid signature also reverts at step 4.
6. No alternative payload can be constructed — any change to `recipient` invalidates the MPC signature.
7. Alice's USDC is permanently locked on NEAR. There is no admin function, no refund path, and no re-signing mechanism in the protocol.

# Security Notes

## Design Decisions (Non-Issues)

These patterns have been reviewed and confirmed as intentional. Do not flag or "fix" them.

- **Fee-on-transfer tokens not supported**: `initTransfer` emits the requested `amount`, not the actual received balance. Fee-on-transfer and rebasing tokens are intentionally unsupported
- **`logMetadata` and `deployToken` are permissionless**: Anyone can call `logMetadata` for any ERC20, and anyone can submit a valid MPC signature to `deployToken`. This is by design — the bridge is fully permissionless
- **`ENearProxy.burn` uses empty NEAR recipient**: `eNear.transferToNear(amount, "")` is intentional — `transferToNear` is a legacy method used purely as a burn mechanism. The actual NEAR recipient is tracked in the OmniBridge `InitTransfer` event
- **`deployToken` signature has no chain ID**: Metadata signatures are intentionally chain-agnostic — one NEAR-side signature deploys the same token on all EVM chains

## Known Issues

Low-severity items acknowledged but not yet addressed:

- **`addCustomToken` can overwrite existing mappings** (H-01): Admin-only function. No existence check — calling with an already-mapped `nearTokenId` silently overwrites `nearToEthToken`. Accepted as operational risk
- **`pause(flags)` replaces all flags** (H-02): `_pause(flags)` does full replacement, not bitwise OR. Calling `pause(PAUSED_INIT_TRANSFER)` when `PAUSED_FIN_TRANSFER` is set will unpause finTransfer. Use `pauseAll()` for emergencies
- **`BridgeToken.initialize` stores metadata redundantly** (L-01): `__ERC20_init(name_, symbol_)` writes to parent storage that is never read (getters are overridden). Minor gas waste on init
- **`require` strings instead of custom errors** (L-02): Several locations use `require` with string messages instead of custom errors (`OmniBridge.sol:150,204,556`, `SelectivePausableUpgradable.sol:100,107`, `ENearProxy.sol:56,76,86`)
- **`OmniBridgeWormhole` has no `__gap`** (L-04): Three storage variables with no gap array. Safe as a leaf contract but would need a gap if inherited from
- **`PayloadType.ClaimNativeFee` defined but unused** (L-05): Enum value 2 is never referenced. Native fees are recovered via `finTransfer` with `tokenAddress=address(0)`
