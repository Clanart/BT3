### Title
Partial `ft_on_transfer` Refund Strands Tokens in Bridge's OmniToken Balance — (`near/omni-bridge/src/lib.rs`, `near/omni-token/src/lib.rs`)

---

### Summary

When the bridge mints tokens via `OmniToken::mint(account_id, amount, Some(msg))`, it first credits itself and then calls `ft_transfer_call`. If the recipient contract's `ft_on_transfer` returns a value in the range `(0, amount)` — a partial refund — the NEP-141 resolver refunds that portion back to the bridge's own balance. The bridge's callback `fin_transfer_send_tokens_callback` then checks `is_refund_required`, which only triggers a burn when `used_amount == 0`. For any partial refund, `used_amount > 0`, so no burn fires, and the refunded tokens are permanently stranded in the bridge's OmniToken balance with no automatic recovery path.

---

### Finding Description

**Step 1 — `OmniToken::mint` with `msg`** [1](#0-0) 

When `msg` is `Some`, `mint` deposits `amount` to `predecessor_account_id` (the bridge itself) and immediately calls `ft_transfer_call(account_id, amount, None, msg)`. The bridge is the NEP-141 *sender* for this transfer.

**Step 2 — Recipient returns partial refund**

The recipient contract's `ft_on_transfer` returns `r` where `0 < r < amount`. Under NEP-141, `ft_resolve_transfer` interprets this as "refund `r` tokens back to the sender (the bridge)" and returns `used_amount = amount - r`. [2](#0-1) 

After resolution: the bridge holds `r` tokens in its OmniToken balance; the recipient holds `amount - r`.

**Step 3 — `is_refund_required` only fires on full rejection** [3](#0-2) 

The function reads the result of `ft_transfer_call` (i.e., `ft_resolve_transfer`'s return value = `used_amount`). It returns `true` **only when `used_amount == 0`**. For any partial use (`used_amount = amount - r > 0`), it returns `false`.

**Step 4 — No burn fires; tokens are stranded** [4](#0-3) 

`fin_transfer_send_tokens_callback` calls `burn_tokens_if_needed` only inside the `is_refund_required == true` branch. When `is_refund_required` is `false`, the `r` tokens sitting in the bridge's OmniToken balance are never burned and never re-transferred. There is no subsequent callback or cleanup path for this case.

**Same issue in `resolve_fast_transfer`** [5](#0-4) 

`burn_tokens_if_needed` is called unconditionally with the full `amount`, but uses `.detach()`: [6](#0-5) 

If the bridge only holds `r < amount` (the partial refund), the burn call panics inside the token contract, but because it is detached, the failure is silently swallowed. The `r` tokens remain in the bridge's balance.

---

### Impact Explanation

- `r` tokens are permanently stranded in the bridge's OmniToken balance with no automatic recovery path.
- The OmniToken total supply is inflated by `r` tokens that are not backed by any locked assets on the source chain, breaking the bridge's collateralization invariant.
- The only recovery is a privileged DAO action (`transfer_token_as_dao` or a manual governance intervention), which is not guaranteed and requires recognizing the incident.

Impact category: **Critical — Permanent irrecoverable lock of protocol funds / unbacked supply inflation.**

---

### Likelihood Explanation

Any user who controls a NEAR contract can be the recipient of a cross-chain transfer. The recipient address is user-supplied in the transfer message. Deploying a contract that returns a value in `(0, amount)` from `ft_on_transfer` is trivial. No privileged access, leaked keys, or colluding validators are required. The attacker only needs to initiate a legitimate cross-chain transfer targeting their own malicious contract.

---

### Recommendation

1. **Track the refunded amount explicitly.** After `ft_transfer_call` resolves, compute `refund = amount - used_amount`. If `refund > 0`, burn `refund` tokens from the bridge's balance (for deployed tokens) or transfer them back to the original sender.

2. **Replace the binary `is_refund_required` check** with a quantity-aware check that handles three cases:
   - `used_amount == 0`: full rejection → burn full amount, revert transfer.
   - `0 < used_amount < amount`: partial use → burn `amount - used_amount`, finalize transfer for `used_amount`.
   - `used_amount == amount`: full use → no burn needed, finalize normally.

3. **Do not use `.detach()` for burns** that are part of correctness invariants. Await the burn result and handle failures explicitly.

---

### Proof of Concept

```
1. Deploy a malicious NEAR contract `attacker.near` implementing `ft_on_transfer`
   that always returns `amount / 2` (partial refund).

2. Initiate a cross-chain transfer (e.g., from EVM) with recipient = `attacker.near`
   and a non-empty `msg` field (to trigger ft_transfer_call path).

3. Relayer submits proof → `fin_transfer` → `fin_transfer_callback`
   → `process_fin_transfer_to_near` → `send_tokens`
   → bridge calls `OmniToken::mint(attacker.near, amount, Some(msg))`

4. OmniToken mints `amount` to bridge, calls `ft_transfer_call(attacker.near, amount, msg)`.

5. `attacker.near::ft_on_transfer` returns `amount / 2`.

6. `ft_resolve_transfer` refunds `amount / 2` to bridge, returns `used_amount = amount / 2`.

7. `fin_transfer_send_tokens_callback` reads `used_amount = amount / 2 > 0`,
   so `is_refund_required` = false → no burn.

8. Assert: bridge's OmniToken balance = `amount / 2` (stranded).
   Assert: OmniToken total_supply = `amount` (only `amount / 2` is in attacker's hands,
   `amount / 2` is unbacked and locked in bridge balance).
   Assert: no recovery path exists without DAO intervention.
```

    pub fn fin_transfer_send_tokens_callback(
        &mut self,
        #[serializer(borsh)] transfer_message: TransferMessage,
        #[serializer(borsh)] fee_recipient: &AccountId,
        #[serializer(borsh)] is_ft_transfer_call: bool,
        #[serializer(borsh)] storage_owner: &AccountId,
        #[serializer(borsh)] lock_actions: Vec<LockAction>,
    ) {
        let token = self.get_token_id(&transfer_message.token);

        if Self::is_refund_required(is_ft_transfer_call) {
            self.burn_tokens_if_needed(
                token.clone(),
                U128(
                    transfer_message
                        .amount_without_fee()
                        .near_expect(BridgeError::InvalidFee),
                ),
            );

            self.revert_lock_actions(&lock_actions);

            self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);

            env::log_str(
                &OmniBridgeEvent::FailedFinTransferEvent { transfer_message }.to_log_string(),
            );
        } else {
            // Send fee to the fee recipient
            if transfer_message.fee.fee.0 > 0 {
                if self.is_deployed_token(&token) {
                    ext_token::ext(token)
                        .with_static_gas(MINT_TOKEN_GAS)
                        .mint(fee_recipient.clone(), transfer_message.fee.fee, None)
                        .detach();
                } else {
                    ext_token::ext(token)
                        .with_attached_deposit(ONE_YOCTO)
                        .with_static_gas(FT_TRANSFER_GAS)
                        .ft_transfer(fee_recipient.clone(), transfer_message.fee.fee, None)
                        .detach();
                }
            }

            if transfer_message.fee.native_fee.0 > 0 {
                let native_token_id = self.get_native_token_id(transfer_message.get_origin_chain());

                ext_token::ext(native_token_id)
                    .with_static_gas(MINT_TOKEN_GAS)
                    .mint(fee_recipient.clone(), transfer_message.fee.native_fee, None)
                    .detach();
            }

            env::log_str(&OmniBridgeEvent::FinTransferEvent { transfer_message }.to_log_string());
        }
    }

        if is_ft_transfer_call {
