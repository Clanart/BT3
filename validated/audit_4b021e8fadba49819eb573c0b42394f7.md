### Title
Stale `proposed_split` Inherited by Child Shards After Shard Split Breaks Chunk Validation Invariant ŌĆö (File: `chain/chain/src/resharding/manager.rs`)

---

### Summary

When a parent shard is split during dynamic resharding, `process_memtrie_resharding_storage_update` creates each child's `ChunkExtra` by cloning the parent's `ChunkExtra` and only overwriting `state_root` and `congestion_info`. The `proposed_split` field ŌĆö which is `Some(TrieSplit{ŌĆ”})` on the parent at the moment of resharding (it is what triggered the split) ŌĆö is silently inherited by both child shards. In epoch N+2, when the child's first chunk is produced, `compute_proposed_split` correctly returns `None` (the resharding cooldown blocks a new proposal), so the chunk header carries `proposed_split = None`. The invariant check in `validate_chunk_with_chunk_extra_and_receipts_root` then compares the stale `Some(TrieSplit)` stored in the child's `ChunkExtra` against the header's `None` and emits `InvalidChunkHeaderShardSplit`, rejecting every block that contains the child shard's first chunk. The network cannot advance past the resharding boundary.

### Finding Description

**Root cause ŌĆö `process_memtrie_resharding_storage_update`** [1](#0-0) 

```rust

// typing. Clarify where it should happen when `State` and
// `FlatState` update is implemented.

*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;

```

The `proposed_split` field lives in `ChunkExtraV5`: [2](#0-1) 

No `proposed_split_mut()` accessor exists on `ChunkExtra`, confirming the field was never wired into the resharding path. The parent's `proposed_split` is `Some(TrieSplit{boundary_account, left_mem, right_mem})` at the resharding block ŌĆö it is the value that caused `get_upcoming_shard_split` to schedule the split.

**Validation that fires** [3](#0-2) 

```rust
if prev_chunk_extra.proposed_split() != chunk_header.proposed_split() {
    return Err(Error::InvalidChunkHeaderShardSplit(ŌĆ”));
}
```

`prev_chunk_extra` for the child's first chunk is the stale child `ChunkExtra` written at the resharding block. `chunk_header.proposed_split()` is `None`, computed by `compute_proposed_split`: [4](#0-3) 

```rust
if !self.epoch_manager.can_reshard(prev_block_hash, config.min_epochs_between_resharding)? {
    return Ok(None);   // ŌåÉ cooldown blocks a new proposal; returns None
}
```

The cooldown is mandatory (`min_epochs_between_resharding > 0`), so `None` is guaranteed for the child's first chunk. The mismatch is therefore deterministic and unconditional.

**The invariant broken**

The protocol invariant is: `ChunkExtra.proposed_split` written after applying chunk at block H must equal `chunk_header.proposed_split` of the next new chunk for that shard. The resharding path violates this by writing a child `ChunkExtra` whose `proposed_split` was never computed for the child ŌĆö it is the parent's stale commitment, carried forward without consent of the child's own chunk application.

This is the direct nearcore analog of the Splitter contract issue: a structural change (shard split / `addPayee`) retroactively corrupts a committed field (`proposed_split` / unclaimed share denominator) that existing downstream consumers (chunk validation / shareholders) depend on.

---

### Impact Explanation

Every block produced in epoch N+2 that contains the child shard's first new chunk will be rejected by every validating node with `InvalidChunkHeaderShardSplit`. No honest chunk producer can produce a valid chunk for the child shard because `compute_proposed_split` will always return `None` while the stored `ChunkExtra` holds `Some(TrieSplit)`. The network cannot finalize any block past the resharding boundary. **Impact: Critical liveness failure triggered unconditionally on every dynamic resharding event.**

---

### Likelihood Explanation

The bug fires on every execution of `process_memtrie_resharding_storage_update` where the parent shard's `proposed_split` is non-`None` ŌĆö which is always true, because a non-`None` `proposed_split` is the precondition for the resharding being scheduled. The feature is currently gated behind `ProtocolFeature::DynamicResharding`; once that gate opens the bug activates on the first resharding. The TODO comment at line 255 of `manager.rs` explicitly acknowledges that not all `ChunkExtra` fields are set correctly.

---

### Recommendation

In `process_memtrie_resharding_storage_update`, after cloning the parent `ChunkExtra` for each child, explicitly reset `proposed_split` to `None`:

```rust
let mut child_chunk_extra = ChunkExtra::clone(&parent_chunk_extra);
*child_chunk_extra.state_root_mut() = trie_changes.new_root;
*child_chunk_extra.congestion_info_mut() = child_congestion_info;
// Add: child shards have no prior split proposal
if let ChunkExtra::V5(ref mut v5) = child_chunk_extra {
    v5.proposed_split = None;
}
```

Alternatively, add a `proposed_split_mut()` accessor (parallel to `congestion_info_mut`) and use it here. Also audit `next_for_old_chunk` and `set_state_finalize_on_height`, which use the same clone-and-patch pattern and would propagate the stale value across missing-chunk blocks: [5](#0-4) [6](#0-5)

### Proof of Concept

1. Enable `ProtocolFeature::DynamicResharding` (set protocol version Ōēź its activation threshold in a test network).
2. Configure one shard with `force_split_shards` so it is guaranteed to split at the next epoch boundary.
3. Run the network through epoch N (parent shard accumulates `proposed_split = Some(TrieSplit{ŌĆ”})` in its `ChunkExtra`).
4. At the epoch N ŌåÆ N+1 boundary, `finalize_epoch` schedules the split; `process_memtrie_resharding_storage_update` writes child `ChunkExtra` with the stale `proposed_split`.
5. In epoch N+2, observe that every block containing the child shard's first new chunk is rejected with `InvalidChunkHeaderShardSplit`; the chain halts.

The exact corrupted value is `child_chunk_extra.proposed_split = Some(TrieSplit { boundary_account: <parent's boundary>, left_memory_usage: <u64>, right_memory_usage: <u64> })` stored at `DBCol::ChunkExtra` keyed by `(resharding_block_hash, child_shard_uid)`.
