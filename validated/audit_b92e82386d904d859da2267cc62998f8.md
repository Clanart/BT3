### Title
Integer Overflow in Relative Timelock Arithmetic Enables Timelock Bypass and Consensus Divergence - (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary
`check_time_locks` uses `wrapping_add` when `nowrap=false` to compute `confirmed_block_index + height_relative` (u32) and `timestamp + seconds_relative` (u64). For attacker-controlled large relative timelock values (e.g., `height_relative = u32::MAX`), the addition wraps around to a small number, causing the timelock comparison to pass when it should fail. This bypasses `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE` timelocks, and inverts `ASSERT_BEFORE_HEIGHT_RELATIVE` / `ASSERT_BEFORE_SECONDS_RELATIVE` checks. Nodes using `nowrap=false` and nodes using `nowrap=true` (saturating) will reach opposite conclusions on the same spend, producing deterministic consensus divergence.

### Finding Description
In `check_time_locks`, the `nowrap: bool` parameter selects between two arithmetic modes for all four relative timelock conditions:

```rust
// nowrap=false path (legacy):
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

When `nowrap=false`, `confirmed_block_index.wrapping_add(height_relative)` silently wraps on overflow. For example, with `confirmed_block_index = 100` and `height_relative = 0xFFFF_FFFF` (u32::MAX, a valid value accepted by `sanitize_uint` with `max_size=4`):

- `100u32.wrapping_add(0xFFFF_FFFF)` = `99` (wraps mod 2^32)
- Check becomes: `prev_height < 99` → false at any height ≥ 99 → spend **accepted**
- With `nowrap=true` (saturating): `100u32.saturating_add(0xFFFF_FFFF)` = `u32::MAX` → `prev_height < u32::MAX` → true → spend **rejected**

The same pattern applies to `seconds_relative` (u64), `before_height_relative` (u32), and `before_seconds_relative` (u64). The tests in the codebase explicitly document and confirm this divergence.

The `sanitize_uint` function accepts `height_relative` up to 4 bytes (u32::MAX = `0xFFFF_FFFF`) and `seconds_relative` up to 8 bytes (u64::MAX), so these extreme values are valid CLVM condition arguments that pass all upstream parsing checks.

### Impact Explanation
- **Timelock bypass**: A coin whose puzzle emits `ASSERT_HEIGHT_RELATIVE(0xFFFF_FFFF)` is intended to be unspendable for ~4 billion blocks. Under `nowrap=false`, the wrapping makes the required height threshold wrap to `confirmed_height - 1`, so the coin is spendable immediately at any height ≥ `confirmed_height`. This is an unauthorized spend acceptance.
- **Consensus divergence**: A node running with `nowrap=false` accepts the spend; a node running with `nowrap=true` rejects it. Both are processing the same valid serialized spend bundle. This is a deterministic, attacker-triggerable consensus split.
- **Inverted before-relative checks**: `ASSERT_BEFORE_HEIGHT_RELATIVE(0xFFFF_FFFF)` wraps to a past deadline under `nowrap=false`, permanently locking coins that should be freely spendable.

### Likelihood Explanation
The `nowrap` parameter is passed from the Python full node via the `py_check_time_locks` binding. The `nowrap=false` path is the legacy behavior retained for backward compatibility. Any node or code path that passes `nowrap=False` to `check_time_locks` is vulnerable. An unprivileged attacker needs only to craft a spend bundle with a large relative timelock value (e.g., `ASSERT_HEIGHT_RELATIVE` with argument `0xFFFF_FFFF`) — a valid CLVM atom of 4 bytes — and submit it to the network. No special privileges, keys, or access are required.

### Recommendation
Replace `wrapping_add` with `saturating_add` unconditionally for all four relative timelock checks, or remove the `nowrap=false` code path entirely. The `nowrap=true` (saturating) behavior is semantically correct: a relative timelock that would overflow the integer type should be treated as "effectively infinite" and always fail (for `ASSERT_*_RELATIVE`) or always pass (for `ASSERT_BEFORE_*_RELATIVE`). The legacy wrapping path should not remain reachable in any production code path.

### Proof of Concept
The existing test suite in `crates/chia-consensus/src/check_time_locks.rs` already proves the divergence:

```
// 200 < 100 + u32::MAX -> Err with nowrap=true (saturates), Ok with nowrap=false (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),  // nowrap=true
    Ok(()),  // nowrap=false: TIMELOCK BYPASSED
)]
```

A concrete attacker spend:
1. Deploy a coin whose puzzle outputs `(ASSERT_HEIGHT_RELATIVE . 0xFFFF_FFFF)` — a 4-byte atom, valid per `sanitize_uint(max_size=4)`.
2. Submit a spend bundle for this coin to a node using `nowrap=False`.
3. The node computes `confirmed_height.wrapping_add(0xFFFF_FFFF)` = `confirmed_height - 1`, finds `prev_height >= confirmed_height - 1`, and accepts the spend — bypassing the intended ~4-billion-block timelock.
