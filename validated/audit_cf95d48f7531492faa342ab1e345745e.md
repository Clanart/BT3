Looking at the full `set_state_header` implementation and the `verify_path` / `merklize` primitives, I need to determine whether the shard-id binding is actually enforced.

Based on my thorough code review, here is the analysis:

**Confirmed facts from the code:**

1. `chunk.shard_id()` is **never called** anywhere in `set_state_header` — grep confirms zero matches in `adapter.rs`.
2. `verify_path_with_index` (the position-aware variant) is **never used** in `set_state_header` — only the position-independent `verify_path` is used.
3. `verify_path` in `core/primitives/src/merkle.rs` simply checks `compute_root_from_path(path, hash_borsh(item)) == root` — it encodes no leaf-index constraint.
4. The `chunk_headers_root` is `merklize` over all `ChunkHashHeight` values for all shards. A chunk from shard B at index B has a perfectly valid Merkle path against this same root.
5. The receipt proof check at line 488 uses the **caller-supplied** `shard_id` (shard A) for `ReceiptList(shard_id, receipts)` — it is entirely independent of `chunk.shard_id()`.
6. The default state sync source is `StateSyncDownloadSourcePeer` (confirmed in `mod.rs` lines 126–132), which sends `NetworkRequests::StateRequestHeader` to **any available P2P peer** — not a specifically configured trusted operator.

### Title
Missing `chunk.shard_id() == shard_id` assertion in `set_state_header` allows cross-shard state root substitution — (File: `chain/chain/src/state_sync/adapter.rs`)

### Summary
`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` for `shard_id_A` but never asserts that the embedded chunk's `shard_id()` equals the requested shard. Because `verify_path` is position-independent, a chunk from shard B with a valid Merkle proof against the same `chunk_headers_root` passes every check. A malicious P2P peer can supply shard B's chunk, causing shard B's `prev_state_root` to be committed to `DBCol::StateHeaders` under key `StateHeaderKey(shard_id_A, sync_hash)`, corrupting the state root used by every subsequent `set_state_part` and `set_state_finalize` call for shard A.

### Finding Description

`set_state_header` performs five checks, none of which bind the chunk to the requested shard:

**Check 1 — `validate_chunk_proofs`** verifies the chunk body hash, tx root, and receipts root. It does not inspect `chunk.shard_id()`. [1](#0-0) 

**Check 2 — `verify_path` against `chunk_headers_root`** proves the chunk hash is *somewhere* in the block's Merkle tree. It does not check which leaf index (shard index) the chunk occupies. `verify_path` is defined as `compute_root_from_path(path, hash_borsh(item)) == root` — purely hash-path equality, no index. [2](#0-1) [3](#0-2) 

The position-aware variant `verify_path_with_index` exists but is **not used** here: [4](#0-3) 

**Check 3 — `verify_path` for `prev_chunk`** — same position-independent proof, same gap. [5](#0-4) 

**Check 4 — receipt proof validation** uses the caller-supplied `shard_id` (shard A) for `ReceiptList(shard_id, receipts)`. This is entirely independent of `chunk.shard_id()`. An attacker supplies valid receipt proofs for shard A from the real chain; this check passes. [6](#0-5) 

**Check 5 — `validate_state_root_node`** validates the supplied `state_root_node` against `chunk_inner.prev_state_root()`. If the attacker supplies shard B's chunk, they supply shard B's root node — which is consistent and passes. [7](#0-6) 

**Storage** then writes the tampered header under `StateHeaderKey(shard_id_A, sync_hash)`: [8](#0-7) 

The state sync source is `StateSyncDownloadSourcePeer`, which routes `NetworkRequests::StateRequestHeader` to any available P2P peer — not a specifically configured trusted operator: [9](#0-8) [10](#0-9) 

### Impact Explanation

The exact corrupted value is: **shard B's `prev_state_root`** stored in `DBCol::StateHeaders` under `StateHeaderKey(shard_id_A, sync_hash)`.

Downstream consequences:
- `set_state_part` reads this header and validates every incoming part against shard B's state root. Legitimate shard A parts fail; shard B parts (also supplied by the attacker) pass.
- `set_state_finalize` installs shard B's trie as shard A's committed state, causing the node to execute shard A transactions against the wrong state. [11](#0-10) 

### Likelihood Explanation

The default state sync path uses `StateSyncDownloadSourcePeer`. Any peer that advertises possession of the epoch snapshot can be selected as the download source. No validator key, no admin access, and no external-storage credentials are required. The attacker only needs: (a) a valid chunk from shard B with its Merkle path (public chain data), (b) valid receipt proofs for shard A (public chain data), and (c) shard B's trie root node (available to any full node tracking shard B).

### Recommendation

Add an explicit shard-id guard immediately after extracting the chunk in `set_state_header`:

```rust
let chunk = shard_state_header.cloned_chunk();
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
```

Optionally, replace the position-independent `verify_path` with `verify_path_with_index` using the shard's index in the epoch layout, which would provide a cryptographic binding rather than just a field check.

### Proof of Concept

```
1. Two-shard testnet: shard A (id=0), shard B (id=1).
2. Victim node enters state sync for shard A at sync_hash H.
3. Attacker peer connects, advertises snapshot for epoch of H.
4. Victim sends StateRequestHeader { shard_id: 0, sync_hash: H }.
5. Attacker responds with ShardStateSyncResponseHeader containing:
   - chunk = real shard-B chunk at height_included in prev(H),
     with its real Merkle path against chunk_headers_root of prev(H)
   - prev_chunk = real shard-B prev chunk, with its real Merkle path
   - incoming_receipts_proofs = real receipt proofs for shard A (public)
   - state_root_node = real root node for shard B's prev_state_root
6. set_state_header(shard_id=0, sync_hash=H, tampered_header) is called.
7. All five checks pass; no chunk.shard_id() == shard_id assertion exists.
8. DBCol::StateHeaders[StateHeaderKey(0, H)] now holds shard B's header.
9. Assert: get_state_header(0, H).chunk_prev_state_root()
           == shard_B_chunk.prev_state_root()   // PASSES — wrong root installed
        != shard_A_chunk.prev_state_root()       // correct root never stored
```
