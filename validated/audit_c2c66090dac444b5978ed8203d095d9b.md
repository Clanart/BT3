Looking at the production code in `chain/chain/src/state_sync/adapter.rs`, I need to trace every guard in `set_state_header` and determine whether the shard-id binding is enforced.

### Title
Missing shard-id binding check in `set_state_header` allows cross-shard state header substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`set_state_header` accepts a `ShardStateSyncResponseHeader` from an untrusted peer and stores it under `StateHeaderKey(shard_id, sync_hash)` without ever asserting that the embedded `chunk.shard_id()` equals the `shard_id` parameter. The `verify_path` call used to prove chunk inclusion is a position-agnostic merkle membership check: it confirms the chunk's `ChunkHashHeight` is *somewhere* in `sync_prev_block.chunk_headers_root`, but not *which leaf position* it occupies. Because `chunk_headers_root` is a single merkle root over all shards' chunks, a valid proof for shard 1's chunk at leaf index 1 passes the check even when `shard_id == 0`. The stored header then drives `set_state_finalize`, which applies the wrong chunk and state root to the victim shard's `ShardUId`.

---

### Finding Description

**Entrypoint and data flow**

Any peer can respond to a `StateRequestHeader` network message. The response is deserialized into a `ShardStateSyncResponseHeader` and passed directly to `set_state_header(shard_id, sync_hash, header)`. [1](#0-0) 

**Guard 1 — `validate_chunk_proofs`**

Checks internal chunk consistency: header hash, tx merkle root, outgoing-receipts merkle root. It does **not** compare `chunk.shard_id()` to the requested `shard_id`. [2](#0-1) 

**Guard 2 — `verify_path` (the critical gap)**

```rust
if !verify_path(
    *sync_prev_block_header.chunk_headers_root(),
    shard_state_header.chunk_proof(),
    &ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included()),
) { … }
```

`verify_path` is defined as:

```rust
pub fn verify_path<T: BorshSerialize>(root: MerkleHash, path: &MerklePath, item: T) -> bool {
    verify_hash(root, path, CryptoHash::hash_borsh(item))
}
pub fn verify_hash(root: MerkleHash, path: &MerklePath, item_hash: MerkleHash) -> bool {
    compute_root_from_path(path, item_hash) == root
}
``` [3](#0-2) 

This is a **position-agnostic** membership check. It does not call `verify_path_with_index`, which is the variant that also validates the leaf's ordinal position in the tree. [4](#0-3) 

`chunk_headers_root` is a single merkle root over **all** shards' `ChunkHashHeight` tuples:

```rust
pub fn compute_chunk_headers_root(&self) -> (CryptoHash, Vec<MerklePath>) {
    merklize(&self.iter().map(|chunk|
        ChunkHashHeight(chunk.chunk_hash().clone(), chunk.height_included())
    ).collect_vec())
}
``` [5](#0-4) 

A valid merkle proof for shard 1's chunk at leaf index 1 will satisfy `compute_root_from_path(proof_1, hash(ChunkHashHeight_1)) == chunk_headers_root`. The check passes even when `shard_id == 0`.

**Guard 3 — receipt validation**

The receipt hash is computed with the *parameter* `shard_id`, not `chunk.shard_id()`:

```rust
let receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts));
``` [6](#0-5) 

An attacker supplies the real on-chain receipts destined for shard 0 (publicly available) with their valid proofs. This check passes regardless of which chunk is embedded.

**Guard 4 — `validate_state_root_node`**

Validates the `state_root_node` against `chunk_inner.prev_state_root()` — the chunk's own state root. If the attacker provides shard 1's chunk with shard 1's valid `state_root_node`, this passes. [7](#0-6) 

**Storage of the corrupted binding**

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
``` [8](#0-7) 

`StateHeaderKey(0, sync_hash)` now stores a header whose embedded chunk has `shard_id() == 1`.

**Downstream corruption in `set_state_finalize`**

```rust
let shard_uid = shard_id_to_uid(self.epoch_manager.as_ref(), shard_id, block_header.epoch_id())?;
// shard_uid is for shard 0, but chunk_header.prev_state_root() is shard 1's state root
let apply_result = self.runtime_adapter.apply_chunk(
    RuntimeStorageConfig::new(chunk_header.prev_state_root(), true),
    …
    ApplyChunkShardContext { shard_uid, … },
    …
)?;
``` [9](#0-8) 

`apply_chunk` is called with shard 0's `ShardUId` but shard 1's `prev_state_root`, shard 1's transactions, and shard 1's gas/validator proposals. The result is written to shard 0's storage.

---

### Impact Explanation

A victim node performing state sync installs shard 1's trie state and applies shard 1's chunk under shard 0's `ShardUId`. The node's local `ChunkExtra`, flat storage, and trie for shard 0 are derived from shard 1's data. All subsequent block processing, receipt routing, and state queries for shard 0 operate on the wrong committed state. The node cannot detect the corruption through normal operation because no post-finalize check re-validates `chunk.shard_id() == shard_id`.

---

### Likelihood Explanation

Any peer that the victim selects as a state sync provider can execute this attack. State sync provider selection is not restricted to privileged nodes — any connected peer can respond to `StateRequestHeader` messages. The attacker needs only publicly available on-chain data (shard 1's chunk, its merkle proof, and shard 0's incoming receipts with proofs), all of which are derivable from block headers and chunk data that any full node possesses.

---

### Recommendation

Add an explicit shard-id binding check immediately after extracting the chunk in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        format!("set_shard_state failed: chunk shard_id {:?} != requested shard_id {:?}",
                chunk.shard_id(), shard_id).into()
    ));
}
``` [10](#0-9) 

Apply the same check to `prev_chunk_header.shard_id()`. Additionally, consider replacing `verify_path` with `verify_path_with_index` (passing the shard's index in the layout) so that the merkle proof is bound to the correct leaf position, not just membership in the tree.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
