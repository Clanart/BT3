### Title
Missing Cross-Validation of `state_diff_commitment` Against `state_diff` in `add_sync_block` Produces Wrong Block Hash - (File: `crates/apollo_batcher/src/batcher.rs`)

---

### Summary

`add_sync_block` in `crates/apollo_batcher/src/batcher.rs` accepts `block_header_commitments.state_diff_commitment` from the sync source and stores it verbatim inside `PartialBlockHashComponents` without verifying it equals `calculate_state_diff_hash(&state_diff)`. When the committer's `verify_state_diff_hash` flag is disabled (the flag is opt-in), the wrong commitment propagates through `finalize_commitment_output` → `calculate_block_hash`, producing and persisting a wrong block hash.

---

### Finding Description

**Root cause – unvalidated initialization of `PartialBlockHashComponents`**

`add_sync_block` destructures the incoming `SyncBlock` into `state_diff` and `block_header_commitments` and immediately constructs `StorageCommitmentBlockHash::Partial(PartialBlockHashComponents { header_commitments, … })` without any cross-check:

```rust
// crates/apollo_batcher/src/batcher.rs  lines 686-697
match block_header_commitments {
    Some(header_commitments) => {
        StorageCommitmentBlockHash::Partial(PartialBlockHashComponents {
            header_commitments,          // ← accepted verbatim, never validated
            block_number,
            l1_gas_price: block_header_without_hash.l1_gas_price,
            ...
        })
    }
    None => return Err(BatcherError::MissingHeaderCommitments { block_number }),
}
``` [1](#0-0) 

The `state_diff_commitment` field inside `header_commitments` is never compared against `calculate_state_diff_hash(&state_diff)`.

**Propagation path**

1. `commit_proposal_and_block` writes the `PartialBlockHashComponents` (containing the unvalidated `state_diff_commitment`) to batcher storage. [2](#0-1) 

2. `write_commitment_results_and_add_new_task` forwards `optional_state_diff_commitment = Some(header_commitments.state_diff_commitment)` to the committer. [3](#0-2) 

3. The committer validates the commitment only when `verify_state_diff_hash = true`:

```rust
// crates/apollo_committer/src/committer.rs  lines 165-180
let state_diff_commitment = match state_diff_commitment {
    Some(commitment) => {
        if self.config.verify_state_diff_hash {   // ← opt-in guard
            let calculated = calculate_state_diff_hash(&state_diff);
            if commitment != calculated { return Err(...); }
        }
        commitment
    }
    None => calculate_state_diff_hash(&state_diff),
};
``` [4](#0-3) 

When the flag is `false`, the wrong commitment is accepted and stored as `ForestMetadataType::StateDiffHash`.

4. `finalize_commitment_output` reads `PartialBlockHashComponents` from batcher storage and calls `calculate_block_hash`, which chains `state_diff_commitment` directly into the Poseidon hash: [5](#0-4) [6](#0-5) 

A wrong `state_diff_commitment` therefore produces a wrong `BlockHash` that is persisted and served.

**Contrast with the committer's own path**

When a block is built locally (proposer path), `BlockExecutionArtifacts::new` computes `state_diff_commitment` itself via `calculate_block_commitments`, so the commitment is always consistent with the actual state diff. [7](#0-6) 

The sync path has no equivalent self-computation step.

---

### Impact Explanation

A wrong `state_diff_commitment` stored in `PartialBlockHashComponents` causes `calculate_block_hash` to produce a wrong `BlockHash`. This wrong hash is:

- Persisted in batcher storage and returned by `get_block_hash` (RPC path) — an authoritative-looking wrong value (High impact: RPC returns wrong block hash).
- Used as `ProposalCommitment` in consensus, causing validators to disagree on the block hash and potentially stalling finality.
- Propagated to the SNOS proof input (`OsBlockInput.new_block_hash`) if the prover reads from the same storage, producing a proof over a wrong block hash.

---

### Likelihood Explanation

The trigger is reachable in P2P sync mode: a malicious peer can serve a `SyncBlock` whose `block_header_commitments.state_diff_commitment` does not match the accompanying `state_diff`. `add_sync_block` performs no cross-check. The only guard is the opt-in `verify_state_diff_hash` flag in the committer, which is disabled by default (it is an explicit config opt-in, not a default-on safety check). The consensus orchestrator's `try_sync` already acknowledges missing validations with a `TODO` comment:

```rust
// TODO(Asmaa): validate starknet_version and parent_hash when they are stored.
``` [8](#0-7) 

---

### Recommendation

Add an explicit cross-validation step inside `add_sync_block` immediately after destructuring the sync block, before constructing `PartialBlockHashComponents`:

```rust
// After destructuring SyncBlock:
if let Some(ref hc) = block_header_commitments {
    let expected = calculate_state_diff_hash(&state_diff);
    if hc.state_diff_commitment != expected {
        return Err(BatcherError::StateDiffCommitmentMismatch {
            block_number,
            provided: hc.state_diff_commitment,
            calculated: expected,
        });
    }
}
```

This mirrors the validation already present in the committer (`verify_state_diff_hash` path) but makes it unconditional and early, before any storage write occurs.

Additionally, enable `verify_state_diff_hash = true` by default in the committer configuration so that the committer acts as a second line of defence.

---

### Proof of Concept

1. Malicious peer constructs a `SyncBlock` for block `N` with:
   - `state_diff = S` (legitimate state diff)
   - `block_header_commitments.state_diff_commitment = H_wrong` where `H_wrong ≠ calculate_state_diff_hash(S)`

2. The consensus orchestrator calls `try_sync(N)`, which calls `state_sync_client.get_block(N)` returning the malicious block.

3. `add_sync_block` constructs `PartialBlockHashComponents { header_commitments: { state_diff_commitment: H_wrong, … }, … }` and writes it to batcher storage via `commit_proposal_and_block`. [9](#0-8) 

4. With `verify_state_diff_hash = false`, the committer accepts `H_wrong` and returns `CommitBlockResponse { global_root }`.

5. `finalize_commitment_output` reads `PartialBlockHashComponents` from storage (containing `H_wrong`) and computes:
   ```
   block_hash = Poseidon(STARKNET_BLOCK_HASH1, block_number, global_root, sequencer,
                         timestamp, packed_lengths, H_wrong, tx_commitment, …)
   ```
   producing a wrong `BlockHash` that is stored and returned by all downstream consumers (RPC, consensus, prover). [10](#0-9)

pub fn calculate_block_hash(
    partial_block_hash_components: &PartialBlockHashComponents,
    state_root: GlobalRoot,
    previous_block_hash: BlockHash,
) -> StarknetApiResult<BlockHash> {
    let block_hash_version: BlockHashVersion =
        partial_block_hash_components.starknet_version.try_into()?;
    let block_commitments = &partial_block_hash_components.header_commitments;
    Ok(BlockHash(
        HashChain::new()
            .chain(&block_hash_version.clone().into())
            .chain(&partial_block_hash_components.block_number.0.into())
            .chain(&state_root.0)
            .chain(&partial_block_hash_components.sequencer.0)
            .chain(&partial_block_hash_components.timestamp.0.into())
            .chain(&block_commitments.concatenated_counts)
            .chain(&block_commitments.state_diff_commitment.0.0)
            .chain(&block_commitments.transaction_commitment.0)
            .chain(&block_commitments.event_commitment.0)
            .chain(&block_commitments.receipt_commitment.0)
            .chain_iter(
                gas_prices_to_hash(
                    &partial_block_hash_components.l1_gas_price,
                    &partial_block_hash_components.l1_data_gas_price,
                    &partial_block_hash_components.l2_gas_price,
                    &block_hash_version,
                )
                .iter(),
            )
            .chain(
                &Felt::try_from(&partial_block_hash_components.starknet_version)
                    .expect("Expect ASCII version"),
            )
            .chain(&Felt::ZERO)
            .chain(&previous_block_hash.0)
            .get_poseidon_hash(),
    ))
}
