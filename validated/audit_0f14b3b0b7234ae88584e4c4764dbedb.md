### Title
Felt Arithmetic Overflow in `compute_max_possible_fee` Enables Complete Fee Evasion — (`crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee` computes the transaction fee ceiling using unchecked felt (modular) arithmetic. Because `max_price_per_unit` is only constrained to be non-negative (≤ (P−1)/2 ≈ 2^250) while `max_amount` is bounded to 2^64−1, their product can silently wrap around modulo the StarkNet field prime P. A user can craft resource-bounds values that make the felt-arithmetic sum equal exactly 0, causing `charge_fee` to return immediately without transferring any tokens — a complete, provably-valid fee bypass.

---

### Finding Description

**Root cause — missing upper bound on `max_price_per_unit`**

`pack_resource_bounds` (called during transaction-hash computation) enforces two constraints on each `ResourceBounds` entry:

```cairo
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // max_amount ∈ [0, 2^64−1]
assert_nn(resource_bounds.max_price_per_unit);            // max_price_per_unit ∈ [0, (P−1)/2]
``` [1](#0-0) 

`assert_nn` only checks non-negativity; it places no upper bound tighter than (P−1)/2 ≈ 2^250. The `tip` field is bounded to 2^64−1 by a separate `assert_nn_le`. [2](#0-1) 

**Overflow in fee computation**

`compute_max_possible_fee` then uses these same `resource_bounds` values in plain felt arithmetic:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
``` [3](#0-2) 

With `max_amount` up to 2^64−1 and `max_price_per_unit` up to (P−1)/2 ≈ 2^250, a single term can reach ≈ 2^314, far exceeding P. Cairo felt arithmetic silently reduces this modulo P. No overflow check or range assertion is applied to the result.

**Fee-bypass gate**

`charge_fee` immediately returns without executing any ERC-20 transfer when `max_fee == 0`:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();
}
``` [4](#0-3) 

The same `resource_bounds` pointer flows from `get_account_tx_common_fields` → `fill_account_tx_info` → `TxInfo.resource_bounds_start` → `compute_max_possible_fee`, so the hash-time constraints and the fee-time arithmetic operate on identical data. [5](#0-4) 

---

### Impact Explanation

**Critical — Direct loss of funds.**

A user who successfully makes `compute_max_possible_fee` return 0 executes any transaction (invoke, declare, deploy-account) with zero fee charged. The sequencer receives no payment. Because the OS Cairo code is the authoritative computation that is proven and verified on L1, a valid STARK proof can be generated for a block containing such a transaction, permanently encoding the fee-free execution into the canonical chain state. The sequencer's off-chain fee validation (which may use big-integer arithmetic rather than felt arithmetic) would compute a large, non-zero `max_fee` and accept the transaction — but the OS would then skip charging.

---

### Likelihood Explanation

**Medium.** The attacker is an unprivileged transaction sender. The required resource-bounds values are fully user-controlled and must only satisfy the two Cairo constraints (`assert_nn_le` for `max_amount`, `assert_nn` for `max_price_per_unit`). No privileged access, key material, or external dependency is needed. The arithmetic to find a zero-sum combination is straightforward modular algebra (shown concretely below). The main friction is that a sequencer's off-chain mempool logic might reject transactions with anomalously large `max_price_per_unit`; however, the OS itself imposes no such rejection, so a sequencer that does not independently guard against felt overflow will include the transaction.

---

### Recommendation

1. **Add an upper-bound assertion on `max_price_per_unit` in `pack_resource_bounds`:**

   ```cairo
   assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);
   ```

   This matches the 128-bit slot allocated for `max_price_per_unit` in the packed representation and keeps the product `max_amount * max_price_per_unit` within ≈ 2^192, well below P.

2. **Add a post-computation non-zero / range assertion in `compute_max_possible_fee`** to make the invariant explicit and catch any future arithmetic path that could produce 0 unexpectedly.

3. **Document the assumed unit and scale** of `max_price_per_unit` and `tip` (analogous to the ChainlinkCalculator fix of documenting decimal assumptions), so that future callers and auditors can verify consistency.

---

### Proof of Concept

StarkNet prime: P = 2^251 + 17·2^192 + 1.

Choose resource bounds:

| Resource | `max_amount` | `max_price_per_unit` | `tip` |
|---|---|---|---|
| L1_GAS | 2 | (P−1)/2 | — |
| L2_GAS | 1 | 0 | 0 |
| L1_DATA_GAS | 1 | 1 | — |

**Constraint checks:**
- `assert_nn_le(2, 2^64−1)` ✓
- `assert_nn((P−1)/2)` ✓ — this is exactly the maximum non-negative felt value
- `assert_nn_le(0, 2^64−1)` ✓ (tip)

**Fee computation (mod P):**

```
A = 2 × (P−1)/2  =  P−1  ≡  −1  (mod P)
B = 1 × (0 + 0)  =  0
C = 1 × 1        =  1

max_fee = A + B + C = −1 + 0 + 1 = 0  (mod P)
```

`compute_max_possible_fee` returns 0. `charge_fee` hits the `if (max_fee == 0) { return (); }` branch and exits without executing the ERC-20 transfer. The transaction is executed and proven with zero fee paid. [6](#0-5) [7](#0-6)
