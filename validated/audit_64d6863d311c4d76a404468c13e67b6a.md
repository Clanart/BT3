### Title
`swap_migrated_token` Executes Token Swap Without Error Handling, Causing Permanent Fund Loss on Mint Failure - (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

The `swap_migrated_token` function in the NEAR bridge contract burns old (migrated) tokens and mints new tokens as a **detached, unchecked promise**. The `ft_on_transfer` handler immediately returns `U128(0)` — consuming all transferred tokens — before the burn/mint promises execute. If the `mint` call fails (e.g., the recipient has not registered storage on the new token), the old tokens are burned from the bridge's balance but new tokens are never delivered. The user permanently loses their funds with no recovery path.

---

### Finding Description

When a user sends old tokens to the bridge with `BridgeOnTransferMsg::SwapMigratedToken`, the `ft_on_transfer` handler dispatches the swap promise and immediately returns `U128(0)`:

```rust
BridgeOnTransferMsg::SwapMigratedToken => {
    self.swap_migrated_token(sender_id, token_id, amount)
        .detach();                                          // ← no error handling
    PromiseOrPromiseIndexOrValue::Value(U128(0))           // ← all tokens kept by bridge
}
``` [1](#0-0) 

Returning `U128(0)` in NEP-141's `ft_on_transfer` means the bridge retains all transferred tokens — none are refunded to the sender. The swap itself is:

```rust
fn swap_migrated_token(&mut self, sender_id: AccountId, old_token: AccountId, amount: U128) -> Promise {
    let new_token = self.migrated_tokens.get(&old_token).near_expect(BridgeError::TokenNotMigrated);
    let burn = ext_token::ext(old_token).burn(amount);
    let mint = ext_token::ext(new_token).mint(sender_id, amount, None);
    burn.and(mint)
}
``` [2](#0-1) 

`burn.and(mint)` schedules two **independent cross-contract receipts** in NEAR. They are not atomic — if `mint` fails, `burn` is not rolled back. Since the promise is `.detach()`, the bridge contract is never notified of failure and cannot refund the user.

The `OmniToken::mint` function calls `self.token.internal_deposit(&account_id, amount.into())`:

```rust
fn mint(&mut self, account_id: AccountId, amount: U128, msg: Option<String>) -> PromiseOrValue<U128> {
    self.assert_controller();
    // ...
    self.token.internal_deposit(&account_id, amount.into());  // panics if account not registered
    PromiseOrValue::Value(amount)
}
``` [3](#0-2) 

`internal_deposit` in the standard NEP-141 `FungibleToken` panics if the recipient account has not called `storage_deposit` on the new token contract. This is a realistic condition: after a DAO-initiated token migration, users holding old tokens are expected to swap them, but the new token is a freshly deployed contract with no pre-registered user accounts.

`migrate_deployed_token` only deposits storage for the **bridge contract itself** on the new token — not for any user:

```rust
ext_token::ext(new_token.clone())
    .with_static_gas(STORAGE_DEPOSIT_GAS)
    .with_attached_deposit(NEP141_DEPOSIT)
    .storage_deposit(&env::current_account_id(), Some(true))  // only bridge, not users
    .detach();
``` [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent, irrecoverable loss of user funds.**

The execution sequence on mint failure:

1. User calls `ft_transfer_call(bridge, amount, "SwapMigratedToken")` — old tokens leave user's wallet.
2. Bridge's `ft_on_transfer` returns `U128(0)` — old tokens are now owned by the bridge, not refundable.
3. `burn(old_token, amount)` receipt executes — bridge's balance of old tokens is destroyed.
4. `mint(new_token, sender_id, amount)` receipt panics — user not registered on new token.
5. NEAR runtime marks the mint receipt failed; burn is already committed and not rolled back.
6. User has lost their tokens with no recovery mechanism.

This matches the allowed impact: **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows."**

---

### Likelihood Explanation

**Medium-High.** After a DAO-initiated token migration via `migrate_deployed_token`, every holder of the old token who attempts to swap without first manually calling `storage_deposit` on the new token contract will trigger this loss. The new token is a freshly deployed contract; no user has storage registered on it by default. There is no on-chain prompt or enforcement requiring users to pre-register. A single announcement gap or UI omission is sufficient to cause widespread losses across all migrating users.

---

### Recommendation

Replace the fire-and-forget detached promise with a callback that handles mint failure and refunds the user's old tokens:

```rust
BridgeOnTransferMsg::SwapMigratedToken => {
    self.swap_migrated_token(sender_id, token_id, amount)
        .then(
            Self::ext(env::current_account_id())
                .with_static_gas(SWAP_MIGRATED_TOKEN_CALLBACK_GAS)
                .swap_migrated_token_callback(token_id, amount, sender_id),
        )
        .as_return();
    // Return amount (refund) only if callback signals failure
}
```

In the callback, check the mint result. If it failed, re-mint the old tokens back to the user (since burn already executed) or, if burn also failed, transfer the old tokens back. Alternatively, perform the burn **only after** confirming the mint succeeded by chaining the promises sequentially: `mint.then(burn_callback)`.

---

### Proof of Concept

1. DAO calls `migrate_deployed_token(ChainKind::Eth, old_token, new_token)`.
2. Alice holds 1000 units of `old_token` and has not called `storage_deposit` on `new_token`.
3. Alice calls `old_token.ft_transfer_call(bridge, 1000, "\"SwapMigratedToken\"")`.
4. Bridge's `ft_on_transfer` returns `U128(0)` — Alice's 1000 old tokens are now in the bridge.
5. Bridge schedules `burn(old_token, 1000)` and `mint(new_token, alice, 1000)` as independent receipts.
6. `burn` succeeds — bridge's 1000 old tokens are destroyed.
7. `mint` panics: `internal_deposit` fails because Alice has no storage on `new_token`.
8. Alice has 0 old tokens and 0 new tokens. Funds are permanently lost. [1](#0-0) [2](#0-1) [5](#0-4)
