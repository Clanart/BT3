Audit Report

## Title
Stranded ETH in Router Consumed by Subsequent WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` treats the router's entire native ETH balance as belonging to the current caller when settling a WETH payment. ETH left in the router from a prior user's multicall that omitted `refundETH` can be silently consumed to fund a subsequent user's WETH swap, with zero pull from the attacker's wallet. The prior user loses their stranded ETH with no recourse.

## Finding Description

In `pay()`, the WETH branch reads the contract's global ETH balance and, if it covers the requested `value`, wraps and forwards it to the pool without any check that the current `payer` deposited that ETH: [1](#0-0) 

The `payer` is always `msg.sender` of the swap entry-point, set via transient storage at line 71: [2](#0-1) 

And forwarded directly to `pay()` in `_justPayCallback`: [3](#0-2) 

There is no per-caller ETH accounting. `address(this).balance` is a shared, unattributed pool. The `receive()` guard (only accepts ETH from WETH) does not prevent ETH from entering via `msg.value` on payable entry-points like `multicall` or `exactInputSingle`. [4](#0-3) 

`refundETH` sends the entire balance to `msg.sender`, not the original depositor, so it cannot safely be called by a third party and does not protect against a front-running attacker: [5](#0-4) 

## Impact Explanation

Direct loss of user principal. User A's ETH is irreversibly consumed to settle User B's swap obligation. User A receives nothing; User B receives full swap output while paying zero WETH from their wallet. Loss equals the stranded ETH amount, up to the full `msg.value` of User A's multicall. This satisfies the "Critical/High direct loss of user principal" impact gate.

## Likelihood Explanation

The precondition — ETH stranded in the router — is realistic and exploitable:
- Users routinely omit `refundETH` from multicalls, especially when they expect the swap to consume the exact amount sent.
- Slippage or partial fills leave residual ETH.
- An attacker can monitor the mempool for multicalls omitting `refundETH` and immediately follow with a WETH swap sized to exactly the stranded amount, requiring zero capital and no special privileges.
- The attack is repeatable and permissionless.

## Recommendation

Track per-call deposited ETH using transient storage. At the start of each top-level entry-point (or multicall entry), record `msg.value` as the "authorized native balance" for this call. In `pay()`, deduct only from that authorized amount rather than from `address(this).balance` globally, and revert if the authorized balance is insufficient. Clear the authorized balance at the end of the call. Alternatively, enforce that `address(this).balance` is zero at the start of every non-multicall entry-point, and that `multicall` tracks and enforces per-call ETH budgets.

## Proof of Concept

```solidity
// Step 1: User A sends 1 ETH, swaps 0.9 ETH worth of WETH, omits refundETH
router.multicall{value: 1 ether}([
    abi.encodeCall(router.exactInputSingle, (ExactInputSingleParams({
        tokenIn: WETH, tokenOut: token1, amountIn: 0.9 ether, ...
    })))
    // no refundETH call
]);
// router.balance == 0.1 ETH (User A's stranded ETH)

// Step 2: Attacker (User B) calls exactInputSingle for WETH, sends 0 ETH
router.exactInputSingle{value: 0}(ExactInputSingleParams({
    tokenIn: WETH, tokenOut: token1, amountIn: 0.1 ether, ...
}));
// Inside pay(WETH, UserB, pool, 0.1 ether):
//   nativeBalance = 0.1 ETH >= value = 0.1 ETH
//   → deposit{value: 0.1 ETH}() + safeTransfer(pool, 0.1 WETH)
//   → UserB pulls 0 WETH from wallet

// Result:
assert(address(router).balance == 0);   // User A's 0.1 ETH is gone
// UserB received full swap output; UserA lost 0.1 ETH
```

The broken invariant — each user's ETH is consumed only for their own swap — is rooted in the `nativeBalance >= value` branch of `pay()` at lines 74–77, which treats the entire contract ETH balance as belonging to the current caller. [6](#0-5)
