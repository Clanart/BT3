### Title
No-op `check_is_reverted` Allows Prover to Falsely Mark Valid Transactions as Reverted — (File: `execution/execution_constraints.cairo`)

---

### Summary

The production OS `check_is_reverted` function is a complete no-op. The `is_reverted` value is loaded exclusively from a prover-controlled hint (`%{ IsReverted %}`) and is never constrained by any Cairo assertion. A malicious prover (sequencer) can set `is_reverted = TRUE` for any valid invoke transaction, causing the execute step to be silently skipped while fees are still charged — resulting in direct loss of user funds with a cryptographically valid proof.

---

### Finding Description

In `execution_constraints.cairo` (the production OS path), the function that is supposed to validate the revert status is:

```cairo
func check_is_reverted(is_reverted: felt) {
    return ();
}
```

It performs no assertion whatsoever. [1](#0-0) 

Compare this to the virtual OS counterpart in `execution_constraints__virtual.cairo`, which actually enforces a constraint:

```cairo
func check_is_reverted(is_reverted: felt) {
    with_attr error_message("Reverted transactions are not supported in virtual OS mode") {
        assert is_reverted = FALSE;
    }
    return ();
}
``` [2](#0-1) 

The production OS call site in `execute_invoke_function_transaction` is:

```cairo
local is_reverted;
%{ IsReverted %}
check_is_reverted(is_reverted);          // ← no-op; is_reverted is unconstrained
if (is_reverted == FALSE) {
    // Execute only non-reverted transactions.
    ...
    non_reverting_select_execute_entry_point_func(...);
} else {
    // Execute step is entirely skipped.
    ...
}

// Charge fee — always runs regardless of is_reverted.
charge_fee(block_context=block_context, tx_execution_context=updated_tx_execution_context);
``` [3](#0-2) 

Because `check_is_reverted` is a no-op, the Cairo verifier places **zero constraints** on the `is_reverted` felt. The prover can assign it any value and the resulting STARK proof remains valid. The `is_reverted` branch then silently skips `non_reverting_select_execute_entry_point_func` while `charge_fee` still executes unconditionally.

This is the direct Cairo analog of the RUSD.sol pattern: a guard function that is supposed to protect downstream logic but performs no actual check, leaving the guarded path completely unprotected.

---

### Impact Explanation

**Critical — Direct loss of funds / Permanent freezing of funds.**

A malicious sequencer can:

1. Accept a valid user invoke transaction (e.g., a token transfer, a DeFi interaction, a withdrawal).
2. Set `is_reverted = TRUE` in the `%{ IsReverted %}` hint.
3. The execute step is skipped; no state changes from the user's intended action are applied.
4. `charge_fee` still runs and deducts the full fee from the user's account.
5. The generated STARK proof is cryptographically valid — the verifier accepts it.
6. On-chain state reflects: fee deducted, transaction "reverted", user's intended action never executed.

If the skipped transaction was a withdrawal or an unlock of escrowed funds, those funds are permanently frozen with no on-chain recourse. If it was a routine transfer, the user suffers fee loss with no execution. This is repeatable for every invoke transaction in every block.

---

### Likelihood Explanation

The attack requires the sequencer to be malicious. In StarkNet's current architecture the sequencer is a centralized operator. The entire purpose of the ZK proof system is to make the sequencer's behavior verifiable and trustless — the verifier is supposed to catch any deviation from correct execution. Because `check_is_reverted` is a no-op, this specific deviation (lying about revert status) is **not caught by the verifier**, defeating the trustless guarantee for this code path. The sequencer needs no leaked key, no external dependency, and no network-level capability — only the ability to set a single hint value, which is already part of normal sequencer operation.

---

### Recommendation

The production OS `check_is_reverted` must enforce a verifiable constraint that ties `is_reverted` to the actual execution outcome. At minimum, the revert status must be derived from a verifiable execution trace rather than an unconstrained hint. The virtual OS version demonstrates the correct pattern — an `assert` that the verifier can check. For the production OS, the revert status should be derived from the return value of the entry-point execution itself (e.g., the `is_reverted` return of `select_execute_entry_point_func`) rather than from a free hint, so that the prover cannot supply an inconsistent value without invalidating the proof.

---

### Proof of Concept

1. Sequencer receives a valid invoke transaction (e.g., `transfer(recipient, 1000_STRK)`).
2. Sequencer sets hint `IsReverted → is_reverted = 1` (TRUE).
3. Cairo execution reaches `check_is_reverted(1)` → returns immediately, no assertion fires.
4. Branch `if (is_reverted == FALSE)` is not taken; `non_reverting_select_execute_entry_point_func` is never called; no storage writes for the transfer occur.
5. `charge_fee` executes normally; the user's fee token balance is reduced.
6. STARK proof is generated and verified successfully — all Cairo constraints are satisfied because `check_is_reverted` imposed none.
7. L1 verifier accepts the proof; block is finalized.
8. On-chain result: user paid fees, recipient received nothing, funds remain locked in sender's account with no protocol-level remedy. [1](#0-0) [4](#0-3) [2](#0-1)
