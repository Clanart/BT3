### Title
Felt Arithmetic Underflow in `check_proof_facts` When `current_block_number < STORED_BLOCK_HASH_BUFFER` - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/execution_constraints.cairo`)

---

### Summary

`check_proof_facts` computes `current_block_number - STORED_BLOCK_HASH_BUFFER` without first verifying that `current_block_number >= STORED_BLOCK_HASH_BUFFER`. During the first 10 blocks of the network, this subtraction underflows in Cairo's prime-field arithmetic, producing a value near the field prime P. The subsequent `assert_nn_le` call then fails because the huge felt value exceeds the range-check bound (2^128), causing the entire OS execution to abort. A block cannot be proven, halting the network.

---

### Finding Description

In `execution_constraints.cairo`, `check_proof_facts` validates that a user-supplied `proof_facts` block number is not too recent:

```cairo
assert_nn_le(
    os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
);
``` [1](#0-0) 

`STORED_BLOCK_HASH_BUFFER` is the constant `10`. [2](#0-1) 

When `current_block_number < 10` (i.e., during the first 10 blocks of the network), the subtraction `current_block_number - STORED_BLOCK_HASH_BUFFER` wraps around in Cairo's felt arithmetic to approximately `P - (10 - current_block_number)`, a value near the Stark prime. `assert_nn_le(a, b)` internally checks that both `a` and `b - a` fit within the range-check bound (< 2^128). A felt near P fails this check unconditionally, causing a hard assertion failure that aborts the entire OS proof.

The same subtraction is performed safely elsewhere in the codebase. `write_block_number_to_block_hash_mapping` in `os_utils.cairo` explicitly guards against this case:

```cairo
tempvar old_block_number = block_context.block_info_for_execute.block_number -
    STORED_BLOCK_HASH_BUFFER;
let is_old_block_number_non_negative = is_nn(old_block_number);
if (is_old_block_number_non_negative == FALSE) {
    // Not enough blocks in the system - nothing to write.
    return ();
}
``` [3](#0-2) 

`check_proof_facts` lacks this guard entirely.

The function is entered only when `proof_facts_size != 0`:

```cairo
if (proof_facts_size == 0) {
    return ();
}
``` [4](#0-3) 

So the trigger condition is: an invoke transaction with a non-empty `proof_facts` payload submitted while `current_block_number < STORED_BLOCK_HASH_BUFFER`.

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

`check_proof_facts` is called from within the OS's transaction processing loop (`transaction_impls.cairo`). A failure inside `assert_nn_le` is a hard Cairo assertion, not a graceful transaction revert. It aborts the entire OS execution, making the block unprovable. No block can be finalized until the condition is resolved, constituting a total network shutdown during the first 10 blocks of the network.

---

### Likelihood Explanation

The vulnerable window is the first 10 blocks of the network (`current_block_number` in `[0, 9]`). Any unprivileged user who submits an invoke transaction carrying a non-empty `proof_facts` field during this window triggers the underflow. No special privilege, key, or operator access is required. The attacker only needs to know the network has just launched and submit a single crafted transaction.

---

### Recommendation

Apply the same guard used in `write_block_number_to_block_hash_mapping`. Before computing the upper bound, verify that `current_block_number >= STORED_BLOCK_HASH_BUFFER`. If not, no valid `base_block_number` can satisfy the constraint, so the check should fail gracefully (or the transaction should be rejected before reaching this point):

```cairo
// Guard against underflow on early blocks.
if (current_block_number < STORED_BLOCK_HASH_BUFFER) {
    // No valid base_block_number exists; reject.
    assert 1 = 0;  // or return an appropriate error
}
assert_nn_le(
    os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
);
```

Alternatively, mirror the `is_nn` pattern from `os_utils.cairo` exactly.

---

### Proof of Concept

1. Network launches; `current_block_number = 5` (< `STORED_BLOCK_HASH_BUFFER = 10`).
2. Attacker submits an invoke transaction with `proof_facts_size > 0` and any `base_block_number`.
3. OS calls `check_proof_facts(..., current_block_number=5, ...)`.
4. `proof_facts_size != 0` → early return is skipped.
5. Line 67 evaluates: `5 - 10` in felt arithmetic → `P - 5` (≈ 2^251, far above 2^128).
6. `assert_nn_le(base_block_number, P - 5)` → range-check on `P - 5` fails → hard abort.
7. The block containing this transaction cannot be proven → network halted. [5](#0-4)
