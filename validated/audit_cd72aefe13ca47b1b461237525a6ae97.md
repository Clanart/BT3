### Title
`batch_insert` Silently Accepts Duplicate Keys, Corrupting DataLayer Merkle Tree Root - (File: `crates/chia-datalayer/src/merkle/blob.rs`)

### Summary

`MerkleBlob::batch_insert` bypasses the duplicate-key and duplicate-hash guards that `MerkleBlob::insert` enforces. When a batch containing a repeated `KeyId` (or repeated leaf hash) is processed via the fast path, `BlockStatusCache::add_leaf` silently overwrites the `key_to_index` and `leaf_hash_to_index` maps, leaving a ghost leaf node physically present in the blob but unreachable through the cache. The resulting tree has two leaves sharing the same key, producing an incorrect Merkle root and invalidating all subsequent inclusion/exclusion proofs derived from it.

### Finding Description

`MerkleBlob::insert` guards against duplicate keys and hashes at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` calls `self.insert()` (with those guards) only for the last two items of the input vector when the tree has ≤ 1 existing leaves. Every other item is written directly through `insert_entry_to_blob` with no duplicate check:

```rust
for ((key, value), hash) in keys_values_hashes {   // fast path – no guard
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ... Node::Leaf(LeafNode { key, value, hash, .. }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` calls `block_status_cache.add_leaf`, which performs an unconditional `HashMap::insert` on both maps:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);        // silently overwrites
    self.leaf_hash_to_index.insert(leaf.hash, index); // silently overwrites
}
```

When a duplicate key appears in the fast-path portion of the batch:

1. The first leaf (index N) is written to the blob and `key_to_index[key] = N`.
2. The second leaf (index M) is written to the blob and `key_to_index[key] = M`, silently overwriting N.
3. Both leaf nodes are physically present in the blob and are wired into the tree's internal-node structure by the subsequent parent-building loop.
4. `get_leaf_by_key(key)` returns only the leaf at M; the leaf at N is orphaned but still contributes its hash to the tree.
5. The computed root hash is wrong: it is derived from a tree that contains two leaves with the same key.
6. Any `ProofOfInclusion` generated afterward is invalid.
7. `check_integrity()` will eventually detect the corruption (`IntegrityKeyToIndexCacheIndex` or `IntegrityKeyToIndexCacheLength`), but only after the bad state has already been committed.

The two conditions that trigger the fast path are:

- The tree already has ≥ 2 leaves (all batch items go through the fast path), **or**
- The tree has ≤ 1 leaves and the batch has ≥ 3 items; the first N−2 items go through the fast path unchecked.

### Impact Explanation

The DataLayer Merkle tree root is the commitment used to prove inclusion and exclusion of key-value pairs. A corrupted root means:

- Proofs of inclusion generated from the tree are structurally invalid (they traverse a tree with a duplicate-key leaf that the cache cannot address).
- The stored root hash diverges from what a correctly-built tree would produce, so any peer that independently recomputes the root from the same key-value set will disagree.
- This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

`batch_insert` is a public API exposed through Python bindings (`MerkleBlob.batch_insert`). DataLayer sync operations that bulk-load key-value pairs from an untrusted peer can supply a list containing repeated keys. No caller-side deduplication is enforced or documented as a precondition. An attacker who can influence the content of a DataLayer delta (e.g., a malicious DataLayer store publisher) can craft a batch with duplicate keys and trigger this path on any syncing node.

### Recommendation

Add the same duplicate-key and duplicate-hash guards to the fast path inside `batch_insert` that already exist in `insert`. Concretely, before calling `insert_entry_to_blob` for each item in the loop, check:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(&hash) {
    return Err(Error::HashAlreadyPresent());
}
```

Alternatively, deduplicate the input vector before entering the fast path, or make `add_leaf` return an error on collision instead of silently overwriting.

### Proof of Concept

```
1. Create a MerkleBlob with 2 existing leaves (so leaf_count > 1 → all batch items use the fast path).
2. Call batch_insert with a Vec containing two entries sharing the same KeyId, e.g.:
       [((KeyId(42), ValueId(1)), hash_a), ((KeyId(42), ValueId(2)), hash_b)]
3. batch_insert returns Ok(()).
4. The blob now contains two LeafNode entries with key=42 at different tree indexes.
5. block_status_cache.key_to_index[KeyId(42)] points only to the second leaf.
6. get_root_hash() returns a hash derived from both leaves, which no honest node
   building the same key-value set would reproduce.
7. get_proof_of_inclusion(KeyId(42)) walks from the second leaf; the first leaf's
   hash is silently included in an ancestor internal node, making the proof
   unverifiable against any canonical root.
8. check_integrity() returns IntegrityKeyToIndexCacheIndex or
   IntegrityKeyToIndexCacheLength, confirming tree corruption.
```

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),
            Ordering::Less => {
                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);
            }
        }

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }

        Ok(())
    }
