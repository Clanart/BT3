### Title
Missing Nonce in `meta_tx_v0` Hash Enables Permanent Signature Replay — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

`compute_meta_tx_v0_hash` computes the hash that a signer must authorize for a `meta_tx_v0` syscall. Unlike every other transaction type in the OS, this hash is computed with `additional_data_size=0`, meaning **no nonce is committed to**. A valid signature over this hash is therefore valid forever and for any number of replays, directly analogous to the MutualConsent bug where stored consent is never invalidated.

---

### Finding Description

In `transaction_hash.cairo`, `compute_meta_tx_v0_hash` calls `deprecated_get_transaction_hash` with `additional_data_size=0` and `additional_data=cast(0, felt*)`:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        ...
        additional_data_size=0,          // ← no nonce
        additional_data=cast(0, felt*),
    );
    return tx_hash;
}
``` [1](#0-0) 

Compare this to `compute_l1_handler_transaction_hash`, which correctly passes `additional_data_size=1, additional_data=&nonce` to bind the hash to a specific nonce: [2](#0-1) 

And compare to all v3 account transactions (`invoke`, `declare`, `deploy_account`), which include the nonce via `hash_tx_common_fields`: [3](#0-2) 

The `meta_tx_v0` syscall is dispatched from `execute_syscalls` and calls `execute_meta_tx_v0`: [4](#0-3) 

Because the hash is a pure function of `(INVOKE_HASH_PREFIX, 0, contract_address, entry_point_selector, calldata_hash, 0, chain_id)` with no nonce, the same hash — and therefore the same valid signature — is produced every time the same call parameters are used. The OS will accept this signature on every replay.

---

### Impact Explanation

**Critical — Direct loss of funds.**

An attacker who observes a single valid `meta_tx_v0` authorization (e.g., a signed token transfer from victim account A to address B) can replay it an unlimited number of times within any future block. Each replay executes the same entry point with the same calldata under the authority of the original signer. If the authorized call moves funds (e.g., ERC-20 `transfer`, `approve`, or any value-bearing call), the victim's entire balance can be drained across successive blocks with no further cooperation from the victim.

---

### Likelihood Explanation

**High.** The `meta_tx_v0` syscall is reachable by any unprivileged transaction sender: a regular invoke transaction calls a contract that issues the `META_TX_V0_SELECTOR` syscall. No privileged role, leaked key, or operator cooperation is required beyond observing one valid signed hash (e.g., from a mempool, a block explorer, or a prior block). The replay requires only submitting a new transaction with the same parameters and the previously-seen signature.

---

### Recommendation

Include a nonce in the `meta_tx_v0` hash, exactly as `compute_l1_handler_transaction_hash` does for L1 handlers. The nonce must be:

1. Committed to in the signed hash (`additional_data_size=1, additional_data=&nonce`).
2. Checked and incremented in the OS execution path for `meta_tx_v0`, analogous to `check_and_increment_nonce` used for account transactions. [5](#0-4) 

This ensures each authorization is single-use, invalidating any previously-captured signature when the nonce advances.

---

### Proof of Concept

1. Victim account `A` authorizes a `meta_tx_v0` call: `transfer(recipient=attacker, amount=100)` on the fee token contract. The OS computes `H = compute_meta_tx_v0_hash(fee_token_address, TRANSFER_SELECTOR, calldata, chain_id)`. Victim signs `H` → signature `σ`.

2. Attacker observes `(H, σ)` from the mempool or a confirmed block.

3. Attacker submits a new invoke transaction that issues the `META_TX_V0_SELECTOR` syscall with the same `(contract_address, selector, calldata)` and presents `σ`.

4. The OS recomputes `H` — identical, because no nonce is included — and accepts `σ` as valid.

5. The transfer executes again, moving another 100 tokens from `A` to the attacker.

6. Steps 3–5 repeat until `A`'s balance is zero. Each iteration requires only a new outer transaction from the attacker; no new signature from the victim is needed. [6](#0-5)
