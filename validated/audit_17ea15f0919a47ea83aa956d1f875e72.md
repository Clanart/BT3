Audit Report

## Title
Public `unwrapWETH9` Sweeps Entire Router WETH Balance to Attacker-Chosen Recipient — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`unwrapWETH9` is a `public payable` function with no caller binding: it reads the router's total WETH balance, withdraws all of it, and forwards the ETH to a caller-supplied `recipient`. Any WETH that reaches the router — from a swap where `recipient: address(router)` was specified — is immediately claimable by any third party who calls `unwrapWETH9(0, attacker)` before the legitimate user does. The same structural flaw exists in `sweepToken`.

## Finding Description
`unwrapWETH9` at `metric-periphery/contracts/base/PeripheryPayments.sol` L37–45:

```solidity
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));   // aggregate router balance
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);   // attacker-controlled address
    }
}
```

Three properties combine into a theft path:

1. **No caller binding** — `recipient` is a free parameter; `msg.sender` is never consulted.
2. **Whole-balance sweep** — `balanceWETH` is `IERC20(WETH).balanceOf(address(this))`, the aggregate of every user's WETH currently held by the router.
3. **Zero-minimum bypass** — passing `amountMinimum=0` disables the only guard.

The documented and tested usage pattern (confirmed in `metric-periphery/test/MetricOmmSimpleRouter.native.t.sol` L135–162) is to batch `exactInputSingle` with `recipient: address(router)` and `unwrapWETH9` atomically inside a single `multicall`. However, `multicall` at `MetricOmmSimpleRouter.sol` L39–44 is a plain delegatecall loop with no transient reentrancy flag and no enforcement that `unwrapWETH9` must be called within it. Any user who issues the swap and the unwrap in two separate transactions leaves WETH stranded on the router, where it is immediately claimable by any caller.

`sweepToken` at L48–55 has the identical structure: it sweeps the entire ERC-20 balance to an arbitrary `recipient` with no caller binding.

## Impact Explanation
Direct, complete loss of the victim's WETH principal. The attacker receives the full ETH equivalent; the victim's subsequent `unwrapWETH9` call either reverts (if `amountMinimum > 0`) or silently sends nothing. No privileged role is required. The attack is deterministic and repeatable for every user who routes WETH output through the router without using `multicall`. This is a direct loss of user principal above Sherlock thresholds, qualifying as High severity.

## Likelihood Explanation
The `multicall`-atomic pattern is the recommended usage, but the router exposes `unwrapWETH9` as a standalone `public payable` function with no enforcement of atomicity. Any wallet or integration that does not support `multicall` and issues the swap and unwrap in separate transactions is immediately vulnerable. Front-running is trivial on any chain with a public mempool; even without front-running, a passive attacker can monitor the router's WETH balance and drain it between blocks. The condition is reachable by any unprivileged caller.

## Recommendation
Bind `unwrapWETH9` (and `sweepToken`) to `msg.sender` as the only permitted recipient, removing the free `recipient` parameter:

```solidity
function unwrapWETH9(uint256 amountMinimum) public payable {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(msg.sender, balanceWETH);
    }
}
```

Alternatively, enforce that these helpers are only callable within a `multicall` context via a transient reentrancy flag set by `multicall` and checked by `unwrapWETH9`/`sweepToken`.
