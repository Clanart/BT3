### Title
Missing `metadata.mint` Validation in `parse_metadata_account` Allows Token Identity Spoofing — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

`parse_metadata_account` verifies that the supplied metadata account's **address** matches the mint's `metadata_pointer` and that the account is **owned by MetaplexID**, but never checks that the Metaplex metadata account's internal `mint` field equals `self.mint.key()`. An unprivileged attacker can create a Token-2022 mint whose `metadata_pointer` points to a *different* mint's Metaplex PDA, call `log_metadata`, and cause NEAR to register the attacker's mint with the victim token's name and symbol.

---

### Finding Description

In `process()`, the Token-2022 branch reads the `MetadataPointer` extension from the caller-supplied mint and, when the pointer is neither self-referential nor zero, delegates to `parse_metadata_account` with the pointer's target address: [1](#0-0) 

`parse_metadata_account` performs two checks:

1. The passed-in `metadata` account key equals the `address` argument (i.e., `metadata_pointer.metadata_address.0`).
2. The account is owned by `MetaplexID`. [2](#0-1) 

After those two checks pass it immediately returns `(metadata.name, metadata.symbol)`. There is **no check** that `metadata.mint == self.mint.key()`. The `MplMetadata` struct deserialized at line 85 carries a `mint` field that identifies which Solana mint the metadata was created for, but it is never read.

The resulting `LogMetadataPayload` is built with `token: self.mint.key()` (the attacker's mint) but `name`/`symbol` harvested from the victim's metadata account: [3](#0-2) 

This payload is serialized and posted via Wormhole to NEAR, where `deploy_token_callback` uses it verbatim to deploy a NEAR token keyed to `M_attacker` but carrying `M_victim`'s name and symbol: [4](#0-3) 

Note that the classic SPL token path (line 117-127) is **not** vulnerable because it derives the Metaplex PDA deterministically from `self.mint.key()`, making it impossible to supply a different mint's PDA: [5](#0-4) 

The vulnerability is exclusive to the Token-2022 third-party metadata path.

---

### Impact Explanation

- NEAR registers `M_attacker` as a Solana-origin token with `M_victim`'s name and symbol.
- Users and front-ends that identify tokens by name/symbol will see `M_attacker` as the legitimate token.
- The attacker controls `M_attacker`'s mint authority (the `!mint.mint_authority.contains(authority.key)` constraint at line 42 only blocks bridge-owned mints, not attacker-owned ones), so they can mint arbitrary supply and bridge it to NEAR under the victim's identity.
- When `M_victim`'s legitimate owner later calls `log_metadata`, NEAR creates a *second* token with the same name/symbol but a different address, permanently fragmenting the token's cross-chain identity and breaking collateralization accounting.

This maps to **High — token-mapping corruption that misdirects value** (not the Critical freeze framing in the question, but a concrete, bounty-grade finding).

---

### Likelihood Explanation

- Fully permissionless: Token-2022 mint creation and `metadata_pointer` configuration require no privileges.
- `M_victim`'s Metaplex metadata account is a public, on-chain PDA — no secret knowledge needed.
- The attacker only needs to pay Solana rent and the Wormhole message fee.

---

### Recommendation

Add a `mint` field check immediately after deserializing the Metaplex metadata in `parse_metadata_account`:

```rust
if metadata.owner == &MetaplexID {
    let data = metadata.try_borrow_data()?;
    let mpl = MplMetadata::try_deserialize(&mut data.as_ref())?;
    require_keys_eq!(
        mpl.mint,
        self.mint.key(),
        ErrorCode::InvalidTokenMetadataAddress,
    );
    Ok((mpl.name.clone(), mpl.symbol.clone()))
}
``` [6](#0-5) 

---

### Proof of Concept

1. `M_victim` is an existing SPL/Token-2022 mint with a Metaplex metadata account at PDA `V_meta = find_program_address([METADATA_SEED, MetaplexID, M_victim], MetaplexID)`.
2. Attacker creates Token-2022 mint `M_attacker` with `metadata_pointer = V_meta`.
3. Attacker calls `log_metadata { mint: M_attacker, metadata: Some(V_meta), … }`.
4. `process()` reads `metadata_pointer.metadata_address.0 = V_meta ≠ M_attacker` → enters the "Third-party metadata" branch.
5. `parse_metadata_account(V_meta)`: key check passes (`V_meta == V_meta`), owner check passes (`MetaplexID`), returns `M_victim`'s name/symbol.
6. `LogMetadataPayload { token: M_attacker, name: "VictimToken", symbol: "VTK", decimals: … }` is posted to Wormhole.
7. NEAR's `deploy_token_callback` deploys a NEAR token for `M_attacker` named "VictimToken"/"VTK".
8. Attacker mints tokens on `M_attacker` and bridges them to NEAR, appearing to be the legitimate "VictimToken".
