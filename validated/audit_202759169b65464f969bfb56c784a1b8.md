### Title
Fee-on-Transfer Token Accounting Discrepancy Breaks Bridge Collateralization — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
`OmniBridge.initTransfer` uses the caller-supplied `amount` parameter directly in the cross-chain message without measuring the actual tokens received by the contract. For fee-on-transfer ERC20 tokens, the bridge locks fewer tokens than it records, causing the NEAR side to mint more bridged tokens than the EVM vault holds, permanently breaking collateralization.

### Finding Description
In `OmniBridge.initTransfer`, when the token is a plain ERC20 (not a bridge token and not a custom minter), the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
``` [1](#0-0) 

Immediately after, the same unmodified `amount` is forwarded to `initTransferExtension` and emitted in the `InitTransfer` event:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message
);
``` [2](#0-1) 

In `OmniBridgeWormhole`, `initTransferExtension` publishes this same `amount` into the Wormhole cross-chain payload:

```solidity
Borsh.encodeUint128(amount),
``` [3](#0-2) 

There is no balance-before/balance-after check. For a fee-on-transfer token (e.g., a token that deducts 1–5% on every transfer), `safeTransferFrom` succeeds but the bridge receives `amount - transfer_fee` while the cross-chain message records `amount`. The NEAR bridge then mints the full `amount` to the recipient.

The NEAR side's `fin_transfer_callback` trusts the proof-verified amount from the EVM event and mints accordingly: [4](#0-3) 

No check exists anywhere in the pipeline to reconcile the actual EVM vault balance against the recorded cross-chain amount.

### Impact Explanation
Every `initTransfer` call with a fee-on-transfer token creates a permanent deficit: the EVM vault holds `amount - fee_on_transfer` but the NEAR-side supply grows by `amount`. Over repeated transfers the vault becomes progressively undercollateralized. When users later bridge back (burning NEAR-side tokens and releasing EVM-side tokens), the vault will eventually be unable to satisfy `finTransfer` payouts — either reverting and permanently locking user funds, or draining tokens belonging to other users. This matches the allowed impact: **balance/accounting corruption that breaks bridge collateralization**.

### Likelihood Explanation
Fee-on-transfer tokens are a well-known ERC20 extension (USDT on some chains, SafeMoon-style tokens, tokens with built-in tax mechanisms). Any unprivileged user can call `initTransfer` with such a token — no special role or permission is required. The bridge has no allowlist or compatibility check for token behavior, so the vulnerability is reachable the moment any such token is used with the bridge.

### Recommendation
Measure the actual received amount using a balance-before/balance-after pattern for the non-bridge-token path:

```solidity
uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
uint256 actualReceived = IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore;
require(actualReceived == amount, "Fee-on-transfer tokens not supported");
```

Alternatively, publish `actualReceived` instead of `amount` in the cross-chain message so the NEAR side mints only what the vault actually holds. Additionally, consider publishing a Token Integration Checklist (analogous to the external report's recommendation) that explicitly flags fee-on-transfer, rebasing, and non-standard ERC20 tokens as incompatible with the bridge's accounting model.

### Proof of Concept

1. Deploy or use an existing fee-on-transfer ERC20 token `FOT` that deducts 5% on every `transferFrom`.
2. Approve `OmniBridge` to spend 1000 `FOT`.
3. Call `initTransfer(FOT, 1000, 0, 0, "near:alice.near", "")`.
   - `safeTransferFrom` moves 1000 `FOT` from the caller; the bridge receives 950 `FOT` (5% fee taken by the token).
   - The `InitTransfer` event records `amount = 1000`.
4. The Wormhole/NEAR relayer picks up the event and the NEAR bridge mints 1000 wrapped `FOT` to `alice.near`.
5. The EVM vault now holds 950 `FOT` but the NEAR supply is 1000 — a 50-token deficit.
6. Repeat N times. After sufficient iterations, the vault cannot cover a `finTransfer` for a user bridging back 1000 tokens, causing the `safeTransfer` to revert and permanently locking those user funds. [5](#0-4)

        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();
        }

        uint256 extensionValue;
        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;

            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount
                );

                    amount
                );

                    amount
                );
