### Title
Missing `shard_id` Binding in `set_state_header` Allows Cross-Shard State Root Substitution — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`ChainStateSyncAdapter::set_state_header` accepts a `ShardStateSyncResponseHeader` for a requested `shard_id` but never verifies that the embedded chunk's `shard_id()` equals the requested `shard_id`. A malicious peer can supply a structurally valid header for shard N while the syncing node requested shard M, causing the wrong shard's `prev_state_root` to be stored under `StateHeaderKey(M, sync_hash)` and subsequently used to validate and install state parts.

---

### Finding Description

**Entrypoint:** `ChainStateSyncAdapter::set_state_header` in `chain/chain/src/state_sync/adapter.rs`.

The function performs five validation steps:

1. **`validate_chunk_proofs`** — verifies the chunk's internal hash consistency (encoded_merkle_root, tx_root, receipts_root). It checks that `chunk.header.hash == compute_header_hash(inner)`, which commits to `inner.shard_id`, but it does **not** compare `inner.shard_id` against the caller-supplied `shard_id` argument. [1](#0-0) 

2. **Merkle path check** — verifies `verify_path(chunk_headers_root, chunk_proof, ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))`. This proves the chunk's hash is *somewhere* in the block's Merkle tree of all chunks. It does **not** verify the leaf's position (shard index). A valid Merkle path for shard 1's chunk at tree-position 1 passes this check even when `shard_id = 0` is requested. [2](#0-1) 

3. **Receipts proofs** — validated using the caller-supplied `shard_id` (step 4e: `CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))`), not the chunk's embedded shard_id. An attacker can supply valid receipts for the requested shard M while the chunk belongs to shard N. [3](#0-2) 

4. **`validate_state_root_node`** — validates the `state_root_node` against `chunk_inner.prev_state_root()`. Since the chunk is shard N's chunk, this validates shard N's state root node against shard N's state root — which passes trivially. [4](#0-3) 

5. **Storage** — the header is stored under `StateHeaderKey(shard_id, sync_hash)` where `shard_id` is the *requested* shard M, not the chunk's actual shard N. [5](#0-4) 

**The missing check:** nowhere in `set_state_header` is there an assertion equivalent to:
```rust
if chunk.shard_id() != shard_id { return Err(...); }
```

**Downstream corruption in `set_state_part`:** the stored header is retrieved and shard N's `prev_state_root` is extracted and used to validate state parts for shard M: [6](#0-5) 

**Downstream corruption in `set_state_finalize`:** the wrong chunk is applied via `apply_chunk`, using shard N's `prev_state_root` and transactions for shard M's `shard_uid`, producing a divergent `ChunkExtra.state_root` for shard M. [7](#0-6) 

**How `chunk_headers_root` is constructed:** it is a Merkle tree over `ChunkHashHeight(chunk_hash, height_included)` for all shards in order. The leaf value does not include `shard_id` explicitly — `shard_id` is only implicit in the chunk hash (via the inner fields). `verify_path` checks inclusion in the tree but not the leaf's position. [8](#0-7) 

---

### Impact Explanation

The exact corrupted value is `prev_state_root` extracted from the wrong shard's chunk header, stored under `StateHeaderKey(requested_shard_id, sync_hash)`. All subsequent `set_state_part` calls for `requested_shard_id` validate trie parts against the wrong state root. `set_state_finalize` applies the wrong chunk and installs the wrong trie, producing a divergent `ChunkExtra.state_root` for the requested shard. The syncing node diverges from the canonical chain for that shard.

---

### Likelihood Explanation

State sync data is downloaded from peers. An unprivileged peer can serve a `ShardStateSyncResponseHeader`. All data needed to construct the malicious header (a real chunk for shard N, its Merkle path, valid receipts proofs for shard M, and shard N's state root node) is publicly observable on the network. No validator or privileged key is required.

---

### Recommendation

Add an explicit shard_id binding check at the start of `set_state_header`, immediately after extracting the chunk:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        format!("set_shard_state failed: chunk shard_id {:?} != requested shard_id {:?}",
                chunk.shard_id(), shard_id)
    ));
}
```

Similarly check `prev_chunk_header.shard_id() == shard_id` when `prev_chunk_header` is `Some`. [9](#0-8) 

---

### Proof of Concept

A cargo test in `chain/chain/src/state_sync/adapter.rs` would:

1. Set up a two-shard chain with `sync_hash` pointing to the first block of a new epoch.
2. Call `compute_state_response_header(shard_id=1, sync_hash)` to obtain a legitimate header for shard 1 (including its chunk, Merkle path, receipts proofs, and state_root_node).
3. Call `set_state_header(shard_id=0, sync_hash, header_for_shard_1)`.
4. Assert the call returns an error indicating shard_id mismatch.

Without the fix, step 4 returns `Ok(())` and the header for shard 1 is stored under `StateHeaderKey(0, sync_hash)`, demonstrating the missing guard.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
