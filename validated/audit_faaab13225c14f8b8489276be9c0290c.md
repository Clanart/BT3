### Title
Unchecked `mint` Account Association in `finalize_transfer` Allows Wrong Token Release and Vault Drain — (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`)

### Summary

The `FinalizeTransfer` Solana instruction accepts a caller-supplied `mint` account that is not constrained to match the token address committed to in the MPC-signed `FinalizeTransferPayload`. An attacker who holds a valid signed payload for token A can substitute token B's mint, causing the bridge to release token B from its vault instead of token A, consuming the nonce and permanently corrupting cross-chain accounting.

### Finding Description

In `FinalizeTransfer`, the `mint` account is declared with only a single constraint:

```rust
#[account(
    mut,
    mint::token_program = token_program,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
``` [1](#0-0) 

The `vault` PDA is derived directly from the caller-supplied `mint`:

```rust
seeds = [VAULT_SEED, mint.key().as_ref()],
``` [2](#0-1) 

The `process` function uses only `data.destination_nonce`, `data.amount`, `data.fee_recipient`, and `data.transfer_id` from the signed payload — no field from the payload is compared against `self.mint.key()`: [3](#0-2) 

Critically, the `FinalizeTransferResponse` emitted back to NEAR via Wormhole uses the caller-supplied mint as the canonical token:

```rust
let payload = FinalizeTransferResponse {
    token: self.mint.key(),   // ← attacker-controlled
    amount: data.amount,
    ...
}
``` [4](#0-3) 

On the NEAR side, `sign_transfer` explicitly includes `token_address` (the Solana mint) in the `TransferMessagePayload` that is MPC-signed:

```rust
let transfer_payload = TransferMessagePayload {
    ...
    token_address,   // ← Solana mint committed in signature
    amount: U128(amount_to_transfer),
    recipient: transfer_message.recipient,
    ...
};
``` [5](#0-4) 

However, the Solana instruction never enforces that the `mint` account passed by the caller equals the `token_address` field from the signed payload. The signature verification only proves the payload bytes are authentic; it does not bind the `mint` account to those bytes because no Anchor constraint performs that check.

### Impact Explanation

**Critical — Direct theft of bridged assets and permanent accounting corruption.**

An attacker who is the legitimate recipient of a signed transfer of token A (e.g., USDC) can:

1. Obtain the valid MPC-signed `FinalizeTransferPayload` for their transfer.
2. Call `finalize_transfer` substituting token B's mint (e.g., WBTC) that also has a bridge vault with locked funds.
3. The vault PDA is derived from the substituted mint, so the bridge releases token B's funds to the recipient.
4. The destination nonce is consumed — the legitimate token A transfer can never be finalized.
5. NEAR receives a Wormhole message claiming token B was released, permanently desynchronizing its locked-token accounting.

Result: attacker receives a more valuable token than entitled, drains token B's vault, and the legitimate token A remains permanently locked on NEAR with no recourse.

For the bridged (mint) path, the check `self.mint.mint_authority.contains(self.authority.key)` limits substitution to bridge-deployed mints, but those are still valid substitution targets that can be minted unboundedly, breaking supply invariants. [6](#0-5) 

### Likelihood Explanation

**High.** Any recipient of a cross-chain transfer to Solana receives a valid signed payload. The substitution requires no privileged access — only knowledge of which other token vaults hold funds (publicly observable on-chain). The attack is a single transaction.

### Recommendation

Add an Anchor constraint that verifies the `mint` account matches the `token_address` field embedded in the signed payload:

```rust
#[account(
    mut,
    mint::token_program = token_program,
    constraint = mint.key() == data.payload.token_address
        @ ErrorCode::WrongMintOnPayload,
)]
pub mint: Box<InterfaceAccount<'info, Mint>>,
```

This mirrors the fix in the reported issue: verify that the provided account matches what is specified in the authoritative signed state (the payload), not just that the account is structurally valid.

### Proof of Concept

1. Alice initiates a transfer of 1,000 USDC from NEAR to Solana. NEAR's MPC signs a `FinalizeTransferPayload` with `destination_nonce=N`, `amount=1000_000000`, `transfer_id=T`, `token_address=USDC_MINT`.
2. Attacker (Alice or a front-runner) observes the signed payload.
3. Attacker calls `finalize_transfer(signed_payload)` but passes `mint=WBTC_MINT` (a different registered token with a funded vault).
4. Anchor derives `vault = PDA([VAULT_SEED, WBTC_MINT])` — this vault exists and holds locked WBTC.
5. `transfer_checked` releases WBTC from the vault to the attacker's WBTC token account.
6. Nonce `N` is marked used in `UsedNonces`.
7. Wormhole message is posted: `FinalizeTransferResponse { token: WBTC_MINT, amount: 1000_000000, transfer_id: T }`.
8. NEAR processes the Wormhole message, records WBTC as finalized for transfer T, and never releases the locked USDC — it is permanently frozen. [7](#0-6) [8](#0-7)
