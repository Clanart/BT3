### Title
Fee-on-Transfer Token Accounting Discrepancy in `initTransfer` Allows Unbacked Minting on NEAR - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` accepts a caller-supplied `amount` and transfers ERC20 tokens via `safeTransferFrom` without verifying the actual balance received. For fee-on-transfer tokens, the contract locks fewer tokens than `amount`, but the emitted `InitTransfer` event (and Wormhole message in `OmniBridgeWormhole`) records the full `amount`. The NEAR bridge processes this event and mints/releases the full `amount` on NEAR, creating unbacked supply and breaking bridge collateralization.

---

### Finding Description

In `OmniBridge.initTransfer`, the ERC20 native token path (lines 407–411) performs:

```solidity
IERC20(tokenAddress).safeTransferFrom(
    msg.sender,
    address(this),
    amount
);
``` [1](#0-0) 

No balance snapshot is taken before or after this call. The function then unconditionally emits `InitTransfer` with the caller-supplied `amount`:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // <-- unverified; actual received may be less
    fee,
    nativeFee,
    recipient,
    message
);
``` [2](#0-1) 

In `OmniBridgeWormhole`, `initTransferExtension` publishes a Wormhole message encoding the same unverified `amount`:

```solidity
Borsh.encodeUint128(amount),
``` [3](#0-2) [4](#0-3) 

The NEAR bridge (`near/omni-bridge/src/lib.rs`) processes the proof/event from the EVM side and mints or releases the full `amount` to the recipient on NEAR, trusting the emitted value without any independent verification of what was actually deposited into the EVM vault. [5](#0-4) 

---

### Impact Explanation

For a fee-on-transfer token with a 10% fee:
- User calls `initTransfer(token, 1000, ...)`.
- EVM bridge receives only 900 tokens (fee deducted by token contract).
- Event/Wormhole message records `amount = 1000`.
- NEAR bridge mints 1000 tokens to the recipient.
- The EVM vault is now undercollateralized by 100 tokens per transfer.
- When NEAR-side holders bridge back, the EVM vault cannot fulfill all withdrawals — funds are permanently locked for the last withdrawers.

This matches the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value"** and **"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds."**

---

### Likelihood Explanation

Any unprivileged user can call `initTransfer` with any ERC20 token address. Fee-on-transfer tokens (e.g., USDT on some chains, deflationary tokens) are widely deployed. No special role or access is required. The attacker only needs to hold such a token and call the public function. [6](#0-5) 

---

### Recommendation

Record the vault balance before and after the `safeTransferFrom` call and use the actual received amount for the event and Wormhole message:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 balanceAfter = IERC20(tokenAddress).balanceOf(address(this));
    uint128 actualReceived = uint128(balanceAfter - balanceBefore);
    require(actualReceived == amount, "Fee-on-transfer tokens not supported");
    // OR: use actualReceived as the canonical amount going forward
}
```

This mirrors the fix recommended in the external report for `L1NativeTokenVault._bridgeBurnNativeToken`. [7](#0-6) 

---

### Proof of Concept

1. Deploy a fee-on-transfer ERC20 token (10% fee on every `transferFrom`).
2. Call `OmniBridge.initTransfer(feeToken, 1000e18, 0, 0, "near:recipient.near", "")`.
3. The bridge receives `900e18` tokens but emits `InitTransfer` with `amount = 1000e18`.
4. The NEAR bridge processes the event and mints `1000e18` tokens to `recipient.near`.
5. Repeat: after 10 such transfers, the EVM vault holds `9000e18` but NEAR has `10000e18` in circulation.
6. When NEAR holders attempt to bridge back, the last `1000e18` worth of withdrawals cannot be fulfilled — funds are permanently locked. [8](#0-7) [4](#0-3)
