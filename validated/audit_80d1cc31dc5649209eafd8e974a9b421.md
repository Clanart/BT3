### Title
Integer Overflow in Relative Timelock Validation Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` When `nowrap=false` — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` exposes a `nowrap: bool` parameter. When `nowrap=false`, relative timelock comparisons use Rust's `wrapping_add` instead of `saturating_add`. An unprivileged spender can craft a coin whose puzzle emits `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` with a near-maximal integer value. The wrapping addition produces a result smaller than the coin's birth height/timestamp, making the guard comparison trivially false and silently accepting the spend at any block height or timestamp — a complete timelock bypass.

---

### Finding Description

In `check_time_locks`, the `nowrap=false` branch for `height_relative` is:

```rust
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
``` [1](#0-0) 

And for `seconds_relative`:

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [2](#0-1) 

**Concrete overflow example (height):** coin confirmed at height `10`, `height_relative = 0xFFFF_FFF6` (`u32`).

```
10u32.wrapping_add(0xFFFF_FFF6) = 0
```

The guard becomes `prev_height < 0`, which is always `false` for any unsigned `prev_height`. The error branch is never taken; the spend is accepted unconditionally.

The project's own test suite documents this behaviour explicitly:

```python
# 10 + (2^32 - 10) overflows to 0, wrapping: 15 < 0 -> Ok
# saturating: clamps to u32::MAX, 15 < u32::MAX -> Err
(make_test_conds(height_relative=0xFFFF_FFF6), 13, None),
# 10 + u32::MAX overflows to 9, wrapping: 15 < 9 -> Ok
(make_test_conds(height_relative=0xFFFF_FFFF), 13, None),
``` [3](#0-2) 

The same overflow applies to `seconds_relative` (`u64`):

```python
# 10000 + u64::MAX overflows to 9999, wrapping: 10150 < 9999 -> Ok
(make_test_conds(seconds_relative=0xFFFF_FFFF_FFFF_FFFF), 105, None),
``` [4](#0-3) 

The condition parser accepts any `u32` value for `ASSERT_HEIGHT_RELATIVE` (4-byte sanitized uint), so `0xFFFF_FFF6` through `0xFFFF_FFFF` are all valid on-chain condition arguments: [5](#0-4) 

`check_time_locks` is exported as a Python binding (`py_check_time_locks`) and is the function the full node calls to enforce timelocks on spend bundles: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

A coin whose puzzle outputs `(ASSERT_HEIGHT_RELATIVE 0xFFFF_FFF6)` (or any value ≥ `2^32 − confirmed_block_index`) is spendable at **any** block height when the caller passes `nowrap=false`. The intended lock — "do not spend until at least N blocks after birth" — is completely nullified. This is a **timelock validation bypass enabling unauthorized spend acceptance**, matching the High impact tier: *"timelock… validation bypass enables unauthorized spend acceptance or replay."*

---

### Likelihood Explanation

- The attacker controls the puzzle/solution of their own coin and can freely choose any `u32` value for `height_relative` or `u64` value for `seconds_relative`.
- The condition is parsed without upper-bound rejection (values up to `u32::MAX` / `u64::MAX` are accepted as `SanitizedUint::Ok`).
- `check_time_locks` is the sole timelock enforcement point exposed to Python; if the full node invokes it with `nowrap=false` for any block range (e.g., pre-fork backward-compatibility), every coin in that range with a crafted relative timelock is immediately spendable.
- No privileged role, leaked key, or network-level attack is required — only a standard coin spend.

---

### Recommendation

Replace `wrapping_add` with `saturating_add` unconditionally in all four relative-timelock branches, or remove the `nowrap=false` code path entirely if it is no longer needed for consensus. The `nowrap=true` (saturating) path already produces the correct, safe semantics: an overflow clamps to `u32::MAX` / `u64::MAX`, making the guard always trigger as expected.

```rust
// Safe replacement for height_relative:
if prev_transaction_block_height
    < unspent.confirmed_block_index.saturating_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

---

### Proof of Concept

1. Craft a puzzle that emits `(ASSERT_HEIGHT_RELATIVE 4294967286)` (`0xFFFF_FFF6`).
2. Fund a coin with this puzzle; it is confirmed at block height `10`.
3. Immediately (block height `11`) submit a spend bundle for this coin.
4. The full node calls `check_time_locks(..., nowrap=false)`.
5. Inside the function: `10u32.wrapping_add(4294967286) = 0`; guard is `11 < 0` → `false` → no error returned.
6. The spend is accepted despite the coin being only 1 block old, bypassing the intended ~4 billion-block timelock.

The same attack applies to `ASSERT_SECONDS_RELATIVE` with `seconds_relative = u64::MAX - coin_timestamp + 1`, causing the wrapped sum to underflow below the current timestamp. [8](#0-7)
