### Title
Write-Only Storage Cells Absent from Prover `initial_reads` Cause Wrong Patricia State Root in `StateCommitmentInfos` - (File: `crates/starknet_transaction_prover/src/running/virtual_block_executor.rs`)

---

### Summary

In `VirtualBlockExecutor::execute`, `get_initial_reads()` is called **before** `to_state_diff()`. The `to_state_diff()` call internally triggers `update_initial_values_of_write_only_access()`, which back-fills write-only storage cells into the cache's `initial_reads`. Because `get_initial_reads()` already cloned the cache before this mutation, the returned `initial_reads` is permanently missing every storage key whose first access was a write (not a read). Downstream, `insert_storage_leaves` only inserts Patricia leaf nodes for keys present in `initial_reads.storage`, so write-only keys are absent from the `FactsDb`. When `commit_state_diff` later updates those keys, the committer finds no pre-existing leaf and silently treats the initial value as `Felt::ZERO`. If the actual on-chain initial value is non-zero, the computed `updated_root` (new storage-trie root) is wrong, and the `StateCommitmentInfos` fed to the virtual OS carry a corrupted post-execution state root.

---

### Finding Description

**Step 1 â€“ Ordering bug in `VirtualBlockExecutor::execute`** [1](#0-0) 

`get_initial_reads()` is called at line 294 and returns a snapshot of `cache.initial_reads` at that moment. `to_state_diff()` is called at line 299 and, as its first action, calls `update_initial_values_of_write_only_access()`: [2](#0-1) 

This mutates `cache.initial_reads` in place, inserting the actual on-chain value for every storage key that was written but never read. The already-cloned `initial_reads` variable is unaffected.

**Step 2 â€“ Missing leaves in `insert_storage_leaves`**

`get_storage_proofs` builds `extended_initial_reads.storage` directly from `execution_data.initial_reads.storage`: [3](#0-2) 

`create_facts_db_from_storage_proof` then calls `insert_storage_leaves` with this incomplete map: [4](#0-3) 

`insert_storage_leaves` iterates only over `initial_reads.storage`: [5](#0-4) 

Write-only keys are absent; no Patricia leaf is inserted for them.

**Step 3 â€“ Committer uses `Felt::ZERO` for missing leaves**

`commit_state_diff` applies `state_diff` (which correctly contains write-only keys, because `to_state_diff()` computed the diff after back-filling) to the `FactsDb`: [6](#0-5) 

For any write-only key whose initial leaf is absent from the DB, the Patricia committer defaults to `Felt::ZERO`. If the actual initial value `V â‰  0`, the committer computes the wrong post-update leaf hash, propagates it up the trie, and returns a wrong `updated_root`.

**Step 4 â€“ Wrong root reaches the OS**

The corrupted `StateCommitmentInfos` (with wrong `updated_root` for one or more storage tries) is assembled into `VirtualOsBlockInput.address_to_storage_commitment_info` and passed to the virtual OS: [7](#0-6) 

The OS uses these commitment infos to verify and output the post-execution global state root, so the final proof attests to a wrong state transition.

---

### Impact Explanation

The virtual OS receives `StateCommitmentInfos` whose `updated_root` fields do not match the actual post-execution Patricia root for any contract whose storage was written without a prior read. The OS outputs a wrong `final_root` (new global state root). Any proof generated from this run is either invalid (if the OS internally cross-checks the root) or attests to a wrong state transition (if it trusts the supplied commitment infos). Either outcome breaks the correctness invariant of the transaction prover: the proof facts embedded in the on-chain Invoke V3 transaction bind to a block hash that commits to a wrong state root, corrupting the authoritative proof output for that transaction.

---

### Likelihood Explanation

Write-only storage access is a normal Cairo pattern: any contract that unconditionally overwrites a storage variable (e.g., a setter that does not read before writing) triggers this path. The condition `initial_value â‰  0` is satisfied whenever the slot was previously set. No special privilege is required; any user submitting an Invoke V3 transaction to the prover service can trigger the bug.

---

### Recommendation

In `VirtualBlockExecutor::execute`, call `get_initial_reads()` **after** `to_state_diff()` so that write-only cells have already been back-filled:

```rust
// Get state diff first (this back-fills write-only cells into cache.initial_reads).
let state_diff = block_state.to_state_diff()?.state_maps;

// Now initial_reads includes write-only cells.
let initial_reads = block_state.get_initial_reads()?;
```

Alternatively, replace `get_initial_reads()` with `get_os_initial_reads()`, which explicitly calls `update_initial_values_of_write_only_access()` before cloning: [8](#0-7) 

---

### Proof of Concept

1. Deploy a contract with a storage variable `counter` initialized to `1` (non-zero).
2. Submit an Invoke V3 transaction to the prover that calls a function which writes `counter = 2` **without** reading `counter` first (pure write, no `storage_read` syscall).
3. `VirtualBlockExecutor::execute` returns `initial_reads.storage` that does **not** contain `(contract_address, counter_key)`, but `state_diff.storage` does contain `(contract_address, counter_key) â†’ 2`.
4. `insert_storage_leaves` inserts no leaf for `counter_key`; the `FactsDb` has no entry for it.
5. `commit_state_diff` treats the initial value of `counter_key` as `0` instead of `1`, computes the wrong leaf hash `H(2)` rooted from `0` instead of from `1`, and returns a wrong `updated_root`.
6. The OS receives `StateCommitmentInfos` with the wrong `updated_root` for that contract's storage trie, producing a proof that attests to a wrong post-execution global state root.

    fn update_initial_values_of_write_only_access(&mut self) -> StateResult<()> {
        let cache = &mut *self.cache.borrow_mut();

        // Eliminate storage writes that are identical to the initial value (no change).
        for contract_storage_key in cache.writes.storage.keys() {
            if !cache.initial_reads.storage.contains_key(contract_storage_key) {
                // First access to this cell was write; cache initial value.
                cache.initial_reads.storage.insert(
                    *contract_storage_key,
                    self.state.get_storage_at(contract_storage_key.0, contract_storage_key.1)?,
                );
            }
        }
        Ok(())
    }

/// Inserts storage leaves derived from execution initial reads.
fn insert_storage_leaves(
    db_map: &mut DbHashMap,
    initial_reads: &StateMaps,
) -> Result<(), ProofProviderError> {
    for ((address, _storagekey), value) in &initial_reads.storage {
        let storage_value = StarknetStorageValue(*value);
        let leaf_hash = TreeHashFunctionImpl::compute_leaf_hash(&storage_value);
        insert_leaf_node(db_map, leaf_hash, storage_value, address)?;
    }

    Ok(())
}

/// Commits the state diff, populates the DB with the new facts and returns the new state roots.
pub async fn commit_state_diff(
    facts_db: &mut FactsDb<MapStorage>,
    contracts_trie_root_hash: HashOutput,
    classes_trie_root_hash: HashOutput,
    state_diff: StateDiff,
) -> Result<StateRoots, ProofProviderError> {
    let warn_on_trivial_modifications = false;
    let build_storage_tries_concurrently = false; // Map storage is used, no need for concurrency.
    let config = ReaderConfig::new(warn_on_trivial_modifications, build_storage_tries_concurrently);
    let initial_read_context =
        FactsDbInitialRead(StateRoots { contracts_trie_root_hash, classes_trie_root_hash });
    let input = Input { state_diff, initial_read_context, config };

    let (filled_forest, _deleted_nodes) = commit_block(input, facts_db, &mut NoMeasurements)
        .await
        .map_err(|e| ProofProviderError::BlockCommitmentError(e.to_string()))?;
    facts_db.write(&filled_forest).await?;

    Ok(StateRoots {
        contracts_trie_root_hash: filled_forest.get_contract_root_hash(),
        classes_trie_root_hash: filled_forest.get_compiled_class_root_hash(),
    })
}
