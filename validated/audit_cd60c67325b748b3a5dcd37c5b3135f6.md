### Title
Excess ETH Permanently Locked When Bridging ERC20/ERC1155 Tokens via `initTransfer` and `initTransfer1155` — (File: evm/src/omni-bridge/contracts/OmniBridge.sol)

---

### Summary

`OmniBridge.initTransfer` and `OmniBridge.initTransfer1155` are `payable` and branch on whether the token is native ETH (`tokenAddress == address(0)`) or an ERC20/ERC1155. In the ERC20/ERC1155 branch, the code computes `extensionValue = msg.value - nativeFee` but never validates that `msg.value == nativeFee`. Any ETH sent beyond `nativeFee` is silently accepted and passed to `initTransferExtension`, which in the base `OmniBridge` contract is a no-op virtual function. The excess ETH is permanently locked with no recovery path.

---

### Finding Description

In `OmniBridge.initTransfer`, the ERC20 branch is:

```solidity
} else {
    extensionValue = msg.value - nativeFee;   // no upper-bound check
    if (customMinters[tokenAddress] != address(0)) {
        IERC20(tokenAddress).safeTransferFrom(...);
        ICustomMinter(...).burn(...);
    } else if (isBridgeToken[tokenAddress]) {
        BridgeToken(tokenAddress).burn(msg.sender, amount);
    } else {
        IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    }
}
``` [1](#0-0) 

`extensionValue` is then forwarded to `initTransferExtension`:

```solidity
initTransferExtension(
    msg.sender, tokenAddress, currentOriginNonce,
    amount, fee, nativeFee, recipient, message, extensionValue
);
``` [2](#0-1) 

In the base `OmniBridge` contract, `initTransferExtension` is declared as an empty virtual no-op:

```solidity
function initTransferExtension(...) internal virtual {}
``` [3](#0-2) 

The identical pattern exists in `initTransfer1155`:

```solidity
uint256 extensionValue = msg.value - nativeFee;
``` [4](#0-3) 

Because the base contract's `initTransferExtension` is a no-op, any ETH in excess of `nativeFee` that a user attaches to an ERC20 bridge call is silently consumed by the contract and permanently locked — there is no sweep, refund, or withdrawal function for it.

The `InvalidValue` error is defined in the contract:

```solidity
error InvalidValue();
``` [5](#0-4) 

but it is never thrown in the ERC20 branch of `initTransfer` or `initTransfer1155` in the base contract. The `OmniBridgeWormhole` extension overrides `initTransferExtension` and does enforce this check (as confirmed by the test expecting `InvalidValue` when `value: 200 > nativeFee: 100`), but the base `OmniBridge` contract — which is concrete and deployable — does not.

---

### Impact Explanation

Any ETH sent by a user in excess of `nativeFee` when calling `initTransfer` or `initTransfer1155` for an ERC20/ERC1155 token on the base `OmniBridge` contract is permanently locked. There is no admin withdrawal, sweep, or refund mechanism for stranded ETH. This constitutes an irrecoverable lock of user funds in the bridge contract, matching the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.**

---

### Likelihood Explanation

The `initTransfer` function is the primary user-facing entry point for bridging tokens. Users bridging ERC20 tokens are required to send `nativeFee` in ETH as a relayer incentive. A user who mistakenly sends `msg.value > nativeFee` (e.g., due to a frontend bug, copy-paste error, or confusion between the `amount` and `nativeFee` parameters) will have their excess ETH permanently locked. The base `OmniBridge` contract is concrete (not abstract) and deployable independently of `OmniBridgeWormhole`, making this reachable by any unprivileged bridge user.

---

### Recommendation

Add an explicit upper-bound check in the ERC20/ERC1155 branch of both `initTransfer` and `initTransfer1155` to revert if `msg.value > nativeFee`:

```solidity
} else {
    if (msg.value != nativeFee) { revert InvalidValue(); }
    extensionValue = 0;
    ...
}
```

Alternatively, enforce this invariant at the top of `initTransferExtension` in the base contract rather than relying solely on overriding subcontracts.

---

### Proof of Concept

1. Deploy the base `OmniBridge` contract (not `OmniBridgeWormhole`).
2. Register a standard ERC20 token (not a bridge token, not a custom minter).
3. Approve the bridge to spend `amount` of the ERC20.
4. Call `initTransfer(tokenAddress, amount, 0, nativeFee, recipient, "")` with `msg.value = nativeFee + 1 ether`.
5. The ERC20 `safeTransferFrom` succeeds, `extensionValue = 1 ether` is computed, `initTransferExtension` is called as a no-op, and the transaction completes successfully.
6. The `1 ether` excess is now held by the `OmniBridge` contract with no mechanism to recover it — permanently locked. [6](#0-5)

        currentOriginNonce += 1;
        if (fee >= amount) {
            revert InvalidFee();

        if (tokenAddress == address(0)) {
            if (fee != 0) {
                revert InvalidFee();
            }
            extensionValue = msg.value - amount - nativeFee;

            extensionValue = msg.value - nativeFee;
            if (customMinters[tokenAddress] != address(0)) {
                IERC20(tokenAddress).safeTransferFrom(
                    msg.sender,
                    customMinters[tokenAddress],
                    amount

                );

                    amount
                );
