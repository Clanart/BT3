### Title
Integer Overflow in Relative Timelock Comparison Bypasses `ASSERT_HEIGHT_RELATIVE` / `ASSERT_SECONDS_RELATIVE` — (`crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` contains two arithmetic paths for relative timelock enforcement, selected by the caller-controlled `nowrap: bool` parameter. When `nowrap = false`, the function uses `wrapping_add` to compute the deadline for `ASSERT_HEIGHT_RELATIVE` and `ASSERT_SECONDS_RELATIVE`. For attacker-chosen condition values near the type maximum, the addition wraps to a small integer, making the "not yet spendable" guard trivially false and allowing the coin to be spent immediately — regardless of how large the declared timelock is.

---

### Finding Description

`check_time_locks` enforces relative timelocks by computing `confirmed_block_index + height_relative` (u32) or `coin_timestamp + seconds_relative` (u64) and checking whether the current block height / timestamp has reached that deadline.

When `nowrap = false` the additions use `wrapping_add`:

```rust
// crates/chia-consensus/src/check_time_locks.rs  lines 64-68
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
```

```rust
// lines 75-77
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
```

If `confirmed_block_index = 10` and `height_relative = 0xFFFF_FFF6` (u32), then:

```
10u32.wrapping_add(0xFFFF_FFF6) = 0   // wraps to 0
prev_height (e.g. 15) < 0  →  false  →  no error  →  spend accepted
```

The coin is accepted at height 15 even though the puzzle demanded waiting ~4 billion more blocks. The same arithmetic applies to `seconds_relative` with u64 overflow.

The test suite explicitly documents and confirms this divergence:

```rust
// lines 276-281
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),   // <-- bypass when nowrap=false
)]
```

The Python integration tests confirm the same for concrete overflow thresholds:

```python
# tests/test_check_time_locks.py  lines 239-251
# 10 + (2^32 - 10) overflows to 0, wrapping: 15 < 0 -> Ok
(make_test_conds(height_relative=0xFFFF_FFF6), 13, None),
# 10 + u32::MAX overflows to 9, wrapping: 15 < 9 -> Ok
(make_test_conds(height_relative=0xFFFF_FFFF), 13, None),
# 10000 + (u64::MAX - 9999) overflows to 0, wrapping: 10150 < 0 -> Ok
(make_test_conds(seconds_relative=0xFFFF_FFFF_FFFF_D8F0), 105, None),
```

The function is exported as a Python binding and is part of the consensus-critical spend-bundle validation path:

```python
# wheel/python/chia_rs/chia_rs.pyi  lines 40-46
def check_time_locks(
    removal_coin_records: dict[bytes32, CoinRecord],
    bundle_conds: SpendBundleConditions,
    prev_transaction_block_height: uint32,
    timestamp: uint64,
    nowrap: bool,
) -> Optional[int]: ...
```

The `nowrap` flag is caller-supplied; if the chia-blockchain passes `nowrap=False` for any block range (e.g., for backward-compatibility with pre-fork blocks), every coin in that range whose puzzle contains a crafted `ASSERT_HEIGHT_RELATIVE` or `ASSERT_SECONDS_RELATIVE` value is vulnerable.

---

### Impact Explanation

A coin whose puzzle includes `ASSERT_HEIGHT_RELATIVE(N)` where `N` is chosen so that `confirmed_height + N` wraps to a value below the current chain height can be spent immediately, bypassing the intended timelock entirely. This is an unauthorized spend acceptance: the coin's own puzzle declares it unspendable for billions of blocks, yet the consensus layer accepts the spend. This maps directly to the allowed High impact: *"timelock validation bypass enables unauthorized spend acceptance."*

---

### Likelihood Explanation

The attacker controls the CLVM puzzle and therefore controls `height_relative` / `seconds_relative` exactly. No privileged role, leaked key, or network-level attack is required. The only prerequisite is that the chia-blockchain calls `check_time_locks` with `nowrap=False` for the block containing the spend. Given that the parameter exists specifically to preserve legacy behavior, this is a realistic production path.

---

### Recommendation

Replace `wrapping_add` with `saturating_add` unconditionally for both `height_relative` and `seconds_relative` checks, eliminating the `nowrap` branch for those two cases. The `nowrap=true` (saturating) path already implements the correct semantics. If backward compatibility truly requires the wrapping behavior for historical blocks, the range of affected block heights should be bounded and documented, and new blocks should always use `saturating_add`.

---

### Proof of Concept

Craft a coin confirmed at height 10 with puzzle condition `ASSERT_HEIGHT_RELATIVE(0xFFFF_FFF6)`. At chain height 15, call:

```python
check_time_locks(
    {coin.name(): coin_record},   # confirmed_block_index=10
    bundle_conds,                  # height_relative=0xFFFF_FFF6
    prev_transaction_block_height=15,
    timestamp=...,
    nowrap=False,
)
# Returns None (no error) — spend accepted despite timelock demanding height ~4 billion
```
