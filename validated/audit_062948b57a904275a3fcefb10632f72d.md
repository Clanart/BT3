### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Counterfactual-Account Invoke Transactions, Admitting Invalid Signatures to Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the Apollo gateway's stateful transaction validator implements a UX shortcut that is structurally analogous to the ERC-6492 "deploy-then-validate" path. Just as ERC-6492 allowed arbitrary calls before signature validation for counterfactual accounts, this path allows an invoke transaction with an invalid or malicious signature to bypass the `__validate__` entry-point check at the gateway level, provided a deploy-account transaction for the same address is already in the mempool. The result is that the mempool admits an invoke transaction that will be rejected by the blockifier during block execution — satisfying the "High: Mempool/gateway/RPC admission accepts invalid transactions before sequencing" impact.

---

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` returns `true` (meaning: skip `__validate__`) when all three conditions hold:

1. The transaction is an `Invoke`.
2. The transaction's nonce is `Nonce(Felt::ONE)`.
3. The account's on-chain nonce is `Nonce(Felt::ZERO)` (account not yet deployed).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When `skip_stateful_validations` returns `true`, the caller `run_pre_validation_checks` propagates the flag to `run_validate_entry_point`: [2](#0-1) 

Inside `run_validate_entry_point`, the flag is used to set `execution_flags.validate = false`: [3](#0-2) 

With `validate = false`, `AccountTransaction::validate_tx` returns `Ok(None)` immediately without calling the account's `__validate__` entry point: [4](#0-3) 

The `account_tx_in_pool_or_recent_block` check in the mempool returns `true` if **any** transaction from the address is in the pool or was seen in a recent committed block — it does not specifically verify that a `DeployAccount` transaction is present: [5](#0-4) 

For a fresh account (on-chain nonce = 0), the only transaction type that can legitimately enter the pool is a `DeployAccount` (since an `Invoke` for a non-existent account would fail `__validate__` at the gateway). However, once the attacker has submitted a valid `DeployAccount` — which requires only a valid `__validate_deploy__` signature — the mempool check is satisfied, and any subsequent `Invoke` with nonce = 1 will have its `__validate__` skipped entirely at the gateway, regardless of the invoke's signature or calldata.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce/fee consistency against the mempool's internal state; it does not invoke the account's `__validate__` logic: [6](#0-5) 

The `StatefulTransactionValidatorConfig` exposes `max_nonce_for_validation_skip` (default `Nonce(Felt::ONE)`), which bounds the window to nonce = 1 in the default deployment: [7](#0-6) 

---

### Impact Explanation

An attacker who controls a fresh key pair can:

1. Submit a valid `DeployAccount` transaction for their address → it enters the mempool.
2. Immediately submit an `Invoke` with nonce = 1 carrying an **invalid signature** or calldata that would fail the account's custom `__validate__` logic.
3. `skip_stateful_validations` returns `true`; `__validate__` is never called at the gateway.
4. The invoke enters the mempool without signature verification.

When the batcher later pulls both transactions and executes the block:
- The `DeployAccount` succeeds.
- The `Invoke` reaches `validate_tx` with `execution_flags.validate = true` (block-execution default), `__validate__` is called, it fails, and the transaction is rejected.

The corrupted value is the mempool state: it contains an invoke transaction whose signature/calldata would not pass `__validate__`, violating the invariant that all mempool transactions have been signature-verified. This matches the **High** impact: "Mempool/gateway/RPC admission accepts invalid transactions before sequencing."

---

### Likelihood Explanation

The trigger is unprivileged. Any external user can:
- Generate a fresh key pair.
- Compute the deterministic contract address.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
