### Title
Missing Cryptographic Signature Verification for P2P-Synced Block Headers Allows Arbitrary Commitments to be Stored as Authoritative - (`crates/apollo_p2p_sync/src/client/header.rs`)

---

### Summary

The P2P sync client's `HeaderStreamBuilder::parse_data_for_block` checks that a received `SignedBlockHeader` contains exactly one signature but **never cryptographically verifies** that signature against the sequencer's public key. A malicious P2P peer can inject headers carrying arbitrary `block_hash`, `state_diff_commitment`, `transaction_commitment`, `event_commitment`, and `receipt_commitment` values. These are written directly to persistent storage as authoritative, corrupting the node's chain view and all downstream consumers (state diff sync, RPC, retrospective block hash checks in consensus).

---

### Finding Description

**Root cause — `parse_data_for_block` in `HeaderStreamBuilder`** [1](#0-0) 

The function performs two checks on a peer-supplied `SignedBlockHeader`:

1. Block number matches the expected value (line 104–113).
2. The `signatures` vector has exactly `ALLOWED_SIGNATURES_LENGTH` (= 1) entries (line 115–119).

It does **not** call `verify_block_signature`, which is the function that cryptographically validates the sequencer's ECDSA signature over `poseidon_hash(block_hash, state_diff_commitment)`: [2](#0-1) 

After `parse_data_for_block` returns `Ok(Some(signed_block_header))`, `write_to_storage` immediately persists the header and the unverified signature: [3](#0-2) 

There is also an acknowledged missing parent-hash chain check: [4](#0-3) 

**Attacker-controlled data flow**

A peer that the syncing node connects to can respond to a `BlockHeadersRequest` with a `SignedBlockHeader` that contains:
- A chosen `block_hash` (any felt value).
- A chosen `state_diff_commitment` (controls the `state_diff_length` field used by `StateDiffStreamBuilder`).
- Chosen `transaction_commitment`, `event_commitment`, `receipt_commitment`.
- A single `BlockSignature` with arbitrary `(r, s)` bytes — the count check passes, the cryptographic check is absent.

The `SyncBlock` type explicitly documents that it is treated as trusted without verification: [5](#0-4) 

This trust assumption is correct for internally-produced `SyncBlock`s (from consensus), but the P2P path receives data from external, potentially adversarial peers and must not inherit the same trust level.

**Downstream corruption**

Once a poisoned header is stored, `StateDiffStreamBuilder` reads `state_diff_length` from it to determine how many state diff chunks to accept from the peer: [6](#0-5) 

A wrong `state_diff_length` causes the node to accept a state diff of the attacker's chosen size, potentially storing a completely fabricated `ThinStateDiff`. The wrong `block_hash` is then served to other peers via the P2P sync server and used in the consensus orchestrator's retrospective block hash comparison: [7](#0-6) 

The `try_sync` path in the consensus orchestrator also skips `starknet_version` and `parent_hash` validation: [8](#0-7) 

---

### Impact Explanation

A malicious P2P peer can store an arbitrary `state_diff_commitment`, `transaction_commitment`, `event_commitment`, `receipt_commitment`, and `block_hash` in the node's persistent storage. This causes:

- **Wrong state diff stored**: `StateDiffStreamBuilder` uses the attacker-controlled `state_diff_length` to accept a fabricated `ThinStateDiff`, corrupting the Patricia trie root computed by the committer.
- **Wrong RPC responses**: `starknet_getBlockWithTxHashes`, `starknet_getStateUpdate`, and related calls return attacker-chosen commitment values as authoritative.
- **Wrong retrospective block hash**: Consensus uses the stored `block_hash` for the retrospective check; a wrong value causes consensus to stall or accept a wrong chain tip.

This matches the scope impact: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value**, and potentially **Critical — Wrong state, receipt, event, class hash, or storage value from blockifier/syscall/execution logic**.

---

### Likelihood Explanation

Any node that the victim connects to via P2P can trigger this. The libp2p layer authenticates the *connection* (peer identity) but does not authenticate the *content* of sync responses. Peer discovery is permissionless; an attacker can join the network and wait to be selected as a sync peer. No privileged position is required.

---

### Recommendation

In `HeaderStreamBuilder::parse_data_for_block`, after the signature-count check, call `verify_block_signature` with the sequencer's public key (obtained from configuration or from a trusted on-chain source):

```rust
use starknet_api::block::verify_block_signature;

let sig = &signed_block_header.signatures[0];
let state_diff_commitment = signed_block_header
    .block_header
    .state_diff_commitment
    .ok_or(/* error */)?;
verify_block_signature(
    &sequencer_pub_key,
    sig,
    &GlobalRoot(state_diff_commitment.0.0),
    &signed_block_header.block_header.block_hash,
)
.map_err(|_| ParseDataError::BadPeer(BadPeerError::InvalidSignature))?;
```

Additionally, implement the acknowledged parent-hash chain check (line 102–103) to prevent chain-fork injection.

### Proof of Concept

1. Attacker runs a node and connects to the victim as a P2P peer.
2. Victim sends a `BlockHeadersRequest` for block N.
3. Attacker responds with:
   ```
   SignedBlockHeader {
     block_header: BlockHeader {
       block_hash: <attacker_chosen_hash>,
       block_header_without_hash: BlockHeaderWithoutHash {
         block_number: N,
         state_root: <attacker_chosen_root>,
         ...
       },
       state_diff_commitment: Some(<attacker_chosen_commitment>),
       state_diff_length: Some(0),   // or any value
       transaction_commitment: Some(<attacker_chosen>),
       event_commitment: Some(<attacker_chosen>),
       receipt_commitment: Some(<attacker_chosen>),
       ...
     },
     signatures: vec![BlockSignature(Signature { r: Felt::ONE, s: Felt::ONE })],
   }
   ```
4. `parse_data_for_block` passes: block number matches, `signatures.len() == 1`. No cryptographic check is performed.
5. `write_to_storage` persists the poisoned header.
6. `StateDiffStreamBuilder` reads `state_diff_length = 0` from the stored header and accepts an empty state diff for block N, storing a fabricated `ThinStateDiff`.
7. The committer computes a wrong global root from the fabricated state diff.
8. RPC calls for block N return the attacker-chosen `block_hash`, `state_diff_commitment`, and `event_commitment` as authoritative values. [9](#0-8) [6](#0-5) [2](#0-1)

pub fn verify_block_signature(

    signature: &BlockSignature,
    state_diff_commitment: &GlobalRoot,
    block_hash: &BlockHash,

        |err| BlockVerificationError::BlockSignatureVerificationFailed {
            block_hash: *block_hash,
            error: err,
        },
    )
