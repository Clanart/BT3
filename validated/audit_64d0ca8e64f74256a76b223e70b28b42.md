### Title
Cross-Chain Replay of MPC-Signed `MetadataPayload` Enables Unauthorized Token Deployment and Permanent Deployment Blocking - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`, `starknet/src/bridge_types.cairo`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `deployToken` / `deploy_token` signed message hash across EVM, StarkNet, and Solana omits any chain identifier (chain ID, contract address, or `omniBridgeChainId`). A valid MPC signature for a `MetadataPayload` produced for one chain can be replayed verbatim on any other supported chain, deploying a spurious bridge token and permanently blocking the legitimate deployment of that token on the target chain.

---

### Finding Description

The MPC signer on NEAR signs a `MetadataPayload` whose borsh encoding is:

```
[PayloadType::Metadata (1 byte)] | [token (string)] | [name (string)] | [symbol (string)] | [decimals (1 byte)]
```

**EVM** (`OmniBridge.deployToken`): [1](#0-0) 

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
```

No `omniBridgeChainId`, no `address(this)`, no `block.chainid`.

No chain ID, no contract address.

**Solana** (`DeployTokenPayload::serialize_for_near`): [3](#0-2) 


No program ID, no chain ID.

**Contrast with `finTransfer`**, which correctly embeds `omniBridgeChainId` twice in the signed payload: [4](#0-3) 

The `MetadataPayload` encoding is byte-for-byte identical across all three destination chains. The same MPC signature that authorizes deployment on Ethereum will pass `ECDSA.recover` / `verify_eth_signature` / `secp256k1_recover` on StarkNet and Solana respectively, because all three verify against the same `nearBridgeDerivedAddress` / `omni_bridge_derived_address` key.

---

### Impact Explanation

An attacker who observes a valid `(signatureData, MetadataPayload)` submitted to any one chain (e.g., from a confirmed transaction or mempool) can immediately replay it on every other supported chain:

1. A spurious bridge token is deployed on the target chain with no NEAR-side collateral backing.
2. The `nearToEthToken` / `near_to_starknet_token` mapping is poisoned: the NEAR token ID is now mapped to the attacker-deployed contract.
3. All future legitimate `deployToken` calls for the same NEAR token ID on that chain revert with `ERR_TOKEN_EXIST` / `ERR_TOKEN_ALREADY_DEPLOYED`. [5](#0-4) [6](#0-5) 

The legitimate token can never be deployed on that chain again without an admin intervention (contract upgrade or state reset). Users who receive the spurious token cannot redeem it for real assets. This constitutes **permanent freezing of the token deployment flow** and **unauthorized token deployment** on every chain other than the one the MPC originally targeted.

---

### Likelihood Explanation

- The MPC signature is publicly visible the moment it is submitted to any chain.
- The attacker needs no privileged access: `deployToken` / `deploy_token` is a permissionless external function on all three chains. [7](#0-6) [8](#0-7) 
- The attacker only needs to copy the calldata from chain A and submit it to chain B before the legitimate relayer does.
- Because the MPC signs the same payload for all chains (no chain binding), every new token deployment leaks a replayable signature for all other chains simultaneously.

---

### Recommendation

Include a destination chain identifier in the signed `MetadataPayload` hash. The simplest fix mirrors what `finTransfer` already does: prepend `omniBridgeChainId` (or the equivalent chain constant) to the borsh-encoded bytes before hashing. Additionally, including the contract address (`address(this)` on EVM, `get_contract_address()` on StarkNet, the program ID on Solana) provides full domain separation and prevents replay across different deployments of the same bridge on the same chain.

For example, on EVM:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // ADD: destination chain binding
    Borsh.encodeAddress(address(this)), // ADD: contract address binding
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

The same fields must be added consistently in the NEAR MPC signing logic, the StarkNet `to_borsh`, and the Solana `serialize_for_near` implementation.

---

### Proof of Concept

1. The NEAR bridge MPC signs a `MetadataPayload` for NEAR token `"usdc.near"` to be deployed on Ethereum (chain ID byte `0x01`). The signed bytes are `[0x01, len+"usdc.near", len+"USD Coin", len+"USDC", 0x06]`.

2. A relayer submits `deployToken(sig, {token:"usdc.near", name:"USD Coin", symbol:"USDC", decimals:6})` to the Ethereum `OmniBridge`. The transaction is confirmed and visible on-chain.

3. An attacker copies the exact same `(sig, metadata)` calldata and calls `deployToken` on the StarkNet `OmniBridge` and the Solana `bridge_token_factory` program.

4. On StarkNet, `_verify_borsh_signature` computes `keccak256([0x01, "usdc.near", "USD Coin", "USDC", 0x06])` — identical to the Ethereum hash — and `verify_eth_signature` passes against `omni_bridge_derived_address`. [9](#0-8) 

5. On Solana, `SignedPayload::verify_signature` computes `keccak::hash(serialize_for_near())` — identical bytes — and `secp256k1_recover` passes against `derived_near_bridge_address`. [10](#0-9) 

6. Both chains now have a spurious `usdc.near` bridge token deployed by the attacker. The `near_to_starknet_token` and Solana mint PDA are permanently poisoned. Legitimate deployment of `usdc.near` on StarkNet and Solana is permanently blocked.

```rust
    pub fn verify_signature(
        &self,
        params: P::AdditionalParams,
        derived_near_bridge_address: &[u8; 64],
    ) -> Result<()> {
        let serialized = self.payload.serialize_for_near(params)?;
        let hash = keccak::hash(&serialized);

        let signature_bytes = &self.signature[0..64];

        let signature = libsecp256k1::Signature::parse_standard_slice(signature_bytes)
            .map_err(|_| ProgramError::InvalidArgument)?;
        require!(!signature.s.is_high(), ErrorCode::MalleableSignature);

        let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)
            .map_err(|_| error!(ErrorCode::SignatureVerificationFailed))?;

        require!(
            signer.0 == *derived_near_bridge_address,
            ErrorCode::SignatureVerificationFailed
        );

        Ok(())
    }
```
