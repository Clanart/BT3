### Title
Unattributed Native ETH Residue in `pay()` Allows Cross-Transaction Theft of Victim's Stranded ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

`PeripheryPayments.pay()` uses `address(this).balance` as a shared, unattributed pool of native ETH when settling a WETH-input swap. Any native ETH left on the router by a prior user's transaction is silently consumed by the next caller whose `tokenIn` is WETH, giving that caller a free (or partially free) swap at the victim's expense.

### Finding Description

`pay()` branches on `token == WETH` and reads `address(this).balance` to decide how much native ETH to deposit before pulling the remainder from the payer's WETH allowance: [1](#0-0) 

There is no per-transaction or per-user accounting. `address(this).balance` reflects the entire router's native balance, including ETH stranded by earlier transactions.

ETH is stranded when a user calls `exactInputSingle{value: X}` with `tokenIn = WETH` and `amountIn = Y < X`. The callback fires `_justPayCallback` → `pay(WETH, victim, pool, Y)`. The function deposits exactly `Y` ETH and sends WETH to the pool; the remaining `X − Y` stays on the router as native ETH with no owner record. [2](#0-1) 

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks plain ETH transfers; it does **not** block ETH sent via a `payable` function call, so `exactInputSingle{value: X}` successfully deposits `X` onto the router. [3](#0-2) 

In a subsequent transaction the attacker calls `exactInputSingle{value: 0}` with `tokenIn = WETH`. The callback fires `pay(WETH, attacker, pool, Z)`. Because `nativeBalance = X − Y ≥ Z`, the function deposits the victim's ETH and sends WETH to the pool — the attacker receives the swap output without spending any WETH. [4](#0-3) 

A second, even simpler theft path exists: `refundETH()` sends the entire router ETH balance to `msg.sender` with no access control, so the attacker can call it directly after the victim's transaction to drain the stranded ETH without needing a WETH swap at all. [5](#0-4) 

### Impact Explanation

Direct loss of user ETH. Any victim who calls `exactInputSingle` (or `exactInput`) with `msg.value > amountIn` and omits a `refundETH()` step loses the excess. The attacker recovers it in the next block with a zero-cost call. Loss magnitude equals `msg.value − amountIn` per victim transaction; with MEV bots watching the mempool this is reliably exploitable.

### Likelihood Explanation

Sending a small ETH surplus to guarantee a swap succeeds is a common user pattern. The router is `payable` and the docs/interface do not enforce that callers must wrap their call in a `multicall` with `refundETH()`. An MEV bot can monitor for any `exactInputSingle{value > 0}` transaction and front-run or back-run with a `refundETH()` call in the same block.

### Recommendation

Track the `msg.value` contributed by the current top-level call and limit `pay()` to consuming only that amount of native ETH. One approach: record `msg.value` in a transient slot at the start of each public entry point and decrement it as ETH is consumed in `pay()`; revert if `pay()` would consume more than the recorded amount. Alternatively, require callers to pre-wrap ETH to WETH before calling the router and remove the native-ETH branch from `pay()` entirely, relying on `multicall` + `unwrapWETH9` for the ETH-in / ETH-out UX.
