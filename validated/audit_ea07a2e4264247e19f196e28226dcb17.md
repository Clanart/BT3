### Title
Unvalidated `l1_gas_price_wei` / `l1_data_gas_price_wei` in `ProposalInit` Allows Wrong Block Hash Commitment — (`File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

---

### Summary

`is_block_info_valid` validates only the FRI-denominated gas prices from a received `ProposalInit` against the oracle. The WEI-denominated prices (`l1_gas_price_wei`, `l1_data_gas_price_wei`) are never checked. Both WEI prices are directly committed into the block hash via `gas_prices_to_hash`, and the wrong WEI price also cascades into a wrong `l2_gas_price_wei` through the ETH/STRK rate derivation. A malicious proposer can set arbitrary WEI prices that pass all validation and produce a permanently wrong block hash in storage.

---

### Finding Description

**Root cause — `is_block_info_valid` discards the oracle WEI prices it already fetches**

In `validate_proposal.rs`, the validator fetches both FRI and WEI prices from the oracle:

```rust
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...)
    .await;
``` [1](#0-0) 

The `_l1_gas_prices_wei` result is silently discarded (note the `_` prefix). Only the FRI prices are then compared against the proposal:

```rust
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, ...)
    && within_margin(l1_data_gas_price_fri_proposed, l1_data_gas_price_fri, ...))
``` [2](#0-1) 

The proposed `init.l1_gas_price_wei` and `init.l1_data_gas_price_wei` are never compared against anything.

**Propagation path 1 — WEI prices enter `BlockInfo` unchecked**

`convert_to_sn_api_block_info` takes the WEI prices directly from `ProposalInit` without any validation:

```rust
let l1_gas_price_wei = NonzeroGasPrice::new(init.l1_gas_price_wei)?;
let l1_data_gas_price_wei = NonzeroGasPrice::new(init.l1_data_gas_price_wei)?;
``` [3](#0-2) 

**Propagation path 2 — WEI prices enter the block hash**

`PartialBlockHashComponents::new` captures both WEI and FRI prices from `BlockInfo`:

```rust
l1_gas_price: block_info.gas_prices.l1_gas_price_per_token(),   // price_in_wei + price_in_fri
l1_data_gas_price: block_info.gas_prices.l1_data_gas_price_per_token(),
``` [4](#0-3) 

`gas_prices_to_hash` then chains `price_in_wei` into the Poseidon hash for all protocol versions:

```rust
.chain(&l1_gas_price.price_in_wei.0.into())
.chain(&l1_gas_price.price_in_fri.0.into())
.chain(&l1_data_gas_price.price_in_wei.0.into())
.chain(&l1_data_gas_price.price_in_fri.0.into())
``` [5](#0-4) 

**Propagation path 3 — wrong WEI price cascades to wrong `l2_gas_price_wei`**

`convert_to_sn_api_block_info` derives the ETH/STRK rate from the proposed WEI and FRI prices:

```rust
let previous_block_info = PreviousBlockInfo::from(init);
let eth_to_fri_rate = calculate_eth_to_fri_rate(&previous_block_info)?;
let l2_gas_price_wei = NonzeroGasPrice::new(init.l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)?)?;
``` [6](#0-5) 

`calculate_eth_to_fri_rate` computes `(l1_gas_price_fri × WEI_PER_ETH) / l1_gas_price_wei`: [7](#0-6) 

A manipulated `l1_gas_price_wei` therefore corrupts the ETH/STRK rate, which corrupts `l2_gas_price_wei`, which is also chained into the block hash.

**Propagation path 4 — wrong WEI prices persist into the next block's gas price calculation**

The committed block header's WEI prices become `previous_block_info` for the next height: [8](#0-7) 

The next block's `calculate_eth_to_fri_rate` re-derives the conversion rate from these stored WEI prices, propagating the error forward.

**The `ProposalInit` struct explicitly carries both WEI and FRI prices for independent validation — but validation never happens for WEI** [9](#0-8) 

The comment states the design intent is to allow distinguishing oracle errors from conversion errors. The WEI validation path is simply missing.

---

### Impact Explanation

A malicious proposer sets `l1_gas_price_wei = 1` (or any value far from the true oracle price) while keeping `l1_gas_price_fri` within the allowed margin. The validator accepts the proposal, executes the block, and commits a `PartialBlockHash` that encodes the wrong WEI price. The committed block hash is permanently wrong. All downstream consumers of the block hash (state sync, proof verifier, L1 verifier) receive a hash that does not correspond to the true gas prices. ETH-denominated fee calculations for L1 handler transactions in that block use the wrong WEI price, producing wrong receipts. The corrupted WEI price also propagates to the next block's ETH/STRK rate.

This matches:
- **Critical**: Wrong state/receipt/revert result — wrong block hash committed to storage.
- **Critical**: Incorrect fee/gas/resource accounting with economic impact — wrong ETH-denominated fees for L1 handler transactions.

---

### Likelihood Explanation

Any validator legitimately selected as proposer in a BFT round can craft this `ProposalInit`. Proposer selection rotates among validators; no special privilege beyond being a validator is required. The manipulation is invisible to the validator node because the only check performed is on FRI prices.

---

### Recommendation

In `is_block_info_valid`, compare the proposed WEI prices against the oracle-computed WEI prices (already available as `_l1_gas_prices_wei`) using the same `within_margin` check applied to FRI prices:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(...).await;

// existing FRI checks ...

if !(within_margin(init_proposed.l1_gas_price_wei, l1_gas_prices_wei.l1_gas_price, margin)
    && within_margin(init_proposed.l1_data_gas_price_wei, l1_gas_prices_wei.l1_data_gas_price, margin))
{
    return Err(ValidateProposalError::InvalidBlockInfo(...));
}
```

Alternatively, validate that the ratio `l1_gas_price_fri / l1_gas_price_wei` (the implied ETH/STRK rate) is within an acceptable range of the oracle-reported rate.

---

### Proof of Concept

1. Validator A is selected as proposer for height H.
2. A crafts `ProposalInit` with `l1_gas_price_fri = X` (within oracle margin) but `l1_gas_price_wei = 1` (far from true oracle WEI price).
3. Validator B receives the proposal and calls `is_block_info_valid`:
   - Fetches oracle FRI price ≈ X → `within_margin` passes.
   - `_l1_gas_prices_wei` is computed but discarded — WEI price `1` is never checked.
4. `convert_to_sn_api_block_info` creates `BlockInfo` with `eth_gas_prices.l1_gas_price = 1`.
5. `PartialBlockHashComponents::new` stores `l1_gas_price.price_in_wei = 1`.
6. `gas_prices_to_hash` chains `1` into the Poseidon hash.
7. `BlockExecutionArtifacts::commitment()` returns a `PartialBlockHash` encoding `price_in_wei = 1`.
8. `decision_reached` commits this wrong hash to storage via `commit_proposal_and_block`.
9. The stored block hash differs from what an honest node would compute using the true WEI price, and all L1 handler fee receipts in the block are computed with `l1_gas_price_wei = 1`.

fn calculate_eth_to_fri_rate(block_info: &PreviousBlockInfo) -> Result<u128, StarknetApiError> {
    let eth_to_fri_rate = block_info
        .l1_prices_fri
        .l1_gas_price
        .0
        .checked_mul(WEI_PER_ETH)
        .ok_or_else(|| {
            StarknetApiError::GasPriceConversionError(format!(
                "Gas price in Fri should be small enough to multiply by WEI_PER_ETH. Previous \
                 block info: {:?}",
                block_info
            ))
        })?
        .checked_div(block_info.l1_prices_wei.l1_gas_price.0)
        .ok_or_else(|| {
            StarknetApiError::GasPriceConversionError(format!(
                "Gas price in Wei should be non-zero. Previous block info: {:?}",
                block_info
            ))
        })?;
    if eth_to_fri_rate == 0 {
        return Err(StarknetApiError::GasPriceConversionError(format!(
            "Eth to fri rate is zero. Previous block info: {:?}",
            block_info
        )));
    }
    Ok(eth_to_fri_rate)

/// This message must be sent first when proposing a new block.
/// This struct differs from `BlockInfo` in `starknet_api` because we send L1 gas prices in ETH and
/// include the ETH to STRK conversion rate. This allows for more informative validations, as we can
/// distinguish whether an issue comes from the L1 price reading or the conversion rate instead of
/// comparing after multiplication.
#[derive(Clone, Debug, PartialEq)]
pub struct ProposalInit {
    /// The height of the consensus (block number).
    pub height: BlockNumber,
    /// The current round of the consensus.
    pub round: Round,
    /// The last round that was valid.
    pub valid_round: Option<Round>,
    /// Address of the one who proposed the block in consensus.
    pub proposer: ContractAddress,
    /// Block timestamp.
    pub timestamp: u64,
    /// Address of the one who builds/sequences the block.
    pub builder: ContractAddress,
    /// L1 data availability mode.
    pub l1_da_mode: L1DataAvailabilityMode,
    /// L2 gas price in FRI.
    pub l2_gas_price_fri: GasPrice,
    /// L1 gas price in FRI.
    pub l1_gas_price_fri: GasPrice,
    /// L1 data gas price in FRI.
    pub l1_data_gas_price_fri: GasPrice,
    // Keeping the wei prices for now, to use with L1 transactions.
    /// L1 gas price in WEI.
    pub l1_gas_price_wei: GasPrice,
    /// L1 data gas price in WEI.
    pub l1_data_gas_price_wei: GasPrice,
    /// Starknet protocol version.
    pub starknet_version: starknet_api::block::StarknetVersion,
    /// Version constant commitment.
    pub version_constant_commitment: StarkHash,
}
