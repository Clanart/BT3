### Title
`prepare_query` Builds Proof Query from Pre-`update_initial_values_of_write_only_access` `initial_reads`, Causing Write-Only Storage Keys to Be Absent from Patricia Trie Proof and Producing Wrong State Roots in Proof Inputs - (`File: crates/starknet_transaction_prover/src/running/storage_proofs.rs`)

---

### Summary

In `VirtualBlockExecutor::execute()`, `initial_reads` is captured from `block_state.get_initial_reads()` **before** `block_state.to_state_diff()` is called. The `to_state_diff()` call internally invokes `update_initial_values_of_write_only_access()`, which retroactively inserts storage entries into `cache.initial_reads` for every storage slot that was **written but never read**. Because `initial_reads` is snapshotted before this update, `execution_data.initial_reads.storage` is missing those write-only keys, while `execution_data.state_diff` (computed after the update) contains them. `prepare_query` then builds the RPC proof query exclusively from the stale `initial_reads.storage`, so write-only storage keys are never requested. The resulting `FactsDb` lacks the Patricia storage-trie nodes for those keys. When `commit_state_diff` applies the full state diff (including write-only keys), it computes wrong storage roots, propagating incorrect state roots into the proof inputs.

---

### Impact Explanation

The corrupted `commitment_infos` (wrong `updated_root` for storage tries) are forwarded directly into `VirtualOsBlockInput.address_to_storage_commitment_info` and `contract_state_commitment_info`.

```rust
// crates/starknet_transaction_prover/src/running/runner.rs  L231-253
let virtual_os_block_input = VirtualOsBlockInput {
    contract_state_commitment_info: storage_proofs.commitment_infos.contracts_trie_commitment_info,
    address_to_storage_commitment_info: storage_proofs.commitment_infos.storage_tries_commitment_infos,
    ...
};
``` [6](#0-5) 

The OS uses these roots to verify that the post-execution global state root is correct. A wrong `updated_root` means the OS either accepts a proof for an incorrect state transition or rejects a valid one. This matches the **Critical** impact: "Wrong state … or revert result from blockifier/syscall/execution logic for accepted input."

---

### Likelihood Explanation

Any Cairo contract that performs a `storage_write` syscall on a slot it has not previously read in the same transaction triggers this path. This is a normal and common pattern (e.g., initializing a counter, emitting a write-only event log slot, or any contract that unconditionally overwrites a slot). An unprivileged user submitting a single Invoke transaction to such a contract is sufficient to trigger the bug; no special privileges or adversarial infrastructure are required.

---

### Recommendation

Capture `initial_reads` **after** `to_state_diff()` has been called (so that `update_initial_values_of_write_only_access` has already populated write-only storage entries), or explicitly call `update_initial_values_of_write_only_access` before snapshotting `initial_reads`. Concretely, in `VirtualBlockExecutor::execute()`:

```rust
// Compute state diff first (this populates write-only initial reads internally).
let state_diff = block_state.to_state_diff()?.state_maps;

// Now snapshot initial_reads — it includes write-only storage entries.
let initial_reads = block_state.get_initial_reads()?;
```

This mirrors the fix applied in the Blend protocol: ensuring the cache is fully populated before it is used as the authoritative source for downstream queries.

---

### Proof of Concept

1. Deploy a Cairo contract with a function `write_slot(key, value)` that calls `storage_write(key, value)` without a prior `storage_read(key)`.
2. Pre-populate `key` with a non-zero value on-chain.
3. Submit an Invoke transaction calling `write_slot(key, new_value)` through the virtual block executor.
4. Observe that `execution_data.initial_reads.storage` does **not** contain `(contract_address, key)`.
5. Observe that `execution_data.state_diff.storage` **does** contain `(contract_address, key) → new_value`.
6. The RPC proof query built by `prepare_query` omits `key` from `contract_storage_keys` for `contract_address`.
7. `create_facts_db_from_storage_proof` builds a `FactsDb` without the storage-trie node for `key`.
8. `commit_state_diff` computes the new storage root for `contract_address` starting from a missing/zero node for `key` instead of the real pre-execution node, producing a wrong `updated_root`.
9. The `VirtualOsBlockInput` carries this wrong root, causing the OS to prove an incorrect state transition.

    fn update_initial_values_of_write_only_access(&mut self) -> StateResult<()> {
        let cache = &mut *self.cache.borrow_mut();

        // Eliminate storage writes that are identical to the initial value (no change).
        for contract_storage_key in cache.writes.storage.keys() {
            if !cache.initial_reads.storage.contains_key(contract_storage_key) {
                // First access to this cell was write; cache initial value.
                cache.initial_reads.storage.insert(
                    *contract_storage_key,
                    self.state.get_storage_at(contract_storage_key.0, contract_storage_key.1)?,
                );
            }
        }
        Ok(())
    }
