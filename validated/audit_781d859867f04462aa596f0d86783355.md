### Title
`validate_virtual_os_state_diff` Rejects Legitimate Zero-Value Storage Writes, Blocking Transaction Prover Proof Generation for Valid Blocks — (`File: crates/starknet_transaction_prover/src/running/committer_utils.rs`)

---

### Summary

The `validate_virtual_os_state_diff` function in the transaction prover unconditionally rejects any storage update whose value is `Felt::ZERO`, treating it as a forbidden "storage deletion." However, a legitimate Invoke transaction (e.g., a full ERC-20 token transfer that drains a balance slot to zero) produces exactly such a zero-value write in the execution state diff. Because `VirtualBlockExecutor::execute()` passes the raw `StateMaps` state diff directly into the proof pipeline without filtering or pre-validating this constraint, the prover returns `ProofProviderError::InvalidStateDiff` and cannot generate a proof for an otherwise valid block.

---

### Finding Description

**Concrete trigger — full ERC-20 drain:**

A user whose STRK balance storage slot holds value `V > 0` submits an Invoke transaction that transfers all `V` tokens to another address. After execution:

- The sender's ERC-20 balance slot `(strk_fee_token_address, ERC20_balances[sender])` is written to `Felt::ZERO`.
- This write appears in `state_diff.storage` and survives `state_maps_to_committer_state_diff` unchanged.
- `validate_virtual_os_state_diff` hits the `value.0 == Felt::ZERO` branch and returns `Err(ProofProviderError::InvalidStateDiff(...))`.
- `create_commitment_infos_with_state_changes` propagates the error, `get_storage_proofs` fails, and `create_virtual_os_hints` in the runner cannot assemble the OS hints needed to produce a proof. [5](#0-4) 

The analogy to the original report is exact:

| Notional vault | Sequencer transaction prover |
|---|---|
| `vaultAccount.tempCashBalance` becomes positive after liquidation + interest | `state_diff.storage` entry becomes `Felt::ZERO` after a full token transfer |
| `require(vaultAccount.tempCashBalance == 0)` in `redeemWithDebtRepayment` reverts | `if value.0 == Felt::ZERO { return Err(...) }` in `validate_virtual_os_state_diff` aborts |
| Owner cannot exit vault; assets stuck | Prover cannot generate proof for valid block; block cannot be finalized on L1 |

---

### Impact Explanation

A valid, accepted Invoke transaction that writes `Felt::ZERO` to any storage slot causes `validate_virtual_os_state_diff` to return an error, aborting the entire proof-generation pipeline for that virtual block. The transaction prover returns a wrong (error) result for accepted, correctly-executed input. No block containing such a transaction can ever be proven, permanently blocking L1 finalization for that block.

This matches: **Critical — Wrong state/revert result from blockifier/syscall/execution logic for accepted input** (the prover produces an error result instead of a valid proof for a correctly-executed block), and **High — RPC execution / simulation returns an authoritative-looking wrong value** (any simulation or fee-estimation path that exercises the same prover pipeline will surface the same error).

---

### Likelihood Explanation

Any Invoke transaction that fully drains a storage slot to zero triggers this. Common real-world cases include:

- Full ERC-20 balance transfer (sender balance → 0)
- Allowance reset to zero
- Any contract that explicitly writes `0` to clear a storage variable

These are routine operations on Starknet mainnet. The trigger requires no special privilege — any unprivileged user submitting a standard Invoke transaction can cause it.

---

### Recommendation

Before passing the execution state diff into `validate_virtual_os_state_diff`, filter out zero-value storage entries from `state_diff.storage` in `state_maps_to_committer_state_diff` (or in the caller), so that legitimate storage deletions are silently dropped rather than causing a hard error. Alternatively, if zero-value writes must be preserved for correctness of the Patricia trie update, the virtual OS constraint should be relaxed to permit them and the trie committer updated accordingly.

---

### Proof of Concept

1. Construct an Invoke V3 transaction from address `A` that calls `transfer(recipient=B, amount=balance_of(A))` on the STRK fee token contract, fully draining `A`'s balance.
2. Pass this transaction to `RpcVirtualBlockExecutor::execute()`.
3. Execution succeeds (no revert); `VirtualBlockExecutionData.state_diff.storage` contains the entry `(strk_fee_token_address, ERC20_balances[A]) → Felt::ZERO`.
4. `get_storage_proofs` is called; it invokes `create_commitment_infos_with_state_changes`.
5. `state_maps_to_committer_state_diff` converts the state diff, preserving the zero value.
6. `validate_virtual_os_state_diff` returns `Err(ProofProviderError::InvalidStateDiff("Storage deletion not allowed: ..."))`.
7. `create_virtual_os_hints` propagates the error; the runner cannot produce `OsHints`; proof generation is permanently blocked for this block. [6](#0-5) [7](#0-6)

/// Validates that the committer state diff contains only allowed state transitions.
///
/// This function enforces the following constraints:
/// * **No Storage Deletions:** Storage entries cannot be updated to `Felt::ZERO`.
/// * **No Class Declarations:** The `class_hash_to_compiled_class_hash` map must be empty.
/// * **No Contract Deployments:** The `address_to_class_hash` map must be empty.
pub(crate) fn validate_virtual_os_state_diff(
    state_diff: &StateDiff,
) -> Result<(), ProofProviderError> {
    // validate no storage deletions.
    for (address, storage_diffs) in &state_diff.storage_updates {
        for (key, value) in storage_diffs {
            if value.0 == Felt::ZERO {
                return Err(ProofProviderError::InvalidStateDiff(format!(
                    "Storage deletion not allowed: try to delete storage at address {address:?}, \
                     key {key:?}"
                )));
            }
        }
    }
    // validate no contract deployments (or replaced classes).
    if !state_diff.address_to_class_hash.is_empty() {
        return Err(ProofProviderError::InvalidStateDiff(format!(
            "Contract deployments not allowed: try to deploy contracts(address to class hash): \
             {0:?}",
            state_diff.address_to_class_hash
        )));
    }
    // validate no contract declarations (or compiled class hash updates).
    if !state_diff.class_hash_to_compiled_class_hash.is_empty() {
        return Err(ProofProviderError::InvalidStateDiff(format!(
            "Contract declarations not allowed: try to declare classes(class hash to compiled \
             class hash): {0:?}",
            state_diff.class_hash_to_compiled_class_hash
        )));
    }
    Ok(())
}
