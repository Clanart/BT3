Audit Report

## Title
Caller-Controlled `pool` Address in `MetricOmmPoolLiquidityAdder` Allows Malicious Pool to Drain User-Approved Tokens - (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

## Summary
`MetricOmmPoolLiquidityAdder` accepts an arbitrary `pool` address from the caller and stores it as the trusted callback counterparty in transient storage without validating it against the factory. A malicious pool contract trivially passes the `msg.sender == expectedPool` guard in `metricOmmModifyLiquidityCallback`, returns attacker-chosen token addresses from `getImmutables()`, and causes the adder to execute `transferFrom(victim, maliciousPool, amount)` for up to `maxAmountToken0`/`maxAmountToken1` of any ERC-20 the victim has approved.

## Finding Description
`addLiquidityExactShares` and `addLiquidityWeighted` both accept a caller-supplied `pool` address and pass it directly to `_addLiquidity`, which stores it as the authoritative callback pool via `_setPayContext`: [1](#0-0) 

The transient-storage guard in `metricOmmModifyLiquidityCallback` only checks that `msg.sender` equals the stored pool address: [2](#0-1) 

Because the stored address was set to the malicious pool by the caller, this check is trivially satisfied. The callback then queries `IMetricOmmPool(msg.sender).getImmutables()` to determine which tokens to pull from the payer: [3](#0-2) 

A malicious pool controls both the return value of `getImmutables()` (choosing any token addresses) and the `amount0Delta`/`amount1Delta` values it passes into the callback (up to the user-supplied caps). The contract's own NatSpec explicitly acknowledges this gap: [4](#0-3) 

By contrast, `MetricOmmSwapRouterBase._requireExpectedCallbackCaller` enforces `FACTORY.isPool(caller)` before accepting any callback, and `_requireFactoryPool` validates every pool before storing it in transient context: [5](#0-4) 

`MetricOmmPoolLiquidityAdder` has no equivalent check. Its constructor takes only `weth` and stores no factory reference: [6](#0-5) 

## Impact Explanation
Any user who has granted a token approval to `MetricOmmPoolLiquidityAdder` can have up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 stolen in a single transaction. The `pay` call executes `transferFrom(victim, maliciousPool, amount)` for both tokens, constituting a direct loss of user principal. This meets the Critical/High direct-loss-of-user-principal threshold.

## Likelihood Explanation
The trigger requires no privileged role. The only preconditions are that the victim has approved the adder (standard UX for any liquidity deposit flow) and is induced to call with a malicious pool address via a phishing front-end or a compromised integration. Deploying a contract that satisfies the `IMetricOmmPoolActions` interface is trivial. Likelihood is Medium.

## Recommendation
Store the factory address in the constructor (as `MetricOmmSimpleRouter` does via `MetricOmmSwapRouterBase`) and call `factory.isPool(pool)` in `_addLiquidity` before `_setPayContext`, reverting if the pool is not registered. Apply the same check in the `addLiquidityWeighted` probe path before calling `pool.addLiquidity`. This mirrors the existing `_requireFactoryPool` pattern in `MetricOmmSwapRouterBase`. [7](#0-6) 

## Proof of Concept
1. Deploy `MaliciousPool(adderAddress, USDC, WETH, attackerAddress)` implementing `IMetricOmmPoolActions` and `IMetricOmmPool`.
2. `MaliciousPool.addLiquidity()` re-enters `metricOmmModifyLiquidityCallback(maxAmountToken0, maxAmountToken1, abi.encode(KIND_PAY))`.
3. `MaliciousPool.getImmutables()` returns `token0 = USDC`, `token1 = WETH`.
4. Victim (who has approved the adder for USDC and WETH) calls `adder.addLiquidityExactShares(maliciousPool, victim, 0, deltas, victimUSDCBalance, victimWETHBalance, "")`.
5. Adder stores `maliciousPool` as `expectedPool`, calls `maliciousPool.addLiquidity()`.
6. `MaliciousPool` calls back `metricOmmModifyLiquidityCallback` — `msg.sender == expectedPool` passes.
7. Adder executes `pay(USDC, victim, maliciousPool, victimUSDCBalance)` and `pay(WETH, victim, maliciousPool, victimWETHBalance)`.
8. Both `transferFrom` calls succeed; victim loses full approved balances.
