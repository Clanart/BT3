### Title
Missing Nonce in `compute_meta_tx_v0_hash` Enables Signature Replay Attack — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/transaction_hash/transaction_hash.cairo`)

---

### Summary

The `compute_meta_tx_v0_hash` function in the StarkNet OS computes the hash for `meta_tx_v0` syscall signatures without including a nonce in the signed data. This is a direct analog to the NFT Port EIP-712 vulnerability: just as that contract allowed signature replay due to missing nonces and domain separators, the OS-level hash for meta transactions omits the nonce entirely, making every valid `meta_tx_v0` signature permanently replayable by any observer.

---

### Finding Description

`compute_meta_tx_v0_hash` is defined as:

```cairo
func compute_meta_tx_v0_hash{pedersen_ptr: HashBuiltin*}(
    contract_address: felt,
    entry_point_selector: felt,
    calldata: felt*,
    calldata_size: felt,
    chain_id: felt,
) -> felt {
    let (tx_hash) = deprecated_get_transaction_hash{hash_ptr=pedersen_ptr}(
        tx_hash_prefix=INVOKE_HASH_PREFIX,
        version=0,
        ...
        additional_data_size=0,
        additional_data=cast(0, felt*),   // ← no nonce
    );
    return tx_hash;
}
``` [1](#0-0) 

The `additional_data_size=0` and `additional_data=cast(0, felt*)` mean **no nonce is committed to in the signed hash**.

Contrast this with `compute_l1_handler_transaction_hash`, which correctly passes the nonce as `additional_data`:

```cairo
additional_data_size=1,
additional_data=&nonce,
``` [2](#0-1) 

The signed fields for `meta_tx_v0` are only: `INVOKE_HASH_PREFIX`, `version=0`, `contract_address`, `entry_point_selector`, `calldata_hash`, `max_fee=0`, `chain_id`. There is no nonce, no sender address, and no replay-prevention mechanism at the OS hash level.

The `meta_tx_v0` syscall is a live, gas-priced feature of the OS: [3](#0-2) 

---

### Impact Explanation

An attacker who observes any valid `meta_tx_v0` signature (e.g., from a submitted transaction or mempool) can replay it an unlimited number of times against the same `(contract_address, entry_point_selector, calldata)` tuple. If the meta transaction encodes a token transfer, asset approval, or any value-bearing operation, the attacker can drain the signer's funds by replaying the signature repeatedly. This constitutes **direct loss of funds (Critical)**.

---

### Likelihood Explanation

The `meta_tx_v0` syscall is a publicly accessible protocol feature. Any unprivileged user who submits or observes a `meta_tx_v0`-bearing transaction gains a permanently valid replay token. No privileged access, key compromise, or network-level attack is required. The attacker only needs to observe one valid signature on-chain or in the mempool.

---

### Recommendation

Include a nonce in the `compute_meta_tx_v0_hash` computation, passed as `additional_data`, exactly as `compute_l1_handler_transaction_hash` does. The nonce must be enforced and incremented at the OS level (not left to the calling contract), so that each signature is bound to a single execution. Additionally, consider binding the signer's address into the hash to prevent cross-account reuse.

```cairo
// Corrected:
additional_data_size=1,
additional_data=&nonce,
```

---

### Proof of Concept

1. User A signs a `meta_tx_v0` authorizing a transfer of 100 tokens: `(contract_address=token, selector=transfer, calldata=[recipient, 100], chain_id=SN_MAIN)`.
2. The OS computes `H = Pedersen(INVOKE_HASH_PREFIX, 0, token, transfer, H(calldata), 0, SN_MAIN)` — no nonce.
3. User A submits the transaction; it executes and 100 tokens are transferred.
4. Attacker observes the signature `sig` from the mempool or on-chain.
5. Attacker submits a new transaction using the same `(contract_address, selector, calldata, sig)`.
6. The OS recomputes the identical hash (no nonce to differ), verifies `sig` successfully, and executes the transfer again.
7. Attacker repeats step 5–6 until User A's balance is zero.

No privileged access is required. The only precondition is observing one valid signature.
