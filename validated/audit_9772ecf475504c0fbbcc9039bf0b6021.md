### Title
Hard-Coded `L1_HANDLER_L2_GAS_MAX_AMOUNT` Gas Cap Causes Permanent Loss of Bridged Funds for L1→L2 Message Senders - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`execute_l1_handler_transaction` executes every L1→L2 message handler with a hard-coded gas budget of `L1_HANDLER_L2_GAS_MAX_AMOUNT = 100_000_000` L2 gas units. The L1 message is consumed (written to OS output) **before** execution begins. If the handler runs out of gas, the message is permanently consumed but its state effects are reverted, causing any bridged funds to be permanently lost with no replay path.

---

### Finding Description

In `execute_l1_handler_transaction`, the OS unconditionally sets the gas budget for the handler entry point to the compile-time constant `L1_HANDLER_L2_GAS_MAX_AMOUNT`:

```cairo
// Consume L1-to-L2 message.
consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
let remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT;
non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
    block_context=block_context, execution_context=tx_execution_context
);
``` [1](#0-0) 

The constant is defined as:

```cairo
const L1_HANDLER_L2_GAS_MAX_AMOUNT = 100000000;
``` [2](#0-1) 

Two structural problems compound each other:

**1. Message consumed before execution.** `consume_l1_to_l2_message` writes the message to the OS output segment at line 444, before the entry point is called at line 446. The OS output is the authoritative record used by the L1 StarkNet core contract to mark a message as processed. Once written, there is no mechanism to un-consume the message or replay it.

**2. No user control over the gas budget.** For regular invoke transactions, the user specifies `resource_bounds[L2_GAS_INDEX].max_amount` and the OS uses `get_initial_user_gas_bound` to derive the execution gas:

```cairo
let initial_user_gas_bound = get_initial_user_gas_bound(common_tx_fields=common_tx_fields);
let remaining_gas = initial_user_gas_bound;
``` [3](#0-2) 

L1 handler transactions carry no `resource_bounds` field (set to null pointers at lines 428–429), so the L1 message sender has zero ability to increase the gas budget beyond the hard-coded ceiling.

**3. Severe asymmetry with regular execution.** `EXECUTE_MAX_SIERRA_GAS = 1_100_000_000` (11× higher) is the cap for regular `__execute__` calls. L1 handlers receive only `100_000_000`, the same budget as `__validate__`, despite potentially needing to perform complex bridging logic (token minting, storage writes, sub-calls, event emissions). [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent loss of funds.**

When a user sends an L1→L2 message (e.g., depositing ETH or ERC-20 tokens via a bridge), the L1 contract locks or burns the funds and emits the message. On L2, the OS:

1. Writes the message to the consumed-messages output (irreversible).
2. Executes the L1 handler with only 100M gas.
3. If the handler exceeds 100M gas (out-of-gas), `non_reverting_select_execute_entry_point_func` returns `is_reverted=1` and all state changes from the handler are discarded.

The L1 funds are already locked. The L2 message is already consumed. There is no cancellation or replay mechanism in the OS. The funds are permanently lost.

A single `STORAGE_WRITE_GAS_COST = 44_970` costs ~45K gas; a `CALL_CONTRACT_GAS_COST = 91_560` costs ~92K gas. A handler that mints tokens, emits events, and calls a registry contract can easily exceed 100M gas in realistic DeFi bridge scenarios. [5](#0-4) 

---

### Likelihood Explanation

**Medium-High.** Any L1→L2 bridge interaction targeting an L2 contract with a non-trivial L1 handler is at risk. The L1 message sender is an unprivileged actor with no ability to increase the gas budget. As L2 contracts grow in complexity (multi-step DeFi protocols, NFT minting with royalty logic, DAO treasury handlers), the probability of exceeding 100M gas increases. The limit is baked into the OS binary and cannot be adjusted per-transaction.

---

### Recommendation

Replace the hard-coded `L1_HANDLER_L2_GAS_MAX_AMOUNT` constant with a value derived from the L1 message itself (e.g., a `gas_limit` field in the L1→L2 message payload that the L1 sender specifies and pays for), or raise the ceiling to match `EXECUTE_MAX_SIERRA_GAS`. At minimum, the message should only be written to the consumed-messages output **after** successful execution, so that out-of-gas failures can be retried.

---

### Proof of Concept

1. Deploy an L2 contract whose L1 handler performs ~25 storage writes and 2 sub-calls:
   - 25 × `STORAGE_WRITE_GAS_COST` (44,970) ≈ 1,124,250 gas
   - 2 × `CALL_CONTRACT_GAS_COST` (91,560) ≈ 183,120 gas
   - Internal Sierra steps for the handler logic: ~98,000,000 gas (achievable with a loop or recursive call pattern)
   - Total: exceeds 100,000,000

2. From L1, call the StarkNet core contract's `sendMessageToL2` targeting this contract, locking ETH in the bridge.

3. The sequencer picks up the message and calls `execute_l1_handler_transaction`.

4. At line 444, `consume_l1_to_l2_message` writes the message to the OS output — it is now permanently consumed.

5. At line 445–448, the handler is invoked with `remaining_gas = 100_000_000`. It runs out of gas; `non_reverting_select_execute_entry_point_func` returns with `is_reverted=1`. No token minting occurs.

6. The block is proven and submitted to L1. The L1 core contract sees the message as consumed. The ETH locked in the bridge is permanently inaccessible. [6](#0-5)

func execute_l1_handler_transaction{
    range_check_ptr,
    builtin_ptrs: BuiltinPointers*,
    contract_state_changes: DictAccess*,
    contract_class_changes: DictAccess*,
    outputs: OsCarriedOutputs*,
}(block_context: BlockContext*) {
    alloc_locals;

    %{ StartTx %}
    local is_reverted;
    %{ IsReverted %}
    // Skip the execution step for reverted transaction.
    if (is_reverted != FALSE) {
        %{ EndTx %}
        return ();
    }

    // TODO(Yoni): currently, the contract state is not fetched for reverted L1 handlers.
    //   Once block hash is supported, we should fetch the contract state for them as well.
    local entry_point_selector;
    %{ TxEntryPointSelector %}
    let (local tx_execution_context: ExecutionContext*) = get_invoke_tx_execution_context(
        block_context=block_context,
        entry_point_type=ENTRY_POINT_TYPE_L1_HANDLER,
        entry_point_selector=entry_point_selector,
    );
    local tx_execution_info: ExecutionInfo* = tx_execution_context.execution_info;

    local nonce;
    %{ LoadTxNonceL1Handler %}
    local chain_id = block_context.os_global_context.starknet_os_config.chain_id;

    let pedersen_ptr = builtin_ptrs.selectable.pedersen;
    with pedersen_ptr {
        let transaction_hash = compute_l1_handler_transaction_hash(
            execution_context=tx_execution_context, chain_id=chain_id, nonce=nonce
        );
    }
    update_pedersen_in_builtin_ptrs(pedersen_ptr=pedersen_ptr);

    %{ AssertTransactionHash %}

    // Write the transaction info and complete the ExecutionInfo struct.
    tempvar tx_info = tx_execution_info.tx_info;
    assert [tx_info] = TxInfo(
        version=L1_HANDLER_VERSION,
        account_contract_address=tx_execution_info.contract_address,
        max_fee=0,
        signature_start=cast(0, felt*),
        signature_end=cast(0, felt*),
        transaction_hash=transaction_hash,
        chain_id=chain_id,
        nonce=nonce,
        resource_bounds_start=cast(0, ResourceBounds*),
        resource_bounds_end=cast(0, ResourceBounds*),
        tip=0,
        paymaster_data_start=cast(0, felt*),
        paymaster_data_end=cast(0, felt*),
        nonce_data_availability_mode=0,
        fee_data_availability_mode=0,
        account_deployment_data_start=cast(0, felt*),
        account_deployment_data_end=cast(0, felt*),
        proof_facts_start=cast(0, felt*),
        proof_facts_end=cast(0, felt*),
    );
    fill_deprecated_tx_info(tx_info=tx_info, dst=tx_execution_context.deprecated_tx_info);
    assert_deprecated_tx_fields_consistency(tx_info=tx_info);

    // Consume L1-to-L2 message.
    consume_l1_to_l2_message(execution_context=tx_execution_context, nonce=nonce);
    let remaining_gas = L1_HANDLER_L2_GAS_MAX_AMOUNT;
    non_reverting_select_execute_entry_point_func{remaining_gas=remaining_gas}(
        block_context=block_context, execution_context=tx_execution_context
    );

    %{ EndTx %}
    return ();
}
