### Title
Unchecked Arithmetic Overflow in `denormalize_amount` Corrupts Bridge Token Accounting - (File: `near/omni-bridge/src/lib.rs`)

### Summary

The `denormalize_amount` function in the NEAR omni-bridge contract performs two unchecked arithmetic operations — a `u8` subtraction and a `u128` multiplication — without any overflow/underflow guards. This is the direct analog of the `weightedAverage` overflow class from the report. When triggered, it silently produces a wrapped (incorrect) token amount that is then minted or transferred to the recipient, breaking bridge collateralization.

### Finding Description

`denormalize_amount` is defined as:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    amount * (10_u128.pow(diff_decimals))
}
``` [1](#0-0) 

Two unchecked operations exist:

**1. `u8` subtraction underflow** — `decimals.origin_decimals - decimals.decimals` is a plain `u8` subtraction. If `decimals > origin_decimals` (e.g., due to a token registered with inverted values), this wraps to a large value like `244`, making `diff_decimals = 244`. In Rust release builds, integer overflow wraps silently.

**2. `u128` multiplication overflow** — `amount * (10_u128.pow(diff_decimals))` has no overflow check. `u128::MAX ≈ 3.4 × 10^38`. If `diff_decimals ≥ 39`, then `10_u128.pow(diff_decimals)` itself overflows before the multiplication. Even with `diff_decimals = 18` (a realistic NEAR-to-EVM decimal gap), any `amount > 3.4 × 10^20` causes overflow.

The `Decimals` struct stores both values as `u8` with no invariant enforcement: [2](#0-1) 

`denormalize_amount` is called in three critical bridge paths:

- **`fin_transfer_callback`** (line 725): scales the proof amount before storing the `TransferMessage` and minting/transferring tokens to the NEAR recipient.
- **`fast_fin_transfer`** (line 771): scales the fast-transfer amount before validation and token dispatch.
- **`claim_fee_callback`** (line 1122): scales the finalized amount to compute the fee paid to the relayer. [3](#0-2) [4](#0-3) [5](#0-4) 

`normalize_amount` has the identical unchecked subtraction: [6](#0-5) 

### Impact Explanation

If `denormalize_amount` overflows:

- In `fin_transfer_callback`, the `transfer_message.amount` is set to a wrapped (much smaller) value. The recipient on NEAR receives far fewer tokens than they sent on the source chain. The difference remains locked/burned on the source chain with no recovery path — **permanent loss of user funds**.
- In `claim_fee_callback`, `fee = transfer_message.amount.0 - denormalized_amount`. If `denormalized_amount` wraps to a small value, the computed fee is inflated, allowing the relayer to claim tokens that belong to the user.
- In `fast_fin_transfer`, the validation `denormalized_amount == amount.0 + denormalized_fee.fee.0` uses the overflowed value, potentially allowing a bypass of the amount integrity check.

This matches the allowed impact: **"Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."**

### Likelihood Explanation

The `decimals` and `origin_decimals` values are set during token registration via `add_token`, called from `bind_token_callback` (processing a `DeployTokenMessage` proof) and `deploy_token_internal`. The decimal values originate from on-chain events (`DeployToken`) emitted by the source-chain bridge contracts. [7](#0-6) 

No validation enforces `origin_decimals >= decimals` anywhere in `add_token` or its callers. A realistic overflow scenario: a token with `origin_decimals = 24` (NEAR) and `decimals = 6` (EVM), `diff = 18`. A user bridging `amount = 10^21` units (1 quadrillion tokens at 6 decimals — plausible for high-supply tokens) produces `10^21 × 10^18 = 10^39 > u128::MAX`, silently wrapping. The `amount` field is fully user-controlled via the source-chain `initTransfer` call.

### Recommendation

Replace both arithmetic operations with checked variants and add an explicit precondition:

```rust
fn denormalize_amount(amount: u128, decimals: Decimals) -> u128 {
    assert!(
        decimals.origin_decimals >= decimals.decimals,
        "origin_decimals must be >= decimals"
    );
    let diff_decimals: u32 = (decimals.origin_decimals - decimals.decimals).into();
    let multiplier = 10_u128
        .checked_pow(diff_decimals)
        .expect("decimal multiplier overflow");
    amount
        .checked_mul(multiplier)
        .expect("denormalize_amount overflow")
}
```

Apply the same fix to `normalize_amount`. Additionally, enforce `origin_decimals >= decimals` in `add_token` at registration time.

### Proof of Concept

1. Register a token with `origin_decimals = 24`, `decimals = 6` (diff = 18) via `bind_token` processing a `DeployTokenMessage` proof.
2. On the source chain, call `initTransfer` with `amount = 4 × 10^20` (400 trillion units at 6 decimals).
3. Submit the proof to NEAR `fin_transfer` → `fin_transfer_callback`.
4. `denormalize_amount(4 × 10^20, {decimals: 6, origin_decimals: 24})` computes `4 × 10^20 × 10^18 = 4 × 10^38 > u128::MAX ≈ 3.4 × 10^38`.
5. In Rust release mode, this wraps to `4 × 10^38 - 2^128 ≈ 6.5 × 10^37` — a value ~6× smaller than the correct result.
6. The recipient on NEAR receives tokens corresponding to the wrapped amount; the remaining value is irrecoverably lost.
