### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new class hash provided by the caller corresponds to a previously declared contract class. Any contract can replace its own class with an arbitrary, undeclared hash, rendering itself permanently non-functional and freezing any funds it holds. The missing check is explicitly acknowledged by a TODO comment in the code.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall. After deducting gas, it reads the new class hash from the request and directly updates the contract's state entry — without consulting `contract_class_changes` to verify that the class hash was ever declared:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The `contract_class_changes` dictionary — which maps declared class hashes to their compiled class hashes — is never consulted. The OS accepts any arbitrary felt value as the new class hash and commits it to state.

This is the direct analog of the original bug: just as `RocketDAONodeTrustedUpgrade._upgradeContract` had an incomplete hardcoded blacklist that could not prevent invalid state transitions, `execute_replace_class` has no whitelist check against declared classes, allowing an invalid class hash to be committed to state with no recourse.

The `execute_declare_transaction` function in `transaction_impls.cairo` correctly enforces that a class may only be declared once and that `compiled_class_hash` must be non-zero:

```cairo
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [2](#0-1) 

But `execute_replace_class` performs no equivalent lookup into `contract_class_changes`, leaving the validation gap open.

---

### Impact Explanation

If a contract calls `replace_class` with an undeclared class hash:

1. The OS updates the contract's class hash in `contract_state_changes` and commits it to the proven state.
2. Future calls to the contract attempt to look up the compiled class for the new hash.
3. Since the class was never declared, no compiled class exists for this hash.
4. All future calls to the contract fail permanently — there is no recovery path, as `replace_class` can only be called by the contract itself, which is now non-functional.
5. Any funds (tokens, ETH, etc.) held by the contract are permanently and irrecoverably frozen.

This matches the **Critical. Permanent freezing of funds** impact category.

---

### Likelihood Explanation

The attack is directly reachable by any unprivileged contract deployer:

- No privileged role, leaked key, or operator access is required.
- Any contract can call the `replace_class` syscall with an arbitrary felt value as the new class hash.
- The attacker fully controls the value passed to `replace_class`.
- The OS, as shown in `execute_replace_class`, performs no validation against declared classes.

A realistic scenario: an attacker deploys a contract that accepts user deposits, collects funds, then calls `replace_class(0x1)` (an undeclared hash). The OS accepts this, the state is proven and committed, and all deposited funds are permanently frozen.

---

### Recommendation

Resolve the existing TODO by adding a validation step in `execute_replace_class` that looks up the new class hash in `contract_class_changes` and verifies it has a non-zero compiled class hash. If the lookup fails, the syscall should return a failure response (analogous to how `execute_call_contract` returns `ERROR_INVALID_ARGUMENT` for invalid selectors). The `contract_class_changes` implicit argument must be threaded into `execute_replace_class` for this purpose.

---

### Proof of Concept

1. Attacker deploys contract `A` with a legitimate declared class hash; users deposit funds.
2. Attacker calls a function on contract `A` that internally invokes `replace_class(0xdeadbeef)`, where `0xdeadbeef` is an arbitrary value never passed through `execute_declare_transaction`.
3. `execute_replace_class` in the OS accepts this: gas is deducted, `contract_state_changes` is updated with `class_hash = 0xdeadbeef`, no check against `contract_class_changes` is performed.
4. The block is proven and the state is committed on-chain: contract `A` now has class hash `0xdeadbeef`.
5. Any future invoke transaction targeting contract `A` fails at class lookup — no compiled class exists for `0xdeadbeef`.
6. All funds in contract `A` are permanently frozen with no recovery path. [3](#0-2)

        // Not enough gas to execute the syscall.
