### Title
Missing Chain ID in Precommit Vote Message Digest Enables Cross-Chain Signature Replay - (File: crates/apollo_signature_manager/src/signature_manager.rs)

### Summary

`build_precommit_vote_message_digest` constructs the BFT consensus precommit vote signing payload from only a static domain separator and the raw block hash bytes, with no chain ID bound into the digest. Because the Starknet block hash specification (`STARKNET_BLOCK_HASH1`) also omits chain ID as a direct field, a validator's precommit signature produced on one Starknet deployment is structurally valid on any other deployment that arrives at the same block hash value.

### Finding Description

In `crates/apollo_signature_manager/src/signature_manager.rs`, `build_precommit_vote_message_digest` at line 129 constructs:

```
MessageDigest = blake2s( b"PRECOMMIT_VOTE" || block_hash.to_bytes_be() )
``` [1](#0-0) 

No chain ID is mixed in at any point. The `SignatureManager::sign_precommit_vote` method calls this function directly and the resulting `RawSignature` is broadcast as the validator's authoritative vote. [2](#0-1) 

The block hash that is signed is produced by `calculate_block_hash` in `crates/starknet_api/src/block_hash/block_hash_calculator.rs`. Its Poseidon preimage is:

```
block_hash_constant | block_number | state_root | sequencer | timestamp |
concat_counts | state_diff_commitment | tx_commitment | event_commitment |
receipt_commitment | gas_prices | starknet_version | 0 | parent_block_hash
``` [3](#0-2) 

Chain ID is absent from this list. The same formula is used in the Cairo OS: [4](#0-3) 

Transaction hashes do include chain ID (see `get_invoke_transaction_v1_hash`, etc.), so the transaction commitment field is indirectly chain-specific when transactions are present. However, this indirect binding fails for:

- **Empty blocks** – transaction commitment is the hash of an empty set, identical across all chains.
- **Blocks whose only transactions are L1 handler messages** – if the same L1 messages are processed on a forked chain, the transaction hashes (and thus the commitment) are identical.
- **Any block where all other fields (block number, sequencer, timestamp, gas prices, state diff, parent hash) happen to collide** across two deployments sharing a common history.

The `verify_precommit_vote_signature` public library function performs the symmetric check with the same chain-ID-free digest: [5](#0-4) 

The `SignatureManagerRequest::SignPrecommitVote` RPC path is wired and reachable: [6](#0-5) 

The consensus state machine currently carries `// TODO(Asmaa): verify the signature` and emits `RawSignature::default()` for self-votes, meaning full enforcement is not yet active. However, the signing path is live and the verification API is the canonical one that will be integrated. [7](#0-6) 

### Impact Explanation

When vote signature verification is enforced, a network observer who collects a quorum of precommit signatures for block H on chain A can replay them verbatim on chain B whenever the two chains share the same block hash for height H. The `verify_precommit_vote_signature` call will return `true` for the replayed signatures because the digest is identical. This satisfies the "High – Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload" impact criterion: the wrong chain's validators are bound as signers of a foreign block commitment, potentially driving chain B to a decision on a block that its own validators never voted for.

### Likelihood Explanation

Medium-low. The attack is easiest on a testnet that was forked from mainnet at a known height: both chains share the same parent hash chain up to the fork point, and an empty block produced immediately after the fork on both chains with the same sequencer address, timestamp, and gas prices will have an identical block hash. A validator key that participates on both networks (common in staging environments) produces a replayable signature with no additional effort.

### Recommendation

Bind chain ID into the message digest before the verification path is activated:

```rust
fn build_precommit_vote_message_digest(
    block_hash: BlockHash,
    chain_id: &ChainId,
) -> MessageDigest {
    let block_hash = block_hash.to_bytes_be();
    let chain_id_bytes = Felt::try_from(chain_id)
        .expect("chain_id to felt")
        .to_bytes_be();
    let mut message = Vec::with_capacity(
        PRECOMMIT_VOTE.len() + block_hash.len() + chain_id_bytes.len(),
    );
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash);
    message.extend_from_slice(&chain_id_bytes);
    MessageDigest(blake2s_to_felt(&message))
}
```

Propagate `chain_id` through `SignatureManager::sign_precommit_vote`, `verify_precommit_vote_signature`, and the `SignatureManagerRequest::SignPrecommitVote` variant. The same fix should be applied to `build_peer_identity_message_digest` for the same reason.

### Proof of Concept

1. Run two Starknet sequencer instances, chain A (`SN_MAIN`) and chain B (`SN_FORK`), both starting from the same genesis state.
2. Both chains produce an empty block at height 1 with identical sequencer address, timestamp, gas prices, and starknet version. The state diff is empty on both; the parent hash is the shared genesis hash. All inputs to `calculate_block_hash` are equal, so `block_hash_A == block_hash_B`.
3. On chain A, a validator calls `sign_precommit_vote(block_hash_A)` → `sig_A`.
4. An attacker observing chain A's P2P traffic captures `sig_A`.
5. The attacker injects `sig_A` as a precommit vote for `block_hash_B` on chain B.
6. `verify_precommit_vote_signature(block_hash_B, sig_A, validator_pub_key)` computes `blake2s(b"PRECOMMIT_VOTE" || block_hash_B.to_bytes_be())` — identical to the digest signed on chain A — and returns `true`.
7. Chain B counts the replayed signature toward its quorum, binding chain A's validator as an authoritative signer of chain B's block commitment. [1](#0-0) [3](#0-2)

```text
func calculate_block_hash{poseidon_ptr: PoseidonBuiltin*}(
    block_info: BlockInfo*,
    header_commitments: BlockHeaderCommitments*,
    gas_prices_hash: felt,
    state_root: felt,
    previous_block_hash: felt,
    starknet_version: felt,
) -> felt {
    static_assert BlockInfo.SIZE == 3;
    static_assert BlockHeaderCommitments.SIZE == 5;

    let hash_state = hash_init();
    with hash_state {
        hash_update_single(BLOCK_HASH_VERSION);
        hash_update_single(block_info.block_number);
        hash_update_single(state_root);
        hash_update_single(block_info.sequencer_address);
        hash_update_single(block_info.block_timestamp);
        hash_update_single(header_commitments.packed_lengths);
        hash_update_single(header_commitments.state_diff_commitment);
        hash_update_single(header_commitments.transaction_commitment);
        hash_update_single(header_commitments.event_commitment);
        hash_update_single(header_commitments.receipt_commitment);
        hash_update_single(gas_prices_hash);
        hash_update_single(starknet_version);
        hash_update_single(0);
        hash_update_single(previous_block_hash);
    }

    let block_hash = hash_finalize(hash_state=hash_state);
    return block_hash;
}
