### Title
Missing Declared-Class Existence Check in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS does not verify that the new class hash supplied by the caller has been previously declared in `contract_class_changes`. An unprivileged contract can call `replace_class` with an arbitrary, undeclared class hash, permanently setting its own class hash to a value for which no compiled class exists. Any funds held by that contract become permanently frozen because the contract can never be executed again.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function accepts the caller-supplied `class_hash` from the syscall request and immediately writes it into `contract_state_changes` without checking whether that hash exists in `contract_class_changes` (i.e., whether it was ever declared):

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The in-code `TODO` comment explicitly acknowledges the missing check. The analogous guard that **is** present for class declarations — `prev_value=0` in `execute_declare_transaction` — enforces that a class may be declared only once:

```cairo
// Note that prev_value=0 enforces that a class may be declared only once.
assert_not_zero(compiled_class_hash);
dict_update{dict_ptr=contract_class_changes}(
    key=[class_hash_ptr], prev_value=0, new_value=compiled_class_hash
);
``` [2](#0-1) 

No equivalent guard exists in `execute_replace_class` to verify the new class hash is present in `contract_class_changes`. The same omission exists in the deprecated path: [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is overwritten with an undeclared value, the sequencer cannot produce a valid execution proof for any call to that contract (there is no compiled class to execute). The contract becomes permanently uncallable. Any ERC-20 balances, ETH, or other assets held by the contract are irrecoverably frozen. Because the class hash is stored in the persistent `contract_state_changes` dict and squashed into the state commitment, there is no protocol-level mechanism to undo the change.

---

### Likelihood Explanation

**Medium.**

Any unprivileged user can:
1. Deploy a contract (via `deploy_account` or the `deploy` syscall).
2. Include logic in that contract that calls `replace_class` with an arbitrary felt value (e.g., `0xdeadbeef`) that has never been declared and can never be declared.
3. Trigger that logic after other users have deposited funds into the contract (e.g., a malicious yield-farming or escrow contract).

No privileged access, leaked key, or network-level attack is required. The attacker only needs to be a normal contract deployer and transaction sender.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, assert that it exists in `contract_class_changes` (i.e., that it was previously declared). Concretely, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the returned compiled class hash is non-zero:

```cairo
// Verify the class has been declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

Apply the same fix to the deprecated `execute_replace_class` in `deprecated_execute_syscalls.cairo`.

---

### Proof of Concept

1. Attacker deploys contract `C` (a fake staking vault) that exposes:
   - `deposit()` — accepts ETH/ERC-20 from users.
   - `rug()` — calls the `replace_class` syscall with `class_hash = 0x1` (never declared).

2. Users call `deposit()`, locking funds inside `C`.

3. Attacker calls `rug()`. The OS executes `execute_replace_class`, writing `class_hash=0x1` into `contract_state_changes` for address `C` with no existence check.

4. The state commitment now records `C.class_hash = 0x1`. No compiled class for `0x1` exists or can ever exist.

5. Any subsequent transaction targeting `C` (withdraw, transfer, etc.) cannot be proven by the sequencer — the contract is permanently uncallable and all deposited funds are permanently frozen.

The root cause is solely in the OS Cairo code at: [4](#0-3)
