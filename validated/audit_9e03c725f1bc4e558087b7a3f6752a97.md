### Title
Missing Validation of `class_hash` Parameter in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in the StarkNet OS accepts any `class_hash` value from a contract's syscall request and commits it to state without verifying that the class hash corresponds to a declared contract class. An unprivileged contract deployer can exploit this to permanently freeze funds held in any contract by replacing its class with an undeclared or zero class hash, rendering the contract permanently uncallable. The OS code itself contains a `TODO` comment acknowledging this missing check.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function processes the `replace_class` syscall:

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
``` [1](#0-0) 

The function:
1. Reads `class_hash` directly from the syscall request (attacker-controlled).
2. Skips any check that the class hash is declared in `contract_class_changes` or the compiled class facts bundle.
3. Unconditionally writes the new (potentially invalid) class hash into `contract_state_changes` via `dict_update`.

The `TODO` comment at line 898 explicitly acknowledges the missing validation. No other OS code performs a post-hoc check on class hashes written into state entries.

By contrast, the `execute_deploy` path (also in `syscall_impls.cairo`) computes the contract address deterministically from the class hash and constructor calldata, and `deploy_contract.cairo` checks reserved addresses — but neither path validates that the class hash being stored is actually declared. [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once `replace_class` succeeds with an undeclared `class_hash` (e.g., `0` or any arbitrary felt), the committed state entry for that contract permanently holds an invalid class hash. Any subsequent call to that contract will fail at class lookup time (the OS cannot find a compiled class for the hash), and since the contract can no longer execute, it cannot call `replace_class` again to recover. All ERC-20 tokens, ETH, or other assets held in the contract's storage are permanently inaccessible — a textbook permanent fund freeze.

---

### Likelihood Explanation

**High.** The attack requires only:
1. Deploying a contract (unprivileged contract deployer role).
2. Having that contract issue a `replace_class` syscall with an undeclared class hash (e.g., `0x1` or any felt not present in `contract_class_changes`).

No privileged role, leaked key, or external dependency is needed. The OS enforces no guard on the class hash value. The `TODO` comment confirms the developers are aware the check is absent. Any contract that holds user funds and exposes a path (even indirectly) to `replace_class` with an unvalidated argument is exploitable.

---

### Recommendation

Before committing the new class hash to `contract_state_changes`, assert that the class hash exists in the `contract_class_changes` dictionary (i.e., it has been declared in the current or a prior block). Concretely, perform a `dict_read` on `contract_class_changes` for `class_hash` and assert the returned compiled class hash is non-zero:

```cairo
// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the existing pattern used in `execute_declare_transaction` where `assert_not_zero(compiled_class_hash)` is enforced before updating `contract_class_changes`. [4](#0-3) 

---

### Proof of Concept

1. **Attacker deploys a contract** (e.g., a vault contract) that accepts user deposits and exposes a `replace_class` call path.
2. Users deposit funds into the vault.
3. Attacker triggers the contract to call the `replace_class` syscall with `class_hash = 0` (or any undeclared felt).
4. The OS executes `execute_replace_class`:
   - `class_hash = 0` is read from the request.
   - The `TODO` check is absent; no validation occurs.
   - `dict_update` writes `StateEntry(class_hash=0, ...)` for the vault's address into `contract_state_changes`.
5. The block is proven and the state is committed on-chain with the vault's class hash set to `0`.
6. Any future call to the vault fails: the OS cannot find a compiled class for hash `0`.
7. All user funds in the vault's storage are permanently frozen — no withdrawal, no recovery. [5](#0-4)

    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
