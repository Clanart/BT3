### Title
Missing NEAR-origin chain special-case in `fin_transfer_send_tokens_callback` native fee distribution — (`near/omni-bridge/src/lib.rs`)

---

### Summary

`fin_transfer_send_tokens_callback` unconditionally calls `get_native_token_id(origin_chain)` and then `mint(...)` when `native_fee > 0`, without checking whether the origin chain is `ChainKind::Near`. The sibling function `send_fee_internal` (used in `claim_fee_callback`) correctly handles this case by sending actual NEAR tokens via `Promise::new(fee_recipient).transfer(...)`. The missing branch causes the callback to panic when a NEAR-native token is finalized back to NEAR with a non-zero native fee, corrupting fee accounting and preventing the relayer from receiving their native fee.

---

### Finding Description

Two code paths distribute native fees after a transfer is finalized on NEAR:

**`send_fee_internal`** (called from `claim_fee_callback`) — correctly handles all origin chains:

```rust
// near/omni-bridge/src/lib.rs lines 2662–2673
if origin_chain.is_utxo_chain() {
    env::panic_str(BridgeError::NativeFeeForUtxoChain...)
} else if origin_chain == ChainKind::Near {
    // Correct: send actual yoctoNEAR
    Promise::new(fee_recipient.clone())
        .transfer(NearToken::from_yoctonear(transfer_message.fee.native_fee.0))
        .detach();
} else {
    // For EVM/Solana: mint wrapped native token
    ext_token::ext(self.get_native_token_id(origin_chain))
        .mint(fee_recipient.clone(), transfer_message.fee.native_fee, None)
        .detach();
}
```

**`fin_transfer_send_tokens_callback`** (called from `fin_transfer` flow) — missing the NEAR-origin check:

```rust
// near/omni-bridge/src/lib.rs lines 1736–1743
if transfer_message.fee.native_fee.0 > 0 {
    let native_token_id = self.get_native_token_id(transfer_message.get_origin_chain());
    // BUG: no check for ChainKind::Near — panics or mints incorrectly
    ext_token::ext(native_token_id)
        .with_static_gas(MINT_TOKEN_GAS)
        .mint(fee_recipient.clone(), transfer_message.fee.native_fee, None)
        .detach();
}
```

The `transfer_message.get_origin_chain()` returns `ChainKind::Near` whenever the bridged token is a NEAR-native token (i.e., `transfer_message.token` is `OmniAddress::Near(...)`). This is the common case when a NEAR-native token (e.g., `usdc.near`) is bridged to EVM and then finalized back to NEAR. The fact that `send_fee_internal` explicitly avoids calling `get_native_token_id(ChainKind::Near)` confirms that calling it for NEAR chain is incorrect — NEAR has no "wrapped native token" to mint; the native fee is actual yoctoNEAR held by the bridge contract.

---

### Impact Explanation

When `fin_transfer_send_tokens_callback` panics (due to `get_native_token_id(ChainKind::Near)` panicking or the subsequent `mint` call failing on a non-mintable contract):

- All state changes within the callback are reverted by the NEAR runtime.
- The relayer's native fee is never distributed.
- The `FinTransferEvent` is never emitted.
- The transfer is already marked finalized (via `add_fin_transfer` in the prior receipt), so the user's tokens were already sent — the transfer cannot be retried.

This is **fee accounting corruption**: the relayer permanently loses their native fee for every NEAR-native token finalization with `native_fee > 0`. Over time, this makes relaying NEAR-native token returns economically unviable and breaks the bridge's fee incentive model.

---

### Likelihood Explanation

NEAR-native tokens (e.g., NEP-141 tokens like `usdc.near`, `wrap.near`) are the primary asset class on NEAR and are commonly bridged to EVM chains. Relayers routinely set `native_fee > 0` to cover gas costs. Any relayer finalizing a NEAR-native token transfer back to NEAR with a non-zero native fee will trigger this bug. The entry path is fully unprivileged — any relayer calling `fin_transfer` with a valid proof can reach this code.

---

### Recommendation

Apply the same three-way branch used in `send_fee_internal` to `fin_transfer_send_tokens_callback`:

```rust
if transfer_message.fee.native_fee.0 > 0 {
    let origin_chain = transfer_message.get_origin_chain();
    if origin_chain == ChainKind::Near {
        Promise::new(fee_recipient.clone())
            .transfer(NearToken::from_yoctonear(transfer_message.fee.native_fee.0))
            .detach();
    } else {
        let native_token_id = self.get_native_token_id(origin_chain);
        ext_token::ext(native_token_id)
            .with_static_gas(MINT_TOKEN_GAS)
            .mint(fee_recipient.clone(), transfer_message.fee.native_fee, None)
            .detach();
    }
}
```

---

### Proof of Concept

1. Deploy a NEAR-native token (e.g., `usdc.near`) and register it with the bridge.
2. Bridge `usdc.near` from NEAR to EVM (origin chain = NEAR, `transfer_message.token = OmniAddress::Near("usdc.near")`).
3. A relayer calls `fin_transfer` on NEAR to finalize the return transfer, including `native_fee > 0` in the fee field.
4. `fin_transfer_callback` → `process_fin_transfer_to_near` → `send_tokens` → `fin_transfer_send_tokens_callback`.
5. Inside `fin_transfer_send_tokens_callback`, `transfer_message.get_origin_chain()` returns `ChainKind::Near`.
6. `get_native_token_id(ChainKind::Near)` is called — this either panics (no native token registered for NEAR) or returns an account that does not support `mint` from the bridge.
7. The callback panics; the relayer's native fee is lost; `FinTransferEvent` is never emitted.
