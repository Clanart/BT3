### Title
Field Overflow in `compute_max_possible_fee` Enables Complete Fee Bypass — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `compute_max_possible_fee` function in `transaction_impls.cairo` computes the maximum chargeable fee as a sum of products of field elements, with no range-checking on the `max_amount` or `max_price_per_unit` inputs. A malicious transaction sender can craft `ResourceBounds` values that cause the entire sum to overflow the StarkNet field prime (p ≈ 2²⁵¹ + 17·2¹⁹² + 1), reducing `max_fee` to 0. When `max_fee == 0`, `charge_fee` returns immediately without executing the ERC-20 transfer, allowing the attacker to execute transactions with zero fee payment.

---

### Finding Description

The vulnerability class is identical to the external report: **unchecked field-element arithmetic allows a value that should be a large positive number to wrap around the field prime to zero**, bypassing a security-critical check.

**Root cause — `compute_max_possible_fee`:** [1](#0-0) 

```cairo
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    ...
    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```

All operands are raw Cairo `felt` values. Cairo arithmetic is modular over the field prime p. There is no `assert_nn_le` or range-check applied to `max_amount` or `max_price_per_unit` before this multiplication and addition. If the attacker chooses values such that the sum ≡ 0 (mod p), the function returns 0.

**Sink — `charge_fee` early-exit on `max_fee == 0`:** [2](#0-1) 

```cairo
func charge_fee(...) {
    ...
    let max_fee = compute_max_possible_fee(tx_info=tx_info);

    if (max_fee == 0) {
        return ();          // ← fee transfer is skipped entirely
    }
    ...
    assert_nn_le(calldata.amount.low, max_fee);   // only reached if max_fee != 0
    non_reverting_select_execute_entry_point_func(...);  // ERC-20 transfer
}
```

When `max_fee == 0`, the function returns before the `assert_nn_le` guard and before the ERC-20 `transfer` call. No fee token is moved from the user to the sequencer.

**No range-check on resource bounds in `fill_account_tx_info`:** [3](#0-2) 

The only array-length guard present is `assert_nn_le(signature_len, SIERRA_ARRAY_LEN_BOUND - 1)`. There is no corresponding check that `max_amount` fits in u64 or that `max_price_per_unit` fits in u128, so the OS accepts arbitrary field-element values for these fields.

---

### Impact Explanation

**Direct loss of funds (Critical).** The fee token (STRK) that the protocol requires to be transferred from the transaction sender to the sequencer is never transferred. The sequencer's fee revenue is eliminated for any transaction exploiting this path. Because the OS is the provable execution environment, a proof generated over such a block is valid and will be accepted by the L1 verifier, permanently recording a state transition in which the sequencer received no fee. Additionally, an attacker who can execute transactions for free can submit an unbounded volume of transactions, threatening network liveness.

---

### Likelihood Explanation

**Moderate.** The attack requires:

1. The attacker selects `ResourceBounds` values (which they fully control as the transaction sender) such that the field-element sum equals p.
2. The attacker signs the transaction with their account key — no privileged access is needed.
3. The transaction is submitted to the network. A gateway that validates resource bounds as u64/u128 would reject it; however, the OS itself imposes no such constraint, so a gateway that validates only field-element well-formedness (or a sequencer that bypasses gateway checks) would include it.
4. The resulting proof is valid and accepted by L1.

The field prime p for StarkNet is publicly known. Constructing a set of `ResourceBounds` whose products sum to p is elementary arithmetic. The attacker does not need to repeat any operation p times (unlike the OpenVM PoC); a single transaction with carefully chosen values suffices.

---

### Recommendation

Add explicit range checks on `max_amount` and `max_price_per_unit` before the fee computation, consistent with the StarkNet transaction specification (u64 and u128 respectively):

```cairo
// In fill_account_tx_info or compute_max_possible_fee:
assert_nn_le(l1_gas_bounds.max_amount, MAX_U64);
assert_nn_le(l1_gas_bounds.max_price_per_unit, MAX_U128);
// ... repeat for l2 and l1_data bounds ...
```

With inputs bounded to u64 × u128, the maximum product per term is 2¹⁹², and the sum of three such terms is at most 3 · 2¹⁹², which is far below the StarkNet prime (≈ 2²⁵¹), eliminating the overflow.

---

### Proof of Concept

Let p = StarkNet prime = 2²⁵¹ + 17·2¹⁹² + 1. Choose:

| Field | Value |
|---|---|
| `l1_gas_bounds.max_amount` | 1 |
| `l1_gas_bounds.max_price_per_unit` | p − k |
| `l2_gas_bounds.max_amount` | 1 |
| `l2_gas_bounds.max_price_per_unit` | k (tip = 0) |
| `l1_data_gas_bounds.max_amount` | 0 |

Then:

```
max_fee = 1·(p−k) + 1·k + 0 = p ≡ 0 (mod p)
```

`charge_fee` hits the `if (max_fee == 0) { return (); }` branch and exits without executing the ERC-20 transfer. The transaction executes fully; the sequencer receives 0 STRK. The OS produces a valid proof over this state transition, which is accepted by the L1 verifier. [4](#0-3) [5](#0-4)

```text
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
```
