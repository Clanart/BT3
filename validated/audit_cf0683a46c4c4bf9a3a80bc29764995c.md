### Title
Unchecked Arithmetic Overflow in `compute_max_possible_fee` Causes Unprovable Blocks - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

### Summary

`compute_max_possible_fee` computes the fee cap as a raw felt multiplication of `max_amount * max_price_per_unit` for each resource type. Because `max_price_per_unit` is only bounded to `[0, 2^128)` and `max_amount` to `[0, 2^64)`, the product can reach ~2^192. The result is then passed to `assert_nn_le(actual_fee, max_fee)`, which internally requires both `actual_fee` and `max_fee - actual_fee` to fit in `[0, 2^128)`. When `max_fee ≥ 2^129`, no valid `actual_fee` exists that satisfies both range checks simultaneously, causing the OS to abort with an assertion failure and making the block unprovable.

### Finding Description

`compute_max_possible_fee` is defined as:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
    (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
    l1_data_gas_bounds.max_price_per_unit;
``` [1](#0-0) 

The only bounds enforced on resource bounds come from `pack_resource_bounds`, called during hash computation:

```cairo
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
assert_nn(resource_bounds.max_price_per_unit);
``` [2](#0-1) 

`assert_nn(max_price_per_unit)` only guarantees `max_price_per_unit ∈ [0, 2^128)`. There is no upper bound on `max_price_per_unit` beyond the range-check width. With `max_amount = 2^64 - 1` and `max_price_per_unit = 2^66`, a single term already yields `max_fee ≈ 2^130`, far exceeding `2^128`.

The result is then used in `charge_fee`:

```cairo
local low_actual_fee;
%{ LoadActualFee %}
local calldata: TransferCallData = TransferCallData(
    recipient=block_context.block_info_for_execute.sequencer_address,
    amount=Uint256(low=low_actual_fee, high=0),
);
// Verify that the charged amount is not larger than the transaction's max_fee field.
assert_nn_le(calldata.amount.low, max_fee);
``` [3](#0-2) 

`assert_nn_le(a, b)` from Cairo's math library expands to:
1. `assert_nn(a)` → `a ∈ [0, 2^128)`
2. `assert_nn(b - a)` → `b - a ∈ [0, 2^128)`

When `max_fee ≥ 2^129`, no value of `actual_fee` satisfies both constraints simultaneously:
- If `actual_fee < 2^128` (required by check 1), then `max_fee - actual_fee > 2^129 - 2^128 = 2^128`, failing check 2.
- If `actual_fee ≥ 2^128`, check 1 fails directly.

The OS execution aborts unconditionally. This is not a graceful transaction revert — it is a hard assertion failure in the OS prover loop itself, making the entire block unprovable.

### Impact Explanation

When the OS Cairo program aborts due to the failed `assert_nn_le`, the STARK proof for the block cannot be generated. The sequencer must discard the block and re-sequence without the offending transaction. During this window, no new transactions can be confirmed. If an attacker can reliably inject such transactions faster than the sequencer can detect and filter them, this constitutes a sustained network halt.

**Impact: High — Network not being able to confirm new transactions (total network shutdown).**

### Likelihood Explanation

A V3 transaction with `max_price_per_unit = 2^66` (a value that is ≥ 0 and < 2^128, so it passes all protocol-level hash validation) is syntactically valid and will be accepted into the mempool. The sequencer's off-chain simulation may not replicate the exact `assert_nn_le` failure if it uses a higher-level fee validation path rather than running the full Cairo OS. An attacker who identifies this gap can repeatedly submit such transactions to stall block production.

### Recommendation

Add an explicit upper bound check on `max_price_per_unit` in `pack_resource_bounds` (or equivalently in `compute_max_possible_fee`) to ensure the computed `max_fee` fits within `[0, 2^128)`:

```cairo
assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 64 - 1);
```

Alternatively, use `Uint256` arithmetic throughout `compute_max_possible_fee` and `charge_fee` so that the fee cap is represented and compared as a 256-bit value, consistent with the `Uint256` type already used for the transfer `amount`.

### Proof of Concept

1. Craft a V3 `invoke` transaction with:
   - `L1_GAS: max_amount = 2^64 - 1`, `max_price_per_unit = 2^66`
   - `L2_GAS: max_amount = 0`, `max_price_per_unit = 0`
   - `L1_DATA_GAS: max_amount = 0`, `max_price_per_unit = 0`
2. `pack_resource_bounds` accepts this: `assert_nn_le(2^64-1, 2^64-1)` passes; `assert_nn(2^66)` passes (2^66 < 2^128).
3. Transaction hash is computed and signed normally.
4. `compute_max_possible_fee` returns `(2^64 - 1) * 2^66 ≈ 2^130`.
5. In `charge_fee`, `assert_nn_le(actual_fee, 2^130)` is called. For any `actual_fee ∈ [0, 2^128)`, `2^130 - actual_fee > 2^128`, so `assert_nn(2^130 - actual_fee)` fails.
6. The OS Cairo execution aborts. The block containing this transaction cannot be proven. [4](#0-3) [5](#0-4) [6](#0-5)

// Returns the maximum possible fee that can be charged for the transaction.
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}

func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
    assert_nn(resource_bounds.max_price_per_unit);
    return (resource_bounds.resource * 2 ** 64 + resource_bounds.max_amount) * 2 ** 128 +
