### Title
Unauthenticated `ProofManager::set_proof` Remote Endpoint Allows Proof-Verification Bypass, Causing Gateway to Accept Transactions with Unverified Proof Facts — (`File: crates/apollo_proof_manager/src/proof_manager.rs`)

---

### Summary

`ProofManager` exposes a `SetProof` operation through an unauthenticated gRPC remote server that binds to `0.0.0.0`. The shared proof-verification helper `run_proof_verification` unconditionally skips cryptographic verification whenever `contains_proof` returns `true`. An attacker who can reach the proof manager port can pre-populate it with an arbitrary (fake) proof keyed to attacker-chosen proof facts, causing the gateway and consensus converter to skip verification for any subsequent transaction that carries those proof facts. The gateway then admits a transaction whose proof was never cryptographically checked.

---

### Finding Description

**Root cause — `ProofManager::set_proof` stores without verifying**

`ProofManager::set_proof` writes any caller-supplied `(ProofFacts, Proof)` pair to the filesystem cache with no cryptographic check:

```rust
pub async fn set_proof(
    &self,
    proof_facts: ProofFacts,
    proof: Proof,
) -> Result<(), FsProofStorageError> {
    if self.contains_proof(proof_facts.clone()).await? {
        return Ok(());          // already present → silent no-op
    }
    let facts_hash = proof_facts.hash();
    self.proof_storage.set_proof(facts_hash, proof.clone()).await?;
    self.cache.insert(facts_hash, proof);
    Ok(())
}
``` [1](#0-0) 

The `handle_request` dispatcher in the communication layer forwards `ProofManagerRequest::SetProof` directly to this function with no additional guard:

```rust
ProofManagerRequest::SetProof(proof_facts, proof) => ProofManagerResponse::SetProof(
    self.set_proof(proof_facts, proof).await
        .map_err(|e| ProofManagerError::ProofStorage(e.to_string())),
),
``` [2](#0-1) 

**Verification-skip logic in `run_proof_verification`**

The shared helper used by both the gateway flow (`spawn_proof_verification`) and the consensus flow (`spawn_verify_and_store_proof`) short-circuits on a positive `contains_proof` result:

```rust
async fn run_proof_verification(
    proof_facts: ProofFacts,
    proof: Proof,
    proof_manager_client: SharedProofManagerClient,
) -> Result<bool, TransactionConverterError> {
    let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

    if contains_proof {
        return Ok(false);   // ← cryptographic verify_proof() is never called
    }
    // ... starknet_proof_verifier::verify_proof(proof_facts, proof) ...
    Ok(true)
}
``` [3](#0-2) 

**No authentication on the remote server**

The proof manager's remote server is a plain `RemoteComponentServer` with no TLS client-auth or token check, and the distributed/hybrid deployment configs bind it to `0.0.0.0`:

```json
"components.proof_manager.remote_server_config.bind_ip": "0.0.0.0"
``` [4](#0-3) 

The server type is:
```rust
pub type RemoteProofManagerServer =
    RemoteComponentServer<ProofManagerRequest, ProofManagerResponse>;
``` [5](#0-4) 

No middleware, no bearer token, no mTLS is applied before `handle_request` is reached.

**End-to-end exploit path**

1. Attacker picks any `ProofFacts` value that will pass `validate_proof_facts` (valid program hash from `allowed_virtual_os_program_hashes`, a real historical block number/hash pair, correct config hash).
2. Attacker computes `facts_hash = proof_facts.hash()` and calls `SetProof(proof_facts, <garbage_proof>)` directly on the proof manager's gRPC port.
3. `ProofManager::set_proof` stores the garbage proof without verification.
4. Attacker submits an Invoke V3 transaction embedding those same `proof_facts` (and any `proof` field — it is irrelevant).
5. Gateway calls `spawn_proof_verification` → `run_proof_verification` → `contains_proof` returns `true` → `verify_proof` is **never called** → task returns `Ok(false)`.
6. `store_proof_and_spawn_archiving` calls `set_proof` again, but the early-return guard fires because the entry already exists.
7. The transaction passes gateway admission and enters the mempool with an unverified proof.

The same skip fires in the consensus path via `spawn_verify_and_store_proof`. [6](#0-5) 

---

### Impact Explanation

The gateway admits a transaction whose proof was never cryptographically verified. The entire client-side proving security model depends on `verify_proof` being called before a proof-bearing transaction is accepted. Bypassing it means the sequencer accepts transactions that assert a virtual-OS execution result without any proof backing that assertion. This falls under:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The proof facts themselves are still validated against chain state (block hash, program hash, config hash) during blockifier execution, so the attacker cannot forge a block hash. However, the cryptographic guarantee that the proof facts correspond to a correctly executed virtual-OS run is entirely absent.

---

### Likelihood Explanation

In distributed and hybrid deployments the proof manager binds to `0.0.0.0` with no authentication. Any process that can reach the proof manager's TCP port — including other pods in the same Kubernetes namespace, a compromised co-tenant, or a misconfigured network policy — can call `SetProof` without credentials. The attack requires no special privilege beyond network reachability.

---

### Recommendation

1. **Verify before storing**: `ProofManager::set_proof` should call `starknet_proof_verifier::verify_proof` before persisting any proof, mirroring the check already present in `run_proof_verification`. This makes the storage endpoint self-defending regardless of who calls it.

2. **Remove `SetProof` from the public remote interface**: The `SetProof` operation should only be reachable through the local in-process channel. The remote `ProofManagerRequest` enum should expose only `GetProof` and `ContainsProof` to external callers.

3. **Add mTLS or token authentication** to the proof manager's remote server so that only the gateway and consensus converter can call it.

---

### Proof of Concept

```
# 1. Attacker constructs valid proof_facts (real block hash at block N-10, allowed program hash,
#    correct config hash) and serialises them.

# 2. Attacker sends a raw gRPC SetProof request to the proof manager port:
grpcurl -plaintext <proof_manager_ip>:<port> \
  ProofManager/SetProof \
  '{"proof_facts": "<hex_encoded_valid_proof_facts>", "proof": "deadbeef"}'

# 3. Proof manager stores the garbage proof keyed by hash(proof_facts).

# 4. Attacker submits an Invoke V3 transaction via the HTTP gateway with:
#    - proof_facts = the same valid proof_facts from step 1
#    - proof      = any bytes (e.g. empty or "deadbeef")

# 5. Gateway calls run_proof_verification:
#      contains_proof(proof_facts) → true   (attacker pre-stored it)
#      → returns Ok(false), verify_proof() never called

# 6. Transaction is admitted to the mempool and sequenced without a valid proof.
```

The `contains_proof` short-circuit is confirmed by the unit test that explicitly asserts verification is skipped when the proof is already present: [7](#0-6)
