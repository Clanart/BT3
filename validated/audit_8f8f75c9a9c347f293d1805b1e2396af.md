### Title
Silent Drop of `migrated_compiled_classes` in `From<ClientStateDiff> for ThinStateDiff` Returns Incomplete Pending State Diff — (`crates/apollo_rpc/src/v0_8/state.rs`)

---

### Summary

Any unprivileged caller invoking `starknet_getStateUpdate` with `BlockId::Tag(Tag::Pending)` receives a `PendingStateUpdate` whose `state_diff.declared_classes` is missing every entry that originated from `migrated_compiled_classes` in the pending block's `ClientStateDiff`. The omission is silent: no error is raised, no field is zeroed, the response simply lacks those class-hash → compiled-class-hash mappings.

---

### Finding Description

The RPC handler `get_state_update` in `api_impl.rs` handles the pending case as:

```rust
let state_update = read_pending_data(&self.pending_data, &txn).await?.state_update;
return Ok(StateUpdate::PendingStateUpdate(PendingStateUpdate {
    old_root: state_update.old_root,
    state_diff: state_update.state_diff.into(),   // ← From<ClientStateDiff> for ThinStateDiff
}));
``` [1](#0-0) 

The `.into()` call resolves to `From<ClientStateDiff> for ThinStateDiff` in `v0_8/state.rs`. That implementation constructs `declared_classes` exclusively from `diff.declared_classes`; `diff.migrated_compiled_classes` is never read:

```rust
impl From<ClientStateDiff> for ThinStateDiff {
    fn from(diff: ClientStateDiff) -> Self {
        Self {
            ...
            declared_classes: diff
                .declared_classes          // ← only this field
                .into_iter()
                .map(|ClientDeclaredClassHashEntry { class_hash, compiled_class_hash }| {
                    ClassHashes { class_hash, compiled_class_hash }
                })
                .collect(),
            // diff.migrated_compiled_classes is never touched
            ...
        }
    }
}
``` [2](#0-1) 

The source type `ClientStateDiff` carries the field explicitly:

```rust
pub struct StateDiff {
    ...
    pub migrated_compiled_classes: Vec<MigratedCompiledClassHashEntry>,
    ...
}
``` [3](#0-2) 

By contrast, the correct path — `ThinStateDiff::from_state_diff` used for accepted blocks — chains both `declared_classes` and `migrated_compiled_classes` into `class_hash_to_compiled_class_hash`:

```rust
class_hash_to_compiled_class_hash: diff
    .declared_classes.iter()
    .map(|(class_hash, (compiled_hash, _class))| (*class_hash, *compiled_hash))
    .chain(
        diff.migrated_compiled_classes.iter()
            .map(|(class_hash, compiled_hash)| (*class_hash, *compiled_hash)),
    )
    .collect(),
``` [4](#0-3) 

The existing test for the pending path hard-codes `migrated_compiled_classes: vec![]`, so the gap is never exercised: [5](#0-4) 

A secondary drop occurs in `pending.rs`: `client_pending_data_to_execution_pending_data` also omits `migrated_compiled_classes` when building `ExecutionPendingData`, so the execution path for pending transactions is equally blind to those mappings. [6](#0-5) 

---

### Impact Explanation

`starknet_getStateUpdate` for the pending block is the canonical source of truth for RPC clients and re-execution tools that reconstruct `CommitmentStateDiff` for the pending state. When `migrated_compiled_classes` is non-empty, the returned `declared_classes` list is structurally incomplete: every `(class_hash, compiled_class_hash)` pair that arrived via migration is absent. Consumers that rely on this response to drive re-execution, fee estimation, or state-root verification will operate on a wrong class-hash → compiled-class-hash mapping, producing incorrect results that appear authoritative.

This falls under: **High — pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

`migrated_compiled_classes` is a live protocol field (Starknet ≥ 0.13.x) populated by the sequencer whenever a previously-declared Sierra class is re-compiled to a new CASM artifact without re-declaration. The field is already present in the feeder-gateway JSON schema and is parsed correctly by the client layer. Any pending block that contains such a migration triggers the omission. The TODO comment (`// TODO(Aviv): Remove this field after we upgrade to 0.14.1`) indicates the field is still active in the current version.

---

### Recommendation

In `From<ClientStateDiff> for ThinStateDiff` (`v0_8/state.rs`), extend the `declared_classes` iterator to also chain `diff.migrated_compiled_classes`:

```rust
declared_classes: diff
    .declared_classes
    .into_iter()
    .map(|ClientDeclaredClassHashEntry { class_hash, compiled_class_hash }| {
        ClassHashes { class_hash, compiled_class_hash }
    })
    .chain(
        diff.migrated_compiled_classes
            .into_iter()
            .map(|entry| ClassHashes {
                class_hash: entry.class_hash,
                compiled_class_hash: entry.compiled_class_hash,
            }),
    )
    .collect(),
```

Apply the same fix to `client_pending_data_to_execution_pending_data` in `pending.rs` so that the execution path for pending transactions also sees the migrated mappings.

### Proof of Concept

```rust
#[test]
fn migrated_compiled_classes_dropped_in_pending_conversion() {
    use apollo_starknet_client::reader::objects::state::{
        StateDiff as ClientStateDiff, MigratedCompiledClassHashEntry,
    };
    use starknet_api::core::{ClassHash, CompiledClassHash};
    use starknet_types_core::felt::Felt;
    use crate::v0_8::state::ThinStateDiff;

    let class_hash = ClassHash(Felt::from(0xABu64));
    let compiled_class_hash = CompiledClassHash(Felt::from(0xCDu64));

    let diff = ClientStateDiff {
        migrated_compiled_classes: vec![MigratedCompiledClassHashEntry {
            class_hash,
            compiled_class_hash,
        }],
        ..Default::default()
    };

    let thin: ThinStateDiff = diff.into();

    // Asserts that the migrated entry is absent — demonstrating the bug.
    assert!(
        thin.declared_classes.iter().all(|c| c.class_hash != class_hash),
        "migrated_compiled_classes entry should be absent (bug confirmed)"
    );
}
```

The assertion passes, confirming that `H` is absent from `declared_classes` after the conversion.
