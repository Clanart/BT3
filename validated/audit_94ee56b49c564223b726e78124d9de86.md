### Title
Unverified `n_transactions` and `state_diff_length` in `concatenated_counts` During P2P Sync Block Ingestion Allows Wrong Block Hash Commitment to Be Stored - (File: crates/apollo_p2p_sync/src/client/header.rs)

### Summary

In `HeaderStreamBuilder::convert_sync_block_to_block_data`, the `n_transactions` and `state_diff_length` fields written into the stored `BlockHeader` are derived independently from the `SyncBlock` payload, while the `concatenated_counts` field (which encodes these same values into the block hash) is taken directly from the peer-supplied `BlockHeaderCommitments`. There is no cross-check that the two sources agree. A peer can supply a `concatenated_counts` that encodes different `n_transactions` or `state_diff_length` values than what the actual block data contains, causing the stored header to carry internally inconsistent metadata. This inconsistency propagates into the block hash commitment stored on disk and into any downstream consumer that reconstructs `concatenated_counts` from the stored header fields.

### Finding Description

`convert_sync_block_to_block_data` in `crates/apollo_p2p_sync/src/client/header.rs` constructs a `SignedBlockHeader` from a `SyncBlock` received over P2P:

```rust
// TODO(Shahak): Verify `n_transactions` and `state_diff_length` match values in
// concatenated_counts.
SignedBlockHeader {
    block_header: BlockHeader {
        ...
        state_diff_length: Some(sync_block.state_diff.len()),   // derived locally
        n_transactions: sync_block.account_transaction_hashes.len()
            + sync_block.l1_transaction_hashes.len(),           // derived locally
        n_events,                                               // extracted from concatenated_counts
        ...
    },
    ...
}
```

The `n_events` field is extracted from `header_commitments.concatenated_counts` (peer-supplied), while `n_transactions` and `state_diff_length` are computed from the actual block data. However, the `concatenated_counts` Felt itself — which encodes all three values plus the L1 DA mode — is stored verbatim in `BlockHeaderCommitments.concatenated_counts` and is later fed directly into `calculate_block_hash` as a single hash input.

The `concatenated_counts` field is defined as:

```
[transaction_count (64 bits) | event_count (64 bits) | state_diff_length (64 bits) | L1 DA mode (1 bit) | 0...]
```

A malicious peer can send a `SyncBlock` where:
- `account_transaction_hashes` has length `A` (so `n_transactions = A` is stored)
- `state_diff` has length `S` (so `state_diff_length = S` is stored)
- But `concatenated_counts` encodes `n_transactions = A'` and `state_diff_length = S'` where `A' ≠ A` or `S' ≠ S`

The stored `BlockHeader` will then have `n_transactions = A` and `state_diff_length = S`, but the `concatenated_counts` Felt (which is the actual input to the block hash) will encode `A'` and `S'`. The block hash computed from this header will be wrong relative to the true block contents.

The acknowledged TODO comment at line 140–141 confirms this check is missing:

```rust
// TODO(Shahak): Verify `n_transactions` and `state_diff_length` match values in
// concatenated_counts.
```

### Impact Explanation

The `concatenated_counts` Felt is chained directly into `calculate_block_hash` at position 6 of the Poseidon hash:

```rust
.chain(&block_commitments.concatenated_counts)
```

A wrong `concatenated_counts` produces a wrong block hash. This wrong block hash is:

1. **Stored in `apollo_storage`** as the authoritative block hash for that block number.
2. **Used as `previous_block_hash`** for the next block's hash computation, poisoning the entire chain of block hashes forward.
3. **Returned by RPC** (`starknet_getBlockWithTxHashes`, `starknet_getBlockWithReceipts`, etc.) as an authoritative value.
4. **Used by the state diff sync** (`StateDiffStreamBuilder`) which reads `state_diff_length` from the stored header to determine how many state diff chunks to accept — a tampered `state_diff_length` in `concatenated_counts` does not affect this check (since `state_diff_length` is stored separately), but the block hash commitment is still wrong.
5. **Used by the proof manager / SNOS path**: `check_proof_facts` in the OS validates `base_block_hash` against the stored block hash. A wrong stored block hash causes valid proof facts to be rejected or invalid ones to be accepted.

This matches the impact scope: **Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic** (wrong block hash stored and returned) and **RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value**.

### Likelihood Explanation

The attack requires a malicious P2P peer. Any node that syncs via P2P (rather than central sync) and connects to a malicious peer is vulnerable. The TODO comment in the code explicitly acknowledges the missing check, confirming the developers are aware the validation is absent. The attack is low-complexity: the peer only needs to craft a `concatenated_counts` Felt with different bit-packed values while keeping the actual transaction and state diff data consistent with the rest of the block.

### Recommendation

In `convert_sync_block_to_block_data`, after computing `n_transactions` and `state_diff_length` from the actual block data, extract the corresponding fields from `concatenated_counts` and verify they match:

```rust
let n_transactions_from_counts = extract_transaction_count_from_concatenated_counts(
    &header_commitments.concatenated_counts
);
let state_diff_length_from_counts = extract_state_diff_length_from_concatenated_counts(
    &header_commitments.concatenated_counts
);
let computed_n_transactions = sync_block.account_transaction_hashes.len()
    + sync_block.l1_transaction_hashes.len();
let computed_state_diff_length = sync_block.state_diff.len();

if n_transactions_from_counts != computed_n_transactions {
    return Err(BadPeerError::...);
}
if state_diff_length_from_counts != computed_state_diff_length {
    return Err(BadPeerError::...);
}
```

Analogous extraction functions for `n_transactions` and `state_diff_length` should be added alongside the existing `extract_event_count_from_concatenated_counts` in `crates/starknet_api/src/block_hash/block_hash_calculator.rs`.

### Proof of Concept

1. A malicious P2P peer sends a `SyncBlock` for block N with:
   - `account_transaction_hashes = [tx1, tx2]` (2 transactions)
   - `state_diff` with 5 entries
   - `block_header_commitments.concatenated_counts` encoding `n_transactions=999`, `state_diff_length=999`, `n_events=<correct>`, `l1_da_mode=<correct>`

2. `convert_sync_block_to_block_data` stores:
   - `n_transactions = 2` (correct, from actual data)
   - `state_diff_length = 5` (correct, from actual data)
   - `concatenated_counts = <encodes 999, 999, ...>` (wrong, from peer)

3. When `calculate_block_hash` is called with this header's `BlockHeaderCommitments`, it chains the tampered `concatenated_counts` Felt, producing a block hash that does not match the true block hash for block N.

4. This wrong hash is stored as the authoritative hash for block N in `apollo_storage`.

5. Block N+1's hash computation uses this wrong value as `previous_block_hash`, propagating the corruption.

6. Any SNOS proof fact referencing block N's hash will fail validation against the stored (wrong) hash, or a proof built against the wrong hash will be accepted as valid. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

// A single felt: [
//     transaction_count (64 bits) | event_count (64 bits) | state_diff_length (64 bits)
//     | L1 data availability mode: 0 for calldata, 1 for blob (1 bit) | 0 ...
// ].
pub fn concat_counts(
    transaction_count: usize,
    event_count: usize,
    state_diff_length: usize,
    l1_data_availability_mode: L1DataAvailabilityMode,
) -> Felt {
    let l1_data_availability_byte: u8 = match l1_data_availability_mode {
        L1DataAvailabilityMode::Calldata => 0,
        L1DataAvailabilityMode::Blob => 0b10000000,
    };
    let concat_bytes = [
        to_64_bits(transaction_count).as_slice(),
        to_64_bits(event_count).as_slice(),
        to_64_bits(state_diff_length).as_slice(),
        &[l1_data_availability_byte],
        &[0_u8; 7], // zero padding
    ]
    .concat();
    Felt::from_bytes_be_slice(concat_bytes.as_slice())
}

fn to_64_bits(num: usize) -> [u8; 8] {
    let sized_transaction_count: u64 = num.try_into().expect("Expect usize is at most 8 bytes");
    sized_transaction_count.to_be_bytes()
}

/// Extracts the event count from concatenated_counts.
pub fn extract_event_count_from_concatenated_counts(concatenated_counts: &Felt) -> usize {
    let bytes = concatenated_counts.to_bytes_be();
    // Extract bytes 8-15 (the event_count field)
    let event_count_bytes: [u8; 8] =
        bytes[8..16].try_into().expect("Felt should have at least 16 bytes");
    u64::from_be_bytes(event_count_bytes).try_into().expect("Expected event count to fit in usize")
}
