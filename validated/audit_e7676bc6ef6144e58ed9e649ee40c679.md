Audit Report

## Title
Unattributed Router ETH Balance Consumed by Any WETH Swap, Enabling Theft of Stranded User Funds — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` function in `PeripheryPayments.sol` uses the router's entire native ETH balance to settle WETH payments without attributing that ETH to the specific caller who deposited it. Any ETH left on the router from a prior `multicall{value: ...}` call where the user omitted `refundETH()` can be silently consumed by any subsequent WETH swap, allowing an attacker to receive a fully or partially subsidized swap at the victim's expense.

## Finding Description
The `pay()` function contains a WETH-specific branch that checks `address(this).balance` before pulling from the declared `payer`: [1](#0-0) 

When `nativeBalance >= value`, the router wraps and forwards its own ETH to the pool without ever calling `safeTransferFrom(payer, ...)`. The `payer` address stored in transient context is completely irrelevant to the actual payment in this branch.

ETH arrives on the router via `multicall{value: X}(...)` because `multicall` is `payable`: [2](#0-1) 

The `receive()` guard only blocks direct ETH transfers from non-WETH addresses; it does not block ETH attached to a `multicall` call: [3](#0-2) 

**Exploit flow:**
1. Victim calls `multicall{value: 1 ETH}([exactInputSingle(WETH→token1, amountIn=0.5 ETH)])` without appending `refundETH()`. The swap consumes 0.5 ETH; 0.5 ETH remains stranded on the router.
2. Attacker (with no ETH sent and no WETH allowance) calls `exactInputSingle(WETH→token1, amountIn=0.5 ETH)`.
3. `exactInputSingle` stores `msg.sender` (attacker) as payer and `WETH` as token in transient context, then calls the pool.
4. The pool calls back `metricOmmSwapCallback`, which dispatches to `_justPayCallback`: [4](#0-3) 

5. `pay(WETH, attacker, pool, 0.5e18)` is called. Since `address(this).balance (0.5 ETH) >= value (0.5e18)`, the router deposits the **victim's** 0.5 ETH as WETH and transfers it to the pool. The attacker receives token1 output and pays nothing.

The same logic applies to the partial-subsidy branch (`0 < nativeBalance < value`): the attacker pays only `value − nativeBalance`, with the difference coming from the victim's stranded ETH. [5](#0-4) 

## Impact Explanation
Direct loss of user principal. The victim's ETH is consumed by the attacker's swap. The attacker receives the full swap output without paying any input. The loss is bounded only by the amount of ETH stranded on the router, which can be arbitrarily large depending on the victim's transaction size. This constitutes a Critical/High direct loss of user funds, meeting Sherlock contest thresholds.

## Likelihood Explanation
The `multicall` pattern is the standard way to compose permit + swap + refund steps. Users who omit `refundETH()` — a common mistake, especially when integrating via SDK or copy-paste — leave ETH on the router. An attacker can monitor the router's ETH balance on-chain and immediately follow any transaction that leaves ETH behind. No special permissions, approvals, or privileged access are required; a plain `exactInputSingle` call with `tokenIn = WETH` suffices.

## Recommendation
Track ETH attribution per-caller in transient storage (analogous to how `TransientCallbackPool` tracks the payer and token). Only allow the ETH deposited by the current `msg.sender` to be used for their own WETH payment in the same transaction. Alternatively, remove the native-ETH-first logic from `pay()` entirely and require callers to wrap ETH themselves (e.g., via a dedicated `wrapETH` multicall step) before invoking the router.

## Proof of Concept
1. Deploy the router with a WETH address.
2. Victim calls `multicall{value: 1 ether}([abi.encodeCall(router.exactInputSingle, (params_weth_to_token1_amountIn_0_5_eth))])` — no `refundETH()` appended. Router now holds 0.5 ETH.
3. Attacker (no ETH, no WETH allowance) calls `router.exactInputSingle(params_weth_to_token1_amountIn_0_5_eth)` with `msg.value = 0`.
4. Pool callback fires; `pay(WETH, attacker, pool, 0.5e18)` is called; `address(this).balance == 0.5e18` satisfies the `nativeBalance >= value` branch; router wraps victim's ETH and sends WETH to pool.
5. Attacker receives token1 output. Victim's 0.5 ETH is gone. Attacker paid nothing.
