### Title
Deprecated (Cairo 0) Declared Classes Silently Omitted from State Diff Commitment in Block Builder — (`crates/blockifier/src/state/cached_state.rs`)

---

### Summary

`CommitmentStateDiff` has no `deprecated_declared_classes` field. The conversion `From<CommitmentStateDiff> for ThinStateDiff` unconditionally sets `deprecated_declared_classes: Vec::new()`. This `ThinStateDiff` is the sole input to `calculate_block_commitments` inside `BlockExecutionArtifacts::new`, so any Cairo 0 class declaration executed in a block is silently excluded from the `state_diff_commitment`, `state_diff_length`, and `concatenated_counts` fields of the block header, producing a wrong `ProposalCommitment` / partial block hash.

---

### Finding Description

**Root cause — structural omission in `CommitmentStateDiff`**

`CommitmentStateDiff` tracks four fields: `address_to_class_hash`, `address_to_nonce`, `storage_updates`, and `class_hash_to_compiled_class_hash`. It has no `deprecated_declared_classes` field. [1](#0-0) 

When the blockifier executes a DeclareV0 or DeclareV1 transaction, the declared class hash is written to `StateMaps.declared_contracts` (a `HashMap<ClassHash, bool>`). The conversion `CommitmentStateDiff::from(StateMaps)` drops `declared_contracts` entirely — only `compiled_class_hashes` (Cairo 1 only) is forwarded. [2](#0-1) 

**Propagation — hardcoded empty vector in `ThinStateDiff` conversion**

`From<CommitmentStateDiff> for ThinStateDiff` unconditionally sets `deprecated_declared_classes: Vec::new()`, with a TODO acknowledging the gap: [3](#0-2) 

**Commitment path — wrong hash, length, and counts**

`BlockExecutionArtifacts::new` passes this deficient `ThinStateDiff` directly to `calculate_block_commitments`: [4](#0-3) 

Inside `calculate_block_commitments`, `calculate_state_diff_hash` chains `deprecated_declared_classes` into the Poseidon hash: [5](#0-4) 

`ThinStateDiff::len()` counts `deprecated_declared_classes` for `state_diff_length` and `concatenated_counts`: [6](#0-5) 

`concatenated_counts` is computed from `state_diff.len()`: [7](#0-6) 

All three corrupted values (`state_diff_commitment`, `state_diff_length`, `concatenated_counts`) feed `PartialBlockHashComponents` and therefore the `ProposalCommitment` that consensus nodes agree on.

**Asymmetry with the RPC path**

The RPC execution path (`induced_state_diff`) correctly handles deprecated declared classes by accepting them as an explicit caller-supplied parameter: [8](#0-7) 

The batcher path has no equivalent mechanism.

**Second TODO confirming uncertainty**

`StarknetClientStateDiff::from(StateMaps)` also silently drops deprecated declared classes with an explicit unresolved TODO: [9](#0-8) 

---

### Impact Explanation

If a DeclareV0/V1 transaction is included in a block:

1. `state_diff_commitment` (Poseidon hash) is wrong — the deprecated class hash is not chained in.
2. `state_diff_length` is wrong — the deprecated class count is missing.
3. `concatenated_counts` is wrong — derived from the wrong length.
4. `partial_block_hash` / `ProposalCommitment` is wrong — derived from all of the above.

Any independent verifier (L1 verifier, SNOS, another sequencer node) that correctly includes the deprecated class in the state diff hash will compute a different block hash, causing consensus failure or acceptance of a block with a wrong commitment. This matches **Critical: Wrong state, class hash, or revert result from blockifier/syscall/execution logic for accepted input** and **Critical: Incorrect commitment construction with economic impact**.

---

### Likelihood Explanation

The code still handles `DeclareV0` and `DeclareV1` in the execution path: [10](#0-9) 

The TODO comment `// TODO(Aviv): Verify that we ignore those fields in purpose.` in `StarknetClientStateDiff` confirms the omission is not definitively intentional. If the gateway admits a DeclareV0/V1 transaction, any user can trigger the wrong commitment by submitting one. Even if the gateway currently rejects them, the absence of an assertion in `BlockExecutionArtifacts::new` means a future gateway change or a direct batcher call could silently corrupt the commitment.

---

### Recommendation

**Option A (preferred):** Add `deprecated_declared_classes` to `CommitmentStateDiff`, populate it from `StateMaps.declared_contracts` (filtering for entries where `compiled_class_hashes` has no corresponding entry, i.e., Cairo 0 classes), and propagate it through `ThinStateDiff::from(CommitmentStateDiff)`.

**Option B:** Explicitly reject DeclareV0/V1 at the gateway and add a `debug_assert!(commitment_state_diff_has_no_deprecated_classes)` guard in `BlockExecutionArtifacts::new` so any regression is caught immediately.

Remove the `// TODO(AlonH)` placeholder and the unresolved `// TODO(Aviv): Verify that we ignore those fields in purpose.` comment only after one of the above is implemented.

---

### Proof of Concept

1. Submit a DeclareV0 transaction with a valid Cairo 0 class to the sequencer.
2. The batcher includes it; the blockifier executes it, writing the class hash to `StateMaps.declared_contracts`.
3. `CommitmentStateDiff::from(state_maps)` drops `declared_contracts`; the class hash is absent.
4. `BlockExecutionArtifacts::new` calls `ThinStateDiff::from(commitment_state_diff)` → `deprecated_declared_classes = []`.
5. `calculate_block_commitments` computes `state_diff_commitment = Poseidon("STARKNET_STATE_DIFF0", ..., 0 deprecated classes, ...)` — missing the declared class hash.
6. An independent verifier computes `state_diff_commitment = Poseidon("STARKNET_STATE_DIFF0", ..., 1 deprecated class, class_hash, ...)` — a different value.
7. The `ProposalCommitment` stored and broadcast by the sequencer does not match the verifier's result, breaking the block hash invariant.
