### Title
Stale `fee_actual` in `finalize_decision` Causes Wrong `next_l2_gas_price` Stored in Block Header and Cende Blob — (`crates/apollo_consensus_orchestrator/src/sequencer_consensus_context.rs`)

---

### Summary

In `finalize_decision`, `update_l2_gas_price` (which computes and stores the next block's L2 gas price) is called **before** `record_fee_proposal` (which inserts the current block's `fee_proposal_fri` into `fee_proposals_window`). Because `fee_actual` — the EIP-1559-style floor used in the gas price formula — is derived as the median of `fee_proposals_window`, the current block's proposal is excluded from the computation. The resulting `next_l2_gas_price` is written into the `BlockHeaderWithoutHash` sent to state sync and into the cende blob's `FeeMarketInfo`, and is also adopted as `self.l2_gas_price` for the next block. Every block is affected.

---

### Finding Description

The analog to the ERC1155 "use-before-assign" bug is exact: in the original report, `saleDetails.amount` was overwritten with `quantityToFill` before the proportional price was computed, so the divisor was already the new value. Here, `self.l2_gas_price` is overwritten with the newly computed next-block price before the current block's `fee_proposal_fri` is inserted into the window that feeds the computation.

**Ordering in `finalize_decision`:**

```
line 517: self.update_l2_gas_price(height, l2_gas_used);
              └─ calls calculate_next_l2_gas_price(height, l2_gas_used)
                     └─ calls compute_fee_actual(&self.fee_proposals_window, height, window_size)
                            // window = [height-window_size .. height-1]  ← current block ABSENT
line 518: self.record_fee_proposal(height, init.fee_proposal_fri);
              // NOW inserts height → fee_proposal_fri into the window
``` [1](#0-0) 

`prune_fee_proposals_window` is called at `set_height_and_round` for the new height, so at the moment `update_l2_gas_price` runs the window holds exactly `window_size` entries covering `[height − window_size, height)` — the current block's proposal is not yet present. [2](#0-1) 

`compute_fee_actual` therefore computes the median over blocks `[height − window_size, height − 1]` instead of the correct `[height − window_size + 1, height]`. The oldest block's proposal is included and the current block's proposal is excluded. [3](#0-2) 

The corrupted `self.l2_gas_price` is then written to two authoritative outputs:

1. **State-sync block header** — `next_l2_gas_price: self.l2_gas_price` in `update_state_sync_with_new_block`, which persists to storage and is returned by RPC. [4](#0-3) 

2. **Cende blob** — `FeeMarketInfo { l2_gas_consumed: l2_gas_used, next_l2_gas_price: self.l2_gas_price }` forwarded to the centralized recorder. [5](#0-4) 

When a node syncs, it reads `next_l2_gas_price` from the stored header and adopts it as `self.l2_gas_price` for the next block: [6](#0-5) 

So both the live proposer and any syncing node use the same wrong gas price for the next block.

---

### Impact Explanation

Every finalized block stores a `next_l2_gas_price` that is computed with a `fee_actual` one block stale. The stored value is authoritative: it is returned verbatim by the RPC `starknet_getBlockWithTxHashes` / `starknet_getBlockWithReceipts` response fields and is consumed by syncing nodes as the gas price for the following block. Transactions whose `max_price_per_unit` falls between the wrong (lower) and correct (higher) gas price are incorrectly accepted or rejected, producing wrong fee accounting and wrong admission decisions. The cende blob carries the same wrong `next_l2_gas_price`, so the centralized pipeline and any downstream proof inputs that consume `fee_market_info` receive a wrong value.

Matches: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value** and **Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact**.

---

### Likelihood Explanation

The bug fires on every block finalization without any special conditions. No privileged access is required; the trigger is the normal consensus `decision_reached` path. The magnitude of the error depends on how much the current block's `fee_proposal_fri` differs from the oldest block being displaced from the window, which can be significant during fee-market transitions.

---

### Recommendation

Swap the two lines so the current block's proposal is recorded before the gas price is computed:

```rust
// finalize_decision — correct order
self.record_fee_proposal(height, init.fee_proposal_fri);   // insert first
self.update_l2_gas_price(height, l2_gas_used);             // then compute with full window
```

This mirrors the ERC1155 fix: assign the quantity after computing the proportional price, not before.

---

### Proof of Concept

1. Let `window_size = 10`. After pruning at height N, `fee_proposals_window` = `{N-10: p0, N-9: p1, …, N-1: p9}`.
2. `finalize_decision` is called for height N with `fee_proposal_fri = P_N`.
3. `update_l2_gas_price(N, gas_used)` runs. `compute_fee_actual` sees the window `{N-10..N-1}` — 10 entries, none for N. `fee_actual = median(p0..p9)`.
4. `record_fee_proposal(N, P_N)` runs. Window becomes `{N-10..N}` — 11 entries.
5. The correct `fee_actual` for computing block N+1's gas price is `median(p1..p9, P_N)` (the 10 most recent proposals including N). The stored `next_l2_gas_price` uses `median(p0..p9)` instead — it includes the stale `p0` and excludes the current `P_N`.
6. If `P_N` is high (e.g., proposer is pushing the fee up) and `p0` is low (old block), the stored `next_l2_gas_price` is lower than correct, causing the next block to accept transactions that should be rejected at the correct price. [7](#0-6) [8](#0-7)

pub fn calculate_next_l2_gas_price_for_fin(
    current_l2_gas_price: GasPrice,
    height: BlockNumber,
    l2_gas_used: GasAmount,
    override_l2_gas_price_fri: Option<u128>,
    min_l2_gas_price_per_height: &[PricePerHeight],
    fee_actual: Option<GasPrice>,
) -> GasPrice {
    if let Some(override_value) = override_l2_gas_price_fri {
        info!(
            "L2 gas price ({}) is not updated, remains on override value of {override_value} fri",
            current_l2_gas_price.0
        );
        return GasPrice(override_value);
    }
    let gas_target = VersionedConstants::latest_constants().gas_target;
    let config_min = get_min_gas_price_for_height(height, min_l2_gas_price_per_height);
    let effective_min = match fee_actual {
        Some(fa) => GasPrice(max(config_min.0, fa.0)),
        None => config_min,
    };
    calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
}
