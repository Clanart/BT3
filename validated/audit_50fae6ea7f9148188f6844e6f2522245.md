Audit Report

## Title
Partial-ETH branch in `pay()` silently consumes stranded router ETH to subsidize subsequent WETH swaps — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` helper in `PeripheryPayments.sol` contains a middle branch (lines 78–81) that fires when `token == WETH`, `payer != address(this)`, and `0 < address(this).balance < value`. This branch unconditionally wraps and spends the router's entire ETH balance as a partial payment before pulling the remainder from the actual payer's allowance. Because the router holds no per-user ETH attribution, any ETH stranded by a prior user (e.g., from a `msg.value` overpayment without a subsequent `refundETH()` call) is silently consumed by the next WETH swap, causing direct, irreversible loss of principal to the original depositor.

## Finding Description
The three-branch structure of `pay()` is:

```
nativeBalance >= value  →  wrap value from ETH, pull nothing from payer        (L75–77)
nativeBalance > 0       →  wrap nativeBalance, pull (value − nativeBalance)    (L78–81) ← root cause
nativeBalance == 0      →  pull all from payer                                 (L82–84)
``` [1](#0-0) 

The middle branch is the root cause. It drains the router's full ETH balance as a partial payment without verifying that the ETH belongs to the current payer. The router's `receive()` guard only restricts ETH from non-WETH senders during unwrap; it does not prevent ETH from accumulating via `msg.value` on any `payable` entry point. [2](#0-1) 

The callback chain that reaches `pay()`:

1. `exactInputSingle` stores `msg.sender` as payer in transient storage via `_setNextCallbackContext`. [3](#0-2) 

2. The pool calls back `metricOmmSwapCallback`, which dispatches to `_justPayCallback`. [4](#0-3) 

3. `_justPayCallback` calls `pay(tokenToPay, _getPayer(), msg.sender, amount)`. [5](#0-4) 

4. Inside `pay()`, when `token == WETH` and `0 < nativeBalance < value`, the partial branch fires, consuming all router ETH before touching the payer's allowance. [6](#0-5) 

The same path is reachable through `exactInput`, `exactOutputSingle`, `exactOutput`, and `multicall`, all of which are `payable` and can leave ETH residue on the router.

## Impact Explanation
User A sends `msg.value = 200e18` to `exactInputSingle` with `amountIn = 100e18` WETH. The callback wraps 100e18 ETH; 100e18 ETH remains on the router. If User A omits `refundETH()`, that 100e18 ETH is stranded. User B then calls `exactInputSingle` with `tokenIn = WETH`, `amountIn = 150e18`. In the callback, `nativeBalance = 100e18 > 0` and `< 150e18`, so the partial branch fires: 100e18 ETH is wrapped and sent to the pool, and only 50e18 WETH is pulled from User B's allowance. User B receives the full swap output while paying 100e18 less than owed. User A's 100e18 ETH is permanently lost. The pool is made whole (no insolvency), but User A suffers a direct, irreversible loss of principal. This is a **High** severity direct loss of user funds, reachable by any unprivileged caller with no special access.

## Likelihood Explanation
- `refundETH()` is not enforced by the router and must be composed manually into a multicall; omitting it is a common mistake.
- Any `payable` entry point can leave ETH residue on the router.
- The exploit requires no privileged access: any address can call `exactInputSingle` with `tokenIn = WETH` to trigger the partial-ETH branch.
- An attacker can monitor the router's ETH balance on-chain and time the call immediately after a victim's transaction, making this repeatable and deterministic.

## Recommendation
Replace the partial-ETH branch with the Uniswap v3 pattern: only use router ETH when it fully covers the payment; otherwise fall through to `safeTransferFrom`:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

This eliminates cross-user ETH consumption while preserving the intended "pay with native ETH" convenience for users who send exactly the right `msg.value`.

## Proof of Concept

```
State before:
  router.balance = 0
  User A WETH balance = 1000e18
  User B WETH balance = 1000e18, allowance to router = 1000e18

Step 1 – User A strands ETH:
  User A calls exactInputSingle{value: 200e18}(
      pool, tokenIn=WETH, amountIn=100e18, ...
  )
  Callback: pay(WETH, UserA, pool, 100e18)
    nativeBalance=200e18 >= value=100e18 → wraps 100e18 ETH, sends to pool
  After swap: router.balance = 100e18  (User A omits refundETH)

Step 2 – User B exploits:
  User B calls exactInputSingle(
      pool, tokenIn=WETH, amountIn=150e18, ...
  )
  Callback: pay(WETH, UserB, pool, 150e18)
    nativeBalance=100e18 > 0, < 150e18 → partial branch fires (L78–81):
      wraps 100e18 ETH → sends to pool
      safeTransferFrom(UserB, pool, 50e18)  ← only 50e18 pulled from User B

Result:
  User A lost 100e18 ETH (stranded, now consumed by User B's swap)
  User B paid 50e18 WETH instead of 150e18 WETH
  Pool received correct 150e18 WETH (no pool loss)
```
