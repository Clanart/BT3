### Title
Missing Upper-Bound Validation on `max_price_per_unit` Causes `charge_fee` Assertion Failure Leading to Block-Proving Halt — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`pack_resource_bounds` validates `max_price_per_unit` only with `assert_nn` (lower-bound only, allowing values up to `2**128 - 1`), while `compute_max_possible_fee` multiplies it by `max_amount` (up to `2**64 - 1`), producing a product up to `~2**192`. The subsequent `assert_nn_le(calldata.amount.low, max_fee)` in `charge_fee` uses Cairo's range-check builtin, which requires both operands and their difference to fit in `[0, 2**128 - 1]`. When `max_fee > 2**128 - 1`, this assertion unconditionally fails regardless of the actual fee value, aborting OS execution and preventing block proving.

---

### Finding Description

**Step 1 — Validation in `pack_resource_bounds` (transaction hash path):**

`pack_resource_bounds` is called from `hash_fee_fields` during transaction hash computation:

```cairo
func pack_resource_bounds{range_check_ptr}(resource_bounds: ResourceBounds) -> felt {
    assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // max_amount ≤ 2^64 - 1
    assert_nn(resource_bounds.max_price_per_unit);            // max_price_per_unit ≥ 0 ONLY
    ...
}
``` [1](#0-0) 

`assert_nn` in Cairo uses the range-check builtin, which constrains the value to `[0, 2**128 - 1]`. There is **no upper-bound** check on `max_price_per_unit`. A user may legally supply `max_price_per_unit = 2**128 - 1`.

**Step 2 — Fee computation in `compute_max_possible_fee`:**

```cairo
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    ...
    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
``` [2](#0-1) 

With `max_amount = 2**64 - 1` and `max_price_per_unit = 2**128 - 1`, the product is `(2**64 - 1) × (2**128 - 1) ≈ 2**192`. The sum of three such terms can reach `~3 × 2**192`, which is a valid field element (< STARK prime ~2**251) but **far exceeds `2**128 - 1`**.

**Step 3 — Assertion failure in `charge_fee`:**

```cairo
assert_nn_le(calldata.amount.low, max_fee);
``` [3](#0-2) 

`assert_nn_le(a, b)` is implemented as:
1. `assert_nn(a)` — checks `a ∈ [0, 2**128 - 1]`
2. `assert_nn(b - a)` — checks `b - a ∈ [0, 2**128 - 1]`

`calldata.amount.low` is the low 128 bits of a `Uint256`, so it is always `< 2**128`. But when `max_fee ≈ 2**192`, the value `max_fee - calldata.amount.low ≈ 2**192` exceeds `2**128 - 1`, causing the range-check builtin to fail. This is **not a transaction-level revert** — it is an OS-level assertion failure that aborts the entire Cairo execution trace, making the block unprovable.

**The inconsistency:** `pack_resource_bounds` (called during hash validation) accepts `max_price_per_unit` up to `2**128 - 1`, but `charge_fee` (called after execution) cannot handle the resulting `max_fee` when it exceeds `2**128 - 1`. The transaction passes hash validation but causes OS failure at fee-charging time — a direct analog to the original report's encoding inconsistency. [4](#0-3) 

---

### Impact Explanation

When the OS execution aborts inside `charge_fee`, the prover cannot generate a validity proof for the block. The sequencer, having already ordered and executed the block, cannot retroactively remove the transaction if it has committed to the block state. This results in the network being unable to confirm new transactions — a total network shutdown matching the **High** impact category.

---

### Likelihood Explanation

Any unprivileged transaction sender can craft a V3 transaction with:
- `max_amount = 2` (for any resource)
- `max_price_per_unit = 2**127`

This yields `max_fee = 2 × 2**127 = 2**128 > 2**128 - 1`, triggering the failure. The values pass all existing OS-level validation (`assert_nn_le(max_amount, 2**64 - 1)` and `assert_nn(max_price_per_unit)`). If the sequencer's off-chain mempool validation does not independently enforce `max_amount × max_price_per_unit < 2**128`, the transaction will be included in a block and cause OS failure.

---

### Recommendation

Add an explicit upper-bound check on `max_price_per_unit` in `pack_resource_bounds` to ensure that `max_amount × max_price_per_unit` cannot exceed `2**128 - 1`. For example:

```cairo
assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 64 - 1);
```

This mirrors the bound already applied to `max_amount` and ensures `max_fee` always fits within the range-check domain. Alternatively, rewrite `charge_fee` to use `Uint256` arithmetic for the fee comparison, avoiding the range-check overflow entirely. [5](#0-4) 

---

### Proof of Concept

1. Attacker submits a V3 `invoke` transaction with:
   - `l1_gas_bounds.max_amount = 2`
   - `l1_gas_bounds.max_price_per_unit = 2**127`
   - Other resource bounds set to 0

2. OS computes transaction hash: `pack_resource_bounds` runs `assert_nn(2**127)` — passes (value is in `[0, 2**128 - 1]`). Transaction hash is accepted.

3. Transaction executes normally.

4. OS calls `charge_fee` → `compute_max_possible_fee` returns `2 × 2**127 = 2**128`.

5. OS executes `assert_nn_le(actual_fee_low, 2**128)`:
   - Step 1: `assert_nn(actual_fee_low)` — passes (actual fee is a normal value < 2**128)
   - Step 2: `assert_nn(2**128 - actual_fee_low)` — **FAILS** because `2**128 - actual_fee_low ≥ 2**128 > 2**128 - 1`

6. OS execution aborts. Block cannot be proven. Network halts. [6](#0-5)

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
