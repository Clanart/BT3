Looking at the production code carefully, I need to trace the full validation path in `set_state_header` and determine whether the `shard_id` parameter is ever cross-checked against `chunk.shard_id()`.

---

## Validation Trace in `set_state_header`

**Step 1-2 — `validate_chunk_proofs`:** [1](#0-0) 

This function validates internal chunk consistency (header hash, tx root, receipts root). It receives only `chunk` and `epoch_manager` — the caller-supplied `shard_id` is **never passed in** and `chunk.shard_id()` is **never compared** to anything external.

**Step 3a — Merkle inclusion in `chunk_headers_root`:** [2](#0-1) 

This verifies `ChunkHashHeight(chunk.chunk_hash(), chunk.height_included())` is in the Merkle tree. The `chunk_headers_root` covers **all shards** in the block. The proof is position-specific (encodes which leaf), but the code never checks that the leaf position corresponds to `shard_id`. A valid chunk for shard_A at index 0 passes this check even when `shard_id = shard_B`.

**Step 3b — `prev_chunk_header` inclusion:** [3](#0-2) 

The attacker can supply shard_B's `prev_chunk_header` with its correct Merkle proof — this passes independently of which chunk is in the `chunk` field.

**Step 4e — Receipt proof validation:** [4](#0-3) 

Uses `ReceiptList(shard_id, receipts)` where `shard_id` is the caller-supplied parameter (shard_B). The attacker can supply valid receipt proofs for shard_B (all public chain data), making this pass.

**Step 5 — `validate_state_root_node`:** [5](#0-4) 

Validates `state_root_node` against `chunk_inner.prev_state_root()` — shard_A's state root. The attacker supplies shard_A's `state_root_node`, which is consistent. Passes.

**Storage — no shard_id cross-check:** [6](#0-5) 

Stored under `StateHeaderKey(shard_id=shard_B, sync_hash)` with shard_A's `prev_state_root` embedded in the chunk.

**`set_state_part` — validates against wrong state root:** [7](#0-6) 

Retrieves the header stored for shard_B, extracts `prev_state_root` from shard_A's chunk, and validates parts against shard_A's state root. shard_A's parts pass.

## Attack Feasibility

The attacker is an unprivileged peer responding to state sync requests. The local node determines `shard_id` (shard_B) locally; the peer only supplies the `header`. [8](#0-7) 

The crafted header the attacker serves:
- `chunk` = shard_A's valid chunk (signed by chunk producer, passes `validate_chunk_proofs`)
- `chunk_proof` = shard_A's Merkle proof against `chunk_headers_root` (valid, shard_A IS in the tree)
- `prev_chunk_header` = shard_B's prev chunk header + its Merkle proof (public data)
- `incoming_receipts_proofs` = shard_B's receipt proofs (public chain data)
- `root_proofs` = shard_B's root proofs (public chain data)
- `state_root_node` = shard_A's state root node (consistent with shard_A's `prev_state_root`)

All five validation steps pass. The header is stored under `StateHeaderKey(shard_B, sync_hash)` containing shard_A's `prev_state_root`. Subsequent `set_state_part` calls for shard_B validate shard_A's trie parts, and `set_state_finalize` applies shard_A's trie into shard_B's `shard_uid` storage slot. [9](#0-8)

### Title
Missing `chunk.shard_id() == shard_id` guard in `set_state_header` allows a malicious peer to install shard_A's trie state into shard_B's storage slot — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` accepts a caller-supplied `shard_id` and a peer-supplied `ShardStateSyncResponseHeader`, but never asserts `chunk.shard_id() == shard_id`. All five validation steps can be satisfied by a crafted header mixing shard_A's valid chunk with shard_B's public receipt proofs, causing the node to store shard_A's `prev_state_root` under shard_B's `StateHeaderKey` and ultimately install shard_A's trie into shard_B's storage.

### Finding Description
The missing guard is at: [10](#0-9) 

After `validate_chunk_proofs` returns `true`, the code comments "Consider chunk itself is valid" but never checks whether the chunk belongs to the requested shard. The `chunk_headers_root` Merkle proof (step 3a) proves the chunk is *somewhere* in the block's chunk tree, not that it is at the position corresponding to `shard_id`. The `StateHeaderKey` is then written with the attacker-chosen `shard_id`: [11](#0-10) 

### Impact Explanation
A node performing state sync for shard_B will have shard_A's trie installed into shard_B's storage. After `set_state_finalize`, the node operates with a completely wrong state for shard_B — wrong account balances, wrong contract storage, wrong validator stakes — for the entire epoch until the next state sync or re-sync.

### Likelihood Explanation
Any peer on the network can serve state sync responses. The attack requires only public chain data (a valid chunk for shard_A, its Merkle proof, and shard_B's receipt proofs). No validator or privileged credentials are needed.

### Recommendation
Add immediately after extracting the chunk in `set_state_header`:
```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
``` [10](#0-9) 

### Proof of Concept
A cargo test in `chain/chain/src/state_sync` would:
1. Build a two-shard test environment with known state for shard_A and shard_B.
2. Compute a legitimate `ShardStateSyncResponseHeader` for shard_A.
3. Replace its `incoming_receipts_proofs`/`root_proofs` with valid proofs for shard_B, and replace `prev_chunk_header`/`prev_chunk_proof` with shard_B's values.
4. Call `set_state_header(shard_B, sync_hash, crafted_header)` — assert it returns `Ok(())`.
5. Call `set_state_part` for shard_B with shard_A's parts — assert they pass validation.
6. Call `set_state_finalize(shard_B, sync_hash)`.
7. Assert the installed state root for shard_B equals shard_A's `prev_state_root`, not shard_B's.

pub fn validate_chunk_proofs(
    chunk: &ShardChunk,
    epoch_manager: &dyn EpochManagerAdapter,
) -> Result<bool, Error> {
    let correct_chunk_hash = chunk.compute_header_hash();

    // 1. Checking chunk.header.hash
    let header_hash = chunk.header_hash();
    if header_hash != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }

    // 2. Checking that chunk body is valid
    // 2a. Checking chunk hash
    if chunk.chunk_hash() != &correct_chunk_hash {
        byzantine_assert!(false);
        return Ok(false);
    }
    let height_created = chunk.height_created();
    let outgoing_receipts_root = chunk.prev_outgoing_receipts_root();
    let (transactions, receipts) = (chunk.to_transactions(), chunk.prev_outgoing_receipts());

    // 2b. Checking that chunk transactions are valid
    let (tx_root, _) = merklize(transactions);
    if &tx_root != chunk.tx_root() {
        byzantine_assert!(false);
        return Ok(false);
    }
    // 2c. Checking that chunk receipts are valid
    if height_created == 0 {
        return Ok(receipts.is_empty() && outgoing_receipts_root == &CryptoHash::default());
    } else {
        let shard_layout = {
            let prev_block_hash = chunk.prev_block_hash();
            epoch_manager.get_shard_layout_from_prev_block(&prev_block_hash)?
        };
        let outgoing_receipts_hashes = Chain::build_receipts_hashes(receipts, &shard_layout)?;
        let (receipts_root, _) = merklize(&outgoing_receipts_hashes);
        if &receipts_root != outgoing_receipts_root {
            byzantine_assert!(false);
            return Ok(false);
        }
    }
    Ok(true)
}
