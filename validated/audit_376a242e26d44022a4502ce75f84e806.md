Audit Report

## Title
Router `pay()` Ignores Designated Payer When Native ETH Balance Is Sufficient, Allowing Theft of Stranded ETH via WETH Swap - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

## Summary
The `pay()` function in `PeripheryPayments` unconditionally uses the router's own native ETH balance to settle WETH payments whenever `address(this).balance >= value`, bypassing the designated `payer` entirely. Because the router is `payable` and holds no per-caller ETH accounting, any ETH left on the router from a prior user's transaction can be consumed by a subsequent attacker's WETH swap at zero cost to the attacker, resulting in direct theft of the victim's stranded ETH.

## Finding Description
In `PeripheryPayments.pay()`, the WETH branch at lines 73–84 checks `address(this).balance` against `value`. When the router's total native balance covers the requested amount, it wraps that ETH and transfers WETH to the recipient without ever calling `transferFrom` on `payer`:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);   // payer never touched
    } ...
}
```

ETH accumulates on the router whenever a caller sends `msg.value` to any `payable` entry point (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`, `multicall`, `addLiquidityExactShares`, `addLiquidityWeighted`) and omits a trailing `refundETH()` call. The `receive()` guard at lines 32–34 only blocks direct ETH pushes from non-WETH addresses; it does not prevent `msg.value` from accumulating across separate transactions.

The transient callback context (stored via `TransientCallbackPool`) tracks the pool, callback mode, payer address, and token to pay, but records **no per-call ETH attribution**. There is no mechanism to distinguish ETH sent by the current caller from ETH left by a prior caller.

The exploit path is:
1. Victim calls `exactInputSingle{value: 2 ether}` for a 1 ETH swap without appending `refundETH()`. The swap consumes 1 ETH; 1 ETH remains on the router.
2. Attacker calls `exactInputSingle{value: 0}` for a 1 ETH WETH-input swap with 0 WETH balance and 0 allowance.
3. The pool calls `metricOmmSwapCallback`, which calls `_justPayCallback`, which calls `pay(WETH, attacker, pool, 1e18)`.
4. `pay()` sees `address(this).balance == 1 ether >= 1 ether`, wraps the victim's ETH, and transfers WETH to the pool. The attacker's address is never debited.
5. The pool's post-callback balance check passes (it received the correct WETH). The attacker receives the full output token amount for free.

## Impact Explanation
Direct loss of user principal: the victim's stranded ETH is consumed to settle the attacker's trade. The pool is unharmed (it receives the correct WETH), so the loss falls entirely on the victim. Loss magnitude equals the stranded ETH balance, up to the victim's full `msg.value`. This constitutes a Critical/High direct theft of user funds, satisfying the allowed impact gate for direct loss of user principal above Sherlock thresholds.

## Likelihood Explanation
ETH stranding is a routine operational outcome of the standard multicall pattern for native-ETH WETH swaps. Any user who omits `refundETH()` after an ETH-input swap leaves excess ETH on the router. The router holds no per-user accounting, so stranded ETH is immediately available to any subsequent caller in the same or a later block. A MEV bot monitoring the mempool can front-run the victim's transaction or back-run it in the same block with a zero-cost WETH swap. No special privileges are required; any unprivileged caller can trigger this path.

## Recommendation
Track the ETH attributed to the current top-level call in transient storage: record `msg.value` at each `payable` entry point and decrement it as it is consumed by `pay()`. In `pay()`, only use native ETH up to the amount attributed to the current call, and fall back to `safeTransferFrom(payer, ...)` for any shortfall. Alternatively, require that WETH swaps always pull from the payer via `transferFrom` and treat native ETH wrapping as an explicit, bounded pre-step (e.g., a dedicated `wrapETH` multicall step that converts exactly `msg.value` to WETH on behalf of the caller before the swap begins).

## Proof of Concept
**Setup:** Deploy a WETH/token1 pool and the `MetricOmmSimpleRouter`.

**Step 1 – Victim strands ETH:**
```solidity
// Victim sends 2 ETH; swap costs 1 ETH; no refundETH() appended
router.exactInputSingle{value: 2 ether}(ExactInputSingleParams({
    pool: pool, tokenIn: address(weth), tokenOut: address(token1),
    zeroForOne: true, amountIn: 1 ether, amountOutMinimum: 0,
    recipient: victim, deadline: block.timestamp + 1,
    priceLimitX64: 0, extensionData: ""
}));
// address(router).balance == 1 ether
```

**Step 2 – Attacker exploits:**
```solidity
// Attacker: 0 ETH sent, 0 WETH balance, 0 allowance
router.exactInputSingle{value: 0}(ExactInputSingleParams({
    pool: pool, tokenIn: address(weth), tokenOut: address(token1),
    zeroForOne: true, amountIn: 1 ether, amountOutMinimum: 0,
    recipient: attacker, deadline: block.timestamp + 1,
    priceLimitX64: 0, extensionData: ""
}));
// pay() sees nativeBalance (1 ETH) >= value (1 ETH)
// Wraps victim's ETH, sends WETH to pool, attacker receives token1 output
// Victim's 1 ETH is permanently lost
```

Assert: `address(router).balance == 0`, attacker holds token1 output, victim's 1 ETH is gone.
