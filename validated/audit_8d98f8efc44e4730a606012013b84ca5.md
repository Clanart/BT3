### Title
Tautological Root-Hash Check in `ProofOfInclusion::valid()` Allows Forged Inclusion Proofs - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary

`ProofOfInclusion::valid()` is the sole API for verifying DataLayer Merkle inclusion proofs. Its final guard — intended to confirm the proof terminates at the correct tree root — is a tautology that is always `true`. Any internally-consistent but root-mismatched proof passes validation, allowing an attacker to forge proof of inclusion for arbitrary key-value pairs.

### Finding Description

`ProofOfInclusion::valid()` iterates over each layer, verifying that `calculate_internal_hash(existing_hash, other_hash_side, other_hash) == layer.combined_hash`, then advances `existing_hash = calculated_hash`. After the loop it executes:

```rust
existing_hash == self.root_hash()
```

`root_hash()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← derived from the proof itself
    } else {
        self.node_hash
    }
}
```

After the loop body, `existing_hash` holds the last `calculated_hash`, which was already asserted equal to `layer.combined_hash` (the last layer's `combined_hash`). `self.root_hash()` returns that same `last.combined_hash`. The comparison therefore reduces to:

```
last_layer.combined_hash == last_layer.combined_hash  →  always true
```

The check is a tautology. `valid()` only verifies internal self-consistency of the proof structure; it never compares the computed root against any externally-trusted tree root. The correct implementation must compare `existing_hash` against a caller-supplied, trusted root hash — not against a value derived from the proof itself.

The function is exposed directly to Python callers via:

```rust
#[pyo3(name = "valid")]
pub fn py_valid(&self) -> bool {
    self.valid()
}
```

No separate `valid_for_root(expected: &Hash)` API exists. The test suite calls only `assert!(proof_of_inclusion.valid())` without any independent root-hash comparison, confirming the design intent is that `valid()` is a complete validity check.

### Impact Explanation

An attacker who can supply a serialized `ProofOfInclusion` (the type is `Streamable` and `from_py_object`) to any consumer that calls only `proof.valid()` can:

1. Choose an arbitrary `node_hash` (e.g., the leaf hash of a key-value pair not present in the tree).
2. Construct any number of layers where each `combined_hash` is computed correctly from the previous hash and a chosen `other_hash` — the layers need only be internally consistent, not anchored to the real tree.
3. Submit the forged proof; `valid()` returns `true`.

The consumer accepts the proof as proving inclusion of a key-value pair that is not in the tree. This directly satisfies the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion and lets untrusted input prove invalid state**.

### Likelihood Explanation

- The `ProofOfInclusion` struct is `Streamable`, meaning it can be deserialized from untrusted network bytes.
- The Python binding exposes `valid()` as the sole verification method with no root parameter.
- The test suite itself does not perform an independent root-hash check, confirming that callers are expected to rely solely on `valid()`.
- Any DataLayer client that receives a proof over the network and calls `proof.valid()` is fully vulnerable.

### Recommendation

Replace the self-referential `root_hash()` comparison with a comparison against a caller-supplied, externally-trusted root:

```rust
pub fn valid_for_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root   // compare against trusted external root
}
```

The Python binding and all call sites must be updated to pass the trusted root obtained from the on-chain committed state, not from the proof object itself.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side, MerkleBlob};
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};

// Build a real tree with one entry and obtain its root.
let mut blob = MerkleBlob::new(vec![]).unwrap();
blob.batch_insert(vec![((KeyId(1), ValueId(1)), real_hash)]).unwrap();
blob.calculate_lazy_hashes().unwrap();
let real_root = blob.get_root_hash().unwrap();

// Forge a proof for a key NOT in the tree.
let fake_node_hash = Hash(/* hash of attacker-chosen leaf */);
let fake_other  = Hash(/* arbitrary */);
// Compute combined so the layer is internally consistent:
let fake_combined = calculate_internal_hash(&fake_node_hash, Side::Right, &fake_other);

let forged = ProofOfInclusion {
    node_hash: fake_node_hash,
    layers: vec![ProofOfInclusionLayer {
        other_hash_side: Side::Right,
        other_hash: fake_other,
        combined_hash: fake_combined,   // consistent with node_hash
    }],
};

// valid() returns true even though forged.root_hash() != real_root
assert!(forged.valid());                          // passes — tautology
assert_ne!(forged.root_hash(), real_root);        // root does NOT match real tree
