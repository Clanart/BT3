### Title
Integer Overflow in Relative Timelock Arithmetic Enables Timelock Bypass When `nowrap=false` — (File: `crates/chia-consensus/src/check_time_locks.rs`)

---

### Summary

`check_time_locks` uses `wrapping_add` for relative timelock arithmetic when `nowrap=false`. An unprivileged spend bundle containing a near-maximum `ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE` value causes the addition to overflow and wrap to a small integer, making the timelock comparison trivially pass and allowing the coin to be spent immediately — bypassing the intended lock period entirely.

---

### Finding Description

`check_time_locks` accepts a `nowrap: bool` parameter that selects between two arithmetic modes for relative timelock conditions:

- `nowrap=true` → `saturating_add` (correct: clamps at `u64::MAX` / `u32::MAX`)
- `nowrap=false` → `wrapping_add` (legacy: silently wraps on overflow)

For `ASSERT_SECONDS_RELATIVE`, the check is:

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [1](#0-0) 

If an attacker sets `seconds_relative` to `u64::MAX - unspent.timestamp + 1`, then:

```
unspent.timestamp.wrapping_add(u64::MAX - unspent.timestamp + 1)
  = (unspent.timestamp + u64::MAX - unspent.timestamp + 1) mod 2^64
  = 0
```

The check becomes `timestamp < 0`, which is always `false`, so the spend is **accepted immediately** regardless of the current block timestamp. The same overflow applies to `ASSERT_HEIGHT_RELATIVE` via `u32::wrapping_add`. [2](#0-1) 

The inverse conditions are also affected. For `ASSERT_BEFORE_HEIGHT_RELATIVE` and `ASSERT_BEFORE_SECONDS_RELATIVE`, overflow wraps the deadline to a small value, making `prev_height >= 0` always true, which **permanently rejects** spends that should be valid — locking coins indefinitely. [3](#0-2) 

The test suite explicitly documents and confirms this divergence:

```
// 2000 < 1000 + u64::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::seconds_relative_wrap(
    Osc { seconds_relative: Some(0xffff_ffff_ffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed)),
    Ok(()),   // <-- bypass: spend accepted when it should be rejected
)]
``` [4](#0-3) 

The Python binding exposes `nowrap` as a caller-controlled parameter, and the full node passes it based on block height (pre-soft-fork blocks use `nowrap=false`): [5](#0-4) 

---

### Impact Explanation

**High — Timelock validation bypass enables unauthorized spend acceptance.**

When `nowrap=false` is in effect (pre-soft-fork block validation), any spend bundle containing `ASSERT_SECONDS_RELATIVE` or `ASSERT_HEIGHT_RELATIVE` with a crafted near-maximum value bypasses the timelock entirely. The coin is accepted as spendable at any block height/timestamp, defeating the puzzle's intended locking semantics. Coins locked by time-sensitive puzzles (vaults, HTLCs, DID recovery, etc.) can be stolen before their unlock time.

The `ASSERT_BEFORE_*` overflow direction causes the opposite: coins that should be spendable are permanently rejected, effectively burning them.

---

### Likelihood Explanation

The attacker controls the `ASSERT_SECONDS_RELATIVE` argument through the puzzle or solution bytes of their own spend bundle — this is a fully unprivileged, on-chain input. The only precondition is that the full node evaluates the spend under `nowrap=false`. Since `nowrap` is selected by block height (legacy path), any node replaying or validating pre-soft-fork blocks, or any node that has not yet activated the `nowrap=true` soft-fork, is vulnerable. The exact overflow threshold is trivially computable from the coin's `confirmed_block_index` / `timestamp`, which are public chain data.

---

### Recommendation

1. Remove the `wrapping_add` branch entirely. Replace all `wrapping_add` calls with `saturating_add` unconditionally, matching the `nowrap=true` path.
2. If backward compatibility with historical blocks is required, add an explicit overflow guard before the comparison rather than silently wrapping:
   ```rust
   let deadline = unspent.timestamp.checked_add(seconds_relative)
       .unwrap_or(u64::MAX);
   if timestamp < deadline { return Err(...); }
   ```
3. Audit all callers of `check_time_locks` in the Python full node to confirm `nowrap=true` is passed for all new block validation paths.

---

### Proof of Concept

**Setup:** Coin confirmed at `timestamp = T = 1_000_000`. Current block `timestamp = 1_000_001`. Attacker crafts spend with:

```
ASSERT_SECONDS_RELATIVE = u64::MAX - T + 1
                        = 18446744073708551616
```

**With `nowrap=false` (wrapping path):**

```
unspent.timestamp.wrapping_add(seconds_relative)
  = 1_000_000 + (u64::MAX - 1_000_000 + 1)  mod 2^64
  = u64::MAX + 1  mod 2^64
  = 0
```

Check: `1_000_001 < 0` → `false` → **spend accepted** (timelock bypassed).

**With `nowrap=true` (saturating path):**

```
unspent.timestamp.saturating_add(seconds_relative)
  = u64::MAX  (saturates)
```

Check: `1_000_001 < u64::MAX` → `true` → **spend rejected** (correct behavior).
