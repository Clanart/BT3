### Title
Stale-Pending Fallback in `read_pending_data` Omits `l1_data_gas_price` and `l2_gas_price` from Latest Header, Causing Fee Estimation to Use `NonzeroGasPrice::MIN` — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

When the in-memory pending block's `parent_block_hash` does not match the latest stored block hash, `read_pending_data` synthesizes a `DeprecatedPendingBlock` fallback. This fallback copies only `l1_gas_price` from the latest header; `l1_data_gas_price` and `l2_gas_price` are left at their `Default` (zero) values. Because `DeprecatedPendingBlock` has no fields for those two prices, `l1_data_gas_price()` and `l2_gas_price()` unconditionally return `GasPricePerToken::default()` for the `Deprecated` variant. Downstream, `create_block_context` silently clamps every zero price to `NonzeroGasPrice::MIN`, so `starknet_estimateFee` and `starknet_simulateTransactions` return artificially low fees for any request that lands during the transient mismatch window.

### Impact Explanation

Any call to `starknet_estimateFee` or `starknet_simulateTransactions` with `BlockId::Tag(Tag::Pending)` that arrives while the in-memory pending block is stale receives fee estimates computed with `l1_data_gas_price = NonzeroGasPrice::MIN` and `l2_gas_price = NonzeroGasPrice::MIN` instead of the actual values from the latest accepted header. For modern Starknet blocks where both prices are non-trivial, the returned fee is materially wrong. This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

The mismatch window is a **normal, recurring transient state**: it opens every time a new block is accepted and closes when the pending-sync loop fetches fresh pending data. On mainnet this window recurs every ~6 seconds. No special privileges are required; any caller can hit it by issuing a pending fee-estimation request at any time. The existing test suite explicitly exercises this path and confirms the fallback fires: [6](#0-5) 

---

### Recommendation

Replace the `Deprecated` fallback block with a `PendingBlockOrDeprecated::Current(PendingBlock { … })` that explicitly copies `l1_data_gas_price` and `l2_gas_price` from `latest_header.block_header_without_hash`, mirroring the existing handling of `l1_gas_price`, `timestamp`, `sequencer_address`, and `starknet_version`.

### Proof of Concept

1. Write a block header with non-zero `l1_data_gas_price` and `l2_gas_price` into storage.
2. Set the in-memory `pending_data.block.parent_block_hash` to a random value that does not match the stored block hash (triggering the fallback).
3. Call `starknet_estimateFee` with `BlockId::Tag(Tag::Pending)`.
4. Observe that the returned `l1_data_gas_price` and `l2_gas_price` in the fee breakdown equal `NonzeroGasPrice::MIN` (1 wei / 1 fri) rather than the values stored in the header.
5. Repeat the call after the pending sync updates (so `parent_block_hash` matches) and observe the correct prices are used.

The existing test at `crates/apollo_rpc/src/v0_8/api/test.rs` lines 653–661 already demonstrates the fallback fires and that `parent_hash` is taken from the latest header; extending it to assert on `l1_data_gas_price` and `l2_gas_price` would immediately expose the defect.
