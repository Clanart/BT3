### Title
Unchecked Constructor Failure in `execute_deploy` Syscall Always Reports Success — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall handler in the StarkNet OS unconditionally writes `failure_flag=0` in the syscall response, regardless of whether the deployed contract's constructor actually succeeded or reverted. This is the direct Cairo analog of the reported "unchecked return value" vulnerability class: a function whose result can indicate failure is called, but its failure status is silently discarded and success is always reported to the caller.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_deploy` function handles the `deploy` syscall. After calling `deploy_contract` (which runs the constructor entry point and can revert), the response header is written with a hardcoded `failure_flag=0`:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` call returns `(retdata_size, retdata)`. When the constructor reverts, `retdata` contains the revert data and the constructor's state changes are rolled back via the `revert_log`. However, the `failure_flag` field in the `ResponseHeader` written to `syscall_ptr` is **always `0` (success)**, regardless of the constructor's actual outcome.

The developer-inserted TODO comment at line 538 explicitly acknowledges this is unimplemented: `// TODO(Yoni, 1/1/2026): support failures.` [2](#0-1) 

By contrast, the `contract_call_helper` function (which handles `call_contract` and `library_call` syscalls) correctly propagates `is_reverted` into the response:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [3](#0-2) 

The `execute_deploy` path is inconsistent: it never sets `failure_flag=1`.

---

### Impact Explanation

**Impact: Critical — Permanent freezing of funds.**

When a calling contract issues a `deploy` syscall and the constructor reverts:

1. The OS rolls back the constructor's storage writes via the `revert_log`.
2. Depending on whether `deploy_contract` also reverts the class-hash assignment, the deployed address is either empty (no class hash) or partially initialized (class hash set, no constructor state).
3. The calling contract receives `failure_flag=0` and a valid `contract_address` in the `DeployResponse`.
4. The calling contract, believing deployment succeeded, may transfer funds (ERC-20 tokens or native ETH bridged via L1→L2 messages) to the returned address.
5. If the address has no class hash (fully reverted), any call to it will fail with `ERROR_ENTRY_POINT_NOT_FOUND`, and funds sent there are permanently unrecoverable — **permanent freezing of funds**. [4](#0-3) 

---

### Likelihood Explanation

**Likelihood: Medium.**

- Any contract that uses the `deploy` syscall and then interacts with the deployed address (e.g., a factory pattern, a wallet deployer, or a vault factory) is affected.
- Constructor failures can be triggered by: out-of-gas conditions (attacker provides a tight gas bound), invalid constructor arguments, or any revert inside the constructor.
- An unprivileged transaction sender can craft a transaction that calls such a factory contract with parameters designed to cause the constructor to revert (e.g., by providing calldata that triggers a revert path in the constructor), while the factory contract proceeds to send funds to the "deployed" address.
- No privileged access is required.

---

### Recommendation

Propagate the constructor's revert status into the `DeployResponse`. The `deploy_contract` function should return an `is_reverted` flag (analogous to `execute_entry_point` returning `is_reverted`), and `execute_deploy` should write:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
```

This mirrors the pattern already used correctly in `contract_call_helper`. [5](#0-4) 

---

### Proof of Concept

1. Deploy a factory contract `F` on StarkNet with the following logic in its `__execute__`:
   - Call `deploy(class_hash=VAULT_CLASS, constructor_calldata=[INVALID_ARG])` where `INVALID_ARG` causes the vault constructor to revert.
   - After the deploy syscall returns, read `response.failure_flag` — it will be `0` (success).
   - Transfer 1000 STRK to `response.contract_address`.

2. Submit an invoke transaction calling `F.__execute__`.

3. The OS processes the block:
   - `execute_deploy` runs the vault constructor, which reverts.
   - The constructor's state changes are rolled back.
   - `failure_flag=0` is written unconditionally.
   - `F` receives success and sends 1000 STRK to the vault address.

4. The vault address has no initialized state (or no class hash). Any subsequent call to withdraw the 1000 STRK fails permanently.
