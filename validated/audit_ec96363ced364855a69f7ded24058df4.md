### Title
`ProofOfInclusion::valid()` Uses Self-Referential Root — Forged DataLayer Inclusion Proof Always Passes - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` contains a tautological final comparison: it checks `existing_hash == self.root_hash()`, where both sides are derived from the proof's own `combined_hash` field in the last layer. The function never compares against an external, caller-supplied trusted root. An attacker who can supply a serialized `ProofOfInclusion` to any code that calls `valid()` can forge a proof claiming any key is included in any DataLayer tree, and the check will pass unconditionally.

### Finding Description

`ProofOfInclusion::valid()` is the sole API for verifying a DataLayer inclusion proof:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;   // ← set to last layer's combined_hash
    }

    existing_hash == self.root_hash()      // ← always true
}
``` [1](#0-0) 

`root_hash()` is defined as:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // ← same field just verified in the loop
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

After the loop completes without returning `false`, `existing_hash` holds `calculated_hash` from the last iteration. The loop body already asserted `calculated_hash == layer.combined_hash`, so `existing_hash` is exactly `last_layer.combined_hash`. `self.root_hash()` also returns `last_layer.combined_hash`. The final comparison is therefore `last_layer.combined_hash == last_layer.combined_hash` — always `true`.

**Edge case — empty `layers`:** When `layers` is empty (single-node tree), the loop body never executes. `existing_hash` stays as `self.node_hash`, and `self.root_hash()` returns `self.node_hash`. The check is `self.node_hash == self.node_hash` — always `true` regardless of what `node_hash` contains.

The correct formula would require an external trusted root parameter:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool { ... }
```

and the final comparison should be `existing_hash == *trusted_root`, not `existing_hash == self.root_hash()`.

This is the direct analog of the external report's bug: the wrong variable is used in the validation formula — a self-referential proof field (`self.root_hash()`) is used instead of an external trusted value, making the check meaningless.

### Impact Explanation

`ProofOfInclusion` is a `Streamable` type exposed through the Python wheel: [3](#0-2) 

Any Python DataLayer client that receives a serialized `ProofOfInclusion` from an untrusted DataLayer server and calls `proof.valid()` will accept a completely fabricated proof. The attacker constructs a `ProofOfInclusion` with:

- `node_hash` = SHA256 of any fake key/value pair
- `layers` = a chain of arbitrary `(other_hash_side, other_hash, combined_hash)` tuples where each `combined_hash` equals `calculate_internal_hash(prev, side, other)` for attacker-chosen inputs

The chain is internally consistent, so `valid()` returns `true`. The attacker can claim any key-value pair is present in any DataLayer store. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion, lets untrusted input prove invalid state.**

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` type, meaning it is routinely serialized and transmitted between DataLayer nodes and clients. The Python method `valid()` is the natural and only API for proof verification. There is no documentation warning callers to also compare `root_hash()` against a separately obtained trusted root. Any DataLayer client that follows the obvious usage pattern — deserialize proof, call `valid()` — is vulnerable. The attacker needs no special privileges; they only need to be a DataLayer server responding to a client's proof request.

### Recommendation

1. Add an external root parameter to `valid()`:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // compare against external trusted root
}
```

2. Either remove the no-argument `valid()` or rename it to `is_internally_consistent()` with a clear doc comment that it does **not** verify against any trusted root.

3. Update the Python binding to expose `valid_against_root(root: bytes32)` and deprecate the current `valid()`.

### Proof of Concept

```python
from chia_rs import ProofOfInclusion, ProofOfInclusionLayer
import hashlib

# Forge a proof claiming fake_key_hash is in some tree
fake_key_hash = bytes([0xAA] * 32)

# Build a single-layer chain with attacker-chosen values
other_hash = bytes([0xBB] * 32)
# compute combined_hash = calculate_internal_hash(fake_key_hash, Left, other_hash)
# (exact hash depends on calculate_internal_hash implementation, but attacker controls all inputs)
combined_hash = hashlib.sha256(fake_key_hash + other_hash).digest()  # simplified

layer = ProofOfInclusionLayer(
    other_hash_side=0,       # Left
    other_hash=other_hash,
    combined_hash=combined_hash,
)

forged_proof = ProofOfInclusion(node_hash=fake_key_hash, layers=[layer])

# valid() returns True despite no real tree existing
assert forged_proof.valid()  # passes — forged proof accepted
```

The `valid()` call succeeds because the loop verifies only that `combined_hash == calculate_internal_hash(fake_key_hash, side, other_hash)`, which the attacker satisfies by construction, and the final check `existing_hash == self.root_hash()` reduces to `combined_hash == combined_hash`.
