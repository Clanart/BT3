### Title
Selfdestruct-then-recreate in same transaction leaves stale storage in KV store ‚Äî (`x/evm/statedb/statedb.go`, `x/evm/keeper/statedb.go`)

### Summary

The external report's vulnerability class is **"remove-then-re-add leaves accumulated state un-reset"**: when an entity is removed and re-added, previously accumulated state is not cleared, causing the re-added entity to inherit stale values it should not see.

The direct analog in Ethermint is: when a contract **selfdestructs and is re-created at the same address within the same transaction** (the canonical CREATE2 re-deploy pattern), Ethermint's `StateDB.Commit()` never calls `DeleteAccount` for the selfdestructed object, so the old contract's storage slots remain in the KV store. The newly-created contract then reads those stale slots through `GetCommittedState`, inheriting storage it should see as zero.

---

### Finding Description

**Root cause ‚Äî `Commit` loses track of the selfdestructed predecessor**

`createObject` replaces the live `stateObject` in `s.stateObjects` with a brand-new object:

```go
// x/evm/statedb/statedb.go  lines 319-330
func (s *StateDB) createObject(addr common.Address) *stateObject {
    prev := s.getStateObject(addr)          // ‚Üê selfdestructed object
    newobj := newObject(s, addr, nil)       // ‚Üê fresh object, empty storage
    if prev == nil {
        s.journal.append(createObjectChange{account: &addr})
    } else {
        s.journal.append(resetObjectChange{prev: prev})  // ‚Üê does NOT dirty addr
    }
    s.setStateObject(newobj)               // ‚Üê overwrites selfdestructed obj
    return newobj
}
``` [1](#0-0) 

`resetObjectChange.Dirtied()` returns `nil`, so the address is dirtied only by the earlier `selfDestructChange`. At `Commit` time the loop resolves `s.stateObjects[addr]` to the **new** object:

```go
// x/evm/statedb/statedb.go  lines 795-844
for _, addr := range s.journal.sortedDirties() {
    obj, exist := s.stateObjects[addr]
    if !exist { continue }
    if obj.selfDestructed {          // ‚Üê FALSE for the new object
        // DeleteAccount / storage wipe ‚Äî NEVER REACHED
    } else {
        // only writes obj.dirtyStorage (empty for new object)
    }
}
``` [2](#0-1) 

Because `obj.selfDestructed` is `false` for the replacement object, `DeleteAccount` is never called:

```go
// x/evm/keeper/statedb.go  lines 195-222
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
    // ...
    k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
        k.SetState(ctx, addr, key, nil)   // ‚Üê clears storage ‚Äî NEVER CALLED
        return true
    })
    k.accountKeeper.RemoveAccount(ctx, acct)
    // ...
}
``` [3](#0-2) 

The new object has empty `originStorage`, so any `GetCommittedState` call falls through to the KV store and reads the **old** value:

```go
// x/evm/statedb/state_object.go  lines 213-228
func (s *stateObject) GetCommittedState(key common.Hash) common.Hash {
    if value, cached := s.originStorage[key]; cached { return value }
    value := s.db.keeper.GetState(s.db.ctx, s.Address(), key)  // ‚Üê reads stale KV
    s.originStorage[key] = value
    return value
}
``` [4](#0-3)

### Impact Explanation

The new contract deployed at the recycled address inherits every storage slot the old contract wrote. This violates the EVM specification (storage must be zero for a freshly created contract). Concrete consequences:

1. **Unauthorized state read / logic hijack**: A contract whose correctness depends on storage being zero at creation (e.g., initializer guards, balance mappings, ownership slots) can be bypassed by an attacker who controls the CREATE2 salt and can trigger a selfdestruct-then-redeploy in one transaction.
2. **Committed invalid state**: The stale slots are written to the committed KV store and persist across blocks, so every future call to the new contract sees wrong storage ‚Äî this is an EVM state transition bug that permits invalid state to commit.
3. **Consensus divergence**: Standard go-ethereum clears storage for selfdestructed accounts; Ethermint does not in this path, causing a state-root mismatch between Ethermint validators and any Ethereum-compatible verifier.

This matches the allowed High impact: *"EVM state transition ‚Ä¶ bug that permits invalid transactions to commit or valid user funds/fees to be mis-accounted."*

---

### Likelihood Explanation

The attack path requires only an unprivileged EVM transaction:

1. Deploy a factory contract that uses CREATE2.
2. In a single transaction: (a) CREATE2-deploy a child contract that immediately selfdestructs (EIP-6780 allows this because the contract was created in the same tx ‚Äî `stateObject.newContract == true`); (b) CREATE2-redeploy a new contract at the same salt/address.
3. The new contract now reads stale storage from step (a).

No privileged role, governance action, or validator collusion is required. The pattern (selfdestruct + CREATE2 redeploy) is well-known and used in production upgrade patterns.

---

### Recommendation

In `Commit`, track whether an address had a selfdestructed predecessor before being replaced by `createObject`. One approach: when `createObject` finds a selfdestructed `prev`, record the address in a separate `pendingDeletes` set. In `Commit`, process `pendingDeletes` first (call `DeleteAccount`) before writing the new object's state. Alternatively, mirror go-ethereum's approach of keeping the selfdestructed object in a separate map so `Commit` can always find it regardless of subsequent re-creation.

---

### Proof of Concept

```
Transaction T:
  1. CALLER ‚Üí Factory.deploy(salt)
       Factory: CREATE2(salt) ‚Üí ContractA at addr X
       ContractA constructor: SSTORE(slot0, 0xdeadbeef)
                              SELFDESTRUCT(beneficiary)   // newContract=true ‚Üí actual destruct
  2. Factory: CREATE2(salt) ‚Üí ContractB at addr X
       ContractB constructor: val := SLOAD(slot0)
                              // val == 0xdeadbeef  (should be 0x0)
                              // stale storage inherited from ContractA

After T commits:
  keeper.GetState(X, slot0) == 0xdeadbeef   // persists in KV store
  ContractB reads 0xdeadbeef on every future SLOAD(slot0)
```

The `selfDestructed` flag on the old object is lost when `createObject` overwrites `s.stateObjects[addr]`, so `Commit` never reaches the `DeleteAccount` branch for address X.

func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
```

func (s *stateObject) GetCommittedState(key common.Hash) common.Hash {
	if s.overrideStorage != nil {
		if value, ok := s.overrideStorage[key]; ok {
			return value
		}
		return common.Hash{}
	}

	if value, cached := s.originStorage[key]; cached {
		return value
	}
	// If no live objects are available, load it from keeper
	value := s.db.keeper.GetState(s.db.ctx, s.Address(), key)
	s.originStorage[key] = value
	return value
}
