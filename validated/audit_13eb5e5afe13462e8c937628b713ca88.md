Looking at the exact code in `_fillBids` and comparing it to `_fillAsks` to understand the invariant.

The bug is real and traceable. Here is the complete analysis.

---

### Title
`_fillBids` subtracts the length of bin `b+1` instead of bin `b` when walking `walkDistE6`, producing systematically wrong `lowerX64`/`upperX64` bid-bin price boundaries for any pool with non-uniform bin lengths — (`metric-periphery/contracts/lens/MetricOmmPoolDataProvider.sol`)

---

### Summary

`MetricOmmPoolDataProvider._fillBids` walks downward from `curBinIdx` to compute per-bin price boundaries. For each bin `b` in the loop it must subtract the length of bin `b` from `walkDistE6` to arrive at bin `b`'s lower boundary. Instead it reads `lenAbove` from bin `b+1` and subtracts that. The error is zero only when every bin has the same length; for any pool with non-uniform `lengthE6` values the bid-side depth ladder returns wrong `lowerX64`/`upperX64` and therefore wrong `binAvgExecPriceX64` / `cumulativeAvgExecPriceX64` for every bin below `curBinIdx`.

---

### Finding Description

**Correct invariant.** Bins are contiguous. The lower boundary of bin `b` equals the lower boundary of bin `b+1` minus the length of bin `b`:

```
lowerDist(b) = lowerDist(b+1) − length(b)
```

**What `_fillAsks` does (correct).** After computing prices for bin `b`, it advances the cursor by the current bin's own length:

```solidity
// _fillAskRow, line 441
ctx.cumDistE6 += int256(uint256(lengthE6));   // lengthE6 is bin b's length ✓
```

**What `_fillBids` does (wrong).** At the top of each loop iteration for bin `b`, it reads `lenAbove` from bin `b+1` and subtracts that:

```solidity
// _fillBids, lines 496-498
(,, uint16 lenAbove,,) = PoolStateLibrary._binState(pool, int8(b + 1));  // bin b+1 ← wrong
walkDistE6 -= int256(uint256(lenAbove));
```

Then it reads `lengthE6` from bin `b` (line 500) and uses it only for the `upperX64` offset — but `walkDistE6` (the lower boundary) is already wrong.

**Error accumulation.** After `k` iterations (processing bin `curBinIdx − k`):

| | `walkDistE6` value |
|---|---|
| **Code** | `curBinDistE6 − Σ length(curBinIdx … curBinIdx−k+1)` |
| **Correct** | `curBinDistE6 − Σ length(curBinIdx−1 … curBinIdx−k)` |
| **Error** | `length(curBinIdx) − length(curBinIdx−k)` |

The error is the difference between the current bin's length and the length of the bin being priced. For a pool with bins of 50 000 E6 (5%) and 10 000 E6 (1%), the first non-current bid bin is mispriced by 40 000 E6 = 4% of mid-price.

**`_fillAsks` is unaffected** — it uses the correct pattern above.

---

### Impact Explanation

`getLiquidityDepth` is the primary depth-ladder API consumed by integrators, aggregators, and UI routing layers to estimate sell-swap execution prices before submitting a transaction. The contest rules explicitly scope this path: *"Lens/quoter path: quoted amounts … must match actual pool execution closely enough that integrations are not induced into fund-losing trades."*

When the depth ladder reports inflated bid prices (e.g., because `length(curBinIdx)` > `length(b)`), an integrator:
1. Reads `binAvgExecPriceX64` / `cumulativeAvgExecPriceX64` from the bid ladder.
2. Concludes the pool will pay more token1 per token0 than it actually will.
3. Sets `amountOutMinimum` below the true execution price, or executes a sell that is economically loss-making relative to the price they believed they were getting.

The error magnitude scales with the difference in bin lengths, which is a free parameter chosen at pool creation. Non-uniform bin lengths are a valid, expected configuration — the factory validates only that each `lengthE6 > 0` and that cumulative distances stay within `±1e6`. No malicious or abnormal setup is required; any legitimately deployed pool with varying bin widths triggers the bug.

---

### Likelihood Explanation

- `createPool` is permissionless; any pool creator can configure non-uniform bin lengths.
- Non-uniform bin widths are a standard AMM design choice (e.g., wider bins near mid, narrower bins at extremes).
- `getLiquidityDepth` is the only public depth-ladder API; every integrator that reads bid-side depth from this contract on a non-uniform pool receives wrong prices.
- The bug is deterministic and reproducible — no race condition or oracle manipulation required.

---

### Recommendation

Replace the `lenAbove` read with the current bin's own length. Since `lengthE6` is already read from `binIdx` on line 500, restructure the loop to subtract it before computing prices:

```solidity
for (int256 b = int256(curBinIdx) - 1; b >= int256(lowCap); b--) {
    int8 binIdx = int8(b);
    (, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) =
        PoolStateLibrary._binState(pool, binIdx);          // bin b ← correct
    walkDistE6 -= int256(uint256(lengthE6));               // subtract bin b's length ✓

    uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(uint256(addFeeSellE6), Q64, ONE_E6);
    uint256 lowerX64 = _priceFromMidAndDistE6(midPriceX64, walkDistE6, Math.Rounding.Floor);
    uint256 upperX64 = _priceFromMidAndDistE6(
        midPriceX64, walkDistE6 + int256(uint256(lengthE6)), Math.Rounding.Floor
    );
    // ... rest unchanged
}
```

This mirrors the correct pattern in `_fillAskRow` (line 441).

### Proof of Concept

Deploy a pool with three bid bins of different lengths, e.g.:
- `curBinIdx = 0`, `length(0) = 50_000` (5%)
- `length(-1) = 10_000` (1%)
- `length(-2) = 20_000` (2%)

Call `getLiquidityDepth(pool, 3)`.

**Expected** `walkDistE6` for bin −1: `curBinDistE6 − 10_000`
**Actual** `walkDistE6` for bin −1: `curBinDistE6 − 50_000` (off by 40 000 E6 = 4%)

**Expected** `walkDistE6` for bin −2: `curBinDistE6 − 10_000 − 20_000 = curBinDistE6 − 30_000`
**Actual** `walkDistE6` for bin −2: `curBinDistE6 − 50_000 − 10_000 = curBinDistE6 − 60_000` (off by 30 000 E6 = 3%)

Compare `depth.bids[1].binAvgExecPriceX64` and `depth.bids[2].binAvgExecPriceX64` against manually computed expected boundaries using `_priceFromMidAndDistE6` with the correct distances. The discrepancy will be proportional to the bin-length differences and will exceed any reasonable Sherlock medium threshold.
