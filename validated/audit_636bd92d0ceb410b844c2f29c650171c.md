### Title
Native Fee (NEAR) Permanently Locked in Pending Transfers With No Cancellation Mechanism — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

When a user initiates a cross-chain transfer with a `native_token_fee > 0`, the corresponding NEAR tokens are deducted from the user's storage balance and held inside the bridge contract. If no trusted relayer ever calls `sign_transfer` for that pending transfer, the native fee NEAR is permanently locked with no recovery path for the user.

---

### Finding Description

**Deposit flow:**

A user who wants to bridge tokens with a native fee must first call `storage_deposit` on the bridge, depositing at minimum `required_balance_for_account + native_token_fee + required_balance_for_init_transfer_message` NEAR. [1](#0-0) 

Inside `init_transfer`, the native fee is consumed from the user's storage balance (via `try_to_transfer_balance_from_message_account` or `update_storage_balance`) and the `TransferMessage` — including `fee.native_fee` — is inserted into `pending_transfers`. [2](#0-1) 

**Payout flow (only path to recover the native fee):**

The native fee is only ever paid out inside `send_fee_internal`, which is reached only after a trusted relayer calls `sign_transfer` and subsequently `claim_fee` once the destination chain finalises the transfer. [3](#0-2) 

**`sign_transfer` is gated to trusted relayers only:** [4](#0-3) 

The user cannot call `sign_transfer` themselves (confirmed by integration test).

**No cancellation mechanism exists:**

A grep for `cancel_transfer`, `withdraw_transfer`, or `refund_transfer` returns zero matches across the entire repository. There is no public function that lets a transfer initiator cancel a pending transfer and reclaim the native fee.

**`storage_withdraw` / `storage_unregister` do not help:**

`storage_withdraw` only returns `available` balance. The native fee was already subtracted from `available` when the transfer was created, so it is not withdrawable. [5](#0-4) 

`storage_unregister(force=true)` refunds only `required_balance_for_account() + available` — again, the native fee is absent from `available` and is not returned. [6](#0-5) 

**`update_transfer_fee` cannot help either:**

`update_transfer_fee` only allows *increasing* the native fee (enforced by `checked_sub` that panics on underflow), so the user cannot reduce the native fee to zero and self-rescue. [7](#0-6) 

---

### Impact Explanation

Any NEAR deposited as `native_token_fee` in a pending transfer that is never processed by a trusted relayer is permanently frozen inside the bridge contract. There is no expiry, no cancel function, and no administrative rescue path visible in the production code. This matches the allowed impact: **Critical — permanent freezing / irrecoverable lock of user funds in bridge flows.**

---

### Likelihood Explanation

The scenario is realistic and reachable by any unprivileged bridge user:

- A user sets a `native_token_fee` that is too low to attract relayers (market conditions change after submission).
- All currently registered trusted relayers become inactive or are removed.
- The destination chain is paused or its factory is deregistered, making `sign_transfer` impossible to complete.
- The bridge itself is paused for an extended period; the pending transfer ages out of relayer interest.

In every case the user has no recourse.

---

### Recommendation

Add a permissionless `cancel_transfer` function callable only by the transfer's original `owner` (stored in `TransferMessageStorage`), subject to a minimum waiting period (e.g., 7 days) to prevent griefing of honest relayers. On cancellation:

1. Remove the entry from `pending_transfers` and restore the storage refund to the owner's `available` balance.
2. Return the native fee NEAR to the owner.
3. Return the bridged token amount to the owner (via `ft_transfer` back to the sender).

---

### Proof of Concept

1. User calls `storage_deposit` depositing `required_balance_for_account + 1 NEAR (native fee) + required_balance_for_init_transfer_message`.
2. User calls `ft_transfer_call` → bridge with `native_token_fee = 1 NEAR`. Transfer is inserted into `pending_transfers`; user's `available` balance is reduced by 1 NEAR + storage cost.
3. No trusted relayer calls `sign_transfer` (fee too low, relayer set empty, chain paused, etc.).
4. User calls `storage_withdraw(amount = available)` — succeeds but returns only the residual `available` (storage cost refund if any), **not** the 1 NEAR native fee.
5. User calls `storage_unregister(force = Some(true))` — receives `required_balance_for_account + available`; the 1 NEAR native fee is **not** included.
6. The 1 NEAR native fee remains permanently locked in the bridge contract's NEAR balance with no on-chain recovery path. [8](#0-7) [6](#0-5)

    #[payable]
    pub fn storage_withdraw(&mut self, amount: Option<NearToken>) -> StorageBalance {
        assert_one_yocto();
        let account_id = env::predecessor_account_id();
        let mut storage = self
            .storage_balance_of(&account_id)
            .near_expect(StorageError::AccountNotRegistered(account_id.clone()));
        let to_withdraw = amount.unwrap_or(storage.available);
        storage.total = storage.total.checked_sub(to_withdraw).near_expect(
            StorageError::NotEnoughStorageBalance {
                requested: to_withdraw,
                available: storage.total,
            },
        );
        storage.available = storage.available.checked_sub(to_withdraw).near_expect(
            StorageError::NotEnoughStorageBalance {
                requested: to_withdraw,
                available: storage.available,
            },
        );

        self.accounts_balances.insert(&account_id, &storage);

        Promise::new(account_id).transfer(to_withdraw).detach();

        storage
    }

    #[payable]
    pub fn storage_unregister(&mut self, force: Option<bool>) -> bool {
        assert_one_yocto();
        let account_id = env::predecessor_account_id();
        let Some(storage) = self.storage_balance_of(&account_id) else {
            return false;
        };

        if !force.unwrap_or_default() {
            require!(
                storage.total.saturating_sub(storage.available)
                    == self.required_balance_for_account(),
                BridgeError::StoragePendingTransfers.as_ref()
            );
        }

        self.accounts_balances.remove(&account_id);

        let refund = self
            .required_balance_for_account()
            .saturating_add(storage.available);
        Promise::new(account_id).transfer(refund).detach();
        true
    }
