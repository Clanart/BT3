### Title
Relayer Reimbursement Permanently Lost When `send_tokens` Fails in Fast-Transfer Finalization — (`near/omni-bridge/src/lib.rs`)

### Summary

In `process_fin_transfer_to_other_chain`, when a fast-transfer relayer is being reimbursed upon on-chain finalization, `send_tokens(...).detach()` is called fire-and-forget and `mark_fast_transfer_as_finalised` is called immediately after — with no callback to handle failure. If the token transfer fails (e.g., a deployed token's `mint` reverts because the token contract is paused), the fast transfer is permanently marked as finalised with no recovery path, and the relayer's pre-paid funds are irrecoverably lost. The bridge's `locked_tokens` accounting is also left in a corrupted state.

### Finding Description

In `process_fin_transfer_to_other_chain` (lib.rs:2027–2040), when a fast transfer record exists for the incoming finalization message, the bridge reimburses the relayer:

```rust
if let Some(relayer) = recipient {
    self.send_tokens(
        token,
        relayer,
        U128(amount_without_fee),
        "",
    )
    .detach();                                                    // ← fire-and-forget
    self.mark_fast_transfer_as_finalised(&fast_transfer.id());   // ← immediately finalised
}
``` [1](#0-0) 

The `detach()` schedules the cross-contract token transfer but discards its result. `mark_fast_transfer_as_finalised` is then called synchronously, regardless of whether the transfer will succeed. There is no callback analogous to `resolve_fast_transfer` that handles the failure case.

Contrast this with the "to-NEAR" fast-transfer path in `fast_fin_transfer_to_near_callback` (lib.rs:877–892), which properly chains a `resolve_fast_transfer` callback that removes the fast-transfer record and returns the amount for refund if `send_tokens` fails:

```rust
self.send_tokens(...)
    .then(
        Self::ext(env::current_account_id())
            .with_static_gas(RESOLVE_FAST_TRANSFER_GAS)
            .resolve_fast_transfer(...)
    )
``` [2](#0-1) 

The "to-other-chain" path has no equivalent. The asymmetry is the root cause.

Before the detached call, the bridge has already:
1. Decremented `locked_tokens[origin_chain][token]` by the full transfer amount via `unlock_tokens_if_needed`.
2. Incremented `locked_tokens[destination_chain][token]` by the fee via `lock_tokens_if_needed`. [3](#0-2) 

If `send_tokens` fails, these accounting mutations are permanent — there is no `revert_lock_actions` call (which exists in the "to-NEAR" failure path at lib.rs:1712) and no way to re-trigger the reimbursement. [4](#0-3) 

For deployed tokens, `send_tokens` calls `mint` on the token contract:

```rust
ext_token::ext(token)
    .with_attached_deposit(deposit)
    .with_static_gas(MINT_TOKEN_GAS.saturating_add(ft_transfer_call_gas))
    .mint(recipient, amount, ...)
``` [5](#0-4) 

If the token contract is paused (independently of the bridge), `mint` reverts. The detached promise silently fails, the fast transfer is already finalised, and the relayer's pre-paid tokens are permanently lost.

### Impact Explanation

- **Permanent loss of relayer funds**: The relayer pre-paid `amount_without_fee` tokens to the user during the fast-transfer phase. Upon finalization, the bridge is supposed to reimburse the relayer from its own holdings. If `send_tokens` fails and is detached, the relayer never receives reimbursement. The fast transfer is already marked finalised, so no retry is possible. The relayer's funds are irrecoverably lost.
- **Bridge accounting corruption**: `locked_tokens[origin_chain][token]` is permanently decremented by the full amount, and `locked_tokens[destination_chain][token]` is permanently incremented by the fee, even though the actual token transfer to the relayer did not occur. This corrupts the bridge's collateralization accounting, potentially causing future legitimate `unlock_tokens` calls to fail with `ERR_INSUFFICIENT_LOCKED_TOKENS`.

This matches the allowed impact: *"Critical. Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows"* and *"High. Balance, decimal, fee, token-mapping, or accounting corruption that breaks bridge collateralization or misdirects value."*

### Likelihood Explanation

The trigger condition is a failed `send_tokens` call. For deployed tokens, this occurs when the token contract's `mint` reverts — a realistic scenario if the token contract is independently paused (e.g., during a security incident on the token side). The bridge's own pause mechanism does not prevent this: the bridge can be unpaused while a specific token contract is paused. Any relayer who executes a fast transfer for a deployed token that subsequently gets paused before the on-chain finalization is processed will permanently lose their pre-paid funds with no recourse.

### Recommendation

Replace the detached `send_tokens` call with a chained callback that handles failure, mirroring the "to-NEAR" path:

```rust
if let Some(relayer) = recipient {
    self.send_tokens(token, relayer, U128(amount_without_fee), "")
        .then(
            Self::ext(env::current_account_id())
                .with_static_gas(RESOLVE_FAST_TRANSFER_GAS)
                .resolve_fast_transfer_to_other_chain(
                    &fast_transfer.id(),
                    &lock_actions_to_revert_on_failure,
                ),
        )
        .detach();
    // Do NOT mark as finalised here; do it inside the callback on success
}
```

The callback should:
- On success: call `mark_fast_transfer_as_finalised`.
- On failure: call `revert_lock_actions` to restore `locked_tokens`, and leave the fast transfer record intact so an admin or retry mechanism can recover.

### Proof of Concept

1. Relayer calls `ft_transfer_call` with `FastFinTransferMsg` targeting a non-NEAR chain (e.g., Solana). Bridge executes `fast_fin_transfer_to_other_chain`, locking `amount_without_fee` on the destination chain and recording the fast transfer.
2. The deployed token contract is paused (independently of the bridge).
3. A relayer (or anyone) calls `fin_transfer` with the origin-chain proof. `process_fin_transfer_to_other_chain` is entered.
4. `unlock_tokens_if_needed(origin_chain, token, full_amount)` decrements origin locked balance. `lock_tokens_if_needed(destination_chain, token, fee)` increments destination locked balance.
5. `send_tokens(token, relayer, amount_without_fee, "").detach()` schedules a `mint` call on the paused token contract — this will silently fail.
6. `mark_fast_transfer_as_finalised(&fast_transfer.id())` is called synchronously — the fast transfer is now permanently finalised.
7. The `mint` call fails. The relayer never receives `amount_without_fee`. The fast transfer cannot be retried. The `locked_tokens` accounting is permanently corrupted. [1](#0-0) [6](#0-5)

    fn unlock_tokens(
        &mut self,
        chain_kind: ChainKind,
        token_id: &AccountId,
        amount: u128,
    ) -> LockAction {
        let key = (chain_kind, token_id.clone());
        let Some(available) = self.locked_tokens.get(&key) else {
            return LockAction::Unchanged;
        };
        require!(
            available >= amount,
            TokenLockError::InsufficientLockedTokens.as_ref()
        );

        let remaining = available - amount;
        self.locked_tokens.insert(&key, &remaining);

        LockAction::Unlocked {
            chain_kind,
            token_id: token_id.clone(),
            amount,
        }
    }
