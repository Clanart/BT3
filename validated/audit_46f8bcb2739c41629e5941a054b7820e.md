### Title
Single-Producer-Only `ContractCodeRequest` With No Retry Permanently Blocks Witness Finalization - (File: chain/client/src/stateless_validation/partial_witness/partial_witness_actor.rs)

### Summary

When `ExcludeContractCodeFromStateWitness` is active, a chunk validator that is missing contract code sends a `ContractCodeRequest` to exactly **one** randomly chosen chunk producer. The `AccessedContractsState` state machine in `PartialEncodedStateWitnessTracker` has no path from `Requested` back to retry with a different producer. If the chosen producer silently drops the request (e.g., because `request.contracts().len() > MAX_CONTRACTS_PER_REQUEST`, because `derive_main_transition_key` returns `None`, or because `StateTransitionData` is absent), the validator's `CacheEntry` stays in `Requested` forever, `try_finalize()` never returns `Some`, and the validator never emits a chunk endorsement.

### Finding Description

**Step 1 – single-producer selection with no retry**

In `handle_chunk_contract_accesses`, after computing `missing_contract_hashes`, the code picks exactly one producer at random and fires a single network message:

```rust
// partial_witness_actor.rs L933-947
let random_chunk_producer = {
    let mut chunk_producers = self
        .epoch_manager
        .get_epoch_chunk_producers_for_shard(&key.epoch_id, key.shard_id)?;
    chunk_producers.swap_remove(rand::thread_rng().gen_range(0..chunk_producers.len()))
};
let request = ContractCodeRequest::new(key.clone(), missing_contract_hashes, ...);
self.network_adapter.send(... NetworkRequests::ContractCodeRequest(random_chunk_producer, request));
```

No retry loop, no timeout, no fallback to another producer.

**Step 2 – state machine has no recovery path**

`set_requested_contracts` transitions `Unknown → Requested` and then ignores all subsequent calls:

```rust
// partial_witness_tracker.rs L223-234
AccessedContractsState::Requested { .. } | AccessedContractsState::Received(_) => {
    tracing::warn!(target: "client", "already received accessed contract hashes");
}
```

`try_finalize` explicitly excludes `Requested` from the ready set:

```rust
// partial_witness_tracker.rs L284
AccessedContractsState::Unknown | AccessedContractsState::Received(_)
```

`Requested` is an absorbing state with no exit unless a `ContractCodeResponse` arrives.

**Step 3 – producer silently drops the request without responding**

In `handle_contract_code_request`, the dedup entry is written *before* the `MAX_CONTRACTS_PER_REQUEST` guard:

```rust
// partial_witness_actor.rs L1130-1134
self.processed_contract_code_requests.push(processed_requests_key, ());

if request.contracts().len() > MAX_CONTRACTS_PER_REQUEST {
    return Ok(());   // silent drop, no response sent
}
```

Other silent-drop paths: `derive_main_transition_key` returns `None` (L1147), `StateTransitionData` absent (L1155-1162), `main_transition` mismatch (L1142-1143). In every case `Ok(())` is returned with no `ContractCodeResponse` sent.

**Step 4 – permanent liveness failure**

Because the dedup entry was already written, even if the validator could retry (it cannot, due to the state machine), the same producer would reject the retry as a duplicate. The validator's `CacheEntry` stays in `Requested`, `try_finalize` returns `None` indefinitely, and no endorsement is ever emitted for that chunk.

### Impact Explanation

A chunk validator that enters `Requested` state and receives no response cannot endorse the chunk. If the condition that causes the silent drop is deterministic (e.g., `MAX_CONTRACTS_PER_REQUEST` exceeded because the chunk calls many distinct contracts not in cache), **all** validators that are missing those contracts will have their requests silently dropped by the same producer, causing a coordinated endorsement failure. Without enough endorsements the chunk is not included in the next block, constituting a liveness failure in the stateless-validation path.

### Likelihood Explanation

The `ExcludeContractCodeFromStateWitness` feature is production-enabled. A chunk that calls many distinct contracts (e.g., a DeFi aggregator touching N > `MAX_CONTRACTS_PER_REQUEST` accounts) will trigger the silent-drop path on every producer that receives the request. Validators whose compiled-contract cache is cold (e.g., after a restart or epoch boundary) will all be affected simultaneously. The condition is reachable by any unprivileged user who can submit transactions.

### Recommendation

1. **Retry with a different producer**: After a timeout, if `AccessedContractsState` is still `Requested`, select a different producer and re-send the request. The state machine should allow transitioning back from `Requested` to retry.
2. **Respond to oversized requests with a partial or error response**: Instead of silently dropping requests that exceed `MAX_CONTRACTS_PER_REQUEST`, send an explicit error response so the validator knows to retry or fall back.
3. **Move the dedup-cache write after all validation checks**: The `processed_contract_code_requests.push` call should occur only after the request is confirmed to be serviceable, so that a silently-dropped request does not poison the dedup cache.
4. **Broadcast to all producers**: Analogously to how the external report recommends allowing multiple borrowers to cumulatively cover a debt, consider sending the `ContractCodeRequest` to all chunk producers for the shard (or at least a quorum), accepting the first valid response.

### Proof of Concept

1. Enable `ExcludeContractCodeFromStateWitness`.
2. Submit a transaction that calls N > `MAX_CONTRACTS_PER_REQUEST` distinct contracts whose code is not in any validator's compiled-contract cache.
3. The chunk producer applies the chunk and sends `ChunkContractAccesses` with N hashes to each chunk validator.
4. Each validator calls `handle_chunk_contract_accesses`, computes `missing_contract_hashes` = all N hashes, transitions `AccessedContractsState` to `Requested`, and sends a `ContractCodeRequest` with N contracts to one random producer.
5. The producer receives the request, writes the dedup entry, then hits `request.contracts().len() > MAX_CONTRACTS_PER_REQUEST` and returns `Ok(())` — no response is sent.
6. Each validator's `CacheEntry` stays in `Requested`. When enough witness parts arrive, `try_finalize` returns `None` because `Requested` is not in the ready set.
7. No validator emits a `ChunkEndorsement`. The chunk is not included.

        let contracts_ready = matches!(
            self.accessed_contracts,
            // We consider `Unknown` to be ready state for the following reasons:

            // - `ChunkContractAccesses` message might have been lost or delayed by
            //   the network. In this case it is better to proceed with a best-effort
            //   attempt to validate witness since in most cases we have all contracts
            //   available in the compiled contracts cache.
            AccessedContractsState::Unknown | AccessedContractsState::Received(_)
        );
        if !(parts_ready && contracts_ready) {
            return None;
