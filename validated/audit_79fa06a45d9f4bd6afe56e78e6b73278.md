### Title
Unchecked `max_price_per_unit` Upper Bound Causes Felt Field Overflow in Fee Cap Computation, Enabling Sequencer to Overcharge Users - (File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo)

---

### Summary

`compute_max_possible_fee` performs felt-field multiplication of `max_amount × max_price_per_unit` without bounding `max_price_per_unit` to the protocol-specified u128 range. Because `pack_resource_bounds` only asserts `assert_nn(max_price_per_unit)` (non-negative, i.e., ≤ (P−1)/2 ≈ 2^250), a user can submit a V3 transaction with `max_price_per_unit` far exceeding 2^128−1. The resulting product wraps around the Stark field prime P, producing an incorrect `max_fee` felt value. The OS then uses this corrupted cap to bound the actual fee transfer, either allowing the sequencer to drain the user's balance or making the transaction permanently unprocessable.

---

### Finding Description

**Root cause — missing upper-bound check in `pack_resource_bounds`:**

In `transaction_hash/transaction_hash.cairo`, `pack_resource_bounds` validates:

```cairo
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);   // ✓ bounded
assert_nn(resource_bounds.max_price_per_unit);             // ✗ only non-negative, no upper bound
```

`assert_nn` only guarantees the value is in `[0, (P−1)/2]` ≈ `[0, 2^250]`. The StarkNet protocol specifies `max_price_per_unit` as a u128, but the OS never enforces `assert_nn_le(max_price_per_unit, 2**128 - 1)`.

**Propagation into fee computation:**

`compute_max_possible_fee` in `transaction_impls.cairo` then computes:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
```

All arithmetic is modular over the Stark prime P. With `max_amount` up to 2^64−1 and `max_price_per_unit` up to ~2^250, the true product can reach ~2^314, which wraps around P to an arbitrary felt value — potentially much larger or much smaller than the intended fee cap.

**Fee enforcement uses the corrupted cap:**

```cairo
assert_nn_le(calldata.amount.low, max_fee);   // max_fee is the wrapped, incorrect value
```

The sequencer-controlled hint `LoadActualFee` sets `low_actual_fee`, and the OS only enforces `actual_fee ≤ max_fee`. If `max_fee` is a large wrapped value (e.g., close to P), the sequencer can set `low_actual_fee` to any value up to `min(max_fee, user_balance)`, draining the user's fee token balance far beyond their intended limit.

---

### Impact Explanation

**Critical — Direct loss of funds.**

A user who submits a V3 transaction with `max_price_per_unit > 2^128−1` (whether due to a wallet bug, off-by-one in unit conversion, or deliberate crafting) will have their fee cap computed incorrectly by the OS. If the wrapped `max_fee` is larger than the user's true intended cap, the sequencer is permitted by the OS proof to charge up to the user's entire fee token balance. The ERC20 transfer executes with `Uint256(low=low_actual_fee, high=0)` for any `low_actual_fee ≤ max_fee`, and the proof remains valid. The user loses funds beyond what they authorized.

Conversely, if the wrapped value is smaller than the actual execution cost, the OS rejects the proof, making the transaction permanently unprocessable — a localized but reachable network disruption for affected transactions.

---

### Likelihood Explanation

Any unprivileged V3 transaction sender is the entry point. The trigger requires `max_price_per_unit > 2^128−1`. This can occur:

1. **Accidentally**: Wallet software that computes `max_price_per_unit` in a different unit (e.g., Gwei vs. Wei analog) or has an integer overflow in its own fee estimation.
2. **Deliberately**: An attacker who wants to craft a transaction that the OS will process with an inflated fee cap, relying on a colluding or compromised sequencer to extract the excess.

The OS program does not reject such transactions at hash-verification time (the hash computation accepts any `assert_nn`-passing value), so the malformed transaction reaches `charge_fee` and the corrupted cap is enforced.

---

### Recommendation

Add an explicit upper-bound check on `max_price_per_unit` in `pack_resource_bounds` (called during hash verification, before fee computation):

```cairo
// In transaction_hash/transaction_hash.cairo, pack_resource_bounds:
assert_nn_le(resource_bounds.max_amount, 2 ** 64 - 1);
assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 128 - 1);  // ADD THIS
```

Additionally, add a corresponding guard in `compute_max_possible_fee` or document that it relies on the invariant established by `pack_resource_bounds`. This ensures the product `max_amount × max_price_per_unit` is at most `(2^64−1) × (2^128−1) < 2^192 ≪ P`, eliminating field overflow.

---

### Proof of Concept

1. Craft a V3 invoke transaction with:
   - `l1_gas_bounds.max_amount = 1`
   - `l1_gas_bounds.max_price_per_unit = B` where `B = (P − 1) / 2` (maximum `assert_nn`-passing value, ≈ 2^250)
   - `l2_gas_bounds` and `l1_data_gas_bounds` set to zero amounts

2. The transaction hash is computed and signed. `pack_resource_bounds` passes because `assert_nn(B)` holds (B ≤ (P−1)/2).

3. `compute_max_possible_fee` computes `1 × B mod P = B` — a value close to P/2 ≈ 2^250, far exceeding any realistic fee.

4. `charge_fee` calls `assert_nn_le(low_actual_fee, B)`. The sequencer sets `low_actual_fee = user_balance` (up to 2^128−1). The assertion passes since `user_balance ≤ B`.

5. The ERC20 transfer executes for `Uint256(low=user_balance, high=0)`, draining the user's entire fee token balance. The OS proof is valid.

```text
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```
