### Title
Cross-Chain Replay of MPC Metadata Signature Enables Unauthorized Token Deployment on Solana — (`solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`, `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

The Borsh-encoded payload that the MPC key signs for `deployToken` on EVM is byte-for-byte identical to the payload that Solana's `deploy_token` verifies. Neither encoding includes a destination chain ID. An unprivileged relayer can extract the MPC signature from a completed EVM `deployToken` transaction and replay it verbatim on Solana's `deploy_token`, creating an unauthorized canonical wrapped SPL mint for the token without any Solana-specific MPC signing event.

---

### Finding Description

**EVM serialization** — `OmniBridge.deployToken` (lines 142–148):

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),  // = 0x01
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) revert InvalidSignature();
``` [1](#0-0) 

`BridgeTypes.PayloadType.Metadata` is enum index `1` (TransferMessage=0, Metadata=1, ClaimNativeFee=2). [2](#0-1) 

`Borsh.encodeString` emits a 4-byte little-endian length prefix followed by the UTF-8 bytes. [3](#0-2) 

**Solana serialization** — `DeployTokenPayload::serialize_for_near` (lines 19–26):

```rust
IncomingMessageType::Metadata.serialize(&mut writer)?;  // Borsh enum index = 0x01
self.serialize(&mut writer)?;  // borsh(token: String, name: String, symbol: String, decimals: u8)
``` [4](#0-3) 

`IncomingMessageType::Metadata` is also enum index `1` (InitTransfer=0, Metadata=1). [5](#0-4) 

Both produce the identical byte sequence:
```
[0x01] [4-byte-LE len(token)] [token] [4-byte-LE len(name)] [name] [4-byte-LE len(symbol)] [symbol] [decimals]
```

**Signature verification** — Solana `SignedPayload::verify_signature`:

```rust
let hash = keccak::hash(&serialized);          // keccak256 — same as EVM
secp256k1_recover(&hash.to_bytes(), ...)       // raw secp256k1 — same as EVM ECDSA.recover
require!(signer.0 == *derived_near_bridge_address, ...);
``` [6](#0-5) 

OpenZeppelin's `ECDSA.recover(hash, sig)` (without `toEthSignedMessageHash`) is also raw secp256k1 recovery over a keccak256 hash — no Ethereum prefix is added. Both chains verify against the same MPC-derived key (`nearBridgeDerivedAddress` / `derived_near_bridge_address`). The hash inputs are identical, so any valid EVM signature is also a valid Solana signature.

**No replay guard on `deploy_token`** — the Solana instruction handler:

```rust
pub fn deploy_token(ctx: Context<DeployToken>, data: SignedPayload<DeployTokenPayload>) -> Result<()> {
    data.verify_signature((), &ctx.accounts.common.config.derived_near_bridge_address)?;
    ctx.accounts.initialize_token_metadata(data.payload)?;
    Ok(())
}
``` [7](#0-6) 

There is no nonce, no used-signature bitmap, and no chain-ID field. The only idempotency guard is that the mint PDA (`[WRAPPED_MINT_SEED, hash(token)]`) is created with `init`, so the call fails if the mint already exists — but that only prevents a second replay, not the first. [8](#0-7) 

---

### Impact Explanation

An attacker who observes a completed EVM `deployToken` transaction for token T can:

1. Read the `signatureData` and `metadata` fields from the EVM calldata (fully public).
2. Submit the identical `SignedPayload<DeployTokenPayload>` to Solana's `deploy_token`.
3. The signature passes `verify_signature` because the serialized bytes, keccak256 hash, and secp256k1 recovery are identical.
4. A canonical wrapped SPL mint is created at the deterministic PDA for token T, with the bridge's authority PDA as mint authority.

The created mint is indistinguishable from one created through a legitimate Solana-specific MPC signing event. Any subsequent legitimate `finalize_transfer` for token T on Solana will mint against this unauthorized mint, because `finalize_transfer` only checks `mint.mint_authority == authority` — it does not verify that the mint was created through an authorized `deploy_token` flow. [9](#0-8) 

This violates the invariant that a signed deploy-token payload must be bound to a single destination chain.

---

### Likelihood Explanation

- All inputs are publicly observable on-chain (EVM calldata is not encrypted).
- No privileged access, leaked key, or colluding MPC signer is required.
- The attacker only needs to call a public Solana instruction with data copied from an EVM transaction.
- The attack is deterministic and locally testable.

---

### Recommendation

Include a destination chain ID in the signed payload on both sides:

**Solana** — add `SOLANA_OMNI_BRIDGE_CHAIN_ID` to `serialize_for_near`:
```rust
fn serialize_for_near(&self, _params: ()) -> Result<Vec<u8>> {
    let mut writer = ...;
    IncomingMessageType::Metadata.serialize(&mut writer)?;
    writer.write_all(&[SOLANA_OMNI_BRIDGE_CHAIN_ID])?;  // <-- add chain ID
    self.serialize(&mut writer)?;
    ...
}
```

**EVM** — add `omniBridgeChainId` to the Borsh-encoded blob in `deployToken`:
```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),   // <-- add chain ID
    Borsh.encodeString(metadata.token),
    ...
);
```

The MPC signer must include the destination chain ID when producing signatures, so a signature for EVM cannot be replayed on Solana and vice versa. Note that `finTransfer` already includes `omniBridgeChainId` in its payload — the same pattern should be applied to `deployToken`. [10](#0-9)

### Proof of Concept

```rust
// Pseudocode unit test
let mpc_key = test_mpc_keypair();
let token = "token.near";
let name = "My Token";
let symbol = "MTK";
let decimals = 18u8;

// Step 1: Construct the EVM payload and sign it (simulating what the MPC does for EVM)
let evm_payload = [
    &[1u8],                          // PayloadType::Metadata
    &borsh_encode_string(token),
    &borsh_encode_string(name),
    &borsh_encode_string(symbol),
    &[decimals],
].concat();
let evm_hash = keccak256(&evm_payload);
let evm_sig = mpc_key.sign_raw(&evm_hash);  // raw secp256k1, no Ethereum prefix

// Step 2: Replay the EVM signature on Solana deploy_token
let solana_signed_payload = SignedPayload {
    payload: DeployTokenPayload { token, name, symbol, decimals },
    signature: evm_sig,
};
// This call succeeds — the signature verifies because the serialization is identical
let result = deploy_token(ctx, solana_signed_payload);
assert!(result.is_ok());

// Step 3: Verify the unauthorized mint was created
let mint_pda = Pubkey::find_program_address(
    &[WRAPPED_MINT_SEED, &hash(token.as_bytes())],
    &program_id,
);
assert!(mint_pda_exists_on_chain());
```

```rust
    pub fn verify_signature(
        &self,
        params: P::AdditionalParams,
        derived_near_bridge_address: &[u8; 64],
    ) -> Result<()> {
        let serialized = self.payload.serialize_for_near(params)?;
        let hash = keccak::hash(&serialized);

        let signature_bytes = &self.signature[0..64];

        let signature = libsecp256k1::Signature::parse_standard_slice(signature_bytes)
            .map_err(|_| ProgramError::InvalidArgument)?;
        require!(!signature.s.is_high(), ErrorCode::MalleableSignature);

        let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)
            .map_err(|_| error!(ErrorCode::SignatureVerificationFailed))?;

        require!(
            signer.0 == *derived_near_bridge_address,
            ErrorCode::SignatureVerificationFailed
        );

        Ok(())
    }
```
