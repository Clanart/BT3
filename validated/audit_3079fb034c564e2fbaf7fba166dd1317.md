Audit Report

## Title
Missing L2 Sequencer Uptime Check in `PriceProviderL2` and `ProtectedPriceProviderL2` Allows Stale-Price Swaps After Sequencer Restart — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

## Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are deployed on L2 networks and supply bid/ask prices to every pool swap via `getBidAndAskPrice()`. Both contracts implement only a timestamp-based staleness check (`_isStale`) with `MAX_TIME_DELTA` configurable up to 7 days, but neither checks whether the L2 sequencer is live before trusting stored oracle data. When the sequencer restarts after an outage shorter than `MAX_TIME_DELTA`, the pre-outage Pyth Lazer price stored in `OracleBase` passes the staleness check and reaches pool swaps, enabling an attacker to trade at a stale price at LP expense.

## Finding Description

`PriceProviderL2._getBidAndAskPrice()` reads price data from the offchain oracle and applies only a timestamp-based staleness check:

```solidity
// PriceProviderL2.sol L215-217
if (_isStale(refTime, block.timestamp, MAX_TIME_DELTA, FUTURE_TOLERANCE)) {
    return (0, type(uint128).max);
}
```

`_isStale` returns `true` only when `(block.timestamp - refTime) > MAX_TIME_DELTA`. `MAX_TIME_DELTA` is bounded at construction to `(0, 7 days]`. If the sequencer was down for a duration shorter than `MAX_TIME_DELTA`, the pre-outage oracle price stored in `OracleBase.oracleData` is still considered fresh when the sequencer restarts.

The Pyth Lazer oracle (`PythOracle` / `LazerConsumer`) is a push-based system: keepers call the `fallback()` to push signed price updates, which are stored via `_verifyAndStore`. During a sequencer outage, no new updates can land on-chain. When the sequencer restarts, the last stored `OracleData.timestampMs` is from before the outage. `OracleBase._readPrice` converts this to `refTime` in seconds and returns it to `PriceProviderL2`, which then passes the staleness check if the outage was shorter than `MAX_TIME_DELTA`.

`ProtectedPriceProviderL2` has the identical gap: `_computeBidAsk` calls the same `_isStale` with no sequencer liveness gate.

The registry ABI for `ChainlinkVerifierL2` (present in `registry.json`) exposes `sequencerUptimeFeed` and `GRACE_PERIOD` public functions, confirming the protocol is aware of the sequencer uptime pattern for L2 deployments, but this check is absent from the Pyth/compressed-oracle provider path.

## Impact Explanation

During the sequencer-restart window — before keepers push a fresh Pyth Lazer update — `getBidAndAskPrice()` returns a stale pre-outage bid/ask. A trader can swap against a stale favorable price, extracting value from LP positions. If the asset price dropped significantly during the outage, the pool quotes a bid above the true market price, allowing a trader to sell at an inflated rate — a direct loss of LP principal. This matches the "Bad-price execution: stale bid/ask quote reaches a pool swap" impact gate.

## Likelihood Explanation

L2 sequencer outages are documented historical events on Arbitrum. The attack requires no special privilege — any user can submit a swap transaction immediately after sequencer restart. The window lasts until keepers push a fresh Pyth Lazer update. `MAX_TIME_DELTA` up to 7 days means even a multi-hour outage leaves prices appearing fresh. The attack is repeatable on every sequencer restart event.

## Recommendation

Add a sequencer uptime check to `getBidAndAskPrice()` in both `PriceProviderL2` and `ProtectedPriceProviderL2`, mirroring the pattern already present in the registry for `ChainlinkVerifierL2`:

```solidity
AggregatorV3Interface public immutable sequencerUptimeFeed;
uint256 public constant GRACE_PERIOD = 3600; // 1 hour

function _isSequencerActive() internal view returns (bool) {
    (, int256 answer, uint256 startedAt,,) = sequencerUptimeFeed.latestRoundData();
    if (answer == 1) return false; // sequencer is down
    if (block.timestamp - startedAt < GRACE_PERIOD) return false; // in grace period
    return true;
}

function getBidAndAskPrice() external override returns (uint128 bid, uint128 ask) {
    if (!_isSequencerActive()) revert SequencerDown();
    (bid, ask) = _getBidAndAskPrice();
    if (bid == 0 || ask == type(uint128).max) revert FeedStalled();
}
```

On chains without a sequencer uptime feed, deploy the non-L2 variants (`PriceProvider`, `ProtectedPriceProvider`).

## Proof of Concept

1. Pool on Arbitrum is configured with `PriceProviderL2`, `MAX_TIME_DELTA = 3600` (1 hour).
2. At `t=0`, ETH/USDC price is 3000 USDC. Pyth Lazer keeper pushes update; `OracleBase.oracleData[feedId].timestampMs` records `t=0`.
3. At `t=1`, Arbitrum sequencer goes down. No new Pyth Lazer updates can be pushed.
4. Off-chain, ETH price drops to 2000 USDC.
5. At `t=1800` (30 min later), sequencer restarts. `block.timestamp - refTime = 1800 < MAX_TIME_DELTA = 3600` → `_isStale` returns `false`, staleness check passes.
6. Attacker immediately calls swap — sells USDC to buy ETH at the stale 3000 USDC/ETH ask price.
7. Pool pays out ETH valued at 3000 USDC when true market value is 2000 USDC — LP suffers a ~33% loss on the trade.
