### Title
P2P Transaction Admission Bypasses `accept_new_txs` Shutdown Check — (`File: crates/apollo_mempool_p2p/src/runner/mod.rs`)

---

### Summary

The HTTP server enforces an `accept_new_txs` flag before forwarding transactions to the gateway. The `MempoolP2pRunner`, which receives transactions from the P2P gossip network, forwards them directly to the same gateway **without** checking this flag. When an operator sets `accept_new_txs = false` to halt transaction admission, the P2P path remains open, allowing any peer to inject transactions into the mempool — exactly the analog of the missing `whenNotPaused` modifier in the original report.

---

### Finding Description

`HttpServerDynamicConfig.accept_new_txs` is the sequencer's admission-control flag. Both HTTP endpoints (`add_rpc_tx` and `add_tx`) read this flag and call `check_new_transactions_are_allowed` before touching the gateway:

```rust
// crates/apollo_http_server/src/http_server.rs:176-177
let HttpServerDynamicConfig { accept_new_txs, .. } = app_state.get_dynamic_config();
check_new_transactions_are_allowed(accept_new_txs)?;
```

`check_new_transactions_are_allowed` returns `Err(HttpServerError::DisabledError())` when the flag is `false`, blocking the transaction before it reaches the gateway.

The `MempoolP2pRunner::start` loop receives `RpcTransactionBatch` messages from the libp2p gossipsub layer and calls `self.gateway_client.add_tx(...)` directly:

```rust
// crates/apollo_mempool_p2p/src/runner/mod.rs:114-116
gateway_futures.push(self.gateway_client.add_tx(
    GatewayInput { rpc_tx, message_metadata: Some(broadcasted_message_metadata.clone()) }
));
```

There is no `accept_new_txs` check anywhere in `MempoolP2pRunner`. The gateway's own `add_tx_inner` also does not check this flag — it only performs stateless validation, stateful validation, and mempool insertion. The flag is exclusively enforced at the HTTP layer.

---

### Impact Explanation

**High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

When `accept_new_txs = false` is set (e.g., during a controlled shutdown, a revert window, or a maintenance pause), the intended invariant is that no new transactions enter the mempool. The P2P path violates this invariant: any connected peer can gossip a transaction and have it admitted into the mempool, altering the mempool state, consuming mempool capacity, and potentially influencing the next block's content — all while the operator believes admission is closed.

---

### Likelihood Explanation

Any peer in the P2P network can trigger this. The `MempoolP2pRunner` is always running alongside the HTTP server; there is no separate flag or lifecycle gate on the P2P runner. The `accept_new_txs` flag is dynamically polled by the HTTP server every `dynamic_config_poll_interval` (default 1 second), but the P2P runner never reads it. The trigger requires only a live P2P connection, which is the normal operating state of any sequencer node.

---

### Recommendation

The `MempoolP2pRunner` should read the `accept_new_txs` flag (or an equivalent admission-control signal) before forwarding P2P-received transactions to the gateway. One approach is to inject a `SharedConfigManagerClient` into `MempoolP2pRunner` (mirroring how the HTTP server polls it) and check the flag in the `broadcasted_topic_server` branch before pushing to `gateway_futures`. Alternatively, the gateway itself could enforce the flag for all callers, removing the split-enforcement pattern entirely.

---
