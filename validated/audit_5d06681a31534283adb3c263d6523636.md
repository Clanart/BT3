### Title
DataLayer `ProofOfInclusion::valid()` Tautological Final Check Accepts Forged Inclusion Proofs - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary

`ProofOfInclusion::valid()` in `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs` contains a tautological final check that makes the function verify only internal self-consistency of the proof chain, never the proof's root hash against any external trusted value. As a result, any attacker who can supply a `ProofOfInclusion` object — including via the `Streamable` deserialization path exposed through Python/wasm bindings — can forge an inclusion proof for an arbitrary key-value pair that passes `valid()`.

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;  // existing_hash = layer.combined_hash
    }

    existing_hash == self.root_hash()    // TAUTOLOGY
}
```

`root_hash()` is:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // exactly what existing_hash equals after the loop
    } else {
        self.node_hash              // exactly what existing_hash equals when layers is empty
    }
}
```

After the loop, `existing_hash` equals the last `layer.combined_hash` (because the loop already asserted `calculated_hash == layer.combined_hash` and then set `existing_hash = calculated_hash`). `root_hash()` also returns `last.combined_hash`. Therefore `existing_hash == self.root_hash()` is **always `true`** when the loop completes without returning `false`.

**Two concrete forgery paths:**

1. **Zero-layer proof (trivially forged):** When `layers` is empty, the loop body never executes, `existing_hash` stays as `self.node_hash`, and `root_hash()` returns `self.node_hash`. The final check becomes `node_hash == node_hash`, which is always `true`. An attacker can claim any key-value pair is in any tree by constructing a `ProofOfInclusion { node_hash: H(fake_leaf), layers: vec![] }` — `valid()` returns `true` unconditionally.

2. **Internally consistent but fake chain:** An attacker can build a multi-layer proof where each `combined_hash` is correctly computed from the previous hash and a chosen `other_hash`, but the final `combined_hash` does not match the actual tree root. `valid()` will return `true` because it only checks the chain's internal consistency, never comparing the terminal hash against a trusted external root.

`ProofOfInclusion` derives `Streamable`, so it can be deserialized from arbitrary bytes via `from_bytes()` / `from_bytes_unchecked()` in both the Rust and Python APIs. The Python binding exposes `valid()` as the sole verification method with no separate "verify against trusted root" function.

### Impact Explanation

Any DataLayer client that receives a `ProofOfInclusion` from an untrusted source (e.g., a DataLayer server) and calls `proof.valid()` to verify it will accept forged proofs. An attacker can prove that arbitrary key-value pairs are included in a DataLayer tree when they are not. This allows untrusted input to prove invalid state — matching the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

### Likelihood Explanation

The `ProofOfInclusion` struct is `Streamable` and fully exposed through Python and wasm bindings. The `valid()` method is the only verification function provided; there is no `verify_against_root(trusted_root: Hash)` API. Any caller who follows the natural API usage pattern of `proof.valid()` is vulnerable. The Python test suite itself calls `proof_of_inclusion.valid()` without separately checking the root hash, demonstrating the expected usage pattern.

### Recommendation

`valid()` must accept an external trusted root hash parameter and verify the terminal hash against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root   // compare against external trusted root
}
```

The existing `valid()` method should either be removed or deprecated, and all call sites updated to pass the trusted root obtained from the blockchain or a trusted local `MerkleBlob`.
