### Title
Unguarded `sweepToken` and `unwrapWETH9` with Arbitrary `recipient` Allow Anyone to Drain Router/Adder Balances - (File: metric-periphery/contracts/base/PeripheryPayments.sol)

### Summary

`sweepToken` and `unwrapWETH9` in `PeripheryPayments` are `public payable` with no access control and accept a fully caller-controlled `recipient` address. Any ERC-20 or WETH balance that lands on `MetricOmmSimpleRouter` or `MetricOmmPoolLiquidityAdder` — whether from an accidental transfer or a two-step usage pattern — can be redirected to an attacker's address by anyone.

### Finding Description

`PeripheryPayments` exposes three withdrawal helpers: [1](#0-0) 

`unwrapWETH9` and `sweepToken` are both `public payable` with no `msg.sender` check and accept an arbitrary `recipient` argument. `refundETH` is `external payable` with no access control and sends the full ETH balance to `msg.sender`. [2](#0-1) 

Both `MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` inherit this contract: [3](#0-2) [4](#0-3) 

The `receive()` guard only blocks direct ETH transfers from non-WETH senders: [5](#0-4) 

It does **not** prevent ERC-20 tokens from being sent directly to the contract. Any ERC-20 balance on the router is immediately drainable by an unprivileged caller.

### Impact Explanation

**Direct loss of user principal.** Two concrete paths:

1. **Accidental ERC-20 transfer**: A user mistakenly sends tokens directly to the router address. An attacker calls `sweepToken(token, 0, attacker)` and receives the full balance. The `recipient` parameter is unchecked, so even if the victim tries to recover their tokens first, an attacker can front-run with a different `recipient`.

2. **Two-step WETH/ETH pattern**: A user sends `exactInput(..., recipient: address(router))` in one transaction to land WETH on the router, intending to follow up with `unwrapWETH9(0, user)` in a second transaction. An attacker observing the mempool front-runs the second transaction with `unwrapWETH9(0, attacker)`, stealing the full WETH balance.

The `MetricOmmPoolLiquidityAdder` documentation explicitly describes the multicall-with-ETH pattern and `refundETH` usage: [6](#0-5) 

Users who do not batch these into a single multicall are exposed.

### Likelihood Explanation

- Accidental ERC-20 transfers to router addresses are a well-documented real-world occurrence.
- The two-step pattern is a natural mistake for users unfamiliar with the multicall requirement.
- No special privilege or setup is required; any EOA can call `sweepToken` or `unwrapWETH9` with an arbitrary `recipient` at any time.
- The `receive()` guard provides a false sense of security (it only blocks ETH, not ERC-20).

### Recommendation

Restrict `sweepToken`, `unwrapWETH9`, and `refundETH` so that the `recipient` is always `msg.sender`, or add a `checkDeadline`/`onlyAuthorized` modifier. The simplest fix matching the Uniswap v3 pattern is to remove the `recipient` parameter from `sweepToken` and `unwrapWETH9` and always send to `msg.sender`, eliminating the front-running redirect vector entirely.

### Proof of Concept

```solidity
// 1. Alice accidentally sends 1000 USDC to the router
usdc.transfer(address(router), 1000e6);

// 2. Attacker (Bob) calls sweepToken with his own address as recipient
// No access control, no msg.sender check, arbitrary recipient accepted
router.sweepToken(address(usdc), 0, bob);

// Result: Bob receives 1000 USDC, Alice loses her funds
assertEq(usdc.balanceOf(bob), 1000e6);
assertEq(usdc.balanceOf(address(router)), 0);
```

The existing payment test confirms the router holds a real balance during operations and that `sweepToken` transfers the full balance to any specified address: [7](#0-6)

  function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);

    if (balanceWETH > 0) {
      IWETH9(WETH).withdraw(balanceWETH);
      _transferETH(recipient, balanceWETH);
    }
  }

  /// @inheritdoc IPeripheryPayments
  function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceToken = IERC20(token).balanceOf(address(this));
    if (balanceToken < amountMinimum) revert InsufficientToken(token, amountMinimum, balanceToken);

    if (balanceToken > 0) {
      IERC20(token).safeTransfer(recipient, balanceToken);
    }
  }
