### Title
Plain `ft_transfer` Failure in `fin_transfer_send_tokens_callback` Permanently Locks User Funds With No Recovery Path - (File: `near/omni-bridge/src/lib.rs`)

### Summary

When `send_tokens` dispatches a plain `ft_transfer` (non-deployed token, empty `msg`), any failure of that promise is silently ignored by `fin_transfer_send_tokens_callback` because `is_refund_required(false)` unconditionally returns `false` without inspecting the promise result. The transfer is already marked finalized via `add_fin_transfer` before `send_tokens` is called, so the user's tokens are permanently locked inside the bridge contract with no retry or recovery mechanism.

### Finding Description

The `process_fin_transfer_to_near` function calls `add_fin_transfer` synchronously (committing finalization state), then dispatches `send_tokens`, then chains `fin_transfer_send_tokens_callback` as the callback. [1](#0-0) 

Inside `send_tokens`, when the token is not a deployed token and `msg` is empty, a plain `ft_transfer` is issued: [2](#0-1) 

The `fin_transfer_send_tokens_callback` is registered with `is_ft_transfer_call = !msg.is_empty()`, which is `false` for the empty-msg path: [3](#0-2) 

Inside `is_refund_required`, when `is_ft_transfer_call` is `false`, the function returns `false` immediately **without reading the promise result at all**: [4](#0-3) 

Therefore, even when `ft_transfer` fails (promise result = `Failed`), `fin_transfer_send_tokens_callback` takes the success branch, emits `FinTransferEvent`, and pays the fee to the relayer: [5](#0-4) 

The recovery path (`remove_fin_transfer`, `revert_lock_actions`) is never reached: [6](#0-5) 

Because `add_fin_transfer` was committed in a prior NEAR receipt, the transfer ID is permanently finalized. The relayer cannot re-submit `fin_transfer` for the same proof, and there is no admin rescue function visible in the contract.

The same silent-failure path applies to the wNEAR branch: if `near_withdraw` fails, `near_withdraw_callback` panics, and `fin_transfer_send_tokens_callback` is still called with `is_ft_transfer_call = false`, again returning `false` from `is_refund_required`. [7](#0-6) 

### Impact Explanation

For any non-deployed token (e.g., USDC, USDT, wBTC) bridged to a NEAR recipient with an empty `msg`:

- The source-chain funds are already consumed (locked or burned on EVM/Solana/StarkNet).
- The NEAR transfer is marked finalized.
- The tokens remain in the bridge contract's balance, inaccessible to the user.
- No admin or user function exists to recover or re-route them.

This constitutes **permanent, irrecoverable lock of user funds** in the bridge.

### Likelihood Explanation

The trigger condition is a `ft_transfer` failure at finalization time. Realistic causes include:

1. **Token blacklist** — USDC and USDT implement account-level blacklists. A user blacklisted after initiating a cross-chain transfer (which can take minutes to hours) will have their `ft_transfer` rejected on NEAR.
2. **Deleted recipient account** — A NEAR account can be deleted between transfer initiation and finalization, causing `ft_transfer` to fail.
3. **Token contract pause** — Some NEP-141 tokens support pausing transfers.

All of these are realistic, user-reachable conditions that require no privileged access to trigger.

### Recommendation

In `fin_transfer_send_tokens_callback`, check the promise result regardless of `is_ft_transfer_call`. When `is_ft_transfer_call` is `false`, read `env::promise_result_checked(0, ...)` and treat a `Failed` result as requiring the refund path (call `remove_fin_transfer`, `revert_lock_actions`, emit `FailedFinTransferEvent`). This mirrors the existing recovery logic already present for the `ft_transfer_call` case.

Alternatively, unify the two branches so that any promise failure — whether from `ft_transfer`, `ft_transfer_call`, or `near_withdraw` — triggers the same cleanup and allows the relayer to retry finalization.

### Proof of Concept

1. User holds USDC (a non-deployed, non-bridge token) and calls `initTransfer` on EVM, locking 1000 USDC. The EVM contract emits `InitTransfer`.
2. Before the relayer finalizes on NEAR, the USDC issuer blacklists the user's NEAR address.
3. Relayer calls `fin_transfer` on NEAR with the proof. `fin_transfer_callback` runs: `add_fin_transfer` commits the transfer ID as finalized; `send_tokens` dispatches `ft_transfer(recipient, 1000)`.
4. The USDC contract rejects `ft_transfer` because the recipient is blacklisted → promise result = `Failed`.
5. `fin_transfer_send_tokens_callback` is called. `is_ft_transfer_call = false` → `is_refund_required(false)` returns `false`. The callback emits `FinTransferEvent` and pays the relayer fee.
6. The transfer ID is permanently finalized. The 1000 USDC sit in the bridge contract. The user's EVM funds are gone. No recovery path exists.
