### Title
Asymmetric `within_margin` Allows Proposer to Inflate L1 Gas Price ~11.1% Above Reference Instead of Intended 10%, Committing Wrong Fee Value to Block Hash - (File: crates/apollo_consensus_orchestrator/src/validate_proposal.rs)

---

### Summary

The `within_margin` function in `validate_proposal.rs` computes the allowed deviation using `number1` (the **proposed** value) as the base for the percentage margin, not `number2` (the locally computed reference). This asymmetry allows a malicious proposer to set `l1_gas_price_fri` and `l1_data_gas_price_fri` up to **~11.11% above** the reference while only allowing ~9.09% below it. The intended symmetric bound is ±10%. The inflated price passes `is_block_info_valid`, is forwarded to `convert_to_sn_api_block_info`, used as the `BlockInfo` for all transaction fee calculations, and is committed to the block hash via `gas_prices_to_hash`.

---

### Finding Description

In `is_block_info_valid`, the validator checks that the proposer's claimed L1 gas prices are within a configured margin of the locally computed reference:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs:302-307
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, l1_gas_price_margin_percent)
    && within_margin(
        l1_data_gas_price_fri_proposed,
        l1_data_gas_price_fri,
        l1_gas_price_margin_percent,
    ))
```

The `within_margin` function is:

```rust
// crates/apollo_consensus_orchestrator/src/validate_proposal.rs:323-333
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;   // ← uses number1 (proposed), not number2 (reference)
    number1.0.abs_diff(number2.0) <= margin
}
```

`number1` is the **proposed** price; `number2` is the **reference** price. The margin is `proposed * 10% / 100`, not `reference * 10% / 100`.

**Algebraic consequence** (with `margin_percent = 10`):

| Direction | Condition | Solved bound |
|---|---|---|
| Proposed > Reference | `proposed − reference ≤ proposed × 0.1` | `proposed ≤ reference / 0.9 ≈ reference × 1.1111` |
| Proposed < Reference | `reference − proposed ≤ proposed × 0.1` | `proposed ≥ reference / 1.1 ≈ reference × 0.9090` |

A symmetric check (margin based on `reference`) would give `[reference × 0.9, reference × 1.1]`. The current asymmetric check gives `[reference × 0.9090, reference × 1.1111]`, allowing the proposer to go **~1.11 percentage points higher** than the protocol intends.

The `l1_gas_price_margin_percent` is set to `10` in all versioned-constants files: [1](#0-0) 

The reference price is the mean of recent L1 blocks computed by `get_price_info`, then clamped by `apply_fee_transformations`: [2](#0-1) 

After passing `is_block_info_valid`, the inflated price is forwarded directly to `convert_to_sn_api_block_info` and used as the `BlockInfo` for execution: [3](#0-2) [4](#0-3) 

The gas prices are then committed to the block hash via `gas_prices_to_hash`, which Poseidon-hashes `l1_gas_price_fri`, `l1_data_gas_price_fri`, and their wei counterparts: [5](#0-4) 

---

### Impact Explanation

**Impact: High — RPC fee estimation and block execution use an authoritative-looking wrong gas price; also fits Critical fee/economic impact.**

1. **Wrong fee calculation**: Every transaction in a block proposed by a malicious validator is charged fees based on a gas price up to 11.11% above the reference instead of the intended 10% ceiling. Users systematically overpay.
2. **Wrong block hash commitment**: The inflated `l1_gas_price_fri` / `l1_data_gas_price_fri` are hashed into the block hash. The committed block hash encodes a price that deviates from what a symmetric protocol would produce.
3. **Wrong RPC fee estimates**: `starknet_estimateFee` and `starknet_simulateTransactions` use the same `BlockInfo` gas prices, returning authoritative-looking inflated estimates.

The `ProposalInit` fields `l1_gas_price_fri` and `l1_data_gas_price_fri` flow directly into `BlockInfo.gas_prices` used by the blockifier for all fee enforcement: [6](#0-5) 

---

### Likelihood Explanation

Any validator can be the proposer in a given consensus round. No special administrative privilege is required. A validator running modified sequencer software can always propose the maximum-allowed inflated price (reference × 1.1111) in every block they lead. With a 10% margin and `l1_gas_price_margin_percent = 10`, the extra 1.11% inflation is small per block but systematic and compounding across all blocks the validator proposes.

The `get_price_info` mean is computed over `number_of_blocks_for_mean` L1 blocks: [7](#0-6) 

The proposer controls the `timestamp` field in `ProposalInit` (within a window), which selects which L1 blocks enter the mean, giving additional fine-grained control over the reference value.

---

### Recommendation

Change the margin base from `number1` (proposed) to `number2` (reference) to make the check symmetric:

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number2.0 * margin_percent) / 100;  // ← use reference, not proposed
    number1.0.abs_diff(number2.0) <= margin
}
```

This enforces a symmetric `[reference × 0.9, reference × 1.1]` window as intended.

---

### Proof of Concept

Let `reference = 1_000_000` fri, `margin_percent = 10`.

**Current (asymmetric) behavior — proposer sets `proposed = 1_111_112`:**
```
margin = 1_111_112 * 10 / 100 = 111_111
|1_111_112 − 1_000_000| = 111_112 > 111_111  → FAILS (just over)
```
Try `proposed = 1_111_111`:
```
margin = 1_111_111 * 10 / 100 = 111_111
|1_111_111 − 1_000_000| = 111_111 ≤ 111_111  → PASSES  (11.1111% above reference)
```

**Symmetric (correct) behavior — same proposed value:**
```
margin = 1_000_000 * 10 / 100 = 100_000
|1_111_111 − 1_000_000| = 111_111 > 100_000  → FAILS  (correctly rejected)
```

The proposer can pass `l1_gas_price_fri = reference × 1.1111` through `is_block_info_valid`. This value is accepted, forwarded to `initiate_validation` → `convert_to_sn_api_block_info` → `BlockInfo`, and used for all fee calculations and the block hash commitment in that block. [8](#0-7) [9](#0-8)

```rust
fn within_margin(number1: GasPrice, number2: GasPrice, margin_percent: u128) -> bool {
    // For small numbers (e.g., less than 10 wei, if margin is 10%), even an off-by-one
    // error might be bigger than the margin, even if it is just a rounding error.
    // We make an exception for such mismatch, and don't bother checking percentages
    // if the difference in price is only one wei.
    if number1.0.abs_diff(number2.0) <= GAS_PRICE_ABS_DIFF_MARGIN {
        return true;
    }
    let margin = (number1.0 * margin_percent) / 100;
    number1.0.abs_diff(number2.0) <= margin
}
```
