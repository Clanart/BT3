Audit Report

## Title
Unguarded `refundETH()` allows any caller to steal excess ETH stranded on the router between transactions — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`refundETH()` has no access control and unconditionally transfers the router's entire native ETH balance to `msg.sender`. The `pay()` function wraps only the exact `value` of ETH needed for a WETH swap, leaving any `msg.value` surplus on the contract after the transaction ends. Any third party can call `refundETH()` in a subsequent transaction to steal that surplus, causing direct loss of user ETH principal.

## Finding Description
`refundETH()` is implemented with no caller restriction: [1](#0-0) 

Inside the swap callback, `pay()` wraps only the exact `value` amount when native ETH is sufficient, leaving any surplus on the router: [2](#0-1) 

When `nativeBalance > value`, the difference `nativeBalance - value` is never returned within `pay()`. It persists on the contract after the transaction completes because the router is a regular (non-transient) contract.

The `receive()` guard only blocks direct ETH transfers from non-WETH senders: [3](#0-2) 

It does not prevent ETH accumulation via `msg.value` on payable entry points (`multicall`, `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`). The `multicall` dispatcher performs no automatic refund after executing sub-calls: [4](#0-3) 

A user who sends excess ETH but omits `refundETH()` from their multicall array leaves the surplus stranded on the router indefinitely, claimable by any address.

## Impact Explanation
Direct loss of user ETH principal. The attacker receives ETH that belongs to the victim with no preconditions other than timing. The amount is bounded only by how much excess ETH the victim sent, which can be arbitrarily large. This meets the Metric OMM Allowed Impact Gate criterion of Critical/High direct loss of user principal.

## Likelihood Explanation
`refundETH()` is an opt-in call that users must explicitly append to their multicall array. Omitting it is a realistic mistake for users interacting directly with the contract or through integrations that do not enforce the pattern. The attack requires no special privileges, no malicious pool, and no non-standard token behavior — only a call to a public function. MEV bots routinely monitor for stranded ETH on router contracts and can execute the steal atomically in the same block.

## Recommendation
Two complementary mitigations:
1. **Track the depositor in transient storage** at the start of `multicall` and restrict `refundETH()` to that address within the same call context.
2. **Auto-refund at the end of `multicall`**: after executing all sub-calls, if `address(this).balance > 0`, transfer the remainder back to `msg.sender` unconditionally, removing the need for users to remember to append the call.

## Proof of Concept
The existing test `test_refundETH_sendsBalanceToCaller` already demonstrates that any arbitrary `swapper` address (not the original depositor) can drain the router's ETH balance by calling `refundETH()`: [5](#0-4) 

End-to-end exploit path:
```
1. User calls:
   router.multicall{value: 1 ether}([
       abi.encodeCall(router.exactInputSingle, (ExactInputSingleParams{
           tokenIn: WETH,
           amountIn: 0.5 ether,   // only 0.5 ETH is wrapped and sent to pool
           ...
       }))
       // refundETH omitted
   ]);

2. After the multicall:
   address(router).balance == 0.5 ether   // surplus sits on router

3. Attacker calls (separate tx, same or next block):
   router.refundETH();

4. Result:
   attacker.balance += 0.5 ether   // stolen from victim
   address(router).balance == 0
```

The correct pattern (shown in `test_multicall_ethInput_exactInputSingle_refundsUnusedEth`) requires the user to explicitly include `refundETH()` as the last call in the multicall array, but this is not enforced by the contract: [6](#0-5)
