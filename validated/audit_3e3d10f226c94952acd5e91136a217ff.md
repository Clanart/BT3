### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Causes Bridge Undercollateralization - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` records and broadcasts the caller-supplied `amount` parameter rather than the actual token balance increase after `safeTransferFrom`. For fee-on-transfer ERC20 tokens, the bridge receives fewer tokens than `amount`, but the cross-chain message instructs the destination to release the full `amount`. This permanently undercollateralizes the bridge vault, eventually causing legitimate `finTransfer` calls to fail and locking user funds.

---

### Finding Description

In `OmniBridge.initTransfer`, for native ERC20 tokens (not bridge tokens, not custom minters), the contract executes:

```solidity
IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
```

and then immediately emits:

```solidity
emit BridgeTypes.InitTransfer(msg.sender, tokenAddress, currentOriginNonce, amount, fee, nativeFee, recipient, message);
``` [1](#0-0) [2](#0-1) 

The `amount` in the event is the **input parameter**, not the actual balance increase. For fee-on-transfer tokens, the bridge receives `amount - transferFee` but the `InitTransfer` event records `amount`. The NEAR bridge reads this event and, when `finTransfer` is called on the destination chain, instructs the release of the full `amount`. The same unchecked `amount` is also encoded into the Wormhole message payload in `OmniBridgeWormhole.initTransferExtension`: [3](#0-2) 

The identical pattern exists in StarkNet's `omni_bridge.cairo`: `transfer_from` is called with `amount`, and then `amount` is emitted in the `InitTransfer` event without measuring the actual received balance: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

Every `initTransfer` call with a fee-on-transfer token causes the bridge to hold `transferFee` fewer tokens than the cross-chain message claims. The NEAR side records the full `amount` as locked/bridged. On the destination chain, `finTransfer` attempts to release `amount` tokens: [6](#0-5) 

After enough such transfers, the bridge vault is drained below the sum of outstanding obligations. Subsequent legitimate `finTransfer` calls revert because the vault cannot cover the full `amount`, permanently freezing user funds. This matches the allowed impact: **High â€” balance/accounting corruption that breaks bridge collateralization**.

---

### Likelihood Explanation

Fee-on-transfer ERC20 tokens are a well-known token class (e.g., tokens with built-in tax/reflection mechanics). The bridge has no allowlist or fee-on-transfer guard. Any unprivileged user can call `initTransfer` with such a token. The entry path requires no special role, no leaked key, and no colluding party â€” only a standard ERC20 `approve` followed by `initTransfer`. Likelihood is **Medium**: the bridge must support the specific token, but no privileged access is required.

---

### Recommendation

Measure the actual balance increase after `safeTransferFrom` and use that value in the event and cross-chain message:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint128 actualReceived = uint128(IERC20(tokenAddress).balanceOf(address(this)) - balanceBefore);
    amount = actualReceived; // shadow input with actual received
}
```

Then use `amount` (now the actual received value) in `initTransferExtension` and the `InitTransfer` event. Apply the same fix to StarkNet's `omni_bridge.cairo` `init_transfer`.

---

### Proof of Concept

1. Token `FOT` has a 1% transfer fee (fee-on-transfer).
2. User calls `OmniBridge.initTransfer(FOT, 1000e18, 0, 0, "near:recipient", "")`.
3. Bridge calls `FOT.safeTransferFrom(user, bridge, 1000e18)` â€” bridge receives `990e18` (1% fee deducted).
4. Bridge emits `InitTransfer(..., amount=1000e18, ...)`.
5. NEAR bridge reads the event, records `1000e18` as the locked amount, and issues a signed `finTransfer` payload for `1000e18` on the destination EVM chain.
6. `finTransfer` calls `IERC20(FOT).safeTransfer(recipient, 1000e18)` â€” but the bridge only holds `990e18`, so the call reverts.
7. The user's funds are permanently locked: the NEAR side has consumed the nonce and recorded the transfer as complete, but the destination `finTransfer` always reverts.
8. Repeated over many transfers, the bridge vault is progressively drained, causing all subsequent legitimate `finTransfer` calls for `FOT` to fail. [7](#0-6) [2](#0-1)
