### Title
Unbounded `compute_max_possible_fee()` Return Value Causes Unconditional `assert_nn_le` Failure in `charge_fee()`, Enabling Network Halt - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

`compute_max_possible_fee()` performs uncapped felt arithmetic over resource bounds that are individually validated but whose product can exceed `2^128 - 1`. The subsequent `assert_nn_le(calldata.amount.low, max_fee)` call in `charge_fee()` uses Cairo's range-check-backed comparison, which is structurally incapable of handling values above `2^128 - 1`. An unprivileged transaction sender can craft a v3 transaction with large-but-valid resource bounds to produce a `max_fee` that makes this assertion unconditionally fail, causing the OS to panic and the block to be invalid.

---

### Finding Description

**Vulnerability class:** Fee/accounting bug — arithmetic result exceeds the implicit bound of the comparison primitive, causing an unconditional OS panic.

**Root cause — `compute_max_possible_fee`:** [1](#0-0) 

The function returns:
```
l1_gas.max_amount * l1_gas.max_price_per_unit
+ l2_gas.max_amount * (l2_gas.max_price_per_unit + tip)
+ l1_data_gas.max_amount * l1_data_gas.max_price_per_unit
```

The only upstream validation of these fields occurs in `pack_resource_bounds()` during transaction hash computation: [2](#0-1) 

- `max_amount` is bounded to `[0, 2^64 - 1]` via `assert_nn_le`.
- `max_price_per_unit` is bounded to `[0, 2^128 - 1]` via `assert_nn` (range-check builtin).
- `tip` is bounded to `[0, 2^64 - 1]` via `assert_nn_le`. [3](#0-2) 

With these validated bounds, the maximum value of `max_fee` is:

```
3 × (2^64 - 1) × (2^128 - 1 + 2^64 - 1) ≈ 3 × 2^192
```

This is well below the Stark field prime (`P ≈ 2^251`), so no field overflow occurs — the result is a valid felt. However, `3 × 2^192 >> 2^128 - 1`.

**Failure site — `charge_fee`:** [4](#0-3) 

`assert_nn_le(a, b)` from `starkware.cairo.common.math` is implemented using the range-check builtin, which can only verify values in `[0, 2^128 - 1]`. Internally it calls `assert_nn(b - a)`, which range-checks `b - a`. If `b - a > 2^128 - 1`, the range check fails and the OS panics.

Concretely: if `max_fee = 2^192` and `low_actual_fee = 0`, then `b - a = 2^192 > 2^128 - 1` → unconditional panic.

The threshold where the assertion becomes unconditionally failing for **any** valid `low_actual_fee ∈ [0, 2^128 - 1]` is:

```
max_fee > 2 × (2^128 - 1) ≈ 2^129
```

This threshold is trivially reachable: e.g., `l1_gas.max_amount = 2`, `l1_gas.max_price_per_unit = 2^128 - 1` already yields `max_fee ≥ 2 × (2^128 - 1) = 2^129 - 2`, and adding any non-zero L2 or L1-data gas bounds pushes it over.

---

### Impact Explanation

**High — Network not being able to confirm new transactions (total network shutdown).**

When the OS panics inside `charge_fee`, the entire block proof fails. The sequencer has already committed to the block; it must discard it and re-sequence. If the attacker continuously submits such transactions and the sequencer lacks pre-validation of `compute_max_possible_fee`, every block that includes one of these transactions will be invalid, preventing the network from finalizing blocks.

---

### Likelihood Explanation

The attacker is an unprivileged transaction sender. The required inputs — `max_amount` and `max_price_per_unit` — are standard v3 transaction fields freely set by the sender. No privileged access, leaked key, or operator cooperation is required. The only dependency is whether the sequencer pre-validates that `compute_max_possible_fee` stays within `2^128`. The OS itself imposes no such guard, meaning any sequencer that relies on the OS as the final arbiter of transaction validity is vulnerable.

---

### Recommendation

Add an explicit upper-bound check on `max_fee` inside `compute_max_possible_fee` or at the start of `charge_fee`, before calling `assert_nn_le`:

```cairo
// Ensure max_fee fits within the range-check domain.
assert_nn_le(max_fee, 2 ** 128 - 1);
```

Alternatively, enforce tighter per-field bounds in `pack_resource_bounds` — e.g., `assert_nn_le(resource_bounds.max_price_per_unit, 2 ** 64 - 1)` — so that the product of any two validated fields cannot exceed `2^128 - 1`.

---

### Proof of Concept

1. Attacker constructs a v3 invoke transaction with:
   - `l1_gas.max_amount = 2` (valid: ≤ 2^64 - 1)
   - `l1_gas.max_price_per_unit = 2^128 - 1` (valid: passes `assert_nn`)
   - `l2_gas.max_amount = 1`, `l2_gas.max_price_per_unit = 1` (minimal, non-zero)
   - `l1_data_gas.max_amount = 0`
   - `tip = 0`

2. `pack_resource_bounds` validates all fields — all checks pass.

3. `compute_max_possible_fee` computes:
   ```
   max_fee = 2 × (2^128 - 1) + 1 × (1 + 0) + 0 = 2^129 - 1
   ```
   This is `> 2^128 - 1`.

4. Sequencer includes the transaction in a block (no OS-level pre-check prevents this).

5. OS calls `charge_fee` → `assert_nn_le(low_actual_fee, 2^129 - 1)` → internally calls `assert_nn(2^129 - 1 - low_actual_fee)`. Since `2^129 - 1 - low_actual_fee > 2^128 - 1` for any `low_actual_fee ≤ 2^128 - 1`, the range check fails.

6. OS panics → block proof fails → network cannot finalize the block.

func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
