Audit Report

## Title
Stranded ETH on Router Consumed by Subsequent Caller's WETH Swap — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`PeripheryPayments.pay()` reads `address(this).balance` — the router's total native ETH balance — when settling a WETH-input swap, rather than only the current transaction's `msg.value`. A user who overpays (sends more ETH than `amountIn`) without calling `refundETH()` leaves ETH stranded on the router. A subsequent attacker can call `exactInputSingle{value: 0}` with `tokenIn = WETH` and have their swap funded entirely by the victim's stranded ETH, causing direct principal loss.

## Finding Description

In `pay()`, when `token == WETH` and `payer != address(this)`, the function reads the router's entire native balance:

```solidity
uint256 nativeBalance = address(this).balance;   // total router ETH, not msg.value
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
``` [1](#0-0) 

`exactInputSingle` is `payable` and stores `msg.sender` as `payer` in transient context, but imposes no constraint that `msg.value == params.amountIn`: [2](#0-1) 

The `receive()` guard only blocks plain ETH transfers (no calldata) from non-WETH addresses; ETH sent alongside a function call is not blocked: [3](#0-2) 

The intended usage pattern shown in tests is `multicall{value}([exactInputSingle(...), refundETH()])`: [4](#0-3) 

This pattern is not enforced on-chain. A user calling `exactInputSingle{value: X}` directly with `X > amountIn` leaves `X - amountIn` ETH stranded on the router with no recovery guarantee.

**Exploit path:**
1. User A calls `exactInputSingle{value: 2 ether}` with `tokenIn = WETH`, `amountIn = 1 ether`. `pay()` deposits 1 ETH as WETH and sends it to the pool. 1 ETH remains on the router.
2. Attacker calls `exactInputSingle{value: 0}` with `tokenIn = WETH`, `amountIn = 1 ether`. `pay()` reads `address(this).balance = 1 ether >= 1 ether`, deposits User A's stranded ETH as WETH, and sends it to the pool. Attacker receives swap output without paying anything.

Secondary vector: `refundETH()` has no access control and sends the entire router balance to `msg.sender`: [5](#0-4) 

An attacker who front-runs User A's `refundETH()` call steals the stranded ETH directly.

## Impact Explanation

User A suffers a direct, silent loss of their overpaid ETH principal. The attacker receives a full swap output without providing any input token or ETH. The pool's accounting is internally consistent (it received the correct WETH amount), so the entire loss is borne by User A. This is a direct loss of user principal meeting the High severity threshold.

## Likelihood Explanation

The overpayment scenario is realistic: users commonly send round ETH amounts for swaps whose `amountIn` is a smaller precise value, and integrators or wallets may call `exactInputSingle` directly without wrapping in `multicall` + `refundETH()`. An attacker can monitor the router's ETH balance on-chain and immediately submit a zero-value WETH-input swap to consume any stranded ETH. The attack requires no special privileges, no malicious setup, and is repeatable against any overpaying user.

## Recommendation

Track the current transaction's ETH contribution explicitly rather than reading `address(this).balance`. Options:

- Pass `msg.value` into the transient callback context alongside `payer` at each payable entry point. In `pay()`, use `min(trackedMsgValue, value)` as the native portion and pull the remainder via `safeTransferFrom` from `payer`'s WETH allowance.
- Alternatively, enforce `msg.value == 0` when `tokenIn != WETH`, and enforce `msg.value == amountIn` when `tokenIn == WETH` in exact-input single-hop calls (exact-output requires a refund path since the exact input amount is unknown upfront).

## Proof of Concept

```solidity
function test_strandedEthConsumedByAttacker() public {
    uint128 amountIn = 1 ether;
    uint256 overpay = 2 ether;

    // User A overpays: sends 2 ETH but amountIn = 1 ETH, no refundETH()
    vm.deal(userA, overpay);
    vm.prank(userA);
    router.exactInputSingle{value: overpay}(ExactInputSingleParams({
        pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
        zeroForOne: true, amountIn: amountIn, amountOutMinimum: 0,
        recipient: userA, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
    }));
    assertEq(address(router).balance, 1 ether); // 1 ETH stranded

    // Attacker sends 0 ETH, swap funded by userA's stranded ETH
    vm.prank(attacker);
    router.exactInputSingle{value: 0}(ExactInputSingleParams({
        pool: address(pool), tokenIn: address(weth), tokenOut: address(token1),
        zeroForOne: true, amountIn: amountIn, amountOutMinimum: 0,
        recipient: attacker, deadline: block.timestamp + 1, priceLimitX64: 0, extensionData: ""
    }));

    assertEq(address(router).balance, 0);
    assertGt(token1.balanceOf(attacker), 0); // attacker received output
    // userA lost 1 ETH silently
}
```
