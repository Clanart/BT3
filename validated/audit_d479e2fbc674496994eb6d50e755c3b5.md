Audit Report

## Title
Stranded native ETH on the router is silently consumed by any subsequent WETH-input swap, enabling theft of prior users' funds - (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`PeripheryPayments.pay()` uses `address(this).balance` — the router's cumulative ETH balance across all transactions — when settling WETH-input swaps. Any ETH left on the router by a prior user who omitted a `refundETH` call is available to the next caller who issues a WETH-input swap, allowing that caller to receive pool output without spending any WETH or ETH of their own.

## Finding Description
In `PeripheryPayments.sol` L74-84, the WETH branch of `pay()` reads `nativeBalance = address(this).balance`, which is the router's total ETH balance, not the current transaction's `msg.value`:

```solidity
uint256 nativeBalance = address(this).balance;
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);   // payer never pulled
} else if (nativeBalance > 0) {
    IWETH9(WETH).deposit{value: nativeBalance}();
    IERC20(WETH).safeTransfer(recipient, nativeBalance);
    IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
} else {
    IERC20(WETH).safeTransferFrom(payer, recipient, value);
}
``` [1](#0-0) 

In the `nativeBalance >= value` branch, `safeTransferFrom` is never called, so the payer's WETH allowance is never checked or consumed. The `receive()` guard at L32-34 only blocks unsolicited ETH pushes from non-WETH addresses: [2](#0-1) 

It does not prevent ETH from accumulating via `msg.value` on any payable entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, `unwrapWETH9`, `sweepToken`, `refundETH`). A user who sends `msg.value = X` for a WETH swap consuming only `Y < X` leaves `X - Y` stranded if they omit `refundETH`.

The transient callback context at `MetricOmmSimpleRouter.sol` L71 correctly binds the payer: [3](#0-2) 

But this payer identity is irrelevant once `nativeBalance >= value` — `pay()` never calls `safeTransferFrom` in that branch. The full callback path is `exactInputSingle` → `pool.swap` → `metricOmmSwapCallback` → `_justPayCallback` → `pay(WETH, attacker, pool, value)`: [4](#0-3) 

## Impact Explanation
Direct theft of user principal. Any ETH stranded on the router is available to the next caller who issues a WETH-input swap with `amountIn ≤ stranded_amount`. That caller receives pool output tokens of full market value while spending zero WETH and zero ETH of their own, requiring no WETH approval. The original depositor permanently loses the stranded ETH. This is a direct loss of user funds meeting Critical/High Sherlock thresholds.

## Likelihood Explanation
Medium. ETH stranding is a realistic outcome of normal usage: a user calls `exactInputSingle` with `tokenIn = WETH` and sends `msg.value`, but the pool partially fills or the user over-estimates `amountIn`, and the user omits `refundETH` from their multicall — a common mistake when composing calls off-chain or when a UI does not append the refund step. An attacker can monitor the router's ETH balance with a single `eth_getBalance` RPC call and immediately exploit any non-zero balance. The attack requires no special permissions, no flash loan, and no privileged role.

## Recommendation
Track only the ETH attributable to the current transaction for WETH settlement. Record `msg.value` at each swap entry point and pass it explicitly to `pay()`, using only that amount for native-ETH wrapping. Alternatively, after each swap, assert `address(this).balance` equals the pre-call balance so that any unspent ETH is detected and refunded atomically rather than left for a future caller to consume.

## Proof of Concept
```
Block N:
  User A calls exactInputSingle{value: 1 ETH}(
      tokenIn = WETH, amountIn = 0.5 ETH, ...
  )
  → pay() sees nativeBalance = 1 ETH ≥ 0.5 ETH
  → deposits 0.5 ETH, transfers 0.5 WETH to pool
  → 0.5 ETH remains on router
  User A does NOT call refundETH.

Block N+1:
  Attacker calls exactInputSingle{value: 0}(
      tokenIn = WETH, amountIn = 0.5 ETH, ...
  )
  → pay() sees nativeBalance = 0.5 ETH ≥ 0.5 ETH
  → deposits 0.5 ETH (User A's), transfers 0.5 WETH to pool
  → safeTransferFrom is NEVER called; attacker needs zero WETH approval
  → Attacker receives pool output worth ~0.5 ETH for free
  → User A's 0.5 ETH is permanently lost
```
