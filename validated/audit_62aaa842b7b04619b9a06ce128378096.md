### Title
Integer Wrapping in `check_time_locks` Bypasses Relative Timelock Conditions When `nowrap=false` - (File: `crates/chia-consensus/src/check_time_locks.rs`)

### Summary

When `check_time_locks` is called with `nowrap=false`, relative timelock conditions (`ASSERT_HEIGHT_RELATIVE`, `ASSERT_SECONDS_RELATIVE`) use `wrapping_add` for the threshold computation. A crafted spend with a near-maximum condition value causes the sum to wrap around to a small integer, making the timelock check trivially pass. A coin that should be unspendable for billions of blocks (or seconds) can be spent immediately.

### Finding Description

`check_time_locks` in `crates/chia-consensus/src/check_time_locks.rs` enforces relative timelocks by comparing the current block height (or timestamp) against the coin's confirmed height (or timestamp) plus the condition value. When `nowrap=false`, the addition uses `wrapping_add`:

```rust
// ASSERT_HEIGHT_RELATIVE path, nowrap=false branch:
} else if prev_transaction_block_height
    < unspent.confirmed_block_index.wrapping_add(height_relative)
{
    return Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed));
}
``` [1](#0-0) 

If `confirmed_block_index = 100` and `height_relative = 0xffff_ffff` (u32::MAX, a valid 4-byte CLVM integer accepted by `sanitize_uint`), then:

```
100u32.wrapping_add(0xffff_ffff) = 99   // wraps around
```

The check becomes `prev_height < 99`, which is `false` at any height ≥ 99 — so the timelock **passes immediately** even though the puzzle demanded ~4.3 billion blocks of waiting. The same wrapping applies to `ASSERT_SECONDS_RELATIVE` with `u64::MAX`:

```rust
} else if timestamp < unspent.timestamp.wrapping_add(seconds_relative) {
    return Err(ValidationErr::Err(ErrorCode::AssertSecondsRelativeFailed));
}
``` [2](#0-1) 

The tests explicitly document and confirm this behavior:

```
// 200 < 100 + u32::MAX -> Err with nowrap (saturates), Ok without (wraps)
#[case::height_relative_wrap(
    Osc { height_relative: Some(0xffff_ffff), ..Default::default() },
    Err(ValidationErr::Err(ErrorCode::AssertHeightRelativeFailed)),
    Ok(()),   // <-- passes when nowrap=false
)]
``` [3](#0-2) 

The condition value `0xffff_ffff` is accepted without error by `parse_args` / `sanitize_uint` for `ASSERT_HEIGHT_RELATIVE` (4-byte limit): [4](#0-3) 

The Python-exposed binding `py_check_time_locks` passes `nowrap` directly from the caller, meaning any Python node that invokes it with `nowrap=False` is vulnerable: [5](#0-4) 

### Impact Explanation

A coin whose puzzle outputs `(ASSERT_HEIGHT_RELATIVE . 0xffffffff)` is intended to be unspendable for ~4.3 billion blocks. Under `nowrap=false`, the wrapping arithmetic makes the threshold `confirmed_height - 1`, so the coin is spendable at any height ≥ `confirmed_height`. Any coin protected solely by a relative timelock with a near-maximum value is immediately spendable, enabling unauthorized coin theft or unauthorized spend acceptance. This matches the allowed High impact: **timelock condition validation bypass enables unauthorized spend acceptance**.

### Likelihood Explanation

The `nowrap=false` path is an explicitly supported, tested code path exposed via the Python binding. The chia-blockchain Python node controls the `nowrap` argument; during any period where it passes `nowrap=False` (e.g., for backward-compatible validation of pre-fork blocks, or if the upgrade has not yet been enforced), an attacker can craft a spend bundle with `ASSERT_HEIGHT_RELATIVE 0xffffffff` and have it accepted immediately. The attacker needs no special privileges — only the ability to submit a spend bundle with a crafted puzzle.

### Recommendation

Replace `wrapping_add` with `saturating_add` unconditionally for all relative timelock checks, or remove the `nowrap=false` code path entirely once the network has fully transitioned. The `nowrap=true` path already uses `saturating_add` correctly: [6](#0-5) 

### Proof of Concept

1. Craft a CLVM puzzle that outputs condition `(82 . 0xffffffff)` (`ASSERT_HEIGHT_RELATIVE` opcode 82, value `u32::MAX`).
2. Fund a coin with this puzzle.
3. Immediately submit a spend of that coin.
4. Call `check_time_locks(..., nowrap=False)` — the check at line 64–68 computes `confirmed_height.wrapping_add(0xffffffff) = confirmed_height - 1`, so `prev_height < confirmed_height - 1` is false, and the function returns `Ok(())`.
5. The spend is accepted despite the coin being locked for ~4.3 billion blocks.

The same attack applies to `ASSERT_SECONDS_RELATIVE` with value `0xffffffffffffffff` (`u64::MAX`), bypassing a ~584-billion-year timelock. [7](#0-6)

        timestamp,
