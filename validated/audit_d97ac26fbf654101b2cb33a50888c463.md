After carefully reading all relevant code, here is my analysis:

## Core Finding: Missing `chunk.shard_id() == shard_id` Guard in `set_state_header`

The entire `set_state_header` function in `chain/chain/src/state_sync/adapter.rs` performs five validation steps, but **none of them check that `chunk.shard_id()` equals the `shard_id` parameter**. [1](#0-0) 

**Step 3a** (`verify_path`) only checks that the chunk's `ChunkHashHeight` is *somewhere* in the block's chunk merkle tree — it does not verify the chunk occupies the leaf position corresponding to `shard_id`. A chunk from shard 1 has a valid merkle proof for shard 1's leaf position, and that proof verifies correctly against `chunk_headers_root`. [2](#0-1) 

**Step 4e** computes `receipts_hash = CryptoHash::hash_borsh(ReceiptList(shard_id, receipts))` using the *parameter* `shard_id`, not `chunk.shard_id()`. The attacker can supply the real on-chain incoming receipts for shard 0 (public data), which pass this check independently of which chunk body is embedded. [3](#0-2) 

**Step 5** (`validate_state_root_node`) only checks internal consistency of the `state_root_node` against `chunk_inner.prev_state_root()` — both from shard 1's chunk, so this passes trivially. [4](#0-3) 

After all checks pass, the corrupted value is written: [5](#0-4) 

`StateHeaderKey(shard_id=0, sync_hash)` is stored in `DBCol::StateHeaders` pointing to shard 1's chunk.

---

## Downstream Corruption

`set_state_part` reads the stored header, extracts `prev_state_root` from shard 1's chunk, and validates state parts against shard 1's state root. The attacker supplies shard 1's state parts. [6](#0-5) 

`set_state_finalize` → `chain_update.set_state_finalize(shard_id=0, ...)` calls `apply_chunk` with `shard_uid` for shard 0 but `prev_state_root` from shard 1's chunk. Since the state parts were applied to shard 0's trie storage under shard 1's state root hash, the apply succeeds and saves a `ChunkExtra` for shard 0 containing shard 1's state root. [7](#0-6) 

---

## Attacker Reachability

The syncing node downloads the header from any responding peer via `StateSyncDownloader::ensure_shard_header`, which sends it directly to `set_state_header` with no pre-check on `chunk.shard_id()`. [8](#0-7) 

An unprivileged peer can craft a `ShardStateSyncResponseHeader` containing:
1. Shard 1's chunk (internally valid, signed by shard 1's chunk producer)
2. Shard 1's merkle proof (valid for shard 1's leaf position in the block)
3. Shard 1's prev chunk header + proof (same)
4. Real incoming receipts for shard 0 (public on-chain data)
5. Shard 1's valid `state_root_node`

All five checks in `set_state_header` pass.

---

## Blocking Factor: Receipt Proof Check

The receipt proof check (step 4e) uses `shard_id` (the parameter, e.g. 0) to compute `receipts_hash`. This means the attacker must supply receipts that are genuinely destined for shard 0, with valid on-chain proofs. This is **not a blocker** — these are public chain data the attacker can read directly. The `to_shard_id` field in `ShardProof` is explicitly ignored (`to_shard_id: _`). [9](#0-8) 

---

## Conclusion

### Title
Missing `chunk.shard_id() == shard_id` check in `set_state_header` allows cross-shard state header substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary
`set_state_header` verifies that the embedded chunk is *included in the block* (via merkle proof) but never verifies that it is the chunk *for the requested shard*. An unprivileged peer can substitute shard N's chunk for shard M's header, causing the syncing node to install shard N's state root into shard M's `ChunkExtra`.

### Finding Description
The `verify_path` call at line 394 checks inclusion in the block's chunk merkle tree without constraining which leaf position the proof covers. No subsequent check compares `chunk.shard_id()` to the `shard_id` parameter. The corrupted `StateHeaderKey(shard_id, sync_hash)` is then used by `set_state_part` and `set_state_finalize` to apply the wrong shard's state.

### Impact Explanation
The syncing node's canonical `ChunkExtra` for `shard_id` would contain the `prev_state_root` of a different shard. All subsequent chunk application, trie reads, and state witness validation for that shard would operate on the wrong state, causing divergence from the canonical chain.

### Likelihood Explanation
Requires the attacker to be a responding peer during state sync (any unprivileged network participant). The crafted header uses only real, publicly available on-chain data — no cryptographic forgery is needed.

### Recommendation
Add an explicit check immediately after extracting the chunk in `set_state_header`:

```rust
if chunk.shard_id() != shard_id {
    return Err(Error::Other(
        "set_shard_state failed: chunk shard_id does not match requested shard_id".into(),
    ));
}
``` [10](#0-9) 

### Proof of Concept
In a two-shard test environment: obtain shard 1's `ShardStateSyncResponseHeader` from `get_state_response_header(shard_id=1, sync_hash)`, then call `set_state_header(shard_id=0, sync_hash, shard_1_header)`. Assert that `get_state_header(shard_id=0, sync_hash)` returns a header whose `chunk_prev_state_root()` equals shard 1's `prev_state_root`, not shard 0's.
