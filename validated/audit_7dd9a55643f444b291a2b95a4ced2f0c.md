### Title
Fee-on-Transfer Token Accounting Mismatch in `initTransfer` Allows Unbacked Supply Minting on NEAR — (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` uses the caller-supplied `amount` parameter to record the locked value in the `InitTransfer` event and Wormhole message, without measuring the actual tokens received. For fee-on-transfer ERC20 tokens, the bridge receives fewer tokens than `amount`, but the cross-chain message claims the full `amount` was locked. The NEAR side processes this message at face value and releases/mints `amount` tokens, creating unbacked supply.

---

### Finding Description

In `OmniBridge.initTransfer`, the branch handling native (non-bridge, non-custom-minter) ERC20 tokens is:

```solidity
} else {
    IERC20(tokenAddress).safeTransferFrom(
        msg.sender,
        address(this),
        amount          // ← user-supplied, not verified against actual receipt
    );
}
```

After this call, the function unconditionally uses the original `amount` in both the on-chain event and the cross-chain message:

```solidity
emit BridgeTypes.InitTransfer(
    msg.sender,
    tokenAddress,
    currentOriginNonce,
    amount,   // ← not the actual received amount
    fee,
    nativeFee,
    recipient,
    message
);
``` [1](#0-0) [2](#0-1) 

In `OmniBridgeWormhole.initTransferExtension`, the same `amount` is encoded into the Wormhole payload published to NEAR:

```solidity
Borsh.encodeUint128(amount),
``` [3](#0-2) 

The NEAR bridge parses this event/message and uses the `amount` field directly to credit the recipient. The `InitTransfer` event ABI confirms `amount` is the sole field used for the credited value: [4](#0-3) 

No balance-before/balance-after check is performed anywhere in the EVM contract to verify that `amount` tokens were actually received.

---

### Impact Explanation

**High — Balance/accounting corruption that breaks bridge collateralization.**

For a fee-on-transfer ERC20 token `T` with a 1% transfer fee:
- Attacker calls `initTransfer(T, 1000, ...)`.
- Bridge receives 990 tokens but records `amount = 1000` in the cross-chain message.
- NEAR side finalizes the transfer and mints/releases 1000 tokens to the recipient.
- Net result: 10 tokens of unbacked supply per transfer, permanently breaking the 1:1 collateral invariant.

Repeated over many transfers, the EVM bridge holds less collateral than the NEAR-side supply, making it impossible for all NEAR-side holders to redeem. This constitutes direct accounting corruption of bridge collateralization.

---

### Likelihood Explanation

**Medium-High.** The `else` branch in `initTransfer` accepts any ERC20 token that is not flagged as a bridge token and has no custom minter — there is no whitelist. Any user can call `logMetadata` permissionlessly to register a fee-on-transfer token on NEAR, then call `initTransfer` with it. Fee-on-transfer tokens (e.g., USDT on some chains, reflection tokens, tax tokens) are common in the wild. The attacker needs no privileged role. [5](#0-4) [6](#0-5) 

---

### Recommendation

Measure the actual received amount using a balance-before/balance-after pattern, and use the measured delta — not the caller-supplied `amount` — in the event and cross-chain message:

```solidity
} else {
    uint256 balanceBefore = IERC20(tokenAddress).balanceOf(address(this));
    IERC20(tokenAddress).safeTransferFrom(msg.sender, address(this), amount);
    uint256 balanceAfter = IERC20(tokenAddress).balanceOf(address(this));
    uint128 actualReceived = uint128(balanceAfter - balanceBefore);
    amount = actualReceived; // use actual received amount downstream
}
```

Then pass `amount` (now the actual received value) to `initTransferExtension` and `emit BridgeTypes.InitTransfer`. This mirrors the recommendation in the reference report and ensures the cross-chain message always reflects the true locked collateral.

---

### Proof of Concept

1. Deploy a fee-on-transfer ERC20 token `FeeToken` (e.g., 1% fee on every transfer) on the EVM chain.
2. Call `OmniBridge.logMetadata(FeeToken)` — permissionless, registers the token on NEAR.
3. Approve the bridge for `1000` `FeeToken`.
4. Call `OmniBridge.initTransfer(FeeToken, 1000, 0, 0, "alice.near", "")`.
   - `safeTransferFrom` transfers 1000 tokens; bridge receives 990 (1% fee deducted by token).
   - `InitTransfer` event is emitted with `amount = 1000`.
5. Relayer submits the proof to NEAR `fin_transfer`.
   - NEAR reads `amount = 1000` from the event and mints/releases 1000 tokens to `alice.near`.
6. Bridge holds 990 `FeeToken` on EVM but 1000 tokens exist on NEAR — 10 tokens of unbacked supply created per transfer.
7. Repeat to drain the bridge's collateral backing. [7](#0-6) [8](#0-7)
