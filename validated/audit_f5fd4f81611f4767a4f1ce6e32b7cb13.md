### Title
Unsigned `MaxPriorityPrice` in `ExtensionOptionDynamicFeeTx` Allows Fee Inflation Without Signature Coverage - (File: `ante/evm/fee_checker.go`)

### Summary
When a Cosmos SDK transaction carries an `ExtensionOptionDynamicFeeTx` extension and is signed with `SIGN_MODE_LEGACY_AMINO_JSON`, the `MaxPriorityPrice` field is not covered by the user's signature. A mempool observer can replace this value with an arbitrarily large number, causing the `NewDynamicFeeChecker` to compute a higher `effectiveFee` and deduct up to the full signed `fee` cap from the user's account — far more than the user intended to pay.

---

### Finding Description

**Root cause — unsigned extension option field consumed by fee computation:**

`NewDynamicFeeChecker` in `ante/evm/fee_checker.go` reads `MaxPriorityPrice` directly from the `ExtensionOptionDynamicFeeTx` extension option attached to the transaction:

```go
// ante/evm/fee_checker.go lines 66-73
if hasExtOptsTx, ok := tx.(authante.HasExtensionOptionsTx); ok {
    for _, opt := range hasExtOptsTx.GetExtensionOptions() {
        if extOpt, ok := opt.GetCachedValue().(*ethermint.ExtensionOptionDynamicFeeTx); ok {
            maxPriorityPrice = extOpt.MaxPriorityPrice
            break
        }
    }
}
``` [1](#0-0) 

This value directly controls the effective fee deducted from the user:

```go
// ante/evm/fee_checker.go lines 91-98
effectivePrice := sdkmath.NewIntFromBigInt(types.EffectiveGasPrice(baseFeeInt.BigInt(), feeCap.BigInt(), maxPriorityPrice.BigInt()))
effectiveFee := sdk.Coins{{
    Denom:  denom,
    Amount: effectivePrice.Mul(sdkmath.NewIntFromUint64(gas)),
}}
``` [2](#0-1) 

The `effectiveFee` returned here is what `DeductFeeDecorator` actually deducts from the user's bank balance:

```go
// ante/evm/nativefee.go lines 57-65
fee, priority, err = dfd.txFeeChecker(ctx, tx)
...
if err := dfd.checkDeductFee(ctx, tx, fee); err != nil {
    return ctx, err
}
``` [3](#0-2) 

**Why `MaxPriorityPrice` is not signed:**

When the ante handler routes a transaction with `ExtensionOptionDynamicFeeTx` it calls `newCosmosAnteHandler`:

```go
// evmd/ante/ante.go lines 83-85
case "/ethermint.types.v1.ExtensionOptionDynamicFeeTx":
    // cosmos-sdk tx with dynamic fee extension
    anteHandler = newCosmosAnteHandler(ctx, options, options.ExtraDecorators...)
``` [4](#0-3) 

`newCosmosAnteHandler` uses the standard Cosmos SDK `ante.NewSigVerificationDecorator`:

```go
// evmd/ante/handler_options.go lines 205-206
ante.NewSigGasConsumeDecorator(options.AccountKeeper, options.SigGasConsumer),
ante.NewSigVerificationDecorator(options.AccountKeeper, options.SignModeHandler),
``` [5](#0-4) 

When the user signs with `SIGN_MODE_LEGACY_AMINO_JSON`, the sign bytes are produced by `legacytx.StdSignBytes` (as seen in the legacy EIP-712 path):

```go
// ante/cosmos/eip712.go lines 197-207
txBytes := legacytx.StdSignBytes(
    signerData.ChainID,
    signerData.AccountNumber,
    signerData.Sequence,
    tx.GetTimeoutHeight(),
    legacytx.StdFee{
        Amount: tx.GetFee(),
        Gas:    tx.GetGas(),
    },
    msgs, tx.GetMemo(),
)
``` [6](#0-5) 

`StdSignBytes` covers `fee.amount`, `fee.gas`, `msgs`, `memo`, `chain_id`, `account_number`, and `sequence`. **Extension options (`TxBody.extension_options`) are not included.** Therefore `MaxPriorityPrice` is entirely outside the signed payload.

The `ExtensionOptionDynamicFeeTx` proto definition confirms this is a standalone field with no binding to the signed fee:

```proto
// proto/ethermint/types/v1/dynamic_fee.proto
message ExtensionOptionDynamicFeeTx {
  string max_priority_price = 1 [...];
}
``` [7](#0-6)

### Impact Explanation

The `effectivePrice` formula is `min(baseFee + maxPriorityPrice, feeCap)` where `feeCap = signedFee / gas`. By setting `maxPriorityPrice` to any value ≥ `feeCap − baseFee`, the attacker forces `effectivePrice = feeCap`, so the user is charged the full signed `fee` amount rather than the intended `baseFee × gas`.

**Concrete example:**
- User signs: `fee = 1,000,000 aphoton`, `gas = 100,000`, intending to pay `baseFee × gas = 100,000 aphoton` (baseFee = 1 aphoton/gas)
- Attacker sets `MaxPriorityPrice = MaxInt64`
- `effectivePrice = min(1 + MaxInt64, 10) = 10 aphoton/gas`
- User is charged `10 × 100,000 = 1,000,000 aphoton` — a 10× overpayment
- The excess `900,000 aphoton` flows to validators as priority tips

This is a direct mis-accounting of valid user funds through the fee market ante handler path, matching the allowed High impact: *"fee market, ante handler, mempool, or proposal handling bug that permits… valid user funds/fees to be mis-accounted."*
