### Title
Missing Destination-Chain Factory and Token-Mapping Validation Before Token Burn/Lock - (`near/omni-bridge/src/lib.rs`)

---

### Summary

`init_transfer` in the NEAR bridge contract burns or locks user tokens before verifying that the destination chain has a configured factory address or a registered token-address mapping. If a user initiates a transfer to a destination chain for which neither is configured, the tokens are irrecoverably destroyed on NEAR while the transfer can never be finalized on the destination chain.

---

### Finding Description

The entry point for outgoing NEAR-side transfers is `ft_on_transfer` → `init_transfer` → `init_transfer_internal`.

`init_transfer` performs only one destination-chain check:

```rust
// near/omni-bridge/src/lib.rs  line 531-534
require!(
    init_transfer_msg.recipient.get_chain() != ChainKind::Near,
    BridgeError::InvalidRecipientChain.as_ref()
);
```

It does **not** verify:
- `self.factories.get(&destination_chain)` — that a bridge/factory contract is deployed and registered for the destination chain.
- `self.get_token_address(destination_chain, token_id)` — that the token being bridged has a registered address on the destination chain.

Control then flows to `init_transfer_internal` (line 1829), which unconditionally burns or locks the user's tokens:

```rust
// near/omni-bridge/src/lib.rs  lines 1850-1857
if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
    self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);
    self.lock_tokens_if_needed(
        transfer_message.get_destination_chain(),
        &token_id,
        transfer_message.amount.0,
    );
}
```

The transfer message is stored in `pending_transfers`. Later, when a relayer calls `sign_transfer`, it attempts to resolve the token address for the destination chain:

```rust
// near/omni-bridge/src/lib.rs  lines 462-469
let token_address = self
    .get_token_address(
        transfer_message.get_destination_chain(),
        self.get_token_id(&transfer_message.token),
    )
    .unwrap_or_else(|| {
        env::panic_str(BridgeError::FailedToGetTokenAddress.to_string().as_str())
    });
```

If no token mapping exists for the destination chain, `sign_transfer` **always panics**. The transfer is permanently stuck in `pending_transfers` with no user-callable cancellation or refund path. Even if a token mapping exists, if no factory is registered for the destination chain (`self.factories` is only checked in `fin_transfer_callback` for *incoming* proofs, never for *outgoing* transfers), the MPC signature is generated but there is no contract on the destination chain to accept it — funds are unclaimable.

The `factories` map is checked only on the inbound path:

```rust
// near/omni-bridge/src/lib.rs  lines 708-713
require!(
    self.factories
        .get(&init_transfer.emitter_address.get_chain())
        == Some(init_transfer.emitter_address),
    BridgeError::UnknownFactory.as_ref()
);
```

There is no symmetric guard on the outbound path.

---

### Impact Explanation

**Critical — Permanent freezing / irrecoverable lock of user funds.**

Tokens are burned (for bridge-deployed tokens) or locked (for native tokens) inside `init_transfer_internal` before any destination-chain readiness is confirmed. Once burned, they cannot be recovered. Once locked, there is no public `cancel_transfer` function that returns them to the user. The transfer sits in `pending_transfers` indefinitely, and `sign_transfer` will revert every time it is attempted for that transfer ID.

---

### Likelihood Explanation

**Low-Medium.** The frontend is unlikely to expose unsupported chains. However, any unprivileged user can call `ft_transfer_call` directly on the token contract with a crafted `InitTransferMsg` whose `recipient` encodes a `ChainKind` that has no factory or token mapping registered. The `ChainKind` enum is a finite set of valid values (Eth, Sol, Arb, Base, Near, Bitcoin, Zcash, etc.), and not all of them are necessarily configured at any given time. No privileged access is required.

---

### Recommendation

Add destination-chain readiness checks at the top of `init_transfer`, before any state mutation:

```rust
fn init_transfer(...) {
    require!(
        init_transfer_msg.recipient.get_chain() != ChainKind::Near,
        BridgeError::InvalidRecipientChain.as_ref()
    );

    // --- ADD THESE GUARDS ---
    let dst_chain = init_transfer_msg.get_destination_chain();
    require!(
        self.factories.get(&dst_chain).is_some(),
        BridgeError::UnknownFactory.as_ref()
    );
    let token_id_near = token_id.clone(); // already known at call site
    require!(
        self.get_token_address(dst_chain, token_id_near).is_some(),
        BridgeError::FailedToGetTokenAddress.as_ref()
    );
    // --- END GUARDS ---

    self.current_origin_nonce += 1;
    ...
}
```

This mirrors the fix applied in the referenced external report: revert early on unsupported destinations rather than accepting tokens optimistically.

---

### Proof of Concept

1. Token `wrapped-sol.bridge.near` is registered on NEAR with a mapping only for `ChainKind::Sol`.
2. User calls `ft_transfer_call` on `wrapped-sol.bridge.near` with:
   ```json
   {
     "receiver_id": "omni-bridge.near",
     "amount": "1000000",
     "msg": "{\"InitTransfer\":{\"recipient\":\"Arb:0xdeadbeef...\",\"fee\":\"0\",\"native_token_fee\":\"0\"}}"
   }
   ```
   `ChainKind::Arb` has no factory and no token mapping registered.
3. `init_transfer` passes the only guard (`get_chain() != Near`).
4. `init_transfer_internal` burns 1 000 000 units of `wrapped-sol.bridge.near` from the user.
5. Transfer is stored in `pending_transfers` with `transfer_id = {Near, nonce}`.
6. Any relayer calling `sign_transfer` for this ID hits `env::panic_str(BridgeError::FailedToGetTokenAddress)` — permanently.
7. No public function exists to cancel the transfer or return the burned tokens.
8. User's 1 000 000 tokens are permanently lost.
