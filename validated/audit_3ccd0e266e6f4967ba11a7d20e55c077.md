### Title
Token-2022 MetadataPointer Cross-Mint Metadata Spoofing in `parse_metadata_account` — (`solana/programs/bridge_token_factory/src/instructions/user/log_metadata.rs`)

---

### Summary

`parse_metadata_account` verifies only that the supplied metadata account is owned by `MetaplexID`, but never checks that the Metaplex metadata account's embedded `mint` field matches the mint being logged. An unprivileged attacker can create a Token-2022 mint whose `MetadataPointer` extension points to a legitimate Metaplex metadata account belonging to a *different* mint (e.g., USDC), causing `log_metadata` to emit a Wormhole VAA that carries the attacker's mint pubkey paired with the victim token's name and symbol.

---

### Finding Description

The Token-2022 branch of `process` reads the `MetadataPointer` extension from the mint and, when the pointer address is non-zero and not self-referential, delegates to `parse_metadata_account`: [1](#0-0) 

`parse_metadata_account` performs two checks before deserializing:

1. The caller-supplied `metadata` account key must equal the address stored in the `MetadataPointer` extension.
2. The account must be owned by `MetaplexID`. [2](#0-1) 

Neither check verifies that `metadata.mint == self.mint.key()`. Because the attacker controls the `MetadataPointer` value at mint-creation time, they can freely aim it at any `MetaplexID`-owned account. The deserialized `name` and `symbol` are then embedded verbatim into the outgoing `LogMetadataPayload`: [3](#0-2) 

Note that the classic SPL path is **not** vulnerable because the metadata address is derived via `find_program_address` using the mint key, cryptographically binding the two: [4](#0-3) 

---

### Impact Explanation

The attacker:

1. Creates Mint B (any SPL token) and registers a legitimate Metaplex metadata account for it with `name = "USD Coin"`, `symbol = "USDC"`.
2. Creates Mint A (Token-2022) with a `MetadataPointer` extension pointing to Mint B's Metaplex metadata account address.
3. Calls `log_metadata` for Mint A, passing Mint B's metadata account as the `metadata` argument.
4. All guards pass: the key equality check passes (the account matches the pointer), the owner check passes (MetaplexID owns it), and `try_deserialize` succeeds.
5. A Wormhole VAA is emitted with `token = Mint A`, `name = "USD Coin"`, `symbol = "USDC"`, `decimals = Mint A's decimals`.

On NEAR, this VAA registers Mint A as "USD Coin / USDC." Any subsequent `init_transfer` of Mint A tokens to NEAR causes NEAR to mint bridged tokens that display as USDC but are backed by a worthless mint. This constitutes unauthorized minting of bridged tokens under a spoofed identity.

---

### Likelihood Explanation

The attack requires no privileged access. Creating a Token-2022 mint with an arbitrary `MetadataPointer` and creating a Metaplex metadata account for a separate mint are both permissionless on-chain operations. The entire exploit is executable by any wallet with enough SOL for rent.

---

### Recommendation

Add a mint-binding check inside `parse_metadata_account` immediately after deserialization:

```rust
let metadata = MplMetadata::try_deserialize(&mut data.as_ref())?;
require_keys_eq!(
    metadata.mint,
    address,   // or self.mint.key() — both must agree
    ErrorCode::InvalidTokenMetadataAddress,
);
Ok((metadata.name.clone(), metadata.symbol.clone()))
```

This mirrors the implicit binding that the classic SPL path already enforces through PDA derivation.

---

### Proof of Concept

```
1. Create Mint B (classic SPL).
2. Call `mpl_token_metadata::create_metadata_accounts_v3` for Mint B
   with name="USD Coin", symbol="USDC".
   → metadata_account_B = PDA([b"metadata", MetaplexID, Mint_B])
3. Create Mint A (Token-2022) with MetadataPointer extension
   set to metadata_account_B.
4. Call bridge `log_metadata` for Mint A,
   passing metadata_account_B as the `metadata` account.
5. Assert the emitted Wormhole message payload deserializes to:
   { token: Mint_A, name: "USD Coin", symbol: "USDC", decimals: <Mint_A decimals> }
```

All steps are permissionless. The assertion at step 5 will pass against the current code, confirming the spoofed VAA is accepted and relayed to NEAR.
