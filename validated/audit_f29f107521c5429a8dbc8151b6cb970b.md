### Title
Missing Deadline Check in `addLiquidityExactShares` and `addLiquidityWeighted` Allows Stale Liquidity Deposits at Unfavorable Oracle Prices - (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

### Summary
`MetricOmmPoolLiquidityAdder` exposes four external liquidity-addition entry points — two overloads of `addLiquidityExactShares` and two overloads of `addLiquidityWeighted` — none of which accept or enforce a `deadline` parameter. The sibling `MetricOmmSimpleRouter` correctly calls `_checkDeadline` at the top of every swap entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`). The liquidity adder has no equivalent guard, so a pending transaction can sit in the mempool and execute arbitrarily late against a stale oracle price.

### Finding Description
`MetricOmmSimpleRouter` enforces a deadline on every swap:

```solidity
// MetricOmmSimpleRouter.sol line 68
_checkDeadline(params.deadline);
```

`MetricOmmPoolLiquidityAdder` has no such check on any of its entry points:

```solidity
// MetricOmmPoolLiquidityAdder.sol lines 56-68
function addLiquidityExactShares(
    address pool, address owner, uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 maxAmountToken0, uint256 maxAmountToken1,
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _validateOwner(owner);
    _validateDeltas(deltas);
    return _addLiquidity(...);   // no deadline check
}
```

The same omission applies to the `msg.sender`-owner overload (lines 71–81) and both `addLiquidityWeighted` overloads (lines 88–149).

Because Metric OMM is an oracle-anchored market maker, the price at which liquidity is deposited is determined by the external oracle at execution time, not at submission time. A transaction delayed in the mempool will execute against whatever oracle price is current at that later block. The user's `maxAmountToken0`/`maxAmountToken1` caps prevent overpaying in absolute token units, but they do not protect against the composition of the deposit shifting dramatically: the user may end up depositing almost entirely the depreciating token into bins that are now far from the live market price, suffering immediate impermanent loss or one-sided exposure that would not have occurred had the transaction executed promptly.

For `addLiquidityWeighted`, the `minimalCurBin`/`maximalCurBin` cursor bounds provide partial protection, but within a non-trivial cursor range the oracle price can move enough to materially change the token composition of the deposit.

### Impact Explanation
A user who submits an `addLiquidityWeighted` or `addLiquidityExactShares` transaction during a period of network congestion may have that transaction executed minutes or hours later. By that time the oracle price may have moved significantly. The user's liquidity lands in bins that are now far off-market, resulting in:
- Immediate one-sided exposure (all token0 or all token1) in bins that will not be traded through at the current price.
- Immediate impermanent loss relative to the price at submission time.
- The user cannot reclaim the "correct" composition without removing and re-adding liquidity, paying gas and potentially fees twice.

This is a direct loss of LP asset value, not a gas-only or theoretical issue.

### Likelihood Explanation
Any period of mempool congestion (gas price spike, network load) can delay a liquidity transaction. MEV searchers can also deliberately delay or sandwich a pending liquidity transaction to force execution at a worse oracle price. The trigger requires no privileged access and no non-standard token behavior.

### Recommendation
Add a `deadline` parameter to all four external entry points in `MetricOmmPoolLiquidityAdder` and enforce it at the top of each function, mirroring the pattern already used in `MetricOmmSimpleRouter`:

```solidity
function addLiquidityExactShares(
    address pool,
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    uint256 maxAmountToken0,
    uint256 maxAmountToken1,
    uint256 deadline,          // ← add
    bytes calldata extensionData
) external payable override returns (uint256 amount0Added, uint256 amount1Added) {
    _checkDeadline(deadline);  // ← add
    ...
}
```

Apply the same change to the `msg.sender`-owner overload and both `addLiquidityWeighted` overloads.

### Proof of Concept
1. Alice calls `addLiquidityWeighted` with `minimalCurBin = -2`, `maximalCurBin = 2`, `maxAmountToken0 = 1000e18`, `maxAmountToken1 = 1000e18` when the oracle price implies the cursor is at bin 0.
2. The transaction is stuck in the mempool for 30 minutes due to a gas price spike.
3. The oracle price drops sharply; the pool cursor moves to bin -2 (the minimum Alice allowed).
4. Alice's transaction executes. The cursor is still within her bounds, so `_validateBinAndBinPosition` passes.
5. The probe reveals that at the new price, `need0 ≈ 0` and `need1 ≈ 1000e18` (almost entirely token1, the depreciating asset).
6. `_scaleWeightsToShares` scales to deposit ~1000e18 token1 and ~0 token0 — within Alice's caps.
7. Alice's liquidity is now entirely in token1 at a price that has already fallen; she suffers immediate impermanent loss relative to the price at which she intended to deposit.

  function _validateBinAndBinPosition(
    address pool,
    int8 minimalCurBin,
    uint104 minimalPosition,
    int8 maximalCurBin,
    uint104 maximalPosition
  ) internal view {
    if (minimalCurBin > maximalCurBin) {
      revert CursorOutOfBounds(0, 0, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }

    (, int8 curBinIdx, uint104 curPosInBin,,,) = PoolStateLibrary._slot0(pool);

    int256 curBin = curBinIdx;
    if (curBin < minimalCurBin || curBin > maximalCurBin) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == minimalCurBin && curPosInBin < minimalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
    if (curBinIdx == maximalCurBin && curPosInBin > maximalPosition) {
      revert CursorOutOfBounds(curBinIdx, curPosInBin, minimalCurBin, minimalPosition, maximalCurBin, maximalPosition);
    }
  }
