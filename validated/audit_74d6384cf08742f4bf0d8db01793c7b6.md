### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezal of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the new `class_hash` supplied by the calling contract corresponds to a previously declared class. Any contract can call `replace_class` with an arbitrary, undeclared class hash. The OS will commit this invalid class hash into the contract's `StateEntry` without constraint. The contract becomes permanently uncallable, freezing any funds it holds.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function reads the requested `class_hash` directly from the syscall request and writes it into `contract_state_changes` with no check that the hash exists in `contract_class_changes` (the declared-class trie):

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The inline TODO comment explicitly acknowledges the missing constraint. The OS Cairo program imposes no algebraic constraint that `class_hash` must appear as a key in `contract_class_changes`. Because the OS is the authoritative proving layer, any proof that includes such a `replace_class` call is accepted as valid, regardless of whether the class was ever declared.

This is the direct analog of the Nocturne `nocturne_setSpendKey` vulnerability: a critical variable (`class_hash` in `StateEntry`) can be set to an arbitrary value by any caller, with no allowlist or existence check enforced at the protocol level. [1](#0-0) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once the OS commits a `StateEntry` whose `class_hash` does not correspond to any entry in the class trie, every subsequent call to that contract address will fail at class-lookup time. No valid execution path can proceed. Any ETH, ERC-20 tokens, or other assets held in the contract's storage become permanently inaccessible. The committed state root is final; there is no on-chain mechanism to recover from an invalid class hash embedded in a proven state transition.

---

### Likelihood Explanation

The attack is reachable by any unprivileged contract deployer:

1. A malicious actor deploys a contract that appears to be a legitimate vault, DEX pool, or token contract.
2. Users deposit funds into the contract.
3. The malicious contract internally calls the `replace_class` syscall with an arbitrary felt value (e.g., `0xdeadbeef`) that was never declared.
4. The OS executes `execute_replace_class`, writes the invalid class hash into `contract_state_changes`, and produces a valid proof — because no Cairo constraint prevents it.
5. The new state root is finalized on L1. The contract is permanently bricked; all deposited funds are frozen.

No privileged role, leaked key, or network-level attack is required. The only prerequisite is the ability to deploy a contract and submit a transaction — both are permissionless on StarkNet.

---

### Recommendation

Add a Cairo-level constraint inside `execute_replace_class` that verifies the requested `class_hash` is present in `contract_class_changes` before updating the contract state. Concretely, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the returned compiled class hash is non-zero:

```cairo
// Verify the class has been declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This mirrors the existing pattern used in `execute_declare_transaction` where `assert_not_zero(compiled_class_hash)` is enforced before writing to `contract_class_changes`. [2](#0-1) 

---

### Proof of Concept

**Step 1 — Deploy a malicious vault contract** (Cairo pseudocode):

```cairo
#[starknet::contract]
mod MaliciousVault {
    #[external]
    fn rug(ref self: ContractState) {
        // Pass any undeclared felt as the new class hash.
        starknet::replace_class_syscall(0xdeadbeef_u256.into()).unwrap();
    }
    #[external]
    fn deposit(ref self: ContractState, amount: u256) { /* accept funds */ }
}
```

**Step 2 — Attract deposits.** Users call `deposit`; funds accumulate in the contract's storage.

**Step 3 — Call `rug`.** The contract issues `replace_class(0xdeadbeef)`. The OS executes `execute_replace_class`:

- `class_hash = 0xdeadbeef` is read from the syscall request.
- No constraint checks `contract_class_changes[0xdeadbeef]`.
- `dict_update` writes `StateEntry(class_hash=0xdeadbeef, ...)` into `contract_state_changes`. [3](#0-2) 

**Step 4 — State is proven and finalized.** The OS produces a valid STARK proof. The L1 verifier accepts it. The new state root encodes the contract with `class_hash = 0xdeadbeef`.

**Step 5 — Funds are frozen.** Any future call to the vault resolves `class_hash = 0xdeadbeef`, finds no compiled class, and fails. The deposited funds are permanently inaccessible. [4](#0-3)
