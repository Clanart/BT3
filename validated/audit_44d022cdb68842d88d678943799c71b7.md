### Title
`batch_insert` Bypasses Duplicate-Key Guard, Causing Dual-Mapping Inconsistency and Corrupted Merkle Root — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips the duplicate-key and duplicate-hash checks that `MerkleBlob::insert` enforces. When a key that already exists in the tree is supplied in a batch, `BlockStatusCache::add_leaf` silently overwrites `key_to_index[key]` with the new leaf's index while leaving the old leaf physically present in the blob and its stale hash entry intact in `leaf_hash_to_index`. The result is a permanently split state between the two cache maps and the underlying blob — an exact structural analog of the token→vault mapping overwrite described in the reference report.

---

### Finding Description

`BlockStatusCache` maintains two parallel maps that must always be in 1-to-1 correspondence with every leaf in the blob:

- `key_to_index: HashMap<KeyId, TreeIndex>` — maps each leaf key to its blob index
- `leaf_hash_to_index: HashMap<Hash, TreeIndex>` — maps each leaf hash to its blob index

`MerkleBlob::insert` enforces uniqueness before touching either map:

```rust
// blob.rs:369-374
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`MerkleBlob::batch_insert` only calls `self.insert()` (with its guards) for the **last two items popped from the vector**, and only when the tree currently has ≤ 1 leaf:

```rust
// blob.rs:578-585
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { return Ok(()); };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;
    }
}
```

All remaining items — and **all items when the tree already has ≥ 2 leaves** — are written directly through `insert_entry_to_blob` with no guard:

```rust
// blob.rs:587-602
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    let new_block = Block { ..., node: Node::Leaf(LeafNode { hash, key, value, ... }) };
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;   // no dup check
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` calls `add_leaf`, which performs a plain `HashMap::insert` — silently overwriting any existing entry:

```rust
// blob.rs:188-193
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);          // overwrites old entry
    self.leaf_hash_to_index.insert(leaf.hash, index);   // adds new hash entry
}
```

After a duplicate-key batch insert the state diverges:

| Structure | State |
|---|---|
| `key_to_index[K]` | → **new** index (overwritten) |
| `leaf_hash_to_index[old_hash]` | → **old** index (stale, never removed) |
| `leaf_hash_to_index[new_hash]` | → new index |
| blob at old index | old leaf still present, still wired into tree |
| `free_indexes` | old index **not** freed |

The old leaf remains a live child of its parent internal node. The tree now contains two leaves with the same `KeyId`. The root hash computed by `calculate_lazy_hashes` covers both leaves, but the cache only acknowledges one.

---

### Impact Explanation

**Corrupted Merkle root committed to the DataLayer.** After the split state is created:

1. `calculate_lazy_hashes()` traverses the actual blob structure (both leaves) and produces a root hash that encodes the duplicate-key tree. This root is what gets committed on-chain.

2. `get_proof_of_inclusion(key)` resolves through `key_to_index` to the *new* leaf only, generating a proof anchored to the new leaf's subtree path. The proof is internally consistent but the committed root was computed over a tree that also contains the old leaf — the two are structurally mismatched.

3. The old leaf is permanently unreachable via the cache: `delete(key)` and `upsert(key, ...)` both resolve through `key_to_index` and will operate on the new leaf, leaving the old leaf stuck in the blob forever.

4. `check_integrity` would detect the divergence (`leaf_count` from blob iteration ≠ `key_to_index.len()`), but `check_integrity_on_drop` is only enabled in `#[cfg(test)]` builds; production code never calls it.

The net effect is a committed DataLayer tree root that is structurally invalid: it encodes a key twice, making the root unprovable against any correct Merkle verifier and corrupting all subsequent delta/proof operations on the store.

**Matched allowed impact:** *High — DataLayer Merkle proof/blob/delta logic corrupts tree roots and lets input prove invalid state.*

---

### Likelihood Explanation

`batch_insert` is the primary bulk-load path exposed to Python callers via `py_batch_insert`. Any caller that supplies a key already present in the tree — whether by mistake or by design — triggers the corruption silently (no error is returned). Because the Python binding accepts arbitrary `(KeyId, ValueId)` pairs, an operator or a higher-level DataLayer routine that calls `batch_insert` without pre-filtering existing keys will corrupt the store. The tree must have ≥ 2 leaves for the fast path to activate, which is the normal operating condition for any non-trivial store.

---

### Recommendation

Add the same duplicate guards to the fast path inside `batch_insert` that `insert` already enforces:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insert_entry_to_blob logic
}
```

Alternatively, pre-validate the entire input vector before any mutation begins, mirroring the pattern used by `insert`.

---

### Proof of Concept

```
1. Create MerkleBlob and insert 3 distinct leaves (key=1, key=2, key=3).
   → leaf_count() == 3, so batch_insert fast path is active.

2. Call batch_insert([((key=2, val=99), new_hash)]).
   → leaf_count() > 1, so the `if` block is skipped entirely.
   → insert_entry_to_blob writes a new leaf at a fresh index.
   → add_leaf overwrites key_to_index[2] → new_index.
   → leaf_hash_to_index[old_hash_of_key2] → old_index  (stale)
   → leaf_hash_to_index[new_hash]         → new_index  (new)
   → old leaf at old_index still in blob, still child of its parent.

3. Call calculate_lazy_hashes().
   → Root hash computed over 4 leaves (key=1, key=2 old, key=2 new, key=3).

4. Call get_proof_of_inclusion(key=2).
   → Resolves to new_index via key_to_index.
   → Proof path does not cover old leaf.
   → Proof is inconsistent with the committed root.

5. check_integrity() (test-only) would report:
   IntegrityKeyToIndexCacheLength(4 leaves in blob, 3 entries in key_to_index)
   IntegrityLeafHashToIndexCacheLength(4 leaves in blob, 4 entries in leaf_hash_to_index)
   → Divergence confirmed.
```

    fn insert_entry_to_blob(&mut self, index: TreeIndex, block: &Block) -> Result<(), Error> {
        let new_block_bytes = block.to_bytes()?;
        let extend_index = self.extend_index();
        match index.cmp(&extend_index) {
            Ordering::Greater => return Err(Error::BlockIndexOutOfBounds(index)),
            Ordering::Equal => self.blob.extend_from_slice(&new_block_bytes),

                self.blob[block_range(index)].copy_from_slice(&new_block_bytes);

        match block.node {
            Node::Leaf(leaf) => self.block_status_cache.add_leaf(index, leaf),
            Node::Internal(..) => self.block_status_cache.add_internal(index),
        }
