### Title
L1 Scraper Stops on Non-Retried `NetworkError`, Causing L1 Handler Events to Be Missed During Restart Window - (File: crates/apollo_l1_provider/src/l1_scraper.rs)

### Summary
In `L1Scraper::run()`, the steady-state polling loop only retries `BaseLayerError`. All other errors — including `NetworkError` (transient IPC failure with the L1 provider) — cause the scraper to stop entirely. During the restart window, new L1 events (`L1HandlerTransaction`, `TransactionCancellationStarted`, `TransactionCanceled`, `TransactionConsumed`) are not scraped and not delivered to the L1 provider. If the downtime exceeds the rewind window (`startup_rewind_time_seconds × 1.5`), those events are permanently absent from the provider's `TransactionManager`, and the corresponding L1 handler transactions are never proposed or validated in any block.

### Finding Description

**Root cause — asymmetric error handling in the steady-state loop.**

`L1Scraper::run()` contains a loop that calls `send_events_to_l1_provider()` on every polling interval:

```rust
loop {
    sleep(self.config.polling_interval_seconds).await;
    match self.send_events_to_l1_provider().await {
        Err(L1ScraperError::BaseLayerError(e)) => {
            warn!("BaseLayerError during scraping: {e:?}");   // retried
        }
        Ok(_) => { ... }
        Err(e) => return Err(e),   // ALL other errors → scraper stops
    }
}
``` [1](#0-0) 

`send_events_to_l1_provider()` calls `handle_client_error()` after `add_events()`. When the IPC channel to the L1 provider is temporarily broken, `handle_client_error` maps the failure to `L1ScraperError::NetworkError`:

```rust
L1ProviderClientError::ClientError(client_error) => {
    Err(L1ScraperError::NetworkError(client_error))
}
``` [2](#0-1) 

`NetworkError` is not `BaseLayerError`, so the `Err(e) => return Err(e)` arm fires, terminating `run()`. The `ComponentStarter` implementation then panics:

```rust
async fn start(&mut self) {
    self.run().await.unwrap_or_else(|e| panic!("Runtime Error: {e}"))
}
``` [3](#0-2) 

**Event gap during restart.**

`scrape_from_this_l1_block` is only updated on a *successful* call to `send_events_to_l1_provider()`:

```rust
// Successfully scraped events up to latest l1 block.
self.scrape_from_this_l1_block = Some(latest_l1_block);
``` [4](#0-3) 

When the scraper stops, `scrape_from_this_l1_block` still points to the last *successfully* processed L1 block. Any L1 events emitted on Ethereum between that block and the moment the scraper restarts are not delivered to the provider.

**Restart re-initialization window.**

On restart, `run()` calls `fetch_start_block()`, which rewinds by `startup_rewind_time_seconds × 1.5` from the current L1 tip. If the scraper was down for less than that window, the events are eventually re-scraped. If the downtime exceeds the window, the events are permanently absent from the provider.

**Cascading failure if the provider is still alive.**

If the scraper restarts while the L1 provider is still running (provider did not crash, only the IPC channel was transiently broken), the scraper calls `initialize()` on the already-initialized provider. The provider panics:

```rust
panic!(
    "Called initialize while not in Uninitialized state. Restart service. Provider \
     state: {:?}",
    self.state
);
``` [5](#0-4) 

This forces both components to restart together, extending the event gap further.

**Exact corrupted value.**

The missing events are never inserted into `TransactionManager::records` / `proposable_index`. The batcher's `get_txs()` call returns an empty set for those L1 handler transactions, so they are never staged, never proposed, and never committed to any block. [6](#0-5) 

### Impact Explanation

Valid L1 handler transactions (messages sent from Ethereum to Starknet) are silently dropped from the sequencer's view. They are never included in any block proposal. From the perspective of the L1 contract the messages were sent and paid for; from the perspective of the Starknet state they never existed. This matches **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing**, and in the permanent-loss scenario also **Critical — wrong L1 message / wrong state** because the sequencer's committed state diverges from the L1 contract's authoritative record of sent messages.

### Likelihood Explanation

A transient IPC failure between the scraper and the L1 provider is a realistic operational event (provider pod restart, network blip, OOM kill). The scraper and provider run as separate microservices in the hybrid deployment topology. Any such blip produces a `ClientError` → `NetworkError` path that terminates the scraper immediately rather than retrying.

### Recommendation

1. Add `NetworkError` to the set of retried errors in the steady-state loop, analogous to `BaseLayerError`:
   ```rust
   Err(L1ScraperError::NetworkError(e)) => {
       warn!("Transient NetworkError during scraping: {e:?}");
       // back-off and continue
   }
   ```
2. Before calling `initialize()` on restart, check whether the provider is already initialized (e.g., via a `is_initialized()` probe) and skip re-initialization if so, resuming from the last known `scrape_from_this_l1_block`.
3. Persist `scrape_from_this_l1_block` to durable storage so that a scraper restart does not depend solely on the time-based rewind heuristic.

### Proof of Concept

1. Scraper is in steady state; `scrape_from_this_l1_block = L1Block(N)`.
2. A user sends an L1→L2 message on Ethereum; it lands in L1 block `N+1`.
3. A transient IPC error occurs when the scraper calls `add_events()` on the provider (e.g., provider pod is briefly restarting). `handle_client_error` returns `NetworkError`.
4. The steady-state loop hits `Err(e) => return Err(e)`; `run()` exits; `ComponentStarter::start` panics.
5. During the restart window (pod restart + re-initialization), L1 blocks `N+2 … N+K` are mined, potentially containing additional L1→L2 messages.
6. Scraper restarts; `fetch_start_block()` rewinds to `L1Block(N - rewind)`. If `K × L1_block_time > startup_rewind_time_seconds × 1.5`, blocks `N+1 … N+K` fall outside the new scrape range and their events are never delivered to the provider.
7. The batcher calls `get_txs()` for those L1 handler transactions; the provider returns an empty list; the transactions are never included in any block. [7](#0-6) [8](#0-7) [9](#0-8)

    pub async fn send_events_to_l1_provider(&mut self) -> L1ScraperResult<(), BaseLayerType> {
        if self.scrape_from_this_l1_block.is_none() {
            panic!(
                "Scraper initialization was skipped. Must provide a value for \
                 scrape_from_this_l1_block. Value is given by a call to fetch_start_block() in \
                 run(). Did you forget to call run()?"
            )
        }
        self.assert_no_l1_reorgs().await?;

        let (latest_l1_block, events) = self.fetch_events().await?;
        // TODO(guyn): remove these _every_n_ms because the polling interval is longer.
        trace!("scraped up to {latest_l1_block:?}");
        info_every_n_ms!(1_000, "scraped up to {latest_l1_block:?}");

        // Sending even if there are no events, to keep the flow as simple/debuggable as possible.
        // Perf hit is minimal, since the scraper is on the same machine as the provider (no
        // network). If this gets spammy, short-circuit on events.empty().
        let add_events_result = self.l1_provider_client.add_events(events).await;
        handle_client_error(add_events_result)?;

        L1_MESSAGE_SCRAPER_LATEST_SCRAPED_BLOCK.set_lossy(latest_l1_block.number);
        // Successfully scraped events up to latest l1 block.
        self.scrape_from_this_l1_block = Some(latest_l1_block);

        Ok(())
    }

```rust
fn handle_client_error<BaseLayerType: BaseLayerContract + Send + Sync + Debug>(
    client_result: Result<(), L1ProviderClientError>,
) -> Result<(), L1ScraperError<BaseLayerType>> {
    let Err(error) = client_result else {
        return Ok(());
    };
    match error {
        L1ProviderClientError::ClientError(client_error) => {
            Err(L1ScraperError::NetworkError(client_error))
        }
        L1ProviderClientError::L1ProviderError(L1ProviderError::Uninitialized) => {
            Err(L1ScraperError::NeedsRestart)
        }
        error => panic!("Unexpected error: {error}"),
    }
}

```rust
    /// Accept new events from the scraper.
    #[instrument(skip_all, err)]
    pub fn add_events(&mut self, events: Vec<Event>) -> L1ProviderResult<()> {
        if self.state.is_uninitialized() {
            return Err(L1ProviderError::Uninitialized);
        }

        // TODO(guyn): can we remove this "every sec" since the polling interval is rather long?
        info_every_n_ms!(1000, "Adding {} l1 events", events.len());
        trace!("Adding events: {events:?}");

        for event in events {
            match event {
                Event::L1HandlerTransaction {
                    l1_handler_tx,
                    block_timestamp,
                    scrape_timestamp,
                } => {
                    self.tx_manager.add_tx(l1_handler_tx, block_timestamp, scrape_timestamp);
                }
                Event::TransactionCancellationStarted {
                    tx_hash,
                    cancellation_request_timestamp,
                } => {
                    if !self.tx_manager.exists(tx_hash) {
                        warn!(
                            "Dropping cancellation request for old L1 handler transaction \
                             {tx_hash}: not in the provider and will never be scraped at this \
                             point."
                        );
                        continue;
                    }

                    self.tx_manager
                        .request_cancellation(tx_hash, cancellation_request_timestamp)
                        .inspect(|previous_request_timestamp| {
                            // Re-requesting a cancellation is meaningful for the L1 timelock, but
                            // for the l2 timelock we only consider the first cancellation
                            // relevant.
                            info!(
                                "Dropping duplicated cancellation request for {tx_hash} at \
                                 {cancellation_request_timestamp}, previous request block \
                                 timestamp still stands: {previous_request_timestamp}"
                            );
                        });
                }
                Event::TransactionCanceled { tx_hash } => {
                    info!(
                        "Cancellation finalized for tx_hash: {tx_hash}. Deleting the tx from the \
                         provider records."
                    );
                    self.tx_manager.finalize_cancellation(tx_hash);
                }
                Event::TransactionConsumed { tx_hash, timestamp: consumed_at } => {
                    if let Err(previously_consumed_at) =
                        self.tx_manager.consume_tx(tx_hash, consumed_at, self.clock.unix_now())
                    {
                        // TODO(guyn): need to check if this is really a critical bug, or if we can
                        // log and ignore.
                        panic!(
                            "Double consumption of {tx_hash} at {consumed_at}, previously \
                             consumed at {previously_consumed_at}."
                        );
                    }
                }
            }
        }
        Ok(())
    }
