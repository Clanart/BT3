### Title
`AllocatedKeys` Over-Counted by 1 in `squash_state_diff` Backward-Compat Path Causes Incorrect Fee Charging — (`crates/blockifier/src/state/cached_state.rs`)

### Summary

When `comprehensive_state_diff=false`, `StateCache::squash_state_diff` naively unions the `AllocatedKeys` sets from each per-cache `to_state_diff()` call. A sequence where cache-1 writes K: 0→V and cache-2 writes K: V→0 causes K to appear in the merged `allocated_keys` even though the net storage transition is 0→0. This inflates `n_allocated_keys` by 1, which flows directly into `TransactionReceipt::from_account_tx` and the actual fee charged to the user.

### Finding Description

**`AllocatedKeys::from_storage_diff`** correctly identifies zero-to-nonzero transitions within a single cache: [1](#0-0) 

For a single cache where `writes[K]=V` (V≠0) and `initial_reads[K]=0`, K is correctly included. For a single cache where `writes[K]=0` and `initial_reads[K]=V`, K is correctly excluded (the `*v != Felt::ZERO` guard).

**`squash_state_diff` with `comprehensive_state_diff=false`** then merges per-cache results by raw set union: [2](#0-1) 

This union does not cancel the allocation from cache-1 when cache-2 deallocates the same key. The result is `{K}` instead of `{}`.

**The `comprehensive_state_diff=true` path** avoids this by first merging the raw caches (giving priority to the earliest initial read and latest write), then calling `to_state_diff()` once on the merged cache: [3](#0-2) 

With the merged cache having `initial_reads[K]=0` and `writes[K]=0`, `from_storage_diff` correctly returns an empty set.

**The production call site** passes the `comprehensive_state_diff` flag from `versioned_constants`, meaning the buggy path is live for any Starknet version where this flag is `false`: [4](#0-3) 

The two caches passed are `validate_state_cache` and `execution_state`'s cache — i.e., the validate and execute phases of a **single transaction**. If the validate phase writes K: 0→V and the execute phase writes K: V→0, the bug triggers within one transaction.

The inflated `StateChanges` is passed directly to `TransactionReceipt::from_account_tx`, which calls `count_for_fee_charge`: [5](#0-4) 

`n_allocated_keys` is 1 instead of 0, causing the user to be charged for a storage slot allocation that did not occur.

**The existing test** explicitly skips asserting `allocated_keys` correctness for the `comprehensive_state_diff=false` path, confirming this is a known gap: [6](#0-5) 

The test case `#[case(true, vec![felt!("0x7"), felt!("0x0")], false)]` (base=0, updates=[7,0], expected charged=false) only asserts the correct result when `comprehensive_state_diff=true`.

### Impact Explanation

`n_allocated_keys` is over-counted by 1 for any transaction whose validate+execute phases net-zero a storage key that was zero in the base state. The fee charged to the user is higher than the correct amount. This is a concrete, deterministic economic error: the wrong fee value is computed and enforced by `PostExecutionReport`.

### Likelihood Explanation

Any unprivileged user can trigger this by submitting a single transaction whose `__validate__` writes a fresh storage slot and whose `__execute__` zeroes it back. No operator privilege is required. The trigger condition is fully user-controlled. The bug is active on any Starknet version where `versioned_constants.comprehensive_state_diff == false`.

### Recommendation

In the `comprehensive_state_diff=false` backward-compat branch, after merging `allocated_keys` by union, subtract any key whose **net** write value (from the final merged `state_maps.storage`) is zero:

```rust
// After the loop:
merged_state_changes.allocated_keys.0.retain(|k| {
    merged_state_changes.state_maps.storage.get(k).map_or(false, |v| *v != Felt::ZERO)
});
```

Alternatively, deprecate the `comprehensive_state_diff=false` path entirely if all supported Starknet versions now use `true`.

### Proof of Concept

Concrete trace (no test code needed — the logic is deterministic from the production code):

```
base_state: storage[K] = 0

cache_1 (validate phase):
  initial_reads.storage[K] = 0
  writes.storage[K] = V  (V ≠ 0)
  → to_state_diff(): from_storage_diff({K:V}, {K:0})
    filter: V≠0 AND base==0 → K included
    allocated_keys_1 = {K}

cache_2 (execute phase):
  initial_reads.storage[K] = V
  writes.storage[K] = 0
  → to_state_diff(): from_storage_diff({K:0}, {K:V})
    filter: 0≠0 is FALSE → K excluded
    allocated_keys_2 = {}

squash_state_diff(false):
  merged.allocated_keys = {K} ∪ {} = {K}   ← BUG: should be {}

squash_state_diff(true):
  merged cache: initial_reads[K]=0, writes[K]=0
  from_storage_diff({K:0}, {K:0}): 0≠0 is FALSE → {}
  merged.allocated_keys = {}                ← CORRECT
```

`count_for_fee_charge` returns `n_allocated_keys=1` in the false path, charging the user for one phantom storage allocation.

    pub fn from_storage_diff(
        updated_storage: &HashMap<StorageEntry, Felt>,
        base_storage: &HashMap<StorageEntry, Felt>,
    ) -> Self {
        Self(
            updated_storage
                .iter()
                .filter_map(|(k, v)| {
                    let base_value = base_storage.get(k).unwrap_or(&Felt::ZERO);
                    if *v != Felt::ZERO && *base_value == Felt::ZERO { Some(*k) } else { None }
                })
                .collect(),
        )
