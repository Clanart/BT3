### Title
Unconstrained `recipient` Account in `FinalizeTransfer` and `FinalizeTransferSol` Allows Redirection of Bridged Assets — (File: `solana/programs/bridge_token_factory/src/instructions/user/finalize_transfer.rs`, `finalize_transfer_sol.rs`)

### Summary

The `recipient` account in both `FinalizeTransfer` and `FinalizeTransferSol` Solana instructions is declared as `UncheckedAccount` with no Anchor constraint linking it to the MPC-signed payload. The `process` function never validates that `self.recipient.key()` matches any recipient field in the signed `FinalizeTransferPayload`. Any caller who possesses a valid signed payload (publicly observable from NEAR events) can pass an arbitrary account as `recipient` and redirect bridged tokens or SOL to themselves, permanently stealing the victim's funds.

### Finding Description

**Root cause — `finalize_transfer.rs`:**

The Anchor accounts struct declares `recipient` as a fully unchecked account:

```rust
/// CHECK: this can be any type of account
pub recipient: UncheckedAccount<'info>,
``` [1](#0-0) 

The `token_account` (destination for bridged tokens) is derived as the ATA of this unconstrained `recipient`:

```rust
#[account(
    init_if_needed,
    payer = common.payer,
    associated_token::mint = mint,
    associated_token::authority = recipient,
    token::token_program = token_program,
)]
pub token_account: Box<InterfaceAccount<'info, TokenAccount>>,
``` [2](#0-1) 

Inside `process`, the only fields consumed from the MPC-signed `data: FinalizeTransferPayload` are `destination_nonce`, `amount`, `fee_recipient`, and `transfer_id`. There is no check of the form `self.recipient.key() == data.recipient`:

```rust
pub fn process(&mut self, data: FinalizeTransferPayload) -> Result<()> {
    UsedNonces::use_nonce(data.destination_nonce, ...)?;
    // tokens transferred to self.token_account (ATA of self.recipient)
    // NO check: self.recipient.key() == data.recipient
    let payload = FinalizeTransferResponse {
        token: self.mint.key(),
        amount: data.amount,
        fee_recipient: data.fee_recipient.unwrap_or_default(),
        transfer_id: data.transfer_id,
    }.serialize_for_near(())?;
``` [3](#0-2) 

**Root cause — `finalize_transfer_sol.rs`:**

The same pattern applies for native SOL transfers. `recipient` is `UncheckedAccount` with only `#[account(mut)]`, and SOL is transferred directly to it with no check against the signed payload:

```rust
/// CHECK: this can be any type of account
#[account(mut)]
pub recipient: UncheckedAccount<'info>,
``` [4](#0-3) 

```rust
transfer(
    CpiContext::new_with_signer(..., Transfer {
        from: self.sol_vault.to_account_info(),
        to: self.recipient.to_account_info(),
    }, ...),
    data.amount.try_into()...?,
)?;
``` [5](#0-4) 

**Analogy to the external report:**

The external report's `sell_market_seat` set `check_signer = false`, meaning the program verified a signer existed but never verified the signer *owned* the account being acted upon. Here, the Solana program verifies the MPC signature over the payload but never verifies the `recipient` account *matches* the intended recipient encoded in that payload. The MPC signature commits to transfer parameters (nonce, amount, fee, transfer ID) but the recipient — the most security-critical field — is left entirely to the caller's discretion.

The `FinalizeTransferResponse` posted back to NEAR also omits the recipient: [6](#0-5) 

This means NEAR cannot detect that the wrong recipient received the funds.

### Impact Explanation

**Critical — Direct theft of bridged assets.**

A valid signed payload is a public artifact: it is emitted as a NEAR event when the MPC signs a transfer. Any observer can extract it. Because the Solana program places no constraint on `recipient`, the first caller to submit the payload to Solana controls where the tokens land. The legitimate recipient's nonce is simultaneously consumed (`UsedNonces::use_nonce`), permanently preventing them from claiming their funds. This satisfies the allowed impact: *"Direct theft, unauthorized release, or unauthorized mint of native or bridged assets"* and *"Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds."*

### Likelihood Explanation

**High.** The signed payload is publicly observable from NEAR chain events. No privileged access, leaked key, or colluding MPC signer is required. Any unprivileged Solana user can:
1. Monitor NEAR for `SignTransferEvent` logs containing a valid MPC-signed payload destined for Solana.
2. Call `finalize_transfer` (or `finalize_transfer_sol`) before the legitimate relayer, substituting their own account as `recipient`.
3. The nonce is consumed; the victim's funds are permanently redirected.

Front-running is straightforward since NEAR finality precedes Solana submission and the payload is broadcast publicly.

### Recommendation

Add an explicit constraint in the Anchor accounts struct that binds `recipient` to the value encoded in the signed payload. Because `#[instruction(...)]` already makes `data` available at the accounts-validation stage, this can be enforced without any changes to `process`:

```rust
// In FinalizeTransfer accounts struct:
#[account(
    constraint = recipient.key().to_bytes() == data.payload.recipient
        @ ErrorCode::InvalidRecipient,
)]
pub recipient: UncheckedAccount<'info>,
```

Apply the equivalent fix in `FinalizeTransferSol`. Ensure `FinalizeTransferPayload` includes the intended recipient as a verified field covered by the MPC signature, so the signature actually commits to the destination.

### Proof of Concept

1. Alice initiates a NEAR → Solana bridge transfer of 1000 USDC, with her Solana address `alice_pubkey` as recipient.
2. NEAR MPC signs a `FinalizeTransferPayload` (nonce=N, amount=1000, transfer_id=T) and emits a `SignTransferEvent`.
3. Attacker Bob observes the event and extracts the signed payload.
4. Bob calls `finalize_transfer` on Solana with:
   - `data` = the valid MPC-signed payload (unchanged)
   - `recipient` = `bob_pubkey` (Bob's own account)
   - `token_account` = Bob's ATA for USDC
5. The program verifies the MPC signature ✓, marks nonce N as used ✓, and transfers 1000 USDC to Bob's ATA.
6. Alice's legitimate relayer later attempts to finalize — nonce N is already consumed → transaction fails.
7. Alice permanently loses 1000 USDC; Bob gains 1000 USDC with no prior relationship to the transfer.

```rust
impl FinalizeTransfer<'_> {
    pub fn process(&mut self, data: FinalizeTransferPayload) -> Result<()> {
        UsedNonces::use_nonce(
            data.destination_nonce,
            &self.used_nonces,
            &mut self.config,
            self.authority.to_account_info(),
            self.common.payer.to_account_info(),
            &Rent::get()?,
            self.system_program.to_account_info(),
        )?;

        if let Some(vault) = &self.vault {
            // Native version. We have a proof of token registration by vault existence
            transfer_checked(
                CpiContext::new_with_signer(
                    self.token_program.to_account_info(),
                    TransferChecked {
                        from: vault.to_account_info(),
                        to: self.token_account.to_account_info(),
                        authority: self.authority.to_account_info(),
                        mint: self.mint.to_account_info(),
                    },
                    &[&[AUTHORITY_SEED, &[self.config.bumps.authority]]],
                ),
                data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?,
                self.mint.decimals,
            )?;
        } else {
            // Bridged version. May be a fake token with our authority set but it will be ignored on the near side
            require!(
                self.mint.mint_authority.contains(self.authority.key),
                ErrorCode::InvalidBridgedToken
            );

            mint_to(
                CpiContext::new_with_signer(
                    self.token_program.to_account_info(),
                    MintTo {
                        mint: self.mint.to_account_info(),
                        to: self.token_account.to_account_info(),
                        authority: self.authority.to_account_info(),
                    },
                    &[&[AUTHORITY_SEED, &[self.config.bumps.authority]]],
                ),
                data.amount.try_into().map_err(|_| error!(ErrorCode::AmountOverflow))?,
            )?;
        }

        let payload = FinalizeTransferResponse {
            token: self.mint.key(),
            amount: data.amount,
            fee_recipient: data.fee_recipient.unwrap_or_default(),
            transfer_id: data.transfer_id,
        }
        .serialize_for_near(())?;

        self.common.post_message(payload)?;

        Ok(())
    }
