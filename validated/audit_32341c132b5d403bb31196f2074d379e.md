### Title
Missing Chain Identifier in `deployToken` Signed Payload Enables Cross-Chain Signature Replay — (`evm/src/omni-bridge/contracts/OmniBridge.sol`, `solana/programs/bridge_token_factory/src/state/message/deploy_token.rs`)

---

### Summary

The `deployToken` function in `OmniBridge.sol` and the equivalent `DeployTokenPayload::serialize_for_near()` in the Solana bridge omit any destination chain identifier from the signed payload. In contrast, `finTransfer` correctly binds its payload to the destination chain via `omniBridgeChainId`. Because the same NEAR MPC-derived key signs metadata for all chains and the payload encoding is identical across EVM chains and Solana, a valid `deployToken` signature obtained from one chain can be replayed verbatim on any other chain where the bridge is deployed.

---

### Finding Description

**EVM — `OmniBridge.sol::deployToken()`**

The borsh-encoded payload submitted to the MPC signer and verified on-chain is:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),  // 0x01
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
bytes32 hashed = keccak256(borshEncoded);
if (ECDSA.recover(hashed, signatureData) != nearBridgeDerivedAddress) {
    revert InvalidSignature();
}
``` [1](#0-0) 

Neither the EVM `block.chainid`, the `omniBridgeChainId` storage variable, nor the contract address appear in this payload. Compare this with `finTransfer`, which correctly embeds `omniBridgeChainId` twice (for token and recipient chain fields):

```solidity
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.tokenAddress),
Borsh.encodeUint128(payload.amount),
bytes1(omniBridgeChainId),
Borsh.encodeAddress(payload.recipient),
``` [2](#0-1) 

**Solana — `DeployTokenPayload::serialize_for_near()`**

```rust
fn serialize_for_near(&self, _params: Self::AdditionalParams) -> Result<Vec<u8>> {
    let mut writer = BufWriter::new(Vec::with_capacity(DEFAULT_SERIALIZER_CAPACITY));
    IncomingMessageType::Metadata.serialize(&mut writer)?;
    self.serialize(&mut writer)?; // token, name, symbol, decimals — no chain ID
    ...
}
``` [3](#0-2) 

The `IncomingMessageType::Metadata` discriminant is `1` (second variant of a borsh enum). [4](#0-3) 

The EVM `BridgeTypes.PayloadType.Metadata` is also `1` (second enum value, zero-indexed). [5](#0-4) 

Both chains use keccak256 over the same borsh layout and verify against the same NEAR MPC-derived address. The `SignedPayload::verify_signature` on Solana uses `keccak::hash(&serialized)`: [6](#0-5) 

The `MetadataPayload` type in `omni-types` confirms no chain field exists in the struct that is signed: [7](#0-6) 

**Result**: The borsh-encoded `deployToken` payload is byte-for-byte identical across all EVM deployments and across EVM↔Solana, because the only differentiating byte (`0x01`) is shared by both `PayloadType.Metadata` and `IncomingMessageType::Metadata`.

---

### Impact Explanation

An attacker who observes a successful `deployToken(signatureData, metadata)` call on chain A (e.g., Ethereum) can immediately replay the identical `(signatureData, metadata)` arguments on chain B (e.g., Arbitrum, Base, BNB, Polygon, Solana) before the NEAR bridge officially deploys the token there.

Consequences:

1. **Unauthorized token deployment**: The token is deployed on chain B at an address determined by the ERC-1967 proxy factory nonce at the time of the attacker's call — not the address the NEAR bridge would have chosen.
2. **Permanent bridge breakage for that token on chain B**: `deployToken` checks `require(!isBridgeToken[nearToEthToken[metadata.token]], "ERR_TOKEN_EXIST")`. Once the attacker's deployment succeeds, the NEAR bridge can never officially deploy the same token on chain B. The NEAR bridge's registered token address for chain B will not match the attacker-deployed address, breaking all `finTransfer` calls for that token on chain B.
3. **Irrecoverable lock of bridged assets**: Users who bridge the token from NEAR to chain B will have their NEAR-side tokens locked but will receive nothing on chain B (or receive tokens at the wrong address), permanently freezing their funds. [8](#0-7) 

---

### Likelihood Explanation

The attack is fully permissionless and requires only:
- Monitoring the mempool or confirmed blocks on any chain where `deployToken` is called.
- Submitting the same calldata to the target chain before the NEAR bridge's official deployment.

No privileged access, leaked keys, or colluding MPC signers are required. The window of opportunity exists whenever a token is deployed on one chain but not yet on another — a common state during bridge expansion to new networks.

---

### Recommendation

Include the destination chain identifier in the signed payload for `deployToken`, mirroring the approach used in `finTransfer`. Concretely, prepend `omniBridgeChainId` to the borsh-encoded metadata before hashing:

```solidity
bytes memory borshEncoded = bytes.concat(
    bytes1(uint8(BridgeTypes.PayloadType.Metadata)),
    bytes1(omniBridgeChainId),          // ADD: bind to this chain
    Borsh.encodeString(metadata.token),
    Borsh.encodeString(metadata.name),
    Borsh.encodeString(metadata.symbol),
    bytes1(metadata.decimals)
);
```

Apply the equivalent fix to `DeployTokenPayload::serialize_for_near()` on Solana and to `MetadataPayloadTrait::to_borsh()` on Starknet. The NEAR bridge's `sign_log_metadata` must also include the destination chain in the payload it submits to the MPC signer so that signatures are chain-scoped.

---

### Proof of Concept

1. NEAR bridge calls `sign_log_metadata` for token `"wrap.near"` targeting Ethereum. MPC signer produces `sig` over `keccak256(0x01 || borsh("wrap.near") || borsh("Wrapped NEAR") || borsh("wNEAR") || 0x18)`.
2. Relayer submits `OmniBridge_Ethereum.deployToken(sig, metadata)` — succeeds, token deployed at address `T_eth`.
3. Attacker observes `sig` and `metadata` on Ethereum.
4. Attacker calls `OmniBridge_Arbitrum.deployToken(sig, metadata)` — signature verifies (same key, same payload bytes), token deployed at address `T_arb_attacker`.
5. NEAR bridge later attempts to officially deploy `"wrap.near"` on Arbitrum — reverts with `ERR_TOKEN_EXIST`.
6. NEAR bridge's registered address for `"wrap.near"` on Arbitrum is `T_arb_official ≠ T_arb_attacker`. All `finTransfer` calls for `"wrap.near"` on Arbitrum fail or mint to the wrong token, permanently locking user funds bridged from NEAR.

```rust
impl<P: Payload> SignedPayload<P> {
    pub fn verify_signature(
        &self,
        params: P::AdditionalParams,
        derived_near_bridge_address: &[u8; 64],
    ) -> Result<()> {
        let serialized = self.payload.serialize_for_near(params)?;
        let hash = keccak::hash(&serialized);

        let signature_bytes = &self.signature[0..64];

        let signature = libsecp256k1::Signature::parse_standard_slice(signature_bytes)
            .map_err(|_| ProgramError::InvalidArgument)?;
        require!(!signature.s.is_high(), ErrorCode::MalleableSignature);

        let signer = secp256k1_recover(&hash.to_bytes(), self.signature[64], signature_bytes)
            .map_err(|_| error!(ErrorCode::SignatureVerificationFailed))?;

        require!(
            signer.0 == *derived_near_bridge_address,
            ErrorCode::SignatureVerificationFailed
        );

        Ok(())
    }
```
