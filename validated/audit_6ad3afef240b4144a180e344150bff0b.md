### Title
Missing Declared Class Validation in `execute_replace_class` Allows Permanent Contract Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS does not verify that the replacement class hash corresponds to a previously declared contract class. An explicit `TODO` in the code acknowledges this missing check. A contract that calls `replace_class` with an undeclared or zero class hash will have its class permanently set to an invalid value, making the contract permanently uncallable and freezing all funds stored within it.

---

### Finding Description

In `syscall_impls.cairo`, the `execute_replace_class` function accepts any arbitrary `class_hash` from the request and writes it directly into the contract state without validating that the hash corresponds to a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

The developer-acknowledged TODO at line 898 confirms this check is intentionally missing from the current implementation.

When a subsequent transaction attempts to call the affected contract, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    elm_size=CompiledClassFact.SIZE,
    n_elms=compiled_class_facts_bundle.n_compiled_class_facts,
    key=compiled_class_hash,
);
```

If `class_hash` is undeclared, `compiled_class_hash` resolves to 0 (default dict value), and `find_element` will fail to locate a matching `CompiledClassFact`, causing the OS execution to abort. The contract becomes permanently uncallable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any contract that calls `replace_class` with an undeclared class hash (whether due to a bug in its upgrade mechanism, a missing input validation, or an attacker-controlled call path) will have its class permanently set to an invalid value. All subsequent calls to that contract — including fund withdrawals — will fail at the OS level. Funds stored in the contract's storage are permanently locked with no recovery path, since the contract itself cannot execute any entry point to migrate or rescue them.

---

### Likelihood Explanation

**Moderate.** The attack requires one of the following realistic conditions:

1. A DeFi protocol or wallet contract implements an upgrade mechanism that calls `replace_class` and does not independently validate that the new class hash is declared (relying on the OS to enforce this invariant, which it does not).
2. An attacker finds a reentrancy or access-control vulnerability in a victim contract's upgrade function, allowing them to supply an arbitrary class hash.
3. A contract developer makes an off-by-one or encoding error when computing the class hash to pass to `replace_class`.

The OS is the correct enforcement layer for this invariant. The explicit TODO confirms the protocol team is aware this check is missing. Any contract that relies on the OS to reject invalid class hashes is vulnerable.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that the class hash is present in `contract_class_changes` (i.e., it has been declared). This mirrors the check already performed in `execute_entry_point` when resolving a class hash to a compiled class. Specifically, after reading `class_hash` from the request, perform a lookup in `contract_class_changes` and assert the result is non-zero:

```cairo
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This resolves the acknowledged TODO and closes the gap between the OS's state-transition logic and the invariant enforced during entry point execution.

---

### Proof of Concept

1. Deploy a contract `VaultContract` that holds user funds and exposes an `upgrade(new_class_hash)` function that calls `replace_class(new_class_hash)`.
2. Call `upgrade(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash. The OS `execute_replace_class` accepts this without validation and writes `class_hash=0xdeadbeef` into the contract state.
3. In the next block, attempt to call any entry point on `VaultContract` (e.g., `withdraw`). The OS `execute_entry_point` reads `class_hash=0xdeadbeef`, looks up its compiled class hash (returns 0), then calls `find_element` with key 0, which fails to find a matching `CompiledClassFact`.
4. The call reverts permanently. All funds in `VaultContract` are frozen with no recovery path.
