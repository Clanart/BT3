### Title
Missing Declared Class Hash Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall implementation in the StarkNet OS accepts any arbitrary class hash from the caller without verifying that the hash corresponds to a previously declared contract class. This is directly analogous to the reported vulnerability where `LSSVMPairFactory` accepted any address for `_propertyChecker` without checking it originated from `PropertyCheckerFactory`. A contract can replace its own class with an undeclared hash, permanently corrupting its state and freezing any funds it holds, because no valid OS proof can ever be generated for subsequent calls to that contract.

---

### Finding Description

In `execute_replace_class` (lines 878–916 of `syscall_impls.cairo`), the OS reads `request.class_hash` from the `ReplaceClassRequest` and immediately writes it into `contract_state_changes` without any check that the hash exists in `contract_class_changes` (the declared class registry):

```cairo
func execute_replace_class{...}(contract_address: felt) {
    ...
    let class_hash = request.class_hash;

    // TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
    local state_entry: StateEntry*;
    %{ GetContractAddressStateEntry %}

    tempvar new_state_entry = new StateEntry(
        class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
    );

    dict_update{dict_ptr=contract_state_changes}(
        key=contract_address,
        prev_value=cast(state_entry, felt),
        new_value=cast(new_state_entry, felt),
    );
    ...
}
```

The developer-acknowledged TODO at line 898 explicitly confirms the missing check. [1](#0-0) 

When a subsequent transaction calls the contract whose class was replaced with an undeclared hash, `execute_entry_point` performs:

```cairo
let (compiled_class_hash: felt) = dict_read{dict_ptr=contract_class_changes}(
    key=execution_context.class_hash   // undeclared hash → returns 0
);
...
let (compiled_class_fact: CompiledClassFact*) = find_element(
    array_ptr=compiled_class_facts_bundle.compiled_class_facts,
    ...
    key=compiled_class_hash,           // find_element(key=0) → assertion failure
);
``` [2](#0-1) 

`dict_new()` initializes `contract_class_changes` with default value `0`. [3](#0-2)  A `dict_read` on an undeclared class hash returns `0`, and `find_element` with `key=0` against the compiled class facts bundle will produce an assertion failure, making the entire block proof invalid. The sequencer is forced to permanently exclude any transaction targeting the corrupted contract, freezing all funds it holds.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared hash:
1. The corrupted class hash is committed to the global state trie via `state_update`.
2. Every future block that includes a transaction calling the contract will fail to generate a valid proof (the `find_element` assertion fails at the OS level, not at the contract level — there is no graceful revert path).
3. The sequencer must permanently exclude all calls to the contract.
4. Any ERC-20 balances, ETH, or other assets stored in the contract's storage are irrecoverably frozen.

---

### Likelihood Explanation

**Medium.**

The entry path is reachable by any unprivileged contract that implements a `replace_class` call. Concrete scenarios:

- A contract with a governance or upgrade mechanism (e.g., a vault, a token contract, a multisig) that allows an authorized internal caller to invoke `replace_class` can be tricked or griefed into supplying an undeclared hash.
- A malicious contract author deploys a contract, attracts user deposits, then triggers `replace_class` with a crafted undeclared hash to freeze deposited funds.
- A buggy contract that computes a class hash incorrectly and passes the result to `replace_class` inadvertently corrupts its own state.

No privileged operator access, leaked keys, or external dependency compromise is required — only the ability to deploy a contract and trigger a syscall from within it.

---

### Recommendation

Before writing the new class hash into `contract_state_changes`, verify that it exists in `contract_class_changes` (i.e., it was previously declared). The check should assert that `dict_read{dict_ptr=contract_class_changes}(key=class_hash)` returns a non-zero compiled class hash:

```cairo
// Verify the class hash is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
```

This mirrors the pattern already used in `execute_entry_point` and closes the gap acknowledged by the existing TODO comment.

---

### Proof of Concept

1. Attacker deploys `VaultContract` (a valid declared class) that accepts ETH deposits and exposes an `upgrade(new_class_hash: felt)` function that calls `replace_class(new_class_hash)`.
2. Users deposit funds into `VaultContract`.
3. Attacker calls `upgrade(0xdeadbeef)` where `0xdeadbeef` is never declared via any declare transaction.
4. The OS executes `execute_replace_class`: no validation is performed; `contract_state_changes` is updated with `class_hash=0xdeadbeef`. [4](#0-3) 
5. The block proof is generated successfully (the replace itself is valid at proof time).
6. In the next block, any transaction calling `VaultContract` causes `execute_entry_point` to do `dict_read(key=0xdeadbeef)` → returns `0`, then `find_element(key=0)` → assertion failure. [5](#0-4) 
7. No valid proof can be produced for any block containing a call to `VaultContract`. All deposited funds are permanently frozen.

        // Not enough gas to execute the syscall.
        return ();
