### Title
`deprecated_declared_classes` Silently Zeroed in `CommitmentStateDiff`→`ThinStateDiff` Conversion Corrupts `state_diff_commitment` and `concatenated_counts` in Block Hash — (`crates/blockifier/src/state/cached_state.rs`)

---

### Summary

`CommitmentStateDiff` has no `deprecated_declared_classes` field. The `From<CommitmentStateDiff> for ThinStateDiff` conversion hard-codes `deprecated_declared_classes: Vec::new()`. Every downstream commitment calculation — `calculate_state_diff_hash` and `concat_counts` — consumes this always-empty list, producing a wrong `state_diff_commitment` and a wrong `state_diff_length` inside `concatenated_counts`. Both values are chained directly into `calculate_block_hash`, so any block that contains a deprecated (Cairo 0) declare transaction receives a permanently incorrect block hash.

---

### Finding Description

**Root cause — the zeroed field**

`CommitmentStateDiff` tracks only the four fields the blockifier writes to the Patricia trie:

```rust
// crates/blockifier/src/state/cached_state.rs  L702-L710
pub struct CommitmentStateDiff {
    pub address_to_class_hash: IndexMap<ContractAddress, ClassHash>,
    pub address_to_nonce: IndexMap<ContractAddress, Nonce>,
    pub storage_updates: IndexMap<ContractAddress, IndexMap<StorageKey, Felt>>,
    pub class_hash_to_compiled_class_hash: IndexMap<ClassHash, CompiledClassHash>,
}
``` [1](#0-0) 

The conversion to `ThinStateDiff` unconditionally sets the fifth field to empty:

```rust
// crates/blockifier/src/state/cached_state.rs  L756-L767
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            ...
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),
        }
    }
}
``` [2](#0-1) 

**Propagation path 1 — wrong `state_diff_commitment`**

`calculate_state_diff_hash` explicitly hashes `deprecated_declared_classes`:

```rust
// crates/starknet_api/src/block_hash/state_diff_hash.rs  L30-L41
pub fn calculate_state_diff_hash(state_diff: &ThinStateDiff) -> StateDiffCommitment {
    ...
    hash_chain = chain_deprecated_declared_classes(
        &state_diff.deprecated_declared_classes, hash_chain);
    ...
}
``` [3](#0-2) 

`chain_deprecated_declared_classes` encodes the count and every class hash:

```rust
// crates/starknet_api/src/block_hash/state_diff_hash.rs  L71-L80
fn chain_deprecated_declared_classes(deprecated_declared_classes: &[ClassHash], ...) -> HashChain {
    hash_chain
        .chain(&sorted_deprecated_declared_classes.len().into())
        .chain_iter(...)
}
``` [4](#0-3) 

With `Vec::new()`, the count is always `0` and no class hashes are chained, producing a `state_diff_commitment` that omits every deprecated declared class.

**Propagation path 2 — wrong `state_diff_length` in `concatenated_counts`**

`ThinStateDiff::len()` counts `deprecated_declared_classes`:

```rust
// crates/starknet_api/src/state.rs  L111-L122
pub fn len(&self) -> usize {
    ...
    result += self.deprecated_declared_classes.len();
    ...
}
``` [5](#0-4) 

`calculate_block_commitments` calls `state_diff.len()` to build `concatenated_counts`:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs  L318-L323
let concatenated_counts = concat_counts(
    transactions_data.len(),
    event_leaf_elements.len(),
    state_diff.len(),   // ← always under-counts by |deprecated_declared_classes|
    l1_da_mode,
);
``` [6](#0-5) 

**Both corrupted values enter the block hash**

`calculate_block_hash` chains `concatenated_counts` and `state_diff_commitment` in positions 6 and 7:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs  L253-L281
HashChain::new()
    ...
    .chain(&block_commitments.concatenated_counts)          // wrong state_diff_length
    .chain(&block_commitments.state_diff_commitment.0.0)    // wrong hash
    ...
``` [7](#0-6) 

**Self-consistency masks the error locally**

`BlockExecutionArtifacts::new` derives the `ThinStateDiff` from `CommitmentStateDiff` before calling `calculate_block_commitments`:

```rust
// crates/apollo_batcher/src/block_builder.rs  L170-L179
let (header_commitments, measurements) = calculate_block_commitments(
    &transactions_data,
    ThinStateDiff::from(commitment_state_diff.clone()),  // deprecated_declared_classes = []
    l1_da_mode,
    &block_info.starknet_version,
).await;
``` [8](#0-7) 

The committer's optional hash-verification check also recomputes from the same zeroed `ThinStateDiff`, so it never detects the mismatch. The wrong commitment is stored, and the wrong block hash is finalized.

The blockifier reexecution tool acknowledges the gap explicitly:

> "Note: Blocks before v0.14.0 may include deprecated (Cairo 0) declared classes which are not represented in `CommitmentStateDiff`; those blocks skip hash comparison below." [9](#0-8) 

---

### Impact Explanation

For every block that contains at least one deprecated (Cairo 0) declare transaction:

- `state_diff_commitment` is computed as if `deprecated_declared_classes = []`, diverging from the value any correct verifier would compute.
- `concatenated_counts` encodes a `state_diff_length` that is smaller than the true length by the number of deprecated declared classes.
- The resulting block hash is wrong and will not match what any independent implementation (full node, proof verifier, L1 verifier) computes for the same block.

This satisfies: **Critical — Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic for accepted input** (specifically: wrong block commitment/hash for accepted transactions).

---

### Likelihood Explanation

Deprecated declare transactions (v0/v1) remain valid Starknet transaction types. Any unprivileged user who submits such a transaction that is included in a sequenced block triggers the incorrect commitment. The gateway has no explicit filter against them in the code examined. The bug fires deterministically on the first such transaction in any block.

---

### Recommendation

Add `deprecated_declared_classes` to `CommitmentStateDiff` and populate it in the blockifier when a deprecated declare transaction is executed. Update `From<CommitmentStateDiff> for ThinStateDiff` to copy the field instead of hard-coding `Vec::new()`. Remove the TODO comment once the fix is in place.

---

### Proof of Concept

```
1. User submits a valid deprecated declare (v1) transaction declaring Cairo 0 class C.
2. Sequencer executes it; blockifier produces CommitmentStateDiff with no deprecated_declared_classes field.
3. BlockExecutionArtifacts::new calls ThinStateDiff::from(commitment_state_diff) → deprecated_declared_classes = [].
4. calculate_block_commitments:
     state_diff_commitment = Poseidon("STARKNET_STATE_DIFF0", ..., 0 /*count*/, ...) ← missing C
     concatenated_counts   = concat_counts(n_txs, n_events, len_without_C, da_mode) ← wrong length
5. calculate_block_hash chains both wrong values → block_hash_actual ≠ block_hash_correct.
6. Any independent verifier (full node, L1 verifier, proof system) recomputes the hash with C included
   and rejects the block or detects a commitment mismatch.
```
