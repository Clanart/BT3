Audit Report

## Title
`MetricOmmPoolLiquidityAdder` accepts arbitrary pool addresses without factory validation, enabling a malicious pool to drain user-approved tokens — (File: metric-periphery/contracts/MetricOmmPoolLiquidityAdder.sol)

## Summary

`MetricOmmPoolLiquidityAdder` accepts a caller-supplied `pool` address in `addLiquidityExactShares` and `addLiquidityWeighted` and interacts with it via `metricOmmModifyLiquidityCallback` without verifying the address was deployed by the legitimate `MetricOmmPoolFactory`. A malicious actor can deploy a fake contract implementing `IMetricOmmPoolActions`, supply it as the `pool` argument, and use the adder's own callback payment logic to drain tokens from any user who interacts with it, up to the caller-provided `maxAmountToken0`/`maxAmountToken1` caps. Note: the router claim is incorrect — `MetricOmmSimpleRouter` already validates every pool via `_requireFactoryPool` inside `_setNextCallbackContext` and `_requireExpectedCallbackCaller`, both of which call `FACTORY.isPool(pool)`.

## Finding Description

`MetricOmmPoolLiquidityAdder` has no `factory` reference and performs no registry check on the `pool` argument. The NatSpec at lines 19–21 explicitly acknowledges this:

```
/// @dev The caller is responsible for supplying a legitimate pool address and other non-malicious parameters.
///      This contract does not verify the pool against the factory; a malicious pool can request token pulls up to
///      the caller-provided max caps during callback settlement.
```

The exploit path:

1. `addLiquidityExactShares(maliciousPool, ...)` or `addLiquidityWeighted(maliciousPool, ...)` is called by a victim.
2. `_addLiquidity` stores `maliciousPool` as the expected callback caller via `_setPayContext(pool, payer, maxAmountToken0, maxAmountToken1)`.
3. `IMetricOmmPoolActions(maliciousPool).addLiquidity(...)` is called.
4. The malicious pool's `addLiquidity` immediately calls back `metricOmmModifyLiquidityCallback(max0, max1, abi.encode(KIND_PAY))`.
5. The callback guard at line 164 checks only `msg.sender == expectedPool` — which is the malicious pool — and passes.
6. The cap check at line 165 passes because the malicious pool requested exactly `max0`/`max1`.
7. The callback reads `token0`/`token1` from `IMetricOmmPool(msg.sender).getImmutables()` — the malicious pool returns attacker-controlled token addresses.
8. `pay(token0, payer, msg.sender, amount0Delta)` and `pay(token1, payer, msg.sender, amount1Delta)` pull tokens from the victim to the malicious pool.
9. The malicious pool's `addLiquidity` returns `(max0, max1)`, the outer `try` succeeds, and the victim receives no LP position.

The existing guard `if (msg.sender != expectedPool) revert InvalidCallbackCaller(...)` is insufficient because `expectedPool` was set to the malicious pool by the victim's own call. The factory's `isPool` registry is never consulted.

By contrast, `MetricOmmSwapRouterBase._setNextCallbackContext` calls `_requireFactoryPool(pool)` which enforces `FACTORY.isPool(pool)` before storing the pool, and `_requireExpectedCallbackCaller` additionally re-checks `FACTORY.isPool(caller)` in the callback — making the router immune to this attack.

## Impact Explanation

Direct loss of user principal. A victim who calls `addLiquidityExactShares` or `addLiquidityWeighted` with a malicious pool address loses up to `maxAmountToken0` of token0 and `maxAmountToken1` of token1 — the exact amounts they authorized as slippage caps. The malicious pool controls which token addresses are returned from `getImmutables()`, so it can target any ERC-20 the victim has approved to the adder. No LP shares are minted in return. This matches the "direct loss of user principal" allowed impact at Medium severity, gated by social engineering to supply the malicious pool address.

## Likelihood Explanation

- Deploying a contract implementing `IMetricOmmPoolActions` and `IMetricOmmPool` is permissionless and trivial.
- Users interacting via front-ends resolve pool addresses from external sources; a phishing UI, a compromised integration, or a misleading pool listing is sufficient.
- Users commonly grant large or unlimited ERC-20 approvals to liquidity adder contracts.
- The `maxAmountToken0`/`maxAmountToken1` caps are set by the victim themselves, so the attacker can drain up to whatever the victim considers an acceptable slippage bound.
- The factory's `isPool` registry is already populated and queryable; the missing check is a single call away.

## Recommendation

Add a `factory` immutable to `MetricOmmPoolLiquidityAdder` (mirroring `MetricOmmSwapRouterBase`) and validate every pool address before `_setPayContext`:

```solidity
IMetricOmmPoolFactory internal immutable FACTORY;

constructor(address weth, address factory) PeripheryPayments(weth) {
    if (factory == address(0)) revert InvalidFactory();
    FACTORY = IMetricOmmPoolFactory(factory);
}

function _requireFactoryPool(address pool) internal view {
    if (!FACTORY.isPool(pool)) revert InvalidPool(pool);
}
```

Call `_requireFactoryPool(pool)` at the top of `_addLiquidity` (before `_setPayContext`) and at the top of `addLiquidityWeighted` before the probe call. This mirrors the pattern already correctly implemented in `MetricOmmSwapRouterBase`.

## Proof of Concept

```solidity
contract MaliciousPool {
    address adder;
    address victim;
    address tokenToDrain;
    uint256 drainAmount;

    // IMetricOmmPoolActions.addLiquidity
    function addLiquidity(address, uint80, LiquidityDelta calldata, bytes calldata, bytes calldata)
        external returns (uint256, uint256)
    {
        // Callback with full cap amounts
        IMetricOmmModifyLiquidityCallback(adder).metricOmmModifyLiquidityCallback(
            drainAmount, 0, abi.encode(uint8(1)) // KIND_PAY
        );
        return (drainAmount, 0);
    }

    // IMetricOmmPool.getImmutables — returns attacker-controlled tokens
    function getImmutables() external view returns (PoolImmutables memory) {
        return PoolImmutables({ token0: tokenToDrain, token1: address(0), ... });
    }
}

// Attack:
// 1. Attacker deploys MaliciousPool targeting victim's USDC approval
// 2. Victim approves adder for 10_000 USDC
// 3. Victim calls adder.addLiquidityExactShares(
//        MaliciousPool, victim, 0, deltas, 10_000e6, 0, ""
//    )
// 4. _addLiquidity stores MaliciousPool as expectedPool, victim as payer, max0=10_000e6
// 5. MaliciousPool.addLiquidity calls back metricOmmModifyLiquidityCallback(10_000e6, 0, KIND_PAY)
// 6. Guard passes: msg.sender == expectedPool == MaliciousPool
// 7. Cap check passes: 10_000e6 <= max0
// 8. getImmutables() returns token0 = USDC
// 9. pay(USDC, victim, MaliciousPool, 10_000e6) executes
// 10. Victim loses 10_000 USDC, receives no LP position
```
