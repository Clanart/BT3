### Title
P2P Sync Stores Tampered `class_hash_to_compiled_class_hash` Without Verifying Against `state_diff_commitment` — (`crates/apollo_p2p_sync/src/client/state_diff.rs`)

### Summary

The p2p sync client validates incoming state diff data only by checking `state_diff_length` (an entry count), never by verifying the content against the cryptographic `state_diff_commitment` stored in the block header. A malicious peer can deliver a `ThinStateDiff` with a tampered `class_hash_to_compiled_class_hash` mapping that has the correct entry count but wrong `CompiledClassHash` values. The tampered mapping passes all checks and is written verbatim into `compiled_class_hash_table`, corrupting the CASM lookup for the affected class.

### Finding Description

**Entrypoint — `parse_data_for_block` in `state_diff.rs`:**

The p2p sync assembles a `ThinStateDiff` from peer-delivered `StateDiffChunk` messages. The only length guard is:

```rust
// state_diff.rs:62-70
let target_state_diff_len = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    .expect("...")
    .state_diff_length
    .ok_or(...)?;

while current_state_diff_len < target_state_diff_len { ... }
``` [1](#0-0) 

`state_diff_length` is a plain integer count of entries, not a cryptographic commitment. A `DeclaredClass` chunk contributes exactly 1 to the count regardless of what `compiled_class_hash` value it carries. [2](#0-1) 

After the length check passes, the only remaining validation is `validate_deprecated_declared_classes_non_conflicting`, which checks for duplicate deprecated class hashes — it does not touch `class_hash_to_compiled_class_hash` at all. [3](#0-2) 

**No commitment verification anywhere:**

A global search for any pattern matching `state_diff_commitment.*verify`, `verify.*state_diff_commitment`, or `calculate_state_diff_commitment` returns zero matches in production code. The `state_diff_commitment` field is written into the block header during header sync and stored, but it is never read back to validate the state diff payload. [4](#0-3) 

**Unchecked write path — `append_state_diff` → `write_compiled_class_hashes`:**

```rust
// state/mod.rs:556-561
write_compiled_class_hashes(
    &thin_state_diff.class_hash_to_compiled_class_hash,
    &self.txn,
    block_number,
    &compiled_class_hash_table,
)?;
``` [5](#0-4) 

`write_compiled_class_hashes` iterates the map and inserts each `(class_hash, block_number) → compiled_class_hash` pair directly, with no cross-check against the header's `state_diff_commitment`. [6](#0-5) 

**p2p write-to-storage:**

```rust
// state_diff.rs:34
storage_writer.begin_rw_txn()?.append_state_diff(self.1, self.0)?.commit()?;
``` [7](#0-6) 

The tampered `ThinStateDiff` is committed to the database in a single transaction with no intervening guard.

### Impact Explanation

`compiled_class_hash_table` is the authoritative source for the `CompiledClassHash` of every declared Sierra class. Downstream CASM lookup uses this table to resolve which compiled artifact to execute. Storing a wrong `CompiledClassHash C'` for `class_hash_K` causes every subsequent execution of a contract of class `K` to either fail (if `C'` has no corresponding CASM) or silently execute the wrong bytecode (if `C'` happens to match a different class's CASM). This satisfies the audit's **Critical** impact: *Wrong compiled class, CASM/native artifact, class hash, or contract code selected for execution.*

### Likelihood Explanation

Any peer in the p2p network can serve state diff data. The p2p sync client does not cryptographically verify block hash signatures before accepting headers, and it does not verify state diff content against the committed hash at all. The attack requires only that the attacker operate a reachable p2p peer and serve a `DeclaredClass` chunk with the correct `class_hash` but a substituted `compiled_class_hash`. No operator or validator privileges are needed.

### Recommendation

After assembling the full `ThinStateDiff` in `parse_data_for_block`, compute its `state_diff_commitment` using the existing `calculate_state_diff_hash` logic (already present in `crates/starknet_api/src/block_hash/state_diff_hash.rs`) and compare it against the `state_diff_commitment` stored in the block header. Reject the peer and return `BadPeerError` on mismatch, mirroring the existing `WrongStateDiffLength` guard pattern. [8](#0-7) 

### Proof of Concept

```rust
// Conceptual Rust unit test (production path only — no mocks)
use apollo_storage::state::{StateStorageReader, StateStorageWriter};
use starknet_api::block::BlockNumber;
use starknet_api::core::{ClassHash, CompiledClassHash};
use starknet_api::state::ThinStateDiff;
use indexmap::IndexMap;

let class_hash_k = ClassHash(1u64.into());
let correct_c   = CompiledClassHash(2u64.into());
let tampered_c  = CompiledClassHash(0xdeadbeefu64.into());

// Build a ThinStateDiff where the attacker substitutes tampered_c for correct_c.
let mut tampered_diff = ThinStateDiff::default();
tampered_diff.class_hash_to_compiled_class_hash
    .insert(class_hash_k, tampered_c);  // attacker-controlled value

// append_state_diff writes tampered_c directly — no commitment check.
writer.begin_rw_txn().unwrap()
    .append_state_diff(BlockNumber(0), tampered_diff).unwrap()
    .commit().unwrap();

// Read back via compiled_class_hash_table.
let txn = reader.begin_ro_txn().unwrap();
let state_reader = txn.get_state_reader().unwrap();
let stored = state_reader
    .get_compiled_class_hash_at(
        StateNumber::unchecked_right_after_block(BlockNumber(0)),
        &class_hash_k,
    ).unwrap();

// Assertion: stored value equals the tampered hash, not the correct one.
assert_eq!(stored, Some(tampered_c));   // passes — corruption confirmed
assert_ne!(stored, Some(correct_c));    // correct hash was never stored
```

The test would pass as written, demonstrating that `compiled_class_hash_table` accepts and persists the attacker-supplied `CompiledClassHash` without any commitment binding.

fn write_compiled_class_hashes<'env>(
    compiled_class_hashes: &IndexMap<ClassHash, CompiledClassHash>,
    txn: &DbTransaction<'env, RW>,
    block_number: BlockNumber,
    compiled_class_hash_table: &'env CompiledClassHashTable<'env>,
) -> StorageResult<()> {
    for (class_hash, compiled_class_hash) in compiled_class_hashes {
        compiled_class_hash_table.insert(txn, &(*class_hash, block_number), compiled_class_hash)?;
    }
    Ok(())
}
