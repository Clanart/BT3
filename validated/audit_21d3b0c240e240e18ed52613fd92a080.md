### Title
Unchecked Felt Arithmetic Overflow in `compute_max_possible_fee` Enables Fee Bypass — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/transaction_impls.cairo`)

---

### Summary

The `compute_max_possible_fee` function in the StarkNet OS computes the maximum chargeable fee using raw felt arithmetic multiplication of `max_amount * max_price_per_unit` without any overflow/range check on the product. Because Cairo's felt arithmetic is modular over a prime field (P ≈ 2²⁵¹), and `max_price_per_unit` is only bounded by `assert_nn` (i.e., ≤ P/2 ≈ 2²⁵⁰) while `max_amount` is bounded to 2⁶⁴−1, the product can wrap around P and produce a value far smaller than the true fee ceiling — including 0. The `charge_fee` function then enforces `assert_nn_le(actual_fee, max_fee)`, so if `max_fee` wraps to 0, the OS immediately returns without charging any fee at all. This is directly analogous to the LP oracle using unguarded spot arithmetic: both compute a critical accounting value without safeguards against the arithmetic producing a manipulated result.

---

### Finding Description

`compute_max_possible_fee` computes:

```cairo
return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit
     + l2_gas_bounds.max_amount * (l2_gas_bounds.max_price_per_unit + tx_info.tip)
     + l1_data_gas_bounds.max_amount * l1_data_gas_bounds.max_price_per_unit;
```

All arithmetic is in felt (mod P). No range check is placed on the product of any `max_amount * max_price_per_unit` term, nor on the final sum. [1](#0-0) 

The only bounds enforced on the inputs come from `pack_resource_bounds`, called during transaction hash computation:

- `assert_nn_le(max_amount, 2**64 - 1)` — caps `max_amount` at 2⁶⁴−1
- `assert_nn(max_price_per_unit)` — only asserts non-negativity (i.e., ≤ P/2 ≈ 2²⁵⁰); **no upper bound** [2](#0-1) 

With `max_amount` up to 2⁶⁴−1 and `max_price_per_unit` up to P/2 ≈ 2²⁵⁰, the product can reach 2³¹⁴, which wraps around P (≈ 2²⁵¹) multiple times. An attacker can choose `max_price_per_unit` such that the product — and therefore the entire sum — reduces to 0 or a negligibly small value modulo P.

`charge_fee` then uses this wrapped result as the fee ceiling:

```cairo
let max_fee = compute_max_possible_fee(tx_info=tx_info);
if (max_fee == 0) {
    return ();  // No fee charged at all.
}
...
assert_nn_le(calldata.amount.low, max_fee);  // Actual fee must be ≤ max_fee.
``` [3](#0-2) 

If `max_fee` wraps to 0, the function returns immediately at line 123–125 without executing any ERC-20 transfer. If it wraps to a small value (e.g., 1), `assert_nn_le(actual_fee, 1)` forces the sequencer to charge at most 1 unit regardless of actual execution cost.

The `tip` field is also added to `l2_gas_bounds.max_price_per_unit` without a combined overflow check, widening the attack surface further. [4](#0-3) 

---

### Impact Explanation

**Direct loss of funds (Critical).** The sequencer is the recipient of all transaction fees. If `compute_max_possible_fee` wraps to 0, `charge_fee` returns without executing the ERC-20 transfer to the sequencer. The sequencer bears the full cost of executing the transaction (L1 data availability, L2 computation) while receiving zero fee. At scale, an attacker can drain sequencer revenue entirely by submitting a stream of crafted transactions that execute arbitrary computation for free.

Additionally, free execution enables unbounded spam: an attacker can flood the network with zero-cost transactions, exhausting block capacity and causing **network inability to confirm legitimate transactions** — a total network shutdown.

---

### Likelihood Explanation

The attack requires only a valid V3 transaction with carefully chosen `max_price_per_unit` values. No privileged access, leaked keys, or external dependencies are needed. The attacker is an ordinary transaction sender. The required arithmetic to find a wrapping `max_price_per_unit` is trivial: since P is public and fixed, the attacker solves `max_amount * x ≡ 0 (mod P)` — or more practically, `max_amount * x ≡ small_value (mod P)` — using modular inverse arithmetic. The off-chain sequencer gateway may validate fees using standard 128-bit integer arithmetic (not felt arithmetic), so the crafted transaction would appear to have a large, legitimate max_fee at the gateway layer but produce a wrapped-to-zero result inside the OS proof.

---

### Recommendation

Add an explicit upper-bound range check on `max_price_per_unit` in `pack_resource_bounds` (e.g., `assert_nn_le(max_price_per_unit, 2**128 - 1)`), matching the 128-bit encoding already implied by the packing formula. Then add a corresponding overflow guard in `compute_max_possible_fee` — either by asserting that each product fits within a safe range before summing, or by computing the fee in a wider representation (e.g., using `Uint256` arithmetic) and asserting the high limb is zero. The final sum should also be range-checked to confirm it does not exceed a protocol-defined maximum fee ceiling.

---

### Proof of Concept

**Setup:** StarkNet prime P = 2²⁵¹ + 17·2¹⁹² + 1.

**Craft resource bounds:**
- Choose `max_amount = 3` for all three resource types (passes `assert_nn_le(3, 2^64-1)` ✓)
- Compute `x = (P + 1) / 3`. Since P ≡ 2 (mod 3), P+1 ≡ 0 (mod 3), so x is an integer ≈ 2²⁴⁹ < P/2, passing `assert_nn(x)` ✓
- Set `max_price_per_unit = x` for L1 gas and L1 data gas
- Set `max_price_per_unit = x`, `tip = 0` for L2 gas

**Result in `compute_max_possible_fee`:**
```
3 * x + 3 * x + 3 * x  (mod P)
= 3 * (P+1)/3 * 3  (mod P)
= (P+1) * 3  (mod P)
= 3  (mod P)
```

This yields `max_fee = 3` — a negligible amount (3 wei) — regardless of actual execution cost. The sequencer is forced by `assert_nn_le(actual_fee, 3)` to charge at most 3 units for arbitrarily expensive computation.

To reach `max_fee = 0`, the attacker sets `max_amount = 0` for all bounds (trivially passes all checks), which is already handled by the `if (max_fee == 0) { return (); }` branch — confirming the OS explicitly supports zero-fee execution with no further validation. [5](#0-4)

// Returns the maximum possible fee that can be charged for the transaction.
func compute_max_possible_fee(tx_info: TxInfo*) -> felt {
    tempvar resource_bounds: ResourceBounds* = tx_info.resource_bounds_start;
    let n_resource_bounds = (tx_info.resource_bounds_end - resource_bounds) / ResourceBounds.SIZE;

    // Only V3 transactions with all resource bounds are supported.
    assert tx_info.version = 3;
    assert n_resource_bounds = 3;

    tempvar l1_gas_bounds: ResourceBounds = resource_bounds[L1_GAS_INDEX];
    tempvar l2_gas_bounds: ResourceBounds = resource_bounds[L2_GAS_INDEX];
    tempvar l1_data_gas_bounds = resource_bounds[L1_DATA_GAS_INDEX];

    return l1_gas_bounds.max_amount * l1_gas_bounds.max_price_per_unit + l2_gas_bounds.max_amount *
        (l2_gas_bounds.max_price_per_unit + tx_info.tip) + l1_data_gas_bounds.max_amount *
        l1_data_gas_bounds.max_price_per_unit;
}
```
