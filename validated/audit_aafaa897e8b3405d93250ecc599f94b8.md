### Title
Deploy Syscall Always Reports Success Regardless of Constructor Outcome — (`File: crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_deploy` syscall handler in the StarkNet OS unconditionally writes `failure_flag=0` in the response header after calling `deploy_contract`, regardless of whether the constructor actually succeeded or reverted. This is the direct analog of the Teller Finance bug: a critical sub-operation's outcome is never checked, so the caller's state machine proceeds as if the operation succeeded even when it silently failed.

---

### Finding Description

In `execute_deploy` (`syscall_impls.cairo`, lines 527–554), the OS calls `deploy_contract` and then writes the syscall response:

```cairo
with remaining_gas {
    let (retdata_size, retdata) = deploy_contract(
        block_context=block_context, constructor_execution_context=constructor_execution_context
    );
}

// TODO(Yoni, 1/1/2026): support failures.
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=0);
``` [1](#0-0) 

The `deploy_contract` call returns only `(retdata_size, retdata)` — there is no `is_reverted` return value in this call path. The OS therefore has no mechanism to propagate a constructor failure to the calling contract. The `failure_flag` field of `ResponseHeader` is hardcoded to `0` (success) unconditionally, and the developer TODO at line 538 explicitly acknowledges this gap.

Compare this with the `contract_call_helper` path, which correctly propagates `is_reverted` into the response:

```cairo
assert [response_header] = ResponseHeader(gas=remaining_gas, failure_flag=is_reverted);
``` [2](#0-1) 

The `deploy` syscall is the only syscall that does not propagate its failure flag.

---

### Impact Explanation

**Direct loss of funds (Critical).**

Any on-chain contract that:
1. Calls the `deploy` syscall to deploy a sub-contract, and
2. Immediately transfers tokens or assets to the returned `contract_address` assuming the constructor succeeded,

will lose those assets if the constructor reverted. The calling contract receives `failure_flag=0` and `contract_address` in the response and has no way to distinguish a successful deployment from a failed one. The revert log mechanism will have undone the constructor's storage writes, leaving the deployed contract in an uninitialized state, while the caller's funds are transferred to it and become permanently inaccessible or mishandled.

---

### Likelihood Explanation

The `deploy` syscall is a standard, publicly accessible syscall callable by any contract. Any protocol that deploys sub-contracts and then funds them (a common DeFi pattern: factory contracts, vault deployers, proxy deployers) is exposed. An attacker who can influence the constructor's inputs or the state it reads can deliberately cause the constructor to revert, then observe the victim contract proceed to fund the uninitialized address. No privileged access is required.

---

### Recommendation

Return an `is_reverted` flag from `deploy_contract` and propagate it into the `ResponseHeader.failure_flag` field, consistent with how `contract_call_helper` handles call failures. The TODO comment at line 538 already identifies this as pending work; it must be treated as a security fix, not a deferred feature.

---

### Proof of Concept

1. Attacker observes a victim factory contract `F` that: (a) calls `deploy(class_hash, constructor_calldata)`, (b) transfers `N` tokens to the returned address.
2. Attacker manipulates the state read by the constructor (e.g., a storage slot `F` checks) so the constructor reverts.
3. `F` calls `deploy`. The OS runs the constructor, which reverts. The revert log undoes constructor storage writes. `deploy_contract` returns `(retdata_size=0, retdata)`.
4. `execute_deploy` writes `ResponseHeader(gas=remaining_gas, failure_flag=0)` and `DeployResponse(contract_address=X, ...)`.
5. `F` reads `failure_flag=0`, concludes deployment succeeded, and transfers `N` tokens to address `X`.
6. Address `X` exists (class hash is set) but is uninitialized. The `N` tokens are permanently lost or controlled by the attacker if the class at `X` has a backdoor path. [3](#0-2)
