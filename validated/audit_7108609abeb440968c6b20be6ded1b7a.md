### Title
`set_state_header` Accepts Cross-Shard State Header Substitution — (`chain/chain/src/state_sync/adapter.rs`)

### Summary

`ChainStateSyncAdapter::set_state_header` never verifies that the chunk embedded in a `ShardStateSyncResponseHeader` actually belongs to the requested `shard_id`. A malicious peer can respond to a header request for shard Y with a structurally valid header containing shard X's chunk and Merkle proof. All existing checks pass. The wrong state root is committed under `StateHeaderKey(shard_Y, sync_hash)`, causing every subsequent state part to be validated against and applied as shard X's trie data into shard Y's storage. This is the nearcore analog of the vault's "treat all assets at parity" bug: the validation treats all shards' state roots as interchangeable because the `shard_id` binding is never enforced at the header commitment step.

### Finding Description

In `set_state_header`, the function receives a caller-supplied `shard_id` and a peer-supplied `ShardStateSyncResponseHeader`. It performs five checks:

1. `validate_chunk_proofs(&chunk, ...)` — validates the chunk's internal encoding consistency.
2. `verify_path(sync_prev_block_header.chunk_headers_root(), chunk_proof, &ChunkHashHeight(chunk.chunk_hash(), chunk.height_included()))` — proves the chunk is *somewhere* in the block's chunk Merkle tree.
3. `verify_path` on `prev_chunk_header` — same, for the previous chunk.
4. Receipt proof chain validation — uses the caller-supplied `shard_id` to hash receipts, not the chunk's own `shard_id`.
5. `validate_state_root_node(state_root_node, chunk_inner.prev_state_root())` — validates the state root node against the chunk's own state root.

**The missing check:** nowhere does the code assert `chunk.shard_id() == shard_id`.

The Merkle proof at step 2 proves the chunk occupies *some* position in the block's chunk tree, but does not prove it occupies the position corresponding to the requested `shard_id`. A malicious peer can supply shard X's chunk with shard X's valid Merkle path, and the `verify_path` call returns `true` regardless of which shard was requested.

After acceptance, the header is stored:

```rust
let key = borsh::to_vec(&StateHeaderKey(shard_id, sync_hash)).unwrap();
store_update.set_ser(DBCol::StateHeaders, &key, &shard_state_header);
``` [1](#0-0) 

The committed state root is now shard X's state root, stored under shard Y's key.

**Downstream propagation in `set_state_part`:**

```rust
let shard_state_header = self.get_state_header(shard_id, sync_hash)?;
let chunk = shard_state_header.take_chunk();
let state_root = *chunk.take_header().take_inner().prev_state_root();
// state_root is shard X's root, but shard_id is Y
self.runtime_adapter.validate_state_part(shard_id, &state_root, part_id, part)
``` [2](#0-1) 

The `validate_state_part` public wrapper passes `shard_id` only to metrics; the actual implementation `validate_state_part_impl` does not receive `shard_id` at all and validates purely against `state_root` and `part_id`:

```rust
fn validate_state_part_impl(
    &self,
    state_root: &StateRoot,
    part_id: PartId,
    part: &StatePart,
) -> StatePartValidationResult {
    ...
    match Trie::validate_state_part(state_root, part_id, partial_state) { ... }
}
``` [3](#0-2) 

Shard X's state parts therefore pass validation when presented as shard Y's parts, and are stored under `StatePartKey(sync_hash, shard_Y, part_id)`.

**Downstream propagation in `apply_state_part`:**

```rust
let key = StatePartKey(sync_hash, shard_id, part_id);  // shard_Y key
// loads shard X's trie nodes
runtime.apply_state_part(shard_id, &state_root, PartId { idx: part_id, total: num_parts }, &state_part, &epoch_id)?;
``` [4](#0-3) 

`apply_state_part` resolves `shard_uid` from the caller-supplied `shard_id` (Y) and writes shard X's trie nodes into shard Y's storage column. [5](#0-4) 

**Downstream propagation in `set_state_finalize`:**

`set_state_finalize` retrieves the stored header for shard Y, extracts shard X's chunk, and calls `apply_chunk` with `chunk_header.prev_state_root()` (shard X's root) and `shard_uid` for shard Y. Shard X's transactions and receipts are applied to shard Y's trie, producing a post-state root that does not match the canonical chain's expected state root for shard Y. [6](#0-5) 

### Impact Explanation

A validator node syncing shard Y (e.g., at an epoch boundary due to shard shuffling) ends up with shard X's state installed as shard Y's state. The node's `ChunkExtra` for shard Y records the wrong state root. When the node subsequently attempts to produce chunks for shard Y, the chunks reference the wrong pre-state root and are rejected by all other validators. The node misses its chunk production slots and may be slashed. If multiple validators are simultaneously syncing shard Y and a malicious peer serves them all substituted headers, chunk production for shard Y can be disrupted for an entire epoch.

**Severity: High.** The corrupted state is durably committed to RocksDB. Recovery requires a full data wipe and resync.

### Likelihood Explanation

State sync header requests are served by any reachable peer, not only validators. The `StateRequestActor` responds to `StateRequestHeader` messages from any connected peer after only a rate-limit and sync-hash validity check. [7](#0-6) 

A single malicious peer that a syncing validator connects to is sufficient. Shard shuffling (enabled in production) causes validators to resync shards every epoch, making this attack surface continuously active.

### Recommendation

Add an explicit shard binding check at the top of `set_state_header`, immediately after extracting the chunk:

```rust
let chunk = shard_state_header.cloned_chunk();
// ADD: verify the chunk belongs to the requested shard
if chunk.shard_id() != shard_id {
    byzantine_assert!(false);
    return Err(Error::Other(format!(
        "set_shard_state failed: chunk shard_id {:?} does not match \
         requested shard_id {:?}",
        chunk.shard_id(), shard_id
    )));
}
```

Additionally, `validate_state_part_impl` should accept and use `shard_id` to verify that the trie nodes being validated belong to the correct shard's storage namespace, preventing cross-shard part substitution independently of the header check.

### Proof of Concept

**Setup:** 4-shard network, epoch boundary, validator V is assigned to shard 1 in the next epoch and must state-sync it. Malicious peer M is connected to V.

1. V sends `StateRequestHeader { shard_id: ShardId(1), sync_hash }` to M.
2. M responds with a `StateResponseInfo` containing a `ShardStateSyncResponseHeader` where:
   - `chunk` = the valid `ShardChunk` for shard 0 at `sync_hash`
   - `chunk_proof` = the valid Merkle path for shard 0's chunk in `sync_prev_block.chunk_headers_root()`
   - `prev_chunk_header` = shard 0's previous chunk header with its valid Merkle proof
   - `incoming_receipts_proofs` = the actual incoming receipts for shard 1 (public chain data), with valid root proofs — these pass the receipt hash check because the hash uses the caller-supplied `shard_id=1`
   - `state_root_node` = the valid state root node for shard 0's trie
3. V calls `set_state_header(ShardId(1), sync_hash, crafted_header)`.
   - `validate_chunk_proofs` → passes (shard 0's chunk is internally valid).
   - `verify_path(chunk_headers_root, shard_0_merkle_path, ChunkHashHeight(shard_0_hash, height))` → passes.
   - Receipt proof loop uses `shard_id=1` to hash receipts → passes (receipts are real shard 1 receipts).
   - `validate_state_root_node(shard_0_state_root_node, shard_0_prev_state_root)` → passes.
   - Header stored under `StateHeaderKey(ShardId(1), sync_hash)` with shard 0's state root.
4. V downloads and validates state parts: `validate_state_part(ShardId(1), shard_0_state_root, part_id, part)` → passes (shard_id ignored in impl).
5. Parts stored under `StatePartKey(sync_hash, ShardId(1), part_id)`.
6. `apply_state_part` writes shard 0's trie nodes into shard 1's RocksDB column.
7. `set_state_finalize(ShardId(1), sync_hash)` applies shard 0's chunk with shard 1's receipts to shard 1's trie.
8. V's shard 1 state root = shard 0's post-state root ≠ canonical shard 1 state root.
9. V produces chunks for shard 1 referencing the wrong state root; all validators reject them.

                StatePartValidationResult::Invalid
            }
        }
    }

    fn apply_state_part(
        &self,
        shard_id: ShardId,
        state_root: &StateRoot,
        part_id: PartId,
        part: &StatePart,
        epoch_id: &EpochId,
    ) -> Result<(), Error> {
        let _timer = metrics::STATE_SYNC_APPLY_PART_DELAY
            .with_label_values(&[&shard_id.to_string()])
            .start_timer();

        let part = part
            .to_partial_state()
            .expect("Part was already validated earlier, so could never fail here");
        let ApplyStatePartResult { trie_changes, flat_state_delta, contract_codes } =
            Trie::apply_state_part(state_root, part_id, part);
        let tries = self.get_tries();
        let shard_uid = self.get_shard_uid_from_epoch_id(shard_id, epoch_id)?;
        let mut store_update = tries.store_update();
        tries.apply_all(&trie_changes, shard_uid, &mut store_update);
        tracing::debug!(target: "chain", %shard_id, values_count = %flat_state_delta.len(), "inserting values to flat storage");
        // TODO: `apply_to_flat_state` inserts values with random writes, which can be time consuming.
        //       Optimize taking into account that flat state values always correspond to a consecutive range of keys.
        flat_state_delta.apply_to_flat_state(&mut store_update.flat_store_update(), shard_uid);
        self.precompile_contracts(epoch_id, contract_codes)?;
        store_update.commit();

async fn apply_state_part(
    store: Store,
    runtime: Arc<dyn RuntimeAdapter>,
    computation_task_tracker: TaskTracker,
    cancel: CancellationToken,
    sync_hash: CryptoHash,
    shard_id: ShardId,
    part_id: u64,
    num_parts: u64,
    state_root: CryptoHash,
    epoch_id: EpochId,
) -> Result<StatePartApplyResult, near_chain::Error> {
    let key = StatePartKey(sync_hash, shard_id, part_id);
    let key_bytes = borsh::to_vec(&key).unwrap();
    let already_applied = store.exists(DBCol::StatePartsApplied, &key_bytes);
    if already_applied {
        tracing::debug!(target: "sync", ?key, "state part already applied, skipping");
        return Ok(StatePartApplyResult::AlreadyApplied);
    }
    return_if_cancelled!(cancel);
    let handle =
        computation_task_tracker.get_handle(&format!("shard {} part {}", shard_id, part_id)).await;
    return_if_cancelled!(cancel);
    handle.set_status("Loading part data from store");
    let bytes = store
        .get(DBCol::StateParts, &key_bytes)
        .ok_or_else(|| {
            near_chain::Error::DBNotFoundErr(format!(
                "No state part {} for shard {}",
                part_id, shard_id
            ))
        })?
        .to_vec();
    let state_part = StatePart::from_bytes(bytes)?;
    handle.set_status("Applying part data to runtime");
    runtime.apply_state_part(
        shard_id,
        &state_root,
        PartId { idx: part_id, total: num_parts },
        &state_part,
        &epoch_id,
    )?;
    tracing::debug!(target: "sync", ?key, "applied state part");

    // Mark part as applied.
    let mut store_update = store.store_update();
    store_update.set_ser(DBCol::StatePartsApplied, &key_bytes, &true);
    store_update.commit();

    Ok(StatePartApplyResult::Applied)

impl Handler<StateRequestHeader, Option<StatePartOrHeader>> for StateRequestActor {
    fn handle(&mut self, msg: StateRequestHeader) -> Option<StatePartOrHeader> {
        let StateRequestHeader { shard_id, sync_hash } = msg;
        let _timer = metrics::STATE_SYNC_REQUEST_TIME
            .with_label_values(&["StateRequestHeader"])
            .start_timer();
        let _span =
            tracing::debug_span!(target: "sync", "StateRequestHeader", ?shard_id, ?sync_hash)
                .entered();

        if self.throttle_state_sync_request() {
            tracing::debug!(target: "sync", "throttling state sync request for shard");
            metrics::STATE_SYNC_REQUESTS_THROTTLED_TOTAL.inc();
            return None;
        }

        if self.validate_sync_hash(&sync_hash) == SyncHashValidationResult::Rejected {
            metrics::STATE_SYNC_REQUESTS_SERVED_TOTAL
                .with_label_values(&["header", "failed"])
                .inc();
            return None;
        }

        let header = self.state_sync_adapter.get_state_response_header(shard_id, sync_hash);
        let Ok(header) = header else {
            tracing::warn!(target: "sync", "cannot build state sync header");
            metrics::STATE_SYNC_REQUESTS_SERVED_TOTAL
                .with_label_values(&["header", "failed"])
                .inc();
            return Some(new_header_response_empty(shard_id, sync_hash));
        };
        let ShardStateSyncResponseHeader::V2(header) = header else {
            tracing::warn!(target: "sync", "invalid state sync header format");
            metrics::STATE_SYNC_REQUESTS_SERVED_TOTAL
                .with_label_values(&["header", "failed"])
                .inc();
            return None;
        };

        metrics::STATE_SYNC_REQUESTS_SERVED_TOTAL.with_label_values(&["header", "success"]).inc();
        let response = new_header_response(shard_id, sync_hash, header);
        Some(response)
    }
