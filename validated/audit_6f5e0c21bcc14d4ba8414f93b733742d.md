### Title
Replay of Finalized Transfer After `ft_transfer_call` Refund Enables Double-Mint / Double-Unlock — (`near/omni-bridge/src/lib.rs`)

### Summary

When a cross-chain transfer to a NEAR recipient includes a non-empty `msg` field, the bridge uses `ft_transfer_call` (via `send_tokens`) to deliver tokens. If the recipient contract rejects the call (returns the full amount), `fin_transfer_send_tokens_callback` removes the transfer ID from `finalised_transfers` via `remove_fin_transfer`. This erasure of the replay-protection record allows the same source-chain proof to be submitted a second time through `fin_transfer`, minting or unlocking tokens again for a single source-chain lock/burn event.

### Finding Description

The NEAR bridge contract's only replay-protection for incoming transfers is the `finalised_transfers` set. A transfer ID is inserted by `add_fin_transfer` at the start of `process_fin_transfer_to_near`, and removed by `remove_fin_transfer` inside `fin_transfer_send_tokens_callback` when `is_refund_required` returns `true`.

`is_refund_required` returns `true` when `is_ft_transfer_call` is `true` (i.e., `msg` is non-empty) and the resolved `ft_transfer_call` promise result indicates zero tokens were consumed — meaning the recipient's `ft_on_transfer` returned the full amount or panicked. [1](#0-0) 

After the refund path executes:
1. `burn_tokens_if_needed` burns the refunded tokens (for deployed tokens).
2. `revert_lock_actions` re-locks any tokens that were unlocked.
3. `remove_fin_transfer` **deletes the transfer ID from `finalised_transfers`**. [2](#0-1) 

Because `finalised_transfers` is the sole guard against replay in `add_fin_transfer`: [3](#0-2) 

…the same EVM proof can now be re-submitted through `fin_transfer` → `fin_transfer_callback` → `process_fin_transfer_to_near`, which will succeed and mint/unlock tokens a second time. [4](#0-3) [5](#0-4) 

### Impact Explanation

**For deployed (bridged) tokens** (origin chain = EVM): `unlock_tokens_if_needed` is a no-op (origin chain matches), so no locked-token accounting is touched. `send_tokens` calls `mint` on the token contract. On replay, `mint` is called again, creating unbacked token supply — a direct unauthorized mint of bridged assets.

**For native NEAR tokens bridged from EVM**: `unlock_tokens_if_needed` decrements `locked_tokens`. On replay, it decrements again, breaking bridge collateralization and allowing more tokens to be released than were ever locked on the source chain.

Both outcomes match the allowed critical/high impact: unauthorized mint of bridged assets, or balance/accounting corruption that breaks bridge collateralization.

### Likelihood Explanation

The attacker must:
1. Initiate a transfer on EVM with a non-empty `message` field (fully user-controlled in `initTransfer`).
2. Control the NEAR recipient address (set as the `recipient` in the EVM event).
3. Deploy a NEAR contract at that address whose `ft_on_transfer` returns the full amount (trivial to implement).

All three conditions are achievable by an unprivileged bridge user. The EVM `initTransfer` accepts arbitrary `message` and `recipient` values. [6](#0-5) 

### Recommendation

Do **not** remove the transfer ID from `finalised_transfers` in the refund path. The finalization record must be permanent once inserted, regardless of whether the downstream `ft_transfer_call` succeeded or failed. If the transfer needs to be retried (e.g., with a different recipient or empty `msg`), a separate mechanism should be used that does not erase the replay-protection record.

```diff
 if Self::is_refund_required(is_ft_transfer_call) {
     self.burn_tokens_if_needed(...);
     self.revert_lock_actions(&lock_actions);
-    self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);
+    // Do NOT remove from finalised_transfers — the proof must not be replayable.
+    // Refund storage cost separately if needed.
     env::log_str(&OmniBridgeEvent::FailedFinTransferEvent { transfer_message }.to_log_string());
 }
```

### Proof of Concept

1. Attacker deploys a NEAR contract `evil.near` whose `ft_on_transfer` always returns the full `amount` (rejecting all tokens).
2. On EVM, attacker calls `initTransfer(token, amount, fee, 0, "evil.near", "trigger_reject")` — the non-empty `message` ensures `ft_transfer_call` is used on NEAR.
3. Relayer submits the EVM proof to NEAR `fin_transfer`. `add_fin_transfer` inserts `{Eth, N}` into `finalised_transfers`. `send_tokens` calls `mint("evil.near", amount, Some("trigger_reject"))`.
4. `evil.near::ft_on_transfer` returns `amount` (full refund). `fin_transfer_send_tokens_callback` fires: `is_refund_required = true`. `burn_tokens_if_needed` burns the refunded tokens. `remove_fin_transfer` **deletes `{Eth, N}` from `finalised_transfers`**.
5. Attacker (or any relayer) re-submits the identical EVM proof to `fin_transfer`. `add_fin_transfer` succeeds (record was deleted). `send_tokens` mints tokens again to `evil.near`.
6. This time `evil.near::ft_on_transfer` returns `0` (accepts tokens). Attacker now holds tokens minted from a single source-chain lock event — unbacked supply created.

    pub fn fin_transfer(&mut self, #[serializer(borsh)] args: FinTransferArgs) -> Promise {
        require!(
            args.storage_deposit_actions.len() <= 3,
            BridgeError::InvalidStorageAccountsLen.as_ref()
        );
        let mut main_promise = self.verify_proof(args.chain_kind, args.prover_args);

        let mut attached_deposit = env::attached_deposit();

        for action in &args.storage_deposit_actions {
            main_promise =
                main_promise.and(Self::check_or_pay_ft_storage(action, &mut attached_deposit));
        }

                .with_attached_deposit(attached_deposit)
                .with_static_gas(FIN_TRANSFER_CALLBACK_GAS)
                .fin_transfer_callback(
                    &args.storage_deposit_actions,
                    env::predecessor_account_id(),
                ),
        )
    }

    pub fn fin_transfer_callback(
        &mut self,
        #[serializer(borsh)] storage_deposit_actions: &Vec<StorageDepositAction>,
        #[serializer(borsh)] predecessor_account_id: AccountId,
    ) -> PromiseOrValue<Nonce> {
        let Ok(ProverResult::InitTransfer(init_transfer)) = Self::decode_prover_result(0) else {
            env::panic_str(BridgeError::InvalidProofMessage.to_string().as_str())
        };
        require!(
            self.factories
                .get(&init_transfer.emitter_address.get_chain())
                == Some(init_transfer.emitter_address),
            BridgeError::UnknownFactory.as_ref()
        );

        let decimals = self
            .token_decimals
            .get(&init_transfer.token)
            .near_expect(BridgeError::TokenDecimalsNotFound);

        let destination_nonce =
            self.get_next_destination_nonce(init_transfer.recipient.get_chain());
        let transfer_message = TransferMessage {
            origin_nonce: init_transfer.origin_nonce,
            token: init_transfer.token,
            amount: Self::denormalize_amount(init_transfer.amount.0, decimals).into(),
            recipient: init_transfer.recipient,
            fee: Self::denormalize_fee(&init_transfer.fee, decimals),
            sender: init_transfer.sender,
            msg: init_transfer.msg,
            destination_nonce,
            origin_transfer_id: None,
        };

        if let OmniAddress::Near(recipient) = transfer_message.recipient.clone() {
            self.process_fin_transfer_to_near(
                recipient,
                &predecessor_account_id,
                transfer_message,
                storage_deposit_actions,
            )
            .into()
        } else {
            self.process_fin_transfer_to_other_chain(predecessor_account_id, transfer_message);
            PromiseOrValue::Value(destination_nonce)
        }
    }

    fn add_fin_transfer(&mut self, transfer_id: &TransferId) -> NearToken {
        let storage_usage = env::storage_usage();
        require!(
            self.finalised_transfers.insert(transfer_id),
            BridgeError::TransferAlreadyFinalised.as_ref()
        );
        env::storage_byte_cost()
            .saturating_mul((env::storage_usage().saturating_sub(storage_usage)).into())
    }
