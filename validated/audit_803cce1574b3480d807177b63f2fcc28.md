### Title
Unbounded Heap Allocation in `run_block_generator2` INTERNED_GENERATOR Path Enables Consensus-Critical OOM via Crafted Back-Reference Generator - (File: crates/chia-consensus/src/run_block_generator.rs)

### Summary

When the `INTERNED_GENERATOR` consensus flag is active, `run_block_generator2` creates a temporary `decode_allocator` using `Allocator::new()` — which has no heap limit — and immediately calls `node_from_bytes_backrefs` on the raw attacker-supplied generator bytes. CLVM back-reference serialization allows exponential tree expansion from a compact input. Because the heap-limit check is bypassed for this allocator, a crafted generator can exhaust all process memory before the cost gate is ever evaluated, crashing validator nodes and causing consensus divergence.

### Finding Description

In `run_block_generator2`, the `INTERNED_GENERATOR` branch allocates a scratch allocator without any heap bound:

```rust
let mut decode_allocator = Allocator::new();                          // no limit
let program_node = node_from_bytes_backrefs(&mut decode_allocator, program)?;
let interned = intern_tree_limited(&decode_allocator, program_node, u32::MAX as usize)
    .map_err(|_| ValidationErr::Err(ErrorCode::GeneratorRuntimeError))?;
let cost = interned_vbytes(&interned) * constants.cost_per_byte;
// cost check happens HERE — after memory is already consumed
subtract_cost(&mut cost_left, base_cost)?;
``` [1](#0-0) 

The normal (non-`INTERNED_GENERATOR`) code path uses `make_allocator(flags)`, which enforces a 500 MB heap cap when `LIMIT_HEAP` is set: [2](#0-1) 

`Allocator::new()` carries no such cap. `node_from_bytes_backrefs` honours only the allocator's internal limit; with an unlimited allocator it will expand back-references until the OS kills the process.

CLVM back-reference serialization is designed for compression: a single 2-byte back-reference can duplicate an arbitrarily large previously-seen subtree. Nesting k such references produces 2^k nodes from O(k) bytes of input. A ~100-byte generator can therefore expand to hundreds of gigabytes of allocator state before `intern_tree_limited` or the cost gate is reached.

The same pattern appears in the Python-exposed `generator_interned_vbytes` binding:

```rust
let mut a = Allocator::new();   // no limit
let node = node_from_bytes_backrefs(&mut a, program)?;
``` [3](#0-2) 

### Impact Explanation

When `INTERNED_GENERATOR` is active (planned hard-fork flag `0x0800_0000`), any block producer can embed a crafted generator in a block. Every validating node that processes the block calls `run_block_generator2` with this flag and hits the unlimited allocator. Nodes with less RAM OOM-crash; nodes with more RAM may survive and continue the chain. This produces **consensus divergence**: the network splits between nodes that crashed and nodes that continued, satisfying the Critical impact criterion "valid unprivileged CLVM program or block can trigger deterministic consensus divergence or chain halt."

Even before divergence, a single such block causes a denial-of-service against all validators simultaneously, halting block processing.

### Likelihood Explanation

- The `INTERNED_GENERATOR` flag is already implemented and tested in the repository; it is activated by a planned hard fork.
- No privileged access is required: any block producer (including a low-stake farmer) can include a crafted generator.
- The crafted input is tiny (tens of bytes) and indistinguishable from a legitimate generator until deserialization begins.
- The cost gate — the only intended resource guard — is evaluated *after* the unbounded allocation, so it provides no protection.

### Recommendation

Replace `Allocator::new()` with a heap-limited allocator in the `INTERNED_GENERATOR` branch of `run_block_generator2` and in `generator_interned_vbytes`:

```rust
// Before (vulnerable):
let mut decode_allocator = Allocator::new();

// After (safe):
let mut decode_allocator = Allocator::new_limited(500_000_000);
```

Apply the same fix to `generator_interned_vbytes` in `wheel/src/run_generator.rs`. Alternatively, route through `make_allocator(flags)` so that the `LIMIT_HEAP` flag is consistently honoured across all deserialization sites.

### Proof of Concept

Construct a CLVM back-reference bomb:

```
ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ff ...
```

Each `ff` byte introduces a cons pair; back-reference opcodes (e.g. `fe <offset>`) duplicate the previously-built subtree. With 60 nesting levels, the expanded tree exceeds 2^60 nodes. Submit this as the `transactions_generator` of a block with `INTERNED_GENERATOR` active. Every validator node calling `run_block_generator2` will attempt `node_from_bytes_backrefs` into an unlimited `decode_allocator`, exhaust available RAM, and be OOM-killed before the cost check at line 243 is reached.
