### Title
Reentrancy in `initTransfer1155` via Malicious ERC1155 `safeTransferFrom` Callback Enables Unbacked Cross-Chain Minting — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer1155` increments `currentOriginNonce` and then calls `IERC1155(tokenAddress).safeTransferFrom(...)` on a **caller-supplied, unvalidated** token address. A malicious ERC1155 contract can reenter `initTransfer1155` during that call, causing multiple `InitTransfer` events to be emitted with distinct nonces while only one (or zero) actual token transfer occurs. The NEAR bridge processes each event independently by nonce, resulting in unbacked minting of bridged assets.

---

### Finding Description

`initTransfer1155` in `OmniBridge.sol` follows this sequence:

```
1. currentOriginNonce += 1                          // line 448
2. IERC1155(tokenAddress).safeTransferFrom(...)     // line 458 — external call to caller-supplied token
3. initTransferExtension(...)                       // line 468
4. emit InitTransfer(...)                           // line 480
``` [1](#0-0) 

There is no reentrancy guard on this function, and no validation that `tokenAddress` is a legitimate or previously registered ERC1155 contract. [2](#0-1) 

The ERC1155 `safeTransferFrom` standard requires calling `onERC1155Received` on the recipient (`address(this)`). A malicious token can also reenter the bridge **before** or **instead of** calling `onERC1155Received`. The bridge's own `onERC1155Received` guard only checks `operator == address(this)`: [3](#0-2) 

Since the bridge itself is the caller of `safeTransferFrom`, `operator == address(this)` is always satisfied, so the guard does not block the reentrant inner call.

No `ReentrancyGuard` or `nonReentrant` modifier is imported or applied anywhere in the EVM bridge contracts. [4](#0-3) 

---

### Impact Explanation

**Critical — Unauthorized minting of bridged assets / broken bridge collateralization.**

The NEAR bridge's `fin_transfer_callback` processes each `InitTransfer` event by its `origin_nonce`. Because the two reentrant calls produce **different nonces** (N+1 and N+2), both are treated as independent, valid transfers: [5](#0-4) 

The NEAR side mints wrapped tokens for both nonces, but the EVM bridge holds only one (or zero) actual ERC1155 tokens. This directly breaks bridge collateralization and constitutes unauthorized minting of unbacked bridged supply.

---

### Likelihood Explanation

**High.** The attack requires no privileged role:

- Any user can deploy a malicious ERC1155 contract.
- `initTransfer1155` accepts any `tokenAddress` with no whitelist check.
- `logMetadata1155` is also permissionless, allowing the attacker to register the malicious token on-chain so the NEAR bridge recognizes it. [6](#0-5) 

For `OmniBridgeWormhole`, the attacker must supply `msg.value = 2 × wormhole.messageFee() + nativeFee` to fund both Wormhole publishes, which is a trivial cost.

---

### Recommendation

Add OpenZeppelin's `ReentrancyGuardUpgradeable` to `OmniBridge` and apply `nonReentrant` to `initTransfer1155` (and, for defense-in-depth, to `initTransfer` and `finTransfer`):

```solidity
import {ReentrancyGuardUpgradeable} from
    "@openzeppelin/contracts-upgradeable/utils/ReentrancyGuardUpgradeable.sol";

contract OmniBridge is ..., ReentrancyGuardUpgradeable {
    ...
    function initTransfer1155(...) external payable nonReentrant whenNotPaused(...) { ... }
    function initTransfer(...)    external payable nonReentrant whenNotPaused(...) { ... }
    function finTransfer(...)     external payable nonReentrant whenNotPaused(...) { ... }
}
```

Additionally, consider maintaining a whitelist of approved ERC1155 token addresses that `initTransfer1155` will accept, preventing arbitrary malicious tokens from being used as reentry vectors.

---

### Proof of Concept

```
Attack setup:
  MaliciousERC1155.safeTransferFrom(from, to, id, amount, data):
    if (reentrancyArmed) {
        reentrancyArmed = false;
        OmniBridge(bridge).initTransfer1155(address(this), id, amount, fee, nativeFee, recipient, message);
        // inner call completes: nonce = N+2, InitTransfer(N+2) emitted
    }
    // outer call continues: nonce = N+1, InitTransfer(N+1) emitted
    // no tokens are actually transferred to the bridge in either call

Attack execution:
  1. Deploy MaliciousERC1155 (implements IERC1155, reenters on safeTransferFrom).
  2. Call bridge.logMetadata1155(maliciousERC1155, tokenId) — permissionless, registers token.
  3. Set reentrancyArmed = true.
  4. Call bridge.initTransfer1155(maliciousERC1155, tokenId, amount, fee, nativeFee, recipient, message)
       with msg.value = 2 * wormhole.messageFee() + nativeFee.
  5. Bridge increments nonce to N+1, calls maliciousERC1155.safeTransferFrom(...).
  6. Malicious token reenters: bridge increments nonce to N+2, inner safeTransferFrom is a no-op,
       InitTransfer(nonce=N+2) emitted, Wormhole message published.
  7. Outer call resumes: InitTransfer(nonce=N+1) emitted, Wormhole message published.
  8. NEAR bridge processes both events → mints 2×amount of wrapped tokens.
     EVM bridge holds 0 actual ERC1155 tokens.
``` [7](#0-6) [8](#0-7)
