### Title
`ft_transfer` Failure Silently Ignored in `fin_transfer_send_tokens_callback`, Permanently Freezing User Funds - (File: near/omni-bridge/src/lib.rs)

### Summary

When finalizing a bridge transfer to a NEAR recipient using a non-deployed (native NEAR-locked) token with an empty `msg`, the `ft_transfer` call result is never checked in the callback. If `ft_transfer` fails, the transfer is permanently marked as finalised, the origin-chain locked-token accounting is decremented, and the fee is paid to the relayer — but the recipient receives zero tokens with no recovery path.

### Finding Description

In `process_fin_transfer_to_near`, the bridge:

1. Calls `add_fin_transfer` — marks the transfer ID in `finalised_transfers` (prevents replay).
2. Calls `unlock_tokens_if_needed` — decrements origin-chain `locked_tokens` accounting.
3. Calls `send_tokens(token, recipient, amount, &msg)` chained to `fin_transfer_send_tokens_callback(..., !msg.is_empty(), ...)`.

When `msg` is empty, `send_tokens` dispatches a plain `ft_transfer` call:

```rust
} else if msg.is_empty() {
    ext_token::ext(token)
        .with_attached_deposit(ONE_YOCTO)
        .with_static_gas(FT_TRANSFER_GAS)   // only 5 TGas
        .ft_transfer(recipient, amount, None)
}
```

The callback is scheduled with `is_ft_transfer_call = false`:

```rust
.then(
    Self::ext(env::current_account_id())
        .with_static_gas(SEND_TOKENS_CALLBACK_GAS)
        .fin_transfer_send_tokens_callback(
            transfer_message,
            &fee_recipient,
            !msg.is_empty(),   // false when msg is empty
            predecessor_account_id,
            lock_actions,
        ),
)
```

Inside `fin_transfer_send_tokens_callback`, the refund gate is:

```rust
if Self::is_refund_required(is_ft_transfer_call) { ... }
```

And `is_refund_required` is:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        // checks promise result
    } else {
        false   // ← always false; promise result is NEVER read
    }
}
```

Because `is_ft_transfer_call` is `false`, the callback unconditionally takes the "success" branch: it pays the fee to the relayer, emits `FinTransferEvent`, and returns — without ever inspecting whether the `ft_transfer` promise succeeded or failed. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

If `ft_transfer` fails (token contract paused, bridge undercollateralized, or the 5 TGas budget exhausted by a complex token contract):

- The recipient receives **zero tokens**.
- The transfer ID is already in `finalised_transfers` — **replay is permanently blocked**.
- `unlock_tokens_if_needed` has already decremented the origin-chain `locked_tokens` — the bridge accounting permanently records the tokens as released.
- `revert_lock_actions` is **never called** (it is only in the refund branch).
- The relayer still receives the fee.

The user's origin-chain tokens remain locked in the origin-chain bridge contract, and the NEAR-side tokens are never minted/transferred. There is no on-chain recovery path without DAO intervention.

This matches the allowed impact: **Permanent freezing / irrecoverable lock of user funds** and **accounting corruption that breaks bridge collateralization**. [4](#0-3) [5](#0-4) 

### Likelihood Explanation

The `ft_transfer` path is taken for every finalization of a transfer to NEAR where:
- The token is **non-deployed** (a native NEAR token locked in the bridge, e.g., USDC.e, wNEAR-wrapped assets), **and**
- The `msg` field is **empty** (the common case — no `ft_transfer_call` receiver logic needed).

Concrete failure triggers reachable by any bridge user:
1. **Token contract paused** — USDC and similar tokens support administrative pausing; a pause during finalization causes `ft_transfer` to revert.
2. **Insufficient gas** — `FT_TRANSFER_GAS` is only 5 TGas; a token contract with non-trivial `ft_transfer` logic (e.g., hooks, storage checks) may exhaust this budget.
3. **Bridge undercollateralization** — if a prior bug or exploit drained the bridge's token balance below the accounted `locked_tokens`, `ft_transfer` reverts with insufficient balance. [6](#0-5) 

### Recommendation

In `fin_transfer_send_tokens_callback`, always check the promise result of the preceding `send_tokens` call, regardless of whether it was an `ft_transfer` or `ft_transfer_call`. When the result indicates failure, execute the same refund/revert logic currently used for the `ft_transfer_call` failure path:

```rust
fn fin_transfer_send_tokens_callback(...) {
    let transfer_failed = if is_ft_transfer_call {
        Self::is_refund_required(true)
    } else {
        // NEW: check ft_transfer result too
        env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT).is_err()
    };

    if transfer_failed {
        self.burn_tokens_if_needed(...);
        self.revert_lock_actions(&lock_actions);
        self.remove_fin_transfer(&transfer_message.get_transfer_id(), storage_owner);
        env::log_str(FailedFinTransferEvent);
    } else {
        // existing success path
    }
}
```

Additionally, consider increasing `FT_TRANSFER_GAS` or making it dynamic (similar to how `ft_transfer_call_gas` is computed in `send_tokens`) to avoid gas-exhaustion failures.

### Proof of Concept

1. Alice bridges 1000 USDC from Ethereum to NEAR. The ETH bridge locks her USDC and emits an `InitTransfer` event.
2. A relayer calls `fin_transfer` on the NEAR bridge with the proof. `fin_transfer_callback` → `process_fin_transfer_to_near` runs:
   - `add_fin_transfer` marks the transfer as finalised.
   - `unlock_tokens_if_needed(ChainKind::Eth, usdc, 1000)` decrements ETH locked_tokens by 1000.
   - `send_tokens(usdc, alice, 1000, "")` schedules `ft_transfer` with 5 TGas.
3. At this moment, USDC on NEAR is paused by Circle (or the 5 TGas budget is exhausted). `ft_transfer` reverts.
4. `fin_transfer_send_tokens_callback` is called with `is_ft_transfer_call = false`.
5. `is_refund_required(false)` returns `false` — the callback takes the success branch.
6. The relayer's fee is paid. `FinTransferEvent` is emitted.
7. Alice's NEAR-side USDC balance: **0**. The transfer ID is in `finalised_transfers`. ETH locked_tokens: decremented. Alice's ETH-side USDC: permanently locked in the ETH bridge contract. [4](#0-3) [7](#0-6) [8](#0-7)
