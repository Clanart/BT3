### Title
Duplicate Key Bypass in `MerkleBlob::batch_insert` Corrupts Tree Root and Enables Forged Proofs of Inclusion — (File: `crates/chia-datalayer/src/merkle/blob.rs`)

---

### Summary

`MerkleBlob::batch_insert` skips all duplicate-key and duplicate-hash validation for the bulk of its input, silently overwriting the `BlockStatusCache` index maps while still writing every leaf node (including duplicates) into the blob and the tree structure. This causes the computed tree root hash to reflect a tree containing phantom duplicate leaf nodes, while the cache only tracks the last occurrence. Any proof of inclusion subsequently generated for the duplicate key is structurally valid against the corrupted root but proves membership in a tree state that does not match the intended committed state — satisfying the DataLayer impact criterion of "corrupts tree roots" and "lets untrusted input prove invalid state."

---

### Finding Description

`MerkleBlob::insert` (the single-item path) guards against duplicates at lines 369–374:

```rust
if self.block_status_cache.contains_key(key) {
    return Err(Error::KeyAlreadyPresent());
}
if self.block_status_cache.contains_leaf_hash(hash) {
    return Err(Error::HashAlreadyPresent());
}
```

`batch_insert` (lines 570–657) uses `self.insert()` — and therefore these guards — only for the first two items, and only when the tree currently has ≤ 1 leaf:

```rust
if self.block_status_cache.leaf_count() <= 1 {
    for _ in 0..2 {
        let Some(((key, value), hash)) = keys_values_hashes.pop() else { ... };
        self.insert(key, value, &hash, InsertLocation::Auto {})?;
    }
}
```

All remaining items (or all items when `leaf_count > 1`) are inserted through the fast-path loop at line 587, which calls `insert_entry_to_blob` directly — bypassing every duplicate check:

```rust
for ((key, value), hash) in keys_values_hashes {
    let new_leaf_index = self.get_new_index();
    ...
    self.insert_entry_to_blob(new_leaf_index, &new_block)?;
    indexes.push(new_leaf_index);
}
```

`insert_entry_to_blob` (lines 1013–1030) calls `block_status_cache.add_leaf`, which performs a plain `HashMap::insert` on both `key_to_index` and `leaf_hash_to_index`:

```rust
fn add_leaf(&mut self, index: TreeIndex, leaf: LeafNode) {
    self.free_indexes.shift_remove(&index);
    self.key_to_index.insert(leaf.key, index);
    self.leaf_hash_to_index.insert(leaf.hash, index);
}
```

`HashMap::insert` silently overwrites the previous entry and returns the old value — no error is raised. The result is:

1. **Both** leaf nodes (original and duplicate) are written to the blob at distinct `TreeIndex` positions.
2. **Both** are included in the `indexes` vector and therefore in the internal-node hash computation that builds the subtree root.
3. The `key_to_index` cache is overwritten to point only to the **last** duplicate's index; the earlier duplicate leaf node is orphaned in the blob (it has a parent pointer set, but the cache no longer knows about it).
4. The final tree root hash is computed from a tree that contains the phantom duplicate leaf, so the root hash does not represent the intended key-value set.

The Python binding `py_batch_insert` (lines 1503–1519) exposes this path directly:

```rust
#[pyo3(name = "batch_insert")]
pub fn py_batch_insert(
    &mut self,
    keys_values: Vec<(KeyId, ValueId)>,
    hashes: Vec<Hash>,
) -> PyResult<()> {
    ...
    self.batch_insert(zip(keys_values, hashes).collect())?;
    Ok(())
}
```

The Python stub confirms the public API: `def batch_insert(self, keys_values: list[tuple[KeyId, ValueId]], hashes: list[bytes32]): ...`

---

### Impact Explanation

After a `batch_insert` with a duplicate `KeyId`:

- The tree root hash is **corrupted**: it commits to a tree with an extra phantom leaf that was never intended to be part of the store.
- `get_proof_of_inclusion(key)` looks up the key in `key_to_index`, finds the last duplicate's index, and walks the tree from that index to the root. The proof is structurally valid against the corrupted root, but the root itself does not represent the intended committed state.
- A verifier who trusts the root hash (e.g., a DataLayer client checking a proof against a published root) will accept a proof that proves membership in a corrupted tree, not the intended one.
- The `check_integrity` function would detect the inconsistency (`leaf_count != key_to_index_cache_length`), but it is not called automatically in production (only in tests via `check_integrity_on_drop`).

This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

---

### Likelihood Explanation

The `batch_insert` Python binding is the primary bulk-insertion API used during DataLayer sync. Any caller — including a DataLayer peer providing a delta — that supplies a `keys_values` list containing a repeated `KeyId` (or a repeated hash) will silently trigger the corruption. No special privilege is required; the input is caller-controlled. The condition is easy to satisfy accidentally (e.g., a buggy sync peer) or intentionally (a malicious peer).

---

### Recommendation

Add duplicate-key and duplicate-hash validation inside the fast-path loop of `batch_insert`, mirroring the checks already present in `insert`:

```rust
for ((key, value), hash) in keys_values_hashes {
    if self.block_status_cache.contains_key(key) {
        return Err(Error::KeyAlreadyPresent());
    }
    if self.block_status_cache.contains_leaf_hash(&hash) {
        return Err(Error::HashAlreadyPresent());
    }
    // ... existing insertion logic
}
```

Alternatively, pre-validate the entire input vector for uniqueness before any insertion begins, so that the tree is never partially mutated on a bad batch.

### Proof of Concept

```python
from chia_rs.datalayer import MerkleBlob, KeyId, ValueId
import hashlib

def h(n: int) -> bytes:
    return hashlib.sha256(n.to_bytes(8, "big")).digest()

# Start with a tree that already has 2 leaves so leaf_count > 1,
# ensuring ALL batch items go through the unchecked fast-path.
blob = MerkleBlob(blob=bytearray())
blob.insert(KeyId(100), ValueId(100), h(100))
blob.insert(KeyId(200), ValueId(200), h(200))
blob.calculate_lazy_hashes()
root_before = blob.get_root_hash()

# batch_insert with a duplicate KeyId(1) — both entries bypass duplicate checks
blob.batch_insert(
    [(KeyId(1), ValueId(10)), (KeyId(2), ValueId(20)), (KeyId(1), ValueId(99))],
    [h(1), h(2), h(3)],   # h(3) is a distinct hash, so hash-dedup won't catch it
)
blob.calculate_lazy_hashes()
root_after = blob.get_root_hash()

# The tree root now commits to 5 leaf nodes (100, 200, 1, 2, 1-duplicate),
# but key_to_index only tracks 4 keys.
# get_proof_of_inclusion(KeyId(1)) returns a proof valid against root_after,
# but root_after does not represent the intended 4-key store.
proof = blob.get_proof_of_inclusion(KeyId(1))
assert proof.valid()   # passes — but root is corrupted

# check_integrity reveals the inconsistency
try:
    blob.check_integrity()
    print("integrity ok (unexpected)")
except Exception as e:
    print(f"integrity failure: {e}")
