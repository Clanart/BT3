### Title
`ProofOfInclusion::valid()` Final Check Is a Tautology — Forged Proofs Pass Validation Without Root Binding — (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

---

### Summary

`ProofOfInclusion::valid()` only verifies internal hash-chain consistency. Its concluding check `existing_hash == self.root_hash()` is a mathematical tautology — it is always `true` after the loop — because `root_hash()` returns the same `combined_hash` value that `existing_hash` was just set to. No external, trusted root hash is ever consulted. An unprivileged attacker who can supply a `ProofOfInclusion` object (via the Python/WASM binding, network deserialization, or direct construction) can forge a proof for any claimed `node_hash` that will unconditionally pass `valid()`.

---

### Finding Description

**Root cause — `valid()` tautology**

```rust
// crates/chia-datalayer/src/merkle/proof_of_inclusion.rs

pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // ← derived entirely from the proof itself
    } else {
        self.node_hash
    }
}

pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {   // loop invariant:
            return false;                             // calculated_hash == layer.combined_hash
        }
        existing_hash = calculated_hash;              // ← existing_hash := layer.combined_hash
    }

    existing_hash == self.root_hash()   // ← TAUTOLOGY
    // After the loop, existing_hash IS last.combined_hash.
    // root_hash() also returns last.combined_hash.
    // This comparison is always true.
}
```

**Two concrete bypass paths**

1. **Empty-layers proof** — `ProofOfInclusion { node_hash: H, layers: [] }` always passes `valid()`. `root_hash()` returns `H`. The attacker chooses `H` freely, claiming any leaf is the entire tree.

2. **Fabricated chain** — The attacker picks any `node_hash` (e.g., the hash of a key-value pair that is *not* in the real tree), then builds a chain of `ProofOfInclusionLayer` values where each `combined_hash` is honestly computed from the previous hash and a chosen `other_hash`. The loop never returns `false`, and the final tautology passes. `root_hash()` returns a fabricated root that does not match the real DataLayer tree root.

**No external root is bound inside `valid()`**

The method is the sole validation API exposed to Python callers:

```python
# wheel/python/chia_rs/datalayer.pyi
class ProofOfInclusion:
    def valid(self) -> bool: ...
    def root_hash(self) -> bytes32: ...
```

A caller who writes `assert proof.valid()` receives no guarantee that the proof corresponds to any real tree. The caller must *additionally* write `assert proof.root_hash() == known_root`, but nothing in the API or documentation enforces or even hints at this requirement. The misleading tautological final check inside `valid()` actively encourages the false belief that `valid()` is a complete check.

**Analog to the external report**

In the Astaria bug, `_isPublicVault(s, to)` was supposed to block transfers to public vaults, but the check relied on on-chain state that did not yet exist at the predicted address, so it silently passed. Here, `valid()` is supposed to block acceptance of forged proofs, but the check relies on an external root hash that is never consulted, so it silently passes for any internally consistent (but fake) proof.

---

### Impact Explanation

An attacker who can deliver a `ProofOfInclusion` to any DataLayer verifier that calls only `proof.valid()` can:

- **Forge inclusion proofs**: claim that an arbitrary key-value pair is present in the DataLayer tree when it is not.
- **Forge exclusion proofs**: the same structural flaw applies to proofs-of-exclusion generated via `MerkleSet.generate_proof` / `validate_merkle_proof` if a caller trusts `valid()` alone.
- **Corrupt committed state**: DataLayer state commitments are on-chain root hashes; accepting a forged proof against a fake root allows an untrusted peer to assert invalid state as proven.

This matches the allowed High impact: *"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."*

---

### Likelihood Explanation

- `ProofOfInclusion` is a `Streamable` type, deserializable from raw bytes received over the network.
- The Python binding exposes `valid()` as the primary (and only named) validation method.
- The tautological final check gives callers false confidence that `valid()` is a complete verification.
- No documentation warns that `proof.root_hash()` must be separately compared to a trusted root.
- The fuzz target and unit tests only call `proof.valid()` after generating proofs from a known-good blob, so the gap is not caught by existing tests.

---

### Recommendation

1. **Add a root-binding overload** that is the canonical validation entry point:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    self.valid_internal() && &self.root_hash() == expected_root
}

fn valid_internal(&self) -> bool {
    // existing loop body, minus the tautological final check
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash, layer.other_hash_side, &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash { return false; }
        existing_hash = calculated_hash;
    }
    true
}
```

2. **Remove or fix the tautological final check** in `valid()` to avoid misleading callers.

3. **Expose `valid_against_root` in the Python binding** and deprecate bare `valid()` for security-sensitive call sites.
