### Title
`CommitmentStateDiff → ThinStateDiff` conversion silently drops `deprecated_declared_classes`, causing `state_diff_length` in `concatenated_counts` / block-hash commitment to undercount Cairo 0 declarations — (`File: crates/blockifier/src/state/cached_state.rs`)

---

### Summary

The `From<CommitmentStateDiff> for ThinStateDiff` conversion in `crates/blockifier/src/state/cached_state.rs` unconditionally sets `deprecated_declared_classes: Vec::new()`. This is the conversion used by `BlockExecutionArtifacts::new` when it calls `calculate_block_commitments`. As a result, any block that contains Cairo 0 (`deprecated`) class declarations will have those classes silently excluded from the `ThinStateDiff` fed into `calculate_block_commitments`, producing a `state_diff_length` that is too small. That wrong length is packed into `concatenated_counts`, which is a direct input to `calculate_block_hash`. The resulting `PartialBlockHash` / `ProposalCommitment` is therefore wrong for any block with deprecated declarations.

---

### Finding Description

**Root cause — the lossy conversion:**

```rust
// crates/blockifier/src/state/cached_state.rs  L756-L768
impl From<CommitmentStateDiff> for ThinStateDiff {
    fn from(commitment_state_diff: CommitmentStateDiff) -> Self {
        Self {
            deployed_contracts: commitment_state_diff.address_to_class_hash,
            storage_diffs: commitment_state_diff.storage_updates,
            class_hash_to_compiled_class_hash: commitment_state_diff
                .class_hash_to_compiled_class_hash,
            nonces: commitment_state_diff.address_to_nonce,
            // TODO(AlonH): Remove this when the structure of storage diffs changes.
            deprecated_declared_classes: Vec::new(),   // ← always empty
        }
    }
}
```

`CommitmentStateDiff` has no `deprecated_declared_classes` field at all — it only tracks `class_hash_to_compiled_class_hash` (Cairo 1 / Sierra). Cairo 0 declarations are tracked separately in `StateMaps::declared_contracts` (a `bool` flag) and are never promoted into `CommitmentStateDiff`. The `TODO` comment acknowledges this is a known structural gap. [1](#0-0) 

**Where the lossy `ThinStateDiff` is used for commitment:**

`BlockExecutionArtifacts::new` calls `ThinStateDiff::from(commitment_state_diff.clone())` and passes the result directly to `calculate_block_commitments`:

```rust
// crates/apollo_batcher/src/block_builder.rs  L170-L176
let (header_commitments, measurements) = calculate_block_commitments(
    &transactions_data,
    ThinStateDiff::from(commitment_state_diff.clone()),  // deprecated_declared_classes = []
    l1_da_mode,
    &block_info.starknet_version,
)
.await;
``` [2](#0-1) 

**How `state_diff_length` enters the block hash:**

Inside `calculate_block_commitments`, `state_diff.len()` is passed to `concat_counts`:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs  L318-L323
let concatenated_counts = concat_counts(
    transactions_data.len(),
    event_leaf_elements.len(),
    state_diff.len(),   // ← wrong: deprecated_declared_classes not counted
    l1_da_mode,
);
``` [3](#0-2) 

`ThinStateDiff::len()` explicitly counts `deprecated_declared_classes`:

```rust
// crates/starknet_api/src/state.rs  L111-L122
pub fn len(&self) -> usize {
    let mut result = 0usize;
    result += self.deployed_contracts.len();
    result += self.class_hash_to_compiled_class_hash.len();
    result += self.deprecated_declared_classes.len();   // ← always 0 from batcher path
    result += self.nonces.len();
    for (_, storage_diffs) in &self.storage_diffs {
        result += storage_diffs.len();
    }
    result
}
``` [4](#0-3) 

`concat_counts` packs `state_diff_length` into a single felt that is chained into the Poseidon block hash:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs  L374-L393
pub fn concat_counts(..., state_diff_length: usize, ...) -> Felt {
    let concat_bytes = [
        to_64_bits(transaction_count).as_slice(),
        to_64_bits(event_count).as_slice(),
        to_64_bits(state_diff_length).as_slice(),   // ← wrong value
        ...
    ].concat();
    Felt::from_bytes_be_slice(concat_bytes.as_slice())
}
``` [5](#0-4) 

The `concatenated_counts` felt is then chained directly into `calculate_block_hash`:

```rust
// crates/starknet_api/src/block_hash/block_hash_calculator.rs  L253-L281
Ok(BlockHash(
    HashChain::new()
        ...
        .chain(&block_commitments.concatenated_counts)   // ← wrong
        ...
        .get_poseidon_hash(),
))
``` [6](#0-5) 

**Downstream propagation — P2P sync uses `state_diff_length` as a termination sentinel:**

The P2P sync client reads `header.state_diff_length` from storage and uses it as the exact byte-count target to know when it has received a complete state diff:

```rust
// crates/apollo_p2p_sync/src/client/state_diff.rs  L62-L70
let target_state_diff_len = storage_reader
    .begin_ro_txn()?
    .get_block_header(block_number)?
    ...
    .state_diff_length
    .ok_or(...)?;

while current_state_diff_len < target_state_diff_len { ... }
``` [7](#0-6) 

If the stored `state_diff_length` in the header is too small (because it was derived from the lossy `ThinStateDiff`), a syncing peer will stop consuming state diff chunks prematurely, accepting a truncated state diff as complete. The P2P server also falls back to computing `state_diff_length` from the stored `ThinStateDiff` when the header field is absent:

```rust
// crates/apollo_p2p_sync/src/server/mod.rs  L230-L237
if header.state_diff_length.is_none() {
    header.state_diff_length = Some(
        txn.get_state_diff(block_number)?.ok_or(...)?.len(),
    );
}
``` [8](#0-7) 

Since the stored `ThinStateDiff` itself was written from the lossy conversion (via `commit_proposal` → `append_state_diff`), the fallback also returns the wrong length. [9](#0-8) 

---

### Impact Explanation

For any block containing at least one Cairo 0 (`deprecated`) class declaration:

1. **Wrong `PartialBlockHash` / `ProposalCommitment`**: The `concatenated_counts` felt encodes an incorrect `state_diff_length`, making the block hash computed by the proposer diverge from what any independent verifier (re-executor, prover, L1 verifier) would compute from the actual state diff. This is a **wrong block commitment** — matching the "Wrong state, receipt, event, L1 message, class hash, storage value, or revert result from blockifier/syscall/execution logic" critical impact.

2. **Wrong `state_diff_commitment` in block header**: `calculate_block_commitments` also computes the Poseidon hash of the `ThinStateDiff` itself as `state_diff_commitment`. With `deprecated_declared_classes` missing, this hash is wrong, so the stored `state_diff_commitment` in the header does not match the actual state diff stored alongside it.

3. **P2P sync truncation**: A syncing node that reads the wrong `state_diff_length` from the header will accept a truncated state diff as complete, silently missing the deprecated class declarations. This causes the syncing node to build a different (wrong) state root — a **wrong state root** impact.

---

### Likelihood Explanation

Cairo 0 (deprecated) class declarations are rare in modern Starknet blocks (post-0.13.x) but are not impossible — any transaction that deploys or declares a Cairo 0 contract triggers this path. The bug is latent and would manifest whenever such a transaction appears in a block produced by this sequencer. The conversion is used unconditionally in the hot path of every block finalization.

---

### Recommendation

`CommitmentStateDiff` must be extended to carry `deprecated_declared_classes`, or the `BlockExecutionArtifacts` path must obtain the deprecated class list from `StateMaps::declared_contracts` (the boolean map tracking Cairo 0 declarations) and inject it into the `ThinStateDiff` before calling `calculate_block_commitments`. The `TODO(AlonH)` comment at line 764 of `cached_state.rs` already acknowledges this gap; it must be resolved before the sequencer handles any block with Cairo 0 declarations. [10](#0-9) 

---

### Proof of Concept

1. Submit a `Declare` transaction for a Cairo 0 contract class to the sequencer.
2. The blockifier executes it; `StateMaps::declared_contracts` records `{class_hash: true}`. `CommitmentStateDiff` records nothing for this class (it has no `deprecated_declared_classes` field).
3. `BlockExecutionArtifacts::new` calls `ThinStateDiff::from(commitment_state_diff.clone())` → `deprecated_declared_classes = []`.
4. `calculate_block_commitments` receives a `ThinStateDiff` with `deprecated_declared_classes.len() == 0`.
5. `state_diff.len()` returns `N` instead of `N + 1` (one fewer than the true count).
6. `concat_counts(tx_count, event_count, N, da_mode)` produces a felt with the wrong `state_diff_length` field.
7. `calculate_block_hash` chains this wrong `concatenated_counts` → wrong `PartialBlockHash` → wrong `ProposalCommitment`.
8. Any independent verifier computing the block hash from the actual on-chain state diff (which does include the deprecated class) will compute a different hash, causing a commitment mismatch.
9. A P2P-syncing peer reads `state_diff_length = N` from the header, stops after receiving `N` state diff chunks, and never receives the `DeprecatedDeclaredClass` chunk — silently accepting a wrong state.

```rust
pub fn concat_counts(
    transaction_count: usize,
    event_count: usize,
    state_diff_length: usize,
    l1_data_availability_mode: L1DataAvailabilityMode,
) -> Felt {
    let l1_data_availability_byte: u8 = match l1_data_availability_mode {
        L1DataAvailabilityMode::Calldata => 0,
        L1DataAvailabilityMode::Blob => 0b10000000,
    };
    let concat_bytes = [
        to_64_bits(transaction_count).as_slice(),
        to_64_bits(event_count).as_slice(),
        to_64_bits(state_diff_length).as_slice(),
        &[l1_data_availability_byte],
        &[0_u8; 7], // zero padding
    ]
    .concat();
    Felt::from_bytes_be_slice(concat_bytes.as_slice())
}
