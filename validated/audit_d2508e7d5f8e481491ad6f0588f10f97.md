### Title
`ProofOfInclusion::valid()` Does Not Verify Against Any Committed Root Hash, Enabling Forged Inclusion Proofs — (`crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

---

### Summary

The `ProofOfInclusion::valid()` method in the DataLayer crate only checks internal hash-chain consistency. Its final equality check is tautological: it compares `existing_hash` against `self.root_hash()`, but `root_hash()` returns the very same `combined_hash` field that the loop just verified. No externally committed root is ever consulted. An attacker who can deliver a crafted `ProofOfInclusion` to any consumer of this API can claim arbitrary key-value inclusion in any DataLayer store.

---

### Finding Description

`ProofOfInclusion` is a `Streamable` type exposed through the Python wheel (`PyStreamable`, `PyJsonDict`). Its `valid()` method is the sole public verification entry point:

```rust
// crates/chia-datalayer/src/merkle/proof_of_inclusion.rs  lines 40-58
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()   // ← tautological
}
```

After the loop, `existing_hash` equals the last `layer.combined_hash` (the loop would have returned `false` otherwise). `root_hash()` is:

```rust
// lines 32-38
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // ← same field
    } else {
        self.node_hash
    }
}
```

So the final comparison `existing_hash == self.root_hash()` is always `true` when the loop completes. The method never accepts a trusted external root to compare against. It only verifies that the proof's own fields are internally self-consistent — not that the proof corresponds to any real committed tree.

The analog to the external report is direct: just as `Balancer._sendToken` decoded and forwarded caller-supplied pool IDs without checking them against the stored `connectedOFTs` values, `ProofOfInclusion::valid()` accepts and verifies caller-supplied hash fields without checking them against any stored/committed tree root.

**Attack path:**

1. Attacker constructs a `ProofOfInclusion` with an arbitrary `node_hash` (claiming any key-value pair is "included").
2. Attacker builds `layers` whose `combined_hash` fields form a self-consistent chain (trivially computable).
3. Attacker serializes the struct (it is `Streamable`) and delivers it to a DataLayer client over the network.
4. Client calls `proof.valid()` → returns `true`.
5. Client accepts the forged inclusion claim; the actual committed root is never consulted.

The `ProofOfInclusion` type is fully deserializable from untrusted bytes via the `Streamable` trait and is exposed to Python via `PyStreamable`/`PyJsonDict`, making the delivery path straightforward.

---

### Impact Explanation

**High.** DataLayer Merkle proof logic accepts a forged inclusion proof, letting untrusted input prove invalid state. Any DataLayer client that calls `proof.valid()` as its sole verification step — a natural reading of the API — can be convinced that an arbitrary key-value pair is present in a store whose actual root is completely different. This undermines the integrity guarantee of the DataLayer's authenticated key-value store.

---

### Likelihood Explanation

The `ProofOfInclusion` struct is a `Streamable` + `PyStreamable` type, meaning it is routinely serialized and exchanged between DataLayer peers. The method name `valid()` strongly implies complete proof verification. Any peer that receives a proof from an untrusted counterpart and calls `valid()` without separately extracting `proof.root_hash()` and comparing it against the locally committed root is vulnerable. The Python binding `py_valid()` makes this equally reachable from Python DataLayer clients.

---

### Recommendation

`valid()` must accept a trusted root parameter and compare the computed root against it:

```rust
pub fn valid(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root   // compare against externally committed root
}
```

Alternatively, rename the current method to `is_internally_consistent()` and add a separate `verify(root: &Hash) -> bool` that performs the full check, making the missing root comparison impossible to overlook.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side};
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};

// Attacker picks any fake node hash (claims this key is "included")
let fake_node_hash = Hash([0xAA; 32]);

// Build a single-layer proof with a self-consistent combined_hash
// (attacker computes calculate_internal_hash(fake_node_hash, Side::Right, other_hash))
let other_hash = Hash([0xBB; 32]);
let combined_hash = calculate_internal_hash(&fake_node_hash, Side::Right, &other_hash);

let forged_proof = ProofOfInclusion {
    node_hash: fake_node_hash,
    layers: vec![ProofOfInclusionLayer {
        other_hash_side: Side::Right,
        other_hash,
        combined_hash,   // self-consistent; root_hash() returns this
    }],
};

// valid() returns true even though this proof has nothing to do with the real tree
assert!(forged_proof.valid());
// forged_proof.root_hash() == combined_hash (attacker-chosen), not the real store root
