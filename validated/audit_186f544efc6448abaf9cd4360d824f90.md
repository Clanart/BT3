Audit Report

## Title
`_fillBids` subtracts the wrong bin's `lengthE6` when walking downward, corrupting `lowerX64`/`upperX64` for all bid bins below the active bin in heterogeneous-length pools — (`metric-periphery/contracts/lens/MetricOmmPoolDataProvider.sol`)

## Summary
In `_fillBids`, the downward walk subtracts `lenAbove` (the length of bin `b+1`) from `walkDistE6` to compute the lower boundary of bin `b`, but `MetricOmmPool`'s own downward traversal subtracts the **new (lower) bin's** `lengthE6`. The two are equal only when all bins share the same length. For any pool with heterogeneous bin lengths — a fully supported factory configuration — every bid bin below `curBinIdx` receives wrong `lowerX64`/`upperX64`, corrupting `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` in the returned depth ladder.

## Finding Description
`_fillBids` initialises `walkDistE6 = curBinDistFromProvidedPriceE6` (the lower boundary of `curBinIdx`) and then for each bin `b < curBinIdx`:

```solidity
// MetricOmmPoolDataProvider.sol L496-498
(,, uint16 lenAbove,,) = PoolStateLibrary._binState(pool, int8(b + 1));
walkDistE6 -= int256(uint256(lenAbove));
``` [1](#0-0) 

`walkDistE6` is then used as the lower boundary of bin `b`, and `walkDistE6 + lengthE6[b]` as the upper boundary:

```solidity
// MetricOmmPoolDataProvider.sol L503-506
uint256 lowerX64 = _priceFromMidAndDistE6(midPriceX64, walkDistE6, Math.Rounding.Floor);
uint256 upperX64 =
  _priceFromMidAndDistE6(midPriceX64, walkDistE6 + int256(uint256(lengthE6)), Math.Rounding.Floor);
``` [2](#0-1) 

The pool itself, when traversing downward, decrements the bin index, loads the **new** (lower) bin's state, and subtracts **that bin's** `lengthE6`:

```solidity
// MetricOmmPool.sol L1197-1200
curBinIdxCache--;
binState = _binStates[curBinIdxCache];
curPosInBinCache = type(uint104).max;
curBinDistE6Cache -= int24(uint24(binState.lengthE6));
``` [3](#0-2) 

The lens subtracts `lengthE6[b+1]` (the bin above) but the pool subtracts `lengthE6[b]` (the bin being entered). With `curBinIdx=0`, `lengthE6[0]=10_000`, `lengthE6[-1]=20_000`:
- Lens: `walkDistE6 = 0 − 10_000 = −10_000` → `lowerX64 = price(−10_000)`, `upperX64 = price(10_000)`
- Pool: `curBinDistE6Cache = 0 − 20_000 = −20_000` → lower = `price(−20_000)`, upper = `price(0)`

Both boundaries are wrong, and the error accumulates for every subsequent bin below. The factory imposes no constraint requiring equal bin lengths: [4](#0-3) 

## Impact Explanation
`getLiquidityDepth` returns a `bids` array whose `binAvgExecPriceX64` and `cumulativeAvgExecPriceX64` are computed from wrong price bounds for every bin below `curBinIdx`. An integrator using this depth ladder to size a sell-token0 swap receives a predicted execution price that diverges from what the pool will actually execute. This meets the lens/quoter impact gate: integrations are induced into fund-losing trades when adjacent bin lengths differ materially (e.g., a 2× difference shifts the reported price range by the full width of one bin), constituting bad-price execution above Sherlock Medium thresholds.

## Likelihood Explanation
Any pool with at least two adjacent bid bins of different `lengthE6` triggers the bug. The factory accepts per-bin `lengthE6` values with no requirement for uniformity; heterogeneous lengths are the expected use case for pools wanting tighter pricing near the mid and wider bins further out. The bug is triggered by a normal, unprivileged `getLiquidityDepth` call with no special setup.

## Recommendation
Replace the `lenAbove` read with the current bin's own `lengthE6`, which is already fetched on the next line, matching the pool's own downward traversal:

```solidity
// After (correct):
(, uint104 t1, uint16 lengthE6,, uint16 addFeeSellE6) = PoolStateLibrary._binState(pool, binIdx);
walkDistE6 -= int256(uint256(lengthE6));
```

This eliminates the extra `_binState` call for `b+1` and aligns the lens geometry with `MetricOmmPool`'s downward traversal.

## Proof of Concept
1. Deploy a pool with `curBinDistFromProvidedPriceE6 = 0`, bin 0 `lengthE6 = 10_000`, bin −1 `lengthE6 = 20_000`, both with token1 liquidity.
2. Call `getLiquidityDepth(pool, 2)`.
3. Observe `bids[1]` (bin −1) has `lowerX64 = price(−10_000)` and `upperX64 = price(10_000)`.
4. Correct values: `lowerX64 = price(−20_000)`, `upperX64 = price(0)`.
5. Execute a sell swap that exhausts bin −1 and compare the actual execution price against `bids[1].binAvgExecPriceX64` — they will not match.
