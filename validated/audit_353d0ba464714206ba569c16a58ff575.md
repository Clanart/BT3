### Title
`is_block_info_valid` Skips WEI Gas Price Validation, Allowing a Malicious Proposer to Embed an Arbitrary ETH/STRK Rate That Corrupts `l2_gas_price_wei` in the Block Hash and Fee Calculations - (File: `crates/apollo_consensus_orchestrator/src/validate_proposal.rs`)

### Summary

The `is_block_info_valid` function validates the proposer's L1 gas prices only in FRI denomination against the oracle, but explicitly discards the WEI prices. The WEI prices from the `ProposalInit` are accepted without any oracle-based check and are then used to derive the ETH/STRK conversion rate, which in turn determines `l2_gas_price_wei`. Because `l2_gas_price_wei` (and `l1_gas_price_wei`) are committed into the block hash via `gas_prices_to_hash`, a malicious proposer can embed an arbitrary WEI price that passes all validation, produces a wrong ETH/STRK rate, corrupts `l2_gas_price_wei`, and permanently alters the block hash and fee accounting for every ETH-denominated transaction in that block.

### Finding Description

**Root cause — the oracle assumption gap**

`is_block_info_valid` calls `get_l1_prices_in_fri_and_wei`, which returns both FRI and WEI prices from the oracle. The WEI result is immediately discarded:

```rust
let (l1_gas_prices_fri, _l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(…).await;
```

Only the FRI prices are compared against the proposer's values:

```rust
if !(within_margin(l1_gas_price_fri_proposed, l1_gas_price_fri, …)
    && within_margin(l1_data_gas_price_fri_proposed, l1_data_gas_price_fri, …))
{
    return Err(ValidateProposalError::InvalidBlockInfo(…));
}
``` [1](#0-0) 

The proposer's `l1_gas_price_wei` and `l1_data_gas_price_wei` are never compared against the oracle's WEI output.

**Downstream corruption — ETH/STRK rate derivation**

After validation passes, `initiate_validation` calls `convert_to_sn_api_block_info(init)`, which derives the ETH/STRK rate directly from the proposer's (unvalidated) WEI and FRI prices:

```rust
let eth_to_fri_rate = calculate_eth_to_fri_rate(&previous_block_info)?;
let l2_gas_price_wei = NonzeroGasPrice::new(init.l2_gas_price_fri.fri_to_wei(eth_to_fri_rate)?)?;
``` [2](#0-1) 

`calculate_eth_to_fri_rate` computes `l1_gas_price_fri * WEI_PER_ETH / l1_gas_price_wei`. A manipulated `l1_gas_price_wei` directly skews this ratio, producing a wrong `eth_to_fri_rate`, which then produces a wrong `l2_gas_price_wei`. [3](#0-2) 

**Block hash commitment**

`gas_prices_to_hash` (Starknet version ≥ 0.13.4) chains all six gas price fields — including `l1_gas_price.price_in_wei`, `l1_data_gas_price.price_in_wei`, and `l2_gas_price.price_in_wei` — into the Poseidon block hash:

```rust
.chain(&l1_gas_price.price_in_wei.0.into())
…
.chain(&l2_gas_price.price_in_wei.0.into())
.chain(&l2_gas_price.price_in_fri.0.into())
``` [4](#0-3) 

`PartialBlockHashComponents::new` populates these fields directly from `block_info.gas_prices`, which was built from the unvalidated proposer values: [5](#0-4) 

Because both the proposer and the validator execute the same `convert_to_sn_api_block_info(init)` with the same manipulated `init`, they both compute the same wrong block hash, so the `built_block != received_fin.proposal_commitment` check passes and the block is committed.

**Cascading fallback propagation**

`PreviousBlockInfo` is derived from the accepted `ProposalInit`, storing the manipulated WEI prices:

```rust
impl From<&ProposalInit> for PreviousBlockInfo {
    fn from(init: &ProposalInit) -> Self {
        Self {
            l1_prices_wei: L1PricesInWei {
                l1_gas_price: init.l1_gas_price_wei,   // manipulated
                …
            },
``` [6](#0-5) 

When the oracle is unavailable for the **next** block, `get_l1_prices_in_fri_and_wei_and_conversion_rate` falls back to `calculate_eth_to_fri_rate(block_info)` using this stored `PreviousBlockInfo`, propagating the wrong ETH/STRK rate to subsequent blocks. [7](#0-6) 

### Impact Explanation

1. **Wrong block hash committed on-chain** — `l1_gas_price_wei`, `l1_data_gas_price_wei`, and the derived `l2_gas_price_wei` are all hashed into the block hash. A manipulated WEI price produces a block hash that diverges from what an honest sequencer would produce, breaking proof verification and state sync for any system that independently recomputes the block hash.

2. **Incorrect ETH-denominated fees** — `l2_gas_price_wei` is used for all ETH-fee-type transactions. A wrong value causes users to be systematically overcharged or undercharged, constituting a direct economic impact.

3. **Cascading fallback corruption** — The wrong WEI prices are stored as `PreviousBlockInfo` and reused as the oracle fallback for the next block, potentially propagating the error across multiple blocks.

These match the Critical impact categories: *Wrong state/receipt/revert result from blockifier/syscall/execution logic* and *Incorrect fee/gas/resource accounting with economic impact*.

### Likelihood Explanation

Any BFT validator that is selected as proposer can trigger this. No external oracle manipulation or network-level attack is required — the proposer simply sets `l1_gas_price_wei` to an arbitrary value while keeping `l1_gas_price_fri` within the accepted margin. The validator nodes have the oracle data needed to check the WEI price but do not use it.

### Recommendation

In `is_block_info_valid`, stop discarding `_l1_gas_prices_wei`. Use the oracle-derived WEI prices to validate the proposed WEI prices with the same `within_margin` check already applied to FRI prices:

```rust
let (l1_gas_prices_fri, l1_gas_prices_wei) = get_l1_prices_in_fri_and_wei(…).await;
// existing FRI checks …
if !(within_margin(init_proposed.l1_gas_price_wei, l1_gas_prices_wei.l1_gas_price, margin)
    && within_margin(init_proposed.l1_data_gas_price_wei, l1_gas_prices_wei.l1_data_gas_price, margin))
{
    return Err(ValidateProposalError::InvalidBlockInfo(…));
}
```

This mirrors the existing FRI validation and closes the assumption gap.

### Proof of Concept

1. Honest oracle returns: `l1_gas_price_wei = 10 gwei`, `eth_to_fri_rate = 800 * 10^18` (1 ETH = 800 STRK), so `l1_gas_price_fri = 8000 fri`.

2. Malicious proposer sends `ProposalInit` with:
   - `l1_gas_price_fri = 8000` (within margin — passes FRI check)
   - `l1_gas_price_wei = 1` (manipulated — not checked)

3. `is_block_info_valid` passes: FRI prices are within margin.

4. `calculate_eth_to_fri_rate` computes: `8000 * 10^18 / 1 = 8000 * 10^18` (instead of `800 * 10^18`).

5. `l2_gas_price_wei = l2_gas_price_fri * 10^18 / (8000 * 10^18) = l2_gas_price_fri / 8000` — 1000× smaller than correct.

6. `gas_prices_to_hash` commits `l1_gas_price_wei = 1` and the wrong `l2_gas_price_wei` into the Poseidon block hash.

7. Both proposer and validator compute the same wrong hash; `built_block == received_fin.proposal_commitment` passes; the block is finalized with a corrupted block hash and wrong ETH-denominated fees.

pub fn gas_prices_to_hash(
