Audit Report

## Title
Router `pay()` Consumes Unattributed Native ETH Balance, Enabling Cross-Transaction Residue Theft of Stranded WETH-Input ETH — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

The `pay()` function in `PeripheryPayments.sol` reads `address(this).balance` — the router's entire native ETH balance — when settling WETH payment obligations, with no per-caller attribution. Any ETH stranded on the router from a prior transaction (due to excess `msg.value` without an atomic `refundETH`) is indistinguishable from the current caller's ETH and will be consumed first. A subsequent attacker can submit a WETH-input swap with `msg.value = 0` and have their full obligation settled from the victim's stranded ETH at zero cost.

## Finding Description

**Root cause — unscoped `address(this).balance` in `pay()`:** [1](#0-0) 

When `token == WETH` and `payer != address(this)`, `pay()` reads `address(this).balance` at line 74. This is the aggregate native balance of the router, not the current caller's `msg.value`. Any ETH left from a prior transaction is consumed before pulling from the payer's WETH allowance.

**How ETH becomes stranded:**

All swap entry points are `payable`: [2](#0-1) [3](#0-2) [4](#0-3) 

The `receive()` guard only blocks direct ETH pushes from non-WETH addresses: [5](#0-4) 

`msg.value` attached to any `payable` entry point bypasses `receive()` entirely and lands in `address(this).balance`. If the swap consumes less than `msg.value`, the surplus persists across transaction boundaries.

**Exploit call path:**

The attacker calls `exactInputSingle` with `tokenIn = WETH` and `msg.value = 0`: [6](#0-5) 

The pool calls back into `metricOmmSwapCallback`, which dispatches to `_justPayCallback`: [7](#0-6) 

`pay()` is called with `payer = attacker`. Because `address(this).balance` already holds the victim's stranded ETH, the `nativeBalance >= value` branch fires and the attacker's WETH obligation is settled entirely from the victim's ETH — the attacker's wallet is not touched.

**`refundETH` is also exploitable directly:** [8](#0-7) 

`refundETH()` is `external` with no access control and sends the entire `address(this).balance` to `msg.sender`. Any attacker can call it directly to steal stranded ETH without even needing a swap — this is a simpler and equally valid theft path that the claim does not fully enumerate.

**Existing guards are insufficient:** `_requireExpectedCallbackCaller` at line 49 only verifies the callback originates from the expected pool; it does not verify that the payer has contributed any ETH to the current transaction.

## Impact Explanation

Direct loss of user principal. The victim loses up to `msg.value − amountIn` ETH per transaction. The attacker receives an equivalent WETH-funded swap at zero cost (swap path) or the raw ETH directly (via `refundETH`). There is no cap other than the victim's stranded balance. This is a Critical/High direct loss of user principal meeting Sherlock thresholds.

## Likelihood Explanation

Medium. The precondition — ETH stranded on the router — arises whenever a user sends `msg.value > amountIn` to a WETH-input swap without atomically including `refundETH` in the same `multicall`, or submits `refundETH` as a separate follow-up transaction. Both patterns are common (users over-provision ETH to avoid reverts). An attacker needs only to monitor `address(router).balance` in the mempool and front-run the victim's `refundETH` with either a WETH-input swap or a direct `refundETH` call.

## Recommendation

1. **Atomic refund enforcement**: Require that every WETH-input swap with `msg.value` includes `refundETH` as the final step of the same `multicall`. Document this as a mandatory integration invariant.
2. **Scoped native balance tracking**: Track the current call's `msg.value` contribution separately (e.g., pass it as a parameter or store it in transient storage at entry) and limit `pay()`'s native ETH consumption to that scoped amount, reverting or pulling from the payer's WETH allowance for any excess.
3. **Guard in `pay()`**: When `payer != address(this)` and `token == WETH`, cap native ETH consumed to `min(nativeBalance, value)` and always pull the remainder from `payer` via `safeTransferFrom`, never silently consuming unattributed router balance.
4. **Access-control `refundETH`**: Restrict `refundETH` to return ETH only to the original `msg.sender` of the enclosing `multicall`, or track per-user ETH contributions in transient storage.

## Proof of Concept

```
Block N:
  Victim calls exactInputSingle{value: 2000}(
      pool=WETH/TOKEN pool, tokenIn=WETH, amountIn=1000, ...
  )
  → pay() fires: nativeBalance=2000 >= value=1000
    → deposits 1000 ETH as WETH, transfers to pool ✓
    → 1000 ETH remains: address(router).balance = 1000
  Victim does NOT include refundETH in this call.

Block N (same or next):
  Attacker observes address(router).balance = 1000 ETH.

  Attack path A (swap-based):
    Attacker calls exactInputSingle{value: 0}(
        pool=WETH/TOKEN pool, tokenIn=WETH, amountIn=1000, ...
    )
    → _setNextCallbackContext(pool, JUST_PAY, attacker, WETH)
    → pool.swap() → metricOmmSwapCallback → _justPayCallback
    → pay(WETH, attacker, pool, 1000)
    → nativeBalance=1000 >= value=1000
    → deposits 1000 ETH (victim's) as WETH, transfers to pool
    → attacker receives TOKEN output, spent 0 ETH/WETH ✓

  Attack path B (direct theft):
    Attacker calls refundETH()
    → address(router).balance = 1000 → _transferETH(attacker, 1000)
    → attacker receives 1000 ETH directly ✓

  Victim calls refundETH:
  → address(router).balance = 0 → victim receives 0 ETH
  → victim net loss: 1000 ETH
```
