### Title
Silent `ft_transfer` Failure in `fin_transfer_send_tokens_callback` Permanently Freezes User Funds and Corrupts Locked-Token Accounting — (File: `near/omni-bridge/src/lib.rs`)

---

### Summary

`fin_transfer_send_tokens_callback` in the NEAR bridge contract never checks whether the underlying `ft_transfer` promise succeeded when `is_ft_transfer_call` is `false` (i.e., when the transfer carries no message). If the `ft_transfer` cross-contract call panics — due to a paused token, a blacklisted recipient, or any other token-level restriction — the callback silently treats the outcome as a success. The transfer is permanently marked as finalized, locked-token accounting is decremented, and the fee is paid out, but the recipient never receives their tokens. The funds are irrecoverably locked in the bridge.

---

### Finding Description

`fin_transfer_send_tokens_callback` delegates the success/failure decision to `is_refund_required`:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    if is_ft_transfer_call {
        match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
            Ok(value) => { ... }
            Err(_) => false,   // ← also a bug: ft_transfer_call panic is ignored
        }
    } else {
        false   // ← BUG: ft_transfer failure is never checked
    }
}
``` [1](#0-0) 

When `msg` is empty, `send_tokens` issues a plain `ft_transfer` and the callback is invoked with `is_ft_transfer_call = false`. The branch `else { false }` returns unconditionally without ever calling `env::promise_result_checked`. A failed (panicked) `ft_transfer` promise is therefore indistinguishable from a successful one.

The state mutations that precede `send_tokens` are:

1. `add_fin_transfer` — permanently records the transfer as finalized (replay-prevention).
2. `unlock_tokens_if_needed` — decrements the locked-token counter for the origin chain. [2](#0-1) 

Both mutations are committed before `send_tokens` is called: [3](#0-2) 

Because `is_refund_required` returns `false`, the callback's refund branch — which calls `revert_lock_actions` and `remove_fin_transfer` — is never entered: [4](#0-3) 

Instead, the callback proceeds to the "success" path: it pays the fee to the fee recipient and emits `FinTransferEvent`, even though the recipient received nothing. [5](#0-4) 

A secondary instance of the same class exists in `process_fin_transfer_to_other_chain`, where `send_tokens` is called with `.detach()` — no callback at all — when reimbursing a fast-transfer relayer. A failed transfer there is completely undetected. [6](#0-5) 

---

### Impact Explanation

When `ft_transfer` fails silently:

- The transfer nonce is permanently consumed — the same proof can never be re-submitted to retry delivery.
- The locked-token counter is decremented even though no tokens left the bridge — bridge collateralization accounting is corrupted.
- The recipient's tokens remain in the bridge contract with no mechanism to recover them.
- The fee is paid to the relayer from the bridge's own balance, compounding the accounting error.

This constitutes **permanent freezing / irrecoverable lock of user funds** and **accounting corruption that breaks bridge collateralization**, both of which are in-scope critical/high impacts.

---

### Likelihood Explanation

The scenario requires the `ft_transfer` cross-contract call to panic after the proof has been verified. Realistic triggers include:

- **Token contract paused** — USDC, USDT, and similar regulated tokens have pause mechanisms that have been activated in production (e.g., USDC during the SVB crisis).
- **Recipient blacklisted** — USDC and other compliant tokens can blacklist addresses; a recipient blacklisted between transfer initiation and finalization triggers this path.
- **Insufficient storage deposit** — if the storage-deposit check passes but the token contract's internal accounting rejects the transfer for any reason.

None of these require privileged access; they are ordinary token-contract behaviors that any user's transfer can encounter.

---

### Recommendation

`is_refund_required` must check the promise result unconditionally, regardless of `is_ft_transfer_call`:

```rust
fn is_refund_required(is_ft_transfer_call: bool) -> bool {
    match env::promise_result_checked(0, MAX_FT_TRANSFER_CALL_RESULT) {
        Err(_) => true,  // ft_transfer or ft_transfer_call panicked → always refund
        Ok(value) if is_ft_transfer_call => {
            near_sdk::serde_json::from_slice::<U128>(&value)
                .map(|a| a.0 == 0)
                .unwrap_or(false)
        }
        Ok(_) => false,  // plain ft_transfer succeeded
    }
}
```

This ensures that any failure of the underlying token transfer — whether a panic or a zero-used-amount return — triggers the refund path that reverts `lock_actions` and removes the finalized-transfer record.

For the `.detach()` call in `process_fin_transfer_to_other_chain`, a proper callback should be added to handle relayer reimbursement failures.

---

### Proof of Concept

1. User bridges 1000 USDC from Ethereum to NEAR (recipient = `alice.near`).
2. Relayer calls `fin_transfer` on the NEAR bridge with a valid proof.
3. `fin_transfer_callback` → `process_fin_transfer_to_near`:
   - `add_fin_transfer(transfer_id)` — nonce consumed.
   - `unlock_tokens_if_needed(Eth, usdc.near, 1000)` — locked balance decremented.
   - `send_tokens(usdc.near, alice.near, 1000, "")` — issues `ft_transfer`.
4. Between step 3 and execution of `ft_transfer`, USDC contract is paused (or `alice.near` is blacklisted).
5. `ft_transfer` panics; NEAR runtime calls `fin_transfer_send_tokens_callback` with a failed promise result.
6. `is_refund_required(false)` returns `false` without inspecting the promise result.
7. Callback emits `FinTransferEvent` and pays fee — bridge believes transfer succeeded.
8. `alice.near` has 0 USDC. The 1000 USDC remain in the bridge but the locked-token counter says 0. The transfer ID is permanently finalized; re-submission is rejected with `NonceAlreadyUsed`. Funds are irrecoverably locked.
