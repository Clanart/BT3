### Title
Proof Verification Bypassed for Any Transaction Reusing a Previously Stored `ProofFacts` Hash — (File: `crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary

`run_proof_verification` skips cryptographic proof verification whenever `contains_proof(proof_facts)` returns `true`. Because `ProofFacts` contains only block-level data (`program_hash`, `block_number`, `block_hash`, `config_hash`) and no transaction-specific binding, any attacker who knows that a proof for a given block has already been stored can submit a new transaction with the same `proof_facts` and a completely invalid `proof` and have it accepted by the gateway/consensus without any verification.

### Finding Description

In `run_proof_verification`:

```rust

if contains_proof {
    return Ok(false);   // ← verification entirely skipped

``` [1](#0-0) 

The `ProofManager` stores proofs keyed by `proof_facts.hash()`, a Poseidon hash of the proof-facts vector:

```rust

`SnosProofFacts` — the only supported variant — contains exclusively block-level fields:

There is no `sender_address`, `calldata_hash`, or `transaction_hash` in `ProofFacts`. The production block data confirms this: multiple transactions from different senders with different calldatas all carry identical `proof_facts` arrays pointing to the same block. [4](#0-3) 

The `proof` field is explicitly **dropped** when converting from the RPC type to the internal type, so it is never committed to in the transaction hash:

```rust

``` [5](#0-4)

### Impact Explanation

This matches the allowed scope: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

Client-side proving transactions carry zero `max_price_per_unit` on all resource bounds (the prover README explicitly requires this). Once a legitimate user has submitted a transaction for block N and its proof has been stored, any attacker can submit a new transaction with the same `proof_facts` (same block N) and a garbage `proof` byte string. The gateway and consensus converter call `run_proof_verification`, find `contains_proof = true`, skip verification, and admit the transaction as if it were proven. The attacker obtains zero-fee transaction execution without having produced a valid proof.

### Likelihood Explanation

`proof_facts` values are public — they appear verbatim in block data returned by the feeder gateway and RPC. Any observer can extract a stored `proof_facts` from any previously accepted client-side-proving transaction and immediately reuse it. No privileged access, no key material, and no interaction with the original prover are required.

### Recommendation

Key the proof-existence check on `(proof_facts_hash, transaction_hash)` rather than `proof_facts_hash` alone. Because the transaction hash already commits to `proof_facts` (via `get_invoke_transaction_v3_hash`), the simplest fix is to store and look up proofs under the transaction hash, or to include the transaction hash inside `ProofFacts` before hashing. This mirrors the fix applied in the original NFT report: adding an ownership check at the point of reservation, not only at the point of first lock.

### Proof of Concept

1. **Setup.** User A submits `Tx_A` with `proof_facts = P` (pointing to block N) and a valid `proof = Q`. `run_proof_verification` calls `starknet_proof_verifier::verify_proof(P, Q)`, succeeds, and stores the proof under `hash(P)` in `ProofManager`.

2. **Attack.** User B constructs `Tx_B` with the same `proof_facts = P` (copied from public block data), a different `sender_address`, `calldata`, and `nonce`, and `proof = vec![0u8; 1]` (arbitrary garbage). `Tx_B` has a distinct transaction hash because `sender_address`/`calldata`/`nonce` differ.

3. **Bypass.** The gateway calls `convert_rpc_tx_to_internal_rpc_tx` → `spawn_proof_verification` → `run_proof_verification`. `contains_proof(P)` returns `true` (the cache/filesystem entry from step 1 exists). The function returns `Ok(false)` without calling `verify_proof`.

4. **Admission.** `Tx_B` passes all gateway checks and enters the mempool with zero fees and an unverified proof. The proof field is dropped before the transaction hash is computed, so no downstream component detects the mismatch. [7](#0-6) [8](#0-7) [9](#0-8)

    /// Runs proof verification: checks if the proof already exists, and if not, verifies it.
    /// Returns `true` if verification was performed, `false` if skipped (proof already stored).
    /// This is the shared verification logic used by both gateway and consensus flows.
    async fn run_proof_verification(
        proof_facts: ProofFacts,
        proof: Proof,
        proof_manager_client: SharedProofManagerClient,
    ) -> Result<bool, TransactionConverterError> {
        let contains_proof = proof_manager_client.contains_proof(proof_facts.clone()).await?;

        if contains_proof {
            return Ok(false);
        }

        let proof_facts_hash = proof_facts.hash();
        let verify_start = Instant::now();
        tokio::task::spawn_blocking(move || {
            starknet_proof_verifier::verify_proof(proof_facts, proof)
        })
        .await
        .expect("proof verification task panicked")?;
        let verify_duration = verify_start.elapsed();
        PROOF_VERIFICATION_LATENCY.record(verify_duration.as_secs_f64());
        info!(
            "Proof verification took: {verify_duration:?} for proof facts hash: \
             {proof_facts_hash:?}"
        );

        Ok(true)
    }
