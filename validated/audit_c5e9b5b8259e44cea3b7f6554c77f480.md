### Title
DataLayer `ProofOfInclusion::valid()` Does Not Verify Against a Trusted Root Hash, Enabling Forged Inclusion Proofs — (`File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary
`ProofOfInclusion::valid()` only checks the internal hash-chain consistency of the proof. It never compares the computed root against any externally-supplied, trusted root hash. An attacker can craft an arbitrary, internally-consistent `ProofOfInclusion` for any `node_hash` they choose, serialize it, and have it accepted as valid by any verifier that calls only `proof.valid()`.

### Finding Description
`ProofOfInclusion::valid()` is implemented as follows:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()   // ← always true if loop passes
}
``` [1](#0-0) 

`root_hash()` returns `last.combined_hash`:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash
    } else {
        self.node_hash
    }
}
``` [2](#0-1) 

After the loop, `existing_hash` is always equal to the last `layer.combined_hash` (because the loop sets `existing_hash = calculated_hash` only after asserting `calculated_hash == layer.combined_hash`). Therefore the final guard `existing_hash == self.root_hash()` is **unconditionally true** whenever the loop completes without returning `false`. The method never compares against any externally-known, trusted root hash.

`ProofOfInclusion` is a `Streamable` type exposed through Python bindings with `from_bytes()` / `from_bytes_unchecked()`: [3](#0-2) [4](#0-3) 

An attacker can therefore:
1. Choose any `node_hash` (e.g., the hash of a key-value pair they want to falsely claim is present).
2. Choose any `other_hash` and compute `combined_hash = calculate_internal_hash(node_hash, side, other_hash)`.
3. Serialize the resulting `ProofOfInclusion` with `to_bytes()`.
4. Deliver it to a verifier.

Any verifier that calls only `proof.valid()` — without separately asserting `proof.root_hash() == <on-chain-trusted-root>` — will accept the forged proof.

### Impact Explanation
This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion … or lets untrusted input prove invalid state."**

A DataLayer consumer that relies on `proof.valid()` alone to gate access decisions (e.g., "is this key present in the store?") can be made to accept a proof for any key-value pair in any fabricated tree. This allows an unprivileged attacker to forge DataLayer state proofs, bypassing inclusion checks and potentially corrupting application-layer state that depends on DataLayer integrity.

### Likelihood Explanation
The `valid()` method's name strongly implies complete proof validation. The Python binding exposes it as the primary verification entry point alongside `root_hash()` as a separate accessor, making it easy for callers to omit the root-hash comparison step. The fuzz target itself calls `proof.valid()` without checking `root_hash()`: [5](#0-4) 

Any DataLayer consumer that follows this pattern in production code is vulnerable.

### Recommendation
`valid()` should accept a trusted root hash as a parameter and compare against it:

```rust
pub fn valid_against_root(&self, trusted_root: &Hash) -> bool {
    // ... existing chain check ...
    existing_hash == *trusted_root
}
```

Alternatively, rename the current method to `is_internally_consistent()` and add a separate `valid_for_root(trusted_root: &Hash) -> bool` that performs the full check. The Python binding should expose only the root-checking variant as the primary API.

### Proof of Concept
```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer, Side

# Attacker-chosen node hash (e.g., hash of a key-value pair not in the real tree)
fake_node_hash = bytes([0xAA] * 32)
other_hash     = bytes([0xBB] * 32)

# Compute a valid combined_hash for this fake pair
# (using the same calculate_internal_hash logic the library uses)
import hashlib
combined = hashlib.sha256(b'\x01' + fake_node_hash + other_hash).digest()

layer = ProofOfInclusionLayer(
    other_hash_side=0,          # Side.Left
    other_hash=other_hash,
    combined_hash=combined,
)
forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[layer])

# valid() returns True even though this proof was never generated from any real tree
assert forged_proof.valid()                  # passes
assert forged_proof.root_hash() == combined  # attacker-controlled root
```

The forged proof passes `valid()` because the internal chain is consistent. A verifier that does not additionally assert `forged_proof.root_hash() == <on-chain-root>` will accept it as genuine.
