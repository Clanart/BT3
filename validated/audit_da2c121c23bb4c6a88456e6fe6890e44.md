### Title
Permanent Fund Freeze When EVM Recipient Is Blacklisted by Native ERC20 Token in `finTransfer` — (`File: evm/src/omni-bridge/contracts/OmniBridge.sol`)

### Summary
When a user initiates a NEAR→EVM transfer of a native ERC20 token with a blacklist (e.g., USDC, USDT), and the designated EVM recipient address is blacklisted by that token contract, the `finTransfer` call on EVM will permanently revert. Because the recipient is encoded in the MPC-signed payload and the NEAR side has already burned or locked the tokens, there is no protocol-level recovery path. The funds are irrecoverably frozen.

### Finding Description
In `OmniBridge.sol`, the `finTransfer` function handles the NEAR→EVM settlement path. For native ERC20 tokens (tokens not deployed by the bridge, not custom minters, not ERC1155), the function reaches the `else` branch:

```solidity
} else {
    IERC20(payload.tokenAddress).safeTransfer(
        payload.recipient,
        payload.amount
    );
}
``` [1](#0-0) 

If `payload.recipient` is blacklisted by the token contract (e.g., Circle's USDC blacklist), `safeTransfer` reverts. The entire transaction reverts, so `completedTransfers[payload.destinationNonce]` is not permanently set. However, the NEAR side has already:

1. Burned the tokens (for deployed/bridge tokens) via `burn_tokens_if_needed`, or
2. Locked the tokens in `locked_tokens` accounting (for native NEAR tokens) via `lock_tokens_if_needed` [2](#0-1) 

The MPC signature covers the exact `recipient` address:

```solidity
Borsh.encodeAddress(payload.recipient),
``` [3](#0-2) 

This means the signed payload is bound to the blacklisted recipient. No protocol mechanism exists to:
- Obtain a new MPC signature redirecting to a different recipient
- Cancel the pending transfer on NEAR and recover the burned/locked tokens
- Rescue the tokens from the EVM bridge contract

The `pending_transfers` map on NEAR retains the entry, but the tokens are already consumed. There is no `cancel_transfer` or emergency recovery function in the NEAR bridge contract. [4](#0-3) 

The same structural issue exists in the StarkNet bridge's `fin_transfer`:

```cairo
let success = IERC20Dispatcher { contract_address: payload.token_address }
    .transfer(payload.recipient, payload.amount.into());
assert(success, 'ERR_TRANSFER_FAILED');
``` [5](#0-4) 

### Impact Explanation
**Critical — Permanent freezing / irrecoverable lock of user funds.**

A user who initiates a NEAR→EVM transfer of a blacklistable token (USDC, USDT, etc.) where the EVM recipient address is subsequently blacklisted will have their tokens permanently frozen:
- Burned on NEAR (unrecoverable from the NEAR side)
- Undeliverable on EVM (every `finTransfer` attempt reverts)
- No admin or emergency path exists to redirect or recover the funds

The tokens held in the EVM `OmniBridge` contract for that transfer are also permanently stranded, as there is no `rescueTokens` or equivalent function. [6](#0-5) 

### Likelihood Explanation
**Medium.** USDC and USDT are among the most commonly bridged tokens and both implement address blacklists. A user's EVM address can be blacklisted by Circle or Tether at any time after a transfer is initiated on NEAR but before the relayer finalizes it on EVM. The window between `init_transfer` on NEAR and `finTransfer` on EVM (MPC signing latency + relayer delay) can be minutes to hours, providing a realistic opportunity for this state to arise. The user does not need to be malicious — regulatory action against an address is sufficient.

### Recommendation
1. **Add a recipient-redirect mechanism**: Allow the MPC to sign a "redirect" payload that maps an original `(originChain, originNonce)` to a new recipient, callable only after a failed `finTransfer` attempt is proven on-chain.
2. **Add an emergency rescue function**: Implement an admin-gated `rescueToken(address token, address to, uint256 amount)` on `OmniBridge` (protected by a multisig with timelock) to recover tokens stranded by permanent transfer failures.
3. **NEAR-side cancellation**: Implement a `cancel_transfer` function on the NEAR bridge that can be invoked after a configurable timeout if the destination-chain finalization has not been confirmed, returning burned/locked tokens to the original sender.

### Proof of Concept
1. User holds USDC on NEAR (bridged representation). User calls `ft_transfer_call` on the NEAR USDC token contract, targeting the NEAR bridge with `InitTransferMsg { recipient: OmniAddress::Eth(0xVictim), ... }`.
2. NEAR bridge burns the USDC tokens and records the pending transfer. [7](#0-6) 
3. MPC signs the `TransferMessagePayload` binding `recipient = 0xVictim`.
4. Before the relayer submits `finTransfer`, Circle blacklists `0xVictim`.
5. Relayer calls `OmniBridge.finTransfer(sig, payload)`. Execution reaches `IERC20(usdc).safeTransfer(0xVictim, amount)` which reverts. [8](#0-7) 
6. Transaction reverts. Nonce is not consumed. Relayer retries — same result indefinitely.
7. USDC tokens are burned on NEAR (gone). USDC tokens in the EVM bridge are undeliverable. No recovery path exists. Funds are permanently frozen.

    fn init_transfer_internal(
        &mut self,
        transfer_message: TransferMessage,
        storage_owner: AccountId,
    ) -> U128 {
        let required_storage_balance = self
            .add_transfer_message(transfer_message.clone(), storage_owner.clone())
            .saturating_add(NearToken::from_yoctonear(transfer_message.fee.native_fee.0));

        if self
            .try_update_storage_balance(
                storage_owner,
                required_storage_balance,
                NearToken::from_yoctonear(0),
            )
            .is_err()
        {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        if let OmniAddress::Near(token_id) = transfer_message.token.clone() {
            self.burn_tokens_if_needed(token_id.clone(), transfer_message.amount);

            self.lock_tokens_if_needed(
                transfer_message.get_destination_chain(),
                &token_id,
                transfer_message.amount.0,
            );
        } else {
            self.remove_transfer_message_without_refund(transfer_message.get_transfer_id());
            return transfer_message.amount;
        }

        env::log_str(&OmniBridgeEvent::InitTransferEvent { transfer_message }.to_log_string());
        U128(0)
    }
