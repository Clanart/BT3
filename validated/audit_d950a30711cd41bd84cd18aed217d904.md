### Title
Unvalidated `global_roots` across chunked RPC storage proof responses silently corrupts Patricia trie commitment inputs - (File: crates/starknet_transaction_prover/src/running/storage_proofs.rs)

### Summary
When a storage proof query exceeds the 100-key per-request limit and is split into multiple chunks, `merge_storage_proofs` unconditionally takes the `global_roots` (containing `contracts_tree_root`, `classes_tree_root`, and `block_hash`) from only the **first** chunk's response without verifying that all subsequent chunks returned the same roots. A race condition between sequential chunk requests — or a misbehaving RPC provider — can cause the merged proof to carry a `global_roots` that is inconsistent with the Merkle proof nodes contributed by the other chunks. This inconsistency is silently accepted and propagates directly into the Patricia trie `CommitmentInfo` structs that are fed to the virtual OS as authoritative proof inputs.

### Finding Description

In `merge_storage_proofs`, line 125:

```rust
let global_roots = proofs[0].global_roots.clone();
```

The `global_roots` from every chunk after the first is silently discarded. No equality check is performed against `proofs[1..]`. [1](#0-0) 

The merged `RpcStorageProof` is then consumed by `create_commitment_infos_with_state_changes`, which extracts the initial Patricia trie roots directly from `rpc_proof.global_roots`:

```rust
let contracts_trie_root_hash = HashOutput(rpc_proof.global_roots.contracts_tree_root);
let classes_trie_root_hash = HashOutput(rpc_proof.global_roots.classes_tree_root);
``` [2](#0-1) 

These values become the `previous_root` fields in the `CommitmentInfo` structs for the contracts trie and classes trie. Those structs are assembled into `VirtualOsBlockInput` and passed to the OS as `contract_state_commitment_info` and `contract_class_commitment_info`: [3](#0-2) 

If the chain advances between the first and a later chunk request — a realistic race condition on a live node, since chunks are fetched sequentially (the `TODO` comment in `fetch_proofs` notes parallel fetching is not yet implemented) — the later chunks return `global_roots` for block N+1 while the first chunk returned roots for block N. The merged proof then contains:

- `global_roots` anchored to block N (from chunk 0)
- Merkle proof nodes from both block N (chunk 0) and block N+1 (later chunks)

`create_facts_db_from_storage_proof` inserts all of these nodes into the `FactsDb` without distinguishing which block they belong to: [4](#0-3) 

The committer then applies the state diff on top of the block-N root using a `FactsDb` that is partially populated with block-N+1 nodes, producing a `CommitmentInfo` with a wrong `previous_root` and a wrong `updated_root`. The `StateCommitmentInfos` returned by `create_commitment_infos` uses these corrupted roots: [5](#0-4) 

This is the direct analog of the SVFHook bug: just as `addLiquidity` accepted a caller-supplied `key` without asserting it belonged to the SVFHook contract, `merge_storage_proofs` accepts the first chunk's `global_roots` without asserting it matches the roots returned by all other chunks.

### Impact Explanation

The corrupted `CommitmentInfo` is the direct input to the virtual OS proof. The OS uses `previous_root` to anchor the Patricia trie verification. A wrong `previous_root` causes the OS to either:

1. Fail to generate a valid proof (denial of service for the transaction prover), or
2. In the case where a misbehaving RPC provider returns self-consistent Merkle nodes for a different block alongside a first-chunk `global_roots` for that same different block, generate a proof that attests to a wrong state transition — a wrong `initial_root` and `final_root` in the OS output, which flows into the block hash and ultimately into L1 settlement.

This matches **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value** (the transaction prover produces an authoritative-looking but wrong proof input), and potentially **Critical — Wrong state … from blockifier/syscall/execution logic for accepted input** if the wrong proof is accepted by the verifier.

### Likelihood Explanation

The chunking path is triggered whenever the total key count (class hashes + contract addresses + storage keys) exceeds 100. On a busy mainnet block with many contracts, this threshold is easily exceeded. The race condition between sequential chunk fetches is a realistic production scenario: a new block can be finalized between requests on a live node. The `TODO` comment in `fetch_proofs` explicitly acknowledges that chunks are currently fetched one at a time, maximising the window for a state change between requests. [6](#0-5) 

### Recommendation

After collecting all chunk proofs, assert that every chunk returned the same `global_roots` before merging:

```rust
pub(crate) fn merge_storage_proofs(
    proofs: Vec<RpcStorageProof>,
    split_queries: &[RpcStorageProofsQuery],
    original_query: &RpcStorageProofsQuery,
) -> Result<RpcStorageProof, ProofProviderError> {
    assert_eq!(proofs.len(), split_queries.len(), "proofs/queries length mismatch");
    assert!(!proofs.is_empty(), "cannot merge zero proofs");

    let global_roots = proofs[0].global_roots.clone();
    for (i, proof) in proofs.iter().enumerate().skip(1) {
        if proof.global_roots != global_roots {
            return Err(ProofProviderError::InvalidProofResponse(format!(
                "Inconsistent global_roots in chunk {i}: \
                 expected contracts_root={}, got {}",
                global_roots.contracts_tree_root,
                proof.global_roots.contracts_tree_root,
            )));
        }
    }
    // ... rest of merge logic
}
```

Alternatively, retry the entire multi-chunk fetch as a single atomic operation when inconsistency is detected, or use a block-pinning RPC parameter (e.g., `starknet_getStorageProof` with a fixed `block_id`) for all chunks.

### Proof of Concept

1. Construct a `VirtualBlockExecutionData` with more than 100 total keys (e.g., 50 contract addresses each with 2 storage keys = 100 keys + 1 class hash = 101 keys total).
2. Configure a mock `RpcStorageProofsProvider` that returns `global_roots.contracts_tree_root = ROOT_A` for the first chunk and `global_roots.contracts_tree_root = ROOT_B ≠ ROOT_A` for the second chunk, with Merkle nodes consistent with `ROOT_B`.
3. Call `get_storage_proofs`.
4. Observe that `merge_storage_proofs` returns a proof with `global_roots.contracts_tree_root = ROOT_A` (from chunk 0) while the Merkle nodes for the second chunk are consistent with `ROOT_B`.
5. `create_commitment_infos_with_state_changes` sets `contracts_trie_root_hash = ROOT_A` and builds a `FactsDb` containing nodes for `ROOT_B`.
6. `commit_state_diff` applies the state diff on top of `ROOT_A` using a `FactsDb` populated with nodes for `ROOT_B`, producing a `CommitmentInfo` with `previous_root = ROOT_A` but commitment facts that do not correspond to `ROOT_A`.
7. The OS receives this corrupted `CommitmentInfo` as its authoritative proof input, anchoring the proof to a wrong state root. [7](#0-6)

pub(crate) fn merge_storage_proofs(
    proofs: Vec<RpcStorageProof>,
    split_queries: &[RpcStorageProofsQuery],
    original_query: &RpcStorageProofsQuery,
) -> RpcStorageProof {
    assert_eq!(proofs.len(), split_queries.len(), "proofs/queries length mismatch");
    assert!(!proofs.is_empty(), "cannot merge zero proofs");

    let global_roots = proofs[0].global_roots.clone();
    let mut classes_proof: IndexMap<Felt, MerkleNode> = IndexMap::default();
    let mut contracts_proof =
        ContractsProof { nodes: IndexMap::default(), contract_leaves_data: Vec::new() };

    let addr_to_idx: HashMap<Felt, usize> = original_query
        .contract_storage_keys
        .iter()
        .enumerate()
        .map(|(i, csk)| (csk.contract_address, i))
        .collect();
    let mut contracts_storage_proofs: Vec<IndexMap<Felt, MerkleNode>> =
        vec![IndexMap::default(); original_query.contract_storage_keys.len()];

    for (chunk_query, proof) in split_queries.iter().zip(proofs) {
        classes_proof.extend(proof.classes_proof);
        contracts_proof.nodes.extend(proof.contracts_proof.nodes);
        contracts_proof.contract_leaves_data.extend(proof.contracts_proof.contract_leaves_data);
        for (csk, storage_proof) in
            chunk_query.contract_storage_keys.iter().zip(proof.contracts_storage_proofs)
        {
            let idx = addr_to_idx
                .get(&csk.contract_address)
                .expect("chunk address not in original query");
            contracts_storage_proofs[*idx].extend(storage_proof);
        }
    }

    RpcStorageProof { classes_proof, contracts_proof, contracts_storage_proofs, global_roots }
}

#[async_trait]
impl StorageProofProvider for RpcStorageProofsProvider {
    async fn get_storage_proofs(
        &self,
        block_number: BlockNumber,
        execution_data: &VirtualBlockExecutionData,
        config: &StorageProofConfig,
    ) -> Result<StorageProofs, ProofProviderError> {
        let query = Self::prepare_query(execution_data);

        let rpc_proof = self.fetch_proofs(block_number, &query).await?;

        // Validate that contract_leaves_data matches contract_addresses length.
        let leaves_len = rpc_proof.contracts_proof.contract_leaves_data.len();
        let addresses_len = query.contract_addresses.len();
        if leaves_len != addresses_len {
            return Err(ProofProviderError::InvalidProofResponse(format!(
                "Contract leaves length mismatch: expected {addresses_len} leaves for requested \
                 contracts, got {leaves_len}"
            )));
        }

        // Update initial reads with class hashes and nonces from the proof.
        // We've validated the lengths match, so this zip is safe.
        let mut extended_initial_reads = StateMaps::default();
        for (leaf, addr) in
            rpc_proof.contracts_proof.contract_leaves_data.iter().zip(&query.contract_addresses)
        {
            extended_initial_reads.class_hashes.insert(*addr, ClassHash(leaf.class_hash));
            extended_initial_reads.nonces.insert(*addr, Nonce(leaf.nonce));
        }

        // Include storage values from execution.
        extended_initial_reads.storage.extend(&execution_data.initial_reads.storage);

        let commitment_infos = match config.include_state_changes {
            true => {
                Self::create_commitment_infos_with_state_changes(
                    &rpc_proof,
                    &query,
                    &extended_initial_reads,
                    &execution_data.state_diff,
                )
                .await?
            }
            false => Self::create_commitment_infos_without_state_changes(&rpc_proof, &query)?,
        };

        Ok(StorageProofs { extended_initial_reads, commitment_infos })
    }

pub(crate) fn create_facts_db_from_storage_proof(
    rpc_proof: &RpcStorageProof,
    query: &RpcStorageProofsQuery,
    initial_reads: &StateMaps,
) -> Result<FactsDb<MapStorage>, ProofProviderError> {
    let mut db_map: DbHashMap = DbHashMap::new();

    insert_inner_nodes(&mut db_map, &rpc_proof.contracts_proof.nodes)?;
    insert_inner_nodes(&mut db_map, &rpc_proof.classes_proof)?;
    for storage_proof in &rpc_proof.contracts_storage_proofs {
        insert_inner_nodes(&mut db_map, storage_proof)?;
    }

    insert_contract_leaves(&mut db_map, rpc_proof, query)?;
    insert_storage_leaves(&mut db_map, initial_reads)?;
    insert_compiled_class_leaves(&mut db_map, initial_reads)?;

    // Add dummy nodes for orphan hashes (sibling hashes without preimages).
    add_dummy_nodes_for_orphan_hashes(&mut db_map, &rpc_proof.contracts_proof.nodes)?;
    add_dummy_nodes_for_orphan_hashes(&mut db_map, &rpc_proof.classes_proof)?;
    for storage_proof in &rpc_proof.contracts_storage_proofs {
        add_dummy_nodes_for_orphan_hashes(&mut db_map, storage_proof)?;
    }

    Ok(FactsDb::new(MapStorage(db_map)))
}
