### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Freezing of Contract Funds — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS accepts any arbitrary `class_hash` value without verifying that the hash corresponds to a previously declared contract class. A contract can call `replace_class` with an undeclared (or zero) class hash, setting its own `state_entry.class_hash` to an invalid sentinel value. Every subsequent call to that contract will fail because the OS cannot resolve the class, permanently freezing any funds held by the contract. This is the direct analog of the external report's pattern: a missing same-identifier guard allows a state variable to be set to a value that makes a subsequent critical operation permanently impossible.

---

### Finding Description

`execute_replace_class` in `syscall_impls.cairo` reads the current `StateEntry` for the calling contract, constructs a new `StateEntry` with the caller-supplied `class_hash`, and writes it to `contract_state_changes` via `dict_update`:

```cairo
// syscall_impls.cairo lines 896–910
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
```

There is **no check** that `class_hash` exists in `contract_class_changes` (the declared-class registry). The TODO comment explicitly acknowledges this missing validation and was due 1 January 2026 — already past as of today.

The revert log records the old class hash for potential rollback:

```cairo
// syscall_impls.cairo line 912
assert [revert_log] = RevertLogEntry(selector=CHANGE_CLASS_ENTRY, value=state_entry.class_hash);
```

But if the transaction is **not** reverted, the contract's class hash is permanently set to the undeclared value. Any future invocation of the contract will fail at class resolution time because the OS cannot find the class body, making the contract permanently inoperable.

The same `execute_replace_class` function is reachable from both the deprecated syscall dispatcher (`deprecated_execute_syscalls.cairo` line 676) and the non-deprecated path, so both old and new contract classes are affected.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's `class_hash` is set to an undeclared hash:
- Every future call to the contract fails at class-resolution time.
- No `__execute__`, `__validate__`, or any other entry point can run.
- Any ERC-20 tokens, ETH, or other assets held in the contract's storage become permanently inaccessible.
- There is no recovery path: the OS has no mechanism to "un-replace" a class after a non-reverted transaction.

This matches the external report's pattern exactly: the missing validation allows a state variable (`class_hash`) to be set to a value that makes all subsequent operations on the contract permanently impossible, analogous to `auctionStartTimestamp` being reset to `type(uint256).max` to make `seizeLoan` always revert.

---

### Likelihood Explanation

**High.** The attack path requires only a single `replace_class` syscall from within a contract execution. A malicious contract deployer can:
1. Deploy a contract that accepts user deposits.
2. Include a function (or constructor logic) that calls `replace_class` with `class_hash = 0` or any other undeclared felt value.
3. After accumulating user funds, trigger that function.
4. The contract's class hash is set to an undeclared value; all funds are frozen.

No privileged role, leaked key, or external dependency is required. The attacker is an ordinary contract deployer — explicitly listed as an in-scope unprivileged actor.

---

### Recommendation

Inside `execute_replace_class`, after reading `class_hash` from the request, verify that the hash is present in the `contract_class_changes` dictionary (or an equivalent declared-class lookup). Concretely, perform a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the returned `compiled_class_hash` is non-zero before proceeding with the `dict_update`. This is exactly what the existing TODO comment calls for.

```cairo
// Proposed fix (pseudocode):
let class_hash = request.class_hash;
// Verify the class is declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
with_attr error_message("Class hash is not declared.") {
    assert_not_zero(compiled_class_hash);
}
// ... proceed with state update
```

### Proof of Concept

1. Deploy contract `A` with a function `freeze()` that executes the `replace_class` syscall with `class_hash = 1` (an undeclared hash).
2. Users deposit funds into contract `A`.
3. Attacker calls `freeze()`.
4. The OS executes `execute_replace_class`:
   - Reads `state_entry` for contract `A` (current class hash = declared hash `H`).
   - Constructs `new_state_entry` with `class_hash = 1`.
   - Calls `dict_update` — **no declared-class check is performed**.
   - Writes `CHANGE_CLASS_ENTRY, value=H` to the revert log.
5. Transaction completes without revert; `state_entry.class_hash` for contract `A` is now `1`.
6. Any subsequent call to contract `A` fails: the OS cannot resolve class `1` → entry point execution is impossible.
7. All funds in contract `A`'s storage are permanently frozen.
