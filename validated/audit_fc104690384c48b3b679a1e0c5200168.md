### Title
Flat Gas Cost for Unbounded `send_message_to_l1` Payload Enables OS Step Exhaustion and Network Halt — (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

`execute_send_message_to_l1` charges a flat, payload-size-independent gas cost (`SEND_MESSAGE_TO_L1_GAS_COST = 14470`) while performing an unbounded `memcpy` of the entire message payload into the OS output segment. Because the OS's own Cairo step cost scales linearly with `payload_size` but the gas deducted from the caller is constant, a malicious contract can force the OS to perform arbitrarily large amounts of work per transaction. This can cause a block's OS execution to exceed the prover's step capacity, making the block unprovable and halting the network.

---

### Finding Description

In `execute_send_message_to_l1`:

```cairo
let success = reduce_syscall_gas_and_write_response_header(
    total_gas_cost=SEND_MESSAGE_TO_L1_GAS_COST,   // flat: 14470
    request_struct_size=SendMessageToL1Request.SIZE
);
...
tempvar payload_size = request.payload_end - payload_start;
// No upper-bound check on payload_size.
memcpy(
    dst=outputs.messages_to_l1 + MessageToL1Header.SIZE,
    src=payload_start,
    len=payload_size          // unbounded
);
``` [1](#0-0) 

The gas deducted is the constant `SEND_MESSAGE_TO_L1_GAS_COST = 14470`. [2](#0-1) 

There is no per-element factor for the payload, unlike `deploy` and `meta_tx_v0` which correctly scale gas with data size:

```cairo
// deploy: DEPLOY_GAS_COST + DEPLOY_CALLDATA_FACTOR_GAS_COST * calldata_size
// meta_tx_v0: META_TX_V0_GAS_COST + META_TX_V0_CALLDATA_FACTOR_GAS_COST * calldata_size
``` [3](#0-2) 

`payload_size` is derived from `request.payload_end - payload_start` with no `assert_nn_le` bound check before the `memcpy`. [4](#0-3) 

The same pattern exists in the deprecated Cairo 0 path, where there is no gas deduction at all for the payload copy: [5](#0-4) 

---

### Impact Explanation

The StarkNet OS runs as a Cairo program whose execution is bounded by the prover's step capacity. The `memcpy` inside `execute_send_message_to_l1` costs one Cairo step per felt element copied. A contract paying only 14470 L2 gas can force the OS to execute millions of `memcpy` steps. If the total OS step count for a block exceeds the prover's capacity, the block cannot be proven. An unproven block means the network cannot advance state — **total network halt**.

**Impact: High — Network not being able to confirm new transactions.**

---

### Likelihood Explanation

Any unprivileged contract can call `send_message_to_l1` with an arbitrarily large payload array. The `EXECUTE_MAX_SIERRA_GAS = 1,100,000,000` allows approximately `1,100,000,000 / 14,470 ≈ 76,000` such calls per transaction, each with a large payload. A single call with a payload of millions of felts costs only 14470 gas to the caller but forces millions of OS-level Cairo steps. This is trivially exploitable by any deployed contract. [6](#0-5) 

---

### Recommendation

Introduce a per-element gas factor for the `send_message_to_l1` payload, analogous to `DEPLOY_CALLDATA_FACTOR_GAS_COST` and `META_TX_V0_CALLDATA_FACTOR_GAS_COST`:

```cairo
let specific_base_gas_cost = (
    SEND_MESSAGE_TO_L1_GAS_COST + SEND_MESSAGE_TO_L1_PAYLOAD_FACTOR_GAS_COST * payload_size
);
```

Additionally, add an explicit upper-bound check on `payload_size` before the `memcpy`, analogous to the `assert_nn_le(calldata_size, SIERRA_ARRAY_LEN_BOUND - 1)` checks used elsewhere. [7](#0-6) 

---

### Proof of Concept

1. Deploy a contract with an `__execute__` function that calls `send_message_to_l1` with a payload array of N = 10,000,000 felts.
2. Submit an invoke transaction calling this contract. Gas cost charged: 14,470 (flat).
3. The OS processes the block: `execute_send_message_to_l1` runs `memcpy(..., len=10_000_000)`, consuming ~10,000,000 Cairo steps in the OS program for this single syscall.
4. The block's total OS step count exceeds the prover's capacity.
5. The block cannot be proven; the network cannot confirm new transactions.

The attacker controls `payload_size` entirely through the contract's Sierra code (constructing a large array and passing `payload_start`/`payload_end` pointers). No privileged access is required. [8](#0-7)

// Sends a message to L1.
func execute_send_message_to_l1{range_check_ptr, syscall_ptr: felt*, outputs: OsCarriedOutputs*}(
    contract_address: felt
) {
    alloc_locals;
    let request = cast(syscall_ptr + RequestHeader.SIZE, SendMessageToL1Request*);
    let success = reduce_syscall_gas_and_write_response_header(
        total_gas_cost=SEND_MESSAGE_TO_L1_GAS_COST, request_struct_size=SendMessageToL1Request.SIZE
    );
    if (success == FALSE) {
        // Not enough gas to execute the syscall.
        return ();
    }

    tempvar payload_start = request.payload_start;
    tempvar payload_size = request.payload_end - payload_start;

    assert [outputs.messages_to_l1] = MessageToL1Header(
        from_address=contract_address, to_address=request.to_address, payload_size=payload_size
    );
    memcpy(
        dst=outputs.messages_to_l1 + MessageToL1Header.SIZE, src=payload_start, len=payload_size
    );
    let (outputs) = os_carried_outputs_new(
        messages_to_l1=outputs.messages_to_l1 + MessageToL1Header.SIZE + payload_size,
        messages_to_l2=outputs.messages_to_l2,
    );

    return ();
}
```
