Audit Report

## Title
Unguarded `refundETH()` / `sweepToken()` / `unwrapWETH9()` Allow Any Caller to Steal ETH and Tokens Stranded on the Router Between Transactions — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
`exactOutputSingle` and `exactInputSingle` are `external payable` functions that accept `msg.value` but do not auto-refund excess ETH after the swap callback consumes only the realized `amountIn`. The remaining ETH sits on the router with no per-user attribution. Because `refundETH()`, `sweepToken()`, and `unwrapWETH9()` carry no access control and operate on the router's full balance, any caller in a subsequent transaction can drain the stranded ETH or tokens.

## Finding Description
`exactOutputSingle` is declared `external payable` and accepts arbitrary `msg.value`. Inside the swap callback, `pay()` inspects `address(this).balance` and wraps only the realized `amountIn` worth of native ETH before pulling the remainder from the payer via `safeTransferFrom`. The excess ETH (`msg.value − amountIn`) is never returned inside `exactOutputSingle` itself — the protocol's intended recovery is `refundETH()` appended as a second call inside a `multicall`, but nothing enforces this pattern.

```solidity
// PeripheryPayments.sol L73-84
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

After `exactOutputSingle` returns, the excess ETH remains on the router. `refundETH()` then sends the entire `address(this).balance` to `msg.sender` with no check that the caller is the original depositor:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) { _transferETH(msg.sender, balance); }
}
```

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks plain ETH transfers; it does not prevent ETH from being deposited as `msg.value` in a call to a `payable` function such as `exactOutputSingle`. Similarly, `sweepToken` and `unwrapWETH9` accept a caller-chosen `recipient` with no restriction, enabling redirection of any stranded ERC-20 or WETH balance to an attacker address.

## Impact Explanation
Direct loss of user ETH principal. A victim who calls `exactOutputSingle{value: amountInMaximum}` without wrapping it in a `multicall` that includes `refundETH()` leaves `amountInMaximum − realizedAmountIn` on the router. An attacker who observes this state calls `refundETH()` in the next block and receives the full stranded balance. The same path applies to WETH output stranded by `exactInput(..., recipient = address(router))` without a subsequent `unwrapWETH9` in the same multicall. Loss is bounded by `msg.value` per victim transaction, which can be the full `amountInMaximum`.

## Likelihood Explanation
Medium. `exactOutputSingle` and `exactInputSingle` are independently callable `external payable` functions — nothing prevents direct invocation with excess ETH. A user who calls `exactOutputSingle{value: X}` and then issues `refundETH()` in a separate transaction faces a front-running window any searcher can exploit. The `MetricOmmPoolLiquidityAdder` exposes the same surface via its own `addLiquidityExactShares{value: X}` payable entry-point, which also relies on the caller including `refundETH()` in the same multicall.

## Recommendation
1. **Auto-refund excess ETH** at the end of every `external payable` entry-point (`exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`, `addLiquidityExactShares`, `addLiquidityWeighted`) rather than relying on the caller to include `refundETH` in a multicall.
2. **Restrict `refundETH`** so it can only be invoked via `delegatecall` from `multicall` (e.g., check `address(this) == implementation`), preventing standalone cross-transaction drains.
3. **Restrict `sweepToken` and `unwrapWETH9`** similarly, or require `recipient == msg.sender` to prevent arbitrary redirection of stranded balances.

## Proof of Concept
```
// Step 1 — Victim calls exactOutputSingle directly (no multicall)
router.exactOutputSingle{value: 1 ether}(
    ExactOutputSingleParams({
        pool: pool,
        tokenIn: WETH,
        tokenOut: token1,
        zeroForOne: true,
        amountOut: 500,
        amountInMaximum: 1 ether,
        recipient: victim,
        deadline: ...,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// pay() wraps only realizedAmountIn (e.g. 0.4 ETH); 0.6 ETH stays on router.
// exactOutputSingle returns without refunding.

// Step 2 — Attacker (next block)
router.refundETH();   // msg.sender = attacker → receives 0.6 ETH
```

Foundry test plan: deploy router with a mock WETH pool, call `exactOutputSingle{value: 1 ether}` from `alice` with `amountInMaximum` set above the quoted input, assert `address(router).balance > 0` after the call, then call `router.refundETH()` from `attacker` and assert `attacker` received the stranded ETH while `alice` received nothing.
