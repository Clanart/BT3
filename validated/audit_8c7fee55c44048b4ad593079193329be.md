### Title
`ProofOfInclusion::valid()` Contains a Tautological Root Check, Allowing Forged DataLayer Inclusion Proofs - (File: crates/chia-datalayer/src/merkle/proof_of_inclusion.rs)

### Summary

`ProofOfInclusion::valid()` performs a self-referential final comparison: after the loop verifies each layer's `combined_hash`, it checks `existing_hash == self.root_hash()`. Because `root_hash()` returns the last `combined_hash` from the proof's own layers — the same value `existing_hash` was just set to — this final check is always `true`. The function therefore only verifies internal consistency of the proof, never that the proof's root matches any externally-known tree root. An attacker can construct a `ProofOfInclusion` with an arbitrary `node_hash` (claiming any key-value pair is in the tree) and internally-consistent layers, and `valid()` will return `true`, enabling forged DataLayer inclusion proofs.

### Finding Description

In `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`, the `valid()` method is:

```rust
pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;  // existing_hash = layer.combined_hash
    }

    existing_hash == self.root_hash()    // always true
}
```

And `root_hash()`:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash          // same value existing_hash was just set to
    } else {
        self.node_hash              // same value existing_hash was initialized to
    }
}
```

**Tautology proof:**

- After each loop iteration, `existing_hash` is set to `calculated_hash`, which was just verified to equal `layer.combined_hash`. So after the loop, `existing_hash == last_layer.combined_hash`.
- `self.root_hash()` returns `last_layer.combined_hash` (non-empty case) or `self.node_hash` (empty case).
- In both cases, `existing_hash == self.root_hash()` is unconditionally `true` when the loop completes without returning `false`.

The function therefore accepts **any** `ProofOfInclusion` whose layers are internally consistent (each `combined_hash` equals the hash of its children), regardless of whether the resulting root matches the actual committed DataLayer tree root.

The analog to the AutoRoller vulnerability is direct: AutoRoller's `eject` used `asset.balanceOf(address(this))` — the total accumulated contract balance including yield from all YTs — instead of the return value of `combine()` (the user's entitled amount). Here, `valid()` uses `self.root_hash()` — a value derived entirely from the proof's own fields — instead of an externally-supplied known root, making the check self-referential and meaningless as a security boundary.

The `ProofOfInclusion` struct derives `Streamable`, so it is fully deserializable from attacker-controlled bytes via `from_bytes()` / `parse_rust()`, both of which are exposed in the Python wheel (`wheel/python/chia_rs/datalayer.pyi`). The `valid()` method is also exposed as a Python binding (`py_valid`).

### Impact Explanation

An attacker acting as a malicious DataLayer peer can craft a serialized `ProofOfInclusion` with:
- `node_hash` = any hash (e.g., `SHA256(fake_key || fake_value)`)
- `layers` = any internally-consistent sequence (or empty)

When a Python client deserializes this and calls `proof.valid()`, it receives `True`. If the client trusts `valid()` as a complete validity check — which the method name implies — it will accept the forged proof and believe the fake key-value pair is present in the DataLayer store, even though the actual on-chain committed root is entirely different. This lets untrusted input prove invalid state, matching the allowed High impact for DataLayer Merkle proof logic.

### Likelihood Explanation

The `valid()` method is exposed via Python bindings and is the only self-contained validity API on `ProofOfInclusion`. Its name strongly implies it is a complete check. Any Python caller that does `if proof.valid(): trust_proof()` without separately checking `proof.root_hash() == known_root` is vulnerable. The DataLayer synchronization protocol involves receiving proofs from remote peers, making attacker-controlled `ProofOfInclusion` bytes a realistic entry path.

### Recommendation

`valid()` must accept an external root hash and verify against it:

```rust
pub fn valid_against_root(&self, expected_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == expected_root   // compare against external root, not self.root_hash()
}
```

The current `valid()` (which only checks internal consistency) should be renamed to `internally_consistent()` or removed, to prevent callers from mistaking it for a complete security check.

### Proof of Concept

```python
from chia_rs.datalayer import ProofOfInclusion, ProofOfInclusionLayer

# Forge a proof claiming node_hash X is in the tree, with no layers.
# root_hash() returns node_hash, existing_hash == node_hash, so valid() == True.
fake_node_hash = bytes([0xAB] * 32)
forged_proof = ProofOfInclusion(node_hash=fake_node_hash, layers=[])

assert forged_proof.valid()          # True — accepted despite being fabricated
assert forged_proof.root_hash() == fake_node_hash  # "root" is whatever attacker chose

# With layers: attacker builds an internally-consistent chain leading to any root.
# valid() returns True for any such chain, regardless of the actual committed root.
```
