### Title
`SuggestGasTipCap` Silent `int64` Truncation Produces Incorrect Gas-Price Suggestions for Large Base Fees - (File: `rpc/backend/chain_info.go`)

### Summary
`SuggestGasTipCap` in `rpc/backend/chain_info.go` uses `baseFee.Int64()` to convert an arbitrary-precision `*big.Int` base fee into a native `int64` before performing arithmetic. When the base fee exceeds `math.MaxInt64`, the conversion silently returns the low 64 bits as a signed integer (undefined behavior per Go spec), causing the multiplication to silently overflow `int64` and produce an incorrect (or zero-clamped) tip-cap. This is the direct Ethermint analog of the `getUsdPrice()` incomplete-path bug: the function only works correctly for a specific sub-range of its input domain and silently returns wrong results outside that range.

### Finding Description
`SuggestGasTipCap` is the backend implementation of `eth_maxPriorityFeePerGas` and is also called by `GasPrice()` (`eth_gasPrice`) and `SetTxDefaults()` (used by `eth_sendTransaction`):

```go
// rpc/backend/chain_info.go:439
maxDelta := baseFee.Int64() * (int64(params.Params.ElasticityMultiplier) - 1) / int64(params.Params.BaseFeeChangeDenominator)
if maxDelta < 0 {
    maxDelta = 0
}
return big.NewInt(maxDelta), nil
```

`baseFee` is a `*big.Int` stored in the feemarket params as an `sdkmath.Int` (unbounded). When `baseFee > math.MaxInt64` (~9.2 × 10¹⁸):

1. `baseFee.Int64()` returns the low 64 bits reinterpreted as `int64` — a large negative number.
2. The subsequent `int64` multiplication can silently overflow (Go integer overflow wraps without panic).
3. The `maxDelta < 0` guard clamps obviously-negative results to 0, but overflow-wrapped positive values pass through unchecked.

Downstream effects:
- `eth_maxPriorityFeePerGas` returns 0 or a garbage value.
- `eth_gasPrice` returns `baseFee + garbage_tip` — either far too low or far too high.
- `SetTxDefaults` sets `MaxPriorityFeePerGas = garbage` and `MaxFeePerGas = garbage + 2×baseFee`, causing `eth_sendTransaction` to build transactions with malformed EIP-1559 fee fields.

The base fee is stored as a full `sdkmath.Int` (big.Int) and grows each block when blocks are full. The `SetBaseFee` path writes it back without any int64 cap.

### Impact Explanation
This matches **High — Public JSON-RPC path feeds incorrect data into transaction execution**. `eth_gasPrice` and `eth_maxPriorityFeePerGas` are the canonical fee-estimation endpoints consumed by every Ethereum wallet and toolchain (MetaMask, ethers.js, web3.py). When these return incorrect values:

- Transactions built via `eth_sendTransaction` / `SetTxDefaults` carry malformed `MaxFeePerGas` / `MaxPriorityFeePerGas` fields.
- If the returned tip is 0 when it should be positive, EIP-1559 transactions may be rejected by the mempool (`feeCap < baseFee` check in `NewDynamicFeeChecker`).
- If the overflow wraps to a large positive value, `SetTxDefaults` inflates `MaxFeePerGas` to an enormous number, causing users to massively overpay fees — a mis-accounting of valid user funds.

### Likelihood Explanation
The base fee must exceed `math.MaxInt64` (~9.2 × 10¹⁸ wei = ~9.2 ETH per gas unit). Starting from the default of 10⁹ wei, this requires sustained full-block pressure over a very long period. While unlikely on a well-operated chain, it is reachable on a chain with low `BaseFeeChangeDenominator`, high `ElasticityMultiplier`, or sustained spam. No privileged action is required — any user submitting transactions that fill blocks contributes to base-fee growth. The RPC endpoints themselves are fully unprivileged.

### Recommendation
Replace the `int64` arithmetic with `*big.Int` arithmetic throughout `SuggestGasTipCap`:

```go
// Safe replacement
elasticity := new(big.Int).SetUint64(uint64(params.Params.ElasticityMultiplier))
denom := new(big.Int).SetUint64(uint64(params.Params.BaseFeeChangeDenominator))
num := new(big.Int).Mul(baseFee, new(big.Int).Sub(elasticity, big.NewInt(1)))
maxDelta := new(big.Int).Div(num, denom)
if maxDelta.Sign() < 0 {
    maxDelta = new(big.Int)
}
return maxDelta, nil
```

Similarly audit `BaseFee()` fallback at line 98 which uses `strconv.ParseInt(..., 10, 64)` — also truncates base fees > MaxInt64.
