Looking at the codebase, I found a direct analog to the `previewWithdraw` bug: a "preview/estimate" function that uses a different (incorrect) formula compared to the actual execution path.### Title
`estimate_minimal_gas_vector` Omits `n_compiled_class_hash_updates` for Declare V2/V3, Causing Gateway to Admit Transactions with Insufficient Resource Bounds — (File: crates/blockifier/src/fee/gas_usage.rs)

---

### Summary

`estimate_minimal_gas_vector` is the sequencer's "preview" function for minimum gas: it is called by `check_fee_bounds` during gateway pre-validation to reject transactions whose resource bounds are below the minimum required. For `Declare` transactions it hardcodes `n_compiled_class_hash_updates: 0`, even though V2/V3 Declare transactions write exactly one compiled-class-hash entry to the DA segment. The actual execution path correctly counts this update. The mismatch causes the gateway to admit Declare V2/V3 transactions whose resource bounds are too low to cover the real DA cost; those transactions then revert on-chain.

---

### Finding Description

**Estimation path (wrong)** — `crates/blockifier/src/fee/gas_usage.rs`, `estimate_minimal_gas_vector`:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    // TODO(Yoni): BLOCKIFIER-RESET: should be 1.
    n_compiled_class_hash_updates: 0,   // ← BUG: should be 1 for V2/V3
    n_modified_contracts: 1,
},
```

The TODO comment is a self-admission of the defect. [1](#0-0) 

**Actual execution path (correct)** — `crates/blockifier/src/transaction/transactions_test.rs`, `declare_expected_state_changes_count`:

```rust
} else if version == TransactionVersion::TWO || version == TransactionVersion::THREE {
    StateChangesCount {
        n_storage_updates: 1,
        n_modified_contracts: 1,
        n_compiled_class_hash_updates: 1, // Also set compiled class hash.
        ..StateChangesCount::default()
    }
``` [2](#0-1) 

**DA segment length formula** — `get_onchain_data_segment_length` adds `n_compiled_class_hash_updates * 2` felts:

```rust
// For each compiled class updated (through declare): class_hash, compiled_class_hash
onchain_data_segment_length += state_changes_count.n_compiled_class_hash_updates * 2;
``` [3](#0-2) 

The estimate therefore underestimates the DA segment by exactly **2 felts**, which translates to:
- KZG DA mode: `2 × DATA_GAS_PER_FIELD_ELEMENT` blob-gas units missing from the minimum bound.
- Calldata DA mode: `2 × SHARP_GAS_PER_DA_WORD` L1-gas units missing.

**Pre-validation call site** — `check_fee_bounds` in `crates/blockifier/src/transaction/account_transaction.rs` calls `estimate_minimal_gas_vector` and rejects the transaction only if `minimal_gas_amount > resource_bounds.max_amount`: [4](#0-3) 

Because the estimate is too low, a transaction whose `max_amount` sits between the underestimated minimum and the true minimum passes this check and enters the mempool.

---

### Impact Explanation

A user submits a Declare V2/V3 transaction with `l1_data_gas.max_amount` (KZG DA) or `l1_gas.max_amount` (calldata DA) set to a value in the gap `[estimated_min, true_min)`. The gateway's `check_fee_bounds` passes because it uses the underestimated value. The transaction is sequenced. During blockifier execution the actual DA cost exceeds the user's resource bounds, causing the transaction to **revert**. The sequencer includes a reverted Declare in the block; the class is never declared; the user is charged for the revert.

This matches: **High — Mempool/gateway admission accepts transactions that should be rejected before sequencing.**

---

### Likelihood Explanation

Any unprivileged user can craft a Declare V2/V3 transaction with resource bounds in the gap. No special role, key, or peer access is required. The gap is fixed and predictable (exactly 2 DA felts), making it trivially exploitable.

---

### Recommendation

In `estimate_minimal_gas_vector`, change the `Declare` arm to:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    n_compiled_class_hash_updates: 1,  // V2/V3 writes compiled class hash
    n_modified_contracts: 1,
},
```

This aligns the estimation with the actual execution path and closes the admission gap. The existing TODO comment already identifies this fix. [1](#0-0)

### Proof of Concept

1. Compute the **true** minimum DA gas for a Declare V3 transaction (KZG DA):
   - `StateChangesCount { n_storage_updates:1, n_modified_contracts:1, n_compiled_class_hash_updates:1 }`
   - DA segment = `1×2 + 1×2 + 1×2 = 6` felts → `6 × DATA_GAS_PER_FIELD_ELEMENT` blob gas.

2. Compute the **underestimated** minimum (as the gateway sees it):
   - `StateChangesCount { n_storage_updates:1, n_modified_contracts:1, n_compiled_class_hash_updates:0 }`
   - DA segment = `1×2 + 1×2 = 4` felts → `4 × DATA_GAS_PER_FIELD_ELEMENT` blob gas.

3. Craft a Declare V3 transaction with `l1_data_gas.max_amount` set to `4 × DATA_GAS_PER_FIELD_ELEMENT` (i.e., exactly the underestimated minimum).

4. Submit to the gateway. `check_fee_bounds` computes `minimal_gas_amount = 4 × DATA_GAS_PER_FIELD_ELEMENT`, which equals `max_amount` → **admission passes**.

5. The blockifier executes the transaction. Actual DA cost = `6 × DATA_GAS_PER_FIELD_ELEMENT > max_amount` → **transaction reverts**. The class is not declared; the user is charged for the failed execution.
