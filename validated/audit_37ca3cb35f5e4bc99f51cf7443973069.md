Audit Report

## Title
Partial-ETH branch in `pay()` silently consumes stranded router ETH to subsidize a subsequent WETH swap — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary
The `pay()` helper in `PeripheryPayments.sol` contains a middle branch (lines 78–81) that fires when the router holds any ETH less than the required payment amount. It unconditionally wraps and spends the router's entire ETH balance as a partial payment, then pulls only the remainder from the actual payer's allowance. Because the router's ETH balance carries no per-user attribution, ETH stranded by a prior user (who sent `msg.value` but omitted `refundETH()`) is silently consumed by the next WETH swap, causing the original depositor an irreversible loss of principal.

## Finding Description
`pay()` in `PeripheryPayments.sol` has three branches when `token == WETH` and `payer != address(this)`:

```
nativeBalance >= value  →  wrap value ETH, send to pool                          (line 75–77)
nativeBalance > 0       →  wrap nativeBalance ETH + pull (value−nativeBalance)   (line 78–81) ← root cause
nativeBalance == 0      →  pull full value from payer                             (line 82–84)
```

The middle branch unconditionally drains `address(this).balance` regardless of which user deposited it. The reachable call path is:

```
exactInputSingle{value: V}(tokenIn=WETH, amountIn=A)   [A < V]
  → _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, msg.sender, WETH)
  → pool.swap(...)
  → metricOmmSwapCallback(...)
  → _justPayCallback(...)
  → pay(WETH, payer=UserA, pool, A)
      nativeBalance=V >= A → wraps A ETH, sends to pool
  After swap: router.balance = V−A  (stranded if UserA omits refundETH)

Later: exactInputSingle(tokenIn=WETH, amountIn=X)  [X > V−A]
  → pay(WETH, payer=UserB, pool, X)
      nativeBalance=V−A > 0, < X → partial branch fires:
        wraps V−A ETH → sends to pool
        safeTransferFrom(UserB, pool, X−(V−A))   ← only X−(V−A) pulled from UserB
```

The `receive()` guard (line 32–34) only blocks direct ETH pushes from non-WETH addresses; it does not prevent ETH accumulation via `msg.value` on any `payable` entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, `multicall`, `unwrapWETH9`, `sweepToken`, `refundETH`). No existing guard checks that the router's ETH balance was deposited by the current payer before consuming it.

## Impact Explanation
User A suffers a direct, irreversible loss of principal equal to the stranded ETH amount (`V−A`). User B receives the full swap output while paying `V−A` less WETH than owed — an unearned discount funded entirely by User A. The pool is made whole (receives the correct WETH amount), so no pool insolvency occurs, but the cross-user ETH theft constitutes a direct loss of user principal, meeting the Critical/High/Medium allowed impact gate for direct loss of user principal above Sherlock thresholds.

## Likelihood Explanation
- `refundETH()` is not enforced by any entry point; it must be manually composed into a multicall, making omission common.
- Any `payable` entry point can leave residual ETH on the router.
- The exploit requires no privileged access: any unprivileged address can call `exactInputSingle` with `tokenIn = WETH` to trigger the partial-ETH branch.
- The router's ETH balance is publicly readable on-chain; an attacker can monitor it and time the call immediately after a victim's transaction.
- The attack is repeatable as long as ETH is stranded on the router.

## Recommendation
Replace the partial-ETH branch with the Uniswap v3 pattern — only use router ETH when it fully covers the payment; otherwise fall through to `safeTransferFrom`:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

This eliminates cross-user ETH consumption while preserving the intended "pay with native ETH" convenience for users who send exactly the right `msg.value`.
