### Title
Cross-Chain Replay of Unprotected Ethereum Transactions When `allow_unprotected_txs=true` — (`rpc/backend/call_tx.go`, `ante/interfaces/setup.go`, `ante/sigverify.go`)

### Summary

When the on-chain EVM parameter `allow_unprotected_txs` is `true`, an attacker can replay a victim's pre-EIP155 (HomesteadSigner-signed) Ethereum mainnet transaction on Ethermint and commit an unauthorized `evm_denom` balance transfer. The root cause is that `VerifyEthSig` uses `ethtypes.MakeSigner(chainID=9000)` which, for unprotected transactions (V=27/28), falls back to chain-ID-agnostic Homestead recovery rules — making the signature check identical across all chains.

### Finding Description

**Step 1 — RPC ingestion (`rpc/backend/call_tx.go:135-163`)**

`SendRawTransaction` has two guards:

```go
// Guard 1: node-level config
if !b.UnprotectedAllowed() && !tx.Protected() {
    return common.Hash{}, errors.New("only replay-protected (EIP-155) transactions allowed over RPC")
}

// Guard 2: chain-ID check — only fires when tx.Protected() == true
if tx.Protected() && tx.ChainId().Cmp(b.chainID) != 0 {
    return common.Hash{}, fmt.Errorf("incorrect chain-id; expected %d, got %d", ...)
}
```

For an unprotected tx (`tx.Protected() == false`), Guard 2 is **never evaluated**. Guard 1 is bypassed when `b.allowUnprotectedTxs == true` (node config) or when the attacker submits directly via CometBFT broadcast, bypassing the JSON-RPC layer entirely.

`FromSignedEthereumTx` is then called with `ethtypes.LatestSignerForChainID(b.chainID)` (chainID=9000). For an unprotected legacy tx, go-ethereum's `LatestSignerForChainID` internally falls back to Homestead rules when V=27/28, so it successfully recovers the victim's address and sets `msg.From`.

**Step 2 — Ante handler (`ante/interfaces/setup.go:126-130`, `evmd/ante/handler_options.go:118-125`)**

```go
// ValidateEthBasic — bypassed when allowUnprotectedTxs=true
if !allowUnprotectedTxs && !tx.Protected() {
    return errorsmod.Wrapf(errortypes.ErrNotSupported, "rejected unprotected Ethereum transaction...")
}
```

```go
// VerifyEthSig — uses MakeSigner with Ethermint's chainID
ethSigner := ethtypes.MakeSigner(blockCfg.ChainConfig, blockCfg.BlockNumber, blockCfg.BlockTime)
if err := evmante.VerifyEthSig(tx, ethSigner); err != nil { ... }
```

`VerifyEthSig` calls `msgEthTx.VerifySender(signer)` → `recoverSender(signer)` → `ethtypes.Sender(signer, tx)`. For an unprotected tx (V=27/28), `MakeSigner` with EIP155Block=0 active still falls back to Homestead recovery (chain-ID-agnostic). The recovered address equals the victim's address (same private key, same Homestead hash), so `VerifySender` passes.

**Step 3 — State transition (`x/evm/keeper/state_transition.go:163-170`)**

```go

    cfg, err := k.EVMConfig(ctx, k.eip155ChainID, ethTx.Hash())
    msg := msgEth.AsMessage(cfg.BaseFee)
    // msg.From = victim's address (set by FromSignedEthereumTx)

    ...

```

`AsMessage` reads `msg.From` directly from the pre-set field — no re-verification of the signature against Ethermint's chain ID occurs here. The value transfer commits.

### Impact Explanation

An attacker who captures a victim's pre-EIP155 signed transaction from the Ethereum mainnet mempool can replay it on any Ethermint chain with `allow_unprotected_txs=true`, draining the victim's `evm_denom` balance. The victim's address and nonce must match on both chains (a realistic condition for new accounts or accounts that have executed the same number of transactions on both chains).

**Impact**: Unauthorized `evm_denom` balance transfer — Critical.

### Likelihood Explanation

- `allow_unprotected_txs` is a governance-controlled on-chain parameter. Chains that enable it to support legacy tooling or Homestead-era wallets are directly exposed.
- The default is `false`, but the parameter exists precisely to be toggled.
- Pre-EIP155 transactions still appear in the Ethereum mempool from legacy wallets and certain DeFi protocols.
- The nonce-matching precondition is realistic for accounts that have performed the same number of transactions on both chains (e.g., fresh accounts, or accounts that mirror activity).

**Likelihood**: Medium (requires governance enabling the param, but the exploit is trivially executable once enabled).

### Recommendation

1. **In `ValidateEthBasic`** (`ante/interfaces/setup.go`): When `allowUnprotectedTxs=true` and EIP155 is active (i.e., `ethCfg.IsEIP155(blockNumber)`), reject unprotected transactions regardless of the param. The param should only permit unprotected txs when EIP155 has not yet activated.

2. **In `SendRawTransaction`** (`rpc/backend/call_tx.go`): Apply the chain-ID check unconditionally for unprotected txs when EIP155 is active — not just when `tx.Protected()`. Since unprotected txs carry no chain ID, they should be rejected outright when EIP155 is active.

3. **Alternatively**: Deprecate `allow_unprotected_txs` entirely on chains where EIP155Block=0 (already active from genesis), as there is no safe way to permit unprotected txs on a chain where EIP155 is active without enabling cross-chain replay.

### Proof of Concept

```
1. Deploy Ethermint with allow_unprotected_txs=true, EIP155Block=0, chainID=9000.
2. Fund victim address V on Ethermint (same address as on Ethereum mainnet).
3. On Ethereum mainnet, victim submits a legacy tx signed with HomesteadSigner:
     tx = LegacyTx{Nonce: N, To: attacker, Value: X, GasPrice: P, Gas: 21000}
     sig = HomesteadSigner.Sign(tx, victimKey)  // V=27/28, no chain ID
4. Attacker captures the RLP-encoded tx from the Ethereum mempool.
5. Attacker submits to Ethermint eth_sendRawTransaction (or CometBFT broadcast_tx_sync).
6. SendRawTransaction: tx.Protected()==false, allowUnprotectedTxs==true → passes.

8. ApplyTransaction: msg.From=V, value transfer to attacker commits.
9. Victim's evm_denom balance drained on Ethermint without their knowledge.

	for _, msg := range tx.GetMsgs() {
		msgEthTx, ok := msg.(*evmtypes.MsgEthereumTx)
		if !ok {
			return errorsmod.Wrapf(errortypes.ErrUnknownRequest, "invalid message type %T, expected %T", msg, (*evmtypes.MsgEthereumTx)(nil))

	}
