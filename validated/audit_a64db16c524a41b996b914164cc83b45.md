### Title
Missing Declared-Class Validation in `execute_replace_class` Allows Permanent Contract Freezing - (File: `crates/apollo_starknet_os_program/src/cairo/starkware/starknet/core/os/execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` syscall handler in the StarkNet OS updates a contract's class hash in `contract_state_changes` without verifying that the new class hash has been declared in `contract_class_changes`. This is the direct analog of the external report's pattern: a weaker condition (the contract exists) is checked, but the required condition (the new class is declared) is not. An attacker-controlled contract can call `replace_class` with an arbitrary, undeclared class hash, permanently bricking the contract and freezing any funds it holds.

---

### Finding Description

In `execute_replace_class`, the OS reads the current state entry for the calling contract and writes a new `StateEntry` with the caller-supplied `class_hash`: [1](#0-0) 

The code contains an explicit, unresolved TODO at line 898:

```cairo
// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
```

No such check is performed. The `class_hash` from `ReplaceClassRequest` is taken directly from the syscall segment (i.e., from the executing contract's code) and written into `contract_state_changes` without any lookup into `contract_class_changes` to confirm the class was ever declared.

By contrast, the `execute_declare_transaction` flow enforces that a class hash is the result of a valid Sierra class hash computation and stores the corresponding `compiled_class_hash` in `contract_class_changes`: [2](#0-1) 

The `replace_class` path has no equivalent guard. The revert log does record the old class hash for rollback: [3](#0-2) 

However, if the transaction completes successfully after calling `replace_class` (without subsequently invoking itself under the new class), the invalid class hash is committed to the global state tree via `compute_contract_state_commitment`: [4](#0-3) 

Once committed, every future call to that contract will attempt to resolve the class hash against `contract_class_changes` and find nothing, causing permanent, unrecoverable execution failure.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A contract whose class hash is set to an undeclared value becomes permanently non-executable. Any ERC-20 tokens, ETH, or other assets held in that contract's storage are irrecoverably frozen. There is no upgrade path because `replace_class` itself requires the contract to execute, which is now impossible.

---

### Likelihood Explanation

**Medium-High.** The entry path requires only that an attacker control (or influence) a contract that calls `replace_class`. This is reachable by:

1. A malicious contract deployer who intentionally calls `replace_class` with a bogus hash before transferring funds into the contract (e.g., a shared vault or multisig).
2. Any contract whose `replace_class` logic accepts a caller-supplied class hash with insufficient on-chain validation — the OS is supposed to be the final enforcement layer, but that layer is absent here.

No privileged role, leaked key, or network-level attack is required. The syscall is available to any executing contract.

---

### Recommendation

Before writing the new `StateEntry`, verify that the requested `class_hash` exists in `contract_class_changes` (i.e., its `compiled_class_hash` is non-zero). Concretely, add a `dict_read` on `contract_class_changes` keyed by `class_hash` and assert the returned value is non-zero, mirroring the invariant enforced by `execute_declare_transaction`:

```cairo
// Verify the class has been declared.
let (compiled_class_hash) = dict_read{dict_ptr=contract_class_changes}(key=class_hash);
assert_not_zero(compiled_class_hash);
```

This resolves the acknowledged TODO and closes the gap between what `execute_declare_transaction` guarantees and what `execute_replace_class` currently enforces.

---

### Proof of Concept

1. Declare a valid class `C_valid` (class hash `H_valid`, compiled class hash `CC_valid`). This populates `contract_class_changes[H_valid] = CC_valid`.
2. Deploy a contract `V` (a shared vault) using `H_valid`.
3. Users deposit funds into `V`.
4. The attacker (who controls `V`'s logic, or can trigger a code path in it) calls `replace_class` with `H_bogus` — a felt value that was never declared and therefore has no entry in `contract_class_changes`.
5. `execute_replace_class` writes `StateEntry(class_hash=H_bogus, ...)` into `contract_state_changes` without any check.
6. The transaction completes successfully (no self-call under `H_bogus` occurs in this transaction).
7. `compute_contract_state_commitment` commits the new state, including `V`'s class hash = `H_bogus`, to the global Merkle tree.
8. Any subsequent `call_contract` or `invoke` targeting `V` causes the OS to look up `H_bogus` in `contract_class_changes`, find nothing, and fail to execute — permanently. All funds in `V` are frozen.
