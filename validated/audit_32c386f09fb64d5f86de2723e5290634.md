### Title
`deploy_token` Permanently DoS'd via Pre-Funding of Deterministic Mint PDA — (`File: solana/programs/bridge_token_factory/src/instructions/deploy_token.rs`)

---

### Summary

The `deploy_token` instruction uses Anchor's `init` constraint (not `init_if_needed`) to create the wrapped mint PDA. Because the PDA address is fully deterministic from the NEAR token name, any attacker can permanently block deployment of any specific bridged token on Solana by transferring even 1 lamport to the mint PDA address before `deploy_token` is ever called. This causes every subsequent `deploy_token` call for that token to fail unconditionally, permanently preventing the token from being bridged and freezing any in-flight NEAR-side locked funds.

---

### Finding Description

In `deploy_token.rs`, the `mint` account is declared with the `init` constraint:

```rust
#[account(
    init,
    payer = common.payer,
    seeds = [WRAPPED_MINT_SEED, data.payload.token.to_hashed_bytes().as_ref()],
    bump,
    mint::decimals = std::cmp::min(MAX_ALLOWED_DECIMALS, data.payload.decimals),
    mint::authority = authority,
)]
pub mint: Box<Account<'info, Mint>>,
``` [1](#0-0) 

The PDA seeds are `[b"wrapped_mint", token.to_hashed_bytes()]`, where `to_hashed_bytes()` is a pure, deterministic function of the NEAR token address string:

```rust
fn to_hashed_bytes(&self) -> [u8; 32] {
    let bytes = self.as_bytes();
    if bytes.len() > 32 {
        let hash = hash(bytes);
        hash.to_bytes()
    } else {
        let mut padded_bytes = [0u8; 32];
        padded_bytes[..bytes.len()].copy_from_slice(bytes);
        padded_bytes
    }
}
``` [2](#0-1) 

Anchor's `init` constraint internally calls the Solana system program's `create_account` instruction. `create_account` requires the target account to have **exactly 0 lamports**. In Solana, any party can transfer SOL to any address — including PDAs — without any signature from the PDA owner. If the mint PDA has even 1 lamport, `create_account` fails with a system program error, and the entire `deploy_token` transaction reverts.

Because the `init` constraint is used (not `init_if_needed`), there is no fallback path. Every future call to `deploy_token` for that token will fail at account validation before any program logic executes.

The `WRAPPED_MINT_SEED` constant is `b"wrapped_mint"`: [3](#0-2) 

---

### Impact Explanation

If `deploy_token` is permanently blocked for a given NEAR token:

1. The wrapped SPL mint for that token never exists on Solana.
2. Any subsequent `finalize_transfer` call for that token fails because the mint account does not exist.
3. Users who have already locked tokens on the NEAR side (via `init_transfer` on NEAR) expecting to receive wrapped tokens on Solana are permanently stuck — their NEAR-side funds are locked with no recourse.

This matches the allowed impact: **Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds in bridge, token, or vault flows.** [4](#0-3) 

---

### Likelihood Explanation

- The attacker needs no privileged access, no MPC signature, and no special role.
- The NEAR token address is public information (it is part of the bridge's public token registry).
- The mint PDA is fully computable off-chain from the token name and the program ID.
- The cost of the attack is 1 lamport (~$0.000000001) per token targeted.
- The attack can be executed preemptively against any token before it is ever deployed, or reactively against any token whose deployment transaction is observed.
- There is no on-chain recovery mechanism without a program upgrade.

---

### Recommendation

Replace the `init` constraint on `mint` with `init_if_needed`, and add a check that if the account already exists it is a valid mint with the correct authority and decimals. This mirrors the pump-science recommendation: check existence before attempting creation, and skip or validate rather than unconditionally failing.

Alternatively, add a guard in the instruction handler that checks whether the mint PDA already has lamports and, if so, attempts to reclaim/reallocate rather than calling `create_account`.

---

### Proof of Concept

```
// Off-chain attacker script (pseudo-code)
let token_name = "usdc.near";                          // public NEAR token address
let token_hash = to_hashed_bytes(token_name);          // deterministic
let (mint_pda, _bump) = Pubkey::find_program_address(
    &[b"wrapped_mint", &token_hash],
    &BRIDGE_PROGRAM_ID,
);

// Transfer 1 lamport to mint_pda — no signature from the program required
system_program::transfer(attacker_keypair, mint_pda, 1);

// Now, any deploy_token call for usdc.near will fail:
// Anchor's `init` → create_account → system program error:
//   "account already has lamports"
// The token can never be deployed. All in-flight NEAR→Solana
// transfers for usdc.near are permanently frozen.
