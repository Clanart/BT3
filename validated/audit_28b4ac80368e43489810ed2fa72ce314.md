Audit Report

## Title
Unguarded `refundETH()` and Aggregate-Balance `pay()` Enable Theft of Stranded Native ETH — (File: `metric-periphery/contracts/base/PeripheryPayments.sol`)

## Summary

`refundETH()` has no access control and sends the router's entire native ETH balance to any caller. `pay()` consumes `address(this).balance` — the router's aggregate ETH — rather than only the current transaction's `msg.value`. When a user calls a `payable` swap function with excess ETH and omits a trailing `refundETH()`, the surplus ETH is stranded on the router and immediately claimable by any third party.

## Finding Description

**ETH stranding path:** `exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`, and `multicall` are all `payable`. A user who sends `msg.value > amountIn` without appending `refundETH()` leaves the surplus on the router. The `receive()` guard at L32–34 only blocks direct ETH pushes (`msg.sender != WETH` reverts); it does not prevent ETH from arriving via `payable` function calls.

**Aggregate balance in `pay()`:** At L74, `pay()` reads `address(this).balance` — the router's total ETH — not the current transaction's `msg.value`. When `nativeBalance >= value`, the router wraps and forwards that ETH to the pool without pulling from the payer's WETH allowance. Any ETH left by a prior user is silently consumed to cover the current caller's obligation.

**Unguarded `refundETH()`:** At L58–63, `refundETH()` sends the router's entire ETH balance to `msg.sender` with no restriction on who may call it. The test at `MetricOmmSimpleRouter.payments.t.sol` L74–85 explicitly confirms this: `vm.deal(address(router), amount)` followed by `vm.prank(swapper); router.refundETH()` drains the full balance to an arbitrary caller.

**Exploit flow:**
1. Victim calls `exactInputSingle{value: 1 ether}(amountIn=500)` without `refundETH()`. `pay()` uses 500 wei; ~1 ether − 500 wei remains on the router.
2a. Attacker calls `router.refundETH()` → receives the victim's stranded ETH directly.
2b. Alternatively, attacker calls `exactInputSingle(tokenIn=WETH, amountIn=500)`. `pay()` sees `address(this).balance >= 500`, wraps the victim's ETH as WETH, and transfers it to the pool. Attacker receives token output at zero WETH cost.

The test `test_multicall_ethInput_exactInputSingle_refundsUnusedEth` at L106–133 of `MetricOmmSimpleRouter.native.t.sol` documents the required pattern (include `refundETH()` as step 2); omitting it is the exact precondition for the exploit.

## Impact Explanation

Direct loss of user ETH principal. Any ETH stranded on the router — from a missed `refundETH()` or an overpayment — is immediately claimable by any address. Both theft paths (direct drain via `refundETH()` and free WETH swap via `pay()`) result in complete loss of the victim's surplus ETH with no recourse. This meets the High severity threshold for direct loss of user principal.

## Likelihood Explanation

Medium-to-High. The pattern that creates stranded ETH is explicitly tested and documented as the expected usage model. Any user who sends `msg.value > amountIn` without a trailing `refundETH()` — a common mistake — leaves ETH on the router. Front-running bots monitoring the mempool can detect such transactions and call `refundETH()` in the next block. The attack requires no special privileges, no malicious setup, and is repeatable indefinitely.

## Recommendation

1. **Track per-transaction ETH via transient storage:** At the start of each `payable` entry point, store `msg.value` in a transient slot. In `pay()`, limit native ETH consumption to that stored amount rather than `address(this).balance`. Clear the slot at the end of the call.
2. **Restrict `refundETH()` to the depositor:** Record the depositing address in transient storage at the `payable` entry point and enforce it in `refundETH()`.
3. **Alternatively, pass `msg.value` explicitly** through the call chain to `pay()` instead of reading `address(this).balance`, so only the current transaction's ETH is eligible for wrapping.

## Proof of Concept

```
Setup:
  - Router deployed with WETH and factory (existing test infrastructure)
  - Pool seeded with WETH/token1 liquidity
  - Victim has 1 ETH

Step 1 — Victim strands ETH:
  victim calls exactInputSingle{value: 1 ether}(tokenIn=WETH, amountIn=500, ...)
  // refundETH() omitted
  → pay() reads address(this).balance = 1 ether >= 500
  → router wraps 500 wei as WETH, transfers to pool
  → 1 ether - 500 wei remains on router

Step 2a — Attacker drains via refundETH():
  attacker calls router.refundETH()
  → attacker receives (1 ether - 500 wei)
  → victim's ETH is gone

Step 2b — Attacker executes free WETH swap:
  attacker calls exactInputSingle(tokenIn=WETH, amountIn=500, ...)
  → pay(WETH, attacker, pool, 500) called
  → address(this).balance = (1 ether - 500 wei) >= 500
  → router wraps victim's ETH as WETH, sends to pool
  → attacker receives token1 output without spending any WETH

Foundry test skeleton:
  function test_strandedEthTheft() public {
      // victim sends excess ETH, omits refundETH
      vm.prank(victim);
      router.exactInputSingle{value: 1 ether}(params_amountIn_500);
      assertGt(address(router).balance, 0);

      // attacker drains
      uint256 before = attacker.balance;
      vm.prank(attacker);
      router.refundETH();
      assertGt(attacker.balance, before);
      assertEq(address(router).balance, 0);
  }
```
