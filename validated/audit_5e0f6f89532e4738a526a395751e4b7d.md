### Title
`ProofOfInclusion::valid()` Never Verifies Against an External Trusted Root — Forged DataLayer Inclusion Proofs Always Pass - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` in the DataLayer crate performs only a self-referential internal consistency check. Its final assertion `existing_hash == self.root_hash()` is tautologically true after the loop body passes, because `root_hash()` returns `last.combined_hash` — a field that is itself part of the attacker-controlled proof. No external trusted root is ever compared. An attacker can construct a fully fabricated `ProofOfInclusion` for any arbitrary key-value pair, in any self-chosen root, and `valid()` will return `true`.

### Finding Description

`ProofOfInclusion` is a `Streamable` type — it is serialized and deserialized across the network boundary. [1](#0-0) 

Its `root_hash()` method derives the root entirely from the proof's own last layer: [2](#0-1) 

The `valid()` method iterates layers, checking that each `calculated_hash == layer.combined_hash`, then performs the final check `existing_hash == self.root_hash()`: [3](#0-2) 

After the loop, `existing_hash` holds the last `calculated_hash`. The loop already enforced `calculated_hash == layer.combined_hash` for every layer, so `existing_hash` is always equal to `last.combined_hash`. Since `self.root_hash()` also returns `last.combined_hash`, the final assertion `existing_hash == self.root_hash()` is **unconditionally true** whenever the loop body passes. No external root is ever consulted.

Contrast this with the consensus-layer `validate_merkle_proof` function, which correctly accepts an external root hash as a parameter and verifies the proof against it: [4](#0-3) 

The DataLayer `ProofOfInclusion` exposes `valid()` and `root_hash()` directly to Python via `pymethods`: [5](#0-4) 

Python callers and any Rust callers that rely solely on `proof.valid()` — as all existing tests do — receive no protection against a forged proof: [6](#0-5) 

### Impact Explanation

An attacker who can supply a `ProofOfInclusion` object (e.g., over the DataLayer peer protocol, via a deserialized `Streamable` message, or through the Python binding) can:

1. Choose any `node_hash` — claiming any key-value pair is a leaf.
2. Choose any `other_hash` and `other_hash_side` values.
3. Compute `combined_hash = calculate_internal_hash(node_hash, side, other_hash)` to satisfy the loop invariant.
4. The resulting proof passes `valid()` and reports an attacker-chosen `root_hash()`.

If the verifier does not separately compare `proof.root_hash()` against a locally-trusted tree root (which the API does not require or document), the forged proof is accepted as valid inclusion evidence. This matches the allowed High impact: **DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion and lets untrusted input prove invalid state.**

### Likelihood Explanation

`ProofOfInclusion` is a `Streamable` type designed for network transmission. The `valid()` method is the sole validation entry point exposed to both Rust and Python callers. No existing call site in the repository checks `proof.root_hash()` against an independent trusted root after calling `valid()`. Any DataLayer peer or client that receives a proof from an untrusted source and calls only `proof.valid()` is exploitable without any cryptographic work.

### Recommendation

Replace the self-referential `valid()` with a method that accepts an external trusted root:

```rust
pub fn valid_for_root(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;
    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );
        if calculated_hash != layer.combined_hash {
            return false;
        }
        existing_hash = calculated_hash;
    }
    &existing_hash == trusted_root  // compare against EXTERNAL root, not self.root_hash()
}
```

Deprecate or remove the current `valid()` method, or make it clearly documented as an internal-consistency-only check that must never be used as a security gate. Update all call sites — including the Python binding — to pass the locally-trusted root.

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side};
use chia_datalayer::ProofOfInclusion;
use chia_datalayer::ProofOfInclusionLayer;

fn forge_proof(fake_node_hash: Hash, other_hash: Hash) -> ProofOfInclusion {
    // Compute combined_hash so the loop invariant holds
    let combined_hash = chia_datalayer::calculate_internal_hash(
        &fake_node_hash,
        Side::Left,
        &other_hash,
    );
    ProofOfInclusion {
        node_hash: fake_node_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Left,
            other_hash,
            combined_hash,
        }],
    }
}

fn main() {
    let fake_node = [0xAA; 32];
    let other    = [0xBB; 32];
    let proof = forge_proof(fake_node, other);
    // valid() returns true for a completely fabricated proof
    assert!(proof.valid(), "forged proof accepted");
    // root_hash() is attacker-controlled — not the real tree root
    println!("Forged root: {:?}", proof.root_hash());
}
```

The forged proof passes `valid()` and reports an attacker-chosen root, with no connection to any real `MerkleBlob` state. [3](#0-2) [2](#0-1)
