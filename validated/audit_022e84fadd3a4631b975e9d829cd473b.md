### Title
Missing Pause Guard in `deploy_token` Allows Token Deployment and Wormhole Message Emission During Emergency Pause — (`solana/programs/bridge_token_factory/src/lib.rs`)

---

### Summary

`deploy_token` in `lib.rs` has no `require!` check against `config.paused`, unlike every other state-mutating instruction (`finalize_transfer`, `finalize_transfer_sol`, `init_transfer`, `init_transfer_sol`). When `config.paused == ALL_PAUSED`, an unprivileged relayer holding a previously-issued valid `SignedPayload<DeployTokenPayload>` can still call `deploy_token`, create a mint PDA, and post a Wormhole message — all operations that should be blocked during an emergency pause.

---

### Finding Description

**Pause flag definitions** (`constants.rs`):

```
INIT_TRANSFER_PAUSED     = 1 << 0  (0x01)
FINALIZE_TRANSFER_PAUSED = 1 << 1  (0x02)
ALL_PAUSED               = 0x01 | 0x02 = 0x03
```

There is no `DEPLOY_TOKEN_PAUSED` bit defined, and no pause check in `deploy_token`. [1](#0-0) 

**`finalize_transfer` and `init_transfer` both have explicit guards:** [2](#0-1) [3](#0-2) 

**`deploy_token` has none:** [4](#0-3) 

The instruction proceeds directly to signature verification and then `initialize_token_metadata`, which:
1. Creates a mint PDA (via `init` constraint on `mint` account)
2. Creates Metaplex metadata accounts
3. Calls `self.common.post_message(payload)` — emitting a Wormhole cross-chain message [5](#0-4) 

**Replay protection analysis**: The mint PDA is seeded by `[WRAPPED_MINT_SEED, token.to_hashed_bytes()]`, so each unique token name can only be deployed once. However, a relayer can hold a valid signed payload for a token that has not yet been deployed and submit it after the pause is set. [6](#0-5) 

---

### Impact Explanation

During an emergency pause (e.g., triggered because a critical exploit was discovered in the bridge), `deploy_token` remains callable by any relayer holding a valid MPC-signed payload. This:

- Creates a new mint PDA on Solana, permanently registering a new wrapped token
- Posts a `DeployTokenResponse` Wormhole message that NEAR will process (NEAR is not paused), registering the Solana mint in the NEAR token registry

This desynchronizes the NEAR token registry with Solana state in a way that cannot be undone without admin intervention, and could enable subsequent exploits if the emergency pause was triggered precisely to prevent new token registrations (e.g., due to a flaw in the token deployment flow itself).

---

### Likelihood Explanation

Relayers routinely hold signed payloads that have not yet been submitted. The window between MPC signing and on-chain submission is normal operational behavior. An attacker (or a relayer acting in bad faith) who holds any unsubmitted `SignedPayload<DeployTokenPayload>` can exploit this immediately after a pause is set. No privileged access, key compromise, or MPC collusion is required — the MPC signature was legitimately issued before the pause.

---

### Recommendation

Add a pause bit for `deploy_token` and enforce it in `lib.rs`:

In `constants.rs`:
```rust
pub const DEPLOY_TOKEN_PAUSED: u8 = 1 << 2;
pub const ALL_PAUSED: u8 = INIT_TRANSFER_PAUSED | FINALIZE_TRANSFER_PAUSED | DEPLOY_TOKEN_PAUSED;
```

In `lib.rs`, at the top of `deploy_token`:
```rust
require!(
    ctx.accounts.common.config.paused & DEPLOY_TOKEN_PAUSED == 0,
    error::ErrorCode::Paused
);
```

---

### Proof of Concept

1. Admin calls `pause()` → `config.paused = ALL_PAUSED (0x03)` [7](#0-6) 

2. Attacker calls `deploy_token(signed_payload)` where `signed_payload` is any previously-issued valid `SignedPayload<DeployTokenPayload>` for a token not yet deployed.

3. `lib.rs:deploy_token` has no pause check — execution proceeds to `verify_signature` (passes, payload is legitimately signed) and then `initialize_token_metadata`. [4](#0-3) 

4. Mint PDA is created, Metaplex metadata is initialized, and `post_message` emits a Wormhole message — all while the bridge is fully paused. [8](#0-7)
