### Title
Non-EVM-Native (IBC) Token Balances Orphaned and Stealable After Contract Self-Destruct — (`x/evm/statedb/statedb.go`)

### Summary

When a contract self-destructs, `StateDB.Commit()` burns only the EVM-denom balance and then calls `DeleteAccount` to remove the auth account and storage. Non-EVM-native tokens (IBC tokens, CosmWasm bridge tokens) held by the destroyed address are explicitly skipped. The bank module retains those balances at the now-ownerless address. Because the auth account is gone, the original contract can never reclaim them; but an attacker who knows the CREATE2 salt can redeploy a new contract at the identical address and drain the orphaned tokens.

### Finding Description

In `StateDB.Commit()`, the self-destruct branch reads:

```go
// Only the EVM denom is burned here. Non-EVM-native tokens (IBC, CosmWasm
// bridge) held by the destroyed address are not drained and may remain as
// orphaned bank balances.
if remaining := s.keeper.GetBalance(cacheCtx, cosmosAddr, s.evmDenom); remaining.Sign() > 0 {
    ...SubBalance(cacheCtx, cosmosAddr, coin)...
}
if err := s.keeper.DeleteAccount(cacheCtx, obj.Address()); err != nil { ... }
writeCache()
``` [1](#0-0) 

`DeleteAccount` removes the auth account and clears EVM storage, but never touches the bank module for any denom other than the EVM denom:

```go
// clear storage
k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool { ... })
// remove auth account
k.accountKeeper.RemoveAccount(ctx, acct)
// NOTE: balance should be cleared separately
``` [2](#0-1) 

Under EIP-6780 (`SelfDestruct6780`), full account deletion only fires when the contract was created in the same transaction (`stateObject.newContract == true`): [3](#0-2) 

The attack sequence in a single transaction:
1. Attacker pre-computes a CREATE2 address and arranges for IBC tokens to be present at that address (e.g., via a prior IBC transfer to the predicted address, or via a precompile/native-action call inside the same tx).
2. Attacker deploys the contract at that address via CREATE2 — `stateObject.newContract` is set to `true`.
3. Attacker calls `SELFDESTRUCT` on the contract in the same transaction — `SelfDestruct6780` delegates to `SelfDestruct`, marking the account for deletion.
4. At `StateDB.Commit()`, the EVM denom is burned and the auth account is deleted, but IBC token balances remain in the bank module at the now-ownerless address.
5. In a subsequent transaction, the attacker redeploys a new contract at the same CREATE2 address. The bank module still holds the IBC tokens there; the new contract can transfer them out via a precompile or native action.

### Impact Explanation

Non-EVM-native Cosmos bank tokens (IBC denoms, CosmWasm bridge tokens) held by a self-destructed contract are permanently orphaned or recoverable only by whoever redeploys at the same address. This constitutes a loss of user funds: tokens sent to a contract address (e.g., via IBC transfer) that subsequently self-destructs are silently removed from the original owner's control and can be stolen by redeploying via CREATE2. This matches the allowed High impact: "EVM state transition… bug that permits… valid user funds/fees to be mis-accounted."

### Likelihood Explanation

- Cosmos/Ethermint chains routinely hold IBC tokens in EVM contracts (DEXes, bridges, wrapped-token vaults).
- The EIP-6780 same-tx create-and-destroy path is reachable by any unprivileged user who can submit an EVM transaction.
- CREATE2 address prediction is deterministic and publicly computable.
- No privileged role, governance action, or validator collusion is required.

### Recommendation

In the self-destruct branch of `StateDB.Commit()`, after burning the EVM denom, iterate over **all** bank balances held by the destroyed address and burn or escrow them before calling `DeleteAccount`. The `BankKeeper` interface already exposes `GetAllBalances` (or equivalent) for this purpose. Alternatively, block `DeleteAccount` from completing if the address holds any non-EVM-native balance, forcing the contract to drain those tokens before self-destructing.

### Proof of Concept

```
Block N, single transaction:
  1. CALL factory.attackInOneTx(salt) { value: 0 }
     a. CREATE2 → childAddr (newContract = true)
     b. IBC tokens already at childAddr (pre-sent in a prior IBC transfer)
     c. CALL childAddr.destroy() → SELFDESTRUCT(factory)
        - SelfDestruct6780 fires, EVM denom burned, account marked deleted
     d. StateDB.Commit():
        - EVM denom at childAddr → burned ✓
        - IBC denom at childAddr → NOT touched ✗
        - DeleteAccount(childAddr) → auth account removed
     e. childAddr now has: no auth account, no EVM balance, but IBC balance > 0

Block N+1, attacker transaction:
  2. CALL factory.redeployChild(salt)
     a. CREATE2 → childAddr (same address, new auth account created)
     b. Bank module still holds IBC tokens at childAddr
     c. New contract calls precompile/native-action to transfer IBC tokens to attacker
```

The root cause is the explicit skip at `x/evm/statedb/statedb.go` lines 813–815, which leaves non-EVM-native bank balances unhandled during the self-destruct commit path. [4](#0-3) [5](#0-4)

// DeleteAccount handles contract's suicide call:
// - remove code
// - remove states
// - remove auth account
//
// NOTE: balance should be cleared separately
func (k *Keeper) DeleteAccount(ctx sdk.Context, addr common.Address) error {
	cosmosAddr := sdk.AccAddress(addr.Bytes())
	acct := k.accountKeeper.GetAccount(ctx, cosmosAddr)
	if acct == nil {
		return nil
	}

	// NOTE: only Ethereum accounts (contracts) can be selfdestructed
	_, ok := acct.(ethermint.EthAccountI)
	if !ok {
		return errorsmod.Wrapf(types.ErrInvalidAccount, "type %T, address %s", acct, addr)
	}

	// clear storage
	k.ForEachStorage(ctx, addr, func(key, _ common.Hash) bool {
		k.SetState(ctx, addr, key, nil)
		return true
	})

	// remove auth account
	k.accountKeeper.RemoveAccount(ctx, acct)

	k.debugLog(ctx, "account suicided",
		"ethereum-address", addr,
		"cosmos-address", cosmosAddr,
	)

	return nil
