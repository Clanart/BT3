### Title
Missing Recipient Validation in `initTransfer` Allows Permanent Irrecoverable Lock of User Assets - (File: `evm/src/omni-bridge/contracts/OmniBridge.sol`)

---

### Summary

`OmniBridge.initTransfer` and `OmniBridge.initTransfer1155` accept a `string calldata recipient` parameter with no validation that it is non-empty or parseable as a valid `OmniAddress`. A user who passes an empty or malformed recipient string will have their tokens burned or permanently locked on the EVM side, while the NEAR side will be unable to finalize the transfer, resulting in an irrecoverable loss of assets.

---

### Finding Description

`initTransfer` burns bridge tokens or locks native ERC20/ETH in the contract before emitting an `InitTransfer` event. The `recipient` field is a free-form string (e.g., `"near:alice.near"`) that is never validated on-chain. [1](#0-0) 

After tokens are burned/locked, the NEAR relayer picks up the `InitTransfer` event and calls `fin_transfer` on NEAR. During proof verification, the NEAR prover parses the recipient string via `event.data.recipient.parse().map_err(stringify)?`: [2](#0-1) 

An empty string `""` cannot be parsed as an `OmniAddress`. This causes `TryFromLog` to return an `Err`, which propagates to `fin_transfer_callback` as `BridgeError::InvalidProofMessage`, causing a panic. The destination nonce is never recorded, so the transfer cannot be retried or finalized. [3](#0-2) 

The same issue exists in `initTransfer1155`: [4](#0-3) 

And in the StarkNet `init_transfer`: [5](#0-4) 

The NEAR `init_transfer` also lacks a zero-address check on the `OmniAddress` recipient — `is_zero()` exists but is never called as a guard: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

- **Bridge tokens** (`isBridgeToken[tokenAddress]`): `BridgeToken.burn` is called before the event is emitted. The tokens are destroyed. No recovery is possible.
- **Native ERC20 tokens**: Locked in the bridge contract with no on-chain rescue function.
- **Native ETH** (`tokenAddress == address(0)`): Locked in the bridge contract.

In all cases the NEAR side cannot finalize the transfer (recipient parse fails), so the assets are permanently unclaimable. This matches: **Critical — Permanent freezing, irrecoverable lock, or unclaimable settlement of user or protocol funds.** [8](#0-7) 

---

### Likelihood Explanation

Any unprivileged EVM user calling `initTransfer` can trigger this by passing `recipient = ""` (empty string) or any syntactically invalid OmniAddress string. This is a realistic user mistake (e.g., forgetting to fill in the recipient field, passing a raw address without the chain prefix). No special privileges or colluding parties are required. The Solana bridge's own `SECURITY.md` explicitly acknowledges the identical issue for Solana, confirming the team is aware of the pattern but has not addressed it on EVM or StarkNet: [9](#0-8) 

---

### Recommendation

Add a non-empty recipient check in `initTransfer` and `initTransfer1155` before any token transfer occurs:

```solidity
require(bytes(recipient).length > 0, "ERR_EMPTY_RECIPIENT");
```

Optionally, add a stricter format check (e.g., require a `:` separator matching a known chain prefix). On the NEAR side, add a `require!(!init_transfer_msg.recipient.is_zero(), ...)` guard analogous to the existing chain-kind check.

---

### Proof of Concept

1. User holds 1000 `wNEAR` bridge tokens on EVM.
2. User calls:
   ```solidity
   OmniBridge.initTransfer(
       wNearTokenAddress,
       1000,
       0,      // fee
       0,      // nativeFee
       "",     // recipient — empty string
       ""
   );
   ```
3. `BridgeToken(wNearTokenAddress).burn(msg.sender, 1000)` executes — tokens destroyed.
4. `InitTransfer` event emitted with `recipient = ""`.
5. NEAR relayer submits proof to `fin_transfer`.
6. `fin_transfer_callback` calls `Self::decode_prover_result(0)`, which internally calls `event.data.recipient.parse()` on `""` — parse fails, callback panics with `InvalidProofMessage`.
7. Transfer is never finalized. Tokens are permanently lost. [10](#0-9) [11](#0-10) [12](#0-11)
