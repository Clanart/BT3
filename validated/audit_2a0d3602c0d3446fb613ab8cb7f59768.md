### Title
Unrestricted `replace_class` Syscall Accepts Undeclared Class Hashes, Enabling Permanent Freezing of Contract Funds — (File: `execution/syscall_impls.cairo`)

---

### Summary

The `execute_replace_class` function in `syscall_impls.cairo` does not validate that the new class hash provided by a contract corresponds to a declared class. Any contract can call `replace_class` with an arbitrary, undeclared class hash. The OS commits this invalid state change. Any subsequent call to the affected contract causes an irrecoverable proof failure inside `execute_entry_point`, permanently bricking the contract and freezing all funds held within it.

---

### Finding Description

In `execute_replace_class` (lines 878–916 of `syscall_impls.cairo`), the OS accepts any class hash from the contract's syscall request and writes it directly into `contract_state_changes` without checking whether it corresponds to a declared class:

```cairo
let class_hash = request.class_hash;

// TODO(Yoni, 1/1/2026): Check that there is a declared contract class with the given hash.
local state_entry: StateEntry*;
%{ GetContractAddressStateEntry %}

tempvar new_state_entry = new StateEntry(
    class_hash=class_hash, storage_ptr=state_entry.storage_ptr, nonce=state_entry.nonce
);

dict_update{dict_ptr=contract_state_changes}(
    key=contract_address,
    prev_value=cast(state_entry, felt),
    new_value=cast(new_state_entry, felt),
);
``` [1](#0-0) 

The TODO comment at line 898 explicitly acknowledges the missing validation. The gas check at line 888 passes, the state entry is updated, and the block is committed — all without any check that `class_hash` is a known, declared class.

When a future transaction calls this contract, `execute_entry_point` performs the following sequence:

1. Reads the compiled class hash from `contract_class_changes` via `dict_read` using the (now-invalid) class hash as the key. Because the invalid hash was never declared, `dict_read` returns `0` (the default uninitialized value).
2. Calls `find_element` on the compiled class facts bundle with `key=0`. [2](#0-1) 

If no compiled class fact with hash `0` exists (the normal case), `find_element` raises a Cairo assertion failure. This is not a graceful revert — it is a proof-generation failure. The sequencer cannot produce a valid STARK proof for any block that includes a call to the bricked contract. The state change (the invalid class hash) was already committed in the prior block, so the contract is permanently inaccessible.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Once a contract's class hash is replaced with an undeclared value and the block is finalized:

- All future calls to the contract cause an irrecoverable OS-level failure (not a revert, but a proof failure).
- The sequencer must permanently exclude all transactions targeting the contract.
- Any ERC-20 tokens, bridged ETH, or other assets held in the contract's storage are permanently frozen with no recovery path.
- The state cannot be corrected without a protocol-level intervention (hard fork or emergency upgrade).

---

### Likelihood Explanation

The attack is reachable by any unprivileged contract deployer:

1. An attacker deploys a contract with a legitimate-looking interface (e.g., a yield vault, bridge, or token contract).
2. Users deposit funds into the contract.
3. The attacker calls a function that internally invokes the `replace_class` syscall with an arbitrary undeclared felt value (e.g., `0xdeadbeef`).
4. The OS accepts the syscall via `execute_replace_class` without validation and commits the state.
5. All future calls to the contract fail at the proof level; user funds are permanently frozen.

No privileged access, leaked keys, or external dependency compromise is required. The `replace_class` syscall is a standard, gas-metered operation available to any contract.

---

### Recommendation

Before updating `contract_state_changes`, validate that `request.class_hash` is present in `contract_class_changes` (i.e., it has been declared in the current or a prior block). The check should assert that `dict_read{dict_ptr=contract_class_changes}(key=class_hash)` returns a non-zero compiled class hash, confirming the class is declared. This is exactly what the existing TODO comment at line 898 calls for.

---

### Proof of Concept

**Block N (attack block):**

1. Attacker's contract calls `replace_class(0xdeadbeef)` where `0xdeadbeef` is not a declared class hash.
2. `execute_replace_class` in `syscall_impls.cairo` (line 896) reads `class_hash = 0xdeadbeef` from the request.
3. No validation is performed (line 898 TODO).
4. `dict_update` writes `StateEntry(class_hash=0xdeadbeef, ...)` for the contract address into `contract_state_changes`.
5. The block is proven and finalized. The state root now encodes `class_hash=0xdeadbeef` for the contract.

**Block N+1 (any call to the bricked contract):**

1. A user submits a transaction calling the bricked contract.
2. `execute_entry_point` calls `dict_read{dict_ptr=contract_class_changes}(key=0xdeadbeef)` → returns `0` (undeclared).
3. `find_element` is called with `key=0` on the compiled class facts bundle.
4. No compiled class fact with hash `0` exists → `find_element` assertion fails.
5. The OS Cairo program cannot complete; no valid proof can be generated for this block.
6. The sequencer must drop the transaction and permanently exclude the contract.
7. All funds in the contract are permanently frozen. [3](#0-2) [4](#0-3)
