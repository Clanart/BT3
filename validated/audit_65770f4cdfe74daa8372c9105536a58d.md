### Title
EIP-7702 Authorization Nonce Permanently Consumed via Front-Running with a Failing Transaction — (`x/evm/keeper/state_transition.go`, `x/evm/keeper/set_code_authorizations.go`)

### Summary
An unprivileged attacker can observe a victim's pending EIP-7702 `SetCode` transaction in the mempool, extract the victim's signed authorization, and include it in their own transaction that deliberately fails (EVM revert or post-hook failure). Because Ethermint commits EIP-7702 authorization effects (nonce bump + delegation code) durably even on failed execution, the victim's authorization nonce is permanently consumed. The victim's own transaction then silently skips the authorization (nonce mismatch), and the victim must sign a new authorization with an updated nonce to recover.

### Finding Description

**Root cause — authorization nonce committed before EVM execution and not reverted on failure:**

In `ApplyMessageWithConfig`, for a CALL transaction carrying `SetCodeAuthorizations`, `applyAuthorization` is called on the `stateDB` **before** `evm.Call()`:

```go
// x/evm/keeper/state_transition.go lines 492-517
for _, auth := range msg.SetCodeAuthorizations {
    authority, err := k.applyAuthorization(&auth, stateDB)
    ...
    validAuths = append(validAuths, validAuth{auth: auth, authority: authority})
}
...
ret, leftoverGas, vmErr = evm.Call(sender, *msg.To, msg.Data, leftoverGas, ...)
```

`applyAuthorization` calls `setAuthorizationDelegation`, which bumps the authority nonce and sets the delegation code in `stateDB`:

```go
// x/evm/keeper/set_code_authorizations.go lines 74-84
func (k *Keeper) setAuthorizationDelegation(...) {
    stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
    stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), ...)
}
```

When `evm.Call()` reverts (`vmErr != nil`), the EVM's internal snapshot rolls back only state changes made **inside** the EVM call. The authorization nonce bump and code write were applied to `stateDB` **outside** the EVM snapshot scope and are therefore not reverted. `stateDB.Commit()` is then called unconditionally at line 603, persisting these effects.

**Ethermint-specific extension — `commitDurableAuthorization` survives post-hook failure:**

`ApplyTransaction` introduces an additional Ethermint-specific mechanism not present in go-ethereum. When `k.hooks != nil`, a sibling cache context (`durableAuthorizationCtx`) is created and the authorization effects are replayed into it via `applyDurableAuthorization`. This durable context is committed even when the post-hook fails and `tmpCtx` is discarded:

```go
// x/evm/keeper/state_transition.go lines 186-191
if len(msg.SetCodeAuthorizations) > 0 {
    var durableAuthorizationCtx sdk.Context
    durableAuthorizationCtx, commitDurableAuthorization = ctx.CacheContext()
    cfg.DurableSetCodeAuthorizationCtx = &durableAuthorizationCtx
}
```

```go
// x/evm/keeper/state_transition.go lines 266-268
if commitDurableAuthorization != nil && !tmpCtxCommitted {
    commitDurableAuthorization()
}
```

`tmpCtxCommitted` is only `true` when both the EVM call and the post-hook succeed. Any failure (vmErr or hook error) leaves `tmpCtxCommitted = false`, triggering `commitDurableAuthorization()`.

**Validation check that fails for the victim:**

```go
// x/evm/keeper/set_code_authorizations.go line 39
if have := stateDB.GetNonce(authority); have != auth.Nonce {
    return authority, core.ErrAuthorizationNonceMismatch
}
```

After the attacker's transaction consumes the nonce (bumping it from N to N+1), the victim's transaction carries `auth.Nonce = N` but the on-chain nonce is now N+1. The authorization is silently skipped (errors are ignored in the loop at line 494-498), and the victim's delegation is not re-applied in their own transaction context.

### Impact Explanation

An attacker can permanently consume a victim's EIP-7702 authorization nonce without the victim's consent. The victim's account delegation code is set by the attacker's transaction (to the address the victim intended), but the victim's own transaction fails to apply the authorization. The victim's authorization is spent and cannot be reused; they must sign a new authorization with nonce N+1. If the victim's transaction logic depended on the delegation being applied atomically with the EVM call (e.g., a self-delegating contract creation), the call executes in an unexpected state. This constitutes unauthorized account/code mutation via EIP-7702 authorization replay.

### Likelihood Explanation

EIP-7702 authorizations are embedded in the transaction body and are fully visible in the public mempool. The outer transaction signer is independent of the authorization signer, so any attacker can extract a victim's signed authorization and include it in their own transaction. The attacker only needs to submit a transaction targeting a reverting contract (trivially achievable) or one that triggers a post-hook failure. This is a low-cost, low-skill attack requiring only mempool observation and a single transaction submission.

### Recommendation

1. **Validate authorization nonce at commit time, not only at apply time**: Re-check the authority nonce against the committed on-chain state inside `stateDB.Commit()` or immediately before committing the durable authorization context, to detect if the nonce was already consumed by a concurrent transaction.

2. **Decouple durable commit from failed execution**: Only call `commitDurableAuthorization()` when the authorization was applied in a transaction that the victim themselves submitted (i.e., where `msg.From == authority`), or require the outer signer to be the authority for self-delegations.

3. **Consider requiring the outer transaction signer to match the authorization authority** for self-delegation cases, preventing third-party inclusion of victim authorizations in failing transactions.

### Proof of Concept

```
1. Victim (authority address A) signs:
   auth = SignSetCode(A_key, {chainID, delegate=D, nonce=N})
   
2. Victim broadcasts SetCode tx:
   Tx_victim = {from: A, to: A, authList: [auth], ...}

3. Attacker observes Tx_victim in mempool, extracts auth.

4. Attacker broadcasts:
   Tx_attacker = {from: Attacker, to: RevertingContract, authList: [auth], ...}
   (RevertingContract always reverts, or attacker triggers post-hook failure)

5. Tx_attacker is included first:
   - applyAuthorization(auth, stateDB) → stateDB.SetNonce(A, N+1), stateDB.SetCode(A, delegation_D)
   - evm.Call() reverts (vmErr) OR post-hook fails
   - stateDB.Commit() OR commitDurableAuthorization() persists nonce=N+1 and code=delegation_D

6. Tx_victim is included:
   - validateAuthorization: stateDB.GetNonce(A) = N+1 ≠ auth.Nonce = N → ErrAuthorizationNonceMismatch
   - Authorization silently skipped
   - Victim's EVM call executes without re-applying the authorization
   - Victim's authorization is permanently consumed; must sign new auth with nonce=N+1

```go
func (k *Keeper) setAuthorizationDelegation(auth *types.SetCodeAuthorization, authority common.Address, stateDB vm.StateDB) {
	// Update nonce and account code.
	stateDB.SetNonce(authority, auth.Nonce+1, tracing.NonceChangeAuthorization)
	if auth.Address == (common.Address{}) {
		// Delegation to zero address means clear.
		stateDB.SetCode(authority, nil, tracing.CodeChangeAuthorizationClear)
		return
	}

	// Otherwise install delegation to auth.Address.
	stateDB.SetCode(authority, types.AddressToDelegation(auth.Address), tracing.CodeChangeAuthorization)
```
